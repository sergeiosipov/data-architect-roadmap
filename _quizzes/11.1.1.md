# Phase 6 · 11.1.1 Identity providers & SSO — 100 self-test questions

Q: What is an identity provider (IdP) and why does every access decision start with it?
A: An IdP is the authoritative system that authenticates users and asserts who they are, issuing tokens or assertions that downstream apps trust instead of running their own login. Every access decision starts with identity because authorization (what you can do) is meaningless until authentication (who you are) is established. Centralising this in one IdP means a single change — a disabled account or a group edit — propagates everywhere, which is exactly what makes leaver and audit questions answerable.

Q: Define SSO (single sign-on) and distinguish it from the IdP itself.
A: SSO is the user experience of authenticating once and then accessing multiple applications without logging in again, achieved by those apps delegating authentication to a shared IdP. The IdP is the component that actually holds the session and issues tokens; SSO is the property that emerges when many relying parties trust that same IdP. So you build an IdP (e.g. Keycloak) and the result your users see is SSO across Trino, Grafana, and OpenMetadata.

Q: What is OpenID Connect (OIDC) and what is it built on top of?
A: OIDC is an interoperable authentication protocol layered on top of the OAuth 2.0 authorization framework, adding a standardised identity layer. OAuth 2.0 alone only deals with delegated authorization (access tokens for APIs); OIDC adds the `id_token` and a defined set of identity claims so a client can actually know who the user is. Because it uses JSON Web Tokens and JSON, it is simpler to consume than the XML-heavy SAML it largely replaces.

Q: What is the authorization code flow in OIDC, at a high level?
A: The authorization code flow has the user authenticate at the IdP, which redirects back to the client with a short-lived single-use `code`, and the client then exchanges that code at the token endpoint for tokens over a back-channel call. The two-step design means tokens never travel through the browser URL, so they are not exposed in history, logs, or the Referer header. It is the recommended flow for web apps and, with PKCE, for native and single-page apps too.

Q: Why is the authorization code flow preferred over the implicit flow for browser apps today?
A: The implicit flow returned tokens directly in the redirect URL fragment, exposing them to the browser, extensions, and logs, and it had no refresh mechanism. The authorization code flow with PKCE keeps tokens off the front channel by exchanging a code on the back channel, and it supports refresh tokens safely. As a result the implicit flow is now deprecated by OAuth security best-current-practice guidance and you should use auth code + PKCE instead.

Q: What is PKCE and which OIDC clients should use it?
A: PKCE (Proof Key for Code Exchange, RFC 7636) is an extension where the client generates a secret `code_verifier` per request and sends its hash up front, binding the authorization code to that one client. It should be used by all public clients — native mobile apps and single-page apps that cannot keep a client secret — and current guidance recommends it for confidential clients too. It defends against authorization-code interception by an attacker who lacks the verifier.

Q: In PKCE, what are `code_verifier` and `code_challenge`?
A: The `code_verifier` is a high-entropy random string of 43 to 128 unreserved characters created by the client for each authorization request and kept secret. The `code_challenge` is a value derived from it — sent in the initial authorization request — that the IdP stores to verify later. At the token exchange the client presents the raw `code_verifier`, and the IdP recomputes the challenge to confirm they match before issuing tokens.

Q: What are the two `code_challenge_method` values and which must you use?
A: The two methods are `plain`, where the challenge equals the verifier, and `S256`, where the challenge is `BASE64URL-ENCODE(SHA256(ASCII(code_verifier)))`. You must use `S256` whenever the client can compute SHA-256, which is essentially always; `plain` exists only as a fallback for technically constrained clients. Using `plain` leaks the verifier in the authorization request, defeating much of PKCE's protection.

Q: What specific attack does PKCE protect against?
A: PKCE protects against the authorization-code interception attack, where a malicious app or network observer captures the authorization code as it is returned to the client. Without PKCE, an attacker who steals the code can redeem it for tokens; with PKCE, redemption also requires the original `code_verifier`, which the attacker never saw. This was the classic risk for mobile apps using custom URI scheme redirects that other apps could register.

Q: What is the client credentials flow and when do you use it?
A: The client credentials flow has a service authenticate as itself — using a client ID and secret (or certificate) — to obtain an access token, with no human user involved. You use it for machine-to-machine calls such as a pipeline job calling an API or one service calling another. There is no `id_token` and no refresh token, because the identity is the service's own client identity, which can simply re-authenticate when the access token expires.

Q: Why is PKCE not used in the client credentials flow?
A: PKCE solves the problem of a public client losing control of an authorization code in a browser redirect, but the client credentials flow has no browser, no user, and no authorization code — the service authenticates directly at the token endpoint. The service is a confidential client holding a real secret, so it proves its identity with that secret rather than a per-request verifier. Adding PKCE there would solve a problem that does not exist.

Q: What is SAML and how does it differ from OIDC?
A: SAML (Security Assertion Markup Language) is an older XML-based SSO protocol where the IdP sends a signed XML assertion, usually via a browser POST, to a service provider. It differs from OIDC in encoding (XML vs JSON/JWT), transport, and ergonomics: OIDC is simpler, mobile- and SPA-friendly, and REST/JSON-native. Many enterprises still run SAML for legacy SaaS, so an architect must read both, but new integrations generally favour OIDC.

Q: A fund-admin SaaS only supports SAML, but your IdP speaks OIDC natively — what is the usual fix?
A: Most IdPs, including Keycloak and Entra ID, can act as a SAML identity provider as well as an OIDC one, so you configure a SAML client/application for that SaaS while keeping OIDC for everything else. The same user directory and groups back both protocols, so SSO and group changes still propagate uniformly. The architect's job is to map the SaaS's expected SAML attributes (e.g. `NameID`, role attributes) to the right claims.

Q: What is an ID token and what is its purpose?
A: An ID token is a JWT issued by the IdP that represents the outcome of an authentication event, carrying the user's identifier (`sub`) and authentication details such as when and how they logged in. Its purpose is to tell the client who the user is — it is consumed by the client itself, not sent to APIs. Treat it as proof of authentication, not as a credential for calling backend services.

Q: What is an access token and how does its purpose differ from the ID token?
A: An access token is a credential the client presents to a resource server (API) to prove it is authorized to call that API on the user's behalf. Unlike the ID token, which is for the client to learn the user's identity, the access token is for the API and is validated by the API against its expected audience and scopes. Sending an ID token to an API, or an access token back to the client as identity, are both common mistakes.

Q: What is a refresh token and why must it be protected carefully?
A: A refresh token is a long-lived credential that lets a client obtain new access tokens (and sometimes ID tokens) without prompting the user to log in again. It must be protected carefully because it effectively represents a standing session: stealing it lets an attacker mint fresh access tokens until it is revoked or expires. Best practice is to store it only in confidential clients, bind it where possible, and support rotation so a leaked token is detectable.

Q: What are claims in an OIDC token?
A: Claims are name/value statements about the subject or the authentication event carried inside a token or returned from the userinfo endpoint, such as `sub`, `email`, `name`, or a `groups` array. They are how the IdP communicates identity attributes to the client and APIs in a structured, signed form. An architect decides which claims to include because over-stuffing tokens leaks data and bloats every request that carries them.

Q: What is the `aud` (audience) claim and why does it matter for security?
A: The `aud` claim identifies the intended recipient(s) of a token, so a resource server can reject a token that was not minted for it. It matters because without audience validation, a token issued for a low-value service could be replayed against a high-value one — a token-confusion attack. Each API must verify that its own identifier appears in `aud` before trusting the token.

Q: What does the `iss` claim assert and what must a client do with it?
A: The `iss` (issuer) claim names the IdP that issued the token, e.g. the Keycloak realm URL. The client and any resource server must verify that `iss` exactly matches the trusted issuer they configured, because accepting tokens from an unexpected issuer would let an attacker's IdP impersonate yours. Issuer validation is paired with verifying the token signature against that issuer's published keys.

Q: What is the `sub` claim and why prefer it over email as a primary user key?
A: The `sub` (subject) claim is the IdP's stable, unique identifier for the user within that issuer. You prefer it over email as the primary key because emails change (marriage, rebrand, domain migration) and can be reassigned, whereas `sub` is meant to stay constant for the same person. Keying internal records on `sub` avoids silently merging or splitting identities when contact details change.

Q: What does the `nonce` parameter protect against in OIDC?
A: The client generates a `nonce`, sends it in the authentication request, and the IdP echoes it into the ID token, letting the client confirm the token belongs to this specific request. It protects against token replay and injection — an attacker cannot substitute a previously captured ID token because its `nonce` will not match the one the client just generated. The client must store the expected `nonce` and reject any ID token whose value differs.

Q: What are scopes in OIDC/OAuth and how do they relate to claims?
A: Scopes are coarse permission labels the client requests, such as `openid`, `profile`, `email`, or a custom API scope, and they shape what the IdP authorizes and which claims it releases. For example requesting the `email` scope is what makes the IdP include the `email` claim. Scopes describe intent and consent; claims are the resulting data — requesting more scopes than needed is a minimisation problem in a fund context handling personal data.

Q: Why must the `openid` scope be present in an OIDC request?
A: The literal `openid` scope is the signal that turns an ordinary OAuth 2.0 authorization request into an OIDC request, instructing the provider to return an `id_token`. Without it you are doing plain OAuth and get only an access token, so the client cannot reliably learn the authenticated user's identity. Forgetting `openid` is a frequent cause of "I get a token but no user info" bugs.

Q: What is the userinfo endpoint and when would you call it?
A: The userinfo endpoint is an OIDC endpoint that returns claims about the authenticated user when the client presents a valid access token. You call it when you want fresh or additional identity attributes that you chose not to pack into the ID token, keeping tokens small. It is the back-channel way to ask "tell me more about this user" without re-authenticating them.

Q: How does a resource server typically validate a JWT access token?
A: It verifies the signature using the issuer's public keys (fetched from the JWKS endpoint advertised in discovery), checks `iss`, checks that its own identifier is in `aud`, checks `exp`/`nbf` for time validity, and confirms required scopes. This local validation avoids a network round-trip per request. The keys are cached and rotated via the JWKS `kid`, so the server can handle key rollover without redeployment.

Q: What is the OIDC discovery document and why is it useful?
A: The discovery document, served at `/.well-known/openid-configuration`, is a JSON file advertising the IdP's endpoints (authorization, token, userinfo, JWKS), supported scopes, claims, and signing algorithms. It is useful because clients can auto-configure from one URL instead of hard-coding endpoints, and it makes issuer changes (like moving a Keycloak realm) less error-prone. Most OIDC libraries bootstrap entirely from this URL.

Q: What is the difference between authentication and authorization, and which protocol layer handles each here?
A: Authentication establishes who the user is; authorization decides what they may do. In this stack OIDC/SAML handle authentication and identity assertion, while authorization is driven by claims (especially group/role claims) that the application or a policy engine evaluates. Conflating them — for example trusting that authentication implies access — is how leavers and over-privileged accounts slip through.

Q: What is a confidential client versus a public client?
A: A confidential client can securely store a secret (a server-side web app or a backend service), so it can authenticate to the token endpoint with a client secret or certificate. A public client cannot keep a secret — native and single-page apps whose code is on the user's device — so it relies on PKCE and registered redirect URIs instead. Misclassifying a public client as confidential leads to a secret that is effectively published.

Q: Why are exact redirect URIs registered and matched for OIDC clients?
A: The IdP only returns the authorization code to a redirect URI that the client registered in advance, and it matches it exactly (no open wildcards). This stops an attacker from causing the IdP to deliver a code to a URL they control. A common misconfiguration is allowing a wildcard or trailing-path redirect, which can be abused to exfiltrate authorization codes.

Q: What is a Keycloak realm and what does it bound?
A: A realm is Keycloak's top-level isolation boundary that owns its own set of users, credentials, roles, groups, clients, and signing keys. Everything inside a realm shares an identity space and trust domain; objects in one realm are invisible to another by default. You typically use one realm per security domain — for example a single realm for your internal data platform tools rather than one realm per tool.

Q: What is a Keycloak client object?
A: A Keycloak client represents an application or service that wants to authenticate users or call APIs through the realm, such as a client each for OpenMetadata, Grafana, and Trino. Each client has its own ID, protocol (OIDC or SAML), allowed redirect URIs, access type (public/confidential), and scope/mapper configuration. Registering a client is how you let a specific tool delegate authentication to Keycloak.

Q: In the lab you stand up Keycloak with clients for OpenMetadata, Grafana, and Trino — why one client per tool rather than one shared client?
A: Each tool has different redirect URIs, secrets, allowed scopes, and possibly protocol needs, so a separate client lets you configure and rotate each independently and revoke one without affecting the others. It also gives each tool a distinct audience, so a token minted for Grafana is not silently accepted by Trino. Sharing one client would blur audiences and make least-privilege and incident isolation much harder.

Q: What is a Keycloak client scope and how does it differ from a realm role?
A: A client scope is a reusable bundle of protocol mappers and scope definitions that controls which claims and permissions go into tokens for clients that use it; a realm role is an authorization label assigned to users or groups. Client scopes shape token contents (e.g. adding a `groups` claim), while roles express what a subject is entitled to do. You often pair them: a role for the entitlement and a client scope/mapper to surface it in the token.

Q: What is a Keycloak protocol mapper?
A: A protocol mapper is a per-client (or per-client-scope) rule that injects or transforms a claim in the issued token or userinfo response, such as mapping group membership into a `groups` claim or copying a user attribute into the token. It is the mechanism that turns Keycloak's internal data model into the exact claim shape a given tool expects. Without the right mapper, the tool authenticates the user but cannot see their groups or roles.

Q: What is groups-to-roles mapping and why is it the core of the lab?
A: Groups-to-roles mapping is the practice of taking an IdP group claim (e.g. membership in `data-steward`) and translating it into the in-tool roles each application understands. It is the core of the lab because it is what lets one group change move a person's permissions across all three tools at once. Get the mapping right and you administer access centrally; get it wrong and each tool drifts into its own ad-hoc permission set.

Q: How do you configure Keycloak so a tool receives group membership in its token?
A: You add a Group Membership protocol mapper (commonly via a `groups` client scope) so that the user's group paths are emitted as a claim, then assign that scope to the relevant clients. The downstream tool is configured to read that claim and map specific group values onto its own roles. The mapper's options — full path vs short name, claim name — must match exactly what each tool expects.

Q: A user is in the `data-steward` group but Trino shows no elevated access — what do you check first?
A: First confirm the token actually carries the group: inspect a decoded token or Keycloak's client scope evaluation to see whether the `groups` claim is present and contains `data-steward`. If the claim is missing, the Group Membership mapper or the client scope assignment is the culprit; if present, the problem is on Trino's side mapping that claim value to a role. Checking the token first cleanly splits "IdP didn't send it" from "the app didn't act on it".

Q: Why map groups to roles instead of assigning individual users to tool permissions directly?
A: Group-based mapping means access is granted by membership, so adding or removing one person from a group updates all tools at once and leaves a clear, auditable basis for their access. Direct per-user assignment in each tool fragments administration, makes leavers hard to fully offboard, and gives no single answer to "who could see this in March?". Groups also align cleanly with role-based access control and joiner/mover/leaver processes.

Q: What is SCIM and what problem does it solve?
A: SCIM (System for Cross-domain Identity Management) is a standard REST protocol for automatically provisioning and deprovisioning users and groups into downstream applications. It solves the problem of manual, drift-prone account management: instead of an admin creating and later remembering to delete accounts in each tool, the IdP pushes lifecycle changes automatically. This closes the orphaned-account gap that leaves leavers with lingering access.

Q: How does SCIM relate to and differ from OIDC/SAML?
A: OIDC and SAML handle runtime authentication — proving who the user is at login — while SCIM handles the lifecycle of the account itself: creating it, updating attributes and group membership, and disabling it. They are complementary: SCIM ensures the account exists with the right groups before the user ever authenticates via OIDC. Relying on just-in-time provisioning at login can miss deprovisioning, which is precisely what SCIM addresses.

Q: Which RFCs define SCIM and what does each cover?
A: SCIM 2.0 is defined by RFC 7643, which specifies the core schema and resource model (User and Group resources and their attributes), and RFC 7644, which specifies the protocol — the REST endpoints, filtering, and bulk/`PATCH` semantics. Knowing which RFC governs which concern helps when an integration complains about a malformed `PATCH` (a protocol/RFC 7644 issue) versus an unknown attribute (a schema/RFC 7643 issue).

Q: What are the two core SCIM endpoints and the typical operations on them?
A: The two core endpoints are `/Users` and `/Groups`. Typical operations include `POST /Users` to create, `PATCH /Users/{id}` to update specific attributes, `PUT` to replace, `GET` with filters to query, and `DELETE` to remove. Group membership is similarly managed via the `/Groups` endpoint, which is how an IdP keeps downstream group lists in sync with its own.

Q: In SCIM, how is a leaver typically deprovisioned, and why not always hard-delete?
A: A leaver is usually deprovisioned by a `PATCH` that sets the user's `active` attribute to `false`, disabling the account rather than removing the record outright. Soft-disabling is preferred because many systems need to retain the account for audit, ownership, and historical attribution, and a hard `DELETE` can orphan related data. In a regulated fund context, you must still be able to answer "who had access in March?" after the person has left.

Q: A SCIM integration creates users fine but deprovisioning never happens — what is the likely cause?
A: The most common cause is that the IdP is configured to provision and update but the deprovisioning action (disable on unassignment or on `active:false`) is not enabled or the target app ignores it. Check the provisioning rules for what triggers removal and confirm the app actually honours `active:false` rather than expecting a `DELETE`. This gap is dangerous because it is exactly how leavers keep access despite "automated" provisioning.

Q: Why is SCIM described as delivering security, efficiency, and compliance simultaneously?
A: Security comes from automatic deprovisioning closing the orphaned-account gap; efficiency comes from eliminating manual account churn for IT; and compliance comes from producing a consistent, reviewable record of who has access to what. These align directly with regulatory expectations around access governance. For a fund administrator, that reviewable record is part of demonstrating control over who could touch investor and NAV data.

Q: Does Keycloak provide SCIM, and how mature is it?
A: Keycloak introduced a built-in SCIM realm API to manage users and groups via the SCIM protocol, and as of 2026 it is shipped as an experimental feature with explicit attention to Microsoft Entra ID compatibility. Historically teams used third-party extensions (such as the scim-for-keycloak module) to add SCIM. So the architect should check the running Keycloak version to know whether SCIM is native-experimental or extension-based.

Q: What are managed identities and what problem do they solve?
A: Managed identities are platform-provided service identities (notably in Azure) whose credentials are issued and rotated by the platform, so a workload can authenticate to other services without any secret in code or config. They solve the "secret zero" problem — the embedded password or key that gets committed, leaked, or never rotated. The workload simply requests a token from the platform, which vouches for its identity.

Q: How do managed identities relate to the client credentials flow?
A: Conceptually a managed identity is service-to-service authentication, the same problem the client credentials flow solves, but with the platform supplying and rotating the credential instead of the developer managing a client secret. So you still end up with a service obtaining an access token for a downstream API, just without a stored secret. The architect's takeaway is to prefer managed/workload identities over long-lived client secrets wherever the platform supports them.

Q: Why are managed identities preferred over storing a client secret in a pipeline?
A: A stored client secret is a static credential that can be leaked, copied, and is often rotated rarely or never; a managed identity removes that artifact entirely because the platform mints short-lived tokens on demand. This shrinks the attack surface and removes a class of "secret in the repo" incidents. In a regulated environment it also strengthens the audit story, since there is no shared secret to track or attest to.

Q: What is the "secret zero" problem and how does federated/workload identity address it?
A: "Secret zero" is the bootstrapping problem that any secret-based scheme needs at least one initial secret to obtain the others, and that root secret is the hardest to protect and rotate. Workload/federated identity addresses it by having the runtime platform attest to the workload (e.g. via a signed token or instance metadata) so no static root secret is stored at all. Trust is anchored in the platform rather than in a file you must guard.

Q: What does it mean to "move one identity's permissions across three tools with one group change"?
A: It means the user's effective access in OpenMetadata, Grafana, and Trino all derive from their group membership claim, so adding them to (or removing them from) a single Keycloak group changes all three at once. This is the architect's proof that identity is truly centralised rather than reconfigured tool-by-tool. The done-when criterion is satisfied only when no per-tool edit is needed beyond the one group change.

Q: Narrate the auth-code + PKCE flow end to end, as the lesson's done-when requires.
A: The client generates a `code_verifier` and its `S256` `code_challenge`, then redirects the user to the IdP's authorization endpoint with the challenge; the user authenticates and the IdP redirects back with a single-use `code`. The client posts that `code` plus the raw `code_verifier` to the token endpoint, the IdP verifies the challenge and returns an ID token, access token, and optionally a refresh token. The client validates the ID token (`iss`, `aud`, `nonce`, signature, `exp`) and then uses the access token to call APIs.

Q: After login a user's `groups` claim is empty even though they are in groups — what are the two prime suspects?
A: The two prime suspects are a missing or misassigned Group Membership protocol mapper / `groups` client scope on that client, or the user's groups not actually being members of the path the mapper emits. Use Keycloak's "evaluate" feature for that client and user to see the exact token that would be issued. If the claim is empty there, it is an IdP-side mapping problem rather than an application bug.

Q: Why can a `groups` claim become very large, and why is that a concern?
A: A user in many nested groups can produce a long `groups` array, and emitting full group paths multiplies the size. This is a concern because the claim travels in every token and, for cookie- or header-carried tokens, can exceed size limits and break requests with header-too-large errors. The fix is to scope which groups are emitted, use role aggregation, or move authorization decisions to a policy service rather than stuffing everything into the token.

Q: What is the difference between front-channel and back-channel in OIDC, and why does it matter?
A: The front channel is the browser redirect path (visible in the URL, history, and to extensions); the back channel is the direct server-to-server call the client makes to the token endpoint. It matters because secrets and tokens must travel on the back channel — the auth code flow deliberately sends only a short-lived code on the front channel and exchanges it for tokens on the back channel. Putting tokens on the front channel was the implicit flow's core weakness.

Q: Why must all of these flows run over TLS, and what specifically breaks without it?
A: TLS protects the confidentiality and integrity of redirects, codes, secrets, and tokens in transit; without it, an attacker on the network can read authorization codes, access tokens, and secrets directly. PKCE explicitly assumes the redirect path may not be TLS-protected and adds defence, but the token exchange and all token use still require TLS. Running OIDC over plain HTTP outside localhost is effectively handing credentials to anyone on the path.

Q: What is token expiry (`exp`) and why are access tokens short-lived?
A: `exp` is the timestamp after which a token must be rejected. Access tokens are kept short-lived (often minutes) so that a stolen token has a narrow window of usefulness and revocation is effectively automatic when it expires. The refresh token, held only by a trusted client, is what obtains a replacement, keeping the user logged in without making the high-exposure access token long-lived.

Q: What is refresh token rotation and why does it improve security?
A: Refresh token rotation issues a new refresh token each time one is used and invalidates the old one, so a refresh token is single-use. It improves security because if both the legitimate client and an attacker try to use the same rotated token, the IdP detects the reuse and can revoke the whole token family. This turns a silent theft into a detectable event, which is valuable when refresh tokens are necessarily longer-lived.

Q: What does token revocation accomplish that expiry alone does not?
A: Revocation invalidates a token (or session) immediately, before its natural `exp`, which is essential when a user is offboarded, a device is lost, or a token is suspected stolen. Expiry only bounds the worst case; revocation lets you cut access now. Note that locally-validated JWTs are not truly revoked until they expire unless you add introspection or short lifetimes, which is a key design tradeoff.

Q: What is token introspection and when would you use it over local JWT validation?
A: Token introspection is an endpoint where a resource server asks the IdP whether a token is currently active and what it represents. You use it over local JWT validation when you need real-time revocation checks or when tokens are opaque rather than self-contained JWTs. The tradeoff is a network call per check (mitigated by caching), versus local validation's speed but inability to know about a just-revoked token.

Q: Distinguish a role from a group in an IdP context.
A: A role is an authorization entitlement — a named permission or capability such as `data-steward` — while a group is a collection of users, often reflecting an org unit or function. Groups are about membership; roles are about what membership grants. The standard pattern is to assign roles to groups and put people in groups, so individuals inherit roles through membership and you administer access at the group level.

Q: How does role-based access control (RBAC) connect to the group claim?
A: RBAC means access is granted to roles and users acquire roles, typically through group membership; the group claim in the token is the bridge that carries that membership to each tool so it can resolve the user's roles. This keeps the policy ("data-steward can edit lineage") separate from the assignment ("Alice is a data-steward"). The IdP owns the assignment; each tool owns the policy mapping from group/role to its own permissions.

Q: A new joiner has an account but cannot see anything in any tool — first thing to check?
A: First check group membership: a freshly provisioned account with no group assignments will authenticate successfully but carry no authorizing claims, so every tool grants only default access. Confirm the joiner is in the right Keycloak groups and that those groups map to roles in each tool. This cleanly distinguishes an authentication success with an authorization gap from a broken login.

Q: Why does centralised identity make the audit question "who could see this in March?" answerable?
A: When access derives from IdP group membership and changes flow through one system, the IdP and SCIM logs hold a time-ordered record of who was in which group when, and therefore what each person could access. Without centralisation you would have to reconstruct state from disparate per-tool configs, many of which keep no history. For a fund administrator under audit, that single authoritative timeline is the defensible answer.

Q: Why is just-in-time (JIT) provisioning on first login not a substitute for SCIM?
A: JIT creates an account the first time a user logs in via OIDC/SAML, which handles onboarding but has no event to trigger removal when the person leaves — they may simply stop logging in while the account lingers. SCIM covers the full lifecycle, including the deprovisioning signal that JIT lacks. Relying on JIT alone is a classic reason leavers retain dormant but live access.

Q: What is the relationship between an OAuth resource server and an OIDC client?
A: The OIDC client authenticates the user and obtains an access token; the resource server (API) is the party that consumes that access token to authorize requests. One application can play both roles, but they are distinct: the client cares about identity (ID token), the resource server cares about authorization (access token, scopes, audience). Keeping the roles straight clarifies which token each component should validate.

Q: Why should an application validate the ID token's signature rather than trusting its contents?
A: A JWT is just base64url-encoded JSON until its signature is checked, so an attacker could forge or tamper with claims unless the application verifies the signature against the issuer's published keys. Trusting unverified contents — for example reading `groups` from an unverified token — is a privilege-escalation hole. Validation must include the signature plus `iss`, `aud`, and time claims before any claim is acted upon.

Q: What is the JWKS endpoint and how does it support key rotation?
A: The JWKS (JSON Web Key Set) endpoint publishes the IdP's current public signing keys, each tagged with a `kid`. Resource servers fetch and cache these keys to validate token signatures, and when the IdP rotates keys it publishes the new key alongside the old so tokens signed with either remain verifiable during the overlap. The `kid` in a token header tells the verifier which published key to use, enabling rollover without downtime.

Q: In Keycloak, what is the practical effect of disabling a user versus removing them from a group?
A: Disabling a user blocks all authentication for that account everywhere immediately, useful for an offboarding or compromise. Removing them from a group leaves the account active but strips the access that membership granted, useful for a role change or mover. The architect picks based on intent: full offboarding disables/deprovisions the account, while a transfer between teams is a group change.

Q: Why might a group change appear in Keycloak but not take effect in a tool until the user re-logs in?
A: Many tools read group/role claims from the token issued at login, so an existing session still carries the old claims until the token (and any cached session) refreshes. The change takes effect on the next token issuance — a re-login or a refresh that re-reads the claim. This is why "I changed the group but nothing happened" is often a stale-session issue, not a mapping failure.

Q: What is consent in OIDC and when is it typically skipped?
A: Consent is the step where the user approves the scopes/claims a client is requesting before the IdP releases them. It is typically skipped for first-party internal clients that you mark as not requiring consent, since the organisation already trusts its own tools, but it is meaningful for third-party clients. In an internal data platform you generally disable interactive consent for your own tools to streamline SSO.

Q: What is the difference between `PUT` and `PATCH` in SCIM and why does it matter?
A: `PUT` replaces the entire resource representation, so any attribute omitted may be cleared, whereas `PATCH` applies targeted operations (add/replace/remove) to specific attributes. It matters because using `PUT` when you intend a small change can wipe attributes the IdP did not send, and malformed `PATCH` bodies are a common source of SCIM interop bugs (notably between Entra and various targets). Knowing which the integration uses helps diagnose unexpected attribute loss.

Q: Why does the lesson tie Keycloak's group claim to roles in three different tools rather than using each tool's own login?
A: Using each tool's own login fragments identity: three password stores, three offboarding steps, and no single source of truth for access. Centralising on Keycloak with a group claim means one credential, one place to change membership, and one audit trail across OpenMetadata, Grafana, and Trino. This is the architectural point of the lesson — identity is a platform capability, not a per-tool afterthought.

Q: For an asset manager, why is fast, provable deprovisioning of a leaver a regulatory concern, not just hygiene?
A: Regulations and frameworks around fund administration (and operational resilience expectations such as DORA) require demonstrable control over access to investor and NAV data, including timely removal of access for leavers. A lingering account that could still read transfer-agency or NAV systems is both a breach risk and an audit finding. Centralised IdP plus SCIM gives the evidence that access was removed on a specific date.

Q: How would you design SSO so a Luxembourg-based fund-admin analyst gets the same access across all data tools?
A: Put all tools behind one Keycloak realm as OIDC clients, define functional groups (e.g. `ta-analyst`, `nav-reviewer`), map each group's claim to each tool's roles, and provision/deprovision membership via SCIM. The analyst then logs in once and their group membership determines identical, role-appropriate access everywhere. A single group change moves them between functions, and the IdP logs provide the audit trail required in a regulated jurisdiction.

Q: Why should access tokens carrying personal data (e.g. investor identifiers) be minimised under data-protection rules?
A: Tokens are transmitted on many requests and may be logged at proxies and gateways, so packing personal data such as investor identifiers into claims spreads that data widely and conflicts with data-minimisation principles. You should include only the claims a tool genuinely needs and fetch sensitive attributes from a controlled source instead. This keeps the token a thin authorization artifact rather than a roaming copy of PII.

Q: What is audience restriction's role when one user accesses both a NAV tool and an unrelated tool?
A: Audience restriction (`aud`) ensures the access token minted for the NAV tool is rejected if presented to the unrelated tool, so a token's reach is bounded to its intended service. Without it, a token leaked from a low-sensitivity tool could be replayed against the high-sensitivity NAV system. Each client/resource server validating `aud` is what keeps cross-tool token misuse from undermining segregation of duties.

Q: What does "claims" versus "scopes" mean when configuring what a fund-data tool receives?
A: Scopes are the request-time permission labels the tool asks for and the user/admin consents to (e.g. `openid groups`), while claims are the actual attribute values delivered (e.g. the `groups` array). When configuring a fund-data tool you grant the scopes it needs and rely on mappers to populate the corresponding claims. Over-broad scopes pull in more claims than necessary, which is both a security and a minimisation issue.

Q: An OIDC login redirects back with `error=invalid_redirect_uri` — what is wrong and how do you fix it?
A: The redirect URI the client sent does not exactly match one registered for that client in Keycloak, so the IdP refuses to return the code. Fix it by registering the precise callback URL (scheme, host, port, and path) for that client, avoiding wildcards. This is a deliberate safety check — a permissive redirect list is a known way to steal authorization codes, so the strictness is by design.

Q: An OIDC login fails with `invalid_client` at the token endpoint — what does that indicate?
A: `invalid_client` means the client failed to authenticate at the token endpoint, typically a wrong or missing client secret, a confidential client misconfigured as public (or vice versa), or the wrong authentication method. Check that the client's access type and credentials in Keycloak match what the application is sending. It is an authentication-of-the-client failure, distinct from problems with the user or the authorization code.

Q: An API rejects a token with an audience error even though the user logged in fine — what is happening?
A: The user authenticated correctly, but the access token's `aud` does not include the API's expected identifier, so the resource server refuses it. Usually the client requested a token for the wrong audience/scope, or a Keycloak audience mapper is missing for that client. Fix it by configuring the token's intended audience (often via an audience protocol mapper or the correct scope) so the API's identifier appears in `aud`.

Q: Why is it dangerous to use the access token to identify the user in your application?
A: An access token is meant for resource servers and its format and claims are not guaranteed to be a stable identity statement for the client; it may be opaque or contain only authorization data. Using it to identify the user can break when the IdP changes token contents and can leak authorization-scoped data into identity logic. The ID token (or a userinfo call) is the correct, contract-defined source of user identity.

Q: What is the practical difference between platform-managed secrets and a secretless managed identity for a Trino-to-storage connection?
A: With a platform-managed secret you still store a credential (rotated by you or a vault) that Trino presents to storage, whereas a managed identity has the platform issue short-lived tokens so no secret is stored in Trino's config at all. The secretless approach removes a leak vector and a rotation burden. For sensitive fund data stores, fewer standing secrets means a smaller attack surface and a cleaner audit.

Q: Why does an architect need to "read OIDC/SAML flows fluently" rather than just enabling SSO?
A: Because when SSO breaks, the failure is almost always in a specific step — a missing scope, a bad `aud`, a stale key, a mismatched redirect, a misconfigured mapper — and you can only diagnose it if you can trace the flow and inspect the token at each stage. Fluency also lets the architect make sound design choices about which flow, which claims, and where to enforce authorization. Treating SSO as a black box leaves you unable to defend or debug access.

Q: How do you inspect what claims a token actually contains during troubleshooting?
A: Decode the JWT (for example by base64url-decoding the payload segment, or using Keycloak's account/admin token tools and the client "evaluate" feature) to see `iss`, `aud`, `exp`, scopes, and `groups`. This shows whether the IdP actually emitted the claims the tool expects, separating an IdP mapping problem from an application consumption problem. Never paste real production tokens into untrusted online decoders, since they are live credentials.

Q: Why is putting authorization decisions entirely inside token claims sometimes a poor design?
A: Claims are fixed at token issuance, so fine-grained or rapidly-changing permissions baked into the token go stale until the next login/refresh, and large permission sets bloat every request. For complex or dynamic authorization, a dedicated policy engine queried at request time scales better and revokes instantly. The token then carries identity and coarse roles, while detailed entitlement decisions live in the policy layer.

Q: What is the role of the `state` parameter in the authorization request?
A: The client generates a `state` value, sends it in the authorization request, and verifies it on the redirect back, ensuring the response corresponds to a request the client initiated. It is the primary CSRF defence for the OAuth/OIDC redirect, preventing an attacker from injecting their own authorization response. A client that ignores `state` is vulnerable to login-CSRF, so validating it is mandatory.

Q: How do `nonce` and `state` differ in purpose?
A: `state` protects the front-channel request/response correlation and defends against CSRF on the redirect, while `nonce` binds the resulting ID token to the specific authentication request to prevent token replay/injection. `state` is checked when the redirect returns; `nonce` is checked inside the ID token. Both are required for a hardened authorization code flow, and they guard different stages.

Q: Why is "log out of the IdP" not always enough to revoke active access?
A: Each application typically maintains its own session and may hold unexpired access tokens, so ending only the IdP session can leave those local sessions and tokens valid until they expire. Full revocation needs single-logout/back-channel logout plus short token lifetimes or token revocation. This gap is why offboarding should disable/deprovision the account, not merely log the user out.

Q: When integrating an external IdP into Keycloak (identity brokering), what is the mapping concern?
A: When Keycloak brokers to an upstream IdP (e.g. Entra ID), you must map the upstream's claims — especially group/role attributes — onto Keycloak's groups/roles so entitlements carry through. A one-to-one identity-provider mapper translates an upstream group into a local role or group. If that mapping is missing, users authenticate via the upstream IdP but arrive with no usable authorization in your realm.

Q: Why might you choose Keycloak (FOSS) for the lab but evaluate Entra ID for the corporate context?
A: Keycloak is open-source and self-hostable, giving full control of realms, clients, and mappers for hands-on learning without licensing, while Entra ID is the managed Azure IdP many enterprises already run as their primary identity store. The protocols (OIDC/SAML/SCIM) are the same, so concepts transfer; the choice is about operations, integration with existing tenancy, and support. An architect should be able to design against either since the standards are shared.

Q: What is the risk of a single shared service account versus per-service identities?
A: A shared service account spreads one set of credentials across many workloads, so a leak compromises everything and the audit trail cannot tell which workload did what. Per-service identities (ideally managed identities) scope blast radius and give attributable logs. The lesson's managed-identity recap is precisely about replacing shared secrets with distinct, secretless service identities.

Q: How does the `prompt` parameter (e.g. `prompt=none` or `prompt=login`) influence the OIDC flow?
A: `prompt=none` asks the IdP to authenticate silently using an existing session and to error rather than show UI, useful for silent token renewal; `prompt=login` forces re-authentication even if a session exists. It lets the client control whether the user sees an interactive step. Mishandling `prompt=none` errors (treating them as failures rather than "needs interactive login") is a common SPA bug.

Q: Why must redirect URIs and post-logout redirect URIs both be registered?
A: The IdP only sends authorization codes to registered redirect URIs and only redirects after logout to registered post-logout URIs, in both cases to prevent open-redirect abuse where an attacker steers the user (and possibly a code) to a hostile site. Forgetting to register the post-logout URI commonly produces a logout that errors or strands the user. Both lists are part of locking down where the IdP will send the browser.

Q: What does it mean that OIDC tokens are "self-contained" and what tradeoff does that create?
A: A self-contained JWT carries its claims and signature so a resource server can validate it locally without calling the IdP, which is fast and scalable. The tradeoff is revocation: because validation is local, a revoked-but-unexpired token still validates until `exp`, so you compensate with short lifetimes, introspection, or a revocation list. Choosing token lifetime is therefore a balance between performance and how quickly you can cut access.

Q: How would you verify the lesson's done-when "one group change across three tools" actually worked end to end?
A: Add a test user to the `data-steward` group in Keycloak, then log in to OpenMetadata, Grafana, and Trino and confirm each now shows the steward-level access — checking each tool reads the group claim and maps it to the right role. Then remove the user from the group, re-authenticate, and confirm access drops in all three. If any tool requires a separate edit, the centralisation is incomplete and the mapper or scope for that client needs fixing.

Q: Why are nested groups in Keycloak both convenient and a potential source of confusion?
A: Nested groups let a child inherit roles and attributes from parents, so you can model hierarchy and reduce repetition, but the inheritance can make it non-obvious why a user has a given role. When the Group Membership mapper emits full paths, the claim values also encode the hierarchy, which downstream tools must match exactly. Documenting the group tree and which level grants which role prevents surprising effective permissions.

Q: What is the difference between an OIDC `access_token` audience and an OIDC `id_token` audience?
A: The `id_token`'s `aud` is the client (the application that requested authentication), since the ID token is for the client to consume, whereas the `access_token`'s `aud` is the resource server(s) the token may be presented to. Confusing the two leads to a client trying to call an API with an ID token, or an API accepting a token whose audience is actually the client. Each component should validate the audience appropriate to the token it is consuming.

Q: For a transfer-agency platform, why is mapping a single `ta-supervisor` group to roles in every tool better than per-tool admin grants?
A: Mapping one `ta-supervisor` group to the appropriate role in each tool means supervisory access is granted and revoked in one place, with one auditable membership record covering NAV review, lineage, and dashboards. Per-tool admin grants scatter the entitlement, make leaver removal error-prone, and leave no unified answer for who held supervisory access on a given date. Centralised group-to-role mapping is what makes segregation-of-duties claims defensible to an auditor.

Q: Summarise the architect's mental checklist when wiring a new tool into the IdP for SSO.
A: Register a client with the right protocol and access type, configure exact redirect and post-logout URIs, assign the scopes/mappers needed so the tool receives identity and `groups` claims, and map those group claims to the tool's roles. Then verify the token contents, confirm a single group change moves access, and ensure deprovisioning (via disable/SCIM) removes access promptly. Finally, validate `iss`/`aud`/signature on the consuming side and run everything over TLS so the integration is both functional and defensible.
