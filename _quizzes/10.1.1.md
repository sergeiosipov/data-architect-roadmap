# Phase 6 · 10.1.1 Metadata management & active catalogs — 150 self-test questions

Q: What is metadata, in one sentence, and why is it the foundation of a data catalog?
A: Metadata is data about data — descriptions of structure, meaning, ownership, freshness, and behaviour of data assets rather than the row values themselves. A catalog is fundamentally a system that collects, organises, and serves metadata so people and machines can find, trust, and govern assets. In a fund platform this is the layer that answers "what is this `nav` table, who owns it, and what feeds it" without opening the data.

Q: What are the three broad kinds of metadata, and give an example of each for a NAV pipeline?
A: Technical metadata describes physical structure (the `nav` table's columns, types, partition keys); business metadata captures meaning and policy (the glossary term "Net Asset Value", the owning fund-accounting domain); operational metadata records runtime behaviour (last ingestion time, row counts, the Dagster run that last refreshed it). All three together let an architect reason about an asset end to end, where any one alone is incomplete.

Q: Distinguish technical metadata from business metadata with a concrete fund example.
A: Technical metadata is the machine-level truth — column `isin` is `VARCHAR(12)`, table is Iceberg, partitioned by `valuation_date`. Business metadata is the human/governance layer — `isin` maps to the glossary term "International Securities Identification Number", the asset is owned by the Security Master domain and tiered Tier-1. The same column carries both: the catalog's job is to bind them so a steward sees meaning while an engineer sees structure.

Q: What is operational metadata and why do regulators care about it?
A: Operational metadata is runtime fact — when a pipeline last ran, how many rows it wrote, whether a check passed, who queried the asset. Regulators and frameworks like BCBS-239 and DORA care because freshness and processing evidence are part of proving a number is current and correctly produced; "the EMT file was generated at 06:12 from a run that ingested 1,204 holdings" is operational metadata. It is also what powers usage ranking and staleness alerts in an active catalog.

Q: Define passive versus active metadata.
A: Passive metadata merely describes assets and sits there to be read — a static schema dump or a wiki page that someone consults. Active metadata fires actions: it triggers alerts, tasks, approvals, and policies directly from changes to the metadata itself. The shift from passive to active is the difference between a catalog that documents governance and one that enforces it.

Q: What makes a catalog "active" rather than just a searchable inventory?
A: A catalog is active when events on metadata cause downstream effects without a separate system — a schema change emits an announcement, a failed quality check opens a task, a new PII tag triggers a masking policy. The metadata is not a passive record but the event source that drives workflows. OpenMetadata implements this through event subscriptions and governance workflows rather than requiring you to bolt on a separate orchestrator.

Q: Why does an architect design the catalog first in a regulated fund platform?
A: Because ownership, glossary, lineage, quality, and policy all hang off the catalog as the control plane, and CSSF/DORA reviewers expect a single authoritative answer to "who owns this and what depends on it". Designing it first means governance is structural rather than retrofitted into stale spreadsheets. Without it, impact analysis becomes guesswork and a schema change can silently break an EMT feed nobody knew was downstream.

Q: A schema change silently broke a downstream EMT feed nobody knew existed — how does an active catalog prevent this?
A: An active catalog with lineage knows the EMT feed consumes the changed table, so a schema-change event can automatically notify the registered consumer (an announcement or task) before or as the change lands. The metadata itself carries the dependency and fires the alert, rather than relying on tribal knowledge. This is the canonical "Do" task for this lesson: wire a schema-change announcement that notifies consumers of a gold table.

Q: What is OpenMetadata in one sentence?
A: OpenMetadata is an open-source metadata management platform and data catalog built on a schema-first metadata standard (JSON Schema), with a unified entity model, a pluggable ingestion framework, and APIs that expose discovery, lineage, quality, and governance. It is the FOSS catalog this lesson has you deploy and ingest into, comparable to Microsoft Purview, Collibra, and Alation.

Q: What does "schema-first" mean in OpenMetadata's design?
A: It means the metadata model is defined as formal JSON Schema specifications first, and the database tables, REST APIs, SDKs, and UI are all generated from or validated against those schemas. This gives a single source of truth for what a `Table`, `Glossary`, or `Domain` entity is, so connectors and clients stay consistent. The practical payoff is that the same entity shape is enforced whether metadata arrives via ingestion, the API, or the UI.

Q: Name the main building blocks of OpenMetadata's architecture.
A: The core is the OpenMetadata server (a Java/Dropwizard service exposing the REST API and entity store), a backing relational database (MySQL or PostgreSQL) for the metadata, a search engine (Elasticsearch or OpenSearch) for discovery, and the ingestion framework (Python) that pulls metadata from sources. The UI sits on top of the API. Together they form the catalog you stand up via Docker.

Q: What backing database and search engine does an OpenMetadata deployment use?
A: The metadata store is a relational database — either MySQL or PostgreSQL — chosen via the env file at deploy time. Discovery and search are powered by Elasticsearch or OpenSearch. The Docker quickstart bundles defaults, but production guidance is to bring your own managed database and search engine rather than relying on the quickstart containers.

Q: What is the OpenMetadata ingestion framework?
A: It is a Python framework (`openmetadata-ingestion`) of pluggable connectors and workflows that connect to a source, extract metadata, and push it into the server via the API. Workflows are configured with YAML and can run from the CLI (`metadata ingest -c config.yaml`), from a bundled Airflow, or from the ingestion container scheduled through the UI. It is what lets you pull metadata rather than type it.

Q: What are the distinct ingestion workflow types in OpenMetadata?
A: The main types are Metadata ingestion (tables, dashboards, topics, pipelines), Usage ingestion (query logs to rank most-used assets), Lineage ingestion (dependencies between assets), Profiler ingestion (column metrics and sample distributions), and Data Quality tests. There is also dbt ingestion (models, tests, descriptions) and a Great Expectations integration that lands test results. Each is a separately schedulable pipeline against the same source.

Q: How do you actually run an OpenMetadata ingestion workflow from the command line?
A: You write a YAML config describing the source connection and what to ingest, then run the `metadata` CLI from the `openmetadata-ingestion` package, e.g. `metadata ingest -c ./postgres-metadata.yaml`. The same workflow can instead be scheduled from the UI, which deploys it to the bundled Airflow or ingestion container. The CLI path is the fastest way to test a connector before scheduling it.

Q: What connector categories does OpenMetadata cover?
A: Connectors span databases (Postgres, Trino, Snowflake, BigQuery, etc.), dashboard/BI tools, messaging systems (Kafka), pipeline/orchestration tools (Airflow, Dagster, dbt), ML and storage services, and API/metadata sources. Each connector knows how to introspect its source's catalog and emit OpenMetadata entities. The breadth is what lets one catalog span a whole lakehouse rather than one engine.

Q: Which four sources does this lesson have you configure connectors for, and why those?
A: Postgres (the OLTP/source-of-truth tables), Trino/Iceberg (the lakehouse query engine and table format), dbt (the transformation layer and its tests/docs), and Kafka (the streaming/CDC topics). Together they cover the full NAV path's metadata — relational sources, the lake, transformations, and streams. The Do task is to run each pipeline and confirm assets land in the catalog.

Q: What does the dbt connector add to OpenMetadata that a plain database connector does not?
A: The dbt connector ingests the `manifest.json` and `catalog.json` artefacts to bring model-to-model lineage, model and column descriptions, and dbt test definitions into the catalog. A plain database connector only sees the resulting tables, not how they were built or documented. So dbt ingestion is how SQL-parsed column lineage and transformation context reach the catalog.

Q: What is the OpenMetadata entity model and why is it called the governance skeleton?
A: The entity model is the typed set of objects the platform manages — `Table`, `Topic`, `Dashboard`, `Pipeline`, `Domain`, `Team`, `User`, `GlossaryTerm`, `Tag`, plus relationships like owner and lineage. It is the governance skeleton because ownership, tiers, glossary links, domains, and policies all attach to these entities, so every governance decision has a place to live. Designing this skeleton well is the architect's core job in this lesson.

Q: What is a Domain in OpenMetadata?
A: A Domain is an organisational grouping that bounds a set of data assets under a business area and its responsible team — for example a "Fund Accounting" or "Security Master" domain. It gives a top-level ownership and discovery boundary and is the unit you assign data products and stewards to. Domains make the catalog navigable by business meaning, not just by physical source.

Q: What does assigning an "owner" to an asset mean and accomplish?
A: An owner is a team or user accountable for an asset; assigning it answers the "who owns this?" question instantly and routes tasks, announcements, and approvals to the right party. Ownership is the single most load-bearing piece of governance metadata because every other workflow needs a responsible party. The Done-when criterion is that every gold asset carries an owner.

Q: What is a "tier" in OpenMetadata and how is it used?
A: A tier is a criticality/importance classification (commonly Tier1 through Tier5) applied to an asset, signalling how business-critical it is. Tier-1 assets — like the `nav` or EMT-output tables — warrant stricter SLAs, more alerting, and tighter change control. Tiers let governance and alerting be proportionate, and the Done-when criterion requires every gold asset to carry one.

Q: What is a glossary term and how does linking one to a column help?
A: A glossary term is a controlled business definition (e.g. "Net Asset Value", "ISIN") stored in the catalog, optionally with synonyms and relationships. Linking it to a column binds the technical field to its agreed meaning, so a consumer reading `nav_per_share` sees the official definition and any policy. This is how business and technical metadata are reconciled, and the lesson requires linking at least one term to a gold asset.

Q: What is the difference between a glossary term and a tag/classification in OpenMetadata?
A: A glossary term is a business-meaning definition organised in a glossary hierarchy, expressing what a thing means. A tag belongs to a classification (a controlled vocabulary like "PII" or "Sensitivity") and expresses a label/policy attribute used for filtering and automation, such as `PII.Sensitive`. Glossary answers "what does this mean", classification/tags answer "what category/policy applies", and both can drive workflows.

Q: How would you answer "who owns this, who consumes it, what breaks if I change it?" from the catalog?
A: Open the asset, read its Owner field (who owns it), follow the downstream lineage edges (who/what consumes it), and inspect those downstream nodes to see what breaks if you change the column. The catalog's job is to make this a sub-two-minute lookup rather than an archaeology dig — that exact two-minute answer is the lesson's first Done-when criterion. It works only if ingestion has populated owners and lineage.

Q: What are catalog-driven workflows in OpenMetadata?
A: They are announcements, tasks, and approval/governance flows that originate from the catalog and are triggered by metadata events or human action — for example announcing a schema change, opening a task to fix a missing description, or routing a glossary-term proposal for approval. They turn the catalog from a reference into a collaboration and control surface. This is the mechanism behind "active" metadata.

Q: What is an announcement in OpenMetadata and when would you use one?
A: An announcement is a time-boxed notice posted on an asset (and surfaced to its owners/followers) about an upcoming or current event — planned downtime, a deprecation, or a schema change. You use it to proactively warn consumers before a change lands. The lesson's active-workflow task is a schema-change announcement that notifies registered consumers of a gold table.

Q: What is a task in OpenMetadata?
A: A task is a tracked unit of work attached to an asset — requesting a description, a tag, or a fix — assigned to an owner or team and closed when resolved. Tasks turn governance gaps into actionable, auditable items rather than informal asks. Combined with the activity feed they make stewardship a workflow inside the catalog.

Q: What is a governance workflow / approval flow in the catalog?
A: A governance workflow is a configurable, multi-step process the catalog runs — for example requiring approval before a glossary term is published or before an asset's classification changes. It encodes who must sign off and emits events (`GovernanceWorkflowChangeEvent`) that alerts can subscribe to. This is how regulated change control lives inside the catalog instead of in email.

Q: How do you wire a schema-change announcement that notifies consumers, conceptually?
A: You configure an event subscription (alert) whose source is schema-change events on the gold table, filter to that asset, and set a destination — a registered consumer's email, Slack, Teams, Google Chat, or a webhook. When ingestion detects the schema change, the subscription fires and the consumer is notified. Triggering it and confirming the consumer receives the notice is the third Done-when criterion.

Q: Which destination types can OpenMetadata alerts/event subscriptions send to?
A: Email (via the server's configured SMTP), Slack (incoming webhook), Microsoft Teams (workflow webhook with Adaptive Cards), Google Chat (webhook), and generic HTTP webhooks. The webhook destination is the escape hatch for custom integrations like paging or a downstream automation. You pick the destination per subscription so different audiences get different events.

Q: What alert types can an OpenMetadata event subscription have?
A: Notification, Observability, ActivityFeed, GovernanceWorkflowChangeEvent, and Custom. Notification covers metadata/collaboration events like ownership or tag changes; Observability covers data-system health like schema drift, quality failures, and pipeline failures; GovernanceWorkflowChangeEvent tracks approval flows. Choosing the right type plus filters is how you avoid alert noise.

Q: What is the difference between a "Notification" alert and an "Observability" alert in OpenMetadata?
A: Notification/System & Governance alerts fire on metadata and collaboration events — a new glossary term, a tag update, an ownership change. Observability alerts fire on the health of data systems — data-quality failures, schema drift, pipeline failures. The split matters because stewards care about governance events while on-call engineers care about observability events, and you route them to different destinations.

Q: How does OpenMetadata expose its data programmatically?
A: Through a comprehensive REST API generated from the JSON-Schema entity model, plus a Python SDK and a GraphQL-less but rich query interface; the ingestion framework itself is just a client of these APIs. You can create, read, patch, and search entities, manage lineage, and configure alerts entirely via the API. This API-first design is why everything in the UI is also automatable.

Q: A connector ingestion pipeline runs green but no assets appear in the catalog — first things to check?
A: First confirm the pipeline scope actually matched something: check the schema/table filter patterns (include/exclude regex) — an over-tight `schemaFilterPattern` is the classic cause of zero assets. Then verify the service connection points at the right database/catalog and that the credentials' role can see those objects. Finally check the ingestion logs for "0 records" and that search indexing (Elasticsearch/OpenSearch) succeeded, since unindexed entities won't surface in search.

Q: An OpenMetadata Postgres ingestion fails to connect — what do you verify first?
A: Verify the connection details the connector uses: host, port (`5432` by default), database name, and that the credentials authenticate and the role has `SELECT` on the system catalogs and target schemas. Check network reachability from wherever ingestion runs (the ingestion container or Airflow), not from your laptop, since container networking differs. Then read the connector log for the precise driver error rather than guessing.

Q: Why might Trino/Iceberg assets show tables but no lineage in OpenMetadata?
A: Table-level metadata comes from the metadata ingestion pipeline, but lineage usually requires a separate lineage/usage pipeline that parses query history, or it must be fed from OpenLineage events. If you only ran metadata ingestion, you get the assets but not the edges. The fix is to run the lineage/usage workflow or wire OpenLineage emission, which lesson 10.3.1 covers.

Q: How does dbt lineage end up inside OpenMetadata?
A: The dbt ingestion reads dbt's `manifest.json` to reconstruct model-to-model dependencies and column references, producing table- and column-level lineage from the compiled SQL. It also imports descriptions and test definitions. This is why SQL-based transformations give richer column lineage than opaque engine steps — the dependency graph is explicit in the manifest.

Q: What is DataHub and how is it positioned relative to OpenMetadata?
A: DataHub is an open-source metadata platform and data catalog, described as a third-generation catalog built around schema-first, stream-based, federated metadata. It is the event-sourced alternative to OpenMetadata in this lesson: where OpenMetadata centres an entity store and ingestion framework, DataHub centres a metadata event stream. Both are OpenLineage-friendly, so they can consume the same lineage events.

Q: What does "event-sourced metadata" mean in DataHub?
A: It means metadata changes are modelled as a stream of events rather than only as the current state of a database — every change is a proposal that, when accepted, produces a log event reflecting the new state. This stream lets subscribers react in near real time and rebuild derived stores (search, graph) by replaying events. It is the architectural difference an architect weighs against OpenMetadata's API-centric model.

Q: What is a Metadata Change Proposal (MCP) in DataHub?
A: An MCP is a request to change a piece of metadata in the Metadata Graph — it carries an entity, an aspect, and the proposed value. Source systems can publish MCPs synchronously over REST or asynchronously over Kafka for highly scalable ingestion. The serving tier (GMS) receives and validates each MCP before persisting it.

Q: What is a Metadata Change Log (MCL) in DataHub and how does it relate to an MCP?
A: An MCL is the event emitted after a change is committed, describing what actually changed in the metadata. GMS receives an MCP, validates it, writes the change, and in normal mode emits an MCL; not every MCP yields an MCL because duplicate/no-op changes are ignored. MCLs are the stream that downstream consumers (like the search/graph index builders) react to.

Q: What is GMS in DataHub?
A: GMS is the Generalized Metadata Service — the core backend that receives MCPs, validates them against the metadata model, persists aspects to storage, and emits MCLs. It also serves the read APIs (REST and GraphQL) used by the UI and clients. It is the equivalent role to OpenMetadata's server, but built around the proposal/log event flow.

Q: Explain DataHub's entity-aspect metadata model.
A: In DataHub an entity (e.g. a dataset) is composed of independent "aspects" — typed slices of metadata like ownership, schema, or tags — each versioned and updatable on its own. A change targets one entity+aspect pair, which is exactly what an MCP carries. This decomposition lets different producers update different aspects of the same asset without conflict.

Q: How does DataHub use Kafka in its architecture?
A: Kafka is the backbone for asynchronous metadata flow: MCPs can be published to Kafka for scalable ingestion, and MCLs are emitted to Kafka so consumers like the search-and-graph index builder (the MAE-style consumer) can apply changes to the search index and graph store. This is what makes metadata changes reflect "within seconds" and enables event-driven, metadata-triggered applications. It is the concrete realisation of "event-sourced".

Q: Does DataHub support both push and pull ingestion? Explain.
A: Yes. Pull ingestion is the framework-driven model where DataHub's ingestion recipes connect to sources and extract metadata on a schedule, like OpenMetadata's connectors. Push ingestion is source systems emitting MCPs directly (over REST or Kafka) as changes happen, which suits real-time and CI-driven metadata. Push fits the event-sourced design; pull fits legacy sources without native emission.

Q: How are MCLs applied to DataHub's search and graph indexes?
A: A consumer job reads the MCL stream from Kafka and, being entity-agnostic, runs the appropriate graph and search index builders to update Elasticsearch and the graph store. This keeps the read/serving stores eventually consistent with the committed metadata. Because it is driven off the log, rebuilding an index is a matter of replaying MCLs.

Q: What is CDC mode for MCL generation in DataHub and why does it exist?
A: In CDC (Change Data Capture) mode, MCLs are generated from database change events captured directly off the metadata storage layer rather than emitted by GMS after processing each MCP. This guarantees MCLs appear in the same order as the database writes, giving stronger ordering guarantees. It exists for deployments that need strict ordering of metadata changes for downstream consumers.

Q: How are both OpenMetadata and DataHub "OpenLineage-friendly"?
A: Both can consume OpenLineage run/job/dataset events to populate lineage, so the lineage you emit from Dagster, Spark, or dbt can land in either catalog. This decouples lineage production from the catalog choice. It also means an architect can evaluate either platform without re-instrumenting pipelines, since the lineage standard is shared.

Q: Microsoft Purview is the corporate alternative to know — what should you understand about it at build-vs-buy level?
A: Know that Purview is the Azure-default governance suite built around a Data Map of scanned assets, with classification, lineage, and a per-asset/consumption cost model that drives total cost as your asset count grows. At architecture level you weigh its native Azure integration against that per-asset pricing and its connector coverage. The lesson asks you to evaluate, not deploy it.

Q: Collibra is the EU financial-services incumbent — what is its recognised strength?
A: Collibra's strength is the operating model and stewardship layer — workflows, roles, policies, and a business-glossary-centric governance program that maps well to regulated EU-FS organisations. It is strong where governance is an organisational process, not just a technical catalog. An architect evaluates it for governance maturity and process fit rather than raw ingestion breadth.

Q: When evaluating build (OpenMetadata) versus buy (Purview/Collibra), what are the key trade-offs?
A: Building on OpenMetadata gives full control, no per-asset licensing, and open APIs, at the cost of running and maintaining the platform and its connectors. Buying gives managed operation, vendor support, and (with Collibra) mature stewardship workflows, at the cost of licensing/per-asset pricing and connector limits. In a regulated fund context you also weigh support SLAs, auditability, and EU data-residency.

Q: Why is per-asset cost a decisive factor in catalog build-vs-buy for a large fund platform?
A: Because a fund platform can have tens of thousands of tables, topics, and columns across many fund ranges, and a per-asset or per-scan pricing model (as in Purview) scales cost directly with asset count. A flat self-hosted FOSS option like OpenMetadata removes that variable cost but adds operational burden. The architect must model expected asset growth before committing.

Q: An auditor asks "who owns the `nav` table and what would break if its schema changed?" — how does the catalog answer in under two minutes?
A: Search the catalog for `nav`, open the asset, read the Owner (a named team like Fund Accounting) and Tier, then walk the downstream lineage to see consumers — for example the EMT-output model and any reports. Those downstream nodes are exactly "what breaks". Achieving this sub-two-minute answer is the lesson's headline Done-when criterion and depends on populated ownership and lineage.

Q: How would you model fund Domains in OpenMetadata for a Luxembourg fund administrator?
A: Create Domains aligned to the business control areas — e.g. Fund Accounting (NAV), Security Master (instruments/ISIN/LEI), Transfer Agency (investor/subscription data), and Regulatory Reporting (EMT/EPT, AIFMD). Assign each a responsible team and tier its critical assets. This makes the catalog navigable by the way the fund admin is organised and routes governance to the right stewards.

Q: How should ISIN and LEI be represented in the catalog's glossary?
A: As controlled glossary terms with their precise definitions and standards — ISIN (ISO 6166, the 12-character security identifier) and LEI (ISO 17442, the 20-character legal-entity identifier) — linked to the columns that carry them across tables. This binds every `isin`/`lei` column to one agreed meaning and validation expectation. It also gives auditors a single definition to reference rather than per-table interpretations.

Q: Why is catalog ownership of EMT/EPT output assets especially important?
A: EMT (European MiFID Template) and EPT (European PRIIPs Template) files are regulatory deliverables consumed by distributors, so a break or a wrong field has external, regulatory consequences. Marking the EMT/EPT output table Tier-1 with a clear owner and lineage means a schema change upstream triggers an announcement to the team accountable for the file. The catalog makes the regulatory dependency explicit rather than tribal.

Q: How does the catalog support a DORA expectation around critical data assets?
A: DORA pushes for knowing your critical assets, their owners, and dependencies for operational resilience and incident response; the catalog provides exactly that register — tiered assets, named owners, and lineage for impact analysis. When an incident hits a source, lineage shows the blast radius and announcements notify owners. The catalog turns resilience obligations into a queryable, evidenced control plane.

Q: What is the practical first step in the lesson's Do list and what confirms success?
A: Deploy OpenMetadata via its Docker quickstart and confirm the UI is reachable. Success is the UI loading at `http://localhost:8585` and `docker ps` showing the server, database, search, and ingestion containers healthy. Only once the UI is up do you move on to configuring connectors.

Q: At what URL is the OpenMetadata UI reachable after the Docker quickstart, and how do you confirm containers are up?
A: The UI is reachable at `http://localhost:8585` once startup completes. You confirm the stack with `docker ps`, which should list the running OpenMetadata server, the metadata database, the search engine, and the ingestion container. If `8585` does not respond, check the server container's logs for a failed database or search connection before anything else.

Q: What Docker tooling versions does the OpenMetadata quickstart require?
A: The documented prerequisites are Docker version 20.10.0 or newer and Docker Compose v2.2.3 or newer. You deploy with `docker compose --env-file ./env-<db> up --detach` using the release's compose file and an env file for MySQL or Postgres. Older Compose v1 (`docker-compose`) can cause issues, so confirm `docker compose version` first.

Q: On native Ubuntu, what is a common reason the OpenMetadata quickstart fails to start, and how is it different on WSL2/macOS?
A: On native Ubuntu the usual culprits are the Docker daemon permissions (your user not in the `docker` group) or insufficient memory/`vm.max_map_count` for Elasticsearch/OpenSearch, which needs raising via `sysctl -w vm.max_map_count=262144`. On WSL2 you set the same sysctl inside the WSL VM and may need to raise the VM's memory in `.wslconfig`; on Docker Desktop (macOS/Windows) you raise memory in Desktop's resource settings. The Elasticsearch memory-map limit is the classic cross-platform gotcha.

Q: The search container exits on startup with a `vm.max_map_count` error — what is the fix on Ubuntu?
A: Elasticsearch/OpenSearch require the kernel's `vm.max_map_count` to be at least `262144`; set it with `sudo sysctl -w vm.max_map_count=262144` and persist it in `/etc/sysctl.conf` (or a file in `/etc/sysctl.d/`). Then restart the stack. This is a host-kernel setting, so it applies to native Ubuntu directly and inside the WSL2 VM, not in the container.

Q: After deploying, what is the standard order of the remaining Do tasks?
A: Configure and run ingestion connectors for Postgres, Trino/Iceberg, dbt, and Kafka and confirm assets land; then assign an owner and a tier to every gold asset and link at least one glossary term; then wire one active workflow — a schema-change announcement that notifies registered consumers of a gold table. This sequence builds the entity skeleton before activating workflows on it. Each maps to a Done-when criterion.

Q: Why assign owners and tiers before wiring alerts?
A: Because alerts, announcements, and tasks route to owners and are often scoped or prioritised by tier — an alert with no owner has nowhere meaningful to go. Populating the governance skeleton first means the active workflows have a responsible party and a criticality signal to act on. Wiring alerts onto unowned, untiered assets produces noise nobody acts on.

Q: How do you verify "every gold asset carries an owner and a tier" at scale?
A: Use the catalog's search/filter to query gold assets lacking an owner or tier — OpenMetadata lets you filter by owner-is-empty or tier-is-empty — and the count should be zero. You can also pull this via the REST API to assert it in CI. The Done-when is a demonstrable "no gold asset is missing owner or tier".

Q: Why is "metadata as code via the API" preferable to manual UI tagging for a regulated platform?
A: Because API/SDK-driven metadata is reproducible, version-controllable, and auditable — you can assert in CI that every gold asset has an owner and tier, and re-apply it on rebuild. Manual UI tagging drifts and cannot be reviewed like code. In a regulated context the ability to evidence and reproduce governance state is itself a control.

Q: What is the risk of relying on stale spreadsheets instead of an active catalog?
A: Spreadsheets describe a point in time and are never re-validated against reality, so ownership, lineage, and quality drift silently and impact analysis becomes guesswork. There is no event to fire when a schema changes, so downstream breaks go unnoticed. The whole point of an active catalog is that the metadata is live and triggers action.

Q: What does "impact analysis" mean in the context of the catalog?
A: Impact analysis is determining everything affected by a proposed change — which downstream tables, dashboards, and regulatory outputs depend on a column you want to alter. The catalog answers it by traversing downstream lineage from the asset. Done well, "what breaks if I change `orders.amount`?" is a graph query, not a meeting.

Q: How does usage/operational metadata improve a catalog beyond static descriptions?
A: Usage ingestion ranks the most-queried tables and surfaces popular columns and frequent users, which helps prioritise tiering, deprecation, and stewardship effort. Operational signals like last-run time and row counts feed freshness and staleness alerts. This is metadata that informs decisions and fires actions, not just documents structure.

Q: What is the difference between "metadata that describes" and "metadata that fires actions"?
A: Describing metadata is passive — a schema, an owner field, a description that someone reads. Action-firing metadata is active — a change to it triggers an announcement, a task, an alert, or a policy evaluation automatically. The lesson's thesis is that an active catalog makes governance a consequence of metadata changes rather than a manual follow-up.

Q: Give an example where a policy fires *from* metadata rather than from a separate system.
A: When a column is tagged `PII.Sensitive` in the catalog, a masking/RLS policy applies to it automatically because the policy is bound to the tag, not maintained separately. Adding the tag is the metadata change; the masking is the action that fires from it. This keeps the policy and the metadata in one system, which is what "active" means.

Q: How does a new `PII` tag in the catalog ideally lead to masking on the lakehouse?
A: The tag is metadata that a policy engine subscribes to; when the column is tagged, the bound column-level masking or access policy is applied on Trino/Iceberg without separate configuration. The catalog is the source of truth and the trigger. This is the active-metadata pattern that lesson connects to later access-control entries (RLS/CLS, policy-as-code).

Q: Why does an architect prefer alerts and policies firing from the catalog rather than from a separate orchestrator?
A: Because keeping triggers in the catalog means governance state and its consequences live in one auditable place, with no drift between "what the catalog says" and "what the separate system enforces". A separate orchestrator must be kept in sync, which fails over time. Single-source-of-truth active metadata is more reliable and easier to evidence to a regulator.

Q: A consumer claims they were never told about a breaking schema change you announced — how do you investigate in the catalog?
A: Check whether the consumer was actually registered as an owner/follower of the gold table and whether the event subscription's filter matched the schema-change event and targeted their destination (email/Slack/webhook). Inspect the subscription's delivery logs/status for a failed send. The common cause is an unfollowed asset or a too-narrow filter, not a missing event.

Q: Why might an OpenMetadata alert never deliver even though the event occurred?
A: Likely the event subscription's filtering rules excluded that asset or event type, the destination is misconfigured (wrong Slack webhook URL, SMTP not set for email), or no recipient/owner was attached. Alerts only fire when the source, filter, and destination all line up. Start by re-checking the subscription's filter scope and then its destination configuration and delivery logs.

Q: What is the role of search (Elasticsearch/OpenSearch) in the catalog, and what breaks without it?
A: Search powers discovery — finding an asset by name, tag, owner, or glossary term — and ranks results. Without a healthy search engine, entities exist in the database but are not findable in the UI, so the "search for `nav`" lookup fails even though ingestion succeeded. A failed search container therefore looks like "ingestion didn't work" when it didn't index.

Q: Ingestion succeeded and the database has the entity, but search returns nothing — what is the likely cause?
A: The search index is out of sync or the search engine is unhealthy — entities were written to the metadata store but not indexed into Elasticsearch/OpenSearch. Check the search container's health and re-run the search-index/reindex job. This decouples "stored" from "discoverable", which is why both must be verified.

Q: Why is the entity model called "extensible", and how would you add fund-specific metadata?
A: Because OpenMetadata's JSON-Schema model supports custom properties on entities and custom classifications/tags, so you can add fund-specific attributes (e.g. a `fund_range` or `regulatory_template` property) without forking the platform. You define the custom property, then populate it via UI or API. This lets the generic catalog carry domain-specific governance metadata.

Q: What is the relationship between the catalog and lineage covered in the next lesson (10.3.1)?
A: Lineage (OpenLineage runs/jobs/datasets) is produced by emitters in your pipelines and fed into the catalog, where it becomes the impact-analysis graph attached to assets. The catalog consumes lineage; it does not by itself produce the column-level edges. So 10.1.1 stands up the catalog and 10.3.1 fills its lineage from OpenLineage events.

Q: Why is column-level lineage richer from dbt than from a black-box SaaS step?
A: dbt's compiled SQL is parseable, so the catalog can derive which output columns came from which input columns, giving column-level edges. A black-box SaaS step emits nothing about its internals, so lineage stops there and must be annotated as a gap. This is a known limit an architect documents rather than fakes.

Q: How would you handle a black-box step that emits no metadata into the catalog?
A: Represent it as an asset/job in the catalog and manually annotate its inputs and outputs (and a note that it is a black box), so lineage continues across the gap even without automatic edges. The catalog supports manually added lineage via the API/UI for exactly this. Honest annotation beats a broken or fabricated graph for audit.

Q: What does "tiering" let you do operationally that a flat asset list cannot?
A: Tiering lets you apply proportionate controls — stricter SLAs, more frequent quality checks, tighter change approval, and louder alerts on Tier-1 assets, while leaving Tier-4/5 lightly governed. A flat list treats everything equally, which either over-governs trivia or under-governs critical assets. For a fund platform, the NAV and EMT outputs are Tier-1 and warrant the strongest controls.

Q: Why is "the catalog is the control plane" a useful mental model for an architect?
A: Because it frames the catalog as the central place where ownership, glossary, lineage, quality, and policy are coordinated and from which workflows are driven — the plane that controls governance, distinct from the data plane that moves data. Designing it first makes every later governance capability attach to a known structure. It is the architect-level reason this lesson comes first in Phase 6.

Q: How do you confirm the Kafka connector landed topics as assets in OpenMetadata?
A: After running the Kafka (messaging) metadata ingestion, search the catalog for the topics and open one to see its schema (if a schema registry is wired) and its `Topic` entity details. If nothing appears, check the bootstrap servers, the topic include/exclude patterns, and schema-registry connectivity. Topics are first-class entities, so they should be searchable and ownable like tables.

Q: What is the difference between a Service and a Pipeline in OpenMetadata ingestion terms?
A: A Service is the configured connection to a source (e.g. your Postgres or Kafka instance), and a Pipeline (ingestion pipeline) is a schedulable workflow that runs against that service to extract a particular kind of metadata (metadata, usage, lineage, profiler). One service can have several pipelines. You create the service once, then attach the pipelines you need.

Q: Why run a profiler pipeline, and what does it add to the catalog?
A: The profiler computes column-level metrics and sample distributions — null counts, min/max, distinct counts, value histograms — and stores them on the asset. This both informs data-quality rule design ("rules from profiling, not imagination") and gives consumers a sense of the data's shape. It is operational/statistical metadata that later DQ lessons build rules against.

Q: How does profiling in the catalog connect to data-quality work later in Phase 6?
A: Profiled distributions are the evidence you write quality expectations against — you set ranges, null thresholds, and uniqueness checks based on observed values rather than guesses. The catalog surfaces those profiles and can also receive DQ pass/fail results back. So the catalog is both the input (profiles) and the destination (results) for DQ.

Q: What governance question does the glossary specifically answer that ownership does not?
A: Ownership answers "who is accountable", while the glossary answers "what does this mean and is it the agreed definition" — semantic consistency across teams. Two tables both called `nav` should map to one glossary definition even if owned by different teams. The glossary is how the business vocabulary stays consistent independent of who owns the physical asset.

Q: In OpenMetadata, what is the difference between a Team and a User as owners?
A: A User is an individual account, while a Team is a group (often mapped from your identity provider) that can own assets collectively and receive routed tasks/alerts. Owning critical assets with a Team rather than a person avoids single-point ownership that breaks when someone leaves. For Tier-1 fund assets, team ownership is the resilient choice.

Q: How does the activity feed support catalog-driven collaboration?
A: The activity feed is the timeline of changes, announcements, tasks, and conversations on an asset, so stewards and consumers see what changed and can discuss or open tasks in context. It turns the catalog into a collaboration surface rather than a static page. `ActivityFeed` is also an alert type you can subscribe to for specific assets.

Q: Why is "metadata versioning" valuable in a regulated catalog?
A: OpenMetadata versions entity metadata, so you can see how a table's schema, owner, tags, or description changed over time and when. For audit and change control this provides an evidenced history — "the EMT-output schema changed on this date, by this pipeline". It supports answering provenance and change questions without external logs.

Q: An architect must choose between OpenMetadata and DataHub for a real-time, event-driven platform — which design points favour DataHub?
A: DataHub's stream-based, event-sourced model (MCP/MCL over Kafka) suits sources that push metadata as changes happen and applications that must react within seconds. Its entity-aspect decomposition lets many producers update different aspects independently. If the platform is already Kafka-centric and needs push ingestion and subscriptions, that favours DataHub.

Q: Which design points favour OpenMetadata over DataHub?
A: OpenMetadata's integrated entity store, broad batteries-included connectors, built-in data-quality/profiling and governance workflows, and simpler operational footprint favour it when you want a complete catalog out of the box without standing up a heavy Kafka-and-aspect pipeline. Its UI-driven announcements, tasks, and alerts cover the active-metadata needs of this lesson directly. For most fund-admin catalogs it is the faster path to the Done-when criteria.

Q: Both catalogs can ingest OpenLineage — why does that reduce lock-in?
A: Because you instrument your pipelines once to emit the OpenLineage standard, and either catalog can consume those events, so switching catalogs does not mean re-instrumenting lineage. The lineage producer and the catalog are decoupled by the standard. This lets an architect defer or change the build-vs-buy decision without rework on lineage.

Q: What is the danger of leaving gold assets untiered in a fund platform?
A: Untiered assets get the same (often low) priority for alerting, quality, and change control as trivia, so a critical NAV or EMT table might not raise the alarm it should when it changes or breaks. Tiering is how criticality drives proportionate governance. Leaving the most important regulatory assets untiered is a direct control gap.

Q: How would you prove to a CSSF/DORA reviewer that your platform has a single answer to "who owns this and what depends on it"?
A: Demonstrate live in the catalog: search the asset, show the named owning team and tier, then traverse downstream lineage to its consumers and regulatory outputs — all in one screen, in under two minutes. Back it with the API/CI assertion that every gold asset carries an owner and tier. The catalog being the single, queryable source is the evidence.

Q: Why is "pull metadata via connectors rather than type it" a governance principle, not just convenience?
A: Hand-typed metadata is immediately stale and unverifiable, whereas connector-pulled metadata is refreshed from the source on a schedule, so the catalog reflects reality. Automation is what keeps the control plane trustworthy over time. A catalog that requires manual data entry decays into the spreadsheet problem it was meant to solve.

Q: What confirms the lesson's third Done-when — the schema-change announcement?
A: You trigger the announcement on a gold table (or make a schema change that fires the configured subscription) and confirm a registered consumer is actually notified via the chosen destination. Seeing the notification arrive — not just the subscription existing — is the proof. This demonstrates active metadata end to end: a metadata change produced an action.

Q: Summarise why metadata management is the first entry in a governance phase.
A: Because the catalog is the control plane everything else hangs off — ownership, glossary, lineage, quality, and policy all attach to its entity model, and active metadata turns changes into governed actions. Standing it up first gives the architect a structure to answer impact and ownership questions and to enforce change control. In a regulated fund context it is the difference between governance as live code and governance as stale prose.

Q: What command runs an OpenMetadata profiler workflow from the CLI, and how does it differ from the metadata-ingestion command?
A: You run `metadata profile -c <config>.yaml` for profiling, where the verb is `profile` rather than the `ingest` used for plain metadata extraction. The profiler config still uses a `metadata-rest` sink to push computed column metrics back to the server, but it adds a `processor` block with profiler options. Using `metadata ingest` on a profiler config silently does the wrong thing, so the verb must match the workflow type.

Q: What CLI command runs OpenMetadata data-quality tests, and where do results land?
A: You run `metadata test -c <config>.yaml`, which executes the test cases defined for a table and writes pass/fail results back through the `metadata-rest` sink onto the asset's Data Quality tab. The tests can be defined inline in the processor section of the YAML or attached to the table via the UI/API. Results become operational metadata that an Observability alert can subscribe to.

Q: Which three filter-pattern keys scope an OpenMetadata database ingestion, and what is their precedence?
A: The keys are `databaseFilterPattern`, `schemaFilterPattern`, and `tableFilterPattern`, each taking `includes` and `excludes` lists of regular expressions. The engine applies `includes` first and then `excludes`, so an explicitly excluded object is dropped even if an include regex matched it. An over-tight `schemaFilterPattern.includes` is the classic reason a green pipeline lands zero assets.

Q: You have two schemas named `public` in different databases and your `tableFilterPattern` is matching the wrong one — what flag fixes this?
A: Enable `useFqnForFiltering: true` so the regex is applied to the fully-qualified name (`service.database.schema.table`) rather than just the bare object name. Without it, a pattern like `public.orders` cannot disambiguate identically-named schemas across databases. This is the standard fix when filter patterns collide on duplicate names in a multi-database fund warehouse.

Q: What are the default OpenMetadata admin credentials after the Docker quickstart, and why must you change them?
A: The seeded basic-auth account is `admin@open-metadata.org` with password `admin`, which is fine for a local laptop demo but is a hard finding in any shared or production deployment. The docs explicitly say to delete the default admin account and wire real SSO/OIDC before exposing the catalog. For a CSSF-regulated platform, leaving the default credential live is an immediate access-control gap.

Q: The OpenMetadata UI is on `8585` but what is on `8080`, and why does this matter when both come up in the quickstart?
A: Port `8080` is the bundled Airflow that runs and schedules ingestion pipelines deployed from the UI, distinct from the catalog UI on `8585`. If scheduled ingestions never run, the Airflow on `8080` is where you inspect the DAG and its logs, not the `8585` server. Confusing the two leads people to debug the wrong container when a pipeline won't fire.

Q: A scheduled ingestion shows "queued" forever in the OpenMetadata UI but never runs — first thing to check?
A: Check the ingestion/Airflow container health and that the server can reach it, because UI-scheduled pipelines are deployed as Airflow DAGs and a dead or unreachable Airflow leaves them stuck queued. Verify the `pipelineServiceClient` configuration (the URL the server uses to reach Airflow) and the Airflow logs on `8080`. The CLI path (`metadata ingest -c`) bypasses Airflow entirely and is the fastest way to confirm the connector itself works.

Q: How would you assign an owner to every gold table in bulk rather than clicking each in the UI?
A: Use the REST API or Python SDK to `PATCH` each table entity's `owner` field, iterating a list of gold-table FQNs from a script so the operation is reproducible and reviewable. This is "metadata as code" — the same script can run in CI to assert and re-apply ownership after a rebuild. Bulk UI clicking drifts and leaves no audit trail, which fails the reproducibility a regulator expects.

Q: What HTTP method does OpenMetadata use to update a single field like `tier` on an existing entity, and why not PUT?
A: It uses `PATCH` with a JSON Patch body to change only the targeted field (e.g. add the `Tier.Tier1` tag), leaving the rest of the entity intact. A `PUT` replaces the whole entity and risks clobbering fields you did not send, so partial updates use `PATCH`. This matters when many producers update different aspects of the same asset and must not overwrite each other.

Q: How would you assert in CI that no gold asset is missing an owner or a tier?
A: Query the search/list API filtering gold assets where `owner` is empty or no `Tier` tag is present, and fail the pipeline if the returned count is non-zero. OpenMetadata's search supports `owner.name` and tier filters, so the assertion is a single API call plus a count check. This turns the "every gold asset carries an owner and a tier" Done-when into an automated, evidenced control.

Q: A Trino/Iceberg ingestion lists databases but the tables are empty shells with no columns — what is the likely cause?
A: The connecting role can see the catalog and schema metadata but lacks privileges to introspect column definitions, or the Iceberg catalog/metastore the connector points at is not the one holding the tables' current snapshots. Verify the role's `SELECT`/describe privileges and that the connector's catalog name matches Trino's. Empty shells almost always mean a permissions or wrong-catalog problem, not a bug.

Q: How do you ingest Iceberg table metadata into OpenMetadata — via Trino, or directly?
A: You can either point the Trino connector at the catalog so Iceberg tables surface as Trino tables, or use the dedicated Iceberg connector that reads the catalog (Hive metastore, Glue, REST, etc.) directly. The direct Iceberg connector captures Iceberg-native details, while the Trino path captures them as the query engine sees them. The choice depends on whether you want engine-centric or format-native metadata.

Q: What is a Data Product in OpenMetadata and how does it relate to a Domain?
A: A Data Product is a curated, owned bundle of assets published within a Domain as a consumable unit — for example a "NAV Dataset" product inside the Fund Accounting domain. It gives consumers a discoverable, governed package rather than loose tables, and inherits the domain's ownership boundary. Domains are the organisational container; data products are the consumable offerings inside them.

Q: How do you wire a custom property like `regulatory_template` onto Table entities in OpenMetadata?
A: You extend the `Table` entity type with a custom property (string or enum) via Settings → Custom Properties or the type API, then populate it per table through the UI or a `PATCH`. Because the model is JSON-Schema based, this adds fund-specific metadata without forking the platform. You could then filter or alert on tables where `regulatory_template = EMT`.

Q: What does the OpenMetadata usage workflow need from the source that plain metadata ingestion does not?
A: It needs access to the source's query history or query logs (e.g. the database's query log table or a log file) to compute which tables and columns are most queried and by whom. Plain metadata ingestion only reads the catalog structure, not access patterns. Without query-log access the usage pipeline produces no rankings, so popularity and most-used columns stay blank.

Q: How does OpenMetadata derive lineage from a query-log/usage source versus from dbt?
A: From usage it parses the SQL in the query history to infer table-to-table (and sometimes column) lineage between assets that appear together in queries. From dbt it reads `manifest.json` to get explicit model-to-model and column references from compiled SQL. dbt gives deterministic column-level edges; query-log parsing is best-effort and depends on the queries actually run.

Q: Why might dbt ingestion land models but no column-level lineage, and how do you fix it?
A: Column-level lineage needs the compiled SQL and a complete `catalog.json` plus `manifest.json`; if you supplied only `manifest.json` or ran `dbt compile` without `dbt docs generate`, the catalog artefact is missing and columns can't be resolved. Regenerate both artefacts with `dbt docs generate` and point the connector at both. Missing or stale dbt artefacts are the usual cause of table-only lineage.

Q: A glossary term won't link to a column because the term is still "Draft" — what is happening?
A: OpenMetadata glossary terms move through a status lifecycle (Draft → Approved), and if a glossary has an approval workflow, terms stay in Draft until a reviewer approves them, limiting their use. Approve the term (or complete its governance workflow) so it becomes available to attach to assets. This is the approval flow working as designed, not a bug.

Q: How does a glossary-term approval workflow encode who must sign off in OpenMetadata?
A: You configure a governance workflow on the glossary with reviewers/approvers, so proposing or editing a term creates a task routed to those reviewers and the term cannot publish until they approve. The workflow emits `GovernanceWorkflowChangeEvent` events that alerts can subscribe to. This puts controlled-vocabulary change control inside the catalog rather than in email threads.

Q: What is the difference between a Classification and a Tag in OpenMetadata's model?
A: A Classification is the controlled vocabulary container (e.g. `PII`, `Sensitivity`), and a Tag is an individual label within it (e.g. `PII.Sensitive`). You create the classification once, define its tags, then apply tags to assets/columns. Policies and auto-classification operate on tags, while the classification groups and governs them.

Q: How does OpenMetadata's auto-PII classification work during ingestion?
A: The profiler/auto-classification feature samples column data and applies tags like `PII.Sensitive` or `PII.NonSensitive` when values match recognised patterns (emails, names, identifiers) or column-name heuristics. You enable it in the profiler/classification workflow config rather than tagging by hand. For a fund platform this is how investor-PII columns in TA data get flagged automatically for masking policies.

Q: An auto-classifier mislabels the `isin` column as PII — how do you correct and prevent it?
A: Remove the incorrect tag via the UI/API and, because `isin` is a public security identifier not personal data, exclude it from the classifier or rely on a corrected pattern; you can also disable auto-classification on that column. Auto-classification is heuristic, so an architect reviews and corrects its output rather than trusting it blindly. Documenting the correction matters for audit defensibility.

Q: How would you scope an event subscription so only schema changes on the EMT-output table fire it, not every table?
A: Set the subscription's source to schema/entity-change events, then add a filter that matches only the EMT-output asset (by FQN or by a tag/domain) so unrelated changes are excluded. Without the filter, the subscription is global and floods consumers with noise. Precise source-plus-filter scoping is how you keep the announcement signal meaningful.

Q: An OpenMetadata Slack alert never arrives though the event fired — what is the most common misconfiguration?
A: The most common cause is a wrong or expired Slack incoming-webhook URL on the subscription's destination, so the server posts to nowhere. Check the destination URL and the subscription's delivery status/logs for the failed POST. Email destinations fail similarly when SMTP is unconfigured on the server — destination misconfiguration, not the event, is usually at fault.

Q: How do you make an OpenMetadata email alert work at all on a fresh deployment?
A: You must configure the server's SMTP settings (host, port, sender, credentials) in the deployment config/env, because email destinations rely on the server's mailer which is unset by default. Until SMTP is configured, any email subscription silently fails to deliver. This is why a brand-new quickstart's email alerts "do nothing" — there is no mail transport wired yet.

Q: What is the difference between an announcement and an event-subscription alert for notifying consumers of a schema change?
A: An announcement is a human-authored, time-boxed notice posted on the asset and shown to its followers/owners — proactive and manual. An event-subscription alert is an automated rule that fires on the schema-change event itself and pushes to a destination like Slack or a webhook. The lesson's active-workflow can use either, but the alert is the truly "active" path where the metadata change itself triggers the notification.

Q: Why is following an asset (or being its owner) a prerequisite for some notifications, and how does that cause "I was never told"?
A: Certain notification paths target an asset's owners and followers, so a consumer who never followed the table and isn't an owner receives nothing even when an event fires. The fix is to register the consumer as a follower/owner or to add them explicitly as a destination on the subscription. An unfollowed asset is the classic reason a consumer claims they were never notified.

Q: In DataHub, what exactly does an MCP carry, and over which two transports can it be sent?
A: An MCP (Metadata Change Proposal) carries an entity URN, a single aspect name, and the proposed aspect value — a targeted change to one entity+aspect pair. It can be sent synchronously over REST to GMS or asynchronously over Kafka for high-throughput ingestion. The REST path gives immediate validation; the Kafka path scales but is eventually applied.

Q: Why does not every DataHub MCP produce an MCL?
A: Because GMS validates and applies the proposal, and if the change is a no-op or duplicate (the aspect value is unchanged), it commits nothing new and emits no MCL. MCLs represent actual committed changes, so they are deduplicated against the current state. This keeps the change-log stream free of meaningless events that would otherwise re-trigger every downstream consumer.

Q: What runs the DataHub CLI ingestion, and what is its default sink if you omit one?
A: You run `datahub ingest -c <recipe>.yml`, and since acryl-datahub `0.8.33.2+` the default sink is the DataHub REST endpoint at `http://localhost:8080` (or `${DATAHUB_GMS_URL}`). The recipe has exactly one source and one sink. So a recipe with only a source still ships metadata to local GMS unless you override the sink.

Q: How do you point a DataHub recipe at a remote GMS with auth, by environment or by config?
A: Either set `DATAHUB_GMS_URL` and `DATAHUB_GMS_TOKEN` env vars before `datahub ingest -c recipe.yml`, or write an explicit `sink: {type: datahub-rest, config: {server: ..., token: ...}}` block in the recipe. Env vars are convenient for CI secrets; the explicit block is self-documenting. Both override the localhost default sink.

Q: Compare DataHub's `datahub-rest` and `datahub-kafka` sinks at an architecture level.
A: The `datahub-rest` sink sends MCPs synchronously to GMS and surfaces validation errors immediately, which suits CI and smaller batches. The `datahub-kafka` sink publishes MCPs to Kafka for asynchronous, high-throughput ingestion at the cost of deferred validation and eventual application. The choice mirrors the synchronous-vs-async ingestion trade-off central to DataHub's event-sourced design.

Q: What is the role of the MAE/MCL consumer in keeping DataHub's search and graph fresh?
A: A dedicated consumer job reads the committed MCL stream from Kafka and, being entity-agnostic, runs graph and search index builders to update Elasticsearch and the graph store within seconds. This is what makes a metadata change reflect in DataHub search almost immediately. Because it is log-driven, a full reindex is just a replay of MCLs.

Q: When would an architect choose DataHub's CDC mode for MCL generation over GMS-emitted MCLs?
A: When downstream consumers need MCLs in exactly the same order as the underlying database writes — CDC mode captures change events directly off the metadata storage layer rather than emitting them after GMS processes each MCP. This gives stronger ordering guarantees at the cost of more infrastructure. It is a deployment choice for strict-ordering requirements, not the default.

Q: An architect is on Azure and the fund platform is small with few tables — does that argue for Purview or OpenMetadata?
A: A small asset count weakens Purview's main drawback (per-asset/consumption pricing) while its native Azure integration is a plus, so it can be the pragmatic buy. As tables scale into the tens of thousands across fund ranges, the per-asset cost model and the flat-cost OpenMetadata alternative re-enter the comparison. The decision hinges on projected asset growth and existing Azure commitment.

Q: At evaluation level, what is Collibra's recognised weakness relative to OpenMetadata?
A: Collibra's strength is stewardship workflows and operating model, but its connector breadth and technical/automated ingestion depth are generally narrower and its licensing heavier than a FOSS catalog. So for an engineering-led lakehouse with many heterogeneous sources, OpenMetadata's batteries-included connectors and APIs can ingest more, faster, without per-asset fees. You weigh Collibra's governance maturity against this ingestion and cost trade-off.

Q: How do you install just the Postgres connector for the OpenMetadata ingestion CLI rather than every dependency?
A: You install the ingestion package with the connector extra, e.g. `openmetadata-ingestion[postgres]`, which pulls only the Postgres driver and the `metadata` CLI rather than every connector's dependencies. Each source has its own extra (`[trino]`, `[kafka]`, `[dbt]`), so you install only what a given pipeline needs. A missing extra is the usual cause of an "unknown connector" or import error when you first run `metadata ingest -c`.

Q: How does the catalog help satisfy BCBS-239's principle that critical data has clear ownership and lineage?
A: The catalog supplies the register BCBS-239 implicitly demands — named owners, tiers, and traversable lineage for every critical data element feeding risk and regulatory numbers. When a number's accuracy is questioned, you walk lineage to its sources and show the accountable owner. This converts the principle's expectations into a queryable, evidenced control rather than documentation.

Q: How would an active catalog support a DORA incident response when a source system fails?
A: Lineage from the failed source shows the blast radius — every downstream table, EMT/EPT output, and report affected — while ownership routes notifications to the accountable teams via announcements/alerts. This turns "what is impacted and who do we tell" into a graph traversal plus an automated notification. The same evidence supports the post-incident report DORA expects.

Q: Why mark the EMT-output table Tier-1 but its upstream staging tables a lower tier?
A: The EMT output is an external regulatory deliverable consumed by distributors, so a defect has regulatory and reputational consequences warranting the strictest SLAs, alerting, and change control. Internal staging is recoverable and lower-stakes, so over-tiering it wastes governance effort and creates alert noise. Tiering aligns control intensity with business and regulatory criticality.

Q: An auditor asks for the history of who changed the EMT-output schema and when — how does the catalog answer?
A: OpenMetadata versions entity metadata, so the asset's version history shows each schema/owner/tag/description change with timestamp and the actor or pipeline responsible. You open the table's versions to evidence "the schema changed on this date via this ingestion". This provenance lives in the catalog without needing external change logs.

Q: Why is team ownership rather than individual ownership the resilient choice for Tier-1 fund assets?
A: A Team owner survives staff turnover and routes tasks/alerts to a group inbox, whereas a single User owner becomes a dead routing target when that person leaves, breaking notifications. For Tier-1 NAV and EMT assets, continuity of the accountable party is itself a control. Mapping teams from the identity provider keeps membership current automatically.

Q: How do you model a corporate action or fund-event reference table's ownership across two domains that both touch it?
A: Assign a single accountable owner (usually the domain that produces or masters it, e.g. Security Master) and represent the consuming domain via lineage and followers rather than co-ownership, so accountability stays unambiguous. Shared ownership dilutes the "who do I call" answer the catalog exists to give. Lineage and follower relationships capture the cross-domain dependency without splitting accountability.

Q: What operational-metadata signals would you put behind a freshness/staleness alert on the `nav` table?
A: The last successful ingestion/run time and the expected cadence (e.g. produced by 06:30 each valuation day), plus row counts versus expected, so an alert fires if the table is stale past its SLA or arrives empty. These are operational metadata the profiler/ingestion records on the asset. A staleness alert on a Tier-1 NAV table catches a missed run before it reaches NAV publication.

Q: How does column profiling concretely inform a later data-quality rule for an `isin` column?
A: Profiling reveals the observed distinct count, null fraction, and string-length distribution, so you can write rules grounded in reality — `isin` should be 12 characters, non-null, and unique per row — rather than guessed thresholds. The catalog surfaces these profiles and later receives the test results. This is the "rules from profiling, not imagination" principle in practice.

Q: A profiler run is extremely slow on a billion-row Iceberg table — what config knobs reduce its cost?
A: Set `profileSample` to sample a percentage or row count instead of scanning the full table, raise `threadCount` to parallelise metric computation, and use `computeMetrics: false` selectively or limit which columns/metrics run. Sampling trades exactness for tractable runtime on huge fact tables. For a fund platform you sample large holdings/transactions tables while fully profiling smaller reference tables.

Q: Ingestion wrote the entity to the database but search still returns nothing — what concrete action restores discoverability?
A: Re-run the search reindex job (or `Recreate Indexes` from the UI's Search Index settings) and confirm the Elasticsearch/OpenSearch container is healthy, because the entity exists in the metadata store but was never indexed. "Stored" and "discoverable" are decoupled, so a reindex bridges them. A failed search container makes successful ingestion look like a failure.

Q: When you manually annotate lineage across a black-box step, what level of detail can you honestly claim and what must you flag?
A: You can honestly claim table-level edges (this feed reads tables A and B, writes table C) because those are observable inputs and outputs, but you must flag that no column-level lineage exists and that the transformation logic is opaque. Asserting column-to-column edges through a black box would be fabricated and indefensible to an auditor. The honest annotation is "table-level only, internals unknown", which is a documented limit rather than a gap pretending to be complete.

Q: Why is "the catalog consumes lineage, it does not produce column edges" an important boundary for 10.1.1 versus 10.3.1?
A: In 10.1.1 you stand up the catalog and its entity skeleton, but the column-level edges come from OpenLineage emitters and dbt/SQL parsing fed in later (10.3.1) — the catalog stores and visualises what producers emit. Expecting the catalog alone to invent lineage leads to disappointment when only table-level or no edges appear. Knowing the boundary tells you to run lineage/usage pipelines or emit OpenLineage, not to wait for the catalog to derive it.

Q: Summarise the end-to-end "active metadata" loop this lesson asks you to demonstrate on a gold table.
A: A schema change on a gold table is itself the metadata event; an event subscription scoped to that asset detects it, applies its filter, and pushes a notification to a registered consumer's destination, who actually receives it. The proof is the notification arriving, not merely the subscription existing. That single loop — metadata change in, governed action out — is the concrete embodiment of an active catalog and the lesson's third Done-when.
