# Phase 1 · 8.5.1 Architecture notations & decision records (C4, ADR) — 150 self-test questions

Q: What does C4 stand for in the C4 model, and who created it?
A: C4 stands for the four hierarchical levels of abstraction: Context, Container, Component, and Code. It was created by Simon Brown as a lightweight, notation-independent way to describe software architecture at different zoom levels. The core idea is that you tell a "map of your code at different scales" story, the way you'd zoom out from a street map to a country map.

Q: Name the four levels of the C4 model in order from most zoomed-out to most zoomed-in.
A: The order is Context (level 1), Container (level 2), Component (level 3), and Code (level 4). Each level zooms in one step: Context shows the system among its users and neighbours, Container shows the deployable/runnable pieces, Component shows the major building blocks inside one container, and Code shows classes or schema inside one component. You almost never draw all four for the same system.

Q: What does a C4 System Context diagram show?
A: It shows your software system as a single box in the middle, surrounded by the people (actors) who use it and the other systems it interacts with. It deliberately hides all internal detail so that non-technical stakeholders can see scope and boundaries. For a NAV-calculation system, the context diagram would show the fund accountant, the order-management system, the market-data feed, and the regulator's reporting gateway, but nothing about how NAV is computed inside.

Q: What does a C4 Container diagram show, and what level does it sit at?
A: The Container diagram is level 2; it zooms into the single system box from the context diagram and shows the separately deployable or runnable units inside it — web apps, APIs, databases, message queues, and scheduled jobs. Every arrow is labelled with the protocol and intent, e.g. "reads EMT files via SFTP" or "publishes NAV events over Kafka". It is the diagram most architecture work stops at because it captures the high-level technology shape without drowning in internal class detail.

Q: In C4, what exactly is a "container"? Is it a Docker container?
A: No — a C4 container is "an application or a data store; something that needs to be running for the overall system to work", i.e. a runtime boundary around executing code or stored data. It is a logical/architectural unit, not a Docker or OS container; Simon Brown chose the word before Docker popularised it and explicitly warns against conflating them. A single C4 container (say a Spring Boot API) might run inside one Docker container, several, or none — deployment is modelled separately.

Q: Give five concrete examples of things that count as C4 containers.
A: Examples include a server-side web application (Node.js, ASP.NET, Java EE), a single-page browser application (Angular, React), a mobile app, a database (PostgreSQL, MongoDB), and a serverless function (AWS Lambda, Azure Function). Shell scripts, file systems, blob stores like S3, and message brokers also count. The unifying test is "does this need to be running, or hold state, for the system to work?".

Q: What does a C4 Component diagram show, and why is it level 3?
A: A Component diagram zooms inside one single container and shows its major structural building blocks — groupings of related functionality behind an interface, such as a "NAV Calculator", an "ISIN Validator", or a "TA Register Loader" — and how they collaborate. It is level 3 because it sits one zoom below the container; you draw one component diagram per container that warrants it, not one for the whole system. Most teams rarely keep these up to date because they drift from the code quickly.

Q: What is at C4 level 4 (Code), and why is it usually not hand-drawn?
A: Level 4 (Code) shows the implementation of a single component — UML class diagrams, entity-relationship diagrams, or similar. It is usually not hand-drawn because it changes constantly and is far better auto-generated from the codebase by an IDE or tool when actually needed. Most teams skip drawing level 4 entirely and let the source code itself be the level-4 "diagram".

Q: Why does C4 say most architecture work stops at the Container level?
A: Because Context plus Container together answer the questions stakeholders and reviewers actually ask — what is the system, who uses it, what are its moving parts, and how do they talk — without incurring the maintenance cost of component and code diagrams that drift the moment someone refactors. Container-level diagrams are stable enough to live in a repo and still meaningful months later. Going deeper is justified only for a specific container that is genuinely complex or contentious.

Q: What would force you to draw a Component (level 3) diagram for a particular container?
A: You drop to component level when one container is internally complex enough that reviewers cannot reason about it from the container diagram alone — e.g. a NAV engine with intricate pricing, accruals, and fee-calculation responsibilities whose internal boundaries are themselves an architectural decision. The trigger is "the interesting decisions now live inside this one box". You draw it only for that container, and only while it stays useful.

Q: What are the three supplementary C4 diagram types beyond the four core levels?
A: They are the System Landscape diagram, the Dynamic diagram, and the Deployment diagram. The System Landscape shows many systems and their relationships across an enterprise; the Dynamic diagram shows the ordered runtime collaboration for one scenario (like a UML communication diagram); and the Deployment diagram maps containers onto infrastructure nodes. They complement the static Context/Container/Component view rather than replacing it.

Q: What does a C4 Deployment diagram add that a Container diagram does not?
A: A Deployment diagram maps the logical containers onto the physical or virtual infrastructure they run on — nodes such as Kubernetes pods, VMs, on-prem servers, or cloud regions — including how many instances of each container exist. The Container diagram says what the runnable units are; the Deployment diagram says where and on how much hardware they live. For a regulated fund platform this is where you'd show data residency, e.g. NAV data confined to an EU region.

Q: What is a C4 System Landscape diagram useful for in an asset-management firm?
A: It gives a bird's-eye view of many systems and how they connect across the whole estate — order management, transfer agency, fund accounting, regulatory reporting, market data — independent of any single system's internals. It is the picture an enterprise architect uses to spot duplicated capabilities or undocumented integrations. It is explicitly a landscape, not a context diagram, because it has no single "system in scope".

Q: What is meant by C4 being "notation independent" and "tooling independent"?
A: C4 prescribes the abstractions (person, system, container, component) and the zoom levels, but not the shapes, colours, or arrows you must use, and not the tool you draw with. You can render C4 with boxes-and-lines in any tool, or with dedicated tooling like Structurizr. The discipline comes from the abstractions and from giving every element and relationship a clear, complete label, not from a fixed visual language.

Q: Why does C4 stress that every element and every arrow must carry an explicit, complete label?
A: Because an unlabelled box or a bare arrow forces the reader to guess, and guessing is exactly what an architecture diagram should eliminate. C4 wants each element to say what it is and its core responsibility, and each relationship to say the intent plus the protocol or technology — for example "TA System sends settlement instructions to Custodian via SWIFT MT54x". A diagram a stranger can read without you in the room is the whole point.

Q: What is "diagrams-as-code", and what core benefit does it give over a drawing tool?
A: Diagrams-as-code means defining the model in a plain-text DSL or markup that a tool renders into the diagram, instead of dragging boxes in a GUI. The core benefit is that the model becomes a versioned text artifact you can diff, review in a pull request, and store next to the code it describes — so the architecture history is captured in Git like everything else. It also means the picture cannot silently drift from the agreed source of truth.

Q: Why is a committed `.dsl` or Mermaid file better than a committed `.png` export of the same diagram?
A: A `.png` is opaque to version control — a diff just shows "binary changed", you cannot see what moved, and a reviewer cannot comment on a specific line. A text model diffs meaningfully, so a reviewer sees exactly that an arrow's protocol changed from `SFTP` to `HTTPS`, and the change is reviewable in a PR. The export is a build output; the text is the source of truth.

Q: What is Structurizr DSL?
A: Structurizr DSL is a text-based domain-specific language for defining a software architecture model based on the C4 model, authored in plain `.dsl` files. You describe the model once — people, systems, containers, components, and their relationships — and then declare which views (context, container, component) to render from it. Because it separates the single model from multiple views, every diagram stays consistent with one source of truth.

Q: In Structurizr DSL, what are the three main top-level blocks inside a `workspace`?
A: Inside `workspace { ... }` the three main blocks are `model`, `views`, and (optionally) `configuration`/`styles`. The `model` block defines the elements and relationships; the `views` block defines which diagrams to generate from that model. Keeping model and views separate is what lets you draw several diagrams (context, container, component) from one definition without duplicating elements.

Q: How do you declare a person and a software system in Structurizr DSL?
A: You assign them inside the `model` block, e.g. `fundAccountant = person "Fund Accountant"` and `navSystem = softwareSystem "NAV Calculation System"`. The left-hand identifier is a variable you reference later when drawing relationships, and the quoted string is the display name. You can add an optional description as a second quoted argument.

Q: In Structurizr DSL, how do you express that one element uses another, and how is the protocol shown?
A: You use the `->` operator with a description and an optional technology, e.g. `navSystem -> custodian "Sends settlement instructions" "SWIFT MT54x"`. The first string is the intent shown on the arrow; the second is the technology/protocol. This is the diagrams-as-code equivalent of labelling every arrow, and it diffs cleanly in Git.

Q: In Structurizr DSL, how are containers nested under a software system?
A: You define them inside the software system's braces, e.g. `navSystem = softwareSystem "NAV" { api = container "NAV API" "REST" "Node.js"; db = container "NAV Store" "" "PostgreSQL" }`. The nesting tells the tool these containers belong to that system, so a container view of `navSystem` automatically shows just those. You then reference the container identifiers (`api`, `db`) in relationships.

Q: How do you declare a container view versus a system-context view in Structurizr DSL?
A: In the `views` block you write `systemContext navSystem "Context" { include *; autolayout }` for the level-1 view and `container navSystem "Containers" { include *; autolayout }` for level-2. Each view names the element it zooms into and which elements to include; `autolayout` lets the renderer position things. The same model thus produces multiple consistent diagrams.

Q: What is Structurizr Lite, and how is it typically run?
A: Structurizr Lite is the free, single-workspace version of Structurizr that renders and serves diagrams from a local `workspace.dsl`, typically run as a Docker container that watches the file and shows the diagrams in a browser. It is the recommended FOSS hands-on tool for this lesson because it keeps the model in your repo and re-renders on save. You point it at a directory containing the `.dsl` file and open `http://localhost:8080`.

Q: What is Mermaid's C4 support, and what is an important caveat about it?
A: Mermaid can render C4 diagrams using directives like `C4Context`, `C4Container`, `C4Component`, `C4Dynamic`, and `C4Deployment`, with syntax modelled on C4-PlantUML. The important caveat is that Mermaid's C4 support is officially experimental — the syntax and behaviour may change in future releases and documentation is provisional until it stabilises. It is attractive because Mermaid renders directly in GitHub/GitLab markdown, but you should pin your tool version.

Q: Why might you choose Mermaid C4 over Structurizr for a small repo, and what do you give up?
A: Mermaid renders inline in GitHub and GitLab markdown with no extra tooling, so a diagram lives right in the README and reviewers see it in the PR without running anything. What you give up is Structurizr's single-model-many-views consistency: in Mermaid each diagram is written separately, so a context and a container diagram of the same system can silently drift out of sync. For one or two diagrams that is fine; for a large model it is not.

Q: In Mermaid C4, how do you declare a person and a system inside a `C4Context` block?
A: You use functions such as `Person(accountant, "Fund Accountant", "Calculates NAV")` and `System(nav, "NAV System", "Computes daily NAV")`. Relationships use `Rel(accountant, nav, "uses", "HTTPS")`. The syntax deliberately mirrors C4-PlantUML so existing C4-PlantUML knowledge transfers, but remember it is experimental in Mermaid.

Q: What is an Architecture Decision Record (ADR)?
A: An ADR is a short document that captures a single significant architectural decision together with its context and consequences, so that future readers understand not just what was decided but why and at what cost. It turns tacit "hallway memory" into a reviewable, auditable artifact. The format was popularised by Michael Nygard's 2011 blog post and is now a widespread lightweight practice.

Q: What are the core sections of Michael Nygard's original ADR template?
A: They are Title, Status, Context, Decision, and Consequences. Context describes the forces and the situation; Decision states the choice in active voice ("We will..."); Consequences records the resulting trade-offs, both good and bad. The deliberate brevity is the point — short enough that people actually write them.

Q: Why does an ADR's "Context" section matter as much as the "Decision" itself?
A: Because the context records the forces — constraints, requirements, regulatory pressures, what you knew at the time — that made the decision reasonable, which is exactly what a future reader needs to judge whether it still holds. A decision with no context is just an assertion; a decision with context can be re-evaluated honestly when circumstances change. For a regulated firm the context is also part of the audit trail.

Q: What goes in the "Consequences" section of an ADR, and why must the bad ones be included?
A: It records the outcomes of the decision — what becomes easier, what becomes harder, and what new constraints or risks are introduced — including the negative ones. The negatives must be there because a list of only upsides is marketing, not an honest decision record, and the trade-offs you accepted are precisely what a reviewer or auditor needs to see. Honest consequences also stop the same trade-off being re-litigated later in ignorance.

Q: Why are ADRs deliberately kept short?
A: Because length is the enemy of adoption — a one-page ADR gets written and read, while a ten-page design document does not. The goal is to capture the decision, its context, and its consequences with just enough detail to be useful, not to produce exhaustive documentation. Short ADRs accumulate into a navigable decision log over the life of a system.

Q: What three (or four) status values describe an ADR's place in its lifecycle?
A: The common statuses are Proposed, Accepted, and Superseded, with Deprecated and Rejected also used. Proposed means under review, Accepted means in force, Superseded means a later ADR replaced it, Rejected means it was considered and turned down, and Deprecated means no longer relevant but not directly replaced. The status makes the document's authority explicit at a glance.

Q: What is the cardinal rule about editing an ADR once it is Accepted?
A: You do not rewrite the substance of an accepted ADR — its decision, context, and consequences are immutable history. If the decision changes, you write a new ADR that supersedes the old one and mark the old one's status as Superseded, leaving its original text intact. This preserves the audit trail of why things were once done a certain way, which slide-decks and edited wikis destroy.

Q: Why does "never rewrite history, supersede it" matter especially in a regulated fund administrator?
A: Because IT governance and regulators expect an auditable trail showing what was decided, when, by whom, and what later changed it — not a single document that was quietly edited to match today's reality. Superseding leaves both the old and new decisions visible with their statuses, so an auditor asking "why was NAV history partitioned this way in 2024, and why did it change in 2026?" gets a complete answer. Silent edits would look like an attempt to hide the trail.

Q: How do you record that ADR-0007 replaces ADR-0003? What changes in each file?
A: You leave ADR-0003's body unchanged but set its status to "Superseded by ADR-0007", and you create ADR-0007 with a reference back to "Supersedes ADR-0003". The two-way link is what makes the trail navigable in both directions. ADR-0003 stays in the repo forever as accurate history of a past decision.

Q: What is MADR, and what does the acronym stand for in its current version?
A: MADR is a popular structured ADR template. As of version 4.0.0 (released 2024-09-17) the acronym stands for "Markdown Architectural Decision Records"; earlier versions expanded it as "Markdown Any Decision Records" before the project refocused on architectural decisions. It adds structure (decision drivers, considered options, pros and cons) on top of Nygard's leaner format.

Q: What are the main sections of the full MADR template?
A: They are an optional YAML front matter (status, date, deciders), Title, Context and Problem Statement, Decision Drivers (optional), Considered Options, Decision Outcome with justification, Consequences (positive and negative), an optional Confirmation, optional per-option Pros and Cons, and optional More Information. The "Considered Options" plus "Pros and Cons" sections are what distinguish MADR from Nygard's template. MADR also ships "minimal" and "bare" variants for lighter use.

Q: What does MADR's "Decision Drivers" section capture, and why is it useful to reviewers?
A: Decision Drivers list the criteria and forces that the chosen option had to satisfy — performance, cost, regulatory constraints, operability, vendor lock-in — that the options are then judged against. They are useful to reviewers because they make the evaluation traceable: you can check that the chosen option actually scores well on the drivers the team claimed to care about. Without them, "we chose X" looks arbitrary.

Q: What does the MADR front-matter status field typically look like for a superseded record?
A: It uses a value like `superseded by ADR-0012`, sitting alongside the other allowed values `proposed`, `rejected`, `accepted`, and `deprecated`. Encoding the superseding link in the status (and in front matter where tooling can parse it) keeps the lifecycle machine-readable. The original record's body is still left intact.

Q: What is the standard file-naming convention for MADR/ADR files?
A: Files are named `NNNN-title-with-dashes.md`, where `NNNN` is a zero-padded sequential number (e.g. `0001`, `0042`) and the title uses lowercase words separated by dashes. The padding to four digits keeps files sorting correctly up to 9,999 records. For example `0007-partition-nav-history-by-fund-and-month.md`.

Q: Why pad the ADR number to four digits (`0001`) rather than using `1`, `2`, `10`?
A: Because zero-padding makes the files sort correctly as plain strings in a directory listing or Git tree — `0002` sorts before `0010`, whereas `2` would sort after `10`. It is a small detail that keeps the decision log readable in the exact chronological order decisions were made. Four digits comfortably covers any realistic project.

Q: Where, by convention, do ADR files live in a repository?
A: The conventional location is a dedicated directory such as `doc/adr/` or `docs/adr/` at the repo root, holding the numbered markdown files. Keeping them in-repo (not in a separate wiki) means the decisions are versioned with the code, reviewed in the same PRs, and travel with a clone. This lesson asks you to set up that `adr/` convention once and reuse it in every capstone.

Q: What is `adr-tools`, and what does `adr new "..."` do?
A: `adr-tools` is a small command-line helper for managing Nygard-style ADRs; `adr new "Use Kafka for NAV events"` creates the next sequentially numbered markdown file from a template with the status pre-set. It also offers `adr new -s 3 "..."` to create a record that supersedes ADR-0003, automatically updating the old record's status. It is optional — many teams just copy a template by hand — but it removes the friction of numbering and superseding.

Q: How would `adr-tools` help you correctly supersede an existing record?
A: Running `adr new -s 7 "Replace SFTP ingestion with event streaming"` creates the new numbered ADR and automatically appends "Superseded by" to ADR-0007's status while linking the new one back to it. This automates the two-way link and the immutability convention so you cannot forget to update the old record. The result is a clean, navigable supersession trail.

Q: What is a lightweight RFC process, and how does it differ from an ADR?
A: An RFC (Request for Comments) is a short proposal document circulated for review and discussion before a decision is made, whereas an ADR records a decision after it has been made. The RFC is the deliberation; the ADR is the outcome. Lightweight RFC processes give a decision the review it needs without a heavyweight committee, and the resulting decision is then captured in an ADR.

Q: When does a decision need an RFC step before an ADR records it?
A: When the decision is significant, contentious, or affects multiple teams enough that it needs explicit review and buy-in before it is locked in — for example choosing the firm-wide market-data ingestion pattern. A small, local, reversible decision can go straight to an ADR; a large, cross-cutting, hard-to-reverse one benefits from an RFC round first. The RFC reduces the risk of an Accepted ADR being immediately challenged.

Q: How do the RFC and ADR artifacts fit together over a decision's life?
A: The RFC opens the question and gathers comments while the decision is still Proposed; once consensus forms, the decision is recorded as an Accepted ADR that may reference the RFC for the fuller discussion. The RFC captures the debate; the ADR captures the conclusion and its consequences. Some teams even keep the RFC as the ADR's "More Information" link.

Q: What does the phrase "the decisions you didn't write down" mean for a reviewer?
A: It refers to implicit architectural choices baked into a diagram or system that were never explicitly recorded — like why a datastore was chosen, why a boundary sits where it does, or why a protocol was selected. A good reviewer hunts for exactly these gaps, because undocumented decisions are the ones that bite later when nobody remembers the rationale. The fix is to surface them as new ADRs.

Q: What three things do experienced reviewers look for in an architecture submission?
A: They look for clear boundaries (is the system scope and each container's responsibility unambiguous?), correct data flows (does every arrow show the right direction, protocol, and intent?), and the decisions that were made but not written down (the missing ADRs). These map directly onto C4 diagrams plus ADRs. A submission that nails all three is reviewable and auditable.

Q: A reviewer says "this arrow has no protocol" on your container diagram. Why does that matter?
A: Because an unlabelled arrow hides a real decision and a real integration risk — whether two containers talk over `HTTPS`, `SFTP`, a message queue, or SWIFT changes security, reliability, and latency, and a reviewer cannot assess the design without it. C4's rule that every relationship carries intent plus technology exists precisely to force this out. The fix is to add the protocol, e.g. "publishes NAV events via Kafka".

Q: Why are C4 diagrams and ADRs described as "the artifacts that define the architect role in practice"?
A: Because they are what make an architect's reasoning reviewable and decisions auditable, rather than living in slide decks and personal memory. The C4 diagrams make the structure legible; the ADRs make the choices and their trade-offs explicit and traceable. In a regulated fund administrator, producing these is much of what distinguishes architecture work from ad-hoc design.

Q: Your NAV platform's architecture lives only in a `.pptx` slide deck. What is the concrete risk?
A: The risk is that the slides drift from reality, cannot be diffed or reviewed in a PR, and carry no record of why decisions were made — so when an auditor asks why NAV history is partitioned a certain way, the only answer is "ask the person who built it". Slides are an export, not a source of truth, and they lose the decision trail entirely. Moving to in-repo C4-as-code plus ADRs fixes all three problems.

Q: For a transfer-agency order flow from EMT file to register, which C4 level best shows the runnable pieces and protocols?
A: The Container diagram (level 2). It would show the SFTP ingestion service, the EMT/EPT parser, the order-validation service, the message broker, and the register database as separate containers, with each arrow labelled with its protocol (e.g. "reads EMT file via SFTP", "publishes validated orders via Kafka"). Context (level 1) would hide these pieces, and Component (level 3) would only be needed inside one especially complex container.

Q: At the Context level, who are the likely external actors and systems for a fund's NAV-calculation system?
A: External actors include the fund accountant and oversight/compliance users; external systems include the order-management/transfer-agency system, the market-data provider, the custodian, the accounting/general-ledger system, and the regulatory-reporting gateway. The context diagram shows the NAV system as one box among these, with labelled relationships, and nothing about its internals. It answers "what is this system and who does it touch?".

Q: Write a one-line ADR title (Nygard style) for the decision to partition NAV history by fund and month.
A: A good title is short, neutral, and states the decision, e.g. "Partition NAV history by fund and calendar month". The corresponding file would be `0007-partition-nav-history-by-fund-and-month.md`. The Context would record the query and retention forces, the Decision the partitioning scheme, and the Consequences the trade-offs such as cross-fund query cost.

Q: A new requirement makes the monthly NAV partitioning scheme wrong. What is the correct ADR action?
A: You write a new ADR proposing the replacement scheme, take it through review, mark it Accepted, and set it to "Supersedes ADR-0007"; then you update ADR-0007's status to "Superseded by ADR-00NN" without altering its body. You never edit the original decision's substance. This leaves a clean, auditable trail of both the old rationale and the new one.

Q: In C4, what is the difference between a "system" and a "container", using a NAV platform as the example?
A: The NAV platform as a whole is one software system at the Context level — the single box stakeholders care about. Its containers are the separately runnable parts inside it: the NAV API, the calculation worker, the PostgreSQL store, the scheduler. Context answers "what is the platform among its neighbours"; Container answers "what does the platform consist of internally".

Q: Why can a single Apache Tomcat instance host several C4 containers in dev but separate them in production?
A: Because C4 containers are logical runtime/data boundaries, not deployment units — in development you might run several web applications inside one Tomcat for convenience, while in production each runs in its own process or node. The Container diagram models the logical separation; the Deployment diagram models the physical placement. This is exactly why C4 keeps "container" decoupled from infrastructure.

Q: What is the relationship between C4's Code level and a UML class diagram?
A: A C4 Code diagram is typically a UML class diagram (or ERD) describing the internals of one component. C4 does not invent a new notation for level 4 — it reuses UML and recommends auto-generating it from the source when actually needed, rather than maintaining it by hand. This is why most teams treat the code itself as the level-4 artifact.

Q: Why should the architecture model and the ADRs live in the same Git repository as the code where possible?
A: Co-locating them means the diagrams-as-code and decision records are versioned, reviewed, and updated in the same pull requests as the code they describe, so they stay in sync and travel with a clone. A reviewer can see a code change and the architecture/ADR change together. Splitting them into a separate wiki is how documentation rots.

Q: A teammate edited the body of an Accepted ADR to reflect a new decision. Why is this wrong, and what should they have done?
A: It is wrong because it destroys the audit trail — the historical record of what was decided and why is overwritten, so nobody can reconstruct the original rationale or see that a change occurred. They should have written a new ADR superseding the old one and set the old one's status to Superseded, leaving its body untouched. The "never rewrite history, supersede it" rule exists precisely to prevent this.

Q: You inherit a system with C4 diagrams but no ADRs. What is the first reviewer instinct?
A: To look for the decisions the diagrams imply but never document — why this datastore, why this boundary, why this protocol — because those are "the decisions you didn't write down" and the ones most likely to cause pain later. You would then write retroactive ADRs capturing the rationale you can reconstruct, so future readers are not left guessing. Diagrams show structure; ADRs explain choice.

Q: What does "Status: Proposed" on an ADR tell a reader about its authority?
A: It tells the reader the decision is still under review and not yet in force, so it should not be relied upon as the current design. Only once it moves to Accepted does it carry authority. Making the status explicit prevents teams from acting on a decision that was merely floated.

Q: Why is the active-voice form "We will use Kafka for NAV events" preferred in an ADR's Decision section?
A: Active voice states the decision unambiguously as a commitment the team is making, which is clearer and more accountable than passive hedging like "Kafka could be considered". It removes doubt about whether a decision was actually taken. Clarity here is what lets a reader, or an auditor, treat the ADR as a real record rather than a musing.

Q: How does diagrams-as-code support code review of architecture changes specifically?
A: Because the model is text, a change to it shows up as a normal diff in a pull request — a reviewer sees that a new container was added or an arrow's protocol changed, and can comment inline on the exact line. This brings architecture under the same review discipline as code, instead of being a separate, unreviewed drawing. The diagram render is just the build output of the reviewed text.

Q: In a regulated environment, why is an auditable ADR trail more valuable than a single up-to-date design document?
A: Because auditors and regulators (under regimes like DORA) want to see what was decided, when, by whom, and how it changed over time — a trail — not just the current state. A single living document shows only "now" and erases the history of "why then". Numbered, superseding ADRs provide exactly the chronological, immutable evidence governance expects.

Q: What is the risk of drawing all four C4 levels for every system?
A: The main risk is maintenance cost and drift: component and code diagrams change with every refactor, so they go stale fast and then actively mislead readers who trust them. Effort spent maintaining low-value diagrams is effort not spent on the context and container diagrams that actually stay useful. The discipline is to draw only the levels that earn their keep.

Q: How would you decide whether a particular architectural choice deserves its own ADR?
A: Ask whether the choice is significant (affects structure, cost, security, or compliance), whether it is hard to reverse, and whether a future reader would benefit from knowing why it was made. If yes to those, it warrants an ADR; trivial, local, easily reversed choices do not. The aim is a useful decision log, not a record of every line of code.

Q: A `C4Container` diagram in Mermaid renders fine today but breaks after a Mermaid upgrade. What is the likely cause?
A: The likely cause is that Mermaid's C4 support is officially experimental, so its syntax and rendering can change between releases without backward-compatibility guarantees. The fix is to pin the Mermaid version your pipeline uses and treat an upgrade as a change to test, not assume. This is a known trade-off of choosing Mermaid C4 for its in-markdown convenience.

Q: Your Structurizr Lite container starts but shows no diagrams. What is the first thing to check?
A: Check that you pointed Structurizr Lite at the directory actually containing your `workspace.dsl` and that the file parses — a DSL syntax error or a misdirected volume mount is the usual cause of an empty workspace. Structurizr Lite watches that one file and re-renders on save, so confirming the mount and a valid model comes first. Then confirm you are looking at `http://localhost:8080`.

Q: In Structurizr DSL, why does defining relationships once in the model save you work across views?
A: Because the model holds elements and relationships in one place, and each view simply selects which of them to show — so a relationship you define once appears correctly in the context, container, and any component view that includes its endpoints. You never redraw the same arrow per diagram. This single-model-many-views design is what keeps the diagrams mutually consistent.

Q: What is the difference between a C4 Dynamic diagram and a Container diagram?
A: A Container diagram is static — it shows what the containers are and which can talk to which. A Dynamic diagram shows the ordered sequence of interactions for one specific scenario, with numbered steps, like "1. order received, 2. validated, 3. NAV recalculated". Use the Dynamic diagram when the runtime flow of a particular use case is what you need to explain.

Q: For an order lifecycle from EMT ingestion to register update, which supplementary C4 diagram best shows the step order?
A: The Dynamic diagram, because it numbers the interactions in the order they occur — SFTP receive, parse EMT, validate order, publish event, update register — for that one scenario. The Container diagram shows the same pieces but not the sequence. Note that step-by-step business-process detail might also be expressed in BPMN, but within C4 the Dynamic diagram is the right tool.

Q: Why is "boundaries" the first thing a reviewer checks on a context or container diagram?
A: Because if the system boundary or each container's responsibility is unclear, nothing else on the diagram can be trusted — you cannot judge data flows or decisions when you do not know what is in scope. Clear boundaries are the foundation the rest of the review stands on. A box whose responsibility you cannot state in one sentence is a red flag.

Q: How do C4 diagrams and ADRs complement each other rather than overlap?
A: C4 diagrams answer "what does the architecture look like" — the structure, boundaries, and flows; ADRs answer "why is it like that" — the decisions, drivers, and trade-offs behind the structure. A diagram without ADRs leaves the reasoning implicit; ADRs without diagrams leave the structure hard to picture. Together they make a design both legible and auditable.

Q: What does "Done when" expect you to be able to explain about C4 levels?
A: You should be able to explain when to stop at the container level — because context plus container answer most stakeholder and reviewer questions cheaply — and what would force a component diagram, namely one container being internally complex or contentious enough that its internal boundaries are themselves a decision. The skill is justifying the chosen zoom depth, not drawing everything. Stopping at the right level is itself an architectural judgement.

Q: What does it mean to "show an ADR pair with the status trail intact"?
A: It means presenting two ADRs where the later one is marked "Supersedes ADR-N" and the earlier one is marked "Superseded by ADR-M", with both bodies preserved unedited. The intact trail proves you modelled the lifecycle correctly — history was extended, not rewritten. This is one of the lesson's explicit completion criteria.

Q: Why write retroactive ADRs for a system you already understand, as this lesson asks?
A: Because it forces you to practise the format, the immutability rule, and especially the supersession lifecycle on a real system whose decisions you can reconstruct, without the pressure of a live debate. Writing two — including one that gets superseded — models the full lifecycle end to end. It also leaves the inherited system better documented than you found it.

Q: An ADR's Context says "we must comply with UCITS reporting deadlines". Why does recording such a driver matter later?
A: Because it ties the architectural choice to a concrete external constraint, so a future reader knows the decision was driven by a real deadline, not preference — and can re-evaluate it correctly if, say, the regulation changes. Without that recorded driver, someone might "simplify" the design and unknowingly break a compliance obligation. Drivers turn a decision into something safely re-examinable.

Q: What is the danger of an architecture diagram that is correct today but never updated?
A: A stale diagram is worse than no diagram, because readers trust it and then make decisions on a false picture of the system. Diagrams-as-code mitigates this by living in the repo and being updatable in the same PRs as code changes, but it still requires the discipline to update it. The cheapest diagram to keep current is the container-level one, which is part of why C4 stops there.

Q: In C4, can two containers in the same system communicate directly, and how is that shown?
A: Yes — direct container-to-container communication is shown as a labelled arrow between them on the container diagram, with the protocol and intent, e.g. "NAV API reads positions from Position Store via JDBC". C4 places no restriction on which containers may talk; the diagram simply records the actual relationships. Every such arrow should carry its technology so the integration is reviewable.

Q: How would you represent an external SWIFT messaging dependency in a C4 context diagram?
A: SWIFT (or the counterparty reached over SWIFT, such as a custodian) appears as an external system box outside your system boundary, with a labelled relationship like "sends settlement instructions via SWIFT MT54x". It is external because you do not own or deploy it. The label captures both the intent and the messaging standard so the dependency is explicit.

Q: Why might "what protocol does this arrow use" be a recurring review question across all your diagrams?
A: Because the protocol determines security, reliability, latency, and failure modes, and it is the single most commonly omitted label on architecture diagrams. Reviewers ask it repeatedly because its absence hides real risk — an unencrypted `FTP` versus `SFTP`, or a synchronous call versus an async queue, are very different designs. C4's rule to label every relationship with its technology is a direct response to this.

Q: What is the smallest viable ADR you could write, and is it still valuable?
A: The smallest viable ADR is essentially Nygard's five sections kept to a few sentences each: a title, status, a paragraph of context, a one-line decision, and a short consequences list. It is still highly valuable because it captures the why and the trade-off, which is what gets lost otherwise. Brevity is a feature — a tiny ADR that exists beats a perfect one that was never written.

Q: How does the ADR lifecycle status `Deprecated` differ from `Superseded`?
A: `Superseded` means a specific later ADR replaced this decision, so there is a forward link to the new record; `Deprecated` means the decision is no longer relevant or in use but is not directly replaced by a named successor. Both keep the original body intact as history. The distinction tells a reader whether to follow a link to a replacement or simply treat the decision as retired.

Q: Why is it useful for an ADR to record options that were considered and rejected (as MADR does)?
A: Because the rejected options and the reasons for rejecting them are exactly what stops a future team from re-proposing a path that was already evaluated and ruled out for good reasons. MADR's "Considered Options" plus pros-and-cons sections capture this. It saves rework and makes the chosen option's justification far more convincing than a bare assertion.

Q: A capstone submission has great C4 diagrams but reviewers still flag it. What are they most likely flagging?
A: They are most likely flagging the missing decisions — the ADRs explaining why the design is the way it is — and possibly unlabelled data flows. Beautiful diagrams still leave "why" implicit, and this lesson's whole point is that diagrams plus ADRs together are the deliverable. The fix is to add ADRs for the key choices the diagrams embody.

Q: What is the practical meaning of "ship C4 + ADRs with every capstone from here on without thinking about it"?
A: It means the habit becomes automatic: every system you design gets at least a context and container diagram as code in the repo, plus ADRs for its significant decisions, as a standard deliverable rather than an afterthought. The later capstones in the plan enforce this. The point is to internalise the practice now so it is muscle memory by the time the systems get large.

Q: Why does C4 deliberately avoid prescribing colours and shapes?
A: Because mandating a visual language would make the model harder to adopt and tie it to specific tools, whereas C4's value lies in its abstractions and the discipline of complete labelling, which work in any tool. Teams are free to add their own consistent styling (e.g. a colour per technology) on top. The model is about clear thinking, not a fixed icon set.

Q: How does the `model` versus `views` split in Structurizr DSL embody the C4 idea of "one model, many diagrams"?
A: The `model` block is the single source of truth for all elements and relationships, while each entry in the `views` block is just a filtered, zoomed projection of that model into a context, container, or component diagram. Because every view draws from the same model, the diagrams cannot contradict each other. This is the structural reason Structurizr keeps multi-level C4 diagrams consistent.

Q: When narrating an unfamiliar C4 container diagram cold, what should you read off first?
A: First the system boundary and what each container is and is responsible for, then each relationship's direction, intent, and protocol, and finally what external systems or actors it touches. Reading in that order — boundaries, then flows — mirrors exactly what a reviewer checks. If you cannot state a container's responsibility in a sentence, that is the first gap to call out.

Q: Why is "data flows" a distinct review concern from "boundaries"?
A: Boundaries are about what the pieces are and where their responsibilities start and stop; data flows are about how data and control actually move between them — direction, protocol, and intent on each arrow. A diagram can have crisp boundaries but wrong or missing flows, and vice versa. Reviewers check both because a design needs the boxes and the arrows to be right.

Q: How would an ADR help when an auditor asks "why is NAV history partitioned this way?"
A: The relevant ADR directly answers it: its Context records the query and retention forces, its Decision states the partitioning scheme, and its Consequences record the trade-offs accepted — all dated and attributable. If the scheme later changed, the superseding ADR shows when and why. This is the difference between an auditable answer and "the person who built it would know".

Q: What is the downside of using a heavyweight RFC process for every small decision?
A: It creates so much review friction that people stop recording decisions at all, or route around the process, which defeats the purpose. RFCs are for significant, contentious, or cross-cutting decisions; small reversible ones should go straight to a lightweight ADR. Matching the process weight to the decision's stakes is the skill.

Q: How can an RFC reduce the chance that an Accepted ADR is immediately challenged?
A: By surfacing objections and alternatives during the RFC review phase, before the decision is locked in, so that by the time it becomes an Accepted ADR the major concerns have already been aired and addressed. The ADR then records a decision that already has buy-in. Skipping the RFC on a contentious decision often means re-opening it as an ADR the week after it's accepted.

Q: Why store the `adr/` directory convention once and reuse it across all capstones?
A: Reusing one convention — location, naming (`NNNN-title.md`), template, and lifecycle rules — means every capstone's decision log looks the same and is instantly navigable, and you stop re-deciding format each time. Consistency is what makes the accumulated body of ADRs across the plan usable as one corpus. It also models the standardisation a real firm's governance imposes.

Q: What single sentence captures why this lesson sits at the start of the architecture phase?
A: Because C4 diagrams and ADRs are the artifacts through which an architect makes reasoning reviewable and decisions auditable, and every later capstone is expected to ship them — so you build the habit before the systems get complex. They convert design from hallway memory into versioned, reviewable, regulator-ready evidence. Master them now and they become invisible infrastructure for everything that follows.

Q: Why does C4 describe itself as "abstraction-first, notation-second"?
A: Because the value of the model is the shared set of abstractions — person, software system, container, component — and the disciplined zoom levels, not the boxes-and-arrows style you render them in. If everyone agrees what a "container" means, any tool or notation will communicate; if they don't, even a beautiful diagram misleads. So C4 invests in getting the abstractions and labels right and treats the visual notation as a free choice on top.

Q: What is the practical difference between linking an RFC from an ADR versus pasting the whole discussion into the ADR?
A: Linking keeps the ADR short and decision-focused while preserving the full deliberation for anyone who wants it, which fits the "short enough that people actually write them" principle; MADR even has a "More Information" section for exactly such links. Pasting the entire thread bloats the record so it stops being read. The rule of thumb is: the ADR records the decision and its consequences, and points to the RFC for the long-form debate.

Q: Write the exact `docker run` command to start Structurizr Lite against a repo directory on your Ubuntu laptop.
A: Run `docker run -it --rm -p 8080:8080 -v /home/you/repo/architecture:/usr/local/structurizr structurizr/lite`, then open `http://localhost:8080`. The `-v` host-path-to-`/usr/local/structurizr` mount is the data directory Structurizr Lite watches, so your `workspace.dsl` must sit directly inside the host folder you mount. The `--rm` flag discards the container on exit, which is fine because all state lives in the mounted directory, not the container.

Q: Your `workspace.dsl` is in a sub-folder `nav-system/` under the mounted directory — how do you point Structurizr Lite at it?
A: Set the `STRUCTURIZR_WORKSPACE_PATH` environment variable, e.g. add `-e STRUCTURIZR_WORKSPACE_PATH=nav-system` to the `docker run` command so it looks in `/usr/local/structurizr/nav-system` for the workspace. By default Structurizr Lite expects `workspace.dsl` at the root of the mounted data directory, so without this it would create a fresh empty one there. There is also `STRUCTURIZR_WORKSPACE_FILENAME` if your file is not named `workspace.dsl`.

Q: Structurizr Lite serves the page but says it cannot find your model and shows a starter workspace instead. What likely went wrong with the volume mount?
A: The host path in `-v` almost certainly does not contain `workspace.dsl` at its root — either you mounted the repo root instead of the `architecture/` sub-folder, or you forgot `STRUCTURIZR_WORKSPACE_PATH`. On finding no workspace file, Structurizr Lite generates a default starter `workspace.dsl` in the data directory, which is the empty model you are seeing. Stop the container, fix the mount so the file is visible, and restart.

Q: After editing `workspace.dsl` while Structurizr Lite is running, the browser still shows the old diagram. What is the fix?
A: Just reload the browser page — Structurizr Lite re-reads the file on each page load rather than pushing live updates, so a manual refresh (`F5`) picks up your saved changes. Confirm the file was actually saved to the mounted host path and that there is no DSL parse error, which would otherwise show as an error banner instead of the new diagram. There is no separate rebuild step; the watch-and-refresh loop is the whole workflow.

Q: In Structurizr DSL, how do you embed ADR markdown files alongside the model so they render in the same workspace?
A: Use the `!adrs` keyword inside a software system or container block, e.g. `!adrs adrs` pointing at a directory of numbered markdown ADRs, and `!docs docs` for general documentation. Structurizr then renders the ADR set and the documentation tabs next to the diagrams, keeping decisions and structure in one navigable workspace. This is the diagrams-as-code way to satisfy this lesson's "C4 + ADRs together" deliverable from a single source.

Q: In Structurizr DSL, how do you apply a tag to a container and then style every element with that tag?
A: Add `tags "Database"` (or pass it as a later positional argument) on the element, then in the `views` block use `styles { element "Database" { shape Cylinder; background #1168bd } }` to style everything carrying that tag. Tags decouple styling from individual elements, so all your datastores get the cylinder shape from one rule. This is how you add a consistent visual convention (e.g. colour per technology) on top of C4's deliberately unstyled abstractions.

Q: How do you give a Structurizr DSL element a `shape Cylinder` to mark it visually as a datastore?
A: You do not set the shape on the element directly; you tag it (e.g. `db = container "NAV Store" "" "PostgreSQL" "Database"`) and then add `element "Database" { shape Cylinder }` in the `styles` block of the `views`. Structurizr separates the model from presentation, so shape, colour, and icon live in styles keyed by tag, not on the element line. Common shapes include `RoundedBox`, `Cylinder`, `Person`, and `Pipe`.

Q: In Structurizr DSL, what does an `!include` directive let you do, and why is that useful for a large fund-platform model?
A: `!include <file-or-url>` pulls the contents of another DSL fragment into the current workspace at that point, so you can split one big model into per-system or per-team files and assemble them. For a firm-wide landscape spanning NAV, transfer agency, and reporting, each team owns its own `.dsl` fragment and a top-level file includes them all. It keeps the single-model-many-views benefit while avoiding one unmanageable monolith file.

Q: In Mermaid C4, how do you draw the system boundary box around your in-scope system in a `C4Context` diagram?
A: Wrap the in-scope elements in a `System_Boundary(alias, "label") { ... }` block, or `Enterprise_Boundary(alias, "label") { ... }` for an organisation-wide boundary; container diagrams use `Container_Boundary`. Everything outside the boundary block is an external person or system. Remember Mermaid C4 is experimental, so test boundary rendering after any Mermaid upgrade.

Q: In Mermaid C4, which element function marks an external system as a database, and how does it differ from `System_Ext`?
A: `SystemDb_Ext(alias, "label", "description")` renders an external system styled as a database (a cylinder), whereas `System_Ext` renders a generic external system box. There are matching internal variants `SystemDb` and `System`, plus `SystemQueue`/`SystemQueue_Ext` for message-queue styling. Picking the right variant communicates the dependency's nature, e.g. an external market-data store versus an external pricing service.

Q: In Mermaid C4, how do you restyle a specific relationship's colour or line after declaring it with `Rel`?
A: Use `UpdateRelStyle(fromAlias, toAlias, $textColor="...", $lineColor="...", $offsetX="...", $offsetY="...")` after the `Rel` line referencing the same two endpoints. There is also `UpdateElementStyle` for elements and `UpdateLayoutConfig($c4ShapeInRow="...", $c4BoundaryInRow="...")` to influence how many shapes pack per row. These tweaks are how you nudge Mermaid's auto-layout, since it has no manual coordinate control.

Q: In Mermaid C4, how do you express a bidirectional relationship between two elements?
A: Use `BiRel(aliasA, aliasB, "label", "technology")`, which draws a double-headed arrow, instead of two separate `Rel` calls. There are also directional hint variants `Rel_U`, `Rel_D`, `Rel_L`, and `Rel_R` (up/down/left/right) to nudge layout. Use `BiRel` when the two containers genuinely call each other, e.g. a request/response pair, but prefer two labelled `Rel` arrows when the intents differ in each direction.

Q: Initialise the ADR directory convention with `adr-tools` on a fresh repo — what command, and what does it create?
A: Run `adr init doc/adr`, which creates the `doc/adr/` directory and writes ADR-0001 recording the decision to use ADRs themselves (the bootstrap record). Omitting the path defaults to `doc/adr`. From then on the directory and that first meta-ADR anchor the convention you reuse across capstones.

Q: With `adr-tools`, how do you link two existing records without superseding — e.g. record that ADR-0012 amends ADR-0010?
A: Run `adr link 12 Amends 10 "Amended by"`, which adds an "Amends ADR-0010" link to record 12 and an "Amended by ADR-0012" back-link to record 10. The general form is `adr link SOURCE LINK TARGET REVERSE-LINK`, so you control the wording of both directions. Unlike supersession, amendment keeps both records in force; it just cross-references them.

Q: How do you generate a Markdown table of contents for your ADR log with `adr-tools`?
A: Run `adr generate toc`, which prints a Markdown table of contents of all records to stdout; redirect it to a file like `adr generate toc > doc/adr/README.md`. Flags `-i INTRO` and `-o OUTRO` add prose before and after the list, and `-p LINK_PREFIX` prefixes each link (useful when the TOC is rendered from a different path). Regenerating it keeps the index in sync as records are added.

Q: How do you visualise the supersession/amendment links between your ADRs using `adr-tools`?
A: Run `adr generate graph`, which emits a Graphviz `dot` description of the links between records to stdout; pipe it to Graphviz, e.g. `adr generate graph | dot -Tpng -o adr-graph.png`. This turns the cross-references into a visual map of how decisions evolved and superseded one another. It is a quick way to audit that your supersession trail is complete and two-way.

Q: On Ubuntu, how would you install `adr-tools`, and what is the one runtime dependency to keep in mind?
A: `adr-tools` is a set of POSIX shell scripts; you install it by cloning the repo or unpacking a release and putting its `src` (or the `adr` binary) on your `PATH` — there is no `apt` package in older Ubuntu, though `snap install adr` exists for some builds. The runtime dependency is a POSIX `sh` plus standard tools like `date`; on Ubuntu that is already present. For the graph feature you also need Graphviz (`apt install graphviz`).

Q: Why does `adr-tools` store the ADR directory location in a `.adr-dir` file, and what breaks if it is missing?
A: After `adr init doc/adr`, the tool writes `.adr-dir` at the repo root containing the path `doc/adr`, so subsequent commands like `adr new` and `adr list` know where records live without re-specifying the path. If `.adr-dir` is missing or you run the command outside the repo, the tool falls back to the default `doc/adr` and may create records in the wrong place. Committing `.adr-dir` keeps the whole team pointed at the same directory.

Q: In MADR 4.x, give the YAML front-matter keys and a valid `status` value for a record that has been replaced.
A: The front matter sits between `---` fences and uses keys like `status`, `date`, `decision-makers`, and `consulted`/`informed`; a replaced record uses a status such as `superseded by [ADR-0012](0012-...)`. Keeping the link in the status string makes the lifecycle both human- and tool-readable. The allowed plain statuses are `proposed`, `rejected`, `accepted`, and `deprecated`.

Q: A teammate proposes using MADR's "minimal" variant. What does it drop compared with the full template, and when is that the right call?
A: The minimal MADR keeps only the essentials — title, context and problem statement, considered options, and decision outcome — dropping decision drivers, per-option pros/cons, confirmation, and more-information sections. It is the right call for a small, low-controversy decision where the full apparatus would be overkill and discourage anyone from writing it at all. You can always start minimal and a later, weightier decision can use the full template.

Q: What is the difference between MADR's "Confirmation" section and its "Consequences" section?
A: "Consequences" records the trade-offs that result from the decision (what gets easier/harder), while "Confirmation" describes how compliance with the decision will be verified — e.g. a code review check, an architecture-conformance test, or a lint rule. Confirmation answers "how will we know this decision is actually being followed?", which matters in a regulated firm where conformance is auditable. It is optional but valuable for decisions that are easy to silently violate.

Q: You want machine-readable ADR statuses so a script can flag every `accepted` record touching NAV. How does the ADR format help, and what is the gotcha?
A: Putting the status in YAML front matter (as MADR does) makes it parseable by a script or tooling, so you can grep/parse `status: accepted` reliably; Nygard's plain "Status: Accepted" line is less structured but still greppable. The gotcha is free-text supersession statuses like `superseded by ADR-0012`, which vary in wording across authors and break naive equality checks. Agreeing a fixed status vocabulary up front is what keeps the log machine-readable.

Q: A capstone reviewer says "your context diagram has eight containers in it." What rule did you break, and how do you fix it?
A: You mixed levels — containers belong on a level-2 Container diagram, not the level-1 System Context diagram, which should show your system as a single box among external actors and systems. The fix is to collapse those eight containers back into one system box on the context diagram and move the internal detail to a separate container diagram. Keeping each level at its own zoom is what makes the set readable.

Q: A relationship in your container diagram is labelled only "uses". What concrete improvement does a reviewer expect?
A: Replace the vague "uses" with the actual intent plus the technology/protocol, e.g. "reads validated orders via Kafka" or "queries positions via JDBC". C4's rule is that every relationship states what flows and over what, because "uses" hides direction, payload, and the integration risk. A stranger reading the diagram should learn the real dependency, not just that two boxes are connected.

Q: Two of your containers both talk to the same PostgreSQL store. Is sharing one database across containers a C4 modelling problem or an architecture smell, and how do you record the concern?
A: C4 has no rule against it — you simply draw two labelled arrows into the one datastore container — but a shared database across service boundaries is often an architecture smell (coupling, unclear ownership) worth flagging. The right place to capture the concern and the trade-off you accepted is an ADR, not the diagram. So the diagram shows the fact; the ADR explains and justifies (or schedules the unwinding of) the shared store.

Q: When would you reach for a C4 System Landscape diagram instead of a System Context diagram in a fund administrator's estate?
A: Use a System Landscape when there is no single system in scope and you want the enterprise-wide picture — order management, transfer agency, fund accounting, market data, regulatory reporting, and their interconnections — for, say, an enterprise architect spotting duplication. A System Context, by contrast, always centres one system as the box in scope. The landscape answers "what does our whole estate look like?"; the context answers "what does this one system touch?".

Q: For documenting the runtime sequence of an end-of-day NAV strike, which C4 diagram fits, and what is its non-C4 alternative?
A: A C4 Dynamic diagram fits, because it numbers the interactions in order — receive prices, value positions, apply fees, strike NAV, publish to GL and reporting — for that one scenario. Its non-C4 alternatives are a UML sequence diagram or, for the business-process angle, BPMN. Within a C4 model the Dynamic diagram is the right tool; BPMN is better when the audience is business analysts modelling the process itself.

Q: You need to show that NAV data must stay in an EU region for two replicas while the calc worker autoscales. Which C4 diagram, and which elements?
A: A Deployment diagram, because residency and instance counts are deployment concerns, not logical structure. You model deployment nodes (e.g. an EU-West region, Kubernetes cluster, VMs) and place container instances inside them, annotating the datastore node with two replicas and the worker node with its autoscaling range. The Container diagram stays unchanged; the Deployment diagram adds the where and how-many.

Q: A reviewer asks why a particular box is inside your system boundary rather than outside it. Why is "is this internal or external?" such a load-bearing question?
A: Because the boundary decides what you own, deploy, and are accountable for versus what you merely depend on — and that changes the diagram, the responsibilities, and the compliance scope. Putting a vendor pricing service inside your boundary wrongly implies you build and run it; putting your own NAV store outside wrongly disclaims ownership. Getting the boundary right is the foundation every other review check stands on.

Q: Your Mermaid C4 diagram renders in the GitHub PR but the boxes overlap and arrows cross messily. What are your realistic options?
A: Mermaid C4 gives only coarse layout control, so options are: add directional `Rel_U`/`Rel_D`/`Rel_L`/`Rel_R` hints, use `UpdateLayoutConfig` to set shapes-per-row, reduce the number of elements per diagram, or accept the limitation. If clean layout matters a lot, switch that diagram to Structurizr, which has stronger auto-layout and manual positioning. The messy layout is a known cost of choosing Mermaid for its in-markdown convenience.

Q: Compare Structurizr DSL and Mermaid C4 on the single dimension of model consistency. Which guarantees it and how?
A: Structurizr guarantees consistency because all diagrams are views projected from one shared model — define an element or relationship once and every view that includes it shows it identically. Mermaid C4 has no shared model: each diagram is authored independently, so a context and a container diagram of the same system can silently disagree. For more than a couple of diagrams, Structurizr's single-model-many-views design is the deciding advantage.

Q: How does a corporate EA suite like Sparx Enterprise Architect differ from Structurizr Lite for this lesson's purpose?
A: Sparx EA (and Visual Paradigm) are full model-repository suites with traceability, governance, multi-notation support (UML, BPMN, ArchiMate, ERD) and access control — the kind EU institutions license — whereas Structurizr Lite is a lightweight FOSS, single-workspace, C4-focused diagrams-as-code tool. For learning C4-in-the-repo and PR-reviewable models, Structurizr Lite is the hands-on choice; the EA suites are what you evaluate for firm-wide, audited model governance. The lesson maps Structurizr Lite as the FOSS counterpart to Sparx EA.

Q: A reviewer reads your ADR and asks "what did you NOT choose, and why?" Which template answers this directly, and in which section?
A: MADR answers it directly in its "Considered Options" section plus the per-option pros-and-cons, which list the alternatives and the reasons each was or was not chosen. Nygard's leaner template has no dedicated options section, so rejected alternatives often live (if at all) inside the Context. Recording the rejected options is what stops a future team re-proposing a path already ruled out.

Q: You are asked to retro-fit ADRs onto a five-year-old NAV system with no records. What is a sane scope so you do not write a hundred ADRs?
A: Write ADRs only for the significant, hard-to-reverse, or compliance-relevant decisions you can reconstruct — the datastore choice, the partitioning scheme, the ingestion protocol, key boundary decisions — not for every line of code. Mark them honestly (e.g. note they are retroactive in the Context) and link any later changes via supersession. The goal is a useful, navigable decision log, not exhaustive archaeology.

Q: In a retroactive ADR, how should you handle the fact that you are reconstructing rationale you were not present for?
A: State plainly in the Context that the record is retroactive and that the rationale is reconstructed from the system's current state and available evidence, rather than asserting certainty about the original intent. This keeps the record honest and auditable — a reader knows its provenance and confidence level. It is far better than presenting a guess as documented historical fact.

Q: What is "architecture conformance" and how do an ADR's Decision plus MADR's Confirmation support checking it?
A: Architecture conformance is verifying that the built system actually matches the decided architecture, not just the diagram. An ADR's Decision states the intended rule (e.g. "all NAV events flow through Kafka"), and MADR's optional Confirmation section says how conformance is checked — an automated test, a fitness function, or a review step. Together they turn a decision into something you can continuously verify rather than hope for.

Q: A decision is reversible and affects only your team's internal service. RFC, ADR, both, or neither?
A: An ADR alone — record it so the rationale is not lost, but skip the RFC, since a small, local, reversible decision does not need cross-team review before being made. RFCs are reserved for significant, contentious, or cross-cutting decisions where buy-in must precede the commitment. Matching process weight to decision stakes is the judgement the lesson is teaching.

Q: A firm-wide market-data ingestion pattern affecting four teams is contested. Sketch the RFC-to-ADR flow.
A: Open an RFC document describing the problem and options, circulate it to the four teams, and gather comments while the decision stays Proposed; once consensus forms, record the conclusion as an Accepted ADR that links back to the RFC for the full debate. The RFC absorbs the disagreement before the decision is locked, reducing the chance the Accepted ADR is immediately re-litigated. The ADR stays short; the RFC holds the long-form discussion.

Q: Why is committing a `.png` export of a diagram into the repo specifically bad for code review, beyond just file size?
A: Because Git treats the `.png` as an opaque binary blob — the diff shows only "binary file changed", so a reviewer cannot see that an arrow's protocol went from `SFTP` to `HTTPS`, cannot comment on a specific change, and cannot trust that the picture matches the agreed text. The image is a build output, not a source; the reviewable artifact is the `.dsl` or Mermaid text that produced it. Diagrams-as-code exists precisely to bring architecture under normal diff-and-review discipline.

Q: How do you keep a rendered diagram image current in a README without committing it manually, in a diagrams-as-code setup?
A: Generate it in CI from the committed text source — e.g. run the Structurizr CLI or Mermaid CLI in a pipeline to export the `.png`/`.svg` from the `.dsl`/`.mmd` on each change — so the image is a reproducible build artifact, not a hand-maintained file. The text remains the single source of truth, and the image cannot drift because it is regenerated. Alternatively, with Mermaid in GitHub markdown you skip the image entirely since it renders from the fenced code block.

Q: In `C4Component` Mermaid syntax, what does `Component(alias, "label", "technology", "description")` represent, and what must it sit inside?
A: It represents a single C4 component — a grouping of related functionality behind an interface inside one container — with its name, implementing technology, and a short description. It should sit inside a `Container_Boundary(alias, "label") { ... }` so the diagram shows it belongs to that container. As with all Mermaid C4, treat the syntax as experimental and pin your Mermaid version.

Q: Your component diagram for the NAV engine has grown to 25 components and is unreadable. What does this tell you, and what are two fixes?
A: Twenty-five components on one diagram usually means either the container genuinely should be split into multiple containers, or the diagram is showing code-level detail that belongs at level 4 (the source itself). Two fixes: decompose the container so each resulting container has a tractable component diagram, or raise the abstraction so only the major building blocks appear and the fine-grained classes drop to code level. An unreadable diagram is feedback about the boundaries, not just the drawing.

Q: When narrating an unfamiliar Structurizr workspace cold, what should you open first — the `model` block or the `views` block — and why?
A: Open the `model` block first, because it is the single source of truth defining every person, system, container, and relationship; the `views` block only selects filtered projections of that model. Reading the model tells you what exists and how it connects; reading views then tells you which slices the authors chose to show. Starting with views alone risks mistaking one diagram's filtered subset for the whole system.

Q: Give a one-line Structurizr DSL relationship for a transfer-agency system sending settlement instructions to a custodian over SWIFT.
A: `taSystem -> custodian "Sends settlement instructions" "SWIFT MT54x"` — the first string is the intent on the arrow, the second is the technology/protocol. This is the diagrams-as-code form of C4's "label every relationship with intent plus technology" rule, and it diffs cleanly in a PR. The custodian would be declared as an external `softwareSystem` outside your system boundary.

Q: How would you model an SFTP-based EMT/EPT file ingestion in a Structurizr container diagram so the protocol is explicit?
A: Declare an ingestion container, e.g. `ingest = container "EMT/EPT Ingestion" "Polls and parses files" "Python"`, and label the inbound relationship with the protocol, e.g. `dataProvider -> ingest "Delivers EMT/EPT files" "SFTP"`. The protocol on the arrow tells a reviewer this is file-based and over SFTP, which carries different reliability and security properties than an API call. The parsed-output relationship to the next container would carry its own protocol label, e.g. Kafka.

Q: Your team debates whether the market-data feed is one external system or several on the context diagram. How do you decide, and where does the rationale go?
A: Decide by ownership and boundary: model it as one external system if you depend on a single provider/interface, or as several if they are genuinely distinct systems you integrate separately — the test is "is this one thing I depend on, or many?". The diagram records whichever is true; the reasoning for the chosen granularity, if non-obvious, goes in an ADR. This keeps the context diagram an honest map and the rationale auditable.

Q: A diagram is technically correct but a newcomer still cannot understand it. By C4's own standards, has it failed, and why?
A: Yes — C4's stated goal is a diagram a stranger can read without the author in the room, so correctness without comprehensibility is a failure of the model's purpose. Usually the cause is missing or vague labels, mixed abstraction levels, or too many elements per diagram. The remedy is C4's discipline: one zoom level per diagram, complete intent-plus-technology labels, and only the elements that earn their place.

Q: Summarise the end-state habit this lesson wants: what exactly ships with every capstone from here, and what does each artifact prove?
A: Every capstone ships at least a C4 Context and Container diagram as committed diagrams-as-code, plus ADRs for the significant decisions, including at least one supersession pair with the trail intact. The diagrams prove the structure, boundaries, and labelled data flows are legible; the ADRs prove the choices and trade-offs are recorded and auditable. Together they convert design from hallway memory into versioned, reviewable, regulator-ready evidence — the artifacts that define the architect role.

Q: How do you fence a Mermaid C4 diagram in a GitHub markdown file so it renders, and what is the first thing to check if it shows as raw text instead?
A: Wrap it in a fenced code block whose info string is `mermaid` (three backticks then `mermaid`), with a valid C4 directive like `C4Context` as the first line inside. If it renders as raw text, first check the fence language is exactly `mermaid` and that the platform actually supports Mermaid C4 — GitHub renders Mermaid but its C4 support is experimental, so a syntax error or an unsupported directive on your Mermaid version drops it to plain text. Validate the snippet in the Mermaid live editor pinned to the same version to isolate the cause.

Q: An ADR you are writing keeps growing past two pages with background detail. What does the format tell you to do, and where does the overflow go?
A: Keep the ADR itself short — title, status, context, decision, consequences — because length is the enemy of adoption, and move the long-form background out of the record. In MADR the overflow goes in the optional "More Information" section as links (to an RFC, a spike, a benchmark), not pasted inline. The discipline is that the ADR captures the decision and its trade-offs; supporting material is referenced, so the record stays short enough that people actually write and read it.
