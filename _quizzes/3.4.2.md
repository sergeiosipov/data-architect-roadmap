# Phase 2 · 3.4.2 Table formats (Iceberg, Delta) — 150 self-test questions

Q: What problem does an open table format like Apache Iceberg solve that plain Parquet files in object storage do not?
A: Bare Parquet files in a directory have no transactional guarantees, no schema-evolution rules, and no consistent snapshot of "which files belong to the table right now". Iceberg adds an ACID metadata layer on top of the data files so that commits are atomic, readers see a consistent set of files, and you get schema evolution, time travel, and partition evolution. This metadata layer is what turns a pile of files into a queryable, governable table — the heart of the lakehouse.

Q: Name the four levels of the Iceberg metadata tree from top to bottom.
A: From top to bottom: the table metadata file (the `metadata.json`), the manifest list, the manifest files, and finally the data files. The catalog holds a pointer to the current `metadata.json`; that file points to a manifest list per snapshot; the manifest list points to manifests; and manifests point to the actual data (and delete) files. Walking this tree is how every engine resolves "what files make up this table".

Q: What does the Iceberg catalog itself store, as distinct from the metadata file?
A: The catalog stores only a small pointer: the mapping from a table identifier (namespace plus name) to the location of the current `metadata.json` file. It does not store schema, snapshots, or file lists — those live inside the metadata file. This is why an atomic commit is just an atomic swap of that one pointer from the old metadata file to the new one.

Q: What information is recorded inside an Iceberg table's `metadata.json` file?
A: The `metadata.json` records the table's full current schema and the history of schemas, all partition specs and sort orders, table properties, the current snapshot ID, and the list of all snapshots with their pointers to manifest lists. It is the single document an engine reads first to understand the table's structure and history. Every commit writes a brand-new `metadata.json` rather than editing the old one in place.

Q: What is a manifest list in Iceberg and how many exist per snapshot?
A: A manifest list is an Avro file that enumerates all the manifest files that make up one snapshot — there is exactly one manifest list per snapshot. For each manifest it records summary statistics such as partition value ranges and added/deleted file counts, which lets an engine skip whole manifests during planning. The current snapshot's `manifest_list` field in `metadata.json` points to this file.

Q: What does a single Iceberg manifest file contain?
A: A manifest file is an Avro file that lists a subset of the table's data files (or delete files), one row per file, together with per-file statistics: partition values, record counts, file size, and column-level lower/upper bounds and null counts. These column stats are what allow predicate pushdown to prune files without opening them. A snapshot is the union of all files referenced by all the manifests in its manifest list.

Q: In what file format are Iceberg manifest lists and manifest files stored, and why does that matter?
A: Both manifest lists and manifests are stored as Avro files (data files are typically Parquet/ORC/Avro). Avro is a compact, schema-evolvable, row-oriented format well suited to the many small metadata records being appended per commit. Knowing this matters operationally: if you want to inspect a manifest by hand you reach for an Avro reader, not a Parquet one.

Q: How do the column-level lower and upper bounds stored in manifests speed up queries?
A: Those bounds let the engine prune at planning time: if a manifest entry says a data file's `nav_date` ranges from `2026-01-01` to `2026-01-31` and your query filters `nav_date = '2026-03-15'`, the engine skips that file entirely without reading it. This file-level skipping (plus partition pruning) is why a well-maintained Iceberg table reads far fewer bytes than a full scan. Bad statistics or no stats means the engine must read everything.

Q: Define an Iceberg snapshot.
A: A snapshot is the complete state of the table at one point in time — effectively the full set of data and delete files valid after a particular commit. Each snapshot has a unique snapshot ID, a parent snapshot ID, a timestamp, an operation summary (append, overwrite, delete), and a pointer to its manifest list. Querying the table normally means querying its current snapshot.

Q: How does each commit to an Iceberg table create a new snapshot?
A: Every write operation produces new data/delete files, writes new manifests and a new manifest list describing the resulting file set, writes a new `metadata.json` that adds this snapshot and marks it current, and then atomically swaps the catalog pointer to that new metadata file. The previous snapshot is left untouched and remains queryable. This append-only, immutable design is what makes commits atomic and time travel possible.

Q: What is "time travel" in Iceberg and what makes it cheap?
A: Time travel is querying the table as it existed at an earlier snapshot, by ID or by timestamp, rather than its current state. It is cheap because old snapshots are not copies — each snapshot just references a set of immutable data files that are still present on storage, so reading an old snapshot is reading files that already exist. No data is duplicated to support it; you only pay storage for files that haven't been expired.

Q: In Trino, what is the syntax to read an Iceberg table at a specific snapshot ID?
A: You use `SELECT * FROM schema.table FOR VERSION AS OF 8954597067493422955`, where the number is a snapshot ID from the `$snapshots` metadata table. This reads the table exactly as it existed at that commit. It is the snapshot-ID form of time travel, deterministic regardless of when you run it.

Q: In Trino, how do you query an Iceberg table as of a particular wall-clock time?
A: You use `SELECT * FROM schema.table FOR TIMESTAMP AS OF TIMESTAMP '2026-01-31 17:00:00.000 Europe/Luxembourg'`, and Trino resolves that to the snapshot current at that instant. This is handy when you know "the state at end of the Luxembourg business day" but not the exact snapshot ID. Internally it still maps to a single snapshot from the snapshot history.

Q: A NAV correction landed this morning; how do you reproduce yesterday's published NAV report from the same Iceberg table?
A: Find the snapshot that was current before the correction commit (from the `$snapshots` or `$history` metadata table) and time-travel to it with `FOR VERSION AS OF <snapshot_id>` or `FOR TIMESTAMP AS OF` the publication time. Re-running the report against that snapshot reproduces the exact figures, because that snapshot still references the pre-correction data files. This is the audit-grade reproducibility that makes time travel a regulatory feature, not just a convenience.

Q: Which Trino metadata table lists the snapshot history of an Iceberg table, and why is it your first stop for time travel?
A: The `"table_name$snapshots"` metadata table lists each snapshot's ID, parent ID, timestamp, operation, and a summary of changes. It is the first stop because time travel needs a concrete snapshot ID or timestamp, and this table is where you find the one you want. You typically `ORDER BY committed_at` and pick the row just before the change you are trying to reproduce.

Q: What does the `"table_name$history"` metadata table show that `$snapshots` does not emphasise?
A: The `$history` table shows the lineage of which snapshot was the table's current state over time, including the parent relationships and whether ancestors were rolled back or replaced. It answers "what was the live snapshot at moment X" rather than just "what snapshots exist". This matters because a rollback can make a later-timestamped snapshot no longer an ancestor of the current one.

Q: What is hidden partitioning in Iceberg and how does it differ from Hive-style partitioning?
A: Hidden partitioning means Iceberg derives partition values from a column via a transform (for example `day(nav_date)`) and tracks them in metadata, instead of requiring physical partition directories and extra partition columns in the path. Unlike Hive, the user does not have to write a separate partition predicate or manage directory layout — querying `WHERE nav_date = '2026-01-31'` automatically benefits from the partition without any special syntax. The partitioning is "hidden" because it is invisible to the query author.

Q: Why does Hive's explicit partitioning often lead to accidental full scans that Iceberg avoids?
A: In Hive you must filter on the literal partition column (often a separate `dt` string) for pruning to kick in; filtering on the underlying timestamp column instead silently scans everything. Iceberg ties the transform to the source column in metadata, so a filter on the real column prunes correctly without the user knowing the partitioning scheme. This removes a whole class of "why is my query reading the entire table" bugs.

Q: Name several partition transforms Iceberg supports for hidden partitioning.
A: Iceberg supports `identity`, `year`, `month`, `day`, `hour`, `bucket[N]`, and `truncate[W]` transforms. For a NAV fact table you might partition by `day(nav_date)` for time pruning and `bucket(16, fund_id)` to spread funds evenly across files. The transform is stored in the partition spec, so the engine knows how to map predicates to partitions.

Q: What is partition evolution in Iceberg and what is its key benefit?
A: Partition evolution lets you change a table's partition spec — say from `month(nav_date)` to `day(nav_date)` — without rewriting any existing data. New writes use the new spec while old data keeps its old spec, and Iceberg tracks both. The benefit is you can refine partitioning as data volume grows without an expensive full rewrite or breaking existing queries.

Q: After you evolve a partition spec, do existing queries need to be rewritten?
A: No — because partitioning is hidden, queries filter on the source columns, not on partition layout, so the same SQL keeps working across the spec change. Iceberg applies whichever spec each data file was written under during planning. This is the headline reason hidden partitioning and partition evolution are designed together.

Q: How does Iceberg store the fact that different data files use different partition specs?
A: Each partition spec is assigned a `spec-id`, and every data file's manifest entry records which `spec-id` it was written with. The `metadata.json` keeps the full list of historical specs keyed by id. At plan time the engine reads each file's spec-id to interpret its partition values correctly, which is how old and new specs coexist in one table.

Q: What is copy-on-write (COW) in Iceberg?
A: Copy-on-write means that when you UPDATE or DELETE rows, Iceberg rewrites every data file that contained an affected row, producing new files with the changes applied and dropping the old ones from the new snapshot. There are no separate delete files — the new snapshot's data files are already correct. It is the default for Iceberg and optimises read performance at the cost of write amplification.

Q: What is merge-on-read (MOR) in Iceberg?
A: Merge-on-read means an UPDATE or DELETE does not rewrite the original data files; instead it writes small delete files that mark which rows are invalidated, and readers merge data files with their delete files at query time. This makes writes cheap and fast but pushes cost onto every read until compaction reconciles them. MOR was introduced with the Iceberg v2 spec.

Q: What is the core read-versus-write tradeoff between COW and MOR?
A: COW makes reads cheap (files are already correct) but writes expensive (whole files rewritten per change), so it suits read-heavy tables with infrequent or batched updates. MOR makes writes cheap (only small delete files appended) but reads more expensive (data and delete files must be merged), so it suits high-frequency row-level changes. The architect picks per workload, and you can even set different modes for delete, update, and merge.

Q: What two kinds of delete files does Iceberg merge-on-read use?
A: Position deletes and equality deletes. A position delete identifies invalidated rows by referencing a specific data file path plus the ordinal row positions within it. An equality delete identifies rows by column values (for example "delete where `holding_id = 123`") without naming a specific file. Each suits a different write pattern.

Q: When is a position delete preferable to an equality delete, and vice versa?
A: Position deletes are efficient to read because they point straight at file-and-row offsets, but they require the writer to know exactly where the row lives, so they are typically slower/heavier to produce. Equality deletes are quick to write — you just record the matching key values — but slower to read because the engine must evaluate that predicate against data files. Streaming upserts often favour equality deletes; batch operations favour position deletes.

Q: Which Iceberg table properties control COW versus MOR, and what are their values?
A: The properties are `write.delete.mode`, `write.update.mode`, and `write.merge.mode`, each set to either `copy-on-write` or `merge-on-read`. You set them independently, so a table can do MOR deletes but COW merges, for example. They are stored in `metadata.json` table properties and read by the writing engine.

Q: If a merge-on-read Iceberg table feels slow to query, what is the likely cause and the fix?
A: The likely cause is an accumulation of delete files that every read must merge against the data files — read amplification grows with each un-compacted upsert. The fix is to run compaction (rewrite data files) so the deletes are applied and the data files become clean again, after which reads no longer pay the merge cost. This is exactly the operational hygiene MOR demands.

Q: What is compaction in the Iceberg context?
A: Compaction is rewriting many small data files (and applying any MOR delete files) into fewer, larger, well-sized data files. It improves read performance by reducing per-file overhead and the number of files an engine must open, and it materialises MOR deletes so reads stop merging them. It is a maintenance operation, not a data change — the logical contents are identical.

Q: In Trino, what statement compacts (optimizes) an Iceberg table's data files?
A: You run `ALTER TABLE table_name EXECUTE optimize`, optionally with `optimize(file_size_threshold => '128MB')` to only rewrite files below a size, and optionally a `WHERE` clause to limit it to certain partitions. This rewrites small files into larger ones for that scope. It is the Trino equivalent of the `rewrite_data_files` action in the Iceberg core API.

Q: Why is "small files" such a common problem on Iceberg tables, and how does compaction help?
A: Frequent streaming or micro-batch commits each write at least one small data file, so over time a table accumulates thousands of tiny files; planning and opening them dominates query cost and inflates metadata. Compaction merges them into appropriately sized files (commonly targeting ~128–512 MB), cutting per-file overhead and shrinking manifests. Without it, read latency and metadata size grow without the table's logical data growing.

Q: What does expiring snapshots do, and why is it necessary?
A: Expiring snapshots removes old snapshots from the table history and deletes the data and metadata files that are referenced only by those expired snapshots and nothing current. It is necessary because every commit retains old files for time travel, so storage grows unbounded until you expire — without it a frequently updated table balloons in size. The tradeoff is you lose the ability to time-travel to expired snapshots.

Q: In Trino, how do you expire Iceberg snapshots older than a retention threshold?
A: You run `ALTER TABLE table_name EXECUTE expire_snapshots(retention_threshold => '7d')`, which expires snapshots older than seven days and deletes files no longer reachable from a retained snapshot. You can also pass `retain_last` to guarantee a minimum number of snapshots are kept. Be deliberate: anything expired can no longer be reached by time travel.

Q: What is the danger of setting an aggressive snapshot-expiry window on a regulated NAV table?
A: If you expire snapshots faster than your reproducibility/audit requirement, you destroy the ability to time-travel to a needed historical state — for example reproducing a NAV as published on a given date for a regulator. Retention must be governed by the audit and correction-reprocessing window, not just storage cost. A common gotcha is matching expiry to a vague "save space" goal and only discovering the gap when an auditor asks.

Q: What does rewriting/optimizing manifests achieve, separate from rewriting data files?
A: Rewriting manifests reorganises and compacts the manifest files themselves so that manifests are well-sized and well-clustered (often by partition), which speeds up query planning. In Trino this is `ALTER TABLE table_name EXECUTE optimize_manifests`. It is distinct from data-file compaction: here you are tidying the metadata layer, not the data.

Q: What problem do orphan files cause, and how does Iceberg clean them?
A: Orphan files are data or metadata files left on storage that no live snapshot references — typically from failed writes or aborted jobs — and they silently consume storage and confuse manual inspection. In Trino, `ALTER TABLE table_name EXECUTE remove_orphan_files(retention_threshold => '7d')` deletes unreferenced files older than the threshold. The threshold guards against deleting files from in-flight commits.

Q: Why must `remove_orphan_files` use a conservative retention threshold?
A: Because a write that is still in progress has already written data files that are not yet referenced by a committed snapshot; if you delete "unreferenced" files too eagerly you can corrupt an in-flight commit. The retention threshold (for example `7d`) ensures only files older than any plausibly running job are removed. This is the classic gotcha that turns a cleanup job into data loss.

Q: How would you debug "why is this Iceberg table slow or huge" using the manifests alone?
A: Query `$files` and `$partitions` to see file counts and sizes per partition, and `$manifests` to see how many manifests and entries exist; a slow/huge table typically shows thousands of tiny data files, many delete files, or bloated manifests. From those numbers you decide the remedy: `optimize` for small files, compaction to apply MOR deletes, `optimize_manifests` for manifest bloat, and `expire_snapshots` plus `remove_orphan_files` for retained/orphaned bytes. The point is you diagnose purely from metadata before touching the data.

Q: What is the rule that makes Iceberg schema evolution safe across engines?
A: Iceberg assigns every column a unique, immutable field ID and tracks columns by that ID, not by name or position. Adding, dropping, renaming, or reordering columns is a metadata-only change because old data files are read by mapping their stored field IDs to the current schema. This is why schema evolution never requires rewriting data and never silently misreads an old file.

Q: Which schema changes does Iceberg consider safe and metadata-only?
A: Adding a new column (it reads as null for old rows), dropping a column, renaming a column, reordering columns, and widening certain types (for example `int` to `long`, `float` to `double`, increasing decimal precision) are safe metadata-only operations. They are safe precisely because resolution is by field ID, so existing data files are reinterpreted rather than rewritten. Narrowing a type or other lossy changes are not allowed.

Q: Why does adding a column to an Iceberg table not require rewriting existing data files?
A: Because the new column simply gets a new field ID that does not appear in any existing data file; when those old files are read, the missing field ID resolves to null. The change is recorded only in `metadata.json` as a new schema. So an `ADD COLUMN` is instantaneous regardless of table size, which is a major operational advantage over rewrite-based formats.

Q: You add a `lei` column to an existing Iceberg holdings table — what do queries return for rows written before the change?
A: They return `NULL` for `lei` on all pre-existing rows, because those data files have no field with the new column's field ID, and Iceberg resolves the absent ID to null. No backfill happens automatically; the old files are untouched. To populate historical `lei` values you would run a separate UPDATE/backfill, which is a real data change with its own snapshot.

Q: Describe the Delta Lake log model at a high level.
A: Delta Lake records table state in a `_delta_log` directory containing ordered JSON commit files, one per transaction, each holding a list of actions such as `add` (a data file added), `remove` (a file removed), `metaData`, and `protocol`. The current table state is the result of replaying these actions in order. Periodically Delta writes a Parquet checkpoint that snapshots the full state so readers don't have to replay every JSON file.

Q: What are Delta checkpoint files and why do they exist?
A: Checkpoint files are Parquet files that Delta writes every N commits (10 by default) summarising the entire table state up to that point. They exist so a reader can load the latest checkpoint and then replay only the few JSON commits after it, instead of replaying potentially thousands of JSON files from the beginning. This keeps read/recovery time bounded as the log grows.

Q: How does the Delta commit/log model contrast with the Iceberg metadata-tree model?
A: Delta is log-structured: state is derived by replaying an ordered sequence of JSON action files (with periodic Parquet checkpoints) in a single `_delta_log`. Iceberg is tree-structured: each commit writes a self-contained snapshot via a new metadata file pointing to a manifest list and manifests, and the catalog pointer is swapped. Both achieve ACID and time travel, but Iceberg's per-snapshot immutable tree and catalog pointer differ from Delta's append-and-checkpoint log.

Q: What is Delta UniForm and what problem does it address?
A: UniForm (Universal Format) lets a Delta table expose itself to Iceberg (and Hudi) readers by asynchronously generating the Iceberg metadata alongside the Delta log when data is written. It addresses interoperability: a single physical Delta table can be queried by Iceberg-native engines without a separate copy or conversion. The data files are shared; only the extra metadata is generated.

Q: At what level should you know Apache Hudi for this lesson, and what is its distinguishing feature?
A: Awareness level: you should recognise Hudi as the third major open table format, originally from Uber, designed around fast upserts and incremental pulls. Its distinguishing feature is record-level indexing and built-in support for copy-on-write and merge-on-read table types tuned for streaming ingestion. You are not expected to operate it here, only to place it alongside Iceberg and Delta in the format landscape.

Q: What goes into an Iceberg-vs-Delta ADR for a given estate?
A: The ADR weighs concrete factors: existing engine and vendor footprint (Databricks favours Delta; multi-engine/open favours Iceberg), catalog strategy and credential vending, who else must read the data, maturity of partition/schema evolution needs, and operational tooling for maintenance. You state the decision, the forces (lock-in, interoperability via UniForm, team skills), and the consequences. The deliverable is a defensible recommendation with reasons, not a feature checklist.

Q: For a multi-engine Luxembourg fund platform where Trino, Spark, and a future Snowflake all read the same tables, which format leans favourable and why?
A: Iceberg leans favourable because it is engine-neutral with broad native support (Trino, Spark, Snowflake managed Iceberg, Flink) and a vendor-independent REST catalog story, minimising lock-in across that heterogeneous estate. Delta is strongest inside the Databricks ecosystem and reaches other engines mainly via UniForm. The ADR would cite multi-engine read access and catalog openness as the deciding forces.

Q: What is the practical effect of Iceberg commits being atomic via a single pointer swap?
A: Because committing means atomically swapping the catalog's pointer from the old `metadata.json` to the new one, a reader either sees the entire previous state or the entire new state — never a half-written mix. Concurrent readers continue on the old metadata file until they refresh. This is the mechanism behind Iceberg's snapshot isolation.

Q: How does Iceberg handle two concurrent writers committing to the same table?
A: Iceberg uses optimistic concurrency: each writer builds its new metadata based on the snapshot it read, then attempts the atomic pointer swap, which succeeds only if the table's current metadata is still the one it started from. If another writer committed first, the swap fails and the losing writer must retry by re-reading the latest metadata and re-applying its changes. This guarantees serializable commits without locking the whole table.

Q: What is snapshot isolation and how does Iceberg provide it?
A: Snapshot isolation means a reader sees a consistent snapshot of the table for the duration of its query, unaffected by concurrent writers. Iceberg provides it because a query binds to the current snapshot (via the immutable metadata file) when it starts, and later commits only swap the pointer forward without altering the files that snapshot references. The reader's files never disappear mid-query.

Q: Which Trino metadata table would you use to see the actual data files in the current snapshot, and what is one column you'd inspect?
A: You query `"table_name$files"`, which lists each data file with columns like `file_path`, `record_count`, `file_size_in_bytes`, and per-column value bounds. Inspecting `file_size_in_bytes` (and the count of rows) quickly reveals a small-files problem worth compacting. It is the most direct way to confirm file sizing before running `optimize`.

Q: What does the `"table_name$partitions"` metadata table tell you in Trino?
A: It aggregates the current snapshot by partition, showing per-partition record counts, file counts, and total sizes. This is how you spot skew — one partition holding far more (or far more files) than others — and decide whether to compact or revisit the partition spec. For a fund table partitioned by `day(nav_date)`, it shows which dates have the most data and the most files.

Q: How do you list the manifests of an Iceberg table in Trino, and what would high manifest counts suggest?
A: You query `"table_name$manifests"` (and `"table_name$all_manifests"` for history), which shows each manifest's path, length, and added/existing/deleted file counts. A high number of manifests, or manifests with few entries each, suggests manifest bloat that slows planning and warrants `optimize_manifests`. It is part of diagnosing "why is planning slow" purely from metadata.

Q: What is the difference between data files and delete files in an Iceberg snapshot?
A: Data files (typically Parquet) hold the actual table rows. Delete files exist only in merge-on-read tables and mark rows in data files as logically removed — either by position (file path plus row offset) or by equality (matching column values). A reader of a MOR snapshot must combine data files with their applicable delete files to produce correct results.

Q: Why is Iceberg sometimes described as "immutable metadata, immutable data"?
A: Because no existing file is ever modified in place: each commit writes new data files (or delete files), new manifests, a new manifest list, and a new metadata file, leaving all prior files untouched. Mutation is expressed as new files plus a pointer swap, never as edits. This immutability is what makes time travel, atomic commits, and easy rollback possible.

Q: How do you roll back an Iceberg table to a previous snapshot in Trino?
A: You run `ALTER TABLE table_name EXECUTE rollback_to_snapshot(<snapshot_id>)`, which sets the table's current snapshot back to that earlier one. Because the old snapshot's files still exist (assuming they weren't expired), this is a fast metadata operation, not a data restore. It is the operational counterpart to "undo the bad commit".

Q: What is the difference between time travel and rollback in Iceberg?
A: Time travel is a read-only query against an old snapshot (`FOR VERSION/TIMESTAMP AS OF`) that does not change the table's current state. Rollback (`rollback_to_snapshot`) actually changes the table's current pointer so the old snapshot becomes the live state again, affecting all future reads. Use time travel to inspect or reproduce; use rollback to recover from a bad commit.

Q: Why does a "botched compaction" risk silently tripling a table's storage?
A: Compaction writes new larger files but does not delete the old small files until those old snapshots are expired; if you compact aggressively without then expiring snapshots and removing orphans, you keep both the old and new file sets. Repeated compaction cycles can multiply retained bytes several times over. The fix is to pair compaction with `expire_snapshots` and `remove_orphan_files` on a governed schedule.

Q: What sequence of maintenance operations keeps an Iceberg table healthy?
A: A typical cadence is: compact data files (`optimize`/rewrite) to fix small files and apply MOR deletes, optionally rewrite manifests (`optimize_manifests`), then `expire_snapshots` to drop old snapshots beyond your retention window, and finally `remove_orphan_files` to clear stragglers. Order matters because expiry and orphan removal reclaim the storage that compaction's rewrites left behind. Run on a schedule tuned to write frequency and your audit retention.

Q: A daily NAV table grows from 50 GB to 150 GB over a month with no extra source data — what is the first thing to check?
A: First check snapshot retention: every daily commit (and any compaction) retains old data files for time travel, so without `expire_snapshots` the table accumulates a month of historical file versions. Query `$snapshots` to see how many are retained and `$files` to see retained bytes, then expire beyond your audit window and run `remove_orphan_files`. Unbounded snapshot retention is the most common cause of silent growth.

Q: How is an Iceberg table's current schema resolved when reading an old data file with fewer columns?
A: The engine reads the old file's columns by their field IDs and projects them into the current schema; any current column whose field ID is absent in the file resolves to null, and dropped columns are simply ignored. This field-ID-based projection is what lets one query span files written under different schema versions. No file is rewritten to "catch up" to the current schema.

Q: Why must you never rely on column position or name for Iceberg correctness across schema changes?
A: Because Iceberg resolves columns by immutable field ID, not by position or name — a rename changes the name but keeps the ID, and reordering changes positions but keeps IDs. Code or assumptions that depend on ordinal position or string name can break after an evolution that Iceberg itself handles cleanly. Always think in field IDs when reasoning about evolution correctness.

Q: What does an UPDATE statement physically produce on a copy-on-write Iceberg table?
A: On COW, the UPDATE rewrites every data file containing at least one matching row into a new file with the updates applied, writes new manifests and a manifest list, and commits a new snapshot whose data files are already correct. The old files remain referenced only by prior snapshots. Even a one-row change can rewrite a large file — the write-amplification cost of COW.

Q: What does the same UPDATE produce on a merge-on-read Iceberg table?
A: On MOR, the UPDATE leaves existing data files in place and writes small delete files marking the old row versions invalid plus new data files holding the updated rows, committing a new snapshot. Reads then merge data and delete files to see the correct result. Writes are cheap, but read cost rises until compaction reconciles the deletes.

Q: A late-arriving holding must be inserted into yesterday's partition of an Iceberg NAV table — does this break time travel?
A: No — inserting it is just another commit that creates a new snapshot with the additional data file; all earlier snapshots still reference their original file sets, so time travel to them is unaffected. Querying the current snapshot now includes the late holding, while a `FOR VERSION AS OF` an earlier snapshot does not. This is exactly the late-arriving-data scenario the lesson's MERGE exercise simulates.

Q: Why is Iceberg's MERGE statement central to handling NAV corrections and late data?
A: MERGE expresses upsert logic in one atomic statement — match existing rows to update/delete them and insert new ones — which maps directly to "apply a correction and add late holdings". It commits as a single snapshot, so the table is never in a partial state, and you can time-travel to before it. The COW/MOR mode you chose decides whether MERGE rewrites files or writes delete files.

Q: What is write amplification and why does it favour MOR for high-frequency updates?
A: Write amplification is the ratio of bytes physically written to bytes logically changed; COW rewrites whole files for tiny edits, so a one-row change in a large file amplifies enormously. MOR avoids this by only appending small delete/data files, keeping write cost proportional to the change. For streaming upserts or frequent corrections, MOR's low amplification is the deciding factor — at the price of read-time merging.

Q: When is COW the better default despite its write cost?
A: COW is better for read-heavy tables that change infrequently or in large batches, because reads never pay a merge penalty and there are no delete files to maintain. A reference table updated nightly and queried thousands of times a day fits COW well. The architect chooses COW to optimise the common read path when updates are rare and batched.

Q: How does Iceberg's design let one query read across multiple snapshots' worth of data files written under different specs and schemas?
A: At plan time the engine reads, per data file from the manifests, that file's partition `spec-id` and the schema (via field IDs) under which it was written, and reconciles each against the table's current spec and schema. This per-file metadata is why old and new files coexist transparently. The query author writes ordinary SQL against the current schema and partitioning.

Q: What is the role of the `$entries` (manifest entries) metadata table in Trino?
A: The `$entries` table exposes the raw manifest-entry rows for the current snapshot — each entry being a data or delete file with its status (added/existing/deleted), partition data, and stats. It is the lowest-level view, useful when you need to see exactly what each manifest records about each file. Most diagnostics use `$files`/`$manifests`, but `$entries` is there for deep inspection.

Q: On MinIO, where would you look on disk to inspect the Iceberg metadata tree, and in what order?
A: Under the table's `metadata/` prefix you find the `*.metadata.json` files (the latest is current), the `snap-*.avro` manifest lists, and the `*.avro` manifest files; the `data/` prefix holds the Parquet data files. You narrate the tree by opening the current `metadata.json`, following its current snapshot's `manifest_list` to the snap Avro, then reading that to the manifest Avros, then to the listed data files. That walk is the lesson's "narrate the tree out loud" exercise.

Q: Why is the catalog (not the format) ultimately what decides openness, even with Iceberg files?
A: Because the catalog owns the pointer to current metadata and brokers access; if your catalog is engine-bundled and proprietary, other engines cannot resolve or write the table even though the underlying files are open Iceberg. The data format being open is necessary but not sufficient for interoperability. This is why the next lesson on catalogs is the real lock-in battleground — here you just note it.

Q: What format are Iceberg data files most commonly stored in, and is that fixed?
A: Most commonly Parquet, though Iceberg also supports ORC and Avro for data files. The choice is independent of the metadata layer — the manifests just record each file's format. Parquet's columnar layout and statistics make it the default for analytical NAV/holdings tables.

Q: How does Iceberg avoid the expensive directory listings that plague Hive tables on object storage?
A: Iceberg tracks the exact set of files for each snapshot in its manifests, so the engine reads a known file list from metadata instead of doing slow, eventually-consistent `LIST` calls against object storage to discover files. On S3/MinIO with millions of objects, this is a major performance and correctness win. The metadata tree is effectively a precomputed, consistent file index.

Q: What is a "snapshot summary" and why is it useful operationally?
A: The snapshot summary is a small set of key/value statistics Iceberg records per snapshot — the operation type plus counts like `added-data-files`, `deleted-data-files`, `added-records`, and total file/record counts. It lets you understand what each commit did without reading the data, straight from `$snapshots`. For audit and debugging, it answers "what changed in this commit" at a glance.

Q: How would you verify, after a partition-spec change, that a query still runs unchanged?
A: Run the same `SELECT ... WHERE nav_date = '2026-01-31'` before and after the spec change and confirm identical results and successful execution; then check `$partitions` to see new data under the new spec and old data under the old spec coexisting. Because partitioning is hidden, no query edit is needed, which is precisely what you are confirming. This is the lesson's "evolve the partition spec; query still runs unchanged" step.

Q: Why does Iceberg keep historical partition specs in `metadata.json` rather than discarding them?
A: Because existing data files were written under those older specs and their partition values must be interpreted with the matching spec at read time; discarding a spec would make older files unreadable in partition terms. Retaining every spec by `spec-id` is what allows partition evolution without rewriting data. The engine selects the right spec per file during planning.

Q: What is the practical meaning of "the Iceberg-vs-Delta decision is an ADR you will own"?
A: It means choosing the table format is an architectural decision with long-lived consequences (lock-in, interoperability, tooling, team skills) that you must justify in writing, not a default you inherit. You document the context, the options, the forces, the decision, and the consequences so others can understand and revisit it. Owning it means defending concrete reasons specific to the estate, not citing generic feature parity.

Q: How does time travel support reproducible regulatory reporting in a fund-administration context?
A: It lets you re-execute a report exactly as it was at the snapshot underlying a prior submission — for example reproducing a UCITS NAV report after a later correction — because that snapshot still references the original data files. This gives bit-for-bit reproducibility and a clear audit trail of what changed and when. Regulators value that the historical state is provable, not reconstructed by guesswork.

Q: If `expire_snapshots` deleted a snapshot you needed for an audit, what is the recovery reality?
A: There is generally no recovery: expiry physically deletes the data and metadata files reachable only from expired snapshots, so unless those exact files survive elsewhere (a backup or a still-referencing snapshot), the historical state is gone. This is why retention must be governed by audit requirements before you ever run expiry. The lesson here is that snapshot retention policy is a compliance control, not an ops afterthought.

Q: What does it mean that Iceberg supports "row-level operations" and why was that historically hard on data lakes?
A: Row-level operations (UPDATE/DELETE/MERGE on individual rows) were hard on classic data lakes because Parquet files are immutable and there was no transactional layer to track which rows changed. Iceberg makes them possible via COW (rewrite affected files) or MOR (delete files), all committed atomically as snapshots. This brings warehouse-style mutation to open files on object storage.

Q: Why might you choose merge-on-read for a transfer-agency upsert stream but copy-on-write for a published NAV table?
A: The transfer-agency stream has frequent, small, late-arriving record changes, so MOR's cheap appends avoid rewriting large files on every upsert. The published NAV table is read far more than written and must be fast and clean for reporting, so COW keeps reads free of delete-file merging. Matching the mode to the read/write ratio per table is the architect's call, set via the `write.*.mode` properties.

Q: What does the `$refs` metadata table expose and how does it relate to branches and tags?
A: The `$refs` table lists the named references on the table — branches and tags — each pointing to a snapshot ID. A tag pins a specific snapshot (for example an end-of-month NAV close) so it survives normal expiry, and a branch supports independent commit lines. Trino can time-travel by tag/branch name via `FOR VERSION AS OF 'tag-name'`.

Q: How could an Iceberg tag help guarantee a month-end NAV snapshot remains reproducible?
A: You create a tag pointing at the snapshot that represents the official month-end close; tags can have their own retention so the underlying snapshot's files are protected from `expire_snapshots`. You then time-travel to that named tag at any future date to reproduce the close exactly. This turns a transient snapshot into a durable, named regulatory checkpoint.

Q: Why is partition evolution preferable to dropping and recreating a table with a new layout?
A: Dropping and recreating forces a full rewrite of all data and breaks every existing query and downstream reference, plus it discards history. Partition evolution changes only the spec going forward, leaving existing files and queries intact and preserving snapshots/time travel. It is the low-risk, in-place way to refine layout as data grows.

Q: What metadata change happens when you run `ADD COLUMN` versus `RENAME COLUMN` on Iceberg?
A: `ADD COLUMN` allocates a new field ID and records a new schema where that ID maps to the new column (null for old data). `RENAME COLUMN` keeps the existing field ID and only changes the name string in the schema, so all existing data files still resolve correctly by ID. Both are metadata-only `metadata.json` updates with no data rewrite.

Q: Why does Iceberg disallow narrowing type changes like `long` to `int`?
A: Because existing data files may contain values that don't fit the narrower type (a `long` value too big for `int`), so reinterpreting them by field ID would lose or corrupt data. Iceberg only allows promotions that are always safe (widening), keeping evolution lossless without rewriting files. To narrow a type you must rewrite the data through an explicit transformation.

Q: How do you confirm a NAV correction produced a new snapshot and identify the snapshot just before it?
A: Query `"table_name$snapshots" ORDER BY committed_at`, find the row whose `operation` and summary correspond to the correction (for example `overwrite` with the expected changed-record counts), and take the `snapshot_id` of the immediately preceding row as the pre-correction state. You then time-travel to that prior snapshot to reproduce the old NAV. The summary counts help you confirm you picked the right commit.

Q: What is the difference between Iceberg's `optimize` and `expire_snapshots` in terms of what they reclaim?
A: `optimize` reclaims read performance by rewriting small/fragmented data files into larger ones (and applying MOR deletes), but it can temporarily increase storage because old files persist until expiry. `expire_snapshots` reclaims storage by dropping old snapshots and deleting files reachable only from them. They are complementary: compact for speed, expire for space.

Q: Why is it dangerous to run `optimize` without a later `expire_snapshots` on a high-churn table?
A: Each `optimize` creates a new snapshot referencing the new compacted files while the old files remain referenced by the prior snapshot for time travel, so storage keeps both copies until you expire. On a high-churn table run frequently, retained pre-compaction files can multiply storage several-fold. Pairing compaction with governed expiry is what prevents the "silently tripled in size" failure.

Q: In Iceberg v2, what made merge-on-read possible at the spec level?
A: The v2 spec introduced delete files as first-class entries in manifests — position deletes and equality deletes — so a snapshot can reference rows-to-ignore separately from the data files. v1 had no delete files, so all mutation was effectively copy-on-write. MOR is therefore a v2-and-later capability, and you confirm your table format version before relying on it.

Q: What is the relationship between an Iceberg snapshot and the manifest list it points to?
A: A snapshot has exactly one `manifest_list` field that points to a single Avro manifest list file, and that manifest list enumerates every manifest (and thus, transitively, every data/delete file) belonging to that snapshot. So "the files of a snapshot" is computed by reading its manifest list, then its manifests. Change a single file and the new commit gets a brand-new manifest list (reusing unchanged manifests where possible).

Q: Why can Iceberg reuse unchanged manifests across snapshots, and what does that save?
A: Because manifests are immutable Avro files describing a fixed set of files, a new commit that only touches some partitions can keep referencing the older manifests for untouched partitions and write new manifests only for the changed ones. This saves metadata write cost and storage — you don't rewrite the whole file index per commit. The new manifest list simply mixes reused and new manifests.

Q: What would you inspect first if a Trino query against an Iceberg table returns stale results after a write from another engine?
A: Check whether your engine/session has refreshed the catalog pointer — a query bound to an older `metadata.json` won't see the newer snapshot until the metadata is reloaded. Confirm the catalog (e.g. the metastore/REST catalog) actually points at the new `metadata.json`, and verify via `$snapshots` that the expected commit landed. Snapshot isolation means each query sees a fixed snapshot; "staleness" is usually a not-yet-refreshed pointer, not lost data.

Q: How do Iceberg, Delta, and Hudi compare in one sentence each for an awareness-level summary?
A: Iceberg is the engine-neutral, spec-driven format with a metadata tree and catalog pointer, strongest for open multi-engine estates; Delta is the log-structured format strongest in the Databricks ecosystem, reaching others via UniForm; Hudi is the upsert/streaming-optimised format with record-level indexing from the Uber lineage. All three offer ACID, schema evolution, and time travel, differing mainly in metadata design, ecosystem, and ideal workloads. The ADR turns these one-liners into estate-specific reasons.

Q: What does the `clean_expired_metadata` option add when running `expire_snapshots` in Trino?
A: Passing `clean_expired_metadata => true` to `ALTER TABLE table_name EXECUTE expire_snapshots(...)` also removes metadata — such as partition specs and schemas — that are no longer referenced by any retained snapshot, not just the data and manifest files. This keeps the `metadata.json` from accumulating dead historical specs and schemas over a long-lived table's life. It is a useful hygiene flag on tables that have undergone many schema or partition evolutions.

Q: Putting it together: outline the end-to-end lifecycle of a single NAV-correction commit on an Iceberg COW table.
A: The engine reads the current snapshot, identifies the data files containing the rows to correct, rewrites those files with the corrected NAV values, writes new manifests plus a new manifest list, and writes a new `metadata.json` adding the new snapshot as current; it then atomically swaps the catalog pointer. The prior snapshot and its files stay intact, so you can time-travel to reproduce the pre-correction NAV, and you confirm the change via the new snapshot's summary counts in `$snapshots`. Later maintenance (`optimize`, `expire_snapshots`, `remove_orphan_files`) keeps the table fast and appropriately sized.

Q: How do you create an Iceberg table on Trino while pinning it to format version 2 and partitioning by day of NAV date?
A: You pass table properties in the `WITH` clause, for example `CREATE TABLE nav (fund_id varchar, nav_date date, nav decimal(18,6)) WITH (format_version = 2, partitioning = ARRAY['day(nav_date)'])`. The `format_version = 2` is what unlocks merge-on-read delete files, and `partitioning = ARRAY['day(nav_date)']` applies the hidden `day` transform. Omitting `format_version` lets the catalog default decide, which on recent Trino is already 2.

Q: What Trino table property sets the sort order written inside each Iceberg data file, and why does it help pruning?
A: The `sorted_by` property — for example `WITH (sorted_by = ARRAY['nav_date', 'fund_id'])` — orders rows within each written file so column min/max bounds in the manifests become tight and non-overlapping. Tight bounds mean predicate pushdown skips far more files, because a query on one `fund_id` no longer matches files whose range was smeared across all funds. It is local (per-file) sorting, distinct from partitioning, and complements it.

Q: What is the default target file size Iceberg writers aim for, and which property changes it?
A: The default is 512 MB, controlled by the `write.target-file-size-bytes` table property (set to `536870912` for 512 MiB). Writers and the `optimize` rewrite use this target to decide when to roll over to a new file, so lowering it produces more, smaller files and raising it produces fewer, larger ones. For a high-concurrency NAV table you might tune this to balance parallelism against per-file overhead.

Q: A `WHERE fund_id = 'LU123'` query on an Iceberg table reads every file despite a `bucket(16, fund_id)` partition — what is the likely cause?
A: The bucket transform only prunes on equality against the exact source column, so a mismatch like filtering a cast/derived expression, a different column, or stale statistics defeats it; more often the file-level min/max bounds on `fund_id` overlap across all buckets so file pruning can't help even though partition pruning narrows to one bucket. Confirm via `EXPLAIN` that the predicate is pushed down and check `$partitions` to see whether one bucket really isolates that fund. Adding `sorted_by` on `fund_id` tightens per-file bounds so the remaining files in the bucket also prune.

Q: How do you migrate an existing Hive table into Iceberg in Trino without rewriting its data files?
A: You call the system procedure `ALTER TABLE hive_schema.tbl EXECUTE migrate`, or `system.migrate('hive_schema', 'tbl')` depending on version, which writes Iceberg metadata over the existing Parquet/ORC files in place rather than copying data. The data files are reused as-is and only the manifest/metadata tree is generated, making it fast even on large tables. After migration the table is Iceberg-native and the original Hive table definition is replaced.

Q: When would you use Trino's `register_table` procedure, and what does it not do?
A: You use `system.register_table(schema_name, table_name, metadata_location)` to add an already-existing Iceberg table (its `metadata.json` is on storage) into a catalog that has no record of it — for example after a disaster-recovery copy or migrating between catalogs. It only creates the catalog pointer to the supplied metadata file; it does not move, rewrite, or validate the data, so a wrong `metadata_location` silently registers stale state. Its inverse is `unregister_table`, which drops the catalog entry while leaving files on storage.

Q: What does Trino's `$properties` metadata table show, and give a fund-platform use for it?
A: The `$properties` table lists the table's key/value configuration — entries like `write.format.default`, `write.delete.mode`, `format-version`, and any custom keys. You query it to confirm, for example, that a published NAV table is actually `copy-on-write` for deletes before trusting read performance, or that `format-version` is `2` before relying on MOR. It is the metadata-driven way to audit a table's write behaviour without reading DDL.

Q: What is the `$metadata_log_entries` metadata table for, and how does it differ from `$snapshots`?
A: `$metadata_log_entries` lists each successive `metadata.json` file the table has had, with its timestamp and the latest snapshot/schema/sequence at that point, whereas `$snapshots` lists data commits. They differ because metadata-only changes (a schema or partition-spec evolution, a property change) write a new `metadata.json` without necessarily creating a data snapshot. So to audit "when did someone add the `lei` column" you look in the metadata log, not the snapshot history.

Q: How do you constrain how many old `metadata.json` files Iceberg keeps, and why bother?
A: Set `write.metadata.previous-versions-max` (default 100) together with `write.metadata.delete-after-commit.enabled = true` so each commit prunes the oldest metadata files beyond that count. Without it, a table committed every few minutes accumulates thousands of `metadata.json` files in the `metadata/` prefix, slowing listing and bloating storage even though data is fine. It is a separate hygiene knob from snapshot expiry, targeting the metadata directory specifically.

Q: A streaming job commits to an Iceberg table every 30 seconds; which two maintenance jobs become mandatory and roughly how often?
A: Data-file compaction (`optimize`) to fix the relentless small-files growth, and `expire_snapshots` to drop the thousands of daily snapshots and reclaim their files — typically run hourly-to-daily for compaction and daily for expiry, governed by audit retention. You would also schedule `remove_orphan_files` periodically for aborted-commit stragglers and possibly prune metadata files. The cadence scales with commit frequency: 30-second commits produce ~2,880 snapshots a day, so neglecting these jobs degrades the table within days.

Q: How does Iceberg's `optimize` decide which files to rewrite, and how do you compact only small files?
A: By default `optimize` considers all files in scope and bin-packs them toward the target size; to limit it to genuinely small files you pass `optimize(file_size_threshold => '128MB')` so only files below that size are rewritten. You can further scope it with a `WHERE` clause on partition columns, e.g. `... EXECUTE optimize WHERE nav_date >= DATE '2026-06-01'`, to compact just recent partitions. This avoids needlessly rewriting already-large files and limits the storage churn before expiry.

Q: Why might `optimize` on a partitioned table only ever produce one large file per partition, and is that desired?
A: Because `optimize` bin-packs within partition boundaries — it will not merge data from two different partitions into one file — so each partition compacts independently toward the target size. That is desired: it preserves partition pruning, since a file must belong to exactly one partition value to carry correct min/max partition bounds. If a partition holds less than the target size you simply get one smaller file for it, which is fine.

Q: What does `drop_extended_stats` do in Trino, and when would you run it?
A: `ALTER TABLE tbl EXECUTE drop_extended_stats` removes the table's extended (e.g. NDV/theta-sketch) statistics that the cost-based optimizer uses. You run it when those stats have become stale or skewed after large data changes and are producing bad query plans, then regenerate them with `ANALYZE`. It targets the optimizer-level statistics, not the per-file manifest bounds, which are always maintained automatically.

Q: How do you add a NOT NULL constraint and what is the catch on an existing Iceberg table?
A: You declare it at creation, e.g. `CREATE TABLE ... (isin varchar NOT NULL, ...)`, which records the column as required in the schema. The catch is that Iceberg/Trino generally cannot add a NOT NULL (required) constraint to an existing column whose old data files might contain nulls, because that would retroactively invalidate existing rows without a rewrite. So plan required columns up front; loosening required-to-optional is allowed, tightening optional-to-required is not.

Q: In an Iceberg manifest, what is the `sequence_number` and why does it matter for merge-on-read correctness?
A: Each commit gets a monotonically increasing sequence number stamped onto its data and delete files, and a delete file only applies to data files with a sequence number less than or equal to its own. This ordering is what guarantees a delete written at commit N invalidates rows from earlier commits but not from data inserted after it. Without sequence numbers, MOR could wrongly drop freshly inserted rows that happen to match an older equality delete.

Q: An equality delete on `holding_id = 123` was committed, then the holding was re-inserted with the same id — why is it still visible?
A: Because the re-insert is a later commit with a higher sequence number, and the equality delete only applies to data files at or below its own sequence number; the new row sits in a newer data file the delete cannot touch. This sequence-number scoping is precisely what makes upsert-by-equality-delete-then-insert correct. It is also why blindly re-running an old equality delete won't re-delete a row added afterwards.

Q: Why are equality deletes considered dangerous on wide-fan-out keys, and what is the mitigation?
A: An equality delete on a non-selective key (say a `currency` or `share_class` value matching millions of rows) forces every read to evaluate that predicate against many data files until compaction, causing severe read amplification and potentially huge logical deletions. The mitigation is to prefer position deletes for large operations, keep equality deletes scoped to selective keys like a primary id, and compact promptly. On a fund table, equality-delete by `isin` is fine; by `fund_currency` is a footgun.

Q: How do you list only the delete files (not data files) of a merge-on-read Iceberg table in Trino?
A: Query the `$files` or `$entries` table and filter on the file content type — in `$entries` the `data_file.content` column is `0` for data, `1` for position deletes, and `2` for equality deletes, so `WHERE data_file.content <> 0` isolates delete files. This tells you how many deletes have accumulated and of which kind, which is your signal to compact. A growing equality-delete count on a hot table is the classic MOR read-amplification warning sign.

Q: What does the `read.split.target-size` (split sizing) influence and how does it relate to file size on Iceberg?
A: Split target size governs how the engine slices scannable work into parallel tasks at read time, ideally aligning splits to row groups within the well-sized data files. If files are tiny, you get one tiny split each and scheduling overhead dominates; if files are huge with few row groups, parallelism suffers. Proper compaction to the target file size plus reasonable Parquet row-group sizing is what lets split sizing produce balanced, efficient parallel reads.

Q: After `expire_snapshots`, time travel to a recent snapshot now errors with "snapshot not found" — what happened and how do you prevent it?
A: Expiry removed that snapshot and the files reachable only from it, so the time-travel reference is gone for good; this is irreversible deletion, not a transient error. Prevent it by setting `retention_threshold` and `retain_last` to cover your audit/reproducibility window, and by pinning critical states (month-end close) with tags that survive expiry. The lesson is that expiry policy is a compliance control, so the window must come from the audit requirement, not a storage target.

Q: How does an Iceberg tag differ from a branch, and which suits an immutable month-end NAV close?
A: A tag is a named, typically immutable reference to one specific snapshot, optionally with its own retention; a branch is a movable reference supporting an independent line of commits (e.g. for WAP/audit fixups). The month-end close suits a tag: you pin the exact close snapshot, give it long retention so expiry can't reclaim it, and time-travel to it by name. Branches are for ongoing parallel work, not for freezing a regulatory checkpoint.

Q: What Trino syntax time-travels to a named tag rather than a snapshot ID?
A: You use `SELECT * FROM nav FOR VERSION AS OF 'month_end_2026_05'`, passing the tag name as a string literal instead of a numeric snapshot ID. Trino resolves the named ref via the `$refs` table to its snapshot and reads that state. This makes audit queries human-readable and stable, since the tag name outlives anyone's memory of the numeric ID.

Q: How do you create a tag on an Iceberg table in Trino, and what is the WAP pattern it supports?
A: You run `ALTER TABLE nav EXECUTE create_branch('audit')` for branches and the analogous tag creation (`create_tag`) where supported, or create refs via the engine's procedures. The Write-Audit-Publish pattern uses a branch: you write new data to an `audit` branch, run validation queries against it in isolation, and only `fast_forward` the main ref to publish once checks pass. This keeps unvalidated NAV data invisible to readers until it is approved.

Q: Why does Iceberg snapshot isolation mean a long-running NAV report never sees a mid-run correction?
A: The report binds to the snapshot that was current when it started (the immutable `metadata.json` it read), and that snapshot's data files are never altered by later commits — a correction just adds a new snapshot and swaps the pointer forward. So the report reads a frozen, consistent state for its whole duration. This is the same property that makes time travel and reproducibility possible: snapshots are immutable file sets.

Q: Two Spark jobs both MERGE into the same Iceberg table concurrently; the second fails with a commit conflict — what is happening and what is the fix?
A: Iceberg uses optimistic concurrency: the second writer built its commit on a metadata version that is no longer current after the first committed, so its atomic pointer swap fails with a validation/commit conflict. The fix is retry — re-read the latest metadata and re-apply the MERGE — which engines do automatically up to `commit.retry.num-retries` (default 4) with backoff. For chronic conflicts, partition writes so jobs touch disjoint partitions, reducing serialization contention.

Q: What table properties tune Iceberg's commit retry behaviour, and why raise them for a busy table?
A: `commit.retry.num-retries` (default 4), `commit.retry.min-wait-ms`, and `commit.retry.max-wait-ms` control how many times and how long a conflicting commit retries with exponential backoff before giving up. On a busy table with many concurrent writers you raise the retry count and waits so transient pointer-swap losses self-heal instead of failing the job. They trade a bit of latency for far fewer surfaced commit-conflict errors.

Q: How would you reduce write conflicts between a streaming upsert job and a nightly compaction on the same table?
A: Schedule compaction in a window when the stream is quiesced, or scope compaction with a `WHERE` clause to older partitions the stream no longer writes, so the two never contend for the same files. Iceberg's conflict detection is at the level of files/partitions a commit validates against, so disjoint partitions commit without conflict. Aligning each writer to its own partition range is the standard way to let maintenance and ingestion coexist.

Q: Contrast how Iceberg and Delta Lake physically represent merge-on-read deletes.
A: Iceberg writes separate position-delete or equality-delete files referenced in manifests, while Delta Lake uses deletion vectors — per-data-file compressed bitmaps (RoaringBitmap) marking deleted row positions, enabled via `delta.enableDeletionVectors`. Both avoid rewriting whole data files for small changes, but Iceberg's delete files are first-class manifest entries whereas Delta attaches a bitmap to each affected file. The conceptual goal — cheap writes, merge at read — is the same.

Q: In Delta Lake, which versions added deletion-vector support for DELETE, UPDATE, and MERGE, and what is the protocol catch?
A: DELETE with deletion vectors arrived in Delta 2.4, and UPDATE/MERGE support followed in the Delta 3.x line; enabling them upgrades the table's reader/writer protocol. The catch is that protocol upgrades are not reversible and older Delta clients that don't understand deletion vectors can no longer read the table. So enabling MOR in Delta is a one-way compatibility decision, much as relying on Iceberg v2 delete files requires v2-capable readers.

Q: How does Delta's checkpoint cadence affect cold-read latency, and how do you tune it?
A: Delta replays JSON commits since the last Parquet checkpoint, so a long gap between checkpoints means more JSON to replay on a cold read; the default writes a checkpoint every 10 commits via `delta.checkpointInterval`. Lowering the interval shortens replay at the cost of more checkpoint writes; raising it does the opposite. This is the Delta analogue of Iceberg's metadata-file accumulation — both formats need metadata hygiene tuned to commit frequency.

Q: Why can a Delta table's `_delta_log` grow huge even with checkpoints, and what cleans it?
A: Old JSON commit files and superseded checkpoints accumulate in `_delta_log` until log retention expires them; `delta.logRetentionDuration` (default 30 days) and `VACUUM` (governed by `delta.deletedFileRetentionDuration`, default 7 days) reclaim log and orphaned data files respectively. Time travel only works within the log-retention window, so shrinking it trades history for size. This mirrors Iceberg's `expire_snapshots`/`remove_orphan_files` split between metadata and data cleanup.

Q: What does Delta UniForm generate, and what is the operational cost of relying on it for Iceberg readers?
A: UniForm asynchronously writes Iceberg (and optionally Hudi) metadata alongside the Delta log when the table is written, so Iceberg-native engines can read the same Parquet files. The cost is the metadata is generated asynchronously, so Iceberg readers may lag the latest Delta commit, and not every Delta feature is mirrored — deletion vectors, for instance, historically constrained UniForm. You treat the Iceberg view as eventually-consistent and verify feature compatibility before depending on it.

Q: For an estate standardising on Databricks but needing Snowflake to read the same tables, how does the UniForm-vs-native-Iceberg choice play out in the ADR?
A: One option is Delta with UniForm — keep Databricks-native authoring and expose an Iceberg read view to Snowflake, accepting async metadata lag and feature gaps; the other is authoring directly in Iceberg via Databricks/Snowflake managed Iceberg, accepting less Delta-specific tooling. The ADR weighs which engine owns writes, freshness requirements for the Snowflake read path, and which advanced features (deletion vectors, liquid clustering) you depend on. You document the forces and pick, rather than assuming UniForm is free interoperability.

Q: Where does Apache Hudi's record-level index fit, and why does it make upserts fast?
A: Hudi maintains an index mapping each record key to the file (and file group) that currently holds it, so an upsert can locate the target file directly instead of scanning to find the row. This is why Hudi excels at high-frequency keyed upserts and incremental pulls from streaming ingestion. Iceberg and Delta lack a built-in per-record index of this kind, which is Hudi's distinguishing design point at awareness level.

Q: What is an "incremental query" in the table-format world and which format pioneered it?
A: An incremental query returns only the records that changed between two commits/snapshots rather than a full table scan, enabling efficient downstream propagation of just the deltas. Hudi pioneered first-class incremental pulls; Iceberg offers an analogous capability via incremental scans between snapshot ranges, and Delta via Change Data Feed. For a NAV pipeline this lets a downstream consumer pull only corrected/late rows since its last read.

Q: How would you implement change-data capture out of an Iceberg table for a downstream consumer?
A: Use an incremental scan between two snapshot IDs (the consumer's last-seen and the current), which yields the appended/changed files since then, or in Spark the `changelog`/incremental read APIs that emit insert/update/delete rows. The consumer records the snapshot ID it last processed and advances it each run, giving exactly-once-style delta propagation. This is the Iceberg counterpart to Delta's Change Data Feed and Hudi's incremental query.

Q: A query against an Iceberg table is slow even though files are well-sized — what metadata-side cause should you check next?
A: Check manifest health: too many manifests, or manifests not clustered by partition, inflate planning time even with healthy data files, so query `$manifests` for high counts and run `optimize_manifests` to cluster and shrink them. Also confirm column statistics exist (no stats means no file pruning) and that the predicate is actually pushed down via `EXPLAIN`. Planning cost, not scan cost, is the usual culprit when data files look fine.

Q: How can you confirm predicate pushdown and file pruning are working on a Trino Iceberg query?
A: Run `EXPLAIN` / `EXPLAIN ANALYZE` and look for the filter pushed into the Iceberg table scan plus the number of splits/files actually read versus the table total; a pruned query reads far fewer files. You can cross-check by querying `$files` to count total files and comparing to the splits reported. If it reads everything, the predicate isn't pruning — suspect a non-pushdownable expression, missing stats, or overlapping file bounds.

Q: What is a "metadata-only" query and give an Iceberg example that returns instantly?
A: A metadata-only query is answered purely from manifests/metadata without scanning data files — for example `SELECT count(*) FROM nav` can be served from per-file `record_count` sums in the manifests, and `SELECT max(nav_date)` may be served from column upper bounds. These return near-instantly regardless of table size because no Parquet is read. They illustrate why accurate manifest statistics are operationally valuable beyond just pruning.

Q: Why might `SELECT count(*)` be served from metadata but `SELECT count(*) WHERE nav > 100` is not?
A: The unfiltered count sums the `record_count` each manifest already stores per file, needing no data read, but a count with a non-partition predicate generally requires opening files to evaluate the condition row by row (unless file bounds let it skip some files entirely). So filtered counts fall back to scanning the surviving files after pruning. Partition-aligned predicates can stay metadata-only; arbitrary value predicates usually cannot.

Q: After widening a column from `int` to `long` on Iceberg, why do old data files still read correctly without a rewrite?
A: Iceberg resolves columns by immutable field ID and permits only safe promotions, so when an old file's `int` values are read under the new `long` schema, each value is promoted upward without loss. The change is a metadata-only schema update; no existing file is touched. Because every representable `int` fits in a `long`, the reinterpretation is always lossless, which is exactly why the promotion is allowed while the reverse is not.

Q: Your team renamed `nav` to `net_asset_value` last month; a colleague's old query using `nav` now errors — what is going on at the field-ID level?
A: Iceberg kept the same field ID and only changed the column's name string, so the column itself is intact, but SQL resolves by current name — the old name `nav` no longer exists in the schema, hence the error. You can still time-travel to a pre-rename snapshot and the underlying file data is unchanged, but queries must use the current name `net_asset_value`. This is why renames are safe for data but require updating query text and any name-based code.

Q: How do you backfill the historical `lei` values after adding the column, and how does that differ from the schema change itself?
A: The `ADD COLUMN` was metadata-only (old rows read null), but populating history is a real data operation — an `UPDATE`/`MERGE` that joins the source of LEIs and writes corrected rows, producing a new snapshot. On COW this rewrites affected files; on MOR it writes delete-plus-data files. So the cheap schema change and the expensive backfill are two distinct steps, and only the backfill creates new data and consumes write capacity.

Q: A regulator asks you to prove a NAV figure submitted on 2026-05-31 — what exact Iceberg steps reproduce it?
A: Query `nav$snapshots ORDER BY committed_at` (or `nav$refs` for a month-end tag) to find the snapshot current at the 2026-05-31 publication time, then run the original report with `FOR TIMESTAMP AS OF TIMESTAMP '2026-05-31 17:00:00.000 Europe/Luxembourg'` or `FOR VERSION AS OF <snapshot_id>`. Because that snapshot still references the original immutable data files, the figures reproduce bit-for-bit. Pinning the close with a long-retention tag beforehand guarantees the snapshot survives any expiry between then and the audit.

Q: How does Iceberg time travel underpin a DORA-style requirement to reconstruct data state after an incident?
A: DORA's operational-resilience expectations include being able to reconstruct and evidence data state around an incident; Iceberg snapshots give you an immutable, timestamped history you can query at any point, plus the snapshot summaries showing exactly what each commit changed. You can prove the state before and after an erroneous write and roll forward or back deterministically. The control is real only if expiry retention is set to cover the required reconstruction window.

Q: Why is matching snapshot-expiry retention to the T+1 settlement and correction window important for a fund table?
A: With a T+1 cadence, corrections and reconciliations for a trade date can land the next business day, so snapshots from at least the prior day's states must remain time-travellable to reproduce and validate figures across the correction window. If expiry retention is shorter than the settlement-plus-correction window, you lose the ability to reproduce the pre-correction state an auditor or counterparty may query. Retention must therefore be derived from the business settlement timeline, not a generic storage budget.

Q: What is the difference between `rollback_to_snapshot` and `set_current_snapshot` style recovery, and when do you choose rollback?
A: `rollback_to_snapshot` moves the table's current pointer to an ancestor snapshot to undo recent bad commits, requiring the target to be an ancestor of the current state; setting current snapshot more generally repoints to any valid snapshot. You choose rollback when a recent commit (a wrong NAV correction) must be undone and the prior good state is a direct ancestor still on storage. Both are fast metadata pointer moves, not data restores, provided the target snapshot's files weren't expired.

Q: An Iceberg `optimize` job failed midway — are partial results visible, and what cleanup is needed?
A: No partial results are visible: `optimize` commits atomically via the pointer swap, so a failure before commit leaves the table on its previous snapshot unchanged, and any files the job wrote are simply unreferenced. Those leftover files are orphans, cleaned later by `remove_orphan_files` once they exceed the conservative `retention_threshold`. This atomicity is why a botched maintenance run wastes some storage but never corrupts the table's logical state.

Q: On Ubuntu, what command-line tool would you reach for to inspect an Iceberg manifest's Avro contents directly off MinIO?
A: You use `avro-tools tojson <file>.avro` (from the Apache Avro tools jar) or the `mc cat` MinIO client piped into an Avro reader to dump a manifest or `snap-*.avro` manifest list as JSON. This is necessary because manifests are Avro, not Parquet, so a Parquet tool like `parquet-tools` will not read them. Inspecting the JSON lets you confirm per-file partition values, bounds, and the data/delete `content` codes by hand during deep debugging.

Q: Why is `EXPLAIN ANALYZE` more useful than `EXPLAIN` when diagnosing a slow Iceberg scan, and what number do you focus on?
A: `EXPLAIN` shows the planned operators and pushed-down filters, but `EXPLAIN ANALYZE` actually runs the query and reports real input rows, bytes, and splits per stage, so you see how many files/bytes were truly read versus the table total. You focus on the table-scan input rows/bytes: if it read nearly the whole table despite a selective predicate, pruning failed and you investigate stats, overlapping file bounds, or a non-pushdownable expression. Planned-versus-actual divergence is exactly what pinpoints a pruning problem.
