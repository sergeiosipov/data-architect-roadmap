# Phase 6 · 10.4.3 Data profiling — 150 self-test questions

Q: What is data profiling in one sentence?
A: Data profiling is the systematic examination of a dataset to discover its actual structure, content, and quality characteristics — distributions, null rates, cardinalities, value patterns, and inter-column relationships. It is descriptive, not corrective: it tells you what the data really looks like so you can decide what rules to enforce. Think of it as gathering evidence about a fund holdings table before you write a single data-quality assertion against it.

Q: Why is profiling described as the "evidence-gathering step before DQ rules"?
A: Because data-quality rules encode assumptions about the data, and profiling replaces guessed assumptions with observed facts. If you write a rule without profiling first, you may encode a myth — for example asserting an `ISIN` column is always populated when 2% are vendor placeholders. Profiling lets you discover that 2% first, so your rule reflects reality and does not generate a flood of false alarms.

Q: What is the practical danger of skipping profiling and writing DQ rules from assumptions?
A: Your rules encode myths instead of facts, producing false positives that fire on legitimate data. Each false alarm forces an analyst to investigate a non-issue, and after enough of them people start ignoring the DQ gate entirely — the classic "alert fatigue" failure. A DQ system that is distrusted is worse than none, because it gives a false sense of coverage while real defects slip through.

Q: Name the five core categories of profiling output highlighted in this lesson.
A: Distributions, null rates, uniqueness, value patterns, and drift over time. Distributions show the shape and range of values; null rates show completeness; uniqueness shows cardinality and key candidacy; value patterns reveal formats and regex-like structure; and drift over time shows how all of these change between snapshots. Together they describe both the static state and the dynamic behaviour of a column.

Q: What does a column's "distribution" tell you in a profile?
A: The distribution describes how values are spread — minimum, maximum, mean, median, quantiles, histogram shape, and outliers for numerics, or the frequency of each category for discrete data. It tells you the plausible range and the typical value, which is exactly what you need to set bounds. For a NAV-per-share column you might learn most funds sit between 8 and 15, with a long tail, before you decide what counts as an implausible value.

Q: What is a "null rate" and why does it matter for DQ?
A: The null rate is the fraction of rows where a column has no value (NULL, empty, or a sentinel). It matters because completeness is a core DQ dimension, and the observed rate tells you whether "not-null" is a realistic rule or one that would fire constantly. A 0% null rate justifies a strict not-null check; a stable 3% rate suggests a tolerance threshold instead of a hard fail.

Q: What does "uniqueness" (cardinality) reveal during profiling?
A: Uniqueness measures how many distinct values a column holds relative to its row count. A column that is 100% distinct is a primary-key candidate; a low-cardinality column (a handful of distinct values) is likely a category or status code. Profiling cardinality tells you which columns can serve as keys and which need referential or allowed-value checks rather than uniqueness checks.

Q: What are "value patterns" in profiling output?
A: Value patterns are the recurring structural shapes of string values — lengths, character classes, and regex-like templates that most values conform to. Profiling can reveal that a code column is consistently 12 alphanumeric characters, hinting it is an `ISIN`, or that dates arrive as `YYYY-MM-DD`. Patterns expose format violations and let you generate format-validation rules without hand-specifying every case.

Q: What is "drift over time" in a profiling context?
A: Drift is the change in a column's statistical profile between two snapshots — a shift in mean, a widening range, a jump in null rate, or a change in the set of categories. It is the signal that something upstream changed: a new feed format, a counterparty onboarding, or a broken transformation. Detecting drift requires comparing profiles across time rather than looking at a single snapshot in isolation.

Q: Why is a single snapshot profile insufficient for ongoing monitoring?
A: A single snapshot tells you the state at one moment but says nothing about change, and most production data problems manifest as a change from a previously-healthy baseline. To catch drift you must profile repeatedly and compare consecutive profiles, so you can alert when this week's distribution diverges from last week's. Continuous, mergeable profiling tools exist precisely to make that comparison cheap.

Q: How does profiling differ from data-quality testing?
A: Profiling is open-ended discovery — it answers "what does this data look like?" without a pass/fail verdict, whereas DQ testing checks "does this data meet a defined expectation?" and returns a result. Profiling has no notion of correct or incorrect; it just reports. DQ tests are the assertions you derive afterward, often informed directly by what profiling surfaced.

Q: What is ydata-profiling, in one line?
A: ydata-profiling is an open-source Python library that generates a comprehensive exploratory data analysis (EDA) report from a pandas or Spark DataFrame with essentially one line of code. It produces distributions, null analysis, correlations, interactions, and an automated alerts section. It is the one-shot profiling tool in this lesson's toolkit, paired with whylogs for continuous monitoring.

Q: What was ydata-profiling formerly called?
A: It was formerly called `pandas-profiling`; the project was renamed to `ydata-profiling` to reflect support beyond pandas (including Spark) and its stewardship by YData. Older tutorials and the package `pandas-profiling` on PyPI now redirect or are deprecated in favour of `ydata-profiling`. If you see `import pandas_profiling`, it is legacy code that should be migrated to `from ydata_profiling import ProfileReport`.

Q: Write the minimal ydata-profiling code to profile a DataFrame and save an HTML report.
A: `from ydata_profiling import ProfileReport`, then `profile = ProfileReport(df, title="Holdings profile")`, then `profile.to_file("holdings.html")`. The `ProfileReport` constructor does the computation, and `to_file()` renders it; passing a `.html` path produces an interactive HTML report. The whole flow is genuinely one or two lines, which is the library's selling point.

Q: What does `ProfileReport(df, minimal=True)` change?
A: The `minimal=True` flag turns off the most expensive computations by default — notably correlations and interactions — producing a much faster, lighter report. It exists for large datasets where the full pairwise analysis is too slow or memory-hungry. You trade some depth (you lose correlation heatmaps and interaction plots) for speed and scalability.

Q: Which output formats does `ProfileReport.to_file()` support?
A: `to_file()` supports HTML and JSON, chosen by the file extension you pass — `to_file("report.html")` renders the interactive report, `to_file("report.json")` emits the structured statistics. The JSON form is the one you parse programmatically to drive rule generation. HTML is for humans; JSON is for machines.

Q: How do you display a ydata-profiling report inline in a Jupyter notebook?
A: Call `profile.to_notebook_iframe()`, which embeds the full HTML report inside an iframe within the notebook output cell. The iframe isolates the report's styling and scripts from the notebook so the heavy HTML does not break the page. Alternatively `to_widgets()` exists in some versions, but `to_notebook_iframe()` is the standard inline display path.

Q: What is the "alerts" section of a ydata-profiling report and why is it the most valuable part for DQ work?
A: The alerts section is an automatically generated list of potential data-quality issues — high correlation, high cardinality, constant values, high zeros, missing values, skewness, uniformity, and duplicates. It is the most valuable part for DQ because it pre-filters thousands of statistics down to the handful that warrant human attention. Reviewing alerts is the fastest route from a raw profile to a candidate list of rules.

Q: List several alert types ydata-profiling can raise.
A: It can raise alerts such as `HIGH_CORRELATION`, `HIGH_CARDINALITY`, `CONSTANT`, `ZEROS`, `MISSING`, `SKEWED`, `UNIFORM`, `UNIQUE`, `DUPLICATES`, and `INFINITE`, among others. Each flags a statistical property that often signals either a data issue or a modelling consideration. For example a `CONSTANT` alert on a column that should vary tells you a transformation may have collapsed it.

Q: A profile fires a `DUPLICATES` alert on your holdings table. What does it mean and what would you check first?
A: A `DUPLICATES` alert means ydata-profiling found exact-match duplicate rows in the dataset. First check whether the duplication is legitimate (genuinely identical line items) or a join fan-out / double-load bug, by inspecting a sample of the duplicated rows and the grain you expected. In a holdings table the grain is usually one row per fund-security-date, so true exact duplicates almost always indicate a pipeline defect.

Q: A `CONSTANT` alert appears on a column you expected to vary. What does that tell you?
A: It tells you every row holds the same single value, which for a column you expected to vary is a strong signal that an upstream transformation, filter, or default override has collapsed it. A common cause is a hard-coded default leaking into production or a join that matched only one partition. You would trace the column back through the pipeline to find where the variability was lost.

Q: How does ydata-profiling report correlations, and when are they suppressed?
A: It computes pairwise correlations between columns (Pearson, Spearman, and categorical measures like Cramér's V) and renders them as heatmaps, raising a `HIGH_CORRELATION` alert for strongly related pairs. They are suppressed when you pass `minimal=True`, because correlation is one of the most expensive computations on wide datasets. High correlation can flag redundant columns or a leak between fields that should be independent.

Q: What does the ydata-profiling "duplicates" table show beyond the alert?
A: Beyond raising the alert, ydata-profiling produces a table of the most frequent exactly-duplicated rows so you can inspect concrete examples. This lets you judge whether duplicates are benign or a bug without writing a separate query. Seeing the actual repeated rows is often the fastest way to diagnose a join fan-out.

Q: How can you compare two datasets with ydata-profiling?
A: Build a `ProfileReport` for each DataFrame and call `compare([report_a, report_b])` (or `report_a.compare(report_b)`), which produces a single side-by-side comparison report. This is how you visualise drift between, say, this week's and last week's holdings extract within ydata-profiling itself. It overlays distributions and statistics so divergences stand out.

Q: ydata-profiling runs out of memory on a 50-million-row table. What are your options?
A: First try `minimal=True` to drop correlations and interactions, the biggest cost; then profile a representative sample rather than the full table, since profiling is descriptive and a good sample reproduces the distribution. You can also profile via the Spark backend if the data is already in Spark, or push profiling down to whylogs, which is built for streaming and large-scale logging. Profiling does not require every row to be useful.

Q: Can ydata-profiling profile Spark DataFrames as well as pandas?
A: Yes — ydata-profiling added Spark DataFrame support (part of why it was renamed from pandas-profiling), so you can profile distributed datasets without first collecting them to the driver. The Spark path supports a subset of the full pandas features, so some sections may be reduced. For very large fund estates this avoids pulling the whole table into memory.

Q: What is whylogs, in one line?
A: whylogs is an open-source data-logging library that produces lightweight, mergeable statistical profiles of datasets for continuous monitoring of data quality and drift. Unlike ydata-profiling it is built for ongoing, scheduled logging at scale rather than one-shot human-readable EDA. It is the "continuous monitoring" half of this lesson's toolkit.

Q: What does it mean that whylogs profiles are "mergeable"?
A: Mergeability means two whylogs profiles can be combined into a single profile that statistically represents both datasets together, as if they had been logged at once. This is possible because whylogs stores compact summaries (sketches and aggregates), not raw rows, so merging is an aggregation rather than a re-scan. It enables profiling distributed and streaming systems and rolling up daily profiles into a weekly or monthly view.

Q: Why does mergeability make whylogs suitable for distributed and streaming pipelines?
A: Because each worker or each micro-batch can log its own profile independently, and those partial profiles are merged afterward into one correct aggregate without re-reading the data. There is no need to centralise all rows before profiling, which is exactly the constraint in distributed and streaming systems. The merge is associative and commutative, so order and partitioning do not matter.

Q: Why are whylogs profiles called "lightweight"?
A: Because they store statistical summaries — counts, null counts, cardinality sketches, quantile sketches, frequent-item sketches — rather than the raw data, so a profile is small and roughly constant-size regardless of how many rows it summarises. This makes them cheap to store, transmit, and retain over time. You can keep a year of daily profiles in the space a single raw extract would take.

Q: Write the basic whylogs call to profile a pandas DataFrame and get a profile view.
A: `import whylogs as why`, then `results = why.log(df)`, then `profile_view = results.profile().view()`. `why.log()` produces a results object, `.profile()` extracts the `DatasetProfile`, and `.view()` gives the immutable `DatasetProfileView` you use for inspection, merging, and serialization. `profile_view.to_pandas()` then renders the summary statistics as a DataFrame.

Q: How do you persist a whylogs profile to disk and read it back?
A: Write with `why.write(profile, "profile.bin")`, which serialises the profile to a binary `.bin` file on the local filesystem, and read with `why.read("profile.bin")`. A gotcha: `why.write` expects a profile, but `why.read` returns a `DatasetProfileView`, meaning the loaded object can be visualised and merged but cannot be further updated with new data. The `.bin` files are what you accumulate over time for trend analysis.

Q: What is the `.bin` file in a whylogs workflow?
A: The `.bin` file is whylogs' serialized binary profile — the on-disk form of a `DatasetProfileView` produced by `why.write(profile, "name.bin")`. It is the durable artifact you store per run so you can later read it back with `why.read()` and compare or merge it with other runs. Storing one `.bin` per day is the standard pattern for building a drift history.

Q: How do you merge two whylogs profile views?
A: You combine them with the merge operation on the views — typically `merged = profile_view_a.merge(profile_view_b)` — which returns a new view representing the aggregate of both. Because the merge is statistically exact for the summarised metrics, the result is identical to having logged both datasets together. This is how you roll daily profiles up into a weekly profile.

Q: Why would you store a profile per run rather than one giant profile?
A: Storing one profile per run preserves the time dimension, so you can diff consecutive runs to detect drift and attribute a change to a specific load. A single merged-forever profile loses that granularity — you can no longer tell when a shift happened. Mergeability still lets you combine per-run profiles on demand for any time window, so you lose nothing by keeping them separate.

Q: What is the difference in intended use between ydata-profiling and whylogs?
A: ydata-profiling is for one-shot, human-facing exploratory analysis — rich HTML reports you read with your eyes during development. whylogs is for automated, continuous, machine-facing monitoring — compact profiles you log on a schedule and diff programmatically. You typically use ydata-profiling to understand a new table once, then whylogs to watch it forever after.

Q: When would you reach for whylogs over ydata-profiling?
A: Reach for whylogs when you need scheduled, ongoing profiling at scale with cheap storage and the ability to merge and diff profiles across time or partitions — for example weekly drift monitoring of every silver table in a fund estate. ydata-profiling would be too heavy and too human-oriented for that loop. Conversely, for a one-time deep look at a single new feed, ydata-profiling's rich report wins.

Q: What are whylogs "constraints"?
A: Constraints are a validation feature built on top of whylogs profiles that let you assert that a profile's metrics meet expectations — for example a numeric column staying within a range, a null count staying below a threshold, or a column matching a format. They turn a descriptive profile into a pass/fail check, bridging profiling and DQ enforcement inside whylogs itself. They are conceptually the same idea as Soda checks, expressed against the profile rather than the raw table.

Q: How do whylogs constraints relate to drift detection?
A: Constraints assert against absolute thresholds on a single profile, while drift detection compares two profiles to see whether the distribution moved. You often use both: constraints catch values that breach known bounds, and drift checks catch a subtler shift that stays within bounds but differs from the baseline. A column can pass every constraint yet still have drifted, which is why distribution comparison is a separate pillar.

Q: What is "profiling as a DQ-rule generator"?
A: It is the practice of turning observed profile statistics into concrete data-quality assertions — taking the min/max a profile reports and writing a range check, taking the observed null rate and writing a completeness threshold, taking the value set and writing an allowed-values check. The profile supplies the empirically-justified numbers so the rules reflect reality. It closes the loop from discovery (profiling) to enforcement (DQ rules).

Q: Give a concrete example of turning a profiling finding into a rule.
A: If profiling shows a NAV-per-share column has observed values between 0.50 and 5000 with no nulls, you generate a Soda check asserting the value is between sensible bounds and is never null. The bounds come from the data, not a guess, so the rule is tight enough to catch a misplaced decimal yet loose enough not to fire on legitimate values. You would still pad the observed bounds slightly to allow for plausible future values.

Q: What is Soda and why is it the target of generated rules in this lesson?
A: Soda (Soda Core / SodaCL) is the open-source data-quality framework used elsewhere in this phase to express checks as declarative YAML against tables. It is the enforcement layer this lesson feeds: profiling discovers the bounds, and you encode them as SodaCL checks that run in the pipeline's DQ gate. So profiling is the upstream evidence and Soda is the downstream gatekeeper.

Q: Write a SodaCL check that encodes an observed not-null and range finding on a NAV column.
A: In SodaCL it reads roughly `checks for fund_nav:` then `- missing_count(nav_per_share) = 0:` and `- invalid_count(nav_per_share) = 0:` with `valid min: 0.5` and `valid max: 5000` configured for the column. The `missing_count = 0` encodes the observed zero null rate, and the valid min/max encode the observed distribution bounds. The exact numbers trace directly back to the profile.

Q: Why should you pad observed bounds slightly when generating range rules?
A: Because a profile only captures the data seen so far, and legitimate future values may sit just outside the historical extremes — a new fund, a market move, a currency. Encoding the raw min and max as hard limits would make the rule brittle and prone to false fails on valid new data. Padding (or using a percentile-based bound) gives headroom while still catching gross errors like a misplaced decimal point.

Q: Why is a generated rule still subject to human review before deployment?
A: Because a profile cannot distinguish "this is normal" from "this is a pre-existing defect we have learned to live with" — it only reports what is, not what should be. If 2% of `ISIN` values are placeholders, naively profiling them into the allowed-value set would bless the defect forever. A human must decide whether each observed pattern is a fact to encode or a problem to fix.

Q: How does the "Done when" criterion about falsified assumptions test real understanding?
A: The criterion asks you to show at least three DQ rules that exist *because* profiling contradicted an assumption you held — proving you used profiling as discovery, not decoration. It forces a before/after: you state what you assumed, what the profile revealed, and the rule that resulted. If you cannot produce three, you likely confirmed your priors rather than testing them.

Q: Give an example of an assumption profiling might falsify in fund holdings data.
A: You might assume every holding row has a populated `ISIN`, but profiling reveals a 2% null rate because OTC derivatives and some money-market instruments legitimately lack an ISIN. That single finding flips your rule from a strict not-null to a conditional check that allows nulls only for specific instrument types. The assumption was a myth; the profile is the evidence that corrects it.

Q: What is the "PII-discovery overlap" mentioned in this lesson?
A: It is the observation that profiling, while looking at distributions and value patterns, also surfaces columns that look like personally identifiable information — emails, names, national IDs, account numbers — because those have recognisable patterns and high cardinality. Those flagged columns become candidate inputs to dedicated PII discovery and classification work (lesson 11.4.1). Profiling thus does double duty: data quality and a first pass at sensitivity tagging.

Q: How can value patterns from profiling hint that a column holds PII?
A: Because PII fields conform to recognisable patterns — emails match an `@`-and-domain regex, IBANs and national IDs have fixed length/checksum structure, and free-text name columns show high cardinality with no numeric structure. When profiling reports such a pattern or flags very high cardinality on a string column, it is a candidate sensitive column. You then route it to formal PII scanning rather than treating the profiler's hint as a final classification.

Q: Why is a profiler's PII hint only a "candidate", not a classification?
A: Because pattern matching produces false positives and false negatives — a column of order references can look like account numbers, and obfuscated PII may not match any known pattern. The profiler narrows the search space; a dedicated discovery tool with context, dictionaries, and validation makes the actual determination. Treating a profile hint as a final PII label risks both over- and under-classifying sensitive data.

Q: Which downstream lesson does PII-discovery profiling feed?
A: It feeds lesson 11.4.1, the dedicated PII discovery and classification work. Profiling in 10.4.3 produces the candidate list of sensitive columns; 11.4.1 takes those candidates and performs proper detection, classification, and tagging. The handoff is the value of doing profiling first — you arrive at PII work with a pre-narrowed set of columns to scrutinise.

Q: In the "Do" steps, what is the first action and what should you focus on in its output?
A: The first action is to run ydata-profiling on the silver holdings table and review the alerts section. You focus on the alerts because they are the automated triage of all the statistics down to the items most likely to be data-quality issues. From there you can identify the surprising findings that become rules.

Q: The lesson asks you to identify the "five most surprising findings". What makes a finding surprising?
A: A finding is surprising when it contradicts what you assumed — an unexpected null rate, a cardinality you did not predict, a range outside what you thought plausible, or a constant where you expected variation. Surprise is the signal that profiling earned its keep by correcting a belief. Findings that merely confirm expectations are not the ones that should drive new rules.

Q: What does the "schedule a weekly whylogs profile and configure a diff alert" step accomplish?
A: It operationalises continuous monitoring: each week a whylogs profile is logged and compared (diffed) against the prior week's profile, alerting when the distribution drifts beyond tolerance. This catches silent changes — a new feed format, a shifted distribution — that a one-shot profile would never see. It is the difference between profiling once and profiling forever.

Q: How would you implement a week-over-week whylogs diff in practice?
A: Each week run `why.log(df)`, take `results.profile().view()`, and `why.write()` it to a dated `.bin`; then load last week's `.bin` with `why.read()` and compare the two views for drift, raising an alert if a metric moves beyond a threshold. The `.bin` history is your time series of profiles. You schedule this with the same orchestrator (e.g. Airflow/cron) that runs the rest of the pipeline.

Q: A profiling run reports 0 nulls in a column that you know sometimes lacks data. What is the first thing to check?
A: Check whether the "missing" values are arriving as a sentinel rather than a true NULL — empty strings, `"N/A"`, `"NULL"` text, `0`, or a placeholder code that the profiler counts as a present value. Profilers measure NULL completeness, so sentinel-encoded missingness reads as 0% null while the data is effectively incomplete. You would profile the value frequencies to spot the sentinel, then decide how to canonicalise it.

Q: Profiling shows an `ISIN` column with cardinality far below the row count. Is that necessarily a problem?
A: Not necessarily — a holdings table has one row per fund-security-date, so the same security held across many funds and many dates legitimately repeats its `ISIN`, giving low cardinality relative to rows. The check is whether the cardinality matches the expected grain, not whether it is high. You would only worry if you expected `ISIN` to be unique (e.g. in a security master) and it was not.

Q: How can profiling reveal a broken join or fan-out in a pipeline?
A: A join fan-out inflates row counts and creates exact-duplicate rows, which profiling surfaces as a `DUPLICATES` alert and as a row count higher than the expected grain. It can also show as a sudden cardinality or distribution shift versus the prior profile. Catching this at the profile stage is far cheaper than discovering it when a downstream NAV is double-counted.

Q: Profiling a date column shows values in the year 1900 and 2099. What likely happened?
A: Those are almost certainly sentinel or default dates — `1900-01-01` and far-future dates are classic placeholders for "unknown" or "no end date" injected by source systems or ETL defaults. The profile's min/max surfaced them as the distribution extremes. You would treat them as encoded missingness, not real dates, when writing rules.

Q: Why is the distribution pillar especially important for a NAV or price column?
A: Because a NAV or price has a plausible range, and the most common serious error — a misplaced decimal, a currency mix-up, a stale price — manifests as a value far outside the normal distribution. Profiling the distribution gives you the empirical bounds to catch those without flagging legitimate values. A NAV that jumps from 12 to 1200 is exactly the kind of outlier a distribution-derived rule should trap.

Q: What is the relationship between profiling and the "distribution" data-observability pillar?
A: Profiling produces the per-column distribution statistics, and the distribution pillar of data observability monitors those statistics over time for anomalies. Profiling is the measurement; the distribution pillar is the continuous monitoring of that measurement. whylogs sits at the junction — it profiles and supports the cross-time comparison the pillar requires.

Q: How does profiling support setting the right DQ "severity" on a generated rule?
A: The observed stability of a metric informs severity: a column that is always 0% null justifies a high-severity failing check, while a metric that naturally fluctuates within a band justifies a lower-severity warning with a tolerance. Profiling shows you which is which by revealing the natural variability. Encoding a noisy metric as a hard fail is how you create the false alarms that erode trust.

Q: How does profiling help choose the correct DQ "dimension" for a rule?
A: The nature of the finding maps to a dimension: an unexpected null rate is a completeness issue, an out-of-range value is a validity/accuracy issue, a duplicate is a uniqueness issue, and a drifted distribution is a consistency/timeliness signal. Profiling categorises the finding, which tells you which DQ dimension the resulting rule belongs to. Tagging rules by dimension is what lets you report DQ coverage meaningfully.

Q: Can profiling alone tell you whether data is "correct"?
A: No — profiling tells you what the data *is*, not what it *should be*; correctness requires an external reference or business rule. A column can be perfectly profiled and internally consistent yet wrong, because the source itself is wrong. Profiling narrows where to look and supplies bounds, but a human or a golden source defines correctness.

Q: What is a representative sampling caveat when profiling huge tables?
A: Profiling a sample is valid for distributions and null rates because those are statistical, but it can miss rare events — a single corrupt row, an outlier `ISIN`, or a once-a-quarter anomaly — that a full scan would catch. So sample for distribution discovery, but be aware uniqueness and rare-defect detection may need the full table or a targeted query. Match the method to the question.

Q: How does whylogs handle high-cardinality columns without storing every value?
A: whylogs uses probabilistic sketches — for example HyperLogLog-style cardinality estimators and quantile sketches — to approximate distinct counts and percentiles in bounded memory. This is what keeps profiles lightweight and mergeable even for billions of rows. The trade-off is that cardinality and quantiles are accurate estimates rather than exact counts, which is acceptable for monitoring.

Q: Why does whylogs use sketches instead of exact statistics?
A: Sketches give bounded, predictable memory and CPU regardless of data volume, and crucially they are mergeable — two sketches combine into one that summarises both datasets. Exact distinct counts and exact quantiles cannot be merged without re-scanning all the raw data. The small loss of precision buys constant-size, distributed-friendly profiles, which is the whole point for continuous monitoring.

Q: whylogs reports an estimated cardinality just below your row count on a key column. Can you declare uniqueness from that estimate alone?
A: No — whylogs' cardinality comes from a probabilistic sketch with a small error margin, so an estimate near the row count is a strong hint but not proof of uniqueness, and it cannot detect the one duplicate that would break a key. Use the estimate to shortlist key candidates, then confirm with an exact uniqueness check (a `COUNT(DISTINCT)` or a Soda `duplicate_count = 0`) on the real table. The sketch narrows the search; the exact check makes the decision.

Q: What does `profile_view.to_pandas()` give you in whylogs?
A: It returns a pandas DataFrame where each row is a column of the original dataset and the columns are the computed metrics — counts, null counts, estimated cardinality, type counts, min/max, quantiles, and so on. It is the convenient, inspectable tabular view of a profile. You use it to eyeball a profile or to drive programmatic rule generation from the metrics.

Q: Profiling reveals two columns with `HIGH_CORRELATION` in your holdings data. Why might that matter?
A: High correlation can indicate redundancy (two columns carrying the same information, like `market_value` and `quantity * price`) or a data leak between fields that should be independent. For DQ it can surface a consistency rule — for instance asserting `market_value` equals `quantity * price` within a tolerance. It can also just be expected domain structure, so you interpret it with business context.

Q: How can profiling catch a currency or unit inconsistency?
A: Profiling the distribution of an amount column can reveal a bimodal or unexpectedly wide spread that signals mixed units or currencies — values clustering around two scales. Profiling a currency code column's value frequencies can reveal an unexpected currency creeping in. Either way the anomaly shows up as a distribution or value-set surprise that a single-currency assumption would have missed.

Q: Why is profiling the silver layer (rather than bronze or gold) a sensible default in this lesson?
A: The silver layer is cleaned and conformed but not yet aggregated, so its columns are at a grain where DQ rules are most meaningful and where profiling findings translate directly into checks before data is served. Bronze is too raw and noisy to derive stable rules from, and gold is aggregated so individual-record issues are already hidden. Silver is where profiling-to-rule generation has the most leverage.

Q: How does profiling output feed both the DQ gate and the data catalog?
A: Profiling statistics — null rates, distributions, cardinalities, value patterns — both seed DQ rules and enrich catalog/column metadata that documents the data for consumers. The same null-rate measurement that drives a completeness check also tells a catalog user how complete the column typically is. Profiling is thus a shared upstream input to quality enforcement and to discovery/documentation.

Q: What is "alert fatigue" and how does profiling reduce it?
A: Alert fatigue is the desensitisation that occurs when a monitoring system fires too many false or low-value alerts, leading people to ignore all of them. Profiling reduces it by grounding rule thresholds in observed reality, so rules fire on genuine anomalies rather than on normal variation the author mis-guessed. Fewer false positives means each alert retains its signal value.

Q: Profiling shows a column is 99.9% one value and 0.1% another. What kinds of rule might this suggest?
A: It suggests either an allowed-values / category check (if the rare value is legitimate) or an anomaly threshold (if the rare value is suspect). You might encode a SodaCL check that the rare value appears in fewer than a small fraction of rows, alerting if it ever spikes. Profiling gave you both the value set and the baseline frequency to set that threshold.

Q: How would you profile to discover candidate primary keys?
A: Look at uniqueness/cardinality: a column (or column combination) whose distinct count equals the row count and has zero nulls is a primary-key candidate. ydata-profiling flags `UNIQUE` columns, and whylogs' estimated cardinality versus row count approximates the same. You then validate the candidate with an actual uniqueness check before declaring it a key.

Q: Why might profiling a column report a `UNIFORM` alert, and what does it imply?
A: A `UNIFORM` alert means the values are spread evenly with no dominant value, which for an identifier or surrogate key is expected but for a measured quantity is unusual. It can imply the column is an artificial sequence or that real signal has been flattened. You interpret it against what the column is supposed to represent.

Q: How does profiling relate to the broader data-quality lifecycle in this phase?
A: Profiling is the discovery stage that precedes rule definition (Soda checks), which precedes monitoring (whylogs/Elementary observability), which precedes incident response. It is the empirical foundation: every well-grounded rule and threshold downstream traces back to something a profile measured. Skipping it means the rest of the lifecycle is built on assumptions.

Q: Why is whylogs described as "privacy-preserving" data logging?
A: Because whylogs profiles store statistical summaries and sketches, not the raw values, so a profile can describe a dataset's shape without retaining or exposing the underlying records. You can share a profile for monitoring without shipping sensitive rows like investor PII. This makes it safer to centralise profiles for cross-environment monitoring than to centralise the raw data.

Q: In a regulated fund-administration context, why is profile retention itself useful for audit?
A: Because a dated series of whylogs profiles is a compact, privacy-preserving record of what the data looked like at each point in time, which can evidence that monitoring was in place and demonstrate when a distribution shifted. Under regimes emphasising operational resilience and data governance, being able to show a drift history supports audit and incident reconstruction. The `.bin` files become an evidentiary trail without storing the underlying client data.

Q: How does profiling support DORA-style operational-resilience expectations on data pipelines?
A: DORA pushes firms to detect, manage, and evidence ICT and data incidents, and continuous profiling provides the detection and the evidence — drift alerts catch silent data failures and the profile history documents that monitoring operated. It does not satisfy DORA by itself, but it is a concrete control that contributes to the data-integrity and incident-detection expectations. Profiling turns "we monitor our data" from a claim into an artifact.

Q: A weekly whylogs diff alerts on a sudden jump in the null rate of a `LEI` column. What is your triage?
A: First confirm the alert is real (not a sampling artifact) by inspecting the actual null count this week versus baseline; then trace upstream to whether a feed changed, a source stopped populating LEIs, or a join now misses matches. A `LEI` null jump often means a counterparty reference feed broke or a new counterparty was onboarded without an LEI. You resolve by fixing the source/join and deciding whether to gate downstream loads.

Q: Why is profiling an `EMT` or `EPT` file before ingestion valuable?
A: European MiFID/PRIIPs template files (EMT/EPT) have many fields with strict expected formats and code lists, and profiling a received file reveals nulls, unexpected codes, and out-of-range values before you load and rely on it. It catches a malformed or partially-populated template at the door rather than after it has polluted downstream reporting. The profile becomes the basis for validation rules on each incoming template.

Q: How could profiling catch a stale-price problem that produces yesterday's NAV?
A: Profiling the as-of/price-date column's distribution and the price distribution over time would show that today's load carries prices dated to yesterday, or that the price distribution is identical to the prior day's — a drift signal of staleness. A freshness-style check derived from the profile (max price date equals today) would fail. Profiling surfaces the staleness that a naive "row count is normal" check would miss.

Q: Distinguish what a one-shot ydata-profiling report and a whylogs diff each catch in a NAV pipeline.
A: The ydata-profiling report catches static issues in a single extract — an out-of-range NAV, an unexpected null, a duplicate — when you read it during development. The whylogs diff catches dynamic issues — a NAV distribution that shifted versus last week, a creeping null rate, a new currency — that only appear as change over time. One is a snapshot inspection; the other is a moving-baseline alarm.

Q: Why can a column pass a static range rule yet still warrant a drift alert?
A: Because the range rule only checks absolute bounds, while drift compares the whole distribution against a baseline — values can stay inside the allowed range but cluster differently, signalling an upstream change. For example NAVs all staying between 8 and 15 but suddenly bunching at 8 indicates something shifted even though no bound is breached. Static rules and drift detection are complementary, not redundant.

Q: What is the risk of generating a DQ rule directly from a profile without business context?
A: You may bless a pre-existing defect as the norm (e.g. encoding placeholder `ISIN`s into the allowed set) or set a bound around a population that includes erroneous outliers. The profile reports what is, including its flaws, so blind automation propagates those flaws into your rules. Business context is what separates "this is a fact to encode" from "this is a problem to fix".

Q: How do you decide whether an observed anomaly becomes a hard failing rule or a soft warning?
A: Judge the business impact and the metric's natural variability: an anomaly that would produce a wrong client-facing number (a bad NAV) warrants a hard fail that blocks the load, while a low-impact or naturally-noisy signal warrants a warning that flags for review. Profiling shows the variability; the business defines the impact. Mismatching these — hard-failing on noise, or warning on a NAV error — is the common mistake.

Q: Why is profiling considered cheaper insurance than debugging a downstream NAV error?
A: Because catching a defect at the profile stage costs minutes, whereas a wrong NAV that reaches a client triggers restatement, regulatory exposure, and loss of trust — orders of magnitude more costly. Profiling shifts detection left, to the cheapest point in the pipeline. The few hours of profiling effort pay for themselves the first time they prevent a published error.

Q: What is the difference between `why.log()` returning a result and `.profile().view()`?
A: `why.log(df)` returns a results object that wraps the logging run; `.profile()` extracts the mutable `DatasetProfile` (which can still be updated with more data), and `.view()` produces the immutable `DatasetProfileView` used for inspection, serialization, and merging. The view is the artifact you persist and compare; the profile is the live accumulator. Reading a `.bin` back gives you a view, not a profile, so it cannot be further updated.

Q: How would you profile incrementally as data streams in, with whylogs?
A: You log each batch as it arrives, keep accumulating into a `DatasetProfile`, and periodically snapshot a view to disk — or log per-batch profiles and merge them. Because profiles are mergeable, the streaming case is just repeated logging plus merge, with no need to retain raw events. This is exactly the streaming-system use case mergeability is designed for.

Q: Why might profiling output differ between a pandas run and a Spark run of ydata-profiling?
A: The Spark backend supports a reduced feature set compared to pandas — some computations (certain correlations, interactions, detailed text analysis) may be unavailable or approximated to remain distributed. So the Spark report can be lighter and omit sections the pandas report includes. You account for this when comparing reports produced by different backends.

Q: How does profiling interact with schema-drift detection?
A: Profiling captures the set and types of columns present, so comparing profiles across runs reveals added, removed, or retyped columns — the schema-drift pillar. A column appearing or vanishing between two profiles is a structural change distinct from a value-distribution change. whylogs profiles naturally encode this because each column's metrics are keyed by column name and inferred type.

Q: A profile shows a numeric column has a non-trivial count of string-typed values. What does that indicate?
A: It indicates mixed types in what should be a numeric column — typically a few rows where the value arrived as text (a code, an error marker, a thousands-separator string) and was not coerced. Profilers report per-column type counts, so a numeric column with a string-type tally is a clear data-quality flag. You trace those rows and add a type/validity check.

Q: Why is the alerts section better than reading every statistic manually?
A: Because a full profile of a wide table contains thousands of numbers, far too many to scan reliably, whereas the alerts section applies heuristics to elevate the few that commonly indicate problems. It is automated triage that focuses scarce human attention. You still read the detail for flagged columns, but you let the alerts decide where to look first.

Q: How do you keep a profiling-derived rule from going stale as the data legitimately evolves?
A: Re-profile periodically and treat the rules as living artifacts — when a drift alert reflects a legitimate business change (a new fund range, a new currency), you update the bound rather than suppress the alarm. Continuous whylogs profiling is what tells you when a rule's assumptions have aged out. The diff that flags drift is also the signal to revisit the rule.

Q: What is the conceptual loop connecting profiling, Soda, and whylogs in this lesson?
A: ydata-profiling discovers the current reality of the data and surfaces surprises; you translate those surprises into Soda checks that enforce expectations on each load; and whylogs continuously re-profiles to detect when reality drifts away from those expectations, prompting you to revisit the checks. Discovery feeds enforcement feeds monitoring feeds discovery again. It is a closed quality loop, not a one-time setup.

Q: Why is "uniqueness" alone insufficient to declare a primary key?
A: Because a column can be unique in the current snapshot by coincidence and collide later, and uniqueness ignores nulls — a key must also be non-null and stable across loads. Profiling flags the candidate, but you confirm with an enforced uniqueness-plus-not-null constraint and domain reasoning about whether the value is truly an identifier. Treating a coincidentally-unique column as a key is a classic modelling error.

Q: How would you present profiling findings to a non-technical fund-operations stakeholder?
A: Lead with business-relevant surprises — "2% of holdings have no ISIN, here is why and here is the rule we added" — rather than raw histograms, and tie each finding to a risk it mitigates (a wrong NAV, a failed regulatory file). The ydata-profiling HTML helps, but the narrative of falsified assumptions and resulting controls is what lands. Profiling is most persuasive when framed as evidence behind specific protections.

Q: What single sentence captures why this lesson sits before DQ-rule lessons rather than after?
A: Because you cannot write trustworthy data-quality rules until you have measured what the data actually is, and profiling is that measurement — it converts assumptions into observed bounds, null rates, and value sets that the rules then encode. Profiling first means your DQ gate reflects reality; profiling never means it reflects guesswork. Evidence precedes enforcement.

Q: On Ubuntu, how would you install both profiling tools into a `uv`-managed project?
A: Run `uv add ydata-profiling whylogs` from the project root, which resolves and pins both into the project's virtual environment and `pyproject.toml`. You then run scripts with `uv run python profile_holdings.py` so the interpreter and dependencies are the locked ones. Avoid `pip install` into a system Python on Ubuntu, since that risks an externally-managed-environment error (`PEP 668`) and pollutes the OS interpreter.

Q: You `import pandas_profiling` on a fresh Ubuntu box and get `ModuleNotFoundError`. What is the fix?
A: The package was renamed, so install and import the new name — `uv add ydata-profiling` then `from ydata_profiling import ProfileReport` — rather than the legacy `pandas_profiling`. The old PyPI package is deprecated and will not be present in a current environment. Migrating the import line is a one-token change but is easy to miss when copying old tutorials.

Q: How do you disable only the correlations computation in ydata-profiling while keeping interactions?
A: Pass a config override such as `ProfileReport(df, correlations={"auto": {"calculate": False}})`, which switches off the correlation block specifically rather than using the blunt `minimal=True` that drops both correlations and interactions. This is the granular alternative when correlations are the bottleneck but you still want interaction plots. The per-section `calculate` flags let you tune cost section by section.

Q: How would you cap ydata-profiling's parallelism to avoid saturating a shared Ubuntu host?
A: Set `ProfileReport(df, pool_size=N)` to fix the number of worker processes, or `pool_size=1` to force single-threaded computation. By default `pool_size=0` means whylogs uses `multiprocessing.cpu_count()`, which can hog every core on a shared box. Capping it keeps a profiling job from starving co-located pipeline tasks.

Q: What does `ProfileReport(df, tsmode=True, sortby="price_date")` change about the report?
A: It enables time-series mode, which orders the data by `price_date`, identifies temporally-dependent columns via autocorrelation, and adds time-series sections (stationarity, seasonality) plus the `NON_STATIONARY` and `SEASONAL` alerts. This is the right mode for a NAV-per-share series where you care about trend and seasonality, not just static distribution. Without `sortby` it assumes the frame is already chronologically ordered.

Q: A NAV time-series profile raises a `NON_STATIONARY` alert. What does that mean for a fund series?
A: `NON_STATIONARY` means the series' statistical properties (mean or variance) change over time, which is normal for a NAV that trends upward but is worth confirming is genuine growth rather than a units break or a feed switch. ydata-profiling derives it from a stationarity test in `tsmode`. You would inspect whether the non-stationarity matches expected performance or signals a data artefact like a re-based series.

Q: What does the `sensitive=True` option do to a ydata-profiling report and when do you use it?
A: `sensitive=True` anonymizes the report by suppressing actual sample values and the most-frequent value listings, so the HTML can be shared without exposing raw records. You use it when profiling investor or counterparty data whose example rows must not leak into a circulated report. It keeps the aggregate statistics while hiding the individual values that could be PII.

Q: How do you force ydata-profiling to treat a numeric-looking code column as categorical?
A: Pass a `type_schema` mapping such as `ProfileReport(df, type_schema={"fund_status_code": "categorical"})`, which overrides the automatic type inference for that column. Without it, a status code stored as integers may be profiled as a numeric distribution, producing meaningless means and histograms. The schema override makes the profiler report value frequencies and an allowed-value set instead.

Q: What is the `low_categorical_threshold` setting and why tune it?
A: `vars.num.low_categorical_threshold` is the distinct-value count below which ydata-profiling treats a numeric column as categorical rather than continuous. Raising or lowering it changes whether a column like an integer rating gets a histogram or a frequency table. Tuning it avoids the profiler misclassifying a small-cardinality numeric code as a continuous measure.

Q: Write the whylogs code to build a constraint that a NAV column never goes below `0.01`.
A: `from whylogs.core.constraints import ConstraintsBuilder` and `from whylogs.core.constraints.factories import greater_than_number`, then `builder = ConstraintsBuilder(dataset_profile_view=view)`, `builder.add_constraint(greater_than_number(column_name="nav_per_share", number=0.01))`, then `constraints = builder.build()` and `constraints.report()`. The `greater_than_number` factory asserts the column's minimum exceeds the threshold against the profile's metrics. `report()` returns the per-constraint pass/fail list.

Q: How do you turn a whylogs constraints result into a human-readable pass/fail artifact?
A: Call `constraints.generate_constraints_report()`, which renders an HTML report showing each constraint and whether it passed or failed, or call `constraints.report()` for the raw tuples in code. The HTML form is what you attach to a run for an auditor; the tuple form is what you branch on in a pipeline. Both evaluate the constraints against the profile, not the raw rows, so they are cheap to recompute.

Q: Why is validating a whylogs constraint against a profile view rather than the raw table significant for a large fund estate?
A: Because the profile view is a compact sketch, so constraint evaluation runs against kilobytes of summary instead of re-scanning millions of rows, making validation cheap and decoupled from the source. You can ship the `.bin` profile to a central monitor and assert constraints there without moving sensitive holdings data. The trade-off is that constraints can only assert on metrics the profile actually captured.

Q: How do you produce a whylogs drift report between last week's and this week's profiles in a notebook?
A: Use `from whylogs.viz import NotebookProfileVisualizer`, instantiate it, call `visualizer.set_profiles(target_profile_view=this_week, reference_profile_view=last_week)`, then `visualizer.summary_drift_report()`. It renders an interactive report comparing each feature's distribution and flags drift severity. The reference is the baseline and the target is the new data.

Q: Which statistical test does whylogs use by default to score drift, and what is the fallback?
A: For columns with distribution metrics whylogs defaults to the Kolmogorov-Smirnov (`KS`) test to score numeric drift, and for columns without them it falls back to a Chi-Square test over frequent items and cardinality. The choice is automatic based on which metrics the profile holds for that column. You can override the algorithm and thresholds via the drift configuration if the defaults do not suit a column.

Q: How do you get drift scores back as data rather than a rendered report in whylogs?
A: Use `from whylogs.viz.drift.column_drift_algorithms import calculate_drift_scores` (or the visualizer's configured scoring) to compute per-column drift metrics returned as a Python dictionary you can threshold programmatically. This is the path for an automated weekly job that must alert, not just display. The dictionary keys are columns and the values carry the test statistic and p-value used to decide drift.

Q: A KS-based drift score is high but the column's range rule still passes. How do you interpret that?
A: The high KS statistic means the distribution's shape shifted relative to baseline even though every value stayed within the allowed bounds, so it is a genuine drift signal the static rule cannot see. For a NAV column this could be a clustering change from a corporate action or a feed switch. You investigate the upstream cause rather than dismissing it because the range check is green.

Q: How would you script a non-interactive weekly drift check on Ubuntu that exits non-zero on drift?
A: In a Python script invoked by `uv run`, load this week's `df`, build the profile view, `why.read()` the prior week's `.bin`, compute `calculate_drift_scores`, and `sys.exit(1)` if any score breaches your threshold so the orchestrator marks the task failed. A `systemd` timer or `cron` entry runs the script weekly. The non-zero exit is what lets Airflow or `cron` mail/alert on the failure.

Q: Schedule the weekly whylogs job with `cron` on Ubuntu — what does a sane entry look like?
A: A line like `0 6 * * 1 cd /opt/dq && /usr/bin/uv run python weekly_drift.py >> /var/log/dq/drift.log 2>&1` runs the check every Monday at 06:00, capturing stdout and stderr to a log. Using an absolute `uv` path and `cd` into the project avoids environment surprises under `cron`'s minimal shell. Redirecting `2>&1` ensures tracebacks are logged, not lost.

Q: Your `cron`-scheduled profiling script works in your shell but fails silently under `cron`. First thing to check?
A: Check the environment and working directory — `cron` runs with a minimal `PATH` and no shell profile, so `uv`, the project venv, and relative paths may not resolve. Use absolute paths, `cd` into the project, and capture `2>&1` to a log to see the real error. A missing `PATH` entry for `uv` is the classic cause of a job that "does nothing".

Q: Two whylogs `.bin` files were written by noticeably different library versions. What risk does merging them carry?
A: Merging across incompatible profile-format versions can fail to deserialize or silently mismatch metric layouts, corrupting the aggregate. whylogs profiles are versioned binary artifacts, so pinning the whylogs version across the whole estate keeps `why.read()` and `merge()` reliable. If versions diverge, re-log the data with a single pinned version rather than merging mismatched binaries.

Q: How do you roll seven daily whylogs `.bin` profiles into one weekly view in code?
A: Read each with `why.read("day_n.bin")` to get a `DatasetProfileView`, then fold them with `merge` — e.g. `weekly = views[0]; for v in views[1:]: weekly = weekly.merge(v)` — producing one view summarising the week. Because merge is associative and commutative, the order does not matter and the result equals logging all seven days at once. You keep the dailies for granularity and compute the weekly on demand.

Q: A merged weekly profile shows a higher estimated cardinality than any single day. Is that a bug?
A: No — merging unions the distinct values across days, so a security that appears on different days legitimately raises the weekly distinct count above any one day's. The HyperLogLog-style sketch merges correctly to estimate the union's cardinality. It would only be suspicious if the weekly estimate exceeded the sum of daily counts, which the sketch cannot produce.

Q: How does profiling distinguish a true schema change from a value-distribution change between two runs?
A: A schema change shows up as a column appearing, disappearing, or switching inferred type in the profile's column set, whereas a distribution change keeps the same columns but shifts their statistics. Comparing the keyed-by-column-name metric maps reveals structural deltas; comparing the per-column distributions reveals value deltas. They are different pillars (schema vs distribution) even though one profile diff surfaces both.

Q: Profiling a freshly ingested `EMT` template shows a mandatory field at 100% null. What is your first hypothesis?
A: The most likely cause is a header/column-mapping mismatch — the field was read from the wrong position or under the wrong name, so a present column reads as entirely empty. `EMT` files have fixed field definitions, so a 100% null on a mandatory field usually means a parsing or mapping defect, not genuinely absent data. You inspect the raw file's columns against the template spec before blaming the provider.

Q: How can profiling a `SWIFT`/MT message-derived table catch a truncated or malformed field?
A: Profiling the length distribution and value pattern of each field exposes values that are shorter than the fixed format or break the expected character class — for example a `BIC` column whose lengths are not the expected 8 or 11. `SWIFT` fields have strict length and format rules, so a length-distribution surprise is a direct format-violation signal. You turn the observed valid lengths into a SodaCL `valid length` check.

Q: Profiling shows an `ISIN` column where 100% of values are exactly 12 characters but some fail checksum. Would profiling catch the bad checksum?
A: Not directly — a length/pattern profile confirms 12 alphanumeric characters but does not validate the `ISIN` Luhn-style check digit, so a structurally-shaped but checksum-invalid `ISIN` passes the profile. The profile narrows you to format-conforming values; you need a dedicated checksum validation (a Soda custom check or a regex-plus-checksum function) to catch the invalid digit. This is a case where profiling is necessary but not sufficient.

Q: How would profiling reveal that a `LEI` column is silently padded with whitespace?
A: The length distribution would show values longer than the expected 20 characters, and the value-pattern analysis would flag trailing or leading spaces, even though the visible code looks correct. Whitespace padding is a common ingestion artefact that breaks exact joins to a reference. You would add a `trim`/length check and canonicalise before matching against the LEI reference data.

Q: Profiling a currency code column surfaces a value `EURO` alongside `EUR`. What does this indicate and what rule follows?
A: It indicates a non-standard code leaking in — `ISO 4217` currencies are three letters, so `EURO` is a data-entry or mapping error against the `EUR` standard. The value-frequency table makes the rogue code visible. You encode an allowed-values check restricting the column to the valid `ISO 4217` set, which fails the load when `EURO` reappears.

Q: A profile of a settlement-date column under `T+1` shows trade-to-settlement gaps of two days. What might that mean?
A: Under a `T+1` settlement regime the expected gap is one business day, so a recurring two-day gap suggests the pipeline is not accounting for the shortened cycle — perhaps stale logic still assuming `T+2`, or a calendar that miscounts holidays. Profiling the distribution of `settlement_date - trade_date` surfaces the anomaly. You would reconcile the date logic against the current `T+1` market convention.

Q: How can profiling help validate that an `EET` file's fields fall within their defined enumerations?
A: Profiling the value-frequency tables of each `EET` field exposes any code outside the template's defined enumeration before you rely on the file for ESG/sustainability reporting. The European ESG Template prescribes coded fields, so an unexpected value in the frequency list is an immediate validity flag. You convert each field's observed-but-valid value set into an allowed-values DQ rule.

Q: Profiling reveals a column with a recognisable `@`-domain pattern in a holdings extract. What is the correct handling?
A: Treat it as a candidate PII column (an email address) flagged by the value-pattern analysis, and route it to formal PII discovery (lesson 11.4.1) rather than concluding it is PII from the pattern alone. The profiler narrows the search; a dedicated scan with context confirms or rejects it. You would also question why an email field is in a holdings extract at all, since it may signal a join leak.

Q: Why might whylogs be preferable to ydata-profiling specifically when the data contains investor PII?
A: Because whylogs stores only statistical sketches and never the raw values, a whylogs profile can be centralised and shared for monitoring without exposing investor records, whereas a full ydata-profiling HTML embeds sample values and frequent items by default. For PII-bearing tables the privacy-preserving profile is the safer artifact to move off the secured environment. You would still run ydata-profiling locally with `sensitive=True` for the one-shot look.

Q: How do you make profiling deterministic so two runs on the same data produce comparable reports?
A: Disable sampling (profile the full frame or fix the sample with a seed), pin the library versions, and keep the same config so the same sections and thresholds apply. Non-determinism usually creeps in via random sampling or a version bump that changes default heuristics. Determinism matters when you diff reports across runs to attribute a change to the data, not the tool.

Q: A profiling job on a 200-column silver table is far slower than expected. What is the most likely culprit and the fix?
A: Pairwise correlations and interactions scale roughly with the square of column count, so a wide table makes them the dominant cost; the fix is `minimal=True` or selectively disabling `correlations` and `interactions` via config. Profiling 200 columns pairwise is tens of thousands of comparisons. Drop the quadratic sections and the runtime collapses to near-linear.

Q: How would you profile only a subset of suspect columns rather than the whole table?
A: Select the columns first — `ProfileReport(df[["nav_per_share", "isin", "currency"]], title=...)` — so the profiler only computes statistics for those, which is faster and focuses the report. This is the targeted approach when a drift alert points at specific fields. It avoids the cost of profiling hundreds of irrelevant columns to inspect three.

Q: Profiling reports an `INFINITE` alert on a ratio column. What likely produced it and what rule follows?
A: An `INFINITE` alert means the column holds `inf`/`-inf`, almost always from a division by zero in a derived ratio such as a yield or a price-to-something computed upstream. The profiler distinguishes infinities from ordinary outliers. You add a validity check rejecting non-finite values and fix the upstream calculation to guard the zero denominator.

Q: How does profiling distinguish "missing" encoded as `NaN` from missing encoded as the string `"NULL"`?
A: A true `NaN` raises the null count, but the literal string `"NULL"` is counted as a present, non-null value and appears in the value-frequency table instead. So a column with sentinel-text missingness shows a deceptively low null rate plus a frequent `"NULL"` token. Reading the frequent-items list alongside the null count is how you catch the disguised missingness.

Q: A profiled boolean-style column has three categories: `Y`, `N`, and `''`. What does the empty string signify?
A: The empty string is sentinel-encoded missingness masquerading as a third category, so the column is effectively not the clean binary you assumed. The value-frequency table exposes the `''` bucket that a null-count metric alone would miss. You would canonicalise `''` to a true null and then decide whether nulls are permitted, rather than blessing three categories.

Q: How would you profile to confirm a candidate composite primary key (e.g. fund-security-date)?
A: Profiling cannot directly test a multi-column combination, so you derive a single key column (concatenate or hash the three fields) and profile its uniqueness, or run an exact `COUNT(*)` versus `COUNT(DISTINCT fund, security, date)` query. The profiler's `UNIQUE`/cardinality signal works on single columns; composites need the derived-column trick or a direct SQL check. You confirm with an enforced uniqueness constraint before declaring the grain.

Q: Profiling shows two date columns where `effective_date` sometimes exceeds `expiry_date`. How does a profile surface this and what rule follows?
A: A single-column profile shows each date's distribution but not the cross-column relation; you surface the inversion either via a correlation/interaction view or, more reliably, by profiling a derived `expiry_date - effective_date` column whose negative values flag the inversions. You then write a SodaCL row-level check asserting `effective_date <= expiry_date`. This is a consistency rule that needs the derived field to be visible to profiling.

Q: When profiling a sample, which findings are trustworthy and which are not?
A: Distribution shape, null rate, and mean/quantiles are statistically robust on a representative sample, so they are trustworthy; exact uniqueness, rare-value presence, and the single corrupt outlier are not, because a sample can miss them. You sample for distribution discovery but run exact checks for key candidacy and rare-defect detection. Matching the method to the question avoids false confidence from a sampled profile.

Q: How would you size a profiling sample so distribution statistics are reliable?
A: Use a random (not head/tail) sample large enough that the quantiles stabilise — often tens of thousands of rows is plenty for a smooth distribution — and verify by re-sampling and checking the statistics barely move. A head sample is dangerous because data is often ordered (by date or fund), biasing the profile. Random sampling preserves the distribution that the profile is meant to describe.

Q: Why is `df.head()` a poor basis for a profile of a holdings feed?
A: Because feeds are frequently sorted by date or fund, so `head()` captures one slice (e.g. the earliest funds) and its distribution, null pattern, and value set differ systematically from the whole. A profile built on it encodes a biased reality. Always profile the full frame or a random sample, never an ordered prefix.

Q: How can profiling reveal that a "stale" feed re-delivered yesterday's data under today's load date?
A: Comparing today's profile to yesterday's would show near-identical distributions and an unchanged max price-date, even though the load timestamp advanced — the tell-tale of a re-served file. A drift report showing zero drift where you expect daily movement is itself the alarm. You add a freshness check asserting the max business date equals the current date.

Q: A whylogs diff shows zero drift across every column for a feed that normally moves daily. Is that good news?
A: No — zero drift in a feed that should change daily strongly suggests a stale or duplicate load delivering identical data, which is a failure, not stability. Drift detection alarms on change, so the inverted reading is that "too little change" is suspicious for a live feed. You confirm by checking the max business date and row-level timestamps against the load date.

Q: How would you profile to detect that a join silently dropped rows (a shrunken load) rather than duplicated them?
A: A fan-out shows as duplicates and inflated counts, but a dropped-rows defect shows as a lower row count and possibly new nulls where the join failed to match, so you compare the current profile's count and null rates against the baseline. A volume drop plus a null-rate spike on the joined columns is the signature. Profiling the count over time (the volume pillar) is what catches the shrink.

Q: What is the cleanest way to attach a profiling report as evidence to a specific pipeline run for audit?
A: Write the artifacts to a run-scoped, dated path — the ydata-profiling `to_file("run_2026-06-15.html")` and the whylogs `why.write(profile, "run_2026-06-15.bin")` — keyed to the orchestrator's run id so each run has its own profile. This builds the privacy-preserving, time-stamped trail that supports DORA-style evidence of monitoring. Overwriting a single fixed filename destroys the audit history, so always date or run-id the output.

Q: Summarise the end-to-end Ubuntu workflow this lesson asks you to stand up, in tool-and-command terms.
A: `uv add ydata-profiling whylogs`; one-shot profile the silver holdings table with `ProfileReport(df).to_file("holdings.html")` and read its alerts; encode the five surprises as SodaCL checks with the right dimension and severity; then `why.log(df)`, `why.write()` a dated `.bin`, and a `cron`/`systemd` weekly job that `why.read()`s the prior week and runs `summary_drift_report()`/`calculate_drift_scores` to alert on drift. Discovery (ydata-profiling) feeds enforcement (Soda) feeds monitoring (whylogs), closing the quality loop.

Q: A `HIGH_CARDINALITY` alert fires on a `counterparty_name` column. Is that a defect, and what would you do with it?
A: Not necessarily a defect — high cardinality is expected for a free-text name field, so the alert is informational rather than an error. The more useful action is to treat it as a PII/standardisation signal: many distinct spellings of the same counterparty suggest a need for an entity-resolution or `LEI`-mapping step. You would route it to reference-data cleanup and PII review rather than writing a uniqueness rule against it.

Q: How would you profile a Parquet file on disk that is too large to load fully into pandas on your Ubuntu laptop?
A: Read it lazily and sample before profiling — e.g. use `pyarrow`/`duckdb` to pull a random subset into a pandas frame, or point whylogs at batches and merge, rather than `pd.read_parquet` of the whole file. Profiling is descriptive, so a representative sample reproduces distributions and null rates without exhausting memory. For exact uniqueness you would still run a `duckdb` `COUNT(DISTINCT)` against the file directly rather than relying on the sampled profile.
