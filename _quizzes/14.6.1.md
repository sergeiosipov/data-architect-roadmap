# Phase 8 · 14.6.1 Data app frameworks (Streamlit) — 100 self-test questions

Q: What is Streamlit in one sentence, and what problem does it solve for a data team?
A: Streamlit is an open-source Python framework that turns a plain Python script into an interactive web app without writing HTML, CSS, or JavaScript. It solves the "last mile" problem of getting a data product in front of stakeholders: an analyst who can write pandas can ship a review queue or what-if tool in hours. The trade-off is that this simplicity comes from a rerun-everything execution model that constrains how the app scales.

Q: What is the single most important thing to understand about Streamlit's execution model?
A: Streamlit reruns your entire script from top to bottom every time a user interacts with a widget. There is no callback-per-component event loop like a traditional frontend; instead the whole script is re-executed and the page is re-rendered from the resulting widget tree. Understanding this "rerun-on-interaction" model is the foundation for reasoning about caching, state, and performance.

Q: What command starts a Streamlit app from a script called `app.py`?
A: You run `streamlit run app.py`, which launches the local Streamlit server (built on Tornado) and opens the app in your browser. Note you do not run the script with bare `python app.py` — that executes the file as a normal script and Streamlit will warn that nothing will render. On this Ubuntu setup you would use `uv run streamlit run app.py` to keep it inside the managed environment.

Q: Why does running `python app.py` instead of `streamlit run app.py` not work?
A: Streamlit needs its own server runtime to manage the session, the websocket connection, and the rerun loop, and `streamlit run` bootstraps all of that. Running the file directly just executes the Python top to bottom once with no server, so `st.*` calls have no runtime to render into and Streamlit prints a warning suggesting you use `streamlit run`. The script is not a standalone program; it is the body of a function the server calls on every rerun.

Q: When exactly does Streamlit rerun the script?
A: Streamlit reruns when a user interacts with a widget (changes a slider, types in a box, clicks a button), when the source file changes and you choose to rerun, or when you call `st.rerun()` programmatically. Each rerun re-executes the whole script in a fresh top-to-bottom pass. Widgets keep their values across reruns so the script "sees" the new input on the next pass.

Q: Walk through what happens, step by step, when a user moves a slider.
A: The frontend sends the new slider value to the server over the websocket, the server starts a fresh top-to-bottom run of the script for that session, and on that run the `st.slider(...)` call returns the new value. The rest of the script executes with that value, producing a new widget tree that is diffed and pushed back to the browser. The previous run's local variables are gone; only session state and caches survive.

Q: What is the client-server architecture of a Streamlit app?
A: The Python backend running `streamlit run` is the server, and the browser tab you view is the client; they communicate over a persistent websocket. The server runs the script, computes the widget tree, and streams render instructions to the client. Because all Python executes server-side, the app cannot directly touch the user's local filesystem or devices.

Q: What web framework and transport does the Streamlit server use under the hood?
A: The Streamlit server is built on the Tornado web framework and uses WebSockets to maintain a persistent, two-way channel between client and server. This is why it can push incremental UI updates without a full page reload. It also means each open browser tab holds an open websocket connection to the server.

Q: How many sessions does opening the same app in two browser tabs create?
A: Two — each browser tab opens its own websocket connection and therefore its own independent session. Session state and widget values are isolated between the tabs; they do not share variables. This is the basis for understanding why caches (shared) and session state (per-session) behave differently.

Q: What is `st.session_state` and what is its lifetime?
A: `st.session_state` is a per-session, dict-like object for sharing variables across reruns within one user session. It persists for the life of that browser-tab connection and resets when the tab closes, the connection drops, or the server restarts. It does not persist across different users or sessions, so it is not a substitute for a database.

Q: Why do you need session state at all given the rerun model?
A: Because every rerun re-executes the script top to bottom, ordinary local variables are recreated from scratch each time and cannot accumulate state. Session state survives reruns, so it is where you keep things like a running counter, a multi-step form's progress, or a selected row. Without it, any state you build up during interaction would be wiped on the next widget click.

Q: Show the standard pattern for initializing a session-state value exactly once.
A: Use the guard `if "key" not in st.session_state: st.session_state.key = initial_value`. This runs the initialization only on the first rerun where the key is absent and skips it on every subsequent rerun, so the value is not clobbered each pass. Initializing without the guard would reset the value on every interaction, defeating the purpose.

Q: What are the two syntaxes for reading and writing session state?
A: Dictionary-style, `st.session_state["count"]`, and attribute-style, `st.session_state.count`; both refer to the same underlying store. Use whichever reads more clearly, but keys with spaces or that collide with method names must use the dictionary form. Reading a key that was never set raises an exception, which is why the initialization guard matters.

Q: How does a widget's `key` parameter relate to session state?
A: Passing `key="celsius"` to a widget binds its current value to `st.session_state.celsius`, so the widget value and the session-state entry stay in sync automatically. You can read `st.session_state.celsius` anywhere after the widget runs. This unifies "widget state" and "session variables" into one store.

Q: Why does Streamlit raise an exception if you try to set `st.session_state` for a `st.button` or `st.file_uploader`?
A: Buttons and file uploaders are transient/action widgets whose value is determined by the interaction itself, not by a stored state you can pre-seed. Streamlit explicitly forbids setting their session-state value via assignment and raises an exception if you try. The pattern instead is to use their return value or an `on_click` callback.

Q: What is a Streamlit widget callback and when does it fire?
A: A callback is a Python function passed via `on_change` or `on_click` that Streamlit runs when that widget changes, and it executes before the rest of the script reruns. This lets you update session state in direct response to the interaction in a well-defined order. For example, `st.button("Increment", on_click=increment_counter)` calls `increment_counter()` first, then reruns the script.

Q: How do you pass arguments to a Streamlit callback function?
A: Use the `args` tuple or `kwargs` dict parameters on the widget, e.g. `st.button("Increment", on_click=fn, args=(5,))` or `kwargs=dict(increment_value=5)`. Streamlit forwards these to your callback when it fires. This avoids relying on closures over loop variables, which is a common source of bugs.

Q: In what order do a callback and the script rerun execute?
A: The callback runs first, then the full script reruns top to bottom. So state mutations you make inside an `on_change`/`on_click` callback are already in place when the script body re-executes. This ordering is what makes callbacks the right place to mutate session state cleanly.

Q: What does `st.rerun()` do and when would you call it?
A: `st.rerun()` immediately stops the current run and triggers a fresh top-to-bottom rerun of the script. You call it when you have changed state programmatically and need the UI to reflect it without waiting for a user interaction — for example after a successful form submit that should reset the page. Overusing it can cause loops, so guard the condition that triggers it.

Q: What does `st.stop()` do?
A: `st.stop()` halts execution of the current script run at that point, so nothing below it renders on this pass. It is useful for short-circuiting: if a required input is missing or a user is unauthenticated, call `st.stop()` to avoid running expensive code or rendering the rest of the app. The next interaction starts a fresh run from the top.

Q: Why is caching essential rather than optional in Streamlit?
A: Because the whole script reruns on every interaction, any expensive call — a database query, an API fetch, loading a model — would re-execute on every widget click without caching. Caching lets those results be reused across reruns and across sessions, turning an unusable app into a responsive one. The Phase-8 "Do" explicitly requires proving a widget interaction does not refetch the world.

Q: What is `st.cache_data` for?
A: `st.cache_data` caches the return value of functions that produce serializable data objects — DataFrames, arrays, strings, dicts, results of API calls, database queries, and transformations. It is the default choice for "I computed some data and don't want to recompute it on every rerun." Each cache hit returns a fresh copy of the stored value.

Q: What is `st.cache_resource` for?
A: `st.cache_resource` caches global, unserializable resources that should be created once and shared — database connections, ML models, file handles, HTTP client/session objects. It behaves as a singleton: every caller gets the same underlying instance rather than a copy. This is what you wrap a connection pool with so you are not reopening connections each rerun.

Q: What is the key behavioral difference between `st.cache_data` and `st.cache_resource` on return values?
A: `st.cache_data` returns a copy of the cached value on each call, so mutations by one session do not corrupt another's view and you avoid cross-session race conditions. `st.cache_resource` returns the actual shared instance, so any mutation directly changes the object in the cache for everyone. Choosing wrong leads either to wasted copies of a connection or to shared mutable data being silently corrupted.

Q: You wrapped a `pandas` DataFrame loader in `st.cache_resource` and now users see each other's edits. What went wrong?
A: `st.cache_resource` returns a shared singleton, so all sessions mutate the same DataFrame object and edits bleed across users. For serializable data like a DataFrame you should use `st.cache_data`, which hands each caller a copy and isolates mutations. The rule of thumb: data → `cache_data`, connections/models → `cache_resource`.

Q: You wrapped a database connection in `st.cache_data`. What problem will you hit?
A: A live connection is generally not serializable, so `st.cache_data` will fail to hash or pickle it, and even if it worked, copying a connection per call defeats the point of pooling. Connections belong in `st.cache_resource`, which keeps a single shared instance alive across reruns and sessions. The serialize-vs-singleton distinction is exactly what separates the two decorators.

Q: What do the `ttl` and `max_entries` parameters do on the cache decorators?
A: `ttl` (time-to-live, in seconds or a timedelta) invalidates a cached entry after that duration, so stale data is refetched — e.g. `ttl=3600` to refresh hourly. `max_entries` caps how many distinct results are stored, evicting the oldest when the limit is reached to bound memory. Both are available on `st.cache_data` and `st.cache_resource`.

Q: Your dashboard shows NAV figures that are an hour stale because of caching. How do you fix it without removing the cache?
A: Add a `ttl` to the cached fetch function so the entry expires and refetches on the desired cadence — for example `@st.cache_data(ttl=300)` to refresh every five minutes. This keeps the performance benefit while bounding staleness to your tolerance. For event-driven freshness you can instead call the function's `.clear()` when an upstream update arrives.

Q: How do you programmatically clear a specific cached function's results?
A: Call the `.clear()` method on the decorated function, e.g. `load_data.clear()`, which empties that function's cache. There is also a global option in the app menu and `st.cache_data.clear()` / `st.cache_resource.clear()` to wipe an entire cache type. This is how you force a refresh after you know upstream data changed.

Q: A cached function takes a database connection argument and Streamlit raises a hashing error. How do you fix it?
A: Prefix the unhashable parameter name with an underscore, e.g. `def load(_conn, query):`, which tells Streamlit to exclude that argument from the cache key. Streamlit then hashes only the remaining arguments (here `query`) to look up the cache. The connection itself should come from a `st.cache_resource`-decorated factory so it is a stable singleton.

Q: What does the leading-underscore convention on a cached function's parameter mean?
A: A parameter whose name starts with `_` is excluded from the cache key computation, so Streamlit does not try to hash it. Use it for arguments that are unhashable or irrelevant to the result identity, like a database connection or a logger. The remaining (non-underscore) arguments still determine cache hits and misses.

Q: When two arguments produce the same result but you can't hash one of them, what besides underscoring can you do?
A: Provide a custom hasher via the `hash_funcs` parameter on the decorator, mapping the argument's type to a function that returns a stable hash for it. This lets Streamlit include the argument in the cache key correctly instead of erroring or ignoring it. Use it when the value genuinely affects the result and you cannot just underscore it away.

Q: How do you prove, as required by the lesson's "Do" step, that a widget interaction is not refetching the world?
A: Add logging inside the expensive cached function (e.g. a log line on every actual execution) and watch that it fires on the first load and on cache expiry but not on each widget click. Cache hits skip the function body, so the absence of the log line on interaction is the proof. You can also surface a cache-hit counter in the UI to demonstrate it live.

Q: What is a Streamlit fragment and what problem does it solve?
A: A fragment, declared with `@st.fragment`, marks a function whose widgets trigger a partial rerun of just that fragment instead of the whole script. It solves the cost of full reruns when only one panel needs to update — for example a filter that should re-render a chart without re-running every other expensive panel. It is a key tool for keeping interactive Streamlit apps responsive.

Q: In your fund data product console, how could fragments help performance?
A: Wrapping each product panel (SLO status, DQ results, lineage, sample preview) so that interacting with one panel's controls only reruns that fragment avoids re-executing the other panels' fetches. For instance, paging through the sample-data preview should not re-pull SLO telemetry. This complements caching: caching reuses results, fragments narrow what reruns at all.

Q: What is the fundamental scaling constraint of Streamlit's execution model?
A: Streamlit runs as a single Python process and runs each user's script in its own thread within that process, so it is bounded by one process's CPU, memory, and the GIL. Heavy concurrent interactivity means many full-script reruns competing for the same process. This is why a naively built Streamlit app cannot transparently serve hundreds of simultaneous interactive users.

Q: Why can the same Streamlit app feel fine for 5 internal users but fall over at a board demo with 200?
A: Each interactive user triggers full-script reruns in the shared single process, so total load scales with users times interactivity, not just users. At low concurrency the process has headroom; at high concurrency reruns queue, the GIL serializes Python work, and latency spikes. Caching reduces the per-rerun cost but does not change that the process is a single shared bottleneck.

Q: How does caching change the scaling story, and how does it not?
A: Caching cuts the cost of each rerun by reusing expensive results across reruns and sessions, so the process does far less work per interaction and can serve more users before saturating. What it does not change is that there is still one process executing every rerun serially-ish under the GIL, so beyond some concurrency you still hit a wall. Caching buys headroom, not horizontal scale.

Q: How do you scale a single Streamlit instance to more users at the infrastructure level?
A: You run multiple Streamlit processes/replicas behind a load balancer and rely on session affinity (sticky sessions) so a user's websocket stays pinned to one replica. Because session state lives in-process per replica, a user must keep hitting the same instance for their session to be coherent. This gives more aggregate capacity but each session is still served by one process.

Q: Why does a load-balanced Streamlit deployment need session affinity (sticky sessions)?
A: Session state and the open websocket live inside one specific server process, so if a load balancer routes a user's next request to a different replica, that replica has none of their session state or connection. Sticky sessions pin each user to the replica that holds their session. Without it, users would see their state vanish mid-interaction.

Q: State the user-count/interactivity threshold where you would leave Streamlit, per the lesson's "Done when".
A: There is no single magic number, but the practical line is roughly low-tens of concurrent interactive users per process: when you need many simultaneous users each doing frequent heavy interactions, or rich multi-step stateful workflows, Streamlit's single-process reruns stop being economical. At that point you move to a horizontally scaled architecture or a real frontend with an API backend. The decision is driven by concurrency × interactivity × state complexity, not raw page views.

Q: What three axes drive the "stay in Streamlit vs leave" decision?
A: Interactivity (how much heavy re-computation each interaction triggers), state complexity (simple selections versus rich multi-user collaborative state), and audience size (a handful of analysts versus hundreds of concurrent users). Streamlit is ideal at the low-interactivity, low-state, small-audience corner. As any axis grows, the case for Dash or a real frontend strengthens.

Q: What is Dash and how does its model differ from Streamlit's?
A: Dash is an open-source Python framework (by Plotly) for building analytical web apps using an explicit callback model: you declare components and reactive callbacks that fire only for the inputs/outputs they declare, rather than rerunning the whole script. This gives finer-grained control over what updates and generally better behavior for complex, highly interactive dashboards. The cost is more boilerplate and a steeper learning curve than Streamlit's script-and-rerun simplicity.

Q: When would you choose Dash over Streamlit?
A: Choose Dash when you need fine-grained reactive control, complex multi-component interactions, or more demanding interactivity where rerunning the whole script is wasteful or hard to reason about. Its declarative callback graph updates only the affected outputs, which suits dense analytical dashboards. You accept more code and ceremony in exchange for that control and efficiency.

Q: When does the decision tip past both Streamlit and Dash toward a "real frontend"?
A: When you need large concurrent audiences, rich and persistent multi-user state, custom UX, complex auth flows, or product-grade reliability and accessibility, a real frontend (React/Vue) over a proper API backend is warranted. At that scale the Python-app frameworks' single-process or callback models become limiting. You trade rapid Python iteration for a separately scalable, fully controllable client and server.

Q: Summarize the decision line "Streamlit vs Dash vs real frontend" in one rule.
A: Streamlit for fast internal tools with small audiences and simple interactivity; Dash when interactivity and component-level reactivity grow but you still want Python; a real frontend with an API when audience size, state complexity, or product polish exceed what either Python framework serves economically. Pick by interactivity, state complexity, and audience size. The architect's failure mode is green-lighting Streamlit for a use case that is clearly past that line.

Q: An architect green-lit a 200-user interactive Streamlit "application" and it fell over at the first board demo. What was the core mistake?
A: They ignored the rerun-on-interaction single-process model: 200 concurrent interactive users each rerun the whole script in one shared process, so reruns queued and latency collapsed under load. The fix was either correct caching plus horizontal replicas with sticky sessions, or recognizing the use case was past Streamlit's line and choosing a different architecture. The lesson's "Why" calls out exactly this anti-pattern.

Q: Why should you not expose a Streamlit app on the public internet with its default settings?
A: Streamlit's built-in server has no real authentication or authorization by default, so anyone who can reach the URL can use the app and any data behind it. In a regulated fund-admin context that is unacceptable. You put it behind your existing gateway and identity plane rather than relying on Streamlit's defaults.

Q: How should the fund data product console be exposed, per the lesson's "Do"?
A: It should be served behind your Kong API gateway with OIDC authentication, so access uses the same identity plane as every other service. This means users authenticate through your central IdP and Kong enforces the policy before traffic ever reaches Streamlit. The benefit is consistent identity, audit, and access control rather than a bespoke per-app login.

Q: What is Kong's role in front of a Streamlit app?
A: Kong is the API gateway that terminates and routes traffic, and it enforces authentication/authorization (here OIDC) and policy before requests reach the Streamlit upstream. Streamlit stays an unauthenticated internal upstream; Kong is the security boundary. This keeps the app simple and centralizes access control with the rest of the estate.

Q: What is OIDC and why use it for the console rather than a custom login?
A: OIDC (OpenID Connect) is an identity layer on top of OAuth 2.0 that lets Kong delegate authentication to your central identity provider and receive verified identity tokens. Using it means the console reuses the organization's existing single sign-on, roles, and audit instead of a separate bespoke credential store. That is the whole point of "use the same identity plane as everything else."

Q: What must you configure on Kong so Streamlit's websocket actually works through the gateway?
A: The gateway and any proxies must allow WebSocket upgrade (the `Upgrade`/`Connection` headers) and not buffer or break the long-lived connection, because Streamlit relies on a persistent websocket. If the proxy treats it as ordinary HTTP and times out or strips the upgrade, the app will fail to render or repeatedly disconnect. Sticky routing also matters so the websocket stays pinned to one Streamlit replica.

Q: Your Streamlit app behind Kong keeps showing "Please wait..." and disconnecting. What is the first thing to check?
A: Check that the gateway/proxy is correctly forwarding the WebSocket upgrade and not timing out the long-lived connection, since Streamlit's UI depends on that persistent socket. A misconfigured proxy that closes idle connections or strips `Upgrade` headers produces exactly this reconnect loop. Also confirm sticky sessions so the socket lands on the same replica each time.

Q: How does putting OIDC at Kong rather than in the app affect your Streamlit code?
A: It keeps the Streamlit code free of auth logic: the app trusts that any request reaching it is already authenticated by Kong, and it can read the propagated identity (e.g. from a header or token) for RBAC decisions. You avoid embedding secrets or login flows in the app. The security boundary and the app are cleanly separated.

Q: What is the "Retool class" of tool and what does it sell?
A: Retool and similar commercial internal-tools platforms sell a fast way to build CRUD interfaces on top of governed APIs and databases, with built-in role-based access control, connectors, and a drag-and-drop UI builder. The pitch is "build internal admin tools quickly with auth and permissions handled for you." They occupy the same internal-tools niche as Streamlit but as a low-code commercial product.

Q: In one phrase, what does Retool give you "at a glance" that you would otherwise hand-build?
A: CRUD on governed APIs and data sources with RBAC baked in — meaning create/read/update/delete screens, pre-built connectors, and per-role permissions out of the box. With Streamlit you would assemble those pieces yourself (forms, API calls, and gateway-level auth). Knowing the Retool class helps you decide build-versus-buy for internal tooling.

Q: How does the Streamlit-vs-Retool choice typically break down?
A: Streamlit (open-source, code-first) wins when you want full Python control, custom analytics/visuals, and no per-seat licensing, and your team is comfortable in Python. Retool (commercial, low-code) wins when you need many standard CRUD admin screens fast, with built-in RBAC and connectors, and want to minimize bespoke code. The decision is build-and-own versus buy-and-configure for internal tools.

Q: What is the fund data product console you are asked to build in this lesson?
A: It is a Streamlit app with one screen per data product showing that product's SLO status, contract version, latest data-quality results, a lineage link, and a sample-data preview. Its purpose is to let a product owner answer "is my product healthy and who consumes it?" without touching the platform. Each panel pulls from an existing service, introducing no new data paths.

Q: Where should each panel of the console get its data, and why "no new data paths"?
A: Each panel reads from an existing service: SLO telemetry, the contract repository, the DQ store, and the OpenMetadata lineage link. Reusing existing services avoids creating new, ungoverned data flows that would themselves need contracts, quality checks, and lineage. The console is a thin presentation layer over governed sources, not a new pipeline.

Q: What is the "Done when" acceptance test for the console's usefulness?
A: A product owner must be able to answer "is my product healthy and who consumes it?" entirely from the console, without touching the underlying platform. If they still have to log into the platform or chase other tools, the console has failed its purpose. It is a self-service health-and-consumption view for product owners.

Q: Which panel of the console answers "who consumes my product," and how?
A: The lineage panel, surfaced via the OpenMetadata link, answers consumption by showing downstream dependencies of the product. The console embeds or links to that lineage so the owner can see consumers without learning the catalog UI. Combined with SLO and DQ panels it gives the full "healthy and consumed?" picture.

Q: Why is a sample-data preview a sensible panel, and what is the caching consideration?
A: A small sample lets the owner sanity-check the data shape without querying the platform, reinforcing the self-service goal. Because pulling samples can be expensive, wrap the fetch in `st.cache_data` (with a `ttl` for freshness) so paging or revisiting the panel does not re-query each rerun. Logging the fetch proves the cache is doing its job.

Q: For the SLO-status panel, why might you choose `st.cache_data` with a short `ttl` rather than no cache?
A: SLO telemetry is relatively expensive to fetch and changes on a slow cadence, so caching with a short `ttl` (say 30–60 seconds) keeps the panel responsive while bounding staleness. Without a cache, every unrelated widget interaction would re-pull telemetry on the full rerun. The `ttl` ensures the status is never more than a minute old.

Q: Why is the contract version best served from the contract repository rather than copied into the app?
A: Reading the live contract version from the repo means the console always reflects the source of truth and cannot drift, whereas a hard-coded or duplicated value would silently go stale. This honors the "no new data paths" rule by pointing at the governed source. If the contract bumps, the console shows the new version automatically.

Q: What is an ISIN and where might it appear in this console?
A: An ISIN (International Securities Identification Number) is the 12-character standard code identifying a security, including funds; it appears in the sample-data preview and as a key for selecting or filtering a fund/product. Using the ISIN as the product key ties the console panel to the right instrument unambiguously. It is one of the canonical identifiers a fund data product is keyed on.

Q: An ISIN passed to a cached lookup is a plain string — does it cause any caching issue?
A: No — strings are hashable and serializable, so an ISIN is a perfectly good part of a `st.cache_data` cache key, and different ISINs produce distinct cache entries. The caching problems arise only with unhashable arguments like connections, which you underscore or move to `cache_resource`. So `@st.cache_data` def `load_fund(isin)` caches cleanly per ISIN.

Q: How would you key the console "one screen per product" navigation in session state?
A: Store the currently selected product identifier (e.g. its ISIN or product id) in `st.session_state` so the selection survives reruns as the user interacts with panels. A `st.selectbox` with a `key` binds the choice directly to session state. The panels then read that key to fetch the right product's SLO, DQ, contract, and lineage.

Q: Why is a NAV figure a good example for the freshness-vs-cache trade-off?
A: NAV (net asset value) is computed on a cycle and consumers care about getting the current published figure, so an over-long cache `ttl` would show a stale NAV and an under-long one would hammer the source. The lesson's caching skill is to pick a `ttl` matched to the NAV publication cadence so the console is both fast and acceptably fresh. It is the canonical "stale data because of caching" scenario in fund admin.

Q: A panel shows a NAV from yesterday's cycle even though today's is published. First thing to check?
A: Check the cache `ttl` on the NAV-fetch function — an entry cached before today's publication is being served until it expires. Either set a `ttl` aligned to the publication cadence or call the function's `.clear()` when a new NAV is published. If there is no cache involved, then check the upstream source itself.

Q: Why should DQ results in the console come from the DQ store rather than being recomputed in the app?
A: Recomputing data-quality checks inside the console would create a new, ungoverned computation path and likely disagree with the platform's official results. Reading the latest results from the DQ store keeps a single source of truth and respects "no new data paths." The console's job is to display the governed result, not to re-derive it.

Q: How do you make a cached fetch refresh exactly when upstream DQ results change, instead of on a timer?
A: Trigger an event-driven invalidation: when the DQ store publishes new results, call the fetch function's `.clear()` (or clear the relevant cache) so the next rerun refetches. This gives precise freshness without a coarse `ttl` polling interval. It is the right pattern when updates are discrete events rather than continuous drift.

Q: What goes wrong if you put a long-lived database connection creation directly in the script body without caching?
A: A new connection would be opened on every single rerun — every widget click — exhausting the connection pool and adding latency, because the script body re-executes each time. Wrapping the connection factory in `st.cache_resource` creates it once and shares the singleton across reruns and sessions. This is the textbook use of `cache_resource`.

Q: Why is `st.cache_resource` the right wrapper for a shared HTTP client used to call your SLO telemetry API?
A: An HTTP client/session is an unserializable resource you want to create once and reuse (for connection pooling and keep-alive), which is exactly the singleton semantics `st.cache_resource` provides. `st.cache_data` would try to copy/serialize it and is for data, not connections. So you cache the client with `cache_resource` and the responses with `cache_data`.

Q: Distinguish what should be `cache_resource` vs `cache_data` in the console.
A: The database connection, the HTTP client to the SLO/DQ services, and any loaded model belong in `st.cache_resource` as shared singletons. The fetched SLO status, DQ results, contract version, and sample DataFrame belong in `st.cache_data` as copied, serializable results. Getting this split right is most of what "cache expensive calls correctly" means here.

Q: Why does Streamlit run each session's script on its own thread, and what concurrency hazard does that create?
A: Per-session threads let the single server process interleave multiple users' reruns without separate processes. The hazard is shared mutable state: anything stored in a `st.cache_resource` singleton or module-level global is shared across those threads, so concurrent mutation can race. That is why `cache_data` copies and why you treat `cache_resource` values as read-mostly.

Q: How can two concurrent users corrupt each other's data through a cached resource?
A: If both call a `st.cache_resource` function returning the same mutable object and one mutates it, the other sees the mutation because they share the instance across threads. There is no copy boundary as there is with `cache_data`. The fix is to not store user-specific mutable state in a shared resource — keep that in per-session `st.session_state`.

Q: Where does per-user state belong, and where does shared, expensive-to-build state belong?
A: Per-user, interaction-driven state belongs in `st.session_state`, which is isolated per session. Shared, expensive-to-build, read-mostly state (connections, models, reference data) belongs in `st.cache_resource`. Confusing the two — putting user state in a shared cache or rebuilding shared resources per session — causes either data bleed or poor performance.

Q: Does `st.cache_data` share results across different user sessions?
A: Yes — caches are process-wide, so a result cached by one session is available to all sessions in that process, which is exactly why a popular query is computed once and reused for everyone. Each caller still gets its own copy of the value with `cache_data`. Session state, by contrast, is never shared across sessions.

Q: If you scale to multiple Streamlit replicas, is the cache shared across them?
A: No — each replica is its own process with its own in-memory cache, so a result cached on one replica is not seen by another. This means cache hit rates depend on routing and that you cannot assume a global cache across replicas. For cross-replica sharing you would need an external cache (e.g. Redis) wired in yourself.

Q: Why might you front Streamlit with an external cache like Redis at scale despite the built-in caches?
A: The built-in `st.cache_data`/`st.cache_resource` are per-process, so across replicas they neither share nor coordinate, and they vanish on restart. An external store like Redis gives a shared, persistent cache so all replicas benefit from one computation and survive restarts. This is part of moving from a single instance toward a horizontally scaled deployment.

Q: What is a common cause of a Streamlit app that "feels slow on every click"?
A: Expensive work (queries, API calls, heavy transforms) running unconditionally in the script body so it re-executes on every rerun. The fix is to wrap those calls in `st.cache_data`/`st.cache_resource` and, where only a panel needs updating, use `@st.fragment` to scope the rerun. Logging actual executions confirms whether the cache is being hit.

Q: How do you confirm whether a slow Streamlit app is rerun-bound or fetch-bound?
A: Add timing/log lines around the expensive functions and at the top of the script to see how often they actually execute per interaction. If the expensive function logs on every click, it is fetch-bound and needs caching; if it does not but the page is still slow, the rerun itself or rendering is the cost and fragments or simplification help. Measuring before optimizing avoids guessing.

Q: Why is it dangerous to do an in-place mutation of a DataFrame returned by `st.cache_data`?
A: Although `st.cache_data` returns a copy so other sessions are protected, mutating the returned object can still surprise you within your own run if you assume it is the cached original, and historically mutating cached values was a known footgun. The safe habit is to treat cached return values as read-only and make explicit copies before mutating. This avoids subtle bugs and keeps the cache semantics clear.

Q: What does `st.write` do, and why is it the "Swiss Army knife" of Streamlit?
A: `st.write` accepts almost anything — strings (rendered as Markdown), DataFrames (as tables), charts, dicts, and more — and renders an appropriate widget for each. It is the quickest way to get something on screen while prototyping a panel. For production panels you usually switch to specific elements like `st.dataframe` or `st.metric` for control.

Q: Which element would you use for the console's SLO-status headline number?
A: `st.metric` is designed for a single key figure with an optional delta, making it ideal for an SLO status or a NAV headline with a change indicator. It communicates "the one number that matters" cleanly. Surrounding context (history, breakdown) can go in adjacent `st.dataframe` or chart panels.

Q: What is the difference between `st.dataframe` and `st.table`?
A: `st.dataframe` renders an interactive, scrollable, sortable grid suited to larger or explorable data like the sample preview, while `st.table` renders a static, fully-drawn table good for small fixed summaries. For the sample-data panel `st.dataframe` is the better fit. Choosing `st.table` for a large preview would render the whole thing statically and bloat the page.

Q: How do `st.columns` and `st.tabs` help lay out "one screen per product"?
A: `st.columns` places panels side by side (e.g. SLO metric next to contract version) and `st.tabs` groups related panels behind tab labels (e.g. an "SLO" tab and a "Quality" tab) to keep the screen uncluttered. Together they structure the five panels into a readable single-product view. Layout containers do not change the rerun model; they only arrange output.

Q: What is `st.spinner` and why use it around the console's fetches?
A: `st.spinner("Loading...")` displays a loading indicator while the code inside its `with` block runs, giving the user feedback during an expensive fetch. It improves perceived responsiveness on a cache miss when data is actually being pulled. On a cache hit the block runs instantly so the spinner barely shows, which is itself a signal the cache worked.

Q: How does `st.form` change the rerun behavior, and why is that useful?
A: Widgets inside a `st.form` do not trigger a rerun on each change; the script reruns only when the form's submit button is pressed, batching all inputs together. This avoids a rerun per keystroke or per field in a multi-field filter. For a console filter with several inputs, a form prevents firing expensive fetches until the user is done.

Q: A multi-field filter refetches data on every keystroke. How do you fix it with Streamlit primitives?
A: Wrap the inputs in a `st.form` so changes are batched and the rerun (and downstream fetch) happens only on submit, not per keystroke. Pair that with caching the fetch so repeated submits with the same values are instant. This combination eliminates the per-interaction refetch storm.

Q: What does `st.secrets` provide and how does it relate to running behind Kong?
A: `st.secrets` reads values from a `secrets.toml` (or environment-backed) store so you keep API keys and connection strings out of source code. Behind Kong+OIDC, the app still needs credentials for the upstream services it calls, and `st.secrets` is where those live rather than hard-coded. The gateway handles user auth; `st.secrets` handles the app's own service credentials.

Q: How does the app know which user is logged in when auth is enforced at Kong?
A: Kong, after OIDC authentication, can propagate the verified identity to the upstream as a header or token, which the Streamlit app reads to know who the user is for display and RBAC. The app trusts this because only authenticated traffic reaches it through the gateway. It should still validate/limit what it does with that identity rather than trusting arbitrary client headers if the gateway is bypassable.

Q: Why must Streamlit not trust a user-supplied identity header if it could be reached without going through Kong?
A: If the Streamlit upstream is directly reachable, a client could forge the identity header that the app reads, bypassing access control. The safe design ensures the app is only reachable via Kong (network isolation) so the header is genuinely set by the gateway. This is why placing the gateway as the sole ingress, not just in front, matters.

Q: How would you implement per-role visibility in the console once you know the user's role?
A: Read the role from the gateway-propagated identity and conditionally render panels or actions — for example, hiding edit controls for read-only roles or restricting which products a product owner sees. Use `st.stop()` or early returns to short-circuit unauthorized access. This is application-level RBAC layered on top of Kong's authentication.

Q: Why is reproducible deployment (pinned dependencies) important for a Streamlit app in a regulated context?
A: A Streamlit app pulls in many packages whose behavior can change between versions, and in a regulated fund-admin estate you need deterministic, auditable builds. Pinning dependencies (here via `uv` and a lockfile) ensures the deployed app behaves identically to what you tested. Unpinned dependencies risk a silent breakage or behavior change between demo and production.

Q: On this Ubuntu setup, how would you launch the console inside the managed Python environment?
A: Use `uv run streamlit run console.py`, which executes Streamlit within the project's `uv`-managed virtual environment and resolved dependencies. This avoids a bare `python`/`pip` invocation and keeps the run reproducible. In containerized deployment the same command is the container entrypoint.

Q: What is the default port Streamlit serves on, and why does it matter behind Kong?
A: Streamlit serves on port `8501` by default, which is the upstream port Kong must route to. Knowing it lets you configure the gateway's upstream target correctly and bind the app appropriately in the container. You can change it with `--server.port` if needed, but the gateway config must match whatever you choose.

Q: Why might you set `--server.headless true` (or the equivalent config) when deploying Streamlit?
A: In a server/container deployment there is no local browser to auto-open, so headless mode stops Streamlit from trying to launch one and suppresses the browser prompt. It is the appropriate mode behind Kong where users reach the app through the gateway URL. Leaving it off in a container just produces a harmless but noisy attempt to open a browser.

Q: Summarize the rerun-on-interaction model in a way you could explain to a product owner, per the "Done when".
A: Every time you touch a control, Streamlit re-runs the whole Python script and redraws the page from scratch, which is why it is so simple to build but also why we cache expensive lookups. Caching means the heavy data fetches run once and are reused on later interactions instead of re-running on every click. Session state remembers your selections across those reruns so the page does not forget what you picked.

Q: In one summary, where did caching save you in the console, and where would you leave Streamlit entirely?
A: Caching saved the SLO telemetry, DQ results, contract lookup, and sample-data fetches from re-running on every widget interaction — proven by logging that the fetch fires on first load and `ttl` expiry but not on each click — while connections and clients live as `cache_resource` singletons. You would leave Streamlit when concurrent interactive users, multi-user persistent state, or product-grade UX exceed its single-process rerun model, moving to Dash for richer Python reactivity or to a real frontend over an API for scale. The trigger is interactivity × state complexity × audience size, not page count.
