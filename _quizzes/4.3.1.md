# Phase 2 · 4.3.1 MPP/federated SQL engines (Trino) — 150 self-test questions

Q: What is Trino, in one sentence, and what category of engine does it belong to?
A: Trino is a distributed, open-source MPP (massively parallel processing) SQL query engine designed to run interactive analytic queries across one or many heterogeneous data sources. It is a query engine only — it has no storage of its own and ships no built-in catalog of data, so it federates over external systems like object stores, Iceberg tables, and relational databases. This decoupling of compute from storage is exactly the "decoupled compute" pattern the lakehouse relies on.

Q: What does "MPP" stand for and what does it imply about how Trino runs a query?
A: MPP stands for massively parallel processing, meaning the work of a single query is split into many parallel units that run simultaneously across many worker nodes. Trino divides a query into stages, each stage into tasks, and each task processes splits of data in parallel using multiple drivers. The implication is that adding workers can increase throughput for a query, in contrast to a single-process engine that runs everything in one address space.

Q: What two roles do nodes play in a Trino cluster, and what does each do?
A: A Trino cluster has exactly one coordinator and one or more workers. The coordinator is the brain: it parses SQL, plans and optimizes the query, schedules work, and aggregates results to return to the client. Workers fetch data from connectors, execute the assigned tasks on splits, and exchange intermediate data with each other.

Q: Can a single Trino node act as both coordinator and worker, and when would you do that?
A: Yes — for development or a tiny deployment you can set `node-scheduler.include-coordinator=true` so the coordinator also processes splits as a worker. This is convenient for a local Docker Compose lab but is discouraged in production because heavy worker work on the coordinator competes with its scheduling and result-gathering duties. For the capstone single-node Trino, running it as combined coordinator/worker is the normal setup.

Q: What is the discovery service in Trino and where does it live?
A: The discovery service is the registry workers use to announce themselves and through which the coordinator finds available workers. By default it runs embedded inside the coordinator process, so you do not deploy it separately. When a worker process starts, it advertises itself to the discovery server in the coordinator, which is how the coordinator builds its picture of the cluster.

Q: A worker is running but the coordinator never schedules work on it. What is the first thing to check?
A: Check that the worker successfully registered with the discovery service — confirm the worker's `discovery.uri` points at the coordinator and that the coordinator shows the node as active (the Web UI lists active workers, or you can query `system.runtime.nodes`). A common cause is a wrong or unreachable `discovery.uri`, a network/firewall block on the HTTP port, or a `node.environment` mismatch between coordinator and worker. If the node never appears, it is a registration problem, not a query problem.

Q: What does the client submit to, and how does it get results?
A: The client (CLI, JDBC/ODBC driver, or a BI tool) submits SQL statements to the coordinator over HTTP and then polls the coordinator for results. The coordinator returns results incrementally as they become available, so the client does not block on the entire query finishing before any rows arrive. The client never talks directly to workers.

Q: Distinguish a "statement" from a "query" in Trino's terminology.
A: A statement is just the SQL text you send, like `SELECT * FROM nav`. A query is the live, instantiated execution of that statement — the configuration, query plan, stages, tasks, splits, and runtime state Trino creates to run it. One statement produces one query when executed; the distinction matters when you read documentation about query lifecycle and the Web UI's per-query metrics.

Q: Name the execution hierarchy from query down to the smallest unit of work.
A: A query is broken into stages, stages are translated into tasks, tasks operate on splits, and within a task work runs through drivers made of operators. Splits are sections of the underlying data; drivers are the parallel execution units inside a task; operators (scan, filter, join, aggregate, etc.) consume and transform pages of data. Understanding this hierarchy is what lets you read the Web UI and a query plan intelligently.

Q: What is a "split" in Trino and where does it come from?
A: A split is an addressable chunk of a data source that a task can process independently — for example a portion of a Parquet file or a range of an Iceberg data file. The connector for each source is responsible for enumerating splits, which is how Trino achieves parallelism: many splits are processed concurrently across tasks and workers. The number and size of splits a connector produces strongly affects parallelism and performance.

Q: What is a "stage" and why don't stages run directly on workers?
A: A stage is a node in the coordinator's tree-shaped distributed query plan, representing one logical phase such as "scan and filter" or "final aggregate." Stages are an abstraction the coordinator uses for planning; they are translated into tasks, and it is the tasks that actually run on workers. So a stage describes work; tasks are the concrete instances of that work distributed across the cluster.

Q: What is an "exchange" in Trino's execution model?
A: An exchange moves intermediate data between tasks, typically across stage boundaries and often across worker nodes — for example shuffling rows by join key so matching keys land on the same worker. Exchanges are where network traffic and redistribution happen, so a plan with large exchanges signals a lot of data movement. Reading exchange types in `EXPLAIN` (e.g. a hash-partitioned exchange before a join) tells you how data is repartitioned.

Q: What is a "page" in Trino and why does it matter?
A: A page is Trino's unit of columnar in-memory data passed between operators — a collection of blocks, each block holding the values for one column for a batch of rows. Operators consume and produce pages, which is what makes Trino's execution columnar and vectorized rather than row-at-a-time. You rarely set pages directly, but the concept explains why Trino is memory- and CPU-efficient on analytic scans.

Q: What is a "driver" versus a "task" in Trino?
A: A task is the unit of work assigned to a worker for one stage on that worker; a driver is one of potentially many parallel pipelines of operators inside that task. A task can run many drivers concurrently to exploit a worker's CPU cores, each driver pushing pages through its operator chain. So tasks give you cross-node parallelism and drivers give you within-node parallelism.

Q: What is a connector in Trino, and how many does a catalog use?
A: A connector is the plugin that adapts Trino to a specific kind of data source — it knows how to list schemas and tables, enumerate splits, read data, and report what pushdown it supports. Each catalog is configured with exactly one connector; for example an `iceberg` connector or a `postgresql` connector. To query two different systems you configure two catalogs, each with its own connector.

Q: Explain the catalog → schema → table naming hierarchy in Trino.
A: Trino addresses every table with a three-part name `catalog.schema.table`. The catalog identifies a configured data source (and its connector), the schema is a logical grouping within that source, and the table is the relation itself. For example `iceberg.funds.nav_history` and `postgres.public.fund_ref` are two tables in two different catalogs that you can join in one statement.

Q: How do you create a catalog in Trino, and what does the file look like?
A: You add a properties file under the `etc/catalog/` directory, named after the catalog (e.g. `iceberg.properties`), containing at minimum `connector.name=iceberg` plus connector-specific settings. The filename (minus `.properties`) becomes the catalog name you reference in SQL. In recent Trino versions catalogs can also be created dynamically via the SQL `CREATE CATALOG` command if the feature is enabled, but file-based config is the standard, restart-required approach.

Q: For the capstone, which two catalogs do you configure and why?
A: You configure an `iceberg` catalog (pointing at MinIO storage plus a metastore/REST catalog) for historical NAV and holdings facts, and a `postgresql` catalog pointing at the Phase-1 Postgres for live transfer-agency reference data. With both configured, a single Trino statement can join the live Postgres dimension to the historical Iceberg facts — the practical demonstration of federation. This is exactly the "no ETL hop" benefit the lesson highlights.

Q: What is "pushdown" in Trino, in general terms?
A: Pushdown is when Trino delegates part of the query work — predicates (filters), projections (column selection), aggregations, limits, or even joins — down to the underlying data source instead of doing it itself after fetching everything. The benefit is that the source returns far less data, cutting network transfer and Trino-side compute. Which pushdowns are possible depends entirely on the connector and the source's capabilities.

Q: What is predicate pushdown and why is it the single biggest performance lever in federation?
A: Predicate pushdown sends `WHERE` conditions to the source so it filters rows before returning them — e.g. pushing `fund_id = 42 AND dt >= DATE '2026-01-01'` into Postgres or into an Iceberg scan. Without it, Trino would pull every row across the network and filter locally, which for a large table is catastrophically slow and expensive. It is the biggest lever because it directly determines how many rows cross the wire.

Q: What is projection pushdown?
A: Projection pushdown sends the list of needed columns to the source so only those columns are read and transferred, rather than `SELECT *`. For columnar sources like Parquet/Iceberg this means only the requested column chunks are read; for Postgres it means a narrower `SELECT` column list. In the Iceberg connector this is controlled by `iceberg.projection-pushdown-enabled`, which defaults to `true`.

Q: What is aggregation pushdown and which connectors typically support it?
A: Aggregation pushdown delegates computations like `COUNT`, `SUM`, `MIN`, `MAX`, or `GROUP BY` to the source so it returns pre-aggregated results. JDBC connectors such as PostgreSQL support a range of aggregate pushdowns, letting the database do the grouping; the Iceberg connector can answer some aggregates directly from file/partition statistics. When it fires, Trino transfers a handful of aggregated rows instead of millions of raw rows.

Q: How do you confirm what actually pushed down to a source?
A: Run `EXPLAIN` (or `EXPLAIN ANALYZE`) on the query and read the plan: pushed-down filters appear inside the `TableScan`/connector node (look for a constraint or `filterPredicate` summary on the scan), while work Trino does itself appears as separate `Filter`, `Aggregate`, or `Project` operators above the scan. If a predicate shows up as a standalone `ScanFilter` or `Filter` node rather than being folded into the scan, it did not push down. This is the concrete skill the lesson's "Done when" requires.

Q: Why might a `WHERE` predicate fail to push down to Postgres even though the column exists?
A: Common reasons: the predicate uses a function or expression the connector cannot translate (e.g. a Trino-specific cast or a function with no SQL equivalent), the column types do not map cleanly, or the predicate involves a value computed from another catalog so it is not known at scan time. The connector only pushes predicates it can faithfully express in the source's dialect; anything else stays in Trino. Rewriting the predicate to a directly translatable form (a plain comparison on the raw column) often restores pushdown.

Q: In a join between Postgres reference data and Iceberg NAV facts, which side's predicates can push down?
A: Predicates on columns of each table push down to that table's own source independently — filters on the Postgres dimension push to Postgres, filters on the Iceberg facts push to the Iceberg scan. The join condition itself is generally executed in Trino (the cross-store join cannot be pushed to either source), so the goal is to push the per-table filters hard enough that each source returns only the rows the join needs. A `dt` range on the Iceberg side and a `domicile = 'LU'` filter on the Postgres side both shrinking their inputs is the ideal shape.

Q: What is a "chatty cross-store join" and why is it an anti-pattern?
A: A chatty cross-store join is a federated join where Trino repeatedly issues many small queries to one source for each batch of rows from the other (or pulls a huge unfiltered table across the network), producing high latency and load on the remote system. It is an anti-pattern because the per-round-trip and per-row transfer costs dominate; the work is bounded by network chatter, not by useful computation. The architect's job is to recognize this in a plan and either push filters harder or copy the dimension local.

Q: How do you spot a chatty cross-store join in a Trino query plan?
A: Look for a `TableScan` on a remote source that has no pushed-down filter and returns a very large estimated/actual row count, feeding a join, plus large `Exchange` nodes moving those rows. In `EXPLAIN ANALYZE` the telltale sign is millions of rows read from a remote catalog with little selectivity, or a dynamic-filter-less scan on the large side. The remedy is more selective pushdown, a dynamic filter, or materializing the small dimension into the same store as the facts.

Q: What is a dynamic filter in Trino and how does it help federated joins?
A: A dynamic filter is a runtime-generated predicate: Trino first reads the small (build) side of a join, collects the set or range of join-key values, then pushes that as a filter into the scan of the large (probe) side. This can dramatically prune the large Iceberg scan — e.g. only reading NAV rows for the handful of `fund_id`s present in the filtered Postgres dimension. Dynamic filtering is a key reason a well-shaped federated join can beat a naive one without copying data.

Q: When does federating beat copying the dimension, and when does copying win?
A: Federating wins when the remote dimension is small, changes frequently (you want live values), and your filters/dynamic filters keep transfer tiny — you avoid an ETL hop and always read current data. Copying wins when the join is run often, the remote side is large or slow, predicates do not push down well, or the remote system cannot tolerate the query load — materializing the dimension into Iceberg makes the join local and fast. The lesson's exercise of timing both versions is how you find the crossover for a given workload.

Q: A federated join is slow; you suspect the Postgres side. What metrics confirm it?
A: In `EXPLAIN ANALYZE`, check the rows and wall time on the Postgres `TableScan` node and whether a filter is folded into it; also watch Postgres-side load (its own `pg_stat_activity`/slow-query log). If Trino is pulling the whole table because the predicate did not push down, you will see a high input-row count and most query time spent in that scan plus its exchange. Fixing pushdown, adding an index on the filtered/join column in Postgres, or copying the dimension are the levers.

Q: Why can hammering a transactional Postgres with federated analytic queries be operationally dangerous?
A: Trino can issue large, long-running scans that compete with the OLTP workload Postgres exists to serve, causing lock contention, buffer-cache pollution, and latency spikes for the application. In a fund admin context, a month-end analytic burst hitting the live transfer-agency database could degrade the system processing subscriptions and redemptions. Mitigations include reading from a replica, limiting concurrency, and pushing filters so scans stay small.

Q: What is Trino's memory model — what are the relevant per-query memory limits?
A: Trino tracks memory used by query operators and enforces limits so one query cannot exhaust the cluster. Key settings include `query.max-memory-per-node` (user memory a single query may use on one worker) and `query.max-memory` (total user memory a query may use across all workers). When a query exceeds these limits it fails rather than crashing the JVM, which is how Trino stays stable under memory pressure.

Q: Why do big joins and aggregations fail with memory errors in Trino, even on a healthy cluster?
A: Joins and `GROUP BY`/`DISTINCT` operations build in-memory hash tables that must hold the build side (or all groups) in RAM, and if that exceeds the per-node or per-query memory limit the query is killed. This is intrinsic to in-memory MPP execution: a join over two huge tables can require more memory than configured. The fix is to enable spill, add memory/workers, increase selectivity so inputs shrink, or restructure the query.

Q: What is "spill to disk" in Trino and which operators support it?
A: Spilling lets memory-intensive operators write intermediate state to local disk when they exceed their memory budget, then read it back, trading speed for the ability to complete instead of failing. It applies to operators like hash joins, aggregations, order-by/sort, and window functions. You enable it with `spill-enabled=true` and configure `spiller-spill-path` to point at fast local disks (ideally not the OS root volume).

Q: A large `GROUP BY` keeps failing with an "exceeded memory limit" error. What are your options in order?
A: First confirm what limit was hit from the error message (per-node vs per-query, user vs total). Then: enable spill so the aggregation can overflow to disk; reduce the input via more selective predicates or pre-aggregation; raise `query.max-memory-per-node`/`query.max-memory` if the cluster genuinely has headroom; or add workers to spread the hash table. Spill is the least invasive first move when you cannot shrink the data.

Q: What is the cost/limit of relying on spill instead of more memory?
A: Spilling writes and re-reads gigabytes from local disk, so it is far slower than in-memory execution and is gated by disk throughput and free space; if the spill volume fills up the query still fails. It keeps queries from dying on memory limits but does not make them fast, so it is a safety valve, not a performance strategy. For repeated heavy workloads the better answer is more memory, more workers, or reshaping the data.

Q: What does "decoupled compute" mean concretely with Trino, and why is it valuable?
A: Decoupled compute means the query engine (Trino) is entirely separate from where data lives (object storage, Iceberg, Postgres) and from the catalog, so you can scale, restart, or replace the compute layer without touching the data. You can spin Trino up only when month-end reporting runs and shut it down after, paying for compute only when used. It also means the same Iceberg tables can be read by Trino, Spark, or DuckDB — no engine owns the data.

Q: What is the relationship between Trino, Starburst, and AWS Athena?
A: Trino is the open-source engine; Starburst is a commercial distribution and managed platform built on Trino (Starburst Galaxy/Enterprise) adding connectors, security, and support; AWS Athena (v3) is a serverless managed query service whose engine is based on Trino. So the architecture and SQL you learn on open Trino transfer directly to those managed offerings — they are the "managed Trino-style" options the lesson lists.

Q: How does Athena's pricing model differ from running your own Trino cluster, and why does it matter?
A: Athena's standard model charges per terabyte of data scanned by each query (with a serverless, no-cluster-to-manage experience), whereas self-hosted Trino charges you for the compute infrastructure you run regardless of bytes scanned. This means Athena rewards aggressive partition pruning and projection (scan-less columns cost nothing) and can be cheaper for sporadic use, while a busy, always-on Trino cluster may be cheaper at high constant utilization. The architect must match the pricing shape to the workload's duty cycle.

Q: What does the Trino Iceberg connector let you do that a raw Parquet-on-S3 setup does not?
A: The Iceberg connector reads the Iceberg metadata tree, so it gives you ACID snapshots, time travel, schema and partition evolution, hidden partitioning, and transactional `UPDATE`/`DELETE`/`MERGE` — all things plain Parquet files lack. It also exposes Iceberg metadata tables and maintenance procedures for inspection and housekeeping. In short, it turns a pile of files into a versioned, mutable, governed table.

Q: How do you time-travel to a specific snapshot ID in Trino's Iceberg connector?
A: Use `SELECT ... FROM table FOR VERSION AS OF <snapshot_id>`, for example `SELECT * FROM nav FOR VERSION AS OF 8954597067493422955`. This reads the table exactly as it existed at that committed snapshot, ignoring any later writes. It is how you pin a query to a known-good state, such as the snapshot just before a NAV correction landed.

Q: How do you time-travel to a point in time (a timestamp) in Trino's Iceberg connector?
A: Use `SELECT ... FROM table FOR TIMESTAMP AS OF TIMESTAMP '2026-06-12 17:00:00.000 Europe/Luxembourg'`. Trino resolves this to the most recent snapshot at or before that timestamp and queries it. This is ideal when you know "as of end of business yesterday" but not the exact snapshot ID.

Q: How would you reproduce yesterday's NAV report after a correction landed today, using Trino?
A: Identify the snapshot that was current at yesterday's cutoff — either by its ID from the `$snapshots`/`$history` metadata table or by timestamp — then run the report query with `FOR TIMESTAMP AS OF` or `FOR VERSION AS OF` pinned to that snapshot. Because each Iceberg commit is an immutable snapshot, the correction made today does not alter the historical snapshot, so the prior figures reproduce exactly. This auditable reproducibility is a core reason Iceberg fits regulated fund data.

Q: Which Iceberg metadata table in Trino lists the table's snapshots, and what does it tell you?
A: Querying `"table$snapshots"` returns one row per snapshot with its committed timestamp, snapshot ID, parent snapshot ID, operation type (append/overwrite/delete), and a summary of changed files/rows. You use it to find the snapshot ID or timestamp to time-travel to, and to see the commit history of the table. It is the starting point for "what changed and when."

Q: What does the `"table$files"` metadata table expose in Trino's Iceberg connector?
A: It lists the data files in the current snapshot with details like file path, format, record count, file size, and per-column value counts and bounds (min/max). Reading it lets you diagnose file-count and file-size problems — for example discovering thousands of tiny files that need compaction. This is the "debug why is this table slow/huge from the manifests alone" skill applied through Trino.

Q: What does `"table$manifests"` show and why might you read it?
A: It describes the manifest files referenced by the current snapshot, including each manifest's path, length, partition summaries, and added/existing/deleted data-file counts. You read it to understand how the metadata is organized and to spot manifest bloat — too many manifests slows planning. It is a layer above `$files` in the metadata tree.

Q: What does `"table$partitions"` give you?
A: It summarizes the table per partition: record counts, file counts, and size per partition value, plus column bounds. For a NAV table partitioned by date, it tells you how rows and files are distributed across `dt`, which is invaluable for spotting skew or partitions with too many small files. It is the quickest way to sanity-check that partitioning is doing what you intended.

Q: What does the `optimize` procedure do in Trino's Iceberg connector and when do you run it?
A: `ALTER TABLE ... EXECUTE optimize` rewrites small data files into larger ones (by default merging files under roughly 100 MB) per partition, improving read performance by reducing file count and metadata overhead. You run it after many small incremental writes — for example frequent NAV corrections or streaming-style appends that produce a small-file problem. You can target specific partitions with a `WHERE` clause to limit the rewrite scope.

Q: What does `expire_snapshots` do, and what is the safety implication?
A: `ALTER TABLE ... EXECUTE expire_snapshots` removes old snapshots and the data/metadata files they exclusively reference, reclaiming storage. The safety implication is that once a snapshot is expired you can no longer time-travel to it, so you must keep enough retention for your audit and reproducibility requirements — the default minimum retention is 7 days. In a regulated fund context, set retention to satisfy the longest "reproduce the report" obligation before expiring.

Q: What does `remove_orphan_files` do and why is it separate from `expire_snapshots`?
A: `remove_orphan_files` deletes files in the table's storage location that are no longer referenced by any table metadata — for example leftovers from failed writes — older than a retention threshold. It is separate because orphans are not tracked by snapshots at all, so expiring snapshots cannot find them; this procedure scans the storage directory directly. Run it cautiously, since an aggressively short retention could delete files from an in-flight write.

Q: How do you evolve an Iceberg table's schema from Trino, and what stays unchanged for readers?
A: You run standard DDL like `ALTER TABLE nav ADD COLUMN currency varchar`, and Iceberg records the change as metadata without rewriting existing data files. Existing queries keep working because Iceberg tracks columns by stable field IDs, so old files simply return null for the new column. This is why "add a column" does not break downstream consumers — a key Iceberg guarantee surfaced through Trino.

Q: What is hidden partitioning and how does it change how you write Iceberg queries in Trino?
A: Hidden partitioning means Iceberg derives partition values from a transform on a column (e.g. `day(nav_date)`) and stores them in metadata, so you query on the natural column and Iceberg prunes partitions automatically. You write `WHERE nav_date = DATE '2026-06-12'` rather than filtering on a separate physical partition column. This avoids the classic mistake of forgetting to filter the partition column and accidentally scanning everything.

Q: What is partition evolution and why does it matter operationally?
A: Partition evolution lets you change a table's partitioning scheme (e.g. from monthly to daily) going forward without rewriting historical data, because Iceberg records which spec each data file was written under. Old files keep their old partitioning, new files use the new one, and queries still work across both. Operationally it means you can fix a poor partition choice without a massive rewrite — though you may still optimize old data later.

Q: Which Iceberg catalog types does the Trino connector support, set via which property?
A: The connector supports Hive Metastore (default), AWS Glue, JDBC, REST, Nessie, and Snowflake catalogs, selected with `iceberg.catalog.type`. For the capstone you move from a Hive-style metastore to a REST catalog (Lakekeeper or Polaris) by setting `iceberg.catalog.type=rest` plus the REST endpoint URI. The data files and Parquet format do not change — only the catalog pointer does.

Q: After swapping Trino's Iceberg catalog from Hive to REST, the same tables must resolve. What changed and what didn't?
A: What changed is the catalog configuration — `iceberg.catalog.type` and the catalog endpoint/URI Trino consults to find table pointers and current snapshots. What did not change is the actual data: the Parquet data files, the Iceberg metadata/manifest files in MinIO, and the table format itself. That clean separation — engine and catalog swappable, data untouched — is precisely why open formats plus an open catalog de-risk lock-in.

Q: What is credential vending and how does it relate to Trino reading Iceberg via a REST catalog?
A: Credential vending is the catalog brokering short-lived, scoped storage credentials to the engine so Trino can read/write only the specific table's files in object storage, rather than holding broad static keys. With a REST catalog like Polaris, Trino asks the catalog for a table and receives both the metadata pointer and temporary credentials to access its files. This centralizes governance — the catalog decides who can touch which regulated data, which is a control a fund administrator needs.

Q: What is the cost-based optimizer (CBO) in Trino and what does it use to make decisions?
A: The CBO chooses an efficient query plan — particularly join order and join distribution type — using table and column statistics such as row counts, distinct-value counts, null fractions, and min/max ranges. With good stats it can put the smaller table on the build side of a join and pick broadcast vs partitioned joins wisely. Without stats it falls back to heuristics and can pick a poor plan that, for example, broadcasts a huge table.

Q: Why are missing or stale statistics a common cause of a bad Trino plan?
A: The CBO relies on stats to estimate intermediate sizes; if they are missing or stale, it may misjudge which side of a join is smaller and choose the wrong join distribution or order, leading to giant broadcasts or shuffles. For Iceberg you can collect stats with `ANALYZE`, and for JDBC sources the connector may read the database's own statistics. When a plan looks wrong, checking and refreshing statistics is an early diagnostic step.

Q: What is the difference between a broadcast join and a partitioned (hash) join in Trino?
A: In a broadcast join the small (build) table is replicated in full to every worker, so each worker can join its slice of the large table locally with no shuffle of the large side — great when one side is genuinely small. In a partitioned join both sides are hash-redistributed by join key across workers via exchanges, which is necessary when both sides are large. Choosing wrongly — broadcasting a big table — blows up memory and network, which is why the CBO's size estimates matter.

Q: How do you read a Trino `EXPLAIN` plan to find the most expensive operation?
A: Read the plan as a tree from the leaves (TableScans) upward, noting estimated rows and CPU/memory at each operator; the operators with the largest row counts and the big exchanges are usually the hotspots. With `EXPLAIN ANALYZE` you also get actual rows, wall time, and CPU per operator, so you can see where real time was spent. Watch especially for unfiltered remote scans, large hash joins, and big partitioned exchanges.

Q: What is the difference between `EXPLAIN` and `EXPLAIN ANALYZE`?
A: `EXPLAIN` shows the planned query — the operator tree and the optimizer's estimates — without running the query. `EXPLAIN ANALYZE` actually executes the query and annotates the plan with real runtime metrics (actual rows, time, CPU, memory per stage/operator). Use `EXPLAIN` to check pushdown and join strategy cheaply; use `EXPLAIN ANALYZE` when you need to know where real time went, accepting that it runs the query.

Q: You run `EXPLAIN` and see a filter as a separate `ScanFilterProject`/`Filter` node above a Postgres scan. What does that tell you?
A: It tells you the predicate did not push down into Postgres — Trino is fetching rows and filtering them itself, which for a large table means heavy network transfer. The fix is to make the predicate translatable (plain column comparison, matching types) or to add the filter in a form the connector accepts, and re-check `EXPLAIN` for the predicate folded into the scan. This is the exact diagnostic the lesson's "confirm it with EXPLAIN" outcome asks for.

Q: How does Trino's `EXPLAIN (TYPE DISTRIBUTED)` help compared with the default logical plan?
A: `EXPLAIN (TYPE DISTRIBUTED)` shows the plan split into stages and fragments with the exchange types between them, revealing how work is distributed and where data is shuffled or broadcast across the cluster. The default `EXPLAIN` (logical) shows operators but not the stage/exchange boundaries. For diagnosing a chatty or shuffle-heavy federated query, the distributed plan makes the data-movement structure explicit.

Q: A federated query works in dev but times out at month-end with concurrency. What cluster-level cause should you suspect?
A: Suspect resource contention: many concurrent queries competing for worker memory and CPU, possibly hitting the concurrent-query limit or the cluster memory pool, so queries queue or fail. Trino's resource groups let you cap concurrency and memory per workload class so an analytics burst does not starve everything. Also check whether each query is well-pushed-down — chatty queries amplify the problem under concurrency.

Q: What are resource groups in Trino and why would a fund admin use them?
A: Resource groups partition cluster capacity into named pools with limits on concurrent queries, queued queries, and memory, optionally with scheduling weights and selectors that route queries by user/source. A fund admin would use them to isolate, say, scheduled regulatory-reporting jobs from ad-hoc analyst queries so neither can monopolize the cluster at month-end. They turn "everyone shares one cluster" into a governed, prioritized arrangement.

Q: Why is Trino described as the "practical face of decoupled compute" for a lakehouse?
A: Because it sits as an interchangeable compute layer over open storage (object store), an open table format (Iceberg), and an open catalog, querying them with standard SQL — and it can be added or removed without migrating data. You prove the decoupling concretely: the same Iceberg tables that Trino reads can also be read by DuckDB or Spark, and you can swap the catalog underneath without touching files. It makes the abstract lakehouse promise tangible in one stack.

Q: In a single Trino statement, how do you join Postgres reference data to Iceberg NAV history?
A: Reference each table by its fully qualified `catalog.schema.table` name and write an ordinary join, e.g. `SELECT r.fund_name, n.nav FROM postgres.public.fund_ref r JOIN iceberg.funds.nav_history n ON r.isin = n.isin WHERE n.dt = DATE '2026-06-12'`. Trino's coordinator plans a federated query that scans each source through its connector and performs the join in the engine. The filter on `n.dt` should push into the Iceberg scan, and `r.isin` becomes a dynamic filter into the NAV scan if the optimizer chooses it.

Q: Why is "no ETL hop" a meaningful advantage for joining live transfer-agency data to historical NAV?
A: Because you read the transfer-agency reference data live from Postgres at query time rather than running a nightly batch to copy it into the lake, so the join always reflects the current account/holder/share-class state. This removes a pipeline to build, schedule, monitor, and reconcile, and eliminates the staleness window between the source change and the copy. The tradeoff is the live-query load on Postgres, which you manage with pushdown, replicas, and concurrency limits.

Q: What is an ISIN, and how is it typically used as a join key across Trino catalogs?
A: An ISIN (ISO 6166) is a 12-character International Securities Identification Number that uniquely identifies a security/share class, structured as a 2-letter country code, a 9-character national security identifier, and a final check digit. In a federated query it is the natural join key linking instrument-level reference data in Postgres to NAV facts in Iceberg. The gotcha is treating it as eternally stable — corporate actions can change an ISIN, so a robust security master also carries an internal surrogate key.

Q: What is NAV and why is reproducibility of NAV history a driver for using Iceberg under Trino?
A: NAV (Net Asset Value) is the per-share value of a fund, the central figure a fund administrator computes and publishes daily. Reproducibility matters because corrections happen, and auditors ask "what NAV did we publish yesterday and can you reproduce it?" — Iceberg snapshots plus Trino time travel answer that precisely. You pin the report to the snapshot that was current at publication, immune to later corrections.

Q: How would Trino help validate or assemble an EMT (European MiFID Template) output?
A: Trino can federate live target-market and costs reference data in Postgres against historical fund facts in Iceberg to assemble the EMT-shaped columns in one query, and you can `EXPLAIN` it to ensure the joins are efficient. It is a read/transform engine, so it is well suited to producing the EMT data product and to spot-checking which upstream attributes are populated. Missing upstream attributes (e.g. a half-populated costs block) surface as nulls in the federated result before distribution.

Q: A federated query that builds an EMT extract returns nulls in the costs block. Is that a Trino bug? How do you check?
A: Almost certainly not a Trino bug — nulls usually mean the upstream attributes are genuinely missing or the join did not match. Check by querying the source tables directly through their catalogs to confirm whether the costs data exists and whether the join keys (e.g. ISIN, share class) line up, and verify no predicate is silently filtering rows. Trino is faithfully reporting the data; the gap is in the upstream completeness the EMT requires.

Q: What is an LEI and could you federate GLEIF LEI data with fund facts in Trino?
A: An LEI (Legal Entity Identifier, ISO 17442) is a 20-character code identifying a legal entity such as a fund's management company, maintained in the GLEIF registry. Yes — you can load the GLEIF golden copy into Iceberg bronze and federate it, or keep it in Postgres, then join it by LEI to fund/entity records in another catalog. Trino lets you enrich fund facts with entity reference data across stores in one statement.

Q: Why might you prefer copying a slowly changing reference dimension (e.g. a country code list) into Iceberg rather than federating it every query?
A: A tiny, rarely changing lookup is cheap to materialize once into Iceberg, after which every join is local, fast, and immune to the remote source's availability or load. Federating it on every query adds a remote round-trip and a dependency on the source being up, for data that almost never changes. The crossover favors copying when the dimension is small and stable and the join is frequent.

Q: How do you add Trino to a Docker Compose stack alongside Postgres and Iceberg?
A: Add a `trinodb/trino` service, mount catalog property files (e.g. `iceberg.properties`, `postgresql.properties`) into the container's `etc/catalog/` directory, expose port `8080`, and put it on the same Compose network as the Postgres and MinIO/metastore services. Each catalog file names its connector and connection details so Trino discovers both sources on startup. You then connect with the Trino CLI or a JDBC client to run federated queries.

Q: What port does Trino's coordinator and Web UI listen on by default?
A: By default the Trino coordinator's HTTP server, which also serves the Web UI and the client API, listens on port `8080`, configurable via `http-server.http.port`. The CLI and JDBC clients connect to that same endpoint, e.g. `--server localhost:8080`. In Compose you publish `8080` to reach both the UI and the query API.

Q: What can you learn from the Trino Web UI when diagnosing a slow query?
A: The Web UI lists running and completed queries with their state, elapsed time, and per-stage details including rows processed, CPU time, memory, and the operator breakdown. You use it to find which stage is the bottleneck, whether a query is blocked or spilling, and how splits are distributed across workers. It is the live counterpart to `EXPLAIN ANALYZE` for queries already in flight.

Q: How do you connect to Trino with the CLI and run a quick federation test?
A: Use the Trino CLI: `trino --server localhost:8080 --catalog iceberg --schema funds`, then run a statement joining across catalogs, e.g. selecting from `iceberg.funds.nav_history` joined to `postgres.public.fund_ref`. The default catalog/schema just set the unqualified-name context; you can always fully qualify names to reach any catalog. This is the fastest way to confirm both connectors resolve their tables.

Q: What does it mean that Trino "has no storage of its own," and how does that shape architecture decisions?
A: It means Trino persists no user data — all tables live in external systems reached through connectors — so it is purely a compute/query tier you can stop and start freely. Architecturally this pushes durability, format, and catalog decisions out to MinIO/Iceberg/Postgres, and makes Trino a swappable component rather than a lock-in point. You design storage and catalog for permanence and treat the engine as ephemeral capacity.

Q: How does Trino achieve parallelism within a single query?
A: It splits each source's data into many splits, runs multiple tasks (one stage's work per worker) across all workers, and within each task runs multiple drivers across CPU cores. So parallelism comes from three nested levels: tasks across nodes, drivers across cores, and splits as the divisible unit of input. More workers and well-sized splits increase the achievable parallelism.

Q: A query is slow and the Web UI shows almost all time in one stage with very few splits. What is the likely cause?
A: Low parallelism — the connector produced too few splits, so only a handful of tasks/drivers do the work while other workers sit idle. This often happens with a small number of large files or a source that does not split well; the fix is to write more, appropriately sized files (or compact tiny ones into right-sized ones), or use a connector/format that splits better. Adding workers will not help if there are not enough splits to keep them busy.

Q: Why can a single huge Parquet/Iceberg data file hurt Trino parallelism?
A: If a file cannot be split into multiple ranges, the whole file may be read by one task, so one worker does all that work while others idle, capping parallelism for that portion of the scan. Iceberg/Parquet readers can often split within a file by row group, but very large files with few row groups still limit concurrency. This is why right-sizing files (via `optimize`) matters for read performance, not just storage.

Q: How is Trino different from a traditional MPP data warehouse like Teradata or Vertica?
A: Traditional MPP warehouses tightly couple their proprietary storage and compute and ingest data into their own format, optimizing within that closed system. Trino separates compute entirely from storage, querying open formats and external databases in place without owning the data, and federates across heterogeneous sources. The tradeoff: warehouses often win on tightly integrated concurrency and indexing, while Trino wins on openness, federation, and storage/compute decoupling.

Q: What does the warehouse "still do better" than a federated engine like Trino, per the lakehouse argument?
A: A mature warehouse typically offers higher sustained concurrency, finer-grained security (row/column-level controls), richer indexing/clustering, and smoother ergonomics for many simultaneous BI users. Trino is excellent for federation and ad-hoc analytics over open data but can lag a tuned warehouse on these dimensions. The architect blends them — lakehouse with Trino for the open core, a warehouse where concurrency and fine-grained security demand it.

Q: When is federation simply the wrong tool, and what do you do instead?
A: Federation is wrong when the join is run constantly, both sides are large, predicates do not push down, or the remote OLTP system cannot bear the load — the chatty cross-store pattern. In those cases you materialize the data into the lake (ingest the dimension into Iceberg via a batch ELT or Airbyte sync) so the join becomes a single-store, well-optimized query. Federation shines for occasional, well-filtered joins against live data, not for hot, heavy, repeated joins.

Q: What is the role of `ANALYZE` in Trino and when should you run it?
A: `ANALYZE table` collects table and column statistics (row counts, distinct values, null fractions, ranges) that the CBO uses to plan joins and distributions well. You run it after significant data changes to an Iceberg table, or when `EXPLAIN` shows wildly wrong row estimates leading to poor join choices. Up-to-date stats are what let the optimizer pick the right build side and avoid broadcasting large tables.

Q: How do you target a single partition when running `optimize` to avoid rewriting the whole table?
A: Add a `WHERE` clause that constrains the partition column, e.g. `ALTER TABLE nav EXECUTE optimize WHERE dt = DATE '2026-06-12'`, so only that partition's small files are rewritten. This limits I/O and runtime, which matters on a large historical NAV table where rewriting everything would be expensive. Scoping maintenance to recently written partitions is the usual operational pattern.

Q: A daily NAV-correction job produces thousands of tiny Iceberg files. What is the symptom in Trino and the fix?
A: The symptom is slow scans and slow query planning because Trino must open and read metadata for thousands of files (high per-file overhead and bloated manifests). The fix is to run `ALTER TABLE ... EXECUTE optimize` to compact the small files into right-sized ones, optionally scoped to recent partitions, and periodically rewrite manifests. You can confirm the small-file problem first by querying `"table$files"` or `"table$partitions"`.

Q: Why does Trino fail a query rather than swap or thrash when it runs out of memory?
A: Trino enforces explicit memory limits and proactively kills a query that exceeds them, choosing predictable failure of one query over an unpredictable, cluster-wide degradation or OOM crash. This protects other concurrent queries and overall cluster stability. The tradeoff for the user is an "exceeded memory limit" error, which you then address with spill, more memory/workers, or a more selective query.

Q: What is the practical difference between user memory and system/total memory limits in Trino?
A: User memory is the memory consumed by operators directly attributable to query logic (hash tables, aggregation state), bounded by limits like `query.max-memory-per-node`; total memory additionally includes system overhead such as buffers used by exchanges and readers. A query can be killed for exceeding either, and the error message indicates which limit was hit. Knowing which one tripped tells you whether to shrink your join/aggregation (user) or reduce concurrency/buffering pressure (system).

Q: How do you decide between adding workers and adding memory per worker for a memory-bound Trino workload?
A: Adding workers spreads partitioned joins/aggregations across more nodes, lowering per-node memory pressure and increasing parallelism — best when work partitions well. Adding memory per worker helps when a single operator's state (e.g. a broadcast build side) must fit on one node regardless of parallelism. Diagnose from the plan: if one node holds a broadcast table, more memory; if partitioned exchanges are saturating nodes, more workers.

Q: Why might `EXPLAIN ANALYZE` itself be undesirable in production for a heavy query?
A: Because `EXPLAIN ANALYZE` actually executes the query to gather real metrics, so running it on an already-slow or resource-heavy query incurs that full cost and load, potentially affecting other users. For planning checks that do not require runtime numbers, plain `EXPLAIN` is free of execution cost. Reserve `EXPLAIN ANALYZE` for when you genuinely need actual rows/time and can afford the run.

Q: What is the danger of running federated analytic queries against the production transfer-agency Postgres at month-end specifically?
A: Month-end is exactly when the transfer-agency system is busiest processing subscriptions, redemptions, and reconciliations, so heavy Trino scans then compound contention and can slow or stall operational processing. The architect mitigates by reading from a read replica, scheduling heavy federation off-peak, or materializing the needed reference data into the lake beforehand. The lesson is that federation's "live data" benefit carries an operational-load cost that peaks at the worst time.

Q: How do branches and tags in Iceberg interact with Trino time travel?
A: Iceberg supports named references — branches and tags pointing at specific snapshots — and Trino can query them, for example `FOR VERSION AS OF 'eom-2026-05'` against a tag. Tags give you stable, human-meaningful pointers to important snapshots (like a month-end close) so you do not need to remember raw snapshot IDs. This is convenient for reproducing regulatory reports tied to known business cut-offs.

Q: What does it mean that Trino executes queries "in memory" and streams between stages, and what is the consequence?
A: Trino is a pipelined, in-memory engine: operators stream pages to downstream operators and across stages via exchanges rather than materializing full intermediate results to disk by default. The consequence is low latency for interactive queries, but it also means very large intermediates must fit in memory unless spill is enabled — hence memory-limit failures on big joins. Spill is the opt-in escape hatch when an intermediate genuinely cannot fit.

Q: How do you simulate a late-arriving holding in an Iceberg NAV/holdings table from Trino, and what command fits best?
A: Use `MERGE INTO holdings AS t USING staged AS s ON t.isin = s.isin AND t.dt = s.dt WHEN MATCHED THEN UPDATE ... WHEN NOT MATCHED THEN INSERT ...`, which upserts the late row in one transactional statement. Trino's Iceberg connector supports `MERGE`, `UPDATE`, and `DELETE` because Iceberg provides the ACID metadata layer underneath. Each such write creates a new snapshot, so the late arrival is fully auditable and you can still time-travel to the state before it landed.

Q: What is the difference between copy-on-write and merge-on-read for an Iceberg table, and why does it matter when Trino reads it?
A: Copy-on-write (COW) rewrites whole data files on `UPDATE`/`DELETE`, making writes more expensive but reads fast because there are no delete files to reconcile. Merge-on-read (MOR) writes lightweight delete files instead, making writes cheap but reads slower because Trino must apply the delete files at scan time. For a NAV table with frequent corrections you trade write cost against read latency; if MOR reads feel slow, running `optimize` compacts away the delete files and restores read speed.

Q: When configuring the Trino Iceberg connector against a Nessie catalog, what makes Nessie distinctive?
A: Nessie is a Git-like catalog that adds branching, tagging, and commits across tables, so you can isolate changes on a branch and merge them, not just time-travel within one table. You select it with `iceberg.catalog.type=nessie` plus the Nessie endpoint, and Trino then resolves tables through Nessie's versioned references. It is useful for staging a multi-table change (e.g. a coordinated reference-data update) before exposing it to consumers.

Q: A SWIFT MT535 holdings statement feeds a transfer-agency table you want to join to Iceberg NAV in Trino. What identifier and pushdown concerns arise?
A: MT535 statements carry instrument identifiers (typically ISIN) and account/safekeeping references that become the join keys to the NAV facts, so the join key quality and consistency drive whether rows match. For performance, ensure the date and ISIN predicates push into the Iceberg scan and the transfer-agency Postgres side respectively, so neither store ships unfiltered rows. The architect must also watch for lossy mapping — an MT535 may carry coarser data than your richer model, producing nulls that are a source-format limitation, not a Trino fault.

Q: Summarize the architect's checklist for keeping a federated Trino join healthy.
A: Confirm predicates push down to each source with `EXPLAIN`; ensure the small side is filtered and used as the build/dynamic-filter side; keep statistics fresh so the CBO picks the right join strategy; watch for chatty unfiltered remote scans and either fix pushdown or materialize the dimension; and protect the cluster and source systems with resource groups, replicas, and off-peak scheduling. Above all, know when federation is the wrong tool and copy the data instead. These map directly to the lesson's three "Done when" outcomes.

Q: Which session property forces a specific join distribution, and what are its three values?
A: Set `join_distribution_type` per session with `SET SESSION join_distribution_type = 'BROADCAST'` (or `'PARTITIONED'` or `'AUTOMATIC'`). `AUTOMATIC` lets the cost-based optimizer choose, `BROADCAST` replicates the right/build side to every worker, and `PARTITIONED` hash-redistributes both sides across the cluster. You override `AUTOMATIC` to `PARTITIONED` when you know stats are unreliable and a broadcast would blow up memory on a deceptively large build side.

Q: A broadcast join is silently chosen for a large table and the query OOMs on one node. What is the session-level quick fix?
A: Run `SET SESSION join_distribution_type = 'PARTITIONED'` to force hash-partitioned joins so neither side must fit on a single node, then re-run the query. The root cause is usually missing or stale statistics making the optimizer think the build side is small, so `ANALYZE` on the Iceberg table is the durable fix. Forcing `PARTITIONED` trades some speed for the ability to complete without exceeding `query.max-memory-per-node`.

Q: What exact error code does Trino raise when a query exceeds a single worker's memory budget, and what does it signal?
A: It raises `EXCEEDED_LOCAL_MEMORY_LIMIT`, signalling that an operator on one worker (commonly a hash-join build side or a broadcast table) tried to use more memory than `query.max-memory-per-node` allows on that node. It is distinct from `EXCEEDED_GLOBAL_MEMORY_LIMIT`, which is the cluster-wide `query.max-memory` cap across all workers. Seeing the local variant points you at per-node operator state, so the fix is partitioned joins, spill, or more workers rather than just raising the global cap.

Q: How do you disable dynamic filtering for one query, and why might you do that during diagnosis?
A: Run `SET SESSION enable_dynamic_filtering = false` before the query; it is enabled by default and this turns off the runtime build-side-to-probe-side predicate. You do this temporarily to measure how much a dynamic filter is actually helping (or to rule it out when a join behaves oddly), comparing the plan and timings with and without it. In normal operation you leave it on, since on a federated NAV join it is often what keeps the large Iceberg scan small.

Q: Which join shapes does Trino's dynamic filtering support, and which one helps an `IN`-style semi-join?
A: Dynamic filtering supports inner and right joins with `=`, `<`, `<=`, `>`, `>=`, or `IS NOT DISTINCT FROM` conditions, plus semi-joins expressed with `IN`. The semi-join case means a query like `WHERE isin IN (SELECT isin FROM filtered_funds)` can push the collected `isin` set into the large NAV scan. A `LEFT JOIN` keeping all probe rows or a non-equi condition outside that list will not generate a useful dynamic filter, so the large side stays unpruned.

Q: When the build side is too large, what does Trino collect for the dynamic filter instead of a value set, and what is the tradeoff?
A: Instead of an exact distinct-value set, Trino falls back to collecting per-column min/max bounds once the build side exceeds the `max-distinct-values-per-driver`/`max-size-per-driver` thresholds. The tradeoff is coarser pruning: a min/max range filter prunes whole files or partitions outside the range but cannot skip individual non-matching keys inside it. For a NAV scan this still helps when the join keys cluster in a narrow range, but a wide-ranging build side yields a near-useless range filter.

Q: How do you list the active nodes and confirm the coordinator/worker split from SQL alone?
A: Query `SELECT * FROM system.runtime.nodes`, which returns one row per node with its `node_id`, HTTP URI, version, `state` (e.g. `active`), and a `coordinator` boolean. This is the SQL equivalent of reading the Web UI's cluster page and is the fastest way to confirm a worker registered after a Compose restart. If a worker you started is missing here, it never reached the discovery service.

Q: How do you inspect or kill a runaway query without the Web UI?
A: Query `system.runtime.queries` to find the query's `query_id`, `state`, elapsed time, and user, then run `CALL system.runtime.kill_query(query_id => '20260615_101500_00007_abcde', message => 'month-end runaway')`. This is invaluable when a chatty federated scan is hammering Postgres and you need to stop it from the CLI. The `query_id` format encodes the date, time, sequence, and a random suffix, which also helps you correlate it with logs.

Q: What is the `system.metadata.materialized_views` versus a plain view in Trino, and which helps a repeated federated join?
A: A regular `VIEW` just stores SQL and re-executes the underlying (possibly cross-store) query every time, so it does not reduce remote load. A materialized view stores the query's results in a backing table (for Iceberg-backed catalogs) and serves reads from that snapshot until refreshed, turning a hot federated join into a local scan. For a dashboard hitting the same Postgres-to-Iceberg join all day, a materialized view refreshed off-peak is the middle ground between live federation and a full ETL copy.

Q: How do you refresh a Trino materialized view, and what governs whether reads are fresh?
A: Run `REFRESH MATERIALIZED VIEW funds.emt_extract`, which recomputes the underlying query and writes a new snapshot to the storage table. Between refreshes, reads return the last materialized result, so freshness is exactly as current as your last `REFRESH` — there is a staleness window you must schedule around. For regulated outputs you document the refresh cadence so consumers know the as-of time of the data they are reading.

Q: A federated query is correct but you want a cheap test harness with no external sources. Which built-in connector do you use?
A: Use the `tpch` connector (catalog usually named `tpch`), which generates standard TPC-H tables on the fly at a chosen scale factor, e.g. `SELECT * FROM tpch.sf1.orders`. It needs no external storage, so it is ideal for testing join distribution, dynamic filtering, and `EXPLAIN` behaviour before pointing at real Iceberg and Postgres. The companion `tpcds` connector does the same for the TPC-DS schema when you need more complex queries.

Q: How does scale factor in the `tpch` connector map to schema names, and why is that handy?
A: Each scale factor is exposed as a schema like `tiny`, `sf1`, `sf100`, `sf1000`, so `tpch.sf100.lineitem` is roughly 100 GB of generated rows. This lets you reproduce a memory-limit failure or a broadcast-vs-partitioned decision at a controlled size without provisioning real data. You scale the schema up until a join starts spilling or OOMing, which is a safe way to learn the cluster's envelope.

Q: What does `EXPLAIN (TYPE IO)` give you that the other EXPLAIN types do not?
A: `EXPLAIN (TYPE IO, FORMAT JSON)` reports, in JSON, the tables and columns the query reads and writes and the estimated bytes scanned per table. It is the planning-time way to predict scan cost — directly relevant on Athena where you pay per terabyte scanned — without executing the query. You use it to confirm partition pruning will limit bytes read before committing to a potentially expensive run.

Q: How do you collect statistics for only specific columns of an Iceberg table in Trino?
A: Use `ANALYZE iceberg.funds.nav_history WITH (columns = ARRAY['fund_id','dt'])` to gather stats just for the join and filter columns. This is cheaper than analyzing every column on a wide NAV table and targets exactly the columns the CBO needs to estimate join sizes. After it runs, re-check `EXPLAIN` to confirm the row estimates on those columns are now realistic.

Q: Where does Trino store the statistics it collects for an Iceberg table?
A: For Iceberg, `ANALYZE` writes table-level statistics (notably NDV/number-of-distinct-values sketches) into the Iceberg table's own metadata as a Puffin statistics file referenced by the current snapshot. Because the stats live with the table metadata, any engine reading the table can use them, and a new snapshot can carry forward or invalidate them. This is why re-running `ANALYZE` after a large write is the way to refresh estimates after the data shifts.

Q: How do you read an Iceberg table's key/value table properties from Trino, and name one property you would check?
A: Query the `"nav_history$properties"` metadata table to see configuration like `write.format.default`, `write.delete.mode`, `write.target-file-size-bytes`, and `format-version`. You would check `write.delete.mode` to learn whether the table is copy-on-write or merge-on-read, which explains read latency after many corrections. It is the metadata-only way to diagnose behaviour without guessing the table's write configuration.

Q: What is the default target file size the Iceberg connector aims for on write, and how do you change it?
A: The default is `1GB`, set by `iceberg.target-max-file-size` (with the per-table override `write.target-file-size-bytes`). For a small-file-prone NAV-correction table you might lower it so `optimize` produces consistently sized files, or confirm writers are not configured to emit tiny files. Knowing the default lets you judge whether the file sizes you see in `"table$files"` are normal or a misconfiguration.

Q: How do you roll an Iceberg table back to a prior snapshot from Trino, and how does that differ from time travel?
A: Run `ALTER TABLE funds.nav_history EXECUTE rollback_to_snapshot(8954597067493422955)`, which makes that snapshot the new current state going forward (a new commit). Time travel with `FOR VERSION AS OF` only reads an old snapshot for one query without changing the table, whereas rollback actually moves the table's head. You rollback when a bad write must be undone for all consumers; you time-travel when you just need to read the past.

Q: What does the `register_table` procedure do, and when is it the right tool?
A: `CALL iceberg.system.register_table(schema_name => 'funds', table_name => 'nav_history', table_location => 's3://lake/funds/nav_history')` makes an existing set of Iceberg metadata/data files known to the catalog without rewriting them. It is the right tool after a catalog swap or migration where the files exist in MinIO but the new catalog has no pointer to them. It complements the catalog change: files are untouched, you simply re-register the table so Trino can resolve it.

Q: What does the Iceberg `migrate` procedure convert, and what is the gotcha?
A: `CALL iceberg.system.migrate(schema_name => 'funds', table_name => 'legacy_holdings')` converts an existing Hive table (Parquet/ORC/Avro) in place into an Iceberg table by generating Iceberg metadata over the existing files. The gotcha is that it is one-way and the source must be a supported Hive table format; you should snapshot/back up first because the table identity changes. It is how you onboard legacy Hive data into the lakehouse without a full data rewrite.

Q: How do you read an Iceberg table's `$history` table, and what is the `is_current_ancestor` column for?
A: Query `SELECT * FROM "nav_history$history"` to get rows of `made_current_at`, `snapshot_id`, `parent_id`, and `is_current_ancestor`. The `is_current_ancestor` boolean tells you whether a snapshot is part of the current table's lineage — after a rollback, snapshots that were abandoned show `false`, so you can distinguish live history from orphaned branches. This is subtly different from `$snapshots`, which lists every snapshot with its operation and file summary regardless of lineage.

Q: How do you constrain `expire_snapshots` to keep at least a chosen retention, and what is the default minimum?
A: Run `ALTER TABLE funds.nav_history EXECUTE expire_snapshots(retention_threshold => '30d')` to keep 30 days of snapshots; the connector enforces a default minimum retention of `7d` via `iceberg.expire-snapshots.min-retention`. Trying to pass a value below that minimum is rejected to protect you from destroying recent history. In a regulated fund context you set the threshold to cover your longest report-reproduction obligation, not just the 7-day floor.

Q: How do you create a named tag on an Iceberg snapshot from Trino so you can pin a month-end close?
A: Run `ALTER TABLE funds.nav_history EXECUTE create_branch` / `create_tag` style procedures, e.g. `ALTER TABLE funds.nav_history EXECUTE create_tag(tag_name => 'eom-2026-05')` against the current snapshot. Then you query it with `SELECT * FROM funds.nav_history FOR VERSION AS OF 'eom-2026-05'`, a stable human-readable pointer instead of a raw snapshot ID. Tags survive snapshot expiry within their retention and are the clean way to bookmark regulatory cut-offs.

Q: A `FOR TIMESTAMP AS OF` query returns an unexpected (older) snapshot. What is the likely cause?
A: Trino resolves the timestamp to the most recent snapshot committed at or before that instant, so if the write you expected committed slightly after your timestamp — or in a different time zone — you get the previous snapshot. The classic gotcha is a naive timestamp without a zone resolving differently than the commit's `made_current_at`; always supply a zoned literal like `TIMESTAMP '2026-06-12 17:00:00 Europe/Luxembourg'`. Cross-check against `"table$history".made_current_at` to confirm which snapshot your timestamp actually maps to.

Q: How do you point a Trino catalog at a read replica of the transfer-agency Postgres, and why is that the standard month-end mitigation?
A: In `postgresql.properties` set `connection-url=jdbc:postgresql://pg-replica:5432/ta` to the replica's host instead of the primary, leaving the SQL unchanged. This routes heavy federated analytic scans to the replica so they do not contend with subscription/redemption processing on the primary at month-end. The tradeoff is slight replication lag, which is acceptable for reference-data joins where second-level staleness does not change the result.

Q: How do you cap how many JDBC connections Trino opens against Postgres for a federated scan?
A: Configure connection-pool sizing in the PostgreSQL catalog (the connector pools JDBC connections), so a parallel scan does not open an unbounded number of sessions against the database. This protects a transactional Postgres from connection exhaustion when many splits each want a connection. Pairing a bounded pool with predicate pushdown keeps both the row count and the connection count that hit Postgres small.

Q: A `JOIN` on `CAST(ta.isin AS varchar) = ice.isin` does not push the filter or use a dynamic filter. Why, and what is the fix?
A: The `CAST` wraps the join key in an expression the connector cannot translate or match against raw column statistics, so neither predicate pushdown nor a dynamic filter can target the underlying column. The fix is to align the column types at rest (store `isin` as the same `varchar(12)` on both sides) so the join is a plain column-to-column equality. Once the cast disappears, the optimizer can push filters and generate the dynamic filter that prunes the NAV scan.

Q: How would you confirm whether an aggregation actually pushed into Postgres versus running in Trino?
A: Run `EXPLAIN` and look for the aggregate folded into the Postgres `TableScan` (the connector node shows the `GROUP BY`/aggregate it computed) rather than a separate `Aggregate` operator above the scan. If you see a standalone `Aggregate` consuming raw rows from the scan, the grouping ran in Trino and you shipped every row across the wire. The fix is to use aggregates and group keys the JDBC connector supports and to ensure aggregation pushdown is not disabled.

Q: Which session property disables JDBC aggregation pushdown, and when would you want it off?
A: Aggregation pushdown for JDBC connectors is governed by a catalog/session toggle (e.g. `<catalog>.aggregation_pushdown_enabled`), which you can set false per session to force aggregation in Trino. You might disable it when the source database computes the aggregate slowly or under load, preferring to pull (well-filtered) rows and aggregate on idle Trino workers. It is a deliberate tradeoff: move CPU off a busy OLTP Postgres at the cost of more network transfer.

Q: How do you write the result of a federated join into a new Iceberg table in one statement?
A: Use `CREATE TABLE iceberg.funds.emt_snapshot AS SELECT ... FROM postgres.public.fund_ref r JOIN iceberg.funds.nav_history n ON ...` (CTAS), which materializes the federated result as a managed Iceberg table. This is exactly how you "copy the dimension into Iceberg first" to compare timings, or how you snapshot a regulatory extract for audit. Each CTAS creates an Iceberg snapshot, so the materialized output is itself versioned and reproducible.

Q: What is `INSERT INTO ... SELECT` across catalogs used for, and what does it create on the Iceberg side?
A: `INSERT INTO iceberg.funds.fund_ref_local SELECT * FROM postgres.public.fund_ref` copies (or appends) the live Postgres dimension into a local Iceberg table, creating a new append snapshot each run. You use it to periodically refresh a materialized copy of a reference dimension so subsequent joins are single-store. Because every insert is a snapshot, you can time-travel to see exactly which version of the dimension a given report used.

Q: How do you measure the wall-clock difference between the federated and copied-dimension versions of the same join from the CLI?
A: Run each query with `EXPLAIN ANALYZE` (or read the elapsed time the CLI prints, and the Web UI's per-query timing) and compare total wall time and the rows/time on each scan node. The federated version will show a remote Postgres scan and exchange; the copied version shows a single-store Iceberg join with no remote round-trip. Recording both, as the lesson's exercise requires, is how you find the crossover where copying starts to win.

Q: What is the difference between `query.max-memory` and `query.max-total-memory`, and which counts exchange buffers?
A: `query.max-memory` caps the user memory a query may use across the whole cluster (operator state like hash tables), while `query.max-total-memory` caps user plus system memory, where system memory includes overhead such as exchange and reader buffers. A query can be killed for breaching either, and the error names which. Knowing the distinction tells you whether to shrink your join/aggregation (user) or reduce buffering/concurrency pressure (total/system).

Q: Why does enabling spill not always rescue a memory-bound join, and what error can still appear?
A: Spilling can still fail with `EXCEEDED_LOCAL_MEMORY_LIMIT` because Trino may be unable to divide the intermediate state into chunks small enough to fit in memory while loading them back, and historically join spilling is less reliable than aggregation/sort spilling. So spill is a best-effort safety valve, not a guarantee. When it fails you fall back to partitioned joins, more workers, more memory, or shrinking inputs with better predicates.

Q: Where should `spiller-spill-path` point, and what failure occurs if you ignore this?
A: Point `spiller-spill-path` at one or more fast, dedicated local disks with ample free space, not the OS root volume. If you ignore it and the spill volume fills, the query still fails (and you can degrade or fill the root filesystem, destabilizing the node). For sustained heavy workloads, spilling gigabytes is also slow, so the better long-term answer is more memory or workers rather than relying on disk.

Q: How do you keep the JVM from OOMing below Trino's own memory limits — what relationship must hold?
A: The JVM `-Xmx` heap must be larger than the sum of `query.max-memory-per-node` and `memory.heap-headroom-per-node`, so Trino's tracked limits trip before the JVM exhausts the actual heap. If you raise `query.max-memory-per-node` without raising `-Xmx`, you invite real JVM OutOfMemoryErrors that crash the worker instead of cleanly failing one query. The headroom setting reserves space for untracked allocations like JIT and connector buffers.

Q: What is a broadcast join's specific memory failure mode, and which side must fit where?
A: In a broadcast join the entire (post-filter) build/right side must fit in memory on every worker simultaneously, so if that side is larger than expected each node hits `EXCEEDED_LOCAL_MEMORY_LIMIT`. The probe/left side is streamed and need not fit, which is why broadcasting is only safe for a genuinely small build side. The remedy is `PARTITIONED` distribution (fits across all nodes) or fresh statistics so the optimizer stops misclassifying the build side as small.

Q: How do resource groups route a query to the right pool, and what configures that mapping?
A: Resource groups are defined with limits (concurrency, queued queries, memory, scheduling weight) and a list of selectors that match on attributes like user, source, or client tags to assign each incoming query to a group. You configure this via the resource-groups manager (a JSON file or a database-backed manager). A fund admin maps scheduled regulatory jobs and ad-hoc analyst queries to separate groups so a month-end burst in one cannot starve the other.

Q: A scheduled regulatory job is being queued behind ad-hoc analyst queries. How do resource groups fix this?
A: Put the regulatory job in its own resource group with a higher scheduling weight (or a guaranteed concurrency slot) and cap the ad-hoc group's concurrent queries, using selectors that route by the job's user or source tag. This guarantees the regulatory workload gets cluster capacity regardless of analyst activity. It converts an unmanaged free-for-all into a prioritized arrangement aligned to business criticality.

Q: How do you set a per-query wall-clock timeout so a runaway federated scan cannot run for hours?
A: Set the `query_max_execution_time` (or `query_max_run_time`) session property, e.g. `SET SESSION query_max_execution_time = '10m'`, so Trino aborts the query once it exceeds the limit. This protects a busy transfer-agency Postgres from an unbounded chatty scan and protects the cluster from a single hung query. You typically bake such limits into the resource group governing scheduled jobs.

Q: A federated query suddenly fails after a Trino version upgrade with `INSUFFICIENT_RESOURCES`. What changed and what do you check?
A: `INSUFFICIENT_RESOURCES` means the cluster could not satisfy the query's resource requirements (often memory) under the new version's accounting, which can change between releases. Check whether memory defaults or the query's plan (join distribution, dynamic-filter behaviour) shifted, compare `EXPLAIN` across versions, and confirm `-Xmx` versus `query.max-memory-per-node` still hold the right relationship. Upgrades can alter cost estimates, so re-running `ANALYZE` and reviewing the new plan is the diagnostic path.

Q: How does Trino fault-tolerant execution (Tasmania/`retry-policy=TASK`) change the month-end concurrency story?
A: Setting `retry-policy=TASK` with an exchange manager (e.g. spooling to object storage) lets Trino retry failed tasks instead of failing the whole query, trading some latency for resilience on long batch jobs. For month-end regulatory runs that must complete despite a transient worker hiccup, this is more appropriate than the default `QUERY` policy tuned for low-latency interactive use. The cost is the overhead of spooling intermediate data, so you reserve it for the batch resource group.

Q: Why is `domicile = 'LU'` a good predicate to push into the Postgres reference side of a fund join?
A: A Luxembourg-domiciled filter is highly selective on a multi-jurisdiction fund register, so pushing it into Postgres makes the database return only the relevant share classes before the join. That both shrinks the network transfer and supplies the dynamic filter that prunes the Iceberg NAV scan to those funds. It is the canonical "filter the small side hard so the join is cheap" move in this domain.

Q: How would you assemble an EET (European ESG Template) extract with Trino, and what new join risk appears?
A: You federate live ESG/sustainability attributes (often in Postgres or freshly loaded Iceberg) against fund and share-class facts, keyed by ISIN/share class, to populate the EET columns in one query. The new risk is sparse coverage: many ESG fields are optional or not yet reported, so the join legitimately yields nulls that are a data-completeness issue, not a Trino fault. As with EMT, you verify against the source tables to distinguish missing upstream data from a mis-joined key.

Q: For a T+1 settlement reconciliation, why does Iceberg time travel under Trino matter specifically?
A: Under T+1 the books move faster, so a same-day correction can land before you have reconciled the prior state, and time travel lets you query the table exactly as it stood at the T+1 cut-off with `FOR TIMESTAMP AS OF`. This lets reconciliation compare the as-published figures against later corrections without the correction overwriting the evidence. The immutable-snapshot model is what makes "what did we have at cut-off" answerable in a compressed settlement window.

Q: Under the EU AI Act, why is Trino's `EXPLAIN` and Iceberg snapshot lineage relevant if a federated query feeds an AI system?
A: If a federated Trino query produces training or inference inputs for an AI system in scope of the Act, you must evidence data provenance and governance, and Iceberg snapshots plus Trino time travel give you a reproducible, point-in-time record of exactly what data was used. `EXPLAIN` documents which sources and predicates shaped that dataset. Together they support the data-governance and traceability expectations rather than the AI Act mandating Trino itself.

Q: How does DORA's resilience expectation shape how you run federated Trino against transfer-agency Postgres?
A: DORA pushes operational resilience for critical financial systems, so you avoid letting analytic Trino scans threaten the live transfer-agency database — you read from a replica, cap concurrency with resource groups, and set query timeouts so a runaway query cannot degrade an in-scope operational system. The principle is isolating the analytic load from the system whose availability is regulated. Materializing the dimension into the lake is the strongest isolation when the join is hot.

Q: An auditor asks you to prove yesterday's published NAV is reproducible after today's correction. Walk through the exact Trino steps.
A: Find the snapshot current at yesterday's publication time from `SELECT made_current_at, snapshot_id FROM "funds.nav_history$history"` (or use a pre-created tag), then run the report with `... FROM funds.nav_history FOR TIMESTAMP AS OF TIMESTAMP '2026-06-13 18:00:00 Europe/Luxembourg'` or `FOR VERSION AS OF <snapshot_id>`. Because today's correction is a new, separate snapshot, the historical snapshot is unchanged and the figures reproduce exactly. You hand the auditor the snapshot ID and the pinned query as the evidence trail.

Q: Summarize, as a decision rule, how to choose between forcing `PARTITIONED`, enabling spill, and adding workers for a failing big join.
A: Force `PARTITIONED` when a misjudged broadcast is the cause (one node holds a too-large build side) — refresh stats so the CBO stops mispicking; enable spill as a low-effort safety valve when you cannot shrink inputs, accepting slower disk-backed execution and knowing join spill may still fail; add workers when the work partitions well and per-node memory pressure, not a single oversized operator, is the bottleneck. Always pair any of these with tighter predicates and fresh `ANALYZE` so the inputs and the plan are right in the first place. The order in practice is: fix stats and pushdown, force a sane distribution, then scale or spill.

Q: How do you view the statistics the optimizer currently holds for a table or query without running it?
A: Run `SHOW STATS FOR funds.nav_history` to see per-column row counts, distinct-value counts, null fractions, and data size, or `SHOW STATS FOR (SELECT ... WHERE ...)` to see estimates for a filtered subquery. Comparing these numbers against reality tells you whether the CBO is working from good information before you trust its join choices. If `SHOW STATS` reports nulls or wildly wrong row counts, run `ANALYZE` and re-check — stale stats here are the usual root cause of a bad plan.
