# Phase 3 · 9.4.3 Integration testing for pipelines — 150 self-test questions

Q: What is "integration testing" for a data pipeline, as distinct from unit testing?
A: Integration testing exercises the seams where your code meets real external systems — the database, object store, message broker — instead of mocked stand-ins. The point is to prove the pipeline actually connects, authenticates, respects the real schema, and survives real type coercion and error codes. In a regulated fund shop, "it passed against the mock" is not evidence the loader works against the actual Postgres.

Q: Why does the lesson argue that pipelines "fail at the seams"?
A: The business logic in a transform is usually simple and well-covered by unit tests; what breaks in production is the boundary — a dropped connection, a permission denied, a column type mismatch, an encoding surprise. Those failures only appear when real infrastructure is in the loop, so mocked tests pass forever while the real loader fails on the first `NUMERIC` overflow. Integration tests are the only honest test of those connections, schemas, and permissions.

Q: What is Testcontainers and what problem does it solve?
A: Testcontainers is a library that spins up real, ephemeral Docker containers (Postgres, MinIO, Kafka, etc.) for the lifetime of a test, then tears them down automatically. It solves the "test against something real but disposable" problem: you get a genuine database with genuine type checking and error codes, not a mock, and you never depend on a hand-maintained shared test server. The Python flavour is the `testcontainers` package on PyPI.

Q: How do you install the Postgres module of Testcontainers in a uv-managed project?
A: Add the extra-qualified package, for example `uv add "testcontainers[postgres]"`, which pulls the core library plus the Postgres module and its driver dependency. Per the project's rule you never call bare `pip`; uv manages the lockfile. You then import with `from testcontainers.postgres import PostgresContainer`.

Q: What is the import path for the Postgres container class in testcontainers-python?
A: `from testcontainers.postgres import PostgresContainer`. The library is namespaced by module, so MinIO comes from `testcontainers.minio`, Kafka from `testcontainers.kafka`, and the generic building block `DockerContainer` from `testcontainers.core.container`. Importing from the wrong module is a common first-run `ModuleNotFoundError`.

Q: What does `PostgresContainer("postgres:16").get_connection_url()` return and why is it convenient?
A: It returns a SQLAlchemy-compatible connection URL (e.g. `postgresql+psycopg2://test:test@localhost:49158/test`) pointing at the container's dynamically mapped host port. It is convenient because you can pass it straight into `sqlalchemy.create_engine(...)` without manually assembling host, port, user, password, and dbname. The port in the URL is the random host port, not the in-container `5432`.

Q: Why does Testcontainers map container ports to random host ports by default?
A: To avoid collisions — if it always bound `5432`, two parallel test runs (or a real Postgres on your machine) would clash. The container exposes `5432` internally, and Docker maps it to a free ephemeral host port that Testcontainers discovers at runtime. You must therefore never hard-code `5432` for the host connection; ask the container what port it got.

Q: How do you find out which host port a container's internal port was mapped to?
A: Call `container.get_exposed_port(5432)`, passing the in-container port number; it returns the host-side port as a string. Combined with `container.get_container_host_ip()` you can build a connection manually when you are not using `get_connection_url()`. Reading the mapped port at runtime is what makes parallel test execution safe.

Q: What does `get_container_host_ip()` give you and why not just hard-code `localhost`?
A: It returns the host address to reach the container, which is usually `localhost` but can differ — for example inside Docker-in-Docker or certain remote-Docker setups it resolves to the daemon's address. Hard-coding `localhost` works on a plain Ubuntu laptop but breaks the moment the test runs inside CI's own container. Asking the API keeps the test portable across environments.

Q: Show the canonical context-manager pattern for a Postgres Testcontainer.
A: `with PostgresContainer("postgres:16") as pg:` starts the container, yields it for the body, and stops it on exit even if an assertion fails. Inside you typically do `engine = sqlalchemy.create_engine(pg.get_connection_url())`. The `with` block guarantees cleanup, which is the whole point of throwaway infra — no orphaned containers after a failing test.

Q: Why pin an explicit image tag like `postgres:16` instead of `postgres:latest`?
A: Pinning makes the test deterministic and reproducible — `latest` silently moves and can change behaviour, default config, or even the major version between runs, breaking tests for reasons unrelated to your code. In a regulated estate you also want the test image to match the production Postgres major version. A floating tag turns "tests went red overnight" into an unsolvable mystery.

Q: What is the Ryuk container and what does it do in a Testcontainers run?
A: Ryuk is a small sidecar (the "resource reaper") that Testcontainers starts to clean up containers, networks, and volumes if your test process dies abruptly without running its teardown. It watches a labelled set of resources and removes them after the session, preventing leaked containers from piling up. You can disable it with `TESTCONTAINERS_RYUK_DISABLED=true`, but then you own cleanup yourself.

Q: A Testcontainers test hangs at startup pulling the image — what is happening and how do you make it predictable in CI?
A: On first use Docker pulls the image from a registry, which can be slow or rate-limited; the test appears to hang until the pull completes. Pre-pull images in a CI cache step (or use a registry mirror) so the test only starts an already-present image. Pinning the tag also lets the layer cache hit instead of re-pulling.

Q: What is `wait_for_logs` and why is it sometimes needed even though the container "started"?
A: `from testcontainers.core.waiting_utils import wait_for_logs` blocks until a given regex appears in the container's logs, e.g. waiting for `database system is ready to accept connections`. A container reporting "running" to Docker does not mean the service inside finished initialising, so connecting too early gives a connection-refused error. The Postgres module already waits, but for a generic `DockerContainer` you often add an explicit log or port readiness check yourself.

Q: Your test connects to the Postgres container and gets connection-refused intermittently — first thing to check?
A: A readiness race: the container is up but Postgres has not yet opened its socket, especially right after `start()`. Confirm you are using a wait strategy (the `PostgresContainer` waits on the "ready to accept connections" log by default) and that you are connecting to the mapped host port from `get_connection_url()`, not `5432`. Retrying the connect with backoff or using `wait_for_logs` removes the flakiness.

Q: What is the test pyramid, in one sentence, and who popularised the modern framing referenced here?
A: It is the idea that you should have many fast, cheap, isolated tests at the bottom, fewer integrated ones in the middle, and very few slow end-to-end tests at the top. The framing in this lesson follows Martin Fowler's "The Practical Test Pyramid". The shape comes from the cost and fragility of tests rising as you move up.

Q: How does the test pyramid map onto a data pipeline specifically?
A: The base is unit SQL and pure-function transform tests; the middle is component/integration tests of the loader against a real containerised Postgres or MinIO; the thin top is an end-to-end run on a small sample dataset. Most of your assertions live at the bottom because they are fast and pinpoint failures, while the expensive full-pipeline runs stay few. The lesson's "why the top is thin" is the cost-and-speed argument.

Q: Why should the top of the data test pyramid be "thin"?
A: End-to-end tests are slow, brittle, and hard to debug — a red E2E test tells you something broke but not where, and it ties up real infra. You keep a few as smoke tests to prove the whole flow wires together, then push detailed coverage down to fast, targeted layers. An inverted pyramid (mostly E2E) gives slow suites that everyone learns to ignore.

Q: What belongs in the unit layer versus the integration layer of a loader's tests?
A: Unit tests cover pure logic with no I/O: parsing a holdings row, computing a NAV total, validating an ISIN check digit — these need no container and run in milliseconds. Integration tests cover the I/O seam: does the loader actually `INSERT`/upsert into real Postgres, does the `NUMERIC(18,4)` column reject overflow, does the MinIO read return the expected bytes. The split is "logic below, real connections above".

Q: What is a "golden dataset" in this context?
A: A small, fixed, known-good input whose correct output you have computed and committed — for a fund pipeline, a tiny holdings file with a NAV total you worked out by hand. The test loads the golden input, runs the pipeline, and asserts the end state matches the expected golden output. It turns "looks about right" into a precise, repeatable pass/fail.

Q: What is snapshot (or "golden master") testing and what is its main risk?
A: Snapshot testing captures the current output as a reference file and fails the test if a later run differs from it. Its strength is catching unintended changes cheaply; its main risk is "snapshot rot" — when output legitimately changes, people blindly re-bless the snapshot without reading the diff, so the test stops protecting anything. Review every snapshot update as carefully as code.

Q: Why use a small golden file rather than a copy of a real production extract?
A: A small file keeps the test fast, lets you reason about and hand-verify the expected totals, and avoids dragging client PII or restricted data into the repo. A production-sized extract makes the test slow, opaque, and a data-governance liability. You want the smallest input that still exercises the edge cases you care about.

Q: How would you assert "row counts, checksums and the NAV reconciliation" after loading the golden dataset?
A: Query `SELECT count(*)` and compare to the known row count; compute a checksum/hash over the loaded rows (e.g. an ordered `md5` of concatenated key columns) and compare to the committed value; and re-sum the position values to confirm the NAV reconciles to the golden total. Each assertion targets a different failure: missing rows, corrupted values, and arithmetic drift respectively.

Q: Why include a checksum assertion when you already check row counts and the NAV total?
A: Row counts catch missing or duplicated rows but not corrupted field values, and a NAV total can coincidentally match even if individual positions are wrong (offsetting errors). A row-level checksum catches value-level corruption the aggregate hides. Layering the three assertions makes it very hard for a silent data error to slip through.

Q: What is idempotency and why is it the property this lesson tests?
A: An operation is idempotent if running it once and running it many times leaves the same end state — re-running the loader must not duplicate rows or change totals. Pipelines get retried (cron overlaps, failed-and-rerun jobs, replayed files), so non-idempotent loaders silently double-count, which in a NAV context is a reconciliation break. The lesson makes idempotency a testable property by running the loader twice and asserting identical state.

Q: Describe the "double-run idempotency" test step by step.
A: Start the containers, run the loader against the golden input, and capture the end state (row counts and a checksum). Run the exact same loader a second time against the same containers, then assert the new end state is byte-identical to the first. If it differs — extra rows, a changed total — the loader is not idempotent and the test fails.

Q: What database mechanism most commonly makes a loader idempotent, and what error does its absence cause in the double-run test?
A: An upsert — `INSERT ... ON CONFLICT (key) DO UPDATE` (or `DO NOTHING`) keyed on a natural/business key — so the second run updates instead of inserting duplicates. Without it, the second run appends a full duplicate set, so row counts double and the NAV total doubles. The lesson explicitly has you break the upsert to watch the idempotency assertion fail.

Q: The double-run test passes but in production you still get duplicates after a rerun — what subtle gap might the test have missed?
A: The conflict target may not match the true business key — e.g. you upsert on a surrogate id that changes per run, or you forgot that a real file can contain the same holding under two as-of dates. Production data has key collisions and edge cases the golden file lacks. The fix is to model the real uniqueness constraint and add golden cases that exercise it.

Q: How do you deliberately break the upsert to demonstrate the idempotency assertion failing?
A: Replace `INSERT ... ON CONFLICT (key) DO UPDATE ...` with a plain `INSERT`, or remove the unique constraint backing the conflict target, then run the suite. The second loader run now inserts a duplicate set, so the row-count and checksum assertions diverge from the first run and the test goes red. That red is the proof the idempotency check actually works.

Q: What is a contract test, at the awareness level this lesson sets?
A: A contract test verifies that a consumer and provider agree on the shape and semantics of the data/messages they exchange, independently, without a full integration. It catches "you changed the field and didn't tell me" breakages before deployment. The lesson previews this idea via Pact and maps it to the data contracts that arrive in Phase 8.

Q: What is "consumer-driven" about Pact's contract testing?
A: The consumer writes its expectations of the provider's responses as a pact; that pact is then replayed against the real provider to verify the provider still satisfies it. So the consumer's actual needs drive what the provider must guarantee, rather than the provider publishing a spec nobody checks. It is awareness-level here — full data-contract work is Phase 8.

Q: How does Pact's consumer-driven idea map onto data pipelines rather than HTTP APIs?
A: Instead of request/response, the "contract" is the schema and semantics of a dataset or event a downstream pipeline consumes — column names, types, nullability, enumerations, units. The producer's tests assert it still emits what consumers depend on, so a renamed `nav_per_share` column fails fast instead of silently breaking a report. This is the bridge to formal data contracts in Phase 8.

Q: Why are real containers preferable to an in-memory or SQLite substitute for Postgres integration tests?
A: SQLite and in-memory fakes have different SQL dialects, type systems, and constraint behaviour — they accept things real Postgres rejects and vice versa, so "passed on SQLite" tells you nothing about `NUMERIC` overflow, `ON CONFLICT`, or `jsonb`. Testcontainers runs the actual Postgres engine, giving you the real error codes and the real upsert semantics. Fidelity at the seam is the entire reason to spend the container startup cost.

Q: What is the prerequisite to run `uv run pytest -m integration`, per the "Done when" criteria?
A: Docker must be installed and the daemon running, because Testcontainers talks to the Docker socket to start containers; no shared database server is needed. The criterion is that the suite passes "on a machine with only Docker installed", proving the tests are self-contained. On native Ubuntu that means the Docker Engine, not Docker Desktop.

Q: What does the pytest `-m integration` flag do?
A: `-m` selects tests by marker expression, so `-m integration` runs only tests decorated `@pytest.mark.integration` (and `-m "not integration"` excludes them). It lets you separate slow container tests from the fast default run. You register the marker in config to avoid the "unknown marker" warning.

Q: How and why do you register a custom marker like `integration` in pytest?
A: Add it under `[tool.pytest.ini_options]` in `pyproject.toml` (or `pytest.ini`) as `markers = ["integration: spins up real containers"]`. Registering it documents the marker and, with `--strict-markers`, turns a typo'd or unregistered marker into an error instead of a silent no-op. Unregistered markers otherwise only emit a `PytestUnknownMarkWarning`.

Q: How do you exclude the integration suite from the default test run?
A: Set the default marker expression in config, e.g. `addopts = "-m 'not integration'"` in `[tool.pytest.ini_options]`, so a bare `pytest` skips them; developers opt in with `-m integration`. This keeps the fast feedback loop fast while heavy container tests run on demand or in CI. The criterion explicitly asks for the suite to be excluded from the default run.

Q: Why keep integration tests out of the fast default loop but still in CI?
A: Container tests cost seconds-to-minutes each, which kills the rapid red-green-refactor loop if run on every save, but they are exactly the tests that catch seam failures before release. So you run the fast pyramid base locally on every change and run the integration layer in CI (and on demand). The CI workflow from 9.3.1 is where you wire the `-m integration` job in.

Q: What is a pytest fixture and why is it the right tool for container lifecycle?
A: A fixture is a function decorated `@pytest.fixture` that supplies setup (and, via `yield`, teardown) to tests that request it by name as an argument. It is ideal for containers because the `yield` cleanly separates "start container / build engine" from "stop container", and the framework guarantees teardown runs. Tests just declare the fixture parameter and receive a ready connection.

Q: Write the shape of a fixture that starts Postgres once and tears it down, and explain the `yield`.
A: `@pytest.fixture(scope="session")` then `def pg():` with `with PostgresContainer("postgres:16") as c: yield c.get_connection_url()`. Code before `yield` is setup, the yielded value is what tests receive, and code after `yield` (here the `with`-exit) is teardown that runs even if a test fails. Session scope starts the container once for the whole run.

Q: What is fixture "scope" and which scope suits an expensive container?
A: Scope controls how often a fixture is created and torn down: `function` (per test), `class`, `module`, `package`, or `session` (once per run). For an expensive container you usually pick `session` or `module` so you pay the startup cost once and share it. The trade-off is isolation, which you restore by resetting data between tests.

Q: If you share one Postgres container across tests for speed, how do you keep tests isolated?
A: Reset state between tests — truncate the loaded tables, wrap each test in a transaction that rolls back, or recreate the schema in a function-scoped fixture layered on the session-scoped container. Sharing the container amortises startup while per-test cleanup prevents one test's rows from leaking into the next. Without isolation you get order-dependent, flaky passes.

Q: What does `@pytest.mark.parametrize` give you for golden-dataset testing?
A: It runs the same test body once per supplied input/expected pair, so you can drive many golden cases (different holdings files, different expected NAVs) through one assertion with clear per-case reporting. Each parameter set is reported as its own test, so you see exactly which golden case failed. It keeps coverage broad without copy-pasting test functions.

Q: How do you spin up MinIO with Testcontainers and why MinIO rather than mocking S3?
A: Use `from testcontainers.minio import MinioContainer`, start it, and point your S3 client (boto3) at its endpoint with the container's access/secret keys. MinIO speaks the real S3 API, so you test actual object PUT/GET, multipart, and error responses instead of a mock that can drift from S3's behaviour. It is the object-store analogue of using real Postgres over SQLite.

Q: Your MinIO-backed test fails with a bucket-not-found error on first run — likely cause?
A: A fresh MinIO container has no buckets; you must create the bucket in setup before the loader writes to it. Mocks often auto-create buckets, hiding this, so the real container exposes the missing provisioning step. Add a `make_bucket`/`create_bucket` call in the fixture after the container is ready.

Q: Why might a Testcontainers suite that passes locally fail in CI with "cannot connect to Docker daemon"?
A: The CI runner has no Docker available to the job, or the job lacks access to the Docker socket / a Docker-in-Docker service. Testcontainers needs a reachable Docker daemon to start anything. Fix by enabling a Docker service in the CI job (or DinD) and ensuring `DOCKER_HOST`/the socket is mounted for the test step.

Q: What environment variable disables the Ryuk reaper, and when might you set it?
A: `TESTCONTAINERS_RYUK_DISABLED=true` turns off the resource-reaper sidecar. You might set it in restricted CI environments that forbid the privileged-ish reaper container or where the platform already guarantees cleanup. The cost is that crashed test runs may leak containers, so you must clean up yourself.

Q: How do you confirm a container is actually being cleaned up after the test?
A: After the run, `docker ps -a` should show no leftover containers carrying the Testcontainers session label, and `docker volume ls`/`docker network ls` should be clean. If using a context manager or a `yield` fixture, teardown runs on exit; orphans usually mean the process was killed before teardown and Ryuk was disabled. Watching for leaks keeps your laptop and CI runners from filling up.

Q: A golden-dataset NAV assertion fails by a rounding-sized amount — what is the first thing to check?
A: Suspect a type/precision mismatch at the seam: the loader stored prices as `float` (binary, lossy) instead of `NUMERIC`/`Decimal`, so summing introduces rounding error. Check the column type and whether the code does arithmetic in `Decimal`. Money and NAV must use exact decimal types end to end; a `float8` column is the classic regulated-data bug.

Q: Why is a real Postgres `NUMERIC` overflow a good example of something only integration tests catch?
A: `NUMERIC(p,s)` enforces precision/scale at insert time and raises a real error if a value exceeds it; a Python mock or in-memory list happily accepts any number. So a unit test passes while the production load throws `numeric field overflow`. Only a test against real Postgres surfaces the overflow before release.

Q: What SQLSTATE does Postgres raise for a numeric value out of range, and why care in a fund loader?
A: It is SQLSTATE `22003` (`numeric_value_out_of_range`). In a fund loader an oversized position or price hitting a too-tight `NUMERIC(p,s)` raises `22003`, failing the load rather than silently truncating — which is the safe behaviour you want to test for. Asserting the loader handles or surfaces `22003` proves it won't quietly corrupt a NAV.

Q: What SQLSTATE indicates a unique-constraint violation, and how does it relate to idempotency testing?
A: SQLSTATE `23505` (`unique_violation`) fires when a duplicate hits a unique/primary-key constraint. A correct upsert with `ON CONFLICT` converts that would-be `23505` into an update, which is what makes the second loader run idempotent. If you removed the upsert, the second run would raise `23505` (or duplicate rows if there is no constraint), exactly the failure the double-run test catches.

Q: What is the difference between `ON CONFLICT DO NOTHING` and `ON CONFLICT DO UPDATE` for an idempotent loader?
A: `DO NOTHING` skips conflicting rows, keeping the first-seen values — fine when re-loads should never change existing data. `DO UPDATE SET ...` overwrites with the new row's values, which you want when a corrected file should refresh prices or quantities. Both make a reload non-duplicating; you choose based on whether reloads are corrections or pure replays.

Q: Why must the `ON CONFLICT` target match a real unique or exclusion constraint?
A: Postgres requires the conflict target to correspond to an existing unique index/constraint, otherwise it raises an error and the upsert won't compile to working behaviour. More importantly, the constraint is what defines "the same row" — if it doesn't match the true business key, the loader either rejects valid rows or fails to deduplicate. The constraint and the loader's idea of identity must be the same thing.

Q: How would you structure a `tests/integration` package per the "Do" steps?
A: Create `tests/integration/` with an `__init__.py`, a `conftest.py` holding the Postgres and MinIO fixtures, and test modules marked `@pytest.mark.integration`. Put golden input files under a `fixtures/`/`data/` subfolder so tests load known inputs. Keeping integration tests in their own package makes the `-m integration` split and CI wiring clean.

Q: What is `conftest.py` and why put the container fixtures there?
A: `conftest.py` is pytest's automatic fixture/plugin file: fixtures defined in it are available to every test in that directory tree without imports. Putting the Postgres/MinIO fixtures there lets all integration tests share them and centralises lifecycle. pytest discovers it by convention, so no explicit import is needed.

Q: How do you wire the integration suite into the CI workflow from 9.3.1?
A: Add a CI job that has Docker available (a Docker service or DinD), installs deps with uv, and runs `uv run pytest -m integration`. Gate it appropriately — it can run on PRs and main — while the fast default job runs `pytest` (which already excludes integration). The criterion is that the marker-selected suite executes in CI, not just locally.

Q: Why run integration tests with `uv run pytest` rather than activating a venv and calling pytest?
A: `uv run` executes inside the project's locked environment, guaranteeing the exact dependency versions from the lockfile and resolving the right interpreter without manual activation. Per the project's standing rule you never invoke bare `python`/`pip`/`pytest`. It also makes the CI command identical to the local command, removing "works on my machine" drift.

Q: A teammate runs the suite and Testcontainers cannot find the Docker socket on their Ubuntu box — what do you check?
A: Confirm the Docker daemon is running (`systemctl status docker`) and that the user is in the `docker` group (or the rootless socket path is exported). Testcontainers reads `DOCKER_HOST`/the default `unix:///var/run/docker.sock`; if rootless Docker is in use, `DOCKER_HOST` must point at the rootless socket. Permission or socket-path mismatches are the usual cause of "cannot connect" on native Linux.

Q: Why is "throwaway infra" framed as "the only honest test of the seams"?
A: Because a mock encodes your assumptions about the dependency, so it can only confirm what you already believe; a real container can contradict you with a real error code or type rejection. Disposable real infra gives you production-fidelity behaviour without a shared, drifting test server. Honesty here means the test can actually fail for the reasons production would.

Q: Give a concrete example where a mocked unit test "passes forever" while the real loader fails.
A: A mock DB accepts a 20-digit position into a column you declared `NUMERIC(18,4)`; the unit test is green. Against real Postgres the insert raises `22003 numeric_value_out_of_range` and the load fails. The mock never models the precision constraint, so only the container test exposes the bug.

Q: How do you assert "byte-identical" end state between the first and second loader runs?
A: Compute a stable digest of the relevant tables after run one — for example `SELECT md5(string_agg(col1||'|'||col2, ',' ORDER BY pk))` — and capture it, then compute the same digest after run two and assert equality. Ordering inside the aggregate is essential so row order doesn't make identical data look different. Equal digests prove the second run changed nothing.

Q: Why must you impose an explicit `ORDER BY` when hashing rows for a snapshot comparison?
A: Without `ORDER BY`, Postgres returns rows in an unspecified, run-to-run-variable order, so the same data can hash differently and the snapshot assertion flaps. A deterministic ordering on the primary/business key makes the digest a stable function of the data alone. This is a classic source of "flaky but actually-correct" snapshot tests.

Q: Where does the integration layer sit in your test pyramid and what stays below it?
A: It sits in the middle: above unit tests of pure logic (parsing, NAV math, ISIN validation) and below the thin end-to-end run on a sample. Below it stay all the fast, no-I/O assertions; the integration layer adds the real-Postgres/real-MinIO seam coverage. The "Done when" asks you to articulate exactly this placement.

Q: Why not just write more end-to-end tests instead of a dedicated integration layer?
A: E2E tests are slow and give poor diagnostics — when one fails you don't know which seam broke, and they tie up the whole stack. A focused integration test targets a single boundary (loader-to-Postgres) so a failure points straight at it and runs in seconds, not minutes. You keep E2E thin and let the integration layer carry seam coverage.

Q: What is the risk of testing exclusively against real containers and dropping unit tests?
A: You invert the pyramid: the suite becomes slow, every change pays full container startup, and tiny logic bugs are diagnosed through heavyweight tests. Pure-logic bugs are cheapest to catch in milliseconds at the base. Containers are for the seam, not for arithmetic you can test in isolation.

Q: For a Luxembourg fund NAV pipeline, what would a golden dataset and its assertion look like?
A: A tiny holdings file for one sub-fund with, say, five positions and a hand-computed total NAV per share; the test loads it, runs the valuation, and asserts the stored NAV per share equals the golden figure to the correct decimal scale. It also checks the row count and a per-position checksum. This proves the loader reproduces a known-correct NAV before it ever touches real client data.

Q: Why should ISIN validation live in the unit layer rather than an integration test?
A: ISIN check-digit validation (Luhn over the 12-char code) is pure string logic with no I/O, so it belongs in fast unit tests parametrized over valid and invalid codes. Pushing it into a container test would be slow and would conflate the validation logic with database behaviour. Reserve the container for what genuinely needs a real database.

Q: How might an EMT/EPT file feed a golden-dataset integration test?
A: You commit a small, known-good EMT (European MiFID Template) sample with expected parsed fields, load it through the ingestion pipeline into the real Postgres, and assert the resulting rows and key totals match the golden expectation. Because the EMT layout and field semantics are fixed, the golden output is precise. This also doubles as a regression guard if someone changes the parser.

Q: How does contract testing relate to receiving an EMT/EPT template from a fund provider?
A: The EMT/EPT layout is effectively a contract between the provider (producer) and your pipeline (consumer): agreed columns, types, and codings. A contract-style test asserts your parser still matches the expected template version and fails loudly if a provider quietly changes a field. At awareness level this is the Pact idea applied to a regulatory file format, formalised as data contracts in Phase 8.

Q: For a transfer-agency dealing feed, why is the idempotency test especially important?
A: TA feeds get reprocessed — files are resent, jobs rerun after a mid-dealing-window failure — so a non-idempotent loader double-books subscriptions/redemptions, corrupting positions and the next NAV. The double-run test proves a resend leaves identical state. In a regulated shop a duplicated dealing record is a reconciliation break, not a cosmetic bug.

Q: A SWIFT-message ingestion test passes on mocks but the real loader rejects messages — what seam is likely at fault?
A: Probably encoding/format strictness or field-length constraints that the mock ignored — real Postgres `varchar(n)` or a `CHECK` on a BIC/field length rejects what the mock accepted. The container surfaces the actual constraint violation (e.g. a value-too-long `22001`). Test against the real schema so SWIFT field constraints are enforced where they will be in production.

Q: What SQLSTATE does Postgres raise when a string is too long for its column, and why does it matter for fixed-width financial codes?
A: SQLSTATE `22001` (`string_data_right_truncation`). Fixed-width codes — a 12-char ISIN, a 20-char LEI, an 11-char BIC — map to tightly sized columns, so a malformed over-length value raises `22001` instead of silently truncating. Asserting the loader hits `22001` on bad input proves it won't store a corrupted identifier.

Q: Why is a 20-character LEI a good candidate for a length/check constraint that integration tests should exercise?
A: An LEI is exactly 20 alphanumeric characters with a defined structure, so a `varchar(20)` plus a `CHECK` is natural; an integration test can confirm that a 19- or 21-char value is rejected by real Postgres. A mock would accept the malformed LEI and hide the bug. This is the kind of constraint that only a real-database test enforces faithfully.

Q: In a DORA / regulated-resilience context, what evidence does an integration suite provide?
A: It gives demonstrable, reproducible proof that the pipeline was tested against production-like infrastructure — including idempotency and failure handling — which supports resilience and change-control claims. Auditable green runs in CI against real containers are stronger evidence than "we mocked it". DORA's focus on operational resilience makes tested recovery/replay behaviour directly relevant.

Q: Why is demonstrating the idempotency assertion failing (by breaking the upsert) a required deliverable, not just a passing test?
A: A test that has never failed might be asserting nothing (a false-green). Forcing it red by breaking the upsert proves the assertion actually detects the duplication it claims to. This "see it fail for the right reason" step is basic test hygiene and is exactly what the "Done when" demands.

Q: What is a common cause of a Testcontainers test that passes alone but fails when run with the full suite?
A: Shared mutable state — a session-scoped container whose data from an earlier test leaks into a later one because cleanup between tests is missing. Order-dependent passes mean isolation is broken. Add per-test truncation/transaction rollback or narrow the fixture scope to restore independence.

Q: Why prefer wrapping each test in a transaction that rolls back over truncating tables between tests?
A: A rolled-back transaction is fast and automatically discards all writes, leaving the schema pristine without enumerating tables; truncation requires you to list and clear each affected table and can miss one. The transaction approach also can't accidentally leave half-cleaned state. The caveat is it won't isolate code paths that commit internally, where truncation or schema recreation is needed.

Q: How do you handle a loader that opens its own connection, when you want each test rolled back?
A: Inject the connection/engine (from the container fixture) into the loader rather than letting it create its own, so the test controls the transaction boundary and can roll it back. Dependency-injecting the connection is also what lets you point the loader at the container's mapped port. A loader that hard-codes its connection is hard to test and should be refactored.

Q: Why is it bad practice to point integration tests at a long-lived shared test database instead of containers?
A: A shared DB accumulates drift, suffers cross-team interference, and makes failures depend on whatever state someone else left behind, so tests become flaky and untrustworthy. Containers give each run a pristine, isolated, disposable database. The "machine with only Docker installed" criterion exists precisely to forbid the shared-server dependency.

Q: What does it mean that "this layer is tool-agnostic; the discipline, not the vendor, is the deliverable"?
A: The value is in having fast unit tests, real-infra integration tests, golden datasets, and idempotency proofs — not in which library provides the containers. Testcontainers and pytest are the chosen tools, but the same pyramid and assertions transfer to any stack. Corp environments use "the same" approach; the practice is what you carry between jobs.

Q: How would you assert NAV reconciliation tolerance correctly when prices have defined decimal scale?
A: Compare using exact `Decimal`/`NUMERIC` arithmetic and assert equality at the prescribed scale, rather than a floating-point "close enough" with an arbitrary epsilon. NAV figures have a contractually defined number of decimals, so the right assertion is exact at that scale after correct rounding. Sloppy float comparison either hides real errors or invents spurious ones.

Q: Why can `assert a == b` on floats give a false failure in a NAV test, and what is the fix?
A: Binary floating point can't represent many decimal fractions exactly, so summing prices accumulates tiny errors and `==` fails even when the value is "right". The fix is to compute and store money as `Decimal`/`NUMERIC` and compare exactly at the defined scale, or use a rounding step that matches the business rule. Never use raw `float` for currency or NAV.

Q: What is "component" testing in the data pyramid, between unit and end-to-end?
A: Component tests exercise one deployable unit — say the loader together with its real database — in isolation from the rest of the system. They are broader than a unit test but narrower than full E2E, which is exactly the integration layer this lesson builds. The term emphasises testing a single component against real collaborators rather than the whole pipeline.

Q: Why is "roll the loader twice" a stronger idempotency check than inspecting the SQL for `ON CONFLICT`?
A: Reading the SQL only confirms the intent; running it twice against real Postgres confirms the actual end state, catching mistakes like a wrong conflict target, a missing constraint, or a side effect outside the upsert. Behavioural assertions beat code inspection because they exercise the real engine. The double run tests the property, not the implementation detail.

Q: What is the danger of golden snapshots that include volatile fields like load timestamps?
A: A snapshot containing `now()` or an auto-incrementing id changes every run, so the comparison always "fails", and people respond by re-blessing it blindly until it protects nothing. Exclude or normalise volatile columns before snapshotting. Snapshot only the deterministic, business-meaningful end state.

Q: Why might `uv run pytest -m integration` collect zero tests, and how do you diagnose it?
A: The marker isn't applied (tests aren't decorated `@pytest.mark.integration`), it's misspelled, or `--strict-markers` plus an unregistered marker errored out collection. Run `uv run pytest -m integration --collect-only` to see what matches, and check the marker registration in `pyproject.toml`. Zero collected usually means a marker/config mismatch, not a real absence of tests.

Q: What does `pytest --collect-only` do and how does it help here?
A: It lists the tests that would run without executing them, so you can confirm your marker expression selects the intended integration tests. It's the fastest way to debug "why did my `-m integration` select nothing/everything". You see collection-time errors too, like an unregistered-marker failure under `--strict-markers`.

Q: Why does the lesson keep the golden holdings file small enough to verify totals by hand?
A: Because a human-verifiable expected value is what makes the test trustworthy — you can prove the golden NAV is correct, not just "what the code produced last time". A large file forces you to trust the code to generate its own oracle, which is circular. Small-and-hand-checked breaks that circularity.

Q: What is the relationship between this integration-testing lesson and the data contracts coming in Phase 8?
A: This lesson previews contract testing (via Pact's consumer-driven idea) at awareness level; Phase 8 formalises it into enforced data contracts between producers and consumers. The integration tests here protect your own seams, while data contracts will protect the boundaries between teams/systems. The conceptual thread is "verify agreements at the boundary".

Q: A container starts but your test cannot reach Postgres because you connected to port 5432 — what's the conceptual error?
A: You used the in-container port instead of the mapped host port, so nothing is listening on host `5432`. You must connect to `get_exposed_port(5432)` or use `get_connection_url()`, which already contains the random host port. Confusing the container-internal port with the host-mapped port is the most common Testcontainers beginner bug.

Q: Why can running the integration suite in parallel (e.g. `pytest -n auto`) be both desirable and tricky with containers?
A: Parallelism cuts wall-clock time, and random host-port mapping makes concurrent containers possible without port clashes; but shared session-scoped fixtures or a shared container break under parallel workers writing the same tables. You either give each worker its own container or use per-test isolation. The random-port design is what makes safe parallelism feasible at all.

Q: How do you prove the integration tests actually exercised the real database and weren't silently skipped?
A: Check the pytest summary for the count of passed integration tests (not "skipped"), and optionally log the container image/port or assert on a real Postgres-specific behaviour (like a `22003` error) that a mock couldn't produce. A green run with zero collected is a false sense of safety. Asserting a real SQLSTATE is strong evidence the real engine ran.

Q: What is "snapshot rot" and how do you defend against it in a regulated codebase?
A: Snapshot rot is the gradual erosion of a snapshot's value as people accept diffs without scrutiny, until it asserts nothing meaningful. Defend by reviewing every snapshot change in code review, keeping snapshots small and deterministic, and pairing them with explicit invariant assertions (row counts, NAV totals) that can't be blindly re-blessed. In a regulated shop, an unreviewed re-bless is an audit gap.

Q: Why is asserting an "end-state invariant" (like a reconciled NAV) more robust than asserting the exact output bytes?
A: An invariant captures the business truth — the NAV reconciles, no duplicates exist, totals balance — so it survives benign output changes while still catching real corruption. Exact-byte snapshots are brittle and flag harmless reorderings or formatting tweaks. The strongest suites combine a few invariants with a deterministic snapshot.

Q: When a transfer-agency file is reprocessed after a partial failure, what should an idempotent loader guarantee, and how do you test it?
A: It must guarantee the final positions are the same as if the file had loaded cleanly once — no double-counted subscriptions, no missing rows. You test it by simulating the partial-then-full reprocess (or simply running the loader twice on the same file) against the container and asserting byte-identical end state. That double-run is the lesson's idempotency proof applied to TA reprocessing.

Q: Why does the lesson tie integration tests to "permissions" as one of the seams?
A: Because production loaders run as a specific database role with limited grants, and a permission gap (no `INSERT` on a schema, missing `USAGE`) only fails against a real database configured like production. A mock ignores grants entirely. Testing with realistic roles in the container catches "permission denied" before deployment.

Q: How could you make an integration test catch a missing-grant bug?
A: Create the role and grants in the container's setup to mirror production, then run the loader as that least-privileged role; a missing grant surfaces as a real `42501 insufficient_privilege` error. Running tests as a superuser hides exactly these bugs. Mirroring production privileges is what makes the permission seam testable.

Q: Why is "run via uv against local Docker" the intended workflow, and what does it imply for reproducibility?
A: It means tests use the project's locked dependencies (`uv run`) and a local Docker daemon to start real containers, so any machine with uv and Docker reproduces the exact environment. There's no shared server, no manual venv activation, and no dependency drift. That reproducibility is what lets the suite be trusted as audit evidence.

Q: What single sentence captures why mocked unit tests are insufficient for a regulated loader?
A: They validate your assumptions about the database rather than the database itself, so they pass while the real loader fails on real constraints, type limits, permissions, and error codes. In a regulated context that gap is the difference between "we tested it" and "it actually works". Real, disposable infrastructure closes the gap at the seam where pipelines actually break.

Q: What is the `driver` argument of `PostgresContainer` and when would you set `driver=None`?
A: `PostgresContainer(image, driver="psycopg2")` controls the SQLAlchemy dialect suffix in the URL produced by `get_connection_url()`, so the default yields `postgresql+psycopg2://...`. You set `driver=None` to get a bare `postgresql://...` URL with no driver suffix, which is what you want when connecting with `psycopg` (v3) since `postgresql+psycopg2` would force the v2 driver. Passing the wrong driver string is a common cause of a `ModuleNotFoundError` for `psycopg2` when only `psycopg` is installed.

Q: How do you set a custom username, password, and database name on a `PostgresContainer`?
A: Pass them as constructor keyword arguments, e.g. `PostgresContainer("postgres:16", username="nav", password="nav", dbname="funddb")`; the container sets the corresponding `POSTGRES_USER`/`POSTGRES_PASSWORD`/`POSTGRES_DB` env vars internally and bakes the values into the URL from `get_connection_url()`. This lets you mirror a production role name so privilege tests are realistic. Leaving them unset gives the module's defaults of `test`/`test`/`test`.

Q: A loader test commits, then a follow-up query in the same transaction raises `25P02` — what happened?
A: SQLSTATE `25P02` (`in_failed_sql_transaction`) means an earlier statement in the current transaction failed and Postgres now rejects every command until you `ROLLBACK`. Typically your loader swallowed the first error (say a `23505`) without rolling back, then kept issuing statements that all bounce with `25P02`. The fix is to roll back on the first error or use a savepoint, and the integration test that surfaces this proves the loader's error handling is wrong.

Q: How would you write a test that asserts the loader rejects an over-length ISIN with a specific SQLSTATE?
A: Wrap the insert in `with pytest.raises(psycopg.errors.StringDataRightTruncation):` (or catch the generic `DBAPIError` and assert `exc.orig.sqlstate == "22001"`), then feed a 13-character value into the `char(12)` ISIN column. Asserting on the SQLSTATE rather than the message keeps the test stable across locales and Postgres versions. This proves the real column constraint fires where a mock would have silently accepted the bad code.

Q: Why is `psycopg.errors` keyed by SQLSTATE useful for writing precise integration assertions?
A: `psycopg` exposes named exception classes mapped to each SQLSTATE (e.g. `UniqueViolation` for `23505`, `NumericValueOutOfRange` for `22003`), and every raised error carries `.sqlstate` so you can assert on the exact code. This lets a test say "this input must fail for exactly this reason" rather than catching a broad `Exception` that could mask an unrelated bug. Matching on the class or the five-character code is far more robust than matching on the English message text.

Q: A foreign-key insert in your loader test fails with `23503` — what does that tell you about the test data or load order?
A: SQLSTATE `23503` (`foreign_key_violation`) means you inserted a child row referencing a parent key that does not exist yet — for a fund loader, a holding pointing at a `sub_fund_id` you never loaded. Either the golden dataset is missing the parent row or the loader inserts children before parents. The test correctly catches an ordering or referential-integrity bug that a constraint-free mock would hide.

Q: How do you simulate and test deadlock handling (`40P01`) in a parallel-loader integration test?
A: Run two transactions that lock the same rows in opposite order against the real container, so Postgres detects the cycle and aborts one with SQLSTATE `40P01` (`deadlock_detected`); assert the loader catches it and retries rather than crashing. Deadlock is a transient error, so the correct behaviour is bounded retry with backoff, not failure. Only a real engine produces a genuine deadlock — no mock will.

Q: What does `MinioContainer.get_config()` return and how do you use it to point boto3 at the container?
A: `get_config()` returns a dict with `endpoint`, `access_key`, and `secret_key` for the running MinIO; you feed those into `boto3.client("s3", endpoint_url=f"http://{config['endpoint']}", aws_access_key_id=config['access_key'], aws_secret_access_key=config['secret_key'])`. This wires your real S3 client at the disposable object store instead of AWS. The `endpoint` already contains the random mapped host port, so you never hard-code one.

Q: What is `MinioContainer.get_client()` and when would you prefer it over building a boto3 client?
A: `get_client()` returns a ready `minio.Minio` client pointed at the container, convenient for setup tasks like `make_bucket` before your loader runs. You would still test the loader through whatever client it uses in production (often boto3), so `get_client()` is best for fixture provisioning rather than for exercising the code under test. Using the production client for the assertion path keeps the test honest about real S3 behaviour.

Q: How do you get the Kafka bootstrap address from a Testcontainers `KafkaContainer`?
A: Call `kafka.get_bootstrap_server()`, which returns the host-and-mapped-port string (e.g. `localhost:49162`) to put in your producer/consumer `bootstrap.servers` config. As with Postgres, the port is dynamically mapped, so you must read it at runtime rather than assume `9092`. A known historical gotcha is the broker advertising `localhost`, which only works when the test client is on the same host as the container.

Q: How would you integration-test that an event-driven NAV pipeline consumes a Kafka message and writes the right row to Postgres?
A: Spin up both a `KafkaContainer` and a `PostgresContainer` in fixtures, produce a known event to the topic, run the consumer/loader, then poll Postgres until the expected row appears and assert its values against the golden expectation. You add a bounded wait because the consume-and-write is asynchronous, so a naive immediate assert races the consumer. This proves the whole seam — broker deserialisation plus the DB write — not just one half.

Q: Why might you add a polling/retry assertion (not a fixed sleep) when testing an async consumer, and how?
A: A fixed `sleep` is either too short (flaky) or too long (slow), whereas polling asserts as soon as the condition is true and fails only after a timeout. Use a helper that retries `SELECT count(*)` every 100 ms up to a few seconds, or a library like `tenacity`, raising if the expected state never appears. This makes async integration tests both fast and reliable.

Q: What generic Testcontainers method exposes a container port, and how does it differ from binding a fixed port?
A: `DockerContainer("img").with_exposed_ports(5432)` publishes the internal port to a random host port (discovered later via `get_exposed_port(5432)`), which is the safe default; `with_bind_ports(5432, 5432)` forces a fixed host port. You prefer `with_exposed_ports` for parallel-safe tests and only use `with_bind_ports` when an external tool must reach a known port. Binding a fixed port reintroduces the collision risk Testcontainers exists to avoid.

Q: How do you pass environment variables into a generic `DockerContainer`?
A: Chain `.with_env("KEY", "value")` calls before `.start()`, e.g. `DockerContainer("postgres:16").with_env("POSTGRES_PASSWORD", "test").with_exposed_ports(5432)`. The dedicated modules like `PostgresContainer` set these for you, but for an arbitrary image you wire the service's expected env yourself. Forgetting a required env var (like `POSTGRES_PASSWORD`) makes the container exit immediately on start.

Q: How would you mount a golden SQL schema or seed file into a Postgres container at startup?
A: Use `.with_volume_mapping(host_path, "/docker-entrypoint-initdb.d/init.sql")` so the official Postgres image runs the script on first init, or copy and execute the DDL from your fixture after the container is ready. The init-dir approach guarantees the schema exists before any test connects. Be aware the entrypoint scripts only run when the data directory is empty, i.e. on a fresh container.

Q: A Testcontainers run dies with `Could not start container: image not found` — what are the two likely causes?
A: Either the pinned tag is wrong/typo'd (e.g. `postgres:166`) so it does not exist in the registry, or you are offline/behind a proxy and the image is not in the local cache to pull. Run `docker pull postgres:16` manually to distinguish a bad tag from a network problem. Pre-pulling in CI and pinning a real tag prevents both.

Q: How do you capture and assert on a container's logs in a generic Testcontainers test?
A: After start, call the container's log accessor (e.g. `container.get_logs()` returning the stdout/stderr tuple) or use `wait_for_logs(container, r"ready")` to block on a readiness pattern. Asserting a known startup line confirms the service actually initialised rather than merely that Docker reported "running". For Postgres specifically the module already waits on `database system is ready to accept connections`.

Q: Why can a session-scoped container fixture plus per-test transaction rollback still leak state, and what is the fix?
A: If a test (or the loader) issues its own `COMMIT`, the outer rollback can no longer undo it, so committed rows persist into later tests. The fix is to make the loader accept an injected connection and not commit, or to fall back to `TRUNCATE ... RESTART IDENTITY CASCADE` in a function-scoped fixture. Code paths that commit internally are exactly why pure transaction-rollback isolation sometimes fails.

Q: What does `TRUNCATE ... RESTART IDENTITY CASCADE` do and why is it useful between integration tests?
A: It empties the named tables, resets their identity/serial sequences to the start, and cascades to tables with foreign keys referencing them, all far faster than `DELETE`. Between tests this returns a shared container to a clean, predictable state including auto-increment ids, so snapshot/checksum assertions stay deterministic. Without `RESTART IDENTITY`, a surrogate `id` keeps climbing and breaks any snapshot that includes it.

Q: How would you structure fixtures so the container starts once (session scope) but the schema/data resets every test (function scope)?
A: Make a `session`-scoped fixture that yields the container/engine, and a `function`-scoped fixture that depends on it and either begins-then-rolls-back a transaction or truncates the tables around each test. pytest resolves the dependency so the cheap reset runs per test while the expensive container runs once. This is the standard pattern that buys both speed and isolation.

Q: Why is `scope="session"` plus `@pytest.fixture(autouse=True)` a risky combination for a container fixture?
A: `autouse=True` forces the fixture into every test in scope whether it needs containers or not, so even your fast unit tests would pay container startup and require Docker. You want integration containers requested explicitly (by naming the fixture or via the `integration` marker), keeping the unit layer Docker-free. Reserve `autouse` for cheap, universally-needed setup, not heavyweight infra.

Q: How does `@pytest.mark.usefixtures("pg_schema")` differ from taking the fixture as a function argument?
A: `usefixtures` applies the fixture for its side effects (e.g. creating the schema) without giving the test the return value, whereas an argument both triggers the fixture and binds its yielded value. You use `usefixtures` when the test just needs the setup to have happened — like a reset table — but does not need a handle to it. It is also applyable at class or module level via the marker.

Q: What is `--strict-markers` and why does the lesson recommend it for the `integration` marker?
A: `--strict-markers` (set in `addopts`) makes pytest error on any marker not registered under `[tool.pytest.ini_options] markers`, instead of merely warning. This catches a typo like `@pytest.mark.integraton` at collection time rather than silently running the test in the default (non-integration) set. In a regulated repo you want a misspelled marker to fail loudly, not quietly skip your seam tests.

Q: How do you run only the integration tests that failed last time, to tighten a debugging loop?
A: Use `uv run pytest -m integration --lf`, where `--lf` (`--last-failed`) reruns just the previously failed tests; pair with `--ff` to run failures first then the rest. This avoids paying full container time on already-green tests while you fix a flaky seam. The cache backing `--lf` lives in `.pytest_cache`, so do not pass `-p no:cacheprovider` if you rely on it.

Q: What does `-x` (or `--maxfail=1`) buy you when a containerised suite starts failing?
A: `-x` stops the run at the first failure (it is shorthand for `--maxfail=1`), so you do not wait for every slow container test to run before seeing the first red. This is valuable when each test costs seconds of startup and the first failure already tells you what to fix. Use `--maxfail=N` to allow a few failures before bailing.

Q: How can `pytest -n auto` (pytest-xdist) speed up the integration suite, and what must you guard against?
A: `-n auto` distributes tests across worker processes equal to the CPU count, cutting wall-clock time; the random host-port mapping lets each worker's containers coexist. You must guard against shared state: a single session-scoped container written by parallel workers corrupts results, so give each worker its own container or enforce per-test isolation. xdist also historically did not support `--lf`/`--ff` cleanly, so combine them carefully.

Q: Why might a Testcontainers suite be much slower in CI than locally even after pre-pulling images?
A: CI often runs Docker-in-Docker or a remote daemon with slower disk and no warm layer cache between jobs, and Ryuk plus each container's init add fixed overhead per test. Reusing a session-scoped container, caching the image layer, and minimising the number of distinct containers all help. The fixed startup cost is why you push most assertions to the Docker-free unit layer.

Q: What is `TESTCONTAINERS_RYUK_DISABLED` versus container reuse, and why not leave Ryuk off to save time?
A: `TESTCONTAINERS_RYUK_DISABLED=true` removes the reaper that guarantees cleanup, which can shave a little startup but risks leaked containers if a run is killed mid-test. Reuse (keeping a container alive across runs) is a separate optimisation that needs explicit opt-in and careful state handling. Disabling Ryuk to save time is a false economy on a laptop or shared runner that then fills with orphans.

Q: How would you confirm that a "passing" integration run actually started a container and is not silently skipping?
A: Check the pytest summary shows `passed` not `skipped` for the marked tests, and add a sentinel assertion that only a real engine can satisfy — e.g. force and catch a `22003` overflow, or read `SELECT version()` and assert it reports PostgreSQL. A run reporting all-green with the integration tests skipped (because Docker was unavailable and you guarded with `skipif`) is a silent false sense of safety.

Q: When is `@pytest.mark.skipif` on the Docker availability check appropriate, and what is its danger?
A: Skipping integration tests when Docker is absent (e.g. `skipif(shutil.which("docker") is None)`) lets the unit suite run on machines without Docker without erroring. The danger is that CI silently skips the seam tests if Docker is misconfigured, reporting green while testing nothing — so CI must assert the integration tests actually ran, not just that the job exited zero. On the native Ubuntu laptop with Docker installed, they should never skip.

Q: How would you assert that a redelivered SWIFT MT message does not create a duplicate dealing record, at the constraint level?
A: Model a unique constraint on the message's idempotency key (e.g. the SWIFT message reference / transaction reference), load the message twice via the real container, and assert the row count stays at one and the values are unchanged. A correct `ON CONFLICT` on that reference turns the second insert's would-be `23505` into a no-op or update. This proves replay-safety on the exact key SWIFT guarantees unique.

Q: For an EET (European ESG Template) golden file, what would an integration test assert beyond row counts?
A: It would load a small known-good EET sample and assert the parsed ESG fields land in the right typed columns with the expected codings (e.g. PAI indicator values, taxonomy alignment percentages) and that any numeric percentage respects its `NUMERIC` scale. Because the EET layout and field semantics are fixed by the template version, the golden output is precise. This doubles as a regression guard against a parser change when a new template version ships.

Q: How does a template version bump (EMT/EPT/EET) interact with your golden-dataset and contract-style tests?
A: A new template version can rename, add, or recode fields, so your golden file is pinned to a specific version and a contract-style test asserts the parser matches that version's layout, failing loudly on drift. When the provider moves versions you deliberately add a new golden fixture for the new version rather than silently re-blessing. This is the awareness-level Pact idea applied to versioned regulatory file formats, formalised as data contracts in Phase 8.

Q: In Pact terms, what is the artefact the consumer produces and where does the provider verify it?
A: The consumer's tests generate a pact file (a JSON contract of expected interactions) which is published to a Pact Broker; the provider then runs verification by replaying those interactions against itself, optionally publishing the results back. For data pipelines the analogue is the consumer recording the dataset shape it depends on, and the producer verifying it still emits that shape. This keeps the producer honest about a renamed column like `nav_per_share`.

Q: What does Pact's `can-i-deploy` answer, and how does that idea transfer to data pipelines?
A: `can-i-deploy` consults the broker's matrix of which consumer and provider versions have been verified against each other and gives a yes/no on whether a given version is safe to release. Transferred to data, it is the question "can I ship this schema change without breaking a downstream consumer", answered by whether the producer still satisfies every consumer's recorded contract. Phase 8 turns this from awareness into enforced data contracts.

Q: How would you test that the loader runs correctly as a least-privileged role, and what error proves the guard works?
A: In the fixture, create a non-superuser role with only the grants production gives it (`USAGE` on the schema, `INSERT`/`SELECT` on specific tables), connect the loader as that role, and run the happy path; then attempt an ungranted action and assert it raises SQLSTATE `42501` (`insufficient_privilege`). Running tests as superuser hides exactly these gaps. Mirroring production privileges in the container is what makes the permission seam testable.

Q: Why is running integration tests as the Postgres superuser a subtle anti-pattern?
A: A superuser bypasses every grant and many constraints, so the test passes regardless of the missing privileges the production role will actually hit, producing a false green that breaks on deploy with `42501`. The test environment must mirror production's least-privilege role to be evidence of anything. This is the privilege analogue of testing against a mock that accepts everything.

Q: How would you integration-test that the loader is safe under a connection drop mid-load?
A: Begin the load in a transaction, force a disconnect (e.g. kill the connection or restart the container's network), and assert that on reconnect the table is either empty or fully loaded — never half — because the aborted transaction rolled back. This proves the load is atomic and recoverable, which is a DORA-style resilience property. A loader that auto-commits per row would leave a partial state the test would catch.

Q: What is the difference between testing atomicity (all-or-nothing) and idempotency (repeatable), and why do you want both?
A: Atomicity means a single run either fully applies or fully rolls back, so a crash never leaves a half-load; idempotency means re-running produces the same end state, so a retry never double-applies. A loader can be atomic but not idempotent (a clean second run still duplicates) or idempotent but not atomic (a crash leaves partial rows). For a NAV/TA feed you need both: the double-run test covers idempotency and a crash-injection test covers atomicity.

Q: How would you assert a stable cross-table checksum that survives benign id reassignment?
A: Compute the digest over business-key columns and values with an explicit `ORDER BY` on the business key, deliberately excluding surrogate `id`s and timestamps, e.g. `md5(string_agg(isin||'|'||quantity::text, ',' ORDER BY isin))`. This makes the snapshot a function of the meaningful data only, so re-running with new auto-increment ids does not flip it. Including the surrogate id is the classic cause of a snapshot that "always fails".

Q: Why should NAV reconciliation in a test sum from the stored `NUMERIC` rows rather than from the in-memory Python values?
A: Summing the persisted `NUMERIC` column proves the database round-trip preserved precision and scale, catching a silent `float8` column or a truncating cast that the in-memory value would hide. The whole point of the integration test is the seam, so the assertion must read back from storage. Reconciling against the hand-computed golden total then confirms end-to-end exactness, not just in-process correctness.

Q: How would you parametrize a single idempotency test across Postgres `DO NOTHING` and `DO UPDATE` loaders?
A: Use `@pytest.mark.parametrize` over the two loader variants (or two conflict clauses) and assert the shared invariant — second run yields identical row count and checksum — for each, with per-case ids so a failure names the variant. This covers both replay semantics in one test body without duplication. It also documents that both clauses are non-duplicating while differing on whether reloads refresh values.

Q: How do you keep client PII out of a golden dataset while still exercising real EMT/holdings parsing?
A: Use synthetic but structurally valid data — fabricated ISINs that pass the check digit, made-up sub-fund names, and round invented prices — so the file exercises every parser branch without containing any real client or position data. This keeps the fixture committable to the repo and audit-safe. A redacted production extract is risky because residual identifiers and data-governance obligations come with it.

Q: A golden NAV test passes locally but fails in CI by a tiny amount — what environment difference would you suspect first?
A: A locale or rounding/precision difference: a different Postgres minor version, a `lc_numeric`/`DateStyle` setting, or a Python/Decimal rounding-context difference between machines. Pin the exact image tag and do money arithmetic in `Decimal` with an explicit rounding mode so the result is environment-independent. Reproducibility via `uv run` plus a pinned image is what removes "works on my machine" NAV drift.

Q: How would you prove your double-run idempotency assertion is not a false-green before trusting it?
A: Deliberately break the loader — swap the `ON CONFLICT` upsert for a plain `INSERT` or drop the backing unique constraint — and confirm the second-run assertion now goes red with doubled counts or a diverging checksum. Seeing it fail for the right reason proves the assertion actually detects duplication. The lesson makes this red demonstration a required deliverable, not an optional nicety.

Q: What CI evidence from the integration suite would you point an auditor to for a DORA change-control claim?
A: A retained CI run log showing `uv run pytest -m integration` passing against pinned real containers, including the idempotency and atomicity assertions, with the commit SHA and image tags recorded. That artefact is reproducible, tied to a specific code version, and demonstrates the seam was tested against production-like infrastructure. It is materially stronger than "we mocked it" for resilience and change-control evidence.

Q: Why is dependency-injecting the database connection into the loader the key enabler for most of these integration techniques?
A: Injecting the engine/connection lets the test point the loader at the container's mapped port, control the transaction boundary for rollback isolation, run it as a chosen least-privileged role, and reuse one container across cases. A loader that constructs its own hard-coded connection can do none of this and is effectively untestable at the seam. So the refactor to accept an injected connection unlocks port wiring, isolation, privilege testing, and reuse all at once.

Q: What `timeout` and `interval` arguments does `wait_for_logs` accept, and why set a finite timeout in CI?
A: `wait_for_logs(container, predicate, timeout=..., interval=...)` polls the container's logs every `interval` seconds until the predicate matches or `timeout` seconds elapse; with `timeout=None` it waits indefinitely. In CI you set a finite `timeout` so a container that never becomes ready fails fast with a `TimeoutError` instead of hanging the whole job until the runner's outer limit kills it. An indefinite wait is the classic cause of a stuck pipeline that "ran for an hour" pulling or initialising a wedged container.

Q: Where exactly do you put `addopts`, `markers`, and `--strict-markers` for the integration split in a uv project?
A: All three live under `[tool.pytest.ini_options]` in `pyproject.toml`: `markers = ["integration: spins up real containers"]`, and `addopts = "--strict-markers -m 'not integration'"` so a bare `uv run pytest` excludes the suite and errors on unknown markers. Developers then opt in with `uv run pytest -m integration`, which overrides the default marker expression. Keeping this config in `pyproject.toml` means there is one source of truth and no separate `pytest.ini`.

Q: A MinIO-backed loader passes on small golden files but a larger-object test intermittently fails — what seam behaviour would you check?
A: Large uploads cross the multipart-upload threshold, so the client splits the object into parts and completes them, and a real bucket enforces part ordering and ETag/completion semantics a mock would skip. Check that the loader handles multipart completion and any retry on a failed part, and that the bucket/region config matches; MinIO speaks the real S3 API so it surfaces these where a fake silently accepts one blob. This is exactly the kind of size-dependent seam bug that only a real object store reveals.
