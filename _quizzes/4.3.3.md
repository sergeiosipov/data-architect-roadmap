# Phase 2 · 4.3.3 Embedded analytical engines (DuckDB) — 150 self-test questions

Q: What does it mean that DuckDB is an "in-process" (embedded) database engine?
A: It means DuckDB runs as a library linked directly into your application's process rather than as a separate server you connect to over a socket. There is no DBMS server software to install, start, or maintain, and queries execute inside your process so data moves to and from the engine without inter-process serialization. The classic analogy is "SQLite for analytics" — you `import duckdb` (or link the C/C++ library) and a full columnar OLAP engine is running inside your program.

Q: Name two practical consequences of DuckDB being embedded rather than client/server.
A: First, there is no network round-trip or server to provision, so query results land directly in your process memory — for example a DuckDB query can return straight into a pandas or Polars DataFrame with zero copy via Arrow. Second, the database file's read-write access is owned by exactly one process at a time, because there is no central server to arbitrate concurrent connections from many clients.

Q: How do you start an in-memory DuckDB database in Python, and what happens to the data on exit?
A: You call `duckdb.connect()` with no path, or `duckdb.connect(':memory:')`, which creates a transient database that lives only in RAM. All tables and data are discarded when the connection closes or the process exits — nothing is persisted to disk. To persist, you instead pass a file path like `duckdb.connect('fund.duckdb')`, which creates or opens a single-file on-disk database.

Q: What is the "single-file database" storage model in DuckDB?
A: A persistent DuckDB database is stored as one self-contained file (commonly `.duckdb` or `.db`) that holds all tables, indexes, and metadata in DuckDB's own columnar storage format. This makes a database trivial to copy, back up, or ship — you move one file. It contrasts with a directory-of-files lake format; the single file is DuckDB's native managed storage.

Q: Why is DuckDB often described as "the architect's pocket warehouse"?
A: Because it gives you a real analytical (OLAP) engine — columnar storage, vectorized execution, full SQL — with zero infrastructure, so it fits local development, fast testing, and genuinely useful mid-size production marts that run on a single node. Knowing where that envelope holds sharpens every "do we actually need a cluster?" decision. For a fund administrator whose marts fit comfortably on one machine, "no cluster" can be the cheapest, most maintainable architecture.

Q: What is a "columnar-vectorized" query execution engine?
A: Columnar means data is stored and processed one column at a time rather than row by row, which is ideal for analytics that scan a few columns across many rows. Vectorized means the engine processes a batch (a "vector") of values per operation instead of one value at a time, amortizing interpreter overhead and using CPU caches and SIMD efficiently. Together they make aggregations and scans over large tables far faster than a row-store like a traditional OLTP database.

Q: Why does columnar storage help a query like `SELECT avg(nav) FROM holdings`?
A: Because only the `nav` column needs to be read from storage; a columnar engine skips every other column entirely, reducing I/O dramatically. In a row store the whole row (ISIN, currency, quantity, etc.) is read even though you only want one field. This column pruning is a core reason DuckDB is fast for analytical scans over wide fund tables.

Q: What transactional guarantees does DuckDB provide?
A: DuckDB provides full ACID guarantees (Atomicity, Consistency, Isolation, Durability) through a custom, bulk-optimized Multi-Version Concurrency Control (MVCC) implementation. This means a failed bulk load rolls back cleanly and concurrent readers see a consistent snapshot. ACID compliance is what makes it safe to use as a real warehouse rather than just a query tool over files.

Q: What is "out-of-core" execution and why does it matter for DuckDB?
A: Out-of-core execution is the ability to run a query whose working set is larger than available RAM by spilling intermediate data to disk instead of failing with an out-of-memory error. It matters because it lets DuckDB process datasets bigger than the machine's memory on a single node, extending the "no cluster" envelope. For example, sorting or hash-aggregating a holdings table larger than RAM still completes, just more slowly.

Q: Which DuckDB operators can spill to disk to support larger-than-RAM queries?
A: The memory-intensive blocking operators can spill: large hash-table aggregations (group-by with many distinct groups), hash joins on large tables, sorts (ORDER BY) on larger-than-memory data, and window functions over large datasets. These operators build big intermediate structures, so when memory runs short they write partitions to the temp directory and stream them back. Simple streaming scans and filters rarely need to spill.

Q: What is the `memory_limit` setting and what is its default?
A: `memory_limit` caps how much memory DuckDB's buffer manager will use before it starts spilling to disk. By default it is set to 80% of the machine's physical RAM (for example, on a 16 GB machine it defaults to about 12.8 GB). You change it with a statement like `SET memory_limit = '4GB';`, which is useful to leave headroom for other processes on a shared box.

Q: What does the `temp_directory` setting control, and where does it default?
A: `temp_directory` is the on-disk location where DuckDB writes spill files when intermediate data exceeds `memory_limit`. For a persistent database it defaults to a directory alongside the database file (a `.tmp` path derived from the database name), and for an in-memory database it defaults to a `.tmp` directory in the working directory. Pointing it at a fast local SSD with plenty of free space is what makes out-of-core queries practical.

Q: What is `max_temp_directory_size` and why might you tune it?
A: `max_temp_directory_size` caps how much data DuckDB may write to the spill (temp) directory; it defaults to 90% of the remaining free disk space on that volume. You tune it down to protect a shared disk from being filled by a runaway spilling query, or up (with caution) on a dedicated scratch disk. If a query needs more spill than this limit allows, it fails rather than filling the disk.

Q: When does DuckDB report an out-of-memory error despite supporting spilling?
A: It reports an out-of-memory error and cancels the query when the memory limit is breached and spilling cannot save it — for example the `temp_directory` is disabled/unwritable, the `max_temp_directory_size` is exceeded, or the operator in question cannot spill. So an OOM in DuckDB usually means "spilling was blocked," which is why the first troubleshooting step is to check that a writable temp directory with free space is configured.

Q: A larger-than-RAM aggregation fails with an out-of-memory error on a fresh DuckDB. What is the first thing to check?
A: Check that `temp_directory` is set to a writable location with enough free disk space, because without a usable spill directory the spilling operators have nowhere to offload and will OOM. Run `SET temp_directory = '/data/duckdb_tmp';` pointing at a fast SSD volume, and confirm `max_temp_directory_size` is not throttling it. Then re-run — the query should now spill and complete instead of failing.

Q: How can you read a Parquet file in DuckDB without first loading it into a table?
A: You query it directly as if it were a table using `read_parquet('file.parquet')`, for example `SELECT * FROM read_parquet('holdings.parquet');`. DuckDB also lets you skip the function when the path ends in `.parquet`, so `SELECT * FROM 'holdings.parquet'` works too. No import/copy step is needed — DuckDB scans the file in place.

Q: What is `parquet_scan` and how does it relate to `read_parquet`?
A: `parquet_scan` is simply an alias for `read_parquet`; both are the table-producing functions that scan one or more Parquet files. You can use either name interchangeably, e.g. `SELECT * FROM parquet_scan('nav.parquet')`. The `read_parquet` form is the more commonly documented spelling.

Q: How do you read many Parquet files at once in DuckDB?
A: You can pass a glob pattern, e.g. `SELECT * FROM read_parquet('nav/*.parquet')`, or pass an explicit list, e.g. `read_parquet(['2026-01.parquet','2026-02.parquet'])`. DuckDB reads all matching files as a single logical table. This is ideal for date-partitioned NAV exports landed as many files in object storage.

Q: What does the `hive_partitioning` option do when reading Parquet?
A: It tells DuckDB to interpret directory names of the form `key=value` (Hive-style partitioning) as additional columns, e.g. a path `…/fund=LU1234567890/date=2026-06-13/part.parquet` yields `fund` and `date` columns. This lets you filter on partition columns in SQL and have DuckDB prune unneeded directories. You enable it with `read_parquet('path/**/*.parquet', hive_partitioning = true)`.

Q: What problem does the `union_by_name` option solve when scanning multiple Parquet files?
A: `union_by_name = true` matches columns across files by name rather than by position, so files with slightly different schemas (extra or reordered columns) are unioned correctly, filling missing columns with NULL. Without it, DuckDB assumes all files share the same column order and a schema drift causes wrong alignment or an error. It is invaluable when EMT/EPT-style exports evolve their column set over time.

Q: How do you inspect a Parquet file's schema and metadata from DuckDB?
A: Use the metadata functions: `parquet_schema('file.parquet')` returns the column types and structure, `parquet_metadata('file.parquet')` returns row-group and column statistics, `parquet_file_metadata('file.parquet')` returns file-level metadata, and `parquet_kv_metadata('file.parquet')` returns custom key-value pairs. For example `SELECT * FROM parquet_schema('holdings.parquet');` shows the logical and physical types without reading the data.

Q: Which DuckDB extension is required to read files over HTTP/S3, and is it bundled?
A: The `httpfs` extension provides HTTP(S) and S3 (and other S3-compatible) access. It is autoloadable and bundled with the standard binaries, so reading `https://…/file.parquet` typically works with no manual install. For S3 you additionally configure credentials, after which `read_parquet('s3://bucket/key.parquet')` streams the file directly.

Q: How do you configure S3 credentials for DuckDB in the modern API?
A: You create a secret with the `CREATE SECRET` statement, for example `CREATE SECRET (TYPE s3, KEY_ID '…', SECRET '…', REGION 'eu-central-1');`. DuckDB then uses that secret automatically for `s3://` paths via `httpfs`. The secrets manager replaced the older approach of `SET s3_access_key_id` global settings and supports provider chains like `CREDENTIAL_CHAIN` for IAM roles.

Q: Can DuckDB read only the needed columns and row groups from a remote Parquet file?
A: Yes — DuckDB pushes down projection (column pruning) and predicate filters into the Parquet reader, and with `httpfs` it issues HTTP range requests to fetch only the relevant byte ranges. This means `SELECT isin, nav FROM read_parquet('s3://…/holdings.parquet') WHERE date = '2026-06-13'` can avoid downloading columns and row groups that statistics rule out. It is what makes querying lake data over object storage practical rather than wasteful.

Q: What is the DuckDB Iceberg extension and what does it provide?
A: The `iceberg` extension lets DuckDB read Apache Iceberg tables; you load it with `INSTALL iceberg; LOAD iceberg;`. It provides functions such as `iceberg_scan('…')` to query a table's current snapshot, plus `iceberg_metadata(...)` and `iceberg_snapshots(...)` to inspect manifest/snapshot history. This lets DuckDB act as a lightweight query engine over an Iceberg lakehouse without Spark.

Q: How does DuckDB read Delta Lake tables, and what is the key function?
A: Through the `delta` extension (`INSTALL delta; LOAD delta;`), which exposes `delta_scan('path')` to read a Delta table including its transaction-log-driven current state. For example `SELECT * FROM delta_scan('s3://bucket/holdings_delta');`. It uses the delta-kernel library so DuckDB respects Delta's protocol for which files are live.

Q: Reading Parquet directly from object storage vs reading Iceberg/Delta — what is the difference?
A: Reading raw Parquet treats files as a flat dataset with no transactional layer: you see whatever files match the glob, including partial or stale writes. Iceberg and Delta add a metadata/transaction log that defines a consistent table snapshot, schema evolution, and time travel, so `iceberg_scan`/`delta_scan` give you exactly the committed state. For regulated, auditable fund data the table-format guarantees usually matter more than raw-file convenience.

Q: What is the extension ecosystem in DuckDB and why is it central to the design?
A: DuckDB keeps a small core and pushes capabilities into a flexible extension mechanism that can add data types, functions, file formats, and even SQL syntax. Features like Parquet, JSON, full-text search, spatial, `httpfs`/S3, Iceberg, and Delta are implemented as extensions rather than baked into the core. This keeps the engine lean while letting you opt into exactly what a workload needs.

Q: How do you install and load a DuckDB extension, and what is the difference between the two steps?
A: `INSTALL name;` downloads the extension binary to local storage (once), and `LOAD name;` makes it active in the current session. For example `INSTALL httpfs; LOAD httpfs;`. Install is persistent across sessions; load must happen per session unless autoloading kicks in for known extensions.

Q: What are "core" versus "community" extensions in DuckDB?
A: Core extensions are maintained by the DuckDB team and signed/distributed through the official repository (e.g. `httpfs`, `json`, `parquet`, `iceberg`). Community extensions are third-party contributions distributed through a separate community repository and may require allowing unsigned extensions. Knowing the distinction matters for governance: a regulated shop may restrict installs to signed core extensions only.

Q: What is the single-writer constraint in DuckDB and what does it rule out?
A: Only one process can hold a read-write connection to a given database file at a time, so DuckDB cannot serve as a shared multi-writer database that many independent processes or services write to concurrently. It rules out patterns like several application instances or a load-balanced API tier all writing to the same `.duckdb` file. The fix is to funnel writes through a single owning process, or use a different store for concurrent writes.

Q: Within a single DuckDB process, can multiple threads write concurrently?
A: Yes — within one process DuckDB supports multiple writer threads using MVCC plus optimistic concurrency control. Appends never conflict, and updates to separate tables or to disjoint sets of rows proceed in parallel. The constraint is across processes (one read-write process per file), not across threads inside that process.

Q: What happens when two threads in the same DuckDB process try to update the same row?
A: Optimistic concurrency control detects the write-write conflict at commit time, and the second transaction to attempt the edit fails with a transaction conflict error rather than silently losing an update. The documented remedy is simply to rerun the conflicting transaction. Appends and edits to different rows do not conflict, so this only affects genuine overlapping updates.

Q: Several processes need to read the same DuckDB file at once — how do you allow it?
A: Open each connection in read-only mode, e.g. with `access_mode = 'READ_ONLY'` (or `read_only=True` in the Python API). With every process read-only, multiple processes can read the same file simultaneously, since none holds the exclusive read-write lock. The trade-off is that no process can write while these read-only readers are attached.

Q: You get `IO Error: Could not set lock on file` opening a DuckDB database. What does it mean?
A: It means another process already holds a read-write lock on that database file, and DuckDB enforces a single owning read-write process per file. Common causes are a leftover process (e.g. a notebook kernel, a still-running script, or an auto-reloading web server like Uvicorn with `reload=True`) holding the file open. Resolve it by closing the other connection/process, or open in read-only mode if you only need to read.

Q: A FastAPI/Uvicorn app with `reload=True` intermittently throws a DuckDB lock error. Why?
A: The auto-reloader runs a watcher process plus a worker process, and if both try to open the same `.duckdb` file read-write you get two read-write attempts on one file, which DuckDB rejects with `Could not set lock on file`. The single-writer constraint is doing exactly what it should. Fixes include disabling reload in that context, ensuring only one process opens the DB, or using read-only connections where appropriate.

Q: What is MotherDuck and how does it relate to DuckDB?
A: MotherDuck is a managed, cloud service built around DuckDB that extends the embedded engine with cloud storage, sharing, and hybrid local/cloud query execution. It is meant to address exactly the gaps of pure embedded DuckDB — shared multi-user access, durable cloud-hosted databases, and collaboration — while keeping the DuckDB SQL experience. In the syllabus it sits at awareness level: know it exists as the managed path when "single process on one laptop" stops being enough.

Q: What is "hybrid execution" in the MotherDuck context?
A: Hybrid execution means a query can run partly on your local DuckDB and partly in MotherDuck's cloud, with the planner deciding where each piece runs based on where the data lives. Local files stay local while cloud-hosted tables are processed server-side, and results are combined. This lets you join a small local CSV against a large cloud table without uploading everything.

Q: What is ClickHouse and when is it relevant versus DuckDB?
A: ClickHouse is a server-based columnar OLAP database known for very high single-node (and clustered) query speed and high-concurrency serving. It is relevant when you need a long-running server that many clients query concurrently, or sustained ingestion with low-latency dashboards — workloads beyond what an embedded single-writer engine serves well. DuckDB stays the choice for in-process analytics, local/ephemeral compute, and querying lake files; the syllabus keeps ClickHouse at awareness level as the server-side single-node speed option.

Q: Give the headline difference between DuckDB and ClickHouse in deployment terms.
A: DuckDB is embedded — it runs inside your application process with no server — whereas ClickHouse is a standalone server that clients connect to over the network. That single distinction drives most of the others: ClickHouse naturally handles many concurrent client connections and continuous writes, while DuckDB excels at single-process, in-process analytics with no operational footprint.

Q: For a mart that fits on one node and is rebuilt nightly, why might DuckDB beat a Spark cluster?
A: Because a single-node columnar engine with no cluster to provision, schedule, or pay for is dramatically simpler and cheaper, and for data that fits comfortably (with out-of-core covering occasional overflow) the distributed coordination of Spark adds latency and cost without payback. The architect's job is to recognize the size/concurrency envelope: nightly batch rebuild of a one-node mart is squarely in DuckDB's sweet spot. You only reach for a cluster when data or concurrency genuinely exceeds one machine.

Q: What is the data-size/concurrency "envelope" where "no cluster" is the right architecture?
A: It is roughly: data that fits on one node's disk (with RAM-sized hot working sets, helped by out-of-core spilling for the rest), batch or low-concurrency analytical access rather than many simultaneous writers, and a single owning process for writes. Within that envelope a single-node engine like DuckDB or a beefy Postgres is cheaper and simpler than a distributed system. You leave the envelope when concurrent writers, multi-terabyte hot data, or high-QPS serving appear.

Q: Why does the single-writer constraint not disqualify DuckDB from production use?
A: Because many production analytical workloads are single-writer by nature — a nightly or hourly batch job that one process owns, rebuilding marts read by many consumers afterward. The constraint only rules out concurrent multi-process writes to the same file, which batch analytics rarely needs. Consumers can read the resulting file (or read-only replicas/copies), so the pattern scales for reads.

Q: How would you serve many concurrent readers from a DuckDB-built mart despite the write lock?
A: Have a single writer process build the database/Parquet output, then distribute read-only access — for example open read-only connections, or publish the result as Parquet files in object storage that any number of consumers query independently. Because the heavy write is isolated to one batch process, the read fan-out is unconstrained. MotherDuck or a copy-per-reader strategy are alternatives when shared concurrent reads of one managed DB are required.

Q: In the Phase-2 "Do," you benchmark the mart on DuckDB over Parquet vs Postgres. What outcome typically favours DuckDB?
A: Large analytical scans and aggregations over columnar Parquet typically favour DuckDB, because its columnar-vectorized engine reads only needed columns and processes them in batches, often beating a row-oriented Postgres on full-table scans. Cold queries straight over Parquet with predicate/projection pushdown also avoid a load step. The win is most pronounced for wide tables where the query touches few columns.

Q: In that same benchmark, what typically favours Postgres over DuckDB?
A: Workloads with many concurrent writers and readers, point lookups/selective indexed reads, and transactional OLTP-style access favour Postgres, because it is a multi-connection server with rich indexing and concurrency control. DuckDB's single-writer model and analytics focus are a poor fit there. So the honest five-bullet conclusion pairs DuckDB-wins (single-writer batch analytics, columnar scans, lake files) against Postgres-wins (concurrency, OLTP, point queries, shared service).

Q: What resource-use differences should you record when benchmarking DuckDB vs Postgres?
A: Record wall-clock query time, peak memory used (and whether DuckDB spilled to the temp directory), CPU utilization/parallelism, and disk I/O, for each engine and query. You should also note setup cost — DuckDB needs no load step over Parquet, whereas Postgres needs the data ingested and indexed first. These numbers make the "when each wins" argument concrete rather than hand-wavy.

Q: How do you observe out-of-core behaviour with a larger-than-RAM holdings table in DuckDB?
A: Lower `memory_limit` (e.g. `SET memory_limit='2GB'`) or use a genuinely larger-than-RAM table, set a `temp_directory` on a disk you can watch, then run a spilling query like a big sort or group-by. Monitor the temp directory growing during execution and observe the query still completing — that growth is DuckDB writing spill partitions. If you disable the temp directory, the same query OOMs, demonstrating the mechanism.

Q: Does querying Parquet over object storage with DuckDB require loading the data first?
A: No — DuckDB scans Parquet in place via `read_parquet('s3://…')` using the `httpfs` extension, applying column pruning and predicate pushdown so it fetches only relevant byte ranges. There is no ingest/copy step as there would be loading into Postgres. This zero-ETL querying is a major part of DuckDB's lakehouse appeal.

Q: What does it mean that DuckDB queries can integrate "zero-copy" with DataFrames?
A: Because DuckDB runs in-process and speaks Apache Arrow, it can read from and return to pandas/Polars/Arrow DataFrames without serializing data across a network or copying buffers unnecessarily. For example `duckdb.sql('SELECT * FROM my_df').df()` queries a Python DataFrame directly and returns one. This tight integration is a direct benefit of the embedded model.

Q: How does DuckDB execute queries in parallel on a single machine?
A: It uses multi-threaded, morsel-driven parallelism: work is split into small chunks (morsels) of a column scan and distributed across CPU cores, with the vectorized operators running in parallel. This lets one node saturate all cores on a big aggregation without any cluster. You can cap it with `SET threads = N;` when you need to limit CPU use on a shared box.

Q: Why is DuckDB a good fit for fast, deterministic testing of a data pipeline?
A: Because spinning up an in-memory DuckDB is instantaneous and needs no server, so each test gets a clean, isolated database in milliseconds, and it speaks standard SQL plus reads Parquet/CSV directly. This makes unit and integration tests for transformation logic fast and hermetic. Many teams use `dbt-duckdb` precisely so model logic can be tested locally without a warehouse.

Q: What is `pg_duckdb` and what problem does it solve?
A: `pg_duckdb` is a Postgres extension that embeds DuckDB's execution engine inside Postgres, so you can run DuckDB-accelerated columnar/analytical queries — including reads of Parquet and lake data — within the same Postgres that serves OLTP. The benefit is analysts querying live operational data with no separate warehouse to load. It is the Postgres-native alternative when you want columnar analytics co-located with transactional data.

Q: When is `pg_duckdb`/Citus columnar a worse choice than DuckDB-on-a-lake or Trino/Spark?
A: They are worse when you need true lakehouse scale, open table formats as the system of record, or independent scaling of storage and compute — that is Trino/Spark/warehouse territory, not "analytics bolted onto one Postgres." `pg_duckdb` shines for live-data analytics inside Postgres but does not give you elastic distributed compute. Recognizing that boundary is exactly the architect skill this lesson builds.

Q: How does Citus columnar differ from `pg_duckdb` as a Postgres analytics path?
A: Citus provides columnar table storage (and optional distributed sharding) inside Postgres, compressing and column-organizing data so analytical scans over Postgres-managed tables are faster, whereas `pg_duckdb` brings DuckDB's whole vectorized engine and lake-file reading into Postgres. Citus is about distribution and columnar storage in Postgres; `pg_duckdb` is about embedding the DuckDB engine. Both keep analytics next to OLTP rather than in a separate warehouse.

Q: Why is "knowing DuckDB's limits" framed as sharpening the cluster conversation?
A: Because the most expensive mistakes in data architecture come from reaching for a distributed cluster when a single node would do, or vice versa. Knowing precisely where DuckDB's single-writer, single-node, out-of-core envelope ends gives you the evidence to say "no cluster needed" confidently — or to justify one. That judgement, backed by benchmarks, is the deliverable of this lesson.

Q: How do you persist an in-memory DuckDB database's tables to a file?
A: You can `ATTACH 'fund.duckdb' AS f;` and create/copy tables into the attached file, or simply open the connection on the file path from the start so writes are durable. To export query results you can also `COPY (SELECT …) TO 'out.parquet' (FORMAT parquet);`. The point is that in-memory data is volatile until written to an on-disk database or to files.

Q: How do you write query results to Parquet from DuckDB?
A: Use the `COPY` statement, e.g. `COPY (SELECT * FROM holdings) TO 'holdings.parquet' (FORMAT parquet);`. You can add options like `COMPRESSION 'zstd'` and, for partitioned output, `PARTITION_BY (fund_id)` to write a Hive-partitioned directory tree. This is how a DuckDB batch job publishes a mart as lake files for downstream readers.

Q: What does `PARTITION_BY` do in a DuckDB `COPY ... TO` statement?
A: It writes the output as a Hive-partitioned directory layout, creating subdirectories like `fund=LU1234567890/date=2026-06-13/` and splitting rows into files by the partition columns. Readers can then prune partitions when filtering on those columns. It mirrors the `hive_partitioning` read option, so write and read sides line up.

Q: Can DuckDB query CSV and JSON files directly like it does Parquet?
A: Yes — `read_csv('file.csv')` (with auto-detection of types/headers) and `read_json('file.json')` let you query those formats in place, and the bare `SELECT * FROM 'file.csv'` shorthand works for known extensions. JSON support comes via the `json` extension, typically autoloaded. This makes DuckDB handy for ad-hoc analysis of raw fund extracts before they are formalized.

Q: What is the difference between DuckDB's native storage format and Parquet?
A: The native `.duckdb` format is DuckDB's own managed, single-file storage optimized for its engine, supporting in-place updates, indexes, and ACID transactions. Parquet is an open, immutable columnar file format meant for interchange and lake storage that DuckDB reads/writes but does not update in place. You use native storage for a managed database and Parquet for portable, queryable data files.

Q: Why might you keep a fund mart as Parquet in object storage rather than in a native DuckDB file?
A: Parquet in object storage is an open, durable, multi-reader format that any engine (DuckDB, Spark, Trino, pandas) can read, and it avoids the single-writer file-lock constraint for consumers. A native DuckDB file is great for a single owning process but is locked while that process writes. For a shared, regulated fund mart, open Parquet files give broad read access and tool independence.

Q: What is the role of predicate pushdown when DuckDB scans Parquet?
A: Predicate pushdown means filters in your `WHERE` clause are evaluated against Parquet row-group statistics (min/max) so DuckDB skips entire row groups that cannot match, reading far less data. For example `WHERE date = '2026-06-13'` can skip every row group whose date range excludes that day. Combined with column projection, it minimizes I/O, which matters most when reading over object storage.

Q: What is projection pushdown in DuckDB's Parquet reader?
A: Projection pushdown means DuckDB reads only the columns referenced by the query from the Parquet file, ignoring all others entirely. Because Parquet stores columns separately, the unneeded columns are never fetched or decoded. Selecting `isin, nav` from a 50-column holdings file therefore reads only two columns' worth of data.

Q: How do you limit the number of CPU threads DuckDB uses, and why would you?
A: Set `SET threads = 4;` (or pass the config at connect time) to cap parallelism. You do this on a shared machine to leave cores for other workloads, or to make benchmarks reproducible, since DuckDB defaults to using all available cores. Reducing threads trades throughput for predictability and neighbour-friendliness.

Q: A DuckDB query over remote S3 Parquet is slow. What are the first things to check?
A: Check that predicate and projection pushdown are actually reducing reads (avoid `SELECT *` and broad scans), that the data is in reasonably sized row groups, and that the network/region latency to the bucket is acceptable (querying eu-central-1 data from far away is slow). Also confirm `httpfs` is using range requests rather than downloading whole files. Caching or copying hot data locally is the fallback when object-storage latency dominates.

Q: Why is DuckDB suitable as a backend for `dbt-duckdb` in local development?
A: Because it provides a real SQL warehouse with zero setup — an in-process engine that runs your dbt models against Parquet/CSV or a local file — so the whole transformation DAG can be built and tested on a laptop. There is no warehouse to provision or pay for during development. The same SQL largely transfers to a server warehouse later, making DuckDB a cheap, fast dev/CI target.

Q: What is the risk of treating a single-writer DuckDB file as a shared team database?
A: The first person to open it read-write locks it; everyone else gets `Could not set lock on file` or must go read-only, so it does not behave like a shared multi-writer service. Teams that try this hit constant lock contention and stale reads. The correct pattern is a single batch writer plus read-only distribution, or a managed option like MotherDuck for shared access.

Q: How does DuckDB's storage handle updates and deletes given its columnar format?
A: DuckDB supports in-place `UPDATE`/`DELETE` with MVCC, storing row versions so concurrent readers see a consistent snapshot while a writer modifies data. This is in its native managed storage; raw Parquet files are immutable and cannot be updated in place. So full SQL DML works on native DuckDB tables but not directly on Parquet you scan.

Q: How would DuckDB help validate a NAV calculation against a Parquet holdings export?
A: You can point DuckDB straight at the export, e.g. `SELECT fund_id, sum(quantity*price) AS gross FROM read_parquet('holdings.parquet') GROUP BY fund_id;`, and compare to the reported NAV — no load step needed. Its fast aggregation over columnar data makes such reconciliation checks quick even on large files. This is a realistic local-analytics use in fund administration.

Q: Why is DuckDB convenient for ad-hoc querying of EMT/EPT files received as Parquet?
A: Because you can query the received file in place with full SQL — filtering, joining against reference data, and aggregating — without standing up a database or running an import. For example you can immediately check `SELECT count(*), count(DISTINCT isin) FROM read_parquet('emt_2026-06.parquet');` to profile the delivery. The zero-setup, file-native model fits irregular, file-based regulatory feeds well.

Q: Can a single DuckDB process attach and query multiple databases at once?
A: Yes — you can `ATTACH 'other.duckdb' AS o;` (and even attach Postgres, MySQL, or SQLite via the respective extensions) and then query across them with qualified names like `o.schema.table`. This lets one process join native DuckDB tables, Parquet files, and an external database in a single SQL statement. It is useful for combining a local mart with reference data living elsewhere.

Q: How can DuckDB read directly from a live Postgres database?
A: Using the `postgres` extension (`INSTALL postgres; LOAD postgres;`) you can `ATTACH 'host=… dbname=…' AS pg (TYPE postgres);` and then query Postgres tables as `pg.schema.table`. DuckDB executes scans against Postgres and can push down some work. This is handy for pulling operational tables into a DuckDB-side analytical query without a separate ETL.

Q: What is the practical meaning of "DuckDB has no server to scale horizontally"?
A: Because it runs in-process on one machine, you cannot add nodes to a DuckDB to handle more data or concurrency — you scale by using a bigger machine (vertical) or by moving to a distributed system. Its parallelism is across the cores of one node, not across a cluster. This is precisely the limit that defines when you must leave the "no cluster" envelope.

Q: How does out-of-core execution change DuckDB's effective data ceiling versus pure in-memory engines?
A: It raises the ceiling from "must fit in RAM" to "must fit on disk (with enough temp space)," because spilling lets larger-than-RAM joins, sorts, and aggregations complete, just more slowly. A pure in-memory engine would OOM on the same query. So DuckDB can credibly handle datasets several times the size of RAM on a single node.

Q: Is DuckDB's out-of-core spilling free in performance terms?
A: No — spilling writes intermediate data to disk and reads it back, so a spilling query is significantly slower than one that fits in memory, and the speed of the temp disk (SSD vs HDD) matters a lot. It prevents failure but is a graceful degradation, not a free lunch. The architecture lesson is that out-of-core extends the envelope but you pay in latency near its edges.

Q: What signals tell you a DuckDB workload has outgrown the single-node model?
A: Persistent need for many concurrent writers, hot data far larger than one node's disk, high-QPS interactive serving to many users, or queries that spill so heavily they are unacceptably slow. Any of these signals you are leaving the embedded envelope toward MotherDuck (managed), ClickHouse (server-side speed/concurrency), or a Trino/Spark lakehouse. Catching these signals early is the architect's job.

Q: Why is "DuckDB is like SQLite for analytics" a useful but imperfect analogy?
A: It is useful because both are embedded, single-file, zero-server libraries you link into a process. It is imperfect because SQLite is a row-store optimized for OLTP/transactional point access, whereas DuckDB is a columnar-vectorized OLAP engine optimized for analytical scans and aggregations. They share the embedding model but target opposite query workloads.

Q: How does DuckDB integrate with Polars or Arrow without copying data?
A: DuckDB can scan an Arrow table or a Polars DataFrame directly because it shares the Arrow memory layout, so the data is referenced in place rather than serialized and re-copied. For example registering an Arrow table and querying it returns results that can flow back into Arrow/Polars zero-copy. This is a concrete payoff of the in-process design.

Q: What does `SET memory_limit='0'` or disabling it imply, and why is it risky?
A: Removing the cap lets DuckDB use unbounded memory, which can starve the rest of the machine and trigger the OS OOM killer instead of DuckDB's controlled spilling. The safer pattern is to keep a sensible `memory_limit` and a writable `temp_directory` so heavy queries spill gracefully. Disabling the limit on a shared host is a common way to cause hard-to-diagnose crashes.

Q: How can you confirm DuckDB actually spilled during a query?
A: Watch the `temp_directory` for files appearing/growing during execution, or inspect query profiling output (`PRAGMA enable_profiling; EXPLAIN ANALYZE …`) which reports operator memory and spilling behaviour. A growing temp directory plus a longer runtime than the in-memory case is the practical evidence. This is exactly the observation the Phase-2 "Do" asks you to make on a larger-than-RAM table.

Q: What is the difference in concurrency story between MotherDuck and embedded DuckDB?
A: Embedded DuckDB enforces a single read-write process per file, so shared concurrent access is awkward. MotherDuck hosts the database in the cloud as a managed service, enabling multiple users and applications to share and concurrently access data, with hybrid local/cloud execution. So MotherDuck is the answer when the single-writer file model becomes the bottleneck.

Q: When evaluating ClickHouse vs DuckDB for a dashboard backend serving 200 concurrent users, which fits and why?
A: ClickHouse fits better, because it is a server designed for high concurrency and continuous low-latency serving of many simultaneous client connections. Embedded DuckDB, with its single read-write process and in-process model, is not built to be a shared high-QPS service. DuckDB would instead be the tool that builds or pre-aggregates the data ClickHouse serves.

Q: Why does the lesson keep MotherDuck and ClickHouse at "awareness level"?
A: Because the hands-on skill being built is mastering embedded DuckDB and its single-node envelope; MotherDuck and ClickHouse are the named escape hatches you must recognize, not tools you implement here. Knowing what each solves (managed sharing vs server-side concurrency/speed) is enough to make the architecture call. Deeper hands-on with them belongs to later, higher-tier work.

Q: How do row-group statistics in Parquet help DuckDB, and what can defeat them?
A: Min/max statistics per row group let DuckDB skip groups that cannot satisfy a filter (predicate pushdown), so well-sorted data on the filter column gives big skips. They are defeated when data is unsorted (every row group spans the whole value range) or when row groups are huge, so almost nothing can be pruned. Writing Parquet sorted/partitioned by common filter columns therefore improves DuckDB read performance.

Q: What is a sensible architecture for a Luxembourg fund mart that fits on one node but is read by many downstream systems?
A: A single DuckDB (or Postgres) batch process builds the mart nightly and writes it to open Parquet in object storage (optionally as an Iceberg/Delta table for transactional snapshots), and downstream systems read those files independently. This respects the single-writer constraint (one writer) while giving unlimited read fan-out and tool independence. It is the cheapest maintainable design when data genuinely fits on one node.

Q: Why might reading Iceberg/Delta be preferable to a glob of raw Parquet for an audited NAV history?
A: Because Iceberg and Delta provide a transaction log and immutable snapshots, so you can time-travel to exactly the committed state as of a date and trust that you are not reading half-written or stale files. For regulated, auditable NAV history this consistency and reproducibility is essential. A raw Parquet glob has no such transactional boundary and can expose partial writes.

Q: How does DuckDB's vectorized engine interact with out-of-core spilling?
A: Vectorized processing handles batches of values in memory for speed, while the buffer manager monitors total memory against `memory_limit` and, when a blocking operator's structures grow too large, partitions and spills them to the temp directory. The two are complementary: vectorization optimizes the in-memory fast path, spilling guarantees correctness past the memory ceiling. Together they give fast queries that still complete when data exceeds RAM.

Q: What does `EXPLAIN` (and `EXPLAIN ANALYZE`) give you in DuckDB?
A: `EXPLAIN` shows the physical query plan (which scans, joins, aggregations, and pushdowns DuckDB chose), and `EXPLAIN ANALYZE` additionally runs the query and reports per-operator timing, rows, and resource use. You use them to confirm projection/predicate pushdown is happening and to find the operator that dominates time or spills. They are the primary tool for the benchmarking and tuning the Phase-2 "Do" requires.

Q: Why can DuckDB read a Parquet file that a row-oriented database would need to import first?
A: Because DuckDB's scan operators understand the Parquet format natively and integrate it through the parquet extension, treating the file as a first-class table source with statistics-driven pushdown. A typical row store needs the data copied into its own storage and indexed before it can query efficiently. This in-place querying removes the load step entirely for read-only analytics.

Q: What is the operational risk of putting `temp_directory` on a small or full disk?
A: If spilling needs more space than the disk (or `max_temp_directory_size`) allows, the query aborts with an out-of-memory/temp-space error even though logically it could have completed. So a too-small spill disk silently caps how large a query you can run. The fix is to point `temp_directory` at a roomy, fast SSD volume sized for your largest out-of-core query.

Q: How does the single-file format help with backups and reproducibility in a regulated context?
A: Because the entire database is one file, a backup or point-in-time copy is just copying that file (when no writer is active), giving a self-contained, reproducible artifact you can archive for audit. There is no multi-file consistency problem to coordinate. For regulated fund data, a single immutable copy of the analytical store simplifies retention and evidence.

Q: Can DuckDB enforce schema and types when reading messy CSV fund extracts?
A: Yes — `read_csv` auto-detects types but you can override with explicit `columns`/`types`, set `header`, `delimiter`, `nullstr`, and use `ignore_errors` or `store_rejects` to handle bad rows. This lets you impose a known schema on a messy extract rather than trusting auto-detection. Explicit typing is important when an ISIN or a date column must not be silently inferred wrong.

Q: Why is DuckDB a good "first thing to reach for" when profiling an unfamiliar data file?
A: Because with one line it gives you full SQL over the file in place — counts, distinct values, min/max, null rates — without any ingestion, schema definition, or server. For example you can profile a delivery instantly with `read_parquet`/`read_csv` and `summarize`. That immediacy makes it the fastest path from "a file arrived" to "I understand its shape."

Q: What does the DuckDB `SUMMARIZE` statement do?
A: `SUMMARIZE table_or_query` produces per-column descriptive statistics — type, min, max, approximate distinct count, null percentage, and quantiles — in a single command. For example `SUMMARIZE read_parquet('holdings.parquet');` profiles a delivery instantly. It is a fast data-quality reconnaissance tool for incoming fund files.

Q: How does out-of-core behaviour relate to the claim that DuckDB can handle "petabyte-scale lakehouse" data?
A: DuckDB can query very large lake datasets (Parquet/Iceberg/Delta) because it reads them in place with pushdown and only materializes the working set, spilling when needed — so total stored data can vastly exceed RAM and even disk, as long as each query's working set fits the out-of-core budget. It is querying, not loading, that scales. The single node still bounds per-query working-set size and concurrency, which is the real limit.

Q: A nightly DuckDB job and an analyst's notebook both need the same `.duckdb` file. How do you avoid lock conflicts?
A: Give the writer (the nightly job) exclusive read-write access during its window, and have the analyst open the file read-only (or query an exported Parquet copy) outside that window. Two read-write opens on one file always collide with `Could not set lock on file`. Scheduling plus read-only/copy access is the standard way to reconcile a batch writer with ad-hoc readers.

Q: Why does DuckDB favour large, set-based SQL over many small row-by-row operations?
A: Because its columnar-vectorized engine is optimized for processing large batches of values per operation, so a single set-based statement runs far faster than thousands of single-row statements that fight the vectorized model. Bulk inserts/updates and big analytical queries are its strength. Row-at-a-time OLTP-style chatter is exactly the workload it is not designed for — that is Postgres territory.

Q: How would you decide between `pg_duckdb` inside Postgres and a standalone DuckDB-over-lake setup for a fund analytics use case?
A: Choose `pg_duckdb` when the priority is querying live operational Postgres data with columnar acceleration and no second system to maintain, keeping analytics next to OLTP. Choose standalone DuckDB-over-lake when the source of truth is open table-format files in object storage and you want tool-independent, multi-reader access. The deciding factors are where the authoritative data lives and whether you need lake openness or co-location with OLTP.

Q: Summarize the single most important architectural lesson from "embedded analytical engines (DuckDB)."
A: That a single-node, in-process columnar engine with out-of-core spilling covers a large and useful envelope of analytical workloads cheaply and simply, and the architect's value is knowing precisely where that envelope ends — single-writer per file, one node's disk, low write concurrency — so you can say "no cluster" with evidence or escalate to MotherDuck, ClickHouse, or a lakehouse when the data size or concurrency genuinely demands it. Benchmarks against Postgres, a demonstrated larger-than-RAM query, and naming the single-writer constraint are how you prove you understand that boundary.

Q: On a fresh native-Ubuntu laptop, how do you install the DuckDB CLI and confirm the version from the shell?
A: Download the CLI with `curl https://install.duckdb.org | sh` (or grab the release zip), which drops a `duckdb` binary you can put on your `PATH`, then run `duckdb -version` to print the build. Inside an open CLI session you can instead run `SELECT version();` or `PRAGMA version;`. Pinning a known version matters for reproducible benchmarks, since storage-format and planner behaviour can shift between releases.

Q: How do you check which DuckDB extensions are installed and loaded in the current session?
A: Query the metadata table function `SELECT extension_name, installed, loaded FROM duckdb_extensions();`, which lists every known extension with boolean flags for whether it is downloaded locally and active in the session. This is the fast way to confirm `httpfs`, `parquet`, or `iceberg` is actually loaded before debugging a read. If `loaded` is false but the query needs it, run `LOAD name;` (autoloading covers most core extensions automatically).

Q: How do you inspect a single DuckDB setting and its current value at runtime?
A: Query `SELECT * FROM duckdb_settings() WHERE name = 'memory_limit';`, which returns the name, current value, description, and scope for that setting. This is how you confirm a `SET memory_limit='2GB';` actually took effect, or read the resolved default like the 80%-of-RAM memory limit. Using `duckdb_settings()` rather than guessing avoids the classic mistake of assuming a `SET` applied to the wrong connection.

Q: Within one DuckDB connection, how do you list all attached databases and the current default one?
A: Run the `duckdb_databases()` table function, e.g. `SELECT database_name, path, type FROM duckdb_databases();`, which shows every attached database (native, plus any attached Postgres/SQLite/MySQL) with its path and type. After `ATTACH 'ref.duckdb' AS ref;` it appears in that list, and you can switch the default catalog with `USE ref;`. This is the introspection step when a qualified name like `ref.schema.table` unexpectedly fails to resolve.

Q: A `read_csv` over a messy fund extract aborts on a bad row. How do you capture the rejects instead of failing?
A: Pass `store_rejects = true` to `read_csv`, which skips faulty lines and records them in two temporary tables named `reject_scans` (scanner parameters) and `reject_errors` (per-line errors with the column and reason). You can rename them via `rejects_scan` and `rejects_table`, and cap volume with `rejects_limit`. Inspecting `SELECT * FROM reject_errors;` tells you exactly which ISIN or date rows broke, which is far better than a blanket `ignore_errors` that silently drops them.

Q: What is the difference between `ignore_errors = true` and `store_rejects = true` in `read_csv`?
A: `ignore_errors` silently skips rows that fail to parse and continues, giving you a clean result but no record of what was dropped. `store_rejects` also continues past bad rows but additionally writes each failure to the `reject_errors` table with the line number, column, and error type, so you can audit and fix the source. For a regulated extract you want `store_rejects` because "we dropped N rows for reason X" is an auditable statement, whereas silent ignoring is not.

Q: How would you force a CSV column like `isin` to be read as text rather than letting auto-detection guess a numeric type?
A: Supply explicit column types, e.g. `read_csv('f.csv', columns = {'isin': 'VARCHAR', 'nav': 'DOUBLE'})`, or use the `types`/`dtypes` override and set `auto_detect = false` when you want full control. This prevents the sniffer from coercing a leading-zero or all-digit ISIN into a `BIGINT` and corrupting it. Explicit typing on identifier columns is a standard defensive habit for fund reference data.

Q: How do you make DuckDB sample more of a large CSV during type auto-detection?
A: Increase the `sample_size` parameter, e.g. `read_csv('f.csv', sample_size = -1)` to scan the whole file, or a large integer to sample more rows before deciding types. The default samples only the first chunk of rows, so a column that looks integer early but contains text or nulls later can be mis-typed. Bumping `sample_size` trades detection time for correctness on heterogeneous extracts.

Q: How do you read a CSV whose delimiter is a semicolon and whose decimal separator is a comma (a common European export)?
A: Pass the field separator and decimal explicitly, e.g. `read_csv('f.csv', delim = ';', decimal_separator = ',')`, so a value like `1.234,56` parses correctly to `1234.56`. European fund and bank exports frequently use `;` plus comma-decimals, and trusting auto-detection on them often yields wrong numbers. Being explicit about `delim` and `decimal_separator` avoids silently mis-scaled NAVs.

Q: How do you read a fixed set of CSV files but tag each row with its source filename in DuckDB?
A: Set `filename = true` in `read_csv` (or `read_parquet`), which adds a `filename` column carrying the path each row came from. For example `SELECT filename, count(*) FROM read_parquet('nav/*.parquet', filename = true) GROUP BY filename;` shows per-file row counts. This is invaluable for reconciling which delivery a given record originated from during an audit.

Q: How do you turn a glob scan of monthly Parquet exports into a profiling pass over each file's row count and distinct ISINs?
A: Combine the `filename` column with aggregation: `SELECT filename, count(*) AS rows, count(DISTINCT isin) AS isins FROM read_parquet('emt/*.parquet', filename = true) GROUP BY filename ORDER BY filename;`. DuckDB reads all matching files as one logical table and the `filename` pseudo-column lets you break results down per delivery. This is a one-liner data-quality reconnaissance over a whole month of feeds.

Q: How do you specify explicit AWS S3 endpoint, region, and URL style for a non-AWS S3-compatible store like MinIO in DuckDB?
A: Create a secret with the endpoint and style, e.g. `CREATE SECRET (TYPE s3, KEY_ID '…', SECRET '…', REGION 'eu-central-1', ENDPOINT 's3.example.lu:9000', URL_STYLE 'path', USE_SSL true);`. The `ENDPOINT` and `URL_STYLE 'path'` keys are what make `httpfs` talk to MinIO or another S3-compatible object store rather than AWS. Path-style URLs are usually required for self-hosted stores that do not support virtual-hosted bucket subdomains.

Q: How do you have DuckDB pick up AWS credentials from the environment or an IAM role instead of hardcoding keys?
A: Create a secret with the credential-chain provider, e.g. `CREATE SECRET (TYPE s3, PROVIDER credential_chain);`, which resolves credentials from the standard chain — environment variables, shared config files, and instance/role metadata. This avoids writing long-lived `KEY_ID`/`SECRET` values into SQL or scripts. On an EC2 box with an attached IAM role it lets DuckDB read `s3://` paths with no static keys at all.

Q: Where does DuckDB persist a secret created with plain `CREATE SECRET`, and how do you make it survive restarts?
A: A plain `CREATE SECRET` is temporary and lives only in the current session. To persist it, use `CREATE PERSISTENT SECRET …`, which stores it (by default under `~/.duckdb/stored_secrets`) so future sessions reuse it. You can list them with `FROM duckdb_secrets();` and drop with `DROP SECRET name;`; persistent secrets are convenient but mean credentials sit on disk, which matters for a regulated host.

Q: How do you query a Parquet file that lives behind a plain HTTPS URL with no credentials in DuckDB?
A: With `httpfs` loaded (it autoloads), just scan it directly: `SELECT * FROM read_parquet('https://host/path/holdings.parquet');`. DuckDB issues HTTP range requests so it fetches only the byte ranges projection and predicate pushdown require, rather than the whole file. No secret is needed for anonymous HTTPS; secrets are only for authenticated S3/GCS/Azure access.

Q: How do you read an Iceberg table's older snapshot for an as-of audit rather than its current state?
A: Use `iceberg_scan` with a snapshot selector — `iceberg_scan('…', snapshot_from_timestamp => TIMESTAMP '2026-03-31 23:59:59')` (or `snapshot_from_id => …`) — to read exactly the committed state at that point. Iceberg's metadata log records every snapshot, which is what enables this time travel. For an audited NAV history this lets you reproduce precisely what the table showed at quarter-end.

Q: How do you list the snapshot history of an Iceberg table from DuckDB?
A: Call `iceberg_snapshots('path_or_table')`, which returns one row per snapshot with its id, timestamp, and manifest pointer. For example `SELECT * FROM iceberg_snapshots('s3://bucket/nav_iceberg');` shows the commit history you can then time-travel to. Pairing this with `iceberg_metadata(...)` lets you see which data files belong to each snapshot, useful when proving lineage to an auditor.

Q: A `delta_scan` returns fewer rows than the underlying Parquet files contain. Why is that actually correct?
A: Because `delta_scan` honours the Delta transaction log, which records that some Parquet files were tombstoned (logically deleted) by later commits, so it reads only the files that are live in the current table version. A raw `read_parquet` glob over the same directory would wrongly include those tombstoned files. The discrepancy is the table format doing its job: giving you the committed state, not every file on disk.

Q: How do you write a Hive-partitioned Parquet dataset from DuckDB and avoid overwrite errors on re-run?
A: Use `COPY (SELECT …) TO 'out' (FORMAT parquet, PARTITION_BY (fund_id, snapshot_date));`, and add `OVERWRITE_OR_IGNORE true` (or `OVERWRITE true` in newer versions) so a re-run replaces existing partition directories instead of failing because the target exists. Without an overwrite flag DuckDB refuses to write into a non-empty partitioned target to protect existing data. This pattern is how a nightly job idempotently republishes only the partitions it rebuilt.

Q: How do you control Parquet file size or row-group size when DuckDB writes output?
A: On the `COPY ... TO` you can set `ROW_GROUP_SIZE` (rows per row group) and, in recent versions, `FILE_SIZE_BYTES` to target a maximum file size before rolling to a new file. Larger row groups improve scan throughput but coarsen predicate-pushdown skipping; smaller ones prune better but add metadata overhead. Tuning these so each row group aligns with your common filter (e.g. one date per group) is what makes downstream reads fast.

Q: Why might sorting fund data by your common filter column before writing Parquet dramatically speed later DuckDB reads?
A: Because Parquet stores per-row-group min/max statistics, and if rows are sorted by, say, `snapshot_date`, each row group covers a narrow date range so a `WHERE snapshot_date = '2026-06-13'` skips almost every other group via predicate pushdown. Unsorted data spreads every value across all row groups, so the min/max ranges overlap and almost nothing can be pruned. Writing `ORDER BY snapshot_date` before `COPY ... TO` is a cheap one-time cost for big repeated read savings.

Q: How do you join a Parquet holdings file directly against a Postgres reference table in a single DuckDB query?
A: Load and attach Postgres — `INSTALL postgres; LOAD postgres; ATTACH 'host=… dbname=ref' AS pg (TYPE postgres);` — then write one statement joining `read_parquet('holdings.parquet') h JOIN pg.public.securities s ON h.isin = s.isin`. DuckDB scans the Parquet in place and pulls the Postgres table over the connection, executing the join in-process. This federates a lake file and live operational reference data with no intermediate ETL.

Q: How can you copy a Postgres table into a local DuckDB native table for fast repeated analytics?
A: After attaching Postgres, run `CREATE TABLE local_sec AS SELECT * FROM pg.public.securities;` (or `COPY FROM DATABASE pg TO memory;` for a whole database), materializing the data into DuckDB's columnar storage. Subsequent queries then hit fast local columnar data instead of round-tripping to Postgres each time. This is the pattern when you want to take an operational snapshot and analyse it offline without hammering the OLTP server.

Q: How do you query a pandas DataFrame named `df` directly in DuckDB without an explicit registration step?
A: In the Python API, DuckDB performs "replacement scans" so `duckdb.sql('SELECT * FROM df')` finds the local variable `df` by name and queries it in place over Arrow, zero-copy. You can also be explicit with `con.register('df', df)`. The replacement-scan convenience is a direct payoff of the in-process model — the DataFrame and the engine share one address space.

Q: What does `.df()` versus `.arrow()` versus `.pl()` return on a DuckDB Python result, and why choose one?
A: `.df()` materializes the result as a pandas DataFrame, `.arrow()` returns an Arrow table, and `.pl()` returns a Polars DataFrame. You pick based on the downstream consumer: `.arrow()` and `.pl()` stay closest to DuckDB's native Arrow layout and avoid a pandas conversion copy, while `.df()` is convenient when the rest of the code is pandas. For large results the Arrow/Polars paths are usually faster and lighter on memory.

Q: How do you stream a huge DuckDB result in Python rather than materializing it all into one DataFrame?
A: Use `con.execute(sql).fetch_record_batch()` (or `fetchmany`) to pull the result in Arrow record batches incrementally, processing each batch and releasing it. This keeps peak memory bounded when the result set is larger than RAM, instead of `.df()` building one giant frame. It is the right approach when exporting a multi-gigabyte mart row-group by row-group to a downstream sink.

Q: How do you read a setting only for one query rather than the whole session in DuckDB?
A: Wrap the statement in a transaction with a `SET LOCAL`, e.g. `BEGIN; SET LOCAL threads = 2; SELECT …; COMMIT;`, so the change reverts at transaction end instead of persisting like a plain `SET`. This lets you throttle one heavy query without permanently changing the connection's parallelism. It is handy in a shared notebook where you do not want a one-off tweak leaking into later cells.

Q: How do you enable and read DuckDB's per-operator profiling for a slow query?
A: Run `PRAGMA enable_profiling = 'json';` (or `'query_tree'`), optionally `PRAGMA profiling_output = '/tmp/prof.json';`, then execute the query, or simply prefix it with `EXPLAIN ANALYZE`. The output gives per-operator timing, cardinalities, and whether operators spilled, so you can find the join or aggregation dominating the runtime. This is the concrete tool the Phase-2 "Do" benchmark relies on to explain a result rather than guess.

Q: In an `EXPLAIN` plan, what tells you projection and predicate pushdown into Parquet are actually happening?
A: The `PARQUET_SCAN` (or `READ_PARQUET`) operator node lists the `Projections` (only the columns you selected) and `Filters` pushed into the scan, and `EXPLAIN ANALYZE` shows far fewer rows scanned than the file contains when row groups were skipped. If you instead see all columns projected and the full row count read, pushdown is not engaging — often because of a `SELECT *` or a non-sargable filter. Confirming this in the plan is how you prove a remote scan is reading minimal bytes.

Q: A DuckDB aggregation is unexpectedly slow and memory-heavy on ordered output. What setting often helps?
A: Disable insertion-order preservation with `SET preserve_insertion_order = false;`, because preserving the original row order constrains the plan and raises memory pressure for large scans and writes. If you genuinely need ordering, do it with an explicit `ORDER BY` at the end instead of paying the implicit preservation cost throughout. This is a documented go-to for both faster bulk reads and lower-memory Parquet writes.

Q: Which knobs would you turn first to make a larger-than-RAM DuckDB query complete on a memory-constrained Ubuntu laptop?
A: Ensure `temp_directory` points at a roomy fast SSD so spilling has somewhere to go, set a realistic `memory_limit` (not unlimited), lower `threads` to cut per-thread memory overhead, and set `preserve_insertion_order = false`. Together these let the blocking operators spill cleanly instead of OOMing, at the cost of some speed. This bundle is the standard out-of-core tuning recipe for a constrained single node.

Q: Why can raising `threads` sometimes make an out-of-core query OOM rather than run faster?
A: Because each parallel thread holds its own working buffers and partial hash tables, so more threads multiply concurrent memory use against a fixed `memory_limit`, pushing operators over the edge before they spill cleanly. On a tight-memory box fewer threads can be the difference between completing and failing. The trade-off is lower CPU parallelism for lower peak memory — exactly the tuning out-of-core workloads need.

Q: How do you cap how much DuckDB may spill so a runaway query cannot fill a shared disk?
A: Set `SET max_temp_directory_size = '50GB';` (or `PRAGMA max_temp_directory_size`), which bounds total spill bytes; a query needing more aborts with a temp-space error instead of filling the volume. By default this is 90% of free space on the temp disk, which on a shared host can still be too generous. Capping it is a guardrail that converts a disk-filling outage into a single failed query.

Q: A spilling query aborts with a temp-directory/disk-space error even though the volume looked fine earlier. What two limits do you check?
A: Check free space on the `temp_directory` volume (the spill files may have already consumed it during the run) and the `max_temp_directory_size` setting that caps spill bytes — either can trigger the abort. Run `SELECT * FROM duckdb_settings() WHERE name IN ('temp_directory','max_temp_directory_size');` and `df -h` on the temp mount. Pointing `temp_directory` at a larger SSD or raising the cap (within reason) resolves it.

Q: How do you create a lightweight, snapshot-like copy of a DuckDB database for safe ad-hoc analysis?
A: When no writer is active, copy the single `.duckdb` file at the OS level (e.g. `cp fund.duckdb fund_snapshot.duckdb`) and open the copy read-only, or use `EXPORT DATABASE 'dir' (FORMAT parquet);` to dump to portable files. Because the whole database is one file, the copy is atomic and self-contained, giving a reproducible artifact. This sidesteps the single-writer lock entirely for an analyst who only needs to read.

Q: What do `EXPORT DATABASE` and `IMPORT DATABASE` do, and when are they useful?
A: `EXPORT DATABASE 'dir';` writes the whole database out as SQL schema plus data files (Parquet or CSV) in a directory, and `IMPORT DATABASE 'dir';` reloads it into a database. They are useful for migrating between DuckDB storage-format versions, archiving a portable open-format copy for retention, or moving a database to another machine. Unlike copying the native file, the export is engine-version-independent and human-inspectable.

Q: You upgraded the DuckDB CLI and now an old `.duckdb` file fails to open with a storage version error. What happened and how do you recover?
A: DuckDB's native storage format is not guaranteed forward/backward compatible across all releases, so a file written by a newer version may refuse to open in an older one (or vice versa). Recover by opening it with the version that wrote it and `EXPORT DATABASE` to portable files, then `IMPORT DATABASE` under the target version, or simply standardize on one DuckDB version. This is why pinning a version and preferring open Parquet for long-lived data matters in a regulated archive.

Q: How do you make a DuckDB session use only signed, official extensions and refuse unsigned community ones?
A: Keep `allow_unsigned_extensions` at its default `false`, which permits only signed extensions from the official repository and blocks loading unsigned binaries. A community extension that is unsigned then fails to load unless you explicitly enable `SET allow_unsigned_extensions = true;`. For a regulated shop this default is the governance control that limits the engine to vetted core extensions.

Q: Beyond `httpfs`, how do you point DuckDB at a specific extension repository or a custom mirror?
A: Use `SET custom_extension_repository = 'https://my-mirror/…';` (or the `INSTALL name FROM 'repo'` form) so installs pull from an approved internal mirror rather than the public endpoint. This lets a locked-down environment with no general internet egress still install vetted extensions from a controlled location. It pairs with disallowing unsigned extensions to give a fully governed extension supply chain.

Q: How would you sandbox a DuckDB instance so untrusted SQL cannot read arbitrary local files or the network?
A: Enable `SET enable_external_access = false;` (and keep `allow_unsigned_extensions = false`), which blocks file-system and network reads/writes outside the attached database, so functions like `read_parquet` on arbitrary paths or `httpfs` URLs are refused. This is the lockdown for running user-supplied SQL against a fixed dataset. Combined with read-only mode it gives a tightly constrained query surface.

Q: How do you open a DuckDB database read-only from the Python API, and when must you?
A: Pass `read_only=True`, e.g. `duckdb.connect('fund.duckdb', read_only=True)` (or set `access_mode='READ_ONLY'` in the config). You must do this when another process already holds the read-write lock and you only need to read, since that is the only way multiple processes can attach to one file concurrently. It is the standard fix for an analyst notebook that must coexist with a running batch writer.

Q: A notebook kernel left a DuckDB connection open and now your script gets `Could not set lock on file`. What is the quickest clean fix?
A: Close or restart the holding process — shut down the stray notebook kernel (or call `con.close()` on the lingering connection) so the read-write lock is released, then reopen. The lock is held by a live process, not stale on disk, so killing or closing that process frees it immediately. If you only need to read, opening read-only avoids the conflict without touching the other process.

Q: How do you run a quick one-off DuckDB query against a file straight from the Ubuntu shell without writing a script?
A: Use the CLI in non-interactive mode, e.g. `duckdb -c "SELECT count(*) FROM read_parquet('holdings.parquet');"`, or pipe SQL via `duckdb < query.sql`. The `-c` flag executes the statement and exits, making DuckDB usable in shell pipelines and cron jobs. Adding `-json` or `-csv` controls the output format for downstream tooling.

Q: How do you change the DuckDB CLI's output to CSV or markdown for piping or pasting into a report?
A: Use a dot-command in the CLI, e.g. `.mode csv` (or `.mode markdown`, `.mode json`, `.mode line`), optionally with `.output results.csv` to redirect to a file. From the shell the flags `-csv`, `-json`, or `-markdown` set the same modes for a `-c` invocation. This is how you turn a quick reconciliation query into a CSV a colleague can open or a markdown table for documentation.

Q: How would you use DuckDB to detect duplicate ISINs within a single fund in a holdings Parquet file?
A: Run a grouping query like `SELECT fund_id, isin, count(*) AS n FROM read_parquet('holdings.parquet') GROUP BY fund_id, isin HAVING count(*) > 1;`, which surfaces every (fund, ISIN) pair appearing more than once. DuckDB's columnar aggregation makes this fast even on a large file, with no load step. It is a typical one-line data-quality check before a holdings file is accepted into the pipeline.

Q: How do you validate that every ISIN in a delivery is a syntactically plausible 12-character code using DuckDB SQL?
A: Filter for violations, e.g. `SELECT isin FROM read_parquet('emt.parquet') WHERE isin IS NULL OR length(isin) <> 12 OR isin !~ '^[A-Z]{2}[A-Z0-9]{9}[0-9]$';`, using a regex matching the two-letter country prefix, nine alphanumerics, and the check digit. Any rows returned are malformed identifiers to reject or investigate. DuckDB's `regexp` support lets such structural contracts run directly over the file before ingestion.

Q: How can DuckDB reconcile two NAV Parquet exports and report only the rows that differ?
A: Use a set operation, e.g. `(SELECT * FROM read_parquet('nav_a.parquet') EXCEPT SELECT * FROM read_parquet('nav_b.parquet')) UNION ALL (SELECT * FROM read_parquet('nav_b.parquet') EXCEPT SELECT * FROM read_parquet('nav_a.parquet'));`, which returns rows present in one file but not the other. For keyed comparison you instead `FULL JOIN` on `(fund_id, isin, date)` and filter where values diverge. This is a fast, load-free diff for catching discrepancies between two pipeline outputs.

Q: When benchmarking DuckDB-over-Parquet against Postgres, why is it unfair to count only DuckDB's query time?
A: Because DuckDB queries the Parquet in place with no ingest, whereas Postgres must first load and index the data — so an honest comparison includes Postgres's load/index time (or amortizes it across many queries) and notes that DuckDB pays that cost only if it materializes. Ignoring setup cost flatters whichever engine you exclude it from. The fair report states cold-query, warm-query, and one-time setup numbers separately so the reader sees the true trade-off.

Q: How do you check whether `httpfs` is even available before debugging a failing `s3://` read in DuckDB?
A: Query `SELECT installed, loaded FROM duckdb_extensions() WHERE extension_name = 'httpfs';` to see if it is downloaded and active in the session. If `installed` is false you run `INSTALL httpfs;` (which needs network access, a common failure on a locked-down host), and if only `loaded` is false you run `LOAD httpfs;`. Confirming this first avoids chasing a credentials or path bug when the real problem is the extension never loaded.

Q: How would you profile an unfamiliar incoming fund Parquet file's full column shape in one DuckDB statement?
A: Run `SUMMARIZE SELECT * FROM read_parquet('delivery.parquet');`, which returns per-column type, min, max, approximate distinct count, null percentage, and quantiles in a single pass. For a freshly arrived EMT or holdings file this instantly reveals an all-null column, an unexpected type, or a date range that looks wrong before you trust the delivery. It is the fastest "what did I just receive?" reconnaissance DuckDB offers, with no schema definition or load step.
