# Phase 5 · 12.1.1 Backup & recovery (cloud-native) — 150 self-test questions

Q: What single question is the litmus test that this lesson says a fund platform must be able to answer about its backups?
A: It must answer "when did we last successfully *restore*?" with a concrete date, not a shrug. The point is that a backup that has never been test-restored is an unverified assumption, not a recovery capability. Auditors in a regulated fund context actually exercise this: they ask for the date and evidence of the last successful restore drill, so the date is the real deliverable.

Q: Why does this lesson stress that lakehouse-era backup is "mostly not file copies"?
A: Modern data platforms store state across object stores, table-format metadata, and external catalogs, so naively copying parquet files misses the catalog pointer and snapshot history that make the data queryable. A backup must capture the system that makes the bytes meaningful, not just the bytes. For example, copying an Iceberg table's data files without its catalog metadata leaves you with files you cannot mount as a table.

Q: In disaster-recovery terms, what does the lesson mean by calling backups "the bottom rung of the DR ladder"?
A: Backups are the most basic, foundational layer of recoverability that everything else builds on; you cannot have meaningful DR, HA, or BCDR without recoverable backups underneath. It is "bottom rung" because it is necessary-but-not-sufficient: replication and failover protect against infrastructure loss, but only backups protect against logical corruption, bad migrations, and ransomware. It is also the rung auditors most reliably test.

Q: What is an Azure Recovery Services vault?
A: It is one of the two Azure Backup vault types, an online storage entity that holds backup copies, recovery points, and backup policies for workloads such as Azure VMs, SQL Server in VMs, SAP HANA in VMs, and Azure Files. It centralizes management, monitoring, and recovery-point retention for those data sources. It predates the newer Backup vault and supports a different (overlapping) set of workloads.

Q: What is an Azure Backup vault, and how does it differ from a Recovery Services vault?
A: A Backup vault is the newer vault type used for data sources like Azure Blobs, Azure Disks, Azure Database for PostgreSQL flexible server, AKS, and Azure Data Lake Storage. It serves a different (and partly newer) set of workloads than the Recovery Services vault, which covers Azure VMs, SQL/SAP HANA in VMs, and Azure Files. Choosing the right vault type depends on which data source you are protecting, and some estates run both.

Q: What is a backup policy in Azure Backup?
A: A backup policy is the configuration object, associated with backup jobs, that specifies the schedule (when backups run) and the retention (how long recovery points are kept). It decouples "what cadence" and "how long" from the workload itself so the same discipline can be applied across many protected items. Editing a policy's retention or schedule is exactly the kind of operation that immutable vaults constrain.

Q: What is a "job" in Azure Backup terminology?
A: A job represents the activity of either backing up or restoring data; backup jobs copy data from the source into the vault, and restore jobs recover data from the vault to a target. Each job has a unique identifier and status tracking so you can monitor progress and troubleshoot failures. The Azure Backup SLA is even framed in terms of jobs: you must retry a failed backup or restore job at least once every 30 minutes to be covered.

Q: What is soft delete in Azure Backup, and what is its default retention?
A: Soft delete is a recycle-bin-style safety net that lets you recover deleted backup items and vaults during a configurable retention window; by default that window is 14 days. It protects against accidental or malicious deletion by keeping the data recoverable for a period after the delete request. Without it, a single delete operation could immediately and permanently destroy recovery points.

Q: A backup item was deleted in Azure Backup 5 days ago and the team panics — what should you check first?
A: Check soft delete: by default deleted backup items remain recoverable for 14 days, so a 5-day-old deletion is almost certainly still in the soft-deleted state and can be undeleted. Confirm soft delete was enabled (it is on by default) and that its retention has not been shortened. This is exactly the scenario soft delete exists to cover, so do not assume the data is gone.

Q: Can Azure Backup soft delete be disabled, and why does enhanced soft delete matter?
A: Standard soft delete can be turned off, which is risky because it removes the recovery window; enhanced soft delete improves on this by letting you make soft delete always-on so it cannot be disabled by an attacker who gains vault access. Making it irreversible matters in a ransomware scenario, where attackers specifically try to disable safety nets before deleting backups. The trade-off is that always-on means you genuinely cannot turn it off, so it must be a deliberate decision.

Q: What is an immutable vault in Azure Backup?
A: An immutable vault blocks any management operation that could lead to loss of recovery points, such as deleting recovery points before expiry or reducing a policy's retention. It is designed so that even a malicious actor with vault access cannot destroy the backups. It can be applied to both Recovery Services vaults and Backup vaults.

Q: What are the three states of the Azure Backup immutable-vault setting?
A: The three states are `Disabled` (no operations blocked), `Enabled` (immutability on but the setting can still be disabled), and `Enabled and locked` (immutability with WORM storage, and the setting can no longer be disabled). Moving to `Enabled and locked` is irreversible, so it is the strongest protection but must be a well-informed decision. The intermediate `Enabled` state lets you evaluate impact before committing.

Q: Why is "Enabled and locked" the strongest immutable-vault state, and what is its catch?
A: It enables WORM (write once, read many) storage and makes the immutability setting irreversible, so an attacker cannot disable immutability and then delete backups. The catch is precisely that irreversibility: once locked, you cannot turn it off, even legitimately, so a misconfiguration or over-long retention becomes permanent. Microsoft recommends evaluating impact in the reversible `Enabled` state first, then locking.

Q: Name three operations an immutable Recovery Services vault blocks.
A: It blocks deleting recovery points before their expiry date (stop-protection-with-delete is disallowed), reducing the retention period in a backup policy, and replacing a policy with one that has lower retention. In other words, anything that shortens how long data is kept or removes it early is refused. You can still increase retention or change the schedule, and you can stop protection while retaining existing data until expiry.

Q: Does Azure Backup immutability apply to operational backups of blobs, files, and disks?
A: No — immutability does not apply to operational backups for resources like blobs, files, and disks; it applies to vaulted backup data. Operational backups keep snapshots in the source's own resource group rather than copying into the vault, so the vault-level immutability control does not govern them. This is a common gotcha: enabling immutable vault does not automatically harden operational-tier blob backups.

Q: What is WORM storage and how does it relate to immutable vaults?
A: WORM stands for write once, read many: data can be written and read but not modified or deleted until any retention expires. The "Enabled and locked" immutable-vault state uses WORM storage so backups physically cannot be altered or deleted by malicious actors, even if they try to disable immutability. WORM is the storage-layer enforcement behind the locked immutability setting.

Q: How does Azure Backup help protect against ransomware specifically?
A: It combines storage isolation (backups live in a vault separate from production workloads), soft delete (a recovery window for deleted items), and immutable vaults with WORM (which block early deletion and retention reduction). Together these mean that even if production is encrypted and an attacker reaches the vault, they cannot quickly delete the backups. The recovery story depends on these being configured and, ideally, the immutability being locked.

Q: Why does keeping backups "isolated from production workloads" matter for ransomware resilience?
A: If backups shared the same access plane and credentials as production, ransomware that compromises production could equally encrypt or delete the backups, defeating recovery. Storing backups in a vault under a separate control plane means a production compromise does not automatically grant the ability to destroy the recovery copies. Isolation is what turns a backup into an independent, attacker-resistant safety net.

Q: What does the lesson mean by a "backup policy matrix," and what columns should it have?
A: It is a table that, for every store on the platform, records what is backed up, the frequency, the retention, whether it is immutable, and the restore-test cadence. It forces an explicit, per-store decision rather than an implicit assumption that "something is backing this up." The matrix is one of the lesson's two "done-when" deliverables and is what you would hand an auditor.

Q: For an Azure VM, which vault type does Azure Backup use?
A: Azure VMs are protected through a Recovery Services vault, using backup extensions for full-VM backups (or the MARS agent for file/folder/system-state). The Recovery Services vault stores the recovery points and applies the backup policy. This is distinct from the Backup vault used for newer data sources like Blobs and PostgreSQL flexible server.

Q: How long can Azure Backup retain Azure Database for PostgreSQL backups?
A: Azure Backup can back up Azure Database for PostgreSQL and retain those backups for up to 10 years. This long-term retention matters for regulated fund administration, where records such as NAV history may need to be kept for many years to satisfy auditors and regulators. It is a vaulted-backup capability distinct from the database service's own short PITR window.

Q: What is point-in-time restore (PITR) for a managed database?
A: PITR lets you restore a database to any moment within a retention window by replaying transaction logs on top of a base backup, so you can recover to, say, "09:42 just before the bad batch ran." It is the mechanism that turns continuous log shipping into fine-grained recovery. Its usefulness is bounded by the PITR retention window — you cannot restore to a time older than the window allows.

Q: What is the key limitation of a database PITR window?
A: PITR can only recover to points inside its retention window; once a moment falls outside that window, it is no longer reachable by PITR and you must rely on longer-term vaulted backups instead. So a 7-day or 35-day PITR window does not protect against a corruption discovered weeks later. This is exactly why PITR is paired with longer retention backups in a complete policy.

Q: A NAV-calculation error corrupted a fund's positions table 40 days ago, but PITR retention is 35 days — can PITR recover it?
A: No — the corruption point is outside the 35-day PITR window, so PITR cannot reach it; you must restore from a longer-term vaulted backup (Azure Backup can retain PostgreSQL backups up to 10 years). This illustrates why a complete policy pairs short PITR with long retention. The fix is to restore the appropriate long-term recovery point and reconstruct, then investigate why the error went undetected for 40 days.

Q: Why is PITR alone not a substitute for vaulted long-term backups?
A: PITR's retention window is short (days to a few weeks), so it protects against recent operational mistakes but not against slow-to-discover corruption, regulatory retention needs, or ransomware that sits dormant. Vaulted long-term backups (up to 10 years for PostgreSQL) cover the long tail PITR cannot. A robust design uses both: PITR for fine-grained recent recovery and vaulted backups for the years auditors require.

Q: What is object-store versioning and how does it act as a backup mechanism?
A: Versioning keeps multiple immutable iterations of an object so that overwrites and deletes do not destroy prior content; each version is retained and addressable, letting you roll back to a previous state. It acts as a lightweight backup against accidental overwrite or delete because the old bytes are still there under an older version ID. It is per-bucket and does not, by itself, protect the bucket's structure or the catalog metadata of a table format.

Q: What is the exact MinIO `mc` command to enable versioning on a bucket?
A: The command is `mc version enable ALIAS/BUCKET`, where `ALIAS` is your configured MinIO server alias and `BUCKET` is the target bucket. Versioning is a per-bucket setting, so each bucket maintains its own versioning state independently. Enabling it is also a prerequisite for configuring object locking and retention rules.

Q: In MinIO/S3 versioning, what is a delete marker?
A: When you delete a versioned object without specifying a version ID, MinIO does not erase the data; instead it creates a 0-byte `DeleteMarker` as the latest version, hiding the object from normal listings while the prior versions remain retrievable by version ID. This is a "soft" delete: the object appears gone but is recoverable. To truly remove data you must delete the specific version ID(s).

Q: How do you permanently and irreversibly delete a specific object version in MinIO?
A: You run the delete and specify the version ID, for example `mc rm --version-id=UUID ALIAS/BUCKET/OBJECT`; this removes that exact version rather than adding a delete marker. Without `--version-id`, a delete on a versioned bucket only creates a delete marker and leaves the data recoverable. To wipe an object across all its versions you can use `mc rm --versions ALIAS/BUCKET/OBJECT`.

Q: How do you list all stored versions of an object in MinIO?
A: Use `mc ls --versions ALIAS/BUCKET/OBJECT`, which lists every version (and any delete markers) along with their version IDs. You need the version IDs to retrieve or permanently delete a specific historical version. This is the first step in any restore-from-versioning drill.

Q: How do you restore a previous version of an object in MinIO?
A: Copy the desired version out by its version ID, for example `mc cp --version-id=UUID ALIAS/BUCKET/OBJECT LOCAL`, then write it back as the current object if needed. The version ID comes from `mc ls --versions`. Doing this end-to-end and timing it is exactly the "run one real restore and time it" exercise the lesson prescribes.

Q: What kind of identifier does MinIO assign to each object version?
A: Each versioned object gets a unique, immutable 128-bit UUIDv4 as its version ID; objects that existed before versioning was enabled carry a `null` version ID. These IDs are how you address a specific historical version for retrieval or permanent deletion. Because they are immutable, a version ID always points to the same bytes.

Q: What is object-store lifecycle management, and how does it relate to backup?
A: Lifecycle rules automatically transition objects between storage tiers or expire (delete) them — including expiring noncurrent versions — after a defined age. As a backup control they cap cost and retention, ensuring old versions do not accumulate forever. But a misconfigured lifecycle rule can also become a foot-gun, silently expiring the very versions you assumed were your backup.

Q: What are the limits of object-store versioning + lifecycle as a backup strategy?
A: Versioning protects against overwrite and delete within the same bucket and account, but it does not protect against bucket deletion, account compromise, region loss, or a credential leak that lets an attacker purge versions, and it does not capture table-format catalog metadata. Lifecycle rules can also delete versions if misconfigured. So versioning is a useful first line of defense, not a complete, isolated, immutable backup.

Q: Why is bucket versioning a prerequisite for object locking and WORM/retention in MinIO?
A: Object locking and retention rules act on specific object versions, so the system must be able to identify and preserve individual versions — which only exists once versioning is enabled. Without versioning there is a single mutable object with no per-version handle to lock. Hence you must enable versioning first, then layer locking/retention to get true WORM behavior.

Q: How does S3/MinIO object locking add protection beyond plain versioning?
A: Object locking enforces WORM at the storage layer using retention modes and legal holds, so a locked version cannot be deleted or overwritten until its retention expires (or a governance override is used) — even by someone with delete permissions. Plain versioning only soft-deletes via delete markers and still allows version-ID deletion. Locking is what turns versioning from "recoverable if nobody purges it" into "genuinely immutable for the retention period."

Q: An Iceberg table has snapshots going back 30 days — does that mean you have 30 days of backups?
A: No. Snapshots are time-travel points within the live table, governed by the same catalog and the same storage; they protect against logical changes you can roll back, but they are not an isolated, immutable copy. If the catalog is lost or the data files are deleted/corrupted, the snapshots go with them. Snapshots complement but do not replace real, isolated backups.

Q: Why are Iceberg snapshots not a backup when the catalog dies?
A: Iceberg's catalog holds the pointer to the current metadata file, which in turn references the table's snapshots; if the catalog is lost, you lose the authoritative reference to the table's current state even though data files may still sit in object storage. The snapshots themselves live inside that same managed structure, so a catalog failure orphans them. A real backup must therefore also protect the catalog, not just the data files.

Q: What does the Iceberg `expire_snapshots` procedure do?
A: It removes old table snapshots and the data/metadata files that only those expired snapshots referenced, reclaiming storage and limiting how far back you can time-travel. Key arguments are `older_than` (drop snapshots created before a timestamp) and `retain_last` (always keep at least N recent snapshots). Because it permanently deletes files, an over-aggressive `expire_snapshots` can destroy the very history someone assumed was a backup.

Q: What is the role of `older_than` and `retain_last` in Iceberg snapshot expiration?
A: `older_than` sets the cutoff timestamp so snapshots created before it become eligible for expiry, while `retain_last` guarantees a minimum number of recent snapshots survive regardless of age, preventing you from accidentally expiring everything. Together they bound the time-travel window. Misjudging them is a classic way to lose rollback history, which is why snapshot retention must be reasoned about separately from backup.

Q: What does Iceberg's `remove_orphan_files` procedure do, and why is it dangerous to misuse?
A: It scans storage and deletes data files that are no longer referenced by any snapshot or metadata — typically leftovers from failed or aborted writes. It is dangerous because if it is pointed at the wrong location or run with a too-recent cutoff, it can delete files that are actually still in use, corrupting the table. For this reason it is usually run conservatively with a safe `older_than` margin.

Q: In Iceberg, what does the catalog actually store, and why is that the crux of backup?
A: The catalog stores the pointer to the table's current metadata file; that metadata file then references the manifest lists, manifests, snapshots, and data files. The crux is that this pointer is the single source of truth for "what the table looks like right now," so losing it means you cannot resolve the table even if every data file survives. A complete backup must capture or be able to reconstruct that catalog pointer.

Q: If you back up only an Iceberg table's parquet data files but not its catalog, what have you actually backed up?
A: You have backed up bytes you cannot easily mount as a table, because without the catalog pointer and metadata you have lost the structure, schema evolution, partition spec, and snapshot lineage needed to read them as an Iceberg table. You would have to reconstruct or re-register the metadata to make the files queryable again. This is the lesson's core warning that snapshots/data-files alone are not a backup.

Q: What additional protection does a table-format catalog need beyond protecting the data files?
A: The catalog (and the metadata it points to) must be backed up or reconstructable independently — for example by backing up the catalog database (the metadata pointer store) and the metadata JSON, so the table can be re-registered after a disaster. Protecting only object-store data files leaves a recoverable blob but an unmountable table. The lesson's "Do" step explicitly asks you to note, per store, what extra protection the catalog needs.

Q: Snapshots, replication, and backup all protect data — what distinct failure does each address?
A: Replication/HA protects against infrastructure loss (a node, zone, or region going down); snapshots protect against recent logical changes you want to roll back within the live system; backups protect against logical corruption, accidental/malicious deletion, ransomware, and long-term retention needs by keeping isolated, ideally immutable copies. They are complementary, not interchangeable. A platform needs all three, and conflating snapshots with backups is the trap this lesson targets.

Q: What is the difference between Azure Backup's GRS and CRR?
A: GRS (geo-redundant storage) asynchronously replicates your backup data to the Azure paired region for durability against a regional outage, while cross-region restore (CRR) is the feature that lets you actually initiate restores from those secondary-region recovery points — even when no outage has occurred. GRS gives you the copy; CRR gives you on-demand access to it. CRR also upgrades the vault storage from GRS to RA-GRS.

Q: What does enabling cross-region restore (CRR) do to your vault's storage redundancy?
A: Turning on CRR upgrades the backup storage from GRS to read-access geo-redundant storage (RA-GRS), which is what allows reads/restores from the secondary region on demand rather than only after Microsoft declares a disaster. This is why CRR is billed at a different rate than plain GRS. It also lets you run resiliency drills by restoring into the paired region without an actual outage.

Q: For Azure Backup, what RPO should you expect for backup data after a full region loss with GRS?
A: Expect up to about 36 hours of potential data loss: the primary-region RPO is 24 hours and asynchronous replication to the secondary region can lag by up to 12 hours, summing to roughly 36. This is the worst-case window of backups that may not have reached the paired region when the primary failed. It is a key number to cite when defining RPO targets for a fund platform's DR plan.

Q: With Azure Backup GRS, how long can it take for backup data to replicate to the secondary region?
A: Backup data can take up to 12 hours to asynchronously replicate from the primary to the paired secondary region after it is committed locally. This replication lag is part of why the region-loss RPO is up to 36 hours (24-hour primary RPO plus up to 12 hours of replication lag). It means very recent backups may not yet exist in the secondary region during a sudden regional disaster.

Q: With CRR enabled and backups already replicated, what is the RTO to start restoring in the paired region?
A: If CRR is enabled and the recovery points are already replicated to the paired region, the RTO for *initiating* a restore there is effectively zero — you can start the restore immediately without waiting for Microsoft to declare a disaster. Without CRR, the RTO depends on how long it takes Microsoft to declare a disaster, which can be lengthy. This is the operational reason to enable CRR for critical workloads.

Q: After enabling CRR, how long can it take before backup items are actually available in the secondary region?
A: After you turn on CRR, it can take up to 48 hours for backup items to become available in the secondary region. So CRR is not an instant safety net the moment you flip it on — you must enable it well ahead of any drill or incident. This lead time should be planned into DR readiness, not discovered during an outage.

Q: Why must you configure ZRS or GRS on an Azure Backup vault *before* the first backup?
A: The vault's storage redundancy is locked once protection begins: for Recovery Services vaults you must set it before protecting any workload, and for Backup vaults it is set at vault creation and cannot be changed afterward. If you miss this, the only fix is creating a new vault with the right redundancy and reassigning workloads, which involves downtime and manual cleanup of old recovery points. So redundancy is a day-zero decision.

Q: What is the minimum storage redundancy Azure recommends for production backups, and why?
A: Azure recommends ZRS (zone-redundant storage) as the minimum tier for production backups because it synchronously replicates the backup data across multiple availability zones, so a single zone outage does not make your backups unavailable for restore. LRS keeps copies only within one zone/datacenter and offers no zone resilience. For regional protection on top of ZRS-class resilience, you add GRS with CRR.

Q: In Azure Backup, what is the difference between the service being zone-resilient and your backup *data* being zone-redundant?
A: The Backup service infrastructure (job orchestration, control plane) is automatically zone-resilient in supported regions, but that built-in resilience does not extend to your backed-up data — the data's zone redundancy depends on whether you chose ZRS for the vault. If you leave the vault as LRS, your backup data is nonzonal and a zone outage could make it unavailable even though the service itself stays up. So you must explicitly opt into ZRS for data resilience.

Q: What is the Azure Backup SLA retry condition you must meet to be covered?
A: To be covered by the Azure Backup SLA, you must retry any failed backup or restore job at least once every 30 minutes. The SLA covers availability of the service for both backup and restore operations, but only if you follow this retry discipline. In practice this means your orchestration should automatically re-trigger failed jobs rather than leaving them failed.

Q: GRS for Azure Backup only works between which kinds of regions?
A: GRS for Azure Backup only works within Azure paired regions — the predefined primary/secondary region pairs Azure maintains. You cannot point GRS at an arbitrary region of your choosing; the secondary is always the designated paired region. This constraint matters for data-residency planning, for example keeping EU fund data within EU pairs.

Q: Why does an Azure data source have to live in the same region as its Backup vault?
A: Backups are always performed in the primary region, which is where both the vault and the data source are deployed; you cannot back up a data source into a vault located in a different region. Cross-region protection is achieved by GRS replicating the *backup data* to the paired region, not by placing the vault elsewhere. This keeps the backup path local and predictable while still allowing geo-redundancy.

Q: If you have a GRS vault but did not enable CRR, when can you restore from the secondary region?
A: Without CRR, the secondary-region data is only made available for restore after Microsoft formally declares a disaster in the primary region, which can take an extended time depending on incident severity. You cannot self-initiate a secondary-region restore on demand. This is the gap CRR closes by giving you immediate, self-service access to paired-region recovery points.

Q: What is operational backup for Azure Blobs versus vaulted backup?
A: Operational blob backup is a local, continuous protection that keeps data in the source storage account (using soft delete, versioning, and change feed) for short-term recovery, while vaulted blob backup copies data into a Backup vault for longer retention and stronger isolation. Operational backup is cheaper and fast for recent recovery but does not give you a separate, off-account copy. Vaulted backup is what you reach for when you need isolated, longer-term, immutable-capable protection.

Q: Why does Azure Backup immutability not protect operational-tier blob/file/disk backups, and what does that imply?
A: Operational-tier backups store snapshots in the source's own resource context rather than copying into the vault, so the vault-level immutability setting does not govern them. The implication is that for true ransomware-grade immutability on blobs you must use vaulted backup (or storage-account-level immutability/object lock), not assume the immutable-vault toggle covers everything. It is a frequent misconfiguration to enable immutable vault and believe operational blob backups are now WORM-protected.

Q: In a backup policy matrix, why is "restore-test cadence" a required column rather than optional?
A: Because an untested backup is an unproven assumption: the only way to know recovery works is to have restored recently, so the cadence column forces a scheduled drill and produces the date that answers "when did we last restore?". Listing it per store prevents the common failure of backing up diligently but never verifying restores. It is also the evidence an auditor will ask to see.

Q: How would you "time a restore" from MinIO versioning, and why does timing it matter?
A: You would record the start, run `mc ls --versions` to find the target version, `mc cp --version-id=UUID ...` to retrieve it, write it back, and record the end, producing an actual elapsed time. Timing matters because RTO is a number you must defend, and a restore that technically works but takes hours may still breach a recovery objective. The measured time turns a vague "we can restore" into a concrete RTO data point.

Q: What is the difference between RPO and RTO?
A: RPO (recovery point objective) is how much data loss you can tolerate, measured as the time between the last recoverable point and the incident; RTO (recovery time objective) is how long you can tolerate being down, measured as the time to restore service. RPO drives backup frequency and replication; RTO drives how fast and accessible your restore path must be. A daily backup gives at most a 24-hour RPO, while CRR can give a near-zero RTO to initiate a paired-region restore.

Q: Why is the question "when did we last restore?" stronger evidence than "when did we last back up?"
A: A successful backup job only proves data was written somewhere; it does not prove the data is complete, uncorrupted, and recoverable, whereas a successful restore proves the entire recovery chain end to end. Auditors and regulators in fund administration care about demonstrated recoverability, not just job-success logs. That is why the lesson makes the restore date — not the backup date — the litmus test.

Q: For a daily NAV pipeline's output table, what does a defensible backup policy row look like?
A: It would name the store (e.g., the Iceberg NAV results table), set frequency to at least daily aligned to the NAV cycle, set retention long enough for regulatory record-keeping (years), mark immutability where ransomware protection is required, and define a restore-test cadence (e.g., quarterly). It would also note that the catalog must be protected separately, since snapshots alone are not a backup. This row is exactly the kind of explicit decision the policy matrix forces.

Q: Why might a transfer-agency register database need both PITR and long-term immutable backups?
A: PITR lets you undo a recent bad batch (e.g., a mis-applied subscription/redemption) to the exact second within its window, while long-term immutable backups satisfy multi-year regulatory retention and protect against ransomware or corruption discovered long after the PITR window closed. The register is both operationally hot and legally significant, so it needs fine-grained recent recovery and durable, tamper-proof history. Relying on only one leaves a gap.

Q: A ransomware event encrypts production and the attacker reaches the Backup vault — what feature determines whether you can still recover?
A: Whether the immutable-vault setting is in the `Enabled and locked` (WORM) state determines it: locked immutability means the attacker cannot delete recovery points or shorten retention even with vault access, so the backups survive. If immutability were merely `Enabled` (reversible) or disabled, the attacker could disable it and purge backups. This is the scenario that justifies locking immutability despite its irreversibility.

Q: Why does the lesson say "Iceberg snapshots are not a backup when the catalog itself is lost"?
A: Because the snapshots are referenced through the catalog's pointer to the current metadata file; losing the catalog severs the only authoritative link to those snapshots, leaving orphaned files you cannot mount as a table. The snapshots provide time travel *within* a working catalog, not independent recoverability of the whole table. Therefore catalog backup is a separate, mandatory concern.

Q: What is the danger of treating object versioning as your only backup for a fund's EMT/EPT file store?
A: Versioning protects against accidental overwrite/delete of individual files but not against bucket deletion, account/credential compromise, region loss, or a lifecycle rule expiring old versions — any of which could wipe the regulatory EMT/EPT history. For files that feed regulatory reporting, you need an isolated, immutable, tested backup in addition to versioning. Versioning is a useful first layer, not the whole defense.

Q: How do storage lifecycle rules interact with versioning to keep version growth from becoming a cost problem?
A: Lifecycle rules can expire or tier *noncurrent* (older) versions after a defined age, so the bucket does not accumulate unbounded history and cost. This makes versioning sustainable, but it also means your effective retention is set by the lifecycle rule, not by versioning alone. The gotcha is setting the expiry too short and silently deleting versions you were relying on as backups.

Q: When defining a backup policy matrix, why must you note per store "whether a snapshot is a backup"?
A: Because some stores' snapshots are isolated, immutable copies that genuinely qualify as backups, while others (like Iceberg table snapshots tied to a live catalog) do not and need additional catalog protection. Recording this per store prevents the false confidence that "we have snapshots, so we are covered." It surfaces exactly where extra backup work is required.

Q: What is the practical first step to verify your Iceberg catalog is recoverable?
A: Identify what your catalog actually is (e.g., a relational metastore database or a REST/Glue/Nessie catalog) and confirm that store is itself backed up — and then test re-registering or restoring a table's current-metadata pointer from that backup. If you cannot restore the pointer, your data files are unmountable. This is the catalog-specific extension of "when did we last restore?".

Q: Why is GRS, not LRS, the default and recommended replication for Azure Backup vaults?
A: GRS replicates backup data to a secondary paired region hundreds of miles away, giving durability even against a full regional outage, whereas LRS keeps all copies within a single region/datacenter. For backups — whose entire purpose is surviving disasters — regional durability is worth the extra cost. ZRS is the in-region middle ground (multi-zone), and GRS adds the cross-region layer.

Q: What happens to active Azure Backup jobs when an availability zone fails and the vault uses ZRS?
A: Jobs for data sources in healthy zones may see only a few seconds of downtime while the platform switches the Backup service to healthy zones, and they then continue; jobs for data sources located in the failed zone may pause or fail until that zone or data source recovers. Backed-up data itself is not lost because ZRS replicates synchronously across zones (zero RPO for the backup data). You are responsible for restoring affected data sources to a healthy zone.

Q: With ZRS, what is the expected backup-data loss (RPO) during a single zone outage?
A: Essentially zero: ZRS replicates backup data synchronously across availability zones, meaning multiple zones acknowledge each write before it completes, so a single zone outage does not lose committed backup data. The exposure during a zone outage is to the *data sources* in the failed zone, not to the already-committed backups. This is the resilience benefit that justifies ZRS as the production minimum.

Q: Who is responsible for restoring your data after an Azure availability-zone outage, you or Microsoft?
A: Microsoft handles detecting and recovering the Backup *service's* own zone resilience, but you are responsible for detecting that your data/resources are unavailable and taking recovery actions, including restoring backups to a healthy zone. The shared-responsibility model means the platform keeps backups durable, but initiating the restore of your workloads is your job. So your runbooks must own the restore step.

Q: Why is "soft delete" sometimes described as a recycle bin rather than true immutability?
A: Soft delete keeps deleted items recoverable for a window (default 14 days) but the items can still ultimately be purged after that window or if soft delete is disabled, whereas immutability/WORM actively blocks deletion before expiry regardless of intent. Soft delete buys time to undo mistakes; immutability prevents the destructive action in the first place. A ransomware-grade posture wants both, with immutability locked.

Q: An attacker with vault permissions tries to shorten a backup policy's retention to 1 day to age out backups — what stops them?
A: An immutable vault stops them: reducing the retention period in a backup policy (or swapping in a policy with lower retention) is one of the operations immutability blocks. Retention can only be increased, not decreased, on an immutable vault. This closes a subtle attack path where backups are not deleted directly but are forced to expire quickly.

Q: Why can you still "stop protection while retaining data" on an immutable Azure Backup vault?
A: Immutability blocks operations that *lose* data, but stopping protection while keeping existing recovery points until their expiry does not delete anything, so it is permitted. What is blocked is stop-protection-*with-deletion* and any early removal of recovery points. This lets you legitimately decommission a workload without violating immutability or losing already-captured backups.

Q: What is the relationship between Iceberg's current-metadata pointer and time travel?
A: The catalog's pointer identifies the current metadata file, which lists all valid snapshots; time travel works by reading an older snapshot referenced in that metadata rather than the current one. So time travel is entirely dependent on the metadata being intact and reachable via the catalog. If `expire_snapshots` removed a snapshot or the catalog pointer is lost, that time-travel target is gone.

Q: After running `expire_snapshots` aggressively, a team loses the ability to roll back a bad write — what went wrong?
A: They expired the snapshots that represented the pre-bad-write state, permanently deleting the files only those snapshots referenced, so the rollback target no longer exists. The fix in design terms is to set `retain_last` and a conservative `older_than` so recent recovery points survive, and to keep real backups separate from snapshot retention. This is a concrete example of why snapshots are not a substitute for backups.

Q: Why should `remove_orphan_files` be run with a conservative `older_than` margin?
A: Concurrent or in-flight writes can create files that look orphaned but are about to be committed; a too-recent cutoff could delete them and corrupt the table. A conservative `older_than` (well in the past) ensures only genuinely abandoned files from old failed writes are removed. The default guidance is to leave a safety margin precisely to avoid racing live operations.

Q: How does the "stop protection with deletion of data" restriction protect a regulated fund's records?
A: It prevents anyone — accidental or malicious — from terminating protection of, say, the NAV history table and simultaneously deleting its recovery points before their retention expiry, which would destroy records the fund is legally required to keep. On an immutable vault this destructive combination is refused. Records can only age out naturally at expiry, preserving the regulatory retention guarantee.

Q: Why might a Luxembourg-domiciled fund care about which Azure paired region GRS replicates to?
A: Because GRS only replicates to the predetermined Azure paired region, and EU/Luxembourg data-residency and regulatory expectations may require the secondary copy to stay within the EU; the fund must verify the pair keeps data in an acceptable jurisdiction. Choosing a primary region whose pair is also EU-based keeps backups compliant. This is a residency decision that must be made before the first backup, since redundancy is locked early.

Q: How does DORA's emphasis on operational resilience connect to this lesson's backup discipline?
A: DORA (Digital Operational Resilience Act) requires financial entities to be able to recover ICT services and data within defined objectives and to test that recovery, which is exactly the tested-restore, defined-RPO/RTO, immutable-backup discipline this lesson teaches. The backup policy matrix and the "when did we last restore?" date are the kind of evidence a DORA-aligned program produces. So the lesson's deliverables map directly onto regulatory resilience-testing obligations.

Q: Why is "immutable, tested backups" specifically called out as the defense against a bad catalog migration?
A: A catalog migration that goes wrong can corrupt or invalidate the pointer to your table metadata, and snapshots within that catalog cannot save you because they depend on the same catalog. Only an independent, immutable, tested backup of the catalog (and metadata) lets you roll the platform back to a known-good state. Untested or mutable backups might be incomplete or already tampered, leaving the migration unrecoverable.

Q: In MinIO, what is the difference between `mc rm` on a versioned object with and without `--version-id`?
A: Without `--version-id`, `mc rm` on a versioned bucket performs a soft delete by adding a 0-byte delete marker as the latest version, leaving prior versions recoverable; with `--version-id`, it permanently removes that exact version's data. So the same command is either reversible or irreversible depending on whether a version ID is supplied. Knowing this prevents both accidental data loss and false confidence that data is "gone."

Q: Why does enabling versioning before incidents matter, given it is per-bucket?
A: Versioning only protects versions created *after* it is enabled — objects written earlier carry a `null` version ID and have no prior versions to fall back to — and it is set per bucket, so an unversioned bucket has no soft-delete safety at all. Enabling it proactively on every relevant bucket is what gives you the overwrite/delete recovery you are counting on. Turning it on after an incident does not retroactively create history.

Q: What does "Iceberg snapshot retention vs backup" mean as a design distinction?
A: Snapshot retention governs how far back you can time-travel *within* a live, catalog-backed table and is tuned with `expire_snapshots` (`older_than`/`retain_last`); backup is about keeping an independent, isolated, ideally immutable copy that survives catalog loss, region loss, or ransomware. They answer different questions — "can I roll back inside the system?" versus "can I rebuild the system from outside?". A good design configures both deliberately and never conflates them.

Q: A scheduled MinIO restore drill fails because `mc ls --versions` shows only a `null` version — what is the likely cause?
A: Versioning was almost certainly not enabled on that bucket when the object was written, so no historical versions exist and there is nothing to restore to. The fix is to enable `mc version enable ALIAS/BUCKET` going forward (it will not recover past history) and to verify versioning is on for every bucket in the policy matrix. This is a common "we thought we had backups" failure caught only by an actual restore drill.

Q: Why is it risky to disable Azure Backup soft delete to "save space" or "clean up faster"?
A: Disabling soft delete removes the 14-day recovery window, so an accidental or malicious deletion becomes immediately permanent with no recycle-bin to undo it. The space/cost saved is trivial compared to the recoverability lost, and it hands attackers a faster path to destroy backups. Enhanced/always-on soft delete exists precisely so this safety net cannot be casually or maliciously turned off.

Q: How do you decide an appropriate retention for a backup store in the policy matrix?
A: You balance regulatory retention requirements (which can mandate years for fund records), the realistic detection window for corruption (so PITR/short retention is not your only line), and cost; the longest binding requirement usually sets the floor. For example, PostgreSQL fund data might need multi-year vaulted retention even if PITR is only 35 days. The matrix forces this to be an explicit, defensible number rather than a default left unexamined.

Q: Why does the lesson pair "object-store versioning + lifecycle" together rather than treating versioning alone?
A: Because versioning without lifecycle leads to unbounded version accumulation and cost, while lifecycle rules expire or tier old/noncurrent versions to keep it sustainable; the two together form the realistic backup-via-versioning pattern. The pairing also surfaces a key limit: lifecycle is what enforces your effective retention, and a misconfigured rule can silently delete your "backups." So you must reason about both the keeping and the expiring.

Q: What is the difference between a snapshot-based operational backup and a vaulted backup in terms of isolation?
A: An operational/snapshot backup typically lives in the same resource or account as the source (fast and cheap, but sharing the source's blast radius), while a vaulted backup is copied into a separate vault under a distinct control plane, giving stronger isolation from a compromise of the source. Isolation is what makes vaulted backups resistant to ransomware that reaches production. The trade-off is cost and restore speed, so many designs use operational for recent recovery and vaulted for durable protection.

Q: For a daily NAV file delivery to investors, why is a tested restore path more important than backup frequency alone?
A: Even hourly backups are worthless if you cannot restore in time to meet the NAV delivery deadline, so a measured, tested restore path (a known RTO) is what actually guarantees you can re-deliver after an incident. Frequency sets RPO (how much you might lose) but the restore drill sets RTO (how fast you recover). For a time-critical investor deliverable, breaching RTO is as damaging as data loss, so both must be proven.

Q: Why does Azure Backup let you increase but not decrease retention on an immutable vault?
A: Increasing retention keeps data longer and cannot cause data loss, so it is allowed; decreasing retention would cause recovery points to expire and be deleted sooner, which is exactly the data-loss outcome immutability is designed to prevent. The asymmetry encodes the principle "you may protect more, never less." This blocks a stealthy way to destroy backups by aging them out rather than deleting them directly.

Q: How would you reconstruct an Iceberg table after losing the catalog but still holding the object-store files?
A: You would re-register the table by pointing a catalog at the most recent valid metadata file (if you backed it up or can locate it in storage), restoring the current-metadata pointer so the manifests, snapshots, and data files become resolvable again. If neither the catalog nor the metadata JSON was backed up, you may have to rebuild the table definition and reload, since the lineage is lost. This is why the metadata pointer itself must be part of the backup, not just the parquet files.

Q: What does it mean that ZRS replication in Azure Backup is "synchronous" while GRS is "asynchronous," and why does it matter for RPO?
A: Synchronous ZRS means every write is acknowledged by multiple zones before it completes, so a zone outage loses no committed backup data (zero RPO across zones); asynchronous GRS means data is committed locally first and replicated to the paired region afterward, so a sudden region loss can lose recent writes (up to ~36 hours RPO). The replication mode directly sets how much you can lose in each failure type. This is why in-region resilience is effectively lossless but cross-region resilience has a real RPO window.

Q: Why does the lesson insist a complete plan covers "every store" rather than just the main database?
A: Recoverability is only as strong as the weakest unprotected store: a NAV pipeline spans a database, object-store landing zones, table-format results, and a catalog, and an unbacked-up catalog or EMT/EPT bucket can make the whole platform unrecoverable even if the database is perfectly protected. Auditors and ransomware both find the gap you forgot. The policy matrix's "every store" requirement exists precisely to eliminate silent blind spots.

Q: Summarize the complete backup posture this lesson expects for a regulated fund data platform.
A: Every store has an explicit policy (what, frequency, retention, immutability, restore-test cadence); databases pair PITR for recent recovery with long-term vaulted backups for retention; object stores use versioning plus lifecycle but are recognized as not fully isolated; table formats are backed up including their catalog/metadata pointer, not just data files; backups are immutable (locked WORM) and geo-redundant where required; and recoverability is proven by a dated, tested restore. The result is that "when did we last restore?" has a real answer and the platform survives ransomware, bad migrations, and regional loss.

Q: What is Azure Backup multi-user authorization (MUA), and what resource implements it?
A: MUA adds a second pair of hands to critical vault operations (like disabling soft delete, reducing immutability, or stopping protection) by requiring authorization through a separate Azure resource called a Resource Guard. A backup admin who lacks the `Backup MUA Operator` role on the Resource Guard cannot perform the protected operations alone, so a single compromised admin account cannot unwind your protections. It is the access-control complement to immutability: immutability blocks the destructive operation, MUA blocks the unauthorized actor from even reaching it.

Q: Why does the Azure Backup guidance recommend placing the MUA Resource Guard in a *separate* tenant?
A: Putting the Resource Guard in a different Microsoft Entra tenant means an attacker who fully compromises the backup admin's tenant still cannot grant themselves the `Backup MUA Operator` role needed to approve destructive operations, because that role lives under a separate identity boundary. It maximizes separation of duties so neither tenant alone can both delete production and unwind backup protection. The trade-off is operational complexity, since legitimate changes now require cross-tenant approval (often via Entra PIM just-in-time access).

Q: How does MUA differ from immutable vault as a ransomware control — do you need both?
A: Immutable vault is a policy-layer guarantee that destructive operations on recovery points are refused outright, while MUA is an identity-layer gate that requires a second authorized party (the Resource Guard) before in-scope operations can even be attempted. They defend different attack steps: immutability stops the action, MUA stops the unauthorized actor, so a hardened posture uses both plus locked soft delete. Relying on MUA alone still lets an authorized-but-malicious approver act, and relying on immutability alone still lets an admin disable other safety nets MUA would have guarded.

Q: An admin requests to disable soft delete on a vault that has MUA enabled — walk through what happens.
A: The request is blocked unless the requesting admin holds the `Backup MUA Operator` role on the linked Resource Guard; if they do not, they must obtain that access (typically a just-in-time grant approved through Entra PIM by the Resource Guard's owners) before the operation succeeds. This forces a deliberate, logged, second-party approval for a security-weakening change. The whole point is that disabling soft delete is exactly the kind of pre-attack step ransomware actors take, so it must not be a single-admin action.

Q: What is the configurable retention range for Azure Backup soft delete, and which part is free?
A: Soft-delete retention is configurable from `14` to `180` days, with `14` days being the default and free tier; any retention beyond 14 days incurs normal backup storage charges for the soft-deleted items. Longer windows help against advanced persistent threats that may sit undetected for weeks before triggering a deletion. So extending to, say, 60 or 90 days buys detection time at a storage cost you must budget for.

Q: For a fund worried about a slow-burn (advanced persistent threat) attack on backups, how would you tune soft-delete retention?
A: Extend soft-delete retention well beyond the free 14-day default — toward the `180`-day maximum — so that even an attacker who deleted backups weeks before being detected leaves the items recoverable for the longer window. Pair this with enhanced (always-on) soft delete so the window itself cannot be shortened or disabled, and accept the storage cost beyond 14 days. The reasoning is that APTs deliberately dwell to outlast short recovery windows, so the window must outlast the dwell time.

Q: What is the relationship between basic soft delete and enhanced soft delete in Azure Backup?
A: Basic soft delete gives a fixed-style recycle-bin window (default 14 days) that an admin can turn off, whereas enhanced soft delete makes the feature configurable (14-180 days) and, critically, lets you make it always-on so it cannot be disabled. Enhanced soft delete is the upgrade you enable to remove the "attacker disables soft delete first" weakness. Once always-on is set, the safety net is permanent for the vault, which is the intended hardening.

Q: You enabled immutable vault in the `Enabled` state but not locked — what real attack can still succeed?
A: An attacker (or rogue admin) with vault permissions can simply set immutability back to `Disabled` and then delete recovery points or shorten retention, because the reversible `Enabled` state offers no defense against someone who can change the setting. Only `Enabled and locked` (WORM) removes that path by making immutability irreversible. The reversible state is meant for evaluating impact, not for production ransomware defense, so leaving it merely `Enabled` is a common false-security gap.

Q: How do MUA and locked immutability combine to close the "disable then delete" attack chain?
A: Locked immutability removes the ability to delete recovery points or shorten retention even after disabling immutability (because it cannot be disabled), while MUA ensures that the very act of trying to weaken protections requires a separate Resource Guard authorization. Together they break the chain at both links: the destructive action is impossible and the weakening action is gated. This layered design is why a regulated fund would deploy both rather than treating either as sufficient.

Q: In MinIO, what is the difference between COMPLIANCE and GOVERNANCE object-lock retention modes?
A: Under `COMPLIANCE` mode a locked object version cannot be deleted or have its retention shortened by anyone — including the MinIO root user — until retention expires, giving true unbreakable WORM; under `GOVERNANCE` mode a privileged user holding the `s3:BypassGovernanceRetention` permission can override, shorten, or lift the lock early. So COMPLIANCE is for hard regulatory immutability and GOVERNANCE is for managed immutability with an authorized escape hatch. Choosing GOVERNANCE where you needed COMPLIANCE leaves a privileged-bypass hole in your "immutable" backups.

Q: For a regulatory EMT/EPT file store that must be tamper-proof for the retention period, which MinIO lock mode fits and why?
A: Use `COMPLIANCE` mode, because regulatory retention demands that not even an administrator (or a compromised root credential) can delete or alter the files before expiry, which is exactly the guarantee COMPLIANCE provides. `GOVERNANCE` would let anyone with `s3:BypassGovernanceRetention` override the lock, defeating the regulatory tamper-proofing. The trade-off is that COMPLIANCE allows no legitimate early correction either, so the retention duration must be chosen carefully up front.

Q: What does the `s3:BypassGovernanceRetention` permission allow, and why is it a control point?
A: It permits a privileged user to delete, overwrite, or shorten the retention of objects locked under `GOVERNANCE` mode before their retention expires, effectively the sanctioned override for managed-immutability locks. It is a control point because anyone granted it can bypass your GOVERNANCE WORM protection, so the permission must be tightly restricted and audited. In a fund context you would grant it sparingly and log every use, treating it like a break-glass capability.

Q: How does a MinIO legal hold differ from a retention lock, and what is the command to set one?
A: A legal hold is an on/off WORM flag with no expiry date — set with `mc legalhold set ALIAS/BUCKET/OBJECT` — that keeps an object version immutable until someone explicitly removes the hold, whereas a retention lock has a fixed expiry timestamp. Critically, a legal hold survives even after a COMPLIANCE/GOVERNANCE retention period expires, so the object stays locked until the hold is lifted. This makes legal holds the right tool for litigation/audit preservation of indefinite, unknown duration.

Q: A regulator demands a fund preserve specific subscription records "until further notice" with no known end date — which MinIO feature fits?
A: A legal hold, set via `mc legalhold set`, because it provides WORM protection with no expiry and is lifted only by an explicit human action — matching an open-ended "until further notice" obligation. A time-based retention lock would require guessing an end date and would auto-expire. The legal hold also stacks on top of any existing retention lock, so the records stay immutable until both the retention expires and the hold is explicitly removed.

Q: Historically, when could MinIO object locking be enabled on a bucket, and what command created such a bucket?
A: Traditionally, object locking had to be enabled at bucket creation time — mirroring S3 behavior — using `mc mb --with-lock ALIAS/BUCKET`, and it could not be added to a bucket that was created without it. This meant retrofitting WORM onto an existing unlocked bucket was impossible without recreating and re-copying data. It is a classic day-zero design decision, just like Azure vault redundancy being locked before the first backup.

Q: A team needs WORM on an existing MinIO bucket that was created without `--with-lock` — what are their options?
A: On older releases the only option is to create a new bucket with `mc mb --with-lock`, copy the data across, and switch over, because locking could not be added after creation. On newer MinIO/AIStor releases (from the May 2025 release onward) object locking and default retention can be applied to an existing bucket via `mc retention set --default`, removing the need to recreate. The fix therefore depends on the server version, so checking the release is the first step.

Q: What does `mc retention set --default` configure on a MinIO bucket?
A: It sets a default retention mode (`GOVERNANCE` or `COMPLIANCE`) and duration that is automatically applied to new object versions written to that bucket, so you do not have to lock each object individually. It is also the mechanism that, on newer releases, enables object locking on an existing bucket. Setting a sensible bucket-level default is how you make WORM the automatic behavior for a backup or regulatory bucket rather than a per-object afterthought.

Q: How would you verify whether a MinIO bucket actually has object locking and what default retention it carries?
A: Run `mc retention info ALIAS/BUCKET`, which reports whether object locking is configured and the default retention mode and duration in effect. This is the check you run during a restore drill or audit to confirm your "immutable" bucket is genuinely WORM-protected rather than just versioned. Skipping this verification is how teams discover, mid-incident, that a bucket they believed was locked never had locking enabled.

Q: A MinIO bucket has versioning on but objects are still being permanently deleted by an attacker with credentials — why, and what would have prevented it?
A: Versioning alone only soft-deletes via delete markers and still allows deletion of specific version IDs, so an attacker with `s3:DeleteObjectVersion` can purge the underlying versions outright. Object locking in `COMPLIANCE` mode (or GOVERNANCE without granting bypass) would have prevented it by making the versions undeletable until retention expiry. The lesson's point stands: versioning is recoverability-if-nobody-purges, while locking is genuine immutability.

Q: What is the Iceberg `rollback_to_snapshot` procedure and when is it the right recovery tool?
A: `rollback_to_snapshot` resets a table's current state to a specific earlier snapshot ID that is an ancestor of the current state, undoing later commits in one metadata operation. It is the right tool to reverse a recent bad write when the pre-error snapshot still exists and has not been expired. Because it only moves the catalog's current-metadata pointer, it is fast and does not rewrite data — but it cannot recover anything `expire_snapshots` already deleted.

Q: How does Iceberg `set_current_snapshot` differ from `rollback_to_snapshot`?
A: `rollback_to_snapshot` requires the target snapshot to be an ancestor of the current state (a true roll-back along the table's history), whereas `set_current_snapshot` can point the table at any snapshot or named ref, even one not on the current ancestry line. So `set_current_snapshot` is more flexible (e.g., jumping to a tagged snapshot) but easier to misuse since it can move the table off its lineage. Both work only on snapshots that still exist, reinforcing that snapshot retention, not these procedures, is what bounds recoverability.

Q: After a bad write, the pre-error Iceberg snapshot exists but a later good write also landed — can `rollback_to_snapshot` cleanly recover, and what is the risk?
A: Rolling back to the pre-error snapshot will discard *every* commit after it, including the later good write, because rollback resets the current pointer to that snapshot's state. The risk is silently losing legitimate work that happened after the corruption point, so you must first decide whether to roll back and re-apply the good change or use `cherrypick_snapshot` to graft only the wanted commit. This is the kind of nuance that makes snapshot recovery a deliberate operation, not a one-click undo.

Q: What does Iceberg `cherrypick_snapshot` do, and what is its limitation?
A: `cherrypick_snapshot` applies the changes from an existing snapshot onto the current table state as a new snapshot, without removing or altering the original, letting you graft a specific change without a full rollback. Its limitation is that only append and dynamic-overwrite snapshots can be cherry-picked, so arbitrary conflicting changes are not always graftable. It is useful when you want to keep most recent history but reintroduce or isolate one particular committed change.

Q: Why must Iceberg `expire_snapshots` be coordinated with your backup/recovery plan rather than run blindly on a schedule?
A: `expire_snapshots` permanently deletes the data and metadata files referenced only by expired snapshots, so a routine maintenance job can silently destroy the very recovery points (`rollback_to_snapshot` targets) you would need after a bad write. If your real recoverability depends on those snapshots and you have no independent backup, the cleanup job becomes a data-loss event. The plan must therefore set `retain_last` and a conservative `older_than` aligned to your recovery window, and keep isolated backups separate from snapshot lifecycle.

Q: In Iceberg terms, what files does `expire_snapshots` actually delete versus what it leaves alone?
A: It deletes manifest lists, manifests, and data files that are referenced *only* by the snapshots being expired; any file still referenced by a surviving snapshot is retained because it is shared history. This is why expiring one old snapshot may free little space (most files are shared) while expiring a long tail can free a lot. Understanding the shared-reference model prevents the wrong assumption that expiring one snapshot reclaims a predictable fixed amount.

Q: Why is a `null` version ID in MinIO a red flag during a restore drill, beyond just "versioning was off"?
A: A `null` version ID means the object was written while versioning was disabled (or pre-existed it), so it has no prior versions and no soft-delete safety — a later overwrite simply replaced the only copy. The deeper red flag is that it reveals versioning was enabled too late, meaning an unknown set of objects in that bucket may share the same gap. The remediation is to audit the whole bucket for `null`-versioned objects and confirm versioning is on going forward, since it cannot retroactively create history.

Q: How would you script a periodic MinIO restore drill so the "when did we last restore?" date is always current?
A: Schedule a job (e.g., cron on the Ubuntu host) that records a start timestamp, runs `mc ls --versions` to pick a known target version, `mc cp --version-id=UUID ...` to retrieve it into a scratch path, validates a checksum against the expected value, records an end timestamp, and writes the elapsed time and pass/fail to a log the policy matrix references. Automating it guarantees the restore date stays fresh and the RTO number stays measured rather than assumed. The checksum step is what upgrades the drill from "a copy ran" to "the restored bytes are correct."

Q: During a restore drill, the copied-back object differs from the original by checksum — what should you suspect first?
A: First suspect that you restored the wrong version ID — a delete marker or an older/newer version than intended — since `mc ls --versions` lists many versions and it is easy to grab the wrong UUID. Next suspect partial-copy or in-flight corruption, then validate the source version's own integrity. The lesson's discipline of timing and verifying restores exists precisely to catch this class of silent mismatch before a real incident.

Q: Why does the Azure Backup SLA's "retry every 30 minutes" rule imply something about your orchestration design?
A: It implies your backup orchestration must automatically detect and re-trigger failed jobs on at least a 30-minute cadence, because the SLA only covers you if that retry discipline is followed — a human noticing a failure the next morning does not qualify. So the design needs alerting plus automated retry, not just dashboards. In practice this means wiring failed-job events to a retry mechanism so a transient failure self-heals within the SLA window.

Q: A scheduled Azure Backup job has been in a failed state for two hours with no retries — what SLA and recovery consequences follow?
A: You fall outside the Azure Backup SLA's coverage because the SLA requires retrying a failed backup or restore job at least once every 30 minutes, and two hours with no retry breaches that condition. Operationally, every failed cycle widens your RPO gap, since the last good recovery point is now older than intended. The fix is to re-trigger immediately and add automated retry so a stuck failure cannot silently age your recovery point.

Q: Why can you not back up an Azure data source into a vault located in a different region, and how is geo-protection actually achieved?
A: Backups are always performed locally in the primary region, so the vault must co-locate with the data source; cross-region protection is achieved not by remote vaults but by GRS asynchronously replicating the *backup data* to the Azure paired region. This keeps the backup path local and predictable while still surviving a regional outage. A common misconception is thinking you place a vault in a remote region for DR — you do not; you enable GRS (and CRR for self-service restore).

Q: A Luxembourg fund must keep both primary and secondary backup copies inside the EU — how does this constrain Azure region choice?
A: Because GRS replicates only to the fixed Azure paired region, you must choose a primary region whose designated pair is also within the EU (e.g., an EU-to-EU pairing), since you cannot redirect GRS to an arbitrary region. Verifying the pair's jurisdiction is a data-residency prerequisite that must be settled before the first backup, because vault redundancy is locked early. Picking a region whose pair sits outside the EU would silently export regulated fund data on every replication.

Q: You discover a production Azure Backup vault is LRS but compliance requires ZRS — what is the remediation and its cost?
A: Redundancy is locked once protection begins, so you cannot upgrade LRS to ZRS in place; the remediation is to create a new vault with ZRS (or GRS), re-register/protect the workloads there, and let the old recovery points age out under the old vault's retention. The cost is operational effort, a re-seeding of full backups, dual-vault running until old points expire, and careful tracking so no protection gap opens. This is exactly why redundancy is described as a day-zero decision.

Q: Why is "Enabled and locked" immutability sometimes a liability, and how do you de-risk choosing it?
A: Because it is irreversible WORM, a mistake baked in before locking — such as a retention that is too long or a wrong policy — becomes permanent and cannot be corrected, so you may carry unwanted, unbillable-to-justify recovery points for years. You de-risk it by first running in the reversible `Enabled` state to evaluate impact (retention, storage cost, operational fit), validating with a restore drill, and only then moving to locked. The deliberate two-phase approach is the recommended path precisely because locking forecloses correction.

Q: A database's PITR window is 35 days but auditors demand 7-year retention — how does a defensible policy reconcile these?
A: It pairs the short PITR window for fine-grained recent recovery with long-term vaulted backups (Azure Backup can retain PostgreSQL backups up to 10 years) to satisfy the 7-year regulatory floor. PITR handles "undo this morning's bad batch to the second," while the vaulted chain handles "produce the NAV record from four years ago." Neither alone is sufficient: PITR cannot reach beyond 35 days and a weekly vaulted backup cannot undo a mid-day error precisely.

Q: When restoring a managed database via PITR, why might the target be a *new* instance rather than an in-place overwrite, and what does that imply for cutover?
A: PITR typically restores to a new server/instance to preserve the original (and because replaying logs onto a fresh base is the safe operation), which means recovery is not complete until you cut applications over to the restored instance and reconcile any data written after the restore point. The implication is that your RTO must include connection-string/cutover time and any downstream reconciliation, not just the restore duration. Teams that forget the cutover step under-estimate true RTO.

Q: After a NAV-corruption incident, you restore the positions table via PITR to just before the bad batch — what reconciliation step is still required?
A: You must account for any legitimate transactions (subscriptions, redemptions, trades) that posted *after* the chosen restore point but *before* you detected the corruption, since rolling back to pre-batch discards them along with the bad data. The reconciliation step replays or re-enters those good transactions onto the restored state so the register is both clean and complete. Skipping it trades a corruption problem for a silent data-loss problem.

Q: How do you decide between rolling back an Iceberg table with `rollback_to_snapshot` versus restoring from an isolated backup?
A: Use `rollback_to_snapshot` when the corruption is recent, the pre-error snapshot still exists, and the catalog is healthy — it is faster and rewrites no data; fall back to an isolated backup when the snapshot was expired, the catalog itself is lost/corrupted, or the damage predates your snapshot-retention window. The decision hinges on whether the recovery point still lives *inside* the live system or only *outside* it. This is the operational form of "snapshots are not a backup when the catalog dies."

Q: Why does a complete catalog backup need both the metadata-pointer store and the metadata JSON, not just one?
A: The catalog (metadata-pointer store) tells you *which* metadata file is current, and the metadata JSON (plus manifests it references) describes the table's schema, partitioning, and snapshot lineage; you need the pointer to know the right metadata, and the metadata to actually resolve the data files. Backing up only the pointer leaves you pointing at metadata you may not have; backing up only metadata leaves you unsure which version was authoritative. Recovery requires re-establishing the pointer→metadata→data chain end to end.

Q: Your Iceberg catalog is a relational metastore database — what does that mean for its backup strategy?
A: It means the catalog inherits database backup discipline: it needs its own PITR plus long-term vaulted backups, because losing or corrupting that database orphans every table whose current-metadata pointer it holds. Treating it as "just config" is the trap — it is a single point of failure for the entire lakehouse. So the metastore database belongs in the policy matrix as a first-class, high-criticality store with tested restores, not as an afterthought.

Q: How would you test that an Iceberg catalog backup is genuinely restorable, end to end?
A: Restore the catalog (metastore DB or REST/Glue/Nessie state) into an isolated environment, point an engine at it, and confirm you can resolve a known table — read its current snapshot, query rows, and verify schema/partition spec match expectations. If the table mounts and queries correctly, the pointer→metadata→data chain survived; if it does not, your "backup" was incomplete. This is the catalog-specific instance of the lesson's "when did we last restore?" applied to the most fragile component.

Q: Why does the lesson treat the object-store landing zone of a NAV pipeline as a store that needs its own policy row?
A: Because the landing zone holds raw inbound files (e.g., price feeds, transfer-agency files) that may be the only copy before ingestion, and losing them can make a NAV run unreproducible even if the downstream table is fine. An incomplete plan that backs up the results table but not the landing zone leaves a reproducibility gap auditors and re-runs will expose. The "every store" rule exists exactly to surface such silent single-copy stages.

Q: For an object-store bucket protected only by versioning, name three failure modes that would still destroy the data.
A: Bucket deletion (versioning protects object versions, not the bucket itself), credential/account compromise that lets an attacker purge specific version IDs, and a misconfigured lifecycle rule that expires noncurrent versions you relied on as backups — plus full region loss if the store is single-region. Each bypasses versioning's overwrite/delete protection. This is why versioning is a first line, and isolated immutable backups (object lock, off-account copies) are still required.

Q: How can a lifecycle rule intended to control cost become a silent backup-destroyer, and how do you guard against it?
A: A lifecycle rule that expires *noncurrent* versions after N days sets your effective backup retention, so if N is shorter than your real recovery needs it quietly deletes the very versions you assumed were safety copies. You guard against it by setting the noncurrent-version expiry to match (or exceed) the retention recorded in the policy matrix, and by treating any change to lifecycle rules as a change to retention requiring review. The gotcha is that the deletion is automatic and silent, so it surfaces only during a failed restore.

Q: When does an Iceberg snapshot genuinely qualify as a backup, and when does it not?
A: It does not qualify when it is a time-travel point inside a live, single catalog and single storage location, because catalog loss, storage loss, or `expire_snapshots` takes it with the table; it begins to approach a backup only if the snapshot's data and metadata are copied to an isolated, immutable, separately-governed location. The deciding properties are isolation and immutability, not the existence of the snapshot. Recording this per store in the matrix prevents the false confidence that "we have snapshots, so we are covered."

Q: How does DORA's threat-led penetration testing and recovery-testing expectation map onto this lesson's drills?
A: DORA expects financial entities to test their ability to recover ICT services within defined objectives, which maps directly onto the lesson's timed restore drills, defined RPO/RTO, and the dated "when did we last restore?" evidence. The backup policy matrix and a logged restore history are the kind of artifacts a DORA-aligned resilience-testing program produces and an examiner can inspect. So the lesson's deliverables are not just good practice — they are the evidence trail a DORA review asks for.

Q: A bad catalog migration corrupted the metadata pointer and the team has only object-store data files plus a pre-migration catalog backup — what is the recovery path?
A: Restore the pre-migration catalog backup into an isolated environment to recover the known-good current-metadata pointer, validate that it resolves the tables against the surviving data files, then promote it back as the authoritative catalog and reconcile any legitimate writes that occurred after the backup. If the catalog backup is consistent with the data files, this reinstates the pointer→metadata→data chain. This is precisely why the lesson calls immutable, tested catalog backups the defense against a bad catalog migration.

Q: Summarize the layered ransomware-defense stack this lesson expects for a regulated fund's backups, naming the control at each layer.
A: Isolation (backups in a separate vault/control plane so a production compromise does not reach them), soft delete extended and made always-on (a recovery window an attacker cannot shorten or disable), immutable vault in `Enabled and locked` WORM (recovery points cannot be deleted or retention reduced), MUA via a cross-tenant Resource Guard (a second authorized party gates weakening operations), and geo-redundancy with GRS plus CRR (survive regional loss with self-service restore) — all proven by timed, dated restore drills. Each layer assumes the one below can be breached, so the stack as a whole, not any single control, is what makes the backups survivable and audit-defensible.

Q: An auditor asks "what is your RTO for the NAV results table and how do you know?" — what should your answer rest on?
A: It should rest on a measured restore time from a recent timed drill, not a target you hope to hit: cite the date of the last successful restore, the elapsed wall-clock time it took, and the recovery method (e.g., `mc cp --version-id=...` from versioning, vaulted-backup restore, or `rollback_to_snapshot`). A defensible RTO is an observed number plus evidence, because an untested estimate is exactly what auditors and DORA reviewers reject. The backup policy matrix's restore-test-cadence column is what guarantees that number stays fresh.

Q: How would you sanity-check a finished backup policy matrix before handing it to an auditor?
A: Walk every store and confirm each row has all five fields populated (what, frequency, retention, immutability, restore-test cadence) with no blanks, that retention meets the longest binding regulatory requirement, that immutability is locked where ransomware protection is claimed, and that each restore-test cadence has produced a real dated restore. Then confirm no store is missing entirely — especially silent ones like the catalog/metastore and object-store landing zones. A row with an empty cell or a store absent from the matrix is precisely the gap an auditor or an attacker will find first.
