# Phase 6 · 10.6.1 Reference data management — 150 self-test questions

Q: What is reference data, and how does it differ from transactional or master data?
A: Reference data is the set of permissible, relatively stable values used to classify or categorise other data — code lists like currencies, countries, and market identifiers. Unlike transactional data (events such as a trade) or master data (the key business entities such as a fund or counterparty), reference data is small and slow-changing but is referenced pervasively, so an error in it has outsized blast radius across every record that uses the code.

Q: Why does the lesson describe reference data as "small data with outsized blast radius"?
A: The volumes are tiny — a few hundred currency or country codes — but the same code is joined into millions of rows downstream. One wrong currency code (for example treating `USD` as `USS`) can mis-price an entire sub-fund's NAV, because every position valued in that currency inherits the error. The cost is governance attention disproportionate to the row count.

Q: What is the distinction between internal and external reference data?
A: External reference data is published and maintained by a standards body or vendor that you do not control — ISO currency codes, ISO country codes, MICs, LEIs. Internal reference data is owned and defined by your own organisation — for example your house fund legal-form taxonomy or an internal product-class list. The governance discipline differs: for external data you track the published version and ingest it; for internal data you are the source of truth and own the approval workflow.

Q: Give two examples of external reference data and two of internal reference data in a fund-administration context.
A: External: ISO 4217 currency codes and ISO 10383 market identifier codes (MICs), both published by recognised maintenance/registration authorities. Internal: a house chart of fund legal forms (SICAV, FCP, RAIF) mapped to your own codes, and an internal sub-fund classification scheme. The line matters because external lists must be re-ingested when the publisher changes them, whereas internal lists change only through your own controlled process.

Q: What does DAMA-DMBOK identify as the core governance challenge for reference data?
A: Establishing a single authoritative source for each code list and distributing it so every consumer reads the same values, rather than each system maintaining its own copy. The DMBOK reference-data chapter frames the problem as managing definition, versioning, and distribution so that "the same country" does not map three different ways across systems. Without that, silver, gold, and the EMT feed silently diverge.

Q: What is "effective dating" of a code list?
A: Effective dating attaches a validity period (a from-date and a to-date) to each version of a code or mapping, so the system records not just the current value but what the value was at any point in time. It lets a code list change going forward without rewriting history, so a NAV struck last year still resolves codes as they stood on that date. This is the temporal backbone of reference-data versioning.

Q: Why can't you simply overwrite a code-list row when a value changes?
A: Overwriting destroys history: any record that referenced the old value, and any report run before the change, can no longer be reproduced or explained. Regulated fund administration requires you to reconstruct exactly what a code meant on a given dealing date. Effective-dated versioning keeps the old row valid for its period and opens a new row, preserving an auditable timeline.

Q: What is the relationship between effective-dated reference data and SCD Type 2?
A: They are the same pattern applied to different data: SCD2 keeps historical versions of a dimension row by closing the old version and inserting a new one with validity timestamps, and effective-dated code lists do exactly that for reference values. The lesson explicitly frames effective dating as "the Phase-1 SCD2 discipline applied to reference data," so the mechanics you learned for slowly-changing dimensions are reused wholesale.

Q: In an SCD2 reference table, what do the validity columns typically hold for the current version of a code?
A: The current (active) version has a populated effective-from date and an open-ended effective-to — often `NULL` or a sentinel like `9999-12-31`. When the code changes, that row's effective-to is set to the change timestamp and a new row is inserted with a fresh effective-from. Querying "as of" a date then selects the row whose validity window contains that date.

Q: What dbt feature implements SCD Type 2 out of the box, and what is it suited to?
A: dbt snapshots implement SCD2: each run detects changed rows and records their history. They are suited to capturing how a source table mutates over time when the source itself only holds the current state. For reference data this lets you snapshot an incoming ISO list and retain every prior version automatically.

Q: What metadata columns does a dbt snapshot add to track history?
A: dbt adds `dbt_valid_from` and `dbt_valid_to` to bound each version's validity window, `dbt_scd_id` as a unique surrogate key per snapshot row, and `dbt_updated_at` recording the source `updated_at` at insertion. Current rows have `dbt_valid_to` set to `NULL` by default. These are the columns that turn a flat extract into a queryable history.

Q: What are the two dbt snapshot strategies, and when do you use each?
A: The `timestamp` strategy uses an `updated_at` column on the source to detect change and is recommended because it tracks a single column and tolerates schema evolution. The `check` strategy compares a named list of columns via `check_cols` (or `check_cols: 'all'`) and is used when the source has no reliable update timestamp. For an ISO code list with no per-row timestamp, the `check` strategy on the value columns is the practical choice.

Q: In a dbt snapshot config, what does `unique_key` specify and why does it matter?
A: `unique_key` names the column (or expression) that identifies a logical record across versions — for a currency list that would be the currency code. dbt uses it to decide whether an incoming row is a new entity or a changed version of an existing one. If it is non-unique or wrong, history splits incorrectly and you get duplicate or orphaned versions.

Q: What changed about hard-delete handling in recent dbt snapshots, replacing `invalidate_hard_deletes`?
A: Newer dbt versions (1.9+) introduced the `hard_deletes` config with options `ignore`, `invalidate`, and `new_record`, superseding the older boolean `invalidate_hard_deletes`. With `hard_deletes: new_record` dbt inserts a closing row flagged via a `dbt_is_deleted` meta column instead of just setting `dbt_valid_to`. This matters for reference data because a retired currency code should be recorded as deleted-as-of a date, not silently dropped.

Q: What is `dbt_valid_to_current` and why might you set it for a code list?
A: `dbt_valid_to_current` (dbt Core 1.9+) lets you store a concrete sentinel such as `'9999-12-31'` in `dbt_valid_to` for current rows instead of `NULL`. This makes "as-of" range queries cleaner because `between valid_from and valid_to` works without special-casing `NULL`. For a code list queried by effective date, a sentinel avoids `NULL`-handling bugs in joins.

Q: What is a dbt seed, and why is it appropriate for governed code lists?
A: A dbt seed is a CSV file checked into the dbt project that `dbt seed` loads into the warehouse as a table. Because it is version-controlled and code-reviewed, every change to a small static list goes through a pull request, giving you who/when/why provenance for free. That governance trail is exactly what reference data needs, which is why the lesson pairs dbt seeds with a governed Postgres schema.

Q: What command loads dbt seeds, and how do you load just one?
A: `dbt seed` loads all seeds; `dbt seed --select country_codes` loads only the named seed. After changing a seed's columns or types you run `dbt seed --full-refresh` to drop and rebuild the table. The resulting table is named after the CSV file and is referenced downstream with `{{ ref('country_codes') }}`.

Q: Where do dbt seed CSV files live, and how do you change that location?
A: By default they live in the `seeds/` subdirectory of the dbt project (for example `seeds/country_codes.csv`). You can override the location by setting `seed-paths: ["custom_seeds"]` in `dbt_project.yml`. Keeping them in the repo is what makes seed changes reviewable in version control.

Q: How do you control the column data types of a dbt seed?
A: Use the `column_types` config (often written `+column_types` under the seed in `dbt_project.yml`) to pin a column's type, for example forcing a zip or numeric currency code to `varchar(5)` so leading zeros survive. Without this, dbt infers types and may strip leading zeros or mis-type a code. Pinning types is essential for codes like ISO 4217 numeric `004` that must stay strings.

Q: What are dbt seeds explicitly NOT recommended for?
A: Seeds are not for loading large files, raw exported data dumps, or production data containing sensitive PII, because they are stored in the repo and are not performant at scale. They are designed for small, static, infrequently changing lists — precisely the shape of reference data. A 200-row currency list is ideal; a million-row trade extract is not.

Q: What is a "distribution pattern" for reference data, and which one does the lesson favour?
A: A distribution pattern is how the single source of truth gets the code list to all consumers. The lesson favours publish-subscribe from one source of truth: the authoritative list is published once and every consumer subscribes to that feed rather than each maintaining a private copy. This guarantees that silver, gold, and the EMT feed all read identical codes.

Q: Why is publish-subscribe preferred over each system holding its own copy of a code list?
A: Independent copies inevitably drift — one system updates a renamed country a week late, another never does — so the same entity ends up mapped differently across the estate. A single published source with subscribers means a change propagates once and consistently, eliminating the "country maps three ways" failure the lesson warns about. It also localises the audit trail to one place.

Q: How do you demonstrate the "Done when" criterion that a code list is consumed everywhere from one place?
A: You show lineage from the single governed code-list table to every downstream model that joins it — for example via dbt's `ref()` graph or a catalog's lineage view — and confirm no model defines its own competing copy. If the lineage graph fans out from exactly one node, the single-source property holds. If two unrelated tables both define currencies, you have failed the criterion.

Q: What is a mapping table in reference-data management?
A: A mapping table reconciles an external party's codes to your canonical codes — for example translating a market-data vendor's proprietary country abbreviations to ISO 3166 codes. It records the correspondence row by row (vendor value → canonical value) so inbound data can be normalised on ingest. It is itself reference data and should be effective-dated and governed like any other.

Q: Why are mapping tables necessary even when an ISO standard exists?
A: Vendors and counterparties frequently send non-standard or proprietary codes — a custodian might send `UK` where ISO 3166 uses `GB` — so you cannot join their feed to your canonical lists directly. The mapping table is the controlled translation layer that absorbs that variation in one auditable place. Without it, every pipeline reinvents ad-hoc translations that disagree.

Q: In the Phase-6 lab, where is the vendor-country-to-ISO-3166 mapping wired in, and why there?
A: It is wired into the silver layer, the standardisation/cleansing tier, so that raw vendor codes are translated to canonical ISO codes as data is conformed and before it reaches gold. Putting it in silver means every gold consumer downstream already sees canonical codes. Translating later, in gold or in reports, would let inconsistent codes leak past the conformance boundary.

Q: A custodian feed sends country `UK`, but your gold model joins on ISO 3166 `GB` and rows drop. What is the first thing to check?
A: Check the vendor-to-ISO mapping table: `UK` is not a valid ISO 3166 alpha-2 code (the correct code is `GB`), so the mapping must contain a `UK` → `GB` entry. If the entry is missing, the silver layer leaves `UK` unmapped and the gold join finds no match. The fix is a governed addition to the mapping table, not a hard-coded patch in the model.

Q: How should a mapping table handle a vendor code that has no valid canonical equivalent?
A: It should fail loudly rather than silently — typically by routing the unmapped value to a quarantine or exception path and raising a data-quality alert, so a human decides whether to add a mapping or reject the record. Silently coalescing to a default code corrupts downstream aggregates. Reference-data hygiene means unknown codes are surfaced, not hidden.

Q: Who maintains ISO 4217 currency codes, and how are changes communicated?
A: SIX (the financial-information arm, on behalf of the Swiss standards body SNV) is the maintenance agency for ISO 4217. Changes are issued as numbered amendments published on the SIX site, and the code list is updated continuously as economic and political changes require. You track the amendment number as the version when you ingest the list.

Q: What is the maintenance cadence of ISO 4217 versus ISO 10383?
A: ISO 4217 is updated as needed via numbered amendments rather than on a fixed calendar — changes happen when a currency is introduced, renamed, or withdrawn. ISO 10383 (MIC) by contrast follows a fixed monthly cycle. Knowing the cadence tells you how often to re-ingest and re-snapshot each list.

Q: Describe the publication and effective-date cycle for the ISO 10383 MIC list.
A: The MIC list is published on the second Monday of the month (or the next business day if that is a holiday), and the modifications become effective on the fourth Monday of the month. Requests must reach the registration authority by the first Monday to make that month's publication. This fixed cadence lets you schedule a monthly snapshot and align effective dates precisely.

Q: Who is the registration authority for ISO 10383 MICs?
A: SWIFT (S.W.I.F.T. SC, based in La Hulpe, Belgium) is the ISO 10383 registration authority and publishes the official MIC list. They accept change requests and maintain the records on the published monthly schedule. Because SWIFT is also central to fund settlement messaging, this ties MIC governance into infrastructure you already touch.

Q: What is the difference between an operating MIC and a segment MIC in ISO 10383?
A: An operating MIC identifies the market operator or exchange as a whole (for example the parent venue), while a segment MIC identifies a specific trading segment or sub-market operated under it. A given operating MIC can have several segment MICs beneath it, and each segment MIC maps to exactly one operating MIC. Mishandling this hierarchy causes venue-level reporting to double-count or mis-attribute trades.

Q: How stable is ISO 3166, and what does that imply for re-ingestion?
A: ISO 3166 country codes are very stable; the maintenance agency changes them only when genuinely necessary, such as a country renaming or splitting. This means you re-ingest less aggressively than for MICs, but you must still subscribe to change notifications because a rare change (for example a new alpha-2 code) ripples widely. You can sign up for notifications on the ISO Online Browsing Platform.

Q: An ISO 3166 alpha-2 code is reassigned years after a country dissolved. Why is effective dating critical here?
A: Because a code that meant one country in 2010 might be reused for a different entity later, a NAV or report from 2010 must resolve the code against the version valid then, not today's meaning. Effective-dated reference data binds each historical record to the code definition in force on its date. Without it, reprocessing old data silently relabels positions to the wrong country.

Q: Why does ISO 4217 alphabetic currency code structure relate to ISO 3166?
A: An ISO 4217 alphabetic code is generally formed from the ISO 3166-1 alpha-2 country code plus a letter for the currency — for example `US` + `D` gives `USD`. This linkage means the two standards are interdependent, and a country-code change can imply a currency-code consideration. It also explains why governance treats them as a coupled family rather than isolated lists.

Q: What columns would you put in a governed code-list schema to meet the audit requirement?
A: At minimum the code value, its description, an effective-from and effective-to (for versioning), an `approved_by` field, a `source`/version field recording where the value came from, and timestamps. Together these let you trace any change to who approved it, when it took effect, and which authoritative source it derived from. That triple of who/when/why is the lesson's audit criterion.

Q: What does recording the `source` and version of an ingested ISO list achieve?
A: It pins each loaded value to a concrete published artefact — for example "ISO 4217 amendment 180" or a dated MIC publication — so you can prove the code list matched the authority at the time. Without it you cannot defend a value's provenance to an auditor or reconcile a discrepancy against the source. Provenance is part of the data, not metadata you can reconstruct later.

Q: What is meant by "approved-by" in a governed code list, and why store it?
A: `approved_by` records the data steward who authorised a particular code-list change, embedding the approval into the row rather than leaving it in a separate ticket. It satisfies the governance requirement to trace every change to a responsible person. In a regulated fund administrator, "who approved this currency change" is a question an auditor will literally ask.

Q: How would you make a "controlled change" to a code list and prove history is preserved?
A: You apply the change through the governed process — edit the seed CSV in a reviewed pull request or run an approved update — which closes the prior version (sets its effective-to) and opens a new effective-dated row with a new approver and source. You then query "as of" a past date and confirm the old value still resolves, while "as of today" returns the new one. Both versions coexisting is the proof.

Q: A code list was changed but an old NAV report now shows the new code. What governance failure does this indicate?
A: It indicates the change overwrote history instead of being effective-dated — the report resolved the code against the current row rather than the version valid on the report's date. The fix is to retrofit effective dating so each version retains its validity window. This is exactly the failure mode the SCD2/effective-dating discipline is designed to prevent.

Q: Why is reference data described as "governance's unglamorous core"?
A: Because it is unexciting, low-volume plumbing that attracts little attention, yet it underpins correctness everywhere — every join to a currency, country, or venue depends on it. Teams under-invest in it precisely because it looks trivial, which is how silent divergence creeps in. The lesson stresses it to counter that neglect.

Q: What is a MIC and where does it appear in fund data flows?
A: A MIC (Market Identifier Code, ISO 10383) is a four-character code identifying the exchange or trading venue where an instrument trades. It appears wherever you record where a security is listed or traded — for example in EMT/EPT instrument records and in trade and position feeds. Getting the operating-versus-segment distinction right matters for accurate venue-level regulatory reporting.

Q: What is an EMT/EPT file, and how does reference data feed into it?
A: The European MiFID Template (EMT) and European PRIIPs Template (EPT) are standardised fund-data exchange files carrying instrument and cost/risk information for distribution. They embed reference codes — currencies, MICs, country codes, ISINs — so any divergence in your code lists shows up as inconsistent EMT/EPT output. Governed single-source reference data is what keeps those templates internally consistent.

Q: Why must currency codes in an EMT file resolve to the same canonical list as the silver and gold layers?
A: If the EMT generator reads a different currency copy than silver/gold, a sub-fund could be reported with a currency that disagrees with its own NAV, breaching the single-source principle and producing a regulatory file that contradicts internal records. Publish-subscribe distribution from one source prevents that. The EMT feed is just another subscriber to the authoritative list.

Q: What is an ISIN, and is the ISIN itself reference data?
A: An ISIN (ISO 6166) is the 12-character International Securities Identification Number for an instrument. The ISIN identifies a specific security (closer to master/instrument data), but the structure of an ISIN embeds reference data: its first two characters are the ISO 3166 country prefix. So validating an ISIN's prefix depends on your governed country-code list.

Q: How does the country prefix of an ISIN tie back to reference-data governance?
A: The leading two characters of an ISIN are an ISO 3166-1 alpha-2 code (or special codes like `XS` for international issues), so checking an ISIN's validity requires resolving that prefix against your governed country list. If your country list is stale or divergent, ISIN validation gives wrong results. This is a concrete example of reference data underpinning instrument-level checks.

Q: What is an LEI and what role does reference data play around it?
A: An LEI (ISO 17442) is the 20-character Legal Entity Identifier for a legal entity such as a fund or management company. While the LEI maps to entity master data, the surrounding attributes — the entity's country (ISO 3166), legal form, and jurisdiction codes — are reference data that must be governed consistently. GLEIF publishes LEI data, making it another external source you ingest and version.

Q: How does the publish-subscribe pattern help compliance and regulatory reporting specifically?
A: Compliance reports (UCITS/AIFMD filings, EMT/EPT, transaction reporting) must agree with each other and with internal books. Sourcing every code from one published authority guarantees that a currency or country in a regulatory file is identical to the one in the underlying NAV. It removes "which copy was right" disputes when a regulator queries a figure.

Q: Why is a centralised, governed code list relevant under DORA?
A: DORA emphasises operational resilience and the integrity of critical data flows in financial entities. A single governed reference-data source with auditable versioning reduces the operational risk that a fragmented, manually-maintained code list silently corrupts downstream processing. It also makes incident reconstruction feasible because you can replay what codes were in force.

Q: In a Luxembourg fund-administration estate, why is consistent country/currency reference data especially load-bearing?
A: Luxembourg funds are cross-border by nature — multi-currency sub-funds distributed across many jurisdictions — so country and currency codes thread through every NAV, distribution file, and regulatory submission. A single divergent code list multiplies into errors across dozens of share classes and markets. Centralised governance contains that combinatorial risk.

Q: What is a CFI code, and is it reference or master data?
A: A CFI (Classification of Financial Instruments, ISO 10962) is a six-character code describing an instrument's type and attributes (category, group, and features). The CFI value attached to a specific instrument is instrument data, but the permissible CFI letters and their meanings are reference data — a governed code list you maintain against the standard. The lesson lists CFI codes as an example of small reference data with large blast radius.

Q: Why might two systems map "the same country" three different ways without RDM?
A: One system might use ISO 3166 alpha-2 (`GB`), another alpha-3 (`GBR`), and a vendor feed a proprietary string (`UK` or `England`), each maintained independently. Without a single governed source and mapping tables, these never converge and joins across systems silently mismatch. The DMBOK pattern of one source plus governed mappings is the cure.

Q: What is the difference between ISO 3166-1 alpha-2 and alpha-3 codes, and why does it matter for mapping?
A: Alpha-2 is the two-letter code (`GB`) and alpha-3 is the three-letter code (`GBR`) for the same country. Different feeds use different representations, so your mapping tables and canonical choice must be explicit about which form is canonical. Mixing them in a join silently fails because `GB` never equals `GBR`.

Q: How does versioning let a code list "change without rewriting history"?
A: Versioning closes the validity window of the outgoing value and opens a new window for the incoming value, so both coexist with non-overlapping effective periods. History is read by selecting the version whose window contains the date of interest. Nothing is deleted, so prior records remain reproducible exactly as they were.

Q: What "as-of" query pattern retrieves the correct code version for a historical date?
A: You filter the effective-dated table to the row where the query date falls between `valid_from` and `valid_to` (treating an open `valid_to` of `NULL` or a sentinel as "still current"). For example `where :as_of_date >= valid_from and (:as_of_date < valid_to or valid_to is null)`. This returns exactly one version per code for the date.

Q: Why might you prefer a sentinel date like `9999-12-31` over `NULL` for the open end of a validity window?
A: A sentinel makes range predicates uniform — `between valid_from and valid_to` works for every row without special-casing `NULL` — which simplifies as-of joins and avoids the three-valued-logic surprises `NULL` introduces. dbt supports this via `dbt_valid_to_current`. The trade-off is that the sentinel must be chosen so it never collides with a real date.

Q: What is the risk of overlapping validity windows in an effective-dated code list?
A: Overlapping windows mean an as-of query can return more than one version for the same code on the same date, breaking the uniqueness an effective-dated lookup depends on and corrupting joins. It usually signals a bug in the update logic that opened a new version without closing the old one. A data-quality test asserting no overlaps per key guards against it.

Q: What data-quality test would you add to guarantee an effective-dated code list has no gaps or overlaps?
A: A test that, for each code key, asserts the validity windows are contiguous and non-overlapping — each version's `valid_from` equals the prior version's `valid_to`, with no gaps and exactly one open-ended current row. In dbt you might implement this as a custom singular test or a dbt-utils-style assertion. It catches both overlap (double-mapping) and gap (a date with no valid code).

Q: How does Collibra's reference-data management relate to the FOSS build in this lesson?
A: Collibra RDM is the commercial benchmark — a governed platform with built-in versioning, distribution, and mapping workflows — that the lesson asks you to evaluate against your own FOSS implementation of dbt seeds plus a governed Postgres schema. The point is to understand the pattern Collibra embodies so your hand-built version delivers the same governance guarantees. Knowing the corp tool lets you articulate the trade-offs.

Q: What governance capabilities does a commercial RDM tool provide that you must replicate manually with dbt seeds plus Postgres?
A: Approval workflows, role-based stewardship, effective-dated versioning, mapping management, and a distribution/subscription mechanism. With FOSS you replicate these via pull-request review (approval), `approved_by`/`source` columns (versioning and provenance), snapshots (history), mapping tables, and `ref()`-based single-source consumption (distribution). The lesson's value is seeing that the capabilities, not the tool, are what matter.

Q: Why is a code-review/pull-request workflow on dbt seeds a legitimate substitute for a formal approval process?
A: Because each change to the CSV is captured as a reviewed, attributable commit with a reviewer and timestamp, it produces the same who/when/why audit trail a formal RDM approval step would. Git history is immutable and signed-off, satisfying the traceability requirement. The discipline is procedural, not tool-specific.

Q: What is the difference between reference data and a business glossary?
A: Reference data is the permissible code values themselves (the list of currencies), whereas a business glossary defines the meaning of business terms (what "NAV" means). They are complementary governance artefacts: the glossary tells you what a concept means, reference data tells you the valid values a coded field may take. A column may link to a glossary term and draw its values from a reference list.

Q: Why is a numeric currency code like ISO 4217 `004` (Afghani) a trap for naive ingestion?
A: Numeric currency codes are zero-padded three-digit strings, but a CSV loader or warehouse may infer them as integers and strip the leading zeros, turning `004` into `4`. That breaks any join expecting the three-character string form. Pinning the column to a string type (for example via dbt seed `column_types`) prevents the silent corruption.

Q: A dbt seed column of ISO 4217 numeric codes loses its leading zeros after `dbt seed`. What is the fix?
A: Set the column's type explicitly in `column_types` (for example `+column_types: {num_code: varchar(3)}`) so dbt stops inferring it as an integer, then run `dbt seed --full-refresh` to rebuild with the correct type. The leading zeros are preserved because the value is now stored as text. Type inference, not the data, was the culprit.

Q: How would you stage an ISO list ingestion so its version is recorded automatically?
A: Load the file as a seed or external load, attach a `source`/version column populated from the published artefact identifier (amendment number or publication date) at load time, and snapshot the result so history accrues. The version travels with every row rather than living in a separate note. Re-ingesting a new publication then produces a new snapshotted version with its own identifier.

Q: What is the danger of letting downstream models hard-code reference values instead of joining the governed list?
A: Hard-coded values (a `case` statement listing currencies inline) bypass the single source, so when the list changes the model silently keeps the stale set, and you lose all audit trail of who put them there. It reintroduces the very divergence RDM exists to prevent. Every reference value should be a join to the governed table, never a literal in code.

Q: How does the single-source principle interact with dbt's `ref()` lineage?
A: Because every consuming model references the governed code-list table via `ref()`, dbt's lineage graph shows a single upstream node fanning out to all consumers, which is both the enforcement mechanism and the evidence for the "consumed from one place" criterion. If a second, competing code table appears in the graph, the single-source property is visibly broken. Lineage thus doubles as a governance audit.

Q: When would a publish-subscribe distribution use an actual message bus versus a shared table?
A: For loosely-coupled or cross-system distribution where consumers are independent services, a message bus (or an API/feed) lets subscribers pull updates on change, decoupling them from your warehouse. Within a single dbt-governed warehouse, a shared governed table consumed via `ref()` is the simpler realisation of the same pattern. The lesson's lab uses the table form; the bus form is the enterprise generalisation.

Q: Why should mapping tables themselves be effective-dated?
A: Because a vendor can change which proprietary code it sends for an entity over time, the correct vendor→canonical translation can differ by date, and you must resolve historical feeds with the mapping that was valid then. Effective-dating the mapping preserves that history just like the code lists themselves. A mapping is reference data and deserves identical governance.

Q: A vendor renames its internal code for a venue mid-year. How does an effective-dated mapping table handle reprocessing old files?
A: The mapping retains the old vendor-code→canonical entry with its validity window closed at the rename date and the new entry opened from that date, so reprocessing a file from before the rename resolves through the old mapping and a recent file through the new one. Both translate to the same canonical venue if the venue is unchanged. History stays correct without manual intervention.

Q: What is the relationship between reference data and the silver layer in a medallion architecture?
A: The silver layer is where raw bronze data is conformed and standardised, so it is the natural place to apply reference-data mappings — translating vendor codes to canonical codes — so that everything reaching gold is already canonical. Reference tables sit alongside silver as governed inputs to that conformance. Doing it in silver enforces canonical codes once, centrally.

Q: Why is it a mistake to apply reference-data mapping in the gold layer or in BI tools instead of silver?
A: Mapping late means each gold model or report repeats the translation, they can disagree, and non-canonical codes have already leaked past the conformance boundary into multiple consumers. Centralising the mapping in silver guarantees every gold model inherits canonical codes consistently. Late mapping reintroduces the divergence RDM exists to kill.

Q: How do you record "why" a code-list change was made, beyond who and when?
A: Capture the rationale in the controlled-change artefact — the pull-request description, a change-reason column, or a linked ticket referencing the source amendment (for example "ISO 4217 amendment 180 added new code"). Tying the change to the authoritative publication gives a defensible reason. The "why" is as auditable as the "who" and "when" when it lives with the change.

Q: How would you detect that a downstream report is using a stale copy of a code list?
A: Trace the report's lineage to its code source; if it resolves to a private table rather than the governed seed, or if the governed list has a newer effective version the report ignores, you have a stale copy. A reconciliation test comparing distinct codes used downstream against the current governed list also surfaces drift. Lineage plus a coverage test together catch it.

Q: Why is "trace any code-list change to who/when/why" listed as a Done-when criterion rather than a nice-to-have?
A: Because in regulated fund administration an auditor or regulator can require you to justify any value used in a NAV or filing, and inability to trace a code change is itself a control failure. The criterion forces the governance columns (`approved_by`, `source`, effective dates) to be present and populated, not optional. It is the minimum evidence of a managed reference-data process.

Q: How does effective dating support reproducing a historical NAV exactly?
A: To reproduce a NAV struck on a past dealing date you must resolve every code — currency, country, venue — as it was defined on that date, which effective-dated reference data provides via as-of lookups. Without it, today's code definitions would silently re-label or re-price the historical positions. Effective dating is what makes the NAV deterministically reproducible.

Q: What is the consequence of a currency code being withdrawn (for example a redenomination) for an effective-dated list?
A: The withdrawn code's row gets its `valid_to` closed at the withdrawal date and, if appropriate, a successor code's row is opened, so historical records still resolve the old code while new records use the successor. Effective dating ensures the transition is bitemporally clean. With `dbt_valid_to: new_record`/hard-delete handling you can also flag the retirement explicitly.

Q: How do you decide what is canonical when two ISO representations both seem authoritative?
A: You make an explicit governance decision — for example "canonical country code is ISO 3166-1 alpha-2" — and document it, then all mappings translate other forms into that canonical choice. The point is not which is objectively best but that exactly one is declared and enforced everywhere. Ambiguity, not the specific choice, is what causes joins to fail.

Q: Why is it dangerous to let analysts add a one-off currency code directly in a report?
A: It bypasses the governed source, so the new code has no `approved_by`, no `source`, no effective date, and no propagation to other consumers — it exists only in that report and silently diverges from everyone else. It defeats the single-source and traceability guarantees simultaneously. Reference values must enter through the governed process, never ad hoc.

Q: How would you reconcile a vendor feed whose country codes partially match ISO 3166 and partially don't?
A: Join the feed through the governed mapping table, route matched codes to canonical ISO values, and quarantine the unmatched ones to an exceptions table with a data-quality alert so a steward can add a mapping or reject the row. You never silently default unmatched codes. Over time the mapping table grows to cover the vendor's full code space.

Q: What is the bitemporal idea, and why might reference data eventually need it?
A: Bitemporal modelling tracks two timelines: when a fact was true in the real world (valid time) and when the system knew it (transaction time). Effective dating gives you valid time; adding transaction time lets you reconstruct what the system believed at a past moment, which matters when you correct a back-dated code error. Full reference-data governance may extend SCD2 effective dating into bitemporal records for that reason.

Q: A back-dated correction to a code list arrives — the value was wrong for a past period. How does effective dating accommodate it?
A: You adjust the validity windows so the corrected value covers the affected past period, ideally retaining the superseded (erroneous) version under transaction-time history so the correction itself is auditable. Pure valid-time effective dating shows the corrected truth; bitemporal records also show that a correction occurred. Either way nothing is destroyed.

Q: Why does the lesson cite DAMA-DMBOK chapter 10 specifically for reference data?
A: DMBOK chapter 10 covers reference and master data management, and the reference-data half lays out exactly the versioning, distribution, and mapping patterns this lesson teaches. It is the authoritative framework the lesson builds its FOSS implementation against. Citing it grounds the hand-built Postgres-and-dbt approach in recognised governance theory.

Q: How is "one source of truth" verified in practice during the lab?
A: You confirm there is a single governed code-list table per domain (currencies, countries, MICs), that every consumer reaches it via `ref()` with no competing copies, and that the lineage graph fans out from that one node. You then make a change in that one place and observe it propagate to all consumers. A single point of change with universal propagation is the verification.

Q: What is the difference between a code list and a hierarchy (or taxonomy) in reference data?
A: A flat code list is an unordered set of permissible values (currencies), whereas a hierarchy or taxonomy adds parent-child structure (for example a fund-classification tree, or operating-MIC to segment-MIC relationships). Hierarchies need extra governance for the relationships, not just the values. The MIC operating/segment relationship is a concrete hierarchical reference structure.

Q: Why must the operating-MIC/segment-MIC hierarchy be modelled, not just stored as two flat columns?
A: Because regulatory venue reporting depends on rolling segment-level activity up to the correct operating MIC, and a flat representation without enforced parent-child integrity lets a segment point to no operator or the wrong one. Modelling the hierarchy with a foreign-key relationship enforces that each segment maps to exactly one operating MIC. That integrity is what makes venue aggregation correct.

Q: How do you handle a brand-new MIC that appears in this month's ISO 10383 publication?
A: Because the publication is on the second Monday and effective on the fourth Monday, you ingest the new list, snapshot it so the new MIC appears as a new effective-dated row from its effective date, and record the publication date as its source/version. Consumers subscribing to the governed table then see it automatically. Aligning the row's effective date to the standard's fourth-Monday effective date keeps as-of lookups accurate.

Q: What test confirms a mapping table covers the full domain of a vendor's codes?
A: An anti-join (or dbt `relationships`/coverage test) from the vendor feed's distinct codes against the mapping table that returns zero unmapped values confirms full coverage; any returned rows are codes needing a mapping. Running it on each load catches new vendor codes early. It turns "did we map everything" into an automated assertion.

Q: Why is the blast radius of a reference-data error larger than that of a single transactional error?
A: A transactional error usually affects one record, but a reference-data error affects every record that joins to the corrupted code — potentially every position in a sub-fund or every fund in a currency. The fan-out is structural: reference data is multiplied across the estate by joins. That asymmetry is why governance attention is concentrated here despite the tiny row counts.

Q: How does versioning reference data help during a regulatory examination?
A: It lets you demonstrate exactly which code values were in force on any examined date, who approved each change, and from which authoritative source it came, answering "prove this currency code was correct on this dealing date" definitively. Without versioning you can only show the current state and must reconstruct the past from memory. Versioning turns governance assertions into queryable evidence.

Q: What is the role of a data steward in reference-data management?
A: The steward owns a code-list domain: approving changes, deciding the canonical representation, maintaining mappings, and being the named accountable party in `approved_by`. They are the human node in the governance process that the tooling records. Stewardship is what makes "who" in who/when/why a real, contactable person rather than an anonymous commit.

Q: How would you migrate an ad-hoc, copied-everywhere currency list into a single governed source?
A: Inventory every existing copy via lineage/search, build the governed effective-dated table seeded from the authoritative ISO list, repoint every consumer to it via `ref()` (replacing hard-coded literals), then add a test asserting no other currency definitions exist. The cut-over is complete when lineage shows a single source. Reconciling the copies' contents first surfaces the divergence you are eliminating.

Q: Why might `check_cols: 'all'` be risky for snapshotting an evolving ISO feed?
A: `check_cols: 'all'` flags a new version whenever any column differs, so a cosmetic change such as a reworded description or a new metadata column triggers spurious versions and schema churn. Restricting `check_cols` to the semantically meaningful value columns avoids versioning on noise. The timestamp strategy avoids this entirely when a reliable `updated_at` exists.

Q: A snapshot of a code list is creating a new version on every run even though nothing changed. What is the likely cause?
A: Likely a non-deterministic or churning column is included in the change detection — for example a load timestamp captured at ingest, or `check_cols: 'all'` picking up a volatile metadata field. Each run sees a "difference" and opens a spurious version. The fix is to exclude volatile columns from `check_cols` or use a stable `updated_at` with the timestamp strategy.

Q: How do you ensure an ingested ISO list's source artefact is itself archived for audit?
A: Store the original published file (the CSV or PDF amendment) in version control or an immutable object store alongside the load, and reference its identifier in the `source` column, so you can re-derive the loaded values from the artefact later. Keeping only the parsed rows is insufficient if a parsing bug is later discovered. The raw artefact plus the version identifier together form complete provenance.

Q: Why is reference-data governance a prerequisite for trustworthy data-quality rules?
A: Many DQ rules assert that a coded field holds a permissible value (a valid currency, country, or MIC), and that assertion is only as good as the governed reference list it checks against. A stale or divergent list makes the DQ rule pass on wrong data or fail on correct data. Governed reference data is the trusted vocabulary the rules depend on.

Q: What is the difference between a "code list changing" and a "code's meaning changing", and why does governance care about both?
A: A code list changes when entries are added or removed (a new currency appears); a code's meaning changes when an existing code is reassigned or redefined (an alpha-2 reused for a different country). Effective dating handles list membership over time, while meaning changes additionally demand that historical records resolve the old meaning. Governance must distinguish them because the second is more dangerous to reproducibility.

Q: How does a governed reference-data layer reduce reconciliation effort across NAV, EMT, and regulatory feeds?
A: Because all three feeds draw their codes from one published source, the codes are identical by construction, so reconciliation no longer has to explain code-level mismatches and can focus on genuine business discrepancies. The single-source pattern eliminates an entire class of false breaks. Less reconciliation noise means real exceptions surface faster.

Q: What is the final synthesis: how do versioning, single-source distribution, and mapping together deliver reference-data governance?
A: Single-source distribution (publish-subscribe via one governed table) guarantees everyone reads the same codes; versioning/effective dating guarantees every historical record resolves the code definition valid at its time; and mapping tables guarantee external feeds are reconciled to your canonical codes on the way in. Together they ensure the same "country" or "currency" never maps three ways and any change is fully traceable to who, when, and why. That triad is the operational definition of reference-data management the lesson teaches.

Q: From dbt Core 1.9, how can you define a snapshot without a `.sql` file and a config block, and why is that cleaner for code lists?
A: dbt 1.9 lets you define snapshots entirely in a properties YAML file under `snapshots-paths` (default `snapshots/`), using a `snapshots:` block with `relation`, `config`, `unique_key`, and `strategy` keys instead of the legacy `{% snapshot %}` Jinja block in a `.sql` file. The YAML form keeps the snapshot's configuration declarative and reviewable in one place, which suits a governed code list whose every change should be diff-able in a pull request. The old `.sql` block still works but is now the legacy method.

Q: In dbt 1.9+, what is the `snapshot_meta_column_names` config and when would you use it for an ISO code list?
A: `snapshot_meta_column_names` lets you rename the snapshot meta columns — mapping `dbt_valid_from`, `dbt_valid_to`, `dbt_scd_id`, `dbt_updated_at`, and `dbt_is_deleted` to names of your choosing such as `valid_from`/`valid_to`. You use it when your governed schema or auditors mandate specific column names so the snapshot output conforms to a house standard without a downstream rename. A known gotcha is that uppercase custom names have triggered compilation errors on some adapters, so keep the names lowercase.

Q: Why did dbt make `target_schema` optional (and effectively unused) for snapshots in 1.9, and what replaces it?
A: Hard-coding `target_schema` forced every developer's snapshot into one shared schema, breaking environment isolation between dev and prod runs. From 1.9 `target_schema` is optional and no longer honoured for routing; you use the `schema` config instead, which respects the `generate_schema_name` macro so dev runs land in a developer-prefixed schema. For a code-list snapshot this means dev experiments no longer overwrite the production reference history.

Q: What is the ISO 4217 "minor unit" (exponent), and why must a governed currency list carry it?
A: The minor unit is the number of decimal places a currency subdivides into — `2` for `USD` (cents), `0` for `JPY` (no sub-unit), and `3` for `BHD` (the Bahraini dinar's fils). NAV and amount fields must round and store monetary values to the currency's exponent, so the governed list must expose it as a column rather than assuming two decimals everywhere. Assuming `2` for `JPY` silently inflates yen amounts by a factor of 100 and mis-states a yen share class's NAV.

Q: A `JPY` sub-fund's NAV per share is being stored and displayed with two decimal places. How does the currency reference list catch this?
A: The governed ISO 4217 list carries `JPY`'s minor unit as `0`, so a data-quality rule joining the amount's scale to the currency's exponent flags the two-decimal storage as non-conformant. The mechanism is that rounding logic should read the exponent from reference data, not hard-code `2`. Without the exponent in the list, the over-precision passes silently and a regulatory file shows yen with phantom sub-units.

Q: What is ISO 4217-3, and how does it differ from the alphabetic and numeric code lists?
A: ISO 4217 is published in parts: the alphabetic codes (`USD`), the numeric codes (`840`), and the funds/historic data, with the standard also defining the minor-unit exponents. The historic-currency list (codes withdrawn, like `DEM` for the Deutsche Mark) is what lets effective-dated history resolve a redenominated currency on an old dealing date. Treating only the current alphabetic list as canonical loses the ability to reprocess pre-euro records correctly.

Q: How would you structure the `unique_key` for a MIC snapshot given the operating/segment hierarchy?
A: The `unique_key` is the MIC itself (the four-character code), because each MIC — whether operating or segment — is globally unique, so one key column suffices for change detection. The operating/segment relationship is then a separate `operating_mic` attribute column on each row, snapshotted alongside, rather than part of the key. Using a composite key of operator-plus-segment would wrongly split history when a segment is re-parented.

Q: A segment MIC is re-parented to a different operating MIC in a monthly ISO 10383 publication. With a `timestamp`/`check` snapshot keyed on the MIC, what happens?
A: The snapshot keyed on the segment MIC detects that its `operating_mic` column changed, closes the prior version's `dbt_valid_to`, and opens a new version with the new parent, preserving both. As-of queries before the change still roll the segment up to the old operator and after the change to the new one. This is exactly why the operating MIC must be a snapshotted attribute, not part of the key.

Q: In an ISO 10383 record, what does a MIC's status of `ACTIVE`, `EXPIRED`, or `UPDATED` tell your ingestion?
A: The status field marks whether a MIC is currently usable (`ACTIVE`), retired (`EXPIRED`), or amended (`UPDATED`), so your snapshot should treat an `EXPIRED` MIC as effective-dated-closed rather than deleting it. Historical trades booked against a now-expired venue must still resolve, so you keep the row with its validity window. Dropping expired MICs would break reprocessing of any position that traded there before retirement.

Q: How does the ISO 10383 file's `OPRT/SGMT` indicator help you build the hierarchy automatically?
A: Each MIC row carries an indicator distinguishing an operating MIC (`OPRT`) from a segment MIC (`SGMT`) plus an `OPERATING MIC` column naming its parent; for an operating MIC the operating-MIC field equals its own code. You use this to populate the foreign-key relationship and assert that every `SGMT` row points to an existing `OPRT` row. The self-referential `OPRT` row whose parent is itself is the root you anchor the integrity test on.

Q: What is the 20-character structure of an LEI, and which parts are reference-data-governed versus opaque?
A: An LEI is 20 alphanumeric characters: characters 1-4 are the LOU (issuer) prefix, 5-6 are reserved `00`, 7-18 are the opaque entity-specific portion, and 19-20 are check digits. The LOU prefix is governed reference data (you can resolve which issuer it denotes), while the middle portion is deliberately meaningless, and the check digits are computational. So validating an LEI means checking the check digits, not parsing the body for meaning.

Q: How do you validate an LEI's check digits, and what standard governs it?
A: The last two characters are computed with the `MOD 97-10` algorithm from `ISO/IEC 7064`: convert all letters to numbers (`A`=10 … `Z`=35), treat the 20-character string as a large integer, and a valid LEI satisfies that integer `mod 97 == 1`. A failing remainder means the LEI is transcribed wrong or fabricated, so the check is a cheap front-line data-quality gate before you trust an entity feed. The same `MOD 97-10` scheme underlies IBAN validation.

Q: Why is GLEIF's published LEI data treated as an external reference source you snapshot, and what cadence?
A: GLEIF publishes the full LEI concatenated file (the "golden copy") daily, and individual LEI records change as entities update registration or lapse, so you ingest and snapshot it like any external list to retain history. Treating it as governed external data means an entity's country or legal-form attribute resolves to what GLEIF held on a given date. The daily cadence is far more frequent than ISO country codes, so snapshot volume and storage planning differ.

Q: What is an LEI's "registration status" of `LAPSED`, and why does effective dating matter for it?
A: `LAPSED` means the entity failed to renew its LEI annually, so the record is stale even though the code still exists; effective dating lets a historical filing resolve the entity's attributes as they stood while the LEI was `ISSUED`/active. You must not retroactively treat a now-lapsed LEI as invalid for past dates when it was valid. Snapshotting the status column captures the lapse transition as its own effective-dated version.

Q: How does the `XS` ISIN country prefix complicate ISIN-prefix validation against your ISO 3166 list?
A: `XS` is allocated for international securities (Eurobonds cleared through Euroclear/Clearstream) and is not an ISO 3166 country, so a naive validator that joins every ISIN's first two characters to the country list rejects valid `XS` ISINs. Your validation must allow the reserved `X`-prefixed codes (`XS`, plus `EU`, `Q*` ranges) as legitimate non-country prefixes. Hard-failing on `XS` would quarantine a large slice of fund holdings.

Q: An ISIN fails your check-digit validation but the prefix is a valid ISO 3166 code. Where is the likely error?
A: The body or the final Luhn check digit is wrong, not the country prefix, so the failure is transcription within the 9-character national security identifier or the computed 12th digit rather than a reference-data gap. ISINs use the Luhn (`mod 10`) algorithm on the first 11 characters to derive the check digit, so a single mistyped character fails it. The reference list governs only the prefix; the rest is structural validation.

Q: What is the difference between ISO 3166-1, ISO 3166-2, and ISO 3166-3, and which matters most for fund reference data?
A: ISO 3166-1 holds country codes (alpha-2/alpha-3/numeric), ISO 3166-2 holds subdivision (state/province) codes, and ISO 3166-3 holds formerly-used country codes for countries that have changed name or merged. For fund data the alpha-2 of 3166-1 is canonical for ISIN prefixes and jurisdictions, but 3166-3 is what lets you resolve a dissolved country's old code on a historical record. Confusing a 3166-2 subdivision code with a country code corrupts jurisdiction joins.

Q: How does ISO 3166-3 support reprocessing a NAV from before a country dissolved or renamed?
A: ISO 3166-3 retains the retired four-letter codes for entities like the former Yugoslavia or Serbia and Montenegro, so an effective-dated country list can map a pre-change record to the code that was valid then rather than to whatever code now occupies the alpha-2 slot. This prevents silent relabelling when an alpha-2 is later reassigned. The lesson's "code's meaning changing" risk is precisely the 3166-3 case.

Q: Write the SQL skeleton of an as-of join from a positions table to an effective-dated currency list.
A: `select p.*, c.minor_unit from positions p join ccy_scd c on p.ccy = c.code and p.dealing_date >= c.valid_from and (p.dealing_date < c.valid_to or c.valid_to is null)`, which binds each position to the currency version valid on its own dealing date. The half-open interval `>= valid_from` and `< valid_to` prevents a date on a boundary matching two versions. Replacing `c.valid_to is null` with a `9999-12-31` sentinel via `dbt_valid_to_current` lets you drop the `or` clause entirely.

Q: Why use a half-open interval (`>= valid_from`, `< valid_to`) rather than `between` for as-of lookups?
A: `between` is inclusive on both ends, so if one version's `valid_to` equals the next version's `valid_from` (the usual contiguous case), a date exactly on that boundary matches both versions and the join duplicates the row. The half-open `[valid_from, valid_to)` convention makes each instant belong to exactly one version. This is the single most common as-of-join bug and the reason snapshot tools store contiguous, non-overlapping windows.

Q: How would you write a dbt singular test asserting no overlapping validity windows per code?
A: A singular test SQL that self-joins the snapshot on the same `unique_key` and returns rows where two distinct versions' windows intersect — for example `where a.code = b.code and a.dbt_scd_id <> b.dbt_scd_id and a.valid_from < coalesce(b.valid_to, '9999-12-31') and coalesce(a.valid_to,'9999-12-31') > b.valid_from`. Any returned row is an overlap, so the test fails when versions double-cover a date. Placing it in `tests/` makes it run on every `dbt build`.

Q: How do you test for gaps (a date with no valid code) in an effective-dated list using window functions?
A: Order each code's versions by `valid_from` and compare each row's `valid_from` to the prior row's `valid_to` with `lag()`: `where valid_from <> lag(valid_to) over (partition by code order by valid_from)` flags any non-contiguous boundary. A returned row means a gap (or overlap) exists between consecutive versions. Combined with a check that each code has exactly one open-ended current row, this guarantees full temporal coverage.

Q: Why might you store both `valid_from` and a separate `recorded_at`/transaction-time column on a reference row?
A: `valid_from` is valid time — when the code's meaning took effect in the real world — while `recorded_at` is transaction time — when your system learned it, which differs for a back-dated correction loaded late. Storing both makes the table bitemporal so you can answer "what did we believe the code was on date X, as of the moment we ran last quarter's report". Pure valid-time storage cannot reconstruct a superseded belief, which an auditor investigating a restated NAV may demand.

Q: A regulator asks you to reproduce a report exactly as it was run on 2025-03-31, including a code value you have since corrected. What do you need?
A: You need bitemporal reference data: the as-of-valid-time gives the code's real-world meaning on the dealing date, but reproducing the report as originally run requires the transaction-time snapshot of what the system believed on 2025-03-31, before the later correction. Querying with both timelines returns the original (even if erroneous) value the report actually used. Valid-time-only effective dating would show the corrected value and fail to reproduce the original output.

Q: How do you load an ISO list into Postgres so leading zeros and exact strings survive, outside of dbt seeds?
A: Use `COPY ... FROM` (or `\copy` in `psql`) into a table whose columns are declared `text`/`varchar` up front, never `integer`, so `004` and `840` stay as strings; for example `\copy ccy(code, num_code, name) FROM 'iso4217.csv' WITH (FORMAT csv, HEADER true)`. Declaring the numeric code as `varchar(3)` is the safeguard, because `COPY` casts to the column type. Loading into an `int` column strips the zeros at cast time exactly as a naive seed would.

Q: What `psql` quoting pitfall can corrupt currency names like `Côte d'Ivoire`'s currency on load?
A: An apostrophe inside a CSV value is fine for `COPY` if the field is properly quoted in the file, but pasting such values into an inline `INSERT` breaks the SQL string literal unless the apostrophe is doubled (`''`) or you use dollar-quoting `$$...$$`. The mechanism is that `'` terminates the string literal mid-name. Preferring `\copy` from a well-formed CSV avoids the issue entirely, which is another reason to load lists as files, not hand-written inserts.

Q: How would you grant least-privilege access so analysts can read but not modify the governed code-list schema in Postgres?
A: Put the governed tables in a dedicated schema and `GRANT USAGE ON SCHEMA ref TO analyst_role; GRANT SELECT ON ALL TABLES IN SCHEMA ref TO analyst_role;` while withholding `INSERT`/`UPDATE`/`DELETE`, and use `ALTER DEFAULT PRIVILEGES` so future tables inherit the grant. Only the stewardship/CI role holds write privileges, enforcing that changes come through the governed pipeline. Read-only grants are the technical backstop to the procedural rule that nobody edits reference data ad hoc.

Q: Why is a database `CHECK` or foreign-key constraint a weaker control for reference data than a governed pipeline, yet still useful?
A: A `FOREIGN KEY` from a fact table to the code list enforces that only existing codes are inserted, but it cannot enforce effective dating, approval, or provenance — it only checks current membership. It is still useful as a last-line integrity guard that rejects a typo'd currency at write time. The governed pipeline adds the who/when/why and history the constraint cannot, so the two layers are complementary, not substitutes.

Q: How do you reconcile a vendor feed against the governed list to find both unmapped codes and obsolete mappings?
A: Run two anti-joins: vendor distinct codes left-anti-joined to the mapping table surfaces codes needing a new mapping, and mapping entries left-anti-joined to the vendor's recent codes surfaces mappings no longer exercised (candidate retirements). The first prevents silent drops; the second prevents the mapping table from accumulating dead entries. Scheduling both on each load turns coverage and staleness into automated assertions.

Q: A vendor adds a brand-new venue code mid-month that you have no mapping for. What should the silver layer do, concretely?
A: The silver mapping join should leave the code unmatched, route the affected rows to a quarantine/exceptions table, and raise a data-quality alert naming the unmapped value, so a steward adds a governed mapping rather than the pipeline guessing. Coalescing the unknown venue to a default MIC would mis-attribute trades in regulatory venue reporting. The Done-when traceability requirement means the new mapping, once added, carries `approved_by` and `source`.

Q: How does `dbt_is_deleted` from `hard_deletes: new_record` differ from simply closing `dbt_valid_to` on a retired code?
A: With `hard_deletes: invalidate`, dbt just sets the disappeared row's `dbt_valid_to`, leaving no explicit signal that the source dropped it; with `hard_deletes: new_record`, dbt inserts an additional closing row flagged `dbt_is_deleted = true`, so the deletion is a first-class, queryable event. For reference data this distinguishes "this code's window ended because it changed" from "the source removed this code entirely". The explicit flag is what lets you report retired-versus-superseded codes separately.

Q: Why is `hard_deletes` only meaningful when the snapshot source is a full extract, not an incremental delta?
A: dbt infers a hard delete by noticing a `unique_key` present in the prior snapshot is absent from the current source query, which is only valid when the source is the complete current list; a delta feed legitimately omits unchanged rows and dbt would falsely flag them all as deleted. So you must snapshot against a full ISO list each run. Pointing `hard_deletes` at a delta would mass-retire every code that simply did not change that day.

Q: How do you migrate an existing pre-1.9 snapshot to the new `hard_deletes` and `dbt_valid_to_current` configs without corrupting history?
A: You add the new configs and, because the meta-column semantics change (current rows move from `NULL` to the sentinel, and a `dbt_is_deleted` column appears), back-fill existing rows to match the new convention in a one-off migration before the next run, then verify as-of queries still return one version per date. dbt's upgrade guide warns these configs are "best suited for new snapshots" precisely because existing history needs careful reconciliation. Skipping the back-fill leaves mixed `NULL`/sentinel rows that break range predicates.

Q: What is the difference between a dbt seed and a dbt snapshot for the same ISO list, and when do you need both?
A: A seed loads the current CSV as a static, version-controlled table (good provenance via git, but no automatic history), whereas a snapshot captures how a table changes across runs into SCD2 rows. You use a seed when you control the file and re-commit each version, and a snapshot when you want dbt to accrue history automatically from a source table. For a governed list you can seed the raw publication and snapshot it to get both git provenance and queryable effective-dated history.

Q: How does committing the raw ISO publication as a seed give you provenance that a snapshot alone does not?
A: The seed CSV lives in git, so each ingested publication is an immutable, attributable commit with author, timestamp, and PR review — the who/when/why — whereas a snapshot only records that values changed, not who authorised the load. Git history ties each value back to the exact source artefact and reviewer. That is why the lesson pairs seeds (provenance) with snapshots/effective dating (history) rather than choosing one.

Q: Why store the original published ISO/MIC file in an immutable object store in addition to the parsed seed?
A: If a parsing bug is later discovered, you must re-derive the loaded values from the authoritative artefact, which requires the original file (the dated MIC CSV or the ISO amendment PDF) preserved byte-for-byte alongside its version identifier in the `source` column. The parsed rows alone cannot prove what the publisher issued. Immutable storage (WORM/object-lock) prevents the artefact itself being altered, closing the provenance chain.

Q: A `dbt snapshot` run errors with a "unique_key is not unique" or duplicate-key failure on an ISO list. What is the first thing to check?
A: Check that the declared `unique_key` is actually unique in the current source extract — an ISO list with both an alphabetic and numeric code can have the wrong column chosen, or the feed may contain duplicate rows for one code. dbt requires the key to identify one logical record, and duplicates make history-splitting ambiguous. The fix is to dedupe the source in a staging model or pick the correct single-valued key before snapshotting.

Q: How do you handle a code list whose natural key is composite, like operating-MIC plus segment for a hierarchy table?
A: For a snapshot, set `unique_key` to a SQL expression concatenating the parts with a separator (for example `operating_mic || '|' || segment_mic`) or build a surrogate key with `dbt_utils.generate_surrogate_key`, so dbt sees one stable key per logical row. The separator must be a character that never appears in either code to avoid collisions. A pipe is safe for MICs because they are alphanumeric only.

Q: Why is `dbt_utils.generate_surrogate_key` preferred over manual concatenation for composite keys?
A: It hashes the listed columns consistently (`md5` over coalesced, separated values), handling `NULL`s uniformly and avoiding the collision and `NULL`-propagation bugs of raw `||` concatenation where a `NULL` part nullifies the whole key. It also produces a fixed-width key regardless of input length. For a code list this guarantees a stable, unique surrogate even as component columns evolve.

Q: How does the silver-layer placement of mapping tables interact with idempotent reprocessing of a historical day?
A: Because the mapping is applied in silver and is itself effective-dated, reprocessing a historical bronze file deterministically re-derives the same canonical codes by resolving the mapping version valid on that file's date, so the silver output is reproducible. If mapping lived in gold or BI, a reprocess might apply today's mapping and produce different canonical codes than the original run. Silver placement plus effective dating is what makes reprocessing idempotent.

Q: What concrete evidence would you show an auditor to prove "the code list is consumed everywhere from one place"?
A: The dbt lineage graph (or catalog lineage) showing every consuming model's edge originating from the single governed reference node, plus a test asserting no other model defines competing currency/country/MIC values, plus a `grep`/search result over the project showing no hard-coded code literals in `case` statements. Together they demonstrate one source fanning out with no rivals. A single change propagating to all consumers in a test run is the live proof.

Q: How would you detect hard-coded currency literals leaking into models, as a CI guard?
A: Add a CI step that greps the compiled SQL or model source for inline currency-like literals — for example `grep -rIEn "'(USD|EUR|GBP|JPY)'" models/` — and fails the build if any model embeds codes instead of joining the governed list. The grep targets string literals that should instead be `ref()` joins. It is a blunt but effective backstop against the single-source principle being quietly bypassed.

Q: Why is the EMT/EPT distinction relevant to which reference lists you govern, and what does the EET add?
A: The EMT (European MiFID Template) carries MiFID II target-market and cost data, the EPT (European PRIIPs Template) carries PRIIPs KID figures, and the EET (European ESG Template) carries SFDR/Taxonomy sustainability data — each embeds reference codes (currencies, MICs, country, ISINs, plus ESG taxonomy codes for the EET). Governing one source means all three templates emit identical codes for the same fund. The EET specifically adds sustainability classification code lists you must govern alongside the financial ones.

Q: How does a governed reference layer reduce false breaks when reconciling an EMT file against the NAV book?
A: Because the EMT generator and the NAV book both join the same single currency/country/MIC source, code-level fields are identical by construction, so a reconciliation between them cannot break on a code mismatch and only surfaces genuine business differences. This eliminates an entire class of spurious exceptions that would otherwise consume analyst time chasing `GB`-versus-`UK` artefacts. Fewer false breaks means real discrepancies surface faster.

Q: Under DORA, why does immutable, versioned reference data help with the ICT incident reporting timeline?
A: DORA requires financial entities to detect, classify, and report major ICT-related incidents within tight deadlines, which demands fast, defensible reconstruction of what data was in force when an incident occurred. Effective-dated, immutably-stored reference data lets you replay exactly which codes drove a corrupted output, accelerating root-cause analysis and the regulatory report. Fragmented, overwriteable code lists would make such reconstruction slow or impossible, raising operational risk.

Q: A UCITS regulatory filing and the internal NAV disagree on a sub-fund's domicile country. How does single-source reference data localise the fault?
A: If both the filing and the NAV draw the country from one governed list via `ref()`, the code cannot differ between them, so any disagreement must be in the business mapping of fund-to-country, not in the code value — narrowing the investigation immediately. The single-source pattern removes "which copy was right" as a possible cause. Were they reading separate copies, you would first have to rule out code drift before reaching the real issue.

Q: How would you stage a controlled change that adds a newly-introduced ISO 4217 currency, end to end?
A: Open a pull request editing the seed CSV (or the governed table via an approved script) to add the code with its minor unit, effective-from set to the amendment's effective date, `approved_by` the steward, and `source` the amendment identifier; CI runs the snapshot and the no-overlap/no-gap tests; on merge the new version propagates to all `ref()` consumers. You then run an as-of query before the effective date to confirm the code is absent historically and after it to confirm presence. The coexistence of both as-of results is the preserved-history proof.

Q: What single end-to-end check best demonstrates the lesson's two Done-when criteria at once?
A: Make one governed change to a code's value, then (criterion one) query the git/`approved_by`/`source`/effective-date trail to show who changed it, when, and why, and (criterion two) run every consumer and the lineage graph to show the change reached all of them from the one source with no stale copies. The change being traceable and universally propagated from a single node satisfies both criteria in a single demonstration. If either the trail or the propagation is missing, the criterion fails.

Q: How does CFI (ISO 10962) code governance differ from currency governance given the CFI's positional structure?
A: A CFI is six positional characters where each position is drawn from a constrained alphabet conditional on the category — position one is the category (`E` equity, `D` debt, `C` collective investment vehicle), position two the group, and positions three to six attributes whose permitted letters depend on the category. So governing CFI means maintaining a context-sensitive grammar of permitted letters per position, not a flat list of whole codes. A naive flat allow-list of full six-character codes is impractical because the valid combinations run to thousands, so you validate position by position against the standard.

Q: When ingesting an ISO list, why record the source identifier per row rather than once per load batch, and when does it matter?
A: Storing `source`/version on each row means a row's provenance survives even after it is merged, snapshotted, or filtered into downstream tables where the batch-level metadata would be lost. If you keep the version only as a load-batch note, a single code's lineage cannot be reconstructed once it leaves the staging table. It matters acutely when two amendments touch the list in one window — per-row source distinguishes which value came from which amendment, whereas a batch tag would conflate them.
