# Phase 7 · 6.4.1 MLOps platforms & the ML lifecycle (MLflow) — 150 self-test questions

Q: Name the six stages of the ML system lifecycle in order, as taught in this lesson.
A: The six stages are data, features, training, registry, serving, and monitoring. Data is ingested and validated; features are engineered; training produces candidate models; the registry versions and governs them; serving exposes predictions; monitoring watches for drift and failures, which can feed back into retraining. As an architect you own how each stage plugs into your lakehouse, orchestrator, and lineage graph rather than the modeling inside training.

Q: In the ML lifecycle, where does the "data" stage touch your existing data platform?
A: The data stage reads from your lakehouse — the Bronze/Silver/Gold tables you already govern with the same partitioning, schema enforcement, and lineage. There is no separate "ML data lake"; training datasets are just queries over governed tables. This is why an architect insists ML reads validated Silver/Gold data rather than re-ingesting raw feeds, so model inputs inherit existing quality and audit controls.

Q: Which orchestrator stage does the "training" step typically map onto, and who should schedule it?
A: Training maps onto a job your orchestrator (Dagster in this plan) schedules and tracks like any other asset. The orchestrator owns when training runs, on what data window, and records the run as a materialization, while the ML framework owns the modeling inside the job. Keeping training under the same orchestrator means retraining, backfills, and dependencies all live in one lineage graph instead of a second hidden scheduler.

Q: Why does the lesson stress that ML stages touch "your" lakehouse, orchestrator, and lineage graph rather than being a separate system?
A: Because the architect's value is integration, not a parallel ML stack — a separate ML platform duplicates ingestion, governance, and audit. Reusing the lakehouse for data, the orchestrator for scheduling, and the lineage graph for provenance means a model's inputs are traceable end to end. The alternative, two orchestrators and two data stores, doubles cost and splits the audit trail a regulator must follow.

Q: What is MLflow, in one sentence, and what two server-side components are central to this lesson?
A: MLflow is an open-source platform for managing the ML lifecycle — experiment tracking, model packaging, a model registry, and serving. The two central components here are the tracking server (runs, params, metrics, artifacts) and the model registry (versioned, governed models). Both can be self-hosted on your platform with a database backend store and an artifact store.

Q: What is an MLflow "run", and what does it record?
A: A run is a single execution of model-development code — typically one training attempt. It records parameters (hyperparameters and config), metrics (like RMSE or MAE), artifacts (the model file, plots, datasets), and metadata such as start time, source, and git commit. Runs are the atomic unit of the audit trail: every number a model produced is traceable back to the run that created it.

Q: What is an MLflow "experiment", and how does it relate to runs?
A: An experiment is a named container that groups related runs — for example all attempts at a fund-flow forecasting model. Each run belongs to exactly one experiment, and the UI lets you compare runs within an experiment side by side. Organising by experiment keeps the registry and tracking server navigable as the number of runs grows into the thousands.

Q: How do you log a parameter, a metric, and an artifact in the MLflow tracking API?
A: You call `mlflow.log_param("max_depth", 6)` for a hyperparameter, `mlflow.log_metric("rmse", 0.42)` for an evaluation number, and `mlflow.log_artifact("plot.png")` for a file. These calls run inside a `with mlflow.start_run():` block so they attach to the current run. Params are write-once per key, whereas metrics can be logged repeatedly with a step to capture a training curve.

Q: What does MLflow autologging (`mlflow.autolog()` / `mlflow.sklearn.autolog()`) do, and why is it useful?
A: Autologging automatically captures parameters, metrics, and the model artifact for supported libraries without explicit `log_param`/`log_metric` calls. For an sklearn regressor over fund flows, calling `mlflow.sklearn.autolog()` before `fit()` records hyperparameters and training metrics automatically. It reduces boilerplate and, importantly, prevents the gotcha of a developer forgetting to log a parameter that later turns out to matter for audit.

Q: Why are runs, params, metrics, and artifacts described as "the audit trail of model development"?
A: Because together they answer how a model came to exist: what data window, what hyperparameters, what evaluation results, and what exact artifact. A regulator or model-risk validator can reconstruct the development decision from the tracked record rather than trusting a developer's memory. This is the model-development analogue of data lineage — provenance for the model itself.

Q: A colleague trained a model in a notebook with no tracking and now cannot reproduce the result. What was missing and what does MLflow give them?
A: They lacked an immutable record of params, metrics, data window, and the exact artifact, so the run is irreproducible. MLflow tracking captures all of these automatically per run, plus the source code reference and environment. The fix going forward is to wrap training in `mlflow.start_run()` (or enable autolog) so every attempt is reconstructable, which is also the minimum bar for any regulated deployment.

Q: What is the MLflow tracking server, and why self-host it with a database backend rather than local files?
A: The tracking server is the service that stores run metadata and serves the UI/API. A file-based store works for one laptop but cannot support concurrent writers, queries, or a team; a database backend (Postgres here) gives concurrency, durability, and SQL-queryable metadata. Self-hosting on your platform keeps fund-data run records inside your governed estate rather than a vendor SaaS.

Q: Write the command to start an MLflow tracking server with a Postgres backend store and a local artifact destination.
A: A typical command is `uv run mlflow server --host 0.0.0.0 --port 5000 --backend-store-uri postgresql://user:pass@localhost/mlflow --artifacts-destination ./mlartifacts`. The `--backend-store-uri` holds run metadata in Postgres, while `--artifacts-destination` is where artifacts land when the server proxies them. In this plan you run it via `uv run` so the MLflow version is pinned in the project environment rather than a bare global install.

Q: What is the difference between the backend store and the artifact store in MLflow?
A: The backend store holds structured metadata — runs, params, metrics, tags, and registry entries — and is typically a relational database like Postgres. The artifact store holds the large binary outputs — model files, plots, serialized datasets — and is typically object storage or a filesystem path. Separating them lets you put cheap bulk artifacts on object storage while keeping queryable metadata in a transactional database.

Q: What is the effect of running `mlflow server` with `--serve-artifacts` (proxied artifact access)?
A: With proxied artifact serving, clients upload and download artifacts through the tracking server using `mlflow-artifacts:` URIs, so they never need direct storage credentials. The server holds the storage credentials centrally, which simplifies access control and auditing of who fetched which artifact. The trade-off is that all artifact traffic flows through the server, which can become a bottleneck at high volume.

Q: Your MLflow UI shows runs but every artifact link 404s. What is the first thing to check?
A: Check that the artifact store location the server was started with is actually reachable and matches where clients wrote — a mismatch between `--artifacts-destination` (or `--default-artifact-root`) and the client's view is the classic cause. Confirm the server has read access to that path/bucket and that proxied serving is configured consistently. Run metadata lives in the backend store and loads fine even when the separate artifact store is misconfigured, which is exactly the symptom you see.

Q: How do you point a training client at a remote MLflow tracking server?
A: Set the tracking URI, either with the environment variable `MLFLOW_TRACKING_URI=http://host:5000` or in code with `mlflow.set_tracking_uri("http://host:5000")`. Without this, MLflow writes to a local `./mlruns` directory and your runs never reach the shared server. Forgetting to set it is a common gotcha: developers see runs locally but the team server stays empty.

Q: What is the MLflow Model Registry, and what problem does it solve beyond tracking?
A: The registry is a centralized store of named, versioned models with lifecycle metadata, layered on top of tracking. Tracking answers "what experiments did we run?"; the registry answers "which model is the official one, what version, who approved it, and where did it come from?". It turns a folder of artifacts into a governed catalog with promotion as an explicit, auditable event.

Q: How do you register a model into the MLflow registry?
A: Either pass `registered_model_name` when logging, e.g. `mlflow.sklearn.log_model(model, name="model", registered_model_name="fund_flow_forecast")`, or call `mlflow.register_model(model_uri, name)` against an existing run's artifact. Registration creates a new model version under that registered-model name. Each registration captures the source run, so the version is linked back to its params, metrics, and data window.

Q: What is a "registered model" versus a "model version" in MLflow?
A: A registered model is the named entity — for example `fund_flow_forecast` — that represents a model's identity over time. A model version is a specific immutable snapshot under that name — version 1, 2, 3 — each tied to the run that produced it. You promote and reference versions; the registered-model name is the stable handle a serving system targets.

Q: Why does MLflow now recommend aliases instead of the old stages, and what were the legacy stages?
A: The legacy registry used four fixed stages — None, Staging, Production, Archived — which are now deprecated because they were rigid and singular. Aliases are mutable named pointers to a specific version that you define freely (e.g. `champion`, `challenger`), giving flexible deployment labeling. Migrating means production code targets `models:/name@champion` instead of the old `models:/name/Production` URI.

Q: What is a model "alias" in MLflow, and how do you set one?
A: An alias is a mutable, named reference to a particular model version, used to mark which version plays a role like production. You set it with `client.set_registered_model_alias("fund_flow_forecast", "champion", 3)`, pointing the `champion` alias at version 3. Reassigning the alias to a new version is how you cut over production without changing the serving code that loads `models:/fund_flow_forecast@champion`.

Q: What does the model URI `models:/fund_flow_forecast@champion` resolve to, and why use it in production?
A: It resolves to whichever model version the `champion` alias currently points at for the registered model `fund_flow_forecast`. Production code references the alias, not a fixed version number, so promotion is a single alias reassignment with no code change or redeploy. This decouples the deployment decision from the codebase and makes "what is in production right now?" a registry query.

Q: What are champion and challenger aliases used for?
A: `champion` conventionally marks the current production model serving the majority of traffic, while `challenger` marks a candidate being validated against it. This supports a controlled comparison: the challenger runs alongside or shadow-scores production data before you promote it by moving `champion`. It is the alias-based expression of a safe-rollout pattern.

Q: How do model version tags differ from aliases, and give a governance example.
A: Tags are key-value annotations on a model version (or registered model), while an alias is a single mutable pointer marking a role. Tags describe state and evidence; aliases mark deployment position. For governance you might set `client.set_model_version_tag("fund_flow_forecast", "3", "validation_status", "approved")` to record that independent validation passed, making approval status a queryable field.

Q: What is the MLflow `pyfunc` (python_function) flavor, and why does it matter for serving?
A: `pyfunc` is MLflow's universal model interface: any logged model can be loaded as a `pyfunc` with a standardized `predict` method regardless of the underlying framework. This means a serving layer calls `model.predict(df)` without knowing whether it is sklearn, XGBoost, or custom code. It is the portability contract that lets one serving stack host heterogeneous models.

Q: How do you load a registered model for inference by alias, and what does `mlflow.pyfunc.load_model` return?
A: You call `model = mlflow.pyfunc.load_model("models:/fund_flow_forecast@champion")`, which fetches the version the alias points at. It returns a `PyFuncModel` exposing a standard `predict` method, so you call `model.predict(input_df)`. Loading by alias means the same code automatically picks up a newly promoted version after you reassign `champion`.

Q: What is a custom `pyfunc` model and when would you build one?
A: A custom `pyfunc` is a class implementing `predict` (subclassing `mlflow.pyfunc.PythonModel`), letting you wrap arbitrary pre/post-processing and artifacts around a model. You build one when inference needs more than a raw framework call — for example normalising ISIN inputs, applying business rules, or combining several models. It still loads through the standard `pyfunc` interface, preserving the uniform serving contract.

Q: What does "packaging" mean in the registry context, and what does an MLflow Model directory contain?
A: Packaging means bundling the model artifact with the metadata needed to reload and run it reproducibly. An MLflow Model is a directory containing an `MLmodel` file describing the flavors, plus the serialized model, a `conda.yaml`/`requirements.txt` for dependencies, and an input/output signature. This package is what gets versioned in the registry so serving can reconstruct the exact runtime.

Q: What is a model "signature" in MLflow and why is it valuable in a regulated pipeline?
A: A signature records the expected input and output schema (column names and types) of a model. At inference MLflow can enforce it, catching a malformed feature frame before it produces a silently wrong NAV-impacting number. In a regulated context the signature is part of the model's documented contract, so a schema drift in upstream fund data surfaces as a validation error rather than a quiet bad prediction.

Q: Describe the promotion workflow from Staging to Production in alias/tag terms.
A: A candidate version is marked `challenger`; once it clears a metric threshold and a named human approves, you record approval as a tag and reassign the `champion` alias to it. Promotion is therefore a deliberate, queryable event — not an overwrite — with the metric evidence and approver captured. This is what makes "promotion a governed event" rather than a developer silently shipping a new model.

Q: Why should a promotion gate include both a metric threshold and a named human approver?
A: The metric threshold is an objective bar (e.g. RMSE below a value on a holdout window) that prevents promoting a worse model. The named approver provides accountability and the "effective challenge" a regulator expects — a human who owns the decision. A purely automated gate has no one to answer for a bad model; a purely human gate has no objective evidence. Recording both as tags/aliases makes the gate auditable.

Q: In the lesson's drill, starting from the Production (champion) alias alone, what chain must you be able to walk back?
A: From the `champion` alias you must reach the model version, then the source run, then that run's params, then the training data window, and finally the named approver recorded in the tags. Each hop is a registry or tracking lookup, not tribal knowledge. Being able to do this quickly is the model-governance analogue of the data-lineage drill: full provenance from the deployed artifact backwards.

Q: How do you find which version an alias points to, and the run behind it, via the client API?
A: Call `client.get_model_version_by_alias("fund_flow_forecast", "champion")` to get the version, whose object carries `run_id`; then `client.get_run(run_id)` returns the params, metrics, and tags. From the run you read the logged data-window param and any artifacts. This two-step lookup is the queryable backbone of the promotion-traceback drill.

Q: What is data drift, and which monitoring signal detects it?
A: Data drift (covariate shift) is a change in the distribution of the input features while the input-to-output relationship stays the same. You detect it by monitoring the input distributions — comparing live feature statistics against the training distribution with tests like population stability index or KS tests. For fund flows, a regime where subscription volumes shift upward is data drift in the input even if the underlying flow dynamics are unchanged.

Q: What is concept drift, and why is it harder to catch than data drift?
A: Concept drift is a change in the relationship between inputs and outputs — the function the model learned no longer holds — so previously learned patterns become invalid. It is harder to catch because inputs can look unchanged while the mapping has shifted, so you usually need labels (actual outcomes) to detect a rise in error. You often only see it as degrading prediction accuracy once ground truth arrives.

Q: Give a fund-context example distinguishing data drift from concept drift for a redemption-forecast model.
A: Data drift: the distribution of monthly inflows shifts because a new share class launched, but the relationship between drivers and redemptions is unchanged. Concept drift: a regulatory or market regime change (say a liquidity event) alters how investors react, so the same inputs now imply different redemption behaviour. The first you spot by watching inputs; the second only shows up as rising forecast error against realized flows.

Q: What monitoring signal best detects concept drift, and what is its practical limitation?
A: The clearest signal is a rise in the model's error against ground-truth labels — predicted versus actual outcomes. Its limitation is label latency: for a redemption forecast you may wait until the actual flows settle before you can compute error, so detection lags the drift. Where labels are delayed, teams use proxy signals or shadow metrics, accepting they are weaker than true error.

Q: What typically triggers a retraining event in a monitored ML system?
A: Retraining is triggered by detected drift (input distribution shift or rising error past a threshold), by a fixed schedule, or by a known regime change such as a new data source or product. The trigger should be an explicit, owned policy rather than ad hoc. In a regulated platform the retraining trigger and the resulting new version both become registry events with their own approval gate.

Q: A monitored fund-flow model's accuracy degrades but input distributions look unchanged. Data drift or concept drift, and what do you do?
A: Unchanged inputs with falling accuracy points to concept drift — the input-to-output relationship has shifted. The response is to investigate what regime changed, gather recent labeled outcomes, and retrain on a data window that includes the new behaviour, then run the candidate through the promotion gate as a challenger. You do not just widen the training window blindly; you confirm the shift is real before promoting a retrained model.

Q: Why is monitoring counted as a first-class lifecycle stage rather than an afterthought?
A: Because a deployed model silently decays as the world changes, and without monitoring you discover failure only when a wrong NAV-impacting number reaches a client. Monitoring closes the loop, feeding drift and error signals back into the retraining trigger. Architecturally it touches your platform like any other pipeline — emitting metrics your observability stack already collects.

Q: SR 11-7 is the canonical model-risk guidance the lesson cites. Who issued it and what is it?
A: SR 11-7, "Supervisory Guidance on Model Risk Management," was issued by the US Federal Reserve in 2011 (jointly with the OCC as OCC 2011-12). It is principles-based guidance for managing the risks that arise when banks use quantitative models in decision-making. Regulators and validators across finance borrow its vocabulary, which is why this lesson uses it as the model-risk reference even outside US banking.

Q: What are the three pillars of model risk management under SR 11-7?
A: The three pillars are sound model development and implementation; effective challenge through independent validation; and governance with clear board/management accountability. Together they ensure models are built well, independently checked, and overseen. The MLflow lifecycle maps onto these: tracking evidences development, the validation gate is independent challenge, and the registry plus approval tags provide governance.

Q: What does SR 11-7 mean by "model risk", and what are its two main sources?
A: Model risk is the potential for adverse consequences from decisions based on incorrect or misused model outputs. Its two main sources are fundamental errors that produce inaccurate outputs, and incorrect or inappropriate use of an otherwise sound model. A fund-flow forecast can be wrong because the model is flawed, or because someone uses it outside the conditions it was validated for.

Q: What is a "model inventory" in SR 11-7 terms, and what is the MLflow analogue?
A: A model inventory is a complete, maintained list of all models in use, with owners, purpose, validation status, and risk rating. The MLflow registry is the technical analogue for ML models: registered models with versions, owner tags, and validation-status tags. The key point for the lesson is that ML models join the same inventory as pricing and valuation models — they are not a separate, ungoverned category.

Q: What is "effective challenge" in SR 11-7, and how does an MLflow promotion gate embody it?
A: Effective challenge is critical analysis by objective, informed parties able to identify a model's limitations and force appropriate changes. A promotion gate embodies it when an independent, named approver — not the model's author — must sign off, with the metric evidence in front of them, before the `champion` alias moves. Recording that approver as a registry tag is the auditable proof the challenge happened.

Q: Why does the lesson say "ML models join the same inventory as pricing models"?
A: Because a regulator does not see ML as special — a model that influences a NAV-impacting forecast carries model risk just like a valuation or pricing model and must be inventoried, owned, and validated the same way. Letting ML models live in a separate ungoverned tool creates a blind spot a CSSF inspector or model-risk validator will find. The architect's job is to wire the ML registry into the firm's existing model inventory and controls.

Q: What three things must you be able to answer about a deployed model from the registry alone (the lesson's headline "Done when")?
A: Which model version produced a given number, what data it was trained on, and who approved its promotion. All three must come from the registry and linked run — not from a person's memory or a wiki. This is the model-governance analogue of the data-lineage drill and the first question a model-risk validator or CSSF inspector asks.

Q: How does the registry map onto an SR 11-7 model inventory for a risk officer's view: model, owner, validation evidence?
A: The registered model is the inventory entry (the model's identity), an owner tag names the accountable party, and validation-status tags plus the linked run's metrics are the validation evidence. The `champion` alias shows what is live, and the approval tag shows who signed off. So the registry can be presented as a self-documenting slice of the firm's model inventory.

Q: Why is "which model version produced this NAV-impacting forecast?" the question a validator asks first?
A: Because a NAV-impacting number must be defensible: the firm has to show the exact model, its inputs, and its approval before relying on it for a regulated calculation. If that question is unanswerable, every downstream figure is unaudited and potentially restatable. A governed registry makes the answer a single query, which is why the lesson treats it as the headline competency.

Q: What is the fund-data experiment the lesson has you build, and what model type does it use?
A: A small fund-flow forecasting experiment: an sklearn regressor predicting monthly subscription and redemption flows. You track 4–6 runs with different hyperparameters, logging metrics and the model artifact each time. The point is not modeling sophistication but exercising the full tracking-to-registry lifecycle on realistic fund data.

Q: When registering the winning fund-flow model, why log a "description of its training data window"?
A: Because the data window is essential provenance — a flow forecast trained only through a calm market period behaves differently from one trained across a redemption shock. Recording the window (e.g. "Jan 2022–Dec 2024 monthly flows") lets a validator judge whether the model was trained on representative conditions. It is exactly the kind of fact the promotion-traceback drill must surface from the registry.

Q: How would you record a written promotion gate so it is queryable in MLflow?
A: Capture the gate's outcome as registry metadata: set a tag like `promotion_approved_by=jane.doe` and `metric_threshold_rmse=0.40`, and move the `champion` alias only when both hold. Because tags and aliases are queryable via the API, an auditor can later filter for approved versions and read who approved each. The gate text itself lives in policy docs, but its application is recorded as structured, searchable registry data.

Q: Why run MLflow via `uv run` in this plan rather than a bare `pip install mlflow`?
A: The plan standardizes on `uv` so every tool runs in a reproducible, pinned project environment; bare `python`/`pip` invocations are deliberately blocked. `uv run mlflow server ...` ensures the MLflow version matches the project lock file, so the tracking server, clients, and CI all use the same version. This avoids the classic gotcha of a model logged under one MLflow version failing to load under another.

Q: What is the risk of a model logged under a much newer MLflow version being loaded by an older client?
A: The serialized model or its `MLmodel`/flavor metadata may use features the older client cannot parse, causing load failures or subtle behaviour differences. MLflow records the version that logged the model, but it cannot retrofit forward-compatibility. Pinning the MLflow version across producers and consumers (via `uv` here) is the practical defence, and the model environment files document the exact dependencies.

Q: What does an MLflow run's "source" metadata capture, and why is it useful for audit?
A: Source metadata records what produced the run — typically the script or notebook path and, when available, the git commit and branch. This links a model back to the exact code that trained it, completing the chain from artifact to source. For audit it means "what code produced this version?" is answerable, not just "what params", which matters when a bug is later found in training code.

Q: How do metrics with a "step" support analysing a training run over time?
A: Logging a metric repeatedly with an increasing `step` (e.g. per epoch or per boosting round) records a time series, which the UI plots as a learning curve. This lets you spot overfitting or non-convergence rather than seeing only a final number. Params, by contrast, are single values set once; the step mechanism is specifically for evolving metrics.

Q: You compare 6 runs in the MLflow UI and pick the lowest validation RMSE, but it overfits in production. What lifecycle gap does this reveal?
A: Picking on a single metric without a proper holdout or out-of-time validation window let an overfit model through — the promotion gate and validation pillar were too weak. The fix is a metric threshold evaluated on a representative out-of-time fund-flow window plus independent review before moving `champion`. It shows that tracking alone is not governance; the registry gate is where overfit candidates should be caught.

Q: Why is the registry promotion event called "governed" while logging a run is not?
A: Anyone can log a run — it is just recording an experiment, with no claim of fitness. Promotion changes which model the platform actually serves, so it must pass a gate and be attributable to an approver, making it a controlled decision. The distinction mirrors a code merge versus a local commit: experimentation is free, but what reaches production is gated.

Q: What is the relationship between MLflow tracking and lineage in your broader platform?
A: Tracking provides model-development lineage — from data window through params to artifact — that should connect to the data lineage your platform already maintains. Ideally the run records which lakehouse tables and snapshot it read, so a model version links back to upstream data lineage. Together they give end-to-end provenance: from a served prediction back through the model to the exact source data.

Q: How would you make an MLflow run record the exact training data snapshot it used?
A: Log the data identifier as a param or tag — for example a Delta/Iceberg table version, a snapshot timestamp, or a query hash — and optionally log the dataset itself as an artifact or via MLflow's dataset tracking. This pins the model to an immutable data version, so "trained on what data?" resolves to a specific snapshot. Relying on "the latest table" instead is the gotcha that breaks reproducibility when the table changes.

Q: What is the difference between an online (real-time) and a batch serving pattern for a fund-flow model, conceptually?
A: Online serving exposes a low-latency endpoint that scores one request at a time, suitable when predictions are needed interactively. Batch serving scores a whole dataset on a schedule, writing results to a table, which fits most fund workloads where flows are forecast periodically. The architect chooses batch by default for fund use cases and reserves online serving for genuine real-time needs, since online adds operational cost.

Q: Why might batch serving via your orchestrator be preferable to a standing online endpoint for monthly fund-flow forecasts?
A: Monthly forecasts have no real-time requirement, so a standing endpoint idles and costs money while adding deployment surface to secure and monitor. A scheduled batch job in your orchestrator loads `models:/name@champion`, scores the period, and writes results into a governed table. This keeps serving inside the lineage graph and avoids paying for an always-on service nobody queries between runs.

Q: What does "promotion is a governed event" require you to record beyond just moving the alias?
A: It requires recording the metric evidence that cleared the threshold, the named approver, and a timestamp — typically as version tags — so the event is reconstructable. Just moving the alias loses the why and the who. The recorded tags turn the alias move into an auditable decision rather than an anonymous overwrite.

Q: How does the MLflow registry help answer a CSSF inspector asking about a NAV-impacting model change?
A: The registry shows the version history, which version is `champion` now and was at the relevant date, the linked run with params and data window, and the approval tags. The inspector can see the model was developed, validated, and approved through a controlled process with named accountability. This is the model-governance evidence the lesson is training you to produce on demand.

Q: What is the gotcha of using the deprecated `transition_model_version_stage` API in new MLflow code?
A: Stages are deprecated in favour of aliases, so building new workflows on `transition_model_version_stage` ties you to a sunset mechanism and the rigid four-stage model. New code should use `set_registered_model_alias` and version tags, and reference `models:/name@alias` URIs. Migrating later means rewriting every production reference, so starting on aliases avoids rework.

Q: How do you programmatically find the best run in an experiment using the MLflow API?
A: Use `mlflow.search_runs(experiment_ids=[id], order_by=["metrics.rmse ASC"], max_results=1)`, which returns a DataFrame of runs sorted by the chosen metric. This lets you select and register the winner in code rather than clicking through the UI, which is essential for an automated promotion pipeline. The same search backs governance queries, since you can filter runs by params, tags, or metric thresholds.

Q: What is the difference between loading a model with `mlflow.sklearn.load_model` versus `mlflow.pyfunc.load_model`?
A: `mlflow.sklearn.load_model` returns the native sklearn estimator with its full API (e.g. `predict_proba`, attribute access), useful when you need framework-specific behaviour. `mlflow.pyfunc.load_model` returns the generic `PyFuncModel` with a single standardized `predict`, which is what a framework-agnostic serving layer uses. Serving code prefers `pyfunc` for portability; debugging or framework-specific inspection uses the flavor-specific loader.

Q: A fund-flow model now also consumes EMT/EPT-derived cost fields that changed format mid-year. How does this present in monitoring, and how does the registry help?
A: A changed EMT/EPT field format shows up as data drift (or, if it silently misfeeds the model, as a model signature/validation failure at serving), and as rising forecast error if the relationship also shifts. The registry lets you pin the affected production version, retrain on a window reflecting the new format, and promote the corrected version through the gate while keeping the old version available for comparison and audit. Tying the data-window tag to the format change documents exactly why the new version exists.

Q: The MLflow UI loads and lists runs, but a client gets an error only when writing artifacts. Which store is at fault and why?
A: The artifact store is at fault, not the backend store — run metadata listing works because that comes from the backend database, while artifact writes go to the separate artifact destination. Check that the configured artifact location is writable by the client (or that proxied serving is enabled so the client routes through the server), and that credentials/path match. The clean split of symptoms — metadata fine, artifacts failing — is the diagnostic clue pointing at the artifact store.

Q: Why does the lesson keep this MLOps lesson at T2 depth rather than deep modeling?
A: Because the architect's seat is the platform — the lifecycle, registry, governance, and integration — not competing with ML teams on modeling. T2 means working-level command of how MLflow plugs into the lakehouse, orchestrator, lineage, and model-risk controls. The deep statistical middle of model building is deliberately skimmed; you architect the platform the modelers run on.

Q: What is the minimum you would log per run to satisfy a model-risk validator?
A: At minimum: the hyperparameters, the evaluation metrics on a defined holdout/out-of-time window, the exact training data window or snapshot, the model artifact with a signature, and the source code reference. Together these let a validator reproduce and challenge the model. Missing any one — especially the data window or the artifact — breaks the chain a validator needs.

Q: How do aliases support a safe rollout (e.g. shadow or canary) for a new fund model?
A: You keep `champion` on the proven model and assign `challenger` to the candidate, then route a fraction of scoring or run the challenger in shadow against the same inputs to compare outputs. Only after the challenger meets the gate do you reassign `champion`. Because serving references aliases, the cutover and any rollback are single alias moves with no redeploy.

Q: If a newly promoted model misbehaves in production, how do you roll back using aliases?
A: Reassign the `champion` alias back to the previous known-good version, e.g. `client.set_registered_model_alias("fund_flow_forecast", "champion", 2)`. Because production loads `models:/fund_flow_forecast@champion`, the next scoring run picks up version 2 with no code change. The fast, auditable rollback is precisely why production references an alias instead of a pinned version.

Q: What does it mean that a model version is "immutable" in the registry, and why does that matter?
A: An immutable version cannot have its bound artifact silently swapped — version 3 always refers to the same model. This is essential for audit: if version 3 produced a NAV-impacting number, that exact artifact is preserved for inspection. Aliases and tags are mutable pointers and labels, but the underlying versioned artifacts stay fixed.

Q: Why log the model artifact every training run rather than only for the winner?
A: Because the "winner" is decided after the runs, and a run whose artifact you discarded cannot be re-examined or promoted later if the chosen model fails review. Logging every artifact keeps all candidates reproducible and comparable, at modest storage cost. The gotcha of logging only the apparent best is discovering later you need a discarded run's exact model.

Q: How does experiment tracking relate to the evaluation/tracing entry (6.7.1) the lesson references?
A: The lesson notes that 6.7.1 "lives here": experiment tracking is the substrate on which evaluation results and traces are recorded. Evals and traces become metrics and artifacts on runs, so the same registry/tracking governance covers LLM and classical-ML evaluation evidence. Tracking is therefore the audit backbone that later LLMOps controls build on.

Q: What is the architectural risk of running MLflow's registry alongside a second, separate model registry?
A: Two registries split the model inventory and the audit trail, so "which is authoritative?" becomes ambiguous and approvals can diverge. It also doubles operational cost and the chance that a model serves from an ungoverned path. The lesson's stance is to consolidate ML models into one governed registry that maps onto the firm's model inventory.

Q: How do you query the registry for all model versions a particular person approved?
A: Tag approvals consistently (e.g. `approved_by=<name>`), then use the search API such as `client.search_model_versions("name='fund_flow_forecast'")` and filter on the tag, or a supported tag filter in the search string. Because tags are structured metadata in the backend store, this is a direct query rather than a manual trawl. Consistent tagging conventions are what make governance questions answerable at scale.

Q: What is the difference between `log_model` and `register_model` in MLflow?
A: `log_model` saves the model as an artifact of the current run in the packaged MLflow Model format; `register_model` creates a new version of a named registered model in the registry, pointing at a logged model URI. You can do both in one step by passing `registered_model_name` to `log_model`. Logging is about producing the artifact; registering is about entering it into the governed catalog.

Q: Why is a Postgres backend store appropriate for a fund-platform MLflow deployment specifically?
A: Postgres gives transactional integrity, concurrent multi-user access, backups, and SQL queryability of the model inventory — all expected of a system holding regulated audit metadata. A file store cannot meet concurrency or durability needs and is not auditable infrastructure. Using the platform's existing managed Postgres also keeps the registry inside the governed, backed-up estate rather than a fragile local file.

Q: What happens to the artifact store if you delete a run or model version in MLflow?
A: Deleting metadata does not necessarily purge the underlying artifact files; MLflow may soft-delete the record while the artifact remains until garbage collection is run. This matters for both storage cost and retention/audit policy — you may be required to retain or, conversely, to provably destroy artifacts. The gotcha is assuming a UI delete fully removes the data; verify the artifact-store retention behaviour explicitly.

Q: How does the registry support the SR 11-7 expectation of "ongoing monitoring" of a model?
A: The registry identifies exactly which version is live, so monitoring results (drift, error metrics) can be attributed to that specific version and logged against it. When monitoring trips a threshold, the retraining produces a new version that re-enters the gate. The registry thus connects ongoing monitoring to a versioned, governed change rather than an untracked tweak.

Q: A regulator asks for the validation evidence behind the production model. Where in MLflow do you point them?
A: Resolve `champion` to its version, open that version's tags for `validation_status` and approver, and its linked run for the metrics computed on the validation window plus any validation-report artifacts. Together these are the documented evidence of independent challenge. If those tags and artifacts are absent, that itself is the finding — the model was promoted without recorded validation.

Q: Why does the lesson frame the promotion-traceback as "the model-governance analogue of your lineage drill"?
A: Data lineage lets you trace a number back through transformations to source data; the promotion traceback lets you trace a deployed model back through version, run, params, data window, and approver. Both are provenance drills measured by speed and completeness, and both answer a regulator's "where did this come from?". Mastering one prepares you for the other; the model is just another node whose origin must be reconstructable.

Q: What is "covariate shift" and which lesson term does it correspond to?
A: Covariate shift is a change in the distribution of input features P(X) while the conditional P(Y|X) stays fixed, which corresponds to data drift in the lesson. Because the input-output relationship is unchanged, a model can still be valid but may be extrapolating into unfamiliar input regions. You detect it by monitoring input distributions rather than waiting for labels.

Q: Can you have data drift without concept drift, and what does it imply for action?
A: Yes — inputs can shift while the input-output relationship holds (pure data drift), so the model may remain accurate even though distributions moved. The implication is that not every detected input shift demands retraining; you should confirm whether error actually degraded. Reacting to every distribution wobble causes needless retraining churn, while ignoring a shift that does coincide with rising error is the opposite failure.

Q: What is the danger of tuning a model only on in-sample fund-flow data, in drift terms?
A: In-sample tuning hides how the model behaves under distribution change, so it can overfit the training regime and break when even mild data drift appears in production. Without an out-of-time validation window you cannot estimate robustness to future flows. The promotion gate should therefore evaluate on a held-out future-like window, not the data the model trained on.

Q: How does the model signature interact with detecting feature schema drift before serving?
A: At inference MLflow can validate inputs against the recorded signature and raise an error if columns or types diverge from training, catching schema drift at the boundary. For a fund pipeline, an upstream change that drops or retypes a feature column then fails loudly rather than feeding the model garbage. This is schema enforcement for the model, complementing distribution-based drift monitoring.

Q: Why is "who approved it" treated as essential metadata rather than optional in a regulated deployment?
A: Because SR 11-7-style governance demands accountability — a named, independent person owns the decision to rely on a model, providing effective challenge. Without a recorded approver there is no one answerable for a model that later produces a bad NAV-impacting number. Capturing the approver as a registry tag makes accountability queryable and auditable, not a matter of recollection.

Q: How would you structure the registry so an LEI- or fund-level model inventory is navigable?
A: Use a clear naming convention per model purpose and tag versions with the relevant scope — owner, fund or entity identifier, and validation status — so inventory queries filter cleanly. Keep one registered model per logical model, with versions for iterations, rather than overloading one name. Consistent tags turn the registry into a queryable model inventory aligned with how the firm catalogs other regulated models.

Q: What does it mean to "sketch the six lifecycle stages from memory and name where each touches your lakehouse and orchestrator" — give the orchestrator touches.
A: Data is validated as lakehouse tables; features are derived assets; training is an orchestrated job; the registry is updated by that job; serving is a scheduled batch (or endpoint) the orchestrator triggers; monitoring emits metrics the orchestrator and observability stack collect. The orchestrator most directly owns training, serving runs, and monitoring schedules. The point of the exercise is to show ML is wired into existing platform components, not bolted on.

Q: Why is reproducibility a precondition for independent validation under SR 11-7?
A: An independent validator must be able to re-run or re-examine the model to challenge it; if the run is irreproducible, validation is impossible and the model cannot be defended. MLflow tracking supplies the params, data window, artifact, and source needed to reproduce. So tracking is not just developer convenience — it is what makes the validation pillar achievable.

Q: What is the difference between logging a metric and setting a tag, and when do you use each?
A: A metric is a numeric measurement of model performance (RMSE, MAE) logged per run, often as a series; a tag is a key-value label for organisation or governance (owner, validation_status, approved_by) on runs, versions, or models. You log metrics to evidence quality and set tags to encode state and accountability. Confusing them — e.g. trying to store an approver as a metric — breaks the queryability the governance workflow relies on.

Q: How does MLflow help you compare 4–6 fund-flow runs to pick a winner objectively?
A: The UI presents runs in a sortable table with their params and metrics side by side, and overlays metric curves, so you sort by validation RMSE/MAE and inspect the trade-offs. Because params are logged alongside, you see which hyperparameters drove the improvement. The objective comparison — on a consistent holdout — is what justifies the registration and later promotion decision.

Q: What is the consequence of not setting `MLFLOW_TRACKING_URI` in a shared-team setting?
A: Each developer's runs land in their local `./mlruns` directory, so the team's central server and registry never see them and the audit trail is fragmented. Promotion and comparison become impossible across people. The fix is to set the tracking URI (env var or `set_tracking_uri`) to the shared server for everyone, ideally baked into the project config so it is not forgotten.

Q: Why might you store validation reports as run/version artifacts rather than only in a separate document system?
A: Attaching the validation report to the exact model version binds the evidence to the artifact it validates, so provenance cannot drift apart. A validator pulling up `champion`'s version sees its report in place, not via a separate cross-reference that can go stale. This keeps the model, its evidence, and its approval colocated in the governed registry.

Q: How does the registry's view of "current production" differ before and after an alias move, and why is that auditable?
A: Before the move, `champion` points at the old version; after, it points at the new one, and the registry records each version and the alias assignment. Because versions are immutable and tag/alias changes are tracked, you can reconstruct what was live on any date. That history is the audit evidence for "which version produced the number on date X".

Q: What is the role of an environment/requirements file in the packaged model, and what failure does it prevent?
A: The packaged model includes a `requirements.txt`/`conda.yaml` capturing the exact dependencies, so serving can rebuild the runtime that the model expects. It prevents the "works on my machine" failure where a library version mismatch silently changes predictions or breaks loading. In a regulated context it also documents the model's runtime as part of its reproducible record.

Q: Summarise how MLflow tracking plus registry together satisfy "which version, trained on what, approved by whom" — the lesson's core competency.
A: Tracking captures, per run, the params, metrics, source, and data window; registering creates an immutable version linked to that run; promotion records the approver and threshold as tags and points `champion` at the version. Resolving the alias walks back to the version, the run, the data window, and the approver in a few queries. That single, reconstructable chain — produced from the registry alone — is exactly the model-governance evidence a CSSF inspector or model-risk validator demands first.

Q: How do you serve a registered fund-flow model as a local REST endpoint with MLflow, and what is the default port?
A: Run `uv run mlflow models serve -m "models:/fund_flow_forecast@champion" -p 5000`, which loads the aliased version and starts a local inference server; the example default in the docs is port `5000`. The model is loaded through the `pyfunc` flavor so the framework is irrelevant to the server. For a fund workload this is mostly useful for testing, since batch scoring via the orchestrator is the default production pattern.

Q: What HTTP endpoints does the `mlflow models serve` inference server expose, and what is each for?
A: It exposes `/invocations` for POST prediction requests, `/ping` and `/health` for liveness/health checks, and `/version` to return the MLflow version serving the model. A load balancer or readiness probe hits `/ping` or `/health`, while scoring traffic posts to `/invocations`. Knowing `/version` exists matters because a version mismatch between the serving image and the logged model is a classic silent-failure source.

Q: What JSON payload formats does the MLflow `/invocations` endpoint accept for a tabular fund-flow input?
A: It accepts a wrapped JSON body keyed by `dataframe_split`, `dataframe_records`, `instances`, or `inputs` (the last two are TensorFlow-Serving-style), plus CSV when you set the `Content-Type` header accordingly. For a pandas feature frame you typically send `{"dataframe_split": {"columns": [...], "data": [...]}}`. Sending a bare unkeyed JSON array is the common gotcha that returns a `400` because the server cannot tell which orientation you meant.

Q: How does the `--env-manager` flag on `mlflow models serve` change how the model's environment is built?
A: `--env-manager` selects how the runtime is recreated from the packaged dependencies: `virtualenv` (the modern default) or `conda` rebuild the model's recorded environment, while `local` skips recreation and uses the current interpreter. Rebuilding guarantees the model runs against the libraries it was logged with, preventing a version-mismatch prediction drift. In this `uv`-pinned plan you often use `--env-manager local` because the project environment is already the pinned one.

Q: What is `mlflow models predict` and when would you use it instead of `mlflow models serve`?
A: `mlflow models predict` is a one-shot CLI command that loads a model, scores an input file, and writes predictions to an output file — e.g. `uv run mlflow models predict -m "models:/fund_flow_forecast@champion" -i input.csv -o preds.csv` — without standing up a server. You use it for batch or ad hoc scoring where no live endpoint is needed, which matches most fund cadences. Like `serve`, it accepts `--env-manager` to rebuild the model's recorded environment so predictions are reproducible.

Q: What does the `mlflow gc` command do, and why is it not automatic?
A: `mlflow gc` permanently purges runs, experiments, and logged models that are in the `deleted` lifecycle stage, removing both backend-store records and (when reachable) their artifacts. It is manual by design because deletion in MLflow is a soft-delete: a UI "delete" only marks records `deleted`, and an operator must run `gc` to make removal permanent. This two-step model protects against accidental loss but means storage is not reclaimed until someone runs the command.

Q: Which `mlflow gc` flag limits garbage collection to records deleted before a certain age, and why use it?
A: The `--older-than` flag restricts `gc` to runs that have been in the `deleted` state longer than the given duration (e.g. `--older-than 30d`), and `--run-ids` / `--experiment-ids` scope it to specific records. Using `--older-than` enforces a grace window so a mistaken deletion can be restored before permanent purge. In a regulated estate this window should align with the retention policy so audit-relevant artifacts are not destroyed prematurely.

Q: A UI "delete" of a fund-flow run did not free any object-storage space. Why, and what completes the cleanup?
A: The UI delete only soft-deletes — it flips the run's lifecycle stage to `deleted` in the backend store but leaves the artifact files in place. Space is reclaimed only when an operator runs `mlflow gc` (with the artifact store reachable) to hard-delete the records and their artifacts. The gotcha is assuming deletion is immediate and complete; verify `gc` ran and that it targeted the artifact destination, not just the database.

Q: What is the difference between the `active` and `deleted` lifecycle stages of an MLflow run, and how do you recover a soft-deleted run?
A: `active` runs appear normally and are queryable; `deleted` runs are hidden from default listings but still exist in the backend store until garbage-collected. You recover one with `client.restore_run(run_id)` (or `mlflow.tracking.MlflowClient().restore_run(...)`), which flips it back to `active`. Recovery is only possible before `mlflow gc` permanently removes it, which is exactly why `gc` is a deliberate manual step.

Q: What does `mlflow.evaluate()` provide that manual metric logging does not, and how would you use it on a fund-flow regressor?
A: `mlflow.evaluate()` runs a standardized evaluation of a model against a labeled dataset, computing a suite of metrics (for a regressor: MAE, MSE, RMSE, R², etc.) and logging them plus diagnostic artifacts to the active run automatically. You call it with the model URI, an evaluation DataFrame, the target column, and `model_type="regressor"`. It standardizes the holdout evaluation so every candidate is scored the same way, which strengthens the promotion gate's objectivity.

Q: How does a nested run (`mlflow.start_run(nested=True)`) help organise a hyperparameter sweep over fund-flow models?
A: A parent run wraps the whole sweep while each trial opens a child run with `nested=True`, so the trials are grouped under one parent in the UI instead of cluttering the experiment as flat siblings. This mirrors the structure of a tuning job and lets you compare children while keeping the parent as the sweep's summary. Forgetting `nested=True` makes every trial a top-level run, which is the usual cause of an experiment with hundreds of unrelated-looking runs.

Q: What is the difference between `mlflow.log_param` and `mlflow.set_tag`, given both take a key and value?
A: `log_param` records an input to the run — a hyperparameter or config value — and is immutable once written for that key; `set_tag` attaches mutable organisational metadata (e.g. `git_branch`, `team`, `data_source`) that you may update. Params are part of the experiment's reproducible record; tags are for searching, grouping, and governance labels. Storing a mutable label as a param, or a hyperparameter as a tag, breaks the semantics queries rely on.

Q: Why can a metric be logged many times for one key but a param only once, and what error do you get if you re-log a param with a different value?
A: Metrics are time series by design — logging the same key with increasing `step` builds a learning curve — whereas a param is a single immutable input recorded once. Re-logging the same param key with a different value in the same run raises an `MlflowException` about a conflicting param value rather than overwriting it. This immutability is deliberate so a run's recorded inputs cannot silently change after the fact.

Q: How do you set the experiment a run belongs to, and what happens if that experiment does not exist yet?
A: Call `mlflow.set_experiment("fund_flow_forecast")` (or pass `experiment_id` to `start_run`); if the named experiment does not exist, MLflow creates it. Without setting one, runs land in the `Default` experiment, which quickly becomes an unnavigable dumping ground. Naming experiments per model purpose is what keeps the tracking server queryable as run counts grow.

Q: What is the `MLmodel` file inside a packaged MLflow Model, and what does its `flavors` section declare?
A: The `MLmodel` file is the YAML manifest at the root of the model directory; its `flavors` section lists the ways the model can be loaded — almost always `python_function` plus a framework flavor like `sklearn` or `xgboost`. Each flavor entry names the loader module and any flavor-specific config (e.g. the pickle path, the serialization format). Serving reads this manifest to decide how to reconstruct and call the model, which is why the `pyfunc` flavor is the portability guarantee.

Q: Besides `MLmodel`, what files does a packaged MLflow Model directory typically contain?
A: It contains the serialized model (e.g. `model.pkl`), dependency files `requirements.txt` and `conda.yaml` (and often `python_env.yaml`), and, when provided, an `input_example.json` and the signature embedded in `MLmodel`. Together these let any consumer rebuild the environment and call the model with the right schema. A missing or stale `requirements.txt` is the usual reason a model that loaded in training fails to load in a fresh serving container.

Q: At inference through `pyfunc`, what does MLflow signature enforcement do when an input column has the wrong name versus the wrong type?
A: For a name-based schema, MLflow matches inputs to the signature by column name and reorders them, raising an exception if a required named input is missing; for types it applies enforcement and will raise when an incompatible type is supplied. The net effect is that a mis-named or absent required feature fails loudly at the boundary rather than producing a silent wrong number. This name-based reordering also means input column order need not match training, which avoids a subtle class of bugs.

Q: Does signature enforcement apply when you load a model with `mlflow.sklearn.load_model`, and what is the practical implication?
A: No — schema/signature enforcement applies when the model is invoked through the `pyfunc` interface (or MLflow's serving tools), not when loaded natively via `mlflow.sklearn.load_model`, which returns the raw estimator with no input validation. The implication is that your governed serving path should call models through `pyfunc` to get the schema guardrail, reserving the native loader for debugging. Bypassing `pyfunc` in production silently drops the schema check that catches upstream fund-data drift.

Q: What is an `input_example` logged with a model, and how does it relate to the signature?
A: An `input_example` is a small sample of representative input saved alongside the model; when you pass it to `log_model`, MLflow can infer the signature from it automatically and also stores it for documentation and serving validation. It doubles as a ready payload for testing the `/invocations` endpoint. Logging one is a cheap way to both pin the schema and give the next person a working example of the expected fund-flow feature frame.

Q: How do you infer a signature explicitly rather than relying on autologging, and why might you?
A: Build it with `signature = infer_signature(input_df, model.predict(input_df))` and pass `signature=signature` to `log_model`. You do this when autolog is off, when the inferred schema needs adjusting, or when you want the signature to reflect a curated input frame rather than whatever happened to be passed. An explicit signature makes the model's input/output contract a deliberate, reviewed artifact instead of an accident of the training code.

Q: How do you authenticate to a tracking server that has basic auth enabled, and where do the credentials go?
A: Set `MLFLOW_TRACKING_USERNAME` and `MLFLOW_TRACKING_PASSWORD` (or a token via `MLFLOW_TRACKING_TOKEN`) in the client environment; the client sends them with each request to the protected server. Hard-coding them in scripts is the gotcha — they belong in environment variables or a secret manager so they are not committed. For a fund platform you front the server with auth so run and registry access is controlled and attributable.

Q: Why might you put MLflow's tracking server behind a reverse proxy with TLS in a fund deployment?
A: A reverse proxy terminates TLS so run metadata, artifacts, and credentials travel encrypted, and it centralizes authentication, rate limiting, and access logging in front of a server whose built-in auth is minimal. This brings the tracking server up to the network-security bar expected of a system holding regulated audit data. Exposing the raw `mlflow server` on a plain HTTP port inside or beyond the platform is the misconfiguration to avoid.

Q: What is the role of the `--default-artifact-root` versus `--artifacts-destination` flags when starting the server?
A: `--default-artifact-root` is the legacy/general setting for where new experiments store artifacts (clients may write there directly), while `--artifacts-destination` is used with proxied serving (`--serve-artifacts`) as the server-side location behind `mlflow-artifacts:` URIs. Mixing them up causes artifacts to be written where clients cannot read them back. The clean modern setup uses proxied serving with `--artifacts-destination` so clients never need direct storage credentials.

Q: A teammate's runs show up in the registry's experiment list but their `champion` alias change is not visible to your client. What do you check first?
A: First confirm both of you point `MLFLOW_TRACKING_URI` at the same tracking server backed by the same backend store — alias state lives in the registry's backend database, so divergent URIs (one local `./mlruns`, one remote) explain the split view. Then verify the registry is enabled on that server and you are querying the same registered-model name. A registry split is almost always a tracking-URI or backend-store-config mismatch, not a propagation delay.

Q: How would you copy a model version from one MLflow registry to another (e.g. staging to production registries)?
A: Use `mlflow.register_model` against the source model's artifact URI while pointing the client at the destination tracking/registry server, or use the `copy_model_version` client API where supported, which clones the version (and artifact) into the target registry. This is how you promote across separate environments without retraining. The gotcha is that a registry reference alone does not move artifacts — the artifact must be reachable or copied, or the new version will register but fail to load.

Q: What is the difference between `runs:/<run_id>/model` and `models:/<name>@<alias>` URIs, and when do you use each?
A: A `runs:/` URI points at an artifact path within a specific run — the raw logged model before any registry governance — while a `models:/` URI resolves through the registry to a version, optionally via an alias. You use `runs:/` for ad hoc loading or to register a model, and `models:/name@champion` in production so promotion is a registry decision, not a code change. Hard-coding a `runs:/` URI in production is the anti-pattern that couples serving to a single run and bypasses the registry.

Q: How do you load a model pinned to an exact registry version rather than an alias, and when is that appropriate?
A: Use `mlflow.pyfunc.load_model("models:/fund_flow_forecast/3")` to load version `3` explicitly. This is appropriate for reproducing a past result, for an audit re-run of a specific NAV-impacting version, or in a test that must not move when `champion` is reassigned. Production scoring, by contrast, should use the alias so promotion and rollback need no redeploy.

Q: How does ECB TRIM relate to SR 11-7, and why does the lesson group them as model-risk vocabulary?
A: ECB TRIM — the Targeted Review of Internal Models — is the ECB's supervisory project and guide setting out expectations for banks' internal models (credit, market, and counterparty credit risk) and general model governance, including an independent validation function. It is the European counterpart to the US Fed's SR 11-7: both demand a model inventory, independent validation, and governance with clear accountability. The lesson groups them because a Luxembourg-supervised firm meets European expectations framed in TRIM/ECB language while SR 11-7 supplies the broader principles regulators borrow.

Q: What does TRIM expect of an "independent validation function", and what is the MLflow analogue?
A: TRIM expects validation to be carried out by a function independent of model development, performing a consistent, meaningful assessment of model performance with both qualitative and quantitative methods. The MLflow analogue is an independent named approver — not the model's author — signing off the promotion gate with the metric evidence recorded as version tags. Recording who validated and what evidence they saw is how you make that independence auditable rather than asserted.

Q: Why does the lesson say ML models "join the same inventory as pricing models" specifically in TRIM/SR 11-7 terms?
A: Both frameworks require a complete model inventory covering every model that feeds decisions, regardless of technique, with owners, purpose, and validation status. A machine-learning fund-flow forecast that influences a NAV-impacting figure is a model in that sense and cannot sit in a separate, ungoverned tool. The architect's job is to make the MLflow registry a slice of that single inventory so an inspector sees ML and valuation models under one control regime.

Q: How would you encode a model's TRIM/SR 11-7 risk rating in the registry so it is queryable?
A: Set a registered-model or version tag such as `risk_rating=high` alongside `owner`, `purpose`, and `validation_status` tags, so the inventory can be filtered by materiality. Because tags are structured backend-store metadata, a risk officer can query for all `high` risk models and check each has a current validation tag. This turns the registry into a queryable, materiality-aware model inventory rather than a flat list of names.

Q: A served model returns predictions but they are systematically shifted by a constant. Distribution monitoring is green. What MLflow-level cause do you check?
A: Check for an environment or version mismatch between training and serving — a different library version reconstructing the model, or a serving image whose `pyfunc` environment was not rebuilt from the model's `requirements.txt`, can change preprocessing or numeric behaviour. Confirm the served version's logged MLflow/library versions match the runtime, ideally by rebuilding via `--env-manager virtualenv`. Distribution monitoring stays green because inputs are unchanged; the defect is in the serving runtime, which is why pinning environments matters.

Q: `mlflow.pyfunc.load_model` raises a dependency or pickling error for a previously working version. What is the likely cause and fix?
A: The serving environment no longer matches the model's recorded dependencies — a library was upgraded, so the pickle or flavor metadata cannot be deserialized. The fix is to recreate the model's environment from its packaged `requirements.txt`/`conda.yaml` (e.g. let MLflow build it with `--env-manager`) instead of loading into a drifted interpreter. This is the concrete form of the "logged under one version, loaded under another" gotcha the lesson warns about.

Q: How do you list all versions of a registered model and see which one each alias points to via the client API?
A: Call `client.search_model_versions("name='fund_flow_forecast'")` to list versions, and `client.get_registered_model("fund_flow_forecast").aliases` (or `get_model_version_by_alias`) to read the alias-to-version mapping. Together they give the full version history and the current `champion`/`challenger` assignments. This is the query backing "what is live now and what was live before", which is the audit view a CSSF inspector asks for.

Q: What does `client.search_runs` with a `filter_string` like `metrics.rmse < 0.4 and tags.validation_status = 'approved'` return, and why is it governance-relevant?
A: It returns the runs that both beat the RMSE threshold and carry the approved validation tag, as a filtered result set. Governance questions — "show me every candidate that met the bar and passed validation" — become a single structured query rather than a manual review. This is why consistent metric and tag conventions matter: they make the audit trail searchable instead of anecdotal.

Q: How would you wire an MLflow training job into Dagster so the run is part of the lineage graph?
A: Implement the training as a Dagster asset (or op) whose function opens an `mlflow.start_run()`, logs params/metrics/model, and returns the run or model URI as the asset's output/metadata. Dagster then records the materialization and the MLflow run id together, so the model version links to the upstream data assets it consumed. This keeps "what data trained which model version" inside one lineage graph instead of a hidden second scheduler.

Q: Why is it better for Dagster to own the training schedule while MLflow owns the run record, rather than scheduling training inside MLflow?
A: MLflow tracks and registers; it is not an orchestrator, so scheduling, dependencies, retries, and backfills belong in Dagster where the rest of the platform's lineage lives. Letting MLflow or a cron drive training creates a second hidden scheduler and splits the audit trail. The clean boundary — orchestrator owns when, MLflow owns what-was-produced — is the lesson's integration stance applied to scheduling.

Q: How do you record in the MLflow run exactly which Delta/Iceberg table snapshot a fund-flow model trained on?
A: Log the table identifier and its version as a param or tag — e.g. `data_table=silver.fund_flows`, `data_version=412` (Delta version) or the Iceberg snapshot id — and optionally use MLflow's dataset tracking via `mlflow.log_input`. This pins the model to an immutable snapshot so "trained on what data?" resolves to an exact, time-travelable version. Logging only "the latest table" is the reproducibility-breaking gotcha when that table changes later.

Q: What does `mlflow.log_input` (dataset tracking) add over logging a data path as a tag?
A: `mlflow.log_input` records a structured dataset entry on the run — name, source, schema, and an optional context like `training` or `validation` — rather than a free-text tag, so the data lineage is first-class and queryable in the UI. It captures the dataset's digest, helping detect if the "same" source actually changed. For governance this gives a typed link from model version to dataset, closer to true end-to-end lineage than a string tag.

Q: A challenger beats the champion on overall RMSE but is much worse on the largest funds. How should the promotion gate handle this?
A: The gate should evaluate segmented metrics — error broken down by fund size, share class, or materiality — not just a single aggregate, because a model worse on high-AUM funds can do more NAV-impacting harm despite a better average. Record the segmented results as version tags/artifacts so the approver sees them. Promoting on aggregate RMSE alone is the overfitting-to-the-average trap that a validator's effective challenge is meant to catch.

Q: How would you implement shadow scoring for a challenger fund-flow model without affecting production output?
A: Keep `champion` serving the official numbers, and in the same batch job also load `models:/fund_flow_forecast@challenger`, score the identical inputs, and write its predictions to a separate comparison table — never to the figures consumers use. After enough periods you compare challenger versus champion error against realized flows before promoting. Because both are loaded by alias, the cutover is later a single alias move with no code change.

Q: For a monthly batch fund-flow forecast in Dagster, sketch how the scoring asset uses the registry.
A: The asset loads `model = mlflow.pyfunc.load_model("models:/fund_flow_forecast@champion")`, reads the period's governed Silver/Gold feature table, calls `model.predict(features)`, and writes results to an output table with the model version recorded as run metadata. Because it loads by alias, a promotion automatically flows into the next run with no redeploy. This keeps serving inside the lineage graph and avoids an always-on endpoint for a monthly cadence.

Q: Why is it important to record the model version number (not just the alias) on each batch scoring run's output?
A: Aliases are mutable, so recording only `champion` loses which concrete version produced a given month's numbers once the alias later moves. Persisting the resolved version id (e.g. via `get_model_version_by_alias` at run time) makes every output row attributable to an immutable version for audit. This is what lets you answer "which version produced the figure dated X?" even after several promotions.

Q: How do you delete (deprecate) an obsolete alias such as a stale `challenger`, and why not just leave it?
A: Call `client.delete_registered_model_alias("fund_flow_forecast", "challenger")` to remove the pointer. Leaving a stale alias risks code or a job still loading `@challenger` and silently scoring with an abandoned candidate, or confusing an auditor about what is actually live. Cleaning up aliases keeps the registry's "what is deployed" view honest, while the underlying versions remain immutable for history.

Q: What governance problem arises if the same person both trains the model and moves the `champion` alias, and how do you prevent it?
A: It collapses development and approval into one party, defeating the independent "effective challenge" TRIM and SR 11-7 require, and leaving no objective sign-off behind a NAV-impacting model. Prevent it with access controls so only a validation/approver role can set the production alias, and require the approver tag to name someone other than the author. Segregation of duties on the alias move is the control that makes promotion a genuine gate.

Q: How would you detect data drift in MLflow-served fund-flow features without ground-truth labels yet?
A: Compute distribution statistics (mean, variance, population-stability-index or KS distance) of live input features against the training distribution baseline and log them as metrics to a monitoring run or your observability stack. Because labels are not needed, this catches covariate shift early — e.g. a new share class shifting inflow distributions — before realized flows arrive. It is the input-side signal that complements label-based error monitoring for concept drift.

Q: Why is population stability index (PSI) a common drift metric, and roughly how is a PSI value interpreted?
A: PSI compares the binned distribution of a feature now versus a baseline, summing the divergence across bins into one number, making it easy to threshold and track over time. Conventionally a PSI under about `0.1` indicates little shift, `0.1`–`0.25` a moderate shift worth investigating, and above `0.25` a significant shift likely warranting action. For fund flows you'd PSI-monitor key inputs and trigger investigation (not blind retraining) when the threshold is breached.

Q: A retrained challenger fails the promotion gate's RMSE threshold but the team wants to ship it anyway for "better recent behaviour". How should the architect respond?
A: Hold the gate: shipping below the agreed objective threshold without recorded justification undermines the control and the audit trail. If recent behaviour genuinely matters, change the gate deliberately — e.g. add an out-of-time recent-window metric with its own threshold and a documented approver decision — rather than waiving the existing one informally. Any exception must itself be a recorded, approved event so the registry still answers "why was this promoted?".

Q: How do you reconstruct, for an audit, which model version was `champion` on a specific past date when the alias has since moved twice?
A: Combine the immutable per-run/per-scoring output metadata (which records the resolved version that produced each dated figure) with the registry's version history, since aliases are mutable and only show the current pointer. If you persisted the resolved version on each batch output, the date-to-version mapping is a direct lookup; otherwise you rely on logged alias-change history and run timestamps. This is exactly why recording the resolved version on every scoring run is non-negotiable for audit.

Q: Summarise the minimal architect's checklist to make an MLflow deployment "regulator-ready" for a fund platform.
A: Self-host the tracking server with a Postgres backend and governed artifact store behind TLS/auth; pin MLflow and model dependencies via `uv` so versions are reproducible; log per run the params, metrics on an out-of-time window, data snapshot, signed model artifact, and source commit; gate promotion on a metric threshold plus an independent named approver recorded as tags; reference production by alias and persist the resolved version on each scoring output; and tag versions with owner, risk rating, and validation status so the registry is a queryable slice of the firm's model inventory. With those in place, "which version produced this number, on what data, approved by whom" is a single set of registry queries — the TRIM/SR 11-7 answer a CSSF inspector wants.
