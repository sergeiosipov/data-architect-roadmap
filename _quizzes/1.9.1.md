# Phase 4 · 1.9.1 + 1.9.2 + 1.9.4 (with 1.9.3, 1.9.5, 1.9.6) Streaming semantics — the Beam model — 150 self-test questions

Q: What is "event time" in a streaming system?
A: Event time is the timestamp at which an event actually occurred at its source — for example the moment a fund order was placed — and it is embedded in the record itself. It is intrinsic to the data and does not change no matter when or how many times the event is processed. Because it is deterministic, event time is the basis for correct, reproducible windowed results regardless of arrival order.

Q: What is "processing time" in a streaming system?
A: Processing time is the wall-clock time of the machine executing the operation when it observes the event, read from the system clock. It is simple and low-latency but non-deterministic: the same input replayed later yields different processing-time windows. It depends on pipeline speed, backlog, and machine scheduling rather than on when the event truly happened.

Q: Why is event time the default temporal domain for regulated finance?
A: Regulated finance demands reproducible, auditable results: a NAV or an intraday fund-flow total must be the same whether computed live, replayed after a crash, or recomputed during a restatement. Only event time gives that determinism, because it groups data by when transactions actually occurred rather than by the accident of when the pipeline saw them. Processing-time results would shift on every replay, which is unacceptable when the number is regulatory evidence.

Q: What is the "skew" between event time and processing time?
A: Skew is the lag between when an event occurred (event time) and when the system observed it (processing time), i.e. the horizontal distance between the two domains on a time-mapping diagram. It is variable and unpredictable, driven by network delays, batching, retries, and sources that go offline. Streaming correctness machinery — watermarks, lateness, triggers — exists precisely to cope with this non-zero, fluctuating skew.

Q: Give a fund example where event time and processing time differ materially.
A: A subscription order is timestamped at the cut-off (event time 16:00) but the transfer-agency feed only delivers it to Kafka at 16:45 because of a batch upload (processing time 16:45). If you aggregated intraday flows by processing time, that order would land in the wrong interval; by event time it correctly falls into the 16:00 window. The 45-minute gap is the skew.

Q: Akidau frames streaming with four questions. What does the "what" question ask?
A: "What results are being computed?" describes the transformations applied to the data — sums, counts, joins, machine-learning models, and so on. It is the classic data-processing concern shared with batch, answering what value you derive from each group of records. For a fund metric this is, for example, "net flow = subscriptions minus redemptions per share class".

Q: In the what/where/when/how framing, what does "where" ask?
A: "Where in event time are results computed?" is answered by windowing — how the unbounded stream is sliced along the event-time axis into finite groups. It determines which events are aggregated together (e.g. one tumbling window per minute). Without windowing, an infinite stream never produces a final group, so "where" is what makes aggregation tractable.

Q: In the what/where/when/how framing, what does "when" ask?
A: "When in processing time are results materialised?" is answered by triggers — they decide the moments at which a window emits output (early, on-time at the watermark, and late). The same window can fire multiple times, so "when" controls the latency-versus-completeness tradeoff of each emission. It is distinct from "where", which is about event-time grouping, not emission timing.

Q: In the what/where/when/how framing, what does "how" ask?
A: "How do refinements of results relate?" is answered by the accumulation mode — discarding, accumulating, or accumulating-and-retracting. It defines whether each firing replaces, adds to, or corrects the previous output for the same window. This matters when a window fires more than once and a downstream sink must combine those panes correctly.

Q: Name the three windowing types Akidau describes.
A: Fixed (also called tumbling) windows, sliding windows, and session windows. Fixed windows slice event time into contiguous non-overlapping segments of equal length; sliding windows have a fixed length and a fixed slide period and may overlap; session windows are bounded by gaps of inactivity. Each fits a different shape of aggregation question.

Q: What is a tumbling (fixed) window?
A: A tumbling window slices the event-time axis into contiguous, equal-length, non-overlapping segments, so every event belongs to exactly one window. For example one-minute tumbling windows produce 16:00:00–16:00:59, 16:01:00–16:01:59, and so on. It is the natural fit for periodic totals such as "net fund flow per minute".

Q: What is a sliding window and how does it differ from a tumbling window?
A: A sliding window has a fixed length plus a fixed slide period; when the slide is smaller than the length the windows overlap, so a single event can fall into several windows. A tumbling window is the special case where slide equals length, giving no overlap. Use sliding windows for moving aggregates, e.g. a 15-minute exposure average recomputed every minute.

Q: What is a session window and when does it fit a fund metric?
A: A session window groups events separated by less than a configured gap of inactivity, closing only after a silence longer than that gap; window boundaries are data-driven, not clock-driven. It fits bursty activity such as a single investor's interaction session or a clustered run of corrections to one order. The gap timeout defines when one session ends and the next begins.

Q: In Flink, which assigner creates one-minute tumbling event-time windows?
A: `TumblingEventTimeWindows.of(Duration.ofMinutes(1))`, passed to `.window(...)` on a keyed stream. It assigns each event to exactly one non-overlapping minute-aligned window based on its event-time timestamp. The event-time variant (rather than the processing-time one) is what makes the result deterministic under replay.

Q: In Flink, how do you declare a sliding event-time window?
A: With `SlidingEventTimeWindows.of(size, slide)` — for example `SlidingEventTimeWindows.of(Duration.ofMinutes(15), Duration.ofMinutes(1))` for a 15-minute window advancing every minute. Because the slide is smaller than the size, the windows overlap and each event is assigned to multiple windows. Overlap multiplies the per-event work, so size/slide ratio drives cost.

Q: In Flink, how is a session window's gap configured?
A: With `EventTimeSessionWindows.withGap(Duration)` — for example `withGap(Duration.ofMinutes(5))` closes a session once five minutes of event-time inactivity pass. Sessions can also merge: a late element arriving in the gap between two sessions can join them into one. This merging is why session windows are stateful and more complex than tumbling ones.

Q: What is a watermark?
A: A watermark is a timestamp `t` that flows in the stream and asserts "event time has progressed to `t`," meaning the system expects no further elements with event timestamps less than or equal to `t`. It is how an out-of-order stream signals event-time completeness so that operators know when a window is safe to close and emit. Watermarks are the heuristic that turns "we will never see all the data" into actionable progress.

Q: What is the difference between a perfect and a heuristic watermark?
A: A perfect watermark guarantees that absolutely no element with a timestamp at or before it will ever arrive — achievable only when the input is fully predictable. A heuristic watermark is an estimate that no more such elements will arrive, accepting that some stragglers may still come (those are late data). Real distributed sources almost always use heuristic watermarks, because true completeness cannot be known in advance.

Q: What is the latency-versus-completeness tradeoff in watermarks?
A: A conservative watermark waits longer before declaring an event-time point complete, capturing more out-of-order data but delaying every result; an aggressive watermark fires sooner with lower latency but marks more arrivals as late. Tuning the watermark is choosing how long to wait for stragglers versus how fresh the answer must be. There is no free lunch: you trade promptness against accuracy.

Q: How does a watermark propagate through an operator with multiple input streams?
A: The operator's current event time is the minimum of the current event times across all its input channels, so a single lagging input holds back the whole operator. Only when every input has advanced past `t` can the operator's watermark advance past `t`. This min rule is why one slow or idle partition can stall downstream windows from closing.

Q: Where do watermarks originate, and how are they emitted by parallel sources?
A: Watermarks originate at the source operators, with each parallel subtask generating its own watermark independently from the data it sees. They then propagate downstream, advancing operator event-time clocks as described by the minimum-across-inputs rule. Because each subtask is independent, an idle or skewed subtask can drag the merged watermark unless idleness is handled.

Q: What is "allowed lateness" and what is its default in Flink?
A: Allowed lateness extends how long a window keeps its state after the watermark passes the window end, accepting elements that are late by up to that amount and re-firing the window for them. In Flink the default is 0, meaning once the watermark passes the window end, later elements are dropped. You set it with `allowedLateness(Duration)` on the windowed stream, trading retained state and extra firings for catching stragglers.

Q: What happens to a late element when allowed lateness is greater than zero?
A: If the element arrives before the watermark passes (window-end + allowed-lateness), it is added to the still-retained window and triggers a "late firing" that produces an updated result for that window. Once the watermark moves beyond window-end + allowed-lateness, the window state is purged and any later element is dropped or routed to a side output. So allowed lateness defines a grace zone after the on-time firing.

Q: What is a side output for late data, and how is it wired in Flink?
A: A side output captures elements that are too late to be processed in their window so they are not silently lost. You declare an `OutputTag<T>` (an anonymous subclass, e.g. `new OutputTag<Order>("late-orders"){}`), call `.sideOutputLateData(tag)` on the windowed stream, and retrieve the stream with `result.getSideOutput(tag)`. Those captured records can then be logged, alerted on, or fed into a restatement path.

Q: Why must the `OutputTag` be created as an anonymous subclass in Flink?
A: `OutputTag` carries a generic type parameter that Java erases at runtime, so Flink needs the type information preserved. Writing `new OutputTag<Order>("late"){}` (note the trailing braces) creates an anonymous subclass whose generic superclass retains the type for Flink's type extraction. Omitting the braces loses the type and causes a runtime error about missing type information.

Q: What is a restatement, and why does late data force one?
A: A restatement is the publication of a corrected result that supersedes a previously emitted one, because new data — a late or amended event — changed the answer for a window that had already fired. In finance this is routine: an intraday flow total reported at the window's on-time firing is restated when a late order arrives. Streaming systems model this with late firings plus an accumulation mode that can update or retract the earlier value.

Q: What is a trigger in the Beam model?
A: A trigger is the rule that decides, in processing time, when a window emits output — it answers the "when" question. A window can fire early (before the watermark, for speculative results), on time (when the watermark passes the window end), and late (when allowed-lateness elements arrive). Triggers therefore turn a single window into a sequence of refinements rather than one final answer.

Q: Distinguish early, on-time, and late firings.
A: An early firing emits a speculative, possibly-incomplete result before the watermark reaches the window end, trading completeness for low latency. The on-time firing happens when the watermark passes the window end, the system's best complete answer. A late firing occurs when a late element (within allowed lateness) updates the window after it already fired on time. Each firing is a "pane" of the same window.

Q: What is the "discarding" accumulation mode?
A: In discarding mode, after each firing the window's accumulated state is reset, so each pane contains only the new data seen since the previous firing. The downstream sink must add these deltas together to reconstruct the total. It minimises per-pane payload but pushes the burden of summation onto the consumer, and a lost pane silently understates the total.

Q: What is the "accumulating" accumulation mode?
A: In accumulating mode the window retains its state across firings, so each pane is the complete result so far including all earlier data plus the new arrivals. Each emission is the full current answer for the window, and a later pane simply overwrites an earlier one for the same window key. This is the easiest mode for an idempotent upsert sink, since the latest pane is authoritative.

Q: What is the "accumulating and retracting" mode?
A: Accumulating-and-retracting emits, alongside the new accumulated value, an explicit retraction of the previously emitted value for that window. This lets a downstream consumer that has already combined panes correctly undo the old contribution before adding the new one — essential when the same window feeds an aggregate further downstream. It is the most correct but most expensive mode, doubling the message volume per refinement.

Q: Which accumulation mode best matches an idempotent upsert into an Iceberg table?
A: Accumulating mode, because each firing is the full current value for the window key and an upsert keyed on (window, share class) simply replaces the prior row. Discarding mode would require the sink to add deltas, and a missed delta would corrupt the total; retracting mode is needed only when a downstream aggregate has already folded the old value in. Accumulating plus upsert gives "last write wins" with the latest pane as truth.

Q: How do allowed lateness, triggers, and watermarks interact for a single window?
A: The watermark determines the on-time firing (when it passes the window end) and, together with allowed lateness, determines when the window's state is finally purged (watermark past window-end + lateness). The trigger governs the emission moments — early before the watermark, on-time at it, late within the lateness grace zone. So watermark sets the "complete" line, lateness sets the grace, and the trigger fires output at each of those points.

Q: A junior asks why yesterday's intraday flow count was too low. What is your structured answer?
A: The likely cause is late orders that arrived after the watermark passed the window end, so they were dropped instead of counted. The fix is one of three knobs: a more conservative watermark (wait longer before closing), a larger allowed lateness (re-fire the window for stragglers), or a side output plus restatement to recover them. Each fix costs latency or extra processing, and you should name which knob and what it costs.

Q: For "catch more stragglers" the knob is allowed lateness — what does choosing a large value cost?
A: A large allowed lateness keeps each window's state in memory (or on the state backend) for longer, increasing state size and checkpoint cost, and it produces more late firings that the downstream sink must absorb as restatements. It also extends the time before a window's result is final, so consumers see updated numbers for longer. You are buying completeness with state, latency-to-finality, and write amplification.

Q: Why does a more conservative watermark reduce late data but raise latency?
A: A conservative watermark trails further behind the maximum observed event time, so it waits longer before declaring any event-time point complete, capturing more out-of-order arrivals before the on-time firing. The cost is that every window's on-time result is delayed by that extra buffer. You move the latency-completeness dial toward completeness for all windows uniformly, not just for the stragglers.

Q: What is keyed state in a stateful streaming operator?
A: Keyed state is per-key state maintained by an operator, partitioned by the stream's key so each key's state is independent and the work parallelises across keys. A windowed aggregation, for instance, keeps each share class's running total as keyed state. Because state is scoped to a key, Flink can redistribute keys across parallel subtasks during rescaling without mixing keys' data.

Q: What is a checkpoint and why does it matter for correctness?
A: A checkpoint is a consistent snapshot of all operator state plus the input positions (e.g. Kafka offsets) at a point in the stream, persisted to durable storage. On failure the job restores from the latest checkpoint and resumes, so the result after recovery matches what it would have been without the crash. This is the mechanism that lets a stateful aggregation feeding a regulated NAV survive restarts without producing different numbers.

Q: Why is event-time windowing inherently a stateful operation?
A: Because events for a window can arrive out of order and over a span of processing time, the operator must buffer and accumulate each open window's partial result until the watermark says it can fire. That retained, per-key, per-window accumulation is state. Without durable state and checkpoints, a restart would lose every open window's in-progress total.

Q: In Flink, what does `WatermarkStrategy.forBoundedOutOfOrderness(Duration)` express?
A: It declares a heuristic watermark that assumes elements arrive out of order but by no more than the given bound — the latest element for timestamp `t` arrives at most that duration after the earliest. Flink tracks the maximum observed timestamp and emits a watermark that trails it by the bound. For example `forBoundedOutOfOrderness(Duration.ofSeconds(10))` tolerates 10 seconds of disorder before counting an element as late.

Q: When would you use `WatermarkStrategy.forMonotonousTimestamps()` instead?
A: When event timestamps are strictly non-decreasing on each partition, i.e. no out-of-order arrivals are expected, such as a single-writer append log with monotonic sequence stamps. It emits a watermark equal to the largest timestamp seen, with zero added delay. Using it on genuinely out-of-order data would mark many legitimate events as late, so it is only safe when ordering is guaranteed at the source.

Q: What problem does `withIdleness(Duration)` solve?
A: It addresses idle sources: if a partition produces no events for the given duration, Flink marks it idle so it stops holding back the minimum-across-inputs watermark. Without it, one quiet Kafka partition — say a share class with no orders this minute — would freeze the merged watermark and stall every window from closing. With it, active partitions advance the watermark while idle ones are temporarily excluded.

Q: A Flink job's windows never emit results even though data is flowing. What do you check first?
A: First check the watermark: most often a timestamp/watermark assigner is missing or misconfigured, or one input partition is idle and pinning the minimum-across-inputs watermark so it never reaches any window end. Confirm a `WatermarkStrategy` is assigned with a correct `withTimestampAssigner`, and consider `withIdleness` for sparse partitions. If the watermark is not advancing, no event-time window can ever fire.

Q: What is the role of `withTimestampAssigner` in a watermark strategy?
A: It tells Flink how to extract the event-time timestamp from each record, since the watermark logic needs a per-event time to track the maximum and compute progress. For example you map an order's `tradeTimestamp` field to the event time. It is sometimes optional when a connector like Kafka already supplies record timestamps, but for a domain field like trade time you usually assign it explicitly.

Q: Distinguish periodic from punctuated watermark generation.
A: Periodic generation emits watermarks on a time interval, computing the current watermark from the maximum timestamp seen so far, which is the common default. Punctuated (per-element) generation emits a watermark in response to special marker records in the stream. Periodic is simpler and cheaper; punctuated suits streams that carry explicit "end of batch" or completeness markers.

Q: Why is processing-time windowing simpler but unsuitable for restatement-heavy finance?
A: Processing-time windows just buffer until the wall clock elapses, so completeness is trivial — the window is done when its clock interval ends — and there are no late elements by definition. But the assignment depends on pipeline speed, so the same order lands in different windows on replay, and out-of-order arrivals are mis-bucketed. For finance that needs deterministic, restatable results, that non-determinism is disqualifying.

Q: Why is the completeness problem fundamentally unsolvable for event-time windows?
A: Elements can be arbitrarily delayed — a mobile device offline for hours, a delayed batch upload — so there is no time at which you can be certain every event for a given timestamp has arrived. You can only estimate completeness with a heuristic watermark and then decide how long to wait. This irreducible uncertainty is exactly why allowed lateness and restatements exist.

Q: Map the four Beam questions to the four configuration choices you make.
A: "What" maps to the transformation/aggregation (e.g. sum of flows); "where" maps to the windowing choice (tumbling/sliding/session and size); "when" maps to the trigger (early/on-time/late firing policy, driven by the watermark); and "how" maps to the accumulation mode (discarding/accumulating/retracting). Together these four fully specify a correct streaming aggregation. A paper design should name a concrete answer for each.

Q: Design the windowing for "intraday fund-flow totals per share class, restated as late orders arrive". What type and key?
A: Use tumbling event-time windows (e.g. one-minute or daily-cut-off buckets) keyed by share class, because each interval's net flow is a discrete periodic total and each order belongs to exactly one interval. Keying by share class gives independent per-class state and parallelism. Tumbling, not sliding, because the totals are non-overlapping reporting intervals.

Q: For that same fund-flow job, what watermark strategy would you justify?
A: A heuristic bounded-out-of-orderness watermark sized to the observed feed delay — e.g. `forBoundedOutOfOrderness` of a few minutes to cover transfer-agency batch latency — with `withIdleness` so share classes that are quiet in an interval do not stall the watermark. It must be heuristic because order arrivals are not perfectly ordered. The bound is a deliberate completeness-versus-latency choice, not a guess.

Q: For the fund-flow job, what trigger and accumulation mode fit, and why?
A: Use the default watermark trigger for the on-time firing plus allowed lateness for late firings, and accumulating mode so each firing emits the full current total per share class. Accumulating pairs cleanly with an idempotent upsert into Iceberg keyed on (window, share class), so a late firing simply restates the row. Add a side output to capture orders too late even for the lateness window, for an explicit restatement path.

Q: Where does a late order land relative to the watermark, and what does the trigger do with it?
A: If the order's event time is before the watermark but within window-end + allowed-lateness, it falls into the grace zone: it is added to the still-retained window and the trigger produces a late firing with the corrected total. If it arrives after window-end + allowed-lateness, the window state is gone, so it is dropped or routed to the side output. The watermark is the line that separates "on-time/grace" from "late/lost".

Q: State the latency cost of choosing a five-minute allowed lateness for the fund-flow job.
A: The window's result is not final for five minutes after the watermark passes its end, so any downstream consumer must treat the on-time number as provisional and accept restated values for that whole grace period. State for each open window is held five minutes longer, increasing checkpoint size, and each straggler triggers an extra restatement write. You give up promptness-to-final and some state budget to catch late orders.

Q: How does session-window merging interact with late data?
A: A late element arriving in the gap between two previously separate sessions can bridge them, causing Flink to merge those sessions into one and re-fire a combined result. This makes session windows more complex under lateness than tumbling windows, since a single late event can restructure window boundaries, not just update a total. The downstream sink must handle a merge as a retraction-plus-new-result, not a simple update.

Q: Why is a "perfect" watermark generally impossible for a Kafka-fed fund-order stream?
A: Because you cannot guarantee that no order with an earlier event time will ever arrive — feeds batch, retry, and back-fill, and an upstream TA system may resend corrections hours later. A perfect watermark requires fully predictable input completeness, which a multi-source order pipeline does not have. Hence you use a heuristic watermark plus allowed lateness and accept restatements.

Q: What is the consequence of setting the out-of-orderness bound too small?
A: A bound smaller than the real disorder means the watermark advances too aggressively, so many legitimately out-of-order orders fall behind it and are treated as late — dropped or restated rather than counted on time. Your on-time totals will be systematically understated and you will see excessive late firings or side-output volume. The symptom is a count that is "wrong yesterday" and gets corrected only via restatements.

Q: What is the consequence of setting the out-of-orderness bound too large?
A: An over-large bound makes the watermark trail far behind real time, so every window's on-time firing is delayed by that buffer even when most data arrived promptly. You pay uniform extra latency on all results to protect against rare stragglers. The fix is to size the bound to the actual observed skew distribution, not to the worst case.

Q: How would you explain to an auditor that a late-arriving order was not lost?
A: Show that the pipeline routes orders too late for the allowed-lateness window into a side output that is persisted and alerted on, and that orders within the lateness window trigger a documented restatement of the affected total. Therefore every order is either counted on time, restated, or captured for manual reprocessing — none is silently dropped. The audit trail is the side-output store plus the restatement log.

Q: Why is processing time still useful despite event time being the default?
A: Processing time is what triggers operate in — early/on-time/late firings are scheduled in processing time even though windows are defined in event time — and it underpins simple low-latency, non-restatement use cases like raw throughput monitoring. It is also the only clock available when records carry no usable event timestamp. The skill is knowing the domain each concept lives in: windows in event time, firings in processing time.

Q: What does it mean that "the watermark drives the on-time firing"?
A: The default event-time trigger emits a window's on-time result at the exact moment the watermark passes the window's end timestamp, because that is when the system believes the window is complete. So the watermark's progress, not the wall clock, decides when each window produces its main answer. If the watermark stalls, the on-time firing never happens.

Q: Contrast accumulating mode with discarding mode for a downstream that re-aggregates panes.
A: If a downstream operator folds panes into a larger aggregate, discarding mode is correct because each pane is a delta that simply adds in. Accumulating mode would double-count, since each pane re-includes earlier data, unless the downstream replaces rather than adds. Accumulating-and-retracting is the safe general choice for such re-aggregation, as it explicitly undoes the prior pane before adding the new one.

Q: Why might you choose early firings for a fund-exposure dashboard?
A: Early firings emit a speculative window result before the watermark closes it, giving the dashboard a low-latency provisional number that refines as more data arrives. For an exposure monitor where seeing an approximate figure now beats an exact figure minutes later, that tradeoff is worth it. You accept that the early value is incomplete and will be updated by the on-time and any late firings.

Q: What is a "pane" in window terminology?
A: A pane is the output produced by a single firing of a window — so a window that fires early, on time, and late produces three panes for the same window. The accumulation mode defines how those panes relate (delta, full value, or value-plus-retraction). Thinking in panes makes explicit that a window result is a sequence of refinements, not a one-shot answer.

Q: Why is "the same rules govern Flink as the Beam model" a useful mental model?
A: Because event time, watermarks, windows, triggers, and accumulation are engine-agnostic concepts from the Beam model, and Flink is one concrete implementation of them — `WatermarkStrategy`, `allowedLateness`, `OutputTag`, and the window assigners are just Flink's names for those ideas. Designing on paper in Beam terms and then mapping to Flink APIs keeps the design portable and the reasoning sound. The concepts transfer to Spark, Dataflow, and managed Flink alike.

Q: How does keyed state enable parallelism in a windowed aggregation?
A: Because state is partitioned by key, each key's accumulation is independent, so the framework can distribute different keys across parallel subtasks and process them concurrently without coordination. A flow-total job keyed by share class therefore parallelises across share classes. This also lets the job rescale by reassigning key groups, since no key's state is entangled with another's.

Q: Why does a watermark that never advances usually point to an assigner or idleness problem?
A: The watermark is computed from observed event timestamps; if no `WatermarkStrategy`/timestamp assigner is attached, there is nothing to advance it, and if one input partition is idle, the minimum-across-inputs rule pins the merged watermark to that stalled partition. Either way windows wait forever. The two first checks are "is a timestamp assigner present and correct?" and "is a partition idle without `withIdleness`?".

Q: What is the difference between "where" (windowing) and "when" (triggering) in one sentence each?
A: "Where" decides which events are grouped together along the event-time axis — the window assignment. "When" decides at which processing-time moments each window emits output — the trigger and its early/on-time/late firings. One is grouping in event time, the other is emission timing in processing time, and conflating them is a common source of design confusion.

Q: An order arrives with an event time inside an already-fired, already-purged window. What happens by default?
A: With Flink's default allowed lateness of 0, the window state was purged at the on-time firing, so the late order is dropped and not counted. To capture it you must either configure `allowedLateness` so the window is retained long enough, or attach `sideOutputLateData` to divert it for reprocessing. Out of the box, "no late handling" means "silently dropped", which is the gotcha to design against.

Q: Why is `allowedLateness` a per-window-state cost, not just a config flag?
A: Each window must keep its accumulated state alive until the watermark passes window-end + allowed-lateness, so a larger value multiplies the number of simultaneously-open windows holding state. That inflates heap or RocksDB usage and every checkpoint. So allowed lateness is a real resource decision sized against straggler frequency, not a costless safety margin.

Q: Give a fund example distinguishing event time from processing time, one each.
A: Event time: a redemption order's official cut-off timestamp of 16:00 is its event time and determines which trading-day window it belongs to. Processing time: the moment the Flink operator actually consumes that order from Kafka, say 16:07, is its processing time and governs when triggers fire. The 16:00 figure is fixed and audit-relevant; the 16:07 figure shifts on every run.

Q: How do triggers let you serve both a fast dashboard and a correct report from one window?
A: An early-firing trigger feeds the dashboard a speculative low-latency value, the on-time firing at the watermark gives the report its best complete value, and late firings within allowed lateness restate it as stragglers arrive. The single window thus emits a refinement sequence consumed differently by each downstream. Accumulating mode makes the latest pane authoritative for both.

Q: Why must a streaming design name all four — windowing, watermark, trigger, accumulation — to be "ready to implement"?
A: Because each one independently changes the result: the window decides grouping, the watermark decides the completeness line and on-time firing, the trigger decides emission moments and lateness behaviour, and accumulation decides how refinements combine. Leaving any unspecified means the implementation will silently pick a default that may be wrong for the metric. A complete paper design pins down one concrete choice for each with a one-sentence justification.

Q: What does "byte-identical results after restart" demand from the streaming model?
A: It demands event-time semantics plus durable checkpointed state, so that after restoring from a checkpoint the operator re-derives exactly the same windows and totals it had before the crash. Processing-time windows could not be byte-identical because re-consumption happens at different wall-clock times. Determinism comes from grouping by event time and resuming from a consistent state snapshot.

Q: How do watermarks relate to the "completeness problem"?
A: The completeness problem is that you can never be certain all data for an event-time point has arrived; the watermark is the system's heuristic answer, declaring "I believe completeness has reached `t`" so windows up to `t` may fire. It does not solve the problem — late data can still arrive — but it makes progress decidable. Allowed lateness then covers the gap between the heuristic and reality.

Q: Why is choosing the windowing type "a data-model decision", not just a tuning knob?
A: The window type encodes what a "result" means for the metric: tumbling for discrete periodic totals, sliding for moving averages, session for activity bursts. Pick the wrong shape and the number answers a different business question, regardless of how well you tune watermarks. So the architect chooses windowing from the metric's semantics first, then tunes lateness and triggers.

Q: A sliding-window exposure metric is doubling CPU versus the tumbling version. Why, and what is the lever?
A: Because sliding windows overlap, each event is assigned to multiple windows (roughly size/slide of them), multiplying per-event aggregation work and state. A 15-minute window sliding every minute puts each event in about 15 windows. The lever is the size-to-slide ratio: widen the slide to reduce overlap, or switch to tumbling if true overlap is not needed.

Q: What is the danger of using processing-time windows for an intraday NAV-flow total?
A: Out-of-order or delayed orders get bucketed by when they were processed, not when they occurred, so an order placed at 15:59 but delivered at 16:01 lands in the wrong interval and corrupts both totals. Replays produce different buckets, breaking reproducibility and auditability. For a regulated flow total this is a correctness defect, not a latency nuisance.

Q: How does the minimum-across-inputs watermark rule cause a "stuck" join or union?
A: A keyed join or union takes the minimum watermark of its inputs, so if one input stream lags or is idle, the operator's event-time clock — and therefore all its windows — cannot advance past that laggard. The whole result stalls even though other inputs are flowing. The remedy is `withIdleness` on idle inputs or fixing the slow source.

Q: Why is "event time is the regulated-finance default" not merely a preference?
A: Because regulators and reconciliation processes require that the same inputs always yield the same outputs and that historical results be reproducible during restatements and audits — only event-time grouping delivers that determinism. Processing-time results depend on run conditions and cannot be defended as "the" answer for a given trading day. Event time makes the result a function of the data alone.

Q: In one sentence, what does each Beam question answer for a NAV-flow job?
A: What = net subscriptions minus redemptions per share class; where = one tumbling window per reporting interval keyed by share class; when = on-time firing at the watermark plus late firings within allowed lateness; how = accumulating mode so each firing emits the full restated total. That fully specifies a correct, restatable design ready to code in Flink.

Q: What is the relationship between a heuristic watermark and the volume of side-output (very-late) records?
A: The more aggressive (earlier) the heuristic watermark relative to true disorder, the more records fall past window-end + allowed-lateness and land in the side output as too-late; a more conservative watermark reduces that volume but delays results. Side-output depth is thus a tuning signal — a growing late-output stream means your watermark or lateness is mis-sized for the feed. You read it as a feedback metric on your completeness assumptions.

Q: Why can the same logical window fire more than once, and what must the sink tolerate?
A: It fires once early (speculative), once on-time (at the watermark), and potentially several times late (within allowed lateness), because new data keeps changing the answer. The sink must therefore tolerate multiple emissions for the same window key — typically by upserting in accumulating mode so the latest pane overwrites prior ones. A sink that blindly appends would duplicate and inflate the metric.

Q: How does allowed lateness differ from the out-of-orderness bound in the watermark?
A: The out-of-orderness bound delays the watermark itself, affecting the on-time firing of every window uniformly and what counts as "on time". Allowed lateness keeps a window's state alive after its on-time firing, governing how long late elements can still update it before being dropped. One shifts the completeness line; the other extends the grace period after that line — they are independent knobs you tune together.

Q: Why is "event time vs processing time" called the central distinction of streaming?
A: Because almost every correctness subtlety — windows, watermarks, lateness, restatements — exists only because event time and processing time diverge by an unpredictable skew. If data always arrived instantly and in order, processing-time windows would be correct and none of the machinery would be needed. The whole Beam model is, in effect, a disciplined way to reconcile the two time domains.

Q: A consumer of your window output is double-counting flows. What mode mismatch is the likely cause?
A: The producer is likely emitting accumulating panes (each pane is the full running total) while the consumer is adding panes together as if they were discarding-mode deltas, so earlier data is counted repeatedly. The fix is to align the contract: either switch the producer to discarding, or have the consumer replace (upsert) rather than add. Accumulation mode is part of the interface, not an internal detail.

Q: How would you preview the role of checkpoints when teaching the Beam model, before the Flink lesson?
A: Explain that windows hold per-key accumulated state that must survive failures, and that a checkpoint is a consistent snapshot of that state plus input offsets, taken periodically and restored on restart. This lets a stateful aggregation resume exactly where it left off, so results are reproducible across crashes — the detailed barrier mechanism is deepened in the Flink entry. The takeaway is that stateful streaming correctness depends on durable, consistent snapshots.

Q: Why does an EMT or EPT data feed benefit from event-time semantics if streamed?
A: EMT/EPT files carry reference and cost figures tied to specific effective dates and as-of points, so any streamed processing must bucket and reconcile them by their effective (event) time, not by ingestion time, to keep restatements correct. Late or corrected EMT records would otherwise mis-bucket against the wrong reporting period. Event-time windowing plus restatement is what keeps the feed's downstream consumers consistent with the file's stated dates.

Q: How does a watermark let you bound, but not eliminate, the wait for completeness?
A: The watermark sets a defensible point at which you stop waiting and fire, encoding your assumption about maximum disorder, so results are produced with bounded latency. It cannot eliminate the wait-for-completeness uncertainty because genuinely late data can still appear afterwards. That residual is handled by allowed lateness and side outputs rather than by the watermark itself.

Q: Why is "restatement-heavy finance demands event-time rigor" the motivating theme of this lesson?
A: Because fund metrics are routinely corrected after the fact — late orders, amended trades, NAV adjustments — and each correction must reproducibly re-derive the affected window from the data, which only event time guarantees. The lesson's machinery (watermarks, lateness, triggers, accumulation) is exactly what makes those restatements principled rather than ad hoc. The architect's job is to choose those knobs so corrections are automatic and auditable.

Q: What two checks form the first move when a windowed result looks wrong?
A: First, is the result wrong because data was dropped as late — check the watermark progression and the allowed-lateness/side-output configuration. Second, is the grouping wrong because of processing-time vs event-time assignment or an idle-partition-stalled watermark — verify the timestamp assigner and idleness handling. Most "wrong window" incidents resolve to one of these two: late-data loss or watermark stall.

Q: Why is naming the accumulation mode part of the data contract between teams?
A: Because a consumer cannot correctly combine multiple panes of a window without knowing whether each pane is a delta (discarding), a full value (accumulating), or a value-plus-retraction (retracting). Get it wrong and you silently under- or over-count. So the producing team must publish the accumulation mode just as it publishes the schema, since it changes how the output must be read.

Q: How do session windows differ from tumbling windows in determinism under late data?
A: Tumbling windows have fixed, data-independent boundaries, so a late element only updates one window's total. Session windows have data-driven boundaries, so a late element can merge two sessions and restructure the grouping itself. Sessions are therefore harder to reason about and restate, since lateness can change which events belong together, not just the value.

Q: What is the practical meaning of "the watermark is a heuristic" for capacity planning?
A: It means you are choosing how long to buffer out-of-order data, and that choice directly sizes how much window state is held open and how big checkpoints get. A more conservative watermark buffers more open windows; a larger allowed lateness extends their lifetime further. So watermark and lateness tuning is also a memory- and checkpoint-capacity decision, not only a correctness one.

Q: Why can two correct streaming jobs over the same data produce different "on-time" numbers?
A: Because they may set different watermark bounds and allowed-lateness values, so they draw the completeness line at different points and thus include different sets of out-of-order events in their on-time firing. Both are internally consistent; they simply encode different completeness-versus-latency tradeoffs. Only after each has caught all its late data (via restatement) do they converge on the same final total.

Q: How does the Beam model treat a "final" answer for a window?
A: It treats finality as conditional: a window's result is final only once the watermark has passed window-end + allowed-lateness and the state is purged, after which no further refinement is possible. Before that, every emission is provisional and may be restated by a later firing. So "final" is a defined point in the lifecycle, not the first time the window fires.

Q: Why does choosing tumbling windows aligned to the fund's cut-off times matter?
A: Aligning window boundaries to the official dealing cut-off makes each window correspond to a regulatory reporting interval, so the aggregate answers exactly the question compliance asks. Misaligned windows would split a trading period across buckets and require error-prone reassembly. Boundary alignment is therefore part of getting the "where" question right for fund data.

Q: Summarise the exact knob-to-symptom mapping for a "count was too low yesterday" incident.
A: If stragglers were dropped after the on-time firing, the fix is larger allowed lateness (cost: more state and restatements). If many events were behind an over-aggressive watermark, the fix is a larger out-of-orderness bound (cost: higher latency on all results). If events were lost entirely past the grace period, the fix is a side output plus restatement path (cost: extra processing and a manual or automated re-feed). Name the knob and its cost, not just the cause.

Q: What is a "trigger" gotcha when combining early firings with discarding accumulation?
A: With early firings in discarding mode, each pane carries only the data since the last firing, so a consumer that displays a single pane as "the total" will show a fraction of the real value. Discarding is meant for consumers that sum panes; pairing it with an early-firing speculative dashboard that reads one pane in isolation understates the metric badly. The safe pairing for a "show me the current total" view is accumulating mode, where every pane is the full running value.

Q: How would you reason about a SWIFT or transfer-agency feed that delivers settlement messages in bursts hours late?
A: Such a feed has large, lumpy event-to-processing skew, so a tight watermark would mark the bursts as late and an over-wide one would delay every result; the better design is a moderate heuristic watermark plus a generous allowed lateness sized to the burst delay, with a side output capturing anything beyond it. Each burst then triggers late firings that restate the affected windows, and the side output guarantees nothing settled is silently lost. The completeness story you tell an auditor is "on-time, restated, or parked for reprocessing — never dropped".

Q: Why is the Beam model described as engine-agnostic, and why does that matter to an architect?
A: Because event time, windows, watermarks, triggers, and accumulation are conceptual primitives that Flink, Spark Structured Streaming, Beam/Dataflow, and managed services all implement under different API names. An architect who reasons in these primitives can specify a correct design once and map it to whatever engine the platform uses, and can audit any pipeline's guarantees by these same questions. It future-proofs the design against engine choice.

Q: In Flink, how would you attach an early-firing trigger to a tumbling event-time window without losing the on-time and late firings?
A: Override the default trigger on the windowed stream with `.trigger(...)`, for example a custom or built-in trigger that combines `EventTimeTrigger` semantics with periodic early fires such as `ContinuousProcessingTimeTrigger.of(Duration.ofSeconds(10))`. Replacing the trigger entirely is the gotcha — the supplied trigger must itself still fire at the window end (the watermark) so the on-time pane is not lost, and you keep `allowedLateness` for late firings. The cleanest path is the Beam-style `AfterWatermark.pastEndOfWindow().withEarlyFirings(...)` pattern, mirrored in Flink with a custom trigger.

Q: Why does wrapping a trigger in Flink's `PurgingTrigger` change the accumulation behaviour of a window?
A: `PurgingTrigger.of(innerTrigger)` makes the window clear its accumulated state immediately after each fire, so every pane contains only data seen since the previous fire — effectively turning the window into discarding mode. Without purging, the window retains state and each fire re-emits the full accumulated value (accumulating mode). So `PurgingTrigger` is the Flink lever that flips a fire-and-keep window into a fire-and-reset one, and using it under a "show me the running total" dashboard would understate the figure.

Q: A teammate set `allowedLateness(Duration.ofMinutes(10))` but late orders are still dropped. What are the two most likely causes?
A: First, the orders may be arriving after `window-end + 10 minutes` in event time, i.e. truly past the grace zone, so check the lateness gap against the actual feed delay distribution. Second, the watermark may already be far ahead because the out-of-orderness bound is too small or a `forMonotonousTimestamps` strategy was used, so events are classified late on arrival before the grace even helps; inspect the `currentInputWatermark` metric versus the order event times. Confirm with the `numLateRecordsDropped` operator metric that records are actually being dropped at that operator.

Q: Which Flink metric tells you, in production, that your watermark or lateness is mis-sized?
A: The `numLateRecordsDropped` operator metric counts records discarded for arriving past `window-end + allowedLateness`, so a rising value means stragglers are being lost rather than restated. Pair it with `currentInputWatermark` (or per-subtask `currentWatermark`) to see whether the watermark is racing ahead of real event time. A growing side-output volume is the same signal seen from the capture side rather than the drop side.

Q: How do you key a Flink stream by share class before windowing, and why must it come first?
A: Call `.keyBy(order -> order.getShareClassId())` (a `KeySelector`) on the `DataStream` before `.window(...)`; only a `KeyedStream` supports keyed windows and keyed state. Keying first is mandatory because window state, timers, and the per-key accumulation are all partitioned by key — without it you would get a single non-parallel `windowAll` with global state. The key also determines parallel distribution, so an even share-class key space gives balanced subtasks.

Q: What is the difference between `windowAll` and a keyed `window` for a fund-flow job, and why avoid the former?
A: `windowAll` applies one global window to the entire stream with parallelism forced to 1, so all orders funnel through a single subtask. A keyed `window` (after `keyBy`) maintains independent state per share class and parallelises across subtasks. For a fund-flow total you almost always want keyed windows; `windowAll` would be a throughput bottleneck and a single point of state, and only makes sense for a genuinely global aggregate with low volume.

Q: How would you compute a per-share-class flow total in Flink SQL rather than the DataStream API?
A: Use a windowing table-valued function such as `TUMBLE(TABLE orders, DESCRIPTOR(event_time), INTERVAL '1' MINUTE)` and then `GROUP BY window_start, window_end, share_class`, with the source table declaring a `WATERMARK FOR event_time AS event_time - INTERVAL '2' MINUTE`. Flink SQL handles the watermark, windowing, and accumulation declaratively, emitting one row per (window, share class). The tradeoff is less control over custom triggers and side outputs than the DataStream API gives.

Q: In Flink SQL, how is the watermark declared and what does the subtracted interval mean?
A: In the `CREATE TABLE` DDL you write `WATERMARK FOR event_time AS event_time - INTERVAL '2' MINUTE`, which defines a bounded-out-of-orderness watermark trailing the maximum observed `event_time` by two minutes. The subtracted interval is exactly the disorder tolerance: events up to two minutes out of order are still on time. Set it too small and legitimate orders are late; too large and every windowed result is delayed by two minutes.

Q: A junior confuses "watermark" with "trigger". Give the one-line corrective using a fund example.
A: The watermark is the event-time progress signal that decides when the 16:00 flow window is believed complete; the trigger is the rule that decides at which processing-time moments that window actually emits a number (e.g. an early figure at 16:01, the on-time figure when the watermark passes 16:00:59, and a restated figure when a late order lands). One measures completeness, the other schedules emission. They interact but live in different time domains.

Q: How does the choice of accumulation mode change the wire format a downstream Kafka topic must carry?
A: In discarding mode each message is a delta the consumer must sum; in accumulating mode each message is the full current total for the (window, key) that the consumer upserts; in accumulating-and-retracting mode you also emit explicit retraction records (in Flink SQL, changelog rows with a `-U`/`-D` op). So the accumulation mode dictates whether the topic is an append-of-deltas, an upsert log, or a changelog with retractions — a schema-level contract, not an internal choice.

Q: What is a Flink changelog stream and how does it relate to retracting mode?
A: A changelog stream carries rows tagged with row-kinds `+I` (insert), `-U` (update-before), `+U` (update-after), and `-D` (delete), which is exactly how Flink SQL materialises accumulating-and-retracting semantics. When a window restates a flow total, Flink emits a `-U` for the old value and a `+U` for the new, so a downstream upsert sink applies the correction. This is the engine-level mechanism behind "retract then re-add" for re-aggregating consumers.

Q: Your sliding-exposure window holds far more state than expected. How do you diagnose and reduce it?
A: Each event is replicated into roughly `size/slide` overlapping windows, so a 15-minute window sliding every minute keeps about 15 live windows per key — multiply by key count for total state. Diagnose by checking the window count and the RocksDB/heap state size per key; reduce by widening the slide (fewer overlaps), shrinking the size, or replacing the sliding aggregation with an incremental running aggregate (`AggregateFunction`) that keeps one accumulator instead of per-window buffers. Using a reducing/aggregating function rather than buffering raw elements is the biggest single lever.

Q: Why is an `AggregateFunction` or `ReduceFunction` preferable to a `ProcessWindowFunction` over the raw iterable for a flow total?
A: An `AggregateFunction` folds each element into a small accumulator incrementally, so the window stores only the running partial (e.g. one sum) rather than buffering every order until firing. A bare `ProcessWindowFunction` receives the full iterable of elements, forcing Flink to retain them all, which inflates state and checkpoints. The idiom is to do the heavy lifting in an `AggregateFunction` and optionally pair it with a `ProcessWindowFunction` only to attach window metadata.

Q: How do event-time timers in a `KeyedProcessFunction` relate to watermarks?
A: An event-time timer registered with `ctx.timerService().registerEventTimeTimer(t)` fires when the watermark passes `t`, not when wall-clock reaches it, so the timer callback runs in the same completeness regime as windows. This lets you implement custom windowing or timeout logic — e.g. emit a per-share-class total once the watermark crosses the cut-off. If the watermark stalls, these timers never fire, exactly like windows.

Q: When would you use a `ProcessingTimeTimer` instead of an event-time timer in a fund pipeline?
A: Use a processing-time timer (`registerProcessingTimeTimer`) for wall-clock concerns independent of data completeness — for example flushing a partial dashboard value every 30 seconds, or timing out an idle session by real elapsed time. It fires on the machine clock regardless of watermark progress, so it is non-deterministic and unsuitable for the authoritative total, but appropriate for liveness, heartbeats, and SLA-style timeouts.

Q: A streamed EMT feed restates a cost figure with a corrected effective date. How should event-time windowing treat the correction?
A: The corrected record carries its true effective (event) date, so it must be routed to the window for that effective period — not the ingestion period — which means the affected window may need a late firing or a restatement if it had already closed. If the correction lands past allowed lateness, it goes to the side output and triggers a documented restatement keyed on the effective date. The principle is that EMT corrections are event-time-addressed, so the file's stated dates, not arrival time, decide bucketing.

Q: How does T+1 settlement pressure interact with watermark and lateness sizing for an order-flow stream?
A: A shorter settlement cycle compresses the window in which corrections and confirmations must be captured, so you cannot afford a generous multi-hour allowed lateness that defers finality past the settlement deadline. The design tension is sizing the watermark and lateness wide enough to catch genuine stragglers yet tight enough that the on-time total is reliable before the T+1 cut-off. In practice you favour faster finality plus an explicit side-output-and-restate path for anything that misses, so settlement is not blocked waiting on rare late data.

Q: Why might you split one watermark strategy per source rather than one for the whole job?
A: Different sources have different disorder profiles — a real-time order gateway is nearly in-order while a batch transfer-agency upload is hours-skewed — so a single global bound would either over-delay the fast source or mark the slow source's data late. Assigning each source its own `WatermarkStrategy` lets each express its true skew, and the minimum-across-inputs rule then merges them correctly downstream. The merged watermark is naturally paced by the slowest relevant input, which is the intended semantics.

Q: What is the effect of `WatermarkStrategy.noWatermarks()` and when is it legitimate?
A: `noWatermarks()` emits no watermarks at all, so event-time windows and timers never fire — legitimate only when the job uses processing-time semantics exclusively or relies on no event-time operations. Accidentally leaving it (or omitting any strategy) is a classic cause of "windows never emit". For any event-time fund aggregation it is wrong; you need a real bounded-out-of-orderness or monotonous strategy.

Q: How do you handle a single hot share class that skews one Flink subtask's load?
A: Key skew concentrates one share class's entire stream and window state on a single subtask, so that subtask lags while others idle. Mitigations include a two-phase aggregation (pre-aggregate on a salted/composite key, then combine), increasing parallelism only helps if keys are balanced, or using a local pre-aggregator (`AggregateFunction` upstream) to shrink the per-key payload. The diagnostic is uneven `currentInputWatermark`/backpressure across subtasks pointing at the hot key.

Q: What does backpressure from a slow Iceberg sink do to your watermark and window firing?
A: Backpressure propagates upstream and slows record consumption, which delays the source emitting newer events and therefore stalls watermark advancement, so on-time firings are deferred even though the logic is correct. The symptom is rising end-to-end latency and a watermark that creeps rather than tracks real time. The fix is at the sink — batch/commit tuning, more sink parallelism — not at the watermark, because the watermark is merely reflecting the throttled intake.

Q: Distinguish "the window is late to fire" from "the data is late" as two different incidents.
A: A late firing of the window means the watermark advanced slowly (idle partition, backpressure, over-wide bound) so the on-time pane emitted later than desired — a latency problem. Late data means an event arrived after its window's completeness line and was dropped or restated — a completeness problem. The first is fixed by watermark/throughput tuning, the second by lateness/side-output tuning; conflating them sends you to the wrong knob.

Q: How would you give an auditor a deterministic replay of yesterday's flow totals?
A: Re-run the same Flink job from the same Kafka offsets (or a savepoint at the start-of-day position) with identical watermark, lateness, trigger, and accumulation configuration; because windows group by event time and state restores from a consistent snapshot, the replay reproduces the same on-time and restated panes. Determinism comes from event-time grouping plus pinned configuration plus the exact input range. The audit artefact is the offsets/savepoint plus the job config hash.

Q: Why can changing only the allowed-lateness value between two runs produce different final totals temporarily but the same eventually?
A: A larger lateness catches more stragglers within the grace zone so its restated totals converge sooner and may differ from the smaller-lateness run while the latter is still dropping or side-outputting those events. Once both runs reprocess everything — the smaller one via its side-output restatement path — the final totals match, because the underlying data is identical. The difference is in how and when each run incorporates the same late events, not in the data.

Q: How does a session window's gap parameter map to a real fund-activity definition?
A: The gap is the maximum event-time silence allowed within one logical activity burst — e.g. five minutes of no further messages closes an investor-interaction session or a cluster of corrections to one order. Choosing it is a domain decision: too small splits one logical burst into many sessions, too large merges unrelated bursts. It is configured with `EventTimeSessionWindows.withGap(Duration.ofMinutes(5))` and is the only boundary control for session windows.

Q: A late element merges two sessions in Flink. What must your sink semantics be to stay correct?
A: Because merging retracts the two prior session results and emits one combined result, the sink must support retraction or upsert keyed on a stable merged-window identity, not blind append. With accumulating mode plus an upsert keyed on the session's namespace, Flink replaces the old rows; with a changelog sink it emits `-D`/`-U` for the merged-away sessions and `+I`/`+U` for the union. A naive append sink would leave the two pre-merge totals plus the merged one, triple-counting.

Q: How do you bound the maximum lateness a window will ever tolerate so state cannot grow unbounded?
A: Set a finite `allowedLateness(Duration)`; once the watermark passes `window-end + lateness`, Flink purges the window state, so retained state per key is bounded by the number of windows whose `window-end + lateness` has not yet been crossed. Anything later goes to the side output, which is an unbounded append you control separately (e.g. to object storage). So lateness bounds in-flight window state while the side output absorbs the long tail without holding window state.

Q: What is the difference between event-time `currentWatermark` and the wall clock when reasoning about a stalled job?
A: `currentWatermark` is derived purely from observed event timestamps and the strategy, so it can sit far behind wall-clock time if data is delayed, idle, or out of order; the wall clock keeps moving regardless. A stalled job typically shows wall clock advancing while `currentWatermark` is frozen, which immediately points at a missing assigner, an idle partition, or backpressure rather than at the window logic. Comparing the two is the fastest triage.

Q: Why is "upsert keyed on (window_start, share_class)" the natural sink key for accumulating mode?
A: Accumulating panes re-emit the full current total for the same logical window, so keying the sink on the window identity plus the business key makes each new pane overwrite its predecessor — last-write-wins gives the latest restated total without summation. Using only the business key would collide across windows; using only the window would collide across share classes. The composite key is what makes idempotent restatement work into Iceberg or a JDBC upsert sink.

Q: How would you detect, from metrics alone, that an upstream partition is idle and stalling windows?
A: Compare the per-subtask `currentInputWatermark` (Flink 1.18+) across source subtasks: one subtask frozen far behind the others while data still flows on the rest indicates an idle or skewed partition pinning the minimum-across-inputs watermark. Confirm by checking that records-in on that partition is zero over the stall period. The remedy is `withIdleness(Duration)` on that source so it is excluded from the watermark computation.

Q: What is the risk of setting `withIdleness` too aggressively (a very short idle timeout)?
A: A short idle timeout marks a merely-slow partition idle prematurely, so the merged watermark advances past data that partition is about to deliver, classifying those legitimate events as late. You trade stall-avoidance for spurious lateness on bursty-but-not-dead partitions. Size the idle timeout above the partition's normal quiet gaps so only genuinely silent partitions are excluded.

Q: How does processing-time windowing eliminate the late-data concept entirely, and why is that a trap for finance?
A: Processing-time windows close when the wall clock crosses the boundary, so every element that has arrived is in some window by definition — there is no "late". The trap is that the same order is assigned to whichever window happened to be open when the pipeline read it, so replays and out-of-order arrivals reshuffle the totals, making them non-reproducible and unauditable. "No late data" here means "wrong bucket silently", which is worse for regulated numbers than visible lateness.

Q: A consumer reports the flow total briefly spikes then settles. Which firing sequence explains this and is it a bug?
A: An early firing can emit a partial-then-overshoot value if combined with discarding panes read in isolation, but the more usual benign pattern is early/on-time/late firings in accumulating mode converging on the true total — the "spike then settle" is the refinement sequence, not a bug. It becomes a bug only if the consumer treats a provisional pane as final. Document that pre-final panes are provisional so the spike is understood as expected restatement behaviour.

Q: How would you express "fire every 10 seconds while the window is open, then a final fire at the watermark" in Flink?
A: Use a trigger that combines `ContinuousEventTimeTrigger.of(Duration.ofSeconds(10))` for the periodic early fires with the watermark-end fire, or build it with the early-firing trigger pattern so the on-time pane still emits when the watermark passes the window end. Keep `allowedLateness` for post-watermark restatements. The point is that the periodic early trigger gives a refreshing provisional figure while the watermark fire gives the authoritative one.

Q: Why does the minimum-across-inputs rule make adding a new lagging source risky mid-design?
A: Because the operator's watermark is the minimum over all inputs, introducing a source with larger skew (e.g. a slow batch feed) drags the merged watermark down to that source's pace, delaying on-time firings for the whole job. A previously prompt pipeline can become sluggish purely from the new input's lag. Mitigate by per-source watermark strategies, `withIdleness`, or isolating the slow source into its own job/window.

Q: How do you reconcile a near-real-time dashboard and a T+1 regulatory report from the same windowed stream?
A: Run one event-time windowed aggregation in accumulating mode; feed early/periodic firings to the dashboard for low-latency provisional figures, and consume the post-final pane (after watermark plus allowed lateness, ahead of the T+1 cut-off) for the report. The single source of truth restates through the day, and the report reads the value once finality is reached for the reporting interval. This avoids two divergent pipelines and keeps the report consistent with what the dashboard last showed.

Q: What goes wrong if you attach `sideOutputLateData` but never read `getSideOutput(tag)`?
A: The late records are diverted into the side-output stream but, if nothing consumes that stream, they are effectively discarded just as if no side output existed — and worse, you may believe lateness is handled when it is not. You must call `result.getSideOutput(tag)` and route it to a sink (log, alert, reprocessing topic). The wiring is two-sided: declaring the tag plus actually consuming the tagged stream.

Q: How does the `OutputTag` generic-erasure gotcha actually manifest at runtime?
A: If you write `new OutputTag<Order>("late")` without the trailing `{}`, Java erases the generic and Flink's type extraction fails, typically throwing an `InvalidTypesException` / "could not determine type" error when the job graph is built. The fix is the anonymous-subclass form `new OutputTag<Order>("late"){}` (or passing an explicit `TypeInformation`). It is a build-time/startup failure, not a silent data issue, which is the saving grace.

Q: Why is event-time alignment of windows to the dealing cut-off more than cosmetic for restatements?
A: If a window boundary falls mid-period, a single trading day's flows split across two buckets, so a restatement must touch and recombine two windows instead of one, multiplying correction complexity and audit surface. Aligning the tumbling boundary to the official cut-off makes each window equal one reportable interval, so a correction restates exactly one window. Boundary alignment is therefore part of making restatements clean and one-to-one with regulatory intervals.

Q: How would you size the out-of-orderness bound empirically rather than guessing?
A: Measure the distribution of `processing_time - event_time` skew on the live feed (e.g. log it and take a high percentile like p99), then set the bound near that percentile so the watermark tolerates the bulk of real disorder without over-waiting for rare extremes. Tail events beyond it are caught by allowed lateness and the side output. Sizing to the observed skew distribution, not the worst case, is what balances latency against late-data volume.

Q: A Spark Structured Streaming team asks how their model differs on late data. What is the key contrast?
A: Spark Structured Streaming uses watermarking via `withWatermark("event_time", "10 minutes")` to bound state and drop late data, similar in spirit, but its micro-batch execution and `outputMode` (`append`/`update`/`complete`) play the role Flink's triggers and accumulation modes play. Append mode only emits a window once it is finalised past the watermark, so it cannot restate; update mode re-emits changed rows like accumulating. The Beam primitives map across, but Spark's late-restatement control is coarser than Flink's trigger/lateness/side-output combination.

Q: What is the consequence of choosing `outputMode("append")` in Spark for a restatement-heavy fund total?
A: Append mode emits each window's result exactly once, only after the watermark guarantees it will not change, so late data past the watermark is silently dropped and there is no restated row. For a fund total that must reflect late orders, this is wrong; you need `update` (or `complete`) mode so corrected window values are re-emitted. The lesson: in Spark the `outputMode` is where the "can it restate?" decision lives.

Q: How do checkpoints and the watermark interact during recovery — does the watermark roll back?
A: On restore from a checkpoint, Flink resumes from the snapshotted input offsets and re-derives watermarks from the reconsumed records, so the effective watermark position is consistent with the restored state rather than jumping ahead. Because windows are event-time grouped and state is consistent, reprocessing the post-checkpoint records reproduces the same firings. The watermark is not separately persisted as authoritative; it is regenerated deterministically from the replayed events.

Q: Why is "exactly-once state" not the same as "exactly-once output", and where does the sink come in?
A: Checkpointing gives exactly-once internal state — after recovery the accumulators match — but the externally visible output is only exactly-once if the sink participates, e.g. via a transactional two-phase-commit sink that commits on checkpoint completion. Without that, a restart can re-emit records produced since the last checkpoint, double-writing to Iceberg or Kafka. So end-to-end exactly-once for a restatable total needs both checkpointed state and a transactional/idempotent sink.

Q: How would you explain why two correct engines (Flink and Beam/Dataflow) can be configured to produce identical flow totals?
A: Because both implement the same Beam-model primitives, you can map a one specification — tumbling event-time window of interval X keyed by share class, bounded watermark of bound B, watermark trigger with early/late firings, accumulating mode, allowed lateness L — onto each engine's API and obtain the same panes over the same input. The result is a function of the data plus these four choices, which are portable. Identical configuration plus identical event-time semantics yields identical totals regardless of engine.

Q: A window emits the right total but the downstream NAV is still wrong. Name two non-windowing causes to check.
A: First, the sink contract: an accumulation-mode mismatch (consumer summing accumulating panes, or appending instead of upserting) can corrupt the downstream figure even though the window pane was correct. Second, late-arriving restatements not yet propagated: the NAV may have read a provisional pane before the late firing, so check whether the consumer waited for finality or re-read after restatement. Both are integration faults at the boundary, not windowing bugs.

Q: Summarise, for this lesson, the exact mapping from each of the four Beam choices to a Flink API surface.
A: "Where" (windowing) maps to the window assigner — `TumblingEventTimeWindows`, `SlidingEventTimeWindows`, `EventTimeSessionWindows`; "when" (triggering) maps to `WatermarkStrategy` plus the window `.trigger(...)` and `allowedLateness(...)`; "how" (accumulation) maps to the trigger's purge behaviour plus the sink mode (upsert vs append vs changelog/retract); and "what" maps to the `AggregateFunction`/`ProcessWindowFunction`/SQL aggregation. Naming a concrete value for each, plus a `sideOutputLateData` path, is a design ready to implement.

Q: Where in the Flink pipeline should you call `assignTimestampsAndWatermarks`, and why does placement matter?
A: Assign timestamps and watermarks as close to the source as possible — ideally on the source via `fromSource(source, watermarkStrategy, name)` or immediately after, before any `keyBy`, window, or rebalance. Placing it after a shuffle or filter that drops or reorders records means the watermark is computed from an already-disturbed stream, and assigning it twice resets progress. Early assignment ensures the event-time clock reflects the true source ordering before partitioning fans the data out.

Q: How do Kafka per-partition watermarks improve on assigning watermarks after the source merges partitions?
A: When the Flink Kafka source generates a watermark per partition, each partition's disorder is tracked independently and the operator takes the minimum, so a well-behaved partition is not penalised by being merged with a skewed one before watermarking. If you instead watermark the post-merge stream, interleaving of partitions inflates apparent disorder and forces a larger bound. Per-partition watermarking (the source-level `WatermarkStrategy`) therefore yields tighter, more accurate watermarks for a multi-partition order topic.

Q: Would you ever key an order-flow window by ISIN or LEI instead of share class, and what changes?
A: You key by whatever granularity the metric is reported at: share class for per-class flow totals, but ISIN if a single security spans the aggregation, or LEI to aggregate exposure by legal entity/counterparty. Changing the key changes the state partitioning and parallelism distribution but not the windowing semantics — the watermark, trigger, and accumulation choices carry over unchanged. The gotcha is key cardinality and skew: a coarse key like LEI can concentrate load, while a fine key like ISIN multiplies the number of open windows and total state.
