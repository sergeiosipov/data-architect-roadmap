# Phase 5 · 3.3.4 Caching — 150 self-test questions

Q: What is a cache, in one sentence, and what fundamental problem does it solve for a serving layer?
A: A cache is a fast, usually in-memory store that holds copies of data so repeated reads do not have to hit the slower system of record behind it. It solves the problem of read amplification: if thousands of requests want the same NAV figure, the cache serves it from memory instead of melting the database or analytics engine with identical queries. The trade-off it introduces is staleness, because the cached copy can drift from the source of truth.

Q: Define the `cache-aside` pattern and say who owns the write to the cache.
A: In `cache-aside` (also called lazy loading) the application owns the cache: on a read it first checks the cache, and on a miss it loads from the database, stores the result in the cache, then returns it. The cache itself is a passive key-value store that knows nothing about the database. Writes go straight to the database and the application invalidates or updates the relevant cache key separately, so the application code carries the consistency burden.

Q: Walk through the exact sequence of a `cache-aside` read on a cache hit versus a cache miss.
A: On a hit, the application requests the key, the cache returns the value, and the database is never touched. On a miss, the application gets a nil reply from the cache, queries the database, writes the fetched value back into the cache (typically with a TTL), and returns it to the caller. The miss path is more expensive, which is why a cold cache or a high miss rate can briefly hammer the backing store.

Q: What is the `read-through` cache pattern and how does it differ from `cache-aside`?
A: In `read-through` the cache sits inline and owns the read-miss logic: the application always asks the cache, and on a miss the cache (via a provider or loader) fetches from the database, stores the value, and returns it. The difference from `cache-aside` is who loads on a miss — the cache library does it transparently in `read-through`, whereas the application does it explicitly in `cache-aside`. Plain Redis/Valkey is `cache-aside` by default; `read-through` requires a client library or module that wires a loader to the cache.

Q: What is the `write-through` cache pattern?
A: In `write-through` the application writes to the cache, and the cache synchronously writes the same data to the backing database before the write is acknowledged. This keeps the cache and database consistent on every write and means a subsequent read is guaranteed a fresh cached value. The cost is write latency, because every write now pays for both the cache write and the database write in the critical path.

Q: What is the `write-behind` (write-back) pattern and what is its main risk?
A: In `write-behind` the application writes to the cache, the write is acknowledged immediately, and the cache asynchronously flushes the change to the database later, often in batches. It gives the lowest write latency and can coalesce many updates into fewer database writes. Its main risk is data loss: if the cache node dies before the flush completes, acknowledged writes that never reached the durable store are gone, which is unacceptable for a record like a posted NAV.

Q: For a regulated fund-administration platform, why is `write-behind` usually the wrong choice for authoritative records?
A: Because `write-behind` acknowledges a write before it is durably persisted, a node failure can silently lose data that the system already told the caller was accepted. In a regulated context — a published NAV, a transfer-agency subscription, a posted accounting entry — that is a correctness and audit failure, not just a performance issue. For such records you want the database to be the synchronous source of truth and treat the cache as a read accelerator only.

Q: In `cache-aside`, why is "write to DB, then delete the cache key" generally safer than "write to DB, then update the cache key"?
A: Deleting the key forces the next read to repopulate from the database, so even under concurrent writes you converge on the true value; updating the key risks two interleaved writers leaving a stale value cached. The delete-on-write pattern trades a guaranteed cache miss (and one extra DB read) for a much smaller window of inconsistency. Updating in place is faster but reintroduces ordering races that are hard to reason about.

Q: What is a "stale read" and why is it the central correctness concern of caching?
A: A stale read is when the cache returns a value that no longer matches the source of truth because the underlying data changed but the cached copy was not invalidated or expired. It is central because caching deliberately trades freshness for speed: every cache is a bet that slightly old data is acceptable. The architect's job is to decide, per dataset, how much staleness is tolerable and to size TTL and invalidation accordingly.

Q: State the core staleness-vs-load trade-off in caching in your own words.
A: A longer TTL or more aggressive caching reduces load on the serving layer because more requests are answered from memory, but it increases the maximum staleness a client can observe. A shorter TTL or stricter invalidation keeps data fresh but pushes more traffic through to the backing store. The decision is per dataset: you cache hard where staleness is cheap and cache lightly (or not at all) where staleness is a bug.

Q: Why is fund NAV data a near-ideal candidate to "cache hard"?
A: A fund's NAV is struck once per valuation cycle, typically once a day after market close, so the value is immutable for the rest of the day until the next strike. That update cadence means a long TTL (hours) introduces essentially zero staleness risk during the day, while caching dramatically cuts load on whatever serves the NAV. The natural correctness boundary is the next NAV strike, which makes invalidation easy to reason about.

Q: For daily-struck NAV, what TTL discipline is defensible and why?
A: A defensible approach is to set the TTL to expire at (or just before) the next expected NAV publication, or use a long fixed TTL plus an explicit invalidation when the new NAV is published. Because the value cannot legitimately change intraday, even a multi-hour TTL produces no incorrect reads; the only risk is serving the prior day's NAV after a new one is struck, which the publish-time invalidation removes. This is the "cache hard, invalidate on the known event" pattern.

Q: Contrast NAV caching with intraday position data — why does the same TTL logic flip?
A: Intraday positions change continuously as trades execute, so any non-trivial TTL means clients can see a position that is seconds or minutes out of date. Unlike NAV, there is no clean daily boundary at which the value freezes, so a stale cached position can be an outright correctness bug for risk or exposure checks. Here you either cache for a very short window with explicit awareness of the staleness, or do not cache and read live.

Q: Why is cache invalidation often called "the hard part" of caching?
A: Because correctly knowing when a cached value has become wrong requires understanding every path that can mutate the underlying data, and missing one path leaves a stale value served indefinitely. TTL is a blunt safety net, but precise event-driven invalidation must be wired to every writer, which is easy to get subtly wrong under concurrency. The famous quip is that there are only two hard things in computer science: cache invalidation and naming things.

Q: Explain TTL choice as a correctness decision rather than just a performance knob.
A: TTL bounds the maximum staleness a consumer can observe, so picking a TTL is implicitly deciding "this data may be up to N seconds wrong and that is acceptable." If the business cannot tolerate that window — say, a compliance check that must see the latest figure — then a TTL that long is a correctness defect, not merely a tuning choice. So TTL must be derived from the data's update cadence and the consumer's tolerance, not from a default.

Q: What is the difference between an absolute TTL and a sliding TTL, and when does each fit?
A: An absolute TTL expires a key a fixed time after it was written regardless of access, so the key lives at most that long; a sliding TTL resets the expiry on each access, so frequently used keys can live indefinitely. Absolute TTL fits data with a known freshness budget like NAV, where you want a hard staleness ceiling. Sliding TTL fits session-like or hot-lookup data where you want to keep popular entries warm but evict cold ones, and where staleness is not a hard constraint.

Q: Why can a sliding TTL be dangerous for data that has a hard freshness requirement?
A: Because a sliding TTL never lets a hot key expire as long as it keeps being read, a value that the source of truth has since changed can be served stale for an unbounded time. For data with a real freshness budget you need a ceiling on age, which only an absolute TTL (or explicit invalidation) guarantees. A common bug is using a sliding TTL "to keep the cache warm" and accidentally pinning stale data forever.

Q: What is the "thundering herd" / cache stampede problem?
A: When a popular key expires, many concurrent requests miss simultaneously and all rush to the backing store to recompute the same value, briefly overloading it. The cache that was protecting the database momentarily inverts into an attack on it. It is worst for expensive-to-compute values with high request rates, such as a heavy semantic-layer aggregation that many dashboards request at once.

Q: Name two techniques to mitigate a cache stampede on a hot key.
A: First, request coalescing / locking: only the first miss recomputes the value while other requests wait for or briefly serve the old value, so the backing store is hit once. Second, early/probabilistic recomputation: refresh the value slightly before it expires (jittered) so the key never goes fully cold under load. Adding random jitter to TTLs also prevents many keys from expiring at the same instant.

Q: Why is adding jitter to TTLs a good default practice?
A: If many related keys are created together with the same TTL, they all expire at the same moment, producing a synchronized wave of misses and recomputations against the backing store. Adding a small random offset (jitter) to each TTL spreads expiries over time so misses are smeared out rather than spiking. It is a cheap one-line change that turns a potential stampede into a gentle trickle.

Q: What does a "cache hit ratio" tell you, and what does a sudden drop usually indicate?
A: The hit ratio is the fraction of reads served from the cache rather than the backing store, and a high ratio means the cache is effectively shielding the source. A sudden drop usually means keys are being evicted or expiring faster than they are reused — for example a TTL that is too short, an undersized cache hitting its memory limit and evicting hot keys, or a deploy that changed key naming so old entries are orphaned. It is one of the first metrics to check when the backing store load spikes.

Q: A dashboard is suddenly slow and the database CPU is pegged, but nothing changed in the queries. What cache-related cause would you check first?
A: First check the cache hit ratio and eviction/expiry metrics: if the hit ratio collapsed, requests are falling through to the database. Likely causes are the cache running out of memory and evicting hot keys, a recently shortened TTL, a key-schema change from a deploy orphaning old entries, or the cache node being unreachable so the client fails open to the database. Confirm by watching whether the database load tracks the cache miss rate.

Q: In a data platform, name three categories of things that are commonly worth caching.
A: API responses (especially read-heavy endpoints returning slowly changing reference or computed data), semantic-layer / aggregation query results (expensive group-bys and rollups that many dashboards reuse), and feature lookups for ML serving (precomputed features keyed by entity id). All three share the profile of being expensive to produce and read far more often than they change. NAV and reference-data lookups fit the first two especially well.

Q: Why are semantic-layer / BI aggregation results a strong caching target?
A: Aggregations like daily revenue by region or fund-level rollups are expensive to compute, change at most once per refresh cycle, and are requested by many users viewing the same dashboard. Caching the result means the heavy scan runs once per cycle instead of once per viewer, protecting the warehouse or query engine from redundant work. Because the refresh cadence is known, the TTL can be tied directly to it.

Q: Why are ML feature lookups a classic caching use case, and what is the gotcha?
A: At inference time you need feature values for an entity (say a customer or instrument) with very low latency, and recomputing them per request would be far too slow, so precomputed features are cached and looked up by key. The gotcha is freshness: a feature cached too long can feed the model stale inputs, silently degrading predictions, so the TTL must match how often the feature is recomputed upstream. This is exactly why feature stores separate an online (cached) store from an offline store.

Q: What kinds of data should you generally NOT cache, and why?
A: Data that changes faster than you can tolerate staleness (live intraday positions, real-time prices used for execution), data that must be transactionally exact on every read (authoritative balances, audit records), and write-mostly data with little read reuse (where the cache earns nothing and just adds invalidation risk). Caching these either introduces correctness bugs or adds complexity without a hit-ratio payoff. The test is: is this read far more than it changes, and is bounded staleness acceptable?

Q: How do you decide *whether* to cache at all for a given endpoint?
A: Ask whether the data is read much more often than it changes, whether some bounded staleness is acceptable to consumers, and whether the backing store is actually under enough load to need protection. If reads vastly outnumber writes and a known staleness budget exists, cache; if every read must be exact, or reuse is low, do not. The architect mostly decides whether and where to cache, not the low-level mechanics.

Q: How do you decide *where* in the request path to place a cache?
A: You place the cache as close to the consumer as is safe while still respecting the staleness budget: a client/CDN edge cache for truly static content, an application-tier cache (Redis/Valkey) for shared computed results, or a database-adjacent cache for query results. Caching closer to the consumer saves more latency and load but widens the set of caches you must invalidate. The placement decision balances how much of the path you short-circuit against how hard invalidation becomes.

Q: What is the difference between a local in-process cache and a distributed cache like Redis/Valkey, and when does each fit?
A: A local in-process cache lives in each application instance's memory: it is the fastest possible but is not shared, so N instances hold N independent copies that can diverge and each must be invalidated. A distributed cache is a separate service all instances share, giving one consistent view and a single place to invalidate at the cost of a network hop. Use local caches for tiny, very hot, low-consistency data and a distributed cache when many instances must agree on shared values.

Q: Why can per-instance local caches make invalidation much harder than a shared cache?
A: Because each application instance holds its own copy, invalidating a value means reaching every instance, not one cache, so a value updated in the database can remain stale in some instances until their local TTLs expire. There is no single authoritative cache to delete a key from. This is why local caches are best reserved for data where short-lived divergence between instances is acceptable.

Q: What does "fail open" versus "fail closed" mean for a cache, and which is usually correct?
A: Fail open means that if the cache is unavailable the application bypasses it and reads directly from the backing store, so requests still succeed (just slower); fail closed means a cache outage causes requests to error. For a read cache you almost always want fail open so a cache hiccup degrades performance rather than availability — but you must size the backing store to survive the full uncached load, or fail-open becomes a self-inflicted outage.

Q: What is the licensing event that triggered the creation of Valkey?
A: In March 2024 Redis Ltd. changed the Redis source license from the permissive BSD 3-Clause to a dual license of RSALv2 and SSPLv1, which restricts cloud providers from offering managed Redis without a commercial agreement. In response the Linux Foundation, with AWS, Google, Oracle, Ericsson and others, forked the last BSD-licensed Redis (7.2.4) into Valkey to keep a truly open implementation. The licensing change, not a technical disagreement, is what forced the fork.

Q: Which license does Valkey use, and why does that matter to a cloud provider or regulated shop?
A: Valkey is licensed under BSD 3-Clause, the same permissive license Redis used before the 2024 change, so anyone can use, modify, run, and offer it as a managed service without a commercial agreement. That matters because it removes the legal restriction the SSPL/RSAL change introduced and keeps the software freely usable. For a regulated firm it also means no per-use commercial licensing entanglement for the open serving-layer cache.

Q: What does SSPL stand for, and why do many organisations treat it as not-truly-open-source?
A: SSPL is the Server Side Public License, created by MongoDB and adopted in Redis's 2024 dual license. The Open Source Initiative does not approve it as an open-source license because it imposes far-reaching copyleft obligations on anyone who offers the software as a service, which the OSI considers a use restriction. So while the source is visible, SSPL is widely classified as "source-available" rather than open source.

Q: Redis 8 changed the licensing picture again — what happened?
A: With Redis 8.0 (released May 2025) Redis Ltd. added AGPLv3 as a third license option alongside RSALv2 and SSPLv1, returning an OSI-approved open-source license to Redis. This was a partial reversal of the 2024 move, restoring an open path for Redis itself. Valkey nonetheless continues as an independent BSD-licensed project with broad vendor backing.

Q: From an architect's seat, why does the Redis/Valkey licensing situation matter even though "it's just a cache"?
A: License terms determine whether you can run, modify, and self-host the software freely, whether your managed-cloud vendor can offer it, and whether copyleft obligations attach to your service. AGPLv3 and SSPL carry service-related obligations that legal teams in a regulated firm will scrutinise, whereas Valkey's BSD license is unencumbered. So the choice between Redis and Valkey is partly a compliance and procurement decision, not only a technical one.

Q: What is Azure Cache for Redis, in one sentence?
A: Azure Cache for Redis is Microsoft's managed, Redis-compatible in-memory cache service that handles provisioning, patching, scaling, and replication so you do not run the cache nodes yourself. It is the corporate, managed counterpart to self-hosting Valkey or Redis, and is the kind of service an architect evaluates for the serving-layer cache. It historically came in Basic, Standard, Premium, and Enterprise/Enterprise Flash tiers.

Q: What is the announced retirement position for Azure Cache for Redis tiers?
A: Microsoft has announced that the Azure Cache for Redis Basic, Standard, and Premium tiers will be retired on 30 September 2028, and the Enterprise and Enterprise Flash tiers will be retired on 31 March 2027. Existing instances keep running until their retirement date, but new architecture should target the successor service. Always confirm the current dates against Microsoft's retirement FAQ, as such timelines can shift.

Q: What is the successor that Azure Cache for Redis customers are being migrated to?
A: Customers are being directed to Azure Managed Redis, a newer managed offering built on Redis Enterprise software that provides features like active geo-replication and Redis modules with improved performance and cost-effectiveness. Microsoft is providing migration tooling (rolling out in phases from early 2026, starting in preview) that can move a cache endpoint while preserving the hostname and access key for a smoother cutover. An architect should design new platforms against Azure Managed Redis rather than the retiring tiers.

Q: Why should an architect treat the Azure Cache for Redis retirement as a design input today, not a future problem?
A: Because building a new platform on a service with a known retirement date bakes in a forced migration, and ADRs and cost models should reflect the successor (Azure Managed Redis) and its migration path. Designing against the retiring tiers risks rework and a compliance conversation about supportability. The architect's move is to note the retirement, choose Azure Managed Redis (or self-hosted Valkey) as the target, and document the rationale.

Q: How does choosing self-hosted Valkey versus Azure Managed Redis change who owns operational burden?
A: Self-hosted Valkey on your own Kubernetes or VMs means you own provisioning, patching, failover, scaling, backups, and monitoring of the cache, in exchange for no per-service licensing entanglement and full control. Azure Managed Redis offloads that operational burden to the provider for a service fee and accepts the provider's licensing and feature set. This is the classic FOSS-hands-on versus corp-managed trade the architect weighs per workload.

Q: What is an eviction policy in Redis/Valkey, and why does it matter when the cache is full?
A: When the cache reaches its memory limit, the eviction policy (set via `maxmemory-policy`) decides which keys to drop to make room — for example `allkeys-lru` evicts the least-recently-used keys across the whole keyspace. It matters because the wrong policy can evict the keys you most needed cached, collapsing the hit ratio, or can refuse new writes if set to `noeviction`. Choosing a policy is part of sizing the cache for its workload.

Q: What does the `noeviction` policy do, and when might it bite you?
A: With `noeviction`, once the cache hits `maxmemory` it stops accepting writes that would grow memory and returns errors instead of dropping keys. It bites you when you are using the cache as a general read cache and assume it will silently make room — instead writes start failing, which can surface as application errors rather than just a lower hit ratio. It is appropriate when every cached item must be kept and you would rather fail loudly than lose data.

Q: What is the difference between `maxmemory` eviction and TTL expiry in Redis/Valkey?
A: TTL expiry removes a key when its time-to-live elapses, regardless of memory pressure, and is about freshness; `maxmemory` eviction removes keys (per the eviction policy) only when the instance runs out of room, and is about capacity. A key can be dropped by either mechanism. Relying on eviction for correctness is a mistake — eviction is non-deterministic, so freshness must come from TTL or explicit invalidation, not from hoping the key gets evicted.

Q: A cached value is wrong even though its TTL has not expired. What is the likely root cause?
A: The underlying data changed and nothing invalidated the key, so the cache happily serves the stale value until TTL expiry. The root cause is a missing invalidation on some write path — a writer that updates the database but does not delete or update the cache key. The fix is to wire invalidation to every mutation path (or shorten the TTL to bound the damage), and to audit all writers for the affected key.

Q: You shortened a TTL to fix staleness and now the database is overloaded. What trade-off did you hit?
A: You moved along the staleness-vs-load curve: a shorter TTL means keys expire sooner, more reads miss the cache, and more traffic falls through to the backing store. You traded freshness for load and pushed the system toward the under-cached failure mode. The better fix is usually event-driven invalidation so values stay cached until they actually change, rather than expiring everything frequently just in case.

Q: What is "cache warming" and when is it worth doing?
A: Cache warming is proactively loading likely-to-be-requested values into the cache before traffic arrives, for example after a deploy that flushed the cache or just after a NAV publish. It is worth doing when a cold cache would cause a damaging miss storm against the backing store, such as the morning rush on freshly struck NAVs. The cost is the upfront work and the need to know which keys will be hot.

Q: Why is a key-naming (key schema) convention important for a shared cache?
A: A consistent, namespaced key schema (for example `nav:{isin}:{date}`) makes keys collision-free, lets you reason about and target invalidation, and avoids accidentally orphaning entries when code changes. A deploy that silently changes the key format turns the entire prior cache into dead weight and forces a cold-start miss storm. A versioned prefix in the key also lets you invalidate a whole class of entries at once by bumping the version.

Q: How can a versioned key prefix be used as a bulk-invalidation lever?
A: If keys are prefixed with a schema or content version, for example `v3:nav:{isin}`, then changing the prefix to `v4:` instantly makes every old key unreachable by the application without deleting anything, so reads miss and repopulate under the new prefix. This is a cheap way to invalidate an entire dataset after a logic change, at the cost of a one-time cold cache. The old keys then age out naturally via TTL or eviction.

Q: Why might you cache the result of an external API call, and what failure does it also protect against?
A: You cache external API responses because the call is slow, rate-limited, or costs money, and the data often changes slowly, so caching cuts latency, cost, and dependence on the remote service's availability. It also gives you a fallback: if the upstream is briefly down, a still-valid cached response keeps your platform serving rather than failing. The matching TTL must respect how often the upstream data legitimately changes.

Q: For a feed of fund reference data (ISINs, fund names, share-class attributes), how would you reason about caching?
A: Reference data like ISINs and share-class attributes changes rarely (new launches, corporate actions, attribute corrections), so it is read-heavy and slow-changing — a strong cache-hard candidate with a long TTL and explicit invalidation on the rare update event. You cache it because every NAV calculation, report, and lookup needs it constantly, and hitting the master on every request is wasteful. The gotcha is making sure corrections (a fixed attribute) trigger invalidation so you do not serve the old value for hours.

Q: A LEI (Legal Entity Identifier) lookup service is hammering its upstream registry. How does caching help and what TTL fits?
A: LEI records change infrequently (registration status, renewal dates, entity attributes), so caching lookups by LEI sharply reduces calls to the upstream registry while staying fresh enough. A TTL on the order of hours to a day is usually fine because the data's update cadence is slow and bounded staleness is acceptable for most uses. The exception is status fields (e.g. lapsed/active) used for compliance gating, where you may want a shorter TTL or explicit refresh.

Q: Why is caching an EMT or EPT file's parsed contents reasonable, while caching a live trade feed is not?
A: EMT (European MiFID Template) and EPT (European PRIIPs Template) files are published on a periodic cadence (e.g. monthly or on change) and are immutable once issued, so the parsed contents are slow-changing reference data that can be cached hard until the next version arrives. A live trade feed, by contrast, changes continuously and is consumed for up-to-the-moment decisions, so any caching introduces staleness that can be a correctness bug. The distinguishing factor is update cadence versus freshness requirement.

Q: How does a SWIFT message flow differ from cacheable data, and where might caching still appear?
A: SWIFT messages are individual, ordered financial transactions (settlements, confirmations) that must each be processed exactly once and in order, so the messages themselves are events, not cacheable read data. Caching still appears around them for reference lookups used during processing — BIC-to-institution mappings, static settlement instructions, currency/holiday calendars — which are slow-changing and read constantly. So you cache the reference context, never the transactional messages.

Q: What is the risk of caching a value that is used for a DORA/regulatory compliance check?
A: If a compliance check reads a cached value, a stale cache can cause the check to pass or fail against out-of-date data, producing an incorrect regulatory determination that is hard to defend in an audit. Under DORA-style operational-resilience expectations you must be able to show the control acted on accurate data. So compliance-critical reads either bypass the cache or use a TTL short and tightly controlled enough that the staleness window is demonstrably acceptable.

Q: Why is "the cache is the source of truth" an anti-pattern for authoritative fund data?
A: A cache is a volatile, evictable, TTL-bounded copy; treating it as the system of record means losing data on eviction, restart, or node failure, which is unacceptable for authoritative records like a published NAV or a transfer-agency holding. The database (or warehouse) must own durability and correctness, with the cache strictly a read accelerator in front of it. Confusing the two leads to silent data loss and unreconcilable records.

Q: What is the relationship between cache TTL and a data product's freshness SLA?
A: The cache TTL is a direct contributor to the worst-case age of data a consumer sees, so it must be no larger than the freshness budget your SLA promises minus other pipeline latencies. If your SLA says "data no older than 15 minutes" and upstream already costs 10 minutes, a 10-minute cache TTL would blow the budget. TTL therefore has to be derived from the SLA, not chosen independently.

Q: In `cache-aside`, what concurrency bug can occur between a reader repopulating a key and a writer updating the database?
A: A classic race is: a reader misses, fetches the old value from the database, and is paused; a writer then updates the database and deletes the cache key; the paused reader resumes and writes the now-stale value back into the cache. The result is a stale value that persists past the write. Mitigations include short TTLs to bound the damage, delete-after-write with a small delay, or versioned/conditional writes.

Q: What is "negative caching" and why is it sometimes necessary?
A: Negative caching means caching the fact that a lookup returned nothing (a miss/not-found), so repeated requests for a non-existent key do not repeatedly hit the backing store. It is necessary when missing keys are requested often — for example bots probing for invalid ISINs — which would otherwise turn every request into a database query. The gotcha is using a short TTL for negative entries so that data which later appears is not masked for too long.

Q: What is the danger of caching personally identifiable or sensitive fund-investor data, and how is it usually handled?
A: Caching PII or sensitive investor data widens the surface where that data lives, must be protected, and must be purged on request, which complicates data-protection and right-to-erasure obligations. It is usually handled by avoiding caching sensitive fields, encrypting the cache and securing access, keeping TTLs short, and ensuring invalidation propagates deletions. The default posture for regulated personal data is to not cache it unless there is a clear, controlled reason.

Q: How does Redis/Valkey persistence (RDB snapshots, AOF) relate to using it as a cache?
A: RDB snapshots and the append-only file (AOF) let Redis/Valkey survive a restart by reloading data, which is useful to avoid a cold-cache stampede after a reboot. But persistence does not make the cache a system of record — it is a warm-restart convenience, not durability you should depend on for correctness. For a pure cache you may even disable persistence to maximise speed, accepting a cold start on restart.

Q: Why does enabling AOF with `appendfsync everysec` not make Valkey safe to use as your primary store?
A: With `appendfsync everysec` the append-only file is flushed to disk about once a second, so up to roughly a second of acknowledged writes can be lost on a crash, and even `always` adds latency without matching a real database's transactional guarantees. Redis/Valkey is single-purpose fast storage, not a transactional system of record with the integrity, querying, and durability semantics of a database. So persistence improves restart behaviour but does not change its role as a cache.

Q: What is the difference between an LRU and an LFU eviction policy?
A: LRU (least-recently-used) evicts the key that has gone longest without access, favouring temporal locality; LFU (least-frequently-used) evicts the key accessed fewest times, favouring keys that are popular over the long run. LRU can evict a long-term-popular key after a burst of one-off accesses, while LFU resists that but adapts more slowly to changing hot sets. Redis/Valkey offers both (e.g. `allkeys-lru`, `allkeys-lfu`); the choice depends on whether your access pattern is bursty or steadily skewed.

Q: How do you size a cache's memory, and what happens if you undersize it?
A: You size it from the working set — the volume of distinct hot keys multiplied by their value size — plus headroom for overhead and replication. If you undersize it, the instance constantly evicts keys to stay under `maxmemory`, so even hot keys get dropped, the hit ratio falls, and the backing store load rises, defeating the cache's purpose. Undersizing often masquerades as "the cache isn't helping," when really it is thrashing.

Q: What is a write to the database followed by a cache invalidation, and why is the order important?
A: It is the standard `cache-aside` write: update the durable source first, then delete (invalidate) the corresponding cache key so the next read repopulates from the new truth. The order matters because if you invalidate before the database write commits, a concurrent reader can repopulate the cache with the old value just before the new one lands, reintroducing staleness. Writing the source first narrows that window.

Q: Why might you deliberately accept some staleness even when you could invalidate precisely?
A: Precise, immediate invalidation on every write adds coupling and load (every writer must reach the cache, every change causes a miss), so for data where a few seconds or minutes of staleness is harmless, a TTL is simpler and cheaper. Accepting bounded staleness reduces invalidation complexity and smooths backing-store load. The architect chooses the cheapest mechanism that still respects the freshness budget.

Q: What is a publish/subscribe-based cache invalidation, and why might a data platform use it across many instances?
A: In pub/sub invalidation a writer publishes an "invalidate key X" message on a channel (Redis/Valkey `PUBLISH`/`SUBSCRIBE` or a message bus) and every subscribing application instance drops its local copy of that key on receipt. A platform uses it when many instances each hold local caches and a single shared-cache delete is not enough to reach them all. The trade-off is that propagation is asynchronous, so there is a brief window where some instances are still stale until the message arrives.

Q: How does a TTL act as a safety net even when you also have explicit event-driven invalidation?
A: Explicit invalidation can miss a write path, drop a message, or race, leaving a stale key that would otherwise live forever; a TTL guarantees that any such missed invalidation self-heals within a bounded time. So TTL and event invalidation are complementary — invalidation gives prompt freshness on known changes, while the TTL caps the damage of any invalidation you forgot or that failed to propagate. Belt-and-braces: invalidate eagerly, but never set a key with no expiry on data that can change.

Q: For the Phase-8 data API serving daily NAVs, give a defensible cache decision: whether, where, and TTL.
A: Yes, cache — NAVs are read-heavy and struck once daily, so caching is almost a free win. Place it in the application/serving tier (a shared Redis/Valkey or Azure Managed Redis) keyed by `nav:{isin}:{date}`, close to the API but shared across instances. Use a long TTL aligned to the valuation cadence (expiring around the next strike) plus an explicit invalidation when a new or corrected NAV is published, so intraday reads are fast and never stale.

Q: Write the one-line justification for caching NAV anchored to the staleness-vs-load trade-off.
A: Because a NAV is immutable from its daily strike until the next, a long TTL produces essentially zero staleness while removing nearly all read load from the serving layer, so the staleness-vs-load trade-off lands overwhelmingly on the side of caching hard. The only correctness risk — serving a superseded NAV after a re-strike or correction — is closed by invalidating the key on publish. That makes NAV the textbook "cache hard, invalidate on the known event" case.

Q: What is a CDN, and how does it relate to caching in a data platform?
A: A CDN (content delivery network) is a geographically distributed cache that stores content at edge locations near users, typically for static assets and cacheable HTTP responses. It relates to platform caching as the outermost layer of a caching hierarchy: static or long-TTL public data (e.g. published fund fact sheets) can be served from the edge, while dynamic per-user data stays behind the application cache. The same staleness-vs-load reasoning applies, just further from the source.

Q: What HTTP mechanism lets clients and intermediaries cache responses, and how does it connect to TTL?
A: HTTP caching is driven by response headers such as `Cache-Control` (with directives like `max-age`, `no-store`, `private`) and validators like `ETag`/`If-None-Match`. `max-age` is effectively a TTL the server hands to clients and CDNs, telling them how long the response may be reused. So the same staleness budget that sets your server-side cache TTL should set the `max-age` you advertise to downstream caches.

Q: What does `Cache-Control: no-store` mean versus `no-cache`?
A: `no-store` tells caches not to store the response at all, so every request goes to the origin — appropriate for sensitive or always-fresh data. `no-cache` is misleadingly named: it allows storing the response but requires revalidation with the origin (via `ETag`/`If-None-Match`) before reuse, so a 304 can avoid re-sending the body while still confirming freshness. They are different tools — one forbids storage, the other forces validation.

Q: How does an `ETag` enable cheap freshness checks?
A: An `ETag` is an opaque version token the server attaches to a response; the client stores it and on the next request sends `If-None-Match` with that token. If the resource is unchanged the server replies `304 Not Modified` with no body, saving bandwidth while confirming the cached copy is still valid. This is a validation-based cache that trades a small round-trip for guaranteed freshness, complementing TTL-based expiry.

Q: What is the difference between caching for latency and caching for load reduction?
A: Caching for latency aims to make individual requests faster by serving from memory instead of a slow backend; caching for load reduction aims to shield an overloaded backing store from repeated identical work. They often coincide but can diverge — a cache might barely improve per-request latency yet hugely cut backend load (or vice versa). Naming the goal helps you place the cache and set TTL correctly, and helps you measure success (p99 latency versus backend QPS).

Q: Why is measuring p99 latency more informative than average latency when evaluating a cache?
A: Averages hide the tail: a cache can leave the average looking great while a small fraction of requests (cache misses, stampedes, evicted hot keys) suffer badly, and those tail requests are what users and SLAs notice. p99 (or p99.9) latency exposes that misses are expensive and how often they happen. A cache that drops the average but leaves a fat tail may still be failing its real job.

Q: What is request coalescing (single-flight) and which caching problem does it solve?
A: Request coalescing, or single-flight, ensures that when many concurrent requests miss on the same key, only one of them actually recomputes/fetches the value while the others wait for and share that result. It solves the cache stampede / thundering-herd problem by collapsing N simultaneous misses into a single backend call. It is typically implemented with a per-key lock or an in-flight registry in the client or cache layer.

Q: How does cache placement interact with the number of caches you must invalidate?
A: The further from the source and the more layers you cache at (edge CDN, app cache, local in-process), the more independent caches hold copies that all must be invalidated when data changes. Each additional layer buys latency and load relief but multiplies invalidation surface and the chance of a stale copy somewhere. So you cache as close to the consumer as the staleness budget allows, but every extra layer is an invalidation liability to weigh.

Q: What does it mean for a cache to be "consistent" versus "eventually consistent" with its source?
A: A strongly consistent cache always reflects the latest committed source value, which usually requires synchronous write-through or invalidation in the write path and costs write latency. An eventually consistent cache may lag the source for a bounded window (a TTL or invalidation-propagation delay) before converging. Most read caches are intentionally eventually consistent, and the architect's job is to bound and justify that window per dataset.

Q: Why can clock skew or a wrong system time break TTL-based caching?
A: TTLs are computed against the cache server's clock, so if a value's freshness is reasoned about using a different machine's clock, skew can make a key appear fresh when it should be expired (or vice versa). For absolute-time invalidation tied to events like a NAV strike, an incorrect clock can serve stale data past the intended boundary. Reliable time (NTP-synced) is a quiet prerequisite for correct TTL discipline.

Q: A new deploy doubled database load even though the cache is up and TTLs are unchanged. What deploy-related cache cause is likely?
A: The deploy probably changed the cache key schema or serialization format, so old entries no longer match the new keys and the whole cache is effectively cold — every read misses and hits the database until it repopulates. Confirm by checking the hit ratio dropping at deploy time and key names in the cache. Fix with key versioning plus a warm-up, or by maintaining backward-compatible keys across the transition.

Q: What is the role of replication in a managed cache like Azure Managed Redis or a Valkey cluster?
A: Replication keeps one or more replica copies of the data so that if the primary node fails, a replica can be promoted and the cache survives with minimal data loss and downtime. It improves availability and can serve reads, which matters when the cache is on the critical path of your serving layer. It does not, however, turn the cache into a durable system of record — replication is about availability, not source-of-truth durability.

Q: Why might reading from a Redis/Valkey replica return slightly stale data?
A: Replication is asynchronous by default, so a replica can lag the primary by a small interval; a read served from a replica may miss the most recent writes that have not yet propagated. For a cache this is usually acceptable, but if you route reads to replicas you add another small staleness window on top of the TTL. If you need read-your-writes consistency, read from the primary.

Q: What is the difference between caching at the query-result level and at the object/row level?
A: Query-result caching stores the full result of a specific query (e.g. a dashboard aggregation) keyed by the query, so it is reused only by identical queries but saves the whole computation. Object/row caching stores individual entities keyed by id (e.g. a fund by ISIN), reusable across many different queries that need that entity but requiring you to assemble results. Query-result caching wins for expensive fixed aggregations; object caching wins for shared entities accessed many ways.

Q: Why does query-result caching become ineffective when queries are highly parameterized?
A: Each distinct parameter combination produces a different cache key, so a query with many parameters (date ranges, filters, user-specific scopes) fragments into countless rarely-reused entries with low hit ratios. The cache fills with one-off results that are never read again, wasting memory without cutting load. In that case object/row caching or caching coarser pre-aggregates is usually more effective.

Q: How would you cache an expensive semantic-layer rollup that refreshes nightly?
A: Cache the rollup result keyed by its definition (and the business date), set a TTL that comfortably spans the day until the next nightly refresh, and explicitly invalidate or refresh the key when the nightly job completes. Because the value only legitimately changes once per night, intraday reads are always fresh and the heavy scan runs once per cycle rather than per dashboard view. Add jitter or warm-up if many such rollups refresh at the same time to avoid a synchronized miss storm.

Q: What is a "cache key cardinality explosion" and why is it a problem?
A: It is when the number of distinct cache keys grows far beyond the working set — for example keying by full request URL including volatile parameters — so the cache holds a huge number of low-reuse entries. The problem is memory pressure and eviction of genuinely hot keys, dragging the hit ratio down and possibly evicting the data you most needed. The fix is to design keys around the reusable dimensions and drop volatile, non-discriminating parameters from the key.

Q: When is it better to cache a pre-aggregate than the raw rows?
A: When consumers nearly always ask for the same rolled-up view (e.g. fund-level totals) and rarely need the underlying rows, caching the pre-aggregate gives a high hit ratio and small footprint, whereas caching raw rows wastes memory on data that is recomputed each time anyway. Pre-aggregation moves the expensive work upstream once. The trade-off is flexibility: a cached pre-aggregate cannot answer a question at a finer grain than it was aggregated to.

Q: How does caching interact with a data product's "freshness" observability metric?
A: Freshness measures how old the data a consumer sees is, and a cache directly inflates that age by up to its TTL plus any replication/invalidation lag. So your freshness monitoring must account for the cache, and ideally measure end-to-end age including the cache, not just the pipeline up to the database. A freshness alert firing while the pipeline looks healthy often points at a cache serving stale entries.

Q: What is the simplest mental model for "cache or not?" that you can apply on the spot?
A: Ask three questions: is this read far more often than it changes, is bounded staleness acceptable to the consumer, and is the backing store under enough load to need protection? If all three are yes, cache it and set the TTL from the update cadence and freshness budget; if any is a firm no, do not. This keeps you from over-caching (serving wrong numbers) or under-caching (melting the serving layer).

Q: Why is "cache everything with a 5-minute TTL" a dangerous default for a data platform?
A: A blanket 5-minute TTL ignores each dataset's actual cadence: it is far too short for daily NAV (needless misses and load) and possibly far too long for intraday positions (correctness bugs). It also synchronizes expiries across keys, inviting stampedes. Good TTL discipline derives each TTL from that data's update cadence and the consumer's tolerance, not from a single platform-wide number.

Q: How do you bound the blast radius of a wrong TTL decision?
A: Keep TTLs short enough that any staleness self-heals within an acceptable window, prefer explicit invalidation for data where staleness is a real correctness issue, and monitor freshness and hit ratio so a bad TTL surfaces quickly. Versioned key prefixes let you flush a class of entries instantly if you discover a mistake. In short: make staleness time-bounded, observable, and quickly reversible.

Q: Why is it usually wrong to cache the result of a write request?
A: A write request changes state, so its response is specific to that mutation and reusing a cached response for a later write would skip the actual state change or return a misleading result. Caches are for idempotent reads, not state-changing operations. HTTP encodes this: GET/HEAD are cacheable, while POST/PUT/DELETE generally are not, and a cached POST response is a correctness hazard.

Q: What is the difference between idempotent and non-idempotent operations in the context of caching, and why does it matter?
A: An idempotent operation (like a GET) produces the same result and no additional side effects when repeated, which is exactly what makes its result safe to cache and replay. A non-idempotent operation (like a POST that creates a record) changes state each time, so caching or replaying it is unsafe. Caching layers therefore key off idempotent reads; mixing in non-idempotent operations is how you get duplicate side effects.

Q: How does multi-tenancy complicate cache key design in a fund-administration platform?
A: Different tenants (management companies, fund ranges) must not see each other's data, so the tenant identity has to be part of the cache key (e.g. `tenant:{id}:nav:{isin}:{date}`) or you risk one tenant's cached value being served to another. Omitting the tenant from the key is a data-leak and correctness bug, not just a cache miss. The cost is reduced cross-tenant reuse, which is the correct trade for isolation.

Q: Why is serializing cached values consistently important, and what breaks if you change it?
A: Cached values are serialized (e.g. JSON or a binary format) before storage and deserialized on read, so the writer's and reader's formats must agree. If a deploy changes the serialization, existing cached entries become unreadable or are misinterpreted, causing errors or silently wrong values. Treat the serialization format as part of the key version, and migrate by versioning keys rather than reusing old ones with a new format.

Q: What does it mean to "cache the computation, not the inputs," and when is that the right call?
A: It means storing the finished, expensive-to-produce result (an aggregate, a rendered report, a feature vector) rather than the raw inputs that everyone would then have to recompute. It is the right call when the computation is costly and widely reused but the inputs change slowly, so one computation serves many reads. The risk is that an input change must invalidate the cached computation, so you trade compute savings for invalidation responsibility.

Q: A monitoring graph shows the cache hit ratio is high but consumers still complain about stale numbers. Where do you look?
A: A high hit ratio with stale complaints points at invalidation, not capacity: the cache is happily serving entries that should have been invalidated when the source changed. Look for write paths that update the database but skip cache invalidation, TTLs longer than the data's real change cadence, or sliding TTLs pinning hot keys. The fix is tightening invalidation, not enlarging the cache.

Q: Why should the decision to cache, and its TTL, be recorded in an architecture decision record (ADR)?
A: Because caching trades correctness (staleness) against load, the rationale — what is cached, where, the TTL, and why that staleness window is acceptable — is a design decision future engineers and auditors must be able to reconstruct. An ADR captures the staleness-vs-load reasoning so a later change does not silently reintroduce stale-data bugs. In a regulated context it also documents that the freshness implications of caching were consciously accepted.

Q: Summarize the architect's caching responsibility in one or two sentences.
A: The architect decides whether and where to cache each dataset and what staleness is acceptable, anchoring every choice to the staleness-vs-load trade-off and the data's update cadence, rather than implementing the cache mechanics. Done well this turns slow-changing data like NAV into nearly free wins while keeping fast-changing data like intraday positions correct, and it records the TTL and rationale so the decision is defensible.

Q: You need to delete all `nav:*` keys after a model change. Why is running `KEYS nav:*` in production dangerous, and what do you use instead?
A: `KEYS` does an O(N) scan of the entire keyspace and, because Redis/Valkey executes commands on a single thread, it blocks every other client for the whole scan — on millions of keys that can freeze the server for seconds and cause connection errors platform-wide. Use `SCAN` (with `MATCH nav:*` and a `COUNT` hint) instead: it is cursor-based and non-blocking, returning a slice of keys per call so the server keeps serving other requests between iterations. The same rule applies to `HSCAN`, `SSCAN`, and `ZSCAN` for collection types.

Q: How exactly do you compute the cache hit ratio from a Redis/Valkey instance, and which `INFO` fields do you read?
A: Run `redis-cli INFO stats` and read the `keyspace_hits` and `keyspace_misses` counters, then compute `keyspace_hits / (keyspace_hits + keyspace_misses)`; a sustained value above roughly 0.9 is generally healthy for a read cache. Those counters are cumulative since the instance started, so for a live picture you sample the delta over a window rather than the lifetime average. A collapsing delta ratio is the fastest signal that TTLs are too short, the working set is being evicted, or keys are orphaned.

Q: In a Redis/Valkey cluster, why might `MGET nav:isin1 nav:isin2` fail, and what is the mechanism?
A: A cluster shards the keyspace across `16384` hash slots by CRC16 of the key, and a multi-key command like `MGET` requires every key to live in the same slot; if the two ISINs hash to different slots you get `CROSSSLOT Keys in request don't hash to the same slot`. The mechanism is that each slot lives on one node, so a single command cannot span nodes. The fix is to co-locate the keys in one slot with a hash tag, or to issue separate single-key gets.

Q: What is a Redis/Valkey hash tag, and how would you use it to keep a fund's keys in one slot?
A: A hash tag is the substring inside curly braces in a key name; when present, only that substring is hashed to choose the slot, so `nav:{GB00B03MLX29}:price` and `nav:{GB00B03MLX29}:ccy` both route to the same slot and can be fetched together with `MGET`. You use it to deliberately co-locate keys that you need to read or invalidate atomically, such as all attributes of one ISIN. The gotcha is cardinality: tagging by a low-variety value (e.g. tenant) crams too many keys onto one node and creates a hot slot, so tag by a high-cardinality id.

Q: Under the OSS cluster policy, which multi-key behaviour differs from the Enterprise cluster policy in Azure Managed Redis?
A: With the OSS cluster policy every multi-key command is strictly slot-bound, so any cross-slot `MGET`/`MSET`/`DEL` raises `CROSSSLOT`; the Enterprise cluster policy is more permissive and allows a small set of multi-key commands — `DEL`, `MSET`, `MGET`, `EXISTS`, `UNLINK`, `TOUCH` — to span slots. The trade-off is throughput: OSS exposes the cluster shape and needs a cluster-aware client but scales bandwidth, while Enterprise uses a single endpoint that is simpler but can bottleneck at high load. The architect picks based on whether the client library is cluster-aware and how multi-key the access pattern is.

Q: Name the four Azure Managed Redis tiers and the one design constraint you must respect when choosing among them.
A: The tiers are Memory Optimized (8:1 memory-to-vCPU, cheapest, dev/test or memory-heavy low-throughput), Balanced (4:1, standard workloads), Compute Optimized (2:1, maximum throughput), and Flash Optimized, which tiers cold data from RAM onto NVMe for large datasets at lower cost. The hard constraint is that you cannot scale between Flash Optimized and any of the in-memory tiers in place, so picking Flash later means a migration. Choose the tier from your throughput-to-memory ratio and whether the working set fits economically in RAM.

Q: When is Azure Managed Redis Flash Optimized the right tier, and what is the catch for a hot serving cache?
A: Flash Optimized fits large datasets where much of the data is cold, because it automatically moves infrequently accessed keys from RAM to NVMe, cutting memory cost for a big keyspace. The catch is latency: a read that has to come from NVMe is slower than an in-memory hit, so if your serving cache needs uniformly low p99 latency on a hot working set, Flash can hurt the tail. Use it for big, skewed datasets, not for a small, uniformly hot cache where Compute or Balanced tiers keep everything in memory.

Q: A consumer reports the NAV API returned yesterday's NAV at 09:05 after a 08:00 strike. Walk through the first three things you check.
A: First confirm the publish-time invalidation actually fired: did the NAV-publish job `DEL` the `nav:{isin}:{date}` key (or bump the version prefix), and did it complete before 09:05? Second check the key itself with `TTL nav:{isin}:{date}` and `GET` to see whether a stale value is still cached and how long it has left. Third check for a replica-read path: if the read came from a lagging replica, the new write may not have propagated yet, so route compliance-sensitive reads to the primary. The usual root cause is a missing or failed invalidation, with TTL as the safety net that did not yet trigger.

Q: How would you implement publish-time invalidation for NAV without a race against in-flight readers?
A: Write the new NAV durably to the database first, then either `DEL` the `nav:{isin}:{date}` key or set the new value with `SET` plus the new TTL in one step, and only acknowledge the publish after the cache step succeeds. To avoid the cache-aside repopulation race — a slow reader writing back the old value after your delete — bump a version prefix (`v4:nav:...`) so any late write lands under a key nobody reads, or use a short delayed second delete. The version-bump approach is cleanest because it makes the stale write unreachable rather than relying on timing.

Q: What is the difference between `DEL` and `UNLINK` in Redis/Valkey, and which is safer for invalidating many keys?
A: `DEL` frees the keys' memory synchronously on the main thread, so deleting a large key or a big batch can block other commands while the reclamation runs; `UNLINK` removes the keys from the keyspace immediately but reclaims their memory asynchronously on a background thread. For invalidating many or large entries, `UNLINK` is safer because it avoids a latency spike on the single command thread. Use `DEL` for small, simple invalidations where the blocking cost is negligible.

Q: You want a read to reflect its own just-committed write in a replicated cache. What command or routing choice helps?
A: Replication is asynchronous by default, so a replica can lag the primary and miss the latest write; for read-your-writes you route the subsequent read to the primary rather than a replica. If you must confirm a write reached replicas before proceeding, `WAIT numreplicas timeout` blocks until the given number of replicas have acknowledged the write or the timeout elapses, giving a bounded durability guarantee. Note `WAIT` adds latency and still is not full transactional durability, so reserve it for writes where replica confirmation genuinely matters.

Q: How does `SET key value EX 3600 NX` differ from a plain `SET`, and where is it useful in caching?
A: `EX 3600` sets a 3600-second absolute TTL in the same atomic command, and `NX` makes the write conditional on the key not already existing, so it succeeds only on the first writer. This combination is useful as a lightweight single-flight lock: the first request to miss acquires the lock key with `NX` and recomputes, while others see the existing lock and wait or serve the old value, collapsing a stampede. The `EX` on the lock guarantees the lock self-expires if the holder crashes, avoiding a deadlock.

Q: Describe a concrete single-flight lock for an expensive semantic-layer rollup using Redis/Valkey primitives.
A: On a miss, a request attempts `SET lock:rollup:{id} <token> NX EX 30`; if it wins the lock it recomputes the rollup, writes the result with `SET rollup:{id} <value> EX <ttl>`, then releases the lock, while losers briefly retry the cache or serve the previous value. The `NX` guarantees exactly one recompute and the `EX 30` ensures the lock cannot wedge if the worker dies mid-computation. To release safely you delete the lock only if the token still matches yours, typically via a small Lua script so the check-and-delete is atomic.

Q: Why use a Lua script (or `EVAL`) to release a single-flight lock instead of a plain `DEL`?
A: A plain `DEL lock:...` can delete a lock that a different worker has since acquired — your lock expired, someone else took it, and you delete theirs — reintroducing concurrent recomputes. A Lua script run via `EVAL` reads the lock's token and deletes it only if it equals your token, and because Redis/Valkey runs the whole script atomically on its single thread, the compare-and-delete cannot be interleaved. This is the standard safe-release idiom for a token-based lock.

Q: What does probabilistic early expiration (the XFetch / early-recompute technique) do, and which problem does it target?
A: It refreshes a cached value slightly before its real TTL, with the probability of an early refresh rising as the key ages, so a hot key gets recomputed by one request while it is still serving the cached value to others. It targets the cache stampede on expiry: instead of every request missing at the exact expiry instant, one early refresh keeps the key warm and the herd never forms. It complements TTL jitter — jitter desynchronises many keys, early recompute protects a single very hot key.

Q: A daily semantic rollup and its NAV inputs both expire at exactly midnight, causing a load spike. Give two fixes.
A: First, add jitter: set each key's TTL to its base value plus a small random offset so expiries smear across a window instead of firing simultaneously at midnight. Second, decouple refresh from expiry by warming the cache when the upstream nightly job finishes — recompute and re-`SET` the rollup proactively so reads never hit a cold key. Tying invalidation to the known refresh event, rather than a synchronized clock-based TTL, removes the synchronized miss entirely.

Q: For an EMT file published monthly, what TTL and invalidation design balances freshness against load?
A: Because an EMT (European MiFID Template) is immutable once issued and changes only on the monthly cycle or an ad-hoc reissue, cache the parsed contents hard with a long TTL (days) and rely primarily on explicit invalidation when a new file lands. Wire the ingestion job to `DEL` or version-bump the affected keys on receipt of a new EMT, so the long TTL is just a safety net. This is the cache-hard pattern: freshness comes from the known publish event, not from expiring frequently and re-parsing the file needlessly.

Q: An EET (European ESG Template) field used in a product-governance check is served from cache. What extra caution applies versus a plain reference field?
A: An EET (European ESG Template) field feeding a regulatory or governance determination is compliance-sensitive, so a stale cached value can produce a wrong control outcome that is hard to defend in audit. Treat it like any compliance read: either bypass the cache for that field, keep its TTL short and explicitly invalidated on EET reissue, and ensure the determination records which EET version it used. The general reference-data caching is fine, but the moment a cached field gates a regulatory decision, the staleness window must be demonstrably acceptable and traceable.

Q: How does a cache TTL interact with a T+1 settlement deadline for shared static-settlement-instruction (SSI) lookups?
A: SSIs are slow-changing reference data, so caching them sharply cuts load during the settlement run, but a T+1 cycle compresses the window in which a corrected SSI must take effect, so a too-long TTL risks settling on superseded instructions. The design is a moderate TTL plus explicit invalidation whenever an SSI is amended, so a same-day correction propagates before the next settlement batch reads it. The principle is that a shorter settlement deadline tightens the freshness budget that the TTL must respect.

Q: Why is `noeviction` a deliberate, sometimes-correct choice rather than always a misconfiguration?
A: With `noeviction`, once `maxmemory` is reached the instance rejects writes that would grow memory and returns errors instead of silently dropping keys, which is wrong for a general read cache that expects to make room. But it is correct when the instance holds data you must not lose to eviction — for example when Redis/Valkey is used as a durable-ish queue or a lock store — where failing the write loudly is safer than discarding a needed key. The architect picks `noeviction` only when every stored item is load-bearing and eviction would be a correctness bug.

Q: Distinguish `volatile-lru` from `allkeys-lru` and say when you would prefer `volatile-lru`.
A: `allkeys-lru` may evict any key under memory pressure, while `volatile-lru` only evicts keys that have a TTL set, leaving keys without an expiry untouched. You prefer `volatile-lru` when the same instance mixes evictable cached entries (which you give TTLs) with a few persistent keys that must survive pressure (which you leave TTL-less), so eviction can never touch the must-keep keys. The risk is that if you forget to set TTLs on cache entries, nothing is eligible to evict and writes start failing like `noeviction`.

Q: A team set `allkeys-lfu` but their hot set changes hourly and the hit ratio is poor. What is the likely mismatch?
A: LFU evicts by access frequency accumulated over time, so it favours keys that were historically popular and is slow to demote a key whose popularity has faded; with an hourly-shifting hot set, yesterday's frequently-hit keys linger while the genuinely hot new keys get squeezed out. LRU, which evicts by recency, adapts faster to a churning working set. The fix is to switch to `allkeys-lru` (or tune LFU decay), matching the policy to a bursty rather than steadily-skewed access pattern.

Q: How do you tell whether a low hit ratio is caused by eviction versus by short TTLs?
A: Compare the `evicted_keys` and `expired_keys` counters from `INFO stats`: a climbing `evicted_keys` with memory pinned at `maxmemory` points to capacity-driven eviction of hot data, while a high `expired_keys` with memory well under the limit points to TTLs expiring entries before they are reused. Eviction says "grow the cache or fix key cardinality"; expiry says "lengthen TTLs or use invalidation instead of short expiry." Reading both counters separates a sizing problem from a freshness-policy problem.

Q: What is the difference between `used_memory` and `maxmemory`, and why watch the ratio?
A: `used_memory` is how much the instance is currently consuming and `maxmemory` is the configured ceiling at which the eviction policy kicks in; the ratio of the two is your headroom. Watching it warns you before the cache starts evicting hot keys or, under `noeviction`, before writes begin failing. A ratio sitting near 1.0 with rising `evicted_keys` is the classic undersized-cache thrash signature.

Q: Why can a single oversized cache key (a "big key") hurt a Redis/Valkey instance, and how would you detect it?
A: Because operations on one key are not parallelised across the single command thread, a multi-megabyte value or a collection with millions of elements makes commands touching it slow, and deleting it with `DEL` can block the whole server while memory is freed. Detect big keys with `redis-cli --bigkeys` (a sampling scan) or `MEMORY USAGE key` for a specific key, and reclaim large ones with `UNLINK` rather than `DEL`. In a cache, prefer splitting a giant aggregate into smaller keyed slices over one monolithic value.

Q: How does caching a cached aggregate (a pre-aggregate) interact with cache-key cardinality, and what is the win?
A: Caching a coarse pre-aggregate (e.g. fund-level totals) collapses many fine-grained query keys into one stable, frequently-reused key, so cardinality stays low and the hit ratio high, whereas caching every raw parameterized query fragments memory into one-off entries. The win is that the expensive scan runs once per refresh and serves all viewers from one warm key. The cost is loss of drill-down: the pre-aggregate cannot answer a finer-grained question than the grain it was rolled up to.

Q: Two app instances each keep a local in-process cache of fund reference data. A correction is published. What goes wrong and how do you fix it?
A: Deleting the key in the shared store does not reach the per-instance local copies, so each instance keeps serving the old attribute until its own local TTL expires, producing inconsistent answers depending on which instance handled the request. The fix is a fan-out invalidation: publish an "invalidate {key}" message on a Redis/Valkey `PUBLISH` channel (or a message bus) that every instance subscribes to and drops its local copy on receipt. Keep local TTLs short as a backstop, because pub/sub delivery is best-effort and a missed message would otherwise pin the stale value.

Q: What consistency caveat applies to pub/sub-based local-cache invalidation that does not apply to deleting one shared key?
A: Pub/sub propagation is asynchronous and best-effort, so there is a brief window where some instances have processed the invalidate message and others have not, meaning two instances can disagree until the message lands, and a dropped message leaves an instance stale until its TTL. Deleting a single shared key is instantaneous and authoritative for every reader of that shared cache. So local caches buy speed at the cost of a propagation window and a need for a TTL backstop.

Q: In `cache-aside`, why is writing the database before deleting the cache key strictly required, not just preferred?
A: If you delete the key first and the database write then fails or is slow, a concurrent reader can miss, read the still-old database value, and repopulate the cache with stale data that now outlives a successful retry — and you have also lost the protection of having committed truth before exposing the miss. Writing the durable source first means that by the time the key is deleted, any repopulating read sees the new value. The order narrows the inconsistency window to the gap between commit and delete rather than opening an unbounded one.

Q: What is "dogpile" protection via serving a stale value, and when is it acceptable?
A: Dogpile protection means that when a key expires under load, one request recomputes while the others are briefly served the just-expired (slightly stale) value instead of all stampeding the backend. It is acceptable whenever a few extra seconds of staleness is harmless — most dashboard aggregates and reference lookups — because the alternative is a synchronized backend overload. It is not acceptable where the value gates a correctness or compliance decision, in which case you block on the recompute or bypass the cache.

Q: How would you cache LEI lookups so that the active/lapsed status used for compliance stays trustworthy while attributes stay cheap?
A: Split the entry by freshness need: cache slow-changing LEI attributes (legal name, address, parent) with a long TTL (hours to a day) since they rarely change, but cache the registration-status field with a short TTL or bypass the cache for it when it gates a compliance decision. This way the hammering of the GLEIF registry for stable attributes is eliminated while the field whose staleness could pass a check on a lapsed entity stays fresh. Keying status separately (`lei:{lei}:status`) lets you give it its own TTL and invalidate it independently.

Q: A new deploy quietly bumped the cache key prefix and now both old and new entries coexist, doubling memory. What happened and what is the clean approach?
A: The deploy effectively orphaned the old-prefix entries — the app now reads and writes only the new prefix, so the old keys are dead weight occupying memory until they age out via TTL or eviction, and during the transition the new prefix is cold so the database load spikes. The clean approach is to treat the prefix bump as a deliberate version invalidation, warm the new prefix proactively, and let TTLs reap the old keys, or explicitly `SCAN`+`UNLINK` the old prefix once cutover is confirmed. Versioning is a feature here; the bug was doing it unintentionally and without a warm-up.

Q: Why should the serialization format be folded into the cache key version, with a fund-data example?
A: Reader and writer must agree on how a value is encoded, so if a deploy changes a NAV record from JSON to a binary format, existing entries become unreadable or silently misparsed, yielding errors or wrong numbers from values that look valid. Folding the format into the version — `v2:nav:{isin}:{date}` for the new encoding — means old and new entries never collide and the old ones age out under their TTL. This turns a risky in-place format change into a safe parallel migration.

Q: How do you cache multi-tenant fund data so one management company can never see another's cached values?
A: Make the tenant identity a mandatory, leading part of the key, e.g. `tenant:{mancoId}:nav:{isin}:{date}`, so two tenants requesting the same ISIN map to distinct keys and a value can never be served across tenant boundaries. Omitting the tenant is not merely a low-reuse miss but a data-leak and correctness bug. In a cluster, you can additionally hash-tag on the tenant to co-locate a tenant's keys, but watch for a hot slot if one tenant dominates.

Q: Why is a sliding TTL on a compliance-relevant reference value a latent bug, framed as a scenario?
A: A sliding TTL resets the expiry on every access, so a heavily-read value — say an instrument's eligibility flag — that the source has since changed can be pinned in cache indefinitely because it never goes long enough without a read to expire. A compliance check then keeps reading the never-expiring stale flag long after the underlying status flipped. The fix is an absolute TTL (or explicit invalidation) so the value has a hard maximum age regardless of how popular it is.

Q: Negative caching of unknown ISINs protects the master, but what failure does a too-long negative TTL cause?
A: Negative caching stores the "not found" result so repeated probes for an invalid ISIN do not hammer the reference master, but if the negative TTL is long and that ISIN later becomes valid (a newly launched share class), the cache keeps masking it as not-found until the negative entry expires. The fix is a deliberately short negative TTL plus invalidation of the negative key when the entity is created, so a genuine new record appears promptly. The principle is that negative entries should expire faster than positive ones because their "truth" is more likely to flip.

Q: How would you measure whether a cache is actually helping, beyond the raw hit ratio?
A: Track p99/p99.9 latency with and without the cache path and the backing-store QPS or CPU, not just the average, because a cache can show a high hit ratio while a fat tail of misses still hurts users and SLAs. Also watch `evicted_keys`, `expired_keys`, and the hit-ratio delta so you can attribute a regression to capacity, freshness, or orphaning. The goal you stated — latency reduction versus load reduction — determines whether you celebrate a lower p99 or a lower backend QPS.

Q: A cache shows a 99% hit ratio but the system is still down. How can both be true?
A: A near-perfect hit ratio only means most reads were served from cache; it says nothing about the latency of the misses, the health of the backing store the misses hit, or whether the cached values are correct. The 1% of misses can stampede an already-fragile backend, or the cache can be confidently serving stale/wrong values that break downstream logic, so the system fails despite the flattering ratio. Hit ratio is necessary but not sufficient — you must also watch tail latency, backend load, and freshness.

Q: When the cache is unavailable and you fail open, what capacity assumption must hold, and how do you verify it?
A: Failing open sends every request straight to the backing store, so the database/warehouse must be sized to absorb the full uncached load — otherwise fail-open turns a cache hiccup into a backend meltdown. Verify it by load-testing the backend at the uncached request rate, or by computing peak QPS times the per-query cost against the backend's headroom. If the backend cannot survive that, you need a circuit breaker, request shedding, or a smaller blast radius, not naive fail-open.

Q: How does an `ETag`/`If-None-Match` revalidation reduce load even on a cache miss?
A: When a client or CDN holds a possibly-stale copy, it revalidates by sending `If-None-Match` with the stored `ETag`; if the resource is unchanged the origin replies `304 Not Modified` with no body, so the expensive payload is not regenerated or re-sent even though the client "missed" its local freshness window. This is a conditional request that saves bandwidth and often backend work while still confirming freshness. It complements `max-age`: `max-age` avoids the round-trip entirely while it lasts, `ETag` makes the post-expiry round-trip cheap.

Q: For a published fund fact sheet served via CDN, what `Cache-Control` directives fit, and why not for a per-investor statement?
A: A public, slow-changing fact sheet can use `Cache-Control: public, max-age=<seconds>` so the CDN and browsers cache it at the edge for its known refresh cadence, maximising offload. A per-investor statement is private and must never be shared across users, so it needs `Cache-Control: private` (or `no-store` for the most sensitive) to keep intermediaries from caching one investor's document and serving it to another. The directive encodes the same staleness-and-isolation reasoning you apply to a server-side cache, pushed to the HTTP layer.

Q: Why does `Cache-Control: no-cache` still allow caching, and when would you choose it over `no-store`?
A: `no-cache` permits a cache to store the response but forbids reusing it without first revalidating with the origin (typically via `ETag`/`If-None-Match`), so you still get a cheap `304` when unchanged but never serve a stale body. You choose it over `no-store` when the data may be reused if confirmed fresh — you want the bandwidth savings of revalidation but cannot tolerate blind reuse. `no-store` is the stricter choice for data that must never sit in any cache at all, such as highly sensitive payloads.

Q: How can RDB/AOF persistence be used to avoid a cold-cache stampede after a planned restart?
A: Enabling RDB snapshots or AOF lets the instance reload its prior keyspace on restart, so after a reboot the cache comes back warm and reads do not all miss into the backing store at once. This is a warm-restart convenience, not durability you rely on for correctness — the cache is still not a system of record. For a maintenance window, taking a snapshot before restart (or relying on AOF replay) is a cheaper warm-up than recomputing the working set from scratch.

Q: Why does `appendfsync everysec` leave a window of possible data loss, and why is that acceptable for a cache?
A: With `appendfsync everysec` the append-only file is flushed to disk roughly once per second, so a crash can lose up to about a second of acknowledged writes; `always` removes the window but adds per-write latency. For a cache this is acceptable because the backing database is the source of truth — losing a second of cache writes just means a few extra misses, not lost records. It would be unacceptable only if you wrongly treated the cache as your durable store.

Q: How do you bound the freshness contribution of caching in an end-to-end data-product freshness SLA?
A: Treat the worst-case cache age — the TTL plus any replication lag and invalidation-propagation delay — as an additive term in the end-to-end freshness budget, alongside pipeline and ingestion latency. So if the SLA is "no older than 15 minutes" and upstream already consumes 10, the cache TTL plus its lags must fit inside the remaining 5. Monitoring should measure freshness end-to-end through the cache, so a freshness alert that fires while the pipeline looks healthy correctly points at the cache.

Q: Give the defensible cache decision for an intraday risk-exposure endpoint, contrasting it with NAV.
A: Intraday exposure changes continuously as trades execute and is read to make risk decisions, so there is no daily freeze and any non-trivial TTL can serve a stale exposure that is an outright correctness bug — so you either do not cache it or cache only with a very short, explicitly-reasoned window and live-read for hard checks. This is the mirror image of NAV, which is immutable from its daily strike and so is cached hard with a long TTL plus publish-time invalidation. The same staleness-vs-load framework yields opposite answers because the update cadence and freshness requirement are opposite.

Q: What goes in the ADR for caching the NAV API so a future engineer or auditor can reconstruct the decision?
A: Record what is cached (`nav:{isin}:{date}`), where (shared serving-tier Redis/Valkey or Azure Managed Redis, not local in-process), the TTL and why (aligned to the daily valuation cadence, expiring near the next strike) plus the publish-time invalidation, and the explicit statement that the resulting staleness window is acceptable because NAV is immutable intraday. Also note the fail-open behaviour and the backend's capacity to absorb uncached load, and the tenant-scoped key design. Capturing this anchors the staleness-vs-load reasoning so a later change cannot silently reintroduce stale-data bugs, and it evidences to an auditor that freshness was consciously accepted.

Q: How would you verify which eviction policy is active and change it at runtime without a restart?
A: Read the current policy with `CONFIG GET maxmemory-policy` and the ceiling with `CONFIG GET maxmemory`, then change either live with `CONFIG SET maxmemory-policy allkeys-lru` so the new behaviour takes effect immediately without dropping connections. Remember that `CONFIG SET` mutates only the running process, so you must also persist the change to the config file (or use `CONFIG REWRITE`) or it reverts on the next restart. On a managed service like Azure Managed Redis you set these through the service's configuration surface rather than raw `CONFIG SET`, which may be restricted.

Q: Why is treating the cache as the audit trail for a regulatory check a mistake, even with persistence enabled?
A: A cache is volatile, TTL-bounded, and evictable, so an entry that recorded "this control read value X at time T" can vanish on expiry, eviction, or node loss, leaving no defensible record of what the check actually saw — and RDB/AOF persistence only aids warm restarts, not durable audit retention. Audit evidence for a DORA-style operational-resilience or compliance determination must live in a durable, append-only store (the database, an event log, or object storage), with the cache used only to accelerate the read. Confusing the accelerator with the record of truth produces exactly the unreconcilable gap an auditor will challenge.

Q: A cached value is correct in the shared cache but wrong at the CDN edge after a NAV correction. Where is the bug and how do you fix it?
A: The layers were invalidated inconsistently: the publish job deleted the application/shared-cache key but the CDN is still serving the old response within its `max-age`, so the edge copy outlives the corrected value. The fix is to invalidate every caching layer on the known event — purge the CDN path (or use a short `max-age` plus `ETag` revalidation for correction-prone resources) at the same time you delete the shared key. This is the general lesson that each additional cache layer is an independent invalidation surface, so a correction must fan out to all of them, not just the one nearest the database.
