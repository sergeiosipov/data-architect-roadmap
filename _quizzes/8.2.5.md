# Phase 2 · 8.2.5 Data Vault modeling (+ 1.4.7 pattern) — 150 self-test questions

Q: What is Data Vault 2.0 and what problem does it solve for a regulated fund-administration warehouse?
A: Data Vault 2.0 is an enterprise data warehouse modeling pattern, defined by Linstedt and Olschimke, built from hubs, links, and satellites and loaded insert-only. It solves auditability-by-construction and multi-source integration: you can absorb a second transfer-agency feed without re-modeling the existing warehouse, and every change is preserved as a new row an auditor can trace. The trade-off is ceremony — on a single small source it is overkill.

Q: What are the three core table types of a raw Data Vault?
A: Hubs, links, and satellites. Hubs store the distinct list of business keys, links store the relationships (associations) between business keys, and satellites store the descriptive, time-variant context and history for hubs and links. This deliberate split is what lets you add sources and attributes independently without rewriting existing structures.

Q: What does a hub table actually store?
A: A hub stores the unique list of a single business key plus a small fixed set of system columns: the hash key (its surrogate primary key), the business key column(s), the load datetime, and the record source. It deliberately holds no descriptive attributes and no relationships — only the agreed identity of a business concept such as a fund, share class, or administrator.

Q: What is a business key in Data Vault terms?
A: A business key is the identifier the business actually uses to recognise an entity in the real world, independent of any source system's surrogate key. For a fund that might be an ISIN; for a share class an ISIN plus class code; for an administrator a LEI. Choosing it well is the single most consequential modeling decision because hubs and integration hinge on it.

Q: Why must a hub's business key be source-system-independent?
A: Because the whole point of the hub is to integrate the same real-world entity arriving from multiple systems into one identity. If you keyed the hub on a source surrogate (e.g. a TA system's internal account number), the same fund from a second TA would land as a separate hub row and integration would fail. A business key like ISIN or LEI is stable across systems, so both feeds map to the same hub record.

Q: What does a link table represent?
A: A link represents a relationship or transaction connecting two or more business keys — for example the association between a fund hub, a share-class hub, and an administrator hub. It is many-to-many by construction and stores the hash keys of the participating hubs, its own link hash key, the load datetime, and record source, but no descriptive context.

Q: Why does Data Vault model every relationship as many-to-many?
A: Modeling links as many-to-many future-proofs the relationship: if the business rule changes so that what was one-to-many becomes many-to-many (e.g. a share class is later serviced by two administrators), no structural change is needed — only new link rows. Cardinality is enforced and reported through the data and business vault, not baked into the link's structure, preserving non-destructive evolution.

Q: What goes in a satellite table?
A: A satellite holds the descriptive attributes and their history for exactly one parent hub or one link. It stores the parent hash key, the load datetime (part of the primary key), the descriptive columns, a hashdiff for change detection, and the record source. Multiple satellites can hang off the same hub, typically split by source or rate of change.

Q: Why are satellites separated from hubs rather than stored as columns on the hub?
A: Because context changes over time and varies by source, whereas the business key's identity does not. Keeping descriptive, time-variant data in satellites lets you keep full history, add a new source's attributes as a new satellite, and load each at its own cadence — all without touching the stable hub. It is the separation of identity, relationship, and context that gives Data Vault its agility.

Q: How is history captured in a satellite without overwriting data?
A: Every change produces a new satellite row keyed by parent hash key plus load datetime; the prior row is left untouched. There is no UPDATE — the current version is simply the row with the latest load datetime for that key. This insert-only history is what makes the full timeline of a fund's attributes reconstructable.

Q: Why might you split one hub's attributes across several satellites?
A: You split satellites by source system and by rate of change. Splitting by source keeps each feed's data isolated and independently loadable; splitting by rate of change avoids re-storing slow-changing attributes every time a fast-changing one moves. For a fund hub you might have one satellite for stable static data and another for frequently updated NAV-related descriptive fields.

Q: What system columns appear on essentially every Data Vault table?
A: A load datetime (when the row was loaded) and a record source (which feed it came from), plus the relevant hash key(s). Hubs and links use the load datetime to mark first-seen; satellites include it in the primary key to version history. The record source is essential for the audit trail — it tells an auditor exactly which system asserted each fact.

Q: What is the record source column for and why does an auditor care?
A: The record source records the originating system or file for each row, so any fact can be traced back to who asserted it. In fund administration, when a NAV or holding figure is questioned, the record source lets you prove whether it came from the transfer agency, the fund accountant, or a market-data feed. It is part of auditability-by-construction, not an afterthought.

Q: Distinguish identity, relationship, and context in one sentence each.
A: Identity is the business key, stored once per concept in a hub. Relationship is the association between business keys, stored in a link. Context is the descriptive, time-variant detail, stored in satellites; separating the three is the foundational idea of Data Vault.

Q: What is the raw vault?
A: The raw vault is the layer of hubs, links, and satellites loaded directly from source data with no business rules applied — a faithful, integrated-by-business-key copy of what the sources said. Data is cleansed only of hard technical issues (type alignment, trimming), never soft business logic, so the raw vault remains a defensible system of record.

Q: What is the business vault and how does it differ from the raw vault?
A: The business vault is a sparse set of additional hubs, links, and satellites that hold the results of applied business rules and computations — derived attributes, computed satellites, mappings. It sits on top of the raw vault and is rebuildable from it, so if a rule changes you recompute the business vault without re-ingesting source data. The boundary is exactly where raw integration ends and computed logic begins.

Q: Why is the boundary between raw vault and business vault important for auditability?
A: Because it cleanly separates "what the source said" (raw, immutable, reproducible) from "what we computed" (business vault, derivable). An auditor can verify that the raw vault matches the source files, then separately review the business rules that produced computed values. If you mixed business logic into the raw load, you could no longer prove the warehouse is a faithful copy of the source.

Q: Can the business vault be rebuilt from the raw vault, and why does that matter?
A: Yes — by design the business vault contains only derived structures, so it can be dropped and regenerated from the raw vault. This matters because a corrected or changed business rule then only requires recomputing the business vault, with no risky re-ingestion of source data and no loss of the original raw record. It localises the blast radius of rule changes.

Q: Where should soft business rules live in a Data Vault, and where should they not?
A: Soft business rules — interpretations, derivations, harmonisation that could change — belong in the business vault and downstream marts, never in the raw vault load. The raw vault applies only hard rules (technical type coercion, splitting, hashing). Keeping soft rules out of the raw layer is what guarantees the raw vault stays an unaltered system of record.

Q: What is a computed satellite?
A: A computed satellite is a business-vault satellite whose attributes are the output of applied business logic rather than raw source values — for example a satellite holding a derived risk classification or a harmonised currency. It hangs off an existing raw hub or link and is fully rebuildable from raw data, keeping the computation explicit and re-runnable.

Q: What is a hash key in Data Vault 2.0 and why is it used instead of a sequence surrogate?
A: A hash key is a deterministic surrogate produced by hashing the business key(s) — by default MD5 in AutomateDV. It is used instead of a database sequence because it can be computed independently on every node in parallel without a central counter, enabling fully parallel, load-order-independent loading. The same business key always yields the same hash key, so loads need no lookups against existing keys.

Q: How is a hub hash key computed?
A: The hub hash key is computed by hashing the standardised business key(s) for that hub — typically uppercased and trimmed, with a defined null placeholder. In AutomateDV the default algorithm is MD5, the multi-column concatenator defaults to `||`, and the null placeholder defaults to `^^`. Determinism is the whole point: identical business keys produce identical hash keys across systems and runs.

Q: How is a link hash key computed?
A: A link hash key is computed by hashing the combined business keys of all hubs participating in the relationship, in a fixed, agreed column order. Because it is derived from the underlying business keys (not the hub hash keys' sequences), it too is deterministic and reproducible independently on every node. Column order must be standardised or two systems would compute different link hashes for the same relationship.

Q: What is a hashdiff and what is it used for?
A: A hashdiff is a hash of all the descriptive attribute values in a satellite row, stored alongside the row. It enables cheap change detection: to decide whether a source record represents a change, you compare its computed hashdiff to the latest stored hashdiff for that key instead of column-by-column comparison. If they match, nothing changed and no new satellite row is inserted.

Q: Why is comparing a hashdiff cheaper than comparing every column?
A: Because a single fixed-width hash comparison replaces an N-column equality check across potentially dozens of wide attributes, which is faster and far simpler to express in SQL. It also handles nulls uniformly through the null placeholder, avoiding the `NULL <> NULL` pitfalls of column-wise comparison. The cost is computing the hash once at load time.

Q: What determinism property must hashing guarantee across systems?
A: Given the same input values, the same standardisation, the same column order, and the same null placeholder, the hash must be byte-for-byte identical on every platform and every run. Without this, two systems loading the same fund would compute different hash keys and integration would silently break. This is why AutomateDV exposes config for casing, concatenator, and null placeholder.

Q: Why does the order of columns matter when computing a multi-column hash?
A: Because hashing is over a concatenated string, swapping column order changes the input and therefore the hash. A hub on ISIN plus share-class code must always concatenate them in the same fixed order, or the same key produces different hashes in different models or systems. AutomateDV uses the order you declare in the staging hash configuration, so it must be standardised.

Q: What is the default hash algorithm in AutomateDV and how do you change it?
A: The default is `MD5`. You change it by setting the package-level `hash` variable in `dbt_project.yml` (e.g. to `SHA` for SHA-256), which switches the algorithm used by the `hash_columns` staging macro across all models. The choice trades collision resistance and width against storage and compute cost.

Q: Why might a team choose SHA-256 over MD5 for hash keys?
A: To reduce the (already tiny) probability of hash collisions and to meet security or governance policies that disallow MD5. The cost is wider keys (32 bytes vs 16) and slightly more compute, which inflates storage and join cost across a large vault. Many teams accept MD5 for surrogate keys because collisions are statistically negligible at warehouse scale, but a regulated shop may mandate SHA-256.

Q: How are nulls handled when computing a hashdiff, and why does it matter?
A: Nulls are replaced with a fixed null placeholder string (AutomateDV default `^^`) before hashing, so a null is hashed as a stable, distinct token. This matters because without it, an actual data value that happened to equal an empty string could collide with a null, or `NULL` concatenation could produce inconsistent results across platforms. The placeholder makes null-versus-value changes detectable and deterministic.

Q: Why does AutomateDV uppercase hash content by default, and when would you disable it?
A: It uppercases (via `hash_content_casing: UPPER`) so that case-only differences in source values do not register as changes — `LU0123456789` and `lu0123456789` hash to the same key. You disable it (`DISABLED`) only when case is genuinely meaningful and a lowercase value must be treated as a different value. For ISINs, uppercasing is appropriate since they are conventionally uppercase.

Q: What are the three defining characteristics of Data Vault loading?
A: Loads are parallel, insert-only, and idempotent in the sense that re-running them does not corrupt history. Parallelism comes from deterministic hash keys removing the need for sequential surrogate lookups; insert-only means no UPDATE or DELETE; and the combination is what makes the warehouse auditable and restartable. These properties follow directly from the hash-key and satellite design.

Q: What does insert-only loading mean and what is explicitly forbidden?
A: Insert-only means every load only appends new rows; existing rows are never UPDATEd or DELETEd. A changed attribute becomes a new satellite row with a later load datetime, and a logically deleted record is recorded by a separate mechanism, not by removing the row. Forbidding in-place mutation is what preserves the complete, tamper-evident history.

Q: Why is insert-only loading inherently auditable?
A: Because history is never destroyed — every version of every fact persists with its load datetime and record source, so the warehouse itself is the audit log. An auditor can reconstruct exactly what was known at any point in time and see who asserted it, without relying on a separate change-data-capture trail. There is no overwrite that could hide a prior value.

Q: How do deterministic hash keys enable parallel loading?
A: Because a hub, link, or satellite key can be computed purely from the source row's business keys, no row needs to look up a previously assigned surrogate from a central sequence. That removes the serialization point, so hubs, links, and satellites — and even multiple sources — can load concurrently and in any order. Re-running a load just re-inserts the same deterministic keys, which can be deduplicated.

Q: Why can hubs, links, and satellites be loaded independently and in any order?
A: Because each derives its keys from business keys via hashing, not from foreign-key lookups into already-loaded tables. A satellite can be loaded before, after, or alongside its hub since both compute the same parent hash key from the same business key. This load-order independence is central to Data Vault's parallel, restartable loading model.

Q: How is a logical delete represented in an insert-only Data Vault?
A: It is not represented by deleting a row but by inserting a status record — commonly a record-tracking satellite or a "deleted" flag in a status satellite — so the deletion itself becomes a timestamped, auditable fact. The original satellite rows remain intact. This preserves the principle that the warehouse never loses history, even for removed source records.

Q: In a fund-administration context, how would insert-only loading appear to an auditor?
A: Each NAV strike, holding update, or static-data change appears as an additional satellite row with its load datetime and record source, never as an overwrite of the previous value. The auditor sees an immutable timeline: the original value, every subsequent restatement, and the source of each. This is exactly the explainability regulated administration demands.

Q: What is a record-tracking satellite?
A: A record-tracking satellite (also driving-key or status satellite variant) tracks the presence or appearance of a business key or relationship in each source load, rather than its descriptive attributes. It is used to detect logical deletes and to know which source last reported a key, supporting status tracking without ever updating prior rows. It is an insert-only mechanism like all satellites.

Q: How does Data Vault let you add a second source system without re-modeling?
A: You add new satellites for the new source's attributes, map its business keys onto the existing hubs, and add links for any new relationships — all without touching existing hubs, links, or satellites. Because integration happens on the business key, the second transfer agency's fund records flow into the same fund hub. Existing structures and their history are untouched, so nothing downstream breaks.

Q: A second transfer agency reuses an ISIN for a different internal account. How does the hub design cope?
A: If the ISIN truly identifies the same fund, both feeds map to one hub row and their differing internal codes live in separate per-source satellites or a separate hub keyed on the internal code. If the ISIN has been reused for a genuinely different instrument, the ISIN alone is an inadequate business key and you must compose it with another stable attribute. The decision turns on whether the key denotes the same real-world entity.

Q: Why can a reused or recycled ISIN be a problem as a hub business key?
A: Because ISINs can be reassigned after an instrument matures or is delisted, so over time one ISIN string may denote two different instruments. Keyed naively, both would collapse into one hub, conflating two funds' histories. The fix is to defend the key choice — for example by composing ISIN with an effective-date or issuer attribute, or by using a more stable internal identifier as the business key.

Q: How would you defend a hub key choice for messy real-world fund identifiers?
A: You document why the chosen key uniquely and stably identifies the business concept, what failure modes (reuse, nulls, formatting variants) exist, and how you mitigated them (composite keys, standardisation, source-priority). For funds you might justify a composite of ISIN and a stable internal fund code precisely because ISINs can be reused. The defence is part of the modeling deliverable, not optional.

Q: When two sources disagree on a fund's attribute, where does that conflict live in a raw vault?
A: It lives as two satellite rows — one per source satellite — each faithfully recording what its source asserted, with no attempt to reconcile in the raw layer. Reconciliation (which value wins) is a soft business rule that belongs in the business vault. This keeps the raw vault an honest record of disagreement and makes the resolution logic explicit and auditable.

Q: Why does Data Vault keep multi-source conflicts unresolved in the raw vault?
A: Because resolving them requires a business rule that could change or be wrong, and the raw vault must remain a defensible, unaltered copy of each source. Storing both assertions preserves the ability to re-resolve later and to prove what each source said. Premature reconciliation in the raw layer would destroy that auditability and lock in a possibly mistaken choice.

Q: What is a PIT (point-in-time) table and what problem does it solve?
A: A PIT table is a query-assistance structure that, for each parent key and each snapshot date, pre-records which satellite row was current at that time. It solves the performance problem of "as-of" queries over a normalized vault, where reconstructing the current row per satellite otherwise requires expensive per-satellite max-load-datetime subqueries. The PIT turns those into simple equality joins.

Q: Why are point-in-time queries slow against a raw vault without a PIT table?
A: Because finding the version of each attribute that was current at a given date means, for every satellite, filtering to rows at or before that date and picking the latest — a correlated max-per-key operation across many satellites, then joining them. With several satellites per hub this becomes many expensive subqueries. A PIT precomputes those choices so the query just joins on stored keys and dates.

Q: What does a PIT table physically contain?
A: For each parent hash key and each snapshot date in a defined grid, the PIT stores the load datetime (and often the hash key) of the satellite row that was current as of that snapshot, for each satellite it spans. Querying then becomes an equi-join from the PIT to each satellite on parent key plus that stored load datetime. It holds no business attributes itself — only the pointers.

Q: What is a bridge table in Data Vault and how does it differ from a PIT table?
A: A bridge table is a query-assistance structure that pre-joins a hub with one or more links (and often related hubs) to flatten frequently traversed relationships and improve join performance. Whereas a PIT optimises as-of access across satellites of one parent, a bridge optimises navigation across multiple hubs and links. Both are derivable business-vault artifacts, rebuildable from the raw vault.

Q: Are PIT and bridge tables part of the raw vault or the business vault?
A: They are business-vault (query-assistance) structures, not raw vault. They contain no new source facts — only precomputed pointers and joins derived from raw hubs, links, and satellites — so they can be dropped and rebuilt at any time. Keeping them out of the raw vault preserves the raw layer as the pure system of record.

Q: How does a PIT table restore performance for "what was the NAV as of month-end" queries?
A: The PIT precomputes, for each fund key and each month-end snapshot, exactly which satellite row was current then, so the as-of query is a direct join rather than a per-satellite max-datetime scan. Reporting NAV history across many month-ends over a large satellite becomes fast and predictable. This is precisely the kind of regulated, repeated as-of reporting where PITs pay off.

Q: How often are PIT and bridge tables typically refreshed?
A: They are rebuilt or extended on a regular cadence aligned to the snapshot grid they serve — often nightly, with the snapshot date range extended each run. Because they are fully derivable from the raw vault, refreshing means recomputing the pointers, not re-ingesting data. The cadence is a performance/storage trade-off, not a correctness one.

Q: When is Data Vault overkill?
A: When you have a single, small, stable source with no real integration need and no strong audit-history requirement, the hub/link/satellite ceremony adds tables and load logic for no payoff. A simple dimensional model or even a flat table loads faster and is easier to query. Data Vault earns its keep with multiple evolving sources and hard auditability needs — exactly fund administration's profile, but not every small mart's.

Q: What is the cost of misapplying Data Vault to a trivial source?
A: You multiply table count and join depth, write and maintain more load code, and pay query-time assembly cost — all to model context that never integrates or needs deep history. The Why note for this lesson puts it bluntly: you build ceremony with no payoff. The skill is recognising when the auditability and integration benefits do not apply.

Q: What is a lakehouse, and what does "Vault-on-lakehouse" mean?
A: A lakehouse combines data-lake storage (columnar files on object storage, often with a table format like Delta or Iceberg) with warehouse-style transactional tables and SQL. Vault-on-lakehouse means implementing the hub/link/satellite Data Vault on that platform — typically with dbt and the AutomateDV package generating and testing the loads — rather than on a classic relational EDW. It brings Data Vault's auditability to modern, scalable, file-based storage.

Q: What is AutomateDV?
A: AutomateDV is a dbt package (by Datavault-UK, available on dbt Hub as `Datavault-UK/automate_dv`) that generates and executes the SQL to load a Data Vault 2.0 warehouse from metadata. You declare staging hashes and source models, and its macros emit insert-only hub, link, and satellite loads plus PIT and bridge logic. It removes the repetitive, error-prone hand-coding of vault loads.

Q: What was AutomateDV previously called?
A: It was previously called dbtvault. The rename to AutomateDV better reflects the tool's purpose, and a key practical consequence is that macros are now namespaced as `automate_dv.*` (e.g. `automate_dv.hub`) instead of `dbtvault.*`. Old tutorials and code using `dbtvault.hub(...)` need updating to the new namespace.

Q: Name the core AutomateDV table macros.
A: `automate_dv.hub`, `automate_dv.link`, `automate_dv.t_link` (transactional/non-historized link), `automate_dv.sat`, `automate_dv.ma_sat` (multi-active satellite), `automate_dv.eff_sat` (effectivity satellite), `automate_dv.xts` (extended tracking satellite), `automate_dv.pit`, and `automate_dv.bridge`. You call them inside dbt models, passing the staged source model and the relevant key and column metadata. They generate the insert-only loading SQL for each structure.

Q: What do the AutomateDV staging macros `hash_columns` and `derive_columns` do?
A: In a staging model, `hash_columns` generates the hash key and hashdiff columns from your business keys and attributes using the configured algorithm and standardisation, and `derive_columns` adds constant or computed columns such as record source and load datetime. The staging layer thus prepares every downstream hub/link/sat to read pre-hashed, standardised rows. Getting the staging hashes right is what makes the vault loads deterministic.

Q: Why pair dbt with AutomateDV rather than hand-writing vault SQL?
A: Because vault loads are highly repetitive yet error-prone — the same insert-only, hashdiff-compare pattern across dozens of satellites — and a single inconsistency in hashing or null handling breaks integration. AutomateDV's macros encode the patterns once, and dbt provides version control, testing, and lineage on top. You get generated, tested raw-vault loads instead of bespoke SQL per table.

Q: How does dbt's testing support the auditability goals of a Data Vault?
A: dbt lets you assert constraints — uniqueness of hub keys, not-null on business keys, relationships between links and hubs, accepted record-source values — as code that runs every build. Combined with AutomateDV's deterministic generation, this gives continuous, documented evidence that the vault's integrity rules hold. For a regulated administrator, automated tests are part of the controls story you show an auditor.

Q: What is a multi-active satellite (`ma_sat`) and when is it needed?
A: A multi-active satellite stores multiple concurrently-active rows for the same parent at the same load datetime — for example several contact methods or several share-class distribution policies effective simultaneously. It needs an additional sub-sequence or business-key-within-parent to make each active row unique. A plain satellite assumes one current row per parent, which would be wrong for genuinely multi-valued context.

Q: What is an effectivity satellite (`eff_sat`)?
A: An effectivity satellite tracks the start and end of the effective period of a relationship recorded in a link — for example when a particular administrator was effectively the servicer of a share class. It records driving keys and effective-from/-to in an insert-only way, so you can answer "which relationship was effective when" without updating prior rows. It is how Data Vault models relationship validity over time.

Q: What is a driving key in an effectivity satellite?
A: The driving key is the subset of the link's hub keys that determines the one-side of a relationship being tracked, used to detect when that relationship changes — for example the share-class key driving which administrator is currently effective. It tells the effectivity logic which existing relationship to "close" when a new one appears. Choosing the right driving key is essential for correct effectivity tracking.

Q: What is a non-historized (transactional) link, `t_link`?
A: A non-historized or transactional link stores immutable transaction or event records — such as an individual subscription or redemption order — directly in the link, including their payload, because such events never change after they occur. Since the record is immutable there is no associated satellite history to track. It models facts that are write-once by nature.

Q: How would you model fund, share class, and administrator as a raw vault?
A: Create a hub each for fund, share class, and administrator on their business keys (e.g. ISIN-based fund and share-class keys, LEI for administrator), a link connecting them to express the servicing relationship, and per-source satellites carrying each entity's descriptive context. Load all of it insert-only with deterministic hash keys via AutomateDV. A second TA source then adds satellites and link rows, not new structures.

Q: Why might an administrator hub be keyed on LEI rather than a name?
A: Because the LEI is a globally unique, stable, regulator-maintained identifier for a legal entity, whereas names vary in spelling, abbreviation, and language and change on rebranding. Keying on LEI ensures the same administrator from two feeds integrates into one hub, and that mergers/renames do not fragment its history. It is the textbook example of choosing a source-independent business key.

Q: How does loading a late-arriving correction work in insert-only Data Vault?
A: The correction is inserted as a new satellite row for the affected key, with its own load datetime (and often a separately tracked applied/effective date), leaving the erroneous prior row in place. The current view then resolves to the corrected row by latest load datetime, while the full history — original, correction, and their sources — remains queryable. Nothing is overwritten.

Q: How do you show the full audit trail of a corrected NAV from the satellites?
A: You query all satellite rows for that fund/share-class key ordered by load datetime: the original NAV row, the later correction row, each with its record source, are all present. That sequence is the audit trail, reconstructable directly from the vault without a separate change log. It demonstrates the Done-when criterion of showing a late correction as a new row, not an overwrite.

Q: Why is a late correction modeled as a new satellite row rather than an overwrite?
A: Because overwriting would destroy the record that the original (wrong) value was once asserted and reported, which is exactly what regulated administration must preserve. A new row keeps both the original and the correction, with their timestamps and sources, so you can prove what was known and reported at each moment. This is the auditability-by-construction the pattern promises.

Q: How would you distinguish the time an event happened from the time you loaded it in a satellite?
A: The load datetime is the warehouse's processing time (when the row was inserted) and is part of the satellite key, while a separate applied/effective date column captures the business event or as-of time. Keeping both lets you answer bitemporal questions — what the value was for a date, and when the warehouse learned it — which matters for late corrections. Conflating them loses the ability to explain corrections.

Q: What is the 1.4.7 pattern referenced in this lesson and how does it relate to Data Vault?
A: It refers to the Phase-1 modeling pattern (lesson 1.4.7) that this Data Vault lesson builds on and reuses — the canonical fund/share-class/administrator modeling you established earlier. Data Vault here is the EDW expression of that same domain pattern, so your hubs, links, and satellites should align to the business keys and relationships already defined. The cross-reference signals continuity, not a new domain.

Q: Why is Data Vault specifically favored in financial services and Luxembourg fund shops?
A: Because it delivers auditability by construction and non-destructive multi-source integration — exactly the controls a regulated fund-administration and asset-management context needs for NAV, holdings, and static data lineage. Luxembourg fund shops ask for it by name in job specs, so it is both a technical and a hiring reality. Its insert-only history makes every change explainable to an auditor.

Q: What does "auditability by construction" mean for Data Vault?
A: It means the audit trail is a structural property of the model, not an add-on: because loading is insert-only and every row carries load datetime and record source, the warehouse inherently preserves who said what and when. You do not bolt on a separate change-tracking system; querying the satellites is the audit. This is why the pattern resonates in regulated finance.

Q: First thing to check when AutomateDV produces duplicate hub rows for the same business key.
A: Check the staging hash configuration: a difference in casing, the concatenator, null placeholder, or column order between loads will produce different hash keys for the same business key, so it appears as a new hub row. Confirm `hash_content_casing`, `concat_string`, `null_placeholder_string`, and the declared column order are identical everywhere. Inconsistent standardisation is the classic cause of "duplicate" hubs.

Q: Source A sends ISIN as lowercase and source B as uppercase. What breaks and how do you fix it?
A: With `hash_content_casing` not normalising, the two ISINs hash to different hub keys, so the same fund splits into two hubs and integration fails. The fix is to ensure uppercasing is enabled (the AutomateDV default) or to standardise ISIN case in staging before hashing. This is why standardising the business key before hashing is mandatory.

Q: Your satellite is inserting a new row on every load even when nothing changed. What is the likely cause?
A: The hashdiff is being computed inconsistently — for example over a different column set, different order, different null handling, or including a volatile column like a load timestamp — so it never matches the stored hashdiff and every load looks like a change. Verify the hashdiff covers exactly the descriptive attributes, in a fixed order, with the standard null placeholder. A non-deterministic hashdiff defeats change detection.

Q: Why must a load datetime or processing timestamp never be part of a satellite's hashdiff?
A: Because the hashdiff exists to detect changes in business content, and a per-run timestamp changes every load, so including it would make every record look changed and insert a redundant row each time. The hashdiff must cover only the descriptive attributes whose change you care about. Excluding system/processing columns from the hashdiff is a fundamental rule.

Q: An auditor asks you to prove no historical NAV was ever altered. How does Data Vault answer this?
A: You show that loading is insert-only with no UPDATE or DELETE on satellites, so the original NAV row, identified by parent key and load datetime, still exists alongside any corrections. The presence of all versions, with record source, is itself the proof of non-alteration. Combined with dbt tests asserting key uniqueness and not-null, you have a controls-backed, structural argument.

Q: Where does NAV calculation logic belong relative to the vault?
A: NAV computation is a business rule, so it belongs in the business vault or downstream marts, never in the raw vault load. The raw vault stores the inputs (holdings, prices, fees) faithfully per source; the computed NAV is a derived, rebuildable result. Keeping the calculation out of the raw layer preserves the raw vault as an unaltered system of record and keeps the calculation auditable separately.

Q: How would EMT/EPT template attributes map into a Data Vault?
A: The funds and share classes the templates describe map to existing hubs; the template-specific attributes (target market, costs blocks, PRIIPs inputs) land in satellites, ideally per-source and per-template-version, off those hubs. Because templates version frequently, satellites let you keep each version's values as new rows without disturbing prior data. This is multi-source integration applied to FinDatEx data.

Q: A new EMT template version adds fields. How does Data Vault absorb it without re-modeling?
A: You add a new satellite (or extend an existing one) on the relevant hub to hold the new fields, loaded insert-only, while existing structures and history stay untouched. The hubs and links for fund and share class do not change because the business keys did not change. This is precisely the "absorb a new source/version without rewriting the warehouse" property.

Q: How do you represent the version of a FinDatEx template that supplied a satellite row?
A: Capture the template version in the record source (or a dedicated version attribute) so each satellite row is attributable to a specific template release. That lets you trace which EMT/EPT version asserted each value and answer questions about partial or superseded files. It is the same record-source-driven traceability the pattern uses everywhere.

Q: How would individual subscription and redemption orders be modeled in the vault?
A: As records in a non-historized transactional link (`t_link`) connecting the investor, share-class, and fund hubs, because each order is an immutable event with its own payload that never changes after it occurs. There is no satellite history needed for the order itself. Order status updates, by contrast, are separate facts and can be tracked via status satellites or further links.

Q: How do SWIFT-message-derived facts fit the raw vault?
A: Business keys carried in the messages (ISIN, LEI, account identifiers) map to hubs, the relationships they describe map to links, and the message field values become satellite or transactional-link payload, with record source identifying the SWIFT message type. The raw vault stores exactly what the message asserted, faithfully per source. Any normalisation of message codes to canonical values is a business-vault rule.

Q: Why might two transfer agencies' fund feeds need separate satellites even though they share a fund hub?
A: Because each source can assert different or conflicting attribute values and may load on a different schedule, and the raw vault must keep each source's assertions intact and independently loadable. Separate per-source satellites preserve who said what and avoid one feed overwriting another's context. Reconciliation across them is a later business-vault concern.

Q: What is the consequence of choosing too narrow a business key for a fund hub?
A: If the key fails to uniquely identify the fund — e.g. ISIN alone where ISINs are reused or shared across share-class structures — distinct funds collapse into one hub and their histories conflate, corrupting every downstream query and report. The remedy is a composite or more stable key, justified explicitly. Too-narrow keys are a leading cause of silent integration errors.

Q: What is the consequence of choosing too broad (over-composite) a business key for a hub?
A: If the key includes attributes that are not actually part of the entity's identity, the same real-world fund can appear as multiple hub rows whenever those extra attributes differ, defeating integration. You end up failing to merge feeds that should merge. The key must be exactly the minimal set that uniquely and stably identifies the business concept — no more, no less.

Q: Why are dbt models the natural home for Vault-on-lakehouse loads?
A: Because dbt expresses each hub/link/sat as a SELECT-based model with dependencies, version control, tests, and documentation, and AutomateDV's macros generate the insert-only SQL inside those models. The lakehouse's transactional table format gives the ACID semantics the inserts need. Together they bring engineering discipline and lineage to a pattern that is otherwise repetitive hand-written SQL.

Q: How does a PIT table interact with multiple satellites on the same hub?
A: For each snapshot date and parent key, the PIT records the current load datetime for each of the hub's satellites, so an as-of query joins the PIT to every satellite on parent key plus that satellite's stored load datetime. This collapses a multi-satellite, per-satellite-max reconstruction into a set of simple equi-joins. The PIT must span exactly the satellites the report needs.

Q: What is the trade-off of building many PIT and bridge tables?
A: They speed queries but add storage and a maintenance/refresh burden, and they must be rebuilt as new snapshots and data arrive. Because they are derived, they never threaten correctness, but over-building them wastes compute and storage on access paths nobody uses. You build them where measured query pain justifies them, not pre-emptively for every report.

Q: Why is "raw vault is never cleansed of business meaning" a hard rule?
A: Because the raw vault's value is being a faithful, defensible copy of the sources; applying business interpretation there makes it impossible to prove the warehouse matches the source files and bakes in possibly-wrong logic irreversibly. Only hard technical fixes (type alignment, hashing, trimming) are allowed in the raw load. All interpretation moves to the business vault, where it is explicit and rebuildable.

Q: How does Data Vault's structure help when a regulator requires reconstructing the warehouse state as of a past date?
A: Because every satellite row carries a load datetime and nothing is overwritten, you can filter every satellite to rows at or before the target date and select the latest per key to reconstruct exactly what was known then. PIT tables make this fast for common snapshot dates. The model is inherently bitemporal-friendly, which is precisely what point-in-time regulatory reconstruction needs.

Q: First thing to check when an as-of query over the vault is unexpectedly slow.
A: Check whether a PIT table exists and covers the snapshot dates and satellites the query uses; without one, the query is doing correlated max-load-datetime scans per satellite. If a PIT exists, confirm the query actually joins through it rather than the raw satellites directly. Missing or unused query-assistance structures are the usual cause of slow as-of queries.

Q: Why does Data Vault separate raw integration from computed logic, in one sentence?
A: So the raw layer stays a provable, source-faithful, immutable record of what every system asserted, while all interpretation lives in a rebuildable business vault — preserving both auditability and the freedom to change business rules without re-ingesting data.

Q: What is an extended tracking satellite (`xts`) in AutomateDV and what is it for?
A: An `xts` is a satellite variant that records, per load, which records appeared in which source feed — an audit of presence across loads — feeding logical-delete and record-tracking logic. It carries tracking metadata rather than business attributes and is loaded insert-only like every satellite. It supports answering "which source last reported this key, and when" without ever mutating prior rows.

Q: How do hash keys behave when a business key value is genuinely null in the source?
A: The null is replaced with the configured null placeholder (AutomateDV default `^^`) before hashing, so a missing key hashes deterministically rather than producing an inconsistent or platform-dependent result. However, a hub keyed on a frequently-null business key is a modeling smell — the key may not actually identify the entity. Consistent null handling guarantees reproducibility, but it does not excuse a poor key choice.

Q: When integrating two TA feeds with different code lists for the same attribute, where does the harmonisation belong?
A: The raw vault stores each feed's native codes verbatim in its own per-source satellite, and the mapping to a single canonical code list is a soft business rule that belongs in the business vault as a computed satellite or mapping structure. This keeps the originals provable and the harmonisation explicit and rebuildable. Doing the mapping during the raw load would destroy the ability to show what each source actually sent.

Q: Why is dbt lineage valuable when defending a Data Vault to an auditor?
A: dbt builds a directed dependency graph from sources through staging, raw vault, business vault, and marts, so you can show exactly how any reported value was derived and which raw satellites and rules contributed. Combined with record source on every row and dbt tests on every build, this turns "trust us" into a documented, reproducible lineage. For regulated fund administration, that end-to-end traceability is a core part of the controls evidence.

Q: Summarise the Done-when criteria for this lesson.
A: You can defend hub business-key choices for messy real-world identifiers such as reused ISINs; you can explain insert-only loading and the audit trail it produces to an auditor; and you can show a late correction as a new satellite row rather than an overwrite. Together these prove you can both model and operate a raw vault. They are the capability bar this Data Vault lesson is aiming for.

Q: In AutomateDV, which `src_*` arguments does the `automate_dv.sat` macro require versus the `automate_dv.hub` macro?
A: The `sat` macro takes `src_pk` (parent hash key), `src_hashdiff`, `src_payload` (the descriptive columns), `src_ldts`, and `src_source`, whereas the `hub` macro takes `src_pk`, `src_nk` (natural/business key), `src_ldts`, and `src_source` and has no hashdiff or payload. The difference reflects their roles: a hub carries only identity, a satellite carries context plus a hashdiff for change detection. Both also take `source_model` naming the staging model to read from.

Q: What does the `src_nk` argument mean in `automate_dv.hub` and what happens if you pass the wrong column?
A: `src_nk` is the natural (business) key column or list of columns that defines the hub's identity, such as the ISIN for a fund hub. If you pass a source surrogate or a non-identifying column, the hub will be keyed on the wrong concept and integration across feeds will fail, since the same fund could land under different keys. It must be the same standardised business key that was hashed into `src_pk` upstream in staging.

Q: In `automate_dv.link`, what is the difference between `src_pk` and `src_fk`?
A: `src_pk` is the link's own hash key (the hash of all participating business keys together), while `src_fk` is the list of the participating hubs' hash keys that the link references. The link row stores both: its identity (`src_pk`) and pointers to each connected hub (`src_fk`). Getting `src_fk` wrong or in the wrong order means the link points at the wrong hubs or fails referential tests.

Q: Why does the `automate_dv.t_link` macro accept a `src_payload` while a historized `link` does not?
A: A transactional (non-historized) link records an immutable event such as a subscription order, so its measurable attributes live directly on the link via `src_payload` because there is no satellite tracking change over time. A historized `link` carries only keys and pushes descriptive context to satellites, since the relationship can gain attributes that change. The payload-on-link design fits write-once events that never need history.

Q: What does the global `concat_string` variable default to and what breaks if two models disagree on it?
A: It defaults to `||`, the separator concatenated between columns before hashing a multi-column key or hashdiff. If two models or two systems use different separators, the same business key produces different hash strings and therefore different hash keys, splitting one entity into two hubs. It must be standardised package-wide in `dbt_project.yml` so every hash is byte-identical.

Q: What is the purpose of AutomateDV's `null_key_required` and `null_key_optional` variables (defaults `-1` and `-2`)?
A: They are ghost-record / null-key sentinel values used when a foreign key into a hub is missing: a required-but-missing key resolves to the `-1` ghost and an optional-but-missing key to the `-2` ghost, so links never carry a true `NULL` foreign key. This keeps joins and referential-integrity tests well-defined even when a relationship's counterpart is absent. You enable the ghost rows via `enable_ghost_records`.

Q: What is a ghost record in AutomateDV and why would you enable `enable_ghost_records`?
A: A ghost record is a single placeholder row inserted into each hub and satellite (with the `system_record_value` record source, default `AUTOMATE_DV_SYSTEM`) representing "unknown", so that outer-join semantics and PIT joins always find a row even when no real data exists. Enabling `enable_ghost_records` makes PIT and bridge logic use inner joins safely without dropping parents that lack a satellite row. Without ghosts, a missing satellite row would silently drop a fund from an as-of report.

Q: A PIT table built with ghost records returns a row for a fund that has no satellite data for that date — is that correct?
A: Yes — that is the designed behaviour: the PIT resolves the missing satellite to the ghost record so the as-of join still produces a row, with the ghost's placeholder values rather than dropping the fund. This keeps the snapshot grid complete and prevents an inner join from silently excluding parents. You then filter or display the ghost as "no data as of this date" downstream.

Q: How would you express a dbt `unique` and `not_null` test on a fund hub to back your audit controls?
A: In the model's YAML you add `tests: [unique, not_null]` under the `columns:` entry for the hub's hash key (`FUND_HK`) and `not_null` on the business key column, so every `dbt build` re-asserts that hub keys are unique and present. A failing test fails the build, giving you continuous, dated evidence the integrity rule held. For regulated administration this is part of the documented control set you show an auditor.

Q: Which dbt test would you use to prove every link row points at a real hub row, and why does it matter for a vault?
A: The built-in `relationships` test (or AutomateDV's referential-integrity tests) asserting that each foreign hash key in the link exists in the target hub, configured in the link model's YAML against `FUND_HK`, `SHARE_CLASS_HK`, and `ADMIN_HK`. It matters because a dangling link key would mean a relationship references a fund or administrator the warehouse never recorded, an integrity break an auditor would flag. Running it every build turns referential integrity into tested evidence.

Q: You run `dbt build` and the satellite test for `not_null` on the hashdiff fails. What is the likely modeling mistake upstream?
A: The staging model probably did not generate the `HASHDIFF` column, or the `hash_columns` macro was given an empty payload list, so the satellite reads a null hashdiff and cannot detect change. Check the staging model's `hash_columns` config includes a hashdiff over the descriptive columns and that the `sat` macro's `src_hashdiff` names that exact column. A null hashdiff defeats change detection and is a hard error, not a warning.

Q: How do you incrementally load only new source rows in an AutomateDV satellite rather than re-scanning all history each run?
A: AutomateDV satellite models are materialised as dbt `incremental` models, and the macro emits SQL that compares each candidate row's hashdiff against the latest stored hashdiff per key, inserting only genuine changes. You set `materialized='incremental'` in the model config, and dbt's incremental logic plus the hashdiff filter keep each run cheap. A full-refresh run rebuilds from scratch when you pass `dbt build --full-refresh`.

Q: What does `dbt build --full-refresh` do to a Data Vault and when is it safe versus dangerous?
A: It drops and rebuilds incremental models from their full source, recomputing every hub, link, and satellite row from the staged source data. It is safe when the source still holds complete history and the deterministic hashes reproduce identical keys; it is dangerous when the source only retains a recent window, because you would lose historical satellite rows that the source no longer provides. In regulated settings you must confirm source completeness before a full refresh of a system of record.

Q: First thing to check when an incremental satellite load inserts zero rows even though the source changed.
A: Check that the `is_incremental()` filter is actually finding changes — most often the hashdiff is being computed identically to the stored one because the changed column was excluded from the payload, or the staging hashdiff list is stale. Confirm the changed attribute is in `hash_columns`' hashdiff payload and that the comparison joins on the correct parent key. A payload that omits the volatile column silently swallows real changes.

Q: How does the AutomateDV `stage` macro relate to `hash_columns` and `derive_columns`?
A: The `stage` macro is a convenience wrapper that, in one call, applies `derive_columns` (constants like record source and load datetime), `hash_columns` (hash keys and hashdiffs), and `ranked_columns` if needed, producing a fully prepared staging model. You give it metadata dictionaries for each, and it emits the staged SELECT that hubs, links, and satellites then read. It centralises the standardisation that makes all downstream hashes deterministic.

Q: What does the `ranked_columns` configuration in AutomateDV staging produce and when do you need it?
A: `ranked_columns` adds window-function rank columns (e.g. row number partitioned by key, ordered by load datetime) used to deduplicate or pick the latest record per business key within a batch. You need it when a single source batch contains multiple versions of the same key and you must load them in order or drop duplicates. Without ranking, a batch with intra-load duplicates could insert redundant or out-of-order satellite rows.

Q: A source file contains two rows for the same ISIN in one batch with different NAVs. How does the vault decide which becomes the current satellite row?
A: Both rows can be inserted as separate satellite versions if their load/applied datetimes differ, and the current row is whichever has the latest load datetime; if they share a load datetime, you must rank within the batch (via `ranked_columns`) on an effective/sequence column to order them deterministically. Leaving intra-batch duplicates unranked risks a non-deterministic "current" value. The applied date, not just load date, drives which NAV is genuinely latest.

Q: Distinguish AutomateDV's `eff_sat` from a status/record-tracking satellite for modeling which administrator services a share class.
A: An `eff_sat` (effectivity satellite) tracks the effective-from/-to validity of the link relationship using a driving key, so it answers "which administrator was effective on date X" by opening and closing relationship periods. A record-tracking or status satellite instead records presence/appearance of the key per source load, used mainly for logical-delete detection. You use `eff_sat` for relationship validity over time and tracking satellites for source-presence auditing.

Q: How does the driving key in an `eff_sat` cause the previous relationship to be "closed" when a new one arrives?
A: The effectivity logic groups link rows by the driving key (e.g. the share-class hash key) and, when a new link with a different other-side key appears for that driving key, it inserts a row that ends the prior relationship's effective period and starts the new one. No prior row is updated; the closure is itself a new inserted record. Pick the wrong driving key and the wrong side gets "closed", corrupting the effective timeline.

Q: Why is choosing LEI over a SWIFT BIC as an administrator hub business key sometimes preferable, and when is BIC fine?
A: A LEI uniquely identifies the legal entity itself and is regulator-maintained, while a BIC identifies a financial institution's connection/branch and one legal entity can hold several BICs, so keying on BIC could fragment one administrator into multiple hubs. LEI is the safer identity key for the legal entity. A BIC is fine when the hub genuinely models the messaging endpoint (a branch/connection) rather than the legal entity.

Q: An ISIN in a source feed is 11 characters instead of 12 — what should the raw vault load do, and where does any fix belong?
A: The raw vault should still load it faithfully (after only hard technical handling like trimming), recording the malformed value with its record source so the defect is preserved and auditable. Validating or correcting the check digit is a business rule that belongs in the business vault or a data-quality layer, not the raw load. Silently "fixing" it in the raw vault would destroy proof of what the source actually sent.

Q: How would you detect, without overwriting, that a share class was logically deleted from one TA feed but still present in another?
A: A record-tracking or extended-tracking satellite (`xts`) records which source reported each key in each load, so the missing-from-feed-A condition appears as the key no longer arriving in A's batch while still arriving in B's. You insert a tracking row capturing that presence pattern rather than deleting anything. The deletion thus becomes a timestamped, source-attributed fact you can query, exactly as insert-only requires.

Q: How do you model an investor in the fund/share-class/administrator vault when investor identity is sensitive (GDPR)?
A: Create an investor hub on a stable business key (e.g. a tokenised investor ID or LEI for institutional investors), keep personally identifiable attributes in a dedicated satellite that can be access-controlled or held in a separate, restricted schema, and link investor to share class via a transaction link for orders. Separating PII into its own satellite lets you apply column- or table-level controls without disturbing the hub or order links. The insert-only history still applies, but retention/erasure rules are handled as governed exceptions.

Q: GDPR erasure ("right to be forgotten") conflicts with insert-only immutability — how is this reconciled in a vault?
A: You isolate erasable PII into a dedicated satellite so that a legally mandated erasure can crypto-shred or null that satellite's content while the hubs, links, and non-PII satellites that carry the audit trail remain intact. The erasure itself is logged as a governed, timestamped action. This keeps the financial audit history whole while satisfying a narrow, lawful deletion, rather than treating it as an ordinary insert-only change.

Q: How does the record source help when the same NAV value arrives via both a SWIFT `semt` statement and a direct TA file?
A: Each arrival lands in its own per-source satellite (or carries a distinct record-source value), so you can see both channels asserted the figure and reconcile or prefer one in the business vault. The record source distinguishes the SWIFT-derived assertion from the file-derived one even when the values agree. This preserves channel provenance, which matters when later only one channel turns out to have been correct.

Q: Why might you keep separate satellites for slow static data and fast NAV-related descriptive data on the same fund hub, in concrete load-cost terms?
A: Splitting by rate of change means a daily NAV-descriptor change inserts a row only into the small fast satellite, while the large static satellite (domicile, legal form, inception date) is untouched, so you do not re-store unchanged wide static columns daily. This cuts storage and load cost dramatically over time. A single combined satellite would write a full new row for the whole attribute set on every fast change.

Q: How would EET (European ESG Template) sustainability attributes be modeled into the vault?
A: The fund and share-class business keys map to the existing hubs, and the EET's SFDR-related fields (Article 6/8/9 classification, PAI indicators, taxonomy alignment) land in a dedicated per-source, per-template-version satellite off those hubs, loaded insert-only. Because the EET versions over time, each release's values become new satellite rows without altering hubs or links. This is the same FinDatEx absorb-a-new-template pattern applied to ESG data.

Q: A fund's SFDR classification changes from Article 8 to Article 9 mid-year. How does the vault record this without losing the prior state?
A: A new satellite row is inserted on the share-class (or fund) hub with the Article 9 value and a later load/applied datetime, while the Article 8 row remains, so the full classification history is preserved and queryable. The current view resolves to Article 9 by latest datetime. An auditor or regulator can see exactly when the reclassification was recorded and from which source, with no overwrite.

Q: Under T+1 settlement, intraday correction volume rises — what property of the vault makes high-frequency corrections safe?
A: Insert-only loading means every correction is just another timestamped satellite row, so a burst of intraday restatements never overwrites or races against prior values and never corrupts history. Deterministic hash keys let those corrections load in parallel and in any order without surrogate-lookup contention. The model scales to T+1's compressed, correction-heavy cycle precisely because it never mutates existing rows.

Q: Why does the EU AI Act push regulated fund shops toward Data Vault-style lineage for any model-fed data?
A: The AI Act's high-risk-system obligations require data governance, traceability, and record-keeping over training and input data, and a vault's record source plus insert-only history give per-row provenance and reproducible point-in-time inputs. You can prove exactly which source asserted each value feeding a model and reconstruct the data state as of any decision date. That auditable, bitemporal lineage is precisely the kind of evidence the AI Act's documentation duties demand.

Q: How would you reconstruct the exact input dataset that fed a risk model on a past date for an AI Act audit?
A: Use the satellites' load datetimes (and a PIT over them) to select, for each business key, the version current as of that date, yielding the precise as-of snapshot the model would have seen. Because nothing was overwritten and every row carries a record source, the reconstruction is exact and provably source-faithful. This bitemporal "what did we know on date X" query is a native strength of the vault.

Q: Compare storing currency as raw codes in the raw vault versus harmonising to ISO 4217 — where does each belong?
A: The raw vault stores each source's currency code exactly as sent (even non-standard or legacy codes) in its per-source satellite, preserving provenance; mapping those to canonical ISO 4217 codes is a soft business rule implemented as a business-vault computed satellite or mapping. Harmonising during the raw load would erase the ability to show what the source sent. The split keeps originals provable and the mapping rebuildable if the rule changes.

Q: Two satellites on the same fund hub disagree on the fund's base currency. Is this a raw vault error?
A: No — it is the raw vault correctly preserving that two sources asserted different values, each in its own per-source satellite, with no reconciliation attempted in the raw layer. Choosing the authoritative value is a business-vault rule. The disagreement being visible and source-attributed is a feature, not a bug, because it lets you re-resolve and prove what each feed claimed.

Q: How does a bridge table speed a "list all share classes serviced by administrator X with their funds" query?
A: The bridge pre-joins the administrator hub through the servicing link to the share-class and fund hubs, flattening the multi-hop navigation into one structure keyed for that traversal, so the query becomes a direct lookup instead of several runtime joins. It holds only derived join keys, no new facts, and is rebuildable from the raw vault. You build it when that navigation is frequent and the runtime joins are measurably costly.

Q: What is the difference in what a PIT versus a bridge precomputes, stated in terms of what they optimise?
A: A PIT precomputes, per snapshot date, which satellite version was current for a parent, optimising as-of access across multiple satellites of one hub; a bridge precomputes joins across multiple hubs and links, optimising navigation across relationships. PITs answer "what was the value on date X", bridges answer "how are these entities connected". Both are derived business-vault structures, never raw.

Q: Your PIT-backed month-end report is missing the latest month — first thing to check.
A: Check that the PIT's snapshot grid was extended to include the new month-end date; PITs only contain the snapshot dates they were built for, so a missing month usually means the refresh job did not append that date. Confirm the snapshot-date driver (often a date-spine or snapshot control table) includes the new period and that the PIT rebuild ran. The raw satellites have the data; the PIT simply has no pointer row for that date yet.

Q: Why can you safely `DROP` and rebuild every PIT and bridge table in production but not a raw satellite?
A: PITs and bridges contain only derived pointers and pre-joins computable from the raw vault, so dropping them loses no source facts and a rebuild reproduces them exactly. A raw satellite, by contrast, holds the only insert-only record of what a source asserted at each load datetime; dropping it destroys history the source may no longer be able to re-supply. The derivability boundary is what makes one disposable and the other sacrosanct.

Q: How do you justify choosing MD5 over SHA-256 for hash keys to a risk officer in a regulated shop?
A: You argue that MD5 is used here only as a non-cryptographic surrogate-key generator, not for security, and that the collision probability across a fund warehouse's key cardinality is statistically negligible, while MD5's 16-byte width halves storage and join cost versus SHA-256's 32 bytes. If governance policy bans MD5 outright (common where MD5 is blanket-prohibited), you switch via the `hash` variable to `SHA`. The decision is a documented trade-off, and AutomateDV makes it a one-line config change.

Q: What concretely changes in storage and joins if you switch the `hash` variable from `MD5` to `SHA`?
A: Every hash key and hashdiff widens from 16 bytes (MD5, 32 hex chars) to 32 bytes (SHA-256, 64 hex chars), inflating every hub, link, and satellite key column and every join predicate that compares them. Across a large vault this measurably increases storage and join CPU. The benefit is stronger collision resistance and policy compliance; you change it once in `dbt_project.yml` and rebuild.

Q: How would you stage a source so the same physical column feeds both a hub natural key and a satellite payload?
A: In the staging model you reference the column once for `hash_columns` to build the hub hash key from it as a natural key, and separately include it (or not) in the satellite's hashdiff/payload list depending on whether it is descriptive context. A column can be both an identity key and, if it also carries descriptive meaning, part of a satellite, but typically the business key is not repeated as payload. The staging metadata makes each role explicit so downstream macros read the right columns.

Q: Why must the load datetime be assigned once per batch and shared across hub, link, and satellite loads from that batch?
A: Using a single batch-level load datetime ties all rows derived from the same source extract to the same point in the warehouse timeline, so an as-of reconstruction and PIT alignment are coherent across hubs, links, and satellites. If each table stamped its own slightly different timestamp, history would be jagged and snapshots could mismatch across tables. AutomateDV typically derives this constant in staging via `derive_columns` so every downstream load shares it.

Q: A satellite's current-row query returns two rows for one key with the same load datetime. What went wrong and how do you fix it?
A: Two rows sharing the parent key and load datetime violate the satellite's primary key, meaning the batch had intra-load duplicates that were not deduplicated, or a multi-active case was modeled as a plain `sat`. Fix by adding `ranked_columns` deduplication in staging, or if the data is genuinely multi-valued, model it as a multi-active satellite (`ma_sat`) with a sub-key. A plain satellite assumes exactly one row per key per load datetime.

Q: When is `automate_dv.ma_sat` the correct choice over adding another hub, for multiple distribution policies per share class?
A: Use `ma_sat` when the several values are concurrently-active descriptive context of the same share class at the same load datetime (e.g. multiple simultaneous distribution policies), distinguished by a sub-key within the parent. Create a new hub instead only if each policy is itself an independently-identified business concept that other entities relate to. The test is whether the items are context of the parent or first-class entities in their own right.

Q: How do you prove to an auditor that no satellite row was ever physically deleted, beyond just asserting insert-only?
A: You show the loading code contains no `DELETE` or `UPDATE` against satellites (only `INSERT`), back it with dbt tests and the materialisation config, and demonstrate that row counts per key are monotonically non-decreasing across loads via the load-datetime history. The platform's transaction log or table-format history (e.g. Delta/Iceberg snapshots) gives an independent, immutable record of appends only. Together these turn the immutability claim into platform-level evidence.

Q: How does a lakehouse table format like Delta or Iceberg reinforce the vault's insert-only guarantee?
A: Delta and Iceberg keep an immutable, versioned commit log of table snapshots, so every append is a recorded transaction you can time-travel to, providing an independent audit of exactly which rows were added and when. This complements the vault's own load-datetime history with platform-enforced versioning. An auditor can verify at the storage layer that operations were appends, not in-place mutations.

Q: First thing to check when AutomateDV macros are not found and dbt errors with an "undefined macro" message.
A: Check the package is installed and namespaced correctly: AutomateDV macros are `automate_dv.*` (e.g. `automate_dv.hub`), and code copied from old `dbtvault.*` tutorials will fail after the rename. Confirm `packages.yml` references `Datavault-UK/automate_dv` and that you ran `dbt deps` to install it. A stale namespace or a missing `dbt deps` is the usual cause of undefined-macro errors.

Q: After renaming dbtvault to AutomateDV, a colleague's model still uses `dbtvault.sat(...)`. What is the minimal fix?
A: Update the macro namespace from `dbtvault.*` to `automate_dv.*` (so `dbtvault.sat` becomes `automate_dv.sat`) and ensure `packages.yml` points at `Datavault-UK/automate_dv` followed by `dbt deps`. The argument signatures are otherwise compatible, so only the namespace prefix changes in most cases. This is the single most common breakage when migrating older dbtvault code.

Q: How would you load a correction that arrives for a NAV whose applied date is in the past but is received today?
A: Insert a new satellite row with today's load datetime (when the warehouse learned it) and the original past applied/effective date in the effective-date column, leaving the erroneous row in place. The current value for that applied date then resolves to the correction, while the load datetime shows the correction was learned later. This bitemporal handling is exactly what late-arriving corrections require and why load datetime and applied date must be separate columns.

Q: Summarise, as an interview answer, why a Luxembourg fund administrator would mandate Data Vault for its NAV and static-data warehouse.
A: Because it gives auditability by construction — insert-only history with record source on every row means any NAV, holding, or static-data value is traceable to who asserted it and reconstructable as of any date — and non-destructive multi-source integration, so a second transfer agency or a new EMT/EPT/EET template version is absorbed by adding satellites, not re-modeling. Deterministic hash keys make the loads parallel and restartable, suiting T+1's correction-heavy cycle, and dbt plus AutomateDV provide tested, lineage-tracked generation. That combination is exactly the controls-and-agility profile a regulated fund shop, and the EU AI Act's data-governance duties, demand.

Q: A business-key column is named with a reserved SQL word and the staged hash query fails to compile — how does AutomateDV help?
A: AutomateDV quotes identifiers using the configurable `escape_char_left` and `escape_char_right` variables so reserved words or special characters in column names are wrapped correctly for the target platform's SQL dialect. You set these in `dbt_project.yml` to the platform's quoting characters (for example double quotes), and the generated staging and load SQL escapes the offending identifier. This avoids hand-editing generated SQL just because a source named a column `ORDER` or similar.
