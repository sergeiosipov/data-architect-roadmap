# Phase 0 · 0.11 Unit testing from zero (pytest) — 100 self-test questions

Q: What is a unit test?
A: A small automated check that runs one piece of code (usually a single function or behaviour) in isolation and asserts that its output or effect matches what you expect.

Q: What is `pytest`?
A: A popular third-party Python testing framework that discovers and runs your tests, lets you use plain `assert`, and gives rich failure output, fixtures, and parametrization.

Q: Why write unit tests at all for a NAV-loading script?
A: They catch regressions automatically — if a refactor breaks the NAV parsing, a test fails immediately instead of you discovering wrong fund prices in production.

Q: How do you run pytest in this course's environment?
A: With `uv run pytest`, which runs pytest inside the project's managed virtual environment without you having to activate it manually.

Q: Why `uv run pytest` instead of just `pytest`?
A: `uv run` guarantees pytest and your dependencies come from the project's locked environment, so the run is reproducible and bare `python`/`pip` is avoided.

Q: How do you add pytest as a development dependency with uv?
A: Run `uv add --dev pytest`, which installs it and records it under the dev dependency group in `pyproject.toml`.

Q: What does `uv run pytest` do when you give it no arguments?
A: It starts test discovery from the current directory (or configured `testpaths`) and runs every test it finds.

Q: What filename pattern does pytest discover by default?
A: Files named `test_*.py` or `*_test.py`.

Q: Would pytest collect a file called `nav_tests.py` by default?
A: No — it does not match `test_*.py` or `*_test.py`, so rename it to `test_nav.py` (or configure `python_files`) for it to be discovered.

Q: What function-name pattern does pytest discover by default?
A: Functions whose name starts with `test`, for example `test_parse_isin`.

Q: Would pytest run a function named `check_isin`?
A: No — it does not start with `test`, so pytest ignores it as a test even if it contains assertions.

Q: What class-name pattern does pytest collect tests from by default?
A: Classes named `Test*` (with no `__init__` method), and inside them it collects methods named `test*`.

Q: Why must a test class not define an `__init__` method?
A: pytest will skip collecting it and warn, because it instantiates test classes itself and an `__init__` interferes with that.

Q: Where does pytest start searching for tests?
A: From the `rootdir` / the arguments you pass, recursing into directories — commonly a `tests/` folder or alongside your modules.

Q: What is the simplest possible passing test?
A: A function named `test_*` containing an `assert` that is true, e.g. `def test_one(): assert 1 + 1 == 2`.

Q: What does the plain `assert` statement do in a pytest test?
A: It raises `AssertionError` if the expression is falsy; pytest catches that, marks the test failed, and prints a detailed comparison.

Q: Why can you use plain `assert` in pytest instead of methods like `assertEqual`?
A: pytest rewrites assert statements so it can show the actual and expected values, so you get rich output without the `unittest`-style assertion methods.

Q: What does `assert parse_isin(raw) == "LU0123456789"` check?
A: That calling `parse_isin(raw)` returns exactly the string `"LU0123456789"`; any other value fails the test.

Q: In the dot output, what does a single `.` mean?
A: One test passed.

Q: In the dot output, what does `F` mean?
A: One test failed (an assertion failed or an unexpected exception was raised in the test body).

Q: What does `E` mean in pytest's per-test status output?
A: An error occurred during setup or in a fixture (not a plain assertion failure in the test body).

Q: What does `s` mean in pytest output?
A: The test was skipped.

Q: What does the final line `3 passed in 0.04s` tell you?
A: Three tests passed and the whole run took 0.04 seconds — the summary of the session.

Q: What does `1 failed, 2 passed` mean for your exit status?
A: pytest exits non-zero because at least one test failed, which is what makes CI pipelines fail the build.

Q: What exit code does pytest return when all tests pass?
A: `0`, the conventional success code that lets CI treat the step as passed.

Q: How do you run only the tests in one file?
A: Pass the path, e.g. `uv run pytest tests/test_nav.py`.

Q: How do you run a single test function by name?
A: Use `::`, e.g. `uv run pytest tests/test_nav.py::test_parse_nav`.

Q: How do you run a single method inside a test class?
A: Use the full path with two `::`, e.g. `uv run pytest tests/test_nav.py::TestNav::test_parse`.

Q: What does the `-k` option do?
A: It runs only tests whose names match a keyword expression, e.g. `uv run pytest -k isin` runs every test with `isin` in its name.

Q: How would you run all tests except those mentioning `slow`?
A: Use a `-k` expression with `not`, e.g. `uv run pytest -k "not slow"`.

Q: What does the `-v` flag do?
A: It runs in verbose mode, printing one line per test with its full node id and PASSED/FAILED instead of single dots.

Q: What does `-x` do?
A: It stops the whole run at the first failure, useful when you want to fix one thing at a time.

Q: What does `--maxfail=2` do?
A: It stops the run after two failures instead of running everything.

Q: What does the `-s` flag do?
A: It disables pytest's output capture so `print` statements show up in the console during the run.

Q: Why does pytest normally hide your `print` output?
A: It captures stdout/stderr per test and only shows it for failing tests, keeping passing runs quiet.

Q: What does `--lf` (`--last-failed`) do?
A: It re-runs only the tests that failed in the previous run, speeding up the fix loop.

Q: What is `@pytest.mark.parametrize`?
A: A decorator that runs the same test function once per set of input/expected values, turning one function into many test cases.

Q: Write a parametrize that checks `is_valid_isin` for two cases.
A: `@pytest.mark.parametrize("isin,expected", [("LU0123456789", True), ("XX1", False)])` then `def test_isin(isin, expected): assert is_valid_isin(isin) == expected`.

Q: What are the two main arguments to `parametrize`?
A: A string naming the parameters (comma-separated) and a list of value tuples, one tuple per generated test case.

Q: How many tests does a parametrize with five value tuples produce?
A: Five — pytest generates one independent test per tuple.

Q: Why is parametrize better than a `for` loop inside one test?
A: Each case becomes its own test, so a failure in one input does not stop the others and the report names exactly which input failed.

Q: How does a single parametrized case appear in `-v` output?
A: With the parameter values in square brackets in the node id, e.g. `test_isin[LU0123456789-True]`.

Q: How do you parametrize over a single parameter (not a tuple)?
A: Use one name and a flat list, e.g. `@pytest.mark.parametrize("isin", ["LU0123456789", "FR0000120271"])`.

Q: Can you stack two `parametrize` decorators on one test?
A: Yes — pytest produces the Cartesian product, so two decorators with 3 and 2 values give 6 tests.

Q: How do you give parametrized cases readable names?
A: Pass `ids=[...]` to `parametrize`, supplying a label per case so the report reads clearly instead of using auto-generated ids.

Q: How would you parametrize a NAV rounding test across several currencies?
A: `@pytest.mark.parametrize("amount,ccy,expected", [(1.005, "EUR", 1.01), (1.005, "JPY", 1.0)])` then assert your `round_nav(amount, ccy)` matches `expected`.

Q: What is a "pure function"?
A: A function whose output depends only on its inputs and which has no side effects (no file/network/global state changes) — same inputs always give the same output.

Q: Why are pure functions the easiest things to unit test?
A: You just call them with inputs and assert on the return value — no setup, no files, no mocking, fully deterministic.

Q: Give a fund-data example of a pure function worth testing.
A: A `parse_isin(raw) -> str` or `compute_nav(assets, liabilities, shares) -> Decimal` that only transforms its arguments and returns a value.

Q: What does "separating IO from logic" mean?
A: Keeping file/network reading in thin wrapper functions and putting the actual parsing/calculation in pure functions, so the logic can be tested without touching disk.

Q: Why refactor `load_and_parse_emt(path)` into two functions before testing?
A: Split it into `read_text(path)` (IO) and `parse_emt(text)` (pure) so you can test `parse_emt` directly on a string without needing a real EMT file.

Q: A function both reads a CSV and computes NAV — how do you make it testable?
A: Extract the calculation into a pure function that takes already-parsed rows, then test that function with in-memory data and keep the thin file-reading part minimal.

Q: Why is code that mixes calculation and `print`/`open` hard to test?
A: You cannot assert on a return value because the result is buried in side effects, so you would need captured output or temp files just to check the logic.

Q: What is "one behavior per test"?
A: The principle that each test should verify a single, specific behaviour, so a failure points to exactly one cause.

Q: Why is one-behavior-per-test better than one giant test with twenty asserts?
A: If the first assert fails the rest never run, and the test name no longer tells you what broke; small focused tests localise failures.

Q: How should you name a test that checks invalid ISINs are rejected?
A: Something descriptive like `test_rejects_isin_with_wrong_length` so the name alone tells you the behaviour under test.

Q: What is the Arrange-Act-Assert structure of a test?
A: Arrange the inputs/setup, Act by calling the code under test, then Assert on the result — a clear three-part shape per test.

Q: In Arrange-Act-Assert, what belongs in "Act"?
A: The single call to the function or behaviour you are testing, kept to one action so the assert clearly maps to it.

Q: What is a pytest fixture?
A: A function decorated with `@pytest.fixture` that provides reusable setup (and optional teardown) to tests by being requested as a parameter.

Q: How does a test use a fixture?
A: It declares the fixture's name as a parameter; pytest calls the fixture and passes its return value in automatically.

Q: Write a basic fixture that returns sample NAV rows.
A: `@pytest.fixture` then `def nav_rows(): return [{"isin": "LU0123456789", "nav": 10.5}]`, used by `def test_x(nav_rows): ...`.

Q: Why use a fixture instead of repeating setup in every test?
A: It removes duplication and gives one place to change the shared setup, keeping tests focused on their assertions.

Q: How does a fixture do teardown (cleanup after the test)?
A: Use `yield` instead of `return`: code before `yield` is setup, the yielded value is given to the test, and code after `yield` runs as teardown.

Q: What is the `tmp_path` fixture?
A: A built-in pytest fixture that gives each test a unique, empty temporary directory as a `pathlib.Path`, automatically created and cleaned up.

Q: What type does `tmp_path` provide?
A: A `pathlib.Path` object pointing to a fresh temporary directory unique to that test.

Q: Why use `tmp_path` instead of writing to a fixed file like `/tmp/test.csv`?
A: It isolates each test in its own directory, avoids clashes between tests or parallel runs, and pytest cleans it up automatically.

Q: How do you write a test EMT file into `tmp_path`?
A: Build a path like `f = tmp_path / "emt.csv"` then `f.write_text("header\nrow")`, and pass `f` to your reader under test.

Q: How would you test a function that reads a file using `tmp_path`?
A: Create the file in `tmp_path` with known contents, call your function on that path, and assert it returns the parsed result.

Q: What is the difference between `tmp_path` and `tmp_path_factory`?
A: `tmp_path` is per-test (function scope); `tmp_path_factory` is session-scoped and lets you create temp dirs shared across many tests.

Q: Where do you put fixtures so many test files can share them?
A: In a `conftest.py` file; pytest discovers fixtures there automatically for all tests in that directory and below, with no import needed.

Q: Do you have to import a fixture defined in `conftest.py`?
A: No — pytest makes `conftest.py` fixtures available to tests in the same directory tree automatically.

Q: What does fixture "scope" control?
A: How often the fixture is created — `function` (default, per test), `class`, `module`, `package`, or `session` (once per run).

Q: When would you use `scope="session"` for a fixture?
A: For expensive setup that is safe to share and never mutated across tests, like a read-only sample dataset loaded once per run.

Q: Can one fixture depend on another fixture?
A: Yes — a fixture can request other fixtures as parameters, and pytest resolves the dependency chain for you.

Q: What is the recommended first step of TDD-style fixing: watch a test fail?
A: Write or run a test that exercises the bug so you see it fail, proving the test actually detects the problem before you fix the code.

Q: Why deliberately watch a test fail before fixing the code?
A: A test that passes for the wrong reason (e.g. never runs, or asserts nothing meaningful) gives false confidence; seeing red first confirms it truly checks the behaviour.

Q: You fixed a bug but the test still passes the same before and after — what went wrong?
A: The test probably never actually exercised the bug (wrong input, wrong assert, or not collected), so confirm it failed against the broken code first.

Q: After fixing code so a test passes, what is the discipline?
A: Re-run the full suite to confirm the fix works and that you did not break any other test (no regressions).

Q: A test passes locally but you suspect it tests nothing — how do you check?
A: Break the production code on purpose and confirm the test goes red; if it stays green, the test is not asserting what you think.

Q: `uv run pytest` says "no tests ran" — what's the first thing to check?
A: That your files match `test_*.py` and functions start with `test`; misnaming is the most common cause of zero collection.

Q: pytest reports `ModuleNotFoundError` importing your own package in a test — likely cause?
A: The package isn't importable from the rootdir; install it editable (`uv pip install -e .` via uv / declare it in `pyproject.toml`) or fix the package layout so imports resolve.

Q: What is a `conftest.py` used for besides fixtures?
A: Shared hooks, plugins, command-line options, and path setup for the tests in its directory tree.

Q: What does `assert a == b` print when it fails on two dicts?
A: A detailed diff showing which keys/values differ, thanks to pytest's assertion rewriting.

Q: How do you assert that a function raises a specific exception?
A: Use `with pytest.raises(ValueError): parse_isin("bad")` — the test passes only if that exception type is raised inside the block.

Q: How do you also check the exception message?
A: Pass `match=` a regex: `with pytest.raises(ValueError, match="length"): ...`, which asserts the message matches.

Q: Why test the error path (e.g. a malformed ISIN), not just the happy path?
A: Because bad input is common in fund feeds; you want to confirm the code rejects it clearly rather than silently producing a wrong NAV.

Q: How do you compare floating-point results safely in a test?
A: Use `pytest.approx`, e.g. `assert compute(...) == pytest.approx(10.5)`, to allow tiny floating-point rounding differences.

Q: Why might `assert 0.1 + 0.2 == 0.3` fail?
A: Binary floating point can't represent those decimals exactly, so the sum is slightly off; use `pytest.approx` or `Decimal` for money/NAV.

Q: Why prefer `Decimal` over `float` for NAV and money in code under test?
A: `Decimal` gives exact decimal arithmetic and predictable rounding, avoiding the float surprises that make monetary tests flaky.

Q: How do you mark a test to be skipped?
A: Use `@pytest.mark.skip(reason="...")` to always skip, or `@pytest.mark.skipif(condition, reason="...")` to skip conditionally.

Q: What is `@pytest.mark.xfail` for?
A: To mark a test you expect to fail (e.g. a known bug not yet fixed); pytest reports it as `xfail` instead of a failure.

Q: How do you register a custom marker like `@pytest.mark.slow`?
A: Declare it under `[tool.pytest.ini_options] markers = [...]` in `pyproject.toml` (or `pytest.ini`) to avoid the unknown-marker warning.

Q: How do you run only tests marked `slow`?
A: Use `uv run pytest -m slow`, where `-m` selects by marker expression.

Q: Where can you configure pytest options for the project?
A: In `pyproject.toml` under `[tool.pytest.ini_options]`, or in `pytest.ini`/`tox.ini`/`setup.cfg`.

Q: What does setting `testpaths = ["tests"]` in config do?
A: Tells pytest to look only in the `tests` directory by default, speeding up discovery and avoiding stray collection.

Q: How do you measure code coverage with pytest?
A: Add the `pytest-cov` plugin and run `uv run pytest --cov=yourpackage`, which reports which lines were exercised by tests.

Q: Is 100% coverage the same as correct code?
A: No — coverage shows lines were executed, not that the assertions meaningfully check behaviour; you can run a line without testing its correctness.

Q: For a transfer-agency subscription processor, what's a good first unit test?
A: Extract the pure calc (shares = amount / nav) and test it with known inputs, including edge cases like a zero or missing NAV that should error.

Q: Why keep tests fast and not hitting real fund databases or SFTP servers?
A: Fast, IO-free tests run on every save and in CI without network flakiness; you test parsing/logic on sample strings and reserve real connections for separate integration tests.

Q: What is the overall fix loop this lesson teaches in one sentence?
A: Write a test for the behaviour, watch it fail with `uv run pytest`, fix the (pure, IO-separated) code, watch it pass, then re-run the whole suite to confirm no regressions.
