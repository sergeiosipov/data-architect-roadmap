# Phase 2 · 9.6.2 Data documentation — 150 self-test questions

Q: In one sentence, what is the core deliverable of dbt's documentation feature?
A: dbt generates an always-current, browsable docs site from the descriptions you write in YAML plus the metadata it already knows (lineage, schema, tests, columns), so the documentation is produced from the same code that builds the models. Because it is generated rather than hand-maintained in a separate wiki, it stays in sync with the project as long as the YAML is kept up to date. This is why the lesson frames docs as a governance deliverable: traceability that a business analyst can self-serve.

Q: Which command generates the dbt documentation artifacts?
A: `dbt docs generate` produces the documentation artifacts in the `target/` directory. It compiles the project, runs a metadata query against the warehouse, and writes `manifest.json`, `catalog.json`, and `index.html`. A common follow-up is `dbt docs serve`, which starts a local web server so you can browse the generated site.

Q: What three primary files does `dbt docs generate` write into the `target/` directory?
A: It writes `manifest.json` (the full project graph, nodes, descriptions, and lineage), `catalog.json` (column-level metadata and statistics pulled from the warehouse), and `index.html` (the static single-page app that reads those two JSON files). The browser app loads `manifest.json` and `catalog.json` at runtime to render the lineage graph and model pages. Knowing this split matters when you publish docs: you must ship all the artifacts, not just the HTML.

Q: Why does `dbt docs generate` need a live connection to the warehouse?
A: To build `catalog.json` it issues a metadata query against the warehouse's information schema (or equivalent) to collect each relation's actual columns, data types, and stats. Without that connection it cannot populate the catalog, so column-level details and the "approximate row count" style stats would be missing. You can skip part of this with flags, but the catalog enrichment is the reason a warehouse round-trip happens at generate time.

Q: What does the `dbt docs serve` command do, and on what default port?
A: `dbt docs serve` starts a lightweight local HTTP server that hosts the generated `index.html` plus the JSON artifacts so you can browse the docs in a browser. By default it serves on port `8080`; you can change it with `--port`. It is a convenience for local viewing only — it is not meant to be your production hosting solution, which is why the lesson plans to publish the static site from CI instead.

Q: Why can you not just open `target/index.html` directly with a `file://` URL in some browsers?
A: The docs page is a JavaScript app that fetches `manifest.json` and `catalog.json` at runtime, and many browsers block those local `fetch` requests under the `file://` scheme for security (CORS/same-origin restrictions). Serving the directory over HTTP, for example via `dbt docs serve` or any static web server, lets the app load its JSON. This is the usual first thing to check when the docs page renders blank.

Q: What is the purpose of the `--no-compile` flag on `dbt docs generate`?
A: `dbt docs generate --no-compile` tells dbt to skip recompiling the project and instead reuse the existing compiled artifacts, only performing the catalog metadata query and writing the docs. It is useful in CI when a previous step already ran `dbt build` or `dbt compile`, so you avoid redundant compilation. The trade-off is that your artifacts must already be fresh, or the docs will reflect a stale graph.

Q: What does the `--static` flag on `dbt docs generate` produce?
A: `dbt docs generate --static` embeds the `manifest.json` and `catalog.json` data inside a single self-contained file named `static_index.html` in `target/`. Because everything is inlined, you can host or share that one file with no separate JSON artifacts and no web server logic, which is ideal for dropping into object storage or emailing a snapshot. The regular `index.html` by contrast still fetches the JSON at runtime.

Q: A colleague shares `index.html` from your docs build but it loads blank for them — what is the likely cause?
A: The plain `index.html` is not self-contained: it expects to fetch `manifest.json` and `catalog.json` from the same directory, and the colleague only received the HTML. Either send all the artifacts together and serve them over HTTP, or regenerate with `dbt docs generate --static` to produce a single `static_index.html` that inlines the data. The static build is the right tool when you need to share one portable file.

Q: How do you attach a description to a model in dbt?
A: You add a `description:` key under the model entry in a properties YAML file, typically `models/<something>.yml`, for example a model named `fct_nav` gets `description: Daily NAV per share class`. dbt reads this at parse time and surfaces it on the model's docs page and in `manifest.json`. The description lives alongside the model definition so it is reviewed and versioned with the SQL.

Q: How do you document an individual column in a dbt model's YAML?
A: Under the model's `columns:` list, each entry has a `name:` and an optional `description:`, so you can write a sentence per column such as documenting that `isin` is the 12-character ISO 6166 instrument identifier. dbt matches the `name` to the actual column from the catalog and shows the description on the model page. Columns without descriptions still appear (from the catalog) but show as undocumented.

Q: What is a dbt "docs block" and how is it defined?
A: A docs block is a reusable chunk of Markdown defined with the Jinja tag pair `{% docs some_name %} ... {% enddocs %}`, placed in a `.md` file inside your project. It lets you write a longer, formatted description once and reference it from many places. This keeps verbose explanations out of the YAML and avoids copy-pasting the same wording across models.

Q: How do you reference a docs block from a model or column description?
A: You set the description to a Jinja call to the `doc()` function, for example `description: "{{ doc('emt_field_overview') }}"`, where the string passed matches the docs-block name. dbt resolves the block's Markdown at parse time and renders it as that resource's description. This indirection is how you share one canonical paragraph across the EMT-related models.

Q: Why would you prefer a docs block over inlining a long description in YAML?
A: Docs blocks let you write rich, multi-paragraph Markdown (lists, links, code) that would be awkward to embed in a YAML scalar, and you can reuse the same block across several models or columns so wording stays consistent. They also keep the YAML compact and readable in PR diffs. The trade-off is a small indirection — the reader of the YAML has to open the `.md` file to see the full text.

Q: In YAML, what is the difference between the `>` and `|` block scalar styles for a description?
A: `>` is a folded scalar that joins interior line breaks into spaces, producing a single flowing paragraph, while `|` is a literal scalar that preserves the line breaks exactly as written. Use `>` for one long sentence wrapped across lines for readability, and `|` when you need real multi-line structure. Picking the wrong one is a common reason a description renders with unexpected or missing line breaks.

Q: What does the `persist_docs` config do?
A: `persist_docs` pushes your YAML descriptions out of dbt and into the warehouse itself as native relation and column comments, so tools and analysts querying the database directly see the same documentation. It turns dbt descriptions into `COMMENT`/`COMMENT ON` metadata on the actual tables and columns. This is the bridge from "docs in dbt" to "docs in the warehouse" that downstream BI catalogs can read.

Q: What is the exact shape of the `persist_docs` config value?
A: It is a dictionary with two boolean sub-keys, `relation` and `columns`, for example `persist_docs: {"relation": true, "columns": true}`. `relation: true` persists the table/view description, and `columns: true` persists each column's description. You can enable one without the other, and both default to off.

Q: Where can you set `persist_docs`?
A: You can set it in `dbt_project.yml` under the `models:` (or `seeds:`/`snapshots:`) path using the `+persist_docs:` prefix, or per resource in a model `{{ config(persist_docs={...}) }}` block, or in a snapshot's config. Setting it project-wide is the usual approach so every model gets database comments by default. Per-model config is handy when only some relations should be commented.

Q: During which dbt command are the `persist_docs` comments actually written to the warehouse?
A: The comments are applied during `dbt run` (and `dbt build`), as part of materializing each model, not during `dbt docs generate`. dbt issues the appropriate `COMMENT` statement against the relation after creating it. A frequent gotcha: people enable `persist_docs`, run `dbt docs generate`, and wonder why the warehouse has no comments — you must re-run the models.

Q: You enabled `persist_docs` but the table comments in the warehouse are stale — first thing to check?
A: Confirm you actually re-ran the models with `dbt run`/`dbt build` after changing the descriptions, because comments are written at materialization time, not at docs-generate time. Also check that the change was to a description that maps to a real column name and that the adapter supports the comment type you enabled. For incremental models, verify the run actually touched the relation rather than no-op'ing.

Q: Name several adapters that support `persist_docs`.
A: `persist_docs` is implemented for Postgres, Redshift, Snowflake, BigQuery, Databricks, and Apache Spark, among others. Each maps the description to that platform's native comment mechanism, such as `COMMENT ON COLUMN` in Postgres or column descriptions in BigQuery. Support and exact behaviour can vary by adapter, so check the adapter docs before relying on it for a given warehouse.

Q: Does `persist_docs` work for dbt sources?
A: No — `persist_docs` is not implemented for sources, since dbt does not own or create the source tables and therefore will not write comments onto objects it did not materialize. You can still document sources in YAML and have those descriptions appear in the dbt docs site; they just will not be pushed into the database. If you need source-table comments in the warehouse, that has to be done by whatever process owns those tables.

Q: For a regulated fund warehouse, why is `persist_docs` a useful governance lever specifically?
A: Pushing descriptions into the warehouse means an analyst or a downstream catalog reading the database directly sees the same authoritative definition for, say, a `nav_per_share` column, without having to open the dbt docs site. This reduces the chance that two tools show conflicting definitions of a regulated figure. It anchors the meaning of sensitive fields like NAV or ISIN at the database layer where audits often look.

Q: What is a dbt exposure?
A: An exposure is a node you declare in dbt to represent a downstream consumer of your project — a dashboard, a report, an application, an ML pipeline — that depends on specific models. It makes the consumer a first-class part of the DAG so lineage extends past the warehouse into "who actually uses this". The lesson's key use is declaring the EMT output's consumer so its dependency is explicit.

Q: In which file and under which top-level key are exposures declared?
A: Exposures are declared in a `.yml` properties file (often a dedicated `models/exposures.yml`) under the top-level `exposures:` key, as a list of entries. Each entry describes one downstream consumer with its name, type, owner, and dependencies. Putting them in YAML means they are reviewed and versioned exactly like model definitions.

Q: Which properties are required on an exposure?
A: An exposure requires `name`, `type`, and `owner`, where `owner` must include at least a `name` or an `email`. Without these dbt cannot parse the exposure. Everything else — `depends_on`, `url`, `maturity`, `description`, `label`, `tags`, `meta` — is optional but usually worth filling in.

Q: What are the valid values for an exposure's `type`?
A: The valid `type` values are `dashboard`, `notebook`, `analysis`, `ml`, and `application`. They categorize the kind of downstream consumer and control the icon/label dbt shows in the docs. For a regulatory file feed like the EMT output, `application` is the natural fit since it is a generated deliverable consumed by another system.

Q: How does an exposure declare the models it depends on?
A: Through the `depends_on:` list, which holds refable nodes expressed as `ref('model_name')`, `source('schema','table')`, or `metric('metric_name')`. For example the EMT exposure would list `ref('emt_output')`. dbt uses these to wire the exposure into the lineage graph and to enable selection by exposure.

Q: What does the `owner` field on an exposure capture and what does it need?
A: `owner` records who is accountable for the downstream consumer and must contain a `name` and/or an `email`. This is the human contact a business analyst or auditor reaches when they have a question about the EMT report, and it shows on the exposure's docs page. Filling in a real distribution-list email is better than an individual so the ownership survives staff changes.

Q: What is the `maturity` property on an exposure and what values can it take?
A: `maturity` signals how production-ready or trusted the downstream consumer is, taking one of `high`, `medium`, or `low`. It is informational metadata shown in the docs, useful for telling consumers whether a dashboard is battle-tested or experimental. For a regulator-facing EMT feed you would typically mark it `high` once it is the official output.

Q: What does the optional `url` property add to an exposure?
A: Setting `url:` to the live location of the consumer (for example the BI dashboard link or the report endpoint) activates a "View this exposure" link on its docs page. This lets an analyst jump straight from the lineage graph to the actual artifact. It is the small touch that makes the docs genuinely navigable rather than just descriptive.

Q: How do you select an exposure and its upstream models on the command line?
A: You use the `exposure:` selector, and prefixing with `+` pulls in upstream dependencies, for example `dbt build -s +exposure:emt_daily_feed` runs everything the EMT exposure depends on. This is the operational payoff of declaring exposures: you can test or rebuild exactly the graph that feeds a given consumer. Without the `+`, you would select only the exposure node itself, which is rarely what you want.

Q: How does an exposure appear in the dbt lineage graph?
A: An exposure renders as a distinct node in the DAG, drawn downstream of the models it depends on, with an "EXP" indicator that visually separates it from models and sources. This is what lets a viewer trace lineage all the way to the consuming dashboard or feed. It is the visual proof that "this EMT report comes from these specific gold models".

Q: Declare a minimal exposure for the EMT output naming its consumer — what entries are essential?
A: At minimum you write `name: emt_daily_feed`, `type: application`, `owner:` with an `email`, and `depends_on: [ref('emt_output')]`; adding `description`, `url`, and `maturity: high` makes it complete. The `type: application` reflects that EMT is a generated file consumed by a distributor or data vendor. Naming the consumer in the `description`/`label` is what satisfies the Done-when criterion for this lesson.

Q: Why does declaring the EMT output as an exposure satisfy a fund-platform governance goal?
A: It makes the regulatory deliverable an explicit, owned node in the lineage so anyone can trace which gold models feed the EMT file and who is responsible for it, without asking the data engineer. That answers "where does this EMT field come from?" through self-service, which the lesson calls a regulatory expectation rather than a nicety. It also lets you rebuild and test exactly the EMT dependency chain via the `exposure:` selector.

Q: What is the difference between what belongs in dbt model YAML versus the enterprise catalog?
A: dbt YAML is the right home for technical, model-local documentation maintained by engineers — column meanings, tests, lineage within the warehouse — versioned next to the SQL. The enterprise catalog is the broader, organization-wide layer for business glossary terms, data ownership across systems, classification, and policies that span far beyond one dbt project. The rule of thumb: model-specific truth lives in dbt; cross-system business and governance metadata lives in the catalog.

Q: Why not try to put the entire enterprise business glossary into dbt model descriptions?
A: dbt descriptions are scoped to a single project and are best at explaining how a specific model and its columns work; an enterprise glossary needs to span every system, be owned by business stewards, and outlive any one dbt project. Cramming glossary-wide definitions into dbt creates duplication and drift when the same term appears in many projects. The catalog is built to be the single canonical glossary, and dbt docs can link to or feed it.

Q: How do dbt docs and an enterprise catalog typically relate in a mature platform?
A: dbt docs is the generated, engineer-maintained source of model and column truth, which a catalog (Phase 6 territory, or a tool like Purview) ingests via dbt's `manifest.json`/`catalog.json` artifacts to enrich the broader business metadata graph. So dbt feeds the catalog rather than competing with it. The lesson explicitly calls dbt docs "the gateway drug before the Phase 6 catalogs".

Q: Which dbt artifact would an enterprise catalog most likely consume to import your model documentation?
A: It would consume `manifest.json`, which contains the full graph of nodes, their descriptions, columns, tests, and lineage edges, often supplemented by `catalog.json` for physical column metadata. These machine-readable JSON files are the integration point that lets a catalog ingest dbt's documentation automatically. This is why keeping descriptions in dbt pays off twice: the docs site and the catalog feed.

Q: What is the "docs-as-code" review flow this lesson advocates?
A: Docs-as-code means model and column descriptions live in the same versioned YAML as the SQL, so every documentation change goes through the same pull-request review as code — a reviewer sees the description diff next to the model diff. This keeps docs from drifting because they cannot be merged without review, and they ship in the same commit as the logic they describe. It is the discipline that makes generated docs trustworthy.

Q: Why is reviewing descriptions in the same PR as the SQL change important?
A: When you change a model's logic, the meaning of its columns can change too; reviewing the description alongside the SQL forces the author to update the docs in the same breath and lets the reviewer catch a description that no longer matches the code. If docs lived in a separate wiki, they would routinely lag behind merged code. Co-locating them is the structural fix for documentation drift.

Q: Why does the lesson say "docs that drift from the models are worse than none"?
A: Because people trust generated docs and act on them; a confidently wrong description of, say, how `nav_per_share` is calculated can mislead an analyst or auditor more dangerously than an obvious gap they would investigate. Missing docs prompt a question; wrong docs end the question with a false answer. This is why the review discipline that keeps docs current is treated as part of the deliverable, not optional polish.

Q: How can you enforce in CI that new models do not ship without descriptions?
A: You can run a check that inspects `manifest.json` (or use a package/test that flags undocumented models and columns) and fails the build when a model or column lacks a `description`. dbt itself does not block on missing descriptions by default, so the gate is something you add. Wiring this into the PR pipeline turns "every model documented" from a hope into an enforced rule.

Q: A reviewer approves a PR that changes a model's SQL but leaves the column description unchanged and now inaccurate — what process gap does this reveal?
A: It reveals that the team is not treating descriptions as part of the reviewable diff: the reviewer focused on SQL and skipped the docs even though they were in the same file. The fix is a review checklist or a CI doc-coverage check that forces description updates when logic changes. This is exactly the drift the docs-as-code flow exists to prevent.

Q: What is the practical Done-when test for this lesson regarding a business analyst?
A: The bar is that a business analyst can answer "where does this EMT field come from?" entirely on their own, by browsing the generated docs and lineage, without interrupting the engineer. That requires complete model and column descriptions, an EMT exposure naming its consumer, and a browsable lineage graph. If they still have to ask you, the documentation is not yet a real governance deliverable.

Q: Where in the dbt docs site would an analyst trace the full path of an EMT field?
A: In the lineage graph (the DAG view), they can click the EMT output node or exposure and walk the edges upstream through the gold, silver, and source nodes that feed it, opening each model's page to read column descriptions. The graph is the visual answer to "where does this field come from". Generating it with up-to-date `manifest.json` is precisely what `dbt docs generate` provides.

Q: What does `dbt docs generate` use the `manifest.json` for versus `catalog.json`?
A: `manifest.json` holds dbt's own knowledge — the node graph, descriptions, tests, configs, and lineage edges — assembled from your project files at parse time. `catalog.json` holds the physical reality from the warehouse — actual columns, data types, and stats — gathered by the metadata query. The docs site overlays the two so you see both "what dbt intends" and "what the database actually has".

Q: A column shows in the docs catalog but has no description — what does that tell you?
A: It means the physical column exists in the warehouse (so `catalog.json` picked it up) but you never wrote a `description` for it in the model YAML, so `manifest.json` has nothing to render. The fix is to add a `columns:` entry with that exact `name` and a `description`. Doc-coverage tooling exists precisely to surface these undocumented-but-present columns.

Q: You added a column description in YAML but it does not appear in the docs — first things to check?
A: Check that the column `name` in YAML exactly matches the real column name (including case/quoting), that you regenerated docs with `dbt docs generate` after the change, and that you are viewing the freshly served artifacts rather than a cached old page. A mismatched name is the most common cause: dbt cannot associate the description with a column it cannot find. Quoting and case sensitivity matter on some warehouses.

Q: How does the `quote: true` column property interact with documentation on case-sensitive warehouses?
A: Setting `quote: true` on a column tells dbt to treat the column name literally with quoting, which matters when the warehouse preserves case or the name has special characters, so the description maps to the right physical column. Without it, the name may be folded to a different case and fail to match the catalog. This is a subtle reason a description silently fails to show up.

Q: Why are dbt tests relevant to documentation quality even though they are not descriptions?
A: Tests like `not_null` and `unique` declared in the same YAML appear on the model's docs page, so a reader sees not just what a column means but what guarantees hold for it — a documented `isin` column that is also tested `unique` and `not_null` is far more trustworthy. Documentation plus tests together communicate intent and enforcement. The co-location in YAML means both are reviewed in the same PR.

Q: When publishing dbt docs from CI later, what must the pipeline produce and where would it go?
A: The pipeline runs `dbt docs generate` to produce the artifacts (or `--static` for a single `static_index.html`) and then publishes them to a static host — object storage with web hosting, GitHub Pages, or similar. Because the plain build needs `manifest.json` and `catalog.json` alongside `index.html`, the publish step must upload all of them unless you used the static build. This is the Phase 3 plan referenced in the Do list.

Q: Why might `dbt docs generate --static` be the better choice for hosting docs from cheap object storage?
A: The `--static` build inlines all metadata into one `static_index.html`, so you can drop a single file into a bucket and share it directly without exposing the JSON artifacts or running a server. Hosting a single self-contained file is simpler to secure and permission than a bucket of files that the page must fetch. The non-static build needs the bucket to serve the JSON, which complicates access control.

Q: What is the risk of letting a model's `description` and its actual transformation logic disagree in a regulated fund context?
A: A description that misstates how a regulated figure such as NAV or a holdings weight is computed can lead an analyst or auditor to certify or report a number on a false understanding, which in fund administration can have compliance consequences. Generated docs are trusted, so the wrong words carry real weight. This is why the lesson ties documentation accuracy to regulatory traceability, not cosmetics.

Q: How would you document a source table's freshness expectation so analysts understand data timeliness?
A: You document the source in YAML with a `description` and configure `freshness` (with `loaded_at_field` and `warn_after`/`error_after`) so dbt can report staleness; the description explains the expectation in words and the config enforces it. The docs page then shows both the meaning and the freshness policy. For a daily NAV feed, stating "expected by 06:00 CET each business day" in the description gives analysts a concrete timeliness anchor.

Q: What metadata can you attach to a model with the `meta` config, and how does it help governance?
A: The `meta` config holds an arbitrary key-value dictionary — for example `meta: {contains_pii: true, regulatory: EMT}` — which surfaces in `manifest.json` and the docs and can be consumed by downstream catalogs or policy tooling. It lets you tag models with governance attributes that dbt has no built-in field for. In a fund platform you might mark which models feed regulated outputs so the catalog can apply controls.

Q: How are dbt `tags` useful for documentation and selection together?
A: `tags` label models, exposures, and other resources with free-form categories that show in the docs and double as selectors, so `dbt build -s tag:emt` rebuilds everything tagged for the EMT pipeline. They give a lightweight grouping that complements descriptions. A reader browsing docs can filter by tag to find all regulatory-output models quickly.

Q: An exposure references `ref('emt_output')` but `dbt parse` errors that the node is not found — what is wrong?
A: The string inside `ref()` must exactly match an existing model's name; the error means there is no model called `emt_output` in the project (typo, wrong name, or the model is disabled). Check the actual file name/`name:` of your EMT model and fix the `depends_on` entry. Exposures are validated against the graph at parse time, so a bad `ref` fails fast.

Q: Why should an exposure's `owner` ideally be a team or distribution-list email rather than a single person?
A: Ownership recorded as one individual's email rots the moment that person changes roles or leaves, breaking the contact path an auditor relies on for the EMT feed. A team distribution list keeps the ownership pointer valid across staff changes. The exposure's whole point is durable accountability, so durability of the contact matters.

Q: What does it mean that dbt docs are "generated" rather than "authored"?
A: The site itself — its structure, lineage graph, column lists, and stats — is assembled automatically by dbt from your project and the warehouse; you author only the prose descriptions in YAML and docs blocks. So you maintain small textual inputs and dbt produces the full document each time. This is what makes the docs cheap to keep current and resistant to the drift that plagues hand-written wikis.

Q: How do you make a longer, multi-paragraph model description with proper formatting?
A: Define the text in a docs block, `{% docs my_model_overview %} ...rich Markdown... {% enddocs %}` in a `.md` file, and set the model's `description: "{{ doc('my_model_overview') }}"`. The block supports Markdown lists, links, and code that a YAML scalar handles awkwardly. This is the idiomatic way to give an important model like the EMT output a thorough write-up.

Q: What happens to columns that exist physically but are absent from the model's YAML `columns:` list?
A: They still appear on the docs page because `catalog.json` captured them from the warehouse, but they show with no description and no tests since dbt has no YAML entry for them. They are effectively undocumented columns flagged by their blank descriptions. Listing them in YAML — even just to add a description — is how you bring them under documentation.

Q: Why is it valuable that the lineage graph extends to exposures rather than stopping at the last model?
A: Stopping at the final model answers "what did we build" but not "who consumes it"; extending the graph to exposures answers "what breaks downstream if this model changes" and "where does this regulatory feed terminate". That downstream visibility is essential for impact analysis and for the self-service traceability the lesson requires. It closes the lineage loop from raw source to consuming application.

Q: In dbt, what is the relationship between a model's `description` and the warehouse column comment when `persist_docs` is on?
A: With `persist_docs: {relation: true, columns: true}`, dbt copies the YAML `description` into the warehouse as the relation's table comment and each column's comment when the model is run. So the YAML remains the single source of truth and the warehouse comment is a generated projection of it. Editing the description and re-running keeps both in sync; editing the comment directly in the warehouse would be overwritten on the next run.

Q: If two BI tools show conflicting definitions of `nav_per_share`, how does this lesson's tooling reduce that?
A: By keeping the canonical definition in the dbt model YAML and pushing it to the warehouse via `persist_docs`, both tools that read database comments see the same authoritative text. Combined with feeding the enterprise catalog from `manifest.json`, you collapse multiple definition sources into one upstream truth. The conflict usually stems from each tool maintaining its own description; sourcing from dbt removes that divergence.

Q: What is the danger of documenting a model in a separate Confluence/wiki page instead of in dbt YAML?
A: A separate wiki is not coupled to the code, so it is updated by a different person at a different time (or never) and silently drifts from the model it describes. There is no PR gate forcing the wiki to change when the SQL changes. dbt's docs-as-code approach exists specifically to bind the description to the code path so they move together.

Q: How does `dbt build` relate to documentation compared to `dbt docs generate`?
A: `dbt build` runs and tests your models and, with `persist_docs` enabled, writes the database comments — but it does not produce the docs website. `dbt docs generate` is the separate step that compiles the browsable docs artifacts. A complete docs pipeline usually runs `dbt build` then `dbt docs generate`, sometimes with `--no-compile` on the latter to avoid recompiling.

Q: Why does the acceptance criterion for Capstone 2 say "dbt build green with tests + docs published" — what does each half prove?
A: "Tests green" proves the data is correct and the models behave as documented, while "docs published" proves the always-current documentation deliverable actually reaches consumers, not just exists in `target/`. Together they show both correctness and traceability. A pipeline that builds but never publishes docs would fail the self-service test this lesson demands.

Q: How would you document the EMT output model so an analyst understands each EMT field's provenance?
A: Give the model a thorough `description` (ideally via a docs block) explaining the FinDatEx EMT context, then add a `columns:` entry per EMT field with a `description` that states the source column and any transformation. Pair it with the EMT exposure so lineage shows the upstream gold models. Then a browsing analyst can read each field's meaning and trace it upstream without asking you.

Q: What is the difference between `label` and `name` on an exposure?
A: `name` is the unique snake_case identifier used in selectors and the graph (for example `emt_daily_feed`), while `label` is an optional human-friendly display string that may contain spaces and capitals (for example "EMT Daily Feed"). The docs site shows the `label` when present, falling back to the `name`. Use `label` to make the exposure read nicely without sacrificing a clean machine identifier.

Q: Which exposure `type` best fits a generated regulatory file like the EMT output, and why?
A: `application` fits best, because the EMT output is a generated artifact consumed programmatically by another system (a distributor or vendor ingestion), not a human-browsed dashboard or notebook. The `type` mostly affects categorization and the icon in docs, but choosing it accurately helps readers understand the nature of the consumer. A reporting dashboard built on the same data would instead use `dashboard`.

Q: What happens if you run `dbt docs generate` without a warehouse connection?
A: The compile step can still run from your project files, but the catalog metadata query that builds `catalog.json` cannot, so column-level physical metadata and stats will be missing or the command can fail at that step. The lineage and descriptions from `manifest.json` may still render, but the catalog enrichment is absent. This is why CI docs jobs need warehouse credentials, not just the repo.

Q: How can `description` content from a docs block help avoid inconsistent definitions across many EMT-related models?
A: By writing the shared concept once in a docs block and referencing it with `{{ doc(...) }}` from every related model and column, you guarantee they all show identical, single-sourced wording. If the definition changes, you edit one block and every reference updates on the next docs build. This is the documentation equivalent of DRY and is ideal for a recurring regulatory concept.

Q: Why is column-level lineage particularly valuable for the question "where does this EMT field come from?"
A: Model-level lineage tells you which models feed the EMT output, but the analyst's real question is often about a single field, and column-level tracing pinpoints which upstream column maps into that EMT field through each transformation. The closer the lineage gets to the column, the less the analyst has to infer. Even where dbt's native graph is model-level, thorough column descriptions plus the DAG together carry the analyst to the answer.

Q: What is the consequence of leaving the EMT exposure undeclared in terms of self-service traceability?
A: Without the exposure, the lineage graph stops at the last gold model and never shows that the EMT file is the consumer, nor who owns it, so an analyst cannot self-serve the "where does this go / who do I ask" half of traceability. They would have to interrupt the engineer — exactly what the Done-when criterion forbids. Declaring the exposure closes that gap and names the responsible party.

Q: How do you ensure the published docs reflect the latest merged changes rather than a developer's local state?
A: Generate and publish docs from CI on the main branch after a merge, so the artifacts always come from reviewed, merged code rather than someone's laptop. This guarantees the served `manifest.json` matches what is in production. Publishing from a local `dbt docs serve` session is fine for development but must not be the source of truth shared with analysts.

Q: A model's docs page shows an old description even after you edited the YAML — what is the likely cause?
A: Most likely you are viewing a previously generated `index.html`/`manifest.json` and did not re-run `dbt docs generate`, or the browser cached the old JSON. Regenerate the docs and hard-refresh the page. If publishing via CI, confirm the latest pipeline actually ran after your merge rather than serving a stale build.

Q: Why does putting tests and descriptions in the same YAML file help the docs-as-code review?
A: Because a single PR diff then shows the logic intent (tests), the meaning (descriptions), and often the model file nearby, the reviewer evaluates correctness and documentation together in one place. This reduces the chance of approving a logic change while overlooking a now-wrong description. Co-location is the structural enabler of meaningful docs review.

Q: What does it mean that an exposure can be selected but not "run" as a model?
A: An exposure is a documentation/graph node, not an executable model, so `dbt run -s exposure:x` does not build the exposure itself but is used to select and operate on the upstream models it depends on (typically with the `+` prefix). The exposure has no SQL to materialize. Its value is metadata and graph connectivity, not computation.

Q: How would `meta` on the EMT exposure help a downstream catalog apply controls?
A: You can attach governance attributes in `meta`, such as `meta: {regulatory_regime: PRIIPs, criticality: high}`, which flow into `manifest.json` and can be ingested by a catalog to drive policies like access restrictions or required approvals. It lets you encode regulatory context where the catalog can act on it. This turns the exposure into a policy-bearing node, not just a label.

Q: Why is "every model and column has a description" a concrete acceptance bar rather than a vague aspiration?
A: It is mechanically checkable — you can scan `manifest.json` and fail CI if any model or column lacks a non-empty `description` — so "documented" becomes pass/fail rather than subjective. This converts documentation from good intentions into an enforced gate. The lesson sets it as a Done-when checkbox precisely because it can be verified objectively.

Q: How does generated dbt documentation reduce the bus-factor risk on a fund data platform?
A: Because the knowledge of what each model and column means lives in versioned YAML and is rendered into a browsable site, that understanding survives independently of any one engineer's memory. A new joiner or an auditor can self-serve the lineage and definitions of regulated outputs. The documentation becomes an institutional asset rather than tribal knowledge.

Q: What is the relationship between `dbt docs serve --port` and running multiple docs sites locally?
A: `dbt docs serve --port 8081` lets you serve on a non-default port, which is how you view two projects' docs simultaneously without the second clashing on the default `8080`. Each server reads the artifacts in its own `target/` directory. It is a small operational convenience, not a deployment mechanism.

Q: Why should the EMT exposure's `description` explicitly name the consumer rather than just say "EMT file"?
A: Naming the actual consumer — for example "Consumed by the distributor's PRIIPs ingestion service" — answers the analyst's downstream question and gives an auditor the real endpoint and responsible party. A generic "EMT file" leaves the trail incomplete. The Done-when criterion specifically requires the consumer to be named, so the description must carry that detail.

Q: How can you tell from the docs site whether a model is a source, a model, or an exposure?
A: The lineage graph and node styling distinguish them: sources, models, and exposures each render differently, with exposures carrying an "EXP" marker that sets them apart from upstream nodes. The node type is also shown on each entity's page. This visual typing is what lets a reader follow lineage across the boundary from data into consumers.

Q: What is a sensible CI ordering for build, test, docs, and publish steps?
A: A common order is `dbt build` (which runs and tests models, and with `persist_docs` writes warehouse comments), then `dbt docs generate` to produce artifacts, then a publish step uploading the artifacts to the static host. Running generate after build means the docs reflect freshly built models. The publish step gates on the build and tests passing so you never publish docs for a red build.

Q: Why is `persist_docs` complementary to, not a replacement for, `dbt docs generate`?
A: `persist_docs` pushes descriptions into the warehouse for tools and people querying the database directly, while `dbt docs generate` produces the rich browsable site with the lineage graph and exposures. They serve different audiences and surfaces. A complete documentation story uses both: comments in the warehouse plus the generated docs portal.

Q: How do exposures help with impact analysis before changing a gold model?
A: By tracing downstream from the model to any exposures that `depends_on` it, you can see immediately which consumers — dashboards, the EMT feed — would be affected, and you can rebuild and test exactly that set with `+exposure:`. This turns a risky change into a scoped, testable one. Without exposures, the downstream blast radius beyond the warehouse is invisible.

Q: What belongs in dbt YAML: the business glossary term "Net Asset Value" or the column note "computed as assets minus liabilities divided by shares for this share class"?
A: The column-specific computation note belongs in dbt YAML because it describes how this particular model's column is derived; the enterprise-wide glossary term "Net Asset Value" belongs in the catalog where it is owned by business stewards and reused across systems. dbt can reference or align with the glossary, but the canonical cross-system definition lives in the catalog. Mixing the two leads to duplicated, drifting definitions.

Q: Why is it risky to document a regulated field only inside an individual analyst's spreadsheet?
A: A spreadsheet is unversioned, unreviewed, and invisible to the lineage graph, so its definition cannot be trusted, traced, or kept current, and it does not satisfy the self-service traceability regulators expect. If that analyst leaves, the definition leaves too. The docs-as-code approach puts the definition where it is reviewed, versioned, and generated into a shared site instead.

Q: How does declaring exposures change the `dbt build` selection ergonomics for a release?
A: With exposures declared, a release that must guarantee a specific consumer works can target it directly via `dbt build -s +exposure:emt_daily_feed`, building and testing precisely the dependency chain that feeds the EMT output. This is more precise and safer than rebuilding everything or guessing the model set by hand. It operationalizes the documentation as a runnable selection.

Q: What does a blank or default model `description` of an empty string signal to a doc-coverage check?
A: An empty-string `description` typically counts as undocumented to a strict coverage check because it carries no information, even though the key technically exists. The check should treat empty or whitespace-only descriptions as failures, not passes. This prevents teams from gaming "every model has a description" by filling in blanks.

Q: Why is generated lineage more reliable than a manually drawn architecture diagram for answering provenance questions?
A: Generated lineage is derived from the actual `ref()`/`source()` dependencies in the code, so it cannot silently disagree with reality the way a hand-drawn diagram does as the project evolves. It updates every time you regenerate docs. A manual diagram is a snapshot that ages; the generated DAG is the living truth, which is exactly what provenance questions demand.

Q: How can docs blocks be organized in a project to stay maintainable?
A: Keep docs blocks in dedicated `.md` files (for example one per domain, like `emt.md` or `nav.md`) so related definitions are grouped and easy to find, and reference them from the YAML via `doc()`. This separates verbose prose from terse YAML structure and avoids scattering long text. It also makes the prose itself diffable and reviewable in PRs.

Q: What is the first thing to verify if `dbt docs serve` starts but the lineage graph is empty?
A: Verify that `dbt docs generate` actually completed and wrote a current `manifest.json` into `target/`, since `serve` only hosts whatever artifacts already exist. An empty or stale graph usually means generate was skipped or failed. Re-run `dbt docs generate`, confirm the artifacts updated, then serve again.

Q: Why does the lesson treat dbt docs as a stepping stone to Phase 6 catalogs rather than the final answer?
A: dbt docs covers documentation and lineage within a single dbt project, but an enterprise needs cross-system cataloging, business glossaries, stewardship, and policy that span far beyond dbt — which is the Phase 6 catalog layer. dbt docs gets you the discipline and the machine-readable artifacts that those catalogs ingest. It is the gateway: necessary, valuable, but scoped to the project, not the enterprise.

Q: How would you handle documentation for a column that is sensitive PII, such as an investor identifier?
A: Document its meaning in the `description`, and use `meta` (and/or `tags`) to mark it as PII, for example `meta: {contains_pii: true}`, so downstream catalogs and policy tools can apply masking or access controls. Avoid putting actual sensitive values in descriptions or docs blocks. The metadata flag travels through `manifest.json` to wherever governance enforcement happens.

Q: Why is it important that exposures, descriptions, and tests are all parsed from the same project at `dbt docs generate` time?
A: Because they are parsed together, the generated docs present a single consistent picture — the same graph drives lineage, the same descriptions appear on pages, and the same tests are listed — with no possibility of three separate sources disagreeing. One generate pass yields one coherent artifact set. This internal consistency is part of why generated docs are trustworthy.

Q: A new EMT field is added to the gold model but its description and the exposure are not updated — what acceptance criteria does this violate?
A: It violates "every model and column has a description" (the new column is undocumented) and undermines "a business analyst can answer where this EMT field comes from" since the field's provenance is not documented. If the new field changes the consumer contract, the exposure's description may also be stale. The docs-as-code review should have blocked the merge until the description was added.

Q: Summarize the end-to-end documentation workflow this lesson expects you to operate.
A: You write model and column descriptions (and reusable docs blocks) in versioned YAML reviewed in PRs alongside the SQL, optionally push them to the warehouse with `persist_docs`, declare exposures like the EMT output naming their consumers and owners, then run `dbt docs generate` and publish the site from CI. The result is always-current, self-service documentation where an analyst can trace any EMT field to its source without asking you. That self-service traceability — not the docs site itself — is the real governance deliverable.

Q: Your CI docs job must run without warehouse credentials — which flag lets `dbt docs generate` skip the metadata round-trip?
A: Use `dbt docs generate --empty-catalog`, which skips the warehouse metadata query and writes an empty `catalog.json` while still producing a full `manifest.json` with descriptions, tests, and lineage. This lets a credential-less CI step build a usable docs site that just lacks physical column types and stats. The trade-off is the docs lose warehouse-sourced details, so a separate credentialed job is needed if you want the full catalog.

Q: How do you generate docs for only a subset of your project, such as just the EMT lineage?
A: Pass `dbt docs generate --select` with a node selector, for example `dbt docs generate -s +exposure:emt_daily_feed`, so the catalog query and graph focus on that subset. This speeds up generation on large projects when you only care about one feed's documentation. Be aware of the historical caveat that selection mainly scopes the catalog query, and a stale or partial manifest can still surface unselected nodes.

Q: `dbt docs generate` is taking many minutes on a large warehouse — what is the bottleneck and how do you cut it?
A: The slow part is almost always the catalog metadata query that interrogates the information schema for every relation's columns and stats, not the compile. You can cut it with `--empty-catalog` to skip the query entirely, or `--select` to scope it to fewer relations. On some warehouses the per-relation `DESCRIBE`/information-schema scan dominates, so narrowing the node set is the most reliable speed-up.

Q: A reviewer wants the docs page to show test results, not just that tests exist — what extra step is needed beyond `dbt docs generate`?
A: The generated docs show which tests are defined on a column, but to show pass/fail status you must run the tests and load `run_results.json` alongside the docs artifacts, since the static site can overlay results from that file. Plain `dbt docs generate` only knows tests exist, not their outcome. Tools and dbt Cloud surface the status by combining `manifest.json` with `run_results.json`.

Q: You set `persist_docs: {relation: true, columns: true}` on a Snowflake model materialized as a `view`, but column comments do not appear — why?
A: Snowflake historically could not `ALTER` column comments on a view the way it can on a table, so column-level `persist_docs` on a view materialization may not persist even though the relation comment does. The mechanism that works for tables (post-create comment statements) does not map cleanly onto views. The workaround is to materialize as a table where column comments matter, or accept relation-only comments on the view.

Q: For BigQuery, how does dbt physically attach a column `description` to the table when `persist_docs` is on, and why does the mechanism matter?
A: On BigQuery, dbt embeds the descriptions inside the `CREATE` statement's `OPTIONS()` (and column `OPTIONS`) at materialization time rather than issuing a separate post-create comment, so the description is set as the relation is built. This matters because the comment cannot drift from the build — it is written in the same DDL. It also means nested/`RECORD` sub-fields have historically been harder to annotate this way.

Q: An analyst opens the docs and the EMT column descriptions are blank, but the warehouse `COMMENT`s are correct — what does this tell you about your pipeline?
A: It means `persist_docs` ran during `dbt run`/`dbt build` and wrote the comments, but you never regenerated and republished the docs site with `dbt docs generate` after the descriptions changed, so the served `manifest.json` is stale. The two surfaces are produced by different commands at different times. Re-run `dbt docs generate` and republish to bring the site in line with the warehouse.

Q: What `resource_type` does an exposure carry in `manifest.json`, and why does knowing it help when a catalog ingests dbt?
A: An exposure appears in `manifest.json` under the `exposures` map with `resource_type: exposure`, distinct from `model`, `source`, `test`, and `metric`. A catalog ingesting the manifest filters on `resource_type` to decide how to render each node, so an exposure is mapped to a downstream-consumer concept rather than a table. Knowing the type lets you write or debug a manifest parser that wires the EMT consumer correctly.

Q: How would you make the EMT exposure disappear from a specific environment without deleting its YAML?
A: Set `enabled: false` on the exposure (or drive it from a variable, e.g. `enabled: "{{ target.name == 'prod' }}"`), which dbt honors at parse time so the node is excluded from that environment's graph and docs. This is the same `enabled` mechanism models use. It lets you keep one YAML definition while showing the exposure only where the EMT consumer actually exists.

Q: Can an exposure's `depends_on` reference a metric, and when is that useful for a fund platform?
A: Yes — `depends_on` accepts `ref()`, `source()`, and `metric()` nodes, so an exposure can declare a dependency on a defined metric such as `metric('total_aum')`. This is useful when the downstream consumer is driven by a governed metric rather than a raw model, making the metric's lineage extend to the consumer. It documents that, say, an AUM dashboard depends on the canonical AUM metric, not an ad-hoc calculation.

Q: A teammate writes `depends_on: ref('emt_output')` as a bare string instead of a list — does dbt accept it, and what is the safer form?
A: `depends_on` is defined as a list, so each dependency must be a list item; writing a single bare `ref()` without the list dash is a YAML/parse error or silently mis-shaped. The safe form is a proper YAML list, e.g. `depends_on:` followed by `- ref('emt_output')`. Treating it as a list from the start avoids surprises when you add a second dependency later.

Q: What does adding `tags: [emt, regulatory]` to an exposure buy you beyond labeling?
A: Exposure `tags` show in the docs and double as selectors, so `dbt build -s +tag:regulatory` rebuilds every upstream model feeding any regulatory-tagged exposure in one command. Tags give a cross-cutting grouping that node names alone cannot. For a release that must protect all regulatory feeds, one tag selector captures the whole blast radius.

Q: How do you list every exposure in a project from the command line without opening the docs site?
A: Run `dbt ls --resource-type exposure` (or `dbt list --resource-type exposure`), which prints the exposure node names dbt parsed from the project. This is a quick inventory check in CI or a terminal. Adding `--output json` gives a machine-readable list you can assert against, e.g. to confirm the EMT exposure exists.

Q: You want CI to fail if the EMT exposure is ever removed — how would you assert its presence cheaply?
A: Run `dbt ls --resource-type exposure --output name` and grep/check that `emt_daily_feed` appears, failing the job if it is missing, which catches an accidental deletion before merge. This is a lightweight guard that does not require the warehouse. It encodes "the regulatory consumer must always be declared" as an enforced contract.

Q: What is the difference between `dbt parse` and `dbt docs generate` when you just want to validate your YAML descriptions and exposures?
A: `dbt parse` reads and validates the project files, building `manifest.json` and catching malformed YAML, bad `ref()`s in exposures, and config errors, without touching the warehouse or writing the docs site. `dbt docs generate` does all that plus the catalog query and HTML. For a fast "did my exposure and descriptions parse" check, `dbt parse` is the cheaper gate.

Q: A `doc()` reference like `{{ doc('emt_overview') }}` errors at parse time with a compilation error — what are the usual causes?
A: Either the docs block name does not exist (typo, or the `.md` file is outside a configured `docs-paths`/project path so dbt never loaded it), or there are two blocks with the same name causing a duplicate-resource error. dbt resolves `doc()` against the parsed set of docs blocks, so an unmatched or duplicated name fails fast. Check the block spelling and that exactly one `{% docs emt_overview %}` exists.

Q: Can a docs block live anywhere, and how does dbt decide where to look for `.md` docs files?
A: dbt searches the paths in `docs-paths` (defaulting to your `models/` and other resource paths) for `.md` files containing `{% docs %}` blocks, so a block only loads if its file sits under a scanned path. Dropping a docs `.md` outside those paths means `doc()` cannot find it. Setting `docs-paths` explicitly in `dbt_project.yml` is how you keep all blocks in a dedicated folder.

Q: Two docs blocks accidentally share the name `nav_definition` in different files — what happens and how do you resolve it?
A: dbt raises a duplicate-resource error at parse time because docs-block names must be unique across the project, just like model names. The fix is to rename one block (for example `nav_definition_gold` vs `nav_definition_source`) so each reference resolves unambiguously. Uniqueness is what lets `doc('name')` map to exactly one block.

Q: How can you embed an image or a link to the regulator's spec inside a model description?
A: Use a docs block, since it renders full Markdown, so you can write `[FinDatEx EMT spec](https://findatex.eu/)` or an image tag and reference it via `{{ doc(...) }}` from the model's `description`. A plain YAML scalar would show the Markdown as literal text only because the docs site renders block content as Markdown. This is the idiomatic way to give the EMT model a richly linked write-up.

Q: Why might a model's `description` render with broken formatting when you used YAML quotes, and how does a docs block avoid it?
A: Inlining Markdown in a quoted YAML scalar is fragesile — colons, `#`, and line breaks can break YAML parsing or fold unexpectedly, so lists and code blocks often render wrong. A docs block keeps the Markdown in a `.md` file untouched by YAML rules, so it renders exactly as written. Moving rich text out of YAML into `doc()` blocks is the reliable fix for mangled formatting.

Q: In `manifest.json`, where do model and column descriptions live, and why does a catalog reader care about that structure?
A: Each model node under `nodes` has a top-level `description` plus a `columns` map where each column entry has its own `description` and `meta`, so a catalog reader walks `nodes[...].columns[...].description` to import field-level docs. Knowing the shape lets you write a reliable importer or a coverage check. It is also how you would script "find every column with an empty description".

Q: How would you script a doc-coverage check against `manifest.json` to fail CI on any undocumented column?
A: Parse `manifest.json`, iterate the `nodes` of `resource_type` `model`, and for each, compare the warehouse columns to the `columns` map, failing if any column's `description` is empty or missing, then exit non-zero. Because the manifest is plain JSON, a short Python/`jq` script suffices and needs no warehouse. This turns "every column documented" into a mechanical, enforced gate rather than a hope.

Q: Why is comparing only `manifest.json` insufficient to know a model has no undocumented columns, and what closes the gap?
A: `manifest.json` only lists columns you wrote in YAML, so a column that exists physically but is absent from YAML is invisible to a manifest-only check and slips through undocumented. You close the gap by also reading `catalog.json`, which has the real warehouse columns, and flagging any catalog column missing from the manifest. The two together reveal "present but undocumented" fields.

Q: A column appears in `manifest.json` but not in `catalog.json` after generate — what does that mismatch usually mean?
A: It usually means you documented a column in YAML that does not actually exist in the built relation — a renamed or dropped column, or a typo in `name`. The catalog reflects the real warehouse, so its absence there means the physical column is gone or misspelled. Fix the YAML `name` to match reality, or remove the stale entry.

Q: How do you document and surface the grain of a fact model like `fct_nav` so analysts do not misuse it?
A: State the grain explicitly in the model `description` (for example "one row per fund per share class per NAV date") and reinforce it with a `unique` test on the composite key, so the docs page shows both the stated grain and the enforced guarantee. Grain is the single most misread property of a fact table. Documenting it in words plus a test stops analysts from double-counting NAV across share classes.

Q: An analyst asks why a `not_null` test on `isin` shows on the docs page even though you never wrote a description for that test — where does that come from?
A: Generic tests like `not_null` declared under the column in YAML are parsed into `manifest.json` as test nodes attached to the column, and the docs site lists them on the model page automatically. You did not describe the test; dbt rendered its existence from the parsed graph. This is why tests and descriptions co-located in YAML both appear together in the generated docs.

Q: How would you give a generic test itself a human-readable explanation in the docs?
A: Configure the test with a `config: {meta: ...}` or a custom `name`/`description` where the adapter and dbt version support it, so the docs and `manifest.json` carry context about why the test exists. For example a `relationships` test can note that `isin` must reference the security master. This turns a bare test into self-documenting intent for an auditor reading the page.

Q: For a source with `freshness` configured, which command actually checks staleness, and how does that relate to the docs?
A: `dbt source freshness` runs the freshness check against `loaded_at_field`, comparing it to `warn_after`/`error_after`, and writes `sources.json`; the docs `description` only explains the policy in words. Generating docs does not run the freshness check. So the timeliness promise shown in the description is enforced by a separate scheduled `dbt source freshness` run, typically before the NAV pipeline.

Q: Your source `freshness` keeps erroring even though data is current — first thing to check?
A: Verify `loaded_at_field` points to a real, populated timestamp column and that it is in the expected timezone, because a wrong or null `loaded_at_field` makes dbt see the data as infinitely stale. Also confirm the `warn_after`/`error_after` units and `count` are sane for a daily feed. A common fund-data trap is comparing a business-date column rather than a true load timestamp.

Q: How do you document a snapshot (SCD2) so analysts understand the `dbt_valid_from`/`dbt_valid_to` columns?
A: Add `description`s for the snapshot's meta columns — `dbt_valid_from`, `dbt_valid_to`, `dbt_scn_id`/`dbt_updated_at` — explaining that a null `dbt_valid_to` marks the current record, and describe the tracked business columns too. Snapshots generate these system columns, and analysts routinely misread them. Documenting that "current row = `dbt_valid_to is null`" prevents querying history incorrectly for, say, share-class fee changes.

Q: Why might `persist_docs` for columns silently do nothing on an incremental model run that processed zero new rows?
A: If the incremental run no-ops because no new rows matched, dbt may not re-issue the comment statements, so a description edited since the last full build is not reflected in the warehouse. Comments are applied as part of materializing the relation, and a skipped/no-op build skips that. A `--full-refresh` run forces the relation rebuild and reapplies the comments.

Q: How do you force `persist_docs` to reapply comments after editing descriptions on a large incremental table without reloading all data unnecessarily?
A: There is no description-only push, so you typically run `dbt run --full-refresh -s the_model` to rebuild and reapply comments, or rely on the next incremental run that touches the relation. For a very large table you weigh the rebuild cost against the staleness. Some teams document this as a known limitation and schedule periodic full refreshes to keep comments current.

Q: What is the difference between `meta` and `tags` for encoding that a model feeds a regulated EMT output?
A: `tags` are flat string labels that also act as selectors (`-s tag:emt`), while `meta` is a structured key-value dict carrying richer attributes like `meta: {regulatory: EMT, contains_pii: false}` that downstream catalogs read but cannot select on. Use `tags` when you need to operate on the group, `meta` when you need structured governance attributes. They are complementary, and many models use both.

Q: How would a downstream policy tool know from dbt metadata that a column must be masked?
A: You encode it in `meta`, for example `meta: {classification: confidential, mask: true}` on the column, which flows into `manifest.json`; a policy tool ingests the manifest and applies masking based on that flag. dbt itself does not mask — it just carries the attribute. This makes the manifest the single declarative source the enforcement layer reads.

Q: Why should the canonical NAV definition live in a docs block referenced by `doc()` rather than copied into each model's YAML?
A: A single docs block referenced from every NAV-related column guarantees identical wording everywhere and means a regulatory wording change is made once and propagates on the next build. Copy-pasted definitions drift the moment one copy is edited and the others are not. For a regulated figure, single-sourcing the definition is what prevents two models from describing NAV differently.

Q: An exposure's `url` points to a dashboard, but clicking "View this exposure" 404s — is that a dbt problem?
A: No — dbt only renders whatever string you put in `url` as a link; a 404 means the URL itself is wrong or the dashboard moved, which is a maintenance issue, not a dbt bug. The fix is to update `url` in the exposure YAML and republish docs. It is a reminder that `url` is unvalidated free text, so it can rot like any hardcoded link.

Q: How do you make the docs site reflect a model that was disabled with `enabled: false`?
A: A disabled model is excluded from the parsed graph, so it does not appear in `manifest.json` or the docs at all — there is nothing to render. If you want it documented but not built, that is a contradiction; you would instead keep it enabled and exclude it from runs via selection. Disabling removes it from lineage entirely, which is the intended behavior.

Q: Why can two analysts see different docs if one uses `dbt docs serve` locally and the other the CI-published site?
A: The local `dbt docs serve` hosts that developer's `target/` artifacts, which reflect their possibly-uncommitted local state, while the CI site reflects merged main-branch code. The two manifests can legitimately differ. The published CI build from main is the authoritative source of truth; local serve is for development only.

Q: How do you keep secrets and connection details out of the published docs artifacts?
A: Descriptions, `meta`, and docs blocks are authored by you, so never write credentials or sensitive values into them, and remember `manifest.json` can contain compiled SQL and configs — review what you publish. Generate with `--empty-catalog` in untrusted CI so no warehouse round-trip leaks schema details if that is a concern. Treat the published `manifest.json` as public-readable and scrub anything sensitive from descriptions.

Q: A model named with mixed case fails to match its YAML `description` on Snowflake — what is going on and how do you fix it?
A: Snowflake upper-cases unquoted identifiers, so a YAML `name` written in lower or mixed case may not match the catalog's stored uppercase column, leaving the description unattached. Set `quote: true` and write the name exactly as stored, or write the name in the case the warehouse uses. Case folding mismatches are the classic reason a description silently fails to render.

Q: How do exposures and the `state:` selector combine to test only what a changed model affects downstream?
A: After computing changed nodes with `dbt build -s state:modified+ --defer --state path/to/manifest`, the `+` pulls downstream nodes including exposures, so a CI run tests exactly the models whose change could reach a declared consumer like the EMT feed. State-based selection plus exposures scopes the run to the real blast radius. This is the efficient CI pattern for large projects.

Q: What does `--defer` add when you generate or build docs against a subset in CI?
A: `--defer` lets unselected, unbuilt nodes resolve against a previously built production manifest instead of being rebuilt, so a CI job can build only changed models while still referencing prod versions of their upstreams. This keeps slim CI runs correct without rebuilding the world. It pairs with `state:modified` to make incremental CI both fast and accurate.

Q: How would you document a column whose meaning depends on the fund's regulatory regime (PRIIPs vs MiFID II)?
A: Write the `description` to spell out the conditional meaning explicitly (for example "cost figure follows PRIIPs methodology for PRIIPs-scope funds, MiFID II ex-ante for others") and tag the model with the relevant regime in `meta`. Ambiguous regulatory fields are exactly where a vague description misleads an auditor. Stating the branching rule in words is the documentation that prevents misreporting.

Q: Why is it dangerous to rely on `catalog.json` row-count stats as documentation of data volume?
A: The catalog stats are a point-in-time snapshot captured at generate time and can be approximate or adapter-dependent, so treating them as authoritative volumetrics misleads anyone sizing or reconciling. They describe the relation when docs were last built, not now. For a daily-changing NAV table, the stat is stale almost immediately and should not be cited as a control figure.

Q: How do you document the relationship that every `isin` in the EMT output must exist in the security master?
A: Add a `relationships` test on the `isin` column pointing to the security-master model's key, which both enforces referential integrity and appears on the docs page as a documented guarantee. The test node in `manifest.json` records the dependency, extending lineage. So the docs show not just what `isin` means but that it is validated against the master.

Q: When publishing static docs, why might `dbt docs generate --static` be preferable to uploading `index.html` plus the JSON to a bucket?
A: `--static` inlines `manifest.json` and `catalog.json` into one `static_index.html`, so you serve a single self-contained file with no separate fetches, which sidesteps CORS, bucket-permission, and missing-artifact problems that break the multi-file build. One file is also far easier to access-control. The multi-file `index.html` only works if the page can successfully fetch its JSON from the same origin.

Q: An auditor wants proof of which exact code produced today's published EMT docs — how does the artifact set help?
A: The published `manifest.json` records the project state — node definitions, descriptions, configs, and the dbt invocation context — so pairing it with the git commit and CI run id that generated it gives a reproducible link from docs to source code. Publishing docs from CI on a tagged merge makes that chain auditable. The manifest is effectively a documentation snapshot tied to a specific build.

Q: Summarize the precise command sequence a credential-aware CI job runs to produce, comment, and publish EMT docs.
A: It runs `dbt build` (executing and testing models and, with `persist_docs` on, writing warehouse comments), then `dbt docs generate` (or `--static`) to produce the artifacts from the freshly built state, then a publish step uploading `static_index.html` (or `index.html` plus `manifest.json` and `catalog.json`) to the static host. The publish gates on build/tests passing so no red build is ever published. A credential-less variant swaps in `dbt docs generate --empty-catalog` and runs the catalog-rich build separately.

Q: A new joiner asks how to find every model that feeds the EMT exposure so they can read its docs — what selector do you give them?
A: Tell them to run `dbt ls -s +exposure:emt_daily_feed`, where the leading `+` walks all upstream ancestors of the exposure and `dbt ls` prints them as a list to open in the docs. This enumerates the exact gold, silver, and source models behind the EMT feed without guessing. It is the command-line counterpart to clicking the exposure node and tracing upstream in the lineage graph.

Q: Why does a docs-as-code workflow make the EMT feed's documentation defensible in a regulatory audit, where a wiki would not be?
A: Because every description, exposure, and test change is a reviewed, timestamped, attributable git commit tied to the SQL it documents, you can show an auditor exactly who approved each definition and when, and prove the docs matched the code at any point in history. A wiki has no such enforced linkage, review gate, or version trail, so its accuracy is unprovable. The audit value is not the docs site but the immutable, reviewed change history behind it.
