# Phase 7 · 6.4.2 Managed MLOps (Azure ML) — 100 self-test questions

Q: What is an Azure Machine Learning `workspace` and why is it called the top-level resource?
A: The `workspace` is the top-level Azure ML resource that groups all your ML artifacts — jobs, experiments, data assets, models, components, environments, and inference endpoints — under one Azure resource with one access-control and networking boundary. It keeps a history of every training run with logs, metrics, output, lineage metadata, and a snapshot of your scripts. Architecturally it is the unit you reason about for cost reporting, RBAC, and data isolation, which is why teams often create one workspace per project.

Q: Which four associated Azure resources does a workspace bring or auto-create, and what does each hold?
A: A workspace depends on an Azure Storage account (job logs, uploaded data, compute-instance notebooks), an Azure Key Vault (secrets and connection strings), Application Insights (endpoint monitoring and diagnostics), and optionally an Azure Container Registry (custom Docker images for environments). If you do not supply them at creation, Azure ML provisions them automatically. The gotcha for a regulated shop: these are real billable resources with their own RBAC and audit trails that your security team must also govern.

Q: A workspace was created and you see no Azure Container Registry attached. Is that a bug?
A: No — ACR is an optional dependency. A workspace can be created without ACR if you never build custom Docker images, and Azure ML can read from external registries. ACR is provisioned automatically the first time you build a custom environment image (for example deploying an AutoML model or triggering a data profile), so it can appear later in the resource group.

Q: What is a `compute target` in Azure ML?
A: A `compute target` is the designated compute resource or environment where a training job runs or a deployment is hosted — it can be your local machine, a managed cluster, serverless compute, or attached Kubernetes. The point of the abstraction is that you can change where code runs without changing the code itself. You reference the target by name in the job or deployment spec and Azure ML schedules the work onto it.

Q: What is the difference between a compute instance and a compute cluster?
A: A `compute instance` is a single-node, personal cloud workstation (a managed VM with notebooks and tooling) that one data scientist uses for development. A `compute cluster` (AmlCompute) is a single- or multi-node managed cluster that autoscales up when you submit a job and back down when idle. Use the instance for interactive work and the cluster for scaled or distributed training and batch jobs.

Q: What is `serverless compute` in Azure ML and what problem does it solve?
A: Serverless compute lets Azure ML provision and tear down training compute on demand for a job without you creating or managing a named `compute cluster`. It offloads the compute lifecycle — sizing, scaling, and idle cleanup — to the platform, so there is no cluster left running with the wrong min-node setting. It is the recommended default when you do not have a specific reason to manage a persistent cluster yourself.

Q: How does a compute cluster avoid charging you when no jobs are running?
A: Set the cluster's minimum node count to `0` so it scales all the way down to zero nodes between jobs; it then autoscales up to the maximum when a job is submitted. While at zero nodes you pay no VM compute. The standard gotcha is leaving a non-zero minimum, which silently bills idle nodes around the clock.

Q: What is the cost trap with a compute instance left running overnight?
A: A `compute instance` is a persistent single-node VM that bills for compute hours the entire time it is running, not just while you are using it. The fix is to enable idle shutdown (`idle_time_before_shutdown` / studio idle-shutdown setting) so it stops automatically. Note that even when stopped you still pay for the attached OS disk, public IP, and standard load balancer — only the compute hours stop.

Q: What is an Azure ML `environment` and why does it matter for reproducibility?
A: An `environment` is a versioned definition of the software context a job or deployment runs in — base Docker image plus Conda/pip dependencies — that Azure ML materializes into a container image. Because it is a registered, versioned asset, the same environment can be reused across training, batch, and online deployment, which is what gives you reproducible runs and a clean audit story. Pinning the environment version is how you guarantee that a redeployed model runs the exact stack it was validated on.

Q: How does Microsoft Entra ID RBAC attach to a workspace?
A: The workspace is an Azure resource, so Azure RBAC role assignments are scoped to it (or to a parent resource group / subscription) using Microsoft Entra ID identities and groups. You assign built-in or custom roles at the workspace scope to control who can create jobs, manage compute, register models, or deploy endpoints. Best practice is to assign roles to Entra security groups rather than individuals so the same group governs related resources.

Q: Name two Azure ML built-in roles beyond Reader/Contributor and what each grants.
A: `AzureML Data Scientist` grants full actions inside the workspace (run jobs, register models, deploy) but cannot create or delete compute or change the workspace itself. `AzureML Compute Operator` grants permission to create, manage, and delete compute resources but not to run experiments. Splitting them lets a platform team own compute provisioning while data scientists consume it, which is exactly the least-privilege separation a regulated audit expects.

Q: How does networking isolation attach to a managed workspace?
A: You can place the workspace behind a managed virtual network (managed VNet) and reach it and its associated resources over private endpoints instead of public IPs, with outbound rules controlling what the managed compute may call. This means training compute and endpoints communicate over the private network rather than the public internet. For a fund administrator, this is the control that keeps NAV-model traffic and data off public endpoints and inside your network boundary.

Q: What is an Azure ML `pipeline`?
A: An Azure ML pipeline is a graph of reusable steps (components) — for example data prep, train, evaluate, register — that Azure ML schedules onto compute targets, with caching, lineage, and parameterization. It is Azure ML's own orchestration construct for ML workflows. The architect's question is not whether it works but whether it should orchestrate jobs your existing Dagster already governs.

Q: Where do Azure ML pipelines and Dagster overlap?
A: Both are DAG orchestrators that schedule steps, pass artifacts between them, cache results, and capture lineage — so each can run a train-evaluate-register sequence. The overlap is real enough that running both naively gives you two schedulers, two retry policies, and two lineage stores for the same work. That doubled cost and split audit trail is precisely the trap the lesson warns against.

Q: State a clean boundary rule between Dagster and Azure ML pipelines.
A: A defensible rule is "Dagster owns data, Azure ML owns training jobs": Dagster orchestrates ingestion, transformation, and quality gates and then triggers an Azure ML job, while Azure ML internally orchestrates the training/registration steps that need its compute and registry. This keeps one system of record for data lineage and one for model lineage, instead of interleaving them. The key is that exactly one orchestrator owns each handoff so retries and lineage are unambiguous.

Q: What are the two places Azure ML pipelines and Dagster would fight over orchestration?
A: First, the data-to-training handoff: both want to own the trigger and lineage of the step that turns curated data into a training run. Second, scheduling and retries: if both are scheduled on the same cadence, a failure can be retried by both, producing duplicate jobs and conflicting run state. You resolve both by designating a single owner of the handoff and making the other system a pure caller.

Q: What is a `registered model` in Azure ML?
A: A registered model is a versioned model asset stored in the workspace model registry, identified by name and version, with metadata and lineage back to the job that produced it. Registration is what lets you reference an immutable model version in a deployment and roll forward or back deterministically. It is the Azure ML equivalent of the MLflow Model Registry stage you would otherwise run yourself.

Q: What is the difference between an `endpoint` and a `deployment` in Azure ML?
A: An endpoint is the stable, durable, authenticated URL clients call; a deployment is the actual set of resources and compute (model + scoring script + environment + instances) that performs inference behind it. One endpoint can hold several deployments, and the endpoint has a routing mechanism that directs requests among them. This separation is what enables blue-green rollout: you add a new deployment under the same URL.

Q: What is a managed online endpoint and when do you use it?
A: A managed online endpoint serves real-time, synchronous, low-latency inference over a stable HTTPS URL, with Azure ML managing the underlying VM compute, scaling, and OS. Use it when requests must be answered in a short time, the inputs fit in the HTTP payload, and you want to scale by request volume. It is the right fit for an interactive scoring service, not for scoring millions of files overnight.

Q: What is a batch endpoint and when do you use it?
A: A batch endpoint operationalizes long-running asynchronous inference: invoking it creates a batch job that scores large volumes of data distributed across many files, reading from a storage account or data asset. Use it for expensive models, pipeline reuse, large parallelizable datasets, and when you have no low-latency requirement. For most fund workloads — scoring a portfolio file or a day's transactions in bulk — a batch endpoint is the natural fit.

Q: Your NAV-anomaly model scores one overnight file of all funds. Online or batch endpoint?
A: Batch. The workload is asynchronous, file-based, large, parallelizable, and has no millisecond-latency requirement — exactly the batch-endpoint profile. A managed online endpoint would mean paying for always-on instances to serve a job that runs once a night, which is the wrong cost shape.

Q: How does autoscaling differ between online and batch deployments?
A: Online deployments autoscale based on resource utilization — CPU, memory, request rate — adding or removing instances to meet live traffic. Batch deployments scale based on the number of jobs/items to process, spinning cluster instances up for the job and down afterward. The mechanisms differ because one is reacting to a request stream and the other to a finite work queue.

Q: Can a managed online endpoint scale its deployment to zero instances?
A: No — managed online deployments cannot scale compute to zero; at least the provisioned instances stay running (and billing) so the endpoint can answer instantly. Batch deployments, by contrast, do scale to zero between jobs because there is nothing to keep warm. This is the core reason an idle online endpoint is the classic "expensive thing that idles overnight."

Q: How does traffic split enable safe rollout under one endpoint?
A: You deploy the new model as a second deployment under the same online endpoint and set the endpoint's traffic allocation, for example 90% to the current deployment and 10% to the new one, then shift the percentages as confidence grows. Because the URL never changes, clients are unaffected during the rollout. This is Azure ML's native blue-green / canary mechanism.

Q: What `az ml` v2 command updates the live traffic split on an online endpoint?
A: `az ml online-endpoint update --name <endpoint> --traffic "<deployment>=<percent>"`, for example `--traffic "blue=90 green=10"`, with resource group and workspace supplied. The percentages across active deployments must sum to 100. You ramp the new deployment up gradually by re-running the update with shifting percentages.

Q: What does mirror traffic do, and how does it differ from a traffic split?
A: Mirror traffic copies a percentage of live requests to a candidate deployment for validation without returning that deployment's responses to callers — it is a shadow test, set via `az ml online-endpoint update --mirror-traffic "green=10"`. A traffic split, by contrast, actually serves a share of users from the new deployment. Mirroring lets you validate latency and errors on real traffic before any user sees the new model's output.

Q: What does `--all-traffic` do on `az ml online-deployment create`?
A: It allocates 100% of the endpoint's traffic to the newly created deployment as part of creating it, so you do not need a separate `online-endpoint update` call. It is convenient for the first deployment under an endpoint. For a safe blue-green rollout you would normally omit it and start the new deployment at a small percentage instead.

Q: A managed online endpoint returns HTTP 429. What does that mean and what do you check first?
A: HTTP 429 is "Too Many Requests" — the deployment is throttling because incoming load exceeds the capacity of the provisioned instances. First check the instance count and SKU versus the request rate, and whether autoscaling is configured and has headroom (max instances). Online endpoints manage overcapacity by throttling, so the fix is more or larger instances or proper autoscale rules, not retry hammering.

Q: How do online and batch endpoints handle overcapacity differently?
A: Online endpoints throttle — excess requests are rejected (HTTP 429) because real-time callers cannot wait. Batch endpoints queue — extra work waits for instances to free up, capped by the cluster's maximum instance count. The difference reflects the latency contract: real-time has none to spare, batch can absorb delay.

Q: What authentication options does a managed online endpoint support?
A: A managed online endpoint supports key-based auth and, as a preview capability, Microsoft Entra ID (token) authentication. Key auth gives a static key in the request header; Entra ID auth uses short-lived tokens tied to identities and RBAC. For a regulated deployment, Entra-based auth is preferable because it ties every call to an auditable identity rather than a shared secret.

Q: What authentication do batch endpoints use?
A: Batch endpoints authenticate with Microsoft Entra ID rather than static keys. Because a batch invocation kicks off a job under your identity, it naturally rides on token-based auth and RBAC. This means access to run batch scoring is governed by the same Entra role assignments as the rest of the workspace.

Q: What does a deployment's scoring script do, and when do you need one?
A: A scoring script defines how requests are turned into model inputs and how predictions are returned — typically an `init()` that loads the model and a `run()` that handles each request. You need one for custom (non-MLflow) models on online or batch deployments. MLflow models can deploy without a custom scoring script because the inference behavior is inferred from the logged model flavor.

Q: Why can you deploy an MLflow model without writing a scoring script?
A: An MLflow model carries a standard flavor and signature that Azure ML knows how to load and invoke, so the platform generates the serving glue for you. This is the practical payoff of logging models in MLflow format from your existing tracking stack — they drop into Azure ML online or batch endpoints with no bespoke `score.py`. It also keeps the input/output contract consistent across tools.

Q: What is the relevance of MLflow to mapping your OSS stack onto Azure ML?
A: Azure ML's tracking and model registry are MLflow-compatible: it implements the MLflow tracking API and accepts MLflow-format models, so your existing MLflow experiments and models map onto Azure ML's experiments and model registry rather than needing a rewrite. This is the key fact for the build-vs-buy table — tracking and registry are not net-new concepts but a managed home for ones you already use. It reduces the lock-in cost of adopting the managed registry.

Q: Which serving servers can back an Azure ML online deployment?
A: Online deployments can use the Azure Machine Learning Inferencing Server, NVIDIA Triton, or a custom server via bring-your-own-container (BYOC). The inference server is the technology that accepts requests, runs the model, and formats responses. Choosing Triton or BYOC matters when you need GPU-optimized serving or a serving stack the default does not provide.

Q: What is the Responsible AI dashboard in Azure ML?
A: It is a single, customizable UI (with CLI v2 / SDK v2 components) that bundles several mature open-source responsible-AI tools — error analysis, fairness and model overview, data analysis, interpretability, counterfactual what-if, and causal inference — so you can debug a model and make data-driven decisions in one place. It is built on packages like Error Analysis, Fairlearn, InterpretML, DiCE, and EconML. It targets the "identify, diagnose, mitigate" model-debugging loop.

Q: Which open-source package powers the dashboard's model-interpretability component?
A: InterpretML — it generates human-understandable global explanations (which features drive overall behavior) and local explanations (why one specific prediction was made). Knowing the underlying package matters for an architect because it means the capability is reproducible outside Azure ML, reducing lock-in. The same explanations could be produced from the OSS library directly.

Q: Which package backs the fairness component, and what does it assess?
A: Fairlearn backs the fairness component; it defines groups by sensitive attributes (for example sex, race, age) and measures how prediction values and performance metrics differ across those groups. It surfaces disparities so you can decide on mitigation. Crucially it assesses group fairness on your chosen attributes — it does not invent the notion of "fair" for your domain.

Q: What does the counterfactual what-if component do and which package provides it?
A: Provided by DiCE, it generates the minimal feature changes that would flip a model's prediction — for example "this loan would have been approved with $10k more income and two fewer open cards" — and supports interactive what-if perturbations. It answers "what is the smallest change to get a different outcome." It is a diagnosis and explanation tool, not a guarantee that the model is correct.

Q: What does the causal inference component estimate, and what package powers it?
A: Powered by EconML, the causal inference component estimates heterogeneous treatment effects from historical observational data — how a real-world outcome would change under an intervention — and helps construct policies for which cohorts benefit. It is about causation versus mere correlation. Its outputs are estimates from observational data, which carry strong assumptions and are not proof of causality.

Q: What does the Responsible AI dashboard offer a regulated buyer?
A: It gives a consolidated, exportable assessment of model errors, fairness across sensitive groups, data representation, explanations, and what-if/causal analysis, plus a PDF scorecard you can attach to compliance reports and share with risk officers and auditors. It standardizes and documents the model-review evidence in one place. That documentation and reproducibility is its real value to a regulated review.

Q: What does the Responsible AI dashboard NOT prove for a regulated buyer?
A: It does not prove your model is compliant, fair in your legal sense, accurate enough for production, or free of harm — it is a debugging and evidence-gathering toolset, not a certification. Fairlearn measures disparities on the attributes you pick; it cannot decide which disparities are lawful or acceptable. The judgement, sign-off, and accountability remain with your humans and your governance process; the dashboard only informs them.

Q: What are the stated data limitations of the Responsible AI dashboard?
A: It currently supports regression and binary/multiclass classification on tabular structured data, MLflow scikit-learn models that implement `predict()`/`predict_proba()`, pandas DataFrames in Parquet format, up to about 5,000 visualized data points, and datasets under ~10,000 columns; it does not support AutoML MLflow models. So deep-learning, unstructured, or AutoML models fall outside it. An architect should not promise it covers a model type it cannot ingest.

Q: How is a managed online endpoint billed?
A: You are billed continuously for the VM instances provisioned for each deployment for as long as the deployment exists and is running — billing is per provisioned instance, independent of how many requests arrive. There is no scale-to-zero, so an endpoint serving zero traffic still costs the full provisioned compute. That is why an idle real-time endpoint is a top cost-review item.

Q: Does Azure ML add a service surcharge on top of the underlying compute?
A: For compute, no — you pay for the underlying Azure resources (VM compute for clusters, instances, and online-endpoint deployments, plus storage, ACR, Key Vault, Application Insights, networking). Azure ML's orchestration of those resources is not separately marked up per node-hour. The exception is standard (serverless foundation-model) deployments, which carry their own per-token / per-minute billing model.

Q: How is a compute cluster billed, and how do you minimize that cost?
A: A compute cluster bills per node for the time nodes are running; with a minimum of `0` nodes it scales to zero between jobs so you pay only while a job runs. You further cut cost by sizing the SKU correctly and capping max nodes. Switching to serverless compute removes the lifecycle-management mistake of a forgotten non-zero minimum entirely.

Q: What still costs money after you stop a compute instance?
A: Stopping a compute instance halts compute-hour billing, but you continue to pay for its attached OS disk, its public IP, and its standard load balancer. So "stopped" is not "free." For long-idle instances the cleaner move is to delete and recreate, or use idle-shutdown to stop automatically rather than relying on people remembering.

Q: What are the main cost rows you would put in an Azure ML build-vs-buy table?
A: Training compute (cluster/serverless node-hours), interactive compute instances, managed online-endpoint instances (always-on), batch-endpoint job compute (transient), storage, ACR/Key Vault/App Insights, and networking egress. Each row maps to an underlying Azure meter, not an Azure ML markup. The always-on online-endpoint row and forgotten compute instances are where the surprises live.

Q: Why does the lesson say two registries and two orchestrators is the failure mode to avoid?
A: If you adopt Azure ML's pipelines and model registry without retiring the equivalent Dagster orchestration and MLflow registry, you pay for and operate both, and your lineage/audit trail splits across two systems — so no single place answers "which model is in production and how was it built." For a regulated fund administrator that split is itself an audit finding. The architect's job is to pick one owner per concern.

Q: Where does Azure ML genuinely add value over running OSS-on-AKS yourself?
A: Managed online endpoints remove the burden of running, patching, scaling, and securing the serving infrastructure, and the whole stack integrates with Microsoft Entra RBAC, private networking, and Azure monitoring out of the box. You trade some flexibility for far less ops toil and a tighter Azure-native security and audit story. For a team without dedicated platform engineers, that managed layer is the buy case.

Q: Where does Azure ML constrain you compared with OSS-on-AKS?
A: You give up portability and fine-grained control: managed endpoints run on Azure's compute and serving stack, you are bound to supported VM SKUs and Azure regions, and moving the workload to another cloud means re-platforming. OSS-on-AKS keeps the serving layer portable and tunable at the cost of running it yourself. The trade is managed-convenience-plus-lock-in versus flexibility-plus-ops-burden.

Q: Which Capstone-7 components map cleanly onto an Azure ML equivalent?
A: Experiment tracking and model registry map onto Azure ML's MLflow-compatible tracking and model registry; model serving maps onto online and batch endpoints; training orchestration maps onto Azure ML pipelines; and environments/compute map onto Azure ML environments and compute targets. These are the rows where "buy" is plausible because the concept is the same. The mapping is what makes the build-vs-buy conversation concrete.

Q: Which Capstone-7 components have no first-class Azure ML equivalent?
A: A general-purpose data orchestrator (your Dagster), a feature/vector store, and LLM evals and tracing are not native Azure ML primitives — you would pair Azure ML with adjacent Azure services or keep the OSS tools. So even with an Azure ML licence, several rows stay open-source. Recognizing the gaps prevents you from claiming Azure ML replaces the whole stack.

Q: You have an Azure ML licence. Which components would you keep open-source and why — give one example.
A: Keep Dagster for data orchestration because it owns ingestion and quality across many non-ML pipelines that Azure ML pipelines do not cover, and a single data-lineage system is worth more than forcing it into Azure ML. You might also keep an OSS vector store and your evals/tracing tooling for portability. Each kept component should earn its place with a one-line justification: ops fit, portability, or lineage ownership.

Q: How would you brief a platform team on what changes if they adopt Azure ML?
A: Tell them serving and the model registry can move to managed endpoints and the Azure ML registry with Entra RBAC and private networking, removing serving ops toil; training jobs can run on managed/serverless compute. What does not change is that data orchestration stays in Dagster, and they must govern the workspace's associated resources (storage, Key Vault, ACR, App Insights) and watch always-on endpoint cost. The net is fewer servers to patch but a new Azure-native cost and RBAC surface to manage.

Q: Why is "where Azure ML overlaps with the Dagster+MLflow stack" the core architect question?
A: Because the overlap is precisely tracking, registry, and ML-job orchestration — adopt the managed versions blindly and you run two of each, doubling cost and splitting the audit trail. The architect must say, per concern, which system is the owner and which is retired or relegated to a caller. Getting that boundary right is the difference between a coherent platform and an expensive duplicate.

Q: Why might idle managed online endpoints be the biggest hidden cost in an Azure ML build-vs-buy analysis?
A: Because they bill per provisioned instance continuously and cannot scale to zero, so an endpoint that serves a low-traffic internal model still costs the full compute when the team goes home. A handful of forgotten real-time endpoints can dwarf the training spend. The mitigation is to prefer batch endpoints for non-interactive scoring and to right-size or decommission low-traffic online endpoints.

Q: What is a `data asset` in Azure ML and how does it relate to lineage?
A: A data asset is a versioned reference to data (a file, folder, or table backed by a datastore) registered in the workspace so jobs can consume it by name and version. Because jobs record which data-asset version they read, the workspace can trace a model back through its training data. That versioned lineage is what lets you reproduce a NAV model and show an auditor exactly which inputs produced it.

Q: What is a `datastore` versus a `data asset`?
A: A datastore is the registered connection to an underlying storage service (for example a blob container or ADLS) including how to authenticate to it; a data asset is a versioned pointer to specific data accessed through a datastore. The datastore answers "how do we connect" and the data asset answers "which data, which version." Separating them keeps credentials in one managed place while data references stay versioned and shareable.

Q: What is an Azure ML `component` and how does it relate to pipelines?
A: A component is a self-contained, versioned, reusable pipeline step with a defined interface (inputs, outputs, command, environment), the building block from which pipelines are assembled. Components let you reuse, say, a "validate-EMT-file" step across multiple pipelines. They are the Azure ML analogue of reusable ops/assets in your existing orchestrator.

Q: What is an Azure ML `registry` (as distinct from the model registry) and when is it useful?
A: An Azure ML registry is an organization-level resource for sharing assets — models, components, environments, data — across multiple workspaces and regions, separate from any single workspace's per-workspace model registry. It is useful when several teams or environments (dev/test/prod workspaces) must promote the same validated model. It centralizes governance of shared assets above the workspace boundary.

Q: A team wants to promote a validated model from a dev workspace to a prod workspace. What Azure ML feature supports this?
A: An Azure ML registry lets you publish a model (and its component/environment) once and consume it from multiple workspaces, so the prod workspace references the same validated artifact the dev workspace produced. This avoids copy-paste promotion that breaks lineage. It gives you one cross-workspace system of record for what was approved.

Q: What does it mean that an endpoint provides a "stable and durable URL"?
A: The endpoint exposes a fixed hostname like `endpoint-name.region.inference.ml.azure.com` that does not change when you swap the deployment behind it. Clients integrate against that URL once and keep working through model upgrades. That stability is what makes blue-green rollout transparent to callers — the model changes, the address does not.

Q: Why does separating endpoint from deployment matter for a regulated change process?
A: Because you can stage a new model as a separate deployment, validate it with mirror traffic or a small split, and promote or roll back by changing traffic percentages — all without touching the client-facing URL or doing a risky in-place swap. Each step is an auditable, reversible action. That controlled, reversible rollout is exactly what change management in a fund administrator demands.

Q: Can one online endpoint host two models written in different frameworks?
A: Yes — each deployment under the endpoint has its own model, environment, scoring code, and even compute (CPU vs GPU), so a PyTorch deployment and a TensorFlow deployment can sit under the same endpoint and be traffic-split. This is the canonical Azure ML example of gradually testing a new model against the old. The endpoint's routing layer directs requests to whichever deployment the traffic split selects.

Q: What is a Kubernetes (AKS/Arc) compute target in Azure ML and why would a regulated firm consider it?
A: It is an attached Kubernetes cluster — AKS in Azure or Arc-enabled Kubernetes on-premises/edge — that Azure ML can target for training and inference instead of fully managed compute. A regulated firm might use it to run inference inside an existing Kubernetes estate, on-premises, or in a specific network boundary it already controls. It trades the simplicity of managed endpoints for placement control and portability.

Q: What is the trade-off between managed online endpoints and Kubernetes online endpoints?
A: Managed online endpoints hide the cluster — Azure runs and scales the VMs for you, with the least ops effort and tight Azure-native integration. Kubernetes online endpoints give you control over the cluster, node pools, and placement (including on-prem via Arc) at the cost of operating Kubernetes yourself. The choice is convenience-and-lock-in versus control-and-portability — the same axis as the whole buy-vs-build question.

Q: What does "managed network isolation" give an endpoint?
A: It lets the endpoint and its compute run inside a managed virtual network with private connectivity, so inference traffic and dependency calls stay off the public internet. Batch endpoints support it with some additional configuration. For fund data, this is the control that keeps model inputs and outputs within your network perimeter and out of public reach.

Q: A managed online endpoint deployment fails health checks at startup. What do you check first?
A: First check the scoring script's `init()` — it most often fails to load the model (wrong path, missing artifact) or the environment lacks a dependency the script imports. Read the deployment logs to see the exception. Test the deployment locally first (online deployments support local testing) before pushing to managed compute, which is faster than debugging in the cloud.

Q: Why is local testing of online deployments useful, and is it available for batch?
A: Online deployments support local testing — you can run the scoring container on your machine to catch `init()`/`run()` and dependency errors before consuming cloud compute, which shortens the debug loop. Batch deployments do not offer local testing, so you validate them by running small jobs. Catching errors locally for online endpoints saves both time and the cost of repeated failed cloud deployments.

Q: How does Application Insights fit into monitoring an online endpoint?
A: Application Insights, one of the workspace's associated resources, collects diagnostics and telemetry from inference endpoints so you can monitor latency, request volume, and errors. It is where you watch for the 429 throttling or latency regressions that signal under-provisioning. For a regulated deployment it is part of the operational evidence that the service met its SLA.

Q: What is the PDF scorecard that accompanies the Responsible AI dashboard?
A: It is an exportable PDF report that captures the dashboard's responsible-AI metadata and insights — performance, fairness, and data findings — so you can share them offline with product, compliance, and audit stakeholders. It turns interactive analysis into a fixed artifact you can attach to a compliance review. It is the deliverable that makes the dashboard useful to people who never open the studio.

Q: Which roles does Microsoft position as users of the Responsible AI dashboard?
A: Data scientists and ML engineers debugging models pre-deployment, product managers and business stakeholders reviewing models, risk officers assessing fairness and reliability, and professionals in regulated spaces presenting models to regulators and auditors. The spread of roles is the point — it is a shared-evidence tool, not just a data-science gadget. That cross-functional review is exactly the governance flow a fund administrator needs.

Q: Why does the Responsible AI dashboard cap visualized data at ~5,000 points, and what should you do?
A: The dashboard renders up to about 5,000 data points for performance and interactivity, so you must downsample larger datasets before passing them in. The implication is that the visualized fairness/error view is a sample, not the full population. You should downsample representatively and be explicit in your review that conclusions rest on a sampled view.

Q: How would you defend keeping your own MLflow registry even with an Azure ML licence?
A: If your model lineage and promotion already run through a tool-agnostic MLflow registry that serves non-Azure consumers too, keeping it preserves portability and a single registry of record across clouds and tools. Azure ML's registry is MLflow-compatible, so you can still push models there for serving without making it the source of truth. The one-line justification is "portability and one system of record outweigh the managed convenience here."

Q: How would you defend moving serving to Azure ML managed endpoints even though you keep other OSS tools?
A: Serving is the component where managed value is highest: managed online and batch endpoints remove infrastructure ops, integrate with Entra RBAC and private networking, and give safe rollout via traffic split out of the box. The portability cost is real but bounded to the serving layer. So "buy serving, keep orchestration and registry open" is a coherent split.

Q: What is the difference between a "training compute target" and an "inference compute target" in Azure ML?
A: Training compute targets (compute clusters, serverless, compute instances, attached Kubernetes/VMs) run jobs that produce models. Inference compute targets (managed-compute online/batch endpoints or Kubernetes) host deployed models to answer requests. The same physical SKUs may be involved, but the billing shape and lifecycle differ — training compute is transient per job, online inference compute is provisioned and persistent.

Q: When choosing an online-endpoint VM SKU, what does the "scale up then scale out" guidance mean?
A: First pick a single VM large enough — start around 150% of the model's RAM, profile, and find the size that meets your latency — then increase the instance count to handle concurrency. In other words, get one instance right before adding more. Doing it in reverse risks paying for many undersized instances that still miss your latency target.

Q: Why does a workspace warn that you cannot move it to a different subscription or tenant?
A: The workspace and its associated resources are wired to a specific subscription and Entra tenant for identity, RBAC, and billing, so a move would break those bindings and can cause errors. The practical consequence is that subscription/tenant placement is a design-time decision, not something to fix later. Plan the regulatory and billing boundary before you create the workspace.

Q: What does deploying an AutoML model do to your workspace's ACR, and why care?
A: Deploying an AutoML model triggers Azure ML to build a custom Docker image, which provisions and uses the Azure Container Registry dependency if it was not already there. You care because that adds a billable resource with its own RBAC and, in subscriptions that mandate resource tags, ACR creation can fail since Azure ML cannot set tags on the ACR it creates. So a tag-enforcing policy can block an otherwise-fine deployment.

Q: Your AutoML model deployment fails to create the ACR. What policy should you suspect?
A: Suspect a subscription policy that requires tags on all resources: Azure ML cannot apply tags to the ACR it auto-creates, so the creation fails under such a policy. The fix is to pre-create a compliant ACR and attach it, or adjust the policy. This is a common governance-versus-automation clash in regulated tenants.

Q: How does the v2 control plane (CLI `ml` extension and SDK `azure-ai-ml`) relate to v1?
A: v2 is the current, supported control plane via the `ml` Azure CLI extension and the `azure-ai-ml` Python SDK; v1 (`azure-cli-ml`, the old SDK) is deprecated. CLI v1 support ended on 30 September 2025 and SDK v1 support ends 30 June 2026. New work, including all the endpoint and traffic commands here, should target v2.

Q: A tutorial uses `az ml` commands but your scripts use `az ml` with different syntax and they fail. What is the likely cause?
A: Likely a v1-versus-v2 mismatch: the old `azure-cli-ml` extension used different command shapes than the current `ml` extension. Confirm which extension is installed (`az extension list`) and use the v2 `ml` extension, since v1 is deprecated and out of support. Aligning on v2 also gets you the online-endpoint traffic and mirror commands.

Q: Why is the model registry's version immutability important for a regulated model?
A: Because deployments reference a specific name and version, and that version is immutable, you can prove exactly which artifact answered production traffic on a given date and roll back deterministically. Mutating "the latest model" in place would destroy that audit trail. For a NAV or risk model under regulatory scrutiny, immutable versions are what make "which model was live" answerable.

Q: How does Azure ML capture lineage from data to model to deployment?
A: A job records the data-asset versions and environment it consumed and the model version it produced; the registered model links back to that job; and a deployment references a specific model version. Following those links traces deployment to model to training job to data. That end-to-end chain is the lineage an auditor walks to validate a regulated model.

Q: For a Luxembourg fund administrator, why does Entra RBAC integration weigh heavily in the Azure ML buy case?
A: Because access to models, compute, and endpoints can be governed by the same Microsoft Entra identities, groups, and conditional-access policies already used across the firm, giving one auditable identity boundary instead of a separate auth system to reconcile. Regulators expect least-privilege access tied to named identities. Native Entra RBAC turns that expectation into a configuration rather than a custom build.

Q: How does DORA-style operational-resilience concern bear on choosing managed endpoints versus OSS-on-AKS?
A: A managed endpoint shifts patching, scaling, and availability of the serving layer to Microsoft, which can simplify your resilience controls but makes the provider a third-party dependency you must assess and document under operational-resilience rules. OSS-on-AKS keeps the resilience controls and the failure surface in your hands. The architect documents which risks are transferred and which are retained, either way.

Q: Why is "no surcharge on compute" an important line to get right in the build-vs-buy table?
A: Because if you wrongly assume Azure ML marks up every node-hour, you will overstate the buy cost and may reject managed serving for the wrong reason; the real cost driver is the shape of usage — always-on online endpoints and forgotten compute instances — not a platform tax on compute. Standard serverless foundation-model deployments are the genuine exception, billed per token. Getting the meter model right keeps the comparison honest.

Q: What is the difference in cost timing between an online deployment and a batch deployment?
A: An online deployment provisions and bills compute at deployment time and keeps billing while it exists, regardless of traffic. A batch deployment consumes no compute at deployment time — cost is incurred only when a job runs, capped at the cluster's maximum instances, and queued jobs cost nothing. So batch cost tracks actual work while online cost tracks provisioned readiness.

Q: How do batch deployments handle a large input split across many files?
A: A batch endpoint parallelizes the work across the cluster's instances, scaling node count by the volume of jobs/items, processing files concurrently, and scaling back to zero when done. This is why it suits scoring a whole day of transactions or a full fund universe. The parallelism and scale-to-zero are exactly what an overnight bulk-scoring job wants.

Q: Why might you NOT buy an Azure ML feature store / vector store row even with a licence?
A: Because Azure ML has no first-class general feature store or vector store primitive, so that row maps to a separate Azure service or an OSS tool you keep — there is no managed Azure ML component to "buy" for it. Forcing it would mean adopting an adjacent product, not Azure ML itself. The honest table entry is "keep OSS / use adjacent Azure service," not "Azure ML equivalent."

Q: What does it mean that batch deployments can use low-priority compute but online cannot?
A: Batch deployments can run on low-priority (evictable, cheaper) VMs because a job can tolerate interruption and retry, whereas online endpoints serving live requests cannot accept eviction and so do not use low-priority compute. This makes batch scoring substantially cheaper for interruptible work. It is another reason to push non-interactive scoring to batch.

Q: How would you summarize the boundary rule you give a platform team in one sentence?
A: "Dagster owns and triggers all data movement and quality; Azure ML owns model training, registry, and serving — neither schedules the other's domain, and the handoff between them has exactly one owner." Said plainly, this prevents the double-orchestrator, double-registry trap. It is the single rule that keeps cost and audit trail single-sourced.

Q: What is the risk of letting Azure ML pipelines call back into your data layer?
A: If Azure ML pipelines start orchestrating ingestion or transformation, you split data lineage between Dagster and Azure ML, so no single system answers "how was this dataset built," and retries can run in both. That undermines the audit trail a regulator expects. Keep data orchestration on one side of the boundary and have Azure ML consume finished data assets only.

Q: How does the responsible-AI scorecard help in a regulator meeting specifically?
A: It packages fairness, error, and data-representation findings into a fixed PDF you can hand to a regulator or auditor as documented evidence of the model assessment you performed. It moves the conversation from "trust us" to a shared artifact. But you must still articulate the judgements — the scorecard documents analysis, it does not substitute for accountable sign-off.

Q: An online endpoint's latency is fine but cost is too high at low traffic. What is the structural fix?
A: The structural fix is to stop using an always-on online endpoint for low-traffic, latency-tolerant scoring and move that workload to a batch endpoint (which scales to zero between jobs) or consolidate it onto a shared endpoint. Online endpoints cannot scale to zero, so trimming instances only goes so far. Match the endpoint type to the traffic and latency profile rather than over-paying for readiness.

Q: How do compute instances and compute clusters differ in audit and shared-use terms?
A: A compute instance is typically a personal, single-node workstation tied to one user, whereas a compute cluster is a shared, multi-node resource that many jobs reuse. For governance, clusters are the shared pool the `AzureML Compute Operator` provisions and meters per project. Instances are the per-developer resource where idle-shutdown discipline matters most.

Q: Why does Azure ML create a load balancer and public IP per compute instance and cluster, and what is the cost catch?
A: They are subresources Azure ML provisions so the compute can receive and manage traffic even while stopped. The catch is that these subresources keep billing after you merely stop the compute — only deletion stops their charges. So stopped-but-not-deleted compute still accrues disk, IP, and load-balancer cost.

Q: Summarize the one-line build-vs-buy verdict you would give for model serving.
A: Buy managed online/batch endpoints — they remove serving ops, give Entra RBAC and private networking, and provide native safe rollout — accepting bounded lock-in to Azure's serving stack and the need to control always-on cost. The justification is "lowest ops burden plus native security integration on the highest-toil component." Pair it with batch endpoints for non-interactive workloads to keep cost in check.

Q: Summarize the one-line build-vs-buy verdict you would give for data orchestration.
A: Keep open-source Dagster — it owns ingestion, transformation, and quality across many non-ML pipelines that Azure ML pipelines do not cover, and a single data-lineage system of record is worth more than forcing data work into Azure ML. The justification is "lineage ownership and breadth of coverage outweigh managed convenience." Azure ML pipelines stay confined to training-job orchestration, triggered by Dagster.
