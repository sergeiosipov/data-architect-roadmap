# Phase 7 · 6.6.2 RAG frameworks & architecture — 150 self-test questions

Q: What does RAG stand for, and in one sentence what does it do?
A: RAG stands for Retrieval-Augmented Generation: instead of relying only on an LLM's parametric memory, you retrieve relevant text from an external corpus at query time and feed it into the prompt as grounding context. The point is to answer from your own up-to-date, governed documents rather than from whatever the model happened to memorise during training, which both improves factuality and lets you cite sources.

Q: Name the canonical six stages of a RAG pipeline in order.
A: The canonical chain is load → chunk → embed → index → retrieve → generate, with retrieve usually expanded into hybrid retrieval plus reranking and generate producing citations. Each stage is an ordinary data-pipeline stage you already know how to govern, schedule, and test, which is exactly why a data architect is well placed to own RAG quality.

Q: Why is it accurate to call RAG "a data architecture problem at heart" rather than an ML problem?
A: Because the levers that actually move quality — how you load and parse documents, how you chunk them, how you index and refresh them, and how you fuse and rerank retrieval — are ingestion, indexing, and freshness concerns, not model-training concerns. The LLM at the end is largely interchangeable, while a bad parser or chunking choice silently degrades every answer, so the discipline is closer to ETL governance than to deep learning.

Q: In the load stage of RAG, what is a "loader" (a.k.a. reader) responsible for?
A: A loader (called a reader/connector in LlamaIndex) is responsible for ingesting a source — a PDF, web page, database table, S3 bucket — and turning it into `Document` objects carrying raw text plus metadata. It is the boundary where format-specific parsing happens, so a weak loader (e.g. one that mangles a PDF table) poisons everything downstream regardless of how good the embeddings or LLM are.

Q: What is the unit of text that gets embedded and indexed in a RAG system?
A: It is the chunk, called a `Node` in LlamaIndex: a bounded span of text (plus metadata and relationships) small enough to embed and to fit usefully in a prompt. Retrieval returns chunks, not whole documents, so the chunk boundary is the atomic unit of relevance and the single most under-rated design decision in the pipeline.

Q: What is an embedding in the RAG context, and what property makes it useful for retrieval?
A: An embedding is a fixed-length numeric vector produced by an embedding model that maps semantically similar text to nearby points in vector space. That property lets retrieval find passages by meaning rather than exact words, so a query about "ongoing charges" can match a chunk that says "OCF" or "fund expenses" even with zero shared keywords.

Q: What does a vector index do, and why not just compute similarity against every chunk at query time?
A: A vector index (e.g. an HNSW graph or IVF structure) stores embeddings in a structure that supports fast approximate nearest-neighbour search, returning the top-k closest vectors in sub-linear time. Brute-force cosine against every chunk is exact but O(n) per query and collapses on large corpora, so the index trades a little recall for the latency you need in production.

Q: Why does each RAG stage map onto governance practices you already use for data pipelines?
A: Because loading is ingestion, chunking is transformation, embedding/indexing is a materialisation, and freshness is incremental re-processing — each is a stage you can version, test, monitor for lineage, and re-run idempotently. Treating them as pipeline stages means you can apply data-quality checks, owners, and SLAs to RAG instead of treating it as an opaque ML black box.

Q: In a KID-extraction workflow, what is the concrete business risk if retrieval returns the wrong fund's fee table?
A: The system can confidently cite a fee or SRI value belonging to a different fund, and an LLM will fluently present it as fact. In a regulated fund-administration context that is not a typo — it is a mis-stated cost disclosure that can amount to a mis-sold product, which is precisely why retrieval-quality discipline is a compliance requirement, not an optimisation.

Q: What is "chunking" and why can't you just feed whole documents to the model?
A: Chunking is splitting documents into retrievable spans before embedding. You can't feed whole documents because embeddings degrade when a vector must represent many unrelated topics, retrieval becomes coarse, and the context window can't hold large documents anyway; chunking restores precision by making each indexed unit topically coherent.

Q: Describe fixed-size chunking and its main parameters.
A: Fixed-size chunking splits text into spans of roughly N tokens, usually with an overlap so context isn't lost at boundaries; in LlamaIndex the `SentenceSplitter` defaults to `chunk_size=1024` and `chunk_overlap=200` tokens. It is simple and fast and ignores document structure, which is both its appeal and its core weakness.

Q: What is the purpose of chunk overlap in fixed-size chunking?
A: Overlap repeats a tail of one chunk at the head of the next so that a sentence or fact straddling a boundary still appears intact in at least one chunk. Without overlap, a fact split across the cut line is embedded as two truncated halves and may be retrieved by neither query, silently dropping information.

Q: What is structure-aware (a.k.a. semantic or layout-aware) chunking?
A: Structure-aware chunking splits on the document's own structure — headings, sections, list items, table rows — so each chunk corresponds to a meaningful unit rather than an arbitrary token count. On a KID this means the "What are the risks?" section or a complete fee table stays together, which dramatically improves retrieval precision over blind fixed-size cuts.

Q: On structured PDFs like KIDs, what is the classic failure mode of naive fixed-size chunking?
A: Tables are where naive chunking dies: a fixed-size splitter cuts a fee or SRI table mid-row, so the header ends up in one chunk and the values in another, and the embedding of either fragment no longer means "this fund's ongoing charge is 1.2%". The retriever then either misses the table or returns a fragment the LLM cannot interpret correctly.

Q: Why does chunk size involve a precision-versus-recall trade-off?
A: Small chunks are topically tight, so they embed precisely and pinpoint the exact fact, but they fragment context and you may need many to answer one question. Large chunks preserve context and reduce fragmentation but dilute the embedding across multiple topics, lowering precision; the right size is corpus-specific and must be measured, not guessed.

Q: A RAG system retrieves the right document but gives a wrong number from its fee table. What chunking-related cause should you suspect first?
A: Suspect that the table was chopped by fixed-size chunking so the retrieved chunk contains a header without its values, or values without their row labels, leaving the LLM to guess the alignment. The fix is structure-aware or table-aware parsing that keeps each table (or row with its header) intact as one chunk.

Q: What is a "semantic chunking" strategy, distinct from fixed-size and structural?
A: Semantic chunking decides split points by measuring where the embedding similarity between consecutive sentences drops, cutting at topic boundaries rather than fixed counts or layout. It can produce more coherent chunks on prose but is slower and can behave unpredictably on tabular or templated documents like KIDs, so it is not a default for structured data.

Q: Why is metadata attached to each chunk important for retrieval quality and governance?
A: Chunk metadata (source filename, fund name/ISIN, section, page, document version) lets you filter retrieval (only this fund), build citations (page span), and trace lineage when an answer is wrong. Without it you cannot enforce "never return another fund's table" or answer the audit question "which document version produced this value".

Q: What is hybrid retrieval and why is it described as "the enterprise default, not an optimization"?
A: Hybrid retrieval runs both keyword (BM25) and dense vector search and fuses the results, because each covers the other's blind spot: vectors miss exact tokens like an ISIN or a specific defined term, and BM25 misses paraphrase. In enterprise corpora full of codes, names, and jargon, relying on vectors alone routinely misses exact-match queries, so hybrid is the baseline you start from rather than a late tweak.

Q: What is BM25 and what kind of matching does it do?
A: BM25 is a probabilistic bag-of-words ranking function that scores documents on term frequency and inverse document frequency with length normalisation; it is sparse, lexical, exact-token matching. It excels precisely where embeddings struggle — rare identifiers, acronyms, and exact phrases — which is why it is the keyword half of a hybrid retriever.

Q: Give a fund-data example where BM25 beats pure vector search.
A: A query for the exact ISIN `LU0123456789` or the LEI of a management company is a token-match problem: BM25 finds the chunk containing that exact string, whereas a dense embedding of a meaningless alphanumeric code carries little semantic signal and may rank an irrelevant chunk higher. For codes, identifiers, and exact defined terms, lexical search is more reliable than semantic.

Q: Conversely, give an example where vector search beats BM25.
A: A user asking "how risky is this fund?" should match a chunk titled "Synthetic Risk Indicator" or "risk and reward profile" that never uses the word "risky"; dense embeddings capture that the concepts are related while BM25, seeing no shared term, scores it near zero. Paraphrase and synonymy are the vector retriever's home turf.

Q: How are the two ranked lists from BM25 and vector search typically fused?
A: A common fusion is Reciprocal Rank Fusion (RRF), which scores each document by summing `1/(k + rank)` across the lists in which it appears, using a constant k commonly set to 60. RRF is attractive because it combines rankings without needing the underlying scores to be on the same scale, which BM25 and cosine emphatically are not.

Q: Why can't you simply add a BM25 score and a cosine similarity together to fuse them?
A: Because they live on incompatible scales — BM25 is an unbounded TF-IDF-style score while cosine similarity is roughly in [-1, 1] — so a raw sum lets whichever metric has the larger magnitude dominate arbitrarily. Rank-based fusion like RRF sidesteps this by using only the position in each list, not the raw score.

Q: What is reranking, and where does it sit in the pipeline?
A: Reranking is a second, more expensive scoring pass applied to the top-N candidates returned by first-stage retrieval, reordering them so the most relevant rise to the top before they go to the LLM. It sits between retrieve and generate: cheap retrieval casts a wide net (say top-50), then a precise reranker selects the best few that actually fit the prompt budget.

Q: What is the difference between a bi-encoder and a cross-encoder in retrieval?
A: A bi-encoder embeds the query and each document independently into vectors and compares them with cosine, which is fast and pre-computable — that's your first-stage retriever. A cross-encoder feeds the query and a candidate document together through the model to produce a single relevance score, which is far more accurate but too slow to run over the whole corpus, so it is used only to rerank the top candidates.

Q: Why run a cheap retriever first and an expensive reranker second, instead of just reranking everything?
A: A cross-encoder reranker scores one query-document pair at a time and cannot be pre-computed, so running it over millions of chunks per query is computationally hopeless. The two-stage design uses cheap approximate retrieval to shortlist a few dozen candidates, then spends the expensive cross-encoder budget only on that shortlist — recall from stage one, precision from stage two.

Q: In LlamaIndex, which component implements hybrid retrieval with rank fusion?
A: `QueryFusionRetriever` fuses multiple retrievers (typically a vector retriever and a `BM25Retriever`) and supports a `reciprocal_rerank` fusion mode; by default it also generates additional query variants (`num_queries=4`: the original plus three rewrites). It is the off-the-shelf way to get hybrid retrieval with RRF without hand-coding the fusion.

Q: What does the `num_queries` parameter of `QueryFusionRetriever` do, and what is the trade-off?
A: It controls query expansion: with `num_queries=4` the retriever uses the original query plus three LLM-generated rewrites, retrieving for each and fusing all results to improve recall on awkwardly phrased questions. The cost is extra LLM calls and latency per query, and you can set `num_queries=1` to disable expansion when you only want pure BM25-plus-vector fusion.

Q: Retrieval hit-rate is fine but answers are still mediocre. How can a reranker help, and how would you confirm it?
A: A reranker helps when the right chunk is being retrieved but buried below near-duplicates or weak matches, so it never makes it into the truncated context the LLM sees. Confirm by measuring a position-sensitive metric like MRR or hit-rate@1–3 before and after reranking on a fixed question set; if top-k recall was already high but top-3 was low, reranking is exactly the right lever.

Q: What are the three frameworks named for the positioning comparison, and what is each best known for?
A: LlamaIndex is data/indexing-centric (loaders, node parsers, retrievers, structured extraction); Haystack is pipeline-first, modelling RAG as explicit, composable component graphs; LangChain is the broad, sprawling toolkit of chains, agents, and integrations. The lesson's point is to know their centre of gravity, not to treat them as interchangeable.

Q: What is the "framework-fatigue" caution the lesson raises?
A: It is the warning that frameworks churn fast, add abstraction layers you may not need, and tempt you to learn the framework instead of the problem; the guidance is "thin frameworks, strong pipelines". You should understand the RAG stages well enough to swap or drop a framework, treating it as a convenience over your own well-governed pipeline rather than a black box you depend on.

Q: What does "thin frameworks, strong pipelines" mean in practice for a data architect?
A: It means investing your engineering effort in the pipeline fundamentals you control — parsing quality, chunking, indexing, freshness, evaluation — and using the framework only as glue for the parts that are genuinely commoditised. That way a framework deprecation or migration is a refactor of glue code, not a rewrite of your core retrieval logic.

Q: Why is Haystack's "pipeline-first" design appealing to someone from a data-engineering background?
A: Because it represents a RAG application as an explicit directed graph of typed components with named inputs and outputs, which mirrors the DAG mental model you already use in orchestrators like Dagster. That explicitness makes data flow, testing, and component swapping legible, as opposed to deeply nested call chains where the data path is implicit.

Q: When is LangChain a reasonable choice despite the framework-fatigue caution?
A: When you need its breadth of integrations and agentic patterns and want to prototype fast across many tools, LangChain's large ecosystem is a genuine advantage. The caution is to keep it at the edges as glue and not let its abstractions own your retrieval core, so you can still measure and replace the pieces that determine quality.

Q: What is "structured extraction" as the capstone RAG pattern in this lesson?
A: Structured extraction means constraining the LLM's output to a predefined schema — in LlamaIndex via a Pydantic model — so that instead of free prose you get validated, typed fields like `fund_name`, `isin`, `ongoing_charges`, `sri`. It turns RAG from a chat answer into a reliable data-extraction step whose output your downstream pipeline can ingest without brittle text parsing.

Q: Why use a Pydantic model rather than parsing free-text LLM output for fund-data extraction?
A: A Pydantic schema gives you typed fields, validation, and a clear contract: if the model returns a string where a float is required or omits a mandatory field, validation fails loudly instead of a malformed value flowing silently into your NAV or KID dataset. It moves extraction from fragile regex on prose to a structured, testable, auditable step.

Q: What does "schema-constrained output" buy you operationally over a well-crafted prompt asking for JSON?
A: Asking for JSON in the prompt is best-effort and the model can still emit invalid or extra-keyed JSON; schema-constrained extraction enforces the structure (via constrained decoding or Pydantic validation with retries) so the result either conforms or errors. Operationally that converts "occasionally broken output we discover downstream" into a hard, early validation boundary you can monitor.

Q: How would you express that an extracted SRI value must be an integer between 1 and 7 using Pydantic?
A: Declare the field with a constrained type, for example `sri: int = Field(ge=1, le=7)`, so any value outside the SRI scale's 1-to-7 range fails validation at extraction time. This encodes a real domain rule from the KID into the schema, catching a hallucinated `sri: 9` before it ever reaches your dataset.

Q: Why is structured extraction called the "capstone" pattern of this lesson?
A: Because it composes everything before it: good parsing and chunking get the right table into context, hybrid retrieval and reranking surface it reliably, and only then can schema-constrained generation turn it into validated, typed data. It is where retrieval quality stops being abstract and becomes a concrete, ingestible record your pipelines can trust.

Q: What is the central claim about document parsing — "parser choice moves quality more than the LLM does"?
A: It means that on real corpora the largest swings in answer quality come from how well your parser extracts text, tables, and layout from the source PDF, not from which LLM you pick. A parser that flattens a two-column page into interleaved gibberish or drops a table corrupts the input to every later stage, and no model can recover information that was never extracted.

Q: Why are PDF tables specifically hard to parse for RAG?
A: PDFs encode visual position, not logical structure, so a table is just text fragments placed at coordinates with no inherent row/column relationships; a naive extractor reads them in stream order and scrambles the cells. Recovering that "this number belongs to this row and column" requires layout analysis, which is exactly where parsers differ most and where fee/SRI tables get mangled.

Q: Why are multi-column layouts a parsing hazard, and where do they appear in fund documents?
A: A naive text extractor reads a two-column page left-to-right across both columns, interleaving lines from column one and column two into nonsense, so a chunk mixes unrelated sentences. Prospectuses and some KIDs use multi-column layouts, and the resulting scrambled text both embeds poorly and confuses the LLM, so the parser must detect and read columns in the correct reading order.

Q: The "Do" step asks you to parse 30 KIDs with two different PDF parsers and compare. What are you actually testing?
A: You are testing how each parser handles the high-value, hard parts — the fee table and the SRI/risk indicator — because those are where parsers diverge and where errors are most consequential. The comparison surfaces, with concrete documents, which parser preserves table structure and column order, giving you evidence rather than a vendor claim for your parser choice.

Q: Why does the lesson specify an EN/FR mix of fund documents?
A: Luxembourg-domiciled funds routinely publish in multiple languages, and French (and other) documents stress your loaders, tokenizers, and embedding model differently — accents, language-specific terms, and layout conventions. Testing on an EN/FR mix prevents you from tuning a pipeline that quietly works only on English and fails on the French KID for the same fund.

Q: What is a "freshness pipeline" in RAG and why is it owned by your orchestrator?
A: A freshness pipeline detects that a source document changed, then re-chunks and re-indexes just that document so retrieval reflects the current version; it is owned by your orchestrator (e.g. Dagster) because it is an incremental data job exactly like any other source refresh. Treating it as a scheduled, dependency-tracked asset is what keeps the index from drifting out of sync with reality.

Q: What is the concrete sequence of the freshness pipeline?
A: Document update → re-chunk → re-index: when a KID or prospectus is republished, you re-run parsing and chunking on that document, then upsert the new chunks into the vector store and remove the stale ones. The hard part is the upsert/delete bookkeeping so old chunks don't linger and get retrieved alongside the new version.

Q: What goes wrong if the freshness pipeline re-indexes new chunks but never deletes the old ones?
A: The index accumulates stale chunks, so retrieval can return last quarter's fee table next to this quarter's, and the LLM may cite the outdated figure with full confidence. You need delete-then-insert (or versioned upsert keyed on document id) so a document update replaces rather than augments its chunks.

Q: How would you sketch the freshness pipeline as a Dagster job over your existing assets?
A: Model the parsed document, the chunks, and the vector index as assets with dependencies, so a change to a source document invalidates the downstream chunk and index assets and Dagster re-materialises only those. A sensor or schedule detects updated source files, partitions can map to documents or funds, and the index asset's materialisation performs the upsert/delete — RAG freshness becomes ordinary asset orchestration.

Q: Why measure retrieval quality before generation rather than just evaluating final answers?
A: Because if the right chunk never reaches the prompt, no amount of prompt or model tuning can produce a correct answer, so a bad generation eval can't tell you whether the failure was retrieval or generation. Measuring retrieval first isolates the dominant failure mode cheaply and deterministically, before you spend money and add the noise of LLM generation on top.

Q: How would you answer a colleague who insists you should "just eval the answers"?
A: Point out that final-answer evals conflate two independent failures — the retriever missed the evidence, or the generator misused good evidence — and they are expensive and non-deterministic because they run the LLM. Measuring retrieval hit-rate@k on a fixed question set is cheap, repeatable, and pinpoints whether the problem is even fixable by generation tuning; you eval answers after retrieval is good, not instead of it.

Q: What is hit-rate@k, and why is it the right first metric for retrieval?
A: Hit-rate@k is the fraction of test questions for which at least one chunk containing the known answer appears in the top-k retrieved results. It directly measures "did we put the evidence in front of the model", is computed without invoking the LLM, and is deterministic, making it the cheapest, most decisive signal for comparing chunking and retrieval choices.

Q: What is the difference between hit-rate@k and MRR (mean reciprocal rank)?
A: Hit-rate@k is binary per question — was a relevant chunk anywhere in the top-k — while MRR rewards ranking the relevant chunk higher by averaging `1/rank` of the first hit. Hit-rate tells you whether the evidence was retrieved at all; MRR tells you how near the top it was, which matters when context is truncated, so MRR is the more sensitive metric once hit-rate is high.

Q: The "Do" step says hand-write 20 questions with known answers. What standard does this become later, and why hand-write it?
A: It becomes the seed of the 6.8.2 golden set — a curated, ground-truth evaluation dataset. You hand-write it because only a human who knows the documents can pin each question to a verified correct answer and source span; an auto-generated set risks circular evaluation where the system is graded against its own assumptions.

Q: When measuring retrieval hit-rate per chunking strategy, what must you hold constant to make the comparison fair?
A: Hold everything except the chunking strategy constant — same documents, same questions, same embedding model, same retriever and k, same query phrasing — so any difference in hit-rate is attributable to chunking alone. Changing the parser or k at the same time confounds the result, and you'd no longer be able to claim chunking moved the number.

Q: After adding hybrid retrieval, what does it mean to "record the delta", and why is the delta the deliverable?
A: It means logging the hit-rate@k before (vector-only) and after (BM25+vector fused) on the identical question set and reporting the change, e.g. hit-rate@5 rose from 0.72 to 0.86. The delta is the deliverable because it is evidence that the architectural choice paid off in numbers, which is what lets you defend the decision instead of asserting that hybrid "feels better".

Q: The capstone "Do" asks which single decision — parser, chunking, or hybrid — moved retrieval quality most. Why insist on a single ranked answer?
A: Forcing one ranked answer turns vague intuition into a defensible engineering judgement backed by measured deltas, and it focuses future effort on the lever that actually matters for this corpus. In fund documents the answer is frequently the parser or table-aware chunking, but the point is that you prove it with numbers rather than assume it.

Q: What does the "Done when" criterion "show retrieval quality numbers (not vibes)" guard against?
A: It guards against the common trap of declaring a RAG system "good" based on a handful of impressive demo answers, which is selection bias dressed up as evaluation. Numbers from a fixed question set are reproducible and comparable across changes, so they survive scrutiny from an auditor or a skeptical colleague in a way that "it seemed to work" never will.

Q: What is a vector store, and how does it relate to the index built in lesson 1.1.4?
A: A vector store is the system that persists embeddings and serves nearest-neighbour queries, such as pgvector in Postgres or a dedicated store; in this plan it is the 1.1.4 component your RAG chunks are indexed into. Reusing your existing governed store means RAG inherits your backup, access-control, and operational practices instead of introducing a new unmanaged data system.

Q: What does Ollama provide in the FOSS hands-on stack for this lesson?
A: Ollama runs open-weight LLMs and embedding models locally, so you can develop and evaluate the RAG pipeline without sending fund documents to a third-party API or incurring per-token cost. It is the local stand-in for managed endpoints like Azure OpenAI or Bedrock, and keeping data on-box matters when the documents are investor-restricted.

Q: Why might running embeddings and generation locally via Ollama be a compliance advantage in fund administration?
A: Because investor-restricted prospectuses, holdings, and KID drafts may not be permitted to leave your environment, and a local model means the document text never transits an external API. That keeps you inside data-residency and confidentiality constraints during development and lets you reserve the managed-cloud decision for a deliberate, reviewed step.

Q: What is the managed-cloud counterpart to the LlamaIndex+Ollama open stack mentioned in this lesson?
A: Azure OpenAI for the models plus Azure AI Search for retrieval, with AWS Bedrock Knowledge Bases as the equivalent on AWS. Knowing the open pipeline first lets you evaluate the managed offering on its real merits — security trimming, ops integration — against a baseline you actually understand and can cost.

Q: How does retrieval quality connect to citations, and why are citations non-negotiable in a regulated context?
A: Citations are produced at the generate stage by carrying each retrieved chunk's source metadata (document, page, span) through to the answer, so every claim points back to the exact text it came from. In a regulated workflow this is the audit trail: a reviewer can verify the cited SRI value against the actual KID page, and an uncited extraction is unverifiable and therefore unusable.

Q: What is a "source-span citation" specifically?
A: A source-span citation pins an answer not just to a document but to the precise character or token span within it that supports the claim, e.g. the exact line in the fee table. It is stronger than a document-level citation because a reviewer can confirm the value without re-reading the whole document, which is what makes extracted fund data defensible.

Q: How does poor chunking undermine the trustworthiness of citations even when the answer happens to be right?
A: If a chunk merges a fee table with unrelated risk text, the citation points at a span that contains the right number surrounded by irrelevant content, so a reviewer can't cleanly verify the claim and may distrust it. Clean, structure-aware chunks make citations tight and checkable, which is part of why chunking quality is a governance concern, not just a retrieval one.

Q: What is "context window" stuffing, and why is it a poor substitute for good retrieval?
A: Context stuffing means dumping large amounts of loosely relevant text into the prompt and hoping the model finds the answer. It is poor because it raises cost and latency, dilutes attention so the model can miss a needle buried in a haystack ("lost in the middle"), and it masks rather than fixes weak retrieval — precise top-k chunks beat a bloated context every time.

Q: What is the "lost in the middle" phenomenon and how does it affect RAG design?
A: LLMs tend to use information at the very start and end of a long context more reliably than information buried in the middle, so a relevant chunk placed mid-prompt can be effectively ignored. It argues for retrieving fewer, better-ranked chunks (hence reranking) and for ordering the most relevant chunks at the edges, rather than padding the prompt with many marginal ones.

Q: What does the embedding model's tokenizer have to do with chunk size?
A: Chunk size is measured in tokens, and the embedding model has a maximum input length, so chunks larger than that limit are silently truncated before embedding — losing the tail of the chunk. You must size chunks to fit the embedding model's context, and a mismatch between your splitter's token counting and the model's tokenizer can cause subtle truncation you only catch by checking.

Q: A query for a specific ISIN returns nothing useful from a vector-only retriever. What is the first thing to check or change?
A: First confirm whether the retriever is vector-only; an ISIN is an exact-token problem that dense embeddings handle poorly, so the fix is to add a BM25 / keyword leg and fuse — i.e. go hybrid. If hybrid is already in place, check that the ISIN survived parsing and chunking intact and wasn't split across a chunk boundary or mangled by the PDF extractor.

Q: Two PDF parsers disagree on a fund's ongoing-charges figure. How do you decide which to trust before involving the LLM?
A: Open the source PDF, read the fee table by eye, and compare each parser's extracted text against the ground truth — this is a parsing-level dispute, not a model-level one, so you resolve it at the parser. Whichever parser preserves the row/column alignment and the exact value wins, and that becomes documented evidence for your parser choice.

Q: Why does the lesson treat the choice of embedding model as less impactful than parser and chunking choices, in many cases?
A: Because if the parser never extracted the table or chunking split it apart, even a state-of-the-art embedding model is embedding the wrong or fragmented text — garbage in, garbage out. Once the right, coherent text is reliably in each chunk, embedding-model differences become a smaller, second-order tuning knob compared with the large swings from parsing and chunking.

Q: What is query expansion / rewriting, and how does it improve recall?
A: Query expansion generates one or more reformulations of the user's question (synonyms, sub-questions, keyword variants) and retrieves for all of them, then fuses the results, so a poorly phrased query still surfaces relevant chunks. In `QueryFusionRetriever` this is the `num_queries` mechanism; the cost is extra LLM calls, traded for higher recall on awkward questions.

Q: How would you decide a good value of k (top-k) for retrieval in your fund-document RAG?
A: Sweep k on your golden question set and watch hit-rate@k: pick the smallest k where hit-rate plateaus, so you maximise the chance the evidence is present while minimising context cost and dilution. Too small and you miss evidence; too large and you stuff the prompt and invite "lost in the middle", so k is an empirical, corpus-specific choice.

Q: What is the difference between recall-oriented first-stage retrieval and precision-oriented reranking, expressed as a goal for each?
A: First-stage retrieval optimises recall — get the relevant chunk into a wide candidate set (e.g. top-50) even at the cost of including junk — because anything not retrieved can never be recovered. Reranking optimises precision — from those candidates, push the genuinely relevant few to the top so the truncated context the LLM sees is dense with signal.

Q: Why is "more chunks retrieved" not the same as "better answers"?
A: Because retrieving more raises recall but lowers the signal-to-noise ratio of the context, and beyond a point the extra chunks are distractors that the model can latch onto, plus they cost tokens and latency. The aim is the right chunks near the top, which is why reranking and a measured, modest k beat simply increasing k.

Q: What metadata filtering capability should you exploit to enforce "never retrieve another fund's documents for a fund-specific question"?
A: Use metadata filters on retrieval so that a query scoped to a given fund only searches chunks tagged with that fund's id/ISIN, hard-excluding all other funds before similarity ranking. This is a pre-filter, not a post-hoc hope, and it is the open-stack way to enforce the kind of access boundary that managed platforms call security trimming.

Q: How does a fixed-size vs structure-aware comparison typically play out on KID fee tables, and what should you expect to report?
A: Structure-aware (table/section-aware) chunking usually wins decisively on fee/SRI questions because it keeps each table intact, so hit-rate@k on those questions jumps while fixed-size leaves the value fragmented and frequently misses it. Expect to report a clear positive delta for structure-aware on table-bearing questions, with little difference on plain prose questions.

Q: What is the risk of evaluating retrieval only on prose questions and not on table/figure questions in a fund corpus?
A: You'd conclude fixed-size chunking is fine because prose retrieves well, then ship a system that fails on exactly the fee and SRI values that matter most commercially and legally. The golden set must over-represent the hard, high-value table and figure questions so the eval stresses the failure mode you actually care about.

Q: Why is RAG generally preferred over fine-tuning a model on your fund documents for a KID-extraction use case?
A: RAG keeps the knowledge in an external, governable index you can update, version, and cite, so a republished KID is handled by re-indexing — minutes — rather than retraining. Fine-tuning bakes facts into weights where they can't be cited or selectively updated, and stale fund data baked into a model is both unauditable and a compliance hazard.

Q: What does "freshness" mean as a service-level concern, and how would you express an SLA for it?
A: Freshness is the maximum acceptable lag between a source document being updated and the index reflecting it, e.g. "a republished KID is searchable within 30 minutes". Expressing it as an SLA turns RAG correctness into a monitorable orchestration target — your Dagster sensor latency and re-index runtime must stay within that bound, with alerting when they don't.

Q: How can document versioning in chunk metadata help when an auditor asks "what did the model see?"
A: If each chunk carries the document version and span it came from, you can reconstruct exactly which version of the KID produced a given extracted value, even months later after the document changed. Combined with LLMOps tracing of the prompt, this answers the auditor's question precisely, which is the regulated-context reason metadata and freshness bookkeeping are not optional.

Q: What is a reasonable order of operations when retrieval quality is poor: parser, chunking, hybrid, reranking — where do you start?
A: Start upstream and work down: verify the parser actually extracted the target text and tables correctly, because nothing downstream can fix missing input; then fix chunking so the evidence stays coherent; then add hybrid retrieval for exact-match recall; finally add reranking for ordering. Fixing reranking while a table is being shredded by the parser is wasted effort.

Q: Why is the LLM choice often the last thing to tune in a RAG system, not the first?
A: Because the LLM only sees what retrieval delivers, so its ceiling is set by parsing, chunking, and retrieval quality; swapping models while retrieval is weak yields fluent wrong answers either way. You tune the data path until the right evidence reliably reaches the prompt, then choose the model that best uses it — model-last, not model-first.

Q: What is the danger of LLM "fluency" specifically in a retrieval failure?
A: When retrieval returns the wrong fund's table, a capable LLM will still produce a confident, well-formatted answer citing that wrong data, so fluency hides the failure rather than signalling it. This is why you can't judge RAG by how good the answers sound, and why retrieval must be measured independently with ground truth.

Q: How do you handle a KID that is a scanned image rather than digital text?
A: A scanned PDF has no extractable text layer, so you must run OCR first to produce text before any chunking or embedding, and OCR quality (especially on tables) becomes the new dominant quality lever. This is an extreme case of "parser choice moves quality most", and you'd evaluate OCR output against the source just as you compare PDF parsers.

Q: Why might two embeddings of the "same" KID section differ between the EN and FR versions, and why does that matter?
A: The English and French texts are different token sequences with language-specific terms, so their embeddings differ even though the meaning is the same; if your embedding model is weak on French, the FR chunk may not be retrieved for an English query or vice versa. It matters because Luxembourg funds are multilingual, so you must test cross-language and possibly use a multilingual embedding model.

Q: What is the role of overlap when chunking a long prospectus narrative, and what overlap is a sensible starting point?
A: Overlap ensures a fact spanning a chunk boundary appears intact in at least one chunk; LlamaIndex's `SentenceSplitter` default of `chunk_overlap=200` tokens against `chunk_size=1024` is a sensible starting point you then tune. Too little overlap drops boundary facts; too much inflates index size and duplicates content across many retrieved chunks.

Q: Why is it dangerous to let chunking duplicate large amounts of content via excessive overlap?
A: Heavy overlap means near-identical chunks, so retrieval returns several copies of essentially the same span, crowding out other relevant chunks within your top-k and wasting context budget on redundancy. It also inflates the index and embedding cost, so overlap should be just enough to preserve boundary facts, not a blanket large value.

Q: How does reranking interact with the "lost in the middle" problem to improve answers?
A: Reranking promotes the most relevant chunk to the top of the list, and if you place top-ranked chunks at the start (and end) of the prompt, you exploit the model's stronger attention there, mitigating lost-in-the-middle. So reranking helps twice: it picks better chunks and it lets you order them where the model actually reads them.

Q: What is the practical meaning of "hybrid is the default" for your initial RAG build decision?
A: It means you should not ship a vector-only retriever as a v1 and "add BM25 later"; on identifier- and jargon-heavy corpora vector-only fails common exact-match queries on day one. Start hybrid, measure the delta to prove it, and treat keyword search as a first-class half of retrieval rather than a bolt-on.

Q: When comparing parsers, why measure on the fee/SRI tables specifically rather than overall text fidelity?
A: Because parsers tend to agree on plain prose and diverge precisely on tables and multi-column layout, and the tables carry the highest-value, most error-prone facts in a KID. Measuring overall character accuracy would average away the very differences that decide the parser choice; you score where it hurts — the structured regions.

Q: What does it mean to "govern" the embedding step like any other pipeline stage?
A: It means versioning the embedding model and parameters, tracking which model produced which vectors, re-embedding deterministically when the model changes, and monitoring for drift or failures. If you silently swap embedding models, old and new vectors become incomparable in the same index, so the embedding model is a governed dependency, not a casual import.

Q: Why must a vector store re-embedding be coordinated across the whole index rather than done piecemeal?
A: Because nearest-neighbour search compares vectors in the same space, and mixing vectors from two different embedding models (or versions) yields meaningless distances and corrupted ranking. Changing the embedding model is therefore a full re-index event you plan and orchestrate, not an incremental per-document change.

Q: How would you detect that a freshness re-index actually worked, as an automated check?
A: Add a post-condition assertion: after re-indexing an updated document, query for a known new value and confirm it is retrieved while the old value is not, and check the chunk count matches expectation. Wiring this as a Dagster asset check turns "I think the re-index ran" into a tested invariant that alerts when freshness breaks.

Q: A reviewer says the RAG answer is right but can't find the cited value on the cited page. What likely went wrong?
A: Likely the citation metadata is misaligned with the chunk text — for example page numbers weren't carried correctly through parsing/chunking, or chunks were merged so the span points to the wrong location. The answer being right but the citation unverifiable is itself a failure in a regulated context, so you fix the metadata propagation, not just the answer.

Q: Why is it useful to separate "retrieval eval" and "generation eval" as two distinct test suites?
A: Because they fail for different reasons and need different metrics — retrieval eval uses deterministic hit-rate/MRR against known source chunks, while generation eval judges faithfulness and correctness of the produced text. Keeping them separate lets you attribute a regression to the right stage and avoids paying for LLM-based generation evals when the cheaper retrieval test already localises the fault.

Q: What is the difference between "context relevance" and "answer faithfulness" as RAG quality dimensions?
A: Context relevance asks whether the retrieved chunks actually pertain to the question (a retrieval property), while answer faithfulness asks whether the generated answer is supported by those chunks rather than hallucinated (a generation property). You can have relevant context with an unfaithful answer, or a faithful answer over irrelevant context, so both must be measured.

Q: Why does the lesson frame ingestion, chunking, indexing, and freshness as "your home turf" for a data architect?
A: Because those four are ingestion, transformation, materialisation, and incremental refresh — the exact lifecycle you already design, schedule, test, and monitor for any data source. The framing tells you to bring your existing pipeline discipline (idempotency, lineage, SLAs, asset checks) to RAG rather than treating it as unfamiliar ML territory.

Q: What is "small-to-big" (parent-child) retrieval, and why is it useful on dense fund documents?
A: Small-to-big retrieval embeds and searches on small, precise child chunks for high retrieval accuracy, but then returns the larger parent chunk (e.g. the whole section) to the LLM so it has enough surrounding context to answer. On dense KIDs it lets you match a precise line while still giving the model the full table or section it sits in, decoupling the precision of the search unit from the richness of the generation context.

Q: Summarise, in one principle, what most determines RAG quality according to this lesson, and what follows from it.
A: The data path determines quality — parser, chunking, and hybrid retrieval move the numbers far more than the LLM — so you measure retrieval before generation, fix problems upstream first, and treat freshness and structured extraction as governed pipeline stages. The model is chosen last, and every quality claim is backed by hit-rate numbers from a hand-built golden set, not vibes.

Q: In LlamaIndex, which node parser implements small-to-big at the sentence level, and what is its key parameter?
A: The `SentenceWindowNodeParser` embeds each individual sentence as the retrieval unit but stores a surrounding window of sentences in metadata, controlled by `window_size` (default `3` sentences on either side). At query time you pair it with a `MetadataReplacementPostProcessor` that swaps each retrieved sentence for its stored window before the text reaches the LLM, so you search precisely on one sentence yet generate over its full context.

Q: What does `MetadataReplacementPostProcessor` do, and why is it useless without the matching parser?
A: It replaces each retrieved node's text with a value stored in its metadata — specifically the window written by `SentenceWindowNodeParser` under `window_metadata_key` (default `window`) — so the LLM sees the surrounding sentences rather than the single embedded one. Without the window-style parser there is no window in metadata to swap in, so the post-processor finds nothing and silently leaves the node unchanged, defeating the small-to-big intent.

Q: Which LlamaIndex node parser would you reach for first on a KID exported to Markdown, and why?
A: `MarkdownNodeParser` splits on the Markdown heading hierarchy, so each `##`/`###` section becomes its own node and a fee table under one heading stays intact. It is the structure-aware option when your loader (e.g. a Markdown-emitting PDF parser like LlamaParse) has already recovered headings, turning layout back into clean chunk boundaries instead of blind token cuts.

Q: You install `llama-index` but `from llama_index.retrievers.bm25 import BM25Retriever` raises `ModuleNotFoundError`. What is the cause and fix?
A: `BM25Retriever` lives in the separate integration package `llama-index-retrievers-bm25`, which the core install does not pull in, so you must `uv add llama-index-retrievers-bm25` (or `uv pip install` it). LlamaIndex split integrations into per-package distributions, so most non-core retrievers, readers, and stores need their own install before the import resolves.

Q: How do you construct a `BM25Retriever` in LlamaIndex over a fixed node list, and what controls stemming?
A: Call `BM25Retriever.from_defaults(nodes=nodes, similarity_top_k=k)`, optionally passing `stemmer=Stemmer.Stemmer("english")` and `language="english"` to enable Snowball stemming and stopword removal. The retriever tokenizes and stems both the query and the indexed text, so `charges` and `charge` collide — useful for prose but irrelevant for an `ISIN`, which you do not want stemmed.

Q: Why might a `BM25Retriever` built from an in-memory node list go stale, and how does that differ from the vector store?
A: `BM25Retriever.from_defaults(nodes=...)` builds its lexical index from the exact node list you pass at construction time and does not watch the vector store, so any chunk added or deleted afterwards is invisible to it until you rebuild. Your pgvector index can be upserted incrementally, but the BM25 leg of a hybrid retriever must be reconstructed on the current node set after every freshness re-index or the two legs disagree.

Q: A user pastes a query in French but your golden set is English; hit-rate collapses. What is the first multilingual check?
A: Confirm your embedding model is actually multilingual and that query and corpus are embedded by the same model — a monolingual English model maps `frais courants` far from `ongoing charges`, so the FR query never retrieves the EN chunk. The fix is a multilingual embedding model (so both languages land in one shared space) or per-language indexes with language routing, then re-measure hit-rate on a bilingual question set.

Q: How does `response_mode="tree_summarize"` differ from `"refine"` when synthesizing an answer over many retrieved chunks?
A: `refine` is sequential — it builds an answer from chunk 1, then feeds that answer plus chunk 2 into a refine prompt, iterating N-1 times — whereas `tree_summarize` recursively summarizes chunks bottom-up into a tree and returns the root, which parallelizes and suits summarization. For a single-fact KID lookup neither matters much, but `refine` can let a later chunk overwrite a correct earlier extraction, so it is a subtle failure mode on extraction tasks.

Q: What does `response_mode="compact"` do, and why is it the sensible default for fund-data lookups?
A: `compact` concatenates (packs) as many retrieved chunks as fit into one context window before calling the LLM, then refines only across the overflow, so it makes far fewer LLM calls than plain `refine`. For a top-3 KID retrieval that all fits in context, it is one call with all evidence present at once, which is cheaper and avoids the refine-overwrite hazard.

Q: A reranker improves MRR but your final answers do not change. What plausibly explains this, and what would you check?
A: If your `similarity_top_k` for generation is large enough that the right chunk was already in context before reranking, reordering it higher changes MRR but not what the LLM sees, so the answer is unchanged. Check whether reranking is actually paired with a smaller final top-k (e.g. retrieve 50, rerank, keep 3) — reranking only pays off when it gates a truncation step that would otherwise drop the best chunk.

Q: In LlamaIndex, how do you wire a cross-encoder reranker into a query engine, and what package provides it?
A: Use a `SentenceTransformerRerank` node post-processor (from `llama-index-postprocessor-sentence-transformers`) with a model like `cross-encoder/ms-marco-MiniLM-L-6-v2` and a `top_n`, then pass it via `node_postprocessors=[reranker]` to `index.as_query_engine(...)`. It runs after retrieval and before synthesis, scoring each query-chunk pair jointly and keeping only the top `top_n` for the LLM.

Q: Your reranker model and your generation model are different sizes; latency spikes only on some queries. What is the likely driver?
A: Cross-encoder rerank cost scales with the number of candidates times their token length, so queries that retrieved long table-bearing chunks (or a larger first-stage top-k) cost far more reranker compute than short-prose retrievals. Cap first-stage top-k and truncate over-long candidates before reranking, and measure per-query candidate count to find the outliers.

Q: How would you use metadata filters in LlamaIndex to hard-scope retrieval to one fund's ISIN?
A: Build a `MetadataFilters` object with a `MetadataFilter(key="isin", value="LU0123456789", operator=FilterOperator.EQ)` and pass it through the retriever or `as_query_engine(filters=...)`, so the vector store pre-filters to matching chunks before similarity ranking. This is a hard exclusion executed in the store, not a post-hoc re-rank, which is exactly the guarantee a compliance reviewer needs against cross-fund leakage.

Q: Metadata filtering by ISIN returns nothing even though the fund is indexed. First thing to check?
A: Verify the metadata key was actually written onto the nodes at ingestion and is queried with the identical key name and value casing — a chunk tagged `ISIN` will not match a filter on `isin`, and a trailing space or differing value type silently zeroes the result set. Inspect one node's `metadata` dict directly and confirm the filter key, operator, and value type line up before suspecting the retriever.

Q: Why can `excluded_embed_metadata_keys` matter when you attach rich metadata to nodes?
A: By default LlamaIndex prepends node metadata into the text that gets embedded, so noisy fields like a long file path or a timestamp pollute the embedding and shift it away from the real content. Listing those keys in `excluded_embed_metadata_keys` keeps them retrievable as filters and citations while excluding them from the embedded string, so the vector reflects the actual passage, not its bookkeeping.

Q: What is the difference between `excluded_embed_metadata_keys` and `excluded_llm_metadata_keys`?
A: `excluded_embed_metadata_keys` controls which metadata is left out of the text sent to the embedding model, while `excluded_llm_metadata_keys` controls which is left out of the text shown to the LLM at generation. You might embed without the page number (noise for similarity) yet still expose it to the LLM for citation, so the two lists serve different stages and are tuned independently.

Q: A query for a defined term like `Synthetic Risk Indicator` matches the glossary definition, not the fund's actual SRI value. How do you bias retrieval toward the value?
A: This is a granularity/precision problem: chunk so the fund's filled-in SRI row is its own node distinct from the boilerplate definition, and add a metadata field marking value-bearing chunks so you can filter or boost them. You can also add the fund identifier to the query and use hybrid retrieval so the exact ISIN token co-locates the value chunk above the generic definition.

Q: What is a "needle in a haystack" eval, and how does it relate to your top-k and chunk-size choices?
A: A needle-in-a-haystack test plants one known fact in a large body of distractor text and checks whether retrieval and the LLM surface it, isolating recall and the lost-in-the-middle effect. It directly informs top-k (too small misses the needle) and chunk size (too large buries the needle inside a diluted vector), giving you a controlled probe before you trust real golden-set numbers.

Q: How does HNSW's `ef_search` (or pgvector's `hnsw.ef_search`) parameter trade off against recall and latency?
A: `ef_search` sets how many candidate nodes the HNSW graph explores per query; a larger value visits more neighbours, raising recall toward exact search at the cost of latency, while a small value is fast but can miss the true nearest chunk. If hit-rate is mysteriously low on a known-present chunk, raising `ef_search` is a cheap thing to test before blaming chunking or embeddings.

Q: In pgvector, what is the difference between an HNSW index and an IVFFlat index for your RAG store?
A: HNSW builds a navigable small-world graph that gives high recall and fast queries without needing the data present at build time, and is generally the default for quality; IVFFlat partitions vectors into lists and probes a subset (`probes`), which builds faster and uses less memory but needs representative data at index time and tends to lower recall. For a fund corpus that grows incrementally, HNSW usually wins on recall and on tolerating inserts.

Q: Your pgvector query is doing a sequential scan instead of using the HNSW index. What is the usual cause?
A: The index is built for a specific distance operator, so the query must use the matching operator class — `<=>` for cosine, `<->` for L2, `<#>` for inner product — and a mismatch makes the planner ignore the index and scan. Confirm with `EXPLAIN` that the index is used, and ensure your LlamaIndex vector store is configured for the same distance metric the index was created with.

Q: How would you make a freshness re-index idempotent so re-running it never duplicates chunks?
A: Key every chunk on a deterministic id derived from document id plus a stable chunk position or content hash, and upsert on that key so re-running replaces rather than appends; pair it with a delete of any chunk ids no longer produced for that document. Idempotency means the index converges to the same state whether the job runs once or five times, which is what lets a Dagster retry be safe.

Q: A document was re-indexed but old chunks linger because their ids changed between runs. What chunking property caused this?
A: Non-deterministic chunk ids — for example ids based on insertion order, a random UUID per run, or boundaries that shift when one upstream byte changes — mean the new run's ids never match the old ones, so the delete-by-id step finds nothing to remove. Use content-stable ids (document id plus content hash or fixed offsets) so a small edit only churns the affected chunks, not the whole document.

Q: Why is content-hash-based change detection better than a modified-timestamp for triggering re-index?
A: A timestamp can change when a file is merely re-saved or re-uploaded with no content change, triggering needless re-embedding cost, and conversely some pipelines rewrite files without bumping the visible mtime. Hashing the extracted text detects real content changes regardless of filesystem metadata, so you re-index exactly when the meaning changed — cheaper and more correct.

Q: A KID is republished with only its fee table changed. How can you avoid re-embedding the unchanged sections?
A: Chunk deterministically and compute a per-chunk content hash, then diff the new chunks against the stored ones and re-embed only the chunks whose hash changed, upserting just those. This incremental, chunk-level freshness keeps embedding cost proportional to the actual edit rather than re-embedding the whole document on every minor republish.

Q: What is the risk of using an LLM to auto-generate your golden-set questions from the same corpus the system reads?
A: It creates circular evaluation: the LLM tends to ask what it can already answer and may inherit the same parsing errors, so the system grades well against questions shaped by its own blind spots. Hand-written questions tied to human-verified source spans break the loop and deliberately probe the hard table/figure cases an auto-generator would skip.

Q: How would you structure a golden-set record so it supports both retrieval and generation evals?
A: Store the question, the verified answer, and the ground-truth source locator (document id, page, and span or chunk id) for each item, so retrieval eval can check whether that chunk appeared in top-k while generation eval checks the produced text against the answer. The shared source pointer is what lets you attribute a failure to retrieval versus generation on the same fixed item.

Q: When comparing two parsers, what concrete artifact should you diff to make the comparison auditable?
A: Diff each parser's extracted text for the same page — especially the serialized fee/SRI table — against the human-read ground truth, and record a per-document pass/fail on whether row/column alignment and the exact values survived. A stored side-by-side diff is reproducible evidence for the parser decision, unlike a remembered impression that the parser "looked better".

Q: What is "reading order" recovery, and which fund documents most need it?
A: Reading-order recovery is reconstructing the logical sequence of text from a PDF's coordinate-placed fragments so a two-column page is read column-by-column rather than line-by-line across both. Prospectuses and some KIDs use multi-column layouts, and without reading-order recovery the extractor interleaves the columns into nonsense that embeds poorly and confuses the LLM.

Q: How would you detect that a parser scrambled a multi-column page programmatically, without reading every document?
A: Add a heuristic check on extracted text — for example flag pages where sentence-tokenization produces a high rate of fragments that do not parse as grammatical sentences, or where known section headers appear interleaved with unrelated lines. You can also assert that specific anchor phrases known to be adjacent in the source remain adjacent in the extraction, failing the document for manual review when they do not.

Q: A scanned KID returns empty text from every PDF parser. What stage is missing and what do you add?
A: A scanned PDF is an image with no text layer, so text extractors return nothing and you must insert an OCR stage (e.g. Tesseract or a layout-aware OCR) before chunking to produce text. OCR quality on tables then becomes the dominant quality lever, so you evaluate OCR output against the source exactly as you compare PDF text parsers.

Q: How does table-aware extraction to Markdown help downstream chunking and citation?
A: Emitting a recovered table as a Markdown table (rows and pipes) preserves the row/column relationships in plain text, so a structure-aware splitter like `MarkdownNodeParser` keeps the whole table as one coherent node and the embedding means "this fund's fee row". The Markdown also gives a clean, human-readable span for citation, so a reviewer can verify the cited value against a faithful rendering.

Q: Why is "one chunk per table" sometimes too coarse for a large prospectus fee schedule?
A: A multi-page fee schedule packed into a single chunk can exceed the embedding model's max input and gets silently truncated, and its vector dilutes across many rows so a query for one share class ranks poorly. The fix is row-group or per-share-class chunking that keeps each row with its header, balancing table integrity against embedding precision and the token limit.

Q: How do you confirm a chunk was not silently truncated by the embedding model's max input length?
A: Count tokens per chunk with the embedding model's own tokenizer and assert each chunk is under the model's documented max sequence length before embedding; log or fail any chunk that exceeds it. Truncation is silent — the API returns a vector regardless — so without an explicit token-count check you only notice when the tail of a long chunk mysteriously never retrieves.

Q: Your splitter's token count and the embedding model's tokenizer disagree, causing occasional truncation. How do you align them?
A: Configure the splitter to count tokens with the same tokenizer the embedding model uses (pass that model's tokenizer/encoding into the `SentenceSplitter`), so the chunk-size budget matches the real limit. A generic word or character heuristic undercounts for code-like or accented text, so a French KID with many sub-word tokens can overflow a chunk that looked safe by character count.

Q: What is the practical reason to store the original source span offsets in chunk metadata, beyond citation display?
A: Storing character/token offsets lets you reconstruct the exact source region for audit, re-highlight it in a viewer, and re-derive citations if you re-chunk, all without re-parsing the document. In a regulated extraction it is the deterministic link from an extracted SRI value back to the precise bytes that supported it, which is what makes the value defensible months later.

Q: How would you express that an extracted ongoing-charges figure must be a non-negative percentage using Pydantic?
A: Declare it as a constrained float, for example `ongoing_charges: float = Field(ge=0.0, le=100.0)`, so a hallucinated negative or absurd value fails validation at extraction time. Encoding the domain bound turns a plausible-looking but wrong number into a hard error before it can flow into your cost-disclosure dataset.

Q: How do you make a Pydantic extraction field optional so a missing KID value does not crash the pipeline, while still flagging it?
A: Type it as `Optional[float] = None` (or `float | None`) so absence yields `None` rather than a validation error, then add a downstream completeness check that counts and reports `None`s per field. This distinguishes "the document genuinely lacks this value" from "extraction failed", letting you route incomplete records to review instead of silently dropping them.

Q: A Pydantic-constrained extraction fails validation on a valid document. What is the first thing to check in the retrieval path?
A: Check whether the chunk containing the value actually reached the LLM — if retrieval missed the fee table, the model has nothing to extract and will either omit the field or hallucinate, tripping validation. Validation failure is often a retrieval symptom, so confirm the source chunk is in context before assuming the schema or prompt is wrong.

Q: How does schema-constrained generation interact with the choice of LLM and provider?
A: Some providers enforce the schema by constrained/structured decoding (the model literally cannot emit invalid JSON), while others only validate after free generation and retry on failure, which differs in reliability and latency. If your local Ollama model lacks robust constrained decoding, you lean harder on Pydantic validation-with-retry, so the extraction's robustness depends on which mechanism is actually in play.

Q: Why is a validation-with-retry loop around extraction not a free lunch?
A: Each retry re-invokes the LLM, adding latency and cost, and a model that keeps producing the same invalid output can loop until a cap is hit, so you need a bounded retry count with a dead-letter path for persistent failures. Retries paper over occasional formatting slips but mask a systematic problem (e.g. a missing chunk) that you should fix upstream instead of retrying around.

Q: In Haystack, what is a "Pipeline" and how do you wire two components together?
A: A Haystack `Pipeline` is an explicit graph of named components; you `add_component(name, instance)` for each and `connect("a.output", "b.input")` to route a named output socket into a named input socket. The data flow is declared edge-by-edge with typed sockets, which is what makes the DAG legible and testable, mirroring the asset wiring you already do in Dagster.

Q: What advantage does Haystack's typed input/output socket model give when a pipeline misconfiguration occurs?
A: Because each component declares typed input and output sockets, a wrong connection (mismatched type or a missing required input) fails fast at wiring or run time with a clear socket-level error, rather than producing a confusing failure deep inside a nested call. This explicitness localizes the fault to a specific edge, which is the data-engineering legibility the lesson values over LangChain's implicit call chains.

Q: How does LlamaIndex's `IngestionPipeline` relate to the freshness story, and what does its `docstore` add?
A: `IngestionPipeline` runs an ordered list of transformations (splitter, metadata extractors, embedder) over documents and can attach a `docstore` so it deduplicates and upserts: documents whose hash is unchanged are skipped and changed ones are reprocessed. That built-in document-management/dedup is exactly the incremental-refresh behaviour you want from a freshness job, executed as a reusable pipeline object.

Q: What does LlamaIndex's document-management upsert in `IngestionPipeline` do when a document's hash is unchanged versus changed?
A: With a `docstore` and `docstore_strategy` set to upsert, an unchanged document hash is skipped to save embedding cost, while a changed hash causes the old nodes for that `ref_doc_id` to be removed and the new ones inserted. This gives delete-then-insert semantics keyed on document identity, preventing the stale-chunk accumulation that re-index pipelines otherwise suffer.

Q: When would you deliberately choose vector-only retrieval over hybrid, despite "hybrid is the default"?
A: On a corpus of pure natural-language prose with no identifiers, codes, or exact-match defined terms — and where you have measured that BM25 adds no hit-rate — the keyword leg only adds latency and maintenance. The point is to decide by the measured delta on your golden set, not dogma; fund corpora almost always keep BM25 because of ISINs and LEIs, but the rule is "measure, then choose".

Q: How does query expansion via `num_queries` interact with metadata filters in a fused retriever?
A: Each generated query variant still runs against the same metadata-filtered candidate set, so the fund-scoping filter holds across all expansions and you cannot leak into another fund's chunks via a rewrite. The expansion broadens phrasing recall within the allowed scope, which is the safe combination: widen wording, never widen the access boundary.

Q: A demo of five questions looks perfect, but you refuse to call the system done. State the governance argument in one line and the next step.
A: Five hand-picked answers are selection bias, not evidence, because they cannot show the failure rate on the table/figure questions that carry the legal and commercial risk; the next step is to run the full hand-built golden set and report hit-rate@k and faithfulness as reproducible numbers. Only numbers from a fixed, hard-case-weighted set survive an auditor's or a skeptic's scrutiny, which is the standard this lesson holds you to.

Q: Why is cosine the conventional distance for text embeddings rather than raw Euclidean, and what must match between query and index?
A: Cosine compares the direction of embeddings and ignores magnitude, which suits text models whose vectors encode meaning in orientation, so it is robust to length differences between a short query and a long chunk. The non-negotiable rule is that the query is embedded by the same model and the index uses the matching distance operator class, since mixing metrics or models yields meaningless rankings.

Q: A second-stage reranker keeps a chunk the first-stage retriever never returned. Is that possible, and what does it tell you?
A: No — a reranker can only reorder candidates that first-stage retrieval already surfaced, so it can never recover a chunk that was missing from the shortlist. If the right chunk is absent, the fault is recall in stage one (fix chunking, go hybrid, or raise first-stage top-k), and adding or tuning the reranker is wasted effort until the evidence is actually in the candidate set.
