# Phase 1 · 1.10.1 Modeling notations overview — 150 self-test questions

Q: What does the abbreviation UML stand for, and which body maintains it?
A: UML stands for Unified Modeling Language, a general-purpose graphical language for visualizing and specifying the structure and behavior of software systems. It is maintained by the Object Management Group (OMG), the same standards body behind BPMN and DMN. The current formal version is UML 2.5.1, released in December 2017.

Q: At this lesson's level, what is your goal with UML, BPMN, DMN, ArchiMate, ERD, and C4?
A: The explicit goal is reading fluency, not authoring mastery — you must name each notation and narrate what a diagram says when shown one cold. The point is that in a fund-industry workshop the business analyst brings BPMN, the vendor brings ArchiMate, and the DBA brings an ERD, and the architect reads all of them without a translator. Deep authoring practice comes later in Phase 4 (BPM) and Phase 8 (ArchiMate).

Q: A UML class diagram shows structure or behavior?
A: A UML class diagram shows static structure: the classes (or types/entities) in a system, their attributes and operations, and the relationships between them such as association, generalization, and composition. It answers "what things exist and how are they related" rather than "what happens over time." Behavior over time is the job of UML's interaction diagrams like the sequence diagram.

Q: A UML sequence diagram shows structure or behavior?
A: A UML sequence diagram shows interaction behavior: how objects or participants exchange messages over time to accomplish a scenario. The vertical axis represents time flowing downward, and horizontal arrows represent messages between lifelines. It answers "in what order do these parties call each other" — for example, how a NAV-calculation service, a pricing feed, and a register are invoked during end-of-day processing.

Q: In a UML sequence diagram, what does a vertical dashed line under a participant represent?
A: The vertical dashed line is the lifeline, representing the existence of that participant (object, actor, or system) over the duration shown. Time flows downward along the lifeline. A thin rectangle drawn over the lifeline is the activation bar, showing when that participant is actively executing in response to a message.

Q: In UML, what is the difference between a solid-arrowhead message and an open-arrowhead message on a sequence diagram?
A: A filled (solid) arrowhead conventionally denotes a synchronous message — the sender waits for the call to complete — while an open (line/stick) arrowhead denotes an asynchronous message or a reply. A dashed line with an open arrowhead is typically the return message. These conventions let you read whether a caller blocks, which matters when narrating a NAV pipeline where some feeds are fire-and-forget and others are awaited.

Q: What notation would you reach for to describe the class structure of a fund's data model at the object/system level?
A: A UML class diagram is the right reading tool for object/system-level structure such as the classes in an application's domain model. However, for a relational database schema specifically, an ERD is the more common and more precise choice. The class diagram shines when the model includes behavior (operations) and object-oriented relationships like inheritance, which a pure ERD does not express.

Q: What does BPMN stand for, and who maintains it?
A: BPMN stands for Business Process Model and Notation, an OMG standard for modeling business processes as flowcharts that both business and technical stakeholders can read. The current widely-used version is BPMN 2.0.2, published in January 2014, which standardized both the visual notation and an XML serialization. It is the notation a business analyst brings to a fund-operations workshop.

Q: What kind of question is BPMN designed to answer?
A: BPMN answers "how does this business process flow end to end" — the sequence of activities, decisions, events, and the participants (lanes) that carry them out. It is the natural choice for describing an order lifecycle or a transfer-agency flow, such as how a subscription order moves from an inbound EMT/order file through validation to settlement and register update. It models process, not data structure or static architecture.

Q: In BPMN, what is the difference between a task and a sub-process?
A: A task is an atomic unit of work in the process — a single activity that is not broken down further at that level of detail. A sub-process is a compound activity that contains its own flow of tasks and can be collapsed (shown as a rounded rectangle with a small plus marker) or expanded inline. Sub-processes let you read a high-level transfer-agency flow first, then drill into, say, the AML-check sub-process.

Q: In BPMN, what does a diamond shape (gateway) represent?
A: A diamond is a gateway, which controls how the sequence flow diverges or converges — it is the branching and merging logic of the process. An exclusive (XOR) gateway marked with an X takes exactly one outgoing path based on a condition; a parallel (AND) gateway marked with a plus splits into or joins concurrent paths. Reading gateways correctly tells you whether two activities happen in sequence, as alternatives, or in parallel.

Q: In BPMN, what do circles represent, and how do start, intermediate, and end events differ visually?
A: Circles are events — something that happens during the process. A start event uses a thin single-line circle, an intermediate event uses a double-line circle, and an end event uses a thick single-line circle. Markers inside the circle (an envelope for a message, a clock for a timer) tell you the trigger, such as a message-start event firing when an inbound order file arrives.

Q: In BPMN, what is the difference between a pool and a lane?
A: A pool represents a participant or organization in the process — for example the transfer agent versus the asset manager — and a message flow (dashed arrow) crosses between pools. A lane is a subdivision within a single pool, used to organize activities by role or system, such as separating the validation team's tasks from the settlement team's. Sequence flow (solid arrow) stays within a pool; only message flow crosses pool boundaries.

Q: In BPMN, what is the difference between a sequence flow and a message flow, and how are they drawn?
A: A sequence flow is a solid line with a filled arrowhead showing the order of activities within a single pool. A message flow is a dashed line with an open arrowhead showing communication of a message between two separate pools (participants). Confusing the two is a classic reading error: a dashed arrow between the asset manager and the transfer agent is them exchanging a message, not one ordering the other's internal steps.

Q: What does DMN stand for, and what is its core idea?
A: DMN stands for Decision Model and Notation, an OMG standard whose core idea is to separate decision logic from process flow. Instead of burying complex business rules inside BPMN gateways, you factor them into reusable decision tables and a Decision Requirements Diagram. The current formal version is DMN 1.5, adopted in August 2024, with a 1.6 beta released the following month.

Q: Why separate DMN decision logic from a BPMN process flow?
A: Separating them keeps each notation readable and each piece independently maintainable: the BPMN process shows the flow, and a single BPMN task or "business rule task" calls out to a DMN decision that holds the rules. When a fund's fee-tier or eligibility rules change, you edit one decision table rather than rewiring gateways across the process. It also lets business users own and audit the rules directly, which matters in a regulated context.

Q: What is a DMN decision table, and what are its main parts?
A: A DMN decision table is a tabular representation of decision logic with input columns, output columns, and rows of rules. Each row maps a combination of input conditions to an output result, and a hit policy (such as Unique, First, or Collect) governs what happens when multiple rows match. For example, inputs of investor type and investment amount could map to an output fee tier.

Q: In a DMN decision table, what does the hit policy control, and what does a "Unique" hit policy require?
A: The hit policy controls how the table resolves when zero, one, or several rules match a given input. The Unique hit policy (often abbreviated U) requires that at most one rule matches any input — the rules must not overlap — which makes the table unambiguous and easy to validate. Other policies like First (F), Any (A), or Collect (C) handle overlaps differently, so reading the hit-policy marker is essential to knowing what output to expect.

Q: What is a DMN Decision Requirements Diagram (DRD)?
A: A DRD is the graph-level view in DMN that shows how decisions depend on one another and on their input data and knowledge sources. Decisions are rectangles, input data are rounded "stadium" shapes, and arrows show information requirements. It lets you read which sub-decisions feed a final decision — for instance, an "eligibility" decision and a "fee tier" decision both feeding a final "accept order" decision.

Q: What is FEEL in the context of DMN?
A: FEEL stands for Friendly Enough Expression Language, the standard expression language defined by the DMN specification for writing the conditions and computations inside decision tables and boxed expressions. It is designed to be readable by business users while still being unambiguous and executable. Knowing FEEL exists explains why DMN cells can contain ranges and expressions rather than just literal values.

Q: What does ArchiMate model, and who maintains it?
A: ArchiMate is an enterprise-architecture modeling language maintained by The Open Group, designed to describe and relate the business, application, and technology layers of an enterprise in one coherent language. The current version is ArchiMate 3.2. It is the notation a vendor or enterprise architect brings when they want to show how a business service is realized by applications and underlying infrastructure together.

Q: What are the three core layers of ArchiMate?
A: The three core layers are the Business layer (business processes, services, actors, and roles), the Application layer (application components and application services), and the Technology layer (nodes, system software, and infrastructure services). ArchiMate uses service orientation and realization relationships to connect concrete elements to more abstract ones across these layers. This layering is exactly what lets one diagram show a fund-accounting business service traced down to the servers that run it.

Q: Beyond the three core layers, what additional aspects does ArchiMate 3.x include?
A: ArchiMate 3.x adds Motivation elements (stakeholders, drivers, goals, requirements, principles), a Strategy layer (resources, capabilities, courses of action), a Physical layer (equipment, facilities, materials) extending technology, and an Implementation and Migration layer for transition planning. These extensions let the language express not just the current structure but why it exists and how it will change, which suits regulatory-driven transformation programs like a DORA remediation roadmap.

Q: What question does ArchiMate answer that BPMN cannot?
A: ArchiMate answers "how does our enterprise hang together across business, application, and technology" — it traces a business service down through the applications that realize it to the infrastructure that runs it. BPMN cannot do this; BPMN only models the flow of a single business process and has no concept of the application components or technology nodes underneath. You reach for ArchiMate for landscape and traceability, BPMN for a specific process flow.

Q: What does ERD stand for, and what does it model?
A: ERD stands for Entity-Relationship Diagram, the classic notation for data modeling — it shows entities (things you store data about), their attributes, and the relationships between them. It is the DBA's and data modeler's primary tool for designing and communicating a relational schema. For example, an ERD for a fund register would show Investor, Holding, Fund, and Transaction entities and the relationships linking them.

Q: In ERD crow's-foot notation, how is a many relationship shown, and how is "one"?
A: The crow's-foot notation draws the "many" end of a relationship as a three-pronged splayed line (the crow's foot) touching the entity, and the "one" end as a single perpendicular bar. Additional marks show optionality: a circle (O) near the end means zero is allowed, a bar means at least one is required. So a line reading "one bar" at Fund and a "crow's foot with a bar" at Holding reads as "one fund has one-or-many holdings."

Q: What does IDEF1X stand for, and where is it still mandated?
A: IDEF1X is a data-modeling notation from the IDEF (Integration DEFinition) family, originally developed for the U.S. Air Force ICAM program and standardized as a U.S. federal standard. It is a rigorous, relational-style entity-relationship notation still mandated in some U.S. government and defense data-modeling contexts. Its existence is why a single "ER diagram" can look quite different depending on whether it follows crow's-foot, Barker, or IDEF1X conventions.

Q: How does IDEF1X distinguish an independent entity from a dependent entity visually?
A: In IDEF1X an independent entity (one whose primary key does not depend on another entity) is drawn as a box with square corners, while a dependent (identifier-dependent) entity is drawn as a box with rounded corners. A dependent entity is one whose primary key includes a foreign key from its parent — for example a Transaction line whose identity depends on the parent Order. Reading the corner shape immediately tells you about identifying relationships.

Q: What does C4 stand for, and what are its four levels?
A: C4 is a software-architecture diagramming approach whose four levels are Context, Container, Component, and Code — the four C's. Each level zooms in one step: Context shows the system and its external users and systems, Container shows the deployable/runnable units and their technologies, Component shows the components inside a container, and Code shows the lowest-level implementation detail. It gives you a consistent way to describe one software system at increasing zoom.

Q: In C4, what does a "container" mean, and why is the term sometimes confusing?
A: In C4 a container is a separately runnable or deployable unit that executes code or stores data — for example a web application, an API service, a database, or a message broker. It is confusing because the word collides with Docker/OS containers, but C4 containers are about application boundaries, not virtualization. A C4 container diagram of a fund-data platform would show, say, the ingestion API, the NAV-calculation service, and the Postgres warehouse as separate containers.

Q: Which C4 level shows the highest-level view and to whom is it usually shown?
A: The System Context diagram is the highest-level C4 view; it shows your software system as a single box surrounded by its users (people) and the external systems it interacts with. It is aimed at the broadest audience, including non-technical stakeholders, because it deliberately hides internal detail. For a fund platform it might show the system, the operations users, and external systems like a SWIFT gateway and a market-data feed.

Q: What supplementary diagrams does the C4 model define beyond the four core levels?
A: Beyond Context, Container, Component, and Code, C4 defines supplementary diagrams: the System Landscape diagram (multiple systems in a broader enterprise context), the Dynamic diagram (runtime collaboration, similar in spirit to a sequence diagram), and the Deployment diagram (how containers map to infrastructure and environments). These cover questions the static four levels do not, such as runtime behavior and physical deployment.

Q: Is C4 a formal notation with fixed symbols like UML or BPMN?
A: No — C4 is deliberately notation-independent and tooling-independent; it prescribes the set of abstractions and the hierarchy of diagrams, not a strict shape grammar. You can render C4 with simple boxes and arrows as long as the diagram is self-explanatory with a clear title, key/legend, and meaningful labels. This contrasts with BPMN and UML, which define precise symbols whose meaning you must memorize.

Q: Which notation best answers "how does an order get from an EMT file to the register"?
A: BPMN is the best fit, because that is a business-process flow question — a sequence of activities, decisions, and handoffs across participants. You would model the inbound EMT/order file arriving as a message start event, validation tasks, gateways for accept/reject, and the register update as the end state. This is exactly the kind of "question to notation" mapping the lesson's cheat sheet is meant to capture.

Q: Which notation best answers "what fee tier applies to this investor and amount"?
A: DMN is the best fit, because that is decision logic, not process flow — it maps combinations of inputs (investor type, amount) to an output (fee tier) in a decision table. Embedding such rules in a BPMN gateway would be unreadable and hard to maintain, which is precisely why DMN separates decisions from process. The BPMN process would simply invoke the DMN decision at the right step.

Q: Which notation best answers "how is our fund-accounting business service realized by applications and infrastructure"?
A: ArchiMate is the best fit, because the question spans business, application, and technology layers and asks about realization — exactly what ArchiMate's layered service-oriented model expresses. No other notation in this lesson can trace a business service through the applications that realize it down to the technology that runs it. BPMN, DMN, ERD, and C4 each cover only one slice of that picture.

Q: Which notation best answers "what is the relational schema of the investor register"?
A: An ERD is the best fit, because schema design — entities, attributes, primary and foreign keys, and cardinalities — is precisely what entity-relationship diagrams express. A UML class diagram could approximate it, but the ERD is the conventional, more precise tool for relational databases. IDEF1X is an alternative ERD dialect you might encounter in stricter or government-adjacent contexts.

Q: Which notation best answers "how do the containers of our internal NAV platform fit together"?
A: C4, specifically the Container diagram, is the best fit because the question is about the high-level software architecture of one system — its runnable units and the technologies and communications between them. C4 was designed exactly to answer this at a glance for a development team. ArchiMate could also show it but operates at enterprise scope; C4 is lighter-weight and software-focused.

Q: You are shown a diagram with rounded-rectangle activities, diamonds, and pools with dashed arrows between them. Which notation is it?
A: This is BPMN. Rounded-rectangle activities (tasks), diamond gateways, swimlanes organized into pools, and dashed message flows crossing between pools are all distinctive BPMN constructs. The combination of pools-with-message-flows plus diamonds is the tell, since neither UML nor C4 uses that exact vocabulary.

Q: You are shown a diagram with lifelines and horizontal arrows where time flows downward. Which notation is it?
A: This is a UML sequence diagram. The defining features are vertical lifelines (often dashed) for participants, time flowing downward, and horizontal message arrows between lifelines in chronological order. Activation bars on the lifelines reinforce the identification. No other notation in this lesson uses a top-to-bottom time axis with lifelines.

Q: You are shown boxes connected by lines ending in three-pronged "crow's feet." Which notation is it?
A: This is an ERD using crow's-foot notation. The crow's-foot symbol at a relationship end denoting "many," combined with entity boxes listing attributes, is unmistakable. If the boxes instead had rounded versus square corners with no crow's feet, you would suspect IDEF1X rather than crow's-foot ERD.

Q: You are shown layered boxes labeled "business," "application," and "technology" with realization arrows. Which notation is it?
A: This is ArchiMate. The explicit business/application/technology layering with service-orientation and realization relationships connecting them is ArchiMate's signature. The distinctive element notation (such as the rounded-corner business-process arrow shape and layered color bands) further confirms it.

Q: You are shown a simple "boxes and arrows" diagram titled "Container diagram" with technologies labeled on each box. Which notation is it?
A: This is a C4 Container diagram. The tells are the explicit C4 level name in the title, plain boxes annotated with their technology choices, people and external systems around the edge, and clearly labeled relationships. Because C4 is notation-independent, the labels and title — not special symbols — are what identify it.

Q: A diagram uses a diamond with a plus sign and two outgoing arrows that both proceed. What does it mean in BPMN?
A: A diamond with a plus marker is a parallel (AND) gateway, and when used to split it activates all outgoing paths simultaneously — both branches proceed in parallel. Later a matching parallel gateway used as a join waits for all incoming branches before continuing. So this reads as "do both of these activities concurrently," for instance run the AML check and the sanctions check at the same time.

Q: A BPMN diamond marked with an "X" splits into three labeled outgoing flows. What does it mean?
A: A diamond marked with an X is an exclusive (XOR) gateway, and on a split it selects exactly one of the outgoing flows based on the conditions labeling each path. Only one branch is taken, unlike the parallel gateway. For an order, an XOR gateway might route to "accept," "reject," or "manual review" depending on validation results.

Q: In UML, what is the difference between association, aggregation, and composition?
A: All three are structural relationships between classes; the difference is ownership strength. A plain association (solid line) just means the classes are related; aggregation (hollow diamond) is a "has-a" whole-part relationship where the part can exist independently; composition (filled diamond) is a stronger whole-part where the part's lifetime is bound to the whole. For example, a Fund composes its NAV-history records (they die with the fund) but merely aggregates the Investors that hold it.

Q: In a UML class diagram, what do the symbols at the ends of an association line tell you?
A: They tell you multiplicity (cardinality) — how many instances of one class relate to one instance of the other, written as numbers or ranges like `1`, `0..1`, `1..*`, or `*`. They may also carry role names and navigability arrows. Reading `1` at Fund and `0..*` at Holding means one fund can have zero or many holdings, mirroring the cardinality an ERD would express with crow's feet.

Q: What does the multiplicity `1..*` mean on a UML association?
A: The multiplicity `1..*` means "one or more" — at least one instance and no fixed upper bound. The `*` (or `0..*`) means "zero or more," and `0..1` means "optional, at most one." So an Order with a `1..*` multiplicity to OrderLine means every order must have at least one line, a constraint you would also enforce in the database the ERD describes.

Q: How does an ERD relationship cardinality compare to a UML class diagram multiplicity?
A: They express the same idea — how many instances relate to how many — but with different notation: ERDs use crow's-foot (or other) symbols at the line ends, while UML uses numeric multiplicities like `1`, `0..1`, and `*`. A crow's foot with a circle equals UML `0..*`; a crow's foot with a bar equals UML `1..*`. Recognizing this equivalence is part of reading both notations without a translator.

Q: Why is BPMN described as "executable," and what makes that possible?
A: BPMN 2.0 is called executable because the standard defines not only the visual notation but also a precise XML serialization and execution semantics, so compliant process engines can run a BPMN model directly. This means a transfer-agency flow drawn in BPMN can, with the right attributes filled in, be deployed to an engine that orchestrates real tasks. UML and C4, by contrast, are descriptive and not executed by an engine.

Q: Can DMN decision tables be executed, and by what?
A: Yes — DMN, like BPMN, has standardized execution semantics, and DMN-compliant engines can evaluate decision tables and DRDs at runtime, often using FEEL for the expressions. This lets a fee-tier or eligibility decision be authored as a table and called by a running process. The separation means the rules can be changed and redeployed without altering the surrounding BPMN process.

Q: What is the relationship between BPMN and DMN in a typical design?
A: They are complementary: BPMN models the process flow and, at a decision point, delegates to DMN rather than encoding complex rules in a gateway. A BPMN "business rule task" invokes a DMN decision, which returns a result the process then acts on. This keeps the BPMN readable and the rules independently maintainable and auditable — valuable when fund rules must be traced for a regulator.

Q: Why might an architect prefer C4 over UML for explaining a system to a mixed audience?
A: C4 offers a small, consistent set of abstractions (Context, Container, Component, Code) with a clear zoom hierarchy and no specialized symbol grammar to learn, so a mixed business-and-technical audience can follow it quickly. UML is precise but its breadth and strict symbols can overwhelm non-specialists. C4's self-explanatory boxes-and-arrows with explicit legends trade formal rigor for communication clarity.

Q: What does a C4 Component diagram show, and what is the common pitfall in keeping it?
A: A C4 Component diagram zooms inside one container to show its components — groupings of related functionality — and how they collaborate. The common pitfall is that component-level diagrams drift out of date fastest because they are closest to the code, so many teams generate or maintain only Context and Container diagrams by hand and derive lower levels from code when possible. This is why C4 emphasizes keeping diagrams cheap to produce.

Q: What does a C4 Code diagram correspond to, and how often is it drawn?
A: The C4 Code diagram is the lowest level, corresponding roughly to a UML class diagram or entity-relationship view of the implementation detail inside a single component. In practice it is rarely drawn by hand because it is volatile and high-effort; when needed it is usually generated from the codebase by an IDE or tool. C4's author recommends most teams stop at the Component level.

Q: Why is the lesson's "notation chooser" cheat sheet structured as question to notation to tool?
A: Because the architect's real skill is matching the question being asked to the notation that answers it and then to a tool that produces it — not memorizing every symbol of every language. A row reads, for example, "how does an order get from EMT file to register?" to BPMN to a BPMN tool. Building this one page forces you to articulate which question each notation answers and, crucially, which it cannot.

Q: Which notation answers "what decision logic applies" and which answers "what is the process flow"?
A: DMN answers "what decision logic applies" via decision tables and DRDs, while BPMN answers "what is the process flow" via activities, gateways, and events. The whole point of DMN's existence is to pull the decision logic out of the BPMN flow so each stays clear. In a fund order pipeline, BPMN shows the steps and DMN holds the eligibility and fee rules invoked at one of those steps.

Q: Which notation answers "what data do we store and how is it related," and name two dialects of it.
A: The ERD (Entity-Relationship Diagram) answers what data is stored and how entities relate, through entities, attributes, keys, and cardinalities. Two common dialects are crow's-foot notation (widely used in commercial tools) and IDEF1X (rigorous, still mandated in some U.S. government/defense contexts). Barker notation is a third dialect you may also encounter.

Q: Which two notations in this lesson can both depict runtime interaction over time, and how do they differ?
A: The UML sequence diagram and the C4 Dynamic diagram both depict runtime interaction. The sequence diagram is a formal UML notation with lifelines, activation bars, and strict message semantics; the C4 Dynamic diagram is a lighter, notation-independent view that numbers interactions on a Context- or Container-level picture. Use the sequence diagram for rigor, the C4 Dynamic diagram for an accessible runtime walk-through.

Q: A vendor's slide shows business processes realized by application components realized by technology nodes. What can you infer about the right phase to study it deeply?
A: The slide is ArchiMate, and the lesson explicitly defers deep ArchiMate practice to Phase 8; at this point you only need to read it and name it. The layered realization chain — business to application to technology — is the recognizable signature. Knowing it is ArchiMate lets you engage the vendor on the same terms even before you can author such models yourself.

Q: In ArchiMate, what does a "realization" relationship express?
A: A realization relationship expresses that a more concrete element realizes (provides the concrete implementation of) a more abstract element — for instance an application component realizes an application service, or an application service realizes a business service. It is the backbone that lets ArchiMate connect the layers vertically. Reading realization arrows is how you trace a business capability down to the system that delivers it.

Q: What is the danger of using BPMN gateways to encode complex business rules instead of DMN?
A: Encoding complex rules as nested gateways produces a tangled, hard-to-read "gateway spaghetti" process that mixes flow with logic and is painful to change or audit. DMN exists precisely to avoid this by holding the rules in a table the process calls out to. In a regulated fund context, the auditable, single-table form of DMN is far easier to review than rules scattered across branches.

Q: What is the difference between a UML use-case diagram and a UML class diagram (awareness)?
A: A use-case diagram shows actors and the use cases (goals) a system supports — a high-level "what can users do" view — whereas a class diagram shows the internal static structure of types and their relationships. Use-case diagrams capture functional scope from the user's perspective; class diagrams capture the design's structure. At reading level, recognize the stick-figure actors and ovals of a use-case diagram versus the attribute/operation boxes of a class diagram.

Q: When narrating a BPMN diagram cold, what should you identify first?
A: First identify the pools and lanes to know who the participants and roles are, then find the start event(s) to know what triggers the process, then trace sequence flow through tasks and gateways to the end event(s). Reading in that order — actors, trigger, flow, outcome — keeps you oriented. For a transfer-agency flow you would say, for example, "the transfer agent's pool starts on receipt of an order message, validates, branches on an XOR gateway, and ends with a register update."

Q: When narrating a UML sequence diagram cold, what should you identify first?
A: First identify the participants across the top (the lifelines) and the scenario the diagram represents, then read the messages top to bottom to follow the order of interaction, noting synchronous versus asynchronous arrows and any return messages. The downward time axis is the key to narration. You would say, for instance, "the scheduler calls the NAV service, which synchronously requests prices from the feed, then asynchronously notifies the register."

Q: When shown an unfamiliar diagram, what three steps does the lesson's self-test prescribe?
A: The self-test is to take three unfamiliar diagrams, (1) identify each notation, (2) name it correctly, and (3) narrate what each diagram actually says. This rehearses the real workshop skill of reading a notation cold without a translator. The "Done when" criteria require exactly this: name the notation and read it correctly when shown a diagram cold.

Q: Why is recognizing the notation a prerequisite to narrating the diagram?
A: Because the same shapes mean different things in different notations — a diamond is a gateway in BPMN but a decision in a flowchart, a box can be a class, an entity, a container, or an ArchiMate element. Until you know which language you are reading, you cannot reliably interpret the symbols. Naming the notation first is what prevents you from mis-narrating a vendor's ArchiMate as if it were a C4 diagram.

Q: Where do the authoritative specifications for UML, BPMN, and DMN live, and what should you read at this stage?
A: They live on the OMG website at omg.org under the respective spec pages (UML, BPMN, DMN). At this stage you read only the intro/overview sections of each spec plus one example diagram each — not the full multi-hundred-page documents. The aim is reading fluency and one worked example per notation, captured into your reference folder.

Q: Where is the authoritative source for ArchiMate, and for C4?
A: ArchiMate's authoritative source is The Open Group's ArchiMate specification and overview pages (opengroup.org / pubs.opengroup.org), with the current version being 3.2. C4's authoritative source is c4model.com, the site maintained by the model's creator. ERD/IDEF1X grounding for this lesson comes from the DAMA-DMBOK, chapter 5.

A: The first deliverable is to collect one example diagram of each notation — UML, BPMN, DMN, ArchiMate, ERD/IDEF1X, and C4 — drawn from the spec intros and c4model.com, into a single reference folder. Having a concrete example of each in one place is what makes the notations recognizable later. It is the raw material for the cheat sheet and the cold-reading self-test.

A: Each row maps a question to a notation to a tool, and must include a fund-industry example question. For instance: "how does an order get from EMT file to register?" to BPMN to a BPMN tool. Adding the fund-flavored question per row is one of the explicit "Done when" criteria, ensuring the sheet is grounded in your real domain.

Q: Why does the lesson stress that the architect reads BA, vendor, and DBA diagrams "without a translator"?
A: Because in fund-industry workshops different roles bring different notations — the BA brings BPMN, the vendor brings ArchiMate, the DBA brings an ERD — and the architect is the one person expected to read all of them. Needing a translator slows decisions and cedes authority. The lesson's reading-fluency goal is precisely to make you that fluent reader across notations.

Q: Which notation is the natural home for "order lifecycle" and "transfer-agency flows" per the syllabus?
A: BPMN is named in the syllabus as the executable business-process notation for the order lifecycle and transfer-agency flows. These are sequences of activities, decisions, and handoffs — exactly process-flow questions. You would model them as BPMN processes with events, tasks, gateways, and lanes for the responsible teams.

Q: How does a UML class diagram differ from an ERD even though both can show entities and relationships?
A: A UML class diagram can include behavior (operations/methods) and object-oriented relationships such as inheritance and composition, and it models classes in a software design. An ERD focuses purely on data — entities, attributes, keys, and cardinalities — and maps naturally to a relational schema. For database design the ERD is conventional and precise; for object-model design the class diagram fits better.

Q: What does it mean that a notation "answers some questions but not others," and give an example with C4 and DMN.
A: It means each notation has a defined scope and using it outside that scope produces a poor or impossible diagram. C4 answers "how is this one software system structured" but cannot express business decision logic; DMN answers "what is the decision rule" but cannot show software containers or deployment. The cheat sheet's value is recording both the question each notation answers and the ones it cannot.

Q: In BPMN, what does an envelope marker inside a start-event circle indicate?
A: An envelope marker inside a start event indicates a message start event — the process is triggered by the arrival of a message. In a fund pipeline this typically models the inbound order or EMT file arriving and kicking off the transfer-agency process. Other markers, like a clock, would indicate a timer start (e.g., a scheduled end-of-day run).

Q: What is a "boundary event" in BPMN at a reading level?
A: A boundary event is an event attached to the edge of an activity that can interrupt or branch off from it when triggered — for example a timer on a task that fires if the task runs too long, or an error event that catches a failure. You read it as "while this activity runs, if X happens, take this alternate path." It lets a process model exceptions like a timeout on waiting for a settlement confirmation.

Q: Why is image pinning unrelated here — what's the genuine cross-notation risk when reading a diagram with no legend?
A: The genuine risk is misidentifying the notation or the cardinality because, without a legend, the same shape (a box, a diamond, a line end) is ambiguous across notations. C4 explicitly requires every diagram to carry a title and key/legend for exactly this reason. When reading a legend-less diagram, your first job is to infer the notation from distinctive constructs before trusting any symbol's meaning.

Q: How would you model "should this subscription order be accepted" cleanly across BPMN and DMN?
A: Model the order's journey as a BPMN process — receive order, validate, settle, update register — and at the validation step place a BPMN business rule task that invokes a DMN "accept order" decision. The DMN decision's DRD might combine an eligibility decision and a fee-tier decision, each a table evaluated with FEEL. This keeps the flow readable in BPMN and the rules auditable in DMN, ideal for a regulated transfer-agency setting.

Q: What is the ArchiMate notion of a "service," and why is it central?
A: In ArchiMate a service is an externally visible unit of functionality that an element exposes to others, defined at each layer (business service, application service, technology service). It is central because ArchiMate is service-oriented: higher layers consume the services that lower layers provide, and realization relationships connect them. This is what lets a business service be traced to the application and technology services that deliver it.

Q: When would IDEF1X be the mandated notation rather than crow's-foot ERD?
A: IDEF1X is mandated chiefly in certain U.S. federal and defense data-modeling environments where it is the prescribed standard, owing to its origin in the Air Force ICAM program and its later federal standardization. In commercial and fund-industry settings, crow's-foot or Barker notation is far more common. Knowing IDEF1X exists prevents you from being thrown when a government-adjacent system delivers models in that dialect.

Q: What is a DRD "knowledge source," and what does it represent?
A: In a DMN Decision Requirements Diagram, a knowledge source represents the authority or origin of the decision logic — a person, document, regulation, or policy that the decision must comply with. It is drawn as a shape with a wavy bottom edge and connected to decisions by an authority requirement. For a fund decision, a knowledge source might be the UCITS eligibility rules or the fund prospectus that governs the logic.

Q: Why does the lesson place this notations overview before Phase 4 (BPM) and Phase 8 (ArchiMate)?
A: Because reading fluency across all the notations is a prerequisite to going deep on any one of them: Phase 4 dives into BPM/BPMN and Phase 8 practices ArchiMate, and both assume you can already recognize and read the surrounding notations. Establishing the map first prevents later phases from stalling on "what am I even looking at." It also makes you immediately useful in mixed workshops well before the deep phases.

Q: A diagram has stick-figure actors connected to ovals inside a system boundary box. Which notation and what does it show?
A: This is a UML use-case diagram; the stick figures are actors and the ovals are use cases representing goals the system supports, with the box marking the system boundary. It answers "who uses the system and for what" at a high level, not how it is built. Recognize it as distinct from a sequence diagram (which has lifelines and a time axis).

Q: What is the practical difference between "reading" a notation and "authoring" one, as this lesson frames it?
A: Reading means you can look at someone else's diagram, name the notation, and correctly narrate what it says; authoring means you can produce correct, well-formed diagrams from scratch. This lesson targets reading only, because in workshops you mostly consume others' diagrams. Authoring fluency in BPMN and ArchiMate is built later, deliberately, in Phases 4 and 8.

Q: Which notation would a DBA most likely bring to a fund-data workshop, and what would it show?
A: A DBA would most likely bring an ERD, showing the relational schema — entities like Fund, Investor, Holding, and Transaction, their attributes and keys, and the cardinalities between them. It answers what data is stored and how it is related, the DBA's core concern. As the architect you must read it fluently alongside the BA's BPMN and the vendor's ArchiMate.

Q: What is the risk of treating a C4 diagram as if it follows a strict symbol standard?
A: Because C4 is notation-independent, two C4 diagrams of the same system may use different shapes and colors, so assuming a fixed symbol grammar leads you to over-read meaning into styling. What carries meaning in C4 is the level (Context/Container/Component), the labels, the relationships, and the legend — not the specific box shape. Read the title and key first to know which level and conventions apply.

Q: How do BPMN message flows help you read an inter-organization fund process?
A: Message flows (dashed arrows crossing pools) show the messages exchanged between separate organizations — for example the asset manager sending an order to the transfer agent, and the transfer agent returning a confirmation. Following them tells you who communicates what to whom, independent of each party's internal steps (sequence flow). This is essential for reading multi-party flows like subscription, redemption, and settlement.

Q: Why is DMN's separation of logic especially valuable in a DORA/regulated context?
A: Because regulators and auditors need to see and verify decision rules clearly, and a self-contained DMN decision table is far easier to inspect, version, and trace than rules buried in process branches or code. DMN also lets business and compliance owners review the logic directly. This auditability and changeability aligns with DORA's emphasis on governance and traceability of automated decisions.

Q: What does a "lane" tell you that a "task" alone does not in BPMN?
A: A lane tells you which role, team, or system is responsible for the tasks placed inside it, adding the "who" to the "what." A task alone says an activity happens; the lane it sits in says who performs it. So reading lanes lets you say, for example, that validation is the operations team's task while final approval sits in the compliance lane.

Q: How can you tell a UML class diagram from a C4 Code diagram if both show classes?
A: You often cannot tell purely from the shapes, because a C4 Code diagram is typically rendered as a UML class (or ER) diagram — C4 borrows existing notations at that level. The distinguishing context is the diagram's stated level and its place in a C4 hierarchy; a Code diagram is labeled as such and zooms into one C4 component. Absent that framing, treat class-box-with-attributes-and-operations as UML.

Q: What is the minimum each diagram should carry so that a cold reader can interpret it, per C4's guidance?
A: Per C4, every diagram should have a clear, descriptive title, a key/legend explaining the notation and any colors or shapes, and meaningful labels on both elements and relationships. This makes the diagram self-explanatory to someone seeing it for the first time. The principle generalizes: a legend-less diagram forces the reader to guess the notation, which is exactly the failure mode this lesson trains you to avoid.

Q: In ArchiMate, what is the Motivation extension used to capture?
A: The Motivation elements capture the "why" behind an architecture: stakeholders, drivers, assessments, goals, outcomes, principles, requirements, and constraints. They let a model justify why elements exist and what they must satisfy, linking architecture to business intent. In a regulated program you might model a DORA requirement as a driver and trace it to the application changes that satisfy it.

Q: What does the Implementation and Migration layer add to an ArchiMate model?
A: It adds elements for planning change — work packages, deliverables, implementation events, plateaus (stable intermediate states), and gaps — so the model can express the transition from a current to a target architecture. This turns ArchiMate from a static snapshot into a roadmap. It is how you would show, for instance, the phased migration of a fund-accounting platform to a new application landscape.

Q: Summarize the one-line scope of each of the six notations in this lesson.
A: UML covers system structure (class) and interaction behavior (sequence); BPMN covers executable business-process flow; DMN covers decision logic separated from flow; ArchiMate covers business, application, and technology layers in one enterprise language; ERD/IDEF1X covers data modeling and relational schema; and C4 covers software architecture at four zoom levels. Each answers a distinct question, and the architect's skill is matching question to notation.

Q: Why is "narrate what the diagram says" a better test of fluency than "name the notation"?
A: Naming proves only recognition, whereas narrating proves you can actually decode the symbols into meaning — read the flow, the cardinalities, the layers, or the rules. The lesson's "Done when" deliberately requires both: name it and read it correctly. Being able to walk a stakeholder through an unfamiliar BPMN or ArchiMate diagram on the spot is the real workshop capability.

Q: If a single diagram tried to show process flow, decision rules, data schema, and infrastructure at once, what would be wrong?
A: It would violate the principle that each notation answers a bounded set of questions — cramming process (BPMN), rules (DMN), data (ERD), and infrastructure (ArchiMate/C4) into one diagram makes all of them unreadable. The correct approach is separate, linked diagrams in the right notation for each concern, as DMN's split from BPMN exemplifies. Recognizing this is part of choosing notations well, the heart of the cheat-sheet exercise.

Q: In a UML class diagram, what does a hollow triangle arrowhead pointing to one class mean?
A: A hollow (unfilled) triangle arrowhead denotes generalization — inheritance — pointing from the subclass to its superclass; it reads "is a kind of." For instance an EquityFund and a BondFund class might both generalize to a Fund superclass, sharing its attributes and operations. Do not confuse it with the open stick arrowhead used for navigation or the diamonds used for aggregation and composition.

Q: How does an ISIN-keyed instrument reference data model illustrate the difference between an ERD and a BPMN diagram?
A: An ERD of instrument reference data shows the static structure — an Instrument entity keyed by ISIN, related to Issuer, Currency, and Price entities with their cardinalities — answering "what do we store and how is it related." A BPMN diagram instead shows the process of how that ISIN gets onboarded, enriched, and validated over time, answering "what happens in what order." The same fund-data domain needs both: the ERD for the schema and the BPMN for the workflow that populates it.

Q: How would you use these notations together to document an end-to-end fund subscription, naming each?
A: Use BPMN for the subscription process flow across the asset manager and transfer-agent pools; DMN for the eligibility and fee-tier decisions invoked within it; an ERD for the register schema the process writes to; ArchiMate to show the business service realized by the applications and infrastructure; and C4 to detail the internal software architecture of your own platform. Each notation answers its own question, and together they form a coherent, role-readable picture.

Q: In BPMN, how is an inclusive (OR) gateway drawn and how does it differ from exclusive and parallel gateways?
A: An inclusive gateway is a diamond containing a hollow circle (O) marker, and on a split it activates every outgoing path whose condition evaluates true — so one, several, or all branches may proceed. This differs from the exclusive (X) gateway, which takes exactly one path, and the parallel (`+`) gateway, which always takes all paths unconditionally. For a redemption order you might use an inclusive gateway to fire both a large-trade notification and a tax-reporting branch when both thresholds are crossed, but only the ones that apply.

Q: What is a BPMN event-based gateway, and when would you read one in a fund settlement flow?
A: An event-based gateway is a diamond enclosing a pentagon-in-a-circle marker, and instead of branching on data it waits and routes based on whichever intermediate event (message, timer, signal) occurs first. You would read one where a process waits after sending an instruction and proceeds down the "confirmation received" path or the "timeout expired" path depending on what happens. In SWIFT settlement this models awaiting an `MT54x`-style confirmation versus a timer firing if the counterparty is silent.

Q: A BPMN diamond contains an asterisk-like marker. What gateway is it and what does it do?
A: A diamond marked with an asterisk (a multi-point star, `*`) is the complex gateway, used for branching or merging behavior that the simpler gateways cannot express — for example "continue once any 3 of 5 incoming branches arrive." It is rare and a tell that the modeller needs nonstandard synchronization. When reading one, look for an annotated condition because the merge/split rule is not implied by the symbol alone.

Q: In BPMN, what is the difference between a thrown and a caught intermediate event, and how do their markers differ?
A: A caught (catching) intermediate event waits for something to happen and is drawn with a hollow/unfilled marker, while a thrown event produces something and is drawn with a filled/solid marker. For example a catching message event (hollow envelope) waits for an incoming confirmation, whereas a throwing message event (filled envelope) sends one out. Reading the fill tells you whether the process is blocking on input or emitting a signal mid-flow.

Q: In BPMN, how does a non-interrupting boundary event differ visually and behaviorally from an interrupting one?
A: An interrupting boundary event is drawn with a solid (single-line) circle border and cancels the activity it is attached to when it fires, diverting the flow; a non-interrupting boundary event uses a dashed circle border and spawns a parallel path while the original activity continues. For a "wait for settlement confirmation" task, a non-interrupting timer could send a chaser reminder without aborting the wait, whereas an interrupting timer would cancel and escalate.

Q: In BPMN, what does a terminate end event look like and what does it do that a plain end event does not?
A: A terminate end event is a thick-bordered end circle containing a solid filled circle, and reaching it immediately ends the entire process instance — killing all other active tokens — rather than just ending its own branch like a plain end event. You read it as "stop everything now," useful for a hard reject where parallel AML and pricing branches must be abandoned. A plain end event only consumes the token on its own path, letting other branches finish.

Q: What is a BPMN call activity, and why might it appear in a transfer-agency model?
A: A call activity is an activity that invokes a separately defined, reusable global process or task, drawn as a rounded rectangle with a thick (bold) border. It differs from an embedded sub-process because the called process is defined once and reused across many models. A transfer-agency model might use a call activity for a shared "KYC/AML check" process that subscription, redemption, and transfer flows all invoke identically.

Q: In a DMN decision table, what does the Priority (`P`) hit policy do when several rules match?
A: The Priority hit policy permits multiple rules to match but returns a single output, choosing the one highest in a predefined ordering of output values rather than by rule order. You define the priority by listing output values from most to least preferred, and the matching rule with the most-preferred output wins. For a risk-rating decision, overlapping rules could all match an investor but `P` ensures the most severe rating (e.g. `High` over `Medium`) is the one returned.

Q: In a DMN decision table, what does the First (`F`) hit policy return and what is its caveat?
A: The First hit policy returns the output of the first matching rule in top-to-bottom row order, even if later rules also match. Its caveat is that the result depends on row order, so reordering rows can silently change behavior — making `F` harder to audit than `Unique`. It is convenient for "fall-through" logic like fee waterfalls, but in a regulated fund context auditors often prefer `U` so correctness does not hinge on ordering.

Q: In DMN, what does the Collect (`C`) hit policy do, and what do the `C+`, `C<`, `C>`, and `C#` variants mean?
A: The Collect hit policy gathers the outputs of all matching rules into a list, and aggregator suffixes reduce that list to one value: `C+` sums them, `C<` returns the minimum, `C>` returns the maximum, and `C#` returns the count. Plain `C` (no suffix) just returns the unordered list. For computing a total fee from several applicable charge rules, `C+` would sum each matched charge into one amount.

Q: In DMN, what does the Any (`A`) hit policy require, and how does it differ from Unique?
A: The Any hit policy allows multiple rules to match but requires that all matching rules produce the same output value; if they agree it returns that value, and if they disagree it is an error. Unique (`U`), by contrast, forbids overlap entirely — at most one rule may match. `A` is useful when overlapping conditions are unavoidable but you can guarantee they never conflict on the result.

Q: In a DMN decision table, what does a dash (`-`) in an input cell mean?
A: A dash in an input cell means "irrelevant" — that input does not constrain the rule, so the rule matches regardless of that input's value. It is DMN's way of writing a don't-care condition, keeping tables compact. For a fee rule that applies to all investor types above a threshold, the investor-type cell would hold `-` while the amount cell holds the range.

Q: In DMN FEEL, how do you write a numeric range such as "1000 up to but not including 5000" in an input cell?
A: FEEL uses interval notation with square and round brackets, so "from 1000 inclusive up to 5000 exclusive" is written `[1000..5000)` — a square bracket includes the endpoint and a round bracket (or reversed bracket) excludes it. This lets adjacent fee tiers tile a number line without overlap, e.g. `[0..1000)` then `[1000..5000)`. Getting the bracket direction wrong is a classic source of gap or overlap bugs that a `U` hit policy would then flag.

Q: What is the completeness requirement on a DMN decision table, and why does it matter for a fund?
A: Completeness means the table's rules cover every possible combination of inputs so that some rule always matches; an incomplete table can return no result for an unanticipated input. It matters because an order whose attributes fall through every rule would yield a null decision, stalling or mis-routing the workflow. Modellers add a catch-all rule (all inputs `-`) or verify completeness with tooling to guarantee, for instance, that every investor-amount combination maps to a fee tier.

Q: In ArchiMate, how is the serving relationship drawn and what does it mean?
A: The serving relationship is drawn as a solid line with an open (line) arrowhead, and it means one element provides functionality that serves (is used by) another — capturing a usage or "provides a service to" dependency. For example an application service serves a business process, or a technology service serves an application component. It replaced the older "used by" relationship and is how you read who consumes whom across layers.

Q: In ArchiMate, how do composition and aggregation relationships differ visually?
A: Composition is drawn as a solid line with a filled (black) diamond at the whole/parent end, indicating the parts cannot exist independently of the whole; aggregation uses a solid line with a hollow (open) diamond, indicating the parts can exist on their own. This mirrors UML's diamond convention. You would read a filled diamond from an application "system" to its modules as "these modules belong to and die with the system."

Q: In ArchiMate, what does the assignment relationship express and how is it drawn?
A: The assignment relationship expresses allocation of responsibility or execution — an active element performs a behavior or a node hosts software — and is drawn as a line with a filled (solid) ball at the source end and an arrowhead at the target. For instance, it assigns a business role to the business process it performs, or a server node to the system software it runs. Reading it tells you "this actor/node is the one that actually does/hosts this."

Q: In ArchiMate, what do the triggering and flow relationships show, and how do they differ from serving?
A: A triggering relationship (solid line, filled arrowhead) shows a temporal or causal sequence — one behavior triggers the next — while a flow relationship (dashed line, filled arrowhead) shows the transfer of information or value between behaviors. Serving, by contrast, is a usage dependency, not a sequence. In a fund process view, triggering reads as "validation then settlement," and flow reads as "the order data passes from intake to enrichment."

Q: In ArchiMate, what is the access relationship and where would it appear in a fund-data model?
A: The access relationship is a dotted line with an arrowhead that connects a behavior element to a data/business object it reads, writes, or accesses, with the arrow direction (or both-ended) indicating read versus write. It appears wherever you want to show which process or application service touches which data object. For example a "calculate NAV" application service would have a read access to the prices object and a write access to the NAV-history object.

Q: In ArchiMate, what is a "junction" and why is it used?
A: A junction is a small filled or hollow circle used to combine or split relationships of the same type — for example to show that one source serves two targets via a shared connector, or that several elements jointly compose a whole. It keeps diagrams tidy by avoiding many parallel lines. Reading a junction means "these relationships are grouped," not that there is an extra element.

Q: How does ArchiMate's "specialization" relationship compare to UML generalization?
A: ArchiMate's specialization relationship means one element is a particular kind of another and is drawn, like UML generalization, as a solid line with a hollow (unfilled) triangle arrowhead pointing to the more general element. It is the cross-layer analogue of inheritance — e.g. a "premium client" business role specializes a generic "client" role. Recognizing the shared triangle convention helps you read inheritance consistently across UML and ArchiMate.

Q: In a UML class diagram, what does a dashed line with an open arrowhead between two classes mean?
A: A dashed line with an open (stick) arrowhead is a dependency — it means the source class uses or depends on the target such that a change to the target may affect the source, without holding a persistent reference. It is weaker than an association (a solid line), which implies a structural link. You would draw a dependency from a `NavCalculator` to a `PricingService` it merely calls, versus an association to the `Fund` it holds a reference to.

Q: In UML, how is a realization (interface implementation) relationship drawn, and how does it differ from generalization?
A: Realization is drawn as a dashed line with a hollow (unfilled) triangle arrowhead pointing to the interface or abstract specification being implemented. It differs from generalization, which uses the same triangle but with a solid line, and from realization's meaning: generalization is "is a kind of" (inheritance) while realization is "implements the contract of." A `PostgresRegisterRepository` realizing a `RegisterRepository` interface is the classic example.

Q: In UML, what is an abstract class shown as, and why does it matter when reading a fund domain model?
A: An abstract class is conventionally shown with its name in italics (or with the `{abstract}` constraint), signalling it cannot be instantiated directly and exists to be specialized. It matters because it tells you the model expects concrete subclasses — for instance an abstract `Fund` with concrete `UCITSFund` and `AIF` subclasses. Misreading an abstract base as an instantiable type would lead you to expect objects that never exist on their own.

Q: In UML, what is the visibility meaning of `+`, `-`, `#`, and `~` prefixes on attributes and operations?
A: They denote access visibility: `+` is public (visible everywhere), `-` is private (visible only within the class), `#` is protected (visible to the class and its subclasses), and `~` is package-level (visible within the same package). Reading these tells you the intended encapsulation — for example a `-navHistory` private attribute versus a `+calculateNav()` public operation. They are a quick cue to a class's interface versus its internals.

Q: In a UML sequence diagram, what does a "combined fragment" labeled `alt` or `loop` represent?
A: A combined fragment is a boxed region of a sequence diagram with an operator label in the top-left corner; `alt` encloses mutually exclusive alternative interaction paths separated by a guard, and `loop` encloses messages that repeat. They add control flow to an otherwise linear message sequence. Reading an `alt` with guards `[valid]` and `[invalid]` tells you the NAV service behaves differently depending on a price-check result.

Q: In a UML sequence diagram, what does an `X` at the bottom of a lifeline indicate?
A: An `X` (a cross) placed on a lifeline marks a destruction occurrence — the participant is deleted or stops existing at that point in time. The lifeline ends at the cross rather than continuing to the bottom of the diagram. You read it as "this object/connection is torn down here," for example a session or temporary calculation context being released after end-of-day processing completes.

Q: In ERD modeling, what is an associative entity (junction table) and when does it appear?
A: An associative entity resolves a many-to-many relationship by becoming its own entity that holds the foreign keys of both parents (and often relationship attributes). It appears whenever two entities relate many-to-many — for example Investor and Fund relate through a Holding associative entity carrying units and acquisition date. In crow's-foot notation it sits between the two parents with a crow's foot toward it from each.

Q: In ERD/IDEF1X, what distinguishes an identifying relationship from a non-identifying one?
A: In an identifying relationship the child's primary key includes the parent's key (the child cannot be identified without the parent), and IDEF1X draws it as a solid line to a round-cornered dependent entity; a non-identifying relationship, where the parent key is only a non-key foreign attribute, is drawn as a dashed line to a square-cornered independent entity. A Transaction whose key is `(OrderID, LineNo)` has an identifying relationship to Order. Reading line style and corner shape tells you whether identity is inherited.

Q: In an ERD, what is a weak entity and how is it depicted?
A: A weak entity is one that cannot be uniquely identified by its own attributes alone and depends on a parent (strong) entity for identification, using part of the parent's key as a partial/discriminator key. In classic Chen notation it is drawn with a double-bordered rectangle and a double-bordered (identifying) relationship diamond; in IDEF1X it is the round-cornered dependent entity. A fund's `DistributionLine` keyed by its parent `Distribution` plus a sequence number is a weak entity.

Q: In ERD crow's-foot notation, how do you read a relationship end drawn as a circle followed by a crow's foot?
A: A circle (zero) next to a crow's foot (many) reads as "zero or many" — optional on that side with no upper bound — equivalent to UML `0..*` or `*`. By contrast a bar next to a crow's foot reads "one or many" (`1..*`). So a Fund-to-Holding line with circle-plus-crow's-foot at the Holding end means a fund may have no holdings or many, a key distinction when validating whether an empty fund is allowed.

Q: In ERD notation, how is an attribute's primary key versus foreign key typically annotated?
A: Conventions vary, but a primary-key attribute is commonly marked `PK` (or underlined), and a foreign key is marked `FK`; an attribute that is both part of the PK and an FK may be shown `PK,FK`. These annotations let you read identifying relationships at a glance. In a Holding entity you would expect `PK` on a surrogate or composite key and `FK` references to Investor and Fund.

Q: What is the difference between a logical and a physical ERD, and why does it matter for a fund register?
A: A logical ERD models entities, attributes, and relationships independent of any database technology (using business names and conceptual types), while a physical ERD adds implementation detail — actual table and column names, datatypes, indexes, and platform-specific keys. It matters because the logical model is the one business stakeholders validate, and the physical model is what the DBA builds. A logical "Investor" entity becomes a physical `investor` table with `varchar`/`numeric` columns and a chosen surrogate key.

Q: In C4, how are people (actors) typically distinguished from software systems and containers?
A: In the common C4 notation people are drawn as stick figures or person-shaped boxes, software systems and containers as rectangles, and external/out-of-scope systems are usually shaded or styled differently (often grey) from the system in focus. Because C4 is notation-independent, the legend defines the exact styling, but the convention is "people on the edges, your system's boxes in the middle, external systems greyed." This lets a reader instantly separate users, your scope, and dependencies.

Q: In C4, how should a relationship arrow between two elements be labeled, and why?
A: A C4 relationship arrow should carry a label describing the nature and often the technology/protocol of the interaction — for example "Sends order file to [SFTP]" or "Reads prices from [HTTPS/JSON]." Unlabeled arrows are a common C4 anti-pattern because the reader cannot tell what the dependency actually is. Direction and label together answer "who calls whom, to do what, over what."

Q: What is "Structurizr," and how does it relate to the C4 model?
A: Structurizr is the tooling created by Simon Brown (C4's author) for defining C4 models as code/text and rendering the Context, Container, and Component diagrams from a single model. Its key idea is a single source of truth from which multiple diagrams are generated, so they stay consistent. Knowing it exists explains the C4 emphasis on cheap-to-maintain, generated diagrams rather than hand-drawn ones that drift.

Q: How would you render a quick BPMN-style or C4-style diagram as text in a Git repo, and name one tool for each?
A: You use a diagram-as-code tool so the source lives in version control and renders to an image — for example PlantUML or Mermaid for UML/sequence/ER/C4-ish diagrams, and Structurizr DSL specifically for C4. Mermaid in particular renders directly in Markdown on platforms like GitHub. This keeps the diagram diffable and reviewable alongside the schema or pipeline code it documents, which fits a regulated, auditable workflow.

Q: Name a commonly used dedicated tool for authoring BPMN and one for DMN, and what file format they share.
A: Camunda Modeler (and the bpmn.io toolkit behind it) is a widely used free BPMN and DMN authoring tool, and Signavio and Trisotech are common commercial ones. BPMN models serialize to standardized BPMN 2.0 XML and DMN models to DMN XML, both OMG-defined interchange formats. Because the XML is standardized, a model authored in one compliant tool can in principle be opened or executed in another.

Q: You open a `.bpmn` file and it is XML, not an image. Why, and what does this enable?
A: BPMN 2.0 defines a standardized XML serialization, so the canonical form of a BPMN model is XML containing both the process semantics and the diagram interchange (`BPMNDI`) layout, with rendering tools turning it into the visual. This enables version control, programmatic generation, and direct deployment to an execution engine. It also means two tools can exchange the same process, though layout fidelity can vary.

Q: You are handed a diagram with boxes containing three compartments (name, attributes, operations) but no crow's feet. Which notation, and what is the giveaway?
A: It is a UML class diagram; the three-compartment box (class name on top, attributes in the middle, operations at the bottom) is the unmistakable giveaway, and the absence of crow's feet rules out an ERD. Operations (methods) in particular are something a pure ERD does not show. If the operation compartment were empty and the relationships used crow's feet, you would lean toward an ERD instead.

Q: A vendor diagram shows layered boxes but every element has the same generic shape and pastel color bands across business/application/technology. Which notation is most likely, and what confirms it?
A: This is most likely ArchiMate, where the standard view stacks business (yellow), application (blue), and technology (green) color bands and uses small element-type icons in the corner of otherwise similar rectangles. The combination of horizontal layering with realization/serving arrows between layers confirms it. The icons (e.g. the role, process, component, or node glyphs) are the precise tell distinguishing ArchiMate elements from plain C4 boxes.

Q: A diagram you are shown could be BPMN or a plain flowchart — both use boxes and diamonds. What single feature most reliably distinguishes BPMN?
A: The presence of pools and lanes with message flows (dashed arrows) crossing between pools is the most reliable BPMN tell, since a plain flowchart has no concept of participants exchanging messages. Standardized BPMN event circles (start/intermediate/end with markers) are a second strong cue. A generic flowchart uses unlabeled diamonds and rectangles without these BPMN-specific constructs.

Q: When narrating a DMN decision table cold, what should you read first and in what order?
A: Read the hit policy marker in the top-left corner first (it governs how matches resolve), then the input columns and their allowed value lists, then the output column(s), then scan the rules row by row matching inputs to outputs. Knowing the hit policy before the rules prevents you from assuming "one row wins" when it might be `Collect` or `Priority`. You would then say, for instance, "this `U` table maps investor type and amount to a fee tier with no overlaps."

Q: A DMN decision returns null for a particular order — what is the first thing to check?
A: First check completeness: whether any rule actually matches that order's inputs, since a gap in the input ranges (e.g. an uncovered amount band or a missing investor-type value) yields no hit and thus a null result. Inspect the boundary brackets of adjacent ranges for an off-by-one gap like `[0..1000)` followed by `(1000..5000]`, which excludes exactly `1000`. Adding a catch-all rule or correcting the interval brackets resolves it.

Q: A BPMN process modeled with a parallel split never completes — what structural mistake should you look for first?
A: First look for a missing or mismatched join: a parallel (`+`) split must be matched by a parallel join that waits for all its branches, and if one branch dead-ends or is joined by an exclusive gateway instead, the process can deadlock or behave incorrectly. The parallel join blocks until every incoming token arrives, so a branch that never produces a token stalls it forever. Reading whether splits and joins are the same gateway type is the key check.

Q: Two architects disagree on whether to use C4 or ArchiMate for documenting a new fund-data platform. How would you frame the choice?
A: Frame it by scope and audience: C4 is lightweight and software-centric, ideal for a development team documenting the internal structure of one system at Context/Container/Component zoom, while ArchiMate is enterprise-wide, modeling how many business services, applications, and infrastructure relate across the organization. For documenting just the new platform's software, choose C4; for situating it in the firm's overall landscape and tracing regulatory drivers, choose ArchiMate. They are complementary, not competing, and many shops use C4 inside an ArchiMate landscape.

Q: How would you decide between embedding a rule in a BPMN exclusive gateway versus extracting it into DMN?
A: Use a BPMN gateway only for simple, stable, single-condition branching (e.g. "is the order amount > 0"), and extract into DMN whenever the logic involves multiple inputs, multiple rules, frequent change, or audit/ownership by business users. The heuristic: if the branch condition would need a table to express, it belongs in DMN. For a fund's fee-tier or eligibility rules — multi-input and regulation-driven — DMN keeps the BPMN flow readable and the rules independently auditable.

Q: In a workshop the vendor's ArchiMate shows an application service but no technology node beneath it. What does that omission tell you, and what would you ask?
A: The omission tells you the model stops at the application layer and does not (yet) trace realization down to infrastructure, so you cannot see what hosts or runs the service. You would ask which technology node or system software realizes that application service and where it is deployed — exactly the cross-layer realization ArchiMate is meant to show. In a DORA context that traceability matters because operational-resilience scoping needs the underlying ICT assets named.

Q: Summarize, as a one-line recognition cheat, the single most distinctive visual tell of each of BPMN, DMN, UML sequence, ERD, ArchiMate, and C4.
A: BPMN: pools/lanes with dashed message flows and event circles; DMN: a grid table with a hit-policy letter in the top-left corner; UML sequence: vertical lifelines with a downward time axis and horizontal message arrows; ERD: crow's-foot (or rounded/square IDEF1X) relationship ends on attribute boxes; ArchiMate: stacked business/application/technology color bands with corner element icons; C4: plainly labeled boxes with an explicit level title and legend. Leading with the distinctive tell lets you name the notation before interpreting any ambiguous shared shape.

Q: In BPMN, what is the difference between a data object and a data store, and how are they drawn?
A: A data object (a rectangle with a folded top-right corner) represents information that flows through and exists within a single process instance, such as the order document being validated, whereas a data store (a drum/cylinder symbol) represents persistent data that outlives any one instance, such as the investor register database. Both connect to activities via dotted data-association arrows rather than sequence flow. Reading them tells you which data is transient to the run and which is durable, e.g. an in-flight order versus the standing register a NAV run reads.
