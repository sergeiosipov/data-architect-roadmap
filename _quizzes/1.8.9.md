# Phase 1 · 1.8.9 Distributed transactions & 2PC — 150 self-test questions

Q: What is a distributed transaction?
A: A distributed transaction is a single logical transaction whose atomic commit spans two or more separate participants — different databases, message brokers, or services — so that all of them commit or all of them abort together. The hard part is achieving atomicity across processes that can each crash or become unreachable independently. Unlike a local transaction, no single node can unilaterally decide the outcome, which is what forces protocols like two-phase commit into existence.

Q: What does `2PC` stand for and what problem does it solve?
A: `2PC` stands for two-phase commit, an atomic commit protocol that coordinates several participants so they reach the same commit-or-abort decision. It solves the distributed atomicity problem: making a transaction that touches multiple nodes appear all-or-nothing even though each node commits its own local data. It does this by splitting commit into a voting phase and a decision phase under a single coordinator.

Q: Name the two phases of `2PC`.
A: The two phases are the prepare phase (phase 1) and the commit phase (phase 2). In prepare, the coordinator asks every participant whether it can commit, and each votes yes or no; in commit, the coordinator broadcasts the final decision — commit if all voted yes, abort otherwise. The phase boundary is the crucial commit point: once the coordinator has collected all yes votes and durably records its decision, the transaction is irrevocably committed.

Q: What role does the coordinator (transaction manager) play in `2PC`?
A: The coordinator, also called the transaction manager, drives the protocol: it issues the prepare request, collects votes, makes the single global commit-or-abort decision, durably logs that decision, and then tells every participant what to do. It is the only entity allowed to decide the outcome, which centralizes authority but also makes it the single point whose failure can block everyone. In XA terminology the participants are resource managers and the coordinator is the transaction manager (TM).

Q: What exactly does a participant promise when it replies "yes" / "prepared" in phase 1?
A: By voting "prepared" a participant promises that it can definitely commit the transaction later if asked, and that it will not abort it on its own initiative — it has surrendered its right to unilaterally decide. To back that promise it must have durably written everything needed (typically the WAL records and locks) so the commit can survive a crash. From that moment the participant is bound to whatever the coordinator decides, even across a restart.

Q: Why is the vote in phase 1 described as "the point of no return" for a participant?
A: Once a participant answers "prepared", it has given up the option to abort unilaterally, so it must wait for the coordinator's decision no matter how long that takes. It cannot time out and roll back, because the coordinator may have already told other participants to commit, and rolling back would break atomicity. This is precisely why a prepared-but-undecided participant is stuck holding locks until it hears back.

Q: What are the two "points of no return" that together make `2PC` atomic?
A: First, when a participant votes "yes/prepared" it can no longer abort on its own. Second, when the coordinator durably logs its commit decision, it can no longer change its mind. These two irrevocable commitments, taken together, guarantee that once the coordinator decides commit, every participant will eventually commit, giving the protocol its all-or-nothing property.

Q: How does `2PC` differ from a single-node commit in a database like Postgres?
A: A single-node commit is atomic for free: one process flushes one WAL record and the transaction is durably committed in one step, with no voting. `2PC` adds a round trip — prepare, collect votes, log decision, broadcast — to coordinate independent nodes that each have their own crash and network failure modes. The extra phase exists only because no single node can speak for the others.

Q: In Postgres, which command places a transaction into the prepared state for external coordination?
A: `PREPARE TRANSACTION 'some_gid'` writes the transaction's state durably to disk and dissociates it from the current session, leaving it prepared but not yet committed. A coordinator can later issue `COMMIT PREPARED 'some_gid'` or `ROLLBACK PREPARED 'some_gid'`. This is exactly the participant side of `2PC`; the prepared transaction keeps holding its locks until one of those commands arrives.

Q: What is the `max_prepared_transactions` setting in Postgres and why does it default to 0?
A: `max_prepared_transactions` caps how many transactions can be in the prepared state at once, and it defaults to `0`, which disables `PREPARE TRANSACTION` entirely. It ships off by default because prepared transactions that are never resolved hold locks and pin resources indefinitely, so the feature is opt-in for systems that actually run an external transaction coordinator. Leaving it at 0 is a deliberate "don't let people accidentally create in-doubt transactions" guardrail.

Q: Describe the happy-path `2PC` message flow at a high level.
A: The coordinator sends `prepare` to all participants; each does its local work, durably records it, and replies `yes`. The coordinator collects every reply, sees all yeses, durably logs a commit decision, and sends `commit` to all participants; each commits locally and acknowledges. The transaction is logically committed the instant the coordinator's decision hits durable storage, even before the acks come back.

Q: What happens in `2PC` if any single participant votes "no" in the prepare phase?
A: A single "no" vote forces a global abort: the coordinator logs an abort decision and sends `rollback` to every participant, including those that voted yes. This preserves atomicity — one participant being unable to commit means none may commit. Participants that had voted yes simply release their locks and discard the prepared work.

Q: For an order-service plus position-service write, sketch where `2PC` inserts itself.
A: The coordinator first tells the order service and the position service to prepare their respective writes — inserting the order row and updating the position row — and each locks those rows and votes prepared. Once both vote yes, the coordinator logs commit and tells both to commit, making the order and the position change atomically visible together. The single transaction guarantees you never see an executed order without its matching position change.

Q: In that order/position example, what is "atomic visibility" and why is it the thing `2PC` buys you?
A: Atomic visibility means no observer can ever see a state where the order is recorded but the position is not, or vice versa — both writes become visible at the same instant. `2PC` delivers this by deferring all commits until the single global decision, so the two services flip from "not committed" to "committed" together. That is the entire value proposition you are weighing against availability when you reject it.

Q: What is an "in-doubt" transaction in `2PC`?
A: An in-doubt transaction is one on a participant that has voted "prepared" but has not yet received the coordinator's final commit-or-abort decision. The participant cannot resolve it on its own because either outcome is still possible and only the coordinator knows the answer. Until the decision arrives, the transaction sits prepared, holding all its locks.

Q: What is the central blocking failure mode of `2PC`?
A: If the coordinator crashes after participants have voted "prepared" but before delivering the decision, those participants are stuck in-doubt: they cannot commit, cannot abort, and must wait for the coordinator to recover. During that wait they hold their locks, blocking any other transaction that needs the same rows. This is why `2PC` is called a blocking protocol — a single coordinator failure can freeze every participant indefinitely.

Q: When a `2PC` coordinator dies mid-commit, what happens to the participants' locks?
A: The locks held by every prepared, in-doubt participant stay held — they are not released, because releasing them could violate atomicity if the coordinator had actually decided commit. So rows touched by the transaction remain locked, and other transactions that need those rows block behind them. The locks persist until the coordinator recovers and replays its decision, or an operator manually resolves the transaction.

Q: Who is able to release the locks of an in-doubt participant, and who is not?
A: Only the coordinator's recovered decision (or a human operator stepping in) can legitimately release them; the in-doubt participant itself cannot, because it lacks the information to know whether commit or abort is correct. A participant that unilaterally guessed and was wrong would break atomicity across the system. This is the heart of why `2PC` blocks: the one actor who could safely decide is exactly the one that is down.

Q: Why can't an in-doubt participant simply time out and abort like a normal transaction would?
A: Because the coordinator may have already collected all yes votes, logged a commit decision, and told other participants to commit before crashing. If this participant aborted while others committed, atomicity is shattered. Having voted "prepared", it forfeited the right to abort on its own, so timing out is not a safe option — it must wait.

Q: How does the coordinator recover its decision after a crash in `2PC`?
A: The coordinator writes its commit-or-abort decision to a durable transaction log before sending it out, so on restart it reads that log to learn the outcome of every in-flight transaction. For any transaction whose decision was logged, it re-sends commit or abort to the participants until they acknowledge. Transactions that crashed before the decision was logged are aborted, since no participant could have been told to commit yet.

Q: Why must the coordinator's log write happen before it contacts any participant in phase 2?
A: If the coordinator told even one participant to commit and then crashed without having logged the decision, it would forget the outcome on restart while a participant had already committed — an unrecoverable inconsistency. Logging first guarantees that any decision visible to a participant is also recoverable by the coordinator. This ordering, decision-to-log then decision-to-participants, is what makes recovery deterministic.

Q: Can participants ask each other to resolve an in-doubt transaction if the coordinator is down?
A: In basic `2PC`, no — participants do not generally know each other and have no protocol to learn the global decision from a peer, so they remain blocked. There is a variant called three-phase commit (3PC) that adds a pre-commit phase to reduce blocking under certain failure assumptions, but it relies on bounded network delays and is rarely used in practice. Plain `2PC` simply blocks until the coordinator returns.

Q: What is a "heuristic decision" in the context of XA / in-doubt transactions?
A: A heuristic decision is when a resource manager, tired of waiting for an absent transaction manager, unilaterally commits or rolls back an in-doubt branch on its own — a heuristic commit or heuristic rollback. It breaks the protocol's guarantee deliberately to free locks, gambling on the likely outcome. If different resource managers guess differently you get a "heuristic mixed" outcome, where the distributed transaction is no longer atomic and a human must reconcile it.

Q: What is a "heuristic mixed" outcome and why is it dangerous?
A: A heuristic mixed (or heuristic hazard) outcome occurs when some participants heuristically committed while others heuristically rolled back the same distributed transaction, so atomicity is permanently violated. It is dangerous because the data is now inconsistent across systems in a way the protocol cannot self-correct — for example one ledger recorded a trade leg while the other did not. Resolving it requires manual investigation and compensating action, exactly the operational nightmare `2PC` was supposed to prevent.

Q: Why does holding locks during the in-doubt window make `2PC` especially bad for a busy fund platform?
A: In a high-throughput book, the same instrument rows (positions, cash balances per ISIN) are contended by many concurrent operations, so an in-doubt transaction holding those locks stalls every other order touching them. A coordinator outage during the trading day therefore cascades from one stuck transaction into a platform-wide freeze on the affected books. This is the "holds locks across the books at the worst possible moment" failure the lesson warns about.

Q: What does `XA` stand for and who defined it?
A: `XA` is the standard interface for distributed transactions defined by the X/Open consortium (now The Open Group) as part of its Distributed Transaction Processing (DTP) model. It specifies how a transaction manager coordinates multiple resource managers using `2PC`. It is the spec behind Java's JTA and the `XA` drivers offered by many databases and message brokers.

Q: In the `XA` / DTP model, what are the roles of the transaction manager and resource managers?
A: The transaction manager (TM) is the coordinator: it begins the global transaction and drives the `2PC` prepare/commit cycle. Each resource manager (RM) — a database, queue, or other transactional store — manages its own local data and participates by voting and committing. The `XA` interface (with calls like `xa_prepare`, `xa_commit`, `xa_rollback`) is the contract between TM and RMs.

Q: Name two or three `XA` interface calls a resource manager must implement.
A: A compliant resource manager implements calls such as `xa_start` and `xa_end` (to associate work with a transaction branch), `xa_prepare` (vote), `xa_commit` and `xa_rollback` (apply the decision), and `xa_recover` (list in-doubt branches after a crash). These C-style entry points are how the transaction manager drives `2PC` across heterogeneous resources. `xa_recover` is what lets a restarted TM find branches still awaiting a decision.

Q: According to DDIA, why do heterogeneous distributed transactions tend to "rot" in real systems?
A: Because `XA` is only as strong as its weakest link: every participating system must implement the protocol correctly, and the transaction manager itself becomes a critical, stateful component whose log must be durable and highly available. In practice the coordinator is often an afterthought, its log lives on one application server, and a crash there blocks the databases it coordinates — so the whole arrangement is fragile and operationally painful. Mixing vendors multiplies the chance that some participant mishandles in-doubt recovery.

Q: Why is the transaction manager's own durability a frequent weak point in `XA` deployments?
A: The TM must durably remember every global decision, yet it is often just a library running inside an application process rather than a replicated, fault-tolerant service. If that process and its log disk are lost, every database it coordinated is left with unresolvable in-doubt transactions. DDIA highlights this irony: you add `2PC` for reliability but introduce a new, often under-engineered single point of failure.

Q: How does `2PC` interact with connection pooling and stateless application servers?
A: Badly — a prepared transaction is bound to a specific database connection and must be completed from the coordinator's perspective, but stateless app servers and pooled connections assume any connection is interchangeable and can be recycled freely. An in-doubt transaction pins its connection and locks until resolved, which clashes with the pool's churn. This impedance mismatch is one of the practical reasons `XA` is awkward to run in modern microservice fleets.

Q: Does `2PC` work cleanly when one participant is a message broker rather than a database?
A: Only if the broker is a fully `XA`-capable resource manager that supports prepared, in-doubt message state — and even then it inherits all the blocking and recovery problems. Many popular brokers either do not support `XA` or support it with caveats, so you cannot reliably enlist them in a heterogeneous distributed transaction. This is a big reason the industry moved to the transactional outbox, which needs only a local database transaction.

Q: Why does adding more participants make `2PC` exponentially less attractive operationally?
A: Each additional participant is another node that can crash or partition during the in-doubt window, and a single one being unreachable blocks the whole transaction. The probability that at least one participant is unavailable grows with the number of participants, so availability degrades as you add resources. You are multiplying failure surfaces while the protocol can tolerate none of them without blocking.

Q: How does `2PC` performance compare with local transactions, and why?
A: `2PC` is substantially slower: it adds network round trips for prepare and commit, forces extra durable log flushes on the coordinator and participants, and holds locks for the full duration of the slowest participant plus any network delay. Throughput is gated by the slowest node and by lock contention during the extended commit window. DDIA notes order-of-magnitude penalties are common, which alone disqualifies it for hot paths.

Q: Is `2PC` an example of a consensus algorithm? Explain.
A: No — `2PC` is an atomic commit protocol, not a consensus algorithm, even though both get nodes to agree on something. The key difference is that `2PC` requires unanimous agreement and a single coordinator whose failure blocks progress, whereas consensus tolerates a minority of failures and never blocks on any single node. They solve related but distinct problems, and conflating them is a common mistake.

Q: What problem does Raft (consensus) solve that `2PC` does not?
A: Raft lets a cluster keep making progress and agreeing on a value even when a minority of nodes — including the leader — fail, because decisions are made by a majority quorum rather than by one indispensable coordinator. `2PC`, by contrast, blocks the moment its single coordinator is unreachable. So Raft removes the single blocking point that is `2PC`'s defining weakness.

Q: State in two sentences what Raft guarantees that `2PC` cannot.
A: Raft guarantees liveness despite the failure of any minority of nodes, including its leader, by electing a new leader from the surviving majority and continuing without manual intervention. `2PC` cannot guarantee this because its single coordinator is a blocking point: if it dies mid-commit, prepared participants are stuck until it returns.

Q: What is a quorum in Raft, and how many failures can a 5-node cluster tolerate?
A: A quorum is a majority of nodes — strictly more than half — required to elect a leader and to commit a log entry. With `2f+1` nodes you tolerate `f` failures, so a 5-node cluster (`f=2`) keeps working with up to 2 nodes down. The overlapping-majority property ensures any two quorums share at least one node, which prevents two conflicting decisions.

Q: How does Raft avoid the single-coordinator blocking that plagues `2PC`?
A: Raft has a leader, but the leader is not indispensable: if it fails, the remaining majority detects the missing heartbeats, runs an election, and elects a new leader, so the cluster never depends on one fixed node surviving. Because commit requires only a majority acknowledgment rather than unanimity, the failure of a minority — leader included — does not block progress. The role of coordinator is replaceable, which is exactly what `2PC` lacks.

Q: Why does `2PC` require unanimity while Raft requires only a majority?
A: `2PC` is solving atomic commit across distinct resources, where every participant owns data that must all change together, so a single "no" or a single absent participant must veto or block the transaction. Raft is solving replicated agreement on a single log among interchangeable replicas, so as long as a majority agrees, the minority can be brought up to date later. The different requirements stem from different problems: distributed atomicity versus fault-tolerant replication.

Q: Can you build an atomic commit that does not block, and how does consensus relate?
A: Yes — DDIA notes that an atomic commit protocol can be made non-blocking if it is built on a consensus algorithm, because consensus survives minority failures. Systems like Google's Spanner effectively layer `2PC` over Paxos/Raft groups so that each "participant" is itself a fault-tolerant replicated group, removing the single-coordinator block. So consensus is the foundation that lets atomic commit stop being a blocking protocol.

Q: Why isn't replacing `2PC` with Raft a drop-in solution for cross-service writes?
A: Raft replicates a single state machine among homogeneous replicas; it does not, by itself, atomically commit writes across two different services that own different data and schemas. You would still need an atomic-commit layer over the consensus groups, plus all the latency and coupling that implies. For ordinary microservice writes, that coupling is usually worse than just embracing eventual consistency via outbox or saga.

Q: What is the transactional outbox pattern?
A: The transactional outbox pattern writes the business change and an "outbox" message row in the same local database transaction, so they commit atomically without any distributed transaction. A separate relay process then reads new outbox rows and publishes them to the message broker, marking them sent. This converts a risky dual-write into one local commit plus reliable, retryable delivery.

Q: How does the outbox pattern give you atomicity without `2PC`?
A: The only atomic operation it needs is a single local transaction in one database, which every relational engine provides for free — the business row and the outbox row commit or roll back together. There is no coordinator, no prepared state, and no cross-system lock, so there is nothing to block. The trade is that the message is published slightly later (asynchronously), giving eventual rather than atomic visibility.

Q: What is a message relay / "polling publisher" in the outbox pattern?
A: It is the component that moves messages from the outbox table to the broker, either by periodically polling the table for unsent rows (polling publisher) or by tailing the database's change log (transaction-log tailing, e.g. with Debezium). It publishes each message and marks it as dispatched, retrying on failure. Because publishing is idempotent-friendly and at-least-once, downstream consumers must deduplicate.

Q: What delivery guarantee does the outbox typically provide, and what does that demand of consumers?
A: It typically provides at-least-once delivery: a message may be published more than once if the relay crashes after publishing but before marking the row sent. Consumers must therefore be idempotent — processing the same message twice must have the same effect as processing it once, often via a unique message id and a processed-ids table. This is the standard price of avoiding distributed transactions.

Q: What is a saga?
A: A saga is a sequence of local transactions across multiple services, where each step commits independently and publishes an event that triggers the next step. If a later step fails, the saga runs compensating transactions to semantically undo the prior steps, since you cannot roll back already-committed local transactions. It trades global atomicity for a managed, eventually-consistent multi-step workflow.

Q: What is a compensating transaction in a saga?
A: A compensating transaction is an explicit, business-level action that semantically undoes a previously committed step — for example, "cancel the reservation" to compensate "make the reservation", since the original local commit cannot be physically rolled back. Sagas rely on these compensations to recover from partial failure. Designing correct compensations (and handling that some actions are hard to undo, like a sent SWIFT message) is the main difficulty of the pattern.

Q: Contrast orchestration-based and choreography-based sagas.
A: In choreography, each service reacts to events and emits the next event with no central brain, which is decentralized but can become hard to follow as steps grow. In orchestration, a dedicated orchestrator tells each service what to do next and handles compensations, which centralizes the logic and is easier to reason about and monitor. The choice trades coupling and visibility against having a central component to maintain.

Q: What does a saga give up compared with `2PC`, and what does it gain?
A: A saga gives up atomic visibility and isolation: intermediate states are observable, so another transaction can see a half-finished saga, and you must compensate rather than roll back. In return it gains availability and loose coupling — every step is a fast local commit with no cross-service locks or coordinator, so one service being down does not freeze the others. This availability-for-isolation trade is exactly the point of an ADR rejecting `2PC`.

Q: Why is "lack of isolation" the headline weakness of sagas?
A: Because a saga's intermediate local commits are immediately visible, another operation can read or act on data that a not-yet-complete saga will later compensate away — analogous to a dirty read across services. You mitigate this with semantic locks, versioning, or status flags ("pending"), but you never get the clean serializable isolation a single transaction offers. Recognizing and designing around this exposure is the core saga skill.

Q: For the order-service/position-service write, how would outbox replace `2PC`?
A: The order service, in one local transaction, inserts the order row and an outbox row describing "order executed"; a relay publishes that event, and the position service consumes it and updates the position in its own local transaction. There is no coordinator and no cross-service lock — each side commits locally — so a position update may lag the order by milliseconds. You accept brief eventual consistency in exchange for not freezing the books when one service is down.

Q: When writing an ADR that rejects `2PC`, what is the minimum it should state?
A: It should name the consistency requirement, state that `2PC`/`XA` would meet it but at the cost of blocking and lock-holding when the coordinator fails, and name the chosen alternative — transactional outbox or saga. Crucially it must state the explicit trade: you give up atomic visibility (intermediate states become observable) and gain availability and decoupling. An ADR that just says "2PC is bad" without naming the trade-off is incomplete.

Q: In an ADR, what exactly do you write that you "give up" by choosing outbox/saga over `2PC`?
A: You give up atomic cross-service visibility: there will be a window where one service has committed and the other has not, so observers can see a partial state, and you must compensate rather than roll back on failure. You also give up clean isolation, accepting at-least-once delivery and the need for idempotent consumers. Naming these concessions honestly is what makes the ADR defensible.

Q: In an ADR, what exactly do you write that you "gain" by choosing outbox/saga over `2PC`?
A: You gain availability and fault isolation: each service commits locally with no coordinator and no cross-service locks, so one service being down or slow does not block the others or hold locks across the books. You also gain operational simplicity — no XA coordinator log to keep highly available, no in-doubt recovery runbooks. The system degrades gracefully instead of freezing.

Q: Why is "steer to outbox/saga with reasons, not taste" the architect's job here?
A: Because anyone can have an opinion that `2PC` is old-fashioned, but an architect must justify the choice with the concrete failure mode — coordinator death leaving in-doubt transactions that hold locks across contended rows — and the precise trade being made. Reasoned rejection survives pushback from a vendor or a senior engineer who proposes XA; taste does not. The lesson's whole point is to be able to defend the decision mechanically.

Q: A teammate proposes `XA` `2PC` between the order and position services. What is your first counter-argument?
A: That `2PC` makes the two services share fate: if the coordinator dies after prepare, both services sit in-doubt holding locks on the contended order/position rows, turning a coordinator blip into a trading-floor outage. That blocking failure mode is unacceptable on a hot path, and outbox gives you the same business outcome with eventual consistency and no shared lock. Lead with the lock-holding outage, because that is the visceral operational cost.

Q: What is the dual-write problem that outbox specifically solves?
A: The dual-write problem is needing to update a database and publish a message as one atomic action, but doing them as two separate operations means a crash between them leaves the system inconsistent — the row changed but the event was lost, or vice versa. Outbox solves it by making the database write and the event-row write a single local transaction, then publishing from that durable row. It removes the gap where the two writes could diverge.

Q: Why is "just publish to Kafka after committing to the database" not a safe alternative to outbox?
A: Because the process can crash in the gap between the database commit and the Kafka publish, losing the event while the data change persisted — the classic dual-write failure. There is no transaction spanning the database and Kafka to make them atomic. Outbox closes the gap by committing the event into the same database transaction and publishing it reliably afterward.

Q: How does log-tailing (CDC) relate to the outbox pattern?
A: Instead of polling an outbox table, you can tail the database's transaction log with change-data-capture (e.g. Debezium reading the Postgres WAL) and turn committed changes into messages. This avoids the polling overhead and the extra outbox table in some designs, while still publishing only durably committed changes. It is an implementation variant of the same "publish from the durable log, not a dual write" idea.

Q: What does it mean that `2PC` provides atomicity but not high availability?
A: It means `2PC` correctly guarantees all-or-nothing commit across participants, but it achieves this by blocking — sacrificing the ability to make progress — whenever the coordinator or a participant is unreachable. So it is safe (never inconsistent) but not live (can freeze). The alternatives flip this: they relax atomic visibility to stay available.

Q: Map `2PC`'s trade-off onto CAP-style thinking.
A: `2PC` favors consistency over availability: faced with a partition or coordinator loss it blocks rather than risk an inconsistent commit, so it gives up availability to preserve atomicity. Saga and outbox lean the other way — they stay available and accept eventual consistency, surfacing intermediate states. This is the same C-vs-A tension expressed at the commit-protocol level.

Q: Scenario: positions are locked and orders are timing out across multiple books at once. What `2PC`-related cause should you check first?
A: Check whether an `XA`/`2PC` coordinator has died or partitioned, leaving prepared transactions in-doubt and holding locks on the shared position rows. Look for prepared transactions on the database — in Postgres, query `pg_prepared_xacts` — and confirm whether the transaction manager is reachable. A dead coordinator is the textbook cause of simultaneous lock-up across contended books.

Q: How would you list in-doubt prepared transactions on a Postgres participant?
A: Query the `pg_prepared_xacts` system view, which shows every transaction that has been `PREPARE TRANSACTION`'d but not yet committed or rolled back, including its global id, owner, and prepare time. Long-lived rows there are your in-doubt transactions holding locks. You then resolve each with `COMMIT PREPARED` or `ROLLBACK PREPARED` once you know (from the coordinator) the correct outcome.

Q: An in-doubt Postgres prepared transaction is blocking trading. The coordinator is permanently gone. What are your options and risks?
A: Your only options are to manually `COMMIT PREPARED` or `ROLLBACK PREPARED` it to free the locks, but you are guessing the global outcome the dead coordinator never delivered. Guess wrong and you create a heuristic outcome — this participant disagrees with another, breaking atomicity across the books and requiring reconciliation. You must determine the intended outcome from other evidence (other participants' state, logs) before forcing it, and document the manual intervention.

Q: Why does the lesson call a blocked `2PC` coordinator "an outage at the worst possible moment"?
A: Because the in-doubt window holds locks precisely on the hottest, most contended rows — positions and cash per instrument — during active trading, so the freeze hits when volume and stakes are highest. A coordinator failure that would be a minor blip in a quiet system becomes a book-wide trading halt. The damage scales with how busy and contended the data is, which in a fund platform is maximal during the trading day.

Q: How does NAV calculation relate to the dangers of distributed transactions across services?
A: NAV depends on consistent, complete inputs — positions, prices, cash, corporate actions — often owned by different services, and a `2PC` arrangement to keep them in lockstep would block NAV-relevant writes whenever the coordinator failed, potentially during the valuation window. An outbox/saga design keeps each input service available and reconciles to a consistent snapshot, accepting brief eventual consistency. You would rather compute NAV from a known-consistent cut than have a coordinator freeze the inputs.

Q: If a saga step is "send a SWIFT settlement message", why is compensation hard?
A: Because a sent SWIFT message is an external, real-world side effect that cannot be un-sent — you cannot roll it back, only issue a follow-up cancellation or correction message (e.g. a reversal), which is itself a new business action that may not always be possible. This is the classic "irreversible step" problem in sagas: some actions have no clean compensator. You must order such steps carefully (often last, as a pivot/retriable step) and design explicit reversal flows.

Q: What is a "pivot transaction" in saga design and why does it matter for irreversible steps?
A: The pivot transaction is the step after which the saga is committed to completing — steps before it are compensatable, steps after it are retriable (must eventually succeed) and cannot be compensated. Placing an irreversible action like a SWIFT send at or after the pivot means the saga is structured to always drive it to completion rather than undo it. Identifying the pivot is how you handle steps that cannot be compensated.

Q: How do EMT/EPT file generation pipelines benefit from outbox over distributed transactions?
A: Producing EMT (European MiFID Template) or EPT (European PRIIPs Template) files draws on cost, risk, and product data from several systems; coordinating those reads/writes via `2PC` would couple their availability and risk blocking during the generation run. An outbox/event-driven approach lets each source publish its data changes reliably and the file-builder consume a consistent set, decoupling failures. You accept eventual consistency in exchange for a generation pipeline that does not freeze when one source is briefly down.

Q: Why might a transfer-agency platform prefer saga/outbox for a subscription workflow?
A: A subscription touches the order book, cash/settlement, the register of holders, and possibly AML checks — often different services — and a `2PC` across them would hold locks and block the whole flow on any coordinator hiccup. A saga lets each step commit locally and compensate on failure (e.g. reverse a provisional allocation), keeping the platform available under partial failure. The intermediate "pending" states are made explicit to the operations team rather than hidden behind a blocking commit.

Q: How does an LEI- or ISIN-keyed reference data update illustrate the dual-write problem?
A: Suppose updating an instrument's reference data must both persist the change and notify downstream systems keyed by ISIN/LEI; doing the database write and the notification as separate steps risks a crash that persists the data but loses the notification, leaving consumers stale. Outbox fixes this by committing the change and an outbox event in one transaction, then publishing reliably. Downstream caches keyed by ISIN then converge once the event is delivered.

Q: Why is idempotency by message id essential in a fund-data outbox pipeline?
A: Because at-least-once delivery means a position or cash event may arrive twice, and applying it twice — double-counting a trade or a fee — would corrupt the books or NAV. Consumers must record processed message ids and skip duplicates, so reprocessing is a no-op. In a regulated context, this deterministic, replayable processing is also what makes the pipeline auditable and recoverable.

Q: Could a regulator's expectation of consistency (e.g. under UCITS/AIFMD reporting) force you back toward `2PC`?
A: Not usually — regulators care about correct, reconcilable, auditable end states, not about microsecond-level atomic visibility across services, which is what `2PC` uniquely buys. An eventually-consistent design with reliable delivery, idempotent processing, and reconciliation produces correct reports while staying available, which is generally preferable. You document in the ADR that the consistency requirement is "eventually reconciled and auditable", not "atomically visible".

Q: How does DORA's emphasis on operational resilience inform the `2PC`-vs-saga choice?
A: DORA stresses that systems must withstand and recover from operational disruptions, and a blocking protocol that freezes the books when a single coordinator fails is the antithesis of resilience. Choosing outbox/saga, which degrades gracefully and recovers via retries and compensations, aligns with that resilience expectation. The ADR can cite operational resilience as an explicit reason to reject `2PC`'s single blocking point.

Q: What is the difference between atomicity and isolation, and which does a saga sacrifice?
A: Atomicity is all-or-nothing completion; isolation is that concurrent transactions do not see each other's partial work. A saga sacrifices isolation first and foremost — intermediate committed steps are visible to others — while it preserves a weaker, eventual form of "atomicity" through compensation rather than rollback. So the precise concession to name is loss of isolation (visible intermediate states), not loss of all atomicity.

Q: Why does `2PC` hold locks for longer than a normal transaction even on the happy path?
A: Because locks acquired during local work must be held all the way through the prepare phase, the network round trip to collect votes, the coordinator's durable log write, and the commit broadcast — a much longer window than a single-node commit. Lock duration is bounded by the slowest participant and the slowest network hop, not by local work. Longer lock hold times mean more contention and lower concurrency even when nothing fails.

Q: What is the relationship between `2PC` and the FLP impossibility / blocking?
A: `2PC`'s blocking is a concrete instance of the general truth that you cannot have a non-blocking atomic commit with a single coordinator under asynchronous failures — there is no safe action for an in-doubt participant when the decider is unreachable. Consensus algorithms sidestep this by using majorities and replaceable leaders, trading the requirement of unanimity for fault tolerance. This is why "just use 2PC" does not escape the fundamental difficulty; it only relocates it to the coordinator.

Q: How would you explain to a non-expert stakeholder why you are not using `2PC`?
A: You can say `2PC` is like requiring two clerks to stamp a document at the exact same instant with a manager watching — if the manager steps out mid-stamp, both clerks freeze with the document locked and nobody else can touch it until the manager returns. We instead let each clerk stamp their own copy and reconcile, so a missing manager never freezes the office. The cost is a brief moment where the copies disagree, which we detect and fix automatically.

Q: What durable state must a `2PC` participant write at prepare time, and why?
A: At prepare it must durably persist all the transaction's effects and its lock state — typically flushing the relevant WAL records — so that even after a crash and restart it can still honor a later commit. This is what backs its promise that it can definitely commit if asked. Without that durable record, the "prepared" vote would be a lie the moment the participant restarted.

Q: After a participant restarts, how does it know which transactions are in-doubt and how to handle them?
A: On restart it reads its own durable prepared-transaction records (in Postgres, these reappear in `pg_prepared_xacts`; via XA, `xa_recover` lists branches) and re-acquires the corresponding locks, then waits to learn each transaction's fate from the coordinator. It must not assume an outcome. The coordinator, recovering its decision log, re-delivers commit or abort so each in-doubt branch is finally resolved.

Q: Why must locks be re-acquired by a participant when it recovers an in-doubt transaction after a crash?
A: Because the transaction is still unresolved and might commit, so the rows it touched must stay protected from conflicting concurrent changes until the decision arrives. If the participant released the locks on restart and the transaction later committed, intervening writes could have produced an inconsistent result. Re-acquiring the locks preserves the same isolation guarantee that existed before the crash.

Q: What is the difference between `2PC` and two-phase locking (`2PL`)?
A: They are unrelated despite the similar names: `2PC` is an atomic commit protocol coordinating multiple nodes, while two-phase locking (`2PL`) is a single-database concurrency-control technique that acquires locks in a growing phase and releases them in a shrinking phase to ensure serializability. One is about distributed commit, the other about local isolation. Confusing them is a classic interview trap.

Q: In the `2PC` flow, at exactly what instant is the transaction considered committed?
A: The transaction becomes committed the moment the coordinator durably writes its commit decision to its log — even before any participant has been told or has acknowledged. From that instant the outcome is fixed and recoverable, so all participants will eventually commit no matter what crashes next. Pinpointing this commit point is essential to reasoning about recovery.

Q: If the coordinator crashes after logging "commit" but before telling participants, what happens?
A: Nothing is lost: on restart the coordinator reads its log, sees the commit decision, and re-sends commit to every participant until they acknowledge. The participants were in-doubt and holding locks during the outage but were never at risk of an inconsistent outcome. This is the "good" coordinator crash — recovery completes the protocol correctly, just with a blocking delay.

Q: If the coordinator crashes after prepare but before logging any decision, what happens?
A: It aborts: on restart it finds no recorded decision for that transaction, and since no participant could yet have been told to commit, the only safe choice is to abort and tell participants to roll back. The in-doubt participants release their locks and discard the prepared work. The protocol stays consistent, again at the cost of a blocking window during the outage.

Q: Why can't you avoid `2PC`'s blocking simply by adding a timeout that aborts in-doubt transactions?
A: Because a participant timing out and aborting might contradict a commit decision the coordinator already logged and possibly delivered to others, producing a split outcome. The safe answer to "what should an in-doubt participant do?" is "wait", and a timeout that acts violates that. This is exactly why automatic timeout-abort is unsafe and why true non-blocking commit needs consensus, not just a clock.

Q: What does "atomic commit requires the same value to be agreed by all" have to do with consensus?
A: Atomic commit is essentially a consensus problem with the extra constraint of unanimity — every participant must agree on the single commit-or-abort value, and a "no" from any one forces abort. Pure consensus relaxes this to majority agreement, which is why it can be non-blocking. Recognizing atomic commit as "consensus plus unanimity" clarifies both why `2PC` blocks and why building it over consensus removes the block.

Q: How does Spanner relate to `2PC` and consensus in your mental model?
A: Spanner runs `2PC` across shards, but each shard is itself a Paxos group, so every participant in the commit is a fault-tolerant replicated entity rather than a single fragile node. This removes `2PC`'s single-coordinator block by making each role survive minority failures via consensus. It is the canonical example that `2PC` is not irredeemable — its blocking comes from non-fault-tolerant participants, which consensus fixes.

Q: Why is "use a saga" not a license to ignore consistency entirely?
A: Because a saga still must reach a correct final state through compensations, and you must design and test those compensations, handle partial-failure ordering, and expose intermediate "pending" states safely. Eventual consistency is a discipline, not an absence of consistency — you owe reconciliation and idempotency. A sloppy saga can corrupt data just as badly as a misused `2PC`.

Q: What is a semantic lock in saga terminology?
A: A semantic lock is an application-level marker — like a `status = 'PENDING'` flag on a record — that tells other operations the data is mid-saga and should be treated cautiously, partially recovering the isolation that sagas otherwise lack. It is not a database lock; it is a business-level signal that the value may still be compensated away. It is one of the standard countermeasures for the lack-of-isolation problem.

Q: Summarize when, if ever, `2PC` is the right choice.
A: `2PC`/`XA` is occasionally justified within a single, controlled environment where all participants are XA-capable, the coordinator is highly available, transactions are short, contention is low, and you genuinely need atomic visibility you cannot reconcile after the fact. Even then, modern systems usually prefer consensus-backed commit or outbox/saga. For cross-service writes on a busy fund platform's hot path, the blocking risk almost always rules it out.

Q: What is the single most important sentence to put in the "Done when" check about coordinator death and locks?
A: "When a `2PC` coordinator dies mid-commit, prepared participants are left in-doubt and keep holding their locks — they can neither commit nor abort on their own — until the coordinator recovers its logged decision (or an operator manually resolves them), and only those parties can safely release the locks." That sentence captures the failure mode, the lock fate, and who can resolve it. It is the lesson's core takeaway in one breath.

Q: How would you phrase, in two sentences, the Raft-vs-`2PC` distinction for the "Done when" check?
A: Raft guarantees the cluster keeps agreeing and making progress despite the failure of any minority of nodes, including the leader, because decisions need only a majority quorum and the leader is replaceable. `2PC` cannot guarantee this: its single coordinator is a blocking point, so its failure leaves prepared participants stuck until it returns.

Q: Why is naming the alternative pattern (not just rejecting `2PC`) required for a complete ADR?
A: Because an architectural decision record exists to record what you will do, not merely what you will avoid — rejecting `2PC` without naming outbox or saga leaves the team without a path forward. Naming the pattern, and the trade it accepts (atomic visibility for availability), makes the decision actionable and reviewable. It also signals you understand the alternative deeply enough to defend it.

Q: A reviewer says "outbox is just eventual consistency, which is risky for money." How do you respond?
A: You acknowledge it is eventual consistency, then explain the risk is bounded and managed: at-least-once delivery plus idempotent consumers plus reconciliation means the books converge to a correct, auditable state, and the visibility lag is milliseconds, not minutes. The riskier alternative is `2PC` freezing the books when a coordinator dies during trading. For money, a brief, detected, self-correcting divergence beats an unbounded lock-up.

Q: What durable component does outbox require that the dual-write does not, and why is that acceptable?
A: Outbox requires an outbox table (or log-tailing of the existing WAL) and a relay process, which is modest extra machinery compared to running a highly available XA coordinator. It is acceptable because the only atomic operation needed is a single local transaction the database already provides, and the relay's failures are recoverable via at-least-once retry. You trade a small, well-understood relay for eliminating distributed-commit blocking.

Q: Why is message ordering a subtle concern in an outbox/saga design, and how do you address it?
A: Because at-least-once asynchronous delivery can reorder or duplicate events, a position-update consumer might apply a later trade before an earlier one and corrupt the running balance. You address it by carrying a monotonic sequence or version per aggregate and having consumers reject or buffer out-of-order events, often pinning a partition key (e.g. by ISIN or account) so a given key's events stay ordered. Designing for reordering up front is part of accepting eventual consistency responsibly.

Q: How does an outbox/saga design support auditability and replay, which `2PC` does not naturally provide?
A: Every state change flows through durable event rows (the outbox/log), giving you an immutable, ordered record you can replay to rebuild state or to investigate a discrepancy — invaluable in a regulated fund context for proving how a NAV or position figure arose. `2PC` leaves only the final committed state, with the coordinator's transient log discarded after recovery, so it offers no comparable replayable history. The event trail is a side benefit that often makes the eventual-consistency design the better audit story as well.

Q: Tie it together: what is the one-line reason an architect steers a fund platform away from `2PC` for cross-service writes?
A: Because `2PC`'s single coordinator is a blocking point whose death leaves prepared participants in-doubt, holding locks on the most contended position and cash rows during trading, so the safe-but-not-available trade it makes is the wrong one for a hot path — outbox or saga keeps the books available at the cost of brief, reconcilable eventual consistency. That sentence names the mechanism, the failure, the trade, and the chosen alternative, which is exactly what the lesson asks you to be able to say.

Q: You enabled `PREPARE TRANSACTION` but it errors with `prepared transactions are disabled`. What is the first thing to check?
A: Check that `max_prepared_transactions` is set to a nonzero value, because it defaults to `0`, which disables the feature entirely and produces exactly that error. It is a startup-only parameter set in `postgresql.conf`, so changing it requires a full server restart, not just a `SELECT pg_reload_conf()`. A common gotcha is that on a replica it must be at least as large as on the primary or recovery will fail.

Q: Which columns does the `pg_prepared_xacts` view expose, and which one do you use to resolve a transaction?
A: It exposes `transaction` (the numeric xid), `gid` (the global transaction identifier string), `prepared` (the timestamp it entered the prepared state), `owner` (the role that prepared it), and `database`. You use the `gid` to resolve it, issuing `COMMIT PREPARED 'thegid'` or `ROLLBACK PREPARED 'thegid'`. The `prepared` timestamp is what you sort by to find the oldest in-doubt transactions causing the longest lock holds.

Q: An orphaned Postgres prepared transaction has been open for hours and `VACUUM` is no longer reclaiming dead tuples. Why, and what is the longer-term danger?
A: A prepared transaction holds an xid that stays visible as "in progress", so it pins the `xmin` horizon and `VACUUM` cannot remove tuples newer than that horizon, causing bloat. Left long enough this blocks transaction-id freezing and can drive the database toward xid wraparound, at which point Postgres refuses new writes to protect data. The fix is to resolve the orphan with `COMMIT PREPARED` or `ROLLBACK PREPARED` so its xid is released.

Q: Can a session other than the one that ran `PREPARE TRANSACTION` issue the `COMMIT PREPARED`? Why does that matter for `2PC`?
A: Yes — `COMMIT PREPARED` and `ROLLBACK PREPARED` can be issued from any session, because preparing deliberately dissociates the transaction from the originating backend. This is precisely what makes external coordination possible: the transaction manager, not the original client connection, delivers the final decision. It is also why an operator can manually resolve an in-doubt transaction even though the application that created it has long since disconnected.

Q: In the `XA` recovery interface, what is the difference between calling `xa_recover` with `TMSTARTRSCAN` versus `TMENDRSCAN`?
A: `TMSTARTRSCAN` begins a fresh scan and returns the first batch of `Xid`s for branches in the prepared or heuristically-completed state, while `TMENDRSCAN` closes the scan and frees the resource manager's scan cursor. A transaction manager typically calls `xa_recover(TMSTARTRSCAN)`, pages through the in-doubt branches, then calls `xa_recover(TMENDRSCAN)` to finish. Forgetting `TMENDRSCAN` can leak the scan state inside the resource manager.

Q: A resource manager returns `XAER_RMFAIL` during `xa_commit` in phase 2. What does the transaction manager do?
A: `XAER_RMFAIL` means the resource manager is currently unavailable (e.g. crashed or unreachable), not that the transaction is bad, so the transaction manager must not abandon the decision — it retains the logged commit and retries `xa_commit` later until the RM recovers. This is the in-doubt branch persisting from the RM side: the branch stays prepared on the RM until the retry succeeds. The contrast is `XAER_RMERR`, which signals an actual error in processing the request.

Q: What is a transaction branch (`Xid`) in `XA`, and why does a single global transaction have several?
A: An `Xid` identifies one branch of a global transaction — the portion of work done against a single resource manager — and is built from a global transaction id plus a branch qualifier. A global transaction touching three databases therefore has three branches, each with the same global id but a distinct branch qualifier. The transaction manager drives `xa_prepare`/`xa_commit` per branch, which is how it tracks each participant's vote and outcome independently.

Q: In `XA`, what does `xa_end` with the `TMSUSPEND` flag let you do, and why is that subtle?
A: `TMSUSPEND` temporarily disassociates the current thread from a transaction branch without ending it, so the branch can be resumed later with `xa_start(TMRESUME)`, supporting interleaved work on one connection. It is subtle because the branch is still alive and holding its locks while suspended — nothing is released. Mismanaging suspend/resume is a classic source of "lock held but no active work" puzzles in `XA` deployments.

Q: How does `xa_start` with `TMJOIN` differ from a plain `xa_start`, and when is it used?
A: Plain `xa_start` begins a brand-new branch, whereas `xa_start(TMJOIN)` attaches the current thread to a branch the resource manager has already seen, letting multiple pieces of work be folded into the same branch. It is used when a transaction manager wants several operations on one RM to share a single prepare/commit rather than become independent branches. Misusing it can merge work you intended to keep as separate votes.

Q: Two participants in one global transaction return conflicting `xa_recover` results after a crash — one lists a branch as prepared, the other does not. What does that tell you?
A: It tells you the global transaction was mid-protocol when the crash hit: the branch that lists the prepared `Xid` is in-doubt awaiting a decision, while the other either never reached prepare or already committed/aborted that branch. The transaction manager's own log is the authority on the intended outcome, and you replay its decision to the in-doubt branch. If the log is gone, you must reconstruct the outcome from the participant that already acted before forcing the in-doubt one to match.

Q: Describe the "presumed abort" optimization in `2PC` and what it saves.
A: Under presumed abort, the coordinator does not force-write a log record for transactions it decides to abort, and on recovery any transaction with no decision record is assumed aborted. This saves a log flush and acknowledgment traffic for the common abort case, since absence of information already means "abort". The symmetric "presumed commit" variant exists but is rarer because it requires logging the participant list before prepare to safely presume commit.

Q: Why does presumed abort mean an in-doubt participant that asks the coordinator about a forgotten transaction is told to roll back?
A: Because under presumed abort the coordinator deliberately discards abort records, so when a participant inquires about a transaction the coordinator has no memory of, the safe and correct answer is "abort" — the absence of a commit record proves it was never committed. This lets the coordinator answer recovery queries without keeping abort logs forever. The participant then rolls back its prepared branch and releases the locks.

Q: A teammate claims `3PC` (three-phase commit) makes the system non-blocking, so it is strictly better than `2PC`. How do you push back?
A: `3PC` adds a pre-commit phase that lets participants reach a decision after a coordinator failure in some scenarios, but its non-blocking guarantee depends on a synchronous model with bounded message delay and reliable failure detection, which real asynchronous networks violate. Under network partitions `3PC` can still produce inconsistent outcomes, which is why it is almost never used in production. The robust answer to non-blocking commit is consensus-backed commit, not `3PC`.

Q: Why does a network partition that isolates the coordinator from a subset of participants create a particularly nasty `2PC` situation?
A: Participants on the far side of the partition that have voted prepared are in-doubt and blocked, holding locks, while the coordinator may have decided and informed the reachable participants — so you get a partial commit visible on one side and frozen rows on the other. The isolated participants cannot safely guess, because the coordinator's decision really exists and just cannot reach them. This split is exactly the availability sacrifice `2PC` makes to preserve atomicity.

Q: In a saga, the "make reservation" step succeeded but the orchestrator crashed before recording it. On restart, how do you avoid a double reservation?
A: The reservation step must be idempotent, keyed by the saga id, so re-issuing it either no-ops or returns the existing reservation rather than creating a second one. The orchestrator persists saga state durably at each step transition, so on restart it replays from the last recorded step and idempotency absorbs any re-execution of an already-completed step. Without idempotent steps, orchestrator crash-and-retry would duplicate side effects.

Q: Compare the failure blast radius of a dead `2PC` coordinator versus a dead saga orchestrator.
A: A dead `2PC` coordinator leaves participants in-doubt holding locks, so it actively freezes contended rows across services until it recovers — the blast radius is everyone touching those rows. A dead saga orchestrator merely pauses in-flight sagas at an already-committed local step; no cross-service locks are held, so unrelated work proceeds and the paused sagas resume on restart. The saga degrades to delayed completion, not a system-wide freeze.

Q: What is a "countermeasure" in saga terminology, and name two beyond the semantic lock?
A: Countermeasures are the techniques sagas use to recover the isolation they lack, mitigating anomalies from visible intermediate state. Beyond the semantic lock (a `PENDING` status flag), two are commutative updates — designing operations so order does not matter, e.g. additive balance deltas — and the pessimistic-view ordering, which reorders saga steps so the riskiest-to-compensate step runs where it least exposes dirty data. A fourth is "re-read value" (version checks) to detect that data changed mid-saga.

Q: How does a "by value" routing decision in a saga help with the lost-update anomaly?
A: By value means the orchestrator inspects the actual data (e.g. the risk level of a subscription) to decide whether to run a low-isolation or a higher-isolation flow, escalating to stricter handling — even `2PC`-like locking — only for the high-risk minority. This concentrates the expensive isolation where it matters and keeps the common path cheap and available. It is the pragmatic middle ground when blanket eventual consistency is too risky for some records.

Q: In the outbox pattern, why is it important that the relay marks a row as sent in a way that survives its own crash?
A: If the relay publishes to the broker and crashes before durably recording "sent", on restart it republishes — that is the source of at-least-once delivery, which is acceptable only because consumers dedupe. The mark-sent write must itself be durable (a committed update or an offset commit) so progress is not lost the other way, causing the relay to skip rows. Getting this wrong in the skip direction loses events, which is worse than a duplicate.

Q: Why might transaction-log tailing (CDC) be preferred over a polling publisher for a high-volume fund event stream, and what is the cost?
A: Log tailing reads committed changes straight from the WAL, so it adds near-zero query load and captures every change with low latency, whereas a polling publisher repeatedly scans the outbox table and trades latency against poll frequency and load. The cost is operational complexity: you run a CDC connector (e.g. Debezium), manage replication slots, and must monitor slot lag — an abandoned slot pins WAL and can fill the disk. For high volume the lower DB load usually justifies that complexity.

Q: An unused Postgres replication slot for your CDC outbox reader is filling the disk. What is the mechanism and the fix?
A: A logical replication slot guarantees the WAL it has not yet confirmed is retained, so if the Debezium consumer is down or lagging, the slot's `restart_lsn` stops advancing and Postgres keeps accumulating WAL segments, eventually exhausting `pg_wal` disk. The fix is to restore the consumer so it advances the slot, or, if the slot is truly abandoned, drop it with `pg_drop_replication_slot`. A `max_slot_wal_keep_size` cap can bound the damage by invalidating slots that fall too far behind.

Q: In the Debezium outbox event router, which outbox column becomes the Kafka message key, and why does that matter for fund data?
A: The `EventRouter` SMT sets the Kafka message key from the aggregate id column (by default `aggregateid`), which routes all events for one entity to the same partition and thereby preserves their order. For fund data keyed by ISIN or account, this keeps a given instrument's or holder's events strictly ordered, so a position consumer never applies a later trade before an earlier one. The aggregate type column similarly selects the destination topic.

Q: You see duplicate position updates in a consumer despite an idempotency table. What are the two most likely bugs?
A: First, the deduplication key may be wrong — keying on something not unique per logical event (e.g. a timestamp) instead of a stable message id — so genuine duplicates slip through. Second, the "record processed id" write and the business update may not be in the same local transaction, so a crash between them lets a redelivery reprocess. The fix is a stable unique message id and committing the dedupe insert and the effect atomically in one local transaction.

Q: How would you make the consumer's dedupe write and its business effect atomic without any distributed transaction?
A: Put both in a single local database transaction on the consumer's own database: insert the message id into a `processed_messages` table (unique constraint) and apply the business update, then commit together. A duplicate hits the unique-constraint violation and the whole transaction rolls back, so the effect is applied at most once per id. This is the consumer-side mirror of the producer's outbox — one local transaction, no `2PC`.

Q: Why is "exactly-once delivery" generally impossible, and what is the achievable target for an outbox pipeline?
A: True exactly-once delivery is impossible across a network because the sender can never be certain a message arrived before a crash, so it must either risk losing it or risk resending it. The achievable target is exactly-once processing (effectively-once), built from at-least-once delivery plus idempotent, deduplicating consumers. Saying "exactly-once" loosely is fine only if everyone understands it means dedupe-on-top-of-at-least-once.

Q: A saga step issues an irreversible SWIFT settlement message, then a later step fails. What design choice prevents needing to "un-send" it?
A: Make the SWIFT send the pivot transaction or place it after the pivot, so every step up to that point is compensatable and every step after it is retriable rather than compensatable. The saga is then structured to drive the post-pivot steps to eventual success instead of ever trying to undo the send. If a genuine reversal is needed it is modeled as a new forward business action (a cancellation message), never as a rollback of the original.

Q: Distinguish "retriable" from "compensatable" steps in a saga and why their ordering matters.
A: Compensatable steps can be semantically undone and must come before the pivot; retriable steps cannot be undone but are guaranteed to eventually succeed and must come after the pivot. Ordering matters because once you cross the pivot you have committed to completing the saga, so any step that is impossible to compensate (like a SWIFT send) belongs in the retriable region. Putting an irreversible step among the compensatable ones creates a saga you cannot correctly recover.

Q: For an NAV publication workflow, why is a saga's "pending until reconciled" state actually a feature, not a bug?
A: It makes the in-progress state explicit and visible to operations, so a half-assembled NAV (some inputs in, others lagging) is clearly marked not-final rather than silently mistaken for an official figure. `2PC` would hide the assembly behind a blocking commit and freeze the inputs if the coordinator died, whereas the saga exposes a `PENDING` snapshot that converges and is then marked official. Explicit pending states match how fund operations actually validate and sign off NAV.

Q: A reviewer asks how you bound the eventual-consistency window in a fund outbox so it cannot silently grow unbounded. What do you monitor?
A: You monitor relay/CDC lag (outbox-row age or replication-slot lag) and consumer lag, alerting when the oldest unprocessed event exceeds a threshold measured in seconds. You also reconcile periodically — comparing source-of-truth balances against the consumer's derived state — so any divergence is detected and quantified. Bounding the window is a monitoring and reconciliation discipline, which is exactly what makes "eventual" defensible to risk and audit.

Q: Why can an at-least-once outbox still lose data if the relay deletes rows before confirming the broker accepted them?
A: If the relay deletes (or marks sent) the outbox row before the broker acknowledges the publish, a broker failure or network drop in that gap loses the event permanently — you have reintroduced a dual-write between "publish" and "mark sent". The correct order is publish, await broker ack, then mark sent, so a crash before the ack merely causes a safe re-publish. Order of operations is the whole game in at-least-once relays.

Q: How does idempotency on the producer side (not just the consumer) help an outbox relay?
A: If the broker supports idempotent producers (e.g. a Kafka producer with `enable.idempotence=true` and a producer id plus sequence numbers), the broker itself deduplicates retries of the same record, so the relay's at-least-once republish does not create broker-level duplicates within a session. This shrinks, though does not eliminate, the duplicate burden on consumers, since cross-session or cross-relay duplicates can still occur. Consumers must still be idempotent as the backstop.

Q: A senior engineer says "we already use Kafka transactions, so we have exactly-once and do not need the outbox". Where does that argument break?
A: Kafka transactions give atomicity among Kafka reads/writes and offset commits within Kafka, but they do not span your external database, so a write to Postgres plus a Kafka produce is still a dual-write that can diverge on a crash. The outbox exists precisely to bridge the database and the broker with a single local DB transaction. Kafka's exactly-once is real but only inside Kafka's own boundary, which is not where the dual-write risk lives.

Q: Contrast the durability requirement of a `2PC` coordinator log with that of an outbox table.
A: The `2PC` coordinator log must be durable AND highly available, because its loss leaves every coordinated database with unresolvable in-doubt transactions — it is a new critical stateful service. The outbox table just lives inside the business database you already operate and back up, and its relay is restartable with at-least-once semantics, so losing the relay process loses nothing. You replace a fragile, must-not-fail coordinator with ordinary, already-maintained database machinery.

Q: Why is the statement "2PC guarantees consistency" imprecise, and what guarantee does it actually provide?
A: `2PC` guarantees atomicity — all participants commit or all abort on the agreed decision — not the broader notion of consistency (preserving invariants), which depends on application logic. It also does not by itself provide isolation across the global transaction beyond what each participant's local concurrency control gives. Precisely: `2PC` provides distributed atomic commit, and conflating that with full ACID "consistency" is a common imprecision.

Q: In CAP terms, place outbox, saga, and `2PC` and note one nuance.
A: `2PC` sits on the CP side — it sacrifices availability (blocks) to preserve atomic consistency under failure; outbox and saga sit on the AP side, staying available and accepting eventual consistency. The nuance is that CAP's "consistency" is linearizability of a single register, which is not exactly atomic commit, so the mapping is an analogy, not a theorem — but it correctly conveys the availability-versus-consistency tension. Use it to communicate the trade, not as a rigorous proof.

Q: A position consumer receives events out of order because they came from different partitions. What partitioning fix preserves correctness?
A: Partition the topic by the aggregate key — ISIN or account — so all events for one instrument or holder land on the same partition and are delivered in order to a single consumer. Cross-key ordering does not matter for per-aggregate balances, so per-key ordering is sufficient and scales horizontally. If you must change keys, you accept reordering across the boundary and need versioning or buffering to cope.

Q: How do monotonic version numbers per aggregate let a consumer safely tolerate duplicate and reordered events?
A: Each event carries an aggregate version, and the consumer stores the last applied version; it applies an event only if its version is the expected next one (or higher, depending on the model), rejecting or buffering out-of-order arrivals and ignoring already-applied versions as duplicates. This turns at-least-once, possibly-reordered delivery into effectively-once, in-order application. It is the standard backstop when partition-key ordering alone is not enough.

Q: Why is a saga's compensation for "debit cash" not simply "credit the same amount" in a regulated ledger?
A: In an audited ledger you generally do not delete or net away the original entry; you post a separate, traceable reversal entry that references the original, preserving the full history of what happened and when. So the compensation is a new forward posting, not an erasure, which keeps the books append-only and auditable. This is why saga compensations in finance are designed as business reversals with their own identifiers, not as physical rollbacks.

Q: A distributed transaction spans a database and a JMS queue via `XA`, and the broker reports a heuristic rollback while the database committed. What state are you in and what must you do?
A: You are in a heuristic-mixed (heuristic-hazard) state: the database committed its branch while the broker unilaterally rolled back its branch, so atomicity is broken and the systems disagree. You cannot fix this automatically; you must investigate which outcome was intended (from the TM log), then issue a compensating action to reconcile — e.g. re-enqueue the lost message or reverse the DB effect. You also record the heuristic outcome for audit, since the protocol's guarantee was deliberately violated.

Q: Why does the `XA` model require participating drivers to be `XA`-capable, and what fails if you mix an `XA` and a non-`XA` resource?
A: An `XA` resource must implement the prepare/commit/recover contract so the transaction manager can vote and recover it; a non-`XA` resource has no prepare step, so it cannot participate in the atomic decision. Mixing them means the non-`XA` resource commits or fails independently of the global decision, reintroducing the dual-write inconsistency `2PC` was meant to prevent. At best you can include exactly one non-`XA` resource via the "last resource commit" optimization, with caveats.

Q: What is the "last resource gambit" (last-resource commit) optimization, and what risk does it carry?
A: Last-resource commit lets one non-`XA` (single-phase) resource join a `2PC` transaction by being committed last, using its own commit as the deciding vote that the transaction manager then propagates to the prepared `XA` resources. The risk is a narrow window: if the TM crashes after the last resource commits but before logging the outcome, the prepared `XA` branches can be left in-doubt with no recorded decision, risking inconsistency. It is a pragmatic but fragile bridge, used sparingly.

Q: For a transfer-agency subscription saga, list a plausible compensation for each forward step.
A: For "reserve units in the order book" the compensation is "release the reservation"; for "earmark incoming cash" it is "release the earmark"; for "provisionally add the holder to the register" it is "remove the provisional register entry". The AML check and the final confirmed allocation are typically arranged around the pivot so the irreversible/confirmable steps are retriable rather than compensatable. Naming each compensation explicitly is the core design deliverable of the saga.

Q: Why is "we will add 2PC later if eventual consistency proves insufficient" usually a weak plan?
A: Because retrofitting `2PC` means making every participant `XA`-capable, standing up a highly available coordinator with a durable log, and rewriting hot paths to hold cross-service locks — a deep architectural change, not a toggle. Worse, the eventual-consistency design's value (availability, decoupling, replay) is forfeited the moment you reintroduce blocking. The honest plan is to escalate isolation surgically by value for the few records that need it, not to bolt on global `2PC`.

Q: How does the transactional outbox interact with database read replicas and the CDC reader's source?
A: The outbox relay/CDC reader should read from the primary's WAL (or a slot on the primary), because the outbox row must be the committed, authoritative change; reading a lagging replica would delay or, with cascading replication, complicate slot management. Tailing the primary ensures events reflect committed state with minimal added latency. You then scale consumers downstream off the broker, not off additional database replicas.

Q: A prepared transaction's locks are blocking a critical batch, and you confirm from the coordinator log it was meant to commit. What exact command resolves it and what must you double-check first?
A: You issue `COMMIT PREPARED 'thegid'` using the exact `gid` from `pg_prepared_xacts`, which applies the committed effect and releases the locks. Before doing so, double-check that every other participant's branch is consistent with "commit" so you are not creating a heuristic split — the coordinator log or the other participants' states must agree. You then document the manual resolution for audit, since you acted outside the normal coordinator flow.

Q: Why is choreography-based saga ordering harder to monitor than orchestration, and how does that bite during an incident?
A: In choreography the workflow logic is implicit in who-listens-to-which-event, so there is no single place that knows the saga's overall state, making it hard to answer "where is this subscription stuck?" during an incident. Orchestration centralizes that state in the orchestrator, which can report the current step and trigger compensations explicitly. The choreography trade is looser coupling at the cost of distributed, harder-to-trace control flow.

Q: An architect must choose between escalating one high-risk record to `2PC` versus keeping the whole flow on saga. What single question decides it?
A: The deciding question is whether a transient, detectable, reconcilable divergence for that record is tolerable to the business and regulator, or whether atomic visibility is genuinely non-negotiable for it. If reconciliation within a bounded window is acceptable, stay on saga with stronger countermeasures; only if atomic visibility is truly required and reconciliation cannot suffice do you escalate that record to a `2PC`/consensus-backed commit. The answer turns on the cost of a brief inconsistency, not on protocol aesthetics.

Q: Summarize the decision tree an architect walks when someone proposes a distributed transaction for a cross-service write.
A: First ask whether a single local transaction suffices (use the outbox to publish) — if so, no distributed transaction is needed; if multiple services must change, ask whether eventual consistency with idempotent consumers and reconciliation meets the real requirement, and if so choose outbox plus saga. Only if atomic visibility is genuinely mandatory and cannot be reconciled do you consider `2PC`, and then prefer a consensus-backed (non-blocking) commit over bare `XA`. The default lands on outbox/saga because the blocking cost of `2PC` is almost never worth it on a hot path.

Q: Why is the prepare phase's durable flush on each participant a hidden latency cost even when nothing fails?
A: Every participant must `fsync` its prepare record to stable storage before it can vote yes, and the coordinator must `fsync` its decision, so a single global commit incurs multiple serialized disk-sync round trips rather than the one a local commit needs. On spinning disks or fsync-heavy storage this dominates commit latency, and it is paid on every transaction, not just failures. This is a core reason `2PC` throughput is often an order of magnitude below local commits even on the happy path.

Q: A consumer applies a "cancel subscription" event before the "create subscription" event it cancels, because of reordering. How do you prevent the cancel from being lost or erroring?
A: Key both events by the same aggregate id (the subscription id) so they share a partition and arrive in order, and additionally carry an aggregate version so the consumer can buffer or reject an out-of-order cancel until the create it references has been applied. If a cancel still arrives first, the consumer parks it keyed by subscription id and replays it once the create lands, rather than dropping it or throwing. This combination of per-key ordering plus version-gated buffering is how you stay correct under at-least-once reordering.
