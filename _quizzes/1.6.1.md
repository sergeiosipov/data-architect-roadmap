# Phase 1 · 1.6.1 ACID & isolation levels — 150 self-test questions

Q: What does the acronym `ACID` stand for?
A: `ACID` stands for Atomicity, Consistency, Isolation, and Durability — the four properties a transaction is expected to provide. They are a guarantee bundle: a transaction either applies fully or not at all, never leaves the database in an invalid state, runs as if alone, and survives a crash once committed. In practice "Consistency" is the odd one out because it depends on application-defined invariants, while the other three are database mechanisms.

Q: What is atomicity in the context of a database transaction?
A: Atomicity means a transaction is all-or-nothing: either every write inside it takes effect, or none of them do. If the transaction aborts halfway — by error, deadlock, or crash — the database rolls back any partial changes so no half-finished work is visible. The mechanism is the ability to discard or undo writes, which is why DDIA notes "atomicity" here really means "abortability".

Q: What is durability in the context of a transaction?
A: Durability means that once a transaction commits, its writes survive even if the machine crashes or loses power immediately afterward. The database must persist the committed data to non-volatile storage (or replicate it) before reporting success. In Postgres this is achieved by flushing the write-ahead log to disk at commit, so the change can be replayed during crash recovery.

Q: What is a write-ahead log (`WAL`) and why does it exist?
A: A `WAL` is an append-only log to which the database writes a record of every change *before* it modifies the actual data pages. It exists so that a commit can be made durable cheaply — appending sequentially is fast — and so the database can recover a consistent state after a crash by replaying the log. Postgres stores it under `pg_wal/`; the principle is "log first, apply later".

Q: How does the `WAL` make a commit survive a crash?
A: At commit time the database forces the WAL records up to that commit point to durable storage, typically via an `fsync`, before acknowledging the client. If the server crashes before the dirty data pages are written, recovery replays the committed WAL records to reconstruct the changes; uncommitted records are ignored or undone. The data pages themselves can be flushed lazily because the log is the authoritative record.

Q: Why is sequential `WAL` writing faster than updating data pages directly at commit?
A: Appending to the WAL is a sequential write to one growing file, which spinning disks and SSDs both handle far faster than the scattered random writes that updating many data pages in place would require. By making only the small, ordered log durable at commit, the database gets durability without paying for random I/O on the hot path. The heavyweight page writes are deferred and batched by the checkpointer.

Q: What is a checkpoint in relation to the `WAL`?
A: A checkpoint is a point at which the database flushes all dirty data pages to disk and records that the WAL up to that position is now reflected in the data files. After a crash, recovery only needs to replay WAL written *after* the last checkpoint, bounding recovery time. Checkpoints trade steady background I/O for shorter recovery and for letting old WAL segments be recycled.

Q: What does "isolation" promise when several transactions run at once?
A: Isolation promises that concurrently executing transactions do not interfere with each other's intermediate state — ideally each runs as if it were the only transaction on the system (serializability). In practice databases offer weaker, cheaper isolation levels that permit certain anomalies, so the real guarantee depends on the level you select. The architect's job is to know exactly what each level still allows.

Q: What is a "dirty read"?
A: A dirty read happens when one transaction reads a row that another transaction has written but not yet committed. If that other transaction later aborts, the reader has acted on data that never officially existed. For example, an order-processing job that reads an uncommitted NAV value and proceeds would be using a number that may be rolled back seconds later.

Q: At which isolation level are dirty reads possible, and where are they prevented?
A: Dirty reads are only possible at `READ UNCOMMITTED`; every level at `READ COMMITTED` and above forbids them. This is the most basic isolation guarantee and almost universally provided. Notably Postgres never actually performs dirty reads — its `READ UNCOMMITTED` behaves exactly like `READ COMMITTED`.

Q: What is a "dirty write"?
A: A dirty write occurs when one transaction overwrites a value that another transaction has written but not yet committed. It corrupts the all-or-nothing logic because two transactions' uncommitted changes get interleaved, often violating an invariant that spans multiple rows. Essentially every isolation level prevents dirty writes, typically by holding a row-level write lock until commit.

Q: What is a non-repeatable read (read skew)?
A: A non-repeatable read is when a transaction reads the same row twice and sees different committed values because another transaction modified and committed it in between. The first read and second read disagree, even though the reader changed nothing. This breaks any logic that assumes a row stays fixed across the life of a transaction.

Q: Give a fund-data failure story for a non-repeatable read.
A: A reconciliation job reads a fund's cash balance to start a calculation, does some work, then re-reads the same balance for a cross-check; between the two reads a settlement transaction commits a new balance, so the two reads disagree and the job flags a false break. The data is internally consistent at each moment, but the transaction saw a moving target. Raising the level to `REPEATABLE READ` pins both reads to one snapshot.

Q: What is a phantom read?
A: A phantom read is when a transaction runs the same query twice and the second run returns rows that did not exist (or no longer exist) the first time, because another transaction inserted or deleted rows matching the predicate. Unlike a non-repeatable read, it concerns the *set* of rows a predicate matches, not a single known row. The classic case is `SELECT COUNT(*) WHERE ...` returning a different count.

Q: Give a fund-data failure story for a phantom read.
A: A compliance check counts all pending subscription orders for a fund above a threshold to enforce a cap, then re-counts before committing; between the counts another session inserts a new qualifying order, so a phantom appears and the cap is breached even though the check "passed". The query predicate matched a new row that wasn't there at first read. Serializable isolation, or a range lock, closes this gap.

Q: What is write skew?
A: Write skew is an anomaly where two transactions each read an overlapping set of rows, make decisions based on what they read, and then each write to *different* rows — so neither overwrites the other, but their combined effect violates an invariant that held when each read. It is subtle precisely because there is no direct write-write conflict. The canonical example is two doctors each going off-call because each saw the other still on-call.

Q: Give a fund-accounting scenario where write skew corrupts data.
A: Two redemption transactions each read a fund cash balance of 100 and a rule "total withdrawals must not exceed the balance"; each independently sees enough cash for its 70 withdrawal and each writes its own withdrawal row, leaving the fund overdrawn by 40. Neither transaction overwrote the other's row, so snapshot isolation sees no conflict, yet the invariant is broken. Only `SERIALIZABLE` (or an explicit lock on the balance) prevents it.

Q: Why does snapshot isolation permit write skew but not lost updates on the same row?
A: Snapshot isolation gives each transaction a consistent point-in-time view and detects conflicts only when two transactions write the *same* object, which catches lost updates. Write skew involves reads of one set of rows and writes to a *different* set, so there is no first-committer-wins conflict to detect — the dangerous dependency is read-write across rows, which plain snapshot isolation ignores. Serializable Snapshot Isolation adds exactly the missing read-write dependency tracking.

Q: What is a lost update?
A: A lost update happens when two transactions read the same value, each computes a new value from it, and each writes back — so the second write silently clobbers the first, and one update is lost. A read-modify-write counter increment is the textbook case. Fixes include atomic update statements, explicit `SELECT ... FOR UPDATE` locking, or an isolation level that detects the conflict.

Q: How do non-repeatable reads, phantoms, and write skew differ from one another?
A: A non-repeatable read concerns a single known row changing value between two reads; a phantom concerns the *membership* of a result set changing because rows are inserted or deleted; write skew concerns two transactions reading overlapping data and writing disjoint data so that a multi-row invariant is broken. They escalate in subtlety: single row, set of rows, then cross-row invariant. Higher isolation levels are needed as you move down that list.

Q: What are the four ANSI/ISO SQL isolation levels, from weakest to strongest?
A: From weakest to strongest they are `READ UNCOMMITTED`, `READ COMMITTED`, `REPEATABLE READ`, and `SERIALIZABLE`. The ANSI standard defines them by which of three phenomena (dirty read, non-repeatable read, phantom read) each must prevent. The standard sets a floor — an implementation may prevent *more* anomalies than required at a given level.

Q: How does the ANSI standard formally define each isolation level?
A: ANSI defines the levels by the phenomena they forbid: `READ UNCOMMITTED` allows all three, `READ COMMITTED` forbids dirty reads, `REPEATABLE READ` additionally forbids non-repeatable reads, and `SERIALIZABLE` forbids phantoms too. It specifies what must *not* happen, not how to implement it, which is why vendors differ in mechanism and even in which extra anomalies they happen to prevent. A famous critique (Berenson et al.) showed these phenomenon definitions are ambiguous and miss anomalies like write skew.

Q: Why is the ANSI standard's phenomenon-based definition considered flawed?
A: Because it defines levels by a short list of named phenomena rather than by serializability, and that list is incomplete and ambiguously worded — it never mentions write skew or lost updates, and it does not cleanly distinguish lock-based from snapshot-based implementations. A 1995 critique by Berenson, Bernstein, Gray, and others demonstrated this and proposed snapshot isolation as a distinct level. The upshot is that two databases can both claim `REPEATABLE READ` yet permit different anomalies.

Q: What is snapshot isolation?
A: Snapshot isolation gives each transaction a consistent view of the database as of the moment it began (or its first read), so it sees a frozen snapshot and never observes other transactions' concurrent changes. Reads never block writes and writes never block reads, which is why it performs well. It prevents dirty reads, non-repeatable reads, and phantoms but still permits write skew, so it is not the same as serializability.

Q: Why is snapshot isolation not the same as `SERIALIZABLE`?
A: Snapshot isolation prevents read-anomalies by freezing a consistent view, but it only checks for write-write conflicts on the same object, so it allows write skew and read-only anomalies that a truly serial execution could not produce. Two transactions can each act on a stale view of rows the other modified and still commit. True serializability requires detecting read-write dependencies, which plain snapshot isolation does not do.

Q: What is multi-version concurrency control (`MVCC`)?
A: `MVCC` is a technique where each update creates a new version of a row rather than overwriting it in place, so different transactions can read different versions according to their snapshot. This lets readers see a consistent old version while writers create newer ones, so reads and writes rarely block each other. Postgres implements snapshot isolation on top of MVCC, keeping old row versions until no snapshot needs them.

Q: In an `MVCC` system like Postgres, why do readers not block writers?
A: Because a writer creates a new version of a row instead of overwriting the old one, a concurrent reader can continue reading the older version that belongs to its snapshot. The reader never needs a lock on data the writer is changing, so they proceed in parallel. This is the core performance benefit of MVCC and why Postgres avoids the read locks that a pure locking scheme would take.

Q: What is the cost of `MVCC` keeping old row versions around?
A: Old, no-longer-visible row versions ("dead tuples") accumulate and consume disk space and slow scans until they are cleaned up. Postgres reclaims them with `VACUUM` (and autovacuum); a long-running transaction can hold back the cleanup horizon and cause bloat. So MVCC trades simple concurrency for a maintenance obligation and sensitivity to long transactions.

Q: What is the default isolation level in PostgreSQL?
A: `READ COMMITTED` is the default isolation level in PostgreSQL. Each statement in a `READ COMMITTED` transaction sees a fresh snapshot taken at the start of that statement, so it sees data committed before the statement began. You can change it per transaction with `SET TRANSACTION ISOLATION LEVEL ...` or via `default_transaction_isolation`.

Q: How many *distinct* isolation levels does PostgreSQL actually implement, and why?
A: Postgres accepts all four standard level names but internally implements only three distinct behaviors, because its `READ UNCOMMITTED` mode behaves identically to `READ COMMITTED`. Postgres never returns uncommitted data, so there is no separate weaker behavior to provide. The three real levels are `READ COMMITTED`, `REPEATABLE READ`, and `SERIALIZABLE`.

Q: Why does Postgres treat `READ UNCOMMITTED` the same as `READ COMMITTED`?
A: Because Postgres's MVCC design never exposes uncommitted row versions to other transactions, so it physically cannot perform a dirty read. Offering a genuinely weaker `READ UNCOMMITTED` would mean building a dirty-read path that the architecture deliberately avoids. The standard permits providing a *stronger* guarantee than requested, so mapping `READ UNCOMMITTED` up to `READ COMMITTED` is conformant.

Q: In Postgres `READ COMMITTED`, when is the snapshot for a query taken?
A: At `READ COMMITTED`, a new snapshot is taken at the start of *each statement*, so every statement sees all data committed before it began. This is why two `SELECT`s in the same transaction can return different results — each got its own snapshot. It is the source of non-repeatable reads and phantoms at this level.

Q: In Postgres `REPEATABLE READ`, when is the snapshot taken?
A: At `REPEATABLE READ`, a single snapshot is taken at the time of the first non-transaction-control statement in the transaction, and all subsequent statements use that same snapshot. So the whole transaction sees one frozen point-in-time view of the database. This eliminates non-repeatable reads and phantoms within the transaction.

Q: Does PostgreSQL's `REPEATABLE READ` prevent phantom reads?
A: Yes — unlike the bare ANSI minimum, Postgres's `REPEATABLE READ` (which is really snapshot isolation) does not allow phantom reads, because the whole transaction reads from one consistent snapshot. The official isolation table marks phantom reads as "allowed by the standard, but not in PostgreSQL". This is a case of Postgres providing a stronger guarantee than the standard requires at that level.

Q: What anomaly does Postgres `REPEATABLE READ` still permit?
A: It still permits the serialization anomaly known as write skew (and certain read-only anomalies), because it is snapshot isolation and only detects write-write conflicts on the same row. Two transactions reading overlapping data and writing disjoint rows can both commit and break a multi-row invariant. To close that gap you must use `SERIALIZABLE`.

Q: What technique does PostgreSQL use to implement `SERIALIZABLE`?
A: Postgres implements `SERIALIZABLE` using Serializable Snapshot Isolation (`SSI`), which builds on snapshot isolation by adding runtime checks for the read-write dependency patterns that could produce a non-serializable outcome. It does not fall back to coarse locking; instead it monitors dependencies and aborts a transaction if a dangerous cycle is detected. This gives true serializability while preserving most of snapshot isolation's concurrency.

Q: What are predicate locks (`SIReadLock`) in Postgres `SERIALIZABLE`?
A: Predicate locks are bookkeeping locks Postgres takes to record which data a serializable transaction *read*, so it can later tell whether another transaction's write would have changed that read had it run first. They appear in `pg_locks` with mode `SIReadLock`. Crucially they do not block other transactions and cannot cause deadlocks — they only flag dependencies used to detect serialization anomalies.

Q: Do Postgres `SIReadLock` predicate locks block other transactions?
A: No — `SIReadLock` predicate locks never block writes or reads and cannot cause deadlocks; they are purely informational, used by SSI to track read-write dependencies. When a dangerous dependency pattern forms among concurrent serializable transactions, Postgres aborts one of them rather than having blocked it earlier. This non-blocking design is what keeps `SERIALIZABLE` performant compared to two-phase locking.

Q: What does `SERIALIZABLE` "buy" you compared to `REPEATABLE READ` in Postgres?
A: `SERIALIZABLE` buys you a guarantee that the concurrent transactions produce a result equivalent to *some* serial order, which eliminates write skew and the remaining serialization anomalies that snapshot isolation permits. You no longer have to reason about specific cross-row anomalies — if your transaction is correct run alone, it is correct under concurrency. The price is that some transactions are aborted with a serialization failure and must be retried.

Q: What does `SERIALIZABLE` "cost" you in Postgres?
A: It costs extra tracking of read-write dependencies (more memory and CPU for predicate locks) and, more importantly, a higher rate of transactions aborted with SQLSTATE `40001` that the application must catch and retry. Under contention these retries can dominate, and the predicate-lock tracking can be coarsened, raising false-positive aborts. So you adopt it where correctness demands it and engineer retry logic accordingly.

Q: What is the exact `SQLSTATE` code raised on a serialization failure in Postgres?
A: It is `40001`, the standard SQLSTATE for "serialization_failure". Postgres returns it under `REPEATABLE READ` or `SERIALIZABLE` when it cannot guarantee the transaction's result is consistent with a serial order. The application's contract is to roll back and retry the whole transaction when it sees this code.

Q: What error message accompanies a `40001` failure caused by a concurrent update?
A: Postgres reports `ERROR: could not serialize access due to concurrent update`. This typically arises at `REPEATABLE READ` when a row the transaction tries to update was changed and committed by another transaction after the snapshot was taken. The fix is to retry the transaction, which gets a fresh snapshot.

Q: What error message accompanies a `40001` failure caused by read/write dependencies?
A: Postgres reports `ERROR: could not serialize access due to read/write dependencies among transactions`. This is the SSI-specific message at `SERIALIZABLE` when the dependency tracker detects a pattern that could not occur in any serial execution. As with the other 40001 case, the correct response is to abort and retry the transaction.

Q: What must an application do when it receives SQLSTATE `40001`?
A: It must roll back the failed transaction and retry the *entire* transaction from the beginning, because the failure means no consistent serial result was possible this time. A fresh attempt gets a new snapshot and usually succeeds. Retries should be bounded with a cap and backoff so a hot conflict does not loop forever.

Q: Why must you retry the whole transaction, not just the failed statement, after `40001`?
A: Because the transaction's earlier reads were taken under a snapshot the database has now declared unsafe; re-running a single statement would mix old reads with new data and could still violate serializability. Only by restarting from `BEGIN` does the transaction get a coherent fresh snapshot for all its reads and writes. This is the defining difference between a transactional retry and a naive statement retry.

Q: What is the difference between an idempotent retry and a transactional retry?
A: A transactional retry re-runs an entire aborted transaction (e.g., after `40001`) because the database guarantees the failed attempt left no committed effect. An idempotent retry is needed when you are unsure whether a prior attempt's effect landed — for example after a network timeout — so the operation is designed to be safe to apply more than once. Serialization-failure retries are transactional; client-side resend-on-timeout retries usually need idempotency.

Q: Why does retrying after a network timeout require idempotency rather than just re-execution?
A: Because a timeout is ambiguous: the transaction may have actually committed before the acknowledgement was lost, so blindly re-running it could double-apply the effect. Idempotency — via a unique request key, an upsert, or a deduplicated insert — makes a second application a no-op. Without it, a "retry" of a placed fund order could book the order twice.

Q: How might you make a fund-order insert idempotent under retries?
A: Attach a unique client-supplied idempotency key (or a natural unique constraint like order reference plus account) and enforce it with a `UNIQUE` index, so a retried insert hits a duplicate-key conflict instead of creating a second order. Use `INSERT ... ON CONFLICT DO NOTHING` to turn the duplicate into a safe no-op and return the existing row. This way an ambiguous network retry cannot double-book the order.

Q: What is the default isolation level of Microsoft SQL Server (on-premises)?
A: On a default on-premises SQL Server install the default is `READ COMMITTED` implemented with locking — readers take short-lived shared locks row by row, so a reader can block a writer and vice versa. This is the pessimistic, lock-based flavor of `READ COMMITTED`. It differs from Postgres, whose `READ COMMITTED` is snapshot-based and non-blocking.

Q: What is the default isolation behaviour of Azure SQL Database, and how does it differ from on-premises SQL Server?
A: Azure SQL Database has Read Committed Snapshot Isolation (`RCSI`) enabled by default, so its `READ COMMITTED` uses row versioning to give each statement a transactionally consistent snapshot without taking read locks. On-premises SQL Server defaults to lock-based `READ COMMITTED` with `READ_COMMITTED_SNAPSHOT` off. So the same isolation-level name behaves optimistically in Azure SQL but pessimistically on a stock on-prem server.

Q: What is the default isolation level of Oracle Database?
A: Oracle's default is `READ COMMITTED`, providing automatic statement-level read consistency: each query sees a consistent view as of the time the statement began, reconstructed from undo data. Oracle does not offer a true `READ UNCOMMITTED`, and its `SERIALIZABLE` is implemented as snapshot isolation. Readers never block writers and writers never block readers, similar in spirit to Postgres MVCC.

Q: What does Oracle's "statement-level read consistency" mean?
A: It means that within a single SQL statement, Oracle guarantees the query sees data as it existed at the moment the statement started, even if other transactions commit changes while the statement runs. Oracle reconstructs older block versions from undo (rollback) data to honor that point-in-time view. At the default `READ COMMITTED`, each statement gets its own consistent point, which is why two statements in one transaction can still differ.

Q: Why can the same isolation-level name behave differently across vendors?
A: Because the SQL standard defines levels by which anomalies they must forbid, not by implementation, so each vendor chooses locking, MVCC, or snapshot mechanics and may prevent extra anomalies. Postgres `REPEATABLE READ` is snapshot isolation (no phantoms), while a lock-based engine's `REPEATABLE READ` might still allow phantoms; Oracle's `SERIALIZABLE` is snapshot isolation, not true serializability. The level name alone does not tell you the real guarantee.

Q: Why does the vendor default difference matter during a database migration?
A: Because application code is often written against the *behaviour* of the source database's default, not the level name, so moving to a database with different mechanics can introduce anomalies that "worked" before. For instance, code relying on Postgres's non-blocking snapshot `READ COMMITTED` may deadlock or block under lock-based on-prem SQL Server `READ COMMITTED`, and code that silently depended on Oracle's snapshot `SERIALIZABLE` may see real serialization failures elsewhere. You must re-test concurrency assumptions, not just port DDL.

Q: Name one concrete migration consequence of Postgres vs SQL Server `READ COMMITTED` differences.
A: Under Postgres, a long-running report at `READ COMMITTED` reads a consistent statement snapshot and never blocks writers; migrating that workload to lock-based on-prem SQL Server `READ COMMITTED` can have the same report acquire shared locks that block or deadlock with concurrent writes. The application that never saw lock contention before suddenly does. Enabling `READ_COMMITTED_SNAPSHOT` on SQL Server restores the snapshot semantics and avoids surprise blocking.

Q: What is `SET TRANSACTION ISOLATION LEVEL` used for in Postgres?
A: It sets the isolation level for the current transaction, e.g. `SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;` issued right after `BEGIN` and before any query. It only affects that transaction; once a query has run you can no longer change the level for it. To change the session or database default you use `SET default_transaction_isolation` or the server parameter.

Q: How do you start a serializable transaction in psql in a single statement?
A: You can write `BEGIN TRANSACTION ISOLATION LEVEL SERIALIZABLE;` (or `BEGIN ISOLATION LEVEL SERIALIZABLE;`) which begins the transaction and sets the level in one step. Alternatively `BEGIN;` followed by `SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;` before the first query has the same effect. The level must be set before any data-accessing statement runs.

Q: How do you inspect the current transaction's isolation level in Postgres?
A: Run `SHOW transaction_isolation;` which returns the level in effect for the current transaction (or session default if none is active). You can also query `SHOW default_transaction_isolation;` to see the configured default for new transactions. These are the quick first checks when an anomaly investigation begins.

Q: To run the two-session ACID lab, how do you start the Phase-1 Postgres container and connect?
A: Start the container (e.g. `docker compose up -d` or `docker run -d --name pg postgres`) so a Postgres server is listening, then open two terminals and connect each with `psql` to the same database. Two independent `psql` sessions let you interleave `BEGIN`/`SELECT`/`UPDATE`/`COMMIT` statements by hand and watch one transaction observe (or fail to observe) the other. Keeping them side by side is the entire concurrency lab.

Q: Why do you need *two* separate psql sessions to demonstrate isolation anomalies?
A: Because isolation anomalies are about how one transaction observes another transaction's concurrent activity, which you cannot show within a single serial session. Two sessions let you deliberately interleave statements — begin A, write in A, read in B, commit A, re-read in B — to expose dirty/non-repeatable/phantom reads and write skew. The manual interleaving is what makes the otherwise abstract definitions concrete.

Q: How would you reproduce a non-repeatable read at `READ COMMITTED` with two sessions?
A: In session A, `BEGIN; SELECT balance FROM fund_cash WHERE id=1;` and leave it open; in session B, `UPDATE fund_cash SET balance=balance-10 WHERE id=1; COMMIT;`; back in session A, re-run the same `SELECT` — it now returns the new value. The two reads in the same transaction disagree because `READ COMMITTED` takes a fresh snapshot per statement. Re-running the experiment at `REPEATABLE READ` makes both reads identical.

Q: How would you reproduce a phantom at `READ COMMITTED` with two sessions?
A: In session A, `BEGIN; SELECT count(*) FROM orders WHERE fund='X';`; in session B, `INSERT INTO orders(fund) VALUES ('X'); COMMIT;`; in session A re-run the count — it increases, a phantom row appeared. The predicate `fund='X'` now matches a row that did not exist at the first read. At `REPEATABLE READ` in Postgres the second count would stay unchanged.

Q: How would you reproduce write skew at `READ COMMITTED` or `REPEATABLE READ`?
A: Use a two-row `fund_cash` table and a rule that the sum stays non-negative; in session A and session B each `BEGIN`, each `SELECT` both balances and sees enough total, then A updates row 1 and B updates row 2, and both `COMMIT`. Neither overwrote the other's row, so both commit and the invariant breaks — that is write skew. Repeating at `SERIALIZABLE` causes one of them to abort with `40001`.

Q: When you re-run the write-skew interleaving at `SERIALIZABLE`, what happens?
A: One of the two transactions commits and the other is aborted with SQLSTATE `40001` and the message about read/write dependencies, because SSI detects the dangerous dependency cycle. The application is expected to retry the aborted transaction, which on its second attempt sees the first transaction's committed effect and correctly refuses to break the invariant. This is the concrete demonstration that `SERIALIZABLE` prevents write skew.

Q: What is the right isolation level for a fund-order workload that must never double-book or overdraw, and why?
A: For invariants that span multiple rows — total redemptions not exceeding cash, no over-allocation of a capped tranche — `SERIALIZABLE` is the safe default because it is the only level that prevents write skew and phantoms in one stroke. You pay with occasional `40001` retries, which is acceptable for correctness-critical fund accounting. Where a constraint can be expressed as a single-row check or database constraint, a lower level plus explicit locking can suffice.

Q: How should a fund-order application respond to a serialization failure?
A: It should catch SQLSTATE `40001`, roll back, and transparently retry the whole transaction a bounded number of times with a small backoff, surfacing an error only if retries are exhausted. The retry must restart from `BEGIN` so it reads a fresh snapshot, and any externally visible side effects (emails, downstream messages) must be deferred until commit or made idempotent. Done right, the end user never sees the transient conflict.

Q: Why is "is this exactly-once and serializable?" a question an architect cannot delegate?
A: Because the answer determines whether the platform can double-book orders, lose updates, or produce NAV figures that auditors will later be unable to reconcile — these are correctness and regulatory risks, not implementation details. Vendors, defaults, and retry behaviour interact subtly, and only someone reasoning at the architecture level can state the guarantee actually delivered end to end. Getting it wrong surfaces as unexplainable corrections that regulators find first.

Q: How does choosing the wrong isolation level lead to "unexplainable NAV corrections"?
A: If concurrent transactions are allowed an anomaly like write skew or a lost update, the persisted figures can violate an invariant nobody coded for, so a later NAV strikes off corrupted balances and must be corrected without a clear cause in the logs. The defect is non-deterministic and timing-dependent, so it resists reproduction and looks like a mystery. The root cause is an isolation gap, and the only durable fix is architectural, not a one-off data patch.

Q: What is a serialization anomaly?
A: A serialization anomaly is any outcome of concurrent transactions that could not have arisen from running those transactions one at a time in some order. Write skew and certain read-only anomalies are examples. `SERIALIZABLE` is precisely the level that forbids all serialization anomalies, which is why it is the gold standard for correctness.

Q: What is two-phase locking (`2PL`) at a high level?
A: `2PL` is a pessimistic concurrency-control protocol where each transaction acquires locks (shared for reads, exclusive for writes) in a growing phase and releases them only in a shrinking phase, typically holding all locks until commit (strict 2PL). Holding locks until commit guarantees serializability but causes readers and writers to block each other and can deadlock. It is the classic alternative to MVCC/SSI.

Q: How does `SSI` differ from two-phase locking as a route to serializability?
A: `2PL` is pessimistic — it blocks transactions by holding locks until commit, trading concurrency and risking deadlocks to guarantee a serial order. `SSI` is optimistic — it lets transactions run on snapshots without read locks, tracks read-write dependencies, and aborts a transaction only if a non-serializable pattern actually forms. SSI keeps read concurrency high but converts some conflicts into retryable aborts instead of waits.

Q: What is the difference between an optimistic and a pessimistic concurrency-control strategy?
A: Pessimistic control (e.g. `2PL`) assumes conflicts are likely and prevents them up front by locking and blocking, accepting reduced concurrency. Optimistic control (e.g. snapshot isolation with SSI) assumes conflicts are rare, lets transactions proceed, and detects conflicts at commit, aborting and retrying when one occurs. Optimistic wins under low contention; pessimistic can win when contention is high and retries would thrash.

Q: When does an optimistic strategy like `SERIALIZABLE`/`SSI` perform poorly?
A: Under high contention on the same data, optimistic control aborts many transactions with `40001`, and the resulting retry storm can waste more work than pessimistic locking would. Each abort discards completed work and the retry may conflict again, so throughput collapses. In such hotspots, explicit locking or queueing the contended operation can outperform repeated optimistic retries.

Q: What does `SELECT ... FOR UPDATE` do, and when would you use it?
A: `SELECT ... FOR UPDATE` takes an exclusive row-level lock on the selected rows, so any other transaction trying to update or lock them must wait until you commit. You use it to serialize a read-modify-write on specific rows — for example, locking a fund's cash-balance row before debiting it — to prevent lost updates without escalating to `SERIALIZABLE`. It is an explicit, targeted alternative to changing the isolation level.

Q: How can explicit locking prevent write skew at a level below `SERIALIZABLE`?
A: By taking a lock that materializes the conflict the anomaly relies on — for instance `SELECT ... FOR UPDATE` on the shared rows both transactions read, or a lock on a summary/guard row — you force the two transactions to serialize on that object. The second transaction blocks until the first commits and then re-reads the updated state, so the invariant holds. This is a deliberate trade: more code and lock contention, but no `SERIALIZABLE` retries.

Q: What is a "materializing conflict" pattern for avoiding write skew?
A: It is deliberately introducing a row whose update both transactions must perform, so a write skew that otherwise touches disjoint rows becomes a detectable write-write conflict on that shared row. For example, both redemption transactions update a single "fund balance guard" row, forcing one to wait or conflict. It converts an invisible read-write dependency into a visible write-write conflict that lower levels catch.

Q: Why is `READ COMMITTED` the common default rather than `SERIALIZABLE`?
A: Because `READ COMMITTED` offers good concurrency with minimal blocking and no serialization-failure retries, while preventing the most jarring anomaly (dirty reads), which suits the majority of workloads. `SERIALIZABLE` adds correctness for multi-row invariants but at the cost of tracking overhead and application retry logic that many simple workloads do not need. Defaults favor broad usability; the architect overrides them where invariants demand.

Q: What does `READ COMMITTED` "silently permit" that can bite you?
A: It permits non-repeatable reads, phantoms, lost updates, and write skew — none of which raise an error, so they corrupt data quietly under concurrency. Code that reads a value, decides, and writes can lose updates or break invariants without any signal. The danger is precisely that nothing fails loudly, so the gap is found later in reconciliation or audit.

Q: What is a read-only serialization anomaly?
A: It is a case where a read-only transaction, by observing a particular snapshot, witnesses a state inconsistent with any serial order even though it writes nothing. Snapshot isolation can produce these, which is one reason it is weaker than serializability. Postgres `SERIALIZABLE` can still abort a read-only transaction (or a concurrent writer) to prevent such anomalies, which surprises people who assume read-only transactions never fail.

Q: Can a read-only transaction be aborted with `40001` under Postgres `SERIALIZABLE`?
A: Yes — under SSI a read-only `SERIALIZABLE` transaction can be rolled back with `40001` if it participates in a dangerous read-write dependency cycle with concurrent writers. To reduce this, you can declare it `READ ONLY DEFERRABLE`, which makes it wait for a safe snapshot and then never be aborted. This is a Postgres-specific tool for long analytical reads that must be serializable.

Q: What does the `READ ONLY DEFERRABLE` transaction mode do in Postgres?
A: Declaring a transaction `SERIALIZABLE READ ONLY DEFERRABLE` makes it wait until it can obtain a snapshot guaranteed free of serialization anomalies, after which it runs without ever being aborted and without taking predicate locks. It trades a possible start-up delay for immunity from `40001` and lower overhead. It suits long-running, serializable, read-only reports such as a consistent NAV or holdings snapshot.

Q: What is the relationship between MVCC, snapshot isolation, and Postgres `REPEATABLE READ`?
A: MVCC is the storage mechanism (multiple row versions); snapshot isolation is the isolation model built on it (each transaction reads one consistent snapshot); and Postgres's `REPEATABLE READ` is its concrete implementation of snapshot isolation. So "Postgres REPEATABLE READ" and "snapshot isolation" are effectively the same guarantee here. `SERIALIZABLE` then layers SSI dependency checks on top of that snapshot model.

Q: Why might a transaction that worked under `READ COMMITTED` suddenly fail under `REPEATABLE READ`?
A: At `REPEATABLE READ` the transaction uses one snapshot for its whole life, so an `UPDATE` targeting a row another transaction has since modified and committed cannot proceed and raises `could not serialize access due to concurrent update` (`40001`). Under `READ COMMITTED` that same `UPDATE` would re-read the latest row version and succeed. The stricter level surfaces the conflict as an error the application must now retry.

Q: What is "first-updater-wins" / "first-committer-wins" in snapshot isolation?
A: When two snapshot-isolation transactions update the same row, the database lets only the first to commit keep its change; the second detects that the row changed under it and is aborted (in Postgres, with `40001` at `REPEATABLE READ`). This rule prevents lost updates on the same object. It does not prevent write skew, which involves writes to *different* objects.

Q: Why does `REPEATABLE READ` prevent lost updates but `READ COMMITTED` may not?
A: At `REPEATABLE READ`, an `UPDATE` to a row changed by a committed concurrent transaction fails with a serialization error instead of silently overwriting, so a read-modify-write cannot lose the other's change unnoticed. At `READ COMMITTED`, the `UPDATE` re-reads the latest version and proceeds, so a naive read-then-write in application code can clobber a concurrent update. To be safe at `READ COMMITTED` you need `SELECT ... FOR UPDATE` or an atomic update expression.

Q: How can you avoid a lost update without raising the isolation level at all?
A: Use an atomic update that computes the new value in the database, e.g. `UPDATE fund_cash SET balance = balance - 10 WHERE id = 1;`, so the read and write happen as one statement under a row lock. Alternatively lock the row first with `SELECT ... FOR UPDATE` or use a compare-and-set `WHERE balance = :expected`. These avoid the read-modify-write race entirely.

Q: What does an optimistic compare-and-set (version column) look like for a fund row?
A: You add a `version` column and update with `UPDATE fund SET nav = :new, version = version + 1 WHERE id = :id AND version = :expected;`, then check the affected-row count. If zero rows changed, someone else updated the row first and you re-read and retry. This is application-level optimistic concurrency that works even at `READ COMMITTED`.

Q: Why does CMU 15-445 frame isolation levels as a contract between database and application?
A: Because the isolation level defines exactly which interleavings the database promises to prevent and which it allows, so the application is responsible for handling everything the chosen level still permits. If you pick `READ COMMITTED`, the database will not stop write skew — that is now your problem to design around. Treating the level as a contract makes explicit who guarantees what, which is the architect's framing.

Q: What is timestamp-ordering concurrency control (briefly)?
A: Timestamp ordering assigns each transaction a timestamp and ensures conflicting operations are processed in timestamp order, aborting transactions that would violate that order. It is an alternative serializability mechanism to `2PL`, avoiding locks but potentially causing more aborts. MVCC borrows the idea of versions tagged with transaction identifiers to decide visibility.

Q: How does MVCC decide which row version a transaction can see?
A: Each row version carries the transaction IDs that created and (if applicable) deleted it, and each transaction has a snapshot describing which transactions had committed when the snapshot was taken. A version is visible if its creating transaction is committed-and-visible to the snapshot and its deleting transaction is not. This visibility check is what implements snapshot isolation on top of MVCC.

Q: What is transaction ID wraparound and why does it matter for durability/correctness?
A: Postgres uses 32-bit transaction IDs for MVCC visibility, and if the oldest unfrozen transaction ID gets too close to the wraparound horizon, old rows could appear to be from the future and become invisible — risking data "loss". Autovacuum freezes old rows to prevent this; if it falls behind, Postgres eventually forces a shutdown to protect data. It is a real operational concern in long-lived, high-write fund databases.

Q: A nightly fund-valuation job sees a different cash balance on its second read of the same row — first thing to check?
A: First check the isolation level the job runs at: at the default `READ COMMITTED`, each statement gets a fresh snapshot, so a committed concurrent update between the two reads explains the discrepancy. Confirm with `SHOW transaction_isolation;` and look for a concurrent settlement transaction. The fix is to run the job at `REPEATABLE READ` (or `SERIALIZABLE`) so all its reads share one snapshot.

Q: A subscription-cap check passes but the cap is breached in production — first thing to check?
A: Suspect a phantom or write skew: at `READ COMMITTED` or even snapshot `REPEATABLE READ`, a concurrent transaction can insert a qualifying order or two checks can each pass on stale views and both write. Confirm the isolation level and look for concurrent inserts matching the cap predicate. The remedy is `SERIALIZABLE`, or a guard lock that serializes the cap evaluation.

Q: Your application logs frequent `ERROR: could not serialize access ... (SQLSTATE 40001)` — is this a bug?
A: Not necessarily — `40001` is the expected, correct signal at `REPEATABLE READ`/`SERIALIZABLE` that the database prevented an anomaly, and the application is supposed to catch it and retry. It becomes a problem only if there is no retry logic (so it surfaces as user errors) or if the rate is so high it indicates a hotspot needing redesign. First check whether a bounded retry-with-backoff is in place around the transaction.

Q: Throughput collapses after switching a hot table to `SERIALIZABLE` — what is happening and what do you do?
A: High contention on the table is producing many `40001` aborts and retries that thrash, so effective throughput drops. Options: revert most operations to a lower level and protect only the truly invariant-critical path, serialize the hot operation with explicit `SELECT ... FOR UPDATE` or a queue, or reduce the conflict footprint (narrower predicates, shorter transactions). The goal is to apply `SERIALIZABLE` surgically rather than globally.

Q: Two psql sessions both `BEGIN` and `UPDATE` the same row and one hangs — what is going on?
A: The second `UPDATE` is blocked waiting for the first transaction's row-level write lock, which is held until that transaction commits or rolls back. This is normal lock waiting, not an isolation anomaly; finishing or aborting the first session releases the lock and the second proceeds (or, at `REPEATABLE READ`, fails with `40001`). You can inspect the wait in `pg_locks` and `pg_stat_activity`.

Q: A long-running report holds a transaction open and table bloat grows — what is the link to isolation/MVCC?
A: An open transaction keeps an old snapshot alive, so MVCC cannot remove dead row versions newer than that snapshot's horizon; `VACUUM` is held back and the table bloats. The report's isolation does not change the data it sees, but its mere longevity pins the cleanup horizon. The fix is to shorten the transaction, use `READ ONLY DEFERRABLE` where appropriate, or run heavy reads against a replica.

Q: Why might you route a NAV-history report to a read replica or a deferrable read-only transaction?
A: To keep a long, consistent read off the primary's MVCC cleanup horizon and out of the way of write contention, while still getting a consistent snapshot. A `SERIALIZABLE READ ONLY DEFERRABLE` transaction gives a serializable, abort-free snapshot for the report; a replica offloads the I/O entirely (at the cost of replication lag). Either keeps the write-heavy order path healthy.

Q: How do isolation levels relate to the regulated requirement that fund records be auditable and reproducible?
A: Auditability depends on the database never persisting states that no serial execution could have produced, because an anomaly leaves a record that cannot be explained by any ordering of legitimate operations. Choosing the right level (often `SERIALIZABLE` for invariant-critical paths) ensures every committed state is reconstructible from a serial history. That reproducibility is exactly what auditors and reconciliation rely on.

Q: Summarize the one-line difference between snapshot isolation and serializability for a design review.
A: Snapshot isolation gives every transaction a consistent point-in-time view and prevents read anomalies and same-row lost updates, but still allows write skew because it ignores cross-row read-write dependencies; serializability additionally guarantees the outcome equals some serial order, eliminating write skew at the cost of retryable serialization failures. In Postgres terms: `REPEATABLE READ` is snapshot isolation, `SERIALIZABLE` adds SSI dependency checks. Use the stronger one wherever a multi-row invariant must hold.

Q: What SQLSTATE code does Postgres raise when a deadlock is detected, and how does it differ from `40001`?
A: A deadlock is reported with SQLSTATE `40P01` (`deadlock_detected`), whereas a serialization failure is `40001` (`serialization_failure`); both are in transaction-rollback Class 40 but signal different causes. `40P01` means two transactions each hold a lock the other needs and Postgres broke the cycle by aborting one; `40001` means a snapshot/SSI conflict, not a lock cycle. Both are retryable, but `40P01` often points to inconsistent lock-acquisition order in your code, which retrying alone will not fix.

Q: What is the default value of `deadlock_timeout` in Postgres, and what does it control?
A: The default is `1s`, and it sets how long a transaction waits on a lock before Postgres runs deadlock detection. Detection is deferred because the check is expensive and most lock waits resolve on their own well within a second, so checking immediately would waste CPU. Setting it too low wastes cycles scanning for deadlocks; too high lets a real deadlock stall transactions longer before it is broken with `40P01`.

Q: What SQLSTATE does `SELECT ... FOR UPDATE NOWAIT` raise when a row is already locked?
A: It raises SQLSTATE `55P03` (`lock_not_available`) with a message like `could not obtain lock on row in relation ...`, instead of blocking. `NOWAIT` tells Postgres to fail fast rather than queue behind the holder's row lock, which suits interactive paths that must not hang. The same `55P03` code is also produced when `lock_timeout` expires while waiting for a lock.

Q: How does `SKIP LOCKED` differ from `NOWAIT` on a `SELECT ... FOR UPDATE`?
A: `NOWAIT` raises `55P03` if any selected row is already locked, failing the whole statement, whereas `SKIP LOCKED` silently omits the locked rows from the result and locks only the free ones. `SKIP LOCKED` therefore returns an intentionally incomplete view, which is exactly what a worker queue wants so concurrent consumers grab disjoint jobs. You would never use `SKIP LOCKED` for a correctness read because it hides rows; you use it to dispatch work.

Q: How would you implement a concurrent fund-order processing queue so workers do not grab the same order?
A: Have each worker run `SELECT ... FROM order_queue WHERE status='pending' ORDER BY id FOR UPDATE SKIP LOCKED LIMIT 1;` inside a transaction, process the row, mark it done, and commit. `SKIP LOCKED` makes each worker skip orders another worker has already locked, so N workers pull N distinct orders without blocking. This is the canonical Postgres queue pattern and avoids both lost work and double-processing.

Q: What is the purpose of `lock_timeout`, and which error does it produce?
A: `lock_timeout` caps how long any statement will wait to acquire a lock before aborting with SQLSTATE `55P03` (`lock_not_available`); the default `0` means wait forever. Setting e.g. `SET lock_timeout = '3s';` prevents a single blocked `UPDATE` on a hot fund-balance row from hanging a request indefinitely. It is distinct from `statement_timeout`, which bounds total statement run time regardless of locking.

Q: What does `idle_in_transaction_session_timeout` do, and why does it matter for MVCC bloat?
A: It automatically aborts a session that has held a transaction open but idle for longer than the configured interval, freeing the locks and snapshot it was holding. This matters because an open transaction pins the MVCC cleanup horizon, blocking `VACUUM` and causing table bloat, and a forgotten `psql` session left after `BEGIN` is a common cause. Setting it to a few minutes is a standard safeguard on write-heavy fund databases.

Q: What does the parameter `default_transaction_isolation` control, and how is it scoped?
A: It sets the isolation level that new transactions start with when they do not override it, and it can be set per session with `SET` or globally in `postgresql.conf`. Changing it does not affect a transaction already in progress, only ones begun afterward. Setting it to `serializable` server-wide is a heavy hammer — every transaction then pays SSI cost and must handle `40001` — so it is usually set per-connection for the paths that need it.

Q: How do you make a serializable transaction also read-only and abort-immune in one statement?
A: Begin it as `BEGIN TRANSACTION ISOLATION LEVEL SERIALIZABLE READ ONLY DEFERRABLE;`, which combines the serializable guarantee with `READ ONLY` and `DEFERRABLE`. The transaction waits for a snapshot proven free of serialization anomalies, then runs without predicate-lock overhead and can never be aborted with `40001`. It is the precise tool for a long, consistent NAV or holdings report that must be serializable yet must not fail.

Q: What is the default value of `max_pred_locks_per_transaction`, and what happens when it is exceeded?
A: The default is `64`, and it sizes the shared predicate-lock table that SSI uses to track what serializable transactions read. When a transaction needs more fine-grained predicate locks than the budget allows, Postgres promotes them to coarser granularity (page, then whole relation) rather than failing. That promotion is safe but increases false-positive `40001` aborts, so heavy serializable workloads sometimes raise this parameter.

Q: How does SSI promote predicate-lock granularity, and what is the side effect?
A: SSI starts with tuple-level predicate locks and, when a transaction accumulates too many on one page or relation (governed by `max_pred_locks_per_page` and `max_pred_locks_per_relation`), it promotes them to cover the whole page or relation. Coarser locks cover rows the transaction never actually read, so they make the dependency tracker flag conflicts that would not really violate serializability. The visible side effect is a higher rate of spurious `40001` retries under memory pressure.

Q: Why can `SERIALIZABLE` produce more aborts on a small table than on a large one with the same query?
A: On a small table the rows likely share a few pages, so SSI predicate locks promote to page or relation granularity quickly, making unrelated transactions appear to conflict. A large table spreads the same rows across many pages, keeping predicate locks fine-grained and conflicts rare. So an aggressively-aborting `SERIALIZABLE` workload sometimes improves simply by spreading data or adding an index that narrows the scanned set.

Q: How does adding an index reduce serialization failures under `SERIALIZABLE`?
A: A selective index lets a query touch only the matching rows instead of sequentially scanning the table, so SSI takes predicate locks on just those rows rather than on every page it reads. Fewer and narrower predicate locks mean fewer read-write dependencies recorded, hence fewer false-positive `40001` aborts. The same query that thrashes on a seq scan can run cleanly once the planner uses an index.

Q: A transaction does `SELECT ... FOR UPDATE` on two rows in different orders across sessions and deadlocks — what is the fix?
A: The deadlock (`40P01`) arises because session A locks row 1 then row 2 while session B locks row 2 then row 1, so each waits on the other. The fix is to impose a consistent lock-acquisition order — for instance always lock rows in ascending primary-key order — so the cycle cannot form. Retrying alone is a band-aid; deterministic ordering removes the deadlock at the source.

Q: How do you inspect which transaction is blocking another in Postgres?
A: Query `pg_stat_activity` for sessions with `wait_event_type = 'Lock'`, and use `pg_blocking_pids(pid)` to get the PIDs holding the lock the blocked session wants. Joining against `pg_locks` shows the exact relation and lock mode involved. This is the standard first step when a `psql` session "hangs" on an `UPDATE` — you confirm it is lock waiting, not an isolation anomaly.

Q: In `pg_locks`, how do you tell SSI predicate locks apart from ordinary row locks?
A: Predicate locks appear with `mode = 'SIReadLock'` and represent reads tracked by SSI, while write locks show modes like `RowExclusiveLock` or the row-level locks recorded in tuple infomask. `SIReadLock` entries never block anyone — they exist only so the dependency checker can detect dangerous read-write patterns. Seeing many `SIReadLock` rows is normal under `SERIALIZABLE` and is not a contention problem by itself.

Q: Why does Postgres not record row-level write locks as individual rows in `pg_locks`?
A: Row-level locks are stored on the tuple itself (in the heap, via the row's `xmax` and infomask bits) rather than in the in-memory lock table, so they do not consume shared-memory lock slots and are effectively unlimited. `pg_locks` shows a single `RowExclusiveLock` at the table level plus a transaction-ID lock, and you find the actual row holder through `xmax`. This is why `max_locks_per_transaction` limits object locks, not the number of rows you may lock.

Q: What is the difference between `FOR UPDATE` and `FOR NO KEY UPDATE`?
A: `FOR UPDATE` takes the strongest row lock and blocks concurrent `FOR KEY SHARE`, which foreign-key checks use, whereas `FOR NO KEY UPDATE` is slightly weaker and does not block those key-share locks. You use `FOR NO KEY UPDATE` when you will modify non-key columns and want to avoid contending with child-row insertions that take key-share locks on the parent. Plain `UPDATE` of non-key columns actually acquires `FOR NO KEY UPDATE` internally.

Q: What is `FOR KEY SHARE`, and when does Postgres take it automatically?
A: `FOR KEY SHARE` is the weakest row lock; it blocks only `FOR UPDATE` (deletion or key change) but allows `FOR NO KEY UPDATE` and `FOR SHARE`. Postgres takes it automatically on a parent row when you insert a child row referencing it via a foreign key, so the parent's key cannot be deleted or changed mid-transaction. This is why heavy child inserts can unexpectedly contend with attempts to `FOR UPDATE` the parent.

Q: A foreign-key insert is blocking on the parent row even though no one updated it — what is happening?
A: The insert takes a `FOR KEY SHARE` lock on the referenced parent row to ensure the key stays valid until commit, and it is waiting because another transaction holds a conflicting `FOR UPDATE`/`FOR NO KEY UPDATE` on that parent. The block is the FK machinery doing its job, not an isolation anomaly. Reducing the window the parent is locked, or locking parents in a consistent order, relieves the contention.

Q: How does `READ COMMITTED` handle an `UPDATE` whose target row was changed by a concurrent committed transaction?
A: At `READ COMMITTED`, the `UPDATE` does not fail; instead Postgres re-evaluates the row at the latest committed version (an "EvalPlanQual" re-check), re-tests the `WHERE` clause, and applies the change if it still matches. This is why `READ COMMITTED` quietly avoids the `40001` that `REPEATABLE READ` would raise in the same race. The subtlety is that the `WHERE` is re-checked against the new version, which can surprise you if it now no longer matches.

Q: Why can the same `UPDATE ... WHERE` statement update a different row version at `READ COMMITTED` than you expected?
A: Because if a concurrent transaction commits a change to the target row mid-statement, `READ COMMITTED` re-reads the newest committed version and re-applies the `WHERE` predicate to it, possibly skipping or altering rows based on the updated values. Your `UPDATE` therefore acts on data committed after your statement began, not on the snapshot you read earlier. This "read latest version on write conflict" behavior is unique to `READ COMMITTED` and absent at `REPEATABLE READ`.

Q: How can the EvalPlanQual re-check at `READ COMMITTED` itself cause a lost update in application logic?
A: If your application reads a balance in one statement, decides in code, then issues a separate `UPDATE` with a stale literal value, the re-check only re-validates the `WHERE`, not your computed value, so you can still overwrite a concurrent change. The database "helpfully" applies your write to the latest row, but using your outdated number. The remedy is an atomic `SET balance = balance - :amt` or `SELECT ... FOR UPDATE` so the computation uses the current value.

Q: What is the difference between `statement_timeout` and `idle_in_transaction_session_timeout`?
A: `statement_timeout` aborts a single statement that runs longer than the limit, protecting against runaway queries, while `idle_in_transaction_session_timeout` aborts a transaction that is open but doing nothing, protecting against held snapshots and locks. The first guards CPU/IO on active work; the second guards MVCC and lock hygiene against abandoned transactions. A robust fund service typically sets both.

Q: Why does a single very long `SELECT` on the primary hurt write throughput even though it takes no write locks?
A: Its open snapshot holds back the MVCC cleanup horizon, so autovacuum cannot remove dead tuples created by concurrent writers, and the tables those writers touch bloat and slow down. The reader never blocks the writers directly, but it starves vacuum of the ability to reclaim space. Routing the long read to a replica or a `READ ONLY DEFERRABLE` transaction, or shortening it, restores write health.

Q: How does `hot_standby_feedback` interact with running long reads on a replica?
A: With `hot_standby_feedback = on`, a replica tells the primary about the oldest snapshot its queries need, so the primary delays vacuuming rows the replica still requires — protecting the read from being cancelled but reintroducing bloat pressure on the primary. With it off, the primary vacuums freely and a long replica query risks cancellation with `ERROR: canceling statement due to conflict with recovery`. It is a direct trade between query survival on the replica and bloat on the primary.

Q: What error indicates a replica query was cancelled because of recovery conflict, and how do you mitigate it?
A: The replica raises `ERROR: canceling statement due to conflict with recovery`, meaning replay needed to remove rows or take a lock that the running query depended on. Mitigations include raising `max_standby_streaming_delay`, enabling `hot_standby_feedback`, or moving the long read elsewhere. For a NAV-history report this is the classic reason a replica-side query dies partway through.

Q: Does running a report on a read replica give you a transactionally consistent snapshot?
A: Yes — a hot-standby replica replays WAL in commit order, so a query there sees a consistent committed state, just one that may lag the primary by the replication delay. You get snapshot consistency but not recency, which is fine for an as-of report and wrong for a read-your-writes check. The architect must distinguish "consistent" from "current" when choosing the replica.

Q: How would you measure whether `SERIALIZABLE` is causing excessive retries in production?
A: Count `40001` (and `40P01`) errors over time from your application's error metrics or the Postgres logs (`log_min_error_statement` capturing them), and compare retry rate to transaction throughput. A few percent is normal; a high and rising fraction on one table signals a hotspot. You then decide between narrowing the level's scope, adding an index, or serializing the hot operation explicitly.

Q: Why is bounded retry with exponential backoff and jitter the right pattern for `40001`?
A: An unbounded retry can loop forever on a persistent hotspot, while immediate retries by many clients re-collide in lockstep (a retry storm); backoff spreads attempts out and jitter de-synchronizes them so they stop colliding. A cap converts a hopeless conflict into a surfaced error rather than an infinite loop. This is the standard contention-control recipe layered on top of the database's correctness guarantee.

Q: When retrying a transaction after `40001`, why must any side effects be deferred or made idempotent?
A: Because the transaction may run several times before it commits, so anything it does outside the database — sending an order confirmation, emitting a Kafka message, calling an external API — would happen once per attempt unless deferred until after commit or made idempotent. A naive retry could fire three confirmation emails for one booked order. The safe pattern is to perform external effects only after a successful commit, e.g. via an outbox table drained post-commit.

Q: What is the transactional outbox pattern and how does it relate to isolation and retries?
A: The outbox pattern writes an intended external event into a database table in the same transaction as the business change, so the event is committed atomically with the data and a separate process publishes it afterward. This ties the "should we send this message" decision to the same atomic, isolated commit, so a rolled-back or retried transaction never emits a phantom message. It is how you get exactly-once-ish external effects on top of transactional retries.

Q: Why is the "C" in `ACID` partly the application's responsibility rather than the database's?
A: Consistency here means application-defined invariants (a fund's positions summing correctly, debits matching credits) hold true, and the database cannot know those rules unless you encode them as constraints. The database guarantees atomicity, isolation, and durability mechanically, but it will happily commit application-logical nonsense that violates no declared constraint. So enforcing "C" means writing the right `CHECK`, `UNIQUE`, and foreign-key constraints, or the right transaction logic, yourself.

Q: How can a `CHECK` constraint or exclusion constraint enforce an invariant that would otherwise require `SERIALIZABLE`?
A: Some invariants can be pushed into the schema — e.g. a `CHECK (balance >= 0)` on a single row, or an `EXCLUDE` constraint preventing overlapping reservations — so the database rejects the violating write directly regardless of isolation level. Where the invariant fits one row or a constraint, this is cheaper and more robust than relying on `SERIALIZABLE` retries. The limitation is multi-row aggregate invariants (sum of withdrawals), which constraints cannot express and which still need serializable logic or a guard row.

Q: Why can't a per-row `CHECK (balance >= 0)` prevent the fund overdraft write-skew scenario?
A: Because in the write-skew case each transaction reads the shared total, decides independently, and writes a *different* withdrawal row, so no single row's `CHECK` is ever violated at write time — the overdraft emerges only from the rows' combined effect. A constraint sees one row's state, not the aggregate across rows two transactions touched. Preventing it needs `SERIALIZABLE`, a `SELECT ... FOR UPDATE` on a balance row, or a materialized guard row.

Q: How does using a single mutable "balance" row instead of summing withdrawal rows change the concurrency picture?
A: If both redemptions update one balance row (`UPDATE fund SET cash = cash - :amt WHERE id=:id AND cash >= :amt`), the conflict becomes a write-write conflict on that row, so the second transaction either blocks then re-checks or fails, and the overdraft cannot occur even at `READ COMMITTED`. You have converted an invisible read-write dependency into a visible same-row contention. The trade is reduced concurrency on that row, which is usually acceptable for a balance.

Q: What does `BEGIN ... COMMIT` actually guarantee about WAL at the moment `COMMIT` returns?
A: By default (`synchronous_commit = on`), when `COMMIT` returns success the WAL record for that transaction has been flushed to durable storage on the primary, so a crash immediately after cannot lose it. The data pages may still be only in memory; durability rests on the flushed log, replayed during recovery. This is the concrete mechanism behind the "D" in ACID.

Q: What does setting `synchronous_commit = off` trade away, and is atomicity affected?
A: It tells Postgres to acknowledge `COMMIT` before the WAL flush completes, so a crash in the small window after acknowledgement can lose the most recent committed transactions — sacrificing durability for lower latency. Atomicity and isolation are unaffected: you never see a partial or inconsistent transaction, you can only lose whole recent commits. For fund bookings this is usually unacceptable; for low-value high-volume telemetry it can be a deliberate choice.

Q: What is the difference between `synchronous_commit = off` and `fsync = off`?
A: `synchronous_commit = off` only delays the WAL flush for commits, risking loss of a few recent transactions but keeping the database recoverable and consistent after a crash. `fsync = off` disables forcing data to disk entirely, so a crash can leave the database corrupt and unrecoverable — it is a benchmarking-only setting. The first is a bounded, principled durability relaxation; the second abandons crash safety altogether.

Q: How do synchronous replication settings turn durability into a multi-node guarantee?
A: With `synchronous_standby_names` set and `synchronous_commit = on` (or `remote_apply`), a commit does not return until at least one standby has received (or applied) the WAL, so a committed fund transaction survives loss of the primary. This extends "D" beyond a single disk to surviving node failure, at the cost of commit latency tied to the standby. It is how a regulated platform meets a "no acknowledged transaction is ever lost" requirement.

Q: Why does the choice between `synchronous_commit` values like `remote_write` and `remote_apply` matter for read-your-writes on a replica?
A: `remote_apply` waits until the standby has actually replayed the commit, so a read routed to that standby immediately after will see the write, whereas `remote_write`/`on` only ensure the WAL reached the standby's memory or disk, not that it is visible to queries yet. So `remote_apply` is what makes a synchronous replica safe for read-your-writes; weaker modes still expose a lag window. The stronger mode costs more commit latency.

Q: How would you decide between using `SERIALIZABLE` globally versus `READ COMMITTED` plus targeted locking for a fund platform?
A: Use `SERIALIZABLE` on the specific transactions that enforce multi-row invariants (caps, redemption-vs-cash, allocation limits) where correctness is non-negotiable, and keep the high-volume, single-row or constraint-protected paths at `READ COMMITTED` for throughput. Global `SERIALIZABLE` is simpler to reason about but makes every transaction pay SSI cost and need retry logic. The architect's answer is "surgical serializability": strongest level only where an invariant actually spans rows.

Q: How do you reproduce a lost update at `READ COMMITTED` with two psql sessions?
A: In both sessions `BEGIN`, then each `SELECT balance FROM fund WHERE id=1;` (say both read 100), then in app-style each computes 100-10 and issues `UPDATE fund SET balance=90 WHERE id=1;`; A commits, then B commits, and B's literal 90 overwrites A's, so one 10-debit is lost. The blind read-then-write with a literal is the bug. Replacing it with `UPDATE fund SET balance=balance-10 WHERE id=1;` makes both debits stick.

Q: At `REPEATABLE READ`, what happens to the second session in that lost-update interleaving?
A: When B issues its `UPDATE` to a row A already updated and committed, B fails with `ERROR: could not serialize access due to concurrent update` (`40001`) instead of silently overwriting. `REPEATABLE READ` applies first-committer-wins on same-row writes, so the lost update is converted into a retryable error. On retry B re-reads 90 and correctly debits to 80.

Q: Why can two `SELECT count(*)` queries in one `READ COMMITTED` transaction disagree even with no error?
A: Because each statement at `READ COMMITTED` takes a fresh snapshot, so an insert committed by another session between the two counts is visible to the second and not the first — a phantom — and Postgres signals nothing because no anomaly is forbidden at that level. The disagreement is "correct" behavior for the level. Running the transaction at `REPEATABLE READ` pins one snapshot and the counts match.

Q: A batch job at `REPEATABLE READ` fails with `40001` on its very first write after many reads — why, and what is the cheapest fix?
A: Its single snapshot was taken at the first read, and by the time it writes, a concurrent transaction has committed a change to a targeted row, so first-committer-wins aborts it with `could not serialize access due to concurrent update`. The cheapest fix is to wrap the job in bounded retry-with-backoff so it restarts from `BEGIN` and gets a fresh snapshot. If conflicts are frequent, shortening the transaction or locking the contended rows up front reduces the abort rate.

Q: Why is "just raise the isolation level" sometimes the wrong fix for a concurrency bug?
A: Because a higher level converts silent anomalies into `40001`/`40P01` errors the application must now catch and retry, and on a hot path it can collapse throughput with retry storms rather than fixing the design. Often the right fix is an atomic update, a targeted `FOR UPDATE`, a constraint, or a guard row that removes the conflict cheaply. Raising the level is correct only when the invariant genuinely spans multiple rows and the contention is low enough to absorb retries.

Q: What single question should an architect ask to decide if a workload needs `SERIALIZABLE`?
A: "Is there an invariant that spans more than one row and is checked by reading then writing different rows?" — if yes, lower levels permit write skew and you need `SERIALIZABLE` or an explicit serializing lock; if the invariant fits one row or a declarative constraint, a lower level suffices. This reframes the decision around the shape of the invariant rather than a vague desire for safety. It is the crisp test that separates "needs SSI" from "needs a `CHECK` or an atomic update".

Q: What does a `SAVEPOINT` give you within a transaction, and how does it relate to atomicity?
A: A `SAVEPOINT` marks a point you can roll back to with `ROLLBACK TO SAVEPOINT name` without aborting the whole transaction, creating a subtransaction whose effects can be undone independently. It refines atomicity from all-or-nothing for the whole transaction to "all-or-nothing per savepoint segment", so you can retry just a failed step. A typical use is attempting a risky insert, catching its error, rolling back to the savepoint, and continuing the outer transaction.

Q: How do you monitor accumulated serialization failures and deadlocks at the database level in Postgres?
A: Query `pg_stat_database`, whose `deadlocks` column counts deadlocks (`40P01`) detected per database since stats reset; serialization failures are not separately counted there, so you track `40001` from application metrics or by parsing the server log. Comparing the `deadlocks` count over time tells you whether lock-ordering problems are growing. You reset the counters with `pg_stat_reset()` to measure a clean window after a change.
