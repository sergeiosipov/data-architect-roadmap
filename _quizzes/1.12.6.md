# Phase 6 · 1.12.6 + 3.2.6 FIBO & knowledge graphs / RDF — 100 self-test questions

Q: What is an RDF triple, and what are its three parts called?
A: An RDF triple is the atomic unit of the Resource Description Framework: a single fact made of a subject, a predicate, and an object. The subject is the thing being described, the predicate is the property or relationship, and the object is the value or related thing. For example `<entity:A> <fibo:hasLegalName> "Acme Fund SICAV"` asserts that entity A has the legal name "Acme Fund SICAV".

Q: Why is RDF often described as a "graph" model rather than a "table" model?
A: Because each triple is a directed edge from the subject node to the object node, labelled by the predicate, so a set of triples forms a labelled directed graph. Nodes are shared across triples whenever the same IRI appears, which lets data from different sources join naturally without predefined schemas or foreign keys. This is the structural reason RDF excels at "who owns whom" entity networks where relationships matter more than rows.

Q: What is an IRI (or URI) in RDF and why does RDF use them for subjects and predicates?
A: An IRI (Internationalized Resource Identifier, a superset of URI) is a globally unique name used to identify resources and properties unambiguously across the web. RDF uses IRIs so that the same concept referenced in two datasets — say a GLEIF entity IRI — denotes exactly the same node and the graphs merge correctly. Without global identifiers you would have name collisions and could not safely combine data from EDM Council, GLEIF, and your own master data.

Q: In RDF, what kinds of things can appear as the object of a triple?
A: The object can be either another resource (an IRI or a blank node) or a literal value such as a string, number, date, or boolean. A resource object creates a link to another node in the graph, while a literal object is a terminal data value with no outgoing edges. For instance `hasLegalName "Acme"` ends in a string literal, whereas `hasDirectParent <lei:529900...>` links to another entity node.

Q: What is a blank node in RDF, and when would you use one?
A: A blank node is an anonymous resource that has no IRI of its own; Jena's tutorial describes them as acting "like existentially qualified variables in first order logic". You use them to group structured data that does not need a stable external identifier, such as an address or a relationship period made of several sub-properties. The gotcha is that blank-node identifiers are local to a document, so they cannot be referenced from outside and complicate merging or diffing graphs.

Q: What is a literal in RDF, and what optional annotations can it carry?
A: A literal is a data value such as text or a number that forms a leaf node in the graph. A literal may carry a datatype IRI (for example `xsd:date` or `xsd:decimal`) or, for plain strings, a language tag like `@en` or `@fr`. So `"2026-06-13"^^xsd:date` is a typed literal and `"Société"@fr` is a language-tagged string — both matter for a multilingual Luxembourg fund register.

Q: What does RDFS stand for and what is its purpose?
A: RDFS is RDF Schema, a lightweight vocabulary layer built on top of RDF for describing classes and properties. It lets you declare that resources are instances of classes and that properties have domains and ranges, giving basic type structure to an otherwise flat set of triples. It is the minimal "schema" mechanism before you reach the richer expressiveness of OWL.

Q: Name two core RDFS constructs and say what each declares.
A: `rdfs:Class` declares that a resource is a class (a type), and `rdf:type` (often written `a` in Turtle) asserts that an individual is an instance of a class. `rdfs:subClassOf` declares a class is a specialization of another, and `rdfs:subPropertyOf` does the same for properties. Together they build taxonomies, so `Fund rdfs:subClassOf LegalEntity` lets a reasoner infer every fund is also a legal entity.

Q: What do `rdfs:domain` and `rdfs:range` mean, and what is a common misconception about them?
A: `rdfs:domain` states the class of subjects that use a property and `rdfs:range` states the class of its objects or the datatype of its values. The common misconception is that they act as validation constraints that reject bad data; in fact they are inference axioms — declaring a domain causes a reasoner to infer the subject's type rather than raise an error. To actually enforce constraints you use SHACL, not RDFS domain/range.

Q: What is OWL and how does it relate to RDFS?
A: OWL is the Web Ontology Language, a W3C standard that extends RDFS with far richer modelling constructs grounded in description logic. Where RDFS offers basic classes and subclass hierarchies, OWL adds things like property characteristics, cardinality, equivalence, disjointness, and complex class expressions. FIBO is published in OWL precisely because financial concepts need this expressiveness to be machine-readable and reasoned over.

Q: Give two OWL property characteristics and explain what each lets a reasoner conclude.
A: `owl:TransitiveProperty` lets a reasoner conclude that if A relates to B and B relates to C, then A relates to C — ideal for an "is an ancestor of" style ownership relation. `owl:inverseOf` declares that one property is the reverse of another, so asserting `hasParent` automatically yields `hasChild` in the other direction. Others include `owl:SymmetricProperty` and `owl:FunctionalProperty`, which constrains a property to at most one value per subject.

Q: What does `owl:sameAs` assert, and why is it both powerful and dangerous?
A: `owl:sameAs` asserts that two IRIs denote the very same real-world entity, so a reasoner merges all their statements. It is powerful for entity resolution — linking your internal counterparty ID to a GLEIF LEI for the same firm. It is dangerous because an incorrect `owl:sameAs` silently fuses two distinct entities and propagates every wrong fact, which in fund data could conflate two sub-funds and corrupt downstream NAV attribution.

Q: What is the difference between `owl:ObjectProperty` and `owl:DatatypeProperty`?
A: An `owl:ObjectProperty` links an individual to another individual, so its object is a resource node in the graph. An `owl:DatatypeProperty` links an individual to a literal value such as a string or date. So `hasUltimateParent` is an object property linking two legal entities, whereas `hasInceptionDate` is a datatype property pointing to an `xsd:date` literal.

Q: In Apache Jena, what does the `Model` interface represent?
A: In Jena the `Model` is the central interface representing an RDF graph as a set of statements (triples). It provides the API to create resources and properties, add and remove statements, query with `listStatements`, and read or write serializations. You typically start with `ModelFactory.createDefaultModel()` and then build or load triples into it.

Q: In Jena's API, what are `Resource`, `Property`, `Literal`, and `Statement`?
A: A `Resource` is a node identified by an IRI or a blank node; a `Property` is a resource used as a predicate; a `Literal` is a data value; and a `Statement` is one complete triple binding subject, predicate, and object. You create them with methods like `model.createResource(uri)`, `model.createProperty(uri)`, and `model.createLiteral(value)`. A `Statement` is what `listStatements()` returns and what `addProperty()` produces.

Q: Which Jena methods would you use to add a fact and to list facts in a model?
A: You add a fact with `resource.addProperty(property, value)` on the subject resource, or `model.add(statement)`. You read facts back with `model.listStatements()`, optionally passing a selector to filter by subject, predicate, or object. To follow outgoing edges of one resource you call `resource.listProperties()` or `resource.getProperty(p)`.

Q: What is Turtle, and why is it preferred over RDF/XML for hand-authoring?
A: Turtle (Terse RDF Triple Language) is a compact, human-readable text serialization of RDF that uses prefixes, semicolons, and commas to abbreviate repeated subjects and predicates. It is preferred over RDF/XML because it is far less verbose and not tangled in XML namespace rules, so a Turtle file mapping a security master to FIBO terms is readable in a diff. RDF/XML remains a valid format but is painful to read or edit by hand.

Q: In Turtle syntax, what do the `@prefix` directive and a trailing `a` keyword mean?
A: `@prefix` declares a short alias for a long namespace IRI, for example `@prefix fibo-be: <https://spec.edmcouncil.org/fibo/ontology/BE/> .`, so you can write `fibo-be:LegalEntity` instead of the full IRI. The keyword `a` is Turtle shorthand for the predicate `rdf:type`, so `:fundX a fibo-fbc:Fund` reads "fund X is of type Fund". These conventions make Turtle concise without changing the underlying triples.

Q: What does the punctuation `;` and `,` do in Turtle?
A: A semicolon `;` repeats the same subject for a new predicate-object pair, letting you describe many properties of one entity in a block. A comma `,` repeats both subject and predicate for a new object, listing several values of the same property. So `:fund a :Fund ; :hasShareClass :A , :B .` says the fund has type Fund and two share classes A and B without rewriting the subject.

Q: What is N-Triples and when is it the right serialization to use?
A: N-Triples is the simplest RDF serialization: one complete triple per line with full IRIs and no prefixes or abbreviations. Its verbosity is a feature for machine processing because it is trivially line-oriented, so you can `grep`, `split`, `sort`, or stream a multi-gigabyte GLEIF dump without an XML or Turtle parser. It is the format of choice for bulk loading and for line-by-line diffs of large datasets.

Q: What is SPARQL and what is its relationship to RDF?
A: SPARQL is the standard W3C query language for RDF, the graph analogue of what SQL is to relational tables. You query by specifying a graph pattern of triples with variables, and the engine returns all bindings of those variables that match the data. It is the native way to ask a triple store like Fuseki for paths and relationships across an entity graph.

Q: In SPARQL, what does a variable look like and what is a basic graph pattern?
A: A SPARQL variable is written with a leading `?` or `$`, for example `?entity`. A basic graph pattern is a set of triple patterns — triples where some positions are variables — that the engine matches against the data, binding variables to the values that satisfy all patterns jointly. For instance `?child fibo-be:hasDirectParent ?parent .` binds `?child` and `?parent` to every direct parent relationship in the graph.

Q: What are the four SPARQL query forms and what does each return?
A: `SELECT` returns a table of variable bindings, like SQL rows. `ASK` returns a single boolean for whether any match exists. `CONSTRUCT` returns a new RDF graph built from a template, and `DESCRIBE` returns RDF describing matched resources as the endpoint sees fit. For reading ownership chains you almost always use `SELECT`, and `CONSTRUCT` when you want to materialize a derived sub-graph.

Q: What does the `PREFIX` clause do at the top of a SPARQL query?
A: `PREFIX` binds a short prefix label to a namespace IRI for the duration of the query, mirroring Turtle's `@prefix`. It lets you write `gleif-L2:hasUltimateParent` instead of the full IRI in every triple pattern. Forgetting a needed prefix, or mistyping the namespace, is a common cause of a query returning zero rows even though the data is present.

Q: Explain the meaning of this SPARQL pattern: `?child gleif-L2:hasDirectAccountingConsolidationParent ?parent .`
A: It matches every pair where a child entity has a recorded direct accounting consolidation parent, binding `?child` and `?parent` to each such pair. In a `SELECT ?child ?parent` query it returns the immediate ownership edges of the loaded GLEIF Level-2 data. It captures one hop only; to walk the full chain to the ultimate parent you need a property path or recursion.

Q: What is a SPARQL property path and why is it central to walking ownership chains?
A: A property path lets a single triple pattern match a route of one or more predicates between two nodes rather than just one edge. It is central to ownership because the chain from a sub-fund to its ultimate parent can be any number of hops, and a path expresses "follow this relationship any number of times" without writing recursive code. This is exactly what the lesson's "walk the ultimate-parent chains" task needs.

Q: What do the `+` and `*` operators mean in a SPARQL property path?
A: `+` matches one or more occurrences of the path, so `?x :hasParent+ ?ancestor` finds every ancestor reachable by at least one hop. `*` matches zero or more occurrences, which additionally binds the node to itself at zero length. For ultimate-parent walking you usually want `+` so an entity is not returned as its own ancestor.

Q: What do the `/`, `|`, and `^` property-path operators do?
A: `/` is sequence, chaining two predicates so `:a/:b` follows `:a` then `:b`. `|` is alternative, matching either path, useful when direct and ultimate parent predicates should both count. `^` is inverse, traversing an edge backwards, so `^:hasParent` walks from parent to child. Combining them lets you express rich navigation in one compact pattern.

Q: Write a SPARQL pattern that returns all ancestors of an entity using a transitive path, and explain it.
A: `?entity gleif-L2:hasDirectAccountingConsolidationParent+ ?ancestor .` returns every entity reachable by following the direct-parent edge one or more times. The `+` performs the transitive closure so a deep chain of holding companies all appear as ancestors. To pick only the topmost you would additionally filter for ancestors that themselves have no parent, or use the ultimate-parent predicate directly.

Q: How would you find the ultimate parent of a specific LEI using the ready-made GLEIF predicate?
A: You bind the specific child and ask for its ultimate-parent object, for example `<lei:529900...> gleif-L2:hasUltimateAccountingConsolidationParent ?ultimate .` in a `SELECT ?ultimate` query. GLEIF publishes the ultimate parent directly as Level-2 data, so you do not always need a recursive path. The path approach is the cross-check that the published ultimate parent matches the top of the direct-parent chain.

Q: What does the SPARQL `FILTER` clause do, and give an example relevant to fund data.
A: `FILTER` restricts solutions to those where a boolean expression is true, applied after pattern matching. For example `FILTER(?country = "LU")` keeps only Luxembourg-domiciled entities, or `FILTER(?nav > 0)` removes zero or negative NAV rows. It operates on bound variables, so a `FILTER` referencing an unbound variable inside an `OPTIONAL` can keep or drop rows in ways that surprise newcomers.

Q: What is the purpose of `OPTIONAL` in a SPARQL query?
A: `OPTIONAL` lets a query include extra triple patterns that bind variables when present but do not eliminate a solution when absent, the graph analogue of a SQL left outer join. For instance you might `SELECT` every entity and `OPTIONAL`-ly its ultimate parent, so entities with no recorded parent still appear with the parent variable unbound. Without `OPTIONAL`, any missing edge would drop the whole row.

Q: What do `DISTINCT`, `ORDER BY`, and `LIMIT` add to a SPARQL `SELECT`?
A: `DISTINCT` removes duplicate solution rows, which is common after transitive paths revisit nodes. `ORDER BY` sorts results by one or more expressions, ascending by default or `DESC(...)` for descending. `LIMIT` caps the number of returned rows, essential when probing a large GLEIF graph so an exploratory query does not stream millions of bindings.

Q: How do `GROUP BY` and aggregate functions like `COUNT` work in SPARQL?
A: `GROUP BY` partitions solutions by the grouping expression, and aggregates such as `COUNT`, `SUM`, `MIN`, `MAX`, and `AVG` compute a value per group. For example grouping by `?ultimate` and counting children tells you how many subsidiaries sit under each ultimate parent. You reference aggregates in the `SELECT` list, typically aliased with `(COUNT(?child) AS ?n)`.

Q: What is SHACL and what problem does it solve that RDFS and OWL do not?
A: SHACL is the Shapes Constraint Language, a W3C standard for validating that an RDF graph conforms to expected structure. It solves the closed-world validation problem: RDFS and OWL use their axioms for inference under an open-world assumption, so they never reject "bad" data, whereas SHACL actively checks constraints and reports violations. It is therefore the graph analogue of data-quality rules over a relational table.

Q: In SHACL, what is the difference between an `sh:NodeShape` and an `sh:PropertyShape`?
A: An `sh:NodeShape` defines constraints that apply to a focus node as a whole, such as which class it must belong to or which property shapes it must satisfy. An `sh:PropertyShape` defines constraints on the values reachable by a specific property path from the focus node. Typically a node shape contains several property shapes via `sh:property`, each governing one attribute.

Q: What does `sh:targetClass` do in a SHACL shape?
A: `sh:targetClass` declares which class of nodes a shape applies to, so the validator selects every instance of that class as a focus node to check. For example a shape with `sh:targetClass fibo-be:LegalEntity` validates every legal entity in the graph. Other targeting options include `sh:targetNode`, `sh:targetSubjectsOf`, and `sh:targetObjectsOf` for finer selection.

Q: In a SHACL property shape, what is the role of `sh:path`?
A: `sh:path` specifies which property (or property path) of the focus node the property shape constrains, and it is mandatory in every property shape. For instance `sh:path fibo-fnd:hasLegalName` aims the constraints at the legal-name values. Like SPARQL, the path can be a sequence or inverse, so you can constrain values several hops away from the focus node.

Q: Name three SHACL cardinality and value constraints and what each enforces.
A: `sh:minCount` enforces a minimum number of values for the property (for example at least one LEI), and `sh:maxCount` enforces a maximum. `sh:datatype` requires values to be of a given datatype such as `xsd:date`, and `sh:nodeKind` requires them to be an IRI, blank node, or literal. `sh:pattern` enforces a regular-expression match, ideal for an ISIN or LEI format check.

Q: What is in a SHACL validation report and what does `sh:conforms` tell you?
A: Running a SHACL validation produces a validation report, itself an RDF graph, with a top-level `sh:conforms` boolean and a list of `sh:ValidationResult` nodes for each violation. `sh:conforms true` means the data graph satisfied every shape; `false` means at least one result is present. Each result carries the focus node, the failing path, the offending value, and a severity, so you can route fixes precisely.

Q: How would you express "every fund must have exactly one LEI in valid format" as SHACL constraints?
A: You target the fund class with `sh:targetClass`, then add an `sh:property` shape whose `sh:path` is the LEI property. On that property shape you set `sh:minCount 1` and `sh:maxCount 1` for "exactly one", `sh:datatype xsd:string`, and `sh:pattern "^[A-Z0-9]{18}[0-9]{2}$"` to enforce the 20-character LEI structure. The validator then flags any fund missing an LEI or carrying a malformed one.

Q: What is the `sh:severity` property and what are its standard values?
A: `sh:severity` declares how serious a violation of a constraint is, letting you distinguish blocking errors from advisory warnings. The standard severities are `sh:Violation` (the default), `sh:Warning`, and `sh:Info`. In a fund pipeline you might make a missing LEI a `sh:Violation` that stops a load, while a missing optional alias is only a `sh:Warning`.

Q: Why is SHACL described as "the graph analogue of DQ rules"?
A: Because data-quality rules in a relational world assert facts like "NAV must be non-null and positive" and reject rows that fail, SHACL does the same over a graph by declaring shapes that nodes must satisfy. Both turn business expectations into machine-checked constraints with a structured report of what failed and why. This lets the same governance discipline you apply to tables extend to RDF master data.

Q: What does the SHACL `sh:class` constraint enforce, and how does it differ from `sh:datatype`?
A: `sh:class` requires that each value of the property be an instance (via `rdf:type`, including inferred types) of a given class — for example that a parent value is a `LegalEntity`. `sh:datatype` instead requires literal values to have a specific datatype like `xsd:decimal`. So `sh:class` governs resource-valued properties and `sh:datatype` governs literal-valued ones.

Q: In SHACL, what is the difference between `sh:not`, `sh:and`, `sh:or`, and `sh:xone`?
A: These are logical constraint components combining other shapes: `sh:and` requires all listed shapes to hold, `sh:or` requires at least one, `sh:not` requires a shape to fail, and `sh:xone` requires exactly one of the listed shapes to hold. They let you build conditional validation, such as "an entity has either a parent or exactly one reporting exception, not both". They mirror boolean logic over shapes rather than over data values.

Q: What does FIBO stand for and who maintains it?
A: FIBO is the Financial Industry Business Ontology, the global standard ontology for financial-services concepts. It is hosted and sponsored by the EDM Council (Enterprise Data Management Council) and standardized through the Object Management Group (OMG). Since January 2020 it has been developed via an open community process on GitHub.

Q: In what language is FIBO expressed and why does that choice matter?
A: FIBO is expressed in OWL (the Web Ontology Language), which is based on description logic. This choice matters because it makes the concepts both human-readable and machine-processable, so a reasoner can infer relationships and a tool can validate usage. It also means FIBO slots directly into the RDF/SPARQL/SHACL stack rather than being a prose glossary.

Q: What is the pragmatic stance the lesson takes toward FIBO — vocabulary source or deployment mandate?
A: The lesson frames FIBO as a vocabulary source, not a deployment mandate: you mine it for shared, precise definitions of securities, funds, and legal entities to harmonise your own data. It explicitly warns against over-investing in a triple store where a relational hub would do. So the architect's job is judgement — borrow FIBO's terms and relationships, but enforce them in whatever store fits the estate.

Q: What are FIBO's main domains, and what do the abbreviations FND, BE, and SEC mean?
A: FIBO is organized into domains, each a top-level module. FND is Foundations (core reusable concepts), BE is Business Entities (legal entities, ownership, control), and SEC is Securities (the instruments themselves). These give you a structured map so you can locate the right concept rather than searching a flat glossary.

Q: What do the FIBO domains FBC, DER, IND, and LOAN cover?
A: FBC is Financial Business and Commerce (financial instruments, markets, and products at large), DER is Derivatives, IND is Indices and Indicators, and LOAN is Loans. Together with FND, BE, and SEC they partition the financial universe so that, for example, you map a fund to BE and SEC concepts and a swap to DER. Knowing the domain map is what makes FIBO navigable in practice.

Q: Which FIBO domain would you consult to model "who controls whom" for a management company, and why?
A: You consult BE (Business Entities), because that domain models legal entities, their formation, and crucially their ownership and control relationships. It defines concepts for parties, holdings, and control that align with the kind of corporate-structure questions GLEIF Level-2 also answers. SEC and FBC by contrast describe the instruments the entity issues or holds, not the control structure.

Q: Map the security-master attribute "ISIN" conceptually to FIBO — what kind of FIBO concept is it?
A: An ISIN maps to FIBO's notion of a security identifier — an identifier assigned to a financial instrument, modelled in the securities and foundations identification concepts. FIBO distinguishes the instrument itself from the identifier that designates it, so the ISIN is a property/identifier of the instrument resource rather than the instrument. This separation is exactly why FIBO helps harmonise: your column "ISIN" becomes a typed identifier linked to a well-defined instrument class.

Q: Map the attribute "legal name of the fund" to FIBO and explain the modelling distinction it makes.
A: The legal name maps to a FIBO legal-name concept attached to a legal entity (a fund being a legal entity or arrangement in BE). FIBO distinguishes a legal entity from its names and from the identifiers (like LEI) that designate it, treating name as one of potentially several names a party may bear. Borrowing this distinction stops your model from conflating "the fund", "its registered name", and "its LEI" into one ambiguous field.

Q: How would mapping five security-master attributes to FIBO terms benefit a fund administrator in practice?
A: It gives those attributes precise, industry-consensus definitions, so when you exchange or reconcile data with a custodian or regulator everyone means the same thing by "issuer", "maturity", or "instrument type". That harmonisation reduces reconciliation breaks and ambiguity in EMT/EPT or regulatory feeds. The benefit is semantic clarity and interoperability, even if you never store a single triple.

Q: When is FIBO "academic for your estate", per the lesson's done-when criterion?
A: FIBO is academic when your data needs are well served by a relational hub with clear column definitions and you would gain no real interoperability from triples. If you have few external semantic-exchange partners, no reasoning requirement, and no entity-graph queries, the overhead of OWL/SPARQL tooling outweighs the benefit. The architect must be able to say in writing exactly where FIBO earns its keep and where it does not.

Q: What is a property graph and how does its data model differ from RDF triples?
A: A property graph models data as nodes and relationships (edges) where both nodes and edges can carry key-value properties directly. This differs from RDF, where everything — including what would be edge attributes — must itself be expressed as triples, often via an intermediate node. So an ownership edge can hold the percentage as an edge property in a property graph, whereas plain RDF needs reification or an intermediate relationship node.

Q: What is Cypher and which kind of graph does it query?
A: Cypher is the declarative query language created for Neo4j and used by property-graph databases. It queries property graphs using an ASCII-art pattern syntax, for example `(a)-[:OWNS]->(b)`, to match nodes and relationships. It is the property-graph counterpart to SPARQL, which queries RDF triple stores.

Q: When does Cypher/Neo4j tend to beat RDF triples?
A: A property graph tends to win when relationships carry rich attributes, when you need fast variable-length traversal and pathfinding, and when a single application owns the graph rather than exchanging it with the world. The intuitive node-and-edge model and edge properties make ownership-percentage or transaction-network analytics ergonomic. The trade-off is weaker standardisation and reasoning compared with the W3C RDF stack.

Q: When does RDF tend to beat a property graph?
A: RDF wins when you need open, standardized data exchange across organizations, global identifiers, formal semantics and reasoning, or alignment to a shared ontology like FIBO. Its W3C standards (RDF, SPARQL, OWL, SHACL) and IRI-based merging make federated, vendor-neutral integration natural. The trade-off is that modelling edge attributes is clunkier and tooling can feel heavier.

Q: How does a property graph typically handle an edge attribute like ownership percentage versus how RDF handles it?
A: A property graph stores the percentage directly as a property on the `OWNS` relationship, so it is a first-class attribute of the edge. RDF, having no native edge attributes, must reify the statement or introduce an intermediate "ownership" node that links owner, owned, and percentage with separate triples. RDF-star (RDF 1.2) is an emerging answer that lets you annotate a triple, narrowing this gap.

Q: For a one-off internal "who-owns-whom" analytics tool with weighted edges, which graph model is the more pragmatic default and why?
A: A property graph such as Neo4j is the more pragmatic default because the weighted ownership edges map directly onto edge properties and Cypher makes variable-length ownership-chain traversal concise. There is no external exchange requirement that would justify the heavier RDF/OWL standards stack. The architect should still note that if regulatory data exchange later appears, RDF and GLEIF's published triples become attractive.

Q: What is GLEIF and what is the LEI?
A: GLEIF is the Global Legal Entity Identifier Foundation, the body that oversees the global LEI system. The LEI (Legal Entity Identifier) is a 20-character alphanumeric code uniquely identifying a legally distinct entity that participates in financial transactions. It underpins counterparty identification in regulatory reporting and is the join key for GLEIF's open entity data.

Q: What is the structure of a 20-character LEI?
A: An LEI is 20 characters: characters 1–4 are a prefix assigned to the issuing LOU (Local Operating Unit), characters 5–6 are reserved as zeros, characters 7–18 are the entity-specific part, and characters 19–20 are check digits. The check digits follow an ISO 7064 MOD 97-10 scheme, so a transposition or typo is detectable. This fixed structure is what a SHACL `sh:pattern` or a validation routine checks.

Q: What is the distinction between GLEIF "Level 1" and "Level 2" data?
A: Level 1 data answers "who is who" — the reference data about each entity such as its legal name, address, and registration. Level 2 data answers "who owns whom" — the parent ownership relationships between entities. The lesson uses Level 2 specifically because it is the ready-made entity graph of corporate structure.

Q: What two parent relationships does GLEIF Level 2 record?
A: Level 2 records the direct accounting consolidation parent — the immediate parent that consolidates the entity in its financial statements — and the ultimate accounting consolidation parent, the top of that consolidation chain. Both are based on accounting consolidation principles rather than simple share ownership. Together they let you walk from a subsidiary to its ultimate parent.

Q: Why does GLEIF base Level-2 relationships on "accounting consolidation" rather than raw share ownership?
A: GLEIF uses accounting consolidation because it is a well-defined, auditable concept already reported in financial statements under accounting standards, giving a consistent global basis. Raw share ownership percentages are harder to source uniformly and may not reflect actual control. The consequence to remember is that a Level-2 "parent" means "consolidates this entity", which can differ from majority shareholder.

Q: What are GLEIF's Level-2 reporting exceptions and why do they exist?
A: When an entity cannot report a parent, it records a reporting exception with a reason; the three common ones are NON_PUBLIC (the parent exists but disclosure is legally or otherwise restricted), NON_CONSOLIDATING (no parent prepares consolidated accounts including it), and NO_KNOWN_PERSON (there is no parent). They exist so that absence of a parent edge is explained rather than ambiguous. When querying, treat an exception as "no edge, here is why" rather than missing data.

Q: How does GLEIF Level-2 data connect two entities structurally?
A: Each ownership relationship is a Relationship Record (RR) that links a start node (the child) to an end node (the parent), both identified by their LEIs. Because the parent's LEI is itself a Level-1 entity, the records chain together into a graph where you can traverse from any entity to its ancestors. The LEI is the shared join key that makes the whole network coherent.

Q: Why is GLEIF Level-2 described as a "ready-made entity graph"?
A: Because GLEIF already publishes the nodes (entities by LEI) and the edges (parent relationships) as open, standardized, high-quality data, you do not have to build the ownership graph yourself. You can load it into a triple store or property graph and immediately query ownership chains. This is why the lesson's lab loads 50 entities' Level-2 data and walks their parent chains rather than scraping registries.

Q: In the lab, after loading GLEIF Level-2 for 50 entities into Fuseki, what does "walk the ultimate-parent chains" actually require?
A: It requires a SPARQL query that follows the direct-parent predicate transitively (or reads the published ultimate-parent edge) from each loaded entity up to the top of its ownership chain. Using a property path with `+` you return every ancestor; selecting the node with no further parent yields the ultimate parent. The done-when criterion is returning these ultimate-parent chains for the loaded entities.

Q: What is Apache Jena and what is its role in this lesson's toolchain?
A: Apache Jena is an open-source Java framework for building Semantic Web and linked-data applications, providing the RDF model API, SPARQL processing (ARQ), and reasoning support. In this lesson it is the FOSS stack for hands-on work, giving you the API to create and read triples and to run SPARQL. It pairs with Fuseki for a server endpoint.

Q: What is Apache Jena Fuseki and what does it provide?
A: Fuseki is Jena's SPARQL server: a triple store with an HTTP SPARQL endpoint that you can query over the network and a web UI for managing datasets. It is where you load the GLEIF Level-2 data and run your ownership-chain queries in this lab. It plays the role that Stardog, Ontotext GraphDB, or Amazon Neptune would in a commercial setting.

Q: Name two alternatives to Jena/Fuseki for an RDF triple store.
A: Stardog and Ontotext GraphDB are two commercial RDF triple stores, and Amazon Neptune is a managed cloud option that supports both RDF and property-graph models. They offer features like scalability, security, and managed operations beyond what a Fuseki proof-of-concept provides. The lesson positions Fuseki as the hands-on FOSS choice with these as the production-grade equivalents.

Q: What HTTP detail matters when sending a SPARQL query to a Fuseki endpoint?
A: A Fuseki dataset exposes a `/query` endpoint that accepts the SPARQL query as an HTTP `GET` parameter or `POST` body, and you select the response format via an `Accept` header such as `application/sparql-results+json`. Updates go to a separate `/update` endpoint, keeping read and write distinct. Pointing a query at the update endpoint or omitting the right `Accept` header is a common first-time mistake.

Q: What does it mean that RDF follows the "open-world assumption" and why does it matter for validation?
A: The open-world assumption means a fact not present in the graph is treated as unknown, not false — the graph is presumed incomplete. It matters because OWL reasoning will never conclude "this entity has no parent" merely from absence, so you cannot rely on inference to flag missing data. That is precisely why you reach for SHACL, which closes the world for validation and can assert "a required value is missing".

Q: How does the open-world assumption interact with GLEIF reporting exceptions?
A: Under the open-world assumption, an entity simply lacking a parent edge means "parent unknown", which is ambiguous. GLEIF's reporting exceptions repair this by explicitly stating why no parent is reported (NON_PUBLIC, NON_CONSOLIDATING, NO_KNOWN_PERSON), turning silence into an asserted fact. So when you query, an entity with a NON_CONSOLIDATING exception is genuinely top-of-chain, not just missing data.

Q: Scenario: your SPARQL query returns zero rows although you can see the triples in the Fuseki UI. What is the first thing to check?
A: First check your `PREFIX` declarations and the exact IRIs in your triple patterns, because a mismatched namespace makes every pattern fail silently and return nothing. Confirm the predicate IRI in your query character-for-character against one real triple in the data. Other frequent culprits are querying the wrong dataset or the default graph when the data is actually in a named graph.

Q: Scenario: a transitive property-path query for ancestors hangs or returns duplicate ancestors. What is happening and how do you address it?
A: A `+` path computes the full transitive closure, so on a densely connected or cyclic graph it can revisit nodes and explode in cost, and shared ancestors appear multiple times via different routes. Add `DISTINCT` to collapse duplicates and `LIMIT` while exploring, and ensure the data has no spurious ownership cycles. If cycles are real, the closure may not terminate cleanly and you must break them or bound the depth.

Q: Scenario: SHACL validation reports `sh:conforms false` for your fund data but you believe the data is correct. What do you inspect first?
A: Inspect the individual `sh:ValidationResult` nodes in the report, which name the failing focus node, the `sh:path`, the offending value, and the constraint component. This tells you whether the data is wrong or the shape itself is too strict — for example a `sh:pattern` that rejects a legitimately formatted LEI. Often the shape, not the data, needs correcting, so read the result before changing source records.

Q: Scenario: you mapped your "issuer" column to a FIBO term but reconciliations still break. What FIBO modelling distinction might you have missed?
A: You may have conflated the issuer legal entity with one of its identifiers or with the instrument it issues, distinctions FIBO keeps separate. The "issuer" is a legal entity (BE domain) related to a security (SEC), not the security's identifier or name. Aligning to the precise FIBO relationship — entity issues instrument — rather than a vague label is what removes the ambiguity causing the breaks.

Q: Why might an architect choose a relational hub over a triple store even after acknowledging FIBO's value?
A: Because FIBO's value is its shared vocabulary and entity-relationship definitions, which you can adopt as column semantics and reference data without paying the operational cost of OWL reasoning and SPARQL tooling. A relational hub is simpler to staff, tune, secure, and integrate with existing fund systems. The lesson's whole point is borrowing FIBO's semantics while deploying in whatever store fits the estate.

Q: What is reification in RDF and why does it come up when modelling ownership percentages?
A: Reification is making a statement about a statement by treating a triple as a resource with its own subject, predicate, and object properties. It comes up for ownership percentages because plain RDF cannot attach a percentage to the ownership edge directly, so you reify or introduce an intermediate node to hang the percentage and effective date on. It is verbose, which is one reason property graphs feel more natural for weighted edges.

Q: What is RDF-star (RDF 1.2) and how does it relate to the edge-attribute problem?
A: RDF-star (standardized as part of RDF 1.2) lets you use a triple itself as the subject or object of another triple, so you can annotate a statement without full reification. This narrows the gap with property graphs by making it concise to say "this ownership relationship has percentage 51 as of this date". It directly addresses the historical awkwardness of attaching attributes to RDF edges.

Q: What does SPARQL federation (`SERVICE`) allow, and when is it useful in entity data?
A: The `SERVICE` keyword lets one SPARQL query delegate part of its pattern to a remote endpoint and join the results locally. It is useful when you want to enrich your local fund data with a remote authoritative graph — for example joining your holdings to GLEIF entity data without copying it. The trade-off is that remote latency and availability become part of your query's reliability.

Q: How would you validate that every loaded GLEIF entity has either a parent or a reporting exception, expressed conceptually in SHACL?
A: You define a node shape targeting the entity class with a constraint requiring at least one of a direct-parent value or a reporting-exception value, for example via `sh:or` over two property shapes each with `sh:minCount 1`. The validation report then flags any entity that has neither, which would indicate incomplete Level-2 loading. This turns "every entity is accounted for" into a machine-checked rule.

Q: Why can RDFS `rdfs:domain` never be used to reject data, and what is the practical consequence?
A: Because `rdfs:domain` is an inference rule: asserting it causes a reasoner to infer the subject's class rather than to test it, so under the open-world assumption no contradiction arises from "wrong" usage. The practical consequence is that you must not treat RDFS as validation — using a property on the "wrong" subject quietly broadens that subject's inferred types instead of erroring. For genuine rejection you use SHACL constraints.

Q: What is an ontology, in one precise sentence, and how does FIBO fit that definition?
A: An ontology is a formal, machine-readable specification of the concepts in a domain and the relationships between them. FIBO fits this exactly: it formally specifies financial-services concepts — legal entities, securities, funds, derivatives — and their relationships in OWL. That formality is what lets tools reason over it and what makes it a candidate shared vocabulary rather than a prose glossary.

Q: How does a class hierarchy in FIBO help harmonise entity data across systems?
A: A class hierarchy (via `rdfs:subClassOf`) lets specific local types map to general shared concepts, so disparate systems that call something a "SICAV", "FCP", or "unit trust" can all align under a common fund or legal-entity superclass. Queries and rules written against the superclass then cover all subtypes automatically. This subsumption is a core mechanism by which FIBO reduces fragmentation in entity data.

Q: Why is a SPARQL property path often preferable to relying on OWL transitive-property inference for ownership chains?
A: A property path computes the traversal at query time with explicit, predictable semantics and works even on a plain store with no reasoner enabled, whereas `owl:TransitiveProperty` requires the endpoint to materialize or reason over the closure. Paths also let you mix predicates and directions in one expression. The practical upshot is that paths are portable and explicit, which is why the lesson's walk-the-chain task uses them.

Q: How would you confirm that a published GLEIF ultimate parent matches the top of the direct-parent chain you computed?
A: You compute the chain with a transitive path on the direct-parent predicate, take the topmost ancestor (the one with no further direct parent), and compare it to the entity's published `hasUltimateAccountingConsolidationParent` value. If they differ, you have either incomplete data, a reporting exception breaking the chain, or a cycle. This cross-check is a good data-quality test of the loaded Level-2 graph.

Q: What is a namespace prefix collision and how can it cause silent query errors against FIBO data?
A: A prefix collision is when the same short label (say `fibo:`) is bound to different IRIs in different files or queries, or when you copy a prefix whose IRI does not exactly match the data's IRIs. It causes silent zero-result queries because the expanded predicate IRI no longer matches any triple. The fix is to copy prefix declarations directly from the data source and verify expansions character-for-character.

Q: What practical advantage does loading GLEIF data as N-Triples give when you only need 50 entities but the full file is huge?
A: Because N-Triples is line-oriented with one triple per line, you can filter the enormous published dump to just the lines mentioning your 50 LEIs using simple line tools before loading, keeping Fuseki small and queries fast. There is no need to parse the whole graph to extract a subset. This streaming-friendliness is exactly why bulk RDF distributions favour N-Triples.

Q: Why might you choose `CONSTRUCT` over `SELECT` when extracting ownership chains for reuse?
A: `CONSTRUCT` returns a new RDF graph rather than a table, so you can materialize a simplified "entity hasAncestor entity" graph that downstream tools load directly as triples. `SELECT` would give you rows you then have to reassemble into a graph. For building a derived ownership sub-graph to share or visualize, `CONSTRUCT` is the natural fit.

Q: What is the difference between an OWL "class expression" and a simple named class, with an entity-data example?
A: A named class is just an IRI standing for a category, whereas a class expression defines a class by conditions, such as "legal entities that have at least one ultimate parent". OWL builds these with restrictions like `owl:Restriction` over a property and cardinality. This lets you define, for example, the class of entities that are intermediate holding companies purely from their relationships rather than by explicit tagging.

Q: How does the choice between RDF and a property graph affect a regulatory data-exchange use case like reporting fund ownership to a supervisor?
A: For regulatory exchange, RDF's standardisation, global LEIs, and alignment to a shared ontology like FIBO make the data portable and unambiguous across institution and regulator, which is often the deciding factor. A property graph's internal ergonomics matter less when the goal is interoperable submission. So the architect leans RDF for cross-boundary regulatory reporting and may still keep a property graph internally for analytics.

Q: What does it mean to say SHACL "closes the world" for the scope of validation, and why is that useful for DORA-style controls?
A: It means that within a shape's scope SHACL treats absent required values as violations rather than as unknown, giving definite pass/fail results suitable for controls and evidence. For DORA-style operational-resilience and data-integrity controls you need provable, repeatable checks with an auditable report — which the SHACL validation report provides. This turns "the data must be complete and well-formed" into a testable, evidenced gate.

Q: How would you express "an entity must not be its own ancestor" (no ownership cycle) as a check, and which technology fits best?
A: This is a graph-reachability constraint, best expressed in SPARQL using a transitive path with a self-match test, for example asking whether `?e gleif-L2:hasDirectAccountingConsolidationParent+ ?e` returns any binding; any result is a cycle. SHACL alone is awkward for transitive reachability, so you pair it with a SPARQL-based constraint or a separate SPARQL `ASK`. An `ASK` returning true means a cycle exists and the ownership data is inconsistent.

Q: How does GLEIF Level-2 data complement FIBO's BE domain in modelling corporate structure?
A: FIBO's BE domain supplies the vocabulary and class definitions for legal entities, ownership, and control, while GLEIF Level-2 supplies the actual instance data — concrete entities and their parent relationships keyed by LEI. Together you get a typed schema (FIBO) populated with real, open ownership facts (GLEIF). This pairing is the practical recipe for a semantic entity graph that is both well-defined and populated.

Q: Why does the LEI make a good shared join key across RDF, property graphs, and relational systems alike?
A: Because the LEI is a single, globally unique, externally governed identifier for a legal entity, every system that adopts it can join on the same key without bilateral mapping tables. In RDF it becomes (part of) the node IRI, in a property graph a node property, and in a relational hub a primary or foreign key. This shared anchor is what lets a fund administrator reconcile counterparty data across heterogeneous stores.

Q: Pulling it together: state in one or two sentences where FIBO and the RDF stack genuinely help versus where they are overkill for a fund administrator.
A: They genuinely help where you need a shared, precise vocabulary for securities and legal entities and where entity relationships ("who owns whom") drive cross-organization or regulatory exchange — there FIBO's definitions and GLEIF's published triples harmonise data and enable graph queries. They are overkill where a well-defined relational hub already serves your needs, you have no semantic-exchange partners, and no reasoning or graph-traversal requirement justifies the OWL/SPARQL operational cost.
