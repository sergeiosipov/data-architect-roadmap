# Phase 5 · 9.10.2 Cloud-native IaC (Bicep) — 150 self-test questions

Q: What is Bicep, in one sentence?
A: Bicep is a domain-specific declarative language for deploying Azure resources, designed as a cleaner, more readable abstraction over Azure Resource Manager (ARM) JSON templates. You author resources in Bicep, and the Bicep CLI transpiles them to ARM JSON, which is what Azure Resource Manager actually consumes. It is Azure-native and maintained by Microsoft, with day-0 support for new resource types and API versions.

Q: What does Bicep compile (transpile) to, and why does that matter?
A: Bicep compiles to standard ARM JSON templates — the intermediate language that Azure Resource Manager executes. This matters because anything ARM can do, Bicep can do, and any ARM template can be decompiled toward Bicep; the runtime engine, deployment model, and REST API are identical. You can run `az bicep build --file main.bicep` to see the generated `main.json` and confirm what will actually be sent to Azure.

Q: Is Bicep imperative or declarative, and what does that imply about element order in a file?
A: Bicep is declarative: you describe the desired end state of resources, not a sequence of steps. Because of this, elements like `param`, `var`, `resource`, `module`, and `output` can appear in any order — the order in the file does not affect how the deployment is processed. Resource Manager resolves dependencies from references rather than from textual position.

Q: What is `targetScope` in a Bicep file and what is its default value?
A: `targetScope` declares the deployment scope of the whole file and accepts `resourceGroup`, `subscription`, `managementGroup`, or `tenant`. The default is `resourceGroup`, so for a resource-group deployment you can omit it entirely. You set a non-default scope at the top of the file, for example `targetScope = 'subscription'` when the file creates resource groups.

Q: In a Bicep `resource` declaration, what are the three required pieces of the header line?
A: A resource header is `resource <symbolic-name> '<resource-type>@<api-version>' = { ... }`, so the three pieces are the symbolic name, the fully qualified resource type, and the API version. For example `resource stg 'Microsoft.Storage/storageAccounts@2025-06-01' = { ... }`. The symbolic name is a Bicep-only identifier used to reference the resource elsewhere in the file; it is not the Azure resource name.

Q: What is the difference between a resource's symbolic name and its `name` property?
A: The symbolic name (the identifier right after the `resource` keyword) is a compile-time handle you use inside the Bicep file to reference the resource, such as reading `stg.properties.primaryEndpoints`. The `name` property is the actual name the resource gets in Azure, like `mystorageacct001`. They are independent: the symbolic name never appears in Azure, and the `name` value is what shows up in the portal.

Q: How do you declare a parameter in Bicep, and how do you give it a default?
A: Use `param <name> <type> = <default>`, for example `param storageSKU string = 'Standard_LRS'`. If you supply a default value, the parameter becomes optional at deployment time; omit the `= <default>` to make it required. Parameters externalize values that vary between deployments (dev versus prod, region, naming prefix).

Q: What is the purpose of a `var` (variable) in Bicep, and how does it differ from a `param`?
A: A `var` holds a value computed inside the file, typically to encapsulate a complex expression and keep the template readable, for example `var uniqueStorageName = '${storagePrefix}${uniqueString(resourceGroup().id)}'`. Unlike a `param`, a variable cannot be supplied or overridden at deployment time — its value is fixed by the file logic. You cannot apply the `@secure()` decorator to a variable.

Q: What does an `output` statement do in Bicep?
A: An `output` returns a value from a deployment so it can be consumed afterward, by a calling module or by a CLI/PowerShell command, for example `output storageEndpoint object = stg.properties.primaryEndpoints`. Outputs are how you surface generated names, endpoints, or resource IDs without hard-coding them. From the parent file you read a module's output via the module's symbolic name and its `outputs` property.

Q: How do you reference an output produced by a module from the parent Bicep file?
A: Use the module's symbolic name followed by `.outputs.<outputName>`, for example `output storageEndpoint object = stgModule.outputs.storageEndpoint`. The parent file can then pass that value into another resource or re-export it as its own output. This is the primary mechanism for wiring modules together — one module emits an ID or connection string, the next consumes it.

Q: What is a Bicep module, fundamentally?
A: A Bicep module is simply a Bicep file that another Bicep file deploys, used to encapsulate and reuse a unit of infrastructure. You declare it with `module <symbolic-name> '<path-to-file>' = { name: '...', params: { ... } }`. Under the hood, modules are compiled into nested deployments within a single ARM template, so each module becomes a `Microsoft.Resources/deployments` resource.

Q: What does the `name` property on a module declaration control, and is it required?
A: The module's `name` becomes the name of the nested deployment resource in the generated ARM template; it is optional. If you omit it, a GUID is generated automatically, which the `no-module-name` linter rule actually encourages. A static module name reused concurrently against the same scope can cause one deployment to clobber another's outputs, so unique or auto-generated names are safer for parallel runs.

Q: How do you point a module at a local file, and what path separator must you use?
A: Provide a relative path string, for example `module stg '../storageAccount.bicep' = { ... }`. You must always use the forward slash `/` as the directory separator in Bicep paths, even on Windows — the backslash `\` is not supported, which keeps templates portable across Linux, macOS, and Windows build agents. Paths may contain spaces.

Q: Can a Bicep module reference an ARM JSON template instead of a `.bicep` file?
A: Yes. A module path can point to a local ARM JSON template, for example `module stg '../storageAccount.json' = { ... }`, and Bicep treats it the same way. This lets you incrementally adopt Bicep over an existing ARM JSON estate or reuse vendor-supplied JSON without rewriting it first.

Q: How do you deploy a module to a scope different from the rest of the file?
A: Add a `scope` property to the module declaration set to a valid scope object, for example `scope: newRG` where `newRG` is a resource-group resource, or `scope: resourceGroup('demoRG')`. A common pattern is a `targetScope = 'subscription'` file that creates a resource group and then deploys a storage module scoped into it. Without `scope`, the module deploys at the parent file's target scope.

Q: Which scope functions can you use to set a module's `scope`?
A: `resourceGroup()`, `subscription()`, `managementGroup()`, and `tenant()`. For example `scope: managementGroup(managementGroupName)` deploys the module at a management-group scope. You can also pass a symbolic name of a resource-group, subscription, or management-group resource declared in the same file.

Q: How do you conditionally deploy a resource or module in Bicep?
A: Append an `if (<condition>)` expression to the declaration, for example `resource dnsZone 'Microsoft.Network/dnsZones@2023-07-01-preview' = if (deployZone) { ... }`. When the condition evaluates to false, the resource or module is skipped. This lets one file serve multiple environments — for instance only deploying a DR storage account when a `deployDr` boolean is true.

Q: How do you deploy multiple instances of a resource or module with a loop?
A: Use a `for` expression, for example `module stg './example.bicep' = [for i in range(0, moduleCount): { name: '${i}deploy', params: {} }]`. You can iterate over an array, an object, or an integer index produced by `range()`. This is how you create N identical subnets, storage accounts, or per-fund resource groups from a single declaration.

Q: What does the `@batchSize()` decorator do on a looped resource or module?
A: By default the deployment engine deploys looped instances in parallel; `@batchSize(int)` forces them to deploy serially in batches of the given size. For example `@batchSize(3)` deploys three at a time. You use it when parallel creation would hit a throttling limit or a dependency that the engine cannot infer, such as resources that must be provisioned in a controlled sequence.

Q: What is a decorator in Bicep and how is it written?
A: A decorator is an annotation written as `@expression(...)` placed on the line above a `param`, `var`, `resource`, `module`, `output`, `func`, or `type` to attach metadata or constraints. Examples include `@description('...')`, `@minLength(3)`, `@allowed([...])`, and `@secure()`. Decorators live in the `sys` namespace, so if you have a parameter named `description` you must disambiguate with `sys.description`.

Q: What does the `@secure()` decorator do, and where can it be applied?
A: `@secure()` marks a string or object parameter (and, from Bicep 0.35.1, a module output) as sensitive so its value is not written to the deployment history or logs. You use it for passwords, connection strings, and API keys — for example `@secure() param adminPassword string`. The what-if operation also refuses to evaluate secure parameter values, treating them as opaque.

Q: What does the `@allowed()` decorator enforce, and on which element only?
A: `@allowed()` restricts a parameter to a fixed set of permitted values, for example `@allowed(['Standard_LRS','Standard_GRS']) param storageSKU string`. It is permitted only on `param` statements; to constrain a `type` or `output` to a set of values you use union-type syntax like `'Standard_LRS' | 'Standard_GRS'` instead. It is the simplest guardrail against typos and unsupported SKUs.

Q: How do you write single-line and multi-line comments in Bicep?
A: Use `//` for a single-line comment and `/* ... */` for a multi-line block comment. Comments are stripped during compilation and do not appear in the generated ARM JSON. They are the right place to record an assumption, such as noting that a referenced Key Vault is expected to already exist.

Q: What does the `existing` keyword do in a resource declaration?
A: `existing` declares a reference to a resource that is already deployed rather than creating or managing it, for example `resource demoRG 'Microsoft.Resources/resourceGroups@2025-04-01' existing = { name: 'demogroup1' }`. You use it to read properties (like a location, a Key Vault URI, or a secret) from a resource owned elsewhere without taking over its lifecycle. No body of deployable properties is supplied — only enough to identify it.

Q: How is a child resource declared inside a parent resource in Bicep?
A: You nest the child inside the parent's body using a shortened type segment, for example a `fileServices` resource named `default` declared inside a `Microsoft.Storage/storageAccounts` block. The nested child uses only the trailing type segment, not the full type path. Alternatively you can declare the child outside the parent and link it with the `parent:` property pointing at the parent's symbolic name.

Q: What is the `uniqueString()` function commonly used for in Bicep names?
A: `uniqueString()` produces a deterministic 13-character hash from its seed arguments, used to generate globally unique, repeatable resource names — for example `'${storagePrefix}${uniqueString(resourceGroup().id)}'` for a storage account whose name must be globally unique. Because it is deterministic, the same inputs always yield the same suffix, so re-deployments target the same resource rather than creating a new one.

Q: What is a `bicepconfig.json` file for?
A: `bicepconfig.json` is the per-project Bicep configuration file that controls linter rules, module registry aliases (including overriding `br/public`), experimental feature flags, and other compiler behavior. For example you can define an alias so a long private-registry path becomes `br/ContosoModules:storage:v1`. It is resolved by walking up the directory tree from the file being built and merged with the default config.

Q: What is the `what-if` operation in Azure Resource Manager?
A: `what-if` is a preview operation that predicts the changes a Bicep/ARM deployment would make to existing resources without actually applying anything. It compares the target state described by your template against the current state in Azure and reports creates, modifications, deletions, and no-changes. It is Azure's equivalent of a Terraform plan — a safety gate before you commit.

Q: What is the exact Azure CLI command to run a what-if at resource-group scope?
A: `az deployment group what-if --resource-group <rg> --template-file main.bicep`. This returns a color-coded text summary of predicted changes and makes no modifications. The parallel commands for other scopes are `az deployment sub what-if`, `az deployment mg what-if`, and `az deployment tenant what-if`.

Q: Which scopes does the what-if operation support?
A: What-if is supported for resource group, subscription, management group, and tenant level deployments, mirroring the four `targetScope` values. The CLI verbs are `az deployment group what-if`, `az deployment sub what-if`, `az deployment mg what-if`, and `az deployment tenant what-if` respectively. PowerShell uses the `-WhatIf` switch on the corresponding `New-Az...Deployment` cmdlets.

Q: What does the `--confirm-with-what-if` (or `-c`) flag do on an `az deployment ... create`?
A: It runs a what-if preview first, prints the predicted change set, and then prompts `Are you sure you want to execute the deployment? (y/n)` before actually deploying. For example `az deployment group create --resource-group ExampleGroup -c --template-file main.bicep`. It combines preview and apply in one gated step, which is ideal for interactive runs where you want a human to eyeball changes.

Q: What are the three symbols what-if uses for resource and property changes?
A: `-` denotes Delete, `+` denotes Create, and `~` denotes Modify; the header of every what-if output lists these symbols. In ResourceIdOnly mode a `!` symbol is also used to denote Deploy. The output is additionally color-coded so the change type is visible at a glance.

Q: Name the seven change types the what-if operation can report.
A: Create, Delete, Ignore, NoChange, NoEffect, Modify, and Deploy. Create and Delete are self-explanatory; Modify means the resource exists and properties will change; NoChange means it exists and will be redeployed with no property change; Ignore means it exists but is not in the template and will be left alone; NoEffect means a read-only property will be ignored by the service; Deploy appears only in ResourceIdOnly mode when ARM cannot determine whether properties change.

Q: When does the what-if `Delete` change type actually appear?
A: The `Delete` change type appears only when you deploy a JSON/ARM template in complete mode, and only for resource types that support complete-mode deletion. In the default incremental mode, resources not in the template are reported as `Ignore`, never `Delete`. This is why you should always run what-if before any complete-mode deployment.

Q: What is the difference between the `Ignore` and `Delete` change types in what-if?
A: Both apply to a resource that exists in Azure but is absent from your template. `Ignore` means it will be left untouched (the behavior in incremental mode, or when nested-template expansion limits are hit). `Delete` means it will be removed — which only happens under complete mode for deletion-eligible types. Confusing the two is dangerous: assuming `Ignore` while actually running complete mode can wipe resources.

Q: What does the `NoChange` change type signify, and what does it imply about idempotency?
A: `NoChange` means the resource exists, is defined in the template, and will be redeployed but with no property changes. It demonstrates that ARM deployments are idempotent — re-running the same template converges to the same state without side effects. It is returned when the result format is `FullResourcePayloads`, the default.

Q: What is the difference between the `--result-format` values `FullResourcePayloads` and `ResourceIdOnly`?
A: `FullResourcePayloads` (the default) returns each changing resource plus the specific properties that will change, giving you a detailed diff. `ResourceIdOnly` returns just the list of resource IDs that will change, using the `!` Deploy symbol, which is faster and less noisy when you only care about which resources are touched. You select it with `--result-format ResourceIdOnly` in the CLI.

Q: How do you get machine-readable JSON from a what-if run instead of the pretty text summary?
A: Add `--no-pretty-print`, for example `az deployment group what-if --resource-group ExampleGroup --template-file main.bicep --no-pretty-print`, which returns a JSON object you can parse programmatically in CI. Capturing it into a variable like `results=$(az deployment group what-if ... --no-pretty-print)` lets a pipeline gate on the change set. PowerShell uses `Get-AzResourceGroupDeploymentWhatIfResult` for the object form.

Q: A what-if run reports a property as deleted, but you know it should not change — what is going on?
A: This is what-if "noise": properties that are not specified in the template but are auto-populated to default values during deployment can be incorrectly reported as deleted. The actual deployed resource will still have those defaults set, so no real change occurs. The output even warns that results may contain false-positive predictions, and Microsoft is progressively filtering these out.

Q: Why does what-if always report a change when a property uses the `reference()` function or `utcNow()`?
A: What-if cannot evaluate certain expressions outside a real deployment context — non-deterministic functions like `utcNow()` and `newGuid()`, secure parameter values, `listKeys()`/resource functions, and `reference()` to resources not in the same template. It compares the current concrete value against the unresolved expression, so they never match and it flags a change. The output shows the raw expression (for example `[utcNow()]`) to signal this.

Q: What are the two ARM deployment modes, and what is the default?
A: The two modes are incremental and complete; incremental is the default. The difference is purely how Resource Manager treats resources that exist in the resource group but are not in your template. Incremental leaves them alone; complete deletes them.

Q: In incremental mode, what happens to resources that exist in the group but are not in the template?
A: They are left unchanged. Incremental mode only adds or updates the resources described in the template and never touches anything outside it. This makes incremental the safe default — for example deploying a new monitoring rule into a fund's resource group without risking the storage accounts already there.

Q: In complete mode, what happens to resources that exist in the group but are not in the template?
A: They are deleted. Complete mode treats the template as the full, authoritative definition of the resource group, so any resource not present is removed (subject to type-specific deletion support and locks). This is powerful for drift elimination but dangerous if the template is incomplete.

Q: Given a group with Resource A, B, C and a template with A, B, D, what is the end state in each mode?
A: In incremental mode the group ends up with A, B, C, and D — D is added, C is preserved. In complete mode the group ends up with A, B, and D — C is deleted because it is not in the template. This is the canonical example illustrating why complete mode requires a what-if first.

Q: Does incremental mode preserve unspecified properties on an existing resource that is in the template?
A: No — this is a common and dangerous misconception. When you redeploy a resource in incremental mode, all properties are reapplied as a whole; properties you omit from the template are reset to their defaults, not left as they were. The resource definition always represents the complete final state, so you must specify every non-default value, not just the ones you are changing.

Q: What is the current Microsoft guidance on using complete mode, and what is the recommended alternative?
A: Complete mode is no longer recommended and is being gradually deprecated; the recommended way to perform deletions with Bicep or ARM is deployment stacks. Deployment stacks track a managed set of resources and can delete or detach resources that fall out of the definition in a controlled, auditable way. For most deployments, stick to incremental mode plus stacks for lifecycle deletion.

Q: Can subscription-level and nested/linked deployments use complete mode?
A: No. Subscription-level deployments do not support complete mode, and only root-level templates support it — linked or nested templates must use incremental mode. The Azure portal also does not support complete mode. These restrictions are another reason complete mode is niche and being phased out in favor of deployment stacks.

Q: How do you set complete mode in an Azure CLI deployment?
A: Use the `--mode Complete` flag, for example `az deployment group create --mode Complete --name ExampleDeployment --resource-group ExampleResourceGroup --template-file storage.json`. In PowerShell the equivalent is `-Mode Complete` on `New-AzResourceGroupDeployment`. Omitting the flag defaults to incremental.

Q: Why does Microsoft strongly advise running what-if before any complete-mode deployment?
A: Because complete mode deletes any resource not in the template, a forgotten or mis-scoped resource can be destroyed silently. What-if surfaces exactly which resources will be created, modified, or deleted before anything happens, letting you catch unintended deletions. The deployment-modes documentation explicitly tells you to always run what-if before deploying in complete mode.

Q: In complete mode, how are parent and child resources treated for deletion?
A: Parent resources not present in a complete-mode template are automatically deleted. Some child resources are not automatically deleted on their own, but they are removed if their parent is deleted. For example, deploying complete mode without a DNS zone deletes both the zone and its CNAME records, but keeping the zone while omitting only the CNAME leaves the CNAME intact.

Q: Does complete mode delete resources when the resource group is locked?
A: No. If the resource group has a lock (for example a CanNotDelete or ReadOnly lock), complete mode does not delete the locked resources. The lock takes precedence, which is one defensive control against accidental complete-mode wipes. This is independent of what what-if predicts, since the lock is enforced at deployment time.

Q: When you deploy to multiple resource groups in one template under complete mode, which resources are eligible for deletion?
A: Only resources in the resource group named in the deployment operation (the primary group) are eligible for deletion. Resources in any secondary resource groups targeted by the same template are not deleted. This scoping limits the blast radius of complete mode to a single group per deployment.

Q: What is the headline trade-off between Bicep and Terraform?
A: Bicep is Azure-only but Azure-native, with no state file to manage and day-0 coverage of new Azure features; Terraform (and OpenTofu) is multi-cloud with a vast provider ecosystem but depends on a state file and provider release cadence. The decision usually turns on whether the estate is Azure-only (favoring Bicep) or spans clouds and SaaS providers (favoring Terraform). A good architect can defend either choice for a given org.

Q: Why is "no state file to manage" cited as a Bicep advantage?
A: Bicep has no separate state file because Azure Resource Manager itself is the source of truth — the live resource state lives in ARM, and what-if diffs against it directly. This removes a whole class of operational problems: state-file storage, locking, drift between state and reality, secrets leaking into state, and state corruption. Terraform, by contrast, maintains its own state that must be stored, locked, and reconciled.

Q: What does "day-0 Azure feature support" mean for Bicep, and why does Terraform sometimes lag?
A: Because Bicep targets the ARM REST API directly, a new Azure resource type or API version is usable in Bicep the moment ARM exposes it — "day 0." Terraform's AzureRM provider is a separate codebase that must add and release support for each new feature, so there can be a lag of days to months. For teams that need bleeding-edge Azure capabilities immediately, this is a point in Bicep's favor.

Q: Give two concrete situations where Terraform clearly wins over Bicep.
A: First, multi-cloud or hybrid estates — Terraform/OpenTofu can manage Azure, AWS, GCP, Kubernetes, Cloudflare, and SaaS providers from one tool and one workflow, whereas Bicep is Azure-only. Second, a rich ecosystem — thousands of community providers and modules, mature tooling (Sentinel/OPA policy, Terragrunt, broad CI integrations) that Bicep's Azure-scoped ecosystem does not match. State-aware orchestration across many provider types is also a Terraform strength.

Q: In a Luxembourg fund-tech shop that is Azure-only and Microsoft-aligned, what is the typical reasoned recommendation, and why?
A: Bicep is usually the pragmatic recommendation: an Azure-only estate gains nothing from Terraform's multi-cloud abstraction, avoids state-file operations entirely, gets day-0 Azure features, and aligns with Microsoft support channels the org already pays for. The incumbent deployment code is likely already Bicep/ARM, so reading and extending it is cheaper. You would still flag that this locks you to Azure and reduces portability if the strategy later changes.

Q: Why must an architect hold a defensible Bicep-vs-Terraform position even if they personally prefer Terraform?
A: In a vendor-aligned (Microsoft-only) organization you will be handed Bicep/ARM as the incumbent, and without evaluation fluency you cannot read the existing deployment code or argue the trade-off credibly — you get overruled by stakeholders who already bought into Azure. Recommending the right tool for the org, not your favorite, is the architect's job. Being able to defend either side against its opposing camp with concrete reasons is the competency this lesson targets.

Q: Bicep and Terraform both claim idempotent deployments — how does each achieve it?
A: Bicep relies on ARM: the template declares desired state, and ARM reconciles the live resources to match on each deployment, with what-if previewing the diff against live state. Terraform reconciles the template against its own state file and then against the provider's reported reality, computing a plan. The key operational difference is the source of truth — ARM (live) for Bicep versus the state file for Terraform.

Q: How does drift detection differ between Bicep/ARM and Terraform?
A: With Bicep, drift surfaces by running what-if against the live ARM state — there is no separate state to fall out of sync. With Terraform, you run `terraform plan` (or `terraform refresh`), which can show drift between the state file and reality, and the state file itself can become stale or corrupted. Bicep avoids the "state versus reality" divergence class, while Terraform's state enables features like importing and tracking resources across many providers.

Q: Is there a "plan" concept in Bicep equivalent to `terraform plan`, and is it stored?
A: Yes — `what-if` is Bicep/ARM's plan-equivalent, previewing changes against live Azure state. Unlike `terraform plan -out`, what-if results are not persisted as an applyable artifact by default; you preview then deploy, optionally combining them with `--confirm-with-what-if`. You can capture the JSON via `--no-pretty-print` for CI gating, but there is no saved plan file that the apply step consumes.

Q: What are Azure Verified Modules (AVM)?
A: Azure Verified Modules are Microsoft's official, curated library of prebuilt, pretested, and preverified IaC modules for deploying Azure resources to a common standard. They are an initiative to consolidate and standardize "what a good IaC module looks like" across languages, owned by Microsoft with a devolved ownership model. Using AVM means you consume modules that already follow Azure best practices like the Well-Architected Framework.

Q: Does AVM cover only Bicep, or other IaC languages too?
A: AVM is multi-language: it indexes and publishes modules for both Bicep and Terraform, each in its own language-specific registry with its own specifications. The naming and quality standards are shared across languages, but the registries and reference syntax differ. So an architect comparing Bicep and Terraform can use AVM in either tool.

Q: Who owns and maintains Azure Verified Modules?
A: Microsoft drives AVM as an official initiative, using a devolved ownership approach — Microsoft sets overall governance and standards while distributed Microsoft teams own and maintain individual modules. The modules are created and owned by Microsoft employees, which is part of why non-AVM community modules were retired from the public Bicep registry. This gives AVM a support and quality posture that ad-hoc community modules lack.

Q: What are the two primary AVM module classes, and how do they differ?
A: Resource modules and pattern modules. A resource module is a flexible, multi-purpose building block for a single Azure resource type (plus its child and extension resources), like a storage account. A pattern module composes multiple resources into a reusable architectural pattern, like a hub-and-spoke network or a multi-tier application. AVM also has utility modules that provide shared helpers across both.

Q: What is the AVM naming convention for a Bicep resource module?
A: Resource modules follow `avm/res/<resource-provider>/<resource-type>`, both segments hyphenated, for example `avm/res/storage/storage-account` or `avm/res/compute/virtual-machine`. The `res` segment signals it is a single-resource module. You append a version tag when referencing it, like `avm/res/storage/storage-account:0.18.0`.

Q: What is the AVM naming convention for a Bicep pattern module?
A: Pattern modules follow `avm/ptn/<grouping-or-category>/<pattern-name>`, hyphenated, for example `avm/ptn/3-tier/web-app` or `avm/ptn/avd-lza/management-plane`. The `ptn` segment distinguishes a multi-resource architectural pattern from a single-resource `res` module. As with resource modules, you reference a specific version via a tag.

Q: How do you reference a public AVM module in Bicep, with exact syntax?
A: Use the `br/public:` alias: `module <symbolic-name> 'br/public:<file-path>:<tag>' = { ... }`, for example `module storage 'br/public:avm/res/storage/storage-account:0.18.0' = { name: 'myStorage', params: { name: '...' } }`. The `<tag>` is the module version, and the params block supplies the module's inputs. The `br/public` alias is predefined by the Bicep tooling.

Q: What does the `br/public:` alias expand to under the hood?
A: `br/public` is an alias for the public Bicep module registry hosted at `mcr.microsoft.com/bicep`, so `br/public:avm/res/...` is equivalent to `br:mcr.microsoft.com/bicep/avm/res/...`. The `br` scheme means "Bicep registry," and you can override or redefine the `br/public` alias in `bicepconfig.json`. Both forms resolve to the same Microsoft Container Registry artifacts.

Q: What is the significance of the version tag on an AVM module reference, such as `:0.18.0`?
A: The tag pins the module to a specific published version, giving you reproducible deployments — the same tag always restores the same module code. AVM modules are independently versioned, and the registry shows the latest version with a dropdown for older ones. Pinning a version (rather than floating) is best practice so an upstream module update cannot silently change your infrastructure.

Q: What happens behind the scenes when you reference a registry module in VS Code with the Bicep extension?
A: The Bicep extension automatically runs `bicep restore` to download the external module into the local cache so IntelliSense and compilation work. It takes a few moments, and IntelliSense for the module may lag until the restore completes. You can also restore manually with `az bicep restore` in CI before building.

Q: Why did Microsoft retire non-AVM community modules from the public Bicep registry?
A: To consolidate around a single, Microsoft-owned, verified standard: the public Bicep module registry now hosts Azure Verified Modules only. This raises the baseline quality, support, and consistency of what teams pull from `br/public` and avoids the maintenance and trust problems of unvetted community modules. Private modules can still be published to your own Azure Container Registry.

Q: How does AVM relate to landing zones and the Cloud Adoption Framework?
A: AVM ships pattern modules that implement Microsoft's landing-zone and Cloud Adoption Framework guidance, so you can stand up an Azure landing zone from curated, best-practice modules rather than hand-rolling the platform. Microsoft has released Bicep AVM pattern modules specifically for the platform landing zone. This is why AVM matters for landing-zone work — it is the Microsoft-sanctioned building-block library.

Q: What is the practical benefit of using AVM resource modules instead of authoring raw `resource` blocks?
A: AVM modules bake in best-practice defaults — diagnostic settings, RBAC, private endpoints, secure configuration — and are pretested, so you inherit Well-Architected alignment without rediscovering every property. You configure them through documented parameters rather than learning every nuance of each resource type's API. The trade-off is less low-level control and a dependency on the module's release cadence.

Q: How do you publish your own (private) module to an Azure Container Registry for reuse?
A: Use `az bicep publish` to push the compiled module to an ACR, then reference it as `br:<registry-name>.azurecr.io/<file-path>:<tag>`, for example `br:exampleregistry.azurecr.io/bicep/modules/storage:v1`. The `<file-path>` becomes the ACR repository and the `<tag>` the version. Consumers need pull permission on the registry, which is how you share internal modules across teams.

Q: A new engineer runs `az deployment group what-if` and nothing under a particular module shows up — what is "short-circuiting"?
A: Short-circuiting is when what-if cannot fully analyze a module or resource because its resource ID or API version cannot be computed outside the deployment context — often due to unresolved expressions or external-state dependencies. When a module short-circuits, all resources inside it are excluded from the what-if results, and the API response includes a diagnostic message. Using a recent CLI (2.75.0+) or Az module (13.1.0+) surfaces those diagnostics instead of failing silently.

Q: What what-if limits cause remaining resources to be reported as `Ignore`?
A: What-if expands nested templates until it hits one of three limits: 500 nested templates, 800 resource groups in a cross-resource-group deployment, or 5 minutes spent expanding nested templates. When a limit is reached, every remaining resource's change type is set to `Ignore`. Very large or deeply nested deployments can therefore produce incomplete previews.

Q: Why might you not see resources from a template-spec or `templateLink` reference in what-if output?
A: During what-if operations, the evaluation and expansion of `templateLink` are not supported, so resources deployed via template links within nested deployments — including template-spec references — do not appear in the results. This is a documented limitation, not a bug. You account for it by knowing those linked resources will still deploy even though what-if was silent about them.

Q: What does the `--validation-level` switch control on a what-if or deployment, and what are its values?
A: `--validation-level` (Azure CLI 2.76.0+; PowerShell `-ValidationLevel` in Az 13.4.0+) controls how thoroughly ARM validates the template. `Provider` (default) does full validation including resource definitions, dependencies, and deployment-permission checks; `ProviderNoRbac` validates fully but only checks read permissions; `Template` does static syntax/structure validation only, skipping preflight and permission checks. Lowering it speeds validation but can miss issues that would later fail the deployment.

Q: What permissions does the what-if operation require?
A: What-if has the same permission requirements as the actual deployment: write access on the resources being deployed plus access to all operations on the `Microsoft.Resources/deployments` resource type. For example, previewing a VM deployment needs `Microsoft.Compute/virtualMachines/write` and `Microsoft.Resources/deployments/*`. Using `--validation-level ProviderNoRbac` relaxes this to read-only checks when you only want to validate configuration.

Q: What is `bicep snapshot`, and how does it differ from what-if?
A: `bicep snapshot` is a local-only testing command that generates and compares a normalized JSON representation of your infrastructure to catch unintended logic changes, without requiring an Azure connection. What-if, by contrast, calls Azure and diffs against live resource state. Snapshot is for fast offline regression checks on the template's compiled output; what-if is for verifying real-world impact before deploying.

Q: A storage-account deployment fails with "name already taken" — how does Bicep naming help you avoid this?
A: Storage-account names must be globally unique, so a hard-coded name collides across environments. The Bicep idiom is to derive the name with `uniqueString()` seeded on a stable scope value, for example `'${prefix}${uniqueString(resourceGroup().id)}'`, which yields a deterministic, collision-resistant suffix that is the same on every redeploy. That keeps the name unique globally yet stable across runs so you update rather than recreate.

Q: You are migrating a fund platform's OpenTofu/Terraform storage-account-plus-Key-Vault-plus-VNet module to Bicep — what maps to what?
A: Each Terraform `resource` block maps to a Bicep `resource` declaration with a type and API version; Terraform `variable`s become Bicep `param`s; `output`s map to Bicep `output`s; and a Terraform module call maps to a Bicep `module` declaration. The state file disappears entirely because ARM tracks live state. You would typically reach for AVM `br/public:avm/res/...` modules for the storage account and Key Vault rather than hand-writing them.

Q: For a Luxembourg fund whose NAV pipeline is entirely on Azure, how does Bicep's no-state-file model reduce regulatory/operational risk?
A: A Terraform state file can contain secrets (connection strings, keys) and must itself be secured, backed up, locked, and access-audited — extra controls a regulated fund administrator must justify under frameworks like DORA. Bicep eliminates that artifact because ARM holds the state, removing a sensitive store and a class of drift/corruption incidents from the operational-resilience picture. Fewer moving parts means a smaller attack and audit surface for the NAV-supporting infrastructure.

Q: An EMT/EPT file-ingestion stack runs on Azure Storage plus Functions — why might Bicep's day-0 feature support matter for compliance deadlines?
A: Regulatory file formats and the controls around them evolve, and you sometimes need a brand-new Azure capability (a new storage feature, a new networking control) the moment it ships to meet a deadline. Bicep can use that resource type or API version on day 0 because it targets ARM directly, whereas the Terraform AzureRM provider may not have added support yet. For a deadline-driven EMT/EPT compliance change, that lag can be the deciding factor.

Q: When provisioning per-fund resource groups (one per sub-fund), which Bicep construct would you use and why?
A: A `for` loop over an array of fund identifiers deploying a module per fund, for example `module fundRg './fund.bicep' = [for fund in funds: { name: 'deploy-${fund.code}', scope: ... }]`. This generates N consistent, parameterized stacks from one declaration, so adding a sub-fund is a one-line array change. Combined with `@batchSize()` you can throttle creation to avoid ARM throttling when onboarding many funds at once.

Q: A complete-mode deployment to a fund's resource group is proposed to "clean up drift" — what is your reaction as architect?
A: I would push back: complete mode is no longer recommended and is being deprecated, and on a regulated fund's resource group an incomplete template could silently delete production resources supporting NAV. The correct approach is incremental mode for normal changes and deployment stacks for controlled, auditable deletions, always preceded by a what-if to review the exact change set. Drift cleanup via complete mode trades a real catastrophic-deletion risk for convenience.

Q: In CI for a fund platform, how would you gate a deployment on the what-if result?
A: Run `az deployment group what-if --resource-group <rg> --template-file main.bicep --no-pretty-print` to get JSON, parse the change set, and fail the pipeline (or require manual approval) if any resource shows a `Delete`. For interactive promotion you would instead use `az deployment group create -c` so an operator confirms the previewed changes. This turns what-if into an automated guardrail against unintended destruction of regulated infrastructure.

Q: How would you store a SWIFT/messaging gateway secret needed by a Bicep deployment without leaking it?
A: Mark the parameter `@secure()` so its value is excluded from deployment history and logs, and pull the actual value from an existing Key Vault using a `Microsoft.KeyVault/vaults` resource declared with the `existing` keyword and a `getSecret()` reference, rather than passing it in plaintext. What-if also refuses to evaluate secure parameter values, so the secret never appears in previews. This keeps the credential out of the IaC artifact entirely.

Q: What is the consequence of forgetting the `@secure()` decorator on a password parameter?
A: Without `@secure()`, the parameter value can be written to the deployment history and surfaced in logs and what-if output, exposing the secret to anyone with read access to the deployment. For a regulated fund administrator this is a credential-leak and audit finding. The fix is to add `@secure()` and ideally source the value from Key Vault rather than passing it directly.

Q: How do you decompile an existing ARM JSON template toward Bicep, and what is the caveat?
A: Run `az bicep decompile --file template.json`, which produces a best-effort `.bicep` file. The caveat is that decompilation is lossy and imperfect — it may emit warnings, fail to map every construct cleanly, and produce code that needs manual cleanup. It is a migration accelerator for moving a fund's incumbent ARM estate to Bicep, not a guaranteed one-shot conversion.

Q: How do you build a Bicep file to ARM JSON manually, and why would you?
A: Run `az bicep build --file main.bicep`, which emits `main.json`. You do this to inspect exactly what ARM will receive, to debug a deployment, or to hand the JSON to a tool or environment that only consumes ARM templates (such as a constrained Azure Stack environment where the intermediate language lags Bicep features). It confirms there is no hidden behavior between your Bicep source and the deployed template.

Q: Why is "the incumbent's deployment code" a key reason to learn Bicep even if you would choose Terraform?
A: In Microsoft-aligned shops, the existing infrastructure is very often already authored in Bicep or ARM, so you cannot maintain, review, or extend production deployments without reading it. An architect who cannot parse the incumbent code is dependent on others and cannot credibly argue any migration. Evaluation fluency lets you read the current state, estimate migration cost, and make a defensible recommendation rather than a vibe-based one.

Q: How does Terraform's provider ecosystem concretely outweigh Bicep when an org runs Azure plus Snowflake plus GitHub?
A: Terraform/OpenTofu has dedicated providers for Snowflake, GitHub, Azure, Datadog, and hundreds more, so one tool and one workflow manage the entire heterogeneous estate with shared variables and dependency graphs. Bicep only speaks Azure (ARM), so it cannot provision the Snowflake warehouse or GitHub repos at all — you would need a second tool. For a genuinely multi-vendor platform, that single-pane-of-glass capability is decisive for Terraform.

Q: What is a deployment stack and why is it the modern answer to complete mode's deletion use case?
A: A deployment stack is an Azure resource that manages a defined collection of resources as a unit, tracking membership so it can delete or detach resources that leave the definition in a controlled, policy-driven way. It replaces complete mode's blunt "delete everything not in the template" behavior with explicit, auditable lifecycle management and deny-settings protection. Microsoft now points you to deployment stacks instead of complete mode for any Bicep/ARM deletion need.

Q: Your `az deployment group what-if` errors with an authentication or context problem — what is the first thing to check?
A: Confirm you are logged in and pointed at the right subscription: run `az account show` and `az account set --subscription <id>` if needed, since what-if requires the same write and `Microsoft.Resources/deployments/*` permissions as a real deployment. On a native Ubuntu laptop also verify the Azure CLI is current (`az version`), because what-if diagnostics and flags like `--validation-level` require recent CLI versions. Only after auth/context/version are confirmed do you investigate the template itself.

Q: `az bicep build` reports a `BCP` error code — what does that prefix mean and how do you handle it?
A: `BCP` codes are Bicep compiler diagnostics, for example `BCP089` for a likely-typo property name. You read the specific code and message to find the offending line, then fix the template; if you have a justified reason to ignore one you can suppress it on the next line with `#disable-next-line BCP089` and a comment explaining why. The codes are case-sensitive and documented in the Bicep core-diagnostics reference.

Q: What do the Bicep directives `#disable-next-line`, `#disable-diagnostics`, and `#restore-diagnostics` do?
A: `#disable-next-line` suppresses one or more diagnostics or linter rules for the immediately following line only; `#disable-diagnostics` suppresses them for the rest of the file (or until restored); and `#restore-diagnostics` re-enables previously disabled ones. You pass identifiers like `no-unused-vars` or `BCP335` after the directive, separated by spaces. The guidance is to use them sparingly and always add a comment explaining the intentional suppression.

Q: On a native Ubuntu laptop, how do you get the Bicep CLI, and how does that integrate with the Azure CLI?
A: The Azure CLI manages Bicep for you: `az bicep install` installs the Bicep CLI into the CLI's own managed location, and `az bicep upgrade` updates it, so you do not need a separate package. Commands like `az deployment group create --template-file main.bicep` automatically invoke Bicep to build the file. This works identically on Ubuntu, macOS, and WSL2, since the path-separator and tooling are cross-platform.

Q: Summarize the one-line decision rule you would give a stakeholder for Bicep versus Terraform.
A: If the estate is and will remain Azure-only and the org is Microsoft-aligned, choose Bicep for its native day-0 coverage, no state file, and first-party support; if you need multi-cloud, a broad provider ecosystem, or state-aware orchestration across vendors, choose Terraform/OpenTofu. Then back the choice with the org's actual cloud footprint, the incumbent codebase, and the team's support posture — not personal preference. Either way, run what-if (or `terraform plan`) before every apply.

Q: What is a `.bicepparam` file and what advantage does it have over a JSON parameters file?
A: A `.bicepparam` file is a Bicep-native parameters file (supported by Azure CLI 2.53.0+ and Bicep CLI 0.22.x+) that supplies values for a template's parameters using real Bicep expressions and IntelliSense, rather than the older `parameters.json` format. Its key feature is the `using` statement at the top that points at the target `.bicep` file, which gives you type-checking and autocomplete against the actual parameters. For a fund platform you keep one `.bicepparam` per environment (dev/uat/prod) so the same template deploys with environment-specific NAV-pipeline settings.

Q: How do you deploy with a `.bicepparam` file, and why must you NOT also pass `--template-file`?
A: Run `az deployment group create --resource-group <rg> --parameters main.bicepparam`; because the `.bicepparam` file already names its template via the `using` statement, the CLI infers the template automatically. Adding `--template-file` produces the error `Only a .bicep file is allowed with a .bicepparam file`, since specifying both is contradictory. This is a common first-time gotcha when migrating from JSON parameters files where `--template-file` was always required.

Q: How do you auto-generate a parameters file from an existing Bicep template?
A: Run `az bicep generate-params --file main.bicep`, which reads every declared `param`, including required ones with no default, and emits a ready-to-edit parameters file. You can control the output format with `--output-format bicepparam` for a `.bicepparam` file or `--output-format json` for the legacy JSON form, and `--include-params all` to also include parameters that have defaults. It saves hand-transcribing a long parameter list and reduces the chance of missing a required value at deploy time.

Q: What does the `using` statement do inside a `.bicepparam` file?
A: The `using` statement declares which Bicep template the parameter file targets, for example `using './main.bicep'`, and it must be the first line. It is what lets the tooling type-check each assigned value against the corresponding `param` declaration and provide IntelliSense. Without it the file has no template context, so the CLI cannot resolve which template the parameters belong to.

Q: How do you pull a Key Vault secret directly into a `.bicepparam` file without exposing it?
A: Use the `az.getSecret()` function inside the `.bicepparam` file, for example `param adminPassword = az.getSecret('<subId>', '<rg>', '<kvName>', '<secretName>')`, which resolves the secret at deployment time so the plaintext never lands in the parameter file or deployment history. This is the modern, preferred way to source secrets and is distinct from the in-template `getSecret()` method on an `existing` vault. For a SWIFT gateway credential it keeps the secret in Key Vault and out of source control entirely.

Q: What are the constraints on the in-template `getSecret()` function for a Key Vault?
A: `getSecret()` can only be called within the `params` section of a `module` declaration, only on a `Microsoft.KeyVault/vaults` resource (typically declared `existing`), and only for a module parameter decorated with `@secure()`. Using it anywhere else in the file, or with string interpolation even inside `params`, raises an error. This deliberately narrow surface stops a secret from being pulled into a non-secure context where it could leak into outputs or logs.

Q: What is the difference between an implicit and an explicit (`dependsOn`) dependency in Bicep?
A: An implicit dependency is created automatically when one resource references another by its symbolic name, for example `properties: { storageId: stg.id }`, so Bicep infers and orders the deployment for you. An explicit dependency is declared manually with the `dependsOn` array property, used only in the rare case where there is no property reference yet an ordering constraint still exists. Best practice is to prefer implicit dependencies, because unnecessary `dependsOn` entries serialize deployments that ARM could otherwise run in parallel, slowing things down.

Q: Why is adding a redundant `dependsOn` when an implicit dependency already exists considered an anti-pattern?
A: Resource Manager deploys independent resources in parallel; an explicit `dependsOn` forces a sequential ordering, so adding one where Bicep already inferred the relationship via a symbolic-name reference needlessly slows the deployment. The `no-unnecessary-dependson` linter rule flags exactly this. Keep `dependsOn` only for genuine ordering with no property reference between the resources.

Q: How do you declare a user-defined function in Bicep, and what keyword does it use?
A: Use the `func` keyword: `func <name>(<arg> <type>) <return-type> => <expression>`, for example `func buildName(prefix string, code string) string => '${prefix}-${code}'`. User-defined functions let you factor a repeated expression — like a fund-naming convention — into one reusable, testable place. You can annotate them with `@description(...)` and share them across files by adding the `@export()` decorator and importing them elsewhere.

Q: How do you declare a user-defined type, and where is it useful in a fund-data context?
A: Use the `type` keyword: `type <name> = <type-expression>`, for example `type fundConfig = { code: string, isin: string, navCurrency: string }`. You then type a parameter with it, like `param fund fundConfig`, giving compile-time validation of the shape and IntelliSense on its members. For onboarding sub-funds this enforces that every fund object carries the required `isin` and `navCurrency` fields rather than relying on loosely typed objects.

Q: What does the `@export()` decorator enable, and how do you consume an exported symbol?
A: `@export()` marks a `type`, `func`, or `var` so it can be shared with other Bicep files, turning a file into a reusable library of compile-time symbols. You consume it in another file with an `import` statement, for example `import { fundConfig, buildName } from './shared.bicep'`. This is how a team centralizes naming functions and shared types instead of duplicating them across templates.

Q: What does the `loadTextContent()` function do, and when does it run?
A: `loadTextContent('file.txt')` reads a non-binary file's contents into a string at compile time, so the data is baked into the generated ARM JSON during `az bicep build` on the author's machine, not at deployment. It is handy for embedding a script, a cloud-init file, or a JSON config alongside a resource. Because it runs at compile time you cannot pass a parameter or variable as the path — the path must be a literal compile-time constant.

Q: What is the difference between `loadTextContent()`, `loadJsonContent()`, and `loadFileAsBase64()`?
A: `loadTextContent()` loads a text file as a string, `loadJsonContent()` parses a JSON file into an object you can index, and `loadFileAsBase64()` encodes a binary file as a base64 string for embedding. All three are compile-time functions resolved during the build. The size limits differ: text and JSON allow up to 131,072 characters, while `loadFileAsBase64()` caps the source file at 96 KB.

Q: Why can't you use a parameter to choose which file `loadTextContent()` loads?
A: Because file-loading functions run at compile time, before any parameter values exist — parameters are only resolved at deployment. The path argument must therefore be a literal compile-time constant; passing a `param` or unresolved expression raises a `BCP032` "value must be a compile-time constant" error. If you need runtime file selection you must restructure to load all candidates and select with a runtime expression, or move the logic outside Bicep.

Q: What is a deployment stack, and which `az` command group manages one at resource-group scope?
A: A deployment stack is an Azure resource (`Microsoft.Resources/deploymentStacks`) that manages a defined set of resources as a single lifecycle unit, tracking membership so resources removed from the template can be cleaned up in a controlled way. You manage it with the `az stack group` command group at resource-group scope, plus `az stack sub` and `az stack mg` for subscription and management-group scopes. It is Microsoft's recommended replacement for complete-mode deletions.

Q: What are the three values of `--action-on-unmanage` on a deployment stack and what does each do?
A: `deleteAll` deletes both managed resources and managed resource groups that fall out of the stack; `deleteResources` deletes only the managed resources but leaves resource groups; and `detachAll` removes them from the stack's management but leaves them in Azure untouched. You pass it as `--action-on-unmanage deleteResources` (alias `--aou`). For a regulated fund platform, starting with `detachAll` is the safe default until you trust the workflow, so an accidental template omission cannot delete production resources.

Q: What does `--deny-settings-mode` on a deployment stack control, and what are its values?
A: `--deny-settings-mode` sets a protection lock on the stack's managed resources: `none` applies no restriction, `denyDelete` blocks delete operations, and `denyWriteAndDelete` blocks both writes and deletes even by users with RBAC permissions. This guards a stack's resources against out-of-band tampering, which is valuable for the infrastructure underpinning a fund's NAV pipeline. You can carve out exceptions with `--deny-settings-excluded-actions` and `--deny-settings-excluded-principals`.

Q: How does a deployment stack with deny-settings differ from a normal Azure resource lock?
A: A resource lock (`CanNotDelete`/`ReadOnly`) is applied per resource or scope independently of any deployment, whereas a stack's deny-settings are bound to the managed set and travel with the stack's lifecycle, including the ability to exclude specific actions or principals. Deny-settings also block writes/deletes for users who otherwise have permission, similar to a lock, but are managed and updated as part of the stack definition rather than as separate lock objects. This makes the protection declarative and tied to the IaC unit rather than bolted on afterward.

Q: How do you preview changes for a deployment stack before applying, and what's the catch?
A: Deployment stacks support what-if via flags on the create/update commands, for example passing the stack template with a preview, but the canonical preview workflow is still to run `az deployment group what-if` against the same template, or use the stack's validate path. The catch is that stacks add their own lifecycle semantics (unmanage actions, deny-settings) that a plain what-if of the template does not show, so you must reason about the action-on-unmanage behavior separately. Always confirm both the resource change set and the unmanage action before applying to regulated infrastructure.

Q: What is the Bicep linter, and how do you run it from the CLI explicitly?
A: The Bicep linter statically analyzes a `.bicep` file for best-practice violations and likely errors, running automatically in VS Code and during `az bicep build`. You can invoke it directly with `az bicep lint --file main.bicep`, which reports rule violations without producing a build artifact. It reads `bicepconfig.json` to decide which rules are active and at what severity.

Q: In `bicepconfig.json`, under which section are linter rules now configured, and what does a rule's `level` do?
A: Linter rules live under the `analyzers` section, specifically `analyzers.core.rules`, where each rule has a `level` of `off`, `info`, `warning`, or `error`. Setting a rule's `level` to `error` makes a violation fail the build, while `off` disables it entirely. For example you might raise `secure-secrets-in-parameters` to `error` so a leaked-secret pattern blocks CI for a regulated fund platform.

Q: What does the `no-hardcoded-env-urls` linter rule catch, and what should you use instead?
A: It flags hard-coded cloud-environment URLs such as `core.windows.net`, `vault.azure.net`, `login.microsoftonline.com`, or `management.azure.com` that differ across Azure clouds (public, government, China). The fix is to call the `environment()` function so the correct URL is resolved at deployment time for the target cloud. Defaulting to `warning`, this rule matters if a fund ever needs to deploy the same templates into a sovereign or government cloud.

Q: What does the `secure-secrets-in-parameters` linter rule enforce?
A: It warns when a parameter whose name suggests it holds a secret (containing tokens like `password`, `secret`, or `key`) is declared without the `@secure()` decorator, or when a secure value appears to be used insecurely. The intent is to stop credentials from being written to deployment history or outputs. Treating this as an error in CI is a cheap guard against an auditable credential-leak finding for a regulated administrator.

Q: How do you suppress a single linter rule on just one line, and what should accompany it?
A: Place `#disable-next-line <rule-name>` on the line immediately above the offending statement, for example `#disable-next-line no-hardcoded-env-urls`, which silences only that rule for only the next line. You should always add a trailing comment explaining the justified reason, because an unexplained suppression is itself an audit smell. For broader scope you would use `#disable-diagnostics`, but per-line suppression is preferred to keep the blast radius minimal.

Q: A `for` loop module deployment fails because all instances use the same `name` — how do you fix it?
A: Each looped module's `name` (the nested-deployment name) must be unique within the deployment, so derive it from the loop variable, for example `name: 'deploy-${fund.code}'` or `name: '${i}-storage'`. Reusing a static name across iterations causes later iterations to collide with or overwrite earlier nested deployments. Including the iteration index or a fund identifier in the name guarantees uniqueness across all instances.

Q: How do you reference a specific item from a looped resource elsewhere in the file?
A: Index into the symbolic name with the loop position, for example `storageAccounts[0].id` to get the first instance's resource ID, or use a nested `for` to project all of them. Bicep treats a looped resource as an array of resources keyed by the same order as the iteration source. This is how you wire, say, the third per-fund storage account into another resource that needs its endpoint.

Q: How do you combine a condition and a loop on the same resource declaration?
A: Put the `if` on each iteration inside the loop body using a ternary, or filter the source array before iterating, because you cannot place a bare `if` after a `for` expression on the same line. The idiomatic approach is `[for fund in funds: (fund.deploy ? { ...real config... } : { ...skipped... })]` is not valid; instead you filter with `filter(funds, f => f.deploy)` and loop over the result. So you typically pre-filter the array and loop only over the wanted items.

Q: What does the `filter()` lambda function do, and give a fund-data example.
A: `filter(array, predicate)` returns a new array containing only the elements for which the lambda predicate is true, for example `filter(funds, f => f.navCurrency == 'EUR')` to deploy resources only for euro-denominated sub-funds. Bicep supports lambda expressions in `filter()`, `map()`, `reduce()`, `sort()`, and `toObject()`. This lets you derive per-fund resource sets declaratively instead of hand-listing them.

Q: What does the `map()` lambda function produce, and why is it useful for naming?
A: `map(array, lambda)` transforms each element of an array and returns a new array of the results, for example `map(funds, f => toLower('${prefix}${f.code}'))` to produce a list of lowercased storage-account names. It is useful for projecting a list of fund objects into a list of derived names or configs in one expression. You can then feed that derived array straight into a `for` loop.

Q: What is the `@minLength()`/`@maxLength()` pair used for, versus `@minValue()`/`@maxValue()`?
A: `@minLength()` and `@maxLength()` constrain the length of a string or array parameter, for example `@minLength(3) @maxLength(24)` to match a storage-account name's allowed range. `@minValue()` and `@maxValue()` constrain the numeric range of an `int` parameter, for example `@minValue(0) @maxValue(10)` on an instance count. Both are guardrails that fail validation early rather than letting Azure reject the value mid-deployment.

Q: Why must a storage-account name parameter often carry `@minLength(3)` and `@maxLength(24)` plus a lowercase constraint?
A: Azure storage-account names must be 3 to 24 characters, lowercase letters and numbers only, and globally unique, so encoding the length bounds as decorators catches an out-of-range value at template validation. The lowercase/alphanumeric rule itself is enforced by Azure, but you often add a `@description` noting it and derive the name via `toLower()` and `uniqueString()`. This prevents a late `StorageAccountAlreadyTaken` or name-format deployment failure.

Q: What does the `@description()` decorator add to the compiled output, and why does it matter for a shared module?
A: `@description('...')` attaches human-readable documentation to a `param`, `output`, `resource`, or other element, and it is emitted into the ARM JSON metadata so it surfaces as IntelliSense and in the portal's custom-deployment UI. For a shared AVM-style module this is how a consumer learns what each parameter means without reading the source. Omitting descriptions on a reusable module is flagged by guidance because it degrades the consumer experience.

Q: How do you import a published module from your private Azure Container Registry, including the alias mechanism?
A: Reference it as `br:<registry>.azurecr.io/<repo>:<tag>`, for example `br:fundregistry.azurecr.io/bicep/storage:v2`, or define an alias in `bicepconfig.json` under `moduleAliases` so it reads `br/FundModules:storage:v2`. The consumer needs `AcrPull` on the registry, and the Bicep tooling runs `bicep restore` to cache it locally. Aliases keep long registry paths out of every template and let you re-point the registry centrally.

Q: A teammate's build fails with a registry module not found, but yours works — what is the likely cache/auth difference?
A: The external module must be restored into the local cache and the user must be authenticated to the registry; your teammate likely hasn't run `az login` against a context with `AcrPull`, or their cache is stale. Have them run `az bicep restore --file main.bicep` (or `az bicep restore --force` to refresh) after `az login`, and confirm they can `az acr login --name <registry>`. CI must also authenticate to the ACR before `az bicep build`, otherwise the restore step fails.

Q: What does `az bicep restore --force` do, and when do you need it?
A: It re-downloads external registry modules into the local cache even if a cached copy exists, overwriting stale entries. You need it when a module tag was re-published (which is discouraged) or when the local cache is corrupted and IntelliSense or builds reference an outdated module body. Routine builds restore automatically, so `--force` is a targeted recovery action rather than a daily command.

Q: How do you decompile a legacy JSON parameters file into the newer `.bicepparam` format?
A: Run `az bicep decompile-params --file parameters.json --bicep-param-file out.bicepparam`, which converts a legacy JSON parameters file into the newer `.bicepparam` format. This is the parameter-file analogue of `az bicep decompile` for templates and is part of migrating an estate fully onto Bicep-native tooling. As with template decompilation, review the output, since the conversion can need minor manual fixes.

Q: What is the practical effect of the `no-unused-params` and `no-unused-vars` linter rules?
A: They warn when a declared `param` or `var` is never referenced anywhere in the file, flagging dead inputs and computed values that add confusion and, for parameters, force callers to supply something with no effect. Cleaning these keeps a module's contract honest — every parameter a consumer sees actually does something. You suppress a deliberate placeholder with `#disable-next-line no-unused-params` plus an explanatory comment.

Q: How would you express "deploy this DR storage account only in production" cleanly in Bicep?
A: Gate the resource with a condition driven by an environment parameter, for example `@allowed(['dev','uat','prod']) param env string` and `resource drStg '...' = if (env == 'prod') { ... }`. The `@allowed` decorator constrains the input to valid environments and the `if` skips the resource outside prod. This keeps one template serving all environments while only provisioning the DR copy where it's required.

Q: What is the difference between `resourceGroup()`, `subscription()`, and `tenant()` as functions versus as scope arguments?
A: As standalone functions they return objects describing the current deployment context — `resourceGroup().location`, `subscription().subscriptionId`, `tenant().tenantId` — used to derive values. As scope arguments to a module or `existing` resource, the same functions (optionally with arguments like `resourceGroup('otherRG')`) target the deployment at a different scope. So `resourceGroup()` with no args reads the current group, while `resourceGroup('demoRG')` names a specific one as a deployment target.

Q: How do you read the deployment region inside a template without hard-coding it?
A: Use `resourceGroup().location` for a resource-group-scoped deployment, which returns the region of the target resource group, and assign it to each resource's `location` property or a `param location string = resourceGroup().location` default. This keeps the template portable across regions and is the idiomatic default. At subscription scope you instead use `deployment().location`, since there is no resource group to read.

Q: What does the `environment()` function return and when must you use it?
A: `environment()` returns details about the Azure cloud the deployment targets, including endpoint suffixes like `environment().suffixes.storage` or the portal/login URLs, which differ between Azure public, US Government, and China clouds. You must use it instead of literal hostnames so the same template works across sovereign clouds, which is exactly what the `no-hardcoded-env-urls` linter rule enforces. For a Luxembourg fund admin staying on Azure public it is still best practice for portability.

Q: A module output you need is marked `@secure()` — how do you consume it and what's the version requirement?
A: From Bicep 0.35.1 onward you can decorate a module `output` with `@secure()`, and you read it the same way as any output, `myModule.outputs.connString`, but its value is excluded from deployment history and logs. Before that version, secure module outputs were not supported and the value would have been exposed. Use this to pass a generated connection string from one module to the next without it surfacing in the deployment record.

Q: Why is exporting a generated secret as a non-secure `output` a finding, even within one deployment?
A: A plain `output` value is written to the deployment history and is readable by anyone with read access to the deployment operations, so emitting a key or connection string as a non-secure output leaks it. The fix is to mark the output `@secure()` (Bicep 0.35.1+) or, better, avoid outputting the secret at all and have downstream resources pull it from Key Vault. For a regulated fund this is the difference between a clean deployment record and an audit issue.

Q: How do you target a deployment at a subscription that creates a resource group and then deploys into it?
A: Set `targetScope = 'subscription'`, declare the group as a resource `resource rg 'Microsoft.Resources/resourceGroups@2025-04-01' = { name: 'fund-rg', location: 'westeurope' }`, then declare a module with `scope: rg` so its resources land inside the new group. The subscription-scoped file orchestrates group creation and intra-group deployment in one run. This is the standard pattern for onboarding a new sub-fund's isolated resource group.

Q: What deployment-name and history considerations apply when running the same template repeatedly in CI?
A: Each `az deployment group create` records a named deployment in the resource group's history (default name derived from the template file), and the history is capped at 800 deployments per resource group before the oldest are auto-deleted. Reusing the same `--name` overwrites that named deployment entry rather than accumulating, which can be intentional for idempotent CI. For traceability you often pass `--name` with a build ID so each pipeline run is auditable.

Q: How would you validate a template in CI without any chance of it deploying?
A: Run `az deployment group validate --resource-group <rg> --template-file main.bicep --parameters main.bicepparam`, which performs preflight validation against ARM (schema, references, and permission checks) but never creates or changes resources. It is stricter than a local `az bicep build` because it talks to Azure, yet safer than `what-if` if you only need a pass/fail gate. You can tune its depth with `--validation-level` (`Provider`, `ProviderNoRbac`, or `Template`).

Q: When migrating a Terraform `for_each` over a map of objects to Bicep, what construct do you reach for?
A: Bicep's `for` loop over an array or over `items(object)` for a map, for example `[for fund in items(funds): { ... }]` where `funds` is an object keyed by fund code. The `items()` function turns a map/object into an array of `{ key, value }` entries you can iterate, mirroring `for_each`'s key/value access. This is the closest idiom to Terraform's keyed iteration, since Bicep loops are positional arrays under the hood.

Q: After deploying with a deployment stack, an operator gets "denied" trying to delete a managed resource — is this a bug?
A: No — this is the stack's `--deny-settings-mode` working as designed: if set to `denyDelete` or `denyWriteAndDelete`, even a user with RBAC delete permission is blocked from removing a stack-managed resource out of band. The intended path is to update or delete via the stack itself, or to add the action/principal to `--deny-settings-excluded-actions`/`--deny-settings-excluded-principals`. For regulated fund infrastructure this is a feature: it prevents accidental or unauthorized tampering with NAV-supporting resources.

Q: Your `az deployment group create` fails immediately with `InvalidTemplateDeployment` and a nested message about a SKU — where does the error originate?
A: An `InvalidTemplateDeployment` returned at deploy time comes from ARM preflight validation rather than the Bicep compiler, so the Bicep file built cleanly but Azure rejected a value such as an unsupported storage SKU in your target region. The nested error details name the offending resource and property, which you read to find, say, a `Standard_GZRS` SKU unavailable in `westeurope`. The fix is to correct the value or constrain it with `@allowed([...])` so the bad input is caught earlier; running `az deployment group validate` would have surfaced it before the create.

Q: How do you pass a complex object or array parameter on the command line versus in a parameters file, and which is preferred?
A: On the CLI you can inline JSON, for example `--parameters funds='[{"code":"LUX1"}]'`, but quoting nested JSON in a shell is fragile and error-prone, especially on a native Ubuntu terminal. The preferred approach is a `.bicepparam` file where the array is written as a typed Bicep literal with IntelliSense and validation against a user-defined type. For anything beyond a scalar — like a list of sub-fund objects — the parameters file keeps the deployment command clean and the values reviewable in source control.
