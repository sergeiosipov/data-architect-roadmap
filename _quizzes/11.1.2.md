# Phase 6 · 11.1.2 Fine-grained access control (RLS/CLS on the platform) — 150 self-test questions

Q: What does "fine-grained access control" mean in a data platform, and how does it differ from coarse table-level grants?
A: Fine-grained access control restricts access at the row, column, or even cell level rather than only at the whole-table or whole-database level. Coarse control answers "may this role read `nav_history`?"; fine-grained control answers "may this role read *which rows* and *which columns* of `nav_history`". In fund administration this is mandatory because two TA agents may both need the same table but must see only their own management company's funds.

Q: Distinguish row-level security (RLS) from column-level security (CLS) in one sentence each.
A: RLS filters *which rows* a principal can see, so a TA agent querying `funds` silently gets only rows for their management company. CLS controls *which columns* a principal can see or whether the values are obscured, so an analyst might see a fund row but with the investor's tax ID masked. RLS narrows the result set by row predicate; CLS narrows it by column visibility or value masking.

Q: Why is "who may see which fund's data at which column grain" a regulatory requirement rather than a nice-to-have?
A: Regulators such as the CSSF and GDPR supervisory authorities treat improper data exposure as a reportable breach, and Chinese walls between sub-funds and clients are explicitly access-control obligations. If a TA agent sees another management company's investors, that is a confidentiality breach no after-the-fact apology undoes. So the architect must both *design* the control and *prove* it under review with evidence.

Q: What is a "Chinese wall" in the fund-administration context, and how does it map to RLS/CLS?
A: A Chinese wall is an enforced information barrier preventing one party from seeing another's confidential data even inside the same firm or system. In a shared TA platform, the wall between management companies maps directly to an RLS predicate (each agent sees only their funds) plus CLS masking of investor PII. Getting the predicate wrong silently leaks data, which is why the wall must be tested, not assumed.

Q: Name the three rungs of the authorization-model ladder this lesson covers, from simplest to most expressive.
A: The ladder runs RBAC (role-based), then ABAC (attribute-based), then ReBAC (relationship-based). Each rung adds expressiveness: roles, then arbitrary attributes of subject/object/environment, then graph relationships between entities. You climb the ladder only as far as your access requirements actually demand, because each step adds power but also complexity to author and audit.

Q: Define RBAC and give its core limitation.
A: RBAC (role-based access control) grants permissions to roles, and users acquire permissions only by holding roles, which simplifies onboarding and offboarding because you change a role membership, not dozens of grants. Its limitation is "role explosion": when access depends on many combinations of attributes (fund, region, sensitivity), you end up minting a separate role per combination. At that point ABAC's attribute predicates are far cleaner than thousands of narrow roles.

Q: Define ABAC and explain what makes it more expressive than RBAC.
A: ABAC (attribute-based access control) decides access by evaluating attributes of the subject, the object, the requested action, and the environment, rather than a fixed role-to-permission map. It is more expressive because a single policy like "allow if `subject.mgmtco == fund.mgmtco`" covers every fund without minting per-fund roles. This is exactly the form the "TA agents see only their management company's funds" rule takes, so ABAC fits fund visibility naturally.

Q: Define ReBAC and give a finance example where relationships drive access.
A: ReBAC (relationship-based access control) decides access by traversing a graph of relationships between subjects and resources, granting access when a path of the right kind exists. A finance example is "an analyst may see a fund if they are assigned to the management company that owns it", where ownership and assignment are edges in a graph. The GLEIF Level-2 "who owns whom" ownership graph is a natural backing store for such relationship-driven decisions.

Q: When would you choose ABAC over ReBAC for a data-access rule?
A: Choose ABAC when access depends on comparing flat attributes that travel with the request and the data, such as a management-company code on both the user and the fund row. Choose ReBAC when access depends on multi-hop relationships that are awkward to flatten into attributes, such as "any ancestor in the ownership chain is permitted". If the decision is a single attribute equality, ABAC is simpler; if it is graph traversal, ReBAC earns its keep.

Q: How does the RBAC→ABAC→ReBAC ladder relate to OPA?
A: OPA (Open Policy Agent) is model-agnostic: its Rego language can express RBAC role maps, ABAC attribute predicates, and ReBAC graph traversals, so the ladder is about *what* you express, not *which engine*. The OPA documentation frames RBAC and ABAC as policy patterns you encode in Rego rather than separate products. That is why later lessons build the fund-visibility policy in Rego and feed it to Trino.

Q: What single SQL statement turns on row-level security for a Postgres table?
A: `ALTER TABLE table_name ENABLE ROW LEVEL SECURITY;` enables RLS so that every row access becomes subject to policies. Crucially, enabling it with no policies yet defined produces a default-deny: the table appears empty to non-owners. You then add `CREATE POLICY` statements to open up exactly the rows each role should see.

Q: With RLS enabled but no policies created, what does a non-owner see when they `SELECT *`?
A: They see zero rows, because Postgres RLS is default-deny: enabling row security without any permissive policy means no row satisfies any policy, so none are returned. This is a deliberate fail-closed design so that forgetting to write a policy never accidentally exposes data. The table owner, by contrast, still sees everything unless `FORCE ROW LEVEL SECURITY` is set.

Q: Write a minimal Postgres RLS policy so that TA agents see only rows where the row's `mgmtco` matches their identity.
A: `CREATE POLICY agent_own_mgmtco ON funds USING (mgmtco = current_setting('app.mgmtco'));` restricts visible rows to those whose `mgmtco` equals a session variable set per connection. The `USING` expression is evaluated per row, and only rows where it is true are returned. You would set `app.mgmtco` at login via the connection pooler or an application hook so the agent cannot spoof another company's code.

Q: In a `CREATE POLICY`, what is the difference between the `USING` clause and the `WITH CHECK` clause?
A: `USING` controls which existing rows are *visible* and thus applies to `SELECT`, `UPDATE`, and `DELETE` (the rows you may touch). `WITH CHECK` controls which *new or modified* rows are *allowed*, so it applies to `INSERT` and `UPDATE` (the values you may write). If `WITH CHECK` is omitted, it defaults to the `USING` expression, which prevents a user from updating a visible row into a state they could not have created.

Q: A user can `SELECT` their own rows but an `INSERT` of a valid-looking row is rejected. Which clause is responsible?
A: The `WITH CHECK` clause (explicit or inherited from `USING`) is rejecting the insert because the new row's values fail the policy predicate. For example, if the policy is `WITH CHECK (mgmtco = current_setting('app.mgmtco'))` and the insert supplies a different `mgmtco`, Postgres raises a `new row violates row-level security policy` error. The fix is to insert a row whose attributes satisfy the policy, not to weaken the check.

Q: How do multiple PERMISSIVE policies on the same table combine?
A: Permissive policies combine with OR, so a row is accessible if *any* permissive policy permits it. This means adding a permissive policy can only ever widen access, never narrow it. `PERMISSIVE` is the default policy type, which is why most teams reason about RLS as "a row is visible if at least one policy says yes".

Q: How do RESTRICTIVE policies combine, and when would you use one?
A: Restrictive policies combine with AND, so a row must satisfy *every* restrictive policy in addition to passing at least one permissive policy. You use `AS RESTRICTIVE` to impose a hard constraint that no permissive policy can override, such as "only from an internal IP" or "never expose flagged investors". The full rule is: a row is visible only if it passes at least one `PERMISSIVE` policy AND all `RESTRICTIVE` policies.

Q: A table has one permissive policy that should expose 100 rows, plus a restrictive policy. The user sees fewer than 100 rows. Why?
A: The restrictive policy is filtering out the difference, because the final visible set is (passes any permissive) AND (passes all restrictive). The permissive policy admitted the candidate rows, but the restrictive predicate then excluded some of them. To debug, evaluate the restrictive `USING` expression against the missing rows to see which condition they fail.

Q: Who bypasses Postgres RLS by default, and why is this a classic gotcha?
A: Table owners and superusers bypass RLS by default, and roles with the `BYPASSRLS` attribute always bypass it. The gotcha is that you test policies as a restricted role, they work, then the application connects as the table owner and silently sees everything. The fix for the owner case is `ALTER TABLE ... FORCE ROW LEVEL SECURITY`, but superusers and `BYPASSRLS` roles still bypass regardless.

Q: What does `ALTER TABLE funds FORCE ROW LEVEL SECURITY` do?
A: It makes the table *owner* also subject to RLS policies, which they normally bypass. This matters when your application connects as the schema owner rather than a restricted login, because without `FORCE` the owner sees all rows and the wall is silently ineffective. Note that superusers and `BYPASSRLS` roles are still exempt even with `FORCE` set.

Q: What is the `BYPASSRLS` role attribute and what operational risk does it carry?
A: `BYPASSRLS`, set with `ALTER ROLE name BYPASSRLS`, lets a role read and write every row regardless of any policy, intended for backups and bulk maintenance. The operational risk is that any login granted it punches straight through every Chinese wall, so it must be tightly held and audited. In a regulated estate you treat `BYPASSRLS` roles like superuser: minimal, named, and reviewed at recertification.

Q: Why can a backup job fail or silently produce partial data when RLS is enabled?
A: If the backup connects as a non-owner without `BYPASSRLS`, RLS filters the dump to only the rows that role can see, producing an incomplete backup. Postgres offers `SET row_security = off`, which makes queries error rather than silently filter when the role cannot bypass RLS, turning a silent data-loss bug into a loud failure. The proper fix is to run dumps as a role that legitimately bypasses RLS.

Q: What is a leakproof function in Postgres RLS, and why does it matter for correctness of filtering?
A: A leakproof function is one guaranteed not to leak information about its arguments through side channels such as error messages or timing. The planner may apply leakproof functions *before* the row-security check, whereas non-leakproof user conditions are applied only after, to avoid revealing filtered rows. This ordering is the mechanism behind the classic RLS "leak" caveat: a crafted non-leakproof predicate or a raised error can disclose the existence of rows you should not see.

Q: Give a concrete example of how an RLS policy can leak the existence of hidden rows.
A: A user runs a query with a custom function or a division that errors only for certain values; because user predicates run after the policy filter, an error triggered by a hidden row reveals that such a row exists. For instance, inserting a value that violates a unique key tells the user a conflicting row is present even though RLS hides it. The defence is to avoid leaking error detail and to treat RLS as confidentiality of *content*, not always perfect concealment of *existence*.

Q: How do you wire a per-session attribute (like the agent's management company) into an RLS policy safely?
A: Set a custom session variable, e.g. `SELECT set_config('app.mgmtco', 'LU123', false);`, and read it in the policy with `current_setting('app.mgmtco')`. The safety requirement is that only trusted code sets it — typically the connection pooler or application middleware after authenticating — so the end user cannot overwrite it. If the user could run `set_config` themselves, the whole wall collapses, which is why this is set out of band, not from user SQL.

Q: Why is `current_user` sometimes a poor key for RLS in a pooled application, and what is the alternative?
A: In a connection-pooled app, every request shares one database login, so `current_user` is the pool's service account, not the human, making it useless as a per-user filter. The alternative is to set a trusted session variable carrying the authenticated principal's attributes and key the policy on `current_setting(...)`. This pushes identity from the database login into application-set context that the pool refreshes per request.

Q: What Postgres error message indicates an `INSERT`/`UPDATE` was blocked by a `WITH CHECK` clause?
A: Postgres raises `new row violates row-level security policy for table "..."`. It means the row you tried to write does not satisfy the policy's `WITH CHECK` (or inherited `USING`) predicate, so the engine refuses to persist a row the user could not legitimately own. First thing to check is whether the inserted attributes match the session context the policy expects.

Q: Are `TRUNCATE` and foreign-key constraint checks subject to RLS?
A: No. `TRUNCATE` is not governed by row-security policies, and referential-integrity checks (unique, primary, and foreign keys) bypass RLS to keep constraints sound. This is a subtle leak vector: a foreign-key or unique violation can confirm the existence of a row the querying user cannot see. So RLS protects normal `SELECT`/`INSERT`/`UPDATE`/`DELETE` visibility but not these integrity-maintenance paths.

Q: Why can RLS policies hurt query performance, and how do you mitigate it?
A: Each policy `USING` expression becomes an extra predicate the planner appends to every query, and if it contains a correlated sub-SELECT it can run per row and defeat indexes. Mitigate by keeping policy predicates simple and indexable (a column compared to a session variable), avoiding sub-selects, and indexing the column the predicate filters on. Measure with `EXPLAIN` to confirm the policy predicate uses an index scan rather than forcing a sequential scan.

Q: A query that was fast becomes slow after enabling RLS. What is the first thing to check?
A: Run `EXPLAIN (ANALYZE)` and look at whether the policy predicate is being applied as an indexable filter or is forcing a sequential scan, and whether any sub-SELECT in the policy is executing repeatedly. RLS adds the `USING` expression to the plan, so a non-sargable predicate (a function on the column, or a sub-query) is the usual culprit. Rewrite the policy to compare an indexed column directly to `current_setting(...)`.

Q: Distinguish "restricting columns" from "masking values" in column-level security.
A: Restricting columns removes the column from the result entirely — the user cannot select it, often enforced by column-level `GRANT`/`REVOKE` or a view. Masking obscures the *value* while the column still appears, returning something like `XXXX` or a partial string to unprivileged users. Restriction hides that the data exists in the result shape; masking keeps the schema stable for applications but blanks the sensitive content.

Q: What is SQL Server Dynamic Data Masking (DDM) and what does it actually change?
A: DDM is a column-level feature that obscures sensitive values in *query results* for non-privileged users, defined with `MASKED WITH (FUNCTION = '...')` on a column. It does not change the data at rest — the real value stays in the table — and it does not encrypt; it only rewrites what unprivileged result sets show. So `SELECT` still returns the column, but masked principals see the masked form instead of the true value.

Q: List the masking function types SQL Server DDM provides.
A: DDM offers `default()` (full mask per data type), `email()` (exposes first letter and a `.com` suffix), `random(start, end)` (random number in range for numeric columns), `partial(prefix, padding, suffix)` (custom string exposing the edges), and `datetime("part")` (masks a chosen date/time component, from SQL Server 2022). Each is supplied in the `MASKED WITH (FUNCTION = '...')` clause on the column. You pick the function whose output shape best matches what the application still needs to render.

Q: What permission lets a user see DDM-masked values in their true form?
A: The `UNMASK` permission. A user with `SELECT` but not `UNMASK` sees masked output; granting `UNMASK` reveals the real values for the columns they can already select. From SQL Server 2022, `UNMASK` can be granted at database, schema, table, or column scope, so you can reveal exactly one sensitive column to a role rather than all of them.

Q: Why is DDM explicitly not a substitute for true access control or encryption?
A: Because a user with ad-hoc query rights can infer the underlying value through range-guessing predicates — for example `WHERE Salary BETWEEN 99999 AND 100001` returns matching rows even though `Salary` shows masked. Microsoft's own documentation warns DDM defends against accidental exposure, not a determined querier, and recommends pairing it with proper permissions, RLS, encryption, and auditing. So DDM reduces casual leakage; it does not contain a motivated insider.

Q: A masked column in SQL Server is copied via `SELECT INTO` by an unprivileged user. What lands in the new table?
A: Masked data lands in the target table, because the source query already returned masked values for a user lacking `UNMASK`. The copy is therefore *statically* masked — the real values were never exposed to that user to copy. This is the same behaviour for `INSERT INTO`, import/export, and BCP performed without `UNMASK`, which is why exports by low-privilege roles are safe but also irreversibly redacted.

Q: How does masking a DiscountCode with `random(1,100)` behave across two queries, and what does that tell you?
A: Each query returns a *different* random value within the range, because `random()` regenerates per result rather than producing a stable mask. This tells you DDM masking is presentation-only and non-deterministic for `random()`, so you must never join or compare on a masked column expecting consistency. It also means masked output is not a stable pseudonym; for stable tokenisation you need a different mechanism.

Q: When you apply an expression to a DDM-masked column, what masking does the result use?
A: The result is always masked with the `default()` function, regardless of which function masked the underlying column. So projecting `LEFT(Email, 5)` over an `email()`-masked column does not inherit the email mask; SQL Server falls back to the default mask on the derived expression. This prevents users from peeling off the original value by wrapping the column in an expression.

Q: How would you express RLS in Postgres and CLS in SQL Server together for an investor table holding PII?
A: Use Postgres RLS so each TA agent's session only returns rows for their management company via a `USING (mgmtco = current_setting('app.mgmtco'))` policy, and use column masking (DDM in SQL Server, or a view/mask in Postgres) so PII columns like tax ID render as `XXXX` unless the role holds `UNMASK`. RLS handles "which investors", CLS handles "which fields of those investors". Together they implement both the Chinese wall and PII minimisation the regulator expects.

Q: What does "lakehouse enforcement reality" mean — name the three layers where access can be enforced.
A: Enforcement can happen at the engine level (the query engine like Trino applies filters and masks), the catalog level (a governance catalog like Unity Catalog evaluates privileges for every engine that goes through it), and the file/object-storage layer (the underlying S3/ADLS/HDFS permissions). The "reality" is that a control is only as strong as the layer that cannot be bypassed beneath it. If the file layer is open, engine-level rules can be sidestepped by reading the Parquet directly.

Q: How does Trino enforce access control, and at which layer?
A: Trino enforces at the engine/query layer: for each query it consults a configured access-control plugin and applies allow/deny decisions, row filters, and column masks before returning results. It does not control the underlying object store, so the enforcement binds queries *that go through Trino*. You configure it via the `etc/access-control.properties` file with a chosen `access-control.name`.

Q: What is Trino's file-based access control and when is it appropriate?
A: File-based access control (`access-control.name=file`) reads a JSON rules file mapping users, groups, catalogs, schemas, and tables to allowed actions, including row filters and column masks. It is appropriate for relatively static, version-controllable rule sets that fit in a file and change through review. For dynamic, attribute-rich, or externally sourced decisions, the OPA plugin is the better fit.

Q: How does the Trino OPA access-control plugin work end to end?
A: You set `access-control.name=opa` and `opa.policy.uri` to an OPA endpoint; for each operation Trino sends a JSON document with a context (user, groups, query ID, version) and an action (for example `SelectFromColumns` with the catalog/schema/table/columns), and OPA returns `{"allow": true}` or `{"allow": false}`. Trino then permits or denies based on that boolean. This externalises the decision so policy lives in Rego, testable and versioned, rather than in Trino config.

Q: Which Trino config properties enable OPA-driven row filtering and column masking?
A: `opa.policy.row-filters-uri` returns filter expressions Trino applies as extra `WHERE` clauses before returning rows, and `opa.policy.column-masking-uri` returns per-column mask expressions; `opa.policy.batch-column-masking-uri` fetches many column masks in one request for wide tables. There is also `opa.policy.batched-uri` for batching resource-filtering decisions. These let OPA drive the same RLS/CLS semantics you would write natively in Postgres.

Q: In a Trino+OPA setup, what does the `opa.policy.batched-uri` endpoint optimise?
A: It optimises bulk filtering decisions, such as which of many catalogs, schemas, or tables a user may list, by sending the full list once and receiving back the indices of permitted items. Without batching, Trino would make one OPA call per resource, which is slow for catalogs with many objects. It reduces round-trips during metadata listing and access checks.

Q: How is catalog-level enforcement (Unity Catalog) fundamentally different from engine-level (Trino)?
A: Unity Catalog enforces at the catalog/metastore layer, so every compute engine that accesses data *through the catalog* inherits the same privilege decisions, whereas Trino enforces only for queries running in Trino. Catalog-level enforcement is engine-agnostic and centralises governance, which prevents a second engine from bypassing rules written for the first. The trade-off is that everything must actually route through the catalog for the guarantee to hold.

Q: Describe Unity Catalog's privilege inheritance model.
A: Privileges are hierarchical — metastore, then catalog, then schema, then tables/views/volumes/functions/models — and a grant at a higher level cascades to current and future children. Granting `SELECT` on a schema therefore covers all its tables, present and future. This inheritance reduces grant sprawl but means you must be deliberate about where in the hierarchy you grant, since a high-level grant is broad.

Q: In Unity Catalog, why is `SELECT` on a table not enough to read it?
A: Because traversal also requires `USE CATALOG` on the parent catalog and `USE SCHEMA` on the parent schema; without those usage privileges the user cannot even reach the table. `USE CATALOG`/`USE SCHEMA` grant the right to *traverse into* the container, not to read its objects, so reading a table needs `SELECT` on the table plus `USE` on each ancestor. Forgetting the `USE` grants is a common "permission denied" cause even when the table grant looks correct.

Q: How does Unity Catalog implement row filters and column masks?
A: It attaches user-defined functions to a table — a row-filter function returning a boolean to gate rows and a column-mask function transforming a column's value — so the catalog applies them for every query regardless of engine. Because the catalog evaluates them centrally, the same RLS/CLS holds whether the query comes from a notebook, SQL warehouse, or external engine going through the catalog. This is the catalog-level analogue of Postgres RLS policies and DDM masks.

Q: "Who actually enforces at the file layer?" — contrast Unity Catalog, Ranger, and Lake Formation on this point.
A: Unity Catalog and AWS Lake Formation broker access to the underlying storage (issuing scoped, temporary credentials), so they can enforce at the file layer for engines that obtain data through them; Apache Ranger primarily enforces at the engine/plugin layer (Hive, Trino, Spark plugins) and does not itself own the object-store credentials. The practical consequence is that with Ranger, anyone with direct object-store access can bypass the policy, whereas credential-vending catalogs aim to close that gap. Always ask which component holds the storage credentials, because that is the real choke point.

Q: What is the residual risk of engine-level enforcement like Trino if the object store is directly readable?
A: The residual risk is bypass: a user or job with direct credentials to the S3/ADLS bucket can read the raw Parquet/ORC files and ignore every Trino row filter and column mask. Engine-level controls only bind queries flowing through the engine, so leaving the bucket broadly readable undermines them entirely. The mitigation is to lock the object store down so the engine (or a credential-vending catalog) is the only path to the bytes.

Q: For the "Done when" item, name where each enforcement layer can be bypassed.
A: Postgres RLS is bypassed by the table owner without `FORCE`, by superusers, and by `BYPASSRLS` roles, and existence can leak via constraint errors. Trino engine-level control is bypassed by anyone with direct object-store access to the underlying files. Catalog-level control is bypassed if an engine reads data without going through the catalog or if the storage credentials are exposed outside the catalog's brokering.

Q: What are the "attribute sources" that drive ABAC decisions on a lakehouse, per this lesson?
A: The two named sources are identity-provider groups (such as Microsoft Entra groups carried as a claim) and catalog tags (classifications attached to columns/tables, such as a `pii` tag). The IdP supplies *who you are and which groups you belong to*; the catalog supplies *what the data is*. An ABAC policy combines them — for example "deny if the column is tagged `pii` and the user is not in the `pii-unmask` group".

Q: How do Entra (IdP) groups reach an enforcement engine like Trino for use in policy?
A: The user authenticates via OIDC/SAML and the IdP issues a token carrying group claims; Trino maps those groups (via its user/group mapping or the token) and includes them in the context document it sends to OPA. OPA's policy then matches the user's groups against the rule. So the group membership lives in Entra, travels in the token, and becomes an attribute the policy evaluates at query time.

Q: How do catalog tags become an ABAC input, and why is that powerful?
A: A governance catalog lets you tag columns/tables with classifications (such as `pii`, `mnpi`, `investor-data`), and policies reference the tag rather than enumerating column names. It is powerful because newly added PII columns inherit protection the moment they are tagged, without editing the policy — the rule "mask anything tagged `pii` for role X" scales automatically. This is far more maintainable than hard-coding column lists that drift as schemas evolve.

Q: Why is tagging PII columns in the catalog a prerequisite for scalable CLS rather than masking column-by-column?
A: Because tag-driven masking lets one policy cover every current and future PII column, whereas column-by-column masks must be re-authored each time the schema changes and inevitably miss new columns. Tags decouple "what is sensitive" (a data classification) from "who may see it" (an access policy). In a fund estate where new investor fields appear regularly, the tag approach is the difference between consistent protection and slow erosion.

Q: What is an access request/review workflow, and why does the lesson call recertification an "audit staple"?
A: It is the controlled process by which access is requested, approved, and then periodically re-reviewed (recertified) to confirm it is still warranted, removing access that is no longer needed. It is an audit staple because regulators expect evidence that access does not accumulate unchecked — reviewers must attest to who can see what, on a schedule. Without recertification, leavers and role-changers keep stale access and the "who could see this in March?" question has no defensible answer.

Q: What four things should a recertification procedure specify?
A: It should specify who reviews access (the accountable owner, such as the fund's data steward or line manager), how often the review runs (for example quarterly for high-sensitivity, annually otherwise), what is reviewed (the actual grants, RLS predicates, and unmask permissions), and what evidence is captured (the reviewer's attestation, date, and any revocations). Naming these turns recertification from a vague intention into an auditable control. The captured evidence is what you show the CSSF reviewer.

Q: Why does recertification need to cover RLS predicates and not just role memberships?
A: Because a stale or over-broad RLS predicate (for example one accidentally widened to all management companies) silently grants the very cross-company visibility the wall is meant to prevent, even if role memberships look correct. Reviewing only roles misses policy-level leaks. So recertification must verify both that the right people hold the right roles *and* that the row/column policies still scope data correctly.

Q: Implement "TA agents see only their management company's funds" as a Postgres policy, including how the agent's company is known.
A: Enable RLS with `ALTER TABLE funds ENABLE ROW LEVEL SECURITY;`, set the agent's company per session via trusted middleware using `set_config('app.mgmtco', '<code>', false)`, then create `CREATE POLICY agent_funds ON funds USING (mgmtco = current_setting('app.mgmtco'));`. The session variable carries the authenticated agent's management company so the policy filters rows to that company. You also run `FORCE ROW LEVEL SECURITY` if the app connects as the table owner.

Q: Implement the same fund-visibility rule as Trino file-based rules over Iceberg — what construct do you use?
A: You use a row filter in the Trino file-based access-control rules file, attaching a filter expression like `mgmtco = current_user_mgmtco` (sourced from the user's mapped attributes/groups) to the `funds` table so Trino appends it as a `WHERE` clause. The rules file maps the user/group to the filter, and Trino enforces it on every query against that Iceberg table. This mirrors the Postgres RLS predicate but at the engine layer over the lakehouse.

Q: How do you demonstrate the same fund-visibility policy at two enforcement layers, and why is that the "Done when" target?
A: You run the rule once in Postgres as an RLS policy and once in Trino as a file-based/OPA row filter over the same Iceberg table, then show an agent querying each gets only their company's funds. The "Done when" target is two layers because regulated platforms rarely have a single engine — proving the wall holds in both the operational database and the analytical lakehouse shows the control is architectural, not incidental to one tool. It also forces you to articulate each layer's bypass risk.

Q: How would you enforce that a restricted role sees masked investor tax IDs while a privileged role sees them in full?
A: Tag the tax-ID column as `pii` in the catalog, define a column mask (DDM `MASKED WITH (FUNCTION='partial(...)')` in SQL Server, a mask function in Unity Catalog, or a view in Postgres) keyed on whether the role holds the unmask privilege/group, and grant `UNMASK` (or membership in the unmask group) only to the privileged role. The restricted role then sees `XXXX`-style output; the privileged role sees the real value. The control is the absence or presence of the unmask grant, which is itself recertified.

Q: An auditor asks you to "prove" the Chinese wall between two management companies. What evidence do you present?
A: You present the RLS policy/row-filter definitions, the mapping from authenticated identity to management-company attribute, a test run showing Agent A querying as B's company returns zero of B's rows, and the recertification record showing the policy was reviewed. Evidence is the policy text plus a reproducible negative test plus the review attestation. This is exactly why the lesson stresses you must *prove*, not merely assert, the control.

Q: Why is "design and prove" emphasised over just "design" for fine-grained access control?
A: Because under CSSF/GDPR review a control that exists but cannot be demonstrated is worthless — the reviewer tests it, and an unprovable wall is treated as no wall. Proving means reproducible negative tests (the wrong user gets nothing), documented bypass analysis, and recertification evidence. Designing without proving leaves you exposed when, not if, the regulator probes.

Q: What is the difference between authentication and authorization, and which one does this lesson focus on?
A: Authentication establishes *who* a principal is (verified identity); authorization decides *what* that principal may do (permitted actions and data). This lesson focuses on authorization at fine grain — RLS, CLS, masking, and the policy models behind them — assuming identity has already been established by an IdP. The two are sequential: you authenticate first, then authorization rules like RLS filter what the now-known principal sees.

Q: Why is a database view sometimes used for column-level security, and what is its weakness versus DDM or catalog masking?
A: A view can expose only a subset of columns (or pre-masked expressions) and you grant access to the view rather than the base table, achieving CLS through projection. Its weakness is maintenance and bypass: you must keep the view in sync with the base table, and anyone with base-table access sidesteps it, whereas DDM/catalog masks bind to the column itself. Views also multiply as roles grow, where tag-driven masking scales with one rule.

Q: An analyst reports they can see an investor's full tax ID that should be masked. Walk through the first checks.
A: First confirm whether masking is even applied to that column (in SQL Server query `sys.masked_columns`, in the catalog check the mask function), then check whether the analyst's role holds `UNMASK` (or membership in an unmask group) that reveals it. Also verify they are not reaching the data via a path that bypasses the mask — a base table, a direct object-store read, or an owner/`BYPASSRLS` connection. The leak is usually a missing mask, an over-granted unmask, or a bypass path.

Q: A TA agent reports they can see another management company's funds. What is the most likely RLS misconfiguration?
A: The most likely causes are that the app connects as the table owner without `FORCE ROW LEVEL SECURITY` (owner bypasses), that the session variable `app.mgmtco` is unset or settable by the user, or that the policy predicate is too broad. Check the connection's role and `BYPASSRLS`, confirm the session context is set by trusted code, and inspect the `USING` expression for an accidental `OR true` or a missing equality. Any one of these silently opens the wall.

Q: Why must the attribute that drives an RLS/ABAC decision be set by trusted infrastructure rather than supplied by the client?
A: Because if the client can set its own management-company attribute, it can claim any company and read everyone's data, defeating the policy entirely. The attribute must be derived from the authenticated session — an IdP claim, a server-set session variable, or a verified token — so the user cannot forge it. This is the core ABAC integrity requirement: trust the attribute source, or the policy is theatre.

Q: Compare enforcing fund visibility in the database (RLS) versus in the engine (Trino) on the dimension of coverage.
A: Database RLS covers every client connecting to that database, including ad-hoc tools, so coverage is broad for that one store but stops at the database boundary. Trino engine enforcement covers analytical queries across many catalogs and file formats but only for traffic that goes through Trino, leaving direct object-store access uncovered. Neither alone is complete in a mixed estate, which is why the lesson requires demonstrating both and naming the gaps.

Q: What is "policy sprawl" and how does fine-grained access control across layers risk it?
A: Policy sprawl is the same logical rule re-implemented in several places (Postgres RLS, Trino rules, catalog masks) that then drift out of sync as they are edited independently. Fine-grained control across layers risks it precisely because you maintain the fund-visibility rule in multiple engines. The forward fix, previewed by the next lesson, is policy-as-code (OPA/Rego) so one authored, tested rule feeds every enforcement point.

Q: How does centralising the rule in OPA reduce the drift created by multi-layer enforcement?
A: With OPA you write the decision once in Rego, version and test it, and have each PEP (Trino, and potentially others) call the same policy decision point, so all layers consult one source of truth. Changing the rule becomes a reviewed PR with passing tests rather than three independent edits that can disagree. This directly addresses the multi-layer drift that pure per-engine RLS/CLS creates.

Q: In OPA/Trino terms, what is the difference between a PDP and a PEP?
A: The PDP (policy decision point) is where the decision is *computed* — OPA evaluating Rego against the request context — and the PEP (policy enforcement point) is where the decision is *applied* — Trino allowing or filtering the query. Trino is the PEP that asks OPA the PDP "may this user select these columns?" and then enforces the answer. Separating decide from enforce lets you change policy without redeploying the engine.

Q: Why might you tag a column `pii` in the catalog even before writing any masking policy?
A: Because tagging records the data classification independently, creating an inventory of sensitive columns that any future policy, audit, or DSAR (data-subject access request) can rely on. The tag is a durable fact about the data; policies that consume it can be added later and will immediately cover everything already tagged. This separation of classification from enforcement is what makes tag-driven CLS scale and what gives auditors a single PII map.

Q: What is the relationship between GLEIF Level-2 ownership data and ReBAC-style access decisions in fund administration?
A: GLEIF Level-2 publishes "who owns whom" entity-ownership relationships keyed by LEI, forming a ready-made graph of parent/subsidiary edges. A ReBAC policy can traverse that graph to decide access — for example permitting a group entity's analyst to see funds owned anywhere down its ownership chain. So the same ownership graph used for entity resolution can back relationship-based authorization decisions.

Q: How might an LEI be used as the join key for an ABAC fund-visibility rule?
A: Each management company and fund can carry the management company's LEI as an attribute, and the policy permits access when `subject.lei == resource.mgmtco_lei`. The LEI is a globally unique, registry-verified identifier, so it is a far more reliable equality key than free-text company names that vary in spelling. Using the LEI as the attribute makes the ABAC predicate unambiguous and auditable against GLEIF.

Q: For a NAV table shared across funds, how would RLS and CLS jointly protect it?
A: RLS would restrict each agent to NAV rows for funds of their own management company via a predicate on the fund's management-company attribute, and CLS would mask or withhold any sensitive components (such as fee breakdowns or large-holder details) from roles not entitled to them. Row security answers "which funds' NAVs", column security answers "which fields of each NAV record". This keeps a single physical NAV table usable by many parties without cross-company leakage.

Q: Why is column-level security on EMT/EPT-style fund data sometimes about *withholding* rather than *masking*?
A: Because some downstream consumers must not even know a sensitive field exists in the file, so the column is removed from their delivered dataset entirely rather than masked to `XXXX`. Masking keeps the schema and hints that data was redacted; withholding changes the projection so the field is absent. The choice depends on whether the consuming process needs a stable schema (mask) or must be blind to the field's existence (withhold).

Q: A regulated report must show the auditor that a restricted role cannot read an investor's tax ID. How do you make this reproducible?
A: Script a test that connects as the restricted role and selects the tax ID, asserting the output is masked or the column is denied, and connects as a privileged role to show the true value, then store both runs as evidence. Reproducibility means the test can be re-run on demand and pinned to the policy version. This turns "trust me" into a repeatable artefact the auditor can execute themselves.

Q: Why does the lesson insist you "name where each layer can be bypassed in writing"?
A: Because a fine-grained control is only meaningful relative to its residual risk, and documenting the bypass (owner/`BYPASSRLS` for Postgres, direct object-store access for Trino, off-catalog reads for the catalog) is what lets reviewers judge whether compensating controls are adequate. Unwritten assumptions about "nobody has direct bucket access" are exactly where breaches originate. Writing it down forces the compensating control (locked storage, audited bypass roles) to exist.

Q: What compensating control closes the "direct object-store access bypasses Trino" gap?
A: Locking the object store so that only the engine or a credential-vending catalog can read the files — users get no standing credentials to the bucket. Then every read must pass through the enforcement layer, restoring the guarantee. Credential-vending catalogs (Unity Catalog, Lake Formation) formalise this by issuing scoped, short-lived credentials only after a policy check.

Q: What is the difference between PERMISSIVE-OR composition and an `OR` inside a single policy's `USING` clause?
A: Multiple `PERMISSIVE` policies combine with OR at the policy level — pass any one and the row is candidate-visible — whereas an `OR` inside one policy's `USING` expression is ordinary boolean logic evaluated within that single predicate. They can produce the same row set but are different mechanisms: one is policy composition, the other is predicate logic. Knowing the difference matters when adding a `RESTRICTIVE` policy, which AND-constrains the whole combined permissive result regardless of internal `OR`s.

Q: Why might `EXPLAIN` show an RLS sub-SELECT executing once per row, and what is the remedy?
A: A policy `USING` clause containing a correlated sub-SELECT (referencing the outer row) cannot be hoisted, so the planner re-evaluates it for every candidate row, often turning an index scan into a slow per-row lookup. The remedy is to rewrite the policy to compare a column directly against a scalar session variable, or to pre-resolve membership into a flat attribute, so the predicate is a simple sargable comparison. Always verify with `EXPLAIN (ANALYZE)` that the sub-SELECT is gone.

Q: How does the choice between RBAC and ABAC affect the recertification burden?
A: With RBAC, recertification reviews role memberships and the (hopefully few) roles' grants; with ABAC, reviewers must also validate the policy logic and the *trustworthiness of attribute sources*, since a wrong attribute silently changes access for everyone. ABAC reduces grant count but shifts review effort to policy and the data quality of attributes. Neither is free — the burden moves from many static grants to fewer but more consequential policies.

Q: What goes wrong if an IdP group used for ABAC is renamed but the policy still references the old name?
A: The policy's group condition stops matching, so users in the renamed group either lose access they should have (fail-closed) or, if the rule is negated, gain access they should not. This is a classic attribute-source drift bug: the policy and the IdP disagree on a string. The defence is to reference stable group IDs rather than display names where possible, and to include group-rename scenarios in policy tests.

Q: Why is "engine-level vs catalog-level vs file-layer" framed as enforcement *reality* rather than theory?
A: Because vendors market governance features that sound airtight, but the actual guarantee depends on which layer is the unavoidable choke point in *your* deployment, and that often differs from the slideware. A catalog only enforces if all engines route through it; an engine only enforces if the storage is locked. The "reality" is that you must trace the real data paths and find where a control can be skipped, not trust the feature name.

Q: How can a column mask leak through a `GROUP BY` or aggregate in an engine that masks naively?
A: If masking is applied only to the displayed value but the engine evaluates predicates or grouping on the *unmasked* column, a user can infer values by grouping or filtering — the same inference attack as SQL Server's range-guessing on DDM. Robust implementations push the mask into the logical plan so even aggregates see masked input, or block ad-hoc predicates on sensitive columns. Naive presentation-only masking is vulnerable to inference, which is why masking alone is never the whole control.

Q: In Trino file-based access control, what is the structural difference between a row filter and a column mask entry?
A: A row filter entry attaches a boolean filter expression to a table for matching users/groups, so Trino appends it as a `WHERE` clause and rows failing it disappear. A column mask entry attaches a transformation expression to a specific column, so Trino rewrites that column's projection to the masked expression. One controls row membership, the other rewrites a value — RLS versus CLS, both authored in the same rules file.

Q: Why is it dangerous to rely on application-layer filtering instead of database/engine RLS for the Chinese wall?
A: Because application-layer filtering only protects access through that one application; any other client — a BI tool, a `psql` session, a leaked connection string — reaches the database directly and sees everything. RLS enforced at the database/engine binds *all* paths to that store, not just the blessed app. For a regulatory Chinese wall you want enforcement as close to the data as possible so no alternate client can route around it.

Q: What is the risk of granting `UNMASK` at database scope versus column scope in SQL Server 2022?
A: Database-scope `UNMASK` reveals *every* masked column in the database the role can select, which is far broader than intended if you only meant to expose one field. Column- or table-scope `UNMASK` (introduced in 2022) lets you reveal precisely one sensitive column to a role, honouring least privilege. The risk of over-scoping is a silent, blanket lifting of masking that an auditor will flag.

Q: How would you test that a new PII column added to a fund table is automatically protected?
A: Tag the new column `pii` in the catalog and run the existing tag-driven mask test as the restricted role, asserting the new column comes back masked without any policy edit. If it returns clear text, either the tag is missing or the masking policy does not consume tags. This test proves the classification-to-enforcement pipeline works for columns the policy author never saw.

Q: Why does the lesson pair Postgres RLS (operational DB) with Trino over Iceberg (lakehouse) specifically?
A: Because a real fund estate has both an operational transactional store and an analytical lakehouse, and the same Chinese-wall rule must hold in both or data leaks through the unprotected one. Postgres RLS is the first enforcement archetype (in-database), Trino is the engine-level archetype (over open table formats). Demonstrating both shows the control is portable across the two dominant enforcement styles you will actually run.

Q: A new analyst is added to the `fund-readers` group but still gets permission-denied in Unity Catalog. First checks?
A: Confirm the group has `SELECT` on the table *and* `USE SCHEMA` and `USE CATALOG` on the parents, since missing traversal privileges deny access even with a valid table grant. Then verify the analyst's group membership actually propagated from the IdP to the catalog, and that no higher-level deny or missing inheritance applies. The usual culprit is forgotten `USE CATALOG`/`USE SCHEMA` grants.

Q: Why is "the same policy at two enforcement layers" stronger evidence to a regulator than one layer plus a promise?
A: Because it demonstrates the control is an architectural invariant that survives across the platform's actual access paths, not a single tool's feature that an alternate route could skip. The regulator's concern is whether a real user, by any path, can cross the wall; two demonstrated layers narrow the unprotected paths and the written bypass analysis names the rest. One layer plus a verbal promise leaves the other path unproven and therefore presumed open.

Q: Summarise how RBAC, ABAC, and ReBAC each express "TA agents see only their management company's funds".
A: RBAC mints a role per management company (`ta_lu123_reader`) and grants it the matching funds — simple but explodes with company count. ABAC writes one rule comparing the agent's management-company attribute to the fund's (`subject.mgmtco == fund.mgmtco`), covering all companies with a single policy. ReBAC permits access when a relationship path exists, such as the agent being assigned to the company that owns the fund in the ownership graph — best when assignment is multi-hop or inherited down an ownership chain.

Q: What is the danger of a SWIFT/payment instruction table being readable across the Chinese wall, and how does CLS help?
A: A payment-instruction table can expose counterparties, account identifiers, and amounts that reveal another client's activity, which crosses the confidentiality wall and may even constitute market-sensitive information. CLS helps by masking or withholding the sensitive identifier columns from roles outside the owning company while RLS removes the rows entirely for unrelated companies. The combination ensures even an accidental query returns neither the rows nor the sensitive fields.

Q: Why does the lesson treat the *attribute source*, not just the policy, as part of the access-control design?
A: Because an ABAC/ReBAC decision is only as trustworthy as the attributes feeding it — a forged group claim or a stale ownership edge produces wrong-but-confident decisions. The design must therefore include where attributes come from (IdP groups, catalog tags, GLEIF ownership), how they are kept current, and who can change them. A perfect policy over bad attributes is still a broken wall.

Q: What is the practical first step when asked to implement fine-grained access control on an existing shared fund table?
A: Identify the discriminating attribute that defines the wall (usually the management-company code or LEI) and confirm it exists, is accurate, and is present on every row, since the entire RLS predicate depends on it. Then establish how the authenticated principal's matching attribute will be supplied to the session in a way the user cannot forge. Only after the attribute and its trusted source are sorted do you write the policy, because a policy keyed on a missing or untrusted attribute is worthless.

Q: What is the single most important takeaway connecting all of 11.1.2's enforcement layers?
A: That a fine-grained control is only as strong as the lowest layer that cannot be bypassed, so you must enforce as close to the data as your estate allows, prove it with reproducible negative tests at every layer data flows through, and recertify it on a schedule. RLS, CLS, masking, the RBAC/ABAC/ReBAC ladder, and engine/catalog/file enforcement are all instruments toward one goal: a Chinese wall that survives audit. Design it, prove it, document its residual bypass, and review it — anything less is a reportable breach waiting to happen.

Q: What command type and target role does a Postgres `CREATE POLICY` default to if you omit `FOR` and `TO`?
A: It defaults to `FOR ALL` (the policy applies to `SELECT`, `INSERT`, `UPDATE`, and `DELETE`) and to `TO PUBLIC` (it applies to every role). So `CREATE POLICY p ON funds USING (...)` silently scopes itself broadly, which is fine for a single per-tenant wall but a gotcha if you intended a read-only `FOR SELECT` rule. Always state `FOR` and `TO` explicitly when the policy should be narrower than the defaults.

Q: Which Postgres catalog view lets you inspect every RLS policy's predicate, and what does it show?
A: The `pg_policies` view exposes one row per policy with columns including `schemaname`, `tablename`, `policyname`, `permissive`, `roles`, `cmd`, `qual` (the `USING` expression), and `with_check` (the `WITH CHECK` expression). Querying `SELECT * FROM pg_policies WHERE tablename = 'funds';` is the fastest way to audit what predicate is actually live. The underlying catalog table is `pg_policy`, but `pg_policies` renders the expressions as readable text.

Q: How do you confirm at the catalog level that RLS is even enabled and forced on a Postgres table?
A: Check `relrowsecurity` (RLS enabled) and `relforcerowsecurity` (FORCE applied) on the table's `pg_class` row, e.g. `SELECT relrowsecurity, relforcerowsecurity FROM pg_class WHERE relname = 'funds';`. If `relrowsecurity` is false, the `CREATE POLICY` statements exist but are inert and everyone sees all rows. This is the first check when a wall you "configured" is not actually filtering.

Q: A policy exists in `pg_policies` but rows are still unfiltered for a restricted role. First thing to check?
A: Confirm `relrowsecurity` is true on the table, because a policy can exist while row security is disabled, making the policy dormant. Run `SELECT relrowsecurity FROM pg_class WHERE relname = '...';`; if false, the `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` step was missed or was reverted. Creating a policy does not implicitly enable RLS — the two are separate operations.

Q: How does a `FOR SELECT` policy differ from a `FOR ALL` policy when a user tries to `UPDATE` a row?
A: A `FOR SELECT` policy gates only read visibility and contributes nothing to `UPDATE`, so without a separate `UPDATE` (or `ALL`) policy the update is denied because no policy permits it. A `FOR ALL` policy covers the read-side via `USING` and the write-side via `WITH CHECK` (defaulting to `USING`) in one statement. Splitting into per-command policies lets you allow reads while forbidding writes, which `FOR ALL` cannot express alone.

Q: You want TA agents to read all their funds but only update a subset flagged `editable`. How do you structure the policies?
A: Create a `FOR SELECT` policy with `USING (mgmtco = current_setting('app.mgmtco'))` for read visibility, plus a separate `FOR UPDATE` policy with `USING (mgmtco = current_setting('app.mgmtco') AND editable)` and a matching `WITH CHECK`. The read policy exposes the whole company's funds while the update policy narrows writable rows to the editable ones. Per-command policies are the idiomatic way to grant asymmetric read versus write scopes.

Q: In `pg_policies`, the `permissive` column reads `PERMISSIVE` for a rule you intended as restrictive. What went wrong and what is the fix?
A: The `AS RESTRICTIVE` clause was omitted, so Postgres created the default `PERMISSIVE` policy, which combines with OR and therefore widens rather than constrains access. A permissive policy can never tighten the result, so your "hard constraint" is doing nothing or even opening rows. Drop and recreate it with `CREATE POLICY ... AS RESTRICTIVE ...` so it AND-combines as a true guardrail.

Q: Write a `RESTRICTIVE` policy that blocks any access to investor rows flagged `do_not_disclose`, regardless of other policies.
A: `CREATE POLICY hide_flagged ON investors AS RESTRICTIVE USING (NOT do_not_disclose);` adds an AND-constraint that every row must satisfy on top of passing some permissive policy. Because restrictive policies combine with AND, no permissive rule can re-expose a flagged row. This is the pattern for a regulator-mandated hard exclusion that must survive other policy edits.

Q: Why does using `current_setting('app.mgmtco')` without the missing-OK flag risk an error, and how do you avoid it?
A: `current_setting('app.mgmtco')` raises `unrecognized configuration parameter` if the variable was never set in the session, which can make every query error rather than return rows. Pass the second argument as true — `current_setting('app.mgmtco', true)` — to return `NULL` instead of erroring when unset. You then design the predicate so a `NULL` context matches no rows, giving fail-closed behaviour without query crashes.

Q: How would you set the per-session management-company context from a PgBouncer-pooled application safely?
A: Because PgBouncer in transaction-pooling mode reuses server connections, set the variable inside the same transaction as the query, e.g. `SET LOCAL app.mgmtco = 'LU123';` issued by trusted middleware right before running the user's statements. `SET LOCAL` scopes the value to the current transaction so it cannot leak to the next borrower of that pooled connection. Setting it with plain `SET` in transaction-pooling mode is the classic bug that bleeds one tenant's context into another's query.

Q: What is the difference between `SET app.mgmtco` and `SET LOCAL app.mgmtco` for RLS in a pooled environment, and why does it matter?
A: `SET` persists the value for the whole session, which in a transaction-pooled setup outlives the request and can attach the wrong tenant's context to a later, unrelated transaction on the same physical connection. `SET LOCAL` confines the value to the current transaction and is rolled back at commit, so each request must re-establish its own context. In pooled architectures `SET LOCAL` (or `set_config(..., true)` with `is_local = true`) is the safe choice.

Q: How can you let a backup role read all rows without granting full superuser, given RLS is enabled?
A: Grant that role the `BYPASSRLS` attribute with `ALTER ROLE backup_svc BYPASSRLS;`, which lets it bypass every row policy for dumps without conferring superuser powers like file access or role management. It is narrower than superuser but still punches through all Chinese walls, so it must be a named, audited, minimally used login. Pair it with `pg_dump` running as that role to produce complete backups.

Q: `pg_dump` run by a non-bypass role produced a suspiciously small dump file. What happened and how do you make the failure loud?
A: RLS silently filtered the dump to only the rows that role could see, producing a partial backup that looks successful. Set `SET row_security = off;` (or run `pg_dump` with a role that has `BYPASSRLS`) so that a role unable to bypass RLS gets an error instead of a silently filtered result. Turning the silent partial dump into a hard error prevents shipping an incomplete backup unnoticed.

Q: What is the security-barrier property of RLS `USING` expressions, and how does it relate to leakproof functions?
A: Postgres treats the RLS `USING` predicate as a security barrier, meaning the planner must not push a user-supplied, non-leakproof condition ahead of it where that condition could observe filtered-out rows. Only functions marked `LEAKPROOF` may be evaluated before the barrier, since they are guaranteed not to leak argument values via errors or timing. This ordering is what prevents a crafted predicate from inferring the existence of rows RLS is hiding.

Q: How can a `UNIQUE` constraint violation leak the existence of an RLS-hidden row, and what is the mitigation?
A: Constraint checks bypass RLS to keep integrity sound, so inserting a value that collides with a hidden row raises a `duplicate key value violates unique constraint` error, confirming the hidden row exists. The mitigation is to treat RLS as protecting content visibility, not existence, and to use surrogate keys or salted/opaque natural keys so a collision reveals nothing meaningful. Where existence must be hidden, partition the data physically per tenant rather than relying on RLS over a shared table.

Q: What does `EXPLAIN` reveal about an RLS predicate, and what plan node signals the policy is being applied?
A: `EXPLAIN` shows the policy's `USING` expression merged into the scan's `Filter` (or as an index condition if sargable), so you see the extra predicate attached to the table scan even though you never typed it. If the policy compares a column to a scalar session variable and the column is indexed, you want an `Index Scan` with the predicate as an index condition. A `Seq Scan` with the policy in `Filter` on a large table is the red flag to optimise.

Q: Rewrite a slow RLS policy `USING (mgmtco IN (SELECT code FROM agent_companies WHERE agent = current_user))` to be sargable.
A: Resolve the agent's company once into a session variable at login — `SET LOCAL app.mgmtco = (lookup)` done by trusted middleware — and change the policy to `USING (mgmtco = current_setting('app.mgmtco', true))`, a direct indexed equality. The original correlated sub-SELECT re-runs per candidate row and defeats the index, whereas the flattened comparison lets the planner use an index on `mgmtco`. Confirm with `EXPLAIN (ANALYZE)` that the sub-plan is gone.

Q: How do you grant column-level read access in Postgres natively, and what is its limitation versus value masking?
A: Use `GRANT SELECT (col_a, col_b) ON investors TO role;` to permit reading only the listed columns, and `REVOKE` to remove a column, so the role simply cannot select the omitted ones. The limitation is that column `GRANT` removes the column entirely rather than masking its value, so an application expecting a stable schema breaks, and it offers no partial reveal like `XXXX1234`. For value-preserving masks you need a view, a generated/expression column, or a tool like DDM.

Q: How would you implement value masking of a tax ID in Postgres without an external tool?
A: Create a view over the base table that projects a masking expression for unprivileged roles, e.g. `CASE WHEN pg_has_role(current_user, 'pii_unmask', 'USAGE') THEN tax_id ELSE 'XXXXX' || right(tax_id, 4) END AS tax_id`, then grant access to the view and revoke direct base-table access. The view keeps the column name stable so applications still bind to `tax_id` while unprivileged roles see the masked form. The weakness is that anyone retaining base-table `SELECT` bypasses the view, so base-table grants must be locked down.

Q: A Postgres masking view works for the app but an analyst sees clear text. What bypass path should you check first?
A: Check whether the analyst still holds `SELECT` on the underlying base table, because a view-based mask is bypassed by anyone who can query the base table directly. Run `\dp investors` (or query `information_schema.role_table_grants`) to see who has base-table privileges, and revoke them so the view is the only path. View masking only works if base-table access is exclusively through the view.

Q: In SQL Server, which catalog view lists masked columns and the function applied, and why query it during an incident?
A: `sys.masked_columns` lists each masked column with its `masking_function` (e.g. `partial(0,"XXXX",4)`), letting you confirm whether masking is actually configured on the column an analyst claims to have read in clear. During a suspected leak this is the first query: if the column is absent from `sys.masked_columns`, no mask exists and the column was never protected. If it is present, the leak is an over-granted `UNMASK` or a bypass path, not a missing mask.

Q: Construct a SQL Server DDM `partial` mask that shows the last four digits of a tax ID and hides the rest.
A: `ALTER TABLE investors ALTER COLUMN tax_id ADD MASKED WITH (FUNCTION = 'partial(0,"XXXXX",4)');` exposes zero leading characters, substitutes the literal `XXXXX`, and reveals the final four characters. The `partial(prefix, padding, suffix)` form lets you tune exactly how many edge characters remain visible. Unprivileged roles then see `XXXXX6789` while `UNMASK` holders see the true value.

Q: What is the difference between SQL Server Dynamic Data Masking and Always Encrypted for protecting a tax ID column?
A: DDM masks values only in result sets for unprivileged readers while the real data sits unencrypted at rest, so it defends against casual exposure but not against anyone with file/backup access or range-guessing queries. Always Encrypted encrypts the column so the engine never sees plaintext and even a DBA cannot read it, at the cost of restricted query operations on the column. For regulated PII at rest you reach for encryption (or tokenisation); DDM is a presentation-layer convenience layered on top.

Q: A user with `SELECT` but not `UNMASK` runs `WHERE tax_id = '123456789'` against a DDM-masked column. Does it match, and what does that prove?
A: Yes, the predicate matches because DDM masks only the displayed output, not the value used to evaluate `WHERE`, so filtering and joins still operate on the true underlying data. This proves DDM is presentation-only and vulnerable to inference: a determined user can confirm or brute-force values through predicates despite the mask. It is exactly why Microsoft says DDM is not a substitute for real access control or encryption.

Q: How do you grant `UNMASK` on just one column in SQL Server 2022, and why prefer that over database scope?
A: Use granular `GRANT UNMASK ON investors(tax_id) TO role;` (column-scoped, introduced in SQL Server 2022) so the role sees only that one column unmasked. Prefer it over `GRANT UNMASK TO role;` at database scope, which lifts masking on every masked column the role can select — a blanket reveal that violates least privilege. Column-scoped unmask is the least-privilege way to expose exactly one sensitive field.

Q: In Trino's OPA plugin, what JSON node carries the batched list of resources for `opa.policy.batched-uri`, and what does OPA return?
A: Trino places the candidate resources under the `action.filterResources` node in a single request to the batched endpoint, and the OPA policy returns the indices (or subset) of the resources the user may access. This lets Trino filter, say, hundreds of tables in one round-trip instead of one call per object. Without `opa.policy.batched-uri` configured, Trino falls back to a separate request per resource, which is slow for metadata listing.

Q: A Trino query is denied with the OPA plugin even though the Rego looks correct. What two endpoints and flags help you debug?
A: Enable `opa.log-requests=true` and `opa.log-responses=true` so Trino logs the exact JSON context it sent and the decision OPA returned, revealing whether the user, groups, or resource fields differ from what the policy expects. Compare that logged input against the policy's package/rule at `opa.policy.uri`, and test the same input directly against OPA's Data API. The mismatch is usually a group name or catalog/schema field the policy keys on but Trino populated differently.

Q: How does Trino decide row filters versus column masks through OPA, and which property governs each?
A: `opa.policy.row-filters-uri` is queried to obtain boolean filter expressions Trino appends as additional `WHERE` clauses (RLS), and `opa.policy.column-masking-uri` is queried to obtain per-column mask expressions Trino substitutes into the projection (CLS). Each returns an expression string rather than a simple allow/deny, so the policy author writes the SQL fragment Trino injects. `opa.policy.batch-column-masking-uri` fetches many masks at once for wide tables.

Q: In Trino file-based access control, give the JSON shape of a row filter that limits `funds` to the user's management company.
A: Under the `tables` array you add an entry such as `{"catalog": "iceberg", "schema": "fund", "table": "funds", "filter": "mgmtco = current_user_mgmtco()", "filter_environment": {"user": "admin"}}`, where the `filter` string is appended as a `WHERE` clause for matching users. The expression must be valid Trino SQL and may call functions or reference session-derived values. Trino enforces it on every query against that table that flows through the engine.

Q: A BI tool reads masked Trino data correctly, but a Spark job on the same Iceberg table sees clear text. Why?
A: Spark reading the Iceberg files (or via its own catalog) bypasses Trino entirely, so Trino's engine-level row filters and column masks never apply — they bind only to queries flowing through Trino. The control is engine-scoped, and a second engine is a parallel, unprotected path. The fix is catalog-level enforcement (so both engines inherit the rule) or locking storage so only credential-vended paths can read the bytes.

Q: In Unity Catalog, which privilege lets a steward apply or edit a `pii` tag on a column, and what else is required?
A: The `APPLY TAG` privilege on the object permits adding and editing tags, including column-level tags on tables and views. The steward must additionally hold `USE CATALOG` on the parent catalog and `USE SCHEMA` on the parent schema to traverse to the object. Without `APPLY TAG` the steward can read the data but cannot classify it, breaking the tag-driven CLS pipeline at the source.

Q: What does the Unity Catalog `BROWSE` privilege grant, and why is it useful for a request-access workflow?
A: `BROWSE` lets a principal discover an object's existence and view its metadata — name, description, tags — without `USE CATALOG`/`USE SCHEMA` and without any access to the underlying data. It is useful for an access-request workflow because users can find and request access to a dataset they cannot yet read, supporting self-service governance. It deliberately exposes metadata only, so the Chinese wall over data content stays intact.

Q: In Unity Catalog, how does the `MANAGE` privilege differ from object ownership, and why does that matter at recertification?
A: `MANAGE` lets a principal grant/revoke privileges, transfer ownership, and delete an object without being the owner, effectively conferring administrative control short of the ownership flag itself. At recertification this matters because a `MANAGE` grant is as powerful as ownership for access decisions and must be reviewed with the same scrutiny — a forgotten `MANAGE` is a backdoor to widen any wall. List `MANAGE` holders alongside owners in the review.

Q: A Unity Catalog row-filter UDF returns rows for the wrong tenant after a refactor. Where do you look first?
A: Inspect the row-filter function's body and the column it is bound to via the table's properties, since the UDF must return a boolean keyed on the caller's identity (e.g. `is_account_group_member`) compared to the tenant column. A refactor often breaks the identity function call or compares against the wrong column. Re-run the function with a known restricted principal and assert it returns false for another tenant's rows before re-binding.

Q: Contrast how Unity Catalog and Apache Ranger relate to the object store, and what that means for the file-layer bypass.
A: Unity Catalog brokers storage access by vending scoped, short-lived credentials only after a policy check, so it can enforce at the file layer for engines that obtain data through it; Apache Ranger enforces via engine plugins (Hive, Trino, Spark) and does not itself hold the object-store credentials. Consequently, with Ranger anyone holding direct bucket credentials bypasses the policy entirely, whereas the credential-vending catalog narrows that path. Always ask which component owns the storage credentials to find the real choke point.

Q: How do AWS Lake Formation's credential-vending and column filtering close the gap that engine-only enforcement leaves open?
A: Lake Formation registers the data location and vends temporary, scoped credentials to integrated engines only after evaluating its table/column/row permissions, so an engine cannot read more than the policy allows even at the storage layer. It supports column filtering and row-level filters (cell-level security via data filters) enforced as data is fetched. Because the engine receives credentials scoped by the policy, direct broad bucket access is removed as a bypass for that path.

Q: How would an Entra (IdP) group claim drive an ABAC deny on a `pii`-tagged column end to end?
A: The user authenticates via OIDC and Entra issues a token carrying group claims; the engine maps those groups and includes them in the context sent to the policy engine, while the catalog supplies the column's `pii` tag, and the policy denies or masks if the column is tagged `pii` and the user's groups lack `pii-unmask`. Identity (groups) comes from Entra, classification (tag) comes from the catalog, and the policy combines them. Newly tagged PII columns are protected automatically without editing the group list.

Q: Why prefer Entra group object IDs over display names in an ABAC policy, and what failure does this prevent?
A: Group display names can be renamed by an admin while the immutable object ID stays constant, so keying the policy on the object ID survives a rename whereas a name-based rule silently stops matching. The prevented failure is attribute-source drift: a renamed group causes users to lose access they should keep (or, with negated rules, gain access they should not). Reference the stable ID and keep a documented mapping from ID to human-readable name for auditors.

Q: How does tag-driven masking automatically protect a newly added EET/PAI sustainability field that contains personal data?
A: Because the masking policy targets the `pii` tag rather than enumerated column names, tagging the new field `pii` in the catalog makes the existing "mask anything tagged `pii` for non-entitled roles" rule cover it the instant the tag is applied — no policy edit needed. This matters as regulatory templates like the EET add fields over time and manual column lists inevitably miss new ones. The control scales with classification, not with schema churn.

Q: Under T+1 settlement, why does shortened processing time raise the stakes on getting RLS context-setting right?
A: With T+1, NAV, reconciliation, and transfer-agency processing compress into a tighter window, so a misconfigured RLS context that fails closed (returns no rows) or fails open (leaks a tenant) is discovered and must be fixed under far more time pressure than under T+2. A pooled-connection context bug that intermittently returns the wrong tenant's funds is both a confidentiality breach and a settlement-timeline risk. The mitigation is deterministic per-transaction context (`SET LOCAL`) and pre-deployment negative tests so the wall never becomes the bottleneck.

Q: How might the EU AI Act bear on column-level access to investor PII used to train or run a model?
A: The EU AI Act imposes data-governance obligations on high-risk AI systems, including controls over the personal data feeding them, so a model consuming investor PII should receive masked or minimised fields unless an entitlement justifies the raw data. CLS that withholds or masks `pii`-tagged columns from the model's service principal enforces data minimisation at the access layer, supporting AI-Act and GDPR documentation. The access policy thus becomes part of the model's governance evidence, not just a database concern.

Q: For an EMT file delivered to two different distributors, when would you withhold a column rather than mask it, and why?
A: Withhold (omit the column from the delivered projection) when a distributor must not even know the field exists in the dataset — for example an internal cost or large-holder field — because masking to `XXXX` still signals its presence and schema position. Masking is preferable when the consuming process needs a stable EMT schema and only the value must be hidden. The choice hinges on whether the consumer requires a fixed schema (mask) or must be blind to the field (withhold).

Q: A SWIFT payment-instruction table is shared across management companies; design the RLS and CLS to prevent cross-wall leakage.
A: Apply RLS with `USING (mgmtco = current_setting('app.mgmtco', true))` so each company sees only its own instruction rows, and apply CLS to mask or withhold counterparty account identifiers and amounts from any role outside the owning company. RLS removes the other company's rows entirely while CLS ensures even an accidental cross-join or mislabeled row exposes no sensitive identifiers. Together they prevent both row-level and field-level disclosure of another client's payment activity, which can be market-sensitive.

Q: How would you use the GLEIF Level-2 ownership graph to implement a ReBAC rule "see funds owned anywhere down my group's chain"?
A: Model parent/subsidiary edges keyed by LEI from GLEIF Level-2 as a graph, then evaluate access by checking whether a path exists from the user's group entity LEI down to the fund's owning entity LEI, granting access if such an ancestor relationship holds. This is genuine ReBAC because the decision is multi-hop traversal, not a flat attribute equality. The same ownership graph used for entity resolution backs the authorization, and the LEI gives each node a registry-verified identifier.

Q: Write a reproducible negative-test script outline that proves Agent A cannot see Management Company B's funds in Postgres.
A: Connect as the restricted role, set the session context to A's company (`SET LOCAL app.mgmtco = 'LU_A'`), run `SELECT count(*) FROM funds WHERE mgmtco = 'LU_B'` and assert the result is `0`, then optionally set context to B and assert B's rows are visible, capturing both outputs with the policy version. The test must run as the role the app actually uses (not the owner) and confirm `FORCE ROW LEVEL SECURITY` is on if the app is the owner. Storing the run pinned to the policy commit turns the wall from an assertion into an executable artefact for the auditor.

Q: What evidence package would you hand a CSSF reviewer to demonstrate the fund-visibility wall at both Postgres and Trino layers?
A: Provide the `pg_policies` dump showing the live `USING` predicate plus `relrowsecurity`/`relforcerowsecurity` flags, the Trino rules file (or Rego) with the matching row filter, the identity-to-`mgmtco` mapping, reproducible negative-test runs at each layer showing zero cross-company rows, and the recertification record with reviewer attestation and date. Together these show the same rule enforced at two layers, the trusted attribute source, and that the control is reviewed. The written residual-bypass analysis (owner/`BYPASSRLS` for Postgres, direct object-store access for Trino) completes the package.

Q: A quarterly recertification finds an RLS predicate was widened to `OR true` during an incident and never reverted. How is this caught and what should the procedure capture?
A: It is caught because recertification reviews the actual `qual` text in `pg_policies`, not just role memberships, so the stray `OR true` (which makes every row visible) is visible to the reviewer. The procedure should capture the reviewer's name, the date, the exact predicate reviewed, any deviation found, and the remediation ticket, so the rollback is evidenced. This is precisely why recertification must inspect policy predicates and not assume correct roles imply a correct wall.

Q: Why is recertifying `BYPASSRLS` roles and Unity Catalog `MANAGE`/owner grants as important as recertifying ordinary read roles?
A: Because `BYPASSRLS` roles and `MANAGE`/owner grants can each defeat or rewrite the access controls themselves, so a stale one is a standing backdoor through every Chinese wall, far more dangerous than a stale read grant that only over-exposes one dataset. Recertification must enumerate these high-privilege principals, confirm each is still warranted, and revoke the rest with evidence. Reviewing only ordinary readers while ignoring the roles that can bypass the wall leaves the most consequential access unexamined.

Q: An access-request workflow grants `UNMASK` for a one-off investigation. What control prevents this becoming permanent over-exposure?
A: Time-bound the grant (an expiry or scheduled review) and ensure the next recertification cycle re-examines all `UNMASK` holders, so a temporary investigative reveal is removed rather than silently retained. The request record should state the justification and end date, and the recertification step should flag any `UNMASK` past its purpose. Without expiry plus recertification, one-off reveals accrete into a blanket loss of masking that an auditor will flag.

Q: Summarise the single discriminating-attribute pre-flight check before writing any fund-visibility RLS policy.
A: Confirm the wall's discriminating attribute (management-company code or LEI) exists on every row, is accurate, and is non-null, because the entire `USING` predicate keys on it and a missing or wrong value silently mis-scopes access. Then confirm how the authenticated principal's matching attribute reaches the session via trusted, un-forgeable means. Only once the attribute and its trusted source are verified do you author the policy, since a policy over a missing or untrusted attribute is worthless.
