# Phase 7 · 6.6.3 Managed RAG platforms — 150 self-test questions

Q: What is a "managed RAG platform" and how does it differ from the open RAG pipeline you built in 6.6.2?
A: A managed RAG platform is a cloud service (Azure AI Search, Amazon Bedrock Knowledge Bases, Vertex AI Search) that bundles ingestion, chunking, embedding, indexing, retrieval, and reranking behind a single managed API, so you operate a service rather than a stack. Your open pipeline in 6.6.2 stitched together your own chunker, embedder, `pgvector`/vector store, and reranker, giving you full control but full ops responsibility. The trade is convenience and integrated security versus lock-in and opacity over the pieces the platform now hides.

Q: Why is "just use Azure AI Search + OpenAI" the default corporate pitch for RAG?
A: It is a single-vendor, fully-managed answer that satisfies procurement, security, and ops with one Azure SKU and Microsoft Entra integration, so it removes the burden of running your own vector store and embedding service. The risk is accepting it without measuring its real capability and cost envelope: you may pay the semantic-ranker meter for a corpus `pgvector` served fine, or you may reject the one managed feature (security trimming) compliance actually required. The lesson's goal is to size that envelope before deciding.

Q: How does Azure AI Search define "hybrid search"?
A: Azure AI Search defines hybrid search as running a vector (similarity) query and a keyword (full-text) query in the same request, where the two execute in parallel and the results are merged into a single ranked response. Vector support is at the field level, so one query can target both vector and nonvector fields in the same index. Hybrid often beats either vector-only or keyword-only retrieval because it captures both semantic likeness and exact-term matches.

Q: What problem does combining vector and keyword search in one query solve?
A: Vector search captures conceptual likeness ("dog" matches "canine") but can miss exact tokens, while keyword/BM25 search nails exact terms and codes but misses paraphrase. Hybrid runs both and fuses them, so a query for an ISIN like `LU0123456789` still hits on the literal code via keyword while a query about "money-market sub-fund liquidity" still hits semantically. For fund documents full of exact identifiers plus prose, that combination matters.

Q: What is Reciprocal Rank Fusion (RRF) and where does Azure AI Search use it?
A: RRF is the algorithm Azure AI Search uses to merge the separately ranked vector and keyword result lists in a hybrid query into one unified ranking. It scores each document by the reciprocal of its rank position in each list (so high-ranked items in either list bubble up), then sums those contributions, which is robust because it uses ranks rather than incomparable raw scores. The fused RRF-ranked set is what feeds the response, and what the semantic ranker reranks if enabled.

Q: Why does RRF use rank position rather than the raw relevance scores from each retriever?
A: Vector cosine/dot-product scores and BM25 keyword scores live on entirely different, non-comparable scales, so summing raw scores would let one retriever dominate arbitrarily. RRF sidesteps this by using only each document's rank within its own list and combining reciprocals of those ranks, making the fusion scale-free and stable. This is why hybrid+RRF is a sensible default when you cannot calibrate the two retrievers against each other.

Q: In a hybrid query, do the vector and keyword searches run sequentially or in parallel?
A: They run in parallel — Azure AI Search executes the vector search and the keyword (full-text) search simultaneously over the same index, then merges the two result sets with RRF. Running them concurrently keeps latency close to that of a single search rather than adding the two together. The merged, ranked response is then optionally handed to the semantic ranker.

Q: Can a single Azure AI Search query filter by metadata at the same time as doing vector search?
A: Yes — a request can include a vector query plus an OData filter expression on text or numeric fields, so you can restrict the vector search to, say, a fund family, document type, or language. The vector field itself is not filterable, but you set up a filterable text/numeric field alongside it. The engine can apply the filter before or after the vector query (pre- vs post-filtering), which has performance and recall implications.

Q: What does Azure mean by "vector support is implemented at the field level"?
A: It means each vector is stored in its own dedicated vector field in the index, and an index can hold both vector and nonvector fields side by side on the same documents. A query targets specific vector fields for similarity while keyword search targets searchable text fields, all in one index. This field-level design is what makes hybrid queries over one index possible.

Q: Is vector search itself a paid add-on in Azure AI Search?
A: No — the docs state vector search is available in all regions and on all tiers at no extra charge; you pay the normal service tier cost, not a separate vector fee. However, generating the embeddings (for example calling Azure OpenAI) or using AI enrichment for vectorization can incur charges from the model provider. The genuinely metered premium add-on is the semantic ranker, not vector search itself.

Q: What is "integrated vectorization" in Azure AI Search?
A: Integrated vectorization is Azure AI Search performing the chunking and embedding inside the indexer pipeline, rather than you generating vectors externally and pushing them in. You supply a data source, an indexer, and a skillset that drives splitting documents into chunks and calling an embedding model (for example Azure OpenAI), and the service handles the transitions. The convenience is real, but it is bought with opacity over exactly how your content was chunked and embedded.

Q: What three components are required to use integrated vectorization?
A: An indexer, a supported data source (such as Azure Blob Storage, Cosmos DB, ADLS Gen2, or OneLake), and a skillset that drives the chunking and embedding. The indexer pulls from the data source, the skillset's skills split text and call the embedding model, and the resulting vectors land in vector fields in the index. Without the skillset there is no chunking/embedding step, so you would have to provide vectors yourself.

Q: What does integrated vectorization "hide," and why does that matter when retrieval quality degrades?
A: It hides the exact chunking boundaries, chunk overlap, and embedding-call details inside the managed indexer/skillset, so when answers go wrong you cannot trivially inspect or tune how a document was split or embedded. In your open 6.6.2 pipeline those parameters are explicit and adjustable; here a bad chunk strategy is a config buried in a managed skill. That opacity is the core trade you must name: convenience now, harder root-cause analysis when retrieval quality drops.

Q: A fund prospectus returns the wrong section as a citation under Azure integrated vectorization. Why is this harder to debug than in your open pipeline?
A: Because the chunk boundaries that produced that section were decided by the managed skillset, not by code you can read line by line, so you cannot immediately see whether the offending text was split mid-table, dropped its heading context, or merged two sub-funds. In the open pipeline you would inspect and re-run your chunker with different size/overlap settings directly. With integrated vectorization you must reverse-engineer the skill configuration and re-index to test changes, which is slower.

Q: What is the alternative to integrated vectorization if you want full control over chunking?
A: External processing: generate embeddings outside Azure AI Search (your own chunker plus embedding calls) and push the pre-vectorized content directly into vector fields in the index using the push API. This gives you the same explicit control you had in 6.6.2 while still using Azure AI Search as the vector store and hybrid/semantic query engine. You trade away the indexer convenience to regain chunking transparency.

Q: What is "security trimming" in Azure AI Search, and why is it the enterprise killer feature for fund documents?
A: Security trimming is filtering search results so a user only ever retrieves documents their identity is allowed to see, by storing principal (group/user) identifiers on each document and filtering on them at query time. For investor-restricted fund documents — where investor A must never see investor B's holdings or a restricted share-class memo — this is the compliance control that makes RAG usable at all. It is often the single requirement that flips a "build vs buy" decision toward the managed platform.

Q: In the Azure AI Search security-filter pattern, what field type holds the principal identifiers and what attributes does it need?
A: It uses a `Collection(Edm.String)` field (a list of strings, e.g. `group_ids`) with `filterable` set to `true` so it can be used in filters, and `retrievable` set to `false` so the identity list is never returned in results. Each indexed document carries the set of group/user IDs allowed to read it. Making it non-retrievable prevents leaking the ACL itself in the API response.

Q: What OData filter does the security-trimming pattern use at query time?
A: It uses `group_ids/any(g:search.in(g, 'group_id1, group_id2, ...'))`, where the listed IDs are the groups the requesting user belongs to. The `any` lambda over the collection field matches documents whose `group_ids` contains at least one of the caller's groups. This returns exactly the documents the user has read access to and trims the rest.

Q: Why is `search.in` preferred over a chain of `Id eq 'x' or Id eq 'y'` equality expressions for security filters?
A: A long disjunction of equality expressions is error-prone, hard to maintain, and — with hundreds or thousands of values — can slow query response by many seconds. The `search.in(field, 'id1, id2, ...')` function is purpose-built for membership tests and keeps response times subsecond even with large identity lists. So for ACL trimming with many groups, `search.in` is both faster and cleaner.

Q: A critical caveat: does Azure AI Search authenticate the user via the security-filter pattern?
A: No — in the security-filter (trimming) pattern the principal is "just a string" used in a filter; there is no authentication or authorization performed by the search service against it. Your application must authenticate the caller, resolve their group memberships, and inject the correct IDs into the filter on every query. If your app passes the wrong or attacker-controlled group list, the search engine will faithfully return documents the user should not see.

Q: Where does the security-filter (trimming) pattern place the responsibility for correctness, and what is the open-stack equivalent you would have to build?
A: It places responsibility on the application layer: authenticate the user, map them to group IDs, and inject `group_ids/any(g:search.in(g,...))` into every retrieval call without exception. In your open `pgvector` stack you would have to build the same thing yourself — store ACLs as a column/array, add a `WHERE` clause keyed off the authenticated user's groups on every retrieval query, and prove that no code path can skip it. The managed platform does not magically authenticate; it gives you the field type and filter function, but the enforcement plumbing is still yours.

Q: How does the newer built-in ACL support differ from the older security-filter pattern in Azure AI Search?
A: The security-filter pattern simulates document-level authorization with a manually populated string field and an OData filter you inject. Azure AI Search also offers built-in document-level access control (ACL) support; the security-filter pattern is the documented fallback for solutions that cannot use the built-in ACL feature. Either way the index must carry per-document identity data, but the built-in option pushes more of the enforcement into the service.

Q: How would you populate the security/`group_ids` field — does Azure AI Search do it for you?
A: Azure AI Search provides no special API for populating the security field; you load it like any other field, either in a single push/pull operation that includes all fields or via multiple operations using `merge`/`mergeOrUpload` keyed on the document ID. So your ingestion code is responsible for attaching the correct group IDs to each fund document, typically derived from the source system's permissions. Getting that mapping right is the heart of correct security trimming.

Q: What is the semantic ranker in Azure AI Search?
A: The semantic ranker is a premium, usage-billed feature that re-scores an initial result set using Microsoft's multilingual deep-learning language models (adapted from Bing) to promote the most semantically relevant documents. It runs as a secondary ("L2") ranking on top of a BM25-ranked or RRF-ranked first pass, and can also return semantic captions and answers. It improves relevance for prose-heavy, information-rich content but adds cost and latency.

Q: How many results does the semantic ranker rerank, and what does it start from?
A: It reranks only the top 50 results from the first-stage ranking — a BM25-ranked set for text queries or an RRF-ranked set for vector/hybrid queries. Even if the initial query returns more than 50, only the top 50 progress to semantic reranking. It cannot re-run retrieval over the whole corpus; it only reorders the candidates the base ranker already surfaced.

Q: A key limitation: what can the semantic ranker NOT do?
A: It cannot rerun the query over the entire corpus to find semantically relevant documents that the base ranker missed; it only reorders the existing top-50 candidates. It also cannot invent new text — captions and answers are extracted verbatim from your indexed content, with no generative model composing new strings. So if a relevant document never makes the first-stage top 50, the semantic ranker will never surface it.

Q: What is `@search.rerankerScore` and what is its range?
A: `@search.rerankerScore` is the relevance score the semantic ranker assigns to each reranked document, ranging from 0.0 (irrelevant) to 4.0 (highly relevant and completely answers the query). Higher means more semantically relevant to the query intent, and results are listed in descending order of this score. You can threshold on it, but the docs warn distributions vary slightly with infrastructure and model updates, so do not set overly granular cutoffs.

Q: How is the semantic ranker priced/billed?
A: It is a premium feature billed by usage: there is a free plan with a monthly free request allowance (default), and a standard pay-as-you-go plan once you exceed that allowance. Charges accrue per qualifying query, so heavy query volume directly drives cost. This usage meter is exactly why you must decide whether the relevance gain justifies turning it on for a given corpus.

Q: Under what exact condition is a semantic-ranker query charged versus free?
A: You are charged when a query sets `queryType=semantic` and the search string is non-empty (for example `search=pet friendly hotels in New York`). If the search string is empty (`search=*`), you are not charged even with `queryType=semantic`. So a wildcard/filter-only request avoids the semantic meter, but any real text query with semantic enabled bills.

Q: When is the semantic ranker genuinely worth turning on for a fund-document corpus, and when is it not?
A: It is worth it when the content is information-rich, prose-structured (prospectuses, KIIDs, policy memos) and base hybrid ranking still buries the right passage, because the L2 rerank measurably lifts the correct answer. It is poor value when queries are dominated by exact-identifier lookups (an ISIN, a LEI, an account number) where keyword/BM25 already ranks the right doc first, so paying the per-query meter buys little. The decision should be driven by an eval showing reranked-vs-not relevance gain against your golden set.

Q: What are "semantic captions" produced by the semantic ranker?
A: Semantic captions are short, verbatim sentences or phrases extracted from the result's fields that best summarize why the document matched, returned in both plain-text and highlighted versions and typically under about 200 words. They let a search UI show a relevant snippet instead of a raw field dump. Crucially they are extracted, not generated — the model never writes new text.

Q: What are "semantic answers," and what must be true for one to be returned?
A: A semantic answer is an optional extra substructure that gives a direct answer when the query is phrased as a question, returned only if some indexed document contains text with the characteristics of an answer. You request it with the `answers` parameter; if no answer-like passage exists in the candidates, none is produced. Like captions, the answer is verbatim text from your index, not generated content.

Q: What is semantic "query rewrite," and how many variants can it create?
A: Query rewrite is an optional semantic-ranker capability that expands the original query string into up to 10 semantically similar variants — correcting typos, fixing spelling, or adding generated synonyms — and runs them on the search engine. The variant results are scored with BM25/RRF and then rescored by the semantic ranker. It is part of the query request (not the response) and is enabled separately.

Q: The semantic ranker's summarization step accepts how much text per document, and from which fields?
A: For each document the summarization model accepts up to 2,000 tokens (a token is roughly 10 characters), assembled from the `title`, `keyword`, and `content` fields named in the semantic configuration. Long fields are trimmed to fit, which is why you add fields to the semantic configuration in priority order. The summary string passed onward to the ranker can be up to 2,048 tokens (raised from 256 in November 2024).

Q: Why does field ordering in the semantic configuration matter?
A: Because the summarization step has token limits (`title` 128 tokens, `keywords` 128 tokens, `content` the remainder) and trims anything beyond the overall maximum, fields are consumed in the order you list them. If your most discriminating text sits in a field listed late, it can be truncated away before the ranker ever sees it. Ordering the most informative fields first ensures the reranker scores on the content that actually matters.

Q: Does the semantic ranker apply synonym maps?
A: Yes — if a field has an associated synonym map and that field is included in the semantic ranker configuration, the configured synonyms are automatically applied during reranking. This lets domain terms (for example "UCITS" expanding to its full form, or fund-specific abbreviations) be honored in the L2 step as well as the first-stage query. It is opt-in via the field's synonym map and the semantic config.

Q: Put the Azure managed pipeline stages in order for a document going in and a query coming out.
A: Ingest side: data source → indexer → skillset (chunking + embedding via integrated vectorization) → index (vector + text fields). Query side: client query → hybrid retrieval (vector + keyword in parallel) → RRF fusion of the two lists → optional semantic ranker (L2 rerank of top 50, captions/answers). Security trimming (the `search.in` filter) is applied during retrieval so out-of-scope documents never reach fusion or reranking.

Q: How does this managed pipeline map onto the open pipeline you built in 6.6.2?
A: Data source+indexer+skillset replaces your loader+chunker+embedder; the Azure index replaces your `pgvector` store; hybrid+RRF replaces your manual fusion of dense and BM25 results; and the semantic ranker replaces your cross-encoder reranker. Security trimming maps to the `WHERE` ACL clause you would hand-roll on every retrieval. Naming these correspondences is the point of the side-by-side sketch in the lesson's Do step.

Q: What is Amazon Bedrock Knowledge Bases, in one line?
A: Amazon Bedrock Knowledge Bases is AWS's managed RAG offering: you connect a data source, it handles ingestion/chunking/embedding into a supported vector store, and it serves retrieval (and optionally generation with citations) through the Bedrock API. It is the AWS counterpart to Azure AI Search's managed RAG stack, making the same managed-vs-open trade on a different cloud.

Q: Which vector stores can Amazon Bedrock Knowledge Bases use?
A: It supports several backends, including Amazon OpenSearch Serverless (the default the Bedrock console can auto-create for you), plus other supported vector stores such as Aurora PostgreSQL with `pgvector` and others. You can let Bedrock provision OpenSearch Serverless or bring your own pre-configured store. The point at a glance is that the managed service still sits on a real vector index you may or may not control directly.

Q: Does Amazon Bedrock Knowledge Bases support access-control-style filtering of retrieval?
A: Yes — it supports metadata filtering on retrieval, which you can use to restrict results, and it offers reranking models to influence which retrieved chunks are returned. As with Azure's security-filter pattern, correct enforcement depends on your application supplying the right filter values for the authenticated user. The managed service provides the filtering mechanism; the identity-to-filter mapping is still your responsibility.

Q: What is the Google Cloud equivalent to managed RAG, mentioned at a glance?
A: Vertex AI Search (formerly Enterprise/Agent Builder search) is Google Cloud's managed retrieval offering, bundling ingestion, embedding, indexing, and retrieval much like Azure AI Search and Bedrock Knowledge Bases. It makes the same managed trade — convenience and integration versus chunking opacity and lock-in — on GCP. The lesson treats all three (Azure, Bedrock, Vertex) as the same pattern across clouds.

Q: Summarize the core "what managed buys vs costs" trade.
A: Managed buys you ops relief (no vector store to run), built-in security/identity integration, and one-API hybrid+rerank retrieval. It costs you lock-in to that cloud's APIs and pricing, opacity over chunking/embedding via integrated vectorization, and a usage meter on premium features like the semantic ranker. The decision turns on which single requirement — usually security trimming or ops capacity — outweighs the lock-in and opacity for your specific corpus.

Q: For a Luxembourg fund administrator, what single requirement most often flips the decision toward a managed platform?
A: Security trimming — guaranteeing investor-restricted documents are never retrieved for the wrong user — because regulated fund administration cannot tolerate cross-investor data leakage, and the managed platform's identity-aware filtering plus the cloud's existing Entra/IAM integration shortens the path to a defensible control. If your open stack would need to build and prove that ACL enforcement from scratch, the managed option's built-in field types and filter functions tip the balance. Conversely, if the corpus is non-sensitive, ops and cost usually flip it back to open.

Q: When does the decision flip back toward your open `pgvector` stack instead of managed?
A: When the corpus is not access-restricted (so security trimming buys nothing), the queries are identifier-heavy (so the semantic-ranker meter buys little), and you already have the ops capacity to run `pgvector` — then lock-in and chunking opacity are pure downside. The open stack also wins when you need fine-grained chunking control for quality or want to avoid per-query rerank billing at scale. The eval should state this counter-case as explicitly as the pro-managed case.

Q: First thing to check: hybrid search returns no vector matches at all in Azure AI Search, only keyword hits. What do you inspect?
A: Check that the index actually has a populated vector field and that the query is sending a vector (or using integrated vectorization to encode the query), because a missing query vector silently degrades to keyword-only. Then confirm the vector field's dimensions match the embedding model and that documents were vectorized during indexing. A common cause is forgetting the query-side vectorizer so no vector is ever produced.

Q: First thing to check: semantic-ranker results look identical to non-semantic results. What is likely wrong?
A: Confirm `queryType=semantic` is actually set and a non-empty search string is supplied, and that a valid semantic configuration naming the right fields exists, because without these the ranker never runs (or runs on the wrong fields). Also check you have not exhausted the free allowance in a way that disabled it, or that the field ordering left the discriminating content truncated. Often the request simply was not flagged semantic.

Q: First thing to check: a user sees a fund document they should not. With security trimming in place, where is the bug?
A: Almost always in the application's identity-to-filter mapping, not the search engine — verify the authenticated user's group IDs are correctly resolved and injected into the `group_ids/any(g:search.in(g,...))` filter on that exact query path. Because the principal is just a string the service trusts, a missing filter on one code path, a stale group cache, or a wrong ID list will leak documents. Check that every retrieval call without exception applies the filter.

Q: First thing to check: integrated vectorization indexing fails partway with embedding errors. What do you look at?
A: Look at the skillset's embedding skill configuration and the linked Azure OpenAI (or embedding) resource — wrong endpoint, key, deployment name, region, or hitting model rate/quota limits are common causes during the indexer's embedding calls. Check the indexer execution history/errors for the failing documents and whether throttling (429) is occurring. Because the embedding call is inside the managed pipeline, the indexer logs are your window into it.

Q: Why might security trimming be "the one managed feature your compliance team actually needed" even if you reject the rest?
A: Because document-level access control is hard to build correctly and prove in a regulated audit, the managed platform's identity-aware filtering plus the cloud's existing directory integration can be the only piece that justifies adoption — even if you would otherwise prefer your open stack. Compliance may insist on it for investor-restricted material regardless of cost. So you might "buy" the platform essentially for that single control.

Q: How does the semantic ranker relate to the reranker you built in your open 6.6.2 pipeline?
A: Both are an L2 reranking stage applied after first-stage retrieval to reorder candidates by deeper relevance, but the semantic ranker is a managed, usage-billed black box using Microsoft's models, while your open reranker (a cross-encoder you host) is transparent and free per query but yours to run and tune. The semantic ranker also bundles captions/answers/query-rewrite that your custom reranker would not. The trade mirrors the whole managed-vs-open theme: convenience and extras versus control and per-query cost.

Q: Why is hybrid+RRF a strong default before even considering the semantic ranker?
A: Because hybrid already combines the complementary strengths of dense (semantic) and sparse (keyword) retrieval via a scale-free fusion, it frequently produces better top results than either alone at no premium charge. Adding the semantic ranker is a further, billed improvement on top, so you should measure whether the L2 rerank actually moves your eval metric beyond hybrid+RRF before paying for it. Starting from hybrid+RRF gives you a free, strong baseline.

Q: What does "lock-in" concretely mean for a managed RAG platform?
A: Lock-in means your retrieval logic, index schema, skillset config, query syntax (OData filters, semantic config), and identity-filter plumbing are written against one cloud's proprietary APIs, so migrating to another platform means re-implementing ingestion, querying, and security trimming. Your data and prompts may be portable, but the surrounding integration is not. This coupling is a real cost to weigh against the ops relief the platform provides.

Q: How is the cost model of a managed RAG platform fundamentally different from your self-hosted stack?
A: A managed platform mixes a service tier cost with per-usage meters (notably the semantic ranker billed per qualifying query, plus embedding-call charges), so cost scales with query volume and feature usage. Your self-hosted `pgvector` stack is mostly fixed infrastructure cost (compute/storage you provision) regardless of query count, plus embedding-generation cost. At high, spiky query volume the metered features can dominate, which the eval's cost row must capture.

Q: In the eval, why include a separate "chunking control" row distinct from "capability"?
A: Because integrated vectorization can match the open stack on retrieval capability while still removing your ability to tune chunk size, overlap, and boundary logic — a difference invisible until quality degrades. Keeping chunking control as its own row forces you to register that you are trading away a tuning knob, not just a feature. For document-heavy fund corpora where table/section integrity matters, that knob can be decisive.

Q: A query for the exact string `ISIN LU0123456789` should rank the right KIID first. Does the semantic ranker help here?
A: Probably not much — exact-identifier lookups are precisely where BM25/keyword already ranks the correct document first, so the L2 semantic rerank of the top 50 adds little while still metering you per query. Hybrid retrieval already ensures the literal code matches via the keyword arm. This is a textbook case where you would leave the semantic ranker off to avoid paying for relevance you do not gain.

Q: Conversely, for "explain the swing-pricing policy for the bond sub-fund," is the semantic ranker likely to help?
A: Yes — this is a prose, intent-heavy question over descriptive content where the right passage may not be the highest BM25 hit, exactly the scenario where the semantic ranker's language understanding promotes the most relevant passage and can extract a caption/answer. Information-rich, prose-structured fund documentation is where the docs say the ranker delivers the most gain. Here the per-query cost is more likely justified by the relevance lift.

Q: What is the indexer's role versus the skillset's role in integrated vectorization?
A: The indexer is the orchestration component that connects to the data source, crawls/pulls documents, drives the pipeline, and writes results to the index, including tracking change detection. The skillset is the set of skills (split/chunk, embed, optionally enrich) the indexer invokes per document to transform content. Integrated vectorization needs both: the indexer to move data and the skillset to chunk and embed it.

Q: Why does the lesson stress sketching the managed and open pipelines "side by side"?
A: Because only by aligning stage-for-stage (chunker↔skillset, vector store↔index, fusion↔RRF, reranker↔semantic ranker, ACL clause↔security filter) can you see precisely what the managed platform absorbs and what it hides. The side-by-side makes the trade-offs concrete instead of abstract marketing claims. It directly feeds the one-page eval's rows.

Q: What does "ACL-aware retrieval via identity filters" mean in one sentence?
A: It means retrieval results are filtered against the caller's identity (their group/user IDs) so that only documents whose stored access list includes one of those IDs are returned, enforcing document-level read permissions at query time. The filter is the `search.in` membership test over the document's principal-ID collection field. It approximates per-document authorization without the search service itself doing auth.

Q: Why must the security/identity field be set `retrievable: false`?
A: Setting `retrievable: false` keeps the list of allowed group/user IDs out of the API response, so the access-control metadata of a document is never leaked to clients even when the document is returned. The field still works in filters because it is `filterable: true`. Leaking the ACL could expose group structure and aid an attacker in crafting a spoofed filter, so hiding it is a deliberate safeguard.

Q: If your open stack must replicate Azure's security trimming, what is the minimal design?
A: Store each document's allowed principal IDs (for example a `text[]` array column in Postgres), and on every retrieval query add a `WHERE allowed_groups && ARRAY[:caller_groups]` (array-overlap) clause keyed to the authenticated user's resolved groups. Authenticate the caller upstream, resolve their groups from your directory, and ensure no retrieval path can bypass the clause. This is functionally the `group_ids/any(g:search.in(g,...))` filter rebuilt in SQL, and proving no path skips it is the hard part.

Q: Why is "the search engine faithfully returns whatever the filter says" a security risk, not a feature, if mishandled?
A: Because the engine performs no authentication on the principal string, it will return restricted documents the instant your application passes an incorrect, overly broad, or attacker-influenced group list — the trust boundary is entirely in your app. A bug that injects an admin group, omits the filter, or uses a stale group cache becomes a data-leak, not a search bug. This is why every retrieval path must be audited to inject the correct, freshly-resolved identity filter.

Q: How does query rewrite interact with billing and latency on the semantic ranker?
A: Query rewrite generates up to 10 variant queries, each run on the search engine and then rescored by the semantic ranker, so it increases work and is part of the semantic feature set you have enabled. It can improve recall on misspelled or awkwardly phrased queries but adds latency and is tied to the semantic-ranker usage you are billed for. You enable it deliberately, weighing the recall gain against the extra cost and time.

Q: Are semantic captions and answers ever AI-generated/hallucinated text?
A: No — the docs are explicit that captions and answers are always verbatim text extracted from your index, with no generative model creating or composing new content in this workflow. The machine-reading-comprehension models select the most representative passages; they do not write. This matters in regulated contexts because what is shown to the user can be traced exactly to indexed source text.

Q: Why is the "captions/answers are verbatim" guarantee attractive in a regulated fund context?
A: Because any snippet surfaced to a user or analyst can be traced exactly to source text in the index, with no risk of a generative model inventing a NAV figure or misstating a policy in the displayed caption. Provenance and faithfulness are easier to defend in audit when the displayed text is extractive, not generated. The generation step (if any) happens downstream and separately, where you apply your own controls.

Q: What is the difference between BM25-based first ranking and RRF-based first ranking for the semantic ranker's input?
A: For a pure text query the first stage is BM25-ranked; for a vector or hybrid query the first stage is RRF-ranked (fusing vector and keyword lists). The semantic ranker takes whichever of these top-50 sets is produced and reranks it using text only. So the base ranker differs by query type, but the L2 input is always the top 50 of that base ranking.

Q: The semantic ranker "uses only text" to rerank — what is the implication for a vector-heavy query?
A: Even when retrieval was driven by vectors, the semantic L2 reranking operates on the text content of the candidates (assembled from the configured title/keyword/content fields), not on the embeddings. So fields must contain meaningful, prose text for the ranker to be effective; sparse or code-only fields give it little to work with. This is why information-rich, descriptive content benefits most from the semantic ranker.

Q: Why warn against setting very granular `@search.rerankerScore` thresholds?
A: Because the docs note that for a given query the `@search.rerankerScore` distribution can vary slightly due to infrastructure-level conditions, and ranking-model updates can shift the distribution over time. A too-tight threshold (say cutting at exactly 2.7) may behave inconsistently or break after a model update. Use coarser, more robust cutoffs if you threshold on it at all.

Q: How does an audit/lineage requirement (Phase 7 theme) interact with choosing a managed RAG platform?
A: A managed platform's integrated vectorization hides chunking/embedding details, which can complicate answering "exactly what text did the model retrieve and why," whereas your open pipeline exposes every stage for lineage capture. You may still get retrieval traces, but the chunk-construction step is opaque. For regulated workloads where you must reconstruct what was retrieved for a given NAV extraction, that opacity is a point against managed unless the platform exposes adequate diagnostics.

Q: What does "convenience bought with opacity" specifically refer to in this lesson?
A: It refers to integrated vectorization: you gain the convenience of not running your own chunker and embedder, but you lose visibility into and control over how documents were split and embedded inside the managed indexer/skillset. The phrase compresses the central trade of integrated vectorization. Naming what it hides (chunk boundaries, overlap, embedding details) and why it matters at degradation time is a "Done when" requirement.

Q: For the eval's "ops burden" row, what does the managed platform genuinely remove?
A: It removes operating the vector store itself — provisioning, scaling, patching, backing up, and tuning a `pgvector`/index cluster — plus much of the embedding-service plumbing, since the indexer makes those calls. You still operate ingestion configuration, query code, and the all-important identity-filter logic. So managed reduces but does not eliminate ops, which the row should reflect honestly.

Q: If two share classes of a fund have different disclosure restrictions, how does security trimming model that?
A: Each document is tagged with the group IDs permitted to read it, so a restricted share-class memo carries only the IDs of teams/investors cleared for it, while the public KIID carries broader IDs. At query time the caller's groups are matched via `search.in`, returning only the documents their identity covers. Different restrictions are just different `group_ids` sets per document.

Q: What happens to documents that exceed the semantic ranker's per-document token limit?
A: They are trimmed — the summarization step accepts up to 2,000 tokens per document assembled from the configured fields, and anything beyond the overall limit is ignored, with `title` capped at 128 tokens, `keywords` at 128, and `content` taking the remainder. So very large, text-heavy documents lose their tail content for reranking purposes. Putting the most discriminating text early (and in the right field) avoids losing it to trimming.

Q: Does turning on the semantic ranker change which documents are retrieved, or only their order?
A: Only their order (and the captions/answers returned) — it reranks the existing top-50 candidates and cannot pull in documents the first-stage retrieval missed. If query rewrite is enabled it can change the candidate set indirectly by running variant queries, but the core L2 ranking just reorders. So a relevant document absent from the base top-50 stays invisible regardless of the semantic ranker.

Q: Why does the lesson want the ACL requirement treated "explicitly" in the eval rather than as a footnote?
A: Because for investor-restricted fund documents, "are restricted documents never retrieved for the wrong user?" is frequently the make-or-break requirement, and how each side enforces it (built-in identity filter vs hand-built `WHERE` clause you must prove) often decides build-vs-buy. Burying it loses the most consequential dimension. The eval must state, for each side, the mechanism and what the open stack would have to build.

Q: What is "RRF fusion on top" in the lesson's one-line description of the Azure stack?
A: It means after the parallel vector and keyword searches produce two ranked lists, Reciprocal Rank Fusion merges them into one ranking before any reranking — RRF sits on top of the two retrievers and below the optional semantic ranker. It is the layer that turns "two searches" into "one hybrid result." The semantic ranker, if enabled, then reranks that fused top-50.

Q: In what sense is the semantic ranker a "secondary" ranker?
A: It is secondary because it always runs after and on top of a primary ranking (BM25 or RRF), re-scoring that primary result set rather than performing retrieval itself. The primary stage decides which documents are candidates; the secondary stage decides their final order using deeper language understanding. This two-stage (retrieve-then-rerank) design is standard in modern search.

Q: How would you empirically decide whether to enable the semantic ranker for the fund corpus?
A: Run your golden-set eval (the 6.8.2 task) with and without `queryType=semantic`, measuring a task metric like field-level extraction accuracy or answer relevancy, and compare the lift against the per-query cost and added latency. If the relevance gain on your actual questions clears your bar and the cost is acceptable at expected volume, enable it; otherwise leave it off. The decision should be data-driven, not assumed.

Q: What are "identity filters" as opposed to ordinary metadata filters?
A: Identity filters are metadata filters whose values are the caller's security principals (group/user IDs), applied to a document's stored ACL field to enforce read permissions, whereas ordinary metadata filters narrow results by attributes like document type, fund family, or date. Mechanically both are OData filters, but identity filters are a security control whose values must come from authenticated identity resolution. Conflating them risks treating an authorization control as a mere convenience filter.

Q: Why might integrated vectorization still be the right call despite its opacity?
A: When you lack the engineering capacity to build and maintain a robust chunking/embedding pipeline, or when the corpus is straightforward enough that default chunking performs well, the convenience and reduced ops can outweigh the lost transparency. Many corpora never push chunking to the point where the opacity bites. The eval should state the conditions under which the convenience wins, not reflexively penalize it.

Q: How does the choice of managed vs open RAG affect DORA-style operational resilience considerations?
A: A managed platform concentrates a critical retrieval dependency on one cloud provider, which under DORA you must assess as a third-party ICT dependency with exit/continuity plans, whereas a self-hosted stack keeps the dependency internal but shifts the resilience burden onto your own operations. Neither is automatically better; both must be documented for concentration risk and recoverability. The eval's lock-in and ops rows feed directly into that resilience assessment.

Q: What is the danger of "paying the semantic-ranker meter for a corpus pgvector served fine"?
A: It is paying a per-query premium for relevance you did not actually need, because for your particular queries (often identifier-heavy lookups) plain hybrid retrieval over `pgvector` already ranked the right document first. The meter then adds cost and latency without improving outcomes. The lesson frames this as the failure mode of accepting the corporate pitch without measuring the capability/cost envelope.

Q: What is the danger of "rejecting the one managed feature your compliance team actually needed"?
A: It is choosing the open stack on cost/control grounds while overlooking that the managed platform's security trimming (ACL-aware retrieval) was the specific control compliance required for investor-restricted documents. You then either cannot ship or must build that control yourself at real cost and risk. The lesson uses this as the opposite failure to over-paying for the ranker, underscoring why the eval must treat ACLs explicitly.

Q: Across clouds, what is "the same managed trade" the lesson refers to?
A: Whether Azure AI Search, Amazon Bedrock Knowledge Bases, or Vertex AI Search, you trade chunking/embedding control and cloud lock-in for managed ops, integrated identity/security, and one-API hybrid+rerank retrieval. The branding and exact features differ, but the fundamental convenience-vs-control trade is identical. Recognizing it as one pattern lets you evaluate any of them with the same one-page rubric.

Q: How does hybrid search improve recall for fund documents specifically?
A: Fund documents mix exact tokens (ISINs, LEIs, share-class codes, NAV figures) with descriptive prose (investment policy, risk factors), so keyword retrieval catches the exact codes while vector retrieval catches the conceptual queries, and hybrid+RRF surfaces both in one ranked list. A vector-only system might miss a literal `LU` ISIN; a keyword-only system might miss a paraphrased policy question. Hybrid covers both failure modes at once.

Q: What does "security trimming" require you to keep in sync between the source system and the index?
A: It requires the per-document group/user ACLs in the index to stay consistent with the authoritative permissions in the source system, so that when access changes (an investor offboarded, a team's clearance revoked) the index's `group_ids` are re-indexed accordingly. Stale ACLs in the index mean either over-exposure or wrongful denial. Your ingestion/sync process must propagate permission changes, since the search service will trust whatever IDs are stored.

Q: If a document's permissions change after indexing, how do you update its security field?
A: Use a `merge` or `mergeOrUpload` action targeting that document's key with the new `group_ids` values, which updates the security field without re-uploading the whole document. This keeps the ACL current without a full re-index of content. Your sync logic should trigger this whenever the source system's permissions for that document change.

Q: Why is the document key relevant in the security-trimming schema?
A: Every Azure AI Search index requires a document key (for example `file_id`), and it is the identifier you target for `merge`/`mergeOrUpload` operations that update the security field or any other field. Without a stable key you cannot reliably patch a document's ACL when permissions change. So the key underpins both indexing and incremental ACL maintenance.

Q: What is the relationship between the semantic ranker and agentic retrieval mentioned in the docs?
A: The semantic ranker is also built into Azure AI Search's agentic retrieval, and the two features are billed independently, so enabling agentic retrieval may invoke semantic reranking as a component. For this lesson the standalone semantic-ranker behavior (top-50 L2 rerank, usage billing) is what you must understand. Just note that the same ranker shows up inside the newer agentic flows.

Q: In the one-page eval, why is "capability" alone an insufficient comparison?
A: Because two systems can be roughly equal on raw retrieval capability while differing sharply on cost, ops burden, chunking control, and lock-in — the dimensions that actually drive the decision. Comparing only capability hides exactly the trade-offs the lesson is about. That is why the eval has explicit rows for each of cost, ops, chunking control, and lock-in alongside capability.

Q: How do you state the eval's conclusion per the lesson's instructions?
A: As a single sentence naming the one requirement that flips the decision either way — for example, "If document-level ACL trimming is mandatory and we lack capacity to build and prove it, Azure AI Search wins; otherwise the open `pgvector` stack wins on cost and control." It must be one crisp flipping requirement, not a hedged list. The "Done when" check is precisely the ability to name that requirement.

Q: Why does the semantic ranker only operate on text even for hybrid queries — what does that imply for index design?
A: Because the reranking models consume the configured `title`/`keyword`/`content` text fields, your index must surface meaningful prose in those fields for the ranker to score on, even when retrieval was vector-driven. If your retrievable text is thin or non-descriptive, the ranker has little signal and the gain shrinks. So when planning to use the semantic ranker, design fields to carry rich, descriptive text.

Q: What is a concrete sign you should switch from integrated vectorization back to external embedding?
A: When retrieval quality problems trace to chunking — answers cut off mid-table, lost section context, sub-funds merged into one chunk — and you cannot fix them within the managed skillset's limited controls, that is the signal to take chunking back in-house via external embedding and push. You regain explicit control over size, overlap, and boundaries. The cost is reintroducing the embedding pipeline you had offloaded.

Q: Summarize the three "Done when" criteria for this lesson in your own words.
A: One: name the single requirement (usually security trimming or ops) that flips the build-vs-buy decision either way. Two: explain what integrated vectorization hides (chunking/embedding control) and why that matters when retrieval quality degrades. Three: state the semantic ranker's pricing model (usage-billed, free monthly allowance, charged on non-empty `queryType=semantic`) and when it is worth enabling for this corpus. Together they prove you have sized the capability/cost envelope rather than accepted the pitch.

Q: Why is RRF described as robust to combining "incomparable" retrievers?
A: Because it ignores the raw, differently-scaled scores of each retriever and uses only rank positions, fusing reciprocals of ranks, so no retriever's score magnitude can dominate the merge. This makes hybrid fusion stable even when you have not calibrated vector similarity against BM25. That robustness is why RRF is Azure's default fusion for hybrid queries.

Q: A stakeholder claims the semantic ranker "searches the whole corpus more deeply." How do you correct them?
A: Politely but firmly: the semantic ranker does not search the corpus at all — it reranks only the top 50 candidates already returned by the BM25/RRF first stage, using language models, and cannot recover relevant documents that first-stage retrieval missed. If recall is the problem, you fix retrieval (better hybrid, query rewrite, chunking), not the reranker. Conflating reranking with retrieval leads to spending on the wrong stage.

Q: Why can a managed RAG platform still leave you owning the generation step's controls even though it manages retrieval?
A: Platforms like Azure AI Search primarily manage ingestion, indexing, and retrieval (and return extractive captions/answers), but the downstream LLM generation — prompts, model choice, faithfulness checks, and PII handling — is typically your application's responsibility unless you also adopt the cloud's full RAG/agent service. So you can buy managed retrieval and still need your own tracing, eval, and masking from 6.8.x. Recognizing this prevents assuming "managed RAG" covers the generation-side regulatory controls, which it generally does not by default.

Q: Final synthesis: for the fund-document corpus, outline the decision logic you would present.
A: Start with hybrid+RRF as a strong, free baseline; enable the semantic ranker only if a golden-set eval shows the L2 rerank materially lifts relevance on your prose questions enough to justify its per-query meter; and make security trimming the pivotal factor — if mandatory ACL-aware retrieval for investor-restricted documents must be built and proven and you lack that capacity, choose the managed platform, otherwise prefer the open `pgvector` stack on cost, chunking control, and reduced lock-in. State that pivot in one sentence, treat integrated-vectorization opacity as a known cost, and document the third-party concentration risk for resilience. That is the capability/cost envelope the lesson asks you to size.

Q: What field type does a vector field use in an Azure AI Search index, and what restrictions apply to it?
A: A vector field is typed `Collection(Edm.Single)` (a list of 32-bit floats), and it must be `searchable: true` and `retrievable` as needed, but it cannot be `filterable`, `facetable`, or `sortable`, nor can it have analyzers, normalizers, or synonym maps assigned. This is exactly why the security/ACL `group_ids` filter lives on a separate `Collection(Edm.String)` field rather than the vector field itself. To filter alongside vector search you always pair the vector field with ordinary filterable text/numeric fields.

Q: What is a `vectorSearchProfile` and how does it connect a vector field to an algorithm and vectorizer?
A: A `vectorSearchProfile` is a named bundle defined in the index's `vectorSearch` section that references an algorithm configuration (HNSW or exhaustive KNN) and optionally a vectorizer, and each vector field names the profile it uses. This indirection lets several vector fields share one tuning/algorithm config and lets you swap the embedding vectorizer without rewriting every field. Without assigning a profile, a vector field has no algorithm to index or query against.

Q: What is the maximum number of dimensions a single Azure AI Search vector field can hold, and why does it matter for embedding-model choice?
A: The maximum is `3072` dimensions per vector field, which comfortably fits common models like `text-embedding-3-large` (3072) and `text-embedding-3-small`/`ada-002` (1536). If you picked a model emitting more than 3072 dims you could not store it natively and would need dimension reduction first. The field's declared dimension count must exactly match the model's output, or indexing/query vectors will be rejected.

Q: First thing to check: query-side integrated vectorization returns the error that vector dimensions do not match. What is wrong?
A: The query vector's dimension count differs from the vector field's declared `dimensions`, almost always because the query is being encoded by a different (or differently-configured) embedding model than the one used at indexing time. Azure requires the index-time and query-time models to produce identically-dimensioned vectors in the same embedding space. Fix it by pointing the vectorizer at the exact same deployment/model that populated the field.

Q: What two vector indexing algorithms does Azure AI Search support, and when would you choose exhaustive KNN?
A: It supports HNSW (a graph-based approximate nearest-neighbor algorithm) and exhaustive KNN (brute-force exact search). You choose exhaustive KNN when you need exact nearest neighbors — for example to compute ground-truth recall for an eval, or for small fields where approximation buys nothing — accepting higher per-query cost. HNSW is the default for production because it trades a little recall for far better latency at scale.

Q: What does the HNSW `m` parameter control in Azure AI Search and what is its default and range?
A: `m` is the bi-directional link count — the number of edges created per node when building the HNSW graph — with a default of `4` and an allowed range of `4` to `10`. Higher `m` improves recall and graph connectivity but increases memory and index size. It is set in the algorithm configuration and cannot be changed without rebuilding the index.

Q: What do `efConstruction` and `efSearch` control, and which one can you tune at query time?
A: `efConstruction` (default `400`, range `100`–`1000`) is the size of the dynamic candidate list used while building the graph, fixed at index time; `efSearch` (default `500`, range `100`–`1000`) is the candidate-list size used during a query. Raising either improves recall at the cost of latency, but only `efSearch`-style search-time behaviour is adjustable per query whereas `efConstruction` requires re-indexing. For a fund corpus where recall on rare share-class terms matters, you tune the search-time parameter first.

Q: A fund corpus's vector index is too large for the tier's quota. What two storage-reduction levers does Azure AI Search give you?
A: Quantization (scalar `int8`, roughly 4x smaller; or binary, up to ~28x smaller) compresses the stored vectors, and setting the field's `stored` property to `false` drops the original full-precision copy from storage. You can also truncate dimensions via `truncationDimension` (MRL). Each saves space at some recall cost, so you measure the recall hit on your golden set before committing.

Q: What is the risk of setting a vector field's `stored` property to `false`, and when is it safe?
A: With `stored: false` the original uncompressed vector is not retained, so a partial document update (`merge`) that omits the vector will lose it — you must re-supply the full vector on every update. It is safe when you always re-upload the complete vector or never partially update, and you accept that you cannot return raw vectors. For fund documents whose ACLs you patch frequently with `merge`, leaving `stored: true` (or always sending the vector) avoids silently blanking embeddings.

Q: What is `oversampling` with `rerankWithOriginalVectors`, and why does it exist?
A: When vectors are quantized, `oversampling` makes the query retrieve more than `k` candidates from the compressed index (e.g. `k=5` with `oversampling=20` pulls 100), then `rerankWithOriginalVectors` rescores those candidates with the original full-precision vectors and returns the top `k`. It recovers recall lost to quantization at modest extra latency. It can only be set when a compression method is configured on the field, and exhaustive KNN does not support it because it already scans everything.

Q: What does the `exhaustive: true` flag on a vector query do even when the field's algorithm is HNSW?
A: It forces that single query to bypass the HNSW graph and run an exact brute-force k-nearest-neighbor scan over all vectors in the field, giving exact results at higher latency. It is useful to establish ground-truth nearest neighbors for measuring your HNSW configuration's recall. You set it per query, so you can keep HNSW for production traffic and flip `exhaustive` on only for eval runs.

Q: How do you test that an Azure AI Search vectorizer is wired correctly without computing an embedding yourself?
A: Send a vector query that passes a plain text string instead of a precomputed vector; if the vectorizer is configured on the field's profile, the service encodes the text at query time and returns matches. If it errors or returns nothing, the vectorizer/profile binding is wrong. This is the fastest smoke test that query-time integrated vectorization is functioning before debugging deeper.

Q: A hybrid query suddenly returns far fewer results after you enabled a metadata filter. Is this pre-filtering or post-filtering, and how do you tell?
A: It is likely post-filtering, where the vector search first finds the top-`k` nearest neighbors and then the filter discards those failing it, so a restrictive filter can leave very few survivors out of `k`. Pre-filtering applies the filter first and searches only the matching subset, preserving result count but costing more per query. Check your `vectorFilterMode` (`preFilter` vs `postFilter`) — switching to pre-filtering restores recall for selective filters like a single fund family.

Q: What is `vectorFilterMode` and which setting suits a highly selective fund-family filter?
A: `vectorFilterMode` chooses whether an OData filter is applied before (`preFilter`) or after (`postFilter`) the vector nearest-neighbor search. For a highly selective filter — say restricting to one small sub-fund — `preFilter` is better because it narrows the candidate space first so you still get `k` relevant hits, whereas `postFilter` might leave almost none. For broad filters `postFilter` can be faster, so the choice is selectivity-dependent.

Q: Why is the security-trimming `search.in` filter best applied as a pre-filter rather than a post-filter?
A: Security trimming must guarantee no out-of-scope document is even considered, and applying it as a pre-filter restricts the searched candidate set to authorized documents before ranking, which is both correct and avoids the post-filter pathology of `k` neighbors collapsing to a handful after ACL removal. Post-filtering could surface a near-empty result for a user with narrow access even when many authorized matches exist. So for ACLs, pre-filter is the safer and higher-recall mode.

Q: When updating only a fund document's `group_ids` ACL, why prefer `merge` over `mergeOrUpload`?
A: `merge` updates fields on an existing document and fails if the document key is not already present, which is the safe choice when you intend only to patch the ACL of a document you know exists, surfacing an error if the key is missing. `mergeOrUpload` would instead create a brand-new, near-empty document from just the key and `group_ids`, silently producing a malformed index entry. So `merge` gives you a guardrail against accidentally inserting partial documents during ACL sync.

Q: What happens to a fund document's searchability if your ACL sync uploads it via `upload` instead of `merge` and omits the content fields?
A: `upload` performs a full replace (upsert) of the document by key, so omitting content and vector fields overwrites the existing document with a sparse one — the prose and embedding are wiped, and it drops out of meaningful retrieval. Only `merge`/`mergeOrUpload` preserve unspecified existing fields. This is a classic ACL-sync bug: a permissions update accidentally blanks the indexed content because the wrong action was used.

Q: In the security-trimming OData filter `group_ids/any(g: search.in(g, 'A, B'))`, what is the default delimiter and what if a group ID contains a comma?
A: The default delimiter for the value list in `search.in` is a comma (and whitespace), so `'A, B'` is two IDs; if any identifier itself contains a comma you must supply a second argument specifying an alternate delimiter, e.g. `search.in(g, 'A|B', '|')`. Without that, an ID with an embedded comma would be split into two wrong IDs and either over- or under-match. For GUID-style group IDs this rarely bites, but it is a real correctness trap.

Q: A fund analyst belongs to 4000 groups; the security filter query times out building the `or` expression. What is the fix?
A: Stop generating a long chain of `Id eq '...' or Id eq '...'` equalities and use `search.in(g, '<comma-list>')`, which is purpose-built for large membership lists and stays subsecond where the disjunction can add many seconds. The single `search.in` passes all IDs as one string argument rather than thousands of clauses the query parser must build. This is precisely the scenario the function was designed for.

Q: How does Azure AI Search's newer built-in document ACL feature reduce the application's burden compared with the manual security-filter pattern?
A: With built-in document-level access control you attach permission metadata (e.g. via permission filters tied to Microsoft Entra group/user identifiers) and the service enforces trimming when the query carries the caller's token, rather than your app hand-injecting a `search.in` filter on every call. It shifts more enforcement into the service, lowering the chance a code path forgets the filter. The manual security-filter pattern remains the documented fallback when you cannot use the built-in feature.

Q: Even with built-in ACLs, what does the application still own for correctness?
A: The application still owns authenticating the user, presenting a valid identity token, and — critically — keeping the index's per-document permission metadata synchronized with the authoritative source-system permissions. The service enforces against whatever ACLs are stored, so stale permissions still cause wrong access. Built-in ACLs remove the per-query injection burden but not the identity-resolution and ACL-sync responsibilities.

Q: For a Luxembourg transfer-agency RAG over investor registers, why is post-indexing ACL latency a compliance concern?
A: When an investor is offboarded or a distribution agreement is terminated, their access must be revoked promptly, but the index only reflects it after the next ACL re-index/`merge`, so the window between source-system change and index update is a period of potential over-exposure. Regulators care that access changes propagate within a defined, auditable SLA. You must design the sync to push ACL updates quickly and log the propagation time.

Q: How would you prove to an auditor that no retrieval path can skip the security filter?
A: Centralize all retrieval through a single repository/client method that always injects the freshly-resolved `group_ids/any(g:search.in(...))` filter (or relies on built-in ACL enforcement), forbid direct calls to the raw search SDK elsewhere, and back it with tests that assert a query without identity returns nothing for restricted docs. Then show the audit a code-coverage map and the centralized chokepoint. The proof is architectural (one enforced path) plus tests, not a promise that every developer remembered the filter.

Q: Bedrock Knowledge Bases offers `FIXED_SIZE`, `HIERARCHICAL`, `SEMANTIC`, and `NONE` chunking. What does `HIERARCHICAL` do and why suit a prospectus?
A: `HIERARCHICAL` chunking builds two layers — large parent chunks and smaller child chunks derived from them — retrieving on the precise child chunks but returning the broader parent to the model for fuller context. For a long fund prospectus where a clause's meaning depends on its surrounding section, this preserves context that flat fixed chunks would sever. You set parent size, child size, and overlap in the data-source config.

Q: What does Bedrock's `NONE` chunking strategy require of you, and when is it appropriate?
A: `NONE` tells Bedrock not to split content, so each source file is treated as a single chunk — meaning you must pre-split your documents upstream so each file already corresponds to one sensible chunk. It is appropriate when you have a custom chunker (as in your open 6.6.2 pipeline) and want the managed KB to index your boundaries verbatim rather than impose its own. It is the managed-platform escape hatch that returns chunking control to you.

Q: How does `SEMANTIC` chunking in Bedrock differ from `FIXED_SIZE`, and what is the trade?
A: `SEMANTIC` groups content into chunks by topical similarity using NLP rather than by a fixed token count, so boundaries fall at meaning shifts instead of arbitrary lengths, which can keep a single swing-pricing explanation intact. The trade is higher ingestion cost/complexity and less predictable chunk sizes versus `FIXED_SIZE` (e.g. ~512 tokens with 20% overlap), which is cheap, deterministic, and a sensible default. You pick semantic only when fixed chunking measurably hurts your eval.

Q: What is the difference between Bedrock's `Retrieve` and `RetrieveAndGenerate` APIs, and why does it matter for control?
A: `Retrieve` returns only the matching chunks (with scores/metadata) for your application to use as it wishes, whereas `RetrieveAndGenerate` also calls a foundation model to produce an answer with citations in one call. For a regulated fund workflow you often want `Retrieve` so your own application owns the generation prompt, model choice, and faithfulness/PII controls. `RetrieveAndGenerate` is convenient but hands the generation step's controls to the managed service.

Q: What is Vertex AI Search's "blended search" and where might it help a fund administrator?
A: Blended search connects multiple data stores (e.g. a structured BigQuery table of NAVs plus an unstructured store of prospectus PDFs) to one search app so a single query spans both structured and unstructured sources. A fund administrator could ask one question and retrieve both the numeric NAV record and the explanatory policy text together. It is Google's answer to querying heterogeneous fund data through a single managed retrieval surface.

Q: In Vertex AI Search, what must you enable at data-store creation time to use it for RAG, and what does that mirror in Azure?
A: You turn on document chunking when creating the data store so the service splits documents into retrievable chunks, then optionally ground a Gemini answer on the results. This mirrors Azure's integrated vectorization (chunking + embedding inside the managed pipeline) — and carries the same opacity trade, since the chunk boundaries are again decided by the platform. Across all three clouds, enabling managed chunking is the convenience-for-opacity step.

Q: A managed RAG platform indexes a fund document containing personal investor data. Which GDPR/DORA control does integrated vectorization complicate, and how?
A: The right-to-erasure / data-minimization control is complicated because the personal data may now exist not only as source text but also embedded in opaque vectors and copied into managed chunks, so proving complete deletion requires the platform to purge all derived representations. You must confirm the service deletes vectors and chunks on document deletion, not just the source. For DORA's data-integrity and exit requirements, you also document that deletion is verifiable and exportable.

Q: How does choosing a managed RAG platform interact with DORA's third-party ICT register and exit-plan obligations?
A: A managed platform is a critical third-party ICT provider, so under DORA you must register the dependency, assess concentration risk, and maintain a tested exit/continuity plan describing how you would migrate retrieval off it. The platform's proprietary index schema, query syntax, and ACL plumbing (the lock-in row of your eval) directly raise exit cost. Documenting that exit path is a regulatory deliverable, not optional hygiene.

Q: Why can't you simply "lift and shift" your Azure AI Search index to another cloud if you later exit?
A: Because the embeddings, vector profiles, skillset chunking config, OData filter syntax, semantic configuration, and identity-filter plumbing are all written against Azure's proprietary APIs; the raw documents are portable but the surrounding retrieval integration must be re-implemented on the target. You would re-chunk, re-embed, rebuild the index schema, and re-create ACL enforcement. This re-implementation cost is the concrete meaning of lock-in for the exit plan.

Q: A query with `queryType=semantic` returns HTTP 403 / feature-not-available. First thing to check?
A: Confirm the semantic ranker is actually enabled on the search service and that the service tier supports it, because semantic ranking is a premium feature you must turn on (selecting the free or standard plan) before any `queryType=semantic` query will run. A 403/feature error usually means the capability is off, not that the query is malformed. Enable the semantic ranker plan, then retry.

Q: You are being billed for semantic queries you thought were free. What likely tripped the meter?
A: A query is charged when `queryType=semantic` is set together with a non-empty `search` string; you are only free when the search string is empty (`search=*`). So filter-only or wildcard requests stay free, but any real keyword/text query with semantic enabled bills per qualifying query — likely you left `queryType=semantic` on for ordinary text searches. Restrict semantic mode to the prose queries that actually benefit.

Q: A semantic query returns no `@search.captions`. Give two plausible causes.
A: First, captions are only produced when the semantic configuration includes content fields with enough prose for the extractive model to summarize, so thin or identifier-only fields yield none. Second, you may not have requested them — captions/answers can require the appropriate parameters (`captions`, `answers`) and a non-empty semantic query. Check both the semantic configuration's field list and that the request actually asked for captions.

Q: Why might raising the semantic configuration's content beyond ~2,000 tokens per document not improve reranking?
A: The semantic ranker's summarization step accepts only up to about 2,000 tokens per document (assembled from the configured `title`/`keyword`/`content` fields, with the summary capped around 2,048 tokens), and trims the rest, so content past that window never reaches the L2 model. Adding more text late in the field order just gets truncated. To influence reranking you must place the discriminating text early, not merely add more of it.

Q: A relevant KIID is retrieved at rank 60 by hybrid search but never appears after semantic reranking. Why, and what is the fix?
A: The semantic ranker only reorders the top 50 first-stage candidates, so a document at rank 60 is outside that window and cannot be promoted no matter how relevant — the reranker never sees it. The fix is at the retrieval stage: improve hybrid recall (better embeddings, query rewrite, chunking, or filters) so the KIID enters the top 50. Reranking cannot recover documents first-stage retrieval ranked below 50.

Q: How does enabling semantic query rewrite change the cost and latency profile of a retrieval call?
A: Query rewrite generates up to 10 semantically similar variant queries, each executed on the engine and then rescored by the semantic ranker, multiplying retrieval work and adding latency, and it is part of the billed semantic feature set. It can lift recall on misspelled or awkwardly phrased fund queries (e.g. a fat-fingered ISIN context). You enable it deliberately, weighing the recall gain against the extra per-query time and cost.

Q: Distinguish `@search.score`, `@search.rerankerScore`, and `@search.rerankerBoostedScore` in an Azure AI Search response.
A: `@search.score` is the first-stage relevance score (BM25 for text, RRF for vector/hybrid); `@search.rerankerScore` is the semantic ranker's L2 score on a `0.0`–`4.0` scale; and `@search.rerankerBoostedScore` reflects additional scoring-profile boosts applied on top of the reranker where configured. Results order by the highest-stage score present. Knowing which score drove ordering tells you whether retrieval, semantic reranking, or a boost is responsible for a ranking you are debugging.

Q: Why are coarse rather than razor-thin `@search.rerankerScore` thresholds recommended for a production gate?
A: Because the docs note the `@search.rerankerScore` distribution can shift slightly with infrastructure conditions and across model updates, a cut at, say, exactly `2.73` can behave inconsistently and silently change recall when the model is updated. A coarse threshold (e.g. drop below `1.5`) is robust to that drift. For a regulated gate you also document and version any threshold so a behaviour change is traceable.

Q: If hybrid+RRF already ranks the right document first for most fund queries, what does turning on the semantic ranker realistically buy?
A: Little to nothing on those queries — when RRF already puts the correct document at rank 1, an L2 rerank cannot improve the top result, so you pay the per-query meter for no measurable gain. The ranker only helps where the right passage is buried below rank 1 (prose/intent queries), not where keyword/identifier matching already nails it. This is why you gate the ranker on a measured lift, not on faith.

Q: For "list the sub-funds whose SRRI is 6 or 7," is the right tool the semantic ranker, a metadata filter, or both?
A: Primarily a metadata filter (or structured query) on an `srri` numeric field — exact attribute selection is a filter problem, not a semantic-relevance problem, so an OData filter like `srri ge 6` answers it deterministically. The semantic ranker adds nothing because there is no prose-intent ambiguity to resolve. This illustrates matching the technique to the query type rather than reflexively enabling reranking.

Q: What index design lets you serve both "find the doc for ISIN `LU0123456789`" and "explain the liquidity policy" well from one Azure index?
A: Store the ISIN and other identifiers in `searchable`/`filterable` text fields (so keyword/BM25 and exact filters nail the code), store the prose in rich `content` fields included in the semantic configuration, and add a `Collection(Edm.Single)` vector field over the prose with a hybrid profile. Then hybrid+RRF handles both arms and the semantic ranker can be toggled for the prose query only. One index, field-level design, covers identifier and intent queries together.

Q: Compare how Azure's semantic ranker and your open cross-encoder reranker handle multilingual fund documents (e.g. French KIIDs, German factsheets).
A: Azure's semantic ranker uses Microsoft's multilingual deep-learning models, so it reranks French/German/English content without you sourcing a model, and honors synonym maps configured on the fields. Your open cross-encoder must be a multilingual checkpoint you select, host, and evaluate yourself, giving control but adding ops. For a Luxembourg corpus spanning several languages, the managed multilingual coverage can be a genuine convenience worth pricing in the eval.

Q: What is the cost implication of running the semantic ranker on a high-QPS public investor portal versus an internal analyst tool?
A: On a high-QPS public portal every qualifying non-empty semantic query bills, so volume directly multiplies the meter and can dominate cost, making it economically risky to leave on globally. An internal analyst tool with modest, prose-heavy query volume may stay within the free monthly allowance or low standard cost while gaining real relevance. So the same feature's value flips with query volume and mix — the eval's cost row must use expected QPS, not a per-query glance.

Q: How does the managed-vs-open decision change if you must also satisfy the EU AI Act's transparency and logging duties for the retrieval-augmented system?
A: The AI Act pushes you toward systems where you can log inputs, retrieved context, and outputs and explain behaviour; integrated vectorization's chunking opacity can weaken your ability to reconstruct "exactly what text grounded this answer," so you must confirm the platform exposes adequate retrieval traces or keep chunking in-house. A self-hosted stack makes that logging fully yours but shifts the burden onto your ops. Either way you document how the chosen architecture meets traceability, since the obligation is on you as deployer, not the cloud.

Q: Final applied check: write the one-sentence conclusion for a corpus that is investor-restricted, prose-heavy, and high-QPS but where you lack ops capacity to build ACL trimming.
A: "Because mandatory investor-level ACL trimming must be enforced and we lack capacity to build and prove it ourselves, Azure AI Search wins — and given the prose-heavy queries we enable the semantic ranker only on the internal, lower-QPS paths where an eval shows it lifts relevance, while accepting integrated-vectorization opacity and registering the platform as a DORA third-party dependency." This names the single flipping requirement (provable ACL trimming), scopes the semantic-ranker meter to where it pays, and acknowledges the opacity and resilience costs in one breath. That is the decision the lesson asks you to defend.

Q: Why must the index-time embedding model and the query-time vectorizer use the same model, and what symptom appears if they diverge?
A: Vector similarity is only meaningful within a single embedding space, so document vectors and query vectors must come from the same model and dimension, or cosine/dot-product distances become noise. If they diverge you see plausible-looking but semantically wrong nearest neighbors — relevant fund documents missed and unrelated ones surfaced — while no error is raised because the dimensions may still match. The fix is to pin one model deployment for both indexing and the query-time vectorizer.

Q: How does the `k` (or `top`) parameter on a vector query interact with the semantic ranker's top-50 ceiling?
A: A vector query's `k` controls how many nearest neighbors retrieval returns, but the semantic ranker reranks at most the top 50 of the first-stage (RRF-fused) result regardless of how large `k` is, so requesting `k=200` does not give the reranker 200 candidates. Setting `k` far above 50 wastes retrieval work for the reranking stage while still only reordering 50. Tune `k` for the fusion and downstream needs, and remember the L2 window is fixed at 50.

Q: For a fund-document RAG, what is the practical risk of relying on `RetrieveAndGenerate`'s built-in citations rather than your own generation layer?
A: `RetrieveAndGenerate` produces the answer and citations inside the managed service, so the prompt, model, faithfulness checks, PII masking, and the exact citation logic are the platform's, not yours, which is hard to defend when a regulator asks how a NAV figure was grounded and masked. Using `Retrieve` plus your own generation keeps those controls and your Langfuse-style tracing in your boundary. The convenience of one-call answers trades away the generation-side governance a regulated fund workflow usually needs to own.
