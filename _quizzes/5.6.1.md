# Phase 3 · 5.6.1 + 5.6.3 Orchestration (Airflow + Dagster, task- vs asset-oriented) — 150 self-test questions

Q: What is the core job of a data orchestrator, and why does this lesson call it the platform's "nervous system"?
A: An orchestrator schedules, sequences, and monitors the units of work in a data platform, guaranteeing that tasks run in the right order, at the right time, with the right dependencies satisfied. It is the nervous system because every other component — ingestion, transformation, quality checks — fires under its control, so a failure there silently corrupts everything downstream. In a fund platform it is what guarantees the NAV pipeline ran, in order, exactly once per valuation day rather than producing yesterday's prices in today's report.

Q: What is an Airflow DAG, and why "directed acyclic graph"?
A: A DAG (directed acyclic graph) is the Airflow object that defines a pipeline as a set of tasks with directional dependencies between them and no cycles. "Directed" means each edge points from an upstream task to a downstream one; "acyclic" means a task can never depend on itself transitively, which would make the schedule undefinable. Airflow uses the DAG structure to decide what can run in parallel and what must wait.

Q: What is the difference between an operator and a task in Airflow?
A: An operator is a template class describing a single unit of work — for example `BashOperator`, `PythonOperator`, or a provider's `SQLExecuteQueryOperator` — while a task is a specific instantiation of an operator inside a DAG. You author with operators, but at runtime Airflow schedules task instances, one per logical date the DAG covers. Confusing the two leads to the common error of thinking the operator "runs" when in fact it is the task instance that does.

Q: What is a sensor in Airflow and what problem does it solve?
A: A sensor is a special operator that waits for a condition to become true before letting downstream tasks proceed — for example a file landing in object storage, a partition appearing, or an external job finishing. It solves the problem of clock-time scheduling being wrong when a pipeline must start on data readiness rather than at a fixed hour. A `FileSensor` or `S3KeySensor` waiting for the day's price file is the canonical fund-data use.

Q: What is the difference between `poke` and `reschedule` mode for a sensor, and why does it matter operationally?
A: In `poke` mode a sensor holds a worker slot for its entire wait, repeatedly checking the condition; in `reschedule` mode it releases the slot between checks and is re-queued, freeing capacity. `reschedule` is strongly preferred for long waits because a fleet of `poke` sensors can deadlock the cluster by occupying every slot. The gotcha is that `reschedule` has coarser timing granularity, so it is unsuitable for sub-minute polling.

Q: What does it mean for an Airflow task to be idempotent, and why is it non-negotiable?
A: An idempotent task produces the same result whether it runs once or many times for the same logical interval — typically by overwriting or deleting-then-inserting a partition rather than blindly appending. It is non-negotiable because Airflow reruns tasks on retry, on backfill, and on manual clear, so a non-idempotent task silently double-counts. A NAV aggregation that `INSERT`s without first clearing the target day's rows will inflate totals on every rerun.

Q: How would you make a "load today's prices into a table" task idempotent?
A: Scope the write to the run's logical interval and make it overwrite that scope: delete the target partition (or rows for that valuation date) and then insert, or use an upsert/`MERGE` keyed on the natural key. The pattern is "delete-write" or "truncate-insert" per partition, never bare `INSERT INTO`. This way a retry or a backfill of that date recomputes the same partition rather than appending duplicates.

Q: What is the Airflow `logical_date` (formerly `execution_date`) and why is it a frequent source of confusion?
A: The `logical_date` is the start of the data interval a run represents, not the wall-clock time the run actually executes. For a daily DAG, the run for the interval covering June 12 typically executes after that interval ends, so the run "for June 12" starts on June 13. The confusion is treating it as "now"; idempotent tasks must template their queries on this interval, not on `datetime.now()`.

Q: Why should an Airflow task never call `datetime.now()` to decide which data to process?
A: Because `datetime.now()` returns the moment of execution, which differs on retries, backfills, and delayed runs, breaking idempotency and reproducibility. The task would process a different window every time it ran, making a backfill of a past date pull today's data. Instead you template on the run's data interval (`{{ data_interval_start }}` / `{{ data_interval_end }}`) so the same logical date always recomputes the same window.

Q: What is the Airflow scheduler responsible for, and what is the executor responsible for?
A: The scheduler parses DAGs, evaluates dependencies and schedules, and decides which task instances are ready to run; the executor is the mechanism that actually places those task instances onto compute and runs them. Separating the two lets the same scheduling logic drive very different runtimes. A common mistake is blaming "the scheduler" for slow task execution when the bottleneck is really an under-provisioned executor.

Q: Compare the `LocalExecutor`, `CeleryExecutor`, and `KubernetesExecutor`.
A: `LocalExecutor` runs tasks as subprocesses on the scheduler host — simple, single-machine, fine for dev or small workloads. `CeleryExecutor` distributes tasks to a pool of long-running worker processes via a message broker (Redis/RabbitMQ), giving horizontal scale with persistent workers. `KubernetesExecutor` launches each task in its own ephemeral pod, giving per-task isolation and resource limits at the cost of pod-startup latency.

Q: When would the `KubernetesExecutor` be the wrong choice despite its isolation benefits?
A: When tasks are numerous and short-lived, because the per-task pod startup overhead (image pull, scheduling, init) can dwarf the actual work, making throughput collapse. Heterogeneous-resource, long-running, or strongly-isolated tasks justify the overhead; thousands of two-second tasks do not. For that profile a `CeleryExecutor` with warm workers, or batching the work into fewer tasks, performs far better.

Q: What does the `catchup` parameter control on an Airflow DAG?
A: `catchup` controls whether, when a DAG is unpaused or newly added, the scheduler creates runs for all past intervals between the start date and now, or only the most recent one. With `catchup=True` it fills in every missed interval; with `catchup=False` it schedules only the latest interval. It governs automatic backfilling on activation, distinct from a manually triggered backfill.

Q: In recent Airflow, what is the practical default behavior of `catchup`, and how can you set it globally?
A: Modern Airflow guidance and the example projects favor `catchup=False`, and you can set the cluster-wide default via the config `scheduler.catchup_by_default=False` so DAGs do not silently backfill on first activation. The reason is that an accidental `catchup=True` on a DAG with an old start date can spawn hundreds of runs at once and overwhelm the cluster. Always set `catchup` explicitly per DAG rather than relying on memory of the default.

Q: What is the difference between `catchup` and a `backfill`?
A: `catchup` is the scheduler automatically creating runs for missed intervals when a DAG becomes active, whereas a backfill is an explicit, operator-initiated rerun of a chosen date range — historically via the `airflow dags backfill` CLI. Catchup is a property of normal scheduling; backfill is a deliberate operations action, typically after a logic fix. Both rely on the same idempotency contract to be safe.

Q: You fixed a bug in a transformation and need to recompute the last 30 days in Airflow. How do you do it safely?
A: Run a backfill over that 30-day date range — for example `airflow dags backfill` with `--start-date` and `--end-date` (or clear those task instances in the UI) — so each daily interval is recomputed against the fixed logic. Safety depends entirely on the tasks being idempotent so each partition is overwritten, not appended. Afterward you verify by checking which partitions changed (row counts, checksums, or a reconciliation total).

Q: What is Airflow data-aware scheduling, and how does it differ from cron scheduling?
A: Data-aware scheduling triggers a DAG when an upstream piece of data is updated rather than when a clock time arrives. A producer task marks a logical data object as updated on completion, and any DAG that consumes that object is scheduled in response. This replaces fragile time guesses ("the file should be ready by 6am") with a true dependency on readiness.

Q: In Airflow 3, what was the "Dataset" concept renamed to, and why does the rename matter conceptually?
A: In Airflow 3 the feature formerly called a Dataset was renamed to an Asset, aligning Airflow's vocabulary with the asset-oriented model and broadening it toward event-driven scheduling. The rename matters because it signals that the consumer-DAG-triggers-on-data idea is now first-class, not a bolt-on, and it eases comparison with Dagster's asset language. Older tutorials using `Dataset` map onto the new `Asset` API.

Q: What is an `AssetWatcher` in Airflow 3 and what new capability does it unlock?
A: An `AssetWatcher` continuously monitors an external source such as a message queue or storage event stream and updates the corresponding asset — and thus triggers consumer DAGs — the moment a relevant event arrives. It unlocks genuinely event-driven workflows, moving beyond both fixed cron and intra-Airflow producer-consumer links. For fund data this means a DAG can fire on the actual arrival of a SWIFT message or a price file rather than polling.

Q: How does asset/dataset-based scheduling let you express "rebuild gold when silver finishes" without a cron?
A: You declare the silver tables as assets that the silver DAG's tasks update on success, and you schedule the gold DAG on those assets rather than on a `schedule_interval`. Airflow then runs gold automatically whenever the upstream silver assets are all updated. This removes the brittle "gold runs at 7am and hopes silver finished by then" coupling that breaks the first time silver is slow.

Q: What is XCom in Airflow and what is it intended for?
A: XCom (cross-communication) is Airflow's mechanism for tasks to pass small pieces of metadata to one another — an identifier, a count, a path — stored in the metadata database. It is intended for lightweight control data, not for moving the actual datasets between tasks. Returning a value from a task pushes an XCom that downstream tasks can pull by key.

Q: Why is "XCom abuse" listed as an orchestrator anti-pattern?
A: Because XCom is backed by the metadata database and was designed for tiny values, pushing large payloads — a DataFrame, a file's contents, thousands of rows — bloats and can destabilize that database and the scheduler. The correct pattern is to write the data to durable storage (object store, table) and pass only a reference through XCom. Treating XCom as a data bus is a reliable way to bring an Airflow deployment to its knees.

Q: Why is "business logic in DAG files" an anti-pattern, and what is the fix?
A: DAG files are parsed by the scheduler frequently — on every parse loop — so heavy logic, imports, or external calls at module top level slow parsing and can time out, and they make the pipeline untestable. The fix is to keep DAG files thin: define structure and dependencies there, and put the actual transformation logic in importable, unit-testable functions or external tools. A DAG file should read like a table of contents, not the book.

Q: What goes wrong when top-level code in a DAG file makes network or database calls at parse time?
A: The scheduler executes top-level code every parse cycle, so a network/DB call there runs constantly — not when the task runs — hammering the external system and slowing or failing DAG parsing. This is distinct from code inside a task's callable, which runs only at execution time. The rule is: nothing expensive or side-effecting at module scope; defer it into the operator's execution.

Q: What are sensible retry settings for a flaky external dependency, and why not just set retries very high?
A: Configure a modest `retries` count with an exponential `retry_delay` (and optionally `retry_exponential_backoff`) so transient blips self-heal without a human. You avoid very high retry counts because they mask real, persistent failures, delay the page that should wake on-call, and can amplify load on a struggling upstream. Retries are for transient faults; a deterministic bug will fail identically every attempt.

Q: What is an Airflow SLA (or its successor concept), and what does it give on-call?
A: An SLA expresses how long a task or DAG is expected to take, so that if it overruns, Airflow records a miss and can alert — surfacing "the NAV pipeline is late" before a human notices the report is missing. It gives on-call a proactive signal about lateness distinct from outright failure. The gotcha is that an SLA miss is not the same as a failure: a task can be on time yet wrong, or late yet eventually succeed.

Q: What information should a sane on-call page contain when a pipeline task fails?
A: It should name the failing DAG/asset and task, the logical date/partition affected, the error class and a link to the logs, the blast radius (what downstream is now blocked or stale), and a hint of the runbook step. The goal is that the responder can triage without first reconstructing context at 3am. A page that says only "task failed" with no partition or downstream impact wastes the critical first minutes.

Q: Define "software-defined asset" (SDA) in Dagster.
A: A software-defined asset is a declarative object describing a persistent thing your pipeline produces — a table, a file, an ML model — together with the computation that materializes it and its dependencies on other assets. You declare the asset and what it depends on; Dagster derives the execution graph from those declarations. This inverts Airflow's task-first model: you describe the data products, and the orchestration falls out.

Q: What is the fundamental difference between task-oriented and asset-oriented orchestration?
A: Task-oriented orchestration (classic Airflow) centers on the work to do — "run this task, then that task" — and any notion of the data produced is implicit. Asset-oriented orchestration (Dagster) centers on the data products themselves — "this table depends on those tables" — and derives the tasks needed to keep them up to date. The shift gives you lineage and freshness as built-in properties rather than things you bolt on.

Q: Give a one-line fund-industry example each of task-oriented and asset-oriented thinking.
A: Task-oriented: "at 18:00 run extract_prices, then run compute_nav, then run publish_report" — the schedule is a sequence of jobs. Asset-oriented: "the nav_per_share asset depends on validated_prices and shares_outstanding, rebuild it whenever either upstream changes" — the schedule is implied by the data graph. The asset framing answers "is today's NAV up to date and what fed it?" directly.

Q: Why does asset-orientation give you lineage "for free"?
A: Because in the asset model you explicitly declare each asset's dependencies in code, the orchestrator already holds the full producer-consumer graph and can render it as lineage without a separate metadata-scraping step. In task-orientation the data dependencies live only in the heads of the engineers, so lineage must be reconstructed externally. The declaration that drives scheduling is the same declaration that draws the lineage.

Q: What is an asset partition in Dagster?
A: A partition slices an asset into independently-materializable units along a dimension — most commonly a date, e.g. one partition per valuation day. Each partition has its own materialization status and can be backfilled or rematerialized in isolation. Daily partitions are exactly how you model "one NAV computation per business day" and recompute a single bad day without touching the rest.

Q: How do partitions make a Dagster backfill precise?
A: Because each partition is an independent unit with its own materialization, a backfill targets a specific set of partition keys (e.g. the last 30 dates) and recomputes exactly those, leaving others untouched. You can then list which partition keys were materialized to prove the scope. This is more legible than clearing a date range of task instances because the unit of recompute is the data slice itself.

Q: What is a Dagster asset check, and how does it differ from a freshness check?
A: An asset check is a test bound to a specific asset that runs against its data — e.g. "row count > 0", "no null ISINs", "NAV total reconciles" — and reports pass/fail/severity alongside the asset. A freshness check specifically verifies the asset was updated recently enough. Binding the check to the asset means its result appears in lineage and gates downstream automation, unlike a standalone test in a separate system.

Q: How would you express a NAV-total reconciliation as an asset check?
A: Define an asset check on the gold NAV asset that recomputes the fund-level total from the underlying share-class rows and compares it, within tolerance, to an independently sourced control total, failing with `ERROR` severity on mismatch. Bound to the asset, the check's red status is visible in the catalog and can block publication. The mechanism turns "we trust the numbers" into a structural gate that fails loudly when share-class NAVs do not sum to the fund total.

Q: What does Dagster's freshness checking do, and which factory builds an update-time-based one?
A: Freshness checking flags an asset as overdue when it has not been updated within an acceptable window, turning staleness into an explicit alertable signal. The factory `build_last_update_freshness_checks` builds a check based on the asset's last-update time against a maximum allowed lag. You must also wire a sensor or schedule (e.g. `build_sensor_for_freshness_checks`) so the check actually evaluates rather than sitting dormant.

Q: A Dagster freshness check never fires even though the asset is stale. What is the first thing to check?
A: Confirm that something is actually evaluating the check — freshness checks do not run themselves, so you need a schedule or the `build_sensor_for_freshness_checks` sensor enabled. A defined-but-unevaluated check is the most common cause of silent staleness. Second, verify the lag threshold and the asset's recorded last-update events are what you expect.

Q: What is declarative automation in Dagster?
A: Declarative automation lets you attach a condition to each asset describing when it should be kept up to date, and Dagster derives "what needs to run" from the assets' states rather than from hand-wired cron triggers. You declare intent ("eager: update whenever a dependency updates") and the daemon computes and launches the necessary materializations. It replaces a web of explicit sensors and schedules with a per-asset policy.

Q: Which API powers Dagster declarative automation, and which older API did it replace?
A: Declarative automation is powered by `AutomationCondition`, attached to assets, which replaced the now-deprecated `AutoMaterializePolicy`. `AutomationCondition`s are composable boolean expressions, making them far more customizable and operable at scale than the old policy enum. Existing `AutoMaterializePolicy` objects are auto-converted to `AutomationCondition`s, so migration can proceed asset by asset.

Q: What does `AutomationCondition.eager()` do, and where would you use it?
A: `AutomationCondition.eager()` tells Dagster to update an asset automatically whenever any of its dependencies are updated, propagating upstream changes downstream without a clock trigger. You use it to wire "rebuild gold when silver materializes" so the gold layer rebuilds on data readiness, not on a cron guess. It is the asset-model equivalent of Airflow asset-based scheduling.

Q: How would you express "rebuild daily on a cron, OR whenever the code version changes" in Dagster?
A: Compose conditions with boolean operators, e.g. `AutomationCondition.on_cron("@daily") | AutomationCondition.code_version_changed()`, and attach the result to the asset. Dagster evaluates the combined condition and materializes when either branch is satisfied. Composability is the headline advantage of `AutomationCondition` over the old single-policy model.

Q: Why is "rebuild gold when silver materializes" with declarative automation better than a cron offset?
A: A cron offset ("run gold at 07:00 because silver usually finishes by 06:45") is a guess that breaks the first time silver is late, producing gold from stale or partial inputs. Declarative automation with `eager` ties gold's rebuild to the actual materialization event of silver, so gold runs exactly when its inputs are ready and never before. This converts a timing assumption into a data-readiness fact.

Q: How does Dagster's local development loop typically differ from Airflow's?
A: Dagster emphasizes a fast local loop where you run the UI (`dagster dev`) and materialize assets interactively, with assets being plain Python functions that are straightforward to unit test in isolation. Airflow's classic loop centers on a scheduler parsing DAG files and is historically heavier to iterate on locally. The asset-as-function design is a large part of why the Dagster dev loop feels tighter.

Q: What command starts the Dagster development server, and what does it give you?
A: `dagster dev` starts the local web UI plus the daemon, giving you the asset graph, the ability to materialize assets and partitions on demand, check results, run logs, and the automation evaluation view. It is the primary inner-loop tool for building and debugging assets locally. The browser-based asset graph is where you see lineage and freshness without extra tooling.

Q: What is a Dagster job, and how does it relate to assets?
A: A job in Dagster is a selectable, runnable target — often a selection of assets to materialize together, or a graph of ops — that schedules and sensors can attach to. With the asset model you frequently define jobs as asset selections (e.g. "all silver assets") rather than authoring ops directly. Jobs give you a unit to schedule and to scope a run, layered on top of the asset graph.

Q: What is the role of the Dagster daemon?
A: The daemon is the long-running process that drives schedules, sensors, run queuing, and automation-condition evaluation — it is what turns declared intent into launched runs. Without the daemon running, schedules do not tick and `AutomationCondition`s never evaluate, so assets that should auto-update sit still. It is the Dagster analog of the part of the Airflow scheduler that emits runs.

Q: Why might an architect prefer Dagster's asset model for a regulated fund platform?
A: Because lineage and freshness are intrinsic, an auditor's question "which inputs produced this NAV, and was each fresh?" is answered directly from the orchestrator rather than reconstructed. Asset checks bind data-quality and reconciliation gates structurally to the products themselves. This makes the platform's correctness story demonstrable, which is exactly what DORA-style operational-resilience and audit expectations push toward.

Q: Why might a fund-admin shop nonetheless run Airflow rather than Dagster?
A: Airflow is the incumbent with the largest hiring pool, the broadest provider/operator ecosystem, and managed offerings on every cloud, so it is often the lower-risk institutional default. A six-person team inheriting an Airflow estate, or one whose vendors ship Airflow operators, may rationally stay. The architect's job is to weigh that incumbency and skills reality against Dagster's lineage/freshness advantages, not to assume the newer tool wins.

Q: What is Prefect, in one or two lines of market awareness?
A: Prefect is a Python-native orchestrator that lets you turn ordinary functions into flows and tasks with decorators, emphasizing a dynamic, code-first model and a hybrid execution where the control plane is hosted but work runs in your infrastructure. It is a credible Airflow alternative, particularly where teams want minimal DAG boilerplate. For this lesson it is awareness-level, not a hands-on target.

Q: What kind of authoring experience does Mage offer, and who is it aimed at?
A: Mage is an open-source pipeline tool whose differentiator is a notebook-like, block-based authoring surface where each block is a step you build and preview interactively, with scheduling built in. It targets teams wanting a lighter, more visual build loop than classic Airflow DAG authoring. For this lesson it is awareness-level — know its block-based, interactive pitch and that it competes in the same lightweight-orchestrator space as Prefect.

Q: What is Azure Data Factory (ADF), and why does it appear as "the Azure reality"?
A: ADF is Azure's managed data-integration and orchestration service, offering pipelines, a large library of connectors, and a low-code visual authoring surface, billed per activity/integration-runtime usage. It appears as the Azure reality because in an Azure-committed enterprise it is frequently already paid for and mandated, so any FOSS recommendation must survive comparison to it. The architect must be able to map a Dagster/Airflow design onto ADF and argue the trade-offs.

Q: What is "Workflow Orchestration Manager" in the Azure context?
A: Workflow Orchestration Manager is Azure Data Factory's managed Apache Airflow offering, letting you run Airflow DAGs as a service within ADF rather than self-hosting the scheduler. It is Azure's answer to "we want real Airflow but managed", comparable in spirit to AWS MWAA and GCP Cloud Composer. It matters because it can blunt the "but ADF is already in our agreement" objection while still giving you genuine Airflow.

Q: Name the three big-cloud managed-Airflow services and their providers.
A: AWS offers MWAA (Managed Workflows for Apache Airflow), GCP offers Cloud Composer, and Azure offers managed Airflow via Workflow Orchestration Manager within Data Factory. All three run upstream Airflow so your DAGs are largely portable across them. The differences that matter at build-vs-buy level are pricing model, upgrade cadence, and the degree of lock-in to each cloud's surrounding services.

Q: What dimensions belong in a build-vs-buy evaluation of a managed orchestrator?
A: Pricing model (per-environment, per-activity, per-DCU/worker, idle cost), upgrade cadence and how quickly new engine versions land, lock-in to the surrounding cloud ecosystem, operational burden removed versus control surrendered, and the team's skills. You weigh the saved ops effort against reduced flexibility and recurring cost. For a small team the "we don't want to run a scheduler at 2am" factor often dominates.

Q: What is `pg_cron`, and when is it the right tool instead of Airflow or Dagster?
A: `pg_cron` is a Postgres extension that schedules SQL commands inside a single database on a cron schedule. It is the right tool when the entire need is periodic SQL in one database — refreshing a materialized view, a nightly cleanup, a scheduled `VACUUM` — with zero extra infrastructure to run. It deliberately stays inside that database and adds no new moving parts.

Q: When is `pg_cron` the wrong tool, and what do you reach for instead?
A: It is wrong as soon as you have cross-system DAGs, inter-task dependencies, backfills, retries, or lineage, because `pg_cron` has no dependency graph and no observability — it just fires SQL on a timer. For anything with real orchestration semantics you reach for Dagster or Airflow. Using `pg_cron` to chain multi-step, multi-system pipelines reinvents an orchestrator badly and blindly.

Q: A daily Airflow DAG produced no runs after you unpaused it, even though its start date is months ago. First thing to check?
A: Check `catchup` and the schedule/start-date interaction: with `catchup=False` the scheduler creates only the latest interval, and if no full interval has elapsed since the start date plus schedule, no run is due yet. Confirm the DAG is actually unpaused and the scheduler is healthy. The classic trap is expecting historical runs to appear when `catchup=False` deliberately suppresses them.

Q: An Airflow sensor seems to "hang forever" and your cluster slows to a crawl. What is the likely cause?
A: Likely a fleet of `poke`-mode sensors each holding a worker slot while waiting, starving the executor of capacity — a sensor deadlock. The waiting condition may never have been met, or there simply aren't enough slots for both sensors and real work. The fix is to switch long waits to `reschedule` mode and/or set a `timeout`, and to consider data-aware/event-driven triggering instead of polling.

Q: After a 30-day backfill you find some daily totals doubled. What does that tell you?
A: It tells you the task is not idempotent — it appended to the target partition instead of overwriting it, so each backfilled date added a second copy on top of the existing one. The fix is to make the write delete-then-insert (or upsert) scoped to the run's partition before reprocessing, then re-backfill from clean. Doubling after a rerun is the signature symptom of a missing idempotency contract.

Q: Your Airflow scheduler is slow and DAGs appear with a lag. The DAG files import heavy libraries at the top. Connection?
A: Yes — the scheduler parses every DAG file on each parse loop, so heavy top-level imports and any top-level work run repeatedly and inflate parse time, delaying when runs are scheduled. Move heavy imports and logic inside task callables so they execute only at task runtime, and keep module scope thin. This is the "business logic in DAG files" anti-pattern manifesting as scheduler lag.

Q: In Dagster you ran a 30-day backfill after a logic fix. How do you prove exactly which partitions recomputed?
A: Inspect the materialization records for the asset and list the partition keys with new materialization timestamps from the backfill — the asset's partition status in the UI or the event log shows precisely which dates were rematerialized. Because each partition is an independent unit, the set of recomputed keys is explicit, not inferred. You can cross-check by diffing row counts or a reconciliation total per partition before and after.

Q: Why does the asset model make "which partitions recomputed and why" easy to answer, where task-orientation makes it hard?
A: In the asset model the partition is the unit of materialization and each one carries its own status and event history, so the recomputed set is literally enumerable. In task-orientation you cleared a range of task instances whose relationship to data slices is conventional, not enforced, so you must reason about it externally. The asset model records the data-level fact; the task model records only the job-level fact.

Q: What does "exactly once per valuation day" mean for a NAV pipeline, and how does the orchestrator help enforce it?
A: It means the NAV for a given valuation date is computed once, completely, in dependency order, and not silently recomputed or skipped — the report reflects that day's data and only that day's. The orchestrator enforces it by tying the run to the valuation-date partition, gating downstream steps on upstream success, and making reruns idempotent so a retry corrects rather than duplicates. Without that discipline a missed upstream file yields yesterday's prices in today's report.

Q: How does an orchestrator catch a "missed upstream file silently produces yesterday's prices" failure?
A: By making the consuming step depend on the actual arrival/freshness of the file — a sensor, an asset update event, or a freshness check — so absence blocks or alerts rather than letting a stale prior partition be reused. The dependency turns "file didn't arrive" into a visible failed/overdue state instead of a quiet success on old data. The alternative, a fixed-time job that reads "whatever is there", is exactly how the transfer agent finds out from a client.

Q: What does idempotency buy you specifically in a regulated restatement scenario?
A: It lets you rerun a corrected computation for an affected valuation date and trust that the result replaces the bad partition cleanly, with no duplication, giving a reproducible recompute you can defend to an auditor. Combined with partitioning, you can recompute exactly the impacted days. Without idempotency, a restatement rerun risks producing new, differently-wrong numbers, compounding the original error.

Q: How would an asset check encode the rule "every published share class must have an ISIN"?
A: Define an asset check on the published holdings/NAV asset asserting that no row has a null or malformed `ISIN`, failing with error severity if any do. Bound to the asset, a violation is visible in lineage and can block publication of that partition. This turns a data-quality rule into a structural gate so a share class missing its `ISIN` never silently reaches the EMT/EPT consumer.

Q: What is an ISIN, and why is it a natural key to validate in a fund orchestration check?
A: An ISIN (International Securities Identification Number) is the 12-character standard identifier for a security, including fund share classes. It is a natural key because downstream consumers — EMT/EPT files, transfer-agency records, reporting — join on it, so a missing, duplicated, or malformed ISIN corrupts the whole feed. An asset check validating ISIN presence and format guards that join surface at the point of production.

Q: What are EMT/EPT files, and where does orchestration fit in producing them?
A: EMT (European MiFID Template) and EPT (European PRIIPs Template) are standardized data exchange files conveying product and cost information for funds to distributors and PRIIP manufacturers. Orchestration ensures the upstream NAV, cost, and reference-data assets are fresh and validated before the EMT/EPT generation step runs, and that it runs once per cycle. Asset checks on completeness (every share class present) protect the feed from silent gaps.

Q: How does data-aware/event-driven scheduling improve handling of an incoming SWIFT message in a fund pipeline?
A: Instead of a clock-based job hoping the SWIFT message has landed, an event-driven trigger (an Airflow `AssetWatcher` on the queue, or a sensor) fires processing the instant the message arrives. This reduces latency and removes the "ran too early, message wasn't there" failure mode. The pipeline reacts to the actual data event rather than to an assumption about timing.

Q: What is a LEI, and why might an orchestrated reference-data pipeline track its freshness?
A: An LEI (Legal Entity Identifier) is a 20-character ISO 17442 code identifying a legal entity in financial transactions, and LEIs have a validity/renewal lifecycle. A reference-data pipeline tracks freshness so that lapsed or stale LEI records are flagged before they flow into regulatory reporting. A freshness check on the LEI reference asset turns "this entity's LEI hasn't been refreshed" into an alert rather than a downstream rejection.

Q: How does orchestration discipline relate to DORA's operational-resilience expectations?
A: DORA (the EU Digital Operational Resilience Act) pushes financial entities toward demonstrable, recoverable, monitored ICT operations, and an orchestrator with explicit dependencies, retries, alerting, lineage, and reproducible backfills is precisely the evidence of that control. Being able to show "the NAV pipeline ran in order, was monitored, and can be recomputed deterministically" is operational resilience made concrete. A pile of unmonitored cron jobs is the opposite of what DORA expects.

Q: A stakeholder asks "prove today's NAV report used the right prices". How does the right orchestration design answer this?
A: With asset lineage you trace the NAV asset back through its declared dependencies to the exact validated-prices partition and tag/snapshot used, and with freshness/asset checks you show those inputs passed their gates for that valuation date. The orchestrator already holds this graph, so the answer is a query, not a forensic reconstruction. This is the core argument for asset-oriented orchestration in a regulated context.

Q: Why is a cron-only "stack of scheduled scripts" risky for a NAV pipeline even if each script works?
A: Because cron has no dependency awareness, no retry/idempotency contract, no lineage, and no visibility into whether upstream actually finished — so a slow or missing upstream silently lets a downstream script run on stale data. Each script working in isolation says nothing about the pipeline being correct as a whole. The orchestrator exists precisely to encode and enforce the between-step guarantees cron cannot.

Q: What should the comparison memo for a 6-person fund-data team on Azure actually compare?
A: It should compare dev loop (iteration speed and testability), backfill ergonomics, lineage and freshness support, ongoing operational burden, and the team's skill demands across Dagster, Airflow, and ADF — ending in a defended recommendation. It must explicitly address the "ADF is already in our Azure agreement" objection rather than ignoring cost. The deliverable is a decision a CAB can act on, not a feature checklist.

Q: How do you counter "but ADF is already included in our Azure agreement" when recommending Dagster?
A: Acknowledge the sunk-cost reality, then argue on total cost and fit: weigh ADF's per-activity billing and weaker lineage/asset semantics against the engineering cost of building lineage, freshness, and testable transforms around it, and the team's productivity on each. If asset-level lineage and freshness materially reduce audit and incident cost for a regulated NAV pipeline, that value can outweigh ADF being "free". The honest answer may also be hybrid — Airflow on Azure's managed offering — if skills and lock-in dominate.

Q: What is the danger of putting non-idempotent business logic directly into operator parameters templated on the wrong date?
A: If the templated date is `now()` or a misunderstood `execution_date`, every retry or backfill processes a different or wrong window, and a non-idempotent write then layers inconsistent results on top of each other. You get partitions that neither match their label nor reproduce on rerun. Correctness requires both the right interval template (`data_interval_start/end`) and an idempotent write.

Q: What is the difference between a task failing and a task producing wrong-but-green output, and why does it matter for on-call?
A: A failing task is loud — it pages and blocks downstream — whereas wrong-but-green output passes, propagates, and is discovered only by a consumer or auditor later. It matters because retries and SLAs catch failures but not silent correctness errors. This is exactly why asset checks (reconciliation, row counts, ISIN completeness) exist: to convert silent data-wrong into a loud, gating failure.

Q: Why can a pipeline be "code-green" yet "data-wrong", and which orchestration feature catches it?
A: Schema and unit tests verify the code's shape and logic but not that the produced data is correct for this run — a valid query can still drop rows or shift a total. Asset checks that run against the actual materialized data (counts, reconciliations, distribution checks) catch the data-level defect the green code did not. Binding those checks to assets makes the catch structural and visible in lineage.

Q: How do retries, SLAs/freshness alerts, and asset checks divide the failure-handling responsibility?
A: Retries absorb transient, self-healing faults (a blip in a network call); SLAs/freshness alerts catch lateness and staleness even when nothing errored; asset checks catch data that is present and on time but wrong. Together they cover the three distinct failure shapes — flaky, late, and incorrect. Relying on retries alone leaves you blind to late and to wrong.

Q: What is a "freshness alert" meant to detect that a retry never would?
A: A freshness alert detects that an asset has not been updated within its acceptable window — staleness — which is invisible to retries because no task errored; the data simply did not refresh on time or at all. It surfaces "the NAV inputs are older than allowed" before a consumer reads stale numbers. Retries only fire on failures, so a quietly-skipped upstream slips past them entirely.

Q: Why is "one comparison DAG" in Airflow worth building even if you choose Dagster?
A: Because Airflow is the incumbent in most fund-admin shops, so market fluency — being able to read, author, and critique a real Airflow DAG with a sensor, idempotent tasks, and deliberate `catchup` — is a hiring and architecture-credibility requirement. Building one forces you to feel the task-oriented model concretely rather than describe it secondhand. It also makes your Dagster-vs-Airflow memo credible because you have run both.

Q: In Airflow, how do you wire a sensor, idempotent tasks, and deliberate catchup into one branch of a pipeline?
A: Start the branch with a sensor waiting on the input's readiness (file/partition/asset), follow with tasks whose writes are partition-scoped delete-then-insert so reruns are safe, and set `catchup` explicitly to match whether you want historical intervals filled. Each task templates on the run's data interval, never on `now()`. The result is a branch that starts on readiness, reruns safely, and backfills predictably.

Q: What does "declarative" mean in "declarative automation", contrasted with imperative triggers?
A: Declarative means you state the desired condition for an asset to be up to date and let the system figure out and execute what is needed, whereas imperative means you hand-wire each trigger, sensor, and schedule yourself. With declarative automation you say "keep gold eager on silver"; with imperative you write a sensor that watches silver and launches gold. The declarative form scales better and drifts less because intent, not plumbing, is the source of truth.

Q: Why does hand-wiring many sensors and schedules become an anti-pattern at scale?
A: Because each hand-wired trigger is a piece of bespoke plumbing that can drift out of sync with the actual data dependencies, multiplying maintenance and creating gaps where something silently fails to fire. As the asset count grows, the combinatorics of cross-dependencies overwhelm manual wiring. Declarative automation derives the triggers from declared dependencies, so the plumbing cannot drift from the graph.

Q: What is the relationship between Airflow asset-based scheduling and Dagster declarative automation conceptually?
A: Both move from clock-time triggering to data-readiness triggering: Airflow schedules consumer DAGs when producer tasks update assets, and Dagster materializes assets when their `AutomationCondition` (e.g. `eager`) is satisfied by upstream updates. They are the same instinct — react to data events, not the clock — expressed in each tool's model. Dagster's version is more deeply integrated because the asset is the primitive, not an annotation on tasks.

Q: How would you decide between daily-partitioned assets and a single unpartitioned asset for NAV data?
A: Use daily partitions when you need to compute, backfill, and reason about each valuation day independently — which is exactly the NAV case, since restatements and freshness are per-day. An unpartitioned asset suits data with no meaningful time slicing or where full recompute is cheap and total. For NAV, partitions give precise backfills and per-day freshness/audit, so they are the right default.

Q: What is the risk of over-partitioning an asset, and how do you spot it?
A: Over-partitioning (e.g. per-minute when you only ever consume daily) explodes the number of partitions, slowing the UI, the daemon's evaluation, and backfills, while adding no analytical value. You spot it when partition counts reach into the hundreds of thousands and automation evaluation or the asset graph becomes sluggish. Choose the partition grain to match how you actually query, backfill, and audit the data.

Q: Why should an asset check that gates publication usually have `ERROR` severity rather than `WARN`?
A: Because `ERROR` is the severity that should block downstream publication and page on-call, whereas `WARN` merely records a concern without stopping the flow. A reconciliation or ISIN-completeness failure on a NAV feed is publish-blocking, so it must be `ERROR`. Using `WARN` for a gate would let bad data through while creating the false comfort that "a check exists".

Q: When is `WARN` the appropriate severity for an asset check?
A: When the condition is a soft signal worth surfacing but not worth halting the pipeline — e.g. a row count slightly outside a typical range that is plausible, or an early-warning threshold ahead of a hard `ERROR` limit. You might pair a `WARN` and an `ERROR` check on the same asset at different thresholds. The principle is: block on what must not ship, alert-only on what merely merits a look.

Q: A Dagster asset shows as "stale" in the UI. What does staleness mean here?
A: Staleness means Dagster has detected that the asset's upstream dependencies or code version changed since the asset was last materialized, so its current materialization may no longer reflect its inputs. It is a derived signal from the dependency graph and versions, distinct from a freshness check on update time. It tells you a rematerialization is warranted to bring the asset back in line with its inputs.

Q: How does code-version awareness help Dagster decide what to rerun?
A: Dagster can track an asset's code version and mark dependents as stale when the version changes, and an `AutomationCondition` like `code_version_changed()` can trigger rematerialization on that change. This means a logic fix can automatically propagate recomputation to affected assets rather than relying on a human to remember. It ties "the transformation changed" directly to "recompute what depends on it".

Q: Why is "the same declaration drives both scheduling and lineage" a powerful property?
A: Because there is a single source of truth — the asset's declared dependencies — there is no second, drift-prone metadata system to keep in sync; what runs and what the lineage shows are guaranteed consistent. In task-oriented setups the lineage is reconstructed separately and can diverge from reality. One declaration, two derived outputs, no drift, is why asset-orientation reduces a whole class of "the docs say X but the pipeline does Y" bugs.

Q: What is the operational cost difference between running self-hosted Airflow and a managed offering like MWAA or Composer?
A: Self-hosted Airflow means you own the scheduler, executor, database, upgrades, scaling, and patching, trading recurring engineering toil for full control and no per-environment vendor markup. Managed offerings remove most of that toil for a recurring fee and a fixed upgrade cadence, at the cost of some configuration flexibility and cloud lock-in. For a small team the removed 2am-pager toil often justifies the managed premium.

Q: What does "lock-in" specifically mean when evaluating ADF versus portable Airflow DAGs?
A: Lock-in is the degree to which your pipelines are bound to one vendor's proprietary surface so that leaving is costly. ADF pipelines authored in its visual/JSON model and tied to Azure-specific activities are harder to lift to another cloud than Airflow DAGs, which run on any managed-Airflow offering. The architect weighs that portability against the convenience of ADF's native Azure integration and existing agreement.

Q: A new Dagster asset with `AutomationCondition.eager()` never auto-materializes when its upstream updates. First checks?
A: First confirm the Dagster daemon is running, since automation conditions are evaluated by the daemon — without it, nothing materializes automatically. Then verify the upstream is actually emitting materialization events the condition can observe, and that the asset's dependency on that upstream is correctly declared. A defined-but-unevaluated condition (no daemon) is the most common cause.

Q: Why is "exactly once" easier to reason about with partitioned assets than with a long-running streaming job?
A: A partitioned batch asset has a discrete, named unit (the valuation-day partition) whose materialization is recorded, so "did June 12 run exactly once?" is a direct lookup. A continuous stream has no such natural per-day boundary, so exactly-once becomes a harder property of offsets and checkpoints. For daily NAV the partitioned-batch framing matches the business cadence and makes the guarantee inspectable.

Q: How does a deliberate `catchup` configuration support a clean 30-day backfill in Airflow?
A: Setting `catchup` deliberately means you control whether normal scheduling already filled those intervals, so a subsequent manual backfill over 30 days has predictable, non-overlapping scope. If `catchup` were accidentally `True` with an old start date, you could get a confusing tangle of auto and manual runs. Deciding `catchup` up front keeps "what the scheduler did automatically" and "what I backfilled deliberately" cleanly separable.

Q: After a backfill in both tools, how do you record "which partitions recomputed and how you proved it"?
A: List the recomputed partition keys (Dagster: the asset's materialized partition statuses/event log; Airflow: the cleared/rerun task-instance date range) and attach evidence per partition — changed row counts, a reconciliation total, or a checksum before and after. The proof is the per-partition delta, not merely "the backfill ran". Recording it makes the recompute auditable and turns the exercise into the kind of evidence a regulator or CAB accepts.

Q: What is the trade-off between ADF's low-code visual authoring and a code-first orchestrator like Dagster or Airflow?
A: ADF's visual/JSON authoring lowers the barrier for analysts and is fast for simple connector-driven movement, but it resists the software-engineering practices — version control diffs, unit testing, code review, modular reuse — that code-first orchestrators make natural. A code-first DAG or asset definition lives in Git, is testable, and is reviewable in a PR, which matters enormously for a regulated, auditable platform. The architect weighs ADF's accessibility against the engineering rigor that Python-defined pipelines bring to change control.

Q: Summarize, in two minutes' worth, the architect's case for choosing asset-oriented orchestration for a regulated fund platform.
A: Asset-orientation makes the data products — NAV, validated prices, reference data — the primitives, so lineage and freshness come built in and answer auditors' "what fed this and was it fresh?" directly. Asset checks bind reconciliation, ISIN completeness, and row-count gates structurally to the products, turning silent data-wrong into loud, publish-blocking failures, and declarative automation rebuilds downstream on real data readiness instead of cron guesses. The trade-off is the incumbency and hiring weight of Airflow plus the gravity of an existing ADF agreement, which a sound memo weighs explicitly before recommending — Dagster where lineage and audit cost dominate, Airflow where skills and ecosystem do.

Q: What is the default `trigger_rule` on an Airflow task, and what does it require of upstream tasks?
A: The default `trigger_rule` is `all_success`, meaning a task runs only when every directly-upstream task has succeeded; any upstream failure or skip blocks it and marks it `upstream_failed`. This is the safe default for a NAV chain because it refuses to compute downstream from a broken input. You override it deliberately — e.g. `all_done` for a cleanup task that must run regardless — but never accidentally, since a loosened rule can let a step run on incomplete data.

Q: You want a "send failure alert" task that runs only when a prior task fails. Which `trigger_rule` do you set?
A: Set `trigger_rule='one_failed'` on the alert task so it fires as soon as at least one upstream task fails, without waiting for the rest of the graph to finish. Unlike `all_done`, `one_failed` does not wait for every upstream to complete, so the page goes out promptly. Pair it with `all_success` on the normal downstream path so the happy path and the alarm path are cleanly separated.

Q: What does `trigger_rule='all_done'` give a final "cleanup" or "publish status" task, and what is the gotcha?
A: `all_done` runs the task once every upstream is in any terminal state — success, failed, or skipped — so a teardown or status-writer always executes. The gotcha is that `all_done` does not check whether upstreams succeeded, so a cleanup task that also publishes results could publish on top of a failed run; use `none_failed_min_one_success` instead when at least one real success must have occurred. Reserve `all_done` for genuinely side-effect-free finalization like releasing a lock.

Q: What is `none_failed_min_one_success` for, and where does it shine after a branch?
A: `none_failed_min_one_success` runs a task when no upstream failed and at least one succeeded (skips are tolerated), which is exactly the join condition after a `BranchPythonOperator` where some branches are skipped by design. With the default `all_success` the join would itself be skipped because skipped upstreams are not successes; `none_failed_min_one_success` lets the merge proceed. It is the canonical "rejoin after a branch" trigger rule.

Q: What is a deferrable operator in Airflow, and how does it improve on `reschedule`-mode sensors?
A: A deferrable operator suspends itself by calling `self.defer(...)`, handing an async trigger to the triggerer process and freeing its worker slot entirely while it waits. It improves on `reschedule` because the wait runs as an async coroutine — thousands can co-exist in one triggerer — and it can resume on criteria beyond fixed time intervals, not just on a clock poke. For long file-arrival waits this is the most slot-efficient option of the three.

Q: What Airflow component must be running for deferrable operators to work, and what breaks if it is absent?
A: The `triggerer` process must be running; it hosts the async event loop that executes the deferred triggers and signals when each is ready to resume. If no triggerer is deployed, any task that calls `self.defer` will defer and then never wake, hanging indefinitely. The fix is to ensure `airflow triggerer` is part of the deployment alongside scheduler and workers.

Q: How does `self.defer(trigger, method_name, ...)` resume execution once the trigger fires?
A: When the trigger condition is met, the triggerer re-submits the task into the executor queue and execution resumes at the operator method named by `method_name`, receiving any event payload the trigger emitted. This split-then-resume design is why a deferrable operator must be written with a clean handoff point rather than a single blocking loop. The `timeout` argument bounds how long the deferral may wait before failing.

Q: What does the `timeout` parameter on a sensor do, and why set it even in `reschedule` mode?
A: `timeout` is the maximum total wall-clock time a sensor will wait before failing the task rather than continuing to poll forever; the default is very large (a week). You set it even in `reschedule` mode so a never-arriving price file becomes a clean, alertable failure instead of a silently stuck task that no one notices. Without a `timeout`, an upstream that simply never lands produces no page — the worst outcome for a NAV deadline.

Q: What is the difference between `poke_interval` and `timeout` on a sensor?
A: `poke_interval` is how often the sensor re-checks its condition (e.g. every 60 seconds), while `timeout` is the total time it may keep checking before giving up and failing. They are independent: a 30-second `poke_interval` with a 4-hour `timeout` checks 480 times before failing. A common mistake is setting a long `poke_interval` with a short `timeout`, so the sensor fails after only one or two checks.

Q: What does `max_active_runs` control on a DAG, and when is `max_active_runs=1` important?
A: `max_active_runs` caps how many DAG runs for that DAG can be in flight simultaneously; setting `max_active_runs=1` serializes runs so run N+1 never starts before run N finishes. It matters when consecutive runs touch shared mutable state — e.g. a NAV table where an overlapping backfill and scheduled run could race on the same partition. For order-sensitive daily pipelines, serializing runs prevents out-of-order partition writes.

Q: What does `depends_on_past=True` do to a task, and what is its risk during a backfill?
A: `depends_on_past=True` makes a task instance wait for the same task in the previous logical interval to have succeeded before it runs, enforcing strict chronological ordering. The risk is that during a backfill or after one failed historical run, every subsequent date stalls because the chain is blocked on the earliest failure. Use it only when a day genuinely cannot be computed without the prior day's success, and be ready to clear the blocking instance.

Q: How do `max_active_tasks` (DAG concurrency) and pool slots together bound parallelism?
A: `max_active_tasks` limits concurrently running task instances within a single DAG, while a pool caps concurrency across all DAGs that share that pool's slots — both must allow a task before it runs. You use a pool to protect a fragile downstream (e.g. a database accepting only N concurrent connections) regardless of how many DAGs hit it. A task can be ready yet queued because either the DAG concurrency or the pool is full.

Q: An Airflow task is stuck in the `queued` state and never starts. What are the first things to check?
A: Check whether the executor has free capacity (workers up, Celery queue not backlogged, or Kubernetes able to schedule pods), then whether a pool or `max_active_tasks`/`max_active_runs` limit is throttling it. Queued means the scheduler wants to run it but the executor has not picked it up, so the bottleneck is capacity or a concurrency cap, not dependencies. Also confirm the worker is listening on the same queue the task was routed to.

Q: A task shows `up_for_retry` then `up_for_reschedule` — what is the difference?
A: `up_for_retry` means the task failed and is waiting out its `retry_delay` before another attempt, whereas `up_for_reschedule` is the normal state of a `reschedule`-mode sensor that released its slot and will re-poke at the next interval. The first signals a fault being retried; the second is healthy waiting, not an error. Confusing a perpetually `up_for_reschedule` sensor for a failure leads to chasing a non-problem.

Q: What does an Airflow task in the `upstream_failed` state tell you, and how is it different from `failed`?
A: `upstream_failed` means the task never ran because a task it depends on failed (under the default `all_success` rule), so the failure is inherited, not original. `failed` means this task itself executed and errored. When triaging, you trace `upstream_failed` tasks back to the first genuinely `failed` task — the root cause — rather than investigating each skipped downstream.

Q: How would you fan out one Airflow task per fund or per ISIN at runtime when the list is dynamic?
A: Use dynamic task mapping with `.expand()`, passing a list resolved at runtime (e.g. from an upstream task's XCom of fund codes) so Airflow generates one mapped task instance per element. This replaces hand-writing N tasks and adapts when the fund list changes between runs. Keep the mapped list to identifiers, not bulky data, so you do not blow up XCom with the expansion input.

Q: What is the danger of dynamic task mapping over an unbounded list, and how do you contain it?
A: If the expanded list can grow without bound — say one mapped task per row — you can spawn thousands of task instances, overwhelming the scheduler and UI much like over-partitioning does. Contain it by capping the fan-out (chunking the list, or `max_active_tis_per_dag` on the mapped task) and by mapping over coarse units like funds, not individual rows. The grain of the map should match a sensible unit of retry and observability.

Q: Why prefer the TaskFlow `@task` decorator over classic `PythonOperator` with manual XCom?
A: The TaskFlow API lets you write plain Python functions decorated with `@task`, and return values are passed as XComs automatically with dependencies inferred from the function calls, removing boilerplate `xcom_push`/`xcom_pull` and explicit `set_upstream`. The code reads like ordinary Python, which is easier to unit-test and review — directly serving the "keep DAG files thin and testable" principle. It still pushes XComs, so the small-payload rule for XCom continues to apply.

Q: How do you keep heavy library imports out of the parse path while still using them in a task?
A: Move the import inside the task's callable (or under the `if __name__` guard / a function) so it executes only at task runtime, not on every scheduler parse cycle. Top-level `import pandas` runs each parse loop and inflates parse time across all DAGs in the file; a deferred import runs once per task execution. This is the concrete fix for the "heavy imports cause scheduler lag" symptom.

Q: What does the `airflow dags test` command do, and why is it useful for local development?
A: `airflow dags test <dag_id> <logical_date>` runs a full DAG for one logical date directly in the foreground without scheduling it, executing tasks in order so you can debug end-to-end locally. Unlike triggering through the scheduler, it does not require the run to be queued and surfaces errors inline. It is the fast inner-loop check that a DAG actually works for a given date before you let the scheduler near it.

Q: What does `airflow tasks test <dag_id> <task_id> <date>` let you do that running the whole DAG does not?
A: It executes a single task instance for a given logical date in isolation, ignoring dependencies and not recording the run in the metadata DB, so you can iterate on one operator without running its upstreams. This is ideal for debugging a single transform's templating or connection without re-running the expensive earlier steps. Because it skips dependency checks, it will run even if upstreams would normally block it.

Q: What is the difference between clearing a task instance in the UI and running `airflow dags backfill`?
A: Clearing resets selected existing task instances to `None` so the scheduler re-runs exactly those, operating on runs that already exist; `airflow dags backfill` creates and executes runs for a date range, materializing intervals that may not have run yet. Clearing is surgical re-execution of known instances; backfill is range creation plus execution. Both still depend on idempotent writes to be safe.

Q: How does the Airflow `Variable` mechanism differ from XCom, and what is the parse-time trap?
A: A `Variable` is global key-value configuration stored in the metadata DB and read with `Variable.get`, whereas XCom passes per-run values between tasks. The trap is calling `Variable.get` at the top level of a DAG file, which hits the DB on every parse loop; read variables inside tasks or via Jinja templates (`{{ var.value.x }}`) so the lookup happens at run time. Top-level `Variable.get` is a classic cause of scheduler-parse load.

Q: Where do Airflow Connections live, and why not hardcode credentials in the DAG?
A: Connections are stored in the metadata DB (or a secrets backend) and referenced by `conn_id`, keeping host/credentials out of code and swappable per environment. Hardcoding credentials in the DAG file puts secrets in Git, breaks dev/prod separation, and forces a code change to rotate a password. Operators take a `conn_id` and resolve the actual connection at run time, which also keeps secrets out of the parsed module scope.

Q: What is the practical effect of setting `schedule=None` on an Airflow DAG?
A: `schedule=None` makes the DAG trigger-only — it never runs on a timer and only executes when started manually or via the API/another DAG. This suits one-off or externally-driven pipelines, such as a restatement DAG you run on demand for a specific valuation date. It is distinct from `schedule='@daily'`, which creates timed runs the scheduler manages.

Q: How does `TriggerDagRunOperator` differ from asset/dataset-based scheduling for chaining DAGs?
A: `TriggerDagRunOperator` makes an upstream DAG imperatively kick off a named downstream DAG, an explicit point-to-point link, whereas asset-based scheduling declares the downstream DAG's dependency on data so it triggers whenever any producer updates that asset. The trigger operator hardcodes the relationship in the producer; the asset model centralizes the dependency on the consumer and supports many producers. Asset-based scheduling scales and drifts less, matching the declarative principle.

Q: In Dagster, what severity does `AssetCheckResult` default to, and how do you make a check non-blocking?
A: `AssetCheckResult` defaults to `severity=AssetCheckSeverity.ERROR`; to make a failing check advisory rather than gating you set `severity=AssetCheckSeverity.WARN`. Severity can be set at runtime on the result, not only on the definition, so the same check can escalate based on how bad the value is. A `WARN` result records the concern in the UI without halting downstream automation.

Q: What does the `blocking=True` parameter on a Dagster asset check actually do?
A: With `blocking=True`, if the check returns a failed result at `ERROR` severity, Dagster prevents downstream assets in the same run from executing, structurally gating bad data. Crucially, gating applies only to an emitted failed result — if the check emits no result at all, downstream proceeds and a warning is logged. So a blocking reconciliation check must actually run and return `passed=False` to stop publication, not merely be defined.

Q: How do you attach evidence (like the computed row count) to a Dagster asset check result?
A: Pass a `metadata` dict to `AssetCheckResult`, e.g. `AssetCheckResult(passed=num_rows > 0, metadata={'num_rows': num_rows})`, and the values render alongside the check in the UI and event log. This turns a pass/fail into an auditable record showing the exact number that was tested. For a NAV reconciliation you would attach the control total and the computed total so the discrepancy is visible without re-running.

Q: By default, how does a daily-partitioned Dagster asset depend on a daily-partitioned upstream, partition-wise?
A: By default each downstream partition depends on all upstream partitions whose time window intersects its own, so a daily-to-daily dependency maps partition `2026-06-12` to upstream `2026-06-12`. When grains differ — daily depending on hourly — the daily partition depends on all 24 of that day's hourly partitions. This automatic time-window matching is what makes per-day backfills line up across the asset graph.

Q: How would you express "today's NAV partition needs yesterday's partition" of an upstream in Dagster?
A: Use a `TimeWindowPartitionMapping` on the dependency with `start_offset=-1, end_offset=0`, so the downstream partition for a date depends on the upstream window from the prior day through today. This models metrics that need the previous valuation day, such as a daily return. Without the mapping, the default same-window rule would only wire today-to-today and miss the prior-day input.

Q: What does `allow_nonexistent_upstream_partitions` control in a Dagster partition mapping, and its default?
A: It controls whether mapping to an upstream partition that does not exist (falls outside the upstream's defined window) raises an error; it defaults to `False`, so a missing mapped partition errors. Setting it `True` is appropriate at the edges of a series — e.g. the first day has no prior-day partition — so the run does not fail on a legitimately absent boundary. Leave it `False` in the steady state so genuine gaps stay loud.

Q: In Dagster, what is the difference between a freshness check failing and an asset being marked "stale"?
A: A freshness check fails when an asset has not been materialized within its allowed lag of wall-clock time — a staleness-of-update signal. "Stale" in the UI is a derived signal that the asset's upstream data or code version changed since its last materialization, so its current value may not reflect its inputs. One is about clock-time lateness; the other is about graph/version drift, and an asset can be one without the other.

Q: How do you get a Dagster asset to recompute automatically when only its transformation code changes?
A: Set a `code_version` on the asset and use an `AutomationCondition` such as `AutomationCondition.code_version_changed()` (or compose it with others) so a version bump marks dependents stale and triggers rematerialization. This ties a logic fix to automatic recomputation of what depends on it, instead of relying on someone to remember to backfill. The same `code_version` drives the "stale" indicator in the UI.

Q: What is a Dagster sensor (as opposed to a schedule), and give a fund-data use for one?
A: A sensor is a function the daemon evaluates on an interval that decides whether to launch runs based on external state — a file appearing, a row in a control table, a queue message — returning `RunRequest`s when it should fire. Unlike a schedule, it reacts to data/events rather than the clock. A fund use is a sensor watching the SFTP drop for the day's price file and materializing the prices asset the moment it lands.

Q: How does a Dagster sensor avoid re-firing on the same event each tick?
A: A sensor uses a `cursor` — a small string it persists between evaluations — to record the last position it processed (e.g. the latest file timestamp or message offset), so each tick only acts on what is new. Without managing the cursor, the sensor would re-launch runs for already-processed inputs every interval. Correct cursor handling is what makes an event sensor idempotent at the trigger level.

Q: What is an `@multi_asset` in Dagster and when would you use one for fund data?
A: An `@multi_asset` is a single computation that materializes several assets at once, declared together because they are produced by the same step. You use it when one query or API call naturally yields multiple outputs — e.g. one ingestion of a fund file that produces both the holdings asset and the share-class reference asset. It avoids artificially splitting an indivisible computation while still exposing each output as a distinct, individually-trackable asset.

Q: What is an asset's IO manager responsible for, and why does it matter for the "don't pass big data through XCom" lesson?
A: An IO manager handles persisting an asset's output and loading it as an input for downstream assets — writing to and reading from object storage, a warehouse table, etc. It matters because Dagster moves the actual data through the IO manager (durable storage) and passes only references in its event metadata, which is the structural version of the Airflow rule "store the data, pass a reference". You configure the IO manager once rather than hand-rolling persistence in every asset.

Q: What does the Dagster `@asset(deps=[...])` form let you do that returning-and-loading does not?
A: `deps=[...]` declares a dependency on an upstream asset for scheduling and lineage without loading that asset's data into the function, useful when the upstream is consumed via SQL inside a warehouse rather than passed in memory. The asset graph and partition mapping still apply, but no IO-manager load happens. This keeps lineage correct for warehouse-resident transforms where Dagster never needs to materialize the bytes into Python.

Q: How does Prefect's execution model differ from Airflow's scheduler-parses-DAGs model?
A: Prefect turns ordinary Python functions into flows and tasks via decorators and builds the graph dynamically at runtime as the code executes, rather than a scheduler statically parsing DAG files ahead of time. Its common deployment is hybrid: a hosted control plane orchestrates while the actual work runs on your own infrastructure via workers/agents. This dynamic, code-first model is its main pitch against Airflow's static-DAG boilerplate.

Q: What does Prefect's hybrid model mean for where your data and credentials live?
A: In the hybrid model the control plane (e.g. Prefect Cloud) stores orchestration metadata and schedules, but flow runs execute on workers inside your own environment, so the data and warehouse credentials never have to leave your infrastructure. This is attractive in a regulated fund shop where data residency and secret control matter. You still evaluate what metadata the control plane sees, but the heavy/sensitive payloads stay local.

Q: Where does Mage's authoring model sit relative to Airflow and Dagster for a small team?
A: Mage offers a notebook-like, block-based authoring surface where each block is an interactively built-and-previewed step with scheduling built in, aimed at teams wanting a lighter, more visual loop than writing Airflow DAGs or Dagster assets. It trades some of the rigorous software-engineering ergonomics (deep testing, composable automation) for speed of assembly. For this syllabus it is awareness-level — know the block-based pitch and that it competes with Prefect in the lightweight space.

Q: In Azure Data Factory, what is an Integration Runtime and why does it affect cost and design?
A: An Integration Runtime (IR) is the compute infrastructure ADF uses to run activities and move data — Azure-hosted, self-hosted (for on-prem/private connectivity), or SSIS IR — and ADF billing is tied to activity executions and IR usage/time. It affects design because reaching private data sources requires a self-hosted IR, and IR hours plus per-activity charges drive cost. An architect comparing ADF to FOSS Airflow must price the IR model, not just the headline "it's included".

Q: How does ADF's pipeline/activity model map onto an Airflow DAG/task mental model?
A: An ADF pipeline corresponds roughly to a DAG and ADF activities correspond to tasks, with control-flow activities (ForEach, If, Until) mirroring Airflow's branching and looping. The big differences are authoring (ADF's visual/JSON vs Airflow's Python) and that ADF lacks first-class asset lineage and idempotency conventions you would enforce yourself. The mapping lets an architect translate a design across, but the translation loses the code-review, unit-test, and asset-lineage strengths.

Q: Why does ADF's JSON/visual authoring weaken change control compared with code-first orchestrators in a regulated shop?
A: ADF pipelines are defined as generated JSON edited through a visual designer, so meaningful PR diffs, modular reuse, and unit testing are awkward, whereas a Python DAG or asset definition is reviewed in a pull request and tested in CI like any code. In a regulated, auditable platform the inability to cleanly diff and review a change is a real control gap. This is the engineering-rigor side of the ADF trade-off, weighed against its low barrier for analysts.

Q: For "periodic refresh of one materialized view", why is `pg_cron` preferable to standing up Airflow?
A: `pg_cron` schedules the `REFRESH MATERIALIZED VIEW` inside the same Postgres with zero extra infrastructure, no scheduler/executor/metadata-DB to run, and no network hop — the whole need is one timed SQL statement in one database. Bringing in Airflow for that adds operational surface and a 2am pager for no dependency-graph benefit. The decision rule is: single database, single SQL, no inter-step dependencies → `pg_cron`.

Q: A `pg_cron` job silently stopped running its nightly SQL. Where do you look first?
A: Check the `cron.job_run_details` table (and the Postgres logs), which records each scheduled job's start, status, and any error message, so you can see whether the job fired and failed or never fired at all. Also confirm the extension is still loaded and the `cron.database_name`/job is pointed at the intended database. Because `pg_cron` has no DAG-level observability, that run-details table is essentially your only built-in audit trail — another reason it does not scale to real pipelines.

Q: With the EU AI Act in force, why would an orchestrator's lineage and freshness records matter if a fund uses an AI model for a data step?
A: The EU AI Act imposes data-governance, record-keeping, and traceability obligations on higher-risk AI uses, and an asset-oriented orchestrator already records which inputs (and which versions) produced a model-fed output and whether each was fresh, supplying exactly that traceability evidence. Being able to show the provenance and freshness of the data behind an AI-assisted classification turns a compliance ask into a query against the asset graph. Unmonitored cron scripts feeding a model give you none of that record.

Q: How does orchestration discipline support a T+1 settlement timeline for fund data, and where is the pressure point?
A: Under a T+1 cycle the time between trade date and settlement is one business day, so dependent reference, price, and instruction data must be produced and validated within a compressed window, leaving little slack for late or rerun-heavy pipelines. The pressure point is freshness and ordering: an orchestrator's freshness checks and dependency gating surface a late upstream immediately and let an idempotent backfill recompute the affected partition fast, rather than discovering lateness when settlement fails. The shrinking window makes "ran on time, in order, exactly once" an operational requirement, not a nicety.

Q: What is an EET, and why might an orchestrated pipeline gate its production on upstream asset checks?
A: An EET (European ESG Template) is the standardized file conveying ESG/sustainability data for funds to distributors, alongside the EMT and EPT in the same template family. An orchestrated pipeline gates EET production on asset checks for completeness and validity (every required share class present, no malformed identifiers) so a partial or malformed ESG feed never reaches distributors. Binding those checks to the EET-producing asset turns "the ESG data looked fine" into a structural, publish-blocking gate per cycle.
