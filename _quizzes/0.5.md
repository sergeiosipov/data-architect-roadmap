# Phase 0 · 0.5 Docker & containers from zero — 100 self-test questions

Q: In one sentence, what is a container?
A: A container is an isolated process running on the host kernel with its own filesystem, network, and process namespace, packaged from an image so it runs the same way anywhere Docker runs.

Q: What is a Docker image?
A: An image is a read-only, layered template (a filesystem snapshot plus metadata like the default command) from which containers are created; it is the build artifact, not the running thing.

Q: What is the core difference between an image and a container?
A: An image is a static, read-only template stored on disk; a container is a running (or stopped) instance of that image with a thin writable layer on top.

Q: Why can you run many containers from a single image at the same time?
A: Each container gets its own writable layer and isolated namespaces while sharing the image's read-only layers, so they are independent processes that do not collide.

Q: What command lists running containers, and what flag also shows stopped ones?
A: `docker ps` lists running containers; `docker ps -a` lists all containers including stopped/exited ones.

Q: What does an image tag like `python:3.12-slim` mean?
A: `python` is the repository name and `3.12-slim` is the tag identifying a specific variant/version; if you omit the tag Docker defaults to `latest`.

Q: Why is relying on the `latest` tag risky in a fund-data pipeline?
A: `latest` is a moving pointer, so a rebuild can silently pull a different version and break reproducibility; pin an explicit tag or digest so a NAV-calculation image is deterministic.

Q: What is an image digest and why prefer it for reproducibility?
A: A digest is a content-addressable SHA256 hash (e.g. `image@sha256:...`) that always points to the exact same bytes, so pinning by digest guarantees the identical image regardless of tag movement.

Q: What does `docker run` do at a high level?
A: It creates a new container from an image and starts it, optionally pulling the image first if it is not present locally.

Q: What are image layers?
A: Layers are stacked, read-only filesystem diffs; each Dockerfile instruction that changes the filesystem typically creates one layer, and the final image is the union of all layers.

Q: What is the build cache?
A: During `docker build`, Docker reuses the result of a previous identical instruction (same instruction plus same inputs) instead of re-executing it, which makes rebuilds fast.

Q: When does a layer's build cache get invalidated?
A: When the instruction text changes or its inputs change (e.g. a `COPY`ed file's contents differ); once a layer is invalidated, every layer after it is rebuilt too.

Q: Why should you `COPY` your dependency manifest and install deps before copying the rest of the source?
A: So that editing application code does not invalidate the dependency-install layer; the expensive install stays cached and only the cheap code-copy layer rebuilds.

Q: In a Python project, what is the cache-friendly order for installing requirements?
A: `COPY requirements.txt .` then `RUN pip install -r requirements.txt`, and only afterwards `COPY . .`, so source edits do not bust the install cache.

Q: What does the port flag `-p 8080:80` mean?
A: It publishes container port `80` to host port `8080`; the format is `-p HOST:CONTAINER`, so traffic to `localhost:8080` reaches the service on `80` inside the container.

Q: If you run `-p 5432:5432` for Postgres but get "port is already allocated", what is the first thing to check?
A: Whether something else (often a locally installed Postgres or another container) already binds host port `5432`; either stop it or map a different host port like `-p 5433:5432`.

Q: How do you load many environment variables from a file at run time?
A: With `--env-file path/to/.env`, which reads `KEY=VALUE` lines and injects them into the container's environment.

Q: Why pass secrets like a transfer-agency DB password via env vars or secrets rather than baking them into the image?
A: Anything baked into an image is stored in a layer and visible via `docker history`, so credentials in the image leak to anyone who can pull it; runtime env/secrets keep them out of the artifact.

Q: What is a bind mount?
A: A bind mount maps a specific host directory or file into a container path, so the container sees and edits the live host files directly.

Q: What is a named volume?
A: A named volume is Docker-managed storage with a name (e.g. `pgdata`) that persists independently of any container and lives under Docker's storage area, not a host path you pick.

Q: What is the key practical difference between a bind mount and a named volume?
A: A bind mount points at an exact host path you control (great for live source code), while a named volume is managed by Docker and portable (great for persistent data like a database).

Q: Which mount type would you use to persist a Postgres data directory, and why?
A: A named volume (e.g. `-v pgdata:/var/lib/postgresql/data`) because it is Docker-managed, survives container removal, and avoids host-path permission and portability issues.

Q: Which mount type would you use to live-edit a NAV-loading script during development?
A: A bind mount of your source directory (e.g. `-v ${PWD}:/app`) so edits on the host are immediately visible inside the container without rebuilding.

Q: On Windows PowerShell, how do you bind-mount the current directory into `/app`?
A: Use `-v ${PWD}:/app` in PowerShell (or `-v "$(pwd):/app"` in bash); `${PWD}` expands to the current path.

Q: What happens to data written inside a container's writable layer (not a volume) when the container is removed?
A: It is destroyed with the container; only data in a volume or bind mount survives container removal.

Q: What is a Dockerfile?
A: A Dockerfile is a text file of instructions that `docker build` executes top-to-bottom to assemble an image layer by layer.

Q: What does the `FROM` instruction do?
A: It sets the base image the build starts from (e.g. `FROM python:3.12-slim`) and must usually be the first instruction.

Q: Why might you choose a `-slim` or `alpine` base image?
A: To get a smaller image with fewer packages, reducing size, attack surface, and pull time; `alpine` is smallest but uses musl libc which can cause compatibility issues for some Python wheels.

Q: What is the build context?
A: The build context is the set of files sent to the Docker daemon for the build, normally the directory passed to `docker build` (e.g. `.`); `COPY`/`ADD` can only use files inside it.

Q: What is a `.dockerignore` file and why does it matter?
A: It lists patterns to exclude from the build context (e.g. `.git`, `__pycache__`, `data/`), which shrinks the context, speeds builds, and avoids copying secrets or huge data files.

Q: What is the difference between `RUN` and `CMD`?
A: `RUN` executes at build time to construct the image; `CMD` specifies the default command that runs when a container starts from the image.

Q: What does the `CMD` instruction do, and how do you override it at run time?
A: `CMD` sets the default command for containers started from the image; you override it by appending a command after the image name, e.g. `docker run myimage python other.py`.

Q: What is the difference between `CMD` and `ENTRYPOINT`?
A: `ENTRYPOINT` defines the fixed executable that always runs; `CMD` provides default arguments to it; with both set, `docker run` arguments replace `CMD` but keep `ENTRYPOINT`.

Q: What does the `ENV` instruction do?
A: It sets environment variables baked into the image, available at build time for later instructions and at run time inside containers, unless overridden by `-e`.

Q: What is the difference between `ENV` in a Dockerfile and `-e` at run time?
A: `ENV` sets a default value persisted in the image; `-e` (or `--env-file`) overrides or supplements it for a specific container run without changing the image.

Q: What is the difference between exec form and shell form for `CMD`/`RUN`?
A: Exec form is a JSON array like `["python", "app.py"]` and runs the binary directly (no shell); shell form like `python app.py` runs via `/bin/sh -c`, enabling shell features but adding a shell process and affecting signal handling.

Q: Why is exec-form `CMD` preferred for the main process?
A: Because the process runs as PID 1 directly and receives signals like `SIGTERM`, allowing graceful shutdown, whereas shell form wraps it in `sh -c` which can swallow signals.

Q: How do you build an image and tag it?
A: `docker build -t myimage:1.0 .` builds from the Dockerfile in the current directory (the context `.`) and tags the result `myimage:1.0`.

Q: What is a multi-stage build and why use it?
A: It uses multiple `FROM` stages so you can build/compile in a heavy stage and `COPY --from=builder` only the artifacts into a slim final image, drastically reducing final image size.

Q: Why combine related commands in a single `RUN` with `&&`?
A: Each `RUN` creates a layer, so combining them (and cleaning caches in the same `RUN`) reduces layer count and image size, since deleting files in a later layer does not shrink earlier ones.

Q: What is Docker Compose?
A: Docker Compose is a tool that defines and runs multi-container applications declaratively from a `compose.yaml` file, so one command can start a whole stack.

Q: What command starts a Compose stack, and how do you run it in the background?
A: `docker compose up` starts it in the foreground; add `-d` (`docker compose up -d`) to run detached.

Q: What does `docker compose down` do, and how do you also delete volumes?
A: It stops and removes the project's containers and networks; add `-v` (`docker compose down -v`) to also delete named volumes declared in the file.

Q: How do containers in the same Compose project reach each other?
A: Compose puts them on a shared default network and they resolve each other by service name as a hostname (e.g. the app connects to `db:5432`).

Q: In Compose, what hostname would an app use to reach a service named `postgres` on port `5432`?
A: `postgres:5432` — the service name resolves via Docker's internal DNS on the project network, with no host port mapping required.

Q: What does `depends_on` do in Compose?
A: It controls startup order so a service starts after the ones it lists, but by default it only waits for the container to start, not for the app inside to be ready.

Q: Why is plain `depends_on` insufficient to guarantee a database is ready?
A: Because it only waits until the dependency's container has started, not until the database accepts connections; you need a `healthcheck` plus `depends_on: condition: service_healthy` for true readiness.

Q: How do you make `depends_on` wait until a dependency is actually healthy?
A: Use the long form `depends_on: { db: { condition: service_healthy } }` combined with a `healthcheck` on the `db` service.

Q: What is a healthcheck in Compose/Docker?
A: A periodic command run inside the container to decide if the service is healthy; its exit code (0 = healthy) sets the container's health status shown in `docker ps`.

Q: What does a Postgres healthcheck typically use?
A: A command like `pg_isready -U postgres` with `interval`, `timeout`, `retries`, and often `start_period`, so the container is marked healthy only once Postgres accepts connections.

Q: What do `interval`, `timeout`, `retries`, and `start_period` mean in a healthcheck?
A: `interval` is time between checks, `timeout` is how long each check may run, `retries` is consecutive failures before marking unhealthy, and `start_period` is an initial grace window where failures do not count.

Q: How do you define a named volume in a Compose file?
A: List it under a top-level `volumes:` key (e.g. `volumes: { pgdata: {} }`) and reference it in a service with `volumes: ["pgdata:/var/lib/postgresql/data"]`.

Q: What is the difference between `ports:` and `expose:` in Compose?
A: `ports:` publishes to the host (`HOST:CONTAINER`), while `expose:` only makes the port reachable to other services on the network without binding it on the host.

Q: How do you tell a Compose service to build from a Dockerfile instead of pulling an image?
A: Use `build:` (e.g. `build: .` or `build: { context: ., dockerfile: Dockerfile.dev }`) instead of, or alongside, an `image:` name.

Q: After changing the Dockerfile, why might `docker compose up` not reflect changes?
A: Because Compose reuses the existing image unless you rebuild; run `docker compose up --build` (or `docker compose build`) to force a rebuild.

Q: What does `docker logs <container>` show?
A: It shows whatever the container's main process wrote to STDOUT and STDERR, the primary way to see application output and errors.

Q: Why should a containerized app log to STDOUT/STDERR instead of a file?
A: Because Docker captures STDOUT/STDERR for `docker logs` and log drivers; writing to a file inside the container hides logs and can fill the writable layer.

Q: Why might `docker exec -it <container> bash` fail with "executable file not found"?
A: Because minimal images (like `slim`/`alpine`) may not include `bash`; try `sh` instead: `docker exec -it <container> sh`.

Q: Your container keeps showing status `Exited (0)` immediately — what does that mean?
A: The main process finished successfully and nothing kept PID 1 alive; a container only runs as long as its foreground process does, so a script that completes will exit cleanly.

Q: A container shows `Exited (1)` — how do you investigate?
A: Run `docker logs <container>` to read the error output; exit code 1 indicates the main process errored, and the logs usually show the traceback or message.

Q: A container shows `Exited (137)` — what does that usually indicate?
A: Exit code 137 means the process received `SIGKILL` (128 + 9), commonly the OOM killer terminating it for exceeding its memory limit, or a forced `docker kill`.

Q: Why can't you `docker exec` into an exited container?
A: `docker exec` needs a running process to attach to; an exited container has no running PID 1, so inspect it with `docker logs`, `docker inspect`, or restart/recreate it.

Q: What is the difference between `docker stop` and `docker kill`?
A: `docker stop` sends `SIGTERM` then `SIGKILL` after a grace period (default 10s) for a graceful shutdown; `docker kill` sends `SIGKILL` immediately.

Q: What does `docker system df` show?
A: It reports disk usage broken down by images, containers, local volumes, and build cache, including how much is reclaimable.

Q: What does `docker system prune` remove by default?
A: It removes stopped containers, unused networks, dangling images, and (in recent versions) build cache, but not named volumes unless you add `--volumes`.

Q: What is the danger of `docker system prune -a --volumes`?
A: `-a` also deletes all images not used by a running container, and `--volumes` deletes unused named volumes — including persistent data like a Postgres volume — so it can wipe important data.

Q: After many rebuilds your disk is full of `<none>` images and old cache — what cleanup sequence is reasonable?
A: Run `docker system df` to see usage, then `docker image prune`, `docker builder prune`, and `docker container prune`, reserving `-a`/`--volumes` for when you are sure nothing needed will be lost.

Q: Your bind-mounted code changes are not appearing in the container — first thing to check?
A: Verify the mount path is correct and actually applied (`docker inspect` mounts), that you mounted over the right container directory, and on Windows/WSL that file sharing for that drive is enabled.

Q: A Python app in a container can't find a package you installed — common cause?
A: The image was not rebuilt after editing `requirements.txt`/`Dockerfile`, so the old layer without the package is still in use; rebuild with `docker compose up --build` or `docker build`.

Q: Your app can't connect to the database using `localhost` inside the container — why?
A: Inside a container `localhost` refers to that container itself, not the host or another container; use the database service/container name (e.g. `db`) on the Docker network instead.

Q: From inside a container, how do you reach a service running on the host machine?
A: Use the special DNS name `host.docker.internal` (supported on Docker Desktop for Windows/Mac and configurable on Linux) to reach host-bound services.

Q: A Compose `web` service starts before `db` is ready and crashes on connect — robust fix?
A: Add a `healthcheck` to `db` and make `web` use `depends_on: { db: { condition: service_healthy } }`, and ideally have the app retry the connection on startup.

Q: You changed a `.env` value but the container still uses the old value — why?
A: Environment variables are read at container creation; you must recreate the container (`docker compose up -d` recreates, or `docker run` again), since `docker restart` alone keeps the old environment.

Q: A pulled image is `linux/amd64` but won't run well on your Apple Silicon Mac — what is happening?
A: An architecture mismatch; the image runs under emulation (slow) or fails, so prefer multi-arch images or build/pull for your platform with `--platform linux/arm64` as appropriate.

Q: How would you containerize a daily NAV-loading Python job that reads EMT files from a host folder and writes to Postgres?
A: Build an image with the script and deps, run Postgres in a service with a named volume for data, bind-mount the EMT input folder read-only into the job container, and pass DB credentials via env/secrets.

Q: For that NAV job, why mount the EMT input folder as read-only?
A: To guarantee the pipeline can read source EMT files but cannot accidentally modify or delete the authoritative inputs, protecting data integrity during the load.

Q: Why run a one-off ISIN-validation script with `docker run --rm`?
A: So the container is automatically cleaned up after the script finishes, avoiding a pile of exited containers from repeated ad-hoc runs.

Q: A transfer-agency batch container is OOM-killed (`Exited 137`) at month-end volumes — what do you adjust?
A: Increase the container/Compose memory limit (e.g. `mem_limit` or `deploy.resources`), and/or optimize the job to stream/batch records instead of loading everything into memory at once.

Q: Why keep the Postgres data on a named volume rather than the writable layer in a fund-data stack?
A: So the NAV/holdings data survives container recreation, upgrades, and `docker compose down`, since the writable layer is destroyed on `docker rm` but the named volume persists.

Q: How do you run database migrations as a separate step before the app starts in Compose?
A: Define a one-off `migrate` service (or command) that `depends_on` the healthy `db`, run it to completion (e.g. `docker compose run --rm migrate`), then start the app.

Q: What does `docker compose run` do differently from `docker compose up`?
A: `docker compose run` starts a one-off container for a single service with the command you give (good for migrations or a shell), instead of bringing up the whole defined stack.

Q: Why should you avoid running the container process as root when possible?
A: Running as root increases blast radius if the container is compromised and can create root-owned files on bind mounts; use a `USER` instruction to drop to a non-root user.

Q: What is the difference between `ARG` and `ENV`?
A: `ARG` defines a build-time variable available only during `docker build` (passed with `--build-arg`); `ENV` sets a variable persisted into the image and present at run time.

Q: Why might `docker build` send a huge context and take long even for a small image?
A: Because the build context includes everything in the directory (e.g. a multi-GB `data/` folder) unless excluded; add a `.dockerignore` to keep large/irrelevant files out of the context.

Q: What does the `HEALTHCHECK` instruction in a Dockerfile do versus a Compose healthcheck?
A: `HEALTHCHECK` bakes a default health probe into the image; a Compose `healthcheck` overrides/defines it per service, and both set the container's reported health status.

Q: What does `docker history <image>` show and why is it useful?
A: It lists the layers of an image with the instruction that created each and its size, helping you spot bloated steps and accidentally embedded secrets or data.

Q: Why do containers start almost instantly compared to virtual machines?
A: Containers share the host kernel and just start a process with isolation, whereas a VM must boot a full guest OS and kernel, which is far heavier.

Q: How do containers differ from virtual machines in terms of isolation?
A: VMs virtualize hardware and run separate kernels via a hypervisor (stronger isolation, more overhead); containers share the host kernel using namespaces and cgroups (lighter, faster, weaker isolation).

Q: On Windows, what actually runs your Linux containers under Docker Desktop?
A: A lightweight Linux VM (typically via WSL 2) hosts the Docker engine and Linux kernel, since Linux containers need a Linux kernel that Windows does not natively provide.

Q: What is the recommended way to handle secrets in Compose beyond plain env vars?
A: Use Compose `secrets:` to mount secret files into containers (e.g. at `/run/secrets/...`) instead of putting them in environment variables or the image, reducing exposure.

Q: What does the `--no-cache` flag on `docker build` do, and when would you use it?
A: It rebuilds every layer ignoring the cache, useful when a cached `RUN` (e.g. `apt-get update`) fetched stale data or you suspect a poisoned cache.

Q: What is the typical pattern to keep apt-based images small in a single `RUN`?
A: `RUN apt-get update && apt-get install -y --no-install-recommends pkg && rm -rf /var/lib/apt/lists/*` so the package index cache is removed in the same layer.

Q: Why is it dangerous to bind-mount a host directory over a container path that the image expects to be populated?
A: A bind mount shadows whatever the image put at that path with the (possibly empty) host directory, so files baked into the image at that location disappear inside the container.

Q: What happens to a named volume the first time it is mounted into a fresh Postgres container?
A: If the volume is empty, Postgres initializes a new data directory into it; if it already has data, Postgres reuses it, which is why a stale volume can keep old credentials/schema.

Q: You changed `POSTGRES_PASSWORD` but the DB still rejects the new password — likely cause?
A: The password is only applied when the data directory is first initialized; an existing named volume keeps the original password, so reset it in-db or recreate with a fresh volume.

Q: How do you completely reset a Compose Postgres to re-run its init with new settings?
A: `docker compose down -v` to remove the data volume, then `docker compose up`, which forces a fresh database initialization (destroying existing data, so back up first).

Q: What is the difference between `docker rm` and `docker rmi`?
A: `docker rm` removes containers; `docker rmi` removes images — a common beginner mix-up.

Q: Why might `docker run myimage` exit immediately with no output even though the build succeeded?
A: The image likely has no long-running foreground process or its `CMD` finished instantly; check the `CMD`/`ENTRYPOINT` and run `docker logs` or run it with a shell to investigate.

Q: What is the single most useful first command when any container "doesn't work"?
A: `docker logs <container>` (or `docker compose logs <service>`), because the application's STDOUT/STDERR usually contains the exact error explaining the failure.

Q: Why keep images small for a fund-data deployment beyond just disk savings?
A: Smaller images pull and start faster, have a smaller attack surface (fewer packages/CVEs), and are easier to audit, which matters for regulated EU fund-data environments.
