# Phase 0 · 0.10 Editor & IDE setup (VS Code) — 100 self-test questions

Q: What is a "workspace" in VS Code?
A: The project context VS Code opens — usually a single root folder (or a multi-root `.code-workspace` file) — that scopes the file explorer, search, settings, and recommended extensions to that project.

Q: What is the difference between User settings and Workspace settings in VS Code?
A: User settings apply globally to every project for your account, while Workspace settings live in `.vscode/settings.json` in the project folder and override User settings only for that project.

Q: Where are project-scoped VS Code settings stored on disk?
A: In a `.vscode/settings.json` file at the root of the workspace folder.

Q: Why commit a `.vscode/settings.json` to a team repo like `fundcli`?
A: So everyone gets the same interpreter, formatter, and lint behaviour — a NAV loader formatted and linted identically on every machine avoids "works on my editor" diffs.

Q: How do you open the integrated terminal in VS Code?
A: Use the Terminal menu, the keybinding Ctrl+backtick, or the command palette command `Terminal: Create New Terminal`.

Q: What working directory does a new integrated terminal open in by default?
A: The root folder of the current workspace, so relative paths in your commands resolve against the project you have open.

Q: Why is the integrated terminal preferable to a separate terminal window for this course?
A: It shares the workspace's folder, environment, and (over WSL) the same filesystem as the editor, so edit-run-debug stays in one window with no context switching.

Q: How do you run several shells side by side in the integrated terminal?
A: Split the terminal (the split icon or "Terminal: Split Terminal"), or open additional terminals from the dropdown — useful for running a loader in one pane and `tail`-ing its log in another.

Q: What is the VS Code command palette and how do you open it?
A: A fuzzy-searchable launcher for every command, opened with `Ctrl+Shift+P` (or `F1`); type part of a command name to find and run it without menus.

Q: How does the command palette differ from "Quick Open" (`Ctrl+P`)?
A: `Ctrl+P` jumps to files by name, while `Ctrl+Shift+P` runs commands; typing `>` in Quick Open switches it into command mode.

Q: What does "Remote - WSL" (now part of the WSL extension) let you do?
A: Run VS Code's UI on Windows while the language server, terminal, extensions, and file access all execute inside your WSL2 Linux distro.

Q: In a WSL-attached window, where does the integrated terminal's shell actually run?
A: Inside the Linux distro (e.g. Ubuntu) — it is a `bash`/`zsh` session in WSL, not Windows PowerShell or CMD.

Q: How can you tell a VS Code window is connected to WSL?
A: The green remote indicator in the bottom-left status bar shows something like `WSL: Ubuntu`.

Q: Why open your `fundcli` repo via the WSL remote rather than from the Windows path?
A: So the editor, terminal, and Python interpreter all sit on the Linux filesystem your lab uses — avoiding line-ending, permission, and `\\wsl$` performance problems.

Q: What command opens the current WSL directory in VS Code from a WSL terminal?
A: `code .` — the WSL extension wires up the `code` command inside the distro.

Q: Why is keeping project files under `~/` in WSL faster than under `/mnt/c/...`?
A: Files on the native Linux ext4 filesystem (`~`) avoid the cross-OS filesystem translation layer that makes `/mnt/c` Windows-drive access slow.

Q: When VS Code is attached to WSL, where do extensions install?
A: Extensions split into "UI" extensions that stay on the Windows side and "Workspace" extensions (like Python) that install into the WSL distro and must be enabled there per-distro.

Q: You open a folder under `/mnt/c/Users/...` in a WSL window and Python is slow to index — what is the likely cause?
A: The files live on the Windows drive accessed through `/mnt/c`, so every read crosses the WSL filesystem boundary; move the repo into the Linux home directory.

Q: What does the Python extension (Microsoft) add to VS Code?
A: IntelliSense/completions, interpreter and environment selection, the run/debug experience, test discovery, and integration with linters and formatters via Pylance.

Q: What is Pylance?
A: The Python language server that powers VS Code's completions, type-checking hints, hovers, and go-to-definition; it ships with the Python extension.

Q: How do you choose which Python interpreter VS Code uses for a project?
A: Run "Python: Select Interpreter" from the command palette and pick the environment (e.g. the project's `.venv`).

Q: Why must VS Code point at the project's `.venv` for a `uv`-managed repo?
A: So completions, debugging, and the integrated terminal use the same locked dependencies as `uv run`, instead of a system Python missing your packages.

Q: How does VS Code usually find a `uv`-created virtual environment automatically?
A: It detects the `.venv` folder in the workspace root and offers it as the recommended interpreter.

Q: What does the Ruff extension do in VS Code?
A: It runs the Ruff linter (and formatter) to surface lint diagnostics inline and can auto-fix and format Python on save.

Q: How do you make Ruff format and fix a file every time you save?
A: Set `editor.formatOnSave` true with Ruff as the default formatter, and enable `source.fixAll`/`source.organizeImports` code actions on save in settings.

Q: Why wire Ruff into the editor in Phase 0 rather than only running it in CI later?
A: So style and obvious errors are caught and fixed as you type a transfer-agency parser, not flagged hours later in a CI run.

Q: What does the Jupyter extension add to VS Code?
A: The ability to open, edit, and run `.ipynb` notebooks and interactive windows directly inside the editor, with a built-in kernel picker and variable explorer.

Q: What does the Docker extension (Container Tools) add?
A: A side panel to view images, containers, volumes, and registries, plus commands to build, run, and attach to containers and edit Dockerfiles with completions.

Q: Which four extensions does this lesson have you install for the `fundcli` setup?
A: WSL, Python, Ruff, Jupyter, and Docker (the WSL extension plus the four toolchain extensions).

Q: What is the `.vscode/extensions.json` file for?
A: It lists recommended extensions for the workspace so collaborators get prompted to install the same toolset when they open the repo.

Q: What is a breakpoint in a debugger?
A: A marker on a line that pauses program execution when reached, so you can inspect state at that exact point instead of guessing.

Q: How do you set a breakpoint in VS Code?
A: Click in the gutter to the left of a line number (a red dot appears) or press `F9` with the cursor on the line.

Q: Why is breakpoint debugging preferred over print-debugging a reconciliation script?
A: A breakpoint lets you inspect every variable live and step through logic in minutes, whereas adding/removing prints is slow, noisy, and easy to leave behind.

Q: What does the "Step Over" action do, and what is its default key?
A: It executes the current line and stops on the next line in the same function without descending into called functions; default `F10`.

Q: What does "Step Into" do, and its default key?
A: It descends into the function called on the current line so you can debug inside it; default `F11`.

Q: What does "Step Out" do, and its default key?
A: It finishes the current function and returns to the caller, stopping at the line after the call; default `Shift+F11`.

Q: What does "Continue" do, and its default key?
A: It resumes execution until the next breakpoint (or the program ends); default `F5`.

Q: How do you inspect a variable's current value while paused at a breakpoint?
A: Read it in the VARIABLES panel, hover over the identifier in the editor, or evaluate it in the DEBUG CONSOLE.

Q: What is the DEBUG CONSOLE used for during a paused session?
A: To evaluate arbitrary expressions in the current frame — e.g. type `len(funds)` or `nav.quantize('0.0001')` to test a fix before editing code.

Q: What is a "watch" expression in the debugger?
A: An expression you pin in the WATCH panel that VS Code re-evaluates and displays each time execution pauses, so you can monitor a value as you step.

Q: When debugging a currency filter that drops the wrong rows, what would you put in a watch?
A: Something like `row['currency']` or `len(filtered)` so you can see exactly which value the comparison is testing on each iteration.

Q: What is the CALL STACK panel?
A: It shows the chain of active function calls that led to the paused line, so you can click up the stack to inspect each caller's local variables.

Q: What is a conditional breakpoint and when is it useful?
A: A breakpoint that only triggers when an expression is true (e.g. `isin == 'LU0000000000'`), so you pause only on the problem fund instead of every row.

Q: How do you set a conditional breakpoint?
A: Right-click the breakpoint (or the gutter) and choose "Edit Breakpoint", then enter the condition expression.

Q: What is a logpoint?
A: A breakpoint variant that logs a message (with embedded expressions) to the debug console instead of pausing — a structured replacement for `print` that you don't have to delete.

Q: What file configures how VS Code launches the debugger?
A: `.vscode/launch.json`, which defines debug configurations (program, arguments, environment, interpreter).

Q: How can you start debugging the current Python file without writing a `launch.json`?
A: Use the Run and Debug panel's "Python File" configuration, or the "Python Debugger: Debug Python File" command.

Q: How do you pass command-line arguments to your script when debugging?
A: Add an `"args"` array to the configuration in `launch.json`, e.g. `"args": ["--currency", "EUR", "navs.csv"]`.

Q: Your breakpoint shows as a hollow grey circle and never hits — what does that mean?
A: VS Code couldn't bind it: the code path didn't run, the file isn't the one actually executing, or the debugger is attached to a different interpreter/environment.

Q: The debugger runs but uses the wrong packages — first thing to check?
A: Confirm "Python: Select Interpreter" points at the project `.venv`; a mismatched interpreter is the usual cause of import or version surprises.

Q: How do you stop a debugging session?
A: Press the stop button in the debug toolbar or `Shift+F5` ("Stop").

Q: What does "Restart" do in the debug toolbar?
A: It stops and relaunches the program with the same configuration; default `Ctrl+Shift+F5`.

Q: What is the value of "keyboard-first" habits in an editor?
A: Keeping hands on the keyboard removes the constant mouse round-trips that tax every edit; a dozen well-drilled bindings save far more time than memorising a hundred.

Q: What does "Go to Definition" do and what is its default key?
A: It jumps to where a symbol is defined; press `F12` with the cursor on the identifier (or `Ctrl+Click` it).

Q: What does "Peek Definition" do?
A: It shows the definition inline in a small embedded window without leaving your current file; default `Alt+F12`.

Q: What is "Go to References" and its key?
A: It lists every place a symbol is used across the project; default `Shift+F12`.

Q: How do you jump back to where you were after a Go to Definition jump?
A: Use "Go Back", `Alt+Left` (and `Alt+Right` to go forward again).

Q: What is multi-cursor editing?
A: Placing several cursors at once so the same edit applies in multiple locations simultaneously.

Q: How do you add a cursor on the next occurrence of the current selection?
A: `Ctrl+D` selects the next matching occurrence and adds a cursor there; repeat to grow the selection set.

Q: How do you select every occurrence of the current word at once?
A: `Ctrl+Shift+L` ("Select All Occurrences").

Q: How do you add a cursor above or below the current line by column?
A: `Ctrl+Alt+Up` / `Ctrl+Alt+Down` (Alt+Click also drops a cursor anywhere).

Q: Why is multi-cursor handy when fixing a hard-coded currency literal repeated across a parser?
A: You can select all `'USD'` occurrences with `Ctrl+Shift+L` and retype them as `'EUR'` in one keystroke, instead of editing each line.

Q: How do you rename a symbol everywhere it is referenced, safely?
A: Use "Rename Symbol" (`F2`) on the identifier — it updates all references via the language server, unlike a blind find-and-replace.

Q: What does `Ctrl+Shift+O` do?
A: "Go to Symbol in File" — a quick fuzzy jump to any function, class, or variable defined in the current file.

Q: What does `Ctrl+T` do?
A: "Go to Symbol in Workspace" — search and jump to symbols across the whole project, e.g. find `parse_emt` no matter which module it lives in.

Q: How do you comment or uncomment the selected lines quickly?
A: `Ctrl+/` toggles line comments on the selection.

Q: How do you open the project-wide search panel?
A: `Ctrl+Shift+F` opens Search, letting you grep across all files with regex, includes, and excludes.

Q: What is a `.ipynb` file?
A: A Jupyter notebook — a JSON document of ordered cells (code, Markdown, and outputs) that a kernel executes interactively.

Q: What is a kernel in the notebook context?
A: The language process (e.g. an IPython process bound to your `.venv`) that runs notebook code cells and holds the in-memory state between them.

Q: How do you run the current cell and move to the next in a VS Code notebook?
A: `Shift+Enter`; `Ctrl+Enter` runs the cell and keeps focus on it.

Q: How do VS Code notebooks and JupyterLab relate to each other?
A: They are two front-ends over the same Jupyter kernels and the same `.ipynb` format — the difference is ergonomics, not the underlying engine.

Q: Name one ergonomic advantage of VS Code notebooks over JupyterLab.
A: You get the full editor's IntelliSense, Git integration, debugging, and your extensions in the same window as the notebook.

Q: Name one thing JupyterLab does that VS Code notebooks are weaker at.
A: JupyterLab's browser UI excels at a dashboard-style multi-panel layout, rich interactive widgets, and a long-running server you can reconnect to remotely.

Q: When would you reach for a notebook over a `.py` script in fund-data work?
A: For exploratory analysis — eyeballing a NAV time series or profiling an EMT file's columns — where you want inline tables and plots; production loaders stay as scripts.

Q: How do you pick which kernel a VS Code notebook runs against?
A: Click the kernel picker at the top-right of the notebook and select the interpreter/environment (typically the project `.venv`).

Q: Why prefer the project `.venv` kernel over a global one for an exploration notebook?
A: So the notebook sees the same locked dependencies as the rest of the repo and your analysis is reproducible by anyone who runs `uv sync`.

Q: How do you launch JupyterLab from a `uv` project?
A: Add it as a dev dependency and run `uv run jupyter lab`, which starts the server using the project environment.

Q: What is the variable explorer / Data Viewer in VS Code notebooks?
A: A panel that lists current variables and lets you open tabular data (like a DataFrame of fund rows) in a scrollable, sortable grid.

Q: Can you set breakpoints inside notebook cells in VS Code?
A: Yes — the Jupyter extension supports cell debugging, so you can step through a parsing cell just like a script.

Q: Why are notebooks awkward under version control compared to scripts?
A: The `.ipynb` JSON embeds execution counts and output blobs, producing noisy diffs and merge conflicts; scripts diff cleanly line by line.

Q: What is one discipline that keeps notebooks reproducible despite out-of-order execution?
A: Periodically "Restart Kernel and Run All" so the notebook actually works top-to-bottom, since cells can otherwise depend on hidden out-of-order state.

Q: After opening `fundcli` in WSL, `code .` works but `python` isn't found in the terminal — first check?
A: Whether you've activated/selected the project `.venv`; the bare `python` may not be on PATH in the fresh WSL shell, and this course runs Python via `uv run`.

Q: Why does this course standardise on `uv run` rather than bare `python` inside the editor terminal?
A: `uv run` guarantees the locked project environment is used every time, avoiding the "wrong interpreter" class of bugs across machines.

Q: The Ruff extension shows no diagnostics on a file you know is messy — first thing to check?
A: That the Ruff extension is enabled in the WSL workspace (not only on the UI side) and that the file is recognised as Python with a Ruff config present.

Q: Format-on-save isn't running Ruff — what setting is most likely wrong?
A: `editor.defaultFormatter` isn't set to Ruff for Python (or `editor.formatOnSave` is off) in the active settings scope.

Q: What is the quickest way to see all keyboard shortcuts and rebind them?
A: Open "Preferences: Open Keyboard Shortcuts" (`Ctrl+K Ctrl+S`) to search, view, and reassign every binding.

Q: How do you open settings as raw JSON to edit precisely?
A: Run "Preferences: Open User Settings (JSON)" or "Open Workspace Settings (JSON)" from the command palette.

Q: What does the Problems panel show?
A: An aggregated list of diagnostics — lint errors, type warnings, and syntax errors — across open files, opened with `Ctrl+Shift+M`.

Q: How do you jump between problems/errors in a file from the keyboard?
A: `F8` goes to the next problem and `Shift+F8` to the previous one.

Q: What does the Source Control panel (`Ctrl+Shift+G`) give you?
A: A Git view of staged/unstaged changes, inline diffs, staging, commits, and branch operations without leaving the editor.

Q: What is the gutter "blame"/inline diff useful for when reviewing a loader change?
A: Seeing exactly which lines changed and their prior content, so a one-character currency-code edit is obvious before you commit.

Q: What is a Quick Fix and how do you invoke it?
A: A context action (auto-import, fix-all, etc.) offered by the language server/Ruff at the cursor, invoked with `Ctrl+.`.

Q: What does the "Testing" panel do for a pytest project?
A: After configuring the test framework, it discovers tests into a tree you can run or debug individually, with pass/fail markers in the gutter.

Q: How do you debug a single failing pytest test from the editor?
A: Use the "Debug Test" gutter/CodeLens action above the test, which launches it under the debugger so you can break inside the code under test.

Q: Why attach VS Code to WSL instead of editing WSL files over the `\\wsl$` share from a Windows-side window?
A: The remote attach runs the toolchain natively in Linux with correct permissions and speed, while the network-share approach suffers slow IO and missing Linux tooling.

Q: You set a breakpoint in a worker thread/async coroutine and it doesn't pause — what is a common cause?
A: The breakpoint is on a code path that runs in a context the debugger didn't attach to, or the line never executes; verify the call actually reaches it and the right config is launched.

Q: What is "justMyCode" in a Python debug configuration?
A: A setting that, when true, steps only through your code and skips library internals; set it false to step into third-party packages like a CSV/parquet reader.

Q: An EMT file parser throws on row 5000 only in production data — how do you pinpoint it fast in the debugger?
A: Set a conditional breakpoint on the parse line with a condition matching that row (e.g. the offending ISIN or `row_num == 5000`) so you pause exactly where it fails.

Q: Why is the integrated terminal's environment important when running a `uv` command interactively?
A: It inherits the workspace/WSL environment and selected interpreter, so `uv run pytest` there uses the same Python and dependencies the editor analyses.

Q: What is the single biggest payoff of completing this lesson before later phases?
A: A frictionless edit-run-debug loop — lint, terminal, debugger, and notebooks in one keyboard-driven window — so tooling never taxes the four years of work that follow.
