# Phase 0 · 0.10 Editor & IDE setup (VS Code) — 100 self-test questions

Q: What is VS Code?
A: A free, cross-platform code editor from Microsoft that runs natively on Ubuntu and gains language support, debugging, and tooling through installable extensions.

Q: How do you install VS Code on Ubuntu using the `.deb` package?
A: Download the `.deb` from the official site and install it with `sudo apt install ./code_*.deb`, which also registers Microsoft's apt repository so future updates arrive through `apt upgrade`.

Q: How do you install VS Code on Ubuntu via Snap?
A: Run `sudo snap install code --classic`; the `--classic` flag is required because the editor needs broad filesystem and tool access that strict snap confinement would block.

Q: What is the difference between the `.deb` and Snap installs of VS Code on Ubuntu?
A: The `.deb` integrates with apt and tends to start faster, while the Snap is sandboxed and self-updating; both give the same editor, so pick whichever fits how you manage software on your machine.

Q: What does the `code` command do from an Ubuntu terminal?
A: It launches VS Code; `code .` opens the current directory as a workspace, and `code myfile.py` opens a single file in the running window.

Q: What is a "workspace" in VS Code?
A: The project context VS Code opens — usually a single root folder (or a multi-root `.code-workspace` file) — that scopes the file explorer, search, settings, and recommended extensions to that project.

Q: How do you open a project folder as a workspace from the terminal?
A: From inside the project directory run `code .`, or run `code /path/to/project` from anywhere to open that folder.

Q: What is a multi-root workspace?
A: A workspace defined by a `.code-workspace` file that holds several top-level folders at once, useful when one task spans separate repositories like a loader repo and a shared library.

Q: Where does VS Code store per-project settings on disk?
A: In a `.vscode/settings.json` file at the root of the workspace folder.

Q: What is the difference between User settings and Workspace settings?
A: User settings apply globally to every project for your account, while Workspace settings live in `.vscode/settings.json` in the project folder and override User settings only for that project.

Q: Why commit a `.vscode/settings.json` to a team repo like `fundcli`?
A: So everyone gets the same interpreter, formatter, and lint behaviour — a NAV loader formatted and linted identically on every machine avoids "works on my editor" diffs.

Q: What shell does the integrated terminal run on Ubuntu by default?
A: It launches your default login shell, which on a standard Ubuntu install is `bash`, opened in the workspace folder.

Q: How do you open the integrated terminal in VS Code on Ubuntu?
A: Use the Terminal menu, the `Ctrl+backtick` keybinding, or the command palette command `Terminal: Create New Terminal`; on Ubuntu it opens a `bash` session in the workspace folder.

Q: How do you run a second terminal alongside the first inside VS Code?
A: Click the `+` in the terminal panel or run `Terminal: Create New Terminal`; you can split the active terminal side by side with `Ctrl+Shift+5`.

Q: Why does running `uv run pytest` in the integrated terminal behave the same as in a standalone terminal?
A: The integrated terminal is a real `bash` session inheriting your environment, so any command that works in a normal Ubuntu terminal works identically here.

Q: What is the VS Code command palette and how do you open it?
A: A keyboard-driven menu for running any command by name, opened with `Ctrl+Shift+P`; almost every action in the editor is reachable through it.

Q: What does Quick Open (`Ctrl+P`) do?
A: It lets you jump to any file in the workspace by typing part of its name, making file navigation fast without using the explorer.

Q: What does Go to Definition (`F12`) do?
A: It jumps from a symbol under the cursor to where that function, class, or variable is defined, even across files in the project.

Q: What does Rename Symbol (`F2`) do?
A: It renames the symbol under the cursor and updates every reference to it across the project in one operation, instead of you doing a risky find-and-replace.

Q: How is `F2` rename safer than a plain text find-and-replace?
A: It uses the language server's understanding of scope, so it renames only the actual symbol references and skips unrelated text that happens to share the name.

Q: What is multi-cursor editing?
A: Placing several insertion points at once so the same edit happens in multiple places simultaneously, for example adding a prefix to many lines.

Q: How do you add a cursor on the next occurrence of the current selection?
A: Press `Ctrl+D` to select and add a cursor at the next matching occurrence; repeat to keep adding.

Q: How do you add cursors above or below the current line?
A: Press `Ctrl+Alt+Up` or `Ctrl+Alt+Down` to stack a cursor on the adjacent line.

Q: What is the Extensions view and how do you open it?
A: The panel for searching, installing, and managing extensions, opened with `Ctrl+Shift+X` or from the activity bar.

Q: What does the Python extension add to VS Code?
A: It provides Python language support — interpreter selection, IntelliSense via Pylance, debugging, test discovery, and the environment picker in the status bar.

Q: What is Pylance?
A: The fast language server bundled with the Python extension that supplies code completion, type checking, hover information, and go-to-definition for Python.

Q: What does the Ruff extension add?
A: It integrates the Ruff linter and formatter into the editor, showing lint diagnostics inline and formatting files on save when configured.

Q: What does the Jupyter extension add?
A: It lets you create and run `.ipynb` notebooks inside VS Code, connecting to Jupyter kernels and rendering cell output in the editor.

Q: What does the Docker extension add?
A: It gives a sidebar view of images, containers, and registries plus commands to build, run, and inspect containers, with syntax help for `Dockerfile` and `compose.yaml`.

Q: How do you install an extension from the command line?
A: Run `code --install-extension <publisher.name>`, for example `code --install-extension charliermarsh.ruff`.

Q: What is the difference between installing and enabling an extension?
A: Installing downloads the extension to your machine, while enabling activates it; you can keep an extension installed but disabled globally or just for one workspace.

Q: Why might you disable an extension for a single workspace?
A: To keep a heavy or noisy extension out of projects that do not need it while leaving it active elsewhere, using the per-workspace Disable option in the Extensions view.

Q: What is an interpreter in the context of the Python extension?
A: The specific `python` executable VS Code uses for IntelliSense, running, and debugging — typically the one inside your project's virtual environment.

Q: How do you select the Python interpreter for a workspace?
A: Run `Python: Select Interpreter` from the command palette and pick the desired environment, or click the interpreter shown in the status bar.

Q: Where does this course's virtual environment live, and what creates it?
A: In a `.venv` folder at the project root, created by `uv`; commands are run through `uv run` so the right environment is always used.

Q: How do you point VS Code at the `.venv` that `uv` created?
A: Run `Python: Select Interpreter` and choose `./.venv/bin/python`; VS Code usually lists it automatically once the folder exists.

Q: Why does this course run Python via `uv run` rather than activating the venv?
A: `uv run` resolves and uses the project environment automatically for each command, so you avoid forgetting to activate and you always run against the locked dependencies.

Q: What is the Run and Debug panel?
A: The sidebar view, opened with `Ctrl+Shift+D`, where you start debug sessions, see call stacks, variables, watches, and breakpoints, and pick a debug configuration.

Q: What is a breakpoint?
A: A marker on a line that pauses execution there when the debugger reaches it, so you can inspect the program's state at that moment.

Q: How do you set or toggle a breakpoint on a line?
A: Click in the gutter to the left of the line number, or put the cursor on the line and press `F9`.

Q: What does Step Over (`F10`) do during debugging?
A: It executes the current line fully — including any function it calls — and stops on the next line in the same function.

Q: What does Step Into (`F11`) do?
A: It descends into the function called on the current line so you can debug that function's body line by line.

Q: What does Step Out (`Shift+F11`) do?
A: It runs the rest of the current function and pauses again at the line that called it.

Q: What does Continue (`F5`) do while paused?
A: It resumes running until the next breakpoint is hit or the program ends.

Q: How do you inspect a variable's value while paused at a breakpoint?
A: Read it in the Variables pane of the Run and Debug panel, or hover the variable in the editor to see its current value.

Q: What is a watch expression?
A: An expression you add to the Watch pane that the debugger re-evaluates each time execution pauses, so you can track a computed value like `nav / shares` across steps.

Q: What is a conditional breakpoint?
A: A breakpoint that only pauses when a condition you supply evaluates to true, for example `isin == "LU0123456789"`, so you skip irrelevant iterations.

Q: How do you create a conditional breakpoint?
A: Right-click the gutter and choose Add Conditional Breakpoint, or right-click an existing breakpoint and choose Edit Breakpoint, then type the condition.

Q: What is a logpoint?
A: A breakpoint variant that logs a message to the Debug Console instead of pausing, letting you trace values without stopping or editing the code with print statements.

Q: How do you add a logpoint?
A: Right-click the gutter, choose Add Logpoint, and enter a message; wrap expressions in curly braces, for example `row={row} nav={nav}`.

Q: Why is a logpoint often better than adding a temporary `print`?
A: It produces no code change, so there is nothing to forget and remove later and no risk of committing stray debug output.

Q: What is `launch.json` and where does it live?
A: A file in `.vscode/launch.json` that defines named debug configurations — what to run, with which arguments, environment, and interpreter — for the Run and Debug panel.

Q: How do you create an initial `launch.json`?
A: In the Run and Debug panel click "create a launch.json file" and pick a Python configuration; VS Code scaffolds the file for you to edit.

Q: How do you pass command-line arguments to a program in `launch.json`?
A: Set the `"args"` array in the configuration, for example `"args": ["--date", "2026-06-12"]` to feed a NAV loader its run date.

Q: How can a `launch.json` configuration set environment variables for the debugged process?
A: Use the `"env"` object to define variables, or point `"envFile"` at a `.env` file whose contents are loaded before the program starts.

Q: You set a breakpoint in a NAV filter that should drop rows with a null ISIN, but it never pauses — what is the first thing to check?
A: Confirm that branch actually runs with your input; if no row has a null ISIN, the line is never reached, so add a logpoint earlier or feed test data that triggers the condition.

Q: A conditional breakpoint `nav < 0` in your pricing loop never fires — what should you verify first?
A: Check the expression evaluates as you expect against the real data types, since a string `"-1.5"` is not less than the integer `0`; cast or compare correctly.

Q: In a transfer-agency loader you want to stop only on the row for one investor account — how do you avoid stepping through thousands of rows?
A: Set a conditional breakpoint such as `account_id == "TA000457"` so execution pauses only on the iteration you care about.

Q: What is the Source Control panel?
A: The sidebar view, opened with `Ctrl+Shift+G`, that shows changed files, lets you stage and unstage changes, write commit messages, and run common Git actions.

Q: How do you stage a changed file in the Source Control panel?
A: Hover the file under Changes and click the `+` (Stage Changes) icon to move it into Staged Changes.

Q: How do you commit staged changes in VS Code?
A: Type a message in the Source Control input box and press `Ctrl+Enter`, or click the commit checkmark.

Q: What is a "task" in VS Code?
A: A defined command — like a build, test, or lint run — configured in `.vscode/tasks.json` and launchable from the command palette so you do not retype it.

Q: How do you run a configured task?
A: Run `Tasks: Run Task` from the command palette and pick the task, or bind the default build task to `Ctrl+Shift+B`.

Q: Write a tasks.json command that runs this course's tests through `uv`.
A: Define a task with `"command": "uv"` and `"args": ["run", "pytest"]`, which executes `uv run pytest` in the integrated terminal when launched.

Q: Why put repeated commands in `tasks.json` instead of retyping them?
A: It standardises the exact command for everyone on the project, reduces typos, and makes the action one keystroke away.

Q: How does running a notebook in VS Code differ from running it in JupyterLab?
A: Both use the same Jupyter kernels and run the same `.ipynb` files, but VS Code embeds notebooks in the editor with its debugging and Git tooling, while JupyterLab is a browser environment with its own multi-document layout.

Q: Can a notebook authored in JupyterLab be opened and run in VS Code?
A: Yes; `.ipynb` is a shared format and both tools connect to the same kernels, so the notebook runs in either with no conversion.

Q: What hidden-state caveat applies to notebooks regardless of which tool you use?
A: Cells run out of order leave variables in the kernel that no longer match the visible code, so always Restart Kernel and Run All to confirm the notebook is reproducible.

Q: How do you select which kernel a VS Code notebook uses?
A: Click the kernel picker at the top right of the notebook and choose the Python environment, such as the project `.venv`.

Q: Why might you prefer VS Code notebooks over JupyterLab for a repo you commit to Git?
A: VS Code shows notebook diffs and lets you commit from the same window, keeping editing, debugging, and source control in one place.

Q: Why might you prefer JupyterLab over VS Code notebooks?
A: JupyterLab's browser layout, multiple synchronized views, and rich extension ecosystem suit exploratory, multi-notebook data work some people find more ergonomic.

Q: How do you format the current file on demand in VS Code?
A: Run `Format Document` from the command palette, which applies the configured formatter — Ruff in this course — to the whole file.

Q: How do you enable format-on-save with Ruff for a workspace?
A: In `.vscode/settings.json` set `"editor.formatOnSave": true` and set Ruff as the default formatter for Python files via `"editor.defaultFormatter": "charliermarsh.ruff"`.

Q: How do you make Ruff fix lint issues and organise imports automatically on save?
A: Add `"editor.codeActionsOnSave"` entries such as `"source.fixAll.ruff": "explicit"` and `"source.organizeImports.ruff": "explicit"` to settings.

Q: How do you set the default formatter specifically for Python without affecting other languages?
A: Use a language-scoped block in settings, `"[python]": { "editor.defaultFormatter": "charliermarsh.ruff" }`, so the choice applies only to Python files.

Q: VS Code does not list your project's `.venv` in `Python: Select Interpreter` — what is the first thing to check?
A: Make sure the `.venv` folder actually exists at the workspace root by running `uv sync` first; then use "Enter interpreter path" to point at `./.venv/bin/python` if it still does not appear.

Q: Ruff is installed but the file is not reformatting on save — what is the first thing to check?
A: Verify `"editor.formatOnSave"` is true and that Ruff is set as the Python default formatter, since without a default formatter VS Code does not know what to run on save.

Q: Ruff shows lint warnings but never reformats — what distinction explains this?
A: Linting and formatting are separate actions; you need format-on-save or `Format Document` configured to apply Ruff's formatter, as diagnostics alone do not rewrite the file.

Q: Breakpoints show as hollow grey circles and say "not bound" — what does that usually mean?
A: The debugger has not associated the breakpoint with running code, often because the file being executed is not the file with the breakpoint or the wrong interpreter is selected.

Q: A hollow "unverified" breakpoint persists when debugging — what should you check first?
A: Confirm you launched the debugger (not just ran the file), that the source path matches the executed module, and that the interpreter is the project `.venv`.

Q: `code .` returns "command not found" in the terminal — for a Snap install, how do you fix the `PATH`?
A: Ensure `/snap/bin` is on your `PATH`; the Snap places the launcher there, and re-logging in or adding it to your shell profile makes `code` resolve.

Q: On a `.deb` install, `code` is missing from `PATH` after a manual extract — how do you add it from inside VS Code?
A: Run the command palette command `Shell Command: Install 'code' command in PATH`, which creates the launcher so `code .` works from `bash`.

Q: What does `Ctrl+Shift+P` give you that menus do not?
A: Searchable access to every command, including ones with no menu entry, so you can find an action by typing part of its name instead of hunting through menus.

Q: How do you split the editor to view two files side by side?
A: Press `Ctrl+backslash` to split the active editor, or drag a tab to the side of the editor area.

Q: How do you search across all files in the workspace?
A: Open the Search view with `Ctrl+Shift+F` and type your query; results group by file and you can replace across the project.

Q: How do you jump to a specific line number?
A: Press `Ctrl+G` and type the line number, or type a colon and number in Quick Open.

Q: What does the Problems panel show?
A: A consolidated list of diagnostics — lint errors, type warnings, syntax issues — across open files, opened with `Ctrl+Shift+M`.

Q: What is the status bar interpreter indicator for?
A: It shows the currently selected Python interpreter and is a one-click shortcut to `Python: Select Interpreter`.

Q: How do you reload the VS Code window without fully restarting it?
A: Run `Developer: Reload Window` from the command palette, which restarts the editor process and reloads extensions, often clearing transient glitches.

Q: How do you see which extensions are recommended for a project?
A: Add an `.vscode/extensions.json` with a `"recommendations"` list; teammates opening the workspace are prompted to install them.

Q: Why list the Python, Ruff, Jupyter, and Docker extensions in `extensions.json` for a fund-data repo?
A: So a new contributor is prompted to install exactly the tools the project relies on, getting consistent linting, notebooks, and container support from day one.

Q: For cross-platform awareness, how does running VS Code natively on Ubuntu differ from a colleague using WSL2 on Windows or macOS?
A: On native Ubuntu the editor and terminal run directly on Linux with no bridge, whereas a Windows colleague's Remote-WSL setup runs the editor UI on Windows and the workspace inside a Linux subsystem; the editing experience is similar but the filesystem and paths differ.

Q: What does `"files.exclude"` in settings do?
A: It hides matching files and folders — like `__pycache__` or `.venv` — from the file explorer to reduce clutter without deleting them.

Q: How do you comment or uncomment the current line or selection?
A: Press `Ctrl+slash`, which toggles line comments using the current language's comment syntax.

Q: How do you move the current line up or down without cutting and pasting?
A: Press `Alt+Up` or `Alt+Down` to shift the line, taking the whole selection with it if several lines are selected.

Q: How do you trigger code completion (IntelliSense) manually?
A: Press `Ctrl+Space` to invoke the suggestion list at the cursor when it does not appear automatically.

Q: How do you find all references to a symbol?
A: Right-click it and choose Find All References, or press `Shift+F12`, to list every place it is used across the project.

Q: How do you run a single test from a Python test file in VS Code?
A: With the Python extension's test discovery enabled, use the green run icon beside the test in the editor gutter or the Test Explorer to run just that test.

Q: How do you debug a single failing test rather than the whole suite?
A: Use the debug icon next to that test in the gutter or Test Explorer, which launches it under the debugger so your breakpoints in the code under test bind.

Q: Why prefer keyboard-first workflows like the command palette and Quick Open?
A: They keep your hands on the keyboard and let you reach files and commands by name in a couple of keystrokes, which is faster and more accurate than mouse-hunting through menus once the shortcuts are habitual.
