# Phase 6 · 11.6.1 Audit logging & SIEM — 100 self-test questions

Q: What is the core promise of audit logging that this lesson is built around?
A: Audit logging means that questions like "who accessed fund X's investor data in March" can be answered from logs that were *designed in advance* to capture access, change, and export events. The mechanism is a planned audit taxonomy plus tamper-evident retention, so the answer is a five-minute query with a documented chain of custody rather than a forensic reconstruction guessed at during a CSSF inquiry. Without that design, you may have logs, but not the *right* logs at the *right* fidelity.

Q: Define "audit-event taxonomy" in the context of a data platform.
A: An audit-event taxonomy is the deliberate, documented classification of which categories of events you capture — typically access (reads), policy changes, exports, and admin/privileged actions. It matters because auditors and investigators ask in those categories ("who changed the masking policy", "who exported investor data"), so your capture layer must map cleanly onto them. Designing the taxonomy first prevents the common failure of capturing high-volume noise while missing the few event types an auditor actually samples.

Q: What four broad event categories does this lesson name for a data-platform audit taxonomy?
A: The four named categories are access events, policy-change events, export events, and admin events. Access events record who read what data; policy-change events record changes to security controls like grants, masking, or row-level security; export events record data leaving the platform; and admin events record privileged operations such as role creation or configuration changes. These map onto the questions a regulator or internal auditor is most likely to pose.

Q: Why is it a mistake to design audit logging "in panic during" an inquiry rather than in advance?
A: Because the events you need must have been captured *before* the period under question — you cannot retroactively log March's reads in June. If the taxonomy, retention, and immutability were not configured in advance, the log either does not contain the events, contains them at insufficient fidelity, or cannot be proven untampered. The result is a reconstruction that "satisfies no auditor," whereas a pre-designed system answers the question directly with a defensible chain of custody.

Q: What is pgAudit?
A: pgAudit is a PostgreSQL extension that provides detailed session and object audit logging through the standard Postgres logging facility. It exists because Postgres's built-in `log_statement` is too coarse for compliance — it cannot, for example, log only reads/writes against specific tables. pgAudit is the *capture layer* for Postgres in this lesson; it does not centralize or correlate, it just emits structured audit records.

Q: How is pgAudit loaded so that it can intercept statements?
A: pgAudit must be added to the `shared_preload_libraries` configuration parameter and the server restarted, because it hooks into Postgres at backend startup via executor and processing hooks. You then run `CREATE EXTENSION pgaudit` in each database where you want it. A common gotcha: `CREATE EXTENSION pgaudit` must be called before `pgaudit.log` is set in a session, or the extension will not function correctly.

Q: What does the `pgaudit.log` setting control, and what is its default?
A: `pgaudit.log` selects which classes of statements are written to the audit log; its default is `none`, meaning pgAudit captures nothing until you configure it. You set it to a comma-separated list of classes such as `'read, write, ddl, role'`. It can be set globally in `postgresql.conf`, per-database, per-role, or per-session, which lets you raise audit fidelity for sensitive roles without flooding the log for everyone.

Q: List the statement classes that `pgaudit.log` accepts.
A: The classes are `READ`, `WRITE`, `FUNCTION`, `ROLE`, `DDL`, `MISC`, `MISC_SET`, and `ALL`. `READ` covers `SELECT` and `COPY` from a table, `WRITE` covers `INSERT`/`UPDATE`/`DELETE`/`TRUNCATE` and `COPY` to a table, `ROLE` covers `GRANT`/`REVOKE` and role management, and `DDL` covers schema changes excluding role statements. You can also subtract, e.g. `pgaudit.log = 'all, -misc'`.

Q: Which pgAudit class maps to a "policy change" in your audit taxonomy, and why?
A: The `ROLE` class is the primary mapping for policy changes, because it captures `GRANT`, `REVOKE`, and role-management statements — the SQL that alters who can access what. `DDL` is also relevant when policies are enforced through objects such as row-level-security policies, views, or masking functions, since those are created and altered via DDL. Together `ROLE` and `DDL` give you the change-event coverage auditors sample.

Q: Which pgAudit class would you enable to capture investor-data *reads* for a transfer-agency table?
A: `READ`, because it logs `SELECT` and `COPY`-from-table statements, which is exactly how someone "accesses" investor records in a transfer-agency schema. To answer "who read fund X's investor data in March" you need `READ` enabled (or object-level auditing on those specific tables) for the whole period. Without it, the access events simply do not exist in the log.

Q: What is the difference between SESSION audit logging and OBJECT audit logging in pgAudit?
A: SESSION logging records *all* statements a user runs that match the configured classes in `pgaudit.log`, giving broad coverage regardless of which object is touched. OBJECT logging is finer-grained: it logs only statements that affect specific relations the audit role has been granted on, acting as a targeted replacement for `pgaudit.log = 'read, write'`. You use SESSION for a baseline and OBJECT to tightly audit a small set of sensitive tables without logging everything.

Q: How is OBJECT-level audit logging actually wired up in pgAudit?
A: You set `pgaudit.role` to a dedicated audit role, then `GRANT` that role the privileges (`SELECT`, `INSERT`, `UPDATE`, `DELETE`) on exactly the relations you want audited. pgAudit then logs any statement that would use those granted privileges on those objects, so the *grants define the audit scope*. OBJECT logging applies only to `SELECT`, `INSERT`, `UPDATE`, and `DELETE`, not to DDL or other commands.

Q: What does `pgaudit.log_catalog` do and why might you turn it off?
A: `pgaudit.log_catalog` controls whether statements that touch only `pg_catalog` system tables are logged; it defaults to `on`. Tools and clients constantly introspect the catalog (e.g. psql tab-completion, ORMs), so leaving it on can flood the audit log with noise that has no compliance value. Many deployments set it to `off` to keep the log focused on real data access.

Q: What is the effect of `pgaudit.log_parameter`, and what is the compliance trade-off?
A: `pgaudit.log_parameter` (default `off`) includes the bound parameter values of a statement in the audit record, so `SELECT ... WHERE isin = $1` shows the actual ISIN queried. This is valuable for proving *which* investor or instrument was accessed, but it can also write sensitive values (PII, account numbers) into logs that then need the same protection as the source data. The trade-off is evidentiary richness versus expanding the data-protection surface of your logs.

Q: When parameters are not logged, what placeholder appears in the pgAudit record?
A: The parameter field shows `<not logged>`. This happens when `pgaudit.log_parameter` is `off` (the default) or when a parameter exceeds `pgaudit.log_parameter_max_size`. Seeing `<not logged>` tells you the statement template was captured but the concrete values were intentionally suppressed, which is often the right default to avoid leaking PII into log storage.

Q: Name the main fields in a pgAudit log record.
A: A pgAudit entry carries `AUDIT_TYPE` (SESSION or OBJECT), `STATEMENT_ID`, `SUBSTATEMENT_ID`, `CLASS` (READ, WRITE, etc.), `COMMAND` (e.g. `SELECT`), `OBJECT_TYPE` (TABLE, VIEW, INDEX), `OBJECT_NAME` (fully-qualified), the `STATEMENT` text, and `PARAMETER`. These fields are emitted as a CSV fragment inside the Postgres log line. Having a stable, structured schema is what makes the log machine-queryable downstream in a SIEM.

Q: How do you read `AUDIT: SESSION,1,1,DDL,CREATE TABLE,TABLE,public.account` at a glance?
A: This is a SESSION audit record, statement id `1`, substatement id `1`, class `DDL`, command `CREATE TABLE`, object type `TABLE`, object name `public.account`. It tells you a schema-change (DDL) event occurred creating the `public.account` table within the first statement of that backend call. Reading the leading tokens in order — type, ids, class, command, object — is how you triage any pgAudit line.

Q: What is the purpose of `STATEMENT_ID` and `SUBSTATEMENT_ID` in pgAudit output?
A: `STATEMENT_ID` is unique per top-level call to the backend, and `SUBSTATEMENT_ID` numbers the sub-statements within it (for example, statements expanded from a function or a multi-statement query). Together they let you reconstruct exactly which logged lines belong to one user action, even when one call fans out into several executed statements. This is essential for honest "who did what in one operation" reconstruction.

Q: Why is pgAudit preferred over simply setting `log_statement = 'all'` in Postgres?
A: `log_statement = 'all'` logs every statement as raw text with no classification, no object resolution, and no ability to scope by table or audit role, producing huge, low-structure logs that are hard to query and easy to flood. pgAudit classifies each statement, resolves object names, supports object-level scoping, and emits a stable field layout. The result is targeted, machine-parseable audit evidence rather than an undifferentiated firehose.

Q: What is the Trino event listener, conceptually?
A: The Trino event listener is a plugin SPI that lets you receive structured events about query lifecycle on a Trino cluster, so you can audit *federated* queries across all the data sources Trino fronts. It is the per-engine capture layer for Trino, parallel to pgAudit for Postgres. Because Trino often sits over many catalogs (Iceberg, Postgres, object stores), its event listener may be the single place where cross-source access is observable.

Q: Which lifecycle events does the Trino `EventListener` interface expose?
A: It exposes `queryCreated` (fired when a query is submitted), `queryCompleted` (fired on success or failure), and `splitCompleted` (fired as individual splits finish). For audit purposes `queryCompleted` is the most important, because the `QueryCompletedEvent` carries the full query text, user, source, timing, and outcome. `splitCompleted` is usually too low-level and high-volume for audit and is more useful for performance telemetry.

Q: How do you register a custom event listener on a Trino cluster?
A: You install the listener as a plugin on the cluster, then create `etc/event-listener.properties` containing `event-listener.name=<factory-name>` matching the name returned by the `EventListenerFactory`, plus any listener-specific properties. Trino loads it at startup. The `event-listener.name` value is the contract that ties your config file to the right factory implementation.

Q: Can a Trino cluster run more than one event listener at once, and how?
A: Yes. You set the config property `event-listener.config-files` to a comma-separated list of properties files, for example `etc/event-listener.properties,etc/event-listener-audit.properties`. Each file configures one listener, and all of them receive events. This lets you, say, send one stream to an audit sink and another to a performance-monitoring system independently.

Q: What audit-relevant information does a Trino `QueryCompletedEvent` carry?
A: It includes the query text, the submitting user, the source/client, session properties, execution timing, resource usage, the catalogs/schemas touched, and whether the query succeeded or failed. That combination lets you answer "who ran what query, against which data, when, and did it succeed" for federated access. The query text plus user plus accessed tables is the audit core; the resource metrics are mostly operational.

Q: Why does federated access make the Trino event listener especially important for audit?
A: Because Trino can query Postgres, Iceberg, object storage, and more behind one SQL endpoint, a user might read fund data through Trino without ever connecting to the underlying engine directly. If you only audited Postgres, you would miss those reads. The Trino event listener captures access *at the federation layer*, ensuring cross-source queries are recorded where they actually originate.

Q: What is a gap to watch for when relying on both pgAudit and the Trino event listener together?
A: Double-counting and clock/identity mismatches: a Trino query against a Postgres catalog can appear in *both* the Trino events and pgAudit, often under different usernames (the Trino service account at the Postgres layer versus the human user at Trino). You must reconcile identities and timestamps when correlating, or "who read this" becomes ambiguous. Documenting which layer is authoritative for which question is part of the taxonomy design.

Q: What does "centralization" of audit logs mean and why is it required?
A: Centralization means shipping audit streams from every engine into one system where they are stored, retained, and queried together, instead of leaving them scattered on individual hosts. It is required because an auditor's question spans engines and hosts, hosts get reimaged or rotated, and a single query surface is what makes the "five-minute answer" possible. It also enables uniform retention and immutability controls.

Q: In this lesson's FOSS stack, what is Loki's role?
A: Loki is the central log aggregation system that both the pgAudit and Trino audit streams are shipped to, serving as the open-source stand-in for Splunk, Sentinel, or Elastic SIEM. It indexes log streams by labels and lets you query them with LogQL. Loki is the *centralization* layer; the SIEM correlation and alerting role is what Sentinel/Splunk add on top.

Q: What does "tamper-evidence" require of an audit log store?
A: Tamper-evidence requires that any modification or deletion of stored audit records be detectable and ideally prevented, so an auditor can trust that what they see is what was recorded. Mechanisms include append-only/immutable storage, write-once policies, cryptographic hashing/chaining, and strict access controls separating who can read logs from who could alter them. Without it, the logs prove nothing because they could have been edited after the fact.

Q: How does Azure Monitor / Microsoft Sentinel describe its data immutability model?
A: Microsoft Sentinel inherits Azure Monitor's tamper-proofing and immutability practices, and Azure Monitor is described as an *append-only* data platform — ingested records cannot be altered. It does, however, include provisions to delete data for compliance purposes (for example to satisfy a GDPR erasure obligation). So the model is "append-only, with controlled compliance deletion," not "absolutely unchangeable forever."

Q: Why does an append-only platform still need a deletion path?
A: Because regulations like GDPR grant a right to erasure, and you may be legally required to remove specific personal data even from logs. An append-only-but-deletable model squares this circle: normal operations cannot tamper with records, preserving evidentiary value, while a narrow, audited deletion capability exists for genuine compliance needs. The deletion itself should be logged so erasure does not become a tampering blind spot.

Q: What is "retention" in audit-log terms, and why must it be documented?
A: Retention is how long audit records are kept before deletion or archival, and it must be documented because regulators expect a stated, enforced period (often years for financial-services security logs). The "Done when" of this lesson requires showing retention and immutability are configured and recorded, because an undocumented or ad-hoc retention period cannot be defended in an examination. Documentation also drives the chain-of-custody narrative.

Q: In Azure Monitor Logs, how do retention and the archive tier relate for security logs?
A: A table can be interactively queryable for an analytics-retention period (commonly up to two years, e.g. 90 days kept interactive then archived), after which data moves to a cheaper archive tier and is retained until a total period such as 730 days before deletion. Archived data is not queryable with ordinary KQL, but a *search job* can run a KQL query against the archive and write results to a new table. This lets you keep audit data affordably for compliance while keeping recent data fast to query.

Q: What is a SIEM and what does it add beyond plain log centralization?
A: A SIEM (Security Information and Event Management) adds correlation, analytics rules, alerting, incident grouping, and response orchestration on top of centralized logs. Where Loki just stores and lets you query, a SIEM like Sentinel or Splunk turns many low-fidelity events across sources into high-fidelity incidents and can trigger automated responses. In short, centralization makes logs queryable; the SIEM makes them *actionable*.

Q: What is Microsoft Sentinel?
A: Microsoft Sentinel is a cloud-native SIEM (and SOAR) that delivers scalable security across multicloud and multiplatform environments, combining AI, automation, and threat intelligence for threat detection, investigation, response, and hunting. It is built on top of an Azure Monitor Log Analytics workspace, ingests data via connectors, and is queried with KQL. In this lesson it represents the SIEM role: correlation, immutability, and KQL querying.

Q: What does "correlation" mean in a SIEM, with an example?
A: Correlation is combining multiple individually-unremarkable events into one meaningful signal. For example, a single `SELECT` on an investor table is normal, but a SIEM rule that correlates "an account read investor PII from 200 funds, from a new IP, outside business hours, then triggered an export" produces one high-fidelity incident. Sentinel's analytics explicitly "connect the dots, combining low-fidelity alerts about different entities into potential high-fidelity security incidents."

Q: How does Sentinel group related alerts, and what is the resulting object called?
A: Sentinel's analytics rules group related alerts into *incidents*, which are the unit analysts investigate. An incident bundles the alerts, the entities involved, and an investigation graph so the scope and root cause can be traced. This grouping is what reduces alert fatigue: instead of hundreds of raw alerts, responders work a smaller set of correlated incidents.

Q: What is SOAR and how does it appear in Sentinel?
A: SOAR (Security Orchestration, Automation, and Response) is the automated-response side of a modern SIEM, and in Sentinel it appears as automation rules and playbooks. Playbooks are built on Azure Logic Apps and can, for instance, open a ServiceNow ticket or disable an account automatically when a particular incident fires. SOAR is why a detected "mass investor-data export" can trigger containment without waiting for a human.

Q: What is KQL and where is it used here?
A: KQL (Kusto Query Language) is the query language of Azure Monitor / Log Analytics and therefore of Microsoft Sentinel, used to search, filter, aggregate, and correlate centralized logs. In this lesson it is the language you would write the "who accessed fund X in March" query in once logs are centralized in a Sentinel/Log Analytics workspace. It ties back to the A.5 KQL material referenced in the syllabus.

Q: Write the shape of a KQL query that finds who read an investor table in March.
A: A typical shape is `AuditEvents | where TimeGenerated between (datetime(2026-03-01) .. datetime(2026-04-01)) | where ObjectName == "public.investor" and Class == "READ" | project TimeGenerated, User, Statement | order by TimeGenerated asc`. KQL uses the pipe `|` to chain operators left to right, with `where` for filtering, `project` to select columns, and `summarize` for aggregation. This is exactly the five-minute, end-to-end "who touched what when" query the lesson targets.

Q: In KQL, what do the `where`, `project`, and `summarize` operators do?
A: `where` filters rows by a predicate, `project` selects and optionally computes the columns you keep, and `summarize` aggregates rows into groups (for example `summarize count() by User`). They are chained with the pipe operator so each stage feeds the next. For audit work you typically `where` down to the time window and object, then `project` the user and statement, or `summarize` counts per user.

Q: What do auditors actually sample from audit logs, according to this lesson?
A: Auditors predominantly sample *access events* and *change events* — who read sensitive data and who altered controls or configuration — rather than exhaustively reviewing every log line. This is why the taxonomy emphasizes access, policy-change, export, and admin categories: those are the buckets a sample is drawn from. Designing for "what auditors actually request" keeps you from over-capturing trivia while under-capturing the events that get inspected.

Q: Why is "chain of custody" part of the lesson's done-when criteria?
A: Chain of custody is the documented, unbroken account of how a piece of evidence (here, an audit record) was produced, stored, protected, and retrieved, proving it was not altered. The lesson requires answering the March-access question "with a documented chain of custody" because an answer an auditor cannot trace and trust is worthless. It combines the immutable store, the retention policy, and the query path into a defensible narrative.

Q: Scenario: pgAudit is configured but the audit log is empty. What is the first thing to check?
A: First confirm `pgaudit.log` is actually set to a non-default value — its default is `none`, so a fresh install logs nothing until you set classes like `'read, write'`. Then verify pgAudit is in `shared_preload_libraries` and the server was restarted, and that `CREATE EXTENSION pgaudit` ran before `pgaudit.log` was set. An empty log almost always means the capture classes were never enabled or the extension was not preloaded.

Q: Scenario: pgAudit floods your log with thousands of catalog lookups. What setting fixes this?
A: Set `pgaudit.log_catalog = off` so statements touching only `pg_catalog` are not audited; that suppresses the introspection noise from clients and ORMs. The default is `on`, which is why a verbose deployment fills with catalog reads. After that, narrow `pgaudit.log` to the classes you actually need (or use object-level auditing) to keep the log focused on real data access.

Q: Scenario: your SIEM shows a Postgres read but the username is a generic service account. What is likely happening?
A: The read probably arrived through Trino (or another middleware) connecting to Postgres with a single pooled service account, so pgAudit records the service identity, not the human. To attribute it to a person you must correlate with the Trino `QueryCompletedEvent`, which carries the original Trino user. This identity-reconciliation step is essential or attribution breaks at the federation boundary.

Q: Scenario: an auditor asks for March access but logs only go back 30 days. What design failure occurred?
A: The retention policy was too short for the obligation — audit/security logs in regulated finance often need years, not 30 days. Either retention was never configured beyond a default or hot data was deleted without archiving to a long-term tier. The fix is a documented retention period aligned to regulatory expectations, with cheaper archive storage for older data and search-job access when needed.

Q: Scenario: someone claims the audit log "could have been edited." How do you rebut this?
A: You demonstrate tamper-evidence: an append-only/immutable store (e.g. Azure Monitor's append-only platform or WORM object storage), access controls separating log readers from anyone who could alter storage, and documented immutability/retention settings. If hashing or chaining is in place, you show the integrity check passes. The rebuttal is the *recorded* configuration plus the access-control separation, which together make undetected edits implausible.

Q: How would you ship pgAudit output into a centralized store like Loki?
A: pgAudit writes through the normal Postgres logging facility, so you point Postgres to emit logs (ideally as `csvlog`/`jsonlog`) and run a log agent such as Promtail (or an OpenTelemetry collector) to read those files and push them to Loki with labels like `engine=postgres`, `env=prod`. The agent handles batching and labelling; Loki indexes the streams for LogQL queries. Keeping a structured format (CSV/JSON) preserves the pgAudit fields for downstream parsing.

Q: How would you get Trino audit events into the same central store?
A: You configure a Trino event listener (registered via `etc/event-listener.properties` with `event-listener.name`) that forwards `queryCompleted` events to your sink — for example an HTTP/OpenTelemetry-based listener that writes to Loki, or a listener that emits to a file a log agent then ships. The key is that the listener serializes the `QueryCompletedEvent` (user, query text, accessed catalogs, timing, outcome) into structured records. Then both Postgres and Trino streams live under one query surface.

Q: Why label audit streams by engine and environment when shipping to Loki?
A: Labels are how Loki indexes and lets you scope LogQL queries efficiently, so labels like `engine=postgres` vs `engine=trino` and `env=prod` vs `env=test` let you query the right stream fast and avoid mixing production audit data with test noise. Without good labels, every query scans everything and attribution gets muddled. Thoughtful, low-cardinality labels are part of making the five-minute query actually fast.

Q: What is the danger of putting high-cardinality values (like full query text) into Loki labels?
A: Loki builds an index per unique label-value combination, so high-cardinality labels (query text, user id, timestamps) cause an index explosion that degrades performance and memory use. Such values belong in the log *line*, queried with LogQL filter expressions, not in labels. Keep labels to a small, bounded set (engine, environment, event class) and filter the rest at query time.

Q: How does an audit-export event differ from an audit-access event, and why track both?
A: An access event records that data was *read* in place (a `SELECT`), while an export event records that data *left* the platform (a download, `COPY TO`, an extract to a file or external system). You track both because reading and exfiltrating are different risk levels: a mass export of investor PII is far more serious than the same volume read interactively. Auditors and DLP rules treat exports as a distinct, higher-scrutiny category.

Q: What is an "admin event" in the audit taxonomy, with an example?
A: An admin event is a privileged operation that changes the platform itself rather than business data — creating or dropping roles, altering server configuration, changing audit settings, or granting superuser. For example, `ALTER ROLE app_user WITH SUPERUSER` or disabling pgAudit is an admin event of the highest interest. These are sampled heavily because they can be used to *defeat* the audit system, so changes to logging config must themselves be logged.

Q: Why is auditing changes to the audit configuration itself important?
A: Because an attacker or insider who can silently disable or narrow audit logging can then act unobserved, so the act of changing audit settings must be captured before the change can blind you. In pgAudit terms, changes to `pgaudit.log` or to `shared_preload_libraries`, and role grants, should be auditable (MISC_SET / ROLE / DDL classes and OS-level config-file controls). This closes the "who turned off the cameras" gap that auditors specifically look for.

Q: Map the pgAudit class for a fund analyst running `SELECT * FROM nav_history`.
A: That statement is captured under the `READ` class (with command `SELECT`, object type `TABLE`, object name `nav_history`), because `READ` covers `SELECT` and table `COPY`-from. In your taxonomy this is an access event. If `pgaudit.log_parameter` is on and the query were parameterized by ISIN or fund id, the specific value would also be recorded, sharpening the "who accessed which fund" answer.

Q: A user runs `COPY investor_data TO '/tmp/extract.csv'`. Which class and taxonomy bucket?
A: `COPY ... TO` a file is captured under the `WRITE`/export side of pgAudit (COPY to a destination), and in the taxonomy it is an *export* event — data leaving the platform to local disk. This is exactly the high-scrutiny case: a bulk extract of investor records. Such events should fire SIEM correlation rules because they are how PII exfiltration typically looks.

Q: Why might `SELECT` be classified as `READ` while `COPY` to a table is `WRITE` — what is the rule?
A: pgAudit classifies by the data effect: `READ` covers statements that read rows out (`SELECT` and `COPY` *from* a table), and `WRITE` covers statements that modify rows (`INSERT`/`UPDATE`/`DELETE`/`TRUNCATE` and `COPY` *to* a table). `COPY` is ambiguous on its own, so its direction determines the class. Understanding the direction-based rule prevents mis-mapping `COPY` events in your taxonomy.

Q: In a fund-administration context, why audit access to ISIN-keyed instrument and NAV tables?
A: NAV figures and the instruments behind them (keyed by ISIN) are price-sensitive and confidential before publication, so unauthorized access can enable front-running or leakage. Auditing reads of these tables lets you prove that only entitled roles saw pre-release NAV and which ISINs were touched. It supports both market-abuse controls and the general "who saw fund X's data" question regulators ask.

Q: How does LEI relate to audit logging in a fund platform?
A: The LEI (Legal Entity Identifier) identifies the legal entities — funds, management companies, counterparties — your data describes, so audit events can be enriched or filtered by LEI to answer entity-scoped questions like "who accessed data for this sub-fund's legal entity." It is reference data rather than an audit mechanism itself, but tagging access events with the relevant LEI sharpens regulatory reporting and entitlement reviews. The same applies to ISIN at the instrument level.

Q: How do EMT/EPT files intersect with export auditing?
A: EMT (European MiFID Template) and EPT (European PRIIPs Template) files are regulated data exchanges sent to distributors, so generating and sending them are *export* events that should be audited end to end. You want a record of who triggered each EMT/EPT extract, what data it contained, and where it went, because these files carry product and cost data subject to MiFID II and PRIIPs. Treating their production as audited exports supports both data-governance and regulatory traceability.

Q: How does DORA motivate centralized audit logging and a SIEM?
A: DORA (Digital Operational Resilience Act) imposes ICT-risk management, incident detection, and tight incident-reporting timelines on EU financial entities, which require the ability to detect, investigate, and reconstruct events quickly. Centralized, immutable audit logs feeding a SIEM with correlation and alerting are precisely the controls that make DORA-grade detection and reporting feasible. Without them, you cannot evidence operational resilience or hit reporting deadlines.

Q: How does a CSSF inquiry concretely test your audit design?
A: A CSSF inquiry typically asks targeted access/change questions — for example, who accessed a specific fund's investor data in a given month, or who changed a particular control — and expects a prompt, evidenced answer. Your design is tested on three axes: did you capture the right events (taxonomy), can you retrieve them fast (centralization + query), and can you prove they are untampered (immutability + chain of custody). The lesson's whole premise is passing that test in five minutes, not a forensic scramble.

Q: Why is SWIFT message flow a candidate for audit logging in this context?
A: SWIFT carries settlement and payment instructions, so access to and changes in SWIFT-related message data are high-value security and compliance events worth auditing. While SWIFT itself is a messaging network rather than your data platform, when those messages land in your stores, reads, modifications, and exports of that data should fall under the same taxonomy. Unusual access to settlement instructions is exactly the kind of low-and-slow signal a SIEM should correlate.

Q: What is the difference between Loki's role and Sentinel's role in this lesson's stack?
A: Loki is the FOSS *centralization and storage* layer you actually deploy hands-on — aggregating both audit streams and serving LogQL queries. Sentinel (alongside Splunk and Elastic SIEM) represents the *SIEM* layer that adds correlation, analytics rules, incident management, immutability guarantees, and KQL querying. The lesson uses Loki to learn the mechanics and Sentinel as the reference for what a full enterprise SIEM provides.

Q: What is LogQL and how does it compare to KQL here?
A: LogQL is Loki's query language, used to select log streams by label and filter their contents, conceptually similar in role to KQL for Sentinel. Both let you scope to a time window, filter to access/change events, and aggregate by user, but KQL is richer for cross-source correlation and joins. In this lesson you practice the "who touched what when" query in LogQL against Loki, knowing the same intent maps to KQL in a Sentinel deployment.

Q: Why must audit-log *read* access itself be restricted and monitored?
A: Audit logs often contain sensitive data (user identities, accessed object names, possibly PII parameters), so unrestricted read access leaks the very information you are trying to protect and lets a bad actor learn what is and isn't observed. Reads of the audit store should be limited to a small group and ideally logged too. Separating "who can read logs" from "who can alter storage" is also part of tamper-evidence.

Q: What does `pgaudit.log_relation` do, and when is it useful?
A: `pgaudit.log_relation` (default `off`) makes pgAudit emit a separate SESSION log entry for *each relation* referenced in a statement, rather than one entry for the whole statement. It is useful when a single query joins several sensitive tables and you need explicit per-table evidence of access. The trade-off is more log volume, so enable it where per-relation attribution matters, such as multi-table investor queries.

Q: What does `pgaudit.log_statement_once` change about output volume?
A: With `pgaudit.log_statement_once = on`, pgAudit logs the full statement text only once per statement/substatement pair rather than repeating it on every related entry, reducing log size. The default is `off`, which repeats the statement for completeness. You enable it when statements are long and the repetition wastes storage, accepting that you must join sub-entries back to the first to see the text.

Q: What is the role of `pgaudit.log_level` and why might you change it?
A: `pgaudit.log_level` sets the severity at which audit entries are written (default `log`). Changing it affects how the entries are filtered and routed by the Postgres logging pipeline and downstream agents — for example, raising it could make entries stand out, though it can also interfere with normal level-based filtering. In practice most deployments leave it at `log`; you change it only with a clear routing reason.

Q: What does `pgaudit.log_client` control, and why is the default `off` a security choice?
A: `pgaudit.log_client = on` would make audit messages visible to the connecting client, which defaults to `off` so that audited users do not see (and cannot rely on or scrape) the audit trail of their own actions. Keeping audit output server-side only is a basic integrity measure — the subject of auditing should not be fed the audit record. You almost never turn this on in production.

Q: Why prefer `csvlog` or `jsonlog` output format for Postgres when using pgAudit?
A: Structured formats like `csvlog` or `jsonlog` keep pgAudit's fields in discrete, parseable columns/keys instead of embedding them in a free-text log line, which makes downstream ingestion into Loki or a SIEM reliable. A log agent can then map fields directly without brittle regex parsing. This preserves the audit schema end to end, so queries on `User`, `Class`, or `ObjectName` actually work.

Q: An auditor asks "show all GRANTs on the investor schema last quarter." Which pgAudit class and how?
A: That is the `ROLE` class (it covers `GRANT`/`REVOKE`/role management), so you query the centralized store for `Class == "ROLE"` and `Command` like `GRANT`, scoped to the investor schema objects and the quarter's time window. These are policy-change events in the taxonomy. Because they are exactly what auditors sample, ensuring `role` is in `pgaudit.log` from day one is essential.

Q: What is the risk if you only enable `WRITE` auditing but not `READ`?
A: You would capture modifications but miss all data *access*, so the central regulatory question "who *read* fund X's investor data" becomes unanswerable. Many compliance and privacy concerns are about reads/exfiltration, not just writes. Enabling only `WRITE` is a common under-scoping mistake; for investor PII you generally need `READ` (or object-level auditing) on the sensitive tables.

Q: Why might object-level auditing be preferable to `pgaudit.log = 'read, write'` for a few sensitive tables?
A: Object-level auditing logs reads/writes only on the specific relations you grant the audit role on, giving you high-fidelity evidence for those tables without the volume and noise of session-wide read/write logging across the whole database. It is described as a finer-grained replacement for `pgaudit.log = 'read, write'`. The result is a smaller, sharper log focused exactly on the regulated tables auditors care about.

Q: How do you reconcile timestamps between Trino and Postgres audit streams?
A: Ensure all hosts use synchronized time (NTP/chrony) and store timestamps in UTC, then correlate on a common time window plus the query/identity link rather than expecting exact equality, since the engines record slightly different lifecycle moments. Document which event's timestamp you treat as authoritative (e.g. Trino's `queryCompleted` time). Clock drift or mixed time zones is a frequent cause of "the events don't line up" confusion.

Q: What is a "search job" in Azure Monitor and when do you use it for audit?
A: A search job runs a KQL query against data in the archive (or large datasets) and writes the results into a new table, because archived data is not directly interactive-queryable. You use it when an audit question reaches back beyond the interactive retention window — for example a two-year-old access request — to retrieve just the matching records affordably. It is how long-retained audit data stays usable without keeping everything in the hot tier.

Q: Why is "the answer should take five minutes" a meaningful design target rather than a nicety?
A: Because the speed of the answer is a proxy for whether the design is right: a five-minute answer means the events were captured, centralized, indexed, and trustworthy, while a slow answer means you are reconstructing from inadequate sources. Under DORA/CSSF timelines, slow also means non-compliant. The target forces end-to-end design — taxonomy, centralization, retention, immutability, and a known query path — rather than just "we have logs somewhere."

Q: What is the difference between an alert and an incident in a SIEM?
A: An alert is a single rule firing on a pattern (e.g. "100 investor reads in a minute"), whereas an incident is a correlated grouping of related alerts and entities that an analyst investigates as one case. SIEMs like Sentinel group alerts into incidents to cut noise and provide investigation context such as an entity graph. Tuning happens at both levels: rule thresholds for alerts, grouping logic for incidents.

Q: Give an example correlation rule a SIEM might run over fund-platform audit logs.
A: "Same user reads investor PII across more than N distinct funds within M minutes, then performs an export" is a classic correlation that turns benign individual reads into one exfiltration incident. The SIEM joins access events and export events across the Postgres and Trino streams, applies the threshold, and raises a high-fidelity incident. This is precisely the low-fidelity-to-high-fidelity dot-connecting Sentinel describes.

Q: Why is data normalization (e.g. Sentinel's ASIM) useful when correlating multi-engine audit logs?
A: Normalization maps differently-shaped logs from Postgres, Trino, and others into a uniform schema so a single correlation rule or query can span them without per-source special-casing. Sentinel does this via query-time and ingestion-time normalization (the Advanced Security Information Model). For audit, it means "user", "object", and "action" mean the same thing regardless of which engine emitted the event, which is what makes cross-engine correlation tractable.

Q: A pgAudit entry shows `PARAMETER:<not logged>` for a sensitive query. Is that a problem?
A: Not necessarily — `<not logged>` is the expected output when `pgaudit.log_parameter` is `off` or a parameter exceeds `pgaudit.log_parameter_max_size`, and suppressing values is often the right privacy choice. It becomes a problem only if your audit *requires* knowing the specific value accessed (e.g. which ISIN), in which case you would enable parameter logging for that scope and accept the added data-protection burden. So whether it is acceptable depends on the question you must answer.

Q: How would you prove retention and immutability are configured for an examiner?
A: You show the documented retention period and the storage's immutability/append-only configuration (e.g. Azure Monitor's append-only platform settings, or a WORM/object-lock policy on the log store), plus the access controls separating log readers from storage admins. You pair this with evidence the settings have been in force across the relevant period. The "Done when" requires this to be *configured and recorded*, so screenshots/policy docs plus the live settings are the artifact.

Q: What goes wrong if audit logs and the data they describe share the same access controls and storage?
A: If a single role can both modify the data and delete the logs of that modification, the audit trail provides no real assurance — the same actor can cover their tracks. Separation of duties (distinct identities and immutable, separately-controlled log storage) is what gives logs evidentiary weight. Co-locating them with shared controls is a tamper-evidence failure auditors will flag immediately.

Q: Why is "documented audit-event taxonomy" itself a deliverable in this lesson's "Do" steps?
A: Because the taxonomy is the contract that says exactly which events you claim to capture and how they map to access/change/export/admin categories, and a compliance officer must be able to read it to understand coverage. The lesson's "Do" step 3 is explicitly "define the audit-event taxonomy you capture." A written taxonomy turns "we log stuff" into a defensible statement of scope, including honestly flagged gaps.

Q: What is the relationship between this lesson's audit logging and a later GDPR DSAR/erasure obligation?
A: Audit logs may themselves contain personal data (user identities, accessed records, parameters), so a GDPR erasure or DSAR can reach into the logs, which is why the append-only platform still needs a controlled deletion path. You must design erasure to handle log copies and backups, and to record that the erasure itself happened. This is the bridge to the regulatory-frameworks lesson (11.7.1), where the erasure path through Iceberg, backups, and Kafka is designed.

Q: Why might `splitCompleted` events be unsuitable for the audit stream specifically?
A: `splitCompleted` fires per split as a query's work is parallelized, producing high-volume, low-level execution telemetry that does not answer "who ran what against which data." Audit cares about the query as a whole — user, statement, accessed catalogs, outcome — which lives in `queryCompleted`. Sending `splitCompleted` to your audit sink would balloon volume and cost with no compliance benefit; it belongs in performance monitoring.

Q: What is the value of capturing `source`/client information in Trino's `QueryCompletedEvent`?
A: The `source`/client field tells you *how* the query arrived — which application, BI tool, or service — which sharpens attribution and anomaly detection beyond just the username. For example, an investor-data read from an unexpected client or an ad-hoc CLI rather than the sanctioned reporting app is a signal worth correlating. It enriches the access event with provenance context auditors and SIEM rules can use.

Q: How does immutability interact with the need to fix a logging misconfiguration?
A: Immutability protects *already-ingested* records, not your ability to change *future* logging behavior — you can and should correct a misconfiguration (e.g. add `READ` to `pgaudit.log`) going forward. What you cannot do is rewrite history to fabricate the events you failed to capture. The honest move is to document the gap and the date it was fixed, which the lesson explicitly endorses ("flag gaps honestly").

Q: What distinguishes "session" pgAudit logging from Postgres's plain server log of all statements, even when both are on?
A: Even with full statement logging, the plain server log lacks pgAudit's classification (`READ`/`WRITE`/`ROLE`/`DDL`), resolved object names, audit-type marker, and object-level scoping, so it cannot answer class- or table-scoped audit questions without heavy reparsing. pgAudit's SESSION mode emits purpose-built, structured audit records. The distinction is "raw statement text" versus "classified, attributable audit evidence."

Q: A compliance officer asks "can you brief me on coverage without hand-waving?" What does that require?
A: It requires that for each audited category (access, policy change, export, admin) you can state exactly what is captured, where it is stored, how long it is retained, that it is immutable, and how you query it — plus an honest list of where no control exists yet. The lesson's done-when is precisely "brief a compliance officer... and where it doesn't yet — without hand-waving." Concrete mappings and named gaps are what distinguish a real briefing from reassurance.

Q: Why are admin/privileged events typically retained longest and reviewed most?
A: Admin events can grant access, alter controls, or disable auditing, so they carry the highest potential impact and are the most attractive to misuse, making them the events auditors and investigators scrutinize most. Long retention ensures that a privilege change made months before an incident is still discoverable. They are the small-volume, high-value tail of the log that justifies extra retention and tighter alerting.

Q: How do you avoid logging sensitive PII into audit records while still proving access occurred?
A: You log the *fact and metadata* of access — user, time, object name, statement template, class — while keeping `pgaudit.log_parameter` off (or capping parameter size) so concrete PII values are not duplicated into the log. This proves "user U accessed table investor at time T" without copying the investor's data into a second, log-shaped store. Where the specific value is genuinely needed for audit, enable parameter logging narrowly and protect those logs accordingly.

Q: What is the consequence of high log volume on the "five-minute query" goal, and how is it mitigated?
A: Excessive volume slows queries, raises storage cost, and buries the events auditors want, undermining the fast-answer goal. Mitigations include scoping `pgaudit.log` to needed classes, disabling `log_catalog` noise, using object-level auditing for sensitive tables, good Loki labels/indexing, and tiered retention (hot vs archive). The aim is *signal-dense* audit logs, not maximal logs.

Q: Why is "answer one who-touched-what-when query end to end" a required hands-on exercise?
A: Because it forces you to validate the entire pipeline at once — capture (pgAudit/Trino), centralization (Loki), query language (LogQL/KQL), and the chain-of-custody narrative — proving the design works rather than just looking plausible on paper. Many audit setups fail only when first queried under pressure. Doing it once in calm conditions is what guarantees the real CSSF question is a five-minute answer.

Q: How would Splunk or Elastic SIEM differ from Loki for this use case?
A: Splunk and Elastic SIEM are full SIEM platforms that, like Sentinel, add correlation rules, alerting, incident management, dashboards, and often richer retention/immutability features, whereas Loki is primarily a cost-efficient log-aggregation and query layer. For learning, Loki teaches the centralization mechanics with the FOSS stack; in an enterprise you would typically use a SIEM for the correlation and alerting on top. The audit *capture* (pgAudit, Trino listener) is identical regardless of sink.

Q: What is the danger of treating audit logging as "just turn on logging" with no taxonomy?
A: You end up with high-volume, unclassified logs that may still miss the specific access/change/export/admin events auditors sample, while drowning the signal in noise. Without a taxonomy you cannot state coverage, cannot tune retention by event value, and cannot answer scoped questions quickly. The taxonomy is what turns logging from a checkbox into an evidence system aligned to real regulatory questions.

Q: Why must export events for regulated files like Annex IV or EMT be especially well audited?
A: Because these are reportable, regulated data flows where the contents and recipients carry legal weight, so you must be able to evidence who generated each extract, what it contained, and where it went. An undocumented or unauthorized export of such files is both a data-governance failure and a potential regulatory breach. Auditing them as first-class export events supports traceability under UCITS/AIFMD, MiFID II, and PRIIPs regimes.

Q: How does the lesson's audit design support DORA's incident-reporting timelines?
A: Centralized, immutable, correlated logs let you detect and scope an ICT incident quickly and reconstruct exactly what happened, which is the prerequisite for filing within DORA's tight reporting deadlines. A SIEM's correlation surfaces the incident early, and the chain of custody supports the report's accuracy. Scattered, mutable, slow-to-query logs would make hitting those timelines effectively impossible.

Q: Final synthesis — name the end-to-end pieces that turn "who accessed fund X in March" into a five-minute answer.
A: You need: an audit-event taxonomy (access/policy/export/admin); per-engine capture (pgAudit with the right classes/object scope, and the Trino event listener on `queryCompleted`); centralization into one queryable store (Loki, or a SIEM like Sentinel/Splunk/Elastic); documented retention with append-only/immutable storage and separated access controls; a query language to ask the question (LogQL/KQL); and a documented chain of custody. With all six in place, the regulator's question is a quick, defensible query instead of a forensic guess.
