# Phase 6 · 11.1.3 Policy-as-code (OPA) — 100 self-test questions

Q: What is policy-as-code, and how does it differ from configuring access rules per system?
A: Policy-as-code expresses authorization rules as version-controlled, testable text (e.g. Rego files) that a single engine evaluates, instead of clicking grants into each tool's own admin UI. The difference is that one rule lives in one repo with tests and review, rather than the "same" rule being re-encoded in three places that silently drift apart. This is the core motivation behind moving the fund-visibility rule out of Trino, the catalogue, and the BI tool and into OPA.

Q: What does OPA stand for, and what kind of component is it?
A: OPA is the Open Policy Agent, a general-purpose policy engine that takes structured input (usually JSON) plus policy written in Rego and returns a structured decision. It is decision-making infrastructure, not an enforcement gateway: it answers "is this allowed?" but something else must actually block the request. It is a graduated CNCF project and the de-facto standard for externalized authorization across APIs, Kubernetes, and data engines.

Q: In OPA terminology, what is a PDP?
A: A PDP is a Policy Decision Point: the component that evaluates policy against input and emits a decision such as allow/deny. OPA itself is the PDP — you send it the request context and it returns the answer. Crucially the PDP only decides; it never enforces.

Q: In OPA terminology, what is a PEP?
A: A PEP is a Policy Enforcement Point: the component sitting in the request path that calls the PDP and then actually allows or blocks the operation based on the decision. In the Trino integration, Trino's OPA access-control plugin is the PEP — it asks OPA for a decision and then permits or rejects the SQL operation. The PEP is where the "yes/no" becomes real behaviour.

Q: Why is PDP/PEP separation (decide vs enforce) considered good architecture?
A: Separating decision from enforcement lets you change authorization logic in one central, testable place without touching every enforcement point, and lets many heterogeneous PEPs (API gateway, Kubernetes admission, Trino) share one policy brain. It also keeps each service simple — the PEP only needs to ask and obey, not understand the rules. The cost is a network call and the need to keep the PDP highly available.

Q: If OPA is "only" a decision engine, what stops an unauthorized query from running?
A: Nothing in OPA itself — the PEP does. OPA returns `false`, but it is the calling system (the PEP) that must honour that answer and refuse the operation. A buggy or bypassable PEP makes a perfect policy useless, which is why the enforcement integration is as important as the Rego.

Q: What is Rego?
A: Rego is OPA's declarative policy language, designed for querying and making decisions over nested JSON/YAML-style data. You write rules that compute values (often a boolean `allow`) from `input` and from loaded `data`. It is purpose-built for authorization rather than being a general programming language.

Q: In Rego, what do `input` and `data` refer to?
A: `input` is the request-specific document the PEP sends for this single decision (e.g. the user, action, and resource), while `data` is the larger, relatively static document OPA holds in memory (e.g. role tables, fund-to-team mappings, allowlists). A rule typically combines the two: it reads the user from `input` and looks them up against `data`. Keeping slow-changing facts in `data` lets policies stay small and reusable.

Q: In Rego, what is a "complete" rule versus a "partial" rule?
A: A complete rule produces a single value (commonly `allow := true`), so the document has one definite value or is undefined. A partial rule builds up a set or object from multiple matching cases (e.g. `allowed_funds contains fund if ...`). You reach for complete rules for yes/no decisions and partial rules for "give me the set of things this user may see".

Q: Why does an undefined Rego rule typically mean "deny"?
A: If no rule body succeeds, the rule produces no value (undefined), and PEPs are written to treat the absence of an explicit allow as denial. This default-deny posture means you must positively grant access; forgetting a case fails closed, not open. It is the safe default for a regulated fund-visibility policy.

Q: In Rego v1 (OPA 1.0+), what keyword is now required to define a conditional rule?
A: The `if` keyword is required before the rule body, e.g. `allow if input.user == "admin"`. OPA 1.0 made `if` and `contains` mandatory so that single-value and multi-value rules are syntactically unambiguous. Older policies without `if` must be migrated or run with a compatibility flag.

Q: In Rego v1, what keyword must a multi-value (set-generating) rule use in its head?
A: It must use `contains`, e.g. `visible_funds contains f if f := data.funds[_]`. OPA 1.0 made `contains` mandatory precisely to distinguish a set-building rule from a single-value rule that just happens to be assigned in a loop. Without it, OPA 1.0 will reject or misinterpret the rule.

Q: How do you write a logical AND of several conditions in a single Rego rule body?
A: You list the conditions on separate lines (or separated by semicolons) inside the body; all must hold for the rule to succeed. For example `allow if { input.action == "read"; input.user_team == resource_team }` requires both. Rego has no explicit `and` operator — conjunction is implicit between statements in a body.

Q: How do you express logical OR for a decision in Rego?
A: You write multiple rules with the same name; if any one succeeds, the value is produced. For instance two separate `allow if ...` blocks mean "allow when block A holds OR when block B holds". This is idiomatic Rego — OR comes from multiple rule definitions rather than an `or` keyword.

Q: What does the underscore `_` mean in a Rego expression like `data.funds[_]`?
A: The underscore is a wildcard variable that iterates over every element (or key) of a collection without binding a named variable. `data.funds[_].isin` effectively means "for some element of `data.funds`, take its `isin`". It is shorthand when you only care that some element matches, not which one.

Q: What is the role of the `default` keyword in a Rego rule?
A: `default allow := false` gives the rule a fallback value used when no conditional definition succeeds, making the deny-by-default behaviour explicit and avoiding an "undefined" result that some callers handle awkwardly. It is best practice to declare `default allow := false` at the top of an authorization policy. The conditional `allow if ...` rules then override it only when access should be granted.

Q: In Rego, how do you reference an array element versus an object field?
A: Both use bracket or dot notation over the document tree: `input.resource.columns[0]` indexes an array element, while `input.resource.schema` reads an object field. Strings keys can use dot (`a.b`) or bracket (`a["b"]`) form; dynamic or non-identifier keys need brackets. The uniformity comes from Rego treating arrays and objects as parts of the same document model.

Q: How would you express in Rego "the user's team owns the fund being accessed"?
A: You bind the relevant facts and compare them, e.g. `allow if { input.action.operation == "SelectFromColumns"; data.fund_owner[input.action.resource.table.schemaName] == input.context.identity.user }`. The rule reads the schema/fund from `input`, looks up its owning team in loaded `data`, and grants only on a match. This is the kind of fund-visibility rule lesson 11.1.2 produced, now in Rego.

Q: What is a Rego "package", and why does it matter for the decision path?
A: A `package` declaration namespaces a set of rules, and it determines the query path the PEP uses, e.g. `package trino` makes the decision reachable at `data.trino.allow`. Trino's plugin queries a specific policy path, so the package name and rule name must match what the plugin expects. Mismatched package/path is a common reason a policy "does nothing".

Q: What does `import rego.v1` do, and when do you need it?
A: `import rego.v1` opts a policy written for OPA v0.x into the v1.0 syntax and semantics (requiring `if`/`contains`) so it behaves identically before and after the 1.0 cutover. In OPA 1.0+ this is the default and the import becomes a no-op; you mainly use it on older engines during migration. Running `opa fmt --rego-v1` can rewrite files to add the needed keywords.

Q: Which CLI command runs OPA's policy unit tests?
A: `opa test` runs all rules whose names begin with `test_` across the given files or directories, reporting pass/fail per test. For example `opa test .` runs every test in the current tree. It is the command you wire into CI so a policy PR cannot merge with failing tests.

Q: How is a Rego unit test written?
A: A test is an ordinary rule named `test_<something>` that asserts the policy's output for a crafted `input`, e.g. `test_admin_allowed if allow with input as {"user": "admin"}`. It succeeds (passes) when its body holds. You write one test per allow case and one per deny case so the suite documents intended behaviour.

Q: What does the `with` keyword do in a Rego test?
A: `with` replaces part of the evaluation environment for that expression, most commonly `with input as {...}` to inject a fake request, or `with data.x as {...}` to mock loaded data. This lets a test exercise the policy against controlled inputs without standing up real data. For example `allow with input as {"action": {"operation": "DropTable"}}` checks the drop-table case in isolation.

Q: How do you write a test that a policy correctly DENIES a case?
A: You assert the negation of the allow decision under the offending input, e.g. `test_outsider_denied if not allow with input as {"user": "stranger"}`. Because deny is the absence of allow, `not allow` succeeding means the policy refused, so the test passes. Pairing explicit allow tests with explicit deny tests is what proves the rule both grants and withholds correctly.

Q: Why are deny-case tests as important as allow-case tests in a fund-visibility policy?
A: Allow tests prove authorized users keep working, but deny tests are what prove the regulated boundary actually holds — that someone outside the fund's team cannot read its NAV or holdings. A policy that allows everyone passes all allow tests yet is catastrophically wrong. Deny tests encode the "must not" requirements an auditor cares about.

Q: What does the `opa coverage` capability tell you, and how do you get it?
A: Coverage reports which lines/branches of your Rego were exercised by the tests, exposing untested paths. You enable it with `opa test --coverage` (often combined with `--format=json`), and it surfaces rules that no test touches. High coverage gives confidence that an authorization change is actually checked, not just present.

Q: A teammate's `opa test` run reports `0 tests` even though test files exist — what is the first thing to check?
A: Check that the test rules are named with the `test_` prefix and live in files OPA actually loaded; OPA only treats `test_`-prefixed rules as tests. Also confirm the path you passed (`opa test .` vs a single file) includes the test files and their package. A common gotcha is naming a rule `admin_test` instead of `test_admin`, so OPA never runs it.

Q: What does `opa eval` do, and when would you reach for it instead of `opa test`?
A: `opa eval` evaluates an arbitrary query against your policy and input and prints the result, e.g. `opa eval -i input.json -d policy.rego "data.trino.allow"`. You use it for ad-hoc debugging — "what does the policy actually return for this exact request?" — whereas `opa test` runs your assertion suite. It is the quickest way to reproduce a PEP's question by hand.

Q: How can you validate that a Rego file is syntactically and type correct before running it?
A: `opa check` parses and type-checks the policy without evaluating it, catching errors like undefined references or v1 incompatibilities early. Adding `--strict` (or `--rego-v1` on older engines) enforces stricter rules. Running it in CI gives fast feedback before the heavier `opa test` stage.

Q: What does `opa fmt` do?
A: `opa fmt` is OPA's autoformatter; it rewrites Rego to a canonical style (indentation, spacing, keyword usage) so diffs stay clean and reviews focus on logic. `opa fmt -w` writes changes in place, and `opa fmt --rego-v1` additionally rewrites legacy syntax to add required `if`/`contains` keywords. Teams usually run `opa fmt` as a pre-commit or CI check.

Q: What is an OPA bundle?
A: A bundle is a gzipped tarball packaging your policies and any `data.json`/`data.yaml` into a single versioned artifact that OPA can load as a unit. It is the reproducible distribution format: the same bundle deployed to every PEP guarantees every PEP evaluates the identical policy. Bundles are how you "ship policy to PEPs reproducibly".

Q: How does OPA typically fetch and refresh bundles in production?
A: OPA is configured with a bundle service URL and polls it on an interval (the bundle plugin), downloading and atomically activating new bundles as they appear. This pull model means you publish a new bundle to the server and every OPA picks it up without a redeploy. It is the mechanism behind "change the Rego in git, and enforcement follows".

Q: How do you build a bundle from the command line?
A: `opa build` compiles a directory of Rego and data into a bundle tarball, e.g. `opa build -b ./policy -o bundle.tar.gz`. The `-b` flag selects bundle mode; the output is the artifact you then publish to a bundle server or registry. You can also produce an optimized/wasm bundle with additional flags.

Q: What is a bundle revision and why does it matter?
A: A bundle carries a `revision` string (often a git SHA or build number) in its manifest, which OPA reports in its status and decision logs. It lets you answer "which exact policy version made this decision?" — essential for audit and for confirming a PEP is on the latest Rego. Matching the revision to a git commit ties a live decision back to a reviewed PR.

Q: How can you confirm a running OPA is serving the newest bundle revision?
A: Query OPA's status — the `/v1/status` API or the status decision log reports the active bundle name and revision — and compare it to the revision you just published. If the revisions match, the PEP is enforcing the latest Rego; if not, OPA hasn't pulled yet or the download failed. This check is the "show enforcement reflects the latest bundle" step.

Q: A new bundle is published but OPA keeps using the old policy — what are the first things to check?
A: Check OPA's status/logs for bundle download errors (bad URL, auth failure, HTTP non-200), confirm the polling interval has elapsed, and verify the new bundle's `revision` actually changed. A bundle that fails activation (e.g. a Rego compile error) is rejected and OPA keeps the last good one — so look for activation errors too. Failing-closed on a bad bundle is by design.

Q: What happens if a freshly published bundle contains a Rego compilation error?
A: OPA rejects the bad bundle during activation and continues serving the last known-good bundle, logging the error. This prevents a typo from taking down enforcement, but it also means a broken policy silently fails to deploy — your `revision` check will show the old version. CI running `opa check`/`opa test` before publish is what keeps broken bundles out.

Q: What is the value of a `.manifest` file in a bundle?
A: The manifest records metadata such as the bundle `revision` and `roots` (which policy paths this bundle owns), which OPA uses for versioning and for delta/multi-bundle ownership. `roots` prevents two bundles from clobbering each other's paths. It is what makes bundle distribution reproducible and auditable.

Q: Which Trino config sets OPA as the access-control provider?
A: In the access-control properties file you set `access-control.name=opa`. This tells Trino to load the OPA system access control plugin instead of the file-based or other providers. Without it, none of the `opa.*` properties take effect.

Q: What does the Trino property `opa.policy.uri` configure?
A: `opa.policy.uri` is the full URL of the OPA decision endpoint Trino queries for each authorization check, e.g. `http://opa:8181/v1/data/trino/allow`. The path after `/v1/data/` selects the Rego package and rule, so it must match your policy's package (`trino`) and rule (`allow`). A wrong path returns undefined, which the plugin treats as deny.

Q: Trino's OPA plugin POSTs a JSON body to OPA — what are its two top-level fields?
A: The body wraps everything under `input`, which contains `context` and `action`. `context` holds the `identity` (the `user` and its `groups`), the `queryId`, and the `softwareStack` (Trino version); `action` holds the `operation`, the `resource`, and related fields like `targetResource` or `grantee`. Your Rego reads `input.context` and `input.action` to decide.

Q: In the Trino-to-OPA request, where do you find the user and their groups?
A: Under `input.context.identity` — `input.context.identity.user` is the authenticated principal and `input.context.identity.groups` is the array of their groups. A typical rule maps a group to allowed operations, e.g. `"data-steward" in input.context.identity.groups`. The groups usually originate from the IdP (Keycloak) mapping covered in 11.1.1.

Q: What does `input.action.operation` carry in a Trino OPA request?
A: It carries the name of the Trino operation being authorized, such as `SelectFromColumns`, `CreateTable`, `DropTable`, or `ExecuteQuery`. Your Rego branches on this string to apply different rules per operation type. Logging unknown operation names is a useful way to discover cases your policy hasn't handled.

Q: What boolean must your Rego ultimately produce for Trino's OPA plugin, and at what path?
A: It must produce `allow` (true to permit, absent/false to deny) at the path the `opa.policy.uri` points to, conventionally `data.trino.allow`. The plugin reads that boolean from OPA's response and enforces accordingly. If the path yields no value, the plugin denies.

Q: What does `opa.policy.batched-uri` enable in the Trino plugin?
A: It points to a batch endpoint so Trino can authorize a whole list of resources (e.g. many tables for filtering) in a single request rather than one call per object. The plugin puts the candidates under `action.filterResources`, and the policy returns the indices of the allowed items. It is a performance optimization for operations like listing/filtering large catalogues.

Q: For a batch request, what must the OPA policy return, and what does an empty list mean?
A: The policy returns a list of the integer indices of the items that are authorized; Trino keeps only those resources. Returning `null` or an empty list denies everything in the batch — they are equivalent. So a batch rule that produces no indices is a full deny, not an error.

Q: What do `opa.log-requests` and `opa.log-responses` do, and why use them carefully?
A: Setting `opa.log-requests=true` / `opa.log-responses=true` makes Trino log the exact JSON it sends to and receives from OPA, which is invaluable when debugging why a query was denied. The caution is that these logs can contain sensitive identity and resource data and are verbose, so you enable them temporarily, not in steady-state production. They default to `false`.

Q: What is the effect of `opa.allow-permission-management-operations` in Trino's OPA plugin?
A: It controls whether Trino permits GRANT/REVOKE-style permission-management operations under OPA control; it defaults to `false`, meaning those operations are denied unless explicitly enabled. Leaving it off is the safe default because managing grants is usually handled by policy/IdP, not ad-hoc SQL. Turn it on only if you intend OPA to govern those operations too.

Q: A `SELECT` in Trino is denied unexpectedly after wiring up OPA — what is the first diagnostic step?
A: Enable `opa.log-requests`/`opa.log-responses` (or run `opa eval` with the captured input) to see the exact request and OPA's response, then confirm whether `allow` was `false` or simply undefined. Undefined usually means a package/path mismatch between `opa.policy.uri` and your Rego, not a logic error. Reproducing the request against OPA directly isolates plugin vs policy problems.

Q: Trino logs show OPA returning HTTP 200 but the query is still denied — what does that indicate?
A: A 200 means OPA answered successfully; denial then comes from the decision content — `allow` was false or absent. So the policy logic (or the path/package the plugin queried) is the culprit, not connectivity. Inspect the response body's `result` and compare against your rule and the `input` it received.

Q: Trino can't reach OPA at all (connection refused) — does the query run?
A: No — if the PEP cannot get a decision it must fail closed and deny, otherwise you'd have an open door whenever OPA is down. This is why OPA availability matters operationally: typically you run an OPA sidecar/agent next to each enforcement point to minimize that failure mode. Check the OPA URL, port `8181`, and network reachability first.

Q: Why is it common to run OPA as a sidecar/local agent rather than one central server?
A: A local OPA per PEP keeps the decision call on localhost — low latency and no shared single point of failure — while bundles still distribute one consistent policy to every agent. Centralizing the PDP as one network service adds latency and an availability risk to every authorization check. The bundle mechanism is what makes the decentralized agents stay consistent.

Q: What is Cedar, in one line?
A: Cedar is an open-source, typed policy language (originating from AWS, used by Amazon Verified Permissions) for expressing and validating authorization policies. It is presented as the typed alternative to Rego when you want schema-checked, more constrained policies. Like OPA it separates decision from enforcement.

Q: What are the three core elements of a Cedar authorization request?
A: Principal, action, and resource — "who is doing what to which thing", plus an optional `context` record. A Cedar policy's scope matches on these, e.g. a given Employee performing `remoteAccess` on a System. This PARC model (principal/action/resource/context) is Cedar's fixed shape, unlike Rego's free-form JSON.

Q: What do the Cedar keywords `permit` and `forbid` mean?
A: `permit` grants access when the policy's scope and conditions match; `forbid` denies it. Cedar evaluates all policies, and a matching `forbid` always overrides any `permit` — forbid wins. This explicit deny-overrides model differs from Rego, where you compose denial yourself.

Q: In Cedar, what happens when both a `permit` and a `forbid` match the same request?
A: The `forbid` takes precedence and the request is denied. Cedar's evaluation is "deny if any forbid matches, else allow if some permit matches, else deny by default". This makes hard prohibitions (e.g. "never expose un-released NAV") robust against an over-broad permit.

Q: What is a Cedar schema, and what does it buy you?
A: A Cedar schema is a JSON declaration of the entity types, actions, and the expected shape of `context` your application uses. It enables typed validation: Cedar can statically check that policies reference real entity types and attributes and apply actions to the right principal/resource types. This catches whole classes of mistakes — a typo'd attribute or applying an action to the wrong resource type — before runtime.

Q: How does `permit`/`forbid` with a schema make Cedar "typed", and why might a fund firm value that?
A: Because the schema declares types, Cedar's validator rejects a policy that, say, checks a non-existent `fund.navStatus` or applies a `redeem` action to an `Investor` instead of a `Position`. In a regulated context that static guarantee reduces the risk that a refactor silently breaks an access boundary. It trades some flexibility for compile-time safety.

Q: When might Cedar be preferable to Rego/OPA?
A: When your authorization domain is well-modelled with stable entity types and you want schema-validated, more constrained, statically checkable policies — Cedar's typing prevents a class of authoring errors. It also fits app-embedded authorization (per-request principal/action/resource) cleanly. The lesson frames Cedar as the typed alternative for exactly these situations.

Q: When might Rego/OPA be preferable to Cedar?
A: When you need to reason over arbitrary nested JSON, integrate many heterogeneous PEPs (Kubernetes, Trino, API gateways), or express logic that doesn't fit a fixed principal/action/resource shape. OPA's flexibility and broad ecosystem (including the Trino plugin) make it the lingua franca across systems. The trade-off is that Rego is untyped, so you lean on tests rather than a schema for correctness.

Q: Both Cedar and OPA separate decision from enforcement — so what is the practical difference for an architect?
A: The PDP/PEP split is shared, but Cedar gives you a typed, schema-validated language with deny-overrides semantics, while OPA gives an untyped, maximally flexible language with a huge integration ecosystem. The choice hinges on whether static typing or breadth of integration matters more for your estate. For a Trino-centric data platform, OPA's existing plugin often decides it.

Q: What are the three headline wins of policy-as-code over per-system grants?
A: Consistency (one rule enforced identically everywhere), auditability (every change is a reviewable, attributable git commit tied to a bundle revision), and CI for policies (automated tests prove the rule before it ships). Together they turn authorization from tribal admin clicks into engineered, verifiable artifacts. This is the "where policy-as-code wins" summary.

Q: How does policy-as-code improve consistency specifically?
A: A single policy artifact (bundle) is distributed to every PEP, so the same rule evaluates the same way in Trino, the API gateway, and elsewhere — no per-tool re-encoding to drift. The fund-visibility rule stops living in three places. Consistency here is structural, not a matter of discipline.

Q: How does policy-as-code improve auditability?
A: Every policy change is a git commit with an author, timestamp, diff, and review, and the deployed bundle carries a `revision` you can tie back to that commit. So "who could see this fund in March, and under what rule?" becomes a `git log` plus a bundle-revision lookup, not guesswork. Decision logs further record what was decided and by which revision.

Q: What is "CI for policies", concretely?
A: It is running `opa fmt --check`, `opa check`, and `opa test` (with coverage) on every PR so a policy change cannot merge unless it is formatted, compiles, and passes its allow/deny assertions. The pipeline then builds and publishes a bundle. This is what lets you change a rule as a reviewed PR with passing tests instead of filing three admin tickets.

Q: Why is "three tickets to three admins" the anti-pattern this lesson targets?
A: Because manual per-system grants mean a single logical change must be re-implemented in each tool by each admin, with no shared test proving they did it consistently or correctly. Drift, leavers retaining access, and unanswerable audit questions follow. Policy-as-code replaces the three tickets with one tested, reviewed PR.

Q: How does policy-as-code help with the "leaver keeps access" problem?
A: When access derives from group membership evaluated by central policy, removing the person from the IdP group (or the data table) instantly changes every PEP's decision via the next bundle/data refresh. There is no per-tool grant to forget to revoke. The single source of truth is what closes the leaver gap.

Q: What is a decision log in OPA, and why is it relevant to regulated funds?
A: A decision log is a record OPA can emit for each decision — the input, the result, the policy bundle revision, and a timestamp. For a fund administrator it provides the evidence trail for "this access was/was not granted under this rule version", which supports DORA-style operational and audit requirements. It turns authorization into reviewable history.

Q: Give an example of expressing a fund-visibility rule's allow path in Rego for the Trino plugin.
A: Something like `package trino` then `default allow := false` and `allow if { input.action.operation == "SelectFromColumns"; some g in input.context.identity.groups; g == data.fund_teams[input.action.resource.table.schemaName] }`. It permits a SELECT only when one of the user's groups matches the team that owns the fund schema. The owning-team mapping lives in `data`, so the rule stays small and the facts are auditable.

Q: How would you load the fund-to-team mapping into OPA so the policy can use it?
A: You ship it as `data.json`/`data.yaml` inside the bundle (or push it via the Data API), placing it under a path like `data.fund_teams`, which the policy reads. Bundling it keeps policy and reference data versioned together and reproducible. Pushing it separately suits data that changes faster than policy.

Q: Why keep slow-changing facts in `data` rather than hard-coding them in Rego rules?
A: Hard-coded values force a policy edit (and re-test/re-deploy) for every data change, and they bloat the rules with lists that aren't logic. Putting fund-to-team mappings in `data` lets the same rule serve changing facts and lets data update on its own cadence. It also keeps the rule readable and the diff meaningful when logic actually changes.

Q: What does it mean to "make the policy change follow enforcement automatically", as the lesson's Do step requires?
A: You edit the Rego in git, CI tests and builds a new bundle, the bundle server publishes it, and each OPA polls and activates it — so Trino's next decision uses the new rule without any manual reconfiguration of Trino. The demonstration is: change a rule, watch a previously-allowed query become denied (or vice versa) once the new revision is live. That end-to-end flow is the lesson's payoff.

Q: After changing a Rego rule from allow to deny, what is the minimum you should verify end to end?
A: That `opa test` passes for the new behaviour, that the published bundle's `revision` increased and OPA's status shows it active, and that running the actual Trino query now reflects the change. Skipping the bundle-revision check is the usual reason "I changed it but Trino still allows it". All three together prove the pipeline works.

Q: What does it mean that Rego is a "declarative" language?
A: You describe the conditions under which a decision holds rather than writing step-by-step control flow; OPA evaluates the rules to find which produce values. There is no explicit loop counter or mutation — iteration comes from variable binding and wildcards. This makes policies read like specifications of "what is allowed".

Q: Why can't you reassign a variable inside a single Rego rule body?
A: Rego variables are bound, not mutated — once a variable takes a value in a rule body it is fixed for that evaluation, like a logical unknown being solved. Trying to "reassign" is a compile error or simply a different binding. This immutability is what makes declarative evaluation predictable and analyzable.

Q: How do you iterate over a collection and apply a condition in Rego?
A: You use comprehensions or `some`/wildcard binding, e.g. `some i; input.action.filterResources[i].schema == "funds"` finds matching indices, and `[x | x := data.funds[_]; x.released]` builds a filtered list. Iteration is expressed through binding a variable across the collection plus a condition, not an imperative `for`. Comprehensions are the workhorse for building sets/arrays/objects.

Q: What is a Rego comprehension, with an example?
A: A comprehension builds a set, array, or object by iterating and filtering, e.g. the array comprehension `released := [f.isin | f := data.funds[_]; f.status == "released"]`. It reads as "collect each fund's ISIN where the fund's status is released". Comprehensions replace imperative loops for constructing collections.

Q: In Rego, what does the `some` keyword introduce?
A: `some` declares one or more local variables that exist for the scope of the body, so they are treated as bound iteration variables rather than references to the global document. For example `some g in input.context.identity.groups` iterates the user's groups. Declaring with `some` (or `some ... in`) avoids accidental references to `data`/`input` paths of the same name.

Q: What is the `in` operator used for in modern Rego?
A: `in` tests or iterates membership, e.g. `"data-steward" in input.context.identity.groups` checks the user is in that group, and `some x in coll` iterates. It is part of the `future`/v1 keyword set that is standard in OPA 1.0. It makes membership checks far more readable than manual indexing.

Q: How does Rego handle a missing key, e.g. `input.action.targetResource` that isn't present?
A: Referencing a non-existent path yields undefined for that expression, which makes the enclosing rule body fail (not error), so the rule simply doesn't produce a value. This is why optional fields naturally cause default-deny rather than a crash. You can guard with checks or use object helpers when you need a default instead of undefined.

Q: What is the difference between `==` and `:=` in Rego?
A: `:=` assigns (binds) a new local variable, while `==` is a comparison that must hold for the body to succeed. So `x := input.user` binds, and `x == "admin"` tests. Using `=` (unification) is older style; modern Rego favours `:=` and `==` for clarity, and OPA 1.0 enforces some of these distinctions.

Q: Why is testing especially critical for an untyped language like Rego?
A: Because there is no schema to catch a misspelled field or a wrong path at compile time, a typo can silently make a rule undefined and thus deny-everything or, worse, an over-broad rule that allows too much. Allow/deny unit tests are your safety net, asserting the actual decisions. This is partly why Cedar's typing is offered as an alternative.

Q: What command and pattern would you put in CI to gate a policy PR?
A: Run `opa fmt --check .`, `opa check --strict .`, and `opa test --coverage .` so the job fails on bad formatting, compile errors, or any failing/uncovered assertion. On success a later step runs `opa build` and publishes the bundle. This pipeline is precisely what replaces "ticket to three admins" with a reviewed, tested PR.

Q: How do bundles relate to the GitOps idea of policy?
A: The git repo is the source of truth for Rego and data; CI turns merges into versioned bundles whose `revision` traces to the commit; OPA agents pull and activate them. So the deployed authorization state is always a function of git, reviewable and revertible. Reverting a bad policy is a `git revert` plus the next bundle pull.

Q: If you need to roll back a policy quickly, what does policy-as-code let you do?
A: Revert the offending commit (or republish a prior bundle revision), and the agents pull the older policy on their next refresh — enforcement rolls back without per-tool edits. Because every bundle is an immutable, revisioned artifact, you can pin to a known-good revision. This is far safer than trying to manually undo grants across three systems.

Q: What is the risk of putting too much logic in the PEP instead of the PDP?
A: If the PEP encodes business rules, you lose the single-source-of-truth benefit — logic drifts back into each system, defeating consistency and auditability. The PEP should stay dumb: gather input, ask OPA, obey. Keeping decisions in Rego is what makes the policy testable and centrally changeable.

Q: How does the input the Trino plugin sends shape how you write your Rego?
A: You write rules against the exact paths Trino sends — `input.context.identity.user`, `input.context.identity.groups`, `input.action.operation`, `input.action.resource.table.schemaName`/`tableName`/`columns` — so your policy must mirror that contract. Misreading a path (e.g. expecting `input.user`) silently yields undefined and denies. Capturing a real request via `opa.log-requests` is the reliable way to learn the shape.

Q: What does the `resource` portion of a Trino OPA `action` typically contain for a table SELECT?
A: It carries the object being accessed, e.g. `resource.table` with `catalogName`, `schemaName`, `tableName`, and for column-level checks a `columns` list. Your rule reads these to decide per catalog/schema/table/column. For a fund schema you'd key off `resource.table.schemaName` against your fund-to-team mapping.

Q: How can OPA do column-level or row-level enforcement with Trino?
A: Beyond the main `opa.policy.uri`, Trino supports `opa.policy.column-masking-uri` (and a batch variant) and `opa.policy.row-filters-uri`, where OPA returns masks or filter expressions rather than a plain boolean. This lets policy hide specific columns (e.g. a sensitive fee field) or restrict rows (e.g. only this team's funds). It extends the same PDP/PEP pattern to finer-grained controls.

Q: What would a row-filter response from OPA conceptually provide to Trino?
A: It provides a filter expression that Trino applies so the user only sees rows they're permitted to — e.g. restricting a positions table to the user's own funds. The policy decides the constraint; Trino enforces it as an added predicate. This pushes a "visibility" rule down to row granularity without per-user views.

Q: Why is batching authorization requests important for catalogue-wide operations?
A: Operations like `SHOW TABLES` or filtering a large schema would otherwise trigger one OPA call per object, which is slow and chatty. The batched endpoint sends all candidates in one request and gets back the allowed indices, cutting round-trips dramatically. Configuring `opa.policy.batched-uri` is the standard fix when listing is sluggish.

Q: An auditor asks "who could read fund X's holdings as of last quarter?" — how does policy-as-code answer it?
A: You check out the policy bundle revision active at that date (traceable from decision logs or release history), read the Rego rule plus the `data` fund-to-team mapping it used, and combine with the IdP group membership of that period. Because all three are versioned, the answer is reconstructable and defensible. Per-tool manual grants, by contrast, leave no such reproducible record.

Q: How does policy-as-code support DORA-style operational resilience and audit expectations?
A: It produces tamper-evident, versioned authorization artifacts and decision logs, enabling you to demonstrate that access controls were defined, tested, reviewed, and consistently enforced — and to reconstruct historical decisions. That evidentiary chain aligns with regulatory expectations around governance and traceability. It converts "trust our admins" into "here is the reviewed change and its test".

Q: Why is mapping IdP groups to policy (rather than naming individual users) the better design?
A: Group-based rules let access change by group membership in the IdP, so onboarding/offboarding is one place and the policy stays stable, while user-by-user rules bloat and rot. It also dovetails with the Keycloak group-to-role work in 11.1.1. The policy says "this group may see fund X"; the IdP says "this person is in the group".

Q: What is the danger of an over-broad `permit` in Cedar or an over-broad `allow` in Rego?
A: It silently grants more access than intended — e.g. allowing a whole catalogue instead of one schema — and because positive grants are easy to write, the error may pass allow-tests while violating an unstated boundary. Cedar mitigates this with `forbid`-overrides and schema typing; Rego mitigates it with explicit deny tests. Either way, you must test the negative cases.

Q: In Cedar, what role does the optional `when`/`unless` clause play?
A: After the scope (principal/action/resource), `when { ... }` adds conditions that must hold for the policy to apply, and `unless { ... }` excludes cases; both are written in Cedar's typed expression language. For example a `permit` could apply `when { resource.fundStatus == "released" }`. These conditions are what the schema type-checks.

Q: How does Cedar's deny-by-default compare with Rego's deny-by-default?
A: Both deny when nothing grants access, but Cedar bakes it into the language (no matching `permit`, or any matching `forbid`, means deny), whereas in Rego you implement it via `default allow := false` plus the convention that undefined means deny. Cedar's is structural; Rego's is by design choice you must remember to make. Forgetting the `default` in Rego is a classic foot-gun.

Q: Where does OPA fit relative to Apache Ranger or Immuta, conceptually?
A: They occupy the same niche — externalized, centrally managed data authorization — but OPA/Rego is an open, general policy engine you integrate via plugins (like Trino's), whereas Ranger and Immuta are more turnkey, policy-management products. The lesson positions OPA as the FOSS, hands-on choice, with Immuta/Ranger/Cedar-AVP as the corresponding alternatives. The PDP/PEP separation idea is common to all.

Q: What is the single most important habit policy-as-code introduces compared to clicking grants?
A: Treating every authorization change as code: written, reviewed in a PR, proven by tests, versioned, and deployed as an immutable artifact. That habit yields consistency, auditability, and reversibility almost for free. It is the difference between "we think the rule is right" and "the test proves the rule is right, and here is the commit".

Q: Summarize the end-to-end loop this lesson asks you to demonstrate.
A: Express the fund-visibility rule in Rego, write allow and deny unit tests with `opa test`, enforce it for Trino via the OPA plugin pointed at `data.trino.allow`, then change the policy in git so CI tests and a new bundle revision flow out to OPA — and watch Trino's enforcement follow without a manual grant. The proof points are passing tests on a reviewed PR and Trino reflecting the latest bundle revision. That loop is what replaces three tickets to three admins.
