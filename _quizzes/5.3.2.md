# Phase 3 · 5.3.2 DataFrame transformation layer (PySpark / Polars / Ibis) — 150 self-test questions

Q: At the architect level, why is "SQL vs DataFrames, and which DataFrame runtime" framed as a standards decision rather than a personal preference?
A: Because every team inherits whatever the architect blesses, so the choice propagates across the whole estate and is expensive to reverse. Get it wrong and you end up with the same NAV logic re-implemented in three dialects, no single source of truth, and no clean answer when an auditor asks where a number was computed. A standard fixes which transformation types live where, so reviewers can enforce it consistently instead of each team improvising.

Q: What is the single biggest "pandas habit" to unlearn when moving to PySpark?
A: Row-by-row iteration — anything like `df.iterrows()`, `apply` over rows, or Python `for` loops over records. Spark is a distributed columnar engine, so row loops force data out of the JVM into Python one record at a time, which serializes everything and destroys parallelism. The PySpark idiom is to express the logic as column expressions that the engine can plan and run across partitions.

Q: What does "columnar thinking" mean in the context of PySpark transformations?
A: It means describing what each output column should be as a function of input columns, using built-in expressions, rather than describing how to walk the rows. You build up `DataFrame` operations like `withColumn`, `select`, `filter`, joins, and aggregations that the Catalyst optimizer can rearrange and the engine can vectorize. The mental model is "transform whole columns at once" instead of "loop and mutate", which keeps work inside the optimized engine.

Q: In PySpark, what should replace a pandas `groupby().apply()` pattern when you need a per-group running calculation?
A: A window specification — define a `Window` with `partitionBy` (and usually `orderBy`) and apply window functions like `row_number`, `lag`, `lead`, `sum`, or `rank` over it. Window specs let Spark compute per-partition ordered results natively without materializing each group into a Python object. `groupby-apply` style code typically drops you into a slow Python UDF path, which a window expression avoids entirely.

Q: Write the PySpark idiom for "running total of `amount` per `account`, ordered by `trade_date`".
A: Define `w = Window.partitionBy("account").orderBy("trade_date")` then `df.withColumn("running_total", F.sum("amount").over(w))`. The `partitionBy` scopes the cumulative sum to each account and `orderBy` fixes the running order. This stays as a native expression so Spark evaluates it across the cluster without any Python round-trip per row.

Q: Why does a Python row loop "melt the cluster" while a native column expression on the same data does not?
A: A Python row loop pulls every row across the JVM-to-Python boundary, serializes it, runs interpreted Python, and serializes it back, which serializes execution onto the driver or chokes executors and defeats parallelism. A native column expression is compiled by Catalyst, runs inside the JVM on whole partitions, and parallelizes naturally. The loop turns a distributed job into an effectively single-threaded one with massive serialization overhead.

Q: What is a `Window` spec in PySpark composed of, and which part is optional?
A: It is composed of an optional `partitionBy` (which groups rows), an optional `orderBy` (which orders rows within each partition), and an optional frame clause such as `rowsBetween` or `rangeBetween`. Ranking and offset functions like `row_number`, `lag`, and `lead` require an `orderBy` to be meaningful. Without `partitionBy` the window spans the entire DataFrame, which can force everything into one partition and is a common performance trap.

Q: What is a Polars `LazyFrame` and how does it differ from a Polars `DataFrame`?
A: A `LazyFrame` records the operations you request as a query plan instead of executing them immediately, whereas a `DataFrame` is eager and computes each step as it is called. The lazy plan is only run when you call `.collect()`, which lets the optimizer see the whole pipeline first. This deferred model is what enables predicate and projection pushdown and other whole-query optimizations.

Q: How do you create a Polars `LazyFrame`, and how do you finally execute it?
A: Create one with a scan function such as `pl.scan_parquet(...)` or `pl.scan_csv(...)`, or by calling `.lazy()` on an existing eager `DataFrame`. You build up the transformation with the usual expressions and then call `.collect()` to run the optimized plan and get back a materialized `DataFrame`. Using `scan_*` rather than `read_*` is what keeps the source itself lazy so the optimizer can push filters and column selection down to the file reader.

Q: What is the difference between `pl.read_parquet` and `pl.scan_parquet`?
A: `pl.read_parquet` eagerly loads the file into an in-memory `DataFrame` immediately, while `pl.scan_parquet` returns a `LazyFrame` that defers reading until `.collect()`. With `scan_parquet` the optimizer can push predicates and column projections into the read, so only the needed row groups and columns are pulled off disk. For large files this means you may read a fraction of the data instead of loading it all and then filtering.

Q: Name two query optimizations the Polars lazy optimizer performs and what each one buys you.
A: Predicate pushdown moves `filter` conditions as close to the data source as possible so rows are dropped early, and projection pushdown selects only the columns the query actually needs so unused columns are never read or carried. Together they cut both I/O and the volume of data flowing through the pipeline. The optimizer also does things like common-subexpression elimination and slice pushdown, all enabled because lazy execution sees the whole plan before running it.

Q: How do you inspect a Polars lazy query plan without running it, and why would you?
A: Call `.explain()` on the `LazyFrame` to print the optimized (and optionally the unoptimized) logical plan as text, or `.show_graph()` to render it visually. You do this to confirm the optimizer actually pushed your filters and projections down and to spot accidental full scans or missing pushdown before paying to execute. It is the Polars equivalent of reading a SQL `EXPLAIN` plan to validate your assumptions.

Q: What does Polars "streaming" execution give you, and when do you need it?
A: Streaming processes the query in batches rather than holding all intermediate data in RAM, so it can handle datasets larger than memory on a single node. You reach for it when a `collect` would otherwise blow the machine's memory but the workload still fits comfortably on one box's disk and cores. It is the mechanism that lets one fat node stand in for a small Spark cluster on moderately large data.

Q: Around what data size does single-node Polars typically become a credible replacement for Spark, per the lesson?
A: Below roughly 100GB, a single beefy node running Polars (especially lazy and streaming) often beats a Spark cluster because you avoid all distributed-execution overhead — shuffles, JVM startup, task scheduling, and serialization between nodes. Spark's value shows up when data genuinely exceeds what one node can hold or process in reasonable time. The architect's job is to know that crossover for their own data and not reach for a cluster reflexively.

Q: Why can a single-node tool like Polars beat a distributed engine on sub-100GB data?
A: Distributed engines pay fixed costs — network shuffles, serialization between executors, task scheduling, and JVM overhead — that only pay off once the data is too big for one machine. On sub-100GB data those costs dominate, while Polars runs everything in one process with a vectorized, multi-threaded, Arrow-based engine and no network. So the "small" tool wins precisely because it skips the machinery designed for "big".

Q: What is Ibis, in one sentence, and what problem does it solve?
A: Ibis is a portable Python dataframe library: one expression language that compiles to 20+ execution engines, so you write your transformation once and run it on DuckDB, Spark, BigQuery, Snowflake, Postgres, Polars, and more. It solves the problem of a multi-engine estate where the same logic would otherwise be rewritten per engine. You change the connection, not the analytical code, to switch backends.

Q: How does Ibis execute an expression — eagerly or deferred — and what happens under the hood for a SQL backend?
A: Ibis uses deferred execution: building an expression just constructs a logical plan, and nothing runs until you trigger execution (for example `.execute()` or `.to_pandas()`). For SQL backends Ibis compiles that expression into a SQL string (using SQLGlot) and sends it to the engine to run. So the same Ibis code becomes BigQuery SQL on BigQuery and DuckDB SQL on DuckDB, each idiomatic to its target.

Q: What is the default Ibis backend, and why is that a sensible default for local development?
A: The default backend is DuckDB, an in-process analytical database that needs no server and runs fast on a single machine. It lets you develop and test the same Ibis expressions locally that you will later run on a heavyweight cloud engine, "scaling up and out" without rewriting. DuckDB-as-default embodies Ibis's local-to-cloud portability story.

Q: In a multi-engine estate, what concretely does Ibis buy an architect?
A: It buys one canonical definition of each transformation that runs across every engine in the estate, so NAV or enrichment logic is written and reviewed once instead of forked into Spark, Snowflake, and Postgres dialects. That collapses maintenance, removes dialect drift, and makes the "where was this number computed" question answerable. It also reduces lock-in, since moving an engine is a connection change rather than a rewrite.

Q: How does Ibis claim to avoid the usual performance penalty of an abstraction layer?
A: Because Ibis does not execute the data itself — it compiles the expression to the backend's native language and the backend does the work, so you get performance "as good as the backend it is connected to". There is no Ibis runtime sitting in the data path adding overhead. The abstraction is at compile time, not at execution time.

Q: What library does Ibis use to turn expressions into SQL for SQL backends, and why does that detail matter?
A: It uses SQLGlot to transpile the Ibis expression into a backend-specific SQL string. It matters because it means Ibis can emit dialect-correct SQL for many engines from one expression tree, and you can even inspect the generated SQL to audit exactly what will run. That inspectability is valuable in a regulated shop where you must show what query produced a figure.

Q: List the three runtimes this lesson asks you to implement the same transformation in, plus the SQL path.
A: PySpark, Polars (lazy), and Ibis are the three DataFrame runtimes, and dbt SQL is the set-based SQL path you compare them against. The exercise is to implement one silver transformation in PySpark, Polars lazy, and dbt SQL and verify identical output on the same sample. Running the same logic three ways is what forces you to understand each engine's idioms and cost model.

Q: What is the "UDF cost model" and what are its three tiers from most to least expensive?
A: The cost model ranks how a custom transformation is executed: a plain Python UDF is most expensive, a pandas/Arrow (vectorized) UDF is cheaper, and a native engine expression is cheapest. The cost is driven by how much data must cross the JVM-to-Python boundary and be serialized, and how much work runs in interpreted Python versus the optimized engine. The architect's rule is to push logic into native expressions whenever possible.

Q: Why is serialization called "the hidden tax" of Python UDFs in Spark?
A: Because a Python UDF forces every input row to be serialized out of the JVM into a separate Python worker process, deserialized, processed in interpreted Python, then serialized back — and that round-trip is invisible in the code but dominates runtime. Native expressions never leave the JVM, so they pay none of it. The tax is "hidden" because the UDF looks like a one-line function call yet quietly moves all your data across a process boundary.

Q: How does a pandas (vectorized) UDF reduce the cost of a plain Python UDF?
A: A pandas UDF transfers data in batches as Arrow record batches and hands the function a whole `pandas.Series` or `DataFrame` at once, so it amortizes the serialization and runs vectorized NumPy/pandas code instead of per-row Python. This drastically cuts the boundary-crossing overhead compared with a scalar Python UDF that processes one value at a time. It is still slower than a native expression but far better than the row-at-a-time path.

Q: What are Arrow UDFs in PySpark and how do they improve on pandas UDFs?
A: Arrow UDFs operate directly on Arrow data without converting inputs into pandas or NumPy objects, preserving the columnar layout end to end and avoiding extra data copies. Per Databricks' benchmarks they run about 10% faster and use roughly 40% less memory than pandas UDFs, with better support for complex data types. They are a leaner replacement for pandas UDFs while still not matching a fully native expression.

Q: Why is a native Spark expression still faster than even an Arrow UDF?
A: Because a native expression executes entirely inside the optimized JVM engine and never crosses the JVM-to-Python boundary at all, whereas any UDF — Arrow included — still ships data to a Python process and back. Eliminating the boundary removes both the serialization cost and the loss of Catalyst's ability to optimize across the operation. So the ordering "native > Arrow UDF > pandas UDF > Python UDF" reflects how much each one avoids leaving the engine.

Q: An architect is enforcing a standard: which transformation types belong in dbt SQL versus DataFrame code?
A: Set-based work — joins, aggregations, filters, and declarative reshaping — belongs in dbt SQL because the warehouse optimizes it and it stays testable and lineage-tracked. Procedural row-level enrichment, complex iterative logic, and ML feature preparation belong in DataFrame code where an imperative API and richer libraries fit. The boundary is roughly "can it be expressed cleanly as a query?" — if yes, keep it in SQL; if it needs procedural logic, move it to DataFrames.

Q: Who enforces the SQL-vs-DataFrame layer ownership standard, and how in practice?
A: The data architect owns the standard and enforces it through code review, templates, and the project's contribution guidelines, so reviewers can reject a procedural pandas blob that should have been a SQL model and vice versa. Without an owner the boundary erodes and logic scatters across layers. Enforcement is social and tooling-based: a written standard, plus reviewers and CI checks that hold the line.

Q: How would you answer the challenge "why not do everything in SQL?" for a fund transformation layer?
A: SQL is ideal for set-based joins and aggregations and gives free lineage and testability in dbt, but it is a poor fit for procedural row-level enrichment, iterative logic, calling Python libraries, or ML feature prep, where DataFrame code is clearer and sometimes the only practical option. Forcing everything into SQL produces unreadable nested CTEs and brittle workarounds. The defensible standard keeps set-based logic in SQL and reserves DataFrames for genuinely procedural work, justified with timings.

Q: For a silver-layer EMT-style share-class enrichment of holdings, which engine and layer would you default to, and why?
A: If it is mostly a set-based join of holdings to share-class reference data, default to dbt SQL in the warehouse for testability and lineage; if it involves procedural per-position logic that does not map cleanly to a query, move it to DataFrame code. EMT enrichment that is essentially a lookup-and-attach is a textbook SQL join. The decision should be backed by your own timings on the 1M-row and 50M-row samples, not by reflex.

Q: How do you prove that the PySpark, Polars, and dbt SQL implementations of one transformation are equivalent?
A: Run all three on the same input sample and verify they produce identical output — same rows, same columns, same values — after sorting to a canonical order, since DataFrame engines do not guarantee row order. A practical check is to compare row counts, then a column-by-column equality or a hash of the sorted result. Any divergence usually reveals a subtle semantic difference such as null handling, join type, or numeric rounding that you must reconcile before trusting the standard.

Q: Why must you sort or canonicalize before comparing outputs of two DataFrame engines?
A: Because DataFrame engines and distributed shuffles do not guarantee a stable row order, so two correct results can differ only in ordering and fail a naive equality check. Sorting on a deterministic key (or comparing as sets/hashes) removes order from the comparison. Skipping this step produces false "mismatch" alarms that send you debugging logic that is actually correct.

Q: When timing the same transformation on a 50M-row dataset versus a 1M-row sample, what pattern usually emerges?
A: On the small 1M-row sample, single-node Polars and even local DuckDB typically win because Spark's fixed startup, scheduling, and shuffle costs dominate at that scale. On the 50M-row set the gap narrows and the winner depends on the operation's shuffle intensity and memory footprint, with Spark's distribution paying off as data grows. The point of measuring both is to locate your own crossover instead of assuming one engine is always best.

Q: A reviewer asks you to name the dataset size and shape where Polars beats Spark — what does a good answer contain?
A: A good answer cites your own measured numbers, for example "on our ~X-row, Y-GB holdings table on a single Z-core node, Polars lazy ran in N seconds versus Spark's M, and held that lead up to ~100GB". It names size, shape, hardware, and the operation, not a generic claim. The "Done when" bar is explicitly that you can quote your own timings, which forces the standard to be evidence-based.

Q: What is the most common cause of a Polars lazy query reading far more data than expected?
A: Predicate or projection pushdown not happening — usually because you used an eager `read_*` instead of `scan_*`, applied a filter after a `collect`, or used an operation the optimizer cannot push through. Inspecting `.explain()` shows whether the filter and column selection reached the scan node. Switching to `scan_parquet` and filtering before `collect` typically restores pushdown.

Q: A PySpark job is far slower than the equivalent SQL — first thing to check?
A: Check whether the job contains a Python UDF (or a row-wise `apply`) that forces a JVM-to-Python serialization round-trip, since that is the classic cause of a DataFrame job underperforming native SQL. Look at the query plan and the stages for a `BatchEvalPython`/UDF stage. If found, the fix is to replace the UDF with a native expression or, failing that, an Arrow/pandas UDF.

Q: In a Spark job profile, how would you detect that a Python UDF is the bottleneck?
A: In the Spark UI's SQL/DAG view you will see a Python evaluation stage (such as a `BatchEvalPython` or `ArrowEvalPython` node) and disproportionate time and data movement around it, with executors spending time in the Python worker rather than the JVM. High serialization/deserialization time and a stage that does not parallelize well are the tells. Removing the UDF and re-profiling, then comparing wall-clock and shuffle metrics, confirms it.

Q: What does the Spark plan node `ArrowEvalPython` versus `BatchEvalPython` tell you?
A: `BatchEvalPython` indicates a plain (scalar) Python UDF running row-at-a-time through pickle serialization, while `ArrowEvalPython` indicates a vectorized pandas/Arrow UDF that transfers data in Arrow batches. Seeing `ArrowEvalPython` means you are already on the cheaper vectorized path; seeing `BatchEvalPython` is a flag to vectorize or go native. Either node means data is still leaving the JVM, so a native expression would remove it entirely.

Q: How do you read a PySpark physical plan, and what are you looking for?
A: Call `df.explain()` (optionally `df.explain(True)` or `mode="formatted"`) to print the physical plan and look for `Exchange` nodes (shuffles), `*EvalPython` nodes (UDFs), the join strategy (broadcast vs sort-merge), and filter/scan pushdown. Excess shuffles and Python eval nodes are the usual performance smells. Reading the plan before optimizing tells you what is actually expensive instead of guessing.

Q: What is the difference between Spark's `withColumn` called in a long loop versus building expressions once?
A: Calling `withColumn` hundreds of times in a Python loop builds a deeply nested logical plan that can make analysis and optimization slow or even blow the plan size, even though each call is "native". The idiom is to construct the columns in a single `select` with a list of expressions, or chain a few `withColumns` (plural) calls. So "native expressions" still require building the plan efficiently, not just avoiding UDFs.

Q: Why does the lesson say NAV logic ending up "in three dialects" is the failure mode to avoid?
A: Because if the same NAV calculation is hand-written separately in Spark, Snowflake, and Postgres SQL, the three copies inevitably drift, produce subtly different numbers, and leave no authoritative definition. When an auditor or transfer agent disputes a figure, you cannot say which copy computed it or that they agree. A single owned definition — ideally one engine-portable expression or one dbt model — is the architectural fix.

Q: How does choosing Ibis differ from choosing Snowpark or the Databricks DataFrame API, in lock-in terms?
A: Ibis is engine-portable: the same expression compiles to many backends, so swapping engines is a connection change. Snowpark and the Databricks DataFrame API are the same concepts rebranded but bound to one vendor's platform, so logic written in them is tied to Snowflake or Databricks respectively. Knowing this trade-off lets the architect decide consciously between portability and a vendor's integrated tooling rather than backing into lock-in.

Q: What is Snowpark, and what should an architect "evaluate" about it before adopting it?
A: Snowpark is Snowflake's DataFrame-style API that runs transformation code inside Snowflake using familiar DataFrame and UDF concepts. The thing to evaluate is the lock-in trade-off: it is convenient and well-integrated, but the code is Snowflake-specific, so the architect weighs that against a portable approach like Ibis. The concepts are the same DataFrame ideas rebranded onto one platform.

Q: Why does running all three runtimes "via uv" matter for this exercise?
A: Because `uv` gives you reproducible, isolated environments so PySpark, Polars, and Ibis (with conflicting transitive dependencies) can each be installed and run cleanly without polluting a global interpreter. The study plan mandates `uv run` for all Python, so each engine's job runs in a pinned environment you can reproduce later. That reproducibility also makes your timing comparisons fair and re-runnable.

Q: What does it mean that a Polars expression is "lazy" but Polars also has an eager API?
A: It means Polars supports both: the eager `DataFrame` API runs each operation immediately for interactive convenience, while the lazy `LazyFrame` API records a plan and optimizes it before `.collect()`. The lazy path is where the query optimizer (predicate/projection pushdown, etc.) lives, so production transformations should generally use it. The eager API is fine for quick exploration where whole-query optimization does not matter.

Q: For a 50M-row holdings enrichment that is a simple join plus a few derived columns, what would you predict about Polars vs Spark and why?
A: If the data fits one node's memory or streams comfortably, predict Polars wins on wall-clock because it avoids shuffle and JVM overhead for what is essentially a hash join and column math. Spark would only pull ahead if the join needs a large shuffle that exceeds one node's capacity. You confirm the prediction with measured timings rather than asserting it, per the lesson's "your own numbers" bar.

Q: What is "projection pushdown" and give a concrete example of its effect on a Parquet scan.
A: Projection pushdown is the optimizer selecting only the columns the query needs and pushing that selection into the source read. For a wide Parquet holdings file with 80 columns where your query uses 5, projection pushdown means only those 5 columns' chunks are read off disk. That can cut I/O by an order of magnitude versus reading the whole file and selecting columns afterward.

Q: What is "predicate pushdown" and how does it interact with Parquet row-group statistics?
A: Predicate pushdown moves filter conditions toward the data source so rows are eliminated as early as possible. With Parquet it can use per-row-group min/max statistics to skip entire row groups that cannot match the filter, so a `WHERE valuation_date = '2026-06-12'` may read only the relevant groups. The result is far less data decoded and carried through the pipeline.

Q: Why does the architect care that dbt SQL gives "free" lineage and testability, in the layer-ownership decision?
A: Because dbt models declare dependencies (lineage) and support schema/data tests as first-class features, so set-based logic kept in SQL is automatically documented, graphed, and testable. DataFrame code does not get that for free — you must add lineage and tests yourself. That asymmetry is a strong reason to keep joins and aggregations in dbt SQL and reserve DataFrames for logic SQL cannot express cleanly.

Q: A pandas UDF returns wrong results only on large inputs but is fine on small samples — likely cause?
A: A common cause is that a grouped or scalar pandas UDF receives data in Arrow batches and the function assumed it saw a whole group when it only saw a batch, so per-batch state (like cumulative computations or `.iloc[0]` assumptions) breaks once data spans multiple batches. It passes on small samples that fit one batch and fails at scale. The fix is to use the correct UDF type (for example a grouped-map or grouped-aggregate UDF) so the engine guarantees whole-group inputs.

Q: How would you implement an EMT/EPT-style enrichment portably so it runs the same on DuckDB locally and Spark in production?
A: Express it as an Ibis expression — the join, derived share-class fields, and filters — built once, then point Ibis at DuckDB for local development and at the Spark (or Snowflake/BigQuery) backend in production by changing only the connection. Ibis compiles the one expression to each engine's native execution, so the EMT logic has a single definition. That is exactly the portability and "computed where?" auditability a regulated fund platform wants.

Q: Why is "where was this number computed" an auditability question that the transformation-layer standard must answer?
A: In a regulated fund-administration context an auditor or regulator can demand the provenance of a NAV, holding value, or EMT field, and you must point to one definitive piece of code in one layer. If the logic is duplicated across dialects or scattered between SQL and DataFrame code with no standard, you cannot give a clean answer. Owning the layer boundary and keeping single definitions makes provenance traceable.

Q: What is the risk of doing procedural NAV-component enrichment as nested SQL CTEs instead of DataFrame code?
A: The SQL becomes deeply nested, hard to read, hard to test in isolation, and brittle to change, because SQL is awkward for genuinely procedural or iterative logic. Reviewers cannot reason about it and small edits cause regressions. The standard's value is recognizing when logic has crossed from set-based into procedural and belongs in DataFrame code instead.

Q: How do `scan_csv` and `scan_parquet` differ in how much the optimizer can help?
A: Both produce a `LazyFrame`, but Parquet is columnar and carries row-group statistics, so the optimizer can do projection pushdown and skip row groups via predicate pushdown far more effectively than with row-oriented CSV. With CSV, projection still helps but the format forces reading and parsing more of the file. So `scan_parquet` generally yields much bigger pushdown wins than `scan_csv`.

Q: Why might `.collect()` in Polars exhaust memory even though the source file is on disk and large?
A: Because the default (non-streaming) collect materializes all intermediate results and the final frame in RAM, so a wide join or aggregation can exceed memory even if the raw file is read lazily. Enabling streaming execution processes the plan in batches and keeps the footprint bounded. Checking the plan for an exploding intermediate (a large join or unbounded group) tells you whether streaming or a redesign is needed.

Q: In Ibis, how do you actually trigger execution and get results back into Python?
A: You build the expression lazily and then call an execution method such as `.execute()` (which typically returns a pandas DataFrame) or backend-aware methods like `.to_pandas()`, `.to_polars()`, or `.to_pyarrow()`. Until you call one of these, nothing runs and no data moves — Ibis has only constructed and (for SQL backends) compiled a query. This deferral is what lets the same expression target any backend.

Q: Why is inspecting Ibis's compiled SQL useful in a regulated environment?
A: Because you can call Ibis's SQL-printing facility (for SQL backends) to see exactly the query that will run against the warehouse, giving you an auditable, reviewable artifact for how a figure is computed. That transparency supports change control and lets reviewers reason about performance and correctness at the SQL level. It turns the abstraction into something you can sign off on rather than a black box.

Q: What is the trap with using `collect_list`/`collect_set` aggregations in Spark on large groups?
A: They gather all values of a group into a single in-memory array on one task, so a skewed or huge group can cause memory pressure or OOM on that executor. They are native expressions but not free — the cost moved from serialization to memory and skew. The mitigation is to bound group sizes, salt skewed keys, or reframe the logic to avoid materializing giant arrays.

Q: How does Spark's choice between broadcast join and sort-merge join show up in a plan, and why care?
A: A small table joined to a large one ideally appears as a `BroadcastHashJoin`, which ships the small side to every executor and avoids shuffling the big side; a `SortMergeJoin` (with `Exchange` nodes) shuffles and sorts both sides and is far more expensive. For an enrichment that joins a big holdings table to a small share-class reference, you want the broadcast plan. Reading `explain()` confirms which you got and whether the broadcast threshold needs adjusting.

Q: Why is a small reference-data join (like share-class lookup) an ideal broadcast-join candidate?
A: Because the reference table is tiny relative to the holdings fact table, so broadcasting it to every executor lets Spark do a local hash join with no shuffle of the large side. That avoids the expensive sort-and-shuffle of a sort-merge join. Ensuring the reference side is recognized as broadcastable (it fits under the broadcast threshold, or you hint it) is a standard optimization for enrichment workloads.

Q: What does "no row loops" not forbid — i.e., when is iterating in Python actually fine?
A: Iterating over a small, fixed set of columns or a handful of partitions/config items in Python to build expressions is fine, because you are constructing the plan, not processing data rows. The rule forbids looping over data rows, which moves the dataset through Python. Looping to generate a list of column expressions that Spark then executes natively is idiomatic and cheap.

Q: How would you decide between Polars and Spark for the gold-layer NAV aggregation step specifically?
A: Measure it: if the aggregated input fits one node (or streams) and the operation is a manageable group-by, Polars likely wins on latency and cost; if the data is genuinely huge or the shuffle exceeds one node, Spark's distribution pays off. NAV aggregation is often a moderate set-based group-by, which favors single-node tools at typical fund sizes. The decision must cite your own timings to survive review.

Q: What is the difference between a scalar Python UDF and a grouped-map pandas UDF in Spark?
A: A scalar Python UDF maps one input value to one output value row-by-row through pickle serialization, while a grouped-map pandas UDF receives an entire group as a `pandas.DataFrame` (via `applyInPandas`/grouped map) and returns a DataFrame. The grouped-map form is vectorized and whole-group, suited to per-group transformations like normalization within a share class. Both still cross the JVM-Python boundary, so prefer native expressions when the logic allows.

Q: Why might two engines disagree on the result of an average due to null handling?
A: Engines differ in whether they include or skip nulls in aggregations and how they treat division and integer-vs-float types, so an `AVG` over a column with nulls can differ if one engine counts nulls in the denominator and another does not. SQL `AVG` ignores nulls, but a hand-rolled DataFrame mean might not. When verifying three implementations are identical, null and type semantics are the first place mismatches appear.

Q: What is the role of Arrow as a common substrate across Polars, Ibis, and pandas/Arrow UDFs?
A: Apache Arrow is the columnar in-memory format that Polars uses natively, that Ibis can hand data back in (`to_pyarrow`), and that pandas/Arrow UDFs use to transfer batches between the JVM and Python with minimal copying. Sharing Arrow means data can move between these tools cheaply and in a columnar layout. It is the reason vectorized UDFs and zero-copy interchange are fast.

Q: A dbt SQL model and a PySpark version of the same logic produce different row counts — first thing to check?
A: Check join semantics and null handling: SQL inner/left joins, equality on nullable keys (SQL treats `NULL = NULL` as not-equal), and implicit deduplication can differ from how the DataFrame code was written. A `DISTINCT` in one and not the other, or a left vs inner join, is a frequent culprit. Reconciling the exact join type and key-null treatment usually closes the count gap.

Q: Why does the lesson tie the half-page standards note to your own timing numbers?
A: Because a standard that says "use SQL for set-based work" is only credible if backed by evidence from your own data, so the note must cite where each engine won on the 1M- and 50M-row samples. Numbers make the standard defensible against "why not all SQL?" and against reviewers' preferences. Opinion without measurement does not survive architectural scrutiny.

Q: What is the difference between `LazyFrame.collect()` and `LazyFrame.fetch()` (or a head/limit-based preview) in Polars?
A: `collect()` runs the full optimized plan over all data, while taking a small preview (historically via a fetch-style call or by adding a `.limit`/`.head` before collecting) runs the plan against a small slice for quick iteration. Previewing lets you validate logic cheaply before paying for the full dataset. The key is that a limited run still goes through the optimizer, so it reflects the real plan, just on less data.

Q: How does Ibis let you "scale up and out" from a laptop to the cloud without rewriting?
A: You develop against the in-process DuckDB backend on your laptop, validate the expression, then point the same Ibis code at a cloud engine like BigQuery, Snowflake, or Spark by changing the connection. "Up" means a bigger single engine, "out" means a distributed one, and neither requires touching the transformation logic. This is the concrete payoff of decoupling the frontend API from the backend execution.

Q: What is the danger of writing transformation logic as a Spark UDF that calls a Python library like `requests` or a model?
A: Besides the serialization cost, such a UDF runs arbitrary Python per row/batch with external side effects or heavy objects, which is slow, hard to make idempotent, and can hammer external services from every executor in parallel. Logic with external calls usually belongs outside the hot transformation path, or batched carefully. If it must be a UDF, an Arrow/pandas form that batches calls is far less damaging than a scalar per-row UDF.

Q: Why is ML feature preparation often placed in DataFrame code rather than dbt SQL in the layer standard?
A: Because feature engineering frequently needs procedural logic, numerical libraries, custom transforms, and tight integration with Python ML tooling that SQL expresses poorly or not at all. DataFrame code (PySpark/Polars) gives the imperative control and library access that feature prep needs. So the standard typically assigns set-based joins/aggregations to SQL and ML feature prep to DataFrames.

Q: What does it mean that Spark's DataFrame API is lazy, and how is that similar to Polars and Ibis?
A: Spark transformations like `select`, `filter`, and joins are lazy: they build a logical plan that Catalyst optimizes, and nothing executes until an action like `count`, `collect`, or `write` is called. This mirrors Polars `LazyFrame` (run on `.collect()`) and Ibis (run on `.execute()`). All three defer execution so a whole-query optimizer can rearrange the plan, which is why each rewards "describe the result" over "step through rows".

Q: Why can a single misplaced `.collect()` (Polars) or action (Spark) inside a loop wreck performance?
A: Each `collect`/action triggers a full execution of the plan, so calling it inside a loop re-runs the whole pipeline repeatedly instead of once, multiplying I/O and compute. It also breaks the optimizer's ability to fuse work across iterations. The fix is to build one lazy plan over the whole job and materialize a single time at the end.

Q: How would you express "rank holdings by market value within each fund" portably in Ibis, and what does it compile to?
A: You build a window expression in Ibis partitioned by fund and ordered by market value, applying a rank function, then execute it on your chosen backend. For a SQL backend Ibis compiles it to a `RANK() OVER (PARTITION BY fund ORDER BY market_value DESC)` window query in that engine's dialect. The same expression runs on DuckDB locally and on Spark or Snowflake in production unchanged.

Q: What is the relationship between Polars and Ibis — can Ibis use Polars as a backend?
A: Yes: Polars is one of Ibis's 20+ backends, so you can write Ibis expressions and have them executed by Polars, just as you could target DuckDB or Spark. This lets you keep one expression language while choosing single-node Polars execution for moderate data. It illustrates Ibis's role as a frontend over many engines rather than a competitor to any single one.

Q: When comparing managed Databricks or Snowpark against the FOSS stack, what does the lesson want you to "know"?
A: That they are "the same concepts rebranded" — DataFrame APIs, lazy execution, UDFs, optimizers — wrapped in a vendor platform, and that adopting them is a build-vs-buy decision with a lock-in cost. You should be able to map each managed feature to its FOSS equivalent and articulate the trade-off. The skill is recognizing the underlying concepts so vendor branding does not obscure the engineering.

Q: A Polars `.explain()` shows your `filter` above the `scan` rather than pushed into it — what likely went wrong?
A: Likely the filter references a column produced later in the pipeline (for example after a join or a computed column) or uses a function the optimizer cannot push through, so it cannot be moved to the scan. Restructuring so the filter uses only source columns, and applying it before transformations that block pushdown, usually lets the optimizer push it down. Re-running `.explain()` confirms the filter moved.

Q: Why is "idempotent and reproducible" relevant even within the transformation layer, not just orchestration?
A: Because a transformation should produce the same output from the same input regardless of how many times it runs, which is essential when reruns happen after a NAV correction or a backfill. UDFs with side effects, nondeterministic ordering, or hidden state break that and produce inconsistent figures. Keeping transformations as deterministic expressions over inputs preserves reproducibility, which a regulated audit trail requires.

Q: What is the cost difference between `df.count()` and `df.cache().count()` in Spark for an iterative comparison job?
A: A bare `count()` re-executes the entire plan from the source each time, while caching materializes the DataFrame once so subsequent actions reuse it from memory/disk. In a timing or comparison job that touches the same frame repeatedly, caching avoids recomputing upstream work on every action. The gotcha is that caching consumes memory and must be sized and unpersisted deliberately, so it is not free.

Q: How do you keep Polars from silently upcasting or changing dtypes when matching a Spark/dbt result?
A: Set schemas explicitly on scan (for example pass `schema`/`dtypes` to `scan_csv`) and cast columns deliberately, because automatic type inference can pick a different numeric type than the other engines and cause value or precision mismatches. Aligning dtypes up front is part of proving the three implementations are identical. Type drift is a frequent, subtle source of "off by a rounding" differences in NAV figures.

Q: Why might a transformation that is correct on Polars eager fail or differ on Polars lazy?
A: Some operations behave differently or are only optimizable in the lazy engine, and the lazy optimizer may reorder or fuse steps in ways that surface a logic dependency you implicitly relied on (such as row order before a non-deterministic step). Most logic is identical, but ordering assumptions can break. Inspecting `.explain()` and pinning any required ordering resolves the discrepancy.

Q: What is the architect's two-minute pitch for adopting Ibis across a multi-engine fund platform?
A: One expression language, written and reviewed once, compiles to every engine in the estate, so NAV and EMT logic has a single auditable definition that runs on DuckDB locally and Spark/Snowflake in production by swapping a connection. It removes dialect duplication, reduces lock-in, and lets you show exactly what query produced a number. The cost is a thinner ecosystem and one more abstraction to learn, which you weigh against the duplication it eliminates.

Q: Why is a wide `select` with a list of expressions preferred over chained `withColumn` calls in production PySpark?
A: Because each `withColumn` adds a layer to the logical plan, and hundreds of them produce a huge nested plan that slows the analyzer/optimizer and can hit limits, whereas a single `select` (or `withColumns` plural) builds the same columns in one flat step. The result is the same data but a far cheaper plan to optimize. It is a "native but still inefficient" trap that an architect should call out in review.

Q: How do you choose UDF type when you genuinely must run custom Python in Spark?
A: Prefer the most vectorized option available: an Arrow UDF or pandas (Series/grouped) UDF over a scalar Python UDF, because batched Arrow transfer slashes the serialization tax versus row-at-a-time pickling. Pick the grouped form when the logic needs whole groups, and the scalar pandas/Arrow form for elementwise work. Reserve the plain Python UDF for cases where no vectorized form fits, knowing it is the most expensive path.

Q: What does "the same concepts rebranded" imply about transferring skills from Snowpark to PySpark or vice versa?
A: It implies the core ideas — lazy DataFrames, lazy evaluation, window functions, UDFs and their cost model, and pushdown optimization — carry across, so an engineer fluent in one can learn the other quickly. The differences are mostly API surface, dialect, and platform integration, not fundamentals. Recognizing this lets the architect hire and train on transferable concepts rather than a single vendor's API.

Q: When does Spark remain the right choice despite single-node tools being faster on small data?
A: When the data genuinely exceeds one node's memory and disk, when shuffles are too large for a single machine, or when you need its mature ecosystem for streaming, complex joins at scale, or an existing cluster investment. The crossover is roughly where data passes what one fat node can handle (around and above ~100GB for many workloads). The architect picks Spark deliberately for that regime, not reflexively for all "big data".

Q: How would you prove to a reviewer that replacing a UDF with a native expression actually helped?
A: Profile both versions on the same input — capture wall-clock, the Spark UI stage breakdown, and whether the `*EvalPython` node disappeared — and show the native version removed the Python evaluation stage and reduced time and shuffle. A controlled before/after on identical data and hardware is the evidence. The "Done when" bar is exactly being able to explain the cost difference and how you detected it in a profile.

Q: Why is it dangerous to let business logic live implicitly inside a UDF rather than as a reviewable expression or SQL model?
A: Because a UDF is opaque to the optimizer and often to reviewers and lineage tooling, so the business rule it encodes (say a fee or NAV adjustment) is hidden from documentation, tests, and provenance tracking. That undermines auditability and makes the rule easy to duplicate or diverge. Expressing the logic as a native expression, a dbt model, or a portable Ibis expression keeps it visible and governed.

Q: What is the practical meaning of Ibis giving performance "as good as the backend it is connected to"?
A: It means Ibis itself does not process data, so your job runs at the speed of DuckDB, Spark, BigQuery, or whichever engine executes the compiled query — there is no Ibis-layer slowdown. If a backend is fast, your Ibis job is fast; if it is slow, Ibis cannot rescue it. The abstraction is a compile-time convenience, not a runtime bottleneck.

Q: How does the layer-ownership standard reduce the chance of NAV logic drift across teams?
A: By declaring where each transformation type must live — set-based logic in dbt SQL, procedural enrichment in DataFrame code — it stops teams from each putting NAV logic wherever is convenient, which is what causes three-dialect drift. A single owned location per logic type means one place to change and one place to audit. Enforcement through review keeps the standard real rather than aspirational.

Q: A Spark window function over a partition with no `orderBy` gives unexpected cumulative results — why?
A: Cumulative and offset functions like `sum().over(w)`, `lag`, and `row_number` depend on a defined order, and without `orderBy` the frame is ambiguous (often the whole partition), so the "running" result is not what you intended and may be nondeterministic. Adding an explicit `orderBy` to the window spec fixes it. This is a frequent bug when porting an ordered `groupby-apply` to a window.

Q: Why is reading the query plan a core architect skill across all three engines?
A: Because Polars `.explain()`, Spark `.explain()`, and Ibis's compiled SQL all reveal whether the optimizer did what you expect — pushdown, join strategy, absence of UDF stages — so you validate performance from the plan rather than by trial. Plan literacy is how you catch a missing broadcast join or an un-pushed filter before it costs cluster time. It is the common diagnostic muscle for the whole transformation layer.

Q: What is the failure mode of "pandas habits" specifically around indexing and joins in Spark?
A: Pandas relies on an index and supports operations like `merge` on the index and positional `.iloc`, none of which translate to Spark, where there is no row index and joins are explicit on key columns across partitions. Carrying those habits leads to attempts at positional access or index joins that are inefficient or impossible. The Spark idiom is explicit key-based joins and column expressions, with no reliance on a row index.

Q: How would you structure the half-page standards note so it survives "why not do everything in SQL?"
A: State the rule (set-based logic in dbt SQL; procedural enrichment and ML feature prep in DataFrame code), justify it with the free lineage/testability of SQL versus SQL's poor fit for procedural logic, and back the boundary with your own 1M/50M-row timings showing where each engine wins. Include a concrete fund example such as EMT enrichment landing in SQL because it is a join. The note must show evidence and a clear owner, not just assertion.

Q: Why might a Polars streaming query produce a different memory profile than a non-streaming one for the same logic?
A: Streaming executes the plan in bounded batches, keeping peak memory roughly constant regardless of data size, whereas the default engine may materialize large intermediates and spike memory. For a big group-by or join, streaming trades some operator support and possibly speed for a flat, predictable footprint. Choosing it is about fitting larger-than-memory work onto one node, not about going faster per se.

Q: What is the concrete risk of running the same NAV transformation in PySpark and Snowpark without a portable definition?
A: You now maintain two implementations of the same business rule in two APIs that can diverge with any edit, and you must keep them provably equivalent forever, doubling test burden and audit surface. A drift between them means two different "official" NAVs depending on which platform ran. A portable definition (one Ibis expression or one dbt model) collapses this to a single source of truth.

Q: How do you give a reviewer confidence that the Polars, PySpark, and dbt SQL results are byte-for-byte equivalent on the sample?
A: Run all three, write outputs in a canonical sorted column order, and compare with an exact equality or a stable hash of the sorted result, then show the comparison passes and the row counts match. Document any deliberate type or null-handling alignment you applied to make them agree. Equivalence proven by code, not by eyeballing a few rows, is what stands up in review and audit.

Q: How do you express a windowed (per-group) calculation in Polars without a `groupby-apply`, and what is the idiom called?
A: Use an expression with `.over("group_col")`, for example `pl.col("amount").sum().over("fund")` to get a per-fund total broadcast back onto each row, or a ranking expression over the group for ordered logic. This is the window-expression idiom and it stays inside Polars's optimized, vectorized engine rather than dropping into per-group Python. It is the Polars analogue of a PySpark `Window` spec and avoids the slow apply path entirely.

Q: Summarize the architect's decision rule for the transformation layer in one sentence.
A: Keep set-based joins and aggregations in dbt SQL for lineage and testability, push custom logic into native engine expressions before reaching for any UDF (and choose Arrow/pandas UDFs over scalar Python when a UDF is unavoidable), pick single-node Polars under ~100GB and Spark above it based on your own timings, and use a portable layer like Ibis when one definition must run across many engines. The whole rule exists so every NAV number has one auditable place it was computed. That is the standard the architect owns and every team inherits.

Q: What Spark config controls when a join is automatically broadcast, and what is its default and disable value?
A: `spark.sql.autoBroadcastJoinThreshold` controls it, defaulting to `10MB`, meaning any side estimated below that size is broadcast for a `BroadcastHashJoin` instead of a shuffled `SortMergeJoin`. Setting it to `-1` disables auto-broadcast entirely, which is sometimes done to diagnose plans or avoid driver OOM when Spark mis-estimates a side's size. For a share-class reference table that is genuinely small but exceeds the threshold, raise the config or use an explicit hint rather than rely on the estimate.

Q: How do you force a broadcast join in PySpark when Spark's size estimate is wrong, and what is the risk?
A: Wrap the small side in `F.broadcast(df_ref)` before the join, or in SQL use the `/*+ BROADCAST(ref) */` hint, which broadcasts regardless of `autoBroadcastJoinThreshold`. The risk is that if the "small" side is actually large, broadcasting it ships a big object to every executor and to the driver first, causing memory pressure or OOM. So only hint a side you are certain is small, such as a curated share-class lookup, not an unbounded result of an upstream filter.

Q: What is `spark.sql.shuffle.partitions`, its default, and why does the default often hurt at fund-data scale?
A: It sets the number of partitions produced by a shuffle (joins, aggregations), defaulting to `200`, so every wide operation fans out to 200 tasks regardless of data size. On a small 1M-row sample that creates 200 tiny tasks with scheduling overhead and many near-empty files; on a huge set 200 may be too few and spill. Adaptive Query Execution mitigates this by coalescing post-shuffle partitions at runtime, but the static default is a classic reason a tiny job is needlessly slow.

Q: What three things does Spark Adaptive Query Execution (AQE) do at runtime, and since which version is it on by default?
A: AQE uses runtime statistics to coalesce post-shuffle partitions, convert a `SortMergeJoin` into a `BroadcastHashJoin` when a side turns out small, and split skewed partitions in a skew join. It is enabled by default since Spark `3.2.0` via `spark.sql.adaptive.enabled`. For fund data with skewed keys (one giant fund, many small ones), AQE's skew-join handling is what keeps one straggler task from dominating wall-clock without manual salting.

Q: A Spark aggregation has one task running for an hour while the rest finished in seconds — what is happening and how do you confirm it?
A: That is data skew: one partition holds a hugely over-represented key (for example a single mega-fund's positions), so its task processes far more rows than the others. Confirm it in the Spark UI by looking at the stage's task summary — the max task's input/shuffle-read bytes and duration will dwarf the median (75th percentile). Fixes are enabling AQE skew-join handling, salting the hot key, or pre-aggregating, after you have proven the skew from the task metrics.

Q: What is key salting and when would you apply it to a NAV-component join?
A: Salting appends a random bucket suffix to a skewed join/group key (and replicates the small side across all buckets) so one hot key is spread across many partitions instead of one overloaded task. You apply it when a single key — say one fund holding a disproportionate share of positions — causes a straggler that AQE cannot fully resolve. The cost is added complexity and a follow-up de-salt step, so reserve it for proven, persistent skew rather than applying it pre-emptively.

Q: In Polars, how do you write a larger-than-memory result to disk, and why is `sink_parquet` different from `collect`?
A: Call `lf.sink_parquet("out.parquet")` on a `LazyFrame`, which executes the plan in streaming mode and writes batches straight to the file so the full result never has to fit in RAM. `collect()` by contrast materializes the entire result in memory before you can write it, which can OOM on a wide aggregation. `sink_parquet` only runs under the streaming engine, so it is the right tool when both inputs and outputs exceed memory on your single node.

Q: What does Polars `collect_batches()` (or a batched collect) give you that a plain `collect()` does not?
A: It yields a Python generator of sub-`DataFrame` chunks instead of one final materialized frame, so you can process and then discard each chunk to keep peak memory bounded. This is useful when even the streamed output must be consumed incrementally — for example feeding NAV rows to a downstream writer or API in batches. A plain `collect()` forces the whole result into memory at once, which a batched approach avoids.

Q: How do you select the streaming engine in newer Polars when calling `collect`, and what is the trade-off?
A: Pass the engine selector to collect (historically `streaming=True`, in newer Polars `engine="streaming"`) so the plan runs in bounded batches rather than the in-memory engine. The trade-off is that not every operator is fully supported in streaming and behaviour/perf can differ, so you verify the result matches the in-memory run on a sample. The streaming engine is slated to become the Polars default, but until then you opt in for larger-than-memory work.

Q: How do you print the exact SQL Ibis will run, and how do you target a different dialect for review?
A: Call `ibis.to_sql(expr)` or `expr.to_sql()` to get the compiled SQL string, and pass `dialect="duckdb"` (or `"spark"`, `"snowflake"`, etc.) to render it for a specific engine even if you are connected elsewhere. Ibis compiles via SQLGlot (since Ibis `9.0`), so the emitted SQL is dialect-correct for the chosen target. In a regulated shop this gives you a concrete, reviewable artifact showing precisely which query computes a figure.

Q: Why might `ibis.to_sql(expr)` and the SQL actually executed against the backend differ in surface detail, and does it matter?
A: `to_sql` produces a formatted, dialect-rendered string for readability, while the backend may receive a functionally equivalent query with different whitespace, quoting, or parameterization, since the compile target is the engine's parser not a human. It matters only that the semantics match, which they do because both come from the same SQLGlot compilation of one expression tree. For audit you cite the `to_sql` output as the canonical, human-readable definition of the computation.

Q: How do you list which backends an Ibis installation can actually use, and why check?
A: Inspect the installed backend entry points (for example `ibis.backends` / the backend registry) or simply attempt `ibis.duckdb`, `ibis.polars`, `ibis.pyspark` and see which import, because each backend is an optional extra you install separately (such as `ibis-framework[duckdb,polars]`). You check so a "portable" expression does not fail at deploy time because the production backend's extra was never installed. A missing extra surfaces as an `ImportError`/missing-backend error at connection time, not at expression-build time.

Q: An Ibis expression works on DuckDB but raises an `OperationNotDefinedError` on the Spark backend — what does that tell you?
A: It tells you the operation you used is not implemented for that backend's compiler, since portability is bounded by what each backend supports — not every function compiles everywhere. The fix is to express the logic with operations the target backend supports, or to confirm support before standardizing on the expression across engines. This is why "write once, run anywhere" in Ibis is qualified: you must validate the expression on every backend in your estate, not just the local default.

Q: How do you bring existing raw SQL into an Ibis pipeline when a piece of logic is easier in SQL?
A: Use the backend's SQL escape hatch — for example `con.sql("SELECT ...")` to get a table expression from a raw query, or `table.sql("...")` to apply a SQL snippet — then continue composing Ibis expressions on top. This lets you keep most logic portable while dropping to native SQL for the rare operation Ibis cannot express. The caveat is that the raw-SQL portion is dialect-specific and breaks portability for that step, so document it as an engine-bound exception.

Q: How do you make a Polars `scan_csv` read the same dtypes as your Spark/dbt result, with the exact argument?
A: Pass an explicit schema via `scan_csv(path, schema_overrides={...})` (or `schema=`/`dtypes=` depending on Polars version) so columns are read as the precise types the other engines use, rather than relying on inference. Inference can pick `Int64` where Spark used a decimal, or `Float64` with different precision, producing off-by-rounding NAV mismatches. Pinning the schema at scan time is part of proving the three implementations are byte-for-byte equivalent.

Q: Why can two engines disagree on a sum or average of monetary amounts even with identical inputs, and how do you prevent it?
A: Floating-point columns accumulate in nondeterministic order under parallelism and round differently, and integer-vs-float-vs-decimal typing differs across engines, so a fund's total can differ in the last digits. Prevent it by computing money in a fixed-precision `Decimal` type with an explicit scale in every engine and avoiding `Float64` for amounts. For NAV figures this matters because a sub-cent discrepancy across two "official" implementations is an audit finding, not a rounding curiosity.

Q: How would you express a decimal-typed share-class fee calculation identically in PySpark and Polars to avoid float drift?
A: In PySpark cast amounts with `F.col("amount").cast("decimal(18,4)")` and in Polars cast to `pl.Decimal(precision=18, scale=4)`, performing the arithmetic in decimal in both so rounding is defined and identical. Doing fee math in `double`/`Float64` instead lets each engine round differently and breaks the equivalence check. The explicit precision and scale are the contract that makes the two results comparable to the cent.

Q: In PySpark, how do you replace a `withColumn`-in-a-loop pattern that builds 100 derived columns, with the exact API?
A: Build a list of expressions and apply them in one `df.select("*", *exprs)`, or use `df.withColumns({...})` (plural, dict form) to add them in a single flat plan node. Looping `withColumn` 100 times nests 100 plan layers and slows the analyzer, even though each call is native. The single-`select`/`withColumns` form produces the same columns with a far cheaper plan to optimize.

Q: What is the difference between `df.withColumn` and `df.withColumns` in modern PySpark, and why prefer the latter for many columns?
A: `withColumn` adds or replaces a single column per call, so adding many means many chained calls and a deeply nested logical plan; `withColumns` takes a dict of name-to-expression and adds them all in one operation. Preferring `withColumns` keeps the plan flat, speeds analysis/optimization, and avoids the plan-size blow-up that hundreds of `withColumn` calls can cause. It is the idiomatic fix for the "native but inefficient" loop trap.

Q: A Spark job logs `Stack overflow` or extremely slow query analysis with no UDFs — what design smell should you suspect?
A: Suspect an over-nested logical plan, typically from calling `withColumn` (or chaining transformations) hundreds of times in a Python loop, which builds a recursively deep plan the analyzer must traverse. Even though no data leaves the JVM, the planning cost explodes before execution starts. Collapsing the columns into a single `select`/`withColumns` flattens the plan and removes the analysis bottleneck.

Q: How do you cache a DataFrame for an iterative comparison and properly release it, with the exact calls?
A: Call `df.cache()` (or `df.persist(StorageLevel...)`) then trigger an action like `df.count()` to materialize it, so subsequent actions reuse the stored result instead of recomputing the plan. Release it with `df.unpersist()` once done, because cached data holds executor memory and silently degrades later stages if left around. In a timing harness that touches the same frame repeatedly, caching is the difference between recomputing upstream work every action and computing it once.

Q: Why might `df.cache()` appear to "do nothing" until much later in a Spark job?
A: Because `cache()` is lazy — it only marks the DataFrame for caching, and nothing is actually stored until the first action materializes it, so the cost and benefit both appear at that action, not at the `cache()` call. Engineers expecting an immediate persist are confused when the first count is slow (it populates the cache) and only later actions are fast. Forcing materialization with an explicit `count()` right after `cache()` makes the timing predictable.

Q: How do you check in the Spark UI whether your cache is actually being used, and what is the tell that it is not?
A: Open the Storage tab to see cached RDDs/DataFrames with their fraction-cached and memory size, and check that the relevant query stage reads from an `InMemoryTableScan` rather than re-reading the source. The tell that caching failed is a `0%` cached fraction (it spilled or never materialized) or a plan that still scans the original files. If the cache evicted under memory pressure, you will see recomputation and should size storage or use disk persistence.

Q: How do you read a Spark physical plan in the most legible form, with the exact argument?
A: Call `df.explain(mode="formatted")` for a structured, node-by-node physical plan that lists operators and their details separately, which is far easier to scan than the default tree, and use `df.explain(True)` to also see parsed/analyzed/optimized logical plans. You are hunting for `Exchange` (shuffles), `*EvalPython` (UDFs), the join type, and pushed filters. The formatted mode is the practical default when validating that the optimizer did what you expect before spending cluster time.

Q: What does an `Exchange hashpartitioning` node in a Spark plan signify, and when is it a smell?
A: It marks a shuffle that repartitions data by a hash of given keys, which joins and aggregations need to co-locate matching keys. It is a smell when you see it for an operation you expected to broadcast (a small-side join that should have been `BroadcastExchange`/`BroadcastHashJoin`) or when there are more exchanges than the logic requires, indicating missed broadcast or repeated repartitioning. Each exchange is network and disk cost, so minimizing unnecessary ones is a core tuning goal.

Q: How would you verify Polars actually pushed a `valuation_date` filter into a Parquet scan, with the exact method and what to look for?
A: Call `lf.explain()` on the lazy plan and read it bottom-up: a pushed filter shows the predicate attached at or inside the `Parquet SCAN` node (often shown as a selection/`SELECTION` on the scan), not as a separate `FILTER` above it. If the filter appears above the scan, pushdown failed and the scan reads everything before filtering. Confirming the predicate sits at the scan is how you prove only the relevant row groups are read.

Q: In a Polars plan, what is the difference between `PROJECT */N COLUMNS` versus `PROJECT k/N COLUMNS` at the scan, and why care?
A: `PROJECT k/N` means projection pushdown succeeded and only `k` of `N` columns will be read from the file, whereas `PROJECT */N` (all columns) means no pruning happened and the full width is read. You care because reading 5 of 80 columns off a wide holdings Parquet is an order-of-magnitude less I/O than reading all 80. Seeing all columns selected when your query needs few is a signal to restructure so the optimizer can prune.

Q: How do you express "year-to-date NAV change per fund" as a Polars window without a `groupby-apply`, with the exact expression?
A: Use an over-expression such as `(pl.col("nav") - pl.col("nav").first()).over("fund")` after ordering by date, or a cumulative form like `pl.col("flows").cum_sum().over("fund")`, so the per-fund calculation runs vectorized inside the engine. The `.over("fund")` scopes the computation to each fund and broadcasts it back per row without dropping into Python. This is the Polars analogue of a Spark `Window.partitionBy("fund")` and avoids the slow apply path.

Q: In PySpark, what is the difference between `F.sum("x").over(w)` with `Window.partitionBy(...).orderBy(...)` and the same without `orderBy`, for a running total?
A: With `orderBy`, the default frame is `rangeBetween(unboundedPreceding, currentRow)`, giving a true running cumulative sum up to each row; without `orderBy`, the frame defaults to the whole partition, so every row gets the partition's grand total, not a running figure. Porting an ordered pandas `groupby().cumsum()` and forgetting `orderBy` silently yields totals instead of running totals. Always specify `orderBy` for cumulative/offset functions.

Q: When do you need an explicit `rowsBetween` versus `rangeBetween` frame in a Spark window, with a concrete fund example?
A: Use `rowsBetween` to count a fixed number of physical rows (for example a 3-row trailing average of the last three trades regardless of dates) and `rangeBetween` to span a value range of the ordering column (for example all trades within the prior 7 days by `trade_date`). They differ when the ordering column has ties or gaps: `rangeBetween` includes all rows with equal ordering values, `rowsBetween` does not. Choosing the wrong one changes which trades fall into a trailing window.

Q: How do you compute the most recent share-class reference row per `isin` as-of a date in PySpark, idiomatically?
A: Use a window `Window.partitionBy("isin").orderBy(F.col("effective_date").desc())` with `row_number() == 1` to pick the latest effective row per ISIN, optionally filtering `effective_date <= valuation_date` first for an as-of join. This avoids a self-join and a `groupby-apply`, staying as native window logic. It is the standard SCD-style "current/as-of record" pattern for slowly changing reference data like fund or share-class attributes.

Q: Why is `F.lag`/`F.lead` preferable to a self-join for "previous valuation's NAV per fund"?
A: `F.lag("nav", 1).over(Window.partitionBy("fund").orderBy("valuation_date"))` computes the prior row's value within the ordered partition in a single pass, whereas a self-join on `date = date - 1` shuffles the table against itself and breaks on non-contiguous dates (weekends, holidays). The window offset is cheaper and handles ordering correctly without assuming consecutive calendar days. It is the idiomatic way to compute day-over-day NAV deltas.

Q: A grouped-map pandas UDF (`applyInPandas`) returns the wrong schema at runtime — what is the likely cause and fix?
A: `applyInPandas` requires you to declare the output schema explicitly, and a mismatch between the declared `StructType` and the columns/dtypes your function actually returns raises a runtime error or silently coerces values. The fix is to make the returned `pandas.DataFrame`'s columns and types exactly match the declared schema, including column order. This is a frequent gotcha because the schema is passed separately from the function and the two drift during edits.

Q: What is the difference between `applyInPandas` and `mapInPandas` in PySpark, and when use each?
A: `applyInPandas` operates per group after a `groupBy`, handing your function one group as a whole `pandas.DataFrame` (good for per-share-class normalization), while `mapInPandas` operates on arbitrary batches of the whole DataFrame without grouping (good for elementwise/batch transforms like enrichment that does not need group context). Use `applyInPandas` when correctness depends on seeing a complete group, and `mapInPandas` for streaming-style batch processing. Both are Arrow-vectorized and cross the JVM-Python boundary, so prefer native expressions when possible.

Q: Why can a scalar pandas UDF silently give wrong results if it relies on row position within a batch?
A: A scalar pandas UDF receives an arbitrary Arrow batch (a `pandas.Series`), not the whole column or group, so any logic using `.iloc[0]`, cumulative state, or an assumed ordering breaks once data spans multiple batches. It passes on small samples that fit one batch and fails at scale. The fix is to use elementwise logic only in a scalar UDF, or switch to a grouped UDF that guarantees whole-group input.

Q: How do you register and call a native Spark SQL function instead of writing a UDF for, say, an ISIN check-digit validation?
A: Express the logic with built-in functions (`F.substring`, `F.regexp_extract`, arithmetic, `F.when`) composed into a column expression, or in Spark SQL use the rich function library, so the check-digit math stays in the JVM. Only if no combination of built-ins can express it do you reach for a UDF, and then a vectorized one. Keeping ISIN/LEI validation as native expressions means it benefits from Catalyst and avoids the per-row serialization tax.

Q: What is the cost difference between `F.regexp_extract` as a native expression and a Python `re` UDF doing the same extraction?
A: `F.regexp_extract` runs inside the JVM across whole partitions with no boundary crossing, while a Python `re`-based UDF serializes every value out to a Python worker, runs interpreted regex per row, and serializes back. For extracting a country code from an ISIN over millions of rows the native version is dramatically faster and parallelizes cleanly. The rule is to exhaust the built-in `regexp_*`/`F` functions before considering a regex UDF.

Q: How do you detect at the plan level that a join silently fell back to a Cartesian/cross product in Spark?
A: `df.explain()` shows a `CartesianProduct` or `BroadcastNestedLoopJoin` node instead of a hash/sort-merge join, which happens when the join condition is non-equi or the keys did not match types, so every row pairs with every row. This is catastrophic on large tables and usually a bug (wrong key, missing condition). Spotting the nested-loop/cartesian node in the plan tells you to fix the join predicate before the job ever finishes.

Q: dbt SQL and PySpark give different row counts on a join — how do null join keys explain it, with the exact semantics?
A: SQL equality treats `NULL = NULL` as unknown (not true), so rows with null keys do not match and an inner join drops them, whereas a DataFrame author might use `eqNullSafe` (`<=>`) which treats two nulls as equal and keeps them. If one implementation uses null-safe equality and the other plain equality, counts diverge exactly on the null-keyed rows. Aligning both to the same null semantics (usually filtering or null-safe by design) closes the gap.

Q: What is Spark's `<=>` / `eqNullSafe` operator and when does it matter for fund reference joins?
A: `<=>` (`Column.eqNullSafe`) is a null-safe equality that returns true when both sides are null, unlike `=` which returns null/unknown for null comparisons. It matters when joining on keys that legitimately contain nulls — for example an optional sub-fund identifier — and you want null-to-null rows to match rather than vanish. Choosing between `=` and `<=>` deliberately is part of making a DataFrame join match (or intentionally differ from) the SQL semantics.

Q: How do you guarantee a deterministic canonical order before comparing two engines' outputs, and why is `orderBy` alone sometimes insufficient?
A: Sort on a key that is unique per row (or the full set of columns) so ties cannot reorder rows nondeterministically — `orderBy` on a non-unique key leaves tied rows in arbitrary relative order, which still breaks a positional comparison. Better still, compare as multisets: hash each sorted row and compare the sorted list of hashes, or use a framework like `chispa`'s `assert_df_equality`. Relying on a non-unique sort is a common reason "equal" results still fail an equality check.

Q: What does `chispa` (or `pyspark-test`) give you over a hand-rolled DataFrame equality check?
A: It provides `assert_df_equality`-style helpers that compare schemas and row contents with options to ignore row order, ignore nullability, and apply float tolerance, raising a readable diff on mismatch instead of a bare boolean. This is far more reliable than eyeballing or a naive `==` that is order- and type-sensitive. For proving PySpark and dbt outputs match, a tolerance-aware, order-insensitive assertion is what stands up in review.

Q: How would you compare a Polars result and a Spark result for equality given they live in different processes?
A: Materialize both to a common neutral format — write each to Parquet or convert to Arrow/pandas — load into one process, sort on a unique key, align column order and dtypes, then assert equality with a float tolerance. Going through Arrow/Parquet avoids re-deriving values and makes the comparison a pure data check. The deliberate dtype and tolerance alignment is what prevents false mismatches from precision or type differences.

Q: When porting pandas code to Polars, what is the key indexing habit that does not translate and what replaces it?
A: Pandas relies on a row `Index` (label-based `.loc`, positional `.iloc`, index-aligned arithmetic), but Polars has no index — rows are positional and operations are column-expression based. Replace index-aligned joins with explicit `join(on=...)`, replace `.loc` filtering with `filter(pl.col(...))`, and replace index arithmetic with explicit key joins. Carrying the index mental model into Polars leads to confusion and inefficient row-oriented code.

Q: How do you do an as-of (time-based) join of trades to the latest FX rate in Polars, with the exact method?
A: Use `join_asof`, for example `trades.sort("ts").join_asof(rates.sort("ts"), on="ts", by="currency", strategy="backward")`, which for each trade picks the most recent rate at or before its timestamp, per currency. Both frames must be sorted on the as-of key. This natively expresses "value each trade at the prevailing rate" without a range self-join, and is the Polars idiom for point-in-time enrichment of valuations.

Q: Does Polars support an as-of join and does Spark have a built-in equivalent — what is the architectural implication?
A: Polars has native `join_asof`, but core Spark has no built-in as-of join (you emulate it with a window or a range join, or use an extension), which is a concrete capability difference. The implication is that a transformation expressed trivially in Polars may need more code and a shuffle-heavy workaround in Spark, affecting both correctness risk and performance. The architect must know these per-engine capability gaps before declaring one standard runtime for point-in-time logic.

Q: How does Ibis handle a window/as-of style ranking portably, and what should you verify per backend?
A: You build an Ibis window with `ibis.window(group_by=..., order_by=...)` and apply `rank()`/`lag()`/`first()`, which compiles to each backend's `OVER (...)` clause. You should verify the generated SQL via `ibis.to_sql(expr, dialect=...)` on each target and confirm the operation is supported, because window-function coverage and exact tie/null semantics can vary by backend. Validating per-engine output on a sample is what justifies trusting one Ibis definition across the estate.

Q: When the same Ibis expression yields different results on two backends, what is the disciplined way to find the cause?
A: Compile the expression to SQL on each backend with `ibis.to_sql(expr, dialect=...)`, diff the two queries, and run each against the same sample to isolate whether the divergence is in null handling, default window frame, type coercion, or ordering. Because Ibis exposes the compiled SQL, you can pinpoint the semantic difference rather than guessing. The fix is then to constrain the expression (explicit casts, explicit frames, explicit null handling) so all backends agree, which you re-verify on the sample.

Q: What is the difference between `repartition` and `coalesce` in Spark, and which do you use to shrink output file count after a filter?
A: `repartition(n)` does a full shuffle to produce exactly `n` evenly-sized partitions (and can increase or decrease the count), while `coalesce(n)` only merges existing partitions without a shuffle and can only decrease the count, producing possibly uneven partitions. To reduce many tiny output files after a heavy filter you usually want `coalesce(n)` because it avoids a shuffle; use `repartition` when you also need to rebalance skew or repartition by a key. Picking `repartition` where `coalesce` suffices adds a needless expensive shuffle.

Q: How do you write a multi-aggregate group-by in Polars idiomatically, and why is it preferable to a pandas-style apply?
A: Use `df.group_by("fund").agg([pl.col("market_value").sum().alias("aum"), pl.col("isin").n_unique().alias("holdings")])`, passing a list of expressions so all aggregates compute in one vectorized pass over each group. This stays inside Polars's optimized engine, unlike a pandas `groupby().apply(func)` that materializes each group into a Python object and runs interpreted code. The expression-list form is also what lets the lazy optimizer fuse and parallelize the aggregation across the whole fund table.
