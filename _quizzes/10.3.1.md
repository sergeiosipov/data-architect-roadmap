# Phase 6 · 10.3.1 Data lineage (OpenLineage standard) — 150 self-test questions

Q: What is data lineage, and why does a fund-administration architect treat it as an impact-analysis weapon rather than just documentation?
A: Data lineage is the recorded provenance of data — which jobs read which datasets and produce which outputs, edge by edge through the pipeline. It is a weapon because it lets you answer "what breaks if I change `orders.amount`?" by walking the downstream graph instead of guessing, and "where did this NAV figure come from?" by walking upstream. Without it, an auditor's "prove this number's provenance" becomes a multi-day archaeology dig across logs and code.

Q: What does BCBS-239 require that makes lineage non-optional in a regulated fund context?
A: BCBS-239 demands data aggregation and risk-reporting principles including accuracy, completeness, and traceability — every regulatory number must be reconstructable back to its sources. Lineage is the mechanism that makes traceability machine-verifiable rather than tribal knowledge, so you can demonstrate to a supervisor exactly how a reported figure was derived. In an asset-management shop this is what lets you defend a NAV or a holdings count when challenged.

Q: What is OpenLineage in one sentence, and what problem does standardising it solve?
A: OpenLineage is an open standard and specification for collecting lineage metadata, defining a common event model that many tools emit and many backends consume. Standardising it solves the N×M integration problem: instead of bespoke connectors between every producer and every catalog, each tool speaks one wire format. That means a Dagster job, a Spark job, and a dbt run can all feed the same lineage graph.

Q: Name the three core entities of the OpenLineage object model.
A: The three core entities are runs, jobs, and datasets. A job is a process that consumes or produces datasets, a run is one time-bound execution of that job, and a dataset is an abstract representation of data identified by namespace and name. Facets attach extensible metadata to any of these.

Q: How does OpenLineage define a "job", and why is it the design-time anchor?
A: A job is a discrete unit of work — a process that consumes or produces datasets, such as a dbt model build or a Spark transform. It is the design-time anchor because its identity (namespace plus name) is stable across executions, so the lineage graph aggregates all runs of the same job under one node. Choosing job granularity well is therefore the main lever for graph readability.

Q: How does a "run" differ from a "job" in OpenLineage?
A: A job is the abstract process; a run is one specific occurrence of that job in time, carrying a unique run identifier. The same job can have thousands of runs, but lineage edges are anchored to the job so the graph stays compact. Runtime facts like start time, end time, and failure messages belong to the run, not the job.

Q: What identifies a dataset in OpenLineage, and why is the namespace part essential?
A: A dataset is identified by the pair (namespace, name). The namespace disambiguates where the data physically lives — for example a Postgres source versus an Iceberg table versus an S3 path — so two datasets named `orders` in different systems are not collapsed into one node. Consistent namespacing is what lets lineage stitch correctly across the NAV path from Postgres through Iceberg to the EMT output.

Q: What is a facet in OpenLineage, and what makes the model "extensible"?
A: A facet is a self-contained piece of metadata that attaches to a run, job, or dataset, identified by a `_producer` and `_schemaURL`. The model is extensible because you can define custom facets without changing the core spec — the spine of runs/jobs/datasets stays fixed while facets carry everything else. This is how new metadata (data quality, schema, column lineage) is added without breaking existing consumers.

Q: Which run states (eventType values) does OpenLineage define?
A: The defined run states are `START`, `RUNNING`, `COMPLETE`, `ABORT`, `FAIL`, and `OTHER`. A run normally begins with `START` and ends in one terminal state — `COMPLETE`, `ABORT`, or `FAIL`. `OTHER` can be sent at any time, even before `START`, to carry supplementary metadata outside the normal progression.

Q: What are the three terminal run states, and what does each signify?
A: The terminal states are `COMPLETE` (the run finished successfully), `ABORT` (the run was cancelled or aborted before finishing), and `FAIL` (the run errored out). Once any terminal state is emitted, no further events for that `runId` should be sent. Distinguishing `ABORT` from `FAIL` matters for ops dashboards because a cancelled run is not the same incident as a crashed one.

Q: What is the purpose of the `RUNNING` eventType?
A: `RUNNING` is an intermediate heartbeat-style event sent after `START` and before a terminal state, used to report progress or attach facets discovered mid-execution. It is optional — many short jobs only emit `START` and `COMPLETE`. It is useful for long Spark jobs where you want incremental metadata before the run ends.

Q: When would you emit an `OTHER` event, and what makes it special?
A: `OTHER` carries metadata that does not fit the normal lifecycle, and uniquely it may be sent at any point, including before `START`. You would use it to attach supplementary facets to a run without asserting a lifecycle transition. Because it is not a terminal state, it does not close the run.

Q: Are OpenLineage events for a single run cumulative or replacements?
A: They are cumulative (accumulative) — facets emitted across multiple events for the same `runId` build up and enrich the run's metadata progressively. This means a backend like Marquez merges the `START` and `COMPLETE` payloads rather than discarding the earlier one. It also means you can emit cheap early facets at `START` and expensive computed ones (like `outputStatistics`) at `COMPLETE`.

Q: What three event types does the spec define beyond the run-cycle, and what is each for?
A: The spec defines `RunEvent` (runtime job-run state updates with inputs/outputs), `JobEvent` (design-time job metadata like source location and declared inputs/outputs, independent of any run), and `DatasetEvent` (design-time dataset metadata like schema or ownership, independent of any run). `RunEvent` is the workhorse; `JobEvent` and `DatasetEvent` let you register static metadata without executing anything.

Q: When would you emit a `DatasetEvent` rather than carrying dataset metadata inside a `RunEvent`?
A: You emit a `DatasetEvent` to publish design-time facts about a dataset — its schema, ownership, or documentation — that are not tied to a particular execution. It is useful when a dataset's metadata changes independently of any job run, or when you want to register a source's schema before any pipeline touches it. A `RunEvent` instead reports what a specific run actually read and wrote.

Q: What HTTP endpoint do OpenLineage HTTP emitters POST events to on a Marquez backend?
A: They POST to `/api/v1/lineage` on the Marquez API, for example `curl -X POST http://localhost:5000/api/v1/lineage`. The request body is a single OpenLineage event in JSON. Marquez parses it, upserts the run/job/dataset nodes, and stores the edges in Postgres.

Q: Which environment variable tells an OpenLineage client where to send events?
A: `OPENLINEAGE_URL` points the client at the consuming service, for example `http://localhost:5000` for a local Marquez. The client appends the `/api/v1/lineage` path itself. If it is unset, the integration has nowhere to emit and silently produces no lineage.

Q: What does the `OPENLINEAGE_NAMESPACE` environment variable control?
A: `OPENLINEAGE_NAMESPACE` sets the default job namespace used when an integration emits events, grouping related jobs under one logical area. If unset, a default namespace is used. Setting it deliberately — for example `nav-pipeline` — keeps the Marquez UI organised instead of dumping everything into `default`.

Q: What is `OPENLINEAGE_API_KEY` for?
A: `OPENLINEAGE_API_KEY` supplies a bearer token when the event consumer requires authentication, sent as an HTTP `Authorization` header. A bare local Marquez does not need it, but a hosted or shared backend often does. Omitting it against a protected endpoint yields 401 responses and no lineage lands.

Q: What is an emitter (integration) in the OpenLineage ecosystem?
A: An emitter is the code embedded in or wrapped around a data tool that observes execution and produces OpenLineage events automatically. Examples are the Dagster, Spark, and dbt integrations. The value is that lineage is captured as a side effect of running the pipeline rather than hand-maintained.

Q: How does the dbt OpenLineage integration capture lineage, and what is the wrapper called?
A: It is a wrapper script, `dbt-ol`, that you substitute for `dbt` — for example `dbt-ol run` instead of `dbt run`. After dbt finishes, the wrapper reads dbt's JSON artifacts and translates them into OpenLineage events sent to `OPENLINEAGE_URL`. Because it runs post-hoc, it captures lineage from a completed dbt invocation rather than streaming it live.

Q: Which dbt artifacts does `dbt-ol` read to build lineage events?
A: It reads dbt's run artifacts — chiefly `manifest.json` (the compiled DAG of models, sources, and their column metadata) and `run_results.json` (what actually ran and its status), plus the catalog where available. From `manifest.json` it derives model-to-model dependencies and column information. This is why column-level lineage from dbt depends on dbt having parsed the SQL into the manifest.

Q: Why does `dbt-ol` emit events only after the dbt run completes, and what is the practical consequence?
A: Because it is a post-run wrapper that parses dbt's artifacts once dbt has finished, it cannot stream per-model `START`/`COMPLETE` events as they happen the way a deeply instrumented tool would. The practical consequence is that if a dbt run is killed mid-flight, the wrapper may not emit, so you can lose lineage for crashed runs. For live per-step lineage you lean on an orchestrator integration instead.

Q: What is the current maintenance status of the OpenLineage Dagster integration, and what version ceiling matters?
A: The legacy OpenLineage Dagster integration is no longer actively maintained and recent Dagster releases introduce breaking changes, with supported Dagster capped around 1.6.9. So you should not assume the old `openlineage-dagster` package works on a current Dagster; verify the integration path against your Dagster version. The modern approach is Dagster's own asset/lineage emission rather than the deprecated plugin.

Q: How does the Spark OpenLineage integration differ in mechanism from the dbt one?
A: The Spark integration instruments the Spark execution itself via a listener on the SparkSession, observing the logical and physical query plans as jobs run. Because it reads the actual execution plan, it can derive lineage for transformations that have no static SQL file, including DataFrame API code. dbt by contrast parses SQL text into a manifest, so the two see different things.

Q: Why can Spark instrumentation produce lineage that dbt SQL parsing cannot?
A: Spark sees the real query plan at runtime, so it captures lineage for programmatic DataFrame transformations, UDFs, and joins expressed in Scala or Python that never appear as parseable SQL. dbt only knows what it can parse from model SQL into `manifest.json`. So a Spark job assembling the Iceberg layer can yield lineage where a pure-code transform would be invisible to a SQL parser.

Q: What is Marquez, and what role does it play relative to OpenLineage?
A: Marquez is the reference backend for OpenLineage — it collects, aggregates, stores, and visualises lineage events. OpenLineage defines the event format; Marquez is one concrete service that ingests those events and renders the graph. It is the default choice for a local hands-on lineage stack.

Q: What datastore does Marquez use to persist lineage, and why does that matter?
A: Marquez stores lineage metadata in PostgreSQL. It matters because the graph — runs, jobs, datasets, and column-level edges — is queryable relationally and survives restarts, and you can inspect or back it up like any Postgres database. It also means scaling and retention are Postgres operational concerns.

Q: How do you typically start Marquez locally for hands-on work?
A: You check out the Marquez source and run the `./docker/up.sh` script, which brings up the API, the web UI, and Postgres via Docker Compose. Flags include `--build` to build images from source and `--seed` to load sample lineage so the UI is not empty. This is the fastest way to get a working backend to point emitters at.

Q: Which ports does Marquez expose for its API and admin interface?
A: The Marquez HTTP API listens on port `5000` and the admin interface on port `5001`. The admin port exposes operational endpoints like `/healthcheck` and `/metrics`. The web UI runs separately (commonly on `3000`) and talks to the API.

Q: On macOS, why might the default Marquez port `5000` fail, and what is the fix?
A: On recent macOS, port `5000` is reserved by the system (AirPlay Receiver), so Marquez cannot bind it. The fix is to start it on another port, for example `./docker/up.sh --api-port 9000`, and point `OPENLINEAGE_URL` at that port. On native Ubuntu — the learner's platform — `5000` is normally free, so this is only cross-platform awareness.

Q: What does the `--seed` flag on Marquez's `up.sh` do, and when is it useful?
A: `--seed` starts Marquez preloaded with sample lineage metadata so the UI shows an example graph immediately. It is useful the first time you stand it up, to confirm the UI renders and to explore the navigation before you have emitted any of your own events. For real work you omit it and emit your own pipeline's lineage.

Q: After emitting events, what three node types should you confirm appear in the Marquez UI?
A: You should confirm that runs, jobs, and datasets all appear — jobs as the process nodes, datasets as the data nodes between them, and runs listed under each job with their states. Seeing all three means the emitter, the transport, and the backend are wired correctly. Missing datasets usually means the emitter did not report inputs/outputs.

Q: What is the difference between table-level and column-level lineage?
A: Table-level (dataset-level) lineage records that dataset A fed dataset B; column-level lineage records which specific input columns produced each output column. Table-level answers "does this report depend on `orders`?", while column-level answers "which source columns feed `report.total_amount`?". Column-level is strictly more detailed and harder to capture.

Q: Which OpenLineage facet carries column-level lineage, and what does it express?
A: The `columnLineage` dataset facet carries it, mapping each output field to the set of input fields it was derived from, often with transformation type information. It answers "which root input columns are used to construct column X?". It attaches to the output dataset because that is where the per-column derivation is known.

Q: Why is SQL parsing the hard part of column-level lineage?
A: Because different warehouses have dialect extensions, and constructs like `SELECT *` require resolving the input schema to know which columns are actually projected. Subqueries, CTEs, window functions, and expression-level derivations all have to be parsed and tracked to map outputs back to inputs. A parser that handles one dialect's edge cases may produce wrong or missing lineage on another.

Q: What library does OpenLineage use for SQL-based column lineage, and what do alternative tools use?
A: OpenLineage uses `openlineage-sql` (a Rust-based SQL parser) to extract column lineage from SQL. DataHub instead uses `sqlglot` for static SQL parsing, and OpenMetadata has used `sqllineage`. They differ in dialect coverage and accuracy, which is why the same query can yield different column lineage depending on the tool.

Q: Why does column-level lineage from dbt depend on what dbt can parse, and where does it fall short?
A: dbt-derived column lineage comes from `manifest.json`, which reflects what dbt compiled and resolved from the model SQL. It falls short where the SQL is too dynamic, uses unsupported dialect features, or where `SELECT *` cannot be expanded because the upstream schema is not resolvable at compile time. In those cases you may get table-level edges but no column mapping.

Q: In the NAV pipeline (Postgres → Debezium → Iceberg → Spark → dbt → EMT), where would you expect column-level lineage and where only table-level?
A: You expect column-level lineage in the dbt layer (model SQL parsed into the manifest) and in Spark where the plan exposes column derivations. You are more likely to get only table-level lineage across the Debezium CDC hop and any code that is opaque to the parser. Stitching these requires consistent dataset namespaces so the dbt and Spark sub-graphs join into one end-to-end picture.

Q: How does lineage flow from OpenLineage events into a catalog like OpenMetadata?
A: The catalog ingests OpenLineage events (or reads them from a backend) and materialises the lineage graph inside its own metadata model, linking it to the dataset entities it already tracks. In OpenMetadata the lineage tab on a table then shows upstream and downstream relationships derived from those events. This is what surfaces lineage next to ownership, glossary terms, and data-quality results in one place.

Q: Why feed lineage into the catalog you already stood up rather than only viewing it in Marquez?
A: Because the catalog unifies lineage with ownership, descriptions, classifications, and data-quality results, so a single asset page tells the whole story to an auditor or analyst. Marquez is excellent for the raw lineage graph, but it does not hold your glossary or DQ verdicts. Centralising in the catalog also means consumers have one place to look.

Q: How does DataHub consume lineage compared with OpenMetadata?
A: DataHub can ingest OpenLineage events and also extract lineage itself using `sqlglot` and dbt's `manifest.json`. Both catalogs surface a lineage graph, but they parse SQL with different engines, so coverage and column-level accuracy differ. The architect's job is to know which source of lineage each catalog trusts and where the gaps are.

Q: What naming discipline keeps a lineage graph readable, and why?
A: Use consistent, meaningful dataset namespaces and names that reflect the physical system and logical identity, and keep job names stable across runs. This matters because nodes are keyed on (namespace, name) — inconsistent naming fragments one logical dataset into several nodes and breaks the end-to-end path. Disciplined naming is what lets the Postgres source and the Iceberg table line up into a single traceable chain.

Q: Why does job granularity matter for lineage graph readability?
A: Too-fine granularity (a job per trivial step) produces a sprawling graph that is hard to read; too-coarse (one job for the whole pipeline) hides the internal edges an auditor needs. The right granularity makes each job a meaningful transformation between datasets. Designing this deliberately is an architectural decision, not an accident of how the tool happens to emit.

Q: What is a "black-box step" in lineage, and why is it a problem?
A: A black-box step is a stage — often a third-party SaaS or an opaque process — that emits no OpenLineage events, so the graph has a gap where data enters and exits without recorded edges. It is a problem because the chain of provenance breaks there, and an auditor's trace stops mid-pipeline. You cannot claim end-to-end traceability if a hidden hop launders the lineage.

Q: How do you handle a black-box step that emits nothing?
A: You annotate the gap explicitly — manually emit a placeholder job/dataset or document the step's declared inputs and outputs so the graph bridges it with a known-approximate edge. The point is to make the gap visible and labelled rather than silent, so reviewers know lineage there is asserted, not observed. The Done-when criterion for this lesson is exactly to identify and annotate such a step.

Q: An auditor asks you to prove an EMT output value's provenance back to source. How does lineage let you answer in minutes?
A: You open the EMT output dataset in the lineage graph and walk upstream edge by edge — dbt model, Spark transform, Iceberg table, Debezium-captured source, original Postgres table — reading each job that transformed it. Where column-level lineage exists, you show exactly which source columns fed the value. This turns a multi-day archaeology dig into a guided walk through the graph.

Q: How do you answer "which reports break if `orders.amount` changes?" purely from the lineage graph?
A: You locate the `orders` dataset (and the `amount` column if column-level lineage is present), then traverse downstream through every job and dataset that consumes it, collecting the terminal outputs — including any EMT report. Column-level lineage narrows the blast radius to consumers of `amount` specifically, avoiding false positives from columns you did not touch. The answer comes from the graph alone, which is the Done-when test.

Q: Why is column-level lineage materially better than table-level for the `orders.amount` impact question?
A: Table-level lineage flags every downstream of `orders` as potentially affected, over-reporting because most consumers may never touch `amount`. Column-level lineage restricts the impact set to the columns actually derived from `amount`, giving a precise blast radius. That precision is the difference between "re-test forty reports" and "re-test the three that use this column".

Q: What is the EMT file in the fund context, and why does its lineage matter to a regulator?
A: The EMT (European MiFID Template) is a standardised data file conveying product and cost information for investment products under MiFID II/PRIIPs distribution. Its lineage matters because regulators and distributors must trust that the figures in it trace back to authoritative fund data. Demonstrable lineage from EMT fields back to source tables is what lets you defend the file's accuracy under scrutiny.

Q: How does an ISIN flowing through the NAV pipeline benefit from lineage?
A: An ISIN is the security identifier threading through holdings, prices, and outputs, so lineage lets you trace any ISIN-keyed figure back to where that instrument's data originated. If an ISIN was mapped or enriched at some step, lineage shows the job that did it. That is essential when reconciling a discrepancy on one instrument across the chain.

Q: How does lineage support answering a transfer-agency query about a subscription figure's origin?
A: Transfer-agency data (subscriptions, redemptions, registers) feeds downstream aggregates, and lineage lets you trace a published figure back to the TA source dataset and the jobs that transformed it. When a figure is questioned, you walk upstream to the TA system of record rather than reconstructing the path from memory. This shortens dispute resolution and gives a defensible audit trail.

Q: Why does DORA make demonstrable lineage tooling relevant to a fund administrator?
A: DORA (the EU Digital Operational Resilience Act) raises expectations around operational resilience and the integrity and traceability of critical data flows in financial entities. Lineage evidence supports demonstrating that critical pipelines are understood, their dependencies mapped, and the impact of changes assessable. It is part of showing control over your data estate, not an isolated nicety.

Q: A Marquez UI shows jobs but no datasets between them. What is the first thing to check?
A: Check whether the emitter is actually reporting `inputs` and `outputs` in its `RunEvent`s — datasets appear only when events declare them. For dbt that means the manifest resolved sources and models; for a manual emitter it means you populated the input/output arrays. A job with empty input/output lists produces isolated nodes with no edges.

Q: You run `dbt-ol run` but nothing appears in Marquez. What do you check first?
A: First check `OPENLINEAGE_URL` is set and points at the reachable Marquez API (for example `http://localhost:5000`), because if it is unset the wrapper has nowhere to emit and fails silently. Then confirm Marquez is up and the `/api/v1/lineage` endpoint accepts a test POST. Also verify `dbt-ol` actually ran (not plain `dbt`) and that dbt produced artifacts.

Q: A run shows `START` in Marquez but never reaches a terminal state. What does that indicate?
A: It indicates the job crashed, was killed, or the emitter failed before sending `COMPLETE`/`FAIL`/`ABORT`, so the run is left dangling open. For a post-hoc wrapper like `dbt-ol`, a process killed mid-run may never emit the terminal event. Operationally you treat lingering non-terminal runs as suspect and investigate the job's actual exit.

Q: Two datasets that should be the same node appear as two separate nodes in the graph. What is the likely cause?
A: The likely cause is inconsistent dataset identity — different namespace or name strings for the same logical dataset, for example `postgres://host/db.orders` from one emitter and `orders` from another. Because nodes are keyed on (namespace, name), any mismatch splits them. The fix is to standardise namespace conventions across all emitters so they converge on identical identifiers.

Q: You get table-level lineage from dbt but no column-level edges on one model. What are the usual reasons?
A: Usual reasons are SQL the parser cannot fully resolve — an unexpanded `SELECT *` whose upstream schema is unavailable, an unsupported dialect feature, or a heavily dynamic/macro-generated query. In those cases the manifest yields the model-to-model dependency but not the per-column mapping. You confirm by checking whether the model's columns are resolved in `manifest.json`.

Q: A POST to `/api/v1/lineage` returns HTTP 401. What is wrong and how do you fix it?
A: The consumer requires authentication and your event lacks a valid bearer token. Set `OPENLINEAGE_API_KEY` so the client sends the `Authorization` header the backend expects. A local unauthenticated Marquez would not return 401, so a 401 means you are hitting a protected endpoint without credentials.

Q: A POST to `/api/v1/lineage` returns HTTP 422 (unprocessable entity). What does that usually mean?
A: It usually means the event payload is malformed or violates the OpenLineage schema — a missing required field like `eventType`, `eventTime`, `run`, or `job`, or a bad structure. The backend parsed JSON but rejected the content. You validate the event against the OpenLineage spec/schema and fix the offending field.

Q: How can you verify a Marquez instance is healthy before debugging missing events?
A: Hit the admin interface on port `5001`, specifically the `/healthcheck` endpoint, which reports component health. A healthy check rules out the backend being down so you can focus on the emitter or transport. The `/metrics` endpoint on the same admin port also confirms it is serving.

Q: What does a manual minimal OpenLineage event need at the top level besides facets?
A: At minimum a `RunEvent` needs `eventType` (one of the run states), `eventTime` (an ISO-8601 timestamp), a `run` object with a `runId`, a `job` object with `namespace` and `name`, and the `producer` URI. `inputs` and `outputs` are how you declare datasets and thus draw edges. Facets are optional enrichments on top of this skeleton.

Q: What is the `producer` field in an OpenLineage event for?
A: The `producer` is a URI identifying the software that generated the event — for example the integration and version that emitted it. It provides provenance about the lineage itself, so when graphs disagree you know which emitter to trust or fix. Facets also carry a `_producer` for the same reason at facet granularity.

Q: What does the `schema` dataset facet record, and why is it useful for lineage?
A: The `schema` facet records the dataset's fields and their types as observed at that event. It is useful because column-level lineage and impact analysis both depend on knowing what columns exist, and a `SELECT *` parser needs the upstream schema to expand. Schema captured over time also lets you see when a column was added or dropped.

Q: What does the `dataSource` dataset facet describe?
A: The `dataSource` facet describes the source system a dataset belongs to — its name and a connection URI/identifier. It anchors the dataset to a concrete system, reinforcing the namespace and aiding correct stitching across the graph. It helps a reader of the graph see, for example, that a node really is the Postgres source versus an Iceberg table.

Q: What does the `outputStatistics` output facet capture, and why emit it at `COMPLETE`?
A: `outputStatistics` captures volume metrics for what a run wrote — for example row count and byte size of the output. You emit it at `COMPLETE` because the figures are only known once the run has finished producing data. Tracking row counts over runs is a cheap signal for volume anomalies feeding into data-quality monitoring.

Q: What does the `nominalTime` run facet represent?
A: `nominalTime` records the logical scheduled window a run is for — its nominal start (and optionally end) time — distinct from the wall-clock execution time. It matters for scheduled pipelines where a run executing now is actually processing yesterday's data partition. It lets you align lineage with the business date the data represents.

Q: What is the `parent` run facet, and how does it help with orchestrated pipelines?
A: The `parent` run facet links a child run to its parent run/job, modelling the hierarchy when an orchestrator triggers sub-jobs. It helps because you can see, for example, that several dbt-model runs belong to one orchestrated pipeline run. This keeps the graph navigable from a top-level run down into its constituent steps.

Q: What does the `sql` facet capture, and why is it valuable for an auditor?
A: The `sql` facet captures the actual SQL a job executed. It is valuable because an auditor (or you, debugging) can see the exact transformation logic that produced an output without hunting through a code repo. Combined with column-level lineage it turns "trust me" into "here is the query and the columns it maps".

Q: What does the `errorMessage` run facet do?
A: The `errorMessage` run facet attaches failure details — message, optional stack trace, and programming language — to a run, typically alongside a `FAIL` event. It turns a bare failure into actionable diagnostics visible in the lineage backend. That means you can triage a failed NAV run from the lineage UI rather than digging through scheduler logs.

Q: What is a dataset's `dataQualityMetrics` / `dataQualityAssertions` facet for, and how does it connect to your DQ stack?
A: Those input/output facets carry data-quality results — measured metrics and pass/fail assertions — attached to the dataset in the lineage event. They connect your DQ framework (for example Soda or Great Expectations) to lineage so a failing check is visible right on the dataset node. That co-location lets you see "this dataset is downstream of a failed check" in one graph.

Q: Why is lineage described as the architect's tool for "impact analysis" specifically?
A: Because impact analysis is the question "if I change X, what downstream breaks?", and lineage is a precomputed downstream graph that answers it deterministically. Without it you reason from incomplete mental models and miss consumers. With column-level lineage you even scope the impact to the specific field changed, not the whole table.

Q: How does lineage reduce the risk of a schema change to a source column?
A: By letting you query, before you make the change, every downstream job and dataset that reads the column, so you can update or coordinate them rather than discovering breakage in production. It converts a risky change into a planned migration with a known consumer list. In a regulated pipeline that is the difference between a controlled change and an incident on a NAV report.

Q: Why is automatic emission preferred over manually maintained lineage diagrams?
A: Automatic emission captures lineage from the actual execution, so it stays accurate as code changes, whereas hand-drawn diagrams drift and become wrong silently. Manual diagrams also cannot realistically reach column granularity. The whole point of OpenLineage emitters is that lineage is a byproduct of running the pipeline, not a separate maintenance burden.

Q: What is the relationship between OpenLineage and Marquez — is Marquez required?
A: OpenLineage is the standard/spec for the events; Marquez is one reference backend that consumes them, but it is not the only option. Other consumers (catalogs like OpenMetadata or DataHub) can ingest OpenLineage too. So you can emit OpenLineage and route it to whichever backend fits, with Marquez being the convenient default for learning.

Q: Why does the lesson have you emit from both Dagster (orchestrator) and dbt rather than one?
A: Because each sees a different slice: the orchestrator captures the run hierarchy and steps it triggers, while dbt captures fine-grained model and column lineage within the transform layer. Emitting from both, with consistent dataset naming, stitches into a fuller end-to-end graph. Relying on one alone leaves gaps that an auditor's trace will hit.

Q: What does "designing for lineage" mean as an upfront architectural activity?
A: It means deciding job boundaries, naming conventions, and namespace schemes before you wire emitters, so the resulting graph is readable and the datasets stitch correctly across tools. It is the difference between a coherent end-to-end path and a fragmented mess of mis-named nodes. Like data modelling, lineage quality is mostly determined by deliberate design, not the tool.

Q: How does consistent namespacing enable end-to-end stitching across emitters?
A: When the dbt integration and the Spark integration both refer to the same physical dataset using the identical (namespace, name), the backend merges them into one node and the sub-graphs connect. If they disagree, you get two disconnected halves. So agreeing namespace conventions across emitters is the linchpin of a single traceable chain.

Q: Why might Spark-emitted lineage and dbt-emitted lineage fail to connect even when both work individually?
A: They fail to connect when they identify the shared dataset differently — Spark might name an Iceberg table by its storage path while dbt names it by its catalog identifier. Both graphs are correct internally but never join at the boundary node. The remedy is to normalise dataset identifiers so the handoff dataset is one node, not two.

Q: What is the limitation of relying on SQL-parsing lineage for a pipeline step written in imperative Python?
A: SQL parsers see nothing in imperative Python — a pandas or plain-Python transform has no SQL to parse, so it is effectively a black box to dbt-style lineage. You would get lineage only if that step runs under an instrumented engine (like Spark) or emits OpenLineage manually. Otherwise it is a gap to annotate.

Q: How would you make a Python-only transform visible in lineage if no integration covers it?
A: Use the OpenLineage Python client directly to emit `START` and terminal `RunEvent`s with explicit `inputs` and `outputs` declaring the datasets it reads and writes. This manually bridges the step so the graph has edges through it. You accept that without a parser the column-level detail is whatever you choose to assert.

Q: Why is the Debezium CDC hop a likely lineage gap in the NAV pipeline?
A: Debezium streams change events from Postgres into the lake, and unless something on that hop emits OpenLineage, the graph may show the Postgres source and the landed Iceberg table without an observed edge between them. CDC plumbing is exactly the kind of step that can launder lineage. You annotate it so the upstream-to-landing link is asserted rather than missing.

Q: What is the difference between a job emitting `JobEvent`s versus `RunEvent`s in terms of what lands in the graph?
A: `RunEvent`s create runs under a job and, via their inputs/outputs, draw the dataset edges that form the live lineage. `JobEvent`s register design-time job metadata (declared inputs/outputs, source location) without an execution. Most operational lineage comes from `RunEvent`s; `JobEvent`s are for static registration.

Q: How does emitting `inputs` and `outputs` on a `RunEvent` actually create lineage edges?
A: The backend reads the run's declared input datasets and output datasets and infers that the job transformed inputs into outputs, drawing edges input→job→output. Repeated across jobs, these edges chain into the end-to-end graph. So a run that lists no inputs/outputs contributes a node but no edges.

Q: Why does OpenLineage attach `columnLineage` to the output dataset rather than the job?
A: Because the per-column derivation is a property of how the output was produced — each output column is mapped to its contributing input columns — so the natural home is the output dataset's facet. The job is the process; the column mapping is the result. This also lets a consumer read column lineage straight from the dataset it cares about.

Q: What does "transformation type" (for example DIRECT/IDENTITY) in column lineage tell you?
A: It describes how an input column contributed to an output column — for example a direct copy versus participation in an expression or aggregation. This adds nuance beyond "X depends on Y", indicating whether the value passed through unchanged or was transformed. It helps assess, for instance, whether a units or formatting change could alter the figure.

Q: An auditor wants column-level lineage but your Spark step only yields table-level for one transform. How do you respond honestly?
A: You explain that the Spark plan for that transform did not expose resolvable column derivations (for example a complex UDF), so lineage there is table-level, and you point to where column-level does hold (the dbt layer). You then annotate the table-level segment as a known limitation rather than claiming column precision you do not have. Honesty about coverage is itself part of defensible lineage.

Q: Why is it important that lineage events carry a stable `runId` per execution?
A: Because the backend correlates the `START`, optional `RUNNING`, and terminal events of one execution by `runId`, merging their cumulative facets into a single run record. A changed or duplicated `runId` would split one execution into multiple phantom runs or lose the terminal state. Stable run identity is what keeps the run-cycle coherent.

Q: How do you confirm column-level lineage actually rendered for the EMT output, per the Done-when criteria?
A: You open the EMT output dataset in the lineage view and inspect a specific output column to see its mapped upstream columns, confirming the `columnLineage` facet populated through the dbt layer. If only dataset-level edges show, column lineage did not capture and you investigate the parser/manifest. The criterion is satisfied only when you can show a column tracing back to source columns.

Q: Why must lineage stitch across the whole path rather than per-tool, for the auditor demonstration?
A: Because the auditor asks for the EMT output back to source tables — a single unbroken chain across Postgres, Iceberg, Spark, and dbt — not four separate per-tool graphs. Stitching requires consistent dataset identity at every handoff so the segments join. A graph that stops at a tool boundary fails the "back to source" test.

Q: What is the risk of over-granular jobs specifically for the impact-analysis use case?
A: Over-granular jobs explode the downstream traversal into many tiny hops, making the impact set harder to read and slower to reason about, even if technically complete. The blast radius answer gets buried in noise. Sensible job granularity keeps the impact query's output to meaningful transformations.

Q: How does lineage in the catalog help a non-engineer (for example a fund accountant) trust a figure?
A: In the catalog, the figure's dataset page shows its upstream lineage alongside ownership, descriptions, and data-quality status in business-friendly terms, so a fund accountant can see where it came from and whether checks passed without reading code. Marquez's raw graph is engineer-facing; the catalog presents it in context. This broadens who can self-serve provenance questions.

Q: Why is a LEI-keyed counterparty dataset a good candidate to trace with lineage?
A: An LEI (Legal Entity Identifier) uniquely identifies legal entities, so a counterparty dataset keyed on LEI threads through many downstream aggregates and reports. Lineage lets you trace any LEI-based figure back to the authoritative entity source and forward to every consumer. That matters when an entity's data is corrected and you must find all affected outputs.

Q: How does lineage assist when a SWIFT-sourced settlement feed changes format?
A: Lineage shows every downstream dataset and job that consumes the SWIFT-derived feed, so you can scope the impact of a format change before applying it. You see which reports and aggregates would break and plan the migration. Without lineage you would discover the breakage only when a downstream NAV or settlement report failed.

Q: What is the difference in trust between "observed" lineage and "asserted/annotated" lineage?
A: Observed lineage is captured automatically from real execution (an emitter saw the run), so it is high-confidence; asserted lineage is manually declared to bridge a black-box gap, so it is only as good as the human's claim. Mixing them is fine, but you must mark which is which. An auditor needs to know an annotated edge is asserted, not measured.

Q: Why might you choose Marquez for hands-on learning but a catalog for production lineage?
A: Marquez is the lightweight reference backend ideal for emitting, inspecting, and learning the OpenLineage event flow quickly. In production you usually want lineage unified with the rest of your metadata — ownership, glossary, classifications, DQ — which the catalog provides. So you learn the mechanics in Marquez and operationalise in OpenMetadata/DataHub.

Q: How do you evaluate a corporate tool like Microsoft Purview or Collibra against OpenLineage-based lineage?
A: You assess connector coverage (which of your systems it can extract lineage from), the depth it reaches (table-level versus column-level), and where it stops, versus the open OpenLineage stack's coverage of the same path. The key questions are gaps and accuracy on your specific Postgres/Iceberg/Spark/dbt path. Evaluation level means knowing their limits, not deploying them.

Q: What does it mean that Collibra and Purview have "connector limits", in lineage terms?
A: It means each only auto-captures lineage from the systems it has built connectors for, and for unsupported systems lineage is partial, manual, or absent. So a tool may give rich lineage in a warehouse it integrates with but a black-box gap across a system it does not. Knowing those limits is what lets you judge whether they cover your end-to-end NAV path.

Q: An emitter sends events but Marquez shows runs under the wrong namespace. What controls this and how do you fix it?
A: The job namespace is controlled by `OPENLINEAGE_NAMESPACE` (or the namespace the integration sets), so an unexpected namespace means it was left at default or set wrong. You fix it by setting `OPENLINEAGE_NAMESPACE` deliberately for the pipeline. Correct namespacing keeps the NAV jobs grouped and the UI navigable instead of dumped into `default`.

Q: After standing up lineage end-to-end, what three Done-when checks prove the lesson is complete?
A: First, demonstrate the EMT output's lineage back to source tables to an auditor, column-level where the tooling allows. Second, answer the `orders.amount` impact query — which reports break — from the graph alone. Third, identify and annotate any black-box step where lineage stops. Meeting all three shows both the captured graph and an honest account of its limits.

Q: Which three fields are required at the top level of every OpenLineage event regardless of event type?
A: Every event — `RunEvent`, `JobEvent`, or `DatasetEvent` — requires `eventTime`, `producer`, and `schemaURL` at the top level. `eventTime` is an ISO-8601 timestamp, `producer` is the URI of the emitting software, and `schemaURL` is the JSON-pointer URL of the schema version the event conforms to. A `RunEvent` additionally needs `eventType`, `run`, and `job`, but those three are the universal baseline.

Q: What exact format must `eventTime` use, and why does including the timezone offset matter for a NAV pipeline?
A: `eventTime` must be an ISO-8601 timestamp such as `2026-06-14T19:52:00.001+02:00`, and including the offset matters because a Luxembourg pipeline running in CET/CEST must not have its events misordered against UTC-emitted upstream events. Dropping the offset lets the backend assume a default zone and can shuffle the apparent order of `START` and `COMPLETE`. Always emit a zone-aware timestamp so the run timeline is unambiguous across the EMT chain.

Q: In the `columnLineage` facet, what distinguishes a `DIRECT` transformation from an `INDIRECT` one?
A: `DIRECT` means the output column value was derived from the input field value — the data actually flowed into it — whereas `INDIRECT` means the input merely influenced the output without being part of its value, for example a column used only in a `WHERE` or `GROUP BY`. So `nav_per_share = aum / shares` is `DIRECT` from both inputs, but the `valuation_date` used only to filter rows is `INDIRECT`. The distinction sharpens impact analysis by separating value-carrying columns from filter/key columns.

Q: Name the three `DIRECT` transformation subtypes in the OpenLineage column-lineage facet and what each means.
A: The `DIRECT` subtypes are `IDENTITY` (the value is copied unchanged), `TRANSFORMATION` (the value is altered, e.g. a cast or arithmetic), and `AGGREGATION` (the value summarises multiple input rows, e.g. `SUM` or `AVG`). For an EMT cost field computed as a sum over fee rows, the subtype would be `AGGREGATION`. Knowing the subtype tells an auditor whether a number passed through untouched or was computed.

Q: Name the `INDIRECT` transformation subtypes and give a fund-data example of one.
A: The `INDIRECT` subtypes are `JOIN`, `GROUP_BY`, `FILTER`, `SORT`, `WINDOW`, and `CONDITIONAL`. For example, an `ISIN` column used as the join key linking a holdings table to a prices table is an `INDIRECT` `JOIN` contributor to the priced-position output, even though the ISIN value itself does not appear in the result amount. These capture columns that shape which rows or ordering produce the output without supplying its value.

Q: What does the boolean `masking` field on a column-lineage transformation indicate, and when is it relevant in fund data?
A: `masking` is a boolean that is `true` when the input value was obfuscated during the transformation — for example hashed, tokenised, or only counted rather than passed through. It is relevant when a personally-identifying TA field (an investor name or account) is hashed before flowing downstream, so lineage records that the link exists but the value was masked. This lets you prove a PII column influenced an aggregate without the raw value leaking into the lineage record.

Q: How would you query Marquez for the lineage graph of a single dataset, and what `nodeId` format does it expect?
A: You call `GET /api/v1/lineage?nodeId=dataset:<namespace>:<name>`, for example `nodeId=dataset:nav-pipeline:emt_output`. The `nodeId` for a job is `job:<namespace>:<name>` and for a single column it is `datasetField:<namespace>:<name>:<field>`. Marquez returns the connected graph as JSON, which is how you script an impact query instead of clicking through the UI.

Q: What does the `depth` parameter default to on the Marquez lineage endpoint, and when would you raise it?
A: `depth` defaults to `20` on `GET /api/v1/lineage`, controlling how many hops of the graph are traversed from the `nodeId`. You raise it when a long NAV chain (Postgres → Debezium → Iceberg → Spark → dbt → EMT) plus catalog jobs exceeds twenty edges and the upstream source is being truncated. You lower it when you only want the immediate neighbours of a node.

Q: Which Marquez release first exposed column-level lineage via the API, and why does the version matter?
A: Column-level lineage landed in Marquez `0.27.0`, exposing dataset column inputs/outputs through the API and a dedicated column-lineage endpoint. The version matters because an older Marquez will ingest `columnLineage` facets but not surface them through that endpoint or the column graph in the UI. If your column lineage "disappears", confirm the backend is recent enough before blaming the emitter.

Q: What is the Marquez column-lineage endpoint's `withDownstream` parameter, and what does it default to?
A: The column-lineage query accepts `withDownstream`, which defaults to `false`, meaning by default you get only the upstream column derivations of the node you ask about. Set it to `true` to also pull the downstream columns that depend on your column — the "what breaks" direction for `orders.amount`. So upstream provenance is the default; downstream impact is opt-in.

Q: In the OpenLineage Python client, how do you wire an `OpenLineageClient` to send events over HTTP to Marquez?
A: You build an `HttpConfig` (or a config dict with `type` `http` and the `url`), pass it to `HttpTransport`, then construct `OpenLineageClient(transport=HttpTransport(http_config))` and call `client.emit(event)`. Authentication is supplied via an `ApiKeyTokenProvider` in the config when the backend requires a token. This is the manual path you use to bridge a Python-only transform that no integration covers.

Q: When emitting manually with the Python client, what objects do you assemble to declare that a job read one dataset and wrote another?
A: You build a `RunEvent` with a `Run` (carrying the `runId`), a `Job` (carrying `namespace` and `name`), and lists of `InputDataset` and `OutputDataset` objects each carrying their own `namespace` and `name`. Emitting `START` then `COMPLETE` `RunEvent`s with those inputs/outputs draws the input→job→output edges. Omitting the dataset lists yields a run node with no lineage edges.

Q: Why must you emit at least two `RunEvent`s (not one) for a manually-instrumented Python step, and what are they?
A: You emit a `START` event when the step begins and a terminal `COMPLETE` (or `FAIL`/`ABORT`) when it ends, sharing one `runId`, because the backend correlates them to form a single run with a start and end time. A lone `START` leaves a dangling open run; a lone `COMPLETE` loses the start boundary. Inputs/outputs can ride on either, but emitting them on both is safe since facets accumulate.

Q: A `POST /api/v1/lineage` returns HTTP 400 rather than 422. What is the practical difference you'd investigate?
A: A `400` typically means the request itself was bad — malformed JSON the parser could not even read, or a wrong content type — whereas a `422` means valid JSON that violated the OpenLineage schema (a missing `eventType` or `eventTime`). For `400` you re-check the body is valid JSON and the `Content-Type: application/json` header is set; for `422` you validate field-by-field against the spec. Confusing the two wastes time validating schema when the JSON never parsed.

Q: A `POST /api/v1/lineage` hangs and eventually times out with no HTTP status. What are the first two things to check?
A: First check network reachability — that Marquez is actually listening on the host/port in `OPENLINEAGE_URL` (a `curl http://localhost:5000/api/v1/namespaces` should respond), since a wrong host silently hangs until timeout. Second check the client's configured timeout and whether a proxy or firewall is swallowing the connection. A hang is a transport problem, not a payload problem, so do not start editing the event JSON.

Q: How do you confirm Marquez ingested your event without opening the web UI?
A: Query the API directly — `GET /api/v1/namespaces` to see your namespace, then `GET /api/v1/namespaces/<ns>/jobs` and `GET /api/v1/namespaces/<ns>/datasets` to confirm the job and datasets landed. If the namespace exists but the dataset list is empty, the event arrived but declared no inputs/outputs. This API-first check isolates emitter problems from UI rendering problems.

Q: The Marquez UI is blank but `GET /api/v1/namespaces` returns your namespace. Where is the fault?
A: The data is in the backend (the API confirms ingestion), so the fault is between the web UI and the API — usually the UI pointing at the wrong API URL or the wrong namespace selected in the UI dropdown. Check the web app's configured API endpoint and that you have selected the correct namespace rather than `default`. The emitter and transport are exonerated by the successful API query.

Q: What is the consequence of reusing the same `runId` across two separate executions of a job?
A: The backend merges the two executions into one run record because it correlates events by `runId`, so their facets and inputs/outputs collide and the run timeline becomes nonsense (two starts, two completes). For the NAV job this could make a successful run appear failed or double-count outputs. Always generate a fresh UUID `runId` per execution; the job identity stays stable, the run identity must not.

Q: How does OpenLineage represent a dataset that is both read and written by the same run, and why does that matter for an Iceberg merge?
A: It appears in both the run's `inputs` and `outputs` lists, which is exactly what an Iceberg `MERGE` does — it reads the target table and writes it back. The lineage then shows a self-edge or a read-then-write cycle on that dataset node. Recognising this prevents you from mistaking a legitimate upsert for a misconfigured loop in the graph.

Q: Why might the Spark integration miss column lineage on a `MERGE INTO` even when table-level lineage is captured?
A: Historically the Spark integration has had gaps capturing column lineage for `MERGE` operations where the transformation logic is not fully exposed in the resolved plan, so you get the dataset-level edge but not the per-column mapping. This is a known limitation, not a misconfiguration. You verify by checking whether the merged columns carry a `columnLineage` facet and, if not, annotate that segment as table-level only.

Q: How do you choose a dataset `namespace` for a Postgres source versus an Iceberg table so the two stitch correctly?
A: Use scheme-based namespaces that reflect the physical system — for example `postgres://host:5432/navdb` for the source and a consistent catalog-based identifier like `iceberg://nav_catalog` for the lake table — and apply the identical string from every emitter touching that dataset. The names stitch only when both Spark and dbt spell the handoff dataset identically. Picking these conventions up front is the design decision that makes end-to-end traversal possible.

Q: dbt names a model `analytics.fct_positions` but Spark writes the same Iceberg table as `s3://lake/positions`. How do you make them one node?
A: You normalise the dataset identifier so both emitters report the table by the same `(namespace, name)` — typically by configuring the Spark integration's dataset naming (or a naming override) to use the catalog identifier dbt uses, rather than the raw storage path. Until they match, Marquez keeps two unconnected nodes and the chain breaks at that handoff. This is the classic Spark-storage-path vs dbt-catalog-name mismatch.

Q: What does the OpenLineage `dataset` naming convention say about how to name a Postgres table, and why follow it?
A: The naming convention specifies a scheme-and-authority namespace (e.g. `postgres://{host}:{port}`) with the dataset name as `{database}.{schema}.{table}`, so identifiers are predictable across tools. Following it means independently-built integrations converge on the same node without manual mapping. Inventing your own ad-hoc names is the fastest route to a fragmented graph.

Q: How would you use lineage to scope a GDPR/PII deletion request for an investor's data across the pipeline?
A: You locate the TA source column holding the investor identifier, then traverse downstream with `withDownstream=true` to enumerate every dataset and job that derived from it, giving the full set of places the data propagated. Column-level lineage narrows this to outputs actually carrying the identifier rather than every table touching the source. That enumerated list is what you act on and what you show to prove the deletion was complete.

Q: An EMT cost figure is wrong. Walk through using lineage to localise whether the bug is in dbt or upstream.
A: Open the EMT output column in the column-lineage view and read its derivation back through dbt; if the dbt mapping and `sql` facet show correct logic, the fault is upstream, so you continue walking to the Spark and Iceberg inputs and inspect their `schema`/`outputStatistics` facets for anomalies. You isolate the layer where the value first goes wrong rather than re-reading all the code. The `sql` facet at each hop lets you confirm the exact transformation without leaving the graph.

Q: How does the `outputStatistics` facet help you catch a silent NAV data-loss bug across runs?
A: `outputStatistics` records the row count and byte size each run wrote, so comparing it across runs flags a sudden drop — say positions falling from 4,000 rows to 40 — that signals an upstream filter or join went wrong. Because it is emitted at `COMPLETE` when the figures are known, it becomes a cheap volume-anomaly signal feeding DQ. A NAV that looks plausible can still be built on a truncated holdings set, and row-count drift surfaces it.

Q: Why is the `nominalTime` facet, not `eventTime`, the right key when reconciling a T+1 NAV run against its business date?
A: `nominalTime` records the logical scheduled window the run processes — yesterday's valuation date for a T+1 calc — whereas `eventTime` is merely when the event was emitted (which could be today, or a backfill weeks later). Keying reconciliation on `nominalTime` aligns the lineage to the business date the data represents. Using `eventTime` would misattribute a late or replayed run to the wrong NAV date.

Q: How does the `parent` run facet let an auditor navigate from a Dagster pipeline run down to a single dbt model run?
A: The `parent` facet on each child run points to the orchestrating run/job, so in the graph the Dagster pipeline run is the parent of the individual dbt-model runs it triggered. The auditor starts at the top-level run and drills into its constituent steps without guessing which model runs belong together. Without `parent`, the dbt runs float as siblings with no recorded membership in the orchestrated execution.

Q: Why does emitting only `JobEvent`s (no `RunEvent`s) leave you unable to answer the `orders.amount` impact query?
A: `JobEvent`s register design-time job metadata and declared inputs/outputs but create no runs, and in practice the live dataset edges and column lineage that the impact query traverses come from `RunEvent` inputs/outputs and facets. With only static `JobEvent`s you may have nodes but lack the observed, run-derived edges and column mappings. The impact query needs the executed lineage, so `RunEvent`s are mandatory.

Q: When is a `DatasetEvent` the better choice than a `RunEvent` for registering an EMT output's schema?
A: When the schema is a design-time fact you want catalogued independently of any execution — for example publishing the agreed EMT field structure before the pipeline first runs — a `DatasetEvent` carrying the `schema` facet registers it without inventing a fake run. A `RunEvent` would tie that schema to a specific execution that did not happen. Use `DatasetEvent` for static dataset metadata, `RunEvent` for what a run actually read and wrote.

Q: How do you make a Debezium CDC hop visible in lineage when nothing on that hop emits OpenLineage?
A: You manually emit a `RunEvent` (or register `JobEvent`/`DatasetEvent`s) for a synthetic "cdc-capture" job declaring the Postgres source as input and the landed Iceberg table as output, so the graph bridges the gap with an asserted edge. You label it clearly as asserted rather than observed. This turns a silent break between the source and the landing table into a visible, documented hop.

Q: An auditor asks whether an annotated CDC edge is "real" lineage. How do you frame the answer?
A: You explain it is asserted lineage — manually declared to bridge a step that emits nothing — not observed lineage captured from an instrumented run, and that its confidence rests on the documented contract of what Debezium moves. You point to where observed lineage resumes (the Iceberg table onward) and why the asserted hop is reliable. Being explicit that the edge is asserted, not measured, is itself part of defensible lineage.

Q: Why can a pure pandas transform between Spark and dbt be a lineage gap, and what is the minimal fix?
A: A pandas step has no SQL for dbt to parse and runs outside Spark's instrumented engine, so no integration observes it and the graph shows input and output datasets with no edge between them. The minimal fix is to wrap it with the OpenLineage Python client, emitting `START`/`COMPLETE` `RunEvent`s that declare its `inputs` and `outputs`. You accept that column-level detail is whatever you choose to assert, since no parser is involved.

Q: How does the `sql` facet plus `columnLineage` together convert "trust me" into evidence for a regulator?
A: The `sql` facet shows the exact query a job ran, and `columnLineage` shows which input columns produced each output column, so together they let a regulator see both the logic and the precise data flow behind an EMT figure. Neither alone is enough — the SQL without the column map still requires the regulator to parse it, and the column map without the SQL hides the transformation. Presented together they form a self-contained, auditable derivation.

Q: Why does column-level lineage attach to the output dataset and not to the input dataset?
A: Because the per-column derivation describes how each output column was produced from its inputs — it is a property of the output's construction — so it naturally lives on the output dataset's `columnLineage` facet. The input dataset has no knowledge of how downstream outputs will combine its columns. A consumer reading the EMT output therefore finds its column provenance right on the node it cares about.

Q: How would you script the `orders.amount` impact answer against Marquez rather than clicking the UI?
A: You call the column-lineage endpoint with `nodeId=datasetField:<ns>:orders:amount` and `withDownstream=true`, then parse the returned graph to collect the terminal downstream dataset fields — the reports that consume `amount`. Scripting it makes the impact answer repeatable and diffable across changes, not a one-off manual trace. The `withDownstream=true` flag is what flips the query from provenance to impact.

Q: Why does column-level impact analysis depend on every intermediate hop also carrying `columnLineage`, and what happens if one hop only has table-level?
A: A column-level downstream traversal can only follow column edges where they exist; if an intermediate Spark `MERGE` hop emits only table-level lineage, the trace loses column granularity there and must widen to the whole dataset beyond that point. So one missing hop degrades the precision of the entire downstream answer. You disclose this by saying the impact set is column-precise up to the Spark hop and table-level thereafter.

Q: How do you tell, from the lineage record alone, whether a dataset edge was observed or asserted?
A: You inspect the `producer` URI on the event and facets — observed edges carry the URI of a real integration (the Spark or dbt OpenLineage producer) running against an execution, while asserted edges carry the URI of your manual client and typically a synthetic job. Documenting which `producer` values are manual lets a reviewer separate measured from declared lineage. This is why the `producer` field is provenance about the lineage itself.

Q: A dbt model uses `SELECT *` from a source and you get table-level but no column lineage. What is the precise cause and fix?
A: The parser cannot expand `SELECT *` into concrete columns unless the upstream schema is resolvable at compile time, so without a populated source schema in `manifest.json` it emits the model-to-model dependency but no column map. The fix is to give dbt the source's column definitions (e.g. via a `catalog.json`/docs generate or explicit source schema) so the star can be expanded, or to enumerate the columns explicitly in the SQL. Confirm by checking whether the model's columns appear resolved in `manifest.json`.

Q: Why does `dbt-ol` produce no lineage when a dbt run is killed by `SIGKILL` mid-execution, and what mitigates it?
A: `dbt-ol` is a post-run wrapper that parses dbt's artifacts after dbt exits, so a hard kill before dbt writes `run_results.json` (and before the wrapper runs) leaves nothing to translate and no events are emitted. The mitigation is to capture live per-step lineage from the orchestrator integration, which emits `START`/`FAIL` as steps run, rather than relying solely on the post-hoc dbt wrapper. This is the structural reason to emit from both Dagster and dbt.

Q: Which two dbt artifacts must exist for `dbt-ol` to emit, and what does each contribute?
A: `manifest.json` (the compiled DAG with model/source dependencies and column metadata) and `run_results.json` (which nodes ran and their status) must exist; the manifest supplies the lineage structure and column info, while run_results supplies the run outcomes. If `dbt-ol` ran but `manifest.json` is stale or missing columns, you get edges without column lineage. Both are written to dbt's `target/` directory during a normal invocation.

Q: How does running `dbt-ol` differ operationally from setting `OPENLINEAGE_URL` for a Spark job?
A: `dbt-ol` is a command substitution — you literally run `dbt-ol run` instead of `dbt run`, and it reads `OPENLINEAGE_URL` to know where to POST after dbt finishes — whereas Spark is configured by attaching the OpenLineage listener via Spark conf (e.g. `spark.openlineage.transport.url`) so it streams events during execution. One wraps the CLI post-hoc; the other instruments the engine live. Both still need a reachable backend URL, but they are configured in different places.

Q: Why is the legacy `openlineage-dagster` plugin risky on a current Dagster, and what is the supported path?
A: The legacy plugin is no longer actively maintained and supported Dagster is capped around `1.6.9`, so newer Dagster releases introduce breaking changes that the plugin does not track. The supported path is Dagster's own native asset/lineage emission rather than the deprecated package. Before wiring it, verify the integration against your exact Dagster version instead of assuming the old plugin works.

Q: How do you evaluate whether Microsoft Purview covers your NAV path's lineage, in concrete terms?
A: You map each system in the path (Postgres, Debezium, Iceberg, Spark, dbt) to whether Purview has a connector that auto-captures its lineage and at what depth — table-level or column-level — then list the hops where it falls back to manual or captures nothing. The evaluation output is a coverage-and-gaps table for your specific path, not a vendor feature list. The decisive questions are connector coverage and column-level accuracy on the systems you actually run.

Q: Collibra shows rich lineage in your warehouse but nothing across the Debezium hop. What does this tell you and how do you respond?
A: It tells you Collibra has a connector for the warehouse but not for the CDC hop, so lineage there is partial or absent — a connector limit, not a configuration bug. You respond by bridging the gap with OpenLineage (emitting an asserted CDC edge) or documenting it, exactly as you would for any black-box step. Knowing each tool's connector boundary is what lets you judge end-to-end coverage honestly.

Q: How does lineage in OpenMetadata combine with its data-quality results to answer "is this EMT figure both traceable and validated"?
A: OpenMetadata materialises the OpenLineage edges on the asset's lineage tab and co-locates them with the test/expectation results it tracks, so one page shows both where the figure came from and whether its checks passed. An auditor sees provenance and DQ verdict together rather than correlating two systems. This is why production lineage usually lives in the catalog rather than only in Marquez.

Q: Why is feeding OpenLineage into the catalog preferable to maintaining a separate hand-drawn lineage diagram for the DORA evidence pack?
A: Catalog lineage is captured from real execution so it stays accurate as code changes and can reach column granularity, whereas a hand-drawn diagram drifts silently and cannot credibly claim column-level traceability. For DORA evidence you need demonstrable, current mappings of critical data flows, which automated lineage provides and a stale diagram undermines. The whole point is lineage as a byproduct of running the pipeline, not a maintenance artifact.

Q: What single naming mistake most often fragments an otherwise-correct end-to-end NAV lineage graph, and how do you prevent it?
A: Spelling the same physical handoff dataset differently across emitters — for example Spark using the S3 path and dbt using the catalog name for the same Iceberg table — fragments the graph into disconnected halves because nodes are keyed on `(namespace, name)`. You prevent it by agreeing namespace and dataset-naming conventions up front and configuring every integration to emit the identical identifier at each handoff. This single discipline is the linchpin of a stitched, traceable chain.

Q: How would you set a per-pipeline lineage namespace without exporting `OPENLINEAGE_NAMESPACE` globally on the host?
A: You scope the variable to the invocation — for example `OPENLINEAGE_NAMESPACE=nav-pipeline dbt-ol run` on the command line, or set it in the orchestrator's run environment so only that job inherits it. This keeps the NAV jobs grouped under `nav-pipeline` while other pipelines on the same Ubuntu host use their own namespaces. A global export risks every unrelated emitter landing in the same namespace and cluttering the Marquez UI.

Q: An EMT field's `columnLineage` lists an input column with `type` `INDIRECT` and subtype `FILTER`. What does that tell an auditor about the figure?
A: It tells the auditor that the column did not supply the field's value but determined which rows were included — it was used in a `WHERE`/filter predicate — so it shapes the result set without appearing in the number. For example a `valuation_date = '2026-06-13'` filter selects the rows that feed an EMT cost aggregate. The auditor learns the figure is conditioned on that column, which matters if the filter was wrong even though the value column was correct.
