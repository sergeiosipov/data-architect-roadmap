# Phase 4 · 2.4.1 Distributed logs (Kafka) — 150 self-test questions

Q: What is the core abstraction Kafka provides, in one sentence?
A: Kafka is a distributed, append-only, replayable commit log: producers append records to the end of a log and consumers read forward at their own pace by tracking a position called an offset. Unlike a queue, reading a record does not remove it — records persist for the configured retention so many independent consumers can replay the same history. For a fund estate this is what lets a trade-event stream be both a live feed and an auditable, reprocessable record of what happened.

Q: What is a `topic` in Kafka?
A: A `topic` is a named logical stream of records — the unit producers write to and consumers subscribe to, analogous to a table name or a category. It is a purely logical grouping; the physical storage and parallelism come from the partitions underneath it. Example: a `fund-orders` topic carries all order events, while `fund-reference` carries reference-data changes.

Q: What is a `partition` and why does Kafka split topics into them?
A: A partition is an ordered, immutable, append-only sequence of records — the actual physical log on disk and the unit of parallelism and ordering. A topic is split into partitions so that throughput can scale horizontally across brokers and so that multiple consumers in a group can each own a distinct slice. The key consequence is that ordering is guaranteed only within a single partition, never across the whole topic.

Q: What is an `offset` and what scope does it have?
A: An offset is a monotonically increasing integer that uniquely identifies a record's position within one partition — it is the per-partition sequence number, starting at 0. Offsets are meaningful only inside their own partition; offset 100 in partition 0 has no relation to offset 100 in partition 1. A consumer's progress is just the set of offsets it has read per partition, which is what makes replay (reset to an earlier offset) possible.

Q: What is a log `segment` and why does Kafka use them instead of one giant file?
A: A partition's log is physically broken into segment files, each a chunk of the log plus its index files; only the newest (active) segment is open for writes while older ones are sealed. Segmentation is what makes retention and compaction tractable: Kafka deletes or compacts whole closed segments rather than rewriting one enormous file. The active segment is never deleted, which is why a very small topic can keep data longer than its retention suggests.

Q: What two configuration knobs roll a segment to a new file?
A: A segment rolls when it reaches `segment.bytes` (size limit, default 1 GiB) or when `segment.ms` elapses since the segment was created (time limit). Whichever triggers first closes the active segment and opens a new one, making the closed segment eligible for retention-based deletion or compaction. Tuning these matters because retention and compaction only ever act on closed segments, so huge segments delay cleanup.

Q: What is the difference between retention and compaction as cleanup policies?
A: Retention (`cleanup.policy=delete`) discards whole segments once they age past `retention.ms` or the partition exceeds `retention.bytes`, regardless of keys — it is time/size-bounded history. Compaction (`cleanup.policy=compact`) instead keeps the latest value per key forever and only removes superseded older values, turning the log into a changelog. They answer different questions: retention bounds how far back you can replay; compaction bounds storage while preserving current state per key.

Q: Can a topic use both retention and compaction at once?
A: Yes — set `cleanup.policy=compact,delete` and Kafka both compacts (keeps the latest value per key) and applies time/size retention to age out old data entirely. This suits a changelog that should keep current state per key but still not retain genuinely ancient keys forever. Without `delete` added, a purely compacted topic never drops a key whose latest value is a real (non-tombstone) record.

Q: What does `retention.ms=-1` mean on a topic?
A: A value of `-1` disables time-based retention, meaning Kafka never deletes segments because of age — data is kept indefinitely (subject to `retention.bytes` if set). This is used for topics meant as a permanent record, such as an event-sourced ledger you intend to replay from the beginning. Be deliberate: infinite retention is a real storage and compliance commitment, not a default to reach for casually.

Q: A consumer asks for offset 5 but the partition's earliest available offset is 200 — what happens?
A: The requested offset is below the log-start offset because retention already deleted those segments, so the broker returns an out-of-range condition and the consumer must follow its `auto.offset.reset` policy. With `auto.offset.reset=earliest` it jumps to the new earliest offset (200); with `latest` it jumps to the end; with `none` it raises an error. The lesson is that retention can silently invalidate stored offsets if a consumer is offline longer than the retention window.

Q: What is the `log-start offset` versus the `log-end offset` of a partition?
A: The log-start offset is the offset of the earliest record still retained (it rises as old segments are deleted), and the log-end offset is one past the last written record — the position the next append will take. The difference between a consumer's committed offset and the log-end offset is its lag. These two markers define the replayable window: you can rewind anywhere between log-start and log-end.

Q: What is a producer `acks` setting and what do the three values mean?
A: `acks` controls how many broker acknowledgements a producer waits for before considering a send successful: `acks=0` waits for none (fire-and-forget, may lose data), `acks=1` waits for the leader only, and `acks=all` (a.k.a. `-1`) waits for all in-sync replicas. It is the durability dial — higher acks trade latency for stronger guarantees against loss on broker failure. For settlement-grade events you use `acks=all`.

Q: What exactly does `acks=all` guarantee, and what does it not guarantee alone?
A: `acks=all` guarantees the record is persisted on the leader and replicated to every replica currently in the in-sync replica set before the producer is told it succeeded, so a single broker loss cannot lose an acknowledged record. It does not by itself guarantee against duplicates from producer retries, nor does it help if the ISR has shrunk to just the leader — that is why it must be paired with `min.insync.replicas` and idempotence. Alone it answers "won't lose," not "won't duplicate" or "enough replicas."

Q: What problem does the idempotent producer solve?
A: It prevents duplicate records caused by producer retries: when a send times out and the producer resends, the broker would normally write the message twice, but an idempotent producer deduplicates so each record is written exactly once per partition. It works by tagging each batch with a producer ID and a per-partition sequence number the broker uses to detect and drop resends. This gives you exactly-once-into-the-partition for a single producer session without transactions.

Q: How do you enable the idempotent producer, and is it on by default in current Kafka?
A: You set `enable.idempotence=true`; in modern Kafka (3.0+) this is the default, which also implicitly requires `acks=all`, `retries` greater than 0, and `max.in.flight.requests.per.connection` at most 5. If you explicitly set conflicting values (e.g. `acks=1`) the client will reject the config or disable idempotence. The practical takeaway is that out of the box you already get retry-safe, no-duplicate writes per partition.

Q: What do Kafka transactions add on top of the idempotent producer?
A: Transactions let a producer write to multiple partitions (and commit consumer offsets) atomically — either all writes become visible together or none do — extending idempotence from one partition to a set. You configure a `transactional.id`, call `beginTransaction`/`commitTransaction`, and consumers reading with `isolation.level=read_committed` see only committed records. This is the building block of consume-process-produce exactly-once within Kafka, e.g. read an order, compute, and write a result atomically.

Q: What does `isolation.level=read_committed` change for a consumer?
A: It makes the consumer skip records that belong to aborted transactions and not read past any open (uncommitted) transaction, so it only ever sees data that was atomically committed. The default `read_uncommitted` shows all records including those later aborted. The cost is a small added latency, because a consumer must wait for a transaction to commit or abort before delivering records beyond the last stable offset.

Q: What is the Last Stable Offset (LSO) and why does it matter to a read_committed consumer?
A: The LSO is the offset up to which all transactional records are decided (committed or aborted); a `read_committed` consumer will not deliver records beyond the LSO because their fate is still unknown. It is what causes a read_committed consumer to appear to "wait" when a long-running transaction is open upstream. A stuck or hung transaction can therefore stall committed consumers until it is committed, aborted, or fenced by the transaction timeout.

Q: For ordering within a single partition under retries, what setting prevents reordering?
A: Enabling the idempotent producer (`enable.idempotence=true`) preserves per-partition order even with retries and multiple in-flight requests, because the broker uses sequence numbers to reject out-of-order or duplicate batches. Without idempotence, you would have to set `max.in.flight.requests.per.connection=1` to guarantee a retried batch cannot land after a later one. So idempotence buys both no-duplicates and no-reordering without throttling to one in-flight request.

Q: What is a consumer group and what does it accomplish?
A: A consumer group is a set of consumers sharing a `group.id` that cooperatively divide a topic's partitions so each partition is consumed by exactly one member of the group at a time. It provides horizontal scaling (add consumers to process more partitions in parallel) and fault tolerance (if one dies its partitions are reassigned). Different groups are independent, so two groups each get a full copy of the stream — this is how Kafka does fan-out.

Q: What is the relationship between partition count and the maximum useful consumers in a group?
A: Within one group, a partition is assigned to only one consumer, so the number of partitions is the hard ceiling on parallelism — extra consumers beyond the partition count sit idle. If `fund-orders` has 3 partitions, a 4th consumer in the group gets nothing until a rebalance moves a partition to it. This is why partition count is a capacity decision made up front: you cannot scale a group's parallelism past it without adding partitions.

Q: What is a rebalance, and what event triggers one?
A: A rebalance is the process of reassigning partitions among the members of a consumer group; it triggers when membership changes (a consumer joins, leaves, or is deemed dead), when the subscribed topics' partition count changes, or on certain metadata changes. During a rebalance the group coordinator redistributes ownership so the new membership covers all partitions. Frequent or slow rebalances are a common operational pain point because they interrupt consumption.

Q: What is the difference between eager and cooperative (incremental) rebalancing?
A: Eager rebalancing has every consumer revoke all its partitions at the start, then everyone rejoins and gets a fresh assignment — a full stop-the-world pause. Cooperative (incremental) rebalancing revokes only the partitions that actually need to move, letting consumers keep processing their unaffected partitions throughout. Cooperative is far less disruptive, which is why the cooperative-sticky assignor is the modern default choice.

Q: What does the `CooperativeStickyAssignor` do specifically?
A: It is the partition-assignment strategy that combines stickiness (try to keep each consumer's existing partitions to preserve local state and caches) with cooperative rebalancing (move only the minimum set of partitions, in two rounds, without a global revoke). You select it via `partition.assignment.strategy`. The result is minimal disruption: most consumers keep their partitions across a rebalance, so a single consumer restart does not stall the whole group.

Q: What is consumer lag and why is it the core health signal?
A: Consumer lag is the difference, per partition, between the partition's log-end offset and the consumer group's committed offset — how many records behind real time the group is. It is the single best health metric because rising lag means the consumers cannot keep up (or are stuck), independent of CPU or memory readings. In a fund pipeline, sustained lag on the order-events consumer is the early warning that NAV-flow totals are falling behind the market.

Q: Lag is high and flat (not growing, not shrinking) — what does that typically indicate?
A: Flat, elevated lag usually means the consumer is processing at exactly the inbound rate but started behind and is not catching up, or it is stuck retrying a single record at the same throughput. You check whether the consumer is actually committing progress (offsets advancing) versus spinning on one poison record. If offsets are advancing steadily but lag stays flat, you need more consumers/partitions; if offsets are frozen, you have a stuck or crashed-and-restarting member.

Q: A consumer group shows one partition with ever-growing lag while others are fine — first thing to check?
A: That pattern points at data skew or a stuck single consumer, so first check whether one key (and thus one partition) is receiving disproportionate traffic, or whether the consumer owning that partition is blocked. Because ordering is per partition, a hot ISIN that all maps to one partition will overload exactly one consumer while the rest idle. The fix is usually a better partition key or, if it is a stuck member, restarting it and checking for a poison record.

Q: What is `session.timeout.ms` versus `max.poll.interval.ms` in triggering a rebalance?
A: `session.timeout.ms` governs the heartbeat: if the coordinator misses heartbeats for that long, the consumer is considered dead and a rebalance starts. `max.poll.interval.ms` governs progress: if the application takes longer than this between `poll()` calls (e.g. a slow per-record operation), the consumer is removed even though heartbeats are fine. A common bug is long per-batch processing tripping `max.poll.interval.ms` and causing repeated rebalances.

Q: What is a replica, a leader, and a follower in Kafka replication?
A: Each partition has `replication.factor` copies (replicas) spread across brokers; one replica is elected leader and handles all reads and writes, while the others are followers that replicate the leader's log. Producers and consumers talk only to the leader; followers exist to take over if the leader's broker fails. Replication is per partition, so different partitions of the same topic have leaders on different brokers for load spreading.

Q: What is the In-Sync Replica set (ISR)?
A: The ISR is the subset of a partition's replicas that are currently caught up with the leader — within `replica.lag.time.max.ms` of the leader's log-end offset. Only replicas in the ISR are eligible to be elected leader without data loss, and `acks=all` waits for all ISR members. A replica that falls behind (slow disk, network) is removed from the ISR until it catches up again, which shrinks the durability guarantee.

Q: What does `min.insync.replicas` do, and what happens when the ISR drops below it?
A: `min.insync.replicas` sets the minimum number of in-sync replicas that must acknowledge a write for an `acks=all` produce to succeed; if the ISR shrinks below it, the broker rejects the produce with a NotEnoughReplicas error. This deliberately chooses unavailability over silent under-replication — the producer is blocked rather than writing data that only lives on too few replicas. It is the guardrail that makes `acks=all` meaningful.

Q: Why is `acks=all` alone insufficient without setting `min.insync.replicas` appropriately?
A: Because `acks=all` only waits for the current ISR, and if replicas have fallen out the ISR could be just the leader — a single broker — so an acknowledged write would live on one disk and be lost if that broker dies. Setting `min.insync.replicas=2` (with replication factor 3) forces at least two copies before acknowledging, restoring real durability. The classic safe trio is `replication.factor=3`, `min.insync.replicas=2`, `acks=all`.

Q: With `replication.factor=3` and `min.insync.replicas=2`, how many broker failures can you tolerate while still accepting writes?
A: You can tolerate one broker failure and keep accepting writes, because two in-sync replicas remain, which meets the minimum. If a second broker holding that partition's replica also fails, the ISR drops to one, falls below `min.insync.replicas`, and produces are rejected to protect durability — reads of already-committed data can still proceed. This is the deliberate availability-vs-durability boundary of that configuration.

Q: What is unclean leader election and what does it trade off?
A: Unclean leader election allows a replica that was NOT in the ISR (i.e. behind the leader) to become the new leader when no in-sync replica is available — restoring availability at the cost of losing the records the old leader had that the new one never received. It trades durability for availability. It is controlled by `unclean.leader.election.enable`, which is `false` by default in modern Kafka precisely because silent data loss is usually worse than a brief outage.

Q: In a regulated fund estate, would you enable `unclean.leader.election.enable=true` on a trade-events topic? Why?
A: No — for settlement or order events, silently losing acknowledged records to restore availability is exactly the failure auditors trace to a design decision, so you keep it `false` and accept that the partition becomes unavailable until an in-sync replica returns. The correct response to "no ISR available" for regulated data is to wait and recover, not to promote a stale replica. You would rather page an operator than quietly drop a settlement instruction.

Q: What is KRaft mode and what does it replace?
A: KRaft (Kafka Raft) is Kafka's built-in consensus mechanism for storing and replicating cluster metadata, replacing the external ZooKeeper ensemble that older Kafka required. Metadata (topics, partitions, configs, ACLs, leader assignments) now lives in an internal `__cluster_metadata` log managed by a quorum of controller nodes using a Raft-based protocol. This removes a whole separate distributed system from the operational picture.

Q: As of which release was ZooKeeper removed, and is KRaft the default now?
A: Apache Kafka 4.0 (released 18 March 2025) removed ZooKeeper support entirely, and KRaft has been production-ready since 3.3 and is the default for new clusters — there is no ZooKeeper option in 4.0. So any cluster you stand up today is KRaft. Migration from ZooKeeper to KRaft had to be completed before upgrading to 4.0, since 4.0 cannot run in ZooKeeper mode at all.

Q: What is the metadata quorum (controller quorum) in KRaft?
A: It is a small set of controller nodes (typically 3 or 5) that run the Raft protocol to elect an active controller and replicate the metadata log among themselves, so cluster metadata survives controller failures. The active controller serves metadata changes; the others are hot standbys with up-to-date copies. Because metadata is now just another replicated Kafka log, controller failover is fast and recovery does not depend on an external ZooKeeper quorum.

Q: What does the `process.roles` config do in a KRaft cluster?
A: `process.roles` declares whether a node acts as a `broker`, a `controller`, or both (`broker,controller`) — the combined mode being what enables a single-binary, all-in-one cluster for development. In production you usually separate the roles so controllers are dedicated, but for a local lab one node with both roles is a complete cluster. This is why a single-binary KRaft broker in Docker Compose is the entire Phase-4 Kafka lab.

Q: Why are single-binary KRaft clusters now the norm for local development?
A: Because KRaft folds metadata management into Kafka itself, you no longer need to run ZooKeeper alongside the broker — one Kafka process with `process.roles=broker,controller` is a fully functional cluster. That halves the moving parts in a Compose file and removes a class of ZooKeeper-connectivity startup failures. The result is the recommended lab setup: one container, KRaft mode, no external coordinator.

Q: When formatting storage for a new KRaft cluster, what one-time step is required before first start?
A: You must generate a cluster ID and format the controller/broker storage with `kafka-storage.sh format` using that ID and the KRaft config, which writes the `meta.properties` and initial metadata records. Skipping it (or reusing an old data dir with a different cluster ID) causes the broker to refuse to start with a cluster-ID mismatch. In Compose this is usually done by an init step or the official image's auto-format on an empty volume.

Q: What is the fundamental rule about message ordering in Kafka?
A: Ordering is guaranteed only within a single partition, never across partitions of a topic. Records with the same key go to the same partition and are therefore strictly ordered relative to each other; records across different partitions can be interleaved arbitrarily on the consumer side. This single fact drives every partition-key design decision: if two events must be ordered, they must share a key.

Q: How does Kafka decide which partition a keyed record goes to by default?
A: For a record with a key, the default partitioner computes a hash of the key (murmur2) modulo the number of partitions, so the same key deterministically lands in the same partition. For a null-key record, the modern default uses sticky batching to spread load across partitions efficiently. The keyed-hash behaviour is what makes "same key, same partition, ordered" hold.

Q: Why is the partition key a data-model decision rather than just a performance tuning?
A: Because the key determines which records are co-located and therefore ordered, choosing the key decides what your system can guarantee about ordering — it is a correctness decision, not just throughput. Pick the wrong key and events that must be processed in order land in different partitions and get reordered. So the architect chooses the key from the domain's ordering requirements first, then checks the load distribution second.

Q: In the fund-orders example, why key by ISIN, and what does that guarantee?
A: Keying order events by ISIN sends every event for a given security to the same partition, guaranteeing that all orders for that ISIN are consumed in the exact sequence they were produced. That is the right guarantee when per-instrument ordering matters (e.g. processing buys and sells for one fund share class in order). It explicitly does not guarantee ordering across different ISINs, which is fine because cross-instrument ordering is rarely required.

Q: What is the risk of keying by ISIN if one security is extremely heavily traded?
A: All events for that hot ISIN map to one partition, so that single partition (and the one consumer owning it) becomes a hotspot while others are underused — partition skew that caps throughput and grows lag on just that key. The ordering guarantee is preserved but capacity is wasted. Mitigations include increasing partition count, or composite keys for sub-streams where strict per-ISIN order is not actually required.

Q: If you increase a topic's partition count later, what happens to keyed ordering?
A: Adding partitions changes the key-to-partition mapping (hash modulo a new partition count), so records for a given key produced after the change may land in a different partition than before, breaking the contiguous per-key history. Existing records do not move, so a key's stream is now split across an old and a new partition with no global order between them. This is why partition count is chosen up front for ordering-sensitive topics rather than scaled casually.

Q: Why can you not reduce the number of partitions on a Kafka topic?
A: Kafka does not support decreasing partition count because it would require moving or merging existing records and would scramble the key-to-partition mapping and per-partition offsets that consumers rely on. The only supported change is increasing partitions. The practical consequence is that over-provisioning partitions is a one-way, hard-to-undo decision, so you size with future growth in mind but avoid wild over-partitioning.

Q: What is a compacted topic conceptually, and why call it "a table from a log"?
A: A compacted topic keeps only the most recent record per key, so reading it from beginning to end gives you exactly one current value for every key — the same shape as a key/value table. The log is still the storage, but compaction makes it equivalent to a materialised snapshot of latest-per-key. This is why a changelog topic keyed by entity ID becomes a queryable reference table you can rebuild state from.

Q: What `cleanup.policy` value turns a topic into a compacted one, and what guarantee does compaction give about the latest value?
A: You set `cleanup.policy=compact`. Compaction guarantees that the latest value for each key is always retained (it is never removed by the compaction process), while older values for the same key are eventually garbage-collected. It does not guarantee removal of every duplicate immediately — a consumer may still see some superseded values in the not-yet-compacted "head" of the log — but it guarantees the head plus compacted tail always contains the current value per key.

Q: What is a tombstone in a compacted topic and what does it do?
A: A tombstone is a record with a non-null key and a null value; in a compacted topic it signals that the key should be deleted, and after compaction the key's data is removed entirely. It is how you express "this entity no longer exists" in a changelog-as-table. Consumers must handle null values specially, treating them as deletes rather than as a value of the key.

Q: How long is a tombstone retained before compaction removes it, and which config controls that?
A: A tombstone is kept for at least `delete.retention.ms` (default 24 hours / 86400000 ms) after it becomes eligible, so that all consumers — including ones reading the compacted tail — have a chance to observe the delete before it disappears. This delay prevents a consumer rebuilding state from missing the delete and resurrecting an old value. After that window, compaction can remove the tombstone and the key is gone.

Q: What does `min.cleanable.dirty.ratio` control and what is its default?
A: It sets the minimum ratio of "dirty" (un-compacted) bytes to total bytes in the log before the cleaner will compact that partition, defaulting to 0.5 — meaning compaction kicks in once roughly half the log is un-compacted. A higher ratio compacts less often (less CPU/IO, more duplicates linger); a lower ratio compacts more aggressively. If you expect quick deduplication and it's not happening, this ratio and the cleaner thread count are the first things to check.

Q: Why does a compacted topic still let a consumer see more than one value per key sometimes?
A: Because compaction only acts on closed segments that have crossed the dirty-ratio threshold; the active segment and recently written, not-yet-cleaned data (the "dirty" head of the log) can contain multiple values for a key. Compaction is a background, eventually-consistent process, not a per-write deduplication. So a state-building consumer must always take the last value it reads per key, never assume exactly one.

Q: In the lab, you write several updates per ISIN to `fund-reference` and expect only the latest to survive — what must hold for the test to actually demonstrate compaction?
A: The cleaner must have run on closed segments, which means you need enough data (or a forced segment roll) to cross `min.cleanable.dirty.ratio` on closed segments, plus time for the cleaner thread. If everything is still in the active segment you'll see all versions and wrongly conclude compaction failed. The reliable demonstration consumes from the beginning after the cleaner has compacted, then confirms exactly one (latest) record per ISIN key survives.

Q: How would a downstream service "serve a table" from a compacted reference topic?
A: It consumes the compacted topic from the beginning, applying each record into a local key/value store (last value per key wins, tombstones delete), producing an up-to-date materialised view of current reference data. Because the latest value per key is guaranteed retained, the service can rebuild its entire state by replaying the topic after a restart. This is the changelog-to-table pattern that backs things like Kafka Streams' KTables.

Q: For fund reference data (e.g. ISIN-to-instrument mappings), why is a compacted topic a good fit?
A: Reference data is naturally keyed (by ISIN) and you only ever care about the current value, not the full history of changes, so compaction keeps storage bounded while always serving the latest mapping per ISIN. Any consumer can rebuild the full reference table by replaying the topic from offset 0. Deletes (a delisted instrument) are clean tombstones, and time-travel is unnecessary, which is exactly the trade compaction makes.

Q: What is Redpanda and in what sense is it Kafka-compatible?
A: Redpanda is a streaming platform that implements the Kafka API wire protocol, so existing Kafka clients, Connect, and tooling talk to it unchanged — it is a drop-in broker, written in C++ with no JVM and no ZooKeeper, aiming for lower latency and simpler ops. Compatibility is at the protocol level: producers/consumers and the admin API behave like Kafka's. It is positioned as a lighter local broker and a managed-cloud alternative for the same workloads.

Q: Where does Kafka-API compatibility typically "end" with an alternative like Redpanda?
A: Compatibility covers the core produce/consume/admin protocol, but ecosystem internals and JVM-specific extensions can differ: things like the exact internals of the JVM broker, some specific configs, certain Kafka Streams/transactional edge behaviours, and broker-side plugins may not map one-to-one. The point for an architect is to validate the specific features you depend on (transactions, compaction semantics, specific Connect transforms) against the alternative rather than assuming full parity. Wire-protocol compatible is not the same as feature-for-feature identical.

Q: What is the Azure Event Hubs "Kafka surface" and how do clients use it?
A: Event Hubs exposes a Kafka-protocol endpoint so Kafka producers and consumers can connect to an Event Hubs namespace by changing only the bootstrap servers and credentials, treating an event hub like a topic and a consumer group like a Kafka group. It is a managed PaaS broker speaking enough of the Kafka protocol to accept standard clients. This lets a team reuse Kafka client code while Microsoft operates the cluster.

Q: Name concrete features the Event Hubs Kafka surface does NOT support that pure Kafka does.
A: Notable gaps include Kafka Connect and Kafka Streams are not run by Event Hubs (and historically not all client features), you cannot add partitions to an existing event hub, log compaction and transactions are tier-limited (supported only in higher tiers/preview), and certain admin operations like size-based retention or the Kafka REST API are absent. So a design assuming in-place repartitioning, Streams, or universally-available compaction will hit walls. The architect checks tier and feature support before committing to the Kafka surface.

Q: Why does "add a partition to an existing topic" being unsupported on Event Hubs matter for design?
A: Because if you cannot increase partitions later, you must fix partition count at creation, and with keyed ordering you also cannot reshuffle keys afterward — so the capacity ceiling is locked in earlier and harder than on self-managed Kafka (where you at least can add partitions, with the ordering caveat). For a growing fund-data stream this means more careful up-front sizing. It is a portability gotcha: code that works on Kafka may assume an operation the managed surface forbids.

Q: An architect is choosing between self-hosted Kafka, Redpanda, and Event Hubs for a fund platform — what is the right framing?
A: Frame it as a build-vs-buy and feature-dependency decision: Event Hubs trades operational control for a managed surface with known feature gaps (compaction/transactions/partition-add limits by tier); Redpanda offers lighter self-hosted ops with broad but not total parity; self-hosted Kafka gives full feature control at full operational cost. You list the features you actually depend on (EOS transactions, compaction, Connect, in-place repartition) and check each candidate against them. This becomes the ADR-010 log-platform decision with the EU/fund constraint that drove it.

Q: A producer is configured `acks=1` — can an acknowledged order event still be lost? When?
A: Yes — with `acks=1` the producer is told the write succeeded as soon as the leader persists it, before followers replicate, so if that leader broker fails before any follower copies the record, the acknowledged record is lost. This is the silent-loss trap behind a config that "looked fine" in testing. For settlement-grade events you need `acks=all` with `min.insync.replicas>=2` so an acknowledged record always exists on multiple brokers.

Q: With auto-commit enabled and at-least-once intent, where does a duplicate enter on the consumer side?
A: If the consumer commits offsets (auto-commit fires) after fetching but before fully processing, then crashes, the restarted consumer resumes past records it never finished — that's a loss. Conversely, committing after processing but crashing before the commit lands causes reprocessing of the last batch — that's a duplicate. Auto-commit ties the commit to a timer rather than to processing completion, which is why precise control needs manual commit-after-process.

Q: How do you make a consumer at-least-once with no loss using manual commits?
A: You disable auto-commit (`enable.auto.commit=false`) and call commit only after the record's downstream effect has fully succeeded, so a crash before commit causes reprocessing (a duplicate) rather than skipping (a loss). At-least-once accepts duplicates and eliminates loss. To then eliminate the duplicate's visible effect you make the downstream sink idempotent (e.g. upsert by order ID), giving effectively-once results.

Q: "Can we lose or duplicate an order event?" — what config matrix do you reason over to answer it correctly every time?
A: You reason over four axes: producer `acks` and idempotence (controls duplicates/loss on send), `min.insync.replicas` plus replication factor and unclean-leader setting (controls loss on broker failure), the consumer's commit strategy (auto vs manual-after-process, controls duplicate/loss on consumer crash), and the sink's idempotency (whether duplicates have visible effect). Loss enters with `acks` too weak, ISR too small, unclean election on, or committing before processing; duplicates enter with retries without idempotence or reprocessing into a non-idempotent sink. Walking those four lets you answer for any given config.

Q: What does "exactly-once" mean precisely in Kafka, and what is its scope?
A: In Kafka, exactly-once semantics (EOS) means the idempotent producer plus transactions plus `read_committed` consumers guarantee that a consume-process-produce loop's writes and offset commits happen atomically and without duplicates — but only within Kafka. It does not extend to an external sink unless that sink participates transactionally or is idempotent. So "Kafka EOS" is a within-broker claim; an end-to-end exactly-once claim into a database needs additional machinery.

Q: A trade-events consumer keeps rebalancing every couple of minutes and never makes progress — first things to check?
A: First check whether per-record/per-batch processing time exceeds `max.poll.interval.ms`, because slow processing makes the coordinator evict the consumer mid-batch and trigger a rebalance loop. Also check `session.timeout.ms`/heartbeat health and whether a single poison record is making one record take forever. The fix is usually to reduce `max.poll.records`, speed up or offload processing, or raise `max.poll.interval.ms` — and to use the cooperative-sticky assignor so rebalances are less destructive.

Q: After a broker restart, one partition stays under-replicated for a long time — what does that mean and what do you check?
A: Under-replicated means the partition's ISR has fewer members than its replication factor, so a follower has not caught up to the leader — durability is temporarily reduced. You check the lagging follower broker's health (disk IO, network, GC pauses), `replica.lag.time.max.ms`, and whether the follower is fetching at all. Until the ISR is restored, `acks=all` writes are riskier and may be rejected if the ISR drops below `min.insync.replicas`.

Q: A consumer's committed offset is reported as `-1` or "unknown" for a partition — what does that indicate?
A: It indicates the group has no committed offset for that partition yet (a brand-new group, or offsets aged out of the `__consumer_offsets` topic via `offsets.retention.minutes`), so on next poll the consumer applies `auto.offset.reset`. This is why a long-idle group can mysteriously "lose its place" and reprocess or skip depending on the reset policy. For critical consumers you keep them active and/or set retention so offsets are not expired.

Q: Where are consumer-group offsets stored in modern Kafka?
A: They are stored in an internal compacted topic called `__consumer_offsets`, keyed by group/topic/partition, so the latest committed offset per group-partition is what survives compaction. Storing offsets in a Kafka topic (rather than ZooKeeper as in very old versions) makes them replicated and durable like any other data. This is itself an example of the changelog-as-table pattern Kafka uses internally.

Q: What does the `kafka-consumer-groups.sh --describe` command show you and why is it your go-to?
A: It lists, per partition for a group, the current committed offset, the log-end offset, the lag (the difference), and which consumer/member owns the partition. It is the go-to because it answers the two operational questions at once: are we behind (lag) and who owns what (assignment), letting you spot skew, stuck members, and unassigned partitions. Example: `kafka-consumer-groups.sh --bootstrap-server localhost:9092 --describe --group nav-flow`.

Q: What does it mean if a partition in `--describe` output shows lag but no assigned consumer (CONSUMER-ID is "-")?
A: It means that partition currently has no live group member consuming it — typically mid-rebalance, or the group has fewer consumers than partitions and this one is genuinely unassigned, or all consumers died. Its lag will grow until a consumer picks it up. If it stays unassigned with a running group, you check whether consumers crashed or whether the assignor failed, because un-consumed partitions silently fall behind.

Q: How does a compacted topic differ from a retention-based topic when a consumer replays from offset 0?
A: Replaying a retention-based topic from 0 gives the full history that still exists within the retention window — every change in order, but bounded in time. Replaying a compacted topic from 0 gives the latest value per key (plus possibly a few un-compacted recent duplicates) — a current-state snapshot, not full history. So you pick retention when you need the event history and compaction when you need only current state per key.

Q: What is the difference between a topic's `retention.bytes` and `segment.bytes`?
A: `retention.bytes` is the maximum total size a partition may grow to before old segments are deleted (a cleanup threshold), while `segment.bytes` is the size at which the active segment rolls to a new file (a file-chunking threshold). They operate at different granularities: segment size controls how data is chunked, retention size controls how much chunked data is kept. Cleanup deletes whole segments, so retention is enforced in segment-sized steps, not byte-exact.

Q: A topic was created with one partition and you now realise order events for many ISINs share it — what is the consequence and the fix?
A: With one partition, every ISIN's events are in the same partition and consumed by a single consumer — you have global ordering but zero parallelism, so throughput is capped and one slow record stalls everything. The fix is to recreate (or expand) the topic with more partitions keyed by ISIN so per-ISIN ordering is preserved while different ISINs spread across partitions and consumers. Note expanding partitions changes future key placement, so for strict per-key history you may need a fresh topic and a migration.

Q: What is the purpose of the producer `linger.ms` and `batch.size` settings?
A: They control batching: `batch.size` is the max bytes per partition batch and `linger.ms` is how long the producer waits to fill a batch before sending, trading a little latency for much better throughput and compression. Larger batches mean fewer requests and higher throughput; `linger.ms=0` sends as soon as possible for lowest latency. For a high-volume fund-event stream, a small `linger.ms` plus compression markedly improves broker efficiency.

Q: What is the broker's role in producer idempotence — what does it track per producer?
A: The broker tracks, per partition, the producer ID (PID) and the last sequence number it accepted from that PID; it accepts a batch only if its sequence is exactly the next expected one, rejecting duplicates (already-seen sequence) and gaps (out-of-order). This is how a retried batch with the same PID and sequence is dropped rather than written twice. It is server-side deduplication scoped to a single producer session.

Q: Why does an idempotent producer's guarantee not survive across a producer restart (without transactions)?
A: Because a plain idempotent producer gets a new producer ID on restart, the broker's per-PID sequence tracking starts fresh, so records resent by application logic after a restart are not recognised as duplicates of the old session's records. Idempotence dedupes within a single producer session only. To get cross-session guarantees you use transactions with a stable `transactional.id`, which fences the old session and gives a recoverable identity.

Q: What does the `transactional.id` enable beyond a plain producer ID?
A: A stable `transactional.id` lets a restarted producer reclaim its transactional state and "fence" any zombie instance of itself, so a crashed-and-restarted producer cannot have an old instance still committing — exactly-once survives process restarts. The broker maps the `transactional.id` to a producer epoch and bumps the epoch on re-init, rejecting writes from the stale epoch. This is what makes transactional EOS robust to crashes, unlike bare idempotence.

Q: What is "fencing" in the context of Kafka transactions?
A: Fencing is the mechanism that prevents a zombie producer (an old instance that hung and came back) from corrupting a transaction: when a new producer initialises with the same `transactional.id`, the broker increments the producer epoch and rejects any writes from the older, now-fenced epoch. It guarantees only one live producer per transactional ID can commit. Without fencing, a split-brain producer could write duplicate or interleaved transactional data.

Q: A compacted `fund-reference` topic shows two records for the same ISIN to a consumer — is this a bug?
A: Not necessarily — compaction is eventual, so a consumer reading the un-compacted head of the log can legitimately see an older and a newer value for the same key before the cleaner runs. A correct consumer applies last-value-wins per key, so seeing two is fine as long as the final one is current. It would only be a real problem if compaction never reduced closed segments, which would point at the cleaner being disabled or the dirty ratio never being crossed.

Q: Why must a state-building consumer of a compacted topic process the entire topic, not just the tail?
A: Because the current value of a key could have been written long ago and never updated since, so it lives deep in the compacted log; reading only recent records would miss keys whose latest value is old. To materialise the full table you must read from offset 0 to the end. This is the standard bootstrap step when a service rebuilds its KTable-like view after a restart.

Q: How does Kafka's per-partition ordering interact with multiple producers writing the same key?
A: Records for the same key always go to the same partition, but if two different producers write that key, the partition orders them by arrival at the leader, not by any global clock — so cross-producer order reflects broker receipt order. Within one producer, idempotence preserves send order. So "same key, ordered" means ordered as the leader received them, which is well-defined per partition but not synchronised to wall-clock across producers.

Q: What problem would you hit using Kafka for synchronous request/reply between two services?
A: Kafka's consumer-group model is built for streaming fan-out and replay, not point-to-point correlation, so request/reply forces you to bolt on correlation IDs, reply topics, and consumer-group gymnastics that a queue gives natively — you fight the model. Latency and per-message acknowledgement semantics also don't fit cleanly. This is the classic "use the log for the wrong job" mistake; a queue/broker (or RPC) is the right tool for request/reply.

Q: Why is a replayable log, not a queue, the right backbone for an audit-and-reprocessing fund integration?
A: Because a log retains records after consumption and lets new or rewound consumers replay history from any offset, you can re-derive downstream state, onboard a new consumer against the full past, and give auditors a reproducible record — none of which a destructive queue offers. The data is the system of record for "what events occurred," not just transient transport. That replay capability is exactly what regulated reprocessing and lineage depend on.

Q: What does `max.poll.records` control and why tune it?
A: It caps how many records a single `poll()` returns to the consumer, bounding the work done before the next poll and heartbeat-progress check. Tuning it down keeps per-poll processing under `max.poll.interval.ms` to avoid rebalance eviction during slow processing; tuning it up improves throughput when processing is fast. It is the first lever when slow processing causes rebalance loops.

Q: How does Kafka achieve high write throughput at the storage layer?
A: Kafka appends sequentially to the active segment (sequential disk IO is far faster than random), relies on the OS page cache rather than a custom in-process cache, and uses zero-copy (`sendfile`) to ship bytes from page cache to socket without copying through user space. Batching and compression further cut per-record overhead. The append-only log design is what makes this sequential, cache-friendly pattern possible.

Q: What is the practical meaning of "the log is the source of truth" in an event-driven fund platform?
A: It means downstream stores (the lakehouse copy, materialised views, read models) are derived projections of the event log, and the log — not any one database — is the authoritative ordered record of what happened. If a projection is corrupted or you need a new view, you rebuild it by replaying the log. This inverts the usual database-as-truth model and is what gives event-driven systems their auditability and reprocessability.

Q: Why is replication factor a per-topic decision rather than cluster-wide?
A: Because different topics have different durability and storage needs — a critical settlement topic may want replication factor 3 while a high-volume, low-value debug topic may tolerate 1 or 2 — Kafka sets `replication.factor` per topic at creation. This lets you spend replication storage where durability matters. The architect maps each topic's regulatory/business importance to a replication factor rather than applying one blanket value.

Q: If `replication.factor=1`, what durability do you have and when is that acceptable?
A: With replication factor 1 there is a single copy per partition, so any broker failure loses that partition's data entirely and makes it unavailable — there is no durability against broker loss. It is acceptable only for truly disposable data (transient test, regenerable cache) where loss costs nothing. It is never acceptable for trade, NAV, or settlement events, where you use at least 3.

Q: What is the difference between a consumer "subscribing" to a topic versus "assigning" specific partitions?
A: `subscribe()` joins a consumer group and lets Kafka dynamically assign partitions and rebalance them across members; `assign()` manually pins specific partitions to the consumer with no group coordination or rebalancing. Subscribe is for scalable, fault-tolerant group consumption; assign is for cases needing precise control (e.g. a single reader replaying one partition). Mixing them or using assign without understanding gives up automatic failover.

Q: How would you replay just the last hour of a topic for reprocessing?
A: Use offsets-for-times to look up the offset corresponding to a timestamp one hour ago in each partition, then seek the consumer to those offsets and consume forward — Kafka indexes records by timestamp per segment to support this. Tools like `kafka-consumer-groups.sh --reset-offsets --to-datetime` do the same for a group. This timestamp-to-offset capability is what makes "reprocess from time T" practical for corrections and backfills.

Q: A NAV-flow consumer must not double-count an order if it reprocesses — how do you make reprocessing safe end-to-end?
A: You make the downstream effect idempotent (e.g. upsert keyed by order ID, or dedupe on event ID) so that consuming the same record twice produces the same final state — at-least-once delivery plus an idempotent sink equals effectively-once. Reprocessing then becomes safe by construction, which is what lets you replay for corrections without inflating totals. Relying on "we won't reprocess" is not a guarantee; idempotent sinks are.

Q: What does a Connect/Flink "two-phase-commit sink" add over a plain idempotent consumer?
A: A two-phase-commit (transactional) sink coordinates the external write with the stream processor's checkpoint/transaction: it pre-commits (phase 1) and only finalises (phase 2) when the checkpoint/transaction commits, so the external effect and the processing position are made durable together. This gives true exactly-once into the external system, not merely effectively-once via dedup. It is heavier than an idempotent upsert but is the honest answer when the sink cannot be made idempotent.

Q: In a design review, how do you justify "3 partitions" for `fund-orders` rather than 1 or 30?
A: You justify it from the ordering and throughput requirements: enough partitions to spread expected load across that many consumers in parallel and to allow growth, but not so many that you waste resources, complicate ordering, and bloat metadata. With per-ISIN keying, partitions only help if many ISINs spread across them; 3 gives modest parallelism for a lab while preserving per-ISIN order. The defensible answer cites measured/expected throughput per partition and the consumer parallelism you need.

Q: What is the metadata cost of having an enormous number of partitions in a cluster?
A: Each partition consumes broker resources — open files, memory for indexes, replication connections, and an entry in the metadata log — so very high partition counts slow leader elections, lengthen rebalances and recovery, and inflate the controller's metadata. KRaft improved metadata scalability over ZooKeeper, but partitions are still not free. So you provision for parallelism and growth, not maximalism.

Q: An auditor asks you to prove no settlement order was lost during a broker outage — what configuration and evidence do you point to?
A: You point to `acks=all` with `min.insync.replicas=2`, replication factor 3, and `unclean.leader.election.enable=false`, which together guarantee every acknowledged order existed on at least two replicas and that no stale replica was promoted to lose committed data. As evidence you show the producer config, the topic config, broker logs showing the ISR never dropped below the minimum (and producers were rejected, not silently degraded, if it had), and a consumer replay confirming completeness. The configuration is the design decision; the logs and replay are the proof.

Q: Summarise the exact trade `min.insync.replicas` plus `acks=all` makes — what it costs and what it protects against.
A: It costs availability and latency: writes wait for multiple replicas and are rejected outright when too few are in sync, so the partition can become temporarily unwritable. In return it protects against silent data loss — every acknowledged record provably lives on at least `min.insync.replicas` brokers, so no single (or, with RF3/ISR2, even some multi-) broker failure can lose committed data. For regulated fund events that is the right trade: refuse the write rather than risk losing a settlement instruction.

Q: On your native Ubuntu lab, which script inspects a partition's raw segment and index files, and what flag prints the actual record payloads?
A: You use `kafka-dump-log.sh` (the launcher for `kafka.tools.DumpLogSegments`), pointing `--files` at the `.log`/`.index` files under the partition's data directory, and add `--print-data-log` to print the message contents rather than just metadata. It is the tool for proving on disk that compaction removed old versions or that records landed in the expected partition. Example: `kafka-dump-log.sh --files /var/lib/kafka/data/fund-orders-0/00000000000000000000.log --print-data-log`.

Q: How do you list only the under-replicated partitions across a cluster in one command, and why is that your first triage step after a broker hiccup?
A: Run `kafka-topics.sh --bootstrap-server localhost:9092 --describe --under-replicated-partitions`, which prints only partitions whose ISR is smaller than their replication factor instead of dumping every topic. It is the fastest way to see whether durability is currently degraded and on which brokers, before lag or producer errors even surface. A companion `--unavailable-partitions` shows partitions with no live leader at all, which is a more severe outage.

Q: Your idempotent producer throws `OutOfOrderSequenceException` — what does it mean and what is the safe response?
A: It means the broker saw a gap between the next sequence number it expected and the one the producer sent (e.g. expected 11 but got 13), which signals that an acknowledged record was lost or the producer's state is corrupt. The safe response is to treat it as fatal: close the producer and recreate it rather than continuing, because continuing risks reordering or silent loss. For settlement events you also alert, since on a properly configured `acks=all`/`min.insync.replicas=2` topic this should not happen in steady state.

Q: A producer intermittently fails with `NOT_ENOUGH_REPLICAS` during normal trading hours — what is happening and where do you look first?
A: That error means an `acks=all` produce was rejected because the partition's ISR had shrunk below `min.insync.replicas`, so a follower is repeatedly falling out of sync. First check broker health for the follower replica — disk IO saturation, network, or GC pauses — and the `replica.lag.time.max.ms` window, because a flapping ISR points at an overloaded or unstable broker, not a producer bug. The producer is correctly refusing to write under-replicated data, so fixing the broker restores both the ISR and the writes.

Q: You expanded `fund-orders` from 3 to 6 partitions and immediately some consumers in the group sit idle — why, and what resolves it?
A: A partition-count change triggers a rebalance, and depending on timing and the assignor a member may briefly hold no partitions while the new assignment settles; if it stays idle the group simply has more partitions than before so capacity exists but assignment is mid-flight. It resolves once the cooperative-sticky rebalance completes and the six partitions spread across members. The deeper gotcha is that new keyed records now hash modulo 6, so per-ISIN history is split across an old and a new partition from this point on.

Q: How do you reset a consumer group to the beginning of every partition without writing code, and what state must the group be in?
A: Use `kafka-consumer-groups.sh --bootstrap-server localhost:9092 --group nav-flow --reset-offsets --to-earliest --all-topics --execute`, which rewrites the group's committed offsets to each partition's log-start offset. The group must have no active members — the command refuses to run while consumers are connected, because moving offsets under a live consumer would be ambiguous. Add `--dry-run` first to preview the offsets it would set before committing with `--execute`.

Q: What is the difference between `--reset-offsets --to-earliest` and `--to-datetime` for a replay?
A: `--to-earliest` rewinds to each partition's current log-start offset (oldest record still retained), giving a full available-history replay, while `--to-datetime 2026-06-14T00:00:00.000` uses Kafka's per-segment timestamp index to seek each partition to the first record at or after that instant. You pick datetime for "reprocess from time T" corrections and earliest for a complete rebuild. Datetime depends on record timestamps being meaningful, so a topic with broker-assigned `LogAppendTime` versus producer `CreateTime` will seek differently.

Q: A consumer group's `--describe` shows negative lag (committed offset greater than log-end offset) — is that possible and what would cause it?
A: It is not a normal steady state and usually means the group's committed offsets were reset forward past current data, the topic was recreated/truncated under the group, or you are reading a stale view mid-rebalance. The fix is to inspect whether someone ran a `--reset-offsets --to-latest` or recreated the topic, then reset the group cleanly. Treating negative lag as "caught up" is a trap — it signals the offset bookkeeping no longer matches the log.

Q: Why does `CreateTime` versus `LogAppendTime` for `message.timestamp.type` matter for time-based replay and retention?
A: `CreateTime` stamps each record with the producer's clock (the event time), while `LogAppendTime` overwrites it with the broker's clock at append; retention by age and `--to-datetime` seeks both use whichever timestamp the topic stores. With `CreateTime`, a producer with a skewed clock or a late-arriving backfill can make retention and timestamp seeks behave unexpectedly. For fund event ordering you usually want `CreateTime` to preserve true event time, accepting that out-of-order producer clocks are then your problem to handle downstream.

Q: In the lab you produce two records with the same ISIN key but see them in different partitions — what almost certainly went wrong?
A: Most likely the producer sent a null key (or a serializer turned the key into bytes inconsistently), so the default partitioner used sticky/round-robin spreading instead of key-hashing, scattering same-ISIN records across partitions. Check that the key is actually non-null and that the key serializer matches what you think you are sending. The fix is to ensure the ISIN is passed as the record key, since only a consistent non-null key gives the same-key-same-partition guarantee.

Q: How would you prove, on disk, that records for one ISIN all went to a single partition?
A: Consume with the partition exposed (or dump each partition's log with `kafka-dump-log.sh --print-data-log`) and confirm every record for that ISIN key appears in exactly one partition's `.log` file and none in the others. This turns the abstract "same key, same partition" claim into a reproducible artifact you can show in a review. It is also how you debug an apparent ordering violation: if the key is split across partitions, the dump reveals it immediately.

Q: What does `unclean.leader.election.enable=true` look like in practice when it fires — what symptom would an operator notice?
A: When it fires, a partition that had no in-sync replica suddenly gets a leader again (availability returns), but the new leader's log-end offset is lower than the old leader's was, so consumers see the high-water mark move backwards and some recently-acknowledged records simply vanish. An operator notices a partition that was unavailable becoming available together with a drop in its end offset and gaps reported downstream. That silent truncation is exactly why it is `false` by default and stays `false` for trade events.

Q: How do you turn off auto-topic-creation so a typo in a topic name does not silently create a junk topic, and why does this matter in a regulated estate?
A: Set the broker config `auto.create.topics.enable=false`, so a produce or consume against a non-existent topic fails instead of quietly materialising an unmanaged, unreplicated topic. In a regulated estate every topic should be provisioned deliberately with a known replication factor and retention; an auto-created topic inherits broker defaults that may not meet durability or compliance requirements. The failure is the desired behaviour — it forces topics through your change process.

Q: What is the `__transaction_state` internal topic and what does it store?
A: `__transaction_state` is the compacted internal topic where the transaction coordinator persists each `transactional.id`'s current state — its producer epoch, the partitions enrolled in the open transaction, and whether it is ongoing, committing, or aborting. It is what lets a coordinator recover transaction status after a restart and fence zombie producers by epoch. Like `__consumer_offsets`, it is a Kafka-as-its-own-database example, replicated and compacted for durability.

Q: A long-running transaction never commits and `read_committed` consumers stall — what bounds how long this can hang?
A: The transaction is bounded by the producer's `transaction.timeout.ms` (capped by the broker's `transaction.max.timeout.ms`); once it expires the coordinator aborts the transaction, which advances the Last Stable Offset and unblocks `read_committed` consumers. Until then those consumers will not deliver records past the open transaction's first record. So a hung producer cannot stall committed consumers forever, but it can for up to the timeout — which is why you keep transactions short.

Q: Why can two consumers in the same group never both read the same partition, and what concrete failure does that prevent?
A: The group protocol assigns each partition to exactly one member so that per-partition offset commits are unambiguous and ordering within the partition is preserved by a single reader. If two members read one partition they would commit conflicting offsets and process records twice, breaking both at-least-once accounting and ordering. To get two independent readers of the same data you use two different `group.id` values, which is Kafka's fan-out, not a within-group split.

Q: A new microservice must read the full history of `fund-orders` without disturbing the existing `nav-flow` consumers — how do you set it up?
A: Give it a brand-new `group.id` and start with `auto.offset.reset=earliest` (or explicitly seek to offset 0), so it forms its own group that reads an independent copy of the stream from the beginning. Because groups are isolated, its offsets live separately in `__consumer_offsets` and do not touch `nav-flow`'s position or lag. This is the standard "add a consumer" pattern that the replayable-log design makes free, unlike a destructive queue.

Q: What happens to a consumer group's committed offsets if no member is active for longer than `offsets.retention.minutes`?
A: The `__consumer_offsets` records for that group can be removed once the group is empty and the retention window elapses, so when a consumer rejoins it finds no committed offset and falls back to `auto.offset.reset` — reprocessing from earliest or jumping to latest. This is how a long-idle group "loses its place." For critical consumers you either keep them running, lengthen the retention, or persist offsets externally so an outage does not silently skip or replay data.

Q: How does `enable.auto.commit=true` with `auto.commit.interval.ms` create a window for both loss and duplication?
A: Auto-commit periodically commits the latest polled offsets on a timer independent of whether processing finished, so if a commit fires after fetch but before processing completes and the consumer then crashes, the skipped records are lost; if processing completes but the consumer crashes before the next timer commit, those records are reprocessed as duplicates. The timer, not the work, drives the commit. Precise delivery therefore requires `enable.auto.commit=false` and committing only after the downstream effect succeeds.

Q: Walk through the exact sequence that yields a duplicate when using manual commit-after-process.
A: The consumer polls a batch, fully applies each record's downstream effect, and is about to commit when it crashes before the commit lands; on restart it resumes from the last committed offset and reprocesses that batch — the same records are applied twice. This is at-least-once: no record is skipped, but the last in-flight batch can repeat. You neutralise the visible duplicate by making the sink idempotent (upsert by order ID), turning at-least-once into effectively-once.

Q: Why is per-partition order preserved across producer retries when `enable.idempotence=true` even with several requests in flight?
A: Because the broker tracks the expected sequence number per producer-and-partition and will only accept the next-in-sequence batch, rejecting a retried duplicate or an out-of-order batch and forcing the client to resend in order. This lets up to five in-flight requests coexist without reordering, whereas a non-idempotent producer would need `max.in.flight.requests.per.connection=1` to be safe. So idempotence gives both no-duplicates and no-reordering without throttling throughput to one request.

Q: A teammate sets `acks=1` with `enable.idempotence=true` and is confused it gets rejected — why?
A: The idempotent producer requires `acks=all`, so explicitly setting `acks=1` alongside `enable.idempotence=true` is a contradictory configuration the client rejects (or it disables idempotence), because the broker's sequence-based dedup needs the stronger replication guarantee to be meaningful. You cannot have retry-safe exactly-once-into-the-partition while only waiting for the leader. The resolution is to leave `acks=all` (the default with idempotence) rather than weaken it.

Q: How would you measure end-to-end throughput and latency of your single-broker lab without writing a client?
A: Use the bundled `kafka-producer-perf-test.sh` to generate load (records, size, target rate) and `kafka-consumer-perf-test.sh` to measure consume throughput, both shipping with Kafka. They report records/sec, MB/sec, and latency percentiles so you can size partitions and batching empirically. Example: `kafka-producer-perf-test.sh --topic fund-orders --num-records 1000000 --record-size 200 --throughput -1 --producer-props bootstrap.servers=localhost:9092 acks=all`.

Q: Your producer's throughput is low and CPU is idle on both sides — which two producer settings do you reach for first?
A: Raise `linger.ms` from 0 to a few milliseconds and ensure `batch.size` is large enough to amortise per-request overhead, so the producer fills bigger batches before sending instead of one tiny request per record. Adding `compression.type` (e.g. `lz4` or `zstd`) compounds the gain by shrinking batches on the wire and disk. Low throughput with idle CPU is the classic under-batching signature, not a broker bottleneck.

Q: What does `compression.type` do, where is it applied, and what is the gotcha with a topic-level override?
A: Set on the producer, `compression.type` (e.g. `zstd`, `lz4`, `snappy`, `gzip`) compresses each record batch once on the client and the broker stores it compressed, so it cuts network, disk, and replication bytes together. The gotcha is the topic/broker-level `compression.type`: if it differs from `producer`, the broker may recompress batches on write, adding CPU; setting the broker side to `producer` keeps the producer's codec and avoids recompression. For high-volume fund streams `zstd` usually gives the best ratio.

Q: How do you cleanly increase a topic's replication factor on a running cluster, and why is it not just a config edit?
A: You generate a reassignment plan that lists new replica brokers for each partition and apply it with `kafka-reassign-partitions.sh --execute`, because raising replication factor means physically copying each partition's data to the added replica brokers, not just changing a number. A plain `--alter` of `replication.factor` is not how it is done; the reassignment tool drives the data movement and you can throttle it. Until the new replicas catch up and join the ISR, durability has not actually improved.

Q: What is the purpose of throttling a partition reassignment, and which knob controls it?
A: Reassignment copies large amounts of data between brokers, which can saturate network and disk and starve live produce/consume traffic, so you cap the replication bandwidth with the reassignment tool's `--throttle` (bytes/sec) to keep production traffic healthy. This trades a slower rebalance for protected latency on the running workload. On a busy fund cluster you reassign during quieter windows and throttle to avoid inducing ISR shrink on unrelated partitions.

Q: Why does a single-broker lab let you create a topic with `replication.factor=1` but fail at `replication.factor=2`?
A: Replication factor cannot exceed the number of available brokers, because each replica of a partition must live on a distinct broker; with one broker the only legal factor is 1. Asking for 2 fails with an error about insufficient brokers. This is why the single-binary KRaft lab demonstrates ordering, partitioning, and compaction but cannot demonstrate ISR/`min.insync.replicas` durability — that needs a multi-broker cluster.

Q: In KRaft, what is the `__cluster_metadata` topic and how does it differ from a normal topic?
A: `__cluster_metadata` is the internal single-partition Raft log where the controller quorum stores all cluster metadata — topic, partition, config, ACL, and leadership records — replicated among controllers by the Raft protocol rather than by the normal ISR replication used for data topics. It is the ZooKeeper replacement, made of the same log primitive Kafka already had. Its health is the cluster's control plane: if the quorum cannot make progress, metadata changes and leader elections stall.

Q: What does `kafka-storage.sh random-uuid` produce and where does that value go?
A: It generates a fresh cluster ID (a UUID), which you then pass to `kafka-storage.sh format --cluster-id <id> --config server.properties` to initialise the storage directory's `meta.properties` before the first start of a KRaft node. Every node in the same cluster must be formatted with the same cluster ID. Reusing an old data directory whose `meta.properties` holds a different ID causes a cluster-ID mismatch and the broker refuses to start.

Q: A KRaft broker refuses to start complaining about a cluster-ID mismatch — what happened and how do you fix the lab?
A: The data directory's `meta.properties` records a cluster ID that does not match what the node now expects — typically a reused volume from a previous `format`, or a Compose volume that outlived a re-init. For a disposable lab you stop the broker, delete the stale data volume, and re-run `kafka-storage.sh format` with the current cluster ID (or let the image auto-format an empty volume). In production you never just wipe it — you investigate why the ID changed.

Q: What is the practical difference between a topic with `cleanup.policy=delete` and one with `cleanup.policy=compact` when an entity is deleted?
A: On a `delete` topic, an entity removal is just another event that ages out with retention and leaves no special marker; on a `compact` topic you express deletion with a tombstone (non-null key, null value), and after `delete.retention.ms` compaction physically removes that key. So a compacted reference topic needs explicit tombstones to drop a delisted ISIN, whereas a retention topic simply stops carrying new events for it. Forgetting the tombstone leaves a stale key in the materialised table forever.

Q: Why might a compacted topic still grow unbounded despite compaction, and how do you bound it?
A: Compaction only deduplicates per key — it never drops a key whose latest value is a live (non-tombstone) record — so a topic that keeps accumulating new distinct keys grows without limit even when fully compacted. To bound it you add `delete` to the policy (`cleanup.policy=compact,delete`) so old data also ages out by time/size, or you tombstone keys you no longer need. A reference topic for an ever-growing universe of identifiers is the realistic case where this bites.

Q: How do you force compaction to actually run in the lab so you can demonstrate it, rather than waiting indefinitely?
A: You make closed segments cross the cleaner's threshold: lower `segment.ms` or `segment.bytes` so segments roll quickly, lower `min.cleanable.dirty.ratio` so the cleaner triggers sooner, write enough updates per key to dirty the log, then consume from offset 0 after the cleaner thread has run. If everything stays in the active segment the cleaner cannot touch it and you wrongly conclude compaction failed. The reliable demo confirms exactly one latest record per ISIN survives once closed segments are compacted.

Q: What is the head versus the tail of a compacted log, and which one can contain duplicate keys?
A: The tail is the already-compacted portion (closed segments the cleaner has processed) holding at most one record per key, while the head is the recently written, not-yet-compacted region (including the active segment) that can legitimately contain several values for the same key. A consumer therefore must always take last-value-wins per key, never assume the log holds exactly one record per key. Seeing duplicates in the head is normal, not a compaction bug.

Q: A state-building consumer of `fund-reference` restarts and a delisted ISIN reappears in its table — what likely went wrong with deletes?
A: The tombstone for that ISIN was probably compacted away before the consumer read it, because the consumer was offline longer than `delete.retention.ms`, so on replay it sees the old value but never the delete that would have removed it. The fix is to size `delete.retention.ms` larger than any expected consumer downtime so every state-builder observes the tombstone. This is precisely why tombstones are retained for a window rather than removed immediately.

Q: How does Kafka's zero-copy path actually move bytes, and what disables it?
A: For a consumer fetch, Kafka uses the `sendfile` system call to transfer record bytes directly from the OS page cache to the network socket without copying them into the JVM and back, which is a big part of its consume throughput. The optimisation is bypassed when the broker must touch the bytes — notably TLS encryption or broker-side recompression — because those require the data in user space. So enabling in-transit encryption trades some of the zero-copy speedup for confidentiality, a real consideration on a regulated cluster.

Q: Why does Kafka rely on the OS page cache instead of a large in-process JVM cache, and what operational consequence follows?
A: Kafka deliberately keeps a small heap and lets the OS page cache hold recently written/read log data, because sequential log access is highly cache-friendly and avoids GC pressure from a giant in-heap cache. The operational consequence is that broker performance depends heavily on free RAM for page cache, so you do not starve the host with other memory-hungry processes and you size memory for the working set. A consumer reading the tail is almost always served from page cache, not disk.

Q: What does `request.timeout.ms` versus `delivery.timeout.ms` bound on the producer side?
A: `request.timeout.ms` bounds how long the producer waits for a single broker response to one in-flight request before considering it failed and possibly retrying, while `delivery.timeout.ms` is the total end-to-end deadline from `send()` to success or failure, encompassing batching time, all retries, and request timeouts. `delivery.timeout.ms` is the one that ultimately decides "give up on this record." A retry storm that keeps hitting `request.timeout.ms` will still terminate at `delivery.timeout.ms`.

Q: A producer's `send()` callback reports a `TimeoutException` after the configured delivery timeout — was the record written or not?
A: It is ambiguous from the client's side: the record may have been persisted on the broker but the acknowledgement was lost, or it may never have been written, which is exactly the uncertainty that makes naive retries duplicate data. With the idempotent producer enabled, a retry is deduplicated by sequence number so this case is safe within the producer session; without it you must rely on an idempotent sink or accept possible duplicates. So you do not assume either outcome — you design the sink to tolerate the ambiguity.

Q: How does `acks=all` interact with `min.insync.replicas` to determine the exact moment a produce is acknowledged?
A: With `acks=all` the leader waits until all replicas currently in the ISR have replicated the record, but the write is only allowed to proceed at all if the ISR size is at least `min.insync.replicas`; if it is smaller the produce is rejected before acknowledgement. So acknowledgement means "persisted on the leader plus all current ISR members, and that set was at least the minimum." It is the conjunction of both settings, not either alone, that defines durability.

Q: Why is `replication.factor=3` with `min.insync.replicas=2` preferred over `replication.factor=2` with `min.insync.replicas=2` for settlement data?
A: With RF3/ISR2 you can lose one broker and still meet the minimum, so writes continue while durability stays strong; with RF2/ISR2 you have no slack — losing any one of the two replicas drops the ISR below the minimum and immediately blocks produces. RF3/ISR2 buys availability headroom on top of the same durability floor. That headroom is why three replicas is the standard safe choice for regulated event topics.

Q: What is the difference between the high-water mark and the log-end offset, and which one do consumers read up to?
A: The log-end offset is one past the last record the leader has appended, while the high-water mark is the highest offset that has been replicated to all ISR members and is therefore safe to expose; ordinary consumers can only read up to the high-water mark, not the log-end offset. Records between the high-water mark and the log-end offset exist on the leader but are not yet committed and stay invisible. This is what makes consumed data durable: you never read a record that could vanish if the leader fails.

Q: A consumer sees no new records even though a producer is actively writing to the topic — list the first three things to check.
A: First confirm the records are committed past the high-water mark (an under-replicated partition can hold writes invisible to consumers until the ISR catches up); second confirm the consumer is assigned the partitions the producer is writing to (a keyed producer may be hitting partitions this member does not own); third, for transactional producers, check whether the consumer is `read_committed` and blocked behind an open transaction at the Last Stable Offset. Each explains "writing but not reading" without any record being lost.

Q: How would you copy `fund-orders` from your on-prem lab cluster to a second cluster for DR, using Kafka's own tooling?
A: You run MirrorMaker 2 (built on Kafka Connect), which replicates topics, consumer-group offsets, and ACLs between clusters and can map topic names with a configurable prefix to avoid loops in active-active setups. It is the Kafka-native cross-cluster replication tool referenced for DR and migration. For a regulated fund estate the data-residency and which records cross which boundary become an explicit design and compliance decision, not just an ops detail.

Q: Why does MirrorMaker 2 rename topics with a source-cluster prefix by default, and when would you turn that off?
A: It prefixes replicated topics with the source cluster alias (e.g. `primary.fund-orders`) so that in active-active replication a record does not get mirrored back to its origin in an infinite loop, keeping each cluster's own and replicated topics distinguishable. You would use an `IdentityReplicationPolicy` (no prefix) for a one-way migration or simple DR where loop prevention is not a concern and you want identical topic names on the target. Choosing the policy is part of the cross-cluster design, not a default to accept blindly.

Q: A `read_committed` consumer's lag looks high but the producer is idle — what subtle cause should you suspect?
A: Suspect an open or hung transaction upstream: a `read_committed` consumer cannot advance past the Last Stable Offset, so even with no new records the apparent lag reflects committed data sitting behind an undecided transaction, not a slow consumer. Check for a long-running or stuck transactional producer and whether its `transaction.timeout.ms` will eventually abort it. This is invisible to a `read_uncommitted` consumer, which is a useful contrast to confirm the diagnosis.

Q: For the "can we lose or duplicate an order event" sign-off, what is the single safest end-to-end configuration you would write into the ADR?
A: Producer: `enable.idempotence=true` (hence `acks=all`) plus transactions with a stable `transactional.id` for multi-partition atomicity; topic: `replication.factor=3`, `min.insync.replicas=2`, `unclean.leader.election.enable=false`; consumer: `enable.auto.commit=false` with commit-after-process and `isolation.level=read_committed`; sink: idempotent (upsert by order ID). That stack gives no loss on broker failure, no duplicates from retries, and effectively-once into the sink, with the explicit cost that produces are refused when the ISR is too small. The ADR records both the guarantee and that deliberate availability trade.

Q: How do you create the `fund-orders` topic with 3 partitions and the safe durability settings in one CLI command on your Ubuntu lab?
A: Run `kafka-topics.sh --bootstrap-server localhost:9092 --create --topic fund-orders --partitions 3 --replication-factor 3 --config min.insync.replicas=2`, which sets partition count, replication factor, and the per-topic minimum ISR in a single call. The `--config` flag attaches topic-level overrides at creation so you do not rely on broker defaults. On the single-broker lab the `--replication-factor 3` part will fail, which is itself the lesson that durability settings need a multi-broker cluster to exercise.

Q: A consumer is stuck on one record that always throws when processed, blocking the whole partition — what is this called and how do you handle it?
A: That is a poison-pill record: because a partition is processed in order by one consumer, a record that always fails halts progress on that partition and lets lag grow indefinitely while the consumer retries forever. You handle it by catching the failure and routing the bad record to a dead-letter topic (then committing past it) so the stream continues, rather than blocking on it. For regulated data the dead-letter path must itself be auditable, since you are deferring, not discarding, a record that needs investigation.
