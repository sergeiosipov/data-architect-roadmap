# Phase 2 · 3.1.1 Cloud data warehouse — 150 self-test questions

Q: What are the three architectural layers of Snowflake, named exactly?
A: Snowflake separates into a database storage layer, a compute layer (virtual warehouses), and a cloud services layer. Storage holds the compressed columnar data, compute runs the SQL, and cloud services coordinates everything — authentication, metadata, query optimization, transaction management, and access control. The whole pitch is that these three scale independently, which is why you can resize compute without touching data.

Q: Why does separating storage from compute matter for cost in a warehouse like Snowflake?
A: Because you pay for the two resources independently: storage is a small, steady, data-shaped cost, while compute is the large, spiky, workload-shaped cost you switch on only when running queries. This is the core of explaining to a CFO why a quiet warehouse storing the same data costs almost nothing until someone runs a heavy month-end report. Legacy coupled systems forced you to over-provision a fixed cluster sized for peak, paying for it 24/7.

Q: In Snowflake, what is a "virtual warehouse"?
A: A virtual warehouse is an independent cluster of compute resources (CPU, memory, local SSD cache) that executes your SQL — it is the compute layer, not where data is stored. You can have many warehouses of different sizes running against the same data simultaneously, and each is isolated so one team's query does not slow another's. Sizing (XS, S, M, L, …) doubles the compute and the credit-per-hour rate at each step up.

Q: What is a Snowflake "credit" and how is warehouse compute billed?
A: A credit is Snowflake's unit of compute consumption; a running warehouse burns credits at a rate set by its size. Billing is per-second, but with a 60-second minimum each time a warehouse starts or resumes — so frequent start/stop churn on a large warehouse wastes money on minimum charges. After the first minute, you are billed per-second for as long as the warehouse runs continuously.

Q: What does the cloud services layer in Snowflake actually do?
A: It is the brain that coordinates the platform: authentication and access control, metadata management, query parsing and optimization, transaction and lock management, and infrastructure management. It is also what lets Snowflake prune micro-partitions using stored statistics before any compute runs. Most cloud-services usage is free, billed only when it exceeds 10% of daily compute credits.

Q: What is a micro-partition in Snowflake?
A: A micro-partition is the immutable, columnar unit of physical storage Snowflake automatically creates as data lands — each holds 50–500 MB of uncompressed data (smaller on disk because it is always stored compressed). Snowflake keeps per-column metadata (min/max, distinct counts, null counts) on each one, which is the basis of pruning. You never create or manage them directly; they are an automatic, internal structure.

Q: Why are Snowflake micro-partitions described as immutable, and what happens on an UPDATE?
A: They are immutable because Snowflake never edits a micro-partition in place; an `UPDATE` or `DELETE` writes new micro-partitions and marks the old ones as no longer current. This copy-on-write behavior is what makes Time Travel and zero-copy cloning possible — old versions still physically exist for the retention window. The gotcha is that heavy churn generates many new partitions and can hurt pruning until reclustering settles.

Q: How does Snowflake prune data without you defining explicit partitions?
A: It uses the min/max (and other) metadata stored per micro-partition: for a `WHERE` predicate, the optimizer checks each micro-partition's range and skips any that cannot contain matching rows, reading only the survivors. This is automatic — there is no `PARTITION BY` clause at table creation like in Hive-style systems. Effective pruning depends on the data being naturally or deliberately clustered on the filter columns.

Q: What is a clustering key in Snowflake and when do you need one?
A: A clustering key is a set of columns you designate so Snowflake co-locates rows with similar values into the same micro-partitions, improving pruning for queries that filter on those columns. You typically only need it on very large tables (multi-TB) where the natural load order does not match the query filter pattern. Automatic Clustering then maintains the ordering in the background as new data arrives, consuming credits to do so.

Q: How does Snowflake's clustering differ from explicit Hive-style partitioning?
A: Explicit partitioning physically splits a table into directories by a column value and you must choose the partition scheme up front, risking skew and small-file problems. Snowflake's clustering is a soft, statistical co-location of rows within automatically managed micro-partitions, with no directory structure and no query rewrite needed. The win is that you avoid the classic "wrong partition key" trap, but you trade it for clustering-maintenance credit cost.

Q: A Snowflake query on a 5 TB table scans the whole table despite a `WHERE trade_date = ...` filter — what is the first thing to check?
A: Check the table's clustering quality on `trade_date` using `SYSTEM$CLUSTERING_INFORMATION`, because poor clustering means the date is spread across most micro-partitions, defeating pruning. If the table was loaded in random order, every micro-partition's min/max range likely spans all dates, so none can be skipped. The fix is to define a clustering key on `trade_date` (or load in date order) so pruning becomes effective.

Q: What does it mean that Snowflake costs are "workload-shaped, not data-shaped"?
A: It means the dominant cost is driven by how much compute you run (warehouse size × time × concurrency), not by how many terabytes you store. You can store huge volumes cheaply and pay almost nothing while idle, but a single oversized warehouse left running, or a flood of month-end reports, spikes the bill. This is the exact line you give a CFO: the budget tracks query activity, so the lever is compute discipline, not data deletion.

Q: How does an idle Snowflake warehouse run up cost, and what setting prevents it?
A: A warehouse keeps burning credits for as long as it is in the running state, even with no queries executing, so one left on overnight is pure waste. The fix is `AUTO_SUSPEND` (e.g. 60 seconds) so it pauses after a short idle period, paired with `AUTO_RESUME` so it wakes on the next query. Beware setting auto-suspend too aggressively, since each resume incurs the 60-second minimum billing and loses warm cache.

Q: Why can a full table scan in Snowflake be a runaway-cost pattern?
A: A query with no selective predicate (or one that defeats pruning) forces compute to read every micro-partition, so a large warehouse churns credits proportional to the data scanned and the time it takes. On a multi-TB holdings table this turns a careless `SELECT *` or a non-sargable filter into a measurable budget hit. The defense is selective, sargable predicates on clustered columns plus right-sized warehouses.

Q: How does an unpartitioned/poorly clustered query specifically cause cost blow-ups in Snowflake?
A: Without effective clustering, the optimizer cannot prune micro-partitions, so even a narrow date filter reads the whole table, multiplying both scan volume and warehouse runtime. The same logical query that should touch 1% of the data ends up touching 100%, and you pay for all of it. Monitoring with `QUERY_HISTORY` for high `BYTES_SCANNED` relative to result size flags these offenders.

Q: What is BigQuery's storage layer called, and what is it separate from?
A: BigQuery stores table data in Colossus, Google's distributed file system, fully separate from the compute layer. This separation means storage scales and is billed independently of the slots that run queries, mirroring the storage/compute split Snowflake pioneered. You are charged for active vs long-term storage by bytes stored, regardless of whether any query is running.

Q: What columnar on-disk format does BigQuery use, and why does it matter?
A: BigQuery stores data in Capacitor, a proprietary columnar format that compresses well and supports column projection so a query reads only the columns it references. Columnar layout plus per-column statistics is what makes scanning a narrow set of columns over huge tables fast and cheap. It is the BigQuery analogue of Parquet's column-chunk design, but managed internally.

Q: What is the Dremel engine in BigQuery?
A: Dremel is the distributed execution engine that runs BigQuery queries by building a multi-level serving tree: a root node plans the query, mixer (intermediate) nodes aggregate, and leaf nodes (the slots) read and filter data in parallel from Colossus. This tree structure is what gives BigQuery massively parallel, serverless execution without you sizing a cluster. The architecture is why adding more slots can speed a query nearly linearly up to a point.

Q: What exactly is a BigQuery "slot"?
A: A slot is BigQuery's unit of compute — roughly a virtual CPU with associated memory — that does the scanning, filtering, and aggregation work in Dremel's leaf and mixer nodes. Slots are a fungible pool, not a fixed-size cluster you provision and keep; BigQuery allocates them to your queries dynamically. Query speed scales with how many slots a query can grab in parallel.

Q: Why is BigQuery called "serverless" while Snowflake is not, strictly speaking?
A: In BigQuery you never create, size, or manage a compute cluster — you submit SQL and Google's shared slot pool runs it, allocating compute transparently. In Snowflake you explicitly create and size virtual warehouses and decide when they run. Both separate storage from compute, but BigQuery hides the compute unit behind a serverless pool whereas Snowflake exposes it as a named, sized resource you control.

Q: What is BigQuery's on-demand pricing model, and what is the per-TB rate concept?
A: On-demand pricing charges by bytes scanned (read) by the query — historically around $6.25 per TB after the first 1 TB free per month — independent of how long the query runs. Because you pay for data read, not time, a poorly written query that scans extra columns or rows directly costs more money. This is why `SELECT *` is a cost anti-pattern in on-demand BigQuery.

Q: What is the minimum data BigQuery bills per table referenced in an on-demand query?
A: BigQuery bills a minimum of 10 MB per table referenced, even if the table is smaller. So a query touching many tiny tables still accrues 10 MB each, which matters for workloads with thousands of small reference tables. For most analytics tables this minimum is irrelevant, but it surprises people running highly fan-out queries over many small dimensions.

Q: What are BigQuery capacity-based "editions," and how do they differ from on-demand?
A: Editions — Standard, Enterprise, and Enterprise Plus — are the capacity (reservation) model where you buy slots by the hour instead of paying per byte scanned. Higher editions add features and higher per-slot-hour rates (e.g. Standard cheapest, Enterprise Plus most expensive and most capable). You pick capacity pricing when workloads are heavy and predictable enough that committed/autoscaled slots beat per-TB scanning costs.

Q: What is a BigQuery reservation and what is slot autoscaling?
A: A reservation is a pool of slots (purchased under an edition) that you assign to projects/workloads so they get dedicated capacity. Autoscaling lets a reservation grow and shrink its slot count automatically with demand, so you pay for baseline plus scaled slots rather than a fixed maximum. Commitments (1-year or 3-year) lower the per-slot rate for the baseline you reserve.

Q: A BigQuery on-demand query costs far more than expected even though it returns few rows — what is the likely cause?
A: On-demand billing is by bytes scanned, not rows returned, so a query that reads many columns or whole partitions scans (and bills for) far more than the small result implies. The usual culprits are `SELECT *`, missing partition/cluster filters, or scanning a wide table for a couple of columns. Use the dry-run estimate (`--dry_run` / the UI's estimated bytes) before running and add partition filters.

Q: How does partitioning help control on-demand cost in BigQuery, unlike Snowflake?
A: BigQuery supports explicit table partitioning (by date/timestamp, integer range, or ingestion time), and a query that filters on the partition column reads only matching partitions, scanning fewer bytes and so costing less. This is a deliberate design choice you make at table creation, in contrast to Snowflake's automatic micro-partitioning. Clustering in BigQuery further sorts data within partitions to cut bytes scanned.

Q: What is the difference between BigQuery clustering and BigQuery partitioning?
A: Partitioning splits a table into separate segments by a column (typically date), and the engine prunes whole partitions on a matching filter. Clustering sorts the data within each partition by up to four columns so the engine can skip blocks inside a partition for filters on the cluster columns. They compose: partition by date, cluster by fund and ISIN, and a query filtering all three scans the least data.

Q: What is the difference between Azure Synapse "dedicated SQL pool" and "serverless SQL pool"?
A: A dedicated SQL pool is a provisioned MPP warehouse sized in DWUs that you pay for by the hour whether or not it is querying (and can pause to stop charges). A serverless SQL pool has no provisioned cluster; it queries data in the lake on demand and bills per terabyte of data processed. Dedicated suits steady, high-concurrency warehousing; serverless suits ad-hoc querying over files in storage.

Q: What is a DWU in Azure Synapse dedicated SQL pool?
A: A DWU (Data Warehouse Unit) is a bundled measure of compute, memory, and I/O that you provision to size a dedicated SQL pool — more DWUs means more compute nodes and throughput. You pay per hour for the provisioned DWU level regardless of utilization, which is why pausing the pool when idle is the key cost lever. It is closer to "sizing a cluster" than to BigQuery's fungible slots.

Q: How is Synapse serverless SQL pool billed, and what is the cost gotcha?
A: It is billed per terabyte of data processed by your queries, with no infrastructure to provision. The gotcha is that querying many small or uncompressed files, or running `SELECT *` over a large lake, processes lots of bytes and runs up cost fast. Using columnar formats (Parquet), filtering on partitioned folder paths, and `OPENROWSET` with explicit column lists keeps bytes processed down.

Q: Where does the Microsoft Fabric Warehouse fit relative to Synapse?
A: Fabric Warehouse is Microsoft's next-generation, fully managed SaaS warehouse that supersedes much of standalone Synapse Analytics, running on the unified Fabric platform with OneLake as its storage. It uses a serverless distributed engine and stores tables in open Delta/Parquet format in OneLake rather than a proprietary store. Microsoft positions it so customers can retire standalone Synapse dedicated pools onto Fabric.

Q: How is compute billed in Microsoft Fabric, and what is a Capacity Unit?
A: Fabric compute is bought as a capacity (an F-SKU such as F2, F8, F64), and each SKU provides a fixed number of Capacity Units (CUs) shared across all Fabric workloads on that capacity. Compute is a single pooled capacity rather than a per-warehouse cluster, and one CU maps to a fixed amount of Warehouse vCores (roughly 1 CU = 0.5 Warehouse vCore). Storage in OneLake is billed separately, similar to ADLS pricing.

Q: What storage does Fabric Warehouse use, and in what format?
A: Fabric Warehouse stores all data in OneLake, the tenant-wide logical data lake, and tables are persisted in open Delta Lake (Parquet) format rather than a proprietary internal store. This is the key differentiator: the warehouse's tables are directly readable by Spark, the SQL endpoint, and external Iceberg consumers via metadata translation. OneLake is "the OneDrive for data," one lake per tenant.

Q: State Snowflake's Iceberg story in one sentence.
A: Snowflake supports native Apache Iceberg tables that you can run either with a Snowflake-managed catalog (full read/write) or against an external catalog (such as AWS Glue or a REST catalog) where Snowflake reads Iceberg data managed elsewhere. This lets a fund platform keep one copy of holdings data in open Iceberg format and query it from Snowflake without ingesting it into proprietary storage.

Q: State BigQuery's Iceberg story in one sentence.
A: BigQuery reads and writes Apache Iceberg through BigLake — external/managed Iceberg tables backed by Cloud Storage and the BigLake Metastore — so it can query the same Iceberg files other engines write. This gives an open-format, interoperable lakehouse path rather than locking data into BigQuery's native storage.

Q: State Microsoft Fabric's Iceberg story in one sentence.
A: Fabric stores tables as Delta in OneLake but provides automatic metadata translation so the same physical Parquet data is exposed as Apache Iceberg (and vice versa), letting Iceberg engines like Snowflake read Fabric tables without copying. This metadata virtualization is how OneLake offers bi-directional Delta/Iceberg interoperability.

Q: Why is the Iceberg/open-format story a vendor-selection question for a fund administrator?
A: Because regulated fund platforms must avoid lock-in and prove they can move or share holdings, NAV, and transaction data across engines and auditors over a long retention horizon. An engine that reads/writes open Iceberg means one governed copy of the data serves BI, ML, and regulatory access without proprietary export. It de-risks both exit and multi-engine architectures, which matters in vendor due diligence.

Q: For a Luxembourg-domiciled fund administrator, why does an engine's EU-region and compliance posture matter?
A: Fund and investor data often must reside and be processed within the EU/EEA for GDPR and regulator expectations, so the warehouse must offer EU regions (e.g. a Luxembourg or other EU data residency) and contractual/compliance commitments. Under DORA, the firm must also assess the provider as a critical ICT third party. A US-only or non-compliant region choice can be a deal-breaker regardless of price.

Q: How do you explain to a CFO why a month-end reporting spike inflates the Snowflake bill?
A: Month-end concentrates many heavy queries (NAV reconciliation, regulatory extracts) into a few days, so warehouses run larger and longer and at higher concurrency, burning far more credits than a quiet mid-month day. Because cost is workload-shaped, the bill follows that activity curve, not the data volume. The mitigation is multi-cluster auto-scaling sized to the spike plus auto-suspend, so you pay for the peak only while it lasts.

Q: What is a multi-cluster (auto-scaling) warehouse in Snowflake and what problem does it solve?
A: A multi-cluster warehouse can spin up additional identical clusters when query concurrency exceeds what one cluster handles, then scale back down when demand falls. It solves the queuing problem during concurrency spikes (like month-end), preserving response time without permanently oversizing. You pay only for the extra clusters while they run, which keeps the cost tied to actual peak demand.

Q: In Snowflake, what is the difference between scaling up and scaling out?
A: Scaling up means choosing a larger warehouse size (e.g. M to L) to make a single complex query faster by giving it more compute. Scaling out means adding more clusters (multi-cluster) to handle more concurrent queries without each one slowing down. Use up for heavy individual queries and out for high-concurrency BI; conflating them leads to either slow reports or wasted credits.

Q: Why might a fund platform keep a warehouse for BI even in a lakehouse-first architecture?
A: Warehouses still deliver superior high-concurrency interactive BI and fine-grained, mature security/governance that many lakehouse engines are only catching up to. For hundreds of analysts hitting NAV and performance dashboards at once with row/column-level controls, the warehouse's optimizer, caching, and access model are battle-tested. The lakehouse holds the open, governed copy; the warehouse serves the demanding serving layer.

Q: What is BigQuery's result cache and how does it affect on-demand cost?
A: BigQuery caches query results for about 24 hours, and a repeated identical query served from cache scans zero bytes, so it is free under on-demand pricing. This rewards deterministic, repeated dashboard queries but the cache misses if the underlying tables change or the query text differs even trivially. It is one reason raw byte-scanned cost can vary between seemingly identical runs.

Q: How does Snowflake's result cache differ in scope from its warehouse cache?
A: Snowflake has a result cache in the cloud services layer that returns identical-query results without using any warehouse (no credits, ~24 hours, invalidated by data changes), and a local SSD data cache on each warehouse that speeds repeated scans while the warehouse is warm. The result cache needs no running warehouse; the local cache is lost when the warehouse suspends. Aggressive auto-suspend therefore trades credit savings for lost warm cache.

Q: What does it mean that BigQuery slots are a "fungible pool, not a sized cluster"?
A: It means compute is not a fixed-size machine you reserve and keep warm; BigQuery draws slots from a shared, elastic pool and assigns however many a query can use at that moment. You do not (in on-demand) decide a cluster size — you submit SQL and the system parallelizes across available slots. With capacity editions you do cap/reserve slots, but they are still allocated as a fungible quantity, not a named server.

Q: Why is Snowflake's compute model better described as a "sized cluster" than a fungible pool?
A: Because you explicitly create named virtual warehouses at a chosen size (XS–6XL), and that size fixes the compute capacity and credit rate until you resize or suspend it. Queries run on the warehouse you route them to, not on an anonymous shared pool. This gives precise workload isolation but puts the sizing decision (and the risk of over/under-provisioning) on you.

Q: Name each engine's primary pricing unit in one line each.
A: Snowflake bills compute in credits (per-second, 60-second minimum) plus storage by TB. BigQuery bills either bytes scanned (on-demand, per TB) or slot-hours (capacity editions), plus storage by bytes. Synapse dedicated bills provisioned DWU-hours and serverless bills per TB processed; Fabric bills Capacity Units via an F-SKU plus OneLake storage.

Q: Name each engine's workload-isolation mechanism in one line each.
A: Snowflake isolates with independent virtual warehouses (and multi-cluster scale-out). BigQuery isolates with reservations/assignments of slots per workload (and project-level on-demand slot quotas). Synapse dedicated isolates by resource classes/workload groups within a provisioned pool; Fabric isolates by capacity (F-SKU) with workload-level smoothing and bursting.

Q: What is a "resource class" / workload group in Synapse dedicated SQL pool?
A: Resource classes (and the newer workload management groups) control how much memory and concurrency a query or user gets within the fixed DWU pool, so heavy ETL does not starve interactive queries. Because a dedicated pool is one provisioned cluster, isolation is achieved by partitioning its resources rather than by spinning up separate clusters. This is a different isolation philosophy from Snowflake's "just add another warehouse."

Q: Why can BigQuery costs run away even though it is serverless?
A: Serverless removes cluster management but not cost: on-demand you pay per byte scanned, so unfiltered `SELECT *`, unpartitioned scans, or full-table joins read enormous volumes and bill accordingly. There is no idle cluster to forget, but a single careless analyst query over a multi-TB table can be expensive. Cost controls (per-query and per-user byte limits, required partition filters, dry-runs) are the guardrails.

Q: What BigQuery feature forces analysts to filter a partitioned table to avoid full scans?
A: The table option "require partition filter" (`require_partition_filter`) makes BigQuery reject any query on the table that does not include a predicate on the partition column. This is a strong cost guardrail on large partitioned tables — for example a holdings table partitioned by `trade_date` — because it prevents accidental whole-table scans. It turns a soft best-practice into an enforced one.

Q: What is the single most common way costs run away in Snowflake?
A: Warehouses left running idle (and/or oversized) so they keep burning credits with no work, compounded by start/stop churn that triggers the 60-second minimum repeatedly. The single biggest lever is `AUTO_SUSPEND` plus right-sizing, then watching `WAREHOUSE_METERING_HISTORY`. Full scans from poor clustering are the second-most-common offender.

Q: What is the single most common way costs run away in BigQuery on-demand?
A: Queries scanning far more bytes than needed — `SELECT *`, missing partition/cluster filters, or repeatedly re-scanning large tables — because billing is per byte scanned. The fix is column projection, partition filters, materialized/aggregate tables, and dry-run estimates before running. Setting maximum-bytes-billed per query caps the blast radius.

Q: What is the single most common way costs run away in Synapse / Fabric?
A: For Synapse dedicated, leaving a large DWU pool provisioned (and not paused) so you pay hourly around the clock; for serverless and Fabric, scanning many small/uncompressed files or oversized capacities sitting idle. The levers are pausing dedicated pools, compacting to Parquet for serverless, and right-sizing the Fabric F-SKU. Idle provisioned compute is the recurring theme.

Q: Why does an architect care that costs are "workload-shaped, not data-shaped" when sizing a NAV reporting platform?
A: Because the budgeting model must follow the reporting calendar: cost peaks around daily/monthly NAV strikes and regulatory deadlines, not steadily with stored history. You size elastic compute (multi-cluster or autoscaled slots) for the spike and let it shrink afterward, instead of provisioning a fixed cluster for peak all month. Getting this wrong turns a predictable monthly cycle into a budget incident.

Q: What is DuckDB and why is it used as a local stand-in for warehouse exercises?
A: DuckDB is an in-process, columnar, vectorized analytical SQL engine — like "SQLite for OLAP" — that runs on your laptop with no server. It reproduces warehouse-style concepts (columnar scans, predicate pushdown over Parquet, analytical SQL) so you can practice query and storage behavior locally before touching a paid cloud warehouse. On native Ubuntu you just `uv run` a Python script or use the CLI; no cloud account or credits needed.

Q: Why is DuckDB a reasonable proxy for warehouse pruning behavior?
A: Because it reads Parquet using row-group statistics and does projection/predicate pushdown, so you can observe how filtering on a well-laid-out column skips data — the same principle behind Snowflake micro-partition pruning and BigQuery partition skipping. You can measure bytes/rows read with different file layouts locally and reason about cost implications. It is not identical to any cloud engine, but the storage-layout intuition transfers.

Q: In Snowflake, what does `SYSTEM$CLUSTERING_INFORMATION` tell you?
A: It returns clustering metrics for a table on given columns — such as the average number of overlapping micro-partitions and clustering depth — indicating how well pruning will work for filters on those columns. High overlap/depth means poor clustering and weak pruning, signaling you should define or reapply a clustering key. It is the diagnostic you run before blaming the engine for slow scans.

Q: Which Snowflake view shows how many credits each warehouse consumed?
A: `WAREHOUSE_METERING_HISTORY` (in `SNOWFLAKE.ACCOUNT_USAGE` or the `INFORMATION_SCHEMA` table function) reports credits used per warehouse over time. It is the first place to look when investigating a cost spike, letting you attribute credits to specific warehouses and time windows. Pair it with `QUERY_HISTORY` to find the offending queries.

Q: How would you estimate a BigQuery query's cost before running it?
A: Run a dry run — `bq query --dry_run` on the CLI or the editor's validator — which returns the bytes the query will scan without executing or billing it. Multiply estimated bytes by the on-demand per-TB rate to get the cost. This is the standard discipline before running anything large in on-demand BigQuery, and it catches `SELECT *` mistakes early.

Q: What is the difference between active and long-term storage pricing in BigQuery?
A: BigQuery automatically discounts storage for a table partition that has not been modified for 90 consecutive days, moving it to a roughly half-price long-term storage tier with no change in performance or access. Active (recently modified) data is billed at the standard rate. This rewards append-only/immutable historical tables, common for fund holdings and transaction history.

Q: Why is `SELECT *` specifically dangerous in BigQuery on-demand and in Synapse serverless?
A: Both bill by data read/processed, and columnar storage means `SELECT *` defeats column projection by forcing the engine to read every column, multiplying bytes scanned and therefore cost. Selecting only the needed columns can cut a query's bill by an order of magnitude on a wide table. The same query in a credit/DWU model wastes time rather than bytes, but it is still wasteful.

Q: How does column projection reduce cost in a columnar warehouse?
A: Because data is stored column by column, the engine reads only the columns a query references and skips the rest, so a query touching 3 of 200 columns reads a fraction of the bytes. In byte-billed engines (BigQuery on-demand, Synapse serverless) this directly lowers cost; in credit/slot engines it lowers runtime. This is why explicit column lists beat `SELECT *` everywhere.

Q: What is predicate pushdown and how does it relate to warehouse pruning?
A: Predicate pushdown means the filter from the `WHERE` clause is applied as early as possible — at the storage/scan level — so the engine reads only blocks/partitions that can satisfy it. In Snowflake this is micro-partition pruning via min/max stats; in BigQuery it is partition/cluster pruning; in DuckDB/Parquet it is row-group skipping. The shared idea is "decide what not to read before reading."

Q: Why does loading a fund holdings table sorted by `trade_date` improve Snowflake pruning?
A: Sorting on load co-locates rows of the same date into the same micro-partitions, so each partition's min/max range for `trade_date` is narrow and non-overlapping. A query filtering one date then prunes to just the relevant partitions instead of scanning all of them. It is the cheap alternative to defining and maintaining a clustering key when you control load order.

Q: A daily NAV job in Snowflake updates yesterday's rows and the bill jumps — why?
A: Because micro-partitions are immutable, every `UPDATE` rewrites whole partitions and marks the old ones stale, generating churn that both costs compute and degrades clustering until reclustering runs. Frequent small updates are an anti-pattern; Snowflake favors append/`MERGE` patterns over row-by-row updates. Batching changes and clustering on the update key reduces the damage.

Q: Why is concurrency, not just data size, central to comparing these warehouses for fund BI?
A: A month-end opens hundreds of simultaneous dashboard and report queries, and an engine's ability to isolate and scale that concurrency (Snowflake multi-cluster, BigQuery slot reservations, Synapse workload groups, Fabric capacity bursting) determines both response time and cost. The same total data can be cheap or a queuing disaster depending on concurrency handling. Architects evaluate the isolation mechanism precisely because of this.

Q: How does Snowflake achieve workload isolation between, say, an ETL team and a BI team?
A: You give each team its own virtual warehouse against the same shared data, so ETL's heavy loads run on one warehouse and BI's interactive queries on another, with no compute contention. Each warehouse is independently sized, suspended, and billed. This "separate compute, shared storage" pattern is the canonical Snowflake isolation answer.

Q: How does BigQuery achieve workload isolation in the capacity model?
A: You create reservations of slots and assign projects or workloads to them, so an ETL reservation cannot starve a BI reservation of slots. Idle slots can optionally be shared across reservations in the same admin project for efficiency. This gives Snowflake-like isolation while keeping the serverless, pool-based allocation underneath.

Q: Why might Synapse dedicated SQL pool be a harder cost story to a CFO than Snowflake?
A: Because a dedicated pool bills for provisioned DWUs by the hour whether or not it is querying, the cost is more "capacity-shaped" — you pay for the cluster you reserved, like a leased machine. The lever is remembering to pause it, which is easy to forget and risky to automate around active reporting. Snowflake's auto-suspend makes idle waste self-correcting, which is an easier conversation.

Q: What does "compute as a fungible pool" let BigQuery do that a sized cluster cannot?
A: It lets a single query burst across thousands of slots for a short heavy operation and then release them, without you pre-sizing or paying to keep a big cluster warm. Capacity scales to the shape of the query rather than to a fixed maximum you chose in advance. The tradeoff is less predictable per-query performance under shared on-demand quotas.

Q: How does Snowflake's separation of services from compute help during a metadata-only operation?
A: Operations like `SHOW`, `DESCRIBE`, listing tables, or queries answerable purely from metadata/statistics can be served by the cloud services layer without resuming a warehouse, so they consume no credits. This is why some "queries" return instantly even with all warehouses suspended. It is the practical payoff of having a distinct services layer.

Q: An analyst complains a Snowflake dashboard got slower after hours — what cache effect explains it?
A: After auto-suspend, the warehouse's local SSD data cache is cleared, so the first queries when it resumes must re-read from remote storage (cold cache) and run slower until the cache warms. The result cache still helps for identical queries, but new or changed queries pay the cold-start penalty. Tuning `AUTO_SUSPEND` higher trades some idle cost for warm caches during business hours.

Q: Why is BigQuery's pricing sometimes hard to predict for a finance team?
A: Under on-demand, cost depends on bytes each query scans, which varies with query patterns, partition filters, and cache hits, making month-to-month spend bumpy. The same dashboard can be free (cached) one run and expensive (full scan) the next. Teams seeking predictability move predictable workloads onto a fixed slot reservation/edition so spend becomes flat capacity rather than variable scanning.

Q: What does it mean to say each of these engines separates storage and compute, and why is it the common thread?
A: All three (Snowflake, BigQuery, Synapse/Fabric in their modern forms) keep data in scalable object/lake storage independent of the compute that queries it, so you scale and pay for each separately. This is the architectural premise that makes elasticity, multi-engine access, and workload-shaped cost possible. The differences are in how compute is packaged: sized warehouses, fungible slots, provisioned DWUs, or pooled capacity units.

Q: In a vendor-selection comparison table, what five attributes does the syllabus ask you to fill per engine?
A: Architecture, pricing unit, workload-isolation mechanism, Iceberg/open-format story, and EU-region/compliance posture. These five let you compare Snowflake, BigQuery, and Synapse/Fabric on the dimensions a regulated fund administrator actually decides on. The sixth practical note is the single most common runaway-cost pattern for each.

Q: Summarize Snowflake's row in such a comparison table.
A: Architecture: three layers (storage, virtual-warehouse compute, cloud services) with separate scaling. Pricing: credits per-second (60s minimum) plus storage; workload isolation via independent/multi-cluster warehouses; Iceberg via managed or external-catalog Iceberg tables; EU regions across AWS/Azure/GCP with strong compliance attestations. Runaway cost: idle/oversized warehouses.

Q: Summarize BigQuery's row in such a comparison table.
A: Architecture: serverless with Colossus storage, Capacitor columnar format, Dremel execution over a fungible slot pool. Pricing: bytes scanned on-demand or slot-hours (editions) plus storage; isolation via slot reservations; Iceberg via BigLake external/managed Iceberg tables; EU multi-regions and region-pinning with GDPR/compliance support. Runaway cost: unfiltered full-table scans billed per byte.

Q: Summarize Synapse/Fabric's row in such a comparison table.
A: Architecture: Synapse dedicated MPP pool (DWU) vs serverless SQL over the lake, evolving into Fabric Warehouse on OneLake (Delta/Parquet). Pricing: DWU-hours or per-TB processed (Synapse) / Capacity Units via F-SKU (Fabric); isolation via workload groups (Synapse) or capacity (Fabric); Iceberg via OneLake Delta-to-Iceberg metadata translation; EU regions with Microsoft compliance posture. Runaway cost: idle provisioned compute.

Q: Why does a fund administrator weigh "fine-grained security" when choosing a warehouse over a bare lakehouse?
A: Investor and holdings data demand row- and column-level access control, dynamic data masking, and auditable governance — for example masking investor PII or restricting one client's NAV from another's analysts. Mature warehouses provide these natively and provably, which matters for GDPR, audit, and segregation-of-duties. A lakehouse engine may need extra tooling to match this, so security maturity is a real selection axis.

Q: What is dynamic data masking and why does it matter for fund investor data?
A: Dynamic data masking applies a masking policy at query time so unauthorized users see obfuscated values (e.g. a redacted investor name or account number) while the underlying data is unchanged. It lets one governed table serve many audiences without copies, which suits transfer-agency data where most analysts must not see investor PII. Snowflake, BigQuery, and Synapse all offer forms of column masking and row-level security.

Q: Why is high-concurrency BI a reason to keep a warehouse rather than query the lake directly?
A: Warehouses are engineered for many short, interactive queries at once with caching, result reuse, and concurrency scaling, giving sub-second dashboards under load. Direct lake querying can be slower and less consistent under hundreds of concurrent users without a serving optimization. So the lakehouse holds the open governed copy and a warehouse (or warehouse-style serving layer) handles the demanding interactive tier.

Q: How does DORA influence cloud-warehouse vendor selection for an EU fund administrator?
A: DORA treats major cloud providers as critical ICT third parties, so the firm must assess concentration risk, ensure contractual rights (audit, exit, sub-outsourcing transparency), and have resilience/exit plans. The open-format (Iceberg) story directly supports a credible exit strategy, and EU-region availability supports data residency. Vendor selection therefore weighs regulatory and exit posture alongside price and performance.

Q: Why does the Iceberg story strengthen a DORA exit plan?
A: If holdings, NAV, and transaction data live in open Apache Iceberg rather than a proprietary store, you can repoint another compliant engine at the same files without a costly export/reload, making the exit credible and testable. Lock-in to a proprietary format is exactly the concentration risk DORA wants firms to mitigate. So an engine's native Iceberg support is a regulatory as well as a technical asset.

Q: What is the practical difference between Snowflake managed Iceberg tables and external-catalog Iceberg tables?
A: With a Snowflake-managed Iceberg table, Snowflake owns the catalog and can fully read and write/compact the Iceberg data. With an external catalog (e.g. Glue or a REST catalog), another system owns the metadata and Snowflake typically reads (and in some configurations writes) the same files. Managed gives full lifecycle control; external maximizes interoperability with engines that write the table.

Q: Why might BigQuery's `require_partition_filter` be set on a regulatory transaction table?
A: Because such tables are huge (every transaction over years) and analysts rarely need a full scan, enforcing a partition filter (e.g. on `event_date`) prevents accidental whole-table reads that would be slow and expensive on-demand. It bakes cost discipline into the schema so a careless query simply errors rather than billing for terabytes. This is a governance control as much as a performance one.

Q: How does storage being billed separately change the architect's mental model of "the cost of a table"?
A: A table's storage cost is small and steady (bytes × rate), while its real expense is the compute every query against it consumes, which can vastly exceed storage over a table's life. So a rarely queried 50 TB archive can be cheaper than a 1 TB hot table scanned thousands of times a day. The architect optimizes query/compute patterns, not just data footprint.

Q: Why can two teams query the same Snowflake data with zero performance interference?
A: Because storage is shared and compute is isolated: each team runs its own virtual warehouse, and warehouses do not share compute resources, so one team's heavy job cannot slow another's. They read the same single copy of the data from the storage layer concurrently. This is the architectural answer to "how do you isolate without copying data."

Q: What goes wrong if you set Snowflake `AUTO_SUSPEND` to a very low value like 1 second?
A: The warehouse suspends almost immediately between queries, so bursty interactive workloads constantly resume — each resume incurring the 60-second minimum billing and losing the warm local cache. You can end up paying more (and running slower from cold caches) than with a sensible 60–300 second value. Tune auto-suspend to the gap between queries in the workload, not to the absolute minimum.

Q: Why does BigQuery's serverless model not eliminate the need for good data modeling?
A: Because cost and speed still depend on how much data each query reads, so partitioning, clustering, column pruning, and avoiding needless joins remain essential. Serverless removes cluster ops, not the physics of scanning bytes. A well-modeled, partitioned table can be an order of magnitude cheaper and faster than a flat unpartitioned dump of the same data.

Q: How would you cap the worst-case cost of a single BigQuery on-demand query?
A: Set a maximum-bytes-billed limit (`--maximum_bytes_billed` / the `maximumBytesBilled` job property), and BigQuery will fail any query that would scan more than that, rather than running and billing it. This caps the blast radius of an accidental full scan. Combined with custom per-user/per-project quotas, it bounds spend without trusting every analyst to dry-run first.

Q: Why does Snowflake store data compressed and columnar inside micro-partitions?
A: Columnar layout lets a query read only the columns it needs and compresses far better than row storage because each column holds homogeneous values, cutting both storage cost and bytes read per scan. Snowflake chooses and applies the compression automatically, so the on-disk micro-partition is much smaller than the 50–500 MB uncompressed figure. This is the same columnar-plus-compression principle behind Parquet and BigQuery's Capacitor, just managed internally.

Q: Why can BigQuery streaming inserts cost more than batch loads?
A: Streaming inserts (the legacy streaming API or the Storage Write API's real-time path) carry a per-row/per-byte ingestion charge, whereas batch loads from Cloud Storage are free of ingestion fees. For a high-frequency feed you pay a premium for immediacy, so if near-real-time is not required, batching is materially cheaper. Architects weigh latency need against this ingestion cost when designing a feed.

Q: What is an external table in Synapse serverless SQL pool, and why does it matter for cost?
A: An external table (or `OPENROWSET`) lets serverless SQL query files directly in the lake (e.g. Parquet in ADLS) without loading them into a provisioned store, billing per terabyte processed. Because cost is bytes processed, the file format and layout matter enormously — Parquet with folder partitioning processes far fewer bytes than raw CSV. It is the pay-per-query way to query lake data without a dedicated pool.

Q: A daily EMT/EPT file feed must land in the warehouse for regulatory reporting at month-end — how does the engine's pricing model shape the design?
A: The feed is a predictable daily batch with a concentrated reporting spike, so you want cheap/free batch ingestion and elastic compute that scales only for the reporting window. On Snowflake that is batch `COPY INTO` plus an auto-suspended warehouse (multi-cluster for the spike); on BigQuery, free batch loads plus on-demand or an autoscaled reservation; on Fabric, a right-sized capacity. Modeling the EMT/EPT data partitioned by reporting date keeps both scan cost and pruning efficient.

Q: Why is "fine-grained security" sometimes cited as a reason warehouses persist despite lakehouses?
A: Mature warehouses ship row-level security, column masking, tag-based policies, and detailed audit out of the box and proven at scale, whereas equivalent lakehouse governance can require extra catalog/policy tooling to match. For regulated investor data, "provably enforced and audited" beats "possible to configure." That maturity is a legitimate, non-performance reason to keep a warehouse in the serving tier.

Q: What is the difference between BigQuery on-demand slot access and a dedicated reservation for predictability?
A: On-demand shares a pool (commonly up to ~2,000 slots per project) with cost driven by bytes scanned and variable performance under contention. A reservation gives you a committed/autoscaled number of slots at a fixed slot-hour cost, decoupling spend from bytes and stabilizing performance. Finance teams pick reservations to convert spiky variable cost into predictable capacity cost.

Q: How does a fund's daily ISIN/holdings load pattern inform warehouse choice and cost?
A: A predictable daily batch load plus a concentrated reporting window favors elastic compute that scales for the window and shrinks after — auto-suspend/multi-cluster (Snowflake), autoscaled reservations (BigQuery), or right-sized capacity (Fabric). Append-friendly modeling (load by `trade_date`/ISIN, avoid row-level updates) keeps clustering and bytes-scanned costs down. The shape of the daily cycle, not the raw data size, drives the design.

Q: Why is it wrong to tell a CFO "we can cut the warehouse bill by deleting old data"?
A: Because cost is workload-shaped: storage of old data is cheap and steady, so deleting it barely moves the bill, while compute on active queries dominates. The real savings come from suspending idle warehouses, right-sizing compute, pruning scans, and reusing caches. Framing it as a data problem leads to deleting useful history for negligible savings.

Q: What single sentence captures why an architect must understand all three pricing models?
A: Because the same workload can be cheapest on different engines depending on whether cost is shaped by credits (time), bytes scanned, or provisioned capacity — and getting the model wrong turns a fund administrator's predictable month-end into a budget incident. The architect's job in vendor selection is to map the firm's actual query and concurrency patterns onto the model that bills them most favorably, with the right isolation and open-format posture for regulatory comfort.

Q: Exactly how many credits per hour does each Snowflake warehouse size from `XS` to `6XL` consume?
A: The rate doubles at every step: `XS` = 1, `S` = 2, `M` = 4, `L` = 8, `XL` = 16, `2XL` = 32, `3XL` = 64, `4XL` = 128, `5XL` = 256, and `6XL` = 512 credits per hour. Because each step doubles both compute and cost, going from `M` to `L` only pays off if the query roughly halves in runtime. The gotcha is assuming a bigger warehouse is always cheaper per query — it is only cheaper if the work parallelizes enough to finish proportionally faster.

Q: A Snowflake `MEDIUM` warehouse runs a query in 8 minutes; you resize to `LARGE` and it now runs in 5 minutes — was that a good trade?
A: No, it was a net loss: `LARGE` burns 8 credits/hr versus `MEDIUM`'s 4, so doubling the rate while only cutting runtime from 8 to 5 minutes means you paid more credits for the faster result (8 × 5/60 ≈ 0.67 vs 4 × 8/60 ≈ 0.53). Scaling up only saves money when runtime drops by more than half. The lesson is to benchmark credit cost (size × time), not wall-clock time alone, before sizing up.

Q: How many days of Time Travel does Snowflake Standard edition give versus Enterprise, and why does it matter for fund data?
A: Standard edition allows a maximum `DATA_RETENTION_TIME_IN_DAYS` of 1 day, while Enterprise edition raises the ceiling to 90 days for permanent tables. For a fund administrator who must reproduce or roll back a corrected NAV table, the 90-day window on Enterprise is what lets you query or restore the table as it was before the correction. The gotcha is that longer retention keeps more old micro-partitions alive, so it raises storage cost.

Q: What is the difference between Time Travel and Fail-safe in Snowflake?
A: Time Travel is the user-accessible window (`AT`/`BEFORE` clauses, `UNDROP`) where you can query or restore historical data, configurable from 0 up to 90 days on Enterprise. Fail-safe is a fixed, non-configurable 7-day period after Time Travel expires during which only Snowflake support can recover data, intended for disaster recovery, not self-service. Both consume storage, so a heavily-churned table can carry weeks of hidden retention cost beyond its visible size.

Q: How do you query a Snowflake table as it existed before a bad NAV correction landed at a known time?
A: Use Time Travel with `SELECT ... FROM holdings AT(TIMESTAMP => '2026-06-14 17:00:00'::timestamp)` or `BEFORE(STATEMENT => '<query_id>')` to read the pre-correction snapshot. This works because every change wrote new immutable micro-partitions and the old ones survive within the retention window. The gotcha is that if retention is set to 1 day and the error is discovered on day 3, the old version is already gone to Fail-safe and only support can retrieve it.

Q: What does Snowflake `UNDROP TABLE` do and what is its time limit?
A: `UNDROP TABLE my_table` restores a dropped table from Time Travel without reloading data, because the drop only marks it as removed while its micro-partitions persist for the retention window. It is the fast recovery path when someone accidentally drops a holdings table. The limit is the table's `DATA_RETENTION_TIME_IN_DAYS`; once that elapses the table moves to Fail-safe and `UNDROP` no longer works.

Q: What is zero-copy cloning in Snowflake and why is it nearly free?
A: `CREATE TABLE clone_t CLONE source_t` (also for schemas and databases) creates an independent, writable copy that initially shares the same underlying micro-partitions, so no data is physically duplicated at clone time. It is near-instant and near-free because only metadata pointers are created; storage grows only as the clone diverges via writes. Fund teams use it to spin up a full test or month-end snapshot environment without doubling storage cost.

Q: A team clones a 10 TB Snowflake holdings table for testing — when does that clone start costing real storage?
A: It costs almost nothing at creation because the clone shares the source's existing micro-partitions, but storage grows as either the clone or the source is modified, since each change writes new partitions unique to that object. So a read-only test clone stays cheap, while a clone subjected to a heavy reload diverges and accrues its own storage. The gotcha is assuming a clone is permanently free — divergence is what bills.

Q: How does Snowflake's `RESULT_SCAN` function relate to the result cache?
A: `RESULT_SCAN(LAST_QUERY_ID())` (or a specific query id) lets you query the stored result set of a previous query as if it were a table, drawing from the cached results held in cloud services. It is useful for post-processing a result without rerunning the expensive query. It is related to but distinct from the automatic result cache that transparently serves identical repeated queries for ~24 hours with no warehouse.

Q: What exactly invalidates Snowflake's result cache for an otherwise identical query?
A: The cache is invalidated if the underlying table data changes (any DML on the referenced micro-partitions), if the query text differs at all (even whitespace or a literal), or if it uses non-deterministic functions like `CURRENT_TIMESTAMP()`. Role and certain session context also affect reuse. So a dashboard that appends new rows each morning will miss the cache on the first post-load run, then hit it for repeats until the next change.

Q: Why does a Snowflake query using `CURRENT_TIMESTAMP()` never hit the result cache?
A: Because the result cache only serves results it can prove are still correct, and a non-deterministic function like `CURRENT_TIMESTAMP()` would produce a different answer on each run, so Snowflake refuses to reuse the cached result. The same applies to `CURRENT_DATE()` in some contexts and to external functions. The gotcha is a "free" dashboard query silently becoming a billed warehouse query because someone added a live timestamp column.

Q: In BigQuery on-demand, what is the exact per-TiB rate and free tier, and what unit is "TiB"?
A: On-demand analysis is billed at about `$6.25` per TiB scanned, with the first 1 TiB per month free per billing account. A TiB is a tebibyte (2^40 bytes, ~1.1 TB), not a decimal terabyte, so the bytes-scanned figure BigQuery reports is in binary units. The gotcha is mixing up TB and TiB when estimating cost — using decimal TB slightly understates the bytes-to-cost conversion.

Q: How do you run a BigQuery dry run from the `bq` CLI and read the estimated bytes?
A: Run `bq query --use_legacy_sql=false --dry_run 'SELECT ...'` and BigQuery returns the estimated bytes the query would scan without executing or billing it. You then multiply that byte estimate by the per-TiB rate to estimate cost. The gotcha is that a dry run cannot account for a result-cache hit (which would be free) — it always reports the full scan estimate.

Q: How do you cap the bytes a single BigQuery query may bill, and what happens when the cap is exceeded?
A: Set `--maximum_bytes_billed=<bytes>` on the CLI (or the `maximumBytesBilled` job property / a query-label limit), and BigQuery aborts the query with an error before running it if the estimate exceeds the cap, billing nothing. This bounds the blast radius of an accidental full scan, e.g. capping at 100 GB on a multi-TB table. The gotcha is the limit is per query, so it does not stop many small queries from adding up.

Q: What error does BigQuery raise when a query would exceed `maximum_bytes_billed`, and is it billed?
A: BigQuery fails the job with an error stating the query would process more bytes than the `maximumBytesBilled` limit allows (a `bytesBilledLimitExceeded`-style failure), and because the query never executes you are not billed for the scan. This is the intended guardrail behaviour — fail fast rather than run expensively. The gotcha is that the cap must exceed the genuine need, or legitimate large reports will be blocked.

Q: What does `require_partition_filter` do at the SQL level, and what error appears without a filter?
A: Setting the table option `require_partition_filter = true` makes BigQuery reject any query that does not include a predicate on the partitioning column, raising an error to the effect that the query must filter the partition column. It enforces partition pruning as a hard rule, e.g. on a `trade_date`-partitioned transaction table. The gotcha is that a filter on a non-partition column does not satisfy it — the predicate must reference the partition column itself.

Q: How does BigQuery clustering prune within a partition, and what is the four-column limit?
A: Clustering sorts data inside each partition by up to four columns in priority order, writing block-level min/max metadata so the engine can skip blocks that cannot match a filter on those columns. You specify it as `CLUSTER BY fund_id, isin` (left-to-right priority). The gotcha is that filtering only the second clustering column without the first gives weaker pruning, because the data is sorted primarily by the first column.

Q: Why does BigQuery clustering give only an "estimated" cost rather than a guaranteed bytes-scanned figure before running?
A: Because partition pruning is known from metadata up front, but block-level cluster pruning depends on the actual data distribution and is only resolved during execution, so the pre-run dry-run estimate reflects partition pruning, not clustering. The real scanned bytes (and bill) can be lower than the estimate once clustering skips blocks. The gotcha is that dry runs over clustered tables can overestimate cost, which is the safe direction but surprises people.

Q: What is the difference between BigQuery ingestion-time partitioning and column partitioning?
A: Ingestion-time partitioning splits a table by the load timestamp, exposed via the pseudo-column `_PARTITIONTIME` (or `_PARTITIONDATE`), with no real column needed. Column partitioning splits by a `DATE`/`TIMESTAMP`/`DATETIME` or integer-range column you choose, like `trade_date`. For fund data the column form is usually better because business date often differs from load date; filtering must reference whichever column actually defines the partition.

Q: How does BigQuery's Storage Write API differ in cost from the legacy streaming insert API?
A: The Storage Write API offers a free default-stream path for high-throughput ingestion plus a paid path, and it generally supersedes the legacy `tabledata.insertAll` streaming API, which charged per-row/per-byte. Batch loads from Cloud Storage remain free of ingestion charges. The architecture choice is: free batch when latency tolerates it, Storage Write API for streaming, and avoid the legacy streaming insert for new designs.

Q: What does BigQuery's `INFORMATION_SCHEMA.JOBS` view let a cost investigator find?
A: It exposes per-job metadata including `total_bytes_billed`, `total_slot_ms`, user, and query text, so you can attribute spend and find the queries scanning the most bytes. It is the BigQuery analogue of Snowflake's `QUERY_HISTORY` for cost forensics. The gotcha is that `total_bytes_processed` and `total_bytes_billed` differ (cache hits and the 10 MB-per-table minimum), so bill from `total_bytes_billed`.

Q: How would you find the top 10 most expensive BigQuery queries last month for a cost review?
A: Query `INFORMATION_SCHEMA.JOBS_BY_PROJECT`, filter on `creation_time` for the month and `job_type = 'QUERY'`, order by `total_bytes_billed DESC`, and limit to 10, optionally joining user and query text. This pinpoints the offenders driving on-demand spend so you can add partition filters or move them to a reservation. The gotcha is region scoping — `INFORMATION_SCHEMA.JOBS` is region-qualified, so you must query the right region's view.

Q: In Snowflake, how do you list the queries that drove a cost spike on a specific warehouse and day?
A: Query `SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY` filtered by `WAREHOUSE_NAME` and the `START_TIME` window, ordering by `BYTES_SCANNED` or `EXECUTION_TIME` to surface the heaviest queries. Cross-reference `WAREHOUSE_METERING_HISTORY` to confirm which warehouse burned the credits. The gotcha is `ACCOUNT_USAGE` has up to ~45 minutes of latency, so the newest queries may not appear immediately; the `INFORMATION_SCHEMA` table functions are fresher but shorter-range.

Q: What is the latency and retention difference between Snowflake's `ACCOUNT_USAGE` and `INFORMATION_SCHEMA` for query history?
A: `SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY` retains roughly 365 days of history but has up to ~45 minutes of data latency, while the `INFORMATION_SCHEMA.QUERY_HISTORY` table function is near-real-time but covers only the last 7 days (and 10,000 rows). For long-term cost trending use `ACCOUNT_USAGE`; for live debugging use `INFORMATION_SCHEMA`. The gotcha is expecting `ACCOUNT_USAGE` to show a query that ran 5 minutes ago.

Q: A Snowflake warehouse shows credits consumed but `QUERY_HISTORY` shows no queries in that window — what explains it?
A: The warehouse was running idle: it stays in the running state and burns credits whenever it is resumed, even between queries, so metering shows credits with no corresponding query execution. The fix is `AUTO_SUSPEND` so it pauses during the gap. The gotcha is a warehouse with `AUTO_SUSPEND` disabled or set very high left on after the last query of the day, silently billing overnight.

Q: How do multi-cluster warehouse `MIN_CLUSTER_COUNT` and `MAX_CLUSTER_COUNT` settings control month-end cost?
A: `MIN_CLUSTER_COUNT` sets the always-on baseline clusters and `MAX_CLUSTER_COUNT` the ceiling Snowflake may scale out to under concurrency; in Auto-scale mode clusters spin up only when queries queue and spin down when demand falls. Setting `MIN` to 1 and `MAX` higher means you pay for extra clusters only during the spike. The gotcha is setting `MIN` equal to `MAX` (Maximized mode), which keeps all clusters running and billing continuously.

Q: What is the difference between Snowflake multi-cluster "Auto-scale" and "Maximized" modes?
A: In Auto-scale mode (`MIN_CLUSTER_COUNT` < `MAX_CLUSTER_COUNT`) Snowflake adds and removes clusters based on query queuing, so you pay for peak only while it lasts. In Maximized mode (`MIN` = `MAX` > 1) all clusters run constantly for steady maximum concurrency, costing more but giving deterministic capacity. For a month-end NAV spike Auto-scale is the cost-right choice; Maximized suits a relentless high-concurrency trading desk.

Q: What does Snowflake's `SCALING_POLICY` of `STANDARD` versus `ECONOMY` change?
A: `STANDARD` scaling adds clusters quickly to minimize query queuing (favouring responsiveness) and starts shutting them down sooner, while `ECONOMY` waits until there is enough queued work to keep a new cluster busy for several minutes before adding one (favouring credit efficiency over latency). For latency-sensitive month-end dashboards use `STANDARD`; for batch-ish workloads where a little queuing is acceptable `ECONOMY` saves credits. The gotcha is `ECONOMY` can make interactive users wait during a spike.

Q: How does query result reuse differ between Snowflake and BigQuery in duration and scope?
A: Both cache results for roughly 24 hours and serve identical repeated queries for free (no warehouse / zero bytes), invalidated when underlying data changes. Snowflake's result cache lives in the cloud services layer and is account-wide across warehouses; BigQuery's cache is per-user/per-project by default and tied to the exact query text and tables. The shared gotcha is that any data change or non-deterministic function defeats both.

Q: Why might disabling BigQuery's query cache (`--use_cache=false`) be a deliberate test-harness choice?
A: Setting `use_cache = false` forces BigQuery to actually scan the data rather than return a cached result, which you want when benchmarking real bytes-scanned/performance so cache hits do not flatter the numbers. It is the equivalent of clearing a warm cache to measure cold performance. The gotcha is leaving it off in production, where you then pay full scan cost for every repeated dashboard query that could have been free.

Q: How does Snowflake Search Optimization Service differ from a clustering key for point lookups?
A: Search Optimization builds a separate, maintained access path (a persistent search structure) that accelerates highly selective point-lookup and equality queries on specific columns, whereas a clustering key co-locates rows to improve range-scan pruning. For finding a single ISIN or investor id in a huge table, Search Optimization can beat clustering. The gotcha is it costs storage and serverless maintenance credits, so it is justified only for frequent selective lookups.

Q: What is a Snowflake materialized view and how does it differ from a regular view for cost?
A: A materialized view stores precomputed query results that Snowflake transparently and incrementally maintains as base data changes, so reading it avoids recomputing the aggregation each time. A regular view is just stored SQL re-executed on every query. Materialized views trade ongoing maintenance credits for cheaper, faster reads of a stable aggregate — useful for a frequently-hit NAV summary, but wasteful if the base table churns heavily.

Q: When would Snowflake automatically rewrite a query to use a materialized view the user did not name?
A: The optimizer can transparently rewrite a base-table query to read a covering materialized view when doing so is cheaper, even though the SQL references the base table, similar to BigQuery's automatic materialized-view rewrite. This delivers the speed-up without changing application queries. The gotcha is that the rewrite only fires when the materialized view actually covers the query's columns and filters, so a slightly different query falls back to the base table.

Q: How does BigQuery BI Engine reduce cost and latency for dashboards?
A: BI Engine is an in-memory analysis accelerator that caches data in RAM so repeated dashboard queries are served from memory at sub-second latency and, for the cached portion, without scanning table bytes. You reserve BI Engine capacity per project/region. For a fleet of NAV dashboards hitting the same aggregates, it cuts both latency and on-demand scan cost. The gotcha is that it accelerates supported query shapes, not arbitrary heavy analytical SQL.

Q: A BigQuery query over a partitioned table still scans every partition despite a date filter — what is the first thing to check?
A: Check whether the filter is "sargable" against the partition column — wrapping the partition column in a function (e.g. `CAST(trade_date AS STRING) = ...` or `DATE(ts_col)` when partitioned on the raw timestamp) or comparing it to a non-constant subquery can prevent partition pruning, forcing a full scan. The fix is to filter the partition column directly with a constant or simple range. The gotcha is that pruning silently fails rather than erroring, so the only signal is the bytes-scanned figure.

Q: Why can a `JOIN` in BigQuery on-demand cost more than expected even with filters?
A: Because each table referenced is scanned for the columns and rows needed, and if the filter is applied after the join or one side lacks a partition filter, that side is read in full, billing its bytes. A broadcast of a large unfiltered dimension is a classic culprit. The fix is to filter and prune each side before joining and to select only needed columns; the gotcha is that a `WHERE` on the joined result does not retroactively reduce what was scanned.

Q: What is the cost effect of a Cartesian/cross join slip in a warehouse, and how is it different between credit and byte models?
A: A missing or wrong join key produces a Cartesian explosion that generates a vast intermediate result, spiking runtime and memory. In a credit/slot model (Snowflake, BigQuery capacity) you pay for the extra time and may spill to disk; in BigQuery on-demand the input bytes scanned may be modest but the query can still run very long or hit resource limits. The gotcha is that on-demand billing is by input bytes, so a Cartesian blow-up shows as a slow/failing query more than a huge bill.

Q: How would you detect and stop a runaway query mid-flight in Snowflake?
A: Use `STATEMENT_TIMEOUT_IN_SECONDS` (at account/warehouse/session level) so long queries self-abort, and `SYSTEM$CANCEL_QUERY('<query_id>')` or the UI to kill an in-flight offender found via `QUERY_HISTORY`. A `RESOURCE_MONITOR` can suspend a warehouse that exceeds a credit quota. The gotcha is that without a statement timeout a pathological query can run for hours on a large warehouse before anyone notices.

Q: What is a Snowflake resource monitor and what action can it take at a credit threshold?
A: A `RESOURCE_MONITOR` tracks credit consumption for an account or assigned warehouses against a quota over an interval, and at configured percentage thresholds it can notify, suspend the warehouse after running queries finish, or suspend immediately and cancel running queries. It is the hard budget guardrail against runaway spend. The gotcha is that monitors act on credit usage, not per-query bytes, so they cap aggregate cost rather than a single query.

Q: How would you enforce a per-user spend cap in BigQuery on-demand?
A: Set custom quotas on the project — `QueryUsagePerDay` per user or per project (bytes processed per day) in IAM/quota settings — so BigQuery rejects further queries once a user or project hits the daily byte limit. This caps aggregate analyst spend, complementing per-query `maximum_bytes_billed`. The gotcha is that the quota is a daily reset, so it bounds the day's damage, not a single catastrophic query (which `maximum_bytes_billed` handles).

Q: Why is loading EMT/EPT files as many small daily CSVs into Synapse serverless an expensive anti-pattern?
A: Serverless SQL bills per terabyte processed, and querying many small, uncompressed CSV files forces the engine to read far more bytes (no columnar projection, no compression, per-file overhead) than the same data in partitioned Parquet. A month-end report over a year of daily CSVs processes enormous bytes. The fix is to compact to Parquet partitioned by reporting date; the gotcha is that the row count looks small while the bytes processed are huge.

Q: How does the `OPENROWSET` column list affect Synapse serverless cost?
A: Specifying explicit columns and a `WITH` schema in `OPENROWSET` lets the engine read only those columns from Parquet (projection pushdown), processing fewer bytes than `SELECT *`, which reads everything. Pointing the path at a partitioned folder and filtering on the folder pattern prunes files too. The gotcha is that on CSV there is no column projection, so format choice dominates — Parquet is what makes projection possible.

Q: How does pausing a Synapse dedicated SQL pool differ operationally from Snowflake auto-suspend?
A: A dedicated pool must be explicitly paused (manually or via an automation/schedule) to stop the hourly DWU charge, and it does not auto-pause on idle; resuming takes minutes and the data stays in place. Snowflake auto-suspends a warehouse after a short idle period and auto-resumes on the next query in seconds. The gotcha is that forgetting to pause a dedicated pool is the classic Synapse cost leak, which Snowflake's self-correcting auto-suspend largely avoids.

Q: How does a Microsoft Fabric capacity "smooth" and "burst" compute, and why does that complicate cost attribution?
A: Fabric smooths a workload's Capacity Unit consumption over time and can burst above the SKU's nominal CUs briefly, then throttle if sustained demand exceeds the purchased capacity (F-SKU). This means a short heavy query can borrow future capacity rather than fail. It complicates attribution because a single F-SKU's CUs are pooled across all Fabric workloads, so warehouse spend is not cleanly isolated like a Snowflake warehouse's credits.

Q: Why does the Fabric Warehouse storing tables as Delta/Parquet in OneLake matter for a multi-engine fund platform?
A: Because the same physical Parquet files in OneLake are directly readable by Spark, the SQL endpoint, and — via metadata translation — by Iceberg consumers like Snowflake, so one governed copy serves many engines without export. This open-format storage is what supports a DORA-friendly exit and multi-engine BI. The gotcha versus a proprietary store is that you must still govern the single copy carefully, since many engines can touch it.

Q: For a Luxembourg fund administrator, which concrete EU regions would you check exist for each of the three engines?
A: You verify in-EU regions and data residency commitments — e.g. Snowflake on AWS/Azure/GCP EU regions (such as Frankfurt, Dublin, or Paris), BigQuery EU multi-region and single EU regions (e.g. `europe-west` locations), and Azure/Fabric EU regions — and confirm the data and metadata stay in-EU. Luxembourg-specific or nearest-EU residency plus contractual GDPR/DORA terms is the real requirement. The gotcha is that a global control plane or US-based support access can still raise residency concerns even with EU storage.

Q: Under DORA, why is an engine's open-format (Iceberg) support evidence for a "tested" exit plan rather than just a paper one?
A: DORA expects firms to demonstrate they can actually exit a critical ICT provider, and if holdings/NAV/transaction data sits in open Iceberg you can repoint a second compliant engine at the same files and run a real exit test, rather than asserting portability you have never exercised. A proprietary store forces a costly export/reload that few firms ever test. The gotcha is that the catalog must also be portable, not just the data files, for the exit to be genuinely engine-agnostic.

Q: How does dynamic data masking let one Snowflake holdings table serve both privileged and restricted analysts without copies?
A: A masking policy attached to a column (e.g. investor name or account number) is evaluated at query time against the querying role, returning the real value to authorized roles and a masked value to others, all from the single underlying table. This avoids maintaining separate redacted copies for different audiences, which suits transfer-agency PII rules. The gotcha is that masking is enforced at query time by role, so misconfigured role grants can leak the unmasked value.

Q: How does row-access security in BigQuery and Snowflake differ in mechanism from column masking?
A: Row-level security (BigQuery row-access policies / Snowflake row-access policies) filters which rows a role can see via a predicate, so one client's analysts never see another client's NAV rows, while column masking obfuscates specific column values within rows a user can already see. They compose: row policy restricts the rows, masking redacts sensitive columns in those rows. The gotcha is testing both together, since a row a user cannot see cannot leak, but a visible row's PII still needs masking.

Q: Why is benchmarking warehouse pruning locally in DuckDB over Parquet a fair proxy but not identical to Snowflake?
A: DuckDB applies projection and predicate pushdown using Parquet row-group min/max statistics, so you can observe how a well-laid-out file skips row groups — the same principle as Snowflake micro-partition pruning — and measure bytes/rows read under different layouts on your Ubuntu laptop with `uv run`. It is a fair intuition-builder for storage layout. The gotcha is that DuckDB's row groups, statistics granularity, and automatic clustering differ from Snowflake's managed micro-partitions, so absolute numbers do not transfer, only the qualitative behaviour.
