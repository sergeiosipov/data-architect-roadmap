# Phase 3 · 9.7.1 Schema migrations & change management — 150 self-test questions

Q: What problem do schema-migration tools like Flyway and Alembic solve?
A: They turn ad-hoc, manually-run `ALTER` statements into versioned, ordered, repeatable scripts that are applied automatically and tracked in a history table. The mechanism is that each script is recorded after it runs, so the tool knows exactly which changes a given database has and which still need applying. Without this you get drift between environments and no audit trail — fatal in a regulated fund-administration estate where every schema change must be evidenced to a change board.

Q: In Flyway, what is a versioned migration and how is it named?
A: A versioned migration is a script that runs exactly once, in version order, named `V<version>__<description>.sql` — for example `V2__add_isin_check_digit.sql`. The `V` prefix, the version number, a double underscore, the description, and the `.sql` suffix are all required parts of the convention. The double underscore separating version from description is the most common naming gotcha; a single underscore makes Flyway reject or ignore the file.

Q: What is a repeatable migration in Flyway and when does it re-run?
A: A repeatable migration uses the `R` prefix with no version number (e.g. `R__refresh_nav_view.sql`) and is re-applied every time its checksum changes, always after all pending versioned migrations. The mechanism is that Flyway compares the file's current checksum against the last-applied checksum stored in the history table; if they differ it re-runs. Repeatables are ideal for objects you redefine in place — views, stored procedures, grants — where the latest definition is what matters, not a sequence of deltas.

Q: What is Flyway's schema history table called by default?
A: It is `flyway_schema_history`, created automatically in the target schema on first run. It stores one row per applied migration with the version, description, type, script name, checksum, who applied it, when, execution time, and success flag. This table is the single source of truth Flyway consults to decide what to run, and in a regulated shop it doubles as the CAB evidence of exactly what changed and when.

Q: What is a migration checksum in Flyway and what is it for?
A: A checksum is a hash Flyway computes over a migration script's contents and stores in `flyway_schema_history` when the script is applied. Its purpose is tamper detection: on every `migrate` or `validate`, Flyway re-hashes the on-disk file and compares it to the stored value, so editing an already-applied migration is caught. This enforces the rule that applied migrations are immutable — you fix forward with a new migration, never by editing history.

Q: Your CI pipeline fails with `Migration checksum mismatch for migration version 3`. What happened?
A: Someone edited the contents of `V3` after it had already been applied to that database, so the on-disk checksum no longer matches the one recorded in `flyway_schema_history`. The first thing to check is the git history of that file — did a teammate "just tweak" an applied migration? The correct fix is almost always to revert the edit and add a new `V4` migration; `flyway repair` only realigns the stored checksum and should be used with care.

Q: What does `flyway validate` do, and how does it differ from `flyway migrate`?
A: `flyway validate` checks that the migrations on disk match what the schema history table says was applied — same versions, same checksums, no missing or extra scripts — and fails without changing anything. `flyway migrate` actually applies any pending migrations and records them. You run `validate` as a fast, read-only guard early in CI so a breaking change is caught before it touches a database.

Q: What does `flyway repair` do?
A: `flyway repair` realigns the schema history table with the migration files: it updates stored checksums to match the current files and removes failed-migration entries so they can be retried. The mechanism makes it useful when a migration failed mid-way or when a checksum mismatch is genuinely benign (e.g. a whitespace-only change). It is a sharp tool — repairing to hide a real, applied content change destroys the integrity guarantee, so in a governed pipeline its use should itself be reviewed.

Q: What is a Flyway baseline migration and when do you use it?
A: A baseline establishes a starting point for Flyway on a database that already has a schema not under Flyway control, marking everything up to a baseline version as already applied. You use it exactly once when onboarding an existing Phase-1 Postgres schema: `flyway baseline` records the current state so future `V` migrations apply on top without Flyway trying to recreate the world. The baseline version is recorded in `flyway_schema_history` as a special baseline row.

Q: Why are already-applied Flyway migrations treated as immutable?
A: Because their checksums are recorded and validated, and because other environments may have already applied the original version — editing it would make environments silently diverge. The enforced model is that schema change is append-only: every correction is a new migration with a higher version. This is what lets you reason about any database purely from its history table and guarantees the same sequence ran everywhere.

Q: What does Flyway do when it finds a migration file with a version lower than the latest applied one?
A: By default Flyway treats it as "out of order" and refuses to run it, because applying an older version after a newer one breaks the linear-history assumption. You can opt in with `flyway migrate -outOfOrder=true`, which tells Flyway to apply any pending lower-versioned migrations it finds. This typically arises when two feature branches both add migrations and merge in an unexpected interleaving.

Q: What is the `validateMigrationNaming` configuration option in Flyway?
A: It controls how Flyway handles files in the migrations directory whose names do not match the expected pattern. When `false` (default), such files are silently ignored; when `true`, Flyway fails fast and lists every misnamed file so you can fix it. Turning it on is good hygiene in CI because a typo like a single underscore would otherwise cause a migration to be silently skipped.

Q: In Flyway terminology, what does it mean that a versioned migration is applied "exactly once"?
A: It means once a `V` migration's row exists in `flyway_schema_history` with success, Flyway never runs it again on that database, even if the file is still present. The mechanism is the version-number bookkeeping: Flyway only applies versions greater than the latest applied version (plus any out-of-order ones if enabled). This is the core distinction from repeatable migrations, which re-run on checksum change.

Q: Why might you choose a repeatable migration over a versioned one for a database view?
A: Because a view's correct state is its latest full definition, not a chain of deltas — putting `CREATE OR REPLACE VIEW` in an `R__` file means each change is a single edit re-applied on checksum change, with no growing pile of `V` files. The mechanism is that Flyway re-runs repeatables after versioned ones whenever the checksum differs. The gotcha is that repeatables must be idempotent (use `CREATE OR REPLACE`, not bare `CREATE`) or the re-run fails.

Q: What information does each row of `flyway_schema_history` contain?
A: Each row records the installed rank, version, description, type (SQL, BASELINE, etc.), script filename, checksum, the OS/DB user who installed it, an installation timestamp, execution time in milliseconds, and a success boolean. This per-migration provenance is precisely what a change board wants as evidence. A failed migration leaves a row with `success = false`, which is why `flyway repair` removes those before a retry.

Q: What is the `U` prefix in Flyway, and what is the catch?
A: `U` denotes an undo migration — the explicit down-script paired with a `V` migration to reverse it. The catch is that undo migrations are a commercial Flyway feature (Teams/Enterprise), not available in the free community edition. Even where available, the discipline-driven view is that you rarely run them in production; you roll forward instead.

Q: How does Flyway decide the order in which to apply migrations?
A: It sorts versioned migrations by their version number (numerically, segment by segment), applies all pending ones in ascending order, then applies any repeatable migrations whose checksum changed. The version string can have multiple dot- or underscore-separated parts, compared left to right. This deterministic ordering is what guarantees every environment receives the same sequence of changes.

Q: What is Alembic, and which ecosystem is it native to?
A: Alembic is the schema-migration tool for SQLAlchemy, the dominant Python ORM, written by the same author. It is the natural choice for a Python-first data shop because migrations are Python files with `upgrade()` and `downgrade()` functions and can be autogenerated from your SQLAlchemy models. It plays the same role Flyway does for the JVM/SQL world: versioned, tracked, ordered schema change.

Q: What does `alembic revision --autogenerate -m "add lei column"` do?
A: It compares your SQLAlchemy model metadata against the current database schema and emits a new migration script containing the detected differences, with the message as the description. The `--autogenerate` flag is what triggers the model-vs-database diff; without it you get an empty revision skeleton to fill in by hand. The output is a draft you must review, because autogenerate cannot detect everything.

Q: Why must Alembic autogenerated migrations always be reviewed before use?
A: Because autogenerate's diff is incomplete and sometimes wrong: it can miss table or column renames (seeing a drop-plus-add instead), server defaults, check constraints, and certain type changes, and it may reorder operations unsafely. The mechanism is that it compares declarative metadata to reflected schema, which loses intent — a rename looks identical to a drop-and-create. Treating the output as a first draft, not a final script, is the documented expectation.

Q: How does Alembic track which migrations a database has applied?
A: It maintains a table called `alembic_version` holding a single row with the revision identifier the database is currently at. Unlike Flyway's full history table, this is a pointer to the current head, and the chain of `down_revision` links inside the scripts encodes the order. To move forward, Alembic walks from the stored revision toward the head, running each `upgrade()` in turn.

Q: What is Alembic's offline (`--sql`) mode and why does it matter in a bank?
A: Offline mode, invoked by adding `--sql` to an `upgrade` or `downgrade` command, makes Alembic emit the SQL to stdout/a file instead of executing it against the database. It matters because in regulated estates DBAs often hold exclusive DDL rights — the developer generates the SQL script, hands it off for review, and a DBA applies it under change control. This separates authoring the change from executing it, which is exactly the segregation a CAB expects.

Q: What command generates an Alembic migration as a SQL script up to a specific revision?
A: `alembic upgrade <revision> --sql`, for example `alembic upgrade ae1027a6acf --sql > change_0042.sql`. The `--sql` flag switches Alembic into offline mode so it prints DDL rather than connecting and running it. You can also use a range like `alembic upgrade <from>:<to> --sql` to script just the delta between two revisions.

Q: What kind of migration logic breaks in Alembic offline mode, and why?
A: Any migration that reads from the database at runtime — e.g. a data backfill that does a `SELECT` into Python and loops — breaks, because offline mode produces static SQL with no live connection. The mechanism is that `--sql` configures the context to write SQL to a stream instead of opening a cursor, so there is nothing to fetch from. Data migrations meant for offline use must therefore be expressed as set-based `UPDATE`/`INSERT ... SELECT` SQL, not row-by-row Python.

Q: In Alembic offline mode, why must you specify a starting revision?
A: Because without a live connection Alembic cannot read `alembic_version` to learn where the database currently is, so it cannot know which scripts to include. You supply the start explicitly, e.g. `alembic upgrade <start>:<head> --sql`, or have the script read `starting_rev` from configuration. The gotcha is that an offline script generated from `base` will include an `alembic_version` setup that conflicts if the table already exists.

Q: What does `alembic current` report?
A: It connects to the database and prints the revision identifier currently stored in `alembic_version` — i.e. the migration head that database is at. It is the online way to discover the starting point you would then feed into an offline `--sql` generation. In troubleshooting, mismatches between `alembic current` and `alembic heads` tell you the database is behind the latest code.

Q: What does `alembic stamp <revision>` do, and when is it appropriate?
A: `alembic stamp` sets the `alembic_version` pointer to a given revision without running any migration logic, marking the database as being at that point. It is appropriate when the schema already matches a revision — for instance after manually applying DDL, or when baselining an existing database — so Alembic's bookkeeping aligns with reality. Misusing it to skip a migration you should actually run leaves the schema and the version pointer inconsistent.

Q: What are the two functions in a typical Alembic migration script and what does each do?
A: `upgrade()` contains the operations to apply the change (e.g. `op.add_column(...)`), and `downgrade()` contains the operations to reverse it (e.g. `op.drop_column(...)`). Alembic calls `upgrade()` when moving forward to that revision and `downgrade()` when moving back past it. Autogenerate populates both, but you are responsible for verifying the `downgrade()` actually reverses the change correctly.

Q: How does Alembic know the order of migrations without version numbers like Flyway's?
A: Each revision file declares its own `revision` identifier and a `down_revision` pointing at its predecessor, forming a linked list (a DAG when branches exist). Alembic walks this chain from the database's current revision toward the head to determine what to run and in what order. This is why two parallel branches can create a "multiple heads" situation that must be resolved with a merge revision.

Q: You run `alembic upgrade head` and get "Multiple head revisions are present". What caused it and how do you fix it?
A: Two migrations were created with the same `down_revision` — typically two feature branches each added a revision off the same parent — so the history has two unmerged tips. You fix it with `alembic merge -m "merge heads" <rev1> <rev2>`, which creates a merge revision whose `down_revision` lists both heads, collapsing them back to one. After merging, `alembic upgrade head` is unambiguous again.

Q: Conceptually, what is the expand–contract (parallel change) pattern?
A: It is a technique for making a breaking schema change as a sequence of individually-safe, deployable steps so the system is never broken at any point. You expand by adding the new structure alongside the old, migrate writes and reads across, backfill existing data, then contract by removing the old structure — each step deployable independently. It is the standard way to change a schema under load without downtime, and it is named "parallel change" by Martin Fowler.

Q: What are the four logical phases of expand–contract for a column rename?
A: Expand: add the new column alongside the old; migrate writes: have the application (or a trigger) write both columns; backfill: copy existing data from old to new and switch reads to the new column; contract: drop the old column. Each phase is a separate deployable so that at every moment the running application and the schema are mutually compatible. The contract step is deliberately last and deliberately separate.

Q: Why is a naive `ALTER TABLE ... RENAME COLUMN` dangerous in a live system?
A: Because it is an instantaneous, atomic break: the moment it commits, any running code still referring to the old name fails, and any code expecting the new name fails until it commits a fraction earlier or later. There is no version of the deploy where both the old and new code work against that single schema. In a fund context this could take the transfer-agency feed down mid-dealing-window, which is exactly what expand–contract prevents.

Q: In the expand phase, how do you keep the old and new columns in sync?
A: Either dual-write from the application (every write sets both columns) or install a database trigger that copies writes from one column to the other automatically. The trigger approach is attractive when you cannot change all writers at once or there are out-of-band writers. Whichever you choose, the goal is that during the transition both columns always hold the same value so reads from either are correct.

Q: What is a backfill in the expand–contract context, and why is it needed?
A: A backfill is a bulk operation that copies existing rows' data from the old column to the new one, because dual-writing only keeps new and updated rows in sync — historical rows are still empty in the new column. It is typically a batched `UPDATE ... SET new = old WHERE new IS NULL`, done in chunks to avoid long locks. Reads can only switch to the new column safely once the backfill is complete and verified.

Q: Why is the contract step deployed last and as its own separate migration?
A: Because dropping the old column is only safe once nothing reads or writes it — that is only true after the new code is fully deployed and the backfill is done. Shipping contract early would break code that still references the old column; shipping it bundled with expand would mean a single migration that both adds and drops, leaving no safe intermediate state. Keeping it separate gives you a stable, fully-functional system to pause at before committing to the irreversible drop.

Q: Your "Done when" says explain what would break if V2 (expand) and V4 (contract) were a single migration. What is the answer?
A: Combining them would add the new column and drop the old one in one atomic step, so there is no window in which both columns coexist — the running application release, which still references the old column, breaks instantly on apply. The whole point of expand–contract is that intermediate state where old and new live side by side; merging V2 and V4 destroys it. You would be back to the naive rename you set out to avoid, with no zero-downtime path.

Q: In expand–contract, when is it safe to switch reads from the old column to the new one?
A: Only after the new column is being kept fully in sync (dual-write or trigger in place) and the historical backfill has completed and been verified. Switching reads earlier means queries can hit rows where the new column is still NULL, returning wrong NAV or holdings data. The ordering — sync first, backfill second, switch reads third — is what guarantees correctness throughout.

Q: How does expand–contract apply to changing a column's type rather than its name?
A: Same pattern: add a new column of the target type, dual-write/convert into it, backfill existing rows with the converted values, switch reads, then drop the old column. You never do an in-place `ALTER COLUMN ... TYPE` on a hot table because it can rewrite the whole table under a lock and breaks code expecting the old type. The new-column approach keeps both representations valid during the transition.

Q: Why can adding a `NOT NULL` column with no default be a breaking change, and how does expand–contract help?
A: On many databases adding a `NOT NULL` column without a default forces a value for every existing row and can require a full table rewrite or simply fail, and old code that inserts without the new column breaks. The expand–contract way is to add the column as nullable first, backfill it, ensure all writers populate it, then add the `NOT NULL` constraint in a later step. Each step is individually safe and the table is never broken for the running release.

Q: What is the deployment-ordering rule "DB before code"?
A: It means you apply the database migration before deploying the application code that depends on it, so the new code never encounters a schema it expects but that does not yet exist. The mechanism is that a migration making purely additive, backward-compatible changes is safe for the old code too, so applying it first harms nothing. This ordering is half of what makes zero-downtime deploys possible; the other half is making every migration backward-compatible.

Q: Why must every migration be backward-compatible with the currently-running release?
A: Because during a deploy there is always a window where the new schema is live but some instances still run the old code, and if the migration broke the old code those instances would error until they are replaced. Keeping each migration additive and non-breaking means old and new code both work against it. This is the invariant that lets you apply DB changes ahead of, and independently from, code rollout.

Q: How do "DB before code" ordering and expand–contract relate to each other?
A: Expand–contract decomposes a breaking change into a series of individually backward-compatible migrations, and "DB before code" is the rule for sequencing each of those migrations relative to its matching code deploy. Together they guarantee that at every point — before, during, and after each step — the live code and the live schema are compatible. Expand–contract supplies the safe steps; DB-before-code supplies the safe ordering of each step.

Q: In a blue-green or rolling deployment, why does backward-compatible schema change matter even more?
A: Because both deployments run old and new application versions simultaneously against the same database for a real interval, not just an instant. If a migration were backward-incompatible, the still-running old version would fail for every request during that interval. Backward-compatible (additive) migrations let both versions coexist, which is precisely what rolling and blue-green strategies require.

Q: What is "roll forward" as a rollback stance, and why is it preferred over rollback scripts?
A: Roll forward means that when a deployed change causes a problem, you fix it by writing and deploying a new migration that corrects the state, rather than running a down-script to revert. It is preferred because down-scripts are rarely tested, can lose data (a dropped column's data is gone), and the database may have moved on since deploy — a blind revert can cause more damage than the original fault. Forward-only keeps history append-only and auditable.

Q: Your "Done when" asks you to defend "roll forward" to a change board demanding a rollback script. What is your argument?
A: A rollback script is a comforting artifact that is almost never exercised, so it is untested code run in a crisis against a database whose state has drifted since deploy — the riskiest possible moment to run unverified DDL. A contract step that dropped a column cannot be undone by a down-script because the data is gone, so the real recovery is a corrective forward migration plus a restore from backup if data was lost. The board's actual goal — safe recovery — is better served by tested forward fixes, monitoring, and backups than by a script nobody has run.

Q: If you favour roll-forward, what is a tested down-path actually for?
A: It is primarily for local development and CI, where you spin schemas up and down repeatedly and a working `downgrade()` keeps iteration fast and reproducible. It is also useful in pre-production for resetting test data between runs. The point is that down-scripts earn their keep in disposable environments, not as the production recovery mechanism — production recovery is roll-forward plus backups.

Q: Why is dropping a column considered effectively irreversible, and what follows from that?
A: Because once the column is dropped the data it held is gone; a "downgrade" that re-adds the column gives you back an empty column, not the original values. It follows that the contract step deserves extra scrutiny, should only run after a verified backfill and a stable period, and that real recovery from a bad drop is a restore from backup, not a down-script. This is the clearest case where roll-forward beats rollback.

Q: What is the difference between Flyway's versioned and repeatable migrations in terms of when they execute?
A: Versioned migrations run once, in ascending version order, and never again once recorded. Repeatable migrations run after all versioned ones and re-execute whenever their checksum changes. The practical rule: deltas to data or table structure go in `V` files; full redefinitions of replaceable objects like views and functions go in `R` files.

Q: How would you put an existing Phase-1 Postgres schema under Flyway control without losing it?
A: Point Flyway at the database, create a baseline that records the current schema as the starting version (e.g. `flyway baseline -baselineVersion=1`), and write future changes as `V2`, `V3`, etc. The baseline tells Flyway "everything up to here already exists, do not try to build it," writing a baseline row into `flyway_schema_history`. From then on only new migrations apply on top, and the history table becomes your change record.

Q: Walk through the three migrations for renaming a loader-written column using expand–contract.
A: `V2` expand: add the new column plus a sync trigger or dual-write so both columns stay equal. `V3` backfill and switch: copy old values into new for existing rows, then change reads to use the new column. `V4` contract: drop the old column once nothing references it. Keeping the loader running on a schedule throughout, you watch for failed loads at each step; zero failures is the success criterion.

Q: While doing a live expand–contract rename, the scheduled loader starts failing right after V2. What do you check first?
A: First check whether the loader's `INSERT`/`UPDATE` is now incompatible with the new column — e.g. you added the new column as `NOT NULL` with no default, so inserts that omit it fail. The expand step must be purely additive: add the column nullable, with a default or a trigger to populate it, so existing writers keep working. Confirm the sync trigger itself is not raising an error on the loader's writes.

Q: Why is keeping the loader running throughout the rename drill the whole point of the exercise?
A: Because expand–contract's promise is zero downtime under live traffic; a rename that only works when the system is quiesced proves nothing. Running the loader on a schedule simulates the dealing-window traffic that a real transfer-agency feed produces, so any step that breaks compatibility shows up immediately as a failed load. Zero failed loads across all three migrations is the evidence that the pattern was executed correctly.

Q: What would a five-line CAB-style change record for this rename contain?
A: The change description (column rename via expand–contract over V2–V4), the risk assessment (low; each step additive and backward-compatible, loader stays up), the rollback stance (roll forward; contract step backed by backup), the evidence (zero failed loads, `flyway_schema_history` rows, before/after schema), and the approver/window. It is deliberately short and decision-oriented, mapping directly onto what a change advisory board signs off. The migration history table supplies the hard evidence behind it.

Q: What does Liquibase add over Flyway/Alembic in a governed enterprise pipeline?
A: Liquibase Pro adds capabilities aimed at governed delivery: drift detection (comparing actual schema against expected), targeted/structured rollback, approval flows, and policy checks, typically integrated into pipelines like Azure DevOps. These address exactly the controls a regulated change process wants beyond plain versioned migrations. For the lab you pick one of Flyway or Alembic; Liquibase is the enterprise tier you evaluate, not necessarily run.

Q: What is schema "drift" and why do enterprise migration tools detect it?
A: Drift is the divergence between the schema a database actually has and the schema your migrations say it should have — caused by manual hotfixes, out-of-band changes, or partial deploys. Tools detect it by comparing the live schema against the expected state derived from the migration history. It matters because undetected drift means your next migration runs against a database that is not what you think it is, which is a common cause of production migration failures.

Q: Why is a migration history table described as "CAB evidence" in a regulated estate?
A: Because it is an immutable, timestamped, per-change record of exactly what schema changes were applied, by whom, when, and with what result — which is precisely what a change advisory board needs to audit. Instead of trusting that a documented change was actually performed, an auditor reads `flyway_schema_history` directly. This is why the tool-managed history, not a wiki page, is the system of record for schema change.

Q: How do migrations fit into an evolutionary database design / CI-CD pipeline?
A: Each schema change is a small, versioned migration checked into source control alongside the application code that depends on it, and the pipeline applies pending migrations automatically as part of deploy. This treats the database as code: every change is reviewed, tested, ordered, and reproducible across environments. Fowler's evolutionary database design names the migration as the atomic unit of database change in a continuous-delivery flow.

Q: Why should migration scripts live in version control alongside application code?
A: So a given commit captures both the schema change and the code that needs it, keeping them reviewable and deployable together and giving you a single history. Reviewers see the `ALTER` next to the code that uses the new column, and CI can validate them together. It also means rolling back code and identifying the corresponding schema state is a matter of reading git, not archaeology.

Q: What is the risk of two developers creating Flyway migrations with the same version number?
A: Flyway will detect a duplicate version and refuse to run, because two migrations sharing a version violate the ordering and one-row-per-version contract of the history table. This is the Flyway analogue of Alembic's multiple-heads problem and usually surfaces at merge time. The fix is to renumber one migration to a unique, higher version before applying.

Q: How does Alembic's branching model differ from Flyway's strictly linear versioning?
A: Alembic explicitly supports branches in its revision DAG via `down_revision` links and even labelled branches, then resolves divergence with merge revisions; Flyway is fundamentally linear by version number, with out-of-order as an opt-in escape hatch. This makes Alembic more flexible for parallel feature work but also means "multiple heads" is a normal state you must actively merge. Flyway instead pushes you to keep one linear sequence.

Q: An autogenerated Alembic migration shows a `drop_column` plus `add_column` for what was meant to be a rename. Why, and what do you do?
A: Autogenerate diffs declarative metadata against the reflected schema and has no concept of identity, so a renamed column looks exactly like dropping the old name and adding a new one — and applying that drop-and-add would destroy the column's data. You must hand-edit the migration to use `op.alter_column(..., new_column_name=...)` (or the expand–contract sequence) instead. This is a canonical reason autogenerate output must be reviewed, never applied blind.

Q: Why does Alembic require a database connection for autogenerate but not necessarily for applying migrations offline?
A: Autogenerate must reflect the current live schema to diff it against your models, so it needs to read the database; applying migrations offline with `--sql` only needs to emit DDL, which is static text. The two phases serve different masters: authoring (online, needs reflection) versus execution (can be offline for DBA handoff). This is why a shop with locked-down production still autogenerates against a dev database, then ships SQL to the DBA.

Q: What is the danger of running data migrations and schema migrations in the same step under load?
A: A long-running data backfill inside a schema migration can hold locks or a transaction open for a long time, blocking the live loader and inflating apply time, and if it fails the whole migration may roll back. The mechanism is that DDL and bulk DML compete for the same locks and transaction. The safer pattern is to separate them: ship the additive DDL as one quick migration, then run the backfill as a batched, restartable operation.

Q: Why should a large backfill be done in batches rather than one big `UPDATE`?
A: A single `UPDATE` over millions of rows takes a long lock, generates a huge transaction/WAL, and on failure rolls back everything, blocking the live loader the whole time. Batching (e.g. `UPDATE ... WHERE id BETWEEN ... LIMIT n`, looped) keeps each transaction short, lets the loader interleave, and makes the operation restartable from where it stopped. In a fund estate this is the difference between a quiet backfill and a stalled dealing window.

Q: In expand–contract, what is "dual-write" and where in the code does it live?
A: Dual-write is application code in the expand phase that writes to both the old and the new column on every insert/update so they stay synchronized. It lives in the data-access layer of the application release that ships with or just after the expand migration. The alternative, a database trigger, moves the same responsibility into the schema, which is useful when you cannot change every writer.

Q: When is a database trigger preferable to application dual-write for keeping columns in sync?
A: When there are multiple or out-of-band writers (other services, manual SQL, a legacy ETL) that you cannot all update to dual-write, a trigger guarantees synchronization regardless of who writes. The trigger fires on every `INSERT`/`UPDATE` and copies the value across, centralizing the logic in the database. The trade-off is that triggers are easy to forget about and must themselves be removed in the contract phase.

Q: What must the contract migration in a rename also clean up besides dropping the old column?
A: It should remove any sync trigger or dual-write scaffolding added during expand, plus any temporary constraints, indexes, or defaults that existed only to support the transition. Leaving the trigger behind means it keeps firing against a now-dropped column and errors, or wastes work. The contract step is the cleanup step: it returns the schema to a clean single-column state.

Q: Why is "every migration backward-compatible" sometimes summarized as "additive-only during transitions"?
A: Because additive changes — adding nullable columns, new tables, new indexes — never break code that does not yet know about them, whereas destructive changes (drop, rename, tighten constraint) do. Restricting in-flight migrations to additive operations is the simplest way to guarantee the running release keeps working. Destructive operations are deferred to contract steps that run only after the old code is gone.

Q: How does expand–contract let you avoid ever taking a maintenance window for a column rename?
A: Because each of its steps — add column, sync, backfill, switch reads, drop column — is individually non-breaking for the currently running release, you never need a window where the system is offline. The system stays fully available and correct at every intermediate state, so the change is applied incrementally under live traffic. The cost is more steps and more deploys in exchange for zero downtime.

Q: What is the relationship between an Alembic `revision` id and a Flyway version number?
A: Both identify a migration and encode ordering, but a Flyway version is a human-chosen, monotonically increasing number, while an Alembic revision is an opaque generated hash whose order comes from `down_revision` links, not the id itself. Flyway's ordering is explicit in the filename; Alembic's is explicit in the script's parent pointer. This is why Flyway forbids duplicate versions while Alembic instead manages branches and heads.

Q: If `flyway migrate` reports "0 migrations applied" but you added a new file, what is the likely cause?
A: The most common causes are a misnamed file (single underscore instead of double, wrong prefix, or wrong extension) so Flyway ignores it, or the file sitting outside the configured `locations` directory. With `validateMigrationNaming` off, a misnamed file is silently skipped, which produces exactly this symptom. Check the filename pattern `V<n>__desc.sql` and the configured migration locations first.

Q: What does it mean that Flyway migrations are applied "transactionally", and what is the Postgres caveat?
A: Flyway wraps each migration in a transaction where the database supports transactional DDL, so a failing migration rolls back cleanly rather than leaving the schema half-changed. Postgres does support transactional DDL, which is a real advantage. The caveat is statements like `CREATE INDEX CONCURRENTLY` cannot run inside a transaction block, so such migrations need special handling or a non-transactional execution.

Q: Why can't `CREATE INDEX CONCURRENTLY` run inside a normal migration transaction in Postgres, and why use it anyway?
A: `CREATE INDEX CONCURRENTLY` is specifically designed to build an index without taking a long write lock, and Postgres forbids it inside a transaction block because it manages its own commits across multiple phases. You use it precisely to avoid locking a hot table while indexing under live load. The migration must therefore be configured to run outside a transaction, which both Flyway and Alembic support via options.

Q: How would expand–contract handle splitting one column into two (e.g. a combined name into first/last)?
A: Expand by adding the two new columns; dual-write or trigger to populate them on new writes; backfill by parsing existing values into the two columns; switch reads to the new columns; then contract by dropping the original. The backfill's parsing logic is the risky part and must be validated against edge cases (no last name, multiple spaces). As always the original column is dropped only after everything reads from the new ones.

Q: What is the role of asset checks or post-migration validation in a safe change?
A: After a backfill or switch, an automated check (row counts, a NAV reconciliation, "no NULLs in the new column") confirms the change produced correct data before you proceed to the irreversible contract step. The mechanism is to gate progression on green checks, so a bad backfill is caught before the old column is dropped. This is the data-quality analogue of "DB before code" — verify before you contract.

Q: Why is it safer to deploy the expand migration well before the code that uses the new column?
A: Because the additive migration is harmless to the old code, deploying it early de-risks the change: by the time the new code arrives, the schema is already in place and proven, so the code deploy carries no schema risk. It also gives you a clean checkpoint to halt at if something looks wrong. Bundling them tightly removes that safety margin and couples two independent risks.

Q: In Alembic, how do you express adding a column in the `upgrade()` function?
A: With the operations object, e.g. `op.add_column('holdings', sa.Column('lei', sa.String(20), nullable=True))`. Alembic's `op` API abstracts the DDL across dialects, and the matching `downgrade()` would call `op.drop_column('holdings', 'lei')`. Keeping the new column nullable here is exactly the additive, backward-compatible expand step the pattern requires.

Q: What is the danger of editing a migration after it has been merged and applied in CI but before production?
A: Production will compute a different checksum than what CI/staging recorded, so Flyway's validate fails there even though the schema looks fine — and environments now disagree about what `V_n` contains. The rule is that once a migration is applied anywhere shared, it is frozen; corrections go in a new migration. This is why fast-forward edits to "just fix a typo" in an applied migration cause environment-divergence incidents.

Q: How does the immutability of applied migrations support reproducibility across environments?
A: Because every environment applies the identical, never-edited sequence of migrations in the same order, any database can be rebuilt to a known state purely by replaying its history. Checksums enforce that the scripts have not changed, so "the same migrations" really means the same bytes. This determinism is what lets dev, CI, staging, and prod be trusted to have equivalent schemas.

Q: What is the practical difference between `alembic downgrade -1` and writing a forward fix migration?
A: `alembic downgrade -1` runs the previous revision's `downgrade()` to move the database back one step, useful in development; a forward fix is a brand-new revision that corrects state while leaving history intact, which is the production-preferred approach. Downgrade depends on a correct, tested `downgrade()` and can lose data; the forward fix is append-only and auditable. The two map exactly onto the down-path-for-dev vs roll-forward-for-prod distinction.

Q: Why might a `downgrade()` function be left unimplemented or raise in a production-focused shop?
A: Because the team has committed to roll-forward and considers production down-migrations unsafe, so some teams deliberately make `downgrade()` raise `NotImplementedError` to prevent anyone running an untested revert in production. The signal is intentional: there is no supported down-path, only forward fixes plus backups. It is a documented stance, not laziness, and forces correct recovery behaviour.

Q: How do you generate the SQL for just the next pending Alembic migration to hand to a DBA?
A: Find the current revision (`alembic current`), then run `alembic upgrade <current>:<next> --sql` (or `alembic upgrade +1 --sql`) to emit only that step's DDL as a script. The `--sql` flag puts Alembic in offline mode so it prints rather than executes. The DBA reviews and applies the script under change control, and afterwards the `alembic_version` row reflects the new head.

Q: What is the consequence of applying the contract step before all application instances have switched to the new column?
A: Any instance still reading or writing the old column will error the instant it is dropped, causing failed transactions or loads until those instances are replaced. This is the exact failure expand–contract exists to prevent, and it happens when contract is rushed ahead of the code rollout. The rule is contract runs only after the entire fleet runs code that uses the new column exclusively.

Q: Why does Flyway store who applied each migration and when?
A: To provide accountability and an audit trail: the `installed_by` and `installed_on` fields tell a change board exactly which user/account applied a change and at what time. In a regulated environment this provenance is mandatory evidence. It also helps incident response — you can correlate a schema change's timestamp with the onset of a problem.

Q: A migration partially failed and left a `success = false` row in `flyway_schema_history`. What now?
A: Flyway will refuse to proceed because the failed migration blocks the version sequence; you fix the underlying problem, then run `flyway repair` to remove the failed row, then re-run `flyway migrate`. On databases without transactional DDL you may also need to manually undo partial effects before retrying. With Postgres's transactional DDL, the failed statement usually rolled back, so cleanup is mostly the history-table repair.

Q: How does expand–contract interact with NOT NULL and CHECK constraints you eventually want?
A: You defer the strict constraint to a late step: add the column nullable, backfill and ensure all writers populate it, verify there are no violating rows, then add the `NOT NULL` or `CHECK` as a separate migration. Adding the constraint early would break inserts from old code or fail outright against existing NULLs. Tightening last keeps every intermediate state valid for the running release.

Q: Why is `ADD COLUMN ... DEFAULT` historically risky on large tables, and how is it mitigated?
A: On older Postgres versions, adding a column with a non-null default rewrote the entire table to populate the default, taking a long lock — dangerous on a large hot table. Modern Postgres optimizes constant defaults to avoid the rewrite, but volatile defaults still rewrite. The mitigation, and the expand–contract way, is to add the column nullable, backfill in batches, then set the default/constraint separately so you control the locking.

Q: What is the "schema as code" idea and how do migration tools embody it?
A: It is the principle that the database schema is defined and evolved through versioned, reviewed, tested artifacts in source control, exactly like application code. Migration tools embody it by making every change a checked-in script applied deterministically by the pipeline, with a history table as the audit log. This replaces hand-run `ALTER` statements and tribal knowledge with a reproducible, governed process.

Q: How would you reconcile a Flyway-managed schema against drift before a risky migration?
A: Compare the live schema against the expected state — using a schema-diff tool (or Liquibase's drift detection in the enterprise tier) — and resolve any out-of-band changes before applying new migrations. The mechanism is to ensure the database actually matches what the history table implies, so the new migration runs against the assumed starting point. Skipping this is a classic cause of "it worked in staging but failed in prod."

Q: Why is it important that a migration to add an index does not block the loader, and how do you ensure that?
A: A plain `CREATE INDEX` takes a lock that blocks writes to the table for the duration, which would stall the live loader on a large table. Using `CREATE INDEX CONCURRENTLY` builds the index without that write lock, at the cost of running outside a transaction and taking longer. Configuring the migration tool to run the statement non-transactionally is what makes the additive index change truly zero-impact.

Q: In the fund context, why is a mid-dealing-window schema break especially serious?
A: During the dealing window the transfer-agency feed and NAV pipeline are actively processing subscriptions, redemptions, and prices, so a schema break causes failed loads exactly when timely, correct NAV depends on them. A delayed or wrong NAV has regulatory and investor-facing consequences, not just an engineering inconvenience. This stakes-raising is precisely why expand–contract and backward-compatible ordering are non-negotiable, not optional polish.

Q: How does the choice between Flyway and Alembic typically follow from the team's stack?
A: A JVM or polyglot SQL-centric team usually picks Flyway because migrations are plain SQL files and it integrates cleanly with non-Python deploys; a Python/SQLAlchemy shop picks Alembic because it autogenerates from models and migrations are Python. Both deliver versioning, ordering, history, and offline SQL handoff. The lesson's lab has you standardize on one, with the concepts transferring directly to the other.

Q: What is the difference between Flyway's `clean` command and a migration, and why is `clean` dangerous?
A: `flyway clean` drops all objects in the configured schemas, wiping the database back to empty — it is a development convenience, not a migration. It is dangerous because run against the wrong target it destroys everything, which is why Flyway lets you disable it via `cleanDisabled` in non-dev environments. In any shared or production-like environment `clean` should be turned off so an accidental invocation cannot wipe the schema.

Q: Why should `flyway clean` be disabled in production configuration?
A: Because it irreversibly drops every object in the managed schemas, and a misconfigured connection string or a stray command could obliterate production data. Setting `cleanDisabled=true` makes Flyway refuse the command outright, removing the foot-gun. This is a standard safety control in regulated estates where no tool should be capable of one-command schema destruction.

Q: How does offline SQL generation support segregation of duties in a regulated shop?
A: The developer authors and tests the migration and generates the SQL with `--sql` (Alembic) or Flyway's dry-run/script output, but a separate DBA reviews and executes it against production. This separates the person who designs the change from the person who applies it, satisfying segregation-of-duties controls. The generated, reviewable SQL artifact is the hand-off boundary between the two roles.

Q: What is the risk of an autogenerated migration silently dropping data through a detected "type change"?
A: Autogenerate may render a type change as an `ALTER COLUMN ... TYPE` that the database implements as a lossy cast or even a drop-and-recreate, discarding data without warning. Because the diff sees only the desired end state, it cannot reason about safe conversion. You must review such operations and, where loss is possible, rewrite them as an explicit expand–contract with a controlled, verified backfill.

Q: How do you verify "zero failed loads" after a live expand–contract change?
A: Inspect the loader's run logs/metrics across the entire change window for any errors, and confirm the loaded row counts and NAV reconciliations stayed green through V2, V3, and V4. Cross-check against `flyway_schema_history` timestamps to confirm the loads spanning each migration succeeded. Zero error rows plus matching reconciliations is the evidence the rename was genuinely zero-downtime.

Q: How does Flyway find migration scripts, and what is the `locations` setting?
A: Flyway scans the directories (and classpath locations) listed in its `locations` configuration for files matching the migration naming pattern. If a migration appears not to run, a misconfigured `locations` pointing at the wrong folder is a common cause, alongside a misnamed file. In the lab you point `locations` at the `sql` directory holding your `V`/`R` scripts so Flyway picks them up deterministically.

Q: Summarize why expand–contract plus DB-before-code plus roll-forward together constitute "change management without drama".
A: Expand–contract breaks a scary change into individually safe steps; DB-before-code with backward-compatible migrations sequences each step so the live system never sees an incompatible schema; roll-forward gives a tested, append-only recovery model backed by backups instead of untested down-scripts. The result is that even a destructive change like a column rename happens under live traffic with zero failed loads and a complete history-table audit trail. That combination is exactly what lets a regulated estate satisfy a change advisory board while keeping the dealing window open.

Q: What does `flyway info` show, and how is it different from `flyway validate`?
A: `flyway info` prints a table of every migration with its version, description, type, installed date, and state — `Success`, `Pending`, `Ignored`, `Out of Order`, `Missing`, etc. — giving a human-readable status of the database versus the scripts. Unlike `validate`, which fails the build on a mismatch and is meant for CI gates, `info` is purely informational and never errors on drift. You run `info` first when troubleshooting "why didn't my migration apply" because it shows at a glance whether the file is `Pending` or being silently `Ignored`.

Q: In `flyway info` output, what does the `Pending` state mean versus the `Ignored` state?
A: `Pending` means the migration has a version higher than the current schema version and will be applied on the next `migrate`; `Ignored` means Flyway found the file but is deliberately skipping it — typically a lower-versioned file when `outOfOrder` is off, or a file excluded by `cherryPick`. The distinction matters in troubleshooting because a migration you expect to run showing as `Ignored` points to an ordering or configuration cause, not a naming typo. Seeing the new `V5` as `Pending` confirms Flyway recognized it and it is simply queued.

Q: What is the Flyway `cherryPick` setting and what is the danger of using it?
A: `cherryPick` lets you specify an explicit array of migration versions to consider during `migrate`, `undo`, or `repair`, overriding the default linear ordering so you can apply (or skip) selected migrations. The danger is that skipping migrations leaves gaps — skipped scripts show a `Skipped` state in `flyway info` and the schema no longer reflects the full linear history, breaking the reproducibility guarantee. It is a Teams/Enterprise feature meant for narrow release-management scenarios, not routine use, and in a regulated estate it undermines the "every environment ran the same sequence" property auditors rely on.

Q: What are Flyway placeholders and what is the default syntax?
A: Placeholders are tokens in a migration script that Flyway substitutes at apply time, written by default as `${placeholder_name}` and supplied via configuration like `flyway.placeholders.schemaName=fund_silver`. They let one script target different schemas or environments without duplicating files — e.g. `GRANT SELECT ON ${schemaName}.holdings TO reporting`. The gotcha is that placeholder replacement changes the script text, which means it also changes the checksum, so the same `V` file resolved with different placeholder values is effectively a different migration across environments.

Q: How does using Flyway placeholders interact with checksum validation across environments?
A: Because Flyway computes the checksum on the resolved script after substitution by default, a migration that uses `${env}` will have a different checksum in dev than in prod, which can trip `validate` if you are comparing across environments. You manage this by keeping placeholder values consistent per environment and understanding that the history table's checksum is environment-local, not globally identical. The cleaner alternative for schema-name differences is often Flyway's `defaultSchema`/`schemas` configuration rather than placeholders inside DDL.

Q: What does Flyway's `baselineOnMigrate` setting do, and how does it differ from running `flyway baseline` manually?
A: `baselineOnMigrate=true` makes Flyway automatically baseline a non-empty schema that lacks a `flyway_schema_history` table on the first `migrate`, treating everything up to `baselineVersion` (default `1`) as already applied. The difference from a manual `flyway baseline` is automation and risk: it silently assumes any populated schema is at the baseline, which is convenient for first-time onboarding but dangerous if pointed at an unexpected database. In a governed pipeline you usually prefer the explicit one-time `flyway baseline` so the baselining is a deliberate, evidenced act rather than an implicit side effect.

Q: You set `baselineOnMigrate=true` against an empty database — what happens?
A: Nothing special: `baselineOnMigrate` only triggers when the schema is non-empty and has no history table, so against a truly empty database Flyway simply creates the history table and applies all migrations from the start. The behaviour is conditional precisely so a fresh build runs the full sequence while an existing schema gets baselined. The pitfall is the reverse case — leaving it on can mask the fact that a database you thought was fresh actually had stray objects, which then get assumed-applied.

Q: What is the difference between Flyway's `schemas`, `defaultSchema`, and where the history table lives?
A: `schemas` lists the schemas Flyway manages (and will `clean`); `defaultSchema` names the one used for unqualified objects and, by default, where `flyway_schema_history` is created. If you do not set them, Flyway uses the connection's default schema for the history table. Getting this wrong is a classic cause of "Flyway can't find its history" or of two projects fighting over one history table — each logically separate schema set should have its own history table.

Q: How do you configure Flyway to point at a Postgres database, and what are the three core connection parameters?
A: You set `flyway.url` (a JDBC URL like `jdbc:postgresql://localhost:5432/funddb`), `flyway.user`, and `flyway.password`, either in `flyway.conf`, as environment variables (`FLYWAY_URL` etc.), or as `-url=` command-line flags. The JDBC URL form is the gotcha for engineers used to libpq strings — Flyway is JVM-based, so it needs the `jdbc:postgresql://` prefix, not a `postgresql://` SQLAlchemy-style DSN. Missing or wrong driver in the URL produces a "no suitable driver found" error rather than an auth failure.

Q: In Alembic, what is `target_metadata` in `env.py` and why must it be set for autogenerate?
A: `target_metadata` is the SQLAlchemy `MetaData` object — typically `Base.metadata` — that Alembic diffs against the reflected database to detect changes during `--autogenerate`. If it is left as `None` (the default in a fresh `env.py`), autogenerate produces empty migrations because there is no model side to compare. Wiring it up by importing your models so all tables are registered on that metadata is the single most common fix for "autogenerate generates nothing."

Q: Your Alembic autogenerate produces an empty `upgrade()` even though you changed a model — what do you check first?
A: First check that the changed model is actually imported so its table is registered on the `MetaData` assigned to `target_metadata` in `env.py`; an unimported model is invisible to the diff. Then confirm `env.py`'s `target_metadata` points at the right `Base.metadata` and that you are connected to the database whose schema you intend to diff against. A model defined but never imported is the textbook cause of silently empty autogenerated migrations.

Q: What does Alembic's `compare_type` option control, and what is its default?
A: `compare_type` in `context.configure()` controls whether autogenerate detects column type changes; it defaults to `True`, so type changes are picked up with good accuracy on most backends. You can set it to `False` to suppress noisy type diffs, or pass a callable for custom comparison logic when using user-defined types. The catch is that even with it on, some dialect-specific type nuances are missed, which is another reason autogenerated type changes must be reviewed by hand.

Q: What does Alembic's `compare_server_default` option do, and why is it off by default?
A: `compare_server_default` controls whether autogenerate detects changes to a column's server-side `DEFAULT`; it defaults to `False` because server-default comparison is dialect-sensitive and prone to false positives. When enabled, some backends like Postgres actually execute both defaults to test equivalence, which is more accurate but heavier. You turn it on deliberately when default drift matters, accepting that you may then have to hand-tune or ignore spurious diffs.

Q: What is Alembic batch mode (`op.batch_alter_table`) and which database makes it necessary?
A: Batch mode wraps a set of `alter_column`/`drop_column`/constraint operations so Alembic can emulate `ALTER` operations that the backend does not support natively — chiefly SQLite, whose `ALTER TABLE` only adds or renames columns. Under the hood on SQLite it creates a new table with the target shape, copies the data across, drops the original, and renames the new one. It matters if your dev or test environment is SQLite even when production is Postgres, because a plain `op.drop_column` that works in Postgres will fail on SQLite without batch mode.

Q: Why might an Alembic migration that runs on Postgres fail on SQLite, and how do you make it portable?
A: SQLite supports only a narrow subset of `ALTER TABLE` (add/rename column), so operations like dropping a column, altering a type, or adding a constraint raise an error there even though Postgres handles them directly. You make the migration portable by wrapping those operations in `with op.batch_alter_table('holdings') as batch_op:` so Alembic falls back to the copy-and-replace strategy on SQLite while still emitting a direct `ALTER` on Postgres. This is a real concern when teams use SQLite for fast local tests against a Postgres production schema.

Q: What does `op.execute()` do in an Alembic migration, and when do you reach for it?
A: `op.execute()` runs an arbitrary SQL string (or SQLAlchemy construct) directly, bypassing the dialect-abstracted `op` helpers. You reach for it when you need DDL or DML that the `op` API does not model — a `CREATE INDEX CONCURRENTLY`, a vendor-specific statement, or a set-based backfill `UPDATE`. The trade-off is that raw SQL in `op.execute()` is not dialect-portable, so it can undermine the cross-database neutrality that `op.add_column` and friends give you.

Q: How do you express a relative move in Alembic, such as upgrading one step or downgrading one step?
A: Use `+N`/`-N` relative specifiers: `alembic upgrade +1` applies the next pending revision and `alembic downgrade -1` reverts the most recent one. You can also combine them with a base, e.g. `alembic upgrade ae10:+2`. The relative form is handy in scripts and DBA hand-offs where you want "just the next change" without looking up the opaque revision hash, but it depends on an unambiguous single head.

Q: What does `alembic history` show, and how do you read its arrows?
A: `alembic history` lists revisions with their `down_revision -> revision` relationships, descriptions, and which one is the current head, so you can see the migration chain top to bottom. The arrow notation shows lineage; branch points and merge revisions appear where multiple revisions share a parent or a merge has two parents. You use it alongside `alembic current` and `alembic heads` to diagnose multiple-heads situations and to find the revision range to script offline.

Q: What is the practical difference between `alembic heads` and `alembic current`?
A: `alembic heads` lists the tip(s) of the revision tree as defined by the migration files on disk — more than one means unmerged branches — while `alembic current` connects to the database and reports the revision stored in `alembic_version`. When `current` is behind `heads`, the database has pending migrations; when there are multiple `heads`, you must merge before `upgrade head` is unambiguous. Comparing the two is the standard "is this database up to date and is the history clean" check.

Q: How do you create a merge revision in Alembic without specifying the heads by hand?
A: Run `alembic merge -m "merge heads" heads`, where the literal token `heads` tells Alembic to merge all current heads automatically, producing a merge revision whose `down_revision` is a tuple of both tips. This avoids copying opaque hashes and is the quickest resolution for a two-branch divergence. After the merge, `alembic upgrade head` again resolves to a single, unambiguous target.

Q: What is a `branch_label` in Alembic and when is it useful?
A: A `branch_label` is an optional human-readable name attached to a revision that begins a branch, letting you refer to that branch by name (e.g. `alembic upgrade reporting@head`) instead of by hash. It is useful when you intentionally maintain parallel migration lineages — for example a shared core schema plus a per-tenant or per-mart branch. Most single-line projects never need it, but it is the mechanism that makes Alembic's DAG genuinely multi-branch rather than just accidentally-divergent.

Q: How do you run an Alembic data backfill safely as part of a migration without holding one giant transaction?
A: Express the backfill as batched set-based statements via `op.execute("UPDATE holdings SET lei = ... WHERE id BETWEEN :lo AND :hi")` in a loop, or better, keep the additive DDL in the migration and run the backfill as a separate, restartable script outside Alembic. Inside a single migration, a row-by-row Python loop reading via the connection both breaks offline `--sql` mode and risks a long-held transaction. The safe split is fast additive DDL in the migration, bulk backfill as its own controlled job.

Q: In Alembic, how do you run a statement like `CREATE INDEX CONCURRENTLY` that cannot be inside a transaction?
A: You disable the per-migration transaction for that revision by setting `transactional_ddl = False` in `env.py` for the relevant configuration, or by isolating the concurrent-index migration so it runs with autocommit, then issue it via `op.execute("CREATE INDEX CONCURRENTLY ...")`. Postgres rejects `CONCURRENTLY` inside a transaction block, so without disabling the wrapping transaction the migration errors. Keeping that statement in its own migration avoids forcing every other change to run non-transactionally.

Q: What is the difference between `alembic upgrade head` and `alembic upgrade heads`?
A: `alembic upgrade head` requires a single unambiguous head and errors with "Multiple head revisions are present" if there are unmerged branches, whereas `alembic upgrade heads` (plural) upgrades to all heads, applying every branch's tip. You normally want the singular form so a stray second head is caught rather than silently both-applied. Reaching for the plural is a smell that you should be merging the branches instead.

Q: A teammate added a Flyway migration as `V2_add_lei.sql` and it never runs — what is wrong?
A: The filename uses a single underscore between version and description; Flyway requires a double underscore, so it should be `V2__add_lei.sql`. With `validateMigrationNaming` off (the default) the misnamed file is silently ignored, producing exactly the "0 migrations applied" symptom with no error. The fix is renaming to the `V<version>__<description>.sql` pattern, and turning on `validateMigrationNaming` in CI so the next typo fails loudly.

Q: How do `outOfOrder=true` and a `Missing` state in `flyway info` relate?
A: A `Missing` state means the history table records a migration that no longer exists on disk (someone deleted the file), whereas `outOfOrder=true` is what lets a newly-added lower-versioned migration actually apply instead of being `Ignored`. They address different problems: `Missing` is a validation concern about deleted history, while `outOfOrder` is about back-filling a version gap from parallel branches. Confusing them leads people to enable `outOfOrder` hoping to fix a `Missing` script, which it does not.

Q: Two branches merge and Flyway reports a duplicate version `V3` — what is the resolution, and how does it differ from Alembic's equivalent?
A: In Flyway you renumber one of the conflicting files to a unique higher version (e.g. `V4`) before applying, because Flyway's linear model forbids two migrations sharing a version. Alembic's equivalent — two revisions off the same `down_revision` — is resolved not by renumbering but by creating a merge revision that joins the two heads. The contrast reflects the models: Flyway enforces one linear sequence; Alembic embraces the branch and reconciles it explicitly.

Q: What is the safe expand–contract approach to adding a foreign key to a busy table?
A: Add the FK column nullable first, backfill and validate referential values, then add the constraint as `NOT VALID` (Postgres) so it applies without a full-table scan, and finally run `ALTER TABLE ... VALIDATE CONSTRAINT` separately to check existing rows under a weaker lock. Doing a plain `ADD CONSTRAINT ... FOREIGN KEY` in one shot takes a lock and scans the whole table, stalling the loader. Splitting creation from validation is the FK analogue of expand–contract's deferred-constraint step.

Q: What is the purpose and risk of Postgres `ADD CONSTRAINT ... NOT VALID` followed by `VALIDATE CONSTRAINT`?
A: `NOT VALID` adds the constraint so it is enforced on new and modified rows immediately but skips the expensive scan of existing rows, which is fast and low-lock; a later `VALIDATE CONSTRAINT` checks the pre-existing rows under a `SHARE UPDATE EXCLUSIVE` lock that does not block writes. The risk is forgetting the validate step, leaving the constraint trusted-but-unverified so old violating rows persist undetected. Used correctly it is the standard zero-downtime way to add CHECK and FK constraints to large fund tables.

Q: How would you safely change a column's type from `VARCHAR` to `NUMERIC` for a price column under load using expand–contract?
A: Add a new `NUMERIC` column nullable, dual-write or trigger to populate it from the text value with an explicit cast, backfill existing rows in batches validating each cast, switch reads to the numeric column, then drop the old `VARCHAR`. A direct `ALTER COLUMN ... TYPE NUMERIC USING ...` rewrites the whole table under an `ACCESS EXCLUSIVE` lock and fails outright on any unparseable value. The new-column path lets you find and quarantine bad rows without blocking the dealing-window loader.

Q: In a regulated estate, why is the segregation between authoring a migration and applying it enforced, and which Flyway/Alembic features support it?
A: Segregation of duties means the developer who designs a change must not be the one who applies it to production, preventing a single actor from both writing and executing unreviewed DDL — a core control under frameworks like DORA. Alembic supports it via offline `--sql` mode so the developer hands a reviewed SQL artifact to a DBA, and Flyway supports it via dry-run/script output plus restricted execution credentials. The reviewable SQL file is the audited hand-off boundary between the two roles.

Q: How does DORA's emphasis on ICT change management map onto schema-migration practice?
A: DORA requires financial entities to manage ICT changes through controlled, documented, testable processes with rollback provisions, which maps directly onto versioned migrations, a history table as evidence, pre-production testing, and a defined recovery stance. The migration tool's audit trail (`flyway_schema_history` rows with `installed_by`/`installed_on`) is precisely the kind of evidence a DORA-aligned change process must produce. Expand–contract plus roll-forward is how you satisfy the "no untested change, safe recovery" expectation without a maintenance window.

Q: Why does adding a column with a volatile default still force a table rewrite on modern Postgres, and what is the workaround?
A: Modern Postgres optimizes a constant default by storing it as table metadata, avoiding a rewrite, but a volatile default like `now()` or `gen_random_uuid()` must produce a distinct value per row, so the table is rewritten under lock to materialize them. The workaround is the expand–contract way: add the column nullable with no default, backfill the values in batches (computing them as you go), then set the default for future inserts in a separate step. This keeps the lock window tiny on a large holdings table.

Q: What is the locking difference between `CREATE INDEX` and `CREATE INDEX CONCURRENTLY` in Postgres, and what is the cost of the concurrent form?
A: Plain `CREATE INDEX` takes a `SHARE` lock that blocks writes to the table for the whole build, while `CREATE INDEX CONCURRENTLY` lets reads and writes continue by building the index in multiple passes. The cost is that the concurrent form is slower, cannot run inside a transaction block, and can leave an `INVALID` index behind if it fails, which you must then drop and rebuild. On a hot table the availability gain is almost always worth those costs.

Q: After a failed `CREATE INDEX CONCURRENTLY`, what state is the index left in and how do you recover?
A: A failed concurrent build leaves an `INVALID` index that exists in the catalog but is not used by the planner and is not maintained correctly, so you must `DROP INDEX` it (ideally `DROP INDEX CONCURRENTLY`) and retry the build. You can spot it by querying `pg_index` for `indisvalid = false` or via `\d tablename` showing the index marked `INVALID`. Leaving the invalid index in place wastes space and can confuse later migrations, so cleanup is mandatory before re-running.

Q: How would you add a unique constraint to a large table without a long lock in Postgres?
A: Build the supporting unique index first with `CREATE UNIQUE INDEX CONCURRENTLY`, then attach it as a constraint with `ALTER TABLE ... ADD CONSTRAINT ... UNIQUE USING INDEX <indexname>`, which is a fast metadata operation. Doing `ALTER TABLE ... ADD CONSTRAINT ... UNIQUE(...)` directly builds the index under an `ACCESS EXCLUSIVE` lock, blocking the loader. The two-step concurrent-index-then-attach is the standard zero-downtime pattern for unique constraints on fund-scale tables.

Q: Why is enforcing a unique constraint on an existing column a potential expand–contract problem if duplicates exist?
A: If the historical data already contains duplicate values, the constraint creation fails outright, so you cannot simply add it — you must first detect and resolve duplicates as a data-cleanup step before the constraint goes on. The expand–contract-style sequence is: add the index concurrently as non-unique to find conflicts, clean the duplicate rows, then build the unique index and attach the constraint. This is common with identifiers like `ISIN` or `LEI` that were not uniquely constrained originally and accumulated dirty rows.

Q: How do you script just the delta between two specific Alembic revisions for a DBA, and what does the colon syntax mean?
A: Use `alembic upgrade <from>:<to> --sql`, where the colon separates the starting revision from the target so Alembic emits only the DDL for that range; for example `alembic upgrade ae1027:f3a9bd --sql > delta.sql`. Offline mode needs the explicit start because, without a connection, it cannot read `alembic_version`. The resulting file is the reviewable artifact handed to the DBA, who applies it under change control.

Q: What goes wrong if you generate an offline Alembic script from `base` against a database that already has an `alembic_version` table?
A: The script generated from `base` includes the initial `CREATE TABLE alembic_version` and the full migration chain, so applying it to a database that is already partway through fails — the `alembic_version` table (and earlier objects) already exist. You must instead generate from the database's actual current revision, e.g. `alembic upgrade <current>:head --sql`, so the script contains only the pending delta. This "from base on a non-empty DB" mistake is a classic offline-mode pitfall.

Q: Why is `flyway repair` not a substitute for fixing a genuine checksum mismatch caused by an edited applied migration?
A: `flyway repair` simply overwrites the stored checksum to match the current file, so it makes `validate` pass without addressing that the applied migration's content has actually diverged from what other environments ran. Using it that way hides real environment divergence and destroys the integrity guarantee that "the same bytes ran everywhere." The correct fix is to revert the edit and ship a new migration; `repair` is for benign whitespace changes or clearing failed-migration rows, not for legitimizing content edits.

Q: A migration failed on Postgres mid-statement but `flyway info` shows no `success = false` row — why?
A: Because Postgres has transactional DDL, the failing migration's whole transaction rolled back, so the partial changes were undone and no failed row was committed to the history table. This is the advantage Postgres has over engines like MySQL, where DDL is auto-committed and a mid-migration failure can leave both partial schema changes and a `success = false` row to repair. So on Postgres you usually fix the script and re-run, with no `flyway repair` needed.

Q: On MySQL, why is a failed multi-statement migration more dangerous than on Postgres, and what does it mean for cleanup?
A: MySQL auto-commits each DDL statement, so a migration that fails on its third statement leaves the first two already committed and irreversible within that transaction, plus a `success = false` history row. Cleanup therefore requires manually undoing the partial schema changes before `flyway repair` removes the failed row and you retry. Postgres avoids this because its transactional DDL rolls the whole migration back atomically, which is why the lesson favours Postgres for safe migrations.

Q: How do asset checks or post-migration validation gate the contract step, and what specific checks fit a NAV table?
A: You make progression to the irreversible contract step conditional on green automated checks run after the backfill and read-switch — for a NAV/holdings table these include "zero NULLs in the new column," row-count parity between old and new columns, and a NAV reconciliation that the recomputed values match the prior gold figures within tolerance. If any check is red, you halt before dropping the old column, preserving the ability to fall back. This is the data-quality gate that turns expand–contract from hopeful into evidenced.

Q: In the live rename drill, the V3 backfill completes but the post-backfill NULL check still finds NULLs in the new column — what is the likely cause?
A: The most likely cause is that rows were inserted or updated between the start of the backfill and its completion through a path that does not populate the new column — i.e. the dual-write or sync trigger is not covering every writer. You must confirm the trigger fires on all `INSERT`/`UPDATE` paths (including any out-of-band loader), re-run the backfill for the stragglers, and only then switch reads. Switching reads with residual NULLs would surface wrong NAV or holdings data, which is exactly what the gate prevents.

Q: Why might you choose a trigger over application dual-write specifically in a fund estate with multiple ingestion paths?
A: A fund estate often has several writers to the same table — the scheduled loader, a corrections tool, manual DBA fixes, and a legacy ETL — and you cannot reliably update all of them to dual-write in lockstep. A database trigger guarantees the new column stays synchronized regardless of which path performs the write, centralizing the invariant in the schema. The cost is that the trigger must be explicitly removed in the contract step, or it will keep firing against the dropped column and error.

Q: How does the immutability of applied migrations support a regulator's "evidence of what changed" requirement under UCITS/AIFMD oversight?
A: Because applied migrations are checksum-frozen and append-only, the history table is a tamper-evident, timestamped record of exactly which schema changes ran, in what order, by whom — the kind of immutable evidence depositary and regulatory oversight expect for systems computing fund NAV. An auditor can read `flyway_schema_history` directly instead of trusting a wiki. Any attempt to edit history is caught by checksum validation, which is what makes the record trustworthy as control evidence rather than mere documentation.

Q: Summarize the end-to-end checklist for executing the lesson's live column rename with zero failed loads.
A: Baseline the schema under Flyway/Alembic; ship `V2` expand (add nullable new column plus sync trigger or dual-write, additive and backward-compatible); verify the loader keeps succeeding; ship `V3` backfill in batches and switch reads only after a green NULL/row-count/NAV-reconciliation check; ship `V4` contract (drop the old column and remove the trigger) only once the whole fleet reads the new column exclusively; throughout, keep the loader on its schedule, apply DB-before-code, hold a roll-forward stance backed by backups, and capture the history-table rows plus zero-error load logs as the five-line CAB evidence.

Q: What does Flyway's `mixed` mode allow, and why is it off by default?
A: `mixed=true` lets a single Flyway run apply both transactional and non-transactional migrations together, which Flyway otherwise refuses because mixing them in one run weakens the all-or-nothing guarantee. It is off by default so that a failure cannot leave you with some transactional changes rolled back and some non-transactional ones committed in the same migration batch. You enable it deliberately when, for example, one migration needs `CREATE INDEX CONCURRENTLY` (non-transactional) alongside ordinary transactional DDL in the same apply, accepting the weaker atomicity.

Q: How can you tell whether a `flyway migrate` actually applied anything versus found nothing pending, from its output?
A: Successful application prints a summary like `Successfully applied N migrations to schema "..."` listing each version, while a no-op prints `Schema "..." is up to date. No migration necessary.` or `Current version of schema ...: N`. If you added a file but see the up-to-date message, the file is being ignored — check the double-underscore naming and the `locations` setting before assuming Flyway is broken. Running `flyway info` then shows whether your file is `Pending` (recognized) or absent (ignored/mislocated).

Q: When backfilling a new column on a 50M-row holdings table, how do you choose a batch size and detect when the backfill is done?
A: Pick a batch size that keeps each `UPDATE` transaction short enough to avoid blocking the loader — often tens of thousands of rows — and loop on a stable key range like `WHERE id BETWEEN :lo AND :hi AND new_col IS NULL`, sleeping briefly between batches to let the loader interleave. You detect completion when a `SELECT count(*) FROM holdings WHERE new_col IS NULL` (excluding rows the trigger now keeps current) returns zero. Tuning batch size is a trade-off: too large and you lock out the dealing-window loader; too small and the backfill takes too long to finish before the read-switch.
