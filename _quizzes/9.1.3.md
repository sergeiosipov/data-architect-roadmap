# Phase 1 · 9.1.3 Local data sandboxes — 150 self-test questions

Q: What is a "local data sandbox" in the context of this study plan, and why does the whole four-year lab depend on it?
A: It is a reproducible, self-contained data stack you can stand up on your own laptop with one command, typically a `docker-compose.yml` describing services like Postgres, MinIO, and an analytical engine. The lab depends on it because every capstone builds on the assumption that a stranger can clone the repo and get the identical stack running; a stack only you can bring up is a stack nobody can trust or review. The discipline of a one-command, reproducible stack is the same discipline that later makes your Azure environments reviewable.

Q: What does the `docker compose up` command do at a high level?
A: It reads the `docker-compose.yml` (and any override files) in the current directory, creates the declared networks and volumes if missing, pulls or builds the images, and starts every defined service as containers in dependency order. By default it runs in the foreground and streams logs; adding `-d` runs the stack detached in the background. It is the single command that turns a declarative file into a running multi-container stack.

Q: What is a Compose "service" versus a "container"?
A: A service is the declarative definition in `docker-compose.yml` — an image, ports, environment, volumes, and dependencies — describing how one logical component should run. A container is a concrete running instance of that service; one service usually maps to one container but can be scaled to several replicas. The distinction matters because you reason and version the service definition, while Compose manages the container instances for you.

Q: What does `depends_on` express in a Compose file?
A: `depends_on` declares that one service depends on another, which controls the order in which Compose starts and stops containers — the dependency is started before the dependent and stopped after it. In short form it is just a list of service names; in long form each dependency carries a `condition`. It is the primary tool for expressing the startup graph of your stack.

Q: With the short-form `depends_on`, what exactly does Compose guarantee about startup?
A: Short-form `depends_on` only guarantees that the dependency container is created and started before the dependent — it is equivalent to the `service_started` condition. It does NOT wait for the dependency's application to be ready to accept connections; Postgres can be "started" while its server is still initialising. This gap is the single most common source of flaky stacks and is exactly why healthchecks exist.

Q: What does Compose explicitly NOT guarantee, even with `depends_on` set?
A: Compose does not guarantee that the dependency's service inside the container is actually ready — only that the container process has started. It does not wait for a database to finish recovery, a port to start listening, or an HTTP endpoint to return 200. To gate on real readiness you must add a `healthcheck` and use the `service_healthy` condition, and even then your application should still tolerate transient connection failures.

Q: What are the three valid `condition` values in the long-form `depends_on` syntax?
A: They are `service_started`, `service_healthy`, and `service_completed_successfully`. `service_started` waits only for the container to start; `service_healthy` waits until the dependency's healthcheck reports healthy; `service_completed_successfully` waits for the dependency to run to a successful exit (exit code 0). You choose the condition that matches what "ready" actually means for that dependency.

Q: When would you use the `service_completed_successfully` condition?
A: You use it when a dependency is a one-shot job rather than a long-running service — for example a migration or seed container that must finish before the app starts. Compose waits for that container to exit with code 0 before starting the dependent. If the job exits non-zero, the dependent is not started, which is the desired fail-fast behaviour.

Q: Write the long-form `depends_on` block that makes a `loader` service wait for `postgres` to be healthy.
A: You write `depends_on:` then under `loader`, `postgres:` with `condition: service_healthy`. Concretely the loader's block reads `depends_on:` followed by an indented `postgres:` then `condition: service_healthy`. This makes Compose hold the loader until the Postgres healthcheck passes, not merely until its container starts.

Q: What is the practical difference between `service_started` and `service_healthy` for a data loader?
A: `service_started` releases the loader the instant the database container starts, which is often before Postgres is accepting connections, so the loader's first connection attempt fails. `service_healthy` releases the loader only after the database's healthcheck succeeds, meaning the server is actually ready. A loader needs `service_healthy` because it connects and runs SQL immediately, with no built-in patience for a not-yet-ready server.

Q: Why is being able to explain `service_started` versus `service_healthy` a "Done-when" criterion for this lesson?
A: Because it is the conceptual crux of reliable local stacks: knowing that "container started" is not "service ready" is what separates a stack that comes up cleanly every time from one that races and fails intermittently. The lesson's acceptance test asks you to articulate why a loader specifically needs `service_healthy`. If you can explain it, you understand both Compose's ordering guarantees and their limits.

Q: What is a Compose `healthcheck` and what problem does it solve?
A: A `healthcheck` is a command Compose runs periodically inside a container to determine whether the service is actually ready and functioning, marking the container `healthy`, `unhealthy`, or `starting`. It solves the readiness-versus-started gap: without it, `depends_on` can only know the container launched, not that the app works. A healthcheck turns a vague "it's up" into a testable signal other services can gate on.

Q: What are the main keys inside a Compose `healthcheck` block?
A: The keys are `test` (the command to run), `interval` (how often to run it), `timeout` (how long each run may take), `retries` (consecutive failures before `unhealthy`), `start_period` (a grace window during which failures don't count against `retries`), `start_interval` (a faster probe cadence during that grace window), and `disable` (turn off an inherited healthcheck). Together they let you tune both how readiness is measured and how forgiving the startup window is.

Q: What does the `test` key in a healthcheck specify, and what forms can it take?
A: `test` specifies the command Compose runs to judge health; a zero exit means healthy, non-zero means failing. It can be a string (run via the shell, equivalent to `CMD-SHELL`) or a list whose first element is `CMD` (run directly, no shell) or `CMD-SHELL` (run through the shell). Using the list `CMD` form avoids shell-quoting surprises, while `CMD-SHELL` is convenient when you need shell features like pipes or variable expansion.

Q: What healthcheck `test` command is conventional for a Postgres service?
A: The conventional probe is `pg_isready`, typically `pg_isready -U <user> -d <db>`, which returns exit code 0 only when the server is accepting connections. It ships with the Postgres image, so no extra tooling is needed. Some teams instead run a trivial `psql -c 'SELECT 1'` for a stronger check that the database actually answers queries, not just that the listener is up.

Q: What does the `start_period` key do, and why does it matter for databases?
A: `start_period` defines an initial grace window after the container starts during which failing healthchecks do not count toward the `retries` limit and do not mark the container `unhealthy`. It matters for databases because first-boot initialisation (creating data directories, running init scripts) can legitimately take many seconds, and you don't want those expected early failures to flip the container to `unhealthy`. A successful probe during the grace period immediately marks the container healthy.

Q: What is the role of `retries` in a healthcheck?
A: `retries` is the number of consecutive failed probes required before Compose marks the container `unhealthy` (after the `start_period`). It debounces transient failures so a single hiccup doesn't fail the service, while still catching a genuinely broken one. For example `retries: 5` with `interval: 5s` means roughly 25 seconds of sustained failure before the container is declared unhealthy.

Q: A dependent service never starts and Compose reports the dependency as `unhealthy` — what is the first thing to check?
A: First run the dependency's healthcheck `test` command yourself inside the container, e.g. `docker compose exec postgres pg_isready -U postgres`, to see what it actually returns. A common cause is a wrong username or database name in the probe so it always exits non-zero, or a `start_period` too short for first-boot init. Inspect `docker inspect --format '{{json .State.Health}}' <container>` to read the recorded failure output.

Q: How do you see a container's current health status from the CLI?
A: Run `docker ps`, which shows the health state in parentheses in the `STATUS` column, such as `Up 30 seconds (healthy)` or `(health: starting)`. For the full probe history and the last few outputs, use `docker inspect` and read the `.State.Health` object. `docker compose ps` similarly surfaces health for the stack's services.

Q: Why should application code still retry connections even when you use `service_healthy`?
A: Because a healthcheck only samples readiness at intervals, and there is always a small window — and edge cases like a database briefly rejecting connections under load or restart — where the dependency can be momentarily unavailable. Healthchecks reduce races but do not eliminate them, so robust apps wrap their initial connection in a short retry-with-backoff loop. Compose's own docs recommend designing services to be resilient to dependency unavailability rather than relying solely on startup order.

Q: What is a Docker named volume and why prefer it for stateful services?
A: A named volume is Docker-managed persistent storage with a stable name (declared under top-level `volumes:` and mounted into a container), whose lifecycle is independent of any single container. You prefer it for stateful services like Postgres or MinIO because data survives `docker compose down` and container recreation, and Docker manages its location and permissions. It is the canonical way to keep a database's data directory durable across stack restarts.

Q: What is the difference between a named volume, an anonymous volume, and a bind mount?
A: A named volume has an explicit name and is managed by Docker in its storage area; an anonymous volume is the same but with a random name Docker assigns when you don't name it, making it easy to orphan; a bind mount maps a specific host path directly into the container. Named volumes are best for service state (portable, managed), bind mounts are best for injecting source code or config from the host. Relying on anonymous volumes for state is a smell because they are hard to track and easy to lose.

Q: Why is putting every stateful service on a named volume a "Done-when" requirement for this lesson?
A: Because reproducibility and durability both depend on it: state on a named volume survives container recreation, and the explicit declaration makes it obvious where each service's data lives. The acceptance criterion asks you to show every stateful service mapped to a named volume so there is no hidden, ephemeral, or host-coupled state. A stack with un-named or accidental state is one you cannot reliably tear down and rebuild.

Q: What happens to a named volume when you run `docker compose down` versus `docker compose down -v`?
A: Plain `docker compose down` stops and removes the containers and the default network but preserves named volumes, so your database data is still there next time. Adding `-v` (or `--volumes`) also removes the named volumes declared in the Compose file, deleting that state. The `-v` flag is what you use deliberately to get a truly clean slate for reproducibility testing.

Q: How do you mount a named volume into the Postgres data directory in Compose?
A: You declare a top-level `volumes:` entry such as `pgdata:` and, under the Postgres service, add a `volumes:` mapping `pgdata:/var/lib/postgresql/data`. The left side is the named volume, the right side is the container path Postgres uses for its data. This keeps the database files durable and lets you wipe them on demand with `docker compose down -v`.

Q: What is a Compose network and what does Compose create by default?
A: A Compose network is a Docker virtual network that connects the stack's containers so they can reach each other by service name. By default, Compose creates a single network for the project and attaches every service to it, so `postgres` can be reached at the hostname `postgres` on its container port. This default makes intra-stack DNS "just work" without you wiring anything up.

Q: How do containers in the same Compose stack address each other?
A: They use the service name as a DNS hostname on the shared network — for example an app connects to Postgres at host `postgres` and port `5432`, not `localhost`. Compose runs an embedded DNS resolver that maps service names to container IPs. This is why your application's connection string inside the stack uses `postgres:5432`, while from your host you'd use `localhost` and a published port.

Q: Why might you define multiple, isolated networks per stack rather than using the single default?
A: To segment traffic and enforce isolation — for instance keeping a database on an internal-only network unreachable from outside, while the app sits on both that network and a front-facing one. Isolated networks per stack also prevent accidental cross-talk between unrelated projects running on the same host. The lesson frames "isolated networks per stack" as part of treating each sandbox as a self-contained unit.

Q: What does publishing a port (`ports: "5432:5432"`) actually do, and how does it differ from `expose`?
A: `ports` maps a container port to a host port so the service is reachable from your laptop (host `localhost:5432`), punching through to the outside of the stack. `expose` only documents/opens a port to other containers on the same network without binding it on the host. For a database you often only need inter-container access, so exposing without publishing reduces your host's attack surface.

Q: What is image pinning and why does the lesson insist on it?
A: Image pinning means referencing an image by an exact, immutable identifier — a specific version tag like `postgres:16.3` or, better, a content digest — instead of a moving target like `latest`. The lesson insists on it because reproducibility across machines and across months requires that everyone pulls the identical bits; `latest` silently drifts and breaks the "same numbers on a clean clone" guarantee. Pinning is what makes the stack a fixed, auditable artifact.

Q: Why is `postgres:latest` dangerous in a Compose file meant to be reproducible?
A: `latest` is a mutable tag that points to whatever the maintainers most recently pushed, so two people pulling on different days can get different major versions with incompatible behaviour or on-disk formats. Your stack then "works on my machine" and fails on a clean clone, defeating the whole point of a reproducible sandbox. Pin to an explicit version (and ideally a digest) so the stack reproduces identically.

Q: What is the difference between pinning by tag and pinning by digest?
A: A tag like `postgres:16.3` is human-readable but still mutable — the maintainer can re-push a new image under the same tag. A digest like `postgres@sha256:<hash>` is content-addressed and absolutely immutable: it always refers to byte-for-byte the same image. Tags are convenient and usually sufficient; digests give the strongest reproducibility guarantee and are preferred for audited or regulated environments.

Q: How do you find the digest of an image you have pulled?
A: Run `docker image inspect <image> --format '{{index .RepoDigests 0}}'`, which prints the `repository@sha256:...` digest reference. You can then copy that into your Compose file in place of the tag. `docker images --digests` also lists digests alongside tags for a quick overview.

Q: Why is "show every image pinned to a specific version tag" an explicit acceptance criterion?
A: Because unpinned images are the quietest way a reproducible stack rots: nothing visibly changes in your file, yet the behaviour drifts as upstream tags move. Requiring a visible, specific tag on every image makes the reproducibility contract auditable at a glance during review. It mirrors the same governance you'll later apply to pinned dependencies and infrastructure versions.

Q: How does image pinning relate to the regulated fund-administration context this plan targets?
A: In a regulated context like Luxembourg fund administration under DORA and UCITS/AIFMD oversight, you must be able to demonstrate exactly which software versions produced a given NAV or report, often years later for audit. Pinned images (ideally by digest) make the computing environment itself reproducible and auditable, so a recomputed NAV can be tied to an unambiguous toolchain. Unpinned `latest` images would make that provenance claim impossible to defend.

Q: What is Testcontainers and what core idea does it embody?
A: Testcontainers is a set of libraries (Java, Python, Go, and others) that let your test code programmatically start, configure, and tear down real Docker containers — a real Postgres, Kafka, or MinIO — for the duration of a test. The core idea is programmatic, throwaway infrastructure: instead of mocking a database or relying on a shared one, each test run spins up genuine ephemeral services and disposes of them automatically. It moves the reproducible-stack discipline from `docker-compose.yml` into the test lifecycle itself.

Q: How does Testcontainers differ from a `docker-compose.yml` development stack?
A: A Compose stack is a declarative, long-lived environment you bring up by hand for development; Testcontainers is imperative and ephemeral, created and destroyed by your test code within a single test run. Compose is for "give me a dev environment to work against"; Testcontainers is for "give this one test a clean, real dependency and throw it away after." They are complementary: Compose for the dev sandbox, Testcontainers for hermetic integration tests.

Q: Why does Testcontainers usually map service ports to random host ports?
A: Because tests must be able to run in parallel and on CI without colliding on fixed ports, Testcontainers binds each container to an ephemeral random host port and exposes a method to look it up at runtime. Your test asks the container object for its mapped port and host rather than hard-coding `5432`. This makes tests robust against other processes and concurrent test runs holding well-known ports.

Q: What guarantees does Testcontainers give about cleanup of containers it starts?
A: It tears down containers at the end of the test/lifecycle, and it ships a companion reaper (commonly called Ryuk) that watches the test session and removes leftover containers, networks, and volumes even if the test process crashes or is killed. This prevents orphaned infrastructure from accumulating across runs. The promise is that what a test starts, the framework reliably cleans up.

Q: In a regulated fund pipeline, what is a concrete use of Testcontainers?
A: You can write an integration test that spins up a throwaway Postgres, applies your migrations, loads a small set of ISIN-keyed securities and a NAV row, runs the loader, and asserts the resulting row counts and checksums — then disposes of the container. Because the database is real and ephemeral, the test exercises actual SQL, constraints, and types rather than mocks, and leaves no shared state behind. That gives high confidence the loader behaves correctly before it ever touches a shared environment.

Q: What is DuckDB and how is it architecturally different from Postgres?
A: DuckDB is an embedded, in-process analytical (OLAP) database that runs as a library inside your application or query tool, storing data in a single file (or in memory). Postgres, by contrast, is a separate server process you connect to over a socket and is optimised for transactional (OLTP) workloads. The key difference is that DuckDB needs no service to manage — you import a library and open a file — whereas Postgres needs a running daemon.

Q: What does "DuckDB as a zero-infra warehouse" mean, and when does a file beat a service?
A: It means using DuckDB's single file plus its in-process engine to do analytical querying without standing up any server, network, or container — no infrastructure to operate. A file beats a service when the data fits comfortably on one machine, the workload is single-user analytics or batch transforms, and you value zero operational overhead and trivial reproducibility over multi-user concurrency. For a local lab building a reporting mart, a `.duckdb` file is often simpler and faster than running a warehouse service.

Q: Why might you include DuckDB in a Compose stack at all if it needs no server?
A: Usually you don't run it as a long-running Compose service; it more often appears as a library used by a Python loader or a dbt run inside a one-shot container. When it does appear in the stack, it is as the analytical engine a transform step uses against a mounted volume, not as a daemon with a port. The point of the lesson is recognising when the right answer is a file (DuckDB) rather than another service.

Q: What is DuckDB's concurrency model for writers?
A: DuckDB is single-writer at the file level: only one process may open the database in read-write mode at a time, using MVCC and optimistic concurrency control within that process. Multiple threads inside that single process can write, but two threads editing the same row trigger a conflict error on the second. For multiple processes to read simultaneously, they must all open the file read-only, during which no process may write.

Q: A second tool fails to open a `.duckdb` file with a lock/conflict error while your loader is running — why?
A: Because DuckDB allows only one read-write process at a time on a database file; your loader holds the read-write lock, so a second writer is refused. The fix is either to let the loader finish, or to have the second process open the file in read-only mode (which is allowed only if no writer holds it). This single-writer constraint is the trade-off for DuckDB's zero-infra simplicity.

Q: Why is DuckDB a natural fit for the Phase-1 capstone's reporting mart?
A: The capstone builds a dimensional mart (a NAV periodic-snapshot star and an SCD2 fund dimension) as dbt models, which is a single-user, batch analytical workload that fits on one machine. DuckDB gives columnar OLAP performance with literally zero infrastructure to stand up, so a stranger can reproduce the mart by running dbt against a file. It stands in for a cloud warehouse like Synapse or Snowflake in the local lab without the operational weight.

Q: What is the recommended way to test that your stack is truly reproducible?
A: Test from a fresh clone on a clean Docker: prune existing images and volumes, clone the repo, and run `docker compose up`, timing it until every service reports healthy. This removes any reliance on cached images or leftover state that only exists on your machine. The lesson's bar is a clean, healthy stack in under two minutes from that fresh state.

Q: What does `docker compose up` doing in the foreground versus `-d` change about your workflow?
A: In the foreground, Compose attaches to and streams all container logs and stays in control of your terminal until you `Ctrl-C`, which is ideal for watching a stack come up and diagnosing startup. With `-d` (detached), it starts the stack in the background and returns your prompt, which you'd use once the stack is known-good or in scripts. For first reproducibility testing, running in the foreground lets you watch healthchecks flip to healthy.

Q: What is the difference between `docker compose down` and `docker compose stop`?
A: `docker compose stop` halts the running containers but leaves them, the network, and volumes in place so you can `start` them again quickly. `docker compose down` removes the containers and the default network entirely (volumes are kept unless you add `-v`), returning closer to a clean slate. You use `stop` to pause work and `down` to tear the stack down.

Q: What does `docker compose ps` show you?
A: It lists the containers belonging to the current Compose project with their state, published ports, and — when healthchecks are defined — their health status. It is the quickest way to confirm which services are up, healthy, or unhealthy for this stack specifically. Unlike `docker ps`, it is scoped to the project defined by your Compose file.

Q: How do you view the logs of just one service in a Compose stack?
A: Run `docker compose logs <service>`, optionally with `-f` to follow in real time and `--tail N` to limit history. For example `docker compose logs -f postgres` streams the database's startup output, which is where you'd look to see why a healthcheck is failing. Scoping to one service keeps the output readable in a multi-service stack.

Q: What is the purpose of the Compose `project name`, and how is it set?
A: The project name namespaces all of a stack's resources — containers, the default network, and volume prefixes — so multiple stacks can coexist on one host without colliding. By default it is derived from the directory name; you can override it with `-p`/`--project-name` or the `COMPOSE_PROJECT_NAME` environment variable, or `name:` at the top of the file. Distinct project names are what let you run two copies of a similar stack side by side.

Q: Why does running the same `docker-compose.yml` from two differently named directories create two independent stacks?
A: Because Compose derives the project name from the directory by default, and the project name prefixes the volume, network, and container names. Two directories yield two project names, hence two isolated sets of resources that don't share state. To force them to be the same stack you would set an explicit shared project name.

Q: What does the `restart` policy on a service control, and how does it differ from `depends_on: restart`?
A: A service-level `restart` policy (`no`, `always`, `on-failure`, `unless-stopped`) tells the container runtime whether to relaunch a container after it exits or the daemon restarts. The `restart: true` under a `depends_on` entry is different: it tells Compose to restart the dependent service after it updates the dependency during a Compose operation, excluding runtime auto-restarts. One governs crash recovery; the other governs reaction to dependency updates.

Q: What is the difference between the Compose `environment` key and an `.env` file?
A: The `environment` key sets variables inside a service's container at runtime. The `.env` file in the project directory provides values for variable interpolation in the Compose file itself (e.g. `image: postgres:${PG_VERSION}`) and is read automatically by Compose. They operate at different layers: `.env` parameterises the file; `environment` configures the container.

Q: Why should secrets like `POSTGRES_PASSWORD` not be hard-coded directly in a committed `docker-compose.yml`?
A: Because the file is version-controlled and shared, a hard-coded credential leaks into history and to anyone with repo access, which is unacceptable in a regulated environment. Instead reference an environment variable or use Docker/Compose secrets so the value comes from an uncommitted `.env` or a secret store. For a throwaway local sandbox a dummy password is fine, but the habit of externalising secrets should be built early.

Q: What is the role of `docker compose config`?
A: `docker compose config` renders and validates the fully-merged, interpolated configuration — applying overrides, `.env` substitution, and defaults — and prints the canonical result. It is the fastest way to catch a YAML or interpolation error and to see exactly what Compose will run, including which image tags resolved. Running it before `up` turns silent config drift into a visible diff.

Q: How do Compose override files (e.g. `docker-compose.override.yml`) work?
A: Compose automatically merges `docker-compose.override.yml` on top of the base `docker-compose.yml`, letting you layer local or environment-specific changes (extra ports, mounts, debug settings) without editing the base. You can also pass multiple files explicitly with `-f base.yml -f prod.yml`, applied left to right. This keeps a clean, shared base while allowing per-context customisation.

Q: A service comes up but your host cannot connect to it on the expected port — what are the first two things to check?
A: First confirm the `ports` mapping actually publishes the container port to the host (e.g. `"5432:5432"`), since without it the service is only reachable from inside the stack. Second confirm the service is bound inside the container to `0.0.0.0` rather than `127.0.0.1`, and that you are connecting to `localhost` on the host-side port. `docker compose ps` and `docker compose logs` quickly reveal whether the container is even listening.

Q: Inside the stack, an app's connection string uses `localhost:5432` and fails — why, and what is the fix?
A: Inside a container, `localhost` refers to that container itself, not the database container, so the app is dialling a port nothing is listening on. The fix is to use the database's service name as the host, e.g. `postgres:5432`, which Compose's DNS resolves to the right container. `localhost` only works from your laptop against a published port, not for inter-container traffic.

Q: What does it mean that Compose creates and removes resources idempotently on repeated `up`?
A: On `docker compose up`, Compose reconciles desired state with what exists: it reuses already-correct containers, networks, and volumes, and only (re)creates what changed. Running `up` again on an unchanged file is largely a no-op rather than a duplication. This reconciliation is part of why a Compose file behaves as a declarative description of the stack rather than a one-shot script.

Q: Why does the lesson tell you to `docker system prune` before timing a fresh `docker compose up`?
A: Pruning removes cached images, stopped containers, dangling volumes, and networks so the timed run starts from a genuinely clean state, like a stranger's machine. Without pruning, your local image cache hides slow or missing pulls and masks reproducibility problems. The honest test of "under two minutes from a fresh clone" requires that nothing is pre-warmed.

Q: What is the risk of `docker system prune -a --volumes` and when should you use it?
A: `prune -a --volumes` removes all unused images (not just dangling ones) and all unused volumes, which can delete data and large images you'd have to re-pull. You use it deliberately when you want a maximally clean slate to validate reproducibility, accepting the time cost of re-pulling. Run it knowing it can wipe state and is not reversible, so never on a machine holding data you need.

Q: How do healthchecks make the under-two-minutes acceptance test meaningful?
A: Because the criterion is "healthy," not merely "started," the timer only stops when every service's healthcheck passes — a real readiness signal. Without healthchecks you could only measure container launch, which says nothing about whether the stack is usable. Healthchecks turn the timing test into a measure of true end-to-end readiness.

Q: What is the difference between a Compose healthcheck and a Kubernetes liveness/readiness probe at the concept level?
A: Both periodically run a command/request to judge whether a container is ready or alive, but Compose has a single `healthcheck` whose state feeds `depends_on` ordering on one host. Kubernetes splits the idea into separate readiness probes (gate traffic) and liveness probes (trigger restarts) across a cluster. The underlying concept — probe for actual readiness rather than assuming "started means ready" — is identical, which is why mastering Compose healthchecks transfers upward.

Q: Why is YAML indentation a common cause of Compose failures, and how do you guard against it?
A: YAML uses indentation to define structure, so a mis-indented key silently lands under the wrong parent (e.g. a `healthcheck` becoming a sibling of `services` instead of a service), changing meaning without an obvious error. Guard against it by running `docker compose config` to render the merged result and confirm keys sit where you intend. Consistent two-space indentation and a linting editor catch most cases before runtime.

Q: What does the top-level `volumes:` declaration do versus a per-service `volumes:` mapping?
A: The top-level `volumes:` block declares named volumes that Compose should create and manage for the project. The per-service `volumes:` list mounts those named volumes (or bind paths) into specific containers at specific paths. You need both: one declares the volume exists, the other attaches it where the service writes data.

Q: Why is it a problem to store database data on a bind mount to a host path in a sandbox meant to be portable?
A: A bind mount couples the stack to a specific host filesystem layout and permissions, so it behaves differently across machines and OSes and breaks the "clone and run anywhere" promise. Named volumes are managed by Docker and portable, so the same Compose file works on another laptop without path assumptions. Bind mounts are great for source code, but state should live on named volumes.

Q: How does running on native Ubuntu rather than Docker Desktop on macOS/Windows affect volume performance?
A: On native Linux, Docker uses the host kernel directly, so bind-mount and volume I/O is native-speed; on macOS/Windows, Docker Desktop runs a Linux VM and bind mounts cross a virtualised filesystem boundary that can be markedly slower. This is one reason the lab assumes native Ubuntu — the local stack is faster and behaves closer to production Linux. Named volumes are less affected than bind mounts, but the native-Linux path is simplest and fastest.

Q: On the native-Ubuntu laptop, why does `docker compose` (the plugin) replace the old `docker-compose` (the script)?
A: Compose v2 is implemented as a Docker CLI plugin invoked as `docker compose` (a subcommand), whereas the legacy `docker-compose` was a separate standalone Python binary that is now end-of-life. The v2 plugin is the current, maintained implementation and is what you install on Ubuntu via the `docker-compose-plugin` package. Scripts and habits should use the space-separated `docker compose` form.

Q: What does the Compose specification's top-level `name:` key set, and why might you commit it?
A: The top-level `name:` sets the project name explicitly in the file itself, overriding the directory-derived default. Committing it makes the project name deterministic regardless of what directory someone clones into, which keeps volume and network names stable across machines. That stability aids reproducibility and avoids accidentally creating duplicate stacks.

Q: Why is pinning the Compose file format / spec version less of a concern than it used to be?
A: Older Compose required a top-level `version:` key (e.g. `'3.8'`) that gated which features were available, and mismatches caused confusion. The current Compose Specification is versionless — the `version` field is deprecated and ignored — so you simply use the features the installed Compose supports. Reproducibility now hinges on pinning your image tags and your Compose/Docker versions, not a schema version string.

Q: How do you confirm which Compose version a fresh machine has, and why does it matter for reproducibility?
A: Run `docker compose version` to print the installed Compose plugin version. It matters because newer healthcheck keys like `start_interval` or certain `depends_on` conditions require a recent enough Compose, so a stack that relies on them can silently behave differently on an older install. Documenting a minimum Compose/Docker version in the README is part of making the stack reproducible.

Q: What is MinIO and why is it in the lab's standard stack?
A: MinIO is an S3-API-compatible object storage server you can run locally in a container, giving you a stand-in for cloud object storage like Azure Blob or AWS S3. It is in the stack so the lab can practise object-store patterns — landing raw files, partitioned data lakes — using the same S3 API you'd use in the cloud, but free and local. It makes the local sandbox a faithful, zero-cost rehearsal for cloud storage.

Q: What does a sensible MinIO healthcheck probe, and why not just rely on the container starting?
A: A MinIO healthcheck typically curls its readiness/health endpoint (for example `/minio/health/live`) and treats a non-200 as failing, so dependents only start once the object store actually answers. Relying on container start would let a loader try to create buckets before MinIO is listening, causing connection-refused errors. The healthcheck closes that race exactly as it does for Postgres.

Q: In the Phase-1 capstone stack, why is a "placeholder service" included in the Compose file from the start?
A: The capstone tells you to write the `docker-compose.yml` you'll grow for all four years, so a placeholder reserves the shape — a third service slot, its network attachment, and its dependency wiring — that later phases will fill (a Spark worker, an Airflow scheduler, etc.). Establishing the pattern now means future additions slot into an already-disciplined file. It also forces you to practise dependency and healthcheck wiring beyond just one database.

Q: How would you wire a loader so it runs once after both Postgres and MinIO are ready, then exits?
A: Define the loader as a service with `depends_on` listing `postgres` and `minio` each with `condition: service_healthy`, give it a `restart: "no"` policy, and have its command run the load and exit. Compose holds the loader until both dependencies' healthchecks pass, then runs it to completion. Downstream services can gate on the loader with `condition: service_completed_successfully` if they must wait for the data to be loaded.

Q: Why does the capstone require the loader to produce identical end state when run twice (idempotency), and how does the sandbox support testing that?
A: Regulated fund pipelines must be safely re-runnable — a retried load after a failure must not double-count orders or NAV rows — so the loader uses a natural dedup key (order refs) and converges to the same state. The sandbox supports testing this by letting you wipe to a clean slate (`down -v`), run the loader twice, and compare committed row counts and checksums. The reproducible stack is what makes that two-run evidence trustworthy.

Q: What is a checksum-based acceptance test, and why is the reproducible stack a prerequisite for it?
A: It is asserting that the data produced (e.g. table row counts plus a hash of key columns) matches a committed expected value, proving the load is deterministic. The reproducible stack is a prerequisite because the same numbers can only be guaranteed if everyone runs the identical images, schema, and seed against the identical engine. Without pinned images and managed state, the checksum could differ for environmental reasons rather than real data changes.

Q: How does the discipline of a reproducible local sandbox map to building "reviewable" Azure environments later?
A: The same instinct — declare every dependency, pin every version, externalise state, and make the whole thing reproducible from a clean start — is exactly what makes a cloud environment auditable via infrastructure-as-code. Compose teaches the muscle memory locally for free before you apply it to Bicep/Terraform and Azure resources. A reviewer trusts an environment they can rebuild identically, whether it's a laptop stack or a subscription.

Q: Why does the lesson describe Compose fluency as "the substrate for every capstone"?
A: Because every later capstone assumes a one-command, reproducible stack underneath it, so if you can't author and bring up a clean Compose stack, you can't reliably run or hand off any of the projects. Compose is the common floor on which Postgres, MinIO, DuckDB, and future services stand. Mastering it once pays off in every subsequent phase.

Q: What is the difference between `service_healthy` gating and simply adding a `sleep` before the dependent starts?
A: `service_healthy` waits exactly as long as needed for the dependency to actually report ready, no more and no less, and fails fast if it never does. A fixed `sleep` is a guess: too short and you still race, too long and you waste time on every startup, and it never adapts to a slow machine. Healthcheck gating is the correct, adaptive mechanism; sleeps are a fragile anti-pattern.

Q: A loader gated on `service_healthy` still occasionally fails its first query — what subtle cause should you consider?
A: Consider that the Postgres healthcheck may be passing on the default `postgres` database while your application database or required tables aren't ready yet, or init scripts are still running. `pg_isready` returns healthy as soon as the server accepts connections, which can precede your schema being fully created. Strengthen the healthcheck to probe the actual target database/object, or have the loader retry briefly.

Q: How can you make a Postgres healthcheck check the specific database the loader will use?
A: Point `pg_isready` at the target with `-d <yourdb>` and the right `-U <user>`, so it only reports healthy when that database accepts connections, e.g. `pg_isready -U appuser -d fundref`. For an even stronger guarantee, use `psql -U appuser -d fundref -c 'SELECT 1'` so the probe confirms the server can actually execute a query against that database. This narrows the readiness signal to what the loader truly needs.

Q: What does `docker compose pull` do and why might you run it separately?
A: `docker compose pull` fetches the images referenced by the stack without starting anything, so you can pre-download (or refresh) images in a controlled step. Running it separately lets you measure or warm pull time, verify that pinned tags/digests resolve, and detect a missing image before `up`. It also lets CI cache images between the pull and the run.

Q: What is the difference between `docker compose run` and `docker compose up` for a one-off task?
A: `docker compose up` starts the whole stack (or named services) as long-running containers per the file; `docker compose run <service>` starts a single service with a one-off command, attached, and respects its `depends_on` but is meant for ad-hoc execution like a migration or a shell. `run` is handy for invoking a tool against the stack without redefining it as a permanent service. By default `run` also doesn't publish the service's ports unless you pass `--service-ports`.

Q: Why might you set `pull_policy` or use digests instead of relying on whatever is cached locally?
A: Because a locally cached image under a tag can be stale relative to the registry, so two machines with the same tag but different cache states diverge. Pinning by digest forces byte-identical content, and an explicit `pull_policy: always` (or pulling fresh) avoids silently running an old cached layer. For strict reproducibility, digests remove cache ambiguity entirely.

Q: How does declaring resources (volumes/networks) explicitly in the file aid teardown and cleanup?
A: Because the resources are named and owned by the project, `docker compose down` knows exactly which network to remove, and `down -v` knows precisely which named volumes to delete, leaving no orphans. Implicit or anonymous resources are harder to track and tend to accumulate as cruft. Explicit declarations make both setup and teardown deterministic.

Q: What is a sign that your "reproducible" stack actually has hidden machine-specific state?
A: It works on your machine but a teammate or a clean clone gets different data, errors, or counts — classic symptoms of leftover named volumes, a bind mount to a local path, or a cached image. The test is to `down -v`, prune, and run from a fresh clone; if the result changes, hidden state existed. Eliminating that gap is the entire point of the volumes/pinning discipline.

Q: Why might you keep your database on an internal-only network with no published port in a more security-conscious stack?
A: To ensure the database is reachable only by other services in the stack and never directly from the host or outside, shrinking the attack surface — useful when the stack handles sensitive fund data. The app service bridges the internal database network and any external-facing network. In regulated contexts, minimising exposed ports is a defensible default even in a sandbox.

Q: How does the LEI/ISIN-keyed reference data in the capstone benefit from a reproducible loader and stack?
A: Reference data keyed by ISIN (securities) and tied to LEIs (legal entities) must be loaded consistently so that downstream NAV and reporting joins are stable; a reproducible loader guarantees the same keys and relationships every run. Because the stack is pinned and state-managed, anyone can reproduce the exact reference set and verify joins resolve. This consistency is what lets the capstone's point-in-time and SCD2 queries be trusted.

Q: Why is "a stranger can run `docker compose up && uv run load.py` from the README and get the same numbers" the ultimate acceptance test?
A: Because it operationalises reproducibility end to end: the stack comes up identically (pinned images, named volumes, healthchecked ordering) and the loader produces deterministic data, verified by matching numbers. It removes every "works on my machine" excuse and proves the sandbox is a shared, trustworthy artifact. This is the same standard you will be held to when handing off real fund-administration pipelines.

Q: How does `uv run load.py` fit into the sandbox, and why `uv` rather than bare `python`?
A: `uv run load.py` executes the loader inside a `uv`-managed, reproducible Python environment resolved from the project's lockfile, so dependencies are pinned just like the Docker images. Using `uv` rather than bare `python`/`pip` guarantees the same package versions on every machine, extending reproducibility from the container layer into the Python layer. It mirrors image pinning: deterministic dependencies are the rule, not whatever happens to be installed.

Q: Why is it better for the loader to connect to Postgres via the published host port when run from the host, but via the service name when run as a container?
A: From the host, the loader reaches Postgres through the port published on `localhost`, because the host is outside the Compose network. When the loader itself runs as a container on the stack's network, it must use the service name `postgres` because that is how Compose DNS resolves the database, and `localhost` would mean the loader's own container. Choosing the right host depends on where the loader executes relative to the network boundary.

Q: What does it mean that Compose `up` will recreate a container when its configuration changes, and why is that desirable?
A: When you edit a service's image tag, environment, or mounts, the next `up` detects the diff and replaces the old container with one reflecting the new config, rather than leaving stale settings running. This is desirable because it keeps the running stack faithful to the declared file — the file remains the source of truth. It is the reconciliation behaviour that makes Compose declarative rather than a fire-and-forget script.

Q: A teammate reports your stack "takes forever to come up" while yours is instant — what reproducibility-relevant explanation is most likely?
A: Most likely your images are already cached locally while theirs must be pulled fresh, so your timing hides the real cold-start cost; the pinned tags are correct but the pull dominates on a clean machine. Confirm by pruning and timing from cold yourself. The lesson's prune-then-time discipline exists precisely to surface this difference.

Q: Why should a healthcheck command be cheap and side-effect-free?
A: Because it runs repeatedly on the configured `interval` for the life of the container, an expensive or stateful probe adds load and can itself cause flakiness or false unhealthies under contention. A good probe is a quick liveness check like `pg_isready` or a lightweight `SELECT 1`, not a heavy query or a write. Cheap, idempotent probes give a reliable signal without perturbing the service.

Q: How do you decide between adding a service to the Compose stack versus using DuckDB-as-a-file for an analytical need?
A: Ask whether the workload needs multi-user concurrency, network access, or to run continuously — if yes, a service (e.g. Postgres or a warehouse) is justified; if it is single-user batch analytics that fits on one machine, a DuckDB file avoids the operational cost entirely. The lesson frames this as "when a file beats a service": prefer the file when it removes infrastructure without losing what you need. Every avoided service is one less thing to pin, healthcheck, and operate.

Q: What is the relationship between this lesson's Compose skills and Phase 0's Docker module?
A: This lesson (9.1.3) builds directly on the container fundamentals from Phase 0's module 0.5, moving from single-container basics to orchestrating a multi-service, healthchecked, volume-backed stack. Phase 0 taught what a container and image are; here you compose them into a reproducible system. The progression is deliberate: container literacy first, then Compose fluency as the substrate for every capstone.

Q: Summarise the minimal checklist that makes a Compose-based local sandbox trustworthy and reproducible.
A: Every image is pinned to a specific tag (ideally a digest); every stateful service writes to a named volume; every dependent gates on `service_healthy` via a real healthcheck, not just `service_started`; and the whole stack comes up clean and healthy from a fresh clone on pruned Docker, with the Python layer pinned via `uv`. Validate it by running from cold and confirming the loader yields identical row counts and checksums on a second run. If a stranger gets the same numbers from the README, the sandbox is trustworthy.

Q: What exit code does `pg_isready` return when the server is still starting up and rejecting connections, and why does that matter for a healthcheck?
A: `pg_isready` returns `1` when the server is reachable but rejecting connections (typically during startup recovery), `0` when accepting them normally, `2` when there is no response at all, and `3` when no connection attempt was made due to bad parameters. It matters because only exit `0` should mark the container healthy, and the `start_period` window is exactly what tolerates those early `1`/`2` results during first boot. A probe that misreads a `1` as ready would release a loader against a server that still refuses queries.

Q: If a healthcheck block omits `interval`, `timeout`, and `retries`, what default values does Compose apply?
A: Compose applies `interval: 30s`, `timeout: 30s`, `retries: 3`, and `start_period: 0s` by default, matching Docker's built-in healthcheck defaults. This means an undertuned database healthcheck can take up to roughly 90 seconds of failures (three 30-second intervals) before going `unhealthy`, and with no `start_period` even expected first-boot failures count immediately. For a fast local stack you usually shorten `interval` to a few seconds and add a `start_period` so the stack reaches healthy quickly without false negatives.

Q: How does the special `CMD-SHELL` form in a healthcheck `test` differ from giving the test as a plain YAML string?
A: A plain string like `test: pg_isready -U postgres` is implicitly run through the shell, which is exactly what `["CMD-SHELL", "pg_isready -U postgres"]` does explicitly, so the two are equivalent. The contrast is with `["CMD", "pg_isready", "-U", "postgres"]`, which execs the binary directly with no shell, avoiding word-splitting and variable-expansion surprises. Use `CMD-SHELL` when you genuinely need pipes, `&&`, or `$VAR` expansion; otherwise the `CMD` list form is safer.

Q: Why can a healthcheck `test` of `["CMD", "curl", ...]` fail even though the service is healthy, and how do you avoid it?
A: It fails when the image is minimal and does not actually contain `curl` (or `wget`), so the probe errors with command-not-found and is scored as unhealthy regardless of the service. You avoid it by probing with a tool the image is guaranteed to ship — `pg_isready` for Postgres, `mc ready` or a built-in for MinIO — or by using the engine's own health endpoint via a tool you have verified is present. Always run the exact probe with `docker compose exec` once to confirm the binary exists before trusting it.

Q: You change `image: postgres:16.3` to `postgres:16.4` and run `docker compose up` — what does Compose do to the running database container and its volume?
A: Compose detects the changed image and recreates the Postgres container from the new image, but the named volume holding `/var/lib/postgresql/data` is reattached, so the data directory persists across the recreation. Within a minor bump like `16.3` to `16.4` the on-disk format is compatible, so the new server starts cleanly on the existing data. A major-version bump (e.g. `16` to `17`) would not be compatible and the new container would refuse to start on the old data directory, requiring a `pg_upgrade` or a fresh volume.

Q: What goes wrong if you pin Postgres to a new major version but keep the old named volume, and how do you recognise it?
A: The new major-version server refuses to start because its expected data-directory layout does not match the old cluster's, and the container logs an error like incompatible data directory or a `PG_VERSION` mismatch, leaving the container in a crash-restart loop. You recognise it from `docker compose logs postgres` showing the version-mismatch message and `docker compose ps` showing the container repeatedly restarting or unhealthy. The deliberate fixes are to run `pg_upgrade`, dump-and-restore into a fresh volume, or for a throwaway sandbox simply `docker compose down -v` to start clean.

Q: How do you get an interactive `psql` shell inside the running Postgres container without publishing a port to the host?
A: Run `docker compose exec postgres psql -U <user> -d <db>`, which executes `psql` inside the already-running container over the local socket, so no host port is needed. This is the cleanest way to inspect a database kept on an internal-only network with no `ports` mapping. Use `docker compose run` only when you want a fresh one-off container; for poking at the live service, `exec` attaches to the existing one.

Q: What is the difference between `docker compose exec` and `docker compose run` when you want a shell in a service?
A: `docker compose exec <service> bash` opens a shell inside the already-running container for that service, sharing its state and network exactly as the live service sees it. `docker compose run <service> bash` starts a brand-new, separate container from the same definition, which is isolated, does not occupy the service's name slot, and by default does not publish ports unless you add `--service-ports`. Use `exec` to debug the live service and `run` for ad-hoc throwaway tasks.

Q: A `service_completed_successfully` dependency keeps the dependent from ever starting — what is the first thing to check?
A: Check that the one-shot dependency actually exits with code `0`, because `service_completed_successfully` only releases the dependent on a clean exit; a non-zero exit (a failed migration, a bad seed path) leaves the dependent blocked forever. Run `docker compose logs <job>` and inspect its exit status with `docker compose ps -a` or `docker inspect`. The fix is to make the job succeed, or to correct an over-strict condition if the job is allowed to exit non-zero.

Q: How do you express that a migration container must finish before the loader starts, while the loader also waits for Postgres to be healthy?
A: Give the loader a `depends_on` with two entries: `migrate` carrying `condition: service_completed_successfully` and `postgres` carrying `condition: service_healthy`. Compose then holds the loader until Postgres reports healthy and the migration job has exited `0`, combining a readiness gate and a completion gate. This is the canonical wiring for a schema-then-load sequence in a reproducible stack.

Q: Why is `restart: "no"` (not the default) the right policy for a one-shot loader or migration container, and what breaks if you use `always`?
A: A one-shot job is meant to run once and exit, so `restart: "no"` lets it stop cleanly after success; `restart: always` would relaunch it endlessly after every exit, re-running the load in a loop and possibly thrashing the database. With `always`, the container also never reaches a settled "done" state, so a `service_completed_successfully` gate downstream may behave unexpectedly. Long-running services use `unless-stopped` or `on-failure`; one-shot jobs use `"no"`.

Q: In YAML, why must `restart: "no"` usually be quoted, and what happens if you write it unquoted?
A: YAML's older boolean handling can interpret the bare word `no` as the boolean `false`, so `restart: no` may be parsed as `restart: false` rather than the string `"no"`, which is not a valid restart policy. Quoting it as `"no"` forces the string value Compose expects. The same trap bites `yes`/`on`/`off`, which is why explicit quoting of such scalar values is a defensive habit in Compose files.

Q: What does `docker compose up --build` do that plain `up` does not, and when do you need it?
A: `--build` forces Compose to rebuild images for services that have a `build:` section before starting them, instead of reusing a previously built image. You need it when you have changed a `Dockerfile` or build context and want the running stack to reflect those changes, since `up` alone will happily reuse a stale local image. For services that only reference a pinned `image:` with no `build:`, `--build` has nothing to rebuild.

Q: How does `docker compose up --force-recreate` differ from a normal `up`, and why might you use it?
A: `--force-recreate` recreates containers even when Compose detects no configuration change, discarding the current container instances and starting fresh ones. You use it to clear transient in-container state or to recover from a container that has drifted, without changing the file. It does not touch named volumes, so persistent data survives; only the container processes are replaced.

Q: What does the `--wait` flag on `docker compose up` do, and how does it interact with healthchecks?
A: `docker compose up --wait` starts the stack detached and blocks until all services with healthchecks report `healthy` (and services without one report running), returning a non-zero exit if any fail to become healthy within the timeout. It turns "bring up and confirm readiness" into a single scriptable command, which is ideal for CI and for the lesson's timed reproducibility test. Without healthchecks defined, `--wait` can only confirm containers started, not that they are truly ready.

Q: How would you script the lesson's under-two-minutes acceptance test so it both times the bring-up and fails if the stack is not healthy?
A: Wrap `docker compose up -d --wait` in a timer, for example `time docker compose up -d --wait`, so the command blocks until every healthcheck passes and exits non-zero on failure, and `time` reports wall-clock duration. Precede it with `docker compose down -v` and a `docker system prune` to start cold, then assert the elapsed time is under two minutes. Because `--wait` keys on healthchecks, the measured time reflects real readiness, not mere container launch.

Q: A container shows `(health: starting)` indefinitely and never becomes healthy or unhealthy — what does that tell you?
A: It means the probe is still inside the `start_period` grace window or has never yet returned exit `0`, so Compose has neither confirmed health nor exhausted `retries` to declare it unhealthy. Usually the `test` command is failing every time (wrong binary, wrong credentials) but a generous `start_period` keeps masking it as `starting`, or the `interval` is so long it has barely run. Inspect `.State.Health.Log` via `docker inspect` to read the actual probe output and exit codes.

Q: How do you read the recorded healthcheck failure output for a container directly from the CLI?
A: Run `docker inspect --format '{{json .State.Health}}' <container>` to print the health object, whose `Log` array holds the last few probe runs with their `ExitCode`, `Start`, `End`, and captured `Output`. This shows you exactly what the probe printed and returned, turning a vague "unhealthy" into a concrete error string. It is the fastest way to discover, say, that `pg_isready` is failing with a role-does-not-exist message.

Q: Why might `docker compose down` leave a network behind with an error about it being "in use," and how do you resolve it?
A: It happens when a container outside the project — or a manually started one — is still attached to that network, so Docker refuses to remove it during `down`. Resolve it by finding and stopping the stray container (`docker ps` filtered by network) or running `docker network rm` once nothing is attached. It is a sign of resource leakage that the project-name namespacing is meant to prevent when every container belongs to the same Compose project.

Q: How do you tear down a stack and also remove the images it pulled, not just containers and networks?
A: Run `docker compose down --rmi all` to remove containers, the default network, and all images used by services, or `--rmi local` to remove only images that were built locally without a tag. Combine with `-v` as `docker compose down -v --rmi all` for a maximally clean slate that also drops named volumes. This is heavier than plain `down` and forces a fresh pull/rebuild next time, which is useful for a truly cold reproducibility test.

Q: What is an orphan container in Compose terms, and what does `--remove-orphans` do?
A: An orphan is a container that belongs to the project but whose service no longer exists in the current Compose file — for example a service you renamed or deleted — so it lingers attached to the project. Passing `--remove-orphans` to `up` or `down` cleans these up so the running stack matches the file exactly. Without it, Compose warns about orphans but leaves them running, which can hold ports or networks and confuse the next bring-up.

Q: Why is the `mc ready` command (or MinIO's `/minio/health/live` endpoint) preferable to a TCP-port check in a MinIO healthcheck?
A: A bare TCP-port check passes the moment the socket is listening, which can be before MinIO has finished initialising its storage backend and can actually serve S3 requests, so a loader could still get errors creating buckets. The `mc ready` command and the `/minio/health/live` endpoint report true service readiness, closing that started-versus-ready gap. It is the same principle as preferring `pg_isready` over checking that port `5432` merely accepts a TCP connection.

Q: When the loader writes to MinIO using the S3 API, what endpoint and credentials does it use inside the Compose network versus from the host?
A: Inside the network it targets the MinIO service name and API port, e.g. an endpoint like `http://minio:9000`, with the root user and password set via `MINIO_ROOT_USER`/`MINIO_ROOT_PASSWORD` environment variables. From the host it instead uses `http://localhost:<published-port>` against the same credentials. The host-versus-service-name distinction is identical to the Postgres case and stems from the network boundary.

Q: How do you create the target bucket in MinIO as part of stack startup so the loader does not fail on a missing bucket?
A: Add a short-lived init service using the `minio/mc` image whose command waits for MinIO healthy, sets an alias with `mc alias set`, and runs `mc mb --ignore-existing <alias>/<bucket>`, gated with `depends_on: minio` `condition: service_healthy`. The loader then depends on that init job with `service_completed_successfully` so it only runs once the bucket exists. This mirrors the migration-before-loader pattern but for object storage.

Q: Why should the bucket-creation step use `--ignore-existing` (or be otherwise idempotent), given the lesson's idempotency theme?
A: Because the stack may be brought up repeatedly against a persisted MinIO volume, a plain `mc mb` would error on the second run when the bucket already exists, failing the init job and blocking the loader. `mc mb --ignore-existing` makes the create a no-op when the bucket is already there, so the step converges to the same end state every run. Idempotent setup steps are part of the same re-runnable discipline the loader itself must follow.

Q: How can you point DuckDB at Parquet files sitting in your MinIO bucket, and what extension enables it?
A: You load the `httpfs` extension in DuckDB (`INSTALL httpfs; LOAD httpfs;`), configure the S3 endpoint, region, and credentials via settings like `SET s3_endpoint`, `SET s3_use_ssl=false`, and the access keys, then query with `SELECT * FROM read_parquet('s3://bucket/path/*.parquet')`. Because MinIO speaks the S3 API, DuckDB treats it exactly like AWS S3. This lets the zero-infra engine read straight from the local object store, rehearsing the cloud lakehouse pattern.

Q: Why must you set `s3_use_ssl=false` (or an `http://` endpoint) when DuckDB reads from a local MinIO over plain HTTP, and what error appears if you forget?
A: Local MinIO in a sandbox typically serves the S3 API over unencrypted HTTP, so DuckDB's default TLS expectation must be disabled with `SET s3_use_ssl=false` and a non-TLS endpoint; otherwise the client tries a TLS handshake against a plaintext port and fails. The symptom is a connection or SSL/handshake error rather than a permissions error, which points you at the protocol mismatch, not the credentials. In production against real S3 you would leave SSL on.

Q: What is the practical effect of opening a DuckDB database with `ATTACH 'file.duckdb' (READ_ONLY)` while another process holds the write lock?
A: Opening read-only is permitted to coexist only when no process holds the database in read-write mode; if a writer holds the single-writer lock, even a read-only open of that same file is refused. The read-only flag is what lets multiple readers share a file, but it does not bypass an active writer's exclusive lock. This is the direct consequence of DuckDB's single-writer, multi-reader file model.

Q: How does running DuckDB purely in memory (`:memory:`) change durability and the single-writer constraint?
A: An in-memory DuckDB database lives only in the process's RAM, so nothing is persisted to disk and there is no file to lock or share — when the process exits, the data is gone. It sidesteps the single-writer file lock because there is no file, but it also cannot be reopened by another process or survive a restart. You choose `:memory:` for ephemeral transforms and tests, and a file path when the mart must persist for the capstone.

Q: When would you prefer DuckDB writing to a single `.duckdb` file versus exporting partitioned Parquet to MinIO for the reporting mart?
A: A single `.duckdb` file is simplest for a self-contained, single-tool mart that dbt builds and queries locally with zero infrastructure. Exporting partitioned Parquet to MinIO is preferable when you want the lakehouse pattern — open columnar files that many engines can read, partition pruning, and a cloud-storage-shaped artifact — at the cost of more moving parts. For the Phase-1 capstone the file is usually right; the Parquet-on-object-store path rehearses later cloud phases.

Q: Why is DuckDB's columnar, vectorised execution a better fit for the NAV periodic-snapshot star than Postgres for the same analytical queries?
A: The NAV snapshot fact and its dimensions are scanned and aggregated in wide analytical queries, which suit DuckDB's columnar storage and vectorised, OLAP-tuned engine that reads only the needed columns. Postgres is row-oriented and OLTP-tuned, so the same aggregate scans touch more data and are typically slower without extra indexing or extensions. Using DuckDB for the mart and Postgres for transactional source data matches each engine to its workload.

Q: In a Testcontainers integration test, how does your code learn the actual host and port to connect to, given the random port mapping?
A: After starting the container, you call its accessor methods — for example `getHost()` and `getMappedPort(5432)` in the Java API, or the equivalent properties on the Python container object — to retrieve the runtime host and the ephemeral host port bound to the container's internal port. Many modules also expose a ready-made connection helper, such as a JDBC URL getter. You never hard-code the port; you always ask the container at runtime.

Q: What is the role of the Ryuk reaper in Testcontainers, and what failure does it protect against?
A: Ryuk is a small sidecar container Testcontainers starts that watches the test session via a labelled connection and removes the containers, networks, and volumes the test created once the session ends or the test process dies unexpectedly. It protects against orphaned infrastructure accumulating when a test crashes, is killed, or the JVM exits abnormally before its own cleanup runs. It is why "what a test starts, the framework reliably cleans up" holds even under abrupt termination.

Q: How do Testcontainers and a `docker-compose.yml` complement each other in this lab rather than compete?
A: The Compose file provides the long-lived, hand-driven development sandbox you work against, while Testcontainers provides per-test, code-driven, throwaway dependencies for hermetic integration tests in CI. You develop and explore against Compose, then prove correctness with Testcontainers spinning up a clean real database per test run. Both embody the same reproducible-real-dependency discipline at different points in the workflow.

Q: Why does Testcontainers require a working Docker daemon, and what is the first thing to check when its containers fail to start on a fresh Ubuntu laptop?
A: Testcontainers drives the local Docker engine to create real containers, so it needs a reachable daemon and permission to use it. On a fresh native-Ubuntu laptop the first check is that the user is in the `docker` group (or the test runs with sufficient privilege) and that `docker ps` works without sudo, since a permission-denied on the Docker socket is the most common cause. Confirming the daemon is running and the socket is accessible resolves most "cannot start container" failures.

Q: How does Compose variable interpolation choose a value, and what does `${PG_TAG:-16.3}` mean in an `image:` line?
A: Compose substitutes `${VAR}` from the process environment first, then from the project `.env` file, before rendering the final config. The form `${PG_TAG:-16.3}` supplies a default: it uses `PG_TAG` if it is set and non-empty, otherwise falls back to `16.3`. This lets you parameterise a pinned tag while keeping a safe, explicit default so the stack still reproduces when the variable is unset.

Q: What is the difference between `${VAR:-default}` and `${VAR-default}` in a Compose file?
A: `${VAR:-default}` uses the default when `VAR` is unset or set but empty, whereas `${VAR-default}` uses the default only when `VAR` is entirely unset and treats an empty value as a valid empty string. The colon variant guards against the common case of an accidentally blank variable, which is usually what you want for an image tag. Choosing the wrong form can let an empty `.env` value silently produce `image:` with no tag.

Q: Why is committing a `.env.example` (without secrets) alongside the Compose file good practice for reproducibility?
A: A `.env.example` documents exactly which variables the stack expects and gives safe placeholder values, so a stranger knows to copy it to `.env` and fill in real values, while the actual `.env` stays gitignored and out of history. It makes the configuration contract explicit and auditable without leaking credentials. This pairs with image pinning: the example file pins the shape of configuration just as tags pin the images.

Q: How do you confirm exactly which image tags and interpolated values Compose will use before bringing the stack up?
A: Run `docker compose config`, which renders the fully merged and interpolated configuration — overrides applied, `.env` substituted, defaults resolved — and prints it, so you can read the concrete `image:` tags and resolved variables. Reviewing this output catches an unpinned or mis-substituted tag before any container starts. It is the deterministic preview that turns silent interpolation into something reviewable.

Q: A teammate's `docker compose up` pulls a different image than yours despite the same tag — what reproducibility gap does this reveal and how do you close it?
A: It reveals that a mutable tag was re-pushed: the registry's image behind that tag changed, so a freshly pulling machine gets newer bits than your cached copy under the same tag. You close the gap by pinning to an immutable digest (`image: postgres@sha256:...`) so the reference is content-addressed and identical everywhere. Tags are convenient but only digests guarantee byte-for-byte the same image across machines and time.

Q: Why does pinning by digest matter specifically for defending a recomputed NAV to an auditor years later?
A: A digest names the exact image bytes, so you can prove the recomputation ran on the identical engine version that produced the original figure, with no possibility that a re-pushed tag silently changed behaviour. Under DORA and UCITS/AIFMD scrutiny, demonstrating an unambiguous, immutable toolchain is part of defending the number's provenance. A tag could have drifted; a `sha256` digest cannot, which is why regulated environments prefer it.

Q: How does using `uv run` with a committed lockfile extend the same reproducibility guarantee that image digests give the container layer?
A: `uv run` resolves and runs against the exact package versions recorded in the project's lockfile, so every machine executes the loader with identical Python dependencies, just as a digest fixes the container image. Together they pin both layers — the OS/engine image and the application's libraries — eliminating "works on my machine" drift from either source. The lockfile is to Python packages what the digest is to images: an immutable, auditable record.

Q: What is the risk of leaving `pull_policy: always` on every service, and when is it actually justified?
A: `pull_policy: always` forces a registry pull on every `up`, which slows startup and, on a mutable tag, can silently swap the image out from under you, undermining the reproducibility you sought. It is justified when you deliberately want the freshest build of a moving tag, or in CI where you must not trust a stale cache. For a pinned, reproducible sandbox, pinning by digest with the default pull policy is usually the better lever.

Q: How would you run two copies of this stack side by side on the same laptop without their volumes or networks colliding?
A: Give each copy a distinct project name — via `-p <name>`, the `COMPOSE_PROJECT_NAME` variable, or the top-level `name:` key — and Compose prefixes that name onto every container, network, and named volume, keeping the two sets fully separate. You must also avoid clashing published host ports, since the host port space is shared regardless of project name. Distinct project names give logical isolation; distinct host ports give physical port isolation.

Q: Two stacks each publish Postgres on host port `5432` and the second `up` fails to bind — what is happening and what are your options?
A: The host port `5432` can be bound by only one process, so the second stack's `ports: "5432:5432"` fails with a bind/address-already-in-use error even though the project names differ. Your options are to map the second stack to a different host port (e.g. `"5433:5432"`), to drop the host publish entirely and reach Postgres via the network or `exec`, or to stop the first stack. Project-name isolation does not extend to the shared host port space.

Q: Why is putting the database on an `internal: true` network a stronger isolation guarantee than simply omitting the `ports` mapping?
A: Omitting `ports` only stops host publishing, but the service is still on a network that may also carry outbound-capable services; marking a network `internal: true` blocks any external connectivity for everything attached to it, so the database cannot reach or be reached beyond the stack. The app then bridges the internal network and a separate front-facing one. For sensitive fund data, `internal: true` enforces "database talks only to in-stack peers" at the network level, not just the publishing level.

Q: How do you confirm two services can actually resolve and reach each other by service name inside the stack?
A: Use `docker compose exec <service-a> sh` and from there resolve and probe the peer, for example `getent hosts postgres` or a connection attempt to `postgres:5432`, confirming Compose's embedded DNS maps the service name to the right container IP. If resolution works but the connection is refused, the issue is the target not listening or being on a different network, not DNS. This isolates name-resolution problems from port/listening problems quickly.

Q: Why does `version: '3.8'` at the top of a Compose file no longer control which features are available, and what should you rely on instead?
A: The current Compose Specification is versionless: the top-level `version` key is deprecated and ignored, so it neither enables nor gates features, and a stack's behaviour now depends on the installed Compose plugin's capabilities. You rely instead on pinning a minimum Compose/Docker version (checked with `docker compose version`) and documenting it, since newer keys like `start_interval` require a recent enough plugin. Leaving a stale `version:` in the file is harmless but misleading.

Q: What is the single end-to-end command sequence from the README that proves the whole sandbox is reproducible, and what does each step verify?
A: It is roughly `docker compose down -v && docker system prune -f && docker compose up -d --wait && uv run load.py && uv run verify.py` from a fresh clone: `down -v` and `prune` clear all state and cache, `up --wait` brings the pinned, healthchecked stack to true readiness within the time bar, `uv run load.py` runs the deterministic loader on pinned dependencies, and a verify step asserts row counts and checksums match committed expected values. Each step removes a class of hidden state or environmental drift, so a matching result proves the stack, data, and toolchain are all reproducible for a stranger.

Q: How does the `start_interval` healthcheck key shorten time-to-healthy, and what does it require to work?
A: `start_interval` sets a faster probe cadence used only during the `start_period` grace window, so instead of waiting a full `interval` between early checks the container is probed frequently and flips to healthy the instant it is ready. This can cut several seconds off a cold bring-up where a database becomes ready partway through a long `interval`. It requires a recent enough Compose/Docker version, since older installs do not recognise the key and would ignore or reject it, so document a minimum version in the README.
