# Phase 6 · 11.4.1 PII detection & redaction (Presidio) — 150 self-test questions

Q: What is Microsoft Presidio and what problem does it solve for a data platform?
A: Presidio is an open-source SDK and set of services for detecting and de-identifying PII (personally identifiable information) in free text, images, and structured data. It matters because you cannot protect what you have not found: automated discovery surfaces national IDs, IBANs, and names hiding in "client notes" or a new bronze feed so your masking and access policies actually cover them. A common framing is that Presidio is the FOSS counterpart to managed scanners like AWS Macie, GCP DLP, BigID, or Microsoft Purview classifiers.

Q: Name the three core detection approaches Presidio combines and why no single one is enough.
A: The three are regex/pattern matching (with checksums) for structured formats, NER (named-entity recognition) models for contextual entities like `PERSON` and `LOCATION`, and context-word enhancement that boosts confidence using nearby words. No single approach suffices because regex catches structured IDs but misses names, NER catches names but is weak on formats it never saw, and context words rescue low-confidence matches that sit next to a giveaway word like `IBAN` or `account`.

Q: What are the two main stages of the Presidio pipeline?
A: The pipeline is analyze then anonymize: the `AnalyzerEngine` detects PII entities and returns where they are and how confident it is, and the `AnonymizerEngine` consumes those findings to de-identify the text. Keeping detection and de-identification as separate stages lets you reuse the same analyzer output to feed a catalog, a masking layer, or an audit log without re-scanning.

Q: What are the four main Presidio packages?
A: The four packages are `presidio-analyzer` (detect entities in text), `presidio-anonymizer` (de-identify detected entities), `presidio-image-redactor` (OCR plus detection plus redaction on images), and `presidio-structured` (PII handling in tabular/semi-structured data). You install only what you need; for a lake-scanning job you typically use `presidio-analyzer` plus `presidio-anonymizer`.

Q: What does the `AnalyzerEngine` return for a single detected entity, and what fields does it contain?
A: It returns a `RecognizerResult` per detection containing `entity_type` (e.g. `IBAN_CODE`), `start` and `end` character offsets into the text, and `score` (a confidence from 0.0 to 1.0). The offsets let the anonymizer replace exactly the right span, and the score lets you apply a threshold so low-confidence noise is dropped.

Q: What is the role of the `RecognizerRegistry` inside the analyzer?
A: The `RecognizerRegistry` holds every recognizer available to the analyzer — the predefined ones plus any custom recognizers you add. When `analyze()` runs, the engine asks each registered recognizer for the requested entities and language, so adding or removing a recognizer is just registering or deregistering it rather than editing engine code.

Q: What is the `NlpEngine` in Presidio responsible for?
A: The `NlpEngine` runs the underlying NLP model to produce linguistic artifacts — tokens, lemmas, and model-detected named entities — that recognizers and the context enhancer consume. Presidio supports spaCy, Stanza, and Hugging Face Transformers backends, so you can trade a small fast spaCy pipeline for a heavier transformer model when recall on names matters more than speed.

Q: What is a `PatternRecognizer` and when would you reach for one?
A: A `PatternRecognizer` is a built-in class that detects entities from one or more regular expressions, optionally with validation logic and context words. You reach for it whenever a PII type has a deterministic format — an internal fund-account number, a Luxembourg phone, a custom client reference — because writing a regex recognizer is far cheaper than training a model.

Q: What is the difference between a predefined recognizer and a custom recognizer?
A: Predefined recognizers ship with Presidio for common entities (`CREDIT_CARD`, `EMAIL_ADDRESS`, `IBAN_CODE`, `PERSON`, etc.), while custom recognizers are ones you author for your own entities or to override default behaviour. You add a custom one by creating a `PatternRecognizer` (or subclassing `EntityRecognizer`) and registering it with the `RecognizerRegistry` so the analyzer starts emitting that entity type.

Q: Why does Presidio use checksum validation for entities like `IBAN_CODE` and `CREDIT_CARD`?
A: A bare regex matches anything shaped like the identifier, producing many false positives, so Presidio validates the structure mathematically: `CREDIT_CARD` uses the Luhn algorithm and `IBAN_CODE` uses the ISO 13616 mod-97 check. Passing the checksum raises confidence and lets the recognizer assign a high score, because a random 16-digit string almost never satisfies Luhn.

Q: Which Presidio entity detects international bank account numbers, and how does it validate them?
A: The `IBAN_CODE` recognizer detects IBANs and validates them with the IBAN mod-97 checksum (rearrange, convert letters to digits, the whole number mod 97 must equal 1). This is directly relevant to fund administration, where investor and settlement records are full of IBANs that must be discovered and tagged.

Q: What is "context-aware enhancement" in Presidio?
A: Context-aware enhancement raises the confidence score of a candidate entity when configured context words appear nearby in the text. The default implementation, `LemmaContextAwareEnhancer`, lemmatizes surrounding tokens so that `accounts`, `account`, and `accounting` all match the context word `account`, letting a borderline regex hit cross the acceptance threshold.

Q: What are the two key tuning parameters of `LemmaContextAwareEnhancer` and their defaults?
A: They are `context_similarity_factor`, the amount added to the score when a context word is found (default `0.35`), and `min_score_with_context_similarity`, the floor the boosted score is raised to (default `0.4`). So a weak pattern hit near a context word is bumped up by `0.35` and never left below `0.4`, which is how a low-base recognizer becomes reliable in the right context.

Q: Why does lemmatization matter for context words rather than plain string matching?
A: Lemmatization reduces inflected forms to a base lemma, so one context word `account` matches `accounts`, `accounting`, and `accounted` without you listing every variant. Plain substring matching would miss the plural or the participle, weakening the boost exactly where natural prose varies its wording.

Q: What does it mean that a pattern recognizer assigns a "base score," and how does context interact with it?
A: Each regex in a `PatternRecognizer` carries a base confidence score reflecting how specific the pattern is; a loose pattern is given a low base score on purpose. The context enhancer then adds `context_similarity_factor` when a context word is present, so you deliberately keep the base low to avoid false positives in isolation while still catching the entity when context confirms it.

Q: What is the analyzer "decision process" and why is it useful?
A: Presidio can record a decision trace explaining each result — which recognizer fired, the pattern or model that matched, the raw score, and any context boost applied. This is invaluable when tuning, because instead of guessing why an IBAN was missed you can read exactly whether the regex matched, the checksum failed, or the score fell below your threshold.

Q: What is a `score_threshold` and how does changing it affect results?
A: `score_threshold` is the minimum confidence a `RecognizerResult` must have to be returned by `analyze()`. Raising it drops low-confidence matches (fewer false positives, lower recall), while lowering it keeps marginal matches (higher recall, more false positives) — it is your primary global precision-recall dial.

Q: How do you call the analyzer on a piece of text in code, conceptually?
A: You construct `AnalyzerEngine()`, then call `analyzer.analyze(text=..., language="en")`, optionally passing `entities=[...]` to restrict the search and `score_threshold=...`. It returns a list of `RecognizerResult` objects, which you then hand to the anonymizer or push to your catalog.

Q: What does the `entities` argument to `analyze()` do, and why restrict it?
A: Passing `entities=["IBAN_CODE", "PERSON"]` tells the analyzer to run only the recognizers for those types and ignore the rest. Restricting the list speeds up scanning and avoids spurious tags — a bronze feed you know contains only IBANs and names should not be scanned for `US_SSN`.

Q: What is the default language for Presidio detection and how do you support others?
A: English (`en`) is the default. To support another language you configure the `NlpEngine` with a model for that language (e.g. a French or German spaCy model) and provide language-specific recognizers and context words, then pass `language="fr"` to `analyze()`.

Q: For a Luxembourg fund dataset that mixes English, French, and German prose, what does multi-language support require in Presidio?
A: It requires an `NlpEngine` configured with a model per language, recognizers whose patterns and context words exist in each language, and code that detects or is told the language so it can pass the right `language=` value. Luxembourg client notes routinely switch among the three, so a single English-only pipeline will silently under-detect names and context in the French and German rows.

Q: What does the `AnonymizerEngine` do, and what does it take as input?
A: The `AnonymizerEngine` de-identifies text by applying operators to the spans the analyzer found; it takes the original `text`, the list of `analyzer_results`, and an optional `operators` mapping of entity type to `OperatorConfig`. It returns the transformed text plus metadata about each replacement so you can audit what was changed.

Q: List the built-in Presidio anonymizer operators.
A: The built-in operators are `replace`, `redact`, `mask`, `hash`, `encrypt`, and `keep`. Each is selected per entity type via an `OperatorConfig`, so you can replace names but mask card numbers and encrypt IBANs in a single pass.

Q: What does the `replace` operator do and what is its default behaviour?
A: `replace` substitutes the detected span with a fixed string given by `new_value`; if you omit `new_value` it defaults to the bracketed entity type such as `<PERSON>` or `<IBAN_CODE>`. It is the most common choice for analytics output because it preserves readability while removing the actual identifier.

Q: What does the `redact` operator do?
A: `redact` deletes the detected span entirely, leaving nothing in its place, and takes no parameters. Use it when even a placeholder token would leak structure or length you do not want exposed downstream.

Q: What does the `mask` operator do and what are its parameters?
A: `mask` overwrites part of the value with a repeated character; its parameters are `masking_char` (the fill character), `chars_to_mask` (how many characters to hide), and `from_end` (whether to count from the end of the string). It is ideal for "show last four" patterns — mask all but the last four digits of a card by masking from the start.

Q: What does the `hash` operator do, what `hash_type` values exist, and what is the default?
A: `hash` replaces the value with a cryptographic hash; `hash_type` may be `sha256`, `sha512`, or `md5`, with `sha256` as the default. Hashing the same input gives the same output, so a hashed investor ID still supports joins and counts across records without exposing the raw value.

Q: Why might you add a `salt` when hashing PII, and what is the trade-off?
A: A salt is appended before hashing to defeat precomputed rainbow-table or brute-force lookups, since unsalted hashes of short, low-entropy values like a national ID can be reversed by guessing. The trade-off is that you must reuse the same salt to keep hashes joinable across datasets, so the salt itself becomes a secret you protect.

Q: What does the `encrypt` operator do and what does it require?
A: `encrypt` performs reversible AES encryption of the detected value and requires a `key`. Unlike hashing it can be reversed later with the same key, which is exactly why it pairs with the deanonymizer for cases where an authorized process must recover the original.

Q: What does the `keep` operator do and when is it appropriate?
A: `keep` leaves the detected value unchanged in the output. You use it when a recognizer flags an entity you have decided is not sensitive in this context — for example a public fund LEI — but you still want the analyzer to detect and tag it for cataloguing.

Q: What is the `OperatorConfig` object?
A: `OperatorConfig` bundles the operator name with its parameters for a given entity type, e.g. `OperatorConfig("mask", {"masking_char": "*", "chars_to_mask": 12, "from_end": True})`. You pass a dict of `{entity_type: OperatorConfig}` to the anonymizer so each PII type gets a tailored treatment.

Q: How does the Presidio `DeanonymizeEngine` work and which operations are reversible?
A: The `DeanonymizeEngine` reverses operations that retained the information, and `decrypt` is the canonical reversible one — given the same `key` used by `encrypt`, it restores the original value. `redact`, `mask`, and `hash` are irreversible by design, so only `encrypt` round-trips.

Q: Why are `hash`, `mask`, and `redact` considered irreversible while `encrypt` is not?
A: Hashing is a one-way function, masking overwrites characters with a fixed symbol, and redaction deletes the span — in all three the original information is destroyed and cannot be reconstructed. Encryption transforms the value with a key in a way that is mathematically invertible, so possessing the key restores the plaintext.

Q: For a NAV report shared with external auditors, which anonymizer operator best handles investor IBANs while keeping records linkable, and why?
A: `hash` with a consistent salt is usually the best fit: it removes the real IBAN yet produces a stable token so the auditor can confirm two rows reference the same account without ever seeing the number. If the auditor must occasionally recover the original under a controlled process, `encrypt` is the alternative, but it introduces a key you must safeguard.

Q: Distinguish "at-rest" scanning from "in-motion" scanning in a PII context.
A: At-rest scanning runs over data already stored — files in the lake, bronze tables, document stores — typically as a batch job that lists objects and analyzes their contents. In-motion scanning intercepts data as it flows through a stream or pipeline and analyzes records before they land, catching PII at the boundary rather than after it has been persisted.

Q: What is the main advantage of in-motion (stream interception) scanning over at-rest scanning?
A: In-motion scanning can stop or tag PII before it ever lands, so an un-tagged national ID never persists in bronze where it could be queried by the wrong person in the gap before the next at-rest scan. The Done-when goal — new PII gets discovered, tagged, and policy-covered "without a human noticing it first" — is best met by catching it in motion.

Q: What is the main advantage of at-rest scanning, despite the latency window?
A: At-rest scanning gives you complete coverage of everything already stored, including historical data that predates any streaming controls and feeds that bypass your pipeline. It is also where you discover the unknown: a CSV someone dropped in the lake last year with national IDs that no in-motion control ever saw.

Q: Why is scanning free-text "client notes" harder than scanning a structured IBAN column?
A: A structured IBAN column has a known position and format you can validate by checksum, whereas client notes are unconstrained prose where a name, phone, or account number can appear anywhere in any phrasing. Free text therefore leans heavily on NER and context enhancement rather than column-level rules, and is where most surprise PII hides.

Q: What is "precision" in the context of tuning a PII recognizer?
A: Precision is the fraction of the recognizer's positive detections that are actually PII — true positives divided by all positives (true plus false). Low precision means the recognizer is noisy, flagging non-PII like order numbers as IBANs, which floods your catalog with bad tags and erodes trust in the masking layer.

Q: What is "recall" in the context of tuning a PII recognizer?
A: Recall is the fraction of real PII the recognizer actually catches — true positives divided by all real PII (true positives plus false negatives). Low recall is the dangerous failure for compliance: real national IDs slip through untagged and unmasked, exactly the leak the lesson exists to prevent.

Q: Describe the precision-recall trade-off when you tighten a recognizer's pattern or raise its threshold.
A: Tightening the pattern or raising `score_threshold` removes marginal matches, which usually increases precision but lowers recall, because some genuine but oddly formatted PII no longer qualifies. Loosening them does the reverse — higher recall, lower precision — so tuning is choosing where on that curve your regulatory risk tolerance sits.

Q: For PII discovery in a regulated fund context, why is recall usually weighted more heavily than precision?
A: A false negative means a real national ID or IBAN goes untagged and unmasked, which is a data-protection breach an auditor will penalise, while a false positive merely over-tags a harmless column that a human can later clear. Under GDPR and CSSF scrutiny the cost of missing PII far exceeds the cost of an extra tag, so you bias toward recall.

Q: How would you raise recall for a Luxembourg IBAN recognizer that is missing some real IBANs?
A: Broaden the regex to tolerate spaces and grouping variations, add relevant context words (`IBAN`, `compte`, `account`, `bank`) so context enhancement boosts weak hits, and lower the `score_threshold`. Then re-measure on your labelled set, because each loosening can drag in false positives you must check.

Q: How would you raise precision for a recognizer that is over-flagging non-IBAN strings?
A: Keep or add the mod-97 checksum validation so only structurally valid IBANs score high, tighten the regex (require the country-code prefix and correct length), and require a context word before accepting low-base matches. Validation is the highest-leverage lever because a random number almost never passes the checksum.

Q: Why are EU national identifiers particularly tricky to detect, and how does Presidio help?
A: EU national IDs vary by country in length, format, and embedded checksums (Spanish NIF/NIE, Italian fiscal code, Polish PESEL, etc.), so one regex cannot cover them all. Presidio ships country-specific recognizers (`ES_NIF`, `IT_FISCAL_CODE`, `PL_PESEL`, and more) and lets you add your own for jurisdictions it does not cover.

Q: How would you measure the precision change after tuning one recognizer, as the lesson's Do step asks?
A: Build a labelled evaluation set where you know which spans are truly PII, run the analyzer before and after the change, and compute precision (and recall) as true positives over predicted positives for each version. Reporting the delta — for example precision rising from 0.78 to 0.93 on the Luxembourg phone recognizer — is the concrete deliverable the lesson requires.

Q: Why must your evaluation set be labelled ground truth rather than just running the scanner and eyeballing it?
A: Without labelled ground truth you have no denominator for recall — you cannot count the PII the scanner missed because you never marked where it was. You also cannot compute precision objectively, since deciding which flags were wrong is exactly the labelling you are trying to avoid; the ground truth turns tuning from opinion into measurement.

Q: What is a confusion matrix and how does it relate to precision and recall?
A: A confusion matrix tabulates true positives, false positives, true negatives, and false negatives for the detector. Precision is TP/(TP+FP) and recall is TP/(TP+FN), so the matrix is the single artifact from which both numbers — and the F1 that balances them — are derived.

Q: What is the F1 score and why might you report it when tuning a recognizer?
A: F1 is the harmonic mean of precision and recall, `2*P*R/(P+R)`, summarising both in one number that is only high when both are high. It is useful when you want a single comparison between a "before" and "after" recognizer, though for compliance you should still inspect recall separately so a good F1 does not hide a recall regression.

Q: How would you build a custom recognizer for a Luxembourg phone number?
A: Create a `PatternRecognizer` with `supported_entity` set (e.g. `LU_PHONE_NUMBER` or reuse `PHONE_NUMBER`), regex patterns covering the `+352` country code and local grouping with appropriate scores, and context words like `phone`, `tel`, `téléphone`, `mobile`. Register it with the `RecognizerRegistry`, then evaluate on labelled data and adjust scores and patterns until precision and recall are acceptable.

Q: Why prefer `phonenumbers`-style validation or context words over a single greedy phone regex?
A: A greedy phone regex matches any run of digits with separators, so it tags order numbers, dates, and reference codes as phones, destroying precision. Adding format-aware validation and requiring a context word means a bare 8-digit string is only accepted as a phone when something nearby actually says phone, sharply cutting false positives.

Q: What is the risk of setting an entity's base pattern score too high?
A: A high base score makes the recognizer accept matches with no supporting context, so every string that merely looks like the pattern is flagged and precision collapses. You generally want the base score low enough that the entity is only confidently accepted when context enhancement or a checksum confirms it.

Q: How do classification tags from Presidio flow into OpenMetadata?
A: After scanning, you map each detected entity type to an OpenMetadata classification tag (for example `PII.Sensitive`) and apply that tag to the affected column or asset via the OpenMetadata API or an ingestion workflow. The catalog then carries the PII label as durable governance metadata that other systems read.

Q: What are Classifications and Tags in OpenMetadata, and how do they relate to PII?
A: A Classification is a named group of related Tags — `PII` is a built-in classification containing tags like `PII.Sensitive` and `PII.NonSensitive`. Applying `PII.Sensitive` to a column records that it holds sensitive personal data, which downstream policies and masking rules can then key off.

Q: What two mechanisms does OpenMetadata's own auto-classification use, and how do they differ?
A: It uses a column-name scanner (regex over column names so `email` or `national_id` gets tagged on naming alone) and, when sample-data ingestion is on, an NLP entity-recognition pass over actual row values to catch PII in generically named columns. The name scanner is cheap but shallow; the data scan is deeper but needs sampled values and a confidence threshold (default 80%).

Q: Why might you push Presidio findings into OpenMetadata rather than relying solely on OpenMetadata's built-in auto-classification?
A: Presidio gives you full control over recognizers, custom EU identifiers, checksums, and context tuning that the built-in classifier does not expose, plus you can scan free-text and streams the catalog ingestion would not sample. Pushing Presidio's findings in as tags lets the catalog remain the single governance source of truth while your detection logic stays as strong as you need.

Q: How does a PII classification tag in the catalog actually drive masking or access control?
A: The masking/access layer reads the catalog tags and applies policy — a column tagged `PII.Sensitive` automatically gets a masking rule or a restricted access policy, with no per-column manual configuration. This is the chain the lesson tests: detect with Presidio, tag in OpenMetadata, and have the policy layer enforce it because of the tag.

Q: The lesson's Done-when says new PII should be "discovered, tagged, and policy-covered without a human noticing it first." What pipeline achieves this?
A: A scan (in-motion interception or a scheduled at-rest job) runs Presidio on new bronze data, maps findings to catalog tags via the OpenMetadata API automatically, and the masking/access layer enforces policy on any newly tagged column. The whole loop is automated so a freshly landed national ID is covered before any analyst queries it.

Q: How would you confirm that a newly tagged column is actually picked up by the masking layer?
A: Land a known PII value in a new column, let the scan tag it `PII.Sensitive`, then query that column as an unprivileged user and verify you see the masked or hashed form rather than the raw value. The end-to-end test must go through a real downstream read, because a tag that no policy consumes proves nothing.

Q: A synthetic "client notes" record contains "Call Jean at +352 691 234 567 re account LU28 0019 4006 4475 0000." Which Presidio entities should fire?
A: `PERSON` for "Jean" (from NER), `PHONE_NUMBER` for the `+352` Luxembourg number (pattern plus context word "Call"/"re"), and `IBAN_CODE` for the `LU28...` IBAN (pattern plus mod-97 checksum, boosted by the context word "account"). This single sentence illustrates all three detection approaches firing together.

Q: Why would the IBAN in that note score higher than a random 20-character alphanumeric string?
A: The IBAN passes the `LU` country-code prefix, correct length, and the mod-97 checksum, so the recognizer assigns a high base score, and the nearby word "account" triggers context enhancement to add up to `0.35` more. A random string fails the checksum and gets no context boost, so it stays below threshold and is correctly ignored.

Q: A recognizer flags every fund order number as a `CREDIT_CARD`. What is the first thing to check?
A: Check whether checksum validation is enabled — a credit-card recognizer without Luhn validation will accept any 12–19 digit run, including order numbers. Order numbers almost never satisfy Luhn by chance, so re-enabling the checksum (or confirming the order format genuinely collides with it) is the fastest precision fix.

Q: Presidio misses IBANs that staff typed with no spaces. First thing to check?
A: Check the IBAN recognizer's regex against your actual data format — many patterns assume grouped, space-separated input and fail on a continuous 20-character string, or vice versa. Confirm by running the decision trace on a missed example to see whether the pattern matched at all before blaming the checksum or score.

Q: Detection works in a notebook but the scheduled scan tags nothing. First things to check?
A: Verify the scheduled job is reading the same data and passing the same `language` and `entities` arguments, and that the `NlpEngine` model is actually installed in the job's environment. A very common cause is the spaCy model present in your notebook but missing from the batch container, so NER silently returns nothing and only regex entities survive.

Q: An auditor asks "are investor national IDs in this new feed masked?" and the column is untagged. What likely failed?
A: Either the scan never ran on that feed, the recognizer for that country's national ID is missing or scored below threshold, or the findings were not pushed to the catalog as tags. Trace it backwards: confirm the scan covered the feed, that Presidio detected the ID on a sample, and that the tag-push step succeeded — the break is in whichever link is empty.

Q: Why is the spaCy NER model a common deployment gotcha for Presidio?
A: Presidio needs the language model downloaded (for example via the spaCy model download step) and present in the runtime; if it is absent, the analyzer can fail to load or fall back so NER entities like `PERSON` never fire. The fix is to bake the model into the image or download it at build time, not assume it is available.

Q: Why must your "synthetic client notes" test set be synthetic and not real client data?
A: Testing PII detection on real investor notes would itself process and potentially expose live personal data, which under GDPR and fund-confidentiality rules is exactly what you are trying to avoid. Synthetic data with known planted PII gives you a safe, fully labelled ground truth to measure precision and recall without handling anyone's real information.

Q: What is the difference between an LEI and an entity Presidio would normally treat as PII?
A: An LEI (Legal Entity Identifier) identifies a legal entity such as a fund or company, not a natural person, so it is reference data rather than personal data and generally not GDPR PII. You would not normally mask it; if a custom recognizer flags it, `keep` is the appropriate operator so it is catalogued but left readable.

Q: Is an ISIN considered PII that Presidio should redact?
A: No — an ISIN identifies a security (a share class or instrument), not a person, so it is instrument reference data and not personal data to redact. Conflating instrument identifiers with PII would over-mask your NAV and holdings reports; only person-linked fields like names, national IDs, and IBANs are the PII targets here.

Q: Why is an investor IBAN PII even though an instrument ISIN is not?
A: An IBAN is tied to a specific person's or entity's bank account and can be used to identify or contact that account holder, making it personal data under GDPR when it belongs to a natural person. An ISIN identifies a tradable security with no link to an individual, so it carries no personal-data risk.

Q: A bronze feed of transfer-agency records has a `notes` column with occasional passport numbers. How does Presidio surface them?
A: NER and pattern recognizers scan the free-text `notes` values; country passport recognizers (for example `US_PASSPORT`, `UK_PASSPORT`) and custom ones you add match the passport formats, with context words like "passport" boosting weak hits. Because the field is unstructured, this is exactly where column-name-only classifiers fail and content scanning earns its keep.

Q: How does `presidio-structured` differ from running the text analyzer over each cell?
A: `presidio-structured` is designed for tabular data: it can detect PII at the column level and apply consistent anonymization across a table, rather than treating every cell as an isolated string. This gives column-aware decisions — once a column is identified as IBANs, the whole column is handled uniformly — which suits bronze tables better than per-cell text scanning.

Q: Why might you scan a column's sampled values rather than every row for at-rest classification?
A: Scanning a representative sample is far cheaper and usually enough to decide a column's type and tag it, because a column of IBANs reveals itself in a handful of rows. The trade-off is that a rare stray PII in an otherwise clean column can be missed, so high-risk feeds may warrant full scans despite the cost.

Q: What does it mean to "trace each decision" in Presidio and how does it aid tuning?
A: Enabling decision-process tracing makes each `RecognizerResult` carry an explanation of which recognizer and pattern matched, the original score, and any context improvement. During tuning you read these traces to see precisely why a result was kept or dropped, turning recall and precision fixes into targeted edits rather than blind trial and error.

Q: Why can NER-based `PERSON` detection produce both false positives and false negatives on fund data?
A: NER models are trained on general prose, so they miss unusual or non-Western names (false negatives) and can flag fund names, place names, or product names that look like people (false positives). Fund datasets are full of entity names like "Aberdeen" or "Luxembourg" that confuse a general model, which is why you tune thresholds and add domain recognizers.

Q: How do context words help disambiguate an 8-digit number that could be a phone or an order id?
A: If the configured phone context words (`phone`, `tel`, `mobile`) appear near the number, the context enhancer boosts the phone recognizer's score above threshold; with no such words the weak base score keeps it from being tagged. The surrounding language, not the digits alone, decides the call — which is exactly how humans read it too.

Q: What is the danger of leaving `score_threshold` at a low global value across all entity types?
A: A single low global threshold may be right for high-stakes recall on national IDs but wrong for noisy entities, so loose recognizers flood the catalog with false positives. Per-recognizer scoring and validation are better tools, letting you keep recall high on critical IDs without dragging in junk from weak patterns.

Q: How would you handle an entity type where you want high recall but the catalog cannot tolerate the resulting false positives?
A: Keep detection recall high but route low-confidence hits to human review or a "suspected PII" tag rather than auto-applying the enforced `PII.Sensitive` tag. This separates discovery from enforcement, so you find everything while only acting automatically on high-confidence findings, with the rest triaged.

Q: Why does the lesson tie PII findings into the catalog (OpenMetadata) rather than handling masking directly from Presidio?
A: Centralising findings as catalog tags makes the catalog the single governance source of truth that many systems consume — masking, access policy, lineage, and audit all read the same tags. If Presidio masked directly and quietly, the catalog and downstream tools would never know the column was sensitive, breaking governance consistency.

Q: What is the relationship between this PII-tagging step (11.4.1) and a masking/policy layer described elsewhere in the plan?
A: 11.4.1 produces the classification tags (the "what is sensitive" signal), and the masking/access layer consumes those tags to enforce "who can see it in what form." The tag is the contract between detection and enforcement, which is why the Done-when test insists a freshly tagged column is automatically policy-covered.

Q: A privileged service account must read raw IBANs for SWIFT payment generation, but analysts must not. How do tags plus policy reconcile this?
A: The IBAN column is tagged `PII.Sensitive`, and the policy layer grants the payment service account an unmasked view while masking or hashing the same column for analyst roles. The tag is one fact; the policy makes role-dependent decisions on top of it, so the SWIFT process works without exposing IBANs to general analytics.

Q: Why is hashing (rather than encryption) often preferred for IBANs used only in analytics joins?
A: Hashing is irreversible, so a leaked analytics table never yields the real IBANs, yet a consistent salted hash still lets you join and count by account. Encryption would also protect the value but introduces a recoverable key and key-management burden you do not need when nobody downstream must read the original.

Q: When is `encrypt` the right choice over `hash` for an entity?
A: Choose `encrypt` when an authorized downstream process must recover the original value — for example a reconciliation job that needs the real account number under controlled access. Because `encrypt` is reversible with the key via the deanonymizer, it supports recovery that a hash cannot.

Q: What operational concern does using `encrypt` introduce that `hash` does not?
A: `encrypt` requires you to generate, store, rotate, and tightly access-control the encryption key, because anyone with the key can decrypt every protected value. Hashing needs only a salt and never produces recoverable plaintext, so it sidesteps the full key-management lifecycle.

Q: How does `mask` differ from `replace` in what it preserves?
A: `mask` keeps the value's length and any unmasked characters (such as the last four digits), substituting the rest with a fill character, while `replace` swaps the whole span for a single fixed token regardless of length. Use `mask` when partial visibility aids verification, `replace` when you want no trace of the original shape.

Q: Why might you anonymize the same text differently for two different downstream consumers?
A: Different consumers have different needs and clearances: a public dashboard might `replace` names with `<PERSON>`, an analyst extract might `hash` IBANs for joins, and a payment job might `keep` them. Because the analyzer output is reusable, you run distinct anonymizer configurations on the same detected spans rather than re-scanning per consumer.

Q: What is the consequence of scanning bronze but never re-scanning when the schema or feed changes?
A: New columns or a changed feed can introduce PII that the original scan never saw, so it lands untagged and unmasked until something flags it — often an auditor. Continuous or trigger-based scanning on schema/feed changes is what keeps the "discovered before a human notices" guarantee from quietly lapsing.

Q: Why is a fund's "client notes" free-text field a higher PII risk than its structured KYC fields?
A: Structured KYC fields have known types and are usually already governed and masked, whereas free-text notes are an uncontrolled channel where staff paste IDs, phones, and account numbers in unpredictable phrasing. That makes notes the field most likely to leak un-tagged PII, which is why the lesson explicitly targets a synthetic client-notes set.

Q: How would you detect that a recognizer change improved precision but quietly hurt recall?
A: Always report both metrics on the same labelled set before and after the change; a precision gain with a recall drop shows up as more missed true positives in the confusion matrix. Reporting only the precision delta the lesson asks for can hide a dangerous recall regression, so track recall alongside it.

Q: What does it mean if the analyzer returns overlapping `RecognizerResult` spans for the same text?
A: Two recognizers matched overlapping or identical spans (e.g. a generic and a country-specific ID recognizer both fired), giving you duplicate or conflicting findings. Presidio resolves overlaps by keeping the higher-scoring or longer match, but you should understand the conflict so your tags do not double-count or mislabel the entity.

Q: Why should context words be chosen from the actual domain language rather than generic English?
A: Context enhancement only fires on words that appear near real entities in your data, so for Luxembourg transfer-agency text you need terms like `compte`, `IBAN`, `téléphone`, and `bénéficiaire`, not just English equivalents. Domain- and language-specific context words are what let the boost work on the prose your fund data actually contains.

Q: How does adding a checksum to a custom recognizer affect its precision and base-score strategy?
A: A checksum lets you assign a high score with confidence because only structurally valid values pass, sharply raising precision and reducing reliance on context. Without a checksum you must keep the base score low and lean on context words, accepting more tuning effort to avoid false positives.

Q: What is the risk of over-redacting fund reports with an aggressive PII scanner?
A: Over-redaction removes or masks values that are not actually personal data — instrument identifiers, fund names, public LEIs — making reports unusable and eroding confidence in the tool. The fix is recognizer scoping, `keep` for known-safe entities, and measuring precision so you can prove the scanner is not destroying legitimate content.

Q: Why does separating "detect" from "anonymize" make Presidio suitable for both tagging and redaction use cases?
A: Because the analyzer's `RecognizerResult` list is a neutral description of where PII is, you can feed it to the anonymizer to redact, or to a catalog-tagging routine to label columns, or to an audit log — all from one scan. Coupling detection to a single fixed action would force a re-scan for each new use.

Q: A streaming bronze ingestion must not block on slow NER. How do you keep in-motion scanning fast?
A: Use a lighter NLP backend (a small spaCy pipeline) or restrict per-record `entities` to the few types that feed expects, and rely on fast regex-plus-checksum recognizers where possible. You can also batch records through the engine, since Presidio's NLP backends support batch processing, to amortise model overhead.

Q: What is the difference between `PII.Sensitive` and `PII.NonSensitive` tags in OpenMetadata, and why does the distinction matter?
A: `PII.Sensitive` marks data whose exposure is harmful and must be masked or restricted (national IDs, IBANs, names), while `PII.NonSensitive` marks personal-but-lower-risk data. The distinction drives proportionate policy — you do not want to lock down a low-risk field as hard as a national ID, nor leave a sensitive one merely labelled.

Q: After Presidio tags a column, why is it important that the policy enforcement be automatic rather than a manual follow-up?
A: Manual enforcement reintroduces the human-in-the-loop gap the lesson exists to close — between tagging and someone remembering to apply masking, the raw PII is queryable. Automatic policy binding off the tag is what makes "discovered, tagged, and policy-covered without a human noticing" actually true.

Q: What is the difference between de-identification and full anonymization in the GDPR sense, and where does Presidio sit?
A: De-identification removes direct identifiers but may leave data re-linkable if you hold extra keys, whereas true anonymization is irreversible so the data falls outside GDPR. Presidio performs de-identification: `replace`, `mask`, and `redact` move you toward anonymization, but reversible `encrypt` or stable salted `hash` keep the result pseudonymized and still in GDPR scope, which you must document honestly.

Q: Why is pseudonymization (e.g. salted hashing) still regulated personal data under GDPR even though the raw value is hidden?
A: Pseudonymized data can be re-associated with an individual by someone holding the salt, key, or a linking table, so GDPR explicitly treats it as personal data with reduced but not eliminated risk. This is why a hashed-IBAN analytics table must still be access-controlled and catalogued as PII rather than treated as freely shareable anonymous data.

Q: Summarise the end-to-end flow this lesson asks you to build and demonstrate.
A: Scan bronze and a synthetic client-notes set with Presidio (regex/checksum, NER, and context approaches), push the findings into OpenMetadata as `PII.Sensitive` classification tags, tune one recognizer (Luxembourg phone or IBAN) and report the precision change, and confirm tagged columns are automatically picked up by the masking/access layer. The proof is landing fresh PII in bronze and watching it get discovered, tagged, and masked before any human intervenes.

Q: What is the full parameter list of `AnalyzerEngine.analyze()` beyond `text` and `language`?
A: The signature accepts `entities`, `correlation_id`, `score_threshold`, `return_decision_process`, `ad_hoc_recognizers`, `context`, `allow_list`, `allow_list_match`, `regex_flags`, and `nlp_artifacts`. Knowing these lets you tune a single call — for example pass `return_decision_process=True` to debug a missed IBAN and `allow_list=["LU0000000000"]` to suppress a known test value — without rebuilding the engine.

Q: What does the `allow_list` parameter of `analyze()` do, and how does `allow_list_match` change its behaviour?
A: `allow_list` is a list of strings that should never be flagged as PII even if a recognizer matches them, useful for whitelisting public reference values or synthetic test tokens. `allow_list_match` defaults to `"exact"` (the matched span must equal a list item) but can be set to `"regex"` so each list entry is treated as a pattern, letting you allow a whole family like all fund codes starting `LU` without enumerating each one.

Q: How do you pass a recognizer for one request only without permanently registering it?
A: Pass it in the `ad_hoc_recognizers` argument of `analyze()`, a list of `EntityRecognizer` instances that apply only to that single call and are discarded afterward. This is ideal for a one-off scan that needs a special internal-reference recognizer you do not want polluting every future analysis run.

Q: What is `BatchAnalyzerEngine` and which method scans an iterable of strings?
A: `BatchAnalyzerEngine` wraps `AnalyzerEngine` to scan many values at once, and its `analyze_iterator` method takes an iterable of texts plus `language`, returning a list of result lists aligned to the inputs. It is the right entry point for scanning a whole bronze column row by row rather than calling `analyze()` in a hand-written Python loop.

Q: Which two parameters of `analyze_iterator` control throughput, and what do they tune?
A: `batch_size` sets how many texts the underlying spaCy/transformers pipeline processes per batch, and `n_process` sets the number of worker processes for parallel NLP. Raising `batch_size` amortises model overhead while `n_process` spreads CPU work across cores — both matter when a nightly at-rest scan must finish in its window.

Q: What is `BatchAnonymizerEngine` and when do you use it?
A: `BatchAnonymizerEngine` is the batch counterpart that anonymizes a list or dict of values given the batch analyzer's results, applying the same operator config across every row. You use it to de-identify a whole structured column or a dictionary of records in one call rather than anonymizing each cell individually.

Q: What does the `global_regex_flags` parameter on `RecognizerRegistry` control, and what is the typical default?
A: `global_regex_flags` sets the Python `re` flags applied to every pattern recognizer's regex, defaulting to `re.DOTALL | re.MULTILINE | re.IGNORECASE`. Setting it once on the registry (or per-recognizer via the `PatternRecognizer` `global_regex_flags` arg) means you do not repeat `(?i)` in every pattern, and `re.IGNORECASE` is why `iban` and `IBAN` context both match.

Q: How does a Presidio deny-list recognizer differ from a regex `PatternRecognizer`?
A: A deny-list recognizer matches against an explicit list of forbidden literal strings rather than a regular expression, built by passing `deny_list=[...]` to `PatternRecognizer`. It suits a closed vocabulary — a fixed set of internal project codenames or known sensitive fund names — where enumerating values is easier and more precise than writing a pattern.

Q: How do you register a custom recognizer so the analyzer starts emitting its entity?
A: Build the recognizer, then call `registry.add_recognizer(my_recognizer)` on the `RecognizerRegistry` before constructing or while configuring the `AnalyzerEngine`, or pass a pre-loaded registry into `AnalyzerEngine(registry=...)`. After that, `analyze()` will return your new `entity_type` whenever the pattern or model fires and the score clears the threshold.

Q: What does `supported_entities()` on a recognizer return, and why query it?
A: It returns the list of entity-type strings that recognizer can produce, e.g. `["IBAN_CODE"]` or several for a multi-entity recognizer. Querying it (or `registry.get_recognizers()`) is how you confirm at runtime that your custom `LU_PHONE_NUMBER` recognizer is actually loaded before blaming the pattern for a miss.

Q: How do you list every recognizer currently loaded in an analyzer to debug a missing entity?
A: Call `analyzer.registry.get_recognizers(language="en", all_fields=True)` (or inspect `registry.recognizers`) to enumerate the loaded recognizers for that language. If your custom recognizer is absent from that list, the bug is registration or language mismatch, not the regex — check this before touching the pattern.

Q: A custom recognizer works for `language="en"` but returns nothing for `language="fr"`. First thing to check?
A: Check the recognizer's `supported_language` — a `PatternRecognizer` is bound to one language and is only consulted when `analyze()` is called with that matching `language` value. You must register a separate instance with `supported_language="fr"` (and French context words) for it to fire on French rows.

Q: How do you supply custom context words to a `PatternRecognizer` so the enhancer boosts its hits?
A: Pass `context=["iban", "compte", "account", "bank"]` to the `PatternRecognizer` constructor; the `LemmaContextAwareEnhancer` then lemmatizes nearby tokens and boosts the score when any context lemma appears close to a match. The context list lives on the recognizer, not the engine, so each entity carries its own domain vocabulary.

Q: What is the practical effect of `return_decision_process=True` on the result objects?
A: Each `RecognizerResult` gains a populated `analysis_explanation` (an `AnalysisExplanation`) describing the recognizer, pattern name, original score, and any context improvement. You read `result.analysis_explanation.textual_explanation` to see exactly why a span scored what it did, turning recall debugging into reading rather than guessing.

Q: Which fields of `AnalysisExplanation` tell you whether a context boost was applied?
A: `original_score` versus `score` shows the delta, `score_context_improvement` gives the boost amount, and `supportive_context_word` names the word that triggered it. If `supportive_context_word` is empty on a borderline IBAN you expected to pass, your context list is missing the word actually present in the text.

Q: How do you detect and resolve two recognizers returning overlapping spans programmatically?
A: `RecognizerResult` exposes geometric helpers — `intersects`, `contains`, `contained_in`, and `has_conflict` — that let you test span relationships, and the analyzer's conflict resolution keeps the higher-scoring or longer match. If a generic `PHONE_NUMBER` and a custom `LU_PHONE_NUMBER` both fire, `has_conflict` flags the overlap so you decide which entity type the catalog should record.

Q: How would you suppress a specific recognizer entirely, such as turning off `US_SSN` for an EU-only dataset?
A: Call `registry.remove_recognizer("UsSsnRecognizer")` (the class name) or build the registry without it, so the analyzer never runs that recognizer or emits `US_SSN`. This avoids spurious US-identifier tags on Luxembourg data and slightly speeds the scan by skipping an irrelevant recognizer.

Q: What is the difference between restricting detection with the `entities` argument and removing a recognizer from the registry?
A: The `entities` argument filters per call — the recognizer is still loaded but only consulted when its type is requested — whereas removing it from the registry unloads it for every call. Use `entities` for a one-off narrow scan and registry removal when an entity type is permanently irrelevant to your platform.

Q: A new spaCy `transformers` model gives better `PERSON` recall but the scan is now too slow. What knobs trade recall for speed?
A: Swap the `NlpEngine` back to a smaller spaCy model (`en_core_web_sm` vs `en_core_web_trf`), raise `batch_size` and `n_process` on `analyze_iterator`, and restrict `entities` to the types each feed needs. The transformer model maximises recall on awkward names but the small pipeline is far faster, so pick per feed based on its risk profile.

Q: Why must the exact spaCy model be pinned and pre-downloaded in the scan container rather than fetched at runtime?
A: A runtime `spacy download` can pull a different model version than you tested against, silently shifting NER behaviour, or fail in an air-gapped fund environment with no internet egress. Pinning the model version and baking it into the image makes the scan reproducible and immune to a network outage breaking compliance scanning.

Q: How does Presidio's `NlpEngineProvider` let you configure the NLP backend from a YAML file?
A: `NlpEngineProvider` reads a YAML config specifying `nlp_engine_name` (`spacy`, `stanza`, or `transformers`) and the per-language models, then builds the `NlpEngine` you pass to `AnalyzerEngine`. Keeping the model choice in config rather than code means ops can switch a French model without a code release.

Q: What does it mean for a recognizer to be a `LocalRecognizer`, and why does it matter for the registry?
A: A `LocalRecognizer` runs in-process using the shared `NlpArtifacts` the engine already computed, as opposed to a remote recognizer that calls out to an external service. Most predefined recognizers are local, which keeps them fast; a remote recognizer adds network latency and a failure mode the registry must tolerate.

Q: How do you scan a tabular dataset column-by-column rather than cell-by-cell with `presidio-structured`?
A: Use `presidio-structured`'s `StructuredEngine` with a `TabularAnalysisBuilder` (e.g. `PandasAnalysisBuilder`) to produce a per-column `StructuredAnalysis` mapping each column to a detected entity type, then apply anonymization uniformly to whole columns. This decides "this column is IBANs" once from sampled values instead of re-detecting on every cell.

Q: What is a `StructuredAnalysis` object and how is it reused?
A: A `StructuredAnalysis` records the mapping from column name to entity type that the builder inferred from the data. You can inspect, edit, or persist it and then feed it to the structured anonymizer, so the same column-classification decision drives both your catalog tags and the masking, with no re-scan.

Q: How would you redact PII that appears as pixels in a scanned KYC document image?
A: Use `presidio-image-redactor`'s `ImageRedactorEngine`, which runs Tesseract OCR to extract text, feeds it to the `AnalyzerEngine`, and draws filled boxes over detected PII regions. A scanned passport or proof-of-address in the document store carries PII no text scanner sees, so image redaction closes that gap.

Q: What is the default fill colour of `ImageRedactorEngine.redact`, and how do you change it?
A: The default `fill` is black, accepted as an int or an RGB tuple `(0, 0, 0)`, and you override it by passing `fill=(255, 0, 0)` or similar to `redact`. For DICOM medical images `DicomImageRedactorEngine` instead accepts `"contrast"` or `"background"` so the boxes either stand out or blend with the page.

Q: Why is OCR-based image redaction more error-prone than text redaction, and what does that imply for recall?
A: OCR can misread characters, merge or split tokens, and miss low-contrast or rotated text, so PII the eye reads clearly may never reach the analyzer, producing false negatives the text path would not have. For high-risk document stores you compensate with higher-resolution scans, OCR-confidence checks, and manual review rather than trusting a single automated pass.

Q: A `mask` operator call raises `InvalidParamError`. What are the likely causes?
A: The anonymizer validates operator params, so `InvalidParamError` typically means `chars_to_mask` is missing or non-integer, `masking_char` is not a single character, or `from_end` is not a boolean. Check that your `OperatorConfig("mask", {...})` dict has exactly `masking_char`, `chars_to_mask`, and `from_end` with the right types.

Q: If `chars_to_mask` exceeds the length of the detected value, what happens with the `mask` operator?
A: The operator masks the whole value — it cannot mask more characters than exist, so a `chars_to_mask` larger than the span simply replaces every character with `masking_char`. This is safe for "hide everything but the last four" only if you compute or cap the count against shorter-than-expected values.

Q: How do you apply one default operator to every entity not explicitly configured in the anonymizer?
A: Include the key `"DEFAULT"` in the operators dict, e.g. `{"IBAN_CODE": OperatorConfig("hash"), "DEFAULT": OperatorConfig("replace")}`, so any detected entity without its own config falls back to the default. This guarantees a newly detected entity type is still anonymized rather than passed through raw because you forgot to configure it.

Q: How can the `replace` operator inject a realistic fake value instead of a static `<PERSON>` token?
A: Use the `lambda`-style callable form of the `replace`/`custom` operator (an `OperatorConfig("custom", {"lambda": fn})`) wired to a generator like Faker, so each `PERSON` becomes a different plausible name. This keeps text natural for testing or ML training while removing real identities, unlike a constant token that flattens every name to the same string.

Q: What does the anonymizer's per-result metadata (`OperatorResult`) tell you, and why log it?
A: Each `OperatorResult` records the entity type, the new start/end offsets, the operator applied, and the resulting text fragment. Logging these gives you an audit trail of exactly what was changed and how, which an auditor reviewing your de-identification process for a NAV extract will expect to see.

Q: How does `decrypt` know which key to use, and what breaks if it is wrong?
A: `DeanonymizeEngine` applies the `decrypt` operator with the `key` you supply in its `OperatorConfig`, which must be the same key (and AES key length) used by `encrypt`. A wrong or rotated key yields garbage or an error, so key versioning matters — you must keep the encrypting key recoverable to ever round-trip an investor account number.

Q: Why is AES key length relevant when using the `encrypt` operator?
A: Presidio's `encrypt` uses AES, which requires a key of valid length (16, 24, or 32 bytes for AES-128/192/256), so an arbitrary-length string passed as `key` will raise an error. You generate a correctly sized key and store it in a secrets manager, never hard-code it, because it can decrypt every protected IBAN.

Q: For a continuous in-motion scan on a Kafka-style stream, where should the analyzer run relative to the bronze sink?
A: Put the analyzer in the consumer or stream-processor stage before the write to bronze, so each record is detected and either tagged or anonymized prior to landing. Running it after the sink reintroduces the latency window where an untagged national ID is already queryable, defeating the in-motion advantage.

Q: How do you keep per-record latency low when the stream processor must call Presidio synchronously?
A: Reuse one long-lived `AnalyzerEngine` per worker (loading the model once), restrict `entities` to the few the feed can contain, prefer regex-plus-checksum recognizers, and micro-batch records through `analyze_iterator`. Constructing a fresh engine or loading the spaCy model per message is the classic latency killer in a streaming PII filter.

Q: An at-rest scan times out on a multi-terabyte bronze table. How do you make it tractable?
A: Scan a statistically representative sample of rows per column for classification, parallelise across partitions with `n_process`/separate workers, and skip columns already confidently tagged from a prior run. Full per-row scanning of every historical row is rarely necessary to classify a column, though high-risk feeds may still warrant it.

Q: Why does sampling for column classification risk missing a rare stray PII, and how do you mitigate it?
A: A column that is mostly clean but holds one pasted IBAN in a million rows may never appear in a small sample, so the column stays untagged. Mitigation is risk-based: full-scan or larger samples for free-text and unknown feeds, and trigger a re-scan on schema or volume changes so a newly introduced stray is eventually caught.

Q: How would you wire Presidio findings into OpenMetadata tags programmatically?
A: Map each detected `entity_type` to a tag FQN like `PII.Sensitive`, then call the OpenMetadata REST API (or the Python SDK's `OpenMetadata` client `patch`/`add_tag` on the column entity) to attach the tag to the specific `table.column`. Driving it through the API rather than the UI is what makes the discover-tag loop automatic and repeatable.

Q: What identifies the exact column to tag when pushing a finding to OpenMetadata?
A: The column's fully qualified name (FQN), of the form `service.database.schema.table.column`, uniquely addresses the asset in OpenMetadata. Your scan must carry that FQN alongside each finding, because a bare entity type with no location cannot be turned into a tag the policy layer can act on.

Q: How do OpenMetadata classification tags relate to PII propagation through lineage?
A: A `PII.Sensitive` tag applied to a source column can be propagated along column-level lineage to derived columns, so a silver table built from a tagged bronze IBAN inherits the sensitivity. This matters because masking that only covers bronze, while silver and gold expose the same value untagged, is a common governance hole.

Q: Why is column-level lineage important specifically for PII coverage in a lake?
A: PII flows downstream into joins, aggregates, and extracts, and a national ID masked in bronze can resurface verbatim in a gold report if the tag did not travel with it. Column-level lineage lets you propagate the classification (and therefore the masking policy) to every derived column, closing the leak that table-level lineage alone would miss.

Q: A column is tagged `PII.Sensitive` but analysts still see raw values. Where do you look first?
A: First confirm the masking/access policy is actually bound to that tag and that the analyst's role is in scope of the masking rule, not the exempt unmasked role. The tag is only metadata; if no policy consumes it or the role mapping is wrong, the tag changes nothing — verify the policy-to-tag binding before re-examining detection.

Q: How would you prove the discover-tag-mask loop runs "without a human noticing" as the Done-when requires?
A: Land a synthetic record with a known planted national ID in a fresh bronze column, let the scheduled or streaming scan tag it, then read the column as an unprivileged role and assert the value is masked — all with no manual step in between. Automating that assertion as a test gives you a repeatable proof the auditor's question is answered by design.

Q: How do you regression-test a recognizer so a future change cannot silently drop recall?
A: Keep a labelled fixtures file of texts with known PII spans and a unit test that runs the analyzer and asserts each expected entity is detected above threshold, failing CI if recall regresses. This turns the manual before/after measurement into a guardrail, so a "precision fix" that quietly stops catching grouped IBANs breaks the build.

Q: Why should you measure recognizer metrics on data that matches your production language mix, not generic English text?
A: A recognizer that scores 0.95 precision on English benchmarks can collapse on Luxembourg trilingual notes where French context words and German name forms appear, so generic metrics overstate real performance. Your evaluation set must mirror the actual English/French/German mix of client notes for the reported precision change to mean anything operationally.

Q: When tuning the Luxembourg IBAN recognizer, why might removing the mod-97 checksum increase recall but be the wrong move?
A: Dropping the checksum makes the recognizer accept anything IBAN-shaped, so it catches malformed real IBANs (recall up) but floods the catalog with false positives from any long alphanumeric string (precision crashes). The right recall fix is broadening the regex to tolerate spacing and adding context words while keeping the checksum, so validity still gates high scores.

Q: How does Presidio's `score` interact with multiple patterns inside one `PatternRecognizer`?
A: Each `Pattern` in the recognizer carries its own `score`, and the recognizer assigns the score of whichever pattern matched, so you give a tight, validated pattern a high score and a loose fallback pattern a low one. The loose pattern then only survives when context enhancement boosts it, giving you recall without the precision cost of a high base score everywhere.

Q: What is the safest way to handle a recognizer with high recall but unacceptable false positives in a regulated catalog?
A: Detect at high recall but route low-confidence hits to a `PII.Suspected` review tag instead of auto-applying the enforced `PII.Sensitive` tag, separating discovery from enforcement. You miss nothing in discovery while only the high-confidence findings trigger automatic masking, and a steward clears or promotes the rest, which keeps both recall and catalog trust high.

Q: Why does the analyzer accept precomputed `nlp_artifacts`, and when would you pass them?
A: The `nlp_artifacts` parameter lets you hand `analyze()` the tokens, lemmas, and entities the `NlpEngine` already produced, so the engine skips re-running the model. You pass them when one piece of text is analyzed by multiple recognizer passes or when an upstream stage already ran NLP, avoiding the dominant cost — the model inference — twice on the same record.
