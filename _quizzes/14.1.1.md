# Phase 8 · 14.1.1 + 14.1.3 Business intelligence (Superset hands-on, Power BI fluency) — 150 self-test questions

Q: What is Apache Superset in one sentence?
A: Superset is an open-source (Apache-licensed) business-intelligence web application for data exploration and dashboarding. It lets users connect to SQL databases, define datasets, build charts, and assemble them into dashboards, all through a browser UI plus a Python backend. For an architect it is the FOSS reference point against which you read commercial tools like Power BI and Tableau.

Q: Why does the architect care about the BI tier even though they "don't build dashboards for a living"?
A: BI is where the platform meets its judges — the fund board sees numbers through dashboards, not through your pipelines, so the BI tier's governance and tooling choices directly shape trust in the whole estate. The architect designs the tier (tool selection, governance, self-service boundaries) rather than authoring individual charts. Get the governance wrong and you inherit the "300 unowned dashboards" estate with contradictory NAV figures and nobody accountable.

Q: Name the four core object types in Superset's data model, from connection to presentation.
A: The chain is `Database` → `Dataset` → `Chart` (also called a slice) → `Dashboard`. A `Database` is a connection to a SQL source; a `Dataset` is a queryable table or saved query on that source; a `Chart` is one visualization built on a dataset; a `Dashboard` arranges many charts on a single page. Understanding this hierarchy is essential before reasoning about where governance and security attach.

Q: What is the difference between a physical dataset and a virtual dataset in Superset?
A: A physical dataset maps directly to one existing table or view in the source database, while a virtual dataset is defined by a saved SQL `SELECT` query that Superset treats as if it were a table. Virtual datasets let you reshape or join data without changing the warehouse, but they also let metric logic leak into the BI tool — which is exactly what the semantic-layer discipline tries to prevent. Prefer thin physical datasets over thick virtual SQL when a semantic layer exists upstream.

Q: In Superset, what is a "slice"?
A: A slice is Superset's older internal term for a single chart — one visualization (bar, line, table, big-number, etc.) built on one dataset. You will see "slice" in URLs, the API, and some menus even though the UI mostly says "chart". Knowing the synonym helps when reading the REST API or older documentation.

Q: What does it mean that Superset has a "semantic-ish" layer rather than a full semantic layer?
A: Superset lets you define calculated columns, metrics (named aggregations like `SUM(net_assets)`), and virtual datasets on top of a database, which gives a light metric-definition capability. But it is per-dataset and tool-local — there is no shared, versioned, governed metric store across tools, so the same metric can be redefined inconsistently in another BI tool. That gap is precisely why an external semantic layer such as Cube is preferred for canonical metrics.

Q: What is a Superset metric versus a calculated column?
A: A calculated column is a row-level expression evaluated before aggregation (e.g. `price * quantity`), while a metric is an aggregate expression that produces a single value over a group (e.g. `SUM(price * quantity)`). Metrics appear in the chart's "Metrics" control and drive the y-axis or value; calculated columns appear as dimensions or feed into metrics. Confusing the two leads to charts that either don't aggregate or aggregate the wrong thing.

Q: How do you typically add a new data source so it appears in Superset?
A: You add a `Database` connection (via Settings → Database Connections, supplying a SQLAlchemy URI), then create one or more `Datasets` against tables or saved queries in that database. Only after a dataset exists can you build charts on it. The SQLAlchemy URI (e.g. `postgresql://user:pass@host:5432/db`) is the standard connection format Superset uses under the hood.

Q: What is Row-Level Security (RLS) in Superset?
A: RLS automatically injects a `WHERE` clause into every query Superset runs against a dataset, restricting which rows a user can see based on their roles. It is configured as rules (Settings → Row Level Security) that bind a clause like `mgmt_company = 'ACME'` to one or more roles and datasets. This lets one shared dashboard show each management company only its own funds without building separate copies.

Q: In Superset RLS, what is the difference between a "regular" filter and a "base" filter?
A: A regular filter adds its `WHERE` clause only for users who belong to the role(s) named in the rule, so it grants scoped visibility to those roles. A base filter adds its clause for everyone EXCEPT the named roles, so it is the default restriction applied to users not otherwise covered. Use regular filters to scope a management-company role to its funds, and a base filter to ensure unscoped users see nothing (or only public rows).

Q: How are multiple RLS filters combined within and across filter groups in Superset?
A: Filters that share the same group key are combined with `OR` (any one matching is enough), while filters in different groups — or with no group key — are combined with `AND` (all must match). This lets you express "see fund family A or fund family B" within a group, while still `AND`-ing an unrelated region restriction from another group. Getting the group keys wrong silently widens or narrows visibility, so always test with real user accounts.

Q: A user reports they can see another management company's funds on a shared Superset dashboard. What is the first thing to check?
A: Check the Row Level Security rules: confirm a rule with the correct `WHERE` clause is bound to that user's role and to the specific dataset the charts use, and that it is a regular filter (not accidentally a base filter excluding their role). Then verify the user actually holds that role and not an over-broad role like `Admin` or `sql_lab`, which can bypass intended scoping. Finally re-test by logging in as that user, because RLS is applied per query at runtime.

Q: Why is testing RLS with two real test users (not just an admin) important?
A: Admins and roles with broad SQL access can bypass RLS entirely, so an admin "looks fine" while a scoped business user is mis-restricted. Logging in as two distinct test users — say one bound to management company A and one to B — proves the `WHERE` clause actually differs per user and that neither leaks the other's rows. This two-user verification is an explicit "Done-when" gate for this lesson.

Q: What does Superset dashboard embedding allow, and what is the typical mechanism?
A: Embedding lets you surface a Superset dashboard inside another web application (e.g. a fund-administration portal) rather than sending users to the Superset UI. Superset provides an embedded SDK that authenticates via short-lived guest tokens minted by your backend, and you can pass RLS rules in that token so the embedded view is already scoped to the viewer. This keeps a single dashboard definition while delivering it tenant-by-tenant inside your own product.

Q: What is a Superset "guest token" and why does it matter for multi-tenant embedding?
A: A guest token is a short-lived signed token your backend issues to authorize an embedded dashboard view for a specific user, and it can carry RLS rules that constrain the rows returned. It matters because it lets you enforce per-tenant data isolation at embed time without creating one dashboard per client. The token is minted server-side using Superset's secret, so the client never sees credentials.

Q: Why is it dangerous to expose a Superset SQL Lab to self-service business users?
A: SQL Lab lets users write arbitrary SQL against connected databases, which can bypass curated datasets, ignore metric definitions, and — depending on database permissions — read rows that RLS on datasets does not cover. For governed self-service you typically restrict SQL Lab to a small data-engineering role and steer business users to certified datasets and charts. The risk is uncontrolled, unowned queries becoming "shadow metrics".

Q: What is "governed self-service" in a BI context?
A: Governed self-service means business users can explore and build content themselves, but only within guardrails: certified datasets as the trusted starting point, clear ownership, a sandbox-versus-production split, and rules for what may be published where. It balances the speed of self-service against the chaos of everyone inventing their own numbers. The architect's job is to design those guardrails, not to lock everyone out.

Q: What is a "certified dataset" and what problem does it solve?
A: A certified dataset is one an accountable owner has reviewed and endorsed as the trusted, correct source for a domain (e.g. official NAV), usually marked with a visible certification badge. It solves the problem of users not knowing which of several similar datasets to trust, steering them to the blessed one. Certification is a workflow and an ownership claim, not just a label.

Q: Describe the "300 unowned dashboards" failure mode and how it arises.
A: It arises when self-service has no governance: anyone can clone datasets, redefine metrics, and publish dashboards with no owner, no certification, and no decommissioning rule. Over time the estate accumulates hundreds of overlapping dashboards showing contradictory numbers, and when the board asks "which AUM figure is right?" nobody is accountable for any of them. The fix is certification, explicit ownership, and a sunset rule for unowned content.

Q: What is a sandbox-versus-production content policy in BI governance?
A: It is an explicit boundary separating a space where users freely experiment (sandbox, personal or team area) from a curated space holding trusted, certified, board-facing content (production). Promotion from sandbox to production requires meeting standards — an owner, certified source datasets, and review. Without this split, half-finished experiments end up in front of the fund board.

Q: What belongs in a BI certification workflow?
A: A certification workflow defines who can request certification, who reviews and approves it (a named data/domain owner), the criteria a dataset or dashboard must meet (correct sources, documented metrics, RLS where needed), and how certification is revoked when something changes. It turns "this dashboard is trustworthy" from folklore into an auditable decision. It is the primary control preventing dashboard sprawl.

Q: Why must every piece of certified BI content have a named owner?
A: Without a named owner, no one is accountable for keeping the numbers correct, answering questions, or retiring the content when it goes stale — which is exactly how the 300-dashboard estate forms. A named owner gives the fund board someone to hold responsible for a figure and gives the platform a contact for change impact. Ownership is the load-bearing governance primitive, more than tooling.

Q: What is a decommissioning (sunset) rule for BI content and why include one?
A: A decommissioning rule states the criteria and process for retiring dashboards — for example, anything unowned, unviewed for N months, or built on a deprecated dataset gets archived then deleted. It is included because content accumulates faster than it is cleaned up, and without a sunset rule the estate only ever grows. Funded, scheduled decommissioning is what stops sprawl from returning after you clean it up once.

Q: Why should BI tools consume metrics from a semantic layer such as Cube rather than re-modelling them?
A: If every BI tool redefines `NAV per share` or `net flows` in its own virtual datasets and calculated metrics, the definitions drift and the board sees contradictory numbers from different tools. A semantic layer defines each metric once, versioned and governed, and serves it to all consumers, so the BI tool becomes a thin presentation layer. "Define the metric once, present it many ways" is the core discipline this lesson enforces.

Q: How does Apache Superset connect to a Cube semantic layer?
A: Superset connects to Cube through Cube's SQL API, treating Cube as if it were a Postgres database — you add a database connection in Superset pointing at Cube's SQL API endpoint and then build datasets on the cubes/views Cube exposes. Cube's SQL API is wire-compatible with the Postgres protocol, so Superset's standard Postgres driver works. This means Superset queries pre-defined metrics instead of writing its own metric SQL.

Q: What wire protocol does Cube's SQL API emulate, and is there a real Postgres engine inside Cube?
A: The SQL API is wire-compatible with the Postgres protocol, so BI tools connect to Cube as though it were a Postgres database. But there is no real Postgres engine inside Cube; it uses Apache DataFusion plus the `egg` e-graph rewriting library to parse incoming SQL, match it against defined cubes, and plan the query. This is why some arbitrary SQL works and some does not — Cube only answers queries it can map to its model.

Q: How does Power BI connect to a Cube semantic layer, and what are the two options?
A: Power BI can connect either via Cube's SQL API (treating Cube as a Postgres database) or via Cube's DAX API for native connectivity. The SQL-API path works but is more limited; the DAX API lets Power BI connect as if to an SSAS / Azure Analysis Services instance, giving native Tabular-model behaviour. For an Azure-shop Power BI deployment, the DAX API is the more capable integration.

Q: What is the advantage of Cube's DAX API for Power BI specifically?
A: The DAX API makes Cube look like a Microsoft Analysis Services (SSAS / Azure Analysis Services) tabular model, so Power BI connects natively and users get the full DAX/measure experience against governed Cube metrics. It avoids the limitations of going through the generic SQL/Postgres path and keeps metric definitions in Cube rather than re-implemented as Power BI DAX measures. This is the cleanest way to keep "no re-modelling in the BI tool" in an Azure estate.

Q: What is Cube's "Semantic Layer Sync" with Superset?
A: Semantic Layer Sync pushes Cube's data model into the BI tool so that cubes, views, and metrics appear there automatically as managed datasets, rather than being hand-built in the BI tool. For Superset/Preset it creates the datasets and keeps them aligned with Cube, reducing drift between the semantic layer and what users see. It operationalizes "model once in Cube, consume everywhere" by syncing definitions instead of copying them.

Q: In the lesson's "Do" task, why must the fund-board dashboard have zero local SQL definitions?
A: Because every figure (NAV trend, net flows, DQ status) must come from semantic-layer metrics in Cube, so the board sees the same numbers any other consumer would and there is a single governed source of truth. Zero local SQL means no virtual datasets or calculated metrics redefining the math inside Superset. Verifying this is an explicit "Done-when" check for the lesson.

Q: For the fund-board dashboard, which three metric families does the lesson call for and where should they come from?
A: NAV trend, net flows (subscriptions minus redemptions), and data-quality status — all sourced from Cube semantic-layer metrics, not local Superset SQL. Pulling them from the semantic layer guarantees the board's NAV matches operational and regulatory views of the same number. The dashboard is then just a presentation of governed metrics.

Q: How would you apply RLS in Superset "by management company" for a multi-manco fund dashboard?
A: Create a role per management company (or a single role parameterized via the embed/guest-token context), then add a regular RLS rule binding a clause like `mgmt_company_id = <X>` to that role on the fund dataset. Add a base filter so users without a manco role see no rows. Then verify with two test users — one for manco A, one for manco B — that each sees only their own funds.

Q: Power BI per-user licensing: what are Power BI Pro and Premium Per User, and their list prices?
A: Power BI Pro is the standard per-user license for authoring and sharing in the Power BI service, listed at about $14 per user per month; Premium Per User (PPU) adds Premium-grade features (larger models, more frequent refresh, paginated reports) at about $24 per user per month. Both prices took effect on 1 April 2025. They are licensed per individual, which makes total cost scale linearly with user count.

Q: What is Fabric capacity (F-SKU) licensing and how does it differ from per-user licensing?
A: F-SKUs are capacity-based licenses that buy dedicated compute (e.g. F2, F8, F64) for Power BI and other Microsoft Fabric workloads, paid for the capacity regardless of how many users hit it. This contrasts with per-user (Pro/PPU) where you pay per individual. Capacity is the right model when you have many viewers, because cost decouples from headcount above a threshold.

Q: What is special about the Fabric F64 capacity tier with respect to viewers?
A: At F64 and above, users with only a free Power BI license and a viewer role can view content hosted on that capacity — they no longer each need a paid Pro license. This dramatically changes total cost of ownership for large read-heavy deployments, because viewers stop being a per-seat cost. Below F64 (or on per-user licensing), every viewer of Pro content needs their own Pro license.

Q: How would you reason about licensing a 300-user Power BI deployment?
A: First split the 300 into authors versus pure viewers. If most are viewers, an F64 (or larger) capacity lets free-licensed viewers consume content, so you pay for the capacity plus Pro/PPU only for the authors — often far cheaper than 300 × Pro. If almost everyone authors or you need Premium features for under ~250 users, per-user PPU may win; the crossover is exactly what the architect must be able to size.

Q: Roughly where is the crossover between Premium Per User and capacity licensing?
A: PPU tends to be more cost-effective when you need Premium features for a relatively small population (commonly cited as fewer than ~250 users), because you pay per head. Above that, or when you have a large viewer base, F-SKU capacity — especially F64 where viewers can be free — usually wins. The exact break-even depends on the author/viewer mix and which Premium features you actually need.

Q: What happened to Power BI Premium "P-SKU" capacities?
A: Microsoft retired the standalone Power BI Premium P-SKUs in favour of Microsoft Fabric F-SKU capacities: new customers could not purchase P-SKUs after 1 July 2024, and existing customers transition to F-SKUs at renewal. So in 2025-2026 capacity sizing is expressed in Fabric F-SKUs (F2…F2048), not the old P1/P2/P3 names. Architects must read pricing in F-SKU terms.

Q: Why is Power BI licensing called "the real selection driver in Azure shops"?
A: In an Azure-centric organization the technical fit of Power BI is rarely in doubt; the decision turns on cost, which is dominated by the licensing model — per-user Pro/PPU versus Fabric capacity. The author/viewer ratio and the F64 free-viewer threshold can swing total cost by an order of magnitude. So the architect's licensing fluency, not feature comparison, drives the recommendation.

Q: Why must a data architect be able to "read the vendor market" for BI even when committed to one tool?
A: Because organizations inherit mixed estates, acquisitions bring other tools, and procurement asks "why not X?" — the architect must explain who each vendor sells to and where it wins. A credible market map (Power BI, Tableau, Looker, Qlik, Superset, Metabase) lets you justify a choice and migrate sensibly rather than defaulting. It also guards against being captured by a single vendor's framing.

Q: In one line each, who is the core buyer for Tableau, Looker, and Power BI?
A: Tableau targets analysts and business users wanting best-in-class visual exploration and storytelling, often regardless of cloud. Looker (Google Cloud) targets engineering-minded teams wanting a governed, code-defined metrics model (LookML) tightly coupled to the cloud warehouse. Power BI targets Microsoft/Azure shops wanting tight Office/Fabric integration and aggressive per-capacity pricing. Each "wins" in a different organizational shape.

Q: What distinguishes Looker's modelling approach from Power BI's and Superset's?
A: Looker centres on LookML, a version-controlled, code-defined modelling layer where metrics and joins are declared once and reused, pushing it toward semantic-layer discipline by design. Power BI models metrics in DAX inside each dataset, and Superset defines metrics per-dataset in its UI; both are easier to fragment. Looker's git-backed model is closer to "define once" but is tied to the Google Cloud ecosystem.

Q: Where does Qlik typically position itself relative to the others?
A: Qlik markets its associative engine (in-memory associative model letting users explore relationships in any direction, not just predefined drill paths) as a differentiator, targeting organizations that value flexible, exploratory analysis over rigid hierarchies. It competes head-to-head with Tableau and Power BI in the enterprise. The associative model is its main selling point in vendor positioning reads.

Q: Where do Metabase and Superset sit in the market map?
A: Both are open-source / low-cost BI tools aimed at teams wanting self-hosted dashboards without per-seat enterprise licensing; Metabase emphasizes ease and quick question-asking, Superset emphasizes extensibility and SQL power. They are the FOSS counterweight to Power BI/Tableau/Looker and are common in cost-sensitive or engineering-led shops. The architect uses them as the "do we even need a commercial tool?" baseline.

Q: What does it mean that "the architect doesn't build dashboards but designs the BI tier"?
A: It means the deliverable is the design of how BI is done — which tools, what licensing model, what governance (certification, ownership, sandbox/prod), and where self-service stops — not a portfolio of charts. The architect sets the rules and the platform so that hundreds of authors can produce trustworthy content. Authoring individual dashboards is a consumer activity within that frame.

Q: What is the BI governance one-pager meant to answer?
A: It must answer "who may publish what, where" — including the certification workflow, ownership rules, the sandbox-to-production promotion path, and a decommissioning rule for unowned content. It is a one-page policy the architect can present to stakeholders so the rules are explicit and auditable. Presenting it (with the sandbox→certified path) is a "Done-when" gate.

Q: Why does the lesson insist the governance pager include "the path from sandbox to certified"?
A: Because the most common governance gap is an undefined promotion path: content either never gets certified (so nothing is trusted) or gets pushed to production without review (so untrusted content reaches the board). An explicit path — owner assigned, certified source datasets, review, badge — makes promotion a controlled gate. It closes the loop between free experimentation and trustworthy production content.

Q: A fund board complains that two dashboards show different "total AUM". As architect, what is your structural fix?
A: Define AUM once as a governed metric in the semantic layer (Cube) and require both dashboards to consume that metric, eliminating local re-definitions; then certify the canonical dashboard and assign a named owner. The contradictory dashboard is either repointed to the metric or decommissioned under the sunset rule. The fix is upstream metric ownership, not editing two charts to match.

Q: Why is "never re-model metrics inside the BI tool" a hard rule rather than a guideline?
A: Because each BI-tool re-model is a place where a metric definition can silently diverge from the governed one, and in a regulated fund context divergent NAV/AUM figures are a control failure, not a cosmetic issue. Making it a hard rule means the BI tier stays a thin consumer of the semantic layer, so there is exactly one definition to audit. Guidelines get ignored under deadline pressure; the rule is enforced by sourcing dashboards only from Cube.

Q: How does a semantic layer help with regulatory consistency (e.g. NAV across reports)?
A: It guarantees that NAV, net flows, and AUM are computed by one definition consumed identically by operational dashboards, regulatory reports, and the board view, so the regulator and the board see the same number. Auditors can review one metric definition instead of dozens of BI-tool re-implementations. This single-definition property is what makes the BI tier defensible under regimes like UCITS, AIFMD, and DORA.

Q: In a Luxembourg fund-administration context, why is BI sprawl a compliance risk and not just an annoyance?
A: Because contradictory NAV or AUM figures presented to a fund board or a regulator undermine the integrity of official reporting, and regulated administrators must be able to show controlled, owned, reproducible numbers. Hundreds of unowned dashboards mean you cannot evidence who produced a figure or how — a governance and audit failure. So the certification/ownership/decommissioning discipline is a regulatory control, not housekeeping.

Q: What is a sensible self-service boundary for transfer-agency (TA) data in Superset?
A: Expose certified datasets/metrics (e.g. subscriptions, redemptions, holder counts) sourced from the semantic layer, enforce RLS by management company and possibly by fund, and keep SQL Lab off for business users so they cannot bypass scoping. Authors build within these certified datasets; raw TA tables are not directly explorable. This lets TA teams self-serve flows analysis without risking cross-client data leakage.

Q: How would RLS interact with an ISIN-scoped view for a sub-advisor who may only see specific share classes?
A: You bind a regular RLS rule to the sub-advisor's role with a clause like `isin IN (<permitted list>)` (or join to an entitlement table) on the relevant dataset, plus a base filter denying everyone else. Each query then silently filters to the permitted ISINs, so a shared share-class dashboard shows the sub-advisor only their classes. Verify with a test user that an unlisted ISIN is invisible.

Q: Why is it risky to let analysts build a "net flows" metric independently in each dashboard?
A: Because "net flows" has subtle definitional choices (timing, pending trades, switches counted once or twice, currency), and independent definitions will disagree, producing different flow numbers for the same fund. Defined once in the semantic layer, every flows chart agrees and the definition is auditable. Independent re-definition is the seed of board-level contradictions.

Q: What does "BI is where your platform meets its judges" mean for design priorities?
A: It means correctness and trust at the BI tier dominate cleverness, because stakeholders judge the entire data platform by the numbers they see on dashboards. So the architect prioritizes governed metrics, ownership, and certification over flashy visuals or maximal self-service freedom. A technically excellent pipeline feeding contradictory dashboards is judged a failure.

Q: What is the difference between Power BI Desktop and the Power BI Service?
A: Power BI Desktop is the free Windows authoring application where you build data models and reports; the Power BI Service is the cloud platform where reports are published, shared, secured, and consumed. Licensing (Pro/PPU/capacity) governs the Service, not Desktop authoring itself. Knowing the split clarifies that "licensing" is about sharing/consumption, not about being allowed to build.

Q: What is a Power BI "dataset" (semantic model) versus a "report"?
A: A Power BI dataset — now branded a semantic model — is the data model with tables, relationships, and DAX measures; a report is the collection of visuals built on top of one semantic model. Multiple reports can share one semantic model, which is the recommended way to keep metric definitions consistent. This mirrors Superset's dataset→chart split and is where DAX re-modelling risk lives.

Q: Why might using Cube's DAX API beat building DAX measures directly in a Power BI semantic model?
A: Building measures in the Power BI model re-implements metrics inside the BI tool, reintroducing the drift the semantic layer exists to prevent; the DAX API instead serves Cube's governed measures to Power BI natively. So you keep one definition in Cube while still giving Power BI users a full DAX experience. It is the Azure-shop way to honour "no re-modelling in the BI tool".

Q: How does row-level security work in Power BI, at a high level?
A: Power BI RLS defines roles with DAX filter expressions on tables in the semantic model, and users are mapped to roles so each sees only permitted rows. Dynamic RLS uses functions like `USERPRINCIPALNAME()` to filter by the logged-in identity against an entitlement table. Conceptually it parallels Superset RLS, but the filter is expressed in DAX inside the model.

Q: What is the architect's concern if RLS is defined in the BI tool rather than upstream?
A: BI-tool RLS protects only that tool's queries, so the same data accessed through another tool, an export, or the warehouse directly may not be filtered, leaving entitlement gaps. For strong, consistent entitlement you push row scoping into the semantic layer or the data platform where every consumer inherits it. BI-tool RLS is then a convenience layer, not the sole control.

Q: When is it acceptable to define a quick metric directly in Superset rather than in Cube?
A: For genuinely throwaway, sandbox-only exploration that will never be certified or shown to the board, a local Superset metric is acceptable as scaffolding. The moment the content is promoted toward production or shared, the metric must move to the governed semantic layer. The rule is: nothing certified or board-facing may carry a BI-tool-local metric definition.

Q: What is "query pushdown" in the context of Cube and a BI tool?
A: Query pushdown means the BI tool's SQL is translated and executed against Cube's model and underlying warehouse, with as much computation as possible delegated ("pushed down") to the data source rather than done in the BI tool. Cube uses DataFusion plus `egg` rewriting to match incoming SQL to cubes and push work down. Effective pushdown is why a thin BI tool over a semantic layer can still perform on large fund datasets.

Q: A Superset chart on top of Cube returns an error that the query can't be planned. What is a likely cause?
A: Cube's SQL API only answers queries it can map onto defined cubes/views, so SQL referencing columns, joins, or functions not represented in the model fails to plan. Likely you wrote (or Superset generated) SQL outside the semantic model — e.g. an ad-hoc join or a dimension that isn't exposed. Fix it by adding the needed measure/dimension to the Cube model rather than forcing raw SQL.

Q: Why does putting metric logic in a virtual dataset undermine the semantic-layer strategy even though it "works"?
A: A virtual dataset embeds metric SQL inside Superset, so the definition now lives in the BI tool and can diverge from the governed one — exactly the drift the semantic layer prevents. It works locally but breaks the single-source-of-truth invariant and is invisible to the metric governance process. Prefer thin physical/sync'd datasets that defer all metric math to Cube.

Q: What is the relationship between "self-service boundaries" and the certification workflow?
A: Self-service boundaries define how far users may go on their own (which datasets, whether SQL Lab is allowed, where they may publish), while the certification workflow defines how their output becomes trusted production content. Together they let exploration be free but make trust earned. The architect designs both as one governance system, not in isolation.

Q: How would you prove to an auditor that the fund-board NAV figure is governed?
A: Show that the dashboard's NAV metric resolves to a single named metric in the semantic layer, that the dashboard is certified with a named owner, and that RLS/entitlements are defined and tested. Then show there is no local SQL re-definition in the BI tool and that a sunset rule removes unowned competing content. The audit trail is metric-definition + ownership + certification + access control.

Q: What is the danger of "embedding" Superset dashboards without scoping each embed?
A: If embedded dashboards aren't scoped per viewer (via guest-token RLS), one client could see another client's funds inside your portal — a serious data-leak in a multi-manco/TA context. Embedding amplifies blast radius because the dashboard is reused across tenants. Always carry the tenant's RLS rules in the guest token so the embedded query is filtered server-side.

Q: Why is "one dashboard, many tenants via RLS" preferable to "one dashboard per client"?
A: A single RLS-scoped dashboard means one definition to maintain, certify, and audit, with each tenant filtered at query time — far less sprawl than cloning a dashboard per client. Per-client copies multiply maintenance, drift, and the very sprawl governance is meant to prevent. RLS turns multi-tenancy into a security configuration rather than a copy-paste explosion.

Q: What's the difference between a "report" being correct and being "certified"?
A: Correctness is whether the numbers are right today; certification is an accountable owner's endorsement that it is the trusted source, with a workflow and revocation behind it. A report can be momentarily correct yet uncertified (so users don't know to trust it) or certified yet later drift (triggering re-review). Certification is about trust and accountability over time, not a one-time correctness snapshot.

Q: How does aggressive self-service freedom trade off against governance, and where should the architect land?
A: Maximum freedom speeds delivery but breeds contradictory, unowned content; maximum lockdown is safe but starves the business and pushes users to spreadsheets. The architect lands on governed self-service: free sandbox exploration on certified datasets, a clear promotion path to owned/certified production. The boundary is set per organization's risk appetite — tighter in regulated fund admin.

Q: What concrete signals would tell you a BI estate is sliding into sprawl?
A: Rising counts of uncertified and unowned dashboards, many near-duplicate dashboards for the same domain, datasets built on ad-hoc SQL rather than the semantic layer, and business questions like "which number is right?". Low view counts on most content and no decommissioning activity are also tells. These are the metrics your governance dashboard (itself built in Superset) should track.

Q: How could you use Superset to monitor its own governance health?
A: Superset stores metadata about dashboards, datasets, owners, and certification, which you can surface as a governance dashboard: counts of certified vs uncertified content, unowned dashboards, datasets bypassing the semantic layer, and stale/low-view content. This makes sprawl visible and drives the decommissioning workflow. The BI tier thus polices itself with the same tooling it provides to the business.

Q: Why is "certified dataset" governance more durable than relying on naming conventions?
A: Naming conventions (e.g. prefixing "OFFICIAL_") are advisory and easily ignored or copied, whereas a certification flag carries an owner, a review record, and revocation semantics enforced by the tool and process. Users can trust a badge backed by accountability far more than a string in a name. Durable governance is a workflow with an owner, not a convention.

Q: In a Power BI shop, how do you keep board-facing measures consistent across many reports?
A: Centralize measures in a single shared semantic model (or serve them from Cube via the DAX API) and have all board reports consume that model rather than redefining DAX per report. This gives one definition of NAV/AUM/flows across every report. Letting each report invent its own DAX measures is the Power BI flavour of the sprawl/contradiction problem.

Q: What does "capacity planning" mean for a Power BI Fabric deployment?
A: It means choosing an F-SKU capacity size (compute) that can sustain your refresh schedules, query concurrency, and model sizes without throttling, and deciding whether free-viewer access (F64+) changes the cost equation. Under-size and users hit throttling/queueing; over-size and you overpay. The architect must size capacity against the author/viewer mix, refresh load, and feature needs.

Q: Why might a heavily-viewer organization prefer capacity over per-user Pro even before F64?
A: Because per-user cost scales linearly with every viewer, a large viewer base quickly exceeds the flat cost of a capacity, and at F64 viewers can be free entirely. So beyond a break-even headcount, capacity is cheaper and simpler to administer than thousands of Pro seats. The architect computes that break-even rather than assuming per-user.

Q: What is the role of OIDC/SSO in front of a BI tool like Superset?
A: SSO via OIDC lets the BI tool authenticate users against the organization's identity provider, so roles (and thus RLS scoping) derive from central identity rather than local accounts. This keeps entitlement consistent with the rest of the platform and simplifies offboarding. Identity is the foundation RLS and certification ownership rely on.

Q: How does role design in Superset connect to RLS and certification?
A: Roles are the unit RLS rules bind to (who gets which `WHERE` clause) and the unit you grant publish/certify permissions to, so role design determines both what data users see and what they may promote. Over-broad roles (e.g. handing out `Admin`) silently bypass RLS and governance. Tight, purpose-named roles are prerequisite to both security and governance working.

Q: A new analyst can publish directly to the production dashboard space. Why is this a governance smell?
A: Because production should require promotion through the certification workflow (owner, certified sources, review), direct-publish rights let uncertified content reach the board and reintroduce sprawl. The fix is to scope the analyst's role to a sandbox space and route promotion through review. Publish-to-production should be a deliberately narrow, governed privilege.

Q: Why does the lesson pair "Superset hands-on" with "Power BI fluency" rather than mastery of both?
A: Because the architect builds and reasons concretely in the FOSS reference tool (Superset) to internalize datasets, RLS, embedding, and semantic-layer consumption, while only needing fluency in Power BI's economics and positioning to make and defend tool decisions in Azure shops. You go deep where you build and broad where you select. Mastery of every commercial tool is neither feasible nor the architect's job.

Q: What's a concise market map of "who wins in which org shape" for BI tools?
A: Microsoft/Azure shops with large viewer bases → Power BI on Fabric capacity (F64 free viewers); analyst-led visual-exploration cultures → Tableau; engineering-led, warehouse-centric, code-defined-metrics teams on Google Cloud → Looker; flexible exploratory enterprises → Qlik; cost-sensitive or engineering-led teams wanting FOSS/self-host → Superset or Metabase. The deciding axes are cloud alignment, author/viewer ratio, and appetite for governed-vs-exploratory analysis.

Q: How do you verify, concretely, that a Superset dashboard "consumes only semantic-layer metrics"?
A: Inspect each chart's underlying dataset: confirm it is a physical/sync'd dataset mapped to a Cube cube/view (or sourced via Semantic Layer Sync), not a virtual dataset containing SQL, and that every metric is a Cube-defined measure rather than a Superset calculated metric. Check the chart's "View query" to see the generated SQL hits Cube's SQL API. If any chart carries local SQL or a local metric, it fails the check.

Q: What does Superset's "View query" feature let you confirm for governance?
A: "View query" shows the exact SQL Superset generated and sent to the source, letting you confirm a chart resolves to semantic-layer metrics (queries against Cube) rather than embedding local metric logic. It is your evidence tool for the "zero local SQL definitions" gate. If the query contains hand-written metric math, the chart violates the semantic-layer rule.

Q: Why is "consume Cube; never re-model metrics inside the BI tool" framed as the architect's discipline rather than a developer rule?
A: Because only the architect controls the design that makes it possible — provisioning the semantic layer, restricting SQL Lab, configuring Semantic Layer Sync, and setting certification rules so re-modelling is unnecessary and disallowed. Developers follow the path the architecture lays down. If the architecture leaves re-modelling easy and metrics absent from Cube, no amount of developer discipline will hold.

Q: What is the danger of certifying a dashboard but not the datasets it depends on?
A: A certified dashboard built on an uncertified or ad-hoc dataset inherits that dataset's untrustworthiness — the badge implies trust the underlying data doesn't justify. Certification must flow from certified, semantic-layer-sourced datasets upward. Certifying presentation without certifying the source is governance theatre.

Q: How does the "single source of truth" idea apply when the board and the regulator want the same NAV?
A: Both views must resolve to the one NAV metric in the semantic layer, so the board dashboard and the regulatory report cannot disagree by construction. If they consume the same governed metric, reconciliation between board and regulatory NAV becomes trivial. Divergence would only arise from a re-modelled figure somewhere — which the no-re-model rule forbids.

Q: What is a "thin" dashboard and why is it the goal over Cube?
A: A thin dashboard does only presentation — layout, formatting, filtering — and pushes all metric and join logic to the semantic layer, holding no SQL or metric definitions of its own. It is the goal because all the governable logic lives in one place (Cube), making the BI tier swappable and auditable. Thick dashboards (heavy virtual datasets, local metrics) re-create the sprawl problem.

Q: How would you handle a stakeholder who insists on a custom metric "just for their dashboard"?
A: Treat it as a request to add a governed metric: define it once in the semantic layer (named, owned, documented), then surface it in their dashboard, so even a niche metric stays consistent and auditable. If it's genuinely throwaway exploration, keep it in their sandbox and never certify it. The answer is "yes, but defined in Cube," not "yes, hard-coded in the BI tool."

Q: Why are "dates-and-hope" not acceptable as the only governance for retiring dashboards?
A: A pure calendar rule ("we'll clean up next year") never happens under operational pressure; durable decommissioning needs criteria (unowned, stale, deprecated source) plus a funded, scheduled process and a named approver. Otherwise the estate only grows. The decommissioning rule must be enforceable and owned, not aspirational.

Q: What organizational forces produce the "300 unowned dashboards" outcome?
A: Easy cloning with no ownership requirement, no certification gate, no sandbox/production split, reward for shipping dashboards but none for retiring them, and no funded decommissioning. Each force individually seems harmless; together they ratchet content upward with no counter-pressure. The architect's governance design must place a counter against each.

Q: For each sprawl-producing force, name a counter-control.
A: Easy unowned cloning → mandatory named owner at publish; no certification → a certification workflow with a visible badge; no sandbox/prod split → enforced spaces with a promotion gate; no retirement incentive → metrics that surface stale/unowned content; no funded cleanup → a scheduled, owned decommissioning process. Together these convert an ever-growing estate into a maintained one.

Q: How does Semantic Layer Sync reduce certification effort?
A: By pushing Cube's governed model into the BI tool as managed datasets, it means users build on already-trusted definitions rather than hand-crafting datasets that each need separate certification. Trust is inherited from the synced semantic model. This shrinks the surface area the certification workflow must police.

Q: Why might an architect still keep some metric logic in Superset for prototypes despite the no-re-model rule?
A: Prototyping new metrics quickly in Superset is faster than the full Cube definition cycle, and sandbox prototypes are explicitly not trusted, so local logic is acceptable scaffolding there. The rule bites at promotion: before anything is certified or board-facing, the metric must be moved into Cube. So local logic is a transient prototyping convenience, never a production state.

Q: A board pack figure differs from the certified dashboard by a small amount. What governance questions do you ask first?
A: Ask whether the board pack pulled from the certified dashboard's semantic-layer metric or from a separate (possibly re-modelled or stale) source, and whether anyone applied a manual adjustment. The discrepancy almost always traces to a figure produced outside the governed metric. The fix is to repoint the board pack at the certified metric and retire the rogue source.

Q: Why is "explicit ownership rules" listed as its own governed-self-service requirement?
A: Because ownership is the accountability hook that makes every other control enforceable — a certified dataset with no owner cannot be maintained or trusted, and an unowned dashboard is the unit of sprawl. Making ownership explicit (and required at publish) ensures someone answers for every figure. It is the keystone of the governance design.

Q: How do RLS and certification together protect a multi-manco Luxembourg platform?
A: RLS ensures each management company sees only its own funds (data isolation), while certification ensures the metrics they see are the trusted, owned definitions (numeric integrity). One without the other is incomplete: correct numbers leaked across clients, or isolated-but-untrusted numbers. Together they give isolated, trustworthy, board-defensible views per tenant.

Q: What is the architect's answer to "why not just let everyone use Excel"?
A: Excel re-implements metrics ungoverned, has no RLS, no certification, no shared definition, and no audit trail — it is sprawl with no controls at all, which is unacceptable for regulated NAV/AUM reporting. The governed BI tier exists precisely to give self-service without those failures. Excel is allowed for genuine throwaway analysis, not for board or regulatory figures.

Q: Why should the architect "read the vendor market in one sitting" rather than deep-evaluate every tool?
A: Because the goal is a defensible market map — who sells what, to whom — sufficient to justify the chosen tool and anticipate procurement questions, not to become an expert administrator of each. A one-sitting positioning read keeps effort proportionate to the decision. Deep evaluation is warranted only for the one or two tools actually in contention.

Q: How does the BI governance one-pager relate to DORA/regulatory expectations?
A: A regulated administrator must evidence controlled, accountable, reproducible reporting; the one-pager documents who may publish what, ownership, certification, and decommissioning — exactly the operational controls an auditor or DORA-style resilience review expects to see. It turns informal practice into a stated, auditable policy. Without it, BI is an ungoverned gap in the control environment.

Q: Summarize the core thesis of this lesson in two or three sentences.
A: The architect designs the BI tier — tool selection, licensing economics, and governance — rather than authoring dashboards, and the central discipline is to consume governed metrics from a semantic layer (Cube) so the BI tool stays a thin, consistent presentation layer. Governance (certification, named ownership, sandbox-to-production promotion, and funded decommissioning) is what prevents the "300 unowned dashboards" estate with contradictory board-facing numbers. In a regulated fund context, this is a compliance control, enforced by RLS for tenant isolation and by sourcing every figure from one definition.

Q: On a native Ubuntu laptop, what is the standard `pip` install sequence to get a local Superset instance running for the first time?
A: After creating a virtualenv you run `pip install apache-superset`, then `superset db upgrade` (builds the metadata schema), `superset fab create-admin` (creates the admin login), `superset init` (seeds default roles and permissions), and optionally `superset load_examples`; finally `superset run -p 8088 --with-threads --reload` starts the dev server. The `FLASK_APP=superset` environment variable must be set so the `superset` CLI finds the app. Skipping `superset init` leaves you with no roles, so RLS and permission management won't work.

Q: What does `superset fab create-admin` actually do, and what is the `fab` part?
A: It creates a Superset admin user (username, name, email, password) by calling Flask-AppBuilder's user-creation command — `fab` is the Flask-AppBuilder CLI namespace Superset embeds for user and role management. Without an admin user you cannot log in to configure databases, datasets, or RLS rules. The created admin holds the `Admin` role, which bypasses RLS, so it must never be the account you use to test scoping.

Q: After editing `superset_config.py` to add `FEATURE_FLAGS = {"EMBEDDED_SUPERSET": True}`, the embedding option still doesn't appear. First thing to check?
A: Confirm `superset_config.py` is actually on the `PYTHONPATH` and that `SUPERSET_CONFIG_PATH` (or the default discovery) points at it, then restart the Superset process, because feature flags are read only at startup. A common Ubuntu mistake is editing a config the running process never loads, or forgetting to restart `gunicorn`/`superset run`. Verify by checking the flag is reflected under Settings or via the `/api/v1/menu` payload after restart.

Q: Walk through the server-side guest-token flow for an embedded fund-administration portal.
A: Your backend authenticates to Superset (e.g. via the `/api/v1/security/login` endpoint to get an access token), then POSTs to `/api/v1/security/guest_token/` with the dashboard's embedded UUID, the user payload, and the RLS rules for that tenant, receiving a short-lived guest token. The frontend passes that token to `embedDashboard()` from `@superset-ui/embedded-sdk` via a `fetchGuestToken` callback. The service account doing the POST must hold the `can_grant_guest_token` permission, and the token must never be minted client-side because it would expose Superset credentials.

Q: How long does a Superset guest token live by default, and what is the practical consequence for an embedded board portal?
A: The default lifetime is about 5 minutes (`GUEST_TOKEN_JWT_EXP_SECONDS`), so the embedded SDK must call your `fetchGuestToken` callback again to refresh before expiry, which it does automatically. The consequence is that your backend token endpoint must stay available for the life of the session, not just at first load. A board member leaving a dashboard open for an hour will trigger many silent re-mints, each re-applying the tenant's RLS rules.

Q: You mint a guest token with RLS rules but the embedded dashboard still shows all funds. Where do you look first?
A: Confirm the RLS rules array in the guest-token request actually references the correct `dataset` id and a valid `clause`, and that the charts on the embedded dashboard are built on that same dataset — guest-token RLS only filters datasets named in the rule. Also verify the service account minting the token is not itself an `Admin`-equivalent whose context bypasses filtering. Decode the issued JWT to inspect the `rls` claim and prove the clause was carried.

Q: What HTTP status does Superset's `/api/v1/security/guest_token/` return on a malformed request, and what commonly triggers it?
A: It returns `422 Unprocessable Entity` when the request body fails schema validation — commonly a missing `resources` array, a wrong `type` (must be `dashboard`), or RLS rules with the wrong shape. The fix is to match the documented payload exactly, including the dashboard's embedded UUID rather than its numeric id. A `401`/`403` instead means the service account lacks `can_grant_guest_token` or a valid session.

Q: How do you find a dashboard's "embedded UUID" needed for the guest token, versus its normal numeric id?
A: You enable embedding on the dashboard (the dropdown "Embed dashboard" option, available with the `EMBEDDED_SUPERSET` feature flag) and Superset generates a separate embedded UUID shown in that dialog, distinct from the numeric id in the dashboard URL. The guest token's `resources` must use this embedded UUID. Passing the numeric id instead yields a token that won't authorize the embed.

Q: A Superset dataset built on Cube returns `column not found` for a field that exists in the warehouse. Why, and what's the fix?
A: Cube only exposes columns it has declared as dimensions or measures in its data model, so a warehouse column not surfaced through a cube/view is invisible to the SQL API even though it physically exists. The fix is to add that field as a dimension in the Cube model and re-sync, not to reach around Cube into the raw table. This is the semantic layer doing its job — only governed fields are queryable.

Q: What environment variable sets Cube's Postgres-compatible SQL API port, and what is the conventional value?
A: `CUBEJS_PG_SQL_PORT` enables and sets the SQL API port, conventionally `15432`, which you point Superset's Postgres connection at. You also need `CUBEJS_SQL_USER` and `CUBEJS_SQL_PASSWORD` for the SQL-API credentials. If the port isn't set, the SQL API simply isn't listening and Superset's connection test fails with a refused connection.

Q: From an Ubuntu shell, how would you quickly prove Cube's SQL API is reachable before wiring Superset to it?
A: Use `psql "postgresql://user:pass@cube-host:15432/cube"` (or `psql -h cube-host -p 15432 -U user`) and run a simple `SELECT` against an exposed cube, since the SQL API speaks the Postgres wire protocol. A successful `psql` session proves connectivity, credentials, and that the model is being served before you touch Superset. If `psql` fails, the problem is Cube/network, not Superset.

Q: Superset's SQLAlchemy URI for Cube uses which driver prefix, and what's a common mistake?
A: It uses the standard Postgres prefix, e.g. `postgresql://user:pass@cube-host:15432/cube`, because Cube emulates the Postgres wire protocol. A common mistake is appending warehouse-specific options or using a non-Postgres driver, which breaks the connection since Cube is not a real Postgres engine. Keep the URI minimal and let Cube handle planning.

Q: What does Superset's "View query" show for a chart correctly sourced from Cube, and what would betray a violation?
A: It shows a plain `SELECT` of Cube-defined measures and dimensions (often a simple aggregate over a view) sent to the SQL API, with no hand-written metric arithmetic. A violation betrays itself as inline metric math — a `SUM(...)/SUM(...)` ratio or a `CASE` computing a flow — meaning the definition lives in Superset, not Cube. That generated SQL is the auditable evidence for the "zero local SQL" gate.

Q: Compare Superset's per-dataset metric definitions against Cube measures for governance auditability.
A: A Superset metric lives in one dataset inside one Superset instance and is editable by anyone with dataset-edit rights, so the same metric can be redefined elsewhere with no record. A Cube measure is defined once in version-controlled model code, reviewed via pull request, and served to every consumer, giving a single auditable definition with git history. For NAV or AUM, only the Cube approach gives an auditor one place and one change-log to inspect.

Q: In Power BI, what is the technical difference between static and dynamic row-level security?
A: Static RLS hard-codes the filter value in the role's DAX expression (e.g. `[mgmt_company] = "ACME"`) and needs one role per value, while dynamic RLS uses `USERPRINCIPALNAME()` (or `USERNAME()`) to filter against a mapping/entitlement table so a single role serves all users. Dynamic is the scalable choice for many management companies because you add a row to the mapping table rather than a new role. Static becomes unmanageable past a handful of tenants.

Q: Write the shape of a dynamic Power BI RLS filter that scopes funds to the logged-in user's management company.
A: On the entitlement table the role's DAX is roughly `[user_email] = USERPRINCIPALNAME()`, and a relationship from that table to the fund table propagates the filter so each user sees only their manco's funds. `USERPRINCIPALNAME()` returns the signed-in user's UPN/email in the Power BI Service, matched against the mapping table. The gotcha is that the relationship must propagate in the right direction (single-direction from entitlement to facts), or the filter won't apply.

Q: Why does Power BI dynamic RLS often need bidirectional or carefully-modelled relationships, and what's the risk?
A: Because the filter must flow from the entitlement table through the model to every fact and dimension a user should be scoped on, and a missed propagation path leaks unfiltered rows. Turning on bidirectional cross-filtering can fix propagation but may open ambiguous paths and performance/security surprises. The architect's concern is that RLS modelled per-tool is fragile compared with scoping pushed into the semantic layer where every consumer inherits it.

Q: How do you test Power BI RLS before publishing, and what does that test NOT cover?
A: In Power BI Desktop you use "View as" → select the role (and for dynamic RLS, type a user UPN to simulate) to see the filtered model, and in the Service you use "Test as role". It does not cover users accessing the same semantic model through XMLA endpoints, exports, or another tool, nor does it protect the underlying warehouse — those remain unfiltered unless scoping is upstream. So passing the in-tool test is necessary but not sufficient for true entitlement.

Q: A Power BI report author is in the RLS role but still sees all rows. Likely cause?
A: Members of the workspace with Admin/Member/Contributor build access, and the dataset owner, are typically not subject to RLS — RLS applies to viewers consuming the report, not to those with edit rights on the model. So the author "sees everything" by design; you must test as a pure viewer or use "View as role". Mistaking the author's view for the viewer's experience is a classic RLS validation error.

Q: What is the smallest Fabric F-SKU that supports Power BI Embedded-style embedding, versus the threshold for free viewers?
A: Embedding capability is available from around `F8` upward (F8 mirroring the old A1 embedded capacity), but free-license viewers of standard shared content require `F64` or higher. So a small embedding scenario can run on a sub-F64 capacity, yet every interactive viewer of regular Power BI content below F64 still needs Pro or PPU. The architect must separate "can I embed" from "do viewers need a paid seat".

Q: Distinguish Fabric F-SKUs from the older A-SKUs and EM-SKUs for Power BI capacity.
A: A-SKUs (Azure) were purpose-built for Power BI Embedded / external app distribution and bill hourly with pause/resume; EM-SKUs were capacity licenses for embedding into Microsoft 365 surfaces like Teams and SharePoint; F-SKUs (Fabric) are the modern unified capacities covering Power BI plus all other Fabric workloads. Since the P-SKU retirement, F-SKUs are the forward path, and only F64+ grants free-viewer access. Recognising the legacy SKU names matters when reading inherited contracts.

Q: How does Fabric capacity pause/resume change the cost calculus versus per-user licensing?
A: F-SKU capacities (especially when bought as pay-as-you-go) can be paused so you stop paying compute during idle hours, which per-user Pro/PPU seats cannot — a seat costs the same whether used or not. For workloads with predictable off-hours, pausing a capacity can materially cut cost, but viewers still can't consume while paused. The architect weighs reserved (cheaper, always-on) versus pay-as-you-go (pausable, flexible) against the usage pattern.

Q: Size the licensing for a Luxembourg TA platform with 12 report authors and 400 read-only fund-board and oversight viewers.
A: With 400 pure viewers, per-user Pro at roughly $14/user/month is about $5,600/month for viewers alone, so an `F64` capacity (free viewers above F64) plus 12 author seats is almost certainly cheaper and simpler. You license the 12 authors with Pro (or PPU if they need Premium features) and host content on F64+, so the 400 viewers need only free licenses. The crossover clearly favours capacity once the viewer base is in the hundreds.

Q: When would PPU still beat an F64 capacity even with a sizeable user base?
A: When you need Premium-grade features (large models, frequent refresh, paginated reports, XMLA) for a population still comfortably under roughly 250 users and most of them author or interact rather than passively view, per-user PPU at about $24/user/month can undercut the flat cost of an F64. PPU also avoids capacity throttling management. The decision hinges on the author/viewer mix and which Premium features are genuinely required.

Q: What operational symptom tells you a Fabric capacity is undersized, and how does it differ from a per-user limit?
A: An undersized capacity shows throttling — interactive delays, queued or rejected operations, and refresh failures once the capacity's compute units are exhausted, because all users share one compute pool. Per-user licensing has no shared-compute throttle; instead the limit is the number of seats you've bought. So capacity problems look like performance degradation under concurrency, while per-user problems look like "this person has no license".

Q: A scheduled Superset alert or report email isn't firing. What components must be running on the Ubuntu host?
A: Superset's alerting/reporting needs Celery workers and Celery Beat running against a message broker (e.g. Redis), plus a headless browser driver (Chromium via `webdriver`/Playwright) for screenshots, and the `ALERT_REPORTS` feature flag enabled. If the worker or beat scheduler isn't running, schedules never execute; if the browser/driver is missing, the email has no rendered image. Check the Celery worker logs first to see whether the task was even picked up.

Q: Why is Redis (or another Celery broker) part of a production Superset deployment?
A: Superset uses Celery for asynchronous work — long-running SQL Lab queries, cached query results, and scheduled alerts/reports — and Celery needs a broker like Redis to queue tasks, plus Redis commonly backs the results/cache layer. Without it, async queries and scheduled reports don't run and dashboards lose caching. For a fund platform with heavy dashboards, the cache layer is also what keeps the board view responsive.

Q: How does Superset caching interact with the "single source of truth" goal — could it show stale NAV?
A: Superset can cache query results (per-dataset/per-chart cache timeout) so a dashboard may display a cached NAV that is older than the semantic layer's current value, creating apparent disagreement. The fix is to set conservative cache timeouts on board-facing NAV charts (or disable caching there) so freshness matches the figure's sensitivity. Caching is a performance feature that must be tuned against the regulatory need for the current official number.

Q: A board NAV chart shows yesterday's figure while the operational dashboard shows today's. Both source Cube. First diagnosis?
A: Suspect caching: compare the cache timeout on each chart/dataset, because one may be serving a cached result while the other re-queried Cube — same definition, different freshness. Force a refresh (the chart's "Force refresh" / clearing cache) and confirm both then agree, proving the definition is identical and the discrepancy was staleness. If they still differ after a forced refresh, then investigate whether one chart carries a hidden local filter or metric.

Q: What is the EU AI Act's relevance to a BI tier that surfaces model-driven outputs in fund dashboards?
A: If a dashboard presents outputs from an AI/ML system used for a regulated purpose, the AI Act's risk-tiering and transparency/record-keeping obligations may attach to that system, and the BI tier becomes where those outputs are shown and must be explained. The architect should ensure model-derived figures are labelled, sourced from a governed metric, and traceable, not silently mixed with deterministic NAV/AUM. BI governance (ownership, certification, lineage) is part of demonstrating the human-oversight and transparency the Act expects.

Q: Under DORA, why is a documented BI decommissioning process more than housekeeping?
A: DORA expects financial entities to manage ICT assets across their lifecycle with clear ownership and controls, so unowned, unmaintained dashboards presenting official figures are an ICT-risk and resilience gap, not just clutter. A documented, funded decommissioning process evidences lifecycle control over reporting assets. It lets the entity show an auditor that stale or orphaned board-facing content is actively retired, not accumulating risk.

Q: How does T+1 settlement raise the stakes on BI freshness for a fund-flows dashboard?
A: Under a T+1 cycle, subscription/redemption settlement happens one business day after trade date, so oversight teams need same-day, accurate net-flows views to spot funding or reconciliation problems before settlement, leaving less slack than the old T+2. A BI tier showing stale or re-modelled flows numbers can mask a break until it's too late to fix. So caching timeouts and a single governed `net_flows` definition matter more, not less, in a T+1 world.

Q: What is the EMT, and why might it appear as a dataset feeding a distribution-oversight dashboard?
A: The European MiFID Template (EMT) is a standardised file the fund industry uses to exchange MiFID II product-governance and cost/charges data (target market, costs, PRIIPs figures) between manufacturers and distributors. A distribution-oversight dashboard might surface EMT-derived fields (e.g. ongoing costs, target-market flags) to monitor that share classes are sold appropriately. As with NAV, those figures should come from a governed metric/dataset, not be re-derived in Superset.

Q: An oversight analyst wants to chart EET (European ESG Template) sustainability indicators alongside NAV. How should the architect handle it?
A: Model the relevant EET fields (e.g. PAI indicators, SFDR Article 8/9 classification) as governed dimensions/measures in the semantic layer, then expose them as a certified dataset the analyst builds on — so ESG figures are as auditable as financial ones. Letting the analyst load a raw EET file into a virtual dataset would create an ungoverned, re-modelled source. ESG data carries its own regulatory exposure (SFDR), so the same single-definition discipline applies.

Q: A chart joining ISIN-level holdings to an entitlement table is slow over Cube. Where does the architect push the fix?
A: Push the join and aggregation into the Cube model (define the relationship/join and a pre-aggregation) so the work is planned and executed at the semantic-layer/warehouse level via pushdown, rather than Superset trying to post-process. Cube pre-aggregations can materialise the rollup so the board chart hits a small, fast table. The wrong fix is a heavy virtual dataset in Superset, which both re-models metrics and performs worse.

Q: What are Cube pre-aggregations and why do they matter for a board dashboard over large fund data?
A: Pre-aggregations are materialised rollups Cube builds and maintains from the model so that matching queries are served from a compact, fast store instead of scanning raw warehouse tables. For a board dashboard aggregating millions of holding rows into a NAV trend, they turn a slow scan into a sub-second read. They keep the BI tier thin and fast without moving any metric logic into Superset.

Q: How would you give a sub-advisor self-service on only their share classes without exposing SQL Lab?
A: Bind a regular RLS rule with an `isin IN (...)` (or entitlement-join) clause to the sub-advisor's role on a certified, Cube-sourced dataset, add a base filter denying everyone else, and ensure their role grants no SQL Lab access. They then build charts only within the scoped certified dataset, never against raw tables. Verify with a test sub-advisor account that an out-of-scope ISIN returns no rows.

Q: Why is granting a business user the `sql_lab` role specifically dangerous in a multi-manco platform?
A: The `sql_lab` role lets the user run arbitrary SQL through SQL Lab against connected databases, which can read tables and rows that dataset-level RLS never covers, potentially crossing management-company boundaries. RLS is enforced per dataset query, not on free-form SQL, so SQL Lab is a bypass. Keep `sql_lab` to a small data-engineering group and route business users to certified datasets only.

Q: Which Superset roles bypass RLS, and how does that shape your test plan?
A: `Admin` (and any role granted broad database/all-datasource access, such as `sql_lab` running raw SQL, or `Gamma` granted overly wide datasource permissions) can bypass intended RLS scoping. Your test plan must therefore use purpose-built scoped roles and two non-admin test users, never validate scoping while logged in as `Admin`. Confirming the right role membership is as important as confirming the rule clause.

Q: Compare Superset RLS, guest-token RLS, and semantic-layer scoping — when does each apply?
A: Superset RLS scopes interactive users by role via injected `WHERE` clauses; guest-token RLS scopes embedded viewers per-token for multi-tenant embeds; semantic-layer (Cube) scoping pushes entitlement upstream so every consumer — Superset, Power BI, exports — inherits it. The first two protect only Superset's own queries, while the third is the strongest, tool-independent control. The architect layers them but treats semantic-layer scoping as the durable line of defence.

Q: How do you express "manco A OR manco B, AND region EU" using Superset RLS filter groups?
A: Put the two manco clauses in the SAME group key so they combine with `OR`, and put the region clause in a DIFFERENT group (or leave it ungrouped) so it `AND`s with the manco result. The engine ORs within a group and ANDs across groups, yielding `(manco='A' OR manco='B') AND region='EU'`. Mis-assigning the region clause into the manco group would wrongly OR it in and widen visibility.

Q: A "certified" badge is missing from a dataset you certified. What governance/config detail likely explains it?
A: Certification in Superset requires setting both the certified-by and certification-details metadata on the object (dataset/metric/dashboard), and the badge only renders when those fields are populated; a partial entry shows nothing. Also confirm you certified the object the dashboard actually uses, not a similarly named duplicate. The badge is metadata-driven, so the fix is completing the certification fields on the correct object.

Q: How can you audit the whole Superset estate for uncertified or unowned content programmatically?
A: Query Superset's metadata database (or the REST API endpoints like `/api/v1/dataset/` and `/api/v1/dashboard/`) for objects whose owners list is empty or whose certification fields are unset, and surface the counts on a governance dashboard. This turns sprawl into a measured number that drives the decommissioning workflow. Doing it via the API/metadata avoids eyeballing hundreds of objects by hand.

Q: Two Superset datasets both claim to be "official NAV". As architect, what is your concrete remediation sequence?
A: Identify which one resolves to the governed Cube NAV measure (via "View query"), certify that one with a named owner, repoint any dashboards off the other, then decommission the duplicate under the sunset rule. Communicate the single certified dataset so authors stop cloning the wrong one. The remediation is consolidate-to-one-definition plus retire, never "make both agree".

Q: Looker's LookML versus a Cube model — are they competitors or complements in this architecture?
A: Both are code-defined semantic layers, but in this architecture Cube is the chosen central semantic layer feeding many BI tools, whereas LookML is Looker's internal model tied to Looker and Google Cloud. They are conceptual cousins (define metrics once, in version control) but you would not run both as the source of truth — that would re-create drift. If Looker were in the estate, you'd either treat Cube as upstream or accept LookML as the layer, not duplicate metric definitions across both.

Q: Why might an architect pick Metabase over Superset for a small, cost-sensitive fund-ops team, and what is the trade-off?
A: Metabase emphasises ease of use and quick natural-language-ish question-asking, so a small non-technical ops team gets productive faster with less setup. The trade-off is less SQL power, weaker extensibility, and a thinner RLS/embedding/feature surface than Superset, which matters as governance and multi-tenant needs grow. The architect picks Metabase for speed-to-value but should anticipate outgrowing it if entitlement complexity rises.

Q: Give a concise "first thing to check" for: a Superset chart over Cube errors with a query-planning failure.
A: Check whether the generated SQL references a dimension, measure, join, or function not declared in the Cube model, because Cube's SQL API can only plan queries that map onto its cubes/views. Open "View query", find the offending field, and either add it to the Cube model and re-sync or rebuild the chart on exposed fields only. The error is almost always "you asked Cube something outside its model", not a Superset bug.

Q: Summarise, as a checklist, what makes a single fund-board dashboard fully defensible to a Luxembourg auditor.
A: Every figure resolves to one named Cube measure (verified via "View query", zero local SQL or virtual datasets), the dashboard and its source datasets are certified with named owners, RLS/entitlement is defined and proven with two scoped test users, caching is tuned so the displayed NAV is current, and a documented decommissioning rule retires competing unowned content. Together these give metric-definition + ownership + access-control + freshness + lifecycle evidence. That bundle is what converts a dashboard from "looks right" into "auditable under UCITS/AIFMD/DORA".

Q: A LEI on a counterparty dashboard renders as null for some rows over Cube. What is the architect's first check, and where is the fix?
A: First confirm whether `lei` is declared as a dimension in the Cube model and populated upstream, because a null over the SQL API usually means the field maps to a warehouse column with missing values rather than a Superset rendering bug. The fix belongs in the data pipeline/semantic layer (backfill or validate the LEI, add a DQ check), not in a Superset `COALESCE` that would hide the gap. Patching display in the BI tool masks a data-quality problem the certified DQ-status panel should be flagging.

Q: How would you stage a Superset-plus-Cube proof of concept on a single Ubuntu laptop before proposing it to the platform team?
A: Run Cube and Superset via Docker Compose locally, set `CUBEJS_PG_SQL_PORT=15432` so Cube exposes its SQL API, point a Superset Postgres connection at `cube:15432`, build one certified physical dataset on a Cube view, and add one RLS rule tested with two non-admin users. This end-to-end slice demonstrates "thin BI over governed metrics" with real isolation on commodity hardware. Keeping it containerised means the platform team can reproduce the exact stack rather than trust a hand-tuned local install.
