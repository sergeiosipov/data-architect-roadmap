# Phase 5 · 2.1.2 + 5.6.2 Azure Data Factory (bulk movement + orchestration) — 150 self-test questions

Q: What is Azure Data Factory (ADF) in one sentence, and what category of tool is it?
A: ADF is Microsoft's managed, serverless cloud data-integration service that orchestrates and automates the movement and transformation of data through pipelines. It is both a bulk data-movement tool and a workflow orchestrator, designed for hybrid ETL, ELT, and data-integration projects. In a Luxembourg fund estate it is the incumbent service you will be asked to evaluate against a code-first orchestrator like Dagster.

Q: What is an ADF `pipeline`?
A: A `pipeline` is a logical grouping of activities that together perform a unit of work, letting you manage the activities as a set rather than individually. For example one pipeline might ingest a daily NAV extract from blob storage and then run a downstream validation. Activities in a pipeline can run sequentially (chained) or in parallel.

Q: What is an ADF `activity` and what three categories exist?
A: An `activity` represents a single processing step inside a pipeline. ADF supports three categories: data-movement activities (the copy activity), data-transformation activities (mapping data flows, plus dispatched compute like HDInsight or Databricks), and control activities (the orchestration glue such as `ForEach`, `If Condition`, and `Until`). Choosing the right category is the difference between moving bytes and shaping control flow.

Q: What is a `trigger` in ADF and why is it distinct from a pipeline?
A: A `trigger` is the unit of processing that determines when a pipeline run is kicked off; it is decoupled from the pipeline so the same pipeline can be invoked by different triggers or manually. Keeping the trigger separate means you can change scheduling without editing pipeline logic. A run started by a trigger receives arguments for the pipeline's parameters.

Q: Name the main `trigger` types ADF offers.
A: ADF offers schedule triggers (run on a wall-clock recurrence), tumbling-window triggers (fixed-size, non-overlapping, contiguous time windows with state and dependency support), and event-based triggers (storage events like blob-created/deleted, or custom events via Event Grid). The tumbling-window type is the one you reach for when each run owns a discrete slice of time, such as a per-day NAV window.

Q: How does a `tumbling window` trigger differ from a plain `schedule` trigger?
A: A schedule trigger simply fires on a recurrence and is fire-and-forget, whereas a tumbling-window trigger creates a series of fixed-size, non-overlapping, contiguous windows, each as a separate run with its own start/end system variables. Tumbling windows support backfill of past windows and window-to-window dependencies, which a schedule trigger cannot do. Use tumbling windows when reprocessing a historical range of fund-valuation dates must be deterministic.

Q: What is a `dataset` in ADF and what is it NOT?
A: A `dataset` is a named, strongly typed view that points to or references a data structure within a data store, used as an input or output to an activity. It is not the connection itself and it does not hold credentials; it only describes the shape and location of data (for example, a specific container and folder of EMT files). The connection lives in a separate `linked service`.

Q: What is a `linked service` and what real-world analogy do the docs use?
A: A `linked service` defines the connection information ADF needs to reach an external resource, and the docs compare it to a connection string. It can represent either a data store (a SQL Server, Oracle, file share, or blob account) or a compute resource that hosts activity execution (such as an HDInsight cluster). A `dataset` rides on top of a `linked service` to describe the actual data structure.

Q: A `dataset` references a `linked service` references an `integration runtime`. In one line, what does each layer supply?
A: The `linked service` supplies the connection (where and how to authenticate), the `dataset` supplies the data shape (what structure to read or write), and the `integration runtime` supplies the compute and network bridge that actually performs or dispatches the work. The activity ties them together to define the action. Getting this layering right is the core mental model for ADF authoring.

Q: What is the `copy activity` used for?
A: The `copy activity` is ADF's data-movement primitive that moves data from a source data store to a sink data store, with built-in connectors, format conversion, column mapping, and optional compression. It is what you use for bulk ingestion such as pulling a transfer-agency extract from on-prem into Azure Data Lake. It runs on, and is billed through, an integration runtime.

Q: What is an `integration runtime` (IR) in ADF, in one sentence?
A: An integration runtime is the compute infrastructure ADF uses to provide data integration across different network environments, acting as the bridge between an activity and its linked services. It determines where data movement, activity dispatch, data-flow execution, and SSIS package execution physically happen. Choosing the wrong IR type is a common cause of connectivity and compliance failures.

Q: What are the three types of `integration runtime`?
A: The three IR types are Azure, self-hosted, and Azure-SSIS. The Azure IR is fully managed serverless compute for public-network (and, with managed VNet, private-link) work; the self-hosted IR runs on your own machine for private-network and on-prem sources; the Azure-SSIS IR is a managed VM cluster that natively runs lift-and-shifted SSIS packages.

Q: What does the Azure (default) `integration runtime` do?
A: The Azure IR runs data flows in Azure, runs copy activities between cloud data stores, and dispatches external transform activities over a public network (or private link when managed VNet is enabled). It is fully managed and serverless, so you do not provision, patch, or scale infrastructure and you pay only during actual utilization. For a copy activity you set the number of DIU and it elastically scales the compute up to match.

Q: When do you NEED a `self-hosted integration runtime` instead of the Azure IR?
A: You need a self-hosted IR (SHIR) whenever a source or sink sits in a private network with no direct line of sight from the public cloud, such as an on-prem transfer-agency database behind a firewall. The SHIR is installed inside that network and bridges it to ADF. It is also required for data stores that need a bring-your-own driver, such as SAP HANA or MySQL.

Q: Why is a `self-hosted integration runtime` the correct placement for an on-prem transfer-agency system?
A: A transfer-agency (TA) system typically lives on-premises behind a corporate firewall with no public endpoint, so the managed Azure IR cannot reach it. Installing the SHIR on a machine inside that network lets it extract data locally and push it out to the cloud sink. Crucially the SHIR only makes outbound HTTP-based connections to ADF, so you do not have to open inbound firewall ports — a key argument when justifying it to security teams.

Q: In which direction does a `self-hosted integration runtime` open connections, and why does that matter for security?
A: The self-hosted IR makes only outbound HTTP-based connections to the internet (to the ADF service); it never requires inbound ports to be opened. This matters because security and DORA-conscious teams resist inbound firewall holes into a regulated TA estate. You can therefore present the SHIR as outbound-only, which usually clears the network-security review.

Q: On which operating system can a `self-hosted integration runtime` run, and what is a common cross-platform gotcha?
A: Per the ADF documentation the self-hosted IR is currently supported only on a Windows operating system. This is a frequent gotcha for a Linux-first or native-Ubuntu shop: you cannot run the SHIR directly on Ubuntu, so you must provision a Windows host or Windows VM (on-prem or in the VNet) to host it. There is no macOS or native Linux build of the SHIR.

Q: What additional runtime dependency must be installed alongside a `self-hosted integration runtime`?
A: The Java Runtime Environment (JRE) is a dependency of the self-hosted IR and must be installed on the same host. It is used for handling certain file formats and operations during copy. A missing or wrong-version JRE is a classic cause of copy failures on a freshly installed SHIR.

Q: How do you achieve high availability and scale-out for a `self-hosted integration runtime`?
A: You scale out by associating the same logical SHIR instance with multiple on-prem machines in active-active mode, which gives both higher throughput and resilience to a single node failing. A single-node SHIR is a single point of failure, so for a production TA feed you register two or more nodes. The nodes share the same logical IR registered to the factory.

Q: What is the `Azure-SSIS integration runtime` for?
A: The Azure-SSIS IR is a fully managed cluster of Azure VMs dedicated to natively executing SQL Server Integration Services (SSIS) packages, enabling a lift-and-shift of existing on-prem SSIS workloads. You bring your own Azure SQL Database or SQL Managed Instance to host the SSIS catalog (SSISDB). You can stop and start it to manage cost, since it is dedicated VMs rather than serverless.

Q: Where should the `Azure-SSIS IR` be located relative to its SSISDB, and why?
A: The Azure-SSIS IR should be in the same location as the Azure SQL Database or SQL Managed Instance that hosts the SSISDB catalog. Co-locating them avoids excessive cross-region traffic between the IR and its catalog, maximizing performance and minimizing data-movement cost. The IR location does not have to match the Data Factory's own location.

Q: What is a `mapping data flow` in ADF and what engine runs it?
A: A mapping data flow is a visually authored graph of data-transformation logic that ADF executes on a Spark cluster which it spins up and tears down on demand. It lets data engineers build transformation graphs (joins, aggregates, derived columns) without writing Spark code or managing clusters. Because it is Spark under the hood, it shines for large, set-based transformations and is heavyweight for tiny ones.

Q: Why is a `mapping data flow` sometimes the WRONG engine choice?
A: Because every mapping data flow run spins up a Spark cluster, there is meaningful cluster start-up latency and vCore-hour cost even for trivial transformations. For small files or simple row-by-row logic, the Spark overhead dominates and a lightweight transform (or the copy activity's own mapping) is cheaper and faster. Reserve mapping data flows for genuinely large or complex set-based work.

Q: What compute meter does a `mapping data flow` bill against?
A: Mapping data flow execution and debugging are billed in vCore-hours, based on the compute type, number of vCores, and execution duration. This is a different meter from the copy activity's DIU-hours. Forgetting that data flows carry their own vCore-hour bill (plus cluster spin-up time) is a common cost surprise.

Q: Can you reduce Spark cluster start-up latency for repeated `mapping data flow` runs, and how?
A: Yes — you configure a Time to Live (TTL) on the Azure IR's data-flow properties so the Spark cluster stays warm between runs and subsequent activities reuse it instead of cold-starting. This trades a small idle cost for much lower per-run latency when many data flows run close together. Note that with TTL configured, those data-flow charges may not file under individual pipelines in per-pipeline billing.

Q: What is the lineage relationship between Azure Data Factory, Synapse pipelines, and Fabric Data Factory?
A: They are three generations of the same pipeline technology: standalone Azure Data Factory, the pipelines embedded inside Azure Synapse Analytics workspaces, and Data Factory in Microsoft Fabric, which Microsoft positions as the next generation of ADF. Recognizing which one an estate is on tells you the migration pressure and feature set you are dealing with. Microsoft advises new users to start with Fabric Data Factory.

Q: How do Synapse pipelines differ from standalone ADF in integration-runtime support?
A: Synapse pipelines currently support only Azure or self-hosted integration runtimes; they do not support the Azure-SSIS IR. So an SSIS lift-and-shift cannot live in Synapse pipelines and must stay in standalone ADF. This is one concrete reason an estate keeps both ADF and Synapse rather than consolidating.

Q: What does Microsoft say about `Data Factory in Microsoft Fabric` versus classic ADF?
A: Microsoft describes Fabric Data Factory as the next generation of Azure Data Factory, with a simpler architecture, built-in AI, and new capabilities, and recommends new users start there. Existing ADF workloads can upgrade to Fabric to gain features across data science, real-time analytics, and reporting. For an architect this means classic ADF is in maintenance-and-migration mode, which should shape any long-horizon recommendation.

Q: An estate is "on Synapse pipelines." What does that tell an architect about its data platform?
A: It signals the estate adopted the unified Synapse Analytics workspace (combining pipelines, dedicated/serverless SQL pools, and Spark) rather than standalone ADF. The orchestration engine is essentially the same as ADF but lives inside Synapse, sharing its security and workspace model. The architect should expect Synapse-specific limits (no Azure-SSIS IR) and a likely future pull toward Fabric.

Q: What are the three primary billing meters for ADF pipelines?
A: The three meters are Orchestration Activity Runs (charged per activity run orchestrated), Data Integration Unit (DIU) Hours (for copy activities on the Azure IR, based on DIU used times duration), and vCore Hours (for data-flow execution and debugging). At the end of the billing cycle each meter is summed as a separate line item. Understanding these three is the foundation of any ADF cost estimate.

Q: What is a `Data Integration Unit` (DIU)?
A: A DIU is a unit representing a combination of CPU, memory, and network-resource allocation applied to a copy activity running on the Azure IR. You set how many DIU a copy activity may use and the Azure IR elastically scales the underlying compute to match, with no manual cluster sizing. You are billed in DIU-hours: DIU allocated multiplied by the copy execution duration.

Q: How is the DIU-hour cost of a `copy activity` computed?
A: It is the number of DIU used multiplied by the copy execution duration in hours. The docs give a worked example where moving 100 GB consumed about `1.2667` DIU-hours, so moving 1 TB/day for a 30-day month works out to `1.2667 * (1 TB / 100 GB) * 30 = 380` DIU-hours. You then plug the DIU-hours plus the activity-run count into the pricing calculator.

Q: Besides the copy itself, what other meter does running a copy pipeline ALWAYS incur?
A: It incurs Orchestration Activity Runs, charged by the number of activity runs orchestrated, on top of the DIU-hours for the data movement. Even a pipeline that only copies one file is billed both for the activity run and for the DIU-hours consumed. For high-fan-out pipelines (for example a `ForEach` over thousands of fund files) the activity-run count, not the data volume, can dominate the bill.

Q: Where do the surprises hide in the ADF pricing model?
A: Surprises hide in three places: high orchestration-activity-run counts from large `ForEach` loops, self-hosted IR hours that accrue per IR-hour for data movement and pipeline/external activities, and data-flow vCore-hours plus Spark cluster spin-up time. A naive estimate that counts only DIU-hours misses all three. The architect must cost activity runs and IR-hours explicitly, not just the bytes moved.

Q: How does running a copy activity through a `self-hosted integration runtime` change the bill versus the Azure IR?
A: With the Azure IR you pay DIU-hours; with a self-hosted IR the meters are per-IR-hour for data movement, pipeline activities, and external activities, and you also bear the cost and maintenance of the host machine itself. So the SHIR removes the DIU-hour line but adds IR-hour charges plus your own infrastructure cost. That trade-off must be made explicit when justifying on-prem connectivity.

Q: A copy activity through `managed virtual network` is quoted at `$0.25` per DIU-hour. What does that number represent?
A: That is the documented price for a copy activity using the Azure integration runtime with managed virtual network enabled, billed per DIU-hour of execution. Managed VNet lets the Azure IR reach private-link endpoints in a private network, and it carries this DIU-hour rate. Treat the figure as a meter rate to multiply by your computed DIU-hours, and always confirm against the live pricing page since regional currency and rates vary.

Q: What does the per-activity `billableDuration` property in the run output tell you?
A: `billableDuration` appears in the JSON output of an activity run (for example a copy or data-flow activity) and reports the duration that is actually billed for that activity. You drill into it from the Monitor tab at activity-run level to find which activity in a pipeline is most expensive. It is the ground-truth signal for cost attribution when a pipeline's bill looks high.

Q: How can you get a per-pipeline cost breakdown in ADF, and is it on by default?
A: It is not on by default; you opt in per factory under Manage → Factory setting → "Showing billing report by pipeline", then publish. Once enabled, each pipeline gets its own entry in the Cost Management billing report, which is essential for chargeback (for example billing each fund or client for its own pipeline). It only affects bills emitted going forward and does not restate past charges.

Q: Why might enabling per-pipeline billing be a mistake in a factory with many pipelines?
A: Per-pipeline billing creates one billing entry per pipeline, so a factory with an excessive number of pipelines produces a long, complicated billing report that is hard to read and slow to process. The docs explicitly warn about this. Enable it where chargeback granularity is worth the report bloat, not blindly everywhere.

Q: Which charges do NOT file under individual pipelines even when per-pipeline billing is on?
A: Data Factory Operations charges (read/write and monitoring), Azure-SSIS node charges, and data-flow activities running on an Azure IR that has Time to Live (TTL) configured all fall back to a factory-level line item rather than a specific pipeline. So those costs cannot be attributed per pipeline. Knowing this prevents you from over-promising chargeback precision.

Q: What is `control flow` in ADF and which control activities matter most?
A: Control flow is the orchestration of pipeline activities — chaining in sequence, branching, looping, parameterizing, and passing arguments on invocation. The key control activities are `ForEach` (iterate over a collection), `If Condition` (branch), `Until` (loop with a condition), `Switch`, `Execute Pipeline` (call a child pipeline), and `Wait`. These are how you build non-linear orchestration without writing code.

Q: What is the practical limit of `ForEach` iteration in ADF, and why does it bite code-first teams?
A: `ForEach` has a bounded batch/parallelism limit and is designed for moderate fan-out, not arbitrary high-cardinality dynamic iteration, and each iteration's activities count as billable activity runs. A code-first orchestrator like Dagster can express dynamic, data-dependent fan-out and partitioning far more naturally and cheaply. This is a textbook place where ADF orchestration "breaks down" versus a code-first DAG.

Q: Name two concrete places where ADF orchestration breaks down versus a code-first orchestrator.
A: First, complex dynamic or data-dependent control flow and high-cardinality fan-out are awkward and costly in ADF's visual `ForEach`/`If`/`Until` model, whereas a code-first DAG (Dagster) expresses them naturally. Second, unit testing, local development, code review, and reuse of orchestration logic are weak in ADF's JSON/visual model compared with versioned Python. The architect should draw the line at the point where logic complexity or testability outweighs ADF's managed convenience.

Q: How does ADF support CI/CD?
A: ADF offers full CI/CD support for pipelines using Azure DevOps and GitHub, letting you develop incrementally in a collaboration branch and deploy via exported ARM templates. This enables a dev/test/prod promotion flow for your ETL artifacts. However, ADF artifacts are JSON definitions, so the developer experience is more click-and-template than code-first.

Q: Why must integration runtimes share the same name and type across CI/CD stages?
A: ADF requires the same IR name and type across all CI/CD stages so that pipeline references resolve consistently when ARM templates are promoted from dev to test to prod. If names diverge, the deployed pipelines point at non-existent IRs and fail. A common pattern is a dedicated shared factory holding the IRs, referenced from each environment as a linked IR.

Q: What pipeline-level construct lets you pass values into a run at trigger time?
A: `parameters` — read-only key-value pairs defined on the pipeline whose arguments are supplied at execution from the trigger or a manual run. They let one pipeline definition handle many runs, for example a `valuationDate` parameter driving a daily NAV pipeline. Activities inside the pipeline consume the parameter values.

Q: How do `parameters` differ from `variables` in ADF?
A: `parameters` are read-only inputs set at the start of a run from the trigger or invoking context and cannot be changed during execution, whereas `variables` are mutable values you can set and update inside a pipeline (for example with the `Set Variable` activity) to hold temporary state. Use parameters for run inputs and variables for in-flight bookkeeping such as accumulating a list. Mixing them up is a frequent authoring error.

Q: What is a `pipeline run` and how is it instantiated?
A: A pipeline run is a single instance of a pipeline's execution, typically instantiated by passing arguments to the pipeline's parameters. Those arguments can be supplied manually or from within a trigger definition. Each run has its own run ID, which you use to monitor and rerun it.

Q: What does "ADF is serverless and elastic" mean for capacity planning?
A: It means there is no fixed-size compute to provision for peak load; you specify how much resource to allocate on demand per operation (for example DIU per copy), and you pay only for what you use. So you size each operation, not a standing cluster, which makes scaling more elastic but also makes cost a function of usage rather than a flat reservation. This is the opposite of running your own always-on Spark or Airflow cluster.

Q: Scenario — a stakeholder says "we already pay for Azure, why not just use ADF instead of Dagster?" What is the architect's framing?
A: The architect answers with a costed, scenario-specific comparison rather than a vibe: ADF is the right default for managed bulk movement and simple orchestration where the team has little code culture, while a code-first orchestrator wins for complex dynamic DAGs, testability, and reuse. You quantify ADF's activity-run, DIU-hour, and IR-hour bill against the operational cost of self-hosting Dagster. The deliverable is "here is where each wins and what it costs," not a blanket yes or no.

Q: For a "lift-and-shift shop" scenario, why might ADF be the right call?
A: A lift-and-shift shop is migrating existing on-prem ETL (often SSIS) with limited appetite to rewrite in code, so ADF — especially the Azure-SSIS IR — lets them rehost packages with little change and gain a managed, monitored orchestrator. The cost is predictable and the team avoids building and operating its own orchestration stack. Here ADF's managed convenience outweighs the flexibility of a code-first DAG.

Q: For a "product team" scenario, why might a code-first orchestrator beat ADF?
A: A product engineering team values version control, unit testing, local development, code review, and dynamic data-dependent pipelines — all areas where ADF's JSON/visual model is weak and a code-first orchestrator like Dagster excels. The team already has the engineering culture to operate it. Here the flexibility and testability of code-first orchestration outweigh ADF's managed-service convenience.

Q: For a "hybrid estate" scenario, how do you reason about ADF?
A: A hybrid estate has both on-prem sources (needing a self-hosted IR) and cloud workloads, so you might use ADF specifically for the bulk on-prem-to-cloud movement via SHIR while orchestrating downstream transformation in a code-first tool. You cost the SHIR IR-hours and host maintenance against the alternative of self-managed connectors. The answer is often a split: ADF for the regulated on-prem ingestion edge, code-first for the rest.

Q: Why is "evaluation fluency" with ADF mandatory for the architect even if they prefer Dagster?
A: Because ADF is the incumbent in nearly every Azure-based Luxembourg fund estate and stakeholders already pay for it, so the architect is constantly asked to justify ADF or its replacement. Without fluency you either rubber-stamp ADF where it genuinely breaks down or reject it without being able to articulate why. The job is to give costed, defensible, scenario-specific answers.

Q: What is the difference between a "pipeline activity" and an "external activity" for billing?
A: Pipeline activities (such as Lookup, Get Metadata, and Delete) execute on the integration runtime itself, while external activities (such as dispatching a Databricks notebook or stored procedure) are dispatched by the IR to run on an external compute service. They are billed under different execution meters, with external activities reflecting dispatch and pipeline activities reflecting in-IR execution. Both also generate orchestration activity-run charges.

Q: What is the `Lookup` activity used for?
A: The `Lookup` activity retrieves a dataset (a single row or a set of rows) from a data source so its output can drive downstream control flow, for example reading a control table of fund codes to feed a `ForEach`. It executes on the IR associated with the data store's linked service. It is a pipeline activity, billed accordingly, and is a staple of metadata-driven pipelines.

Q: What is the `Get Metadata` activity used for?
A: `Get Metadata` returns metadata about a data store or dataset — such as item name, size, last-modified time, or the list of child items in a folder. It is commonly used to check whether an expected file has landed or to enumerate files for a `ForEach`. For a fund feed you might use it to confirm today's EMT file exists before triggering downstream processing.

Q: What is a metadata-driven copy pattern and why is it valued in fund administration?
A: A metadata-driven pattern stores a control table describing each source-to-sink mapping (connection, path, schedule) and uses `Lookup` plus `ForEach` to drive generic, parameterized copy activities, so adding a new fund or feed is a data change, not a pipeline change. In fund administration where new share classes, funds, and TA feeds onboard constantly, this avoids hand-building a pipeline per feed. The trade-off is that the `ForEach` fan-out drives up activity-run cost.

Q: Scenario — an on-prem TA SQL Server feed must land in ADLS Gen2 nightly. Sketch the ADF design.
A: Install a self-hosted IR (ideally two nodes for HA) on a Windows host inside the TA network; create a `linked service` for the SQL Server using that SHIR and a `linked service` for ADLS Gen2; define `datasets` for source table and sink folder; build a pipeline with a copy activity, parameterized by `valuationDate`; and fire it with a schedule or tumbling-window trigger nightly. The SHIR makes only outbound connections, so no inbound firewall ports are opened. Cost is SHIR IR-hours plus orchestration activity runs.

Q: First thing to check when a copy activity through a self-hosted IR fails to connect to an on-prem source?
A: First confirm the self-hosted IR node is registered and shows "Running"/online in the ADF Manage hub, since a stopped or unregistered node fails every copy. Then check that the host can reach the source on the right port and that the JRE dependency is installed. Because the SHIR is outbound-only to ADF, inbound firewall rules are rarely the cause — look at the SHIR-to-source path instead.

Q: A copy activity fails with a driver-related error against SAP HANA via the Azure IR. What is the likely fix?
A: SAP HANA (like MySQL) needs a bring-your-own driver, which the managed Azure IR cannot host, so the copy must run through a self-hosted IR with the required driver installed on the host. Switching the linked service to use the SHIR and installing the driver typically resolves it. This is a documented reason to choose the SHIR even when the source has a public endpoint.

Q: A freshly installed self-hosted IR copies small files fine but fails on certain file formats. What dependency should you check first?
A: Check that the Java Runtime Environment (JRE) is installed on the SHIR host, since the JRE is a documented dependency used for handling certain formats during copy. A missing or incompatible JRE produces format-specific failures even though basic connectivity works. Install the supported JRE version on the same host and retry.

Q: How does ADF decide which IR runs a copy when source and sink use different IRs?
A: For a copy between a cloud source and a private-network source, if either the source or sink linked service points to a self-hosted IR, the copy runs on that self-hosted IR. When both are in private networks, both linked services must point to the same IR instance. More broadly, a self-hosted IR takes precedence over a managed-VNet Azure IR, which takes precedence over the global Azure IR.

Q: What is the "autoresolve" Azure IR and what does it do for a copy activity?
A: Autoresolve is the default Azure IR setting where ADF makes a best effort to detect the sink data store's region and run the copy on an Azure IR in the same or closest region, falling back to the factory's region if detection fails. This minimizes egress and latency without manual configuration. For strict data-residency it can be the wrong default — see the explicit-region pattern.

Q: For a Luxembourg fund with strict data-residency, why might you pin the Azure IR region instead of using autoresolve?
A: Autoresolve may run the copy in a region you did not intend if region detection misfires, which can move regulated fund data out of the required geography. To guarantee data does not leave, say, UK South or a specific EU region, you explicitly create an Azure IR in that region and bind the linked services to it via the `connectVia` property. This is the documented pattern for strict data-compliance requirements.

Q: What is the `connectVia` property used for?
A: `connectVia` is the linked-service property that pins which integration runtime is used for that connection, overriding autoresolve. You use it to force a copy to run on a specific regional Azure IR (for data residency) or on a particular self-hosted IR (for private-network access). It is the explicit control behind IR selection.

Q: Why does the location you pick when creating a Data Factory matter, and what is stored there?
A: The Data Factory's location is where its metadata is stored and from where pipeline triggering is initiated; that metadata stays only in the chosen region. However, pipelines can still access data stores and run compute in other regions via the globally available IR. So the factory region governs metadata residency and trigger origin, not necessarily where every byte moves.

Q: Does the integration-runtime location have to match the Data Factory location?
A: No — the IR location defines where data movement, activity dispatch, and SSIS execution physically happen, and it can differ from the factory's location. This lets you place compute close to the data for performance and residency while keeping the factory metadata elsewhere. For a self-hosted IR there is no explicit location property because its compute is the host you provide.

Q: How would you implement the equivalent of an Airbyte + Dagster ingestion in ADF, conceptually?
A: Map Airbyte's connectors to ADF copy activities with appropriate linked services and integration runtimes (self-hosted IR for on-prem sources), and map Dagster's orchestration to ADF pipelines, control-flow activities, and triggers. Each source becomes a parameterized copy, ideally metadata-driven via a control table, scheduled by tumbling-window or schedule triggers. The architect then costs DIU-hours, activity runs, and IR-hours to compare against the FOSS stack.

Q: What goes into a monthly ADF cost estimate for such an ingestion?
A: You sum three things: DIU-hours for the copy activities (computed from a POC, e.g. DIU-hours per GB scaled to volume and run frequency), orchestration activity runs (count of activities times runs per month, watching `ForEach` fan-out), and self-hosted IR hours for any on-prem copies plus the host's own infrastructure cost. If mapping data flows are used, add vCore-hours and account for Spark spin-up. The pricing calculator turns these quantities into a currency figure.

Q: How do you derive DIU-hours for a cost estimate without guessing?
A: Run a proof-of-concept copy on a representative sample (the docs use 100 GB), read the actual DIU-hours from pipeline-run consumption monitoring, then project linearly to full volume and the operational schedule. The docs' example reads `1.2667` DIU-hours for 100 GB and scales it to `380` DIU-hours for 1 TB/day over a month. Measuring beats guessing because throughput depends on dataset shape, network, and egress limits.

Q: Where in the ADF UI do you see the consumption (meter quantities) for a single pipeline run?
A: In the Monitor experience, the default view is a list of pipeline runs, and clicking the Consumption button next to a pipeline name opens a pop-up showing the meter quantities (DIU-hours, activity runs, vCore-hours) aggregated across that run's activities. This shows quantities consumed, not the priced amount, because price depends on your account type and currency. To attribute cost to a specific activity, drill into the activity run's `billableDuration`.

Q: Why does the pipeline-run consumption view show quantities but not the actual price?
A: The consumption view reports meter quantities (such as DIU-hours and activity runs) because the price billed depends on your Azure account type and currency, which the run view does not know. You take those quantities into the pricing calculator or Cost Management to get a money figure. This separation keeps the run telemetry account-agnostic.

Q: What is the role of the Azure pricing calculator in an ADF cost exercise?
A: The pricing calculator turns measured meter quantities — activity runs, DIU-hours, data-flow vCore count and duration, IR instances — into an estimated monthly bill in your currency. You feed it POC-derived numbers rather than guesses. The "Do" task for this lesson is exactly to design the ADF implementation on paper and produce a monthly estimate with the calculator.

Q: An ADF bill is dominated by Orchestration Activity Runs, not data movement. What is the likely cause?
A: A high-fan-out pattern, typically a `ForEach` iterating over thousands of items (for example one activity per fund file), multiplies the activity-run count, and each iteration's activities are billed as runs. The fix is to batch work (fewer, larger copies), reduce per-item activities, or move high-cardinality dynamic iteration to a code-first orchestrator. Always count activity runs, not just bytes, when estimating.

Q: An ADF bill shows unexpected vCore-hour charges. What feature is responsible?
A: Mapping data flows are responsible — they execute on Spark and bill vCore-hours plus carry cluster spin-up cost, so even a few data-flow runs add a meter many people forget. Check whether transformations that could be done lightweight are routed through data flows. Consider an Azure IR TTL to amortize spin-up, but weigh the idle cost.

Q: An ADF bill shows self-hosted IR hours higher than expected. What should you investigate?
A: Investigate how many copy activities and pipeline/external activities route through the SHIR and for how long, since the SHIR bills per IR-hour for data movement and activities in addition to the host's own infrastructure cost. Long-running or frequent on-prem copies, or routing cloud-to-cloud copies needlessly through the SHIR, inflate it. Move public-endpoint copies back to the Azure IR where possible.

Q: Why might routing a cloud-to-cloud copy through a self-hosted IR be a costly mistake?
A: If both source and sink are public cloud stores, the Azure IR (DIU-hours) is the efficient path; forcing the copy through a self-hosted IR adds IR-hour charges and your host's cost while also funneling data through your on-prem machine, hurting throughput. The SHIR should be reserved for copies that actually touch a private-network endpoint. Check the linked services' `connectVia` to ensure cloud-to-cloud copies use the Azure IR.

Q: How does ADF support monitoring, and through which Azure service?
A: ADF has built-in monitoring via the Monitor tab in the portal and integrates with Azure Monitor, including Azure Monitor logs, plus access through the API and PowerShell. You can track success and failure rates of pipeline and activity runs and build health panels and alerts. For a regulated estate this is how you evidence that the nightly NAV feed ran and succeeded.

Q: What is the difference between a `data movement` activity, a `data transformation` activity, and a `control` activity?
A: A data-movement activity (the copy activity) moves bytes between stores; a data-transformation activity reshapes data (mapping data flows on Spark, or dispatched HDInsight/Databricks/stored-proc work); a control activity directs orchestration flow (`ForEach`, `If Condition`, `Until`, `Execute Pipeline`, `Wait`). Every activity falls into one of these three buckets. Knowing the bucket tells you which meter and IR behavior applies.

Q: What does the `Execute Pipeline` activity enable architecturally?
A: `Execute Pipeline` lets one pipeline invoke another as a child, enabling modular, reusable orchestration — for example a master pipeline that calls a parameterized ingestion sub-pipeline once per fund. It supports passing parameters and optionally waiting for completion. This is ADF's main mechanism for composing pipelines into larger workflows.

Q: How do tumbling-window dependencies help with backfilling historical fund data?
A: Tumbling-window triggers support window-to-window dependencies and can run past (backfill) windows, so reprocessing, say, every NAV date in a historical range becomes a deterministic series of contiguous, non-overlapping runs. Each window owns its slice and can depend on prior windows completing. A plain schedule trigger cannot express this ordered backfill.

Q: What is an event-based trigger and a fund-data use case for it?
A: An event-based trigger fires a pipeline in response to a storage event (such as a blob being created or deleted) or a custom Event Grid event, rather than on a clock. A natural fund use case is firing the ingestion pipeline the moment an EMT or EPT file lands in the landing container, instead of polling on a schedule. This reduces latency between file arrival and processing.

Q: Scenario — EMT files arrive at unpredictable times from multiple management companies. Schedule trigger or event trigger?
A: An event-based (storage-event) trigger fits better because arrival times are unpredictable; the pipeline fires when each file lands rather than waiting for a fixed schedule window. This minimizes processing latency and avoids empty scheduled runs. You would still add validation (for example `Get Metadata`) to confirm the file is complete before processing.

Q: Why is data compression in the copy activity relevant to bandwidth cost?
A: The copy activity can compress data during the copy and write the compressed form to the target, reducing the volume transferred and thus bandwidth usage. For large bulk movements such as full TA extracts over constrained on-prem links, compression can materially cut transfer time. It is a built-in copy-activity feature, not a separate step.

Q: What security model does ADF use for access control to its artifacts?
A: ADF integrates with Microsoft Entra ID and uses role-based access control (RBAC) to govern who can author and operate pipelines and data flows. This lets you separate, for example, who can edit a NAV pipeline from who can only run or monitor it. For regulated fund administration this RBAC plus Entra ID integration is part of the auditability story.

Q: Why is managed identity / Entra ID authentication preferred over storing credentials in linked services?
A: Using a managed identity or Entra ID lets ADF authenticate to data stores without embedding secrets in the linked-service definition, and secrets that are needed can be referenced from Azure Key Vault rather than stored inline. This reduces the risk of leaked credentials in exported ARM templates and aligns with DORA-style operational-resilience and secrets-hygiene expectations. It also simplifies rotation.

Q: How does ADF integrate with Azure Key Vault?
A: ADF can reference secrets stored in Azure Key Vault from its linked services so that connection strings, keys, and passwords are not stored in the factory's JSON. The factory authenticates to Key Vault (typically via managed identity) and pulls the secret at runtime. This keeps credentials out of source control and out of exported ARM templates.

Q: What is the difference between ADF's "managed virtual network" and a self-hosted IR for reaching private endpoints?
A: With managed VNet, the Azure IR itself runs inside a managed virtual network and reaches private resources over private-link endpoints, all serverless and managed by Microsoft. A self-hosted IR instead runs on a machine you operate inside the private network. Managed VNet suits private-link-reachable Azure resources; the SHIR is required when the source is genuinely on-prem with no line of sight from the cloud.

Q: For an on-prem mainframe-style transfer-agency system with no cloud connectivity, which IR and why?
A: A self-hosted IR, because the system has no public or private-link endpoint reachable from Azure, so only a runtime installed inside the on-prem network can extract from it and push outbound to the cloud. Managed VNet cannot help when there is no line of sight at all. You would size the SHIR host, install the needed driver and JRE, and run two nodes for HA.

Q: Why is the self-hosted IR being Windows-only a real planning constraint for a native-Ubuntu team?
A: Because the team cannot host the SHIR on their Ubuntu workstations or Linux servers; they must stand up and maintain a Windows host or VM purely to run the runtime, adding licensing, patching, and operational burden. This is a concrete cost line in the on-prem-ingestion estimate and a point of friction worth naming explicitly. On macOS or WSL2 the SHIR likewise is not the native path — a real Windows host is required.

Q: How does ADF differ from a pure ELT tool like dbt?
A: ADF is primarily a movement-and-orchestration service (copy data, schedule, branch) with optional Spark-based transformation via data flows, whereas dbt is a transformation framework that runs SQL inside a warehouse and does no data movement or scheduling itself. In practice ADF or another orchestrator triggers dbt rather than replacing it. Confusing the two leads to picking the wrong tool for movement versus in-warehouse transformation.

Q: How does ADF differ from Airbyte for ingestion specifically?
A: Airbyte is a connector-focused EL tool (extract-load) with a large catalog of open-source connectors, while ADF's copy activity provides built-in connectors plus full orchestration, IR-based network reach, and managed billing. ADF wins on managed on-prem reach (self-hosted IR) and tight Azure integration; Airbyte wins on connector breadth and being FOSS/self-hostable. The architect compares connector coverage, on-prem story, and cost.

Q: Why is "rubber-stamping ADF where it breaks down" a specific failure mode to avoid?
A: Because ADF's managed convenience masks weak spots — complex dynamic control flow, testability, reuse, and high-fan-out cost — so approving it for a workload that hits those limits bakes in expensive, brittle orchestration. The architect must recognize the break-down points and either design around them or draw the line at a code-first orchestrator. The opposite failure, rejecting ADF without articulating why, is equally unprofessional.

Q: A pipeline must copy 5,000 small fund files nightly. What ADF cost risk should you flag, and a mitigation?
A: The risk is that a `ForEach` over 5,000 files generates a very large orchestration-activity-run count, so the bill is driven by activity runs rather than the tiny data volume. Mitigations include using the copy activity's wildcard/folder copy to move many files in fewer activities, batching, or reconsidering whether a code-first orchestrator would handle the fan-out more cheaply. Always model the activity-run count for high-cardinality patterns.

Q: What does "activity dispatch" mean for an integration runtime, and is it heavy?
A: Activity dispatch is the IR routing an external transform activity (such as a Databricks notebook or stored procedure) to its target compute service and monitoring it; the actual heavy work runs on that external compute, not the IR. Dispatch is described as a lightweight operation, so the IR does not need to scale up for it. The compute cost lands on the external service, while ADF bills the orchestration/external-activity meters.

Q: When promoting ADF pipelines dev → test → prod, what artifact carries the definitions and what is the gotcha?
A: Promotion uses exported ARM templates of the factory, deployed through Azure DevOps or GitHub pipelines. A gotcha is that the per-pipeline detailed-billing setting is not included in exported ARM templates, so CI/CD will not overwrite it — by design, letting dev/test/prod have different billing behaviors. Another gotcha is that IR names and types must match across stages or references break.

Q: How would you argue ADF-versus-code-first for a regulated NAV-calculation feed specifically?
A: You weigh ADF's managed, auditable, Entra-ID/RBAC-secured movement with a self-hosted IR for the on-prem TA source against a code-first orchestrator's superior testability and reproducibility for the calculation logic. Often the answer is a split: ADF (with SHIR) owns the regulated on-prem-to-cloud ingestion edge, while the NAV transformation and validation live in versioned, unit-tested code. You back the recommendation with DIU-hour, activity-run, and IR-hour costs for the ADF portion.

Q: Why might a regulated fund administrator prefer ADF's managed monitoring over a self-built orchestrator for audit?
A: ADF's built-in run history, Azure Monitor integration, and per-pipeline billing give out-of-the-box, retained evidence that scheduled feeds ran, succeeded or failed, and what they cost — useful for DORA-style operational-resilience and audit. A self-built orchestrator can match this but you must build and maintain the observability yourself. The managed audit trail is a genuine point in ADF's favour for regulated workloads.

Q: Summarize the architect's one-paragraph verdict on ADF for a Luxembourg fund estate.
A: ADF is the incumbent, fully managed bulk-movement and orchestration service you will meet everywhere on Azure, and it is the right default for lift-and-shift, on-prem ingestion via a Windows-hosted self-hosted IR, and teams without code culture — provided you cost the activity runs, DIU-hours, IR-hours, and any data-flow vCore-hours honestly. It breaks down for complex dynamic control flow, high-fan-out, testability, and reuse, where a code-first orchestrator like Dagster wins, and it is in maintenance-and-migration mode as Fabric Data Factory becomes the next generation. The architect's job is the costed, scenario-specific call — lift-and-shift shop, product team, hybrid estate — not a blanket yes or no.

Q: What is the documented maximum number of items a single `ForEach` activity can iterate over, and why is that ceiling a design constraint?
A: A `ForEach` activity can iterate over at most `100000` items in one run, so a feed that enumerates more files or fund codes than that in a single collection will not fit in one loop. The constraint forces you to either batch the collection upstream (for example partition by management company or valuation date) or push the high-cardinality iteration into a code-first orchestrator. For a transfer-agency estate onboarding thousands of share classes, hitting this ceiling is a signal you have outgrown ADF's visual fan-out model.

Q: What are the default and maximum values of the `ForEach` `batchCount` property, and what does it govern?
A: When `isSequential` is `false`, `batchCount` caps how many iterations run in parallel, defaulting to `20` with a maximum of `50` concurrent iterations. Raising it increases throughput but also increases pressure on the integration runtime and the source's connection pool, so starting near `10` and increasing cautiously is the safe pattern. Setting `isSequential` to `true` ignores `batchCount` and forces one-at-a-time execution, which you use when later iterations must see the effects of earlier ones.

Q: When would you deliberately set `ForEach` `isSequential` to `true` despite the throughput cost?
A: You set `isSequential` to `true` when iterations have ordering dependencies — for example each NAV window must complete before the next so running balances or audit sequence numbers stay consistent, or when the downstream sink cannot tolerate concurrent writers. It trades parallel throughput for deterministic, ordered execution. The common gotcha is leaving it sequential by accident on an independent fan-out, which silently serialises thousands of copies and inflates wall-clock time.

Q: What are the four `dependsOn` dependency conditions between activities, and what does `Completed` mean?
A: The four conditions are `Succeeded`, `Failed`, `Skipped`, and `Completed`. `Completed` means the downstream activity runs whether the upstream finished `Succeeded` or `Failed`, which is how you wire a clean-up or notification step that must always execute. `Succeeded` is the default happy-path link, `Failed` builds an error branch, and `Skipped` fires only when the upstream was bypassed.

Q: In ADF, are multiple `dependsOn` conditions on one activity combined with AND or OR logic, and why does that trip people up?
A: Multiple activity dependencies are always evaluated as a logical AND, so an activity with `dependsOn` pointing at two predecessors runs only when both predecessors satisfy their stated conditions. People expect OR behaviour and are surprised when a join-style step never fires because one branch was `Skipped`. To get OR-style "run if either path completed" you typically restructure with an `If Condition` or route both paths through a single intermediate activity.

Q: What is the `Skipped` status and when does an activity acquire it?
A: An activity gets the `Skipped` status when an upstream activity it depends on (via a `Succeeded` link) fails, so the downstream never executes and is reported as skipped rather than failed. In a chain `X -> Y -> Z` on success links, if `X` fails then both `Y` and `Z` show `Skipped`. You exploit this with a `Skipped` dependency to build fallback or compensating logic that only runs when the normal path was abandoned.

Q: What is the `parallelCopies` (Degree of copy parallelism) setting on a copy activity, and how does it relate to DIU?
A: `parallelCopies` is the maximum number of threads within a single copy activity that read from the source or write to the sink concurrently, set in the JSON or as "Degree of parallelism" in the Settings tab. It is orthogonal to DIU and to self-hosted IR nodes — the parallel copies are counted across all DIUs or nodes, not per unit. If the source store or the SHIR is overwhelmed you lower `parallelCopies` to relieve load, which is a distinct lever from raising DIU.

Q: A copy into Azure Synapse Analytics is slow. What copy-activity feature is designed to speed bulk loads there, and how?
A: Enabling staged copy with PolyBase routes the data through interim blob storage and then bulk-loads it into Synapse via PolyBase, which is far faster than row-by-row inserts for large loads. You turn on staging and specify a staging linked service; it is also the documented path for performant loads from Snowflake, Redshift, and HDFS. The trade-off is the extra storage round-trip and a transient copy of the data in the staging account, which you must account for under data-residency rules.

Q: What does the `Auto` DIU setting do, and what is the valid DIU range for a copy activity?
A: Choosing `Auto` lets ADF dynamically pick the optimal DIU for your source-sink pair and data pattern instead of you fixing a number. The explicit range you can set is `4` to `256` DIU, and the actual DIU used at runtime may be lower than configured if the data pattern cannot exploit more. DIU applies only to the Azure IR — it has no meaning for a self-hosted IR, where throughput is governed by node count and concurrent-job limits instead.

Q: How is throughput scaled on a self-hosted IR if DIU does not apply there?
A: On a self-hosted IR throughput is governed by the number of nodes and each node's concurrent-jobs limit, which defaults from the host's CPU and RAM and is configurable up to `96` per node. With multiple nodes the effective concurrent-jobs ceiling is the sum across all available nodes. So you scale a SHIR by adding nodes or raising the per-node concurrent-job value, not by allocating DIU.

Q: What is the maximum number of nodes you can register to a single self-hosted IR, and why register more than one?
A: A single logical self-hosted IR supports up to `4` nodes, spread across machines on-prem or in a VNet. You register more than one node for high availability (surviving a node failure) and for higher aggregate throughput, since concurrent-job capacity sums across nodes. A single-node SHIR is both a performance bottleneck and a single point of failure for a production TA feed, so two-to-four nodes is the regulated-estate norm.

Q: Where in the portal do you change a self-hosted IR node's concurrent-jobs value, and what is the cap?
A: You edit it under Manage (or Author) → Connections → Integration runtimes → open the SHIR → Nodes → modify the concurrent-job value per node. The maximum settable value is `96` per node. Raising it lets one node run more simultaneous copies, but only up to what the host's CPU, memory, and the source's connection limits can sustain, so it is a tuning lever to apply with monitoring, not a free win.

Q: A daily NAV pipeline must not start until yesterday's window finished successfully. Which trigger feature expresses this, and how?
A: A tumbling-window trigger with a self-dependency expresses it: you add a window-to-window dependency on the trigger itself so each window waits for the prior window to complete before running. This serialises the series deterministically, which a schedule trigger cannot do. It is exactly the pattern for valuation feeds where each NAV date must build on a confirmed prior date.

Q: What system variables does a tumbling-window trigger expose to its pipeline, and how do you use them?
A: A tumbling-window trigger exposes `WindowStart` and `WindowEnd` system variables marking the window's boundaries, which you pass into pipeline parameters to scope each run's data. For a per-day NAV window you bind `valuationDate` to `WindowStart` so each run reads exactly its slice. This is what makes tumbling-window runs self-describing and backfill-safe, since every historical window carries its own start and end.

Q: How would you configure a tumbling-window trigger to reprocess a month of historical NAV dates safely?
A: You set the trigger's `startTime` back to the first day of the month with a one-day window size, and ADF materialises one contiguous, non-overlapping window per day that you can run as a backfill series. Each window passes its own `WindowStart`/`WindowEnd`, so the reprocessing is deterministic and idempotent per date. A schedule trigger could not do this — it has no concept of owning a past slice — which is why tumbling windows are the backfill primitive.

Q: An event-based trigger fires twice for what looks like one file landing. What is a likely cause and a mitigation?
A: Storage-event triggers can fire on each blob-created event, and some upload patterns (multi-block uploads, overwrite-then-rename, or tools that write a temp file then move it) emit more than one event, causing duplicate runs. Mitigate by filtering the trigger on a specific path prefix and file suffix (for example only `.csv` in the final landing folder), and make the downstream pipeline idempotent so a duplicate run is harmless. For EMT/EPT drops, validating with `Get Metadata` before processing also absorbs partial-write events.

Q: How do you make an event-triggered ingestion pipeline tolerate a partially-written file that arrives in two events?
A: You decouple arrival from processing: have the event trigger only record or stage the arrival, then use a `Get Metadata` plus a size or last-modified check (or a sentinel/marker file) to confirm the file is complete before the copy. Many fund-data senders write a zero-byte then full file, so reacting to the first event copies an empty file. Requiring a marker file or a stable size across two checks is the robust guard.

Q: What does the `Wait` activity do, and what is a better alternative for waiting on a file?
A: The `Wait` activity simply pauses the pipeline for a fixed number of seconds, which is a blunt instrument that wastes time if the wait is too long and fails if too short. For waiting on a file to land, an event-based trigger or an `Until` loop polling `Get Metadata` for existence is better because it reacts to the actual condition rather than a guessed delay. Hard-coded `Wait` steps are a common smell in fragile file-arrival pipelines.

Q: What is the `Until` activity, and what must you always pair it with to avoid an infinite loop?
A: The `Until` activity repeats its inner activities until a boolean expression evaluates to true, making it ADF's polling/retry construct. You must pair it with a `timeout` so a never-satisfied condition (for example a file that never lands) terminates instead of looping forever, and usually an inner `Wait` to avoid hammering the source. Without a timeout an `Until` polling for a delayed feed can run until the pipeline's own limit, burning activity runs.

Q: What is the `Switch` activity and when is it preferable to nested `If Condition` activities?
A: The `Switch` activity evaluates an expression and routes to one of several named case branches (plus a default), which is cleaner than deeply nested `If Condition` activities when you have more than two mutually exclusive paths. For example, routing a fund file to a different sub-pipeline per instrument type or per management company reads far better as a `Switch`. Nested `If`s quickly become unreadable and harder to maintain than a single `Switch`.

Q: How do you implement retry-on-failure for a flaky on-prem copy without writing code?
A: You set the copy activity's built-in `retry` count and `retryIntervalInSeconds` properties so ADF automatically re-runs the activity on transient failure before marking it failed. This is the first-line mechanism for a flaky TA network link, distinct from re-running the whole pipeline. For non-transient errors (bad credentials, missing driver) retries just delay the inevitable failure, so retries address transience, not misconfiguration.

Q: What is the difference between re-running a failed pipeline from the start and re-running from a failed activity?
A: ADF lets you rerun a pipeline from the beginning or rerun from the point of failure, where the latter skips already-succeeded activities and resumes at the failed one. Rerun-from-failed saves time and cost on a long pipeline where only a late step broke, for example a validation after a successful large copy. The gotcha is that rerun-from-failed assumes the earlier successful steps are still valid; if upstream data changed you may need a full rerun.

Q: How do you alert the team when a nightly NAV pipeline fails, using native Azure tooling?
A: You create an alert rule in Azure Monitor on the ADF metric for failed pipeline runs (or route diagnostic logs to a Log Analytics workspace and alert on a query), wiring it to an action group that emails or pages on failure. This gives evidence and notification without a custom watcher. For a regulated feed this is also part of the audit story — you can show that a failed valuation feed would have raised an alert.

Q: What is the difference between an activity-level `timeout` and a pipeline being stuck, and what is the default activity timeout?
A: Each activity has its own `timeout` that bounds how long it may run before ADF fails it; the historical default for many activities was 7 days (`7.00:00:00`), which is far too long for a nightly feed. A copy that hangs on a slow source will sit until that timeout unless you shorten it, masking a stuck pipeline. Setting a realistic timeout (for example a couple of hours for a NAV copy) makes failures surface fast instead of silently blocking the next window.

Q: How do you pass a value computed in one activity into a later activity in the same pipeline?
A: You reference the upstream activity's output in an expression, for example `@activity('LookupFundList').output.firstRow.fundCode`, binding it to the downstream activity's parameter or property. The `Lookup` and `Get Metadata` activities are the usual producers, and the consumer reads their `.output`. For multi-row results you typically feed `activity('Lookup').output.value` into a `ForEach` `items` expression.

Q: What is the `@item()` expression and where is it valid?
A: `@item()` references the current element being processed inside a `ForEach` loop's activities, letting each iteration act on its own value (for example `@item().filePath`). It is only valid within the scope of the `ForEach`. A frequent error is referencing `@item()` outside the loop or forgetting that nested `ForEach` is not allowed, which forces you to call a child pipeline via `Execute Pipeline` for two-level iteration.

Q: Why can't you nest a `ForEach` inside another `ForEach` directly in ADF, and what is the workaround?
A: ADF does not allow directly nesting a `ForEach` (or an `If`/`Until`) inside another `ForEach`, so a two-level fan-out such as per-fund-then-per-file cannot be expressed inline. The standard workaround is to move the inner loop into a separate pipeline and call it with `Execute Pipeline` from inside the outer `ForEach`, passing the current `@item()` as a parameter. This adds pipeline-run overhead and more activity runs, which is another place high-cardinality work gets expensive in ADF.

Q: What does the `Validation` activity do and how does it differ from `Get Metadata`?
A: The `Validation` activity blocks pipeline progress until a referenced dataset meets conditions you specify (existence, minimum size, or a child-item count) within a timeout, whereas `Get Metadata` simply returns properties for you to branch on yourself. `Validation` is purpose-built for "wait until the EMT file has actually landed and is non-empty before continuing." It combines the wait, the existence check, and the timeout into one step rather than hand-wiring `Until` plus `Get Metadata`.

Q: A copy activity reports a schema/column mapping error. Where do you look first, and what feature controls it?
A: You look at the copy activity's mapping (the explicit source-to-sink column mapping in the Mapping tab or `translator` in JSON); a mismatch or a renamed/removed source column breaks it. ADF can do automatic mapping when names align, but explicit mapping is brittle to source schema drift, which is common when a fund administrator changes an EMT layout. Switching to flexible/late-binding mapping or adding schema-drift handling in a data flow is the typical fix.

Q: How does ADF handle source schema drift, and which feature is built for it?
A: Mapping data flows support schema drift, where columns not known at design time are still carried through using rules and patterns rather than fixed mappings. The plain copy activity is less forgiving — a changed source schema can break its explicit column mapping. For fund feeds whose layouts mutate (extra fields appended to an EPT export), schema-drift-aware data flows or wildcard column patterns prevent breakage that a rigid copy mapping would suffer.

Q: What is the practical difference in fault tolerance between the copy activity and a data flow when a few rows are malformed?
A: The copy activity has fault-tolerance settings (`skipErrorFile`, skip incompatible rows, and a log location) that let it skip bad rows or files and continue, writing the rejects to a log, whereas a data flow handles row-level errors via its own error-row handling on the sink. For a TA extract with occasional corrupt rows, copy-activity fault tolerance lets the bulk land while quarantining the rejects. Silently skipping rows in a regulated feed is dangerous, so you log and reconcile the skipped count.

Q: How do you keep a copy idempotent so a rerun does not duplicate fund records in the sink?
A: You make the sink write deterministic per key — for example use an upsert/merge in a data flow keyed on a natural key like ISIN plus valuation date, or write to a partition folder named by `valuationDate` and overwrite that partition on rerun. A plain append copy is not idempotent and a rerun double-loads, corrupting NAV history. Designing the sink write so reruns are safe is essential because reruns and backfills are routine in fund operations.

Q: Why is overwriting a date-partitioned folder a clean idempotency pattern for NAV data in ADLS?
A: Writing each valuation date to its own folder (for example `/nav/year=2026/month=06/day=14/`) and configuring the copy to overwrite that folder means a rerun replaces exactly that date's data with no duplicates and no effect on other dates. It aligns naturally with tumbling-window runs that own one date each. The pattern also makes downstream partition pruning efficient and makes a single-date reprocess trivially scoped.

Q: A copy activity through the Azure IR fails to reach an Azure SQL Database behind private endpoints. What two configurations should you check?
A: Check whether the Azure IR has managed virtual network enabled with a managed private endpoint approved to that SQL Database, since a plain (public) Azure IR cannot reach a private-endpoint-only resource. Also confirm the SQL Database firewall and "Deny public network access" settings actually permit the private path. The alternative is routing the copy through a self-hosted IR inside the VNet; managed VNet is the serverless option, the SHIR the self-operated one.

Q: What is a managed private endpoint in the context of an ADF managed VNet?
A: A managed private endpoint is a private endpoint that ADF creates inside its managed virtual network to connect privately, over the Microsoft backbone, to a specific Azure resource such as a storage account or SQL Database. The target resource's owner must approve the private-endpoint connection before traffic flows. This is how a managed-VNet Azure IR reaches private resources without you operating a self-hosted IR, and it keeps the regulated data off the public internet.

Q: When source and sink both sit in private networks, what is the rule about which IR they must use?
A: Both linked services must point at the same self-hosted IR instance, because a copy between two private-network stores has to run on a runtime that can see both, and that is a single SHIR with line of sight to each. You cannot split such a copy across two different self-hosted IRs. If the two private networks cannot both be reached by one SHIR, you stage through an intermediate reachable store.

Q: In IR-selection precedence, which IR wins when source and sink imply different runtimes?
A: A self-hosted IR takes precedence over a managed-VNet Azure IR, which in turn takes precedence over the global (public) Azure IR. So if either the source or sink linked service is bound to a self-hosted IR, the copy runs on that SHIR. This precedence is why a cloud-to-cloud copy can accidentally run on a SHIR — and rack up IR-hours — if one of its linked services is still pointed there.

Q: How do you force a specific copy to run in a chosen region for data residency, overriding autoresolve?
A: You create an Azure IR pinned to the required region and bind the relevant linked services to it via the `connectVia` property, so ADF runs the copy there instead of autoresolving to the sink's detected region. This guarantees regulated fund data is processed only in the permitted geography (for example an EU region). Relying on autoresolve risks region detection misfiring and moving data out of the required jurisdiction.

Q: What is the difference between the Data Factory's region and where the copied bytes actually move?
A: The factory's region only fixes where its metadata lives and from where triggers originate; the bytes move wherever the chosen integration runtime runs, which can be a different region entirely. So a factory in West Europe can run a copy on an Azure IR pinned to UK South. This separation means residency questions must be answered at the IR level, not by pointing at the factory's region.

Q: How do you reference an Azure Key Vault secret in a linked service, and what authenticates the lookup?
A: In the linked service you select "Azure Key Vault" for the secret field and point to a Key Vault linked service plus the secret name, so the connection string or password is pulled at runtime rather than stored in the factory JSON. The factory's managed identity (granted a Key Vault access policy or RBAC role like Key Vault Secrets User) authenticates the lookup. This keeps secrets out of exported ARM templates and out of source control.

Q: What permission does the Data Factory managed identity need on a storage account to copy without keys?
A: The factory's managed identity needs an Azure RBAC data-plane role on the storage account, typically `Storage Blob Data Contributor` for read/write to blobs, rather than an account key. Granting the role and using managed-identity auth in the linked service removes embedded secrets entirely. A common failure is assigning a control-plane role like `Contributor` and finding data-plane operations still denied, because blob data access needs the data-plane role.

Q: When promoting via ARM templates, how are environment-specific values like connection strings handled?
A: They are parameterised in the exported ARM template and supplied per environment through a template parameters file, so dev/test/prod each inject their own linked-service values at deploy time. You override the autogenerated parameterisation with `arm-template-parameters-definition.json` when the defaults do not expose the right properties. This is how the same pipeline definition points at different storage accounts or databases across stages.

Q: Why is a dedicated shared factory for integration runtimes a common CI/CD pattern?
A: Because ADF requires IR name and type to match across stages, teams put the self-hosted IR in one shared factory and reference it as a linked (shared) IR from each environment's factory, so promotion never breaks on IR references. It also avoids installing the SHIR agent multiple times for non-prod. The pattern centralises the on-prem connectivity edge while keeping per-environment pipeline factories clean.

Q: A data-flow pipeline is slow because each run cold-starts Spark. Beyond TTL, what cluster setting also affects start time and cost?
A: Besides setting a Time to Live on the Azure IR's data-flow properties to keep a warm cluster, the compute type and core count of the data-flow IR (General Purpose vs Memory Optimized, and the number of cores) affect both spin-up behaviour and the vCore-hour rate. Over-provisioning cores raises cost without helping small flows; under-provisioning slows large ones. You right-size cores to the data and use TTL to amortise spin-up across closely-spaced runs.

Q: How do you debug a mapping data flow without running the whole pipeline, and what does it cost?
A: You turn on Data Flow Debug, which spins up an interactive Spark cluster so you can preview transformations and inspect data at each step before publishing. Debug sessions bill vCore-hours for as long as the debug cluster stays alive, not just while you click, so leaving a debug session running idle is a quiet cost leak. You turn the session off when finished rather than relying on it timing out.

Q: What does the Consumption report on a pipeline run show versus what Cost Management shows?
A: The per-run Consumption pop-up shows meter quantities — DIU-hours, orchestration activity runs, vCore-hours — consumed by that single run, but no money figure, because price depends on your account and currency. Cost Management aggregates the priced charges across runs into the actual bill and supports per-pipeline breakdown when you have enabled it. You use Consumption to attribute usage to a run and Cost Management to attribute money.

Q: An ADF estimate must compare against self-hosted Dagster. What ADF cost lines do you tabulate, and what is the matching Dagster cost?
A: For ADF you tabulate orchestration activity runs (watch `ForEach` fan-out), DIU-hours for Azure-IR copies, IR-hours plus the Windows host for any self-hosted-IR copies, and data-flow vCore-hours including Spark spin-up. The matching Dagster cost is the always-on compute and storage for the orchestrator plus the connector compute, the engineering time to build and operate it, and on-call. The honest comparison weighs ADF's per-use metered bill against Dagster's mostly-fixed run-it-yourself cost, scenario by scenario.

Q: Summarise the decision rule for choosing between an event trigger, a schedule trigger, and a tumbling-window trigger for a fund feed.
A: Use an event trigger when files arrive at unpredictable times and you want to react on arrival (EMT/EPT drops), a schedule trigger for simple fire-and-forget wall-clock recurrence with no per-window state, and a tumbling-window trigger when each run must own a discrete, contiguous time slice with backfill and window-to-window dependencies (daily NAV that must build on the prior date). The deciding factors are arrival predictability, whether each run owns a time slice, and whether you need ordered backfill. Matching trigger type to these properties is the core scheduling design call in an ADF fund pipeline.

Q: You need an audit trail of exactly which files a copy activity moved each night. What copy-activity feature provides it?
A: You enable copy-activity session logging, pointing it at a log storage location so ADF writes a per-run log of the files copied, skipped, and their status. This gives a retained, queryable record of what landed each valuation night, which supports DORA-style operational-resilience evidence and reconciliation against the source manifest. It is distinct from pipeline run history because it captures file-level detail, not just activity-level success.

Q: There is a cap on how many activities a single pipeline can contain. Why does it matter and what do you do when you approach it?
A: A single pipeline is limited to a maximum number of activities (a soft limit raised over time from `40` to `80` and now `120` in standalone ADF, including the inner activities of containers; Fabric Data Factory currently caps at `80`), so a sprawling metadata-driven pipeline can hit the ceiling. When you approach it you decompose into child pipelines invoked via `Execute Pipeline`, which both stays under the limit and improves modularity and reuse. Trying to cram an entire estate's orchestration into one pipeline is an anti-pattern that this limit usefully discourages.
