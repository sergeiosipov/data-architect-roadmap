# Phase 2 · 2.1.1 Batch ELT connectors — 150 self-test questions

Q: What is a batch ELT connector platform like Airbyte, in one sentence?
A: It is a tool that moves data from many sources into destinations on a schedule, extracting and loading raw records (the E and L of ELT) while leaving most transformation to a downstream tool like dbt. The platform supplies a library of pre-built connectors so you do not hand-code extraction logic for every API or database. For a fund administrator it is how you land transfer-agency exports, NAV feeds, and reference data into bronze without writing a bespoke script per source.

Q: In Airbyte's model, what is a `source`?
A: A `source` is the system you ingest data from — Airbyte defines it as "an API, file, database, or data warehouse that you want to ingest data from." It is one half of a pipeline and is configured from a source connector plus the credentials and settings that connector's spec requires. Example: your Phase-1 Postgres holding NAV and holdings tables is a `source`.

Q: Where does ingested data land, and what does Airbyte call that endpoint?
A: It lands in a `destination` — "a data warehouse, data lake, database, or an analytics tool where you want to load your ingested data." It is the second endpoint of a pipeline and, like a source, is an instance of a destination connector with its own configuration. In the capstone, MinIO holding bronze Parquet plays this role.

Q: What does the term `connector` mean as distinct from a source or destination?
A: A `connector` is the reusable software component that knows how to talk to a particular system — it pulls records from sources or pushes records to destinations and declares what configuration it needs. A `source` or `destination` is a configured instance of a connector with your specific credentials and options. So "Postgres source connector" is the reusable code; "my fund Postgres source" is the configured source.

Q: What is a `connection` in Airbyte?
A: A `connection` is "an automated data pipeline that replicates data from a source to a destination" — it binds a configured source to a configured destination and adds the sync schedule, stream selection, and sync-mode choices. It is the object you actually run and monitor. One source and one destination can be combined into multiple connections with different stream selections.

Q: What is a `stream` in Airbyte terms?
A: A `stream` is "a group of related records," which maps to a table in a relational database, a file, an API endpoint, or a blob depending on the source type. You select which streams a connection replicates and set the sync mode per stream. For your fund Postgres source, each table (funds, share_classes, nav) surfaces as a separate stream.

Q: What is a `field` in a stream?
A: A `field` is "an attribute of a record in a stream" — a database column, a JSON property in an API response, or a key in a file record. Fields are what schema-change handling watches: a new or retyped field is a field-level schema change. In a NAV stream, `nav_date`, `share_class_isin`, and `nav_per_share` are fields.

Q: What are the four pillars commonly described as the connector model: source, destination, spec, and state?
A: They are the contract every Airbyte connector implements: the `source`/`destination` say where data comes from and goes; the `spec` declares how the connector is configured; and `state` records how far an incremental sync has progressed. Together they let the platform run, resume, and reconfigure any connector uniformly. Learning these four is the core mental model for this lesson.

Q: What is a connector's `spec` (specification)?
A: The `spec` is the connector's self-description of how it can be configured — it is returned by the `spec` operation as a `ConnectorSpecification` and is expressed in JsonSchema. Airbyte renders that JsonSchema into the configuration form you fill in (host, port, credentials, options). It is how the platform supports arbitrary connectors without hard-coding each one's settings.

Q: Where does a Python connector usually define its spec, and in what format?
A: A connector typically declares its spec in a `spec.yaml` or `spec.json` file at the module root, written as a JsonSchema document. The platform reads this file when it runs the `spec` command and uses it both to validate your config and to generate the setup form. So the same JsonSchema both documents and enforces the connector's configuration.

Q: What does the `spec` command actually return at the protocol level?
A: It returns a `ConnectorSpecification` wrapped in an `AirbyteMessage` of type `spec`. That message carries the JsonSchema connection specification plus metadata such as which features the connector supports. The platform consumes this to know how to configure and drive the connector.

Q: What is `supported_destination_sync_modes` in a spec, and why does it matter?
A: It is a field in the specification that lists which sync modes a destination connector can support. It matters because a connection's available sync modes are the intersection of what the source offers and what the destination accepts — a destination that cannot do dedup limits your options. Checking it explains why some source/destination pairs cannot run incremental append + deduped.

Q: What is `protocol_version` in a connector spec?
A: `protocol_version` describes which version of the Airbyte Protocol the connector implements, with a documented default of `0.2.0`. It lets the platform and connector negotiate compatible message formats. You rarely set it by hand, but a mismatch is a thing to check when a connector behaves oddly after an upgrade.

Q: What is the Airbyte Protocol, conceptually?
A: It is the standardized set of commands (`spec`, `check`, `discover`, `read`/`write`) and JSON message types that every connector speaks, so the platform can drive any connector uniformly. Connectors emit and consume `AirbyteMessage` objects (records, state, logs, specs, catalogs) over stdio. This protocol is exactly what decouples the connector library from the orchestration platform.

Q: What is the `check` operation a connector exposes?
A: `check` validates that the configuration you supplied actually works — it attempts a connection to the source or destination using your credentials and returns success or a failure reason. It is what the UI runs when you click "Test connection." A failing `check` on a fund Postgres source usually means a credential, host, or network/firewall problem, not a sync logic problem.

Q: What is the `discover` operation and what does it produce?
A: `discover` inspects the source and returns an `AirbyteCatalog` describing the available streams and their fields and types. It is how Airbyte learns the source's schema so you can select streams and the UI can show field lists. Re-running discover is also how the platform detects schema changes against the previously stored catalog.

Q: What is the difference between the discovered `AirbyteCatalog` and the `ConfiguredAirbyteCatalog`?
A: The `AirbyteCatalog` is everything the source offers (all streams and fields from `discover`); the `ConfiguredAirbyteCatalog` is your selection — the streams you chose, each annotated with its sync mode, cursor field, and primary key. The connection runs against the configured catalog, not the full one. So enabling a new stream means adding it to the configured catalog.

Q: What is `state` in Airbyte, in plain terms?
A: `state` is the bookmark that records how far an incremental sync has progressed — typically the maximum cursor value seen so far per stream. The platform persists it after each successful sync so the next run only fetches records beyond that point. Without state, every sync would be a full refresh.

Q: Why does full refresh not use state, but incremental does?
A: Full refresh modes "always retrieve all available information requested from the source, regardless of whether it has been synced before," so there is nothing to remember between runs. Incremental modes must remember the last cursor position to fetch only newer records, so they read and advance `state` each sync. State is therefore the defining mechanism of incremental sync.

Q: Name the four sync modes Airbyte offers.
A: They are full refresh overwrite, full refresh append, incremental append (history), and incremental append + deduped (dedup). The "full refresh vs incremental" axis decides whether state is used; the "overwrite/append vs deduped" axis decides what the destination ends up holding. Your choice per stream is recorded in the configured catalog.

Q: What does full refresh overwrite do to the destination?
A: Each sync destroys all existing data in the destination table and reloads the complete current source dataset — "new syncs will destroy all data in the existing destination table and then pull the new data in." A consequence is that rows deleted at the source disappear from the destination too. It is the simplest mode and fine for small, fully reloadable reference tables.

Q: How does full refresh append differ from full refresh overwrite?
A: Both pull the entire source each sync (no state), but append adds the full pulled dataset alongside what is already there instead of replacing it. The result is accumulating snapshots — useful if you want a crude history of how a small table looked over time. The cost is unbounded growth, since every sync re-lands every row.

Q: What does incremental append do?
A: It replicates only new or modified records since the last sync using a cursor, and appends each one to the destination — "a copy of each new or updated record is appended." It uses state but does not deduplicate, so the destination can hold multiple copies of the same logical record over time. It is the right mode when you want full change history per record.

Q: Why can incremental append produce multiple rows for the same logical entity?
A: Because an updated source record is appended as a new row rather than replacing the old one — "you can find multiple copies of the same record in the destination warehouse." Each NAV correction for the same `share_class_isin` and `nav_date` lands as an additional row. That is a feature if you want an audit trail, and a problem if downstream consumers assume one row per key.

Q: What does incremental append + deduped (dedup) guarantee at the destination?
A: It syncs only new or modified data and guarantees the final table is "unique per primary key," keeping only the latest version of each entity. Behind the scenes it appends the delta to a history table and upserts the deduped final table by primary key. It is the mode you want for a silver NAV table where one row per share class per date is the contract.

Q: In dedup mode, what two stream-level settings must you supply?
A: A cursor field (to detect what changed) and a primary key (to deduplicate). The cursor — for example `updated_at` — decides which records to pull; the primary key — for example (`share_class_isin`, `nav_date`) — decides which rows collapse into one in the final table. Missing either makes correct dedup impossible.

Q: What is a `cursor field` versus a `cursor`?
A: The cursor field is "the field or column in the data where that cursor can be found," such as an `updated_at` timestamp column. The cursor is "the actual timestamp value used to determine if a record should be replicated" — the concrete value stored in state. So the field names the column; the cursor is the bookmark value within it.

Q: What is a `primary key` in Airbyte dedup terms?
A: It is "one or multiple (called composite primary keys) fields or columns that is used to identify the unique entities of a table," ensuring only one row per primary key value in the final destination. It is the dedup key, not necessarily the database's declared PK — you choose it per stream. For a NAV stream the natural choice is the share class identifier plus the valuation date.

Q: Mechanically, what query does cursor-based incremental sync run each time?
A: It pulls records where `cursor_field > 'last_sync_max_cursor_field_value'` — that is, everything strictly newer than the highest cursor value stored in state. After the sync it advances state to the new maximum cursor it observed. This is why the cursor field must be monotonically updated whenever a row changes.

Q: How does `state` advance during an incremental sync?
A: During each sync the source emits a delta — "the set of records that the source identifies as being new or updated" — and the platform tracks the maximum cursor value seen. On success it persists that new maximum as the stream's state, so the next sync starts just past it. The advance is per stream, so streams progress independently.

Q: A connection is configured incremental but re-fetches every row each sync. What is the first thing to check?
A: Check that a valid cursor field is selected and that the source actually provides it for every record — if no cursor is set, or the field is null, the sync degrades toward pulling everything. Then confirm state is being persisted between runs. A frequent cause is choosing a cursor column that the source does not update on change.

Q: A row was edited at the source but the incremental sync did not pick it up. Why?
A: Because the edit did not bump the cursor field — "if modifications to the underlying records are made without properly updating the cursor field, then the updated records won't be picked up." Since the sync only pulls `cursor_field > last_max`, an update that leaves `updated_at` unchanged is invisible. This is the classic silent-data-loss trap with cursor-based incremental.

Q: What makes a good cursor field?
A: One that is updated on every insert and update, is monotonically non-decreasing, and ideally has reasonable time resolution — an `updated_at` timestamp maintained by the database is the canonical example. A `created_at` is wrong for updates because it never changes after insert. A cursor with coarse resolution can cause boundary records to be missed or duplicated.

Q: Why is a `created_at` column a poor cursor for catching updates?
A: Because `created_at` is set once at insert and never changes, so an updated row keeps an old cursor value and is never re-pulled by a `cursor_field > last_max` query. You would capture new rows but silently miss every edit. For a NAV table where corrections happen, that means restated figures would never propagate.

Q: Why must a cursor field be monotonically increasing?
A: Because state stores a single high-water mark and the next sync fetches only values strictly above it; if the field can go backwards, records with smaller cursor values written after the bookmark advanced are skipped forever. A clock that jumps backward (NTP correction, timezone confusion) can therefore drop records. This is why server-side, monotonic timestamps are preferred over client-supplied ones.

Q: What is a risk of using a low-resolution cursor like a date-only column?
A: Records sharing the same cursor value at the sync boundary can be either missed (if the boundary is exclusive) or re-emitted (if inclusive), because the high-water mark cannot distinguish among same-value rows. With a date-only `nav_date`, many NAV rows share one value, so a mid-day sync boundary is ambiguous. A timestamp with sub-second precision largely avoids this.

Q: How does cursor-based incremental handle deletes at the source?
A: It does not — a deleted row simply stops appearing in future deltas, and nothing instructs the destination to remove it, so the stale row persists. Cursor-based methods typically miss deletions entirely. If a share class is closed and its row deleted in the source, a cursor-based sync leaves it in your destination indefinitely.

Q: What is the practical takeaway about deletes and cursor-based incremental for a fund mart?
A: Closed funds, withdrawn ISINs, or removed holdings will linger in your destination because the cursor never sees the deletion event. You must either run periodic full refreshes, model soft-deletes upstream, or use log-based CDC to capture deletes. Otherwise a "closed" share class quietly stays "open" in downstream reporting.

Q: What is the difference between "CDC-flavored" cursor incremental and true log-based CDC?
A: CDC-flavored incremental infers changes by polling a cursor column and only sees inserts and cursor-bumping updates; true log-based CDC reads the database's transaction log and sees every committed change including deletes. The first is a query-time approximation; the second is a faithful replay of the write-ahead history. Their guarantees differ sharply, especially around deletes and missed updates.

Q: What does log-based CDC read to capture changes?
A: It reads the database's transaction/replication log — "many common databases support writing all record changes to log files for the purpose of replication" — and keeps track of its current position in the log. For Postgres that is the write-ahead log via logical replication; for MySQL it is the binlog. Because it reads committed log entries, it captures changes the application made regardless of any cursor column.

Q: Which databases does Airbyte support for log-based CDC?
A: Airbyte documents log-based CDC support for Postgres, MySQL, Microsoft SQL Server, MongoDB, Oracle DB, SAP HANA, and IBM Db2. Other sources fall back to cursor-based incremental or full refresh. So whether you can use CDC depends on which source system the fund feed comes from.

Q: What can log-based CDC capture that cursor-based incremental cannot?
A: Deletes — CDC emits a metadata column `_ab_cdc_deleted_at` that "is only present for records from DELETE statements," letting the destination know a row was removed. It also captures every update faithfully, independent of whether an application bumped a cursor column. This is the central reason to prefer CDC when deletes and hard updates matter.

Q: What is `_ab_cdc_deleted_at` and when does it appear?
A: It is an Airbyte CDC metadata column that is populated only for records originating from a DELETE statement, marking when the source row was deleted. Its presence lets downstream logic remove or tombstone the corresponding destination row. In dedup mode the final table "will not show the records whose most recent entry has been deleted."

Q: What is a key requirement for CDC incremental on most sources?
A: "CDC incremental is only supported for tables with primary keys for most sources." The primary key is needed to identify which row each log event applies to so updates and deletes can be matched. A source table without a primary key generally cannot be replicated via CDC.

Q: Which source changes does log-based CDC NOT capture?
A: Changes from `TRUNCATE` and `ALTER` statements do not appear in the change log the way row-level DML does, so a truncate or a column type change is not reliably propagated as data events. CDC also requires data in tables, not views. So a schema migration on the source still needs schema-change handling, not just CDC.

Q: Why can't you run CDC against a database view?
A: Because log-based CDC reads row-level changes from the transaction log of base tables, and views have no independent log presence — "data must be in tables, not views." A view's apparent rows are computed from underlying tables at query time and are not written to the log as their own changes. If a fund feed is exposed only as a view, you must point CDC at the underlying tables or use cursor-based incremental.

Q: Why is CDC described as not being treated like an infinite streaming source, and what follows?
A: Airbyte runs CDC as periodic batch reads of the log rather than a continuously open stream, so you must schedule syncs frequently enough that the source's log is not rotated or purged before Airbyte reads it. If the log overflows between runs, CDC can require a costly resync from scratch. For a busy transfer-agency database this means tuning both sync frequency and the source's log retention.

Q: For Postgres specifically, what must be configured to enable log-based CDC?
A: Logical replication must be enabled (for example `wal_level = logical`), and a replication slot and a publication are typically created so Airbyte can read row changes from the WAL. The replication slot pins the WAL until consumed, so an unread slot can cause WAL to accumulate and fill disk. That makes monitoring slot lag an operational must for a CDC pipeline.

Q: What operational hazard does a Postgres replication slot introduce?
A: An active but unread (or slow) slot prevents the WAL from being recycled, so transaction log files pile up and can fill the source database's disk, risking an outage. If an Airbyte CDC connection is paused or broken for too long, the slot keeps holding WAL. You must monitor slot lag and clean up abandoned slots — a real "self-hosting maintenance" cost.

Q: When should you choose CDC over cursor-based incremental for a fund source?
A: Choose CDC when you need deletes captured, when updates may not bump a reliable cursor column, or when you need near-real-time low-latency change capture. Choose cursor-based when the source lacks CDC support, lacks primary keys, or when a clean `updated_at` exists and deletes are handled by soft-delete flags. The decision is about which guarantees the downstream NAV/holdings model actually needs.

Q: What is schema-change detection in Airbyte?
A: It is the platform re-running `discover` and comparing the new source catalog against the stored one to spot added, removed, or retyped streams and fields. Detection is configured per connection in its Settings tab. It is the first half of schema evolution; what to do about a detected change is the second half.

Q: How often does Airbyte check for schema changes?
A: For Cloud, it checks immediately before a sync, at most every 15 minutes per source; for self-managed deployments, at most every 24 hours. The cadence differs because cloud can afford frequent discovery while self-hosted minimizes load on the source. So a self-hosted fund deployment may not notice an upstream column change until the next daily check.

Q: What are the propagation options for detected schema changes?
A: "Propagate field changes only" applies field additions/changes but ignores new or removed streams; "Propagate all field and stream changes" applies both field and stream changes automatically. There are also options to ignore changes or to disable the connection on certain changes. You pick the policy per connection based on how much drift you tolerate.

Q: What is the difference between "Propagate field changes only" and "Propagate all field and stream changes"?
A: "Propagate field changes only" automatically applies new or altered fields but leaves new and removed streams out of the sync; "Propagate all field and stream changes" additionally brings in new streams and removes dropped ones. The first is conservative — you opt into new tables manually; the second keeps the destination fully in step with the source. Regulated estates often prefer the conservative setting so a new upstream table is reviewed before it lands.

Q: What is the distinction between breaking and non-breaking schema changes?
A: Non-breaking changes (a new field, a widened type) can be auto-propagated to the next sync; breaking changes (such as a removed or retyped cursor or primary key) are not auto-applied because they would invalidate the sync. Breaking changes pause the connection and require a human to review and refresh the configuration. So the breaking/non-breaking line is essentially "can the existing sync still run safely?"

Q: For a non-breaking change, what options does Airbyte give beyond propagating?
A: For non-breaking schema changes you can configure the connection to "Ignore" the change (sync proceeds, change not applied) or to "Disable connection" (sync stops until reviewed). This lets you trade automation against control. A fund platform that wants every new column reviewed might choose ignore-or-disable rather than auto-propagate.

Q: What happens when a brand-new column appears in a source table?
A: It is detected as a non-breaking field change; depending on policy it is auto-propagated into the stream and destination, ignored, or it pauses the connection for review. If propagated, future syncs include the new field and the destination schema is extended. So a new `swing_factor` column on a NAV table can flow through automatically or wait for sign-off, by your choice.

Q: What typically makes a column type change a breaking issue?
A: If the retyped column is the cursor field or part of the primary key, the change can break incremental/dedup logic and is treated as breaking, pausing the connection. A general field retype may be non-breaking and widened at the destination, but a narrowing or incompatible change can cause load errors. The safe move is to review type changes on key columns before resuming.

Q: Why is removing or renaming the cursor column especially dangerous?
A: Because the incremental query depends on `cursor_field > last_max`; if the column vanishes or is renamed, the sync loses its bookmark mechanism and is flagged as breaking. Resuming usually requires reselecting a cursor and may force a full re-sync. This is exactly the kind of "break the cursor on purpose, then repair it" scenario in the lesson's Do steps.

Q: After a breaking schema change is resolved, what does a re-sync often involve?
A: Often a full refresh of the affected stream to rebuild a consistent state, because the old cursor/PK basis is no longer valid. Airbyte may reset the stream's state so the next sync re-reads everything before resuming incremental. For a large holdings table that re-sync is a real cost, which is why breaking changes are gated behind human review.

Q: What does "resetting a stream" do, and when would you do it?
A: Resetting clears the destination data and the stored state for that stream so the next sync starts clean as a full load, then resumes incrementally. You do it after a breaking schema change, a cursor change, or to recover from corrupted/duplicated data. It is the standard repair when a cursor has been broken or mis-selected.

Q: A connection's sync now errors after the source added a NOT-NULL column with no default. First thing to check?
A: Check the schema-change status on the connection: detection likely flagged the new field, and your propagation policy determines whether it was applied or the connection paused. Confirm whether the destination accepted the new column and whether any load constraints rejected nulls for historical rows. The fix is to propagate the schema and backfill or relax the constraint, not to keep retrying the same failing sync.

Q: Why is "the hidden maintenance line item" the real cost of self-hosting connectors?
A: Because the software is free but keeping it running is not — a self-hosted deployment typically needs ongoing engineering time to patch connectors, handle schema changes, and chase failures, with no vendor support desk. That labor, at a loaded engineering rate, can exceed the price of a managed service for many teams. The whole point of running Airbyte yourself in this lesson is to feel that cost concretely.

Q: Roughly how much ongoing effort does a self-hosted Airbyte deployment tend to require?
A: Reported figures are on the order of 1–5 hours of engineering maintenance per week depending on connector count and schema stability, which at a fully loaded rate translates into hundreds to a few thousand dollars per month in labor. The variance is driven mostly by how often upstream schemas drift and how flaky the connectors are. That is the line item a managed vendor like Fivetran absorbs for you.

Q: Why does connector reliability vary in an open-source connector ecosystem?
A: Because many connectors are community-contributed and differ in maturity and upkeep, whereas a managed vendor staffs a large dedicated team maintaining its connectors around the clock. A long-tail SaaS connector you depend on may lag API changes or carry bugs. That risk is the maintenance you are insourcing when you self-host.

Q: What are you "really paying Fivetran to absorb" when you choose managed over self-hosted?
A: Connector maintenance, schema-change handling, infrastructure operation, monitoring, and on-call response to breakages — the operational toil rather than the bytes moved. Fivetran prices that as a managed service; you trade money for not staffing it. Running Airbyte yourself once is how you learn the size of that toil before signing a vendor contract.

Q: When does self-hosted Airbyte clearly make economic or compliance sense despite the maintenance cost?
A: When you have spare engineering capacity, when sources must stay inside a data-residency perimeter, or when the workload is core/on-prem data that cannot be sent to a managed cloud. For a Luxembourg fund administrator, EU data-sovereignty constraints on core banking and transfer-agency feeds can make self-hosting mandatory regardless of cost. The cost question is then secondary to the legal one.

Q: How do you stand up Airbyte OSS for this lesson's lab?
A: You run it locally in Docker Compose (or via the `abctl` local install), which brings up the platform services so you can configure sources, destinations, and connections through the web UI. The lesson's first Do step is exactly "stand up Airbyte in Docker Compose." On native Ubuntu this is straightforward with Docker installed; WSL2 would be the Windows equivalent and Docker Desktop on macOS.

Q: For the lab, how would you configure an incremental Postgres-to-MinIO Parquet sync with a cursor?
A: Create a Postgres source pointing at the Phase-1 database, a destination writing Parquet to a MinIO bucket, then a connection selecting the relevant stream with sync mode incremental and a cursor field such as `updated_at` (and a primary key if using dedup). Run an initial sync, then an incremental one. The goal is to confirm that the second sync moves only changed rows.

Q: How do you prove an incremental sync moved only changed rows?
A: Run an initial sync, change a small known set of source rows (and bump their cursor), then run a second sync and inspect that only those rows landed — the connection's sync logs show record counts and the destination shows only the delta. Comparing emitted record counts between the two runs makes it concrete. This satisfies the "demonstrate an incremental sync moving only changed rows" done-when criterion.

Q: The lesson asks you to "break the cursor on purpose." What is a clean way to do that, and how do you repair it?
A: Alter or reset the cursor column at the source (or change which column is the cursor) so the sync loses its bookmark, then observe the failure or full re-pull; repair it by reselecting a valid cursor field and resetting the stream's state to rebuild a clean baseline. The exercise teaches what breaks and what the recovery costs. Being able to explain what broke is an explicit done-when criterion.

Q: Why might you choose dedup over plain incremental append for a silver NAV table?
A: Because silver should expose one conformed row per share class per valuation date, and dedup keeps the final table unique per primary key while still preserving change history in the raw/history table. Plain append would leave multiple NAV rows per key, pushing the dedup burden downstream. Dedup puts the one-row-per-key contract right at the load boundary.

Q: When is plain incremental append actually the better choice for fund data?
A: When you want a full, immutable change history — for example every restated NAV value over time — rather than only the latest. Append preserves each version as its own row, which is useful for audit reconstruction and bronze immutability. You can then derive a deduped view downstream in dbt when you need the current value.

Q: How does Airbyte's incremental loading relate to the bronze layer of a medallion lakehouse?
A: Incremental append fits bronze's "immutable, replayable" contract by accumulating raw deltas without overwriting, preserving everything the source emitted. Dedup or conformance belongs to silver, not bronze. So you typically land raw incremental Parquet in bronze, then conform and deduplicate in a downstream transform.

Q: Why is full refresh overwrite risky for capturing source deletes versus convenient for small dimensions?
A: It is convenient because it naturally reflects deletes — anything gone from the source is gone from the destination after the reload — but risky because it discards history and re-reads the whole table every run. For a tiny reference dimension (a few hundred fund codes) that is fine; for a large holdings table it is expensive and loses any audit trail. The mode choice is a cost-versus-history tradeoff.

Q: A nightly incremental sync of NAV data is missing some same-day corrections. What is the likely cause?
A: The corrections probably did not advance the cursor (or share the cursor value at the sync boundary), so the `cursor_field > last_max` query skipped them. Check that corrections update `updated_at` and consider a higher-resolution cursor or a periodic full refresh to catch stragglers. This is the cursor-bump trap applied to NAV restatements.

Q: Why might a transfer-agency feed that does hard deletes need CDC rather than cursor incremental?
A: Because cursor-based incremental never sees deletes, so redeemed or closed positions deleted at the source would persist in your destination and distort holdings. Log-based CDC captures the delete via `_ab_cdc_deleted_at`, letting you tombstone the row. If the source supports CDC and has primary keys, that is the correct tool for accurate position data.

Q: How does Airbyte's role as the E and L map onto an ELT architecture with dbt?
A: Airbyte extracts from sources and loads raw records into the destination (bronze) with minimal transformation; dbt then transforms in-warehouse to produce silver and gold models. This separation is why Airbyte focuses on connectors, sync modes, and state rather than business logic. The clean handoff keeps transformation in version-controlled SQL rather than buried in the ingestion tool.

Q: Why is incremental sync important for cost and load on a fund source database?
A: Because pulling only changed rows minimizes load on the operational source, network transfer, and destination write volume, which matters when the source is a live transfer-agency or accounting database you must not hammer. A full refresh of a large holdings table every hour could degrade the OLTP system. Incremental keeps ingestion light enough to run often.

Q: What is the relationship between a connection's schedule and incremental state?
A: The schedule decides how often the connection runs; each run reads the persisted state, fetches records past the cursor, loads them, and writes back the advanced state. More frequent schedules mean smaller deltas per run and lower latency. State is what makes those frequent small runs correct rather than redundant.

Q: If two connections share the same source, do they share state?
A: No — state is tracked per stream within a connection, so two connections each maintain their own bookmarks even against the same source table. This lets you sync the same stream to two destinations independently. It also means resetting one connection's stream does not reset the other's.

Q: What does it mean operationally that state is stored by the platform, not the source?
A: It means resumability is the platform's responsibility: Airbyte persists the cursor high-water mark in its internal database, so losing or corrupting that store loses your incremental position. A platform database backup is therefore part of protecting your pipelines. Restoring a stale state could cause re-pulls or, worse, skipped records if the store is ahead of reality.

Q: Why might a sync that was incremental suddenly perform a full refresh on its own?
A: Common triggers are a stream reset, a breaking schema change that invalidated the cursor/PK, a changed cursor field, or lost/cleared state in the platform database. Any of these removes the valid bookmark and forces a fresh full load before incremental resumes. Diagnosing which one happened is the first step before worrying about performance.

Q: What does "append vs dedup" mean for downstream consumers querying the destination?
A: With append, consumers must themselves pick the latest version per key (e.g., a window function on the cursor) because multiple rows per entity exist; with dedup, the final table already holds one row per primary key. Append pushes correctness work downstream; dedup centralizes it at load time. Choosing wisely avoids every analyst re-implementing the same "latest NAV per share class" logic.

Q: In dedup mode, what is the purpose of the history (raw) table behind the final table?
A: It retains the appended delta of all changes so the deduped final table can be rebuilt and so change history is not lost; the final table is derived from it by keeping the latest record per primary key. This gives you both a single-row current view and a full audit trail. For NAV corrections, the history table is where the superseded values still live.

Q: Why must the primary key you choose for dedup truly be unique per entity?
A: Because dedup collapses all rows sharing a primary key to the latest one; if the key is not actually unique per entity, legitimately distinct records merge and data is lost. Picking (`share_class_isin`) alone for a NAV stream is wrong because the same ISIN recurs every valuation date — you need the date too. Composite keys exist precisely for this.

Q: How do you decide the dedup primary key for a NAV-per-share-class stream?
A: The grain is one value per share class per valuation date, so a composite key of the share-class identifier (e.g., ISIN or an internal surrogate) plus `nav_date` is correct. Using only the identifier would overwrite every prior day's NAV. Getting the grain right is the same modeling discipline you applied to fact tables in Phase 1.

Q: Why might you prefer an internal surrogate over the ISIN as a dedup/join key in fund data?
A: Because ISINs can be reused or reassigned by corporate actions, so keying on ISIN alone can merge or split instruments over time and break continuity. An internal surrogate stays stable across identifier lifecycle events. This is the same security-master argument from the LEI/ISIN lesson, applied at the ingestion layer.

Q: What happens to your incremental sync if the source's `updated_at` is set by application code that sometimes forgets to bump it?
A: Those un-bumped updates are silently skipped by the `cursor_field > last_max` query, so the destination drifts out of sync with the source for exactly the edited rows. Because nothing errors, the drift is invisible until someone notices stale data. The fix is a reliable, database-maintained `updated_at` (or CDC), not trusting application discipline.

Q: How would you detect that incremental drift has occurred without an error being raised?
A: Periodically reconcile counts and checksums between source and destination, or run an occasional full refresh and diff it against the incremental result. Row-count or hash mismatches reveal records the cursor missed. Building such a reconciliation check is part of treating ingestion as a data-quality boundary in a regulated platform.

Q: Why is a periodic full refresh sometimes used alongside incremental sync?
A: To self-heal drift from missed updates, missed deletes, or cursor gaps by occasionally reloading the whole table and reconciling it. It trades extra load for correctness guarantees that pure incremental cannot give. A common pattern is frequent incremental with a weekly full refresh of critical tables like NAV.

Q: What is the tradeoff between sync frequency and source impact for CDC versus cursor incremental?
A: CDC must run frequently enough that the source log is not purged before it is read, so very infrequent CDC risks a costly resync; cursor incremental can run less often but with bigger deltas and higher latency. CDC trades steady low-latency log reading against the need for adequate log retention. You tune frequency to the source's log/WAL retention and your latency needs.

Q: Why does CDC require careful coordination with the source DBA in a fund shop?
A: Because enabling logical replication, sizing log/WAL retention, granting replication privileges, and managing replication slots all touch the production source database that operations depends on. A mis-managed slot can fill disk and threaten an outage on a system running NAV calculations. So CDC is as much an operational agreement as a technical setting.

Q: What is the connection-level "Settings" tab used for regarding schema changes?
A: It is where you set the detection-and-propagation policy — which kinds of changes auto-apply, which are ignored, and whether non-breaking changes should disable the connection. The setting is per connection, so different pipelines can have different drift tolerance. A strict fund pipeline can disable on change while a sandbox auto-propagates.

Q: Why might a regulated fund platform prefer "ignore" or "disable" over auto-propagating schema changes?
A: Because unreviewed schema changes can silently alter what lands in a regulated dataset, undermining lineage and auditability — an auditor expects deliberate, reviewed evolution. Pausing on change forces a human to assess and document the impact before data flows. The slight operational friction is the price of a defensible audit trail.

Q: How does schema-change handling interact with the cursor field specifically?
A: A change to the cursor field — removal, rename, or incompatible retype — is typically breaking because it invalidates incremental state, so it pauses the connection rather than auto-propagating. Adding unrelated fields is non-breaking and can flow through. This is why the cursor and primary key columns deserve special change-control attention.

Q: What is the first diagnostic step when an Airbyte connection turns red after an upstream release?
A: Inspect the connection's sync logs and schema-change status to see whether a detected schema change (especially breaking) paused it, or whether `check` now fails on credentials/connectivity. The status usually names the cause directly. Reading the log beats blindly retrying, which just re-hits the same wall.

Q: A self-hosted connector silently stopped fetching new rows after a source API version bump. What is the likely root cause and remedy?
A: The community connector probably lags the new API version, so its extraction no longer matches the source — a classic self-hosted maintenance hit. The remedy is to upgrade the connector image (or patch/fork it) and re-test, which is exactly the unbudgeted labor managed vendors absorb. It illustrates why connector currency is an ongoing operational cost.

Q: How does the connector model (source/destination/spec/state) make Airbyte extensible to new systems?
A: Because any new system only has to implement the same protocol contract — declare a `spec`, support `check`/`discover`/`read` or `write`, and emit/consume `state` — the platform drives it without bespoke orchestration code. New connectors slot into the same engine. This uniformity is what lets a connector library scale to hundreds of sources.

Q: Summarize when batch ELT connectors are the right ingestion choice versus streaming or hand-coded extraction.
A: Batch ELT connectors fit scheduled, non-streaming ingestion of database tables, files, and SaaS APIs where minute-to-hour latency is acceptable and you value pre-built connectors over custom code. Streaming (Kafka/CDC-to-stream) fits sub-second event flows; hand-coded extraction fits one-off or highly bespoke sources. For most fund reference, NAV, and holdings feeds, scheduled incremental batch ELT is the pragmatic default — which is exactly why this lesson makes you run one.

Q: On your native Ubuntu laptop, what is the simplest supported command to install Airbyte OSS locally without hand-writing a compose file?
A: Use the `abctl` CLI: run `abctl local install`, which provisions a local Kubernetes (kind) cluster and deploys the platform, then exposes the UI on port `8000` by default. You can pick another port with `abctl local install --port <PORT>` if `8000` is taken. After it finishes, fetch the login with `abctl local credentials` rather than guessing the default user.

Q: After `abctl local install`, you do not know the web UI login. Which command prints it?
A: Run `abctl local credentials`, which displays the auto-generated email/password (and the instance's client ID/secret) for the local Airbyte instance. The credentials are generated at install time, so you cannot assume a fixed default. If you ever rotate them, this is also the command to re-read the current values.

Q: You want to tear down a local `abctl` install but keep your configured sources and connections. Which command, and what is the gotcha?
A: Run `abctl local uninstall`, which removes the running instance but deliberately leaves the underlying data so you can reinstall later with everything intact. The gotcha is that to actually wipe the data you must add the data-removal flag (e.g. `--persisted`), and that deletion cannot be undone. So a plain uninstall is safe and reversible; the destructive variant is not.

Q: `abctl local install` hangs or fails partway on Ubuntu — what are the first two things to check?
A: First confirm the Docker daemon is running and your user can talk to it (`docker ps` should work without `sudo`), because `abctl` builds its kind cluster on Docker. Second check that the chosen port (`8000` by default) is free and that you have enough disk and memory, since the install pulls large images and can take up to ~30 minutes. A stale or half-created cluster from a prior failed run may also need `abctl local uninstall` before retrying.

Q: What three platform-added metadata columns does an Airbyte Destinations-V2 sync attach to every record, regardless of source?
A: `_airbyte_raw_id` (a UUID Airbyte assigns to each record), `_airbyte_extracted_at` (a timestamp of when the record was pulled from the source), and `_airbyte_meta` (a structured object holding per-record metadata such as typing changes). A newer `_airbyte_generation_id` integer also increments on each refresh. These always exist in the output, so your bronze schema must expect them alongside the business fields.

Q: What goes into the `_airbyte_meta.changes[]` array, and why is that useful for a regulated NAV feed?
A: When a source value cannot be coerced into its destination type, Airbyte nulls the value but records the problem in `_airbyte_meta.changes[]` rather than failing the whole sync. For a NAV feed this means a single malformed `nav_per_share` does not block the night's load, yet you retain an auditable trail of exactly which fields were dropped. You can query `_airbyte_meta` downstream to surface and triage those coercion failures.

Q: How does Airbyte's current "Typing and Deduping" differ from the old "Basic Normalization" it replaced?
A: Typing and Deduping (Destinations V2) expands the raw JSON into typed final-table columns and keeps one row per primary key, but it does not flatten nested objects the way Basic Normalization tried to. It also routes type-coercion failures into `_airbyte_meta` instead of erroring the sync. The practical shift is that deeply nested structures are now your job to unpack in dbt, not the connector's.

Q: In a typing-and-deduping destination, where do the raw records live versus the final table?
A: The connector first lands untyped JSON in a raw table (the `_airbyte_raw_` layer), then derives a typed, deduplicated final table from it. The raw layer preserves everything as received, while the final table holds one typed row per primary key. This two-tier structure is what lets a corrupted final table be rebuilt by re-running typing against the retained raw data.

Q: Name the three Airbyte CDC metadata columns added on a Postgres CDC source and what each means.
A: `_ab_cdc_lsn` records the WAL log-sequence-number position where the change was read, `_ab_cdc_updated_at` is the transaction timestamp present on every INSERT/UPDATE/DELETE, and `_ab_cdc_deleted_at` is populated only for DELETE events. Together they let the destination order changes and distinguish deletes from upserts. For holdings data, `_ab_cdc_deleted_at` is what tells you a position was redeemed and removed at source.

Q: A Full Refresh stream from a Postgres source still shows `_ab_cdc_*` columns even though you are not running CDC. Is that a bug?
A: No — Airbyte includes the CDC metadata columns by default on Postgres-source full-refresh tables too, even when CDC is not the replication method, so their presence alone does not mean changes are being captured incrementally. You should treat those columns as informational, not as proof of delete capture. Only an actual CDC incremental sync populates `_ab_cdc_deleted_at` meaningfully.

Q: Which replication plugin does Airbyte require for Postgres CDC, and how do you create the slot?
A: Airbyte requires the `pgoutput` logical-decoding plugin, and you create a dedicated slot with `SELECT pg_create_logical_replication_slot('airbyte_slot', 'pgoutput');`. The slot must be reserved for Airbyte's exclusive use — sharing it across sources corrupts each consumer's log position. This is why the slot name is configured per connection and should not be reused.

Q: Beyond `wal_level = logical`, which two Postgres settings commonly need raising for CDC, and why?
A: `max_replication_slots` and `max_wal_senders` must be high enough to accommodate Airbyte's slot and sender, or slot creation/replication connection fails. They cap how many logical slots and WAL-sender processes the server allows concurrently. On a shared fund database already feeding other replicas, the defaults can be exhausted, so you size these with the DBA.

Q: Setting `wal_level = logical` on the source has one property worth flagging to the DBA — what is it?
A: Once `wal_level` is raised to `logical` it cannot simply be silently reverted without a server restart and consideration of all dependent replication, and it increases the volume of WAL the server writes. The extra WAL means more disk churn on a busy NAV-calculation database. So enabling logical replication is a deliberate, reviewed change, not a toggle to flip casually on production.

Q: What is a Postgres `REPLICA IDENTITY` and why does Airbyte CDC care about it?
A: `REPLICA IDENTITY` controls which old-row columns Postgres writes to the WAL for UPDATE and DELETE events; without an adequate identity, the log lacks the key needed to match a change to a destination row. Airbyte (or the role it uses) needs each replicated table to have a usable identity, typically the primary key. A table set to `REPLICA IDENTITY NOTHING` will not yield workable UPDATE/DELETE events.

Q: Write the query you would run to measure how far behind an Airbyte logical replication slot is.
A: Use `SELECT slot_name, active, pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), confirmed_flush_lsn)) AS slot_lag, wal_status FROM pg_replication_slots WHERE slot_type = 'logical';`. The `pg_wal_lsn_diff` between the current WAL position and the slot's `confirmed_flush_lsn` is the unconsumed lag in bytes. A growing lag plus a non-`reserved` `wal_status` is the early warning that WAL is piling up.

Q: In `pg_replication_slots`, what do `restart_lsn` and `wal_status` tell you operationally?
A: `restart_lsn` is the oldest WAL position the slot still needs, so it is the floor below which WAL cannot be recycled; a `wal_status` of `lost` means required WAL has already been removed and the slot can no longer resume. Watching `restart_lsn` falling far behind `pg_current_wal_lsn()` warns of disk-fill risk before it happens. A `wal_status` other than `reserved`/`extended` is a signal to act immediately.

Q: An Airbyte Postgres CDC connection was paused for two weeks and now errors on resume. What likely happened and how do you recover?
A: The slot's required WAL was probably purged while it sat unconsumed, so `wal_status` became `lost` and Airbyte can no longer continue from its last LSN. Recovery means resetting the stream to force a fresh snapshot (full re-read) and recreating/cleaning the slot, accepting the resync cost. The lesson here is to monitor slot lag and avoid leaving CDC connections paused.

Q: Your Postgres disk is filling and you trace it to an abandoned Airbyte slot. What is the safe cleanup command?
A: Drop the orphaned slot with `SELECT pg_drop_replication_slot('airbyte_slot');` after confirming no live connection still needs it, which releases the WAL it was pinning. You must be certain the connection is truly dead, because dropping a slot in use breaks that pipeline and forces a resync. This is exactly the abandoned-slot maintenance burden that self-hosting puts on you.

Q: How would you prove, at the destination, that a source DELETE was actually captured by CDC?
A: Inspect the affected record in the destination and confirm `_ab_cdc_deleted_at` is populated for it; in a dedup final table the row whose most recent entry is a delete is removed entirely. So you verify either the tombstone column in the history/raw layer or the row's absence in the final table. If `_ab_cdc_deleted_at` is always null after deleting source rows, CDC is misconfigured.

Q: A NAV row deleted at source still appears in your destination and `_ab_cdc_deleted_at` is null. First thing to check?
A: Confirm the connection is genuinely running log-based CDC and not cursor-based incremental, because cursor incremental never emits deletes and leaves `_ab_cdc_deleted_at` empty. Then verify the source table's `REPLICA IDENTITY` exposes the key and that the slot/publication actually include that table. A common cause is a connection mislabeled "CDC" that is really polling a cursor column.

Q: How do you decide between Full Refresh Overwrite and dedup for the small `funds` reference dimension specifically?
A: For a few-hundred-row `funds` table that changes rarely and where you only need the current state, Full Refresh Overwrite is simpler and naturally reflects deletes, with negligible reload cost. Dedup is only worth its cursor/PK overhead when the table is large or you need change history at the load boundary. Match the mode to the table's size and history requirement, not a blanket policy.

Q: Compare incremental append vs incremental dedup purely from the destination storage-growth angle.
A: Incremental append stores every changed version as a new row, so storage grows with the change rate and the table accumulates indefinitely; dedup keeps a bounded final table of one row per key while still retaining history in the raw layer. Append therefore trades disk for a complete in-line audit trail. For high-churn NAV restatements, append can balloon faster than people expect.

Q: A stakeholder wants "the latest NAV per share class" from an append stream. What SQL pattern do you give them, and why is dedup nicer?
A: Use a window function such as `ROW_NUMBER() OVER (PARTITION BY share_class_isin, nav_date ORDER BY updated_at DESC)` and keep `row_num = 1`, because append holds multiple rows per key. Dedup is nicer because the connector already collapses to the latest row, so every analyst is spared re-implementing that window. Pushing this logic to load time avoids inconsistent hand-rolled "latest" queries across teams.

Q: For a Postgres-to-MinIO Parquet pipeline, why is dedup at the destination sometimes not available, forcing append?
A: A blob/object destination writing raw Parquet files may not support the upsert/merge needed for deduped final tables, so the available sync modes are the intersection of source and destination capabilities and exclude dedup. You then land append-style raw Parquet in bronze and deduplicate downstream in dbt. Checking `supported_destination_sync_modes` upfront tells you whether dedup is even on the table.

Q: Your incremental sync to MinIO writes a new Parquet file each run. How do you reconcile that with bronze's "immutable, replayable" contract?
A: Treat each run's Parquet output as an immutable append — never rewrite prior files — so bronze accumulates the full delta history exactly as the source emitted it. Conformance and dedup happen in a downstream silver transform, not by mutating bronze. This is why incremental append (not overwrite) is the natural bronze loading mode for object storage.

Q: You changed a cursor field from `created_at` to `updated_at` on a running connection. What does Airbyte typically force, and why?
A: Changing the cursor field invalidates the existing state bookmark, so Airbyte generally treats it as requiring a stream reset and a full re-read before incremental resumes against the new cursor. The old high-water mark is meaningless for the new column, so continuing would skip or duplicate rows. Plan for the full reload cost when you correct a badly chosen cursor.

Q: Concretely, how do you "break the cursor on purpose" in the lab and observe the failure?
A: At the source, either drop/rename the cursor column (e.g. `ALTER TABLE nav DROP COLUMN updated_at`) or reset its values so the high-water mark no longer advances, then run a sync and watch it either error on the missing field or silently stop pulling edits. The connection's schema-change status or sync logs name what broke. Repair by reselecting a valid cursor and resetting the stream's state.

Q: After repairing a broken cursor by resetting the stream, what evidence confirms a clean baseline was rebuilt?
A: The reset clears destination data and state, so the next sync's logs should show a full record count (a complete re-read) rather than a small delta, and subsequent syncs should drop back to small deltas. Comparing the post-reset run's emitted record count to source row count confirms completeness. Seeing deltas shrink again proves incremental has correctly resumed from the new bookmark.

Q: Where in the Airbyte UI do you trigger a stream reset, and what exactly does it clear?
A: You trigger it from the connection's status/streams view by choosing to clear or reset the stream's data, which removes the destination's data for that stream and the stored incremental state. The next sync then runs as a full load before resuming incremental. It does not delete the source — only the destination copy and the platform's bookmark.

Q: Two connections replicate the same `nav` table to two destinations; you reset one. What happens to the other?
A: Nothing — state is tracked per stream within each connection, so resetting one connection's `nav` stream does not touch the other connection's bookmark or destination data. Each maintains its own high-water mark independently. This isolation is also why the same stream can sync to MinIO and to Postgres at different cadences without interfering.

Q: An Airbyte `check` succeeds but `discover` returns no streams for your fund Postgres. What is the usual cause?
A: The connecting role can authenticate (so `check` passes) but lacks `USAGE`/`SELECT` on the target schema or tables, so `discover` finds nothing to enumerate. Schema/search-path scoping or a wrong schema name also yields an empty catalog. Grant the role read access to the intended schema and re-run discover.

Q: `check` fails immediately on a self-hosted source while psql connects fine from your laptop. First hypothesis?
A: Network reachability from inside the Airbyte containers/cluster differs from your laptop — `localhost`/`127.0.0.1` inside a container is the container itself, not your host Postgres. Use the host's reachable address (e.g. `host.docker.internal` or the LAN IP) and ensure Postgres listens and `pg_hba.conf` permits that source. The fix is networking/config, not sync logic.

Q: How would you make incremental ingestion a data-quality boundary for a regulated NAV pipeline?
A: Periodically reconcile source-vs-destination row counts and column checksums (or run an occasional full refresh and diff it) to catch records the cursor silently missed. Surface `_airbyte_meta.changes[]` to flag type-coercion losses, and alert on any mismatch. Treating each sync as a checkpoint with reconciliation is what makes silent cursor drift visible before it reaches reporting.

Q: A weekly full refresh of `nav` was added to self-heal drift, but it now overwrites this week's incremental rows with stale source data. What went wrong?
A: The full refresh likely points at a source that lags the incremental feed, or it runs Overwrite mode and clobbers fresher rows because the refresh and incremental write the same destination table. Separate the reconciliation refresh into its own table/connection for diffing, rather than overwriting the live incremental table. The refresh should validate, not replace, the authoritative incremental data.

Q: Why is a date-only `nav_date` cursor especially dangerous when intraday NAV corrections arrive?
A: With date-only resolution, many rows share one cursor value, so a sync whose boundary lands on today cannot distinguish corrections already pulled from ones still pending, risking skipped or re-pulled rows at the boundary. A correction posted after the boundary but with the same date may never be seen. A sub-second `updated_at` timestamp resolves the ambiguity.

Q: An application-maintained `updated_at` is sometimes not bumped on correction. What two-pronged fix do you propose?
A: Move cursor maintenance off the application: add a database trigger (or `DEFAULT now()` plus `ON UPDATE`-style logic) so `updated_at` is set on every write, or switch to log-based CDC which ignores cursor columns entirely. The database-enforced timestamp removes reliance on application discipline. Either way you stop trusting code that "sometimes forgets."

Q: Why might you key dedup on an internal surrogate plus `nav_date` rather than `share_class_isin` plus `nav_date`?
A: Because ISINs can be reassigned or reused across corporate actions, so keying on ISIN risks merging two distinct instruments or splitting one over its identifier lifecycle. A stable internal surrogate preserves entity continuity through identifier changes. This mirrors the security-master discipline: resolve to a durable internal key, carry ISIN as an attribute.

Q: A LEI on the issuing-entity reference stream lapsed and was reissued. Why does that argue against using LEI as a dedup primary key?
A: An `LEI` can change status (lapse) and a successor identifier can appear, so keying entity rows on `LEI` risks fragmenting one legal entity's history across identifier values. Dedup would treat the new LEI as a different entity and lose continuity. Use a stable internal entity key and treat `LEI` as a versioned attribute instead.

Q: Self-managed Airbyte only checks for schema changes at most every 24 hours. What does that mean for a same-day upstream column addition?
A: Self-hosted schema detection runs at most once every 24 hours (versus as often as every 15 minutes on Cloud), so a column added this morning may not be noticed until the next daily discover. Until then the new field is simply not in your configured catalog and is not synced. For time-sensitive regulatory fields you may need to trigger discovery manually rather than wait.

Q: Under "Propagate field changes only", a brand-new upstream `eet_field` table appears. Does it start syncing automatically?
A: No — "Propagate field changes only" applies field additions and changes but deliberately ignores new and removed streams, so a new table is detected but not added to the sync until you enable it. This is the conservative default a regulated estate prefers, so a new EET/EPT extract is reviewed before it lands. To auto-add streams you would need "Propagate all field and stream changes".

Q: Why might a regulated Luxembourg fund platform set non-breaking schema changes to "Disable connection" rather than auto-propagate?
A: Because auto-propagation can silently alter what lands in a regulated dataset, undermining lineage and the deliberate, reviewed evolution an auditor expects under UCITS/AIFMD record-keeping. Disabling on change forces a human to assess, document, and approve the impact before data flows. The operational friction is the price of a defensible audit trail.

Q: A retype of the `nav_per_share` column from `numeric` to `text` arrives. Breaking or not, and what is the practical risk?
A: A general field retype is usually non-breaking unless that column is the cursor or part of the primary key, so the sync continues but the destination may widen the type or null un-coercible values into `_airbyte_meta`. The practical risk is silent loss of precision or values on a money field, so you should review numeric-to-text changes before resuming. A breaking flag only fires if `nav_per_share` were the cursor/PK.

Q: A breaking schema change paused your `holdings` connection overnight, missing the T+1 settlement cut-off. How do you reason about the recovery cost?
A: A breaking change (e.g. the cursor/PK column removed or retyped) requires human review, reconfiguration, and often a full stream reset and re-read before incremental resumes, which on a large `holdings` table is slow. Under T+1 that delay can push settlement data past its cut-off, so you weigh an emergency full refresh against waiting. The episode is exactly why key columns deserve strict change-control on settlement-critical feeds.

Q: How does a self-hosted Airbyte's reliance on community connectors create a hidden T+1 operational risk?
A: Community connectors can lag upstream API or schema changes, so a connector that silently stops fetching after a source bump may go unnoticed until reporting breaks — and under T+1 there is little slack to catch up. The remedy is upgrading or patching the connector image, which is unbudgeted labor. Vendors like Fivetran absorb exactly this currency risk that tight settlement windows make costly.

Q: Quantify, roughly, the self-hosting maintenance you are insourcing versus a managed vendor, and what drives the variance.
A: Self-hosted Airbyte typically consumes on the order of 1–5 hours of engineering time per week — patching connectors, handling schema drift, chasing failures — which at a loaded rate is hundreds to a few thousand dollars per month. The variance is driven mainly by how often upstream schemas change and how flaky the connectors are. That recurring toil is the line item a managed service prices in for you.

Q: When does EU data residency make self-hosted Airbyte the correct choice for a Luxembourg fund administrator despite the maintenance cost?
A: When core banking, accounting, or transfer-agency feeds must remain inside an EU/Luxembourg data perimeter, self-hosting keeps the data plane on-prem or in-region, which a managed cloud may not guarantee. Under that constraint the legal/residency requirement dominates the cost comparison. DORA's third-party and resilience expectations further reinforce keeping critical pipelines under direct control.

Q: For a connector handling personal investor data, what DORA/governance-flavored practice should wrap your incremental state store?
A: Because Airbyte persists incremental state (cursor high-water marks) in its own internal database, that store must be backed up and access-controlled like any critical system — losing or corrupting it loses your replication position and can cause re-pulls or skipped records. Under DORA's resilience expectations, a tested restore of the platform DB is part of protecting the pipeline. Treat the state store as in-scope for backup, monitoring, and recovery drills.

Q: End-to-end, walk through what one incremental dedup sync does from trigger to advanced state.
A: On schedule the connection reads the stored per-stream state, queries the source for `cursor_field > last_max`, emits the delta as records, lands them in the raw layer, then types-and-dedups into a final table that is unique per primary key. On success it persists the new maximum cursor as the stream's state so the next run starts just past it. Deletes (if CDC) are tombstoned via `_ab_cdc_deleted_at` and removed from the final table.

Q: A connection completes with `succeeded` status but the destination got zero rows on a first sync. What are the prime suspects?
A: For an incremental stream, an inherited or pre-seeded state whose cursor is already at/above the source maximum will make the very first sync a no-op, so check whether state was carried over from a prior connection. Also verify the stream is actually enabled in the configured catalog and that the source table is non-empty for the selected schema. A "succeeded but empty" outcome is almost always state or stream-selection, not a transport failure.

Q: How can you cap the source-side load of an initial Airbyte snapshot of a huge `holdings` table so you do not hammer the production OLTP system?
A: Schedule the initial full snapshot during a low-traffic window and, where the source connector supports it, narrow scope by selecting only the needed streams/columns so less is read in one pass. For Postgres CDC the initial snapshot is the heavy part — afterwards only WAL deltas flow — so the goal is to survive that one snapshot gently. Coordinating timing and column selection with the DBA keeps the live NAV/holdings OLTP system responsive during that first read.
