# Phase 7 · 1.1.4 + 3.2.9 Vector data & vector storage — 100 self-test questions

Q: What is an embedding in the context of vector storage?
A: An embedding is a fixed-length array of floating-point numbers that encodes the semantic meaning of an input (text, image, etc.) as a point in a high-dimensional space, produced by a learned model. Points whose inputs are semantically similar land close together under a chosen distance metric, which is what makes nearest-neighbour search behave like "find me related content". For a fund corpus, two paragraphs both describing ongoing costs end up near each other even if they share no exact words.

Q: Why does the lesson call embeddings a "first-class data type" in your storage portfolio?
A: Because a vector is not just a blob you stash and forget; it has its own storage footprint, its own index type (ANN), its own distance operators, and its own lifecycle (re-embedding when the model changes). Treating it as first-class means it gets the same governance you give tables: schema, versioning, lineage, and retention. The architectural consequence is that "where do embeddings live and how are they indexed?" becomes a real design decision alongside choosing row vs columnar storage.

Q: What does ANN stand for and why don't we use exact nearest-neighbour search at scale?
A: ANN stands for Approximate Nearest Neighbour: an index that returns *probably* the closest vectors rather than provably the closest. Exact search compares the query against every stored vector (a full scan), which is O(n) per query and becomes too slow once you have millions of vectors. ANN trades a small, measurable loss of recall for orders-of-magnitude lower latency, which is the only way interactive RAG works at corpus scale.

Q: Name the three quantities in the "recall–latency–memory triangle" and what it means.
A: The three quantities are recall (fraction of the true top-k that the index actually returns), latency (query response time), and memory (RAM the index occupies). The triangle means you cannot maximise all three at once: pushing recall up usually costs latency or memory, and shrinking memory usually costs recall or latency. Every ANN index and every tuning parameter is just a way of choosing where you sit inside that triangle for your workload.

Q: Give the intuition for how an HNSW index is structured.
A: HNSW (Hierarchical Navigable Small World) is a multi-layer graph where each vector is a node connected to a small set of near neighbours, with sparse upper layers acting like an express network and dense lower layers giving fine-grained navigation. A search enters at the top, greedily hops toward the query through the express layers, then descends to refine, so you touch a tiny fraction of nodes instead of scanning all of them. This graph-walk design is why HNSW gives high recall at low latency, at the cost of significant memory for the edges.

Q: Give the intuition for how an IVF (inverted-file) index works and how it differs from HNSW.
A: IVF first clusters the vector space into a fixed number of cells (lists) around centroids; at query time it finds the few nearest centroids and only scans the vectors inside those cells. It differs from HNSW by being partition-based rather than graph-based, so it is cheaper to build and lighter on memory but typically needs more probes to reach the same recall. In pgvector this is `ivfflat`, where you set `lists` at build time and `ivfflat.probes` at query time.

Q: In pgvector, which operator computes cosine distance and which computes Euclidean (L2) distance?
A: `<=>` computes cosine distance and `<->` computes Euclidean (L2) distance. pgvector also offers `<#>` for negative inner product, `<+>` for taxicab (L1) distance, and `<~>`/`<%>` for Hamming/Jaccard on binary vectors. Choosing the right operator matters because it must match the metric your embedding model was trained for, otherwise "nearest" is measured in the wrong geometry.

Q: A query uses `ORDER BY embedding <=> '[...]' LIMIT 10` but the HNSW index isn't being used. What is the most likely cause?
A: The index's operator class must match the operator in the query: an index built with `vector_cosine_ops` only accelerates `<=>`, so if your query uses `<->` (the `vector_l2_ops` operator) the planner falls back to a full scan. The first thing to check is that the `CREATE INDEX ... USING hnsw (embedding vector_cosine_ops)` operator class lines up with the distance operator in `ORDER BY`. A close second is whether a `WHERE` clause or `LIMIT` shape is defeating index use.

Q: Write the pgvector statement to create an HNSW index for cosine similarity on column `embedding`.
A: `CREATE INDEX ON items USING hnsw (embedding vector_cosine_ops);` builds an HNSW graph index using the cosine operator class so that `<=>` queries are accelerated. You can tune build quality with `WITH (m = 16, ef_construction = 64)`, the defaults. The operator class (`vector_cosine_ops`, `vector_l2_ops`, or `vector_ip_ops`) must match the distance you actually query with.

Q: What do the pgvector HNSW build parameters `m` and `ef_construction` control, and what are their defaults?
A: `m` is the maximum number of connections (edges) per node per layer, defaulting to 16; higher `m` improves recall and query speed but increases memory and build time. `ef_construction` is the size of the dynamic candidate list while building the graph, defaulting to 64; higher values build a better-connected graph (better recall) but make index creation slower. Both are set once at `CREATE INDEX` time and cannot be changed without rebuilding the index.

Q: What does the pgvector query-time parameter `hnsw.ef_search` do and what is its default?
A: `hnsw.ef_search` sets the size of the dynamic candidate list explored during a search, defaulting to 40; raising it makes the graph walk consider more candidates, improving recall at the cost of latency. You set it per session or transaction with `SET hnsw.ef_search = 100;`, which lets you move along the recall–latency curve without rebuilding the index. It is the single most useful knob when your benchmark shows recall is too low.

Q: In pgvector IVFFlat, what does `lists` control and what does `ivfflat.probes` control?
A: `lists` (set at `CREATE INDEX ... WITH (lists = 100)`) is the number of inverted lists / centroid cells the data is partitioned into, fixed at build time. `ivfflat.probes` (default 1) is how many of those cells a query scans, set per session at query time; raising it improves recall by scanning more cells at the cost of latency. A common rule of thumb is `lists ≈ rows/1000` for up to ~1M rows, with `probes ≈ sqrt(lists)` as a starting point.

Q: What is the maximum number of dimensions pgvector can build an index on for the `vector` type?
A: pgvector can index `vector` (and `halfvec`) columns of up to 2,000 dimensions; you can store wider vectors but you cannot build an HNSW or IVFFlat index on them. This matters when picking an embedding model: a 3,072-dimension model would store fine but force exact scans, so you would need dimensionality reduction or `halfvec` tricks. `bit` indexes go up to 64,000 dimensions and `sparsevec` up to 1,000 non-zero elements.

Q: What is the difference between the pgvector `vector` and `halfvec` types and why would you use `halfvec`?
A: `vector` stores each component as a 4-byte single-precision float, while `halfvec` stores each as a 2-byte half-precision float, roughly halving storage and index memory. You use `halfvec` to fit larger corpora or higher dimensions in RAM and to speed up the memory-bound parts of ANN search, accepting a small precision loss that is usually negligible for retrieval. It is one concrete lever on the "memory" corner of the recall–latency–memory triangle.

Q: What is the core problem of "metadata filtering at scale" that vendors compete on?
A: The problem is combining a vector similarity search with ordinary structured predicates (e.g. "nearest chunks *where* fund_family = 'X' and language = 'fr'") without wrecking recall or latency. Naive approaches either filter then can't navigate the graph efficiently, or search then throw away most results, and both degrade badly when the filter is selective. "Filterable HNSW" — making the graph itself filter-aware — is the genuinely hard engineering that dedicated vector databases differentiate on.

Q: Define pre-filtering vs post-filtering in vector search and the failure mode of each.
A: Pre-filtering applies the metadata predicate first and searches only the surviving subset, which is correct but can be slow or break HNSW navigation when the surviving set is small and disconnected in the graph. Post-filtering runs the ANN search first and then discards results that fail the predicate, which keeps graph navigation intact but can return far fewer than k results — or none — when the filter is selective. The right engine adapts between them or uses a filter-aware index so you get both correctness and recall.

Q: How does Qdrant make HNSW filterable rather than just bolting a filter on afterwards?
A: Qdrant extends the HNSW graph with *additional edges* derived from indexed payload values, so the graph stays navigable even when you constrain to a subset of points. This means filtered search walks a graph that already accounts for the filter, avoiding the disconnected-subgraph failure of naive pre-filtering and the empty-results failure of post-filtering. The catch is that those extra edges only exist if the payload index was created, so ordering matters.

Q: In Qdrant, why should payload indexes be created *before* ingesting data?
A: Because the filter-aware extra edges in the HNSW graph are only generated when a payload index already exists at the time points are indexed. If you create the payload index after ingestion, you must rebuild the HNSW index for the filterable benefit to apply, otherwise filtered queries fall back to slower behaviour. The practical rule is: define your `keyword`/`integer`/etc. payload indexes (e.g. on `fund_family`) up front, then load the vectors.

Q: What is Qdrant's `full_scan_threshold_kb` and why does it exist?
A: `full_scan_threshold_kb` is a configuration value at which Qdrant's planner switches from HNSW graph traversal to an exact full scan, used when the candidate set surviving a filter is small. The reasoning is that for a tiny, highly selective subset, scanning it exactly is faster and more accurate than navigating a graph that the filter has fragmented. It is the mechanism that lets a filterable engine stay fast across both broad and narrow filters.

Q: What are the names of the Qdrant HNSW configuration parameters and the query-time search parameter?
A: At index configuration Qdrant uses `m` (edges per node) and `ef_construct` (neighbours considered while building), mirroring pgvector's `m` and `ef_construction`. At query time you set `hnsw_ef` in the search request to control the candidate-list size, and if unspecified it defaults to `ef_construct`. Naming differs slightly from pgvector (`ef_construct` vs `ef_construction`, `hnsw_ef` vs `hnsw.ef_search`), which is an easy trap when porting a benchmark.

Q: Why is "re-embedding on model change" a lineage and pipeline problem rather than a one-off script?
A: Because an embedding is only comparable to other vectors produced by the *same model and version*; changing the model invalidates every stored vector, so you must re-embed the entire corpus and atomically swap indexes. That makes it a scheduled, orchestrated job with its own inputs (raw documents), outputs (new vectors), and lineage edges, not a manual run. Treating it as a pipeline lets you answer "which model produced this vector?" and roll forward or back deliberately.

Q: Why can't you mix vectors from two different embedding models in one searchable index?
A: Because different models place inputs in different coordinate systems with different dimensionalities and geometry, so distances between a model-A vector and a model-B vector are meaningless. A nearest-neighbour query would return nonsense, silently degrading retrieval rather than erroring. This is exactly why an embedding-model change forces a *full* re-embed and why you pin the model name and version next to the vectors.

Q: An embedding-model upgrade is approved. Sketch the safe cutover for a production RAG index.
A: Build the new index in parallel from the same source documents (dual-write or a side collection), validate retrieval quality on your golden set against the old index, then atomically switch reads to the new index and retire the old one. You keep the old index available for rollback until the new one is proven, and you record the model name+version and the cutover timestamp in lineage. The whole thing is orchestrated by your scheduler so it is repeatable and auditable, not a manual swap.

Q: What does it mean to say "your SCD2 instincts apply" to vector lifecycle management?
A: It means an embedding is effective-dated against a model version much like an SCD2 row is effective-dated against a business validity window: a vector is "the embedding of this chunk *under model X from date Y*". To answer historical questions you keep the model/version as a dimension and never silently overwrite, just as SCD2 keeps prior rows rather than mutating in place. The discipline of versioned, queryable history transfers directly from dimensional modelling to embeddings.

Q: State the lesson's default architectural recommendation for vector storage and the condition for overturning it.
A: The default is "pgvector until proven otherwise": start with vectors inside your existing Postgres so embeddings sit next to relational metadata under one transactional, governed system. You overturn it only when your *own* recall/latency/scale numbers show pgvector cannot meet the workload — for example filtered top-k at high QPS over tens of millions of vectors with tight latency SLOs. The point is to decide on measured numbers, not on a vendor's benchmark slide.

Q: Why is "decide it on vendor slides instead of your own recall/latency numbers" called out as the failure mode here?
A: Because vendor benchmarks are tuned to flatter the vendor's product on workloads that may not match yours — different dataset, dimensionality, filter selectivity, and hardware. Buying on those numbers risks either over-provisioning a dedicated cluster that Postgres could have replaced, or stranding a workload on an index that can't filter by fund family at your scale. The architect's job is to reproduce the decision on your corpus and your queries.

Q: What three things does the lesson's hands-on benchmark actually measure?
A: It measures filtered top-k latency, recall against an exact-scan baseline, and how the recall–latency point moves as you vary one HNSW parameter (`ef_search` or `m`). The exact scan provides ground truth so recall is a real number, not a guess. Sweeping one parameter shows you the shape of the trade-off curve for *your* data, which is the evidence behind the "pgvector until proven otherwise" verdict.

Q: Why must you compute an exact-scan baseline when benchmarking an ANN index?
A: Because recall is defined relative to the true nearest neighbours, and only an exact (brute-force) scan tells you what those truly are. Without that ground truth you can report latency but you cannot report recall — you would have no idea whether your fast index is returning the right documents or confidently wrong ones. The exact scan is slow but you only need it once per query set to label correctness.

Q: For a filtered top-k benchmark like "top-10 within fund family X", what makes it harder than an unfiltered top-10?
A: The filter shrinks the candidate set, which can fragment the HNSW graph so greedy navigation either misses good neighbours (low recall) or has to wander widely (high latency). An unfiltered search walks the full, well-connected graph and is the easy case the index was built for. Measuring the *filtered* case is the whole point because real fund queries are almost always scoped to a family, language, or document type.

Q: You raise `hnsw.ef_search` from 40 to 200 and recall improves but p99 latency doubles. How do you read this result?
A: You have moved up-and-right on the recall–latency curve: more candidates explored means more true neighbours found but more work per query. The right call depends on your SLO — if the workload tolerates the latency and needed the recall, keep it; if not, this is evidence you may need a different index (or hardware) rather than just a bigger `ef_search`. Crucially, this is exactly the kind of own-numbers evidence that should drive the pgvector-vs-dedicated decision.

Q: What is "hybrid search" in the RAG context and why is it the modern default?
A: Hybrid search fuses keyword (lexical, e.g. BM25) retrieval with dense vector retrieval and then reranks the merged candidates, rather than relying on vectors alone. It is the default because the two methods fail differently: vectors catch paraphrase and semantics but miss exact tokens like an ISIN, while keyword search nails exact identifiers but misses synonyms. Combining them recovers both, which is why the taxonomy calls "keyword+vector reranked" the RAG default rather than an optimisation.

Q: Give a fund-data example where pure vector search fails but keyword search succeeds.
A: Searching for an exact ISIN like `LU0123456789` or an exact LEI: a dense embedding may not place that precise token near a query containing it, so vector search can miss the document that literally contains the code. BM25/keyword search matches the exact string immediately. This is the canonical argument for hybrid retrieval in a fund corpus full of identifiers, codes, and regulated terminology.

Q: Give a fund-data example where pure keyword search fails but vector search succeeds.
A: A query for "ongoing charges" should also retrieve passages that say "frais courants" (French) or "running costs" — synonyms and translations that share no tokens with the query. Keyword/BM25 matching misses these because the surface strings differ, whereas a multilingual embedding places them near each other in vector space. This is why a Luxembourg EN/FR corpus benefits especially from the vector half of hybrid retrieval.

Q: When does a dedicated vector database genuinely earn its place over pgvector?
A: When you need very large scale (tens of millions to billions of vectors), high write/query throughput, advanced filterable-HNSW performance under selective filters, or operational features like distributed sharding and replication that Postgres extensions don't provide cleanly. The trigger should be measured limits on your workload, not anticipation. Below those thresholds, the cost of running a second stateful system rarely beats keeping vectors in the database you already operate and govern.

Q: What are the hidden costs of adding a dedicated vector database next to your existing Postgres?
A: You now run a second stateful system: separate backups, monitoring, access control, upgrades, and a second source of truth that can drift from your relational metadata. You also lose transactional joins between vectors and the rest of your data, so consistency between a chunk's metadata and its vector becomes an integration concern. These operational and governance costs are why "pgvector until proven otherwise" defaults to fewer moving parts.

Q: What is the convergence trend among Postgres, dedicated vector DBs, and search engines for RAG?
A: They are converging on the same capability set: keyword + vector retrieval fused and reranked, with metadata filtering. Postgres adds vectors via pgvector, dedicated stores add payload filtering and sometimes lexical search, and search engines (Elasticsearch/OpenSearch) add vector fields alongside their BM25 core. The practical upshot is that the *capability* (hybrid, filtered, reranked) matters more than the product label, so you choose on operational fit and your own numbers.

Q: Why is a search engine like OpenSearch a credible third option for RAG storage?
A: Because it already provides a mature inverted index and BM25 for the keyword half of hybrid search, and it has added dense-vector fields and ANN for the vector half, so one engine can serve both. If your organisation already runs it for log or document search, reusing it avoids standing up a new system. The trade-off is that its ANN and filtering maturity, and its operational profile, must still be benchmarked against pgvector and dedicated stores for *your* workload.

Q: In pgvector, how do you combine a vector similarity search with an ordinary `WHERE` filter, and what's the gotcha?
A: You write a normal query such as `SELECT ... FROM chunks WHERE fund_family = 'X' ORDER BY embedding <=> :q LIMIT 10;`, mixing the structured predicate with the distance order-by. The gotcha is that depending on selectivity the planner may filter first then exact-sort the survivors, or use the HNSW index then filter, and with a selective filter you can silently get fewer than the requested k or worse recall. Partial indexes or `iterative scan` options exist to mitigate this, and you should benchmark the filtered path rather than assume it.

Q: What is "post-filtering returns too few results" and how do you detect it in pgvector?
A: It happens when the HNSW search retrieves k candidates and the `WHERE` filter then removes most of them, leaving you with fewer than k matching rows even though more exist in the table. You detect it by comparing the count and identity of results against an exact-scan baseline for the same filter, where the exact path returns the true full set. The fix is iterative/relaxed scanning or pre-filtering so the index keeps searching until k filtered matches are found.

Q: Why does the lesson tie vector storage to "lineage" so insistently?
A: Because in a regulated fund context you must be able to answer where a retrieved fact came from: which document, which chunk, which embedding model and version, and which index. Lineage turns the vector store from an opaque similarity black box into an auditable component where every retrieval traces back to a source. That traceability is what lets an extracted NAV or SRI value survive a model-risk or CSSF review.

Q: What metadata should you store alongside each vector in a fund-document RAG store?
A: At minimum the source document id and version, the chunk id and character span, the fund family/ISIN, the language, the document type (KID, prospectus, etc.), and the embedding model name+version. This metadata is what powers filtering ("within fund family X"), citation (source span), and lineage (which model produced this vector). Without it you can retrieve a vector but you cannot say what it represents or whether you're allowed to show it.

Q: Why is storing the fund family as a filterable field, not just in the chunk text, important?
A: Because real queries are scoped — "the fee table for this specific fund" — and you must constrain retrieval to that fund family rather than hoping similarity keeps other funds out. A structured, indexed `fund_family` field lets the engine pre/filter correctly and prevents the system from confidently citing the wrong fund's fee table, which in a KID workflow is a mis-sold product. Relying on the text alone leaves correctness to chance.

Q: How would re-embedding interact with a fund document that gets a new prospectus version?
A: The new prospectus version is a new source: you re-chunk it, embed those chunks with the pinned model, and index them, while deciding whether to retire or retain the prior version's vectors for historical queries. This is the same freshness pipeline you run for any source, owned by your orchestrator, with lineage linking document-version to chunk to vector. Versioning here lets you answer "what did the KID say last quarter" as well as "what does it say now".

Q: Why does EU data residency apply to the *embeddings*, not just the source documents?
A: Because an embedding is derived from and represents the content of the document, and sending text to a third-country embedding API to produce vectors is itself a cross-border processing of that data. So a French KID's residency and processing constraints follow it into the embedding step, the stored vectors, and any API call used to create them. This is a key reason an architect may prefer a local sentence-transformers model over a third-country embedding API for a Luxembourg corpus.

Q: How does the distance metric you index with relate to how the embedding model was trained?
A: Embedding models are trained so that a specific metric reflects semantic similarity — many sentence-transformers models are normalised and intended for cosine similarity, others for dot product. Indexing with a metric the model wasn't trained for (e.g. L2 on vectors meant for cosine) silently degrades retrieval quality. So you read the model card, pick the matching operator class (`vector_cosine_ops`, `vector_ip_ops`, etc.), and keep query and index metric consistent.

Q: If embeddings are L2-normalised, why are cosine distance and inner product effectively equivalent?
A: When every vector has unit length, the dot product equals the cosine of the angle between vectors, so ranking by inner product and ranking by cosine similarity produce the same order. This lets you use the cheaper inner-product operator (`<#>`) and `vector_ip_ops` index while getting cosine semantics. It is a common optimisation, but it only holds while the vectors are actually normalised, so you must enforce normalisation in the embedding step.

Q: What is the difference between data drift and concept drift, and which one applies to a re-embed decision?
A: Data drift is a change in the input distribution (e.g. new fund types appear in the corpus), while concept drift is a change in the relationship between inputs and the target. For embeddings the relevant trigger is usually a deliberate *model* change (a better embedding model is adopted) plus input drift that makes the old model's vectors less representative. Either way the response is a governed re-embed, which is why drift monitoring feeds the vector lifecycle.

Q: Why is choosing a smaller embedding dimensionality not automatically worse for retrieval?
A: Because vector width drives storage and ANN latency, and a well-trained smaller model can match a larger one on a specific corpus while being cheaper and faster to index and search. "Bigger is not reliably better" — the extra dimensions can encode noise or capabilities irrelevant to fund text. You decide on your own retrieval set rather than assuming higher dimensions win, since they directly inflate the memory corner of the triangle.

Q: How does vector dimensionality affect the recall–latency–memory triangle concretely?
A: Wider vectors mean more bytes per vector (memory), more arithmetic per distance computation (latency), and a larger HNSW graph to hold in RAM. So a 1024-d model can cost roughly twice the storage and distance work of a 512-d model for the same number of vectors. This is why dimensionality is an architectural choice, not a detail: it moves your whole operating point in the triangle before you tune a single index parameter.

Q: What is the "first thing to check" when a pgvector similarity query is unexpectedly slow on a large table?
A: First confirm the query is actually using the ANN index via `EXPLAIN`: look for an Index Scan on the HNSW index rather than a Seq Scan with a sort. Common causes of a missed index are an operator/operator-class mismatch, a `LIMIT` that's absent, or a `WHERE` filter the planner handles by sequential scan. Only after confirming the plan should you tune `hnsw.ef_search` or revisit `m`.

Q: Why does pgvector require a `LIMIT` (or `ORDER BY ... LIMIT`) to use the HNSW index effectively?
A: HNSW is an approximate *top-k* index: it answers "give me the nearest k" by walking the graph, so without an `ORDER BY distance LIMIT k` shape there is no k to navigate toward and the planner has no reason to use it. A query that wants all rows beyond a distance threshold doesn't map onto the index's design and will scan. So similarity queries are written as `ORDER BY embedding <=> :q LIMIT k` to engage the index.

Q: What does `EXPLAIN (ANALYZE)` help you see when validating a vector index, and what's a caveat?
A: It shows the actual plan and timing, confirming whether the HNSW index scan is used and how long it took on real data, so you can verify index engagement rather than guessing. The caveat is that `EXPLAIN ANALYZE` reports latency but not recall — a fast plan can still be returning approximate, incomplete results. You pair it with an exact-scan recall check to know the answer is both fast and correct enough.

Q: What is `pgvector` and how is it deployed?
A: `pgvector` is an open-source PostgreSQL extension that adds the `vector` data type, distance operators, and ANN index types (HNSW, IVFFlat) to a standard Postgres database. You enable it per database with `CREATE EXTENSION vector;`, after which embeddings live in ordinary columns you can join and filter like any other data. Its appeal is that it brings vectors into a system you already run, back up, and govern.

Q: What is Qdrant and what is its core unit of organisation?
A: Qdrant is an open-source, purpose-built vector database written in Rust, designed for ANN search with rich payload filtering. Its core unit is a *collection*, which holds points; each point has a vector plus a JSON *payload* of metadata you can index and filter on. It competes on filterable-HNSW performance and operational features aimed specifically at vector workloads, which is the dedicated-store half of the lesson's comparison.

Q: In Qdrant, what is a "payload" and how does it relate to filtering?
A: A payload is the arbitrary JSON metadata attached to a point — for fund data that's fields like `fund_family`, `language`, and `doc_type`. To filter efficiently you create a payload index of the appropriate type (`keyword`, `integer`, `float`, `bool`, `datetime`, etc.) on the fields you'll filter by. Those indexes are what let Qdrant build filter-aware HNSW edges and run fast filtered searches.

Q: Which Qdrant payload index type would you use for filtering by ISIN, and which for a numeric SRI score?
A: Use the `keyword` payload index for ISIN, since it is an exact-match string identifier you filter on equality. Use an `integer` (or `float`) payload index for an SRI risk score so you can filter by exact value or range. Picking the right type matters because it determines how the filter is evaluated and whether the filter-aware HNSW edges can be built for it.

Q: Why is "filterable HNSW is the hard part vendors compete on" a fair summary of the market?
A: Because plain ANN at scale is largely a solved, commoditised problem, but keeping high recall *and* low latency *while* a selective metadata filter is applied is genuinely difficult engineering. Vendors differentiate on how gracefully they handle the spectrum from broad to highly selective filters without falling back to slow exact scans or empty results. For fund RAG, where every query is scoped to a family or language, that exact capability is what you're really buying.

Q: How does cosine *distance* relate to cosine *similarity*, since pgvector reports distance?
A: pgvector's `<=>` returns cosine distance, defined as 1 minus cosine similarity, so smaller distance means more similar. Cosine similarity ranges from -1 to 1, so the distance ranges from 0 (identical direction) to 2 (opposite). When you display a "relevance score" you typically convert back with `1 - distance`, and you must keep the direction straight: you `ORDER BY distance ASC` to get the most similar first.

Q: What is the role of reranking after hybrid retrieval, and why isn't ANN ranking enough?
A: Reranking takes the merged top candidates from keyword and vector retrieval and rescores them with a stronger model (often a cross-encoder) that reads query and passage together, producing a more accurate final order. ANN ranking is fast but coarse because it scores each passage independently of the query's full context, so the top result by vector distance isn't always the most relevant. Reranking the top ~50 is cheap relative to its quality gain, which is why it's part of the enterprise default.

Q: What is Reciprocal Rank Fusion (RRF) and where does it fit in hybrid search?
A: RRF is a simple, robust method to merge two ranked result lists (e.g. BM25 and vector) by summing reciprocals of each item's rank across the lists, so items ranked highly by either method bubble up without needing comparable raw scores. It fits as the fusion step that combines lexical and dense results before reranking. Its appeal is that it sidesteps the problem that BM25 scores and cosine distances live on incompatible scales.

Q: Why can't you simply add a BM25 score and a cosine distance together to combine results?
A: Because they live on different, unbounded-and-bounded scales with opposite directions — BM25 is an unbounded relevance score where higher is better, while cosine distance is bounded where lower is better. Summing them is dimensionally meaningless and dominated by whichever has the larger numeric range. Rank-based fusion (RRF) or careful score normalisation is used instead precisely to avoid this trap.

Q: For a 1,000-chunk fund corpus, does the lesson expect you to reach for a dedicated vector DB? Why or why not?
A: No — at a thousand chunks even an exact scan is fast, and pgvector with an HNSW index is overkill-but-fine, so a dedicated cluster would add operational cost for no measurable benefit. The lesson's whole point is that scale and filter selectivity, demonstrated with your own numbers, are what justify dedicated storage. A 1k corpus is firmly in "pgvector until proven otherwise" territory.

Q: What does "embeddings as learned features" mean and how does it connect to the rest of the ML platform?
A: It means embeddings are model outputs that serve as features for downstream tasks (retrieval, classification, clustering), so they belong to the same feature-lifecycle thinking as any engineered feature. The connection is that producing and versioning them is a pipeline managed by your orchestrator, their provenance is tracked in lineage, and a model change triggers regeneration — the same governance you apply to a feature store. They are features that happen to be high-dimensional and similarity-searched.

Q: Why is recall, not just latency, the metric an architect must insist on when evaluating a vector index?
A: Because a fast index that quietly drops the right documents produces confident, wrong retrievals — and in a fund-extraction workflow that means citing the wrong fee table or missing a disclosure. Latency alone is easy to optimise and easy to be fooled by; recall against ground truth is what tells you the system is actually finding what it should. The discipline is to always report both, with recall measured against an exact baseline.

Q: How would you explain `m` in HNSW to a colleague using a road-network analogy?
A: `m` is roughly how many roads connect each town (node) to its neighbours: more roads (higher `m`) mean you can almost always find a fast route to anywhere, improving recall and speed, but building and maintaining all those roads costs more land (memory) and construction time. Too few roads and the graph has dead-ends where the search gets stuck and misses good neighbours. The right `m` gives enough connectivity for high recall without paying for redundant edges.

Q: What practical effect does increasing `ef_construction` have, and when is it worth it?
A: A higher `ef_construction` makes the index builder consider more candidate neighbours per node, producing a better-connected, higher-quality graph that yields better recall at query time for the same `ef_search`. It is worth it for a relatively static corpus you index once and query many times, where the one-time build cost is amortised. For a frequently re-embedded corpus you weigh that build cost against how often you rebuild.

Q: Why is the embedding-model name and version "pinned next to the vectors" rather than stored once globally?
A: Because a corpus can contain vectors produced at different times by different model versions during a migration, and you must know per-vector which model made it to interpret, compare, or selectively re-embed. Pinning it as a column/payload field makes that queryable and is the lineage anchor for the re-embed pipeline. A single global setting would lie the moment a partial re-index is in flight.

Q: A RAG system returns chunks from the wrong fund despite a fund-family filter. What do you check first?
A: First confirm the filter is actually being applied at the storage layer (an indexed `fund_family` predicate), not just hoped for via similarity, and that post-filtering isn't silently passing through unfiltered candidates. Then verify the metadata on the offending chunks is correct — a mis-tagged `fund_family` at ingestion will defeat any filter. The combination of "is the filter enforced" and "is the metadata right" catches most cross-fund leakage.

Q: Why does the lesson frame the vector-store decision as "today's recurring architecture question"?
A: Because nearly every team adopting RAG hits the same fork — dedicated vector DB vs pgvector vs hybrid search engine — and the market actively pitches dedicated products. An architect is repeatedly asked to make or defend this call, so having a principled, numbers-driven method (benchmark, recall vs latency, filter behaviour) is a reusable skill rather than a one-off. The wrong default here either wastes money on a cluster or strands a workload.

Q: What is the difference between `IVFFlat` build cost and `HNSW` build cost, and why does it matter for re-embedding?
A: IVFFlat builds quickly (clustering into lists) and uses less memory, whereas HNSW builds more slowly and uses more memory because it constructs a rich neighbour graph. This matters for re-embedding because a corpus that is re-indexed frequently pays the HNSW build cost each time, so the cheaper-to-build IVFFlat can be attractive despite usually needing more probes for equal recall. The choice trades build/query/recall against how often you rebuild.

Q: Why might IVFFlat recall be poor right after loading data, and how is that different from HNSW?
A: IVFFlat must first see representative data to compute good centroids, so building the index before enough rows are loaded (or with too few `lists`) yields poorly placed cells and weak recall. HNSW doesn't depend on pre-computed centroids — it builds its graph incrementally as points are added — so it doesn't have this "build on populated table" requirement. The practical advice for IVFFlat is to create the index after loading representative data.

Q: How do you decide between cosine, L2, and inner-product distance for a fund-document corpus?
A: You follow the embedding model's intended metric first — most sentence-transformers text models target cosine similarity, so you'd use `<=>` / `vector_cosine_ops`. If the vectors are normalised, inner product is an equivalent, sometimes faster choice. L2 is appropriate only for models trained for Euclidean geometry; using it on cosine-trained vectors quietly degrades retrieval, so the model card, not preference, drives the choice.

Q: What is the operational risk of "buy a cluster Postgres could have replaced"?
A: You take on the recurring cost and complexity of a separate distributed system — provisioning, scaling, backups, security reviews, upgrades — for a workload that a single Postgres with pgvector handled within SLO. Beyond money, it splits your source of truth and governance surface, making audits harder. The lesson's antidote is to prove the need with your own filtered recall/latency numbers before procuring.

Q: What is the risk of "strand the RAG workload on an index that can't filter by fund family at scale"?
A: If you pick a store whose filtered-search degrades under selective predicates, then at production scale "top-k within fund family X" either becomes slow or returns too few/wrong results — the system can't reliably scope to a fund. That breaks correctness for a regulated extraction workflow where citing the right fund is mandatory. This is the failure that filterable-HNSW benchmarking is designed to catch *before* you commit.

Q: How does keeping vectors in Postgres simplify enforcing access controls on fund documents?
A: Because the vectors live in the same database as the relational metadata and permissions, you can apply row-level security, schema permissions, and joins to enforce "this user may only retrieve documents for funds they're entitled to" in one place. A separate vector store would need its own ACL model kept in sync, risking drift where a vector is retrievable even though the underlying document isn't permitted. Single-system governance is a real point on the pgvector side of the ledger.

Q: Why is "filter by fund family" a recurring requirement specifically in fund administration RAG?
A: Because documents are organised and regulated per fund — a KID, fact sheet, or prospectus belongs to one fund — and any extracted value (NAV, costs, SRI) must come from *that fund's* document, never a neighbour's. Cross-fund leakage isn't a quality nit; it's a compliance and mis-selling risk. So scoped, filtered retrieval is a first-class requirement, which makes filterable indexing central to the storage choice.

Q: What is a sensible starting `lists` value for an IVFFlat index over ~500k fund-chunk vectors?
A: A common heuristic is `lists ≈ rows / 1000` for datasets up to about a million rows, so around 500 lists for 500k vectors, with `ivfflat.probes` then tuned upward from 1 to hit your recall target. These are starting points to refine against your own recall/latency benchmark, not fixed rules. The reason for the rule is to keep each cell's contents small enough that probing a few cells scans a manageable fraction of the data.

Q: How does the choice of `k` (top-k) interact with filtered retrieval quality?
A: A larger `k` gives the reranker and the filter more candidates to work with, improving the odds that enough *filtered* relevant chunks survive, but it costs latency and can dilute precision. With selective filters you often over-retrieve (a larger ANN `k`) precisely so that post-filtering still leaves enough matches. Tuning `k` is part of the same recall-vs-latency balancing as the index parameters.

Q: Why does the lesson recommend embedding KIDs specifically rather than arbitrary text for the benchmark?
A: Because KIDs (Key Information Documents) are real, public, multilingual, table-heavy fund documents, so they exercise exactly the retrieval challenges your production system faces — French/English mix, identifiers, structured fee/SRI tables. Benchmarking on representative data is the only way your recall/latency numbers transfer to the real workload. Generic text would give numbers that flatter or mislead the storage decision.

Q: What is the relationship between chunking and the vector store you choose?
A: Chunking decides what unit of text becomes one vector; finer chunks mean more vectors (more storage, more index work) but more precise retrieval, while coarser chunks mean fewer vectors but blurrier matches. The vector store must handle the resulting count and the metadata you attach per chunk, so a chunking strategy that explodes the vector count can push you toward needing a more scalable store. The two decisions are coupled and should be benchmarked together.

Q: Why is it dangerous to store the embedding but not the source span (character offsets) of a chunk?
A: Without the source span you can retrieve a relevant chunk but cannot cite *where* in the document it came from, which breaks the auditability a regulated extraction needs — every extracted value should trace to a source span and a trace id. It also makes human review impossible, since a reviewer can't see the exact text that justified a field. So the span is stored as metadata alongside the vector precisely to support citation and review.

Q: How does "pre-filtering" help correctness but potentially hurt HNSW navigation?
A: Pre-filtering guarantees correctness because only points satisfying the predicate are ever considered, so you can't return a forbidden document. But restricting the searchable set can disconnect the HNSW graph — the surviving nodes may not be reachable from the entry point via the normal edges — so greedy navigation stalls or misses neighbours, hurting recall and/or latency. Filter-aware indexes (extra edges) or a full-scan fallback for tiny subsets exist to rescue this.

Q: What does it mean that Qdrant's filtered search must "balance weak and strict filters"?
A: A weak (low-selectivity) filter keeps most points, so the normal HNSW graph still works well and you mostly want fast traversal; a strict (high-selectivity) filter keeps very few points, where the graph fragments and an exact scan of the small survivor set is faster and more accurate. Qdrant's planner picks the right strategy per query, using filter-aware edges in between and `full_scan_threshold_kb` to switch to exact scan when the survivor set is tiny. Handling both extremes well is the engineering that distinguishes a serious filterable engine.

Q: Why is "which model version produced this vector?" a question an architect must be able to answer?
A: Because retrieval quality, comparability, and any downstream extracted value all depend on the embedding model, so a model-risk validator or auditor will ask exactly which model and version generated the vectors behind a result. If you can't answer, you can't defend the pipeline or reproduce results, and you can't scope a partial re-embed. Pinning the model name+version next to vectors and recording it in lineage makes the answer a lookup.

Q: How does a vector store fit into the broader Phase-6 control plane (lineage, DQ, orchestration)?
A: The vector store is just another governed asset: its population is an orchestrated job, its inputs/outputs appear in the lineage graph, and the documents and extracted outputs flow through the same data-quality and lineage tooling. Re-embedding is a scheduled pipeline run, not a script, and the model version is a lineage attribute. So embeddings inherit the platform's controls rather than living in an ungoverned side system.

Q: What does `CREATE EXTENSION vector;` do and where is it run?
A: It installs the pgvector extension's types, operators, and index access methods into the current PostgreSQL database, after which you can declare `vector(N)` columns and build HNSW/IVFFlat indexes. It is run once per database (by a role with sufficient privilege), not per session, and the extension binaries must already be installed on the server. On native Ubuntu you typically install the `postgresql-NN-pgvector` package first, then run this statement.

Q: On native Ubuntu, what is the typical way to get pgvector available to your Postgres server?
A: You install the distribution package matching your Postgres major version (for example `postgresql-16-pgvector` via `apt`), which places the extension files where the server can load them, then run `CREATE EXTENSION vector;` in the target database. Alternatively you build the extension from source against your `pg_config`. The key point is the server-side extension must be present before `CREATE EXTENSION` succeeds — a missing package is the usual cause of an "extension not available" error.

Q: How do you run Qdrant locally for the benchmark, and what's the simplest option?
A: The simplest option is the official Qdrant Docker image — `docker run -p 6333:6333 qdrant/qdrant` — which exposes the REST/gRPC API for loading collections and querying. You then create a collection with your vector size and distance metric and define payload indexes on filter fields before ingesting. Running it in a container keeps it isolated from your Postgres benchmark so the two sides are compared cleanly.

Q: When creating a Qdrant collection, what two vector properties must you specify and why?
A: You must specify the vector *size* (dimensionality, which must match your embedding model exactly) and the *distance* metric (Cosine, Dot, or Euclid, matching how the model was trained). They matter because a mismatched size will reject vectors outright and a mismatched metric will silently rank results in the wrong geometry. Getting these right is the Qdrant analogue of choosing the correct pgvector operator class.

Q: Why does the lesson pair pgvector *and* Qdrant in the same hands-on exercise?
A: So you load the identical vectors and metadata into both and benchmark the same filtered top-k queries, producing a like-for-like comparison on recall, latency, and filtering behaviour. That head-to-head is the evidence behind the ADR for the vector-store choice and behind "pgvector until proven otherwise". Comparing them on *your* corpus is what turns a vendor debate into a decision grounded in numbers.

Q: What single sweepable parameter does the lesson suggest varying, and what does the sweep teach?
A: It suggests varying one HNSW parameter — `ef_search` (query-time) or `m` (build-time) — and observing how the recall–latency operating point moves. The sweep teaches the *shape* of the trade-off curve for your data: how much latency you pay per point of recall, and whether you can hit your SLO at acceptable recall at all. That curve, not a single number, is what justifies the storage verdict.

Q: Why is `ef_search` a better first knob to sweep than `m` when tuning an existing index?
A: Because `ef_search` is a query-time setting you can change instantly with `SET hnsw.ef_search = N;` and re-run the benchmark, whereas changing `m` requires rebuilding the entire index. So you explore the recall–latency curve cheaply with `ef_search` first, and only rebuild with a different `m` if `ef_search` alone can't reach your target. It's the fastest path to mapping your operating options.

Q: What would you put in the "10-line when-does-Postgres-suffice verdict" the lesson asks for?
A: The corpus size and growth, the measured filtered top-k recall and latency at your target `ef_search`, the filter selectivity you tested, whether pgvector met the SLO, and the threshold (scale, QPS, or filtering) at which you'd revisit. Each claim cites your own numbers, not a vendor's, and ends with a clear default (e.g. "pgvector suffices below ~X million vectors at our filter selectivity; reassess above it"). It's a decision record, not a benchmark dump.

Q: How does the "fund family" filter requirement interact with the choice between pre- and post-filtering?
A: Because fund-family filters are often selective (one family out of many), post-filtering risks returning too few results and pre-filtering risks fragmenting the graph — so you need a store that handles selective filters well, i.e. filterable HNSW or a sensible full-scan fallback. This is precisely why you benchmark the *filtered* path and why filterability dominates the fund-RAG storage decision. The requirement turns an abstract index property into a concrete acceptance test.

Q: Why is reproducibility (offline, no paid APIs) emphasised for the vector benchmark?
A: Because a regulated, auditable result must be re-runnable by a reviewer without depending on a third-party service that could change, charge, or lose access — and because EU residency concerns argue against routing fund text through external embedding APIs. A fully local stack (sentence-transformers, pgvector, Qdrant) means anyone can reproduce your recall/latency numbers and your retrieval results. Reproducibility is what makes the benchmark defensible rather than anecdotal.

Q: What is product quantization (PQ) and how does it relate to the memory corner of the triangle?
A: Product quantization compresses vectors by splitting each into sub-vectors and replacing them with small codebook indices, so a vector that took kilobytes can be stored in a handful of bytes. It directly attacks the memory corner of the recall–latency–memory triangle, letting far more vectors fit in RAM, at the cost of some recall because distances are computed on approximations. Dedicated stores like Qdrant offer it as a tunable option; for most fund corpora pgvector's `halfvec` is a simpler first lever before reaching for PQ.

Q: Why is normalising vectors before storage a deliberate pipeline step, not an afterthought?
A: Because whether your similarity metric behaves correctly often depends on the vectors being unit-length — normalisation is what makes inner product equal cosine similarity and keeps scores comparable across the corpus. If some vectors are normalised and others aren't, ranking is silently corrupted. So normalisation is a fixed, tested step in the embedding pipeline, applied identically at index time and query time, and the choice is recorded with the model version.

Q: In a fund-RAG store, why might you keep both the raw chunk text and its vector, rather than the vector alone?
A: Because hybrid search needs the raw text for BM25/keyword matching of exact tokens like ISINs, citation needs the text to show the source span to a reviewer, and re-embedding on a model change needs the original text to regenerate vectors without re-parsing documents. The vector alone is a one-way projection you cannot invert back to text. Storing both makes the same row serve lexical search, vector search, citation, and re-embed — which is another quiet advantage of keeping everything in Postgres.

Q: What does `EXPLAIN` showing a `Seq Scan` instead of an `Index Scan` on a vector query tell you, and what's the fix path?
A: It tells you the planner decided not to use the HNSW/IVFFlat index, so every row is being compared by brute force — fine for tiny tables but fatal at scale. The fix path is to check, in order: that the query has an `ORDER BY embedding <op> :q LIMIT k` shape, that the index's operator class matches the operator used, that the index exists and is valid, and that a `WHERE` filter isn't forcing a scan. Only once you see an `Index Scan` do you move on to tuning recall via `hnsw.ef_search`.

Q: Summarise the architect's defensible position on vector storage in one or two sentences.
A: "pgvector until proven otherwise": keep embeddings in the governed Postgres you already run, benchmark filtered top-k recall and latency against an exact baseline on your own fund corpus, and only adopt a dedicated store when measured scale or filtering needs exceed what Postgres meets. Always pin the embedding model+version with the vectors and treat re-embedding as an orchestrated, lineage-tracked pipeline. Decide on your numbers and your filter behaviour, not on vendor slides.
