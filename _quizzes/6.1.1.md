# Phase 7 · 6.1.1 Feature stores (Feast) — 100 self-test questions

Q: What is a feature store, in one sentence?
A: A feature store is a system that manages the definition, storage, and serving of ML features so the same feature logic produces consistent values for both model training (offline) and model inference (online). Its core value is solving training-serving skew: ensuring the feature your model sees at scoring time matches what it was trained on. Feast is an open-source implementation that orchestrates your existing offline store (e.g. the lakehouse) and an online store (e.g. Redis or Postgres) rather than storing the raw data itself.

Q: What are the two "stores" a feature store coordinates, and what is each optimized for?
A: The offline store holds the full feature history and is optimized for large scans — reading months or years of values to build training sets. The online store holds only the latest value per entity and is optimized for millisecond key-lookups at serving time. A feature store like Feast reads from the offline store for training and serves from the online store at inference, keeping both consistent.

Q: Why does training read from the offline store but serving read from the online store?
A: Training needs to reconstruct historical feature state across many timestamps, so it requires a system that can scan large volumes of dated rows cheaply — exactly what a columnar lakehouse offline store does well. Serving needs a single entity's current features in a few milliseconds inside a request budget, which a scan-oriented store cannot meet, so a key-value online store is used. The split exists because the two access patterns have opposite optimization targets.

Q: What is "training-serving skew" and why is it the central problem a feature store addresses?
A: Training-serving skew is any divergence between the feature values a model trains on and the values it receives at inference. It silently degrades production accuracy because the model learned relationships that no longer hold on the skewed live inputs. A feature store reduces it by making one feature definition the single source for both paths, so the offline training value and the online serving value are computed identically.

Q: In Feast, what is an entity?
A: An entity is the object features describe and the key features are looked up by — for example an `investor`, a `fund`, or a `driver`. It defines the join key (e.g. `investor_id`) used both to join features into a training set and to fetch them at serving time. Every feature view references one or more entities so Feast knows what to key on.

Q: In Feast, what is a feature view?
A: A feature view is a versioned definition that groups related features, ties them to one or more entities, points at a data source, and sets a TTL. It is the unit you write in Python and register with `feast apply`, analogous to a dbt model in that it is code, reviewable in a pull request. A feature view tells Feast both how to read training history and what to materialize into the online store.

Q: Why is "features defined as code" considered an architectural advantage over ad-hoc feature SQL?
A: Defining entities, feature views, and TTLs in versioned Python makes features reviewable, diffable, and reproducible — the same governance you already apply to dbt models. Ad-hoc SQL scattered across notebooks drifts between training and serving and has no single owner, which is exactly how skew creeps in. Code-as-definition gives one canonical source that both paths compile from, plus change history.

Q: What does a TTL on a feature view control?
A: TTL (time-to-live) sets the maximum lookback window when joining features to an event timestamp — how stale a feature value is allowed to be before Feast treats it as missing. In a point-in-time join it bounds how far back Feast searches for the most recent value before each row's `event_timestamp`. A short TTL means features that haven't updated recently produce nulls rather than silently using very old data.

Q: Is the TTL window measured relative to "now" or relative to the event timestamp?
A: It is measured relative to each row's `event_timestamp` in the entity dataframe, not relative to the wall-clock time when you run the query. This is essential for reproducibility: re-running a historical query next month must return the same training set it did today. If TTL were anchored to "now", every re-run would shift the eligible window and change the data.

Q: What is a point-in-time-correct join?
A: It is a join that, for each labelled event, attaches only the feature values that were knowable at or before that event's timestamp — never any value recorded afterward. Feast scans backward in time from each entity row's `event_timestamp` and picks the most recent feature value within the TTL window. This is what guarantees the training set reflects "the state of the world" the model would actually have seen at scoring time.

Q: How does a point-in-time join prevent future-data leakage?
A: By enforcing temporal ordering: for every row it only considers feature records whose timestamp is less than or equal to that row's event timestamp. A value recorded after the event is structurally excluded, so a feature computed with hindsight can never enter the training set. Leakage is prevented by construction, not by a downstream cleanup step.

Q: A teammate from your SCD2 background asks how point-in-time correctness relates to dimensional modelling. What do you say?
A: It is the same idea as an effective-dated lookup against a Type-2 slowly-changing dimension: you join the version of the row that was current as of the business event date, not the latest version. Latest-value joins are the SCD1 mistake — they overwrite history and answer "what is true now" instead of "what was true then". Feast's `event_timestamp` plays the role of your as-of date and TTL bounds how far back the valid-from window may stretch.

Q: What is the Feast method to build a training set with point-in-time correctness?
A: `store.get_historical_features()`. You pass it an `entity_df` (the rows you want labels for, each with its entity keys and an `event_timestamp`) and a `features` list naming which feature-view columns to pull. Feast returns a dataframe where each row is enriched with the point-in-time-correct values for those features.

Q: What two things must the `entity_df` passed to `get_historical_features` contain?
A: It must contain the entity join-key column(s) — e.g. `investor_id` — so Feast knows which entity each row is about, and an `event_timestamp` column giving the as-of moment for the point-in-time join. You typically also carry the label/target column through so the returned dataframe is a complete training set. Missing or misnamed `event_timestamp` is the classic reason the join behaves wrongly.

Q: What is materialization in Feast?
A: Materialization is the scheduled process of computing feature values from the offline store and loading the latest one per entity into the online store, so serving has fresh values to look up. It is what bridges the batch world (history in the lakehouse) and the request world (millisecond lookups). Without materialization the online store is empty and `get_online_features` returns nothing.

Q: What is the difference between `feast materialize` and `feast materialize-incremental`?
A: `feast materialize <start> <end>` processes feature values over an explicit time range you supply. `feast materialize-incremental <end>` tracks the end time of the previous run and uses it as the new start, so it only loads data that has arrived since — making it the natural fit for a recurring schedule. Incremental avoids re-scanning history you already pushed online.

Q: After `feast materialize-incremental` runs daily, what determines its start time on the next run?
A: The end time of the previous incremental run becomes the start time of the next, because Feast records how far it has materialized. This is what makes it incremental: each run only picks up newly-arrived offline data rather than re-materializing the whole window. The first run with no recorded history materializes from the feature view's earliest available data up to the given end.

Q: What is the "freshness window" or freshness lag created by materialization?
A: It is the gap between when a feature value becomes true in the offline store and when materialization pushes it into the online store. If you materialize once an hour, the online value can be up to roughly an hour stale at serving time. This lag is a design parameter: tighter freshness costs more frequent (or streaming) materialization, and some use cases tolerate hours while others need minutes.

Q: What Feast method fetches features at serving time, and from which store?
A: `store.get_online_features()` reads the latest feature values from the online store by entity key, for low-latency inference. It does not do a point-in-time scan — it returns whatever materialization last pushed for that entity. You pass it the entity rows and the feature references you need for the live prediction.

Q: Why does `get_online_features` not perform a point-in-time join like `get_historical_features` does?
A: At serving time you want the single current value as fast as possible, and the online store deliberately holds only the latest value per entity — there is no history there to scan. Point-in-time correctness is a training-time concern, ensuring labels don't see the future; at inference the "current" value is by definition the only correct one. So online reads are simple key lookups, not temporal joins.

Q: When is a feature store overkill for a fund workload?
A: When scoring is batch-only — you compute predictions on a schedule reading directly from the lakehouse — you never need millisecond lookups, so the entire online store and materialization machinery adds operational cost for no benefit. Most fund use cases (overnight risk runs, periodic investor segmentation) are batch. Buying a feature store there means standing up and paying for an online database nobody queries.

Q: Name a fund use case that justifies an online store and one that does not.
A: A real-time fraud or AML check on an incoming subscription/redemption — where you must score within the request — justifies an online store for millisecond feature lookups. A nightly investor-churn batch score does not: it reads history straight from the lakehouse and writes results to a table, needing no online machinery. The deciding question is whether a prediction must happen inside a synchronous request.

Q: Why might a feature-store purchase be a "premature vendor pitch" to veto?
A: Because feature stores solve online/offline consistency and low-latency serving — problems a batch-only shop does not have. If your scoring is all scheduled batch, the pitch adds an online store, materialization jobs, and a serving SLA you'll pay for and never exercise. The veto condition is: no real-time scoring use case exists, so there is no online consistency problem to buy a solution for.

Q: Show concretely how a naive "latest value" SQL join leaks future data.
A: A naive join attaches each entity's current feature value to every historical event regardless of date, so a churn event from January gets joined to a NAV-staleness or flow-volatility feature computed in June. The model then trains on information that did not exist at the January event, learning a relationship it can never reproduce live. The leak shows up as rows where the joined feature's source timestamp is later than the row's `event_timestamp`.

Q: In the "Do" exercise, how do you isolate the rows where the naive join cheated?
A: Build the training set both ways — once with `get_historical_features` (point-in-time correct) and once with the naive latest-value SQL join — then diff the two outputs on the same (investor, event_timestamp) keys. The rows that differ are exactly the ones where the naive join pulled a feature value whose timestamp is after the event, i.e. future data. Inspecting those rows makes the leakage concrete rather than theoretical.

Q: Three fund features the lesson asks you to define in Feast are what?
A: 30-day flow volatility, an investor churn signal, and NAV staleness, each modelled as a feature view over the lakehouse keyed on the relevant entity with an explicit TTL. They are chosen because they are dated, slowly-changing quantities where point-in-time correctness actually matters. Committing them as code makes the definitions reviewable and reproducible across training and any serving path.

Q: For a "30-day flow volatility" feature, why does point-in-time correctness matter at training time?
A: Because the 30-day window must be computed as of each event's date — a churn event on 1 March must use flow volatility through 1 March, not a value that secretly includes April flows. A latest-value join would feed the model post-event volatility, inflating apparent predictive power and collapsing in production. The TTL and `event_timestamp` together ensure each row sees only its own as-of window.

Q: What backends can Feast use as an online store?
A: Common online-store backends include Redis, SQLite (the default for local development), PostgreSQL, and DynamoDB, among others. SQLite is convenient for the hands-on lab; Redis or Postgres are typical for low-latency production serving. The online store only needs fast key lookups, so a key-value or simple relational store fits.

Q: What backends can Feast use as an offline store?
A: Offline-store backends include local files/Parquet, BigQuery, Snowflake, and Redshift, among others — generally scan-oriented analytical stores where the full feature history lives. In this lesson the offline store is your lakehouse. The offline store must support the large historical scans that point-in-time joins require.

Q: For the hands-on lab on a native Ubuntu laptop, what is a sensible Feast stack?
A: Feast over PostgreSQL (or local Parquet files) as the offline store and Redis — or SQLite for the simplest setup — as the online store, all installed via `uv` and run with `uv run`. This mirrors the managed equivalents (Databricks/Tecton/SageMaker feature stores) without cloud dependencies. SQLite/file backends let you complete the whole exercise locally with no services to provision.

Q: What does the `feast apply` command do?
A: `feast apply` parses your feature definition files (entities, feature views, feature services) and registers them in the Feast registry, creating or updating the infrastructure Feast tracks. It is the step that turns your versioned Python definitions into the live registered state the store reads from. You run it after editing definitions and before materializing or querying.

Q: What is the Feast registry?
A: The registry is the catalogue of registered objects — entities, feature views, data sources, feature services — that Feast persists when you run `feast apply`. It is the source of truth for what features exist and how they are defined, and both the offline and online paths consult it. Backing it with a shared store (e.g. a database) lets a team collaborate on one consistent set of definitions.

Q: What is a feature service in Feast and why is it useful?
A: A feature service is a named bundle of features (often spanning multiple feature views) that a particular model consumes, giving the model one stable handle to request. It decouples the model's feature contract from the underlying view layout, so you can reorganize views without changing model code. It also documents exactly which features a given model version depends on.

Q: What is a data source in a Feast feature view?
A: The data source tells the feature view where the underlying feature data physically lives — for example a Parquet path, a BigQuery table, or a Postgres table — and which columns carry the timestamp and entity keys. Feast reads from it for both historical retrieval and materialization. Pointing a view at the wrong timestamp column is a frequent cause of incorrect joins.

Q: Why must each feature record carry an event/feature timestamp in its data source?
A: The timestamp is what point-in-time joins and materialization use to decide which value was current as of a given moment. Without a reliable per-record timestamp, Feast cannot order values in time, so it can neither prevent leakage nor know which value is "latest" to push online. It is the temporal backbone of the whole feature-store mechanism.

Q: A churn model trained with a naive join scores beautifully offline but fails in production. What is the first thing to check?
A: Suspect feature leakage from a latest-value join: check whether training features were attached as of each event's timestamp or whether current values bled in. Re-build the training set with `get_historical_features` and diff against the naive version; if the rows where future data was pulled are the ones driving the lift, that's your skew. Inflated offline metrics that collapse live are the signature of point-in-time violations.

Q: Why does buying a feature store for batch-only scoring add cost without value?
A: A feature store's distinctive components are the online store and the materialization pipeline that keeps it fresh for real-time lookups. Batch scoring reads history directly from the lakehouse and never issues an online lookup, so those components run, get paid for, and are never queried. You take on an extra database and scheduled jobs to operate for a serving path that does not exist.

Q: What components does a managed feature store (e.g. Databricks or SageMaker) add beyond open-source Feast?
A: Managed offerings typically add serving SLAs, a lineage and discovery UI, monitoring, and operational handling of the online store. These are the things the vendor pitch emphasizes. Whether they justify the cost depends on whether you actually have the real-time serving and team-discovery problems they solve.

Q: How would you write the "verdict note" the exercise asks for?
A: State which fund use cases justify an online store — those requiring a prediction inside a synchronous request, like real-time AML/fraud checks — and which do not, like nightly batch scoring. Then list the conditions under which you'd veto a purchase: no real-time scoring need, so no online-consistency problem to solve. Anchor it in your own observed leakage diff so the recommendation is evidence-based.

Q: What is the relationship between Feast and your existing lakehouse?
A: Feast does not replace the lakehouse; it uses it as the offline store, reading historical feature values from it for training and materialization. Feast adds the definitions layer, point-in-time retrieval, and (optionally) the online store on top. So adopting Feast is mostly about disciplined definitions and the online path, not migrating your data.

Q: Why are "effective-dated" lookups the right mental model for online stores too?
A: At serving time the online store holds the value effective as of the last materialization — the "current" effective-dated record for that entity. The freshness lag is exactly how stale that effective date can be relative to true business time. Thinking in effective dates keeps you honest about whether the online value is current enough for the decision being made.

Q: What happens at serving time if you fetch an entity that was never materialized to the online store?
A: `get_online_features` returns missing/null values for that entity because the online store only contains what materialization pushed. New entities, or entities outside the materialized window, simply have no online row. The fix is to ensure materialization covers them — this is a common cause of "the model gets nulls in production".

Q: If your TTL is too short, what symptom appears in training sets and online serving?
A: In training, features that update infrequently will frequently fall outside the lookback window and come back null, starving the model of valid signal. In serving, an entity whose latest value is older than the TTL can also read as missing. Too short a TTL trades staleness protection for coverage; too long lets genuinely stale values masquerade as current.

Q: If your TTL is too long, what risk do you introduce?
A: A long TTL lets Feast reach far back and attach a very old value as if it were current, so the model may train and serve on stale features without any null to warn you. For a fast-moving feature like 30-day flow volatility this means using values that no longer reflect reality. The TTL should match how long a feature value stays meaningfully valid.

Q: Why is re-running a historical Feast query expected to return identical results months later?
A: Because point-in-time joins are anchored to each row's `event_timestamp` and the TTL is relative to that timestamp, not to "now". The state of the world as of a past date does not change, so the reconstructed training set is reproducible. This determinism is what lets you re-audit or retrain on exactly the same data — a real asset in a regulated fund context.

Q: How is point-in-time correctness a regulatory/audit asset in fund administration?
A: It guarantees that a model's training data reflects only information available at each historical decision point, which is precisely the "no hindsight" standard auditors and DORA-style governance expect for model risk. You can demonstrate, row by row, that no future NAV or flow data leaked into a churn or risk model. The same mechanism makes the dataset reproducible for inspection.

Q: A naive join shows zero rows differing from the point-in-time join. What does that tell you?
A: Either the feature genuinely never changed across the relevant period, or your entity events are all newer than every feature update so no future value could be pulled — or the join keys/timestamps are misaligned and nothing is actually matching. Investigate before concluding the naive join is safe. Usually a slowly-moving feature plus recent events explains a small or empty diff; a totally empty result more often signals a join bug.

Q: Why does a feature store treat re-computation logic as the single source of truth for both paths?
A: Because the only durable guarantee against skew is that training and serving compute features from the same definition. If the offline path and online path each had their own logic, they would drift and reintroduce the skew the store exists to prevent. One feature view definition, materialized for online and scanned for offline, keeps them aligned.

Q: What does "millisecond lookups by entity key" buy you that a lakehouse scan cannot?
A: It lets you fetch one entity's current features inside a synchronous request's latency budget, which a scan-oriented lakehouse — built for throughput over many rows, not single-key latency — cannot meet. That is the whole reason the online store exists. If no request ever needs sub-second features, you don't need this capability and thus may not need a feature store.

Q: For a real-time AML check on a redemption, sketch the serving path with Feast.
A: A redemption request arrives, you extract the `investor_id`, call `get_online_features` for that investor's risk-relevant features (e.g. NAV staleness, recent flow volatility), and feed them to the model — all within the request. The online store returns the values materialization last pushed, so freshness depends on your materialization cadence. This is a case where the online store earns its keep.

Q: Why does materialization cadence matter more for an AML use case than for a churn batch?
A: An AML decision acts on near-real-time behaviour, so a stale feature could miss an emerging pattern; you'd materialize frequently to keep the freshness lag small. A churn batch reads history directly and is recomputed wholesale each run, so online freshness is irrelevant to it. Cadence is a function of how time-sensitive the decision is.

Q: How does Feast scan "backward in time" during a point-in-time join, conceptually?
A: For each entity row at its `event_timestamp`, Feast looks among that entity's feature records for the most recent one whose timestamp is at or before the event, and within the TTL window. It then attaches that value. Records after the event are ignored, which is what enforces correctness.

Q: What is the role of the entity join key in both training and serving?
A: The join key (e.g. `investor_id`) is how Feast matches feature records to the right entity — in training it joins history to each labelled row, in serving it looks up the row in the online store. It is defined on the entity and referenced by feature views. A mismatched or wrong join key produces silently empty or wrong feature columns.

Q: Why might you define multiple feature views over the same entity rather than one giant view?
A: Different features may come from different sources, update on different schedules, or warrant different TTLs, so separate views keep each definition coherent and independently materializable. It also lets a feature service compose just the views a given model needs. Smaller, focused views are easier to review and version, like well-factored dbt models.

Q: How does a feature store relate to dbt in your pipeline?
A: dbt builds and tests the curated tables in the lakehouse that become Feast's offline data sources; Feast then layers feature definitions, point-in-time retrieval, and online serving on top. They are complementary: dbt owns transformation-to-table, Feast owns feature-to-model with temporal correctness. Many shops generate Feast's source tables as dbt models.

Q: A stakeholder says "we need a feature store to do ML." How do you respond?
A: ML does not require a feature store; you can train and batch-score directly from the lakehouse. A feature store earns its place only when you have an online/offline consistency problem or a real-time serving latency requirement. Frame the decision around those specific needs rather than treating the store as a prerequisite.

Q: What is the danger of computing the online feature with different code than the offline feature?
A: It reintroduces training-serving skew: the model trains on one definition and serves on another, so production inputs subtly differ from training inputs and accuracy drifts. This is the exact failure a feature store is meant to prevent by having one definition feed both. If you hand-roll online features in app code, you've defeated the point.

Q: How would you demonstrate point-in-time correctness to a skeptical data engineer in five minutes?
A: Show two training sets for the same (investor, event_timestamp) rows — one from `get_historical_features`, one from a naive latest-value SQL join — and diff them. Point at the rows where the naive feature's source timestamp is after the event date: those are leaked future values. Then explain it as an effective-dated SCD2 lookup, which they already understand.

Q: What is the consequence of NAV-staleness as a feature being joined with a latest-value rather than point-in-time join?
A: A latest-value join would attach today's NAV-staleness to a historical event, so a label from last quarter would "know" how stale the NAV became afterward. The model learns a relationship using information unavailable at the historical decision point, inflating offline performance. Point-in-time correctness restricts each row to the staleness known as of its own event date.

Q: Why is "bigger online store" never the answer to a batch-only workload's problems?
A: Because a batch-only workload issues no online lookups, so scaling the online store improves nothing it does. The relevant resource for batch is offline-store/lakehouse scan throughput. Spending on online infrastructure for a path that never queries it is the classic over-provisioning the lesson warns against.

Q: What does it mean that feature views are "reviewable like dbt models"?
A: They live in version control as Python, so a change to an entity, a feature's logic, or a TTL goes through a pull request where reviewers can see the diff and reason about its effect on training and serving. This brings governance, change history, and accountability to feature logic. It contrasts with features defined inline in notebooks that no one reviews.

Q: How do you choose a TTL for the "investor churn signal" feature?
A: Pick a window over which a churn-signal value stays meaningfully representative of the investor — long enough to avoid spurious nulls when the signal updates infrequently, short enough that you don't attach a stale signal as if current. If churn signals refresh weekly, a TTL of a few weeks may be reasonable; validate by checking null rates and the point-in-time diff. TTL is a feature-specific decision, not a global constant.

Q: Why is materialization described as "pushing offline values into the online store on a schedule"?
A: Because the online store does not compute features; it just holds the latest precomputed value per entity for fast lookup. Materialization is the recurring job that reads the current value from the offline store and writes it online. The schedule's frequency sets how fresh those online values are.

Q: If serving latency is fine but predictions are subtly wrong, where do you look first in a Feast setup?
A: Check freshness: how stale are the online values relative to the decision, and is your materialization cadence too slow? Then check definition parity — is the online feature computed from the same view as training? Latency being fine rules out the online store's speed and points at staleness or skew.

Q: What is the difference between "latest value" semantics and "as-of value" semantics?
A: Latest-value returns whatever the current value is regardless of any reference date; as-of returns the value that was current as of a specified timestamp. Online serving correctly uses latest-value (the current state is the decision input), while training must use as-of to avoid leakage. Confusing the two in training is the core point-in-time error.

Q: Why does Feast separate the definition layer from the storage layer?
A: So the same versioned definitions can drive different physical backends — Parquet or BigQuery offline, SQLite or Redis online — and so the definitions, not the storage, are the reviewable source of truth. It also means adopting Feast doesn't force a data migration; it sits over your existing stores. This separation is what lets the local lab and a cloud deployment share definitions.

Q: A new entity appears in production but `get_online_features` returns nulls for it. What is happening and how do you fix it?
A: The entity has no row in the online store because materialization hasn't pushed values for it yet — perhaps it arrived after the last run or falls outside the materialized window. Run (or schedule more frequent) materialization covering it, and ensure the offline source actually contains its data. Cold-start entities producing online nulls is a routine operational gap.

Q: How does TTL prevent "silently using very old data" in a point-in-time join?
A: If the most recent value before an event is older than the TTL window, Feast treats the feature as missing rather than attaching that ancient value. This forces stale data to surface as a null you can detect and handle, instead of quietly degrading the model. Without TTL, the join would grab any prior value no matter how outdated.

Q: Why is the online store said to hold "only the latest value per entity"?
A: Because its job is millisecond serving, not history — keeping just the current value per entity key keeps lookups simple and fast and storage small. History lives in the offline store for training. This is also why online reads can't do point-in-time joins: there is no past to scan there.

Q: What would you put in the entity dataframe for the "Do" exercise step 2?
A: A list of (investor, `event_timestamp`) rows for the labelled events you want to train on — each row naming the `investor_id` and the timestamp at which you want features reconstructed, plus the label column. You pass this to `get_historical_features` with the three feature references. The timestamps drive the point-in-time join, so they must reflect the real decision moments.

Q: How does point-in-time correctness change your conclusions about a feature's predictive power?
A: A feature that looked highly predictive under a naive join often loses much of its apparent power once future leakage is removed, because the "predictiveness" was partly hindsight. Evaluating with `get_historical_features` gives an honest estimate of live performance. This is why correcting the join can deflate metrics — and why that deflation is good news, not bad.

Q: In Feast terms, what is the analogue of an SCD2 "valid_from"/"valid_to" window?
A: A feature record's timestamp acts as its effective-from, and the TTL bounds how long that record remains the valid answer before it is treated as expired — together approximating a valid window for point-in-time lookups. The `event_timestamp` is the as-of date you query against. So your SCD2 effective-dating instincts map almost directly onto Feast's temporal join.

Q: Why is "feature store is overkill for most fund use cases" a defensible architectural stance?
A: Because the dominant fund ML patterns — risk runs, segmentation, churn — are batch and read history directly from the lakehouse, needing neither millisecond serving nor online/offline consistency. The feature store's online machinery only earns value when a synchronous, real-time scoring need exists, which is the exception in fund admin. So the default is "no online store until a real-time use case proves otherwise".

Q: What does it mean to "veto a feature-store purchase" and on what basis?
A: It means recommending against buying or standing up a feature store, on the basis that there is no online serving or online/offline consistency problem to solve — i.e. all scoring is batch. The veto is conditional and evidence-based: name the use cases, show they are batch, and note the unused online-store cost. If a genuine real-time use case later emerges, the analysis is revisited.

Q: How does a feature store help with feature discovery and reuse across teams?
A: A central registry of defined feature views lets multiple teams find and reuse existing features rather than re-implementing them inconsistently, reducing duplicate logic and skew. Managed stores often add a discovery UI on top of this. The benefit only materializes if you have multiple teams and models — for a single batch model it is marginal.

Q: Why might Redis be chosen over Postgres as the online store?
A: Redis is an in-memory key-value store offering very low-latency lookups, which suits tight real-time serving budgets. Postgres can serve as an online store too and may be preferred when you want fewer moving parts or already operate it. The choice trades latency against operational simplicity and existing infrastructure.

Q: For the local lab, why is SQLite a reasonable online store choice?
A: SQLite is the default, file-based online store that needs no server, letting you complete the materialize-and-fetch exercise on a single Ubuntu laptop with zero services to provision. It is not for production latency or concurrency, but it faithfully demonstrates the online-store mechanics. You can later swap to Redis by changing config, not definitions.

Q: What is the difference between the offline store and the lakehouse in this lesson?
A: In this lesson they are effectively the same thing: your lakehouse serves as Feast's offline store, the place historical feature values are scanned for training and materialization. "Offline store" is Feast's role-name for whatever backend holds that history. So configuring Feast's offline store means pointing it at your lakehouse tables/files.

Q: A model's offline AUC drops sharply after you switch from a naive join to `get_historical_features`. Is this a bug?
A: Usually not — it is the expected effect of removing future-data leakage that had been inflating the naive AUC. The point-in-time number is the honest estimate of how the model will perform live. Treat the drop as a correction, then investigate only if the corrected performance is unusably low.

Q: Why does the lesson call online/offline consistency "the architecturally interesting part of ML serving"?
A: Because keeping training and serving features identical across two very different storage systems, time semantics, and update cadences is a genuinely hard distributed-data problem — unlike the model code, which is often routine. Point-in-time correctness and materialization freshness are where the real design decisions live. Getting them wrong silently breaks production models.

Q: How do you set the materialization schedule for the three fund features?
A: Choose a cadence matching how fresh each feature must be at serving time and how often the offline source updates — e.g. daily incremental materialization if features refresh daily — typically via `feast materialize-incremental` driven by a scheduler like cron or Airflow. The schedule defines the freshness window. For batch-only use you may not materialize at all.

Q: Why is materialization a "pipeline" concern rather than a one-off command?
A: Because the online store must be kept continuously fresh as new offline data arrives, materialization has to run on a reliable recurring schedule with monitoring and failure handling — it is an operational data pipeline. A one-off run only freezes a snapshot that immediately starts aging. Treating it as a scheduled job is what keeps serving values trustworthy.

Q: What is the risk of letting application code compute serving features instead of using `get_online_features`?
A: The app's logic will inevitably diverge from the training-time feature view, reintroducing skew that the feature store existed to eliminate. You also lose the single source of truth and the registry's documentation of what each model consumes. If you must compute features in app code, you've undermined the consistency guarantee.

Q: How would you explain to management why batch scoring needs "none of the online machinery"?
A: Batch scoring runs on a schedule, reads feature history straight from the lakehouse, and writes predictions to a table — there is no synchronous request, so there is nothing for an online store or materialization to serve. The online machinery exists solely to answer real-time lookups, which batch never makes. Paying for it would be paying for a capability you never call.

Q: What entity and key would you use for the NAV-staleness feature?
A: Likely a `fund` entity keyed on something like the fund identifier (e.g. its ISIN), since NAV staleness is a property of a fund/share class rather than an investor. The feature view then computes how stale the latest NAV is as of each event timestamp. Choosing the right entity ensures the feature joins to the correct grain.

Q: Why is ISIN a natural join key for fund-level features in a Feast deployment?
A: An ISIN uniquely identifies a security (here a fund share class), so it gives a stable, standardized entity key for fund-level features like NAV staleness across systems. Using a recognized identifier reduces ambiguity and join errors compared with internal codes. Feast just needs a consistent key; a standardized one like ISIN is ideal.

Q: How does the freshness window interact with regulatory expectations on data timeliness?
A: If a regulated decision (e.g. an investor-protection or AML check) depends on a feature, the materialization freshness window becomes part of your control: you must show the served value was sufficiently current for the decision. A lag that's acceptable for marketing churn may be unacceptable for a compliance check. So the cadence is partly a compliance, not just a performance, parameter.

Q: Why is point-in-time correctness "the training-serving skew killer"?
A: Because it eliminates the most damaging source of skew — features in training that secretly contain information unavailable at serving time. By restricting each training row to values knowable at its event, the model learns only relationships it can reproduce live. Other skew sources remain, but this one is the silent, high-impact killer it specifically defeats.

Q: What is the first diagnostic when `get_historical_features` returns unexpected nulls?
A: Check the TTL versus the gap between feature updates and event timestamps — if values are older than the TTL window they are correctly returned as null. Then verify the `event_timestamp` and join-key columns are named and typed correctly so the join matches. Nulls are often the join working as designed, not a failure.

Q: How does the offline/online split mirror the OLAP/OLTP distinction you already know?
A: The offline store is OLAP-like — scan-heavy, history-rich, optimized for large analytical reads (training). The online store is OLTP-like — point-lookup, latest-state, optimized for low-latency single-row access (serving). A feature store is essentially the bridge that keeps an OLAP history and an OLTP cache consistent for ML.

Q: Why is "diff the two outputs" the heart of the leakage demonstration?
A: Because the diff between the point-in-time join and the naive join is precisely the set of rows where future data entered, making an abstract correctness argument visible as concrete, countable rows. You can then point at a specific investor/event where the naive feature post-dates the event. Evidence beats assertion when you're justifying an architectural stance.

Q: When evaluating a managed feature store, what specific questions tie back to your batch-only context?
A: Ask whether you have any real-time scoring use case, since the serving SLA, online store, and monitoring they sell address exactly that — and if you don't, those features are dead weight. Also ask whether team-scale feature discovery is a real pain you have. If both answers are "no", the managed pitch doesn't fit your context.

Q: How would you phrase the one fund workload that wouldn't allow batch-only scoring?
A: A workload requiring a prediction inside a synchronous transaction — for example a real-time risk or AML decision on an incoming subscription or redemption that must complete before the transaction proceeds. It cannot wait for a nightly batch, so it needs online feature lookups. That is the single class of fund use case where the online store is justified.

Q: Why does the lesson stress that "your SCD2 instincts apply directly" to point-in-time joins?
A: Because the hard part — reconstructing the value that was effective as of a past date rather than the latest value — is identical to effective-dated lookups against a Type-2 dimension you already build. Recognizing the equivalence lets you reason about leakage and correctness with familiar tools instead of treating feature stores as exotic. It demystifies the "feature store magic" into known data-modelling.

Q: What is the relationship between materialization frequency and online-store cost/load?
A: More frequent materialization keeps online values fresher but increases compute and write load on the offline scan and online store, while less frequent materialization is cheaper but increases the freshness lag. You tune the cadence to the freshest the use case actually requires. For batch-only workloads the optimal frequency is effectively zero.

Q: How do you keep a Feast definition change from silently breaking an in-production model?
A: Treat feature views like dbt models: review the change in a pull request, reason about its effect on both training and serving, and re-run `feast apply` and re-materialize deliberately. Where a model depends on a specific contract, a feature service pins the set of features it consumes. Version control plus the registry give you the change history to audit.

Q: Summarize the architect's decision framework for whether to adopt a feature store at all.
A: First ask if any use case requires synchronous real-time scoring; if not, there is no online store need and the feature store is likely overkill. If yes, weigh whether the online/offline consistency and discovery benefits justify Feast or a managed store. Anchor the recommendation in your own leakage demonstration and freshness numbers, and be willing to veto when scoring is purely batch.

Q: What is the single biggest correctness mistake to avoid when building ML training sets from time-stamped data?
A: Joining the latest feature value to historical events instead of the value effective as of each event — i.e. skipping point-in-time correctness and leaking future data. It inflates offline metrics and destroys live performance. Always reconstruct features as-of the event timestamp, which is exactly what `get_historical_features` does for you.

Q: What is a stream feature view, and when would a fund workload need one?
A: A stream feature view ingests features from a streaming source so the online store stays fresh in near-real-time rather than only at batch-materialization cadence, shrinking the freshness lag toward seconds. A fund workload needs it only when a real-time decision depends on very recent behaviour — e.g. AML scoring on a live redemption — where even hourly materialization would be too stale. For batch-only scoring it is unnecessary complexity, consistent with the lesson's "online machinery is overkill" point.

Q: Why is an idle online store an operational liability, not just a wasted cost?
A: Beyond the licence/compute bill, an online store you never query still demands patching, monitoring, backups, capacity planning, and on-call coverage, and any materialization job feeding it is one more pipeline that can fail and page someone. So a feature store bought for a batch-only shop doesn't just sit unused — it actively enlarges the surface area you must operate and secure. That ongoing burden is a core reason to veto the purchase when no real-time use case exists.
