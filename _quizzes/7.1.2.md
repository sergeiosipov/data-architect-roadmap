# Phase 8 · 7.1.2 Platform semantic layers (Power BI semantic models) — 100 self-test questions

Q: What is a Power BI "semantic model" and why is it often called the de-facto semantic layer in Azure shops?
A: A semantic model is a Power BI/Analysis Services tabular model that holds tables, relationships, and reusable measures, presenting business-friendly entities and metrics on top of raw data. In Azure-centric organisations it frequently *is* the semantic layer because reports, Excel, and other tools all query the same model, so metric definitions like NAV and AUM live in one place. The risk is that without governance the same model gets re-created in dozens of workspaces, and the certified definition forks.

Q: Name the three classic semantic-model storage modes Power BI documents, plus the Fabric-era fourth.
A: The three documented modes are Import, DirectQuery, and Composite, and the Fabric-era fourth is Direct Lake. Import caches a compressed copy in memory, DirectQuery leaves data at source and federates queries, Composite mixes storage modes per table, and Direct Lake reads Delta/Parquet files from OneLake directly into the VertiPaq engine. Choosing among them trades latency, freshness, and cost.

Q: In Import mode, where does the data physically live and what must happen before a query runs?
A: Imported data is compressed by the VertiPaq engine and stored to disk, but it must be fully loaded into memory of the capacity before any query or refresh runs — there is no partial-load concept for an Import model. This gives very fast in-memory query performance but means the whole model competes for capacity memory. A NAV mart that compresses to 1 GB still has to fit entirely in RAM to be queried.

Q: Roughly what compression ratio does VertiPaq typically achieve on imported data, and why does it matter for capacity sizing?
A: Microsoft documents that it is reasonable to expect around 10x compression, so about 10 GB of source data can compress to roughly 1 GB in the model, with on-disk storage a further ~20% smaller than the compressed in-memory size. It matters because the compressed-in-memory size, not the raw source size, is what must fit within the capacity's memory guardrail. You compare the Power BI Desktop file size against Task Manager memory usage to gauge it.

Q: What is the single biggest data-freshness limitation of Import mode?
A: Import data is only as current as the last refresh, so the model must be refreshed (usually on a schedule) to stay current and a full refresh truncates and reloads every table. For a fund NAV that finalises late in the evening CET, you must schedule the refresh after the official striking, or reports show stale figures. Incremental refresh mitigates the cost of full reloads.

Q: How many scheduled refreshes per day does Power BI allow on shared capacity versus Premium/Fabric capacity?
A: Scheduled refresh is limited to eight times a day on shared capacity and 48 times a day on a Premium/Fabric capacity. This is a hard service limit on the *scheduled* refresh path; the XMLA endpoint is not bound by the 48/day cap. For near-real-time needs beyond 48 daily refreshes you generally move to DirectQuery, hybrid tables, or Direct Lake instead.

Q: In DirectQuery mode, what does the semantic model actually store, and how are results produced?
A: A DirectQuery model stores only metadata defining structure (tables, relationships, measures) and imports no data; at query time it translates DAX into native source queries (for example SQL) and runs them on the underlying database. Report users therefore see the latest source data on every interaction without a refresh. The trade-off is that the source must handle the report query load and may be slower than the in-memory engine.

Q: Give two legitimate reasons to choose DirectQuery over Import.
A: The two main reasons are data volumes too large to load or practically refresh even after reduction techniques, and a need for near-real-time data beyond what scheduled refresh limits allow. For example, a multi-billion-row tick history that cannot fit a capacity's memory, or a dashboard that must reflect intraday flows within minutes. You accept slower queries and DAX/M function restrictions in return.

Q: Why are DAX and Power Query functions restricted in DirectQuery models?
A: Because every expression must be translatable into a native query the source understands, DAX and Power Query M are limited to functions that can be transposed to native queries, and calculated tables are not supported. If a function cannot be pushed to the source, it simply is not available in DirectQuery. This is why complex time-intelligence or text logic often forces Import or a Direct Lake gold layer instead.

Q: What is Composite mode and what storage modes can an individual table take within it?
A: Composite mode mixes Import and DirectQuery (and can integrate multiple DirectQuery sources) by letting you set storage mode per table; a table can be Import, DirectQuery, or Dual. Dual means a table behaves as both Import and DirectQuery, and the engine picks the most efficient one per query. The classic pattern is Import/Dual dimensions over DirectQuery fact tables.

Q: In a composite model, why are dimension tables often set to Dual while the fact table is DirectQuery?
A: A Dual dimension can serve a slicer instantly from memory yet also participate in a single efficient native SQL join when the DirectQuery fact table is queried, so the engine generates one push-down query that joins and filters at source. If the dimension were pure Import, a join with the DirectQuery fact would force a less efficient two-step query. Dual gives the best of both for star-schema-shaped models.

Q: What is a "hybrid table" and how does it relate to incremental refresh?
A: A hybrid table is a single table with one or more Import partitions plus one DirectQuery partition, so most data is queried from fast in-memory while the newest rows come live from the source. The easiest way to create one is an incremental refresh policy in Desktop with "Get the latest data in real time with DirectQuery (Premium only)" enabled. It is supported only on Premium/Fabric capacity workspaces.

Q: Define Direct Lake mode in one or two sentences.
A: Direct Lake is a Fabric semantic-model table storage mode that loads data on demand straight from Delta tables in OneLake into the VertiPaq engine, giving Import-like query speed without replicating the data or refreshing a cached copy. It bridges the freshness of DirectQuery with the in-memory performance of Import. It is the recommended choice for the gold layer of a medallion lakehouse.

Q: Why is a Direct Lake "refresh" fundamentally cheaper than an Import refresh?
A: A Direct Lake refresh is just *framing* — it copies only metadata, analysing the latest version of the Delta tables and re-pointing the model at the newest Parquet files in OneLake, which takes seconds. An Import refresh, by contrast, produces a full cached copy of the data, consuming significant source and capacity CPU and memory and taking far longer. So Direct Lake pushes data-prep cost upstream into the lake.

Q: What is "framing" in Direct Lake?
A: Framing is the low-cost Direct Lake operation that scans the Delta table metadata and updates the model's references to point at the current set of Parquet files, defining the data the model exposes until the next frame. Because it moves only references and no rows, a frame typically completes in seconds. New data prepared in the lake becomes visible to the model once it is reframed (manually, on schedule, or via automatic updates).

Q: What is the key difference between "Direct Lake on OneLake" and "Direct Lake on SQL endpoint"?
A: Direct Lake on OneLake can read from one or more Fabric data sources backed by Delta tables and does *not* fall back to DirectQuery, whereas Direct Lake on SQL uses the SQL analytics endpoint of a single source for table/view discovery and permission checks and *will* fall back to DirectQuery when it cannot read a Delta table directly. On OneLake you can mix in Import tables for a composite model; on SQL you cannot combine with Import/DirectQuery/Dual in the same model. The fallback behaviour on SQL is governed by the model's "Direct Lake behavior" property.

Q: What causes a Direct Lake on SQL model to fall back to DirectQuery, and what happens if fallback is disabled?
A: Fallback occurs when the engine cannot load directly from a Delta table — for example a non-materialised SQL view, SQL-based granular access control, SQL row-level security, or exceeding a per-query guardrail. With fallback enabled, refresh succeeds with a warning and queries still return results but slower; with fallback disabled, those queries fail instead. Direct Lake on OneLake never falls back, so exceeding guardrails there fails the refresh entirely.

Q: What capacity licence does Direct Lake require, and does Import or DirectQuery share that requirement?
A: Direct Lake (both flavours) requires a Fabric capacity subscription (F SKU), whereas Import and DirectQuery work on any Fabric or Power BI licence including the free one. This is because Direct Lake is a Fabric-only OneLake feature. So a free Desktop user can build Import and DirectQuery models but cannot publish a working Direct Lake model without an F SKU capacity.

Q: On an `F2` Fabric SKU, what is the documented rows-per-table guardrail for a Direct Lake model, and what is the on-disk model-size cap?
A: On `F2` (and `F4`, `F8`) a Direct Lake model is capped at 300 million rows per table, with a maximum model size on disk/OneLake of 10 GB and roughly 3 GB max memory. The rows-per-table and Parquet-file guardrails are evaluated per query, while the model-size cap is evaluated at model level. Exceeding them on Direct Lake on OneLake fails the refresh, while on Direct Lake on SQL it triggers DirectQuery fallback if enabled.

Q: Which engine processes queries for Import and Direct Lake, and which one is different?
A: Import and Direct Lake are both processed by the in-memory VertiPaq engine, which is why their query performance is comparable, while DirectQuery federates the query to the underlying data source instead. That federation is what makes DirectQuery typically slower and dependent on source tuning. Knowing this distinction explains why Direct Lake can match Import speed yet avoid full data replication.

Q: A Direct Lake table was created via an XMLA external tool and queries error out. What is the likely cause?
A: Direct Lake tables created by XMLA applications start in an unprocessed state until the application sends a refresh (process) command, and queries against unprocessed tables error (on OneLake) or fall back to DirectQuery (on SQL). The fix is to issue a refresh so the tables get framed and become queryable. Always process a newly created model's tables before querying.

Q: Why can Direct Lake "analyse data volumes that exceed the capacity's max memory"?
A: Because Direct Lake pages only the columns needed to answer a query into memory rather than loading the whole model, the total data volume can exceed the capacity memory limit as long as no single query's working set does. This column-level, on-demand loading is the ROI argument for Direct Lake over Import. The downside is that if working sets are large, excessive paging in and out of OneLake can slow queries.

Q: Why is a star schema described as the prerequisite for a healthy Power BI semantic model?
A: Power BI's VertiPaq engine and DAX filter-propagation model are optimised for a Kimball-style star schema — narrow fact tables surrounded by dimension tables joined on single-column keys. Modelling this way yields simpler relationships, smaller models, faster queries, and predictable measure behaviour, whereas snowflaked or wide flat tables degrade compression and complicate filtering. Your Kimball dimensional work is therefore directly reused as the model's backbone.

Q: For the NAV mart exercise, what dimensions and which explicit measures should the star schema contain?
A: The star should have fund, share-class, and date dimensions surrounding a NAV/positions fact, with explicit measures for NAV, AUM, and net-flows defined in DAX. Explicit measures (rather than implicit drag-and-drop aggregations) are essential so every report computes the metric identically. Date should be a marked date table so time-intelligence works correctly.

Q: Why prefer explicit DAX measures over implicit aggregations for metrics like AUM?
A: Explicit measures centralise the calculation logic (filters, aggregation, timezone handling) in one named definition that every visual and tool reuses, which is exactly what makes the model a semantic layer rather than a pile of charts. Implicit aggregations are recomputed ad hoc per visual and silently drift, so two reports can show different AUM. A certified AUM measure must be explicit to be authoritative.

Q: What is row-level security (RLS) in a semantic model?
A: RLS restricts which rows a user can retrieve by attaching DAX filter expressions to roles, so members of a role only see the data their filter permits — for example a management-company filter so each admin sees only its own funds. It is enforced in the model (VertiPaq or live-connected AS), not just hidden in the report. Without RLS, anyone with report access can see every fund's data.

Q: What is the difference between a "dynamic" and a "static" RLS role?
A: A static role applies the same filter to every member of the role (for example, ManagementCompany = "ManCoA"), whereas a dynamic role filters by the report user's identity, typically using `USERPRINCIPALNAME()` in the DAX filter joined to a security mapping table. Dynamic roles scale to many users without one role per company, which is why they suit multi-tenant fund administration. Static roles are simpler when there are only a handful of isolation groups.

Q: Write the shape of a dynamic RLS filter that isolates a user to their own management company.
A: On a security or bridge table you apply a table filter DAX expression such as `[UserEmail] = USERPRINCIPALNAME()`, and relationships propagate that filter to the fund and fact tables. The `USERPRINCIPALNAME()` function returns the signed-in user's UPN, which you match against a mapping of users to management companies. This means one role serves every tenant, each seeing only its own funds.

Q: How do you verify RLS during development without logging in as another user?
A: In Power BI Desktop you use the "View as" (view as role) feature to test the report exactly as a member of a chosen role — or as a specified user for dynamic roles — and confirm the rows are filtered correctly. This lets you prove two management companies are isolated before publishing. In the service, role membership is then assigned to security groups or users.

Q: Where is RLS role *membership* assigned — in Desktop or in the service?
A: Role definitions (the DAX filters) are authored in Power BI Desktop before publishing, but role membership (which users or security groups belong to each role) is assigned in the Power BI service on the semantic model's security settings. SSMS or TMSL over the XMLA endpoint can also create and manage roles. Separating definition from membership lets the same model serve different tenants per environment.

Q: In a Direct Lake model, why is a "fixed identity" cloud connection strongly recommended when using RLS?
A: Direct Lake on OneLake reads files via OneLake access that does not observe SQL-based RLS, and the recommended pattern is a fixed-identity (single-sign-on-bypassing) cloud connection so the model itself enforces semantic-model RLS consistently rather than relying on each user's file permissions. Microsoft explicitly recommends a fixed identity for Direct Lake RLS. Without it, security can behave inconsistently between the two Direct Lake flavours.

Q: A Direct Lake on OneLake model ignores the SQL endpoint's row-level security — is that a bug?
A: No; with Direct Lake on OneLake, queries succeed and SQL-based RLS is *not* applied because access is granted at the OneLake file level, which does not observe SQL RLS. You must instead define semantic-model RLS in the model to isolate management companies. Direct Lake on SQL, by contrast, does honour SQL RLS but then falls back to DirectQuery for those queries.

Q: What is object-level security (OLS) and how does it differ from RLS?
A: OLS hides entire tables or columns from a role (the object is invisible, not just empty), whereas RLS filters which *rows* a role can see while the table and columns remain visible. You might use OLS to hide a sensitive fee-schedule column from one role and RLS to limit which funds another role can see. Both are model roles managed through Desktop, the service, or the XMLA endpoint.

Q: What is the XMLA endpoint and what protocol does it use?
A: The XMLA endpoint is the open-platform connection a Premium/PPU/Embedded/Fabric workspace exposes so external tools can talk to the underlying Analysis Services engine using the XML for Analysis (XMLA) protocol, the same protocol used by Azure AS and SQL Server AS. Traffic is fully encrypted. It turns a Power BI semantic model into something you can manage like an enterprise tabular database.

Q: What is the difference between XMLA read-only and read-write, and which is the default?
A: Read-only is the default for the semantic-models workload and lets tools query model data, metadata, events, and schema; read-write must be explicitly enabled and additionally permits management, advanced modelling, scripting, debugging, and deployment. Read-write is what unlocks pro-grade ALM. A capacity admin enables read-write in the Admin portal, and it applies to all workspaces and models on that capacity.

Q: Name several tools that connect over the XMLA endpoint and what each is for.
A: SSMS (DAX/MDX/XMLA queries, TMSL scripting, fine-grain refresh), Tabular Editor 2.x (open-source metadata editing), DAX Studio (DAX authoring and performance tuning), ALM Toolkit (schema compare/deploy for ALM), SQL Server Profiler (event tracing), Visual Studio/SSDT (enterprise model authoring), and Excel PivotTables (read-only analysis). Read-only suffices for querying; read-write is required for metadata and deployment operations. They use the MSOLAP/ADOMD/AMO client libraries as an abstraction layer.

Q: Which XMLA operations need read-only and which need read-write?
A: Query operations (browsing data, running DAX/MDX, Excel PivotTables, Report Builder) need only read-only, while metadata operations — scripting/altering the model with TMSL, deploying from Visual Studio, creating roles, refresh via XMLA, ALM schema merges — require read-write. So Tabular Editor and ALM Toolkit query with read-only but must have read-write to save changes back. Enabling read-write is a capacity-level governance decision.

Q: What licence or capacity is required to use the XMLA endpoint at all?
A: The XMLA endpoint is available on Premium, Premium Per User (PPU), Power BI Embedded, and Fabric capacity workspaces, not on shared capacity, and the connecting identity needs a PPU licence unless the model sits on Premium/Fabric capacity. Build permission on the model is the minimum for read access. This is why "pro-grade ALM" of semantic models presumes a capacity, not just Pro licences.

Q: Besides enabling read-write on the capacity, which tenant setting must also be on for XMLA to work?
A: The tenant-level Integration setting "Allow XMLA endpoints and Analyze in Excel with on-premises semantic models" must be enabled in the admin portal in addition to the capacity's read-write toggle; both are on by default. If a tool cannot connect even though read-write is set, this tenant switch is the next thing to check. Workspace/app security-group membership is still honoured on top.

Q: A connection to a workspace's XMLA endpoint fails with a duplicate-workspace-name error. How do you resolve it?
A: Power BI normally blocks duplicate workspace names, but if a collision exists you append the workspace `ObjectIDGuid` to the connection URL — for example `powerbi://api.powerbi.com/v1.0/myorg/Contoso Sales - <guid>` — copying the GUID from the workspace's `objectID` in its URL. The same trick (appending the model GUID) resolves duplicate semantic-model names within one workspace. Workspace names in the URL must also be URI-encoded per RFC 3986.

Q: Which kinds of semantic models do NOT appear over the XMLA endpoint?
A: Live-connection models to Azure AS or SQL Server AS, live connections to a Power BI model in another workspace, push (REST API) datasets, Excel-workbook models, and default Fabric semantic models cannot be managed via the XMLA endpoint and will not show under the workspace in SSMS. They are essentially pass-throughs or special types without a manageable tabular database. If a model is missing in SSMS, check whether it is one of these types.

Q: After an XMLA write operation on a Desktop-authored model, what can you no longer do, and what is the safeguard?
A: A write operation via XMLA on a PBIX-authored model prevents it from being downloaded back as a PBIX file, so the round-trip to Desktop is lost. The safeguard is to retain your original PBIX in source control before doing XMLA writes. This is a common gotcha when teams start editing published models with Tabular Editor.

Q: What authentication mode must SSMS use to connect to a Power BI workspace XMLA endpoint?
A: SSMS must use "Active Directory - Universal with MFA" authentication, with the workspace URL entered as the Analysis Services server name and your organisational user ID as the username. Once connected, the workspace shows as an AS server and each semantic model as a database. Standard SQL or Windows auth does not work against the Power BI XMLA endpoint.

Q: What are Fabric/Power BI deployment pipelines and what is the default stage layout?
A: Deployment pipelines are Fabric's built-in ALM tool that lets creators promote content through ordered stages in the service; you can have 2 to 10 stages, with a default starting layout of Development, Test, and Production. Content is cloned from one stage to the adjacent one, typically dev → test → prod. They are the no-code path to staged releases of semantic models and reports.

Q: What is "item pairing" in a deployment pipeline and why does it matter?
A: Pairing associates an item in one stage with the same item in the adjacent stage, so deploying overwrites the paired item instead of creating a duplicate; pairing happens when you assign a workspace to a stage or deploy unpaired content. If items are not paired, even identical-looking ones get duplicated rather than updated on deploy. Paired items appear on the same line and stay paired even if renamed.

Q: What are deployment-pipeline "deployment rules" used for?
A: Deployment rules let you change item configuration per stage on deploy — most importantly data source rules and parameter rules that rebind a semantic model to the stage-appropriate database (test DB in Test, prod DB in Production) instead of carrying the dev connection forward. This is how one model definition serves multiple environments cleanly. Note Direct Lake on OneLake does not support data-source rebind rules directly; you use a parameter expression in the connection string instead.

Q: How do deployment pipelines relate to git integration?
A: Deployment pipelines handle promotion of content *between environments* within the service, while git integration handles *version control* of workspace items in a repo; they are complementary, not the same feature. You typically use git for source control and history and pipelines (or the pipeline REST APIs) to move validated content to test and production. The pipeline docs explicitly point to git integration for versioning.

Q: Can deployment pipelines be automated, and how?
A: Yes — you can deploy programmatically using the deployment pipelines REST APIs, integrating them into Azure DevOps or other CI/CD so promotions are scripted and gated. This pairs with XMLA-based tooling (Tabular Editor, ALM Toolkit, `Invoke-ASCmd`) for the model-build side. Together they give a fully automated, auditable release path for the NAV semantic model.

Q: Beginning February 2026, what semantic-model requirement do deployment pipelines enforce?
A: From February 12, 2026, Fabric deployment pipelines retire support for semantic models that have not been upgraded to Enhanced Metadata, so older models must be upgraded to keep deploying through pipelines. Enhanced metadata is also what XMLA write operations on Desktop models require. If a model suddenly stops deploying, check whether it needs the enhanced-metadata upgrade.

Q: What does it mean to "endorse" content in Fabric/Power BI, and what are the three badges?
A: Endorsement marks high-quality, trustworthy content so users can find it, attaching one of three badges: Promoted, Certified, or Master data. Endorsed items get a visible badge and, in some lists, precedence (they appear first). It is the governance feature that fights the "forty AUM definitions" sprawl by signalling which model is authoritative.

Q: Who can apply the "Promoted" badge versus the "Certified" badge?
A: Any user with write permission on an item can Promote it (the creators saying it is ready to share), but only users explicitly specified by a Fabric administrator can Certify an item, although any user may *request* certification. Certification therefore reflects an organisation-authorised review, not just author confidence. Promotion is self-service; certification is gated.

Q: What does the "Master data" endorsement badge signify and where can it be applied?
A: The Master data badge marks an item as a core, authoritative single source of truth for key organisational data such as product codes or, in funds, the canonical instrument or fund master. It can only be applied to items that contain data — such as lakehouses and semantic models — and only by users the Fabric admin specifies. It is the strongest "this is the system of record" signal of the three badges.

Q: Are certification and master-data endorsement available by default?
A: No — certification and master-data endorsement are available only if a Fabric administrator has explicitly enabled them for the organisation, whereas promotion is available by default. Certification enablement can be delegated to domain administrators, allowing a different reviewer set per domain. So if certification is missing, an admin must turn it on and nominate certifiers.

Q: How does endorsement aid discovery and reduce model sprawl?
A: Endorsed items are clearly badged and, in discovery surfaces like the OneLake data hub and the "Power BI semantic models" connect dialog, are given precedence so users find the certified model first instead of cloning yet another copy. By steering consumers to one certified, authoritative model, endorsement directly counters the proliferation of redundant workspace models. The badge is the discoverability lever that makes governance stick.

Q: Name the semantic-model-sprawl failure mode this lesson warns about and the control that contains it.
A: The failure mode is certified metric definitions (like AUM or NAV) forking across dozens of redundant workspace semantic models, so different reports disagree and auditors notice. The containing control is a certified (or master-data) dataset combined with endorsement-driven discovery, so there is one authoritative model and consumers are guided to it. The governance rule names which layer owns which metric and the review step that enforces it.

Q: When does a redundant *headless* semantic layer become the wasteful choice, per this lesson's "Why"?
A: If an Azure shop already standardises on the Power BI semantic model as its layer, bolting on a separate headless layer (for example Cube) that nobody queries just duplicates definitions and cost. The "rule wrongly" warning is symmetric: pay for a redundant headless layer no one uses, or let Power BI models sprawl into forty conflicting AUM definitions. The rubric exists to pick deliberately.

Q: What is a "headless" semantic layer and how does it differ from the Power BI model?
A: A headless semantic layer (such as Cube) defines metrics and a data model centrally but exposes them to *any* BI tool via APIs (SQL/REST/GraphQL) rather than being bound to one visualisation product. The Power BI semantic model bundles the layer with Power BI's own consumption tools. Headless wins when many heterogeneous tools must share one metric definition; the Power BI model wins when the org is Power-BI-centric.

Q: For an Azure-only 50-person fund admin with everyone in Power BI, which layer does the rubric favour and why?
A: The Power BI semantic model, because the consumer base is homogeneous (everyone uses Power BI), the tool mix is single-vendor, and adding a separate headless layer would be pure overhead with no extra reach. One certified model with explicit NAV/AUM measures and RLS suffices. A headless layer would be a redundant cost no one queries.

Q: For a multi-tool 300-person admin (Power BI plus Tableau plus notebooks), what does the rubric suggest?
A: When consumers are diverse and span multiple BI tools, a headless layer (Cube-style) that serves all tools from one metric definition becomes attractive, because pinning the semantic layer to Power BI would leave Tableau and notebook users to redefine metrics and drift. The decision turns on consumer diversity and tool mix, not just headcount. Power BI can still consume the headless layer as one of many clients.

Q: For a multi-cloud group with data outside Azure, which way does the rubric lean?
A: A multi-cloud group typically leans toward a headless/decoupled semantic layer because the Power BI model ties you to the Azure/Fabric stack, whereas a tool- and cloud-neutral layer keeps definitions portable across clouds and consumers. Org shape (multi-cloud), tool mix, and consumer diversity all push away from a single-vendor binding. The architect must still weigh the operational cost of running a separate layer.

Q: What three axes does the lesson's decision rubric score "Power BI model vs headless layer" on?
A: It scores the choice by org shape, tool mix, and consumer diversity — essentially, is the organisation Power-BI-and-Azure-centric with homogeneous consumers, or multi-tool/multi-cloud with diverse consumers. Homogeneous and Azure-centric favours the Power BI model; diverse and multi-cloud favours headless. The point is to rule deliberately by these factors rather than defaulting.

Q: When comparing a Power BI model against a Cube model measure by measure, what categories of drift should you document?
A: You should document drift in measure logic (the aggregation/calculation itself), filter context (which filters each layer applies by default), and timezone handling (for example, NAV-date boundaries across CET/UTC). These are the places two independently-authored layers silently diverge. The deliverable is a per-measure mapping plus a governance rule naming the authoritative layer for each metric.

Q: What should a governance rule preventing metric drift between two semantic layers actually specify?
A: It must specify which layer is authoritative for which metric (e.g. NAV owned by the certified Power BI model, some cross-tool KPI owned by Cube) and the concrete review step that enforces it before a definition changes. Naming authority per metric, not per tool, prevents both layers claiming the same metric. Without the enforcing review step the rule is aspirational only.

Q: Why is "view as role" insufficient on its own to prove production RLS, and what extra check matters?
A: "View as role" proves the DAX filters work, but it does not prove the right *users* are mapped to the right roles, since membership is assigned in the service, not Desktop. You must additionally verify role membership (security-group mapping) in the published model and, for dynamic RLS, that the security mapping table is correct and current. A correct filter with wrong membership still leaks data.

Q: A published Import model shows yesterday's NAV. What is the first thing to check?
A: Check the scheduled refresh — Import data is only as fresh as the last successful refresh, so a failed or mistimed refresh leaves stale NAV. Look at the refresh history for failures, verify the schedule runs after the official NAV striking time in CET, and confirm data-source credentials are valid. The model is behaving correctly; the refresh pipeline is the issue.

Q: You deployed a model to the Test stage but it still reads the Development database. What did you forget?
A: You forgot a deployment rule (data source / parameter rule) on the Test stage to rebind the model's connection to the test database; without it, deployment carries the dev connection forward. Configure the data-source rule per stage so each environment points at its own source. For Direct Lake on OneLake, use a connection-string parameter expression instead of a direct rebind rule.

Q: In SQL Server Profiler over XMLA, what must you specify that other tools may not require?
A: You must specify an Initial Catalog — the specific semantic model (database) within the workspace — via Options > Connection Properties > "Connect to database" in the Connect to Server dialog. Profiler needs to know which model to trace because a workspace exposes many models as databases. Forgetting it leaves Profiler unable to attach to the right model.

Q: Why are XMLA refresh operations not subject to the 48-per-day scheduled-refresh limit?
A: The 48/day cap applies to the *scheduled* refresh path in the service, whereas refreshes triggered through the XMLA endpoint (via SSMS, PowerShell, TOM, Azure Functions) are not limited to 48 per day and are not bound by the scheduled-refresh timeout. This lets you do frequent, fine-grain partition refreshes — for example refreshing only the latest NAV partition many times a day. It is a key reason XMLA is "pro-grade".

Q: What is fine-grain (partition-level) refresh and why is it valuable for a fund NAV mart?
A: Fine-grain refresh uses the XMLA endpoint (TMSL/TOM) to process individual partitions — for example only today's NAV partition — without reloading all historical data. This is far cheaper and faster than a full model refresh and avoids re-striking unchanged history. With incremental refresh policies, partitions are created automatically and you can re-process a single historical partition when a late correction arrives.

Q: When using the XMLA endpoint for write operations on a large model, what setting should you enable and why?
A: Enable the model's "large models" (large semantic model storage format) setting, because it materially reduces the overhead of XMLA write operations — significantly so for models over ~1 GB compressed. Write operations become much faster against large models with this on. It is a recommended companion whenever you start managing big models via XMLA.

Q: Can the XMLA endpoint set data-source credentials when it deploys a model?
A: No — metadata pushed through XMLA can create the data-source *connections* but cannot set their credentials; credentials must be entered afterward in the model's settings page in the Power BI service. This is why a first-time Visual Studio/SSDT deployment creates the model but the processing step fails until credentials are supplied in the service. Set credentials, then refresh.

Q: A first-time SSDT deployment to a Premium workspace succeeds in creating the model but processing fails. Why, and what is the fix?
A: Processing fails because, unlike Azure/SQL Server AS, deploying to a Premium workspace cannot prompt for data-source credentials, so the load step has no credentials. The fix is to let metadata deploy (optionally with Processing Option set to "Do Not Process"), then set credentials in the service's data-source settings, after which subsequent deployments and refreshes process successfully. It is expected behaviour, not an error in your model.

Q: What does the audit-log operation `CreateDatasetFromExternalApplication` indicate?
A: It records that a semantic model was created through the XMLA endpoint by an external application (for example Tabular Editor or a script), one of several XMLA audit operations alongside `ConnectFromExternalApplication`, `EditDatasetFromExternalApplication`, `RefreshDatasetFromExternalApplication`, and `DeleteDatasetFromExternalApplication`. These let governance teams trace who changed models outside the portal. In a regulated fund context this audit trail supports change-control evidence.

Q: How does the XMLA endpoint enable git-based ALM for semantic models?
A: With read-write XMLA, tools like Tabular Editor serialise the model to text (TMDL/TMSL/JSON) that you commit to git, and ALM Toolkit does schema compare/merge to deploy changes across environments while retaining incremental-refresh history. This gives semantic models the same branch/PR/review lifecycle as code. It is the "pro-grade ALM" the syllabus calls out, beyond no-code deployment pipelines.

Q: What is TMDL and why does it matter for source-controlling a model?
A: TMDL (Tabular Model Definition Language) is a human-readable, folder-and-text representation of a tabular model, far more diff-friendly in git than a monolithic JSON BIM file, and it is supported in Desktop's TMDL view and by Tabular Editor. It lets reviewers see exactly which measure or RLS filter changed in a pull request. This is central to treating the NAV semantic model as reviewable code.

Q: For a Direct Lake model, why does well-tuned Delta-table layout (V-Order, row groups) matter?
A: Direct Lake reads Parquet/Delta files directly, so its performance depends on the physical layout — V-Order optimisation and appropriately sized row groups make column paging into VertiPaq fast and keep you within the per-table file/row-group guardrails. Poorly compacted Delta tables (many tiny files) blow past file-count guardrails and slow framing and queries. Table maintenance (OPTIMIZE/V-Order) is part of operating a Direct Lake gold layer.

Q: What is the medallion-architecture role that Direct Lake is described as ideal for?
A: Direct Lake is positioned as the ideal storage mode for the *gold* analytics layer of a medallion lakehouse, because the gold layer is curated, Delta-table data ready for BI and Direct Lake exposes it at Import-like speed without copying. Bronze/silver layers feed the gold tables via Spark, T-SQL, dataflows, or pipelines upstream. This keeps data-prep logic in the lake and the model thin.

Q: Why does Direct Lake "move data preparation to OneLake," and what is the architectural benefit?
A: Because Direct Lake reads prepared Delta tables rather than running Power Query inside the model, data preparation must happen upstream in OneLake using Spark, T-SQL DML, dataflows, or pipelines, which maximises reuse of that logic across many consumers. The benefit is one place to fix a transformation instead of duplicating M across models. The model becomes a thin semantic veneer over governed lake data.

Q: A self-service analyst lacks write access to the IT-managed lakehouse but needs an extra column. How should they extend a Direct Lake model?
A: Since Direct Lake depends on upstream lake prep they cannot do without write access, they can add Import-mode tables (which support Power Query inside the model) to a Direct Lake on OneLake model to build a composite model. This lets them augment without modifying the governed source. Direct Lake on SQL cannot mix storage modes in the same model, so the OneLake flavour is required for this composite.

Q: Which data types are unsupported in Direct Lake tables and how do you work around them?
A: Direct Lake does not support complex/nested Delta column types, nor Binary or GUID semantic types, so you must convert them to strings or other supported types upstream. String column values are also capped at 32,764 Unicode characters and non-numeric floats like `NaN` are unsupported. Encountering a model error on a GUID/ISIN-as-binary column points straight at this limitation.

Q: A Direct Lake on SQL model's relationship suddenly errors with duplicate values on the one-side. What rule was violated?
A: The one-side column of a relationship must contain unique values, and Direct Lake fails queries when duplicates are detected (it does not silently de-duplicate like some Import scenarios). For a fund dimension keyed by ISIN, a duplicate ISIN row breaks the relationship and the query. Enforce uniqueness in the upstream Delta table to fix it.

Q: Why must you mark a date table and avoid Desktop's auto date/time in a Direct Lake model?
A: Direct Lake on SQL does not support Power BI Desktop's auto date/time feature (which would create hidden calculated date tables), so you must mark your own date dimension as the date table and build relationships on real date columns. A proper marked date table is also best practice for correct time-intelligence on NAV-by-date measures. Relying on auto date/time will simply not work in that mode.

Q: Why can a Direct Lake model not be created in "My Workspace"?
A: Direct Lake models cannot be created in personal workspaces (My Workspace); they require a Fabric-capacity-backed collaborative workspace. This is consistent with Direct Lake's dependence on a Fabric capacity and OneLake. Attempting it forces you into a proper capacity workspace, which is also better for governance and sharing.

Q: Can you refresh a Direct Lake model through an on-premises or VNET data gateway?
A: No — Direct Lake (both on OneLake and on SQL) supports only cloud connections and cannot operate through any gateway, on-premises or VNET. Because the data already resides in OneLake, there is no gateway hop. If your source genuinely needs a gateway, Direct Lake is not the right mode; use Import or DirectQuery instead.

Q: What compatibility level must an XMLA-compliant tool support to work with Direct Lake semantic models?
A: The tool must support `compatibilityLevel` 1604 or higher to work with Direct Lake models and expose Direct Lake-specific metadata. Older client tooling below that level will not handle Direct Lake correctly. So if Tabular Editor or SSMS mishandles a Direct Lake model, check that the tool and its AMO/TOM libraries are current.

Q: How does "Direct Lake behavior" property let you tune the OneLake vs SQL fallback trade-off?
A: On a Direct Lake on SQL model the "Direct Lake behavior" property controls fallback: you can allow automatic fallback to DirectQuery (queries always return, slower when they fall back) or disable it (queries fail rather than silently degrade). Disabling fallback is useful when you want to guarantee in-memory performance and surface guardrail breaches loudly. Direct Lake on OneLake has no fallback, so this property is a SQL-flavour concern.

Q: What does it mean that Direct Lake on SQL "can use data from a single Fabric data source," and why is that a constraint?
A: Direct Lake on SQL is scoped to one Fabric lakehouse or warehouse (its SQL analytics endpoint), so it cannot natively span multiple sources, whereas Direct Lake on OneLake can use Delta tables from one or more Fabric sources. If your NAV model needs to join tables that live in two different warehouses, Direct Lake on SQL forces you to consolidate or shortcut first. Direct Lake on OneLake or a composite design is more flexible.

Q: Why should you prototype before committing a model to Direct Lake?
A: Microsoft explicitly recommends a proof of concept because Direct Lake performance hinges on well-tuned Delta tables and on staying within SKU guardrails (rows, files, model size), and a prototype de-risks discovering at production scale that you need a higher F SKU or better table maintenance. For a fund NAV mart, a POC reveals whether framing and paging meet the morning-deadline latency. It is cheaper to learn this before the rebuild.

Q: How does a semantic model enforce data permissions when it is a *live connection* to Analysis Services?
A: With a live connection, Power BI passes the report user's identity through to the external AS model, which enforces its own data permissions (including AS-defined RLS) using that identity. So security lives in the AS model, not in Power BI. This is why live-connection models do not appear over the Power BI XMLA endpoint — they are managed at the AS server.

Q: What is "ownership takeover" of a semantic model and when do you need it?
A: If you are not the model owner you see configuration in read-only mode and cannot change gateway/cloud connections; you must either ask the owner or take over ownership, including via the "Take Over In Group" REST API. You need it when the original owner leaves the organisation or to let automation manage refresh. In regulated shops this is governed because takeover changes who controls the model's credentials.

Q: Service principals cannot be used for which security feature in a semantic model?
A: Service principals do not work with RLS or OLS and cannot be added as model role members, so dynamic security must key off real user identities, not an app identity. They can still automate management tasks (provisioning, deploy, refresh) over XMLA. If a service-principal-driven embed needs row filtering, you implement it through fixed-identity connections and the embed token's effective identity, not by adding the SPN to a role.

Q: What does `USERPRINCIPALNAME()` return, and what is a gotcha when matching it in RLS?
A: `USERPRINCIPALNAME()` returns the signed-in user's UPN (typically their email-style identity); the gotcha is that the value in your security mapping table must match the UPN exactly (case and domain), or the filter returns no rows and the user sees nothing. Cloud vs on-prem identity formats can differ. Always test with "view as" specifying a real user to catch a mismatch before users complain.

Q: Why is "everyone sees everything" the dangerous default if you forget RLS on a multi-tenant fund model?
A: A semantic model with no RLS roles applies no row filter, so any user with build/read access to the model can query every management company's funds — a confidentiality breach in a multi-client administration context. RLS is opt-in: absence of roles means no restriction, not a safe default. The control is to define and test management-company isolation roles before sharing.

Q: How does Direct Lake's "automatic updates" feature reduce data latency, and what is the alternative?
A: With automatic updates enabled, the model automatically reframes to the latest committed version of its Delta tables so new data appears without a manual refresh schedule; the alternative is programmatic reframing (a refresh/framing call via API or XMLA) when you want to control exactly when new data becomes visible. For a NAV mart you often want controlled reframing so reports flip to final figures only after striking. Automatic updates suit dashboards that should always reflect the newest lake data.

Q: Why does the lesson tie storage modes to "A.4" / the Fabric context specifically through Direct Lake?
A: Direct Lake is the storage mode that only exists in Fabric and reads from OneLake, so it is the concrete link between semantic-layer choices and the Fabric platform decisions made elsewhere (the A.4 Fabric entry). Picking Direct Lake commits you to Fabric capacity and a lake-centric architecture. That coupling is exactly why the rubric weighs platform shape, not just performance.

Q: A regulator asks which Power BI model is the authoritative source for reported AUM. What governance artefacts answer this?
A: The Certified (or Master data) endorsement badge on the one authoritative model identifies it, backed by the documented governance rule naming that model as authoritative for AUM and the review step controlling changes to its measures. Endorsement plus the authority rule together give an auditable answer. This is precisely the certified-dataset control that contains definition sprawl.

Q: How would you demonstrate, end to end, that one model isolates two management companies?
A: Define an RLS role (static per ManCo or one dynamic role keyed on `USERPRINCIPALNAME()`), assign each company's users/security group to the appropriate membership in the service, and prove isolation with "view as" in Desktop and by signing in as a test user from each company — each sees only its own funds, NAV, and flows. Confirm no cross-company rows leak in either direction. That round-trip (define, assign, verify both sides) is the demonstration the "Done when" asks for.

Q: Why can implicit measures plus no certification produce "forty AUM definitions," and how do explicit certified measures fix it?
A: When each report author drags fields to create implicit aggregations and no model is certified, every workspace re-derives AUM with subtly different filters and timezone logic, and there is no authoritative one to discover. Explicit measures centralise the logic in a single certified model that endorsement surfaces first, so consumers reuse it instead of re-inventing. The fix is organisational (certify + endorse) as much as technical (explicit DAX).

Q: An ISIN-keyed fund dimension is snowflaked across three tables and reports are slow. What modelling change does this lesson prescribe?
A: Flatten the snowflake into a single star-schema dimension keyed by a clean fund/ISIN key joined directly to the fact, because VertiPaq and DAX filter propagation are optimised for star schemas, not chained dimension-to-dimension joins. The Kimball dimensional work is the prerequisite for a healthy, fast model. Fewer, wider dimensions with single-column relationships beat normalised chains here.

Q: Summarise the one-line decision: when does the Power BI semantic model suffice as the semantic layer, and when does headless win?
A: The Power BI model suffices when the organisation is Azure/Fabric-centric with a homogeneous Power-BI consumer base and a single-vendor tool mix; headless wins when consumers are diverse across multiple BI tools or clouds and need one metric definition served to all of them via APIs. Rule by org shape, tool mix, and consumer diversity. Choosing wrong costs either a redundant unused layer or sprawling, conflicting metric definitions.
