# Phase 2 · 3.1.4 Data lakehouse — 150 self-test questions

Q: What is a data lakehouse in one sentence?
A: A lakehouse is an analytical architecture that puts warehouse-style management features — ACID transactions, schema enforcement, indexing, versioning — directly on top of low-cost object storage holding open file formats, so one platform serves both BI/SQL and data science without copying data into a separate warehouse. The point is to get warehouse guarantees on lake economics. It is defined in the CIDR 2021 paper "Lakehouse: A New Generation of Open Platforms" by Zaharia, Ghodsi, Xin and Armbrust.

Q: Name the four decoupled layers of a lakehouse.
A: Object store (the physical storage substrate, e.g. S3/MinIO/ADLS), open table format (e.g. Apache Iceberg, Delta Lake or Hudi, which adds ACID and metadata over files), a catalog (which tracks which table points at which metadata and is the namespace), and one or more query engines (e.g. Trino, Spark, DuckDB). Each layer talks to the next through an open contract, so any one can be swapped without rewriting the others — that decoupling is the architectural thesis.

Q: Why is "decoupled layers" the central selling point of the lakehouse, architecturally?
A: Because each of the four layers can be replaced independently, you avoid a single vendor owning storage, format, catalog and engine all at once, which is exactly the lock-in a monolithic warehouse imposes. You can keep data in S3 while swapping Trino for Spark, or change catalogs without rewriting your Parquet files. The architect's job is to know precisely what each layer guarantees so they can reason about a swap's blast radius.

Q: In the lakehouse stack, what does the object store layer actually provide?
A: Durable, cheap, near-infinite byte storage addressed by keys, with no native notion of tables, transactions, or schema — just immutable objects (typically Parquet files) plus PUT/GET/LIST/DELETE. Everything table-like (atomicity, snapshots, schema) is added by the table-format layer above it. That is why you can point MinIO, S3 or ADLS Gen2 at the same Iceberg tables: the store is deliberately dumb.

Q: What does the table-format layer add that raw Parquet on object storage cannot give you?
A: It adds a metadata layer over the files that provides ACID transactions, snapshot isolation, schema enforcement and evolution, partition tracking, and time travel — turning a directory of files into a real table. Without it, two writers can corrupt a table because object stores have no multi-file atomic commit. Iceberg, Delta Lake and Hudi are the three mainstream implementations of this layer.

Q: What is the job of the catalog layer in a lakehouse?
A: The catalog is the namespace and pointer: it maps a logical table name like `funds.nav_daily` to the location of that table's current metadata file, and it is the component that performs the atomic commit when a table changes. Engines ask the catalog "where is table X right now?" rather than scanning storage. Examples include the Iceberg REST catalog, Apache Polaris, AWS Glue, Databricks Unity Catalog and Snowflake Open Catalog.

Q: What does the query-engine layer contribute, and why can several engines share one table?
A: The engine plans and executes SQL or dataframe queries — pruning, joining, aggregating — by reading the table format's metadata to know which data files to touch. Several engines (Trino for interactive SQL, Spark for heavy ETL, DuckDB for local analysis) can read the same Iceberg table concurrently because they all speak the open table-format spec and ask the same catalog. No engine owns the data; the data lives in open files in the store.

Q: According to the CIDR 2021 lakehouse paper, what was the core problem with the "two-tier" lake-plus-warehouse architecture it sought to replace?
A: Teams kept a cheap lake for raw/ML data and a separate warehouse for BI, which meant continuous ETL copying between the two, extra storage cost, staleness, and two sources of truth that drift. The paper argues a single lakehouse tier eliminates that duplication and the reliability problems of moving data twice. The fix is to add management features to the lake rather than copy data out of it.

Q: Who are the authors of the CIDR 2021 lakehouse paper, and what company are they associated with?
A: Matei Zaharia, Ali Ghodsi, Reynold Xin and Michael Armbrust, all associated with Databricks (and UC Berkeley for several). The syllabus cites it as "Armbrust et al.", but the full author list is Zaharia et al. They are the same people behind Apache Spark and Delta Lake, which colours the paper's framing.

Q: The paper says a metadata layer like Delta Lake stores its transaction log where? Why is that significant?
A: It stores the transaction log in the same object store that holds the data files (for example, a `_delta_log` directory in the same S3 bucket), rather than in a separate database. This means the table's history and the data travel together and you need no extra always-on metadata server, but it also means the object store's consistency behaviour matters for correctness. It is what lets the table format work on plain S3.

Q: According to the lakehouse paper, transactions in Delta/Iceberg/Hudi span how many tables at once?
A: One table at a time — these formats support single-table ACID transactions only, not cross-table atomic commits. The paper notes it should be extensible to multi-table transactions but that, as designed, you cannot atomically commit a change touching two tables. For a fund pipeline this means you cannot atomically update a `holdings` table and a `nav` table in one transaction; you orchestrate that at the job level.

Q: Name three warehouse capabilities you lose when you run queries directly against open-format files in a raw lake.
A: ACID transactions, data versioning (time travel / rollback), and indexing — and, more broadly, fine-grained security and management ergonomics. The lakehouse paper frames these as exactly the "rich management features" the table-format metadata layer is designed to add back. Without them, concurrent writers corrupt tables and you cannot audit what a table looked like last month.

Q: The lesson says the architect must articulate "which component carries which guarantee." Give an example of doing that.
A: For example: durability and cheap retention is carried by the object store; atomicity and snapshot isolation is carried by the table format, not the store; namespace, table-level access control and the atomic commit pointer is carried by the catalog; query performance and concurrency limits come from the engine. So if NAV files are corrupting under concurrent writes, you look at the table format and catalog, not the bucket. Naming the carrier tells you where to debug and what a swap risks.

Q: What does the lakehouse paper say the warehouse still does better — name the three areas the lesson highlights.
A: High concurrency (many simultaneous BI users), fine-grained security (row-/column-level access control), and ergonomics (mature tooling, governance, predictable SQL behaviour). The paper is honest that lakehouses had not fully closed these gaps, which is why even lakehouse-first shops often keep a warehouse for month-end BI. The architect should not oversell the lakehouse on these axes.

Q: Why is high concurrency historically a warehouse strength rather than a lake strength?
A: Warehouses were built around a managed, elastic compute service with a query optimizer, result caching, admission control and workload management tuned for hundreds of concurrent short BI queries. A raw lake with an open engine had to read metadata and prune files on every query with fewer of those built-in concurrency controls. At a fund administrator's month-end, when many analysts hit NAV dashboards at once, that concurrency engineering is exactly what you are paying the warehouse for.

Q: Why is fine-grained security cited as a warehouse advantage, and how do lakehouses try to close the gap?
A: Warehouses offer mature row-level, column-level and dynamic-masking controls enforced at the engine. Lakehouses historically only had file/object-level permissions, so masking a single column (say an investor's tax ID) was awkward. Modern catalogs — Unity Catalog, Polaris, Snowflake Horizon — close the gap by enforcing table/column/row policies at the catalog so every engine inherits them, but the architect must verify the chosen engine actually honours catalog-enforced policies.

Q: What does "ergonomics" mean as a warehouse advantage, in practical terms?
A: It means the day-to-day experience is smoother: a single SQL dialect that just works, automatic statistics and tuning, built-in workload management, mature BI-tool connectors, and fewer moving parts to operate. An open lakehouse asks you to assemble store, format, catalog and engine yourself and keep them compatible. The lesson's "just buy Snowflake" objection is largely an ergonomics-and-headcount argument.

Q: Per the lesson's "Why," what is the cost trap if you pick the wrong architecture for cold regulatory archives?
A: If you keep seven-year regulatory archives (which must be retained but are rarely queried) inside a priced warehouse, you pay warehouse storage and compute rates for cold data that should sit in cheap object-store tiers. The lakehouse lets those archives live as Iceberg/Parquet on cheap S3/Glacier-class tiers, queried only on demand. Picking warehouse-everything turns a compliance retention requirement into an ongoing budget line.

Q: Conversely, what do you lose by going "just use files on a share" for fund data?
A: You lose the concurrency, atomic multi-writer safety, schema enforcement and fine-grained security a fund administrator needs at month-end — flat files give you no atomic commit, so a half-written NAV file can be read as final, and no access controls beyond filesystem permissions. There is no time travel to prove what a figure was on a given date for an auditor. The table-format and catalog layers exist precisely to fix the "files on a share" failure modes.

Q: Why do open formats "de-risk vendor lock-in" according to the paper and lesson?
A: Because the data sits in open, documented formats (Parquet files, an open table spec) that many independent engines can read, no single vendor controls your ability to access your own data. Storage, format, catalog and engine can each be swapped independently, so leaving a vendor does not require a data migration. Lock-in shrinks to whichever single layer you let a vendor own.

Q: If storage, format, catalog and engine are each swappable, which one tends to be the real lock-in point in modern vendor bundles?
A: Increasingly the catalog — because the catalog enforces governance, security policies and the atomic commit, and proprietary catalogs (e.g. a vendor's managed Unity Catalog or Horizon) may gate write access or policy features behind the vendor even when the underlying files are open Parquet/Iceberg. Open files do not help if only one vendor's catalog can safely write to them. The architect should ask "who can write to and govern these tables if I leave?"

Q: A vendor says "your data is in open Parquet, so there is no lock-in." How do you pressure-test that claim?
A: Ask which catalog governs the tables, whether external engines can both read and write through an open catalog API, and whether row/column security travels with the data or lives only in the vendor's engine. Open file format is necessary but not sufficient; if only the vendor's catalog can commit writes or enforce policy, the lock-in has just moved up a layer. Verify by trying to register and write the same tables from a second engine.

Q: What is a "data swamp," and which author warns about it?
A: A data swamp is a data lake that has degraded into an unusable mess: tables with unclear ownership, cryptic column names, no quality enforcement, no lineage, and analysts building distrusted shadow copies. Bill Inmon, in "Building the Data Lakehouse," warns that without lakehouse discipline a lake turns into a swamp. The lesson uses this as the warehouse-veteran objection you must be able to answer.

Q: What is the core of the Inmon / warehouse-veteran distrust of data lakes?
A: That lakes historically lacked the disciplines warehouses enforce — governance, data quality, modelling, lineage and conformed definitions — so they accumulate ungoverned data nobody trusts. Inmon is also skeptical of pure ELT dumping raw data in and sorting it out later. His point is that the lakehouse only works if it borrows the warehouse's governance discipline, not just its SQL.

Q: How does the lakehouse answer the Inmon objection rather than just dismissing it?
A: By adding back the missing disciplines at the table-format and catalog layers: schema enforcement stops bad data landing, time travel and the transaction log give lineage and auditability, and the catalog gives ownership and governance. So the lakehouse is not "lake without rules" but "lake with the warehouse's guarantees re-implemented on open storage." Conceding that ungoverned lakes do become swamps is more persuasive than denying it.

Q: At an architecture level, how does Databricks position its lakehouse?
A: Databricks centres on Delta Lake (and now Iceberg interop) as the table format with Unity Catalog as the governance/catalog layer and Spark/Photon plus its SQL engine as compute, all over your cloud object store. The pitch is one governed platform for data engineering, BI and ML. The architect should note that the governance and operational glue (Unity Catalog) is where Databricks' stickiness lives.

Q: At an architecture level, how does "Snowflake-with-Iceberg" position itself differently from classic Snowflake?
A: Classic Snowflake stored data in its own proprietary internal format; "Snowflake-with-Iceberg" lets tables live as open Apache Iceberg in your object store, with Snowflake Open Catalog (a managed Polaris) or external catalogs governing them, so other engines can read the same tables. The pitch is "warehouse ergonomics and concurrency, but your data stays open." The lock-in shifts from the storage format toward the catalog and the compute pricing.

Q: At an architecture level, where does Microsoft Fabric / OneLake sit?
A: OneLake is a single tenant-wide logical data lake (built on ADLS Gen2) storing data in open Delta/Parquet, with Fabric providing the engines (Spark, Warehouse, Power BI) over it. The pitch is "one lake for the whole organization, deeply integrated with Power BI and the Microsoft stack." Its stickiness is the tight Microsoft 365 / Power BI integration and the Fabric capacity pricing model rather than the file format.

Q: A 50-person Luxembourg fund administrator asks "why not just buy Snowflake and be done?" Give the two-sentence architect rebuttal.
A: Snowflake is excellent for concurrent BI and fine-grained security, but pricing is workload-shaped — month-end NAV reporting spikes and seven-year regulatory archives can become an expensive line if cold data and idle warehouses are not managed, and historically its proprietary format risked lock-in. With Iceberg-on-object-store you keep cold regulatory data cheap and your tables open while still being able to attach Snowflake as one engine if you want its concurrency. The answer is "you can use Snowflake as an engine without surrendering your storage and format."

Q: Same client asks "why not just keep NAV files on a network share?" Give the architect rebuttal.
A: A share gives you no atomic multi-writer commit, so a partially written NAV file can be read as final; no schema enforcement, so a malformed EMT/EPT file silently corrupts downstream; no time travel, so you cannot prove to an auditor what a figure was on a given date; and only filesystem-level security. A lakehouse adds exactly those — ACID, schema, versioning, fine-grained access — on top of storage that is just as cheap. Files-on-a-share fails precisely at month-end concurrency and regulatory audit.

Q: For the build-vs-buy memo, what are the four scoring axes the lesson asks you to use?
A: Cost, in-house skills required, lock-in, and EU compliance / data-residency posture. Each candidate (open lakehouse, Databricks, Fabric) is scored on all four so the recommendation is defensible rather than a gut call. For a Luxembourg fund administrator, data-residency and EU compliance often carry disproportionate weight.

Q: Why does "in-house skills required" weigh against the fully open lakehouse for a 50-person shop?
A: Assembling and operating MinIO/S3 plus Iceberg plus a catalog plus Trino yourself demands engineers who can keep four layers compatible, tune compaction, run the catalog, and handle upgrades — scarce skills for a 50-person fund administrator. A managed platform trades cost and lock-in for far lower operational headcount. The architect must weigh whether the team can actually run what they build.

Q: Why is EU compliance / data-residency posture a first-class axis for a Luxembourg fund administrator?
A: Fund data is subject to EU data-protection and sector rules (and DORA-era operational-resilience expectations), so where data physically sits and who can access it matters legally, not just technically. You must confirm the platform offers EU regions, keeps data resident, and supports the access controls and audit trails regulators expect. A cheaper platform that cannot guarantee EU residency may be disqualified regardless of its technical merits.

Q: The "Do" asks, for each lakehouse component, "what breaks if you swap that vendor." What breaks if you swap the object store?
A: Generally the least breaks if the table format is open: pointing Iceberg at a new S3-compatible store mostly means re-homing files and updating locations, since the format does not depend on a specific store. The gotchas are store-specific semantics — atomic rename on ADLS Gen2 vs none on S3, different consistency and per-request pricing — and any lifecycle/retention rules that must be re-created. Engines and catalog can usually stay put.

Q: What breaks if you swap the table format (e.g. Iceberg to Delta)?
A: A lot: the metadata layout, transaction log, snapshot mechanics and engine connectors all change, so you must rewrite or migrate every table and re-point every engine and catalog at the new format. Time-travel history may not carry over cleanly. This is the most invasive swap because the format defines the table's on-disk contract that everything above depends on.

Q: What breaks if you swap the catalog?
A: You change the namespace and the component that performs atomic commits and (often) enforces governance, so every engine must be re-pointed at the new catalog, table registrations re-created, and security policies re-implemented if they lived in the old catalog. If history of who-owns-what or column-level policies were catalog-resident, those must be migrated. Because the catalog is increasingly the lock-in point, this swap is where vendor portability is really tested.

Q: What breaks if you swap the query engine?
A: Usually the least disruptive swap with open formats: the new engine reads the same Iceberg tables via the same catalog, so data does not move. What breaks is SQL-dialect differences, engine-specific functions and tuning, performance characteristics, and any engine-only features (materialized views, caching) you relied on. You re-test queries and dashboards rather than migrate data.

Q: The "Done when" asks you to "name, for each component, what breaks if you swap vendors." Summarize the swap blast radius from smallest to largest.
A: Engine swap is smallest (re-test SQL, data stays put), object-store swap is small-to-medium (re-home files, mind store semantics and lifecycle), catalog swap is medium-to-large (re-point engines, migrate policies and registrations), and table-format swap is largest (rewrite/migrate every table, re-connect everything). Knowing this ordering is exactly what lets an architect reason about portability. The open formats are what keep the engine and store swaps cheap.

Q: What is Apache Iceberg, in one line?
A: Apache Iceberg is an open table format that sits over Parquet/ORC/Avro data files in object storage and adds a metadata tree giving ACID transactions, snapshot isolation, schema and partition evolution, and time travel. It is the table-format layer in the FOSS stack the lesson assembles (MinIO + Iceberg + Trino + DuckDB). As of 2026 it has become the de-facto open table-format standard.

Q: What is the `metadata.json` file in an Iceberg table?
A: It is the table's current metadata file holding the schema, partition specs, sort orders, the list of snapshots, and table properties; it is the root of the metadata tree that points down to manifest lists, manifests, and finally the data files. Every change writes a new metadata file (for example `v2.metadata.json`) and the catalog atomically swaps its pointer to it. Reading a table starts from whichever `metadata.json` the catalog currently points at.

Q: How does Iceberg achieve an atomic commit on an object store that has no multi-file atomic operation?
A: It writes new immutable data and metadata files first, then performs a single atomic pointer swap in the catalog from the old `metadata.json` to the new one; readers see either the old or the new state, never a half-written mix. Because the only atomic step is the single pointer swap, no atomic rename of many files is needed. If two writers race, only one pointer swap wins and the loser retries — that is optimistic concurrency.

Q: What is an Iceberg snapshot, and what feature does it enable?
A: A snapshot is a complete, immutable view of the table's data at one commit — the set of data files valid at that moment. Keeping past snapshots enables time travel (query the table "as of" a timestamp or snapshot id) and rollback. For a fund administrator this is how you reproduce exactly what a NAV table showed on a given valuation date for an auditor.

Q: What is the Iceberg REST catalog, and why does it matter for the catalog layer?
A: It is a standardized HTTP/JSON catalog API (an open spec) so any compliant client can create, read and commit tables against any compliant catalog server, regardless of the storage behind it. It matters because it makes the catalog layer itself open and swappable, reducing catalog lock-in. Implementations include Apache Polaris, Snowflake Open Catalog, and others that speak the REST spec.

Q: As of 2026, what notable capability did Iceberg v3 add, and what does it speed up?
A: Iceberg v3 (GA across major platforms in the first half of 2026) added deletion vectors, which mark rows as deleted instead of rewriting whole data files. This speeds up updates, merges and deletes because you avoid the cost of rewriting large Parquet files for small row-level changes. For high-churn fund tables (corrections to holdings or NAV) this materially reduces write amplification.

Q: What is Trino's role in the FOSS lakehouse stack?
A: Trino is a distributed, in-memory SQL query engine that connects to Iceberg tables (and many other sources) and is tuned for interactive, federated analytics across data it does not store itself. In the stack it is the high-concurrency SQL engine over the open tables, complementing heavy batch engines. It reads the Iceberg metadata to prune files and pushes down filters.

Q: What is DuckDB's role in the FOSS lakehouse stack, and how does it differ from Trino?
A: DuckDB is an embedded, single-node analytical SQL engine — it runs in-process (for example via `uv run` in Python) and is ideal for local development, testing and moderate-sized analysis directly over Parquet and Iceberg. Unlike Trino, it is not a distributed cluster; it trades horizontal scale for zero-ops simplicity on one machine. In the stack it is the local/laptop engine, Trino the distributed one.

Q: What is MinIO's role in the FOSS lakehouse stack?
A: MinIO is an S3-compatible object store you can run locally (e.g. in Docker) that serves as the physical storage substrate for the whole lakehouse, standing in for ADLS Gen2 or S3 in the cloud. Because it speaks the S3 API, code written against MinIO ports to real S3 with little change. It is the bottom layer the table format and engines sit on.

Q: Why is the lakehouse called "the target architecture for most fund platforms" in this lesson?
A: Because fund platforms need both cheap, auditable, long-retention storage for regulatory data and warehouse-grade concurrency and security for month-end NAV reporting — the lakehouse's lake-economics-plus-warehouse-guarantees is exactly that combination. The architect is expected to design one and defend it against both "just buy a warehouse" and "just use files." It balances cost, compliance and concurrency in one tier.

Q: Why does the lesson stress that you must "defend" your lakehouse design, not just build it?
A: Because in a vendor-selection or steering-committee setting you will face credible counter-pressures ("Snowflake is simpler," "files on a share are cheaper") and a CFO who sees cost, and your design only survives if you can name specific consequences of each alternative. An undefended design loses to whoever argues most confidently. The whole T1 lesson is about being able to articulate trade-offs, not just draw boxes.

Q: A fund's bronze NAV files land continuously and an ML job reads them while a BI job queries them — what does the table format buy you here?
A: Snapshot isolation: the BI and ML readers see a consistent snapshot of the table even while a writer commits new NAV files, and the writer's commit is atomic so no reader ever sees a half-written batch. On a raw share, a reader could pick up a partially written file and report a wrong NAV. This is the concurrency guarantee that makes a multi-consumer fund pipeline safe.

Q: Schema enforcement: how does it protect a fund pipeline ingesting EMT/EPT files?
A: The table format rejects or quarantines writes whose columns or types do not match the table's declared schema, so a malformed or changed EMT/EPT field cannot silently land as garbage and propagate into NAV or regulatory reports. This is one of the warehouse disciplines Inmon says lakes lack and lakehouses restore. The architect pairs enforcement with explicit schema evolution for the cases where the format legitimately changes.

Q: Why is time travel specifically valuable in a regulated fund-administration context?
A: Because regulators and auditors ask "what did this figure show on this date, and can you prove it wasn't altered" — time travel lets you query a table as of a past snapshot and demonstrate the exact state used for a given NAV or filing. The transaction log gives an immutable audit trail of every change. On flat files you would have to keep manual dated copies and hope nobody overwrote them.

Q: An ISIN-keyed holdings table needs corrections without losing history. How does the lakehouse support that?
A: You commit updates as new snapshots (using deletion vectors / merge-on-read in Iceberg v3 to avoid rewriting whole files), and every prior snapshot remains queryable for audit, so a correction does not erase what was previously reported. The catalog's atomic commit ensures concurrent readers never see a partial correction. You get mutability with a full, auditable history — exactly what flat files cannot offer.

Q: "Storage/format/catalog/engine can each be swapped independently" — why does the lesson treat this as a risk-management statement, not just a technical one?
A: Because independent swappability is what bounds your exposure to any one vendor failing, raising prices, or being acquired — it converts a potential full-platform migration into a single-layer change. For a regulated administrator, being able to credibly answer "what is our exit plan from vendor X" is a governance and resilience requirement (think DORA). So the decoupling is a continuity-of-operations argument as much as an engineering one.

Q: Two Spark jobs write to the same Iceberg table at once and one fails with a commit conflict. What happened and is the data safe?
A: Iceberg uses optimistic concurrency: both jobs prepared commits against the same base snapshot, only one atomic pointer swap could win, and the loser's commit was rejected — so the data is safe and consistent, not corrupted. The losing job should retry against the new current snapshot. This is the expected, correct behaviour, not a bug; the architect plans retries rather than trying to prevent the conflict.

Q: A NAV query over Iceberg is slow and scanning far more files than expected. What is the first thing to check?
A: Check partitioning and file layout / clustering and whether the query's predicates align with the partition spec and column statistics so Iceberg can prune files — and check for many tiny files needing compaction. The lakehouse paper lists data layout/clustering and auxiliary structures (statistics, indexes) as the performance levers. If predicates do not match partitioning, the engine cannot prune and reads everything.

Q: According to the lakehouse paper, what performance techniques let a lakehouse approach warehouse speed despite using open formats?
A: Caching hot data, optimizing data layout (clustering/ordering files so queries skip irrelevant ones), and maintaining auxiliary data structures such as statistics, zone maps and indexes to enable file/row-group skipping. These are added by the format and engine layers over plain Parquet. The argument is that openness need not mean slow, because the optimizations live above the file format.

Q: Why does keeping data in Parquet (a columnar format) matter for the lakehouse performance story?
A: Columnar storage lets engines read only the columns a query needs and skip the rest, and it compresses well and carries per-row-group statistics (min/max) that enable predicate-based skipping. That gives the lakehouse much of a warehouse's scan efficiency on open files. The table format layers metadata on top of these Parquet files rather than replacing them.

Q: What is "small files" / compaction, and why does it bite fund pipelines specifically?
A: Streaming or frequent micro-batch writes (e.g. NAV updates landing through the day) create many tiny data files, and queries that must open thousands of small files are slow and rack up object-store request costs. Compaction rewrites them into fewer, larger files. Fund pipelines with frequent corrections and intraday landings are prone to this, so the architect schedules compaction as routine maintenance.

Q: Why can't you just rely on the object store's permissions for fine-grained security in a lakehouse?
A: Object stores grant access at the bucket/prefix/object level, which cannot express "this user may see all columns except investor tax IDs" or "only rows for fund X." Fine-grained row/column security has to be enforced higher up — at the catalog and/or engine. This is a known gap versus warehouses, which is why catalog-enforced policies (Unity Catalog, Polaris, Horizon) are central to a regulated design.

Q: "Even lakehouse-first shops keep a warehouse." Why might a fund administrator still run one alongside the lakehouse?
A: For the warehouse's strengths the paper concedes — very high concurrency for many simultaneous BI users at month-end, mature fine-grained security, and predictable ergonomics for finance/reporting teams. The lakehouse holds the cheap, open, long-retention data and ML workloads; the warehouse serves the high-concurrency governed BI layer. The two-tier setup can be deliberate, not just legacy.

Q: How would you explain to a CFO that warehouse cost is "workload-shaped, not data-shaped"?
A: Cost is driven by how much compute you run — concurrent queries, full scans, idle-but-running warehouses — far more than by how many terabytes you store, so a month-end reporting spike or an unpartitioned full-table scan can cost more than a year of cold storage. The lever is controlling compute and query shape, not deleting data. This is why a runaway query, not a big dataset, is usually the budget incident.

Q: In a managed lakehouse vendor's bundle, how do you identify "the genuine lock-in point"?
A: Find the single layer the vendor controls that you cannot replace without a migration — typically the layer that holds governance and the atomic commit (the catalog) or the proprietary compute pricing, even when files are open. Ask: if I cancel tomorrow, can another engine still read and write these tables under the same policies? Whatever answer is "no" is your lock-in point.

Q: Databricks' bundle: where does the lesson-style lock-in tend to sit?
A: In the governance/operational layer — Unity Catalog and the surrounding platform glue — rather than the file format, since data can be Delta or open Iceberg in your own object store. You can read the files elsewhere, but the policies, lineage and managed-table conveniences are Databricks-native. So the question on exit is whether your governance survives the move.

Q: Snowflake-with-Iceberg: where does lock-in tend to sit?
A: In the compute/pricing model and the catalog governance (Horizon / Open Catalog) rather than the storage format, since the tables can be open Iceberg in your bucket. You keep the data, but Snowflake's concurrency engine, security features and credit pricing are the sticky parts. The architect checks whether external engines can write the same tables through an open catalog API.

Q: Fabric / OneLake: where does lock-in tend to sit?
A: In the tight integration with Power BI and the Microsoft 365 / Fabric capacity ecosystem, plus the Fabric capacity pricing model, rather than the open Delta/Parquet files in OneLake. The data is portable; the deep Power BI/governance integration and the capacity-based billing are the gravity. For a Microsoft-centric fund shop that integration is a feature, but it is also the exit cost.

Q: Why is "open table format" necessary but not sufficient to avoid lock-in?
A: Open files mean other engines can read your data, but if only the vendor's catalog can safely commit writes, or governance/security policies live only in the vendor's platform, you still cannot fully operate the tables elsewhere. Lock-in moves from the file format up to the catalog/governance layer. So the architect must check the catalog and write path, not just the file format, before claiming portability.

Q: How does the lakehouse's single-tier design reduce a class of data-quality bugs versus the two-tier lake-plus-warehouse setup?
A: By removing the repeated ETL copies between lake and warehouse, you remove the windows where the two copies drift, a load silently drops rows, or transformations diverge — there is one governed source of truth. Fewer copies means fewer reconciliation breaks between, say, the lake's NAV and the warehouse's NAV. The paper frames eliminating that duplication as a reliability win, not just a cost win.

Q: What is the difference between schema enforcement and schema evolution in a table format?
A: Enforcement rejects writes that do not conform to the current schema (protecting against bad/changed data), while evolution lets you deliberately and safely change the schema over time — add a column, rename, widen a type — without rewriting all data files. Iceberg tracks columns by stable IDs so evolution does not break old data. A fund pipeline uses enforcement to catch malformed feeds and evolution to absorb a legitimate new EMT/EPT field.

Q: An auditor asks you to reproduce yesterday's NAV table exactly. Which lakehouse feature do you reach for and how?
A: Time travel: query the Iceberg table as of yesterday's snapshot id or timestamp to return precisely the rows valid then, backed by the immutable transaction log proving no later edits altered that view. You do not need a manual dated backup. This is a concrete reason the format layer, not the bare object store, must carry the versioning guarantee.

Q: Why does the lesson pair "object store + open table format + catalog + query engines" as decoupled rather than as one product?
A: To make explicit that a lakehouse is an architectural pattern of independent, contract-bound layers — not a single SKU — so you reason about each layer's vendor and guarantee separately. A managed product bundles all four, which is convenient but concentrates lock-in. Seeing the four layers is what lets you compare a bundle against an assembled-open stack honestly.

Q: A stakeholder says "Iceberg, Delta, Hudi — they're all the same, just pick one." How do you respond as an architect?
A: They solve the same problem (ACID/metadata over files) but differ in maturity of features, engine support, catalog ecosystems and write mechanics, and swapping format later is the most invasive change — so the choice is consequential. As of 2026 Iceberg has the broadest cross-engine and catalog support, which lowers lock-in. The decision deserves a real evaluation, not a coin flip, because it is the hardest layer to swap.

Q: What does "open direct-access data formats" mean in the paper, and why is it foundational to the lakehouse claim?
A: It means data is stored in open, documented formats (like Parquet) that any tool can read directly from the object store without going through a proprietary engine's API. This is foundational because it is what makes the format, catalog and engine independently swappable and breaks the warehouse's "data is hostage inside our system" model. Without direct-access open formats there is no real decoupling.

Q: Why does the lakehouse paper emphasize first-class ML/data-science support as a motivation?
A: Because data scientists need direct file access to large datasets for training and feature engineering, which classic warehouses (SQL-only, data locked in proprietary format) made awkward, forcing yet another export. The lakehouse keeps data in open files an ML framework can read directly while still offering SQL/BI on the same tables. One copy serves both BI and ML — a core selling point.

Q: How does running MinIO locally help you reason about a cloud lakehouse for a Luxembourg fund?
A: MinIO speaks the S3 API, so a local MinIO + Iceberg + Trino/DuckDB stack lets you prototype the exact layering, partitioning and lifecycle behaviour you would run on ADLS Gen2 or S3 in an EU region — testing concurrency, schema enforcement and time travel before committing to a cloud bill or vendor. It de-risks the design cheaply. You validate the pattern locally, then re-home the object store to an EU-resident cloud store.

Q: What is the practical meaning of "the warehouse still does ergonomics better" for a small fund team's day-to-day?
A: It means with a warehouse the analysts mostly write SQL that just works, with tuning, security and connectors handled, whereas an open lakehouse asks the team to operate four layers and keep them compatible — a real burden for a 50-person shop. Ergonomics translates directly into how much engineering headcount you need. This is often the deciding factor in build-vs-buy for small administrators.

Q: How can a lakehouse cause "warehouse-priced cold archives" if you design it badly?
A: If you put cold seven-year regulatory data into a priced warehouse engine's storage instead of into cheap object-store tiers with lifecycle rules, or you query archives with an always-on expensive engine, you pay warehouse rates for data that should be near-free at rest. The fix is keeping cold data as open files on cheap tiers, queried on demand by a serverless/scaled-down engine. Bad tiering, not the lakehouse pattern itself, creates the cost.

Q: Why is the atomic commit specifically a property of the catalog/format rather than the object store?
A: Because object stores offer no multi-file atomic operation, the table format arranges that the only atomic step is a single pointer swap, and that swap is performed/coordinated by the catalog. So atomicity is engineered above the store, using the store's one cheap guarantee (atomic single-object write/pointer update). The store stays dumb; the catalog/format make the table transactional.

Q: A new engine you want to add cannot write to your tables, only read them. What does this tell you about your catalog?
A: It signals the write path and governance are gated by your current catalog/vendor and not exposed through an open, multi-writer catalog API — so the catalog is your lock-in point even though reads are open. To make writes portable you need an open catalog (e.g. an Iceberg REST catalog like Polaris) that any compliant engine can commit through. Read-openness alone does not give you a real exit.

Q: How does the lakehouse pattern support DORA-style operational-resilience and exit requirements?
A: Because storage, format, catalog and engine are independently swappable and data sits in open formats, you can demonstrate a credible exit/continuity plan from any single vendor without a wholesale data migration — a key operational-resilience and concentration-risk concern under DORA. The transaction log and time travel also give the auditability regulators expect. The architecture turns "what if our vendor fails" from a crisis into a documented single-layer swap.

Q: Why might "just use files on a share" actually pass a naive cost comparison, and why is that misleading?
A: A share looks cheap because you only count storage and ignore the cost of the failures it invites — corrupted NAVs from non-atomic writes, no audit trail for regulators, manual reconciliation, and no fine-grained security. The lakehouse adds those guarantees on storage that is just as cheap, so the real comparison is "cheap-but-unsafe" versus "cheap-and-governed." The misleading part is pricing storage while ignoring the risk and labour the share imposes.

Q: What is the difference between the catalog and the metadata file in Iceberg?
A: The `metadata.json` metadata file is per-table state (schema, snapshots, partition spec) stored alongside the data in the object store; the catalog is the external service that knows, for a named table, which metadata file is currently authoritative and performs the atomic pointer swap on commit. The catalog is the namespace and the commit coordinator; the metadata file is the table's own record. You need both — the catalog points at the metadata file.

Q: Why does the architect care which layer enforces security policies — catalog versus engine?
A: If policies are enforced only in one engine, swapping engines (or letting a second engine in) bypasses your row/column security; if policies live in the catalog, every engine that uses the catalog inherits them. For regulated fund data, catalog-enforced policy is far safer and more portable. So "where is security enforced" is a design decision with direct lock-in and compliance consequences.

Q: How does the lakehouse let cold regulatory archives get cheap while staying queryable?
A: The archives sit as open Parquet/Iceberg files on cheap, infrequent-access object-store tiers (with lifecycle rules moving them down), and an engine reads them only on demand rather than keeping them in priced warehouse storage with running compute. Time travel and the table format still make them auditable. You pay near-storage-only prices at rest and compute only when an auditor actually queries.

Q: Why is "the table format question is settled — Iceberg won" (a 2026 view) relevant to an architect's lock-in reasoning?
A: Because broad, converging engine and catalog support for Iceberg means choosing it minimizes the risk that your format becomes a stranded niche, and it maximizes the number of engines that can read/write your tables — directly reducing lock-in. It also means vendors increasingly compete on catalog and compute, not format. The architect can lean on Iceberg as the low-lock-in default while still evaluating the catalog layer carefully.

Q: A vendor demo shows blazing query speed on open Iceberg files. What lets that be fast despite open formats?
A: Caching of hot data, optimized data layout/clustering so queries skip irrelevant files, and auxiliary structures (statistics, zone maps, indexes) for file/row-group pruning — the exact techniques the lakehouse paper cites. The speed comes from the engine and format layers, not from a proprietary storage format. So fast does not require closed, which is the whole performance argument of the lakehouse.

Q: Why does the lesson frame this as a T1 (architect-tier) lesson rather than a hands-on build lesson?
A: Because the deliverable is judgement and articulation — a build-vs-buy memo, scoring on cost/skills/lock-in/compliance, and the ability to rebut "just buy Snowflake" and "just use files" with specific consequences — not assembling the stack yourself. The "Done when" checks are all "can you explain/name/state," i.e. defend a design. The hands-on assembly lives in the T2 lessons around it.

Q: For the memo's recommendation, the lesson asks for "the one risk that would change it." What does that phrase force you to do?
A: It forces you to name the single most important uncertainty whose resolution would flip your recommendation — for example, "if the team cannot hire two platform engineers, switch from open lakehouse to managed Databricks," or "if EU data-residency for vendor X cannot be guaranteed, drop it." This makes the recommendation honest and testable rather than absolute. An architect who can name the pivot risk has actually weighed the decision.

Q: Compare the lock-in profile of an assembled-open stack (MinIO/Iceberg/Trino/DuckDB) versus a managed bundle.
A: The open stack pushes lock-in to nearly zero at every layer but demands real in-house operational skill to keep the layers compatible and maintained; the managed bundle concentrates lock-in in one layer (usually catalog/compute) in exchange for far lower operational burden. The trade is portability-and-effort versus convenience-and-stickiness. For a 50-person shop the operational burden often outweighs the lock-in saved.

Q: A fund's NAV table must support both heavy nightly batch loads and many concurrent morning BI reads. How do the lakehouse layers divide that work?
A: The object store holds the Parquet files cheaply; the table format gives the nightly batch atomic commits and the morning readers a consistent snapshot; the catalog enforces who can read which columns; and you can use a batch engine (Spark) for the load and a high-concurrency engine (Trino, or a warehouse) for the morning reads over the same tables. Each layer carries its guarantee, and engines are chosen per workload. That separation is the lakehouse's answer to mixed workloads.

Q: Why does Inmon's perspective remain useful even though he is a "warehouse veteran" skeptical of lakes?
A: Because his critique pinpoints exactly what lakes must not lose — governance, data quality, lineage, conformed definitions — and the lakehouse is strongest when it deliberately re-implements those disciplines rather than assuming open files alone are enough. Treating his objection as a checklist (do we have ownership, quality gates, lineage?) makes your design swamp-proof. Dismissing the warehouse veterans is how you build a swamp.

Q: What is the risk of letting "ELT, sort it out later" govern a regulated fund lakehouse?
A: Inmon's warning applies: dumping raw feeds in with no enforcement, ownership or modelling lets the lake degrade into a swamp where analysts distrust the data and regulators cannot trace a figure's lineage. In a fund context that means unreliable NAVs and un-auditable reports. The lakehouse permits ELT but pairs it with schema enforcement, catalog governance and lineage so "later" actually happens with guarantees.

Q: How do you answer "can't we just put everything in one big warehouse and skip the lake entirely" for ML workloads?
A: Warehouses historically lock data in a proprietary format that ML frameworks cannot read directly, forcing yet another export and a second copy for data science, plus you pay warehouse rates for large training datasets. The lakehouse keeps one open copy that both SQL/BI and ML tools read directly. So warehouse-only re-creates the two-tier copy problem the lakehouse was meant to remove, specifically for ML.

Q: In a vendor selection, why ask "does fine-grained security travel with the data or only with the engine?"
A: Because if masking and row filters are enforced only inside one engine, any other engine reading the open files bypasses them — a serious risk for investor PII or restricted fund data — whereas catalog-enforced policy protects the data regardless of engine. The answer tells you both your compliance posture and your engine-swap freedom. For regulated data, "travels with the data/catalog" should be a hard requirement.

Q: What is Apache Polaris, and why does it matter to the catalog layer's lock-in story?
A: Apache Polaris is a vendor-neutral, open-source implementation of the Iceberg REST catalog spec governed under the Apache Software Foundation, so multiple engines can register, read and commit tables through a common open API rather than a proprietary catalog. It matters because an open catalog is what turns "open files" into genuinely portable tables — you can leave a vendor and keep both your data and your governance. Snowflake Open Catalog is a managed Polaris, which is why the architect checks whether a vendor's catalog speaks the open REST spec.

Q: A scheduled compaction job on a fund's Iceberg table conflicts with a NAV writer and one commit is rejected. Is this a failure to fix or expected behaviour?
A: It is expected optimistic-concurrency behaviour: compaction and the writer both committed against the same base snapshot, only one atomic pointer swap won, and the loser must retry against the new snapshot — no data is lost or corrupted. You design compaction to retry or to run in a low-write window rather than trying to eliminate the conflict. The gotcha is treating a normal commit conflict as an outage and over-engineering around it.

Q: Why does the lesson tell you to state consequences "with specific consequences" rather than general pros and cons?
A: Because a steering committee discounts vague trade-offs but acts on concrete ones — "files-on-a-share means a half-written NAV can be read as final and you cannot prove yesterday's figure to an auditor" is decisive in a way "lakes are less reliable" is not. Specific, fund-relevant consequences are what let an architect actually win the build-vs-buy argument. The "Done when" check is explicitly the ability to rebut each alternative with named consequences.

Q: How does choosing an open catalog change your answer to "what is our exit plan from this vendor?"
A: With an open catalog (Iceberg REST / Polaris-compatible) the exit plan is "re-point engines at a new catalog endpoint and keep the same open tables," because reads, writes and policies are not gated by one vendor — turning a wholesale migration into a single-layer swap. With a proprietary catalog, the exit plan involves migrating governance and possibly rewriting the write path, which is far costlier. For DORA-era resilience reviews, that difference is exactly what you must be able to demonstrate.

Q: Summarize, in architect terms, the single sentence the lakehouse paper is built to support.
A: That you can add the warehouse's management features — ACID transactions, versioning, indexing, governance, performance optimizations — directly on top of cheap open-format data in object storage, so one platform unifies data warehousing and advanced analytics without copying data between two tiers. Everything else (formats, catalogs, engines, performance tricks) is in service of making that claim true on open storage. The architect's job is to know where each of those features actually lives in the stack.

Q: Final integration question — walk through what each lakehouse layer guarantees for a Luxembourg fund's month-end NAV pipeline, and name the layer that is your real lock-in risk.
A: The object store (EU-resident S3/ADLS) guarantees cheap durable retention of NAV/holdings files; the table format (Iceberg) guarantees atomic month-end commits, schema enforcement on EMT/EPT feeds, and time travel for auditors; the catalog guarantees the namespace, the atomic commit, and column/row security on investor data; the engines (Trino/DuckDB or an attached warehouse) guarantee query performance and concurrency. The real lock-in risk is the catalog/governance layer, since open files do not help if only one vendor can safely write and govern the tables — so that is the layer whose exit plan you must prove for DORA.

Q: Walk the Iceberg metadata tree from root to data, naming each file level.
A: Reading starts at the current `metadata.json` (root: schema, partition specs, snapshot list, table properties), which points at a snapshot's manifest list (one Avro file per snapshot listing every manifest), each manifest then lists a group of data files with per-file partition values and column min/max stats, and the leaves are the Parquet data files. The engine traverses this tree to prune at every level — skipping whole manifests via the manifest list's partition bounds, then whole files via the manifest's stats — without ever LISTing directories. This is why Iceberg scales pruning to millions of files where a directory-listing lake would choke.

Q: Why does the manifest list store partition value ranges, and how does an engine exploit that for a NAV query?
A: The manifest list records the lower/upper partition bounds of every manifest it points to, so a query like `WHERE valuation_date = '2026-05-31'` can skip an entire manifest (and all the data files it tracks) when that manifest's date range does not overlap — pruning at the manifest granularity before ever opening a manifest. This is a coarse first pass; the manifest's per-file stats then prune individual data files. The two-tier skip is what lets a single-day NAV query touch only a handful of files in a multi-year table.

Q: Iceberg coordinates commits through a catalog pointer swap; how does Delta Lake's commit mechanism differ at the file level?
A: Delta writes a new numbered JSON commit file into the table's `_delta_log` directory (for example `00000000000000000027.json`) and relies on the storage system's atomic PUT-if-absent so only one writer can create that exact version number — there is no external catalog pointer involved for the commit itself. Iceberg instead writes a new `metadata.json` and has the catalog atomically swap its pointer to it. So Delta's atomicity lives in the log-file naming plus object-store conditional put, while Iceberg's lives in the catalog's pointer swap.

Q: What storage guarantee does Delta Lake's commit protocol require that plain eventually-consistent S3 historically did not give?
A: It requires atomic PUT-if-absent (create-the-object-only-if-it-does-not-already-exist) on the numbered commit JSON, so two writers racing to create version N cannot both succeed. S3 only added native conditional writes relatively recently, which historically forced Delta on S3 to use an external coordination service (like DynamoDB) to guarantee mutual exclusion on the log file. The gotcha: a Delta table on a store lacking that primitive can silently corrupt under concurrent writers.

Q: Contrast Iceberg's optimistic-concurrency retry with Delta's, given two writers hit the same table.
A: Both use optimistic concurrency — each prepares against a read base version, and on commit only one wins. In Iceberg the loser fails the catalog pointer swap (the base snapshot moved) and retries against the new snapshot; in Delta the loser fails to create the next numbered JSON (PUT-if-absent rejected because that version now exists) and retries by reading the new version and re-validating its changes. The architect's takeaway is identical: design writers to retry, not to lock.

Q: A stakeholder asks why Iceberg picked a single-pointer-swap design instead of locking the table during writes. What is the architectural reason?
A: Object stores offer no table-level lock and no multi-file atomic operation, so a lock-based design would need an always-on external lock manager and would serialize writers, hurting throughput. The single atomic pointer swap lets many writers prepare data files concurrently and only contend at the very last cheap step, maximizing parallelism while still guaranteeing readers see one consistent state. Optimistic concurrency on a cheap atomic primitive is the natural fit for dumb storage.

Q: Distinguish Iceberg copy-on-write from merge-on-read for a row-level correction on a holdings table.
A: Copy-on-write rewrites every data file that contains an affected row at write time, so reads are fast (no merge) but a one-row correction can rewrite a large Parquet file (high write amplification); merge-on-read instead writes a small delete file / deletion vector plus any new rows and defers the merge to read time, so writes are cheap but readers must reconcile. For a high-churn holdings table with frequent small corrections, merge-on-read (v3 deletion vectors) cuts write cost; for a mostly-read NAV snapshot, copy-on-write keeps reads simplest.

Q: What is the difference between Iceberg position deletes and equality deletes?
A: A position delete marks a row dead by its data-file path plus row position (for example "row 42 of file X"), which is precise but requires knowing where the row physically sits; an equality delete marks rows dead by column value (for example `isin = 'LU0000000000'`) regardless of which file they live in, which is cheaper to write for streaming CDC but forces readers to evaluate the predicate against many files. Equality deletes can become a read-time performance problem (the "equality delete problem") if they accumulate. v3 deletion vectors are the modern replacement for position delete files.

Q: What are Puffin files in Iceberg v3 and what do they store?
A: Puffin is Iceberg's blob format for auxiliary statistics and indexes, and in v3 it stores deletion vectors (compact bitmaps of deleted row positions per data file) as well as things like NDV/theta sketches for stats. Storing deletion vectors as bounded, stable bitmaps in Puffin is what lets v3 avoid rewriting position-delete files and makes merge-on-read deletes both compact and fast to apply. So when you hear "v3 deletion vectors," the on-disk home is a Puffin blob.

Q: How does Hudi's copy-on-write table type differ from its merge-on-read type?
A: Hudi copy-on-write stores data only in columnar base files and synchronously rewrites the affected base file on every update or delete — read-optimized, write-heavy; merge-on-read keeps columnar base files plus row-based (Avro) log/delta files for updates and compacts them to new base files periodically (sync or async) — write-optimized, with reads paying a merge cost until compaction. The CoW-vs-MoR trade in Hudi mirrors Iceberg's, letting you tune per table for read-latency versus ingest-latency.

Q: Why does the choice between merge-on-read and copy-on-write make compaction more or less urgent?
A: Merge-on-read defers work to read time by piling up delete files / log files, so without regular compaction read latency degrades as those accumulate — compaction is operationally mandatory; copy-on-write does the merge at write time, so there are fewer pending delete/log artifacts and compaction is mostly about consolidating small files rather than rescuing read performance. For an intraday-corrected fund table on MoR, you schedule compaction precisely because skipping it slowly poisons query speed.

Q: An auditor needs the table "as of 31 May 2026 23:59." Show the two ways Iceberg lets you express that and which is more robust.
A: You can time-travel by timestamp (`AS OF TIMESTAMP '2026-05-31 23:59:00'`, which resolves to the snapshot current at that instant) or by an explicit snapshot id (`AS OF VERSION <snapshot_id>` / Spark's `VERSION AS OF`). The snapshot id is more robust for audit because it pins an exact immutable commit that cannot be reinterpreted, whereas a timestamp depends on commit clock alignment and which snapshot was current then. For a regulatory reproduction you record the snapshot id alongside the figure.

Q: Time travel only works while the old snapshot still exists. What routine operation can silently destroy an auditor's snapshot, and how do you guard it?
A: Snapshot expiration / `expire_snapshots` (and the underlying retention property) physically removes old snapshots and their orphaned data files to control storage, which can delete the very snapshot an auditor needs. You guard it by setting retention long enough to cover your regulatory window (for example multi-year for fund records), tagging or branching snapshots that must be preserved, and never running aggressive expiry on regulated tables. The gotcha is assuming "time travel" is unlimited — it is bounded by your expiration policy.

Q: What is Iceberg's `rewrite_data_files` action and when do you run it on a fund table?
A: It is the compaction action that reads many small data files and rewrites them into fewer, larger, optimally sized files (optionally re-sorting/clustering), reducing file count and improving scan and pruning efficiency. You run it on tables fed by frequent micro-batches — intraday NAV landings, streaming corrections — where small files accumulate, typically on a schedule in a low-write window so commit conflicts with live writers are minimized. It is the standard answer to "queries are slow and opening thousands of tiny files."

Q: Beyond data-file compaction, what other Iceberg maintenance routine controls metadata bloat, and what does it remove?
A: `expire_snapshots` removes old snapshots and the data/manifest files only they referenced, and there are companion actions to rewrite/compact manifests (`rewrite_manifests`) and to remove orphan files (`remove_orphan_files`) left by failed jobs. Over a high-churn table the manifest list and manifests themselves can bloat and slow planning, so rewriting manifests keeps the metadata tree lean. Neglecting metadata maintenance is a subtler cause of slow planning than just small data files.

Q: A failed Spark job left data files in storage that no snapshot references. What are these called and which action cleans them?
A: They are orphan files — Parquet files written by a job that crashed before (or instead of) committing, so no snapshot's manifests point at them; they waste storage but are invisible to queries. Iceberg's `remove_orphan_files` action lists storage, compares against referenced files, and deletes the unreferenced ones older than a safety threshold. You run it carefully with a conservative age cutoff so you do not delete files from an in-flight commit.

Q: Why is `remove_orphan_files` dangerous if run with too small an age threshold?
A: Because a legitimately in-progress write has already put data files in storage but not yet committed the snapshot that references them, so a too-aggressive orphan cleanup can delete files belonging to a live transaction and corrupt the impending commit. The default threshold (commonly three days) exists precisely to stay clear of any in-flight writer. The architect sets the threshold longer than the longest possible write, never shorter for convenience.

Q: How does Iceberg hidden partitioning prevent a class of bug that Hive-style explicit partitioning suffers from?
A: Iceberg derives the partition value from a transform on a real column (for example `days(valuation_date)`) and tracks it in metadata, so a query filtering on `valuation_date` is pruned automatically — the user never writes a separate partition predicate. Hive-style partitioning exposes a physical partition column (like `dt`) that the query must filter on explicitly; forget it and you full-scan. Hidden partitioning eliminates the "analyst filtered the data column but not the partition column" full-scan footgun.

Q: With Iceberg partition evolution, you change a table's partitioning from monthly to daily. What does NOT happen to old data, and why does that matter?
A: Old data files are not rewritten — they keep their original (monthly) partitioning, while new writes use the new (daily) spec, and Iceberg stores both specs so the planner prunes each set of files by the spec under which it was written. This matters because you can re-partition a huge historical fund table instantly without a costly rewrite, and queries still prune correctly across the boundary. The gotcha is that pruning on old data is only as fine-grained as its original spec.

Q: How does Iceberg schema evolution track columns so that renaming a column does not corrupt old files?
A: Iceberg assigns every column a stable unique field id and maps reads by id, not by name or position, so renaming a column just changes the name attached to an id while old Parquet files (written under the same id) still read correctly. This is what makes add/drop/rename/reorder safe without rewriting data. Contrast with name- or position-based formats where a rename or reorder can silently misalign columns — exactly the kind of error that would mis-key an ISIN column.

Q: A new EPT field must be added to a live table mid-quarter without breaking running readers. Which Iceberg operation and what is the safe ordering?
A: Use an additive schema evolution — `ALTER TABLE ... ADD COLUMN` — which is a metadata-only change assigning a new field id; old data files simply return null for the new column and existing readers ignore it. The safe path is add the nullable column first, then start writing the EPT value, never reorder or retype existing columns in the same step. Additive, nullable changes are the only ones guaranteed backward-compatible for in-flight consumers.

Q: Why is dropping or narrowing a column type riskier than adding one, in schema-evolution terms?
A: Adding a nullable column is backward- and forward-compatible (old readers ignore it, old files return null), but dropping a column or narrowing a type (for example `long` to `int`) can break readers that still expect the column or can lose/overflow data in existing files. Iceberg permits safe widening (`int`→`long`, `float`→`double`) but blocks unsafe narrowing for this reason. For regulated feeds you treat drops/narrowing as breaking changes requiring a coordinated consumer migration.

Q: The catalog returns the wrong (stale) metadata location for a table and reads look outdated. Where do you look first?
A: First confirm the catalog's pointer — the catalog is authoritative for "which `metadata.json` is current," so a stale read usually means a commit did not update the pointer (a failed/partial commit) or you are reading a different catalog/namespace than the writer used. Check that writer and reader share the same catalog endpoint and namespace, and inspect the table's current metadata location in the catalog versus the latest `metadata.json` in storage. The data files are rarely the culprit; the pointer is.

Q: Two engines disagree on a table's contents — one is current, one is a snapshot behind. What configuration mistake most commonly causes this?
A: They are pointed at different catalogs (or a catalog vs. a direct hadoop/filesystem table path), so they resolve "current metadata" independently and drift. Iceberg's consistency guarantee holds only when all engines commit and read through the same catalog that owns the atomic pointer; bypassing the catalog with a raw metadata path defeats it. The fix is to standardize every engine on one catalog endpoint and namespace, never mixing catalog-managed and path-based access to the same table.

Q: Why is a HadoopCatalog (filesystem-pointer) discouraged for production multi-writer fund tables, and what is preferred?
A: A filesystem/hadoop catalog infers the current metadata from a file in storage and depends on atomic rename/listing that object stores like S3 do not reliably provide, so concurrent commits can race and corrupt the pointer. A service catalog — the Iceberg REST catalog (Polaris), or Glue/Nessie/JDBC — provides a real atomic commit and is the production choice. For multi-writer NAV pipelines on S3 you choose a REST/service catalog precisely to get a safe atomic commit.

Q: How does Project Nessie's "Git-for-data" model extend the catalog layer, and where might a fund use it?
A: Nessie is an Iceberg catalog that adds Git-like branches, tags and commits across tables, so you can create a branch, run and validate a multi-table load in isolation, then atomically merge it — giving cross-table-style isolation the single-table formats lack natively. A fund could load and reconcile holdings and NAV on a branch and only publish (merge) once month-end checks pass, avoiding readers ever seeing a half-finished close. It turns "orchestrate multi-table consistency at the job level" into a catalog-level branch-and-merge.

Q: The lesson says formats give single-table ACID only. Sketch how you achieve a consistent multi-table month-end close anyway.
A: You orchestrate it above the format: load each table to a staging branch/snapshot, run cross-table reconciliation (holdings tie to NAV, totals balance), and only then publish all tables — either via a catalog with branching (Nessie) that merges atomically, or via an orchestration pattern where downstream consumers read a published "as-of" pointer table that you flip once all components are committed. The guarantee comes from process design plus an indirection pointer, not from the format. The architect must state this explicitly because no single Iceberg/Delta commit spans the two tables.

Q: Iceberg branches and tags (the table-level ones, not Nessie) — what auditing use do they serve for a fund?
A: Iceberg supports named refs: a tag pins a specific snapshot under a stable name (for example `eom-2026-05`) and a branch is a movable line of commits; tagging the exact month-end-close snapshot gives auditors a durable, named handle that survives even as the table keeps changing. You also use a long retention on tagged snapshots so expiration cannot remove them. Tags turn "snapshot id 81923..." into a human-meaningful, audit-safe reference.

Q: What is the WAP (write-audit-publish) pattern on Iceberg, and why does it fit regulated data?
A: Write-audit-publish writes new data to a staging branch, runs data-quality/audit checks against that branch in isolation, and only fast-forwards the main branch (publishes) if the checks pass — so consumers never see unvalidated data. For fund data this means a malformed EMT batch or a NAV that fails reconciliation is caught on the branch and never becomes visible to BI or downstream reports. It operationalizes "schema enforcement plus validation" as a gated promotion rather than a hopeful post-load check.

Q: A Trino query against Iceberg errors with a missing-metadata or "table not found" despite files existing in the bucket. First diagnosis?
A: The catalog has no registration (or the wrong one) for that table — Trino resolves tables through its configured Iceberg catalog connector, so files in the bucket are invisible until a catalog entry points at their current `metadata.json`. Check the catalog connector config, the schema/namespace, and whether the table was created path-based outside this catalog. The fix is usually `register_table` (or recreating it through the catalog), not touching the data files.

Q: How do you register an existing set of Iceberg metadata into a catalog so an engine can see it?
A: Use the catalog's register-table capability (for example Spark's `register_table` procedure or the REST catalog's create-with-existing-metadata) pointing at the table's current `metadata.json`, which creates the namespace entry and pointer without rewriting data. This is the standard move after migrating a table between catalogs or recovering one created path-based. The data and metadata files stay put; you are only giving a catalog the pointer.

Q: Snowflake reads an external Iceberg table fine but cannot write to it. What catalog arrangement explains read-only, and how do you get writes?
A: When Snowflake uses an externally-managed Iceberg table (an external catalog owns the commits), Snowflake treats it as read-only because it is not the catalog of record for that table; to write you must make Snowflake the managing engine via Snowflake-managed Iceberg tables or have all writers commit through a shared open catalog (Snowflake Open Catalog / Polaris) that both Snowflake and external engines use. The lock-in question is exactly "who owns the write/commit path," and read-only-from-one-side is the symptom. You resolve it by choosing one open catalog all writers share.

Q: Why can letting two different engines each "own" writes to the same Iceberg table corrupt it, and what is the rule?
A: If two engines commit through different catalogs (or one through a catalog and one path-based), neither sees the other's atomic pointer move, so they can both think they hold the current snapshot and clobber each other's commits — the optimistic-concurrency guarantee only holds within one catalog. The rule is single catalog of record per table: exactly one commit coordinator, which every writer goes through. Multi-writer safety is a property of sharing the catalog, not of the file format.

Q: How does Unity Catalog's "lakehouse federation" / external-table reading complicate the simple "swap the engine, data stays put" story?
A: If governance, row/column masking and (for managed tables) the commit path live in Unity Catalog, then reading the open files from a non-Databricks engine bypasses those policies and may not be permitted for writes, so the engine swap that looks free for data movement is not free for governance — your security and write path are tied to the catalog. The portability you actually have is "other engines can read the bytes," not "other engines can operate the governed table." This is why the architect interrogates the catalog, not the file format, when assessing an engine swap.

Q: Distinguish Unity Catalog managed tables from external tables in lock-in terms.
A: For Unity Catalog managed tables, Databricks owns the storage location lifecycle and the commit/optimization path, maximizing convenience but binding operation to the platform; external tables point at a storage location you control, loosening the storage tie but still routing governance and (often) the write path through Unity Catalog. The lock-in difference is "who controls the table lifecycle and commit," and external tables are the lower-lock-in choice when portability matters. The architect asks which kind a vendor's "open Iceberg" actually is.

Q: BigQuery offers "BigLake Iceberg" / managed Iceberg tables. At the catalog/lock-in level, what is the architect's check?
A: Confirm whether the Iceberg tables are governed by an open catalog that external engines can also commit through, or whether BigQuery is the catalog of record (so other engines get read-only and governance lives in BigQuery/Dataplex). The data being open Iceberg in GCS is necessary but, per the lesson, not sufficient — the write/commit and policy path is the lock-in. The check is identical across clouds: who can write and govern these tables if I leave.

Q: A CFO sees a huge object-store bill that is mostly request charges, not storage. Which lakehouse anti-pattern causes this and what is the fix?
A: The small-files / over-listing anti-pattern: thousands of tiny data files and frequent metadata listing drive up per-request GET/LIST charges even though stored bytes are modest. The fix is scheduled compaction (`rewrite_data_files`), fewer larger files, and relying on Iceberg metadata pruning instead of directory listing so the engine touches far fewer objects per query. The lesson's point that "request costs are a layout choice" shows up here as a real budget incident.

Q: How does Iceberg's metadata-based planning reduce object-store LIST costs versus a Hive-style lake?
A: A Hive-style lake discovers files by LISTing directories/partitions on every query, which on S3 is many billed LIST calls and is eventually-consistent; Iceberg instead reads the manifest list and manifests (a few known objects) to enumerate exactly the data files, so planning is a handful of GETs with no directory scans. Fewer, predictable metadata reads cut both cost and latency and avoid stale-listing bugs. This is a concrete reason the table format, not just cheaper storage, lowers the bill.

Q: A streaming NAV ingest commits every few seconds and planning has become slow even after data compaction. What metadata problem is this and the fix?
A: Frequent commits create a long snapshot history and many small manifest files, bloating the manifest list so query planning spends time reading metadata — a metadata-bloat problem distinct from data small-files. The fix is `rewrite_manifests` to consolidate manifests plus `expire_snapshots` to trim history (within your retention window). Tuning commit frequency (micro-batch instead of per-record) also reduces metadata churn at the source.

Q: How do auxiliary statistics (min/max, zone maps, optional NDV sketches) in Iceberg metadata translate into file skipping?
A: Each manifest stores per-data-file column bounds (min/max) and counts, so a predicate like `nav BETWEEN x AND y` lets the planner skip any file whose stored min/max cannot satisfy it, reading only candidate files; NDV/theta sketches (Puffin) further help the optimizer estimate cardinality for join/order decisions. This is the lakehouse paper's "auxiliary data structures for skipping" made concrete. Good clustering makes these bounds tight; randomly ordered data makes min/max useless because every file's range overlaps.

Q: Why does sort order / clustering of data on write determine whether min/max stats actually prune anything?
A: If rows are clustered so each file holds a narrow range of the filter column (for example each file is one valuation_date range), the per-file min/max are tight and disjoint, so most files can be skipped; if data is written randomly, every file's min/max spans the whole range and overlaps, so no file can be pruned and you full-scan despite having stats. Clustering is therefore a prerequisite for stats-based skipping, not an optional nicety. Compaction with a sort order is how you fix a badly-laid-out fund table.

Q: ADLS Gen2 has a hierarchical namespace with atomic directory rename; does that change how Iceberg commits compared to flat S3?
A: Iceberg deliberately does not rely on directory rename for its commit — atomicity comes from the single catalog pointer swap (or, for hadoop catalog, a single metadata file operation), so the core commit path is the same on flat S3 and on ADLS Gen2. The hierarchical namespace mainly helps non-Iceberg/Hive-style operations and some maintenance ergonomics, not Iceberg's atomicity. The takeaway: choosing Iceberg means you do not need ADLS's atomic rename to be safe, which is why the same tables run on plain S3/MinIO.

Q: Where does a vendor's "serverless" Iceberg query engine actually save the fund money, and where can it still surprise the bill?
A: It saves money by charging compute only while a query runs (no idle warehouse), which suits spiky month-end and on-demand auditor queries over cold archives; it can surprise you when a poorly-pruned full scan over a huge unclustered table bills enormous bytes-scanned, because serverless cost is usually bytes-scanned or slot-seconds, not data size at rest. The lever is query shape and layout, not turning the engine off. A single unpartitioned NAV-history scan is the classic serverless cost spike.

Q: How would you prove EU data-residency for an open lakehouse to a Luxembourg regulator, layer by layer?
A: Show the object store bucket/account is pinned to an EU region (and not silently replicated out), the catalog service runs in-region and stores its metadata in-region, the query engines execute in EU regions, and that backups, logs and any vendor telemetry also stay in-EU; document this per layer with the provider's region attestations. Because the layers are independent, residency must be verified for each — an EU bucket behind a US-hosted catalog still leaks metadata. The architect treats residency as a four-layer checklist, mirroring the four-layer architecture.

Q: For DORA exit-testing, what concrete artifact demonstrates a credible catalog exit, beyond "files are open"?
A: A tested runbook showing you can register the same Iceberg tables into a second, independent open catalog (for example Polaris elsewhere) and have a second engine both read and write them under equivalent policies — ideally rehearsed, with timings — proving the exit is a single-layer re-point, not a data migration. Open files alone do not satisfy the test; the demonstrated write/govern path on an alternate catalog does. Regulators value a rehearsed exit over an asserted one.

Q: Why might a fund deliberately keep its EMT/EPT validation logic in the catalog/table-format gate rather than in each consuming engine?
A: Because enforcing schema and quality at the table format (schema enforcement, WAP gating, constraints where supported) and policy at the catalog means every engine inherits the same guarantees, so a second BI tool or a Python job cannot bypass them — whereas validation duplicated per engine drifts and leaves gaps. Centralizing the gate is both a correctness and a portability decision: it travels with the data. For regulated feeds, a single enforced gate is far easier to evidence to an auditor than per-engine checks.

Q: Summarize the architect's portability checklist distilled from this lesson as four questions, one per layer.
A: Object store: can I re-home the files to another EU-resident S3/ADLS account and recreate lifecycle rules? Table format: am I on the broadly-supported open format (Iceberg) so any engine can read/write, making a format change unnecessary? Catalog: can a second engine read AND write these tables through an open catalog under the same policies (the real lock-in test)? Engine: can I re-point dashboards at another engine and only re-test SQL dialect, with no data move? If every answer is yes, your exit is a set of single-layer swaps — which is exactly the DORA-grade resilience posture the lesson asks you to defend.

Q: How does a DuckDB analyst on a laptop read the same EU-resident Iceberg fund tables that Trino serves in production, and what is the catch?
A: DuckDB reads Iceberg via its `iceberg` extension and reaches the object store with `httpfs` and S3-style credentials (or an Iceberg REST catalog connection), so it queries the very same files and snapshots Trino uses — ideal for local validation against production data layout. The catch is that DuckDB is single-node and historically stronger at reads than at committing Iceberg writes, and it has no part in governance, so a laptop session can read sensitive investor columns unless catalog-enforced row/column policies actually gate that access path. You validate locally but never assume the laptop engine inherits production security for free.

Q: Iceberg, Delta and Hudi all add ACID over files, but which one was designed first around streaming upserts/CDC, and why does that origin show?
A: Hudi was designed at Uber first for incremental, upsert/CDC-heavy ingestion, which is why it foregrounds record-level indexes, primary keys and merge-on-read with log files to make frequent updates cheap. Iceberg originated at Netflix to fix Hive's correctness and scale problems for large analytic tables (hence its metadata-tree pruning and snapshot model), and Delta originated at Databricks around the Spark/`_delta_log` experience. The origin shows in defaults: reach for Hudi-style thinking when the workload is a high-rate keyed upsert stream, Iceberg when it is large multi-engine analytic tables — which is the common fund-platform case.

Q: A fund must guarantee that an already-published month-end NAV snapshot can never be silently altered. Which lakehouse mechanisms together provide that, and what is the residual gap?
A: Combine an immutable tagged snapshot (a named ref like `eom-2026-05` so the exact commit is pinned), long/locked retention so expiry cannot remove it, catalog-enforced write/governance policy so only authorized jobs commit, and the table's immutable transaction log as the audit trail proving any later change is a new snapshot, not an edit. The residual gap is that the format protects against silent in-table edits but not against a privileged operator deleting files in the bucket or expiring the snapshot, so you also need object-store immutability/object-lock (WORM) and least-privilege on storage. True tamper-evidence needs the storage layer's WORM controls plus the table-format guarantees, not either alone.
