# Phase 6 · 11.3.1 + 11.3.2 Data masking & tokenization — 100 self-test questions

Q: What is data masking, in one sentence?
A: Data masking is the process of replacing real sensitive values with structurally similar but non-sensitive substitutes so that data stays usable for development, testing, and analytics without exposing the original PII. The shape (length, format, data type) is typically preserved so applications and queries still behave realistically, while the actual investor names, national IDs, or account numbers can no longer be read. It is a core GDPR data-minimization control for lower environments.

Q: What is the core difference between static and dynamic masking?
A: Static masking permanently rewrites the data at rest — it produces a masked copy of the database where the original values are physically gone. Dynamic masking leaves the stored data intact and obscures it only in the query result, deciding per-user whether to reveal or mask. So static is "mask the copy you hand to dev," and dynamic is "mask what this particular caller is allowed to see right now."

Q: For a masked replica of the Phase-1 investor tables shipped to a dev laptop, is static or dynamic masking the right tool, and why?
A: Static masking is the right tool because the dev environment must never contain real PII at rest — the masking has to be baked into the copy itself. Dynamic masking only hides values at query time while the real data still sits in the table, so a developer with direct access or a backup of that DB would still hold real investor data. For lower environments you want the original PII to be genuinely absent, which only static masking guarantees.

Q: What is SQL Server Dynamic Data Masking (DDM)?
A: DDM is a SQL Server feature (2016 and later, plus Azure SQL) that masks designated columns in the result set of queries while leaving the stored data unchanged. You define masking functions on columns with simple T-SQL, and non-privileged users see obfuscated values while users holding the `UNMASK` permission see the real data. It is dynamic masking implemented natively in the engine.

Q: Which five mask function types does SQL Server DDM provide?
A: The five built-in functions are `default()` (full masking by data type), `email()` (exposes first letter and a `.com` suffix), `random([start], [end])` (random value in a range for numeric types), `partial(prefix, [padding], suffix)` (a custom-string mask exposing some leading and trailing characters), and `datetime()` (masks a chosen part of a date/time value, available from SQL Server 2022). Each is supplied as a string inside `MASKED WITH (FUNCTION = '...')`.

Q: What does the `default()` mask do to a `varchar` column versus a numeric column?
A: For string types `default()` returns `xxxx` (or fewer characters if the field is shorter than four), and for numeric types it returns a zero value. For date/time types it returns `1900-01-01 00:00:00.0000000`, and for binary types a single zero byte. So `default()` adapts its masked output to the column's data type rather than emitting a fixed token.

Q: How would you add a `default()` mask to an existing `Gender` column?
A: You alter the column to add the mask, for example `ALTER TABLE Data.Membership ALTER COLUMN Gender ADD MASKED WITH (FUNCTION = 'default()');`. The `ADD MASKED WITH` clause attaches the masking function without changing the stored data. Adding a mask requires the `ALTER ANY MASK` permission plus `ALTER` on the table.

Q: What does `email()` reveal of the value `RTamburello@contoso.com`?
A: It exposes only the first letter and a constant `.com` suffix, producing something like `RXXX@XXXX.com`. The point is to keep the value recognisable as an email address shape while hiding the local part and the real domain. It is a fixed pattern, so it is not configurable beyond choosing the `email()` function.

Q: Why can a `partial()` mask leak information when the value is short or highly structured?
A: `partial()` exposes the configured prefix and suffix characters in plaintext, so for a short value much of it is revealed, and the docs note that if the value is too short part of the prefix or suffix simply is not shown. For a structured value, exposing the first few characters can disclose meaningful segments — for example the country prefix of an IBAN or the branch portion of an account number. The gotcha is choosing a prefix/suffix length large enough to be useful but small enough not to reveal an identifying segment.

Q: What does the `partial(1,"XXXXXXX",0)` mask do to the phone number `555.123.1234`?
A: It exposes the first one character, replaces the middle with the literal padding `XXXXXXX`, and exposes zero trailing characters, yielding `5XXXXXXX`. The signature is `partial(prefix, [padding], suffix)` where prefix is how many leading characters to keep, padding is the literal middle string, and suffix is how many trailing characters to keep. So `partial(5,"XXXXXXX",0)` on the same value would yield `555.1XXXXXXX`.

Q: What does `random(1, 12)` produce, and on what column types is it valid?
A: It produces a random number within the specified inclusive range (here 1 through 12) and is valid only on numeric column types. Because the value is regenerated per query, the same row can show a different masked number on each `SELECT`. It is useful for masking things like a discount code or a numeric month without preserving the real value.

Q: What does the `datetime("Y")` mask do, and from which SQL Server version is it available?
A: `datetime("Y")` masks the year portion of a date/time column, and the `datetime()` function supports masking `Y`, `M`, `D`, `h`, `m`, or `s` (year, month, day, hour, minute, second). It is available starting with SQL Server 2022 (16.x). For example `ALTER COLUMN BirthDay ADD MASKED WITH (FUNCTION = 'datetime("M")')` masks the month.

Q: Which permission lets a user see the real, unmasked values in a DDM column?
A: The `UNMASK` permission. A user with `SELECT` but without `UNMASK` sees masked output; granting `UNMASK` reveals the true values for the masked columns they can read. Administrative principals with `CONTROL` (such as `sysadmin` or `db_owner`) always see unmasked data because `CONTROL` implicitly includes `UNMASK`.

Q: Why must `UNMASK` always be paired with `SELECT` to have any effect?
A: `UNMASK` on its own does not disclose any data or metadata; it only changes whether masking is applied to columns the user can already read. So if you grant `UNMASK` at database scope but `SELECT` on only one table, the user can see unmasked data from just that one table. Without a `SELECT` grant there is nothing to unmask.

Q: What permission is needed to add, change, or remove a mask on a column?
A: You need the `ALTER ANY MASK` permission together with `ALTER` permission on the table. Microsoft recommends granting `ALTER ANY MASK` to a security officer role so that masking policy is controlled separately from ordinary schema changes. Note that creating a brand-new table with a mask only needs the standard `CREATE TABLE` and `ALTER` on schema.

Q: How do you grant column-level `UNMASK` only on the `FirstName` column to one user?
A: Use `GRANT UNMASK ON Data.Membership(FirstName) TO ServiceAttendant;`. Granular `UNMASK` at database, schema, table, and column level was introduced in SQL Server 2022 (16.x), letting you reveal one sensitive column to a role while keeping the rest masked. Before 2022, `UNMASK` could only be granted at database scope.

Q: How do you drop a mask from a column?
A: Use `ALTER TABLE Data.Membership ALTER COLUMN LastName DROP MASKED;`. The `DROP MASKED` clause removes the masking function so the column once again returns its true values to all readers with `SELECT`. Like adding a mask, this requires `ALTER ANY MASK` and `ALTER` on the table.

Q: Which system view tells you which columns have a mask, and what does it add?
A: The `sys.masked_columns` view lists only columns that have a masking function applied, inheriting from `sys.columns` and adding `is_masked` and `masking_function`. You can join it to `sys.tables` and filter `WHERE is_masked = 1` to inventory every masked column and its function. This is how you audit your masking coverage programmatically.

Q: A developer copies a masked column with `SELECT INTO` and the target table also shows masked data. Why?
A: Because DDM masking is applied at read time, a user without `UNMASK` who runs `SELECT INTO` or `INSERT INTO` reads masked values and therefore writes masked values into the destination. The masking effectively propagates through the copy. The same applies to SQL Server Import and Export run by a non-`UNMASK` user — the export file contains masked data.

Q: Does a DDM mask stop a user from updating the masked column?
A: No. Masking only affects what is returned on read; it does not block writes. A user who sees `xxxx` in a column can still overwrite it if they hold update permission, so you must still enforce a proper access-control policy on writes. Masking is a read-side disclosure control, not an integrity control.

Q: What is the documented inference attack against DDM, using a salary example?
A: An ad hoc user without `UNMASK` cannot read `Salary` directly, but can binary-search it with predicates like `WHERE Salary > 99999 AND Salary < 100001`; the rows that come back reveal the real value even though the displayed `Salary` is `0`. DDM masks the projected value but not the `WHERE`-clause filtering, so a determined querier infers the underlying data. This is why Microsoft says DDM alone does not secure data against malicious ad hoc queriers.

Q: Given the inference limitation, what does Microsoft recommend alongside DDM?
A: Microsoft recommends not relying on DDM alone for users with ad hoc query rights, and instead applying least-privilege permissions and enabling Auditing to monitor all database activity. DDM is positioned as a defence against accidental exposure in predefined application queries, complementary to encryption, row-level security, and auditing. The masking is a convenience layer, not a hard boundary.

Q: Why do cross-database joins on DDM-masked columns return wrong results?
A: When a query spans two different databases or instances, the remote server returns the column already in masked form, so any join or comparison happens on masked values rather than real ones. The masked tokens do not match the real keys, so the join silently produces incorrect results. This is a documented restriction of cross-database queries involving masked columns.

Q: Name two column types on which a DDM mask cannot be defined.
A: A mask cannot be defined on Always Encrypted (encrypted) columns or on `FILESTREAM` columns; it also cannot be set on a `COLUMN_SET` or a sparse column that is part of a column set, nor on a column used as a key for a `FULLTEXT` index, nor directly on a computed column. If a computed column depends on a masked column, though, the computed column itself returns masked data. These restrictions exist because masking is a schema change on the base column.

Q: Adding a DDM mask fails with `ALTER TABLE ALTER COLUMN columnname failed because one or more objects access this column`. What is the cause and fix?
A: The column has a dependency (for example an index referencing it), and because adding a mask is implemented as a schema change on the underlying table it cannot run against a column with dependencies. The fix is to remove the dependency, add the mask, then recreate the dependency — e.g. drop the dependent index, add the mask, recreate the index. The error names the blocking dependency situation explicitly.

Q: Why does the lesson treat DDM as "warehouse-native dynamic masking" with inference limits rather than a complete solution?
A: DDM is built into the engine and applies masks declaratively at query time, which is why it is "warehouse-native," but its masking is bypassable by inference and brute-force on `WHERE` predicates and breaks on cross-database joins. So it is excellent for preventing accidental exposure within a known application's query set, yet insufficient as the sole control for untrusted ad hoc access. The lesson wants you to know both the convenience and the boundary.

Q: What is tokenization?
A: Tokenization replaces a sensitive value with a surrogate token that has no exploitable mathematical relationship to the original, where the mapping is held separately (in a vault) or derived deterministically. Unlike a one-way mask, tokenization can be designed to be reversible by an authorised party who can resolve the token back to the original. It is the technique that lets you de-identify investor identifiers while still allowing controlled re-identification.

Q: What is vault tokenization versus vaultless (deterministic) tokenization?
A: Vault tokenization stores the token-to-value mapping in a secure lookup table (the vault), so reversal is a lookup and tokens can be random. Vaultless tokenization derives the token from the value using a deterministic algorithm (such as keyed hashing or format-preserving encryption) so no mapping table is needed and the same input always yields the same token. The trade-off is operational: a vault is a stateful store to protect and scale, while vaultless trades that for cryptographic key management.

Q: What is the "technique zoo" the lesson references, and roughly what is each?
A: It is the set of de-identification techniques: redaction (remove the value entirely), substitution (swap in a realistic fake value), shuffling (permute a column's real values across rows), format-preserving encryption (encrypt to a value of the same format), and vault tokenization (replace with a surrogate resolved via a vault). Each trades realism, reversibility, and referential consistency differently, and you pick per column and per use case. Presidio's anonymizer operators implement several of these.

Q: What is redaction as a masking technique?
A: Redaction removes the sensitive value completely, leaving nothing or a fixed placeholder in its place. It maximises privacy but destroys any analytical value of the field and breaks any join that relied on it. Presidio implements this with the `redact` operator, which simply removes the detected PII from the text.

Q: What is substitution, and when is it useful?
A: Substitution replaces the real value with a realistic fake of the same kind — a real-looking but invented name, address, or IBAN. It keeps data looking authentic for demos and UI testing where realism matters more than join-stability. The gotcha is that naive substitution is usually not referentially consistent unless you make the mapping deterministic.

Q: What is shuffling, and what risk does it carry?
A: Shuffling permutes the real values of a column across rows, so the set of values is genuine but each value is detached from its original row. It preserves column-level distributions and statistics for analytics. The risk is re-identification: because the real values are still present, an attacker who knows one row's true value or can correlate other columns may unshuffle some records.

Q: What is format-preserving encryption (FPE) and why use it for masking?
A: FPE encrypts a value into ciphertext of the same length and format — a 16-digit card-like number encrypts to another 16-digit number, an IBAN to an IBAN-shaped string. This lets masked data pass format validation and fit existing schemas while remaining reversible with the key. It is the technique to reach for when downstream systems enforce a strict format that random tokens would fail.

Q: What is referential consistency in masking, in one sentence?
A: Referential consistency means a given real value always maps to the same masked value everywhere it appears, so the same investor becomes the same token in every table. This is what keeps cross-table joins working after masking, since the join keys still match. Without it, masked joins silently return wrong or empty results.

Q: Why does deterministic tokenization preserve cross-table joins while random substitution breaks them?
A: A deterministic token is a pure function of the input, so investor `INV-7781` becomes the same token in the `accounts` table and the `transactions` table, and a join on that key still matches. Random substitution assigns an unrelated value per occurrence, so the same investor gets different tokens in different tables and the join keys no longer line up. Determinism is the property that makes masked referential integrity possible.

Q: With Presidio, which operator gives you deterministic tokenization and what must you control?
A: The `hash` operator with a fixed `salt` gives deterministic tokenization — the same input plus the same salt always produces the same hash. You must control the salt as a secret and keep it identical across every table you mask so the tokens line up. The salt also prevents trivial precomputed-dictionary reversal of common values.

Q: From Presidio version 2.2.361, what changed about the `hash` operator's salt and why does it matter for referential integrity?
A: From that version the `hash` operator uses a random salt by default, which means without intervention the same input would hash differently across runs and break referential consistency. To get consistent, joinable tokens you must explicitly pass a fixed `salt` parameter. This is a common gotcha: relying on defaults silently destroys your join keys.

Q: What hash types does the Presidio `hash` operator support, and what is the default?
A: It supports `sha256` and `sha512`, with `sha256` as the default. You set this through the `hash_type` parameter. For tokenization the choice mainly affects token length and collision margin; both are one-way, so `hash` is for irreversible tokens, not for cases needing reversal.

Q: List the built-in Presidio anonymizer operators.
A: The built-in operators are `replace` (substitute a fixed `new_value`), `redact` (remove the PII), `mask` (replace characters with a masking character), `hash` (salted one-way hash), `encrypt` (reversible encryption with a key), `custom` (apply a `lambda`), and `keep` (leave the PII unmodified). Each maps to a de-identification technique, and you configure them per entity type in the anonymizer. There is also a `decrypt` operator on the deanonymizer side.

Q: What does the Presidio `replace` operator do and what parameter does it take?
A: `replace` substitutes the detected PII with a desired fixed value supplied via the `new_value` parameter, for example turning every detected name into `<NAME>`. It is substitution to a constant, so it is not referentially consistent — every match becomes the same placeholder. Use it when you want to signal that a field held PII without preserving any per-entity identity.

Q: What does the Presidio `mask` operator do, and what are its parameters?
A: `mask` replaces part of the PII with a repeated masking character, controlled by `chars_to_mask` (how many characters to mask), `masking_char` (which character to use), and `from_end` (whether to mask from the end of the string). For example you can mask all but the last four digits of an account number. It preserves the visible portion for recognisability while hiding the rest.

Q: Which Presidio operator is reversible, and how do you reverse it?
A: The `encrypt` operator is reversible: it encrypts the PII with a supplied `key`, and the matching `decrypt` operator on the deanonymizer reverses it using that same cryptographic key. Anyone holding the key can recover the original value, so the key is the access-control boundary. This is the operator to use when authorised re-identification is a requirement.

Q: What is the Presidio `custom` operator for?
A: `custom` replaces the PII with the result of an arbitrary function you supply via the `lambda` parameter. It is the escape hatch when none of the built-in operators implement the transform you need — for example a bespoke tokenization scheme or a lookup into your own vault. The trade-off is that you own the correctness and consistency guarantees of whatever the lambda does.

Q: What does the Presidio `keep` operator do, and when is it useful?
A: `keep` preserves the detected PII unmodified, passing it through untouched. It is useful when an entity type is detected but you have decided it does not need de-identification in a given context — for example keeping an organisation name while masking person names. It lets you express "detected but intentionally retained" explicitly in the operator config.

Q: How does the analyzer/anonymizer split of responsibilities work in Presidio?
A: The analyzer detects PII entities and their spans in text and returns results with entity types and confidence; the anonymizer then takes those results and applies an operator per entity type to de-identify them. Detection and transformation are deliberately separate stages, so you can tune recognition independently from how you mask. For tokenization you run the analyzer to find the investor identifiers, then apply `hash` or `encrypt` in the anonymizer.

Q: When should you mask data versus generate synthetic data?
A: Mask when you need real-world fidelity and referential integrity tied to actual records — true distributions, real edge cases, joinable keys — but must remove the identifying PII. Generate synthetic data when you need volume, must guarantee no real record can ever be reconstructed, or want to share data with parties who should see no derivative of production at all. Masking transforms real rows; synthesis fabricates new ones, and they sit on a spectrum of fidelity versus disclosure risk.

Q: Why is masked production data often preferred over fully synthetic data for a fund-NAV pipeline test?
A: Because masked production data retains the genuine shape, skew, and corner cases of real fund and investor records — odd ISINs, multi-currency holdings, fractional units — that a synthetic generator may not reproduce faithfully, so your NAV aggregation logic is exercised against realistic data. Synthetic data can miss the very edge cases that break NAV calculations. The cost is that masking must be airtight so no real PII leaks downstream.

Q: What is a residual re-identification risk of masking that synthesis avoids?
A: Masked data is still derived from real records, so quasi-identifiers (postcode, birth year, fund holdings) left unmasked can sometimes be combined to re-identify an individual even when the direct identifiers are tokenized. Well-formed synthetic data has no one-to-one link to a real person, so this linkage attack does not apply. This is why masking must also consider quasi-identifiers, not just the obvious PII columns.

Q: For the lesson's Do step, what does "deterministic tokenization" concretely mean when building the masked investor replica?
A: It means each investor identifier is transformed by a function (such as Presidio `hash` with a fixed salt, or FPE with a fixed key) so the same investor identifier always yields the same token across every table. That determinism is exactly what lets you later prove the same investor maps to the same token everywhere and that cross-table joins still resolve. You are trading random unpredictability for join-stability on purpose.

Q: How would you prove the same investor maps to the same token across two masked tables?
A: Tokenize the investor key in both tables with the identical deterministic function and salt/key, then run a query that selects the token for a known investor from each table and assert the tokens are equal. Equivalently, join the two masked tables on the token and confirm the expected matched rows appear. If the tokens differ, your salt, key, or function diverged between the two masking runs.

Q: After masking, a cross-table join returns zero rows that previously matched. What is the first thing to check?
A: Check that both sides used identical deterministic masking parameters — same algorithm, same salt or key, same normalisation of the input before hashing — because any divergence makes the same investor produce different tokens and breaks the join. A frequent culprit is Presidio's default random salt or trimming/case differences in the input. Confirm a known investor produces the same token on both sides.

Q: Why must input normalisation come before deterministic tokenization?
A: Because the token is a function of the exact input bytes, ` INV-7781 ` with stray whitespace, `inv-7781` in lowercase, and `INV-7781` will all hash to different tokens and break joins. Normalising (trim, fix case, canonical format) before tokenizing ensures the same logical investor always presents identical input to the function. Skipping this is a classic cause of "the join used to work" failures on masked data.

Q: How do you access-control the reversibility of tokenization?
A: Make de-tokenization require a secret only an authorised role holds — for `encrypt`/`decrypt` that is the cryptographic `key`, for vault tokenization it is access to the vault lookup. Store that key or vault access behind role-based access control so only the authorised role can resolve tokens back to real values. The mask is reversible only to whoever holds the secret, which is exactly the control boundary you want.

Q: Why is an irreversible `hash` token sometimes preferable to a reversible `encrypt` token for a dev environment?
A: If dev never needs to recover the real investor identity, an irreversible salted `hash` removes reversal as an attack surface entirely — there is no key whose leak would re-expose PII. You still get referential consistency for joins because the hash is deterministic. You only choose `encrypt` when some authorised workflow genuinely must re-identify, accepting the obligation to protect the key.

Q: The "Done when" requires correct aggregates with zero real PII downstream. How do you verify zero real PII?
A: Run the full pipeline on the masked replica and scan every output and intermediate dataset for any original PII pattern (real names, national IDs, IBANs), for example with a Presidio analyzer pass or targeted regex checks, expecting no hits. Confirm that the only identifiers present are tokens, and that no unmasked column slipped through the masking config. Finding real PII downstream means a column was missed in the masking specification.

Q: How can you confirm aggregates computed on masked data are still correct?
A: Pick aggregates that do not depend on the masked PII values themselves — counts, sums of amounts, group-bys keyed on the deterministic token — and compare them against the same aggregates computed on the real data; they should match exactly because masking changed identities, not amounts or row membership. If a count or sum differs, masking altered a join key or dropped rows, which is a bug. Referential consistency is what makes the two sets of aggregates agree.

Q: In the fund context, why must investor names and national IDs be masked but ISINs usually need not be?
A: An ISIN identifies a security (an instrument), not a person, so it is reference data rather than personal data and carries no GDPR PII concern, whereas investor names and national IDs directly identify individuals and must be minimized in lower environments. Masking ISINs would also break legitimate joins to instrument reference data for no privacy benefit. The principle is to mask what identifies people, not what identifies instruments.

Q: How does deterministic tokenization let analysts "see funds without seeing investors"?
A: You tokenize the investor identifiers while leaving fund-level reference data (fund, ISIN, NAV, share class) in the clear, so an analyst can aggregate holdings and flows per fund and even per investor-token without ever reading a real name or national ID. The token preserves the grouping so per-investor analytics still work, but the human identity is hidden. This satisfies minimization while keeping the analytics meaningful.

Q: Why is tokenizing the investor key, not just the name, important for a transfer-agency dataset?
A: A transfer-agency record links an investor across registrations, subscriptions, and redemptions via the investor key, so masking only the display name while leaving the key in the clear still lets anyone trace an individual's entire holding history. Tokenizing the key itself breaks that traceability to a real person while keeping the records joinable under the token. You mask the identifier that ties the records together, not only the cosmetic name field.

Q: An EMT or EPT file feeds the masked test pipeline. Which fields are the masking concern?
A: EMT (European MiFID Template) and EPT (European PRIIPs Template) files are largely instrument- and product-level reference and cost data keyed on ISIN, so they typically carry no investor PII and need little masking. The masking concern arises when such files are joined to investor-level subscription or holding data, where the investor key must be tokenized. So you mask on the investor side of the join, not in the EMT/EPT reference fields themselves.

Q: Why might a LEI be left unmasked in a masked test dataset while an investor's national ID is tokenized?
A: A LEI (Legal Entity Identifier) identifies a legal entity such as a fund or corporate counterparty and is public reference data, not personal data, so it is not a GDPR PII concern. A natural person's national ID directly identifies an individual and is sensitive personal data that must be tokenized. The distinction is again person versus entity, and public-reference versus private-personal.

Q: How does masking support GDPR data minimization specifically?
A: Data minimization requires that you process no more personal data than necessary for a purpose; a developer or analyst rarely needs real investor identities, so masking lets the non-production purpose be served with tokens instead of PII. By ensuring lower environments hold no real personal data at rest, you minimise both the exposure and the breach blast radius. Masking operationalises minimization for every environment that does not strictly need real identities.

Q: Why is running dev on un-masked production data treated as a breach risk even if nobody copies it out?
A: Because the personal data is now present in an environment with weaker access controls, more users, and looser logging than production, so the likelihood of accidental disclosure or unauthorised access is materially higher — under GDPR that exposure itself is a processing risk, not just an exfiltration. A lost laptop, an over-broad dev grant, or a shared backup each becomes a reportable incident. Masking removes the personal data so these scenarios no longer expose individuals.

Q: Why is randomised (non-consistent) masked data described as making test results "meaningless"?
A: If the same investor gets different random tokens in different tables, every join on that investor fails or mismatches, so counts, per-investor aggregates, and reconciliation results computed in the test no longer reflect what the real pipeline would produce. The test "passes" or "fails" on artefacts of broken joins rather than on real logic. You cannot trust any result derived from referentially-broken data, so the test is worthless.

Q: What is the privacy versus utility trade-off across the technique zoo?
A: Stronger de-identification (redaction, random tokens) maximises privacy but destroys utility like joins, distributions, and format validity, while utility-preserving techniques (FPE, deterministic tokenization, shuffling) keep the data useful but retain more re-identification risk. Each column's choice balances how much privacy you need against how much realism the downstream test requires. Good masking design picks the strongest technique that still meets the test's utility needs per field.

Q: Why does redaction break referential integrity?
A: Redaction removes the value entirely, so a redacted investor key becomes empty or a constant placeholder and can no longer distinguish one investor from another. Any join that depended on that key collapses — all redacted rows look identical or null. Therefore redaction suits free-text fields you will not join on, not the keys that hold your relational model together.

Q: Why does shuffling preserve column statistics but not row-level truth?
A: Shuffling keeps the exact multiset of real values in the column, so the mean, distribution, and frequency counts are unchanged, but it detaches each value from its original row so no individual record is truthful. Aggregates over the single column stay valid while any cross-column relationship or per-row analysis becomes wrong. It is good for distribution-sensitive testing, bad for anything joining or correlating that column with others.

Q: Why can format-preserving encryption pass schema and validation checks that random tokens fail?
A: FPE outputs ciphertext of the same length, character set, and structure as the input, so an FPE-masked IBAN still has the right length and check-digit format and an FPE-masked account number still fits a `CHAR(16)` column with the expected pattern. Random tokens often violate length, type, or checksum constraints and get rejected by validation or won't fit the column. FPE is the technique when downstream format enforcement is strict.

Q: How is static masking related to "import the masked database results in statically masked data" in SQL Server?
A: When a non-`UNMASK` user exports a DDM database, the export reads masked values, so importing that file produces a database whose columns are statically (permanently) masked — the dynamic mask has been frozen into the data. This is a way to derive a static masked copy from a dynamically masked source. It illustrates how a dynamic-masking feature can yield a static artefact through an export/import cycle.

Q: Why is `EXECUTE AS USER = 'MaskingTestUser'` used when testing DDM, and what is the Fabric caveat?
A: `EXECUTE AS USER` impersonates a non-privileged user so you can confirm masked output without that user holding `UNMASK`, then `REVERT` returns to your own context. In SQL database in Microsoft Fabric the `EXECUTE AS USER` clause is not supported, so you must instead connect as a different real user lacking `UNMASK` to verify the masking. The verification goal is the same; only the impersonation mechanism differs.

Q: When a masked expression is projected, why does it always come back with default masking regardless of the original function?
A: Whenever you wrap a masked column in an expression (a concatenation, a function call), SQL Server applies masking to the resulting expression using the `default()` function rather than the column's configured function. So an `email()`-masked column inside `CONCAT(...)` returns default-masked output, not the email pattern. This is a documented behaviour and can surprise you when a derived column looks differently masked than the base column.

Q: Why doesn't `UNMASK` on its own expose any metadata?
A: Because masking and metadata visibility are independent — `UNMASK` only governs whether stored values are revealed for columns the user can already select, and grants no rights to discover objects. So granting database-scope `UNMASK` plus table-scope `SELECT` lets the user see unmasked rows of that one table and the metadata of only that table, not other tables' metadata. Metadata visibility is controlled separately, per the metadata-visibility configuration.

Q: How does DDM relate to the other security layers it is meant to complement?
A: DDM is explicitly complementary to auditing, encryption, and row-level security, addressing only result-set disclosure to non-privileged readers. Encryption protects data at rest and in transit, row-level security filters which rows a user can see at all, and auditing records who did what; DDM masks which values are shown. A robust design layers all of them rather than relying on masking alone.

Q: In Presidio, why is a salt also a security control, not just a determinism knob?
A: The salt is mixed into the hashed input so that identical plaintext values do not produce a hash an attacker could reverse with a precomputed dictionary of common values. A fixed secret salt gives you both determinism (same input to same token) and resistance to dictionary attacks on the tokens. If the salt leaks, an attacker with the algorithm could brute-force low-entropy values, so the salt must be protected like a key.

Q: Why is a one-way hash token not truly anonymous for a low-cardinality column?
A: If the column has few possible values (for example a country or a small set of fund codes), an attacker who knows the hashing scheme can simply hash every candidate and match the tokens, recovering the originals. Determinism plus low entropy makes the tokens guessable even with a salt if the salt is known. For such columns you may need generalisation, a secret vault, or treating the hash itself as sensitive.

Q: What is the difference between Presidio `mask` and SQL Server `partial()`?
A: Both expose part of a value and hide the rest, but Presidio `mask` operates on detected entities in arbitrary text at processing time (controlled by `chars_to_mask`, `masking_char`, `from_end`), producing a transformed copy, whereas SQL Server `partial()` is a declarative dynamic mask on a database column (signature `partial(prefix,[padding],suffix)`) applied to query results. One is an application-side text de-identifier; the other is an in-engine result-set mask. They implement the same idea in different layers.

Q: Why might you choose Presidio over warehouse-native DDM for the masked investor replica?
A: Because you need static, referentially-consistent tokenization baked into the copy (deterministic `hash` or reversible `encrypt`), which DDM's result-time dynamic masking does not provide, and because Presidio also detects PII in free text where column-level DDM cannot reach. DDM is bypassable by inference and breaks on cross-database joins, while Presidio lets you produce a genuinely de-identified at-rest dataset. The lesson pairs them: DDM for native dynamic masking concepts, Presidio for the technique zoo and consistent tokenization.

Q: What does "masking at rest versus masking in the query result" mean precisely?
A: Masking at rest (static) physically rewrites the stored values so the database itself no longer holds the originals; masking in the query result (dynamic) leaves stored data unchanged and transforms only what a given query returns. The first protects the data even against someone with direct file or backup access; the second depends on every access going through the masking-aware engine. They defend against different threat models.

Q: A dev DB built by static masking still leaks a real email in one column. What likely happened?
A: That column was almost certainly omitted from the masking specification, so it was copied verbatim — static masking only de-identifies the columns you explicitly target. The fix is to add the column to the masking config (and ideally to run a PII detector like Presidio over the whole replica to catch any other missed columns). The root cause is coverage, not the masking technique itself.

Q: Why is FPE often the right choice for masking an IBAN in a payments-testing dataset?
A: An IBAN has a strict country-prefix, length, and check-digit structure that downstream validators enforce, and FPE produces a same-format substitute that passes those checks while being reversible with the key. A random token would fail IBAN validation or break a fixed-length column, and plain redaction would remove a field the payment flow needs. FPE preserves both format validity and controlled reversibility.

Q: Why is masking applied during SQL Server Import and Export when run by a non-`UNMASK` user, and what is the security implication?
A: Because the export reads through the masking layer, a user without `UNMASK` exports masked values, and importing that file yields a statically masked database. The security implication is positive — it means an unauthorised export cannot smuggle out real values — but you must ensure the exporting account genuinely lacks `UNMASK`, or the export will contain real data. Always verify the export was done under a non-`UNMASK` identity.

Q: Why does the lesson tie masking-versus-synthesis back to lesson 9.8.1?
A: Because 9.8.1 covers synthetic data generation, and the choice between masking and synthesis is a single design decision on the fidelity-versus-disclosure spectrum, not two unrelated topics. Knowing when masked real data suffices versus when you must fabricate synthetic data is the same skill viewed from the privacy side. The cross-reference signals you should reason about both options together when provisioning lower environments.

Q: What is a "masked replica" as required by the lesson's first Do step?
A: It is a copy of the Phase-1 investor tables where the identifying columns have been transformed by deterministic tokenization, producing a dataset that is structurally identical and join-consistent but contains no real PII at rest. It is static masking applied once to create a reusable lower-environment dataset. The full pipeline then runs against this replica instead of production.

Q: Why run the full pipeline (not just a spot check) on masked data per the Done-when?
A: Because masking bugs often surface only in end-to-end behaviour — a broken token join might still let a spot query succeed but cause aggregates, reconciliations, or NAV roll-ups to silently diverge downstream. Running the whole pipeline and getting correct aggregates is the only convincing proof that referential integrity survived masking. A spot check can miss a join that breaks two stages later.

Q: How would you demonstrate that reversibility is access-controlled, as the Done-when requires?
A: Show that a normal role attempting to de-tokenize fails for lack of the key or vault access, while only the designated authorised role can run the `decrypt` (or vault lookup) and recover the original. Concretely, attempt de-tokenization under two identities and prove only the privileged one succeeds. The control is demonstrated by the failure of the unauthorised attempt as much as the success of the authorised one.

Q: Why is the salt or key a single point of failure in deterministic/reversible tokenization?
A: Because every token's relationship to its real value depends on that one secret: leak the `encrypt` key and all tokens become reversible; leak the `hash` salt and low-entropy tokens become brute-forceable. The secret concentrates the entire scheme's confidentiality. Therefore it must be stored in a secrets manager or vault with tight RBAC, rotated carefully, and never committed to code or shipped to dev.

Q: If you rotate the tokenization key or salt, what breaks and how do you handle it?
A: Rotating the secret changes the token for every value, so newly-masked data will not join to previously-masked data and historical tokens become unresolvable unless you retain the old key for decryption. You handle it by re-tokenizing all affected datasets together under the new secret, or by versioning keys and keeping old ones available for reversal. The gotcha is partial rotation, which silently breaks joins between old and new masked data.

Q: Why must the same masking function and parameters be applied across all tables in one masking run?
A: Because referential consistency is only guaranteed when every table transforms the investor key with an identical function, salt/key, and input normalisation; any divergence yields different tokens for the same investor and breaks joins. Treating the masking parameters as one shared configuration for the whole run is what keeps the tokens aligned. This is why masking is best driven by a single config rather than per-table ad hoc commands.

Q: How could Presidio detection feed the masking step in an automated flow?
A: The Presidio analyzer detects which columns or text spans contain PII entities (names, national IDs, IBANs), and those detections drive which fields the anonymizer must transform and with which operator. This ties to the classification-tag idea in the neighbouring lesson, where detected PII becomes catalog tags that the masking layer consumes. Detection answers "what to mask," and the operators answer "how to mask it."

Q: Why is masking only the obvious PII columns insufficient for true de-identification?
A: Because combinations of quasi-identifiers — birth date, postcode, fund holdings, nationality — can re-identify an individual even when names and IDs are tokenized, a linkage risk masking of direct identifiers alone does not remove. Effective de-identification considers these quasi-identifiers, applying generalisation or suppression where needed. Masking design must look beyond the columns labelled "PII" to the columns that, together, single someone out.

Q: What is the difference between the Presidio `hash` and `encrypt` operators for tokenization?
A: `hash` is a one-way salted hash — deterministic and joinable but irreversible — so use it when no authorised party ever needs the original back. `encrypt` is reversible with a `key` (recovered via `decrypt`), so use it when controlled re-identification is required, at the cost of having a key to protect. Both can give referential consistency; the deciding factor is whether reversal must be possible.

Q: Why does the lesson pair SQL Server DDM with Presidio rather than treating one as sufficient?
A: Because they illustrate complementary halves of the topic: DDM teaches warehouse-native dynamic masking and its real-world inference limits, while Presidio teaches the technique zoo, deterministic tokenization, and referential consistency for at-rest masking and free text. Neither alone covers both dynamic query-time masking and static consistent tokenization. Together they map the full landscape the lesson's Learn bullets demand.

Q: A masked dataset passes joins but an analyst can still recover a few real investors. What technique gap likely exists?
A: The direct identifiers were tokenized but quasi-identifiers (e.g. exact holding amounts, rare fund combinations, birth dates) were left in the clear, enabling linkage re-identification. The gap is failing to generalise or suppress quasi-identifiers, not the tokenization of the keys. The fix is to coarsen or remove the columns that, combined, uniquely fingerprint an individual.

Q: Why is "the same investor → same token everywhere" the single most important property for masked fund data?
A: Because fund data is deeply relational — investors link to accounts, subscriptions, redemptions, and holdings across many tables — and if the token is not consistent, every cross-table operation that reconstructs an investor's position fails. Consistency is what keeps NAV roll-ups, reconciliations, and per-investor reporting correct on masked data. Lose it and the masked dataset cannot support the very pipeline you are testing.

Q: How would you handle a free-text "client notes" field that must be masked but is not a clean column of identifiers?
A: Run it through the Presidio analyzer to detect embedded PII entities (names, phone numbers, IBANs) within the prose, then apply anonymizer operators — `redact` or `replace` for unstructured mentions, or deterministic `hash` where the entity must stay joinable. Column-level DDM cannot reach PII buried in free text, which is exactly where Presidio's NER-based detection is needed. The notes field illustrates why detection must precede masking.

Q: Why is dynamic masking dependent on "every access going through the masking-aware engine"?
A: Because dynamic masking transforms data only at query time inside the engine, so any path that bypasses it — a raw file copy, a backup restore, replication, or a direct storage read — sees the unmasked stored values. The protection is only as strong as the guarantee that all access flows through the masking layer. Static masking does not share this weakness because the originals are gone from storage.

Q: What does it mean operationally that DDM "doesn't change the data in the database"?
A: It means the real values remain on disk and in backups, only the result returned to non-`UNMASK` users is obscured, so backups, replicas, and storage-level access still contain the true PII. Operationally you must protect those at-rest copies separately, since DDM gives them no protection. For a dev replica that must hold no PII at rest, this is precisely why DDM alone does not satisfy the requirement.

Q: Why are SOX-style "who can de-tokenize" controls part of masking design rather than an afterthought?
A: Because the value of tokenization as a privacy control collapses if anyone can reverse it; the access control on de-tokenization is what makes the masked data genuinely de-identified to unauthorised users. In a regulated fund context an auditor will ask who can re-identify investors and how that is restricted and logged. So restricting and auditing reversal is a first-class design requirement, captured directly in the lesson's Done-when.

Q: A masked replica's NAV-by-fund totals differ from production by a few rows. What masking failure should you suspect first?
A: Suspect that the investor or fund key tokenization was not perfectly deterministic across all the joined tables, so a few investors got mismatched tokens and their holdings dropped out of or double-counted in the join. Because NAV roll-ups depend on every holding joining correctly to its fund and investor, even a small tokenization inconsistency shifts the totals. Verify a sample of known keys produce identical tokens on every table involved in the roll-up.

Q: Summarise the decision rule for picking a masking technique per column.
A: Choose the technique by what the column must still do downstream: redact or `replace` for fields you neither join on nor analyse, deterministic `hash`/tokenization for join keys that must stay consistent but never reversed, `encrypt`/FPE for values needing reversibility or strict format, and shuffling only for single-column distribution tests with no cross-column use. Always also mask or generalise quasi-identifiers that could re-identify in combination. The rule is to apply the strongest de-identification a column's required utility can tolerate.
