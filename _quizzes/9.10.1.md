# Phase 5 · 9.10.1 Infrastructure as code (OpenTofu/Terraform) — 100 self-test questions

Q: What does "infrastructure as code" (IaC) mean, and why does it matter in a regulated fund-administration estate?
A: IaC means defining infrastructure — networks, storage, compute, policies — in version-controlled declarative files instead of clicking through a portal, so that environments are reproducible and auditable. In a regulated estate it doubles as the audit trail: reviewers want the exact resources, firewall rules, and IAM policies as reviewed, version-controlled code, not portal screenshots. Without it environments drift, DR rebuilds become guesswork, and there is no four-eyes review on infrastructure change.

Q: What is the difference between declarative and imperative infrastructure tooling, and which category does OpenTofu fall into?
A: Imperative tooling runs ordered steps ("create this, then that"), whereas declarative tooling describes the desired end state and the engine computes the steps to reach it. OpenTofu and Terraform are declarative: you write the resources you want and `tofu plan` figures out what to create, change, or destroy to converge reality onto that desired state. The gotcha is that ordering still matters internally — the engine builds a dependency graph from references rather than from the line order in your files.

Q: What is HCL, and what is its role in OpenTofu?
A: HCL is HashiCorp Configuration Language, the declarative, block-based language in which you write `.tf` files that OpenTofu reads. It expresses providers, resources, variables, outputs, and expressions in a human-friendly syntax that is also machine-parseable as JSON. HCL is what you author; the OpenTofu engine parses it, builds a graph, and reconciles it against state.

Q: In HCL, what is a `provider` and what does it do?
A: A provider is a plugin that teaches OpenTofu how to talk to a specific platform's API — for example `azurerm` for Azure, `aws` for AWS, `docker`, `kubernetes`, or `helm`. It exposes the resource and data-source types you can declare and handles authentication and the CRUD calls behind them. You configure it in a `provider` block and pin its version in `required_providers` so plans are reproducible.

Q: What is a `resource` block in HCL, and what does each instance of one represent?
A: A `resource` block declares one managed object that OpenTofu will create, update, and destroy on your behalf — for example `resource "azurerm_storage_account" "nav_files" { ... }`. The two labels are the resource type and a local name; together they form the address `azurerm_storage_account.nav_files` used in references and in state. OpenTofu owns the full lifecycle of that object, so removing the block schedules it for destruction.

Q: What is a `data` source, and how does it differ from a `resource`?
A: A data source reads information about something that already exists without managing its lifecycle — for example `data "azurerm_key_vault" "shared"` looks up an existing vault. A `resource` is created and destroyed by OpenTofu; a `data` source is read-only and never modified or deleted by your config. Data sources are how you reference infrastructure managed elsewhere, such as a central networking team's VNet.

Q: What are input variables in OpenTofu and how do you declare one?
A: Input variables parameterise a configuration so the same code can be reused across environments; you declare them with a `variable "name" { type = ... default = ... }` block. They are referenced as `var.name` and can be set via `-var`, a `.tfvars` file, `TF_VAR_` environment variables, or interactive prompt. Adding a `type` and `description`, and marking secrets `sensitive = true`, makes them self-documenting and safer.

Q: What are outputs, and why are they useful in a composed estate?
A: Outputs expose selected values from a configuration with an `output "name" { value = ... }` block, printed after apply and queryable with `tofu output`. They are the public interface of a module: a network module can output a subnet id that a downstream storage module consumes. In a multi-stack estate, outputs from one state are read by another via `terraform_remote_state`, wiring stacks together without hardcoding ids.

Q: What is a local value (`locals`) and when would you use one instead of a variable?
A: A `locals` block defines named expressions computed inside the configuration, referenced as `local.name`, used to avoid repeating a complex expression or to assemble derived values like a naming prefix. Unlike variables, locals cannot be overridden from outside — they are internal helpers, not inputs. Use a variable when a caller should set the value; use a local when you are just naming a computed expression such as `local.tags = merge(var.base_tags, { env = var.env })`.

Q: How does OpenTofu decide the order in which to create resources if you never specify it?
A: It builds a directed acyclic dependency graph from the references between resources: if resource B uses `azurerm_subnet.a.id`, OpenTofu infers B depends on A and creates A first. Order in the file is irrelevant; only references and `depends_on` matter. This is why expressing real references (rather than hardcoding ids) is important — it lets the engine sequence and parallelise correctly.

Q: When and why would you use `depends_on`?
A: `depends_on` declares an explicit ordering dependency that OpenTofu cannot infer from references — for example a resource that needs an IAM role to exist first even though it does not reference the role's attributes. It forces the named dependency to be fully applied before the dependent resource. Overusing it is a smell; prefer natural references and reserve `depends_on` for hidden, side-effect dependencies.

Q: What does the `count` meta-argument do, and what is its main pitfall?
A: `count` creates N instances of a resource indexed `[0]`, `[1]`, …; you reference them by index and address them as `resource.name[0]`. Its pitfall is that instances are tracked by position, so removing an element from the middle of a count-based list shifts every later index and forces unnecessary destroy/recreate. For keyed collections prefer `for_each`, which tracks instances by a stable map key instead of position.

Q: How does `for_each` differ from `count`, and when is it the better choice?
A: `for_each` iterates over a map or set of strings, creating one instance per key addressed as `resource.name["key"]`, so instances are tied to stable keys rather than ordinal positions. It is the better choice whenever the collection can change in the middle, because adding or removing one element does not disturb the others. For example a `for_each` over a map of fund codes keeps `["LU0001"]` stable even when `["LU0002"]` is added.

Q: What is the difference between `tofu validate` and `tofu plan`?
A: `tofu validate` checks that the configuration is internally consistent — syntax, types, references — without contacting any provider or reading state, so it is fast and offline. `tofu plan` goes further: it reads current state, refreshes against the real provider APIs, and produces the concrete set of changes it would make. Validate catches authoring errors in CI early; plan is the review artifact that shows real-world impact.

Q: What does `tofu fmt` do, and why run it in CI?
A: `tofu fmt` rewrites `.tf` files to the canonical HCL style — consistent indentation, alignment, and spacing — so diffs reflect real changes, not formatting noise. Running `tofu fmt -check` in CI fails the build if any file is unformatted, keeping the whole estate stylistically uniform. It is purely cosmetic and never changes behaviour, which makes it safe to enforce strictly.

Q: What is OpenTofu state, and what fundamental problem does it solve?
A: State is the JSON file (`terraform.tfstate`) that maps each resource in your configuration to the real object it manages, recording ids and last-known attribute values. It solves the binding problem: OpenTofu has no other way to know that `azurerm_storage_account.nav_files` corresponds to a specific storage account it created earlier. Without state, every plan would try to recreate everything because it could not match config to reality.

Q: Why should production state live in a remote backend rather than a local file?
A: A local `terraform.tfstate` on one laptop cannot be shared by a team, is easily lost, and offers no locking — two engineers applying at once would corrupt it. A remote backend (S3, Azure Blob, GCS, an HTTP backend) stores state centrally so the whole team and CI share one source of truth, and most support locking. In a regulated estate it also centralises the state for backup, encryption, and access control.

Q: What is state locking and what failure does it prevent?
A: State locking takes an exclusive lock for the duration of a `plan` or `apply` so that only one operation mutates state at a time. It prevents two concurrent applies from interleaving writes and corrupting the state file, which can leave resources orphaned or duplicated. Most backends acquire the lock automatically; if a previous run crashed mid-lock you may need `tofu force-unlock <LOCK_ID>` after confirming no apply is actually running.

Q: How does state locking work for the S3 backend in current versions, and what changed from the older approach?
A: The modern S3 backend supports native S3 locking via `use_lockfile = true`, which writes a companion lock object next to the state with a `.tflock` extension. This replaced the older pattern of a separate DynamoDB table referenced by `dynamodb_table`, which is now deprecated and slated for removal. Native locking went GA in Terraform 1.11; OpenTofu offers the equivalent, so new estates should prefer `use_lockfile` over a DynamoDB lock table.

Q: What is drift, and how can it arise in a fund estate?
A: Drift is any divergence between the real infrastructure and what state records — for example someone widened a storage account's firewall in the Azure portal during an incident. It arises from out-of-band changes: manual portal edits, other tools, or automated remediation. Drift is dangerous in regulated estates because the code no longer reflects reality, so the next apply may silently revert a hotfix or, worse, the audit trail is wrong.

Q: How do you detect drift without proposing any configuration changes?
A: Run `tofu plan -refresh-only`, which refreshes state against the real APIs and reports differences between recorded and actual attributes, without trying to apply your configuration's desired changes. Pair it with `-detailed-exitcode` in CI to get a machine-readable signal. This is the standard scheduled drift-detection job — it tells you reality moved without touching it.

Q: With `-detailed-exitcode`, what do exit codes 0, 1, and 2 mean?
A: Exit code 0 means the plan succeeded with no changes (no drift), 1 means an error occurred, and 2 means the plan succeeded and changes are present (drift or pending changes). CI uses this to branch: 0 is green, 1 fails the job, and 2 raises an alert or opens a ticket. The trichotomy is the reason `-detailed-exitcode` exists — a plain plan returns 0 even when there are changes.

Q: A scheduled `tofu plan -refresh-only -detailed-exitcode` job returns exit code 2 every night even right after a clean apply — what would you check first?
A: First confirm whether real drift exists by reading the plan output: a provider that normalises or recomputes a value on read (a known class of false positives) can report a change even with no human edit. Check the specific attribute flagged, the provider version, and whether a `lifecycle { ignore_changes = [...] }` is warranted for that noisy attribute. If a genuine out-of-band change is shown, remediate by reapplying or by codifying the change; do not blanket-ignore exit code 2.

Q: What is the purpose of `tofu import`, and what does it do to your configuration?
A: `tofu import` brings an already-existing resource under OpenTofu management by writing its real id into state, so future plans manage it instead of trying to recreate it. Crucially, the classic CLI command only writes state — it does not generate the matching HCL, so you must author the `resource` block yourself to match. It is imperative and one resource at a time, which is why it is error-prone for large brownfield estates.

Q: How does the config-driven `import` block differ from the legacy `tofu import` command?
A: The `import` block (introduced in Terraform 1.5 and supported by OpenTofu) is declarative: you add an `import { to = ... id = ... }` block to your config and the import becomes part of the normal plan/apply cycle, previewable before it touches state. Unlike the imperative command it is plannable, reviewable in CI, can run in bulk, and can generate starter HCL with `-generate-config-out`. This makes adopting a large brownfield estate far safer than the one-at-a-time command.

Q: You inherit a Luxembourg fund platform built entirely by hand in the portal and must bring it under IaC — what is your high-level import strategy?
A: Inventory the existing resources, then bring them under management incrementally using config-driven `import` blocks, generating starter HCL and refining it until `tofu plan` shows no changes for each imported resource. Work stack by stack (network, storage, compute) so blast radius stays small, and never run an apply until plan is clean, to avoid recreating live infrastructure. Capture each clean plan as the evidence that import matched reality before you hand over change control.

Q: After importing a resource, `tofu plan` shows it wants to destroy and recreate it — what does that indicate and how do you fix it?
A: It indicates your hand-written HCL does not match the real object on an attribute that forces replacement, so OpenTofu thinks it must create a new one. Compare the plan's diff to find the mismatched force-new attribute and align your configuration to the imported reality, or move it into `lifecycle { ... }` if appropriate. The goal of an import is a clean, no-op plan; a replace plan means the config and reality still disagree.

Q: What is a module in OpenTofu, and what is the root module?
A: A module is a reusable container of `.tf` files with defined input variables and outputs — a unit like "a VNet" or "a storage account with a key vault" you can instantiate many times. The root module is simply the directory where you run `tofu`; it can call child modules with `module "name" { source = ... }`. Modules are how you avoid copy-pasting infrastructure and enforce one approved pattern across many environments.

Q: What goes in the typical `main.tf`, `variables.tf`, and `outputs.tf` split of a module?
A: By convention `variables.tf` holds the module's input `variable` blocks, `outputs.tf` holds its `output` blocks, and `main.tf` holds the resources and data sources; the split is organisational, since OpenTofu loads all `.tf` files in a directory together regardless of name. This convention makes a module's interface — its inputs and outputs — easy to find without reading the implementation. The filenames carry no special meaning to the engine; only the directory boundary defines the module.

Q: What does the `source` argument of a `module` block accept?
A: `source` tells OpenTofu where to fetch the module's code: a local path (`./modules/network`), a Git URL, a Terraform/OpenTofu registry reference, or an archive over HTTP. Combined with a `version` constraint for registry modules, it pins exactly which module code is used. Local paths are best for modules private to one repo; registry or Git sources let you share a versioned module across many repos.

Q: Why is a network module a good candidate for factoring out, and what should its inputs and outputs be?
A: A network module (VNet/VPC, subnets, security groups) is identical in shape across environments and is consumed by almost everything else, so factoring it out enforces one approved network design. Its inputs are things like address space, environment name, and tags; its outputs are subnet ids and the VNet id that storage and compute modules consume. Centralising it means a firewall-rule change is reviewed once and applied uniformly, which matters for regulated network controls.

Q: What is the practical difference between OpenTofu workspaces and a directory-per-environment layout?
A: Workspaces let one configuration carry multiple named state files (`dev`, `prod`) selected with `tofu workspace select`, while directory-per-env gives each environment its own directory, backend config, and variable files. Workspaces are lightweight but share code and backend, making accidental cross-environment changes easier and per-env differences awkward. Directory-per-env is more verbose but gives strong isolation, separate state and credentials per environment, and clearer blast-radius boundaries — usually preferred for regulated prod separation.

Q: Why might a regulated fund-admin shop prefer directory-per-environment over workspaces for prod isolation?
A: Because prod must be strongly isolated: separate state, separate backend, separate credentials, and an independent change-approval path, all of which are clearer with a dedicated directory than with a workspace sharing one backend. Workspaces make it easy to run the wrong command against prod from a dev checkout, and they complicate per-environment policy and access controls. The directory boundary becomes the natural place to attach four-eyes review and least-privilege credentials for prod.

Q: What does the `terraform { required_version = ... }` setting do and why pin it?
A: `required_version` constrains which OpenTofu/Terraform CLI versions may run the configuration, failing fast if someone uses an incompatible binary. Pinning it prevents a teammate or CI runner on a different version from producing a subtly different plan or using unsupported syntax. In a regulated estate, version-pinning the CLI and providers is part of making plans reproducible and auditable.

Q: Why pin provider versions in `required_providers`, and what is the role of the lock file?
A: Pinning with a version constraint in `required_providers` stops an unexpected provider upgrade from silently changing planned behaviour between runs. The dependency lock file (`.terraform.lock.hcl`) records the exact provider versions and their checksums, and is committed to the repo so every machine and CI run resolves identical providers. Together they make plans deterministic — a core requirement when the plan is your audit artifact.

Q: Why is `terraform plan` (or `tofu plan`) called the review artifact in a CI-gated workflow?
A: Because the plan shows exactly what will change — every create, update, replace, and destroy — before anything happens, so it is the document reviewers approve. In a gated pipeline the plan is generated and posted on the pull request, a human reviews and approves it, and only then does a separate gated `apply` run. This separation gives four-eyes control over infrastructure change and a durable record of what was approved.

Q: What is the value of running `tofu plan -out=plan.tfplan` and applying that exact file later?
A: Saving the plan to a file with `-out` captures the precise set of changes that was reviewed; passing that file to `tofu apply plan.tfplan` applies exactly those changes with no re-planning. This closes the gap where state or reality drifts between review and apply, so what is applied is provably what was approved. It is the standard pattern for a gated CI pipeline where plan and apply are separate, reviewed steps.

Q: In a typical PR-driven OpenTofu pipeline, which step runs on the pull request and which is gated behind a merge or manual approval?
A: `tofu plan` runs on the pull request and posts its output for review, acting as the read-only proposal; `tofu apply` is gated behind merge to the main branch and/or an explicit manual approval. This keeps applies off feature branches and ensures every change to real infrastructure passes human sign-off. The gate is where regulated four-eyes and segregation-of-duties controls attach.

Q: In `tofu plan` output, what do the symbols `+`, `-`, `~`, and `-/+` mean?
A: `+` means a resource will be created, `-` means it will be destroyed, `~` means it will be updated in place, and `-/+` (also shown as "replace") means it will be destroyed and then recreated. Reading these symbols is how you assess blast radius before approving. A surprise `-/+` on a stateful resource like a database is a red flag that demands scrutiny before apply.

Q: What is the difference between an in-place update and a replacement in a plan, and why does it matter?
A: An in-place update (`~`) modifies the existing object while keeping its identity and data; a replacement (`-/+`) destroys and recreates it, which can mean downtime and data loss. It matters because some attributes are "force new" — changing them requires replacement — so an innocent-looking edit can silently trigger destruction. Always check whether a plan replaces stateful resources like a storage account or database before approving.

Q: A teammate's plan shows `azurerm_storage_account.nav_files must be replaced` — how do you explain it and what is the risk?
A: It means an attribute that cannot be changed in place (a "force new" property such as the account name or certain network settings) was modified, so OpenTofu will destroy the existing account and create a new one. The risk is data loss and downtime — any NAV or EMT/EPT files in that account would be destroyed with it. Before approving, identify which attribute forces replacement and decide whether a safer migration path is needed instead.

Q: What does `lifecycle { prevent_destroy = true }` do and where would you use it?
A: It makes OpenTofu error out rather than destroy that resource, so a plan that would delete it fails instead of proceeding. You put it on irreplaceable, stateful resources — a production database, a state-storing account, a vault — to stop an accidental destroy or replace. The gotcha is you must remove the flag deliberately when you genuinely intend to delete the resource, which is itself a useful speed bump.

Q: What does `lifecycle { create_before_destroy = true }` change about a replacement?
A: It reverses the default replace order: the new resource is created first and the old one destroyed only after, minimising downtime during a replacement. The default is destroy-then-create, which can cause an outage window. Use it where a brief overlap is acceptable and downtime is not — but beware unique-name or capacity constraints that prevent two instances coexisting.

Q: Why must secret values never be stored in clear text in OpenTofu state?
A: Because state records the last-known values of all managed attributes, including passwords and keys, in plain JSON; anyone who can read state can read those secrets. In a regulated estate this is a serious exposure since state is shared and backed up. The rule is to keep secrets out of clear-text state by referencing a secrets manager instead of materialising the secret into a managed attribute.

Q: How should a configuration obtain a database password or API key instead of hardcoding it?
A: Reference a secrets manager at apply time — for example a `data "azurerm_key_vault_secret"` for Azure Key Vault, or an equivalent OpenBao/HashiCorp Vault data source — so the value is fetched rather than written into code. The secret lives in the vault under its own access controls and rotation, and the configuration only holds a reference. Even so, the resolved value can still land in state, so combine this with sensitive marking and state encryption.

Q: What does marking a variable or output `sensitive = true` actually do, and what does it not do?
A: `sensitive = true` redacts the value in CLI output and plan diffs so it does not appear in logs or PR comments. What it does not do is encrypt or remove the value from the state file — a sensitive value can still be stored in clear in state. So sensitive marking protects logs and screen output, not the state-at-rest exposure, which needs encryption or keeping secrets out of state entirely.

Q: What is OpenTofu's state encryption feature, and which version introduced it?
A: OpenTofu added client-side state and plan encryption in version 1.7.0, configured in a `terraform { encryption { } }` block (or via the `TF_ENCRYPTION` environment variable). It encrypts the state and plan files end-to-end before they reach any backend, so the stored bytes are ciphertext regardless of where they live. This directly addresses the clear-text-secrets-in-state problem and is a meaningful differentiator from classic Terraform.

Q: What key providers and encryption method does OpenTofu state encryption support?
A: It uses AES-GCM as the encryption method and supports several key providers: PBKDF2 (a passphrase, requiring at least 16 characters and a high iteration count), plus key-management backends such as AWS KMS, GCP KMS, Azure Key Vault, and OpenBao. PBKDF2 is convenient for a passphrase but a managed KMS/OpenBao key is preferable in production for rotation and access control. The key provider supplies the key; AES-GCM does the actual symmetric encryption.

Q: What does setting `enforced = true` in an OpenTofu encryption block guarantee?
A: With `enforced = true` under the `state` (and/or `plan`) section, OpenTofu refuses to read or write unencrypted state — if the encryption key is missing it errors instead of silently falling back to plaintext. This prevents the accidental write of a clear-text state file, which is exactly the failure you want to make impossible in a regulated estate. Without enforcement, a misconfiguration could quietly produce an unencrypted state.

Q: Even with a secret fetched from a vault and a sensitive marking, what is the residual exposure and what closes it?
A: The residual exposure is that the resolved secret value can still be written into the state file in clear, because state records attribute values regardless of sensitive marking. What closes it is state encryption (encrypting the state at rest) combined, where possible, with designs that avoid placing the secret into a managed attribute at all. Defence in depth here is: secret in vault, sensitive marking for logs, and encrypted state for the at-rest copy.

Q: How does a `for_each` over fund identifiers help keep secret references clean across many funds?
A: Using `for_each` over a map keyed by stable identifiers (such as fund codes or ISINs) lets each instance pull its own vault secret by key, e.g. a Key Vault secret named per fund, without copy-pasting blocks. Because instances are keyed, adding or removing a fund does not disturb the others' secret bindings or force needless replacements. The pattern keeps the secret in the vault and the config holding only a per-key reference.

Q: What is the historical event behind the OpenTofu fork of Terraform?
A: In August 2023 HashiCorp changed Terraform's licence from the open-source MPL 2.0 to the Business Source License (BSL) 1.1, which restricts using Terraform to build competing products. The community responded by forking the last MPL-licensed code into OpenTofu, now governed by the Linux Foundation. The fork is the reason there are two compatible engines whose CLI commands map one-to-one.

Q: What licence does OpenTofu use, and what licence does current Terraform use?
A: OpenTofu is licensed under the Mozilla Public License 2.0 (MPL 2.0), an OSI-approved open-source weak-copyleft licence. Current Terraform is licensed under the Business Source License 1.1 (BSL), which the OSI does not classify as open source. This licence divergence — MPL versus BSL — is the core of the Terraform↔OpenTofu split.

Q: What does the BSL's "competitive use" restriction actually limit, and why does it matter for vendor selection?
A: The BSL forbids using the licensed Terraform code to build a product that competes with HashiCorp's offerings, with each release converting to a permissive licence only after about four years. For most internal users this changes little day to day, but for vendors, platform builders, and risk/legal reviewers it is a material constraint to assess. It matters for vendor selection because some regulated organisations prefer the unambiguously open MPL-licensed OpenTofu to avoid future licence risk.

Q: Why do Terraform skills and commands transfer almost directly to OpenTofu?
A: OpenTofu forked from Terraform's codebase, so the HCL language, command structure, and concepts are deliberately compatible — `terraform plan` maps to `tofu plan`, modules and providers work the same way, and most learning resources apply to both. The binaries differ and they have diverged in features (for example OpenTofu's state encryption), but the mental model and most workflows are shared. This is why a Terraform-centric book teaches OpenTofu concepts effectively.

Q: Beyond licence, name one notable feature OpenTofu has that classic Terraform lacks.
A: OpenTofu shipped client-side state and plan encryption (in 1.7.0), letting you encrypt state end-to-end regardless of backend, which classic Terraform does not provide natively in the same way. OpenTofu has also added features like provider-defined functions and early evaluation as it diverged. These divergences mean the two engines are compatible but no longer identical, which is part of the build-vs-buy and vendor-selection calculus.

Q: What does `tofu init` do, and when must you re-run it?
A: `tofu init` initialises a working directory: it downloads the required providers and modules, configures the backend, and writes the dependency lock file. You must re-run it after changing provider/module sources or versions, after changing the backend, or in a fresh checkout or CI runner. Forgetting to re-init after a backend or provider change is a common cause of confusing errors.

Q: A `tofu plan` fails with an error that the backend is not initialised — what is the first thing to do?
A: Run `tofu init` in that directory, because plan requires an initialised backend and downloaded providers, and the error means initialisation is missing or stale. This happens on a fresh clone, in a CI job that did not cache `.terraform`, or after someone changed the backend configuration. If init itself fails, check backend credentials and the backend config block next.

Q: What is the recommended workflow loop for making an infrastructure change with OpenTofu?
A: Edit the HCL, run `tofu fmt` and `tofu validate`, then `tofu plan` to review the proposed changes, and finally `tofu apply` once the plan is approved. In a team setting plan runs in CI on the PR and apply is gated behind approval/merge. The discipline is: never apply a plan you have not read, and never apply on a feature branch in a regulated pipeline.

Q: What does `tofu destroy` do, and why is it dangerous in shared environments?
A: `tofu destroy` plans and executes the deletion of every resource the configuration manages, returning the environment to nothing. It is dangerous because in a shared or production environment it will tear down live infrastructure — and any data in stateful resources — if pointed at the wrong state. Guard rails include `prevent_destroy` on critical resources, separate prod credentials, and never running destroy outside an isolated, disposable environment.

Q: How do you reproduce a platform byte-for-byte to prove reproducibility, and what does success look like?
A: Run `tofu destroy` to tear the platform down, then `tofu apply` from the same code and pinned versions to rebuild it, and confirm the resulting platform is functionally identical. Success is a clean rebuild with no manual steps and a final `tofu plan` showing no changes, demonstrating the code fully describes the platform. Capturing the plan output as evidence turns this into an audit-ready reproducibility proof.

Q: What is the goal of putting the entire compose-era local stack under OpenTofu using the docker, kubernetes, and helm providers?
A: The goal is that a single `tofu apply` builds the full local platform — containers, Kubernetes objects, and Helm releases — so the environment is reproducible and version-controlled rather than assembled by ad-hoc scripts. The `docker`, `kubernetes`, and `helm` providers let OpenTofu manage each layer declaratively. This makes the local stack a faithful, rebuildable mirror of the intended architecture.

Q: Why validate an `azurerm` module with `tofu plan` against a free-trial subscription as plan-only, no spend?
A: Because `tofu plan` exercises the provider, schema, and your module against the real Azure API and shows exactly what would be created, without calling apply, so no resources are provisioned and no cost is incurred. This proves the module is correct against the live API while staying within a no-spend constraint. It is how you validate cloud IaC safely before any budget approval to actually apply.

Q: What is the difference between the `azurerm` provider and the `azuread` provider?
A: The `azurerm` provider manages Azure Resource Manager resources — VNets, storage accounts, key vaults, VMs — the infrastructure plane. The `azuread` provider manages Microsoft Entra ID (formerly Azure AD) objects like applications, service principals, and groups — the identity plane. A full landing zone often uses both: `azurerm` for resources and `azuread` for the identities that access them.

Q: What does the `helm` provider manage, and how does it relate to the `kubernetes` provider?
A: The `helm` provider manages Helm releases — installing and upgrading packaged charts into a cluster — while the `kubernetes` provider manages individual Kubernetes objects like deployments, services, and config maps directly. They are complementary: use `helm` for off-the-shelf charts and `kubernetes` for bespoke manifests. Both typically authenticate to the same cluster via a kubeconfig or in-cluster credentials.

Q: What is the role of the `terraform_remote_state` data source in a multi-stack estate?
A: `terraform_remote_state` reads the outputs of another configuration's state, letting one stack consume values produced by another — for example a compute stack reading a network stack's subnet ids. It is how you compose independently-applied stacks without hardcoding ids or merging everything into one giant state. The trade-off is a coupling between stacks; changing a consumed output can ripple downstream.

Q: Why is splitting a large estate into multiple smaller states (stacks) often preferable to one monolithic state?
A: Smaller states limit blast radius (a mistake affects one stack, not everything), speed up plan/apply, reduce lock contention, and let teams own and apply their stacks independently with least-privilege credentials. A single monolithic state becomes slow, risky, and a coordination bottleneck. The cost is wiring stacks together via outputs and `terraform_remote_state`, plus disciplined dependency management between them.

Q: What is the OpenTofu/Terraform registry, and why pin a `version` constraint on registry modules and providers?
A: The registry is the public (and private) catalogue from which providers and modules are downloaded by `source`. You pin a `version` constraint (for example `~> 4.0`) so upgrades are deliberate and plans stay reproducible, rather than silently pulling a new major version that changes behaviour. Combined with the lock file, version constraints make builds deterministic across machines and over time.

Q: What does a version constraint like `~> 4.1` mean?
A: The pessimistic operator `~> 4.1` allows `4.1` and any higher patch within the same minor series — that is, `>= 4.1, < 4.2` at the patch level — while `~> 4.0` would allow `>= 4.0, < 5.0`. It lets you take bug-fix updates automatically while blocking breaking minor or major bumps. Choosing the right granularity balances getting fixes against avoiding surprise behaviour changes.

Q: What is a sensitive output, and how does OpenTofu treat it?
A: An `output` marked `sensitive = true` is redacted in CLI output so its value is not printed after apply or in `tofu output` without `-raw`. Like sensitive variables, this protects logs and screens but does not encrypt the value in state. Marking module outputs sensitive also prevents the value from being accidentally exposed when a parent module surfaces it.

Q: What does `tofu state list` show and when would you use it?
A: `tofu state list` prints the addresses of every resource currently tracked in state, like `azurerm_storage_account.nav_files`. You use it to discover what is managed, to find the exact address for a targeted operation, or to audit state contents. It is a read-only inspection command and a good first step when reconciling config against state.

Q: What does `tofu state show <address>` do?
A: `tofu state show azurerm_key_vault.shared` prints the full recorded attributes of that one resource from state, including ids and current values. It is how you inspect what OpenTofu believes a resource looks like, useful when debugging drift or a confusing plan. Remember the output can contain sensitive values stored in state, so treat it with the same care as the state file itself.

Q: When and why would you use `tofu state mv`?
A: `tofu state mv` renames or moves a resource's entry in state — for example after you refactor a resource into a module or rename its local name — so OpenTofu keeps managing the same real object instead of destroying and recreating it. It edits state only, not your config, so you change the HCL and then move the state entry to match. The declarative alternative for refactors is the `moved` block, which records the rename in code.

Q: What is the `moved` block and how does it differ from `tofu state mv`?
A: A `moved` block (introduced in Terraform 1.1, supported by OpenTofu) records in code that a resource's address changed — `moved { from = ... to = ... }` — so the refactor is applied automatically and reviewably during plan/apply. Unlike the imperative `tofu state mv`, it lives in version control, is visible in code review, and runs consistently for everyone. It prevents the destroy/recreate that a bare rename would otherwise cause.

Q: What does `tofu state rm` do, and what is the danger?
A: `tofu state rm <address>` removes a resource from state so OpenTofu forgets it manages that object — without touching the real infrastructure, which keeps existing. The danger is that the object is now unmanaged (orphaned): future plans will not see it, and re-adding it requires an import. Use it deliberately, for example when handing a resource off to another stack, not as a way to "fix" a confusing plan.

Q: What is the `-target` flag on plan/apply, and why is it a last resort?
A: `-target=resource.address` restricts the operation to the named resource and its dependencies, applying only part of the configuration. It is a break-glass tool for recovering from a broken state or a partial apply, not normal workflow, because it produces a state that does not reflect the full configuration and can mask broader problems. The plan even warns that targeting should be exceptional; over-reliance on it hides drift.

Q: What does `terraform { backend "azurerm" { ... } }` configure, and what cannot you do inside it?
A: It configures the remote backend where state is stored — for `azurerm`, a storage account, container, and blob key. A key limitation is that the backend block cannot use input variables or most expressions; its values must be literals or supplied via `-backend-config` at `init` time. This is why backend settings are often passed as a separate `-backend-config` file per environment rather than templated with `var.`.

Q: Why can't the backend configuration reference `var.` values, and how do you parameterise it per environment?
A: The backend is initialised before variables are evaluated, so it must be statically resolvable and cannot reference `var.` or most expressions. You parameterise it by passing a partial backend config and supplying the rest with `tofu init -backend-config=prod.backend.hcl`, choosing the file per environment. This is the canonical way to point dev and prod at different state locations without templating the block.

Q: What does the dependency lock file `.terraform.lock.hcl` record, and should it be committed?
A: It records the exact provider versions selected and their checksums (hashes) so every machine resolves identical providers, protecting against tampering and version drift. Yes — it should be committed to version control so the whole team and CI use the same locked providers. You update it deliberately with `tofu init -upgrade`, which is itself a reviewable change.

Q: What does `tofu init -upgrade` do, and why is it a deliberate action?
A: `tofu init -upgrade` re-resolves providers and modules to the newest versions allowed by your constraints and rewrites the lock file accordingly. It is deliberate because it can change planned behaviour, so it should be a reviewed change (a PR that updates `.terraform.lock.hcl`), not something done silently in CI on every run. Pinning plus deliberate upgrades is what keeps plans reproducible.

Q: How do you make CI runs reproducible with respect to providers and avoid surprise upgrades?
A: Commit the lock file, pin provider versions in `required_providers`, and run `tofu init` without `-upgrade` in CI so it installs exactly the locked versions. Optionally cache the `.terraform` directory or use a provider mirror for speed and supply-chain control. The result is that the same code plus the same lock file produces the same providers every run, which is what reproducibility demands.

Q: A `tofu apply` fails midway leaving a partially-created environment — what state are you in and how do you recover?
A: You are in a partial-apply state: some resources were created and recorded in state, others were not, and OpenTofu's state reflects whatever completed before the failure. Recover by reading the error, fixing the root cause (quota, permission, dependency), then re-running plan/apply — OpenTofu is idempotent and will create only what is still missing. Avoid manual portal edits, which add drift on top of the problem.

Q: A `tofu apply` errors with "Error acquiring the state lock" — what does it mean and how do you proceed?
A: It means another operation holds the state lock, or a previous run crashed without releasing it. First verify no apply is genuinely running (check CI and teammates); if the lock is stale, release it with `tofu force-unlock <LOCK_ID>` using the id shown in the error. Never force-unlock while a real apply is in progress, as that reintroduces the corruption the lock was preventing.

Q: A plan shows a destroy on a resource you never edited — what are the likely causes to investigate?
A: Likely causes are that the `resource` block was removed or renamed (so OpenTofu thinks it should be deleted), a `count`/`for_each` index shifted, a module source changed, or someone deleted it out of band. Investigate by checking recent diffs to the config, the resource's address in `tofu state list`, and whether a rename should have used a `moved` block. The fix depends on the cause: restore the block, add a `moved` block, or import if it was recreated manually.

Q: What does it mean that OpenTofu is idempotent, and why does that property matter?
A: Idempotent means applying the same configuration repeatedly converges on the same end state — once reality matches the config, further applies make no changes (a no-op plan). It matters because it makes re-runs safe after failures and makes "no changes" a meaningful signal that reality equals intent. This is the foundation of drift detection and of trusting that `apply` does only what `plan` showed.

Q: Why is `apply` gated behind approval while `plan` is allowed freely in a regulated pipeline?
A: `plan` is read-only — it inspects and proposes without changing infrastructure — so it is safe to run on every PR and gives reviewers the diff. `apply` mutates real infrastructure and so must pass human approval to satisfy four-eyes and segregation-of-duties controls. Gating apply, and applying a saved reviewed plan file, ensures only approved changes reach the regulated estate.

Q: How does IaC strengthen disaster-recovery posture compared with manually-built infrastructure?
A: With IaC the entire environment is described in version-controlled code, so a DR rebuild is `tofu apply` against the same modules rather than a manual reconstruction from tribal knowledge. This makes recovery faster, testable (you can rebuild a copy to verify), and consistent with the original, which manual builds cannot guarantee. For regulated funds it turns "can we rebuild from scratch?" from a hope into a demonstrable, time-bounded procedure.

Q: How does code-based IaC support the four-eyes / segregation-of-duties controls regulators expect for infrastructure change?
A: Because every change is a reviewable pull request whose `tofu plan` is the proposed diff, approval is a distinct person signing off before a gated `apply` executes. This separates the author of a change from its approver and produces a durable, version-controlled record of who changed what and why. Portal clicks provide none of this, which is why "architecture not in code" fails regulated review.

Q: Why does the lesson say "architecture that isn't in code is a slide"?
A: Because a design that lives only in diagrams or portal clicks cannot be reproduced, reviewed, version-controlled, or audited — it is aspirational rather than operational. IaC is the medium in which a platform actually ships, so the architect's design must land as modules, plans, and applies. Without it there is no reproducibility guarantee and no audit trail, exactly the things a regulated estate requires.

Q: How does OpenTofu handle a provider's eventual-consistency where a resource exists in the API but a dependent read returns "not found"?
A: OpenTofu's dependency graph creates the resource before its dependents, but some providers add explicit waits because cloud APIs are eventually consistent; if a read races ahead you may see transient "not found" errors. The remedy is usually to express the real dependency so ordering is correct, or rely on the provider's built-in retries, and re-run apply since the operation is idempotent. Adding gratuitous `depends_on` rarely fixes true eventual-consistency races and can mask the real issue.

Q: When composing modules, why prefer passing explicit inputs/outputs over having a child module read remote state directly?
A: Passing explicit inputs and consuming outputs keeps the module's interface clear, testable, and decoupled from where the caller's data lives, so the module can be reused in different contexts. Having a child module reach into `terraform_remote_state` itself hardwires it to a specific external stack and hides the dependency. Explicit wiring makes the dependency visible in code review and easier to change.

Q: What is a common reason a `for_each` errors with "Invalid for_each argument" and how do you fix it?
A: It typically means the value passed to `for_each` is not known until apply (it depends on a resource attribute computed at apply time), and `for_each` keys must be known at plan time. Fix it by keying on values known at plan time — static maps, variables, or `for_each` over a set derived from inputs rather than from yet-to-be-created resource attributes. If you truly cannot know the keys early, restructure so the collection comes from inputs, not from apply-time outputs.

Q: Why should you tag resources (for example with environment, owner, and cost-centre) in IaC, especially in a fund estate?
A: Tags drive cost allocation, ownership, lifecycle, and regulatory inventory, and putting them in code (often via a shared `local.tags` merged into every resource) makes them consistent and enforceable. In a fund estate, tagging by fund, environment, and data-classification supports cost reporting and the audit requirement to know what holds regulated data. Centralising tags in a local or module input prevents the drift and gaps that manual tagging produces.

Q: How would you prove to an auditor that no secret value is stored in clear in your OpenTofu state?
A: Show that secrets are referenced from a vault (Key Vault/OpenBao data sources) rather than hardcoded, that state encryption is configured with `enforced = true` so unencrypted state cannot be written, and demonstrate the stored state object is ciphertext. You can also inspect `tofu state show` to confirm no plaintext secret attributes are exposed and document the key provider and rotation. The combination of vault references, sensitive marking, and enforced encryption is the auditable control.

Q: How do you capture a plan as a durable review artifact for change governance?
A: Generate the plan with `tofu plan -out=plan.tfplan`, render a human-readable version (and JSON via `tofu show -json plan.tfplan` if you need machine processing), and attach both to the change record or PR. The saved binary plan is then applied verbatim with `tofu apply plan.tfplan`, proving what was approved is what ran. Storing the plan output alongside the approval gives auditors the before/after diff they want.

Q: What is the difference between `tofu show` against state versus against a saved plan file?
A: `tofu show` with no argument renders the current state (what is managed now), while `tofu show plan.tfplan` renders the saved plan's proposed changes. Adding `-json` produces machine-readable output for either, useful for policy checks or posting diffs in CI. Knowing which you are showing avoids confusing "current reality" with "proposed change" when reviewing.

Q: Where does policy-as-code (for example Sentinel or OPA/Conftest) fit in the plan/apply pipeline?
A: Policy-as-code runs against the plan — typically the JSON plan — between `plan` and `apply`, automatically enforcing rules like "no public storage" or "must be tagged with cost-centre" before any change is approved. Sentinel is HashiCorp's enterprise option; OPA/Conftest is an open alternative usable with OpenTofu. It turns governance into an automatic gate rather than a manual checklist, which scales four-eyes controls.

Q: At a build-vs-buy level, what do HCP Terraform / Terraform Enterprise add over a self-hosted OpenTofu setup?
A: They provide a managed state backend, run governance (remote, gated runs with approvals), policy-as-code via Sentinel, a private module registry, and access controls out of the box. With self-hosted OpenTofu you assemble these from a remote backend, your CI system, OPA/Conftest, and a private registry yourself. The build-vs-buy question is whether the platform's convenience and governance features justify the cost and the BSL-licensed dependency versus an open self-managed stack.

Q: In a Luxembourg Azure-only fund-tech shop that mandates Bicep, where does OpenTofu/Terraform still matter for you as an architect?
A: Even where Bicep is mandated for Azure-native deployment, you need OpenTofu/Terraform fluency to read multi-cloud or FOSS-stack code, to argue the Bicep-vs-Terraform trade-off credibly, and to manage anything outside Azure (Kubernetes, SaaS providers, Git, DNS). The state model, modules, and provider ecosystem concepts also transfer to evaluating any IaC tool. Holding a reasoned position requires understanding both camps rather than defaulting to whichever the incumbent chose.

Q: Putting it together: walk through the end-to-end lifecycle of a single reviewed infrastructure change in a regulated OpenTofu pipeline.
A: An engineer edits HCL on a branch, runs `fmt`/`validate` locally, opens a PR where CI runs `tofu plan -out` and posts the diff; policy-as-code checks the plan and a separate reviewer approves it for four-eyes. On merge, a gated job runs `tofu apply plan.tfplan` against encrypted, locked remote state, and the saved plan plus approval become the audit record. Scheduled `tofu plan -refresh-only -detailed-exitcode` then watches for drift, closing the loop between intended and actual infrastructure.
