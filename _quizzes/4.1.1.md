# Phase 3 · 4.1.1 Distributed batch compute (Spark) — 150 self-test questions

Q: What is the role of the Spark driver in a cluster application?
A: The driver is the process that runs your `main()` and holds the `SparkSession`; it builds the logical and physical plans, schedules work by breaking the job into stages and tasks, and tracks their progress. It is the single coordination point — if the driver dies the whole application dies — so it must reach the cluster manager and every executor. A common gotcha is collecting a huge result with `df.collect()`, which pulls all data back to the driver's JVM and can OOM it.

Q: What is a Spark executor and what two things does it provide?
A: An executor is a JVM process launched on a worker node that does the actual computation: it runs the tasks the driver assigns and provides memory for caching/shuffle data. Each executor has a fixed number of cores (task slots) and a heap, so total cluster parallelism is roughly `num_executors × cores_per_executor`. Sizing these wrong — too few fat executors or too many tiny ones — is one of the most common causes of slow or wasteful Spark jobs.

Q: Define the hierarchy jobs → stages → tasks in Spark.
A: One Spark action (e.g. `count()`, `write()`) triggers a job; each job is split into stages at shuffle boundaries; each stage is a set of tasks, one per partition, that run the same code on different data. Tasks are the unit of parallel execution scheduled onto executor cores. Understanding this chain is the basis for reading the Spark UI: you trace a slow job to its slow stage, then to the straggler task.

Q: What exactly creates a stage boundary in Spark?
A: A stage boundary is created by a wide transformation that requires a shuffle — data must be repartitioned across the network so rows with the same key land on the same task. Narrow transformations (`map`, `filter`, `select`) stay within a stage because each output partition depends on one input partition. So every `groupBy`, `join`, `distinct`, or `repartition` typically introduces a new stage.

Q: Why is the shuffle described as "where the money goes" in Spark?
A: The shuffle writes intermediate data to local disk on the map side, transfers it across the network, and re-reads it on the reduce side, so it costs disk I/O, network, and serialization — the most expensive operations in a distributed job. Most slow and over-billed Spark jobs are dominated by one or two heavy shuffles, not by CPU. For a NAV pipeline, minimizing and balancing shuffles (avoiding needless `repartition`, broadcasting small dimensions) is usually the biggest single saving.

Q: What is a task in Spark and how many tasks does a stage have?
A: A task is the smallest unit of work: it processes exactly one partition of the data within a stage and runs in one executor core slot. A stage has as many tasks as the stage's input has partitions, so a 200-partition shuffle produces a 200-task stage. This is why partition count directly controls parallelism and why "stuck at task 199 of 200" points to a single straggler partition.

Q: What is the difference between a wide and a narrow transformation?
A: A narrow transformation produces output partitions that each depend on a single input partition (`map`, `filter`, `union`), so no data moves between nodes. A wide transformation produces output partitions that depend on many input partitions (`groupBy`, `join`, `repartition`), forcing a shuffle. Narrow transformations are pipelined within a stage; wide ones cut the job into separate stages.

Q: In the Spark architecture, what is the cluster manager's job?
A: The cluster manager (YARN, Kubernetes, Spark standalone, or a vendor's like Databricks) allocates executor resources to the application — it grants the driver a set of executors with cores and memory. Spark itself is agnostic to which manager you use; the same job runs on any of them. The manager handles physical resource scheduling, while Spark's driver handles task scheduling within the granted resources.

Q: What does it mean that the DataFrame API is "lazy"?
A: Transformations like `select`, `filter`, and `join` do not execute; they build up an unresolved logical plan describing the computation. Nothing runs until you call an action (`count`, `show`, `collect`, `write`), at which point Spark optimizes and executes the whole plan. This laziness lets Catalyst optimize across many operations at once, but it also means errors and timings surface at the action, not at the line where you wrote the transformation.

Q: Name three Spark actions and three transformations and explain the difference.
A: Actions include `count()`, `collect()`, `show()`, `write()`, and `take()` — they trigger execution and return a value or write data. Transformations include `select()`, `filter()`, `withColumn()`, `join()`, and `groupBy()` — they are lazy and just extend the plan. The rule of thumb: if it returns a new DataFrame it's a transformation; if it returns a non-DataFrame result or side effect it's an action.

Q: A junior says their `df.filter(...)` line is slow but the program flew past it instantly — what's happening?
A: Because transformations are lazy, the `filter` line only added to the plan and did no work, so it returned instantly; the time is actually spent at the next action that forces execution. To find the real cost you look at the action's job in the Spark UI, not the source line of the transformation. This lazy-evaluation gotcha is the single most common confusion for engineers coming from pandas.

Q: What is the difference between the logical plan and the physical plan?
A: The logical plan is the abstract sequence of operations Catalyst derives from your DataFrame code, independent of how it runs; the physical plan is the concrete execution strategy — which join algorithm, which exchange (shuffle), which scan. Catalyst rewrites the logical plan with rule- and cost-based optimization, then selects a physical plan. You can inspect both with `df.explain(True)` to see the analyzed, optimized, and physical stages.

Q: What does `df.explain()` show and why is it the first tool for plan debugging?
A: `df.explain()` prints the physical plan; `df.explain(True)` (or `explain(mode="extended")`) prints parsed, analyzed, optimized logical plans plus the physical plan. It reveals the chosen join strategy (`BroadcastHashJoin` vs `SortMergeJoin`), exchanges, and filter pushdown — before you run anything. It is the cheapest way to confirm Spark is doing what you intended without paying for a full execution.

Q: What is the Catalyst optimizer?
A: Catalyst is Spark SQL's query optimizer that transforms the logical plan through rule-based rewrites (predicate pushdown, constant folding, projection pruning) and cost-based decisions, then generates an efficient physical plan. It is what lets the DataFrame and SQL APIs produce the same optimized execution. Because it optimizes the whole plan from one action, writing idiomatic DataFrame code usually beats hand-tuned RDD code.

Q: Give two concrete rewrites Catalyst performs automatically.
A: Predicate pushdown moves filters as close to the data source as possible — e.g. pushing a `WHERE fund_id = 'X'` down into a Parquet/Iceberg scan so unmatched row groups are never read. Projection (column) pruning drops columns you never reference so only needed columns are read. Both reduce I/O before any compute happens, which is why selecting only the columns you need and filtering early still matters even though Catalyst helps.

Q: What is Adaptive Query Execution (AQE) and what changed in its default?
A: AQE re-optimizes the physical plan at runtime using actual statistics gathered from completed shuffle stages, rather than relying only on compile-time estimates. It is controlled by `spark.sql.adaptive.enabled`, which has defaulted to `true` since Spark 3.2. This matters because AQE can fix planner mistakes — like a join planned as sort-merge that should have been a broadcast — that static optimization could not catch.

Q: Name the three main optimizations AQE applies at runtime.
A: AQE (1) coalesces small post-shuffle partitions to avoid thousands of tiny tasks, (2) converts a sort-merge join to a broadcast hash join when runtime stats show one side is small enough, and (3) splits skewed partitions into sub-partitions to defuse straggler tasks. All three rely on real shuffle statistics that only exist at runtime. They reduce the manual tuning you'd otherwise do with shuffle-partition counts and broadcast hints.

Q: What does AQE's coalescing of post-shuffle partitions do, and which config drives the target size?
A: After a shuffle, AQE merges adjacent small partitions so each task processes a sensible amount of data instead of many near-empty tasks, cutting scheduling overhead. The target is governed by `spark.sql.adaptive.advisoryPartitionSizeInBytes`, which defaults to 64 MB. This is why with AQE you can leave `spark.sql.shuffle.partitions` high and let AQE coalesce down rather than hand-tuning it per job.

Q: With AQE enabled, do you still need to set `spark.sql.shuffle.partitions` carefully?
A: Far less so — AQE's coalescing uses your shuffle-partition count as an upper bound and merges down to the advisory size at runtime, so a generously high value (the default is 200) is usually fine. You set it high to give AQE room, not low. The old practice of hand-tuning it per job for partition skew is largely replaced by AQE on Spark 3.2+.

Q: What is a partition in Spark and why does partition count matter?
A: A partition is a chunk of the dataset that one task processes; the number of partitions sets the maximum parallelism of a stage. Too few partitions underuses the cluster and risks OOM on large tasks; too many creates scheduling overhead and tiny tasks. The goal is partitions sized so each task handles a balanced, reasonable amount of data — neither a 4 GB straggler nor a million 1-row tasks.

Q: What is data skew in the context of a shuffle?
A: Skew is when shuffle keys are unevenly distributed, so one or a few partitions receive far more rows than the rest. The stage can't finish until that oversized partition's task finishes, so the whole job waits on a single straggler. In fund data this is classic: one mega-fund or one hot `fund_id` holds most of the positions, so its partition dwarfs the others.

Q: Why does a 200-task stage "stall at task 199 of 200"?
A: Because 199 tasks got balanced partitions and finished quickly, but the 200th got a skewed, oversized partition — a hot key — and runs far longer, holding up the whole stage. The symptom in the UI is a long task-duration tail: max far above median. This is the signature of skew, and the fix is to break up the hot key, not to add more nodes.

Q: What is salting and how does it fix a skewed join?
A: Salting appends a random suffix (a "salt") to the hot key so its rows spread across many partitions instead of one; the other side of the join is replicated across the same salt values so matches still occur. This converts one giant straggler partition into several balanced ones. The cost is extra rows on the replicated side and added complexity, so you salt only the keys that are actually hot.

Q: Walk through salting a positions × fund-reference join where one `fund_id` is hot.
A: On the skewed `positions` side, add a column `salt = floor(rand() * N)` and key on `(fund_id, salt)`; on the small `fund_ref` side, explode each row into N copies, one per salt value, keyed the same way. The join now spreads the hot fund's rows across N tasks instead of stalling on one. Choose N roughly proportional to how badly that fund dominates, and drop the salt column after the join.

Q: How does AQE's automatic skew-join handling work, and which two thresholds define a "skewed" partition?
A: AQE detects a partition as skewed when its size exceeds both `skewedPartitionFactor × the median partition size` (factor default 5.0) and `skewedPartitionThresholdInBytes` (default 256 MB), then splits that partition into smaller sub-partitions handled by separate tasks. It is enabled by `spark.sql.adaptive.skewJoin.enabled` (default `true`). It often removes the need for manual salting, but only for sort-merge joins and only above those thresholds.

Q: When does manual salting still beat AQE skew-join handling?
A: When the skew is below AQE's 256 MB threshold but still causes a meaningful straggler, when the join isn't a plain sort-merge join AQE can split, or when you need deterministic control in a regulated pipeline rather than runtime heuristics. AQE is the first thing to try, but it has thresholds and join-type limits. Knowing both lets you say in a review exactly why AQE did or didn't rescue a given job.

Q: What is the broadcast hash join strategy?
A: In a broadcast hash join Spark ships the entire small table to every executor, building a local hash table, so the large table is joined with no shuffle of itself. It is the fastest join when one side is small because it eliminates the expensive shuffle of the big side. The risk is broadcasting something too large, which OOMs executors or the driver during the broadcast.

Q: What is the sort-merge join and when is it used?
A: Sort-merge join shuffles both sides on the join key, sorts each partition, then merges them; it is Spark's default for joining two large tables. It scales to big-vs-big joins that can't be broadcast, but it pays for two shuffles plus sorting. Most heavy, slow joins in a Spark NAV pipeline are sort-merge joins, and they're where skew bites.

Q: What is `spark.sql.autoBroadcastJoinThreshold` and its default?
A: It is the size threshold below which Spark automatically broadcasts a table in a join rather than using sort-merge; the default is `10485760` bytes (10 MB). Spark uses table size statistics to decide. Setting it to `-1` disables auto-broadcast entirely, which you might do to stop Spark broadcasting something whose size it underestimated and OOMs on.

Q: How do you force or prevent a broadcast join explicitly?
A: Wrap the small side in `broadcast(df)` (from `pyspark.sql.functions`) or use the SQL hint `/*+ BROADCAST(t) */` to force it; raise or lower `spark.sql.autoBroadcastJoinThreshold` to change the automatic behavior, or set it to `-1` to disable auto-broadcast. Forcing is useful when Spark's size estimate is wrong — for instance a small fund-reference table whose stats are stale after a write.

Q: When does the planner pick the wrong join and broadcast a table that's too big?
A: When table statistics are missing or stale, Spark may underestimate a side's size and broadcast it, causing executor OOM or a driver bottleneck while it collects the broadcast. This often happens after writing a table without refreshing stats, or with compressed columnar files whose on-disk size hides a large in-memory footprint. The fix is `ANALYZE TABLE`/refreshed stats, an explicit threshold, or disabling auto-broadcast for that query.

Q: What is the difference between `repartition()` and `coalesce()`?
A: `repartition(n)` does a full shuffle to produce exactly `n` evenly-sized partitions and can increase or decrease partition count; `coalesce(n)` only reduces partitions by merging existing ones without a full shuffle. Use `repartition` to fix skew or increase parallelism, and `coalesce` to cheaply cut down many small partitions before a write. A gotcha: `coalesce` to a very small number can starve upstream parallelism because it pulls the narrow-dependency boundary back.

Q: Why might `repartition(col)` before a write be useful for an Iceberg/Parquet output?
A: Repartitioning by the column you partition the table on (e.g. `valuation_date`) groups rows so each output partition writes to one table partition, avoiding many small files per partition directory. This reduces the small-file problem that slows later reads and metadata. The tradeoff is the extra shuffle, so you balance file-size health against shuffle cost.

Q: What are the main regions of Spark executor memory?
A: Unified memory is split between execution memory (for shuffles, joins, sorts, aggregations) and storage memory (for cached data), which share one region and can borrow from each other; outside that sits user memory for your data structures and a reserved portion. The split between execution and storage is dynamic. Understanding it explains why heavy caching can squeeze out shuffle memory and trigger spill.

Q: What is "spill" in Spark and is it always bad?
A: Spill is when a task's execution data (e.g. a sort or aggregation buffer) exceeds available execution memory and Spark writes it to local disk, then reads it back. It is a safety valve that prevents OOM, so it's not catastrophic, but heavy spill means slow disk I/O dominating the stage. In the UI, large "spill (memory)" and "spill (disk)" numbers on a stage are a sign to give the job more memory or reduce partition size.

Q: Distinguish `cache()` from `persist()` and state the default storage level for a DataFrame.
A: `cache()` is shorthand for `persist()` with the default storage level; `persist(StorageLevel...)` lets you choose the level explicitly. For a DataFrame/Dataset the default is `MEMORY_AND_DISK` (not `MEMORY_ONLY` as it is for raw RDDs), because recomputing the columnar cache is expensive so Spark prefers to spill it to disk rather than drop it. This is a frequently-missed distinction in interviews and tuning.

Q: List the main persistence storage levels and what they trade off.
A: `MEMORY_ONLY` keeps deserialized objects in heap (fast, but recomputes if it doesn't fit); `MEMORY_AND_DISK` spills overflow to disk instead of recomputing; the `_SER` variants store serialized bytes (less memory, more CPU); `DISK_ONLY` keeps everything on disk; `OFF_HEAP` stores serialized data off-heap. The `_2` suffix replicates each partition on two nodes for fault tolerance. You trade memory footprint against CPU and recompute risk.

Q: When is caching a DataFrame actually worthwhile?
A: Cache only when the same DataFrame is reused by multiple actions or branches and recomputing it is expensive — otherwise the cache overhead is pure waste, and a single-action job never benefits. Caching also consumes storage memory that competes with execution memory, so over-caching can cause spill or slow shuffles. A common gotcha is caching then immediately using the data once; that's slower than not caching.

Q: A Spark job dies with `java.lang.OutOfMemoryError` — how do you debug "from the error upward"?
A: First read which JVM died: driver OOM usually means a large `collect()`/broadcast pulled too much back, while executor OOM points to oversized partitions, skew, or too much cached data squeezing execution memory. Check the failed stage's task sizes and spill in the UI, look for a skewed partition or an over-large broadcast, then fix the cause (repartition, raise memory, lower broadcast threshold) rather than blindly increasing `--executor-memory`. The error's stack and the UI together localize whether it's data volume, skew, or a broadcast.

Q: Driver OOM vs executor OOM — what distinguishes their causes?
A: Driver OOM typically comes from pulling data to the driver — `collect()`, `toPandas()`, very large broadcast tables, or huge query plans — because the driver's heap is small and not meant to hold data. Executor OOM comes from data-side pressure: a skewed partition too big to process, excessive caching, or wide aggregations exceeding execution memory. The fix differs: shrink what you bring to the driver vs rebalance/resize executor work.

Q: What does the Spark UI "Stages" tab tell you about a slow stage?
A: It shows each stage's task count, duration distribution (min/median/max and percentiles), shuffle read/write sizes, and spill, so you can see at a glance whether tasks are balanced or one straggler dominates. A max task time far above the median signals skew; large shuffle read on one task confirms a hot partition. This is the core screen for the UI → stages → shuffle/skew → fix workflow.

Q: In the Spark UI, what evidence proves a skewed join (not "Spark is slow")?
A: You point to the offending stage's task summary where one task's shuffle-read size and duration are many times the median while the rest finished early — a long tail with a single outlier. The stage timeline shows most tasks done and the job waiting on that one. That concrete max-vs-median gap and shuffle-read size is the evidence an architect cites instead of a vague "it's slow".

Q: What is the SQL tab in the Spark UI good for?
A: The SQL/DataFrame tab shows each query's plan as a DAG with per-node metrics — rows, time, and which physical operators ran (e.g. `BroadcastHashJoin`, `Exchange`, `SortMergeJoin`) — so you confirm the chosen join strategy and where time went. It links the plan to actual runtime numbers, complementing `explain()` which only shows the plan. It's where you verify AQE actually converted a join or split a skew partition.

Q: How do you write to an Iceberg table from Spark, and what must be configured first?
A: You configure an Iceberg catalog via `spark.sql.catalog.<name>` properties (the implementation class and a catalog `type` such as `hive`, `hadoop`, or `rest`), then write with SQL (`INSERT INTO`, `MERGE INTO`) or the DataFrame `writeTo(...)` API. The catalog tells Spark where table metadata lives. Without the catalog configured, Spark can't resolve the Iceberg table and you get a table-not-found error.

Q: What is the difference between Iceberg's `SparkCatalog` and `SparkSessionCatalog`?
A: `SparkCatalog` exposes Iceberg tables under a dedicated catalog name you reference explicitly (e.g. `my_catalog.db.table`); `SparkSessionCatalog` wraps Spark's built-in session catalog so Iceberg tables coexist with non-Iceberg tables under the default `spark_catalog`. Use `SparkCatalog` for a clean Iceberg-only namespace and `SparkSessionCatalog` when you need Iceberg alongside existing Hive tables. Both are set through `spark.sql.catalog.*` configs.

Q: What does `MERGE INTO` do in Iceberg from Spark, and why is it central to a NAV pipeline?
A: `MERGE INTO target USING source ON <cond> WHEN MATCHED THEN UPDATE ... WHEN NOT MATCHED THEN INSERT ...` performs an atomic upsert, applying inserts, updates, and deletes in one transactional commit. For NAV data this is how you apply late-arriving corrections or restated positions without rewriting the whole table. It replaces fragile delete-then-insert patterns and gives you a clean, auditable snapshot per run.

Q: What is write-audit-publish (WAP) and which Iceberg/Spark setting enables it?
A: WAP writes new data to a staged branch, audits/validates it, then publishes (fast-forwards the main branch) only if checks pass, so readers never see unvalidated data. In Iceberg you set `spark.wap.branch` (or `spark.wap.id`) so Spark writes commit to a branch instead of `main`. This pattern is the regulated-pipeline answer to "how do you guarantee a bad NAV never reaches consumers" and is the hook this lesson sets up for later.

Q: How does `spark.wap.branch` change where a Spark write lands?
A: When `spark.wap.branch` is set to a branch name, Spark routes the table's writes to that Iceberg branch rather than the table's main line, creating staged snapshots you can query and validate in isolation. After your audit passes you publish by fast-forwarding `main` to that branch. This decouples "data written" from "data visible", which is exactly what an audited NAV publish needs.

Q: What does INSERT OVERWRITE's dynamic vs static partition behavior mean in Iceberg from Spark?
A: Dynamic overwrite replaces only the partitions present in the incoming data, leaving other partitions untouched — ideal for reprocessing a single `valuation_date`. Static overwrite replaces the whole table (or an explicitly named partition predicate). Choosing the wrong one can wipe partitions you meant to keep, so for daily NAV reloads you almost always want dynamic, partition-scoped overwrite.

Q: What are the four main Spark deployment modes you should compare as an architect?
A: Local mode (single JVM, for dev/test), standalone (Spark's own simple cluster manager), Kubernetes (executors as pods, the modern container-native option), and managed platforms like Databricks (Spark plus a control plane, autoscaling, and proprietary engines). They differ operationally in resource management, autoscaling, isolation, and cost model — not in your DataFrame code, which stays the same. The architect's job is picking the mode whose operations and bill fit the team.

Q: What is the difference between client and cluster deploy mode?
A: In client mode the driver runs on the machine that submitted the job (e.g. your laptop or an edge node), convenient for interactive work but tying the job to that machine's lifetime and network; in cluster mode the driver runs inside the cluster, which is more robust for production batch jobs. The choice affects where logs appear and what happens if the submitting machine disconnects. Production scheduled NAV jobs typically run cluster mode.

Q: Why does local mode matter even though production is a cluster?
A: Local mode runs the full Spark engine in one JVM with no cluster manager, so you can develop, unit-test, and reproduce plan/optimizer behavior on a laptop — the same DataFrame code and `explain()` output as production. It's the entire lab for this lesson before you scale up. The caveat: local mode hides shuffle-network and executor-sizing problems, so some issues only appear on a real cluster.

Q: What operationally changes when you move a Spark job from standalone to Kubernetes?
A: On Kubernetes executors become pods scheduled by the K8s API server, so you gain container isolation, dynamic allocation, and integration with cluster autoscaling and namespaces/quotas; you also inherit K8s concerns like image management, RBAC, and pod scheduling. The Spark code is unchanged. The operational model shifts from Spark managing workers to Kubernetes managing pods, which is why many firms standardize on it.

Q: What does Databricks add on top of open-source Spark that an architect should know to challenge a quote?
A: Databricks bundles managed Spark with autoscaling clusters, a notebook/workspace control plane, Unity Catalog governance, and the proprietary Photon vectorized engine; pricing is by DBU on top of cloud compute. To challenge a "just add nodes" or vendor quote you need to know cluster types, Photon's where-it-helps, and the DBU multiplier so you can compare against self-managed Spark. The lock-in and cost model, not raw Spark capability, are the decision levers.

Q: What is Photon and when does it help?
A: Photon is Databricks' C++ vectorized query engine that accelerates SQL and DataFrame workloads by processing columnar batches with SIMD, helping most on scan-, filter-, join-, and aggregation-heavy analytical queries. It does not accelerate arbitrary Python UDFs and carries a higher DBU rate. So the architect's question on a quote is whether the workload is the kind Photon speeds up enough to justify its premium.

Q: A nightly NAV Spark job takes four hours when it could take twenty minutes — what is your diagnostic order?
A: Go UI → find the dominant job → its slowest stage → inspect the stage's task distribution and shuffle for skew (max ≫ median) or excessive shuffle/spill → then apply the targeted fix (broadcast a small dimension, salt or let AQE split the hot key, fix partitioning). You quantify before/after with stage timings. The point is a methodical UI-driven diagnosis, never "add nodes" without evidence.

Q: Why is "just add more nodes" usually the wrong answer to a skewed Spark job?
A: Skew means one task is overloaded; adding executors gives more idle slots but the single straggler partition still runs on one task, so wall-clock barely improves while the bill rises. You must rebalance the data (salt, AQE skew split, broadcast) so work spreads across tasks. Recognizing this lets an architect push back on a vendor's reflexive scale-up recommendation.

Q: When is DuckDB on one node the right tool instead of a Spark cluster?
A: When the data fits comfortably on a single machine (roughly tens of GB, well under ~100 GB) and the workload is a self-contained query, a single-node columnar engine like DuckDB avoids cluster startup, shuffle-network, and scheduling overhead entirely and often wins on wall-clock and cost. Spark's distribution only pays off when data genuinely exceeds one node or you need its ecosystem. The rule of thumb: don't pay the distributed tax for single-node-size data.

Q: What concrete numbers would you cite to argue Spark is the wrong tool for a given job?
A: You compare your own runs: DuckDB wall-clock and peak RAM on one node versus the Spark job's wall-clock plus cluster-minutes (and therefore cost), for the same NAV computation on the same data. If DuckDB finishes faster and cheaper because the data fits in memory and there's no shuffle benefit, that's the evidence. Architects win these arguments with measured numbers from their own benchmark, not assertions.

Q: What is a broadcast variable's lifecycle risk in a long-running job?
A: A broadcast (whether an explicit broadcast variable or a broadcast-join table) is held in executor memory for the life of its use, so broadcasting a too-large or growing table can accumulate memory pressure and contribute to OOM. If a "small" reference table grows past the threshold over time, a job that worked yesterday can OOM today. Re-checking broadcast sizes and stats is part of operating the pipeline, not a one-time setup.

Q: What does `spark.sql.shuffle.partitions` control and what is its default?
A: It sets the number of partitions produced by shuffles in Spark SQL/DataFrame operations (joins, aggregations); the default is 200. Too low underuses a big cluster; too high makes tiny tasks. With AQE on, you can leave it high and let coalescing right-size partitions, so it's now more of a ceiling than a precise knob.

Q: Why can the same Spark code give different physical plans on two clusters?
A: Physical plans depend on runtime configuration and statistics: `autoBroadcastJoinThreshold`, AQE settings, available memory, table stats, and Spark version all influence join choice, partitioning, and AQE rewrites. So a join that broadcasts on one cluster may sort-merge on another with a lower threshold or stale stats. This is why you debug with that environment's `explain()` and UI, not assumptions.

Q: What is predicate pushdown into an Iceberg/Parquet scan and why does it save money?
A: Predicate pushdown sends your filter (e.g. `WHERE valuation_date = '2026-06-12'`) down to the file format and table metadata so Spark skips row groups, files, and partitions that can't match, reading far less data from object storage. With Iceberg's partition and column-level stats, entire data files are pruned before any compute. Less I/O from MinIO/S3 is often the biggest single cost lever in a lakehouse Spark job.

Q: What is the difference between partition pruning and predicate pushdown?
A: Partition pruning eliminates whole partition directories/files using the table's partition columns (e.g. skip every date but the one queried); predicate pushdown additionally filters within files using column statistics and the file format's skipping. Pruning is coarse (partition level) and pushdown is fine (row-group/page level). Together they minimize bytes read, which is why partitioning a NAV table by `valuation_date` pays off.

Q: A join produces far more output rows than expected and the job explodes — what's the likely cause?
A: A key that isn't unique on one side (or nulls/duplicates in the join key) produces a cartesian-like fan-out, multiplying rows and inflating shuffle and output. Check key cardinality and for duplicate or null keys before blaming Spark. In fund data this happens when a reference table has multiple rows per `fund_id` (e.g. by share class) and the join condition forgot a column.

Q: Why can nulls in a join key create skew?
A: All null-keyed rows hash to the same partition in a shuffle, so a column with many nulls dumps them onto one task, creating a hot partition exactly like a single hot key. The fix is to filter or specially handle nulls before the join, or use a null-safe strategy. It's a frequently-missed skew source because the "hot key" is null, not a real value.

Q: What does a "straggler task" look like in the stage timeline and what causes it besides skew?
A: A straggler is one task whose bar in the timeline extends well past the others; besides data skew it can come from a slow or failing executor node, disk pressure causing heavy spill, or GC pauses on that JVM. The UI's per-task metrics (GC time, spill, shuffle read) tell you which. So you distinguish data skew (big shuffle read on that task) from infrastructure skew (one bad node) before fixing.

Q: How does Spark achieve fault tolerance for a lost executor mid-job?
A: Spark tracks lineage — the deterministic transformations needed to recompute any partition — so when an executor is lost it reschedules that partition's tasks on another executor and recomputes from the last available shuffle output or source. Cached data with replication (`_2` levels) survives one node loss without recompute. This is why a transient node failure usually retries rather than killing the whole job.

Q: What does it mean that Spark uses lazy lineage rather than materializing every step?
A: Because transformations build a lineage graph and only actions execute it, Spark can recompute any lost partition from its lineage instead of needing intermediate results saved to disk. This is the basis of fault tolerance and of whole-plan optimization. The tradeoff is that without caching, reusing a DataFrame across actions recomputes the lineage each time.

Q: Why might caching a DataFrame that's used once actually slow the job down?
A: Materializing the cache costs memory and time, and if the DataFrame is consumed by only one action you pay that cost without any reuse benefit — and you may evict other useful data or trigger spill. Caching only helps when the same computed result is read multiple times. The gotcha is "cache everything" cargo-culting; cache deliberately where the lineage is reused.

Q: What is `spark.sql.adaptive.advisoryPartitionSizeInBytes` for?
A: It tells AQE the target size for shuffle partitions after coalescing — Spark merges small partitions toward this size so tasks aren't tiny; the default is 64 MB. Raising it produces fewer, larger tasks; lowering it produces more, smaller ones. It's the main knob for tuning AQE's post-shuffle partition sizing without touching `spark.sql.shuffle.partitions`.

Q: Your broadcast join unexpectedly OOMs the executors — first thing to check?
A: Check the actual size of the "small" side versus `spark.sql.autoBroadcastJoinThreshold` and its current table statistics — Spark may have under-estimated it (stale stats, compressed columnar files) and broadcast something too large. The immediate mitigations are to refresh stats, lower the threshold, or set it to `-1` to force a sort-merge join for that query. Confirm in `explain()` which side is being broadcast.

Q: How can stale table statistics cause a bad plan, and how do you refresh them?
A: Catalyst uses size and row-count stats to choose join strategies and broadcast candidates; if stats are missing or stale after a write, it can mis-size a table and pick sort-merge when broadcast was right, or vice versa. You refresh with `ANALYZE TABLE <t> COMPUTE STATISTICS` (and `FOR COLUMNS` for column stats), or rely on AQE which uses runtime stats instead. Keeping stats fresh is part of operating a stable pipeline.

Q: What is the role of the DAG scheduler vs the task scheduler in the driver?
A: The DAG scheduler turns the logical plan into a DAG of stages split at shuffle boundaries and submits stages in dependency order; the task scheduler then assigns each stage's tasks to executor slots and handles retries and locality. Together they implement jobs → stages → tasks. Understanding the split clarifies why a stage waits (its parent shuffle) versus why a task is slow (scheduling/skew).

Q: What is data locality and why does it affect task scheduling?
A: Locality is how close a task runs to its input data — process-local, node-local, rack-local, or any — and Spark prefers scheduling tasks where the data already resides to avoid network transfer. With object storage (S3/MinIO) most reads are "any" locality, so locality matters less than it did with HDFS. It still influences cached-partition reuse and shuffle reads.

Q: Why is `collect()` dangerous and what should you use instead for inspection?
A: `collect()` pulls every row to the driver's JVM, which OOMs the driver on large results since the driver heap is small and not meant to hold data. For inspection use `show(n)` or `take(n)` to bring back a bounded sample, and write full results to storage rather than collecting them. This is a top cause of driver OOM in notebooks.

Q: What does `df.show()` actually trigger under the hood?
A: `show()` is an action that runs a job to compute and return the first rows (default 20), so it forces execution of the whole lazy plan up to a `limit`. Because it's an action, it's also where transformation errors and timings surface. Spark can short-circuit by reading only enough partitions to fill the requested rows, so `show()` is cheaper than `collect()` but still a real execution.

Q: How does columnar storage (Parquet/Iceberg) interact with Spark's optimizer to reduce work?
A: Columnar formats let Spark read only the referenced columns (projection pruning) and skip row groups via min/max statistics (predicate pushdown), so Catalyst's pruning translates directly into less I/O. Iceberg adds table-level metadata for partition and file pruning on top. The result is that selective queries on wide NAV/holdings tables read a small fraction of the bytes.

Q: What's the difference between `union()` and `join()` in terms of shuffle cost?
A: `union()` is a narrow transformation that concatenates partitions with no shuffle (it just stacks rows), while `join()` is wide and shuffles both sides on the key (unless one is broadcast). So unioning daily NAV files is cheap, but joining holdings to reference data is where shuffle cost and skew appear. Knowing which operations shuffle is half of reading a plan.

Q: Why might increasing `--executor-memory` not fix a job's OOM?
A: If the OOM is caused by skew (one partition too big for any reasonable heap) or by a too-large broadcast, more memory per executor just delays the failure or wastes resources, because the problem is data imbalance, not total memory. The real fix is to rebalance the data (salt/AQE) or shrink the broadcast. Throwing memory at a skew problem is a classic anti-pattern.

Q: What does the "shuffle read" metric on a task tell you?
A: Shuffle read is the volume of data a reduce-side task pulled from map-side outputs over the network; a task with shuffle read far above the median is processing an oversized partition — the signature of skew. Comparing shuffle-read across tasks in a stage is how you prove which partition is hot. Large total shuffle read across the stage also flags a join or aggregation you might reduce or broadcast away.

Q: How does `MERGE INTO` give you auditability that delete-then-insert does not?
A: `MERGE INTO` commits inserts, updates, and deletes as one atomic Iceberg snapshot, so the table moves cleanly from one consistent version to the next and you can time-travel to any prior snapshot. Delete-then-insert is two non-atomic steps that can leave the table half-updated and harder to audit. For regulated NAV restatements, the single atomic, snapshotted commit is the audit trail.

Q: What is an Iceberg snapshot and how does it relate to a Spark write?
A: Each successful Spark write to an Iceberg table creates a new immutable snapshot capturing the table state and a summary of the operation; readers see a consistent snapshot and you can time-travel or roll back to earlier ones. This is what makes Iceberg writes atomic and auditable. In WAP, snapshots created on a branch are exactly what you validate before publishing to main.

Q: Why does writing many small files to Iceberg from Spark hurt later reads?
A: Each small file adds metadata and a per-file open/read cost, so a table with thousands of tiny files has slow planning and scan performance and bloated metadata. Spark creates one output file per task per partition, so high task counts plus many table partitions multiply files. The mitigations are repartitioning by the partition column before write and periodic compaction.

Q: What does `writeTo(...).append()` vs `.overwritePartitions()` do for an Iceberg table in the DataFrame API?
A: `append()` adds the DataFrame's rows as a new snapshot without touching existing data; `overwritePartitions()` (dynamic overwrite) replaces only the partitions present in the DataFrame, leaving others intact. For a daily NAV reload you typically `overwritePartitions()` so reprocessing one `valuation_date` doesn't disturb history. Choosing `append()` when you meant to replace causes duplicate rows for that date.

Q: How would you verify, before running, that Spark will broadcast your small fund-reference table?
A: Run `df.explain()` on the join and look for `BroadcastHashJoin` with a `BroadcastExchange` on the reference side; if you instead see `SortMergeJoin`, the table is over the threshold or its stats are missing. You can then force it with `broadcast()` or fix stats. Checking the plan first avoids paying for a full run to discover the wrong join.

Q: What is the difference between `cache()` taking effect lazily vs eagerly?
A: `cache()`/`persist()` only mark the DataFrame for caching; the data is actually materialized the first time an action reads it, so the first action pays the full cost and later ones benefit. If you want to force population immediately you trigger an action like `df.count()` after caching. Forgetting this leads people to think the cache "isn't working" when really nothing has read it yet.

Q: Why can a `groupBy().agg()` on a high-skew key be slow even without a join?
A: Aggregation also shuffles rows by the grouping key, so a dominant key (one mega-fund) sends a huge partition to one reduce task — the same straggler problem as a skewed join. Partial aggregation helps but can't fully fix extreme skew. AQE skew handling and salting the aggregation key are the same tools you'd use for the join case.

Q: What is map-side (partial) aggregation and why does it help?
A: Spark pre-aggregates within each map task before the shuffle (a combiner), so it shuffles partial results instead of every raw row, drastically cutting shuffle volume for `sum`, `count`, `avg`, etc. This is automatic for standard aggregations on the DataFrame API. It's why a `groupBy().sum()` is far cheaper than collecting raw rows, though it can't rescue a single dominant key's final partition.

Q: How do you decide executor count and cores per executor at a high level?
A: Balance parallelism against overhead: enough total cores to run your partitions in a few waves, but not so many tiny executors that JVM/GC and shuffle-connection overhead dominate, nor so few fat ones that one GC pause stalls many cores and memory is wasted. A common starting point is moderate executors of 4–5 cores each. The right answer comes from observing stage parallelism and spill in the UI, then adjusting.

Q: What is "speculative execution" in Spark and when is it useful?
A: Speculative execution launches duplicate copies of slow-running tasks on other executors and takes whichever finishes first, mitigating stragglers caused by a slow/flaky node (not by data skew). It's controlled by `spark.speculation`. It helps with infrastructure stragglers but wastes resources on data-skew stragglers, since a duplicate of an oversized partition is just as slow.

Q: A stage shows huge "spill (disk)" — what does that mean and how do you reduce it?
A: It means execution memory was too small for the task's sort/aggregation/join buffers, so Spark wrote intermediate data to local disk and read it back, slowing the stage. You reduce spill by giving tasks more execution memory (more memory per core, fewer cores, or less caching competing for the unified region) or by making partitions smaller so each task's working set fits. Confirm by watching the spill metrics drop after the change.

Q: Why is the same DataFrame transformation code portable across local, K8s, and Databricks?
A: Spark separates the logical API from execution: your DataFrame/SQL code compiles to the same Catalyst plan regardless of deployment, and only the cluster manager and resource layer differ underneath. This is why the lab runs the identical NAV transformation in local mode and on a cluster. The differences you must manage are operational — sizing, autoscaling, cost — not the transformation logic itself.

Q: What's the practical difference between `persist(MEMORY_AND_DISK_SER)` and `MEMORY_AND_DISK` on the JVM?
A: `MEMORY_AND_DISK` stores deserialized objects — fast to read but larger and heavier on GC; `MEMORY_AND_DISK_SER` stores serialized bytes — more CPU to read/write but a much smaller footprint and less GC pressure. On memory-constrained executors the serialized level often fits more data and reduces GC stalls. In PySpark the distinction is blurred because Python objects are pickled regardless.

Q: How does AQE convert a sort-merge join to a broadcast join at runtime?
A: After the input shuffle stages run, AQE has exact size statistics; if one side turns out smaller than the adaptive broadcast threshold it rewrites the remaining plan to a broadcast hash join, skipping the sort and the second shuffle. This rescues cases where compile-time estimates were too high to broadcast. It's a key reason AQE-on can outperform a hand-tuned static plan when stats were wrong.

Q: Why might you still see a `SortMergeJoin` even though one table looks small?
A: The "small" table's estimated size may exceed `autoBroadcastJoinThreshold` (10 MB default) due to stale or missing stats or because its compressed on-disk size hides a larger uncompressed footprint, so neither static planning nor AQE chose to broadcast it. Check the estimated size in `explain()`, refresh stats, raise the threshold, or force `broadcast()`. The lesson: trust the plan's size estimate, then correct it.

Q: When reprocessing a single bad `valuation_date` in an Iceberg NAV table, what write pattern avoids touching other dates and stays auditable?
A: Use a partition-scoped dynamic overwrite (`MERGE INTO` matched on date, or `writeTo(...).overwritePartitions()`/`INSERT OVERWRITE` with the date predicate) so only that date's partition is rewritten as a new atomic snapshot, leaving history untouched. Optionally stage it on a WAP branch and publish after validation. This gives a clean, reversible, time-travelable correction — exactly what an auditor expects for a restated NAV.

Q: What does `spark-submit` do and name two flags you'd set for a production batch job?
A: `spark-submit` is the launcher that packages your application and submits it to a cluster manager, creating the driver and requesting executors. Key flags include `--master` (the cluster manager URL or `local[*]`), `--deploy-mode cluster`, and resource flags like `--executor-memory`, `--executor-cores`, and `--num-executors`. You also pass `--conf` to set Spark properties such as `spark.sql.shuffle.partitions`. Getting these right is how a scheduled NAV job gets the resources and deploy mode you intend.

Q: What is the Tungsten engine and why does it make Spark SQL fast?
A: Tungsten is Spark SQL's execution backend that uses off-heap binary memory layouts, cache-aware processing, and whole-stage code generation to compile a chain of operators into tight JVM bytecode, cutting per-row virtual-call and boxing overhead. This is why DataFrame/SQL operations dramatically outperform equivalent Python RDD code. The flip side: dropping to a Python UDF breaks whole-stage codegen and reintroduces serialization cost, which is a common performance regression to watch for.

Q: Summarize the architect's end-to-end method for an unfamiliar slow Spark job.
A: Open the Spark UI, find the job triggered by the slow action, drill into its longest stage, and read the task distribution and shuffle metrics: a max-≫-median tail or huge single-task shuffle read means skew, heavy spill means memory pressure, an unexpected sort-merge means a missed broadcast. Apply the matching fix — salt or let AQE split the hot key, broadcast a small dimension, resize partitions/memory — then re-measure stage timings to prove the improvement. The discipline is evidence-first (UI → stages → shuffle/skew → fix), explainable to a junior, never "Spark is just slow" or "add nodes".

Q: What is `spark.memory.fraction`, its default, and what the rest of the heap is for?
A: `spark.memory.fraction` sets the size of Spark's unified memory region (execution plus storage) as a fraction of `(JVM heap − 300 MB)`, and its default is `0.6`. The remaining ~40% is user memory for your own data structures, UDF state, and internal metadata, plus the fixed 300 MB reserved buffer guarding against OOM on unusually large records. Lowering it gives Python/UDF-heavy jobs more user memory; raising it favours shuffle- and cache-heavy SQL.

Q: What does `spark.memory.storageFraction` control and what does its `0.5` default mean in practice?
A: It sets the fraction of the unified region (`M`) that cached/storage blocks are guaranteed and immune to eviction by execution; the default `0.5` means half of `M` is protected for cache, and execution can borrow the rest — and even evict cache beyond that line when it needs room. So if your job spills heavily while a big cache sits idle, execution is being starved because cached blocks under this floor can't be evicted. Tuning it down frees more guaranteed space for shuffles and joins.

Q: A PySpark job that does heavy Python UDF work keeps OOMing the executor even with big heaps — what memory region is the real culprit?
A: Python UDFs run in separate Python worker processes outside the JVM heap, so their memory comes from overhead/off-heap, not from `--executor-memory`; raising the heap doesn't help and can starve the Python side. The knob is `spark.executor.memoryOverhead` (or `spark.executor.pyspark.memory`), which sizes the non-heap allowance per executor. The gotcha: people keep bumping `--executor-memory` when the Python workers need overhead memory instead.

Q: What `spark.executor.memoryOverhead` default does Spark use and why does it exist?
A: It defaults to `max(384 MB, 0.10 × executorMemory)` and covers off-heap needs the JVM heap doesn't: Python/R worker processes, native libraries, netty shuffle buffers, and direct byte buffers. On Kubernetes/YARN, exceeding it gets your executor container killed by the resource manager (a `Container killed... exceeded memory limits` message) even though the JVM heap looked fine. Wide shuffles and PyArrow UDFs are the usual reasons you must raise it.

Q: An executor is killed with exit code `137` / "Container killed by YARN for exceeding memory limits" — what does this signal versus a JVM `OutOfMemoryError`?
A: Exit `137` (128 + SIGKILL 9) means the OS/resource-manager killed the whole container for exceeding its total memory limit, which is heap plus overhead plus off-heap — not a clean JVM `OutOfMemoryError` thrown inside the heap. The fix is raising `spark.executor.memoryOverhead` or reducing off-heap pressure, whereas a true heap OOM points to data volume or skew inside the JVM. Confused engineers raise `--executor-memory` for a `137`, which can make it worse by leaving less room for overhead within the container.

Q: What is Kryo serialization and when should an architect bother enabling it?
A: Kryo (`spark.serializer=org.apache.spark.serializer.KryoSerializer`) is a faster, more compact serializer than the default Java serialization — roughly 10x faster and smaller — used when Spark must serialize objects for shuffle, cache (`_SER` levels), and broadcast. It matters most for RDD/dataset code and network-intensive jobs; for pure DataFrame/SQL work Tungsten already uses an efficient binary format, so the gain is smaller. You register custom classes with `registerKryoClasses` to avoid storing full class names per object.

Q: With AQE on, what is `spark.sql.adaptive.coalescePartitions.minPartitionSize` and how does it differ from the advisory size?
A: `minPartitionSize` (default `1 MB`) is the floor below which AQE will not produce a coalesced partition, preventing pathologically tiny partitions, while `advisoryPartitionSizeInBytes` (default `64 MB`) is the *target* size AQE aims for when merging. So advisory is the goal and minPartitionSize is the lower guard rail. Together they keep post-shuffle tasks neither tiny nor uneven without you hand-setting `spark.sql.shuffle.partitions`.

Q: What does `spark.sql.adaptive.localShuffleReader.enabled` do and when does AQE use it?
A: When AQE converts a sort-merge join to a broadcast join at runtime, the local shuffle reader (default `true`) lets reduce tasks read shuffle blocks from the local map output without a network re-partition, since the shuffle is no longer needed. This avoids a redundant network round-trip after the join strategy changes. It is one of the quiet reasons an AQE-rescued join is faster than re-running with a static broadcast hint.

Q: What does the `MERGE` join hint force, and how does it rank against `BROADCAST` and `SHUFFLE_HASH`?
A: `/*+ MERGE(t) */` forces a sort-merge join on that relation; Spark's hint priority order is `BROADCAST` > `MERGE` > `SHUFFLE_HASH` > `SHUFFLE_REPLICATE_NL`, so if both sides carry conflicting hints, broadcast wins. You'd force `MERGE` to stop Spark broadcasting a table whose size it underestimates. There is no guarantee the hint is honoured if Spark can't support that join type for the query (e.g. a non-equi join).

Q: When would you reach for the `SHUFFLE_HASH` join hint instead of letting Spark pick?
A: `/*+ SHUFFLE_HASH(t) */` forces a shuffled hash join — both sides shuffled on the key, then one side built into an in-memory hash table — which can beat sort-merge when one side is moderately sized (too big to broadcast) and you want to skip the sort. It's third in hint priority, behind `BROADCAST` and `MERGE`. The risk is the build side not fitting in memory, so it's a deliberate choice for a known size profile, not a default.

Q: What is the `SHUFFLE_REPLICATE_NL` hint for and why is it a last resort?
A: `/*+ SHUFFLE_REPLICATE_NL(t) */` forces a shuffle-and-replicate nested-loop join, the only equi-key-free strategy, used for non-equi join conditions (range or inequality joins) where hash/merge can't apply. It's the lowest hint priority because nested-loop is `O(n×m)` and replicates data across partitions, so it's expensive. You accept it only when the join condition genuinely isn't an equality.

Q: A `join` with a condition like `a.value BETWEEN b.lo AND b.hi` is catastrophically slow — what join is Spark forced into and what's the mitigation?
A: A non-equi (range) join can't use sort-merge or hash strategies, so Spark falls back to a broadcast or shuffled nested-loop join that compares many row pairs, which explodes on large inputs. Mitigations include broadcasting the small side, bucketing the range into join keys so part of it becomes an equi-join, or pre-filtering. For fund data, range-joining prices to effective-date windows is a classic case where you add an equality key (e.g. a date bucket) to rescue it.

Q: How do you actually measure a DataFrame's in-memory size to decide if it's broadcast-safe?
A: Cache it, run an action to materialize, then read the cached size from the Spark UI Storage tab or `spark.sessionState... ` size estimate; alternatively inspect the `sizeInBytes` shown in `df.explain("cost")`'s statistics. On-disk Parquet size badly underestimates the in-memory footprint because of compression and columnar encoding, so trust the cached/estimated in-memory number against the 10 MB `autoBroadcastJoinThreshold`. This stops you broadcasting a table that's 8 MB on disk but 200 MB in memory.

Q: What does `df.explain("cost")` add over a plain `explain()`?
A: `explain(mode="cost")` (or `explain("cost")`) prints the optimized logical plan annotated with statistics — estimated row counts and `sizeInBytes` per operator — so you can see why Catalyst chose a join strategy and whether its size estimate is realistic. Plain `explain()` shows only the physical plan with no stats. When a broadcast/sort-merge decision looks wrong, the cost mode reveals whether stale stats produced a bad size estimate.

Q: What are the `explain()` modes available and what does each give you?
A: `df.explain(mode=...)` accepts `simple` (physical plan only, the default), `extended` (parsed, analyzed, optimized logical plans plus physical — same as `explain(True)`), `codegen` (the generated Java/whole-stage codegen), `cost` (optimized logical plan with statistics), and `formatted` (a clean physical plan with a numbered operator tree and per-node details). You pick `formatted` for readability, `cost` for stats-driven join debugging, and `codegen` to confirm whole-stage codegen wasn't broken by a UDF. Knowing the modes turns `explain` from a wall of text into a targeted tool.

Q: In a `formatted` explain plan, what does an `Exchange` node mean and why do you count them?
A: An `Exchange` node is a shuffle — data being repartitioned across the network — so each one marks a stage boundary and a cost centre. Counting `Exchange` nodes tells you how many shuffles a query pays for; collapsing two joins so they share a partitioning, or broadcasting to remove an exchange, is how you cut that count. A plan with three `Exchange` nodes where you expected one is the signal to look for an avoidable repartition or a missed broadcast.

Q: What is bucketing in Spark and how does it avoid shuffles on repeated joins?
A: Bucketing pre-shuffles a table into a fixed number of hash buckets on the join key at write time (`df.write.bucketBy(n, "fund_id").saveAsTable(...)`), so later joins or aggregations on that key read co-located buckets and skip the shuffle entirely. It trades a one-time write cost for repeated read-time savings. The catch: both sides must be bucketed the same way on the same key and bucket count, and it's a managed-table feature, which is why many lakehouse teams prefer Iceberg partitioning/sorting instead.

Q: Why is bucketing less common in Iceberg-based lakehouses despite its shuffle savings?
A: Classic Hive-style bucketing is tied to the Spark/Hive metastore table format and brittle across engines, whereas Iceberg offers hidden partitioning and the `bucket(N, col)` partition transform plus sort orders that achieve similar clustering without the fragile co-bucketing contract. So in an Iceberg estate you express the same intent through partition transforms and `sortBy`/compaction rather than `bucketBy`. The portability and engine-agnosticism is why architects steer toward Iceberg transforms.

Q: How do `repartition(n, col)` and `repartitionByRange(n, col)` differ, and when do you want range?
A: `repartition(n, col)` hash-partitions rows so the same key lands together but partition boundaries are arbitrary; `repartitionByRange(n, col)` sorts and assigns contiguous ranges of the column to each partition, giving ordered, range-clustered output. Use range partitioning before writing a table you'll query by ranges (e.g. `valuation_date` windows) so each file holds a contiguous date range and min/max stats prune tightly. Range partitioning samples the data to pick boundaries, adding a small cost.

Q: What's the difference between `sort` / `orderBy` (global) and `sortWithinPartitions`?
A: `orderBy`/`sort` produces a total global ordering, which forces a range-partition shuffle so all data is sorted across partitions; `sortWithinPartitions` only sorts each partition locally with no shuffle. For writing well-clustered files you usually want `sortWithinPartitions` (or a partition-then-local-sort) because you need ordering within each output file, not a globally sorted dataset, and you avoid the expensive global shuffle. Using `orderBy` before a write when you only needed local order is a common needless-shuffle mistake.

Q: Why can a global `orderBy` before a `write` quietly serialize your job?
A: A global sort range-partitions the data and, combined with a single output, can funnel work toward few tasks, and a final `orderBy().limit()` or `coalesce(1)` collapses parallelism to one task that does all the writing. The symptom is a fast multi-task stage followed by one slow single-task stage. If you only need ordered files, `sortWithinPartitions` keeps full parallelism; if you need one sorted file you accept the serial cost deliberately.

Q: Your output has thousands of tiny files even though the job is small — first thing to check before a write?
A: Check the partition count of the DataFrame right before the write (`df.rdd.getNumPartitions()`); each partition becomes at least one file per table partition, so a 200-partition frame writing to a partitioned table can yield thousands of files. The fix is `coalesce`/`repartition` to a sensible count, or `repartition(col)` matching the table partition column, before writing. With Iceberg you can also enable write-side fanout controls and run compaction.

Q: How do you find the number of partitions of a DataFrame, and why isn't there a direct `df.getNumPartitions()`?
A: You go through the underlying RDD: `df.rdd.getNumPartitions()`, because partitioning is an execution-layer property of the RDD, not a logical property of the DataFrame — Catalyst may re-plan partitioning, so the DataFrame API deliberately doesn't expose it as a stable attribute. Calling `df.rdd` also forces analysis of the plan. For shuffle outputs the count is governed by `spark.sql.shuffle.partitions` (modulated by AQE), not by this call.

Q: What does the AQE skew-join split actually produce in the Spark UI, and how do you confirm it fired?
A: After a skewed sort-merge join, the SQL DAG node shows the join with a "skew" annotation and the stage has *more* tasks than the shuffle-partition count because the hot partition was split into sub-partitions; task durations become balanced where before one dominated. You confirm it fired by seeing the split partitions in the AQE-adjusted plan (the SQL tab shows the final adaptive plan) and the disappearance of the max-≫-median tail. If you still see the tail, the partition was below the 256 MB / 5.0× thresholds.

Q: A skewed join still stalls even with AQE skew-join enabled — give two concrete reasons and the next step.
A: First, the hot partition may be below `skewedPartitionThresholdInBytes` (256 MB) or under `skewedPartitionFactor × median` (5.0×), so AQE doesn't classify it as skewed; second, the join isn't a plain sort-merge join AQE can split (e.g. it got broadcast, or it's an outer join on the wrong side). The next step is to confirm the plan and thresholds, then fall back to manual salting or lower the threshold for that query. Null-keyed skew also evades it because nulls are a "real" hot key you must filter first.

Q: How would you implement salting end-to-end in PySpark for a skewed `groupBy().sum()` (not a join)?
A: Add a salt with `withColumn("salt", (rand()*N).cast("int"))`, do a first `groupBy("fund_id","salt").agg(sum("amount").alias("partial"))` to spread the hot key across `N` tasks, then a second `groupBy("fund_id").agg(sum("partial"))` to combine partials. This two-stage aggregation defuses the single dominant key. The cost is an extra aggregation pass, justified only for keys that are genuinely hot.

Q: Why does map-side partial aggregation reduce but not eliminate skew on a dominant key?
A: Partial aggregation shrinks each map task's contribution to one partial row per key, so the *volume* shuffled for the hot key is tiny — but all those partials for one key still converge on a single reduce task for the final combine, so that one task is still the last to finish. The straggler is now cheap per row but still serial. That residual is exactly what salting's two-stage combine or AQE skew-split removes.

Q: For a windowed calculation like 30-day rolling NAV performance, why can a `Window.partitionBy(fund_id)` reintroduce skew that a `groupBy` salt trick can't fix?
A: A window function must place *all* rows of a partition key together and ordered to compute the running window, so a mega-fund's entire history lands on one task and can't be salted the way a sum can (salting would break the ordering/frame). The fix is different: ensure the window partition key is high-cardinality enough, push as much pre-aggregation upstream as possible, or split the computation by an additional time bucket. This is why window-heavy NAV jobs are a distinct skew case from aggregations.

Q: What does `spark.sql.files.maxPartitionBytes` control on the read side and when do you tune it?
A: It sets the maximum bytes Spark packs into a single input partition when reading files (default `128 MB`), so it governs read-side parallelism and task count before any shuffle. Lowering it creates more, smaller read tasks (more parallelism for CPU-heavy parsing); raising it creates fewer, larger tasks (fewer tasks, less overhead) for simple scans. It's the read-side analogue of shuffle-partition tuning and matters when a scan stage is the bottleneck.

Q: A scan stage reads a single huge Parquet file with only one task — why, and how do you parallelize it?
A: A single unsplittable or single-row-group file (or gzip-compressed text, which isn't splittable) gives Spark one input split and therefore one task, no matter how many cores you have. The fix is to rewrite the source into multiple files or larger row-group-count Parquet, use a splittable codec (Snappy/zstd Parquet is splittable at row-group level), or repartition after the read. Unsplittable gzip CSV is a recurring "why is my 100-core cluster using one core" trap.

Q: Why are Snappy/zstd-compressed Parquet files splittable while gzip-compressed CSV is not, and why does it matter for parallelism?
A: Parquet stores independently compressed row groups, so Spark can assign different row groups to different tasks regardless of the codec; a gzip-compressed CSV is one continuous compressed stream with no internal boundaries, so it can't be split and must be read by one task. This is why columnar Parquet parallelizes reads that a single large gzip CSV cannot. For ingest, landing data as Parquet (or many smaller files) preserves read parallelism.

Q: What is dynamic partition pruning (DPP) and how does it help a star-schema NAV join?
A: DPP (`spark.sql.optimizer.dynamicPartitionPruning.enabled`, default `true`) uses the result of filtering a small dimension at runtime to prune partitions of the large partitioned fact table before scanning it, so a filter on a dimension (e.g. a fund-attributes table) prunes the partitioned holdings table even though the filter isn't directly on its partition column. This avoids reading partitions that can't survive the join. It's a major saving when you filter facts indirectly through a dimension.

Q: What does `spark.sql.shuffle.partitions = 200` mean for a tiny dataset, and why is that wasteful without AQE?
A: Without AQE, every shuffle produces exactly 200 partitions regardless of data size, so a 5-row aggregation spawns 200 mostly-empty tasks — pure scheduling overhead and 200 tiny output files. AQE's coalescing merges these back toward the advisory size, which is why the default is safe to leave high on Spark 3.2+. On older Spark or with AQE disabled, you'd lower it for small jobs to avoid the tiny-task tax.

Q: How does `spark.dynamicAllocation.enabled` change executor behaviour and what does it need to work?
A: Dynamic allocation lets Spark add executors when there's a task backlog and release idle ones, sizing the cluster to the workload instead of a fixed `--num-executors`. It requires an external shuffle service (or shuffle tracking) so shuffle files survive after an executor is removed, otherwise releasing an executor would lose its shuffle output. It's how managed platforms autoscale a job, and the trade-off is allocation latency at stage boundaries.

Q: Why must an external shuffle service exist before you remove executors during a job?
A: Shuffle map outputs are written to an executor's local disk and read by later reduce tasks; if the executor is decommissioned, those files vanish and dependent tasks fail and must recompute. An external shuffle service (or shuffle-file migration/tracking) holds or relocates the shuffle data independently of the executor lifecycle, making safe down-scaling possible. Without it, dynamic allocation either can't shrink or triggers recompute storms.

Q: A job fails with `FetchFailedException` during a shuffle — what does it mean and what are the usual causes?
A: A `FetchFailedException` means a reduce task couldn't fetch shuffle blocks from a map-side executor — the producing executor died, ran out of memory/disk, was preempted, or the shuffle data was lost — so Spark retries the failed map stage to regenerate the blocks. Common causes are executor OOM/GC killing the producer, decommission without a shuffle service, or disk pressure. You diagnose by checking which executor died and why, not by retrying blindly.

Q: What is `spark.task.maxFailures` and why does a skewed task sometimes exhaust it?
A: `spark.task.maxFailures` (default `4`) is how many times a single task may fail before Spark fails the whole stage and job. A skewed, oversized partition can repeatedly OOM the same way on each retry, burning all four attempts and killing the job — retrying doesn't help because the data, not luck, is the problem. The fix is to address the skew/size, since raising `maxFailures` just delays the inevitable.

Q: Speculative execution is enabled but a job is still stuck on one task — why didn't speculation help?
A: Speculation launches a duplicate of a slow task on another node, but if the slowness is data skew the duplicate processes the *same* oversized partition and is just as slow, so it can't win. Speculation only helps infrastructure stragglers (a slow/flaky node), where a copy on a healthy node finishes first. Mistaking a skew straggler for an infra straggler and enabling speculation wastes resources without fixing wall-clock.

Q: How do you tell a data-skew straggler from an infrastructure straggler in the UI?
A: Compare the slow task's metrics to its peers: a data-skew straggler has a far larger *shuffle read* / *input size* than the median, while an infra straggler has normal input but high *GC time*, repeated failures, or always sits on the same executor/host. Skew is fixed by rebalancing the data; infra is fixed by speculation, blacklisting the bad node, or replacing it. The shuffle-read column is the single most decisive piece of evidence.

Q: What does the Spark UI Storage tab tell you, and how do you spot a cache that isn't paying off?
A: The Storage tab lists each cached/persisted RDD or DataFrame with its storage level, the fraction cached in memory vs spilled to disk, and total size. A cache showing low "fraction cached" (mostly on disk) or one that's never read again (no downstream reuse) signals wasted memory that's likely causing spill elsewhere. You'd `unpersist()` it or pick a smaller serialized level. This is how you audit "cache everything" cargo-culting with evidence.

Q: Why should you call `unpersist()` and when does it matter in a long pipeline?
A: `unpersist()` evicts a cached DataFrame from executor memory/disk so it stops competing with execution memory and later caches; in a long pipeline with several cache points, stale caches accumulate and squeeze shuffles, causing spill. Spark eventually evicts via LRU, but explicit `unpersist()` after the last use is deterministic and avoids the pressure window. It matters most when memory is tight and the cached frame is large.

Q: What is whole-stage code generation and how does a Python UDF break it?
A: Whole-stage codegen (a Tungsten feature) fuses a chain of operators in a stage into a single compiled Java function, eliminating per-row virtual calls and intermediate objects. A Python UDF can't be compiled into that JVM function — Spark must serialize each row out to a Python worker, run the UDF, and serialize back — so codegen breaks at the UDF boundary and you pay serialization per row. In the plan you see the operator outside the `WholeStageCodegen` box, which is the visual cue to replace it with a native expression.

Q: How does a pandas/Arrow UDF reduce the cost a plain Python UDF imposes?
A: A pandas (vectorized) UDF transfers data to Python in Arrow-format batches and runs your function on whole pandas Series at once, so serialization happens per batch instead of per row and the Python code is vectorized — often an order of magnitude faster than a row-at-a-time Python UDF. It still breaks whole-stage codegen and crosses the JVM↔Python boundary, so a native Spark expression beats it when one exists. Arrow batching is the mitigation when you genuinely need Python logic.

Q: What does `spark.sql.execution.arrow.pyspark.enabled` change and what's the gotcha?
A: Enabling Arrow (`spark.sql.execution.arrow.pyspark.enabled=true`) makes conversions between Spark and pandas — `toPandas()` and pandas UDFs — use Arrow columnar transfer instead of slow row-by-row pickling, dramatically speeding them up. The gotcha is that `toPandas()` still collects the entire result to the driver, so Arrow makes a dangerous operation faster, not safer — a large `toPandas()` still OOMs the driver. It also silently falls back to non-Arrow on unsupported types unless you watch for the warning.

Q: How does Iceberg time travel let you audit a NAV value as of a past run from Spark?
A: You query a prior snapshot with `... VERSION AS OF <snapshot_id>` / `TIMESTAMP AS OF '<ts>'` (or the DataFrame `option("snapshot-id", ...)`), reading the exact table state that produced an earlier NAV without restoring anything. Each Spark write created an immutable snapshot, so an auditor's "what did this fund's NAV look like before the restatement" is answered by reading the pre-restatement snapshot. This is the auditability payoff of `MERGE INTO` committing atomic snapshots.

Q: After many `MERGE INTO` writes the Iceberg table's read planning gets slow — what maintenance fixes it?
A: Frequent small commits accumulate many data files, delete files, and metadata/manifest entries, slowing query planning; you run compaction (`rewrite_data_files`), manifest rewriting (`rewrite_manifests`), and snapshot expiry (`expire_snapshots`) to merge small files, prune deletes, and drop old snapshots/orphan files. These are Iceberg maintenance procedures callable from Spark SQL. Skipping them is why a heavily-merged NAV table degrades over weeks even though each write was correct.

Q: In a WAP flow, what exactly happens at "publish" and how is it different from a normal write?
A: Publish fast-forwards the table's `main` branch reference to the validated branch's snapshot — a metadata-only pointer move, atomic and instantaneous — so readers switch from the old snapshot to the audited one with no data rewrite. A normal write commits directly to `main`, making data visible the moment it lands; WAP decouples "written" (on the branch) from "visible" (after publish). That gap is precisely where validation gates a bad NAV before any consumer can read it.

Q: What is the practical difference between `spark.sql.autoBroadcastJoinThreshold` and `spark.sql.adaptive.autoBroadcastJoinThreshold`?
A: The first is the *static* (compile-time) threshold Catalyst uses from table statistics to decide a broadcast before running, default `10 MB`; the AQE one is the *runtime* threshold applied after shuffle stats are known, and if unset it falls back to the static value. The runtime version is what lets AQE broadcast a side whose true size only became known after a shuffle. You can set the AQE threshold higher to let runtime broadcasts happen that compile-time stats were too pessimistic to allow.

Q: A nightly job that worked for months suddenly OOMs on a broadcast — what changed and how do you confirm?
A: The "small" reference side likely grew past the broadcast threshold over time (more share classes, more funds), so Spark now broadcasts something too large; stats drift or a data growth pushed it over the line. Confirm by checking the side's current size against `autoBroadcastJoinThreshold` and reading `explain()` to see which side is broadcast. The durable fix is to re-evaluate the threshold or force `MERGE`/sort-merge for that join, and to monitor reference-table size as an operational metric.

Q: Why might the same Spark job be faster on DuckDB for a 20 GB single-day NAV recompute but slower for a 2 TB full-history rebuild?
A: At 20 GB the data fits one machine's memory, so DuckDB avoids Spark's shuffle-network, executor JVM startup, and scheduling overhead and wins on wall-clock and cost; at 2 TB it exceeds single-node memory and must spill or can't hold the working set, where Spark's distributed shuffle and many-node parallelism are the right tool. The crossover is roughly where data no longer fits one node comfortably (tens of GB into low hundreds). The architect's rule is to measure both at the actual data size, not assume one engine always wins.
