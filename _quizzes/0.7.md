# Phase 0 · 0.7 What data physically is — 100 self-test questions

Q: At the lowest level, what is any file on disk physically made of?
A: A sequence of bytes — each byte is 8 bits, an integer from 0 to 255 — and nothing in the bytes themselves says whether they mean text, an image, or a number; that meaning comes only from how a program chooses to interpret them.

Q: What is the difference between "bytes" and "text"?
A: Bytes are raw numeric values stored on disk; text is what you get when those bytes are decoded using a character encoding that maps byte values to characters, so the same bytes can become different text under different encodings.

Q: What is a character encoding?
A: A defined mapping between characters (letters, digits, symbols) and the byte sequences that represent them, e.g. UTF-8 or `ISO-8859-1`, used to turn text into bytes (encode) and bytes back into text (decode).

Q: What is the difference between a character set and an encoding?
A: A character set (like Unicode) is the catalogue of characters and their code points; an encoding (like UTF-8) is the rule for turning those code points into actual bytes — one character set can have several encodings.

Q: What is Unicode?
A: A universal character set that assigns a unique number (code point, written like `U+00E9`) to virtually every character in every writing system, plus symbols and emoji, so text is not tied to one language's legacy encoding.

Q: What is a Unicode code point?
A: The abstract number assigned to a character in Unicode, written as `U+` followed by hex, e.g. `U+20AC` is the euro sign `€`; it is independent of how that character is stored as bytes.

Q: What is UTF-8 and why is it the default choice today?
A: UTF-8 is a variable-length encoding of Unicode using 1 to 4 bytes per character; it is ASCII-compatible for the first 128 characters, space-efficient for Latin text, and the de facto standard for files, web, and APIs.

Q: How many bytes does UTF-8 use for a plain ASCII character like `A`?
A: Exactly one byte, with the same value as ASCII (`A` is byte `0x41`/65), which is why valid ASCII text is also valid UTF-8.

Q: Why can a single accented character like `é` take more than one byte in UTF-8?
A: Because any code point above `U+007F` falls outside the single-byte ASCII range, so UTF-8 encodes it across 2 to 4 bytes; `é` (`U+00E9`) is two bytes (`0xC3 0xA9`).

Q: What is ASCII?
A: A 7-bit encoding defining 128 characters (English letters, digits, punctuation, control codes); it covers no accented or non-Latin characters, which is why broader encodings like UTF-8 exist.

Q: What is `ISO-8859-1` (Latin-1)?
A: A single-byte legacy encoding covering ASCII plus 128 extra Western-European characters; every byte 0–255 maps to exactly one character, so it never errors on decode but happily produces wrong characters for non-Latin-1 data.

Q: What is "Windows-1252" and why does it matter for legacy fund files?
A: A single-byte Microsoft encoding very similar to Latin-1 but using the 0x80–0x9F range for characters like curly quotes and the euro sign; old Windows-exported CSVs often use it, so decoding them as UTF-8 can fail or mangle those symbols.

Q: What is UTF-16?
A: A Unicode encoding using 2 or 4 bytes per character (16-bit code units); it is common inside Windows and Java internally but uncommon for interchange files, and it is not ASCII-compatible at the byte level.

Q: What does it mean that UTF-8 is "self-synchronizing"?
A: The leading bits of each byte indicate whether it starts a character or continues one, so a decoder can find character boundaries even after starting mid-stream — a property single-byte legacy encodings lack.

Q: What is mojibake?
A: Garbled text that appears when bytes are decoded with the wrong encoding, e.g. `é` showing up as `Ã©` because UTF-8 bytes were read as Latin-1; the data is intact but mis-interpreted.

Q: You open a CSV and see "Société Générale" instead of "Société Générale" — what happened?
A: The file is UTF-8 but was decoded as Latin-1/Windows-1252, so each multi-byte UTF-8 sequence was shown as separate single-byte characters; re-open it declaring the correct UTF-8 encoding.

Q: Mojibake shows `Ã©` where you expected `é`. Which direction was the encoding mistake?
A: The producer wrote UTF-8 and the reader decoded it as Latin-1/Windows-1252; `Ã©` is exactly the two UTF-8 bytes of `é` interpreted as two single-byte characters.

Q: What is a BOM?
A: A Byte Order Mark — a special character (`U+FEFF`) optionally placed at the start of a file to signal its encoding/byte order; in UTF-8 it appears as the three bytes `0xEF 0xBB 0xBF`.

Q: Why does a UTF-8 BOM cause trouble in CSV files?
A: Many parsers do not strip it, so the BOM bytes get attached to the first column name (e.g. header becomes `﻿ISIN` instead of `ISIN`), breaking column lookups and joins.

Q: How do you read a BOM-prefixed UTF-8 CSV correctly in Python?
A: Open it with encoding `utf-8-sig`, which recognises and discards a leading UTF-8 BOM if present and otherwise behaves like plain UTF-8.

Q: Why does Excel sometimes add a BOM when you "Save As CSV UTF-8"?
A: Excel writes the UTF-8 BOM so that it can itself later detect the file as UTF-8 on re-open; this helps Excel but can surprise downstream tools that don't expect the BOM.

Q: How can you check whether a file starts with a UTF-8 BOM from the command line?
A: Inspect the first bytes with a hex viewer such as `xxd file.csv | head -1` or `hexdump -C file.csv | head -1` and look for `ef bb bf` at offset 0.

Q: Why should encoding always be specified explicitly when reading files in a pipeline?
A: Because the default encoding varies by OS, locale, and tool (Windows may default to a code page, Linux to UTF-8), so relying on the default makes the same code produce mojibake or crashes on a different machine.

Q: What is the safest general rule for encodings in a new data pipeline?
A: Standardise on UTF-8 everywhere, declare it explicitly on every read and write, and convert legacy inputs to UTF-8 at the ingestion boundary rather than letting unknown encodings propagate.

Q: What error does Python raise when it cannot decode bytes with the chosen encoding?
A: A `UnicodeDecodeError`, naming the codec and the byte position that failed — a strong signal that the file's real encoding differs from the one you assumed.

Q: Why does decoding never "fail" with Latin-1 even on non-Latin-1 data?
A: Because Latin-1 maps all 256 possible byte values to a character, so it always succeeds at the byte level but silently yields wrong characters — making it dangerous as a fallback that hides encoding bugs.

Q: What is CSV?
A: Comma-Separated Values — a plain-text tabular format where rows are lines and fields are separated by a delimiter; it is simple and ubiquitous but has no formal standard, so dialects vary.

Q: Why is "CSV has no schema" an important caveat?
A: Because the file carries no declared data types, lengths, or constraints — every field is just text, so the meaning (date format, decimal, ISIN vs free text) lives only in convention and downstream parsing code.

Q: What is a CSV "dialect"?
A: The specific combination of delimiter, quote character, escape rules, line terminator, and quoting policy a particular CSV file uses; mismatching the dialect on read produces misaligned columns.

Q: Why are European CSV files often semicolon-delimited?
A: Because many European locales use the comma as the decimal separator (e.g. `1.234,56`), so spreadsheets default to `;` as the field delimiter to avoid clashing with decimals.

Q: A NAV file from a Luxembourg system uses `;` but your parser assumes `,` — what symptom do you see?
A: The whole row lands in a single column (or columns are wildly misaligned), because the parser found no commas to split on; set the delimiter explicitly to `;`.

Q: How does CSV handle a field value that itself contains the delimiter?
A: The field is wrapped in quote characters (usually `"`), so `"Paris, France"` is one field; a parser that ignores quoting would wrongly split it into two.

Q: How is a literal quote character represented inside a quoted CSV field?
A: By doubling it, so `"He said ""hi"""` decodes to the value `He said "hi"`; some dialects instead use a backslash escape, which is a dialect difference to watch for.

Q: How can a single CSV field legitimately contain a line break?
A: When the field is quoted, a newline inside the quotes is part of the value, not a row separator — so a "row" can span multiple physical lines, which naive line-by-line splitting breaks.

Q: Why is splitting a CSV on newlines and commas with simple string operations risky?
A: Because it ignores quoting, embedded delimiters, embedded newlines, and escaped quotes, so any field containing those characters corrupts the parse; use a real CSV parser instead.

Q: What is the safe way to parse CSV in Python?
A: Use the standard-library `csv` module (or `pandas.read_csv`), specifying delimiter, quotechar, and encoding, so quoting and embedded newlines are handled per the dialect.

Q: What line-ending difference between Windows and Unix can affect CSV reading?
A: Windows uses `\r\n` (CRLF) and Unix uses `\n` (LF); a parser or tool expecting one may leave stray `\r` characters on field values or mis-count lines if it expects the other.

Q: How do you detect a stray trailing `\r` on the last column of each row?
A: Values look fine in print but comparisons/joins fail; inspect with a hex view or check `repr()` of the value to reveal a hidden `\r`, then normalise line endings or set the parser to universal newlines.

Q: Why can leading zeros disappear from an ISIN or account number opened in Excel?
A: Excel auto-detects the column as numeric and drops leading zeros (and can switch to scientific notation), so `0001234` becomes `1234`; import the column as text or avoid editing identifiers in Excel.

Q: Why might `DE000` style identifiers turn into `4.5E+11` in a spreadsheet?
A: The spreadsheet guessed the field was a number and reformatted it in scientific notation; this is a data-typing trap of opening text files in spreadsheet apps, and it can silently corrupt identifiers.

Q: What does it mean that CSV is "untyped"?
A: Every value is text on disk; whether `2026-06-13`, `42`, or `True` is a date, integer, or boolean is decided entirely by the reader, so the same column can be parsed differently by different tools.

Q: How do you tell a CSV reader to treat a column as text to preserve identifiers?
A: In `pandas.read_csv`, pass `dtype={"ISIN": str}` (and consider `keep_default_na=False`) so the column is not coerced to numbers or NaN, preserving leading zeros and codes.

Q: Why can a CSV with a thousands separator like `1,234.56` break a comma-delimited parse?
A: The embedded comma is read as a field delimiter unless the value is quoted, splitting one number into two columns; this is a classic reason numeric exports must be quoted or use a different delimiter.

Q: What is a header row and why is its presence ambiguous in CSV?
A: The first row often holds column names, but CSV has no flag declaring this, so a tool must be told whether row 1 is a header or data — guessing wrong shifts every column or loses a record.

Q: Two CSV exports of the same NAV report have different column orders — why is that a CSV-specific risk?
A: Because CSV columns are positional with no enforced schema, so code that relies on column position rather than header name breaks when the order changes; always select by header name.

Q: What is JSON?
A: JavaScript Object Notation — a text format for structured data built from objects (key/value maps), arrays (ordered lists), strings, numbers, booleans, and null; widely used in APIs and config.

Q: What are the two container types in JSON?
A: Objects, written with `{ }` as unordered collections of string-keyed values, and arrays, written with `[ ]` as ordered lists of values; values can nest arbitrarily.

Q: What primitive value types does JSON natively support?
A: Strings, numbers, booleans (`true`/`false`), and `null` — and that's all; there is no native date, time, decimal, or binary type.

Q: Why is the lack of a native date type in JSON a problem?
A: Because dates must be smuggled as strings (e.g. ISO-8601 `"2026-06-13"`) or numbers (epoch), with no guarantee of format, so producers and consumers must agree on a convention or dates parse wrongly.

Q: Why are JSON numbers risky for monetary or NAV values?
A: JSON numbers are typically read as IEEE-754 floats, which cannot represent many decimals exactly (e.g. `0.1`), so amounts can lose precision; transmit money as strings or scaled integers to preserve exactness.

Q: How are large identifiers like big integers a hazard in JSON?
A: Many parsers map JSON numbers to 64-bit floats, which lose precision beyond 2^53, so a long numeric ID can be silently rounded; send such IDs as strings.

Q: Are JSON object keys ordered?
A: The JSON spec treats object members as unordered, so code must not rely on key order (though many parsers preserve insertion order in practice); arrays, by contrast, are ordered.

Q: Can JSON object keys be duplicated, and what happens?
A: The spec doesn't forbid duplicate keys but says behaviour is undefined; most parsers keep the last value, so duplicates can silently drop data — treat them as a malformed input.

Q: What is the difference between JSON and JSON Lines (NDJSON)?
A: Plain JSON is usually one document (often a big array); JSON Lines stores one independent JSON object per line, which streams better and lets you process huge datasets record-by-record without loading everything.

Q: Why must strings in JSON use double quotes?
A: The JSON spec only allows double-quoted strings; single quotes (valid in JavaScript) make the document invalid JSON, a common cause of parse errors when hand-editing.

Q: How are special characters escaped in JSON strings?
A: With backslash escapes such as `\"`, `\\`, `\n`, `\t`, and `\uXXXX` for arbitrary code points; an unescaped control character or lone backslash makes the JSON invalid.

Q: Does JSON allow comments or trailing commas?
A: No — standard JSON forbids both, so a trailing comma after the last array/object element or a `//` comment causes a parse error; relaxed variants like JSON5 allow them but aren't standard JSON.

Q: What encoding should JSON be in?
A: JSON text is exchanged as Unicode and UTF-8 is the standard and assumed default; a BOM is not recommended, and many strict parsers reject a leading BOM.

Q: What is XML?
A: eXtensible Markup Language — a verbose, tag-based text format for hierarchical data, using nested elements and attributes; it predates JSON and remains common in finance, including EMT/EPT fund-data files.

Q: What is the difference between an XML element and an attribute?
A: An element is a tagged node like `<price>100</price>` that can contain text and child elements; an attribute is a name/value pair inside a start tag like `<price currency="EUR">100</price>` and cannot itself contain child elements.

Q: When should data be an element vs an attribute in XML?
A: Convention favours elements for substantive, possibly repeating or structured data and attributes for simple metadata about an element; but XML doesn't enforce this, so consumers must read both.

Q: What is an XML namespace and why does it exist?
A: A namespace qualifies element/attribute names with a URI to avoid name collisions when documents combine vocabularies, declared via `xmlns`; e.g. two schemas can both define `<id>` without clashing.

Q: Why do XML namespaces complicate parsing and XPath queries?
A: Because elements become qualified names (prefix maps to a URI), so an XPath like `//id` may match nothing unless you register the namespace and query the fully qualified name; ignoring namespaces is a frequent XML bug.

Q: What is the difference between a well-formed and a valid XML document?
A: Well-formed means it obeys XML syntax rules (proper nesting, closed tags, quoted attributes); valid means it additionally conforms to a declared schema (DTD/XSD) defining allowed elements, types, and structure.

Q: What is an XSD?
A: An XML Schema Definition — itself an XML document that formally defines the structure, element/attribute names, data types, cardinality, and constraints a conforming XML file must satisfy; it gives XML a real schema.

Q: What is CDATA in XML?
A: A `<![CDATA[ ... ]]>` section that holds text verbatim without the parser interpreting `<`, `>`, or `&`, useful for embedding markup or raw content without escaping each character.

Q: How are reserved characters like `<` and `&` represented in normal XML text?
A: As entity references — `&lt;` for `<`, `&gt;` for `>`, `&amp;` for `&`, `&quot;` and `&apos;` for quotes — because the raw characters would otherwise be parsed as markup.

Q: What is the XML prolog/declaration and what does it tell you?
A: The optional first line like `<?xml version="1.0" encoding="UTF-8"?>` declaring the XML version and the character encoding of the document, which parsers use to decode the bytes correctly.

Q: In fund data, what is an EMT file and what format is it typically?
A: The European MiFID Template, a standardised file of product cost/target-market data exchanged between asset managers and distributors; it is commonly distributed as CSV but the data model is schema-defined, with newer versions also available as XML.

Q: Why does a schema-defined format like EMT reduce data-exchange errors compared with ad-hoc CSV?
A: Because the agreed schema fixes field names, order, types, and allowed values, so producers and consumers interpret each column identically rather than relying on undocumented local conventions.

Q: What does "schema" mean in data engineering?
A: The formal definition of a dataset's structure — its fields, their data types, constraints, relationships, and sometimes order — that tells any program how to interpret and validate the data.

Q: What is the difference between "schema-on-write" and "schema-on-read"?
A: Schema-on-write enforces structure when data is stored (as in a relational database), rejecting bad data up front; schema-on-read applies structure only when the data is queried (as with raw files in a data lake), accepting anything and validating later.

Q: Which formats are schema-on-write and which are schema-on-read?
A: A relational table is schema-on-write; raw CSV/JSON/XML files dropped into storage are schema-on-read because nothing enforces structure until something parses them.

Q: What does it mean for data to be "structured"?
A: It conforms to a fixed, predefined schema of rows and typed columns — the classic relational table or a well-defined fixed-format file — so its shape is known before you read it.

Q: What does "semi-structured" data mean?
A: Data with organisational markers (tags, keys, nesting) but no rigid fixed schema, so structure can vary record-to-record; JSON and XML are the canonical examples.

Q: What does "unstructured" data mean?
A: Data with no predefined data model or field structure, such as free-text documents, PDFs, emails, images, and audio; meaning must be extracted rather than read off a schema.

Q: Where does CSV sit on the structured/semi-structured spectrum?
A: It is tabular and looks structured, but because it carries no enforced schema or types it behaves like semi-structured/loosely-structured text; the structure is convention, not guarantee.

Q: Why does the structured/semi/unstructured distinction matter for storage choices?
A: Because structured data fits relational databases and columnar formats, semi-structured fits document stores or flexible columns, and unstructured fits object storage with separate indexing — picking wrong makes querying painful.

Q: What is the core difference between a spreadsheet and a database?
A: A spreadsheet is a single-user, visual, free-form grid where formatting and formulas mix with data and types are loose; a database enforces a schema, types, and constraints, supports concurrent multi-user access, and scales to large, queryable datasets.

Q: Why are spreadsheets risky as a "system of record" for fund data?
A: Because they silently coerce types (dropping leading zeros, reformatting dates/IDs), lack enforced constraints and audit trails, allow manual overwrites, and don't scale or support reliable concurrent updates, leading to errors that are hard to trace.

Q: What is a key reliability advantage of a database over a spreadsheet?
A: A database enforces a schema and constraints (types, keys, not-null, uniqueness) and provides transactions, so invalid data is rejected and concurrent changes stay consistent — guarantees a spreadsheet cannot give.

Q: Why does a database give you typed columns while CSV does not?
A: Because a database column has a declared type (e.g. `DECIMAL`, `DATE`) enforced on write, whereas CSV stores everything as text and leaves typing to whatever reads it.

Q: How does a database handle concurrent edits that a spreadsheet cannot?
A: Through transactions and locking/MVCC that isolate concurrent changes and keep data consistent, whereas a shared spreadsheet typically forces last-writer-wins or file conflicts.

Q: For a daily NAV feed consumed by many systems, why prefer a database or schema-defined file over an emailed spreadsheet?
A: Because the schema-defined source enforces consistent types and structure, supports automated validation and access control, and avoids the silent corruption and version chaos of manually edited spreadsheets.

Q: What is a fixed-width (fixed-position) file?
A: A text file where each field occupies a fixed number of character columns rather than being delimiter-separated; common in legacy banking/fund feeds, and parsed by column offsets defined in an accompanying spec.

Q: Why is a fixed-width file fragile if the spec is wrong by even one column?
A: Because parsing is purely positional, an off-by-one in any field's start/length shifts every subsequent field on the line, silently corrupting all downstream values without raising an error.

Q: What is TSV and when is it preferable to CSV?
A: Tab-Separated Values uses a tab delimiter; it's handy when fields commonly contain commas (so a comma delimiter would need heavy quoting), though tabs inside values still require care.

Q: What is Parquet and how does it differ from CSV?
A: Parquet is a binary, columnar, schema-carrying file format with compression and typed columns; unlike text CSV it stores its own schema and types, is far more space- and query-efficient, but isn't human-readable.

Q: Why is a self-describing format like Parquet or Avro safer than CSV for data interchange?
A: Because the schema and types travel with the data, so consumers don't have to guess delimiters, encodings, or types — eliminating a whole class of CSV/encoding ambiguity bugs.

Q: A CSV import puts all data in one column — what is the first thing to check?
A: The delimiter: the file likely uses `;` or a tab while the parser assumed `,` (or vice versa); confirm by viewing raw bytes and set the delimiter to match the actual file.

Q: A CSV has more parsed fields in some rows than the header has columns — what's the likely cause?
A: A delimiter character appears unquoted inside a field (e.g. a comma in a name or a decimal), so that field was split; the fix is correct quoting in the source or a parser that honours the file's quoting.

Q: Rows in a CSV "disappear" or merge — what trap should you suspect?
A: Embedded newlines inside quoted fields being treated as row breaks by a tool that doesn't respect quoting, splitting one record into two or merging text; use a quoting-aware parser.

Q: A date column reads as `13/06/2026` in one file and `06/13/2026` in another — what's the underlying issue?
A: CSV has no date type, so date format is just a textual convention that varies by locale; you must know and specify the format on parse or risk swapping day and month silently.

Q: Numbers like `1.234,56` are misread as `1.234` — what's wrong?
A: The file uses European decimal notation (comma decimal, dot thousands) but the parser assumes US notation; set the decimal and thousands separators (e.g. pandas `decimal=","` `thousands="."`) to match the locale.

Q: An API returns `"amount": 0.1 + 0.2` style values that don't add up — what concept explains it?
A: Floating-point representation: JSON numbers parsed as IEEE-754 floats can't store many decimals exactly, so sums drift; use decimal types or string-encoded amounts for money.

Q: A JSON parse fails with "Expecting property name enclosed in double quotes" — what's the likely mistake?
A: The JSON uses single quotes, has a trailing comma, or includes a comment — all invalid in standard JSON; fix the source to strict JSON or use a tolerant variant deliberately.

Q: Your XPath query against a valid XML file returns nothing despite the element existing — what's the usual culprit?
A: An unhandled namespace: the elements are namespace-qualified, so you must register the namespace prefix/URI and query the qualified name rather than the bare local name.

Q: A column of identifiers loaded from CSV lost its leading zeros — what should you have done on read?
A: Forced the column to a string/text type on import (e.g. `dtype=str` in pandas, or "Text" column type in Excel's import wizard) so numeric coercion never strips the zeros.

Q: Bytes look correct in a hex dump but the displayed text is garbled — bytes problem or interpretation problem?
A: An interpretation problem: the data on disk is fine but is being decoded with the wrong encoding, so choose the correct encoding rather than trying to "repair" the bytes.

Q: As a Data Architect, what is the single most reliable defence against the whole family of "what data physically is" bugs?
A: Make structure explicit at boundaries — pin the encoding (UTF-8), the dialect/delimiter, and a declared schema on every interface, validate on ingestion, and prefer self-describing typed formats over guessing from untyped text.
