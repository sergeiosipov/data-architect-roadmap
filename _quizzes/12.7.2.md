# Phase 5 · 12.7.2 Observability stack (OpenTelemetry, Prometheus, Grafana) — 100 self-test questions

Q: What are the "three pillars" of observability?
A: The three pillars are metrics, logs, and traces — sometimes called the three core telemetry signals. Metrics are aggregated numeric measurements over time (e.g. rows processed per minute), logs are timestamped text records of discrete events, and traces follow a single request or run as it flows across components. Together they let you answer "is it healthy?" (metrics), "what happened?" (logs), and "where did it happen?" (traces) without a forensic dig.

Q: What is OpenTelemetry (OTel) and what problem does it solve?
A: OpenTelemetry is a CNCF-hosted, vendor-neutral standard and toolkit for generating, collecting, and exporting telemetry — traces, metrics, and logs — from applications. It solves vendor lock-in: you instrument once against the OTel API and can ship to any backend (Prometheus, Grafana, Datadog, Azure Monitor) by swapping the exporter, not the code. For a fund platform that may move from a FOSS stack to a managed one, this means the NAV-pipeline instrumentation survives the migration.

Q: What does the acronym OTLP stand for and what is it?
A: OTLP is the OpenTelemetry Protocol — the wire format and transport that OTel components use to send traces, metrics, and logs to each other and to backends. It is the standardised, vendor-neutral protocol an SDK uses to push telemetry to the Collector, and that the Collector uses to push onward to a backend. Using OTLP end-to-end is what makes the pipeline portable across observability vendors.

Q: What is the OpenTelemetry Collector and why use it instead of exporting straight to a backend?
A: The Collector is a standalone proxy that receives telemetry, processes it (batching, filtering, attribute enrichment, sampling), and exports it to one or more backends. Putting it between your app and the backend decouples instrumentation from destination, so you can change vendors, fan out to several backends, or add a sampling policy without redeploying the pipeline. It also offloads buffering and retry from your Python process, which matters when a NAV run must not block on a slow telemetry endpoint.

Q: What are the three stages of an OTel Collector pipeline?
A: A Collector pipeline is built from receivers, processors, and exporters, wired together per signal type in the `service.pipelines` section of the config. Receivers ingest data (e.g. the `otlp` receiver), processors transform it (e.g. `batch`, `memory_limiter`), and exporters send it onward (e.g. `prometheus`, `otlp`, `debug`). You can define separate `traces`, `metrics`, and `logs` pipelines, each chaining its own receivers/processors/exporters.

Q: What are the two default OTLP network ports and which transport does each use?
A: Port `4317` is the default for OTLP over gRPC, and port `4318` is the default for OTLP over HTTP. The OTel Collector's `otlp` receiver listens on both by default, and SDK exporters default to `http://localhost:4317` (gRPC) or `http://localhost:4318` (HTTP). If your spans never arrive, confirming the SDK and Collector agree on port and protocol is the first thing to check.

Q: In the OTel signals model, what is a "span"?
A: A span is the basic unit of a trace — it represents a single named, timed operation, such as "read EMT file" or "write Iceberg snapshot". Each span has a start time, duration, status, and a set of attributes (key/value metadata), and spans are linked by parent/child relationships to form the trace tree. A NAV run instrumented end-to-end produces one root span with child spans for each stage.

Q: What is a "trace" and how does it relate to spans?
A: A trace is the full tree of spans that together describe one end-to-end execution — for example a single Dagster→Spark→Iceberg pipeline run. All spans in a trace share the same `trace_id`, and each span carries a `span_id` plus the `span_id` of its parent, which is how the tree is reconstructed. The trace is what lets you see exactly where a NAV pipeline's latency or failure occurred across components.

Q: What is "context propagation" in OTel and why is it essential for distributed traces?
A: Context propagation is the mechanism that carries the trace context (notably `trace_id` and the current `span_id`) across process and service boundaries so child spans attach to the right parent. Without it, each service would start its own disconnected trace and you could never see a single run spanning Dagster, Spark, and Iceberg. OTel propagates context via standards like W3C Trace Context (the `traceparent` HTTP header) so the trace stays intact across the hops.

Q: What is the W3C `traceparent` header and what does it carry?
A: `traceparent` is the standard HTTP header defined by W3C Trace Context that carries the propagated trace context between services. It encodes the version, the `trace-id`, the parent `span-id` (the calling span), and trace flags such as the sampled bit, in a single dash-delimited string. OTel uses it by default so that an HTTP call from one instrumented service continues the same trace rather than starting a new one.

Q: What is a "metric" in the OTel signals model and how does it differ from a log?
A: A metric is a measurement captured as a numeric value, typically aggregated over time — e.g. a counter of rows processed or a gauge of consumer lag. It differs from a log in that a metric is pre-aggregated and cheap to store and query at scale, whereas a log is a discrete, high-cardinality text event. You watch metrics on a dashboard for trends and alerting, and drill into logs only when a metric tells you something is wrong.

Q: What is a "resource" in OpenTelemetry?
A: A resource is the set of attributes that describe the entity producing telemetry — for example `service.name`, `service.version`, host, and deployment environment. These attributes are attached to every span, metric, and log from that process, so you can filter all signals by the service that emitted them. Setting a clear `service.name` like `nav-pipeline` is the first thing to get right, otherwise everything shows up as `unknown_service`.

Q: What is the difference between the OTel API and the OTel SDK in Python?
A: The API (`opentelemetry-api`) defines the interfaces you call to create spans and record metrics; it is a no-op by default and adds almost nothing if unconfigured. The SDK (`opentelemetry-sdk`) is the concrete implementation that actually samples, batches, and exports telemetry once you wire up providers and exporters. Libraries instrument against the API so they don't force an SDK on consumers, and the application chooses and configures the SDK at startup.

Q: What is "auto-instrumentation" in OTel Python and how is it typically invoked?
A: Auto-instrumentation wraps common libraries (HTTP clients, DB drivers, frameworks) to emit spans without you editing their call sites, using the `opentelemetry-instrument` command to launch your program. You install `opentelemetry-distro` and run `opentelemetry-bootstrap -a install`, then run `opentelemetry-instrument python my_pipeline.py`. It is the fastest way to get baseline traces, though pipeline-specific spans like "compute NAV" still need manual instrumentation.

Q: Which OTel Python packages give you the API, SDK, and OTLP exporter?
A: The core trio is `opentelemetry-api`, `opentelemetry-sdk`, and `opentelemetry-exporter-otlp` (which pulls in the gRPC and HTTP OTLP exporters). With `uv`, you would add them via `uv add opentelemetry-api opentelemetry-sdk opentelemetry-exporter-otlp`. The exporter package is what actually ships your spans and metrics over OTLP to the Collector.

Q: In OTel Python, what is a `TracerProvider` and why must you set one?
A: The `TracerProvider` is the SDK factory that creates tracers and holds the configuration for how spans are processed and exported (span processors, exporters, sampler, resource). You set it once at startup with `trace.set_tracer_provider(...)`; without it, the API returns no-op tracers and no spans are ever exported. It is the central object you attach a `BatchSpanProcessor` and OTLP exporter to.

Q: What does a `BatchSpanProcessor` do and why prefer it over a simple processor in production?
A: A `BatchSpanProcessor` queues finished spans and exports them in batches on a background thread, rather than exporting each span synchronously as it ends. This reduces export overhead and keeps your pipeline's hot path from blocking on the network, which matters when a NAV run is racing a cut-off. The `SimpleSpanProcessor` exports one span at a time and is fine for local debugging but too slow for production volumes.

Q: How would you manually create a span around a stage in a Python pipeline?
A: You get a tracer and open a span as a context manager, e.g. `tracer = trace.get_tracer(__name__)` then `with tracer.start_as_current_span("load_emt_file") as span:`. Code inside the `with` block runs within the span, the span ends automatically on block exit, and you can call `span.set_attribute("rows", n)` to record metadata. Using `start_as_current_span` also makes it the active parent for any nested spans.

Q: How do you attach contextual metadata to a span, and what is a good example for a NAV run?
A: You call `span.set_attribute(key, value)` to add key/value pairs, using stable keys and primitive values. For a NAV run you might set `span.set_attribute("fund.isin", "LU0123456789")`, `span.set_attribute("nav.rows", 4200)`, and `span.set_attribute("emt.file", "EMT_20260613.csv")`. These attributes are then queryable in the trace backend, so you can find the exact run for a given fund or file.

Q: How do you record that a span failed, and why is just catching the exception not enough?
A: You set the span status to error and record the exception, typically with `span.set_status(Status(StatusCode.ERROR))` and `span.record_exception(exc)` inside the `except` block. Just catching the exception in Python leaves the span marked OK, so the trace would falsely look healthy and the failure would not surface on the dashboard. Recording it makes the failing span visibly red in the trace view with the stack trace attached.

Q: What is the difference between an OTel counter and a gauge instrument?
A: A counter is a monotonically increasing value you add to — ideal for "rows processed" or "runs completed", where you care about totals and rates. A gauge (or observable gauge) records a value that can go up or down at the moment of observation — ideal for "current consumer lag" or "queue depth". Choosing the wrong instrument type breaks your math: applying `rate()` to a gauge or treating a counter as an absolute level both mislead.

Q: For a Dagster→Spark→Iceberg run, which three pipeline metrics does the lesson call out, and what instrument fits each?
A: The three are rows processed (a counter, since it only accumulates), run duration (a histogram, so you can see the distribution and percentiles), and lag (a gauge, since it rises and falls). Modelling them with the right instrument lets you alert on "rows dropped to zero", chart "p95 run duration creeping up", and "lag exceeding the cut-off budget" respectively. These map directly to the data-freshness and consumer-lag alert rules the lesson asks you to build.

Q: How do you make a single trace span the full Dagster→Spark→Iceberg run rather than three disconnected traces?
A: You start a root span at the start of the run and ensure the trace context is propagated into each stage so Spark and Iceberg work attach as child spans under the same `trace_id`. In practice this means setting up the SDK once, opening the root span in the Dagster op/asset, and propagating context (via OTel's context APIs or instrumentation) into the Spark job. The Done-when criterion — one trace spanning the whole run — fails the moment a stage starts its own provider and breaks propagation.

Q: A pipeline emits spans but they appear as separate traces per stage — what is the likely cause?
A: The trace context is not being propagated across the stage boundary, so each stage starts a fresh root span with a new `trace_id`. Common causes are spawning a subprocess or remote Spark job without carrying the context, or re-initialising a `TracerProvider` inside the child. The fix is to propagate the active context (e.g. inject/extract the W3C `traceparent`) into the downstream stage so its spans share the parent's `trace_id`.

Q: What is the difference between OTel logs and metrics, and why might you still send logs to Loki rather than through OTel?
A: OTel logs are structured log records carried over OTLP and correlatable with traces via `trace_id`; metrics are aggregated numbers. Many teams still ship logs to Loki because Loki's label-indexed model and LogQL are purpose-built for high-volume log search and are cheap to operate. A common hybrid is OTel for traces/metrics and an agent shipping logs to Loki, with the `trace_id` embedded in log lines so you can pivot from a log to its trace.

Q: What does it mean that Prometheus uses a "pull" (scrape) model?
A: Prometheus periodically fetches metrics by making an HTTP GET to each target's metrics endpoint, rather than having targets push to it. This pull model gives Prometheus control over scrape frequency and makes a target's health observable (a failed scrape is itself a signal), but it requires Prometheus to be able to reach every target. Short-lived jobs that die between scrapes are the classic gap, which is why a Pushgateway exists for them.

Q: What is the conventional HTTP path Prometheus scrapes, and what format does it expect?
A: By convention targets expose metrics at `/metrics`, returning the Prometheus text exposition format — plain-text lines of `metric_name{labels} value`. Prometheus scrapes that endpoint on the configured `scrape_interval` and parses each line into a time series. Your Python pipeline either exposes its own `/metrics` via a client library or, more commonly here, the OTel Collector's `prometheus` exporter exposes the metrics for scraping.

Q: What is the default port for the Prometheus server's own web UI and API?
A: Prometheus listens on port `9090` by default for its web UI, HTTP API, and its own `/metrics` endpoint. You reach the expression browser at `http://localhost:9090` to run PromQL interactively. Note this is distinct from the ports your targets expose; `9090` is the Prometheus server itself.

Q: In a `prometheus.yml`, what does a `scrape_config` define?
A: A `scrape_config` defines a set of targets to scrape and how — the `job_name`, the target addresses (statically via `static_configs` or via service discovery), the metrics path, and any scrape-interval or relabeling overrides. Each job groups related targets under one logical name that becomes the `job` label on every scraped series. Getting the target address and port right here is the usual first fix when "no data" appears.

Q: What is a Prometheus "counter" and which PromQL function turns it into a per-second rate?
A: A counter is a cumulative metric that only increases (and resets to zero on restart), such as total rows processed. You almost never graph a raw counter; instead you apply `rate(metric[5m])` to get the per-second average rate over a window, which also correctly handles counter resets. So `rate(rows_processed_total[5m])` gives throughput rather than an ever-climbing line.

Q: What is the difference between `rate()` and `increase()` in PromQL?
A: `rate(v[5m])` returns the per-second average rate of increase of a counter over the window, while `increase(v[5m])` returns the total increase over that window (essentially `rate × window-seconds`). Use `rate` for throughput-style graphs and alerting thresholds in per-second terms, and `increase` when you want "how many rows in the last hour". Both expect a counter and a range-vector selector like `[5m]`.

Q: Why should you never apply `rate()` to a gauge in PromQL?
A: `rate()` assumes a monotonically increasing counter and interprets any decrease as a counter reset, so applying it to a gauge (which legitimately goes up and down) produces nonsense. For a gauge like consumer lag you query the value directly or use functions such as `avg_over_time`, `max_over_time`, or `deriv`. Mixing up instrument types is a classic PromQL bug that yields plausible-looking but wrong graphs.

Q: What is a Prometheus "histogram" and how do you compute a p99 from it?
A: A histogram observes values into cumulative buckets and exposes `_bucket{le="..."}`, `_sum`, and `_count` series; it is how you track distributions like run duration. You compute a percentile with `histogram_quantile(0.99, rate(run_duration_seconds_bucket[5m]))`, which interpolates the 99th percentile from the bucketed counts. The accuracy depends on having buckets placed around the latencies you care about.

Q: What do the `_sum` and `_count` series of a Prometheus histogram let you compute cheaply?
A: Dividing `rate(metric_sum[5m]) / rate(metric_count[5m])` gives the average observed value over the window — e.g. average run duration. The `_count` series also gives you the throughput of observations, and `_sum` the total magnitude. This average is cheaper and exact (unlike quantiles, which are bucket-interpolated), so it is a good companion metric on a dashboard.

Q: What is a "range vector" versus an "instant vector" in PromQL?
A: An instant vector is a set of time series with a single sample per series at the query time (e.g. `rows_processed_total`), while a range vector selects a window of samples per series using a duration like `rows_processed_total[5m]`. Functions like `rate`, `increase`, and `_over_time` require a range vector, whereas arithmetic and most aggregations operate on instant vectors. Passing the wrong one is a common "expected type range vector" error.

Q: How does PromQL `sum by (label)` aggregation work, and give a fund-data example?
A: `sum by (label) (expr)` collapses series, summing across all label combinations except the ones you list, keeping a separate result per value of `label`. For example `sum by (fund) (rate(rows_processed_total[5m]))` gives per-fund throughput aggregated across pipeline instances. The complementary `without (label)` form keeps all labels except the ones named.

Q: What is a Prometheus alerting rule and where is it evaluated?
A: An alerting rule is a PromQL expression with a threshold and a `for` duration, defined in a rule file loaded by Prometheus; when the expression has been true for the `for` window, the rule fires. Prometheus evaluates rules on its own `evaluation_interval` and pushes firing alerts to Alertmanager for routing. Example: an alert named `ConsumerLagHigh` with `expr: consumer_lag_seconds > 300` and `for: 5m`.

Q: What does the `for:` field do in a Prometheus alerting rule?
A: `for:` requires the alert expression to remain true continuously for that duration before the alert transitions from "pending" to "firing". It suppresses flapping on brief spikes — a 10-second lag blip won't page anyone if `for: 5m`. This is how you encode "sustained, not transient" into a data-freshness or consumer-lag alert.

Q: What is Alertmanager and how does it relate to Prometheus?
A: Alertmanager is a separate component that receives firing alerts from Prometheus and handles deduplication, grouping, silencing, inhibition, and routing to receivers (email, Slack, PagerDuty, webhook). Prometheus only decides whether an alert is firing; Alertmanager decides who gets told and how. Separating them keeps notification policy out of the metrics engine.

Q: What is the difference between a Prometheus recording rule and an alerting rule?
A: A recording rule precomputes a PromQL expression on a schedule and saves the result as a new time series, speeding up dashboards and reusing expensive expressions; an alerting rule evaluates an expression and fires an alert when a condition holds. Recording rules are about performance and reuse, alerting rules are about notification. Both live in rule files but serve different purposes.

Q: What is Grafana's role in this stack?
A: Grafana is the visualization and dashboarding layer — it queries data sources like Prometheus and Loki and renders panels, and it also has its own alerting engine. It does not store the telemetry itself; it reads from the stores and presents the platform-health view. In this lesson it is where you build the dashboard that lets you diagnose a seeded failure in under five minutes.

Q: How is a Prometheus data source connected to Grafana?
A: You add Prometheus as a data source in Grafana by pointing it at the Prometheus server URL (e.g. `http://prometheus:9090`), after which panels can run PromQL against it. Grafana then proxies or direct-queries that endpoint when a panel renders. If panels show "No data", confirming the data-source URL and that Prometheus is actually scraping is the first check.

Q: What is a Grafana panel versus a dashboard?
A: A panel is a single visualization — a graph, stat, table, or gauge — backed by one or more queries against a data source. A dashboard is a collection of panels laid out on one page, often with variables, that together tell a story such as "platform health". The lesson's deliverable is a platform-health dashboard whose panels cover freshness, lag, throughput, and run duration.

Q: What is a Grafana dashboard variable and why is it useful for a multi-fund platform?
A: A dashboard variable is a parameter (e.g. `$fund` or `$env`) that you select from a dropdown, and panels reference it in their queries to filter dynamically. For a multi-fund platform you can template `$fund` so one dashboard serves every fund's NAV pipeline, switching the ISIN filter without editing panels. Variables can be populated from a query (e.g. `label_values(fund)`) so the list stays current.

Q: What is Grafana Loki and how does its indexing differ from a traditional log system?
A: Loki is a horizontally scalable, multi-tenant log aggregation system inspired by Prometheus, designed to be cheap to run. The key difference is that Loki indexes only a small set of labels (metadata) per log stream, not the full log contents, which keeps storage and indexing costs far lower. You filter to a stream by labels first, then grep within it — trading rich full-text indexing for operational simplicity.

Q: What is LogQL and how does it relate to PromQL?
A: LogQL is Loki's query language, designed to feel like PromQL: a query is a log-stream selector in curly braces (label matchers) optionally followed by a filter pipeline. For example `{service="nav-pipeline"} |= "ERROR"` selects the stream and keeps lines containing "ERROR". LogQL also has a metric-query form that turns log matches into rates and aggregations, e.g. counting errors per minute.

Q: What does the LogQL expression `{job="nav"} |= "NAV cut-off missed"` do?
A: It selects all log streams labelled `job="nav"` and then keeps only the lines containing the literal substring `NAV cut-off missed`. The `{...}` part is the stream selector matched against indexed labels, and `|=` is a line filter applied to the log content. This is the typical way to jump from a fired freshness alert to the exact log lines explaining the missed cut-off.

Q: Why is putting high-cardinality values (like an ISIN) into a Loki label a known anti-pattern?
A: Each unique combination of label values creates a separate Loki stream, so a label with thousands of ISINs explodes the number of streams and destroys Loki's performance and storage efficiency. The guidance is to keep labels low-cardinality (e.g. `service`, `env`, `level`) and put high-cardinality identifiers inside the log line, then filter on them with `|=` or a parser. The same cardinality discipline applies to Prometheus labels.

Q: As of 2026, what is the recommended log collector for Loki, given Promtail's status?
A: Promtail reached end-of-life in March 2026 and is superseded by Grafana Alloy, a Grafana distribution of the OpenTelemetry Collector that is OTLP-compatible and handles metrics, logs, and traces. New deployments should use Alloy rather than Promtail, and Grafana ships a migration tool to convert existing Promtail configs. On a fresh Ubuntu build today you would install Alloy, not Promtail.

Q: What is Grafana Alloy and why does it matter that it is an OTel Collector distribution?
A: Alloy is Grafana's open-source telemetry collector built on the OpenTelemetry Collector, supporting OTLP plus native Prometheus and Loki pipelines for metrics, logs, traces, and profiles. Because it is OTLP-native, it can act as your OTel Collector and your Loki log shipper in one agent, simplifying the stack. It replaces both the older Grafana Agent and Promtail.

Q: What is a Grafana "contact point" in its alerting model?
A: A contact point defines where and how a notification is sent — the integration (email, Slack, PagerDuty, webhook, etc.) and its settings — when an alert fires. Notification policies route firing alerts to the appropriate contact point based on labels. So the contact point answers "send it here", and the notification policy answers "for alerts like this".

Q: How does a Grafana notification policy decide which contact point an alert goes to?
A: A notification policy is a tree that matches alert labels against routing rules; the matching branch determines the contact point and timing settings (grouping, repeat interval). For example, alerts with `severity=critical` and `team=data-platform` route to a PagerDuty contact point, while warnings go to email. This label-based routing is how you keep a NAV-freshness page going to the on-call rather than a shared inbox.

Q: What is the difference between Grafana-managed alerts and data-source-managed (Prometheus) alerts?
A: Grafana-managed alerts are defined and evaluated inside Grafana against any data source, with state and notifications handled by Grafana's own alerting engine. Data-source-managed alerts are the rules defined in Prometheus rule files, evaluated by Prometheus and routed by Alertmanager, which Grafana can display but not own. Choosing one keeps your alerting logic in a single place rather than split across two engines.

Q: What does SLI stand for and what is it?
A: SLI stands for Service Level Indicator — a carefully chosen quantitative measure of some aspect of service quality, such as the fraction of NAV runs that finish before the cut-off. It is the metric you actually compute from telemetry, typically expressed as a ratio of good events to total events. The SLI is the raw number that an SLO then sets a target on.

Q: What does SLO stand for and how does it relate to an SLI?
A: SLO stands for Service Level Objective — a target value or range for an SLI over a window, e.g. "99% of NAV runs complete before cut-off over a rolling 30 days". The SLI is the measurement; the SLO is the goal you hold it to. Breaching the SLO is what should trigger action, and ideally an alert, rather than alerting on every transient blip.

Q: What is an "error budget" and how is it derived from an SLO?
A: An error budget is the allowed amount of failure implied by an SLO over its window — `1 - SLO`. A 99.9% availability SLO over 30 days gives roughly 43 minutes of budget; once consumed, you slow risky changes until reliability recovers. It turns "is it reliable enough?" into a number you can spend and track rather than an argument.

Q: How does an SLA differ from an SLO, especially in a regulated fund context?
A: An SLA (Service Level Agreement) is a contractual commitment to a customer or regulator with consequences (penalties, remediation) for breach, whereas an SLO is an internal target you set tighter than the SLA. In fund administration, the SLA might be "NAV delivered by the contractual cut-off", and the internal SLO is stricter so you catch trouble before the SLA is breached. Phase 8 builds on this vocabulary for formal data SLAs.

Q: What is the "data freshness" alert in this lesson actually measuring?
A: Data freshness measures how stale the data is — typically the time since the last successful pipeline run or the lag between event time and ingestion time. The alert fires when that staleness exceeds a budget tied to the NAV cut-off, signalling that downstream consumers are about to see (or are seeing) out-of-date data. It is the early warning that a cut-off is at risk.

Q: How would you express a data-freshness condition in PromQL using a "last success" timestamp gauge?
A: If your pipeline emits a gauge of the last successful run as a Unix timestamp, freshness is `time() - last_success_timestamp_seconds`, and you alert when that exceeds your budget, e.g. `> 1800` for 30 minutes. `time()` returns the evaluation timestamp, so the difference is the age of the freshest data. Wrapping it in an alerting rule with a sensible `for:` avoids paging on a single late scrape.

Q: What is "consumer lag" in a streaming sense and why alert on it?
A: Consumer lag is how far behind a consumer is from the latest available data — in Kafka terms, the difference between the latest offset and the consumer's committed offset, often expressed in records or seconds. Rising lag means the consumer can't keep up and downstream freshness will degrade, so it is a leading indicator of a missed cut-off. Alerting on it lets you intervene before freshness actually breaches.

Q: Why is consumer lag better modelled as a gauge than a counter?
A: Lag goes up when the consumer falls behind and down when it catches up, so it is an instantaneous level, not a cumulative total — exactly what a gauge represents. Modelling it as a counter and applying `rate()` would misread every catch-up as a counter reset and yield garbage. As a gauge you alert directly, e.g. `consumer_lag_seconds > 300`.

Q: The lesson asks you to "seed a failure and confirm it shows up with the trace attached" — what does success look like?
A: Success is that you deliberately break a stage (e.g. make the Iceberg write fail), the corresponding alert fires on the dashboard, and clicking through takes you to the failing trace with the error span marked red and the exception recorded. This proves the instrumentation, metrics, alerting, and trace correlation all work together. The Done-when bar is diagnosing it from the dashboard alone in under five minutes.

Q: What is "exemplar" support and how does it help you jump from a metric to a trace?
A: An exemplar is a sample trace ID attached to a specific metric observation (commonly a histogram bucket), letting a dashboard offer a direct link from a latency spike to the exact trace that produced it. In Grafana, exemplars appear as clickable points on a graph that open the corresponding trace in Tempo or another tracing backend. This is the mechanism behind "click the spike, see the trace" that makes five-minute diagnosis realistic.

Q: What is Grafana Tempo and where does it fit relative to Prometheus and Loki?
A: Tempo is Grafana's distributed tracing backend — the store for traces, complementing Prometheus (metrics) and Loki (logs). It receives spans (e.g. over OTLP) and lets Grafana display full traces and correlate them with metrics and logs via shared IDs. The common Grafana "LGTM" stack is Loki, Grafana, Tempo, and Mimir.

Q: How do you correlate a log line in Loki with a trace in Tempo?
A: You ensure each relevant log line includes the `trace_id`, and you configure a Grafana "derived field" on the Loki data source that turns that ID into a link to Tempo. Clicking the ID in the log jumps straight to the trace, closing the loop from "what happened" (log) to "where" (trace). This requires that your instrumentation injects the active `trace_id` into the log record.

Q: Why is structured (e.g. JSON) logging important when correlating logs with traces?
A: Structured logs let you reliably extract fields like `trace_id`, `level`, and `fund` with a parser rather than fragile regex, enabling correlation and label-based filtering. With JSON logs, LogQL's `| json` parser exposes those fields for filtering and metric extraction. Unstructured free-text logs make the trace-to-log pivot brittle and slow.

Q: What is the difference between a span attribute and a span event in OTel?
A: A span attribute is a key/value pair describing the whole span (e.g. `fund.isin`), set with `span.set_attribute`, whereas a span event is a timestamped annotation marking a point in time within the span, added with `span.add_event("retry attempted")`. Attributes answer "what is this operation about"; events answer "what happened at moment X during it". A recorded exception is in fact stored as a special span event, which is why it shows up on the timeline.

Q: Why does a Prometheus histogram always include a bucket with `le="+Inf"`, and what does it equal?
A: Buckets are cumulative — each `_bucket{le="x"}` counts all observations less than or equal to `x` — so the `le="+Inf"` bucket counts every observation and therefore equals the histogram's `_count`. `histogram_quantile` relies on these cumulative counts to interpolate percentiles, and the `+Inf` bucket anchors the top of the distribution. If your buckets top out below your real latencies, everything piles into `+Inf` and high quantiles become inaccurate.

Q: What is Azure Monitor and how does it map to this open-source stack?
A: Azure Monitor is Microsoft's managed observability platform — it ingests metrics and logs from Azure resources and applications and provides dashboards, alerting, and querying. It is the corporate-stack analogue of the Prometheus+Grafana+Loki combination: managed, integrated, and billed, versus FOSS that you run yourself. Knowing both lets the architect reason about the build-vs-buy line.

Q: What is Application Insights and how does it relate to OpenTelemetry?
A: Application Insights is Azure Monitor's application performance monitoring (APM) feature for traces, requests, dependencies, and exceptions. Microsoft now recommends instrumenting with OpenTelemetry and exporting to Application Insights via the Azure Monitor OTel distro, so your OTel instrumentation is portable between the FOSS stack and Azure. This is precisely why instrumenting against the vendor-neutral OTel API pays off.

Q: What is KQL and where does it appear in this corporate mapping?
A: KQL is the Kusto Query Language, the query language for Azure Monitor Logs (Log Analytics) and Application Insights. It is the Azure analogue of PromQL/LogQL for slicing logs and metrics, and the lesson ties it back to appendix A.5. An architect fluent in both can move a query intent between the FOSS and Azure worlds.

Q: What is the build-vs-buy trade-off between running Prometheus/Grafana/Loki yourself and using a managed service like Datadog or Azure Monitor?
A: Self-hosting the FOSS stack avoids per-host/per-GB SaaS bills and keeps data in your environment (relevant for EU residency), but you own the operational burden of scaling, retention, and upgrades. Managed services like Datadog or Azure Monitor remove that toil and add polish, at a cost that grows with telemetry volume and can surprise you. The architect's job is to place the line deliberately, not by default.

Q: Why does EU data residency matter when choosing an observability backend for a Luxembourg fund administrator?
A: Telemetry can contain personal data and operational detail that, under GDPR and sector rules, may need to stay within the EU; a US-region SaaS backend could violate residency requirements. A self-hosted EU-region FOSS stack, or a managed service with an EU region, keeps logs and traces compliant. This is a real constraint that can tip a build-vs-buy decision regardless of cost.

Q: How does DORA make observability a regulatory concern rather than just good engineering?
A: DORA (the EU Digital Operational Resilience Act) requires financial entities to detect, manage, and report ICT-related incidents, which presupposes that you can actually see the state of your systems. Without metrics, logs, and traces you cannot evidence detection times or root cause within mandated reporting windows. So an observability stack is part of demonstrating operational resilience to a regulator, not a nice-to-have.

Q: Your OTel-instrumented Python pipeline runs but no spans reach the backend — what are the first checks?
A: Confirm the SDK is actually configured (a `TracerProvider` set with an exporter and span processor, not the default no-op), then check the exporter endpoint and port match the Collector (`4317` gRPC or `4318` HTTP). Next verify the Collector is up and its `otlp` receiver is listening, and look at the Collector logs or a `debug` exporter to see if data arrives. A `BatchSpanProcessor` also flushes on shutdown, so a hard kill can lose the last batch.

Q: Spans appear in the Collector but never reach the backend — where do you look?
A: This isolates the problem to the Collector's exporter half: check that the exporter is wired into the relevant pipeline under `service.pipelines`, that its endpoint and credentials are correct, and read the Collector logs for export errors or retries. Adding a `debug` exporter alongside confirms the data is in the pipeline, proving the issue is the downstream exporter, not the receiver. A frequent cause is the exporter defined but not listed in the pipeline.

Q: Prometheus shows a target as "down" — what does that mean and what do you check first?
A: "Down" means Prometheus could not successfully scrape that target on the last attempt, and `up{job="..."}` equals `0` for it. First check network reachability and the exact target `address:port` and metrics path in the scrape config, then curl the `/metrics` endpoint yourself from where Prometheus runs. The Targets page in the Prometheus UI shows the last error, which usually names the cause (connection refused, 404, TLS).

Q: A Grafana panel shows "No data" but Prometheus has the metric — what is the usual cause?
A: Usually the panel's PromQL doesn't match any series — a typo in the metric or label name, a label filter that matches nothing, or the wrong data source selected on the panel. Test the exact query in Prometheus's expression browser first; if it returns data there but not in Grafana, the data source or time range is the culprit. Time range mismatches (querying a window with no data) are an easy thing to overlook.

Q: An alert never fires even though the condition looks true on the graph — what would you check?
A: Check that the `for:` duration hasn't been outlasting the condition (the spike is real but too brief), that the rule file is actually loaded by Prometheus or the rule enabled in Grafana, and that the alert expression evaluates to a non-empty vector when the condition holds. Also verify the alert is wired to a notification policy and contact point if "fire" means "notify". The Prometheus `/alerts` page or Grafana's alert state view shows pending-versus-firing status.

Q: An alert fires but nobody is notified — where is the break?
A: The break is in the notification path, not the rule: in Prometheus that means Alertmanager isn't receiving or routing the alert (check Prometheus's Alertmanager config and Alertmanager's routing tree and receivers), and in Grafana it means no notification policy matches the alert's labels or the contact point is misconfigured. Send a test notification to the contact point to isolate whether the integration itself works. Silences or inhibition rules can also legitimately suppress it.

Q: What is "cardinality" in a metrics context and why is high cardinality dangerous?
A: Cardinality is the number of distinct time series, which multiplies with every unique combination of label values. Putting a high-cardinality value like ISIN or run-ID into a Prometheus label can create millions of series, blowing up memory and query time. The discipline is to keep labels low-cardinality and push unique identifiers into traces or logs instead.

Q: Why is it wrong to put a NAV run's unique run-ID as a Prometheus metric label?
A: Every distinct run-ID creates a new, never-reused time series, so the metric's cardinality grows without bound and Prometheus's memory and index suffer (a "cardinality explosion"). Run-level identity belongs in a trace attribute or a log field, where high cardinality is expected and cheap. Metrics should carry only stable, bounded labels like `fund`, `env`, and `stage`.

Q: What is "sampling" in tracing and why might you sample traces but not metrics?
A: Sampling keeps only a fraction of traces to control cost and volume, since storing every span at high throughput is expensive and often redundant. Metrics are already aggregated and cheap, so you typically keep all of them, whereas traces are voluminous and individually large. The trade-off is that aggressive sampling can drop the very trace you needed, so tail-based sampling (keep all error/slow traces) is often preferred for pipelines.

Q: What is the difference between head-based and tail-based sampling?
A: Head-based sampling decides whether to keep a trace at its start, before the outcome is known, using a fixed probability — simple but blind to errors. Tail-based sampling buffers spans and decides after the trace completes, so you can keep all traces that errored or exceeded a latency threshold. For a NAV pipeline you usually want tail-based so every failed or slow run is retained for diagnosis.

Q: Why might you choose OTLP/HTTP over OTLP/gRPC for exporting telemetry?
A: OTLP/HTTP (port `4318`) is easier to route through proxies, load balancers, and firewalls that don't speak gRPC, and avoids gRPC client dependencies. OTLP/gRPC (port `4317`) is typically more efficient with streaming and bidirectional flow control. In constrained or proxied corporate networks, HTTP is often the pragmatic choice even if gRPC is marginally faster.

Q: What is the Prometheus `up` metric and why is it useful?
A: `up` is a synthetic metric Prometheus generates for every scrape: `1` if the target scraped successfully, `0` if not. It is the simplest health signal for your scrape targets and the basis of a "target down" alert like `up == 0`. Because it is always present, it is a reliable canary that doesn't depend on the target exposing any other metric.

Q: What does `histogram_quantile(0.95, sum by (le) (rate(run_duration_seconds_bucket[5m])))` compute, and why the `sum by (le)`?
A: It computes the 95th-percentile run duration over the last 5 minutes aggregated across instances. The `sum by (le)` aggregates the bucket counts across label dimensions while preserving the `le` (bucket boundary) label, because `histogram_quantile` needs the per-bucket counts to interpolate. Forgetting to keep `le` (or summing it away) breaks the quantile calculation.

Q: What is the difference between a Prometheus summary and a histogram, and why prefer histograms here?
A: A summary computes quantiles on the client and exposes them directly, so you cannot aggregate those quantiles across instances; a histogram exposes buckets and computes quantiles at query time, which aggregates correctly. For a distributed Dagster→Spark pipeline with many workers, histograms let you compute a fleet-wide p99, whereas summary quantiles can't be meaningfully combined. That aggregability is why histograms are the default recommendation.

Q: What does it mean to instrument "across" a Dagster→Spark→Iceberg run as opposed to "within" each component?
A: Instrumenting across means producing one connected trace that follows the run through all three components via context propagation, so you see the whole journey. Instrumenting within means each component has its own isolated traces that don't link up, leaving gaps at the boundaries. The lesson's Done-when criterion — a single trace spanning the full run — is specifically the "across" outcome.

Q: How can OTel metrics be exposed to Prometheus given Prometheus pulls and OTel pushes?
A: You use the Collector's `prometheus` exporter, which exposes the OTel-received metrics on an HTTP endpoint that Prometheus then scrapes, bridging push to pull. Alternatively the Collector's `prometheusremotewrite` exporter pushes directly into a Prometheus-compatible store. Either way the Collector mediates between OTel's push model and Prometheus's pull model.

Q: What is the difference between the Collector's `prometheus` exporter and `prometheusremotewrite` exporter?
A: The `prometheus` exporter exposes a `/metrics` scrape endpoint that a Prometheus server pulls from, fitting the standard pull model. The `prometheusremotewrite` exporter actively pushes metrics to a remote-write-compatible endpoint (e.g. Mimir or Prometheus's remote-write receiver), fitting a push topology. You pick based on whether your backend wants to scrape or be written to.

Q: Why is a `debug` (formerly `logging`) exporter useful when bringing up a Collector pipeline?
A: The `debug` exporter writes received telemetry to the Collector's own stdout, so you can confirm data is actually flowing into the pipeline before any backend works. It is the fastest way to answer "is the Collector receiving anything at all?" and isolate receiver problems from exporter problems. You add it to the pipeline temporarily and remove it once the real exporter is confirmed.

Q: What is the `batch` processor in the Collector and why is it recommended?
A: The `batch` processor groups telemetry into batches before export, improving compression and reducing the number of outgoing requests. Grafana and OTel docs recommend it on virtually every pipeline because it materially cuts export overhead and load on the backend. It sits in the `processors` list between receivers and exporters.

Q: What does the `memory_limiter` processor protect against?
A: The `memory_limiter` processor caps the Collector's memory use by refusing or dropping data when usage approaches configured limits, preventing the Collector from being OOM-killed under a telemetry spike. It is recommended as the first processor in a pipeline so back-pressure kicks in before memory runs out. Without it, a burst of spans from a failing pipeline can take the Collector down with it.

Q: Why might a "missed NAV cut-off" be diagnosable in minutes with this stack but take hours without it?
A: With the stack, a freshness alert fires, the dashboard shows which stage's metric degraded, and the attached trace pinpoints the failing or slow span — a guided path from symptom to cause. Without it, you start from raw scattered logs and reconstruct the timeline by hand, which is the "archaeology" the lesson warns against. The five-minute Done-when target is only realistic with metrics, traces, and logs correlated.

Q: What telemetry would you attach to a span to make a transfer-agency subscription/redemption run debuggable?
A: Useful attributes include `ta.run_id`, `fund.isin`, `order.count`, `order.type` (subscription vs redemption), and a status, plus an error recorded on failure. These let you filter traces to a specific fund or order type and see exactly where a batch stalled. The IDs live as span attributes (high cardinality is fine in traces), not as Prometheus labels.

Q: How would you alert that an EMT/EPT file failed to arrive before its expected time?
A: Emit a "last successful EMT ingest" timestamp gauge per fund and alert when `time() - emt_last_ingest_timestamp_seconds` exceeds the budget before the deadline. This is a data-freshness pattern: the absence of a fresh file is detected as growing staleness rather than waiting for a downstream error. Keep the per-fund dimension low enough cardinality, or aggregate, to avoid label explosion.

Q: Why include a `service.version` resource attribute on telemetry, and how does it help during an incident?
A: `service.version` tags every signal with the deployed code version, so you can correlate a spike in errors or latency with a specific release. During an incident you can confirm whether a new NAV-pipeline version coincides with the regression and decide on a rollback quickly. It turns "did the deploy break it?" into a chart you can read.

Q: What is the risk of telemetry itself becoming a single point of failure, and how do you mitigate it?
A: If the pipeline blocks on a slow or down telemetry endpoint, observability could take out the very process it's meant to watch. Mitigation is asynchronous export (the `BatchSpanProcessor` and a local Collector that buffers and retries), plus bounded queues that drop telemetry rather than back-pressuring the pipeline. The principle is that losing some spans is acceptable; missing the NAV cut-off because of telemetry is not.

Q: What is the recommended order of operations to satisfy this lesson's four "Do" steps?
A: First instrument the Dagster→Spark→Iceberg path with OTel traces and the rows/lag/duration metrics; second stand up Prometheus to scrape and Grafana to visualize; third build the platform-health dashboard with the data-freshness and consumer-lag alert rules; fourth seed a failure and confirm it surfaces with the trace attached. Each step depends on the previous, ending in a demonstrable diagnose-in-five-minutes outcome. The Done-when checks then verify the single end-to-end trace and both alert rules firing on real conditions.
