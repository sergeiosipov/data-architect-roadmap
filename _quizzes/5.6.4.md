# Phase 4 · 5.6.4 Durable execution (Temporal) — 150 self-test questions

Q: What problem does "durable execution" solve that plain application code does not?
A: Durable execution guarantees that a long-running, stateful workflow runs to completion despite crashes, restarts, and infrastructure failures, without you hand-rolling retry, timeout, and state-persistence logic. The engine persists every step so that after a process dies the workflow resumes from exactly where it stopped, with local variables and call stack intact. For fund workflows like a corporate-action or KYC process, this means a worker crash mid-run does not leave the business in an unknown half-applied state.

Q: In Temporal terminology, what is a Workflow?
A: A Workflow is your business-logic orchestration expressed as a function in a general-purpose language (Workflow-as-Code), describing the sequence of steps and their control flow. The Workflow function itself must be deterministic and never perform side effects directly; instead it issues Commands that Temporal records as Events. Temporal replays this function against its Event History to reconstruct state, which is why determinism is mandatory.

Q: In Temporal terminology, what is an Activity?
A: An Activity is a single unit of work that performs side effects — API calls, database writes, file I/O, sending a SWIFT message — the things that can fail or be slow. Activities are where all non-deterministic and external work belongs, because Temporal runs an Activity once, records its result in the Event History, and reuses that recorded result on replay rather than re-executing it. This separation is the heart of Temporal's model: deterministic Workflow code orchestrates, side-effecting Activities do the risky work.

Q: Why must Workflow code be deterministic but Activity code need not be?
A: Workflow code is re-executed (replayed) from the start every time state is reconstructed, so it must produce the identical sequence of Commands given the same Event History — otherwise the replay diverges from history and the Workflow fails. Activities are never replayed; their results are recorded once and read back from history, so an Activity is free to call the network, read the clock, or generate randomness. The mental rule is: orchestration is deterministic and replayable, side effects are isolated into Activities.

Q: What is the Temporal Event History?
A: The Event History is the durable, append-only log of Events (Workflow started, Activity scheduled, Activity completed, Timer fired, Signal received, and so on) that the Temporal Service persists for each Workflow Execution. It is the single source of truth for a Workflow's progress and the input to replay. Because every meaningful state transition is an Event, the history is what lets a brand-new Worker recreate the exact in-memory state of a Workflow after a crash.

Q: Explain event-history replay in your own words.
A: Replay is the mechanism by which a Worker reconstructs a Workflow's state: it re-runs the Workflow function from the beginning, and each time the code would have produced a Command, Temporal feeds back the already-recorded result Event instead of doing the work again. Timers that already fired do not wait again, Activities that already completed return their stored result, and so on, so the code fast-forwards deterministically to the point just before the crash and then continues live. This is why a worker can die mid-Workflow and a replacement resumes seamlessly.

Q: A Worker crashes while a Workflow is running. What happens when a new Worker picks it up?
A: The new Worker pulls the Workflow's Event History from the Temporal Service and replays the Workflow function against it, reusing every recorded Activity result and timer outcome to rebuild local variables and execution position. Once replay reaches the last recorded Event, execution continues live from that point as if nothing happened. No business step is re-done or skipped, which is exactly the guarantee you want when a corporate-action saga is interrupted halfway.

Q: Why can you not call `random.random()` or generate a UUID directly inside Workflow code?
A: A direct random call would return a different value on replay than on the original run, so the replayed code would take a different branch or produce different Commands and diverge from the recorded Event History, failing with a non-determinism error. You must instead use the SDK's deterministic randomness (a workflow-safe seeded API) or generate the value inside an Activity and pass it back. The same logic forbids any source of fresh, unrecorded variation in Workflow code.

Q: Why can you not call `datetime.now()` directly in Workflow code, and what do you use instead?
A: The system clock would return a different timestamp on replay than at first execution, breaking the determinism replay depends on, so a direct `datetime.now()` is forbidden in Workflow code. You read time through the deterministic Workflow API (for example `workflow.now()` in the Python SDK), which returns the time as recorded in the Event History so it matches on every replay. Anything genuinely needing wall-clock side effects belongs in an Activity.

Q: Why must Workflow code never make a network or database call directly?
A: A direct network or database call is a side effect whose result is not recorded as an Event, so on replay it would either run again (duplicating the effect) or return a new value that diverges from history. All such I/O must be wrapped in an Activity, whose single execution and result are recorded once and replayed deterministically. Keeping I/O out of the Workflow body is the practical rule that keeps replay safe.

Q: Why is spawning OS threads or relying on wall-clock `sleep` inside Workflow code a determinism hazard?
A: Native threads and real `sleep` introduce timing and ordering that the engine cannot record or reproduce, so replay cannot reconstruct the same interleaving and the Workflow becomes non-deterministic. Temporal provides deterministic equivalents — durable Timers via the SDK's sleep API and the SDK's own concurrency primitives — which are recorded as Events and replay correctly. The principle: every wait or concurrency decision in a Workflow must be expressed through Temporal so it lands in the Event History.

Q: What is a Worker in Temporal and where does it run?
A: A Worker is a process you run as part of your own application, embedding the Temporal SDK, that polls the Temporal Service for tasks and executes your Workflow and Activity code. Workers are stateless and horizontally scalable — the durable state lives in the Temporal Service, not the Worker — which is precisely why a Worker can be killed and replaced freely. You register your Workflow and Activity definitions with a Worker and point it at a Task Queue.

Q: What is a Task Queue and why does it matter?
A: A Task Queue is the named queue the Temporal Service uses to hand Workflow Tasks and Activity Tasks to Workers; Workers poll a specific Task Queue and only receive work routed to it. It is the routing and load-balancing mechanism that decouples the Temporal Service from your Workers, letting you scale Workers independently and isolate workloads (for example a heavy NAV-calculation queue separate from a lightweight notification queue). Workflow and Activity options specify which Task Queue to dispatch to.

Q: What is the difference between the Temporal Service (Cluster) and your application?
A: The Temporal Service is the separate backend infrastructure that persists Event History, schedules tasks, enforces timeouts, and coordinates retries; your application is the Workers and SDK code that contain your actual Workflow and Activity logic. This split is the core of the architecture: durable state and orchestration guarantees live in the Service, while your business code stays in your deployable. You operate (or buy as Temporal Cloud) the Service; you write and deploy the Workers.

Q: Name the four Activity timeout types in Temporal.
A: They are Schedule-To-Close, Start-To-Close, Schedule-To-Start, and Heartbeat. Schedule-To-Close and Start-To-Close are the two commonly set; Schedule-To-Start and Heartbeat are for specific cases. Together with the Retry Policy these timeouts give fine control over how long an Activity may take and when it is considered failed.

Q: What does the `StartToCloseTimeout` on an Activity limit?
A: `StartToCloseTimeout` caps the maximum duration of a single Activity attempt — from when a Worker starts executing it to when that attempt completes. If an attempt exceeds it, that attempt is timed out and, subject to the Retry Policy, a new attempt may be scheduled. It is the most important timeout to set because it bounds how long any one execution of your side-effecting code may run.

Q: How does `ScheduleToCloseTimeout` differ from `StartToCloseTimeout`?
A: `ScheduleToCloseTimeout` bounds the total wall-clock time for the entire Activity including all retry attempts and any time spent waiting in the queue, whereas `StartToCloseTimeout` bounds only a single attempt. So Schedule-To-Close is the overall deadline ("give up on this Activity entirely after N") while Start-To-Close is the per-try deadline. Setting Start-To-Close without Schedule-To-Close lets retries continue indefinitely as long as each attempt is fast enough.

Q: What is the `ScheduleToStartTimeout` for, and what is its default?
A: `ScheduleToStartTimeout` limits how long an Activity Task may sit in the Task Queue waiting for a Worker to pick it up, before it ever starts executing; its default is infinity. You set it when you want to detect that no Worker is available to drain the queue — for example to fail fast if your Activity Worker fleet is down — rather than letting tasks back up silently. It does not bound execution time, only the queue wait.

Q: What is an Activity Heartbeat and why use it?
A: A Heartbeat is a periodic ping from the Worker executing a long-running Activity back to the Temporal Service, signalling that the Activity is still making progress and the Worker has not crashed. If the Service does not receive a Heartbeat within the `HeartbeatTimeout`, the Activity is considered failed and may be retried, so Heartbeats let Temporal detect a dead or stuck Worker quickly instead of waiting for a long Start-To-Close timeout. Heartbeats can also carry checkpoint details so a retried attempt resumes from where the last one left off.

Q: How can a heartbeating Activity resume from where it stopped after a Worker failure?
A: When an Activity heartbeats it can include details (a checkpoint payload) that the Temporal Service stores; if that attempt fails and is retried, the new attempt receives those last-reported details and can pick up from that point rather than restarting from scratch. This is valuable for an Activity processing a large file — say a multi-megabyte EMT delivery — so a crash near the end does not force reprocessing the whole file. Without heartbeat checkpointing, each retry of an Activity starts from its initial state.

Q: What is a Retry Policy in Temporal and when does it apply by default?
A: A Retry Policy controls how a failed Activity (or Workflow) is retried: the initial backoff interval, the backoff coefficient, the maximum interval, and the maximum number of attempts. Activities are automatically given a default Retry Policy if you do not specify one, so transient failures are retried with exponential backoff out of the box. You tune it per Activity — for example fewer retries on a non-idempotent external call, more on a flaky read.

Q: What do `initialInterval`, `backoffCoefficient`, `maximumInterval`, and `maximumAttempts` control in a Retry Policy?
A: `initialInterval` is the delay before the first retry; `backoffCoefficient` multiplies the interval after each attempt to produce exponential backoff; `maximumInterval` caps how large that backoff can grow; and `maximumAttempts` limits the total number of attempts (0 meaning unlimited). Together they shape the retry curve, letting you start fast, back off under sustained failure, and eventually give up. A coefficient of 2.0 with a 1s initial interval yields roughly 1s, 2s, 4s, 8s delays up to the maximum interval.

Q: Why should Activities be idempotent in Temporal?
A: Because Temporal may retry an Activity after a timeout or a Worker crash even if the original attempt actually completed its side effect, so the same logical operation can run more than once. If the Activity is idempotent — for example keyed on an order ID so a duplicate "place subscription order" is a no-op — these retries are safe. Non-idempotent Activities risk double-charging, double-booking, or sending a duplicate SWIFT instruction, so you design idempotency keys in deliberately.

Q: What is a Signal in Temporal?
A: A Signal is a message sent into a running Workflow Execution that delivers data asynchronously and is recorded as an Event in the Event History, so it can alter the Workflow's course. Signals are fire-and-forget from the sender's perspective and durable — they survive replay because they live in history. You use a Signal to, say, deliver an external approval or a late-arriving price into a Workflow that is waiting on it.

Q: What is a Query in Temporal, and how does it differ from a Signal?
A: A Query reads the current state of a running Workflow synchronously and returns a value to the caller; unlike a Signal it must be read-only — it must not mutate Workflow state or perform side effects. Crucially Queries are not recorded in the Event History, whereas Signals are, because a Query is just inspecting state rather than changing it. You use a Query to ask "what stage is this subscription saga in right now?" without disturbing it.

Q: Why must a Query handler never mutate Workflow state or have side effects?
A: A Query runs against the replayed Workflow state purely to report it, and because Queries are not part of the Event History, any mutation a Query made would not be durably recorded and would vanish on the next replay — corrupting consistency between live state and history. Keeping Queries strictly read-only preserves the invariant that Event History is the sole source of state changes. If you need to change a running Workflow, use a Signal (or an Update), not a Query.

Q: What does a Temporal Update provide beyond Signals and Queries?
A: An Update combines sending input into a running Workflow with receiving a response, and supports an optional validator that can reject the request before it is admitted. It is effectively a tracked, request/response interaction — like a Signal that returns a result and can be validated — whereas a Signal is one-way and a Query is read-only. Updates are useful when a caller must both change Workflow state and get an acknowledgement, for example submitting a correction and confirming it was accepted.

Q: What is "versioning a running Workflow" and why is it hard?
A: It is the problem of deploying changed Workflow code while executions started under the old code are still in flight — because those in-flight Workflows will be replayed against history produced by the old logic, naive changes cause the replay to diverge and fail with a non-determinism error. You cannot simply edit the Workflow function and redeploy. Temporal therefore provides explicit versioning mechanisms so old histories keep replaying correctly while new executions use the new code.

Q: How do the `GetVersion`/patch APIs version Workflow code, and what is the recommended use?
A: The patch API (`GetVersion` in some SDKs, `patched`/`workflow.patched` in others) lets you branch within the Workflow code: it writes a marker into the Event History the first time it runs so all future replays of that execution take the same branch, while new executions take the patched branch. Because these APIs clutter code and are easy to misuse, Temporal recommends them mainly for fixing bugs that would otherwise block a Workflow, and prefers Worker Versioning where possible. On replay, a mismatch between the code's patch marker and the recorded history triggers a Workflow-task failure.

Q: What is Worker Versioning (Build IDs) and when should you prefer it over patching?
A: Worker Versioning pins each Workflow Execution to the Worker Deployment Version (identified by a Build ID plus deployment name) that started it, so in-flight executions keep running on Workers with their original code while new executions go to the new version — no in-code branching needed. For teams that can run versioned Worker deployments, Temporal recommends it as the default way to ship Workflow changes, reserving patching for cases where you cannot version deployments. It avoids the code clutter and replay risk of manual patch branches.

Q: What is a non-determinism error in Temporal, and what typically causes it?
A: A non-determinism error occurs when, during replay, the Commands the Workflow code produces do not match the Events already recorded in history — for example a different order of Activity calls, a changed branch, or a removed step. Common causes are editing Workflow logic without versioning, using a forbidden non-deterministic operation (clock, random, direct I/O), or upgrading code so an existing history no longer lines up. The fix is to either version the change (patch or Worker Versioning) or remove the offending non-deterministic construct.

Q: When does Temporal beat a BPM engine?
A: Temporal beats a BPM engine when the orchestration logic is complex, code-driven, developer-owned, and benefits from a general-purpose language — heavy branching, dynamic fan-out, tight integration with services, and rapid iteration. BPM's visual model adds overhead and friction for that style of logic. Temporal shines where engineers, not business analysts, own the process and need maximum expressiveness and testability.

Q: When does a BPM engine beat Temporal?
A: A BPM engine wins when the process must be readable and auditable by non-developers — a compliance officer signing off a settlement or onboarding flow can read and approve a BPMN diagram in a way they never could read Temporal code. BPM also fits human-task-heavy workflows with formal review queues and model-driven governance. In regulated fund administration that auditability is often the deciding factor even when Temporal would be technically cleaner.

Q: When does a plain message queue suffice instead of Temporal?
A: A plain queue (or pub/sub) suffices when each unit of work is a short, stateless, independently retryable task with no long-lived multi-step state to coordinate — fire a message, a consumer processes it, done. The moment you need durable cross-step state, compensation, timers spanning days, signals into a running process, or guaranteed run-to-completion of a multi-stage saga, a queue forces you to hand-build exactly what Temporal provides. Temporal is the upgrade when "orchestrate many steps reliably over time" replaces "process one message."

Q: What is Azure Durable Functions and how does it relate to Temporal conceptually?
A: Azure Durable Functions is Microsoft's durable-execution analogue: an extension of Azure Functions where you write orchestrator functions (deterministic, replayed via event sourcing) and activity functions (side-effecting), with the runtime managing state, checkpoints, retries, and recovery. The orchestrator/activity split and the determinism-via-replay model mirror Temporal's Workflow/Activity design. It is the Azure-native option an architect evaluates when the estate is already serverless on Azure rather than running a Temporal cluster.

Q: Why must Azure Durable Functions orchestrator code also avoid `DateTime.Now`, randomness, and direct I/O?
A: Because Durable Functions replays the orchestrator function using event sourcing exactly as Temporal replays a Workflow, so any non-deterministic call would produce different results on replay and break the reconstruction of state. The same constraints apply: read time and randomness through the durable context APIs and push all I/O into activity functions. It is the same determinism contract under a different vendor's branding.

Q: Name two common application patterns supported by Azure Durable Functions.
A: Common patterns include function chaining (run activities in sequence, each feeding the next), fan-out/fan-in (run many activities in parallel then aggregate), human interaction (wait, possibly for days, for an external approval), and the monitor pattern (recurring polling with state). These map directly onto Temporal idioms — sequential Activities, parallel Activity fan-out with aggregation, waiting on a Signal, and durable timers. Recognising the shared patterns helps an architect translate a design between the two engines.

Q: What is DBOS and how does it differ architecturally from Temporal?
A: DBOS is durable execution delivered as a library rather than a separate cluster: you annotate functions as durable workflows and steps, and DBOS persists their execution state in PostgreSQL, recovering by reading that state back after a crash. The key difference is operational — there is no Temporal Service to run, because the durable state lives in a Postgres database you already operate. That makes DBOS far lighter to run while Temporal offers a richer standalone platform.

Q: Why is DBOS attractive when you already run PostgreSQL?
A: Because DBOS stores its workflow and queue state in the same Postgres you use for business data, the workflow's state changes can commit transactionally alongside your application writes, eliminating the dual-system consistency problem and the latency of talking to a separate orchestrator. You get durable workflows without operating or paying for a Temporal cluster — fewer moving parts, lower latency, and one database to back up. For a team standardised on Postgres this is a major operational simplification.

Q: When does Temporal win over a Postgres-native option like DBOS?
A: Temporal wins at scale and breadth: massive parallel fan-out, very long-lived workflows, rich Signals/Queries and mature visibility tooling, and multi-language or multi-region deployments. In those cases a single Postgres instance backing DBOS becomes a bottleneck, and Temporal's purpose-built, horizontally scalable Service plus its ecosystem and operational maturity pay off. The trade is operational weight for headroom and features.

Q: Name two Postgres-native durable-execution tools mentioned as alternatives to Temporal.
A: DBOS is the primary one — durable execution as a library with workflow and queue state in Postgres — and the syllabus also names Hatchet and pgflow as Postgres-backed alternatives. They share the appeal of keeping orchestration state in a database you already run rather than operating a separate cluster. They sit on the lighter-ops end of the spectrum, with Temporal at the heavier, higher-scale end.

Q: In the orchestration trichotomy, what is the headline trade-off between Temporal and a scheduler like Airflow?
A: A scheduler like Airflow is built for time- or dependency-triggered batch DAGs (run this pipeline at 02:00, then these downstream tasks), whereas Temporal is built for event-driven, long-lived, stateful business workflows with fine-grained retries, signals, and compensation. Use the scheduler when work is periodic batch orchestration; use Temporal when work is a durable saga reacting to events over time. They overlap rarely once you frame the workload correctly.

Q: How would you reimplement a subscription-order saga as a Temporal Workflow?
A: Write a deterministic Workflow function that calls one Activity per step of the saga — validate order, check compliance, reserve units, confirm allocation, settle — passing results between steps, and configure each Activity with timeouts and a Retry Policy. Put all side effects (database writes, external calls) in the Activities and keep branching and orchestration in the Workflow body. The Workflow then becomes a durable, resumable description of the whole order lifecycle.

Q: How do you add a compensation path for a step-2 failure in a Temporal saga?
A: You implement the saga pattern: if a later step's Activity fails, the Workflow code explicitly invokes compensating Activities that undo the effects of the steps already completed (for example release reserved units after a failed allocation). Temporal does not auto-rollback side effects, so you express compensation as ordinary Workflow logic catching the failure and running the inverse Activities. The Workflow's durability guarantees the compensation itself runs to completion even across crashes.

Q: You kill the Temporal Worker mid-run. What do you expect to observe on restart?
A: You expect the Workflow to resume from its Event History on the new (or restarted) Worker, re-using all already-completed Activity results and not re-executing them, then continue from the exact point of interruption to completion. The business outcome should be identical to an uninterrupted run, demonstrating run-to-completion durability. If instead you saw an Activity re-run with a duplicate side effect, that points to a non-idempotent Activity, not a Temporal failure.

Q: A Workflow that worked yesterday now fails with a non-determinism error after a deploy. First thing to check?
A: First check whether the Workflow code was changed in a way that affects the sequence of Commands — added, removed, or reordered Activity calls, timers, or branches — for in-flight executions that still have old histories. The fix is to either version the change with a patch (`GetVersion`/`patched`) or Worker Versioning, or revert the structural change. The error means replay of an old history no longer matches the new code's Commands.

Q: An Activity keeps timing out with `StartToCloseTimeout`. What should you investigate first?
A: First confirm the timeout is realistic for how long the Activity genuinely takes — a too-short `StartToCloseTimeout` will time out healthy work and trigger needless retries. Then check whether the Activity is actually slow (a slow downstream API, a large payload) or stuck, and whether it should be heartbeating so the Service can distinguish progress from a hang. Raising the timeout blindly without understanding the cause just hides a real performance or hang problem.

Q: A long-running Activity is occasionally retried even though it eventually succeeds. What is likely missing?
A: It is likely missing Heartbeats together with a `HeartbeatTimeout`, so the Temporal Service cannot tell the difference between a Worker that is making slow progress and one that has died, and a coarse timeout fires. Adding regular Heartbeats lets the Service see continuous progress and avoid premature retries, while a sensible `HeartbeatTimeout` still catches a genuine crash quickly. Heartbeat details can also let any retry resume from a checkpoint rather than restarting.

Q: Why does keeping a Workflow function "thin" (orchestration only) make testing and replay easier?
A: A thin Workflow that only orchestrates and delegates all side effects to Activities is deterministic and pure with respect to its inputs and recorded results, so it replays reliably and can be unit-tested by mocking Activity results. Heavy logic or hidden I/O in the Workflow body invites non-determinism and makes both replay and testing brittle. The discipline of "decisions in the Workflow, effects in Activities" is what keeps durable execution sound.

Q: How does Temporal handle an Activity that fails transiently versus one that fails permanently?
A: Transient failures are retried automatically according to the Retry Policy with exponential backoff until success or `maximumAttempts`, so flaky network blips self-heal. For permanent failures you raise a non-retryable application error (or configure non-retryable error types) so Temporal stops retrying immediately and surfaces the failure to the Workflow, which can then compensate or escalate. Distinguishing the two is a design decision you encode in the Activity and its policy.

Q: What is the practical meaning of "exactly-once" semantics in the context of a durable Workflow?
A: It means the Workflow's overall logic executes to completion once and its recorded decisions are applied once, even though the underlying Workers and Activities may crash and retry — the durable Event History deduplicates progress on replay. It does not magically make a non-idempotent external side effect happen exactly once; that still requires idempotent Activities. So "exactly-once" is a guarantee about workflow progress, paired with your responsibility to make Activity effects idempotent.

Q: Why is hand-rolled retry-and-timeout logic across microservices considered the thing durable execution replaces?
A: Because spreading retry, timeout, backoff, and state-tracking across many services produces scattered, inconsistent, hard-to-reason-about reliability code that rots and leaves processes in unknown states after partial failures. Durable execution centralises those primitives into the engine and makes the workflow itself the durable unit, so a crash resumes cleanly instead of stranding a half-applied corporate action. The architect's job is recognising when that centralisation is worth adopting.

Q: For a multi-day KYC workflow that waits on an external reviewer, what Temporal feature carries the wait durably?
A: A durable Timer (the SDK's workflow-safe sleep) combined with a Signal: the Workflow can sleep for days awaiting input and resume exactly when a Signal delivers the reviewer's decision, all recorded in Event History so a crash during the wait loses nothing. Real `sleep` or in-memory waits would be lost on restart; Temporal's durable timers and signals are not. This is the canonical "human interaction" pattern in durable execution.

Q: Why is running Temporal OSS in Docker Compose a reasonable way to learn it?
A: Compose lets you stand up the Temporal Service, its dependency (a database) and the Web UI alongside your own Worker container in one declarative file, so you can run, inspect, and kill components locally without cloud setup. It mirrors the real architecture — separate Service plus your Worker — at laptop scale. You can then exercise the full loop: start a Workflow, watch the Event History in the UI, kill the Worker, and watch it resume.

Q: What would you look for in the Temporal Web UI to confirm a Workflow resumed correctly after a crash?
A: You inspect the Workflow Execution's Event History and confirm there is no duplicate Activity execution for steps that completed before the crash — completed Activities show a single completed Event and were reused on replay, not re-run. You also confirm the Workflow reached its final completed Event. Seeing continuous history across the kill, with the same Run ID and no re-executed side effects, is the evidence of clean resumption.

Q: How does Temporal's model help a corporate-actions workflow specifically?
A: Corporate actions (dividends, splits, mergers) are long-running, multi-step, and retry-heavy, often spanning days with external confirmations — exactly the stateful saga durable execution is built for. Temporal keeps the whole process durable so a Worker crash mid-event does not leave holdings half-adjusted, retries transient feed failures automatically, and lets the workflow wait durably on confirmations via signals and timers. The alternative — hand-rolled cross-service state — is where these processes historically rot.

Q: What is the difference between a Workflow Type and a Workflow Execution?
A: A Workflow Type is the definition — the named Workflow function and its logic — while a Workflow Execution is a single running (or completed) instance of that type, identified by a Workflow ID and a Run ID. One Workflow Type can have many concurrent Executions, just as one class has many instances. The Event History belongs to a specific Execution, not to the type.

Q: What roles do the Workflow ID and Run ID play?
A: The Workflow ID is the business identifier you assign (for example the subscription order number) and is used for deduplication and addressing a Workflow, while the Run ID uniquely identifies a particular run of that Workflow ID, changing when a Workflow continues-as-new or is retried at the Workflow level. You signal or query by Workflow ID; the system tracks history by Run ID. Reusing a Workflow ID lets you enforce that only one active Execution exists for a given business entity.

Q: Why might you set a Workflow ID equal to a fund order number?
A: Setting the Workflow ID to a natural business key like the order number gives you idempotent Workflow starts — attempting to start a second Workflow with the same ID can be rejected or reused, preventing duplicate sagas for the same order. It also makes the Workflow directly addressable by that key for Signals and Queries. This is a common pattern for enforcing one-active-process-per-entity in fund operations.

Q: What is Continue-As-New and what problem does it solve?
A: Continue-As-New atomically completes the current Workflow Execution and starts a fresh one with the same Workflow ID but a new Run ID and a fresh, empty Event History, carrying forward whatever state you pass. It solves the problem of an ever-growing Event History in very long-lived or looping Workflows, which would otherwise become huge and slow to replay. You use it for entity workflows that run effectively forever, periodically resetting their history.

Q: Why can an unbounded Event History become a problem, and how is it mitigated?
A: A history that grows without bound (a Workflow that loops forever accumulating Events) increases replay time, storage, and the risk of hitting size limits, hurting performance. It is mitigated by Continue-As-New, which periodically starts a fresh Execution with a clean history while preserving logical continuity via the same Workflow ID. Designing long-running Workflows to continue-as-new on a sensible boundary keeps replay cheap.

Q: What is a Workflow Task (as distinct from an Activity Task)?
A: A Workflow Task is the unit of work a Worker pulls to advance the Workflow's own logic — it replays the Workflow function up to the latest history and produces the next batch of Commands (schedule an Activity, start a timer, complete). An Activity Task, by contrast, is the unit that executes your side-effecting Activity code. Workflow Tasks drive orchestration decisions; Activity Tasks do the actual external work.

Q: Why are Activity results, but not the Activity's internal logic, what replay depends on?
A: Replay reconstructs Workflow state by reading recorded Activity result Events, so it only needs the outcome each Activity already produced, never to re-run the Activity's body. This is exactly why Activities can be non-deterministic and do I/O: their internals are invisible to replay, only their recorded result matters. It cleanly isolates the messy, side-effecting world from the deterministic orchestration that must replay.

Q: How does Temporal differ from a Saga library you might write yourself on top of a queue?
A: A hand-built saga on a queue forces you to persist step state, implement compensation tracking, manage retries and timeouts, and reconstruct state after crashes — all yourself, usually with bugs. Temporal provides durable state, replay-based recovery, built-in retries and timeouts, and signals/queries as platform primitives, so you write the saga as ordinary code and the engine handles durability. The trade is operating the Temporal Service versus carrying your own reliability code.

Q: What determinism risk arises from iterating over an unordered collection in Workflow code?
A: If the iteration order of, say, a hash map can differ between the original run and replay, the Workflow may issue Commands in a different order than recorded in history, causing a non-determinism error. You must use deterministically ordered structures or sort before iterating in Workflow code. It is a subtle pitfall because the code looks innocent but its ordering is implicitly non-deterministic.

Q: Why is upgrading a library that changes serialization or ordering risky for in-flight Workflows?
A: Such an upgrade can change how data is serialized or how collections are ordered during replay, making the replayed Workflow produce Commands that no longer match the recorded Event History and failing with non-determinism. Even a dependency bump can silently break in-flight executions. This is why Workflow code and its determinism-affecting dependencies must be versioned carefully, often via Worker Versioning to pin executions to their original build.

Q: How does Temporal's approach to timeouts differ from a cron job that simply re-runs on failure?
A: A cron re-run restarts the whole job from the top with no memory of prior progress, so completed steps re-execute and partial side effects can duplicate. Temporal's timeouts apply per Activity and per Workflow with replay preserving completed steps, so only the failed unit is retried and prior work is reused. You get fine-grained, stateful recovery instead of coarse, stateless re-execution.

Q: For a NAV-calculation pipeline, when is Temporal overkill versus a good fit?
A: It is overkill if the NAV calc is a simple scheduled batch that either succeeds or is re-run wholesale — a scheduler handles that fine. It is a good fit when the NAV process is a long, multi-step, stateful flow with external dependencies, partial-completion concerns, manual checks, and compensation needs (for example a multi-source pricing and reconciliation saga). The deciding question is whether you need durable per-step state and run-to-completion, not just periodic triggering.

Q: What does "the workflow itself is durable" mean compared to making each service durable?
A: It means the orchestration — the sequence, the local variables, the position in the process — is what Temporal persists and resumes, rather than asking each downstream service to remember the overall progress. Individual services and Activities stay stateless and replaceable while the durable thread of the business process lives in the Event History. This inverts the usual burden: durability is a property of the workflow, not bolted onto every participant.

Q: Why can you not catch a non-determinism error inside the Workflow and "handle" it?
A: A non-determinism error is detected by the framework during replay because the code's Commands diverge from recorded history; it is not an ordinary application exception you can catch and recover from in business logic. The only real fixes are to make the code match history again (revert) or to version the change so old histories take the old path. Treating it as a catchable error misunderstands that replay integrity, not your logic, is what failed.

Q: How do Signals get delivered to a Workflow that is currently being replayed or not yet loaded?
A: Signals are durably recorded as Events by the Temporal Service as soon as they arrive, independent of whether a Worker currently has the Workflow in memory, so they are not lost if no Worker is active. When a Worker next processes the Workflow, the Signal Event is part of the history and is delivered in order during replay/execution. This durability is why Signals are reliable for external input even across crashes and quiet periods.

Q: Why are Queries safe to run against a Workflow without affecting its history?
A: A Query causes the Worker to replay the Workflow to its current state purely to evaluate the read-only query handler and return a value — it produces no Commands and writes no Events, so the history is untouched. That is what makes Queries a non-intrusive way to inspect live state. The cost is that a Query forces a replay, so very frequent queries on huge histories can be expensive.

Q: What is the difference between a Workflow failing and a Workflow timing out at the Workflow level?
A: A Workflow failure means the Workflow logic raised an unhandled error (or hit a non-determinism error) and ended in a failed state, whereas a Workflow-level timeout (such as a Workflow Execution Timeout) means the whole Execution exceeded its allowed wall-clock duration and was terminated by the Service. Both end the Execution, but for different reasons — application/logic versus an outer time budget. You set Workflow timeouts to bound total lifetime, separate from per-Activity timeouts.

Q: Why might you isolate heavy Activities onto a dedicated Task Queue?
A: Routing CPU- or memory-heavy Activities (for example bulk EMT/EPT file parsing) to their own Task Queue lets you run a Worker fleet sized and tuned specifically for that load, without starving lightweight Activities sharing a pool. It gives independent scaling, resource isolation, and the ability to apply different concurrency limits per queue. This is standard capacity-management practice once different Activities have very different cost profiles.

Q: How does Temporal support running the same Workflow logic in different languages?
A: Temporal provides SDKs in multiple languages (including Python, Go, Java, TypeScript, .NET) that all speak the same Service protocol and Event-History model, so different services can host Workers in different languages against the same Temporal Service. This multi-language reach is one reason Temporal scales across a heterogeneous estate where a single-language library would not. An architect weighs this against a Postgres-native library that may be tied to fewer languages.

Q: In Temporal, what is the consequence of an Activity that is not registered with any running Worker?
A: If no Worker polls the Task Queue for that Activity, the Activity Task sits in the queue undispatched; with the default infinite `ScheduleToStartTimeout` it can wait indefinitely without erroring, leaving the Workflow stalled. Setting a finite `ScheduleToStartTimeout` makes Temporal fail the Activity when no Worker picks it up in time, surfacing the missing-Worker problem. The first thing to check when a Workflow hangs at an Activity is whether a Worker for that Task Queue is actually running.

Q: Why is "replay" cheaper than re-doing the actual work?
A: Replay re-executes only the deterministic Workflow function and substitutes recorded Event results for every Activity, timer, and signal, so no external calls, no I/O, and no waiting actually occur — it is pure in-memory fast-forwarding. The expensive side effects happened once and are read back from history. That asymmetry is what makes resuming a crashed Workflow fast and side-effect-free.

Q: How would you justify Temporal over Azure Durable Functions for a particular platform?
A: You would justify Temporal where you want a vendor-neutral, self-hostable (or Temporal Cloud) platform spanning multiple languages and clouds, with rich signals/queries/visibility and the freedom from Azure lock-in. Azure Durable Functions makes more sense when the estate is already committed to Azure Functions and you want the native, serverless, fully managed option. The decision is largely about existing platform commitment and portability requirements, since the durable-execution model is similar.

Q: Why are signals preferred over polling an external system inside a Workflow?
A: Polling would require the Workflow to repeatedly invoke Activities that hit an external system, wasting work and complicating logic, whereas a Signal lets the external system push the event into the Workflow exactly when it happens, recorded durably in history. The Workflow can simply wait (via a durable timer or condition) until the Signal arrives. Push-via-Signal is cleaner, cheaper, and more responsive than poll-via-Activity.

Q: How does compensation in a Temporal saga differ from a database transaction rollback?
A: A database rollback undoes uncommitted changes within one transaction automatically, whereas a Temporal saga's steps commit independently across services and cannot be rolled back atomically — so you must run explicit compensating Activities to semantically undo completed steps. Compensation is forward-recovery logic you write (release the reservation, reverse the booking), not an automatic rollback. Temporal makes running that compensation reliable; it does not make it automatic.

Q: What is the risk of putting a large blob directly into Workflow inputs, signals, or Activity results?
A: Those payloads are stored in the Event History, so oversized blobs bloat history, slow replay, and can hit payload-size limits. The practice is to pass references (an object-store key, a database ID) through the Workflow and have Activities fetch the actual data. Keeping the durable history small keeps replay fast and storage manageable — relevant when processing large EMT/EPT or pricing files.

Q: How does the Temporal Service detect that a Worker has died mid-Activity without a heartbeat?
A: Without heartbeating, the Service cannot distinguish a dead Worker from a slow one until the Activity's `StartToCloseTimeout` (or `ScheduleToCloseTimeout`) elapses, at which point the attempt is timed out and retried. That is why long-running Activities should heartbeat with a short `HeartbeatTimeout` — it gives the Service near-real-time crash detection instead of waiting out a long execution timeout. The choice of timeout strategy directly determines failure-detection latency.

Q: Why is Temporal described as a "modern saga orchestrator"?
A: Because it directly supports the saga pattern — a sequence of local transactions across services with compensating actions on failure — but adds durable state, automatic retries, durable timers, and crash recovery as platform features rather than library plumbing. It modernises the saga by making the orchestration itself durable code you can read and test, instead of an event-choreography web that is hard to reason about. That is its core value proposition for trade-lifecycle and corporate-action workflows.

Q: What is the operational cost an architect accepts when choosing Temporal OSS over a Postgres-native library?
A: Choosing Temporal OSS means operating a separate, stateful distributed system — the Temporal Service plus its persistence — including upgrades, scaling, monitoring, and backups, which is real ongoing work. A Postgres-native library like DBOS folds that into a database you already run. The architect must judge whether Temporal's scale and feature headroom justify the extra operational surface for the workloads at hand.

Q: For a transfer-agency subscription saga, why is durable run-to-completion important for regulatory correctness?
A: In regulated fund administration, a half-applied subscription — units reserved but allocation not confirmed after a crash — can produce an inconsistent register and reconciliation breaks that auditors and regulators care about. Durable execution guarantees the saga either runs to a consistent completion or compensates, never silently stranding state, which supports accurate, auditable transfer-agency records. This reliability is part of meeting operational-resilience expectations such as those under DORA.

Q: How might durable execution support DORA-style operational-resilience requirements?
A: DORA emphasises that critical financial processes withstand and recover from ICT disruptions; durable execution directly delivers crash-resilient, automatically recovering, retryable workflows with a complete Event History as an audit trail of every step. That recorded history and guaranteed recovery help demonstrate that a corporate-action or settlement process can survive component failures without manual reconstruction. It is a technical control that supports the resilience and traceability DORA expects.

Q: Why is the Event History also valuable as an audit artefact in a regulated context?
A: Because it is a durable, ordered, immutable record of every decision and step — which Activities ran, what results they returned, when signals arrived — it provides a precise, replayable audit trail of how a workflow reached its outcome. In fund administration that traceability supports investigations, reconciliations, and regulatory inquiries far better than scattered logs. The same property that enables replay (a complete history) doubles as compliance evidence.

Q: What happens if you change an Activity's signature but not the Workflow that calls it, for in-flight Workflows?
A: If a running Workflow's history expects the old Activity call and the new code now calls it differently, replay can diverge and the Workflow may fail with non-determinism, or the Activity may fail to bind. Activity changes that affect how the Workflow schedules them are determinism-affecting and need versioning just like Workflow changes. Safer changes keep the call shape stable or are guarded behind a patch/Worker Version.

Q: Why does Temporal recommend short Activities and short timeouts where feasible?
A: Short Activities with tight timeouts mean failures and hangs are detected quickly, retries are cheap and granular, and progress is checkpointed frequently, so recovery is fast and little work is lost on a crash. Very long monolithic Activities delay failure detection and force large re-do on failure unless they heartbeat. Decomposing work into smaller Activities improves both responsiveness and resilience.

Q: How do durable Timers in Temporal survive a crash?
A: A durable Timer is recorded as an Event when the Workflow schedules it, with its fire time stored in the Event History, so even if every Worker crashes the Timer's existence and deadline persist in the Service. On replay a Timer that already fired is not waited on again; one still pending resumes counting toward its recorded deadline. This is why Workflow `sleep` can safely span days where a real `sleep` could not.

Q: What is the danger of performing logging or metrics with side effects directly in Workflow code?
A: Ordinary logging is usually fine because it does not change Commands, but anything that branches Workflow behaviour on an external or time-dependent value — or that performs real I/O — risks non-determinism on replay. Worse, logs will appear to "run" on every replay, which can confuse operators if not handled by the SDK's replay-aware logging. The safe stance is to keep side-effecting observability out of Workflow logic and rely on the SDK's replay-aware mechanisms.

Q: Why is "deterministic by construction" a better goal than "tested to be deterministic"?
A: Because non-determinism can hide in rarely taken branches, dependency behaviour, or data-dependent ordering that tests miss, only surfacing as a production replay failure mid-workflow. Building Workflows from deterministic SDK primitives (workflow time, workflow random, durable timers, Activities for all I/O) removes whole classes of these bugs structurally. Replay tests against recorded histories help, but designing for determinism is the primary defence.

Q: How would you decide between Temporal and Camunda BPMN for a settlement-exception process?
A: If the exception handling is human-task heavy and must be auditable and editable by operations and compliance staff, Camunda BPMN's visual, governable model is likely the better fit. If the exception flow is complex, code-driven, integration-heavy, and owned by engineers who need expressiveness and testability, Temporal fits better. The deciding lens is who owns and audits the process and how much human-readable governance the regulator and business demand.

Q: What does it mean that Activities can be invoked with both a Retry Policy and timeouts together?
A: Timeouts decide when a given attempt (or the whole Activity) is considered failed, and the Retry Policy decides whether and how that failure is retried — they cooperate. For example a `StartToCloseTimeout` fires on a slow attempt, then the Retry Policy schedules the next attempt after its backoff interval up to `maximumAttempts`. Tuning them together lets you bound both per-attempt latency and total retry effort precisely.

Q: Why might a freshly started Worker briefly "replay" many Workflows on startup?
A: When a Worker picks up Workflow Tasks for executions it has never held in memory — including after a deploy or a crash — it must replay each one's Event History to reconstruct state before continuing. This is normal and fast because replay does no real I/O, but a large fleet restart can cause a burst of replay activity. Sticky execution caches recently run Workflows on a Worker to avoid replaying from scratch on every task.

Q: What is sticky execution and what does it optimise?
A: Sticky execution keeps a Workflow's reconstructed state cached on the Worker that last ran it, so subsequent Workflow Tasks for that execution can continue from the cached state instead of replaying the entire Event History each time. It optimises latency and CPU for active Workflows. If the sticky Worker is unavailable, another Worker simply replays from history — correctness is preserved, only the optimisation is lost.

Q: Why is durable execution a poor fit for ultra-low-latency, high-throughput stateless request handling?
A: Durable execution adds the overhead of persisting Event History and round-tripping through the Service for each orchestration decision, which is the wrong trade for a hot path that just needs to process a request in microseconds and forget it. That workload belongs behind a plain service or queue. Temporal pays for itself when workflows are long-lived and stateful, not when they are short, stateless, and latency-critical.

Q: How does choosing idempotency keys per Activity interact with retries during a crash?
A: An idempotency key (for example the order ID plus step name) lets a retried Activity recognise that the same logical operation already happened and skip or safely repeat it, so a crash-induced retry does not double-apply the side effect. Without a key, Temporal's retry — which may follow a partially completed attempt — can duplicate the effect. Designing the key around the business identity of the operation is what makes the retry-safe guarantee real.

Q: What is the first thing to verify if killing the Worker causes the Workflow to start over from step 1 instead of resuming?
A: First verify that the Workflow's progress was actually being recorded as Events — that the steps were real Activities (whose results persist) rather than plain in-Workflow code with no recorded outcome, and that you are reusing the same Workflow ID/Run rather than starting a new Execution. If completed steps re-ran, also check the Activities are idempotent so any unavoidable retry is harmless. A "restart from step 1" usually signals a new Execution being started, not Temporal failing to resume.

Q: Summarise the architect's decision rule among Temporal, BPM, and a scheduler for a fund workflow.
A: Use a scheduler when the workflow is periodic batch orchestration triggered by time or data dependencies; use BPM when the process must be human-readable, auditable, and editable by business and compliance, especially with human tasks; use Temporal (or a Postgres-native durable-execution library) when the workflow is a long-lived, code-driven, stateful saga needing fine-grained retries, durable timers, signals, and run-to-completion. The architect picks per workflow, with explicit reasons about ownership, auditability, statefulness, and scale — never one tool for everything.

Q: What command starts a local single-binary Temporal dev server, and where does its Web UI appear by default?
A: You run `temporal server start-dev`, which boots the Service, creates the `default` Namespace, and serves the Web UI at `http://localhost:8233` while the Frontend gRPC endpoint listens on `127.0.0.1:7233`. The UI port is the gRPC port plus 1000, overridable with `--ui-port`. By default it uses an in-memory SQLite database, so all Workflows vanish on restart unless you pass `--db-filename` to persist them.

Q: What is the difference between `temporal workflow start` and `temporal workflow run` on the CLI?
A: `temporal workflow start` launches a Workflow Execution and returns immediately with the Workflow ID and Run ID without waiting, whereas `temporal workflow run` starts it and blocks, streaming progress until the Workflow completes and printing the result. You use `start` for fire-and-forget kicks (the normal production pattern) and `run` for interactive testing when you want to watch the outcome. Both require `--task-queue` and `--type` (the Workflow Type name).

Q: The legacy `tctl` tool is mentioned in old tutorials — what should you use instead and why?
A: Use the newer `temporal` CLI; `tctl` is deprecated and reached end of support on `2025-09-30`, after which its repository was archived. The modern CLI restructures commands (for example `temporal workflow list`, `temporal workflow describe`) and adds JSON/table output via `--output` and pager control via `--pager`. Following a tutorial that calls `tctl` is a signal it predates the current tooling.

Q: What is the hard limit on Temporal Event History size, and what happens when it is hit?
A: A single Workflow Execution's Event History is capped at `51200` Events or `50MB`, whichever comes first, and on breaching either limit the Service terminates the Workflow with a "history size/count exceeds limit" error. This cap exists because a cold Worker must replay the entire history to reconstruct state, so unbounded histories would make recovery unworkable. The defence is Continue-As-New, which resets to a fresh history before the limit.

Q: At what thresholds does Temporal warn you that an Event History is getting large?
A: The Service emits a warning once a Workflow's history passes `10240` Events or `10MB`, well before the hard `51200`-Event / `50MB` termination limit. The warning is your cue to introduce Continue-As-New or rethink a loop that accumulates Events without bound. Treating the warning seriously avoids a surprise termination of a long-running entity Workflow in production.

Q: How does Continue-As-New differ from simply starting a new Workflow from within an Activity?
A: Continue-As-New is an atomic Workflow-level operation that closes the current Execution and starts a fresh Run of the same Workflow ID with a clean history in one recorded step, preserving logical continuity and any carried-forward state you pass. Starting a new Workflow from an Activity is a separate, non-atomic side effect with no inherent linkage, no guaranteed handoff of in-memory state, and a risk of duplication on retry. For history-truncation purposes you always use Continue-As-New, not a manual restart.

Q: When Continue-As-New runs, what carries over to the new Run and what does not?
A: Only the explicit input arguments you pass into the Continue-As-New call carry over — you choose what state to seed the next Run with — while the entire Event History, accumulated local variables, and the old Run ID are discarded and replaced. Pending Signals received but not yet processed are also carried over so none are lost. Anything not threaded through the new input is gone, so you must deliberately serialise the state you need to continue.

Q: How do you raise a failure inside an Activity that Temporal should never retry?
A: You raise an `ApplicationError` with `non_retryable=True` (Python) or the equivalent flag, which makes that failure terminal regardless of the Retry Policy — Temporal stops immediately and surfaces it to the Workflow. This is the right tool for genuinely permanent errors like malformed input or a business rejection, where retrying only wastes time. It overrides even a generous `maximumAttempts`, so use it precisely.

Q: How does setting `non_retryable_error_types` on a Retry Policy differ from raising a non-retryable `ApplicationError`?
A: `non_retryable_error_types` is configured on the Retry Policy and lists error type names that should not be retried, applied by the Service when an Activity fails with a matching type. Raising an `ApplicationError(non_retryable=True)` is decided inside the Activity code at throw time and is always non-retryable even if the policy does not list it. The first is a declarative policy filter; the second is an imperative, per-throw decision — and the imperative flag wins.

Q: An Activity should retry on a `ConnectionError` but fail fast on a `ValueError` — how do you express that?
A: Configure the Activity's Retry Policy with `non_retryable_error_types=["ValueError"]` (or your domain's error type name) so a `ValueError` ends the Activity immediately, while a `ConnectionError`, not being listed, follows the normal exponential-backoff retry up to `maximumAttempts`. Type names are matched as strings, so they must correspond to the actual exception class names your Activity raises. This separates transient infrastructure faults from permanent logic faults declaratively.

Q: How would you call multiple Activities in parallel inside a Workflow and wait for all of them?
A: You start each Activity without awaiting it to get a list of awaitables (for example `workflow.start_activity(...)` in Python or returning Promises in TypeScript), then await them together with the SDK's gather/all primitive (`asyncio.gather` in Python, `Promise.all` in TS). This is the fan-out/fan-in pattern: dispatch N Activities concurrently, then aggregate. The key is that the concurrency primitive must be the deterministic, SDK-provided one so the ordering is reproducible on replay.

Q: How do you implement a timeout-or-signal race ("wait up to 24h for approval, else escalate") in a Workflow?
A: You start a durable Timer for 24h and concurrently wait on the approval Signal, racing them with the SDK's wait-condition or select primitive, then branch on whichever resolves first. In Python you commonly use `workflow.wait_condition(lambda: self._approved, timeout=timedelta(hours=24))`, which returns when the Signal handler sets the flag or raises on timeout so you take the escalation path. Both the Timer and the Signal are recorded as Events, so the race resolves identically on replay.

Q: What does `workflow.wait_condition` do and why is it determinism-safe?
A: `workflow.wait_condition(predicate, timeout=...)` blocks the Workflow until the supplied predicate becomes true (typically flipped by a Signal handler) or the optional timeout elapses, without busy-looping. It is determinism-safe because it only resumes in response to recorded Events — a Signal arriving or a durable Timer firing — so replay re-evaluates the predicate at exactly the same history points. It is the idiomatic way to make a Workflow await external state.

Q: What is signal-with-start and when is it useful?
A: Signal-with-start atomically delivers a Signal to a Workflow, starting that Workflow first if no running Execution with the given Workflow ID exists, so you never race between "does it exist yet?" and "send the Signal." It is ideal for entity Workflows keyed by a business ID — for example routing an incoming price update to a per-instrument Workflow that may or may not already be running. It removes a check-then-act race that plain signal plus start would otherwise have.

Q: Why can a Query that takes a long time to evaluate be a performance concern on a large history?
A: Answering a Query may require a Worker to replay the Workflow's full Event History to reconstruct current state before the read-only handler runs, so on a Workflow with tens of thousands of Events each Query is expensive. Sticky execution caching mitigates this when the Workflow is hot, but a cold Workflow pays the full replay cost per Query. Frequent Queries on huge histories are therefore a reason to keep histories small (Continue-As-New) or expose state another way.

Q: What is the practical difference between terminating and cancelling a Workflow Execution?
A: Cancellation requests a graceful stop — it is delivered as an Event the Workflow can observe and respond to, running cleanup and compensation before exiting, so it is cooperative. Termination is a hard, immediate kill by the Service that records a terminated Event and runs no further Workflow code, so no compensation executes. You cancel when you want orderly teardown of a saga; you terminate only when you must stop a stuck or runaway Workflow outright.

Q: What is a Child Workflow and when would you choose one over an Activity?
A: A Child Workflow is a separate Workflow Execution started and managed by a parent Workflow, with its own Event History, identity, and replay lifecycle. You choose a Child Workflow over an Activity when the sub-process is itself a long-lived, multi-step orchestration that benefits from independent history, its own retries/timeouts at the Workflow level, or separate scaling — for example a full per-investor onboarding running under a batch parent. An Activity is for a single side-effecting unit of work, not a sub-orchestration.

Q: What is the `ParentClosePolicy` and what does its default do?
A: `ParentClosePolicy` controls what happens to running Child Workflows when their parent closes; the default `TERMINATE` kills the children when the parent completes, fails, or is terminated. Alternatives are `ABANDON` (let children keep running independently) and `REQUEST_CANCEL` (send them a cancellation). You set `ABANDON` when a child should outlive its parent, which matters for long-lived sub-processes spawned by a short-lived dispatcher.

Q: Why is `workflow.now()` preferred over `datetime.now()` even for something as innocent as a log timestamp inside Workflow code?
A: `workflow.now()` returns the deterministic time recorded in Event History, so it yields the same value on the original run and every replay, whereas `datetime.now()` returns fresh wall-clock time that diverges on replay. Even if a timestamp only feeds a log, using the real clock is a latent non-determinism hazard the moment that value ever influences a branch or a Command. Consistently reading time through the SDK keeps the Workflow uniformly replay-safe.

Q: How do you generate a deterministic UUID or random number inside a Temporal Workflow?
A: You use the SDK's workflow-safe random API — in Python `workflow.random()` (a seeded `random.Random` recorded in history) and `workflow.uuid4()` for UUIDs — so the value is reproduced on replay rather than freshly generated. The seed is captured in the Event History, making the sequence deterministic. Generating the value with the standard library `random` or `uuid` modules in Workflow code is forbidden because it would differ on replay.

Q: How can you test that a Workflow still replays cleanly against an old Event History before deploying changed code?
A: You use the SDK's replay testing facility — for example the Python `Replayer` / `WorkflowReplayer` — feeding it a captured Event History JSON and your new Workflow code; if the new code diverges it raises a non-determinism error in the test rather than in production. You export real histories with `temporal workflow show --output json` to build a regression corpus. This replay-test gate catches determinism-breaking edits in CI before they strand in-flight Workflows.

Q: How do you export a Workflow's Event History from the CLI for replay testing or audit?
A: You run `temporal workflow show --workflow-id <id> --output json` (optionally with `--run-id`) to dump the full Event History as JSON, which you can archive as an audit artefact or feed into a replay test. The same history is viewable interactively in the Web UI. Capturing histories of representative production Workflows gives you a determinism regression suite for future code changes.

Q: How does `patched`/`workflow.patched` actually decide which branch a given Execution takes?
A: The first time `workflow.patched("my-change-id")` executes for an Execution it writes a marker Event into history and returns `True`; on replay of that same Execution the recorded marker makes it return `True` again, so the Execution always takes the new branch consistently. Executions whose histories predate the marker do not have it, so `patched` returns `False` for them and they take the old branch. The marker in history, not the deploy state, drives the per-Execution decision.

Q: What is `deprecate_patch` for, and what is the lifecycle of a patch?
A: The lifecycle is: introduce `patched("id")` to branch old vs new while old Executions drain, then once no Executions on the old code remain, call `deprecate_patch("id")` (which marks the patch deprecated without branching) for a release, and finally remove the patch entirely. `deprecate_patch` lets you stop writing new markers while still tolerating histories that have them, easing the cleanup. Skipping straight to deletion while old histories still carry the marker would reintroduce non-determinism.

Q: Why is Worker Versioning with Build IDs generally cleaner than scattering `patched` calls?
A: Worker Versioning pins each Execution to the Worker Deployment Version that started it, so old Executions keep replaying on Workers running their original code and new Executions go to new Workers — the branching happens at the infrastructure level with no `patched` markers cluttering the Workflow function. Patching, by contrast, accumulates branch code and markers you must later deprecate and delete. Versioning trades a deployment-routing mechanism for in-code complexity, which scales better across many changes.

Q: With Worker Versioning, how does a new Workflow Execution choose its version, and how do old ones stay pinned?
A: A Task Queue has a Current Deployment Version; new Executions are routed to Workers of that current version and become pinned to it for their lifetime, while Executions already pinned to an earlier version continue to be served only by Workers of that earlier version. You promote a new build to current to start sending new work to it, and you keep the older Workers running until their pinned Executions drain. This guarantees each history is only ever replayed by code that can reproduce it.

Q: What is the `WorkflowExecutionTimeout` and how does it differ from `WorkflowRunTimeout`?
A: `WorkflowExecutionTimeout` bounds the total time for the whole logical Workflow including all Continue-As-New runs and retries, while `WorkflowRunTimeout` bounds a single Run only. So an entity Workflow that continues-as-new repeatedly is governed across all its Runs by the Execution timeout but each individual Run by the Run timeout. Both default to unlimited, and you set them to cap lifetime separately from the per-Activity timeouts.

Q: A Workflow appears stuck and the Web UI shows repeated "WorkflowTaskTimedOut" events — what is the likely cause?
A: Repeated Workflow Task timeouts usually mean the Workflow function is blocking or running heavy work directly — a real `sleep`, a synchronous network call, or a tight CPU loop — so it cannot finish the task within `WorkflowTaskTimeout` and the Service keeps retrying it. The fix is to move that work into an Activity and use durable Timers instead of blocking sleeps. It can also indicate a Worker that is overloaded or deadlocked on the event loop.

Q: What is the Workflow "deadlock detector" and what trips it?
A: The SDK runs Workflow Task execution under a deadlock detector (around a one-second budget by default) that fails the task if the Workflow code does not yield back control in time — typically because it performed a blocking call or an unbounded CPU loop on the Workflow thread. It exists because Workflow code must be cooperative and fast. Seeing a deadlock/`PotentialDeadlock` error means real blocking work leaked into the Workflow body and belongs in an Activity.

Q: How does the `maximumAttempts` value of `0` behave in a Retry Policy?
A: `maximumAttempts=0` means unlimited attempts — Temporal keeps retrying the Activity indefinitely (subject to any `ScheduleToCloseTimeout` or Workflow-level timeout) following the backoff curve. This is the default for Activities, which is why a flaky dependency is retried forever unless you bound it. To cap retries you set a positive integer, and to bound total time you add a `ScheduleToCloseTimeout`.

Q: Why might you deliberately set a finite `ScheduleToCloseTimeout` even when each attempt is fast?
A: Because with the default unlimited `maximumAttempts`, a persistently failing dependency would retry forever; a `ScheduleToCloseTimeout` puts an absolute wall-clock cap on the whole Activity so it eventually fails the Workflow and lets compensation or alerting fire. It bounds total effort even when no single attempt is slow. This is the timeout that prevents a "fast but always-failing" Activity from retrying into eternity.

Q: How do you make a long file-processing Activity both crash-detectable and resumable?
A: You heartbeat periodically with a checkpoint payload (for example the byte offset or last-processed record) via the SDK's heartbeat call, and set a short `HeartbeatTimeout` so the Service detects a dead Worker quickly. On retry the new attempt reads `activity.info().heartbeat_details` (the last reported checkpoint) and resumes from there instead of restarting. This combination gives fast failure detection plus minimal rework for a large EMT/EPT file.

Q: Why should secrets or PII generally not be passed as plain Workflow/Activity payloads, and what is the mechanism to protect them?
A: Payloads are persisted verbatim in the durable Event History, so plaintext secrets or investor PII would sit readable in storage and in the Web UI, a problem under data-protection rules. Temporal's answer is a custom Data Converter with a Codec (often a Codec Server) that encrypts payloads before they leave your process and decrypts them on the way back, so the Service only ever stores ciphertext. This keeps sensitive fund data confidential while preserving durability.

Q: How does Temporal's at-least-once Activity execution reconcile with the "exactly-once workflow" promise for a SWIFT instruction?
A: Temporal guarantees the Workflow's orchestration progresses exactly once (history deduplicates on replay), but an Activity may execute more than once because a retry can follow an attempt that actually completed its side effect before the result was recorded. So sending a SWIFT instruction must be made idempotent — keyed on a unique message reference — so a duplicate Activity run is a no-op at the counterparty. The platform gives at-least-once Activity delivery; you supply idempotency for exactly-once effect.

Q: Why is using the Workflow ID itself as part of an Activity's idempotency key a strong pattern?
A: The Workflow ID is a stable business key (for example the order number) that is identical across every retry and replay of that saga, so combining it with a step identifier yields a deterministic idempotency key the downstream system can dedupe against. Because Workflow IDs can be made unique per business entity, the key naturally maps one external effect to one business operation. This anchors idempotency in business identity rather than in volatile attempt or Run IDs.

Q: What is the Schedules feature and why prefer it over an external cron triggering Temporal?
A: Temporal Schedules are a built-in mechanism to start Workflow Executions on a cron-like or interval timetable, with pause/unpause, backfill, overlap policies, and jitter, all managed inside Temporal and visible in the Web UI. Preferring it over an external cron keeps the trigger durable, observable, and co-located with the Workflows rather than depending on a separate scheduler that can drift or fail silently. For a recurring NAV or reconciliation run it gives you missed-run backfill and overlap control out of the box.

Q: For a recurring Schedule, what does the overlap policy control and why does it matter for a NAV run?
A: The overlap policy decides what happens when a scheduled Workflow is still running when the next trigger fires — options include `SKIP` (don't start the new one), `BUFFER_ONE`/`BUFFER_ALL` (queue it), `CANCEL_OTHER`, or `ALLOW_ALL` (run concurrently). For a NAV calculation you typically do not want two overlapping runs producing conflicting figures, so `SKIP` or `BUFFER_ONE` prevents concurrent execution while still honouring the cadence. Choosing the wrong policy could double-run a valuation or silently drop a cycle.

Q: How do Search Attributes improve operating Temporal at fund-administration scale?
A: Search Attributes are indexed key/value fields you attach to a Workflow (for example `fundId`, `orderType`, `navDate`) that the Service indexes via its Visibility store, letting you run filtered List queries like "all failed subscription sagas for fund X today." They turn the Web UI and `temporal workflow list --query` into an operational search tool over millions of Executions. Without them you can only address Workflows by ID, which does not scale for triage and reporting.

Q: Why can adding more Workers reduce Activity latency but not, by itself, speed up a single Workflow's sequential steps?
A: More Workers increase the concurrency of polling and executing Tasks, so a backlog of pending Activity Tasks drains faster and queue-wait (`ScheduleToStart`) shrinks across many Workflows. But a single Workflow whose steps are sequential is gated by each step's own duration, not Worker count — adding Workers cannot make step 3 start before step 2 finishes. You speed a single Workflow by parallelising its independent Activities, not by scaling Workers.

Q: An Activity sits pending and never starts while Workflows pile up — what queue-level things do you check first?
A: First confirm a Worker is actually polling that Activity's Task Queue and that the Activity Type is registered on it, since a missing or misrouted Worker leaves Tasks undispatched; the Web UI's Task Queue / pollers view shows whether any poller is present. Next check Worker concurrency limits (`max_concurrent_activities`) and resource saturation that could throttle pickup. Setting a finite `ScheduleToStartTimeout` would also surface this as an explicit failure instead of a silent stall.

Q: What does the Workers / pollers panel in the Temporal Web UI tell you when diagnosing a stuck Workflow?
A: It shows which Worker identities are currently polling a given Task Queue for Workflow Tasks and Activity Tasks, with last-access times, so you can confirm whether any live Worker is available to make progress. If the panel shows no pollers for the queue your Workflow needs, the Workflow is stalled because nothing is draining its Tasks — a missing or crashed Worker fleet. It is the fastest first check when a Workflow hangs with no error.

Q: Why is the Temporal Web UI history view a better debugging tool than application logs for a stalled saga?
A: The history view shows the exact ordered sequence of Events — which Activities were scheduled, started, completed, timed out, or failed, with inputs, results, retry attempts, and timestamps — giving an authoritative, replayable picture of where the Workflow actually is. Application logs are scattered, may be incomplete, and re-emit on replay, whereas the Event History is the single durable source of truth. For "why is this subscription stuck?", the last Event and any pending Activity tell you immediately.

Q: How does Temporal handle a Worker that is healthy but simply slow to keep up with load?
A: Tasks it cannot pick up remain on the Task Queue and are taken by other Workers or wait until capacity frees up, while per-Worker concurrency limits (`max_concurrent_workflow_task_executions`, `max_concurrent_activity_task_executions`) cap how much one Worker pulls so it is not overwhelmed. If `ScheduleToStart` times are climbing, that is the signal to add Workers or raise limits. The durable state is unaffected — backpressure manifests as latency, not lost work.

Q: Why does Temporal recommend keeping Activity inputs and outputs small and passing references for large data?
A: Activity inputs and results are recorded in the Event History, so large payloads bloat history toward the `10MB` warning and `50MB` hard limit, slow replay, and increase storage. Passing a reference — an object-store key or database ID — and fetching the bytes inside the Activity keeps history lean while still moving the data. For large EMT/EPT or pricing files this reference pattern is essentially mandatory.

Q: What is a "local Activity" and when is it appropriate?
A: A local Activity runs in the same Worker process as the Workflow Task without a separate Activity Task round-trip through the Service, making it lower-latency and lighter on history for very short, idempotent operations. It is appropriate for quick calls like a fast in-memory lookup or a tiny side effect where the overhead of a normal Activity dominates. It is not appropriate for long-running work, anything needing independent retries with long backoff, or heartbeating, where a regular Activity is correct.

Q: How would you model a multi-day corporate-action workflow that must wait for an external custodian confirmation that may never come?
A: You combine a durable Timer as a deadline with a Signal (or Update) carrying the custodian confirmation, racing them via `wait_condition`: if the confirmation Signal arrives you proceed, and if the Timer fires first you take an exception/escalation path to a human queue. All of it persists in Event History, so a Worker crash during the multi-day wait loses nothing. This is the canonical durable human/external-interaction pattern applied to corporate actions.

Q: For a transfer-agency batch that opens thousands of per-investor sub-sagas, why might you choose a parent Workflow with Child Workflows over one giant Workflow?
A: One giant Workflow tracking thousands of investors accumulates an enormous Event History (risking the `50MB`/`51200`-Event limit) and couples all investors into one replay unit, so a single problem affects everyone. A parent that spawns one Child Workflow per investor gives each its own bounded history, independent retries, and isolated failures, while the parent coordinates and aggregates. This scales and isolates far better, at the cost of more Executions to operate.

Q: When the syllabus says "kill the worker mid-run and watch it resume," what specific UI evidence distinguishes a true resume from a silent restart?
A: A true resume shows one continuous Event History under the same Workflow ID and Run ID spanning the kill, with previously completed Activities appearing exactly once and no duplicate executions after the restart, ending in a single completed Event. A silent restart would appear as a new Run ID (or new Workflow ID) with Activities re-executing from step one. Same Run ID plus no re-run side effects is the proof of durable replay rather than a fresh start.

Q: Why is "the Activity ran but its result was lost to a crash before recording" the classic cause of a duplicated side effect, and how is it mitigated?
A: If a Worker completes an Activity's side effect but crashes before the completion Event is persisted, the Service never sees success, so on retry the Activity runs again and the effect happens twice — at-least-once delivery in action. The mitigation is Activity idempotency: key the operation so the second run detects the first and becomes a no-op. This is exactly why non-idempotent external calls (payments, SWIFT messages, register writes) must carry an idempotency key in any durable-execution design.

Q: A Workflow is wedged after a bad deploy and you have already fixed the code — how can `temporal workflow reset` recover it?
A: `temporal workflow reset` rewinds an Execution to an earlier point in its Event History (you target an `--event-id` of a `WorkflowTaskStarted`/`WorkflowTaskCompleted` boundary, or use a `--reset-type` like `FirstWorkflowTask` or `LastWorkflowTask`) so history up to that point is kept and everything after is re-executed against the fixed code. This recovers a Workflow blocked by a non-determinism error without losing the completed work before the reset point. You typically reset to just before the offending change, then let the corrected Workers replay forward.
