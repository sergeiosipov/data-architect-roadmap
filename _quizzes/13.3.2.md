# Phase 8 · 13.3.2 Cross-org data sharing (Delta Sharing) — 150 self-test questions

Q: What is Delta Sharing in one sentence?
A: Delta Sharing is an open, REST-based protocol for securely sharing live data — primarily Delta Lake tables — across organizations without copying the underlying files. The provider runs a sharing server that hands recipients short-lived pre-signed URLs pointing directly at the Parquet objects, so consumers read the same physical data the provider owns rather than receiving a duplicate. This makes it the modern alternative to the SFTP-and-CSV file exchange that fund administration still leans on.

Q: Why is "open protocol" the load-bearing phrase in Delta Sharing's description?
A: Open means the wire protocol is a published spec (`PROTOCOL.md`) with a permissive license, so any client or server can implement it without buying into one vendor's platform. That is the contrast with Snowflake Secure Data Sharing or Databricks-native shares, which only work account-to-account inside their own ecosystems. For a fund manufacturer with 40 distributors on heterogeneous stacks, an open protocol means consumers can use pandas, Spark, Power BI, or their own client rather than all being forced onto one walled garden.

Q: What does "format-level access without copying" mean in Delta Sharing?
A: The recipient reads the provider's actual Parquet files in place — the sharing server returns pre-signed object-store URLs and the client downloads those bytes directly from S3/ABFS/GCS. There is no ETL job duplicating the table into the consumer's storage, so there is no second copy to drift, re-key, or get stale. The "format level" part means the consumer works with columnar Parquet plus Delta metadata, not a flattened CSV export that loses types and statistics.

Q: Name the core REST endpoints a Delta Sharing server exposes.
A: The protocol defines `GET {prefix}/shares` to list shares, `GET .../shares/{share}` to fetch one share, `GET .../shares/{share}/schemas` to list schemas, `GET .../shares/{share}/schemas/{schema}/tables` to list tables, `GET .../shares/{share}/all-tables` for all tables in a share, plus per-table `.../version`, `.../metadata`, a `POST .../query` to read data, and `GET .../changes` for the change data feed. The read path is a `POST` because the client sends a JSON body with hints; the list and metadata paths are plain `GET`s.

Q: Why is the table read endpoint a `POST` while listing shares is a `GET`?
A: Reading a table accepts a JSON request body carrying `predicateHints`, `jsonPredicateHints`, `limitHint`, and version/timestamp parameters, and bodies belong on `POST` not `GET`. Listing shares or schemas takes no body — only optional `maxResults` and `pageToken` query params — so a cacheable, idempotent `GET` is correct. Keeping the semantics aligned with HTTP verbs is part of why the protocol is easy to implement against any gateway.

Q: What is a Delta Sharing profile file?
A: It is a small JSON file the recipient holds that tells their client where the sharing server is and how to authenticate. The fields are `shareCredentialsVersion`, `endpoint`, `bearerToken`, and `expirationTime`. You point any Delta Sharing client at this file and it can then list and read whatever the token is authorized to see.

Q: What does each field in the profile file mean?
A: `shareCredentialsVersion` is the schema version of the file (currently always `1`); `endpoint` is the sharing server URL and is identical for every share from the same server; `bearerToken` is the secret that authenticates the recipient; and `expirationTime` marks when that token stops working. A value like `9999-12-31` in `expirationTime` is the convention for a token that effectively never expires, which you should treat as a governance smell.

Q: How does a Delta Sharing client authenticate to the server?
A: It sends the `bearerToken` from the profile file in the standard HTTP `Authorization: Bearer <token>` header on every request. The server maps that token to a recipient and the access grants that recipient holds, then scopes every response accordingly. There is no separate handshake — the bearer token is the whole credential, which is exactly why short expiry and revocation matter.

Q: In the Python client, what is the table URL format?
A: It is `profile_file_path#share.schema.table` — the path to the profile JSON, a `#` separator, then the fully qualified three-part name. For example `/home/me/config.share#fund_gold.nav.daily_nav` selects the `daily_nav` table in schema `nav` of share `fund_gold`. The profile path itself may be local or a cloud URL via fsspec, and `/dbfs/` is supported on Databricks.

Q: What does `delta_sharing.load_as_pandas(table_url)` do?
A: It reads the shared table addressed by the `profile#share.schema.table` URL into a pandas DataFrame, fetching the pre-signed file URLs and downloading the Parquet directly. It accepts optional arguments like `limit` to cap rows, `version` for time travel, and `use_delta_format`/`convert_in_batches` for large reads. It is the fastest way for a non-Spark "distributor" environment to consume a share.

Q: What does `delta_sharing.SharingClient(profile)` give you that `load_as_pandas` does not?
A: `SharingClient` is the discovery handle: `client.list_shares()`, `list_schemas(share)`, `list_tables(schema)`, and `list_all_tables()` let a recipient enumerate exactly what their token can see before reading anything. `load_as_pandas`/`load_as_spark` are the data-fetch functions that act on a specific table URL. In a consumer onboarding flow you typically `SharingClient.list_all_tables()` first to confirm scope, then load the table you want.

Q: When would you use `load_as_spark` instead of `load_as_pandas`?
A: Use `load_as_spark` when the consumer runs on a Spark cluster and the shared table is too large to fit one machine's memory, since Spark reads the Parquet files in parallel across executors. `load_as_pandas` pulls everything into a single process, which is fine for a daily NAV table but not for a multi-billion-row tick history. Choosing the client is a function of consumer sophistication and data volume — a theme of the channel rubric.

Q: What is a "share" in Delta Sharing governance terms?
A: A share is the named, top-level container of what a provider exposes — a logical bundle of one or more schemas and tables that recipients can be granted. It is the unit you grant to a recipient, so it functions as the boundary of a data product. You might have a `nav_distribution` share holding only the gold NAV schema, kept separate from an internal `risk` share.

Q: What is a "recipient" in Delta Sharing?
A: A recipient is the identity the provider grants a share to — it owns a bearer token (delivered via a profile file) and the set of shares mapped to that token. In the open reference server the recipient is essentially the token plus its authorized shares; in Databricks Unity Catalog a recipient is a first-class object with its own credential lifecycle. Granting a share to a recipient is what turns a share into something the outside world can read.

Q: How is access granted, scoped, and revoked at a high level?
A: Access is granted by mapping a recipient (token) to one or more shares; it is scoped by which schemas and tables a share contains and by table-level partition filters where supported. Revocation is done by invalidating the recipient's token or removing the share-to-recipient grant, after which the next request fails authorization. The whole lifecycle — grant, scope, rotate, revoke — is the governance story SFTP feeds historically lacked.

Q: What governance evidence does a Delta Sharing setup produce that an SFTP feed does not?
A: The server can log every authenticated request — which recipient/token read which share, schema, and table, and when — giving an auditable access trail per consumer. Token issuance, rotation, and revocation are explicit, timestamped events, so you can prove when access started and ended. With anonymous SFTP-and-CSV you typically have only a file-transfer log and no concept of a revocable, per-recipient credential.

Q: How is the reference sharing server configured?
A: Through a YAML file, conventionally `delta-sharing-server.yaml`, that lists each share, its schemas, and each table's name and storage location (an `s3a://`, `abfs://`, `gs://`, or local `file:` path), plus an `authorization.bearerToken`. You declare the catalog statically in this file; the server then serves exactly those tables to anyone presenting the configured token. Cloud credentials for reaching the storage come from environment variables or Hadoop config, not the YAML itself.

Q: How do you run the open-source reference server?
A: Either from the packaged distribution with `bin/delta-sharing-server --config <yaml-file>` (a JVM process, since the reference server is JVM-based) or via Docker with `docker run -p <port> --mount <config> deltaio/delta-sharing-server`. It then exposes the REST endpoints on the configured host and port. For the lesson's `Do` step you point it at a gold Delta table directory and start serving immediately.

Q: The reference server starts but a recipient gets `401 Unauthorized` on every call — first thing to check?
A: Confirm the `bearerToken` in the recipient's profile file exactly matches the `authorization.bearerToken` configured on the server, with no trailing whitespace or stale value after a rotation. Bearer auth is an exact-string match, so a single mismatched character fails every request. Also verify the client is actually sending the `Authorization` header — a malformed profile `endpoint` can cause the client to hit the wrong path.

Q: The recipient can list tables but `load_as_pandas` fails to download files — what is the likely cause?
A: Listing only touches the sharing server, but downloading hits the object store via pre-signed URLs, so a failure here usually means the server cannot mint valid URLs or the URLs have expired. Check that the server's own cloud credentials (env vars / Hadoop config) are correct and that the configured `s3a://`/`abfs://` table path exists. If the URLs were generated long before the client used them, the short pre-signed expiry may have elapsed.

Q: How does the server return table data to the client?
A: As newline-delimited JSON (`application/x-ndjson`): a protocol-version object, a metadata object carrying the schema, then one object per data file with its pre-signed URL, partition values, size, and statistics. The client parses these lines, downloads the referenced Parquet files directly from storage, and assembles the result. The server never streams the row bytes itself — it only hands out signed references.

Q: Why does Delta Sharing use pre-signed URLs rather than proxying the data through the server?
A: Pre-signed URLs let the recipient pull Parquet straight from the object store, so the sharing server stays lightweight and does not become a bandwidth bottleneck or a single point of failure for large reads. It also means the heavy egress is a storage-layer concern, which matters for both scaling and cost attribution. The trade-off is that those URLs are short-lived secrets that grant temporary read access to specific objects.

Q: What is the security risk of pre-signed URLs and how is it mitigated?
A: A pre-signed URL is a bearer capability — anyone who obtains it can read that object until it expires, regardless of the sharing token. Mitigation is short expiry (minutes), so a leaked URL is useless quickly, and scoping each URL to one specific object rather than a whole bucket. This is why you cannot rely on the pre-signed URL alone for long-term access control; the recipient's sharing token is the durable, revocable credential.

Q: What is `predicateHints` in a table query?
A: It is a list of SQL boolean expressions the client sends in the read request so the server can skip data files whose partition values cannot match, reducing the number of pre-signed URLs and bytes downloaded. For example `date >= '2026-01-01'` lets the server drop older partitions. Crucially it is BEST EFFORT — the server may ignore it — so the client must still apply the filter to the data it receives.

Q: How does `jsonPredicateHints` differ from `predicateHints`?
A: `jsonPredicateHints` expresses the same partition-filtering intent as a structured JSON tree rather than a raw SQL string, which is less ambiguous to parse and safer for servers to evaluate. Both are partition-pruning hints and both are best-effort. The JSON form was introduced to make predicate handling more robust across implementations than free-text SQL.

Q: What is `limitHint` and why is it only a hint?
A: `limitHint` tells the server the maximum number of rows the client ultimately wants, letting the server return fewer files when a small sample suffices. It is a hint because file-level granularity means the server returns whole Parquet files, so the actual row count handed back is usually more than the limit. The client still enforces the exact `limit` locally after download.

Q: Why does the protocol stress that filtering is "BEST EFFORT"?
A: Because the server prunes at file/partition granularity using statistics, not row-by-row, so it can never guarantee that returned files contain only matching rows. A correct client therefore always re-applies the predicate after downloading, treating hints purely as an optimization to cut transfer volume. Relying on the server to fully filter would risk returning extra rows the consumer should not assume away.

Q: How do you read a specific historical version of a shared table?
A: Pass a `version` (or `timestamp`) parameter — in the Python client, `delta_sharing.load_as_pandas(table_url, version=N)` — and the server resolves the read against that Delta table version. This is time travel over the share, useful for reproducing a NAV as it stood on a prior valuation date. The provider must have retained that version (Delta history not yet vacuumed) and enabled history sharing.

Q: What does the change data feed endpoint provide?
A: `GET .../changes` (exposed in the client via `load_table_changes_as_pandas`/`_as_spark`) returns the row-level inserts, updates, and deletes between a `startingVersion` and `endingVersion` (or timestamps), rather than a full table snapshot. This lets a distributor incrementally sync only what changed since their last pull. It requires Change Data Feed to be enabled on the source Delta table.

Q: What is the `Delta-Table-Version` response header for?
A: It tells the client which Delta table version the returned data corresponds to, so the consumer can record exactly what they read and detect when the table has advanced since. A distributor can store this version and on the next pull request only the changes after it via the change feed. It turns each read into a reproducible, version-stamped event.

Q: What is the `delta-sharing-capabilities` header used for?
A: It is how client and server negotiate optional features — for example whether responses come as pre-signed Parquet URLs or in native Delta format with deletion vectors. The client advertises what it supports and the server responds in a compatible mode. This negotiation lets the protocol evolve without breaking older clients.

Q: How do `maxResults` and `nextPageToken` implement pagination?
A: List endpoints accept `maxResults` to cap items per response and return a `nextPageToken` when more results remain; the client passes that token back as `pageToken` to fetch the next page. You loop until no `nextPageToken` is returned. This keeps responses bounded when a provider exposes many shares or tables.

Q: A consumer paginating shares loops forever — what is the bug to look for?
A: Almost certainly the client is not feeding the returned `nextPageToken` back as the `pageToken` on the next request, so it keeps re-fetching page one. Correct loops stop precisely when the response omits `nextPageToken`. A second possibility is treating an empty-string token as "more pages" rather than as the terminating condition.

Q: What is "directory-based access" / temporary table credentials in the protocol?
A: Instead of pre-signing each file URL, the server can issue short-lived cloud credentials (AWS STS tokens, Azure SAS, GCP OAuth) scoped to the table's storage directory, via `POST .../temporary-table-credentials`. The client then reads the table directory using normal cloud-storage APIs. It suits clients that prefer native Delta reads with features like deletion vectors over a list of per-file URLs.

Q: Walled gardens — what is Snowflake Secure Data Sharing?
A: It is Snowflake's in-ecosystem sharing where a provider account grants a consumer account access to databases, schemas, tables, and secure views without copying data. Sharing happens through Snowflake's services layer and metadata store, so the consumer queries the provider's data live. It is account-to-account inside Snowflake — a walled garden, not an open protocol any client can speak.

Q: How does Snowflake achieve zero-copy sharing?
A: No data is physically moved between accounts; the consumer creates a read-only database from the share that is just a set of metadata pointers into the provider's storage, resolved through Snowflake's services layer. The consumer sees live data and pays no storage for it. This is conceptually the same "no second copy" benefit Delta Sharing gives, but it only works between Snowflake accounts.

Q: What is a Snowflake reader account and why does it exist?
A: A reader account is a special Snowflake account a provider creates and pays for so that a consumer who is not a Snowflake customer can still query a share. The reader can only consume from the provider that created it. It exists precisely to reach non-Snowflake consumers without forcing them to buy their own Snowflake — the gap an open protocol closes more naturally.

Q: What are Snowflake Listings and the Snowflake Marketplace?
A: Listings are the packaging mechanism for offering a share more broadly than one-to-one account grants, and the Marketplace is the public catalog where providers publish data products for many consumers to discover and request. They are the discovery and distribution layer on top of secure shares. The Databricks Marketplace plays the analogous role for Delta Sharing inside the Databricks ecosystem.

Q: What is the Databricks Delta Sharing relationship to the open protocol?
A: Databricks implements the open Delta Sharing protocol natively through Unity Catalog, so it can both serve shares to any open-protocol client (open sharing) and exchange shares with other Databricks/Unity-Catalog accounts (Databricks-to-Databricks sharing). The wire protocol is the same spec the reference server uses. The difference from the bare reference server is the governance, recipient management, and auditing Unity Catalog layers on top.

Q: What does the open protocol buy you over Snowflake/Databricks-native shares?
A: It frees consumers from your platform choice — any pandas, Spark, or BI client that speaks the protocol can read, so 40 distributors on different stacks all work without onboarding to your vendor. It is vendor-neutral, license-permissive, and self-hostable. The native shares give richer built-in governance and zero operational burden, but only to consumers already inside that same ecosystem.

Q: What does the open protocol cost you to run compared to a native share?
A: You must operate the sharing server yourself — deploy it, patch it, secure the endpoint, manage object-store credentials, and build the recipient/token lifecycle and audit logging the reference server does not provide out of the box. Snowflake and Databricks shares offload all of that to the platform. So the open protocol trades operational responsibility for ecosystem independence.

Q: What is "egress economics" in the context of data sharing?
A: It is the question of who pays for the bytes and compute when data is read, and how the answer changes with the sharing model. With Delta Sharing the recipient downloads Parquet directly from your object store, so the provider typically eats cloud egress charges per read. Knowing where the cost lands is essential before promising "free" access to 40 distributors.

Q: In Snowflake Secure Data Sharing, who pays for what?
A: The provider bears the storage cost for the shared data, while the consumer pays for the compute (virtual warehouses) used to query it. So read cost is pushed onto the consumer, and the provider's marginal cost per consumer query is largely their own storage, not per-read egress. This is a different cost shape from Delta Sharing, where provider-side egress scales with how much each recipient downloads.

Q: Why does no-copy sharing change the cost conversation versus file drops?
A: With CSV/SFTP file drops you pay to materialize, store, and transfer a full export to every consumer on every cycle, multiplying storage and transfer by the number of consumers. No-copy sharing keeps one authoritative copy and serves reads from it, so you pay for storage once plus per-read egress. The cost driver shifts from "number of exported copies" to "how much each consumer actually reads."

Q: In Delta Sharing, can heavy reads by one distributor create a surprise cloud bill — and why?
A: Yes, because every `load_as_pandas`/`load_as_spark` pulls Parquet from the provider's object store, generating cloud data-egress charges the provider is billed for. A distributor that re-downloads the full table hourly instead of using the change feed can drive significant egress. This is why the rubric scores consumer sophistication and why nudging consumers toward incremental CDF reads is a real cost lever.

Q: What is the "channel rubric" this lesson asks you to write?
A: A decision framework for choosing among sharing protocol, API, and file-drop for a given consumer, scored on volume, latency, consumer sophistication, and audit needs. It exists so an architect picks the right channel per consumer class instead of defaulting to 40 bespoke SFTP feeds. The output is a defensible mapping from "what this consumer is like" to "which channel they get."

Q: When does a sharing protocol like Delta Sharing win over an API?
A: When consumers want large, analytical, table-shaped reads — full datasets or big slices — and are sophisticated enough to run pandas/Spark/BI clients. The protocol gives them columnar Parquet at scale with no per-row request overhead and no copy. An API is a poor fit for "give me the whole table" because pagination and JSON serialization make bulk pulls slow and expensive.

Q: When does an API beat a sharing protocol for fund data?
A: When the consumer needs small, operational, low-latency point lookups — "the current NAV for this ISIN right now" for a website or partner system — rather than bulk analytics. An API gives a stable request/response contract, fine-grained authz, quotas, and chargeback per call. That is the subject of the next lesson (data APIs and gateways); the rubric is about knowing which consumer gets which.

Q: When is an EMT/EPT file-drop still the right channel?
A: When the consumer is a regulated counterparty whose process is built around receiving a specific standardized file on a schedule — the European MiFID Template or European PRIIPs Template — and where the file format itself is the contractually mandated deliverable. Low sophistication, fixed cadence, and a regulator-defined layout favor file-drop. The rubric should acknowledge that some consumers cannot or will not adopt a protocol no matter how superior it is.

Q: Score "a regulator ingesting your full transaction history nightly" on the rubric.
A: High volume, batch latency tolerance, and an audit requirement point toward either a sharing protocol or a controlled file-drop rather than an API. If the regulator can run a protocol client, Delta Sharing gives them no-copy access plus a clean revocation and audit trail; if they mandate a fixed file, you drop the file. The decisive factors are volume (high) and the consumer's ability and willingness to adopt a protocol.

Q: Score "a distributor's website showing today's NAV per share class" on the rubric.
A: Low volume per request, low-latency need, point lookups by ISIN/share class, and a relatively simple consumer point to an API, not a sharing protocol. Bulk no-copy access is overkill and would expose more than the consumer needs. This is the classic case where the rubric routes the consumer to the API/gateway channel rather than Delta Sharing.

Q: How does "consumer sophistication" change the channel choice concretely?
A: A sophisticated consumer running Spark can consume a Delta Sharing share directly and incrementally; an unsophisticated consumer who can only drop a file onto an SFTP server and open it in Excel needs a file-drop or a simple API. Forcing a protocol on a consumer who cannot run a client just recreates the failed-feed problem in a new wrapper. The rubric makes you match the channel to what the consumer can actually operate.

Q: What is the "contractual overlay" on a share?
A: The principle that a share is a data product, so it carries a contract: an agreement on schema, refresh SLOs, license/usage terms, and support, not just a token. This ties back to data-contract and data-product thinking (13.1.1) and license/legal concerns (A.18). The technical revocation story is necessary but not sufficient — the legal and SLO terms ride alongside the bearer token.

Q: Why is "a share is still a product" more than a slogan?
A: Because consumers build real processes on it, an unannounced schema change or a missed refresh breaks them exactly like a broken API would. Treating the share as a product means versioning its schema, publishing SLOs for freshness, and defining license terms and a deprecation path before you grant access. Without that overlay you have a technically clean pipe with no accountability around what flows through it.

Q: What SLOs would you attach to a daily NAV share?
A: Freshness (the table reflects the new valuation date by an agreed cutoff each business day), completeness (all expected share classes present), availability of the sharing endpoint, and a defined notice period for schema changes. These are the promises a distributor's downstream NAV publication depends on. They belong in the data contract that accompanies the share, not in tribal knowledge.

Q: Why is license/usage terms (A.18) relevant to a Delta Sharing arrangement?
A: Because granting read access does not automatically grant rights to redistribute, resell, or commingle the data, and fund/benchmark data often carries strict licensing. The contractual overlay must state permitted use, redistribution limits, and retention obligations. The protocol enforces who can read; the license governs what they may do with what they read.

Q: Walk through the lesson's hands-on `Do` step 1.
A: Stand up the reference sharing server over a gold Delta table, writing a `delta-sharing-server.yaml` that defines one share exposing one schema, with that schema's table(s) pointing at the gold table's storage path, and an `authorization.bearerToken` for one recipient. Then start it with `bin/delta-sharing-server --config delta-sharing-server.yaml`. This is the minimal provider setup: one share, one schema, one recipient token.

Q: Walk through `Do` step 2.
A: From a separate "distributor" environment, install the `delta-sharing` Python client, write a profile file with the server `endpoint` and the recipient `bearerToken`, and call `delta_sharing.load_as_pandas("profile.share#share.schema.table")`. Then compare the resulting DataFrame against the source gold table to prove the data matches. The point is that there is no copy pipeline in between — the distributor reads the provider's actual files.

Q: Walk through `Do` step 3 (the revocation demo).
A: Rotate the recipient credential — issue a new `bearerToken`, confirm the consumer must update their profile to keep working — then revoke it by removing/invalidating the token and show the next `load_as_pandas` failing with an authorization error. This demonstrates the lifecycle SFTP never cleanly had: access that can be ended on demand and proven to have ended. You also capture the server-side log entries as governance evidence.

Q: On the reference server, what is the practical way to "revoke" a token?
A: Since the reference server reads its token from the YAML config, you revoke by changing/removing that `bearerToken` (or the recipient's grant) and restarting/reloading the server so the old token no longer authenticates. After that the consumer's next request returns `401`. In a production system like Unity Catalog you would rotate or delete the recipient credential through the catalog's API instead of editing a file.

Q: Walk through `Do` step 4.
A: Write the channel rubric: a table or scoring scheme that, for each consumer class, weighs volume, latency, consumer sophistication, and audit needs to recommend sharing protocol vs API vs EMT file-drop. The deliverable is a reusable decision tool, not a one-off answer. It is what lets you defend the dissemination design for 40 distributors per consumer class rather than building 40 ad-hoc feeds.

Q: Design challenge — disseminating to a fund manufacturer's 40 distributor relationships: what is the failure mode you are avoiding?
A: The folklore failure mode is 40 bespoke SFTP-and-CSV feeds, each with its own format drift, its own ad-hoc schedule, and no per-recipient revocation. Every new distributor adds another bespoke pipeline to maintain and another way the format can quietly diverge. The architecture goal is to collapse those into a small number of governed channels chosen by consumer class.

Q: For those 40 distributors, how do you defend the channel per consumer class?
A: Segment distributors by what they actually consume and can operate: large platforms that want full datasets and can run Spark get a Delta Sharing share; operational consumers needing live point lookups get the gated API; and counterparties bound to standardized regulatory templates get the EMT/EPT file-drop. Each assignment is justified by volume, latency, sophistication, and audit needs from the rubric. The defense is that every consumer gets the cheapest channel that meets their real requirements.

Q: How does Delta Sharing improve the format-drift problem of 40 SFTP feeds?
A: With no-copy sharing every recipient reads the same authoritative gold Delta table with its enforced schema, rather than each receiving a separately generated CSV that can drift. There is one schema to govern and version, not forty export formats. Drift becomes a single contracted change to the shared table instead of forty silent divergences.

Q: ISIN scenario — a distributor complains your shared NAV table changed a column and broke their load. What governance failure does this expose?
A: It exposes a missing or unenforced data contract on the share — schema changes were made without versioning, advance notice, or a deprecation path, even though the share is a product. The fix is the contractual overlay: pin the share's schema, publish a change-notice SLO, and offer a versioned transition. The protocol guarantees secure access, but only the contract guarantees the column stays where the distributor expects.

Q: A distributor needs only the rows for ISINs they distribute — how can you scope that?
A: Partition the gold table so the distributor's ISINs are a filterable dimension, expose a share whose grant or partition filter limits them to those partitions, and rely on `predicateHints` for efficient pruning on read. For row-level isolation beyond partition scope you would expose a per-recipient secure view or a dedicated table rather than the whole dataset. The principle is to grant the narrowest share that meets the need.

Q: Why might you expose a secure/gold view to a recipient rather than the raw table?
A: A view lets you project only the columns and rows a given recipient is licensed to see, mask sensitive fields, and present a stable contract even if the underlying physical table changes. The recipient reads the view as their table without knowing the internals. This is the standard way to enforce least privilege and decouple the shared contract from internal storage.

Q: Why share the gold layer rather than bronze or silver?
A: Gold is the cleaned, conformed, business-ready data with an agreed schema and quality — exactly what an external consumer should depend on. Bronze/silver are internal, raw or in-progress, and their schemas churn, so sharing them would leak internal mess and break consumers constantly. Disseminating gold keeps the external contract stable and the internal layers free to evolve.

Q: How does token rotation work and why do it on a schedule?
A: Rotation issues a new bearer token to a recipient and retires the old one after a transition window, so the consumer updates their profile while access stays continuous. Doing it on a schedule limits the blast radius if a token leaks — a stale credential stops working soon regardless. It is the same hygiene as rotating API keys; the profile file just makes the new token easy to distribute.

Q: What is the difference between rotating and revoking a credential?
A: Rotation replaces a recipient's token with a new one while preserving their access — continuity with a fresh secret. Revocation ends the recipient's access entirely by invalidating the token/grant with no replacement. The lesson asks you to demonstrate both because together they prove access can be refreshed and definitively terminated, the lifecycle SFTP lacked.

Q: How do you prove to an auditor that a former distributor can no longer read your data?
A: Show the revocation event (token/grant removed with a timestamp), then a captured request from that recipient returning `401 Unauthorized` after that time, and the absence of any successful read in the server access log afterward. That trio — revocation record, failed request, clean log — is the governance evidence. It is exactly the assurance an open SFTP drop could not give.

Q: Why is per-recipient tokenization better than one shared SFTP credential for many partners?
A: A unique token per recipient means you can scope, rotate, and revoke each partner independently and attribute every read to a specific recipient in the logs. One shared SFTP login gives you no attribution and forces you to change everyone's credential to cut off one partner. Per-recipient identity is the foundation of both audit and least-privilege revocation.

Q: How does Delta Sharing relate to DORA-style operational-resilience expectations?
A: DORA pushes financial entities toward controlled, monitored, and revocable third-party data access with clear audit trails — exactly the grant/scope/rotate/revoke lifecycle and access logging Delta Sharing supports, in contrast to opaque SFTP folklore. Running the sharing server also makes the dissemination dependency explicit and testable. It does not by itself make you DORA-compliant, but it supplies controls regulators expect to see.

Q: How does LEI fit a Delta Sharing recipient model?
A: The Legal Entity Identifier gives each counterparty a globally unique identity, so you can map each recipient/token to a specific legal entity rather than an anonymous login. That makes audit logs meaningful ("entity with this LEI read this share") and ties technical access to a real legal party for the contract and license. It strengthens the contractual overlay by anchoring the recipient to an externally verifiable identity.

Q: Could SWIFT or a messaging network replace Delta Sharing for fund data dissemination?
A: No — SWIFT is a secure financial-messaging network optimized for structured transactional messages (settlement, payments, order instructions), not for serving large analytical datasets. Delta Sharing addresses bulk, table-shaped, no-copy analytical access. They are complementary channels for different payloads; the rubric would route a structured transactional instruction to messaging and a full dataset to the sharing protocol.

Q: A fund administrator in Luxembourg must share NAV data with distributors across the EU — why does region matter for the sharing model?
A: Cross-border reads can incur cross-region egress costs and trigger data-residency considerations under EU rules, and with native shares (e.g. Snowflake) cross-region sharing requires replicating the data into the consumer's region. With Delta Sharing the consumer pulls from your object store, so region affects egress and latency but not a forced copy. Either way, region is a cost-and-compliance factor the channel design must account for.

Q: For Snowflake cross-region/cross-cloud sharing, what extra step is required?
A: You must replicate the shared database to an account in the consumer's region or cloud before sharing, because Snowflake's zero-copy sharing works within a region/cloud; cross-region introduces a replicated copy. That replication adds storage and transfer cost and a freshness lag. It is a concrete way the walled garden's "no copy" guarantee weakens at region boundaries — a point worth contrasting with the open protocol.

Q: Why is Delta Sharing often called "the revocation story SFTP never had"?
A: With SFTP-and-CSV, once a file is delivered you cannot recall it, and shutting off a partner means changing a shared login with no per-recipient granularity or proof. Delta Sharing replaces the file handoff with per-recipient tokens you can rotate and revoke, plus access logs, so you can end and evidence the ending of access. The data is read live from your store, so cutting the token cuts future reads.

Q: What does "no copy pipeline in between" mean for data freshness?
A: Because the recipient reads the provider's live table rather than a periodically exported copy, they always see the latest committed Delta version with no batch lag introduced by an ETL hop. When the provider commits a new NAV, the next read reflects it. This removes a whole class of "the file is stale" complaints inherent to scheduled exports.

Q: What are the trust/security implications of running your own open sharing server on the public internet?
A: You own the attack surface: the endpoint must terminate TLS, validate bearer tokens, rate-limit abusive recipients, and keep object-store credentials from leaking, since a compromised server can mint pre-signed URLs to your data. The reference server is a starting point, not a hardened gateway, so you typically front it with proper auth, logging, and network controls. This operational burden is part of "what it costs you to run" the open protocol.

Q: Why might you put the sharing server behind an API gateway or reverse proxy?
A: To add TLS termination, centralized authentication/rate-limiting, request logging, and WAF protections the bare reference server does not provide, and to get per-recipient quotas and analytics. The gateway becomes where operational controls live, leaving the sharing server to do protocol logic. This mirrors the gateway discipline covered for data APIs in the following lesson.

Q: How does Delta Sharing differ from physically replicating a table into a partner's warehouse?
A: Replication makes a second physical copy in the partner's storage that you must keep in sync, that drifts, and that you cannot revoke once landed; Delta Sharing keeps one copy and serves governed, revocable reads from it. Replication shifts storage cost and control to the consumer permanently. Sharing keeps the provider authoritative and retains the ability to cut access.

Q: What is the latency profile of a Delta Sharing read versus a streaming feed?
A: Delta Sharing is pull-based batch-style access: the consumer fetches the current (or a versioned) snapshot when they ask, so freshness is bounded by how often they read, not by sub-second streaming. For near-real-time needs you would use a streaming channel or an event feed instead. The rubric's latency axis is precisely about steering low-latency consumers away from the sharing protocol.

Q: A consumer wants minute-level updates of changed rows without re-reading the whole table — what do you point them to?
A: The Change Data Feed: enable CDF on the source table and have them call `load_table_changes_as_pandas`/`_as_spark` between their last-seen `Delta-Table-Version` and the current one. They get only inserts/updates/deletes since their last pull, which is far cheaper than full snapshots and reduces your egress. This is the incremental pattern that keeps a "frequent reader" affordable.

Q: Why might a `version=N` time-travel read fail even though the table has that history?
A: Delta's `VACUUM` may have removed the physical files for old versions, or history sharing may be disabled on the share, so the server can no longer materialize that version. Time travel only works while the underlying files and metadata are retained and exposed. If reproducibility of past NAVs is a requirement, you must set retention and enable history sharing deliberately.

Q: How would you let a consumer verify that what they read matches your source with no copy in between?
A: Have them load the share and compare against the source table on a stable key and a checksum or row count for a fixed table version, confirming identical rows and values. Because they are reading your actual Parquet files via pre-signed URLs, a match demonstrates there is no transforming copy pipeline distorting the data. Pinning a `version` makes the comparison deterministic even if the table is being updated.

Q: What is the role of `client.list_all_tables()` during recipient onboarding?
A: It lets the new recipient confirm exactly which tables their token can see before they wire up any read, validating that the provider scoped the grant correctly. If it returns more (or fewer) tables than agreed, the share configuration is wrong and you fix it before go-live. It is the cheap pre-flight check that the grant matches the contract.

Q: Why is the profile file a sensitive artifact, and how should it be handled?
A: It contains the `bearerToken`, which is the full credential to read everything the recipient is granted, so leaking it is equivalent to leaking access. Treat it like any secret: deliver over a secure channel, store in a secrets manager not in source control, and rotate if exposure is suspected. The short `expirationTime` limits damage but does not replace good handling.

Q: What happens when a token reaches its `expirationTime`?
A: Requests authenticated with that token start failing authorization, so the recipient's reads stop until they receive a refreshed token/profile. A near-term expiry is a deliberate control that forces periodic re-issuance and bounds the lifetime of a leaked credential. This is why an `expirationTime` of `9999-12-31` (never expires) is generally a governance red flag.

Q: Why is `shareCredentialsVersion` in the profile file even though it is always `1`?
A: It is forward-compatibility plumbing: the field lets the credential file format evolve, so a future version `2` could add fields and clients can dispatch on the version. Today it is always `1`, but encoding the version up front means old and new clients can coexist when the format changes. It is the same versioning hygiene you would apply to any wire format.

Q: How does Delta Sharing handle a consumer who only speaks SQL/BI, not Python or Spark?
A: Tools like Power BI, Tableau, and other BI clients have native Delta Sharing connectors that consume a profile file directly, and you can also stand up a thin query layer in front of the share. So "not a coder" does not force you to a file-drop, as long as their BI tool has a connector. Where no connector exists and sophistication is low, the rubric may still route them to a file-drop.

Q: Contrast the audit trail of Delta Sharing with an EMT file-drop on SFTP.
A: Delta Sharing logs per-recipient, per-table reads with timestamps tied to an identity, supporting precise access attribution and revocation evidence. An EMT file-drop typically logs only that a file was placed or fetched, often via a shared login, with little per-recipient read attribution. For audit needs the protocol is stronger; for a regulator who mandates the EMT file itself, the file-drop is still required regardless.

Q: Why is "the industry still runs on SFTP-and-CSV folklore" a problem for an architect?
A: Because each bespoke SFTP feed has its own undocumented format, schedule, and credential, the estate becomes unmaintainable, drift-prone, and impossible to govern or revoke cleanly at scale. An architect who cannot compare protocol vs API vs file-drop just keeps adding to that pile. The lesson's whole point is to give you the rubric to escape it deliberately rather than by default.

Q: In the channel rubric, how does "audit needs" tip the decision?
A: High audit needs favor channels with per-recipient identity and read-level logging — a sharing protocol or a gated API — over an anonymous file-drop, because you must be able to prove who accessed what and when. A regulator-facing flow with strict accountability should not ride on a shared SFTP login. The audit axis can override a pure cost/latency choice.

Q: When does GraphQL or a bespoke API beat Delta Sharing for fund data?
A: When consumers need flexible, shape-shifting, low-latency reads of small slices and you want strict per-call authz, quotas, and chargeback — operational rather than analytical access. Delta Sharing is built for bulk table reads, not arbitrary fine-grained queries. The rubric routes such consumers to the API/gateway channel, which is the focus of the next lesson.

Q: A distributor reports they are reading "yesterday's NAV" — how do you triage with version headers?
A: Check the `Delta-Table-Version` they received against the current table version: if it lags, either the source table has not committed the new NAV yet (a provider-side freshness/SLO issue) or the consumer is pinning an old `version`/cached profile. The version header turns a vague freshness complaint into a precise comparison. Resolution is either fixing the provider's publish cadence or the consumer's read parameters.

Q: Why must even a no-copy share still respect data-residency and privacy law?
A: Because the recipient downloads real bytes across a network boundary, the data physically moves to wherever the consumer reads it, so residency, GDPR, and cross-border-transfer rules still apply. "No copy in the provider's storage" does not mean "no transfer." The contractual overlay and channel design must account for where each recipient pulls data to.

Q: What single dependency does standing up the open reference server add to your operational risk register?
A: The sharing server itself becomes a critical dissemination component — if it is down, mis-credentialed, or compromised, every distributor relying on Delta Sharing loses access or is exposed. That is a concentration risk you must monitor, patch, and have a recovery plan for, which DORA-style resilience expects you to document. The native-share alternative offloads that risk to the platform vendor at the cost of ecosystem lock-in.

Q: How can a consumer cheaply poll whether a shared table has new data before downloading it?
A: They call the lightweight `GET .../tables/{table}/version` endpoint, which returns just the current Delta table version without transferring any data files. If that version is unchanged since their last read they skip the download entirely; if it advanced they fetch only the change feed from their last-seen version. This is far cheaper than re-reading the whole table to discover nothing changed, and it directly cuts provider egress.

Q: Summarize the architect's takeaway: protocol vs API vs file-drop in one frame.
A: Use a sharing protocol like Delta Sharing for high-volume, batch-tolerant, sophisticated consumers who want governed no-copy table access; use an API/gateway for low-volume, low-latency, operational point lookups needing per-call control; and use an EMT/EPT file-drop for low-sophistication or regulation-mandated fixed-format consumers. Score each consumer on volume, latency, sophistication, and audit needs, and treat every channel as a contracted product. That rubric is what replaces forty bespoke SFTP feeds with a defensible dissemination architecture.

Q: The query response includes an `endStreamAction` with an `errorMessage` even though the HTTP status was `200` — what must the client do and why does this exist?
A: The client must FAIL the query, because an `endStreamAction.errorMessage` means the server hit an error while streaming the NDJSON body after it had already committed a `200` status and could no longer change the HTTP code. It exists precisely so a server can signal mid-stream failures (e.g. a storage error part-way through listing files) that the status line cannot express. A naive client that ignores the terminal action would silently return a truncated, incomplete result set.

Q: What is the `refreshToken` returned in a query response and when does a client use it?
A: `refreshToken` is an opaque value carried in the `endStreamAction` of a snapshot query that lets the client re-request the same table version and get freshly minted pre-signed URLs without re-running the whole query plan. A client uses it when the original pre-signed URLs have expired (or are about to) before it finished downloading all the files. It keeps a long-running bulk read of a large NAV history alive past the short URL expiry window.

Q: What does `minUrlExpirationTimestamp` in the response tell the client, and how should the client use it?
A: `minUrlExpirationTimestamp` is the earliest expiry time across all the pre-signed URLs in that response, so the client knows the deadline by which it must finish downloading before any URL goes stale. A well-behaved client compares it to the current time and, if the window is too tight for the data volume, refreshes URLs via the `refreshToken` rather than racing the clock. It turns "URLs expire eventually" into a concrete, machine-checkable deadline.

Q: A `load_as_pandas` on a very large shared table fails partway with expired-URL errors despite the table being fine — what is the fix?
A: The pre-signed URLs minted at query time expired before the client finished downloading all the Parquet files, so the fix is to use a client that honours the `refreshToken`/`minUrlExpirationTimestamp` to re-sign URLs, or to read with Spark which fetches in parallel and finishes within the window. Pruning with `jsonPredicateHints` or `limitHint` to cut the file count also helps. The underlying cause is download wall-clock time exceeding the URL TTL, not an auth or data problem.

Q: What is `responseFormat=parquet` versus `responseFormat=delta` in a query, and why pick `delta`?
A: `responseFormat=parquet` (the default, compatible with older clients) returns plain pre-signed Parquet file URLs, while `responseFormat=delta` returns the table in native Delta form so the client can use advanced reader features like Deletion Vectors and Column Mapping. You pick `delta` when the shared table uses those features and the consumer's client supports them; otherwise the Parquet form would either error or silently mis-read deleted rows. The client signals which formats it accepts through the `delta-sharing-capabilities` header.

Q: How does a client tell the server it can handle Deletion Vectors and Column Mapping?
A: It advertises `responseformat=delta` (and the relevant reader-feature flags) in the `delta-sharing-capabilities` request header; the server then returns data in native Delta format and echoes its chosen capabilities in the response header. Without that negotiation the server falls back to `responseformat=parquet`, which cannot faithfully represent a table that relies on deletion vectors. This handshake is how the protocol adds features without breaking clients that predate them.

Q: A shared table with Deletion Vectors is read by an old Parquet-only client and returns rows that should have been deleted — what happened?
A: The old client only requested `responseformat=parquet`, so the server handed back the underlying Parquet files but the client never applied the deletion vectors that logically remove those rows, yielding "ghost" rows. The fix is a client that requests `responseformat=delta` so deletions are honoured, or the provider rewriting the table to physically drop the rows. This is a real correctness trap when sharing tables that use modern Delta features to soft-delete.

Q: A query against a `version=N` time-travel read returns `404` on some file downloads — what is the specific cause?
A: The pre-signed URL points at a Parquet file that Delta's `VACUUM` has already physically removed, so the object no longer exists in storage even though the metadata referenced it; the documented workaround is to query the latest snapshot instead. Old versions are only readable while their files survive the retention window. If reproducing a past NAV version matters, the provider must lengthen `VACUUM` retention and not prune that history.

Q: A recipient's read fails with an error about exceeding active-file or RemoveFile limits — what is going on and who fixes it?
A: The shared Delta table has grown past the protocol's scale ceiling — on the order of 400,000 active files or 100,000 `RemoveFile` actions in the version being read — so the metadata response is rejected. The fix is on the PROVIDER side: run `OPTIMIZE` to compact small files and `VACUUM` to clear tombstones, shrinking the file count. It is a maintenance/file-hygiene problem on the source table, not a recipient permission issue.

Q: Why does sharing a table backed by an S3 bucket named with dots (e.g. `my.fund.bucket`) cause TLS failures?
A: A bucket name containing dots breaks AWS's virtual-hosted-style HTTPS wildcard certificate (`*.s3.amazonaws.com` does not match a multi-dot host), so the pre-signed-URL download fails SSL verification. The fix is to use a bucket whose name has no dots for shared data. It is a storage-naming gotcha that surfaces only at download time, not when listing tables.

Q: An open-source Spark client hits a schema-mismatch error reading a share — what config resolves it?
A: Set `spark.delta.sharing.client.useStructuralSchemaMatch` to `true` (on a recent enough `delta-sharing-client`, around `1.2.3+`) so the client matches columns structurally rather than by exact metadata, which reconciles differences between how the server and the client model the schema. Without it, otherwise-compatible tables can fail to read. It is a client-side compatibility flag, not a change to the share itself.

Q: How would you confirm an activation/profile token is genuinely working before handing it to a distributor?
A: Point a throwaway client at the profile and call `SharingClient(profile).list_all_tables()`; a successful enumeration that returns exactly the agreed tables proves the `endpoint` is reachable, the `bearerToken` authenticates, and the grant scope is correct. A `401` means the token is wrong, an empty list means the grant is missing, and extra tables mean the grant is too broad. This pre-flight catches mis-issued credentials before they reach the partner.

Q: A distributor's profile file has `shareCredentialsVersion: 2` but their client errors on load — what is the likely mismatch?
A: The client library is too old to understand a version-`2` credential file format, so it cannot parse fields a newer profile introduced; the `shareCredentialsVersion` field exists exactly so clients can detect and reject formats they do not support. The fix is upgrading the `delta-sharing` client to a version that handles v2 profiles (or re-issuing a v1 profile if the server supports it). It is a forward-compatibility guard doing its job.

Q: Give the exact Python call to read only rows changed between version 12 and version 20 of a shared table.
A: `delta_sharing.load_table_changes_as_pandas("profile.share#share.schema.table", starting_version=12, ending_version=20)` returns the CDF rows for that range, provided Change Data Feed is enabled on the source table. The result includes `_change_type` (`insert`, `update_preimage`, `update_postimage`, `delete`), `_commit_version`, and `_commit_timestamp` metadata columns. This is the incremental pull that keeps a frequent distributor's egress down.

Q: What do the `_change_type` values in a Change Data Feed result mean for a distributor applying an incremental NAV update?
A: `insert` and `delete` are straightforward adds/removals, while an update appears as a pair: `update_preimage` (the row before) and `update_postimage` (the row after), so the consumer can apply the change as a delete-then-insert or an upsert keyed on the row's identity. A distributor merging CDF into their own store must treat the pre/post pair as one logical update, not two separate rows. Ignoring the preimage/postimage distinction corrupts the merged NAV table.

Q: Why can `endingVersion` in a CDF query be treated as "not enforced" by the server?
A: The protocol documents `endingVersion` as an optional hint; the server returns changes up to the current state and is not obligated to hard-stop exactly at the requested version, so the client must check the `_commit_version` of returned rows and trim anything beyond its intended ceiling. Relying on the server to clip precisely could let through changes past the boundary the consumer wanted. It mirrors the same best-effort philosophy as the predicate hints.

Q: `predicateHints` is described as deprecated — what should you use instead and why?
A: Use `jsonPredicateHints`, which expresses partition filters as a structured JSON tree rather than a free-text SQL string, because the JSON form is unambiguous to parse and safer for servers to evaluate consistently across implementations. The SQL-string `predicateHints` was error-prone and is retained only for backward compatibility. Both are still best-effort partition pruning, so the client re-applies the filter regardless.

Q: Construct a `jsonPredicateHints` to keep only partitions where `valuation_date` equals `2026-06-15`.
A: It is a JSON tree like `{"op":"equal","children":[{"op":"column","name":"valuation_date","valueType":"date"},{"op":"literal","value":"2026-06-15","valueType":"date"}]}`, sent in the query body so the server can skip partitions whose `valuation_date` cannot match. The `op` nodes compose (`and`, `or`, `equal`, `lessThan`, etc.) over `column` and `literal` leaves. Because it is best-effort, the client still filters the returned rows to exactly that date.

Q: For a 40-distributor estate, why might you run multiple sharing servers (or Unity Catalog) rather than one giant YAML config?
A: A single static `delta-sharing-server.yaml` listing every share and token becomes an unauditable, restart-to-change bottleneck and a single point of failure for all 40 recipients, with no per-recipient credential lifecycle. Splitting by tenant/domain or moving to Unity Catalog gives independent blast radius, hot recipient management, and proper audit per recipient. The reference server is a teaching tool; production dissemination needs dynamic recipient governance.

Q: How do you load a profile file that lives in cloud storage rather than on the local disk?
A: Pass an fsspec-style URL as the profile path, e.g. `delta_sharing.load_as_pandas("s3://bucket/config.share#share.schema.table")` or an `abfss://`/`gs://` path, because the client resolves the profile location through fsspec; on Databricks a `/dbfs/` path also works. This lets a distributor keep the profile (and its `bearerToken`) in a governed secrets/object location rather than scattered on laptops. The `#share.schema.table` suffix stays the same regardless of where the profile lives.

Q: A recipient can read table A in a share but gets `403 Forbidden` (not `401`) on table B in the same share — what does that distinguish?
A: A `403` means the token authenticated successfully but is not authorized for that specific resource, whereas a `401` would mean the token itself is missing or invalid — so table B is simply outside this recipient's grant, while their credential is fine. The fix is a grant/scope change on the server, not a new token. Distinguishing the two codes tells you whether to touch authentication or authorization.

Q: A recipient gets `404 Not Found` listing a schema that you believe exists — what are the two likeliest causes?
A: Either the schema/table is genuinely not part of any share the recipient's token is granted (so the server hides it as non-existent rather than leaking that it exists), or there is a name typo in the three-part `share.schema.table` reference. The server deliberately returns `404` for ungranted resources to avoid disclosing the catalog to unauthorized callers. Cross-check the exact names against `list_all_tables()` for that token.

Q: Why does the server return `404` rather than `403` for a share a recipient was never granted?
A: Returning `404` avoids confirming the existence of shares/schemas the caller is not entitled to see, which prevents an attacker from enumerating your catalog by probing names; `403` would leak that the resource exists but is off-limits. It is a deliberate information-hiding choice in the protocol's authorization model. Only resources within the token's grant are acknowledged at all.

Q: How would you set a `limitHint` to fetch a quick 1,000-row sample, and why might you still get far more rows?
A: In the Python client call `delta_sharing.load_as_pandas(table_url, limit=1000)`, which sends `limitHint=1000` so the server can return fewer files; but because pruning is at whole-file granularity, the server hands back complete Parquet files and you typically receive more than 1,000 rows before the client trims to exactly 1,000. The hint cuts transfer volume, not the precise count. The client enforces the real cap after download.

Q: Compare `temporary-table-credentials` against pre-signed per-file URLs for a Spark consumer reading a large Delta table.
A: `POST .../temporary-table-credentials` issues short-lived cloud creds (AWS STS, Azure SAS, GCP OAuth) scoped to the table's storage directory, so Spark reads the whole Delta directory natively — picking up deletion vectors and parallelism — instead of juggling a long list of per-file pre-signed URLs. Per-file URLs are simpler and client-agnostic but explode in number for big tables and cannot easily support directory-level native Delta reads. The directory-credential path suits sophisticated, high-volume consumers.

Q: What is the security trade-off of `temporary-table-credentials` versus per-file pre-signed URLs?
A: Directory-scoped temporary credentials grant read over the table's whole storage prefix for their lifetime, a broader capability than a pre-signed URL that unlocks exactly one object, so a leaked directory credential exposes more. The mitigation is the same short TTL plus tight directory scoping and the durable, revocable sharing token gating who can mint them at all. You trade a slightly larger per-grant blast radius for native Delta reading.

Q: A consumer needs to read the share through Power BI with no Python — what do you give them?
A: Hand them the profile file and have them use Power BI's native Delta Sharing connector, which consumes the `endpoint`/`bearerToken` directly and lists the granted tables for selection. No code is required, so "not a programmer" does not automatically force a file-drop. Where a consumer's BI tool lacks a connector and they cannot script, the rubric may still route them to EMT/EPT.

Q: How do you cheaply detect "did the NAV table change since my last read?" without downloading any data?
A: Call the lightweight `GET .../tables/{table}/version` endpoint (or read the `Delta-Table-Version` response header from a prior read) and compare it to your stored last-seen version; if unchanged, skip the pull entirely, and if advanced, fetch only the CDF from your last version. This costs one metadata request rather than re-reading the table to discover nothing changed. It is the single biggest egress-saving habit you can give a frequent distributor.

Q: A distributor polls `GET .../version` every minute and the number never advances even though you publish hourly — is that a bug?
A: Not necessarily — the version only increments on a Delta commit, so if your NAV publish writes once per hour the version is correctly stable for ~59 of every 60 polls; an unchanging number across a full publish cycle, however, would point to a stalled provider job. Pair the version check with your known publish cadence to interpret it. The endpoint reports table commits, not wall-clock time.

Q: Why prefer `GET .../version` over `GET .../metadata` for a cheap freshness check?
A: `.../version` returns just the current Delta version as a tiny scalar, while `.../metadata` returns the full schema and table properties, so for a pure "has it changed?" poll the version endpoint is the minimal, cheapest call. You only need `.../metadata` when you actually want to inspect or validate the schema. Using the heavier endpoint for routine polling wastes requests and bandwidth.

Q: A new distributor's `list_all_tables()` returns MORE tables than the contract specified — what do you do before go-live?
A: Treat it as a scoping defect: the share grant is too broad and is exposing tables the recipient is not licensed for, so you tighten the share/grant on the server (remove the extra schemas/tables or split into a narrower share) and re-verify before issuing the profile. Over-granting is a least-privilege and licensing violation even if the distributor never reads the extras. Fix the grant, not the client.

Q: How does sharing a per-recipient secure view enforce row-level isolation that partition filters alone cannot?
A: A secure view can apply an arbitrary `WHERE` (e.g. only this distributor's ISINs) and column projection/masking, then you share the VIEW as the recipient's table, so isolation is enforced by the view definition regardless of how the physical table is partitioned. Partition-based `jsonPredicateHints` only prune at partition granularity and are best-effort, not an access control. The view is the durable least-privilege boundary; hints are an optimisation.

Q: Map each of grant, scope, rotate, revoke to a concrete server-side action on the reference server.
A: Grant = add the recipient's `bearerToken` and its share mapping to `delta-sharing-server.yaml`; scope = define which schemas/tables that share lists (or point it at a narrowed view); rotate = replace the `bearerToken` value and have the recipient update their profile during a transition window; revoke = delete/invalidate that token and reload the server so the next request returns `401`. All four are config edits plus a reload on the bare reference server, versus catalog API calls in Unity Catalog.

Q: After you rotate a recipient's token, their reads suddenly `401` — what step was missed?
A: Rotation requires the recipient to update their PROFILE FILE with the new `bearerToken`; if you swapped the server token but the distributor is still presenting the old one, every request fails authentication. A clean rotation overlaps both tokens for a transition window so the consumer can switch without downtime, or you coordinate the cutover. The fix is distributing the refreshed profile, not reverting the rotation.

Q: How do you produce a tamper-evident audit trail of who read a share, suitable for a DORA examiner?
A: Capture the sharing server's per-request access logs (recipient/token identity, share/schema/table, timestamp, version returned) into an append-only, retained log store, and correlate each recipient token to a legal entity via its LEI. That gives "entity X read table Y at time Z, version N" records plus the grant/rotation/revocation events as timestamped entries. The trio of identity, action, and time is what turns operational logs into examiner-grade evidence.

Q: Why is mapping a recipient token to an LEI more than cosmetic for audit?
A: The LEI is a globally unique, externally verifiable identifier for the legal entity, so binding a token to an LEI lets you state in audit and contract terms exactly which real-world counterparty held access, not just an opaque token id. It survives token rotation (the entity is stable while the credential changes) and ties technical access to the license/contract holder. Anonymous tokens give attribution to a string; LEIs give attribution to an accountable party.

Q: A distributor wants the NAV "as it was published on the 2026-03-31 valuation date" months later — what must the provider have done in advance?
A: The provider must have retained the Delta history covering that commit (not `VACUUM`-ed those files away) and enabled history/version sharing, so the distributor can issue a `timestamp=2026-03-31...` or `version=N` time-travel read that the server can still materialize. Reproducibility of past NAVs is a deliberate retention decision made before the fact. Once the underlying files are vacuumed, that version is unrecoverable through the share.

Q: How does `timestamp`-based time travel differ from `version`-based, and which is safer for a fixed audit point?
A: `version=N` pins an exact Delta commit, whereas `timestamp` resolves to whatever version was current at that wall-clock moment, which can be ambiguous around commit boundaries and shifts if you re-interpret time zones. For a fixed, reproducible audit point a specific `version` is safer because it names one immutable commit. Use `timestamp` only when you genuinely think in clock time and accept the resolution rules.

Q: Why does reading the share NOT mean the data stays in the provider's region only, and what compliance step follows?
A: Because the recipient downloads the actual Parquet bytes to wherever their client runs, the data physically crosses into the consumer's region/jurisdiction on every read, so "no copy in my storage" is not "no transfer." The follow-up is treating each recipient pull as a cross-border transfer for GDPR/data-residency purposes and documenting lawful basis and location in the data contract. Channel design must record where each distributor reads to.

Q: Contrast Delta Sharing's transfer model with Snowflake cross-region sharing for an EU residency requirement.
A: With Delta Sharing the consumer pulls from your single object store, so there is no provider-side replica but the bytes transit to the consumer's region on read — a transfer to manage, not a stored copy. Snowflake cross-region sharing instead forces you to REPLICATE the database into an account in the consumer's region first, creating a managed copy with its own residency footprint and freshness lag. Both have a residency story; one is a transfer-on-read, the other a standing replica.

Q: For a near-real-time intraday NAV consumer, why is Delta Sharing the wrong channel and what do you offer instead?
A: Delta Sharing is pull-based snapshot/CDF access whose freshness is bounded by how often the consumer reads and by commit cadence, so it cannot deliver sub-second intraday updates; you would offer a streaming feed or an event/push channel for that latency class. Forcing such a consumer onto polling the share wastes egress and still lags. The rubric's latency axis exists to steer this consumer away from the sharing protocol.

Q: Why is an EMT or EPT file-drop sometimes mandatory even when Delta Sharing would be technically superior?
A: The European MiFID Template and European PRIIPs Template are standardized, regulator-and-counterparty-mandated file deliverables whose exact format is the contractual artifact, so some counterparties' ingestion processes are legally and operationally built around receiving that specific file on a schedule. Technical superiority does not override a regulatory format mandate or a counterparty that will not adopt a protocol. The rubric must concede that compliance and consumer capability can trump elegance.

Q: Where does the European ESG Template (EET) fit the dissemination channel choice?
A: The EET is another standardized fund-data exchange template (carrying SFDR/ESG data points between manufacturers and distributors), so like the EMT/EPT it is a fixed-format, regulator-aligned file deliverable that typically rides the file-drop channel rather than a live protocol. A counterparty whose process expects an EET file each cycle is a low-sophistication-or-mandated case in the rubric. You would not replace a contractually required EET with a Delta Sharing grant unless the counterparty agreed to consume the underlying data product instead.

Q: How does a T+1 settlement regime sharpen the freshness SLO you attach to a shared NAV table?
A: Under T+1 the downstream settlement window collapses to one business day, so any distributor or TA process depending on your shared NAV needs it published and readable earlier and with tighter, enforceable cutoffs than under T+2. The share's freshness SLO must guarantee the new valuation-date version is committed by a cutoff that leaves the consumer time to settle. A late commit that was tolerable under T+2 can break the T+1 chain, so the SLO and your publish cadence must tighten together.

Q: Could the EU AI Act bear on a Delta Sharing arrangement, and how?
A: Indirectly — if a recipient uses the shared fund data to train or operate a high-risk AI system, the Act's data-governance and record-keeping expectations attach to that downstream use, so your contractual overlay should state permitted purposes and whether AI training is allowed, and your access logs help evidence what data was supplied and when. The sharing protocol governs access; it does not by itself satisfy AI-Act obligations, which sit with the deploying party. Naming permitted use (including AI use) in the license is the lever the provider controls.

Q: When two providers in a fund group both share overlapping data, why can per-recipient tokenisation still leave a governance gap?
A: Tokens scope and revoke ACCESS, but if both providers grant the same distributor the same underlying data under different contracts, you can have inconsistent license terms, SLOs, or schema versions for one consumer — a contract/product gap the bearer token cannot close. The fix is treating the share as a product with a single owning contract and reconciling overlapping grants. Technical revocation is necessary but does not enforce contractual coherence.

Q: Summarise the end-to-end troubleshooting order when a distributor says "the share just stopped working."
A: Check authentication first (does a fresh `list_all_tables()` succeed, or is it `401`/expired token), then authorization (`403`/`404` on the specific table = grant/scope issue), then download (URLs expiring, `404` from vacuumed files, S3 dot-bucket TLS, or provider cloud-cred failure), then freshness (`Delta-Table-Version` lag or a stalled publish job). Work outward from credential to grant to storage to data, because each layer has a distinct signature (`401` vs `403`/`404` vs download error vs stale version). That ordering localises the fault fast instead of guessing.

Q: What is the minimal install and setup for a non-Spark distributor to consume a share?
A: Install the client with `pip install delta-sharing` (preferably into an isolated environment), drop the provider's profile JSON somewhere readable, then call `delta_sharing.load_as_pandas("profile.share#share.schema.table")` — no cluster, no ETL job, just one library and one credential file. That is the whole consumer footprint for a daily NAV table, which is why the protocol is attractive for lightweight distributor environments. For larger reads the same package also offers `load_as_spark` when a Spark session is available.

Q: If you reload the reference server to revoke a token while a recipient is mid-download, is their current read interrupted?
A: Typically not for files already in flight, because the recipient is downloading Parquet directly from object storage via pre-signed URLs that remain valid until their own short TTL expires, independent of the sharing server's auth state. The revocation bites on the NEXT call to the sharing server (a new query, metadata, or version request), which then returns `401`. This is why short pre-signed URL expiry matters: it bounds how long an already-issued read can outlive a revocation.

Q: How does Microsoft Fabric external data sharing compare to Delta Sharing as a walled-garden equivalent?
A: Fabric external data sharing lets a tenant grant another Fabric tenant in-place access to OneLake data (which is stored in Delta/Parquet) without copying, conceptually like Snowflake secure shares but inside the Fabric/OneLake ecosystem and tenant-to-tenant. It is in-ecosystem governance and discovery at the cost of requiring the consumer to also be on Fabric, the same trade-off as Snowflake and Databricks-native shares. The open Delta Sharing protocol is what frees a consumer who is not on your platform from that constraint.
