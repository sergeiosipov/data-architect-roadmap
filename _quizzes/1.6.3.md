# Phase 1 · 1.6.3 + 1.6.4 + 1.6.5 The consistency spectrum — 150 self-test questions

Q: What does "eventual consistency" actually promise?
A: It promises only that if writes stop, all replicas will *eventually* converge to the same value — given enough time and no new updates. It says nothing about *when* convergence happens, nor about what any single read returns in the meantime, so a read may legitimately see arbitrarily stale data. The guarantee is a liveness property (it happens someday), not a safety property bounding how stale or out-of-order any given read is.

Q: What does eventual consistency explicitly *not* promise?
A: It never bounds staleness, never guarantees you read your own writes, and never guarantees monotonic (non-going-backwards) reads. A client can read a fresh value, then read an older value a moment later from a lagging replica, and that is still legal under pure eventual consistency. Because of this, "eventual" is the weakest useful model and almost always needs strengthening with session guarantees for user-facing reads.

Q: Why is eventual consistency described as a liveness property rather than a safety property?
A: A safety property says "nothing bad ever happens" and must hold at every instant; a liveness property says "something good eventually happens" with no deadline. Eventual consistency only promises convergence *eventually*, which is liveness — you can never point to a moment and prove it has been violated, only observe that it has not yet converged. This is exactly why it is so weak: an unbounded "eventually" is impossible to test against in finite time.

Q: What is the practical difference between strong and eventual consistency for a reader?
A: Under strong consistency every read returns the most recent committed write, so the reader behaves as if there were a single up-to-date copy. Under eventual consistency a read may return any past value that has not yet been overwritten by convergence, so the reader can see stale or out-of-order data. The cost of strong is higher latency and lower availability during partitions, because the system must coordinate before answering.

Q: Define replication lag and name the anomalies it causes.
A: Replication lag is the delay between a write committing on the leader and that write becoming visible on a follower/replica. It causes three classic read-after-write anomalies: reading-your-own-writes violations (your write seems lost), monotonic-read violations (time appears to go backwards), and causal violations (effects appear before causes). All three are consequences of asynchronous replication, and all three are fixable with the corresponding session guarantee.

Q: What is the "read-your-writes" guarantee?
A: Read-your-writes (also called read-after-write) guarantees that once a user has made a write, any subsequent read *by that same user* will see that write or a later state — never an older one. It is a per-user, per-session guarantee, not a global one: other users may still see stale data. It does not require the whole system to be up to date, only that a writer is never shown a version older than their own write.

Q: Give a concrete user-visible bug when read-your-writes is violated in a fund platform.
A: An investor submits a subscription order, sees the confirmation, then refreshes the order-status page and the order has vanished because the refresh hit a lagging read replica that has not yet received the write. The investor reasonably concludes the order failed and may resubmit, causing a duplicate. This is an architectural read-your-writes violation, not a UI bug, and the fix is to route that user's reads to the leader or to a replica known to have their write.

Q: How do you typically implement read-your-writes against read replicas?
A: Common techniques: route reads for recently-written keys to the leader for a short window; track the write's log position (LSN/version) on the client and only read from replicas that have caught up to that position; or pin the user's session to a single sufficiently-fresh replica. Azure Cosmos DB does this automatically via *session consistency* using a session token that acts as a minimum-version barrier. The unifying idea is that the read must carry enough information to reject too-old replicas.

Q: What is the "monotonic reads" guarantee?
A: Monotonic reads guarantees that if a client reads a value, any later read by that client returns that same value or a newer one — never an older one, so the client never sees time go backwards. It is weaker than read-your-writes: it does not require seeing your own writes, only that successive reads do not regress. Without it, a client bouncing between a fresh replica and a lagging replica can watch a comment appear and then disappear.

Q: Give a user-visible bug when monotonic reads is violated.
A: A user views the NAV-history dashboard and sees yesterday's published NAV, then refreshes and the latest data point disappears because the second read landed on a more-lagged replica. The published value appears to "un-publish," which in a regulated reporting context looks like data corruption or a recalculation. The fix is to pin the session to one replica or require each read to be at least as fresh as the previous one.

Q: Why is monotonic reads strictly weaker than read-your-writes?
A: Monotonic reads only constrains the *order among a client's reads* (never go backwards), whereas read-your-writes additionally constrains reads relative to that client's *writes* (you must see at least your own write). A system can satisfy monotonic reads while still failing to show you your latest write, if all your reads happen to land on a replica that has your earlier state but lacks the most recent write. So read-your-writes implies a stronger freshness floor.

Q: What are "session guarantees" and which ones break first under replication lag?
A: Session guarantees are per-client-session consistency promises — read-your-writes, monotonic reads, monotonic writes, and writes-follow-reads — that hold for one client's view without requiring global consistency. They are the first guarantees to break when you add read replicas, because a client whose reads scatter across replicas of differing freshness immediately violates read-your-writes and monotonic reads. They are also the cheapest to restore, which is why session consistency is the most widely used productized level.

Q: Define causal consistency.
A: Causal consistency guarantees that operations related by a cause-and-effect (happens-before) relationship are seen by every replica in the same causal order; concurrent operations may be seen in any order. Concretely, if write B was made after observing write A, no reader ever sees B without also seeing A. It preserves "question before answer" ordering without requiring a single global total order, making it the strongest model still compatible with high availability.

Q: Give a user-visible bug when causal consistency is violated.
A: In a messaging or comment thread, a reply ("the NAV looks wrong") appears before the message it answers ("today's NAV is 12.34"), so the conversation reads nonsensically. Causally, the reply *happened-after* the original, but a replica delivered them out of order. Preserving causal order ensures the answer never shows up without the question that prompted it.

Q: What does causal consistency allow that linearizability forbids?
A: Causal consistency allows concurrent (causally unrelated) operations to be observed in different orders on different replicas, and it does not require any global notion of "most recent." Linearizability forbids both: it imposes a single real-time total order on all operations, so every observer agrees on recency. The payoff of the weaker causal model is that it remains available during network partitions, which linearizability cannot.

Q: Why is causal consistency considered the strongest model that survives a partition?
A: Because tracking and respecting happens-before relationships only requires propagating causal metadata (e.g., version vectors), not real-time global coordination, replicas can keep accepting reads and writes while a partition heals. Any guarantee stronger than causal — such as linearizability — requires cross-replica agreement on recency, which is exactly what a partition makes impossible. This makes causal the practical ceiling for highly-available systems.

Q: Define linearizability.
A: Linearizability is a single-object recency guarantee: it makes a replicated register behave as if there were exactly one copy, and every operation appears to take effect atomically at some instant between its invocation and its response, consistent with real-time order. Once a write completes, every subsequent read (anywhere) must return that write or a later one. It is about *recency of one object*, not about transactions spanning multiple objects.

Q: Define serializability.
A: Serializability is a multi-object transaction-isolation guarantee: the outcome of concurrently executing transactions is equivalent to *some* serial (one-at-a-time) execution of those transactions. It constrains how transactions over many objects interleave, ensuring no anomaly that a serial schedule could not produce. Crucially, the equivalent serial order need not match real-time order, so serializability alone says nothing about recency.

Q: What different questions do linearizability and serializability answer?
A: Linearizability answers "is this *single* object's value as recent as real time demands?" — a recency question about one register. Serializability answers "did these *multi-object transactions* interleave as if run one at a time?" — an isolation question about transaction ordering. They are orthogonal: a system can be serializable but not linearizable (stale snapshots) or linearizable but not serializable (single-object, no transactions).

Q: What is strict serializability and how does it combine the two?
A: Strict serializability is serializability plus linearizability's real-time constraint: transactions are equivalent to a serial order, *and* that order respects the real-time order of non-overlapping transactions. It is the gold-standard guarantee delivered by systems like Google Spanner. It answers both questions at once — correct multi-object isolation and single-object recency — at the cost of the most coordination.

Q: Can a system be serializable without being linearizable? Give an example.
A: Yes. Serializable snapshot isolation can serve a transaction from a slightly stale consistent snapshot: the result is equivalent to *some* serial order, but a read may not reflect the very latest committed write, violating linearizability's real-time recency. PostgreSQL's `SERIALIZABLE` isolation level on a read replica is a practical example — serializable per transaction, yet a replica read can be stale. So "serializable" does not imply "always reads the freshest value."

Q: Can a system be linearizable without being serializable? Give an example.
A: Yes. A single-object atomic register (e.g., an atomic compare-and-set on one key) can be linearizable — every read sees the latest write in real-time order — yet offer no multi-object transactions at all, so it cannot be serializable across keys. Many key-value stores provide per-key linearizability without cross-key transaction isolation. Linearizability is about one object; serializability is about whole transactions.

Q: Why is "strong consistency" an ambiguous term that an architect should pin down?
A: "Strong" is marketing/colloquial and can mean linearizability, strict serializability, or merely "stronger than eventual," depending on the vendor. An architect must name the *specific* guarantee actually delivered — linearizable? serializable? read-your-writes? — because each has different latency, availability, and failure semantics. Saying a system is "strongly consistent" without the precise model is exactly the vague claim this lesson trains you to reject.

Q: Where does replication lag bite first in a typical web architecture?
A: It bites on read-after-write: a user performs a write to the leader, then an immediately-following read is served by an asynchronous replica that has not yet applied the write, so the user's own change appears missing. This is the single most common production complaint when read replicas are added for scaling. The second-most-common is monotonic-read violations as a user's reads scatter across replicas of different freshness.

Q: What is the "lost-write window" during a cross-region failover?
A: When an asynchronous replica is promoted after the primary region fails, any writes that committed on the old primary but had not yet replicated are lost — that gap is the lost-write window, quantified as the Recovery Point Objective (RPO). With asynchronous replication the RPO is non-zero; with synchronous replication it can be zero but at the cost of write latency and availability. In a fund platform this window is precisely the set of orders or NAV updates that could silently disappear in a DR event.

Q: How does the choice between synchronous and asynchronous replication affect the lost-write window?
A: Synchronous replication waits for the replica to acknowledge before the write is considered committed, so a failover loses nothing (RPO ≈ 0) but every write pays cross-replica latency and stalls if the replica is unreachable. Asynchronous replication acknowledges the write immediately and ships it later, giving low latency and high availability but a non-zero lost-write window on failover. This is the core durability-versus-latency trade the architect must state explicitly.

Q: In a geo-DR setup, why can a lower consistency level mean a larger RPO?
A: Weaker consistency levels (session, consistent prefix, eventual) replicate asynchronously across regions, so on a region outage the un-replicated tail of writes is lost — Cosmos DB documents this as an RPO of under 15 minutes for those levels with a single write region. Strong consistency replicates to a global majority before acknowledging, giving RPO = 0 but the highest write latency. Thus the same knob that buys you low-latency writes also widens the data you can lose in a disaster.

Q: List Azure Cosmos DB's five consistency levels from strongest to weakest.
A: Strong, then Bounded staleness, then Session, then Consistent prefix, then Eventual. This ordered spectrum is the vocabulary Azure design reviews use, sitting between the two extremes most NoSQL stores offer. Each step down trades a weaker guarantee for lower latency, higher availability, and (for reads) higher throughput.

Q: Which textbook guarantee does Cosmos DB's Strong level implement?
A: Strong implements linearizability — reads are guaranteed to return the most recent committed write, and a client never sees an uncommitted or partial write. It is the single-object recency guarantee, achieved by committing writes to a global majority of regions before acknowledging. This is why multi-region Strong incurs write latency equal to roughly two round-trips between the two farthest regions plus about 10 ms at the 99th percentile.

Q: What does Cosmos DB's Bounded staleness level guarantee, and how is the bound configured?
A: Bounded staleness guarantees that reads lag the latest write by no more than a configured bound, expressed as *K* versions (updates) of an item or *T* time interval, whichever is hit first. Within the bound, reads are consistent-prefix-ordered; outside regions never exceed *K* or *T* of staleness. If a partition's lag would exceed the bound, Cosmos DB throttles writes for that partition until it catches up.

Q: For a multi-region single-write account, what are the minimum K and T values for Bounded staleness?
A: Per Microsoft's documentation, for multi-region accounts the minimum bounded-staleness values are 100,000 write operations (*K*) or 300 seconds (*T*). For a single-region account the minimums are much smaller — 10 write operations or 5 seconds. These minimums also define the floor of the achievable RPO when using bounded staleness, because the lag bound is exactly the data you can lose on failover.

Q: Which textbook guarantees does Cosmos DB's Session level provide?
A: Session consistency provides the session guarantees within a single client session: read-your-writes, monotonic reads, monotonic writes, and writes-follow-reads. It does this with a session token returned after each write, which the client passes on reads as a minimum-version barrier so it never reads data older than its own session. It is the default and most widely used level because it gives user-correct behavior at eventual-consistency-like latency and throughput.

Q: How does the Cosmos DB session token actually enforce read-your-writes?
A: After each write the server returns an updated session token (a per-partition logical version); the client caches it and attaches it to subsequent reads. If the replica handling the read lacks data for that token, the client retries against another replica — and if needed other regions — until a replica at or beyond that version answers. Because the token is a *minimum* barrier, you always get your own write or newer, never older.

Q: What happens to Cosmos DB reads if the client has no session token for a partition?
A: If the client never wrote to that physical partition, or its token cache was lost (for example the client was re-created), it holds no token, so reads to that partition behave as Eventual consistency until subsequent writes rebuild the cache. This is a subtle gotcha: a freshly restarted service can momentarily lose read-your-writes guarantees it previously enjoyed. Session tokens are also partition-bound, so a token for one partition does nothing for another.

Q: What does Cosmos DB's Consistent prefix level guarantee?
A: Consistent prefix guarantees that reads never see writes out of order: if writes happened in order A, B, C, a reader sees some prefix — A, or A,B, or A,B,C — but never A,C without B. Single-document writes get eventual ordering, but multi-document transactional batches are always visible together (all or nothing) and in committed order. So you might see stale data, but never a causally-scrambled sequence.

Q: Map Cosmos DB Consistent prefix to a textbook guarantee.
A: Consistent prefix corresponds to the consistent-prefix property within causal/sequential consistency: replicas apply writes in an order consistent with their commit order, so no reader observes a later write without the earlier ones it followed. It is weaker than full causal consistency because it does not track cross-session causal dependencies, but it rules out the out-of-order "answer before question" anomaly within a write sequence. It sits above pure eventual, below session.

Q: Which textbook guarantee does Cosmos DB's Eventual level implement, and what is its risk?
A: Eventual is plain eventual consistency: reads hit any one replica in the region, which may be lagging and may even return values *older* than a previous read — there are no ordering or monotonicity guarantees. The documented risk is that a client can read values older than ones it read before, so it is suitable only where ordering is irrelevant, such as a like count or non-threaded comment count. For fund order status or anything a user expects to persist, it is unsafe.

Q: Why does Cosmos DB cost the same request units for writes across all consistency levels but differ for reads?
A: Writes always go to a local majority (or, for Strong, a global majority) regardless of level, and the RU cost of a write of a given type is identical across levels. Reads differ because Strong and Bounded staleness must read from two replicas (a local minority quorum) to verify recency, doubling read RU cost and halving read throughput, while Session, Consistent prefix, and Eventual use single-replica reads. So you pay for stronger reads in throughput, not just latency.

Q: Why is multi-region-write plus Strong consistency disallowed in Cosmos DB?
A: A multi-write-region account cannot guarantee both RPO = 0 and RTO = 0 with strong consistency, so Cosmos DB blocks the combination. Moreover, Strong would require every write to commit to all regions anyway, giving the same write latency as a single-write-region account with no benefit. The intended pattern for multi-write accounts is to read and write in the same region, where cross-region lag is irrelevant.

Q: Why is Bounded staleness considered an anti-pattern for multi-region-write Cosmos DB accounts?
A: Bounded staleness depends on bounding *cross-region replication lag*, but in a multi-write-region design each application server should read and write in its own local region, where that lag is irrelevant. Coupling correctness to inter-region lag in such a topology adds throttling risk and cost for a guarantee you do not need. Microsoft therefore recommends Session (or local Strong-equivalents) for multi-write accounts and reserves Bounded staleness for single-write-region accounts wanting near-strong cross-region reads.

Q: In Cosmos DB, can overriding the consistency level make a strongly-configured account weaker for one request?
A: Yes — an SDK client or per-request override can read at Session or weaker even when the account default is Strong, and that read then uses a single replica. The override affects only *reads within the SDK client*; the account still writes and replicates synchronously to every region. So you can opportunistically read cheaper/faster on a strong account, but you cannot make a session-configured account read stronger than its writes were committed.

Q: A fund order-status API and a NAV-history dashboard sit on the same platform — which needs the stronger guarantee and why?
A: The order-status API needs at least read-your-writes (ideally session or stronger), because an investor must always see an order they just placed; a vanished order triggers duplicate submissions and a compliance incident. The NAV-history dashboard tolerates staleness, because a few seconds' lag on a historical, already-published series has no user-correctness or regulatory impact. Defended in one line: order status is read immediately after a write by the same actor, NAV history is not.

Q: Defend in one sentence why NAV history tolerates staleness but order status does not.
A: NAV history is a slowly-changing, already-published series read long after it was written, so seconds of lag are invisible; order status is read seconds after the same user wrote it, so any lag manifests as a missing order and an apparent failure. The difference is the temporal gap between write and read by the same actor. That gap, not the data's importance, determines the minimum guarantee.

Q: For a transfer-agency register lookup, what consistency consideration matters most?
A: A transfer-agency (TA) register lookup is read after registrations/transfers are posted, and it underpins ownership and entitlement decisions, so it needs read-your-writes and monotonic reads at minimum so a just-recorded holding is never shown as absent. For settlement and regulatory ownership questions, you may push toward linearizable/strong reads on the authoritative register, accepting higher latency. The gotcha is serving TA lookups from a lagging analytics replica, which can show stale ownership.

Q: Why can a NAV publication that "un-publishes" on refresh be a monotonic-read violation rather than a recalculation?
A: If the first read landed on an up-to-date replica showing the published NAV and a later read landed on a more-lagged replica missing it, the value appears to disappear — that is monotonic reads going backwards, not the business recalculating anything. The data was correct; the reads regressed because they scattered across replicas of different freshness. The fix is to pin the dashboard session to one replica or require each read to be at least as fresh as the last.

Q: What is the relationship between the session guarantees and causal consistency?
A: The session guarantees (read-your-writes, monotonic reads, monotonic writes, writes-follow-reads) are the per-client building blocks; together, writes-follow-reads in particular captures causality within a session. Full causal consistency generalizes this to a system-wide happens-before order across all clients, not just within one session. So session consistency is a pragmatic, per-session approximation of causal ordering that is cheap to implement with tokens or sticky routing.

Q: Why does adding read replicas for scale silently downgrade your consistency guarantee?
A: A single-leader database read from the leader gives strong reads, but the moment you offload reads to asynchronous replicas, those reads may be stale and unordered relative to the leader — you have silently moved from strong to eventual (or session at best) for replica reads. Nothing in the API announces this change; it manifests only as user-visible anomalies. The architect's job is to name the new, weaker guarantee explicitly rather than assume "strong" still holds.

Q: How can a cache in front of a database downgrade consistency, and what is the user-visible symptom?
A: A cache returns a previously-stored value that may be older than the current database state, so reads served from cache can violate read-your-writes (your write went to the DB but the cache still serves the old value) and monotonic reads. The user-visible symptom is the classic "I changed it but it still shows the old value," or a value that flips between fresh (cache miss) and stale (cache hit). Mitigations include write-through caching, cache invalidation on write, or short TTLs.

Q: A user reports "my submitted order disappeared after I refreshed" — what is the first thing to check?
A: First confirm whether the read after submission was served by a different (lagging) replica or cache than the leader the write hit — i.e., suspect a read-your-writes violation from replica lag, not a failed write. Check the order actually committed on the leader (it almost certainly did), then check the read routing and replication lag for that path. The fix is architectural: route post-write reads to the leader or enforce session consistency, not "retry the order."

Q: Two users see the same fund's holdings in different orders of recent transactions — which guarantee is missing?
A: If unrelated transactions appear in different orders to different users, that is acceptable under causal/eventual models; but if *causally related* events (e.g., a buy then its matching settlement) appear out of order, causal consistency is being violated. Determine whether the swapped events have a happens-before relationship. If they do, you need at least causal consistency (or consistent-prefix ordering for a single write sequence) to forbid the anomaly.

Q: Why does linearizability reduce availability during a network partition?
A: To guarantee every read returns the latest write in real-time order, a linearizable system must coordinate with a majority/quorum before answering; if a partition isolates a node from that quorum, it must refuse to serve rather than risk returning stale data. Refusing is unavailability. This is the CAP trade in concrete terms: under partition, linearizability forces you to sacrifice availability for some nodes.

Q: What is "monotonic writes" and why is it sometimes the missing guarantee?
A: Monotonic writes guarantees that writes from a single client are applied to replicas in the order the client issued them, so an earlier write is never overwritten by a later replica applying them out of order. It is part of the session guarantees. It is the missing piece when, for example, two rapid updates to the same record by one client land out of order on a replica, leaving the stale value.

Q: What is "writes-follow-reads" and how does it preserve causality?
A: Writes-follow-reads (also called session causality) guarantees that if a client reads value X and then makes a write W, W is ordered after the write that produced X on all replicas. This preserves the natural "I saw this, therefore I wrote that" causal chain — for instance, replying to a message only after having read it. It is the session guarantee most directly responsible for keeping "question before answer" ordering intact.

Q: Why is "eventually consistent" insufficient language for a design review, and what should you say instead?
A: "Eventually consistent" hides whether the system offers read-your-writes, monotonic reads, or any bounded staleness — the very properties that determine whether users see correct behavior. In a review you should state the precise model (e.g., "session consistency with per-user session tokens, RPO under 15 minutes on regional failover"). Naming the exact guarantee turns a hand-wave into a verifiable, costable design decision.

Q: How does bounded staleness differ from eventual consistency in what it promises a reader?
A: Eventual promises only eventual convergence with no bound, so a read can be arbitrarily stale and even non-monotonic. Bounded staleness promises that reads lag the latest write by at most *K* versions or *T* time, and within that window observe a consistent prefix (no out-of-order writes). So bounded staleness gives you a *quantified worst-case staleness*, which is exactly what makes it usable for "near-strong but available" requirements.

Q: Why is bounded staleness attractive for a single-write-region, globally-read fund platform?
A: It lets remote regions serve fast local reads while guaranteeing they are never more than *K* versions or *T* seconds behind the write region — near-strong freshness without paying Strong's global-commit write latency. For a fund platform with one authoritative write region and worldwide read access, this caps how stale a remote NAV or position read can be, which you can map directly to a regulatory tolerance. The cost is write throttling if a region falls behind the bound.

Q: What does "consistent prefix" forbid that "eventual" permits?
A: Consistent prefix forbids out-of-order observation of a write sequence: you can never see a later write without the earlier writes it followed, so no "answer before question." Eventual permits exactly that — a replica could surface a newer value while still missing an older one, scrambling the sequence. Consistent prefix thus rules out the most jarring ordering anomalies while still allowing overall staleness.

Q: Map each Cosmos DB level to its textbook guarantee in one pass.
A: Strong = linearizability (single-object recency); Bounded staleness = consistent-prefix reads bounded by *K*/*T* staleness; Session = the per-session guarantees (read-your-writes, monotonic reads/writes, writes-follow-reads) via session tokens; Consistent prefix = ordered-prefix (no out-of-order writes) but otherwise eventual; Eventual = eventual consistency with no ordering or monotonicity. This mapping is the core deliverable of the lesson and the vocabulary Azure reviews assume.

Q: Why is "the system is consistent" a meaningless claim without qualification?
A: "Consistent" spans a whole spectrum — from eventual to causal to linearizable to strict serializable — each with radically different recency, ordering, latency, and availability properties. Saying a system is "consistent" tells you nothing about which anomalies it permits. The architect's discipline is to always attach the specific model name (and, for transactions, the isolation level) so the claim is testable.

Q: What is the difference between consistency models (replication recency) and isolation levels (transaction interleaving)?
A: Consistency models (linearizability, causal, eventual) describe how *recent and ordered* reads of replicated single objects are. Isolation levels (read committed, snapshot/repeatable read, serializable) describe how concurrent *multi-statement transactions* may interleave and which anomalies (dirty read, write skew, etc.) are prevented. They are different axes — DDIA stresses that conflating them is a common error — and a complete design names both.

Q: Why does snapshot isolation not provide linearizability?
A: Snapshot isolation serves each transaction from a consistent snapshot taken at start time, so a long-running transaction can read values that are already stale by commit time, and two snapshots can disagree on recency. Linearizability requires every read to reflect the latest committed write in real time, which a fixed snapshot cannot. Hence a system can give strong isolation (snapshot/serializable) yet still return non-linearizable, slightly-stale reads.

Q: In DDIA's framing, why is linearizability "the strongest single-object guarantee" yet still not enough for everything?
A: Linearizability makes one register behave like a single up-to-date copy, which is the strongest you can say about *one object's recency*. But it says nothing about *atomicity across multiple objects* — you still need serializability/transactions to coordinate multi-object invariants. So a correct multi-object design often needs both linearizability (for recency, e.g., on a lock or counter) and serializability (for transaction isolation).

Q: What is a quorum, and how do read/write quorums relate to consistency?
A: A quorum is a minimum number of replicas that must respond for an operation to be considered successful. If the read quorum R and write quorum W satisfy R + W > N (total replicas), every read overlaps at least one replica that saw the latest write, enabling strong/linearizable-ish reads. Cosmos DB applies this idea: Strong and Bounded staleness read a local minority (two of four) to verify recency, while weaker levels read a single replica.

Q: Why does reading from two replicas (a minority quorum) let Cosmos DB guarantee recency for Strong reads?
A: Within a region, writes commit to a local majority — three of four replicas — so any two replicas read together are guaranteed to include at least one carrying the newest data. Consulting two replicas (a local minority quorum) therefore always surfaces the most up-to-date value available in the region. That overlap is precisely why Strong and Bounded staleness reads cost double the RUs of single-replica reads.

Q: What does it mean that Cosmos DB "guarantees 100% of reads meet the chosen level," yet you often get stronger?
A: The guarantee is a floor: every read satisfies at least the configured level's promise. In practice, when there is little or no concurrent write activity, a Session or Eventual read may happen to return fully up-to-date data — stronger than promised. Cosmos DB exposes a Probabilistically Bounded Staleness (PBS) metric to quantify how often you actually get stronger-than-configured reads, but you must design against the floor, not the lucky case.

Q: What is Probabilistically Bounded Staleness (PBS) and why is it useful?
A: PBS estimates the probability (in milliseconds) that a read under a weaker level actually returns up-to-date data — "how eventual your eventual really is." Cosmos DB surfaces it as a portal metric so you can see, for a given write/read region pair, how often you are getting effectively strong reads. It helps you judge whether a cheaper level is empirically safe enough for a workload without paying for Strong.

Q: Why can two clients legitimately disagree about "the current value" under eventual consistency, and is that a bug?
A: Under eventual consistency each client may read a different replica at a different point in convergence, so disagreement is expected, not a bug — the model never promised a single agreed-upon current value at any instant. It only promises they will converge if writes stop. Whether that disagreement is *acceptable* depends on the use case; for a like count it is fine, for an account balance it is not.

Q: How would you choose a consistency level for a fund-order status API on Cosmos DB, and what does it cost?
A: Choose Session at minimum, so the investor always reads their own just-placed order via the session token; consider Bounded staleness or Strong if regulators require cross-region freshness on order state. The cost of Session is essentially eventual-level latency and throughput, which is cheap; stepping up to Strong roughly halves read throughput and adds multi-region write latency. The memo should state this trade explicitly: Session buys correctness for the writer at negligible cost.

Q: How would you choose a consistency level for a NAV-history dashboard, and why?
A: Eventual or Consistent prefix is appropriate, because the data is historical, already published, and read long after it was written, so staleness of seconds is invisible to users and regulators. This gives the lowest latency, highest read throughput, and best availability. Choosing Strong here would needlessly halve read throughput and add latency for a guarantee no one needs — the classic over-provisioning of consistency the lesson warns against.

Q: What is the latency cost of choosing Strong consistency in multi-region Cosmos DB, concretely?
A: For a multi-region Strong account, write latency is approximately two round-trip times between the two farthest regions plus about 10 ms at the 99th percentile, because the write must commit to a global majority before acknowledging. So two regions ~5,000 miles apart pay tens of milliseconds per write just in speed-of-light/RTT terms. Cosmos DB even blocks Strong for accounts spanning more than 5,000 miles by default due to this cost.

Q: What is "dynamic quorum" in Cosmos DB Strong consistency and why does it exist?
A: With three or more regions on a Strong account, if some regions become slow or unresponsive, Cosmos DB can temporarily remove them from the quorum set to keep committing writes against the remaining majority — preserving Strong without waiting on stragglers. This improves write availability and replication latency during partial regional trouble. Removed regions cannot serve reads until they catch up and rejoin the quorum.

Q: Why does the number of regions affect how many can be dropped under dynamic quorum?
A: A quorum needs a majority, so the count droppable is total minus majority. In a three- or four-region account the majority is two or three, leaving room to drop only one; in a five-region account the majority is three, so up to two unresponsive regions can be removed. More regions therefore give more slack to tolerate slow/failed regions while still maintaining Strong.

Q: What RPO does each Cosmos DB consistency level give on a region-wide outage (single write region)?
A: Strong gives RPO = 0 (committed everywhere before ack). Bounded staleness gives RPO equal to its configured *K* and *T*. Session, Consistent prefix, and Eventual give RPO under 15 minutes. Single-region accounts give under 240 minutes for any level. This table is the bridge from "consistency level" to "how much data a disaster can lose," which is what business-continuity planning actually needs.

Q: Why is RPO = 0 impossible for a multi-write-region Cosmos DB account?
A: With multiple write regions accepting writes independently, a region can commit writes locally that have not yet replicated when it fails, so some recently-committed data is unavoidably lost on failover — RPO cannot be zero, and neither can RTO. That is exactly why Strong (which demands RPO = 0) is disallowed for multi-write accounts. The architecture trades zero data loss for multi-region write availability.

Q: Distinguish RPO from RTO in the consistency context.
A: RPO (Recovery Point Objective) is how much *data* you can lose — the lost-write window measured in time or versions before the failure point. RTO (Recovery Time Objective) is how long *recovery takes* — how long the service is down before it is restored. Consistency level directly drives RPO (stronger = less data loss); it does not by itself set RTO, which depends on failover automation.

Q: An EMT (European MiFID Template) file is generated nightly from positions read off a replica — what consistency risk applies?
A: If positions are read from an asynchronous replica that has not caught up to the cut-off time, the EMT file may be built from stale holdings, producing a regulatory file that disagrees with the authoritative book — a monotonic/read-your-writes hazard at the batch level. The fix is to read the EMT inputs from the leader or a replica proven caught up to the snapshot timestamp (a bounded-staleness or LSN check). Stale regulatory output is a compliance issue, not a cosmetic one.

Q: Why might a SWIFT message confirming a trade and the internal position update need causal ordering?
A: The position update is *caused by* the confirmed trade, so any system observing both must never show the position change without the corresponding confirmation — that is a happens-before relationship requiring at least causal consistency. If a reconciliation view shows an updated position before (or without) the confirming message, it looks like an unexplained, possibly fraudulent change. Causal (or consistent-prefix) ordering forbids that out-of-order appearance.

Q: How does an LEI (Legal Entity Identifier) reference-data store's consistency requirement differ from a transactional order store?
A: LEI reference data changes rarely and is read widely, so eventual or consistent-prefix consistency is usually fine — a few seconds of staleness on an entity's LEI record rarely affects correctness. A transactional order store is written and immediately read by the same actor and underpins money movement, demanding at least read-your-writes/session. Matching the consistency level to the read/write pattern — not to the data's perceived importance — is the architect's discipline.

Q: Why is "ISIN + valuation date" a useful natural key when reasoning about consistency of NAV reads?
A: Because each (ISIN, date) NAV is written once and effectively immutable thereafter, reads of historical NAVs are safe under weak consistency — there is no later write to be stale relative to, so eventual reads are correct. Consistency anomalies only bite on *mutable, recently-written* keys. Recognizing which keys are append-only/immutable lets you safely down-tier consistency (and cost) for the bulk of read traffic.

Q: A Luxembourg fund's order cut-off relies on a timestamp written then read across regions — what guarantee prevents accepting a late order as on-time?
A: You need at least monotonic reads plus read-your-writes (session), and ideally linearizable reads of the authoritative clock/cut-off state, so a region never reads a stale "still open" status after the cut-off has been recorded closed. A weak read could let a late order slip through against a lagging replica that still shows the window open. For a regulated cut-off, the recency of that single status object is exactly a linearizability requirement.

Q: Why might a regulated NAV-publication workflow justify Strong consistency despite its latency cost?
A: Once an official NAV is published it is authoritative for subscriptions/redemptions, so every consumer must see the same single value with no staleness window — a linearizability requirement on the published value to avoid two investors transacting on different NAVs. The latency/throughput cost of Strong is acceptable because publication is infrequent and correctness is regulatory. This is a case where over-provisioning consistency is the *right* call, justified explicitly.

Q: What is the danger of serving transfer-agency register reads from an analytics/reporting replica?
A: Analytics replicas are typically the most lagged (optimized for throughput, not freshness), so a TA register lookup there can show stale ownership — a holding that was just transferred may still appear under the old holder. For entitlement, settlement, or regulatory ownership decisions this is a read-your-writes/monotonic-read violation with legal consequences. Authoritative TA lookups should hit the leader or a freshness-bounded replica, reserving the analytics replica for non-authoritative dashboards.

Q: When designing read paths, why list the replication topology each path sits on before choosing a guarantee?
A: The topology (single-leader, leader-follower replicas, multi-region, cache-fronted) determines which anomalies are even possible and how stale a read can be, so it scopes the minimum guarantee a path must enforce. Choosing a consistency level without knowing whether reads hit the leader, a lagging replica, or a cache leads to either silent staleness bugs or needless over-provisioning. The Do-step's path-by-path table exists precisely to force this mapping.

Q: For each violated guarantee, what is the canonical user-visible bug — summarize the three.
A: Read-your-writes violated: a user's own just-made change appears missing (e.g., a submitted order vanishes on refresh). Monotonic reads violated: data goes backwards in time (a value seen then unseen on the next read). Causal violated: effect appears before cause (a reply shows up before the message it answers). Being able to attach a concrete bug to each name is a Done-when criterion of this lesson.

Q: Why does the lesson insist the fix for a read-your-writes violation is "architectural, not a UI bug"?
A: Because the data did commit correctly — the defect is in *where reads are routed* and *what freshness they require*, which is a property of the replication and routing design, not the front-end. Patching the UI to hide the symptom (e.g., re-fetch loops) leaves the underlying anomaly and can mask real failures. The genuine fix is to route post-write reads to a sufficiently-fresh replica or enforce session consistency at the data layer.

Q: How does Jepsen's consistency map help an architect communicate guarantees?
A: Jepsen provides a precise, agreed-upon vocabulary and hierarchy of consistency models (linearizable, sequential, causal, read-your-writes, monotonic, etc.) and how they relate, so two engineers can refer to "monotonic reads" or "linearizability" and mean exactly the same thing. Using its names avoids the ambiguity of "strong" or "consistent." It is the reference map that disciplines design-review language.

Q: Why does causal consistency require tracking metadata, and what metadata is typically used?
A: To respect happens-before, a replica must know which writes a given write depends on, which means carrying causal metadata — commonly version vectors (vector clocks) or dependency lists — so it can withhold a write until its dependencies are applied. Without that metadata it cannot tell which orderings are causal versus concurrent. The overhead of this metadata is the price of causal ordering, and it is why causal is more expensive than eventual but cheaper than linearizable.

Q: Why is sequential consistency stronger than causal but weaker than linearizable?
A: Sequential consistency requires all clients to agree on a *single total order* of operations consistent with each client's program order, but it does *not* require that order to match real time. Causal only orders causally-related operations (concurrent ones may differ across replicas), so it is weaker. Linearizable adds the real-time constraint on top of a total order, so it is stronger. The ladder is: eventual < causal < sequential < linearizable.

Q: A read replica is consistently 30 seconds behind. Which consistency models can it still satisfy and which not?
A: A 30-second-lagged replica can still satisfy eventual and consistent-prefix (ordered but stale), and bounded staleness *if* the bound is ≥ 30 seconds; with sticky routing it can give monotonic reads and even read-your-writes for that client. It cannot satisfy linearizability/Strong, because its reads are not real-time recent. The lag is fine or fatal entirely depending on which guarantee the read path requires.

Q: What does "consistent prefix preserved but data stale" mean for a fund transaction feed?
A: It means a consumer reading the feed sees transactions in their true committed order but possibly trailing the latest — e.g., it has processed transactions up to 10:00:00 correctly ordered, while the leader is at 10:00:05. No transaction appears out of order or skipped within the prefix, so ordering-dependent logic (running balances) stays correct; only the *frontier* is stale. This is exactly why consistent prefix is acceptable for ordered event processing where some lag is tolerable.

Q: Why is "exactly the latest write everywhere instantly" physically impossible across regions?
A: Information cannot propagate faster than the speed of light, so a write in one region cannot be reflected in a distant region until at least the one-way latency has elapsed; "instant global recency" would violate physics. Strong consistency copes by *waiting* (paying that latency) before acknowledging, not by avoiding it. This physical floor is why every cross-region consistency choice is fundamentally a latency trade.

Q: How do you verify a read-your-writes fix actually works?
A: Write a value, then immediately issue the read through the *same* path a user would, repeatedly and across replica routing, asserting the read always returns the value just written or newer. Inject replica lag (or test during real lag) and confirm the read still never regresses below your write. For Cosmos DB Session, confirm the session token is being propagated on the read so stale replicas are rejected.

Q: Why can restarting a service reintroduce a consistency anomaly that was previously absent?
A: Session-style guarantees often depend on client-side state — a cached session token or LSN watermark — that is lost on restart; without it, the client cannot reject too-stale replicas and reads degrade to eventual until the cache rebuilds. Cosmos DB documents exactly this: a re-created client with an empty token cache reads as Eventual for those partitions until new writes repopulate tokens. The gotcha is intermittent post-deploy "missing write" reports.

Q: Summarize the one-line memo conclusion contrasting order-status and NAV-history consistency choices.
A: Order status needs Session (read-your-writes) so an investor always sees their just-placed order, at near-zero latency/throughput cost; NAV history needs only Eventual/Consistent prefix because it is historical, already-published, and tolerant of seconds of staleness, buying the best latency, throughput, and availability. Each choice is justified by the temporal gap between write and read by the same actor — not by the data's importance. Stronger than needed wastes throughput; weaker than needed breaks correctness.

Q: Why is matching consistency level to read/write *pattern* more reliable than matching it to data *importance*?
A: Importance is subjective and tempts you to slap Strong on everything "critical," over-provisioning latency and throughput needlessly; the pattern (is the data read soon after a write by the same actor? is it mutable? is order observable?) objectively determines which anomalies can actually harm correctness. NAV publication is critical yet read long after write, so weak reads of history are safe; order status is mundane yet read immediately after write, so it needs session. Pattern, not gravitas, predicts the anomaly.

Q: Why does DDIA argue that "eventual consistency" is a poorly chosen name?
A: The word "eventual" describes only the happy ending — convergence once writes stop — while saying nothing about the user-facing anomalies along the way, which is where all the engineering difficulty lives. DDIA suggests "convergence" would be a more honest label, since the hard part is not that replicas converge but the unbounded, unordered staleness they exhibit before they do. The name lulls designers into ignoring read-your-writes and monotonic-read failures that bite real users.

Q: After a leader failover, a client that read fresh data now reads stale data from the new leader — which guarantee broke and why?
A: Monotonic reads broke: the new leader was an asynchronous follower that had not yet applied the most recent writes, so reads regressed to an older state after the client had already seen newer data. This is a classic failover anomaly — the promotion exposed the lost-write window as a backwards jump in the client's view. Mitigations include waiting for the follower to fully catch up before promotion, or tracking the client's last-seen version and rejecting a leader behind it.

Q: Why can a serializable batch report still disagree with the live book in a fund platform, and is that a defect?
A: Serializability only guarantees the report is equivalent to *some* serial order, not that its snapshot is real-time current, so a long-running report reading a consistent snapshot can lag writes committed after the snapshot opened. That is not a defect if the report is explicitly an as-of-snapshot view — it is internally consistent, just not linearizable. The defect would be presenting it as "live" when it is a fixed-point-in-time snapshot, which is a labeling problem, not an isolation failure.

Q: A reconciliation finds a position update with no matching trade confirmation after a cross-region failover — what consistency explanation should you check first?
A: First suspect a consistent-prefix or causal-ordering violation combined with the lost-write window: the failover promoted a region that received the position update's dependency partially, or the confirming write fell inside the un-replicated tail and was lost (non-zero RPO). Check whether the confirmation actually committed before the outage and whether replication ordering was preserved across the failover. The fix is either stronger cross-region ordering (causal/consistent-prefix) or a lower RPO via more synchronous replication for that critical pair.

Q: What is the single most common mistake architects make about consistency, per this lesson?
A: Assuming "strong" by default — believing replicated platforms still deliver linearizable reads after read replicas, caches, or geo-DR have silently downgraded the actual guarantee to eventual or session. The discipline is to name the guarantee *actually delivered* on each read path and to attach the concrete user-visible bug that the gap permits. Everything else in the lesson — the Cosmos spectrum, the session guarantees, RPO mapping — is tooling for making that named claim precise and defensible.

Q: In a single-leader Postgres setup, which configuration knob most directly controls the read-after-write anomaly on replicas, and what are the options?
A: The `synchronous_commit` setting on the primary, combined with `synchronous_standby_names`, controls whether a commit waits for a standby before acknowledging — set it to `remote_apply` and a read on that synchronous standby is guaranteed to see the just-committed write. The default `on` only flushes WAL locally, so asynchronous replicas can still lag and break read-your-writes. The trade is that `remote_apply` adds the replica's apply latency to every commit, which is exactly the synchronous-replication cost in concrete Postgres terms.

Q: A team sets Postgres `synchronous_commit = remote_apply` but read-your-writes still fails on a reporting replica — first thing to check?
A: Check whether the reporting replica is actually named in `synchronous_standby_names`; `remote_apply` only waits for the *synchronous* standbys, so any asynchronous replica not in that list still lags and can serve stale reads. A common mistake is having one synchronous HA standby plus several asynchronous read replicas, then routing reads to an async one expecting freshness. The fix is to route post-write reads to the primary or a synchronous standby, or track the write's LSN and gate the read on it.

Q: How does an application use a Postgres LSN to enforce read-your-writes against an async replica explicitly?
A: After the write, capture `pg_current_wal_lsn()` (or the value returned by the commit) on the primary, carry it on the client, then before serving the read poll the replica's `pg_last_wal_replay_lsn()` and only read once it is greater than or equal to the captured LSN. This turns the LSN into a minimum-version barrier identical in spirit to a Cosmos session token. If the replica never catches up within a timeout, fall back to reading the primary so the user never sees their write disappear.

Q: What is the difference between `pg_last_wal_receive_lsn()` and `pg_last_wal_replay_lsn()` on a standby, and which one gates a correct read?
A: `pg_last_wal_receive_lsn()` is how far WAL has been *received* over the network, while `pg_last_wal_replay_lsn()` is how far it has actually been *applied* to the visible database state. Only the replay LSN gates a correct read-your-writes check, because received-but-not-yet-replayed WAL is not yet queryable. Gating on the receive LSN is a subtle bug that lets a read proceed before the data is visible, reintroducing the very anomaly you tried to prevent.

Q: A standby's `pg_last_wal_replay_lsn()` stops advancing while `pg_last_wal_receive_lsn()` keeps climbing — what is the likely cause?
A: Replay is paused or blocked while WAL keeps streaming in, most often because of `max_standby_streaming_delay` conflict handling (a long-running query on the standby is blocking recovery) or an explicit `pg_wal_replay_pause()`. The receive LSN advances because the network stream is fine; the replay LSN stalls because apply is held back. The user-visible effect is growing replication lag and stale reads, fixable by tuning the conflict-delay settings or stopping the offending standby query.

Q: How does MySQL's `WAIT_FOR_EXECUTED_GTID_SET()` help implement read-your-writes on replicas?
A: After a write, the application reads the primary's last GTID (via `@@GLOBAL.gtid_executed`) and, before serving the read on a replica, calls `WAIT_FOR_EXECUTED_GTID_SET(gtid, timeout)` so the read blocks until that replica has applied the transaction or the timeout fires. This is MySQL's native version-barrier mechanism, equivalent to the Postgres LSN-gate and the Cosmos session token. The gotcha is choosing a timeout: too short and you fall back to the primary often, too long and a stuck replica stalls user reads.

Q: What is the role of `read_your_writes` consistency in PgBouncer or a proxy layer, and why is connection pooling a hazard here?
A: Connection poolers can route a user's write and their immediately-following read to *different* backend servers (primary vs replica), so without sticky routing the read lands on a lagging replica and breaks read-your-writes. The hazard is invisible because the pooler is transparent to the application. The fix is read/write splitting that pins post-write reads of the same session to the primary for a freshness window, or LSN-aware routing in the proxy.

Q: Why is a write-through cache not automatically sufficient for read-your-writes, and what failure window remains?
A: Write-through updates the cache synchronously with the database, so a subsequent read of the *same key* by the same node sees the write — but in a distributed cache the writing node may update one cache shard while a reader hits a different node or region whose cache was not updated, leaving a stale entry. There is also a window where the DB commit succeeds but the cache write fails, leaving them divergent. So write-through narrows but does not eliminate the read-your-writes gap across a distributed cache tier.

Q: Compare cache invalidation on write versus short TTLs for preventing stale reads, and name a failure mode of each.
A: Invalidation-on-write deletes or updates the cache entry the moment the DB write commits, giving tight freshness but failing if the invalidation message is lost or arrives out of order, leaving a stale entry indefinitely. Short TTLs bound staleness to the TTL regardless of lost messages, but cannot prevent serving stale data *within* the TTL window and increase DB load via more misses. Many systems combine both: invalidate on write for correctness, short TTL as a safety net for missed invalidations.

Q: A Redis cache in front of a fund position store occasionally serves a position that "flips back" to an older value — which session guarantee is broken and what is the cache-specific cause?
A: Monotonic reads is broken: a fresh value served on a cache miss (read from the DB) is followed by a stale value served on a later cache hit from a node that still holds the old entry, so the reader sees time go backwards. The cache-specific cause is an un-invalidated or replica-lagged cache entry surviving past the DB update. The fix is invalidate-on-write plus reading the writer's own session through a path that bypasses stale shards.

Q: How does DynamoDB expose the consistency choice, and what does the default give you?
A: DynamoDB reads default to *eventually consistent*; you opt into a *strongly consistent* read per request by setting `ConsistentRead = true` on `GetItem`, `Query`, or `Scan`. The default eventual read may return stale data from a replica that has not yet converged, while the strong read returns the latest committed value at roughly double the read-capacity-unit cost. Global Secondary Indexes are *always* eventually consistent and cannot be read strongly, which is a frequent source of stale-index surprises.

Q: Why can you not get a strongly consistent read from a DynamoDB Global Secondary Index, and what is the workaround?
A: A GSI is maintained asynchronously — writes to the base table propagate to the index after the fact — so the index is fundamentally eventually consistent and `ConsistentRead = true` is rejected on GSI queries. The workaround when you need a strong read on an indexed attribute is to query the base table directly with a strongly consistent read, or to model the access pattern with a Local Secondary Index (which *can* be read strongly) when the partition key matches. This is a concrete case where the index topology forces the weaker guarantee.

Q: In DynamoDB global tables (multi-region, multi-active write), what consistency model applies across regions and what conflict-resolution is used?
A: Cross-region replication in global tables is *eventually consistent*, and conflicting concurrent writes to the same item in different regions are resolved by *last-writer-wins* using a reconciliation timestamp. Within a single region you can still request a strongly consistent read, but you can never get a strong read across regions. The last-writer-wins rule means a slower clock or a later-but-logically-earlier write can silently overwrite data, which is the classic multi-active-write hazard.

Q: What anomaly does last-writer-wins conflict resolution permit that a fund platform must guard against?
A: Last-writer-wins can silently discard a concurrent update — two regions update the same holding, the one with the larger timestamp wins and the other's change is lost with no error — which in a fund context can drop a legitimate transaction or amount. It assumes wall-clock timestamps reflect real order, but clock skew can invert that. Architects guard against it by avoiding multi-active writes on authoritative records, or by using conflict-free data types (CRDTs) or application-level merge logic for fields that must not be silently overwritten.

Q: What is a CRDT and why does it matter for the consistency spectrum?
A: A CRDT (Conflict-free Replicated Data Type) is a data structure — like a grow-only counter, OR-set, or LWW-register — whose concurrent updates on different replicas always merge to the same result regardless of order, giving *strong eventual consistency*: once all updates are delivered, replicas converge with no conflicts and no coordination. It matters because it lets an AP system avoid last-writer-wins data loss for the structures it covers. The limit is that not every business rule fits a CRDT — you cannot express "balance must not go negative" as a simple CRDT.

Q: Define "strong eventual consistency" and how it differs from plain eventual consistency.
A: Strong eventual consistency guarantees that any two replicas that have received the *same set* of updates are in the same state, with convergence achieved by deterministic conflict-free merge (as in CRDTs), so there is never a conflict to resolve. Plain eventual consistency only promises replicas converge *eventually* if updates stop, and may require conflict resolution (like last-writer-wins) that can lose data. The "strong" qualifier is about deterministic, conflict-free convergence, not about recency or linearizability.

Q: A version vector reports two writes as "concurrent" — what does that mean and how must the system respond?
A: Concurrent means neither write is in the other's causal history — each was made without having seen the other, so the version vectors are incomparable (neither dominates). The system cannot pick a "correct" order from causality alone, so it must either apply a deterministic resolution rule (last-writer-wins, CRDT merge) or surface both *siblings* to the application for semantic resolution. Dynamo-style stores return siblings; resolving them in application logic is how shopping-cart-style merges avoid losing data.

Q: How does a version vector differ from a Lamport timestamp, and which can detect concurrency?
A: A version vector (one counter per replica) captures partial causal order and can *detect concurrency* — it shows when two events are causally unrelated because neither vector dominates. A Lamport timestamp is a single scalar that gives a total order consistent with causality but *cannot* distinguish "concurrent" from "causally ordered," because it collapses everything into one number. So you use Lamport clocks when you only need a consistent tie-broken order, and version vectors when you must detect and surface conflicts.

Q: Why does a NAV recalculation that supersedes an earlier intraday estimate need monotonic writes rather than just monotonic reads?
A: Monotonic writes guarantees the two writes by the producing service — the preliminary NAV then the corrected NAV — are applied to every replica in issue order, so a replica never ends up with the preliminary value after having received the correction. Monotonic reads alone would not help if the *writes* themselves landed out of order on a replica, leaving the stale estimate as the final state. The producer-side ordering is what prevents a corrected NAV from being silently overwritten by a late-arriving earlier estimate.

Q: Two rapid updates to the same order record by one service land out of order on a replica, leaving the stale value — exact guarantee to add and one mechanism?
A: Add monotonic writes: the replica must apply a single client's writes in the order issued. One mechanism is to route all writes for a given key through the same partition leader in commit order, or to attach a per-session sequence number the replica uses to reject or reorder out-of-sequence applies. Kafka achieves the analogous property per partition via the producer's monotonic offsets plus idempotent-producer sequence numbers.

Q: How does Kafka's per-partition ordering relate to consistent-prefix consistency for a fund transaction feed?
A: Kafka guarantees messages within a single partition are delivered to consumers in the exact offset order they were produced, which is precisely consistent-prefix on that partition — a consumer sees some prefix of the log, never a later record without the earlier ones. Cross-partition ordering is *not* guaranteed, so causally related events must share a partition key to preserve their order. For a transaction feed, keying by account or fund ID onto one partition gives ordered, consistent-prefix delivery for that entity.

Q: Why does keying a Kafka topic by `ISIN` preserve per-instrument ordering but not global ordering, and when does that bite?
A: Keying by `ISIN` hashes all events for one instrument to the same partition, so they keep their produced order, but events for different ISINs land on different partitions whose relative order is undefined. It bites when a consumer needs a *global* causal order across instruments — for example a fund-level event that depends on two instruments' updates — because those updates arrive from independent partitions with no cross-partition ordering. The fix is either a single partition (sacrificing throughput) or application-level reordering using sequence/timestamp metadata.

Q: What is Spanner's TrueTime and how does it deliver external (strict-serializable) consistency?
A: TrueTime is an API that returns a time *interval* `[earliest, latest]` bounded by a known uncertainty, backed by GPS and atomic clocks, so Spanner knows the real time to within a few milliseconds. To commit, Spanner picks a timestamp and *waits out the uncertainty* (commit-wait) before releasing the transaction, guaranteeing that any later transaction gets a strictly greater timestamp — yielding strict serializability across regions. The cost is the commit-wait latency, which is the price of global real-time ordering without a single coordinator.

Q: How does Spanner's "commit-wait" turn clock uncertainty into a correctness guarantee?
A: After choosing a commit timestamp `s`, Spanner deliberately waits until TrueTime guarantees `s` is in the past everywhere (i.e., until `TT.now().earliest > s`) before making the transaction's effects visible, so no other transaction can be assigned an earlier-or-equal timestamp. This converts the clock's uncertainty window into a short, bounded delay that preserves real-time order. Without commit-wait, overlapping uncertainty intervals could let a later transaction get a smaller timestamp, breaking external consistency.

Q: A globally distributed fund ledger needs strict serializability — what is the unavoidable per-write latency floor and why?
A: The floor is roughly the cross-region round-trip time for the commit quorum plus, in Spanner-style systems, the TrueTime commit-wait (a few milliseconds), because strict serializability requires committing to a majority and respecting real-time order before acknowledging. You cannot beat the speed of light between regions, so a write spanning continents pays tens of milliseconds minimum. This is why even Spanner encourages keeping transactions within a region when strict global ordering is not required.

Q: When would you deliberately choose Cosmos DB Bounded staleness over Session for a regulated cross-region read, and what do you gain?
A: Choose Bounded staleness when a regulator or business rule demands a *quantified* worst-case staleness across regions for a globally-read value — e.g., remote regions must never read a position more than `T` seconds or `K` versions behind the write region. Session only guarantees per-session correctness for the writer, not a bound for *other* readers in remote regions. Bounded staleness converts "how stale can a remote NAV be?" into a number you can put in a control document, at the cost of write throttling if a region falls behind.

Q: Why does Cosmos DB Bounded staleness throttle *writes* (not reads) when a region falls behind, and what is the operational signal?
A: To honor the `K`/`T` bound, Cosmos DB must prevent the write region from getting further ahead than the bound allows, so when replication to a region lags toward the limit it back-pressures by throttling new writes until the lagging region catches up. The operational signal is elevated `429` (request-rate-too-large) responses or rising write latency tied to replication lag rather than to RU exhaustion. This is the concrete cost of pinning correctness to inter-region lag, and it is why Microsoft steers multi-write accounts toward Session instead.

Q: How would you empirically decide whether Eventual is "safe enough" for a read path instead of guessing?
A: Use Cosmos DB's Probabilistically Bounded Staleness (PBS) metric for the specific write/read region pair to measure how often an Eventual read actually returns up-to-date data and the distribution of staleness in milliseconds, then compare that to the path's tolerance. If PBS shows, say, 99.9% of reads are effectively strong within your tolerance window, Eventual is empirically safe; if not, step up to Session. The discipline is measuring the actual staleness distribution rather than assuming the worst or the best case.

Q: A monitoring dashboard shows Cosmos DB read latency suddenly doubling with no traffic change — how could a consistency *override* explain it?
A: A code change may have started issuing reads at the account default (Strong or Bounded staleness) instead of a per-request Session override, and Strong/Bounded reads contact two replicas (a local quorum) and cost roughly double the RUs and latency of single-replica Session/Eventual reads. So removing or forgetting a per-request weaker override silently doubles read cost. Check whether the SDK client or request `ConsistencyLevel` override was dropped, since you can only relax — never strengthen — from the account default.

Q: Can a per-request override make a Session-configured Cosmos DB account serve a Strong read, and why does this matter for design?
A: No — overrides can only *relax* consistency to the account default or weaker, never strengthen it, because the data was only ever replicated to the quorum that the account level requires. A Session account never wrote to a global majority, so there is no fresher state for a Strong read to return. This matters because you cannot retrofit linearizability per request; if a single read path genuinely needs Strong, the *account* (or that container's policy) must be provisioned for it.

Q: How does Cosmos DB's consistency guarantee interact with a multi-region read for a client that has a session token from a different region?
A: The session token is a logical version barrier, so if the client's read lands on a region whose replica has not yet caught up to the token's version, the SDK retries against other replicas and, if necessary, other regions until one at or beyond that version answers — preserving read-your-writes even across regions. The cost is added latency for the cross-region retry when the local region is behind. If no region can satisfy the token within the SDK's retry budget, the read surfaces an error rather than returning stale data.

Q: What does it mean that consistency models constrain *replicas of one object* while isolation levels constrain *transactions over many objects* — give a fund example where you need both.
A: Publishing an official NAV needs *linearizability* on that single published value (every consumer sees the one latest NAV) and, simultaneously, *serializable* isolation on the transaction that writes the NAV alongside the fund's units-outstanding and accrual records (no concurrent transaction sees a half-updated set). Linearizability handles single-object recency; serializability handles multi-object atomic invariants. A design that names only one leaves the other class of anomaly open, which is why DDIA insists you specify both axes.

Q: Why can snapshot isolation suffer write skew, and how does that differ from a consistency-model anomaly?
A: Write skew occurs under snapshot isolation when two transactions read an overlapping set, each makes a decision based on a snapshot that does not reflect the other's not-yet-committed write, and both commit, jointly violating an invariant neither violated alone (e.g., two on-call doctors both go off duty). It is an *isolation* anomaly about transaction interleaving, not a *consistency* (recency/ordering of one replica) anomaly. The fix is `SERIALIZABLE` isolation or explicit predicate locks, not a stronger replica consistency level.

Q: In a fund subscription/redemption system, how could write skew over snapshot isolation breach a capacity limit?
A: Two redemption transactions each read the fund's current outstanding-units snapshot, each independently confirms there is enough liquidity, and both commit, together breaching a redemption-gate or minimum-capital invariant that neither violated in isolation. Snapshot isolation hides their mutual conflict because each reads a snapshot taken before the other committed. Preventing it requires `SERIALIZABLE` isolation (which detects the read-write conflict and aborts one) rather than any change to replica consistency.

Q: How does PostgreSQL Serializable Snapshot Isolation (SSI) detect a conflict that plain snapshot isolation misses, and what does the application see?
A: SSI tracks read-write dependencies between concurrent transactions and aborts one when it detects a dangerous structure of dependencies that could produce a non-serializable outcome, catching write skew that plain snapshot isolation allows. The application sees a serialization-failure error — `SQLSTATE 40001` (`could not serialize access due to read/write dependencies`) — and is expected to retry the transaction. So serializable correctness comes at the price of application-level retry logic on `40001`.

Q: What SQLSTATE codes signal a serialization or deadlock retry in Postgres, and why does this matter for an idempotent loader?
A: `40001` (`serialization_failure`) and `40P01` (`deadlock_detected`) are the transient errors a `SERIALIZABLE` or conflicting transaction can raise, signalling the application should retry rather than treat the write as permanently failed. It matters for a loader because the retry must be idempotent — re-running the same logical write must not double-apply — otherwise retrying a `40001` could duplicate a trade or position. So serializable isolation and idempotent writes are complementary, not interchangeable.

Q: A cross-region failover promotes a replica and afterward an investor's two reads of their balance go newer-then-older — name the guarantee and the precise cause.
A: Monotonic reads is violated, and the precise cause is that the promoted node was an asynchronous follower missing the tail of writes (the lost-write window / non-zero RPO), so after the client had seen the newer balance, reads from the new leader regressed to the older state it actually held. The failover exposed the un-replicated gap as a backwards jump. Mitigation is to promote only a fully caught-up replica, or to track the client's last-seen version and reject a leader behind it.

Q: How do you size the maximum acceptable replication lag for a read path from a regulatory tolerance, concretely?
A: Translate the regulatory or business freshness requirement into a time bound (e.g., "remote NAV reads must be no more than 60 seconds stale"), then set the bounded-staleness `T` (or the LSN/GTID gate timeout) at or below that, and alert when measured lag approaches it. The lag bound also fixes the RPO for that path, so the same number answers both "how stale can a read be?" and "how much can a disaster lose?" Sizing from the tolerance — not from defaults — is what makes the choice defensible in a control review.

Q: What metric would you alarm on to catch a silent read-your-writes regression after a deploy, and at what threshold?
A: Alarm on replication lag for each read replica — Postgres `pg_last_wal_replay_lsn()` delta versus the primary's current LSN (or the time form via `now() - pg_last_xact_replay_timestamp()`), or the cloud provider's replica-lag metric — with the threshold set to the read path's freshness tolerance. A post-deploy regression (e.g., a lost session token cache, or reads newly routed to an async replica) shows up as lag exceeding the threshold or as a spike in "missing write" user reports. Pair the lag alarm with a synthetic write-then-read canary that asserts the write is always visible.

Q: Design a synthetic canary that proves read-your-writes on a live read path — what does it do?
A: The canary writes a uniquely-keyed marker through the production write path, then immediately reads it back through the exact production read path (same routing, pooler, cache, region) and asserts the value returned is the one just written or newer, recording any failure. Running it continuously across replica routing and during real lag detects regressions the moment they appear. For a Cosmos Session path it should also assert the session token is propagated; for Postgres it should assert reads are LSN-gated or hit the primary within the freshness window.

Q: For an EPT (European PRIIPs Template) file built from a positions snapshot, what consistency check belongs in the pipeline before the file is published?
A: Before building the EPT, gate the read of position inputs on a proven freshness marker — confirm the replica's replay LSN/GTID (or Cosmos session token / bounded-staleness bound) has reached the snapshot cut-off timestamp — so the regulatory file is built from holdings at least as fresh as the cut-off, not from a lagging replica. Publishing an EPT from stale positions yields a regulatory artifact that disagrees with the authoritative book, a compliance defect rather than a cosmetic one. The check belongs in the pipeline as a hard precondition, failing the run if freshness is not met.

Q: Why must an EET (European ESG Template) generated from sustainability data still respect read-your-writes if the data was just updated, and where is the trap?
A: If an ESG attribute was corrected just before the EET run and the pipeline reads it off a lagging replica, the template can carry the old value — a read-your-writes violation at batch scale that publishes superseded sustainability figures to distributors. The trap is treating reference-style ESG data as "slowly changing" and therefore safe under weak consistency, when a recent correction makes the key freshly-mutable. Gate the read on the correction's version, or read authoritative ESG inputs from the leader for the run.

Q: A reconciliation between two systems shows a holding present in one and absent in the other right after a batch — how do you tell a consistency-lag artifact from a genuine break?
A: Re-read both systems after forcing reads to the *authoritative leader* (bypassing lagging replicas/caches) and compare again; if the discrepancy disappears, it was a replication-lag artifact (one side read a stale replica), whereas if it persists against fresh reads it is a genuine data break. Also check each side's replica lag and the batch's read-routing at the moment of the run. The discipline is to rule out the weaker-than-assumed read path before opening a data-integrity incident.

Q: Why is "the replica is only a few seconds behind" not a sufficient answer when a user reports a missing write?
A: Because read-your-writes is violated by *any* lag greater than the gap between the user's write and their read — even sub-second lag breaks it if the read follows the write closely enough on a different replica. The relevant question is not the absolute lag but whether the read path guaranteed the user's own version; "a few seconds" can be fatal for an immediate post-write read and irrelevant for a historical read. The fix is per-session freshness (token/LSN gate or leader routing), not merely reducing average lag.

Q: How does idempotency on the write path interact with a serialization-failure retry to avoid duplicating a fund order?
A: When a `SERIALIZABLE` write aborts with `40001` and is retried, the retry must carry the same idempotency key (e.g., the client order reference) so the sink — via `INSERT ... ON CONFLICT DO NOTHING` or an upsert — recognizes the order if the first attempt actually partially applied, preventing a duplicate booking. Without the key, a retry of a transaction that committed-then-failed-to-report could double-insert. So serializable retries and idempotent writes must be designed together for a safe rerun.

Q: When promoting a Postgres standby during failover, what step preserves monotonic reads for already-connected clients, and what is the trade-off?
A: Wait until the standby has fully replayed the primary's WAL up to the last known LSN before promoting (or only promote the most-caught-up standby), so no client that already saw newer data is served an older state by the new leader. The trade-off is increased recovery time (RTO) while you wait for replay to finish, versus the data-correctness/monotonicity benefit. In a regulated platform the monotonicity guarantee usually outweighs a few extra seconds of failover time, but the choice should be explicit in the DR runbook.

Q: Summarize the architect's end-to-end procedure for assigning a consistency guarantee to a single read path.
A: First identify the path's replication topology (leader, sync/async replica, cache, region) to scope which anomalies are possible; second determine the read/write pattern (is it read soon after a write by the same actor? is the key mutable? is order observable across clients?); third pick the *minimum* guarantee that closes the harmful anomalies (read-your-writes, monotonic, consistent-prefix, causal, or linearizable) and name the concrete user-visible bug it prevents; fourth choose a mechanism (leader routing, LSN/GTID/session-token gate, bounded-staleness bound) and an enforcement check (canary, lag alarm). The output is a named, costed, testable guarantee per path — never a bare "strongly consistent."

Q: Why does Azure SQL's read-scale-out (the `ApplicationIntent=ReadOnly` secondary) carry the same replica-lag risk as any async replica, and how do you mitigate it?
A: The readable secondary in a Business Critical tier is kept up to date *asynchronously*, so a connection opened with `ApplicationIntent=ReadOnly` can serve data that lags the primary, breaking read-your-writes for a user whose write went to the primary. The mitigation is to route reads that must reflect a just-made write to the primary (default `ReadWrite` intent) and reserve the read-only secondary for staleness-tolerant reporting. The gotcha is assuming the secondary is "just a copy"; it is a lagging copy, and the lag is unbounded under load.

Q: For a globally-read fund reference-data store, why might Consistent prefix be preferable to Eventual even though both are cheap?
A: Consistent prefix costs the same single-replica read price as Eventual but additionally forbids out-of-order observation of a write sequence, so a consumer never sees a later reference update without the earlier ones it followed — avoiding a scrambled history of, say, an instrument's successive attribute changes. Eventual permits exactly that backwards/scrambled view. Since the ordering guarantee is essentially free relative to Eventual, choosing Consistent prefix removes a class of confusing anomalies at no throughput cost.

Q: A design review claims "we use Cosmos DB Session, so reads are always correct" — what is the precise hole in that claim?
A: Session only guarantees correctness *within a client session that holds the relevant session token*; a freshly-restarted client with an empty token cache, a read of a partition the client never wrote to, or a read by a *different* user all fall back to Eventual-level behavior and can be stale. So "always correct" overstates it — Session protects the writer's own session, not every reader. The review should specify whose reads, on which partitions, with what token lifecycle, rather than treating Session as a blanket guarantee.
