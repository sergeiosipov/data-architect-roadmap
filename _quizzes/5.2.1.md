# Phase 2 · 5.2.1 + 5.3.1 dbt & the SQL transformation layer — 150 self-test questions

Q: What is dbt at its core, and what does it deliberately NOT do?
A: dbt (data build tool) is a transformation framework that lets you define models as `SELECT` statements and manages their dependency graph, materialization, testing, and documentation. It deliberately does NOT extract or load data — it is the "T" in ELT, running transformations inside a warehouse you already loaded. This separation is why dbt pairs with an ingestion tool (Fivetran, Airbyte) and an orchestrator (Airflow, Dagster) rather than replacing them.

Q: What is the difference between dbt Core and dbt Cloud?
A: dbt Core is the open-source, command-line tool you install and run yourself (e.g. via `uv pip install dbt-core dbt-duckdb`), while dbt Cloud is the managed SaaS that adds an IDE, scheduler, hosted docs, CI, and column-level lineage on top of Core. For learning and for a sovereignty-constrained fund estate you run Core; dbt Cloud is the "evaluate" corporate option whose scheduler and lineage features SQLMesh notably offers for free.

Q: Why does the lesson insist on "mastery, not mere usage" of dbt?
A: Anyone can write a few models, but an architect must set transformation standards for a whole team — consistent layering, deliberate materialization choices, and tests as contracts. Mastery means you can justify every design decision and keep the warehouse reasoned-about rather than a pile of views. Without that discipline the DAG becomes unreadable and the warehouse rots into something nobody trusts.

Q: What are the three conventional layers of a dbt project and how do they map to the medallion architecture?
A: The convention is `staging` → `intermediate` → `marts`, mirroring bronze/silver/gold of medallion. Staging does light, 1:1 cleanup of raw sources (renaming, casting, deduping), intermediate holds reusable joins and computations, and marts are the business-facing fact/dimension models. This layering keeps each model's responsibility narrow and makes the DAG legible to a stranger.

Q: What belongs in a dbt staging model and what does NOT?
A: A staging model should be a thin, one-to-one wrapper over a single source: rename columns to a consistent style, cast types, apply trivial cleanup, and nothing more. It should NOT join multiple sources, aggregate, or contain business logic — that belongs in intermediate or marts. Keeping staging atomic means downstream models have a stable, predictable foundation and source changes are absorbed in one place.

Q: In dbt project structure, why are staging models usually materialized as views?
A: Staging models are cheap, frequently-rebuilt, 1:1 transformations, so materializing them as views avoids storing duplicate data and always reflects the latest source without a rebuild. The cost is paid at query time, but because staging logic is light the overhead is small. The heavier, reused marts are where you switch to tables to amortize compute.

Q: What is the purpose of the `intermediate` layer, and why not put everything in marts?
A: The intermediate layer holds the "verbs" — joins, unions, and reusable calculations — that more than one mart needs, so the logic is written once and `ref`-ed everywhere. Pushing it all into marts would duplicate logic across models and make them sprawling and hard to test. Intermediate models are typically not exposed to BI tools; they exist to keep marts thin and DRY.

Q: What does the `{{ ref('model_name') }}` function do, and why never hard-code a table name?
A: `ref` resolves to the fully-qualified relation of another dbt model and, crucially, registers a dependency edge so dbt knows build order. Hard-coding a schema.table name breaks the DAG: dbt cannot infer ordering, cannot rebuild downstream models when an upstream changes, and you lose environment portability between dev and prod. Always use `ref` for models and `source` for raw tables.

Q: What is a dbt `source` and how does it differ from a `ref`?
A: A `source` (declared in YAML and called as `{{ source('schema', 'table') }}`) points at raw tables that dbt did NOT build — the landing zone from your loader. `ref` points at models dbt itself builds. Sources give you a named, documented, freshness-checkable entry point so staging models depend on declared sources rather than raw string identifiers.

Q: How do you configure source freshness checks, and what command runs them?
A: You add a `freshness` block to the source in YAML with `warn_after` and `error_after` thresholds plus a `loaded_at_field` (e.g. a `_loaded_at` timestamp), then run `dbt source freshness`. dbt computes the max of `loaded_at_field` and compares the lag to your thresholds, warning or erroring if the data is stale. For a fund feed this is how you catch a NAV file that failed to land before downstream models even run.

Q: What are the four core materializations in dbt?
A: `view`, `table`, `incremental`, and `ephemeral`, with `snapshot` as a related but separately-defined construct. You set them with `config(materialized='...')` in the model or in `dbt_project.yml`. Each trades freshness, build cost, and query cost differently, and choosing the right one per model is exactly what "Done when" asks you to be able to explain.

Q: What does the `view` materialization do, and when is it the right default?
A: A `view` rebuilds the model as a database view — no data is stored, the `SELECT` runs every time the view is queried. It is the right default for lightweight, infrequently-queried, or staging models because it costs nothing to build and is always current. The trade-off is repeated compute on every read, which becomes expensive for heavy or hot models.

Q: What does the `table` materialization do, and what is its main downside?
A: A `table` materialization runs `create table as select`, fully rebuilding and persisting the result on every `dbt run`. It makes queries fast because the result is precomputed and stored, which suits heavy marts read by many dashboards. The downside is a full rebuild each time — expensive and slow for very large datasets — which is what `incremental` exists to fix.

Q: What is an `ephemeral` materialization and when would you use it?
A: An `ephemeral` model is never built as its own object; instead dbt inlines its SQL as a CTE into any downstream model that `ref`s it. Use it for small, reusable logic you do not want cluttering the warehouse with extra objects. The trade-off is it cannot be queried or tested directly and, if reused widely, can bloat compiled SQL into deeply nested CTEs.

Q: How does the `incremental` materialization work at a high level?
A: On the first run it builds the full table; on subsequent runs it processes only new or changed rows and inserts/merges them into the existing table rather than rebuilding everything. You gate the new rows with an `is_incremental()` block (typically a `where` filter on a timestamp greater than `max()` already loaded) so reruns stay cheap. This is the standard pattern for append-heavy fact tables like daily NAV or transaction events.

Q: What does the `is_incremental()` macro return, and when is it false?
A: `is_incremental()` returns true only when the model already exists as a table, the run is not a `--full-refresh`, and the materialization is `incremental`. On the very first build, after `dbt run --full-refresh`, or for non-incremental models it returns false, so the SQL inside the `{% if is_incremental() %}` block is skipped. This lets one model both build from scratch and do delta loads.

Q: What is the role of the `unique_key` config in an incremental model?
A: `unique_key` tells dbt which column(s) identify a row so that, instead of blindly appending, it can `merge`/`delete+insert` to update existing rows when they reappear in the new batch. Without it, late-arriving updates to an already-loaded row would create duplicates. For a NAV table keyed by `(fund_id, nav_date)`, the `unique_key` ensures a corrected NAV overwrites the prior value rather than doubling it.

Q: Name two common incremental failure modes the syllabus flags.
A: First, schema drift: if the source gains or changes a column, an incremental run can fail or silently skip the new column because the target table's shape no longer matches — often needing `on_schema_change` config or a `--full-refresh`. Second, a wrong or non-unique `unique_key`, or a flawed `is_incremental()` filter, which causes duplicates or silently dropped rows. Both fail quietly, so incrementals demand tests and occasional full-refresh validation.

Q: What does the `on_schema_change` config control for incremental models?
A: `on_schema_change` decides how dbt reacts when the model's columns differ from the existing table: `ignore` (default-ish, leave the table as-is), `append_new_columns`, `sync_all_columns`, or `fail`. Setting it to `append_new_columns` or `sync_all_columns` lets a source gaining a column flow through without a manual full-refresh. Choosing `fail` makes drift loud, which can be the safer choice in a regulated pipeline.

Q: An incremental model is silently dropping rows after a backfill. What is the first thing to check?
A: Check the `is_incremental()` filter and the `unique_key`: a `where event_time > (select max(event_time) from {{ this }})` filter will skip late-arriving rows whose timestamp is older than the current max. The fix is usually a lookback window (e.g. `> max - interval '3 days'`) plus a `unique_key` so re-seen rows update rather than duplicate, then a `--full-refresh` to repair. Confirm with a row-count or uniqueness test afterward.

Q: What does `{{ this }}` refer to inside an incremental model?
A: `{{ this }}` is the relation (schema.table) of the model currently being built — the existing target table. It is used inside `is_incremental()` blocks to read the already-loaded data, e.g. `select max(loaded_at) from {{ this }}`, so you only pull newer source rows. It only makes sense once the table exists, which is why it lives inside the incremental branch.

Q: What does `dbt run --full-refresh` do to an incremental model and when is it needed?
A: `--full-refresh` forces dbt to drop and rebuild the incremental table from scratch, ignoring the delta logic. You need it after a logic change to historical rows, a schema change the incremental strategy cannot absorb, or to repair a corrupted/duplicated table. It is the escape hatch that re-establishes correctness at the cost of a full, expensive rebuild.

Q: What incremental strategies does dbt offer and how do they differ?
A: Common strategies (set via `incremental_strategy`) include `append` (just insert new rows, no dedup), `merge` (upsert on `unique_key`, default on warehouses that support it like Snowflake/BigQuery/DuckDB), `delete+insert`, and `insert_overwrite` (replace whole partitions). `append` is fastest but allows duplicates; `merge` handles updates; `insert_overwrite` suits partitioned date-based reloads. The right choice depends on whether rows mutate and on adapter support.

Q: Why are dbt tests described as "executable data-quality contracts"?
A: A dbt test is a SQL assertion that runs in CI and on every build, so a quality rule (e.g. "`isin` is never null", "NAV per share is positive") is enforced automatically rather than living in a wiki. Because `dbt build` fails when a test fails, the contract is executable — a breach stops bad data from propagating. This turns DQ from documentation into an enforced gate.

Q: What is a generic (schema) test in dbt and what are the four built-in ones?
A: A generic test is a reusable, parameterized assertion attached to a column or model in YAML; the four built-ins are `unique`, `not_null`, `accepted_values`, and `relationships` (referential integrity). For example `not_null` and `unique` on an `isin` column, or `relationships` from a holdings model to a fund dimension. They are the cheapest, highest-leverage tests and you typically apply several per key column.

Q: What does the `relationships` generic test assert?
A: `relationships` checks referential integrity: every value in a column exists in a referenced column of another model (a foreign-key check). You configure it with `to: ref('dim_fund')` and `field: fund_id`. It catches orphaned facts — e.g. a holdings row whose `fund_id` has no matching fund — which in fund data signals a broken load or a missing dimension record.

Q: What is a singular test in dbt and where does it live?
A: A singular test is a one-off SQL query saved as a `.sql` file in the `tests/` directory that returns the rows which VIOLATE your rule; if it returns zero rows the test passes. It is for bespoke logic that no generic test covers, e.g. "rows where `sum(share_class_nav * shares)` does not reconcile to fund total NAV". Singular tests encode complex invariants that are specific to one relationship.

Q: A singular test passes when it returns how many rows, and why?
A: It passes when it returns ZERO rows, because a dbt data test is written to SELECT the failing/offending records — the assertion is "there should be no rows that break this rule". Any returned rows are violations and dbt reports their count as failures. This is the opposite intuition from many test frameworks, so it is a common beginner gotcha.

Q: What is a dbt unit test, which version introduced it, and how does it differ from a data test?
A: A unit test (introduced in dbt Core v1.8) validates a model's SQL LOGIC against small, hand-crafted mock inputs and an expected output, rather than running against real warehouse data. Data tests assert properties of already-materialized real data; unit tests run before materialization on synthetic rows to prove the transformation is correct. They are ideal for testing edge cases in complex CASE logic or a NAV calculation without needing production data.

Q: What are the key YAML blocks of a dbt unit test?
A: A unit test is defined under the top-level `unit_tests` key with `name`, `model` (the model under test), `given` (a list of mocked inputs, each an `input: ref(...)` with `rows`), and `expect` (the expected output rows). Each `given` input and the `expect` can set `format` to `dict`, `csv`, or `sql`. dbt feeds the mock rows through the model's compiled SQL and asserts the result equals `expect`.

Q: Write the idea of a unit test for a NAV-per-share calculation model.
A: You would mock a `given` input with, say, `{net_assets: 1000000, shares_outstanding: 50000}` and assert `expect` rows of `{nav_per_share: 20.0}`. This proves the division and rounding logic is correct on a known case, including edge cases like zero shares (expecting a null or guarded value). It runs without touching the real fund tables, so you can test the math deterministically in CI.

Q: How do you run only tests in dbt, and how do tests fit into `dbt build`?
A: `dbt test` runs all tests; `dbt build` runs models, snapshots, seeds, AND tests in DAG order, stopping downstream models if an upstream test fails. So `dbt build` is the gate that proves correctness end to end, which is exactly the "Done when" criterion. Running `dbt test` alone is useful for a quick assertion pass without rebuilding models.

Q: Why does `dbt build` stop building downstream models when a test fails, and why is that good?
A: `dbt build` interleaves models and their tests by DAG position, so if a model fails its test, dbt skips the models that depend on it rather than propagating bad data. This "fail fast, contain the blast radius" behavior means a NAV invariant breach upstream never reaches the EMT-shaped mart. It is the practical reason `build` is preferred over running `run` and `test` separately.

Q: What is the difference between `severity: warn` and `severity: error` on a test?
A: `severity: error` (default) makes a failing test fail the run, breaking `dbt build`; `severity: warn` logs the violation but lets the run succeed. You can also set `warn_if`/`error_if` thresholds (e.g. warn at >0 failures, error at >100). Warnings suit soft quality signals you want visibility on without blocking deployment, but core regulatory invariants should stay at `error`.

Q: What is a dbt snapshot and what modeling pattern does it implement?
A: A snapshot captures the state of a mutable source table over time and implements Slowly Changing Dimension Type 2 (SCD2): when a tracked row changes, the old version is closed off and a new version is opened, preserving full history. dbt does this by adding validity columns and inserting new versions rather than overwriting. It is how you keep an auditable history of attributes that the source overwrites in place.

Q: Why are snapshots essential for auditable fund/share-class dimensions?
A: Source systems usually overwrite an attribute (e.g. a share class's management fee or distribution policy) in place, destroying the prior value. A snapshot preserves every historical version with valid-from/valid-to ranges, so an auditor can ask "what was this share class's fee on a given NAV date" and get the answer. In a regulated fund-admin context that point-in-time reconstructability is often a hard requirement.

Q: What meta-columns does a dbt snapshot add to the output table?
A: dbt adds `dbt_valid_from` (when this version became current), `dbt_valid_to` (when it was superseded, NULL/`dbt_valid_to_current` for the live row), `dbt_scd_id` (a surrogate key unique per version), and `dbt_updated_at`. With `hard_deletes='new_record'` it also adds `dbt_is_deleted`. These columns are what turn a flat extract into queryable SCD2 history.

Q: What are the two snapshot strategies and how do they detect change?
A: The `timestamp` strategy uses an `updated_at` column — if the source row's timestamp is newer than the snapshot's, it records a new version; this is preferred when a reliable last-modified column exists. The `check` strategy compares a list of `check_cols` (or `'all'`) value-by-value and records a new version when any differ; use it when there is no trustworthy timestamp. Timestamp is cheaper and more reliable; check is the fallback.

Q: In a snapshot, what does the `unique_key` config identify and why must it be truly unique?
A: `unique_key` identifies the business entity whose history you are tracking (e.g. `share_class_id`), so dbt can match a source row to its existing snapshot versions. If it is not actually unique per entity, dbt mismatches rows and corrupts the history — opening and closing the wrong versions. For composite keys you may concatenate or hash the parts into one deterministic key.

Q: How are dbt snapshots now defined in recent versions (v1.9+) versus older ones?
A: In dbt Core v1.9+, snapshots are defined in YAML files under a `snapshots:` key with `name`, `relation: ref(...)`, and a `config` block (strategy, unique_key, updated_at/check_cols), replacing the older `.sql` file with a `{% snapshot %}` block. The YAML approach is the recommended modern form and makes configs consistent with the rest of the project. You run them with `dbt snapshot` or as part of `dbt build`.

Q: What does the `dbt_valid_to_current` config do, and why might you set it for downstream joins?
A: By default a current (open) snapshot row has `dbt_valid_to = NULL`; setting `dbt_valid_to_current` (e.g. to `'9999-12-31'`) writes a sentinel high-date instead. This makes point-in-time `between dbt_valid_from and dbt_valid_to` joins simpler because you avoid special-casing NULL. It is a v1.9 convenience for cleaner SCD2 range queries.

Q: What does `hard_deletes='new_record'` change about snapshot behavior?
A: By default snapshots do not detect rows deleted from the source. With `hard_deletes='new_record'` (v1.9+), when a previously-snapshotted row disappears, dbt inserts a final version marked with `dbt_is_deleted = true` and closes the prior version. This lets you audit "this share class existed until date X then was removed", which a plain snapshot would silently leave open forever.

Q: Why must a snapshot run reliably and on schedule rather than ad-hoc?
A: A snapshot only records the states it actually observes; if it misses a run, an intermediate value that existed between runs is lost forever, leaving a gap in the SCD2 history. For audit-grade fund dimensions this means snapshots must run on a dependable cadence aligned to how often the source can change. Unlike a model, you cannot reconstruct missed history with a `--full-refresh`.

Q: What does `dbt docs generate` produce and what does `dbt docs serve` do?
A: `dbt docs generate` compiles a documentation site (a `catalog.json` plus `manifest.json`) describing every model, column, test, source, description, and the lineage graph. `dbt docs serve` then hosts that as a local browsable website with an interactive DAG. Together they turn your project into self-documenting, navigable artifacts — part of the "Done when" docs criterion.

Q: How do you document a model and its columns in dbt?
A: You add `description` keys in the model's YAML — at the model level and per column — and these flow into the generated docs and lineage site. You can embed reusable doc blocks with `{{ doc('block_name') }}` defined in `.md` files for longer descriptions. Good descriptions are what make the DAG "readable by a stranger", the explicit done-when condition.

Q: What is a dbt exposure and why would you add one for an EMT-shaped output?
A: An exposure is a YAML-declared downstream consumer of your dbt models — a dashboard, a report, or in this case an EMT (European MiFID Template) file feed — that appears in the lineage graph and docs. Declaring it makes the dependency explicit so anyone can see which models feed the regulatory EMT output and what breaks if they change. It also lets you run `dbt build --select +exposure:emt_feed` to rebuild everything that exposure depends on.

Q: What keys does a dbt exposure require?
A: An exposure needs `name`, `type` (e.g. `dashboard`, `application`, `analysis`, `ml`), `owner` (with `name`/`email`), and `depends_on` listing the `ref()`s it consumes; `url`, `description`, and `maturity` are optional. The `depends_on` is what wires it into the lineage graph. For the EMT feed you would list the mart model(s) that produce the template's rows.

Q: What is the dbt-utils package and name two macros it provides.
A: dbt-utils is the most widely-used dbt package, providing reusable macros and generic tests; examples include `generate_surrogate_key` (deterministic hashing of business keys into a surrogate) and `equality`/`unique_combination_of_columns` tests, plus `star`, `union_relations`, and `date_spine`. You add it via `packages.yml` and `dbt deps`. It saves reinventing common SQL patterns and standardizes them across the team.

Q: What does the dbt-expectations package add, and what is it modeled on?
A: dbt-expectations is a package of generic tests modeled on the Python Great Expectations library, giving you assertions like `expect_column_values_to_be_between`, `expect_column_values_to_match_regex`, and `expect_column_values_to_be_in_set`. It dramatically expands testable DQ rules beyond dbt's four built-ins — e.g. asserting an `isin` matches a 12-character regex or a NAV is within a plausible range. It is the go-to package for richer data-quality contracts.

Q: How do you declare and install dbt packages?
A: You list them in a `packages.yml` file with each package's source and version (e.g. a `package:` name and `version:`, or a git/local source), then run `dbt deps` to download them into `dbt_packages/`. Pinning versions keeps builds reproducible. After installation their macros and tests are available throughout your project.

Q: Why pin package versions in `packages.yml` rather than tracking latest?
A: Pinning (e.g. a specific `version:` range) makes builds reproducible and prevents a surprise upstream change from breaking your tests or macros between runs. In a regulated environment, reproducibility and a controlled upgrade path are themselves requirements. Tracking latest invites silent behavior changes that are hard to audit.

Q: How would you use dbt-expectations to assert ISIN format on a column?
A: You attach the generic test `dbt_expectations.expect_column_values_to_match_regex` to the `isin` column with a regex for a 12-character alphanumeric code (two-letter country prefix, nine alphanumerics, one check digit). This enforces format at build time so a malformed ISIN fails `dbt build` before it reaches the EMT output. You could pair it with `not_null` and `unique` for a complete contract on the identifier.

Q: What is a surrogate key and why generate one with `dbt_utils.generate_surrogate_key`?
A: A surrogate key is a single deterministic value derived (hashed) from one or more business-key columns, used as a stable join key independent of the source's natural keys. `dbt_utils.generate_surrogate_key(['fund_id','share_class_id'])` produces the same hash for the same inputs every run, so it is reproducible across environments. It standardizes composite-key handling and is heavily used in dimensional and Data-Vault-style modeling.

Q: What does `dbt compile` do versus `dbt run`?
A: `dbt compile` renders your Jinja+SQL into the final executable SQL in the `target/` directory WITHOUT executing it against the warehouse, letting you inspect exactly what will run. `dbt run` compiles and then executes, building the models. `compile` is invaluable for debugging a macro or `is_incremental()` block before paying compute.

Q: What does `dbt parse` do and why is it useful in CI?
A: `dbt parse` reads the project and builds the `manifest.json` (the graph and metadata) without running anything, validating that refs, configs, and YAML resolve. In CI it is a fast structural sanity check that catches a broken `ref`, a typo'd config, or a missing dependency before you spend compute. It is also how tools that consume the manifest (lineage, catalogs) get their input.

Q: What is the dbt manifest (`manifest.json`) and why does it matter?
A: `manifest.json` is the compiled representation of your entire project — every node (model, test, source, snapshot, exposure), its config, dependencies, and compiled SQL. It is the source of truth for the DAG, docs, state comparison, and external integrations. Many advanced workflows (state-based selection, lineage tools, CI) read it.

Q: What does `dbt build --select state:modified+ --defer` enable?
A: This is "slim CI": `state:modified+` selects only models changed since a stored manifest plus their downstream children, and `--defer` resolves unchanged upstream refs to the production relations instead of rebuilding them. The result is a CI run that builds only what your change affects, not the whole project. It dramatically cuts CI time and cost on large dbt projects.

Q: How do you select a model and everything downstream of it, versus everything upstream?
A: dbt's graph selectors use `+` as a position operator: `model_name+` selects the model and all descendants (downstream), `+model_name` selects the model and all ancestors (upstream), and `+model_name+` selects both. You can also bound depth with a number, e.g. `2+model_name`. This is how you rebuild exactly the affected slice of the DAG, e.g. `+emt_mart` to rebuild everything feeding the EMT output.

Q: What is the difference between `dbt seed`, `dbt run`, and `dbt snapshot`?
A: `dbt seed` loads small static CSV files from the `seeds/` directory into tables (e.g. a country-code or fund-type lookup); `dbt run` builds models from `SELECT` statements; `dbt snapshot` captures SCD2 history of mutable sources. `dbt build` runs all of these plus tests in dependency order. Seeds are for small reference data only, not bulk loading.

Q: Where do model configs come from and what is the precedence?
A: Configs can be set in `dbt_project.yml` (broad, folder-level defaults), in a model's YAML properties file, or inline via `{{ config(...) }}` in the model SQL, with inline overriding YAML overriding project-level. This lets you set a default materialization for a whole folder (e.g. `staging` as views) and override individual models. Knowing the precedence prevents surprises about which materialization actually applies.

Q: How do you set all models in the `staging` folder to materialize as views by default?
A: In `dbt_project.yml` under the `models:` key, nest your project name then the folder, e.g. `models: my_project: staging: +materialized: view`. The `+` prefix marks it as a config that cascades to everything in that path. Any individual staging model can still override with an inline `config(materialized='table')`.

Q: What is a dbt profile and where does `profiles.yml` live?
A: `profiles.yml` holds the connection details (adapter type, host, database, schema, credentials, threads) for each target environment (dev, prod), and lives by default in `~/.dbt/` on your Ubuntu machine — outside the project so secrets are not committed. The project's `dbt_project.yml` names which `profile` to use. Keeping it out of the repo is the standard way to avoid leaking warehouse credentials.

Q: What does the `threads` setting in a dbt profile control?
A: `threads` sets how many models dbt builds in parallel — dbt walks the DAG and runs independent nodes concurrently up to that limit. More threads speed up builds on a warehouse that can handle the concurrency, but too many can overwhelm a small engine like a single DuckDB process. For local DuckDB work a modest number (e.g. 4) is typical.

Q: Why is dbt-duckdb a good adapter for local hands-on learning?
A: dbt-duckdb runs the warehouse in-process as an embedded DuckDB database — no server to stand up, fast, and it reads/writes Parquet and CSV directly, which suits a local medallion lake. It lets you rebuild the Phase-1 mart entirely on your laptop with zero infrastructure. The trade-off is it is single-process, so it is for development and small data, not a shared production warehouse.

Q: What is dbt-trino and when would you reach for it over dbt-duckdb?
A: dbt-trino is the adapter that runs dbt transformations through a Trino (distributed query engine) cluster, federating across many sources and scaling to large data. You reach for it when DuckDB's single-process model is too small or when you must query a real lakehouse/multiple catalogs. For learning you prototype on DuckDB; for a production shared compute layer Trino is a realistic target.

Q: `dbt build` fails with "Compilation Error ... depends on a node named X which was not found". What does it mean?
A: It means a `ref('X')` (or `source(...)`) points at a model/source that does not exist in the project — a typo, a deleted model, or a file not saved. dbt resolves the DAG at parse time and cannot find node X. Fix the name, confirm the model file exists, and re-run; `dbt parse` would have caught it early.

Q: A model builds but a `unique` test fails on its primary key. What does that tell you and how do you investigate?
A: It tells you the model is producing duplicate key values — usually a fan-out join (a one-to-many you treated as one-to-one) or a missing dedup in staging. Investigate by querying the model grouped by the key with `having count(*) > 1` to find offending rows, then trace which join multiplied them. The fix is correcting the join grain or adding a dedup, not loosening the test.

Q: What is "fan-out" in a dbt model and why is it a frequent test-failure cause?
A: Fan-out is when a join to a table with multiple matching rows multiplies the result set, inflating row counts and breaking `unique` tests and aggregations. It happens when you join on a key that is not unique on the joined side. The cure is to either aggregate the many-side to the correct grain first (often in an intermediate model) or join on the correct unique key.

Q: Why is "the DAG is readable by a stranger" an explicit done-when criterion?
A: Because an analytics team inherits and extends each other's models; if the lineage is a tangle of cryptic names and skipped layers, onboarding and debugging cost explode. A medallion-layered DAG with clear staging/intermediate/marts naming and descriptions lets a newcomer trace any number back to its sources unaided. Readability is a maintainability property, not cosmetics.

Q: How do you decide a model should be `table` rather than `view`?
A: Choose `table` when the model is queried often enough that recomputing it on every read is more expensive than periodically rebuilding and storing it, or when its logic is heavy (large joins/aggregations) and downstream consumers need fast reads. Views win when freshness matters more than read latency and the logic is cheap. The deciding question is read frequency × compute cost versus rebuild cost.

Q: When is `incremental` the right materialization over `table`?
A: Choose `incremental` when the model is large and append-dominant — most rows never change once written (event logs, daily NAV, transactions) — so rebuilding the whole table each run is wasteful. Incremental processes only the delta, slashing build time and cost. If rows frequently mutate across their whole history or the table is small, a plain `table` is simpler and less error-prone.

Q: For an audited share-class attribute history, which dbt construct do you choose and why?
A: A snapshot, because it implements SCD2 with validity ranges and preserves every prior value of attributes the source overwrites — exactly what auditors need to reconstruct point-in-time state. An incremental model only accumulates new rows; it does not version mutable attributes. The snapshot's `dbt_valid_from`/`dbt_valid_to` columns are the audit trail.

Q: What is a dbt macro and give a practical use.
A: A macro is a reusable piece of Jinja-templated SQL (defined in `macros/`) callable from models, like a function. A practical use is a `cents_to_currency` formatter, a dynamic `pivot`, or a `limit_data_in_dev` macro that caps row counts in the dev environment to speed local builds. Macros keep SQL DRY and encode team conventions.

Q: What is the dbt `target` variable and a common use for it?
A: `target` exposes the current run's profile context — `target.name` (e.g. `dev` or `prod`), `target.schema`, `target.database`. A common use is `{% if target.name == 'dev' %}` to limit data or change behavior only in development, e.g. filtering to recent NAV dates so local builds stay fast. It lets one codebase behave differently per environment.

Q: What is the difference between dbt's `vars` and `env_var`?
A: `vars` are project variables defined in `dbt_project.yml` or passed with `--vars` on the CLI, read via `{{ var('name') }}`, for configurable build-time values. `env_var` reads operating-system environment variables via `{{ env_var('NAME') }}`, the standard way to inject secrets (like warehouse passwords) without committing them. Use `vars` for logic toggles, `env_var` for secrets and environment-specific config.

Q: How do dbt tags help and how do you run by tag?
A: A `tags` config labels models/tests (e.g. `tags: ['nav','daily']`) so you can select a logical group regardless of folder, via `dbt build --select tag:daily`. This is handy for scheduling subsets — running only the daily-NAV slice in a morning job, say. Tags cut across the DAG where folder structure cannot.

Q: What is the difference between `dbt run --select` and `--exclude`?
A: `--select` builds only the nodes matching the selector (and is the primary way to scope a run), while `--exclude` removes matching nodes from whatever `--select` chose (or from the full set). For example `dbt build --select marts --exclude tag:slow` builds the marts but skips the slow ones. They compose, with exclude applied after select.

Q: What does a freshness `error_after` breach mean operationally for a fund pipeline?
A: It means the source's newest row is older than your tolerance — the upstream feed (e.g. the daily NAV or positions file) failed to land or arrived late. `dbt source freshness` errors, and your orchestrator should halt downstream transformation rather than compute NAV on stale data. Catching it at the source gate prevents publishing a stale figure to the EMT/EPT output.

Q: Why might you NOT test every column with `not_null`, and what is the discipline instead?
A: Over-testing creates noise and slow builds, and some columns are legitimately nullable, so blanket `not_null` produces false failures or trains people to ignore tests. The discipline is to test invariants that actually matter: keys (`unique`, `not_null`), referential integrity, accepted value sets, and domain rules like positive NAV. Tests should encode contracts, not decorate every column.

Q: What is the medallion-to-dbt mapping for a NAV pipeline, concretely?
A: Bronze/staging = lightly-cleaned raw feeds (`stg_nav_raw`, `stg_positions`), silver/intermediate = conformed joins and computed share-class figures (`int_share_class_nav`), gold/marts = business-facing facts and dimensions (`fct_nav`, `dim_share_class`) and the EMT-shaped output. Each layer adds exactly one kind of responsibility. This makes the lineage explain itself and isolates where any error entered.

Q: Why are dbt snapshots loaded insert-only / non-destructively, and how does that aid audit?
A: Snapshots never update or delete historical versions; they only close a version (set `dbt_valid_to`) and insert the new one, so the table is an append-only history. This immutability means an auditor can trust that a past version was never silently rewritten. It mirrors the insert-only philosophy that makes patterns like Data Vault auditable.

Q: How would you encode a NAV invariant as a singular test?
A: Write a `.sql` file in `tests/` selecting rows that violate the rule, e.g. `select * from {{ ref('fct_nav') }} where nav_per_share <= 0 or nav_per_share is null`. If any rows return, `dbt build` fails, flagging an impossible NAV. You could extend it to reconcile sum of share-class assets against total fund net assets within a tolerance.

Q: What is "column-level lineage" and why is it valuable for regulated data?
A: Column-level lineage traces how each output column is derived from specific upstream columns, not just which tables feed which. For regulated outputs it lets you prove provenance — "this EMT field comes from these source columns through these transforms" — and assess the precise blast radius of a change. dbt Core gives model-level lineage; column-level is a dbt Cloud / SQLMesh capability.

Q: What is SQLMesh and why is it positioned as dbt's challenger in this lesson?
A: SQLMesh is an open-source SQL transformation framework, backwards-compatible with dbt projects, that adds capabilities dbt charges for in Cloud — a built-in scheduler, automatic column-level lineage, and a Terraform-style plan/apply workflow with virtual environments. It is the "evaluate at eval level" alternate you compare against dbt to understand where dbt's model ends. The lesson wants you aware of it, not migrated to it.

Q: What are SQLMesh "virtual data environments" and what problem do they solve?
A: Virtual data environments let developers test changes in an isolated environment WITHOUT physically copying production data — SQLMesh reuses already-built tables and only materializes the models that actually changed, swapping views to point at the right physical objects. This makes spinning up a dev environment near-instant and cheap, versus dbt typically rebuilding into a separate dev schema. It is SQLMesh's headline efficiency feature.

Q: How does SQLMesh's plan/apply workflow differ from `dbt run`?
A: SQLMesh separates `plan` (compute and display the exact blast radius of a change — which models are breaking vs non-breaking, what will be backfilled) from `apply` (execute it), like Terraform. `dbt run` compiles and executes in one step with no pre-execution preview of impact. The plan step lets you see and approve changes before spending warehouse compute.

Q: How does SQLMesh derive column-level lineage automatically?
A: SQLMesh parses your raw SQL with SQLGlot (a SQL parser/transpiler), giving it a real understanding of the query's structure, so it can compute column-level lineage and validate models at compile time without running them. dbt treats SQL more as opaque text it templates, so it does not get column-level lineage natively. The SQLGlot foundation is why SQLMesh can offer compile-time checks dbt cannot.

Q: SQLMesh claims it rebuilds only changed tables — how does it know what changed?
A: SQLMesh fingerprints each model (a snapshot of its definition and dependencies) and tracks state, so on a `plan` it compares fingerprints and intervals to determine which models are new, modified, or unchanged, rebuilding only what differs. Combined with virtual environments, unchanged models are reused rather than recomputed. This is the mechanism behind its speed/cost advantage in benchmarks.

Q: Name one real trade-off of adopting SQLMesh over dbt today.
A: SQLMesh introduces several new concepts — plan/apply, model kinds, interval/state tracking, virtual environments, snapshot fingerprinting — that carry a steeper learning curve and a smaller ecosystem/community than dbt's. The payoff (cost, speed, free lineage) is real, but the upfront investment and ecosystem maturity are genuine costs. That is why the lesson keeps it at evaluation level rather than mandating a switch.

Q: Why does the lesson combine 5.2.1 and 5.3.1 (dbt and the SQL transformation layer) into one block?
A: Because dbt IS the SQL transformation layer in this stack — learning dbt's project structure, materializations, tests, and snapshots is learning how the transformation layer is built, modeled, and governed. Treating them separately would artificially split tool mechanics from the architectural role they serve. The combined block teaches dbt as the modeling-to-production bridge, not as an isolated CLI.

Q: How does dbt help "data-quality contracts become executable" specifically?
A: A contract like "every NAV row has a non-null positive value and a valid ISIN" is written as `not_null`/`accepted_values`/dbt-expectations tests and singular tests, which run on every `dbt build` and fail the pipeline on breach. The contract is no longer prose — it is code the CI enforces. This is the concrete sense in which dbt operationalizes data quality.

Q: Your `dbt build` is green locally but fails in CI on `state:modified`. What is the likely cause?
A: CI compares against a stored production manifest (the deferred state); if that artifact is missing, stale, or the `--state` path is wrong, `state:modified` selects nothing or the wrong set, and `--defer` cannot resolve upstream refs. Check that CI downloads the latest production `manifest.json` and that `--state`/`--defer` point at it. A mismatch between local full builds and CI slim builds is almost always a state-artifact problem.

Q: Why should staging models rename and recast even when the source "looks fine"?
A: Standardizing names (snake_case, consistent prefixes) and types at the boundary insulates every downstream model from source quirks, so when the source changes a column name or type you fix it once in staging. It also makes the warehouse vocabulary consistent for the whole team. "Looks fine now" is not a contract; the staging layer makes the shape explicit and stable.

Q: What does `dbt build` do when a snapshot is included, and in what order?
A: `dbt build` runs seeds, models, snapshots, and tests together in dependency order — snapshots run at their DAG position so models that `ref` a snapshot get the freshly-captured history, and tests on each node run right after it builds. This single command is why the done-when criterion is "`dbt build` is green from scratch". It guarantees the whole project, including SCD2 capture, is consistent in one pass.

Q: How do you verify "only changed rows moved" after an incremental run?
A: Inspect the run's row-count/insert metrics (dbt logs rows affected), or compare the table before and after, or query for rows with the latest `loaded_at`/batch marker. You can also add a test or a `dbt run --select model+` with logging and confirm the inserted count matches the source delta. If the count equals a full rebuild, your incremental filter is not actually filtering.

Q: What is the risk of using `check` strategy with `check_cols: 'all'` on a wide table?
A: Comparing every column on every run is expensive and brittle — any incidental change (even a reformatted but semantically identical value, or a volatile audit column) triggers a new SCD2 version, bloating history with noise. It is better to list only the business-meaningful `check_cols` you actually want to version. Reserve `'all'` for narrow tables where every column is significant.

Q: Why declare an exposure for a regulatory feed even though dbt does not "build" it?
A: The exposure makes an otherwise-invisible downstream dependency explicit in lineage and docs, so the regulatory EMT/EPT consumer is part of the impact analysis when an upstream model changes. It also documents ownership and lets you run `+exposure:name` to rebuild its full upstream chain. Without it, someone could break the regulatory output by changing a model and never realize it fed a filing.

Q: What is the gotcha with `ephemeral` models and debugging?
A: Because ephemeral models are inlined as CTEs and never materialized, you cannot query them directly to inspect intermediate results, and errors surface inside the consuming model's compiled SQL where they are harder to locate. Heavy reuse also produces deeply nested CTEs that some engines optimize poorly. When debugging, temporarily switch an ephemeral model to `view` to inspect it.

Q: How do generic tests and singular tests divide responsibility in a mature project?
A: Generic tests cover the broad, repeatable invariants on keys and columns (`unique`, `not_null`, `relationships`, accepted values, dbt-expectations ranges/regex), applied densely and cheaply. Singular tests encode the few complex, relationship-spanning business rules that no parameterized test expresses — like cross-model NAV reconciliation. Most assertions are generic; a handful of singular tests carry the bespoke domain logic.

Q: Final synthesis: how would you justify the materialization of each layer to a stranger reviewing your DAG?
A: Staging as `view` because it is cheap 1:1 cleanup that should always reflect the source; intermediate as `view` or `ephemeral` when light, `table` when reused heavily; marts as `table` for fast BI reads, switching append-heavy facts like daily NAV to `incremental` to avoid full rebuilds; and mutable dimensions as `snapshot` for auditable SCD2 history. The rule is freshness-vs-read-cost-vs-rebuild-cost, decided per model — and being able to state that reasoning is the explicit "Done when" criterion.

Q: What does the `store_failures: true` test config do and where do the failing rows land?
A: `store_failures: true` makes dbt write the rows a test flagged into a table instead of discarding them, so you can inspect the actual offending records after a `dbt test`. By default they land in a schema suffixed `_dbt_test__audit` (e.g. `myschema_dbt_test__audit`) in a table named after the test. For a NAV pipeline this lets you pull the exact rows where `nav_per_share <= 0` rather than just seeing a failure count.

Q: What is `store_failures_as` and how does it relate to `store_failures`?
A: `store_failures_as` (introduced as a dedicated config) specifies the materialization of the stored failures — `ephemeral` (nothing stored, the default), `table`, or `view`; if both are set, `store_failures_as` takes precedence over `store_failures`. So `store_failures_as: table` is the modern, explicit way to persist failing rows. It is configurable in the same places as `store_failures`: generic test YAML, singular test SQL, or `dbt_project.yml`.

Q: What does the `--limit` flag on `dbt show` do and when is it useful?
A: `dbt show --select model_name --limit 5` compiles and runs the model's query and previews the first few result rows in the terminal without materializing it, with `--limit` capping how many rows come back (default 5). It is a fast way to eyeball a transformation's output during development. Unlike `dbt run` it does not build the object, so it is non-destructive inspection.

Q: What is the `incremental_predicates` config and what performance problem does it solve?
A: `incremental_predicates` is a list of SQL expressions added to the `merge`/`delete+insert` statement to filter the DESTINATION table during an incremental run, e.g. `["DBT_INTERNAL_DEST.nav_date > dateadd(day, -7, current_date)"]`. Without it, a merge scans the entire target to find matching keys; the predicate enables partition pruning so the warehouse only scans recent partitions. On a large `fct_nav` table it is the single biggest lever to keep merge cost bounded.

Q: What do `DBT_INTERNAL_DEST` and `DBT_INTERNAL_SOURCE` alias in an incremental merge?
A: During a `merge`-strategy incremental run dbt aliases the existing target table as `DBT_INTERNAL_DEST` and the temporary table of new rows as `DBT_INTERNAL_SOURCE`. You reference these aliases inside `incremental_predicates` to filter which side of the merge is scanned. Knowing them is what lets you write a pruning predicate like `DBT_INTERNAL_DEST.fund_id = DBT_INTERNAL_SOURCE.fund_id`.

Q: How would you debug a Jinja macro that produces wrong SQL without burning warehouse compute?
A: Run `dbt compile --select the_model` and open the rendered file under `target/compiled/...` to read the exact SQL dbt will execute, including the macro's expansion. Because `compile` never touches the warehouse you can iterate on the macro for free until the SQL is right. You can also use `{{ log(..., info=True) }}` inside the macro to print intermediate values during compilation.

Q: What does `dbt run-operation` do and give a fund-data use for it.
A: `dbt run-operation macro_name --args '{...}'` invokes a macro directly from the CLI outside of a model build, typically for maintenance or DDL tasks. A practical use is a macro that grants `select` on all marts to a BI role, or one that vacuums/optimizes the `fct_nav` table on a schedule. It lets you script warehouse housekeeping with the same Jinja you already use in models.

Q: What does `dbt debug` check and when do you run it first?
A: `dbt debug` validates your setup — it confirms `dbt_project.yml` and `profiles.yml` are found and parse, tests the warehouse connection, and reports the adapter and dbt versions. You run it first whenever a build fails for an unclear reason, because it isolates "is my connection/config even valid" from "is my SQL wrong". A failed connection here points at credentials, host, or a missing `profiles.yml` rather than model logic.

Q: What does `dbt clean` remove and why might you run it after switching branches?
A: `dbt clean` deletes the directories listed under `clean-targets` in `dbt_project.yml` — typically `target/` (compiled artifacts, `manifest.json`) and `dbt_packages/`. After switching branches the cached `manifest.json` or installed packages can be stale relative to the new code, causing confusing parse errors. Running `dbt clean` then `dbt deps` gives you a clean slate so the next build reflects the current branch.

Q: A model compiles fine but `dbt run` errors with a warehouse "column does not exist" message. Where is the bug, and how does this differ from a parse error?
A: The bug is in the SQL semantics — a `ref` resolves and Jinja renders (so parse and compile pass), but the actual column you selected is misnamed or absent in the upstream relation, which only the warehouse catches at execution. A parse error, by contrast, is a dbt-side failure (bad `ref`, malformed Jinja) caught before any SQL runs. Inspect the compiled SQL in `target/compiled/` and check the real column names in the upstream model.

Q: How do you apply a generic test to a combination of columns rather than a single column?
A: Use the `dbt_utils.unique_combination_of_columns` test (from dbt-utils) at the model level in YAML, passing `combination_of_columns: ['fund_id', 'nav_date']`. dbt's built-in `unique` only tests one column, so for a composite grain you either use this dbt-utils test or build a surrogate key and test `unique` on it. It is the right tool to assert that `(fund_id, nav_date)` is unique in `fct_nav`.

Q: What does the `dbt_utils.accepted_range` test assert and when would you use it over `accepted_values`?
A: `dbt_utils.accepted_range` checks a numeric column stays within `min_value`/`max_value` bounds (e.g. `min_value: 0` for a NAV per share), whereas `accepted_values` checks membership in a discrete set of allowed strings/codes. Use the range test for continuous quantities and `accepted_values` for categoricals like a fund-type or currency code. They are complementary domain-rule tests.

Q: How do you write a `where` filter on a generic test to scope it to a subset of rows?
A: Add a `config: { where: "nav_date >= '2024-01-01'" }` block under the test in YAML, so the test only evaluates rows matching that predicate. This is useful when historical data legitimately violates a rule you only want enforced going forward, or to skip a known-bad backfill region. It scopes the assertion without weakening it for current data.

Q: What is the `--store-failures` flag versus the config, and how do you query the result?
A: `dbt test --store-failures` (or `dbt build --store-failures`) turns on failure persistence for that invocation even if not set in config, writing offending rows to the audit schema. You then query the produced table directly in your warehouse (e.g. `select * from myschema_dbt_test__audit.not_null_fct_nav_isin`) to triage. It is the fastest path from "a test failed" to "here are the exact bad rows".

Q: How does `dbt build` order a seed, a snapshot, a model that refs both, and that model's test?
A: `dbt build` topologically sorts the whole DAG, so the seed and snapshot build first (they are upstream), then the model that `ref`s them, then the model's tests run immediately after it materializes. If the snapshot or seed fails, the dependent model is skipped; if the model's test fails, anything downstream of the model is skipped. This per-node interleaving is what contains a bad NAV before it spreads.

Q: What is the difference between `dbt list` and `dbt ls`, and what does it output?
A: They are the same command (`ls` is the alias) and they print the nodes matching a selector WITHOUT building anything, e.g. `dbt ls --select +fct_nav` lists `fct_nav` and all its ancestors. It is invaluable for previewing exactly what a selector will touch before you run an expensive build. You can change the output with `--output name|path|json|selector`.

Q: You want to rebuild only the models a single source feeds. What selector do you use?
A: Use the `source:` selector with a downstream operator, e.g. `dbt build --select source:raw_nav+`, which selects every model descended from that source. This is how you rebuild exactly the slice affected by a re-landed NAV file without touching unrelated marts. Pair it with `--exclude` if some descendants should be skipped.

Q: What does `dbt build --select result:error+ --state ./prev` do and when is it used?
A: The `result:` selector reads the previous run's `run_results.json` (pointed to by `--state`) and selects nodes by their last status, so `result:error+` rebuilds everything that errored plus its downstream. It is the retry-after-failure pattern: re-run only what broke and what depends on it, not the whole project. `result:fail` and `result:warn` work the same way for test outcomes.

Q: Why is `+` used both before and after a model name, and what does `2+fct_nav` mean?
A: `+model` selects ancestors (upstream), `model+` selects descendants (downstream), and a number bounds the depth — `2+fct_nav` selects `fct_nav` plus up to two levels of ancestors. This lets you rebuild a precise neighborhood rather than the entire lineage. `+fct_nav+` would take all ancestors and all descendants.

Q: What is the `@` selector operator in dbt and how does it differ from `+`?
A: The `@model` operator selects the model, all its descendants, AND all ancestors of those descendants — the full set needed to build everything downstream from scratch. Plain `model+` only takes the model and its descendants, which can leave a descendant's other upstream dependency unbuilt. `@` is the safe choice when you must build a subtree in a fresh schema with no deferred state.

Q: How do you parametrize a singular test so it is reusable, turning it into a custom generic test?
A: Move the SQL into a file under `tests/generic/` (or `macros/`), wrap it in `{% test my_rule(model, column_name) %} ... {% endtest %}`, and reference `{{ column_name }}` and `{{ model }}` inside; it then becomes attachable in YAML like a built-in. This converts a one-off `tests/*.sql` query into a parameterized assertion you can reuse across columns. For example a `positive_value` generic test applied to both `nav_per_share` and `total_net_assets`.

Q: What practical difference does materializing a test failure `as: view` versus `as: table` make?
A: `store_failures_as: view` recreates the failing-rows query as a view, so it always reflects the latest data each time you query it but stores nothing; `store_failures_as: table` snapshots the failing rows at test time into a persisted table. Use `view` for lightweight, always-current triage and `table` when you need a frozen record of exactly what failed during a given run — useful for a regulated audit trail. The view re-runs the test SQL on read; the table does not.

Q: How would you make a model behave differently in CI versus production using only built-in context?
A: Branch on `target.name`, e.g. `{% if target.name == 'ci' %}` to add a `limit` or a recent-date filter so CI builds on a small slice, while prod processes everything. The CI target is just another profile in `profiles.yml` selected with `dbt build --target ci`. This keeps one codebase that scales its data volume to the environment without duplicating models.

Q: What is the risk of putting business logic in a `view`-materialized staging model that many marts query?
A: Every downstream query re-executes the view's SQL, so heavy logic placed in staging is recomputed once per consuming query, multiplying compute. The convention keeps staging 1:1 and light precisely so its view materialization stays cheap; expensive joins belong in `table`-materialized intermediate/marts where the cost is paid once per build. Misplaced logic in a hot staging view is a classic silent cost leak.

Q: When a snapshot's `unique_key` is genuinely composite, what is the recommended way to supply it?
A: Provide a list, e.g. `unique_key: ['fund_id', 'share_class_id']`, or build one deterministic key with `dbt_utils.generate_surrogate_key([...])` and pass that single column. A reliably-unique key is essential because the snapshot matches source rows to existing versions by it; a non-unique key corrupts the SCD2 history. Hashing the parts guarantees a stable, collision-resistant single key across runs.

Q: How do you snapshot a source that has no reliable `updated_at` column?
A: Use the `check` strategy with an explicit `check_cols` list of the business-meaningful columns to version, e.g. `check_cols: ['management_fee', 'distribution_policy']`. dbt compares those columns value-by-value each run and opens a new version when any differ, so you do not need a trustworthy timestamp. Avoid `check_cols: 'all'` on wide tables because incidental changes create noisy versions.

Q: A `relationships` test from `fct_holdings.fund_id` to `dim_fund.fund_id` fails. What are the two most likely root causes?
A: Either the fact contains a `fund_id` that genuinely has no matching dimension row (an orphaned holding from a broken or out-of-order load), or `dim_fund` was not rebuilt/loaded before the test ran so legitimate keys appear missing. Investigate by selecting `fund_id`s in the fact that are absent from the dimension. The fix is correcting the load order or sourcing the missing fund record, not deleting the test.

Q: How do you exclude one expensive model from a full build while still building everything else?
A: Run `dbt build --exclude fct_heavy_reconciliation`, which builds the entire project minus that node (and you can append a selector to also drop its downstream with `fct_heavy_reconciliation+`). This is how you skip a slow nightly model during an ad-hoc rebuild. `--exclude` is applied after `--select`, so it composes with a scoped selection too.

Q: What does setting `+enabled: false` on a model in `dbt_project.yml` do, versus deleting the file?
A: `+enabled: false` disables the model so dbt ignores it during parse, run, and DAG resolution without removing the SQL file, and any `ref` to it will error as a missing node. It is a clean way to temporarily retire a model — e.g. a deprecated EPT variant — while keeping its code in version control. Deleting the file is permanent and loses the history of the logic.

Q: How do you control the database schema a dbt model is built into, and what is the default naming behavior?
A: By default dbt builds into the target schema from `profiles.yml`, and a custom `+schema:` config is APPENDED to that target schema (so `+schema: marts` in dev schema `dbt_seniorose` becomes `dbt_seniorose_marts`) via the `generate_schema_name` macro. You override that concatenation by redefining `generate_schema_name` if you want literal schema names. Knowing this prevents surprise at why your prod schema name has a prefix.

Q: What is the purpose of a `--vars` override on the command line, and give a NAV example.
A: `--vars '{nav_cutoff_date: "2024-12-31"}'` injects a project variable at run time, read in models via `{{ var('nav_cutoff_date') }}`, letting one build be parameterized without code changes. You might use it to reprocess NAV up to a specific reporting date, or to toggle a backfill window. CLI `--vars` overrides any default set in `dbt_project.yml`.

Q: How do you provide a default for a `var()` so a model does not break when the variable is unset?
A: Pass a second argument: `{{ var('lookback_days', 3) }}` returns `3` when `lookback_days` is not defined anywhere. Without a default, referencing an undefined var raises a compilation error. Defaults make a model robust to being run without explicit `--vars`, which matters for scheduled jobs that omit them.

Q: Why might `dbt build` succeed but `dbt docs generate` show a model with no description or columns?
A: Because documentation comes from YAML `description` keys and the catalog, not the SQL — a model with no properties file simply has nothing to display, even though it builds fine. The columns appear in `catalog.json` only after `dbt docs generate` introspects the warehouse, so the model must exist there too. The fix for the "readable by a stranger" criterion is adding model- and column-level `description`s in YAML.

Q: What does the `persist_docs` config do and why is it useful in a regulated warehouse?
A: `persist_docs: {relation: true, columns: true}` pushes your dbt `description`s into the warehouse's native table and column COMMENT metadata, so the documentation is visible to anyone querying the database directly, not just in the dbt docs site. In a regulated estate this means data dictionaries live with the data and survive outside the dbt tooling. It keeps the warehouse self-describing for auditors and BI users alike.

Q: How do you tell dbt to treat a warning-severity test as an error in CI only?
A: Set `severity: warn` plus `error_if` thresholds, or use `--warn-error` (or `--warn-error-options`) on the CI invocation so warnings are promoted to failures. `dbt build --warn-error` makes the CI run fail on anything that would only warn locally, enforcing a stricter gate before merge. This lets developers iterate with soft warnings while CI stays strict.

Q: What is the difference between a model `description` and a doc block referenced with `{{ doc(...) }}`?
A: A `description` is inline YAML text for one model or column, while a doc block is reusable Markdown defined in a `.md` file inside `{% docs block_name %}...{% enddocs %}` and pulled in via `description: "{{ doc('block_name') }}"`. Doc blocks let you write a long explanation once (e.g. the definition of NAV) and reference it from many columns. They keep documentation DRY and consistent across the project.

Q: An `accepted_values` test on a `currency` column fails after onboarding a new fund. What happened and what is the correct fix?
A: A legitimately new but unlisted value (e.g. a new share class denominated in `CHF`) appeared, and since `accepted_values` asserts membership in a fixed set, the unexpected code fails the test. The correct fix is to add the new currency to the test's `values` list once you confirm it is valid, not to drop the test. This is the test doing its job — forcing a human to confirm a domain change.

Q: How do you make an incremental model absorb a newly-added source column without a manual full refresh?
A: Set `on_schema_change: 'append_new_columns'` (or `'sync_all_columns'`) in the model config, so when the source gains a column dbt alters the target table to include it on the next incremental run. With the default `'ignore'`, the new column is silently dropped from the incremental insert. `sync_all_columns` also handles removed/type-changed columns, at the cost of more aggressive table alterations.

Q: What is the danger of `incremental_strategy: append` for a corrected NAV that is re-sent by the source?
A: `append` blindly inserts the incoming rows with no deduplication, so a re-sent corrected NAV for an existing `(fund_id, nav_date)` produces a duplicate row rather than overwriting the old value. For mutable facts you need `merge` (or `delete+insert`) with a `unique_key` so the correction updates in place. `append` is only safe for truly immutable, never-re-sent event data.

Q: How would you reconcile share-class NAVs against fund-total NAV as a dbt test?
A: Write a singular test in `tests/` selecting fund/date groups where `abs(sum(share_class_assets) - fund_total_net_assets) > tolerance`, returning the breaking rows so `dbt build` fails on a reconciliation gap. The tolerance accounts for rounding. This encodes a cross-model business invariant that no generic test expresses, exactly the niche singular tests fill.

Q: What does `dbt seed --full-refresh` do differently from a plain `dbt seed`?
A: `dbt seed` by default appends/upserts or, depending on adapter, rebuilds the seed table, but `--full-refresh` forces it to drop and recreate the table from the CSV, which you need after changing the seed's columns or types. Without it a structural change to the CSV can clash with the existing table schema. Seeds are for small static reference data, so a full refresh is cheap here.

Q: Why should you avoid loading a large CSV with `dbt seed`, and what is the alternative?
A: `dbt seed` parses the CSV through dbt and issues row-by-row or batched inserts, which is slow and memory-heavy for anything beyond small lookups (country codes, fund-type maps). For bulk data you use a proper loader/ingestion tool into a source table, then `ref` it — dbt's "T", not "EL", responsibility. Seeds are explicitly for small static reference data only.

Q: How do you find which models a given column-level change could break before merging?
A: Run `dbt ls --select +the_changed_model+` to list all ancestors and descendants, and in dbt Cloud or SQLMesh use column-level lineage to narrow it to the exact downstream columns derived from the changed one. dbt Core gives model-level blast radius; column-level requires the Cloud/SQLMesh capability. This impact analysis is essential before touching a model feeding a regulatory EMT field.

Q: What problem does SQLMesh's automatic backfill detection solve that dbt leaves to you?
A: When you change an incremental model's logic, SQLMesh's `plan` detects the affected time intervals and proposes the precise backfill needed, whereas in dbt you must reason about whether to `--full-refresh` and over what range yourself. SQLMesh tracks intervals/state per model, so it knows exactly which dates are stale. This removes a common source of silent incremental drift in dbt.

Q: How does SQLMesh classify a change as "breaking" vs "non-breaking", and why does it matter?
A: During `plan`, SQLMesh parses the SQL with SQLGlot to determine whether a change alters downstream results (breaking — requires backfilling dependents) or is cosmetic/additive (non-breaking — dependents can be reused without rebuild). This classification means non-breaking edits skip needless backfills, saving compute. dbt has no equivalent pre-execution impact classification.

Q: A teammate ran `dbt run` instead of `dbt build` and bad data reached the mart. Explain precisely why.
A: `dbt run` builds models but does NOT execute tests, so the upstream data-quality assertions that would have failed and skipped downstream models never ran, letting a bad NAV flow into the mart. `dbt build` interleaves tests at each node and stops descendants when a test fails. The fix is to standardize on `dbt build` (or `dbt test` after `run`) so the quality gate is never bypassed.

Q: What is the difference between `manifest.json` and `run_results.json`, and which powers `result:` selectors?
A: `manifest.json` is the static compiled graph (every node, config, dependency, compiled SQL) produced at parse time, while `run_results.json` records the OUTCOME of the last invocation (each node's status, timing, rows affected). The `result:` selectors read `run_results.json` to re-select errored/failed/warned nodes, whereas `state:modified` compares against a prior `manifest.json`. They serve different selection needs — state-based vs outcome-based.

Q: For a daily NAV `incremental` model with occasional late corrections, what is a robust config combination?
A: Use `materialized: incremental`, `incremental_strategy: merge`, `unique_key: ['fund_id', 'nav_date']`, an `is_incremental()` filter with a lookback window (e.g. `nav_date >= dateadd(day, -3, (select max(nav_date) from {{ this }}))`), and `on_schema_change: 'append_new_columns'`. The lookback re-pulls recent dates so late corrections are caught, the `merge` and `unique_key` overwrite rather than duplicate them, and the schema config absorbs new columns. A `unique_combination_of_columns` test then guards the grain.

Q: What does the `--defer` flag actually resolve, and why is it useless without `--state`?
A: `--defer` tells dbt that any `ref` to a model NOT selected in the current run should resolve to that model's relation in a different (usually production) environment, read from the manifest at the `--state` path. Without `--state` there is no prior manifest to point those unselected refs at, so dbt has nothing to defer to and the flag is inert. It is what lets slim CI build only changed models while their unchanged upstreams resolve to prod tables.

Q: How do you spot a fan-out join before it breaks a `unique` test, during development?
A: Run `dbt show --select the_model --limit 0` is not enough; instead query the compiled model grouped by its intended grain with `having count(*) > 1`, or add the `dbt_utils.unique_combination_of_columns` test and run `dbt build` to surface duplicates immediately. Catching it early means checking the join's many-side cardinality — if the joined table is not unique on the join key, the result multiplies. Aggregating the many-side to the right grain in an intermediate model is the structural fix.
