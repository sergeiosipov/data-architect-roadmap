# Phase 5 · 4.2.2 Cloud-native streaming (Event Hubs, Stream Analytics) — 150 self-test questions

Q: At the highest level, what is this lesson asking the architect to be able to do?
A: It asks you to give a feature-by-feature verdict on whether Azure Event Hubs can replace open-source Kafka for a given workload, rather than a vibe-based yes or no. You must know the Event Hubs model (namespaces, throughput/processing units, partitions, Capture), where its Kafka-protocol surface stops short of real Kafka, and when Azure Stream Analytics SQL fits versus Apache Flink. The payoff is that in a fund platform streaming order/trade events you never quietly pick Event Hubs where you needed log compaction or full transactional semantics.

Q: What is Azure Event Hubs in one sentence?
A: Azure Event Hubs is a fully managed, cloud-native event-streaming platform (PaaS) that ingests millions of events per second with low latency and exposes Apache Kafka, AMQP 1.0, and HTTPS protocols on the same service. It is a partitioned, append-only log built for high-throughput ingestion with time-based retention. Because it is managed, there are no brokers, disks, or clusters for you to provision or patch.

Q: What is the phrase "Kafka on Azure" usually shorthand for, and why does the nuance matter?
A: In practice "Kafka on Azure" usually means Event Hubs with its Kafka-protocol endpoint, not a self-run Apache Kafka cluster on VMs or AKS. The nuance matters because Event Hubs implements the Kafka wire protocol but is a different engine underneath, so a handful of Kafka features behave differently or are absent. An architect who treats the two as identical can sign off a design that silently lacks log compaction or specific transactional guarantees.

Q: What is an Event Hubs namespace?
A: A namespace is the top-level management container that holds one or more event hubs and exposes a single fully qualified domain name like `NAMESPACE.servicebus.windows.net`. It is where you set streaming capacity (throughput or processing units), network security, and geo-disaster-recovery configuration. In Kafka terms the namespace maps to a Kafka cluster, and each event hub inside it maps to a Kafka topic.

Q: How do Kafka concepts map onto Event Hubs concepts?
A: A Kafka cluster maps to an Event Hubs namespace, a Kafka topic maps to an event hub, a Kafka partition maps to an Event Hubs partition, a Kafka consumer group maps to an Event Hubs consumer group, and a Kafka offset maps to an Event Hubs offset. This near one-to-one mapping is why existing Kafka clients can often switch with only a configuration change. The conceptual gaps appear in features like compaction and admin operations rather than in the core log model.

Q: What is a partition in Event Hubs and why does it exist?
A: A partition is an ordered, append-only commit log within an event hub; new events are added to the end of the sequence. Partitions exist to multiply throughput (parallel logs spread the IO ceiling) and to give your consumers parallel processing lanes with a clear ownership boundary. Ordering is guaranteed only within a single partition, never across partitions.

Q: What is a throughput unit (TU) and exactly how much capacity does one provide?
A: A throughput unit is a prepurchased unit of capacity on the Standard tier; one TU provides up to 1 MB/s or 1,000 events/s ingress (whichever comes first) and up to 2 MB/s or 4,096 events/s egress. You pay for TUs by the hour with a one-hour minimum. All event hubs and partitions in the namespace share the namespace's TU pool.

Q: What happens if you exceed your purchased throughput units on ingress?
A: Ingress is throttled and Event Hubs raises an `EventHubsException` with a `Reason` value of `ServiceBusy` (the older Kafka/AMQP surface calls this a server-busy / `ServerBusy` condition). Egress does not throw a throttling exception, but it still cannot exceed the egress ceiling of your purchased TUs. The fix is to add TUs, enable auto-inflate, or reduce the publish rate.

Q: What is the maximum number of throughput units for a Standard-tier namespace, and what is auto-inflate?
A: You can purchase up to 40 throughput units for a single Event Hubs namespace, shared across all event hubs in it. Auto-inflate automatically scales the TU count up when load rises beyond the configured minimum, preventing `ServerBusy` throttling without failing requests. Note that auto-inflate scales up automatically but does not scale back down on its own.

Q: What is a processing unit (PU) and which tier uses it?
A: A processing unit is the capacity model for Event Hubs Premium; it is a CPU-and-memory-isolated resource container in a managed multitenant environment. You can buy 1, 2, 4, 6, 8, 10, 12, or 16 PUs per Premium namespace. As a rough figure, one PU with a 100-partition event hub offers roughly 5-10 MB/s ingress and 10-20 MB/s egress for AMQP or Kafka workloads.

Q: What is a capacity unit (CU) and which tier uses it?
A: A capacity unit is the scaling unit for the Dedicated tier, which provisions a single-tenant cluster with reserved resources. Dedicated suits very high, predictable throughput and supports the largest message size. You scale a Dedicated cluster by adding CUs, independent of the partition count of any event hub inside it.

Q: Does the number of partitions affect Event Hubs pricing?
A: No. Pricing depends on scaling units — TUs (Standard), PUs (Premium), or CUs (Dedicated) — not on partition count. A Standard event hub with 32 partitions and one with a single partition cost the same at one TU. You scale TUs, PUs, or CUs independently of partitions.

Q: How is the partition count for an event hub set, and can it be changed later?
A: Partition count is chosen at creation and must be between one and the per-tier maximum. On Standard (and lower) tiers it is fixed for the life of the event hub and cannot be changed. On Premium and Dedicated tiers you can increase the partition count after creation, but you can never decrease it.

Q: Why is increasing partition count on a live event hub risky for ordering?
A: Partition assignment uses a static hash of the partition key, so changing the partition count remaps which key lands on which partition. Events that previously shared a partition (and thus a guaranteed order) may scatter to different partitions after the change. If relative order matters in your application — for example trade-event sequencing — you should avoid changing partition count rather than rely on it.

Q: How does Event Hubs decide which partition an event goes to?
A: If the producer supplies a partition key, Event Hubs runs it through a static hash function to pick the partition, keeping all events with that key together and in arrival order. If no key is supplied, events are distributed round-robin across partitions. The publisher knows only its key, never the resulting partition number, which decouples senders from the partition layout.

Q: What makes a good partition key in a streaming fund platform?
A: A key that groups events whose relative order you must preserve and that spreads load evenly, such as a per-instrument `ISIN`, an order id, or a per-account identifier. Using `ISIN` as the key keeps all events for one security in order within a single partition, which matters when you reconstruct a position or NAV-affecting trade sequence. Avoid keys with severe skew (for example a single hot fund) that would overload one partition.

Q: What does "the number of partitions equals the maximum number of parallel consumers per consumer group" mean?
A: Within one consumer group each partition is owned by at most one active consumer at a time, so you cannot usefully run more concurrent consumers than there are partitions — extra consumers sit idle. This holds for both AMQP epoch consumers and Kafka consumer-group members. So you must size partition count for your peak consumer parallelism, not just for ingress throughput.

Q: What is a consumer group in Event Hubs?
A: A consumer group is an independent logical view of the entire event hub that lets multiple applications read the same stream in parallel, each tracking its own offsets. It is the mechanism for fan-out: a real-time pricing app and an archival app can each have their own consumer group over the same trade stream without interfering. Within a group, partition ownership is still one-consumer-per-partition.

Q: What is checkpointing in the Event Hubs consumer model?
A: Checkpointing is the consumer storing its last successfully processed offset (typically in a durable store like Azure Blob Storage for the AMQP `EventProcessorClient`) so it can resume from there after a restart or rebalance. Because Event Hubs guarantees at-least-once delivery, consumers may see an event again after a failure, so processing should be idempotent. Checkpoint frequency trades off reprocessing volume against checkpoint write cost.

Q: What is an epoch (exclusive) consumer and what error appears when ownership is stolen?
A: An epoch consumer claims exclusive ownership of a partition with an owner level; the `EventProcessorClient` (.NET, Java) and `EventHubConsumerClient` (Python, JavaScript) use this pattern. If a new consumer connects with a higher owner level, the service disconnects the current owner with a `ConsumerDisconnected` error and hands the partition to the newcomer. This is the recommended production pattern for AMQP workloads.

Q: How many non-epoch receivers can read one partition at once, and what is the catch?
A: Up to 5 non-epoch receivers can read the same partition concurrently within a consumer group, each seeing the same events (fan-out). Because every receiver gets the full event stream, this does not increase processing throughput per partition — it duplicates work. Connecting an epoch consumer to that partition disconnects all non-epoch consumers on it.

Q: What is Event Hubs Capture?
A: Capture automatically writes the incoming event stream in near real time to Azure Blob Storage or Azure Data Lake Storage (ADLS), typically as Avro files, for long-term archival and batch analytics. It runs on the same stream used for real-time processing, so you get cheap archival without a separate consumer. This is how a fund platform retains raw order/trade events for regulatory replay while still processing them live.

Q: In what format does Event Hubs Capture write files, and why does that matter for the lake?
A: Capture writes Avro files (a self-describing, schema-carrying binary format) into your blob or ADLS container, partitioned by a configurable folder path including namespace, hub, partition, and time. The Avro format means downstream lake engines can read the events with their schema intact. For a lakehouse you often convert these raw Avro captures to Parquet or Delta in a bronze-to-silver step.

Q: Does Capture replace or run alongside real-time consumers?
A: It runs alongside them. Capture is an additional sink that mirrors the stream to storage; it does not consume from or compete with your live consumer groups. So you can simultaneously process trade events for real-time NAV impact and capture the raw stream to ADLS for audit and reprocessing.

Q: What retention periods does Event Hubs support by tier?
A: Standard tier retains data for 1 to 7 days; Premium and Dedicated tiers retain for 1 to 90 days. Retention is time-based — events age out after the window regardless of consumption. For retention beyond the tier maximum you use Capture to land data in ADLS or Blob storage.

Q: What is the maximum message size by tier?
A: Standard and Premium support events up to 1 MB; Dedicated supports up to 20 MB (large-message support, which has been in preview). Most streaming events should stay well under 1 MB. If your fund messages routinely exceed 1 MB, you typically store the payload in blob storage and stream a reference (claim-check pattern) rather than the full document.

Q: Which Event Hubs tiers expose the Kafka protocol endpoint?
A: The Kafka endpoint is available on the Standard, Premium, and Dedicated tiers. For full Kafka protocol compatibility Microsoft recommends Premium or Dedicated, because the Standard tier has limitations that can affect some Kafka workloads. The Basic tier does not offer the Kafka surface in the same way.

Q: What network endpoint and port do Kafka clients use to reach Event Hubs?
A: Clients point `bootstrap.servers` at `NAMESPACE.servicebus.windows.net:9093`. Port `9093` is the TLS-secured Kafka endpoint; Event Hubs requires TLS, so `security.protocol` is `SASL_SSL`. Because Event Hubs presents a single stable virtual IP, you do not open firewall rules per broker as you would for a real Kafka cluster.

Q: Which Kafka client versions does Event Hubs support?
A: Event Hubs for Apache Kafka supports Apache Kafka version 1.0 and later client protocols. Most existing Kafka producer and consumer applications connect with only a configuration change, no code change. Very old (pre-1.0) clients are not supported.

Q: What two SASL mechanisms can a Kafka client use against Event Hubs, and which is preferred?
A: Over `SASL_SSL`, clients can use `OAUTHBEARER` (OAuth 2.0 via Microsoft Entra ID) or `PLAIN` (Shared Access Signature connection string). OAuth 2.0 with Entra ID and Azure RBAC is preferred because it gives fine-grained, role-based, centrally managed access and avoids hand-managed ACLs and shared keys. `PLAIN` uses `$ConnectionString` as the username and the namespace connection string as the password.

Q: In a `PLAIN` SASL config, what goes in the username and password fields?
A: The username is the literal `$ConnectionString` and the password is your full Event Hubs namespace connection string, for example `Endpoint=sb://mynamespace.servicebus.windows.net/;SharedAccessKeyName=...;SharedAccessKey=...`. This is set in `sasl.jaas.config` with the `PlainLoginModule`. It is a convention specific to the Event Hubs Kafka endpoint.

Q: A Kafka producer is rejected at the Event Hubs endpoint with an authentication failure — what are the first things to check?
A: Confirm `security.protocol=SASL_SSL` (not `PLAINTEXT`), `bootstrap.servers` ends in `:9093`, and the SASL mechanism matches the credential type (`PLAIN` with `$ConnectionString` and a valid connection string, or `OAUTHBEARER` with a working token callback). Also verify the connection string has not been scoped to the wrong entity or had its key rotated. Note that generated SAS *tokens* are not supported on the Kafka endpoint — use the connection string form.

Q: The single biggest Kafka feature gap in Event Hubs is what?
A: Log compaction. Event Hubs is a time-retention partitioned log and does not implement Kafka's compacted-topic semantics, where the broker retains only the latest value per key indefinitely. If your design depends on a compacted changelog (for example a keyed "current state" topic), Event Hubs cannot provide it natively.

Q: Why does the absence of log compaction matter for a stateful changelog use case?
A: Compacted topics are how Kafka stores the latest-known value per key forever (think a key-value snapshot rebuilt by replaying the topic). Without compaction, Event Hubs only keeps events for the retention window and then drops them, so you cannot rebuild full current state by replaying the hub after data ages out. The workaround is to materialize current state externally — for example into a database, Azure Data Explorer, or a Delta table — and treat Event Hubs as the transport, not the store of record.

Q: How would you work around the lack of log compaction for a "latest NAV per ISIN" feed?
A: Stream the NAV events through Event Hubs as transport and continuously upsert the latest value per `ISIN` into a durable store — a SQL table, Cosmos DB, or a Delta table keyed by `ISIN`. That external store plays the role Kafka's compacted topic would, holding the current value indefinitely. Use Capture to ADLS if you also need the full historical event log for audit and replay.

Q: Does Event Hubs support Kafka transactions, and with what caveat?
A: Yes — the Event Hubs Kafka endpoint supports Kafka transactions, but as of the current documentation this is in public preview and available on the Premium and Dedicated tiers. So you cannot assume the same generally-available, cross-partition transactional guarantees as a mature self-run Kafka cluster on every tier. For a design that hinges on exactly-once transactional writes, verify the current preview/GA status and tier before committing.

Q: Does Event Hubs support idempotent producers?
A: Yes. Event Hubs for Apache Kafka supports idempotent producers (and idempotent consumers), which deduplicate producer retries so a message is not written twice on a retry. This is separate from transactions, which span multiple writes atomically. Even so, the broader service contract is at-least-once delivery, so consumers should still be idempotent.

Q: What compression types does the Event Hubs Kafka endpoint support?
A: On the broker side Event Hubs supports `gzip` (and `none`); `gzip` compression is supported in the Premium and Dedicated tiers. Clients configured for `snappy`, `lz4`, or `zstd` are accepted, but the documented guidance is to use `gzip` or `none`. So a Kafka producer relying on `zstd` for bandwidth savings cannot count on that exact codec being preserved.

Q: Is Kafka Streams usable against Event Hubs?
A: The Kafka Streams client library is supported, but as of current docs it is in public preview on the Premium and Dedicated tiers. Customers often want it as a path to Confluent's ksqlDB, but ksqlDB's license forbids competing managed services from offering it, so you cannot get ksqlDB on Event Hubs. Microsoft positions Stream Analytics, Flink, Databricks, or Spark as the stream-processing layer instead.

Q: Why can you not run Confluent ksqlDB on Event Hubs?
A: ksqlDB is a shared-source product whose license prohibits any vendor offering competing SaaS/PaaS/IaaS from using or offering ksqlDB support. Event Hubs is exactly such a competing managed service, so it cannot provide ksqlDB. If you require ksqlDB you must run Kafka yourself or use Confluent Cloud; otherwise use Stream Analytics or Flink for streaming SQL.

Q: Does Event Hubs support the Kafka admin/management APIs for changing broker or topic configuration?
A: No, not in the way a self-managed Kafka cluster does. Event Hubs is a managed service with no brokers to configure, so broker-level and many topic-level config operations (log.message.format, replication factor, arbitrary topic configs) are not available — partitions and retention are governed by the Event Hubs model and Azure control plane. You create event hubs through Azure (portal, ARM/Bicep, CLI), not by arbitrary Kafka admin commands.

Q: Can a Kafka producer write to Event Hubs and an AMQP or Stream Analytics consumer read it?
A: Yes. Event Hubs runs Kafka, AMQP 1.0, and HTTPS concurrently on the same hub, so you can produce with a Kafka client and consume with the native AMQP SDK, Stream Analytics, or Azure Functions. This protocol interop is a genuine advantage over a pure Kafka cluster. It lets you migrate producers gradually while immediately using Azure-native consumers.

Q: How do you migrate historical data from an existing Kafka cluster into Event Hubs?
A: Use Kafka MirrorMaker pointed at the Event Hubs Kafka endpoint to replicate topics; it can run as a one-time backfill or continuously during a phased migration. You create one event hub per Kafka topic and update client `bootstrap.servers`. For a cutover you typically run both systems in parallel, validate throughput and consumer lag, then decommission Kafka.

Q: When Kafka consumers first connect to Event Hubs with no prior offsets, what controls where they start?
A: The `auto.offset.reset` client setting: `earliest` starts from the beginning of available data, `latest` reads only new messages from connection time. Because freshly migrated consumers have no committed offsets in Event Hubs, this setting decides whether they reprocess history or skip it. Choosing wrong can cause either a flood of reprocessed events or silent data loss for the gap.

Q: A migrated Kafka consumer using `snappy` compression now sees errors or unexpected behavior — what is the likely cause?
A: Event Hubs supports `gzip` (and `none`) on the broker side, so a workload assuming `snappy`/`lz4`/`zstd` end-to-end may not behave as on native Kafka. Reconfigure producers to `gzip` or `none` per the migration guidance. Also confirm you are on a tier that supports compression at all (Premium/Dedicated for gzip).

Q: Why is Event Hubs' single stable endpoint an operational advantage over a Kafka cluster?
A: Event Hubs presents one stable virtual IP and FQDN, so all traffic for all partitions routes through that single endpoint and clients never need broker discovery or per-broker firewall rules. A self-run Kafka cluster requires reachable host/port for every broker, complicating network and firewall design. This simplifies private networking, Private Link, and IP filtering in a regulated fund environment.

Q: What is Azure Schema Registry and how does it relate to Event Hubs Kafka use?
A: Azure Schema Registry is a centralized repository in the Event Hubs namespace for managing Avro and JSON schemas, enforcing compatibility and supporting schema evolution across producers and consumers. Kafka applications can use it the way they would a Confluent Schema Registry. For a fund feed, it pins the agreed shape of a trade or NAV event so producers cannot silently break consumers.

Q: Summarize the "can we just use Event Hubs instead of Kafka?" verdict at a high level.
A: For ordinary high-throughput produce/consume, consumer groups, Capture-to-lake, and Azure-native integration, yes — Event Hubs is an excellent managed replacement. The hard "no or verify first" cases are log compaction (unsupported), and transactions / Kafka Streams (supported but preview and tier-limited), plus broker-level admin control and non-gzip compression. The architect's job is to enumerate the exact Kafka features the workload uses and check each against that list.

Q: What is Azure Stream Analytics in one sentence?
A: Azure Stream Analytics is a fully managed (PaaS) stream-processing engine that runs SQL-like queries with temporal extensions over streaming inputs at sub-millisecond latencies. You wire inputs (like Event Hubs or IoT Hub), write a query, and route results to outputs (Blob, ADLS, SQL, Cosmos DB, Event Hubs, Power BI). It targets real-time analytics, anomaly detection, and continuous transforms without you managing infrastructure.

Q: What engine is Azure Stream Analytics built on?
A: It is built on Trill, a high-performance in-memory streaming analytics engine developed with Microsoft Research. This in-memory design is why all processing happens without persisting the incoming data and why latencies are very low. Trill underpins the SQL-temporal model the service exposes.

Q: What kind of query language does Stream Analytics use?
A: A SQL query language augmented with temporal constraints for data in motion, supporting selection, aggregation, joins, geospatial functions, pattern matching, and anomaly detection (complex event processing). It feels like T-SQL but adds windowing, `TIMESTAMP BY`, and temporal joins. You can extend it with JavaScript or C# user-defined functions and aggregates, or call Azure Machine Learning.

Q: What are typical inputs and outputs for a Stream Analytics job?
A: Inputs include Event Hubs and IoT Hub for streaming data, plus Blob/ADLS Gen2 for historical or reference data, and reference data from Blob or SQL Database for lookups. Outputs include Blob, ADLS, Azure SQL Database, Cosmos DB, Event Hubs, Synapse/HDInsight for batch, and Power BI for live dashboards. This makes a job a self-contained ingest-transform-deliver pipeline.

Q: What is reference data in Stream Analytics and why is it useful for fund data?
A: Reference data is static or slow-changing lookup data (from Blob or SQL Database) that you join against the live stream to enrich events. For a trade stream you might join on `ISIN` to attach instrument metadata, fund identifiers, or an `LEI` from a reference table. It lets you decorate fast-moving events without calling out to an external service per event.

Q: What is a streaming unit (SU) in Stream Analytics?
A: A streaming unit is the unit of compute (CPU and memory) you allocate to a job; you pay only for the SUs you consume with no upfront commitment or cluster to provision. More SUs give more parallel processing capacity for heavier queries. You scale SUs up or down based on load, and partitioned queries parallelize across streaming nodes.

Q: What delivery and processing guarantees does Stream Analytics provide?
A: Stream Analytics guarantees exactly-once event processing and at-least-once delivery, so events are never lost. End-to-end exactly-once *delivery* is guaranteed only with selected outputs that support it; with other sinks you can see duplicates on recovery. Built-in, encrypted checkpoints maintain job state for repeatable results.

Q: What is the availability SLA for Stream Analytics?
A: As a managed service, Stream Analytics guarantees 99.9% availability at a minute-level granularity. In availability-zone-enabled regions it automatically spreads job resources across zones at no extra cost or config for zone-failure resilience. This is the figure to cite when an architect documents the streaming tier's reliability.

Q: What are the five window types in Stream Analytics?
A: Tumbling, hopping, sliding, session, and snapshot windows. All windows must appear in a `GROUP BY` clause and each emits its output event at the window's end. The maximum size for any window is 7 days.

Q: What is a tumbling window?
A: A tumbling window segments the timeline into fixed-length, non-overlapping, contiguous intervals; every event belongs to exactly one window. For example a 5-minute tumbling window yields `(12:00-12:05]`, `(12:05-12:10]`, and so on, each aligned to the zeroth hour. It is the default choice for periodic aggregates like "trade count per 5 minutes."

Q: How does a hopping window differ from a tumbling window?
A: A hopping window has a fixed size but advances by a separate hop interval, so consecutive windows can overlap and an event can fall into multiple windows. A tumbling window is the special case where the hop equals the window size (no overlap). You use hopping windows for moving aggregates, like a 10-minute average computed every 1 minute.

Q: What is a sliding window in Stream Analytics?
A: A sliding window produces output only at the moments when events enter or leave the window, rather than on a fixed schedule, so each output reflects a content change over the trailing window length. It is event-driven, which makes it efficient for "in the last N minutes" conditions that change with arrivals. Unlike tumbling/hopping it does not emit on empty intervals.

Q: What is a session window?
A: A session window groups events that arrive close together in time, closing the window after a configurable gap of inactivity (and optionally a maximum duration). Window length is not fixed — it expands while events keep arriving and ends when the gap timeout elapses. It models bursts of activity, such as a flurry of order amendments around a single client session.

Q: What is a snapshot window?
A: A snapshot window groups events that share the exact same timestamp, so you aggregate over all events at a given instant. Unlike the others you do not specify a size; you get it by using `System.Timestamp()` in `GROUP BY`. It answers "for each distinct event time, aggregate the events at that time."

Q: What does it mean that "every window operation outputs an event at the end of the window"?
A: A windowed aggregate emits a single result event when the window closes, timestamped at the window end, not continuously as events arrive. After windowing, the output event's timestamp differs from the original event time and is read via `System.Timestamp()`. This is why downstream latency includes the window length plus any late-arrival tolerance.

Q: What does `TIMESTAMP BY` do in a Stream Analytics query?
A: `TIMESTAMP BY` tells the engine to use a column in the payload as the event time instead of the default arrival time. This is essential when events can arrive out of order or delayed, because windowing and temporal joins should reason on when the event occurred, not when it landed. For trade events you would `TIMESTAMP BY` the exchange execution time.

Q: What is the difference between event time and arrival time?
A: Event time is when the event actually occurred (a field in the payload, selected via `TIMESTAMP BY`); arrival time is when Event Hubs/the job received it. They diverge when networks delay or reorder events. Using arrival time for windows can mis-bucket late trades, so financial pipelines almost always key on event time.

Q: How does Stream Analytics handle out-of-order and late-arriving events?
A: You configure event-ordering policies: a late-arrival tolerance window (how long to wait for stragglers before closing a window) and an out-of-order tolerance (how much reordering to absorb), plus an action (Adjust or Drop) for events outside those bounds. Larger tolerances improve correctness for delayed events at the cost of higher latency. This is the practical knob behind the watermark concept.

Q: What is a watermark in stream-processing terms?
A: A watermark is the engine's notion of "event time has progressed to here," used to decide when a window is complete and can emit. Events arriving with a timestamp behind the watermark are late and handled by the late-arrival policy. Stream Analytics manages watermark progression internally based on your `TIMESTAMP BY` and ordering tolerances.

Q: Can Stream Analytics run outside the Azure cloud?
A: Yes — it runs on the Azure IoT Edge runtime and on Azure Stack for ultra-low-latency edge analytics, using the same query language as the cloud. This enables hybrid architectures where some processing happens near the device and some in the cloud. The portability of the SQL is the point: one dialect, two deployment targets.

Q: Where does Stream Analytics' SQL-temporal model stop, motivating a move to Flink?
A: It is a declarative SQL service: great for windowed aggregates, temporal joins, pattern matching, and enrichment, but it is not a general programmable dataflow engine. Complex custom state machines, large keyed state, iterative algorithms, bespoke connectors, and fine-grained control over state backends and exactly-once across arbitrary sinks push you toward Apache Flink. When the transform exceeds what SQL plus a UDF can express cleanly, that is the crossover.

Q: What is Apache Flink and how does it contrast with Stream Analytics?
A: Apache Flink is an open-source distributed stream-processing framework offering low-level DataStream and Table/SQL APIs, rich managed keyed state, event-time processing, and exactly-once via checkpointing across many connectors. Compared with Stream Analytics it is more powerful and flexible but requires you to manage (or pay a managed-Flink service for) a cluster and to write more code. The trade is expressiveness and control versus the serverless simplicity of Stream Analytics.

Q: For a simple "5-minute average price per ISIN" transform, would you pick Stream Analytics or Flink, and why?
A: Stream Analytics, because it is a textbook tumbling-window aggregate keyed by `ISIN` that the SQL-temporal model expresses in a few lines, with no cluster to run. You get exactly-once processing and a serverless cost model paying only for SUs. Reaching for Flink here adds operational weight with no benefit.

Q: For a transform needing large keyed state, custom operators, and a bespoke sink, Stream Analytics or Flink?
A: Flink, because it exposes the DataStream API, managed RocksDB-backed keyed state, and custom source/sink connectors that Stream Analytics SQL cannot express. The SQL-temporal model does not give you arbitrary stateful operators or control over the state backend. The reason to accept Flink's operational cost is that the workload genuinely needs programmable state and connectors.

Q: How does Stream Analytics achieve parallelism, and what limits it?
A: It parallelizes partitioned queries across multiple streaming nodes, scaling with allocated SUs; an embarrassingly parallel job partitions input, query, and output on the same key. The ceiling is the input partition count and whether the query can be expressed as partition-local (a "fully parallel" topology). Cross-partition operations (non-partitioned aggregates) force a serializing step that limits scale.

Q: Why does the partition count of an Event Hub input affect a downstream Stream Analytics job?
A: The input partition count caps how parallel the job can be: a fully parallel Stream Analytics topology assigns one processing lane per input partition. Too few partitions throttles the job no matter how many SUs you add; mismatched partitioning forces repartition shuffles. So you design Event Hub partitioning with the downstream Stream Analytics (or Flink) parallelism in mind.

Q: Name the three Azure messaging services this lesson asks you to triage between.
A: Azure Event Hubs, Azure Service Bus, and Azure Event Grid. Each targets a different pattern: Event Hubs for high-throughput event streaming, Service Bus for enterprise messaging with delivery guarantees, and Event Grid for reactive event routing. The architect's triage skill is matching the workload's pattern to the right one rather than forcing everything through one service.

Q: What is the core pattern Event Hubs is built for?
A: Many producers emitting a high-throughput, time-ordered event stream consumed by multiple independent consumers — telemetry, logs, clickstream, and financial transaction ingestion. It is a partitioned log with consumer-controlled offsets, optimized for volume and replay within retention. Think "firehose to be analyzed," not "work queue to be drained."

Q: What is the core pattern Azure Service Bus is built for?
A: Enterprise messaging with strong delivery semantics: point-to-point queues and pub/sub topics, transactions, sessions, dead-lettering, scheduled and deferred messages, and the competing-consumer pattern. It is about reliable per-message processing and workflow control, not high-volume analytics streams. Use it where each message represents a unit of work that must be processed exactly and can be sidelined on failure.

Q: What is the core pattern Azure Event Grid is built for?
A: Reactive, push-based event routing: it delivers discrete events from sources to handlers with server-side filtering, triggering serverless functions, webhooks, or workflows. It is event-driven glue for "when X happens, react," not a durable high-throughput stream or a work queue. Think lightweight notifications and choreography across services.

Q: A fund platform must stream millions of order/trade events per second for real-time analytics — which service?
A: Event Hubs. It is purpose-built for high-throughput, time-ordered event streaming with partitioned parallelism, multiple consumer groups, and Capture-to-lake for audit. Service Bus would be the wrong tool (per-message queue semantics, far lower throughput) and Event Grid is for reactive routing, not a sustained firehose.

Q: A workflow must process each NAV-approval message exactly once, with retries and a dead-letter path for failures — which service?
A: Azure Service Bus. Its queues give competing-consumer processing, transactions, and dead-lettering so a failed NAV-approval message is sidelined rather than lost, and sessions can preserve per-fund ordering. Event Hubs offers no dead-letter queue or per-message lifecycle; Event Grid is push-routing, not durable work processing. This is the canonical "enterprise messaging, not streaming" case.

Q: "When a new EMT file lands in blob storage, trigger a validation function" — which service?
A: Azure Event Grid. The blob-created event is a discrete, reactive trigger that Event Grid routes (with filtering) to an Azure Function handler. Event Hubs is for streaming volumes of events, not single file-drop reactions, and Service Bus is for queued work units. This is the textbook reactive-routing pattern.

Q: Which features does Apache Kafka (and thus a streaming log) NOT provide that push you toward Service Bus or Event Grid?
A: Kafka/Event Hubs do not natively implement the competing-consumer queue pattern, server-rule-based pub/sub filtering beyond offsets, per-message job lifecycle tracking, or dead-lettering of poison messages. Those are foundational enterprise-messaging features, which is why Service Bus exists alongside Event Hubs. Event Grid covers the reactive push-with-filter routing that a log also does not do.

Q: Why is "default everything to Event Hubs because it scales" a design smell?
A: Because throughput is only one axis; if the workload needs per-message dead-lettering, transactions, sessions, or reactive filtered routing, a log is the wrong shape and you bolt on fragile workarounds. The triage discipline matches pattern to service: streaming to Event Hubs, reliable work units to Service Bus, reactive routing to Event Grid. Forcing all three through Event Hubs produces hidden complexity and missing guarantees.

Q: In a streaming order/trade pipeline, what concrete harm comes from picking Event Hubs where you needed log compaction?
A: You silently lose the ability to keep the latest value per key indefinitely, so any consumer that rebuilds current state by replaying the topic breaks once events age out of retention. A "current position per `ISIN`" rebuild would return incomplete state after the retention window. The corruption is quiet because everything works in testing with fresh data and only fails after retention elapses in production.

Q: How does Event Hubs Capture support regulatory replay and audit in a fund context?
A: Capture continuously archives the raw event stream to ADLS/Blob as Avro, giving you a durable, complete record beyond the short streaming retention. For DORA/UCITS-style audit you can replay or re-derive NAV-impacting computations from the captured raw events long after they left the live stream. It separates "fast retention for processing" from "long retention for compliance."

Q: How might `LEI`, `ISIN`, and EMT/EPT data flow through this streaming stack?
A: Trade/order events keyed by `ISIN` (with `LEI` for counterparties) stream into Event Hubs, a Stream Analytics or Flink job enriches them against reference data and computes windowed aggregates, and Capture lands the raw stream in ADLS for audit. EMT/EPT files are typically batch artifacts, but a file-drop can be turned into an event (via Event Grid) that triggers ingestion into the same lake. The streaming layer handles the high-volume flow; file-based regulatory artifacts ride the reactive/batch paths.

Q: What is the difference between Event Hubs at-least-once delivery and Stream Analytics exactly-once processing?
A: Event Hubs delivers events at least once, meaning a consumer may legitimately see the same event again after a failure or rebalance, so consumers must be idempotent. Stream Analytics adds exactly-once *processing* on top (state is not double-counted) and at-least-once delivery, with end-to-end exactly-once delivery only for selected outputs. So even reading from an at-least-once Event Hub, a Stream Analytics aggregate does not double-count within its state.

Q: A Stream Analytics job's output is missing recent events even though the input is flowing — first thing to check?
A: Check the event-ordering and late-arrival settings and your `TIMESTAMP BY` column: if windows are waiting for the late-arrival tolerance, results lag by that window plus tolerance, and events past tolerance may be dropped. Also confirm windows are emitting (output only fires at window end) and that watermark progression is not stalled by a partition with no incoming events. Misconfigured event time is the most common cause of "missing" late data.

Q: A Stream Analytics job won't scale past a point no matter how many SUs you add — likely cause?
A: The query is probably not fully parallel — a non-partitioned aggregate or a join across keys forces a serializing step — or the Event Hub input has too few partitions to feed more parallel lanes. SUs cannot exceed the parallelism the input partitioning and query topology allow. The fix is to partition input by the aggregation key and write a partition-local (embarrassingly parallel) query, or add input partitions.

Q: Your Kafka producer to Event Hubs intermittently gets `ServerBusy`/`ServiceBusy` errors under load — what is happening and how do you fix it?
A: You are exceeding the namespace's purchased throughput-unit ingress ceiling, so Event Hubs is throttling. Add throughput units, enable auto-inflate to scale TUs automatically, or implement backoff-and-retry in the producer. Also verify you have enough partitions to spread the ingress, since a single hot partition can bottleneck even within TU limits.

Q: You need to increase partitions on a live Standard-tier event hub — what do you discover?
A: You cannot: partition count is immutable after creation on Standard (and lower) tiers. Only Premium and Dedicated allow increasing (never decreasing) partition count after creation. The remedy on Standard is to create a new event hub with more partitions and migrate, accepting that the key-to-partition remapping changes ordering.

Q: A team wants a "topic of current account balances" they can replay to rebuild state — why is this a red flag on Event Hubs?
A: That is a compacted-changelog pattern, and Event Hubs has no log compaction, so the hub only holds events for the retention window and cannot serve as the indefinite latest-value-per-key store. Replaying it after retention elapses yields incomplete state. The architect should redirect "current state" to a database, Cosmos DB, Azure Data Explorer, or a Delta table, using Event Hubs only as transport.

Q: When does serverless Stream Analytics beat a self-managed Flink cluster on total cost of ownership?
A: When the workload is expressible in SQL-plus-temporal-windows, has bursty or moderate volume, and the team values no infrastructure to run — you pay only for SUs and skip cluster ops, patching, and Flink expertise. Flink wins when you need programmable state/operators or already operate a Flink platform at scale where its flexibility amortizes the operational cost. The deciding question is expressiveness needed versus operational burden accepted.

Q: How do you turn an EMT file drop into the streaming/event world cleanly?
A: Use Event Grid to emit a blob-created event when the file lands, routed to an Azure Function that validates the EMT file and emits a downstream event (which could go onto Event Hubs for fan-out). This keeps the reactive trigger (Event Grid) separate from the high-volume stream (Event Hubs) and the light glue (Functions). It is the canonical "file drop becomes an event" pattern across the three services.

Q: A Kafka consumer group has more members than the event hub has partitions — what happens?
A: The excess members receive no partition assignments and sit idle until a future rebalance frees a partition, because each partition is owned by exactly one member at a time. This mirrors native Kafka group behavior. To avoid wasted instances, size your consumer count to the partition count, and use `group.instance.id` (static membership) to reduce churn from transient disconnects.

Q: What is the Event Hubs emulator and why is it useful for a learner on a native Ubuntu laptop?
A: The Event Hubs emulator provides a local development experience so you can build and test code against an Event Hubs-like endpoint in isolation, free of cloud dependencies and Azure cost. It lets you iterate on producers, consumers, and Kafka-protocol config without provisioning a namespace, which suits offline practice on Ubuntu. It does not replicate every production quota or scale behavior, so still validate throughput and limits against the real service before relying on a design.

Q: Where does Event Hubs Capture write, and how is the path structured?
A: Capture writes to Azure Blob Storage or ADLS in a configurable folder path that includes the namespace, event hub, partition, and time components (year/month/day/hour/minute). This time-and-partition layout makes the captured Avro files easy to partition-prune in downstream lake queries. You set a time and/or size window that controls how often a new Capture file is rolled.

Q: Why convert raw Capture Avro to Parquet/Delta in the lake?
A: Avro is row-oriented and self-describing — ideal for streaming capture — but analytical queries over the gold layer favor columnar Parquet/Delta for predicate pushdown, partition pruning, and cheaper per-TB-scanned reads. So a bronze (raw Avro from Capture) to silver/gold (Parquet/Delta) conversion is standard. This is also where you key by date and `ISIN` to keep downstream scan costs low.

Q: How would you express "alert when more than N trades for one ISIN occur in any 1-minute window" in Stream Analytics?
A: Use a tumbling (or hopping for overlap) window of 1 minute, `GROUP BY ISIN, TumblingWindow(minute, 1)`, with a `HAVING COUNT(*) > N`, timestamping events with `TIMESTAMP BY` the execution time. The window emits one alert row per `ISIN` per minute that breaches the threshold. Hopping would let you slide the 1-minute window every few seconds for earlier detection at higher cost.

Q: Final synthesis: outline the architect's decision flow for the streaming tier of a fund platform.
A: First classify the pattern: high-volume time-ordered stream (Event Hubs), reliable per-message work with dead-lettering (Service Bus), or reactive filtered routing (Event Grid). For Event Hubs, enumerate every Kafka feature the workload uses and check each against the gaps — log compaction (no), transactions/Kafka Streams (preview, tier-limited), compression (gzip), partition immutability — and design workarounds. Then pick the processor: Stream Analytics for SQL-expressible windowed transforms with serverless economics, Flink when you need programmable state, custom operators, or connectors beyond SQL — and use Capture to ADLS for the audit/replay record throughout.

Q: Microsoft has since added a compacted event-hub option — what is the `Compact` cleanup policy and how does it change the old "no log compaction" verdict?
A: Event Hubs now supports key-based retention via a `Compact` cleanup policy, the alternative to the default `Delete` (time-based) policy, so the service does retain the latest event per key like a Kafka compacted topic. You set the policy at the event hub / Kafka topic level, it is unsupported on the Basic tier, and the partition key doubles as the compaction key. The architect's updated verdict is "compaction is available but is a per-hub mode you must explicitly enable and validate," not the blanket "unsupported" of older guidance.

Q: In a compacted Event Hubs hub, what plays the role of Kafka's compaction key, and what is a tombstone?
A: The event's partition key is used as the compaction key, so all events sharing a key are candidates for collapsing to the latest value. A tombstone is a delete marker the client creates by sending a new event with an existing key and a `null` payload, telling the compaction job to remove that key. Consumers can still see tombstoned events for the configured `Tombstone Retention Time (hours)` before they are purged.

Q: What are the "clean" and "dirty" portions of a compacted event hub's log?
A: At any instant a compacted partition log has a clean portion already processed by the compaction job (one latest event per key) and a dirty portion of newer events not yet compacted. Because the Event Hubs service triggers compaction non-deterministically and you cannot control its timing, the dirty portion can still hold multiple events with the same key between job runs. So a consumer reading the tail may transiently see several values for one key until compaction catches up.

Q: What ordering and offset guarantees does Event Hubs compaction preserve?
A: Compaction never reorders messages and never changes a message's sequence number or offset — it only discards older events of the same key. A consumer reading from the start of the log is guaranteed to see at least the final state of every key in write order. This is the same contract Kafka offers, which is what makes a compacted hub usable as a rebuildable keyed snapshot.

Q: What is the maximum size of a compacted event hub per partition by tier?
A: A compacted event hub is capped at `1 GB per partition` on Standard and `250 GB per partition` on Premium and Dedicated. This is a storage ceiling on the retained latest-value-per-key set, distinct from the time-based retention windows that govern non-compacted hubs. For a large keyspace like every active `ISIN`, size the tier and partition count so the compacted state fits under that per-partition cap.

Q: Given that compaction now exists, how would you re-answer "build a latest-NAV-per-ISIN topic we can replay" on Event Hubs?
A: Create the hub with the `Compact` cleanup policy keyed on `ISIN`, which keeps the latest NAV per `ISIN` indefinitely and lets a consumer rebuild current state by replaying from the start. Confirm you are off the Basic tier and that the keyspace fits the per-partition size cap, and remember compaction timing is non-deterministic so the tail may briefly show stale duplicates. The older "materialize into an external store" workaround is still valid but is now a choice rather than a forced one.

Q: Why is `Compact` a poor fit for an append-only audit log of every trade event?
A: Compaction deliberately discards all but the latest event per key, so an audit trail that must retain every individual trade would lose history the moment two events share a key. For full-fidelity audit you keep the default `Delete` policy plus Capture to ADLS, which preserves every event. Compaction is for "current state per key," not "complete event history."

Q: What is the maximum partition count for an event hub on Standard versus Dedicated, and why does the gap matter?
A: Standard (and Basic) cap at `32` partitions per event hub; Dedicated allows up to `1,024` per event hub (and `2,000` per CU). Premium allows `100` per event hub but `200` per PU at the namespace level. The gap matters because consumer and Stream Analytics parallelism is bounded by partition count, so an ultra-high-fan-out workload can outgrow Standard's 32-lane ceiling and need Premium/Dedicated.

Q: How many consumer groups can one event hub have on Standard versus Premium, and what is the Kafka-consumer-group namespace cap?
A: Standard allows `20` consumer groups per event hub, Premium `100`, and Dedicated `1,000`; Basic allows only `1`. Separately, the number of Kafka consumer groups per namespace is capped at `1,000` on Standard, Premium, and Dedicated. If a fund platform fans a trade stream out to many independent reader apps, count those groups against the per-hub limit before choosing a tier.

Q: What is the cap on concurrent receive requests on a single event hub, and what happens past it?
A: The limit is `5,000` concurrent receive requests across all consumers and consumer groups on the hub; subsequent receive requests are throttled. This is separate from the 5-non-epoch-receivers-per-partition rule. A design that spins up thousands of pollers against one hub can hit this ceiling even within throughput limits, so consolidate readers rather than over-fan-out.

Q: How many brokered connections does a namespace allow by tier, and why does it matter for many small producers?
A: Standard allows `5,000` brokered connections per namespace, Premium `10,000 per PU`, and Dedicated `100,000 per CU`. Each producer/consumer AMQP or Kafka connection counts, so a fleet of thousands of lightweight publishers can exhaust a Standard namespace's pool. The fix is connection pooling, batching publishers behind fewer connections, or moving to Premium/Dedicated for more headroom.

Q: How much retained event storage does one TU or PU or CU give you, and how does that interact with retention?
A: Standard/Basic give `84 GB per TU`, Premium `1 TB per PU`, and Dedicated `10 TB per CU` of event storage for retention. If your data within the retention window exceeds that storage allotment, older events can be trimmed before the time window elapses. So retention is bounded by both time and the storage your scaling units provide.

Q: A Kafka producer must publish a 5 MB trade-batch message — which Event Hubs tier can accept it and what is the alternative?
A: Only Dedicated accepts publications up to `20 MB`; Standard and Premium cap at `1 MB` and Basic at `256 KB`, and oversized publications are rejected outright. If you are not on Dedicated, use the claim-check pattern — store the 5 MB payload in blob storage and stream a small reference event. The publication limit applies whether you send one event or a batch, so batching does not raise the ceiling.

Q: Write the `az` CLI command shape to create a Standard event hub with 16 partitions and 3-day retention.
A: Use `az eventhubs eventhub create --resource-group myRg --namespace-name myNs --name trades --partition-count 16 --message-retention 3`. The `--partition-count` is fixed at creation on Standard (allowed range up to `32`) and `--message-retention` is in days (1-7 on Standard). You manage hubs through Azure CLI/portal/ARM because the Kafka admin API cannot create or configure them on Event Hubs.

Q: Which Azure RBAC roles grant send and receive on Event Hubs data, and why prefer them over connection strings?
A: Use `Azure Event Hubs Data Sender` for publish and `Azure Event Hubs Data Receiver` for consume, assigned to a Microsoft Entra ID identity, with `Azure Event Hubs Data Owner` for full data access. These give centrally managed, auditable, least-privilege access and avoid distributing long-lived SAS keys. For the Kafka endpoint this pairs with the `OAUTHBEARER` SASL mechanism rather than `PLAIN`.

Q: How do you grant a Kafka client send-only (not receive) access to one event hub on Event Hubs?
A: Scope an `Azure Event Hubs Data Sender` role assignment to that specific event hub resource (not the whole namespace), or create a SAS authorization rule with only the `Send` claim. RBAC scoping at the hub level enforces least privilege so a producer cannot read other hubs. Note a namespace allows at most `12` authorization rules, so SAS-rule-per-entity does not scale and RBAC is preferred.

Q: A consumer's checkpoints live in Blob Storage and the consumer restarts reprocessing thousands of old events — what likely went wrong?
A: Either checkpoints were not being written (or were written too infrequently), or the checkpoint blob container/connection was misconfigured so the `EventProcessorClient` found no stored offset and fell back to its default start position. Because Event Hubs is at-least-once, a consumer with no valid checkpoint resumes from the configured start (often `earliest`) and replays. Verify the checkpoint store is writable, that checkpoints commit after successful processing, and that processing is idempotent.

Q: In the Kafka consumer model on Event Hubs, what is the difference between `enable.auto.commit=true` and manual offset commits, and which is safer for a fund pipeline?
A: With `enable.auto.commit=true` the client periodically commits offsets on a timer regardless of whether processing finished, risking lost events if it crashes after commit but before processing. Manual commit (`enable.auto.commit=false`, then commit after the record is durably handled) gives at-least-once with no silent loss, at the cost of possible reprocessing. For NAV-affecting trade data you choose manual commit plus idempotent processing.

Q: A Kafka consumer keeps getting kicked out of the group and rebalancing under load — which timeouts do you tune?
A: Tune `max.poll.interval.ms` (raise it if processing a batch takes longer than the default and the broker assumes the consumer died), and `session.timeout.ms` / `heartbeat.interval.ms` for liveness detection. Frequent rebalances usually mean processing per poll exceeds `max.poll.interval.ms`, so the member is evicted mid-batch. Also consider `group.instance.id` static membership to avoid rebalances on transient restarts.

Q: What does `group.instance.id` (static membership) do and when is it valuable on Event Hubs?
A: Setting `group.instance.id` makes a consumer a static group member, so a brief disconnect or restart does not trigger a full group rebalance — the same partitions are re-assigned to the returning member within the session timeout. This reduces churn and reprocessing in deployments where pods restart frequently. It is valuable for stable, long-running fund-stream consumers where rebalance storms would otherwise cause lag spikes.

Q: A migrated Kafka producer using `acks=all` for durability — does that setting carry meaning on Event Hubs?
A: Event Hubs accepts the `acks` config for client compatibility, but durability is provided by the managed service's internal replication rather than a user-tunable replication factor, so you do not set `replication.factor` or manage in-sync replicas as on self-run Kafka. The client still benefits from `acks=all` semantics for confirmed writes. You cannot, however, change broker-side replication via Kafka admin — that is governed by the Azure platform.

Q: Your Kafka producer to Event Hubs throws `TimeoutException: Expiring N records` under burst load — first checks?
A: This usually means records sat in the producer buffer past `delivery.timeout.ms` because the broker was throttling you (TU/PU ingress ceiling) or the connection stalled. Check for `ServerBusy`/`ServiceBusy` throttling, add throughput units or enable auto-inflate, and tune `linger.ms`/`batch.size` and `retries`/backoff. Also confirm enough partitions so ingress spreads instead of bottlenecking one partition.

Q: How does enabling Capture interact with a compacted (`Compact`-policy) event hub?
A: Capture mirrors the live event stream to ADLS/Blob as it arrives, so on a compacted hub it captures events before compaction collapses them — giving you both the latest-value-per-key view in the hub and a fuller raw record in storage. This lets you keep a compacted "current state" feed while still landing history for audit. The two retention models (key-based in the hub, time/size-rolled files in Capture) operate independently.

Q: What controls how often Capture rolls a new file, and what is the empty-file behavior?
A: Capture rolls a file when either a configured time window (for example every 5 minutes) or a size window (for example 300 MB) is reached, whichever comes first. By default Capture can also emit empty files during idle windows; there is a "do not emit empty files" option to suppress them. Tuning these windows trades file count and small-file overhead in the lake against capture latency.

Q: How do you read Event Hubs Capture Avro output in Spark on your Ubuntu laptop, and what is the gotcha?
A: Point Spark at the Capture folder with `spark.read.format("avro")` (the `spark-avro` package), then parse the `Body` column, which holds the original event payload as raw bytes that you must decode (for example `CAST(Body AS STRING)` then JSON-parse). The gotcha is that Capture wraps each event in an Avro envelope with metadata fields (`SequenceNumber`, `Offset`, `EnqueuedTimeUtc`, `Body`), so the business payload is nested inside `Body`, not at the top level.

Q: In the Capture Avro envelope, what is the difference between `EnqueuedTimeUtc` and an event-time field in the payload?
A: `EnqueuedTimeUtc` is the Event Hubs arrival/ingest timestamp added by the service, whereas the business event time (for example exchange execution time) lives inside the decoded `Body`. For correct windowing and ordering you key downstream processing on the payload event time, not `EnqueuedTimeUtc`, which can lag the actual occurrence. Confusing the two re-creates the arrival-vs-event-time bug at the batch layer.

Q: For a Stream Analytics job at compatibility level 1.2+, how do you make a job embarrassingly parallel without writing `PARTITION BY` in the query?
A: At compatibility level `1.2` or above you set the partition key as an input property in the input settings, and the engine parallelizes automatically with no `PARTITION BY PartitionId` clause needed. The requirement is still that all inputs, query steps, and outputs align on the same partition key. For 1.0/1.1 you must add `PARTITION BY PartitionId` in every step explicitly.

Q: What is the maximum SU V2 count for a Stream Analytics job, and roughly what throughput does that represent?
A: A job can scale up to `66 SU V2s`, which Microsoft cites as roughly `400 MB/s` or about `38 trillion events per day`. All nonpartitioned steps together cap at one SU V2, and you add one SU V2 per partition in a partitioned step. So the only way to climb toward 66 is a well-partitioned, fully parallel query over a high-partition input.

Q: Given an input partitioned by 3 and a two-step query where step one is partitioned and step two is not, what is the max SUs?
A: `4 SU V2s` — three for the partitioned step (one per input partition) plus one for the nonpartitioned aggregation step. The pattern is the standard "partitioned local aggregate, then a single nonpartitioned step to combine across partitions." This is exactly how you fix results that would otherwise be split across partitions when the `GROUP BY` key is not the input partition key.

Q: Why can a `GROUP BY TollBoothId` (or `GROUP BY ISIN`) produce duplicate per-key rows when partitioned by `PartitionId`?
A: If the input is partitioned by `PartitionId` but events for one `ISIN` are spread across several partitions, each partition aggregates that `ISIN` independently and emits its own row, so you get multiple partial counts per key per window. The fix is to either partition the input by `ISIN` so each key lands in one partition, or add a nonpartitioned second step that sums the partial aggregates. This is the most common "my counts are wrong" Stream Analytics bug.

Q: When are two joined streams NOT embarrassingly parallel, and what is the rule to fix it?
A: If the two inputs are partitioned by different keys (say one by `WarehouseId` and one by `ProductId`), the join forces a shuffle and the job is not parallel. The rule is to partition both streams by the join key and give them the same number of partitions. For a trade-vs-reference join, key both on `ISIN` so matching records co-locate.

Q: For an Event Hubs OUTPUT from Stream Analytics to stay fully parallel, what must you configure?
A: Set the Event Hubs output's "Partition key column" to the same partition key the query uses (often `PartitionId`), and make the output partition count equal the input partition count. If input has 8 partitions, the output hub should have 8; a mismatch (for example 8 in, 32 out) breaks embarrassing parallelism regardless of the query. Power BI output cannot be partitioned at all.

Q: Which Stream Analytics outputs auto-parallelize and which need explicit partition configuration?
A: ADLS, Azure Functions, Azure Table, and Service Bus auto-parallelize; Blob storage, Cosmos DB, Event Hubs, and IoT Hub require you to set the partition key explicitly; SQL and Synapse offer optional parallelization (Inherit Partitioning). Power BI does not support partitioning at all. So when a job won't scale at the sink, check whether that specific output type needs a partition key set.

Q: Past 8 partitions a fully parallel pipeline to Azure SQL stops scaling linearly — what is the technique to push further?
A: Repartitioning before the SQL output using the `INTO <shard count>` clause, which reshards the stream into more write lanes ahead of the sink. SQL write throughput is bounded by database tier and table schema, so beyond ~8 partitions you repartition and enable Inherit Partitioning. Premium SQL SKUs are also needed to sustain the IO under frequent log backups.

Q: Which Stream Analytics metrics tell you the job is falling behind, and what do they mean?
A: `Watermark Delay` (how far the job's event-time watermark lags real time) and `Backlogged Input Events` (events queued but not yet processed) are the lateness/backpressure signals, alongside Input/Output Events for throughput. A rising watermark delay or growing backlog means the job is under-provisioned or not parallel enough. You also watch Event Hubs `Throttled Requests` to catch TU exhaustion upstream.

Q: A Stream Analytics job shows high SU % Utilization and growing watermark delay — what are the levers, in order?
A: First confirm the query is fully parallel (partition input/query/output on the same key) so added SUs actually help, then raise SU V2s, then increase input partition count if you are already partition-bound. If the bottleneck is the sink, repartition before it or scale the output resource. Throwing SUs at a non-parallel query does nothing because nonpartitioned steps cap at one SU V2.

Q: How do you join a live trade stream against slowly-changing reference data in Stream Analytics, and what is the refresh gotcha?
A: Define a reference-data input (Blob or SQL Database) and `JOIN` it to the stream on the key (for example `ISIN`); the engine loads the snapshot into memory for enrichment. The gotcha is reference data is read by a path/snapshot pattern (often time-tokenized blob paths), so updates only take effect when a new snapshot is produced and picked up — a stale reference blob silently enriches with old values. Size the reference set to fit the SU memory budget.

Q: How would you express a temporal join "match each trade to a quote within the prior 5 seconds" in Stream Analytics?
A: Use `JOIN` with `DATEDIFF` in the `ON` clause bounding the time difference, e.g. `... ON t.ISIN = q.ISIN AND DATEDIFF(second, q, t) BETWEEN 0 AND 5`. The mandatory `DATEDIFF` bound is what makes it a temporal join over data in motion, limiting how far back the engine must hold state. Both inputs should be timestamped via `TIMESTAMP BY` their event time for correct matching.

Q: When would you choose a hopping window over a tumbling window for a fund-platform alert, with concrete parameters?
A: Choose hopping when you want overlapping coverage and earlier detection — e.g. `HoppingWindow(second, 60, 10)` evaluates a 60-second window every 10 seconds, so a breach surfaces within 10 seconds rather than waiting up to a full minute. A tumbling window (`TumblingWindow(minute, 1)`) is the no-overlap special case. The trade is more frequent, overlapping output (more cost) for lower detection latency.

Q: How do `LAG` and `LIMIT DURATION` work in Stream Analytics, and what is a fund use case?
A: `LAG` accesses a previous event in the stream within a `LIMIT DURATION` lookback window, letting you compare the current event to the prior one for the same key. For example, detect a NAV jump by comparing the current NAV for an `ISIN` to its previous value: `LAG(nav) OVER (PARTITION BY ISIN LIMIT DURATION(hour, 24))`. The `LIMIT DURATION` is required because the engine must bound how much history it retains.

Q: How does Stream Analytics' `AnomalyDetection_SpikeAndDip` function fit anomaly detection on a price stream?
A: It is a built-in machine-learning operator you call over a windowed stream (via `OVER` with a `LIMIT DURATION`) that flags spikes and dips with a score and an is-anomaly flag at a configured sensitivity. You apply it to, say, per-`ISIN` price to surface abnormal moves without training your own model. It is one of the complex-event-processing capabilities that keeps simple anomaly detection inside Stream Analytics rather than pushing to Flink.

Q: Stream Analytics SQL can call out to scoring — how, and where is the boundary versus Flink?
A: You can invoke an Azure Machine Learning endpoint as a function from the query, or use JavaScript/C# UDFs and UDAs, to enrich or score events inline. This covers lightweight per-event scoring and custom scalar logic. The boundary to Flink appears when you need large evolving keyed state, iterative algorithms, custom operators, or fine control over the state backend and exactly-once across arbitrary sinks.

Q: How does exactly-once differ between Stream Analytics and a Flink job, in practical terms?
A: Stream Analytics gives exactly-once processing (internal state isn't double-counted) plus at-least-once delivery, with end-to-end exactly-once delivery only on selected outputs — you get it declaratively with no checkpoint tuning. Flink gives exactly-once via its own checkpointing and two-phase-commit sinks across many connectors, but you configure checkpoint intervals, state backends (e.g. RocksDB), and sink semantics yourself. The trade is managed simplicity versus configurable end-to-end guarantees over arbitrary sinks.

Q: Why might a regulated Luxembourg fund platform prefer Event Hubs + Stream Analytics PaaS over self-hosting Kafka + Flink, beyond cost?
A: The managed services reduce the DORA operational-resilience and patching burden — no brokers/clusters to harden, plus built-in availability-zone spreading, geo-disaster-recovery, Private Link, IP firewall, and customer-managed keys on Premium/Dedicated. That shrinks the audit surface and the number of self-run components an architect must evidence for ICT-risk controls. Self-hosting Kafka/Flink can still win on feature control, but it puts the resilience and patching obligations back on the firm.

Q: How does Private Link / IP filtering on an Event Hubs namespace support a regulated networking posture?
A: Standard and above support a Private Endpoint (Private Link) so traffic to the namespace stays on the Azure backbone with no public exposure, plus an IP firewall to allowlist source ranges. Because Event Hubs presents one stable FQDN and virtual IP, you secure a single endpoint rather than per-broker rules. This is how a fund platform keeps its trade stream off the public internet for compliance.

Q: What is geo-disaster recovery on Event Hubs, and what does it replicate versus not?
A: Geo-DR pairs a primary and secondary namespace under an alias and replicates the metadata/entity configuration (event hubs, consumer groups, auth rules), giving you a failover alias to flip to the secondary. It does not replicate the event data itself — only the configuration — so in-flight messages are not carried over. For data-level replication you need the newer geo-replication feature (Premium/Dedicated) or your own Capture-based recovery.

Q: A consumer must process a poison trade event that repeatedly fails — Event Hubs has no dead-letter queue, so what is the pattern?
A: Because Event Hubs is a log with no per-message dead-letter, you implement it in the consumer: catch the failure, write the bad event (with error context) to a side "dead-letter" sink (a blob container, a Service Bus dead-letter queue, or a quarantine hub), checkpoint past it, and continue. This prevents one poison message from blocking the partition's progress. It is exactly the lifecycle Service Bus gives natively, which is why genuinely poison-prone work belongs there.

Q: Pulling it together for the fund streaming tier — when is the right answer "Event Hubs with a `Compact` hub" versus "Event Hubs + external state store"?
A: Use a `Compact` hub when you want the stream itself to serve as the rebuildable latest-value-per-key feed (e.g. current NAV or current position per `ISIN`), you are off Basic tier, the keyspace fits the per-partition size cap, and brief pre-compaction duplicates are tolerable. Use Event Hubs as transport plus an external store (SQL, Cosmos DB, Azure Data Explorer, Delta) when you need transactional point queries, secondary indexes, joins, or stronger read consistency than a compacted log offers. Either way keep `Delete`-policy Capture to ADLS for the full audit history the compacted view discards.

Q: How does the 5-non-epoch-receivers-per-partition limit differ from the per-namespace Kafka consumer group cap, and why confuse them?
A: The `5` non-epoch receivers per consumer group per partition is a concurrency cap on duplicate AMQP readers of one partition (they all see the same events), whereas the `1,000` Kafka consumer groups per namespace is a cap on how many independent group offsets you can track. One is "how many sockets fan out the same partition," the other is "how many distinct logical views of the stream." Conflating them leads architects to think they can only have a handful of reader applications, when in fact they can have up to 1,000 groups, each limited to one active consumer per partition.

Q: A Stream Analytics job needs to replay historical Capture data and live events together — how do you structure that, and what is the catch?
A: Configure the job to start from a custom timestamp ("start time") so it reads the retained Event Hubs backlog, or add a Blob/ADLS input pointed at the Capture Avro and `UNION` it with the live stream after aligning schemas via `TIMESTAMP BY` the payload event time. The catch is that Event Hubs only retains data for the tier window (up to 7 days on Standard), so anything older must come from Capture storage, not the live input. You also throttle the replay so the historical surge does not blow past SU capacity and inflate watermark delay.
