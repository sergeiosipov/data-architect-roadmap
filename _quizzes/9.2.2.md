# Phase 3 · 9.2.2 Data versioning & write-audit-publish — 150 self-test questions

Q: What is the write-audit-publish (WAP) pattern in one sentence?
A: WAP is a data-loading pattern where you first write new data to a staging area that consumers cannot see, then run automated checks (audit) against it, and only publish it to the consumer-visible table if the checks pass. The point is that the audit gate is structural rather than a promise: bad data is physically isolated until proven good. In a fund context this is why a corrupted price file can be caught before it ever reaches the gold layer that feeds the NAV calculation.

Q: Why do auditors and regulators favour the WAP pattern over a plain "load then validate" pipeline?
A: Because in WAP the isolation is structural — the data simply is not visible to consumers until the check passes — whereas "load then validate" relies on humans reacting to an alert after bad data is already live. Auditors want a control they can see in the system design, not a procedure someone might skip under deadline pressure. A demonstrable audit branch that never fast-forwarded on red is far stronger evidence than a log line saying someone meant to roll back.

Q: In a fund-administration platform, what is the concrete cost of NOT having a WAP gate before the gold layer?
A: Without WAP a corrupted price file lands directly in the gold layer, the NAV is computed and published wrong, and the remediation becomes a formal restatement letter to investors and regulators rather than a quietly deleted staging branch. The restatement is expensive, reputation-damaging, and may trigger regulatory reporting obligations. The architectural difference is whether a bad load is a non-event or an incident.

Q: What is an Iceberg snapshot?
A: A snapshot is an immutable, point-in-time record of the complete state of an Iceberg table — the set of data files that make up the table at the moment of a commit. Every write operation creates a new snapshot, and the table's metadata keeps a list of them, forming the lineage you can time-travel through. Branches and tags are simply named references that point at specific snapshots.

Q: What is an Iceberg branch?
A: An Iceberg branch is a named, independently mutable reference to a snapshot that has its own lifecycle and retention policy; writes to a branch create new snapshots on that branch without affecting other branches. This lets you stage changes on, say, an `audit` branch in isolation from `main`. Branches are the mechanism that makes table-level write-audit-publish possible inside a single Iceberg table.

Q: What is an Iceberg tag, and how does it differ from a branch?
A: A tag is a named reference to a single, fixed snapshot — it points at one snapshot and is not advanced by writes, making it effectively immutable, whereas a branch moves forward as you commit to it. You use a branch for ongoing work (staging, audit) and a tag to permanently mark a snapshot of record, such as the exact state used for a given valuation date's NAV. Tags are the natural fit for immutable audit markers.

Q: What is the name of Iceberg's default branch?
A: The default branch is called `main`. When you query a table normally you read from `main`, and operations that don't specify a branch write to `main`. Every Iceberg table has a `main` branch implicitly, and other branches are created relative to it.

Q: Write the Spark SQL to create an Iceberg branch named `audit` on table `prod.db.prices`.
A: `ALTER TABLE prod.db.prices CREATE BRANCH audit` (the branch name is normally back-quoted in real DDL). This creates a branch pointing at the current snapshot of `main` with default retention. You would then direct your staging writes to that branch, validate, and only fast-forward `main` if the checks pass.

Q: How do you create an Iceberg branch from a specific historical snapshot rather than the current one?
A: Use the `AS OF VERSION` clause, for example `ALTER TABLE prod.db.prices CREATE BRANCH `replay` AS OF VERSION 3`. This roots the new branch at snapshot id 3 instead of the current head of `main`. It is useful when you need to re-stage work from a known-good prior state, for instance to reproduce a NAV run from a particular point in the lineage.

Q: Write the Spark SQL to create an Iceberg tag named `EOM-01` at snapshot version 30, retained for 180 days.
A: `ALTER TABLE prod.db.prices CREATE TAG `EOM-01` AS OF VERSION 30 RETAIN 180 DAYS`. This pins an immutable marker to snapshot 30 and tells Iceberg to keep that reference for 180 days. End-of-month tags like this are exactly how you preserve the snapshot a month-end NAV was computed from.

Q: What happens if you try to `CREATE TAG` with a name that already exists, and how do you intentionally move a tag?
A: A plain `CREATE TAG` on an existing name fails because the tag already exists; to intentionally repoint it you use `CREATE OR REPLACE TAG` (or `REPLACE TAG`), which advances the named reference to a new snapshot. The gotcha is that replacing an audit tag silently rewrites your evidence of which snapshot a figure used, so audit tags should be treated as write-once and never replaced. Use replace only for moving working markers, not for tags of record.

Q: What does the `RETAIN <n> DAYS` clause control when creating a branch or tag?
A: `RETAIN <n> DAYS` sets the maximum reference age — how long the named branch or tag reference itself is kept before it becomes eligible for expiry. It governs the lifetime of the pointer, not directly the data files, though expiring the reference can make its snapshots collectable. For an audit tag you want this set long enough to cover your regulatory retention window.

Q: What does `WITH SNAPSHOT RETENTION 2 SNAPSHOTS` mean in `CREATE BRANCH ... RETAIN 7 DAYS WITH SNAPSHOT RETENTION 2 SNAPSHOTS`?
A: It sets a branch-level snapshot retention policy: keep at least 2 of the most recent snapshots on that branch regardless of age. This is independent of the 7-day reference age and protects a minimum number of snapshots from expiration. It lets a busy branch retain just a couple of recent states without holding everything forever.

Q: Which two retention dimensions can a branch carry, and what is the difference?
A: A branch can carry a maximum reference age (how long the branch reference lives, set via `RETAIN ... DAYS` / `max-ref-age-ms`) and a snapshot retention policy on the branch (a minimum number of snapshots and/or a maximum snapshot age via `min-snapshots-to-keep` and `max-snapshot-age-ms`). The first decides when the branch pointer itself expires; the second decides which snapshots on the branch survive expiration runs. Tuning both lets you keep an audit branch lean without losing the snapshots you must retain.

Q: How do you read from a specific Iceberg branch in Spark SQL?
A: Use the branch identifier, for example `SELECT * FROM prod.db.prices VERSION AS OF 'audit'` or the dot-suffix form `prod.db.prices.branch_audit`. This lets you validate the staged data on the `audit` branch with the same queries you would run on `main`. Being able to point your existing DQ checks at the branch is what makes WAP cheap to bolt on.

Q: How do you read the state of a table as of a tag named `EOM-01`?
A: Use `SELECT * FROM prod.db.prices VERSION AS OF 'EOM-01'`, which resolves the tag to its pinned snapshot and reads that exact state. Because a tag never moves, this query is reproducible forever (until the tag is expired). That reproducibility is the whole audit story: the same tag always returns the same data a NAV report was computed from.

Q: What is the `fast_forward` procedure in Iceberg and when do you call it?
A: `fast_forward` advances one branch to the head of another when the target is an ancestor of the source — for example `CALL catalog.system.fast_forward('prod.db.prices', 'main', 'audit')` moves `main` to the latest snapshot of `audit`. You call it as the "publish" step of WAP, after the audit checks on the `audit` branch pass. Because it is a fast-forward (no merge conflict resolution), the published state on `main` is exactly the validated state.

Q: In the WAP flow, what is the consequence of the audit checks failing on the staging branch?
A: If the checks fail you simply do not call `fast_forward`, so `main` never advances and the bad data stays quarantined on the `audit` branch where no consumer reads it. You then alert and hold the branch for inspection, or drop it. The defining property is that "do nothing" already protects production — there is no rollback to perform because nothing was published.

Q: What does the table property `write.wap.enabled` do?
A: Setting `write.wap.enabled=true` puts the table into write-audit-publish mode so that writes tagged with a WAP id are staged only — they create snapshots that are not reflected in the table's current state until explicitly published. It is the table-level switch that turns on the staged-write behaviour. Without it, the WAP id mechanism does not stage writes.

Q: What does the Spark session property `spark.wap.id` control?
A: `spark.wap.id` assigns an identifier to the writes performed in that session so that, with `write.wap.enabled=true`, the resulting snapshot is staged under that id instead of becoming the table's live state. Later you publish by referencing that id. It is how you label a batch of staged writes for a subsequent audit-and-publish decision.

Q: What does the `publish_changes` procedure do?
A: `publish_changes` takes a staged WAP id and publishes its changes into the current table state by cherry-picking the staged snapshot, creating a new snapshot from it without altering or removing the original. For example `CALL catalog.system.publish_changes('prod.db.prices', 'wap-2026-06-13')`. It is the WAP-id flavour of "publish", as opposed to the branch-fast-forward flavour.

Q: There is a known issue with `publish_changes` and multiple snapshots sharing one WAP id — what is it?
A: If multiple snapshots in the table carry the same `wap.id`, `publish_changes` only cherry-picks the earliest matching snapshot and silently ignores the others (tracked as Iceberg issue #14953). The gotcha is that staged changes can be quietly dropped, so you should ensure one WAP id maps to one staged snapshot, or prefer the branch-based WAP flow. Silent data loss in a regulated pipeline is exactly the kind of surprise to design out.

Q: What does the `cherry_pick_snapshot` procedure do?
A: `cherry_pick_snapshot` applies the changes from an existing snapshot onto the current table state, creating a new snapshot, without removing the original — for example `CALL catalog.system.cherry_pick_snapshot('prod.db.prices', 8)`. It is the lower-level building block that publish-style operations use. Only append and dynamic-overwrite snapshots can be cherry-picked.

Q: Which kinds of snapshots can be cherry-picked or published in Iceberg, and why does that limit matter?
A: Only append and dynamic-overwrite snapshots can be successfully cherry-picked or published. This matters because if your staged write was, say, a full overwrite or a row-level delete, the simple publish procedures may not apply and you must rethink the WAP mechanics. Knowing the limit up front prevents designing a WAP pipeline that cannot actually publish its staged writes.

Q: Compare the two ways Iceberg can implement WAP: branch-based vs WAP-id-based.
A: Branch-based WAP writes to a named branch (e.g. `audit`), validates by reading that branch, and publishes with `fast_forward`; WAP-id-based WAP uses `write.wap.enabled` plus `spark.wap.id` to stage snapshots under an id and publishes with `publish_changes`/`cherry_pick_snapshot`. The branch approach is more explicit, easier to inspect, and avoids the multiple-same-id gotcha. Most modern Iceberg WAP designs prefer branches for that clarity.

Q: What is an "audit branch" use case in Iceberg's own terminology?
A: The audit-branch use case is writing incoming data to a dedicated branch, running data-quality validations against that branch in isolation, and fast-forwarding `main` only when the validations pass. It is Iceberg's documented canonical example of branching, and it is precisely the WAP gate. The branch is the quarantine; the fast-forward is the release.

Q: What is the difference between Iceberg time travel via snapshot id and via a tag?
A: Time travel by snapshot id (`VERSION AS OF <id>`) targets a numeric snapshot that means little to a human and may be expired, whereas a tag gives that snapshot a stable, human-meaningful name with its own retention. For audit you want tags, because "the snapshot for valuation date 2026-06-12" is a far stronger reference than "snapshot 41872" that someone might expire. Tags turn time travel into a durable audit narrative.

Q: How would you demonstrate, using tags alone, which snapshot a given day's NAV report was computed from?
A: You tag the published snapshot with the valuation date at publish time, for instance a tag named `nav-2026-06-12` created with `CREATE TAG`, and later run `SELECT * FROM ... VERSION AS OF 'nav-2026-06-12'` to reproduce the exact input data. Because the tag is immutable and never moves, the query reproduces the precise state the NAV used. That is the audit deliverable: a named, reproducible pointer to the data of record.

Q: What is snapshot expiration in Iceberg and why is it relevant to a versioning strategy?
A: Snapshot expiration (e.g. via `expire_snapshots`) removes old snapshots and their orphaned data files to reclaim storage, but it can delete the very history you rely on for audit. It is relevant because branch and tag retention policies protect referenced snapshots from expiration, so your audit tags survive while unreferenced clutter is cleaned up. Getting retention wrong can either bloat storage or destroy required history.

Q: A teammate created an audit tag but it disappeared after a maintenance job. What is the first thing to check?
A: Check the tag's reference age / retention against the maintenance schedule — a tag created with a short `RETAIN ... DAYS` (or default) can become eligible for expiry, after which a snapshot-expiration job removes it. The fix is to recreate audit tags with a retention long enough to cover the regulatory window. Always set explicit, generous retention on audit markers rather than relying on defaults.

Q: Why does WAP at the Iceberg level only protect a single table?
A: Iceberg branches and tags are per-table references, so a branch isolates and an audit gate protects exactly that one table's writes — there is no cross-table atomicity. If a load must update several tables consistently (e.g. prices and FX rates together), table-level branching cannot guarantee they publish as one unit. That gap is precisely what lake-level tools like lakeFS and Nessie address.

Q: What problem does lakeFS solve that Iceberg branching does not?
A: lakeFS provides Git-like versioning across the whole data lake — many objects and many tables at once — rather than within a single table, so you can branch, validate, and merge a consistent set of files spanning multiple datasets atomically. Iceberg branches are scoped to one table; lakeFS branches span the repository. You reach for lakeFS when a change must be all-or-nothing across several tables or raw files.

Q: In lakeFS, what is a repository?
A: A lakeFS repository is the top-level versioned namespace over a location in object storage — conceptually like a Git repo for a portion of your data lake. It contains branches, commits, and the objects they reference, all backed by a bucket in S3, Azure Blob, GCS, or an S3-compatible store like MinIO. Everything you version in lakeFS lives inside a repository.

Q: What is a branch in lakeFS, and is creating one expensive?
A: A lakeFS branch is an isolated, writable line of development over the repository's data, and creating one is a cheap metadata-only operation that copies no objects (zero-copy). You get a consistent view of the whole repository that you can modify without affecting other branches. This is what makes spinning up a full staging environment for an entire lake nearly free.

Q: What is a commit in lakeFS?
A: A commit in lakeFS is an immutable checkpoint capturing a complete, consistent snapshot of the repository at a point in time, identified by a commit id. It records the exact state of all objects on a branch, so you can always return to it. Commits are what give lakeFS its reproducibility and rollback guarantees across many tables at once.

Q: What is a merge in lakeFS and why is it useful for WAP across many tables?
A: A lakeFS merge atomically updates one branch with the changes from another, so a whole set of multi-table changes either becomes visible together or not at all. For WAP this means you stage a coordinated load on a branch, validate it, and merge to `main` in one atomic step — no half-published state. That cross-table atomicity is the key thing table-level Iceberg branching cannot give you.

Q: How does lakeFS achieve zero-copy branching over object storage?
A: lakeFS stores version metadata that maps logical paths to the underlying immutable objects in your bucket, so a branch is just a new set of metadata references rather than duplicated data. It acts as a virtual filesystem that routes reads to the correct physical objects. Because branching only writes metadata, it is fast and atomic regardless of data size.

Q: What are lakeFS hooks and how do they relate to WAP?
A: lakeFS hooks are configured actions that run automatically on events such as pre-merge or post-create-branch, allowing you to run validations and block a merge if they fail. A pre-merge hook is effectively the audit gate of WAP at the lake level: the merge (publish) cannot complete unless the checks pass. This makes the data-quality gate a structural part of the merge, not an external step someone runs by hand.

Q: Which object stores does lakeFS support, and why does that matter for an on-prem or EU deployment?
A: lakeFS works over AWS S3, Azure Blob Storage, Google Cloud Storage, and any S3-compatible store such as MinIO or Dell ECS. This matters because a regulated EU fund administrator may need data resident in Azure or in an on-prem MinIO cluster, and lakeFS layers on top without forcing a specific cloud. The same Git-for-data model runs wherever the bucket lives.

Q: What is Project Nessie?
A: Nessie is a transactional catalog for data lakes that adds Git-like semantics — branches, tags, commits, and hashes — at the catalog level, primarily for Iceberg tables. Instead of versioning files or a single table, it versions the catalog's view of all tables and namespaces. It is the catalog-level seam for multi-table consistency.

Q: How does Nessie differ from lakeFS in where it sits in the stack?
A: Nessie operates at the catalog level — it versions table pointers and metadata for tables (typically Iceberg) registered in the catalog — whereas lakeFS operates at the object-storage level, versioning the raw files themselves. Nessie knows about tables and commits each Iceberg transaction as a Nessie commit; lakeFS knows about objects and paths. They solve overlapping problems from different layers.

Q: How does Nessie enable something close to a multi-table transaction?
A: You create a Nessie branch, perform a series of table operations (each an Iceberg/Nessie commit) on that branch, and then merge the branch back into `main` atomically, so all the table changes become visible together. This approximates a multi-table transaction with optimistic concurrency control. It is how you keep, say, a fact table and its dimension updates consistent on publish.

Q: In Nessie terminology, what is a commit hash?
A: A Nessie hash is a hexadecimal string identifying a particular commit — a consistent snapshot of all tables at that point in time. Branches and tags are human-friendly names that ultimately resolve to a hash. You can reference a hash directly for fully reproducible, immutable reads, much like a Git commit SHA.

Q: When you use Nessie as the catalog for Iceberg, what happens to each Iceberg transaction?
A: Each Iceberg transaction becomes a Nessie commit, so the catalog records a version of the whole namespace rather than only the single table's snapshot. This lifts versioning from per-table to catalog-wide. As a result you get cross-table point-in-time consistency that plain Iceberg snapshots, scoped to one table, cannot provide.

Q: What is DVC and what is it primarily used for?
A: DVC (Data Version Control) is a tool that brings Git-like versioning to large files, datasets, and ML model artifacts by storing lightweight pointer files in Git while the actual data lives elsewhere. It is file/artifact oriented, aimed at ML reproducibility, and for this lesson it is awareness-only rather than a core platform tool. You version a dataset or model the way you version code, without bloating the Git repo.

Q: What is a `.dvc` file and what does it contain?
A: A `.dvc` file is a small text pointer that DVC commits to Git in place of the large data file; it contains the data's content hash (typically an MD5) plus metadata needed to locate the real data. Git tracks the tiny pointer while the bytes stay in the DVC cache and remote. This is how DVC keeps Git repositories small while still versioning gigabyte-scale data.

Q: How does DVC store the actual data, and what is content-addressable storage in this context?
A: DVC moves the real file into a local cache (by default `.dvc/cache`) that is content-addressable — files are stored under their content hash, so identical content is stored once. The `.dvc` pointer's hash maps to the cached object's path. Content addressing deduplicates data and guarantees that a given hash always refers to exactly the same bytes.

Q: What do `dvc push` and `dvc pull` do?
A: `dvc push` uploads the data objects from your local DVC cache to a configured remote (S3, Azure Blob, GCS, etc.), and `dvc pull` downloads them back, typically after a `git pull` or `git clone` resolves the `.dvc` pointers. This separates code sync (Git) from data sync (DVC remote). It lets collaborators reconstruct the exact dataset that matches a given commit.

Q: Why is DVC a poor fit as the primary versioning layer for a regulated fund-data warehouse?
A: DVC versions whole files as opaque blobs with no notion of tables, schemas, transactions, or SQL-level time travel, so it cannot give you a query like "show the table as of valuation date." It is built for ML datasets and model artifacts, not for the transactional, table-aware, audit-gated workflows a fund warehouse needs. For that you want Iceberg branches/tags or lakeFS/Nessie, not DVC.

Q: What is the core trade-off between table-level versioning (Iceberg) and lake-level versioning (lakeFS/Nessie)?
A: Table-level Iceberg versioning is built into the table format with no extra service to run, but it cannot coordinate changes across multiple tables atomically; lake-level versioning (lakeFS/Nessie) adds cross-table/cross-file atomicity at the cost of operating another moving part in your stack. You choose based on whether your consistency boundary is a single table or many. Each granularity buys a different guarantee at a different operational cost.

Q: When is table-level Iceberg branching sufficient, and how would you justify that to a change-advisory board?
A: It is sufficient when each load's consistency boundary is a single table — the corrupted-price scenario where you only need to gate writes to one table before they reach consumers. You justify it by noting there is no extra service to deploy, monitor, or secure, so the operational and audit surface is minimal. The argument is: simplest control that fully covers the risk wins.

Q: When is lakeFS-style lake-level versioning worth the extra moving part?
A: It is worth it when a single logical change must update several tables or raw files consistently and publish atomically — for example a fund load that updates prices, FX, and holdings together, where any partial publish would corrupt the NAV. lakeFS/Nessie give you one branch and one atomic merge across all of them. The extra service earns its keep precisely when table-by-table gating would leave windows of inconsistency.

Q: How do Snowflake zero-copy clones approximate data branching, and where do they fall short?
A: A Snowflake clone (`CREATE TABLE ... CLONE`) creates new metadata pointers to the source's existing micro-partitions, giving an isolated, independently writable copy with virtually no storage cost — similar in spirit to a branch. It falls short because there is no built-in merge or fast-forward back to the source, and no native branch/merge workflow; you would script promotion yourself. So clones give cheap isolation but not true branch-merge versioning.

Q: What is Snowflake's zero-copy cloning mechanism under the hood?
A: Cloning creates a new set of metadata pointers that reference the same immutable micro-partitions as the source, so the clone initially consumes almost no additional storage. When either the clone or the source changes, Snowflake writes new micro-partitions for the changed data (copy-on-write) and the clone diverges only for what was modified. Unchanged data continues to be shared via pointers.

Q: Can a Snowflake clone be taken from a past point in time, and how?
A: Yes — the `CLONE` statement can include Time Travel syntax such as `AT (TIMESTAMP => ...)` or `BEFORE (STATEMENT => ...)` to clone the object as it existed at that moment. This combines cloning with Time Travel to reproduce a prior state cheaply. It is how Snowflake approximates "branch from an old snapshot," within the Time Travel retention window.

Q: What is the default and maximum Time Travel retention in Snowflake, and why does it constrain its use as a versioning tool?
A: Snowflake Time Travel retention is configurable per object from 0 up to 90 days (the 90-day maximum requires Enterprise edition). It constrains versioning because once data ages past the retention window it is no longer queryable via Time Travel and clones from before that point are impossible. For multi-year regulatory retention you cannot rely on Time Travel alone; you need explicit, durable markers like Iceberg tags.

Q: In the lesson's "Do" exercise, what is step one of adding WAP to the Dagster pipeline?
A: Step one is to make every load write to an Iceberg `audit` branch instead of writing directly to `main`. This converts the pipeline so that new data is staged and invisible to consumers by default. It is the structural change that makes the subsequent audit-and-publish gate possible.

Q: After writing to the `audit` branch in the Dagster WAP pipeline, what runs next?
A: The existing data-quality asset checks run against the `audit` branch — the same checks you already had, just pointed at the staged data instead of `main`. On green you fast-forward publish to `main`; on red you alert and hold the branch for inspection. Reusing the existing checks is why bolting WAP onto an Iceberg pipeline is relatively cheap.

Q: How do Dagster asset checks fit the WAP audit step?
A: Dagster asset checks are the automated validations that constitute the "audit" in write-audit-publish; you run them against the staged `audit` branch and gate the publish on their result. A passing set of checks triggers the `fast_forward` to `main`; a failing set blocks it and raises an alert. The asset-check result is the green/red signal that drives the gate.

Q: In the corrupted-load exercise, what failure should the WAP gate catch — and what is the expected outcome?
A: You deliberately inject a load with prices off by 100x for one ISIN; the DQ checks on the `audit` branch should flag the anomaly, so the gate never fast-forwards to `main`. The expected outcome is that the corrupted data is never visible to the gold consumer and remains quarantined on the branch with a full audit trail. That is the "done-when" proof that the control works.

Q: A price-off-by-100x corruption passes your WAP checks and reaches gold. What is the first thing to investigate?
A: First check whether your data-quality checks actually cover that failure mode — a 100x error implies a missing or too-loose range/reasonableness check on price, so the audit step had nothing to fail on. The WAP plumbing can be perfect, but it only protects against what the checks test. The fix is to add a bounded-range or day-over-day variance check that catches order-of-magnitude price jumps.

Q: Why does the lesson have you tag each published snapshot with the valuation date?
A: Tagging each published snapshot with its valuation date creates an immutable, human-meaningful audit marker so you can later time-travel back to the exact data a given day's NAV used. It turns the snapshot lineage into an auditable story keyed on the business date auditors care about. Demonstrating that time-travel-by-tag is one of the lesson's done-when criteria.

Q: What are the three done-when criteria for this lesson, in essence?
A: First, a deliberately corrupted load never becomes visible to the gold consumer and you can show the audit trail; second, you can demonstrate which snapshot a given day's NAV report was computed from using tags alone; third, you can explain to a CAB when table-level branching suffices versus when lake-level versioning is worth the extra moving part. Together they prove the gate works, the lineage is auditable, and you can reason about granularity. They map directly onto the WAP control, the audit story, and the architectural judgement.

Q: What is an ISIN and why is per-ISIN data quality central to the WAP exercise?
A: An ISIN (International Securities Identification Number) is the 12-character standard identifier for a security, and the WAP exercise corrupts the price of one specific ISIN. Per-ISIN checks matter because a single bad instrument can poison a NAV even when the rest of the load is fine, so the audit step must validate at instrument granularity. WAP isolates the whole load, but the checks must be fine-grained enough to catch one rogue ISIN.

Q: How does WAP relate to the integrity of a Luxembourg fund's NAV publication?
A: NAV must be struck from validated inputs, and WAP guarantees that prices and other inputs reaching the gold layer have passed the audit gate, so a bad input cannot silently flow into the published NAV. For a Luxembourg-domiciled fund this protects against a wrong NAV going to the transfer agent and investors. The structural gate is the difference between a caught load and a restatement.

Q: How could published Iceberg tags support an EMT/EPT file production audit trail?
A: EMT and EPT (European MiFID/PRIIPs templates) are produced from validated fund data, and tagging the exact snapshot used to generate a given file lets you reproduce precisely the inputs behind that template later. If a regulator questions a value, you time-travel to the tagged snapshot and show the source data unchanged. The tag is the immutable evidence linking the template to its data of record.

Q: Why is multi-table atomicity (lakeFS/Nessie) attractive for a transfer-agency load that updates several related tables?
A: A transfer-agency load may update subscriptions, redemptions, and holdings tables that must stay mutually consistent for downstream NAV and reporting, and lakeFS/Nessie let you stage all of them on one branch and merge atomically. Table-level Iceberg branching would publish each table independently, risking a window where holdings and subscriptions disagree. The lake-level branch makes the whole consistent set go live as one unit.

Q: How might a LEI-keyed reference-data update benefit from branch-based staging?
A: An LEI (Legal Entity Identifier) reference-data refresh can be large and occasionally wrong, so staging it on an `audit` branch lets you validate entity records (e.g. no orphaned LEIs, no duplicate mappings) before any consumer sees them. Only on green do you fast-forward to `main`. This keeps a faulty counterparty-reference load from cascading into downstream reports.

Q: In a SWIFT-message-driven settlement feed, where does WAP add value?
A: SWIFT-driven feeds can carry malformed or out-of-sequence messages, and staging the parsed records on an audit branch lets you run validation (field formats, mandatory tags, reconciliation totals) before publishing to the consumer tables. A failing batch is held, not published, so downstream settlement views never ingest corrupt data. WAP turns "react to a bad feed" into "never expose a bad feed."

Q: How does data versioning support DORA-style operational-resilience expectations?
A: DORA (Digital Operational Resilience Act) pushes financial entities toward demonstrable controls and recoverability, and versioned data with audit branches and immutable tags gives you both a structural quality gate and a provable rollback/recovery point. You can show the regulator the control that stopped a bad load and the tag you would recover to. Versioning thus turns resilience claims into evidence.

Q: Under UCITS/AIFMD oversight, why is reproducibility of a historical NAV input so important?
A: UCITS and AIFMD frameworks demand that fund administrators can substantiate reported values, so being able to reproduce the exact data that fed a past NAV is a compliance necessity, not a nicety. Immutable Iceberg tags (or Nessie/lakeFS commits) give you that reproducibility by name and date. Without it, answering "what did the NAV on date X use?" becomes a forensic reconstruction.

Q: How do you "walk the failure path end to end" for a corrupted load in the WAP pipeline?
A: You inject the corruption, observe the write land on the `audit` branch, watch the DQ asset checks fail, confirm `main` is never fast-forwarded, verify the gold consumer still reads clean data, and inspect the held branch and alert. End-to-end means you trace from injection through quarantine to the absence of any consumer impact. Doing this once is how you prove the control rather than assume it.

Q: A `fast_forward` call fails with an error that `main` is not an ancestor of `audit`. What does that mean and how do you fix it?
A: Fast-forward requires the target branch (`main`) to be an ancestor of the source (`audit`); the error means `main` advanced independently after `audit` was created, so the histories diverged. The fix is to re-create or rebase the `audit` branch from the current `main`, re-run the checks, and fast-forward again. In a pipeline you avoid this by branching `audit` from the latest `main` at the start of each load.

Q: Why must the consumer-facing read path point at `main` (or a published tag) rather than the `audit` branch?
A: Because the whole protection of WAP rests on consumers reading only the published reference; if a consumer query targets the `audit` branch it would see unvalidated, possibly corrupt staged data. The read path must be pinned to `main` or a specific published tag. A misconfigured consumer reading the staging branch silently defeats the gate.

Q: What is the risk of leaving many stale `audit` branches around, and how do you manage it?
A: Stale audit branches accumulate snapshots and pin data files, bloating storage and cluttering the metadata, and they can confuse operators about which branch is live. You manage it with branch reference-age retention so unmerged branches expire automatically, plus a pipeline step that drops a branch once it is published or definitively failed. Hygiene here keeps expiration jobs effective and the table readable.

Q: Why is "merge" in lakeFS atomic, and why does atomicity matter for publishing?
A: A lakeFS merge updates the target branch's metadata to incorporate the source's committed state in a single operation, so either the entire change set is applied or none of it is. Atomicity matters because partial publication of a multi-table change would create a window of inconsistency that downstream NAV or reporting could read. Atomic merge guarantees consumers never see a half-published state.

Q: How does a lakeFS pre-merge hook implement the "audit" gate of WAP at lake scale?
A: A pre-merge hook runs a validation when a merge into a protected branch is attempted and can block the merge if the validation fails, so the publish (merge) cannot happen on red. This is exactly the audit step, enforced structurally at the lake level across all the files in the change. The merge is the publish; the hook is the audit it must pass.

Q: What does it cost operationally to run lakeFS compared to relying on Iceberg branches alone?
A: lakeFS is an additional service you must deploy, secure, monitor, scale, back up, and integrate with your engines and authentication — operational surface that Iceberg branches (built into the table format) do not add. You also take on its metadata store as a dependency. The trade is real cross-lake versioning in exchange for a new component in your reliability and security perimeter.

Q: What does it cost operationally to run Nessie compared to Iceberg branches alone?
A: Nessie is a separate catalog service with its own backing store that must be deployed and made highly available, and all your engines must be configured to use it as the catalog. In return you get catalog-level, multi-table Git semantics. The cost is another stateful service on the critical path between engines and tables, which you must operate to the same standard as the warehouse.

Q: Why is "the audit step is structural, not a promise" the key phrase for explaining WAP to an auditor?
A: Because it captures that the control is enforced by the system design — data is physically isolated until the check passes — rather than depending on a person following a procedure that could be skipped. Auditors can verify a structural control by inspecting the pipeline; they cannot verify a promise. This framing is what makes WAP a credible control in a regulated environment.

Q: How do Iceberg branches give you environment isolation (e.g. dev/test) within one table?
A: You can create separate branches off the same table and read/write each independently, so a dev or test workflow operates on its own branch without touching `main` or other branches. This isolates experimentation from production state. It is a lighter-weight alternative to copying the whole table for a sandbox.

Q: How does branch-level retention differ from tag-level retention in intent?
A: Branch retention typically protects a working line that moves forward — you may keep a minimum number of recent snapshots and expire old ones — whereas tag retention protects a single fixed snapshot you want to preserve for a long, often regulatory, period. Intent differs: branches are about an evolving line's lifecycle, tags about preserving a specific moment. Audit markers therefore live on tags with long retention.

Q: What is the relationship between Iceberg metadata files and the branch/tag references?
A: Iceberg's table metadata records all snapshots plus the named references (branches and tags) and the snapshot each points to, so branches and tags are entries in metadata rather than separate copies of data. Resolving a branch or tag is a metadata lookup that yields a snapshot id. This is why creating a branch or tag is cheap — it just adds a reference.

Q: If two writers commit to the same Iceberg branch concurrently, what guarantees keep the branch consistent?
A: Iceberg uses optimistic concurrency with atomic metadata swaps: each commit is validated against the current branch state and only one wins a conflicting race, with the loser retrying. This keeps the branch's snapshot lineage linear and consistent. It is the same mechanism that protects `main`, applied per branch.

Q: Why can table-level versioning leave a "window of inconsistency" that lake-level versioning closes?
A: When two related tables are published independently (table-level), there is a moment where one is updated and the other is not, so a reader can see an inconsistent combination. Lake-level versioning publishes the related change as one atomic merge, eliminating that window. For coupled fund data like prices-and-FX, closing that window can be the difference between a correct and an incorrect NAV.

Q: How would you choose between Nessie and lakeFS when you need multi-table versioning?
A: Choose Nessie when your data is Iceberg tables in a catalog and you want catalog-aware, table-level transactions and Git semantics that understand tables; choose lakeFS when you must version arbitrary objects and many file types across the lake, not just catalog-registered tables. The deciding factor is whether your consistency boundary is "tables in a catalog" or "objects in a bucket." Nessie is table-aware; lakeFS is file/object-aware.

Q: What is "branch retention" protecting against if you set it too aggressively low?
A: If branch reference age is too low, the branch pointer expires quickly and a maintenance run can collect its snapshots before you finish staging or auditing, effectively losing your in-flight work. The symptom is staging branches vanishing mid-pipeline. Set retention comfortably longer than your longest load-plus-audit cycle, and longer still for anything that must persist for inspection.

Q: Why is it dangerous to expire snapshots without accounting for tags in a regulated platform?
A: An aggressive `expire_snapshots` run that ignores retained references could, if a required tag's retention has lapsed, remove the snapshot an auditor needs to reproduce a historical NAV. The danger is destroying evidence you are obliged to keep. The safeguard is long, explicit retention on audit tags so expiration can never reach them.

Q: How do you demonstrate the audit trail of a blocked corrupted load?
A: You show the `audit` branch still holding the staged corrupt snapshot, the DQ check result recording the failure, the absence of a fast-forward on `main`, and the consumer view returning only clean data. Together these artefacts are the audit trail: what was caught, where it was held, and that it never reached gold. The trail is the proof that the control fired.

Q: What is the difference between Snowflake Time Travel and true branch-merge versioning?
A: Time Travel lets you read or clone a past state within a retention window, but it has no notion of an isolated mutable branch you validate and then merge back — there is no fast-forward or atomic publish of staged changes. True branch-merge (Iceberg/lakeFS/Nessie) gives you a parallel line you can build on and publish deliberately. Time Travel is read-back-in-time; branch-merge is stage-and-promote.

Q: Why does the lesson position DVC as "awareness only" rather than hands-on?
A: Because DVC's file/artifact orientation does not match this platform's needs — transactional, table-aware, audit-gated versioning of warehouse data — so you should recognise what it is and where it fits (ML datasets/models) without building the pipeline on it. The hands-on tools are Iceberg branches and lakeFS. Knowing DVC exists prevents you from reaching for it on the wrong problem.

Q: A consumer reports seeing data that later "disappeared." How could a misconfigured WAP setup cause this?
A: If the consumer query was inadvertently pointed at the `audit` branch (or read a staged WAP-id snapshot) and that branch was later dropped or never published, the data would appear and then vanish. The root cause is a read path not pinned to `main`/published tags. The fix is to enforce that consumers only ever resolve to the published reference.

Q: Why is fast-forward (rather than a content merge) the right publish primitive for single-table WAP?
A: Fast-forward simply moves `main` to the validated head of `audit` without recomputing or reconciling content, so what becomes live is byte-for-byte the exact state you audited. A content merge could combine in changes you did not validate. For WAP you want the published state to equal the audited state, which fast-forward guarantees.

Q: How does versioning help with the "deleted branch instead of a restatement letter" outcome the lesson highlights?
A: When a bad load is isolated on a branch and never published, remediation is just dropping or holding that branch — a quiet internal action with no consumer impact, hence no restatement. Without versioning the bad load is already live, so remediation escalates to a formal restatement to investors and regulators. Versioning converts a potential incident into a non-event.

Q: What metadata would you capture at publish time to make a snapshot fully auditable?
A: At minimum the valuation/business date (as a tag), the snapshot id, the set of DQ checks that passed, the pipeline run id, and the source files ingested. This ties the published state to a business date and to the evidence that it passed the gate. With that captured, reproducing and defending a historical figure is a lookup, not an investigation.

Q: Why might you tag both the source/input snapshot and the published output for a NAV run?
A: Tagging the validated input snapshot lets you reproduce exactly what fed the calculation, while tagging the published output records the result of record; together they document the complete computation provenance. If a figure is questioned you can show both the inputs and the published state by name. This is stronger audit evidence than a single tag.

Q: How does optimistic concurrency in Nessie compare to Iceberg's per-table concurrency?
A: Nessie applies optimistic concurrency at the catalog/commit level across potentially many tables, so a branch merge or commit either succeeds against the current catalog state or is rejected for retry, coordinating multiple tables; Iceberg's optimistic concurrency operates per table on a single table's metadata. Nessie's scope is the namespace; Iceberg's is the table. That broader scope is what enables multi-table consistency.

Q: What does it mean that lakeFS "acts as a virtual filesystem" for your data?
A: It means lakeFS presents versioned, branch-scoped logical paths to clients and internally routes each read or write to the correct immutable object in the underlying bucket. Clients address data through lakeFS (e.g. via its S3-compatible gateway) and get a consistent branch view. The virtual-filesystem layer is what makes branches and commits transparent to engines.

Q: When a CAB asks "why add lakeFS instead of using Iceberg branches," what is the crisp answer?
A: Because the change in question must update several tables or raw files consistently and publish atomically, which per-table Iceberg branches cannot guarantee, whereas a single lakeFS branch and atomic merge can. If the change were single-table, Iceberg branching would suffice and lakeFS would be unjustified overhead. The answer turns on the consistency boundary of the change.

Q: Summarise the decision rule for choosing data-versioning granularity in this lesson.
A: Use table-level Iceberg branches/tags when the consistency and audit boundary is a single table — the cheapest control with no extra service; step up to lake-level lakeFS/Nessie when a change must be atomic and versioned across multiple tables or files, accepting the operational cost of another service. Match the granularity to the consistency boundary, and never pay for cross-lake versioning you do not need. The architect's job is justifying that choice to the CAB, not defaulting to the heaviest tool.

Q: Which Spark session property routes ordinary `INSERT INTO prod.db.prices` writes to a staging branch without changing the SQL, and what is the prerequisite?
A: Set `SET spark.wap.branch = audit-branch` in the session, and from then on plain writes like `INSERT INTO prod.db.prices` land on the `audit-branch` reference instead of `main`. The prerequisite is that `write.wap.enabled=true` is set on the table and the branch already exists, otherwise the write either errors or hits `main`. This is the cleanest WAP wiring because the pipeline SQL is unchanged — only the session config decides where the data stages.

Q: Write the Spark SQL to write directly to an Iceberg branch using the table-identifier suffix form rather than a session property.
A: Use the dot-suffix `branch_` identifier, for example `INSERT INTO prod.db.prices.branch_audit VALUES (...)`, and the same form works for `UPDATE prod.db.prices.branch_audit ...` and `DELETE FROM prod.db.prices.branch_audit WHERE ...`. The literal prefix is `branch_` followed by the branch name, so a branch called `audit` is addressed as `branch_audit`. This per-statement targeting is handy when only some writes in a job should stage while others go to `main`.

Q: What is the difference between `spark.wap.id` and `spark.wap.branch`, and which should a new WAP pipeline prefer?
A: `spark.wap.id` labels staged snapshots with an opaque id that you later publish via `publish_changes`/`cherry_pick_snapshot`, whereas `spark.wap.branch` stages writes onto a named branch you publish via `fast_forward`. A new pipeline should prefer `spark.wap.branch` because branches are inspectable, retention-managed, and avoid the same-id cherry-pick pitfall. The id form predates the branch form and is mostly kept for backward compatibility.

Q: A teammate complains that the WAP config behaviour is "undocumented and confusing" — which interactions actually trip people up?
A: The sharp edges are that `write.wap.enabled` must be `true` for either `spark.wap.id` or `spark.wap.branch` to stage, that you cannot meaningfully set both id and branch at once, and that the target branch must pre-exist before `spark.wap.branch` will route to it (the gaps tracked in Iceberg issue #11528). The gotcha is silently writing to `main` because one of these preconditions was missed. Always assert the table property and branch existence at pipeline start rather than trusting the session to fail loudly.

Q: Write the Spark SQL to delete an Iceberg branch named `audit` after a load is published.
A: `ALTER TABLE prod.db.prices DROP BRANCH audit` removes the named branch reference (the branch name is normally back-quoted in real DDL), after which its now-unreferenced snapshots become eligible for expiration. You run this as a cleanup step once `fast_forward` has published the validated state to `main`. Dropping spent branches keeps metadata small and stops operators confusing a stale staging branch with the live one.

Q: Write the Spark SQL to delete a tag named `EOM-01`, and what is the audit caution before doing so?
A: `ALTER TABLE prod.db.prices DROP TAG EOM-01` removes the tag reference (the tag name is normally back-quoted in real DDL). The audit caution is that dropping an audit tag destroys the named, reproducible pointer to a snapshot a regulator may later request, so tags of record must never be dropped inside their retention window. Reserve `DROP TAG` for transient working markers, not month-end or NAV tags.

Q: How do you repoint an existing Iceberg branch to a specific snapshot in one statement, and what clause sets its new retention?
A: Use `ALTER TABLE prod.db.prices REPLACE BRANCH `audit` AS OF VERSION 4567 RETAIN 60 DAYS`, which moves the `audit` reference to snapshot 4567 and resets its reference-age retention to 60 days. `REPLACE BRANCH` is the idempotent "create-or-move" you reach for when re-staging from a known-good snapshot. Unlike a tag of record, moving a working staging branch this way is safe.

Q: Which Iceberg table property sets the minimum number of snapshots kept during expiration, and what is its default?
A: The property is `history.expire.min-snapshots-to-keep` and its default is `1`, meaning by default only the single most recent snapshot is guaranteed to survive an `expire_snapshots` run. For an audit-friendly table you raise this so recent history is not pruned to a single point. Pair it with `history.expire.max-snapshot-age-ms` to bound retention by age as well.

Q: What does the table property `history.expire.max-snapshot-age-ms` control, and how does it interact with branch retention?
A: `history.expire.max-snapshot-age-ms` sets the default maximum age of snapshots before they become expirable on the table as a whole, while per-branch `max-snapshot-age-ms` and `min-snapshots-to-keep` override that for snapshots on a given branch. The branch-level policy wins for that branch, so an audit branch can retain more (or fewer) snapshots than the table default. Getting these layered correctly is how you keep evidence without bloating storage.

Q: You run `expire_snapshots` and your `audit` branch's in-flight snapshot vanishes mid-load. What two settings would have prevented it?
A: Either a generous branch reference age (`max-ref-age-ms` / `RETAIN ... DAYS`) so the branch pointer itself is not collected, plus a branch `min-snapshots-to-keep` high enough to protect the staged snapshot from being aged out. The root cause is that expiration only spares referenced snapshots and adequately retained references. Set both comfortably longer than your longest load-plus-audit cycle so maintenance never races a live load.

Q: Show the exact `CALL` to fast-forward `main` to the head of `audit` on a Glue/Hadoop catalog named `prod`.
A: `CALL prod.system.fast_forward('db.prices', 'main', 'audit')` advances `main` to `audit`'s head, where the first argument is the table, the second is the branch to move, and the third is the branch to move it to. The procedure lives under the catalog's `system` namespace, so the catalog name (`prod` here) prefixes `system.fast_forward`. It only succeeds when `main` is an ancestor of `audit`.

Q: After `fast_forward` publishes, how do you confirm in SQL that `main` and `audit` now point at the same snapshot?
A: Query the metadata refs table, for example `SELECT name, snapshot_id FROM prod.db.prices.refs`, and confirm the `main` and `audit` rows share the same `snapshot_id`. The `.refs` metadata table lists every branch and tag with the snapshot it resolves to. This is the cheap, scriptable assertion to add as a post-publish check in the pipeline.

Q: Which Iceberg metadata table lists all branches and tags, and what columns identify a reference's type and target?
A: The `.refs` metadata table (e.g. `SELECT * FROM prod.db.prices.refs`) lists every named reference with a `type` column (`BRANCH` or `TAG`), the `name`, the `snapshot_id` it points at, and the retention columns `max_reference_age_in_ms`, `min_snapshots_to_keep`, and `max_snapshot_age_in_ms`. It is the single source of truth for "what references exist and how long they live." Auditing this table is how you prove your audit tags carry sufficient retention.

Q: How would you list the full snapshot history of a table to find the snapshot id you want to tag?
A: Query `SELECT snapshot_id, committed_at, operation FROM prod.db.prices.snapshots ORDER BY committed_at`, which exposes each snapshot's id, commit timestamp, and operation type (`append`, `overwrite`, `delete`). You pick the id you need and feed it to `CREATE TAG ... AS OF VERSION <id>`. The `.snapshots` metadata table is the lineage you time-travel through.

Q: Why does the `operation` column of `.snapshots` matter when planning a cherry-pick-based publish?
A: Because only `append` and dynamic-overwrite snapshots can be cherry-picked or published, so a snapshot whose `operation` is a row-level `delete` or a full `overwrite` cannot be promoted by `cherry_pick_snapshot`/`publish_changes`. Checking the `operation` value before designing the publish step prevents building a WAP flow whose staged writes can never go live. For deletes you must use the branch-and-fast-forward flow instead.

Q: In a Dagster job, how do you make the "publish" decision conditional on an asset check using a fast-forward?
A: You gate the `CALL ... system.fast_forward(...)` behind the asset-check result: if the check passes (green) the op issues the fast-forward, and if it fails (red) the op short-circuits, raising a failure and leaving `main` untouched. Dagster's check result is the boolean that drives the branch in your op logic. The structural property is that the publish op simply never runs on red.

Q: How can you reuse one set of DQ SQL checks against either `main` or `audit` without duplicating the queries?
A: Parameterise the table reference so the same query template runs against `prod.db.prices` or `prod.db.prices.branch_audit` (or via `SET spark.wap.branch`), swapping only the branch identifier. Because Iceberg lets you point any read at a branch, the validation logic is identical and only the target changes. This is precisely why bolting WAP onto an existing pipeline is cheap — you re-aim the checks, you do not rewrite them.

Q: Two concurrent loads both try to stage on the same `audit` branch. What happens, and how do you isolate them?
A: They contend on the same branch's optimistic-concurrency commits, so one wins and the other must retry against the new head, and worse, their staged data can interleave into one ambiguous state. The fix is to give each load its own uniquely named branch (e.g. `audit-<runid>`), validate and fast-forward independently, then drop it. Per-run branches keep loads isolated and make the audit trail attributable to a single run.

Q: A `fast_forward` returns "cannot fast-forward: target is not an ancestor of source." Beyond rebasing, what pipeline design avoids it entirely?
A: Branch the `audit` reference from the current head of `main` at the very start of each load via `CREATE OR REPLACE BRANCH audit AS OF main`, so `main` is always an ancestor by construction. The error arises only when `main` advances after the branch was cut; recreating the branch fresh per run removes the window. Combined with per-run branch names, divergence simply cannot occur.

Q: How do you compute exactly which rows changed between the staged `audit` branch and live `main` for a sanity check before publishing?
A: Run a set-difference query across the two references, for example `SELECT * FROM prod.db.prices.branch_audit EXCEPT SELECT * FROM prod.db.prices` (and the reverse), to surface added or removed rows. Because both are addressable as tables, ordinary SQL diffs them. Reviewing this delta before fast-forward is a cheap human-readable confirmation that the load changed only what you expected.

Q: A NAV recalculation must reproduce inputs from three months ago, but the snapshot id is long expired. How could tags have saved you?
A: An immutable tag created at publish time with a retention spanning the regulatory window (e.g. `CREATE TAG `nav-2026-03-14` AS OF VERSION <id> RETAIN 2557 DAYS`) keeps that snapshot referenced and therefore safe from expiration. Without the tag, `expire_snapshots` reclaimed the files and the state is unrecoverable. The lesson's rule is to tag at publish, not to hope the numeric snapshot survives.

Q: Roughly how many days is a seven-year regulatory retention, and why express tag retention in days?
A: Seven years is about `2557` days (7 × 365 + 2 leap days), and Iceberg's `RETAIN <n> DAYS` clause takes a day count, so you translate the regulatory window into days explicitly. Expressing it in days makes the control auditable — anyone can read the DDL and see the reference outlives the obligation. Always round up to cover the full window plus a margin.

Q: How would you script a nightly check that every audit tag still has enough retention left to cover the regulatory window?
A: Query `prod.db.prices.refs` filtered to `type = 'TAG'`, compute remaining life from `max_reference_age_in_ms` against each tag's creation, and alert on any whose remaining retention drops below the required window. This catches a tag created with too-short `RETAIN` before an expiration job can collect it. Treat it as a control on the control — proving the audit markers themselves are durable.

Q: In lakeFS, write the `lakectl` commands to create a staging branch, then merge it to `main` after validation.
A: Create with `lakectl branch create lakefs://repo/load-2026-06-14 -s lakefs://repo/main`, then after checks pass merge with `lakectl merge lakefs://repo/load-2026-06-14 lakefs://repo/main`. The `-s` (source) flag roots the new branch on `main`, and `merge` takes source then destination. This is the lake-level analogue of cut-branch, validate, fast-forward.

Q: Write the `lakectl` command to commit staged objects on a lakeFS branch with a message.
A: `lakectl commit lakefs://repo/load-2026-06-14 -m "stage prices+fx+holdings for 2026-06-14"` records an immutable checkpoint of the branch's current object state with that message. Until you commit, changes are uncommitted on the branch and not yet a referenceable point in history. The commit is what a later merge promotes atomically to `main`.

Q: What lakeFS `lakectl merge` flag forces conflicting changes to favour the source branch, and when is that risky?
A: `lakectl merge ... --strategy source-wins` makes conflicts resolve in favour of the source (the other option is `dest-wins`). It is risky because automatically overwriting destination changes can silently discard data someone else published to `main`, which in a fund warehouse could mean clobbering a concurrent correction. Prefer resolving conflicts deliberately and reserve auto-strategies for branches you know cannot truly conflict.

Q: How is a lakeFS pre-merge hook configured to gate a merge into `main`, and in what language are the checks typically written?
A: You add an action YAML with `on: pre-merge: branches: [main]` listing hooks that run before the merge finalizes, commonly `type: lua` pointing at a `script_path` such as a schema or row-count validator. If the hook fails, the merge is blocked, so the publish cannot happen on red. Lua runs in an embedded VM inside the lakeFS server, so no external service is needed for the gate.

Q: A lakeFS merge into `main` is rejected by a pre-merge hook. Where do you look to see why?
A: Inspect the action run output in the lakeFS UI or via `lakectl actions runs describe` / the runs log, which records each hook's output and the failure reason (e.g. a Parquet schema mismatch the Lua validator flagged). The merge stays blocked until the offending commit on the source branch is fixed and a new merge attempted. This run record is the lake-level equivalent of the failed Dagster asset-check evidence.

Q: How does a lakeFS commit id differ from an Iceberg snapshot id in scope?
A: A lakeFS commit id captures the entire repository's object state at a point in time across all paths and datasets, whereas an Iceberg snapshot id captures only one table's data files. The lakeFS commit is therefore the right reference when you must reproduce a multi-table, multi-file state as one unit. Iceberg's snapshot is the right reference when a single table's lineage is the boundary.

Q: A consumer reads through the lakeFS S3 gateway and sees stale data. What branch-related misconfiguration is the likely cause?
A: The consumer's gateway path likely pins a specific branch or commit that has not been merged forward, so it reads an old committed state rather than the latest `main`. lakeFS routes reads by the branch/commit in the path, so a hard-coded staging branch or pinned commit serves stale objects. The fix is to point consumers at `main` (or an intentionally pinned published commit), never at a transient staging branch.

Q: In Nessie, write the steps (conceptually) to perform a two-table change atomically using a branch.
A: Create a Nessie branch (e.g. `etl`), perform the Iceberg writes to both tables on that branch so each becomes a Nessie commit, then `MERGE BRANCH etl INTO main`, which applies all those commits to `main` atomically. Either both table updates appear on `main` together or none do. This is how Nessie approximates a multi-table transaction with optimistic concurrency.

Q: A Nessie merge fails with a conflict. What concurrency model causes this and what is the remedy?
A: Nessie uses optimistic concurrency at the catalog level, so if `main` advanced after your branch was cut the merge detects a conflicting state and rejects it. The remedy is to rebase or re-cut your branch from the new `main`, re-apply or re-validate the changes, and merge again. It is the catalog-wide analogue of Iceberg's per-table commit retry.

Q: How do you address a fully reproducible read in Nessie that survives branch movement, and what is the analogy?
A: Reference the commit hash directly (a hex string) rather than a branch or tag name, so the read resolves to a fixed catalog state regardless of how branches later move. It is analogous to checking out a Git commit SHA instead of a branch name. For an immutable audit read of multiple tables at a point in time, the hash is the durable handle.

Q: A regulator asks you to prove FX rates and security prices used for one NAV were mutually consistent. Which tool's primitive answers this cleanly and why?
A: A Nessie commit hash (or a lakeFS commit) answers it cleanly because it pins the state of both tables as one catalog/repository snapshot, so you reproduce the exact mutually consistent pair. Two independent Iceberg tags could in principle have been created at slightly different states, leaving a consistency gap. The cross-table commit is the single reference that removes any doubt about alignment.

Q: For a load that updates only the single prices table, why is reaching for lakeFS over-engineering, and what do you tell the CAB?
A: Because the consistency boundary is one table, Iceberg branch-and-tag already gives a structural audit gate with no extra service to deploy, secure, or monitor, so lakeFS adds operational and security surface for no additional guarantee. You tell the CAB the simplest control that fully covers the risk wins, and a single-table load does not need cross-lake atomicity. Choosing the lighter tool is the defensible architectural decision.

Q: Distinguish a Snowflake zero-copy clone from a true Iceberg branch in terms of the publish step.
A: A Snowflake clone gives an isolated writable copy but has no native fast-forward or merge to promote validated changes back to the source — you would hand-script the promotion — whereas an Iceberg branch publishes deterministically with `fast_forward`. The clone is cheap isolation without a branch-merge lifecycle. That missing atomic publish is exactly why clones approximate but do not equal WAP.

Q: Within Snowflake, how would you approximate a WAP gate given clones and Time Travel, and where does it break down?
A: You clone the target, load and validate on the clone, then swap it in (e.g. `ALTER TABLE ... SWAP WITH clone`) on green, using Time Travel to recover if needed. It breaks down because the swap is a manual scripted promotion rather than a structural fast-forward, Time Travel retention caps at 90 days (Enterprise), and there is no cross-table atomic merge. For multi-year reproducibility you still need durable markers like Iceberg tags.

Q: What is Snowflake's `SWAP WITH` and why is it relevant to a clone-based publish?
A: `ALTER TABLE a SWAP WITH b` atomically exchanges the metadata of two tables, so a validated clone can replace the live table in a single metadata operation. It is the closest Snowflake primitive to an atomic publish of staged data. The relevance is that it gives the "make validated state live" step, but only for one table and without a branch lineage you can time-travel by name.

Q: Compare how Iceberg, lakeFS, and Nessie each define the "publish" operation.
A: In Iceberg, publish is `fast_forward` (or cherry-pick/publish_changes) advancing a table's branch; in lakeFS it is an atomic `merge` of a branch into `main` across all objects; in Nessie it is a `MERGE BRANCH` applying many table commits to `main` atomically. The scope grows from one table, to all files in a repo, to all tables in a catalog. Same WAP shape, three different consistency boundaries.

Q: How would you prove to an auditor that the audit gate "cannot be skipped" in an Iceberg WAP pipeline?
A: Show that consumers read only `main`/published tags, that the publish op's sole trigger is a passing asset check, and that on a failed run `main`'s `snapshot_id` in `.refs` is unchanged while the staged snapshot sits on the held `audit` branch. The evidence is structural — the data was physically isolated and the publish op never fired. That is stronger than any log line claiming a human rolled back.

Q: An auditor asks "show me every dataset state used for the 2026-05-31 month-end." Which artefacts do you produce per tool?
A: For Iceberg, the `EOM` tags on each table resolved via `.refs` plus the time-travel reads they enable; for lakeFS, the committed `main` commit id for that date; for Nessie, the commit hash pinning all tables. Each is an immutable, named pointer reproducing the exact state. The deliverable is reproducibility by name, not a forensic reconstruction.

Q: How does versioning support DORA's expectation of recovery to a known-good state after an incident?
A: Immutable tags/commits give you explicit, named recovery points, so after a corruption incident you can fast-forward or check out the last validated state and demonstrate the recovery objective was met. DORA pushes for provable recoverability, and a tag of record is the provable artefact. You show both the gate that caught the bad load and the marker you would recover to.

Q: How does T+1 settlement pressure change the operational requirements of a WAP gate on a fund data load?
A: T+1 compresses the window between trade date and settlement, so the audit checks must run and the publish decision must be made fast enough that validated prices and positions are available the next business day. WAP still applies, but the DQ checks have to be performant and the fast-forward automated, because there is no slack to hand-investigate a held branch and still meet the deadline. The architecture choice favours quick, automated branch-based publishing over heavier multi-service flows where latency would threaten the cutoff.

Q: Under the EU AI Act, why would a versioned, time-travellable training dataset be valuable for a model used in fund operations?
A: The EU AI Act expects data governance and traceability for AI systems, so being able to reproduce the exact dataset version a model was trained or validated on — via an Iceberg tag, lakeFS commit, or Nessie hash — provides the auditable data provenance the regime asks for. If a model's behaviour is questioned, you time-travel to the tagged training snapshot and show the inputs unchanged. This is the same WAP/versioning machinery repurposed as AI-governance evidence.

Q: How could the EET (European ESG Template) production process benefit from tagging the snapshot used per file?
A: EET files carry ESG data points derived from validated fund data, so tagging the exact snapshot used to generate a given EET (e.g. `eet-2026-06-30`) lets you reproduce the precise inputs if an ESG figure is later challenged. The immutable tag links the template to its data of record, just as for EMT/EPT. It turns "which ESG values did we report and from what?" into a single time-travel query.

Q: A daily SWIFT settlement feed occasionally arrives with messages out of sequence. How does staging on a branch help beyond format checks?
A: Staging the parsed messages on an `audit` branch lets you run sequence and reconciliation checks (e.g. running balances, expected message counts, no gaps in sequence numbers) across the whole batch before any consumer sees it, not just per-message format validation. A batch that fails the cross-message reconciliation is held on the branch, never published. WAP thus catches integrity problems that only emerge when the batch is viewed as a set.

Q: For a transfer-agency load touching subscriptions, redemptions, and holdings, why is per-table Iceberg WAP insufficient and what symptom would prove it?
A: Because the three tables must stay mutually consistent for NAV and reporting, but per-table fast-forwards publish them independently, leaving a window where, say, holdings reflect a redemption that subscriptions do not yet. The proving symptom is a downstream read during that window showing units and cash that do not reconcile. lakeFS/Nessie close it by merging all three as one atomic unit.

Q: Summarise the end-to-end WAP control as a one-line decision-and-evidence checklist an architect would present.
A: Stage every load on an isolated, retention-protected branch; gate the publish solely on automated DQ checks; fast-forward (or atomically merge) only on green; tag the published snapshot with the business date for durable reproducibility; and keep consumers pinned to `main`/published tags — with `.refs`, check results, and the held branch as the audit trail. Scope the branch to a single table for Iceberg, or to many via lakeFS/Nessie when the consistency boundary spans tables. The whole thing is defensible because the gate is structural and every state is reproducible by name.

Q: How would you write the `CREATE OR REPLACE BRANCH` statement that cuts a fresh `audit` branch from the current head of `main` at the start of every load?
A: `ALTER TABLE prod.db.prices CREATE OR REPLACE BRANCH `audit` AS OF VERSION main` repoints (or creates) the `audit` reference at `main`'s current head idempotently, so each run starts from the latest published state. Doing this per run guarantees `main` is always an ancestor of `audit`, which is the precondition `fast_forward` needs. It is the one-line guard that prevents the "not an ancestor" publish failure entirely.

Q: A scheduled `expire_snapshots` and a running WAP load overlap nightly and occasionally collide. How do you sequence maintenance to remove the race safely?
A: Schedule `expire_snapshots` in a maintenance window that does not overlap active loads, and have it honour reference retention so it only collects snapshots not protected by a branch or tag — for example run it after the day's loads have published and their staging branches are dropped. The collision arises when expiration runs while a staging branch's snapshot is still in flight and under-retained. Sequencing maintenance after publish, plus generous branch retention, makes the two operations independent.
