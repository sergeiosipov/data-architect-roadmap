# Phase 8 · 13.4.2 + 2.6.1 Data APIs & gateways (data-as-a-service) — 100 self-test questions

Q: What is "data-as-a-service" and why does an architect treat the API as a product surface rather than an afterthought?
A: Data-as-a-service means exposing curated, governed data through a stable API contract that operational consumers (websites, partner systems, downstream apps) build against, instead of letting them query the database directly. The architect treats the API as the product surface because it is the externally-visible boundary where auth, quotas, versioning, and a stable contract must live. Get it wrong and consumers couple to your internal schema, so every refactor becomes a breaking change and you have no usage attribution or back-pressure.

Q: In the fund-data domain, what are the natural REST resources to model, and how do they relate?
A: The natural resources are `/funds`, `/share-classes`, and `/navs`, modeled as a hierarchy where a fund has many share classes and each share class has a time series of NAVs. Modeling them as distinct resources (rather than one mega-endpoint) lets consumers fetch exactly what they need and lets you paginate and filter each independently. A common gotcha is conflating fund-level and share-class-level identifiers, since the ISIN identifies a share class, not the umbrella fund.

Q: Why should NAV figures be served from a "gold" view rather than from raw or silver tables?
A: A gold view is the curated, business-validated layer that has already passed data-quality and reconciliation checks, so it presents the official NAV a consumer is allowed to trust. Serving from gold keeps the API contract stable even as upstream raw/silver schemas churn, and it ensures you never accidentally expose an unvalidated or intra-day draft NAV. The view also becomes the single place to enforce column selection and naming for the public contract.

Q: What does "resource modeling" mean in Web API design, and what is the architect's main concern?
A: Resource modeling means deciding which nouns become addressable endpoints, how they nest, and what their representations contain — for example funds, share-classes, and navs as resources. The architect's main concern is that resources map to durable business concepts rather than to internal tables, so the contract survives database refactoring. Good modeling also keeps URLs predictable (`/funds/{id}/share-classes`) so consumers can guess them.

Q: Why is pagination mandatory for a list endpoint like `/navs` rather than optional?
A: Without pagination a single request to `/navs` could return millions of rows, exhausting server memory, saturating the network, and giving the consumer an unbounded, slow response. Pagination bounds each response to a page size, making latency and resource use predictable and protecting the database from full-table scans. It also maps directly to the OWASP risk of unrestricted resource consumption, so it is a security control as much as a usability one.

Q: What does PostgREST use the `Range` header for, and what `Range-Unit` does it expect?
A: PostgREST reads the request `Range` header with `Range-Unit: items` to select a slice of rows, so `Range: 0-19` requests the first twenty rows (zero-indexed, inclusive). Open-ended ranges like `Range: 10-` are supported for an offset-only query. This gives consumers row-range pagination without needing custom query parameters.

Q: What does the `Content-Range` response header from PostgREST tell the consumer?
A: `Content-Range` reports the returned slice and the total in the form `start-end/total`, so `Content-Range: 0-14/*` means rows 0 through 14 were returned and the total is unspecified. The `*` for total appears when the client did not request an exact count. Consumers use it to know whether more pages remain and, when a number replaces `*`, the full result-set size.

Q: How does a PostgREST consumer ask for the exact total row count, and what is the cost?
A: The consumer sends `Prefer: count=exact`, and PostgREST then runs a counting query and returns the precise total in `Content-Range` (e.g. `0-14/3000`). The cost is that the larger the table the slower this exact count runs, because it forces the database to count all matching rows. For big tables prefer `Prefer: count=planned` or `count=estimated`, which use PostgreSQL planner statistics instead of a full scan.

Q: What HTTP status code does PostgREST return for a full result versus a partial/counted one?
A: PostgREST returns `200 OK` when the whole result set is delivered, and `206 Partial Content` when only a slice is returned or a count was requested. The `206` signals the consumer that the `Content-Range` matters and that more rows may exist beyond the returned slice. Treating `206` as an error is a common consumer bug — it is a normal success status for paginated reads.

Q: What is the difference between `Prefer: count=planned` and `Prefer: count=estimated` in PostgREST?
A: `count=planned` returns the row estimate straight from the PostgreSQL query planner's statistics, which is fast but approximate and depends on `ANALYZE` being current. `count=estimated` is a hybrid: it returns an exact count when the result is below a configurable threshold and falls back to the planned estimate above it. Use these on large fund-history tables where an exact `count=exact` would scan too many rows on every page request.

Q: How should a list API expose filtering, and why keep the filter syntax in the contract?
A: Filtering should be expressed as query parameters that map to resource fields, for example `/navs?fund_id=eq.123&nav_date=gte.2026-01-01` in PostgREST's operator syntax. Documenting the filter syntax in the contract lets consumers retrieve only the rows they need, cutting payload size and database load. Keeping it explicit also prevents consumers from inventing brittle client-side filtering over full dumps.

Q: What is a "consistent error shape" and why does it matter to consumers?
A: A consistent error shape is a single, documented JSON structure for all errors — typically a code, a human-readable message, and optional details — returned with the correct HTTP status. It matters because consumers can write one error-handling path instead of guessing at each endpoint's idiosyncratic format. PostgREST, for instance, returns errors with `code`, `message`, `details`, and `hint` fields, which you should document so a stranger can program against them.

Q: Why is returning a 200 with an error body inside it considered an anti-pattern?
A: HTTP status codes are part of the contract that intermediaries (gateways, caches, monitoring) and client libraries act on automatically, so a 200 carrying an error makes failures invisible to all of them. The client's generic "if status is 2xx, succeed" logic will then treat the failure as success. Always map errors to the right status — `400` for bad input, `404` for missing resources, `409` for conflicts, `422` for validation — so the failure is machine-detectable.

Q: What is the architect's argument for URI versioning (e.g. `/v1/navs`)?
A: URI versioning puts the version in the path, which makes it trivially visible, cacheable, and easy to route at the gateway, and a consumer can see exactly which version they are calling just from the URL. The downside is that the version leaks into every link and some purists argue it violates REST resource identity. For fund data where a small set of long-lived versions coexist, the operational simplicity usually wins.

Q: What is header-based versioning, and what is its main trade-off?
A: Header-based versioning keeps a single clean URL and selects the version via a request header such as `Accept: application/vnd.fund.v2+json` or a custom version header. Its advantage is that resource URIs stay stable; its trade-off is reduced visibility and harder caching/routing, since the version is hidden from the URL and from casual log inspection. It also makes it easier for a consumer to forget to set the header and silently get the default version.

Q: How does an API versioning policy connect to a schema-evolution policy (the 1.9.7 reference)?
A: The API version is the consumer-facing expression of the same compatibility rules your schema-evolution policy enforces internally: additive (backward-compatible) changes need no new version, while breaking changes require one. Tying them together means a breaking schema change cannot silently reach consumers — it forces a new API version and a deprecation path. The contract and the schema thus evolve under one coherent compatibility discipline rather than two contradictory ones.

Q: What is a "deprecation path" for an API version and what does a responsible one include?
A: A deprecation path is the managed lifecycle for retiring an old version: announce the deprecation, run old and new in parallel for a stated window, emit deprecation signals (e.g. a `Deprecation` or `Sunset` header), and only then remove it. A responsible path gives consumers enough lead time and a clear migration guide, never a silent removal. For regulated fund consumers, the timeline often has to be contractually agreed and auditable.

Q: Give an example of an additive change that does NOT need a new API version, and one that does.
A: Adding a new optional field to the NAV response (e.g. `nav_per_share_gross`) is additive — existing consumers ignore it — so it does not need a new version. Renaming `nav` to `net_asset_value`, removing a field, or changing a field's type or meaning is breaking and requires a new version with a deprecation path. The rule of thumb: if a well-behaved existing consumer could break, it is a major version change.

Q: What is OpenAPI and why does an architect insist it lives in version control?
A: OpenAPI is a machine-readable specification (YAML or JSON) that describes every endpoint, parameter, schema, and response of a REST API — the formal contract consumers onboard from. Keeping it in version control makes the contract diffable, reviewable, and tied to releases, so a breaking change shows up in a pull request rather than as a surprise in production. It also enables generated clients, mock servers, and contract tests from one source of truth.

Q: Why is a generated OpenAPI document usually not enough on its own?
A: Auto-generation captures structure (paths, types, status codes) but produces empty or terse descriptions, so a stranger cannot tell what `nav_date` means versus `valuation_date`, or which filters are supported. The architect's job is to hand-edit the descriptions, examples, and constraints until the document is self-sufficient for onboarding. The "Done when" bar is precisely that an external consumer can onboard from the OpenAPI doc plus contract alone.

Q: PostgREST can emit an OpenAPI document automatically — what should you check before publishing it?
A: PostgREST serves an OpenAPI spec at the API root derived from the database schema, so check that it exposes only the intended gold views and not internal tables, that descriptions are filled in (it pulls some from PostgreSQL `COMMENT`s), and that security schemes reflect the real JWT/gateway setup. Publishing the raw generated spec risks leaking internal structure and shipping an undocumented contract. Treat the generated file as a starting draft to curate, then commit it.

Q: What is an API gateway, and what duties does it centralize that the service itself should not?
A: A gateway is a reverse proxy in front of one or more services that centralizes cross-cutting concerns: authentication/token validation, rate limiting and quotas, usage analytics, caching, and routing/versioning. Centralizing them means the upstream service (e.g. PostgREST) stays a thin data layer and you do not re-implement auth and quotas in every service. The architect's checklist is exactly these gateway duties, applied uniformly.

Q: What does "OIDC token validation" at the gateway mean, and why validate there rather than in the service?
A: It means the gateway verifies the OpenID Connect / JWT access token — checking signature against the issuer's keys, expiry, audience, and issuer — before the request reaches the backend. Validating at the gateway gives one trusted enforcement point, rejects bad tokens before they cost backend resources, and lets backends trust an authenticated identity. In this lesson the issuer is Keycloak, so Kong validates Keycloak-issued tokens via its OIDC/JWT plugin.

Q: How does Kong's OIDC plugin relate to Keycloak in this setup?
A: Keycloak is the identity provider that issues OIDC/JWT tokens to consumers after they authenticate, and Kong's OIDC (or JWT) plugin validates those tokens on every request by checking the signature against Keycloak's published JWKS, plus claims like `exp`, `aud`, and `iss`. This decouples authentication (Keycloak's job) from enforcement (Kong's job) and from data access (PostgREST's job). The backend never sees an unauthenticated request.

Q: What is the difference between a rate limit and a quota?
A: A rate limit caps requests over a short rolling window (e.g. 100 requests/second) to protect against bursts and abuse, while a quota caps total usage over a longer billing/entitlement period (e.g. 1M requests/month) tied to a consumer's plan. Rate limits defend availability; quotas enforce commercial entitlement and feed chargeback. A gateway typically implements both, and exceeding either returns `429 Too Many Requests`.

Q: What HTTP status code does Kong's rate-limiting plugin return when a limit is exceeded, and what body?
A: Kong returns `HTTP/1.1 429 Too Many Requests` and a JSON body indicating the rate limit was exceeded once any configured limit is reached. The `429` is the standard signal telling the client to back off and retry later. Designing consumers to honor `429` (and any `Retry-After`/reset header) is what makes back-pressure actually work.

Q: Which response headers does Kong's rate-limiting plugin add, and what do they convey?
A: Kong adds `RateLimit-Limit`, `RateLimit-Remaining`, and `RateLimit-Reset` (based on the IETF RateLimit Header Fields draft), and when multiple windows are configured, per-window headers like `X-RateLimit-Limit-Minute` and `X-RateLimit-Remaining-Minute`. `RateLimit-Remaining` tells the client how many requests remain before throttling, and `RateLimit-Reset` when the window resets. You can suppress these with `config.hide_client_headers` if you do not want to reveal the policy.

Q: Why might you set `config.hide_client_headers` to true on the Kong rate-limiting plugin?
A: Setting `config.hide_client_headers: true` stops Kong from returning the `RateLimit-*` / `X-RateLimit-*` headers, hiding the exact limits and remaining counts from clients. You might do this to avoid revealing your throttling policy to potential abusers or partners you do not want probing the boundaries. The trade-off is that well-behaved consumers lose the signal they need to self-pace, so it is usually left enabled for trusted partner tiers.

Q: What does "consumer tier" mean at the gateway, and why have more than one?
A: A consumer tier is a named class of API client (e.g. internal, standard partner, premium partner) with its own credentials, rate limits, and quotas configured at the gateway. Multiple tiers let you offer different service levels and price/charge accordingly, and they let you throttle a noisy free consumer without affecting a premium one. In Kong this is modeled with consumers grouped and plugins scoped per consumer or per group.

Q: How does per-consumer usage analytics at the gateway enable chargeback?
A: The gateway logs each request attributed to a specific authenticated consumer, so you can aggregate request counts, data volume, and endpoints per consumer over a period. That per-consumer usage is the evidence base for chargeback — billing or internally allocating platform cost to the team or partner that actually consumed the NAV API. Without an attribution point at the gateway, all consumers blur into anonymous database connections and cost allocation becomes guesswork.

Q: What concrete evidence proves quota enforcement is working, per this lesson's "Done when"?
A: The evidence is twofold: driving traffic past the configured quota and observing `429 Too Many Requests` responses, and showing per-consumer usage counts in the gateway logs. Together they prove the gateway both blocks over-limit traffic and attributes usage to the right consumer. This is the chargeback-and-enforcement proof the architect must be able to demonstrate, not just assert.

Q: What is PostgREST and why can it serve a REST API "in minutes"?
A: PostgREST is a standalone web server that automatically exposes a PostgreSQL schema as a RESTful API, deriving endpoints, filtering, and pagination directly from tables and views with no hand-written endpoint code. It serves a REST API in minutes because once you point it at a gold view and grant the right role, `/navs` and its filters exist immediately. The trade-off is that the database schema becomes the API design, so you must shape views deliberately.

Q: Why is a PostgREST API over a gold view typically configured read-only, and how?
A: A data-as-a-service NAV API is a consumption surface, not a write path, so it should be read-only to prevent any consumer from mutating official records. You enforce this by granting the API's database role only `SELECT` on the gold view (no `INSERT`/`UPDATE`/`DELETE`), and optionally exposing only views rather than base tables. PostgREST then physically cannot perform writes because the underlying role lacks the privilege — defense at the database, not just the application.

Q: How does PostgREST decide which database role a request runs as?
A: PostgREST authenticates with a low-privilege "authenticator" role, then reads the JWT's `role` claim and runs `SET LOCAL ROLE <rolename>` for the duration of that request, so the request executes with exactly that role's privileges. All authorization then happens in the database via grants and row-level security. If no valid JWT or no `role` claim is present, it uses the configured anonymous role.

Q: What is the `db-anon-role` in PostgREST and what is the danger of misconfiguring it?
A: `db-anon-role` names the database role used for requests that arrive without a JWT or without a `role` claim — the anonymous identity. The danger is granting that role too many privileges: if `db-anon-role` can `SELECT` sensitive tables, anyone can read fund data with no token at all. The mitigation is to grant the anon role nothing (or only truly public views) and require a valid token for everything else.

Q: What status code does PostgREST return when JWT validation fails, e.g. a bad `kid` or wrong `aud`?
A: PostgREST returns `401 Unauthorized` for JWT validation failures, including a missing or mismatched `kid`, a failed audience (`aud`) check, an expired token, or a bad signature. The `401` signals an authentication problem — the caller has not proven who they are. This is distinct from a `403`, which would mean the identity is known but lacks permission.

Q: In PostgREST's model, where does authorization actually happen, and why is that powerful?
A: Authorization happens entirely in PostgreSQL through role privileges and row-level security policies, not in PostgREST code, because PostgREST just impersonates the JWT's role. This is powerful because the database is a single, auditable enforcement point that applies the same rules to PostgREST, direct SQL, and any other client. The gotcha is that you must actually write the `GRANT`s and RLS policies — PostgREST grants nothing by default.

Q: How would you use PostgreSQL row-level security so a transfer-agency consumer sees only its own funds?
A: You enable RLS on the underlying table/view and write a policy whose `USING` clause filters rows by a value derived from the request's role or a JWT claim, for example matching `management_company_id` to a claim exposed via `current_setting`. Because PostgREST runs the request as that role with the claim available, the policy automatically restricts every query, list, and filter. The consumer cannot bypass it because the filter is enforced by the database, not by trusting a client-supplied parameter.

Q: When choosing FastAPI instead of PostgREST for the serving layer, what do you gain and lose?
A: FastAPI gives you full control: custom business logic, request validation with Pydantic, bespoke auth, computed fields, and composition across multiple data sources, plus auto-generated OpenAPI docs. You lose PostgREST's zero-code immediacy — you now write and maintain endpoint code. The architect's rule of thumb: use PostgREST when the API is essentially "a view over a view," and reach for FastAPI when you need real logic, orchestration, or non-trivial transformations.

Q: What is "edge caching" at the gateway and which 3.3.4 decision does it apply?
A: Edge caching means the gateway stores and serves responses for cacheable requests so repeated reads never reach the backend, applying your earlier caching strategy (the 3.3.4 decision) at the API tier. For fund data, a daily NAV that does not change after striking is highly cacheable, so caching it at the gateway cuts database load dramatically. The decision is which responses are cacheable, for how long, and how to invalidate them when a NAV is corrected.

Q: Why is caching NAV data both attractive and dangerous, and how do you reconcile it?
A: It is attractive because an official NAV for a given date is immutable once struck, so it can be cached for a long time with a huge hit rate. It is dangerous because NAVs do get corrected/restated, and serving a stale wrong NAV from cache could mislead a consumer making investment or reporting decisions. You reconcile it by caching with a key that includes the version/correction status and by actively invalidating the cache entry when a correction is published.

Q: What cache controls would you use to make a NAV response safely cacheable at the edge?
A: Use `Cache-Control` with an appropriate `max-age` for stable, struck NAVs and a validator like `ETag` or `Last-Modified` so the gateway can revalidate cheaply with conditional requests. For draft or intra-day values use `Cache-Control: no-store` so they are never cached. The key discipline is that cacheability is a per-resource decision encoded in headers, and a correction event must bust the relevant cache entries.

Q: What is the OWASP API Security Top 10, and which version is current?
A: It is OWASP's ranked list of the most critical API-specific security risks, used as a review checklist; the current edition is the 2023 version, with items labeled `API1:2023` through `API10:2023`. The architect runs the API against this list and either fixes or explicitly risk-accepts each finding. It is API-specific (not the general web Top 10) because APIs fail in distinct ways like object-level authorization and resource consumption.

Q: What is `API1:2023 Broken Object Level Authorization`, and how could it bite the NAV API?
A: Broken Object Level Authorization (BOLA) is when an endpoint trusts a client-supplied object ID without verifying the caller is allowed to access that specific object. On the NAV API it would bite if `/funds/{id}/navs` returned any fund's NAVs to any authenticated caller just because they guessed an `id`. The fix is to enforce per-object access in the database (RLS) or service layer so the caller's identity, not the URL, decides what they can read.

Q: What is `API2:2023 Broken Authentication` in API terms?
A: Broken Authentication covers flaws in how the API verifies identity: weak or missing token validation, accepting unsigned or expired JWTs, no signature verification against the issuer's keys, or credentials in URLs. In this stack it is mitigated by Kong validating Keycloak-issued OIDC tokens (signature, `exp`, `aud`, `iss`) at the gateway. A classic failure is a backend that trusts any token the gateway forwards without the gateway actually verifying it.

Q: What is `API3:2023 Broken Object Property Level Authorization`, and give a fund-data example?
A: This risk combines excessive data exposure and mass assignment: the API returns or accepts object properties the caller should not see or set. A fund-data example is the NAV response leaking an internal `cost_basis` or `pre_publication_draft` column, or a write endpoint letting a consumer set `is_official`. The fix is to project only the public columns in the gold view and never bind client input to sensitive fields.

Q: What is `API4:2023 Unrestricted Resource Consumption`, and which gateway controls address it?
A: It is the risk that a client can consume disproportionate resources — huge result sets, no pagination, unbounded query complexity, or no throttling — leading to denial of service or runaway cost. Pagination on `/navs`, query-complexity/size limits, and Kong's rate limiting and quotas directly address it. The lesson's `429` enforcement demo is precisely a control against this item.

Q: What is `API5:2023 Broken Function Level Authorization`?
A: Broken Function Level Authorization is when a caller can invoke operations or endpoints their role should not reach — for example a standard consumer hitting an admin-only route or an unauthorized HTTP verb. It is enforced by ensuring each function/endpoint checks the caller's role, e.g. the read-only NAV role having no privilege on write paths and the gateway restricting admin routes. The fix is deny-by-default authorization per function, not per object.

Q: What is `API6:2023 Unrestricted Access to Sensitive Business Flows`?
A: This risk is about business logic abused through the API even when each individual request is technically authorized — for example scraping the entire NAV history or automating a flow at a scale that harms the business. Mitigations include rate limiting, anomaly detection, quotas, and device/consumer fingerprinting that make bulk automated abuse expensive. It recognizes that "allowed per request" is not the same as "allowed at scale."

Q: What is `API7:2023 Server Side Request Forgery (SSRF)` and when does it apply to a data API?
A: SSRF is when an API takes a client-supplied URL or reference and fetches it server-side, letting an attacker reach internal services or cloud metadata endpoints. It applies if your API accepts a URL parameter (e.g. a webhook target, an external lookup, or a "fetch from this source" feature). A pure read-only PostgREST-over-gold-view API rarely fetches external URLs, but any enrichment feature that does must validate and allow-list destinations.

Q: What is `API8:2023 Security Misconfiguration` in an API/gateway context?
A: Security Misconfiguration covers insecure defaults and operational mistakes: permissive CORS, verbose error messages leaking internals, missing TLS, default credentials, unnecessary HTTP methods enabled, or an over-privileged `db-anon-role`. It is the catch-all for "the components are fine but configured wrong." The fix is hardened defaults, least privilege, minimal error detail to clients, and reviewing the actual gateway and PostgREST configuration against a checklist.

Q: What is `API9:2023 Improper Inventory Management`, and how does the OpenAPI doc help?
A: Improper Inventory Management is losing track of which API versions, endpoints, and environments exist — the forgotten `v1` still live, the undocumented debug endpoint, the staging API exposed to the internet. A version-controlled OpenAPI document plus a gateway that fronts all traffic gives you an authoritative inventory of what is exposed. This is why publishing and maintaining the spec is a security control, not just documentation.

Q: What is `API10:2023 Unsafe Consumption of APIs`?
A: This risk flips the perspective: it is about your API trusting data it consumes from third-party/upstream APIs without validation, applying weaker scrutiny to "internal" or partner sources than to end users. For a fund platform, blindly trusting a market-data or pricing feed and passing it through unvalidated would qualify. The fix is to validate, sanitize, and rate-limit data coming from upstream sources as rigorously as data from clients.

Q: Which OWASP API risk number one is most directly mitigated by PostgreSQL row-level security, and why?
A: `API1:2023 Broken Object Level Authorization` is most directly mitigated by RLS, because RLS makes the database itself filter every query to only the rows the caller's role/claims permit. Since PostgREST runs each request as the caller's role, an attacker cannot read another management company's funds even by guessing IDs — the policy excludes those rows. This pushes object-level authorization to a single enforceable layer rather than scattering checks across endpoint code.

Q: When does GraphQL "earn its complexity," and why is it usually overkill for fund data?
A: GraphQL earns its complexity when there are many diverse consumers needing flexible, shape-shifting reads across a richly connected graph, so they can request exactly the fields and relations they need in one round trip. It is usually overkill for fund data because the access patterns are stable and tabular — funds, share classes, NAVs — which REST plus filtering serves cleanly. The lesson keeps GraphQL (2.6.2) at T3 precisely because it rarely pays off here and adds caching, auth, and query-cost-control burdens.

Q: What new problems does GraphQL introduce that REST gateways already solved?
A: GraphQL collapses everything onto a single endpoint and lets clients compose arbitrary queries, which breaks per-endpoint rate limiting, HTTP caching, and simple authorization — you instead need query-cost analysis, depth limiting, and field-level auth. It also makes the OWASP "unrestricted resource consumption" risk sharper because one query can fan out enormously. REST's resource-per-URL model gives you caching and throttling almost for free, which is why it is the default for stable data.

Q: What is an ISIN and why is it the natural identifier for a share-class resource?
A: An ISIN (International Securities Identification Number) is a 12-character code (2-letter country prefix, 9 alphanumeric NSIN, and a check digit) that uniquely identifies a security, including a fund share class. It is the natural key for the share-class resource because consumers already use it to reference instruments, so `/share-classes/{isin}` is meaningful and stable. A common gotcha is treating the ISIN as identifying the whole fund — it identifies one share class, so the umbrella fund needs its own identifier.

Q: How would you validate an ISIN check digit, and why bother in the API layer?
A: The check digit is computed with the Luhn (mod-10) algorithm over the digit-expanded code, so you can validate it before trusting an inbound ISIN. Bothering at the API layer turns a typo into a clean `400 Bad Request` with a clear message instead of a confusing empty `404` or a database lookup miss. It is cheap input validation that improves both the error contract and protection against malformed-input abuse.

Q: What is a LEI and where might it appear in the fund-data API contract?
A: A Legal Entity Identifier (LEI) is a 20-character alphanumeric code identifying a legally distinct entity, such as the fund's management company or the umbrella legal entity. It might appear as a property on the fund or management-company resource, or as a filter (`/funds?lei=...`). Exposing it lets regulated consumers reconcile your fund data against entity registries; it is reference data, not sensitive, so it is safe to publish.

Q: Why is a NAV endpoint a high-value target that deserves extra gateway discipline?
A: The NAV is the official price at which fund units transact, so a tampered, leaked-early, or denial-of-serviced NAV endpoint has direct financial and regulatory consequences. The lesson frames skipping gateway discipline as "an OWASP-API-Top-10 incident waiting on the NAV endpoint." Extra discipline — strict auth, quotas, caching with correction-busting, and least-privilege reads — is proportionate to that blast radius.

Q: How would you design pagination for the NAV time series so a consumer can reliably page through history?
A: For a large, append-mostly time series, prefer keyset (cursor) pagination ordered by `nav_date` (and a tiebreaker) so each page says "give me rows after this date," which stays correct as new NAVs are inserted. Offset/`Range` pagination is simpler but can skip or repeat rows if data shifts between page requests. Whichever you choose, document the ordering and the cursor/range mechanism in the OpenAPI spec so consumers page deterministically.

Q: Why is stable, documented sort order essential for a paginated NAV endpoint?
A: Pagination only makes sense relative to a deterministic order; without an explicit `ORDER BY` the database may return rows in any order, so page 2 could overlap or skip page 1. For NAVs you would order by `nav_date` with a stable tiebreaker (e.g. `share_class_id`) so paging is reproducible. Documenting the default order in the contract prevents consumers from assuming an order that the engine does not guarantee.

Q: A consumer reports they "only get 20 NAV rows no matter what" from PostgREST — what is the first thing to check?
A: First check the PostgREST `db-max-rows` configuration (and any default `Range`/limit), because it caps the rows returned per request to protect the database, so the consumer is hitting that cap and must page with the `Range` header. Confirm by inspecting the `Content-Range` response header, which will show the slice and total. This is expected back-pressure, not a bug — the fix is to teach the consumer to paginate, not to remove the cap.

Q: A partner gets `429` on every call almost immediately — what are the first things to check?
A: Check whether the partner is in the wrong consumer tier (a free/low tier with a tiny limit), whether the rate-limit window/policy is misconfigured (e.g. per-second instead of per-minute), and whether multiple partners are sharing one credential so their traffic aggregates against a single limit. Inspect the `RateLimit-Limit`/`RateLimit-Remaining` headers and the gateway logs to see which limit fired. Often the fix is correct credential-to-consumer mapping or moving them to the right tier.

Q: A consumer's JWT is rejected with `401` at the gateway but "works in their app" — what do you investigate?
A: Investigate the token's `aud` (audience) and `iss` (issuer) claims against what Kong is configured to accept, the signing key/JWKS (a rotated Keycloak key not yet picked up), and clock skew causing premature `exp` failures. "Works in their app" usually means it is a valid token but for a different audience or against a different realm. Comparing the decoded token claims to the gateway's expected issuer/audience/key id resolves most of these.

Q: A consumer gets `401` from PostgREST even though Kong accepted their token — what is the likely cause?
A: The likely cause is that PostgREST is validating the JWT independently and its configured secret/JWKS, audience, or `role` extraction does not match the token, or the gateway forwarded a token PostgREST does not trust. Either align PostgREST's JWT config with Keycloak, or terminate auth at the gateway and have Kong inject a token/role PostgREST does trust. Mismatched audiences and a missing `role` claim are the usual culprits.

Q: A consumer sees fewer columns than the documentation promises — where do you look first?
A: Look first at the gold view definition behind the endpoint, because PostgREST exposes exactly the view's columns, so a missing column means the view does not select it (or the role lacks column privileges). Also check column-level `GRANT`s, since a role can be granted `SELECT` on only some columns. The OpenAPI doc must be regenerated from the actual view, so a drift between docs and reality usually traces to an un-republished spec.

Q: How does the gold view act as a security boundary in addition to a contract boundary?
A: The view exposes only the columns and rows you choose to publish, so it physically prevents the API from returning internal or sensitive fields even if a query asks for them. Granting the API role `SELECT` on the view (and not on base tables) means a vulnerability in the API cannot reach unexposed data. It thus enforces both `API3` (property-level exposure) and the principle of least privilege at the database, not just in application code.

Q: Why should error responses to external consumers be deliberately terse?
A: Verbose errors that leak stack traces, SQL, table names, or internal hostnames feed `API8:2023 Security Misconfiguration` and help an attacker map your system. External consumers need a stable error code, a safe message, and the right status — not your internals. The pattern is to log full detail internally for debugging while returning a sanitized, contract-defined shape to the client.

Q: How does CORS configuration become a security concern for a data API?
A: CORS controls which web origins a browser will let call your API, so a permissive `Access-Control-Allow-Origin: *` on an authenticated NAV API can expose it to scripts on any site under `API8` misconfiguration. You should allow only the specific origins of legitimate browser consumers and never reflect arbitrary origins with credentials enabled. For server-to-server partners CORS is irrelevant, so do not loosen it just to satisfy one browser client.

Q: Why is TLS (HTTPS) non-negotiable for the gateway in a regulated fund context?
A: Without TLS, tokens, NAV data, and consumer identifiers traverse the network in clear text, exposing credentials and confidential fund information to interception — a direct breach of confidentiality and of regulatory expectations like DORA on ICT security. TLS termination at the gateway encrypts the consumer-facing channel and is part of basic security configuration. Serving any authenticated API over plain HTTP would be an immediate, indefensible finding.

Q: How do API quotas and usage logs support DORA-style ICT risk and resilience expectations?
A: DORA emphasizes operational resilience, monitoring, and control over ICT services and third parties; gateway quotas provide back-pressure that prevents one consumer from degrading the platform, and usage logs provide the auditable record of who accessed what and how much. Together they demonstrate that access is bounded, attributed, and observable. This is the kind of evidence a regulator or internal audit expects for a critical data service.

Q: How does the gateway help with auditability and incident response for the NAV API?
A: Because all traffic flows through one point, the gateway can log every request with consumer identity, timestamp, endpoint, status, and rate-limit decisions, giving a single authoritative audit trail. During an incident you can answer "who called the NAV endpoint, when, and how often" without correlating scattered service logs. That centralized, attributable log is both a security and a compliance asset.

Q: Why is letting operational consumers hit the database directly the failure mode this lesson warns against?
A: Direct database access gives no quotas, no usage attribution for chargeback, no version isolation, and couples consumers to your internal schema — exactly the situation the gateway exists to prevent. One heavy consumer can then starve everyone, you cannot bill or throttle anyone, and any refactor breaks people silently. The API-plus-gateway pattern interposes a governed, observable, throttleable contract instead.

Q: What does "the API is the product surface" imply for backward compatibility commitments?
A: If the API is the product, then its contract is a promise consumers build businesses on, so you owe them backward compatibility within a version and a managed deprecation path across versions. You cannot change field meanings or remove endpoints on a whim, because real partner systems depend on them. This product mindset is what forces the versioning policy, OpenAPI contract, and deprecation discipline.

Q: How would you prove an external consumer can onboard from the contract alone, per the "Done when"?
A: You hand the OpenAPI document and any contract notes to someone who has never seen the system and confirm they can authenticate, call `/funds`, `/share-classes`, `/navs`, paginate, filter, and interpret errors without asking you questions. Success means the spec's descriptions, examples, auth scheme, and error shapes are complete enough to self-serve. Any question they have to ask you is a gap to fix in the document.

Q: What is the difference between a `400` and a `422` response, and how would you use each on the NAV API?
A: `400 Bad Request` signals the request is malformed or unparseable (bad syntax, invalid query operator), while `422 Unprocessable Entity` signals the syntax is fine but the content fails validation rules (e.g. `nav_date` after today, or an ISIN failing the check digit). Using them consistently lets consumers distinguish "I sent garbage" from "I sent a well-formed but invalid request." Document which conditions trigger which, so the error contract is predictable.

Q: When is `404` the right response versus `403` on the NAV API, and what is the privacy nuance?
A: `404 Not Found` means the resource does not exist (or you choose to hide its existence), while `403 Forbidden` means it exists but the authenticated caller may not access it. The privacy nuance is that returning `403` can confirm a resource exists to an unauthorized caller, so for sensitive objects you may deliberately return `404` to avoid leaking existence. Choose one policy and apply it consistently to avoid an enumeration oracle.

Q: What does `409 Conflict` indicate, and would a read-only NAV API ever return it?
A: `409 Conflict` indicates the request conflicts with the current state of the resource, typically on writes (e.g. a version/concurrency clash). A strictly read-only NAV API generally would not return `409` because there is no state to conflict with. If you later add write paths (e.g. publishing a NAV), `409` becomes relevant for concurrent-update protection.

Q: Why is idempotency a contract concern for any write endpoints you might add later?
A: Idempotency means repeating the same request yields the same result without side effects, which matters because networks retry — a consumer that resubmits a "publish NAV" after a timeout must not create duplicates. `GET` is naturally idempotent; for unsafe methods you provide an idempotency key so retries are de-duplicated. Even on a read-only API today, documenting that all operations are safe and idempotent reassures consumers their retries are harmless.

Q: How does content negotiation (the `Accept` header) fit a fund-data API?
A: Content negotiation lets a client request a representation via `Accept`, e.g. `application/json` for programmatic consumers or `text/csv` for analysts pulling NAV history into a spreadsheet — PostgREST supports CSV output, for instance. Honoring `Accept` keeps one resource URL serving multiple formats instead of inventing `/navs.csv`. It also intersects with header-based versioning when you use vendor media types.

Q: Why must the OpenAPI document live in the same version control and review flow as the code?
A: Keeping the spec in version control makes every contract change a reviewable diff tied to a release, so a reviewer (and a contract test) can catch a breaking change before it ships. It also keeps the published contract and the deployed behavior in lockstep, preventing the "docs say one thing, API does another" drift. This directly mitigates `API9` improper inventory management by making the inventory authoritative and current.

Q: How can the OpenAPI document drive contract testing, and why is that valuable?
A: The spec defines expected request/response schemas, so you can generate tests (or use a contract-testing tool) that assert the live API conforms, and generate typed clients that fail to compile when the contract changes. This is valuable because it catches accidental breaking changes automatically and gives consumers confidence the documented contract is enforced. It turns the contract from prose into an executable check.

Q: What is the risk of exposing PostgREST's auto-generated OpenAPI without curation?
A: The auto-generated spec mirrors the database schema, so without curation it can reveal internal structure, expose endpoints you did not mean to publish, and ship empty descriptions that make onboarding impossible. It also pins your public contract to internal naming. The mitigation is to expose only deliberate gold views, add PostgreSQL `COMMENT`s and hand-edited descriptions, and review the file before publishing it as the contract.

Q: How do PostgreSQL `COMMENT`s improve the PostgREST-generated OpenAPI?
A: PostgREST incorporates `COMMENT ON` text for tables, views, and columns into the generated OpenAPI descriptions, so adding comments to your gold view and its columns populates the spec with meaningful documentation from the database. This keeps the documentation next to the definition and regenerable. It is the cheapest way to make a generated spec self-explanatory before hand-editing the rest.

Q: Why is "least privilege" the unifying principle across PostgREST roles, gateway tiers, and the gold view?
A: Each layer grants only what is necessary: the API database role gets `SELECT` on one view, consumer tiers get only their entitled rate and quota, and the view exposes only public columns and rows. The principle limits blast radius — a compromise of any single layer yields minimal access. It is the through-line connecting `API1`, `API3`, `API5`, and `API8` mitigations into one coherent design stance.

Q: What is the difference between authentication and authorization, mapped to this stack?
A: Authentication proves who the caller is — handled by Keycloak issuing tokens and Kong validating them (`401` on failure). Authorization decides what that authenticated caller may do — handled in PostgreSQL via role grants and RLS (`403`/empty result on denial). Keeping them in distinct layers means you can reason about identity and permission independently, and the gateway can reject unauthenticated traffic before it ever reaches authorization logic.

Q: Why is it dangerous to let consumers filter on arbitrary columns, and how do you constrain it?
A: Allowing arbitrary filtering can expose unindexed columns to slow full scans (resource consumption) and can let a consumer infer sensitive values by probing filters even if those columns are not displayed. You constrain it by exposing a curated view with only public, appropriately indexed columns, and by documenting the supported filters in the contract. The gold view thus bounds both performance and information exposure of the filter surface.

Q: How does indexing on the gold view's underlying tables affect API performance and safety?
A: Filters and sorts the API offers (e.g. by `fund_id`, `nav_date`) must be backed by indexes, or each request triggers a sequential scan that is slow and amplifies `API4` resource-consumption risk under load. Proper indexes keep paginated, filtered reads cheap and predictable. The architect's gotcha is that exposing a filterable column you have not indexed is a latent denial-of-service surface.

Q: What is a "noisy neighbor" problem on a shared data API, and how does the gateway address it?
A: A noisy neighbor is one consumer whose heavy traffic degrades performance for everyone sharing the backend. The gateway addresses it with per-consumer rate limits and quotas so one client's burst returns `429` to them rather than slowing others, and with usage analytics to identify the offender. Edge caching further insulates the shared backend from repeated identical reads.

Q: Why is per-consumer attribution impossible if everyone shares one API credential?
A: If many consumers authenticate with the same key, the gateway sees one identity and cannot separate their usage, so rate limits aggregate them and logs cannot attribute traffic for chargeback or incident response. Each consumer (or tier) must have its own credential mapped to its own gateway consumer object. Shared credentials also undermine `API2` authentication and make revocation all-or-nothing.

Q: How would you stage a breaking change so consumers are not surprised, end to end?
A: Announce the change with a timeline; ship the new behavior under a new version (`/v2` or new media type) while keeping `/v1` running; emit deprecation signals (e.g. a `Deprecation`/`Sunset` header on `/v1`); provide a migration guide and updated OpenAPI; monitor `/v1` usage in gateway logs until it drops to zero or the sunset date; then retire `/v1`. This satisfies the "show where a breaking change would surface for consumers" done-when by making the surface explicit and observable.

Q: Where, concretely, does a breaking change "surface" for consumers in this architecture?
A: It surfaces in the OpenAPI document diff (a removed/renamed field or a new version), in the URL or media type (`/v2` or a new vendor type), and operationally in deprecation headers on the old version plus its declining usage in gateway logs. The architect must be able to point at each of these. If a breaking change could ship without surfacing in any of them, the versioning policy is broken.

Q: What is the role of mock servers generated from OpenAPI in consumer onboarding?
A: A mock server generated from the OpenAPI spec returns example responses for each endpoint, letting a consumer build and test their integration before the real API or their credentials are ready. This decouples onboarding from backend availability and proves the contract is complete enough to code against. If the mock is unusable, the spec has gaps — which is itself useful feedback.

Q: Why does the architect care about the "shape" of pagination metadata being consistent across endpoints?
A: If `/funds`, `/share-classes`, and `/navs` each report paging differently, consumers must write three paging implementations and are likely to get one wrong. A consistent mechanism — the same `Content-Range`/cursor convention everywhere — lets a consumer write one generic pager. Consistency is a contract-quality property the architect enforces, exactly as with the consistent error shape.

Q: How do you decide between exposing a database view via PostgREST and building a FastAPI endpoint for a given resource?
A: If the resource is essentially a projection/filter over existing tables with no business logic, PostgREST over a view is the fastest, lowest-maintenance choice. If it needs computed fields, multi-source composition, custom validation, side effects, or bespoke auth flows, FastAPI is warranted despite the extra code. The architect mixes them: PostgREST for the straightforward read surfaces, FastAPI where logic genuinely lives.

Q: Why is "instant serving" a double-edged sword that still needs the gateway and view discipline?
A: PostgREST turning a view into a REST API in minutes is powerful, but "instant" also means it is easy to accidentally expose internal tables, skip auth, or ship an undocumented contract. The discipline — least-privilege roles, a curated gold view, a curated OpenAPI doc, and a gateway for auth/quotas — is what turns a quick demo into a production product surface. Speed without that discipline is precisely the OWASP-incident risk the lesson warns about.

Q: What is a "certified" or contract-tested dataset in the context of a published data API, and why does it matter to fund consumers?
A: A contract-tested dataset is one whose API representation is guaranteed by automated tests to conform to the published OpenAPI schema, so consumers can trust the field names, types, and semantics will not silently drift. It matters to fund consumers because they feed your NAV and reference data into regulated reporting (e.g. EMT/EPT files, board packs) where an unexpected null or renamed field can corrupt a downstream filing. The architect's discipline is that what is published is exactly what is contract-tested, with no undocumented behavior.

Q: How does an EMT or EPT file consumer benefit from a stable, versioned NAV API rather than a nightly file drop?
A: EMT (European MiFID Template) and EPT (European PRIIPs Template) producers need current, validated NAVs and share-class reference data, and a versioned REST API lets them pull exactly the funds they cover, on demand, against a stable contract — instead of parsing a brittle nightly file whose layout can change without warning. The API's pagination and filtering let them fetch only their in-scope ISINs, and versioning guarantees a field they map will not vanish mid-cycle. If a breaking change is needed, the deprecation path gives them time to re-map before the template generation breaks.

Q: Summarize the end-to-end request path for an external NAV read in this architecture.
A: The consumer authenticates with Keycloak to obtain an OIDC/JWT token, then calls `https://gateway/.../navs?...` with the token; Kong validates the token, applies the consumer's rate limit/quota (returning `429` if exceeded), optionally serves a cached response, and otherwise forwards to PostgREST. PostgREST sets the role from the JWT, runs the query against the gold view under RLS/grants, paginates per the `Range` header, and returns `200`/`206` with `Content-Range`. Kong logs the attributed request for usage analytics, and the consumer reads the JSON against the OpenAPI contract.
