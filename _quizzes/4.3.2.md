# Phase 5 · 4.3.2 Cloud query services — 100 self-test questions

Q: What is a "cloud query service" / serverless SQL engine in the lakehouse sense?
A: It is a query engine that runs SQL directly over files in object storage without any always-on warehouse or pre-loaded tables, billing you per query rather than per running hour. Azure Synapse serverless SQL pool, Amazon Athena, and self-hosted Trino are the canonical examples. The defining trait is that compute is decoupled from storage and you pay only when a query runs, which is why people call it the cheap-and-cheerful consumption tier.

Q: What does "serverless" actually mean here — is there really no server?
A: There are servers, but you never provision, size, or pay for them while idle; the cloud allocates compute on demand for the duration of each query and tears it down afterward. You are billed on a usage metric (data scanned), not on instance hours. The mental shift is that capacity planning becomes data-layout planning, because the meter runs on bytes read, not on a cluster you keep warm.

Q: What is the headline pricing model for Azure Synapse serverless SQL pool?
A: You pay per terabyte of data processed, currently `$5` per TB, where "processed" means bytes read from storage to satisfy the query. There is no charge for the engine sitting idle because there is no provisioned engine. This is the same per-TB-scanned shape as Amazon Athena, which makes the two directly comparable for cost reasoning.

Q: In serverless SQL, what does "data processed" or "data scanned" mean precisely?
A: It is the volume of bytes the engine reads from object storage to answer the query, not the bytes returned to the client. A query that reads `1 TB` of Parquet and returns ten rows still costs for the full terabyte scanned. This is the single most important billing fact: your bill tracks input read, so reducing what gets read is the entire game.

Q: Does the result set size affect the bill in per-TB-scanned pricing?
A: No — only the data read from storage is metered, so returning ten rows or ten million rows from the same scan costs the same. This is counter-intuitive for people used to row-based or egress-based pricing. The lever you control is not output volume but how few bytes the engine must read, via partition pruning and columnar formats.

Q: What is the minimum charge per query in Synapse serverless and Athena?
A: Each query has a `10 MB` minimum data-processed charge, and the amount processed is rounded up to the nearest `1 MB`. So even a trivial query touching a few kilobytes bills as if it read `10 MB`. This minimum makes a flood of tiny ad-hoc queries surprisingly non-free, which matters for dashboards that fire many small queries.

Q: Are DDL statements like `CREATE EXTERNAL TABLE` charged in serverless SQL?
A: No — metadata-only operations such as `CREATE`, `ALTER`, and `DROP` of external tables and views do not read data files, so they incur no data-processed charge. Only statements that actually scan storage are billed. This is why you can freely define and redefine external tables and views without worrying about cost.

Q: Are failed queries billed in Synapse serverless and Athena?
A: No — queries that fail (for example, a syntax error or a permission failure before any scan) are not charged the data-processed fee. Likewise, cancelled queries generally do not bill for the scan. However, a query that fails partway after already reading data may still have read bytes, so the safe rule is "no charge for failures that never scanned."

Q: Why is it said that in serverless SQL "file layout literally is the cost model"?
A: Because the bill is bytes read, and bytes read is determined entirely by how your files are partitioned, formatted, and sized, the physical layout of the gold layer directly sets the per-query cost. A badly partitioned layout forces full scans on every filter, turning a cheap lookup into a four-figure bill. The architect who signs off file layouts is effectively signing off the consumption tier's cost structure.

Q: What is `OPENROWSET` in Synapse serverless SQL pool?
A: `OPENROWSET` is a table-valued function that reads files in storage in place and exposes them as a rowset you can query, without first defining a table. You point it at a path with `BULK` and a `FORMAT` (for example `'PARQUET'`, `'CSV'`, or `'DELTA'`). It is the ad-hoc entry point — you can explore a file with one statement, no schema setup required.

Q: Show the basic shape of an `OPENROWSET` query over Parquet in Synapse serverless.
A: It looks like `SELECT * FROM OPENROWSET(BULK 'https://acct.dfs.core.windows.net/container/path/*.parquet', FORMAT='PARQUET') AS rows`. The `BULK` argument is the storage path (wildcards allowed), and `FORMAT` tells the engine how to parse. You can append a `WITH (...)` clause to pin column names and types instead of relying on schema inference.

Q: What does the `WITH` clause add to an `OPENROWSET` call and why use it?
A: `WITH` lets you declare the exact column names, types, and lengths the engine should use instead of inferring them from the file. It matters because inference can over-size types — for example Parquet has no max length metadata, so a string infers as `varchar(8000)`, which wastes memory and slows sorts and joins. Pinning `varchar(20)` instead reduces row width and improves performance and concurrency.

Q: What is an external table in serverless SQL and how does it differ from `OPENROWSET`?
A: An external table is a persisted metadata object that names a storage location, file format, and schema, so consumers query it like a normal table without repeating the path. `OPENROWSET` is inline and ad-hoc; an external table is reusable and discoverable in the catalog. Both read the same underlying files in place — neither copies data into the engine.

Q: What three objects do you typically create to define an external table in Synapse serverless?
A: You create an `EXTERNAL DATA SOURCE` (the storage location and credential), an `EXTERNAL FILE FORMAT` (for example Parquet), and then the `EXTERNAL TABLE` itself binding a schema to a `LOCATION` under that data source. The data source and file format are reusable across many tables. This separation lets you change credentials or format definitions in one place.

Q: Why convert a CSV gold table to Parquet before querying it serverlessly?
A: Parquet is a compressed columnar format, so files are smaller and the engine reads only the columns a query references rather than every byte of every row. A `SELECT col1 FROM ...` over Parquet reads roughly one column's worth of bytes, while the same over CSV must read the entire file. Both the smaller size and column pruning cut data-processed directly, which cuts the bill.

Q: How does columnar storage reduce data scanned compared to row storage?
A: Columnar formats store each column's values contiguously, so the reader can skip the file segments for columns the query does not select. Row formats interleave all columns per row, forcing the engine to read everything to extract one field. Selecting four of forty columns from Parquet can read roughly a tenth of the bytes, which is a tenth of the cost.

Q: What is a row group in Parquet and why does it matter for serverless cost?
A: A row group is a horizontal partition of rows within a Parquet file, each carrying per-column min/max statistics in its footer. The engine reads those statistics and skips entire row groups whose min/max range cannot satisfy the `WHERE` predicate, reading less data. This is the file-internal layer of pruning that complements folder-level partition pruning.

Q: What is predicate pushdown in serverless SQL over Parquet?
A: Predicate pushdown means the engine evaluates `WHERE`-clause filters against Parquet row-group statistics before reading the data, skipping row groups that cannot match. Because skipped row groups are never read, they are never billed, so pushdown reduces both latency and cost. It works automatically for numeric and date predicates when the file carries the relevant statistics.

Q: What collation is required for predicate pushdown on character columns in Synapse serverless Parquet?
A: Predicate pushdown for string columns in Parquet is supported only with the `Latin1_General_100_BIN2_UTF8` collation, specified per column in the `WITH` clause. If you do not set it, the database collation is used and string filters may not push down, so the engine reads more row groups than necessary. This is a classic silent cost leak on string-filtered queries.

Q: A string-filtered Synapse query scans far more than expected. What is the first thing to check?
A: Check the collation on that string column — predicate pushdown for character columns in Parquet only kicks in under `Latin1_General_100_BIN2_UTF8`. Without it, row-group skipping on the string predicate is disabled and the engine reads everything. Set the collation in the `WITH` clause for the filtered column and re-measure the data processed.

Q: What is the `filepath()` function in Synapse serverless and what is it for?
A: `filepath()` returns part of the storage path of the file a given row came from, letting you filter on folder-encoded partition values such as a date or ISIN folder. `filepath(1)` returns the value matched by the first wildcard in the path, `filepath(2)` the second, and so on. Filtering on it prunes whole folders so the engine never lists or reads files outside the requested partitions.

Q: What does the `filename()` function return?
A: `filename()` returns the name of the source file for each row, which is useful for filtering to specific files or for auditing which file a record came from. Like `filepath()`, it lets you target a subset of the data and reduce what gets read. A common use is restricting a query to one day's drop file by name pattern.

Q: How does partitioning by folder enable partition elimination in serverless SQL?
A: When data is laid out as `.../puYear=2024/puMonth=03/...`, a query that filters on `filepath(1) = 2024 AND filepath(2) = 3` makes the engine list and read only the matching folders. Folders outside the filter are never opened, so their bytes are never scanned or billed. This folder-level elimination is the coarsest and usually largest cost lever.

Q: Why should you push wildcards as low as possible in the storage path?
A: Serverless SQL lists files starting from the first asterisk in the path using the storage API, then eliminates non-matching files. The more of the path you specify literally before the first wildcard, the smaller the initial file list and the faster and cheaper the listing and scan. Putting the wildcard high (early) forces a large listing the engine must then filter.

Q: A query filters on a partition column but still does a full scan. What is the likely cause?
A: The filter is probably applied to a regular data column rather than to `filepath()`/`filename()`, or the folder layout does not encode that value, so partition elimination cannot trigger. Serverless SQL prunes folders only when the predicate references the path functions tied to the wildcard positions. Rewrite the filter against `filepath(n)` matching the partition folder to enable elimination.

Q: Are `filepath()` and `filename()` supported on external tables in Synapse serverless?
A: Generally no — partition-elimination via `filepath()`/`filename()` is not supported on external tables, except those auto-created for Apache Spark tables. To get partition pruning with ad-hoc paths, use `OPENROWSET` with wildcards and the path functions, or use partitioned views. This is a real gotcha: people define an external table expecting pruning and instead get full scans.

Q: What is a partitioned view in Synapse serverless and why use one?
A: A partitioned view is a view built over `OPENROWSET` with wildcards and `filepath()` predicates so that consumers query a single named object while the engine still prunes folders. It gives the convenience of an external table plus the pruning that external tables lack. It is the recommended pattern when you have partitioned files and want both discoverability and cost control.

Q: What file size range does Synapse serverless recommend, and why does it matter for cost?
A: Microsoft recommends keeping files roughly between `100 MB` and `10 GB`. Too-small files create excessive storage-listing and per-file overhead (the small-files problem), while too-large single files cannot be parallelized well. Right-sized files keep both the listing overhead and the per-query latency reasonable without inflating scanned bytes.

Q: What is the "small files problem" and how does it bite serverless query cost?
A: It is having thousands of tiny files instead of a few right-sized ones, which forces the engine to make many storage requests and list operations, adding latency and overhead per query. While the metered bytes may be similar, the per-file overhead and the `10 MB`-per-query minimum on any tiny ad-hoc reads degrade efficiency. Compaction into `100 MB`–`10 GB` files fixes it.

Q: Why prefer equally sized files within one `OPENROWSET` path or external-table `LOCATION`?
A: Equal-sized files let the engine parallelize work evenly across them, avoiding a straggler where one giant file dominates the query time. Skewed file sizes mean some workers finish early while one grinds through an oversized file. Even sizing improves throughput without changing the bytes scanned.

Q: Why does Synapse serverless rely on statistics, and how are they created?
A: The optimizer uses column statistics (cardinality, distribution) to choose good join orders and plans, which affects how much data is read and how fast the query runs. For Parquet and CSV with parser 2.0, statistics are auto-created via sampling; for CSV with parser version 1.0, sampling is unsupported so auto-stats may not happen. In those cases you create statistics manually on columns used in `JOIN`, `WHERE`, `GROUP BY`, `ORDER BY`, and `DISTINCT`.

Q: When should you create statistics manually in Synapse serverless?
A: When automatic statistics are not created — notably CSV read with parser version 1.0, where sampling is not supported — you should manually create statistics on the columns the optimizer needs. Target columns used in joins, filters, grouping, ordering, and distinct operations. Good statistics let the optimizer avoid plans that read or shuffle more data than necessary.

Q: What is `PARSER_VERSION` in Synapse serverless and which should you use for CSV?
A: `PARSER_VERSION` selects the CSV parsing engine; version `2.0` is the performance-optimized parser and is preferred for most CSV reads. Parser `1.0` lacks features like sampling-based automatic statistics. Using `2.0` improves CSV read performance and enables auto-stats, though Parquet remains the better choice for cost.

Q: What is CETAS in Synapse serverless and how does it help cost?
A: CETAS (`CREATE EXTERNAL TABLE AS SELECT`) runs a `SELECT` and writes the results to a new set of Parquet files in storage, plus external-table metadata. You use it to materialize expensive, frequently repeated joins or aggregations once, then query the cheap pre-computed Parquet instead of re-scanning raw data every time. Statistics are auto-created on the CETAS output on first query, further helping subsequent reads.

Q: A nightly report repeats the same costly join across raw files. How would you cut its serverless cost?
A: Materialize the join once with CETAS into partitioned Parquet, then point the report at that external table so each run scans the small pre-joined output instead of re-reading raw inputs. This trades a one-time write cost for cheap repeated reads. It is the serverless analogue of a materialized view, justified when query frequency times scan size is high.

Q: When does serverless SQL beat keeping a warehouse running?
A: When query frequency is low to moderate and scans are well-bounded, serverless wins because you pay nothing while idle and a modest per-TB fee when you run. A warehouse that sits mostly idle but bills by the hour is wasteful for sporadic ad-hoc analytics. The crossover is governed by query frequency multiplied by scan size versus the warehouse's hourly run cost.

Q: When does serverless SQL bankrupt you compared to a running warehouse?
A: When you run many queries, frequently, each scanning large volumes — high frequency times large scan size — the per-TB meter accumulates faster than a warehouse's flat hourly rate, especially if layout forces full scans. Dashboards refreshing every minute over an unpartitioned multi-terabyte table are the classic disaster. At that point a provisioned, cached warehouse is cheaper and faster.

Q: Express the serverless-versus-warehouse crossover as a rule of thumb.
A: Compare expected monthly serverless cost (queries per month times average TB scanned times the per-TB rate) against the monthly cost of a right-sized always-on or auto-suspending warehouse. If the product of frequency and scan size is small, serverless wins; if it is large, a warehouse with caching wins. The lever that moves serverless cost down is reducing average TB scanned through layout.

Q: How does good partitioning change the serverless-versus-warehouse crossover?
A: By slashing average bytes scanned per query, good partitioning pushes the crossover far to the right, letting serverless stay cheaper even at higher query volumes. A query that scans `50 GB` instead of `15 TB` because of date and ISIN partitioning is roughly three hundred times cheaper. So layout is not just a cost optimization — it changes which engine you should even choose.

Q: What is Amazon Athena and how does it relate to Synapse serverless?
A: Athena is AWS's serverless interactive query service that runs SQL over data in Amazon S3, billed per TB scanned with a `10 MB` minimum per query — the same economic shape as Synapse serverless. It uses the AWS Glue Data Catalog for table metadata. The pattern is identical across clouds: serverless SQL over object storage, priced on bytes read.

Q: What query engine does Amazon Athena descend from?
A: Athena is built on Presto/Trino, the open-source distributed SQL engine; its SQL dialect and execution model derive from that lineage. This is why understanding Trino transfers directly to reasoning about Athena. The "serverless SQL over object store" pattern is essentially managed, metered Trino.

Q: What is Trino?
A: Trino is an open-source distributed SQL query engine designed to query large data sets spread across one or more heterogeneous data sources, separating compute from storage. You run it yourself (for example over MinIO or HDFS), so it makes the per-query cost trade-off explicit in your own infrastructure rather than hidden behind a cloud meter. Athena and, conceptually, Synapse serverless are managed embodiments of this pattern.

Q: Describe Trino's coordinator-and-worker architecture.
A: A Trino cluster has one coordinator that parses SQL, plans the query, and schedules work, and multiple workers that fetch data from sources and execute query fragments in parallel. The coordinator returns results to the client after aggregating worker output. This MPP design lets a single query fan out across many workers reading object storage concurrently.

Q: What is a connector in Trino?
A: A connector is a plugin that adapts Trino to a specific data source — Hive/object storage, Iceberg, Delta, PostgreSQL, Kafka, and so on — exposing it as schemas and tables Trino can query. Connectors are why one Trino query can join object-storage files against a relational database. The Hive and Iceberg connectors are the usual route to lakehouse files in object storage.

Q: What is a catalog in Trino?
A: A catalog is a named configuration of a connector pointing at a particular data source, for example a `hive` catalog over your MinIO bucket. You reference tables as `catalog.schema.table`, and one Trino instance can host many catalogs at once. This is how Trino federates queries across multiple stores in a single SQL statement.

Q: Why is the Trino-over-MinIO setup pedagogically useful for understanding serverless cost?
A: Because you operate the engine and the storage yourself, you can directly observe how partitioning and Parquet conversion change the bytes read and the query time, with no opaque cloud meter in between. It makes the abstract per-TB pricing of Athena and Synapse concrete. The same layout decisions that speed up your Trino query are the ones that lower your managed serverless bill.

Q: What is the common pattern uniting Synapse serverless, Athena, and Trino?
A: All three are serverless-style SQL over object storage: compute is decoupled from storage, tables are defined as metadata over files read in place, and performance and cost are dominated by how little data the engine must read. Trino is the open engine; Athena is managed Trino on AWS; Synapse serverless is the Azure equivalent. Learning one transfers almost entirely to the others.

Q: Does Trino itself charge per TB scanned?
A: No — Trino is open-source software you run on your own compute, so there is no per-TB meter; you pay for the servers it runs on. The per-TB-scanned pricing is a feature of the managed services (Athena, Synapse serverless) built on or like Trino. But the same data-read efficiency that lowers managed bills also lowers your Trino infrastructure load and latency.

Q: For a gold layer with no partitioning, what is the per-query scan behaviour?
A: Every query that filters on any column must read all files in the table because there is no folder structure to eliminate, so each query scans the entire dataset. A `1 TB` gold table costs roughly `$5` per query regardless of how selective the filter is. This is the baseline against which partitioning savings are measured.

Q: Estimate the cost of one query over an unpartitioned `2 TB` gold table at `$5`/TB.
A: With no partitioning the engine scans the full `2 TB`, costing about `2 × $5 = $10` per query, even if the result is a single row. Columnar Parquet would reduce the bytes via column pruning, but without partitioning every folder is still read. This shows why "none" is the worst of the three partitioning strategies.

Q: Estimate the cost delta when partitioning a `2 TB` gold table by date, querying a single day out of 365.
A: If the data is evenly spread, one day is roughly `2 TB / 365 ≈ 5.6 GB`, costing about `0.0056 × $5 ≈ $0.028` instead of `$10` — over three hundred times cheaper. Partition elimination skips the other 364 day-folders entirely. This single change is usually the largest cost lever available.

Q: A query filters by date and ISIN; how does adding ISIN as a second partition help?
A: Partitioning by `date` then `ISIN` lets a query for one date and one ISIN read only that leaf folder, pruning both the other dates and the other instruments. The scanned bytes shrink to a single instrument-day instead of a whole day. The win is largest for queries that always filter on both dimensions, which fund-data lookups typically do.

Q: What is the risk of over-partitioning, for example by date plus ISIN plus currency?
A: Over-partitioning fragments data into many tiny files, triggering the small-files problem: excessive storage listing, per-file overhead, and files below the efficient `100 MB`–`10 GB` range. Beyond a point, the listing and request overhead outweighs the pruning benefit. Partition on the columns you actually filter on, and stop before files become tiny.

Q: How do you choose partition columns for a serverless gold table?
A: Pick the columns that appear in the `WHERE` clause of the most frequent and most expensive queries, typically a date and one high-selectivity business key, ordered coarse-to-fine. The goal is to maximize folders eliminated per query while keeping leaf files in the `100 MB`–`10 GB` range. Partitioning on a column nobody filters by adds cost without benefit.

Q: Why is high-cardinality partitioning (for example by a unique transaction id) a mistake?
A: It creates one folder per value — potentially millions — each with a tiny file, producing a catastrophic small-files problem and huge listing overhead with little pruning benefit per typical query. The engine spends its time enumerating folders rather than reading data. Reserve partitioning for low-to-moderate-cardinality columns that queries actually filter on.

Q: In a NAV reporting workload, why is partitioning by valuation date especially valuable?
A: NAV queries almost always scope to a date or a date range (today's NAV, month-end NAV), so a `valuationDate` partition lets each report read only the relevant day-folders and skip the rest of the history. Over years of daily NAV data this turns a multi-terabyte scan into a few gigabytes. It aligns the physical layout with the dominant query predicate.

Q: How would you lay out a gold fund-holdings table queried by date and ISIN in serverless SQL?
A: Store it as Parquet partitioned `valuationDate=YYYY-MM-DD/ISIN=.../` with leaf files in the `100 MB`–`10 GB` range, and query via `OPENROWSET` or a partitioned view using `filepath()` on both levels. Date-then-ISIN ordering prunes history first, then narrows to one instrument. This matches the typical holdings lookup and keeps scanned bytes minimal.

Q: An EMT file feed lands daily; how should the gold layer be partitioned for cheap serverless queries?
A: Partition by the reporting date (and possibly by fund or share class), so a query for a given EMT reporting period reads only that period's folder rather than the full multi-year history. EMT (European MiFID Template) data is queried by reporting cut and instrument, which maps cleanly to date and instrument partitions. This keeps regulatory-template queries in the cents-per-query range.

Q: Why is per-TB pricing a governance concern for a regulated fund-admin platform, not just a tech detail?
A: Because an unbounded ad-hoc query against a poorly laid-out regulatory dataset can silently generate a large bill, and because cost predictability and access control are part of operational-risk management under regimes like DORA. The architect must design layouts and guardrails so consumption cost is bounded and attributable. Cost runaway is an operational-resilience and budget-control issue, not merely an efficiency one.

Q: How can you estimate a query's scan size in Synapse serverless before trusting the bill?
A: Run the query and inspect the data-processed metric reported by the service (visible in the Synapse monitoring/query history and via DMVs), which tells you the bytes read. You can also reason a-priori from partition pruning and column selection: estimated bytes ≈ (selected partitions' size) × (selected columns' fraction). Comparing the estimate to the reported metric validates that pruning actually fired.

Q: How would you confirm that partition elimination actually happened for a query?
A: Compare the data-processed reported for the partition-filtered query against a full-scan baseline; if pruning worked, the filtered query reports dramatically fewer bytes. You can also vary the filter and watch the metric scale with the number of partitions selected. If the metric does not drop when you narrow the filter, pruning is not firing — check that you filtered on `filepath()` and not a data column.

Q: A Synapse serverless query is far more expensive than expected. What is your troubleshooting checklist?
A: Confirm the format is columnar Parquet not CSV; confirm the filter targets `filepath()` so folders prune; confirm string predicates use the `BIN2_UTF8` collation for pushdown; confirm files are right-sized; and confirm you select only needed columns. Then read the data-processed metric to see which layer is leaking bytes. Each item maps to a distinct lever on bytes read.

Q: Why does selecting only the columns you need matter for serverless cost, beyond tidiness?
A: Over Parquet, the engine reads only the columns referenced, so `SELECT col1, col2` reads far fewer bytes than `SELECT *`, directly lowering data processed and cost. `SELECT *` defeats the main advantage of columnar storage. On a wide gold table this difference alone can be an order of magnitude.

Q: Why is `SELECT *` over a wide CSV gold table the worst case for serverless cost?
A: CSV is row-based so there is no column pruning, and `SELECT *` returns every column, meaning the engine must read every byte of every file regardless of the filter. You pay full scan with none of Parquet's skipping. Converting to Parquet and selecting specific columns attacks both halves of the problem.

Q: Does data egress factor into Synapse serverless or Athena query cost?
A: The headline per-TB-scanned charge is for reading from storage to the engine within the region; cross-region or internet egress of results is a separate networking charge. Co-locating storage, engine, and client in the same region avoids extra egress and latency. So keeping everything in one region is both a performance and a cost practice.

Q: Why should storage and the serverless SQL endpoint be in the same region?
A: Cross-region reads add network latency and potential egress charges and can slow large scans significantly, since every byte travels between regions. Co-location keeps the high-volume storage-to-engine traffic intra-region and fast. Synapse provisions its default storage and endpoint in the same region for exactly this reason.

Q: What is storage throttling and how does it affect serverless query performance?
A: Throttling occurs when combined IOPS or throughput from all workloads exceeds the storage account's limits, degrading query performance as serverless backs off its request rate. It is not a billing event but a latency one. Avoid running heavy unrelated workloads against the same account during big queries, or spread data across accounts.

Q: How does Delta Lake log layout affect serverless query performance in Synapse?
A: Query performance over Delta depends on the number of JSON files in the `_delta_log` directory; too many uncheckpointed JSON commits slow reads because the engine must replay them. Maintaining roughly ten JSON files between checkpoints balances read and write performance. You can set `delta.checkpointInterval` to `10` to force regular checkpoints.

Q: What `TBLPROPERTIES` setting controls Delta checkpoint frequency for serverless read performance?
A: Setting `ALTER TABLE tableName SET TBLPROPERTIES ('delta.checkpointInterval' = '10')` makes Delta write a Parquet checkpoint after every ten JSON log files. This keeps the `_delta_log` from accumulating many JSON files that slow serverless reads. It trades a little write overhead for consistent read performance.

Q: What does it mean that serverless SQL has "no local storage or ingestion"?
A: Serverless SQL pool cannot store data or ingest it into itself; every file it queries lives in external object storage and is read in place each time. There is no warehouse table to load into, no cache you provision. This is precisely why repeated heavy queries re-scan storage and why CETAS materialization is the way to avoid repeated reads.

Q: How does schema inference work in serverless SQL and what is its cost-of-convenience?
A: Schema inference lets you query files without declaring columns by reading the file's own metadata or sampling, which speeds exploration. The cost is over-sized inferred types — Parquet strings infer as `varchar(8000)` because the format lacks length metadata — which inflates memory, sorts, and joins. For production queries you should pin tighter types with `WITH`.

Q: How do you discover the inferred types of an `OPENROWSET` query?
A: Run `EXEC sp_describe_first_result_set` against the `OPENROWSET` query to get each column's inferred `system_type_name` and `max_length`. You then use those results to write a `WITH` clause with smaller, accurate types. This is the standard workflow for tightening an exploratory query into an efficient production one.

Q: Why pick the smallest adequate numeric type, for example `smallint` over `int`, in serverless queries?
A: Smaller types narrow each row, so more rows fit per memory page and sort/join/group operations move less data, completing faster. If the max value fits in `smallint`, using `int` wastes space for no benefit. Integer operations are also faster than character operations, so prefer integer keys where possible.

Q: What concurrency does Synapse serverless support and how does query weight affect it?
A: Concurrency is not hard-capped; one serverless pool can handle on the order of 1,000 active sessions running lightweight queries, but the practical number drops as queries get more complex or scan more data. Heavy scans consume more shared resources per query. So efficient layout indirectly raises how many users you can serve at once.

Q: Why avoid Azure Synapse Studio for returning large result sets?
A: Synapse Studio is a web tool that transfers results over HTTPS and is not designed for large result sets, so it is slow and can choke on big returns. For large outputs use SQL Server Management Studio or Azure Data Studio instead. This is a client-side performance concern, separate from the per-TB scan cost.

Q: Does using a more efficient client application change the per-TB scan cost?
A: No — the scan cost depends on bytes read from storage by the engine, which is independent of the client; switching from Synapse Studio to SSMS only improves result-transfer speed. The scan is the same whether you view results in a fast or slow client. Client choice affects experience, not the data-processed bill.

Q: What is the relationship between query latency and query cost in serverless SQL?
A: They are correlated through bytes read: a query that scans more data both costs more (more TB processed) and usually runs slower (more IO). Optimizing layout to read fewer bytes improves both simultaneously. This alignment is convenient — the same partitioning and Parquet work that cuts cost also cuts time.

Q: Are cached or repeated identical queries free in serverless SQL?
A: Where the service serves a query from cache without re-reading storage, the scan is not re-billed; but serverless pools have limited result caching compared to a warehouse, so most repeated heavy queries re-scan and re-bill. Do not rely on implicit caching for cost control. Use CETAS materialization to make repeated reads genuinely cheap.

Q: Why is a Power BI Direct Query dashboard over serverless SQL a cost risk?
A: Each interactive filter or refresh fires queries that scan storage and bill per TB, so a busy dashboard can generate many scans and a large bill, especially over a poorly partitioned source. Serverless cannot give a snappy interactive experience for complex Direct Query loads anyway. Import mode or a cached layer is recommended to bound both cost and latency.

Q: How does Power BI Import mode reduce serverless query cost compared to Direct Query?
A: Import mode loads a snapshot into Power BI's own engine and serves interactions from that cache, so the serverless scan happens only on scheduled refresh rather than on every user interaction. This collapses thousands of interactive scans into a few daily refresh scans. The trade-off is data freshness bounded by the refresh interval.

Q: What is the danger of allowing analysts unrestricted ad-hoc serverless SQL over the raw lake?
A: Without guardrails, an analyst can run `SELECT *` over an unpartitioned multi-terabyte raw zone repeatedly, each query scanning everything and billing accordingly, producing a surprise four-figure bill. The architect should expose a well-partitioned, Parquet gold layer and views rather than the raw zone. Layout plus access design is the cost guardrail.

Q: How can cost-control results-set caches like Azure Analysis Services reduce serverless scans?
A: By materializing query results into a separate caching layer that periodically refreshes from serverless, you serve user queries from the cache and limit serverless scans to the refresh cadence. This decouples user query volume from scan volume. It is the same idea as Import mode applied as a shared semantic-model tier.

Q: In a transfer-agency reporting context, why does query frequency matter for the serverless decision?
A: Transfer-agency subscription/redemption reporting can be run many times a day across many funds, and at high frequency the per-TB meter accumulates; if scans are large, a running warehouse may be cheaper. The frequency-times-scan product determines the choice. Good date and fund partitioning shrinks scan size enough to keep serverless competitive even at high frequency.

Q: How does laying out data by ISIN help fund-data serverless queries specifically?
A: Because most instrument-level queries filter on a specific ISIN, an `ISIN` partition lets each query read only that instrument's folder and skip every other instrument. Combined with a date partition, an ISIN-on-a-date lookup reads a tiny leaf. This maps the dominant fund-data access pattern directly onto folder pruning.

Q: Why might partitioning by ISIN alone be insufficient for a holdings history query?
A: A query asking for one ISIN across all dates still reads every date under that ISIN folder, which can be large over years; and a date-range query that does not fix an ISIN cannot prune by ISIN. You generally need both date and ISIN partitions, ordered to match the dominant filter. Single-dimension partitioning only helps queries that filter on that dimension.

Q: How would you estimate the three-strategy cost delta the lesson's "Do" step asks for?
A: For each strategy — none, by date, by date plus ISIN — multiply the bytes a representative query would scan by the `$5`/TB rate: none scans the whole table, by-date scans one day, by-date-plus-ISIN scans one instrument-day. Reading the data-processed metric for each validated layout confirms the estimate. The delta between full scan and leaf scan is typically two to three orders of magnitude.

Q: Why convert just one CSV gold table to Parquet as a learning exercise?
A: Converting one table and re-estimating its scan makes the cost effect of columnar storage and compression concrete and measurable without rebuilding the whole lake. You directly observe data processed drop for the same query. It also surfaces practical issues — schema, types, file sizing — on a manageable scope before scaling the pattern.

Q: What would you write in the "design pressure" paragraph about per-TB pricing and file layout?
A: That per-TB-scanned pricing makes physical layout the cost model: partitioning on the columns queries filter on, converting to compressed columnar Parquet, right-sizing files, and exposing pruned views are not optional polish but the primary cost-control mechanism. The architect's layout decisions directly set the consumption tier's bill. Therefore layout reviews are budget reviews.

Q: Compare per-TB-scanned serverless pricing to a provisioned warehouse's pricing model.
A: Serverless bills on bytes read per query with nothing while idle; a provisioned warehouse bills on compute time (hourly or per-credit) whether or not it is fully utilized, often with caching that makes repeated queries cheap. Serverless shifts cost to data layout; warehouses shift cost to right-sizing and auto-suspend. The right choice depends on query frequency and scan size.

Q: What is the analogue of per-TB pricing in Databricks or Snowflake terms?
A: Those engines bill on compute units consumed (DBUs for Databricks, credits for Snowflake) over time rather than purely on bytes scanned, though efficient layouts still reduce the compute time needed. The serverless-SQL per-TB model is more directly tied to data read. Understanding both lets the architect reason about cost across the engine matrix.

Q: Why does the lesson call serverless SQL the "cheap-and-cheerful" consumption tier?
A: Because it requires no infrastructure to stand up, costs nothing when idle, and is inexpensive for occasional, well-laid-out queries, making it the easy entry point for lakehouse consumption. It is "cheerful" because it is simple and on-demand. The caveat is that "cheap" holds only when file layout keeps scans small.

Q: What completes the "engine matrix" by adding serverless SQL to it?
A: The matrix of consumption engines spans always-on MPP warehouses, Spark clusters, and serverless SQL; adding serverless fills the low-frequency, pay-per-scan corner with its distinctive price/perf envelope. Knowing where serverless sits — cheap when idle, dangerous under heavy frequent scans — lets the architect place each workload on the right engine. It is the final piece of the cost-model comparison.

Q: How does Athena's use of the Glue Data Catalog parallel Synapse external tables?
A: Both keep table metadata (location, schema, format) separate from the data files, so the engine queries files in place using catalog definitions. Glue plays the catalog role for Athena that external-table/database metadata plays in Synapse serverless. The shared idea is metadata-over-files, with data never copied into the engine.

Q: Why do DDL operations not incur scan charges across these serverless services?
A: Because creating, altering, or dropping table metadata reads no data files — it only touches the catalog — there are no bytes processed to bill. The meter is strictly on data read during query execution. This is consistent across Synapse serverless and Athena and is why schema management is effectively free.

Q: A serverless query bill spiked overnight with no code change. What would you investigate first?
A: Check whether the underlying data grew (more files or larger partitions now scanned), whether a partition-pruning predicate stopped matching the new folder layout, or whether a scheduled job started firing more or broader queries. Read the data-processed metric per query to localize the increase. Data growth silently raises scan cost even when the SQL is unchanged.

Q: Why can a single bad `WHERE` clause turn an ad-hoc query into a four-figure bill?
A: If the filter does not align with the partition layout — for example filtering on a non-partition column or using a function that defeats pruning — the engine falls back to a full scan of a multi-terabyte table, costing roughly `$5` per TB scanned. The mechanism is loss of partition elimination. Always confirm the filter targets the partition keys via `filepath()`.

Q: How does applying a function to a partition column in the `WHERE` clause hurt pruning?
A: Wrapping the partition value in a function can prevent the engine from matching the predicate to folder values, so partition elimination does not fire and the engine scans everything. Filter on the raw `filepath()`/partition value directly, casting only as the docs advise, so the comparison stays sargable. This is a common silent cause of full scans.

Q: Why does the lesson advise casting the result of `filepath()`/`filename()` to an appropriate type?
A: The functions return string values, and comparing them to the right typed value with an appropriately sized type ensures the predicate is efficient and correctly matches; using an oversized character type or wrong type can hurt performance. Casting to, say, a small `int` for a year partition keeps comparisons tight. Microsoft explicitly recommends casting these results.

Q: How would you bound a junior analyst's worst-case serverless cost by design?
A: Expose only a well-partitioned, Parquet gold layer and partitioned views (not the raw zone), so even an unfiltered `SELECT *` scans a bounded, columnar dataset rather than the entire raw lake. Combine with workspace cost alerts and result caches. Design the surface area so the worst plausible query is still cheap.

Q: Summarize the single most important cost lever in serverless SQL and why.
A: Reduce bytes read — primarily through partition pruning on the columns queries filter, plus columnar Parquet with compression and right-sized files — because the bill and the latency both track data processed. Everything else (collation for pushdown, statistics, column selection) is a refinement of "read less." File layout is therefore the cost model, and getting it right is the architect's core responsibility for the consumption tier.
