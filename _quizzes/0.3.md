# Phase 0 · 0.3 SQL from zero — 100 self-test questions

Q: What is the basic purpose of a `SELECT` statement?
A: It reads (queries) rows from one or more tables and returns a result set; it does not change the stored data.

Q: In `SELECT fund_name, nav FROM funds;`, what does the part between `SELECT` and `FROM` specify?
A: The column list (projection) — exactly which columns appear in the output and in what order; `fund_name` then `nav`.

Q: What does `SELECT *` return and why is it discouraged in production code?
A: It returns every column of the table; it is fragile because added/reordered columns silently change results and it pulls more data than needed.

Q: How do you return every row of a table with no filtering?
A: Omit the `WHERE` clause entirely, e.g. `SELECT * FROM funds;`.

Q: What does the `WHERE` clause do?
A: It filters rows, keeping only those for which its condition evaluates to `TRUE` (rows that are `FALSE` or `UNKNOWN` are excluded).

Q: Write a condition that selects funds with a NAV above 100.
A: `WHERE nav > 100` — comparison operators like `>`, `<`, `>=`, `<=`, `=`, `<>` filter on the column value.

Q: How do you match a range of NAV values inclusively between 90 and 110?
A: `WHERE nav BETWEEN 90 AND 110`, which is inclusive of both endpoints (equivalent to `nav >= 90 AND nav <= 110`).

Q: What does `WHERE currency IN ('EUR','USD','GBP')` do?
A: It keeps rows whose `currency` equals any value in the list — a concise alternative to several `OR` comparisons.

Q: What does the `LIKE` operator do and what do `%` and `_` mean?
A: `LIKE` does pattern matching on text; `%` matches any sequence of characters (including none) and `_` matches exactly one character.

Q: Write a filter for ISINs that start with `LU` (Luxembourg-domiciled).
A: `WHERE isin LIKE 'LU%'` — the `%` allows any characters after the `LU` prefix.

Q: How does `ORDER BY` change a result set?
A: It sorts the returned rows; without it the row order is not guaranteed by SQL even if it looks consistent.

Q: What is the default sort direction of `ORDER BY`, and how do you reverse it?
A: Ascending (`ASC`) is the default; add `DESC` for descending, e.g. `ORDER BY nav DESC`.

Q: How do you sort by multiple columns, e.g. currency then NAV descending?
A: `ORDER BY currency ASC, nav DESC` — rows are sorted by the first key, ties broken by the next.

Q: Can `ORDER BY` reference a column alias or output column number?
A: Yes; because `ORDER BY` runs after the `SELECT` list is computed, you can use an alias (`ORDER BY total`) or position (`ORDER BY 2`), though aliases are clearer.

Q: What does `LIMIT 10` (or `FETCH FIRST 10 ROWS ONLY`) do, and why pair it with `ORDER BY`?
A: It caps the result to 10 rows; without `ORDER BY` the "top 10" is arbitrary because row order is undefined.

Q: In what logical order are `FROM`, `WHERE`, `GROUP BY`, `HAVING`, `SELECT`, and `ORDER BY` evaluated?
A: `FROM` → `WHERE` → `GROUP BY` → `HAVING` → `SELECT` → `ORDER BY`; this explains why `WHERE` can't see aggregates and `ORDER BY` can see `SELECT` aliases.

Q: What is a `JOIN` used for?
A: To combine rows from two or more tables based on a related column (the join condition), producing a wider result set.

Q: What does an `INNER JOIN` return?
A: Only the rows where the join condition matches in both tables; unmatched rows from either side are dropped.

Q: What does a `LEFT JOIN` (LEFT OUTER JOIN) return?
A: All rows from the left table plus matching right-table columns; where there is no match, the right-side columns are `NULL`.

Q: A fund has no holdings yet. With `funds LEFT JOIN holdings`, what happens to that fund?
A: It still appears once, with all `holdings` columns as `NULL`; an `INNER JOIN` would have dropped it.

Q: What is the join key in `funds f JOIN nav_history n ON f.fund_id = n.fund_id`?
A: `fund_id` — the `ON` condition specifies which columns must match to pair rows from the two tables.

Q: Why can a join make the row count multiply (a "fan-out")?
A: If one left row matches many right rows, the left row is repeated once per match, so the output can be larger than either input table.

Q: One fund has 30 daily NAV rows; you join `funds` to `nav_history`. How many rows for that fund?
A: 30 — the single fund row is duplicated once for each matching NAV row.

Q: A many-to-many join unexpectedly returns far too many rows. What is the likely cause?
A: A row on one side matches multiple rows on the other (and vice versa), so counts multiply; check whether the join key is unique on at least one side.

Q: How do you avoid double-counting NAV when joining a 1-to-many relationship before aggregating?
A: Aggregate the "many" side first (e.g. in a subquery/CTE) so each left row matches exactly one pre-aggregated row, then join.

Q: What does a `CROSS JOIN` produce?
A: The Cartesian product — every row of the left table paired with every row of the right (rows = left_count × right_count), with no join condition.

Q: What accidental SQL pattern silently produces a Cartesian product?
A: Listing tables comma-separated in `FROM` without a join condition in `WHERE`, e.g. `FROM funds, holdings` — every combination is returned.

Q: What is the difference between filtering in the `ON` clause vs the `WHERE` clause of a `LEFT JOIN`?
A: A condition in `ON` only affects which right rows match (left rows are still kept); the same condition in `WHERE` runs after the join and can drop the unmatched left rows.

Q: Putting `WHERE h.quantity > 0` after a `LEFT JOIN` — what unintended effect can it have?
A: It turns the `LEFT JOIN` into an effective `INNER JOIN`, because rows with no match have `NULL` quantity, and `NULL > 0` is `UNKNOWN`, so they are filtered out.

Q: What is a self-join and give a fund example?
A: A table joined to itself with table aliases; e.g. joining `nav_history` to itself to compare each day's NAV with the prior day's row.

Q: How do you find funds in table A that have no matching row in table B using a `LEFT JOIN`?
A: `A LEFT JOIN B ON ... WHERE b.key IS NULL` — the anti-join keeps only left rows whose right side never matched.

Q: What does `GROUP BY currency` do conceptually?
A: It collapses rows into one output row per distinct `currency`, so aggregate functions compute per group.

Q: Name four common aggregate functions and what they return.
A: `COUNT()` counts rows, `SUM()` totals, `AVG()` averages, `MIN()`/`MAX()` return the smallest/largest value in the group.

Q: What is the difference between `COUNT(*)` and `COUNT(column)`?
A: `COUNT(*)` counts all rows in the group; `COUNT(column)` counts only rows where that column is not `NULL`.

Q: What does `COUNT(DISTINCT isin)` return?
A: The number of distinct non-`NULL` ISIN values in the group, eliminating duplicates before counting.

Q: Which columns may appear unaggregated in the `SELECT` list when using `GROUP BY`?
A: Only the columns listed in `GROUP BY`; every other selected column must be inside an aggregate function (in standard SQL).

Q: Why does `SELECT currency, fund_name, SUM(nav) ... GROUP BY currency` raise an error?
A: `fund_name` is neither grouped nor aggregated, so the database cannot pick a single value per currency group.

Q: What is the difference between `WHERE` and `HAVING`?
A: `WHERE` filters individual rows before grouping; `HAVING` filters whole groups after aggregation and can reference aggregate functions.

Q: Write a query for currencies whose total NAV exceeds 1,000,000.
A: `SELECT currency, SUM(nav) FROM funds GROUP BY currency HAVING SUM(nav) > 1000000;` — the group-level filter belongs in `HAVING`.

Q: Can you put an aggregate like `SUM(nav) > 100` in a `WHERE` clause?
A: No; `WHERE` runs before grouping so aggregates aren't available there — use `HAVING` instead.

Q: To count holdings per fund and keep only funds with more than 50 holdings, what clauses do you use?
A: `GROUP BY fund_id` with `HAVING COUNT(*) > 50` — group then filter the groups.

Q: What does a subquery (nested query) mean?
A: A `SELECT` embedded inside another statement, used to produce a value, a list, or a derived table that the outer query consumes.

Q: What is a scalar subquery?
A: A subquery that returns exactly one row and one column, usable wherever a single value is expected, e.g. `WHERE nav > (SELECT AVG(nav) FROM funds)`.

Q: What does `WHERE isin IN (SELECT isin FROM blacklist)` do?
A: It keeps rows whose `isin` appears in the set of values returned by the subquery — a membership test against a list.

Q: What is the difference between a correlated and a non-correlated subquery?
A: A non-correlated subquery runs once independently; a correlated subquery references the outer row and is conceptually re-evaluated per outer row.

Q: What does `EXISTS (SELECT 1 FROM holdings h WHERE h.fund_id = f.fund_id)` test?
A: Whether at least one matching holding row exists for the current fund; `EXISTS` returns `TRUE`/`FALSE` and stops at the first match.

Q: What is a CTE (Common Table Expression) and how is it written?
A: A named temporary result defined with `WITH name AS (SELECT ...)` that you can reference later in the query, improving readability over nested subqueries.

Q: Give one advantage of a CTE over a nested subquery.
A: Readability and reuse — a CTE is named, can be referenced multiple times in the main query, and reads top-to-bottom instead of inside-out.

Q: How do you chain multiple CTEs?
A: Separate them with commas after one `WITH`, e.g. `WITH a AS (...), b AS (SELECT ... FROM a) SELECT ... FROM b;`.

Q: What is a recursive CTE typically used for?
A: Hierarchical/iterative data such as fund-of-funds ownership chains or org trees, using `WITH RECURSIVE` and a `UNION ALL` between an anchor and a recursive member.

Q: What is `NULL` in SQL?
A: A marker for unknown or missing data — it is not zero, not an empty string, and not equal to anything (including another `NULL`).

Q: Why does `WHERE nav = NULL` return no rows?
A: Comparing anything to `NULL` with `=` yields `UNKNOWN`, never `TRUE`, so no row passes the filter; you must use `IS NULL`.

Q: How do you correctly test for missing values?
A: Use `IS NULL` and `IS NOT NULL`, which are the only operators that return `TRUE`/`FALSE` for `NULL`.

Q: What are the three truth values in SQL's logic?
A: `TRUE`, `FALSE`, and `UNKNOWN` — three-valued logic, where comparisons involving `NULL` produce `UNKNOWN`.

Q: What does `NULL = NULL` evaluate to?
A: `UNKNOWN`, not `TRUE` — two unknown values cannot be asserted equal.

Q: What does `NOT IN (subquery)` do if the subquery contains a `NULL`?
A: It can wrongly return no rows, because a comparison with the `NULL` yields `UNKNOWN`; prefer `NOT EXISTS` to avoid this trap.

Q: How does `NULL` behave in arithmetic, e.g. `100 + NULL`?
A: The result is `NULL` — any arithmetic with `NULL` propagates to `NULL`.

Q: Does `SUM(nav)` count `NULL` NAV values?
A: No; aggregate functions (except `COUNT(*)`) ignore `NULL` inputs, so `SUM`/`AVG` skip them.

Q: How can ignoring `NULL` in `AVG()` mislead you?
A: `AVG` divides by the count of non-`NULL` values, so missing data is excluded rather than treated as zero, which can inflate the average.

Q: How do you substitute a default value for `NULL`?
A: Use `COALESCE(nav, 0)` (or `IFNULL`/`ISNULL` in some dialects), which returns the first non-`NULL` argument.

Q: What does `COALESCE(a, b, c)` return?
A: The first of `a`, `b`, `c` that is not `NULL`; if all are `NULL`, it returns `NULL`.

Q: How does `ORDER BY` handle `NULL` values?
A: Dialect-dependent — they sort either first or last; control it explicitly with `NULLS FIRST` / `NULLS LAST` where supported.

Q: What is a window function?
A: A function that computes a value across a set of rows related to the current row (the "window") without collapsing them, unlike `GROUP BY` which reduces rows.

Q: What is the key behavioural difference between an aggregate with `GROUP BY` and a window function?
A: `GROUP BY` returns one row per group; a window function returns one value per input row, so every original row is retained.

Q: What does the `OVER ()` clause do?
A: It defines the window for the function; an empty `OVER ()` treats all rows as one partition, e.g. `SUM(nav) OVER ()` gives the grand total on every row.

Q: What does `PARTITION BY` do inside `OVER`?
A: It splits rows into groups; the window function restarts independently within each partition, e.g. running totals per fund.

Q: What does `ROW_NUMBER() OVER (ORDER BY nav DESC)` produce?
A: A unique sequential integer per row in the specified order (1, 2, 3, …), with no ties.

Q: How do you get the latest NAV row per fund using `ROW_NUMBER`?
A: `ROW_NUMBER() OVER (PARTITION BY fund_id ORDER BY nav_date DESC)` then filter the outer query to `WHERE rn = 1`.

Q: What is the difference between `RANK()` and `DENSE_RANK()`?
A: Both give ties the same rank; `RANK()` then skips numbers (1,1,3) while `DENSE_RANK()` does not (1,1,2).

Q: What does `LAG(nav) OVER (PARTITION BY fund_id ORDER BY nav_date)` return?
A: The previous day's NAV for the same fund; the first row of each partition has no predecessor, so it returns `NULL`.

Q: How do you compute the daily NAV change per fund using `LAG`?
A: `nav - LAG(nav) OVER (PARTITION BY fund_id ORDER BY nav_date)` gives the difference from the prior NAV in date order.

Q: What does `LEAD()` do compared to `LAG()`?
A: `LEAD()` looks forward to a following row, while `LAG()` looks back to a preceding row, both within the ordered window.

Q: How do you give `LAG`/`LEAD` a default for missing neighbours?
A: Pass the offset and default, e.g. `LAG(nav, 1, 0)` returns 0 instead of `NULL` when there is no prior row.

Q: Write a running total of subscriptions per fund over time.
A: `SUM(amount) OVER (PARTITION BY fund_id ORDER BY trade_date)` — adding `ORDER BY` turns the window into a cumulative (running) sum.

Q: Why does adding `ORDER BY` inside `OVER` change `SUM` from a total to a running total?
A: With `ORDER BY` the default frame becomes `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`, so it sums from the start up to the current row.

Q: What does the frame clause `ROWS BETWEEN 2 PRECEDING AND CURRENT ROW` do?
A: It defines a sliding window of the current row plus the two before it — useful for a 3-row moving average of NAV.

Q: Can you use a window function in a `WHERE` clause?
A: No; window functions are computed after `WHERE`, so you must wrap the query in a subquery/CTE and filter on the result (e.g. `WHERE rn = 1`).

Q: What does the `INSERT` statement do?
A: It adds new rows to a table.

Q: Write an `INSERT` adding one fund with id, name, and currency.
A: `INSERT INTO funds (fund_id, fund_name, currency) VALUES (1, 'EU Equity', 'EUR');` — name the columns then supply matching values.

Q: Why should you list column names explicitly in `INSERT`?
A: So the statement keeps working if columns are added or reordered, and so it is clear which value maps to which column.

Q: How do you copy rows from one table into another in a single statement?
A: `INSERT INTO archive (...) SELECT ... FROM source WHERE ...;` — the `SELECT` supplies the rows to insert.

Q: What does `UPDATE funds SET currency = 'EUR' WHERE fund_id = 1;` do?
A: It changes the `currency` to `EUR` for the single row whose `fund_id` is 1.

Q: What is the danger of `UPDATE funds SET currency = 'EUR';` with no `WHERE`?
A: It overwrites the currency of every row in the table, not just the intended one — a classic destructive mistake.

Q: What does `DELETE FROM holdings WHERE fund_id = 1;` do?
A: It removes only the holding rows belonging to fund 1.

Q: What happens with `DELETE FROM holdings;` (no `WHERE`)?
A: It deletes every row in the table; the table remains but is emptied.

Q: How can you preview which rows an `UPDATE`/`DELETE` will affect before running it?
A: Run the same `WHERE` clause inside a `SELECT` first (e.g. `SELECT * FROM funds WHERE ...`) to confirm the row set.

Q: What is a transaction and how do `COMMIT` and `ROLLBACK` relate?
A: A transaction groups statements into an all-or-nothing unit; `COMMIT` makes changes permanent and `ROLLBACK` undoes them since the transaction began.

Q: How can transactions protect you from a mistaken mass `UPDATE`?
A: Run inside `BEGIN`/`START TRANSACTION`, check the row count or results, and `ROLLBACK` if wrong instead of `COMMIT`.

Q: What does `UPSERT` (e.g. `INSERT ... ON CONFLICT DO UPDATE`) achieve for daily NAV loads?
A: It inserts a new NAV row or, if the key already exists, updates it instead — idempotent loading without duplicate-key errors.

Q: What is a database index, in one sentence?
A: An auxiliary data structure (often a B-tree) that lets the engine locate rows by key quickly without scanning the whole table.

Q: Which operations does an index typically speed up?
A: Lookups and range scans in `WHERE`, join matching on the indexed column, and sometimes `ORDER BY`/`GROUP BY` by avoiding a sort.

Q: What is the main cost of adding an index?
A: Extra storage and slower writes (`INSERT`/`UPDATE`/`DELETE`), because the index must be maintained on every data change.

Q: Why does a primary key usually come with an index automatically?
A: The database needs to enforce uniqueness efficiently, so it builds a unique index on the primary-key column(s).

Q: Why does `WHERE UPPER(isin) = 'LU123'` often fail to use an index on `isin`?
A: Wrapping the column in a function prevents the engine from matching it to the plain-column index, forcing a full scan unless a matching function/expression index exists.

Q: For a query filtering `fund_id` and ordering by `nav_date`, what kind of index helps most?
A: A composite index on `(fund_id, nav_date)`, so the engine can seek by fund and read rows already in date order.

Q: How do you check whether a query actually uses an index?
A: Run `EXPLAIN` (or `EXPLAIN ANALYZE`) and look for an index scan/seek versus a sequential/full table scan in the plan.

Q: What is `information_schema`?
A: A standard set of read-only views describing the database's own metadata — its tables, columns, data types, constraints, and more.

Q: How do you list all columns and their data types for a `funds` table via `information_schema`?
A: `SELECT column_name, data_type FROM information_schema.columns WHERE table_name = 'funds';`.

Q: How would you find every table in the database using `information_schema`?
A: `SELECT table_name FROM information_schema.tables WHERE table_schema = 'public';` (adjust the schema filter to your database).

Q: An EMT file load fails because a column is missing — how can `information_schema` help diagnose it fast?
A: Query `information_schema.columns` for the target table to confirm the exact column names, types, and nullability before re-checking the file mapping.
