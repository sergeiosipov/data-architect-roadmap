# Phase 5 · 4.1.2 Serverless compute (Azure Functions) — 100 self-test questions

Q: What is Azure Functions in one sentence, and what does "serverless" actually mean here?
A: Azure Functions is an event-driven, serverless compute service that runs small units of code ("functions") in response to triggers without you managing the underlying servers. "Serverless" means the platform handles provisioning, scaling to zero, and patching — you pay for execution rather than for an always-on VM. In a fund platform it is the natural home for glue work like reacting to an EMT file landing in storage, not for running the heavy NAV-calculation engine itself.

Q: What is a "trigger" in Azure Functions and how many can a single function have?
A: A trigger is the event that causes a function to run, and it defines how the function is invoked and what data flows in as a parameter. Every function must have exactly one trigger — for example an `HttpTrigger`, `TimerTrigger`, or `BlobTrigger` — and no more. The trigger is what turns an external event (a file drop, a queue message, an HTTP request, a schedule) into a code execution.

Q: What is the difference between a trigger and a binding?
A: A trigger starts the function and is mandatory (exactly one per function), whereas a binding is an optional declarative connection to data that the runtime wires up for you. Bindings come in two directions: input bindings read data into the function, output bindings write data out, and you can have zero, one, or many of each. The point of both is to avoid hand-writing SDK client/connection code for the common cases.

Q: What does `direction: in` versus `direction: out` mean in a binding definition?
A: `direction: in` declares an input binding that supplies data to the function (for example reading a blob or a Cosmos DB document), while `direction: out` declares an output binding that the function writes to (for example enqueuing a message). The trigger itself is always `direction: in`. These directions appear explicitly in the legacy `function.json` model and are implied by the attribute or decorator type in code-first models.

Q: Are bindings required to use Azure Functions?
A: No — bindings are entirely optional. You can ignore them and create an Azure SDK client directly in your code (for example `BlobClient` or `QueueClient`) to read and write data yourself. Bindings just remove boilerplate for the common cases; the trigger is the only mandatory wiring.

Q: How are triggers and bindings declared across the different language models?
A: In C# class libraries you decorate methods and parameters with attributes like `[BlobTrigger(...)]` and `[QueueOutput(...)]`; in the Python v2 and Node.js v4 models you use decorators/registration objects such as `@app.blob_trigger(...)` or `app.storageBlob(...)`; and the legacy model uses a separate `function.json` file per function with a `bindings` array. All three describe the same underlying trigger-and-binding contract, just in different syntaxes.

Q: Give a concrete trigger/input/output combination for a "scheduled job reads a blob and writes a Cosmos DB document" scenario.
A: The trigger is a Timer trigger (it fires on a schedule), the input binding is Blob Storage (to read the file contents), and the output binding is Azure Cosmos DB (to write the resulting document). This illustrates the pattern: one trigger to start, input bindings to fetch data, output bindings to emit results — all declared rather than hand-coded.

Q: Name several common Azure Functions trigger types beyond HTTP.
A: Besides `HttpTrigger`, common triggers include `TimerTrigger` (schedule-based), `BlobTrigger` (a blob is created or updated), `QueueTrigger` (a Storage queue message), `EventGridTrigger`, `ServiceBusTrigger`, `EventHubTrigger`, and Cosmos DB change-feed trigger. Each maps an external event source to a function invocation, and many have matching input/output bindings for the same service.

Q: What schedule format does the Timer trigger use, and what is a gotcha versus standard cron?
A: The Timer trigger uses an NCRONTAB expression, which has six fields — `{second} {minute} {hour} {day} {month} {day-of-week}` — adding a leading seconds field that classic five-field Unix cron lacks. A common gotcha is copying a five-field Unix cron string and getting an off-by-one-field schedule; for example `0 */5 * * * *` means "every 5 minutes", not the Unix `*/5 * * * *`.

Q: In a fund platform, what would an event-triggered function that "validates an incoming EMT file drop and emits an event" look like in terms of wiring?
A: It is triggered by a Blob trigger (preferably Event Grid-sourced) on the container where EMT files land; the function reads the blob, runs schema/field validation (for example checks ISINs and required EMT fields), and uses an output binding — to Event Grid, Service Bus, or a queue — to emit a "validated" or "rejected" event. The function is pure glue: it gates the file and signals downstream, but does not itself perform the heavy transformation.

Q: What does the Blob trigger path `samples-workitems/{name}` mean?
A: The first segment (`samples-workitems`) is the container to watch, and `{name}` is a binding expression that captures the blob's name so you can use it in code or in other bindings. For example if `incoming-emt/{name}` triggers on `fund123_20260613.csv`, the `name` variable holds `fund123_20260613.csv`. A container name itself cannot contain a binding-expression resolver.

Q: How do you filter a Blob trigger to only certain files, such as only `.csv` EMT files?
A: You add a name pattern to the path, for example `incoming-emt/{name}.csv` to fire only on `.csv` blobs, or `incoming-emt/EMT-{name}` to fire only on blobs whose names start with `EMT-`. You can also split name and extension with `input/{blobname}.{blobextension}`. This is declarative filtering, so non-matching blobs never invoke the function at all.

Q: What are the two implementations of the Blob storage trigger, and which is recommended?
A: There is a polling-based implementation (`source = LogsAndContainerScan`, the default) and an event-based implementation (`source = EventGrid`). Microsoft recommends the Event Grid-based trigger because it has much lower latency, and importantly the Flex Consumption plan supports only the event-based blob trigger. For a fund platform that drops files on a schedule and needs prompt processing, Event Grid is the right choice.

Q: How does the polling-based Blob trigger actually detect new blobs, and what latency can it incur?
A: It works as a hybrid of inspecting storage analytics logs and running periodic container scans, scanning blobs in groups of 10,000 with a continuation token between intervals. On a Consumption plan an idle function app can take up to about 10 minutes to notice a new blob. Because storage logs are "best effort", some events can even be missed — another reason to prefer the Event Grid source for reliable, prompt processing.

Q: What are "blob receipts" and which container stores them?
A: Blob receipts are how the Functions runtime guarantees a blob-trigger function isn't invoked more than once for the same blob version; the runtime records a receipt per processed blob. They live in a container named `azure-webjobs-hosts` in the storage account referenced by the `AzureWebJobsStorage` app setting. A receipt records the function name, container, blob type, blob name, and the blob's ETag (the version identifier).

Q: How would you force a Blob trigger to reprocess a file it has already handled?
A: Delete that blob's receipt from the `azure-webjobs-hosts` container; reprocessing is then guaranteed to happen at some later point, though not necessarily immediately. To force an immediate rescan you can update the `scaninfo` blob under `azure-webjobs-hosts/blobscaninfo`, since any blob with a last-modified timestamp after the `LatestScan` property gets scanned again. This is the lever to re-run validation on a corrected EMT file with the same name.

Q: What happens to a blob whose trigger function keeps failing — what are "poison blobs"?
A: By default Azure Functions retries a failing blob-trigger function up to five times; if all attempts fail it writes a message to a Storage queue named `webjobs-blobtrigger-poison`. That poison message is a JSON object containing `FunctionId`, `BlobType`, `ContainerName`, `BlobName`, and the `ETag`. Monitoring this queue is essential in a fund pipeline so that a malformed EMT file that fails validation five times surfaces an alert rather than silently disappearing.

Q: When binding a blob to `string` or `byte[]`, what memory gotcha should you watch for?
A: Binding to `string` or `byte[]` loads the entire blob into memory, and the runtime may load it more than once during processing, so large files can blow your instance's memory limit. For anything but small blobs you should bind to a streaming type such as `Stream` or `BlobClient` instead. With large EMT/NAV files this matters: a multi-hundred-MB CSV bound as `string` can OOM the worker.

Q: What is an output binding, and give an example relevant to emitting an event after EMT validation.
A: An output binding lets the function write to a service declaratively, by setting a return value or an output parameter rather than calling an SDK directly. For example a `QueueOutput("emt-validated")` binding or an Event Grid output binding lets the validation function emit a "file validated" event simply by returning the message object. The runtime takes care of the actual send to the queue or topic.

Q: What is the `AzureWebJobsStorage` app setting used for?
A: It is the connection to the Storage account the Functions runtime uses for its own internal bookkeeping — blob receipts, poison queues, blob-scan info, timer state, and (for some plans) the deployment package and lease coordination. It is distinct from any storage your business logic reads or writes. If it is misconfigured the host itself may fail to start or triggers may not fire, even though your code is correct.

Q: List the current Azure Functions hosting plans you should know.
A: The GA plans are Flex Consumption (the recommended serverless plan), Premium, Dedicated (App Service plan / ASE), and Azure Container Apps hosting; the original Consumption plan is now legacy (Windows still GA, Linux retiring). For new serverless function apps Microsoft directs you to Flex Consumption rather than the legacy Consumption plan. The plan you pick dictates scaling, timeout limits, networking, cold start, and the billing model.

Q: What is the Flex Consumption plan and what does it bill you for?
A: Flex Consumption is the recommended serverless plan offering fast horizontal scaling (up to 1,000 instances), VNet integration, selectable instance memory sizes (512 MB, 2,048 MB, or 4,096 MB), and always-ready instances. Billing is based on the number of executions plus the memory of instances while they are actively executing functions (GB-seconds), plus the cost of any always-ready instances you keep warm. It is pay-as-you-go and can scale to zero when idle.

Q: What are the default and maximum function timeouts on the Flex Consumption plan?
A: On Flex Consumption the default `functionTimeout` is 30 minutes and the maximum is effectively unbounded — there is no enforced maximum execution timeout. However, during scale-in a running execution gets only a 60-minute grace period, and a 10-minute grace period during platform updates. This is a big change from the legacy Consumption plan's hard 10-minute cap.

Q: What are the default and maximum timeouts on the legacy Consumption plan, and why does it matter?
A: On the legacy Consumption plan the default timeout is 5 minutes and the maximum is 10 minutes. This hard cap is exactly why heavy or long-running data movement is an anti-pattern there — a long NAV reconciliation or a multi-gigabyte transform will simply be killed at 10 minutes. If your workload routinely approaches that limit, it is a signal to move to a different plan or, better, a pipeline engine.

Q: What is the 230-second HTTP limit, and how is it different from the function timeout?
A: Regardless of the configured `functionTimeout`, an HTTP-triggered function has roughly 230 seconds to send a response because of the default idle timeout of the Azure Load Balancer fronting the service. This is independent of the plan's execution timeout. For longer HTTP-initiated work you return an immediate `202 Accepted` and process asynchronously — for example via the Durable Functions async HTTP pattern.

Q: What is a "cold start" and which plans suffer from it?
A: A cold start is the extra latency incurred when a request arrives but no warm instance exists, so the platform must provision an instance and start the host and language worker. Plans that scale to zero — legacy Consumption and Flex Consumption when idle — can experience cold starts, while Premium and Dedicated keep instances warm so cold start isn't really an issue. Flex Consumption mitigates it with always-ready instances and improved cold-start behavior even from zero.

Q: What are "always-ready instances" and which plans offer them?
A: Always-ready (or pre-provisioned / prewarmed) instances are instances kept perpetually warm so requests don't pay a cold-start penalty. Flex Consumption lets you configure one or more always-ready instances, and the Premium plan keeps prewarmed workers warm by design (at least one instance per plan is always kept warm). You pay for these instances even when idle, which is the cost you trade for predictable low latency.

Q: When should you choose the Premium plan over Flex Consumption?
A: Choose Premium when your apps run continuously or near-continuously, when you need to avoid cold starts entirely with perpetually warm instances, when you need more CPU/memory than Flex offers, or when you have a high count of small executions whose per-execution bill is high but GB-seconds are low. Premium bills on core-seconds and memory across needed and prewarmed instances, with at least one instance always warm, giving the most predictable pricing among the dynamic plans.

Q: When is the Dedicated (App Service) plan the right choice for functions?
A: Pick Dedicated when you have existing underutilized App Service capacity to reuse, need fully predictable billing or manual scaling, want to co-host web apps and function apps on the same plan, need larger compute sizes, or need full isolation/secure networking via an App Service Environment. It is also the fallback for long-running scenarios where Durable Functions can't be used. Note that to get the unbounded timeout you must enable Always On.

Q: What does enabling "Always On" do on a Dedicated plan, and why is it needed?
A: Always On keeps the function host process running continuously instead of letting it idle out, which is what allows the unbounded timeout and prevents the app from going cold. Without it, the App Service plan can unload the app after inactivity and triggers like timers or blobs may not fire reliably. It is a prerequisite for treating a Dedicated-plan function app as an always-available worker.

Q: How does Flex Consumption decide to scale, and what is "per-function scaling"?
A: Flex Consumption makes fast, event-driven scaling decisions on a per-function basis rather than per-app, which is more deterministic. Most trigger types scale on independent instances, but HTTP triggers all scale together as a group, Blob (Event Grid) triggers scale together, and all Durable Functions triggers share instances and scale together. The maximum scale-out is 1,000 instances, subject to a regional subscription memory quota.

Q: What is the maximum instance count on Flex Consumption versus the legacy Consumption plan?
A: Flex Consumption scales out to a maximum of 1,000 instances (Linux only, subject to a regional memory quota). The legacy Consumption plan caps at 200 instances on Windows and 100 on Linux, with a separate limit of 500 instances per subscription per hour during scale-out for Linux. So Flex Consumption offers an order-of-magnitude higher ceiling for bursty workloads.

Q: Why is the Consumption plan now called "legacy", and what does Microsoft recommend instead?
A: The Consumption plan is legacy because Flex Consumption supersedes it with higher scale, VNet integration, configurable memory, longer timeouts, and better cold-start behavior; Linux Consumption is even being retired. Microsoft recommends using Flex Consumption for all new serverless function apps and provides a migration path for existing Consumption apps. Knowing this prevents you from architecting new fund-platform glue on a deprecated plan.

Q: For the "Do" task of documenting an Azure cost profile at 10k files/month, what cost components must you sum on a serverless plan?
A: You sum (1) the execution count — here 10,000 invocations/month, (2) the resource consumption in GB-seconds, computed as instance memory in GB times execution duration in seconds across all invocations, and (3) the cost of any always-ready/prewarmed instances you keep configured. On Flex Consumption the first two are the pure pay-as-you-go components and the third is the warm-instance floor. The plan choice itself (Flex vs Premium vs Dedicated) sets which of these apply.

Q: At 10k EMT files/month, why is a serverless plan usually cheap, and when would that flip?
A: 10,000 short validation executions per month is a tiny volume — each costs a fraction of a cent in executions plus a small GB-seconds charge — so a scale-to-zero plan is essentially free. It flips when files get huge or processing gets long: GB-seconds scale with memory times duration, so a 4 GB instance running for minutes per file, or always-ready instances kept warm 24/7, can dominate the bill. That crossover is exactly the glue-versus-engine line.

Q: What is a "GB-second" and why does it drive serverless cost?
A: A GB-second is one gigabyte of allocated memory consumed for one second of execution; serverless plans bill resource consumption in GB-seconds rather than per-VM-hour. Cost therefore scales with both how much memory each instance uses and how long the function runs, multiplied across all executions. This is why long-running, memory-hungry data movement is expensive on serverless even when execution count is low.

Q: What is Durable Functions and what problem does it solve?
A: Durable Functions is an extension of Azure Functions that lets you write stateful, long-running workflows in code in a serverless environment. The runtime manages state, checkpoints, retries, and recovery so a workflow can reliably run for seconds, days, or longer despite restarts. It solves the problem that plain functions are stateless and short-lived, which makes multi-step coordination (chaining, fan-out/fan-in, waiting for events) awkward and fragile to build by hand.

Q: What are the four function types in a Durable Functions app?
A: They are the client (or starter) function, which starts an orchestration and is triggered by a normal trigger like HTTP or a queue; the orchestrator function, which defines the workflow by scheduling other functions; the activity function, which does the actual units of work (I/O, computation) the orchestrator calls; and the entity function, which manages small pieces of explicit, addressable state. The orchestrator coordinates, activities execute, the client launches, and entities hold state.

Q: What does an orchestrator function do, and what does it notably not do?
A: An orchestrator function defines the control flow of a workflow — calling activity functions, waiting on timers, waiting for external events, and combining results — using ordinary code constructs like loops and conditionals. It notably does not perform I/O, call external services, or do non-deterministic work itself; all of that must be delegated to activity functions. The orchestrator is the brain that schedules; activities are the hands that act.

Q: What is an activity function?
A: An activity function is the basic unit of work in a durable orchestration — it is where you perform I/O, call external services, do CPU-heavy computation, or anything non-deterministic. Activities are invoked only by an orchestrator (via `CallActivityAsync` or the language equivalent), and their return values are recorded in the orchestration history so they are safe across replays. Restrictions that apply to orchestrators (no random, no clock, no I/O) do not apply to activities.

Q: What is the function-chaining pattern?
A: Function chaining runs a sequence of activity functions in a specific order, passing each one's output to the next, all coordinated by an orchestrator. For example: validate EMT file, then enrich with reference data, then compute a check, then publish — each step is an activity and the orchestrator awaits them in sequence. Durable Functions makes this reliable: if the host crashes between steps, replay resumes from where it left off without re-running completed activities.

Q: Describe the fan-out/fan-in pattern and a fund-data use for it.
A: Fan-out/fan-in runs many activity functions in parallel (fan-out) and then aggregates their results once all complete (fan-in), with the orchestrator awaiting the whole batch. A fund-data use is validating a batch of, say, 200 EMT files in parallel — one activity per file — then aggregating the pass/fail counts into a single reconciliation summary. The durable framework tracks all the parallel tasks and resumes correctly even if instances are recycled mid-batch.

Q: What is the async HTTP API pattern in Durable Functions?
A: It addresses long-running operations started over HTTP: a client function starts the orchestration and immediately returns `HTTP 202 Accepted` with a `Location` header pointing at a status-query endpoint, instead of blocking past the 230-second HTTP limit. The caller then polls that `statusQueryGetUri` to check whether the orchestration is running, completed, or failed, and to retrieve the result. This is the standard way to expose a multi-minute fund process behind a quick HTTP call.

Q: What is the monitor pattern in Durable Functions?
A: The monitor pattern is a recurring, flexible-interval poll inside an orchestration — the orchestrator checks a condition, then sleeps with a durable timer, then checks again, possibly for a long time or until a deadline. For example it could poll a transfer-agency endpoint until a settlement status appears, backing off between checks. Unlike a regular Timer trigger, the interval and lifetime are dynamic and stateful per orchestration instance.

Q: What is the human-interaction (approval) pattern, and which API enables it?
A: The human-interaction pattern pauses an orchestration to wait for an external signal — typically a human approval — using `WaitForExternalEvent`, often combined with a durable timer for a timeout/escalation. For instance, a NAV that breaches a tolerance could halt and wait for an oversight officer's approval event before publishing, escalating if no response arrives in time. The orchestration stays durably suspended without consuming compute while it waits.

Q: What are durable entities (the aggregator pattern) for?
A: Durable entities (entity functions) manage small pieces of explicit, addressable, long-lived state, processing operations one at a time to avoid race conditions. The aggregator pattern uses them to accumulate event data over time — for example tallying received-vs-expected EMT files per fund per day into a single entity. They give you stateful objects in a serverless model without standing up a database for tiny coordination state.

Q: Why must orchestrator functions be deterministic?
A: Because Durable Functions uses event sourcing and replay: the orchestrator code is re-executed from the beginning many times as the workflow progresses, replaying its history to rebuild local state. If the code produced different results on each run — different timestamps, random numbers, or external reads — the replay would diverge from the recorded history and corrupt the workflow. Determinism guarantees that every replay reproduces exactly the same sequence of decisions.

Q: What is the replay / event-sourcing mechanism behind Durable Functions?
A: Each time an orchestrator awaits a durable task (an activity call, a timer, an external event), the runtime checkpoints and the function may unload; when the awaited result is ready the orchestrator is re-run from the top, replaying the recorded history to fast-forward back to where it paused. Completed activity results are read from the history rather than re-executed. This event-sourced replay is what makes long-running, crash-resilient workflows possible on stateless compute.

Q: Why can't you call `DateTime.Now` (or `datetime.now()`) inside an orchestrator, and what do you use instead?
A: Time-of-day APIs are non-deterministic — each replay would read a different current time and break replay consistency. Instead you use the context's deterministic clock: `context.CurrentUtcDateTime` in .NET in-process, `TaskOrchestrationContext.CurrentDateTimeUtc` in .NET isolated, `context.current_utc_datetime` in Python, or `context.df.currentUtcDateTime` in JavaScript. These return the same value on every replay so any time-based logic is reproducible.

Q: Why can't you call `Guid.NewGuid()` or `uuid.uuid4()` directly in an orchestrator?
A: A fresh GUID/UUID is different on every call, so generating one in orchestrator code would produce a different value on each replay and break determinism. Instead use the replay-safe context API — `context.NewGuid()` in C#, `context.df.newGuid()` in JavaScript, `context.new_guid()`/`ctx.new_uuid()` in Python — which yields a deterministic Type 5 UUID, or generate the value inside an activity and return it. The activity's return value is stored in history and is therefore stable across replays.

Q: Why must you avoid I/O, network calls, and bindings inside an orchestrator function?
A: Because the orchestrator replays multiple times, any direct I/O — HTTP calls, database access, input/output bindings — would execute repeatedly and produce duplicate side effects and non-deterministic results. All such work belongs in activity functions, which run exactly once per logical step and have their results recorded. If you must call an HTTP API from the workflow, use the orchestrator's durable HTTP API rather than a raw client.

Q: How do you create a delay or timeout inside an orchestrator without blocking, and why not `Thread.Sleep`?
A: You use a durable timer — `context.CreateTimer(...)` in .NET, `ctx.create_timer(...)` in Python, `context.df.createTimer(...)` in JavaScript — which is replay-safe and does not consume execution time or incur charges while waiting. Blocking calls like `Thread.Sleep`, `time.sleep`, or `Task.Delay` are forbidden because they tie up the worker, cost money on consumption plans, and aren't part of the durable replay model. Durable timers let an orchestration wait minutes, hours, or days for free.

Q: Why are environment variables and static variables unsafe inside an orchestrator?
A: Both can change between replays — an env var might be updated and a static variable's value can drift across executions — making the orchestrator non-deterministic. Configuration an orchestrator needs must be passed in as orchestration input or returned from an activity function, and mutable shared state should be confined to activities or entities. Constants are fine; anything that can vary over time is not.

Q: In .NET, why must orchestrator code avoid extra threading and `ConfigureAwait(false)`?
A: The Durable Task Framework runs orchestrator code on a single thread to keep replay deterministic, and it cannot safely interact with other threads. Running continuations on a worker-pool thread — for example via `ConfigureAwait(continueOnCapturedContext: false)` — can cause non-deterministic execution or deadlocks. You should keep continuations on the orchestrator's original `SynchronizationContext` and never spin up your own threads in orchestrator code.

Q: What is a `NonDeterministicOrchestrationException`?
A: It is the exception the Durable Task Framework throws when it detects that orchestrator code used a non-orchestrator thread or otherwise behaved non-deterministically during replay. It is a best-effort guardrail — it catches some but not all determinism violations — so you must not rely on it to police your code. Seeing it usually means you did something like direct I/O, a wall-clock read, or threading inside an orchestrator.

Q: In Python, why must an orchestrator be a generator (`yield`) rather than an `async def` coroutine?
A: Python Durable Functions orchestrators must be declared as generators that `yield` on context APIs, because coroutine `async`/`await` semantics do not align with the replay model and would not behave deterministically. Declaring the orchestrator with `async def` and using `await ctx.call_activity(...)` simply won't work. The same idea applies to JavaScript, where orchestrators are synchronous generator functions, not `async`.

Q: What does "versioning" mean as a Durable Functions hazard?
A: Because a durable orchestration can run for days, months, or even indefinitely, deploying a code change that alters an in-flight orchestration's structure can break its replay and corrupt running instances. You must plan changes carefully — common strategies are side-by-side deployment or version-specific task hub names. This matters for long fund workflows: you can't naively redeploy an orchestrator while month-end reconciliations are mid-flight.

Q: What is a "task hub" in Durable Functions?
A: A task hub is the logically isolated set of storage resources (queues, tables, history) that a Durable Functions app uses to store orchestration state and pass messages. Different apps — or different versions — should use separate task hubs so their orchestrations don't collide. It is the backing store boundary that keeps one app's durable state independent of another's.

Q: How is a specific orchestration instance identified and queried?
A: Each orchestration has an instance ID (auto-generated or supplied at start), and the client function returns management URLs — notably a `statusQueryGetUri` — that let you query that instance's runtime status, output, and history. You poll the status endpoint to see whether it is Running, Completed, Failed, or Terminated. The instance ID is also how you raise external events to or terminate a specific running workflow.

Q: When do Azure Functions become an anti-pattern? State the core rule.
A: Functions become an anti-pattern for long-running work, heavy data movement, and stateful processing — anything that pushes against execution timeouts, consumes large GB-seconds, or needs durable in-process state. The core rule: functions are the glue tier (event handlers, light transforms, API shims); when the work is a sustained pipeline of moving or transforming large volumes, it belongs in a pipeline engine (ADF, Spark, a warehouse). Misplacing that line yields timeouts, cold-start pain, and surprise bills.

Q: Why is running heavy data movement inside a plain Consumption-plan function a bad idea specifically?
A: Heavy data movement tends to be long-running and memory-intensive, but the legacy Consumption plan caps execution at 10 minutes, so the job gets killed; even where it fits, GB-seconds for a large in-memory transform make it expensive, and you get no parallel-engine optimizations. A pipeline engine (Spark, ADF copy, a SQL warehouse) is built for bulk movement with checkpointing and scale-out. The function should trigger and hand off the bulk work, not perform it.

Q: A NAV-calculation batch that takes 40 minutes and reads gigabytes is proposed as an Azure Function. Why decline, and what do you suggest?
A: Decline because it is long-running, heavy, and stateful — the legacy Consumption timeout (10 min) would kill it, GB-seconds make it costly even on a longer-timeout plan, and a single function gives you no native parallelism or checkpointing. Suggest a pipeline engine (Spark/Databricks or a Synapse/SQL job, orchestrated by ADF) for the compute, with a small function acting only as the trigger that kicks it off and as the event emitter on completion. Glue triggers; the engine computes.

Q: Where exactly is the "glue versus engine" line for a function?
A: The function-glue side is event reaction, validation, routing, light enrichment, API shims, and emitting events — short, mostly stateless, bounded work. The pipeline-engine side is bulk transformation, large joins, multi-gigabyte data movement, and long stateful jobs that need scale-out, checkpointing, and a real execution engine. If a workload needs sustained compute, lots of memory, or long duration, it crosses the line into engine territory.

Q: Classify five workloads as glue or engine: (a) validate an EMT file on drop, (b) nightly reprocess a year of NAV history, (c) post a webhook when a fund is approved, (d) join 50M transactions to reference data, (e) resize a thumbnail.
A: Glue: (a) validate EMT on drop — short, event-driven; (c) post a webhook — a light API shim; (e) resize a thumbnail — small, stateless transform. Engine: (b) reprocess a year of NAV history — long, heavy, bulk; (d) join 50M transactions — a large set-based operation needing a query/Spark engine. The pattern: bounded reactions are glue, sustained bulk compute is engine.

Q: If a function is "stateful" — it accumulates state across invocations in memory — what is the architectural smell?
A: It's a smell because Azure Functions instances are stateless and ephemeral: there is no guarantee in-memory state survives between executions, and the platform can scale in or recycle instances at any time. Genuine cross-invocation state belongs in durable storage, a Durable Entities object, or an orchestration — not in static variables or instance fields. Relying on in-process state leads to lost-state bugs that appear only under scale-in.

Q: How does Durable Functions change the "anti-pattern" calculus for multi-step serverless workflows?
A: Plain functions are a poor fit for multi-step, long-running coordination, but Durable Functions extends them to handle exactly that — chaining, fan-out/fan-in, waits, and human approval — with managed state and replay. So a multi-step EMT-to-NAV coordination that would be fragile as ad-hoc queue-chained functions becomes a legitimate serverless workflow as a durable orchestration. The bulk compute still belongs in activities or external engines; Durable Functions handles the coordination, not the heavy lifting.

Q: What is the Azure Functions Core Tools and why use it for the "Do" exercise?
A: Azure Functions Core Tools is the command-line toolset for creating, running, and debugging functions locally for free, without deploying to Azure. The "Do" task — locally run an event-triggered function that validates an EMT drop and emits an event — uses it so you can iterate at zero cost and learn the trigger/binding model before incurring any cloud bill. It mirrors the managed Functions service so behavior carries over.

Q: On a native Ubuntu laptop, how do you typically install and verify Azure Functions Core Tools?
A: You install the `azure-functions-core-tools-4` package (Microsoft provides an apt feed for Ubuntu) and verify with `func --version`, which should report a 4.x version. You do not need WSL2 on native Ubuntu — that is only relevant on Windows; macOS uses Homebrew. Then `func init` scaffolds a project and `func start` runs the host locally.

Q: What does `func init` do versus `func new`?
A: `func init` scaffolds a new function-app project (host configuration, language worker setup, `host.json`, `local.settings.json`), while `func new` adds an individual function — prompting for a trigger template — into an existing project. So the typical flow is `func init MyApp --python`, then `cd MyApp`, then `func new` to add, say, a blob-triggered function. They operate at the app level and the function level respectively.

Q: What does `func start` do and what does its output tell you?
A: `func start` launches the Functions host locally and loads your functions, printing the bound trigger endpoints — for HTTP functions it shows the local URLs, and it logs when triggers fire. It is how you exercise the function end-to-end on your laptop before deploying. Seeing your function listed with its trigger on startup confirms the trigger and bindings were parsed correctly.

Q: What is `local.settings.json` and what must you never do with it?
A: `local.settings.json` holds local-development settings and connection strings — including `AzureWebJobsStorage` and your binding connections — under a `Values` collection, and is read only when running locally. You must never commit it to source control or rely on it in Azure, because it can contain secrets and is not deployed; production settings live in the function app's application settings. It is typically gitignored by the templates.

Q: For local development with no real Azure Storage, what stands in for `AzureWebJobsStorage`?
A: You run Azurite, the local Storage emulator, and set `AzureWebJobsStorage` to `UseDevelopmentStorage=true` (or the Azurite connection string). This lets blob/queue triggers and the runtime's internal bookkeeping work entirely offline. For the EMT-validation exercise you can drop a test file into an Azurite blob container to fire the trigger without touching the cloud.

Q: What is the role of the extension bundle in non-.NET functions?
A: The extension bundle is a versioned package, referenced in `host.json`, that brings in the binding extensions (Blob, Queue, Event Grid, Service Bus, etc.) so you don't have to install each one individually for Python/JavaScript/PowerShell apps. Without a compatible bundle, the runtime can't recognize a binding type and the function fails to load. For example using an Event Grid blob trigger requires a recent enough bundle/extension version.

Q: What does the `functionTimeout` setting in `host.json` control?
A: `functionTimeout` sets the maximum duration a single function execution may run before the runtime times it out and restarts the language worker process (or, for C# in-process, the host). It applies to function executions after the trigger fires, and its allowed range depends on the plan — bounded on Consumption (max 10 min), unbounded on Flex/Premium/Dedicated. Hitting it produces a timeout error and a worker restart, which can also disrupt other in-flight executions.

Q: What is the worker-process startup timeout, and can you change it?
A: Separate from `functionTimeout`, there is a 60-second timeout for the language-specific worker process to start, and it is not currently configurable. If a worker — for example a slow-loading Python app with heavy imports — takes longer than 60 seconds to initialize, startup fails. This is a gotcha when functions pull in large dependencies; keep cold-path initialization lean.

Q: A blob-triggered function "doesn't fire" when a file lands. What are the first things to check?
A: Check that the `AzureWebJobsStorage`/connection points at the right account and the path/container name is correct; confirm whether you're on the polling source (which can lag up to ~10 minutes when idle) and consider switching to the Event Grid source; verify the blob isn't already recorded in `azure-webjobs-hosts` as a processed receipt; and on Flex Consumption remember that only the Event Grid blob trigger is supported. Also confirm the extension bundle/version supports the trigger.

Q: A blob trigger fires repeatedly for the same EMT file. What is likely happening?
A: Most often the function is failing partway and the runtime is retrying (up to five times by default), so it appears to "fire repeatedly" before the blob lands in the poison queue. Make the function idempotent and check logs for the failure; if you genuinely deleted the blob receipt or reset the scan info, that would also cause reprocessing. Confirm by inspecting `azure-webjobs-hosts` receipts and the `webjobs-blobtrigger-poison` queue.

Q: An HTTP function returns a `502`/timeout after about 4 minutes even though the plan timeout is 30 minutes. Why?
A: Because HTTP-triggered functions are bounded by the ~230-second (about 3.8-minute) Azure Load Balancer idle timeout, independent of the plan's execution timeout. Long synchronous HTTP work will be cut off there regardless of `functionTimeout`. The fix is to return `202 Accepted` immediately and process asynchronously — for instance with the Durable Functions async HTTP pattern and a status-query endpoint.

Q: Your serverless bill is much higher than expected despite low file counts. What likely drives it?
A: GB-seconds, not execution count: a large instance memory size times long execution durations across the month dominates the bill, so a few heavy, long-running invocations cost far more than many short ones. Always-ready/prewarmed instances kept warm 24/7 also add a fixed floor. Investigate by checking per-function average duration and memory, and move heavy work out of the function into an engine.

Q: A Python function app fails to start locally with a worker error and no functions load. What do you check first?
A: First check the language-worker startup: heavy imports or a slow init can exceed the 60-second worker startup timeout, and a bad dependency or wrong Python version commonly breaks worker boot. Verify the virtual environment and dependencies, the `host.json` extension bundle, and the `func start` log for the actual worker error. In this study plan you run Python via `uv`, so ensure the interpreter and packages the Core Tools host uses are the intended ones.

Q: A Durable Functions orchestration produces inconsistent results or throws `NonDeterministicOrchestrationException`. First thing to check?
A: Check the orchestrator for non-deterministic code: wall-clock reads (`DateTime.Now`/`datetime.now()`), `Guid.NewGuid()`/`uuid4()`, random numbers, direct I/O or bindings, env/static variables, `Thread.Sleep`/`Task.Delay`, or extra threading. Move any of those into activity functions and use the context's deterministic APIs (`CurrentUtcDateTime`, `NewGuid()`, durable timers) instead. The exception specifically flags use of non-orchestrator threads during replay.

Q: After deploying a code change, in-flight durable orchestrations start failing on replay. What happened and how do you prevent it?
A: You changed the shape of the orchestrator (added/removed/reordered awaited steps) while instances were mid-flight, so their recorded history no longer matches the new code and replay breaks. Prevent it with a versioning strategy — deploy the new orchestrator side by side or under a new task-hub name and let old instances drain on the old version. Never make breaking orchestrator changes while long-running instances are active.

Q: An EMT file fails validation five times and then "disappears" from processing. Where did it go and how do you find it?
A: It went to the blob-trigger poison queue, `webjobs-blobtrigger-poison`, as a JSON message containing `FunctionId`, `BlobType`, `ContainerName`, `BlobName`, and `ETag`. Monitor or drain that queue to detect and triage failed files. In a regulated fund pipeline you'd alert on poison-queue depth so a malformed EMT submission is investigated rather than silently dropped.

Q: Why is "the function quietly grew into a shadow pipeline" a real architectural risk?
A: Because functions are easy to add, glue tends to accumulate — one validation grows a transform, then a join, then a schedule — until an unowned, untested mesh of functions is doing pipeline work it was never designed for, with no lineage, checkpointing, or observability. In a fund platform that means NAV/EMT data flowing through fragile, undocumented glue. The architect's job is to draw and police the glue-versus-engine line so glue stays glue.

Q: How does choosing Event Grid as the blob-trigger source change reliability versus polling for EMT files?
A: The Event Grid source reacts to storage events with much lower latency and doesn't depend on best-effort storage logs or periodic scans, so files are processed promptly and reliably rather than possibly waiting up to ~10 minutes (or being missed) under the polling model. It is also the only blob-trigger source supported on Flex Consumption. For a regulated, time-sensitive EMT drop, Event Grid is the dependable choice.

Q: What is the difference between an Event Grid trigger and an Event Grid-sourced blob trigger?
A: An `EventGridTrigger` is a general trigger that fires on any Event Grid event delivered to the function, where you handle the event payload yourself. An Event Grid-sourced blob trigger is still a `BlobTrigger` (with `source = EventGrid`) that uses Event Grid under the hood to deliver storage-change notifications but presents you the blob directly, as the classic blob trigger does. The latter keeps the convenient blob binding while gaining Event Grid's low latency.

Q: Why might you prefer a Queue trigger over a Blob trigger for decoupling EMT processing?
A: A Queue trigger gives you explicit control over message ordering, visibility timeout, retry/dequeue behavior, and back-pressure, and decouples the producer from the consumer so spikes are buffered. A pattern is to have a lightweight blob/Event Grid handler enqueue a work item and a separate queue-triggered function do the processing, which smooths load and isolates failures. The queue's built-in poison handling (`maxDequeueCount`) also makes retry semantics explicit.

Q: How do you control concurrency for a Blob trigger to avoid memory pressure?
A: With Storage extension 5.x and later you set `maxDegreeOfParallelism` in the `blobs` section of `host.json`; on older extensions you control it via the internal `queues` configuration. Lowering concurrency reduces simultaneous in-memory blob loads (helpful for large EMT/NAV files) at the cost of a larger processing backlog. Limits apply separately per blob-triggered function.

Q: Why does the Blob trigger require both blob and queue permissions on an identity-based connection?
A: Because the trigger mechanism itself relies on an internal Storage queue (for retries and poison-blob handling) in addition to reading the blob, so the managed identity needs `Storage Blob Data Owner` plus `Storage Queue Data Contributor`. Granting only blob access leaves the trigger unable to manage its internal queue and it fails to operate. This is a frequent least-privilege misconfiguration.

Q: In a Luxembourg fund-administration context, why is "scale to zero" attractive but the timeout a constraint for scheduled EMT processing?
A: Scale-to-zero is attractive because EMT/NAV files arrive on a schedule with quiet periods in between, so a serverless plan costs nothing when idle and bursts up when a batch lands. The timeout is a constraint because if any single file's processing is heavy or slow it can hit the plan's execution limit (10 minutes on legacy Consumption), so the function must stay lightweight and hand bulk work to an engine. The pattern: cheap event-driven glue, bounded per-file work.

Q: How would you keep a function "warm" to avoid cold starts for a latency-sensitive fund API shim?
A: Use a plan and feature that maintain warm capacity: configure always-ready instances on Flex Consumption, or use the Premium plan with its prewarmed instances, both of which keep at least one instance running so requests skip the cold-start path. You pay for the warm instance even when idle, which is the trade for predictable latency. On a pure scale-to-zero plan you'd instead accept occasional cold-start latency.

Q: Why is VNet integration a factor when choosing between legacy Consumption and Flex Consumption for a regulated platform?
A: Regulated fund platforms often require private network access to storage, databases, and key vaults rather than public endpoints. The legacy Consumption plan does not support outbound VNet integration or private endpoints, whereas Flex Consumption (and Premium/Dedicated) does. So for a DORA-conscious, network-isolated environment Flex Consumption is preferred over legacy Consumption partly for its networking capabilities, not just scale.

Q: What does "idempotent" mean for a blob-triggered validation function and why must you design for it?
A: Idempotent means processing the same file more than once produces the same end state without harmful duplicate side effects (no double-emitted events, no double-written rows). You must design for it because triggers retry on failure (up to five times) and blob receipts are a best-effort de-dup, not an absolute exactly-once guarantee. Use a dedup key (for example file name plus ETag) so a re-fired EMT validation doesn't publish the event twice.

Q: Are blob triggers exactly-once or at-least-once in practice?
A: In practice they are effectively at-least-once: blob receipts make the runtime avoid re-invoking for the same blob version under normal conditions, but retries, manual receipt deletion, and edge cases mean your code should not assume strict exactly-once delivery. Design the function to be idempotent so a duplicate invocation is harmless. Treating "no more than once" as a strict guarantee is a common and dangerous assumption.

Q: Why is the orchestrator/activity split also a cost and reliability strategy, not just a code constraint?
A: Pushing all I/O and heavy work into activities (which run once and are recorded) while the orchestrator only schedules means the orchestrator's frequent replays are cheap and side-effect-free, and durable timers let it wait for free. This both controls GB-seconds (no compute burned while waiting) and improves reliability (crash-safe resume). It is the same glue-versus-engine discipline applied inside a workflow: coordinate cheaply, execute deliberately.

Q: What is the relationship between Durable Functions and lesson 5.6.4 (orchestration)?
A: Durable Functions is the serverless orchestration option — code-first workflow coordination (chaining, fan-out/fan-in, waits) within the Functions runtime — and ties into the broader orchestration topic of 5.6.4, which weighs it against dedicated orchestrators like ADF or Airflow. The architect's choice is whether lightweight in-code durable orchestration suffices or a heavier external orchestrator is warranted. Knowing Durable Functions lets you place that boundary correctly.

Q: Give one example each of a workload that should be a plain function and one that should be a Durable Functions orchestration.
A: A plain function: validate a single EMT file on drop and emit an event — one trigger, bounded, stateless. A Durable Functions orchestration: coordinate a multi-step end-of-day flow that validates a batch of files in parallel, waits for an oversight approval, then publishes — multi-step, long-running, stateful coordination. The first is glue; the second is glue-with-coordination, with the heavy compute still delegated to activities or engines.

Q: How does the 60-minute scale-in grace period on Flex Consumption affect long-running functions?
A: On Flex Consumption (and Premium) a function execution in progress during a scale-in event is given up to a 60-minute grace period to finish before its instance is removed, and a separate 10-minute grace during platform updates. This means moderately long executions can complete during scale events, but it is not a license to run truly long jobs — relying on it for sustained heavy work is still an anti-pattern. It is a safety margin, not a design target.

Q: Why is the LEI/ISIN-aware validation in an EMT-drop function still "glue" and not "engine"?
A: Because validating identifiers and required fields on a single incoming file is a bounded, mostly stateless check — it inspects the file, applies rules, and emits a pass/fail event — exactly the light-transform/event-handler shape functions are built for. It does not move bulk data, run large joins, or maintain long-lived state. The moment you'd cross into engine territory is enriching against a 50-million-row reference set or reprocessing history in bulk.

Q: Summarize the architect's one-line rule from this lesson for placing serverless functions.
A: Use Azure Functions as the glue tier — event handlers, light transforms, API shims, and serverless orchestration via Durable Functions — and keep long-running, stateful, memory-heavy, bulk-data work in dedicated pipeline engines. Choose the hosting plan (Flex Consumption by default, Premium/Dedicated for warmth or isolation) by matching its timeout, scaling, networking, and cost profile to the workload. Police that line, or glue silently grows into an unmaintainable shadow pipeline with timeouts and surprise bills.
