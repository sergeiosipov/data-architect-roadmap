# Phase 0 · 0.4 Git & GitHub (version control from zero) — 100 self-test questions

Q: What is version control, in one sentence?
A: A system that records changes to files over time so you can recall specific versions later, see who changed what, and collaborate without overwriting each other's work.

Q: What is Git?
A: A distributed version control system that tracks file changes as a series of snapshots (commits), where every clone contains the full history rather than depending on a central server.

Q: What is the difference between Git and GitHub?
A: Git is the version control tool that runs locally on your machine; GitHub is a web service that hosts Git repositories remotely and adds collaboration features like pull requests, issues, and access control.

Q: What are Git's "three states" a file can be in?
A: Modified (changed but not staged), staged/indexed (marked to go into the next commit), and committed (safely stored in the local history/database).

Q: What are the three corresponding areas in a Git project?
A: The working tree (the files you edit), the staging area/index (the snapshot you're preparing), and the `.git` directory (the repository where committed history lives).

Q: What is the working tree?
A: The directory of actual files you can see and edit on disk — a single checked-out version of the project that Git compares against the index and history.

Q: What is the staging area (index)?
A: An intermediate area that holds the exact set of changes you've selected for the next commit, letting you craft a commit from a subset of your working-tree changes.

Q: What does `git init` do?
A: It creates a new, empty Git repository in the current directory by making a `.git` subfolder, after which Git starts tracking changes there.

Q: What does `git clone <url>` do?
A: It downloads a full copy of an existing remote repository, including all history and branches, and sets up a remote named `origin` pointing back to the source.

Q: What does `git status` show?
A: Which branch you're on, which files are staged, which are modified but unstaged, which are untracked, and whether your branch is ahead of or behind its remote.

Q: How do you stage a single file called `nav_loader.py`?
A: Run `git add nav_loader.py`, which copies that file's current content into the staging area for the next commit.

Q: How do you stage all changes in the current directory and below?
A: Run `git add .` (or `git add -A` to also catch deletions across the whole repo) to stage everything modified, new, or removed.

Q: What does `git commit` do?
A: It permanently records the staged snapshot into the repository's history as a new commit, along with a message, author, and timestamp.

Q: What is `git commit -m "message"` for?
A: It commits the staged changes and supplies the commit message inline, avoiding the editor that would otherwise open.

Q: What does `git commit -a -m "message"` do, and what's the catch?
A: It stages and commits all tracked files that are modified in one step; the catch is it does NOT include new untracked files, which must still be added with `git add`.

Q: What makes a good commit message for a fund-data ETL repo?
A: A concise imperative summary line under ~50 characters (e.g. `Fix ISIN validation for null tickers`) optionally followed by a blank line and a body explaining the why.

Q: What is a commit hash (SHA)?
A: A 40-character SHA-1 (or SHA-256) checksum that uniquely identifies a commit based on its content, parent, author, and message; you can usually abbreviate it to the first 7 characters.

Q: What does `git log` do?
A: It lists the commit history of the current branch in reverse chronological order, showing each commit's hash, author, date, and message.

Q: How do you get a compact one-line-per-commit history?
A: Run `git log --oneline`, which shows the abbreviated hash and the first line of each commit message.

Q: What does `git log --oneline --graph --all` show?
A: A compact ASCII graph of all branches and their merge structure, useful for visualising how branches diverged and came back together.

Q: What does `git diff` (with no arguments) show?
A: The differences between your working tree and the staging area — i.e. the changes you have made but not yet staged.

Q: What does `git diff --staged` (or `--cached`) show?
A: The differences between the staging area and the last commit — i.e. exactly what will be recorded if you commit now.

Q: How do you see what changed in a specific commit?
A: Run `git show <commit-hash>`, which displays that commit's metadata and the diff it introduced.

Q: How do you compare two branches?
A: Run `git diff branchA..branchB` to see the changes that differ between them, or `git diff main` while on a feature branch.

Q: In a unified diff, what do lines starting with `+` and `-` mean?
A: `+` lines were added in the newer version and `-` lines were removed from the older version; context lines without a sign are unchanged.

Q: What is a branch in Git?
A: A lightweight, movable pointer to a specific commit; creating a branch just makes a new pointer, so it's cheap and fast.

Q: What is HEAD?
A: A pointer to the commit you currently have checked out, usually by pointing at the current branch, which in turn points at its latest commit.

Q: How do you create a new branch called `feature/emt-parser`?
A: Run `git branch feature/emt-parser` to create it, or `git switch -c feature/emt-parser` to create and switch to it in one step.

Q: What is the modern command to switch branches, and the older one?
A: The modern command is `git switch <branch>`; the older, multi-purpose command is `git checkout <branch>`.

Q: How do you list all local branches and see which one you're on?
A: Run `git branch`; the current branch is marked with an asterisk (`*`).

Q: How do you delete a fully merged local branch?
A: Run `git branch -d <branch>`; Git refuses if it isn't merged, in which case `git branch -D <branch>` force-deletes it.

Q: What does `git merge <branch>` do?
A: It integrates the changes from `<branch>` into your current branch, creating either a fast-forward or a new merge commit depending on the history.

Q: What is a fast-forward merge?
A: When the current branch has no new commits since the other branch diverged, Git simply moves the branch pointer forward to the other branch's tip — no new commit is created.

Q: When does Git create a merge commit instead of fast-forwarding?
A: When both branches have new commits since they diverged, Git must combine the two histories with a new commit that has two parents.

Q: How can you force a merge commit even when a fast-forward is possible?
A: Use `git merge --no-ff <branch>`, which always creates a merge commit to preserve the fact that a branch existed.

Q: What is a merge conflict?
A: A situation where the same lines (or a delete-vs-edit) changed differently on both branches, so Git can't automatically combine them and asks you to resolve it.

Q: What do the conflict markers `<<<<<<<`, `=======`, and `>>>>>>>` mean?
A: `<<<<<<<` to `=======` is your current branch's version (HEAD), `=======` to `>>>>>>>` is the incoming branch's version, and you edit the file to keep the correct result.

Q: How do you finish resolving a merge conflict?
A: Edit the files to the desired final content, remove all conflict markers, `git add` the resolved files, then run `git commit` to complete the merge.

Q: How do you abort a merge that has conflicts you don't want to resolve right now?
A: Run `git merge --abort`, which restores the branch to the state it was in before you started the merge.

Q: A merge conflict appears in `etl/load_nav.py` — what's the safe first step?
A: Run `git status` to list which files are conflicted, then open each one and look for the `<<<<<<<` markers to see both versions before deciding what to keep.

Q: What is a remote in Git?
A: A named reference to another copy of the repository, typically hosted elsewhere (like on GitHub), that you push to and pull from.

Q: What is `origin`?
A: The conventional default name Git gives to the remote you cloned from; it's just a name, not a special keyword.

Q: How do you list your configured remotes and their URLs?
A: Run `git remote -v`, which prints each remote name with its fetch and push URLs.

Q: What does `git push` do?
A: It uploads your local commits on the current branch to the corresponding branch on the remote, updating the remote's history.

Q: What does `git push -u origin main` do, and why the `-u`?
A: It pushes `main` to `origin` and the `-u` (`--set-upstream`) records the tracking relationship so future `git push`/`git pull` need no arguments.

Q: What does `git fetch` do?
A: It downloads new commits and branch updates from the remote into your local remote-tracking branches (like `origin/main`) without changing your working tree or local branches.

Q: What does `git pull` do, and how does it relate to fetch?
A: It runs `git fetch` followed by a merge (or rebase) of the upstream branch into your current branch — i.e. it downloads and integrates remote changes in one step.

Q: What is the difference between `git fetch` and `git pull` in terms of safety?
A: `git fetch` is non-destructive and just updates your view of the remote, letting you inspect before integrating; `git pull` immediately changes your working branch, which can cause unexpected merges.

Q: What is a remote-tracking branch like `origin/main`?
A: A local read-only pointer that records where the remote's `main` branch was the last time you fetched or pulled; it's how Git knows whether you're ahead or behind.

Q: Your `git push` is rejected as "non-fast-forward" — what does that mean and what do you do?
A: The remote has commits you don't have locally, so pushing would lose them; run `git pull` (fetch + integrate) to incorporate them, resolve any conflicts, then push again.

Q: What is `.gitignore`?
A: A file listing path patterns that Git should not track, used to keep generated files, secrets, and local clutter out of the repository.

Q: Give three things you'd typically put in `.gitignore` for a Python fund-data project.
A: Virtual environments and caches (`.venv/`, `__pycache__/`), secrets/config (`.env`, credentials), and large data or output files (`*.csv`, `data/raw/`).

Q: Why must credentials for the transfer-agency API never be committed?
A: Once committed they live in history forever and anyone with repo access (or a leaked clone) can read them; they must be ignored via `.gitignore` and stored in environment variables or a secrets manager.

Q: A file is already tracked but you just added it to `.gitignore` — why is it still showing changes?
A: `.gitignore` only affects untracked files; you must run `git rm --cached <file>` to stop tracking it, then commit, after which the ignore rule takes effect.

Q: What does a pattern like `data/raw/` (with trailing slash) match in `.gitignore`?
A: It matches the directory named `data/raw` and everything inside it, ignoring the whole folder's contents.

Q: What does `*.csv` in `.gitignore` do, and how do you keep one exception?
A: It ignores all files ending in `.csv`; to keep a specific one, add a negation line like `!schema_sample.csv` after it.

Q: What is GitHub Flow?
A: A lightweight workflow: create a branch off `main` for each change, commit to it, open a pull request, get review/CI, then merge to `main` and deploy — keeping `main` always deployable.

Q: What is a pull request (PR)?
A: A GitHub feature that proposes merging one branch into another, providing a place for discussion, code review, automated checks, and a record of the change before it's merged.

Q: What is the typical lifecycle of a PR?
A: Push a feature branch, open a PR against `main`, address review comments and failing checks with new commits, get an approval, then merge and delete the branch.

Q: What are the common PR merge strategies on GitHub?
A: "Create a merge commit" (preserves all commits plus a merge commit), "squash and merge" (combines the branch into one commit), and "rebase and merge" (replays commits onto `main` with no merge commit).

Q: Why is squash-and-merge popular for feature branches?
A: It collapses messy work-in-progress commits into a single, clean commit on `main`, keeping the main history readable while preserving the full detail in the PR.

Q: What is a "draft" pull request used for?
A: To open a PR early for visibility or to run CI while signalling it isn't ready for review or merge yet.

Q: What is a fork on GitHub?
A: Your own server-side copy of someone else's repository, which lets you push changes and open pull requests back to the original when you lack write access.

Q: What is the difference between `git restore` and `git revert`?
A: `git restore` discards changes in the working tree or unstages files (it doesn't make a commit); `git revert` creates a new commit that undoes a previous commit, preserving history.

Q: How do you discard unstaged changes to a single file `report.sql`?
A: Run `git restore report.sql`, which overwrites it with the version from the index (or last commit), losing your unsaved edits to it.

Q: How do you unstage a file you accidentally `git add`-ed?
A: Run `git restore --staged <file>` (or the older `git reset HEAD <file>`), which removes it from the index but keeps your working-tree changes.

Q: What does `git revert <commit>` do?
A: It creates a brand-new commit that applies the inverse of the specified commit, undoing its effect while leaving the original commit in history.

Q: Why is `git revert` the safe way to undo a commit that's already pushed/shared?
A: Because it adds a new commit rather than rewriting history, so collaborators' clones stay consistent and nobody's work is dropped underneath them.

Q: What are the three modes of `git reset`?
A: `--soft` moves the branch pointer but keeps changes staged; `--mixed` (default) moves the pointer and unstages changes but keeps them in the working tree; `--hard` moves the pointer and discards all changes.

Q: What does `git reset --hard HEAD~1` do, and why is it dangerous?
A: It moves the current branch back one commit and discards that commit's changes from the index and working tree; it's dangerous because uncommitted work is permanently lost and it rewrites history.

Q: What does `HEAD~1` (or `HEAD~3`) mean?
A: `HEAD~1` is the commit one step before HEAD (its parent), and `HEAD~3` is three commits back along the first-parent line.

Q: Why should you never `git reset` (rewrite) commits that others have already pulled?
A: Rewriting shared history changes commit hashes, so collaborators' branches diverge and pulling becomes a tangle of conflicts and duplicated commits; use `git revert` instead.

Q: What is `git push --force` and when is it risky?
A: It overwrites the remote branch with your local version, discarding remote commits; it's risky on shared branches because it can erase teammates' pushed work.

Q: What safer alternative to `--force` should you prefer?
A: `git push --force-with-lease`, which refuses to overwrite if the remote has commits you haven't seen, protecting against clobbering someone else's recent push.

Q: What is `git reflog` and why is it a lifesaver?
A: It logs where HEAD and branch tips have pointed recently, including "lost" commits after a bad reset, so you can recover them with `git reset` or `git checkout` to the recorded hash.

Q: You ran `git reset --hard` and lost a commit — how do you try to recover it?
A: Run `git reflog` to find the commit's hash from before the reset, then `git checkout <hash>` or `git reset --hard <hash>` (or branch from it) to restore it.

Q: What is `git stash`?
A: A command that shelves your uncommitted working-tree and staged changes onto a stack and reverts your tree to a clean state, so you can switch tasks without committing.

Q: How do you get stashed changes back?
A: Run `git stash pop` to reapply the most recent stash and remove it from the stack, or `git stash apply` to reapply without removing it.

Q: What does `git rm <file>` do versus deleting the file in your file manager?
A: `git rm` removes the file from disk and stages the deletion in one step; deleting it manually only changes the working tree and still requires `git add`/`git rm` to stage the removal.

Q: What is the difference between a tracked, untracked, and ignored file?
A: A tracked file is one Git already knows about from a prior commit or staging; untracked is a new file Git sees but isn't tracking; ignored is an untracked file matched by `.gitignore` so Git hides it.

Q: How do you tell Git your name and email for commit authorship?
A: Run `git config --global user.name "Your Name"` and `git config --global user.email "you@example.com"`; the `--global` flag applies it to all your repos.

Q: What's the difference between `--global`, `--local`, and `--system` config?
A: `--system` applies to all users on the machine, `--global` to all repos for your user, and `--local` (the default) only to the current repository, with local overriding global overriding system.

Q: What is a detached HEAD state?
A: When HEAD points directly at a commit instead of a branch (e.g. after `git checkout <hash>`); new commits there belong to no branch and can be lost unless you create one.

Q: How do you save work made in a detached HEAD?
A: Create a branch at the current position with `git switch -c <new-branch>` (or `git branch <name>`) before switching away, so the commits are anchored.

Q: What does `git log --oneline origin/main..main` tell you?
A: It lists the commits you have on local `main` that haven't been pushed to `origin/main` yet — i.e. what a push would upload.

Q: What does "your branch is ahead of 'origin/main' by 2 commits" mean?
A: You have made 2 local commits that aren't on the remote yet; running `git push` will send them.

Q: What does "your branch is behind 'origin/main' by 3 commits" mean?
A: The remote has 3 commits you don't have locally; running `git pull` will bring them in.

Q: What does "your branch and 'origin/main' have diverged" mean?
A: Both your local branch and the remote have commits the other lacks, so you'll need to merge or rebase to reconcile them before pushing.

Q: What is `git rebase` at a high level?
A: It replays your branch's commits on top of another base commit, producing a linear history as if you'd started your work from the newer base.

Q: Why prefer rebase over merge for keeping a feature branch up to date locally?
A: Rebase avoids extra merge commits and yields a clean, linear history; but it rewrites your branch's commits, so you shouldn't rebase commits that others have already based work on.

Q: What is the golden rule of rebasing?
A: Never rebase (or otherwise rewrite) commits that have been pushed to a shared branch others rely on; only rebase local, unshared commits.

Q: What is a tag in Git, and when would a fund-data team use one?
A: A tag is a named pointer to a specific commit, typically used to mark releases (e.g. `v2025.06` for the NAV pipeline version that went to production).

Q: What's the difference between a lightweight tag and an annotated tag?
A: A lightweight tag is just a name pointing at a commit; an annotated tag (`git tag -a v1.0 -m "..."`) is a full object with tagger, date, and message, and is preferred for releases.

Q: How do you create a repository on GitHub from existing local code?
A: Create an empty repo on GitHub, then run `git remote add origin <url>` locally and `git push -u origin main` to upload your existing history.

Q: What is the role of CI checks on a pull request?
A: Automated workflows (e.g. GitHub Actions) run tests, linters, and builds on the PR's commits so reviewers and authors see whether the change is safe before merging.

Q: What is a `CODEOWNERS` file?
A: A file that maps paths in the repo to people or teams who are automatically requested for review when those paths change in a PR.

Q: What does "protected branch" mean on GitHub?
A: A branch (often `main`) configured with rules such as required reviews, passing checks, and no force-pushes, to prevent unreviewed or destructive changes.

Q: A teammate force-pushed and rewrote `main`; your local `main` won't pull cleanly — what's the recovery?
A: Use `git reflog`/`git fetch` to inspect, then reset your local branch to the new remote with `git reset --hard origin/main` after backing up any local-only commits to a separate branch.

Q: Why commit early and often in small, logical units?
A: Small commits make history easy to read, reviews easier, reverts surgical, and conflicts smaller — far better than one giant commit mixing the EMT parser, NAV loader, and config edits.

Q: What's the first thing to check if `git push` asks repeatedly for a password or fails to authenticate?
A: Verify your remote URL and credentials — GitHub no longer accepts account passwords over HTTPS, so use a personal access token or set up SSH keys, and confirm `git remote -v` points to the right place.
