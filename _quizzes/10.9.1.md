# Phase 6 · 10.9.1 + 10.9.2 Data observability (pipeline health) — 100 self-test questions

Q: What is data observability, in one sentence?
A: Data observability is the practice of continuously monitoring the health of data assets and pipelines — detecting freshness, volume, schema, distribution, and lineage problems — so that silent failures are caught before consumers see wrong numbers. It answers "is the data itself healthy and trustworthy right now?" rather than "is the server up?". The goal is to surface a stalled load or a shrunken table proactively, not after a client queries a wrong NAV.

Q: How does data observability differ from the infrastructure observability you set up in Phase 5?
A: Infra observability watches the compute and transport plane — CPU, memory, container restarts, request latency, error rates — while data observability watches the data plane: is the table fresh, did the row count collapse, did a column type change, did a distribution shift? A pipeline can be green at the infra level (every job exited zero) yet produce stale or shrunken data, so the two layers are complementary, not substitutes.

Q: Why is data observability described as "the data plane's heartbeat"?
A: Because it provides a continuous pulse on whether data is actually flowing and arriving correctly, the same way a heartbeat signals a living system. A missed beat — a load that did not land, or a connector that silently stalled — is exactly the kind of failure that infra dashboards miss but data observability catches. Without it you only learn the heart stopped when a downstream report shows yesterday's prices.

Q: What are the five pillars of data observability?
A: The five pillars are freshness, volume, schema, distribution, and lineage. Freshness asks "is the data recent enough?", volume asks "is the row count in range?", schema asks "did the structure change unexpectedly?", distribution asks "do the values look statistically normal?", and lineage maps how datasets depend on each other so you can trace impact. Together they cover the most common silent-failure classes in a pipeline.

Q: Define the freshness pillar.
A: Freshness measures how recently a table was updated, typically by comparing the maximum timestamp (or load time) against an expected cadence. A freshness failure means data is stale — the table exists and the job may even have run, but the newest record is older than the SLA allows. In a fund context, a stale prices table can silently produce a NAV computed from yesterday's market data.

Q: Define the volume pillar.
A: Volume tracks the number of records (row count) landing in a table over time and flags sudden increases or decreases versus the historical baseline. A drop can mean a truncated or partial load, duplication can mean a re-run gone wrong, and a spike can mean a fan-out bug. Volume monitoring catches "shrunken load" failures where the job succeeds but only loads a fraction of the expected rows.

Q: Define the schema pillar.
A: Schema observability detects structural changes to a table — added, dropped, or renamed columns, and changed data types — that can silently break downstream models. The danger is that an upstream vendor renames a column and a join quietly returns nulls instead of erroring. Schema monitoring turns that silent drift into an explicit, alertable event.

Q: Define the distribution pillar.
A: Distribution monitoring watches the statistical shape of column values — null rate, min, max, mean, cardinality, string length, category frequencies — and flags when those metrics drift outside their normal range. Unlike volume, the row count can be perfectly normal while the values themselves are corrupted, for example a currency column suddenly going 90% null. It catches content-level rot that structural checks miss.

Q: Define the lineage pillar.
A: Lineage maps the dependency graph between datasets — which sources feed which models feed which dashboards — so that when one node fails you can immediately see upstream causes and downstream impact. It turns "table X is stale" into "table X is stale because connector Y stalled, and it affects the NAV dashboard and the EMT export." Lineage is the connective tissue that makes the other four pillars actionable during an incident.

Q: Which pillar would catch a Debezium connector that silently stalled and stopped streaming changes?
A: Freshness is the primary detector — the target table's newest record stops advancing while the clock keeps moving, so the max timestamp falls behind the SLA. Volume may also catch it once the expected periodic load fails to arrive. The infra layer might show the connector task still "running," which is exactly why a data-plane freshness check is needed.

Q: Which pillar catches a "shrunken load" where only 10% of expected rows arrive?
A: Volume is the detector — the row count drops far below its historical baseline even though the job exited successfully. Freshness would not catch it because the few rows that did land are recent, and schema would not catch it because the structure is unchanged. This is the classic case where only volume anomaly detection saves you.

Q: Which pillar catches an upstream vendor renaming a column from `nav_price` to `price_nav`?
A: The schema pillar catches it as a column drop plus a column add. Downstream models referencing `nav_price` would otherwise either error loudly (best case) or silently produce nulls if the reference is loose. Schema-change detection makes the rename an explicit incident before it corrupts a report.

Q: Which pillar catches a feed where the row count is normal but the `currency_code` column is suddenly 80% null?
A: The distribution pillar catches it via the null-rate metric on that column. Volume sees a normal count, schema sees the column still present with the same type, so neither fires — only distribution monitoring notices that the content degraded. A null currency code can mis-price an entire sub-fund, so this is a high-value detector.

Q: A NAV is computed from yesterday's prices even though every dbt job ran green. Which observability gap let this happen?
A: A missing freshness check on the prices source table. The jobs succeeded (infra green, dbt green) but the upstream feed never refreshed, so the model consumed stale inputs. Freshness anomaly detection on the source — not just on the final model — is what closes this gap.

Q: What is a "silent failure" in pipeline terms?
A: A silent failure is one where the pipeline reports success — jobs exit zero, tests pass, no error is raised — yet the data produced is wrong: stale, incomplete, or structurally drifted. They are dangerous precisely because nothing alerts; the failure surfaces only when a human or client notices a wrong number. Data observability exists to convert silent failures into loud, owned alerts.

Q: Name three common silent-failure classes that data observability targets.
A: Stale data (a table that stopped refreshing despite green jobs), shrunken/partial loads (far fewer rows than expected), and schema drift (an upstream column renamed, dropped, or retyped). Each maps cleanly to a pillar — freshness, volume, and schema respectively — and each can pass through ordinary success checks unnoticed. A fourth is distribution rot, such as a column going mostly null.

Q: Why is "the job succeeded" insufficient as a data-quality signal?
A: Because exit code zero only proves the code ran without crashing, not that it ran on the right or complete data. A `COPY` into a table can succeed while loading an empty file; a Debezium task can stay "running" while emitting nothing. Observability adds outcome checks on the data itself, independent of the orchestrator's success signal.

Q: What is Elementary?
A: Elementary is a dbt-native data observability tool that runs anomaly-detection tests inside your existing dbt project and produces an observability report and alerts. It collects dbt run artifacts and test results into tables in your warehouse, then layers freshness, volume, schema, and distribution monitoring on top. The open-source package is `elementary` (repo `elementary-data/dbt-data-reliability`), complemented by the `edr` CLI.

Q: What does "dbt-native" mean for Elementary?
A: It means Elementary runs as a normal dbt package whose anomaly checks are authored and executed as ordinary dbt tests in your `.yml` files, using `dbt test` and the same `ref`/`source` graph. There is no separate engine to deploy — the monitors live alongside your models and run in the same invocation. This keeps observability in version control and CI rather than in a disconnected external tool.

Q: What two components make up Elementary's open-source offering?
A: The dbt package `elementary` (which adds tests, on-run-end artifact collection, and the metadata tables) and the `edr` command-line tool (which queries those tables to generate the HTML observability report and send alerts). The package runs inside `dbt`, while `edr` runs after the dbt invocation. Cloud-only features such as column-level lineage and AI triage sit on top of this OSS core.

Q: How do you add Elementary to a dbt project?
A: You add the package to `packages.yml` (the `elementary` / `elementary-data/dbt-data-reliability` entry), run `dbt deps` to install it, then add the on-run-end hook to `dbt_project.yml` and run `dbt run --select elementary` to build its models. After that, anomaly tests are configured in your model `.yml` files and run with `dbt test`. The `edr` CLI is installed separately via `pip` to generate reports.

Q: What does Elementary's `on-run-end` hook do, and where is it configured?
A: The `on-run-end` hook is set in `dbt_project.yml` and runs after every `dbt run`/`dbt test`, parsing dbt's artifacts (`run_results.json`, `manifest.json`) and loading model run results and test results into Elementary's tables. This is how Elementary builds a historical record of executions without you instrumenting anything manually. If the hook is missing, the metadata tables stay empty and the report has nothing to show.

Q: Into which schema does Elementary write its models and metadata tables?
A: By default Elementary writes to a dedicated `elementary` schema (commonly your target schema suffixed with `_elementary`), keeping its run-results, test-results, and monitoring tables separate from your business models. Keeping it isolated avoids cluttering analytics schemas and makes it easy to grant the `edr` CLI read access to just that schema. The exact name follows your dbt custom-schema configuration.

Q: What is the `edr` CLI?
A: `edr` (Elementary Data Reliability) is the command-line tool, installed via `pip install elementary-data`, that connects to your warehouse using a dbt profile, queries Elementary's tables, and produces outputs. Its main jobs are generating the HTML observability report (`edr report`) and sending alerts for failures and anomalies (`edr monitor`). It is the piece that turns the collected metadata into something a human or a Slack channel sees.

Q: What does `edr report` produce?
A: `edr report` generates a single self-contained HTML file — the observability dashboard — summarizing model run history, test results, anomalies, freshness, volume trends, and lineage. Being one HTML file, it can be opened locally or uploaded to S3/GCS/Azure Blob or a web server for the team to view. It is the human-facing window into the metadata Elementary has been collecting.

Q: What does `edr monitor` do?
A: `edr monitor` reads recent test results and anomalies from Elementary's tables and sends alerts about new failures to the configured channel, such as Slack or Microsoft Teams. It is typically scheduled to run after your dbt jobs (for example in Airflow or a cron) so incidents are pushed to people rather than waiting to be discovered. It is the routing mechanism that connects detection to notification.

Q: How are Elementary anomaly tests configured in a dbt project?
A: They are declared like any dbt test — under a model's or column's `tests:`/`data_tests:` block in a `.yml` file — using Elementary test names such as `volume_anomalies` or `freshness_anomalies`. Model-level tests (volume, freshness, schema) attach to the model; column-level tests (column anomalies) attach to a specific column. Because they are ordinary dbt tests, they run with `dbt test` and respect selection syntax like `--select`.

Q: Name the Elementary volume anomaly test and what it does.
A: The test is `volume_anomalies`. It computes the row count of a model per time bucket, builds a baseline from the training period, and flags buckets whose counts deviate anomalously from history. This is what catches partial loads, accidental truncation, or duplication where the structure and freshness look fine but the count is wrong.

Q: Name the Elementary freshness anomaly test and what it monitors.
A: The test is `freshness_anomalies`. It monitors the latest update timestamp of a table per time bucket and flags when freshness deviates from the historical pattern, for example a load arriving far later than usual or not at all. It is the primary guard against stale source data feeding downstream calculations.

Q: What is the `event_freshness_anomalies` test for?
A: `event_freshness_anomalies` measures the gap between an event's own timestamp (when something happened) and when it was loaded into the table, then flags anomalies in that lag. It is suited to event/streaming data where you care about ingestion latency, not just the wall-clock age of the newest row. For example, it can catch a trade-event stream whose events are arriving but with a growing delay.

Q: What does the `dimension_anomalies` test detect?
A: `dimension_anomalies` monitors the distribution of row counts across the values of one or more dimension columns (a `GROUP BY`), flagging when the mix shifts unexpectedly. For instance, if a fund-administration feed normally splits 60/40 across two custodians and suddenly goes 95/5, this test fires even though the total volume may be unchanged. It catches category-level skew that aggregate volume misses.

Q: What do `column_anomalies` and `all_columns_anomalies` tests monitor?
A: They monitor column-level data metrics over time — such as null rate, percentage of zeros/negatives, min, max, average, and string length — and flag anomalous shifts. `column_anomalies` targets a chosen column, while `all_columns_anomalies` applies the monitoring across all (or all matching) columns of a model. These implement the distribution pillar at the column granularity.

Q: How does Elementary detect schema changes?
A: It provides `schema_changes` (and `schema_changes_from_baseline`) tests that compare a table's current columns and types against the previously observed schema (or a defined baseline) and flag additions, removals, and type changes. This converts silent upstream restructuring into an explicit test failure. It is the dbt-native implementation of the schema pillar.

Q: What statistical method does Elementary use for anomaly detection by default?
A: Elementary uses a Z-score approach: it computes a metric per time bucket, derives a mean and standard deviation from the training history, and scores how many standard deviations the current value sits from that mean. Values with an absolute Z-score at or above the threshold are marked anomalous. The default `anomaly_score_threshold` is 3, meaning a deviation of three standard deviations or more.

Q: What is the default `anomaly_score_threshold` in Elementary, and what does raising it do?
A: The default is 3 (three standard deviations). Raising it makes the test less sensitive — fewer points are flagged, reducing false positives but risking missed real anomalies; lowering it makes it more sensitive with the opposite trade-off. You tune it per metric to balance alert noise against detection coverage for that specific dataset.

Q: What is the `training_period` in an Elementary anomaly test?
A: The `training_period` (often expressed as `days_back` or a training window) is the span of historical data used to learn the baseline mean and standard deviation against which current buckets are scored. Too short a window makes the baseline noisy and unstable; too long a window can drown out a recent legitimate level shift. It defines "what normal looks like" for the detector.

Q: What is the `detection_period` in an Elementary anomaly test?
A: The `detection_period` is the recent window of time buckets that are actually evaluated for anomalies against the trained baseline. The training period teaches normality and the detection period is where you look for deviations. Keeping the detection period aligned with your run cadence ensures each new load gets scored shortly after it lands.

Q: What is a `time_bucket` and why does its size matter?
A: A `time_bucket` is the interval Elementary groups data into when computing a metric over time, for example one bucket per day or per hour. The bucket size sets the resolution of detection: daily buckets smooth out intraday noise but delay detection, while hourly buckets catch problems faster but can be noisier. You match the bucket to the table's natural load cadence.

Q: Why does Elementary need historical data before anomaly tests are useful?
A: Anomaly detection is baseline-relative — it needs enough history in the training period to estimate a stable mean and standard deviation before it can judge whether a new value is unusual. On a brand-new table with only a day or two of data, the baseline is too thin and the test will either skip or produce noisy results. Coverage matures as the metadata tables accumulate runs.

Q: A `volume_anomalies` test fires the morning after you backfilled a table. Why, and is it a real incident?
A: The backfill loaded an unusually large number of rows into recent buckets, so the row count deviates sharply from the trained baseline and the Z-score crosses the threshold. It is typically a false positive caused by a legitimate operational event, not a pipeline failure. You handle it by acknowledging the incident, optionally excluding the backfill window, or letting the baseline re-absorb it.

Q: What is the difference between an Elementary anomaly test and a classic dbt schema test like `not_null`?
A: A classic dbt test (`not_null`, `unique`, `accepted_values`) is a deterministic rule — it passes or fails against a fixed assertion you wrote. An Elementary anomaly test is statistical and baseline-relative — it has no fixed threshold you author; it learns "normal" from history and flags deviations. Rules catch known, expected violations; anomaly tests catch unexpected shifts you did not think to assert.

Q: When would a rule-based test catch something an anomaly test never would, and vice versa?
A: A rule catches a hard, known constraint regardless of history — for example "currency must be one of ISO 4217 codes" — even on day one with no baseline. An anomaly test catches the unforeseen — a 30% volume drop or a creeping null-rate rise — that no static rule was written for. The robust design uses both: rules for the known-bad, anomaly detection for the unknown-unknown.

Q: Why does Elementary store run and test results in warehouse tables rather than only in dashboards?
A: Persisting results in tables gives a queryable, version-controlled history that powers baselines for anomaly detection, the observability report, and ad-hoc SQL analysis. It also means the data survives beyond a single run and can feed alerting and lineage. A purely in-memory dashboard could not build the historical baselines the Z-score method depends on.

Q: How does Elementary fit into a dbt CI/CD pipeline?
A: Because its checks are dbt tests, they run with `dbt build`/`dbt test` in the same CI step as your other tests, and the on-run-end hook captures results automatically. In production runs (for example via Airflow), you schedule `edr monitor` afterward to alert and optionally `edr report` to refresh the dashboard. This keeps observability in the same deployment path as the models themselves.

Q: Why run `edr monitor` as a separate scheduled step rather than inside `dbt test`?
A: `dbt test` produces and stores results, but it does not itself route alerts to Slack or Teams — `edr monitor` is the component that reads the latest results and pushes notifications. Separating them means dbt focuses on execution while `edr` focuses on alerting and reporting, and you can run `edr monitor` on its own cadence. Typically it runs immediately after the dbt job in the same orchestrator DAG.

Q: What is "unified alerting" in this context?
A: Unified alerting means routing data incidents — freshness, volume, schema, distribution failures — through the same alerting and on-call path as your infrastructure alerts, rather than into a separate, ignored channel. Concretely, Elementary's `edr monitor` (or a webhook) feeds the same Slack/PagerDuty/Grafana flow Phase 5 set up. The point is one pane of glass and one escalation policy for all incident types.

Q: Why is unified alerting preferable to a separate data-alerts channel?
A: A separate channel tends to become a graveyard nobody watches, so data incidents go unnoticed — defeating the purpose of detection. Routing through the existing, paged infra path means data incidents inherit the same urgency, ownership, and escalation as a server outage. It also reduces tool sprawl and gives responders a single place to triage.

Q: Which alerting integrations does Elementary support out of the box?
A: Elementary can send alerts to Slack, Microsoft Teams, and generic webhooks, and integrates with incident tools such as PagerDuty, Opsgenie, Jira, and Linear. The webhook option lets you fan into almost any system, including a Grafana or alertmanager pipeline. This breadth is what makes unified alerting practical regardless of your existing stack.

Q: How would you route an Elementary data incident into the same Grafana alerting flow used for infra?
A: You point Elementary's alerting at a webhook (or push a metric/state that Grafana's alerting can evaluate), so a data anomaly raises an alert through the same Grafana notification policies and contact points as infra alerts. The Do step in this lesson does exactly this — firing a synthetic stale/shrunken load through Grafana alerting with a runbook link attached. The result is one alerting path for both planes.

Q: Why attach a runbook link directly to a data alert?
A: Because the moment an alert fires is when the responder most needs to know what to check and how to resolve it, and a link removes the scramble to find the runbook. It turns a bare "volume anomaly on `prices`" into an actionable page with triage steps. This is a core part of making detection lead to fast resolution rather than confusion.

Q: What is a data-incident triage runbook?
A: It is a documented procedure that states, for a given class of data incident, who is paged, what they should check first, and how the incident is resolved or escalated. It removes guesswork at 3 a.m. by encoding the diagnostic path — for example "volume anomaly on `prices`: check the upstream loader, confirm source file size, compare to prior run." A good runbook is short, specific, and linked from the alert.

Q: What three questions must a data-incident runbook answer?
A: Who is paged (the owner/on-call for that detector), what they check (the concrete first diagnostic steps and lineage to inspect), and how it is resolved (remediation, backfill, or escalation criteria). If any of the three is missing, the runbook stalls during a real incident. Clear ownership in particular prevents the "everyone assumes someone else" failure.

Q: Why does every detector need an assigned owner?
A: Without an owner, an alert fires into a void and no one feels responsible for triaging it, so detection produces noise instead of resolution. Assigning an owner (a team or on-call rotation) per detector ensures someone is accountable for tuning thresholds, responding to incidents, and keeping the runbook current. The lesson's Done-when explicitly requires each silent-failure class to have a detector and an owner.

Q: In the lesson's "Do," what does firing a synthetic incident accomplish?
A: Forcing a stale or shrunken load deliberately verifies the whole chain end-to-end — detector fires, alert routes through Grafana, runbook link is attached, owner is notified — before a real incident tests it. It is the data equivalent of a fire drill, proving the pipeline of detection-to-resolution actually works. Untested alerting paths routinely fail silently when first needed.

Q: How would you simulate a "shrunken load" to test a volume detector?
A: Load a deliberately partial batch into the monitored table — for example insert only 10% of a normal day's rows, or run the loader against a truncated source file — so the row count drops well below baseline. The `volume_anomalies` test should then flag it on the next `dbt test`, and `edr monitor` should route the alert. You confirm the alert reaches the channel with the runbook link.

Q: How would you simulate a "stale load" to test a freshness detector?
A: Skip or delay the upstream refresh so the table's newest timestamp stops advancing past its SLA, or directly hold the max load time back. The `freshness_anomalies` test should detect that freshness has fallen behind the historical cadence. Verifying it fires and alerts confirms the freshness detector and its owner assignment are real, not theoretical.

Q: A freshness anomaly alert fired but the source table actually updated on time. First thing to check?
A: First check the timestamp column the test is keyed on — a freshness test reads a specific update/load column, and if that column is itself stale or wrong (for example the loader did not set it), the test reports staleness even though rows arrived. Confirm the loader populates the freshness column correctly and that the timezone matches. Misconfigured freshness columns are the most common cause of false freshness alerts.

Q: A `volume_anomalies` test on a daily table never fires despite obvious drops. First thing to check?
A: Check that the table has accumulated enough training history and that the `time_bucket` and selection actually cover the dropped period — a too-short baseline or a mis-aligned bucket means the test silently skips or has no reference. Also confirm the test is enabled in the `.yml` and is actually being run by `dbt test`. A detector that quietly skips is as dangerous as no detector.

Q: Elementary's `edr report` is empty even though dbt ran. First thing to check?
A: Confirm the `on-run-end` hook is present in `dbt_project.yml` and actually executed — if the hook is missing or errored, no run/test results were loaded into the `elementary` schema, so the report has nothing to render. Also verify `edr` is connecting to the same profile/schema where the package wrote its tables. An empty report almost always traces back to the artifact-collection hook or a schema/profile mismatch.

Q: `edr monitor` runs without error but no Slack alert arrives. First things to check?
A: Verify the Slack integration is configured (webhook URL or token and channel) and that there are actually new, un-sent failures for it to alert on — `edr monitor` only notifies on new incidents, so a clean run sends nothing. Check the `edr` logs for delivery errors and confirm the channel/webhook is valid. Often the "missing alert" is simply that nothing failed, or the alert was already sent on a prior run.

Q: Why might a schema-change test be the most business-critical detector in a fund feed?
A: Because a silent schema change upstream — a renamed or retyped identifier column — can cause downstream joins to drop rows or return nulls, corrupting positions and NAV without any error. The blast radius is large and the failure is invisible until reconciliation. Catching the change at ingestion, before it propagates, prevents a wrong number reaching a client.

Q: Map each silent-failure class to its Elementary detector: stale, shrunken, schema-drifted.
A: Stale maps to `freshness_anomalies`, shrunken maps to `volume_anomalies`, and schema-drifted maps to `schema_changes`/`schema_changes_from_baseline`. The lesson's Done-when requires showing that each of these classes has both a detector and an assigned owner. This mapping is the backbone of a data-plane heartbeat.

Q: How does lineage help during a data incident?
A: Lineage lets you trace from a failing table upstream to the root cause and downstream to everything affected, turning a single alert into a scoped impact assessment. Instead of guessing, you see that "stale `prices`" came from "stalled connector Y" and impacts "NAV model, EMT export, client report Z." This speeds both diagnosis and stakeholder communication.

Q: What is OpenLineage and how does it complement Elementary?
A: OpenLineage is an open standard and set of integrations for emitting lineage events (which job read and wrote which datasets) from orchestrators and engines like Airflow and Spark. It complements Elementary's dbt-graph lineage by capturing lineage across tools beyond dbt, giving cross-system context during triage. Together they let you follow an incident across the whole estate, not just within the dbt project.

Q: Why can't Elementary's dbt-graph lineage alone cover a full enterprise estate?
A: Because the dbt graph only knows about transformations expressed in dbt — it does not see ingestion tools, external Spark jobs, or BI consumers. A connector stall or a Spark failure upstream of dbt is invisible to the dbt graph. OpenLineage events fill that gap by emitting lineage from those external systems, so the picture spans the whole pipeline.

Q: What does a typical OpenLineage event describe?
A: An OpenLineage event describes a run of a job along with the input datasets it read and the output datasets it wrote, plus facets carrying schema, statistics, and metadata. Emitted at job start and completion, these events let a backend stitch together a cross-tool lineage graph keyed on dataset identity. That dataset-centric model is what lets lineage span Airflow, Spark, and dbt rather than being trapped in one tool.

Q: What is column-level lineage and why is it valuable during triage?
A: Column-level lineage maps how individual columns flow and transform from source to consumer, not just table-to-table, so you can see exactly which downstream fields a changed or broken column feeds. During triage it pinpoints blast radius precisely — for example showing that a corrupted `fx_rate` column reaches only the NAV and EMT cost fields. It is an Elementary Cloud feature, whereas the OSS package gives table/model-level lineage via the dbt graph.

Q: What is the relationship between data observability and data-quality rules?
A: They are layered, complementary disciplines: DQ rules assert known constraints deterministically, while observability watches the broader health signals (freshness, volume, distribution, lineage) statistically. Rules catch the violations you anticipated; observability catches the silent, unanticipated failures. A mature platform runs both, and Elementary conveniently hosts both inside dbt.

Q: Why is freshness/volume/schema monitoring "distinct from infra observability and from DQ rules"?
A: Infra observability watches compute health, DQ rules assert business constraints on values, and data observability watches operational health of the data itself — is it recent, complete, and structurally stable. A pipeline can be infra-green and rule-passing yet still produce stale or shrunken data, which only the observability layer catches. The lesson stresses this distinction because conflating the three leaves the silent-failure gap open.

Q: In a Luxembourg fund-administration platform, why is freshness on the prices source especially sensitive?
A: Because the daily NAV is struck from current market prices, and a stale prices table yields a NAV computed from yesterday's data — a regulator-relevant and client-visible error. UCITS funds publish NAVs that investors transact on, so a stale-price NAV can cause mispriced subscriptions and redemptions. A freshness detector on the prices source is therefore a frontline control.

Q: How would a volume anomaly on a transfer-agency (TA) feed manifest as a business problem?
A: A shrunken TA feed means some subscription/redemption transactions did not load, so positions and cash are understated and the NAV per share or the order book is wrong. The job may succeed while the file was truncated or partially delivered. A `volume_anomalies` detector on the TA load catches the missing transactions before they corrupt downstream holdings.

Q: How could a schema-drift incident corrupt an EMT or EPT file?
A: The EMT/EPT (European MiFID/PRIIPs Template) export is built from upstream models with specific fields; if an upstream column is renamed or retyped and the export silently maps nulls or wrong values, the regulatory file goes out malformed. Distributors consuming the EMT then receive bad cost or risk data. A `schema_changes` detector on the source models guards the export's inputs.

Q: Why is an ISIN column a prime candidate for distribution monitoring?
A: Because ISINs are the join key linking instruments across feeds, and a rise in null or malformed ISINs silently breaks joins and double-counts or drops positions. Volume and freshness can look normal while the ISIN null-rate climbs. A `column_anomalies` test on ISIN null rate (alongside a format rule) catches this content rot early.

Q: How does data observability support DORA's operational-resilience expectations?
A: DORA (the EU Digital Operational Resilience Act) expects financial entities to detect, manage, and report ICT-related incidents promptly, and data observability provides the detection-and-evidence layer for data incidents specifically. Freshness/volume/schema detectors plus unified alerting and runbooks demonstrate that silent data failures are monitored and have defined response paths. The persisted run/test history also provides an audit trail of incidents and resolutions.

Q: Why does a regulated fund administrator need an audit trail of data incidents?
A: Because regulators and clients require evidence that data issues were detected, triaged, and resolved within control timelines, not merely that nothing visibly broke. Elementary's warehouse tables of run results, test results, and anomalies provide that queryable, durable record. It turns "we think it was fine" into demonstrable monitoring with timestamps and owners.

Q: What does it mean to monitor freshness on a SWIFT-sourced cash feed?
A: It means checking that the cash-movement messages (for example MT940/MT942 balances) keep arriving on their expected cadence, so the latest record stays within the freshness SLA. A SWIFT feed that stalls leaves cash positions stale, which distorts NAV and reconciliation. A `freshness_anomalies` detector on that table catches a stalled or delayed SWIFT delivery.

Q: Why might LEI reference data warrant volume and distribution monitoring?
A: Because counterparty and issuer identification depend on the LEI, and a feed that drops a chunk of LEIs (volume) or where LEIs go increasingly null (distribution) silently breaks counterparty linkage and aggregation. The structure can look unchanged while coverage degrades. Monitoring both row count and null/coverage metrics on the LEI feed protects downstream counterparty reporting.

Q: A UCITS NAV is published, then found to be wrong due to a stale FX rate the next day. How would observability have prevented it?
A: A freshness detector on the FX-rates source would have flagged that the rates table had not refreshed before the NAV run, firing an alert with a runbook pointing to the FX loader. Caught pre-publication, the NAV run could have been held. This is the canonical "stale input → wrong NAV" failure the lesson targets, and freshness is the specific control.

Q: Why is "assign an owner to each detector" a Done-when criterion rather than a nice-to-have?
A: Because detection without ownership produces alerts no one acts on, which is operationally indistinguishable from having no detection at all. The criterion forces accountability: each silent-failure class has both a detector and a named responder. It is what makes the heartbeat actually trigger a response when it stops.

Q: What is the risk of over-tuning anomaly thresholds too sensitively?
A: Too-sensitive thresholds produce frequent false positives, and alert fatigue sets in — responders start ignoring the channel, so a real incident gets missed. The fix is to tune per-table thresholds, suppress known operational events like backfills, and route only meaningful anomalies. A noisy detector erodes trust as surely as a missing one.

Q: What is the risk of setting anomaly thresholds too loosely?
A: Too-loose thresholds (a high `anomaly_score_threshold`) miss real anomalies — a genuine 30% volume drop slips under the bar and no alert fires. You gain quiet at the cost of coverage, reintroducing the silent failures observability is meant to catch. Tuning is a balance, validated by firing synthetic incidents to confirm the detector still trips.

Q: How do you decide volume thresholds for a table with strong weekly seasonality?
A: You ensure the training period spans enough weeks for the baseline to learn the weekly pattern, or use a time bucket and detection logic that compares like-to-like (for example same weekday) rather than raw day-over-day. Otherwise a normal weekend low triggers a false volume anomaly every week. Seasonality must be in the baseline, not fought against with a static rule.

Q: Why might you exclude weekends or holidays from a freshness check on a market-data table?
A: Because markets and the vendor feed do not publish on non-trading days, so the "newest record" legitimately ages over a weekend without being a real freshness failure. If you do not account for the trading calendar, the detector floods you with false staleness alerts every Monday. You align the freshness SLA with the market calendar, not the wall clock.

Q: What is the difference between a freshness SLA and a freshness anomaly?
A: A freshness SLA is a fixed rule — "data must be no older than N hours" — that you assert explicitly, while a freshness anomaly is a statistical deviation from the table's historical update pattern. The SLA catches a hard miss against a contract; the anomaly catches an unusual delay even if it is technically within SLA. Many setups use both, an SLA rule plus `freshness_anomalies`.

Q: How does Elementary's `dbt source freshness` relate to its `freshness_anomalies` test?
A: dbt's built-in `source freshness` is a rule-based check against `warn_after`/`error_after` thresholds you configure on a source, while Elementary's `freshness_anomalies` is statistical and works on models and sources without a hand-set threshold. They are complementary: the SLA rule for the hard contract, the anomaly test for unexpected drift. You can run both in the same project.

Q: Why is it important that observability tests live in version control with the models?
A: Because keeping detectors as code in the same repo means they are reviewed, diffed, and deployed alongside the models they watch, so coverage does not silently rot when models change. A new gold model can ship with its detectors in the same pull request. Tool-based detectors defined outside source control tend to drift out of sync with the pipeline.

Q: In the lesson's "Do," why enable volume and freshness anomaly tests specifically on gold models?
A: Gold models are the consumption layer — what dashboards, exports, and clients actually read — so a silent failure there is the closest to a wrong number reaching a user. Monitoring gold gives the highest-leverage early warning. You typically also monitor key sources, but gold is the priority because it is the last line before the consumer.

Q: Should you monitor only gold models, or also sources and silver?
A: You should monitor critical sources and key intermediate models too, because catching a stale source or a shrunken silver load earlier pinpoints the root cause and prevents propagation. Monitoring only gold tells you something is wrong but not where. A layered approach — sources for root cause, gold for consumer impact — gives both early detection and fast localization.

Q: What is "build vs buy" in the data-observability context?
A: It is the decision between assembling a FOSS stack (Elementary plus OpenLineage, wired into your own alerting) versus purchasing a platform like Monte Carlo, Bigeye, or Databand that provides ML anomaly detection, lineage, and alerting out of the box. Build favors control, cost, and dbt-nativeness; buy favors broad automated coverage and less maintenance. Estate size and team capacity drive the choice.

Q: How does estate size influence the build-vs-buy decision for observability?
A: At small-to-mid estates, a dbt-native FOSS stack (Elementary) is cheap, controllable, and sufficient; at large estates with hundreds of pipelines across many tools, the maintenance and breadth needs tilt toward a bought platform with automated, ML-driven coverage. The crossover is where the labour of hand-maintaining detectors and lineage exceeds a license cost. The architect justifies the call in one line tied to estate size.

Q: What do commercial tools like Monte Carlo add over a FOSS Elementary stack?
A: They typically provide broader automatic coverage across many systems (not just dbt), ML-based anomaly detection with less manual configuration, automatic table-level monitoring, richer cross-tool lineage, and built-in incident management. The trade-off is cost and less transparency/control. Elementary closes much of the gap for dbt-centric estates while keeping everything in your warehouse and repo.

Q: What is a hybrid build-and-buy observability posture?
A: A hybrid uses FOSS like Elementary for dbt-native, in-warehouse detectors and version-controlled tests, while buying a platform for broad estate-wide automatic coverage, cross-tool lineage, or incident management. You get dbt-native depth plus turnkey breadth without committing fully to either. This is often the pragmatic answer for a growing regulated estate.

Q: Why is dbt the natural host for observability in a modern stack?
A: Because dbt already owns the transformation graph, the model definitions, and the test framework, so adding observability there means detectors share the same lineage, selection, scheduling, and CI as the models. There is no separate system to keep in sync. Elementary leverages exactly this — its checks are dbt tests and its lineage is the dbt graph.

Q: What metadata does the on-run-end hook collect, and why is each useful?
A: It collects model run results (timings, status, rows affected) and test results (pass/fail, the failing rows context) from dbt's artifacts. Run results power volume/freshness baselines and run-history views; test results power the anomaly and quality reporting and alerting. Together they are the raw material the report and `edr monitor` consume.

Q: If you delete the `elementary` schema, what happens to anomaly detection?
A: Anomaly detection loses its historical baselines and run/test history, so tests that depend on accumulated metrics will have no reference and effectively reset, and the report will be empty until history rebuilds. You would need to re-run the pipeline to repopulate the tables before detection is meaningful again. It is the persistent memory of the system, so it should be backed up like any production data.

Q: Why schedule `edr report` to upload its HTML to object storage?
A: Uploading the single HTML file to S3/GCS/Azure Blob (or a web server) makes the latest observability dashboard available to the whole team at a stable URL without anyone running the CLI locally. It keeps the report current after each production run. This turns a local artifact into a shared operational view.

Q: How does unified alerting reduce mean-time-to-resolution (MTTR) for data incidents?
A: By delivering data alerts into the same paged on-call path as infra, with a runbook link and lineage context attached, responders start triage immediately instead of first discovering the issue and hunting for context. The detection-to-action gap shrinks. Lower MTTR is the practical payoff of pillars plus unified alerting plus runbooks working together.

Q: What goes wrong if detectors exist but alerts route to an unwatched channel?
A: Detection happens but no one sees it, so incidents persist exactly as if there were no monitoring — the classic "alerts into a graveyard" failure. The fix is unified alerting into the paged, watched path the team already uses. This is precisely why the lesson pairs detectors with routing through Grafana and an owner.

Q: How would you prove end-to-end that your data-observability setup works?
A: Fire a synthetic incident — force a stale or shrunken load — and confirm the detector trips on `dbt test`, `edr monitor` (or the webhook) routes the alert through Grafana, the runbook link is attached, and the assigned owner is notified. This is the lesson's Done-when: a synthetic incident driven end-to-end into Grafana with a runbook link. An untested chain cannot be trusted.

Q: Summarize how the four Learn topics combine into a working data-observability practice.
A: The five pillars (freshness, volume, schema, distribution, lineage) define what to watch; Elementary as dbt-native observability provides the anomaly tests and report to watch it; unified alerting routes the resulting incidents through the same paged path as infra; and the triage runbook plus owners ensure each incident is diagnosed and resolved. Together they convert silent failures — like a stale-price NAV — into owned, fast-resolved, auditable incidents. That end-to-end loop, proven with a synthetic incident, is the deliverable.
