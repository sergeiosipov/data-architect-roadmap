# Phase 8 · 7.1.1 + 7.2.1 Semantic & metrics layer (Cube, MetricFlow) — 100 self-test questions

Q: What is a semantic layer, in one sentence?
A: A semantic layer is a centrally-defined, query-time abstraction that maps raw warehouse tables to business concepts — dimensions, measures, joins, and access rules — so every consumer computes a metric like AUM the same way. It sits between the physical gold marts and the BI tools, APIs, and notebooks that read them. Its value is that the definition lives in one governed place instead of being re-implemented in each tool's local SQL.

Q: Why is "one definition of AUM" described as a governance promise only a semantic layer can keep?
A: Because AUM (assets under management) appears in dashboards, board packs, REST consumers, and notebooks, and without a shared definition each one re-derives it in its own SQL and the numbers drift. A semantic layer makes the definition executable in a single model file, so all consumers inherit the same logic and any change propagates everywhere. The architect then spends review meetings designing rather than arbitrating whose AUM figure is correct.

Q: Define a dimension in semantic-layer terms.
A: A dimension is an attribute you slice or group measures by — for fund data that is typically fund name, share class, ISIN, management company, currency, or NAV date. Dimensions are defined once on the entity that owns them and reused by every measure, so a filter on `management_company` means the same thing in every query. They are the "by what" half of a metrics query, where measures are the "how much" half.

Q: Define a measure in semantic-layer terms.
A: A measure is a quantitative aggregation — a sum, count, average, or ratio — computed over the underlying rows, such as `SUM(net_asset_value)` rolled up to AUM. The measure carries its aggregation logic and any filters as part of its definition, so consumers ask for the measure by name rather than writing the aggregation themselves. Centralising the aggregation is exactly what stops two dashboards from summing NAV differently.

Q: What does it mean that joins are "defined once at the semantic level"?
A: It means the relationships between entities — for example funds to share classes to daily NAV — are declared once in the model, so the engine knows how to traverse them and consumers never hand-write join keys. A query can then combine a measure from one table with a dimension from another and the layer assembles the correct join automatically. This removes the most common source of silent miscounts: fan-out and wrong-grain joins written ad hoc in each report.

Q: What is access control "at the semantic level" and why put it there?
A: It is row- or column-level security defined inside the semantic model so that, for example, a user only sees rows for their own management company regardless of which client tool they use. Putting it at the semantic level means the rule is enforced once for the REST API, the SQL API, and every BI consumer, instead of being re-implemented per dashboard. This is the only way to guarantee a transfer-agency analyst cannot see another fund group's holdings no matter how they connect.

Q: What is "headless BI" and how does it relate to the semantic layer?
A: Headless BI is the architectural pattern where the metric definitions and modelling live in a standalone layer with no built-in visualisation, exposing data through APIs that any front end can consume. The semantic layer is the "head-less" engine; Superset, a notebook, or a custom app are the interchangeable heads. It decouples "what a metric means" from "how it is drawn," which is why one AUM definition can serve many tools.

Q: What is Cube, in this lesson's context?
A: Cube is an open-source headless semantic layer that you define in data model files, connects to your warehouse, and exposes metrics over REST, SQL, GraphQL, and MDX APIs. In this lesson it is the FOSS hands-on tool for defining NAV, AUM, and net-flows once over the gold marts. It is the open analogue of proprietary stacks like Looker/LookML or AtScale.

Q: What lives in a Cube data model file?
A: A Cube data model file (a cube or view) declares the SQL source table, its dimensions, its measures, its joins to other cubes, and optionally pre-aggregations and access policies. These files are version-controlled, so a change to the AUM measure is a reviewable diff. Cubes describe the underlying entities; views compose selected dimensions and measures into a clean, consumer-facing surface.

Q: What is the difference between a Cube "cube" and a Cube "view"?
A: A cube maps to an underlying dataset and holds its dimensions, measures, and joins; a view is a curated, denormalised facade that exposes a chosen subset of members from one or more cubes to consumers. Views give you a stable public contract — you can refactor cubes underneath without breaking dashboards. The recommended pattern is to query views, not raw cubes, from BI tools.

Q: What is a pre-aggregation in Cube?
A: A pre-aggregation is aggregated data that Cube builds and maintains ahead of query time so that queries hit a small rollup instead of scanning the full fact table. It is defined inside the data model and is the second of Cube's two caching levels. For a NAV mart with millions of daily rows, a pre-aggregation on AUM by fund and date turns a heavy scan into a sub-second lookup.

Q: What are Cube's two levels of caching?
A: The first level is an in-memory result cache that stores query result sets keyed on the generated SQL; the second level is pre-aggregations, which are materialised rollups stored in Cube Store or the source database. The in-memory cache absorbs bursts of identical concurrent queries, while pre-aggregations cut the cost of the underlying scan itself. They work together: pre-aggregations make the heavy query cheap, the in-memory cache makes the repeated query instant.

Q: What is Cube Store?
A: Cube Store is Cube's purpose-built distributed storage and query engine for pre-aggregations, separate from your source warehouse. Storing rollups in Cube Store offloads analytical serving from the operational database and gives faster, more concurrent reads than keeping pre-aggregations in the source system. It is what lets Cube serve dashboards at scale without hammering Postgres or Snowflake.

Q: What does the Cube REST API let a consumer do?
A: The REST API accepts a JSON query object naming the measures, dimensions, filters, and time dimensions you want, and returns the computed result with metadata. It is the general-purpose, language-agnostic way to fetch a metric — any app that can make an HTTP call can request AUM and get the canonical number. In the lesson's demo it is the "raw REST call" client proving the figure matches the others.

Q: What does the Cube SQL API expose, and to whom?
A: The Cube SQL API speaks the PostgreSQL wire protocol, so any Postgres-compatible client — a BI tool, `psql`, or a notebook using a Postgres driver — can issue SQL `SELECT` statements against the semantic layer as if it were a database. This lets tools that only know SQL still consume governed metrics. In the demo it is how the notebook queries AUM via the SQL API and gets the identical figure.

Q: Why is it significant that Cube's SQL API uses the Postgres wire protocol specifically?
A: Because the Postgres protocol is the most widely supported database interface, almost every BI tool, ORM, and notebook library can connect with no custom driver. Cube can therefore present the semantic layer as "just a Postgres endpoint," dramatically lowering the integration cost for SQL-native consumers. The trade-off is that you query Cube views/measures through SQL semantics rather than the richer REST query object.

Q: Name the four query APIs Cube exposes.
A: Cube exposes a REST API, a SQL API (Postgres-compatible), a GraphQL API, and an MDX API. REST and SQL are the two the lesson exercises; GraphQL suits typed front-end clients and MDX lets Excel and some BI tools connect via the analytical-expression protocol. All four resolve to the same underlying measures, so the AUM definition is identical regardless of which API is used.

Q: What is MetricFlow?
A: MetricFlow is dbt's semantic layer engine: you define semantic models and metrics in YAML inside your dbt project, and MetricFlow compiles requested metrics into SQL against your warehouse at query time. The metric definitions live as nodes in the dbt DAG alongside the models they sit on. It is the dbt-native alternative to a standalone tool like Cube.

Q: How does a MetricFlow metric relate to the dbt DAG?
A: A MetricFlow semantic model corresponds to a dbt model and appears as a node on the DAG, and the semantic graph is effectively a subset of that DAG. This means metric definitions are version-controlled, tested, and lineage-tracked with the same tooling as the rest of the dbt project. Changing an upstream model's grain is therefore visible to the metric's lineage rather than hidden in a BI tool.

Q: What are the three core pieces of metadata in a MetricFlow semantic model?
A: Entities, dimensions, and measures. Entities are the join keys (the edges that let MetricFlow traverse between semantic models), dimensions are the attributes you group and filter by, and measures are the aggregatable numeric columns. Metrics are then defined on top of measures. This mirrors Cube's cube anatomy of joins, dimensions, and measures with different names.

Q: List the MetricFlow metric types.
A: MetricFlow supports simple, ratio, cumulative, derived, and conversion metrics. Simple wraps a single measure; ratio divides one measure by another; cumulative accumulates a measure over a window; derived computes from other metrics with an expression; and conversion measures the rate at which a base event leads to a target event. Choosing the right type is how you express, say, net-flow ratios or rolling AUM without bespoke SQL.

Q: What is a MetricFlow simple metric?
A: A simple metric is the thinnest wrapper around one measure — for example AUM defined as the `sum` of a `net_asset_value` measure. It is the starting point for most metrics and the building block other types compose from. Even a simple metric earns its keep because the aggregation and any constraints are now defined once and inherited everywhere.

Q: What is a MetricFlow ratio metric, with a fund example?
A: A ratio metric divides a numerator measure by a denominator measure, each of which can carry its own filters. A fund example is expense ratio as total fees over average AUM, or the share of net flows attributable to a single management company. Defining it as a ratio metric keeps the numerator and denominator definitions consistent rather than letting each dashboard pick its own.

Q: What is a MetricFlow cumulative metric?
A: A cumulative metric aggregates a measure over a sliding or to-date window rather than a single period — for instance month-to-date net flows or a trailing-30-day average AUM. MetricFlow handles the window logic so consumers do not write window functions by hand. This is exactly the kind of definition that drifts when each analyst codes the rolling window themselves.

Q: What is a MetricFlow derived metric?
A: A derived metric is computed by an expression over other metrics — for example net-flows defined as subscriptions minus redemptions, where both are themselves metrics. It lets you compose business logic on top of already-governed metrics, so a change to "subscriptions" flows into "net-flows" automatically. Derived metrics keep the dependency explicit in the DAG instead of buried in BI SQL.

Q: What is a MetricFlow conversion metric?
A: A conversion metric measures how often a base event is followed by a related conversion event within a window — classically signups to purchases, but in fund data, say, the rate at which subscription orders settle within the standard cycle. It joins the two event streams on an entity and a time bound. It is the one metric type least likely to appear in NAV reporting but useful in transfer-agency operational metrics.

Q: State the consistency argument for a metrics layer over BI-tool-local definitions.
A: When metrics are defined inside each BI tool, every tool — and often every dashboard within a tool — holds its own copy of the logic, so the same metric forks and the copies silently diverge as filters and edge cases are added unevenly. A metrics layer defines the logic once upstream and every tool consumes it, so there is exactly one source of truth. The argument is structural: local definitions drift because nothing forces them to stay in sync.

Q: Why do BI-tool-local metric definitions "always drift"?
A: Because each is edited independently by different people at different times, and there is no mechanism that propagates a fix in one to all the others. A timezone tweak, an excluded share class, or a changed filter applied in one dashboard never reaches the forty other places the metric was re-implemented. Drift is the default outcome of duplication without a single governed definition.

Q: Distinguish a semantic layer from a business glossary.
A: A glossary documents what a metric means in prose; a semantic layer executes that meaning in code against the warehouse. The glossary is the human-readable definition, the semantic layer is the machine-enforced one. The two must be linked so the documented definition and the executed definition describe the same thing.

Q: What is the risk if the glossary and the semantic layer are not linked?
A: They rot independently — the glossary text says AUM excludes side-pockets while the executed measure still includes them, or vice versa, and nobody notices until an auditor compares them. Linking them in both directions makes a divergence visible and gives each a reason to be kept current. The lesson's exercise builds exactly this two-way link.

Q: How do you link a metric to its glossary term in both directions?
A: From the glossary entry you reference the semantic-layer object name (the Cube measure or MetricFlow metric), and from the metric's description you store the glossary URL. That way a reader of either artefact can jump to the other, and a reviewer can confirm the executed logic matches the documented intent. Bidirectional linking is what stops one side from silently going stale.

Q: When is a semantic layer premature?
A: When there is one consumer, one tool, and no contested metric — for example a single analyst running one Superset dashboard off a clean mart. With no second consumer to keep consistent and no metric being argued over, the abstraction adds maintenance cost without buying consistency. Cube's own introduction frames the layer as earning its place when definitions must be shared across tools or teams.

Q: State the rule for when a metric belongs in the semantic layer versus a local dashboard calculation.
A: A metric belongs in the semantic layer when it is consumed by more than one tool or audience, or when its definition is contested or governed (AUM, NAV, net-flows); it can stay a local dashboard calculation when it is single-use, single-tool, and not authoritative. The test is "would drift here cause a credibility or compliance problem?" — if yes, centralise it. Throwaway exploratory math does not need to be promoted to the layer.

Q: A board pack and a regulator report show different AUM for the same fund. How does a semantic layer prevent this?
A: By having both artefacts pull AUM from the same governed measure rather than each computing it in local SQL, so there is one calculation and one answer. If the definition changes — say to exclude a share class — both update from the single source. Without the layer, reconciling the two figures becomes a recurring manual investigation.

Q: What does it mean to define NAV "once over the gold marts" in Cube?
A: It means the NAV measure's SQL points at the curated gold-layer NAV mart (already cleaned and conformed) and is declared a single time in a Cube data model file. Consumers then request `nav` by name and inherit that exact logic. Building on gold rather than raw tables keeps the measure simple and the grain trustworthy.

Q: Why model the join topology explicitly when defining AUM in Cube?
A: Because AUM rolls daily NAV up across share classes within funds, and the join from NAV rows to fund and management-company dimensions must be declared so the engine aggregates at the right grain. An implicit or wrong join would fan out rows and inflate AUM. Declaring the topology once means every consumer's AUM uses the correct join.

Q: How would you express a management-company access rule at the semantic level in Cube?
A: You add a security context (query rewrite/row-level filter) to the cube or view so that the user's management-company claim is injected as a `WHERE` filter on every query. Then no matter which API or BI tool connects, a user only sees their own fund group's rows. This is the semantic-layer equivalent of database row-level security but enforced uniformly across all consumers.

Q: Why mirror the same three metrics in MetricFlow after defining them in Cube?
A: To learn where the two specification styles diverge — for example how each names entities versus joins, how each expresses a ratio, or how each handles time grains and filters. Building NAV, AUM, and net-flows in both tools surfaces the modelling trade-offs concretely. It also de-risks a future stack choice by proving the same business logic is expressible either way.

Q: Name a place where Cube and MetricFlow specifications can diverge.
A: They diverge in vocabulary and structure: Cube models joins and measures inside cubes/views, while MetricFlow uses entities, measures, and a separate metric layer with explicit metric types. Filter placement, time-dimension handling, and how derived/ratio logic is expressed also differ. The exercise is to document each divergence so a migration or comparison is not a surprise.

Q: In the three-client demo, what are the three clients and what must they prove?
A: A Superset chart, a raw REST call, and a notebook via the SQL API — three different consumption paths into the same Cube model. They must all return identical AUM, NAV, and net-flow numbers. Identical figures across three transports is the proof that the semantic layer, not each client, owns the definition.

Q: Why use Superset specifically as the BI client in the demo?
A: Superset is the open-source BI tool that consumes the Cube semantic layer over its APIs, standing in for any production BI front end. It lets you show a real dashboard pulling governed metrics rather than re-deriving them in chart-level SQL. It is the "head" in the headless-BI demonstration.

Q: Walk through what happens when you change the AUM definition in one place.
A: You edit the single AUM measure (in the Cube data model file or the MetricFlow metric YAML), commit the change to version control, and on the next query every client — Superset, REST, notebook — returns the new figure. Because all three resolve the metric through the layer, none of them needed editing. The version-control diff is the audit trail showing who changed AUM, when, and why.

Q: Why is changing AUM to "exclude side-pocket share classes" a good exercise case?
A: Because side-pockets are illiquid sleeves that funds sometimes carve out, and whether they count toward headline AUM is a genuine definitional choice with reporting consequences. Making that one edit and watching all three clients update demonstrates the layer's single-point-of-change promise on a realistic regulated-fund decision. It also shows the audit trail capturing a change auditors would care about.

Q: What audit trail does the single-definition change leave, and why does it matter?
A: The change is a commit in version control showing the diff, author, timestamp, and (via the message or linked ticket) the rationale. In a regulated fund-administration context that history is what lets you answer "why did AUM move on this date?" without forensic reconstruction. The semantic layer turns a metric change into a reviewable, attributable event.

Q: Explain what pre-aggregations cache.
A: They cache pre-computed aggregate rollups — measures grouped by a chosen set of dimensions and time grains — so a query for, say, daily AUM by fund reads a compact materialised table instead of re-scanning the full NAV fact. You define which measure/dimension/grain combinations to roll up based on your query patterns. They trade storage and refresh cost for dramatically faster, higher-concurrency reads.

Q: How does a pre-aggregation get invalidated and refreshed?
A: Its refresh key determines staleness; when the key's value changes, Cube rebuilds the rollup. The key can be time-based (refresh every N minutes) or SQL-based (e.g. `SELECT MAX(updated_at)` so it rebuilds only when new data lands). A Refresh Worker performs the rebuild in the background in production so queries are never blocked waiting for a rebuild.

Q: What is a refresh key in Cube?
A: A refresh key is the signal Cube evaluates to decide whether a cached result or pre-aggregation is still valid. A time-based key invalidates on a fixed interval; a SQL-based key runs a cheap query (like a max timestamp or a row count) and invalidates only when the result changes. Choosing a SQL-based key tied to data arrival avoids needless rebuilds while keeping freshness tight.

Q: State the freshness trade-off pre-aggregations buy.
A: Pre-aggregations give fast, concurrent reads at the cost of serving data that is only as fresh as the last rebuild — between refreshes, results can be stale. Tightening the refresh interval reduces staleness but increases load on the source and on Cube Store. The architect picks the point on that curve that meets the consumer's freshness requirement, for example "AUM fresh by 08:00 CET."

Q: Why should the pre-aggregation refresh worker run as a separate process in production?
A: So that cache invalidation and rebuilds happen in the background without delaying user queries; in development Cube can refresh inline, but in production a dedicated Refresh Worker keeps serving latency predictable. Without it, a user query could trigger a slow rebuild and time out. Separating refresh from serving is standard for a production semantic layer.

Q: Cube's default in-memory cache returns a stale number after the source updated. What is the first thing to check?
A: Check the refresh key — the in-memory result cache returns a cached result while the refresh key is unchanged, so if your key is time-based with a long interval (or not tied to data arrival) you will serve stale results until it ticks. Switch to a SQL-based refresh key that tracks the source's max update timestamp so the cache invalidates when new data actually lands. Also confirm the refresh worker is running.

Q: A pre-aggregation is rebuilding far too often and loading the warehouse. What is the likely cause?
A: A refresh key set too aggressively — either a very short time interval or a SQL key that changes on every minor write — forces constant rebuilds. The minimum recommended refresh interval is around one minute, and the SQL key should reflect meaningful data arrival, not noise. Loosen the interval or anchor the key to the partition/batch that actually matters for freshness.

Q: What is the in-memory cache keyed on, and what does that imply?
A: It is keyed on the generated SQL statement plus the refresh-key value, so two semantically identical requests that compile to the same SQL share a cache entry. This means a burst of concurrent users asking for the same AUM tile hits the database once and is served from memory thereafter. It also means a tiny change to the query (a different filter) is a different key and a separate cache entry.

Q: How does the semantic layer help a transfer agency enforce that one fund group cannot see another's data?
A: By defining the management-company isolation rule once at the semantic level, every consumer — operations dashboards, REST integrations, ad-hoc notebooks — has the same filter applied automatically from the user's security context. There is no client where an analyst can bypass it by writing their own SQL, because the SQL API itself enforces the rule. This single enforcement point is what makes the isolation defensible to an auditor.

Q: How would an ISIN be represented in the Cube model?
A: ISIN (the 12-character International Securities Identification Number that uniquely identifies a share class) is naturally a dimension on the share-class entity, used for filtering and grouping NAV and AUM. Modelling it as a dimension once means every consumer filters on the same canonical ISIN field. It is also a good join key between the NAV mart and reference data.

Q: Where would LEI fit in a fund semantic model?
A: The LEI (Legal Entity Identifier, the 20-character ISO 17442 code identifying a legal entity such as the fund or management company) is a dimension on the management-company or fund entity. It is useful both for grouping and as the key behind the management-company access rule. Defining it once avoids the common mess of LEIs formatted or sourced inconsistently across reports.

Q: Why is a semantic layer especially valuable for an EMT/EPT reporting flow?
A: EMT (European MiFID Template) and EPT (European PRIIPs Template) files must carry consistent fund figures — costs, AUM-derived values, NAV — and if each downstream report re-derives those numbers they can disagree with the templates and with each other. A semantic layer makes the figures fed into EMT/EPT generation come from the same governed measures as every dashboard. That consistency is exactly what regulators and distributors expect across the templates.

Q: How does a semantic layer support a "fresh by 08:00 CET" NAV serving commitment?
A: By tuning pre-aggregation refresh keys so the AUM/NAV rollups rebuild as soon as the morning NAV batch lands, the layer can serve the day's figures the moment they are valid. The refresh trade-off is set to favour freshness for that window. Combined with the audit trail, the layer turns the deadline from marketing copy into something measurable and enforceable.

Q: What is the difference between a Cube cube and a Cube view from a governance standpoint?
A: A cube is the internal model you can refactor freely; a view is the stable public contract consumers query, so governance focuses on view definitions as the surface that must not break. By querying views, downstream tools are insulated from cube refactors, and access policies can be applied at the view level. This separation lets the architect evolve the model without renegotiating every dashboard.

Q: In MetricFlow, why is it an advantage that metrics are tested with dbt?
A: Because the same `dbt test` machinery that validates the underlying models can assert properties about the metrics' inputs, so a broken grain or a null surge fails CI before it reaches consumers. Metrics inherit dbt's lineage, documentation, and version control rather than living outside the engineering workflow. This is the core appeal of a metrics layer embedded in the dbt DAG.

Q: What does "definition executed vs definition documented" capture?
A: It captures the core distinction between a semantic layer (the definition is code that runs against the warehouse) and a glossary (the definition is prose a human reads). Both are needed: the executed definition guarantees consistency, the documented one explains intent to non-engineers. Linking them keeps the running logic and the explained meaning in agreement.

Q: A Superset chart and a notebook disagree on AUM despite both using Cube. What is the first thing to check?
A: Confirm both actually resolve to the same Cube measure/view rather than one of them computing AUM in chart-level or notebook-level SQL that bypasses the layer. The whole point of the layer is defeated if a client re-derives the metric locally. Next, check that filters, the security context (management company), and any timezone handling are identical between the two requests.

Q: Why is timezone handling a classic source of metric drift, and how does the semantic layer help?
A: Because a NAV "as of date" or a flows cut-off depends on a timezone, and two tools applying different timezones to the same timestamp will bucket rows into different days and report different daily figures. Defining the time dimension and its timezone once in the semantic layer makes every consumer bucket identically. The lesson explicitly flags timezone handling as a place Cube and a Power BI model could drift.

Q: Compared to Looker/LookML, what is Cube's positioning?
A: Cube is the open-source headless semantic layer that plays the role LookML plays inside Looker, but without being tied to one BI front end — it exposes metrics over REST, SQL, GraphQL, and MDX to any consumer. LookML defines metrics for Looker; Cube defines them for everything. AtScale is another proprietary point of comparison in the same headless category.

Q: What is the relationship between the gold marts and the semantic layer?
A: The gold marts are the curated, conformed physical tables (Kimball-style facts and dimensions); the semantic layer sits on top and defines the business metrics over them. The layer does not replace the marts — it depends on their clean grain and conformed dimensions to keep measures simple. Good marts make the semantic model thin; messy marts push complexity into the layer.

Q: Could you build the semantic layer directly on raw or silver tables instead of gold? What is the trade-off?
A: You can point measures at any source, but building on raw/silver means the layer absorbs all the cleaning, deduplication, and conforming logic, making measures complex and fragile. Building on gold keeps the metric definitions thin and trustworthy because the grain and quality are already guaranteed. The architectural norm is gold-as-foundation, semantic-layer-as-definition.

Q: How does the semantic layer differ from a database view?
A: A database view is a single SQL definition in one database; a semantic layer is a modelling layer that knows dimensions, measures, joins, access rules, and metric types, and serves them over multiple APIs with caching. A view returns rows of a fixed shape; the semantic layer composes arbitrary measure-by-dimension queries on demand. The layer is closer to a query-generation engine than to a static view.

Q: What does it mean that the semantic graph is "a subset of the DAG" in MetricFlow?
A: It means the semantic models and metrics are themselves nodes within the larger dbt DAG, so they participate in dependency resolution and lineage like any model. The semantic graph adds relationships about how information can be combined for metric consumption, on top of the build-order dependencies the DAG already tracks. Practically, metric lineage is just dbt lineage.

Q: Why might a 50-person Azure-only fund admin not need a headless layer like Cube?
A: If they have effectively one BI tool (Power BI) and a small, agreed set of metrics, the platform's own semantic model can be authoritative and a separate headless layer is redundant overhead. The premature-semantic-layer rule applies: few consumers, one tool, little contention. Cube earns its place when tool diversity or contested metrics appear — a point sharpened in the companion 7.1.2 lesson.

Q: What is the cost of adopting a semantic layer too early?
A: You pay ongoing maintenance — model files, refresh tuning, deployment, and review overhead — for consistency you do not yet need because there is only one consumer to be consistent with. The abstraction also slows iteration when a single analyst could just edit one dashboard. The discipline is to introduce the layer when a second consumer or a contested metric makes drift a real risk.

Q: How does access control in the semantic layer differ from RLS in a BI tool?
A: Semantic-layer access control is enforced once for every consumer of the layer — REST, SQL, GraphQL, BI — whereas BI-tool RLS only protects users of that one tool and must be re-implemented elsewhere. Centralising it removes the gap where a notebook hitting the SQL API could bypass a dashboard's row filter. For management-company isolation in fund admin, the single enforcement point is the safer design.

Q: What query shape does the Cube REST API expect?
A: A JSON query object listing `measures`, `dimensions`, `filters`, `timeDimensions` (with a granularity), and ordering/limits; Cube compiles it to SQL, applies caching and security, and returns rows plus annotation metadata. Consumers describe what they want declaratively rather than writing the join and aggregation. This declarative shape is what guarantees they cannot accidentally redefine the measure.

Q: How would a notebook consume AUM via Cube's SQL API?
A: It connects with any Postgres driver to Cube's SQL API endpoint and runs a `SELECT` against a Cube view, treating measures and dimensions as columns; Cube generates the underlying warehouse SQL and returns the governed result. The notebook author writes ordinary SQL but inherits the central AUM definition. This is the "notebook via the SQL API" client in the three-client demo.

Q: What is the danger of letting a notebook re-implement AUM in its own SQL?
A: It re-creates exactly the drift the semantic layer exists to prevent — the notebook's AUM can quietly disagree with the board pack's, and there is no audit trail tying the two together. The fix is to make the notebook query the layer (REST or SQL API) so it inherits the canonical measure. Governance only holds if every path goes through the layer.

Q: Why are net-flows a good third metric to model alongside NAV and AUM?
A: Net-flows (subscriptions minus redemptions over a period) are naturally a derived or ratio-style metric, so modelling them exercises composition on top of base measures rather than a plain sum. They are also a contested, period-bounded figure prone to timezone and cut-off drift, making them a realistic governance target. Together NAV, AUM, and net-flows cover a stock measure, a rolled-up stock measure, and a flow measure.

Q: How does a pre-aggregation differ from the in-memory cache in what it stores?
A: The in-memory cache stores finished query result sets for specific queries; a pre-aggregation stores reusable aggregate rollups that many different queries can be answered from. A new but related query can be served from an existing pre-aggregation without a fresh warehouse scan, whereas the in-memory cache only helps if the exact same query repeats. Pre-aggregations are about reuse across queries; the in-memory cache is about repetition of one query.

Q: What is "pre-aggregation matching" in Cube?
A: It is Cube's logic that decides whether an incoming query can be answered from an existing pre-aggregation based on its measures, dimensions, time grain, and filters. If the query's grain is a rollup of the pre-aggregation's grain, it matches and is served fast; if it needs finer detail, it falls through to the source. Designing pre-aggregations to match your common query patterns is how you get the speed-up.

Q: Why might a pre-aggregation not be used even though it exists?
A: Because the query does not match it — it asks for a dimension or finer time grain the pre-aggregation does not contain, or a filter that the rollup cannot satisfy, so Cube falls back to the source database. The first check is to compare the query's measures, dimensions, and granularity against the pre-aggregation's definition. Adding the missing dimension or grain (or restructuring the query) restores the match.

Q: In MetricFlow, where do filters/constraints on a metric live?
A: They live in the metric or measure definition (and can be applied per-input for ratio and derived metrics) in the YAML, so the constraint is part of the governed definition rather than the consumer's query. For example AUM could constrain to active share classes once. Centralising the filter is what keeps two consumers from disagreeing on which rows count.

Q: How do you keep the Cube and MetricFlow definitions of AUM from diverging if you maintain both?
A: You write down the divergence points (vocabulary, filter placement, time/timezone handling) and a governance rule naming which layer is authoritative for which metric, then enforce it in review. In practice most shops pick one as canonical and treat the other as a comparison or migration target rather than maintaining two sources of truth. The lesson's exercise is to document, not to bless permanent duplication.

Q: What makes AUM a "contested metric" in fund administration?
A: Because reasonable definitions differ — whether to include side-pocket or gated share classes, how to treat committed-but-undrawn capital, which valuation timestamp and FX to use — so different stakeholders legitimately compute different AUM. That contention is precisely why it must be defined once and changed in one place. A contested metric is the textbook case for promoting a metric into the semantic layer.

Q: How does the semantic layer interact with SWIFT-driven settlement data?
A: SWIFT messages drive subscription/redemption settlement events that feed the net-flows fact, and the semantic layer defines net-flows over that conformed gold mart rather than over raw message logs. Consumers then get one definition of flows regardless of message volume or format. The layer's job is the metric definition; SWIFT ingestion and parsing happen upstream in the pipeline.

Q: What is the role of version control in the semantic-layer workflow?
A: Cube data model files and MetricFlow YAML are committed to git, so every change to a measure is a reviewable, attributable diff with history. This turns metric governance into the same pull-request discipline as code: review, approve, deploy, audit. It is what supplies the audit trail when AUM changes and what makes "change in one place" safe to do.

Q: Why is "change AUM in one place and see it everywhere" considered the headline proof of the lesson?
A: Because it demonstrates the layer's entire value proposition in one action: a single governed edit propagates to every consumer with an audit trail, which no per-tool definition can achieve. If the change had to be replicated in Superset, the REST consumer, and the notebook separately, drift would already be possible. The single-point-of-change behaviour is the consistency guarantee made visible.

Q: How does headless BI change the architect's review meetings?
A: Instead of arbitrating whose AUM SQL is correct, the architect designs and governs the one definition and reviews changes to it. Debates shift from "which number is right" to "what should the definition be," which is a design conversation rather than a reconciliation chore. Removing per-tool re-derivation is what frees the architect to architect.

Q: What is a pre-aggregation's "partition" and why partition them?
A: Partitioning splits a pre-aggregation by time (or another key) so that only the partitions whose data changed are rebuilt on refresh, instead of the whole rollup. For a daily NAV pre-aggregation, only today's partition rebuilds as new NAV lands, keeping refresh cheap. Partitioning is the main lever for keeping freshness high without expensive full rebuilds.

Q: When would you store a pre-aggregation in the source database instead of Cube Store?
A: When you want to leverage the warehouse's own compute and avoid moving data into Cube Store, or when the source already serves analytical loads well. The default and usually faster path is Cube Store, which offloads serving from the operational database. The choice is a trade-off between source-warehouse cost/coupling and Cube Store's optimised serving.

Q: How does the semantic layer relate to UCITS/AIFMD reporting consistency?
A: UCITS and AIFMD reporting draws on fund figures (AUM, NAV, exposures) that must be consistent across regulatory filings and internal reports; sourcing those figures from one governed semantic definition prevents a filing from disagreeing with a dashboard. The layer does not implement the regulation but it removes definitional drift in the numbers the regulation consumes. Consistency plus an audit trail is what supports the regulatory narrative.

Q: What does a "security context" do in Cube?
A: A security context carries the authenticated user's attributes (for example their management company) into the query so that data model rules can filter or restrict results per user. It is the mechanism behind semantic-level access control. Because it is evaluated in the layer, the same isolation applies whether the user arrives via REST, SQL, or a BI tool.

Q: Why prefer a SQL-based refresh key over a purely time-based one for a NAV pre-aggregation?
A: A SQL-based key like `SELECT MAX(nav_date)` or a batch-id check rebuilds only when new NAV data actually arrives, so you avoid both stale results (waiting for a timer) and wasteful rebuilds (refreshing when nothing changed). A purely time-based key either lags the data or hammers the source on a tight interval. Anchoring freshness to data arrival is the precise control a NAV deadline needs.

Q: What is the difference between a measure and a metric in MetricFlow?
A: A measure is the raw aggregatable input defined on a semantic model (for example `sum(net_asset_value)`); a metric is the consumer-facing definition built on measures, with a type (simple, ratio, cumulative, derived, conversion) and any constraints. Consumers query metrics, not measures. The split lets one measure feed several metrics — AUM, a ratio, a cumulative window — without redefining the underlying sum.

Q: A new dimension is needed on AUM (e.g. by domicile) but only one report needs it. Layer or local?
A: If only one report needs it and it is not a governed figure, slicing AUM by domicile can stay local — but because AUM itself is authoritative, the AUM measure stays in the layer and you add the domicile dimension there if it is reusable. The rule is per-element: the contested measure belongs in the layer, a one-off slice can be local. Promote the dimension to the layer once a second consumer wants the same cut.

Q: How does the semantic layer help when onboarding a new BI tool?
A: The new tool connects to the existing APIs (commonly the SQL API via Postgres protocol) and immediately inherits every governed metric, so no metric logic is re-implemented during onboarding. The integration cost drops to "point it at the endpoint," and consistency is guaranteed from day one. This is a direct payoff of headless BI's decoupling of definition from front end.

Q: What is the audit question a semantic layer lets you answer that local definitions cannot?
A: "Show me exactly how AUM was defined on this date, who changed it, and why" — answerable from the version-controlled model history, whereas with local definitions the logic is scattered across tools with no unified history. The layer turns metric definitions into auditable artefacts. For regulated fund administration, that traceability is often a compliance requirement, not a nicety.

Q: Why is it risky to expose raw cubes rather than views to BI consumers in Cube?
A: Because cubes are internal modelling objects you may refactor, and binding dashboards directly to them means a refactor can break consumers; views provide a stable contract that insulates them. Views also let you apply access policies and curate exactly what is public. Querying views, not cubes, is the recommended governance boundary.

Q: How do pre-aggregations improve concurrency, not just latency?
A: By serving many queries from a small materialised rollup (often in Cube Store) instead of each query running a heavy scan on the source, the source database is freed from concurrent load and Cube Store handles the fan-out. More users can be served the same metric simultaneously without contention on the warehouse. Concurrency is the second, often underappreciated, win of pre-aggregations.

Q: What is the minimum sensible refresh interval in Cube and why does it exist?
A: Around one minute is the recommended floor; below it the constant refresh-key checks and rebuilds put excessive load on the source for little freshness gain. The floor protects the database from being hammered by an over-eager refresh configuration. For genuinely sub-minute needs, Cube offers streaming pre-aggregations rather than tighter polling.

Q: What is the GraphQL API in Cube best suited for?
A: The GraphQL API lets typed front-end clients request exactly the measures and dimensions they need in a single typed query, which suits modern web apps and component-based dashboards. It resolves to the same governed measures as REST and SQL, so AUM is identical regardless of API. It is one of four consumption paths — REST, SQL, GraphQL, MDX — that all share the single definition.

Q: Why does the lesson pair a FOSS tool (Cube) with a dbt-native one (MetricFlow) rather than teaching just one?
A: Because the architect must recognise the semantic-layer pattern independent of any one product, and seeing the same NAV/AUM/net-flows logic in both a standalone engine and an embedded-in-dbt engine teaches the pattern, not the syntax. It also informs the build-versus-adopt decision: a shop already on dbt may prefer MetricFlow, while a multi-tool shop may want Cube's broad API surface. The divergences you document are the real learning.

Q: How does a streaming pre-aggregation differ from a normal refresh-key-driven one?
A: A streaming pre-aggregation keeps the rollup continuously updated from a stream rather than rebuilding on a refresh-key interval, so it serves near-real-time results without the staleness window of periodic refresh. You reach for it when freshness needs are sub-minute and a tighter polling interval would overload the source. For most NAV reporting, a SQL-based refresh key tied to the daily batch is sufficient and far cheaper.

Q: Summarise why the semantic/metrics layer is the architectural keystone for "one definition of AUM."
A: It moves the definition of contested fund metrics out of individual tools and into one governed, executable, version-controlled place that every consumer reads through APIs, so the number cannot drift and every change is auditable. Cube and MetricFlow are two implementations of this pattern — headless and dbt-native respectively — over the gold marts. The result is consistency by construction, with pre-aggregations buying performance and the glossary link keeping the documented and executed definitions aligned.
