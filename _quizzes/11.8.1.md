# Phase 5 · 11.8.1 Secrets management — 100 self-test questions

Q: What is a "secret" in the context of a data platform?
A: A secret is any credential or sensitive value that grants access to a system: database passwords, API keys, TLS private keys, SWIFT connection credentials, OAuth client secrets, and encryption keys. The defining property is that disclosure to an unauthorised party causes harm, so secrets need confidentiality, controlled distribution, auditing, and rotation. Treating a market-data API key the same as ordinary config is the root cause of most credential leaks.

Q: Why is storing a database password in an environment variable considered unacceptable in a regulated fund platform?
A: Environment variables are visible to anyone who can read `/proc/<pid>/environ`, are inherited by child processes, frequently leak into logs and crash dumps, and are dumped by debugging endpoints. They also have no access audit, no rotation, and no expiry, so a leaked value lives forever. In a fund-administration audit, a credential in an env var is an immediate finding because there is no record of who read it or when it last changed.

Q: Why are secrets in config files or `.env` files dangerous even when the file is not committed to git?
A: Files on disk are readable by every process running as that user, are picked up by backups and container image layers, and routinely end up in git history through a mistaken `git add`. There is no access log telling you who copied the file, and rotating the value means editing and redeploying every file copy. A `.env` baked into a Docker image is especially bad because the credential ships to every registry and node that pulls the image.

Q: What three capabilities does a vault give you that environment variables and files fundamentally cannot?
A: A vault provides access policies (only authorised identities can read a given path), an audit log (every read, write, and delete is recorded with who and when), and lifecycle management (rotation, TTLs, and revocation). Env vars and files have none of these — they are inert storage. These three properties are exactly what an auditor or DORA assessor asks you to demonstrate.

Q: What does the lesson state is "the only acceptable answer" for secrets architecture?
A: A vault — a dedicated secret store with access policies and audit logs. The reasoning is that only a vault can enforce who may read a secret, record every access for audit, and manage rotation and revocation centrally. Env vars and files are explicitly rejected because they lack policies, lack audit trails, and make rotation effectively impossible.

Q: What is OpenBao?
A: OpenBao is an open-source secrets-management platform forked from HashiCorp Vault, providing a vault with secret engines, dynamic secrets, leasing, policies, and audit logging. It keeps API and CLI compatibility with the Vault version it forked from, so most Vault tutorials and tooling apply directly. It is the FOSS hands-on tool this lesson uses, mapping conceptually to Azure Key Vault and AWS Secrets Manager.

Q: From which version of HashiCorp Vault was OpenBao forked, and why does that version matter?
A: OpenBao was forked from Vault 1.14.0, the last release published under the open-source MPL 2.0 licence before HashiCorp's licence change. The fork point matters because it is the dividing line: everything up to 1.14.0 is open source and forkable, while later Vault releases carry the restrictive Business Source Licence. This is why OpenBao can legitimately reuse the codebase and remain a free, redistributable project.

Q: What licence change did HashiCorp make to Vault in 2023, and what was the previous licence?
A: In 2023 HashiCorp relicensed Vault (and Terraform, Consul, and others) from the open-source Mozilla Public License 2.0 (MPL 2.0) to the Business Source License 1.1 (BSL). The BSL is a source-available but non-open-source licence that restricts using the software to compete commercially with HashiCorp. This move is what triggered community forks: OpenTofu for Terraform and OpenBao for Vault.

Q: Why does the BSL versus MPL distinction matter for tool selection in a regulated platform?
A: The BSL is not OSI-approved open source, so it restricts commercial competitive use and can create procurement, redistribution, and embedding concerns that legal and compliance teams scrutinise. Choosing OpenBao under MPL 2.0 sidesteps those restrictions and keeps you on a genuinely open, community-governed codebase. For a fund administrator, predictable licensing and avoiding vendor lock-in are real architectural criteria, not just developer preference.

Q: Under which open-source licence is OpenBao distributed?
A: OpenBao is distributed under the Mozilla Public License 2.0 (MPL 2.0), the same OSI-approved licence Vault used before its 2023 relicensing. MPL 2.0 is a weak copyleft licence that permits commercial use, redistribution, and embedding without the competitive-use restrictions of the BSL. This is the licensing reason OpenBao is the recommended FOSS choice in this lesson.

Q: What is the name of the OpenBao command-line binary?
A: The OpenBao CLI binary is `bao`. It mirrors the Vault CLI structure, so commands like `bao secrets enable`, `bao kv put`, `bao auth enable`, and `bao token revoke` work as the Vault equivalents do with `vault`. If you copy a Vault tutorial, the main change is swapping `vault` for `bao`.

Q: What command enables a secrets engine in OpenBao, and give an example?
A: You enable a secrets engine with `bao secrets enable`, for example `bao secrets enable database` to enable the database engine or `bao secrets enable -path=kv kv-v2` to mount a KV version 2 engine at the `kv` path. Enabling mounts the engine at a path so you can then configure roles or write secrets under it. You list mounted engines with `bao secrets list`.

Q: How do you write and read a static key/value secret in OpenBao's KV v2 engine?
A: You write with `bao kv put secret/nav/db username=svc password=s3cr3t` and read with `bao kv get secret/nav/db`. KV v2 keeps version history, so `bao kv get -version=2` retrieves an older version and `bao kv metadata get` shows version info. The path after the mount (`secret/`) is what policies grant access to.

Q: What is the difference between a static secret and a dynamic secret?
A: A static secret is a fixed value you store and read back, such as a password you typed in; it only changes when you rotate it. A dynamic secret is generated on demand by the vault when a client requests it, is unique per request, and is automatically revoked when its lease expires. Dynamic secrets shrink the blast radius because each credential is short-lived and tied to one consumer.

Q: How does OpenBao's database secrets engine produce dynamic credentials?
A: You configure the engine with an admin connection and a role whose creation statements are SQL that create a user and grant privileges. When a client calls `bao read database/creds/<role>`, OpenBao runs those statements against the database, returns a freshly generated username and password with a lease, and schedules automatic revocation at lease expiry. The application never sees a long-lived shared password.

Q: What is a lease in OpenBao and what is its purpose?
A: A lease is metadata OpenBao attaches to a dynamic secret, including a lease ID and a TTL (time-to-live in seconds) for which the credential is valid. Its purpose is to bound the credential's lifetime so it self-expires and is automatically revoked, limiting exposure if it leaks. You can read the lease duration alongside the secret, renew it before expiry, or revoke it early.

Q: What do the `ttl` and `max_ttl` role parameters control for dynamic secrets?
A: `ttl` is the default lease duration assigned to credentials issued by the role, and `max_ttl` is the absolute ceiling beyond which a lease cannot be renewed. They let you make credentials short-lived by default while capping how long any single credential can live even with renewals. Setting a tight `max_ttl` on a database role forces regular re-issuance and bounds the damage from any single leaked credential.

Q: What is the difference between renewing and revoking a lease in OpenBao?
A: Renewing extends a lease's TTL (up to `max_ttl`) so the credential stays valid longer, typically done by a still-running workload that needs continued access. Revoking immediately invalidates the secret, runs the configured revocation statements (for example dropping the database user), and prevents any further renewal. When a lease expires naturally, OpenBao automatically revokes it, which is the self-cleaning property that makes dynamic secrets safe.

Q: A dynamic database credential leaks into a log file. Why is the damage bounded compared with a static password leak?
A: The dynamic credential carries a short TTL and is automatically revoked when its lease expires, so the leaked username and password stop working within minutes or hours rather than living forever. You can also revoke that specific lease immediately without affecting other consumers, because each consumer holds its own distinct credential. A static shared password, by contrast, is valid until someone manually rotates it everywhere it is used.

Q: Why is rotation "impossible" if you get secrets architecture wrong, as the lesson warns?
A: If credentials are scattered across env vars, config files, image layers, and CI definitions with no central registry, you cannot reliably find every copy, so changing the value breaks unknown consumers and gets reverted. Without a vault tracking who holds what, rotation becomes a high-risk, all-hands operation that teams avoid. The result is that leaked credentials are never rotated and live indefinitely, which is exactly the breach scenario the lesson describes.

Q: What is a rotation runbook and why must it be documented?
A: A rotation runbook is a written, step-by-step procedure for replacing a specific credential: where it lives, which consumers use it, the exact commands, and how to verify success and roll back. Documenting it turns rotation from a risky improvisation into a repeatable five-minute operation that anyone on the team can execute under pressure. It is also evidence for auditors that the organisation can rotate credentials on demand, which DORA and fund-administration audits expect.

Q: The lesson's "Done when" says you should rotate a credential as a five-minute documented operation. What makes that target achievable?
A: It is achievable when secrets live in a vault with a single authoritative copy, consumers fetch at runtime rather than caching, and a runbook lists the precise steps. Then rotation is: generate the new value in the vault, let consumers pick it up (or restart them), and revoke the old one — no hunting through files. Dynamic secrets make it even easier because rotation is continuous and automatic by design.

Q: What is "secret zero" (the bootstrap secret problem) and why is it hard?
A: Secret zero is the very first credential a workload needs to authenticate to the vault itself, before it can fetch any other secret — you cannot store it in the vault you are trying to reach. It is hard because that bootstrap token or role credential becomes the one secret you must still distribute and protect by other means. Managed identities and platform-issued tokens solve it by letting the platform vouch for the workload's identity so no static secret zero exists.

Q: What is a managed identity in Azure, and why does the lesson call it "the no-secret ideal"?
A: A managed identity is an identity for an Azure resource (a VM, App Service, AKS workload, function) that Azure itself manages, so the resource obtains tokens from Microsoft Entra ID without any client secret or certificate stored anywhere. It is the no-secret ideal because there is literally no credential to leak, rotate, or audit for that path — the platform handles authentication. This is why the lesson ranks it above any secret store: the best secret is the one that does not exist.

Q: What is the difference between a system-assigned and a user-assigned managed identity?
A: A system-assigned managed identity is created with and tied to the lifecycle of a single Azure resource, and is deleted when that resource is deleted. A user-assigned managed identity is a standalone resource you create independently and can attach to multiple resources, so its lifecycle is decoupled and it can be shared. User-assigned is preferred when several workloads need the same identity or when you want the identity to outlive a recreated resource.

Q: Why does eliminating a credential with a managed identity beat even storing it well in a vault?
A: A vault still holds a real credential that must be protected, audited, rotated, and bootstrapped, so there is always residual risk and operational overhead on that path. A managed identity removes the credential entirely: there is nothing to store, nothing to leak, and nothing to rotate, because the platform issues short-lived tokens on demand. Fewer secrets in existence is strictly fewer things that can be stolen, which is why "no secret" beats "well-stored secret".

Q: Name one service where a managed identity removes a secret entirely (as the lesson asks).
A: A data pipeline running in Azure (for example an AKS workload or an Azure Function) that reads from Azure Storage or queries Azure SQL can use a managed identity instead of a connection string with an embedded key. The workload obtains an Entra token for the storage or database resource at runtime, so no account key or SQL password exists anywhere. That removes the most common leaked credential in a lakehouse ingestion path.

Q: What is workload identity federation and what problem does it solve?
A: Workload identity federation lets an external workload — running outside Azure, such as in GitHub Actions or another cloud's Kubernetes — exchange a signed OIDC token from its own identity provider for an Entra access token, without any stored Azure secret. It solves the cross-platform secret-zero problem: a CI job or non-Azure cluster can authenticate to Azure resources with no client secret to leak or expire. The trust is configured as a federated identity credential on a user-assigned managed identity or app registration.

Q: Can you configure a federated identity credential on a system-assigned managed identity?
A: No. Configuring a federated identity credential is supported on user-assigned managed identities (and app registrations), but not on system-assigned managed identities. This is a concrete gotcha: if you plan to use workload identity federation, you must create a user-assigned managed identity, because attempting it on a system-assigned one is unsupported.

Q: How does a managed identity authenticate without a stored credential — what does the platform actually do?
A: The Azure platform issues the resource a short-lived token from Entra ID through a local metadata or token endpoint that only that resource can reach, and the resource presents that token to the target service. Identity is proven by the platform vouching for the resource (and by cryptographic OIDC signing in the federated case), not by a shared secret. Because tokens are short-lived and minted on demand, there is no long-lived credential to manage.

Q: What is the Secrets Store CSI Driver and what problem does it solve in Kubernetes?
A: The Secrets Store CSI Driver is a Kubernetes container-storage-interface driver that mounts secrets from an external store (such as Azure Key Vault) into a pod as files in a volume, fetched at pod start. It solves the problem of getting secrets to workloads without baking them into images, ConfigMaps, or env vars on disk. The pod reads its credentials from the mounted path at runtime, so the secret never lands in the image or the Kubernetes manifest.

Q: What is a `SecretProviderClass` in the Azure Key Vault CSI provider?
A: A `SecretProviderClass` is the custom resource that tells the CSI driver which external store and which specific secrets to fetch, plus how to mount or sync them. It declares the Key Vault name, the objects (secrets, keys, certificates) to retrieve, and the identity to authenticate with, and a pod references it on its CSI volume. It is the portable, declarative binding between a pod and the secrets it needs.

Q: When the Azure Key Vault CSI driver mounts secrets into a pod, where does the content actually live?
A: The driver fetches the secret content over gRPC from the external store and mounts it into the pod as a `tmpfs` volume, meaning the content lives in memory, not on the node's disk. Because `tmpfs` is RAM-backed and per-pod, the secret is not written to persistent storage and disappears when the pod stops. This is the mechanism that satisfies "get secrets to workloads without writing them to disk".

Q: What is the recommended authentication method for the Azure Key Vault CSI driver in AKS, and why?
A: The recommended method is Microsoft Entra Workload Identity, which uses federated identity credentials linking a Kubernetes ServiceAccount to a user-assigned managed identity, rather than a stored client secret or node-pool pod-managed identity. It is recommended because it eliminates static credentials in the cluster: the pod presents a signed ServiceAccount token that Entra verifies directly. Enabling it requires `--enable-oidc-issuer` and `--enable-workload-identity` on the AKS cluster.

Q: What is the default rotation poll interval for the Azure Key Vault CSI driver's autorotation, and what does autorotation do?
A: With autorotation enabled, the driver polls the store on a configurable interval — the default is two minutes — and updates the mounted file content and any synced Kubernetes secret when the value changes in Key Vault. This means a rotated secret can reach running pods without a redeploy, narrowing the window in which a stale credential is in use. You must still ensure the application re-reads the mounted file or restarts to pick up the new value.

Q: The CSI driver can sync mounted secrets to a native Kubernetes `Secret`. What is the trade-off?
A: Using the `secretObjects` field, the driver can mirror mounted content into a Kubernetes `Secret`, which is convenient because workloads can consume it as an env var or by existing tooling. The trade-off is that a Kubernetes `Secret` is only base64-encoded in `etcd`, not encrypted by default, so syncing reintroduces a copy outside the pod's `tmpfs` mount. If you sync, you should enable encryption at rest for `etcd` and tightly RBAC the synced secret.

Q: Why is a Kubernetes `Secret` object by itself not a sufficient secrets-management solution?
A: A core Kubernetes `Secret` is merely base64-encoded (not encrypted) and stored in `etcd`, readable by anyone with the right RBAC or `etcd` access, with no rotation, no per-access audit, and no lifecycle policy. Base64 is encoding, not encryption, so it offers zero confidentiality. It is an injection mechanism at best; the source of truth and the policy/audit/rotation must come from a real vault or managed identity.

Q: How should secrets be delivered to a CI/CD pipeline without writing them to disk?
A: The pipeline should authenticate to the vault or cloud using a short-lived, platform-issued identity — OIDC federation from the CI provider to the vault or to Entra, rather than a stored long-lived token — and fetch secrets into memory for the job. The secrets are injected as masked, in-memory variables scoped to the job and never committed to the repo or a file. This pattern means even a fully compromised pipeline definition contains no usable standing credential.

Q: Why is hardcoding a long-lived API token in a GitHub Actions secret weaker than OIDC federation?
A: A stored Actions secret is a standing credential: it is valid until manually rotated, can be exfiltrated by a malicious workflow change, and must be copied to every repo or org that needs it. OIDC federation instead has the CI provider mint a short-lived signed token per run, which the vault or Entra exchanges for an access token with no stored secret. The federated approach removes the long-lived credential entirely, which is the whole point of the no-secret ideal.

Q: What is the access-policy model in a vault, conceptually?
A: A vault attaches policies to authenticated identities that grant fine-grained capabilities (read, create, update, delete, list) on specific paths, following least privilege. An identity can only touch the exact secrets its policy allows, and everything is denied by default. This is what lets you say, for auditors, that only the NAV service identity can read the NAV database path and nothing else.

Q: What is an OpenBao policy and how does it differ from a Kubernetes RBAC rule?
A: An OpenBao policy is an ACL document mapping paths (like `secret/data/nav/*`) to capabilities (`read`, `create`, `update`, `delete`, `list`), attached to tokens or auth roles, governing access to secrets inside the vault. Kubernetes RBAC governs access to Kubernetes API objects (pods, secrets, deployments) within the cluster, not the contents of an external secret store. They are complementary: RBAC controls who can deploy the pod, the OpenBao policy controls which secrets that pod's identity can read.

Q: What is an audit log in a secret store and why is it non-negotiable for fund administration?
A: An audit log is an append-only record of every operation against the vault — each secret read, write, lease, and revocation with the identity, timestamp, and request details. It is non-negotiable because regulated fund administration and DORA require demonstrable evidence of who accessed credentials to transfer-agency, market-data, and warehouse systems, and when. Without it you cannot answer an auditor's "who read this credential" question, and you cannot detect or investigate a breach.

Q: An auditor asks "who has read the production NAV database password in the last 90 days?" How does a vault answer this, and how would env vars answer it?
A: With a vault, you query the audit log filtered by the secret path and time window, returning the exact identities and timestamps of each read. With env vars or files there is no answer at all — there is no record of reads, so the only honest response is "we cannot tell," which is itself an audit failure. This contrast is the core argument for a vault in a regulated platform.

Q: What is the principle of least privilege as applied to secrets?
A: Least privilege means each identity is granted access only to the specific secrets it needs to do its job and nothing more, and only the capabilities required (often read-only). Applied to secrets, the transfer-agency ingestion service can read only the transfer-agency credential, not the warehouse admin password. This limits the blast radius: a compromised service exposes only its own narrow set of secrets.

Q: Why does the lesson say `git grep -i password` should "return nothing real"?
A: Because credentials must never live in the repository; the check confirms no password, key, or token has been committed to source control where it would persist in history forever and propagate to every clone. "Nothing real" allows incidental matches like the word `password` in documentation or a variable name, but no actual secret value. Passing this check is a baseline "Done when" condition and a standard pre-audit hygiene step.

Q: `git grep -i password` returns nothing, but you suspect a credential was committed and later removed. Why is the grep insufficient?
A: `git grep` only searches the current working tree or a given revision, not the full history, so a secret deleted in a later commit still sits in earlier commits and is recoverable by anyone with the repo. You must search history (for example `git log -p` scanning or a dedicated history scanner like `gitleaks` or `trufflehog`) and, if found, rotate the secret immediately and purge history. Removing a leaked secret from HEAD does not make it safe — only rotation does.

Q: A secret was committed to git two years ago. The fix is not "delete the line." What is the correct remediation?
A: The credential must be treated as compromised and rotated immediately, because it has been exposed in every clone, fork, and backup since the day it was committed and may already be in attacker hands. Purging it from git history (with filter-repo or BFG) and force-pushing is secondary cleanup; it does not undo the exposure. The rule is: once a secret hits a shared repo, it is burned — rotate first, scrub second.

Q: Why must secrets be kept out of container image layers specifically?
A: Docker image layers are immutable and shared: a secret added in any layer remains in the image even if a later layer deletes the file, and the image is pushed to a registry and pulled to every node. Anyone who can pull the image can extract the credential from its layers. This is why secrets are mounted or fetched at runtime, never `COPY`-ed or `ARG`-ed into the build.

Q: What is the risk of passing a secret via Docker `--build-arg` or `ARG`?
A: Build args are recorded in the image history and metadata and can be inspected with `docker history`, so a secret passed as a build arg is effectively embedded in the image even though it never appears in the final filesystem. Anyone with the image can recover it. The correct approaches are BuildKit secret mounts (which are not persisted to layers) or, better, fetching the secret at runtime rather than build time.

Q: A teammate suggests storing the OpenBao root token in the CI system so jobs can administer the vault. Why is this dangerous, and what is the alternative?
A: The OpenBao root token has unlimited privileges over the entire vault, so storing it in CI makes the CI system a single point of total compromise — anyone who reads that token owns every secret. The alternative is to authenticate CI with a narrowly scoped auth method (AppRole with a tight policy, or OIDC/JWT federation) that grants only the specific paths the job needs, with short TTLs. The root token should be generated for emergencies, used rarely, and then revoked.

Q: What is the AppRole auth method in OpenBao and when is it appropriate?
A: AppRole lets a machine or application authenticate using a `role_id` (relatively stable, like a username) and a `secret_id` (sensitive, like a password, ideally short-lived and delivered out-of-band), exchanging them for a token with a scoped policy. It is appropriate for automated workloads and CI that cannot use a cloud-native identity. The security depends on protecting the `secret_id` delivery — for example having a trusted orchestrator issue a wrapped, single-use `secret_id`.

Q: How do you rotate an AppRole `secret_id` in OpenBao?
A: You generate a fresh `secret_id` with `bao write -f auth/approle/role/<role>/secret-id`, distribute it to the workload, and then destroy or let expire the old one. Because `secret_id`s can be short-lived and multiple can exist per role, you can roll without downtime by issuing the new one before retiring the old. This is the mechanical basis of a quick AppRole rotation runbook.

Q: How do you revoke a leased token or credential in OpenBao if you suspect compromise?
A: You revoke the token with `bao token revoke <token>` or revoke a specific lease with `bao lease revoke <lease_id>`, which immediately invalidates it and runs any revocation statements (such as dropping a generated database user). For broad incidents you can revoke by prefix to invalidate many leases at once. Immediate revocation is the incident-response advantage dynamic secrets give you over static passwords.

Q: What does it mean that OpenBao must be "unsealed," and why does this matter operationally?
A: OpenBao encrypts its storage with a master key that is itself encrypted, and on startup it is "sealed" — it cannot read any secret until the master key is reconstructed by providing a quorum of unseal key shares (Shamir's Secret Sharing) or an auto-unseal mechanism. This matters because a restart leaves the vault unavailable until unsealed, so you need an unseal runbook or auto-unseal (using a cloud KMS) to avoid an outage. It is a deliberate safety property: stolen storage is useless while sealed.

Q: What is auto-unseal and what trade-off does it introduce?
A: Auto-unseal delegates the master-key decryption to an external KMS or HSM (for example a cloud key vault), so OpenBao unseals itself on startup without an operator supplying key shares. The trade-off is operational availability versus the manual-quorum security boundary: you avoid downtime and human key-handling, but you now trust the external KMS and the identity used to reach it. In practice auto-unseal is standard for production because manual unsealing does not scale and is itself a credential-handling risk.

Q: What is an HSM and how does it relate to a secret store?
A: A Hardware Security Module is a tamper-resistant device (or validated cloud equivalent) that generates and stores cryptographic keys such that the private key material never leaves the device; operations are performed inside it. A secret store can use an HSM to protect its master/encryption keys, raising assurance from software-only protection. Azure Key Vault offers an HSM-backed tier (Managed HSM) for exactly this higher-assurance requirement, which regulated finance often mandates.

Q: How does Azure Key Vault map conceptually onto OpenBao for an architect comparing them?
A: Both are vaults: they store secrets behind access policies, log access for audit, support keys and certificates, and integrate with Kubernetes via the CSI driver. Key Vault is the managed corporate offering with HSM tiers, Entra RBAC, and tight integration with managed identities, while OpenBao is the self-hosted FOSS option with rich dynamic secret engines you operate yourself. The architect's job is the build-versus-buy decision: managed convenience and managed-identity integration versus self-hosted control and dynamic-secret flexibility.

Q: Where does Azure Key Vault get its access control, and how does that differ from OpenBao policies?
A: Azure Key Vault uses Entra ID identities with Azure RBAC (or legacy access policies) to authorise operations, so a managed identity or user is granted a role like Key Vault Secrets User. OpenBao uses its own auth methods and ACL policies attached to tokens. The practical difference is that Key Vault leans on the cloud identity plane (managed identities work natively), whereas OpenBao manages identity and policy internally and must be wired to whatever auth you choose.

Q: Why is "secrets in env vars" especially risky in a containerised microservice architecture?
A: Container env vars are visible in the container's `/proc`, in orchestrator API responses (`kubectl describe pod` shows them), in crash dumps, and are inherited by any subprocess or sidecar. With many small services, each leak point multiplies, and an exposed orchestrator dashboard can reveal every service's credentials at once. Runtime fetching into memory or `tmpfs` mounts avoids this broad exposure surface.

Q: In a Luxembourg fund-administration platform connecting to a transfer-agency system, why is a vault the only acceptable place for that connection's credentials?
A: The transfer-agency link grants access to investor subscription/redemption and registry data, so its credential is high-value and its access must be auditable and rotatable to satisfy CSSF and DORA expectations. A vault enforces that only the TA-ingestion identity can read it, records every access, and lets you rotate it on a documented schedule or instantly on suspicion. Putting it in an env var or config file would be an immediate audit finding and would make rotation effectively impossible.

Q: How would dynamic secrets reduce risk for the database holding daily NAV records?
A: Each NAV pipeline run could fetch a short-lived database credential scoped to exactly the privileges it needs, automatically revoked when its lease expires after the run. There is no shared standing password to leak, and a credential exposed in run logs stops working within minutes. The audit log also shows precisely which run obtained access at which time, which is valuable for reconstructing a NAV-calculation incident.

Q: A market-data vendor API key is currently in a `docker-compose.yml`. Walk through moving it to OpenBao per the lesson's "Do".
A: First write the key into OpenBao, for example `bao kv put secret/marketdata/vendor api_key=<value>`, and remove it from the compose file. Then grant the consuming service a scoped policy and an auth method (AppRole or a cloud identity) so it can read only that path. Finally have the service fetch the key from OpenBao at startup via the API or an injected mount rather than from an env var, and confirm `git grep -i api_key` finds no real value.

Q: The lesson says wire "one service to fetch its secret at runtime, not from disk." What does fetching at runtime actually mean?
A: It means the application, at startup or on demand, authenticates to the vault with its identity and pulls the secret into memory (or reads it from a `tmpfs` CSI mount), rather than reading a value baked into a file, image, or env var. The credential then exists only in the running process's memory and is gone when it stops. This removes the persistent on-disk copy that backups, images, and logs would otherwise capture.

Q: What is the difference between encoding (like base64) and encryption, and why does it matter for secrets?
A: Encoding (base64) is a reversible transformation with no key, so anyone can decode it instantly; it provides zero confidentiality. Encryption transforms data with a key so that only key-holders can recover the plaintext. It matters because a base64-"protected" Kubernetes `Secret` or config value is not protected at all — treating encoding as security is a classic and dangerous mistake.

Q: What is "encryption at rest" for a secret store, and does it remove the need for the store itself?
A: Encryption at rest means the stored secret data is encrypted on disk so that stealing the storage media or `etcd` snapshot does not reveal plaintext. It does not remove the need for the store, because the store still provides the access policies, audit log, and rotation that encryption alone does not. Encryption at rest is one necessary layer; the vault's policy and audit machinery is the other.

Q: Why should a leaked credential always be rotated rather than just "removed from where it leaked"?
A: Once a credential is exposed — in a log, a repo, a screenshot, a Slack message — you must assume an attacker has captured it, so the only way to neutralise it is to make the value itself invalid by rotating it. Deleting the visible copy does nothing to the copy an attacker already took. This is why the rotation runbook is the centre of incident response, and why fast, documented rotation is a "Done when" criterion.

Q: What is the danger of a secret with no expiry or rotation schedule?
A: A non-expiring secret accumulates exposure over time — it ends up in more logs, more backups, more developer machines, and more former employees' memories — while you lose track of everywhere it lives. If it leaks you may never know, and even if you suspect it you may be unable to rotate it safely. Short-lived, automatically rotated credentials structurally cap this exposure window.

Q: Why are short-lived (ephemeral) credentials inherently safer than long-lived ones, even with a perfect vault?
A: A short-lived credential is only useful to an attacker for the brief window before it expires, so even a successful theft yields little, and there is no manual rotation step that can be forgotten. Long-lived credentials remain valuable indefinitely and depend on humans remembering to rotate them. Ephemerality moves security from a process you must remember to perform into a property the system enforces automatically.

Q: What is credential sprawl and why is it the enemy of rotation?
A: Credential sprawl is the same secret copied into many places — env files, CI variables, developer laptops, image layers, wikis — with no central registry of where it lives. It is the enemy of rotation because you cannot rotate what you cannot find: changing the value breaks unknown consumers, so teams avoid rotating at all. Centralising on a vault with runtime fetch eliminates sprawl by keeping a single authoritative copy.

Q: Why does centralising secrets in a vault make rotation tractable?
A: With one authoritative copy in the vault and consumers fetching at runtime, rotating means updating that single value (or letting a dynamic engine reissue it) and consumers transparently pick up the new one. There is no hunt across files and pipelines, and the audit log confirms the change propagated. This is precisely why the lesson positions the vault as the prerequisite for a five-minute documented rotation.

Q: What is a secret-scanning tool and where in the workflow should it run?
A: A secret-scanning tool (such as gitleaks or trufflehog) detects credential patterns and high-entropy strings in code, commits, and history before they reach a shared location. It should run as a pre-commit hook and as a CI gate, and periodically over full history, so a secret is caught before push and the build fails if one slips in. It is a detective control that backs up the preventive control of never putting secrets in code.

Q: Pre-commit secret scanning passes but a secret still reaches production. Name two realistic gaps.
A: First, scanners are pattern-based and miss novel or unusual secret formats, custom tokens, or values that look like ordinary strings (low entropy), so a real credential can slip through. Second, developers can bypass the hook (committing with `--no-verify`) or the secret enters through a path the scanner does not cover, such as a manually set CI variable or a deployment manifest. Scanning is a safety net, not a substitute for not having secrets in code at all.

Q: How does a vault's audit log support DORA's incident-reporting and ICT-risk requirements?
A: DORA requires financial entities to manage ICT risk and report serious incidents with evidence, which includes knowing what was accessed during an event. A vault's audit log provides the precise, tamper-evident record of which identity read which credential and when, supporting both detection and the forensic reconstruction a report demands. Without it, an entity cannot evidence the access trail regulators expect.

Q: Why is "passwords in config files is an immediate audit finding" stated so categorically in this lesson?
A: Because a plaintext credential in a config file violates the basic control expectations of any financial-services audit: it has no access control beyond filesystem permissions, no audit trail, no rotation, and is trivially copyable. Auditors do not need to assess context — its mere presence demonstrates the absence of secrets management. It is a deterministic finding, which is why the lesson treats it as a hard line.

Q: What is the relationship between secrets management and the Phase-6 security work the lesson references?
A: The lesson frames secrets architecture as a prerequisite: you cannot do meaningful security hardening, identity, or compliance work if credentials are scattered in env vars and files. Getting secrets into a vault with policies and audit, and eliminating credentials via managed identities, is the foundation the later security and audit work builds on. In short, you must control your secrets before you can claim any security posture.

Q: How do you list which secret engines are currently mounted in OpenBao?
A: You run `bao secrets list`, which shows each mounted engine, its path, type, and configuration such as default and max lease TTLs. This is how you inventory what is enabled — for example confirming a `database` engine and a `kv` engine are mounted at their expected paths. Knowing the mount paths is essential because policies grant access by path.

Q: What does `bao status` tell you, and why check it first when OpenBao seems unreachable?
A: `bao status` reports whether the vault is initialised, sealed or unsealed, the seal type, and HA leadership status. Check it first because the most common reason clients cannot read secrets after a restart is that the vault is sealed — a sealed vault accepts the status call but refuses all secret operations. If it reports sealed, the fix is unsealing (or fixing auto-unseal), not debugging the client.

Q: A workload gets a 403/permission-denied when reading a secret from OpenBao. What is the first thing to check?
A: Check the policy attached to the workload's token or auth role against the exact path it is requesting, because a 403 almost always means the policy does not grant the needed capability on that path. Confirm the path (including the KV v2 `data/` segment, which is a frequent gotcha), the capability (`read`), and that the token is still valid and not expired. The denial is an authorisation problem, not a connectivity one.

Q: A workload's OpenBao token works at first but later all calls fail with permission errors. What likely happened?
A: The token's TTL most likely expired, so it is no longer valid and must be renewed before expiry or re-acquired through the auth method. Long-running workloads must renew their token (and any leased secrets) on a schedule, or use an auth method that re-authenticates automatically. The fix is to add token/lease renewal to the workload, not to lengthen the TTL indefinitely, which would weaken the security model.

Q: In KV v2, why does reading at `secret/myapp` sometimes fail while `secret/data/myapp` works?
A: KV version 2 inserts a `data/` segment into the API path, so the API read path is `secret/data/myapp` even though the CLI `bao kv get secret/myapp` hides this by adding it for you. Policies must grant capabilities on the `data/` path, and direct API calls or misaligned policies that omit `data/` will fail. This path-versioning detail is a classic KV v2 stumbling block.

Q: Why should application logs and error handlers be reviewed as part of secrets hygiene?
A: Applications frequently log connection strings, full request URLs with embedded tokens, or exception messages containing credentials, turning the log store into a secret leak. Reviewing and scrubbing logs (and configuring libraries to redact secrets) prevents credentials from being copied into a system that is widely readable and long-retained. A credential in a centralised log aggregator can be more exposed than the original config.

Q: What is the danger of a debugging endpoint that dumps environment variables or config?
A: Such an endpoint (a misconfigured actuator, a `/debug` route, or a verbose error page) exposes every env var and config value — including secrets — to anyone who can reach it, often unauthenticated. This is a common breach vector precisely because secrets sit in env vars. Removing secrets from env vars and disabling debug dumps in production both mitigate it.

Q: How does a managed identity simplify the secret-rotation runbook for an Azure-hosted pipeline?
A: It removes that credential from the runbook entirely: there is no password to rotate because the platform issues short-lived tokens automatically, so the rotation step for that connection simply does not exist. The runbook shrinks to the secrets that genuinely cannot be eliminated. This is the operational payoff of "the best secret is no secret."

Q: Why is it better for an AKS pod to use workload identity than to mount a Key Vault client secret?
A: A mounted client secret is a standing credential that must be created, stored, rotated, and protected, and can be exfiltrated from the pod. Workload identity instead links the pod's ServiceAccount to a managed identity via a federated, signed token, so there is no stored client secret to leak or rotate. It is strictly fewer credentials in existence, which is the security goal.

Q: What does "least exposure window" mean and how do dynamic secrets minimise it?
A: The exposure window is the span during which a leaked credential remains usable by an attacker. Dynamic secrets minimise it by issuing credentials with short TTLs that auto-revoke, so even an undetected leak is only exploitable for that brief lease, and you can revoke the specific lease instantly on suspicion. Minimising this window is the core security argument for short-lived over static credentials.

Q: Why is sharing one database account across all microservices an anti-pattern from a secrets standpoint?
A: A shared account means one leaked credential compromises every service, you cannot tell from the database which service performed an action, and rotating it requires coordinating a simultaneous change across all consumers. Per-service identities (ideally dynamic, per-request) give isolation, attribution, and independent rotation. Shared credentials maximise blast radius and defeat least privilege.

Q: How do you tighten the blast radius of a single OpenBao policy?
A: Scope the policy to the narrowest paths the identity needs, grant only the required capabilities (prefer `read` over write), and avoid wildcards that widen access beyond intent. Pair this with short token/lease TTLs so even a leaked token is briefly useful. The combination of narrow path, minimal capability, and short lifetime is what shrinks the damage a compromised identity can do.

Q: What role does a region-pairing or residency requirement play when choosing where to host secrets for EU funds?
A: EU fund administration often requires that sensitive data, including secrets governing access to investor data, stays within EU jurisdictions for residency and regulatory reasons. This constrains where you deploy OpenBao or which Key Vault region you use, and informs DR replication choices for the secret store. The architect must ensure the secret store's location and its backups respect EU residency just like the data it protects.

Q: Why must the secret store itself have a disaster-recovery plan?
A: If the vault is unavailable, every workload that fetches secrets at runtime may fail to start or reconnect, so the vault becomes a critical dependency whose outage halts the platform. Its DR plan must cover backup/restore of its storage, recovery of unseal keys or auto-unseal access, and replication within residency constraints. A secrets architecture that ignores vault DR trades one risk (leaked secrets) for another (platform-wide outage).

Q: What is wrapping (response wrapping) in the OpenBao/Vault model and when is it useful?
A: Response wrapping returns a single-use wrapping token instead of the actual secret, and the intended recipient unwraps it to retrieve the value, with the wrap being detectably consumed. It is useful for securely delivering a `secret_id` or other bootstrap material across a trust boundary, because if the wrapping token is intercepted and used, the legitimate consumer's unwrap fails, revealing the interception. It narrows the secret-zero distribution risk.

Q: How does a vault support certificates and PKI, and why does that matter for SWIFT or TLS connectivity?
A: A vault can run a PKI secrets engine that issues short-lived certificates on demand, so services obtain TLS certs that auto-expire rather than holding long-lived ones. This matters for SWIFT and other secure financial connectivity where mutual TLS is common: short-lived, centrally issued certs reduce the risk of a stolen long-lived private key and automate renewal. It applies the dynamic-secret philosophy to certificates.

Q: Why should developers never have standing read access to production secrets?
A: Standing human access means every developer is a leak point and a target, and an auditor sees a large, untracked population that can read production credentials. Access should be just-in-time, brokered through the vault with approval and short TTLs, and fully logged. In regulated fund administration, demonstrating that no human holds standing production-secret access is a meaningful control.

Q: What is "break-glass" access and how does it fit secrets management?
A: Break-glass is an emergency, highly privileged access path (such as the OpenBao root token or an emergency admin identity) used only when normal automated paths fail, with its use heavily logged and alerted. It fits secrets management as the controlled exception: it exists, is protected, and any use triggers review and rotation afterward. The goal is that it is essentially never used and always noticed when it is.

Q: How would you verify, for the "Done when" check, that a service truly fetches its secret at runtime and not from disk?
A: Inspect the running container: confirm no secret appears in its image layers, env vars (`kubectl describe pod` / `docker inspect`), or on-disk config, and that the only place the value exists is in process memory or a `tmpfs` CSI mount. You can also rotate the secret in the vault and confirm the service picks up the new value on restart or via autorotation. If you can change it in one place and the service follows, it is genuinely fetching at runtime.

Q: Why is a vault's audit log typically configured to write to multiple destinations?
A: If a single audit destination is unavailable, a strict vault can refuse operations to avoid acting without an audit record, so multiple destinations prevent a logging outage from becoming a vault outage while still guaranteeing every operation is logged somewhere. Redundant audit sinks (file plus syslog, for example) also protect the integrity requirement that no access goes unrecorded. For regulated use, a guaranteed, redundant audit trail is the point.

Q: What is the security argument for `tmpfs` (memory-backed) secret mounts over writing secrets to a regular file?
A: A `tmpfs` mount keeps the secret in RAM, so it is never written to the node's persistent disk, does not survive a reboot, and is not captured by disk backups or forensic disk imaging. A regular file persists, can be backed up, and can be recovered from the underlying storage. Keeping secrets in `tmpfs` satisfies the "without writing them to disk" requirement and shrinks where the secret can be found.

Q: How does eliminating credentials with managed identities improve auditability rather than hurt it?
A: It might seem to remove an audit point, but Entra logs token issuance and the target resource logs access by that identity, so you retain a trail without a credential to manage. You trade the audit of a secret's reads for the audit of an identity's token requests and resource accesses, which is cleaner because there is no shared password to attribute. Fewer secrets means fewer ambiguous, shared-credential accesses in the logs.

Q: What is the difference between the `kv-v1` and `kv-v2` secrets engines in OpenBao?
A: `kv-v1` is a simple key/value store that overwrites a secret in place with no history, while `kv-v2` adds versioning, soft-delete, and metadata, keeping previous versions you can read or roll back to. The cost of `kv-v2` is the extra `data/` path segment in API calls and policies, which trips people up. For most platforms `kv-v2` is preferred because version history aids audit and recovery from a bad write.

Q: When pulling a secret in a CI job, why prefer a job-scoped, masked variable over exporting it to a shared environment?
A: A job-scoped, masked variable exists only for that job's duration and is redacted in logs, so it does not persist into later stages, the runner's environment, or build artefacts. Exporting it broadly lets subsequent steps, caches, and possibly forked workflows read it, widening exposure. Scoping the secret to the smallest unit of work that needs it, in memory and masked, matches the least-exposure principle.

Q: Summarise the lesson's ranked hierarchy for handling a credential, best to worst.
A: Best is to eliminate the credential entirely with a managed identity so there is nothing to leak or rotate; next is a vault with access policies, audit logging, and short-lived dynamic secrets fetched at runtime; worse is a vault holding static secrets; and unacceptable is plaintext credentials in environment variables or config files. The whole lesson is about climbing this ladder: remove what you can, vault the rest, and never leave secrets in env vars or files.
