# Phase 6 · 9.4.2 + 10.4.1 Data quality engineering (Great Expectations / Soda) — 100 self-test questions

Q: What are the six data-quality dimensions this lesson maps to executable rule types?
A: They are accuracy, completeness, consistency, timeliness, validity, and uniqueness. Each dimension answers a different question about a dataset — is the value correct, is anything missing, do related values agree, is it fresh, is it well-formed, and are there no improper duplicates. Naming the dimension a rule belongs to is what lets you defend in an audit why a given check exists and at what severity.

Q: Define the completeness dimension and give the rule type that enforces it.
A: Completeness asks whether all required data is present — no missing values where a value is mandatory. It is typically enforced with a not-null / missing-count check, e.g. a SodaCL `missing_count` or a Great Expectations `expect_column_values_to_not_be_null`. In a NAV pipeline, a completeness rule on the holdings table guards against a partial load that would silently shrink the portfolio.

Q: Define the validity dimension and contrast it with accuracy.
A: Validity asks whether a value conforms to its expected format, type, or domain — for example that an `ISIN` is 12 characters matching the right pattern. Accuracy is stronger: it asks whether the value is actually correct in the real world, not merely well-formed. A typo'd-but-syntactically-valid ISIN passes validity yet fails accuracy, which is why validity checks are cheap to automate while accuracy usually needs a reference source.

Q: What does the uniqueness dimension guard against, and what rule type expresses it?
A: Uniqueness guards against improper duplicate rows or duplicate key values where each should appear once. It is expressed as a duplicate check on a column or set of columns, e.g. SodaCL `duplicate_count` or `expect_column_values_to_be_unique`. For a positions table keyed by (`fund_id`, `isin`, `valuation_date`) a uniqueness violation usually signals a double-load or a broken upsert.

Q: What does the timeliness dimension assert and why is it easy to miss?
A: Timeliness asserts the data is sufficiently fresh — loaded or updated within an acceptable window relative to "now" or to a business cut-off. It is easy to miss because stale data is often perfectly valid and complete; nothing is malformed, the rows are just old. A timeliness check on an FX-rates table catches yesterday's rates feeding today's NAV, a failure no format or null check would see.

Q: What does the consistency dimension cover, and give a fund-data example?
A: Consistency covers agreement between related values, across columns, rows, or systems — the data does not contradict itself. A fund-data example is that the sum of position market values must reconcile to the fund's total net assets used in the NAV, or that a security's currency in the holdings table matches the same security's currency in the instrument master. It is often the hardest dimension to encode because it requires a cross-dataset or aggregate rule, not a single-column assertion.

Q: How does accuracy as a DQ dimension typically get enforced when you have no ground truth?
A: Without an authoritative reference you cannot check accuracy directly, so you approximate it with bounds and reasonableness checks derived from observed history — a price within a plausible range, a return below a threshold, a count near the prior day's. These are validity/reasonableness proxies; true accuracy needs a golden source such as a vendor price or a counterparty confirmation to compare against. Always document that a "price in range" rule is a proxy, not a guarantee of correctness.

Q: Why does mapping each rule to a named dimension matter for a regulated architect?
A: Because regulators and auditors expect "accuracy, completeness, timeliness" to be enforced, not just asserted in prose, and dimension labels turn that prose into a traceable control framework. When asked "how do you ensure completeness of NAV inputs?" you point at the specific completeness checks and their severities rather than waving at the pipeline. The mapping is the bridge from audit language to executable code.

Q: A check verifies an `LEI` is exactly 20 alphanumeric characters. Which dimension is that?
A: That is a validity check — it tests conformance to the LEI format (20 characters, the ISO 17442 structure) without judging whether the LEI is the correct one for the entity. Confirming the LEI actually belongs to the right legal entity would be an accuracy check needing a reference lookup against the GLEIF database. Conflating the two is a common mislabel.

Q: A check verifies the holdings load arrived before the 18:00 cut-off. Which dimension, and what is the gotcha?
A: That is timeliness. The gotcha is that timeliness must be measured against a business deadline or freshness SLA, not merely "is the table non-empty" — a table can be full of stale data and still pass a naive presence check. Express it against a max-age or an expected load timestamp so an old-but-present load still fails.

Q: What is Great Expectations, at the level of "what layer does it provide"?
A: Great Expectations (GX) is an open-source Python data-validation framework that provides the rule-and-report layer for data quality: you declare assertions called Expectations, group them, validate batches of data against them, and publish human-readable results. It is not an orchestrator or a database; it sits beside your pipeline and answers "did this data meet its declared expectations." In GX 1.x the workflow is built around a `DataContext` as the entry point.

Q: What is the atomic building block called an Expectation, and give one concrete instance?
A: It is a single declarative, verifiable assertion about data — the smallest unit of a rule. A concrete instance is `expect_column_values_to_be_between` applied to `nav_per_share` with min 0 and max 10000, or `expect_column_values_to_not_be_null` on `isin`. GX ships a large library of these built-in types, and each is human-readable and individually testable.

Q: What is an Expectation Suite in Great Expectations?
A: An Expectation Suite is a named collection of Expectations that together describe what "good" looks like for a dataset. It is the rule set you version, review, and run as a unit — for example a `silver_holdings_suite`. Grouping Expectations into a suite is what lets you treat data quality as a reviewable artifact rather than scattered assertions.

Q: What is a Checkpoint in Great Expectations and why is it the production entry point?
A: A Checkpoint is the primary object for validating data in production: it bundles one or more Validation Definitions (pairing data with a suite) together with an `action_list` of things to do with the results. It is the production entry point because a single Checkpoint run both validates and triggers follow-up actions such as updating Data Docs or sending alerts. You schedule the Checkpoint, not the individual Expectations.

Q: In GX 1.x, what is a Validation Definition and what does it link?
A: A Validation Definition is a fixed, reusable reference that links a Batch Definition (which data) to an Expectation Suite (which rules). Running it produces a Validation Result. It exists so a Checkpoint can refer to a stable "validate this data with these rules" pairing rather than re-wiring data and suite each run.

Q: What is a Batch Definition in GX 1.x?
A: A Batch Definition contains all the information needed to identify and retrieve a specific set of data — a Batch — from a configured Data Source, such as a whole table or a daily partition. It is how GX knows precisely which slice of data to validate, e.g. "the holdings rows for `valuation_date = today`." It is paired with a suite inside a Validation Definition.

Q: What are Data Docs in Great Expectations?
A: Data Docs are auto-generated, human-readable static HTML pages that render your Expectation Suites and Validation Results. They turn raw validation output into a browsable report showing which Expectations passed or failed and on what data. They are the "report" half of GX's rule-and-report layer and are often published to a shared location so non-engineers can read DQ outcomes.

Q: What is the `DataContext` in Great Expectations 1.x?
A: The `DataContext` is the primary entry point for a GX deployment — it holds the configuration and provides the methods to create and manage data sources, suites, validation definitions, checkpoints, and actions. You typically start a script with something like `context = gx.get_context()`. Almost every GX operation goes through the context.

Q: How do you make a Great Expectations Checkpoint update Data Docs after it runs?
A: You include an `UpdateDataDocsAction` in the Checkpoint's `action_list`, for example `UpdateDataDocsAction(name="update_all_data_docs")`. When the Checkpoint runs, that action rebuilds the Data Docs sites so the newest Validation Result appears in the HTML report. Without it, validation runs but the published docs do not refresh.

Q: What does a GX Validation Result tell you beyond pass/fail?
A: A Validation Result records, per Expectation, whether it succeeded plus the observed values that drove the outcome — counts, unexpected value samples, and the percentage that failed. This detail is what makes a failure actionable: instead of "completeness failed" you see "3.2% of `isin` were null, sample rows attached." Data Docs render exactly this so an analyst can triage without rerunning code.

Q: What is Soda Core?
A: Soda Core is the open-source command-line data-quality tool that scans a data source against checks you write in SodaCL (the Soda Checks Language). You point it at a data source and a checks file, it runs the checks, and it returns pass/fail/warn/error results. It is designed to drop into orchestration as a scan step in a pipeline.

Q: What is SodaCL?
A: SodaCL (Soda Checks Language) is the human-readable, YAML-based domain language you use to declare Soda checks. You write checks such as `row_count`, `missing_count`, and `duplicate_count` in a checks file, and Soda Core executes them during a scan. Its readability is deliberate so analysts, not just engineers, can review the rules.

Q: What is the basic `soda scan` command form?
A: The form is `soda scan -d <data_source> -c <configuration.yml> <checks.yml>`, where `-d` names the data source defined in the configuration, `-c` points at the configuration file with connection details, and the trailing argument is the checks file. The scan connects, runs every check, and prints results with pass/fail/warn/error statuses. This is the command you wire into an orchestration task.

Q: What does the `-c` flag supply to `soda scan` and why is it kept separate from checks?
A: The `-c` flag supplies the configuration YAML that holds the data-source connection details — host, credentials, schema. It is kept separate from the checks file so secrets and connection config live apart from the human-reviewable rules, letting you commit checks to git while injecting the configuration (often with environment-variable references) at runtime. Mixing them risks leaking credentials into the rule repository.

Q: In SodaCL, how do you scope a set of checks to a particular table?
A: You open a block with `checks for <dataset_name>:` and list the checks beneath it, each as a YAML list item. For example `checks for silver_holdings:` followed by `- row_count:` and `- missing_count(isin):`. The header names the dataset the indented checks apply to.

Q: Write a SodaCL check that fails if `silver_holdings` is empty.
A: `checks for silver_holdings:` then `  - row_count > 0`. The `row_count > 0` form asserts at least one row exists, so a completely empty load fails the scan. This is a minimal volume/completeness guard you would put on every critical table.

Q: What does a SodaCL `missing_count` check measure?
A: `missing_count` counts the rows where the specified column is missing — null, or in the set you define as missing. Written as `missing_count(isin) = 0` it enforces completeness on the `isin` column. You can also use `missing_percent(isin)` to threshold on a proportion rather than an absolute count.

Q: What does a SodaCL `duplicate_count` check enforce, and on which dimension?
A: `duplicate_count` counts duplicate values in a column or combination of columns, enforcing the uniqueness dimension. For example `duplicate_count(fund_id, isin, valuation_date) = 0` asserts that no position key repeats within a day. A non-zero result usually points at a double-load or a failed deduplication.

Q: What four statuses can a Soda check return during a scan?
A: A check returns `pass`, `fail`, `warn`, or `error`. `pass` and `fail` are the rule outcome, `warn` is a softer threshold breach you configured deliberately, and `error` means the check itself could not run — a bad column name, a connection failure, or a SQL error. Distinguishing `fail` from `error` matters because an `error` is a broken check, not bad data.

Q: How do you express both a warn and a fail threshold on a single SodaCL check?
A: You attach `warn:` and `fail:` alert-configuration blocks under the check, each with a `when` condition. For example a `missing_percent(isin)` check can carry `warn: when required missing percent > 5%` and `fail: when required missing percent > 10%`. This lets one rule page softly at 5% and hard-stop at 10% without writing two separate checks.

Q: In SodaCL, what is the practical difference between a `warn` result and a `fail` result?
A: A `warn` flags a condition worth attention that should not stop the pipeline, while a `fail` is a hard breach that should. The difference is purely how you wire the downstream gate to react to each status — Soda itself just reports them. In a WAP design you let warnings notify on-call while failures block the publish.

Q: How does Soda Core get wired into orchestration?
A: You add a scan as a task in your orchestrator (Airflow, Dagster, etc.) that shells out to `soda scan -d ... -c ... checks.yml` or calls the Soda Python API, and you branch the DAG on the scan's exit/result. The orchestrator runs the scan at the right pipeline stage — typically after a load, before publish — and uses the result to decide whether to proceed. That placement is what turns Soda from a manual tool into an enforced gate.

Q: A Soda scan returns exit-code non-zero but you expected a pass — what is the first thing to check?
A: First confirm whether the non-zero came from a `fail` (bad data) or an `error` (broken check), because they need opposite fixes. Read the scan output: an `error` line naming a missing column or SQL problem means the check or schema is wrong, while a `fail` line with observed values means the data genuinely breached the rule. Treating an `error` as if it were dirty data sends you chasing the wrong problem.

Q: What does a dbt test do, mechanically?
A: A dbt test compiles to a SQL query that selects the rows violating an assertion; if the query returns any rows, the test fails. Built-in generic tests include `not_null`, `unique`, `accepted_values`, and `relationships`, applied to columns in the model's YAML. Mechanically it is "find the bad rows; non-empty result = failure," which keeps tests close to the transformation logic.

Q: Why is the choice framed as "dbt tests vs dedicated DQ frameworks — layering, not either/or"?
A: Because the two operate at different layers and complement each other rather than compete: dbt tests live inside the transformation project and catch simple column-level violations as models build, while frameworks like Soda or GX handle richer cross-dataset, distributional, and severity-tiered checks with reporting and routing. Using both gives fast in-pipeline guards plus a deeper, auditable DQ layer. Picking one and dropping the other leaves a gap.

Q: Give a concrete example of a check best left to dbt versus one best handled by Soda/GX.
A: A `not_null` or `unique` on a primary key inside a gold model is best as a dbt test — it runs as the model builds and needs no extra tooling. A check that the NAV-input distribution has not drifted from last week, or that fires a warn at 5% and a fail at 10% with results routed to a catalog, is better in Soda or GX. The simple, build-time, single-column assertions go to dbt; the severity-tiered, distributional, reportable ones go to the dedicated framework.

Q: What advantage do dbt tests have from living inside the dbt project?
A: They are version-controlled alongside the model definitions, run as part of `dbt build`/`dbt test` in the same DAG, and fail fast right where the transformation happens, so a broken model never produces downstream rows. They also need no separate infrastructure beyond dbt itself. The cost is that their expressiveness is limited compared with a purpose-built DQ framework.

Q: What is severity in a DQ context, and what two basic levels does this lesson use?
A: Severity is the declared importance of a rule's failure, deciding what happens when it breaks. The two basic levels here are block-publish (critical/fail) which halts the run, and warn which notifies but lets the run continue. Assigning severity per rule is what separates a noisy alert stream from an enforced gate.

Q: What is "circuit-breaking" in DQ engineering?
A: Circuit-breaking is the mechanism by which a critical check failure stops the pipeline — it breaks the circuit so bad data cannot flow downstream to publication. It is the enforcement teeth behind a severity model: a block-publish rule that fires aborts the run. In a NAV pipeline the circuit-breaker is what prevents a wrong holdings count from ever reaching a client report.

Q: How does severity tie into a WAP (write-audit-publish) gate?
A: In WAP you write data to a staging/branch location, audit it with DQ checks, and only publish if the audit passes. Severity decides the audit verdict: a critical failure fails the audit and the publish step is skipped, while a warning lets the publish proceed but raises an alert. The WAP gate is precisely where circuit-breaking is enforced.

Q: Why publish to a staging area first instead of running DQ checks after publishing?
A: Because checking after publish means bad data has already reached consumers before you detect it — the audit becomes forensic, not preventive. WAP audits the candidate data while it is still isolated, so a critical failure blocks promotion and consumers never see the bad version. The whole point is to gate before exposure, not to apologize after.

Q: In the lesson's NAV scenario, what must a seeded completeness failure demonstrate?
A: It must block publish, appear in the catalog, and page on-call with a link back to the failing expectation. That single end-to-end path proves the severity model, the catalog routing, and the alerting are all wired together, not just that a check can fail. It is the "Done when" acceptance test for the whole DQ build.

Q: Why should warnings page without blocking while criticals halt the run?
A: Because not every quality issue justifies stopping a time-critical NAV publish — some are advisory signals you want a human to see but not act on immediately. Warnings preserve flow while surfacing degradation, and criticals enforce hard correctness. Conflating them either floods on-call with non-blocking noise or, worse, blocks the pipeline on trivia.

Q: What is the danger of making too many rules block-publish?
A: You create a brittle pipeline that halts on minor or noisy issues, eroding trust and tempting operators to disable or bypass the gate entirely. Over-blocking turns the circuit-breaker into an obstacle rather than a control, and a bypassed gate protects nothing. Reserve block-publish for rules where shipping the bad data is genuinely worse than missing the cut-off.

Q: What does "DQ result routing" mean in this lesson?
A: It means pushing each check's pass/fail outcome out of the DQ tool into the wider system — into the data catalog and into the alerting channels. Routing is what makes a result visible and actionable beyond the scan log: a failure shows up against the dataset in the catalog and pages the right people. Without routing, results die in a CI log no one reads.

Q: Why route a DQ failure into the data catalog rather than only to a chat alert?
A: Because the catalog is where consumers and stewards look to judge whether a dataset is trustworthy, so a failure recorded there warns anyone about to use the data, persistently, not just whoever saw the transient alert. The chat alert pages the responder now; the catalog entry informs the next analyst tomorrow. You want both: alert for response, catalog for standing context.

Q: What is the value of a DQ failure that "links back to the failing expectation"?
A: The link lets a responder jump straight from the alert or catalog entry to the exact rule that broke and its observed values, collapsing triage time. Instead of "DQ failed somewhere," they see "the completeness expectation on `isin` failed, 4% null, here is the rule." This traceability is also what you show an auditor to prove the control fired and was investigated.

Q: What does "rules from profiling, not imagination" mean?
A: It means you derive DQ thresholds from the data's actually-observed distributions, cardinalities, and null rates rather than from assumptions about how the data "should" look. You profile first, then encode bounds that reflect reality. Rules invented from imagination encode myths — and produce false alarms when the real data disagrees.

Q: Give an example of an imagined rule that profiling would falsify in fund data.
A: Assuming the `isin` column is always populated and writing a hard `missing_count(isin) = 0` rule, when profiling reveals 2% are vendor placeholders for unlisted or pending instruments. Profiling shows the real null rate, so you instead set a `warn` at the observed level and `fail` only above it. The imagined rule would have paged on every normal load.

Q: How does profiling produce a defensible threshold instead of a guessed one?
A: Profiling gives you the observed distribution — min, max, mean, percentiles, null rate, distinct count — so you set bounds at, say, the historical range plus a margin rather than a number you made up. The threshold is then justifiable: "fail when daily row count drops below the 1st-percentile of the last 90 days." You can point to the evidence behind every number.

Q: Why do false alarms from imagined rules specifically erode the DQ gate's value?
A: Because each false alarm trains responders to ignore the gate, and an ignored gate eventually lets a real failure through unnoticed — the boy-who-cried-wolf failure mode. In a regulated NAV process that is the worst outcome: a control that exists on paper but is muted in practice. Profiling-grounded thresholds keep the signal-to-noise high enough that people still trust a page.

Q: You must add a uniqueness rule to the positions table. What do you profile first?
A: You profile the cardinality of the candidate key — how many distinct combinations of (`fund_id`, `isin`, `valuation_date`) exist versus total rows — and check the observed duplicate rate. If profiling shows legitimate duplicates exist (e.g. lot-level rows), a naive `duplicate_count = 0` would be wrong and you must key the uniqueness rule differently. Profiling tells you whether the key you imagined is actually unique.

Q: Map "the FX rate used for NAV must be from today" to a dimension and a rule type.
A: That is the timeliness dimension, enforced with a freshness/max-age rule on the FX-rates table's timestamp — for example asserting the latest rate's date equals the valuation date. In SodaCL this is a freshness check; in GX it is an Expectation on the max of the date column. It catches the silent failure of pricing today's NAV with yesterday's rates.

Q: Map "every position's `isin` must be a syntactically valid ISIN" to a dimension and rule type.
A: That is the validity dimension, enforced with a regex/format check on the `isin` column — 12 characters, two-letter country prefix, nine alphanumerics, and a check digit. SodaCL supports `invalid_count(isin)` with a `valid format` or regex configuration; GX offers pattern-matching Expectations. It catches malformed identifiers without judging whether the instrument is the right one.

Q: Map "the sum of position values equals the fund's reported net assets" to a dimension.
A: That is the consistency dimension — a cross-aggregate reconciliation asserting two related figures agree. It is typically a custom SQL-backed check (SodaCL user-defined check or a dbt test with a custom query) comparing the summed market values to the NAV's net-asset figure within a tolerance. Consistency rules like this are where dedicated frameworks earn their place over simple dbt generics.

Q: Map "no `lei` appears twice for the same fund share class" to a dimension and rule.
A: That is the uniqueness dimension, enforced with a `duplicate_count` over the relevant key. You would write `duplicate_count(share_class_id, lei) = 0` so a single share class never carries two LEIs. A breach usually means a master-data merge created a conflicting identifier.

Q: In a GX 1.x script, what is the typical sequence to validate a table end-to-end?
A: Get the `DataContext`, define or retrieve a Data Source and a Batch Definition for the table, create an Expectation Suite and add Expectations, build a Validation Definition linking the Batch Definition to the suite, then create and run a Checkpoint with an `action_list`. The Checkpoint run produces Validation Results and, via `UpdateDataDocsAction`, refreshes Data Docs. That chain — context, batch, suite, validation definition, checkpoint — is the canonical 1.x flow.

Q: Why does GX separate the Expectation Suite from the Batch Definition rather than binding rules to one table inline?
A: Separating them lets the same suite validate different batches (today's partition, a backfill, another environment) and lets one batch be checked by different suites. The Validation Definition is the explicit join of a specific suite to a specific batch, so reuse stays clean. Binding rules inline to a single table would force duplication every time the data or rule set varied.

Q: What is the difference between a GX Expectation failing and a GX run erroring?
A: An Expectation failing means the data did not satisfy a valid assertion — bad data, surfaced in the Validation Result with observed values. A run erroring means GX could not evaluate something — a missing column, a type mismatch, a connection problem — so there is no meaningful pass/fail for that Expectation. As with Soda, an error is a broken check to fix, not dirty data to clean.

Q: Your `soda scan` reports `error` on a check referencing column `nav_per_share`. First thing to check?
A: First check that the column actually exists with that exact name and case in the scanned dataset, because an `error` (not `fail`) on a column-referencing check most often means the column is misnamed, was renamed upstream, or lives in a different schema. Confirm against the table's current schema before suspecting the data. A schema drift upstream is a frequent cause.

Q: A Checkpoint runs and Data Docs do not update. What is the likely cause?
A: The Checkpoint's `action_list` most likely lacks an `UpdateDataDocsAction`, so validation ran but nothing rebuilt the docs sites. Add `UpdateDataDocsAction(name="update_all_data_docs")` to the `action_list`. If the action is present, next check that the Data Docs site is configured and writable in the `DataContext`.

Q: How would you wire a Soda critical failure to actually block a publish in an orchestrator?
A: Run the `soda scan` as a task before the publish task and make the publish depend on the scan task's success, so a `fail` (non-zero result) stops the DAG before publish executes. In Airflow that is an upstream dependency plus letting the scan task fail the run; in Dagster an asset check with `blocking=True`. The dependency edge is the circuit-breaker.

Q: How would you let a Soda warning page without blocking in the same orchestrator?
A: Configure the warn-level checks so a `warn` result does not fail the scan task, and route warn results to a notification (the catalog/alert channel) via Soda's result handling or a downstream branch. The publish proceeds because the task did not fail; the alert fires because you routed the warn status. The key is that only `fail` stops the DAG while `warn` only notifies.

Q: Why is profiling described as the evidence-gathering step before writing DQ rules?
A: Because you cannot set a defensible threshold without first knowing the data's real shape — its ranges, null rates, cardinalities, and surprises. Profiling produces that evidence, and the rules then encode observed reality. Skipping it means your rules rest on assumptions you never tested.

Q: A completeness rule on `isin` fires on every load though analysts say the data is fine. Likely cause and fix?
A: The likely cause is an imagined hard rule (`missing_count(isin) = 0`) that ignores a legitimate, profiled null rate such as vendor placeholders for unlisted instruments. The fix is to profile the actual null rate and set a `warn` at the normal level with a `fail` only above it, so genuine spikes still page. This is "rules from profiling, not imagination" in action.

Q: What is the risk of writing DQ rules without severity, treating every check as equal?
A: Either everything blocks — making the pipeline fragile and prone to bypass — or nothing blocks and the checks become advisory noise that protect nothing at publish time. Without a severity axis you cannot express "this must stop the run, that should just warn." Severity is what turns a flat list of checks into an enforceable gate.

Q: Why might you run a complex distributional check in Great Expectations even if Soda is your primary tool?
A: Because GX's Expectation library and its statistical/distributional Expectations can express richer, observed-distribution checks than a quick SodaCL threshold, and the lesson explicitly suggests reaching for GX for complex distributional checks. You keep Soda for the bulk of simple checks wired into the WAP gate and lean on GX where you need distribution-aware assertions plus rich Data Docs. The two layer rather than compete.

Q: What does it mean that GX provides "rule-and-report" while Soda is "scan-and-result"?
A: GX emphasizes declared Expectations plus rich human-readable Data Docs reporting, so the report artifact is first-class. Soda emphasizes a lightweight CLI scan that returns statuses you wire into orchestration, with the result handed off to your own routing. Both validate data; they differ in how much of the report layer they own versus delegate.

Q: How do dbt tests, Soda, and GX form a layered DQ stack in this lesson?
A: dbt tests run in-pipeline as models build, catching simple column violations fast; Soda scans wire severity-tiered checks into the WAP gate for block/warn enforcement; GX handles complex distributional checks with rich Data Docs. Each layer catches a class the others handle less well, and together they cover build-time, gate-time, and report-time quality. The lesson's stance is layering, not picking one.

Q: Why is it dangerous to encode a NAV row-count rule as a fixed constant like "exactly 1,250 rows"?
A: Because the real holdings count legitimately varies day to day as positions open and close, so a fixed constant will false-fail constantly. Profiling the historical row-count distribution lets you set a relative bound — e.g. fail if today's count drops more than a profiled percentage below the trailing median. The rule should track observed volatility, not a frozen guess.

Q: What is a "seeded" or synthetic failure and why deliberately trigger one?
A: A seeded failure is an intentional fault you inject — e.g. nulling some `isin` values — to prove the DQ gate detects it, blocks publish, records it in the catalog, and pages. You trigger it deliberately because an untested control is an assumed control; only an observed end-to-end failure proves the wiring works. It is the acceptance test for the whole DQ build.

Q: In SodaCL, how would you express "warn if more than 5% of `isin` are missing, fail if more than 10%"?
A: Under `checks for silver_holdings:` write a `missing_percent(isin)` check carrying both alert configurations: `warn: when required missing percent > 5%` and `fail: when required missing percent > 10%`. Soda evaluates the column's missing percentage and assigns `warn` or `fail` accordingly. This single check encodes a two-tier severity for one completeness rule.

Q: What is the difference between `missing_count` and `missing_percent` in SodaCL, and when prefer each?
A: `missing_count` thresholds on an absolute number of missing values, while `missing_percent` thresholds on the proportion missing relative to total rows. Prefer `missing_percent` when row volume varies day to day, since a fixed count threshold becomes meaningless at different table sizes. For a small, fixed-size reference table an absolute `missing_count = 0` may be clearer.

Q: How does a data catalog become part of the DQ control loop rather than just a documentation site?
A: When DQ results are routed into it, the catalog records each dataset's latest check outcomes, so it reflects live trustworthiness, and consumers can see a dataset is currently failing before they query it. Combined with links to the failing expectation, it becomes the standing record of quality state. That turns a passive glossary into an active part of the gate.

Q: Why is "justify every rule's dimension and severity in writing" a Done-when criterion?
A: Because an architect must be able to defend, in an audit or design review, why each check exists, which quality dimension it protects, and why it blocks or merely warns. Written justification forces deliberate rule design instead of accreted, unexplained checks. It is also the artifact you hand a regulator asking how NAV-input quality is controlled.

Q: What failure does a freshness/timeliness check on the holdings table specifically prevent in a NAV pipeline?
A: It prevents producing a NAV from a stale load — yesterday's holdings or prices silently reused because today's load stalled but the old data is still present and valid. No completeness or validity check catches this, since the stale rows are full and well-formed. Only an age/timeliness assertion against the valuation date exposes it.

Q: Why can a column be 100% complete and 100% valid yet still be inaccurate?
A: Because completeness only checks presence and validity only checks format, while accuracy is about real-world correctness. A price that is non-null and within a plausible range can still be the wrong price for that security. This gap is why accuracy usually requires comparison against an authoritative source rather than a self-contained rule.

Q: When wiring Soda into orchestration, why run the scan after load but before publish specifically?
A: Because that placement is the audit step of write-audit-publish: the data has been written to staging and is available to check, but has not yet been promoted to consumers. A critical failure there blocks promotion so bad data never publishes, while passing data proceeds. Scanning earlier has nothing to check; scanning after publish is too late to gate.

Q: A dbt `unique` test on a gold model passes but Soda's `duplicate_count` on the same table fails. What is a likely explanation?
A: They are probably keyed differently — the dbt `unique` test is on a single surrogate column that is genuinely unique, while the Soda `duplicate_count` is on a business key combination that is not. Reconcile by deciding which key truly defines a row's identity for the business and aligning both checks to it. Disagreements like this often reveal an ambiguous grain definition.

Q: Why does the lesson treat severity as tied to the WAP gate rather than a property of the check alone?
A: Because severity only has meaning at the point where it changes behaviour — the gate that decides publish-or-block. A check labelled "critical" matters only because the WAP audit fails and skips publish when it fires. The severity model and the gate's branching logic are two halves of one mechanism.

Q: What does a GX `expect_column_values_to_be_between` enforce, and which dimensions can it serve?
A: It asserts a column's values fall within a min/max range, failing rows outside it. Depending on intent it can serve validity (values within an allowed domain) or proxy accuracy/reasonableness (a NAV per share between sane bounds). The same Expectation type maps to different dimensions according to what business rule it encodes, which is why dimension labelling is a design decision, not automatic.

Q: How do you keep DQ rules from drifting out of sync with the data they protect?
A: Re-profile periodically and update thresholds against fresh distributions, ideally on a schedule, so rules track the data's evolving shape rather than freezing at first authorship. Pair this with monitoring how often each rule fires — a rule that never fires or always warns is a candidate to retune. Static rules over a moving dataset slowly become either blind or noisy.

Q: Why is it useful that SodaCL is human-readable YAML rather than code?
A: Because data stewards, analysts, and auditors who are not Python engineers can read and review the checks, raising the chance that domain experts catch a wrong threshold or a missing rule. Readability also makes the checks a credible audit artifact on their own. The trade-off is less programmatic flexibility than a code-based framework like GX.

Q: What is the role of `action_list` in a GX Checkpoint, with two example actions?
A: The `action_list` defines what happens with the Validation Results after a Checkpoint validates — the post-validation side effects. Examples include `UpdateDataDocsAction` to refresh the HTML report and notification actions that alert a channel on failure. It is the hook through which GX routes results to docs and alerts.

Q: Why might you prefer relative thresholds (vs prior period) over absolute ones for NAV-pipeline volume checks?
A: Because absolute thresholds assume a stable size the data does not have — fund holdings counts and trade volumes legitimately fluctuate, so a relative check ("within X% of the trailing median") flexes with normal variation while still catching abnormal drops. Profiling supplies the baseline. Absolute thresholds either false-fail on normal growth or miss real shrinkage.

Q: What is the consequence of a DQ failure that pages but provides no link to the failing rule?
A: The responder wastes triage time hunting for which check fired and on what data, slowing resolution and increasing the chance of a wrong fix — exactly when a NAV cut-off is tight. The link-back collapses "something failed" into "this completeness expectation on `isin` failed, here are the rows." Without it, the alert is an interruption rather than a head start.

Q: How does profiling feed PII discovery, and why mention it in a DQ lesson?
A: Profiling surfaces candidate sensitive columns — value patterns that look like names, account numbers, or other personal data — which feeds downstream PII-classification work. It is mentioned because the same scan that gathers DQ evidence doubles as a discovery pass for governance, so you do not run the data twice. In a regulated fund context that overlap matters for both DQ and data-protection controls.

Q: Distinguish data quality (DQ rules) from data observability at a high level.
A: DQ rules are explicit, hand-written assertions about known correctness criteria ("isin must not be null"), enforced at a gate. Observability is broader monitoring of pipeline health — freshness, volume, schema drift, distribution — often anomaly-detected rather than rule-defined, catching the unknown-unknowns. This lesson is the rule-based DQ side; observability complements it for silent failures no one wrote a rule for.

Q: Why does a consistency check between holdings and the instrument master often need a custom check rather than a built-in?
A: Because consistency is relational — it compares values across two datasets or aggregates — and built-in single-column checks like `not_null` or `missing_count` cannot express a join-and-compare. You write a user-defined SodaCL check or a dbt test with custom SQL that performs the reconciliation. The cross-dataset nature is exactly what pushes such checks into the dedicated-framework layer.

Q: What does it mean to "tie criticals block-publish vs warn to your WAP gates"?
A: It means mapping each check's severity onto the gate's branching: critical (fail) checks make the WAP audit fail and skip publish, while warn checks let publish proceed but raise an alert. The severities are not abstract labels — they are the conditions the gate evaluates to decide promotion. That binding is what makes severity operational.

Q: After profiling reveals five surprising findings in the holdings table, what does the lesson have you do with them?
A: Turn each surprising finding — an unexpected null rate, a strange cardinality, an out-of-range value — into a Soda rule with the correct dimension and severity. The surprises become evidence-grounded checks rather than ignored anomalies. This is the direct loop from profiling to rule authorship the lesson teaches.

Q: Why label a profiling-derived check's dimension even though profiling generated it automatically?
A: Because the dimension label is what lets you defend and organize the rule later — "this is a completeness check at warn severity because profiling showed a 2% baseline null rate." Profiling tells you the number; the dimension tells you what quality property the rule protects and how to justify it. The two together make the check auditable.

Q: A scenario: criticals halt the run but warnings are silently swallowed. What is wrong and what should happen?
A: The warn-routing is broken — warnings are meant to page or be recorded without blocking, not to vanish. You should route warn results to the alert channel and the catalog so on-call sees them while the publish continues. A swallowed warning is a missed early signal; the design goal is "warn notifies, critical blocks," not "warn does nothing."

Q: Why is it better to derive an upper bound for a return-rate check from the 99th percentile of history than from a textbook maximum?
A: Because the textbook or theoretical maximum rarely matches the instrument's actual behaviour, so it either lets through anomalies (bound too loose) or false-fails on normal extremes (bound too tight). The 99th percentile of observed history reflects what this fund actually does, plus you can defend the number with the profile. It is "rules from profiling, not imagination" applied to a reasonableness bound.

Q: In GX, why is publishing Data Docs to a shared location valuable in a regulated context?
A: Because it gives auditors, stewards, and analysts a persistent, human-readable record of which Expectations ran, passed, or failed, on which data, without anyone rerunning code. That standing evidence is exactly what a regulator expects when asking how data quality is controlled and demonstrated. The static-HTML form means the record survives independently of the pipeline.

Q: What is the single most important reason to distinguish `error` from `fail` when operating a DQ gate?
A: Because they demand opposite responses: a `fail` means clean or reject bad data, while an `error` means fix a broken check or schema before the gate can be trusted at all. Treating an `error` as a data problem leaves the actual rule non-functional, so genuinely bad data may pass undetected. The distinction protects the gate's integrity, not just the data.

Q: How would you store Soda data-source credentials so checks stay committable to git?
A: Keep connection details in the `-c` configuration file and reference secrets via environment variables rather than hard-coding them, so the configuration can inject credentials at runtime while the checks file stays secret-free. Soda supports `${VAR}`-style environment-variable substitution in the configuration YAML. Commit only the checks file and a template configuration; the real credentials come from the environment or a secrets manager.

Q: For an EMT/EPT file feed into the silver layer, which DQ dimensions would you prioritise and why?
A: Validity and completeness lead, because EMT/EPT (European MiFID/PRIIPs Template) files have a defined field structure where malformed or missing mandatory fields break downstream regulatory reporting. Timeliness matters too, since these templates feed periodic disclosures with deadlines, and a stale file misses a filing window. You profile a few real deliveries first to learn which "optional" fields are actually always populated before hardening any completeness rule.

Q: Why might the same Soda check belong at `warn` in one environment and `fail` in another?
A: Because severity reflects the business cost of shipping the data in that context: in a development or backfill run a completeness dip might only warn so work continues, while in the production NAV path the same dip must `fail` and block publish. Severity is environment- and stage-aware, not an intrinsic property of the rule. You parameterise the threshold/severity per environment rather than duplicating the check.

Q: Pulling it together: describe the end-to-end path a critical NAV completeness failure should travel.
A: A load arrives in staging, the WAP audit runs the Soda/GX completeness check, the critical rule fails, the gate breaks the circuit and skips publish so no client sees the bad data, the failure is routed into the catalog against the dataset and pages on-call, and the alert links straight to the failing expectation with its observed values for fast triage. That path — write, audit, block, route, page, link — is the lesson's whole DQ-engineering thesis, with severity and profiling underpinning every threshold along the way.
