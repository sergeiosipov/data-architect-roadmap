# Phase 4 · 4.2.1 Stateful stream processing (Flink) — 150 self-test questions

Q: What does it mean for a stream processor to be "stateful," and why is Flink the reference engine for it?
A: Stateful means the engine remembers information across events — running counts, window contents, last-seen values — rather than treating each record independently. Flink is the reference because it bakes durable, fault-tolerant keyed state and exactly-once semantics into the runtime, so a windowed aggregation survives crashes; many managed services (AWS Managed Service for Apache Flink, Azure Stream Analytics internals, Confluent Flink) run Flink or a Flink-like engine underneath.

Q: Why does stateful streaming matter specifically for a regulated NAV pipeline?
A: Real-time NAV estimates and exposure monitoring accumulate state — positions per share class, running flow totals — that must be identical after a restart as before it. If checkpointing or state evolution is wrong, a job restart can produce different numbers than the pre-crash run, which is unacceptable when those numbers feed a regulated NAV that auditors will reconcile.

Q: What is the Flink dataflow graph?
A: A Flink job is expressed as a directed graph of operators (sources, transformations, sinks) through which records flow; the API graph is the logical "JobGraph," and the runtime expands it into an "ExecutionGraph" of parallel subtasks. Edges between operators carry records, and an edge may be a forward, a rebalance, or a keyed (hash) partition depending on the transformation.

Q: What is a keyed stream in Flink and how do you create one?
A: A keyed stream partitions records by a key (for example ISIN or share-class id) so that all records with the same key are routed to the same operator subtask, where per-key state lives. You create one with `keyBy(...)`, which hash-partitions the stream; only on a keyed stream can you use keyed state (`ValueState`, `ListState`, `MapState`) and keyed timers.

Q: Why must state be accessed on a keyed stream rather than an arbitrary operator?
A: Keyed state is scoped to the current key, and Flink guarantees all records for a key land on one subtask, so the state for that key is local and consistent. Without `keyBy`, there is no key context, so `ValueState` and friends are unavailable; operator (non-keyed) state exists but follows different scoping and redistribution rules on rescale.

Q: What is operator parallelism in Flink and what controls it?
A: Parallelism is the number of parallel subtasks an operator runs as, each handling a slice of the stream; it is set per-operator with `setParallelism(...)`, per-job with the default parallelism, or by the cluster default. For a keyed operator, parallelism bounds how many key-groups can be processed concurrently, and changing it requires redistributing keyed state across subtasks.

Q: What is a key group in Flink and why does it exist?
A: A key group is the atomic unit of keyed-state assignment: every key hashes into one of `maxParallelism` key groups, and key groups are distributed across subtasks. This indirection lets Flink rescale a job (change parallelism) by reassigning whole key groups without rehashing every individual key, which is why `maxParallelism` is fixed at job start and caps future rescaling.

Q: What is the difference between keyed state and operator state?
A: Keyed state is partitioned by key and only available on keyed streams (`ValueState`, `ListState`, `MapState`, `ReducingState`, `AggregatingState`); operator state is bound to a parallel operator instance regardless of key, used by things like the Kafka source to track offsets per subtask. On rescale, keyed state moves by key group, while operator state is redistributed via list-style or union-style schemes the operator defines.

Q: Name the common keyed state primitives Flink exposes.
A: `ValueState<T>` holds a single value per key, `ListState<T>` an appendable list, `MapState<K,V>` a per-key map, `ReducingState<T>` and `AggregatingState<IN,OUT>` maintain an incrementally combined value. Each is obtained from the `RuntimeContext` via a corresponding state descriptor (for example `ValueStateDescriptor`) inside `open()` of a `RichFunction`.

Q: What is a state backend in Flink?
A: A state backend determines how working (in-flight) keyed state is stored and accessed while a job runs — on the JVM heap as objects, or in an embedded on-disk store. It is a separate concern from checkpoint storage (where snapshots are durably written), a distinction Flink made explicit when it split the old combined backends.

Q: What are the two main Flink state backends today and how do they differ?
A: `HashMapStateBackend` keeps state as Java objects on the TaskManager JVM heap — fastest access but bounded by heap and subject to GC pressure. `EmbeddedRocksDBStateBackend` stores state in an embedded RocksDB on local disk, serializing keys/values, scaling working state to terabytes at the cost of per-access serialization and disk I/O latency.

Q: Which state backend is the default if you configure nothing?
A: `HashMapStateBackend` is the default. It is appropriate for development and for jobs whose per-slot state stays in the hundreds of megabytes to single-digit gigabytes range; beyond roughly 10 GB per slot you should move to RocksDB to avoid GC stalls and out-of-memory failures.

Q: When would you pick `EmbeddedRocksDBStateBackend` over `HashMapStateBackend` for a fund job?
A: Choose RocksDB when keyed state is large — for example one running position and a window buffer per ISIN across tens of thousands of instruments, or long allowed-lateness windows that retain many in-flight aggregations. RocksDB also uniquely supports incremental checkpoints, so for large state it dramatically cuts checkpoint size and duration versus a full heap snapshot.

Q: What is the size/latency tradeoff between heap and RocksDB state backends?
A: Heap (`HashMapStateBackend`) gives the lowest per-record latency because state is live Java objects, but capacity is limited by heap and large state causes long GC pauses. RocksDB trades latency — every access serializes/deserializes and may hit disk — for near-unbounded capacity and incremental checkpointing, so you accept slower point lookups to hold terabytes safely.

Q: Why did Flink separate "state backend" from "checkpoint storage"?
A: The old API conflated where live state lives with where snapshots are persisted, which confused configuration. The split lets you pick, independently, a working-state store (`HashMapStateBackend` or `EmbeddedRocksDBStateBackend`) and a checkpoint storage target (`JobManagerCheckpointStorage` for tiny state, or `FileSystemCheckpointStorage` pointing at S3/MinIO/HDFS) so you can run heap state but still snapshot durably to object storage.

Q: What does a checkpoint achieve in Flink?
A: A checkpoint is a periodic, consistent, asynchronous snapshot of all operator state plus the source positions (offsets) that produced it, written to durable checkpoint storage. On failure Flink restarts the whole job from the last completed checkpoint, restoring state and rewinding sources, which is the foundation of its exactly-once-within-Flink guarantee.

Q: Explain the checkpoint barrier mechanism from memory.
A: The `JobManager`'s checkpoint coordinator periodically injects numbered barriers into the source streams. A barrier flows through the dataflow with the records; when an operator has received the barrier for checkpoint N on all its input channels, it snapshots its state to durable storage and forwards the barrier downstream. When every operator and sink has acknowledged barrier N, the checkpoint is complete and becomes the recovery point.

Q: What is checkpoint barrier alignment?
A: When an operator has multiple input channels, alignment means it stops processing records from a channel once that channel's barrier N arrives, buffering them, until barrier N has arrived on all channels — only then does it snapshot and emit the barrier. This guarantees the snapshot reflects exactly the records before barrier N on every input, giving a clean exactly-once cut, but it stalls fast channels while waiting for slow ones.

Q: What is an aligned checkpoint and what is its cost under backpressure?
A: An aligned checkpoint performs the alignment described above, producing a snapshot with no in-flight data captured. Under backpressure a barrier travels slowly because it is stuck behind buffered records, and alignment makes operators wait, so checkpoint duration balloons and can time out exactly when the system is already struggling — the problem unaligned checkpoints address.

Q: What is an unaligned checkpoint and what does it capture that aligned does not?
A: An unaligned checkpoint lets the barrier overtake buffered in-flight records, and it snapshots that in-flight buffer data as part of the checkpoint state. This decouples checkpoint duration from throughput/backpressure because the barrier no longer waits behind queued records, at the cost of larger checkpoints (the buffered data is now persisted).

Q: When should you prefer unaligned over aligned checkpoints?
A: Prefer unaligned when the job suffers backpressure and aligned checkpoints are timing out or growing unacceptably long, since unaligned keeps checkpoint time roughly constant regardless of load. Prefer aligned (the default) when the job is healthy and you want the smallest checkpoints, because unaligned persists in-flight buffers and increases I/O and state size.

Q: What is the adaptive (timeout-based) unaligned checkpoint behaviour?
A: Flink can start a checkpoint as aligned and automatically fall back to unaligned if the alignment phase exceeds a configured timeout (`execution.checkpointing.aligned-checkpoint-timeout`). This gives you small aligned checkpoints in the common healthy case and the backpressure-resilience of unaligned only when alignment is actually slow, rather than paying the unaligned cost on every checkpoint.

Q: How do checkpoints enable exactly-once within Flink?
A: At a checkpoint, every operator's state and the source offsets that produced it are captured at the same logical cut (via barriers). On recovery, Flink restores that state and rewinds sources to those exact offsets, so every record is reflected in state exactly once — no record is double-counted or lost relative to the snapshot, which is exactly-once state semantics inside the engine.

Q: What does "exactly-once within Flink" not by itself guarantee?
A: It guarantees state consistency inside the engine but not end-to-end exactly-once to external sinks, because on replay Flink may re-emit output records. End-to-end exactly-once additionally requires either an idempotent sink or a transactional/two-phase-commit sink that only makes output visible when the corresponding checkpoint completes.

Q: What is at-least-once checkpointing mode and when is it acceptable?
A: In at-least-once mode Flink skips barrier alignment, so an operator processes records past a barrier before all barriers arrive, meaning some records may be counted twice after recovery. It is acceptable when downstream is idempotent or small duplication is tolerable, and it lowers latency and checkpoint cost — but for a NAV total you almost always want exactly-once.

Q: What is incremental checkpointing and which backend supports it?
A: Incremental checkpointing uploads only the state changes (new RocksDB SST files) since the last checkpoint instead of the full state, drastically reducing checkpoint size and time for large state. Only `EmbeddedRocksDBStateBackend` supports it, because it relies on RocksDB's immutable SST file structure; `HashMapStateBackend` always writes a full snapshot.

Q: What is a savepoint and how does it differ from a checkpoint?
A: A savepoint is a manually triggered, self-contained, durable snapshot of job state intended for operational use — upgrades, migrations, A/B redeploys — whereas checkpoints are automatic and owned by the running job for failure recovery. Savepoints are not auto-deleted, are designed to be portable and restorable into a (possibly modified) job, and you trigger one with `flink savepoint <jobId>` or stop-with-savepoint.

Q: What are the CANONICAL and NATIVE savepoint formats?
A: `CANONICAL` is a backend-independent uniform format that lets you restore into a different state backend (for example switch heap to RocksDB), at the cost of slower creation. `NATIVE` writes the backend's own format (RocksDB SST files), which is faster to take and restore but ties the savepoint to that backend; you choose with `--type canonical` or `--type native` when triggering.

Q: How do you perform a savepoint-based upgrade without losing state?
A: Take a savepoint (ideally stop-with-savepoint to drain cleanly: `flink stop --savepointPath <dir> <jobId>`), deploy the new job jar, then start the new job from that savepoint with `flink run -s <savepointPath>`. Flink remaps state to operators by their stable `uid`s, so state survives the code change as long as operator UIDs and state schemas remain compatible.

Q: Why must you assign stable `uid`s to operators for upgrades?
A: On restore Flink matches saved state to operators by their UID, not by position in the graph; if you let Flink auto-generate UIDs, any change to the job graph reshuffles them and state can no longer be matched, causing restore to fail or silently drop state. Calling `.uid("flow-total-agg")` on each stateful operator pins identity so savepoint restores stay stable across code changes.

Q: What is state schema evolution in Flink?
A: It is changing the type/shape of state (adding or removing fields, widening types) and still restoring old snapshots into the new job. Flink supports it for state serialized with Avro or POJO serializers under compatible changes; the serializer's compatibility check decides whether old state can be read, and incompatible changes require a migration (for example an upcaster) rather than a direct restore.

Q: What is an event-time operator in Flink?
A: An event-time operator triggers logic based on timestamps carried in the data and the watermark, not wall-clock time — for example a window that fires when the watermark passes the window end. This makes results deterministic and replayable, which is why event time is the regulated-finance default: re-running the same input yields the same windows regardless of when processing happens.

Q: What is a watermark in Flink?
A: A watermark is a marker flowing in the stream asserting "no more events with timestamp ≤ T are expected," letting event-time operators decide when a window is complete enough to fire. It is the engine's estimate of event-time progress; choosing it trades completeness (waiting longer catches stragglers) against latency (firing sooner emits results faster).

Q: What is a `WatermarkStrategy` and how do you configure a bounded-out-of-orderness watermark?
A: A `WatermarkStrategy` tells Flink how to extract event timestamps and how to generate watermarks. For heuristic out-of-order streams you use `WatermarkStrategy.forBoundedOutOfOrderness(Duration.ofSeconds(n))` plus `.withTimestampAssigner(...)`, which emits a watermark of (max-seen-timestamp − n), tolerating up to n of lateness before an event is considered late.

Q: What is the difference between a perfect and a heuristic watermark?
A: A perfect watermark guarantees no event later than T will ever arrive (possible only when the source has total event-time knowledge, like a sorted file). A heuristic watermark — what real streams use — estimates progress and may be wrong, so some events arrive after the watermark has passed their timestamp; those are "late" and must be handled with allowed lateness or side outputs.

Q: What is allowed lateness in a Flink window and what does it cost?
A: Allowed lateness (`window(...).allowedLateness(Duration.ofMinutes(m))`) keeps a window's state alive for m after the watermark passes the window end, so late events that arrive within m re-fire the window and update (restate) the result. The cost is latency-to-finalize and memory: window state is retained longer, and downstream sees corrected re-emissions, so the consumer must handle restatements.

Q: What is a side output for late data and why use it?
A: A side output is a separate stream tagged with an `OutputTag` to which a windowed operator routes events that arrive even after allowed lateness expires (truly late). You use it so stragglers are never silently dropped — they can be logged, alerted, or reconciled in batch — which matters when a late trade would otherwise quietly understate a fund-flow total.

Q: How would you compute per-share-class windowed flow totals with heuristic watermarks in Flink?
A: Key the order stream by share class with `keyBy`, assign event-time timestamps and a `forBoundedOutOfOrderness` watermark strategy, apply a tumbling event-time window (for example `TumblingEventTimeWindows.of(...)`), and aggregate net flow with an `AggregateFunction`. Add `allowedLateness` plus a late-data side output so stragglers restate the total rather than being lost.

Q: What is a Flink timer and when is it used?
A: A timer is a callback registered in a `KeyedProcessFunction` to fire at a specific event-time or processing-time instant, invoking `onTimer(...)`. You use timers for custom time-based logic — emitting a result when no event arrived for a key within a timeout, expiring stale per-key state, or implementing custom windowing — and event-time timers fire when the watermark passes their timestamp.

Q: What is a `ProcessFunction` / `KeyedProcessFunction` and why is it the low-level escape hatch?
A: It is the most general stateful operator, giving direct access to keyed state, event-time and processing-time timers, the current watermark, and side outputs via its `Context`. It is the escape hatch when windows and built-in operators cannot express the logic — for example bespoke session logic, custom lateness handling, or emitting on a timeout — at the price of writing the state and timer bookkeeping yourself.

Q: What does the Flink DataStream window API let you express?
A: It expresses keyed (or non-keyed) windowed computations: pick a window assigner (tumbling, sliding, session, global), optionally a trigger and evictor, then apply `reduce`, `aggregate`, or `process` to compute per-window results. It is the declarative path for event-time aggregations where you do not need the full manual control of a `ProcessFunction`.

Q: What is Flink SQL / the Table API and when do you reach for it?
A: Flink SQL and the Table API are the declarative layer where you express windowed aggregations, joins, and filters as SQL or fluent Table operations, and the planner compiles them to the same DataStream runtime. Reach for them when the logic is relational and standard (windowed `GROUP BY`, temporal joins) — they are far less code than DataStream and easier for analysts to read and audit.

Q: How are windowed aggregations expressed in Flink SQL?
A: Modern Flink SQL uses windowing table-valued functions such as `TUMBLE`, `HOP` (sliding), and `CUMULATE`, then groups by the `window_start`/`window_end` columns: for example `SELECT window_start, share_class, SUM(amount) FROM TABLE(TUMBLE(TABLE orders, DESCRIPTOR(event_time), INTERVAL '1' HOUR)) GROUP BY window_start, window_end, share_class`. Watermarks are declared on the source table's event-time column.

Q: How do you declare event time and watermarks for a Flink SQL source table?
A: In the `CREATE TABLE` DDL you mark a column as the event-time attribute and add a `WATERMARK FOR event_time AS event_time - INTERVAL '5' SECOND` clause, which generates a bounded-out-of-orderness watermark. The column must be a `TIMESTAMP(3)` or `TIMESTAMP_LTZ(3)`, and that watermark drives all windowing TVFs over the table.

Q: What is the Kafka source/sink EOS wiring in Flink?
A: The Kafka source (`KafkaSource`) tracks consumer offsets in Flink's checkpointed state (not via Kafka auto-commit) so on recovery it rewinds to the checkpointed offset. The `KafkaSink` with `DeliveryGuarantee.EXACTLY_ONCE` writes records inside a Kafka transaction per checkpoint cycle, pre-committing at the barrier and committing only when the checkpoint completes — a two-phase commit.

Q: When configuring `KafkaSink` for exactly-once, what extra setting is mandatory and why?
A: You must call `setTransactionalIdPrefix(...)`, because exactly-once uses Kafka's transactional producer and Flink needs a stable, unique transactional-id prefix per sink to fence zombie producers and resume transactions correctly after recovery. Omitting it (while requesting `EXACTLY_ONCE`) is a configuration error; the prefix must be unique across all applications writing to that Kafka cluster.

Q: What is the two-phase-commit (2PC) sink and how does Flink coordinate it with checkpoints?
A: A 2PC sink splits output into a pre-commit and a commit phase tied to checkpointing: as records process, the sink writes them into an external transaction (pre-commit), at the checkpoint barrier it persists enough to resume the transaction, and only when the checkpoint completes (notified by `notifyCheckpointComplete`) does it commit. Flink abstracts this in `TwoPhaseCommitSinkFunction` (and the newer unified-sink `TwoPhaseCommittingSink`), giving end-to-end exactly-once into transactional sinks.

Q: Why does committing a 2PC sink transaction on checkpoint completion (not on the barrier) give exactly-once?
A: Because the checkpoint is the recovery point: if the job fails after pre-commit but before checkpoint completion, recovery restarts from the prior checkpoint and the uncommitted transaction is aborted, so no output leaks. Output becomes externally visible only for checkpoints that actually completed, so each record is committed exactly once and replayed records never produce a second visible commit.

Q: How do you write Flink results exactly-once into Iceberg?
A: Use the Flink Iceberg sink (`FlinkSink` with a `TableLoader`), which buffers writes and commits new data files to the Iceberg table as an atomic snapshot tied to Flink checkpoints. Because the Iceberg commit is atomic and aligned to checkpoint completion, a job restart re-creates the same files without double-committing, giving exactly-once results in the lakehouse table on MinIO.

Q: A restored Flink job must produce byte-identical results to the pre-crash run — what makes that achievable?
A: Determinism plus exactly-once recovery: event-time logic makes results depend on data and watermarks rather than wall clock, checkpoints restore state with sources rewound to the matching offsets, and a 2PC/idempotent sink ensures no duplicate or missing output. As long as the computation is deterministic (no random ordering, no processing-time dependence), replaying from the checkpoint yields identical output.

Q: What can make a Flink job non-deterministic and thus produce different results on restore?
A: Reliance on processing time, wall-clock timers, random or `now()` values, non-deterministic UDFs, or non-deterministic input ordering across keys/partitions all break determinism. For reproducible NAV-feeding results you use event time, deterministic aggregations, stable partitioning, and avoid embedding the current time into outputs unless it is itself part of the checkpointed, replayable state.

Q: How does Spark Structured Streaming compare to Flink at a high level?
A: Spark Structured Streaming defaults to a micro-batch model — it discretizes the stream into small batches executed by the Spark SQL engine — whereas Flink processes event-at-a-time natively. Spark reuses the batch engine and DataFrame API (familiar, strong ecosystem) but imposes a per-batch latency floor; Flink offers lower latency and richer event-time/state primitives at the cost of a separate operational model.

Q: What is the micro-batch latency floor and why does event-at-a-time avoid it?
A: In micro-batch, results cannot appear faster than the trigger interval because records are buffered into a batch before processing, so end-to-end latency is at least one batch (often ~100 ms and up). Flink's event-at-a-time pipeline processes and emits each record as it arrives, so it has no inherent batch boundary and can reach lower latencies.

Q: What is Spark's continuous processing mode and its tradeoff?
A: Continuous processing is an experimental Spark mode targeting ~1 ms end-to-end latency by running long-lived tasks instead of micro-batches. The tradeoff is it provides only at-least-once fault tolerance (versus exactly-once for micro-batch) and supports a limited set of operations, so for exactly-once stateful aggregations the micro-batch engine remains the production choice.

Q: When is Spark Structured Streaming "enough" versus needing Flink?
A: Spark is enough when latency requirements are seconds-to-minutes, the team already runs Spark, and the logic is batch-shaped (windowed aggregations, joins) — for example minute-grained reconciliation. Flink wins when you need sub-second latency, complex event-time handling, large evolving keyed state with fine control, or per-record processing — for example a sub-second exposure monitor.

Q: How does Spark Structured Streaming persist and recover state?
A: It writes progress and state to a configured `checkpointLocation` (offsets, commit log, and the state store) at the end of each micro-batch; on restart it reads the offset log to resume from the last committed batch and reloads state. The watermark is checkpointed at the end of every micro-batch and restored at the start of the next, so event-time cleanup continues correctly across restarts.

Q: Scenario — after a Flink job restart, the flow totals differ from before the crash. What do you check first?
A: First confirm the job actually restored from the last completed checkpoint/savepoint rather than starting fresh (check the restore log line and the restored checkpoint id), and that the Kafka source resumed from checkpointed offsets, not Kafka-committed or `latest`. Then check for non-determinism (processing-time logic, unstable ordering) and that the sink is exactly-once/idempotent so no duplicates were re-emitted on replay.

Q: Scenario — a Flink windowed count is consistently too low for late-arriving fund orders. Which knob fixes it and at what cost?
A: The window is firing before stragglers arrive because the watermark advances too aggressively, so increase the out-of-orderness bound and/or add `allowedLateness` so late orders re-fire and restate the window — and capture truly-late events in a side output. The cost is latency: results finalize later and the consumer must accept restatements of previously emitted totals.

Q: Scenario — Flink checkpoints keep timing out and the job is under heavy backpressure. First mitigations?
A: Because aligned checkpoints stall behind buffered data under backpressure, enable unaligned checkpoints (or the adaptive aligned-checkpoint timeout) so barriers overtake in-flight data and checkpoint duration decouples from throughput. Also address the root backpressure (scale the slow operator, tune the sink) and consider RocksDB with incremental checkpoints if large state is part of the cost.

Q: Scenario — a savepoint restore fails after a code change with a state-incompatibility error. Likely causes?
A: Most often the operator `uid`s changed (auto-generated and reshuffled by editing the graph) so state cannot be matched, or a state type's serializer changed incompatibly (renamed/retyped a field with no migration path). Fix by pinning stable `.uid(...)` on stateful operators, and for type changes use a compatible serializer (Avro/POJO) or an explicit migration rather than an in-place incompatible change.

Q: Scenario — the Iceberg table fed by Flink shows duplicate rows after a job restart. What is the first thing to check?
A: Verify the sink is genuinely exactly-once: confirm the Iceberg/Flink sink commits are tied to checkpoint completion and that checkpointing is enabled and succeeding, because without checkpoints the 2PC sink cannot commit transactionally. Also check the source rewound to checkpointed offsets — if it replayed records into a non-transactional or append-only write path, you get duplicates.

Q: Why does enabling checkpointing matter for exactly-once sinks specifically?
A: Two-phase-commit sinks only commit their transaction when a checkpoint completes, so if checkpointing is disabled or never completes, the sink either never commits (data stuck) or falls back to at-least-once behaviour. Exactly-once therefore requires a working checkpoint cycle with a reasonable interval; the checkpoint is literally the commit trigger.

Q: What is the relationship between checkpoint interval and end-to-end latency for a 2PC sink?
A: Output becomes visible only at checkpoint completion, so the externally observable latency of a transactional sink is roughly bounded by the checkpoint interval plus commit time. A shorter interval lowers visibility latency but increases checkpoint overhead; you tune it to balance how fresh downstream NAV results must be against the I/O cost of frequent snapshots.

Q: What is backpressure in Flink and how is it signalled?
A: Backpressure is a slow downstream operator forcing upstream operators to slow down because output buffers fill and cannot be drained. Flink propagates it through its credit-based flow control rather than dropping data, and you observe it in the web UI's backpressure metrics and via rising input-buffer usage and growing consumer lag at the source.

Q: How does Flink's credit-based flow control prevent data loss under backpressure?
A: Each receiving subtask advertises buffer "credits" to senders, and a sender only transmits when the receiver has credit, so a slow consumer naturally throttles producers instead of overflowing buffers and dropping records. The slowdown propagates upstream to the source (which reads Kafka slower, growing lag) — flow is paused, never silently discarded.

Q: How would you point to backpressure in a running Flink job and prove it is not dropping data?
A: In the Flink UI, the job graph colours operators by backpressure status and shows busy/backpressured percentages and buffer-pool usage; a backpressured operator plus rising Kafka consumer lag at the source confirms throttling. Because credit-based flow control pauses upstream rather than dropping, no records are lost — they wait in Kafka (durable) until the consumer catches up.

Q: Why is consumer lag the early-warning signal even for a stateful Flink job?
A: Lag (committed/processed source offset versus log end offset) rises when Flink cannot keep up, which is the earliest external symptom of backpressure or an undersized job — well before checkpoints start failing. Monitoring lag per partition lets you act (scale, tune the slow sink) before the pipeline falls so far behind that NAV results become stale.

Q: What is a tumbling event-time window and when does it fit a fund metric?
A: A tumbling window is fixed-size and non-overlapping (for example hourly buckets), assigning each event to exactly one window. It fits metrics defined over disjoint periods — intraday fund-flow totals per hourly bucket, or daily subscription/redemption counts — where each event must be counted once in exactly one period.

Q: What is a sliding window and when would a fund use it?
A: A sliding window has a fixed size but advances by a smaller slide, so windows overlap and an event can belong to several windows (for example a 1-hour window sliding every 5 minutes). A fund would use it for a moving exposure or rolling-flow indicator that must update frequently while still summarizing a fixed look-back period.

Q: What is a session window and what defines its boundary?
A: A session window groups events separated by gaps smaller than a configured inactivity gap, closing when no event arrives for that gap duration; window length is data-driven, not fixed. For fund data it suits bursty per-investor activity — grouping a flurry of related orders from one investor into one logical session for analysis.

Q: Why is event time, not processing time, the default for regulated finance windows?
A: Because regulators care when a trade actually occurred, not when the pipeline happened to process it, and restatements demand reproducibility. Event time makes a window's membership depend on the embedded timestamp, so reprocessing the same input always yields the same totals; processing time would change results based on transient system speed, which is indefensible to an auditor.

Q: Give a fund example distinguishing event time from processing time.
A: Event time is the timestamp on an order ("this subscription was placed at 14:59:50"); processing time is when Flink ingested it ("the message reached the consumer at 15:02 after a broker lag"). Bucketing intraday flows by event time keeps a 14:59 order in the 14:00–15:00 window even if it arrives at 15:02, which is what NAV-flow correctness requires.

Q: What is a trigger in the Beam/Flink windowing model?
A: A trigger decides when a window emits a result relative to the watermark — on-time when the watermark passes the window end, optionally early (speculative) firings before completion, and late firings as stragglers arrive within allowed lateness. Triggers let you trade freshness against completeness, for example emitting a provisional intraday flow total early and restating it as late orders land.

Q: What are discarding, accumulating, and accumulating-and-retracting accumulation modes?
A: Discarding emits only the delta since the last firing (downstream must sum); accumulating re-emits the full updated window result each firing (downstream overwrites); accumulating-and-retracting emits a retraction of the previous result plus the new one so downstream can correct exactly. For restated NAV-flow totals, accumulating (overwrite) or retracting modes are typical so the latest figure replaces the earlier provisional one.

Q: Why do late firings require thinking about accumulation mode downstream?
A: Because a late firing re-emits or amends a window result, the sink must know whether it is receiving a delta, a full replacement, or a retraction-plus-new — get it wrong and you double-count or fail to correct. With accumulating mode the sink upserts on the window key; with retractions it applies the retract then the new value, keeping the stored total exact.

Q: What is the watermark latency-versus-completeness tradeoff in one sentence?
A: A more conservative watermark (longer out-of-orderness bound) waits longer so it captures more late events and is more complete, but it delays window firing and thus increases latency; a more aggressive watermark fires sooner but risks treating real events as late.

Q: How do watermarks propagate through a multi-input Flink operator?
A: An operator's output watermark is the minimum of the watermarks across all its input channels, because it can only assert completeness up to the slowest input. This is why one idle or lagging source partition can stall the whole job's event-time progress — its low (or absent) watermark holds back the minimum.

Q: What problem does watermark idleness handle and how do you configure it?
A: If a Kafka partition receives no data, its watermark never advances, and since downstream takes the minimum, event time stalls and windows never fire. You mark such sources idle with `WatermarkStrategy.withIdleness(Duration.ofMinutes(n))`, so an idle partition is temporarily excluded from the minimum computation, letting active partitions advance the watermark.

Q: What is the role of `RichFunction.open()` in stateful Flink operators?
A: `open()` is the lifecycle hook called once per subtask before processing begins, where you obtain state handles from the `RuntimeContext` (for example `getRuntimeContext().getState(descriptor)`) and initialize resources. State descriptors must be registered here because state access requires the runtime context, which is only available after the operator is opened.

Q: What is `notifyCheckpointComplete` and why do 2PC sinks rely on it?
A: It is a callback Flink invokes on operators after a checkpoint has fully completed and been confirmed durable. A two-phase-commit sink uses it as the signal to commit the external transaction it pre-committed at the barrier, because only at this point is the checkpoint guaranteed recoverable — committing earlier could leak output that a failed checkpoint would have had to abort.

Q: What happens to a 2PC sink's pre-committed transaction if the job fails before the checkpoint completes?
A: On recovery from the previous checkpoint, the pre-committed-but-not-committed transaction is aborted (or simply never committed and times out), so its output is not externally visible. This is precisely what preserves exactly-once: only transactions whose checkpoint completed are committed, so replayed records do not produce duplicate visible writes.

Q: What is the difference between checkpoint storage `JobManagerCheckpointStorage` and `FileSystemCheckpointStorage`?
A: `JobManagerCheckpointStorage` keeps checkpoint data in the JobManager's heap memory — only viable for tiny state and testing, with a bounded size. `FileSystemCheckpointStorage` writes checkpoints to a durable file system or object store (S3/MinIO/HDFS) via a configured path, which is what production jobs use so checkpoints survive a full cluster restart.

Q: For the Phase-4 lab, where would you checkpoint Flink state and why?
A: To MinIO (S3-compatible object storage) via `FileSystemCheckpointStorage` with an `s3://` path, because object storage is durable and survives TaskManager/JobManager loss, which heap-based storage does not. This mirrors the production pattern (S3/ADLS) and lets a restarted job recover state independently of which node it lands on.

Q: Why can a heap state backend still checkpoint to object storage?
A: Because state backend (where live state lives — JVM heap for `HashMapStateBackend`) and checkpoint storage (where snapshots are persisted) are independent settings. You can run fast heap working state and still configure `FileSystemCheckpointStorage` pointing at MinIO so snapshots are durable — the split exists exactly to allow this combination.

Q: What is `maxParallelism` and why does it cap rescaling?
A: `maxParallelism` fixes the number of key groups, which is the granularity at which keyed state is distributed and reassigned on rescale; an operator's actual parallelism can never exceed it. Because changing it would require rehashing all keys (invalidating existing snapshots), it is set at job creation and cannot be changed by a savepoint restore, so choose it generously up front.

Q: How do you rescale a stateful Flink job and what enables it?
A: Take a savepoint, then restart the job with a new parallelism from that savepoint; Flink redistributes keyed state by reassigning whole key groups to the new set of subtasks. Rescaling is bounded by `maxParallelism` (you cannot exceed the key-group count), which is why that value is the true ceiling on a job's scalability.

Q: What is an `AggregateFunction` in Flink and why prefer it for windowed sums?
A: `AggregateFunction<IN, ACC, OUT>` defines incremental aggregation — `add`, `merge`, `getResult` — so the window maintains only a small accumulator rather than buffering every event. For per-share-class flow totals it is far more memory-efficient than a `ProcessWindowFunction` that holds all events, and you can combine the two to get incremental aggregation plus window metadata.

Q: How do you get both incremental aggregation and window-context metadata?
A: Combine an `AggregateFunction` with a `ProcessWindowFunction` via the windowed `aggregate(aggFn, processFn)` overload: the aggregate keeps a compact accumulator, and the process function receives the pre-aggregated result plus the window's start/end and key. This gives memory-efficient totals while still emitting, say, the share class and window boundaries needed to upsert into Iceberg.

Q: What does it mean that managed cloud streaming "runs Flink underneath," and why care?
A: Services like AWS Managed Service for Apache Flink and Confluent Flink expose Flink (SQL or DataStream) as a managed offering, handling cluster ops while preserving Flink semantics. As architect you care because the concepts you debug locally — checkpoints, watermarks, state backends — transfer directly, and a build-vs-buy decision is largely about ops burden and lock-in, not a different programming model.

Q: How does Azure Stream Analytics fit a build-vs-buy comparison against self-managed Flink?
A: Azure Stream Analytics is the Azure-native, fully managed streaming option with a SQL-like language and tight Azure integration, attractive for low-ops simple windowed jobs. Compared to self-managed Flink it trades flexibility and portability (you cannot drop to arbitrary `ProcessFunction` code, and you are tied to Azure) for far lower operational overhead — the architect weighs that per workload.

Q: What is the difference between a window's allowed lateness and the watermark's out-of-orderness bound?
A: The out-of-orderness bound (in the `WatermarkStrategy`) delays the watermark so the window fires later and counts more events before its on-time firing. Allowed lateness applies after the window's on-time firing, keeping state alive so even-later events re-fire and restate it; together they form a two-stage tolerance — wait a bit before firing, then correct for a while after.

Q: Why might you choose RocksDB even when state fits in heap?
A: To get incremental checkpoints (only RocksDB supports them), which keep checkpoint size and duration small and predictable for a continuously updated state, and to avoid GC pauses that hurt latency tails. The point lookups are slower, but for jobs where checkpoint stability and bounded GC matter more than per-record latency, RocksDB is the safer operational choice.

Q: What is the practical difference between stop-with-savepoint and a plain cancel?
A: Stop-with-savepoint (`flink stop --savepointPath ...`) drains the pipeline, advances watermarks/finishes in-flight work, takes a final savepoint, and stops cleanly — ideal for upgrades. A plain cancel stops immediately without a fresh snapshot, so you would have to restore from the last automatic checkpoint, potentially losing the most recent progress and any clean drain.

Q: In the EOS chain Kafka source → Flink → Kafka/Iceberg sink, where exactly does each guarantee live?
A: The source guarantees no-loss/no-double via checkpointed offsets (rewind on recovery); Flink guarantees exactly-once state via barriers and checkpoints; the sink guarantees no duplicate external output via a transactional/2PC commit tied to checkpoint completion. Break any link — for example a non-transactional sink — and the end-to-end claim degrades to at-least-once at that edge.

Q: Why is "exactly-once" a property of the whole pipeline, not just Flink?
A: Flink's checkpointing only makes its internal state exactly-once; on replay it re-emits output, so without a transactional or idempotent sink the external world sees duplicates. End-to-end exactly-once requires every boundary to cooperate — checkpointed sources, exactly-once internal processing, and a sink that commits transactionally or deduplicates — which is why architects audit the claim edge by edge.

Q: How does an idempotent sink achieve effectively-once without 2PC?
A: An idempotent sink makes re-applying the same record a no-op — for example upserting on a primary key (ISIN + window) so a replayed record overwrites rather than appends. Combined with Flink's at-least-once delivery on replay, the net external effect is exactly-once ("effectively-once") without coordinating a distributed transaction, which is often simpler than 2PC.

Q: What determines whether you use a 2PC sink versus an idempotent sink for a fund result?
A: Use an idempotent sink when the target supports natural upserts/dedup on a stable key (most lakehouse merges, keyed stores) — it is simpler and avoids transaction coordination. Use a 2PC/transactional sink when output must be atomic and the target cannot dedupe append-only writes (for example transactional Kafka output or systems without an upsert key), accepting the extra coordination cost.

Q: What is the danger of running Flink at-least-once into a non-idempotent NAV store?
A: On recovery Flink replays records, and a non-idempotent store (plain append/insert) will record each replayed flow twice, silently inflating a NAV-flow total — exactly the kind of duplicated figure compliance discovers downstream. The fix is to make the store idempotent (upsert on key) or wrap output in a transactional sink so replays do not double-apply.

Q: How does a Flink keyed window cause memory growth if misconfigured, and how do you bound it?
A: Long allowed-lateness, many distinct keys, or session windows that never close retain window state indefinitely, growing memory until the job fails. Bound it by setting realistic allowed-lateness, using `EmbeddedRocksDBStateBackend` to spill to disk for large state, applying state TTL where appropriate, and using incremental aggregation so each window holds an accumulator rather than raw events.

Q: What is state TTL in Flink and when is it useful?
A: State TTL (`StateTtlConfig`) automatically expires keyed-state entries after a configured time-to-live, freeing state for keys that go silent. It is useful for unbounded key spaces — for example per-investor or per-ISIN state where old, inactive keys would otherwise accumulate forever — preventing slow state growth that eventually exhausts the backend.

Q: Which `StateTtlConfig` update type controls when the TTL clock resets, and which is the default?
A: The update type is set with `setUpdateType(...)`, taking `OnCreateAndWrite` (TTL resets on every write — the default) or `OnReadAndWrite` (also resets on reads), so the choice decides whether merely reading a key keeps it alive. For a per-ISIN running total that you write each event but rarely read, the default is fine; use `OnReadAndWrite` only when read activity should count as "still active," otherwise frequently-read-but-never-updated state expires unexpectedly.

Q: With Flink state TTL, when is expired state actually removed and what is the visibility default?
A: Expired entries are lazily removed on access by default and, for RocksDB, also cleaned in the background during compaction (`cleanupInRocksdbCompactFilter`); the heap backend uses an incremental cleanup that scans entries on each access. The visibility default is `NeverReturnExpired`, so a read of expired-but-not-yet-physically-removed state returns nothing — important because relying on lazy cleanup alone can let dead state linger and inflate checkpoint size until compaction runs.

Q: Why can a job with TTL configured still grow its checkpoints, and what is the fix?
A: TTL removal is lazy/background, so for a backend or access pattern where expired entries are never touched again they sit on disk until compaction reclaims them, keeping snapshots large. The fix is to enable an aggressive background cleanup — for RocksDB ensure `cleanupInRocksdbCompactFilter` runs often enough (tune the query-after-N-entries threshold) — so dead per-investor keys are physically purged rather than only logically expired.

Q: How do you enable RocksDB incremental checkpoints, and what does it not make incremental?
A: Enable it with `state.backend.incremental: true` (or the `EmbeddedRocksDBStateBackend(true)` constructor), after which each checkpoint uploads only new/changed SST files referencing prior ones. It does not make recovery itself incremental — restore still reads the full state — and it does not apply to savepoints in canonical format, which are always full; only the periodic RocksDB checkpoints benefit.

Q: A Flink job's checkpoint size keeps climbing even though incremental checkpoints are on — what mechanism explains this and how do you bound it?
A: Incremental checkpoints retain old SST files as long as any still-referenced checkpoint needs them, so a slowly-compacting RocksDB or a large `state.checkpoints.num-retained` keeps historical files alive and total checkpoint storage grows. Bound it by lowering retained checkpoints, ensuring compaction reclaims dead SSTs, and verifying TTL actually purges silent keys; the symptom is shared-state directory growth in the checkpoint folder, not the latest delta.

Q: What does `execution.checkpointing.tolerable-failed-checkpoints` default to, and what is the operational risk of leaving it there?
A: It defaults to `0`, meaning a single failed (timed-out or async-failing) checkpoint triggers a full job failover and restart from the last successful checkpoint. The risk is that one transient object-store hiccup restarts the whole NAV job; for production you often raise it (for example to `3`) so a brief blip is tolerated, while still failing if checkpoints are persistently broken.

Q: Which checkpoint failure phase always forces a failover regardless of `tolerable-failed-checkpoints`?
A: A failure in the synchronous phase on a TaskManager always fails the affected task and forces failover; the tolerance setting only covers async-phase failures, JobManager `IOException`, and expiration-by-timeout. So if your operator throws while snapshotting state synchronously, raising the tolerance does not protect you — you must fix the underlying snapshot error.

Q: How do you set a minimum pause between Flink checkpoints, and why does it matter under load?
A: Set `execution.checkpointing.min-pause` (the minimum gap between the end of one checkpoint and the start of the next); it matters because without it a slow checkpoint that nearly fills the interval leaves almost no time for actual processing, so checkpoints pile up back-to-back and starve the job. A sensible pause guarantees the job always gets processing time even when individual checkpoints run long under backpressure.

Q: What does `execution.checkpointing.max-concurrent-checkpoints` control and why is `1` the safe default for a 2PC pipeline?
A: It caps how many checkpoints can be in flight simultaneously; `1` is the default and the safe choice for transactional/2PC sinks because overlapping checkpoints would create ambiguous, interleaved pending transactions that complicate exactly-once commit ordering. Raising it can improve throughput for stateless-ish jobs but is generally avoided when a 2PC sink ties each commit to one completing checkpoint.

Q: How do you configure the checkpoint interval in Flink SQL or via config, not just the Java API?
A: Set `execution.checkpointing.interval` (for example `'30 s'`) in the job configuration or `SET` it in a Flink SQL session, which enables periodic checkpointing without touching `StreamExecutionEnvironment.enableCheckpointing(...)`. This matters for SQL-only jobs where there is no DataStream code to call — exactly-once sinks need a configured interval, and forgetting it silently leaves checkpointing disabled.

Q: A Flink SQL job writing to an exactly-once Kafka sink produces no output and never commits — what is the first config to check?
A: Check that checkpointing is actually enabled (`execution.checkpointing.interval` is set), because the transactional Kafka sink only commits when a checkpoint completes, and a SQL job with no interval set never checkpoints, so transactions stay open and nothing becomes visible. The tell-tale is data sitting in uncommitted Kafka transactions; setting an interval makes the pending output commit on the next completed checkpoint.

Q: When a downstream consumer must read a Flink exactly-once Kafka topic, what `isolation.level` must it use and why?
A: The consumer must set `isolation.level=read_committed`, otherwise it reads uncommitted (aborted-or-pending) records from Flink's in-flight transactions and sees duplicates or phantom rows. `read_committed` makes the consumer skip records belonging to open or aborted transactions, which is what actually delivers the exactly-once guarantee to the reader — the producer side alone is not enough.

Q: Why can Flink exactly-once Kafka output add latency for `read_committed` consumers, and how is it bounded?
A: A `read_committed` consumer cannot advance past an open transaction, so records stay invisible until Flink commits at checkpoint completion, making end-to-end visibility latency roughly the checkpoint interval. It is bounded by shortening `execution.checkpointing.interval`, trading more frequent (costlier) checkpoints for fresher visible output — the classic freshness-versus-overhead knob for a 2PC Kafka sink.

Q: What is `transaction.timeout.ms` / `transaction.max.timeout.ms` interaction for a Flink Kafka EOS sink, and what failure does mismatch cause?
A: Flink's producer transaction timeout must be less than or equal to the broker's `transaction.max.timeout.ms`; if Flink requests a longer timeout the broker rejects the producer at startup. Conversely, if the timeout is shorter than the worst-case time between checkpoints, a transaction can expire before it is committed, so the broker aborts it and you lose that output — set the timeout comfortably above the max expected checkpoint interval.

Q: After a long Flink outage, a restarted EOS Kafka job logs that transactions could not be resumed — what likely happened and what is the consequence?
A: The pending Kafka transactions exceeded `transaction.timeout.ms` while the job was down, so the broker aborted them; on restart Flink cannot recommit the lost transactions, meaning the output from the last incomplete checkpoint cycle was discarded. The consequence is acceptable for exactly-once (no duplicates, and the rewound source will reproduce that output), but it shows why the transaction timeout must exceed realistic recovery/downtime windows.

Q: How does Flink's `KafkaSource` set its starting position, and which option breaks exactly-once on a fresh start?
A: `KafkaSource` offsets are configured via `setStartingOffsets(...)` with `OffsetsInitializer` options like `earliest()`, `latest()`, `committedOffsets(...)`, or `timestamp(...)`, but on restore from a checkpoint these are ignored in favour of the checkpointed offsets. Using `latest()` is the trap on a redeploy without state — it skips unprocessed records, so a fresh start (no savepoint) silently drops data rather than continuing where the old job left off.

Q: In Flink, what is the difference between restarting from a checkpoint and restarting from a savepoint regarding source offsets?
A: Both restore the source position from the snapshot — the checkpointed/savepointed offsets override any `OffsetsInitializer` — so the job resumes exactly where it stopped in either case. The practical difference is lifecycle: checkpoints are auto-managed for failure recovery (and may be retained or cleaned per `num-retained`), while savepoints are operator-triggered, durable, and meant for planned restarts and upgrades.

Q: How do you tell Flink to restore a job while ignoring an operator whose state no longer exists in the new code?
A: Start the job with `--allowNonRestoredState` (the `-n` flag on `flink run -s`), which lets restore succeed even though the savepoint contains state for an operator you removed. Without it, Flink fails the restore to protect you from silently dropping state; you opt in only when you have deliberately deleted a stateful operator and accept losing its state.

Q: A savepoint restore fails with "Cannot map checkpoint/savepoint state for operator X" — give two distinct fixes depending on intent.
A: If operator X still exists but its auto-generated UID changed, the fix is to assign a stable `.uid("...")` matching the original (or re-take the savepoint after pinning UIDs) so state maps again. If X was intentionally removed, the fix is to restore with `--allowNonRestoredState` so the orphaned state is discarded deliberately — the right fix depends on whether you meant to keep that state.

Q: Why does Flink recommend pinning both `.uid(...)` and `.name(...)`, and what does each affect?
A: `.uid(...)` is the stable identifier used to match savepoint state on restore and must never change for a stateful operator; `.name(...)` is the human-readable label shown in the web UI and logs and has no bearing on state mapping. Pinning the UID protects upgrades, while a good name aids debugging — confusing the two leads people to rename for clarity and accidentally break state if they only set the name.

Q: What serializer underlies Flink's most robust state schema evolution, and what change does it still reject?
A: Avro-serialized state (via `AvroSerializer`) gives the strongest evolution support because Avro's schema-resolution rules govern compatibility — you can add fields with defaults, remove fields, and reorder. It still rejects incompatible changes like changing a field's type to something Avro cannot resolve or removing a field that lacks a default on the read schema; those require an explicit migration step rather than an in-place restore.

Q: What evolution does Flink's POJO serializer support, and what is its key limitation versus Avro?
A: The POJO serializer supports adding and removing fields across restores (new fields get default/zero values, removed fields are dropped), so simple shape changes evolve cleanly. Its key limitation is that it cannot change the declared type of an existing field and is sensitive to the class definition; for richer, contract-driven evolution and explicit defaults you move state types to Avro.

Q: Why are Kryo-serialized state types a schema-evolution trap in Flink?
A: Kryo is Flink's generic fallback serializer and does not support state schema evolution — if your state falls back to Kryo (because the type is not a recognized POJO/Avro/registered type), you cannot evolve that type's shape across savepoints without a custom migration. The fix is to make the type a proper POJO or use Avro, and to detect the trap early by checking logs for "using generic Kryo serializer" warnings on stateful types.

Q: How do you migrate state when a serializer change is genuinely incompatible, without losing data?
A: Use the State Processor API (`org.apache.flink.state.api`) to read the old savepoint as a dataset, transform each key's state into the new shape, and write a new savepoint that the upgraded job restores from. This offline "upcaster" path handles changes the in-line serializer compatibility check rejects — for example splitting one field into two — so you migrate rather than discard a NAV job's accumulated state.

Q: What is the Flink State Processor API used for beyond migration?
A: Besides migrating/transforming state, the State Processor API lets you read existing savepoints offline to inspect or audit state (for example dump every per-ISIN running total for a reconciliation), and bootstrap a brand-new savepoint to pre-seed a job's state before it starts. It treats savepoint state as a batch dataset, which is invaluable when you must prove to an auditor what a job's state contained at a point in time.

Q: Why is `keyBy` with a lambda that returns a non-deterministic or mutating key dangerous in Flink?
A: `keyBy` must return a stable, deterministic key for each record because the key determines partition routing and which subtask holds the state; a key derived from mutable fields or wall-clock time would send the same logical entity to different subtasks, scattering its state. The result is corrupted per-key aggregates (a single ISIN's total split across partitions) — keys must be immutable, deterministic functions of the record.

Q: What goes wrong if your `keyBy` key type has an inconsistent `hashCode`/`equals`, and how does it manifest?
A: Flink partitions by the key's hash, so an unstable or mismatched `hashCode`/`equals` (for example a key object without proper overrides, or one whose hash varies across JVMs) routes records for the same logical key inconsistently, fragmenting keyed state. It manifests as aggregates that are mysteriously too low or split, and is a classic reason a "per-share-class total" is wrong even though the code looks right — always key by a primitive or a value type with correct equality.

Q: What is key skew in a Flink keyed stream and how would you detect and mitigate it?
A: Key skew is when a few keys (for example one dominant share class) carry most records, overloading the subtasks that own those key groups while others idle, causing backpressure and uneven checkpoint times. Detect it via per-subtask record/throughput metrics in the UI; mitigate by pre-aggregating with a salted/two-stage key (local combine then re-key) or by choosing a finer key so load spreads across more key groups.

Q: How would you implement a two-stage (local-then-global) aggregation to fight skew for a hot share class?
A: Add a salt to the key for a first `keyBy(salt+shareClass)` aggregation that partial-aggregates each shard in parallel, then strip the salt and `keyBy(shareClass)` again to combine the partials into the final total. This spreads the hot key's load across many subtasks in stage one and only the small set of partial results converges in stage two, trading a little extra shuffle for removing the single-subtask bottleneck.

Q: What is the difference between `reduce`, `aggregate`, and `process` on a Flink window, in terms of state held?
A: `reduce` and `aggregate` are incremental — they keep only a running accumulator per window and combine eagerly as events arrive — while `process` (a `ProcessWindowFunction`) buffers all events in the window so it can iterate over them at firing time. For large windows you prefer incremental functions to avoid holding every event; use `process` only when you genuinely need the full element collection or window metadata, ideally combined with an aggregate.

Q: Why might a `ProcessWindowFunction` alone cause memory blowups that `aggregate(aggFn, processFn)` avoids?
A: A bare `ProcessWindowFunction` retains every element of the window in state until it fires, so a long or busy window holds thousands of raw orders, exhausting the backend. The `aggregate(aggFn, processFn)` overload runs the incremental aggregate as events arrive and passes only the single pre-aggregated result to the process function, so window state is one accumulator plus metadata — orders of magnitude smaller for high-volume fund flows.

Q: What is the purpose of `merge` in a Flink `AggregateFunction`, and which window type requires it?
A: `merge` combines two accumulators into one and is required by session windows, because sessions can be merged when a new event bridges the inactivity gap between two previously separate sessions. Tumbling and sliding windows do not merge, but implementing `merge` correctly is mandatory for session windows — an incorrect or missing merge corrupts results when sessions coalesce, for example joining two bursts of one investor's orders.

Q: How do processing-time timers behave differently from event-time timers across a Flink restart?
A: Event-time timers fire when the watermark passes their timestamp, so after a restart they fire deterministically as the replayed stream advances the watermark — reproducible. Processing-time timers fire on wall clock, so on restart any that were "due" during downtime fire roughly immediately and their timing is not reproducible; this is why processing-time timers break the byte-identical-replay requirement for a NAV job.

Q: A Flink event-time window never fires in production though it works in tests — what is the most common cause?
A: The watermark is not advancing, almost always because one or more source partitions are idle (no data) and the operator takes the minimum watermark across inputs, so event time stalls and on-time firings never trigger. The fix is `WatermarkStrategy.withIdleness(Duration.ofMinutes(n))` so idle partitions are excluded from the minimum; in tests a single bounded source advances cleanly, which is why the bug hides until production with multiple live partitions.

Q: Beyond idleness, name two other reasons a Flink watermark may stall and starve windows.
A: First, no watermark strategy is assigned (or the timestamp assigner returns wrong/constant timestamps), so the watermark never progresses; second, a single very late/out-of-order partition or a stuck upstream operator holds the minimum watermark down for the whole job. Both manifest as windows that buffer state forever without firing — you diagnose by inspecting the per-operator current-watermark metric in the Flink UI to find which input is lagging.

Q: How do you inspect the current watermark of a running Flink operator to debug a stalled window?
A: The Flink web UI exposes a `currentInputWatermark`/`currentOutputWatermark` metric per operator (also available via the metrics system), so you walk the job graph and find the operator whose watermark is frozen or far behind wall clock. A watermark stuck at `Long.MIN_VALUE` means it never advanced (missing strategy or all-idle inputs); a watermark pinned below others points to the specific lagging input channel to fix.

Q: What does a watermark value of `Long.MIN_VALUE` (or "no watermark") on an operator indicate?
A: It indicates the operator has not yet received or generated any valid watermark — typically because no `WatermarkStrategy` is attached, the source has produced no data, or every input is still idle — so event-time progress is zero and no event-time window or timer can fire. Seeing it on the source is the fastest confirmation that watermark generation, not the windowing logic, is the root cause of missing output.

Q: In Flink SQL, how do you route late records that fall outside the watermark, and how does it differ from DataStream side outputs?
A: Modern Flink SQL window TVFs do not expose a per-window side output the way DataStream does; instead you control lateness through the table's `WATERMARK ... AS ts - INTERVAL` bound, and records later than that are dropped by the windowing operator. To capture stragglers you either widen the watermark delay, handle them in a separate query/branch, or drop to the DataStream API where `OutputTag` side outputs give explicit late-data routing — a real limitation to weigh when choosing SQL over DataStream for auditable late-trade handling.

Q: What does the Flink `numLateRecordsDropped` metric tell you, and what action does a rising value imply?
A: `numLateRecordsDropped` counts records that arrived after the watermark passed their window end and beyond any allowed lateness, so the window had already closed and dropped them. A rising value means real events (for example late fund orders) are being silently discarded from totals; the action is to increase the out-of-orderness bound or `allowedLateness`, and add a side output so dropped records are captured for reconciliation instead of lost.

Q: How do you set RocksDB to use a predefined performance profile in Flink, and give an example option.
A: Use `state.backend.rocksdb.predefined-options` with a profile such as `SPINNING_DISK_OPTIMIZED_HIGH_MEM` or `FLASH_SSD_OPTIMIZED`, which presets block cache, write buffers, and compaction settings tuned for that storage class. On a laptop or SSD-backed lab you would pick the flash profile; choosing the spinning-disk profile on SSD leaves performance on the table, while finer tuning uses the `RocksDBOptionsFactory` for explicit control.

Q: What is RocksDB write amplification in the Flink context and why does it affect checkpoint and disk behaviour?
A: Write amplification is RocksDB rewriting the same data multiple times as LSM levels compact, so a high update rate on keyed state (frequent per-ISIN total updates) causes background compaction and extra disk I/O. It affects Flink because compaction competes with checkpoint uploads and processing for disk bandwidth, and it is also what eventually purges TTL-expired and overwritten entries — so tuning compaction balances steady-state latency against how fast dead state is reclaimed.

Q: How does managed-memory allocation interact with RocksDB in Flink, and what breaks if it is too small?
A: RocksDB by default draws its block cache and write buffers from Flink's managed memory (`taskmanager.memory.managed`), with usage bounded so it does not blow the container limit. If managed memory is too small, RocksDB thrashes — tiny block cache means more disk reads, small write buffers mean more frequent flushes/compaction — degrading throughput and inflating checkpoint times; the fix is to raise the managed-memory fraction or container memory for state-heavy jobs.

Q: Compare how Flink and Spark Structured Streaming each persist the watermark across a restart.
A: Flink carries watermarks as in-band markers and reconstructs event-time progress from the replayed stream after restoring state from a checkpoint, so watermark progress resumes as data flows. Spark checkpoints the watermark value to its `checkpointLocation` at the end of every micro-batch and restores it at the start of the next, so the global watermark survives restart explicitly; both keep event-time cleanup correct, but Flink's is per-operator and stream-driven while Spark's is a single global value persisted per batch.

Q: In Spark Structured Streaming append output mode, when is a windowed aggregation actually emitted, and why does that surprise people?
A: In append mode a windowed aggregation row is emitted only after the watermark passes the window end, finalizing it — so nothing appears for a window until the watermark has moved beyond it plus the late-tolerance. It surprises people because they see no output during the window's lifetime and assume the job is broken; the alternative `update` mode emits running results each trigger, which is what you want for live intraday flow figures versus finalized append rows.

Q: Why is Spark append mode incompatible with non-windowed (unbounded) aggregations, and what mode do you use instead?
A: Append only emits rows once they are finalized and will never change, but an unbounded aggregation (a grand total with no window/watermark) can always change, so there is no point at which a row is safe to append. You therefore use `update` (or `complete`) output mode for such aggregations, accepting that the engine re-emits changing rows — append is reserved for watermark-finalized windows or stateless transforms.

Q: How does Spark Structured Streaming achieve exactly-once for file/Delta sinks, and how is the mechanism different from Flink's 2PC?
A: Spark uses idempotent, batch-versioned commits: each micro-batch has an offset range and writes are committed atomically via the sink's commit log (for Delta, an atomic transaction-log entry) keyed to that batch id, so re-running a batch overwrites rather than duplicates. Unlike Flink's per-checkpoint two-phase transactional commit across operators, Spark's guarantee comes from deterministic batch boundaries plus an idempotent/transactional sink replaying the same batch id, which is why it requires sinks that support that commit protocol.

Q: For a sub-second exposure monitor versus a minute-grained reconciliation, which engine fits each and why — give the deciding factor.
A: The sub-second exposure monitor fits Flink, because its event-at-a-time pipeline has no micro-batch latency floor and offers fine event-time/state control needed for tight, per-record alerting. The minute-grained reconciliation fits Spark Structured Streaming, since seconds-to-minutes latency is fine, the logic is batch-shaped, and reusing the existing Spark/DataFrame stack lowers operational cost — the deciding factor is the latency requirement weighed against existing team tooling and ops burden.

Q: A managed Flink service (for example AWS Managed Service for Apache Flink) reports your job is stuck restarting after a config change — what state-related cause should you suspect first?
A: Suspect a state/UID incompatibility from the new code: if the deployment restored from a snapshot but the operator UIDs or state serializers changed incompatibly, the job fails on restore and loops restarting. The check mirrors self-managed Flink — confirm stable `.uid(...)`s and compatible serializers, and if you intentionally removed an operator use `allowNonRestoredState` — because the managed wrapper preserves Flink's exact restore semantics underneath.

Q: What restart strategy should a production Flink NAV job use, and why is `none`/`fixed-delay` with one attempt wrong?
A: Use a resilient strategy such as `exponential-delay` or `failure-rate` (via `restart-strategy: exponential-delay` or `failure-rate`), so transient failures retry with backoff and the job self-heals from each successful checkpoint instead of staying down. A strategy of `none`, or `fixed-delay` capped at one attempt, means the first transient error (a broker blip, a brief object-store timeout) permanently stops the job — unacceptable for a continuously-running NAV feed that must recover unattended.

Q: What is the `_metadata` file in a Flink checkpoint/savepoint directory, and why must you never delete it?
A: `_metadata` is the manifest that records the operator-state handles and pointers to the actual state data files for that snapshot, so it is the entry point Flink reads to restore — you pass its directory to `flink run -s`. Deleting or losing `_metadata` orphans the state data and makes the snapshot unrestorable even though the data files remain, which is why checkpoint cleanup must treat a checkpoint directory atomically rather than pruning individual files.
