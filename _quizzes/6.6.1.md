# Phase 7 · 6.6.1 Embedding models — 100 self-test questions

Q: What is a text embedding, in one sentence?
A: A text embedding is a fixed-length vector of floating-point numbers that represents the meaning of a piece of text, produced by a model so that semantically similar texts land near each other in the vector space. Retrieval works by embedding both the query and the documents and finding the document vectors closest to the query vector. The whole point is that geometric closeness (cosine or dot product) stands in for semantic relevance.

Q: Why does the lesson call embedding-model choice "an architect decision dressed as a detail"?
A: Because the model you pick quietly fixes your retrieval quality, your running cost, your storage footprint, and how painful every future re-index will be — decisions with budget and compliance consequences, not just a config line. A wrong English-only pick on a French Luxembourg corpus silently returns garbage for French KIDs, and a casual model upgrade triggers an unbudgeted full re-index through a third-country API. None of that is visible at the moment you type the model name, which is exactly why it gets under-weighted.

Q: What are the three corners of the "cost/latency/sovereignty triangle" for embeddings?
A: Cost (per-million-token API fees plus storage), latency (network round-trips and throughput), and sovereignty (where the text and vectors are processed and stored, and under whose jurisdiction). Local `sentence-transformers` models trade upfront GPU/CPU effort for zero per-call fee and full data residency; API embeddings trade convenience and quality for recurring cost and sending your corpus to a third party. You rarely optimise all three at once, so the architect picks which corner the use case can least afford to lose.

Q: What is `sentence-transformers` and where do you get its models?
A: `sentence-transformers` (often imported as `from sentence_transformers import SentenceTransformer`) is a Python library that runs embedding and reranking models locally, on your own CPU or GPU, with pretrained weights pulled from the Hugging Face Hub. You call `model.encode(texts)` to get vectors and never send the text off-box. Its docs at `https://www.sbert.net/` list pretrained multilingual models and semantic-search examples and are the primary resource for this lesson.

Q: Name one concrete advantage of local `sentence-transformers` embeddings over an API in a regulated fund context.
A: Data sovereignty: the corpus and the resulting vectors never leave your infrastructure, so French KIDs, investor lists, and prospectuses are not transmitted to a third-country processor. That sidesteps a whole class of GDPR/data-residency and DORA third-party-risk questions before they start. The trade is that you own the hardware, the model updates, and the throughput tuning.

Q: Name one concrete advantage of an API embedding model over a local one.
A: You get a typically stronger, regularly improved model with zero infrastructure to run — no GPU to provision, no `sentence-transformers` version to maintain, just an HTTP call. For a small or bursty corpus this is cheaper and faster to ship than standing up local inference. The cost is recurring per-token billing, network latency, and sending your text to an external processor.

Q: Does EU data residency apply to embeddings, or only to raw documents?
A: It applies to embeddings too — that is the lesson's explicit warning ("EU data residency applies to embeddings too!"). When you call an API to embed text, the text itself is transmitted and processed by the provider, so the input is subject to residency rules even though the output is "just numbers". Treating the vector as anonymous is a mistake: the source text crossed the border to produce it.

Q: Can the embedding vector itself leak the source text, or is it safely anonymous?
A: It is not safely anonymous. Embedding-inversion research shows vectors can be partially reconstructed back into recognisable text, so a leaked vector store can disclose investor or fund content. More to the point for residency, the text had to be sent to the provider to be embedded in the first place, so the privacy exposure happens at embed time regardless of what the stored vector reveals.

Q: What does the Azure OpenAI privacy documentation say about whether your embeddings train Microsoft's or OpenAI's models?
A: It states that prompts, completions, embeddings, and uploaded training data are not used to train or improve Microsoft's or OpenAI's foundation models, and are not shared with other customers or model providers, without your permission. This matters because "your data won't be used for training" is a different guarantee from "your data stays in the EU" — the two are independent. You should cite the documentation, not assume, when a compliance reviewer asks.

Q: What problem do Azure OpenAI "Data Zones" solve, and which is relevant for a Luxembourg fund?
A: Data Zones let you pin processing and storage of inputs/outputs to a specific geography, such as the EU, so data does not leave that boundary during inference. For a Luxembourg fund the EU Data Zone is the relevant deployment type, because a default global deployment could route processing outside the EU. Choosing the wrong deployment type is how an API embedding pipeline quietly violates residency expectations.

Q: A reviewer asks "where exactly is our text processed when we embed it via the API?" — how should an architect answer?
A: Not from memory — by reading the provider's current data-privacy and residency documentation for the specific deployment type and region you configured, and citing it. For Azure OpenAI you would point to the deployment's data-zone or regional setting; the lesson's deepening resource is the Azure OpenAI data, privacy, and security page. The professional answer names the document and the configured region, because residency depends on exactly how the service was set up.

Q: What does "abuse monitoring" retention mean for API embeddings, and why might a fund care?
A: Some API providers retain a sampled subset of inputs and outputs for a limited window (Azure OpenAI documents up to 30 days) for abuse monitoring, unless you are approved for a zero-retention configuration. For confidential fund documents even a short retained copy at a third party can be a compliance concern. The architect should check whether the deployment qualifies for, and is configured with, the no-retention option.

Q: What is "data sovereignty" as opposed to "data residency"?
A: Residency is the physical location where data is stored and processed; sovereignty is whose laws and jurisdiction govern that data, which can differ from where the bytes physically sit. A US-headquartered provider may store EU data in an EU region (residency satisfied) while still being subject to US legal process (a sovereignty concern). For regulated funds both questions need separate answers, and the lesson's triangle names sovereignty deliberately, not just residency.

Q: What does "dimensionality" mean for an embedding model?
A: It is the number of components in each output vector — for example 384, 768, 1536, or 3072 floats per text. The model's architecture fixes a native dimension, though some models support producing shorter vectors. Dimensionality directly drives how many bytes each vector costs to store and how much work an approximate-nearest-neighbour (ANN) index does per query.

Q: How many bytes does a single 768-dimensional embedding occupy in 32-bit float storage?
A: 768 × 4 = 3072 bytes, i.e. about 3 KB per vector before any index overhead, because each `float32` is 4 bytes. Multiply by your document count to size the store: one million 768-dim vectors is roughly 3 GB of raw vectors plus index structures. This back-of-envelope is exactly the storage-versus-width tradeoff the lesson flags.

Q: Is a higher-dimensional embedding reliably better than a lower-dimensional one?
A: No — "bigger is not reliably better" is an explicit Learn point. Extra dimensions cost storage and ANN latency and can add noise, while a smaller well-trained model often matches or beats a larger one on your actual corpus. You decide width by measuring retrieval quality on your own queries, not by assuming more dimensions wins.

Q: How does vector width affect approximate-nearest-neighbour (ANN) latency?
A: Wider vectors mean more arithmetic per distance computation and more memory traffic per comparison, so both index build time and per-query latency rise roughly with dimension. An index over 3072-dim vectors does about four times the distance work of one over 768-dim vectors. At scale this turns a "harmless" model upgrade into a measurable latency and cost regression.

Q: What is Matryoshka Representation Learning (MRL) and why is it useful here?
A: MRL trains a model so that the most important information is packed into the leading components of the vector, meaning you can truncate the vector to a shorter length and still get usable embeddings without retraining. OpenAI's `text-embedding-3` models support this via a `dimensions` API parameter — for example shortening `text-embedding-3-small` to a value between 256 and 1536. It lets you trade a little accuracy for big storage and latency savings on demand.

Q: For `text-embedding-3-large`, what native dimension does it produce and what can you shorten it to?
A: It natively produces 3072-dimensional vectors and supports requesting fewer dimensions (for example 1536) via the `dimensions` parameter thanks to Matryoshka training. Halving the dimension halves the bytes stored per vector and roughly halves ANN distance work, at a small accuracy cost. The per-token API price is unchanged by shortening — you only save on storage and query cost, not on the embedding call.

Q: What native dimension does `text-embedding-3-small` produce?
A: It produces 1536-dimensional vectors natively, and supports shortening down to as low as 256 dimensions via the `dimensions` parameter. It is the cheaper of OpenAI's two current third-generation models, useful when you want API quality without `text-embedding-3-large` storage. As always you confirm the width and price against current OpenAI documentation rather than memory.

Q: What dimension does the local model `paraphrase-multilingual-mpnet-base-v2` output?
A: It outputs 768-dimensional vectors. It is built on XLM-RoBERTa, covers about 50 languages including French and English, and is a common `sentence-transformers` multilingual choice for a French/English corpus. Knowing the width up front lets you size storage (768 × 4 bytes per vector) before you embed anything.

Q: What is the maximum input sequence length of `paraphrase-multilingual-mpnet-base-v2`, and why does it matter?
A: Its `max_seq_length` is 128 tokens, so any text longer than that is silently truncated at encode time. That is short, which means you must chunk documents into small pieces before embedding or you will lose the tail of every long passage. A common gotcha is feeding whole KID pages to a 128-token model and wondering why retrieval misses content near the bottom of the page.

Q: Why must "domain fit" beat "leaderboard rank" when choosing an embedding model for fund text?
A: Because a model that tops a general benchmark may have been trained mostly on English web text and handle French regulatory vocabulary or finance jargon poorly, while a lower-ranked multilingual model nails your actual queries. The Learn point is explicit: multilingual reality and finance vocabulary beat leaderboard rank. You only learn domain fit by testing on your own French/English fund queries.

Q: Why is multilingual capability non-negotiable for a Luxembourg fund corpus specifically?
A: Luxembourg fund documentation routinely mixes French and English (and sometimes German), so KIDs, prospectuses, and factsheets exist in multiple languages and users query in both. An English-only model embeds French text into a space where it cannot match the right French passages, silently returning wrong or empty results. The lesson's worst-case is exactly this: French KIDs "silently retrieve garbage".

Q: Give an example of finance vocabulary that a general embedding model might handle poorly.
A: Terms like `frais courants` (ongoing charges), `SRI`/`SRRI` (the risk indicator on a KID), `valeur liquidative` (NAV), or `ISIN` carry precise regulatory meaning that a general-web model may treat as rare or ambiguous tokens. If the model has weak representations for these, a query about ongoing charges may fail to retrieve the fee table. This is why you evaluate on domain queries, not generic sentences.

Q: How would you actually test "domain fit" rather than trusting a model's reputation?
A: Write a set of real domain queries (the lesson asks for 20 in a French/English mix), embed your corpus and those queries with the candidate model, and measure hit-rate@k — how often the correct passage appears in the top k results. The Done-when criterion is showing how the French queries changed the choice versus an English-only evaluation. The reputation of the model is irrelevant if it scores poorly on your set.

Q: What is `hit-rate@k` and why is it the right metric for shortlisting an embedding model?
A: Hit-rate@k is the fraction of queries for which a known-correct passage appears in the top k retrieved results, measured before any LLM generation. It isolates retrieval quality — the embedding model's job — from the language model's ability to paper over bad context. Because it is cheap and objective, it is exactly the signal you want when comparing two embedding models on the same corpus.

Q: Why measure retrieval quality of the embedding model before plugging in an LLM at all?
A: Because if the right passage never gets retrieved, no amount of LLM cleverness can produce a correct, citable answer — and a fluent-but-wrong answer is worse than an obvious failure. Measuring retrieval in isolation tells you whether the embedding/index layer is sound before generation hides or masks its faults. It also makes A/B comparison of embedding models clean: same queries, same k, just swap the model.

Q: The lesson says "model change = full re-index = a pipeline design issue." Why is changing the embedding model so expensive?
A: Because vectors from different models (or even different versions of the same model) live in incompatible spaces, so you cannot mix old and new vectors in one index — every document must be re-embedded and re-indexed. For a large corpus that is a heavy batch job, and via an API it is a real per-token bill you must budget. Treating a model upgrade as a one-line change is how you incur an unplanned full re-index and a possibly large invoice.

Q: Why can't you compare a vector embedded by model A against a vector embedded by model B?
A: Each model defines its own coordinate space, so the same text gets different vectors and distances are only meaningful within one model's space — cross-model cosine similarity is noise. Mixing them in one index means queries embedded by the new model are compared against documents embedded by the old one, which breaks retrieval. This is precisely why a model change forces a complete re-index rather than an incremental top-up.

Q: What does "pin model name+version next to the vectors" mean in practice?
A: Store, alongside each vector (or each index/collection), the exact model identifier and version used to produce it — for example a metadata column or collection tag recording `paraphrase-multilingual-mpnet-base-v2` plus a revision or date. Then any code reading the store can verify the query is embedded with the same model, and you can detect when a re-index is required. Without this pin, you eventually mix model versions and silently corrupt retrieval.

Q: A new model version ships and quietly replaces the old one. Why is "just upgrade in place" dangerous?
A: Because new query vectors will be compared against documents still embedded by the old version, so retrieval quietly degrades with no error thrown — the worst kind of failure for a regulated workflow. Even a "minor" version bump can shift the space. The safe path is a planned full re-index with the version pinned, never an in-place swap of the encoder while the existing vectors remain.

Q: What are the two main re-embedding rollout strategies the lesson asks you to consider?
A: A dual-write (write vectors to both old and new indexes during a transition, query one, then flip) or a cutover window (build the new index offline, then switch reads atomically). Dual-write minimises downtime and allows shadow comparison but costs double storage and write effort during the window; cutover is simpler but needs a controlled switch and a fallback. The choice depends on tolerance for downtime versus extra cost and complexity.

Q: What three things should a complete "re-embedding strategy" specify?
A: The trigger (what event forces re-indexing — a model upgrade, a new model version, or new documents), the batch re-index job (how the corpus gets re-embedded and re-indexed, owned by your orchestrator), and the transition plan (dual-write or cutover window, with rollback). The Done-when criterion requires your model pick to ship with this strategy stated. It turns "we changed the model" from a fire drill into a routine pipeline run.

Q: Where should the re-index job live in an orchestrated data platform, and why?
A: In your orchestrator (for this plan, a Dagster job over the vector-store asset), so it is scheduled, observable, retryable, and auditable like any other pipeline — not a hand-run script. A model upgrade then becomes a tracked materialisation with logs and lineage, which matters for a regulated environment. Treating re-embedding as ad-hoc shell work is how you lose reproducibility and auditability.

Q: What is MTEB and what is it for?
A: MTEB (the Massive Text Embedding Benchmark) is a public benchmark and leaderboard that scores embedding models across many task types — retrieval, clustering, classification, reranking, and more — across datasets and languages. It is hosted as a Hugging Face Space and is the standard first stop for comparing models. Its purpose in this lesson is shortlisting candidates, not making the final decision.

Q: The lesson calls MTEB "a (gameable) benchmark." How is a leaderboard like MTEB gameable?
A: Because public benchmark datasets can leak into training data (contamination), models can be tuned specifically to the benchmark's tasks, and authors choose which configuration to report — so a high rank can reflect benchmark-fitting rather than general quality. Average scores also hide weakness on the task type and language you actually care about. The rank is a signal, not a verdict.

Q: What is the correct, disciplined way to use MTEB?
A: "Shortlist with it, decide with your own retrieval set" — use MTEB to narrow a huge field to a few plausible candidates filtered by task type, language coverage, dimension, and licence, then run those finalists on your own labelled fund queries and pick by your measured hit-rate. The leaderboard chooses your bracket; your data chooses the winner. Skipping the second step is the classic mistake the lesson warns against.

Q: Explain to a sceptical colleague why MTEB rank alone would be a bad selection method for a Luxembourg fund corpus.
A: Because the top MTEB model may be English-centric or strong on tasks unlike retrieval over French regulatory PDFs, so it can rank first overall yet retrieve French KID content poorly. The benchmark's averages and language mix do not match your corpus, and contamination can inflate ranks. The only trustworthy evidence is hit-rate@k on your own French/English fund queries — which is exactly a Done-when criterion for this lesson.

Q: Why filter MTEB by task type before reading the ranking?
A: Because a model that excels at classification or clustering may be mediocre at retrieval, and retrieval is what RAG needs — the overall average blends task types you do not use. Reading the retrieval-specific results gives a relevant ranking, while the headline average can be misleading. You want the model that ranks well on the task you will actually run.

Q: Why also filter MTEB by language before shortlisting?
A: Because a model that tops English retrieval can be untrained or weak on French, and your Luxembourg corpus is French/English — an English-only leader is the trap the lesson's "Why" describes. Filtering to multilingual results (or a multilingual MTEB variant) keeps only candidates that can represent both languages in one space. This is the difference between a model that works on your corpus and one that quietly fails on the French half.

Q: Besides quality scores, what two model attributes does the lesson require you to record when shortlisting?
A: Dimensions and licence, plus the published price for the API option — step 1 of the Do says to record dimensions, licence, and published price. Dimensions drive storage and latency, licence determines whether you may legally use the model in a commercial regulated product, and price drives ongoing API cost. These non-quality attributes often decide the pick as much as accuracy does.

Q: Why is the model's licence a first-class selection criterion, not an afterthought?
A: Because a model with a non-commercial or restrictive licence cannot legally back a production fund-administration product no matter how well it retrieves, and discovering that after building is expensive. Some strong open-weight models carry usage restrictions or require attribution. Recording the licence at shortlist time prevents picking a model you are not allowed to deploy.

Q: How does `model.encode()` in `sentence-transformers` turn into stored vectors, conceptually?
A: You pass a list of texts to `model.encode(texts)` and it returns a NumPy array (or tensors) of shape (number_of_texts, dimension), which you then write to your vector store with each document's id and metadata. You typically batch the calls for throughput and may pass `normalize_embeddings=True` so cosine similarity reduces to a dot product. The vectors plus the pinned model version are what you persist for retrieval.

Q: What does `normalize_embeddings=True` do, and why might you set it?
A: It scales each output vector to unit length (L2 norm of 1), so that the dot product between two vectors equals their cosine similarity. This lets your vector store use the cheaper inner-product distance while still ranking by cosine, and keeps similarity values in a comparable range. You set it when your index and similarity metric expect normalised vectors, which is common for semantic search.

Q: Why is cosine similarity (not raw Euclidean distance) the usual metric for text embeddings?
A: Because cosine measures the angle between vectors and ignores their magnitude, capturing semantic direction rather than length, which suits text where vector magnitude carries little meaning. On normalised vectors cosine and Euclidean rank identically, but cosine is the conventional default and what most embedding models are tuned for. Picking a metric the model was not designed for can quietly hurt retrieval.

Q: When embedding a large corpus locally, what is the single most important throughput lever?
A: Batching — passing many texts to `encode()` at once with an appropriate `batch_size` so the model processes them in parallel on the GPU/CPU, rather than one call per document. A larger batch raises throughput up to the memory limit; too large and you hit out-of-memory errors. Embedding one text at a time in a Python loop is the classic reason a corpus takes hours instead of minutes.

Q: Your local embedding job crashes with a CUDA out-of-memory error mid-corpus. First thing to check?
A: The batch size — lower the `batch_size` passed to `encode()` so each forward pass uses less GPU memory, since OOM during embedding is almost always too many texts (or too-long texts) batched together. Also confirm inputs are truncated to the model's `max_seq_length` and that no single giant string is being sent. Reducing batch size is the fastest fix before reaching for a bigger GPU.

Q: How would you size storage for one million documents embedded at 1536 dimensions in `float32`?
A: 1536 × 4 bytes = 6144 bytes per vector, so one million vectors is about 6.1 GB of raw vectors, plus ANN index overhead (HNSW graphs add a substantial multiple) and metadata. That is the number you take to capacity planning before choosing the model. Halving the dimension (e.g. via Matryoshka shortening to 768) roughly halves this raw storage.

Q: Two candidate models score within a point of each other on your hit-rate test, but one is 768-dim and the other 3072-dim. Which do you lean toward, and why?
A: The 768-dim model, because it stores four times fewer bytes per vector and does roughly four times less ANN distance work per query for essentially the same retrieval quality. The lesson's "bigger is not reliably better" applies directly: equal quality at lower width is a strictly better operating point. You would still confirm licence and language coverage before committing.

Q: For 20 fund-domain retrieval queries, why does the lesson insist on a French/English mix rather than English only?
A: Because evaluating only in English would hide the exact failure mode the lesson is teaching — that an English-strong model can collapse on French regulatory text. A mixed set surfaces whether the model retrieves French KID and `frais courants` content as well as English, which is the real production condition for a Luxembourg corpus. Done-when explicitly asks you to show how the French queries changed the model choice.

Q: Give an example of a Luxembourg-flavoured retrieval query you might include in the evaluation set.
A: Something like "Quels sont les frais courants du compartiment ?" (what are the ongoing charges of the sub-fund), an `ISIN` lookup such as "find the factsheet for `LU0123456789`", or "what is the SRI of this fund". These hit French finance vocabulary, identifier matching, and risk-indicator terminology at once. A model that handles all three on your corpus is doing the real job, not a benchmark proxy.

Q: An `ISIN` like `LU0123456789` appears in a query — is semantic embedding the right tool to retrieve documents containing it, and what is the gotcha?
A: Pure semantic embedding is poor at exact-identifier matching, because an `ISIN` is a precise 12-character code, not a concept, and tokenisers may split or blur it. The robust pattern is hybrid retrieval that adds keyword/lexical matching (e.g. BM25) so exact codes are found reliably, with vectors handling the fuzzy semantic part. Relying on embeddings alone for `ISIN` lookups is a common, silent retrieval failure.

Q: Why is naming and pinning the model version especially important under a regulated, auditable workflow like fund administration?
A: Because auditors and DORA-style controls expect reproducibility — you must be able to show which exact model produced the vectors behind a given retrieval result and answer at a given time. A pinned model name and version next to the vectors gives you that lineage; an unrecorded silent upgrade destroys it. It is the difference between "we can prove what the system did" and "we think it was probably the new model".

Q: How does embedding-model choice interact with DORA third-party / ICT risk concerns?
A: An API embedding provider is an ICT third party whose availability, data handling, and jurisdiction fall under DORA-style oversight, so depending on it for embedding makes it part of your resilience and concentration-risk picture. A local `sentence-transformers` model removes that third-party dependency for the embedding step entirely. The architect weighs the convenience of the API against the third-party governance burden it adds.

Q: What is the difference between "the model won't train on our data" and "our data stays in the EU"?
A: They are independent guarantees: a provider can promise not to train on your inputs while still processing them in a non-EU region, and conversely an EU-region deployment says nothing by itself about training use. You need both confirmed separately from current documentation for a regulated corpus. Conflating them is a common compliance mistake when evaluating API embeddings.

Q: Why does API embedding price being "per million tokens" matter for re-index budgeting specifically?
A: Because a full re-index re-embeds the entire corpus, so its cost equals total corpus tokens times the per-million-token rate — every model upgrade is a fresh bill of that size, not a one-time cost. With `text-embedding-3-small` at roughly $0.02 per million tokens versus `text-embedding-3-large` at roughly $0.13, the per-token rate directly sets how affordable re-indexing is. This is why the lesson ties model choice to re-index burden.

Q: Roughly what is the published API price difference between `text-embedding-3-small` and `text-embedding-3-large`?
A: At the prices reported in the lesson's research window, `text-embedding-3-small` is about $0.02 per million tokens and `text-embedding-3-large` about $0.13 per million tokens — large is several times more expensive per token. You should always confirm the current figure from OpenAI's pricing page rather than quoting from memory, since prices change. The order-of-magnitude gap is the point: width and quality cost real money at scale.

Q: Why does shortening a `text-embedding-3` vector with the `dimensions` parameter not reduce the API call price?
A: Because you are still sending the same input text and the model still computes the full representation before truncating — the per-token billing is on input tokens, not output width. Shortening saves you storage bytes and ANN query cost downstream, not the embedding API fee. So Matryoshka shortening is a storage/latency optimisation, not an API-cost optimisation.

Q: A French KID retrieval returns confident but wrong fee tables. How do you tell whether the embedding model or something downstream is at fault?
A: Measure retrieval in isolation first: embed the known query and check whether the correct fee-table passage appears in the top-k results before any generation. If the right passage is absent from retrieval, the embedding model (or chunking) is the culprit; if it is present but the answer is still wrong, the problem is downstream in generation. Separating retrieval from generation is exactly why the lesson measures hit-rate@k before adding an LLM.

Q: After switching from an English-only model to a multilingual one, French hit-rate@k jumps but English stays flat. What does that tell you?
A: That the old model was the bottleneck specifically for French — it was embedding French text into a space where it could not match, which is the lesson's silent-garbage failure. English being unchanged confirms the multilingual model did not regress on the language that already worked. This is precisely the evidence the Done-when criterion asks you to show: how the French queries changed the choice.

Q: Why can't you trust an embedding model's reported "average MTEB score" when your corpus is one narrow domain in two languages?
A: Because the average blends dozens of datasets, task types, and languages unlike yours, so it can be high while the model is weak on French financial retrieval specifically. Averages hide the variance that matters to you. The disciplined move is to read the retrieval-and-language-specific sub-scores for shortlisting, then validate on your own labelled queries.

Q: What is "re-indexing" and how does it differ from re-embedding?
A: Re-embedding is producing new vectors for documents (e.g. with a new model); re-indexing is rebuilding the ANN structure (such as an HNSW graph in pgvector) over those vectors so queries can search them. A model change forces both: you re-embed every document and then rebuild the index over the new vectors. They are distinct steps, and a complete strategy budgets time and resources for each.

Q: Why is the corpus's chunk size tied to the embedding model's `max_seq_length`?
A: Because if a chunk exceeds the model's token limit, the model truncates it and silently drops the overflow, so any text past the limit never gets embedded or retrieved. With a 128-token model like `paraphrase-multilingual-mpnet-base-v2` you must chunk small; with a longer-context model you can use larger chunks. Mismatching chunk size and model limit is a frequent cause of "the answer was in the document but never retrieved".

Q: How would you empirically confirm a French/English model actually embeds the two languages into a shared space?
A: Embed a known French passage and its English translation and check that they are near each other (high cosine similarity) relative to unrelated text — a genuinely multilingual model places translations close together. Then run cross-lingual retrieval: an English query should retrieve the relevant French passage and vice versa. If translations land far apart, the model is not truly bridging the languages for your corpus.

Q: What does it mean that embeddings from two different model versions are "in different spaces"?
A: It means the axes and geometry differ, so the vector for the same text is different and distances are only comparable within one version — a cosine score between a v1 vector and a v2 vector is meaningless. There is no transform you can apply at query time to make them compatible. This incompatibility is the root reason a version change demands a full re-embed rather than an incremental update.

Q: When is a local `sentence-transformers` model clearly the right call over an API, all else equal?
A: When the corpus is confidential and regulated (so sending text to a third party is a residency/DORA problem), when volume is high enough that per-token API fees dominate, or when you need full reproducibility and version control over the encoder. In those cases the operational burden of running inference locally is worth the sovereignty and cost control. The lesson's fund context pushes hard toward this corner of the triangle.

Q: When is an API embedding model the more sensible choice despite the residency cost?
A: When the corpus is small or public (so residency is not a hard constraint), when you lack GPU infrastructure or the team to maintain local inference, or when you need top-tier multilingual quality quickly and can configure an EU-residency deployment. For a low-volume internal tool over non-confidential text, the API's zero-ops convenience often wins. The decision is contextual, which is exactly why the architect tabulates the triangle.

Q: What goes wrong if you pick an English-only API model "because it topped the leaderboard" for a Luxembourg corpus?
A: French KIDs and French queries embed into a space the model represents poorly, so they silently retrieve garbage while English appears to work, masking the failure until a French user complains. On top of that, every model upgrade triggers an unbudgeted full re-index through a third-country API, raising both cost and residency risk. This double failure is the lesson's headline cautionary scenario.

Q: How does picking a smaller-dimension model help the re-index burden, not just steady-state cost?
A: Smaller vectors mean each re-embed-and-re-index pass writes fewer bytes and builds a lighter ANN graph, so the periodic full re-index is faster and cheaper to run and store. Width compounds: it taxes you every query and every rebuild. Choosing the lowest width that meets your hit-rate target reduces both the recurring and the episodic costs.

Q: What is the risk of evaluating embedding models on a tiny or unrepresentative query set?
A: Small or skewed query sets produce noisy hit-rate numbers and can crown a model that happens to do well on those few queries but fails broadly — overfitting your decision to the test. The lesson asks for 20 queries spanning French/English and real fund tasks precisely to make the signal trustworthy. A handful of English-only queries would reproduce the very blind spot you are trying to avoid.

Q: Why might a model that is excellent at English semantic similarity still be wrong for cross-lingual retrieval?
A: Because monolingual quality says nothing about whether French and English are mapped into the same shared space — a model can be great within English yet place French translations far from their English equivalents. Cross-lingual retrieval needs a model explicitly trained for multilingual alignment. Testing only same-language similarity would never expose this gap.

Q: What does "thin frameworks, strong pipelines" imply for how you treat the embedding step?
A: That the embedding step is a governed pipeline stage you own — versioned, measured, re-runnable — rather than a black box buried inside a framework's defaults. You should know exactly which model, which version, which dimension, and which residency posture is in play, and control them. Leaning on an undocumented framework default for embeddings is how the architect-level decisions get made by accident.

Q: How do you decide the final `k` (top-k) when measuring hit-rate for model comparison?
A: Pick a `k` that reflects how many passages your downstream generation step will actually consume (e.g. top 5 or top 10), and keep it fixed across all candidate models so the comparison is fair. Measuring hit-rate@k at the k you will use in production makes the number actionable rather than abstract. Changing k between models would make the comparison meaningless.

Q: A colleague proposes upgrading the embedding model the night before a regulatory reporting run. What is your objection?
A: That an embedding-model change is a full re-embed plus re-index of the corpus, not a quick swap, and doing it un-tested before a reporting run risks silently degraded retrieval with no audit trail of which model produced which result. The safe path is a planned, version-pinned re-index in the orchestrator with hit-rate validation, scheduled away from a critical run. The "quick upgrade" framing is the exact misconception the lesson dispels.

Q: What metadata, beyond the vector itself, should every record in the vector store carry for a regulated RAG system?
A: At minimum the embedding model name and version that produced the vector, the source document id and a span/citation locator, the language, and a timestamp — so any answer is traceable to its model, source, and time. The model-version pin enables re-index detection and audit; the source span enables citation. Storing the bare vector with no provenance is what makes a regulated system unauditable.

Q: Why does the lesson tie embedding-model versioning to "a pipeline design issue" rather than a model-quality issue?
A: Because the consequence of changing models is operational — a full re-index, a transition strategy, budget, and lineage — which is pipeline design, not model accuracy. Even if the new model is strictly better, mishandling the rollout corrupts retrieval. So the architect's main work around versioning is designing the re-embedding pipeline, not just judging which model is best.

Q: How would you implement re-embedding as a Dagster job over your existing vector-store asset?
A: Model the vector store as an asset whose upstream dependency includes the pinned embedding-model identifier, so changing that identifier (or arriving documents) marks the asset stale and triggers a materialisation that re-embeds and re-indexes the corpus. The job is observable, retryable, and logged like any other, and can implement dual-write or cutover in its logic. This turns a model upgrade into a tracked, auditable run rather than a manual script.

Q: What is the relationship between "freshness" of documents and re-embedding?
A: When source documents change, the affected chunks must be re-embedded and re-indexed so retrieval reflects the new content — that is incremental re-embedding driven by document updates, distinct from a model-driven full re-index. The same pipeline that handles model upgrades should also handle document freshness. Both are triggers your re-embedding strategy must enumerate.

Q: Why might you keep the old index live (dual-write) during a model migration rather than cutting over immediately?
A: To compare retrieval quality between old and new on real traffic, to keep a working fallback if the new model regresses, and to avoid any downtime while the new index builds. The cost is double storage and write effort during the window. For a regulated workflow that can't tolerate a retrieval outage, dual-write's safety often justifies the temporary expense.

Q: What is the danger of letting a framework silently default your embedding model?
A: You may end up sending confidential text to a default API endpoint (a residency breach), or using an English-centric default on a French corpus (silent retrieval failure), without ever making a conscious decision. Defaults also tend to drift across framework versions, breaking the version-pin discipline. The architect must explicitly set and record the model, never inherit it from a framework default.

Q: How do you reconcile "shortlist with MTEB" with "domain fit beats leaderboard rank" — aren't they in tension?
A: They are sequential, not contradictory: MTEB efficiently narrows thousands of models to a handful that plausibly fit on task type, language, dimension, and licence, and then your own domain retrieval set picks the winner among those. The leaderboard does coarse filtering; your data does the final selection. Using either step alone is the mistake — MTEB-only ignores domain fit, and corpus-only would require testing an unmanageable number of models.

Q: What is the storage and latency consequence of choosing `text-embedding-3-large` at full 3072 dimensions for a multi-million-document corpus?
A: Each vector is 3072 × 4 = 12288 bytes (about 12 KB), so a few million documents run to tens of gigabytes of raw vectors plus a large HNSW index, and every query does the maximum distance work, raising latency. Unless your hit-rate measurably needs that width, it is an expensive default. Shortening via the `dimensions` parameter or choosing a 768-dim model can cut this dramatically at little quality cost.

Q: Why is "licence + price + dimensions recorded at shortlist time" framed as part of doing the lesson properly?
A: Because these three attributes frequently override raw quality — an unusable licence, an unaffordable per-token price, or an unacceptable storage width can eliminate the top-scoring model — and discovering them late is costly. Recording them up front (Do step 1) makes the tradeoff explicit and decision-ready. It is the difference between an engineering choice and a gamble.

Q: A stakeholder says "embeddings are just numbers, residency rules don't apply." How do you correct them?
A: By explaining that to produce those numbers the source text was transmitted to and processed by the embedding model, so residency applies at embed time to the input regardless of the output's form — the lesson states residency applies to embeddings too. Additionally, vectors can partially disclose their source via inversion, so even the stored output is not risk-free. The "just numbers" framing is a compliance trap.

Q: How would you present the final model decision so it satisfies the lesson's Done-when criteria?
A: As a side-by-side table of quality (hit-rate@k on your French/English set), cost (per-million-token price or local compute), and residency/processing terms, leading to a chosen model that ships with a stated re-embedding strategy (trigger, batch job, dual-write or cutover) and an explicit data-residency position. You would also include a short argument for why MTEB rank alone would have misled, and how the French queries changed the choice. That package is precisely what Done-when asks for.

Q: Why is it risky to assume a multilingual model covers a specific language well just because it is "multilingual"?
A: Because multilingual coverage is uneven — a model trained on 50 languages may represent French strongly but a low-resource language weakly, so "multilingual" is not a guarantee for your particular language. You confirm by testing cross-lingual retrieval on your actual French/English queries. Trusting the "multilingual" label without measurement reintroduces the silent-failure risk in a subtler form.

Q: What is the simplest experiment to show a colleague that "bigger dimensions" did not help your corpus?
A: Run hit-rate@k on the same French/English query set for a high-dimension and a low-dimension candidate, holding everything else fixed, and show the scores are within noise while the high-dimension model costs several times more storage and query work. The numbers make "bigger is not reliably better" concrete. This is the data-driven version of the lesson's claim, on your corpus rather than a benchmark.

Q: How does sending text to an embedding API differ, residency-wise, from sending it to a chat completion API?
A: It does not differ in the way that matters: both transmit your text to the provider for processing, so embeddings carry the same residency and data-handling exposure as chat calls. People mentally file embeddings as "preprocessing" and forget this, which is the gap the lesson closes. The same deployment-region and retention questions apply to the embedding endpoint.

Q: What does pinning the embedding model version enable when you later need to debug a bad retrieval result?
A: It lets you reproduce the exact vector for the query and documents by re-running the same pinned model, so you can confirm whether the retrieval was the encoder's fault and compare against a candidate replacement deterministically. Without the pin you cannot be sure which model produced the stored vectors, so debugging becomes guesswork. Reproducibility is the practical payoff of the version pin.

Q: Why might two different PDF parsers produce different embedding quality even with the same model?
A: Because the parser determines the text that reaches the embedder — garbled tables, dropped columns, or merged headers produce noisy chunks that embed poorly no matter how good the model is. On structured documents like KIDs, parser choice can move retrieval quality more than the model does. So evaluating the embedding model fairly requires holding the parser fixed and clean.

Q: In the cost/latency/sovereignty triangle, which corner does a high-throughput, latency-sensitive internal search tool most want to protect, and how?
A: Latency, typically by running a smaller local model (lower dimension, fast inference) close to the data so there is no network round-trip per query, and by keeping vectors narrow to speed ANN search. You accept the local-ops cost to keep responses fast. The triangle framing forces you to name which corner the use case cannot sacrifice before choosing the model.

Q: A model upgrade is proposed that improves MTEB rank by two points. What questions decide whether it is worth a full re-index?
A: Does it actually improve hit-rate@k on our own French/English fund queries; what is the re-index cost in compute or API tokens; does it keep our licence and residency posture; and can the orchestrator run the migration safely with rollback? A two-point MTEB gain that doesn't move your hit-rate, or that breaks residency, is not worth a full re-index. The decision is driven by your data and operational cost, not the leaderboard delta.

Q: What is the consequence of forgetting to truncate inputs to `max_seq_length` before embedding long KID text?
A: The model truncates silently to its limit (e.g. 128 tokens), so anything beyond that — often the fee or SRI detail lower on the page — is never embedded and therefore never retrievable, while no error warns you. You either chunk to fit the limit or choose a longer-context model. This silent truncation is a classic cause of "the data is in the corpus but the system can't find it".

Q: Why does the lesson place embedding-model choice upstream of, and decisive for, everything else in a RAG system?
A: Because retrieval quality is capped by how well your vectors represent meaning, and no chunking trick, reranker, or LLM can recover information the embedding failed to place near its query. The model also fixes storage width, cost, residency, and re-index burden for the whole system. Get this one decision wrong and you pay for it at every query and every rebuild — which is why it is an architect decision.

Q: Summarise the four artefacts that prove you have truly completed lesson 6.6.1.
A: A side-by-side quality/cost/residency comparison of one local multilingual model and one API model with dimensions, licence, and price recorded; a chosen model carrying a stated re-embedding strategy (trigger, batch job, dual-write or cutover) and a data-residency position; an explanation of why MTEB rank alone would have been a bad selection method for this corpus; and evidence of how the French/English queries changed the model choice versus English-only evaluation. Together these are exactly the lesson's Done-when criteria, and they reflect treating the embedding choice as the architect decision it is.

Q: How does the embedding dimension interact with an HNSW index's memory footprint specifically?
A: HNSW stores the full vectors plus a multi-layer graph of neighbour links, so memory grows with both the number of vectors and the dimension that fixes each vector's byte size — wider vectors enlarge the dominant vector-storage term directly. A pgvector HNSW index over 3072-dim vectors therefore needs far more RAM than one over 768-dim vectors at the same row count. This is another reason "bigger dimension" is not free: it inflates index memory, not just raw storage.

Q: If an API offers a cheaper "batch" embedding tier, when is it appropriate for a re-index and when not?
A: A batch tier (e.g. roughly half price but asynchronous, completed within a longer window) suits a full corpus re-index, which is a bulk offline job where latency does not matter, so you take the discount. It is wrong for live query-time embedding, where a user is waiting and you need the synchronous endpoint. Matching the right tier to the job — batch for re-index, sync for queries — controls API cost without hurting user-facing latency.

Q: Give a concrete example of how a model licence could block an otherwise ideal pick for a fund product.
A: A high-scoring open-weight model released under a research-only or non-commercial licence cannot legally power a commercial fund-administration service, so even a top hit-rate disqualifies it; you would need a permissively licensed alternative such as one under Apache-2.0 or MIT. Some licences also impose attribution or use restrictions that conflict with regulated deployment. Recording the exact licence at shortlist time (Do step 1) is what catches this before you build on it.

Q: Why does exact-identifier retrieval (ISIN, LEI, SWIFT BIC) argue against relying on embeddings alone, beyond just ISINs?
A: Codes like an `ISIN`, a 20-character `LEI`, or an 8/11-character SWIFT `BIC` are exact-match tokens whose meaning is identity, not semantics, and embedding models blur or split them so cosine ranking is unreliable for precise lookups. The robust answer is hybrid retrieval pairing the vector search with lexical/keyword matching so the exact code is found deterministically. This generalises the `ISIN` gotcha: any structured financial identifier needs a lexical path, not embeddings alone.
