# Phase 4 · 2.3.1 + 1.8.10 Log-based CDC (Debezium) — 150 self-test questions

Q: What is change data capture (CDC) and why is log-based CDC preferred over query-based polling in a fund estate?
A: CDC is the practice of capturing every row-level insert, update, and delete from a source database and emitting them as an ordered stream of change events. Log-based CDC reads the database transaction log (the Postgres WAL) directly, so it captures every change — including deletes and rapid updates a poll would miss — without adding load via repeated `SELECT` queries or needing `updated_at` columns. In a transfer-agency system this means you can mirror the source of record completely and without touching fragile core application code.

Q: What is the PostgreSQL write-ahead log (WAL) and why is it the foundation of Debezium's Postgres connector?
A: The WAL is an append-only log to which Postgres writes every change before applying it to data files, guaranteeing durability and crash recovery. Because every committed change is recorded there in commit order, the WAL is the authoritative changelog Debezium reads to reconstruct row-level events. Logical decoding turns those low-level WAL records into a readable stream of logical changes the connector can consume.

Q: What is logical decoding in PostgreSQL?
A: Logical decoding is the Postgres mechanism that extracts the WAL's physical change records and presents them as a stream of logical, row-level changes through an output plugin and a replication slot. Unlike physical (streaming) replication, which ships raw byte-level WAL, logical decoding gives you per-table inserts/updates/deletes that a consumer like Debezium can map to change events. It is the bridge between the internal WAL format and a usable change stream.

Q: What is the `pgoutput` logical decoding plugin and why does Debezium default to it?
A: `pgoutput` is the logical decoding output plugin built into PostgreSQL 10 and later, maintained by the Postgres community and always present, so no extra extension needs to be compiled or installed. Debezium defaults to it because it requires zero extra installation on the database server, which matters when a regulated DBA team will not install third-party C extensions. The alternative `decoderbufs` plugin must be installed separately.

Q: What is a Postgres logical replication slot and why must Debezium keep one?
A: A replication slot is a server-side object that tracks the last WAL position a logical consumer has confirmed, and it tells Postgres how far back WAL must be retained. Debezium creates and holds a slot (default name configurable via `slot.name`) so the database does not discard WAL the connector still needs after a restart. The gotcha: if the connector is stopped for a long time the retained WAL grows and can fill the disk, so an abandoned slot must be dropped deliberately.

Q: What configuration property names the Debezium replication slot, and why should you set it explicitly?
A: The property is `slot.name`, defaulting to `debezium`. You should set it explicitly (and uniquely per connector) because each logical slot can be consumed by only one connector at a time, and two connectors sharing a slot name will conflict. A meaningful name like `dbz_ta_orders` also makes it obvious which connector a slot belongs to when an operator later has to drop an orphaned one.

Q: What is a Postgres publication and what does Debezium's `publication.autocreate.mode` control?
A: A publication is a Postgres object that defines the set of tables whose changes are exposed through `pgoutput`. `publication.autocreate.mode` controls whether Debezium creates the publication for you and how: `all_tables` publishes every table, `filtered` (the typical choice) publishes only the tables matching the connector's include/exclude lists, and `disabled` requires a DBA to create it manually. In a locked-down fund database you usually pre-create a `filtered` publication so the connector account needs no `CREATE` rights.

Q: What does the `table.include.list` property do and how is a table named in it?
A: `table.include.list` is a comma-separated allowlist of fully qualified table identifiers in `schemaName.tableName` form (regex-matched) that the connector should capture; everything else is ignored. For example `public.orders,public.nav` captures only those two tables. Restricting capture to exactly the tables you need keeps the publication small and avoids streaming sensitive tables you have no business mirroring.

Q: What is REPLICA IDENTITY in Postgres and why does it matter for CDC?
A: REPLICA IDENTITY is a per-table setting that controls how much of the old row is written to the WAL for `UPDATE` and `DELETE` operations, which directly determines what Debezium can put in the event's `before` field. The default only logs the primary-key columns of the old row, so non-key old values are absent on updates/deletes. Getting this wrong means your downstream `before` image is incomplete, which can break audit reconstruction.

Q: What are the four REPLICA IDENTITY settings and when would you use `FULL`?
A: They are `DEFAULT` (logs only primary-key columns of the old row), `FULL` (logs the entire old row), `NOTHING` (logs no old-row info), and `USING INDEX` (logs columns of a named unique index). You set `REPLICA IDENTITY FULL` with `ALTER TABLE ... REPLICA IDENTITY FULL` when downstream consumers need the complete `before` image — for example to compute which fields actually changed in a NAV record. The trade-off is more WAL volume per change.

Q: A table has `REPLICA IDENTITY DEFAULT` and you see `before` is null for an update — is that a bug?
A: With `DEFAULT`, Postgres only logs the primary-key columns of the old row, so for an update the `before` field will contain just the key and null for unchanged non-key columns it could not log — and for tables with no primary key the `before` image can be entirely absent. It is not a Debezium bug; it is the source REPLICA IDENTITY limiting what reaches the WAL. The fix is `ALTER TABLE ... REPLICA IDENTITY FULL` if a full before-image is required.

Q: What is Kafka Connect and what role does it play in a Debezium deployment?
A: Kafka Connect is Kafka's framework for running reusable connectors that move data into and out of Kafka, providing the worker runtime, REST API, configuration management, and offset storage. Debezium connectors are Kafka Connect source connectors, so Connect is the operational host that starts, restarts, and rebalances them. This means you never run Debezium as a standalone process; you submit a connector config to a Connect cluster.

Q: What is the difference between a Connect connector and a task?
A: A connector is the logical unit you configure (one Debezium Postgres connector per database), while tasks are the units of parallel work the connector spawns to actually do the copying. The Debezium Postgres connector is constrained to a single task because logical decoding from one slot is inherently sequential, so `tasks.max` greater than one has no effect for it. Other connectors (like sinks) can split work across many tasks.

Q: Where does Kafka Connect store a Debezium connector's offsets in distributed mode?
A: In distributed mode, Connect stores source offsets in an internal compacted Kafka topic, configured by `offset.storage.topic`, alongside `config.storage.topic` and `status.storage.topic`. For a Debezium Postgres source the stored offset is essentially the last confirmed WAL LSN plus snapshot position. Because the offset lives in Kafka, any worker in the cluster can resume the connector after a crash, not just the original host.

Q: Why does Debezium store offsets in Kafka rather than relying solely on the Postgres replication slot?
A: The replication slot tracks the WAL retention boundary on the database, but Connect's offset topic records exactly which LSN the connector last committed to Kafka and whether a snapshot was in progress. Together they let the connector resume precisely: the slot guarantees the WAL is still there, and the offset says where to resume reading. Relying on the slot alone would not capture snapshot progress or the consumer's committed position.

Q: What is the change event envelope and what are its top-level fields?
A: The change event envelope is the structured payload Debezium emits per change, with the key fields `before`, `after`, `source`, `op`, and `ts_ms`. `before` and `after` hold the row state pre- and post-change, `source` carries metadata, `op` is the operation code, and `ts_ms` is the connector's processing timestamp. This consistent shape lets one downstream consumer handle inserts, updates, and deletes uniformly.

Q: What does the `op` field contain and what does `op:"c"` mean?
A: `op` is a single-character operation code identifying the kind of change. `c` means create (insert), `u` means update, `d` means delete, `r` means read (a row emitted during a snapshot), `t` means truncate, and `m` means a logical decoding message. So `op:"c"` is a freshly inserted row, with `before` null and `after` populated.

Q: For an insert, an update, and a delete, what do `before` and `after` contain?
A: For an insert (`op:"c"`), `before` is null and `after` holds the new row. For an update (`op:"u"`), `before` holds the old row image (subject to REPLICA IDENTITY) and `after` holds the new row. For a delete (`op:"d"`), `before` holds the deleted row and `after` is null. Reading these two fields together tells a consumer exactly what changed.

Q: What does `op:"r"` signify and when do you see it?
A: `op:"r"` (read) marks a row emitted during a snapshot rather than from the live transaction log, with `after` populated and `before` null. You see a burst of `r` events when the connector first starts and performs its initial snapshot of existing rows. Distinguishing `r` from `c` lets downstream logic know a row is historical backfill, not a new live insert.

Q: What is in the `source` block of a change event and why is it important for auditing?
A: The `source` block carries provenance metadata: the Debezium version, connector name, database/schema/table, the WAL LSN, transaction id, commit timestamp, and a `snapshot` flag. This is what lets you prove to an auditor exactly which source table and WAL position a given lakehouse row came from. For fund data it ties each NAV or order event back to a precise point in the source of record.

Q: What is the difference between `ts_ms` in the envelope and the commit timestamp in `source`?
A: The envelope `ts_ms` is when the Debezium connector processed the event, whereas `source.ts_ms` is when the change was committed in the source database. They differ by the connector's processing lag, so during a backlog catch-up `ts_ms` can be much later than the source commit time. For event-time analytics you should use the source commit time, not the connector processing time.

Q: What is a tombstone event and why does Debezium emit one after a delete?
A: A tombstone is a record with a non-null key but a null value (null payload) emitted immediately after a delete event. It exists so Kafka log compaction can physically remove all records for that key, since compaction retains the latest value per key and a null tells it to drop the key entirely. Without the tombstone, a compacted topic would keep the last non-null value of a deleted row forever.

Q: What does the `tombstones.on.delete` property control and what is its default?
A: `tombstones.on.delete` controls whether Debezium emits the extra null-value tombstone record after each delete; it defaults to `true`. You set it to `false` when your downstream consumer treats the `op:"d"` event itself as the delete signal and does not rely on compaction semantics. Leaving it `true` is correct when the topic is compacted and must garbage-collect deleted keys.

Q: A delete produces two records on the topic — is that expected?
A: Yes, with `tombstones.on.delete` at its default `true`, a delete yields a normal delete change event (`op:"d"`, `before` populated, `after` null) followed immediately by a tombstone (same key, null value). The first conveys the delete and the old row image; the second enables compaction to purge the key. A downstream merge job must treat both correctly and not mistake the tombstone for a malformed record.

Q: What is an initial snapshot and why is it necessary?
A: An initial snapshot is the connector's first-pass read of all existing rows in the captured tables, emitted as `op:"r"` events before streaming begins. It is necessary because the WAL only contains changes from a point in time forward, so without a snapshot the consumer would have no record of rows that existed before CDC started. The snapshot plus subsequent streaming gives a complete picture of the table.

Q: What does `snapshot.mode = initial` do, and what happens on a normal restart?
A: `initial` (the default) performs a full snapshot only on the very first start when no offsets exist, then switches to streaming WAL changes. On a normal restart where offsets already exist, the snapshot is skipped entirely and the connector resumes streaming from the stored LSN. This is why a routine restart does not re-read the whole table.

Q: What is the difference between `snapshot.mode = initial` and `initial_only`?
A: Both perform a full snapshot on first start, but `initial` then continues into streaming the ongoing WAL changes, whereas `initial_only` completes the snapshot and stops without streaming. You use `initial_only` for a one-time bulk migration or backfill where you do not want continuous replication. For live CDC you want `initial` so the stream keeps flowing after the snapshot.

Q: What does `snapshot.mode = no_data` (formerly `schema_only`) do?
A: `no_data` captures the table schema and starts streaming from the current WAL position without emitting `op:"r"` read events for existing rows. It replaces the deprecated `schema_only` value. You choose it when you only care about changes from now on and already have the historical data loaded by other means, so you skip the cost of a full snapshot.

Q: What does `snapshot.mode = always` do and what is the downstream requirement?
A: `always` performs a full snapshot on every connector start, re-emitting all rows as `op:"r"` regardless of stored offsets. Because the same rows are re-read on each restart, downstream consumers must be idempotent — typically an upsert keyed on the primary key — or they will duplicate data. It is rarely used in production except to forcibly re-baseline a target.

Q: What is `snapshot.mode = when_needed` and when does it re-snapshot?
A: `when_needed` snapshots only when it determines one is required — for example when stored offsets are missing or point to a WAL/LSN position the server has already purged. In normal operation with valid offsets it just resumes streaming. It is a safety mode that self-heals after the connector falls behind far enough that the needed WAL is gone.

Q: What is an incremental snapshot and what problem does it solve over a blocking initial snapshot?
A: An incremental snapshot reads existing rows in bounded chunks while the connector continues streaming live WAL changes concurrently, instead of blocking all streaming until a full snapshot finishes. This avoids long lock/latency windows on huge tables and lets you start capturing live changes immediately. It is the modern approach for re-snapshotting a large transfer-agency table without stalling the pipeline.

Q: How is an ad-hoc incremental snapshot triggered in Debezium?
A: You send an `execute-snapshot` signal through Debezium's signaling mechanism, typically by inserting a row into a configured signaling table (named via `signal.data.collection`) or sending a signal over a Kafka signal topic. The signal names the data collections (tables) to snapshot. This lets an operator re-snapshot a specific table on demand without redeploying or restarting the connector.

Q: What is the signaling table and why does Debezium use it?
A: The signaling table is a small auxiliary table the connector watches (configured by `signal.data.collection`) into which you insert command rows such as `execute-snapshot` or `pause`/`resume`. Because the connector already watches the database via CDC, signals delivered as table rows are processed in WAL order alongside data changes, giving deterministic, ordered control. It is how you trigger incremental snapshots and other runtime actions safely.

Q: Name two events that force a re-snapshot in a Debezium Postgres deployment.
A: A re-snapshot is forced when the connector's stored offsets are lost or deleted (so it cannot resume from a known LSN), and when the replication slot is dropped or the required WAL has been purged so the prior position no longer exists. In `when_needed` mode the connector detects these and re-snapshots automatically; in `initial` mode a wiped offset topic on first-like start triggers a fresh full snapshot. Either way the consequence is re-reading existing rows as `op:"r"`.

Q: How does an incremental snapshot avoid conflicting with concurrent live changes to the same row?
A: During an incremental snapshot Debezium reads a chunk of rows and buffers them, then deduplicates against any live change events for the same primary key that arrive in the same window, so the most recent state wins. This windowing/deduplication means a row updated by live traffic while being snapshotted is not overwritten by the older snapshot read. The result is a consistent merge of snapshot reads and streamed changes.

Q: What per-table ordering guarantee does Debezium provide and what is its mechanism?
A: Debezium guarantees that change events for a single table arrive in commit order, achieved by keying each event by the table's primary key and routing all of a table's events to one topic partition (since the key hashes to the same partition). Kafka preserves order within a partition, so all changes to one row are strictly ordered. Cross-table or cross-partition global ordering is not guaranteed.

Q: Why is there no global total ordering across all tables in a Debezium stream, and how do you reconstruct transactions?
A: Because each table typically goes to its own topic and events are partitioned by key for parallelism, there is no single ordered stream across all tables. To reconstruct a multi-table transaction you enable Debezium's transaction metadata, which emits begin/end markers and a per-event transaction id on a transaction topic. Consumers can then group events that share a transaction id to see a consistent commit boundary.

Q: What is a schema-change event and why does Debezium emit one?
A: A schema-change event records a DDL change to a captured table — for example a new column added to the `nav` table — so downstream consumers know the structure of subsequent data events evolved. Debezium emits these in order relative to the data events, so a consumer sees the schema change before rows shaped by it. This keeps the consumer's understanding of the table layout synchronized with the source.

Q: Why does schema-change ordering relative to data events matter for a NAV pipeline?
A: If a column is added to the `nav` table and you start receiving data events carrying that new field, a consumer that processed the data before learning of the schema change could mis-parse or drop the new field. Debezium orders the schema-change event ahead of the data events that depend on it, so the consumer updates its schema first. This prevents silent data loss when fund schemas evolve mid-stream.

Q: What is transaction metadata in Debezium and which fields identify a transaction?
A: When transaction metadata is enabled, Debezium writes `BEGIN`/`END` marker events to a transaction topic and stamps each data event with a `transaction` block containing the transaction `id`, a total event count, and per-data-collection counts. The `id` lets consumers correlate every change that committed together. This is how you reassemble an atomic multi-row business operation, like a subscription that touches `orders` and `nav` together.

Q: What is the transactional outbox pattern and what problem does it solve?
A: The outbox pattern has a service write its business state change and an "event" row into an outbox table within the same local database transaction, instead of trying to update the database and publish to a message broker in one distributed transaction. This solves the dual-write problem: because both writes are one atomic DB commit, you can never publish an event without the matching state change or vice versa. A separate process then relays the outbox rows to the broker.

Q: How does outbox-via-CDC relay the outbox table, and why is it reliable?
A: Instead of polling the outbox, you point Debezium at the outbox table so every inserted outbox row becomes a CDC change event captured from the WAL. Because the outbox insert and the business change committed in the same transaction, CDC guarantees the event is published exactly when (and only when) the business state actually persisted. It turns the outbox into a guaranteed, ordered event source with no extra polling component.

Q: What is the Debezium Outbox Event Router and what is its transform class name?
A: The Outbox Event Router is a single-message transform (SMT) that reshapes raw outbox-table change events into clean domain events on the right topics; its class is `io.debezium.transforms.outbox.EventRouter`. Rather than emitting the literal outbox row structure, it extracts the payload and routes by aggregate type. You add it to the connector config as a transform so consumers see business events, not the outbox table's schema.

Q: What columns does the Outbox Event Router expect by default in the outbox table?
A: By default it expects `id` (a UUID, not null), `aggregatetype` (varchar, not null), `aggregateid` (varchar, not null), `type` (varchar, not null), and `payload` (typically `jsonb`). `aggregatetype` selects the destination topic, `aggregateid` becomes the event key, and `payload` becomes the message body. If your column names differ you remap them via the SMT's field-mapping options rather than renaming database columns.

Q: How does the Outbox Event Router decide which topic an event goes to?
A: It routes by the value of the `aggregatetype` column (overridable via the route-by-field option), so all events sharing an aggregate type land on the same topic. By default the topic name is `outbox.event.<aggregatetype>`. This lets one outbox table feed many logically separate event topics — for example `outbox.event.order` and `outbox.event.nav` from a single table.

Q: Why does the outbox event's key matter, and what becomes the key by default?
A: By default the `aggregateid` column becomes the Kafka message key, which means all events for the same aggregate (e.g. one order) hash to the same partition and stay ordered. Per-aggregate ordering is essential so that consumers see an aggregate's events in the sequence they were produced. Choosing the aggregate id as key is what gives you correct ordering without a global lock.

Q: Why use the outbox pattern instead of capturing the business tables directly with CDC?
A: Direct CDC of business tables couples your event contract to internal table structure, leaks physical schema, and forces consumers to interpret raw row changes. The outbox lets the application emit a deliberately shaped domain event (the `payload`) with a stable contract, decoupled from how the data is physically stored. You get clean, versioned events while still inheriting CDC's transactional guarantees.

Q: What is the difference between merge (upsert) and append-plus-dedupe when landing change events into Iceberg?
A: Merge (upsert) applies each change against the target Iceberg table by primary key — inserting new keys, overwriting updated ones, and deleting on tombstones — so the table always holds the current state mirroring the source. Append-plus-dedupe instead writes every change event as a new row and resolves the latest state at query time (or in a later compaction) by picking the highest version per key. Merge keeps a clean current-state table; append keeps full history and defers reconciliation.

Q: When landing CDC into Iceberg with an upsert, how do you handle a delete event and its tombstone?
A: For an `op:"d"` event you issue a delete (or a soft-delete marker) against the Iceberg row matching that primary key, removing it from current state. The following tombstone record carries the same key with a null value and should be ignored by the merge logic since the delete was already applied — treating the tombstone as data would error or no-op. Mishandling this is a common cause of "deleted rows reappear" bugs.

Q: Why might append-plus-dedupe be safer than in-place merge for an auditable fund lakehouse?
A: Append-plus-dedupe never loses prior versions, so you retain a complete, immutable history of every change with its source LSN and timestamp — exactly what an auditor wants to reconstruct point-in-time NAV. A pure upsert overwrites old values and keeps only current state. With append you can always derive current state via a dedupe view while still proving how a value changed over time.

Q: In an append-plus-dedupe model, how do you compute current state per primary key?
A: You select, per primary key, the change event with the highest ordering value — typically the source WAL LSN or the source commit `ts_ms`, with the operation type so a delete wins over an earlier update. A window function like `ROW_NUMBER() OVER (PARTITION BY pk ORDER BY lsn DESC)` keeping rank one yields current state. Ordering by the source LSN (not arrival time) is what guarantees correctness under out-of-order processing.

Q: Why must dedupe ordering use the source LSN/commit time rather than the Kafka arrival time?
A: Arrival/processing time can be reordered by retries, rebalances, or parallel consumers, so two updates to the same row might be processed out of source order. The source WAL LSN (or `source.ts_ms`) reflects the true commit order in the database, so using it as the dedupe key guarantees the latest source change wins regardless of processing jitter. Using arrival time risks resurrecting a stale value.

Q: Why is Iceberg's snapshot/ACID model a good fit for landing CDC merges?
A: Iceberg gives atomic table commits and snapshot isolation, so a batch merge of change events either fully applies or not at all, and readers never see a half-applied state. Its hidden partitioning and metadata also make per-key upserts and time-travel queries efficient. For CDC this means a crashed merge job leaves the table consistent and you can compare a specific Iceberg snapshot against the source.

Q: The Debezium Postgres connector ignores `tasks.max = 4` — why?
A: The Postgres connector reads from a single logical replication slot, and logical decoding is inherently a single ordered stream, so it always runs exactly one task no matter what `tasks.max` is set to. Parallelism for Postgres CDC comes from partitioning downstream topics, not from multiple source tasks. Setting `tasks.max` higher simply has no effect for this connector.

Q: Postgres disk usage is climbing steadily after deploying Debezium — what is the first thing to check?
A: Check the replication slot's retained WAL: query `pg_replication_slots` for the slot's `restart_lsn` and the lag, because an inactive or lagging slot forces Postgres to keep WAL it cannot recycle. If the connector is down or far behind, WAL accumulates until the disk fills. Resolve by restarting the connector to advance the slot, or drop the orphaned slot if the connector is gone for good.

Q: Why can a low-traffic captured table cause unbounded WAL growth, and how do heartbeats fix it?
A: Debezium only advances its confirmed flush LSN when it processes events for the captured tables, but Postgres WAL is shared across all tables; if your captured table is quiet while other tables churn, the slot never confirms newer LSNs and WAL is retained. Heartbeat messages, enabled via `heartbeat.interval.ms`, are periodic events Debezium emits even when the captured tables are idle, periodically pushing the connector to flush a newer LSN. This unblocks WAL recycling without needing real changes to the captured table.

Q: After restarting the Debezium connector mid-stream, how does it know where to resume?
A: On restart, Connect reads the connector's last committed offset from the `offset.storage.topic`, which records the last processed WAL LSN and whether a snapshot was in progress. The connector then asks Postgres to stream from that LSN via the still-present replication slot. Because the slot retained the WAL and the offset records the position, it resumes exactly where it left off with no gap.

Q: An auditor asks how you prove the lakehouse copy is complete after a connector crash. What is the argument?
A: Completeness rests on three guarantees: the initial snapshot captured all pre-existing rows, the WAL plus a durable replication slot captured every subsequent change with no gaps, and the offset topic let the connector resume from the exact last-committed LSN after the crash so no change was skipped or required re-reading data lost from WAL. You then back it with a row-by-row reconciliation showing zero divergence. Together these prove no change between snapshot and now was missed.

Q: Why is at-least-once, not exactly-once, the realistic delivery model for Debezium, and how do you cope?
A: Exactly-once would require atomic coordination between committing the read position and the Kafka write, which Debezium does not guarantee across a crash, so a replayed window after restart can duplicate recently emitted events. You cope by making the sink idempotent: upsert by primary key so reapplying a change is a no-op, or append and dedupe by source LSN. This turns at-least-once into effectively-once at the table level.

Q: Your row-by-row comparison after a restart shows the Iceberg table has rows Postgres deleted. What likely went wrong?
A: The merge job probably failed to apply delete events: either it ignored `op:"d"` records, or it mistook the following null-value tombstone for a malformed message and dropped the actual delete, or REPLICA IDENTITY left it unable to identify the key to delete. The fix is to handle `op:"d"` as a delete keyed by `before`'s primary key and to skip (not error on) tombstones. Then re-reconcile to confirm zero divergence.

Q: How would you run a full row-by-row comparison to prove Iceberg equals Postgres?
A: Quiesce or pick a consistent cutoff (e.g. a known source LSN), then compare the dedup'd current-state view of the Iceberg table against the Postgres tables key by key — for example by computing a per-row hash of all columns on both sides and full-outer-joining on the primary key to surface any mismatched, missing, or extra rows. Zero rows in the symmetric difference proves equivalence. Doing this after a deliberate restart is exactly the "Done when" check for this lesson.

Q: What does `op:"t"` represent and why does TRUNCATE need special handling?
A: `op:"t"` is a truncate event, signalling that the whole table was emptied by a `TRUNCATE` rather than per-row deletes. It needs special handling because it carries no per-row `before` image, so a merge job must interpret it as "delete all current rows for this table," not as a single-row change. Ignoring truncate events leaves stale rows in the lakehouse that no longer exist in the source.

Q: What is `op:"m"` and when does it appear?
A: `op:"m"` is a message event produced from a Postgres logical decoding message — for example one emitted via `pg_logical_emit_message()` — rather than from a normal row change. It lets applications inject out-of-band markers into the change stream. Most pipelines never see these unless the application deliberately emits logical messages.

Q: Why does dropping and recreating the replication slot risk data loss?
A: A logical replication slot only retains WAL from its current position forward; when you drop it, Postgres can recycle the WAL it was holding, and a freshly created slot starts from the current LSN. Any changes that occurred while no slot existed are gone from the WAL and will not be streamed. The safe recovery is a re-snapshot, because streaming alone cannot recover the lost window.

Q: A consumer renames a column in Postgres and downstream parsing breaks. How does Debezium's schema handling help?
A: Debezium emits a schema-change event capturing the DDL before the affected data events, and each change event carries its own schema (or a schema-registry reference), so consumers can adapt to the new structure in order. The break usually means the consumer is not honoring the evolved schema or the schema registry compatibility was bypassed. Properly consuming schema-change events keeps the consumer's view synchronized with the source.

Q: Why does Debezium key change events by primary key, and what breaks if a table has no primary key?
A: Keying by primary key routes all changes for a row to one partition (preserving order) and lets compaction and upserts target the right record. A table with no primary key and `REPLICA IDENTITY DEFAULT` produces events with null/empty keys, so ordering per logical row is lost and upserts have nothing to merge on. The fix is to add a primary key or set `REPLICA IDENTITY FULL` and configure a message key explicitly.

Q: What is the practical difference between `pgoutput` and `decoderbufs` as logical decoding plugins?
A: `pgoutput` is built into Postgres 10+ and needs nothing installed on the server, while `decoderbufs` is a Protobuf-based plugin that must be compiled and installed as a Postgres extension. Functionally both feed Debezium, but in a regulated database where DBAs will not add third-party C extensions, `pgoutput` is the pragmatic default. Debezium selects the plugin via `plugin.name`.

Q: What does the connector property `plugin.name` configure and what value do you typically use on modern Postgres?
A: `plugin.name` selects the logical decoding output plugin the connector uses, with `pgoutput` being the typical value on Postgres 10 and later because it ships with the server. The alternatives are `decoderbufs` and the older `wal2json`. Choosing `pgoutput` avoids any extension-installation step on the database host.

Q: What permissions does the Postgres role used by Debezium need at minimum?
A: The role needs `REPLICATION` (or be a superuser) to create and read logical replication slots, `LOGIN`, and at least `SELECT` on the captured tables for the snapshot. If Debezium must create the publication it also needs `CREATE` on the database; pre-creating a `filtered` publication lets you avoid granting that. Scoping these tightly is part of least-privilege design for a regulated source.

Q: How would you capture both the Phase-1 `orders` and `nav` tables and route each to its own topic?
A: Set `table.include.list = public.orders,public.nav` so only those two tables are captured, and rely on Debezium's default topic naming `<topic.prefix>.public.orders` and `<topic.prefix>.public.nav` to land each table on its own topic. You configure `topic.prefix` to namespace the connector. Each table then streams to a dedicated, primary-key-keyed topic that downstream merge jobs consume independently.

Q: What does `topic.prefix` do and why must it be unique per connector?
A: `topic.prefix` is a mandatory logical name prepended to every topic the connector produces and embedded in the `source` metadata, identifying the source system. It must be unique across connectors because two connectors sharing a prefix would collide on topic names and offsets, corrupting the streams. A clear prefix like `ta_prod` makes the topic namespace self-describing.

Q: Why does Debezium include the WAL LSN in the `source` metadata, and how do you use it downstream?
A: The LSN is the exact byte offset of the change in the WAL, giving a monotonically increasing, source-authoritative ordering key for every event. Downstream you use it as the tiebreaker for dedupe (latest LSN wins per key) and as the proof-of-position when reconciling against Postgres. It is the most reliable ordering field because it reflects true commit sequence, unlike processing timestamps.

Q: A NAV correction updates the same fund's row three times in one second. How do you ensure the lakehouse reflects the final value?
A: Because all three updates share the fund's primary key, they go to the same partition in commit order, and your merge keyed on that primary key applies them in LSN order so the last update wins. In append-plus-dedupe you keep all three but resolve current state by the highest LSN. Either way ordering by source LSN — not arrival time — guarantees the final corrected NAV is what the lakehouse shows.

Q: What is the risk of running multiple Debezium connectors against the same publication or slot, and how do you avoid it?
A: A logical replication slot can be read by only one consumer at a time, so two connectors on the same `slot.name` will conflict and one will fail or stall. The fix is a distinct `slot.name` (and usually a distinct publication) per connector. This is a common production trap when someone clones a connector config without changing the slot name.

Q: How does Debezium behave when the database is unreachable for a while, and what recovers it?
A: The connector retries the database connection (with backoff) and its task may transition to a failed state if retries are exhausted; meanwhile the replication slot keeps the WAL on the server. When the database returns, restarting the connector via the Connect REST API resumes streaming from the stored offset and retained WAL. The slot plus offset are what make the gap-free recovery possible.

Q: Why might you enable transaction metadata even though it adds a topic and overhead?
A: Transaction metadata is what lets you reconstruct atomic commit boundaries across multiple tables, which matters when a single business operation (say a subscription) writes to `orders` and `nav` together and consumers must see them as one unit. Without it you only have per-table ordering and cannot tell which cross-table changes committed together. The extra transaction topic is the price of cross-table consistency.

Q: What does a `before` image being entirely null on an update tell you about the source table configuration?
A: It indicates the table likely has no primary key and `REPLICA IDENTITY DEFAULT`, so Postgres logs no old-row identity to the WAL, leaving Debezium with nothing to populate `before`. To get a usable old image you must set `REPLICA IDENTITY FULL` (or add a primary key / `USING INDEX`). Until then, change-detection logic that diffs `before` against `after` cannot work for that table.

Q: How does keying by primary key plus single-partition routing give per-row ordering even with multiple partitions on the topic?
A: Because Debezium keys each event by the row's primary key, Kafka's default partitioner sends all events for one key to the same partition, and Kafka guarantees order within a partition. So even on a many-partition topic, every change to a given row stays strictly ordered relative to itself. Different rows may interleave across partitions, but no single row's history is ever reordered.

Q: Why must the merge job into Iceberg be idempotent, and how is that achieved?
A: Because Debezium is at-least-once and restarts can replay events, the merge job will sometimes see the same change twice, so it must produce the same result whether a change is applied once or many times. Idempotence is achieved by upserting on the primary key (re-applying overwrites with the identical value) or by appending and deduping on source LSN. This is what lets duplicates after a restart cause zero divergence.

Q: How do you decide between landing CDC as a current-state Iceberg table versus a full-history one for fund data?
A: Choose current-state (merge/upsert) when consumers only need the latest mirror of the source — fast and compact. Choose full-history (append-plus-dedupe) when regulation or audit demands reconstructing point-in-time values, such as showing what a fund's NAV was on a past date. Often you keep the append table as the system of record and materialize a current-state view from it.

Q: What is the consequence of setting `snapshot.mode = never` (where supported), and why is it not offered for Postgres?
A: `never` skips the snapshot and streams only from the current log position, so pre-existing rows are never captured — useful only when you already have the historical data loaded elsewhere. Debezium documents it as available for MySQL and MariaDB, not the Postgres connector. On Postgres you instead use `no_data` to start streaming without read events while still capturing schema.

Q: During an initial snapshot the connector restarts. Does it start the snapshot over?
A: With a non-incremental (blocking) snapshot, a restart before the snapshot completed generally re-runs it, because the snapshot is not resumable and the offset only marks streaming progress, not partial snapshot rows. Incremental snapshots, by contrast, track chunk progress and can resume mid-snapshot. This is one reason incremental snapshots are preferred for very large tables.

Q: What is the difference between a Connect `pause` and `restart` for a Debezium connector's tasks?
A: `pause` stops the connector's tasks from processing while keeping them assigned and their state intact, so they resume from the same offset on `resume`. `restart` stops and re-starts the task(s), re-reading configuration and resuming from the stored offset. Pause is for temporarily halting flow (e.g. during downstream maintenance); restart is for recovering a failed task or applying certain changes.

Q: Why does pausing a Debezium Postgres connector risk WAL growth, and what mitigates it?
A: While paused, the connector stops confirming new LSNs, so the replication slot's `restart_lsn` does not advance and Postgres retains all WAL produced during the pause. On a busy database that can grow quickly. Mitigate by keeping pauses short, monitoring slot lag, and not pausing a CDC connector against a high-write database for long without watching disk.

Q: How does the outbox approach preserve event ordering for a single aggregate across a restart?
A: Outbox rows are captured from the WAL in commit order and keyed by `aggregateid`, so all events for one aggregate go to the same partition and stay ordered; after a restart the connector resumes from its offset and continues emitting in the same order. At-least-once may replay the tail, but the per-key ordering and idempotent consumers keep the aggregate's event sequence correct. The transactional write of the outbox row is what anchors the ordering to the business commit.

Q: What goes wrong if the application updates the outbox row instead of inserting a new one per event?
A: The outbox pattern relies on each event being a distinct insert so CDC emits one change event per event; updating a single row in place would emit updates that overwrite the prior event, and consumers would lose intermediate events. The outbox is meant to be append-only — insert one row per domain event, optionally deleting old rows later. In-place updates defeat the guaranteed-publish semantics.

Q: Can you delete rows from the outbox table after they are captured, and does that affect consumers?
A: Yes — once Debezium has captured an outbox insert from the WAL, the source row can be deleted to keep the table small, and the delete itself is just another (ignorable) change you typically filter out. The captured event already lives on the Kafka topic, so downstream consumers are unaffected by the source cleanup. Many teams run a periodic purge of already-relayed outbox rows.

Q: Why is the EMT/EPT file flow a poor candidate for direct table CDC but a good fit for the outbox pattern?
A: EMT/EPT (European MiFID/PRIIPs Template) outputs are produced business artifacts whose meaningful unit is a generated file/record set, not raw table mutations, so direct CDC of the underlying tables would leak storage structure and miss the "a template was produced" semantics. An outbox row written in the same transaction that finalizes a template gives a clean, contract-stable "EMT produced" event. CDC then reliably publishes that domain event without polling.

Q: How does CDC help reconcile a transfer-agency subledger against a lakehouse copy for ISIN-level positions?
A: CDC streams every insert/update/delete on the subledger position tables in commit order, so the lakehouse can maintain an exact mirror keyed by ISIN (and account), and the per-table ordering guarantee plus LSN-based dedupe ensure the latest position per key is correct. A row-by-row hash comparison keyed on ISIN then proves zero divergence. This gives auditors a continuously reconciled, point-in-time-capable copy without batch extracts.

Q: Why is capturing deletes correctly especially important for regulatory positions like LEI or holdings records?
A: If a holding or counterparty record is deleted in the source but the delete is dropped by the pipeline, the lakehouse retains a position that no longer exists, overstating exposure or misreporting a counterparty's LEI status. Log-based CDC captures deletes from the WAL (which query-based polling cannot), and the merge must apply `op:"d"` as a real removal. Correct delete handling is what keeps regulatory reporting from diverging from the source of record.

Q: What is the danger of using `updated_at`-based polling instead of log-based CDC for a SWIFT message store?
A: Polling on `updated_at` misses hard deletes entirely, can miss rapid intra-poll updates (only the last state survives), and can miss rows whose timestamp was not bumped, so a message store mirrored this way silently diverges. It also adds query load to the source. Log-based CDC reads every committed change from the WAL in order, capturing deletes and every intermediate state, which is why it is the correct pattern here.

Q: How do schema-change events plus a schema registry together prevent silent corruption of a NAV pipeline?
A: Debezium orders schema-change events ahead of dependent data events so consumers learn of structural changes first, and a schema registry enforces compatibility so an incompatible change (like dropping a required NAV field) is rejected at registration rather than silently breaking consumers. Together they make the schema contract a hard gate. Without both, a producer-side change can corrupt downstream parsing before anyone notices.

Q: Why does an incremental snapshot of a large `nav_history` table not block the live NAV change stream?
A: Incremental snapshots read existing rows in bounded chunks interleaved with live WAL streaming, so new NAV changes keep flowing to the topic while the backfill proceeds in the background. The connector deduplicates any overlap between a snapshotted chunk and concurrent live changes by source ordering. This means you can re-baseline a huge history table without a maintenance window or a streaming stall.

Q: What single artifact lets a Debezium Postgres connector resume after both the worker and database restart, and why?
A: The combination of the Kafka offset (last committed LSN in `offset.storage.topic`) and the durable replication slot on Postgres. The offset tells the connector exactly where to resume, and the slot guarantees the WAL from that LSN onward was retained on the database despite the outage. Neither alone is sufficient: the slot without the offset loses consumer position, and the offset without the slot may point to purged WAL.

Q: How would you size a Connect cluster to run several Debezium Postgres connectors plus their sinks?
A: Each Debezium Postgres source runs one task, so source throughput is bounded by single-stream decoding; you scale by spreading connectors across workers and giving sink connectors multiple tasks matched to topic partitions. Size workers for peak event rate, serialization cost, and headroom for rebalance, and ensure enough memory for buffering. The sources do not parallelize, so the cluster's parallelism budget mostly goes to sinks and topic partitioning.

Q: What happens to Connect's in-flight work when a worker dies in distributed mode?
A: Connect detects the worker loss and rebalances the failed worker's connectors and tasks onto the surviving workers, which resume each from its last committed offset in the offset topic. For a Debezium Postgres source that means the single task restarts on another worker and continues from the stored LSN against the same replication slot. Distributed mode plus durable offsets are what make this failover gap-free.

Q: A change event's `after` shows a column missing that exists in Postgres. What should you check first?
A: First check the connector's column-level filters (`column.include.list`/`column.exclude.list`) and any masking/truncation transforms, since an excluded or masked column simply will not appear in `after`. Also confirm the schema-change event for that column was processed if it was added recently. The missing column is almost always a deliberate filter or a schema-evolution lag, not a decoding failure.

Q: What is the purpose of the `decimal.handling.mode` setting for fund data, and what is the risk of the default?
A: `decimal.handling.mode` controls how Postgres numeric/decimal values are represented in change events — as `precise` (default, encoded as byte-backed exact decimals), `double`, or `string`. The risk with `double` is silent loss of precision on monetary values like NAV per share, which is unacceptable for financial figures. Keep `precise` (or `string`) so exact decimal amounts survive into the lakehouse.

Q: Why does landing CDC into Iceberg via append require periodic compaction or dedupe maintenance?
A: Append writes one row per change event, so the table accumulates many versions per key and many small files, which slows queries and inflates storage over time. Periodic compaction merges small files and an optional dedupe/expire pass collapses or ages out superseded versions while preserving the history you must keep. Without maintenance, the append table degrades in query performance even though it stays correct.

Q: Summarize the end-to-end restart-correctness story you would present to a regulator for a Debezium-to-Iceberg pipeline.
A: The initial snapshot captures all pre-existing rows as `op:"r"`, then logical decoding streams every committed change from the WAL in commit order, held safe by a durable replication slot; on any connector or worker restart, Connect resumes from the last LSN in the offset topic with no gap, and at-least-once replays are absorbed by an idempotent Iceberg upsert (or LSN dedupe). Finally a row-by-row hash reconciliation keyed on the primary key demonstrates zero divergence between Postgres and the lakehouse after a deliberate mid-stream restart. That chain — snapshot completeness, ordered durable streaming, exact-position resume, idempotent landing, and a clean reconciliation — is the proof of completeness and ordering.

Q: Debezium fails to start with an error that logical decoding is not enabled — which Postgres parameter is wrong and what does fixing it require?
A: The server's `wal_level` is not set to `logical` (it is likely `replica` or `minimal`), so Postgres is not writing the logical change information Debezium needs. You fix it by setting `wal_level = logical` in `postgresql.conf` (or via `ALTER SYSTEM`), which requires a full Postgres restart — not just a `SELECT pg_reload_conf()` — because `wal_level` is not reloadable. On a managed service like RDS you set the equivalent parameter (e.g. `rds.logical_replication = 1`) and reboot the instance.

Q: A connector cannot create its replication slot and errors about reaching the slot limit — which two server parameters do you raise?
A: You raise `max_replication_slots` so the server permits enough logical slots (each Debezium connector needs one), and `max_wal_senders` because every active slot consumes a WAL sender process. A common safe rule is to set `max_wal_senders` at least equal to `max_replication_slots` plus any physical replicas. Both changes take effect only after a Postgres restart, so plan a maintenance window before deploying many connectors.

Q: How do you drop an orphaned logical replication slot left by a removed connector, and what must be true first?
A: You run `SELECT pg_drop_replication_slot('slot_name')` after confirming the slot is inactive — its `active` column in `pg_replication_slots` must be `false`, because an in-use slot cannot be dropped. Dropping it lets Postgres recycle the WAL the dead connector was pinning and stops the disk filling. The gotcha: never drop a slot whose connector you intend to resume, because the resume will then fail and force a re-snapshot.

Q: Which columns of `pg_replication_slots` tell you a slot is the cause of WAL bloat, and how do you read them?
A: Look at `active` (is anything consuming it), `restart_lsn` (the oldest WAL the slot still pins), and on newer Postgres `pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)` to get the retained-bytes lag. An `active = false` slot with a `restart_lsn` far behind `pg_current_wal_lsn()` is pinning a large WAL window and is your culprit. The fix is to resume the consumer (advancing `restart_lsn`) or drop the slot if it is truly abandoned.

Q: What does `slot.drop.on.stop` do, and why must it stay `false` in production?
A: `slot.drop.on.stop` controls whether the connector drops its logical replication slot when it shuts down cleanly; it defaults to `false`. If set to `true`, a graceful stop discards the slot and lets Postgres recycle the pinned WAL, so on the next start the connector cannot resume from its old position and any changes during the gap are lost — forcing a re-snapshot. It is only safe in throwaway dev or test environments, never against a regulated source of record.

Q: What is the default Postgres publication name Debezium uses with `pgoutput`, and which property overrides it?
A: The default publication name is `dbz_publication`, and you override it with the `publication.name` property. You set an explicit, unique name (for example `pub_ta_orders`) when running several connectors so each owns a clearly identifiable publication and they do not collide on the default. Pre-creating that named publication with `CREATE PUBLICATION ... FOR TABLE ...` also lets the connector role avoid needing `CREATE` privilege.

Q: How do you trigger an incremental snapshot via a Kafka signal topic instead of a signaling table, and what does the message look like?
A: You enable the `kafka` signal channel and configure `signal.kafka.topic`, then produce a message whose key equals the connector's `topic.prefix` and whose value is JSON like `{"type":"execute-snapshot","data":{"data-collections":["public.nav"],"type":"INCREMENTAL"}}`. The connector consumes that signal and begins chunked snapshotting of the named tables while streaming continues. This is useful when you cannot or do not want to grant the connector write access to a signaling table in the source database.

Q: The `source` channel for signals is always enabled — why, and what is it required for?
A: The `source` (in-database signaling table) channel is enabled by default because the incremental-snapshot mechanism uses it internally to mark chunk window open/close boundaries in WAL order. Even if you trigger snapshots over the `kafka` channel, the connector still relies on the source channel's deterministic ordering to deduplicate snapshot chunks against live changes. That is why `signal.data.collection` (the signaling table) is generally still required for incremental snapshots on Postgres.

Q: What does `heartbeat.action.query` add on top of `heartbeat.interval.ms`, and when is it essential?
A: `heartbeat.interval.ms` makes Debezium emit periodic heartbeat events, but those only advance the slot if the connector actually flushes; `heartbeat.action.query` lets you specify a DML statement (e.g. an `UPDATE` of a dedicated heartbeat table) that Debezium runs on each heartbeat so a real change appears in the WAL of a captured table. It is essential when your captured tables are idle but other databases/tables on the same cluster are busy, because it forces the slot's `restart_lsn` to advance and prevents WAL from accumulating. Without it, a quiet captured table can pin WAL indefinitely.

Q: Why might `heartbeat.interval.ms` alone fail to advance the slot on a low-traffic capture, and what is the more reliable fix?
A: Heartbeat events advance the connector's committed offset, but on Postgres the slot's `restart_lsn` only moves forward when there is decodable activity the connector confirms, so a purely idle capture can still pin WAL. The more reliable fix is `heartbeat.action.query`, which injects an actual write into the WAL stream the connector reads, giving it a concrete position to confirm. This is the standard remedy for the "busy neighbour table" WAL-growth pattern.

Q: An auditor asks for exact source provenance of a lakehouse row — which `source` field gives the byte-precise WAL position on Postgres?
A: The `source.lsn` field carries the WAL log sequence number, a monotonically increasing byte offset that uniquely positions the change in the transaction log. Combined with `source.txId` (the transaction id) and `source.ts_ms` (the commit time), it lets you point to the exact commit in Postgres that produced the row. For an auditor this is the cryptographic-grade pointer back to the source of record, far stronger than a processing timestamp.

Q: How do you confirm a delete actually removed a position from the source via the event stream alone?
A: You look for a change event with `op:"d"` whose `before` block carries the deleted row's primary key (subject to REPLICA IDENTITY) and whose `after` is null, optionally followed by a tombstone with the same key and null value. The `source.lsn` on that delete proves the WAL position at which the removal committed. If you only ever see `op:"u"` and `op:"c"` for a table that does experience deletes, the deletes are being lost upstream of the topic.

Q: A consumer sees `op:"u"` events but `before` always equals `after` for changed rows — what is misconfigured?
A: The table is almost certainly at `REPLICA IDENTITY DEFAULT` or has no full old-image, so Postgres logs only the primary key in the old row and Debezium fills `before` with key-plus-nulls rather than the true prior values, making naive before/after diffs look identical or unusable. Set `ALTER TABLE ... REPLICA IDENTITY FULL` to log the entire old row so `before` reflects real prior state. Until then, change-detection by comparing `before` to `after` is unreliable for that table.

Q: How would you capture only a subset of columns from a sensitive `clients` table and exclude PII?
A: Use `column.include.list` to allowlist exactly the columns you need, or `column.exclude.list` to drop the PII columns, both expressed as `schema.table.column` regex entries. Excluded columns simply never appear in the `after`/`before` payloads, so the PII never leaves the source. For columns you must keep but obscure, you instead apply a masking transform rather than excluding them.

Q: What is the difference between excluding a column and masking it with `column.mask.hash`, and when do you choose masking?
A: Excluding a column removes it from the event entirely, so downstream consumers never see the field at all, whereas `column.mask.hash` replaces the value with a deterministic hash (using a configured algorithm and salt) so the field is present but its real value is hidden. You choose masking when consumers still need to join or detect change on that field — equal inputs hash equally — but must not see cleartext, such as a hashed client identifier. Exclusion is for fields nobody downstream should know exist.

Q: For a NAV value stored as Postgres `numeric(18,6)`, which setting keeps full precision and what is the wire representation under the default?
A: Keep `decimal.handling.mode` at its default `precise`, which encodes the value as an exact byte-backed decimal (a base64-encoded two's-complement unscaled integer plus the scale in the schema), preserving every digit. Switching to `double` would risk silent rounding of monetary figures, and `string` keeps exact text but forces consumers to parse. For fund NAV per share, `precise` (or `string`) is mandatory because a rounding error in a per-share price is a reportable data-quality defect.

Q: A consumer using a JSON converter complains the NAV decimal arrives as an unreadable base64 string — what is happening and what are the options?
A: With `decimal.handling.mode = precise`, the exact decimal is serialized as a base64-encoded byte array (the `org.apache.kafka.connect.data.Decimal` logical type), which a plain JSON reader shows as opaque text rather than a number. You can either teach the consumer to decode the logical type using the schema's scale, or switch `decimal.handling.mode` to `string` so the value arrives as a human-readable exact decimal string. Avoid `double` for money, since that trades readability for precision loss.

Q: How does `time.precision.mode` affect a Postgres `timestamp` column landing in the lakehouse, and what is the safe choice for fund timestamps?
A: `time.precision.mode` controls how temporal types are encoded: `adaptive` (default) matches the column's own precision, `adaptive_time_microseconds`, and `connect` which downgrades to Kafka Connect's millisecond-based logical types. The risk with `connect` is truncating microsecond-precision source timestamps to milliseconds. For fund data where you may need microsecond ordering or exact audit timestamps, `adaptive`/`adaptive_time_microseconds` preserves the source precision.

Q: You add a `NOT NULL` column with a default to the captured `orders` table mid-stream — what does the consumer see and in what order?
A: Debezium emits a schema-change event recording the DDL before any data events carry the new column, then subsequent `op` events include the new field in `after` populated with the default for rows written after the change. The consumer must process the schema-change event first to widen its schema, which is why Debezium guarantees the ordering. Backfilling the new column onto historical rows already landed in the lakehouse, if needed, is a separate reconciliation step the stream does not do for you.

Q: A `DROP COLUMN` on a captured table breaks downstream Avro consumers despite Debezium emitting the schema change — what compatibility rule was violated?
A: Dropping a field that consumers require is a backward-incompatible change under the registry's compatibility setting, so even though Debezium correctly emits the DDL, the new schema version fails to register or breaks readers expecting the old field. With `BACKWARD` compatibility a producer may drop a field only if it had a default, and removing a required field violates that. The architectural fix is to enforce compatibility at the registry as a hard gate and phase column removals as deprecate-then-drop.

Q: How does the Postgres connector differ from the MySQL connector in where it gets table schema history?
A: The MySQL connector must maintain a separate database schema history topic because it parses DDL from the binlog and needs to remember prior schemas, whereas the Postgres connector does not use a schema history topic — it reads current table structure from the live catalog and from `pgoutput`/replica identity metadata. This means a wiped schema-history topic is a MySQL recovery concern, not a Postgres one. For Postgres, the recovery artifacts that matter are the slot and the Connect offset topic.

Q: What does `snapshot.select.statement.overrides` let you do, and give a fund-data use case?
A: It lets you supply a custom `SELECT` per table that the initial/incremental snapshot uses instead of reading the whole table, for example to filter or order the snapshot rows. A fund-data use case is snapshotting only currently-open positions — `SELECT ... WHERE status = 'OPEN'` on a large `positions` table — to avoid backfilling years of closed records you reload elsewhere. You configure it together with the per-table property naming the override statement; live streaming after the snapshot is unaffected by the override.

Q: How do you pause and later resume an in-progress incremental snapshot, and why is that useful?
A: You send a `pause-snapshot` signal (via the signaling table or Kafka signal channel) to halt the chunked backfill while live streaming continues, then a `resume-snapshot` signal to pick it back up from where it stopped. This is useful when the backfill is adding too much load during a busy NAV-cut window — you pause it, let the time-critical streaming proceed, and resume the backfill off-peak. Because incremental snapshots track chunk progress, the resume does not restart the table from scratch.

Q: How do you stop an incremental snapshot entirely if it was started on the wrong table?
A: You send a `stop-snapshot` signal naming the data collection, which aborts the chunked read for that table without affecting ongoing live streaming or other tables' snapshots. Unlike `pause-snapshot`, it does not retain progress to resume later. This is the corrective action when an operator triggered `execute-snapshot` against the wrong table and you want to cancel before it consumes more resources.

Q: Two connectors must capture the same database — what three properties must differ to keep them isolated?
A: Each connector needs a distinct `slot.name` (a slot serves one consumer), a distinct `publication.name` (or at least non-conflicting publication scope), and a distinct `topic.prefix` (so topic names and offsets do not collide). Sharing any of these causes a slot conflict, publication confusion, or overlapping offset state. The safest pattern is to template the config and vary all three plus the table include lists per connector.

Q: A cloned connector config caused the original connector to stall — what is the most likely single cause?
A: The clone almost certainly reused the same `slot.name`, and because a logical replication slot can be consumed by only one connection at a time, the two connectors fight over it and one stalls or errors. The fix is to give the clone its own unique `slot.name` (and publication). This is the classic copy-paste trap when scaling out CDC.

Q: How does Debezium expose connector and task health, and what would you poll to detect a failed task?
A: Debezium runs on Kafka Connect, so you query the Connect REST API endpoint `GET /connectors/{name}/status`, which returns the connector state plus each task's state and trace. A task in the `FAILED` state with a stack trace in `trace` is your alert condition; `RUNNING` for connector and all tasks is healthy. You typically scrape this (or the JMX metrics) so an operator is paged before WAL backs up.

Q: Which Connect REST call restarts only a failed task without disturbing a healthy one, and why prefer it?
A: You call `POST /connectors/{name}/tasks/{taskId}/restart` to restart a single task by id, leaving other tasks and the connector configuration untouched. You prefer it over restarting the whole connector because it is the least disruptive recovery — it resumes that task from its stored offset without re-evaluating config or rebalancing the cluster. For the single-task Postgres source there is effectively one task (id `0`), so this targets it precisely.

Q: How do you pause a Debezium connector through the REST API, and what state do its tasks enter?
A: You call `PUT /connectors/{name}/pause`, and the connector's tasks transition to the `PAUSED` state, stopping consumption while keeping their assignment and offsets intact. They resume from the same offset on `PUT /connectors/{name}/resume`. The Postgres caveat is that a paused source stops advancing the slot, so WAL accumulates during the pause on a busy database.

Q: What is the risk of leaving a Debezium Postgres connector paused over a weekend on a high-write database?
A: While paused, the connector stops confirming new LSNs, so the slot's `restart_lsn` is frozen and Postgres retains every WAL segment written across the whole weekend, which on a busy database can exhaust disk and take the database down. The mitigation is to monitor slot lag and either keep pauses short or delete the connector (and slot) rather than pausing it for long maintenance. Pausing is for brief downstream maintenance, not extended outages.

Q: How would you re-baseline a single drifted table to the lakehouse without restarting the connector or re-reading every other table?
A: You trigger an ad-hoc incremental snapshot for just that table via an `execute-snapshot` signal naming the one data collection, so Debezium chunk-reads its current rows as snapshot events interleaved with ongoing live streaming. The merge job upserts those rows by primary key, correcting the drift, while all other tables keep flowing untouched. This is the surgical fix versus the blunt instrument of a full connector re-snapshot.

Q: During an incremental snapshot, two records for the same key arrive — one snapshot read and one live update. Which wins and how?
A: The live change wins if it is newer, because Debezium uses a windowing/deduplication algorithm that, for each chunk, drops snapshot reads of keys that also received a live event in the same window, so the streamed (more recent) state takes precedence. This prevents an older snapshot read from clobbering a fresher live update of the same row. The mechanism relies on the source-channel signal markers that bound each chunk window in WAL order.

Q: Your merge job lands rows but query latency degrades over weeks on the append table — what maintenance is missing and what runs it?
A: Append-mode writes one small file per micro-batch, so the Iceberg table accumulates many small data files and many versions per key, which slows scans; the missing maintenance is file compaction (and snapshot expiration). You run Iceberg's `rewrite_data_files` to bin-pack small files and `expire_snapshots` to drop old metadata/orphan files, optionally a dedupe/compaction of superseded versions you no longer need. Scheduling these keeps the auditable history correct while restoring query performance.

Q: How does Iceberg snapshot isolation let you reconcile against Postgres at a precise cutoff?
A: Each Iceberg commit produces an immutable snapshot id, so you can run your dedup'd current-state reconciliation query against a specific snapshot (via time travel) that corresponds to a known ingestion cutoff, rather than a moving target. You then compare that frozen Iceberg state against Postgres as of the matching source LSN. This pinning is what makes the "zero divergence after restart" proof reproducible rather than racing live writes.

Q: Write the dedupe expression that yields current state when a delete must beat an earlier update at the same LSN ordering.
A: Use a window like `ROW_NUMBER() OVER (PARTITION BY pk ORDER BY source_lsn DESC, op_rank DESC)` keeping rank `1`, where `op_rank` ranks a delete above an update/insert so that when ordering ties on `source_lsn` the delete is treated as latest. Then filter out rows whose winning event is `op:"d"` to drop deleted keys from current state. Ordering by `source_lsn` (not arrival time) plus the delete-precedence tiebreak is what guarantees a removed position does not resurrect.

Q: Why does keying the Kafka topic by primary key still preserve order when the topic has many partitions?
A: Debezium sets the message key to the row's primary key, and Kafka's default partitioner hashes that key to a fixed partition, so every event for a given row lands in the same partition where Kafka guarantees in-order delivery. Different rows spread across partitions for throughput, but no single row's event sequence is ever reordered. This is how you get per-row ordering and partition-level parallelism simultaneously.

Q: A table you must capture has no primary key — what two things break and how do you remediate?
A: Without a primary key (and with `REPLICA IDENTITY DEFAULT`) the change events carry a null/empty key, so per-row ordering is lost (events scatter across partitions) and upsert/compaction have nothing to key on. You remediate by adding a real primary key, or by setting `REPLICA IDENTITY FULL` and configuring a custom message key (via a key-defining SMT or `message.key.columns`) on the columns that uniquely identify a row. Until one of those exists, the lakehouse cannot reliably maintain current state for that table.

Q: How would you set a custom message key on columns other than the primary key?
A: You use the `message.key.columns` property, listing `schema.table:col1,col2` entries that tell Debezium which columns form the Kafka key for that table. This is how you give a primary-key-less table (or one keyed differently for the lakehouse) a deterministic key so ordering and upserts work. The chosen columns must uniquely identify a row, or events for different rows will collide on the same key and reorder.

Q: What does `op:"t"` (truncate) require the merge job to do, and what breaks if it is ignored?
A: A `TRUNCATE` emits an `op:"t"` event with no per-row `before`, so the merge job must interpret it as "delete all current rows for this table" rather than as a single-row change. If ignored, every row that existed at truncate time stays in the lakehouse even though the source table is now empty, leaving phantom positions. Correctly handling truncate is essential for any table that gets bulk-cleared, like a staging or reload table.

Q: How do you make Debezium not emit truncate events at all, and when would you?
A: You configure `skipped.operations` to include `t` (or set the truncate-handling so truncates are filtered), so the connector omits `op:"t"` events from the stream. You might do this when truncates on the source are operational noise that should not propagate to the lakehouse — for example a staging table the source app truncates and reloads, but whose intermediate empty state you do not want mirrored. You accept that the lakehouse will not reflect the momentary truncate.

Q: What does `skipped.operations` let you filter, and give a reason to skip deletes (`d`).
A: `skipped.operations` is a comma-separated list of operation codes (`c`, `u`, `d`, `t`, `r`) the connector should drop before emitting, letting you, say, ignore deletes entirely. You might skip `d` when the lakehouse is intentionally an append-only historical archive where source deletes must not remove history — a soft-delete/audit model. The risk is that current-state views then never reflect real removals, so this is a deliberate, documented design choice, not a default.

Q: A poison change event keeps killing the merge job on restart — how do you keep the pipeline moving without losing the bad record?
A: On the Connect sink side you enable error tolerance with `errors.tolerance = all` and route failures to a dead-letter queue via `errors.deadletterqueue.topic.name`, so the offending record is parked with diagnostic headers instead of halting the task. The pipeline advances past the poison message while you inspect the DLQ to fix the root cause. This converts a hard stop into an isolatable, replayable failure — critical so one malformed record does not stall NAV ingestion.

Q: How do diagnostic headers on a DLQ record help you debug a failed CDC sink message?
A: When `errors.deadletterqueue.context.headers.enable = true`, Connect stamps the dead-lettered record with headers giving the original topic, partition, offset, and the exception class and message that caused the failure. That lets you trace the exact source event and the precise error (e.g. a schema mismatch or null key) without guessing. It turns a vague "task failed" into an actionable, reproducible diagnosis tied to one record.

Q: After wiping the Connect offset topic by mistake, what happens on the next connector start and what is the data-integrity consequence?
A: With no stored offset, the connector behaves as a first start: in `initial` mode it performs a full snapshot again, re-emitting every row as `op:"r"`. The integrity consequence is mass duplication downstream unless the sink is idempotent — an upsert keyed on the primary key absorbs the replay as no-ops, while a naive append would double every row. This is exactly why the merge job must be idempotent and why the offset topic should be protected and replicated.

Q: Distinguish what the replication slot guarantees versus what the Connect offset topic guarantees on resume.
A: The replication slot guarantees the WAL from the connector's last confirmed position forward is still physically retained on Postgres, so the data to resume from exists. The Connect offset topic guarantees the connector knows exactly which LSN it last committed to Kafka and whether a snapshot was mid-flight, so it knows where to resume. Neither alone suffices: the slot without the offset loses your position, and the offset without the slot may point at WAL Postgres already recycled.

Q: An EMT template is regenerated nightly and consumers must get exactly one "template produced" event per run — why is an outbox insert better than capturing the template table?
A: Capturing the underlying template tables directly would emit many low-level row changes per regeneration and leak the storage layout, making "one template produced" ambiguous and brittle. An outbox row inserted in the same transaction that finalizes the EMT produces exactly one clean, contract-stable domain event, and CDC guarantees it is published if and only if that transaction committed. Consumers then react to a single well-shaped `EMT produced` event rather than reverse-engineering table mutations.

Q: Why must outbox rows be inserted (never updated in place), and what does the Outbox Event Router key the message by?
A: Each domain event must be a distinct `INSERT` so CDC emits exactly one change event per event; updating one outbox row in place would overwrite earlier events and consumers would lose the intermediate ones, defeating the guaranteed-publish semantics. The Outbox Event Router keys the resulting message by the `aggregateid` column (overridable), so all events for one aggregate hash to the same partition and stay ordered. The router routes to a topic chosen by `aggregatetype` and uses `payload` as the message body.

Q: Compare building CDC on open-source Debezium versus buying Qlik Replicate or Oracle GoldenGate for a regulated fund estate.
A: Debezium plus Kafka Connect is free, transparent, and gives you full control of the change envelope and landing logic, but you own the operational burden — slot/WAL monitoring, connector ops, and upgrades — and must build the audit story yourself. Qlik Replicate and Oracle GoldenGate are EU-bank incumbents offering vendor support, GUIs, heterogeneous-source coverage, and packaged compliance features, at license cost and with less low-level control. The architect's build-vs-buy call weighs in-house Kafka skills and the need for control against vendor support obligations and existing GoldenGate/Qlik footprint in the bank.

Q: For a heterogeneous source mix (Postgres plus an Oracle core), what capability would push you toward GoldenGate over a Debezium-only stack?
A: Oracle GoldenGate's mature, vendor-supported Oracle CDC (deep redo-log integration and certified support for complex Oracle features) is the capability that pushes you toward it when the Oracle core is mission-critical and the bank already runs GoldenGate. Debezium does offer an Oracle connector, but in a regulated estate the certified support, established operational tooling, and existing licenses can outweigh its openness. You might still run Debezium for the Postgres side and GoldenGate for Oracle, or standardize on one to reduce operational surface — that trade-off is the architect's decision.

Q: How would you prove SWIFT message-store CDC captures every state transition where polling on `updated_at` would silently diverge?
A: You point Debezium at the message-store table so every committed insert/update/delete is read from the WAL in commit order, capturing hard deletes and rapid intra-second transitions a timestamp poll would miss, then prove completeness with a per-row hash reconciliation keyed on the message identifier after a deliberate restart. Polling on `updated_at` misses deletes entirely, collapses multiple fast updates into the last visible state, and skips rows whose timestamp the application forgot to bump. The WAL-plus-slot chain plus a zero-divergence comparison is the evidence that no SWIFT message transition was lost.
