# Phase 4 · 1.8.3 + 1.8.4 Outbox & saga (implemented) — 150 self-test questions

Q: What is the "dual-write" problem the transactional outbox pattern exists to solve?
A: Dual-write is when a service must update its own database and also publish an event to a broker as two separate writes that are not in one atomic unit. If the process crashes between the database commit and the publish, the state change is persisted but the event is lost (or vice versa), corrupting every downstream consumer. The outbox eliminates the race by making the event a row written inside the same local database transaction as the state change.

Q: In one sentence, how does the transactional outbox pattern work?
A: Instead of publishing to the broker directly, the service inserts the event as a row into an `outbox` table within the same database transaction that performs the business state change, so both commit or neither does. A separate relay process then reads new outbox rows and publishes them to the broker. This converts an unsafe distributed dual-write into a single local atomic write plus an at-least-once relay.

Q: Why does writing the event into the same transaction as the state change give you atomicity?
A: A single database transaction is ACID, so the business-row update and the outbox-row insert either both commit or both roll back as one unit. There is no window in which the state changed but the event row is missing. The broker publish becomes a downstream concern handled by the relay, decoupled from the transactional boundary.

Q: What are the two distinct responsibilities the outbox pattern separates?
A: First, capturing the event reliably — inserting it into the outbox table inside the business transaction so it can never be lost. Second, relaying the event — a separate, asynchronous process that reads committed outbox rows and publishes them to the broker. Keeping these separate is what lets the capture be transactional while the publish is at-least-once.

Q: What delivery guarantee does the outbox relay give you, and why not exactly-once?
A: It gives at-least-once delivery: the relay may crash after publishing but before marking the row processed, so it republishes on restart and the same event can appear more than once. True end-to-end exactly-once across a database and a broker would need distributed transactions, which the outbox deliberately avoids. The duplicates are made harmless by idempotent consumers downstream.

Q: Name the two common families of outbox relay implementations.
A: The "polling publisher", where a relay process periodically queries the outbox table for unsent rows, publishes them, and marks them sent. And "transaction log tailing" (change data capture), where a tool like Debezium reads the database's write-ahead log and emits an event for every outbox row insert. CDC tailing avoids polling lag and load but requires log-level access to the database.

Q: What is the main drawback of a polling-publisher outbox relay versus log tailing?
A: Polling adds latency (you only publish as often as you poll) and load (every poll queries the table even when empty), and you must manage marking or deleting sent rows to avoid rescanning them. Log tailing reads the write-ahead log instead, so it reacts to commits with near-zero added query load and lower latency. The trade-off is that tailing needs replication-level database privileges and a CDC connector.

Q: What does CDC stand for and how does it relate to the outbox?
A: CDC is change data capture — reading a database's transaction log (the WAL in PostgreSQL) to stream every committed row change as an event. For the outbox, a CDC tool tails the log and emits an event for each new outbox-table insert, so you get the relay "for free" without polling. Debezium is the canonical open-source CDC platform used this way.

Q: Which PostgreSQL feature does Debezium use to capture the outbox table's inserts?
A: Logical replication via a replication slot reading the write-ahead log (WAL) through an output plugin such as `pgoutput`. Debezium decodes the WAL into change events, so an `INSERT` into the outbox table becomes a create event. You must run PostgreSQL with `wal_level = logical` for this to work.

Q: What is the role of the Debezium Outbox Event Router?
A: It is a Kafka Connect Single Message Transform (SMT) that takes the raw CDC change event for an outbox-row insert and reshapes it into a clean, business-meaningful Kafka message: extracting the payload, routing it to the right topic, and setting the key and headers. Without it, consumers would see a generic Debezium envelope describing a row insert rather than a domain event. It turns the outbox table into a set of clean event topics.

Q: What is the fully qualified class name of the Debezium relational Outbox Event Router SMT?
A: `io.debezium.transforms.outbox.EventRouter`. You add it to the Kafka Connect connector configuration as a transform of this type. There is a separate `io.debezium.connector.mongodb.transforms.outbox.MongoEventRouter` for MongoDB sources.

Q: What columns does the Debezium Outbox Event Router expect by default in the outbox table?
A: By default it expects `id`, `aggregatetype`, `aggregateid`, `type`, and `payload`. `id` is the unique event id, `aggregatetype` drives the topic, `aggregateid` becomes the message key, `type` is the event type, and `payload` is the event body. Each of these is configurable if your column names differ.

Q: By default, which outbox column does the Debezium router use to choose the Kafka topic?
A: The `aggregatetype` column, controlled by the `route.by.field` option whose default value is `aggregatetype`. So a row with `aggregatetype = 'Order'` is routed based on that value. This lets one outbox table feed many topics, one per aggregate type.

Q: What is the default Debezium topic-naming expression and what topic does `aggregatetype = 'Order'` produce?
A: The `route.topic.replacement` option defaults to `outbox.event.${routedByValue}`, so `aggregatetype = 'Order'` produces the topic `outbox.event.Order`. The `${routedByValue}` placeholder is substituted with the value of the routing column. You override `route.topic.replacement` to change the prefix or naming scheme.

Q: Which outbox column becomes the Kafka message key by default in the Debezium router, and why does the key matter?
A: The `aggregateid` column, set by `table.field.event.key` (default `aggregateid`). Using the aggregate id as the key means all events for the same aggregate (e.g. one order or one investor account) land in the same partition and stay strictly ordered. That ordering guarantee is essential for consumers that fold events into per-aggregate state.

Q: What does the `table.field.event.id` option default to, and where does that value end up?
A: It defaults to `id`, the unique event identifier column. The router places this id into a message header (by default a header named `id`) so consumers can deduplicate on it. This event id is the linchpin of idempotent-consumer deduplication.

Q: What does `table.field.event.payload` default to, and what happens to that column?
A: It defaults to `payload`. The router uses that column's contents as the Kafka message value (the event body), typically JSON, rather than emitting Debezium's generic row-change envelope. So consumers receive the clean domain payload your service wrote, not a CDC wrapper.

Q: What is `table.fields.additional.placement` used for in the Debezium router?
A: It lets you carry extra outbox columns into the emitted message, placing each either in a Kafka header or in the message envelope. The syntax is `column:placement[:alias]`, where placement is `header` or `envelope`. This is how you propagate things like a trace id, tenant, or schema version alongside the payload.

Q: What does `route.tombstone.on.empty.payload` control and what is its default?
A: It controls whether an outbox row with an empty or null payload causes the router to emit a Kafka tombstone (a record with a null value), and it defaults to `false`. Tombstones are used to signal deletion in log-compacted topics. With the default `false`, an empty payload does not produce a tombstone.

Q: Why is JSON or Avro in the `payload` column preferable to letting Debezium emit the raw row-change envelope?
A: The raw CDC envelope describes a database operation ("a row was inserted with these columns") and is coupled to your table schema, leaking internal storage details to consumers. A curated `payload` is a stable, versioned domain event contract decoupled from the table. The outbox router exists precisely to swap the storage-shaped envelope for the contract-shaped payload.

Q: In a fund order service, what would a typical row in the outbox table look like conceptually?
A: For booking a subscription order you might write a row with `aggregatetype = 'Order'`, `aggregateid` = the order id, `type = 'OrderCreated'`, and `payload` = JSON containing the ISIN, investor id, amount, and currency. It is inserted in the same transaction that inserts the order itself. Debezium then routes it to `outbox.event.Order` keyed by the order id.

Q: An idempotent consumer is one that does what?
A: It processes the same message more than once without producing a different or duplicated effect — applying the event twice yields the same final state as applying it once. This is required because the outbox relay is at-least-once and brokers redeliver after failures. Idempotency is what makes at-least-once delivery safe to build on.

Q: How do you typically make a consumer idempotent in the outbox pattern?
A: Give every event a unique id (the outbox `id`), and have the consumer record processed ids in a deduplication table or store; before applying an event it checks whether that id was already handled and skips it if so. The processed-id record and the resulting state change are written in the same local transaction. This guarantees the effect happens exactly once even though delivery is at-least-once.

Q: Why must the dedup-id record and the state change be written in the same transaction in an idempotent consumer?
A: If you applied the state change first and then crashed before recording the id, a redelivery would re-apply the change and double its effect; if you recorded the id first and then crashed, the change would be lost but treated as done. Writing both atomically closes both windows. It is the consumer-side mirror of the outbox's producer-side atomicity.

Q: Besides a dedup table, name another way to achieve idempotent application of an event.
A: Make the operation itself naturally idempotent — e.g. an upsert keyed by a stable id, or a "set position to X" assignment rather than "add to position". Conditional updates guarded by a version or sequence number also work. When the operation is inherently idempotent you may not need an explicit dedup table at all.

Q: What is a saga?
A: A saga is a way to maintain data consistency across multiple services without a distributed (two-phase-commit) transaction, by breaking a business transaction into a sequence of local transactions. Each local transaction commits in its own service and triggers the next step via an event or command. If a step fails, the saga runs compensating transactions to undo the previously committed steps.

Q: Why can't you just use a single ACID transaction or two-phase commit for a multi-service trade lifecycle?
A: Each microservice owns its own database, so there is no single transaction manager spanning them, and two-phase commit (2PC) is a blocking, tightly-coupled protocol that scales and fails poorly across independent services and brokers. The saga trades the strong isolation of 2PC for availability and loose coupling, accepting that consistency is achieved over time via compensations. The architect's job is to design those compensations so the eventual state is correct.

Q: Define choreography-based saga coordination.
A: In choreography there is no central coordinator: each service performs its local transaction and publishes domain events, and other services subscribe to those events and react with their own local transactions. The workflow emerges from the chain of events. Control is fully distributed across the participants.

Q: Define orchestration-based saga coordination.
A: In orchestration a central orchestrator object tells each participant which local transaction to execute next, stores and interprets the state of the overall saga, and drives compensations on failure. Participants do work when commanded and report back. The flow logic lives in one place rather than being implicit in event subscriptions.

Q: Give two advantages of choreography over orchestration.
A: It needs no extra coordination service and introduces no single point of failure because responsibility is spread across participants, and it is well suited to simple workflows with few services. It also keeps each service autonomous, reacting only to events it cares about. The cost is that the overall flow is implicit and harder to see.

Q: Give two drawbacks of choreography.
A: The workflow becomes hard to follow as you add steps — it is difficult to know which service responds to which event — and there is a risk of cyclic dependencies because participants consume each other's events. Integration testing is also harder because you must run all participating services to exercise one transaction. These costs grow with workflow complexity.

Q: Give two advantages of orchestration over choreography.
A: It handles complex workflows and the addition of new services more cleanly, and it avoids cyclic dependencies because the orchestrator centrally manages the flow. It also gives a clear separation of responsibilities and a single place to observe and reason about saga state. The flow is explicit rather than emergent.

Q: What is the principal drawback of orchestration?
A: The orchestrator is additional design and infrastructure that you must build and operate, and it concentrates the coordination logic in one place that becomes a potential single point of failure. You also risk pushing too much business logic into the orchestrator, turning participants into anaemic CRUD services. Mitigations include making the orchestrator highly available and keeping business rules in the services.

Q: When should you reach for orchestration instead of choreography?
A: When the workflow is complex, has many steps or participants, requires clear visibility into saga state, or is likely to change and grow — orchestration's centralized control and explicit flow pay off. Choreography is preferable for short, stable, low-fanout flows. A 3-step subscription saga could go either way, but a full trade lifecycle with many participants argues for orchestration.

Q: A 3-step subscription saga is "reserve cash, book units, confirm". Sketch how this runs as a choreography.
A: The order service emits `CashReservationRequested`; the cash service reserves cash and emits `CashReserved`; the register service books units and emits `UnitsBooked`; the order service then confirms and emits `SubscriptionConfirmed`. Each step is a local transaction triggered by the prior event. There is no orchestrator — the flow is the chain of events.

Q: What is a compensating transaction in a saga?
A: It is a local transaction that semantically undoes the effect of a previously committed step when a later step fails — for example releasing reserved cash after unit booking fails. Because the original step already committed in its own service, you cannot roll it back automatically; you must explicitly perform an inverse action. Compensations are how a saga reaches a consistent state after a partial failure.

Q: Why is "semantic undo" the right phrase for compensation rather than "rollback"?
A: A database rollback removes an uncommitted change as if it never happened, but saga steps have already committed and may have had observable side effects, so you cannot literally rewind them. Instead you apply a new transaction whose business meaning negates the original — release the reservation, issue a refund, cancel the booking. The history still shows the original action plus its compensation, which is exactly what an auditor wants to see.

Q: In the subscription saga, what is the compensation for "reserve cash" when "book units" fails?
A: A "release cash reservation" local transaction in the cash service that frees the previously reserved amount back to the available balance. It is triggered when the saga learns that unit booking failed. After it commits, no cash is left reserved against units that were never issued.

Q: What are the three types of transaction in Richardson's saga model?
A: Compensatable transactions, which can be undone by a later compensating transaction; the pivot transaction, which is the go/no-go point of no return; and retriable transactions, which follow the pivot and are guaranteed to eventually succeed. Ordering steps into these three categories is central to a clean compensation design. Everything before the pivot must be compensatable; everything after must be retriable.

Q: What is the pivot transaction in a saga?
A: It is the point of no return: once it commits, the saga is committed to completing and there is no going back via compensation. A pivot may be an irreversible step (it cannot be undone), or simply the boundary — the last compensatable step or the first retriable one. Designing where the pivot sits is a key part of saga design because it fixes which steps need compensations.

Q: What is a retriable transaction and why must it never fail permanently?
A: A retriable transaction comes after the pivot and is designed to be idempotent and to eventually succeed through retries even after transient failures. It must not fail permanently because there is no compensation path after the pivot — the saga is already committed to finishing. So steps that genuinely could fail in a business sense belong before the pivot, as compensatable transactions.

Q: Why should you order saga steps so that all compensatable steps precede the pivot and all retriable steps follow it?
A: Because anything before the pivot can be cleanly undone, you put the steps most likely to fail (validations, reservations, business-rule checks) there. Once you pass the pivot you can no longer compensate, so everything after must be guaranteed-to-succeed retriable work. Getting this ordering right is what makes a partial failure recoverable rather than corrupting.

Q: What is the fundamental ACID property a saga gives up, and what does that cause?
A: It gives up Isolation — the "I" in ACID — because the intermediate, uncommitted-from-a-business-view results of one saga are visible to other concurrent sagas. This lack of isolation causes anomalies such as lost updates, dirty reads, and fuzzy (non-repeatable) reads. Countermeasures must be designed in deliberately to restore enough isolation.

Q: Name the three classic data anomalies that arise from a saga's lack of isolation.
A: Lost updates, where one saga overwrites a change made by another saga; dirty reads, where a saga reads data another saga has modified but not yet finalized (and may later compensate away); and fuzzy or non-repeatable reads, where two reads in the same saga see different values because another saga wrote in between. These are the concurrency hazards the saga's missing isolation exposes.

Q: What is the "dirty read" anomaly in a saga, with a fund example?
A: A dirty read is when a saga reads a value that another in-flight saga has changed but might still compensate (undo). For example saga A reserves cash, lowering an investor's available balance; saga B reads that lowered balance and rejects a second subscription — but then A fails and releases the cash, so B rejected based on a value that was rolled back. The read was of dirty, uncommitted-from-the-business-view data.

Q: What is the "dirty compensation" / dirty-write hazard that compensation design must avoid?
A: It is when a compensating transaction overwrites or releases something based on a stale assumption, undoing more than it should because another saga touched the same data in between. For instance releasing a fixed cash amount when part of it was meanwhile legitimately consumed by another operation. Avoiding it requires countermeasures like rereading the value or using semantic locks so compensation acts only on what its own saga actually changed.

Q: What is the "semantic lock" countermeasure?
A: A semantic lock is an application-level flag a compensatable transaction sets on any record it creates or updates, marking it as "in progress / not yet final". Other sagas that encounter the flag know the data may still change and can wait, fail, or handle it explicitly. It restores a form of isolation without a database lock, at the cost of having to manage and clear the flag.

Q: What are "commutative updates" as a saga countermeasure?
A: They are updates designed so that applying them in any order produces the same final result, which removes the lost-update and ordering hazards between concurrent sagas. For example crediting and debiting a balance commute, whereas "set balance to X" does not. Where you can model state changes as commutative deltas, two sagas interleaving cannot corrupt each other.

Q: Explain the "pessimistic view" countermeasure.
A: It reorders the saga so that the riskier updates — those that grant a resource or increase a value others might read — are moved to retryable transactions after the pivot, so no other saga can dirty-read a temporary grant that later gets compensated away. By the time the value is visible it will not be rolled back. It trades some flexibility in step ordering for elimination of certain dirty reads.

Q: Explain the "reread value" countermeasure.
A: Before a step updates a record, it rereads the record and verifies it has not changed since the saga last saw it; if it changed, the saga aborts or restarts that step rather than overwriting another saga's update. This prevents lost updates and dirty writes. It is essentially optimistic concurrency applied within the saga.

Q: Explain the "version file" countermeasure.
A: A version file records the sequence of operations performed on a record so that operations which are not naturally commutative can be reordered and applied in the correct logical order, effectively making them commutative. It lets out-of-order saga messages be sorted before they take effect. This protects against anomalies caused by reordering of concurrent operations.

Q: What is the "by value" (risk-based concurrency) countermeasure?
A: It chooses the concurrency mechanism dynamically based on the business risk of each request: low-risk operations use sagas with their relaxed isolation, while high-risk operations (large amounts, sensitive accounts) fall back to stricter mechanisms such as distributed transactions. You spend the cost of strong isolation only where the stakes justify it. It is a pragmatic, value-driven mix rather than one policy everywhere.

Q: Why does a regulated fund platform care intensely about getting compensation design right?
A: Because a half-completed subscription that reserves cash but never issues units, with no clean compensation, leaves money tied up against nothing — a reconciliation and reporting problem the transfer agent and auditors will surface. Incorrect compensation can also double-release or mis-state investor positions, breaching the accuracy obligations around NAV and the share register. The architect must design failures to resolve automatically into a provably consistent state.

Q: What does "timeout and dead-man handling" mean in a saga?
A: It is what the saga does when a participant never replies — neither success nor failure — because it crashed, the message was lost, or it is hung. The saga cannot wait forever, so it sets a timeout after which it treats the step as failed and triggers compensation, or routes the stuck message for human handling. Without this, one silent participant freezes the whole business transaction.

Q: What happens to a saga step that hangs indefinitely if you have no timeout?
A: The saga blocks on that step forever, leaving earlier steps' changes (e.g. reserved cash) committed but uncompensated and the business transaction neither completed nor reversed. Resources stay locked or reserved, and the inconsistency accumulates silently. A timeout converts the silent hang into a definite outcome the saga can act on.

Q: After a saga step times out, what are the safe options the design must choose between?
A: Either treat the step as failed and run compensations to unwind the saga, or retry the step if it is past the pivot and must eventually succeed, or park the case for human intervention with full diagnostic context. The right choice depends on whether the step is compensatable or retriable. The decision must be deterministic and recorded, never left to chance.

Q: Why is idempotency a prerequisite for safely retrying or timing out saga steps?
A: Because after a timeout you do not know whether the participant actually performed the work before going silent, so a retry might be a duplicate. Only if the operation is idempotent can you retry safely without double-booking units or double-reserving cash. Idempotency is therefore baked into both the consumer side and the retriable steps of a saga.

Q: Where do saga orchestrators like Temporal and Camunda fit?
A: They are workflow/orchestration engines that act as the saga orchestrator: they durably persist the saga's state, drive each step, enforce timeouts, and run compensations automatically on failure, so you do not hand-code that coordination. They turn the orchestration pattern into managed, restartable, observable workflows. They are previewed here and revisited in later lessons as the production way to run complex sagas.

Q: What does an orchestration engine give you over hand-rolled choreography for a complex trade lifecycle?
A: Durable state that survives crashes and resumes exactly where it left off, built-in timers and timeouts, automatic retry and compensation handling, and a single place to observe and audit the whole workflow. Hand-rolled choreography scatters this across many event handlers and is hard to see or debug. For a long, multi-step regulated workflow, the engine's durability and visibility are decisive.

Q: Why is debugging a saga harder than debugging a single transaction, and how do you mitigate it?
A: Because the business operation is spread across several services and (in choreography) several event hops, no single stack trace or log shows the whole flow, and partial failures interleave with compensations. You mitigate it with correlation/trace ids carried on every event, centralized saga-state tracking or an orchestrator, and an audit log that records each step and compensation. The lesson's "Done when" requires the audit log alone to explain a failure.

Q: What does "the audit log alone explains the whole failure-and-compensation sequence" require of your event design?
A: Every step — each local transaction, each failure, and each compensation — must emit a durable, correlated, timestamped record, so that reading the log end to end tells the full story without inspecting service internals. Events need a shared correlation id and meaningful types. This is exactly what event sourcing and a disciplined outbox give you, and it is the regulated-finance acceptance bar.

Q: How do the outbox pattern and sagas combine in this lesson's design?
A: Each saga step is a local transaction that, in the same database commit, writes its state change and an outbox event; Debezium relays those events to topics, and the next participant (or orchestrator) consumes them. This guarantees that every saga step's event is published exactly when and only when the step committed — no dual-write loss. The outbox is what makes the saga's event-driven coordination reliable.

Q: A consumer is processing duplicate events and double-booking units. What is the first thing to check?
A: Whether the consumer is idempotent — does it deduplicate on the event id before applying the effect? The outbox relay is at-least-once, so duplicates are expected; the consumer, not the producer, must absorb them. Check that a dedup record keyed by the outbox `id` is written in the same transaction as the booking and that the check actually runs before the effect.

Q: Your Debezium outbox events are all landing on one topic instead of per-aggregate topics. What do you check first?
A: Check `route.by.field` and the `aggregatetype` column: if every row has the same (or null) `aggregatetype`, or `route.by.field` points at the wrong column, all events route to the same `route.topic.replacement` result. Confirm the outbox rows actually carry distinct aggregate types. Then verify `route.topic.replacement` is producing the topic names you expect.

Q: Debezium is emitting verbose row-change envelopes instead of clean payloads on your topics. What is likely misconfigured?
A: The Outbox Event Router SMT is probably not applied (or not first) in the connector's `transforms` chain, so consumers see the raw CDC envelope rather than the extracted `payload`. Confirm the transform of type `io.debezium.transforms.outbox.EventRouter` is configured and that `table.field.event.payload` points at your payload column. Without the router, the payload extraction and routing do not happen.

Q: Orders are committing but no events ever reach Kafka. Walk through the first checks.
A: Confirm the outbox rows are actually being inserted in the same transaction (look in the table), then confirm PostgreSQL has `wal_level = logical` and the Debezium connector's replication slot is active and not erroring. Check the connector status and logs for the outbox table being captured. The failure is usually a CDC/connector problem, not the application, if the rows exist but no topic messages appear.

Q: After a Debezium connector restart you see duplicate outbox events. Is this a bug?
A: No — CDC and the outbox relay are at-least-once, so on restart Debezium may re-emit events it had read but not fully committed offsets for. The correct fix is not to chase exactly-once but to ensure consumers deduplicate on the event id. Duplicates after restart are expected and must be absorbed downstream.

Q: A saga left cash reserved but no units booked, and nobody noticed for days. What design failures does this reveal?
A: The compensation for the failed unit-booking step was missing or never triggered, and there was no timeout/dead-man handling to detect the stuck step, and no alerting or audit visibility to surface the dangling reservation. The reserved-cash-without-units state should have been compensated automatically and, if not, raised an alert. It is exactly the reconciliation nightmare the lesson says the architect must design out.

Q: In choreography, two services keep triggering each other's events in a loop. What anti-pattern is this?
A: A cyclic dependency between saga participants — a known drawback of choreography where services consume each other's events and form a cycle. It makes the flow confusing and can create infinite or unintended loops. The standard remedy is to switch to orchestration, where a central orchestrator manages the order and breaks the implicit cycle.

Q: Your compensation released a fixed reserved amount, but the investor's available cash is now wrong. What likely went wrong?
A: A dirty-write / dirty-compensation anomaly: another concurrent operation changed the balance between reservation and compensation, and the compensation released a stale fixed amount rather than reconciling against the current state. The fix is a countermeasure such as rereading the value before compensating, or a semantic lock so only this saga's reservation is released. Compensation must act on what this saga actually changed, not a remembered number.

Q: Why does the outbox `id` need to be globally unique and stable, and what breaks if it is reused?
A: Idempotent consumers deduplicate on that id, so if two distinct events share an id, the second is wrongly skipped as a duplicate and its effect is lost; if the same logical event gets different ids on redelivery, dedup fails and it is applied twice. A UUID generated once at insert time gives the needed uniqueness and stability. The id is the contract between the outbox and every idempotent consumer.

Q: For the subscription saga, where would you place the pivot transaction and why?
A: The pivot is naturally at "book units" or the confirmation: before it you have only reserved cash (fully compensatable by releasing the reservation), and once units are issued in the share register you are committed to finishing and confirming. Cash reservation and validations sit before the pivot as compensatable steps; confirmation and notifications sit after as retriable. This ordering means any business failure happens while you can still cleanly unwind.

Q: How does writing events into an outbox preserve event ordering for a single aggregate?
A: Events for one aggregate are inserted in commit order, and because the Debezium router keys messages by `aggregateid`, all of an aggregate's events go to the same partition where Kafka preserves their order. Consumers folding those events therefore see them in the right sequence. Cross-aggregate global ordering is not guaranteed and usually not needed.

Q: Why might an architect deliberately keep a simple workflow as a choreography rather than introducing an orchestrator?
A: For a short, stable, low-fanout flow, an orchestrator is extra infrastructure, an extra failure point, and extra operational burden that buys little; choreography keeps services autonomous and avoids a central coordinator. The drawbacks of choreography (poor visibility, cyclic risk) only bite as the workflow grows. Matching the coordination style to the workflow's complexity is the design judgment being taught.

Q: What is the difference between a command and an event in saga messaging?
A: A command is an instruction to do something that may still be rejected (e.g. "reserve cash"), whereas an event is a statement that something already happened and is a fact (e.g. "cash reserved"). Orchestration tends to send commands to participants; choreography propagates events that participants react to. Mixing the two clearly in your topic and message design keeps intent and history distinct.

Q: How do you make the relay step of the outbox tolerant of broker outages?
A: The outbox rows are already durably committed in the database, so if the broker is down the relay simply retries publishing later; nothing is lost because the source of truth is the table, not the in-flight publish. For CDC, Debezium tracks its log offset and resumes from where it left off. This is the whole point of decoupling capture from relay.

Q: What gotcha arises if you delete outbox rows immediately after publishing in a polling relay?
A: If you delete before the publish is confirmed durable on the broker, a crash can lose the event with no trace; if you delete inside the same transaction as the business change you re-introduce a dual-write. The safe pattern is to publish, confirm, then mark or delete the row, accepting at-least-once and relying on consumer idempotency. With CDC tailing you can often retain or prune rows independently of publishing.

Q: Why does at-least-once plus idempotent consumers give effectively-once processing, and how is that different from exactly-once delivery?
A: At-least-once guarantees no event is lost but allows duplicates; an idempotent consumer makes duplicates harmless, so the observable effect happens exactly once even though delivery did not. Exactly-once delivery (no duplicate ever reaches the consumer) is far harder and usually unnecessary. The pragmatic, production answer is at-least-once delivery with effectively-once processing.

Q: In a transfer-agency context, why is an event-sourced/outbox-backed audit trail attractive to regulators?
A: Because every state change — order, allocation, cash movement, compensation — is captured as a durable, ordered, timestamped event you cannot have skipped, the log itself becomes the evidence of what happened and when. Auditors can reconstruct any investor's position and the full lifecycle of a subscription from the events alone. This "audit by construction" is far stronger than reconstructing history from mutable tables.

Q: What could go wrong if the outbox table and business tables live in different databases?
A: You lose the single local transaction that gives atomicity, reintroducing the very dual-write race the outbox was meant to remove — the business write could commit while the outbox write fails. The outbox only works when its insert shares one transaction with the state change in the same database. Keeping them co-located is a non-negotiable invariant of the pattern.

Q: How would you propagate a correlation id through an outbox-driven saga so the audit log ties together?
A: Put the correlation (saga) id in the outbox payload and/or as an additional column surfaced via `table.fields.additional.placement` into a Kafka header, so every event for the saga carries it. Each participant copies it onto the events it emits. Then a query or trace on that id reconstructs the entire saga, including failures and compensations.

Q: A retriable post-pivot step keeps failing permanently. What does that tell you about the saga design?
A: That a step which can genuinely fail in a business sense was wrongly placed after the pivot, where there is no compensation path, so the saga is stuck committed but unfinished. The fix is to move that risk before the pivot as a compensatable step, or to make the post-pivot step truly always-eventually-succeed (only transient failures). Post-pivot steps must not have business-rule failure modes.

Q: Why should compensating transactions themselves be idempotent?
A: Because a timeout or retry may cause the same compensation to be requested more than once (e.g. "release reservation" twice), and a non-idempotent compensation could over-release or error. Making it idempotent means re-running it has no extra effect once the undo is done. The same dedup or conditional-update discipline applies to compensations as to forward steps.

Q: What is the danger of putting too much business logic into a saga orchestrator?
A: The participants degrade into anaemic CRUD services and the orchestrator becomes a god object that is hard to change and a heavy single point of failure. The intended balance is for the orchestrator to coordinate sequencing and compensation while each service keeps its own business rules. Keeping logic in the services preserves their autonomy and testability.

Q: How does Debezium's CDC approach to the outbox reduce load compared with application polling?
A: Polling repeatedly queries the outbox table even when there is nothing new, adding query load and latency proportional to the poll interval; CDC instead reads the write-ahead log the database already writes, reacting to commits with essentially no extra query load. There is no "is there anything new?" loop. The trade-off is the operational cost of running and securing the CDC connector and replication slot.

Q: Why is an outbox event's `type` column useful to consumers?
A: It names the kind of event (e.g. `OrderCreated`, `OrderCancelled`) so a consumer can dispatch on type without inspecting the payload structure, and so a single topic can carry several event types for an aggregate. Debezium surfaces it (commonly as a header) for routing and handling. It is part of the stable event contract.

Q: A subscription saga must reverse "book units" if a later AML check fails. Is "book units" compensatable, and what is its compensation?
A: Yes, if you place the AML check after unit booking but before the pivot, then booking is compensatable and its compensation is a "reverse units" local transaction in the share register that removes the issued units. The original booking and its reversal both remain in the event log for audit. If instead unit issuance is your point of no return, the AML check must run before it.

Q: Why is "the system is consistent after the failure path" a stronger claim than "no error was thrown"?
A: An error-free run only means nothing crashed; consistency means that after a partial failure and its compensations, no resource is left half-committed — no cash reserved against unissued units, no orphaned bookings. Sagas deliberately allow steps to fail, so success is measured by the final reconciled state, not by absence of errors. That is why the lesson's acceptance test inspects the end state and audit log.

Q: How do snapshots of saga state in an orchestrator help with crash recovery?
A: A durable orchestrator persists the saga's current step and accumulated state, so after a crash it reloads that state and resumes the workflow exactly where it stopped rather than restarting or losing in-flight sagas. This durability is what distinguishes an engine like Temporal from coordinating in process memory. It is the reason orchestration engines are preferred for long-running, multi-step sagas.

Q: Why can the lack of isolation in sagas be acceptable in many fund-platform flows despite the anomalies?
A: Because many flows are not highly contended on the same records, the anomalies are rare, and targeted countermeasures (semantic locks, rereads) cover the cases that do contend, while the gain in availability and loose coupling is large. The architect applies the "by value" idea: strong isolation only where business risk demands it. Blanket distributed transactions would be far costlier for little benefit on most paths.

Q: What is the difference between "compensatable" and "irreversible" in saga design?
A: A compensatable step can be semantically undone by a later compensating transaction; an irreversible step cannot be undone at all and therefore acts as (or sits at) the pivot, after which the saga must complete. Recognizing which real-world actions are irreversible (sending a SWIFT payment, issuing units to a registrar) tells you where the pivot must be. You arrange compensatable risk before any irreversible action.

Q: Why might sending an external SWIFT payment force the pivot earlier in a payment-bearing saga?
A: Because a dispatched SWIFT message generally cannot be un-sent, that step is effectively irreversible and becomes a point of no return, so all compensatable validation and reservation must precede it. After it, the saga can only move forward with retriable steps. The architect orders the saga so that anything that might fail in a business sense happens before the irreversible external send.

Q: Why is "every consumer survives a replayed event without double-effect" one of the lesson's done-when criteria?
A: Because replays are inevitable — broker redelivery, CDC restarts, reprocessing — and the entire reliability story rests on consumers being idempotent. If a single replay can double-book units or double-release cash, the outbox/saga design is unsafe in production. The criterion forces you to prove idempotency rather than assume it.

Q: How can you test the saga's failure path deliberately as the lesson's "Do" step requires?
A: Force the step-2 unit booking to fail (e.g. inject an error or a rejecting stub), then observe that the saga triggers the cash-release compensation, the final state shows no reservation and no units, and the audit log records the failure and compensation in order. You verify consistency from the end state and the log, not from the absence of exceptions. Repeating it confirms the compensation is idempotent.

Q: Why is the outbox pattern preferable to "publish then write to the database" ordering?
A: Publishing first risks announcing an event for a state change that then fails to commit, so consumers act on something that never happened (a phantom event). Writing the database first then publishing risks the crash-before-publish lost-event case. The outbox sidesteps the whole ordering dilemma by making the event part of the single committed transaction and relaying it afterward.

Q: In a saga, what is the consequence of a participant processing a stale (out-of-order) compensation message?
A: It may undo work that was never done or has already moved on, corrupting state — for example releasing a reservation that a newer saga has already legitimately consumed. Countermeasures like a version file (ordering operations) or rereading current state before compensating guard against this. Compensations must be guarded so they act only when the original effect is actually present.

Q: How would you size the alerting around a saga so a stuck step is noticed?
A: Track saga age and per-step latency, and alert when a saga or a step exceeds its expected timeout, plus alert on reserved-but-not-confirmed resources (e.g. cash reserved beyond a threshold age). These signals surface dead-man steps that timeouts should be acting on. The point is that a dangling reservation pages someone rather than sitting unnoticed for days.

Q: Why is co-locating the dedup store with the consumer's business data important?
A: So the processed-id record and the resulting business write commit in one local transaction, giving true exactly-once effect; a remote or separate dedup store reintroduces a dual-write between marking-processed and applying-effect. Locality is what makes the idempotency atomic. It mirrors the producer-side rule that the outbox row must share the business transaction.

Q: What is the simplest way to verify your subscription saga's audit log "tells the whole story" without manual digging?
A: Filter the event log by the saga's correlation id and read it top to bottom: it should show cash reserved, the booking attempt, the booking failure, the cash-release compensation, and a terminal failed state, each timestamped. If you can narrate the entire incident from that filtered log alone, the criterion is met. If you must inspect service databases to understand it, the event design is incomplete.

Q: Summarize, in one breath, why the outbox plus a well-designed saga turns event-driven theory into production-shaped reflexes for fund administration.
A: The outbox makes every state-change-plus-event atomic so no event is ever lost to a dual-write crash, idempotent consumers make at-least-once delivery effectively-once, and a saga with the right pivot, compensations, timeouts, and countermeasures makes a partial failure resolve into a provably consistent state explained entirely by the audit log. Together they ensure a half-completed subscription never leaves cash reserved against unissued units. That reliability and auditability is exactly what a regulated transfer-agency platform must guarantee.

Q: What does the Debezium router's `table.op.invalid.behavior` option control, and what is its default?
A: It controls how the SMT reacts when the outbox table receives a non-`INSERT` change such as an `UPDATE` or a `DELETE` it cannot simply filter, and it defaults to `warn`. The other values are `error` (the connector raises an error for that record) and `fatal` (the connector stops). The default `warn` reflects the design assumption that outbox rows are insert-only and append-only — you write a row once and never mutate it.

Q: By default, how does the Debezium Outbox Event Router treat a `DELETE` on the outbox table?
A: It silently filters the `DELETE` out, emitting nothing, because the router is built on the assumption that the outbox is insert-only and deletes are just housekeeping/pruning of already-relayed rows. This lets you safely purge old outbox rows without producing spurious events. Only `INSERT` rows become domain events; `UPDATE` rows fall under `table.op.invalid.behavior`.

Q: What does `table.field.event.timestamp` default to, and when would you set it?
A: It defaults to empty, meaning the emitted message simply uses Debezium's own event timestamp (when the change was captured). You set it to an outbox column when you want the message timestamp to reflect a business time you wrote into the row — e.g. the order's trade timestamp — rather than the CDC capture time. This matters when consumers or audit tooling key off event time and you need it to match the domain event, not the relay moment.

Q: What is `table.expand.json.payload`, what does it default to, and why might you enable it?
A: It is a boolean (default `false`) that tells the router to parse a JSON string in the payload column and expand it into a structured Kafka Connect record rather than passing it through as an opaque string. You enable it when you want a real schema'd message (e.g. for Avro/schema-registry consumers) instead of a raw JSON blob. If the content is not valid JSON it is kept as-is, so malformed payloads do not crash the transform.

Q: When `table.expand.json.payload` is enabled, what does `table.json.payload.null.behavior` govern, and what is its default?
A: It governs how a JSON `null` inside the expanded payload is represented in the emitted record, and it defaults to `ignore`, which drops the null field. The alternative behaviour optionally keeps the null using an optional schema so the field is present with a null value. You choose based on whether downstream consumers need to distinguish "field absent" from "field explicitly null".

Q: Where does the Debezium router put the outbox event id by default, and under what header name?
A: It places the value of the `table.field.event.id` column (default column `id`) into a Kafka message header under the key `id` by default. Consumers read this header to deduplicate. Because it travels as a header rather than inside the payload, idempotency tooling can inspect it without parsing the message body.

Q: How would you carry a saga correlation id and a schema version from outbox columns into Kafka headers with the router?
A: Set `table.fields.additional.placement` to something like `correlationid:header,schemaversion:header`, listing each extra column with the `header` placement (the syntax is `column:placement[:alias]`). Each named column is then emitted as a header on every message from that table. You would add `correlationid` and `schemaversion` columns to the outbox table and populate them in the same insert as the business change.

Q: In `table.fields.additional.placement`, what is the difference between the `header` and `envelope` placements?
A: `header` puts the extra column's value into a Kafka record header, where it is metadata consumers can read without deserializing the value; `envelope` puts it inside the message value structure alongside the payload, making it part of the event body. Use `header` for routing/trace metadata like correlation id and tenant, and `envelope` when the field is genuinely part of the domain event's data. The optional `alias` lets you rename the column in the output.

Q: Your outbox SMT is logging warnings every time you update an outbox row's `processed` flag. Why, and what is the cleaner design?
A: The warnings come from `table.op.invalid.behavior = warn` firing on each `UPDATE`, because the router expects insert-only rows and treats the update as an invalid operation. The cleaner design with CDC tailing is to never mutate outbox rows in place — leave the relay/offset tracking to Debezium and prune old rows with `DELETE` (which is filtered), rather than maintaining a `processed` column that belongs to the polling-relay model.

Q: Why does a polling-publisher relay typically need a `processed` flag or status column while a Debezium CDC relay does not?
A: A polling relay must remember which rows it has already published so it does not rescan and re-send them, so it marks each row processed (or deletes it) after a confirmed publish. Debezium instead tracks its position in the WAL via a replication-slot offset, so "what has been relayed" lives in the connector's offset, not in a table column. That is why CDC outbox tables can stay pure insert-only with no status column.

Q: If you choose a polling relay, what query pattern and index keep it efficient?
A: Query for unprocessed rows ordered by insertion, e.g. `SELECT ... WHERE processed = false ORDER BY id LIMIT n`, and back it with an index on the `processed` flag (or a partial index `WHERE processed = false`) so each poll touches only pending rows. Without that index every poll scans the whole growing table. A partial index keyed to the unprocessed predicate stays small because relayed rows drop out of it.

Q: What concrete problem does a partial index `WHERE processed = false` solve for a polling outbox over a high-volume order table?
A: As millions of orders accumulate, a full index on `processed` would index every relayed row too, growing unboundedly and slowing scans; a partial index only contains the small set of not-yet-relayed rows, so each poll's lookup stays fast and the index stays tiny. It directly targets the polling relay's hot query. Once a row is marked processed it leaves the partial index entirely.

Q: A consumer's dedup table is growing without bound. How do you keep it from becoming a liability?
A: Deduplicate ids only need to be retained for as long as a duplicate could plausibly be redelivered, so you can age out old dedup records (e.g. by a TTL or by partitioning the table by time and dropping old partitions) once redelivery is no longer possible. You must keep the window comfortably longer than the broker's and relay's maximum redelivery delay. Trimming too aggressively risks re-applying a late duplicate.

Q: How can you make a consumer idempotent without a separate dedup table when writing to PostgreSQL?
A: Use an `INSERT ... ON CONFLICT (event_id) DO NOTHING` against a uniqueness constraint on the event id, or fold the dedup into the business write with a unique key, so a duplicate insert is a no-op rather than a second effect. The database's unique constraint becomes the deduplication mechanism. This works when each event maps to at most one row keyed by the event id.

Q: Why is a database `UNIQUE` constraint on the event id a robust idempotency guard even under concurrency?
A: Because the constraint is enforced atomically by the database across all concurrent transactions, two simultaneous attempts to process the same event id cannot both insert — one commits and the other gets a unique-violation it can treat as "already done". A read-then-write dedup check, by contrast, has a race window where both readers see "not present" and both apply. The constraint closes that window without an explicit lock.

Q: In Kafka, what `enable.idempotence` producer setting relates to the outbox, and what does it and does it not solve?
A: A Debezium/Kafka producer with `enable.idempotence=true` prevents the producer from writing the same record twice to a partition due to internal retries, de-duplicating at the produce step. It does not give end-to-end exactly-once across a relay restart or offset replay, so the consumer must still deduplicate on the business event id. Producer idempotence narrows one duplication source but is not a substitute for idempotent consumers.

Q: Why is keying outbox messages by `aggregateid` not enough on its own to guarantee correct per-aggregate ordering at the consumer?
A: Keying sends an aggregate's events to one partition so Kafka preserves their order on the broker, but a multi-threaded or rebalancing consumer can still process them out of order or concurrently unless it processes a partition's records sequentially. You must also ensure the consumer applies records from a partition in offset order and does not parallelize within a key. Ordering is a property of both the partition and the consumer's processing discipline.

Q: If you need strict global ordering of all outbox events, what is the cost, and why is it usually avoided?
A: Strict global ordering forces a single partition (so all events share one ordered log), which caps throughput at one consumer's pace and removes parallelism. It is usually avoided because correctness only requires per-aggregate ordering — events for one order or one investor — which partitioning by `aggregateid` already gives while still scaling across aggregates. You pay a large throughput price for an ordering guarantee the domain rarely needs.

Q: A new microservice needs the order events but starts consuming today. How does the outbox/CDC design let it read history?
A: If the events topic is retained long enough (or is log-compacted/tiered), the new consumer joins with `auto.offset.reset=earliest` and replays from the beginning of the topic to build its state. The outbox plus durable Kafka topic means the event history is the shared source of truth a late joiner can fold. If retention has already expired, you instead reseed from a snapshot or replay from the source database's outbox if rows were retained.

Q: What is the risk of relying on Kafka topic retention as your only event history, and how does it interact with late consumers?
A: If retention is time- or size-bounded, events age out and a consumer that joins or needs to rebuild after that window cannot replay the full history, breaking the "fold the log to get state" guarantee. Mitigations are longer retention, log compaction (keeps the latest per key), tiered storage, or keeping the outbox rows as the durable archive. The architect must decide where the long-term source of truth actually lives.

Q: For an outbox-driven event-sourced share register, why might you enable Kafka log compaction on the topic?
A: Log compaction retains at least the most recent event per key indefinitely, so a consumer rebuilding state always finds the latest value for each aggregate even after older records are cleaned. For a keyed register where you need the current per-aggregate state to survive, compaction gives a compact, replayable snapshot of latest-per-key. It is not appropriate where every historical event must be preserved for audit — there you keep full retention.

Q: How would you wire a trace/correlation id end-to-end so a single saga is traceable across all participants' events?
A: Generate the correlation id at saga start, store it as an outbox column, surface it via `table.fields.additional.placement` as a header, and have every consumer copy that header onto the outbox rows it writes for its own step. Then a search on that one id in the event store or tracing system returns every step and compensation in order. The discipline is that no participant ever drops or regenerates the id.

Q: What is the difference between a correlation id and a causation id in saga event tracing?
A: A correlation id is constant for the whole saga and ties all its events together, while a causation id on each event points to the specific event or command that directly caused it, forming a parent-child chain. Correlation answers "which saga?"; causation answers "what triggered this exact step?". Carrying both lets you both group a saga and reconstruct the precise sequence of cause and effect.

Q: A saga's compensation ran but the original forward step's effect is still present. What ordering or guard failure does this suggest?
A: It suggests the compensation message was processed out of order (before or without the forward effect being applied) or the compensation was not guarded to check the original effect's presence before acting — so it either no-op'd against absent state or the forward step landed afterward. The guard is to make compensation conditional on the original change actually existing, and to order messages (version file / sequence) so a compensation never overtakes its forward step. Compensation must be both idempotent and order-aware.

Q: Why must a compensation be safe to run even if the forward step it undoes never actually committed?
A: After a timeout you often cannot tell whether the forward step committed, so the saga may compensate "to be safe"; if the forward step never happened, a naive compensation could create a negative effect (e.g. releasing cash that was never reserved). Making compensation check current state and act only on what is present (or making it a no-op when there is nothing to undo) keeps it safe under that uncertainty. This is why compensations are written as conditional, idempotent reconciliations rather than blind inverses.

Q: In the subscription saga, give a concrete idempotent design for the "release cash reservation" compensation.
A: Model the reservation as a row keyed by the saga/order id with a status, and have the compensation execute `UPDATE reservations SET status='released' WHERE order_id=? AND status='reserved'`, returning the freed amount only on the transition. Re-running it finds status already `released` and changes nothing, so it is idempotent, and it releases exactly the amount this saga reserved rather than a remembered number. This guards against both double-release and dirty-compensation.

Q: Why is releasing "the amount this saga reserved" safer than releasing "the original requested amount" in a compensation?
A: The originally requested amount may differ from what was actually reserved (partial fills, fees, rounding), and other operations may have touched the balance, so releasing the request can over- or under-credit; releasing exactly the recorded reserved amount for this saga reverses precisely its own effect. Tying the compensation to the persisted reservation record, not the request, makes it correct under concurrency. This is the reread/own-effect discipline applied to money.

Q: How do you choose between modeling cash reservation as a balance decrement versus a separate reservation hold?
A: A separate hold (a reservation record that earmarks funds without mutating the spendable balance derivation) keeps the original balance event immutable and makes compensation a clean status flip, which suits an event-sourced, auditable register; a direct decrement is simpler but entangles the reservation with the balance and complicates compensation and audit. For a regulated TA platform the hold model is usually preferred because it leaves an explicit, reversible audit artefact. The hold also naturally carries the saga id for own-effect compensation.

Q: A choreography saga has grown to seven services and nobody can say what reacts to `UnitsBooked`. What concrete refactor addresses this?
A: Introduce an orchestrator so the flow becomes explicit and centrally visible, replacing the implicit web of event subscriptions with commands issued in a known order. This directly cures the "hard to follow / cyclic risk" drawbacks that bite as choreography grows. As a lighter first step you can document the event-to-handler map and add correlation-id tracing, but the durable fix for a complex flow is orchestration.

Q: How can you migrate an existing choreography saga to orchestration incrementally without a big-bang rewrite?
A: Stand up the orchestrator and have it drive the steps via commands while the participants keep their existing event handlers, then move one step at a time under orchestrator control, retiring the corresponding event subscription as each step is taken over. The outbox events remain the transport, so participants barely change. You shift the locus of control gradually, validating consistency at each step.

Q: Why is "the orchestrator issues commands, participants emit events" a clean division in an orchestrated saga?
A: Commands express the orchestrator's intent ("reserve cash") which a participant may accept or reject, while the participant's resulting event ("cash reserved" / "reservation rejected") is a fact the orchestrator consumes to decide the next step. This keeps intent (rejectable) and outcome (factual) cleanly separated and gives the orchestrator authoritative knowledge of saga state. Mixing the two — having the orchestrator listen to commands or participants issue commands at each other — reintroduces the choreography tangle.

Q: How does an orchestration engine like Temporal typically implement a saga step timeout, and why is that better than a cron sweep?
A: The engine has durable timers as a first-class workflow primitive: the workflow awaits a step result with a timeout, and if the timer fires first the engine deterministically resumes the workflow on the timeout branch to compensate or retry. This is better than an external cron sweeping for stuck rows because the timer is part of the durable, replayable workflow state and survives process restarts without a separate scanning job. The timeout logic lives with the saga, not in ad-hoc infrastructure.

Q: In Temporal terms, how do compensations usually get registered as the forward saga progresses?
A: As each forward activity succeeds, the workflow appends its corresponding compensation to a list (a saga/compensation stack), so that on a later failure the workflow runs the accumulated compensations in reverse order. This guarantees you only attempt to undo steps that actually ran, in last-in-first-out order. It is the engine-managed form of the "everything before the pivot is compensatable" rule.

Q: Why run compensations in reverse (LIFO) order of the forward steps?
A: Later steps may depend on the state established by earlier ones, so undoing in reverse order unwinds dependencies safely — you reverse the most recent effect first, the same way a stack unwinds. Releasing in forward order could try to undo an earlier step while a later step still relies on it. Reverse-order compensation mirrors how nested operations are correctly unwound.

Q: What is the "set-based" or batch hazard when one outbox event represents many aggregates, and how do you avoid it?
A: If a single business transaction touches many aggregates but you emit one coarse event, you lose the per-aggregate key needed for ordered, parallel consumption and idempotent application, and a partial failure is hard to compensate granularly. The fix is to emit one outbox row per aggregate change, each keyed by its `aggregateid`, so each is independently routed, ordered, and dedup'd. Granular events preserve the per-aggregate guarantees the pattern depends on.

Q: Your Debezium connector's replication slot is growing the WAL on the PostgreSQL primary. What is happening and what do you check?
A: A replication slot holds the WAL until the consumer (Debezium) confirms it has read past it, so if the connector is down, lagging, or its offsets are stuck, the slot's `confirmed_flush_lsn` does not advance and WAL accumulates, risking disk exhaustion. Check the connector is running and consuming, inspect `pg_replication_slots` for `active` and the retained WAL size, and remedy by restoring the connector or dropping an abandoned slot. An orphaned slot from a deleted connector is a classic cause.

Q: Why must you drop a stale Debezium replication slot after permanently removing a connector?
A: An inactive slot still pins WAL on the primary, so PostgreSQL keeps accumulating and cannot recycle log segments, eventually filling the disk and threatening the database. Removing the connector does not automatically drop its slot, so you must `SELECT pg_drop_replication_slot('slot_name')` yourself. This is a standard operational gotcha of CDC-based outbox relays.

Q: After a failover to a PostgreSQL replica, your Debezium outbox connector cannot find its replication slot. Why?
A: Logical replication slots are not, by default, replicated to physical standbys in older setups, so when the standby is promoted the slot does not exist there and the connector loses its position. Mitigations include failover slots / slot synchronization features in newer PostgreSQL versions or re-establishing the connector against the new primary. It is a known operational risk of WAL-tailing outbox relays across failovers.

Q: A duplicate event slipped past your idempotent consumer and double-booked units. What are the two most likely dedup bugs?
A: Either the dedup check and the business write were not in the same transaction (so a crash between them let a redelivery re-apply), or the dedup key was not actually stable/unique — e.g. you keyed on something regenerated per delivery rather than the immutable outbox `id`. Verify the processed-id insert and the booking commit atomically and that the key is the outbox `id`. A read-then-write check with a race window is a third possibility.

Q: Why is the outbox `id` (not the Kafka offset or partition+offset) the right dedup key for consumers?
A: The Kafka offset identifies a position in a topic, which changes across topic recreation, mirroring, or reprocessing, whereas the outbox `id` is the stable, business-level identity of the event that stays constant across any redelivery or replay path. Dedicating on offset would fail the moment the same event arrives via a different offset. The `id` is the contract; the offset is transport plumbing.

Q: How do you prevent a "phantom" event where the outbox row commits but the business change is later found invalid?
A: You keep them in one transaction so they share fate — if the business change rolls back, the outbox row rolls back with it, so no phantom is ever emitted. The whole pattern hinges on never writing the outbox row outside the business transaction. If you find yourself wanting to "cancel" an already-committed outbox event, that signals the outbox write escaped the transaction boundary.

Q: In a fund context, why is emitting `OrderCancelled` as a new outbox event preferable to deleting the original `OrderCreated` outbox row?
A: Append-only events preserve the full audit history — both the creation and the cancellation are facts the regulator and downstream consumers must see — whereas deleting the original row erases history and, in a CDC relay, the delete is filtered and emits nothing, so consumers never learn of the cancellation. You model lifecycle changes as new events, never as mutations of past ones. This is the immutability discipline that makes the log trustworthy.

Q: A consumer must enrich an order event with the investor's LEI before forwarding. Where does idempotency risk appear in this enrichment step?
A: If the consumer publishes the enriched event and then crashes before recording that it processed the source event, a redelivery re-enriches and re-publishes, duplicating downstream — so the enriched output needs the same idempotency discipline (dedup on source event id, or publish via its own outbox in the same transaction as recording the source id). Treat the enricher as both a consumer (must dedup input) and a producer (should use an outbox for its output). Otherwise the duplicate input becomes a duplicate output.

Q: Why is chaining outboxes (consumer writes its own outbox in the same transaction it records the processed id) the clean pattern for a multi-hop event flow?
A: Each hop then has the same atomic guarantee — recording "I handled input X" and emitting "here is output Y" commit together — so neither a lost output nor a re-emitted duplicate can arise from a crash between them. The at-least-once-plus-idempotent property composes cleanly across hops. Without it, every intermediate service reintroduces a dual-write between consuming and producing.

Q: How would you load-test that your saga's compensation path is correct under concurrency, not just in a single happy run?
A: Run many concurrent subscription sagas against the same investor/cash account while injecting step-2 failures, then assert from the event log that every failed saga released exactly its own reservation and the final balance equals the start balance minus only successfully booked subscriptions. Concurrency surfaces dirty-compensation and lost-update anomalies a single run hides. The check is on reconciled end state and per-saga own-effect releases, not on absence of errors.

Q: What metric best signals that compensations are silently misbehaving in production?
A: A persistent or growing count of "reserved but neither confirmed nor released" resources past their expected lifetime — dangling reservations — because correct sagas either confirm or compensate every reservation promptly. Tracking reservation age and a reconciliation between reserved holds and confirmed bookings exposes leaks. A nonzero, aging backlog means a compensation or timeout path is failing.

Q: Summarize the operational invariants you must monitor for an outbox-plus-CDC saga platform to be production-safe.
A: Watch replication-slot lag and retained WAL size (so the slot never fills the disk), connector liveness and consumer lag (so events keep flowing), dedup-store growth and consumer duplicate-handling (so idempotency holds), and dangling reservations / per-step saga latency (so stuck or uncompensated steps page someone). Together these cover the relay, the broker path, the consumer, and the saga's failure handling. Each maps to a concrete failure the lesson's design is meant to prevent — lost events, replays, and half-completed subscriptions.

Q: When does `route.tombstone.on.empty.payload` actually matter, and what must be true of the topic for the tombstone to do its job?
A: It matters when you model deletion of an aggregate by writing an outbox row with an empty/null payload and want a Kafka tombstone (null-value record) emitted so a compacted topic prunes that key; you set it to `true` to enable that. The target topic must be log-compacted for tombstones to remove the key — on a non-compacted topic the null record just sits there with no cleanup effect. With the default `false`, an empty payload yields no tombstone at all.

Q: A T+1 settlement deadline means a subscription saga that gets stuck must resolve fast. How does that shorten your timeout and alerting design?
A: Under `T+1` the cash-and-units lifecycle must complete by the next business day, so a saga step that hangs cannot sit for hours awaiting a manual sweep — the dead-man timeout and the dangling-reservation alert must fire well inside the settlement window, not after it. You size per-step timeouts and reservation-age alerts against the `T+1` cut-off so a stuck step is detected and compensated or escalated with time to spare. The compressed settlement cycle turns timely timeout handling from good practice into a regulatory necessity.

Q: How do you decide whether a given outbox/saga component is in scope of DORA's operational-resilience expectations?
A: `DORA` (the Digital Operational Resilience Act) targets ICT systems supporting critical financial functions, so the outbox relay, broker, and saga orchestrator that carry subscription/settlement events are squarely in scope because their failure disrupts a critical business service. That pulls in expectations for ICT risk management, incident detection/reporting, resilience testing, and oversight of third-party providers (e.g. the managed Kafka or CDC service). Concretely it means the monitoring, replay runbooks, and failover tests for this pipeline are not optional engineering niceties but documented resilience controls.
