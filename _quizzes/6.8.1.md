# Phase 7 · 6.8.1 LLM tracing & prompt management (Langfuse) — 100 self-test questions

Q: What is Langfuse, in one sentence?
A: Langfuse is an open-source LLMOps platform for tracing, evaluating, and managing prompts for LLM applications. It captures every model call with its inputs, outputs, cost, and latency so an otherwise opaque pipeline becomes observable and auditable. In this lesson you self-host it next to your RAG pipeline to answer "what exactly did the model see and answer?".

Q: Why is untracked LLM use a problem in a regulated fund-administration context?
A: Without traces, "what exactly did the model see when it extracted this SRI value from document X?" has no answer, so the extraction cannot be audited or defended to a regulator. Regulated workflows (UCITS KIDs, EMT/EPT data, NAV inputs) require reproducibility and lineage, and an LLM call that leaves no record is the LLMOps equivalent of an undocumented manual override. Langfuse traces turn that mystery into a lookup.

Q: What is a `trace` in the Langfuse data model?
A: A trace is the top-level container representing a single request or operation end-to-end — for example one question through your RAG pipeline. It holds all the nested observations (retrieval, generation, tool calls) for that request, plus trace-level metadata like `user_id`, `session_id`, and tags. You typically create one trace per user question so the whole flow is grouped under one record.

Q: What is an `observation` in Langfuse, and how does it relate to a trace?
A: An observation is an individual step inside a trace, such as a retrieval step or an LLM call, and observations can be nested to mirror your pipeline's structure. A trace is the container; observations are its children. Langfuse defines LLM-specific observation types like generations, spans, and events to give each step the right shape.

Q: Name the three observation types you will use most when instrumenting a RAG pipeline.
A: `span` for arbitrary units of work like a retrieval step, `generation` for an LLM call that has model/usage/cost, and `event` for a single point-in-time marker. In a RAG trace you typically wrap retrieval in a span and the model call in a generation. Choosing the right type matters because only generations carry token-usage and cost semantics.

Q: What distinguishes a `generation` from a plain `span`?
A: A generation is a specialised observation for an LLM (or embedding) call and carries model-specific fields: `model`, `input`/`output` messages, `usage_details` (token counts), and `cost_details`. A span is a generic unit of work with no built-in notion of tokens or model. Use a generation for the actual model call so cost and latency telemetry aggregate correctly; use a span for retrieval, parsing, or orchestration steps.

Q: What is a `session` in Langfuse used for?
A: A session is an optional grouping that ties together multiple traces belonging to the same user interaction or conversation. For a multi-turn chat, each turn is its own trace but all share one `session_id`, letting you view the whole conversation. For single-shot KID extraction you often won't need sessions, but they help when one document drives several follow-up questions.

Q: In Langfuse v3, what underlying standard is the Python SDK built on?
A: The Python SDK v3 is built on OpenTelemetry (OTEL), so Langfuse observations are real OTEL spans under the hood wrapped by convenience objects like `LangfuseSpan` and `LangfuseGeneration`. This means context propagation (parent/child nesting) is handled by OTEL automatically, and spans from other OTEL-instrumented libraries can be captured inside your Langfuse traces. SDK v3 reached General Availability in mid-2025.

Q: What practical benefit does the OTEL foundation give you when other libraries are already instrumented?
A: Because v3 rides on OpenTelemetry, libraries already emitting OTEL spans (for example an instrumented HTTP or vector-store client) have their spans picked up and correctly nested inside your Langfuse trace without extra glue code. You get a unified view spanning your code and third-party calls. This is why the SDK can support many languages without hand-writing a bespoke SDK per language.

Q: What does the `@observe()` decorator do?
A: `@observe()` wraps a function so Langfuse automatically captures its inputs, outputs, timing, and any errors as an observation, without you editing the function body. It works on sync and async functions and accepts parameters such as `name` and `as_type` (for example `as_type="generation"`). It is the fastest way to instrument an existing pipeline function with one line above it.

Q: How do you turn an `@observe`-decorated function into a generation rather than a span?
A: Pass `as_type="generation"` to the decorator, e.g. `@observe(as_type="generation")`, so the resulting observation carries model/usage/cost semantics. You then set `model`, `usage_details`, and `cost_details` from inside the function via the update helpers. Without `as_type`, `@observe` produces a generic span that cannot record token usage.

Q: What does `langfuse.start_as_current_observation()` (or `start_as_current_span`) give you?
A: It opens a context manager that creates an observation and makes it the active OTEL context for the duration of the `with` block, so any child observations created inside automatically nest under it. This is how you build the trace tree explicitly rather than via decorators. When the block exits, the observation's end time and any final attributes are recorded.

Q: How do you set attributes on the currently active observation without holding a reference to it?
A: Use `update_current_span()` / `update_current_observation()` (and `update_current_trace()` for trace-level fields), which mutate whatever observation is active in the current context. This is convenient inside an `@observe` function or a `with` block where you don't have the object in a variable. For example you can attach `output=...` or `usage_details=...` mid-function.

Q: What is `get_client()` for in the v3 SDK?
A: `get_client()` returns the configured singleton Langfuse client so you can create observations, fetch prompts, or flush from anywhere in your code without re-instantiating. It reads configuration (host and keys) from environment variables or a prior `Langfuse(...)` initialisation. Using the shared client avoids spinning up multiple exporters and duplicate background threads.

Q: Which three environment variables typically configure the Langfuse client?
A: `LANGFUSE_PUBLIC_KEY`, `LANGFUSE_SECRET_KEY`, and `LANGFUSE_HOST` (the base URL of your self-hosted instance, e.g. `http://localhost:3000`). The public/secret pair scopes the client to one project, and the host points at your Docker-Compose deployment rather than Langfuse Cloud. Setting these lets `get_client()` work with no arguments.

Q: Why must short-lived scripts call `flush()` before exiting?
A: Langfuse batches observations and sends them asynchronously on a background thread, so a process that exits immediately can terminate before the buffer is sent, silently losing traces. `flush()` forces all buffered observations to be transmitted and blocks until done. For a one-shot extraction script over 20 questions, call `langfuse.flush()` (or `shutdown()`) at the end.

Q: What is the difference between `flush()` and `shutdown()`?
A: `flush()` sends all currently buffered observations and returns, leaving the client usable afterwards. `shutdown()` flushes and then gracefully stops the background worker threads, so it is what you call once at true program exit. In a long-running service you might `flush()` periodically; in a batch job you `shutdown()` at the end.

Q: How do you record per-call token usage on a generation?
A: Set `usage_details` on the generation, e.g. `usage_details={"input": 1200, "output": 180}` (or `input_tokens`/`output_tokens` keys), either via `.update()` on the generation object or `update_current_observation()`. Langfuse stores these counts per generation and aggregates them across the project. Accurate token capture is the basis for the cost and chargeback dashboards.

Q: How does Langfuse arrive at a cost figure for a generation?
A: It can either take the cost you supply directly via `cost_details`, or infer cost by multiplying the recorded `usage_details` by the unit prices for the named `model`, using its model-price definitions. So if you set `model="gpt-4o"` and provide token counts, Langfuse computes the dollar cost from its price table. For a custom or self-hosted model, you supply `cost_details` or define the model pricing yourself.

Q: A generation shows zero cost despite many tokens — what is the first thing to check?
A: Check that the generation's `model` field is set to a name Langfuse recognises in its model-price list (a typo or a custom model name means no price match, so cost stays zero). Then confirm `usage_details` actually populated with non-zero token counts. If you use a self-hosted model with no built-in price, define a model-price mapping in Langfuse or pass explicit `cost_details`.

Q: What is captured for latency, and how?
A: Latency is derived from each observation's start and end timestamps, which the SDK records automatically when you use `@observe`, a `with` context manager, or explicit start/end calls. The trace's total latency and each span/generation's duration appear in the UI without extra instrumentation. This is what lets you compare wall-clock cost between two prompt versions straight from the traces.

Q: How would you instrument the 6.6.2 RAG pipeline so each question is fully traceable?
A: Create one trace per question, then within it open a `span` around retrieval (recording the query and the retrieved chunk ids) and a `generation` around the LLM call (recording `model`, `input`, `output`, `usage_details`, and `cost_details`). Use `@observe` on the retrieve and generate functions or `start_as_current_*` context managers to get correct nesting. The result is a trace tree showing exactly what was retrieved and what the model produced.

Q: After instrumenting, how do you answer "what did the model see and answer for document X last Tuesday?"
A: You search the Langfuse UI by trace metadata — for example the document id you attached as a tag or metadata field, filtered to last Tuesday's date range — and open the generation to read its exact `input` and `output`. Because the prompt, retrieved context, and model output are all persisted per call, it becomes a lookup rather than a reconstruction. This directly satisfies the lesson's first "Done when" criterion.

Q: Why attach the source document id as trace metadata or a tag?
A: Without a stable business identifier on the trace, you cannot later find the call that processed a specific KID or ISIN — you would be scrolling blind. Attaching the document id (and perhaps the ISIN) as `metadata` or a tag makes every trace queryable by the artifact it acted on. In a fund context this links the LLM call back to the exact instrument or filing under audit.

Q: What is Langfuse prompt management, conceptually?
A: It treats prompts as managed, versioned artifacts stored in Langfuse and fetched at runtime, rather than as string literals baked into code. Each save creates a new immutable version, and labels point at the version that should be served. This decouples a prompt change from a code deploy and gives you an auditable history of every wording change.

Q: Why move the extraction prompt out of code and into Langfuse prompt management?
A: Because an untracked prompt edit can silently change extraction behaviour across the whole corpus with no record of who changed what or when — exactly the kind of undocumented change a regulated workflow forbids. Centralising the prompt gives version history, staged rollout, and rollback, and lets non-engineers iterate without a code deploy. It also lets you link each generation to the precise prompt version that produced it.

Q: How do you fetch a prompt at runtime in the Python SDK?
A: Call `langfuse.get_prompt("extraction-prompt")` on the client returned by `get_client()`; by default this returns the version currently carrying the `production` label. You can override with `label=` or `version=` arguments. The returned prompt object is then compiled with your variables before being sent to the model.

Q: How do you fill a fetched prompt template with runtime values?
A: Call `.compile(...)` on the prompt object, passing the variables as keyword arguments, e.g. `prompt.compile(criticlevel="expert", movie="Dune 2")`. Templates use `{{variable}}` placeholders, which `compile` substitutes. The compiled string (or message list for chat prompts) is what you pass to the model call.

Q: What two prompt types does Langfuse support, and how do they differ?
A: `text` prompts are a single string template with `{{variable}}` placeholders, while `chat` prompts are an array of role/content message objects (system, user, assistant). You choose the type at creation with `type="text"` or `type="chat"`. Chat prompts let you version the full message structure, which matters for models that distinguish system and user roles.

Q: What does a Langfuse prompt `label` actually do?
A: A label is a movable pointer to a specific version, so `production` (or `staging`, `latest`) names whichever version should currently serve that role. Fetching by label means your code asks for "the production prompt" and Langfuse resolves it to the right version. Promoting a new version is just moving the label, which is what enables staged rollout and instant rollback.

Q: How do you roll back a bad prompt change with Langfuse?
A: Move the `production` label back to the previous, known-good version; because versions are immutable and the old one still exists, the rollback is instantaneous and leaves an audit trail. Your runtime code is unchanged — it always fetches by label — so no redeploy is needed. This is the operational payoff of fetching prompts by label rather than pinning a version in code.

Q: How do you create a new prompt version in code, and how is versioning numbered?
A: Call `langfuse.create_prompt(name=..., type=..., prompt=..., labels=[...])`; reusing an existing `name` automatically creates the next version of that prompt. Version numbers increment automatically and each version is immutable. You can attach labels at creation (for example promote straight to `production`) or move labels later.

Q: For the lesson's exercise, how do you make a v2 of the extraction prompt and serve it intentionally?
A: Create a second version of the same prompt name with a deliberate wording change (this becomes v2), then assign it a label such as `production` or a dedicated `experiment` label. At runtime fetch by that label so you control which version is live. This is exactly step 3 of the Do list: a v2 with a deliberate change, fetched by label.

Q: Why fetch a prompt by label rather than by a hard-coded version number?
A: Fetching by label decouples "which version is live" from your code, so promoting or rolling back is a metadata change in Langfuse rather than a deploy. Pinning `version=2` in code means every rollout requires a code change and redeploy, defeating the point of prompt management. Labels are how you do staged rollout safely.

Q: What is the default client-side prompt cache behaviour, and why does it exist?
A: The SDK caches fetched prompts client-side with a default TTL of around 60 seconds, so most `get_prompt` calls hit the cache instead of the network. When the TTL expires it serves the stale prompt immediately and revalidates in the background, so latency stays low. This keeps prompt fetching from adding a network round-trip to every request.

Q: You moved the `production` label to a new version but your app still serves the old prompt — what is the likely cause?
A: The client-side cache is still serving the previous version until its TTL expires, so the change isn't picked up instantly. Wait for the TTL, lower `cache_ttl` for faster propagation, or restart the process to clear the cache. This is a well-known gotcha documented in Langfuse's FAQ on stale prompt caching.

Q: How do you control or disable the prompt cache?
A: Pass `cache_ttl_seconds` (the per-fetch cache TTL) to `get_prompt`; set it to `0` to bypass the cache and always fetch fresh, or to a larger value to cache longer. A `0` TTL trades a network round-trip for immediate label propagation. Choose based on how quickly you need rollouts to take effect versus how much latency you can absorb.

Q: What is the fallback prompt feature and why does it matter for availability?
A: You can pass a `fallback` prompt (a template string or chat messages) to `get_prompt`, used if the Langfuse API is unreachable and the cache is empty. This guarantees your pipeline still has a prompt to run even during a Langfuse outage, giving effectively 100% availability. It's a resilience pattern: prompt management must never become a single point of failure for production extraction.

Q: How do you tie a generation to the exact prompt version that produced it?
A: Pass the fetched prompt object to the generation when you create it (the SDK links them), so the trace records which prompt name and version was used for that call. In the UI you can then jump from a generation to its prompt version and vice versa. This linkage is what lets you compare cost and quality per prompt version directly from traces.

Q: After running the 20-question set under v1 and v2, how do you compare them from traces alone?
A: Filter traces by the prompt version (or the experiment label) attached to each generation, then read the aggregated cost and token metrics per version, and inspect outputs for quality. Because each generation is linked to its prompt version and carries `usage_details`/`cost_details`, the cost/quality comparison comes from the trace store, not from memory. This satisfies the "show two prompt versions with a cost/quality comparison derived from traces" criterion.

Q: What is PII masking in Langfuse and when does it run?
A: Masking is a hook that redacts sensitive fields from observation inputs, outputs, and metadata before the data is persisted to the Langfuse backend. You register a custom `mask` function on the client; it runs synchronously as observations are created, so masked content never leaves your process unmasked. This is how you keep investor names and account numbers out of the trace store.

Q: How do you register a masking function with the Langfuse client?
A: Pass it as the `mask` argument to the client constructor, e.g. `Langfuse(mask=masking_function)`; from then on it is applied automatically to all trace data. There is no per-call wiring needed once it's set on the client. In v3 the function runs as spans are created, before export.

Q: What is the contract of a Langfuse mask function — what does it receive and return?
A: It receives `data` (the value being logged, of any type) plus optional `**kwargs`, and must return the masked equivalent, ideally preserving the original type (string stays string, dict stays dict). For nested structures you typically recurse so emails, names, or account numbers are redacted at every level. Returning the wrong type or shape can corrupt how the trace renders.

Q: For the fund-data exercise, what would your mask function specifically remove?
A: Investor names and any other PII that appears in retrieved context or model output — for example beneficial-owner names, addresses, and account/IBAN numbers — replacing them with a redaction token before persistence. You verify by inspecting raw traces in the UI and confirming none of those fields appear. This meets the "masked fields appear nowhere in the trace store" criterion.

Q: Why is "don't log what you masked elsewhere" a critical rule for traces?
A: If you carefully mask PII in your pipeline outputs but then log the unmasked model input/output into traces, you have just re-created the leak in a second store that is often less access-controlled. The trace store must respect the same masking as the primary data flow. Langfuse's `mask` hook exists precisely to enforce this at the observability boundary.

Q: A reviewer reports masked data is appearing in a trace — what is the first thing to check?
A: Confirm the `mask` function is actually registered on the client that created those traces (a second un-masked client instance is a common cause), then check the function handles the data type/shape of that field, since a recursion gap can let nested values through. Also verify the masking didn't throw silently. Reproduce by inspecting the raw trace JSON, not just the rendered view.

Q: Why test the mask function against nested dicts and lists, not just strings?
A: LLM inputs and outputs are frequently structured (chat message arrays, tool-call JSON, retrieved-chunk lists), so a mask that only handles top-level strings will leak PII buried inside those structures. Robust masking recurses through dicts and lists and type-checks each value. The Langfuse docs explicitly show recursive processing for complex structures for this reason.

Q: What is a `score` in Langfuse?
A: A score is a numeric, boolean, or categorical evaluation attached to a trace or a specific observation, used to capture quality signals such as human feedback, an LLM-judge result, or a metric. Scores let you quantify and filter traces by quality alongside cost and latency. They are how reviewer judgments become structured data on each call.

Q: How does user feedback get captured in Langfuse, and why does it matter for 6.8.2?
A: Reviewers (or end users) attach scores to traces — for example a thumbs-up/down or a correctness rating — which Langfuse stores against that trace or observation. These accumulated scores become a labelled dataset of model behaviour. That labelled feedback is exactly the seed for the golden-set evaluation work in lesson 6.8.2.

Q: How can a score be attached to a specific generation rather than the whole trace?
A: You target the observation by its id when creating the score (or use the current-observation helpers), so the rating lands on that generation instead of the trace container. This matters when one trace has several model calls and only one is being judged. Observation-level scores give finer-grained quality data than trace-level ones.

Q: Why is captured human feedback described as "tomorrow's eval data"?
A: Each reviewer score is a human label on a real production input/output pair, and a collection of such labels is precisely what you need to build or validate a golden set and to calibrate an LLM-as-judge. So feedback gathered today through Langfuse scores becomes the training/validation signal for the evaluation system in 6.8.2. It turns ad-hoc review into reusable ground truth.

Q: What does "self-host Langfuse next to the RAG pipeline" entail in this lesson?
A: Running Langfuse via Docker Compose on your Ubuntu machine so the trace backend lives alongside the pipeline it observes, then creating a project and generating API keys. Self-hosting keeps potentially sensitive fund data on infrastructure you control rather than sending it to a cloud SaaS. The keys and host URL then configure your SDK client.

Q: What core services does a Langfuse self-hosted deployment include?
A: A Langfuse web/server application backed by a database stack — Postgres for transactional data plus, in recent versions, ClickHouse for high-volume trace/analytics storage, with Redis and an S3-compatible blob store. Docker Compose wires these together for a local deployment. You don't hand-manage them individually; the compose file brings the stack up.

Q: After `docker compose up`, how do you create a project and get API keys?
A: Open the Langfuse web UI (by default `http://localhost:3000`), sign in, create an organisation/project, and in the project settings generate a public/secret API key pair. Those keys plus the host URL go into your environment so the SDK can authenticate. Each project scopes its own traces and prompts.

Q: A self-hosted Langfuse trace shows nothing despite running your pipeline — what do you check first?
A: Verify the SDK is pointed at your self-hosted `LANGFUSE_HOST` (not the default cloud host) and that the public/secret keys match the project you're viewing. Then confirm the process called `flush()`/`shutdown()` before exiting so the buffer was actually sent. A keys/host mismatch or an un-flushed short-lived script are the two most common causes.

Q: Why prefer self-hosting Langfuse over the SaaS cloud in a regulated fund context?
A: Self-hosting keeps trace contents — which may include extracted investor data and document text — inside your own controlled environment, satisfying data-residency and confidentiality requirements (relevant under DORA and Luxembourg fund-data rules). You also control retention and access. The trade-off is you operate the stack yourself, but for regulated PII that control is usually mandatory.

Q: How is Langfuse's role analogous to the data-lineage work earlier in the plan?
A: Lineage tracks how a data value flowed through your batch/streaming pipelines so any figure can be traced to its source; Langfuse tracing does the same for LLM calls, tying each output to the exact prompt, context, model, and version that produced it. It is the LLMOps analogue of column-level lineage. Both exist so that "where did this number come from?" is always answerable.

Q: How would you record cost telemetry suitable for per-team chargeback?
A: Attach a team or project identifier to traces (via `metadata` or tags) and ensure every generation has accurate `usage_details`/`cost_details`, then aggregate cost in Langfuse grouped by that identifier. Because cost is captured per call, the platform can sum it per team for chargeback. This is the operational use of the cost/latency telemetry bullet.

Q: How do traces help you define and monitor an SLO for the extraction service?
A: Each observation's latency and each generation's cost are recorded per call, so you can aggregate p95 latency and cost over time and set thresholds (the SLO) on those aggregates. Breaches become visible in the trace metrics rather than discovered by users. This makes latency and cost first-class, measurable service properties.

Q: What does `propagate_attributes()` (or trace-level update with `user_id`/`session_id`) accomplish?
A: It sets attributes like `user_id`, `session_id`, `metadata`, `version`, and tags so they propagate to the trace and its nested observations. This lets you later filter and group traces by user, session, app version, or business key. For example tagging a trace with the document version lets you slice cost by document cohort.

Q: Why tag traces with an application `version`?
A: Tagging the app or pipeline `version` lets you correlate a change in cost, latency, or quality with the deploy that caused it. When a regression appears you can filter traces to "before vs after vX" to confirm the culprit. It's the trace-side analogue of release tagging in your code.

Q: How do labels enable a staged (canary) rollout of a prompt?
A: You can point a small slice of traffic at a new version using a label your code selects for a subset of requests (e.g. `canary`), keep the rest on `production`, then compare metrics per label before promoting. Because fetching is label-driven, shifting traffic is a metadata change, not a deploy. Once the canary's cost/quality looks good you move `production` to it.

Q: In the trace tree, where should retrieved chunk ids and the query live?
A: On the retrieval `span`'s input/output — the query as input, the retrieved chunk ids (and optionally scores) as output — so the span records exactly what context fed the generation. Keeping this on the span, separate from the generation, preserves the pipeline structure. It lets you later inspect whether a wrong answer was a retrieval failure or a generation failure.

Q: A generation's recorded output doesn't match what your app returned to the user — what likely happened?
A: Most often the output was logged at the wrong point (before post-processing/parsing) or a masking function altered it, so the trace shows the raw model output while the app returned a transformed value. Check where in the code you set `output` and whether the `mask` function rewrote it. Align the logged value with the value the contract actually cares about.

Q: Why is one-trace-per-question better than one-trace-per-batch for the 20-question set?
A: One trace per question keeps each question's retrieval, generation, cost, and any score isolated and individually queryable, so you can compare per-question behaviour across prompt versions. A single trace for the whole batch flattens 20 distinct flows into one, losing per-question cost and quality granularity. Granularity is exactly what you need for the v1-vs-v2 comparison.

Q: How would you confirm that v2 of the prompt actually changed model behaviour and not just cost?
A: Compare the linked generations' outputs (and any quality scores) between the v1-labelled and v2-labelled traces over the same 20 questions, not just their token/cost totals. A wording change can alter cost without changing answers, or change answers without much cost — you need both dimensions. The per-version trace comparison surfaces both.

Q: What is the risk of editing a prompt directly in code instead of through Langfuse versioning?
A: A direct code edit changes extraction behaviour across the entire corpus with no immutable version record, no label-based rollback, and no link from existing traces to "which wording produced this". If the change degrades ISIN or SRI extraction, you can't cleanly revert or prove what ran before. Versioned prompts in Langfuse remove that risk.

Q: Why is `chat`-type prompt versioning useful when extracting fields from a KID?
A: A chat prompt lets you version the system instruction (role, output schema, refusal rules) separately from the user content, which is where most extraction-behaviour changes live. Versioning the whole message array means a system-prompt tweak is captured and rollback-able like any other change. This is cleaner than concatenating everything into one text blob.

Q: What is the relationship between Langfuse `generation` cost and the lesson's chargeback goal?
A: Because each generation records token usage and a derived (or supplied) cost, summing those costs across traces — grouped by team, document type, or prompt version — produces the chargeback figures. Without per-generation cost you'd be guessing from invoices. The data model makes cost an attributable, per-call fact.

Q: How do you verify masking worked "in the trace store" specifically, not just in the UI rendering?
A: Inspect the raw stored trace data (via the UI's raw/JSON view or the API) for the affected observations and confirm the PII tokens are absent, since the rendered view could theoretically hide rather than remove data. The masked value should be the redaction token at rest. Checking the raw record is what the "appear nowhere in the trace store" criterion demands.

Q: Why might you mask in the SDK rather than rely on a self-hosted "data masking" feature alone?
A: SDK-side masking redacts before the data ever leaves your process, so PII never traverses the network or lands in the backend at all, whereas server-side masking still receives the raw data first. For strict PII (investor names, account numbers) keeping it out of transit and storage entirely is safer. The two can be combined, but client-side masking is the stronger guarantee.

Q: What goes wrong if your mask function raises an exception?
A: An unhandled exception in the mask hook can break observation creation or, worse, cause the SDK to skip masking and risk leaking the raw value — so the function must be defensive and never throw on unexpected types. Wrap risky operations and default to redaction on uncertainty. Test it against the actual shapes your pipeline produces (chat arrays, tool JSON, None values).

Q: How do you keep cost telemetry accurate when calling a self-hosted or local model?
A: Since Langfuse may not have built-in pricing for a custom model name, either provide explicit `cost_details` on each generation or define a model-price entry so token counts convert to cost. Otherwise cost shows zero even with real usage. For a local Llama-style model you typically supply your own cost basis or accept token-only telemetry.

Q: Why record both `input` and `output` on every generation, even when verbose?
A: The whole point of the trace is to answer "what did the model see and what did it produce" for audit and debugging, which requires the exact input (compiled prompt plus retrieved context) and exact output. Omitting either breaks reproducibility. Masking handles the privacy concern so you can keep the full record safely.

Q: How does linking prompts to generations help during an incident review?
A: When an extraction goes wrong, you open the failing generation and jump straight to the exact prompt name and version it used, so you can see whether a recent prompt change is implicated. Without the link you'd have to reconstruct which wording was live at that time. The linkage turns "which prompt ran?" into one click.

Q: What is the difference between a label and a version, in one line each?
A: A version is an immutable, numbered snapshot of a prompt's content; a label (like `production`) is a movable pointer that names which version currently plays a role. You promote and roll back by moving labels, never by editing a version. Code fetches by label so it always gets the intended version.

Q: Why is immutability of prompt versions important for auditability?
A: If a version could be edited in place, a trace claiming "produced by v2" could no longer be trusted because v2's content might have changed since. Immutable versions guarantee that the wording linked to any historical generation is exactly what ran. This is essential for defending an extraction to a regulator months later.

Q: How would you measure whether v2 of a prompt is cheaper per question than v1?
A: Aggregate `cost_details` (or token usage) across the v2-labelled generations and compare the mean per question to the v1-labelled set over the identical 20-question corpus. Because cost is recorded per generation and generations are linked to prompt versions, this is a direct query in Langfuse. No spreadsheet of manual estimates is needed.

Q: What's the danger of relying on the prompt cache during a regulated rollout?
A: Different processes may hold different cached versions of `production` until their TTLs expire, so for a brief window your fleet runs a mix of old and new wording — undesirable when you need a clean, auditable cutover. Lower `cache_ttl` or restart workers to force convergence at the moment of promotion. Know the cache exists so a "stale prompt" surprise doesn't undermine the rollout record.

Q: How do you ensure a trace can be located by ISIN later?
A: Attach the ISIN as a tag or in `metadata` on the trace (and validate it's a well-formed 12-character ISIN before logging), so the trace is queryable by that instrument identifier. Then a query for that ISIN returns every LLM call that touched it. This connects your LLM observability to the same business keys used across the fund-data platform.

Q: Why might you also capture the source filename and document version in trace metadata?
A: Because the same instrument can have multiple KID revisions, the filename and document version disambiguate which exact document the model read, which matters when extraction results differ across revisions. Without it, "document X" is ambiguous. Logging it makes the "what did the model see for document X" lookup precise.

Q: How does the trace model support comparing retrieval quality separately from generation quality?
A: Because retrieval lives in its own span (with the query and retrieved chunks) and generation lives in its own generation observation, you can inspect each independently. A wrong field can then be attributed to bad retrieval (the right chunk wasn't fetched) versus bad generation (the chunk was there but the model misread it). This separation is the first diagnostic step when an extraction fails.

Q: What is the benefit of using `@observe` over manual `start_as_current_*` for existing pipeline functions?
A: `@observe` instruments a function with a single decorator and auto-captures inputs, outputs, timing, and errors, so you touch the least code to get a trace. Manual context managers give finer control (custom names, mid-block attributes, partial nesting) at the cost of more code. A common pattern is `@observe` for coarse steps and explicit context managers where you need precise control.

Q: How do you attach the model name to a generation, and why does it matter beyond display?
A: Set `model="gpt-4o"` (or whatever you call) on the generation; beyond labelling, the model name is what Langfuse uses to look up unit prices and convert token usage into cost. A wrong or missing model name therefore breaks cost telemetry, not just the label. Always set it from the actual model invoked.

Q: Your traces appear but have no cost despite correct model and tokens — what subtle issue could cause this?
A: The `model` name may not exactly match Langfuse's model-price table entry (case, suffix, or a date-stamped variant), so no price matches and cost stays zero even though tokens are present. Align the name to a known price entry or define a custom model price. This is distinct from the missing-tokens case where `usage_details` itself is empty.

Q: Why keep prompts in Langfuse rather than in a Git file, given Git also versions text?
A: Git versions the prompt but couples a change to a code deploy and offers no runtime label-based rollout, no link to live traces, and no aggregated cost/quality per version. Langfuse fetches at runtime so prompt changes go live (or roll back) without redeploying, and every generation links to the version used. The two are complementary, but runtime management is what Langfuse adds.

Q: How does capturing reviewer scores close the loop toward continuous evaluation?
A: Reviewers score live traces, those scores accumulate as labelled examples, and that labelled data feeds the golden-set and CI-gate evaluation built in 6.8.2 — so production review continuously enriches the eval set. Each judged trace makes tomorrow's regression test stronger. Tracing and feedback are the upstream half of the eval pipeline.

Q: What's the simplest way to confirm two prompt versions were actually exercised over the 20 questions?
A: Filter traces by each prompt version (or experiment label) and confirm each filter returns 20 generations, then spot-check that the inputs were the same questions. Because generations are linked to versions, the counts and inputs are directly visible. This guards against accidentally running all 40 traces under one version.

Q: How does Langfuse help defend the statement "this SRI value was extracted correctly"?
A: You open the generation that produced the SRI value and read the exact compiled prompt, the retrieved KID context, the model, and the output — the full provenance of that extraction. If challenged, you can show precisely what the model saw and which prompt version produced it. That evidentiary trail is the regulated-context payoff of the whole lesson.

Q: Why is it risky to log full retrieved context without masking in a fund pipeline?
A: Retrieved KID or filing chunks can contain investor names, account numbers, or other PII, which would then be persisted into the trace store unmasked. The `mask` hook must process the generation/span input so that context is redacted before storage. Otherwise the observability layer becomes a new uncontrolled copy of regulated data.

Q: What is the practical meaning of "stale-while-revalidate" in the prompt cache?
A: When a cached prompt's TTL expires, the SDK returns the stale (cached) prompt immediately for the current call and refreshes it from the server in the background, so no request pays the network latency. The next call gets the refreshed version. It optimises latency at the cost of a brief window of staleness after a label move.

Q: How would you organise traces so two prompt experiments don't get confused?
A: Tag each generation (or trace) with the prompt version or a distinct experiment label, and optionally use distinct trace tags per experiment, so each experiment's traces are cleanly filterable. The prompt-to-generation link does much of this automatically. Clean tagging is what makes the per-version cost/quality comparison trustworthy.

Q: Why are token counts not enough on their own for cost, and what completes the picture?
A: Token counts must be multiplied by per-token prices, which differ by model and by input-vs-output direction, so the same token count costs different amounts on different models. Langfuse completes the picture by mapping `(model, usage_details)` to `cost_details` via its price table, or you supply `cost_details` directly. Direction-aware pricing is why `input`/`output` token counts are recorded separately.

Q: What should you check first if `get_prompt` is slow on every call?
A: Verify the cache is actually being used — an unusually low or zero `cache_ttl` forces a network fetch every time, adding latency to each request. Restore a sensible TTL so most calls hit the cache. Also confirm you aren't constructing a fresh client per call, which would defeat caching; use the shared `get_client()`.

Q: How does the trace model make an LLM pipeline auditable in the DORA/operational-resilience sense?
A: Every model interaction is recorded with inputs, outputs, model, cost, latency, prompt version, and any human scores, giving a complete, queryable operational record of the LLM component. That record supports incident reconstruction, change attribution, and evidence of controls — the kind of operational resilience and auditability DORA expects. It turns the LLM from an opaque dependency into a monitored, governed service.

Q: What is the single biggest reason an instrumented batch job ends with empty traces?
A: The process exited before the asynchronous send completed because it never called `flush()` or `shutdown()`, so the buffered observations were dropped on exit. Always flush/shutdown at the end of a short-lived script. This is the most common first-day Langfuse mistake and the first thing to check when "I ran it but see nothing".

Q: How does grouping multi-turn follow-up questions about one KID under a `session_id` help review?
A: A `session_id` ties the separate traces (one per follow-up question) into a single conversation view, so a reviewer can read the whole exchange about that document in order rather than hunting for related traces. This is useful when extracting one field prompts a clarifying follow-up. For single-shot extraction you can skip sessions, but for interactive review they restore conversational context.

Q: Why might you set a longer `cache_ttl` in a stable production pipeline but a short one during active prompt iteration?
A: A long TTL minimises network fetches and latency when the `production` label rarely moves, which suits a settled pipeline. During iteration you want label changes to take effect almost immediately, so a short (or zero) TTL trades a little latency for fast propagation of each new version. Matching the TTL to your change cadence avoids both stale-prompt surprises and needless fetches.

Q: What is the first diagnostic you reach for when an extracted ISIN is wrong but the trace looks complete?
A: Open the trace and compare the retrieval span's output against the generation's input and output: if the correct ISIN never appeared in the retrieved chunks, it's a retrieval failure; if it was present but the model returned something else, it's a generation/prompt failure. The separated span/generation structure makes this attribution immediate. From there you fix either retrieval (chunking, embeddings) or the prompt version accordingly.

Q: Summarise how the five Learn topics fit together into one auditable LLM workflow.
A: The trace model records every call (traces/spans/generations); prompt management versions and labels the wording so changes are controlled and reversible; cost/latency telemetry attributes spend and performance per call; masking keeps PII out of the trace store; and feedback scores capture human judgment that seeds evaluation. Together they make "what did the model see, what did it cost, which prompt produced it, and was it right?" all answerable from one system — the LLMOps analogue of lineage plus quality control.
